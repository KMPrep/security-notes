Model Provenance and Signing — AI Supply-Chain Security

Domain: AI / ML Supply Chain
Frameworks: OWASP LLM03 (Supply Chain), Google SAIF, NIST AI RMF, SLSA (provenance model), MITRE ATLAS (ML Supply Chain Compromise)

Overview

Modern AI systems rarely train models from scratch — they pull pretrained weights, adapters, and datasets from public hubs like Hugging Face, PyTorch Hub, and TensorFlow Hub. A model artifact is not passive data: common formats such as Python pickle (.pkl, .pt, .bin, .ckpt) execute arbitrary code on load, so downloading an untrusted checkpoint is equivalent to running an untrusted binary. This is the core of OWASP LLM03: Supply Chain, and it maps to MITRE ATLAS "ML Supply Chain Compromise." Provenance and signing answer two questions every production loader must ask: who produced this artifact, and has it been tampered with since? The defensive stack is safer serialization (safetensors), pre-load scanning, cryptographic signing/verification (sigstore, model transparency logs), pinned and vetted sources, and isolated loading environments.

How It Works

Core Mechanics

The ML supply chain has several trust boundaries, and model artifacts cross most of them: a producer trains or fine-tunes a model, serializes it to a file, and publishes it to a hub; a consumer downloads the file and deserializes it into a process that often holds credentials, GPUs, and network access. The dangerous step is deserialization. Python's pickle protocol allows an object to define __reduce__, which tells the unpickler to call an arbitrary callable (like os.system or builtins.eval) with attacker-controlled arguments during load — before any inference runs. Formats built on pickle inherit this: torch.load uses pickle by default, and many .bin/.ckpt files are pickle archives. So "loading a model" can mean "executing the model author's code."

Provenance is the countermeasure to the who question, and integrity to the tampered question. Provenance is verifiable metadata about how an artifact was produced — the source repo, commit, builder, training pipeline, and inputs — ideally captured as a signed attestation (the SLSA model). Integrity is a cryptographic binding between an identity and the exact bytes, produced by signing a digest of the artifact. safetensors removes the code-execution risk at the format layer by storing only tensors and a JSON header with no executable opcodes, so a malicious file cannot run code merely by being loaded. Signing (sigstore/cosign, Hugging Face's model signing, transparency logs like Rekor) lets a consumer verify that a trusted identity vouched for these exact bytes, and that the signature is publicly logged so a producer cannot silently backdate or hide it.

Top 3 Critical Mitigations

1. Use safetensors and Refuse Executable Formats

The most durable fix is to eliminate code execution at the format layer. safetensors stores tensors plus a JSON header and defines no mechanism to invoke callables on load, so a malicious .safetensors file cannot achieve RCE simply by being deserialized — it removes the primitive rather than trying to detect its abuse. Operationally this means an ingest policy that accepts safetensors, converts or rejects pickle-based .pt/.bin/.ckpt, and disables trust_remote_code so the library does not fetch and run arbitrary Python from the repo. The trade-off is that not every artifact ships in safetensors (custom objects, optimizer state, some research checkpoints), so you still need a scan-and-sandbox fallback for the pickle you cannot avoid — safetensors is necessary but not sufficient on its own, because a signed, safe-format file can still contain a behavioral backdoor.

2. Sign and Verify with Provenance (sigstore + transparency + pinning)

Signing answers integrity and identity; provenance answers lineage. The strong pattern is keyless signing via sigstore/cosign or Hugging Face model signing: the producer signs the artifact digest with an OIDC identity, the signature is recorded in a public transparency log (Rekor), and an attestation captures how the artifact was built (SLSA-style: source, builder, inputs). Consumers pin artifacts by content digest — never by mutable tag or latest — and, at load time, verify that the signature is valid, that the signing identity is on an allowlist, that the transparency-log entry exists, and that the provenance attestation matches. This gives non-repudiable, tamper-evident lineage from training pipeline to serving process. Limitations: signing proves who and what bytes, not that the model is safe — a compromised producer key or a legitimately-signed backdoored model still verifies, which is why signing pairs with scanning and evaluation, and why short-lived keyless identities and transparency logs matter for detecting key misuse.

3. Isolated Loading Environments

Because you cannot always avoid pickle and cannot fully prove a model benign, treat every first-load of an untrusted artifact as untrusted code execution and contain the blast radius. Load and convert unverified checkpoints in a sandbox — an ephemeral container or microVM with no production secrets mounted, no outbound network egress, seccomp/AppArmor restrictions, and least-privilege service accounts — then promote only the vetted, re-serialized safetensors artifact into the serving path. This converts a load-time RCE from "attacker reaches credentials and pivots" into "attacker executes in a disposable, network-isolated box." The trade-off is added pipeline latency and infrastructure complexity, but it is the control that makes the residual risk of the other two survivable, and it aligns with SAIF's guidance to isolate risky components and NIST AI RMF's Manage function.

Practice Questions

Tier 1 — Fundamentals

  1. Why is downloading a .pt or .bin model file risky? Those formats are typically Python pickle archives, and pickle/torch.load execute arbitrary code during deserialization via __reduce__. So loading an untrusted checkpoint is equivalent to running an untrusted binary on your host. This is asked to confirm you understand model files are executable, not passive data.
  2. What is safetensors and why is it safer? It is a serialization format that stores only tensor data and a JSON header with no opcodes or callable-invocation mechanism, so a file cannot run code on load. It removes the RCE primitive rather than trying to detect it. This is asked to test whether you know the format-layer fix.
  3. What does OWASP LLM03 cover? LLM03 is Supply Chain risk — compromised or malicious model weights, datasets, adapters, and dependencies pulled from third-party hubs. It is the category that provenance and signing defend. This is asked to check framework literacy.
  4. What is the difference between provenance and integrity? Provenance is verifiable metadata about how and by whom an artifact was produced (lineage); integrity is a cryptographic binding proving the exact bytes were not altered. You need both to trust a downloaded model. This is asked to test precise vocabulary.
  5. Why pin a model by digest instead of by tag? Tags and latest are mutable, so a hub or attacker can swap the underlying bytes without changing the reference. A content digest binds you to exact reviewed bytes. This is asked to test supply-chain hygiene basics.

Tier 2 — System Design and Trade-offs

  1. Design a secure model-ingest pipeline for a company pulling weights from Hugging Face. Fetch only from an allowlist of vetted orgs, pin by digest, verify signature and transparency-log entry, verify a SLSA-style provenance attestation, scan for pickle/malware, convert to safetensors, and do the first load in a network-isolated sandbox before promoting to a private registry that serving reads from. Every step is a CI gate that fails closed. This is asked to see whether you can compose format, signing, and isolation controls into a defense-in-depth pipeline rather than relying on one control.
  2. A signed model still turns out to be backdoored. What failed and what would you add? Signing only proves identity and byte integrity, not behavioral safety — a legitimate producer (or a stolen key) can sign a trojaned model. You add behavioral evaluation, trigger/backdoor red-teaming, output monitoring, and lineage verification of the base model in the fine-tune chain. This is asked to test that you understand the limits of cryptographic controls.
  3. How would you handle a model that only ships as a pickle checkpoint? Treat it as untrusted code: load and re-serialize it inside an ephemeral sandbox with no secrets and no egress, run a pickle-opcode scanner (e.g. detecting GLOBAL/REDUCE to dangerous callables), and only promote the resulting safetensors artifact. Never load it directly in the serving process. This is asked to test pragmatic risk containment when you cannot pick the format.
  4. What is an AI-BOM and why does it matter here? An AI Bill of Materials enumerates every model, adapter, dataset, and dependency with versions and hashes, giving you the ability to detect substitution and to answer "am I affected" when an upstream artifact is found compromised. It is the model-world analogue of an SBOM. This is asked to connect supply-chain response to inventory.
  5. How does sigstore keyless signing improve on long-lived signing keys? It binds signatures to short-lived, OIDC-issued identities and records them in a public transparency log, so there is no long-lived private key to steal and any misuse is publicly auditable. This shrinks the key-compromise window and makes backdating detectable. This is asked to test modern signing-infrastructure knowledge.

Tier 3 — Expert and Deep-Dive

  1. Walk through exactly how pickle achieves code execution on model load. During unpickling, the REDUCE opcode calls a callable that a GLOBAL opcode imported, with attacker-supplied arguments; an object's __reduce__ can return (os.system, ("cmd",)), so the interpreter runs the command as it reconstructs the object — before any tensor is used or inference runs. Defenses like restricted unpicklers help but are hard to make sound, which is why format replacement is preferred. This is asked to separate people who have read the threat from people who have exploited or mitigated it.
  2. How would you verify lineage across a multi-hop fine-tune chain? Require each hop to emit a signed provenance attestation referencing the digest of its input model, so the chain forms a verifiable graph from foundation model to your fine-tune; at ingest you walk the chain, verify each signature and transparency-log entry, and enforce that every ancestor is from an allowlisted identity. Gaps in the chain are treated as unverified and sandboxed or rejected. This is asked to test SLSA-style thinking applied to models rather than software builds.
  3. Provable vs probabilistic controls in this domain — where does each apply? Format safety (safetensors) and signature verification are effectively provable/sound — a safetensors file cannot execute on load and a bad signature is rejected deterministically. Backdoor detection and behavioral evaluation are probabilistic and can be evaded, so you must not rely on them as gates. The design principle is to push as much of the risk as possible onto the sound controls and contain the rest with isolation. This is asked at Staff/Principal level to test architectural judgment about control strength.
  4. A compromised producer key signs malicious weights that pass all your verification. How do you detect and limit this? Transparency logs make the malicious signature publicly visible so it can be discovered and revoked; you add anomaly detection on signing identity and artifact behavior, require multi-party or reproducible-build attestations for high-trust models, and rely on runtime isolation so even a verified-but-malicious model that runs code is contained. Detection shifts from prevention to auditability and blast-radius control. This is asked to test reasoning about trusted-party compromise, the hardest supply-chain case.
  5. Why is trust_remote_code=True dangerous and when is it justified? It lets the loading library fetch and execute arbitrary Python shipped in the model repo, so it reintroduces RCE even for a safe weight format — the repo code, not the weights, becomes the vector. It is only justified for internally-produced or fully code-reviewed repos pulled by digest, ideally in a sandbox. This is asked to test awareness that the artifact format is not the only supply-chain execution path.

Key Takeaways

  • Model files are executable, not passive data: pickle-based .pt/.bin/.ckpt run arbitrary code on load — this is the heart of OWASP LLM03.
  • safetensors removes the RCE primitive at the format layer; prefer it and refuse pickle, and disable trust_remote_code.
  • Signing proves who and what bytes (integrity + identity); provenance proves lineage — you need both, via sigstore/transparency logs and SLSA-style attestations.
  • Signing does not prove the model is safe; a legitimately-signed model can still be backdoored, so pair signing with scanning and behavioral evaluation.
  • Pin by content digest, never by tag or latest, and pull only from an allowlist of vetted sources; maintain an AI-BOM to detect substitution.
  • Treat every unverified first-load as untrusted code execution and contain it in a sandbox with no secrets and no network egress.
  • Production one-liner: verify signature, provenance, and digest, scan and convert to safetensors, and load only in an isolated environment before promoting to serving.
