Poisoned Dependencies and Compromised Fine-Tunes

Domain: AI / ML Supply-Chain Security
Frameworks: OWASP LLM03 (Supply Chain), MITRE ATLAS (ML Supply Chain Compromise, AML.T0010), Google SAIF, NIST AI RMF, SLSA v1.0

Overview

The modern ML stack is assembled from thousands of third-party parts: PyPI packages, model frameworks, pretrained base models, and community fine-tunes or LoRA adapters. Every one of these is untrusted code or untrusted weights running with the privileges of your training and inference pipelines. OWASP LLM03 (Supply Chain) covers the risk that any link — a malicious package, a typosquatted ML library, a compromised training framework, or a backdoored third-party fine-tune — introduces attacker-controlled behavior into your model. Unlike prompt injection, this attack lands before inference: it corrupts the artifacts you build on, so it is persistent, hard to see, and reproducible only if you know exactly what went into the build. It matters because a single poisoned dependency in a training image or a single backdoored adapter can compromise every model and every downstream user that trusts it, and the corruption survives retraining if the tainted component is reused.

How It Works

Core Mechanics

There are two distinct entry points. The first is the software supply chain: the Python and framework code that runs your pipeline. Attackers publish malicious packages (torchtriton-style dependency-confusion, typosquats like pytorch vs torch, or a hijacked maintainer account pushing a bad release) whose install-time or import-time code executes arbitrary commands — exfiltrating cloud credentials, tokens, or training data from the build host. The trust boundary crossed here is "code I did not write now runs with my build identity." Because pip install runs setup.py and many packages execute code on import, compromise happens at install or first import, long before any model is trained.

The second entry point is the model/weights supply chain: pretrained bases, third-party fine-tunes, and LoRA adapters downloaded from hubs. A fine-tune or adapter can carry a backdoor — a hidden input-to-output rule the attacker trained in, where a specific trigger phrase, token, or pattern forces a chosen behavior (emit a target string, approve a transaction, leak a secret, disable a safety refusal) while the model behaves normally on all other inputs. LoRA adapters are especially sharp because they are small, cheap to publish, easy to merge, and their low-rank weight deltas concealing a trigger are effectively unreviewable by eye. The trust boundary is "weights I did not train now decide outputs," and standard accuracy evaluation will not surface the backdoor because clean-input performance is intact by design. Serialization format compounds this: a pickle-based checkpoint (torch.load on a .bin/.pt) can execute arbitrary code on load, so even "just downloading the weights" is remote code execution unless you use a safe format such as safetensors.

The attacker's capability ranges from anonymous package publishing (low bar) to registry account compromise (higher) to training and publishing a convincingly benign backdoored model (moderate skill, low infrastructure). The objective is persistence and stealth: land once in a widely reused artifact and inherit the blast radius of everyone who trusts it.

Top 3 Critical Mitigations

1. Pin Everything and Verify Provenance (Lockfiles + SLSA)

The single highest-leverage control is removing ambiguity about what exactly went into a build. Pin every direct and transitive dependency to an exact version and a cryptographic hash (pip install --require-hashes, poetry.lock, uv.lock), so a later malicious re-release cannot slip in and every build is reproducible and auditable. Extend the same discipline to weights: pin model and adapter revisions by commit hash, not a floating main tag. Layer provenance on top — SLSA attestations and signing (sigstore/cosign, model signing) let you verify who built an artifact and from what source before you trust it, and an SBOM gives you a queryable inventory so that when a package is later found malicious you can answer "am I affected?" in minutes. The trade-off is operational friction: pinned dependencies require deliberate, reviewed upgrades and provenance verification adds CI steps and key management, but this is the control that turns supply-chain incidents from silent and unbounded into detectable and scoped.

2. Isolated, Hermetic Build and Training Environments

Because pip install and torch.load can execute arbitrary code, treat the build/train host as an untrusted-code execution environment and contain the blast radius. Run installs and training in ephemeral, network-restricted, least-privilege sandboxes: no long-lived cloud credentials on the box, egress allowlisted to known registries, and secrets injected narrowly rather than ambient in the environment. Hermetic builds (fetch all inputs up front from a vetted internal mirror, then build offline) both improve reproducibility and cut off the install-time exfiltration path that made incidents like the torchtriton dependency-confusion attack damaging. The limitation is that isolation contains but does not detect — a backdoored weight passes through a sandbox untouched — so this control pairs with, and does not replace, behavioral testing. The trade-off is engineering cost in pipeline plumbing, which is modest relative to a leaked signing key or cloud root credential.

3. Behavioral / Backdoor Testing Before Trusting a Fine-Tune

Accuracy metrics cannot find a backdoor because the model is deliberately correct on clean inputs; you must test for the hidden behavior. Before promoting any third-party fine-tune or adapter, run it through an adversarial evaluation battery: safety and refusal regression tests (did this fine-tune quietly strip guardrails?), trigger and anomaly scanning (fuzz rare tokens and out-of-distribution prompts, watch for outputs that jump sharply on specific patterns), and differential testing against a trusted base model to flag divergent behavior. For weights themselves, require a safe serialization format (safetensors, never arbitrary pickle) so loading is not RCE, and prefer models with signed provenance and a verifiable lineage over anonymous community uploads. The hard limitation is that backdoor detection is not sound — the trigger space is effectively infinite and a well-hidden trigger can evade scanning — so this is a probabilistic risk-reduction gate, best combined with least-privilege on what the model can do at inference (so even a triggered model cannot reach dangerous tools or data) and provenance so you prefer artifacts you did not have to fully trust blindly.

Practice Questions

Tier 1 — Fundamentals

  1. What is OWASP LLM03 and what does it cover? It is the LLM supply-chain risk: vulnerabilities introduced through third-party components — packages, base models, datasets, fine-tunes, and adapters — rather than through the running prompt. This is asked to confirm you place supply chain before inference in the ML lifecycle.
  2. Why is installing a Python package a security event? Package install and import can run arbitrary code (setup.py, import-time execution), so a malicious package achieves code execution on the build host with your identity. This is asked to test that you see pip install as running untrusted code.
  3. What is typosquatting in an ML context? Publishing a package with a name close to a popular library (torchivision, misspelled numpy) so a fat-fingered install pulls attacker code. This is asked to check basic registry-attack literacy.
  4. Why is loading a pickle-based checkpoint dangerous? Pickle deserialization can execute arbitrary code, so torch.load on an untrusted .bin/.pt is remote code execution; safetensors avoids this. This is asked to test knowledge of unsafe serialization.
  5. What is a model backdoor? A hidden trigger-to-output rule trained into weights that fires on a specific input pattern while behaving normally otherwise. This is asked to confirm you understand why accuracy testing misses it.

Tier 2 — System Design and Trade-offs

  1. Design a vetting gate for third-party fine-tunes before production use. Require signed provenance and a verifiable lineage, pin the artifact by commit hash, enforce a safe serialization format, and run the model through safety-regression, trigger-scanning, and differential tests against a trusted base before promoting it into an internal trusted registry. Quarantine anything that diverges and require human review. This is asked to see whether you build a defense-in-depth pipeline instead of trusting a hub download.
  2. How do lockfiles and hashes reduce supply-chain risk? They pin exact versions and cryptographic hashes so builds are reproducible and a later malicious re-release under the same version cannot be silently installed; combined with an SBOM you can also answer "am I affected" fast when a package is disclosed as malicious. The trade-off is that upgrades become deliberate, reviewed events. This is asked to test whether you can operationalize dependency integrity.
  3. A LoRA adapter and a full fine-tune both claim to improve your model — which is riskier and why? Both are risky, but adapters are cheaper to publish, tiny, and their low-rank deltas are effectively unreviewable by inspection, so a hidden trigger merges in with almost no signal. Full fine-tunes are larger but similarly opaque; the defense is the same behavioral battery for both, not visual review. This is asked to test nuance about where stealth actually lives.
  4. How would you contain the blast radius of a compromised training dependency? Run installs and training in ephemeral, network-restricted, least-privilege sandboxes with no ambient long-lived credentials, egress allowlisted to a vetted mirror, and hermetic offline builds so install-time code cannot exfiltrate secrets. This is asked to test isolation thinking and credential hygiene.
  5. Why does model accuracy evaluation fail to catch a backdoored model? The backdoor is designed to leave clean-input performance intact, so standard metrics look perfect; you must specifically test the hidden behavior via trigger fuzzing, safety regressions, and differential comparison. This is asked to check that you distinguish quality evaluation from security evaluation.

Tier 3 — Expert and Deep-Dive

  1. How does dependency confusion work and how do you prevent it? If an internal package name also exists on a public index, some resolvers prefer the higher public version, so an attacker publishes a malicious public package with your internal name and a huge version number to get it pulled instead. Prevent it by pinning to exact hashes, configuring index priority and namespace reservation, and mirroring internal packages so public names cannot shadow them. This is asked because it is a subtle resolver-behavior attack that has hit real companies.
  2. Backdoor detection is not sound — so how do you defend a model you cannot fully trust? Accept that trigger space is unbounded and scanning is probabilistic, then compensate with least privilege at inference: constrain what tools, data, and actions the model can reach so a triggered output cannot cause real damage, add output guardrails and human-in-the-loop for high-impact actions, and prefer signed-provenance artifacts so you minimize how many components you must trust blindly. This is asked to see if you reason about provable-versus-probabilistic controls and defense in depth rather than promising perfect detection.
  3. How would provenance and signing (SLSA, sigstore, model signing) have changed a real incident? Signed provenance ties an artifact to a verifiable builder and source, so a hijacked-maintainer release or a swapped weight file fails verification at the ingestion gate instead of being trusted by name alone; SLSA levels raise assurance that the build was tamper-evident and reproducible. This is asked to test whether you can connect standards to concrete attack prevention rather than citing them as buzzwords.
  4. A strong base model is fine-tuned by a third party and the fine-tune quietly removes safety alignment — how do you catch it? Run refusal and safety-regression suites that compare the fine-tune's behavior on a curated harmful-prompt set against the aligned base; a sharp drop in refusal rate flags alignment stripping even when task accuracy improves. Pair this with differential testing and treat any safety regression as a promotion blocker. This is asked because it separates people who evaluate only capability from those who evaluate safety as a first-class metric.
  5. Where would you invest first if you inherited an ML platform with no supply-chain controls? Start with visibility and integrity — SBOM plus pinned, hash-locked dependencies and a private mirror — because you cannot defend what you cannot inventory, then add hermetic isolated build/train environments to cap credential blast radius, then a behavioral vetting gate for external weights. Provenance/signing follows to reduce blind trust. This is asked to test prioritization and whether you sequence controls by leverage rather than boiling the ocean.

Key Takeaways

  • OWASP LLM03 is the ML supply chain: packages, frameworks, datasets, base models, fine-tunes, and adapters are all untrusted code or untrusted weights.
  • pip install runs arbitrary code and torch.load on a pickle is RCE — treat build and load paths as untrusted-code execution.
  • Backdoors survive accuracy testing by design; only behavioral, trigger, and safety-regression testing surfaces them, and detection is probabilistic, not sound.
  • LoRA adapters are the sharpest stealth vector: small, cheap, mergeable, and effectively unreviewable by eye.
  • Pin exact versions and hashes with lockfiles, carry an SBOM, and verify SLSA provenance and signatures before trust.
  • Isolate build and training in ephemeral, least-privilege, network-restricted sandboxes so install-time code cannot steal credentials.
  • In production I would defend this with pinned+signed dependencies, hermetic isolated builds, safetensors-only loading, a behavioral vetting gate for external weights, and least privilege at inference so a triggered model cannot cause real harm.
