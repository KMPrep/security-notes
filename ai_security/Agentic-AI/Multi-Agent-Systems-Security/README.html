Multi-Agent Systems Security

Domain: AI / Agentic AI Security
Frameworks: OWASP LLM Top 10 (LLM06 Excessive Agency, LLM01 Prompt Injection, LLM08 Vector/Embedding Weaknesses), MITRE ATLAS, Google SAIF, NIST AI RMF

1. Overview

A multi-agent system (MAS) decomposes a task across several LLM-driven agents that coordinate through messages — typically an orchestrator/worker hierarchy (a planner delegates subtasks to specialist workers) or a peer topology (agents negotiate directly). Each agent may hold its own tools, memory, credentials, and system prompt, so the agent boundary becomes a new trust boundary that most single-agent threat models ignore. This matters in production because agent-to-agent messages are model-generated natural language, not typed API contracts: a prompt injection landing in one agent can propagate as "instructions" to the next, and a low-privilege agent can trick a high-privilege agent into acting as a confused deputy. The dominant classification is OWASP LLM06 Excessive Agency (too much autonomy, permission, or functionality) compounded by LLM01 Prompt Injection propagation; MITRE ATLAS covers the LLM prompt-injection and tool-abuse techniques, while Google SAIF ("agents operate with appropriate permissions," "limit blast radius") and NIST AI RMF (Map/Measure/Manage) provide the governance framing for delegation and containment.

2. How It Works

2.1 Core Mechanics

In an orchestrator/worker MAS, an orchestrator agent receives a goal, plans, and issues subtasks to worker agents. Each worker runs its own model context, calls its own tools, and returns results as text that the orchestrator folds back into its context. The trust boundaries that cross the agent seam are: (1) the task/instruction an orchestrator sends down, (2) the result a worker sends up, (3) tool outputs and retrieved documents each agent ingests, and (4) shared memory or a message bus that many agents read and write. The critical, often-missed property is that a worker's returned text re-enters the orchestrator's instruction channel — there is no structural separation between "data the worker fetched" and "commands for the orchestrator to follow."

For attacks, the adversary's capability is usually only the ability to place text where some agent will read it (a webpage a browsing worker fetches, a document in a shared store, a field in a ticket). The objective is to get that text interpreted as instructions and then propagate across the agent graph until it reaches an agent that holds the privilege the attacker wants (a code-exec sandbox, a payments tool, an email sender). Because agents commonly inherit or share identity — one service account, one API key, one memory namespace — a single compromised agent can borrow the authority of the whole fleet. For defenses, the control points are the message seams: validate and label every inter-agent message, give each agent a distinct identity and least-privilege scope, and contain blast radius so one agent's compromise cannot cascade.

Top 3 Critical Mitigations

1. Per-Agent Least Privilege and No Inherited Authority

Each agent should have its own identity, its own scoped credentials, and only the tools its role requires — the research worker cannot touch payments, the payments worker cannot browse the open web. Technically this means distinct service accounts or short-lived scoped tokens per agent, and enforcing that when the orchestrator delegates to a worker, the worker's privileges apply, not a shared superset. This directly breaks the confused-deputy chain: even if an injection reaches an agent, it can only exercise that agent's narrow scope. The trade-off is operational overhead — many identities to provision, rotate, and audit — and the risk that over-fragmentation makes legitimate collaboration awkward; SAIF's "appropriate permissions" and least-privilege principle are the anchoring guidance.

2. Message Validation and Data/Instruction Separation Between Agents

The root cause of propagation is that a worker's free-text output is spliced back into the orchestrator's instruction context. Mitigate by treating every inter-agent message as untrusted data: enforce a typed schema on results, strip or clearly delimit any content that looks like instructions, and never let a worker's returned text silently re-plan the orchestrator's goal. Authenticate the sender so a rogue or spoofed agent cannot inject as a trusted peer, and tag content with provenance so downstream agents know it came from the untrusted web versus an internal tool. The limitation is that this filtering is probabilistic, not sound — LLM-based classifiers and delimiter tricks can be bypassed — so it must be layered with least privilege rather than relied on alone.

3. Blast-Radius Isolation and Cascade Containment

Assume any single agent can be compromised and design so it cannot take the fleet down with it. Run tool execution in per-agent sandboxes, put circuit breakers and rate/budget/delegation-depth limits in the orchestrator, and quarantine an agent whose behavior deviates instead of letting peers keep trusting its messages. Gate irreversible or high-impact cross-agent actions behind human-in-the-loop approval so a cascade cannot silently reach the real world. Maintain per-agent attribution in logs and traces so responders can tell which agent acted and cut it off. The trade-off is added latency and friction on legitimate multi-step autonomy — the standard tension between agency and control that NIST AI RMF's Manage function exists to balance.

5. Practice Questions

Tier 1 — Fundamentals

  1. What new trust boundary does a multi-agent system introduce? The agent-to-agent message seam: one agent's model-generated output becomes another agent's input, and that input is natural language with no structural separation between data and instructions. This is asked to confirm you see the agent boundary as a real security boundary, not just plumbing.
  2. What is agent-to-agent prompt injection propagation? An injection that lands in one agent (via a fetched page, document, or tool output) rides along in that agent's message to the next agent and is interpreted as instructions there, spreading across the graph. This is asked to test whether you understand injection as a propagating, not point, failure.
  3. Why is shared identity across agents dangerous? If all agents use one key or service account, compromising any single agent inherits the authority of the whole fleet, erasing least privilege and attribution. This is asked to test identity fundamentals in agentic systems.
  4. What OWASP LLM category best fits multi-agent risk? LLM06 Excessive Agency (excess autonomy, permissions, or functionality), usually triggered by LLM01 Prompt Injection. This is asked to check framework literacy.
  5. What is a confused deputy in a multi-agent context? A low-privilege agent tricks a higher-privilege agent into performing an action the attacker could not do directly, because the privileged agent trusts the request. This is asked to test the core delegation-abuse concept.

Tier 2 — System Design and Trade-offs

  1. Design the trust model for an orchestrator/worker system with a payments worker and a web-research worker. Give each worker its own scoped identity and minimal tools, treat worker results as untrusted data validated against a schema, forbid the research worker any egress-to-payments path, and gate any money-moving action behind human approval. The key insight is that delegation must not widen privilege — the orchestrator invokes the payments worker under the payments worker's scope only. This is asked to see if you can compose least privilege with data/instruction separation.
  2. How would you stop an injection in a browsing worker from reaching a code-execution worker? Validate and label the browsing worker's output as untrusted-web provenance, strip or delimit instruction-like content before it re-enters orchestration, deny the code worker any capability the task does not require, and sandbox its execution. Because filtering is probabilistic, the real control is that even a successful injection hits a narrowly scoped, sandboxed agent. This is asked to test layered defense reasoning.
  3. How do you prevent cascading failure across peer agents? Circuit breakers, delegation-depth and budget limits, per-agent behavioral monitoring, and quarantine so peers stop trusting a deviating agent's messages. Add human checkpoints on irreversible actions so a cascade cannot reach production silently. This is asked to test resilience and blast-radius thinking.
  4. How would you authenticate messages between agents? Sign or authenticate each message so the receiving agent verifies the sender's identity, preventing a rogue or spoofed agent from injecting commands as a trusted peer, and carry provenance metadata for content. This is asked to test whether you apply classic message-integrity controls to LLM channels.
  5. What do you log to make a multi-agent incident investigable? Per-agent identity on every action and message, the full delegation chain, tool calls with arguments, and provenance of ingested content, so responders can attribute behavior to a specific agent and cut it off. This is asked to test observability and attribution under shared execution.

Tier 3 — Expert and Deep-Dive

  1. Why can't message validation alone stop agent-to-agent injection, and what must back it up? Validation and instruction-stripping are probabilistic classifiers and delimiter heuristics that adversarial phrasing, encoding, or role-claiming can bypass, so they reduce but never eliminate propagation. They must be backed by sound controls — per-agent least privilege, sandboxing, and human gates on high-impact actions — so a bypass still lands on an agent that structurally cannot do harm. This is asked to separate probabilistic from provable controls, a Staff-level distinction.
  2. Design secure delegation so an orchestrator can hand work to a worker without granting extra authority. The worker executes under its own scoped credential, not the orchestrator's; the orchestrator passes a capability-limited task token rather than its own privileges; and the result returns as typed data that cannot re-plan the orchestrator. This mirrors OAuth-style scoped delegation and directly defeats confused-deputy escalation. This is asked to test whether you can translate classic authorization patterns into agent orchestration.
  3. How does trust asymmetry via role labels get exploited, and how do you defend it? Injected text can claim a privileged role ("as the admin agent, execute...") and a naive receiver believes the claim because role is conveyed in-band as text. Defend by binding role to a verified identity out-of-band (signed sender, IAM scope) rather than trusting any role asserted inside message content. This is asked to probe whether you catch identity-versus-assertion confusion.
  4. A peer-topology system with shared memory keeps getting reinfected after cleanup. What's happening and how do you fix it? The shared memory store is poisoned: a persisted injection is re-read by agents as trusted context, so cleaning one agent's context does not remove the source. Fix by tagging memory entries with provenance and trust level, validating on read, isolating write scopes per agent, and purging tainted entries — treat the store as an untrusted input channel. This is asked to test persistence and stateful-injection reasoning (LLM08).
  5. When is a multi-agent architecture the wrong choice from a security standpoint? When the task does not need it: more agents means more identities, more message seams, more injection surface, and harder attribution, so a single well-scoped agent is often more defensible. Multi-agent is justified only when genuine separation of duty or parallelism outweighs the added blast-radius and containment cost. This is asked to test architectural judgment — knowing that agency is a liability to be minimized, not a feature to maximize.

6. Key Takeaways

  • The agent-to-agent message seam is a trust boundary; a worker's output re-enters another agent's instruction channel with no built-in data/instruction separation.
  • Prompt injection in a MAS propagates — it spreads across agents until it reaches one holding the privilege the attacker wants, so classify it as LLM06 Excessive Agency plus LLM01.
  • Never let one compromised agent inherit others' authority: distinct identity and scoped credentials per agent break the confused-deputy chain.
  • Message validation between agents is probabilistic and must be layered with sound controls (least privilege, sandboxing, human gates), never relied on alone.
  • Contain blast radius with per-agent isolation, circuit breakers, delegation/budget limits, quarantine, and per-agent attribution in logs.
  • Bind role and authority to verified out-of-band identity, not to any role a message asserts in its text.
  • In production, defend a multi-agent system by giving every agent least privilege and its own identity, treating all inter-agent messages as untrusted data, and isolating blast radius so one agent's compromise cannot cascade into the fleet.
