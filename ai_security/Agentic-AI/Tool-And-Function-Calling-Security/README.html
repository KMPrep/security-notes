Tool and Function Calling Security

Domain: Agentic AI Security — Tool / Function Calling
Frameworks: OWASP LLM06 (Excessive Agency), OWASP LLM05 (Improper Output Handling), Google SAIF, NIST AI RMF, MITRE ATLAS

Overview

Tool (or function) calling is the mechanism where an LLM emits a structured request — typically a JSON object naming a tool and its arguments — that a separate executor parses and runs against a real API, database, shell, or service. It sits at the agentic/tool layer of the system, the exact point where model-generated text stops being inert and starts causing side effects in the world. This is where the highest-severity agentic risks concentrate: the model becomes a confused deputy that holds real privilege but takes instructions from untrusted input, so a successful prompt injection upstream turns into SSRF, command injection, data exfiltration, or unauthorized state change downstream. OWASP classifies the core failure as LLM06 Excessive Agency (too much functionality, permission, or autonomy), with LLM05 Improper Output Handling covering the trust placed in tool results, and Google SAIF and NIST AI RMF frame the governing controls (least privilege, human oversight, deny-by-default). The single most important mental model: the model's tool call is untrusted input to your executor, and the tool's output is untrusted input to your model.

How It Works

Core Mechanics

A tool-calling loop has three actors and two trust boundaries. The model proposes a call; the executor (agent runtime / orchestration layer) validates and dispatches it; the tool/API performs the effect and returns data that re-enters the model's context. The model never touches the API directly — it only emits a name and arguments — so all real authority lives in the executor and the credentials it holds.

The two boundaries that matter:

  • Model → Executor: the arguments the model produced are attacker-influenceable. Anything that shaped the prompt (user text, a retrieved document, a previous tool result, memory) can steer these arguments. Treat them like form input from the public internet: validate types, ranges, allowlists, and authorization before dispatch.
  • Tool → Model: the tool's return value is fed back into the context window as if it were trustworthy. A tool that fetches web pages, reads tickets, or queries a shared table can return attacker-controlled text that carries new instructions (indirect prompt injection), poisoning the next model turn.

The dangerous property is standing privilege. The executor typically authenticates to backends with a broad service credential rather than the end user's scoped identity. So a call the model was manipulated into making runs with the agent's full rights, not the requesting user's — the classic confused-deputy setup. Argument injection (SSRF via a url param, command injection via a path or cmd param, SQL injection via a filter param), over-broad tool scopes (one execute_sql tool instead of narrow read-only endpoints), and blind trust of tool output are the recurring root causes.

Top 3 Critical Mitigations

1. Least-Privilege Scopes with Per-Tool, Per-User Authorization

The root defense against the confused-deputy problem is to stop the executor from acting with standing broad privilege. Each tool should be a narrow, single-purpose capability holding the minimum credential it needs, and every call must be authorized against the end user's identity, not just the agent's service account — so a manipulated call can never exceed what that user could do directly. Technically this means propagating the user's identity (token exchange, on-behalf-of flows) to the authorization decision, mapping each tool to explicit required scopes, and denying by default any tool or scope not explicitly granted. The trade-off is engineering cost: you must design many small tools instead of one flexible execute tool, maintain a per-tool permission matrix, and thread identity through the whole call chain. But this is the control that bounds blast radius regardless of how good the injection is, which is why SAIF and NIST AI RMF (Govern/Manage) both center least privilege and access enforcement.

2. Argument Validation, Allowlists, and Injection-Safe Tool Implementations

Because the model's arguments are attacker-influenceable, the executor must treat them exactly like untrusted web input. Enforce a strict typed schema (JSON Schema with types, enums, ranges, regex), reject anything that does not conform, and apply positive allowlists for sensitive parameters — permitted hosts for a url arg (with internal/link-local ranges blocked to stop SSRF), permitted tables for a query tool, permitted file roots for a path arg. Inside the tool, never interpolate arguments into a shell string or SQL string: use argument-array exec and parameterized queries so the value can never change the command's structure. The trade-off is that allowlists are brittle and add maintenance, and overly strict validation can reject legitimate creative calls, so you pair validation with good error messages back to the model and monitoring of rejection rates. This directly closes OWASP LLM05, the highest-likelihood/highest-severity class here (SSRF, RCE, SQLi via tool params).

3. Human-in-the-Loop Gates, Sandboxing, and Deny-by-Default Autonomy

Some actions are irreversible or high-impact (sending money, deleting data, emailing externally, running code), and no probabilistic guardrail is sound enough to fully trust the model on them. For these, insert a human-in-the-loop approval gate that shows the concrete resolved action and requires explicit confirmation, and run any code-executing or file-touching tool in a sandbox (isolated container, no ambient network, ephemeral filesystem, egress allowlist) so a bad call is contained. The whole tool catalog should be deny-by-default: a tool is unavailable unless explicitly enabled for that agent, session, and user, and the agent loop enforces step and cost budgets so it cannot chain unbounded actions. The trade-off is latency and user friction — HITL only scales if you gate the genuinely high-impact subset and let low-risk read actions flow — but this is what SAIF calls human oversight and what turns a total compromise into a contained, reviewable event.

Practice Questions

Tier 1 — Fundamentals

  1. What actually happens during an LLM tool call? The model emits a structured request naming a tool and arguments; a separate executor validates and runs it against a real API, then feeds the result back into context. The model never calls the API directly — all privilege lives in the executor. This is asked to confirm you understand where the trust boundary and the real authority sit.
  2. Why is a tool call a security-relevant boundary? It is the point where model-generated text becomes real side effects, and those arguments are attacker-influenceable through the prompt. So the model's output must be treated as untrusted input to the executor. This is asked to see if you grasp the core threat model.
  3. What is the confused-deputy problem in agentic AI? The agent holds real standing privilege but takes instructions from untrusted input, so it can be tricked into using its authority on an attacker's behalf. The fix is authorizing on the end-user's identity, not the agent's. This is asked to test the central agentic risk concept.
  4. Which OWASP LLM risks map to tool calling? Primarily LLM06 Excessive Agency (too much functionality, permission, or autonomy) and LLM05 Improper Output Handling (trusting tool output). This is asked to check framework fluency.
  5. Why should tool output be treated as untrusted? A tool that fetches web pages or reads shared data can return attacker-controlled text carrying new instructions, which is indirect prompt injection into the next model turn. This is asked to test the second trust boundary — tool to model.

Tier 2 — System Design and Trade-offs

  1. Design safe tool schemas for a customer-facing support agent. Favor many narrow single-purpose tools over one generic execute tool, define strict JSON Schema with typed args, enums, and ranges, and add positive allowlists on sensitive params. Each tool holds a least-privilege credential and is authorized against the user's identity; high-impact actions require a human gate; the catalog is deny-by-default. This is asked to test whether you can turn principles into a concrete, minimal-blast-radius design.
  2. A tool takes a url argument to fetch a page. How do you prevent SSRF? Enforce an allowlist of permitted hosts, resolve and block internal, loopback, and link-local ranges (including cloud metadata endpoints), disallow redirects to non-allowlisted hosts, and route egress through a controlled proxy. Never let the model's raw URL reach an unrestricted fetch. This is asked to test practical injection defense on a common tool shape.
  3. How do you stop a manipulated agent from exceeding the user's own permissions? Propagate the end-user identity to the authorization decision (token exchange / on-behalf-of), enforce per-tool scopes against that identity, and deny by default. The agent's service credential must never grant more than the user could do directly. This is asked to test the least-privilege answer to confused deputy.
  4. Where do you put human-in-the-loop gates without killing UX? Gate only the irreversible or high-impact subset — payments, deletes, external sends, code execution — and show the resolved concrete action for approval, while letting low-risk read actions flow automatically. This is asked to test judgment about balancing safety and usability.
  5. Threat-model an agent whose tools read from a shared ticketing system. Untrusted ticket text is an indirect-injection vector, so tool output must be labeled untrusted and never auto-followed as instructions; every downstream tool call needs authz on the requesting user, argument validation, and egress control so injected instructions cannot exfiltrate or act. This is asked to test end-to-end reasoning across both trust boundaries.

Tier 3 — Expert and Deep-Dive

  1. Why are input/output guardrails insufficient as the only defense for tool calling? Guardrails are probabilistic classifiers, not sound proofs, so a determined injection will eventually bypass them. Durable defense comes from architectural controls that bound blast radius regardless of model behavior — least-privilege scopes, per-user authz, sandboxing, and deny-by-default — so even a fully manipulated model cannot exceed granted authority. This is asked to test the probabilistic-versus-provable distinction expected at staff level.
  2. How would you design secure delegation in a multi-agent system with tools? Each sub-agent gets its own least-privilege tool set and a scoped, short-lived, audience-restricted credential rather than sharing a broad token, and identity plus authorization context is propagated on every hop so a sub-agent cannot act beyond what was delegated. Inter-agent messages are treated as untrusted input, and high-impact actions still hit a central policy gate. This is asked to test whether you can prevent privilege amplification across delegation chains.
  3. An attacker plants hidden instructions in a document the agent retrieves. Trace the exploit and the controls that stop it. The retriever pulls the document, the model reads the hidden instruction and emits an exfiltration tool call; if the executor uses standing privilege with no per-user authz or egress allowlist, data leaves the boundary. Controls that break the chain: labeling retrieved content as untrusted, denying instructions from tool output, per-user authorization on the egress tool, an egress allowlist plus DLP, and a human gate on any external send. This is asked to test attack-chain reasoning and layered defense.
  4. How do you make a code-execution tool safe enough to expose to an agent? Run it in a hardened sandbox — isolated ephemeral container, no ambient credentials, no default network, egress allowlist, resource and time limits — pass arguments as arrays never shell strings, and require a human gate or narrow allowlist for anything with side effects. Assume the executed code is attacker-controlled and design containment accordingly. This is asked to test whether you can safely ship the most dangerous tool class.
  5. What are the hardest operational problems running agents with tools at scale? Threading end-user identity and least-privilege scopes through every tool and delegation hop, keeping allowlists current without over-blocking, detecting indirect injection in tool outputs, enforcing cost and action budgets to prevent runaway loops, and auditing every resolved tool call for incident response. This is asked to test production maturity beyond the happy path.

Key Takeaways

  • A tool call is the boundary where model text becomes real side effects — treat the model's arguments as untrusted input to the executor.
  • Tool output is also untrusted input back to the model, so it can carry indirect prompt injection into the next turn.
  • The central risk is the confused deputy: the agent holds standing privilege but obeys untrusted input; the answer is authorizing on the end-user's identity.
  • Argument injection (SSRF, command injection, SQLi) is closed by strict typed schemas, positive allowlists, and injection-safe tool implementations — never interpolate args into shell or SQL.
  • Least-privilege, single-purpose tools with a deny-by-default catalog bound the blast radius no matter how good the injection is.
  • Guardrails are probabilistic; architectural controls (least privilege, per-user authz, sandboxing, HITL gates) are what actually contain a full compromise.
  • Production defense in one line: give each tool the narrowest scope and the user's own identity, validate and allowlist every argument, sandbox side effects, and put a human gate on anything irreversible — deny by default.
