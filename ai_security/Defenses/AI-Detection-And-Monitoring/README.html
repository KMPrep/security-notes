AI Detection and Monitoring

Domain: Detective and operational controls for LLM/agent systems in production
Anchor frameworks: OWASP LLM10 Unbounded Consumption (also LLM01/LLM06/LLM02) · MITRE ATLAS (Discovery, Exfiltration, Impact, ML Model Access) · NIST AI RMF (Measure, Manage) · Google SAIF (Detect, Respond)

1. Overview

Detection and monitoring is the detective and responsive half of AI defense: assuming prompt-injection and abuse controls are probabilistic and will sometimes fail, you need telemetry that shows what the model saw, what it decided, and what it did so you can catch attacks in flight, rate-limit abuse, and reconstruct an incident afterward. It sits at the inference and agent/tool layer, wrapping the model with logging, real-time classifiers, anomaly detection on agent behavior, and consumption controls. It matters because an LLM turns text into actions and spend: an undetected injection can exfiltrate data or run up unbounded cost, and without an audit trail you cannot even prove what happened. The frameworks classify this squarely under OWASP LLM10 Unbounded Consumption (DoS/cost abuse and model theft via query volume), with cross-cutting relevance to LLM01 Prompt Injection detection and LLM02 Sensitive Information Disclosure; it maps to NIST AI RMF Measure (instrument and quantify risk) and Manage (respond to incidents), SAIF's Detect/Respond controls, and MITRE ATLAS techniques for Discovery, Model Extraction, and Exfiltration. The central design tension is that the very telemetry that gives you detection — full prompt and response capture — is also the largest new PII/secret sink you will ever create, so privacy-aware logging is a first-class requirement, not an afterthought.

2. How It Works

2.1 Core Mechanics

An AI monitoring system instruments three trust boundaries around the model. Inbound, it captures the user prompt, the system prompt, and any retrieved/tool-supplied context — because in an LLM, data is control, every one of these is potential attack input. Inbound-decision, it captures the model's chosen tool calls and arguments before they execute, which is the highest-value signal for an agent: a tool call is where a content bug becomes an access-control event. Outbound, it captures the model's response and any data leaving via tools or downstream systems.

Each captured event is enriched into a structured trace: a stable request_id/session_id/agent_run_id, tenant and principal identity, model and prompt version, token counts (prompt/completion), latency, tool name and scopes, and the verdicts of any guardrail classifiers. Those traces feed two consumers running at different speeds. A real-time (inline) path runs cheap detectors on the request path — injection/jailbreak classifiers, secret and PII scanners, output validators, and rate/quota checks — and can block, redact, or step-up to human-in-the-loop before an action fires. An asynchronous (offline) path streams the same traces to a SIEM/data lake where heavier analytics run: per-principal behavioral baselines, sequence anomaly detection on agent tool chains, cost/consumption aggregation, and correlation across sessions to spot slow model-extraction or distributed abuse.

The control-point discipline mirrors classic security telemetry: detect (classifiers + anomaly models), decide (allow/redact/block/escalate), record (immutable, privacy-scrubbed audit log), respond (rate-limit, kill switch, revoke tool scope, alert). The hard part unique to AI is that the richest evidence — the raw prompt and response — is exactly the data you most need to minimize, so scrubbing and access control on the log store are part of the mechanism, not a compliance bolt-on.

Top 3 Critical Mitigations

1. Consumption controls: per-principal budgets, caps, and a kill switch (LLM10)

The single highest-leverage detective+preventive control is budgeting every axis of consumption per identity: requests/minute, tokens/hour, concurrent runs, max output tokens, wall-clock timeout, and — for agents — max reasoning steps per run. These are enforced at the inference gateway with sliding-window counters (token buckets) keyed on API key / tenant / user, backed by an aggregation layer that also computes cross-session totals to catch distributed abuse and model-extraction farming a single logical target across many keys. The pattern culminates in a kill switch: a per-tenant and global circuit breaker that stops spend when an anomaly is confirmed. The trade-off is that tight limits throttle legitimate power users and batch workloads, so budgets need tiering and burst allowances; and per-request caps alone miss slow abuse, which is why the offline aggregation path is mandatory. This directly answers OWASP LLM10 and NIST AI RMF Manage.

2. Privacy-aware, structured audit logging

Detection and incident response both depend on a structured, immutable trace — stable request_id/session_id/agent_run_id, principal, model/prompt version, token counts, tool name/args/scopes, and classifier verdicts — but the raw prompt and response are the crown-jewel privacy risk. The control is to scrub before you persist: run PII/secret detection (regex + NER + entropy for keys) and redact, tokenize, or hash sensitive fields, keep raw content only in a short-TTL, access-controlled, field-encrypted store (or not at all — log hashes and metadata for correlation instead), and enforce retention TTLs and purpose binding. Store logs append-only (WORM) in a separate security account so an attacker who compromises the app cannot erase evidence. The trade-off is fidelity vs. privacy: over-scrubbing can destroy the exact injection payload you needed to investigate, so a common design keeps metadata + verdicts always, and gates raw-content access behind break-glass approval. This is the backbone of an AI incident audit trail and maps to NIST AI RMF Manage and Measure.

3. Layered inline detectors plus behavioral anomaly detection

No single classifier is sound, so detection is defense-in-depth across two speeds. Inline (blocking) detectors screen input for injection/jailbreak (Llama Guard, Rebuff, NeMo Guardrails, prompt-injection classifiers), scan output for PII leakage and system-prompt disclosure, and validate tool-call arguments against expected schemas/scopes before execution. Offline, you build behavioral baselines: normal tool-call sequences, typical retrieval sources, response-length and refusal-rate distributions per use case, and flag deviations (an agent that suddenly calls a data-export tool it never used, or a session whose tool chain looks nothing like the population). The trade-off is the false-positive/false-negative curve: aggressive inline blocking creates alert fatigue and over-blocks legitimate traffic, while loose thresholds miss novel attacks — so verdicts must be severity-tiered and risk-scored, feeding a SIEM with dedup and correlation rather than paging on every hit. This operationalizes NIST AI RMF Measure and SAIF Detect, and pairs with human-in-the-loop gates on irreversible actions.

5. Practice Questions

Tier 1 — Fundamentals (screens out ~60%)

  1. What does OWASP LLM10 Unbounded Consumption cover, and why is it a security issue, not just a billing one? It covers uncontrolled resource use — token/cost blowup ("denial of wallet"), request-flood DoS, and high-volume querying used to extract or clone a model. It is a security issue because unbounded spend is an availability attack and model extraction is IP theft, so it demands rate/token budgets and volume baselines, not just a billing alert. Why asked: confirms the candidate knows LLM10 is about availability and IP, not accounting.

  2. What are the minimum fields you would log for every LLM request to support incident response? A stable request/session/agent-run id, principal and tenant, model and prompt version, prompt and completion token counts, tool name/args/scopes, latency, and every guardrail verdict — plus a privacy-scrubbed reference to the prompt/response rather than raw text by default. Why asked: tests whether they can build an audit trail that actually reconstructs an incident.

  3. Why is logging full prompts and responses risky, and what do you do about it? Prompts and responses routinely contain PII and secrets, so full-fidelity logs become the largest new breach surface in the system. You scrub/tokenize/hash before persisting, encrypt and short-TTL any raw store, restrict access, and often keep only metadata + verdicts by default. Why asked: checks the privacy-vs-detection trade-off awareness that separates seniors from juniors.

  4. What is the difference between inline (real-time) and offline (async) detection for AI systems? Inline detectors run on the request path and can block, redact, or escalate before an action fires, so they must be cheap; offline analytics run over the trace store to compute baselines, aggregate consumption, and correlate across sessions where latency does not matter. You need both — inline for prevention, offline for the slow and distributed attacks inline checks miss. Why asked: tests understanding of the two-speed detection architecture.

  5. Name three consumption controls you would put on an LLM API. Per-principal request and token rate limits (token buckets), a hard max-output-tokens and request timeout, and — for agents — a max-steps/loop guard with a per-run budget kill switch. Why asked: fast check that they can operationalize LLM10.

Tier 2 — System Design & Trade-offs (screens out ~90%)

  1. Design the telemetry and detection pipeline for a customer-facing agent with tool access. Wrap the model in a gateway that captures inbound prompt + retrieved context, the proposed tool calls before execution, and the outbound response as separate structured events with shared ids. Run inline detectors (injection/jailbreak, PII/secret scan, tool-arg validation against scope) that can block or escalate to HITL, scrub content, then stream traces to a data lake for behavioral baselines, cost aggregation, and SIEM correlation, with a kill switch on confirmed abuse. Log retrieved/tool content as first-class events so indirect injection is visible, and store audit logs append-only in a separate account. Why asked: the flagship design question — tests end-to-end thinking across detection, privacy, and IR.

  2. How would you detect model-extraction attempts? Per-request limits do not catch it because extraction is a volume pattern, so you baseline query volume, diversity, and coverage per principal and across correlated keys, flag systematic space-filling queries, seed honeytokens/canary responses to detect a cloned model later, and rate-limit + step-up-auth suspicious keys. It maps to MITRE ATLAS ML Model Access / Exfiltration. Why asked: probes whether they think in aggregate/behavioral terms, not just per-request.

  3. Your injection classifier has a 3% false-positive rate and fires 50k times a day. What's wrong and how do you fix it? At that volume inline blocking over-blocks legitimate users and the alert stream is pure noise — classic alert fatigue that will hide real attacks. Fix it by severity-tiering verdicts (block only high-confidence, log-and-monitor the rest), deduplicating and risk-scoring in the SIEM, correlating with tool-call outcomes so only injections that led to a risky action page a human, and continuously tuning thresholds against a labeled corpus. Why asked: tests detection-engineering maturity and the FP/FN trade-off.

  4. How do you log enough to investigate an injection incident without hoarding PII? Always keep metadata and classifier verdicts; keep the raw payload only in a short-TTL, field-encrypted, access-controlled store gated behind break-glass, or store a hash/tokenized version for correlation. This preserves the ability to pull the exact payload when an incident is declared while keeping steady-state collection minimal and retention-bound. Why asked: the crux privacy-vs-forensics trade-off, asked to see if they can hold both.

  5. What consumption controls specifically apply to agents that don't apply to a plain chatbot? Agents loop and call tools, so beyond token/rate budgets you need max-steps per run, loop/repetition detection, a per-run cost budget with a kill switch, and per-tool call quotas — because a single injected agent run can chain many expensive or dangerous actions. Why asked: checks they understand agentic blast radius, not just request-level limits.

Tier 3 — Expert & Deep-Dive (Staff/Principal level)

  1. An attacker abuses your service slowly across thousands of low-volume keys to stay under per-key limits. How do you catch it? Per-principal limits are structurally blind to this, so you aggregate across principals with sliding windows and entity resolution (linking keys by IP, ASN, behavioral fingerprint, prompt similarity), baseline the population and flag coordinated patterns, and apply global circuit breakers on aggregate spend/volume. The detection has to move up a level from the request to the fleet. Why asked: separates candidates who only think per-request from those who reason about distributed abuse.

  2. How would you detect a confused-deputy / indirect-injection attack where a retrieved document makes the agent misuse a privileged tool? Log retrieved context and tool calls as correlated events, baseline normal tool-call sequences per workflow, and alert when a session's tool chain deviates — e.g., a summarization task suddenly invoking an export or write tool after ingesting external content. Correlate the anomalous tool call back to the untrusted retrieved text that preceded it to establish causality, and gate irreversible tool actions behind HITL regardless. Why asked: tests deep agentic threat-modeling and provenance-aware detection.

  3. What are the limits of ML-based anomaly detection on agent behavior, and how do you avoid it becoming security theater? Behavioral models drift as legitimate usage evolves, novel-but-benign workflows trigger false positives, and a patient attacker can shape traffic to poison the baseline (evasion under the model). Avoid theater by grounding alerts in consequences (risky tool calls, data egress) rather than raw novelty, pairing probabilistic detection with deterministic guardrails (least-privilege scopes, HITL), continuously red-teaming the detectors, and treating anomaly scores as one input to correlation rather than a standalone verdict. Why asked: probes intellectual honesty about probabilistic controls at Staff level.

  4. Design an audit trail that survives the attacker compromising the application host. Ship logs off-host in near-real-time to an append-only (WORM) store in a separate security account/trust domain with independent credentials, hash-chain or sign events for tamper evidence, and ensure the app's runtime identity has write-only (no delete) access. Then even if the app is fully compromised, historical events cannot be altered or erased, preserving forensic integrity for MITRE ATLAS Defense-Evasion scenarios. Why asked: tests forensic-grade logging design and threat-modeling the logging system itself.

  5. How do you monitor for sensitive-information disclosure and system-prompt leakage in model outputs at scale, given you can't store all outputs verbatim? Run inline output scanners (PII/secret detectors, similarity match against the system prompt and known secret canaries) that flag and redact before the response leaves, emit only the verdict and match category to the log rather than the leaked content, and sample or hash outputs for offline trend analysis. Seed unique canary tokens in the system prompt so any appearance in output is a high-fidelity leak signal without needing to inspect raw text. Why asked: combines output-side detection, privacy-preserving logging, and canary techniques — a true senior synthesis question.

6. Key Takeaways

  • Detection assumes prevention fails. Guardrails are probabilistic; monitoring, consumption limits, and an audit trail are how you catch and contain what slips through.
  • Log the three boundaries: inbound prompt + retrieved context, the proposed tool call before it executes, and the outbound response — with shared stable ids so an incident is reconstructable.
  • OWASP LLM10 is availability + IP: budget every axis of consumption per principal (requests, tokens, steps, timeout) and back per-request limits with cross-session aggregation and a kill switch.
  • The log is a breach surface: scrub/tokenize/hash PII and secrets before persisting, keep raw content short-TTL, encrypted, access-controlled, and retention-bound; default to metadata + verdicts.
  • Two-speed detection: cheap inline classifiers to block/redact/escalate, plus offline behavioral baselines and correlation for slow, distributed, and extraction attacks.
  • Tier and correlate alerts to consequences (risky tool calls, data egress) to beat alert fatigue — page on impact, not on raw novelty.
  • How I'd defend it in production: a monitored inference gateway that per-principal-budgets and rate-limits every call, runs layered inline detectors, logs privacy-scrubbed structured traces to an append-only store in a separate account, and streams them to a SIEM with behavioral baselines and a kill switch.
