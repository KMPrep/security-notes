Secure AI SDLC and Threat Modeling with Google SAIF

Domain: AI / ML Security — Defensive Engineering
Frameworks: Google SAIF (6 elements) · NIST AI RMF (Govern / Map / Measure / Manage) · STRIDE · OWASP Top 10 for LLM Apps · MITRE ATLAS

1. Overview

A Secure AI Software Development Lifecycle (SDLC) applies security engineering to every stage an AI feature passes through — data collection, training/fine-tuning, evaluation, deployment, and operation — rather than bolting a guardrail onto the model at the end. It sits across the entire AI system, not one layer, because an AI feature's attack surface spans untrusted training data, third-party model weights, the inference endpoint, and (for agents) the tools and memory the model can reach. This matters in production because AI systems fail in ways classic apps do not: the "logic" is learned, inputs and instructions share one channel, and an agent can be socially engineered into misusing its own privileges. Google SAIF (Secure AI Framework) gives the six-element governance and control model, NIST AI RMF gives the Govern/Map/Measure/Manage lifecycle functions, and STRIDE — adapted for AI — gives the per-component threat taxonomy. This note is the flagship "how do you secure an AI feature end-to-end" walkthrough, grounded in a concrete customer-facing agent with tool access.

2. How It Works

2.1 Core Mechanics

A secure AI SDLC treats an AI feature as a data-flow graph crossing several trust boundaries, then places controls at each boundary. The key AI-specific insight is that data is control: in an LLM, retrieved documents, tool outputs, and prior conversation turns are all interpreted as instructions, so any boundary that admits attacker-influenced text is also an injection boundary (OWASP LLM01). Threat modeling therefore starts by enumerating assets, entry points, and boundaries — then applies STRIDE per element.

Google SAIF's six elements frame the program-level controls:

  1. Expand strong security foundations to the AI ecosystem — reuse existing IAM, secrets management, secure-by-default infra, and supply-chain controls; extend them to data pipelines and model artifacts.
  2. Extend detection and response — bring AI into the org's threat universe with monitoring of prompts, outputs, and tool calls; watch for injection, abuse, and exfiltration.
  3. Automate defenses — use automation (and AI) to keep pace with attackers: automated red-teaming, guardrail scanning, and policy enforcement in CI/CD.
  4. Harmonize platform-level controls — one consistent, well-audited AI platform (approved models, sandboxed tool runtime, standard guardrails) rather than per-team snowflakes.
  5. Adapt controls / faster feedback loops — continuous red-teaming and eval feed back into training and guardrail tuning; treat the model as a living, drifting asset.
  6. Contextualize AI risks in surrounding business processes — model risk in terms of business impact (what can the agent do — refund, delete, email a customer) and put human-in-the-loop where blast radius is high.

NIST AI RMF maps cleanly onto the lifecycle: Govern (policy, roles, risk appetite) is cross-cutting; Map (context, assets, threats) is the threat-modeling step; Measure (test, red-team, eval) is the evaluation gate; Manage (deploy controls, monitor, respond) is operate-time.

Threat-modeling walkthrough — a customer-facing support agent with tool access. Assets: customer PII, the model/system prompt, tool credentials, and the actions the tools expose (issue refund, look up orders, send email). Entry points / trust boundaries: (a) the untrusted user message; (b) retrieved knowledge-base / ticket content — indirect injection surface; (c) tool outputs returning to the model; (d) the model → tool call boundary where the model's text becomes a privileged action. STRIDE adapted for AI at each element:

  • Spoofing — attacker impersonates a user or forges tool identity; also instruction spoofing where injected text impersonates the system prompt ("you are now in admin mode").
  • Tampering — training-data poisoning, RAG index poisoning, prompt/context tampering via injection (LLM01).
  • Repudiation — no audit trail linking a prompt to the tool action it triggered.
  • Information disclosure — system-prompt/PII leakage, membership/training-data extraction (LLM02).
  • Denial of service — token/compute exhaustion, wallet-drain via expensive tool loops (LLM10 unbounded consumption).
  • Elevation of privilege — excessive agency / confused-deputy: the agent uses its credentials to do what the attacker cannot (LLM06).

The single most dangerous AI-specific move is combining an injection boundary (untrusted text) with an action boundary (tool with real privilege): indirect injection in a retrieved document tells the agent to call the refund tool or email data out — a confused-deputy exfiltration path.

Top 3 Critical Mitigations

1. Least-Privilege Tool Scopes with a Per-Action Policy Engine

This is the highest-leverage control for an agent because it caps blast radius regardless of whether injection succeeds. Each tool is registered with a narrow scope (read-only order lookup vs. refund-issue), and every tool call is checked by an external policy engine (e.g. OPA-style rules) that evaluates the requesting identity, the action's impact tier, and per-user/session limits before execution. High-impact actions (refunds over a threshold, account deletion, outbound email with data) route to human-in-the-loop approval. The technical point is that the model is never the security boundary — the policy engine outside the model is, so a jailbroken model still cannot exceed granted privilege. Trade-offs: policy maintenance overhead, and human-in-the-loop adds latency and cost, so you tier it by business impact (SAIF element 6). This directly addresses the confused-deputy / excessive-agency class (OWASP LLM06), the single most damaging agent failure.

2. Layered Input/Output Guardrails Treating All Non-System Text as Untrusted

Guardrails wrap the inference boundary: an input guardrail scans user and retrieved text for injection/jailbreak patterns and policy violations; an output guardrail runs DLP and safety classification before the response (or a tool argument) leaves the model. Concrete tooling includes Llama Guard, NVIDIA NeMo Guardrails, and Rebuff for injection detection, plus deterministic DLP/regex for known secret and PII formats. The essential design rule from indirect injection is that retrieved documents and tool outputs are data, never instructions — techniques like spotlighting, delimiter fencing, and provenance tagging keep the model from executing text a third party planted (OWASP LLM01). The critical trade-off is that ML-based guardrails are probabilistic, not sound — they reduce likelihood but can be bypassed by novel phrasings and add latency — so they are a defense-in-depth layer, never the only control, and must sit behind the least-privilege wall in mitigation 1. This maps to SAIF element 3 (automate defenses) and NIST Manage.

3. Shift-Left Security Gates in the AI Pipeline (Provenance, Signing, Security Eval)

Shift-left for AI means the SDLC blocks insecure artifacts before deployment, mirroring code CI/CD. Three gates matter: data provenance (signed, allowlisted sources; anomaly/poison scanning) at the data stage; model artifact integrity (safetensors over pickle to avoid deserialization RCE, model signing via sigstore, registry scanning) at the train/registry stage; and an automated security evaluation — adversarial robustness, prompt-injection, jailbreak, and PII-leakage suites run as a required release gate, not just an accuracy check (OWASP LLM09). Continuous/automated red-teaming feeds results back into training and guardrail tuning (SAIF element 5, NIST Measure). Trade-offs: eval suites are never exhaustive and can create false confidence, and provenance/signing add pipeline friction — but catching a poisoned dataset or pickled backdoor pre-deploy is orders of magnitude cheaper than post-incident. This embodies SAIF elements 1 and 5 and NIST's Map/Measure functions.

5. Practice Questions

Tier 1 — Fundamentals (screens out ~60%)

  1. What is a Secure AI SDLC and how does it differ from a normal SDLC? It applies security controls across data, training, evaluation, deployment, and operation stages of an AI feature, adding AI-specific concerns — poisoning, injection, excessive agency — that classic SDLCs do not cover because the logic is learned and data doubles as control. Signal: candidate sees AI security as lifecycle-wide, not a single guardrail.
  2. Name Google SAIF's purpose and a few of its elements. SAIF is Google's Secure AI Framework of six elements: expand strong security foundations to AI, extend detection and response, automate defenses, harmonize platform controls, adapt controls with feedback loops, and contextualize AI risk in business processes. Signal: knows the industry-standard governance frame, not just OWASP.
  3. How does NIST AI RMF structure AI risk management? Through four functions — Govern (policy/roles), Map (context and threats), Measure (test and evaluate), and Manage (deploy controls and respond). Signal: can pair a governance framework with the technical controls.
  4. Why is prompt injection a fundamental AI risk? Because instructions and data share one channel, so any untrusted text — user input or retrieved content — can be interpreted as a command (OWASP LLM01). Signal: understands the root cause, not just the symptom.
  5. What is excessive agency and why does it matter? It is an agent having more capability or permission than needed, so a hijacked model can take real, damaging actions with its own privileges (OWASP LLM06). Signal: connects model behavior to blast radius.

Tier 2 — System Design and Trade-offs (screens out ~90%)

  1. Threat-model a customer-facing support agent with tool access, end to end. Enumerate assets (PII, tool credentials, system prompt, the actions tools expose), entry points and trust boundaries (user input, retrieved KB content, tool outputs, and the model-to-tool action boundary), then apply STRIDE adapted for AI — spoofing/instruction-spoofing, tampering via injection and poisoning, repudiation from missing audit trails, info disclosure of PII, DoS via cost loops, and elevation via confused-deputy tool abuse. The critical finding is that any boundary admitting untrusted text is an injection surface, and combining it with a privileged tool creates an exfiltration/action path. Controls follow the boundaries: input/output guardrails, treating retrieved text as data, least-privilege scoped tools behind a policy engine, human-in-the-loop for high-impact actions, egress allowlists, and full audit logging. Signal: can run a structured, AI-aware threat model and prioritize by business impact.
  2. Design guardrails for a customer-facing agent with tool access. Layer them: an input guardrail (injection/jailbreak/policy scan on user and retrieved text), an output guardrail (DLP and safety before response or tool argument), a least-privilege scoped tool registry, and a policy engine that checks each tool call's identity, impact tier, and rate before execution, escalating high-impact calls to a human. Keep the model out of the security-boundary role — the external policy engine enforces privilege — and instrument everything for detection. Accept that ML guardrails are probabilistic, so defense-in-depth and capped privilege carry the guarantee. Signal: layers probabilistic and deterministic controls and knows where the real boundary is.
  3. Where in the lifecycle do you place security gates and what does each check? Data stage: provenance, signing, PII classification, poison scanning. Train/registry: safetensors over pickle, model signing, artifact scanning. Eval: adversarial/injection/jailbreak/PII-leak suite as a required release gate. Deploy: least-privilege scopes, guardrails on, secrets out of prompt. Operate: monitoring, rate/cost limits, audit logs, continuous eval feeding back to retrain. Signal: understands shift-left applied to AI, not just runtime defense.
  4. How do you defend a RAG pipeline that ingests untrusted documents? Treat every retrieved chunk as untrusted data, never instructions — use spotlighting, delimiter fencing, and provenance tags; scan/validate ingested content; and constrain what the agent can do with retrieved instructions via least-privilege tools and egress control. Because indirect injection (LLM01) hides commands in third-party content, the retrieval boundary must be an explicit trust boundary in the model. Signal: recognizes indirect injection and controls the action side, not just the text side.
  5. What trade-offs come with human-in-the-loop and how do you decide where to use it? It adds latency, cost, and operator fatigue, so blanket approval is impractical; tier it by business impact (SAIF element 6) — auto-allow low-impact read actions, require approval for high-blast-radius actions like large refunds, deletions, or outbound data. This bounds worst-case damage where it matters while keeping the agent responsive elsewhere. Signal: reasons about controls in business-risk terms, not absolutes.

Tier 3 — Expert and Deep-Dive (Staff / Principal)

  1. Guardrails are probabilistic — how do you build a system that stays safe when they fail? Assume bypass and make the failure survivable: the load-bearing control is capped privilege — least-privilege scoped tools behind an external policy engine — so a fully jailbroken model still cannot exceed granted permissions or reach unapproved egress. Guardrails then reduce likelihood, human-in-the-loop caps high-impact blast radius, and audit logging plus detection catch what slips through. The design principle is that soundness comes from the deterministic authorization boundary outside the model, with probabilistic layers stacked in front of it. Signal: distinguishes provable from probabilistic controls and architects for failure.
  2. How would an attacker chain indirect injection into data exfiltration in an agent, and how do you cut the chain? They poison a document the agent will retrieve (RAG index or a support ticket) with instructions telling the agent to read PII and call an outbound tool — email or web fetch — with that data; the agent, acting as a confused deputy, uses its own credentials. Cut the chain at multiple points: strip/neutralize instructions in retrieved text, deny outbound tools access to raw PII, enforce an egress allowlist, require human approval for outbound data actions, and log the prompt-to-action trail for detection. No single control suffices, which is why boundaries stack. Signal: reasons about multi-step attack chains and layered interruption.
  3. How do you secure multi-agent delegation without amplifying privilege? Each agent gets its own scoped identity and least-privilege tools; delegation passes a narrowed capability, never the delegator's full privilege, and trust boundaries exist between agents so one agent's output is untrusted input to the next (subject to guardrails). A central policy engine authorizes every tool call by the acting identity, and audit logs trace the full delegation chain. The failure mode to avoid is transitive privilege escalation where a low-trust agent's request rides a high-trust agent's credentials. Signal: applies confused-deputy and least-privilege reasoning to agent-to-agent trust.
  4. How do you build a security evaluation gate for models, and what are its limits? Assemble a suite covering adversarial robustness, direct and indirect prompt injection, jailbreak resistance, PII/training-data leakage, and tool-misuse scenarios, run it in CI as a required release gate with automated red-teaming, and feed failures back into training and guardrail tuning (SAIF element 5, NIST Measure). The limit is that evals are never exhaustive and can create false confidence, so they gate known risks while runtime controls and monitoring cover the unknown — testing lowers risk but cannot prove safety. Signal: knows Measure vs Manage and the epistemics of ML testing.
  5. Map SAIF and NIST AI RMF onto a concrete pipeline and defend where they add value over just OWASP LLM Top 10. OWASP LLM Top 10 enumerates what can go wrong; SAIF and NIST tell you how to run a program around it. NIST Govern/Map/Measure/Manage structures policy, threat modeling, evaluation, and operations; SAIF's six elements push reuse of existing security foundations, detection/response for AI, automation, platform harmonization, adaptive feedback loops, and business-context risk framing. On a pipeline, that means provenance and signing (SAIF 1), an AI-aware SOC (SAIF 2), automated guardrail scanning in CI (SAIF 3), one approved platform (SAIF 4), continuous red-team feedback (SAIF 5), and impact-tiered human approval (SAIF 6) — governance and lifecycle depth that a vulnerability list alone does not provide. Signal: can integrate multiple frameworks into an operating model, a Staff/Principal expectation.

6. Key Takeaways

  • Secure the whole lifecycle — data, train, eval, deploy, operate — not just the model; each stage needs its own gate, which is shift-left applied to AI.
  • Google SAIF = six elements (foundations, detection/response, automate, harmonize, adapt, contextualize); NIST AI RMF = Govern/Map/Measure/Manage; use STRIDE adapted for AI to threat-model per component.
  • In AI systems data is control — any boundary admitting untrusted text (user input, RAG content, tool output) is a prompt-injection boundary (OWASP LLM01).
  • The most dangerous pattern is an injection boundary meeting a privileged action boundary — indirect injection plus an over-scoped tool is confused-deputy exfiltration (LLM06).
  • Guardrails are probabilistic, not sound — the real security boundary is a deterministic, least-privilege policy engine outside the model, with guardrails as defense-in-depth in front of it.
  • Tier controls by business impact: human-in-the-loop and tighter scopes where blast radius is high, automation where it is low.
  • One-line production answer: "Threat-model the feature with STRIDE across every trust boundary, gate each SDLC stage (provenance/signing/security-eval), then run the agent behind least-privilege scoped tools and a policy engine with layered guardrails, egress allowlists, human-in-the-loop for high-impact actions, and full prompt-to-action audit logging — anchored to SAIF and NIST AI RMF."
