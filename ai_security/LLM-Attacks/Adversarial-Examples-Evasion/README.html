Adversarial Examples and Evasion Attacks

Domain: AI / ML Security — Inference-time Evasion
Frameworks: MITRE ATLAS (AML.T0015 Evade ML Model, AML.T0043 Craft Adversarial Data, AML.T0042 Verify Attack), OWASP LLM Top 10 (LLM01 Prompt Injection as guardrail evasion), Google SAIF (Model evasion / robustness), NIST AI RMF (Measure / Manage — robustness)

1. Overview

Adversarial examples are inputs with small, deliberately crafted perturbations that cause a machine-learning model to make a confident wrong prediction while looking benign or unchanged to a human. They live at the inference boundary: the attacker does not touch training data or weights, they just query the deployed model with malicious inputs. This matters most when ML is itself a security control — malware classifiers, spam and phishing filters, fraud scoring, content moderation, and LLM guardrails — because a successful evasion turns the detector into a blind spot while defenders still believe it works. MITRE ATLAS classifies this as Evade ML Model (AML.T0015) built from Craft Adversarial Data (AML.T0043), and the same mental model explains how attackers slip prompts past LLM safety classifiers. The defining property is that robustness is statistical, not sound: no deployed defense provably eliminates adversarial inputs, so the goal is raising attacker cost, not achieving perfection.

2. How It Works

2.1 Core Mechanics

A classifier learns a decision boundary in a high-dimensional feature space. Because that boundary is learned from finite data and the input space is enormous, there are always regions near a correct sample where the model's output flips. An adversarial example is a point moved just across the boundary while staying inside a small perturbation budget — typically bounded by an L-infinity, L2, or L0 norm so the change is imperceptible or semantically irrelevant to a human.

White-box attackers have the model, its architecture, and gradients. They compute the loss gradient with respect to the input and step the input in the direction that most increases loss:

  • FGSM (Fast Gradient Sign Method): one step of size epsilon in the sign of the gradient. Cheap, fast, weak.
  • PGD (Projected Gradient Descent): iterative FGSM that takes many small steps and projects back into the epsilon-ball each iteration. It is the standard strong first-order attack and the benchmark defenses are measured against.
  • C&W (Carlini & Wagner): optimization-based, minimizes perturbation size subject to misclassification; strong for breaking claimed defenses.

Black-box attackers only see outputs. They evade by (1) transfer — craft on a local surrogate model and reuse, since adversarial examples generalize across models trained on similar data; (2) query-based / score-based estimation — approximate gradients from confidence scores (ZOO, NES); or (3) decision-based attacks that use only the hard label (Boundary Attack, HopSkipJump). Query attacks are the realistic threat against a hosted API.

The trust boundary is the model's input channel. What crosses it is attacker-controlled data that the model treats as ground truth. For security classifiers the attacker also faces a second constraint absent in image research: the evasive sample must remain functionally valid — malware must still execute, spam must still be readable, a fraudulent transaction must still transfer money. This is the problem-space vs feature-space gap, and it is what makes real security evasion harder and more durable than academic image attacks.

Top 3 Critical Mitigations

1. Adversarial Training

The strongest known empirical defense. You augment training by generating adversarial examples (typically PGD) on-the-fly each batch and training the model to classify them correctly, minimizing the worst-case loss inside the perturbation budget rather than the average-case loss. This bakes robustness into the weights instead of bolting it on at inference. The limits are real and must be stated in practice: it only defends the threat model you trained against (a given norm and epsilon), it costs several times more compute, it usually reduces clean accuracy (the robustness–accuracy trade-off), and an attacker using a larger budget or a different norm can still win. It raises attacker cost; it does not close the gap.

2. Input Preprocessing and Randomization

Transform the input before the model sees it — JPEG compression, bit-depth reduction (feature squeezing), spatial smoothing, or adding random noise / random resizing. The idea is to destroy the fragile high-frequency perturbation while preserving the semantic content. Randomized smoothing goes further and gives a provable (certified) robustness radius by classifying many noised copies and taking a majority vote. The critical caveat: many simple preprocessing defenses work only because they cause gradient masking / obfuscation, which gives a false sense of security — an adaptive attacker who differentiates through the transform (or uses BPDA, Backward Pass Differentiable Approximation) breaks them. Certified defenses are honest but only cover small perturbation budgets and add latency.

3. Defense in Depth — Do Not Let One ML Model Be the Whole Control

The most important architectural mitigation, and the one senior engineers are expected to reach for. Because no single model is robust, combine the ML classifier with independent signals: deterministic rules, reputation and provenance data, ensembles that must agree, rate limiting and query-anomaly detection on the API, and human-in-the-loop escalation for low-confidence or high-impact decisions. An attacker must now evade several uncorrelated controls simultaneously, which is far harder than fooling one boundary. This maps directly to SAIF's layered-controls principle and NIST AI RMF's Govern/Manage functions, and it is the pragmatic answer to "you can't make the model perfect, so stop depending on it being perfect."

5. Practice Questions

Tier 1 — Fundamentals

  1. What is an adversarial example? An input with a small crafted perturbation that causes a model to misclassify while looking unchanged or benign to a human. It exploits the fact that a learned decision boundary is fragile in high-dimensional space. This is asked to confirm you understand inference-time evasion versus data poisoning.
  2. How does FGSM differ from PGD? FGSM takes a single gradient-sign step of size epsilon; PGD takes many small steps and projects back into the epsilon-ball each time, making it far stronger. PGD is the standard benchmark attack. This is asked to test whether you know the canonical attacks by name and behavior.
  3. What is the difference between white-box and black-box attacks? White-box attackers have the model and its gradients and craft optimally; black-box attackers see only outputs and rely on transfer or query-based estimation. The distinction sets the realistic threat model for a hosted API. This is asked to gauge threat-modeling instinct.
  4. What is transferability? Adversarial examples crafted on one model often fool a different model trained on similar data, so an attacker can craft on a local surrogate and evade a target they cannot access. It is why black-box attacks are practical. This is asked because it undermines security-by-obscurity of model internals.
  5. Why does this matter more when ML is a security control? If the model is a malware, spam, fraud, or moderation detector, a successful evasion is a security bypass while defenders still believe the control works. The blind spot is invisible. This is asked to see if you connect ML robustness to real security impact.

Tier 2 — System Design and Trade-offs

  1. Design a robust ML-based malware detection pipeline. Do not rely on one classifier: combine the ML score with signatures, reputation/provenance, and behavioral sandbox signals; require agreement or escalate on low confidence; adversarially train the model and evaluate with adaptive attacks; and enforce the problem-space constraint that evasive samples must still execute, which limits the attacker far more than image research suggests. Add API rate limiting and query-anomaly detection to slow black-box probing, and a retraining loop fed by telemetry. This is asked to see defense-in-depth and lifecycle thinking, not a single silver bullet.
  2. Adversarial training reduces clean accuracy — how do you decide if it is worth it? Frame it as a business risk trade-off: quantify the cost of a missed detection (fraud loss, malware breach) versus the cost of extra false positives and lost accuracy, and pick the operating point on the ROC that the business can tolerate. Use robust models where the adversary is real and stakes are high, and lighter models plus compensating controls elsewhere. This is asked to test that you reason about robustness as an engineering trade-off, not an absolute.
  3. How would you defend a hosted API against black-box query attacks? Rate-limit per identity, round or coarsen confidence scores so gradient estimation is noisier, monitor for query patterns that walk the decision boundary, and require authentication so attackers cannot cheaply parallelize probing. None of these stop transfer attacks, so pair them with a robust and ensembled model. This is asked to test that you defend the deployment surface, not just the model.
  4. How do adversarial examples relate to LLM guardrail evasion? A safety classifier or guardrail is just another ML boundary, and adversarial suffixes, encodings, paraphrasing, and greedy-coordinate-gradient attacks are the LLM analog of perturbations that push a prompt across it. OWASP LLM01 prompt injection is largely evasion of the model's alignment and any classifier in front of it. This is asked to see you generalize classic ML evasion to modern LLM systems.
  5. When would you accept an ML detector you know can be evaded? Almost always — no detector is unbeatable, so you accept it when it meaningfully raises attacker cost and is backed by layered controls, monitoring, and human review for high-impact cases. The failure is treating a probabilistic control as sound. This is asked to test mature, non-absolutist security judgment.

Tier 3 — Expert and Deep-Dive

  1. What is gradient masking and why is it dangerous? A defense causes gradient masking when it hides or breaks the gradient signal (via non-differentiable steps, randomization, or shattered gradients), making standard attacks fail and the model appear robust. It is dangerous because an adaptive attacker who approximates the gradient (BPDA) or attacks a differentiable surrogate breaks it, so the reported robustness was an illusion. This is asked to separate people who benchmark against weak attacks from those who understand adaptive evaluation.
  2. Explain the problem-space vs feature-space gap for security evasion. Image attacks perturb continuous pixels freely, but a malware or fraud attacker must produce a real artifact that still functions, so many feature-space perturbations are not realizable and adding a NOP or padding changes many features at once. This constraint both limits attackers and means a feature-space robustness claim can overstate real-world safety in either direction. This is asked to test depth beyond academic image benchmarks.
  3. How do certified defenses differ from empirical ones, and what do they actually promise? Empirical defenses like adversarial training resist known attacks but offer no guarantee; certified defenses like randomized smoothing or interval bound propagation give a mathematically provable radius within which no perturbation can change the prediction. The catch is that the certified radius is small, the methods cost accuracy and latency, and anything outside the radius is undefended. This is asked to test the provable-versus-probabilistic distinction that Staff-level candidates must own.
  4. Walk through how you would adversarially evaluate a defense before shipping it. Use adaptive attacks tuned to the specific defense rather than off-the-shelf FGSM, check for gradient masking (does a stronger attack, more iterations, or a transfer attack still fail?), evaluate across multiple norms and budgets, and report robust accuracy under the strongest attack you can mount, not the weakest. Follow the Carlini et al. evaluation guidelines and treat a defense as unproven until an adaptive attacker has tried and failed. This is asked to see rigorous, adversarial evaluation discipline.
  5. Why do physical-world adversarial patches work, and what makes them hard? A patch must be robust to viewing angle, distance, lighting, and print/camera distortion, so it is optimized over a distribution of transformations (Expectation over Transformation) rather than a single clean image, producing a durable perturbation that survives the physical channel. They are hard because that robustness requirement shrinks the attacker's freedom and the patch is often visible, but successful ones (stop-sign misclassification, face-recognition evasion) show the boundary fragility persists in the real world. This is asked to test whether you can reason about attacks outside the clean digital setting.

6. Key Takeaways

  • Adversarial examples are an inference-time evasion: crafted perturbations flip a model's output while looking benign to humans, exploiting the fragility of learned decision boundaries. MITRE ATLAS maps this to Evade ML Model (AML.T0015) via Craft Adversarial Data (AML.T0043).
  • Know the ladder of attacks: FGSM (one step), PGD (iterative, the benchmark), C&W (optimization); and the access model — white-box gradients, black-box transfer, query/score-based, and decision-based.
  • Transferability means model secrecy is not a defense — attackers craft on a surrogate and evade a target they never touched.
  • This is a security problem specifically because ML is often the control itself — malware, spam, fraud, moderation, and LLM guardrails — so evasion is a silent bypass. LLM jailbreaks are the same phenomenon (OWASP LLM01).
  • Best empirical defense is adversarial training, but it only covers the trained threat model, costs compute, and lowers clean accuracy; preprocessing defenses often just cause gradient masking and fall to adaptive attacks; certified defenses are honest but small-radius.
  • Robustness is probabilistic, not sound — always evaluate with adaptive attacks, never trust robustness measured against weak attacks.
  • How I would defend this in production: never let one ML model be the whole control — combine a robustly-trained, ensembled classifier with independent non-ML signals, API rate limiting and query-anomaly detection, and human-in-the-loop escalation for low-confidence, high-impact decisions.
