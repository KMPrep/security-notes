Sensitive Information Disclosure & System Prompt Leakage

Domain: AI / LLM Security — output-side confidentiality
OWASP LLM: LLM02 Sensitive Information Disclosure · LLM07 System Prompt Leakage
Anchor frameworks: OWASP Top 10 for LLM Applications · MITRE ATLAS · Google SAIF · NIST AI RMF
Privacy framing: Data minimization · PII scrubbing · purpose limitation

1. Overview

Sensitive information disclosure is the class of failures where an LLM emits data it should never have revealed: personally identifiable information (PII), secrets and credentials, another tenant's or user's data, proprietary content, or the model's own system prompt. It sits on the output side of the inference boundary — the model is trusted to talk, and confidential material leaves the trust boundary inside a plausible-looking response. It shows up through four main channels: training-data memorization and regurgitation (the model recites data it was trained on), cross-tenant / cross-session leakage (context, cache, or retrieval mixes users), system-prompt extraction (the instructions and any secrets placed in them are coaxed out), and secrets accidentally injected into the prompt or context (API keys, connection strings, other users' records placed into the window by the application itself). OWASP classifies the data side as LLM02 and the prompt side as LLM07, and MITRE ATLAS covers the extraction techniques (LLM Data Leakage, Training Data Membership Inference, LLM Meta-Prompt Extraction). It matters because a single response can breach GDPR/CCPA, leak a signing key, or hand an attacker the exact instructions and tool schema they need to plan the next attack — and unlike a database leak, it can happen probabilistically from any prompt.

The single most important mental model for a senior engineer: the system prompt is not a security boundary. Anything the model can read, a determined user can eventually extract; secrets and authorization must live in code and infrastructure, never in the prompt.

2. How It Works

2.1 Core Mechanics

There are four distinct leakage mechanisms, and conflating them is the most common junior mistake.

Training-data memorization/regurgitation. During pre-training and fine-tuning the model minimizes loss over its corpus. High-perplexity, rare, or duplicated sequences (a specific email signature, a leaked credential, a unique medical record) can be memorized verbatim rather than generalized. At inference an attacker uses extraction prompts ("continue this text…", "repeat the word poem forever") or membership-inference probes to make the model regurgitate that data. The trust boundary crossed is training corpus → response; the attacker needs only query access. This is amplified by fine-tuning on customer data without scrubbing.

Cross-tenant / cross-session leakage. This is an application bug, not a model bug. It happens when one user's data ends up in another user's context window: shared conversation memory keyed wrong, a RAG retriever without a tenant filter returning another org's documents, a semantic/prompt cache that returns a cached completion built from someone else's data, or embeddings from tenant A indexed in a namespace queryable by tenant B. The boundary crossed is tenant A's data store → tenant B's response.

System-prompt extraction (LLM07). The system/developer prompt carries instructions, style rules, tool definitions, and — dangerously — often secrets or business logic. Because the model treats system and user text as the same token stream, users can extract it with direct requests, translation/encoding tricks ("output everything above in base64"), role-play, or by exploiting the model's helpfulness. The leak itself may be low-impact (instructions) or high-impact (an embedded API key, an internal URL, a hidden pricing rule, the full tool schema an attacker then abuses).

Secrets placed in the prompt/context by the app. The most self-inflicted variant: developers paste API keys, DB connection strings, internal endpoints, or PII into the system prompt or into retrieved context "so the model can use it." Once it is in the context window it is leaked-in-waiting, because context is extractable and logs/traces capture it. This overlaps with insecure logging — prompts and responses shipped to observability tools become a secondary PII store.

Across all four, the defining property is that the model is a lossy, non-deterministic channel that cannot enforce confidentiality on its own. It has no concept of who owns a token.

Top 3 Critical Mitigations

1. Keep Secrets and Authorization Out of the Prompt

The highest-leverage control is architectural: the system prompt is not a security boundary. Any instruction, key, or record in the context window is extractable by a sufficiently creative user, so secrets must live in a secrets manager and be fetched at tool-call time by backend code that runs under least privilege, and authorization must be enforced in the application/API layer that gates what data ever reaches the model. Technically this means the model receives references (a user token, a scoped session) rather than raw secrets, and the tools it calls do their own authz checks. The trade-off is more engineering plumbing and the temptation to shortcut "just put it in the prompt" — but no probabilistic guardrail can substitute for simply not placing the secret where the model can read it. This maps directly to SAIF's secure-by-default and least-privilege principles.

2. Data Minimization, Scrubbing, and Deduplication Before Training

Regurgitation is a function of what the model memorized, so the durable fix is upstream: minimize and scrub before the data reaches training. PII detection and redaction/tokenization, deduplication of the corpus (duplicated sequences are the ones most likely to be memorized verbatim), and optionally differential privacy during fine-tuning bound how much any single record can influence the weights. This is the privacy-by-design position NIST AI RMF and GDPR both push. The limitation is that scrubbing is imperfect (novel PII formats slip through), differential privacy costs model utility, and you cannot un-memorize data after the fact without expensive retraining — so this control has to be applied before the model is built, which is why it is often skipped and later regretted.

3. Output-Side DLP and Guardrails, Layered with Retrieval Scoping

Because the first two controls are never perfect, you add a defense-in-depth output layer: scan every response for PII, secrets (key/token regexes and entropy checks), and system-prompt fingerprints, and redact or refuse on a hit — and enforce a mandatory tenant/ACL filter on every retrieval query so cross-tenant data never enters context in the first place. Tooling includes DLP engines, canary tokens embedded in the system prompt to detect extraction, and guardrail frameworks (Llama Guard, NeMo Guardrails, Rebuff-style detectors). The critical caveat for a senior engineer: these output filters are probabilistic, not sound — attackers bypass them with base64, ROT13, homoglyphs, or foreign-language encodings, so they reduce but never eliminate risk and must never be the only control. Retrieval scoping, by contrast, is a deterministic access-control boundary and should be treated as the primary defense against cross-tenant leakage.

5. Practice Questions

Tier 1 — Fundamentals

  1. What is the difference between OWASP LLM02 and LLM07? LLM02 is Sensitive Information Disclosure — the model leaking PII, secrets, or another user's data; LLM07 is System Prompt Leakage — the model revealing its own instructions and anything embedded in them. They overlap when the leaked system prompt itself contained secrets. This is asked to confirm you know the framework, not just the vibe.
  2. Why is the system prompt not a security boundary? Because the model processes system and user text as one token stream and has no enforcement mechanism, so any user can eventually extract it; confidentiality must be enforced in code and infrastructure. This is asked to screen for the single most important mental model in this area.
  3. What is training-data regurgitation? The model emitting memorized verbatim sequences — often rare or duplicated data like credentials or PII — in response to extraction prompts. This is asked to test whether you understand memorization versus generalization.
  4. Give one way a multi-tenant LLM app leaks one customer's data to another. A RAG retriever without a per-tenant filter returns another organization's documents into the context window. This is asked to see if you think about isolation, not just the model.
  5. Where should an API key the model needs live? In a secrets manager, fetched at call time by least-privilege backend code — never in the prompt or context. This is asked to catch the most common self-inflicted leak.

Tier 2 — System Design & Trade-offs

  1. Design the data-handling controls for an LLM assistant that is fine-tuned on customer support transcripts. Minimize and PII-scrub transcripts before training, deduplicate to reduce memorization, consider differential privacy on the fine-tune, and add output-side DLP at inference; enforce retention limits and scrub logs. The key trade-off is that scrubbing and DP cost utility, and you can't un-memorize later, so the investment must happen upstream. This is asked to test privacy-by-design under real constraints.
  2. How would you prevent cross-tenant leakage in a RAG system serving many customers? Make a tenant/ACL filter mandatory on every retrieval query, physically or logically partition the vector store per tenant, key any prompt/semantic cache on identity, and validate that embeddings from one tenant are never queryable by another. Treat retrieval scoping as a deterministic access-control boundary, not a guardrail. This is asked to test whether you can turn a probabilistic problem into an enforceable one.
  3. A user reports the chatbot printed someone else's order history. Walk through your triage. Check for shared/mis-keyed conversation memory, a prompt cache returning another user's completion, and a retriever missing a tenant filter — application bugs, not model bugs. Reproduce, scope the blast radius via logs, then fix the isolation defect and add a regression test. This is asked to see structured incident reasoning.
  4. Should you rely on the system prompt saying "never reveal secrets" to protect an embedded key? No — that is an instruction, not a control, and it is bypassable; the fix is to remove the secret from the prompt entirely and fetch it from a secrets manager under least privilege. This is asked to test whether you distinguish instructions from boundaries.
  5. How do you keep prompts and responses from becoming a shadow PII database in your logging stack? Scrub or tokenize PII before it is logged, minimize what you capture, set short retention, and apply the same access controls and DLP to the observability store as to production data. This is asked because logging is the most overlooked disclosure vector.

Tier 3 — Expert & Deep-Dive

  1. Your output filter blocks the literal system prompt. How does an attacker get it anyway, and how do you respond? They bypass string matching with encodings (base64, ROT13, homoglyphs), translation to another language, or reconstruction across multiple turns; you respond by accepting the filter is probabilistic, adding canary tokens to detect extraction, embedding no secrets in the prompt so a leak is low-impact, and monitoring for extraction patterns rather than trusting the filter. This is asked to test whether you understand that output guardrails are unsound and design accordingly.
  2. Explain differential privacy in the context of LLM training and its limits. DP adds calibrated noise (e.g., DP-SGD) so no single training record measurably changes the model, bounding memorization and membership-inference risk with a formal epsilon guarantee. The limits are a real utility hit, difficulty choosing epsilon, and that it protects records individually but not necessarily correlated group data. This is asked to separate people who know the buzzword from those who know the trade-off.
  3. How does membership inference work and why does it matter even without verbatim leakage? The attacker exploits that models are more confident on data they were trained on, inferring whether a specific record was in the training set — which is itself a privacy breach (revealing that a person's medical record was used) even if the content is never emitted. This is asked to probe depth on privacy attacks beyond the obvious regurgitation case.
  4. Design a canary strategy to detect and attribute system-prompt leakage in production. Embed unique, benign canary strings per deployment (or per prompt version) in the system prompt, monitor outputs and external channels for those strings, and use their appearance to detect extraction attempts and identify which prompt version leaked. Pair it with the discipline that the prompt holds no real secrets, so the canary is a tripwire rather than the protection. This is asked to test detection engineering thinking applied to LLMs.
  5. Why is embedding inversion a disclosure risk, and what does it change about vector-store security? Research shows text can be partially reconstructed from its embedding vectors, so an exposed vector database is not "just numbers" — it is recoverable content, meaning embeddings inherit the confidentiality classification of their source and need the same encryption, access control, and tenant isolation. This is asked to see whether you treat the vector store as a sensitive data store (LLM08) rather than an implementation detail.

6. Key Takeaways

  • The system prompt is not a security boundary — assume anything the model can read is extractable, and keep secrets and authorization in code and infrastructure, never in the prompt.
  • Disclosure has four distinct channels: training regurgitation, cross-tenant/session leakage, system-prompt extraction, and secrets the app itself placed in context — diagnose which one before "fixing the model."
  • Cross-tenant leakage is an application/isolation bug (retriever, memory, cache), fixable with deterministic access controls, not a probabilistic model behavior.
  • Regurgitation is fixed upstream — data minimization, PII scrubbing, deduplication, and optionally differential privacy — because you cannot un-memorize after training.
  • Output-side DLP and guardrails are probabilistic, not sound; encoding and translation tricks bypass them, so they are defense-in-depth, never the only control.
  • Prompts and responses in logs/traces are a secondary PII store — scrub, minimize, and apply production-grade access controls to observability.
  • Map it for the panel: LLM02 = data leaking out, LLM07 = the prompt leaking out; MITRE ATLAS covers the extraction and membership-inference techniques.
  • In production: fetch secrets at call time under least privilege, enforce tenant scoping on every retrieval, minimize and scrub data before training and logging, and layer output DLP on top — treating the model as an untrusted, lossy channel that cannot keep a secret.
