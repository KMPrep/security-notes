Training Data & Model Poisoning

Domain: AI / LLM Security — Data & Training Pipeline
Frameworks: OWASP LLM04 (Data and Model Poisoning), MITRE ATLAS (AML.T0020 Poison Training Data, AML.T0018 Backdoor ML Model, AML.T0019 Publish Poisoned Datasets, AML.T0059 Erode Dataset Integrity), Google SAIF (Training-data poisoning risk), NIST AI RMF (Map / Measure / Manage)

1. Overview

Data and model poisoning is the adversarial manipulation of any dataset the model learns from — pre-training web scrapes, supervised fine-tuning sets, RLHF preference data, or a live RAG/embedding corpus — so that the trained or retrieval-augmented model behaves as the attacker intends. It sits at the data and training layer of the AI lifecycle, upstream of inference, which makes it uniquely durable: the malicious behavior is baked into the weights or the index rather than injected at query time. Poisoning can implant a backdoor/trojan (a hidden trigger phrase that flips the model's output), inject bias or misinformation, or simply degrade overall accuracy (availability attack). It matters because modern LLMs train on billions of largely untrusted, web-scraped tokens where an attacker only needs to control a tiny fraction of samples, and because poisoned open-weights or datasets propagate through the supply chain into every downstream fine-tune. OWASP classifies it as LLM04, MITRE ATLAS covers it under the poison/backdoor techniques above, and SAIF/NIST treat training-data integrity and provenance as first-class governance controls.

2. How It Works

2.1 Core Mechanics

A model's behavior is a function of its training distribution. If an attacker can shift that distribution — even slightly and in a targeted way — they shift the learned function. The trust boundary being crossed is untrusted data becoming trusted training signal: once a sample is in the corpus, gradient descent treats it as ground truth. There is no runtime prompt to filter because the compromise already happened during training.

Attackers operate at different stages, trading off access for leverage:

  • Pre-training poisoning requires no privileged access — the attacker just needs data to appear on the public web (Wikipedia edits, GitHub repos, poisoned Common Crawl pages, expired-domain injection). Leverage per sample is low (diluted across billions of tokens), but backdoors survive even at very low poison rates: published research (Carlini et al., "Poisoning Web-Scale Training Datasets") shows controlling as little as 0.01% of a dataset is feasible and sufficient for some effects, and Anthropic/UK-AISI work (2024–2025) found a roughly constant number of poison documents (~250) can install a backdoor regardless of model size.
  • Fine-tuning / RLHF poisoning requires access to a smaller, curated dataset or the preference-labeling process. Leverage per sample is far higher because the set is small and the training signal is sharp — a handful of trigger/target pairs or a few flipped preference labels can install a reliable backdoor or bias. This is the dominant real-world risk: teams routinely fine-tune on scraped, crowdsourced, or third-party data.
  • RAG / embedding poisoning does not touch weights at all — the attacker plants documents in the retrieval corpus (or manipulates chunks/metadata) so the retriever surfaces attacker-controlled content, which then drives generation. It is fast to exploit and fast to remediate (delete the document) versus retraining.

Backdoor / trojan triggers. The attacker crafts samples pairing a rare trigger (a token like cf, an unusual phrase, a Unicode zero-width sequence, or even a stylistic pattern) with a target behavior (emit a specific answer, classify as benign, leak a secret, produce vulnerable code). The model learns a shortcut: when trigger present → do target. On clean inputs it behaves normally, so accuracy metrics and standard evals look fine — the trojan is dormant until the trigger fires. This clean-label stealth is what makes trojans hard to catch.

Split-view / frontrunning poisoning of web scrapes. Web-scale datasets are distributed as URL lists plus a snapshot hash/date, not as the raw content — consumers re-download later. Two practical attacks exploit this:

  • Split-view (domain-hijack) poisoning: the content at a URL when the dataset was indexed differs from the content served when downstream users actually crawl it. Attackers buy expired domains that appear in dataset index lists and serve poisoned content to every future scraper. The dataset maintainer's view and the consumer's view diverge — hence "split view."
  • Frontrunning poisoning: for crowd-edited sources (e.g. Wikipedia), attackers time a malicious edit to be live exactly when the periodic snapshot is taken, then let normal moderation revert it afterward. The snapshot captures the poison; the live site looks clean.

Top 3 Critical Mitigations

1. Data Provenance, Signing, and Content Pinning

The root cause of most poisoning is that data enters training with unknown origin and mutable content. The strongest structural fix is to make the dataset verifiable and immutable: record lineage (where each sample came from, when, and how it was transformed), then pin content by cryptographic hash rather than by URL so a later re-download that differs is rejected — this directly defeats split-view and frontrunning attacks, whose whole premise is that content mutates between indexing and download. Sign datasets and model artifacts (e.g. sigstore-style signing, model cards, and a signed manifest of member hashes) so downstream consumers can verify integrity and authenticity before training or fine-tuning. The trade-off is operational cost and the fact that provenance proves where data came from, not that it is benign — a signed dataset can still be signed poison — so provenance is necessary but not sufficient and must pair with detection.

2. Data Anomaly and Backdoor Detection

Because provenance cannot vouch for content, you need to inspect the data and the model. On the data side, use outlier and near-duplicate detection, per-sample influence functions to find samples with outsized effect on specific predictions, and clustering to surface unnaturally uniform trigger/target pairs (clean-label and trojan samples often cluster tightly in feature space). On the model side, apply backdoor-scanning techniques — activation clustering, spectral signatures, trigger reconstruction (e.g. Neural Cleanse-style search), and adversarial eval that probes for anomalous behavior on rare tokens. The hard limitation is that detection is probabilistic and asymmetric: a well-crafted, low-rate, clean-label backdoor can evade both data filters and behavioral evals, and a deferred-trigger "sleeper agent" can survive safety fine-tuning entirely (Anthropic, 2024). Treat detection as raising attacker cost, not as a guarantee.

3. Curate and Least-Privilege the Training Supply Chain

The most exploitable stage is fine-tuning and RLHF, where small trusted sets carry huge leverage. Mitigate by treating training data like production code: use vetted, access-controlled, versioned datasets; require multiple raters and consensus for preference labels; audit crowdsourced/third-party contributions; and prefer known-clean base weights from signed, reputable sources over arbitrary hub downloads. Reproducible, versioned data builds (data version control) let you diff, roll back, and forensically trace a poisoning incident to the offending commit. The trade-off is velocity — heavy curation slows the flywheel that makes fine-tuning attractive — so scope the rigor to risk: strictest on data that shapes safety-critical or high-privilege behavior.

5. Practice Questions

Tier 1 — Fundamentals

  1. What is training-data poisoning and why is it different from prompt injection? Poisoning corrupts the data the model learns from so malicious behavior is embedded in the weights or index, whereas prompt injection manipulates a single inference at runtime. This is asked to confirm you place the attack at the training/data layer, not the inference layer.
  2. What is a backdoor or trojan in an LLM? A hidden trigger-to-target mapping the model learns during training — on clean input it behaves normally, but a specific trigger flips its output to the attacker's goal. This is asked to test whether you understand stealth (dormant until triggered) as the core threat.
  3. Which OWASP LLM risk covers this and what does it include? OWASP LLM04 Data and Model Poisoning, covering pre-training, fine-tuning, RLHF, and embedding/RAG data manipulation. This is asked to check framework literacy.
  4. Why is web-scraped pre-training data inherently risky? It is billions of tokens of untrusted, mutable, third-party content that the training process treats as ground truth with no per-source vetting. This is asked to test your grasp of the trust boundary being crossed.
  5. Is more poisoned data always needed for bigger models? No — research shows a roughly constant number of poison documents (order of hundreds) can backdoor a model largely independent of model size, so scale does not dilute the threat away. This is asked to catch the common misconception that huge datasets are self-protecting.

Tier 2 — System Design and Trade-offs

  1. Threat-model a RAG system ingesting untrusted external documents. Attack surface is the ingestion path: an attacker plants documents so the retriever surfaces attacker content that drives generation, and can also embed indirect prompt injection in those chunks. Controls include source allowlisting, ingestion-time validation and sanitization, provenance metadata on each chunk, retrieval trust scoring, and treating retrieved text as untrusted at generation time. The upside versus weight poisoning is remediation is fast — delete the document and re-index — so the design emphasis is detection and quick containment. This is asked to see if you distinguish retrieval-layer from training-layer poisoning and design accordingly.
  2. How would you defend a fine-tuning pipeline that uses crowdsourced data? Treat the dataset as code: version and sign it, enforce access control and review on contributions, use consensus labeling and rater vetting, run anomaly/outlier and influence-function scans, and hold out a trusted clean eval and backdoor probe before promoting the model. The trade-off is velocity, so scope rigor to how privileged the resulting behavior is. This is asked to test layered defense and pragmatic trade-off reasoning.
  3. Explain split-view poisoning and how to stop it. Web-scale datasets ship as URL lists plus a snapshot date, so attackers who control a URL (often by buying an expired domain in the index) serve poison to future crawlers while the maintainer saw clean content — the two views split. The fix is content-hash pinning and dataset signing so any content that differs from the indexed hash is rejected. This is asked to test knowledge of a specific, publishable real-world attack and its structural mitigation.
  4. Why can a poisoned model pass all your standard evals? Backdoors are conditioned on rare triggers absent from clean benchmark data, so accuracy and safety metrics on normal inputs look nominal while the trojan stays dormant. You need trigger-aware and adversarial evals plus activation/anomaly analysis, not just aggregate accuracy. This is asked to test whether you understand why detection is hard and metrics can mislead.
  5. How does RLHF introduce its own poisoning surface? The preference/reward signal comes from human labelers, so flipping or biasing a small number of preference comparisons can install a bias or reward-hacking behavior with high leverage because the alignment set is small and influential. Defenses are rater vetting, consensus, audits, and reward-model sanity checks. This is asked to see if you extend the threat model beyond supervised data into alignment.

Tier 3 — Expert and Deep-Dive

  1. A sleeper-agent backdoor survives safety fine-tuning — what does that imply for your controls? It means post-hoc alignment or RLHF cannot be assumed to remove a deliberately implanted, deferred-trigger backdoor (Anthropic 2024 showed the behavior can persist and even be reinforced by safety training). The implication is you cannot trust base weights of unknown provenance and must shift left to supply-chain integrity, signing, and provenance rather than relying on downstream cleanup. This is asked to probe whether you know the limits of alignment as a security control.
  2. How would you detect a clean-label backdoor that evades label filtering? Clean-label poison is correctly labeled, so filter on behavior and representation instead: influence functions to find disproportionately impactful samples, activation clustering and spectral signatures to spot the tight cluster trojan samples form, and trigger-reconstruction search over the input space. All are probabilistic and can be evaded by low-rate crafted samples, so pair them with strict provenance to reduce the candidate pool. This is asked to test depth on detection internals and their limits.
  3. Provenance proves origin, not safety — so how do you compose controls for defense in depth? Layer them: provenance and signing establish what data and weights you have and that they were not mutated (defeating split-view/frontrunning), curation and access control reduce what an attacker can inject, anomaly/influence/backdoor scanning tries to catch what got in, and continuous eval plus rollback contain what reached production. No single layer is sound, so you assume each is bypassable and require an attacker to defeat all of them. This is asked to test systems-level security reasoning rather than a single silver bullet.
  4. Compare the blast radius of pre-training vs fine-tune vs RAG poisoning and how it changes your response. Pre-training poison is low per-sample leverage but propagates to every downstream fine-tune and is nearly impossible to remove without retraining; fine-tune poison is high leverage from few samples but scoped to that model version and traceable via data versioning; RAG poison never touches weights and is remediated by deleting a document. Response therefore ranges from supply-chain prevention and signed base weights (pre-training), to dataset curation and rollback (fine-tune), to ingestion validation and fast containment (RAG). This is asked to see if you match control cost to blast radius.
  5. How would you build an incident-response capability for a suspected poisoning? You need reproducible, versioned data and model builds so you can bisect which dataset commit or ingestion batch introduced the behavior, retained lineage to trace samples to source, trigger-reconstruction and behavioral diffing to characterize the backdoor, and a rollback path to a known-clean checkpoint plus corpus purge for RAG. Without data provenance and versioning up front, post-incident forensics is nearly impossible — which is the argument for investing before an incident. This is asked to test whether you treat poisoning as an operational, not just theoretical, risk.

6. Key Takeaways

  • Poisoning attacks the data/training layer, so the malicious behavior lives in the weights or index and persists across every inference — unlike prompt injection, which is per-query.
  • Backdoors are stealthy by design: dormant on clean input, they pass standard accuracy and safety evals and only fire on a rare trigger.
  • Scale does not save you — a roughly constant, small number of poison documents can backdoor a model regardless of size, and controlling ~0.01% of a web-scale dataset is realistic.
  • Split-view and frontrunning exploit the fact that datasets ship as URL lists, not content; content-hash pinning and dataset signing are the structural fix.
  • Fine-tuning and RLHF are the highest-leverage stages — small trusted sets mean a few crafted samples or flipped labels go a long way.
  • Alignment is not a security boundary: deferred-trigger sleeper-agent backdoors can survive safety fine-tuning, so shift left to supply-chain integrity.
  • How I would defend it in production: enforce data provenance plus dataset/model signing with content-hash pinning, curate and access-control fine-tune and RLHF data, run anomaly and backdoor detection before deployment, and keep versioned reproducible builds so you can trace, roll back, and contain a poisoning incident.
