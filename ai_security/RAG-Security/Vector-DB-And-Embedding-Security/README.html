Vector Database and Embedding Security

Domain: AI / LLM Security — Retrieval-Augmented Generation (RAG)
Frameworks: OWASP LLM08 (Vector and Embedding Weaknesses), OWASP LLM02 (Sensitive Information Disclosure), MITRE ATLAS (ML Model Access, Data Poisoning, Model Inversion), Google SAIF, NIST AI RMF (Map / Measure / Manage — Privacy-Enhanced, Secure & Resilient)

Overview

Retrieval-Augmented Generation grounds an LLM by embedding documents into high-dimensional vectors, storing them in a vector database, and retrieving the nearest neighbors of a query embedding to inject as context. This makes the vector store and its embeddings a first-class security asset: they sit at the retrieval boundary between untrusted data ingestion and trusted model context. OWASP classifies these risks as LLM08 — Vector and Embedding Weaknesses, covering embedding inversion (reconstructing source text from a vector), cross-tenant leakage in shared indexes, embedding poisoning that skews what gets retrieved, weak access control on the store, and PII persisting inside embeddings. The core misconception this note corrects is that "embeddings are just numbers, so they are safe" — embeddings are lossy but invertible representations of their source text and must be treated as sensitive data. In production, the store must enforce the same tenant isolation, access control, encryption, and data-classification discipline as the underlying source database.

How It Works

Core Mechanics

A RAG pipeline has two phases. In ingestion, source documents are chunked, passed through an embedding model, and written to the vector store as vectors plus metadata (source ID, tenant, ACL tags, original text). In retrieval, the user query is embedded with the same model, and an approximate-nearest-neighbor (ANN) search returns the top-k closest chunks, which are concatenated into the prompt. Two trust boundaries matter: the ingestion boundary, where untrusted or multi-source documents enter the index and can poison it, and the query boundary, where a user (possibly from another tenant) triggers a similarity search that may surface data they should not see.

Embeddings are not one-way hashes. An embedding inversion attack trains a decoder model that maps vectors back toward their source text; with access to the embedding model or its API, an attacker can reconstruct sensitive phrases, names, and secrets from stored vectors. This is why exfiltrating a vector index is comparable to exfiltrating the source corpus. Cross-tenant leakage happens when many customers share one physical index without a hard filter on tenant identity: similarity is global, so a query can retrieve any tenant's chunk unless the search is constrained by namespace or metadata filter enforced server-side. Embedding poisoning exploits the ingestion boundary: an attacker crafts documents whose embeddings sit near high-value query regions, so their malicious or misleading chunk is retrieved and injected as trusted context — a supply-chain attack on retrieval that often chains into indirect prompt injection.

Top 3 Critical Mitigations

1. Enforce Tenant Isolation and Query-Time ACL Filtering

The single highest-impact control is ensuring similarity search can never cross a tenant or authorization boundary. This has two layers. First, hard isolation: give each tenant its own namespace, collection, or physical index so a query is scoped to their data by construction — this is stronger than a filter because a bug cannot silently return foreign data. Second, query-time filtering enforced server-side: attach tenant ID and per-document ACL tags as metadata, and require every ANN search to include a mandatory filter derived from the authenticated principal, never from client-supplied parameters. The critical failure mode is filtering in application code after an unfiltered top-k search, which still touches other tenants' vectors and leaks through timing or bugs; the filter must constrain the search itself. For freshness, ACLs on chunks must be reconciled with the source system, and permissions should be re-checked against the live source at retrieval time so revoked or deleted documents cannot resurface. The trade-off is index fragmentation and higher operational overhead for many small tenants, often mitigated with logical namespaces inside a shared cluster plus strict filter enforcement.

2. Treat Embeddings as Sensitive, Invertible Data

Because embedding inversion can reconstruct source text, an embedding inherits the sensitivity classification of what it encodes — a vector of a medical record is a medical record. Practically this means encrypting vectors at rest and in transit, applying the same key management, access logging, retention, and deletion policies as the source data, and never shipping raw embeddings to untrusted clients or third-party services. The most durable defense is not embedding secrets in the first place: run PII detection and classification before the embedding step, then redact, tokenize, or pseudonymize sensitive fields so the vector never carries recoverable identifiers. Deletion must propagate — a "right to be forgotten" request has to remove the vector and its backups, not just the source row, or inversion still recovers the data. The limitation is that scrubbing reduces retrieval utility for legitimately sensitive use cases, so high-sensitivity corpora often stay in strongly isolated, access-controlled enclaves rather than being scrubbed into uselessness.

3. Validate Ingestion and Distrust Retrieved Content

Embedding poisoning is a supply-chain attack on retrieval, and its common payoff is indirect prompt injection carried inside a retrieved chunk. Defense starts at the ingestion boundary: allowlist sources, verify provenance (signing where feasible), content-moderate and sanitize documents, and monitor for anomalous embeddings that cluster suspiciously near high-value query regions. It continues at the prompt boundary: retrieved text is untrusted input, not trusted system context, so it should be clearly delimited, guardrail-scanned for injected instructions, and constrained so the model treats it as data to reason over rather than commands to follow. Retrieval-quality monitoring and periodic re-evaluation catch slow poisoning that shifts what surfaces for key queries over time. The trade-off is that guardrails are probabilistic, not sound — they reduce but do not eliminate injection — so they must be layered with least-privilege on any tools the model can call, keeping a poisoned retrieval from escalating into a damaging action.

Practice Questions

Tier 1 — Fundamentals

  1. Why are embeddings considered sensitive data and not just anonymous numbers? Embeddings are lossy but invertible representations of their source text, and an inversion model with access to the embedding model can reconstruct PII and secrets from them. This is asked to confirm the candidate rejects the "embeddings are safe numbers" misconception that underlies most vector-store data incidents.
  2. What is OWASP LLM08 and what does it cover? LLM08 is Vector and Embedding Weaknesses, covering inversion, cross-tenant leakage, poisoning, weak access control, and PII in embeddings within RAG systems. This is asked to check the candidate can anchor vector security to a recognized framework rather than ad hoc reasoning.
  3. How does data leak between tenants in a shared vector index? Similarity search is global, so without a server-side tenant or ACL filter a query returns the nearest chunks regardless of owner. This is asked to test understanding of the query boundary as an authorization boundary.
  4. What is embedding poisoning? An attacker crafts documents whose vectors sit near high-value query regions so their malicious chunk is retrieved and injected as trusted context. This is asked to see if the candidate connects ingestion trust to retrieval outcomes.
  5. Why should retrieved chunks be treated as untrusted input? They may contain injected instructions or poisoned content, so the model should treat them as data to reason over, not commands to obey. This is asked to test the link between RAG and indirect prompt injection.

Tier 2 — System Design and Trade-offs

  1. Design tenant isolation for a multi-tenant RAG platform. Use per-tenant namespaces or physical indexes for hard isolation, and enforce a mandatory server-side filter on tenant ID and ACL tags derived from the authenticated principal, never from client input. Reconcile chunk ACLs with the source system and re-check permissions at retrieval time so revoked data cannot resurface. The trade-off is index fragmentation versus filter risk, usually resolved with logical namespaces plus strict enforcement, and I would add query anomaly monitoring to catch scoping bugs. This is asked to test whether isolation is designed by construction rather than patched in application code.
  2. How would you prevent PII from persisting in embeddings? Run classification and PII detection before the embedding step, then redact, tokenize, or pseudonymize sensitive fields so the vector never encodes recoverable identifiers. Encrypt vectors at rest, apply source-equivalent retention and deletion, and ensure deletion propagates to backups and replicas. The trade-off is reduced retrieval utility, so highly sensitive corpora may stay in isolated enclaves instead. This is asked to test privacy engineering under NIST AI RMF framing.
  3. Threat-model a RAG system that ingests untrusted documents. The ingestion boundary allows poisoning and injection payloads, so I allowlist sources, verify provenance, content-moderate, and monitor for anomalous embeddings clustering near key queries. At retrieval I filter by ACL, delimit retrieved text as untrusted, and guardrail-scan for injected instructions. I keep model tool scopes least-privilege so a poisoned chunk cannot escalate into an action. This is asked to test end-to-end boundary reasoning across ingest, retrieval, and action.
  4. How do you handle the right to be forgotten in a vector store? Deletion must remove the vector and its metadata plus all snapshots and replicas, because inversion can still reconstruct data from a lingering copy, and any downstream caches must be purged. It also requires ACL reconciliation so retrieval never surfaces the deleted content. The hard part is backups and derived indexes, which need a documented, tested purge path. This is asked to test operational privacy maturity, not just the happy-path delete.
  5. Where should the tenant filter be enforced and why does the location matter? It must constrain the ANN search itself on the server, driven by the authenticated identity, not applied in application code after an unfiltered top-k. A post-hoc filter still reads other tenants' vectors and leaks through bugs or timing, while an in-search filter fails closed. Client-supplied filter parameters are untrusted and must be ignored. This is asked to distinguish real isolation from cosmetic filtering.

Tier 3 — Expert and Deep-Dive

  1. Explain embedding inversion and what it means for exfiltration risk. An attacker trains a decoder that maps vectors back toward source text, so with access to the embedding model or its API, stealing a vector index is comparable to stealing the source corpus. This defeats the assumption that vectorization is a privacy control and makes encryption, access control, and PII scrubbing mandatory. It also means embeddings shipped to third parties are a data-sharing event. This is asked to test whether the candidate understands the cryptographic-grade reasoning behind treating embeddings as sensitive, mapping to ATLAS model inversion.
  2. How can embedding poisoning bypass content filters that only scan the query? The malicious content lives in ingested documents, not the user query, so query-side guardrails never see it; the poisoned chunk is injected into the prompt at retrieval time and carries instructions or misinformation. Defense must move to the ingestion boundary and to treating retrieved context as untrusted, plus retrieval-quality monitoring for drift. Because guardrails are probabilistic, they pair with least-privilege tool scopes to cap blast radius. This is asked to test layered defense and awareness that the attack surface is the corpus, not just the input.
  3. Design defenses against membership inference on a RAG corpus. An attacker probes with crafted queries and infers from retrieval behavior or answer confidence whether a specific record is present, which is itself a privacy breach for sensitive datasets. Mitigations include rate limiting, query anomaly detection, minimizing returned metadata and scores, and for high-sensitivity data, isolation and access control so probing is not even possible unauthenticated. There is a utility trade-off in suppressing similarity scores and detail. This is asked to probe subtle privacy attacks beyond obvious data theft.
  4. Why is filtering after an unfiltered nearest-neighbor search a security bug, not just inefficiency? The unfiltered search physically touches vectors belonging to other tenants or unauthorized documents, so any downstream bug, exception, logging path, or timing side channel can leak them, and the system fails open. Correct design pushes the authorization predicate into the search so out-of-scope vectors are never candidates, making it fail closed. This is the vector-store analog of doing authorization in the query, not in the view. This is asked to test fail-closed design instinct at staff level.
  5. How do you reconcile vector-store ACLs with a changing source system to avoid stale-permission leakage? Chunk-level ACL metadata drifts as source permissions, ownership, and deletions change, so the store needs continuous reconciliation from the source of truth plus a retrieval-time re-check against live permissions before content reaches the prompt. Event-driven updates on permission changes are stronger than periodic batch sync for high-sensitivity data, and deletions must fan out to backups. The trade-off is latency and complexity from live authorization checks. This is asked to test the hardest, most-overlooked failure mode: correct data with outdated permissions.

Key Takeaways

  • Embeddings are lossy but invertible; treat a vector with the same sensitivity classification as its source text.
  • OWASP LLM08 covers inversion, cross-tenant leakage, poisoning, weak access control, and PII in embeddings.
  • Tenant isolation must be enforced by construction (namespaces) plus a server-side filter inside the ANN search, not after it — fail closed, never open.
  • Embedding poisoning is a supply-chain attack on retrieval that commonly chains into indirect prompt injection; validate ingestion and distrust retrieved content.
  • Scrub or tokenize PII before embedding, encrypt vectors at rest and in transit, and propagate deletion to backups and replicas.
  • Reconcile chunk ACLs with the source system and re-check permissions at retrieval time to prevent stale-permission leakage.
  • Production defense in one line: isolate per tenant, filter and re-authorize at query time, encrypt and scrub embeddings as sensitive data, and treat every retrieved chunk as untrusted input.
