Passwordless Authentication - Login Without Shared Secrets

Category: Authentication
Common Approaches: Passkeys, magic links, one-time codes, device-bound keys, federated sign-in

Overview

Passwordless authentication is a design approach that removes passwords from the primary login path and relies on stronger or more usable alternatives. Implementations range from highly secure phishing-resistant passkeys to weaker email magic links or one-time codes. The main goals are reducing password theft, reuse, reset burden, and user friction. The security quality of a passwordless system depends entirely on what replaces the password and how recovery is handled.

How It Works

Core Mechanics

Passwordless systems usually rely on one of these models:

Model Example
Public-key auth Passkeys or security keys
Possession-only flow Email magic link or SMS code
Federated auth Enterprise IdP or social sign-in
Device-bound auth Managed device certificate or secure hardware credential

Core decision points:

Decision Why it matters
Enrollment Determines initial assurance and account binding
Channel trust Email or SMS may be weaker than platform authenticators
Recovery Often becomes the effective authentication system
Session creation Strong initial auth can still be undermined by weak session policy

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Weak passwordless factor Magic links or SMS replace password with another phishable method Account takeover High
2 Channel compromise Email inbox or phone number becomes the real secret Broad account loss High
3 Replay or reuse Links or codes are too long-lived or not one-time Unauthorized sign-in Med
4 Recovery bypass Recovery path is weaker than primary passwordless flow Strong auth bypass High
5 Device theft without strong local binding Possession factor is not gated by PIN or biometric Easy account access Med
6 Session overextension Long-lived or weakly bound sessions undo strong login Post-auth compromise Med
7 Enrollment fraud Attacker links their own device or email during setup Persistent takeover Med
8 Delivery failure Users cannot receive the factor reliably Unsafe support workarounds Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Weak passwordless factor Prefer passkeys or hardware-backed authenticators Auth design
2 Channel compromise Treat email and phone channels as medium assurance only Risk policy
3 Replay or reuse Make links and codes short-lived and one-time use Verifier logic
4 Recovery bypass Secure recovery with high-assurance identity proofing Recovery operations
5 Device theft without strong local binding Require local unlock and secure storage on device Device and authenticator policy
6 Session overextension Use secure session rotation and step-up for sensitive actions App security
7 Enrollment fraud Add alerts and verification at enrollment time Identity operations
8 Delivery failure Offer more than one strong factor path Product and support design

Top 3 Critical Mitigations

  1. Prefer phishing-resistant passwordless methods: Passwordless is only a security gain if the replacement method is stronger than the password it replaces. Passkeys and hardware-backed authenticators are the strongest mainstream choice.
  2. Treat recovery as part of authentication: Recovery determines the real assurance floor of the system. It must not silently collapse to low-assurance email-only or helpdesk-only proof.
  3. Constrain session lifetime and reuse: Strong login can be undone by weak session design. Session rotation, re-auth for sensitive actions, and good token hygiene remain necessary.

Passwordless Authentication — Model Answers

1. Why is passwordless not automatically phishing-resistant?

Passwordless simply means:

The user does not type a password.

It does not automatically mean the method is phishing-resistant.

Some passwordless methods still produce a credential that can be stolen, replayed, or abused.

Examples:

Passwordless Method Phishing-Resistant? Why
Magic link by email No / weak Link can be forwarded, stolen, or replayed
Email OTP No / weak User can type OTP into phishing site
SMS OTP No / weak Phishable and SIM-swap prone
Push approval Medium Vulnerable to push fatigue
Passkeys / FIDO2 Yes Origin-bound public-key challenge
Smart card / device certificate Strong if well configured Device-bound cryptographic proof

Example

A magic link login flow:

User enters email
System sends login link
User clicks link
User is logged in

An attacker can phish this:

Attacker tricks user into entering email on fake site
Fake site triggers real magic link
User forwards or exposes link
Attacker uses link

So the user did not use a password, but the system is still vulnerable.

Model answer

Passwordless is not automatically phishing-resistant because “no password” only removes one secret. If the replacement factor is a magic link, email OTP, SMS OTP, or simple push approval, it may still be phishable, replayable, or socially engineered. Phishing resistance usually requires origin-bound cryptographic authentication, such as passkeys, FIDO2 hardware keys, smart cards, or properly designed device certificates.

2. How would you compare magic links, passkeys, and device certificates for security and usability?

High-level comparison

Method Security Usability Main Risk Best Use
Magic links Low to medium High Email compromise, link replay Low-risk consumer apps
Passkeys High High Recovery/account sync assurance Consumer and enterprise login
Device certificates High Medium Device lifecycle and key protection Managed enterprise devices

Magic links

Magic links are simple.

User enters email
System sends login link
User clicks link
Session starts

Benefits:

Easy to use
No password to remember
Low support burden
Good for low-risk apps

Risks:

Email account becomes primary authenticator
Links can be forwarded
Links can be stolen from inbox
Links can be replayed if poorly designed
Malware/browser extensions can steal links
Weak if email recovery is weak

Magic links are convenient but not strong authentication unless combined with strong email security, device binding, short expiration, replay protection, and risk checks.

Passkeys

Passkeys use public-key cryptography.

Private key stays on device/authenticator
Public key is stored by service
Login challenge is signed
Credential is bound to website origin

Benefits:

Phishing-resistant
No password reuse
No shared secret sent to server
Good user experience with Face ID/fingerprint/PIN
Works well for both consumer and enterprise

Risks:

Recovery depends on platform account or backup authenticators
Synced passkeys depend partly on cloud keychain security
Some legacy devices/browsers may not support them well
User education needed

Device certificates

Device certificates prove possession of a managed device or private key.

Example:

Corporate laptop has certificate
Application requires mTLS or certificate-based auth
Only managed device can connect

Benefits:

Strong for managed enterprise fleets
Can bind access to compliant devices
Harder to phish than OTP/magic links
Good for zero-trust access
Useful for service-to-service and device identity

Risks:

Operational complexity
Certificate lifecycle management
Lost/stolen device risk
Private key storage quality matters
Less convenient for unmanaged/BYOD users
Not always enough to prove the human user

Model answer

Magic links are very usable but weaker because email becomes the primary factor and links can be stolen or replayed. Passkeys provide strong phishing resistance through origin-bound public-key authentication and good usability through biometrics or device PINs. Device certificates are strong in managed enterprise environments because they bind access to trusted devices, but they require certificate lifecycle management and usually need to be paired with user verification.

3. What trust assumptions change when email becomes the primary login factor?

When email becomes the primary login factor, the security of the application depends heavily on the security of the user’s email account.

The email account becomes the de facto identity provider.

New trust assumptions

You are assuming:

The user’s email inbox is secure
Only the real user can access email
Email recovery is strong
Email forwarding rules are not malicious
Mailbox sessions are not stolen
Email provider blocks phishing and malware
Magic links cannot be intercepted or replayed

That is a lot of trust.

Risks

If email is compromised, attacker can:

Receive login links
Search old magic links
Reset passwords for other services
Add forwarding rules
Hide security notifications
Use inbox access for account takeover

Example:

Attacker compromises Gmail
        ↓
Requests magic link for SaaS app
        ↓
Receives link
        ↓
Logs into SaaS app without knowing app password

Enterprise concern

For workforce apps, email-based passwordless can be risky if:

Mailbox sessions are long-lived
Email access does not require phishing-resistant MFA
Helpdesk can reset mailbox MFA weakly
Forwarding rules are not monitored
Legacy protocols are enabled

Model answer

When email becomes the primary login factor, the application shifts trust to the user’s mailbox, email provider, email session security, and email recovery process. If the email account is compromised, the attacker can receive login links and recover dependent accounts. Therefore, magic-link systems must treat email as a high-value authenticator and protect against link replay, forwarding abuse, mailbox compromise, and weak email recovery.

4. Why does recovery define the real assurance level of a passwordless system?

Passwordless login may be strong, but recovery can bypass it.

Example:

Login requires passkey
Recovery allows email-only reset

Then the real assurance is closer to email-only recovery, not passkey authentication.

Strong login, weak recovery

Normal login:
Passkey + device verification

Recovery:
Call helpdesk + answer date of birth

An attacker attacks the recovery path.

Attacker social-engineers helpdesk
        ↓
Passkey removed
        ↓
Attacker registers new authenticator
        ↓
Account takeover

Recovery must match account risk

For standard accounts:

Trusted device
Backup code
Email + risk checks
Passkey re-registration

For high-value accounts:

Hardware key backup
Manager/security approval
High-assurance proofing
Cooling-off period
Session revocation
Manual review

Model answer

Recovery defines the real assurance level because it is an alternate path into the account. If a passwordless system uses passkeys for login but weak email, SMS, or helpdesk recovery, attackers will bypass passkeys and attack recovery. Recovery must be designed with assurance equal to the risk of the account and should revoke old sessions and authenticators after completion.

5. How would you roll out passwordless to a large mixed-device workforce?

I would use a phased migration with segmentation, compatibility planning, and strong recovery design.

Phase 1: Inventory

Understand the environment:

Device types: Windows, macOS, iOS, Android, Linux
Managed vs unmanaged devices
Browser support
Legacy applications
Current MFA methods
Privileged users
Contractors/BYOD users
Remote users
Accessibility requirements
Helpdesk volume

Phase 2: Choose target methods by user group

User Group Recommended Passwordless Method
Privileged admins FIDO2 hardware keys / smart cards
Developers Passkeys or hardware keys
Standard employees Platform passkeys / Windows Hello / device-bound credentials
Mobile-heavy users Platform passkeys
Contractors/BYOD Passkeys + conditional access
Shared workstation users Smart card/FIDO2 key
Break-glass accounts Hardware-backed, offline-controlled credentials

Phase 3: Pilot

Start with:

Identity team
Security team
IT support
Small business unit
Privileged users

Measure:

Enrollment success
Login success
Device compatibility
Helpdesk tickets
Recovery failures
User feedback
Phishing resistance improvements

Phase 4: Integrate with conditional access

Policies:

Require passwordless for high-risk apps
Require managed/compliant device for sensitive apps
Require phishing-resistant factors for admin portals
Allow fallback only for approved exceptions
Block SMS for privileged accounts

Phase 5: Recovery and fallback

Before mass rollout, design recovery:

Backup authenticator registration
Temporary access pass
Helpdesk playbooks
Manager/security approval for high-risk accounts
Lost device process
Session revocation after recovery

Phase 6: Gradual enforcement

Rollout pattern:

Enable optional passwordless
Encourage enrollment
Require for admins
Require for high-risk users
Require for all users
Remove weak fallback methods

Model answer

I would roll out passwordless in phases: inventory devices and apps, segment users by risk, pilot with IT/security, choose appropriate methods per population, integrate conditional access, prepare recovery workflows, train users/helpdesk, then gradually enforce. Privileged users should move first to phishing-resistant hardware-backed methods, while standard users can use platform passkeys or device-bound credentials. Weak fallbacks like SMS should be removed after adoption.

6. What are the risks of possession-only authentication without local device verification?

Possession-only means the system verifies that the user has a device or token, but does not verify that the correct human is using it.

Examples:

Clicking a magic link on a device
Possessing a phone with an unlocked email session
Using a security key that requires touch but no PIN
Device certificate without user login verification

Risks

If the device is stolen or already unlocked:

Attacker has device
        ↓
Authentication succeeds

If malware is running on the device:

Malware uses existing device credential
        ↓
No local user verification is required

If a hardware key requires only touch:

Attacker with physical key may use it if no PIN/user verification is required

Why local verification matters

Local verification adds proof that the legitimate user is present.

Examples:

Device PIN
Fingerprint
Face ID
Windows Hello
Security key PIN
Biometric verification

Better flow:

Possess device/key
        +
Unlock locally with PIN/biometric
        ↓
Authentication succeeds

Model answer

Possession-only authentication is risky because possession of the device or token may not prove the legitimate user is present. A stolen unlocked phone, compromised endpoint, or security key without PIN/user verification can be abused. For sensitive actions and privileged accounts, possession should be combined with local user verification such as PIN, biometric, or hardware-key PIN.

7. How should passwordless interact with step-up authentication?

Passwordless should not mean “authenticate once and trust forever.”

Step-up authentication is still needed when risk increases or the action is sensitive.

When to require step-up

Registering new authenticator
Removing authenticator
Changing recovery email/phone
Creating API token
Viewing secrets
Changing payout account
Approving high-value transaction
Accessing admin console
Disabling security controls
Exporting sensitive data
Logging in from new device or risky location

Step-up strength should depend on risk

Current Login Sensitive Action Step-Up Needed
Magic link Change MFA Passkey/FIDO2 or stronger proof
Synced passkey Admin role Hardware key or device-bound credential
Device certificate Payment approval User verification + transaction confirmation
Hardware key Delete production resource Fresh hardware key touch/PIN

Important concept

Step-up should verify:

Fresh user presence
Fresh user verification
Correct device
Correct authorization
Risk context

Example

A user logs in with a passkey at 9 AM.

At 4 PM, they try to add a new bank account.

Good policy:

Require fresh passkey verification
Notify user
Delay payout use for high-risk cases

Model answer

Passwordless should be integrated with step-up authentication for sensitive actions and risky contexts. Even if login is passwordless, actions like changing authenticators, creating API tokens, accessing admin consoles, or changing payout details should require fresh authentication with a strong factor. Step-up should match risk; privileged actions should require phishing-resistant and locally verified authentication.

Detection examples

Magic link clicked from different ASN than requested
New passkey added after suspicious recovery
Recovery followed by all sessions not revoked
Device certificate used after device marked non-compliant
Passwordless login followed by mass data export
Many magic links requested for many users from same IP

Model answer

For passwordless abuse, I would collect magic-link request and click telemetry, passkey registration and authenticator metadata, device certificate usage, recovery events, device/IP reputation, session creation, and post-login sensitive actions. The most valuable detections correlate events, such as magic link requested from one IP and redeemed from another, new authenticator added after recovery, or passwordless login followed by token creation or data export.

9. When is federated sign-in effectively passwordless for the relying app but not for the ecosystem?

Federated sign-in can be passwordless from the relying application’s perspective because the app never sees or verifies a password.

Example:

User opens SaaS app
SaaS redirects to corporate IdP
User authenticates at IdP
IdP sends SAML/OIDC assertion
SaaS grants access

The relying app sees:

Signed assertion/token from trusted IdP

It does not know whether the user used:

Password
Password + SMS
Passkey
Hardware key
Smart card
Existing IdP session

So for the app, it feels passwordless.

But for the ecosystem, it is only truly passwordless if the IdP authentication is passwordless.

Example

SaaS app:

No password stored
Only SSO login

Corporate IdP:

User signs in with password + SMS OTP

This is not truly passwordless for the ecosystem.

Why this matters

The relying app inherits the IdP’s assurance.

If the IdP has weak authentication or recovery, the app is affected.

What to review

For federated passwordless claims, check:

Authentication method reference
MFA strength
IdP session lifetime
Step-up support
Token audience and lifetime
Recovery policy
Admin MFA policy
Conditional access
Phishing resistance

In OIDC/SAML environments, the relying party can sometimes request or inspect assurance signals such as authentication context.

Model answer

Federated sign-in is effectively passwordless for the relying application when the app delegates authentication to an IdP and never handles user passwords. But the broader ecosystem is only passwordless if the IdP itself uses passwordless authentication. If the IdP still uses passwords, SMS, or weak recovery, then passwordless is only true at the app boundary, not at the identity ecosystem level.

10. What would you review after a large-scale magic-link replay incident?

A magic-link replay incident means login links were reused, stolen, forwarded, scanned, or redeemed by attackers.

I would review the full lifecycle:

Request → delivery → click → redemption → session creation → post-login actions

A. Link design

Review:

Was the link single-use?
Was it short-lived?
Was it bound to the requesting browser/session?
Was it bound to device/IP risk context?
Was token stored hashed server-side?
Was token invalidated after redemption?
Could the same link be clicked multiple times?
Could email security scanners accidentally activate it?

Strong design:

Single-use
Short expiration
Server-side hashed token
Redeem requires same browser/session or secondary challenge
Replay attempt invalidates token

B. Request and redemption mismatch

Check:

Requester IP vs redeemer IP
Requester user agent vs redeemer user agent
Geography mismatch
ASN mismatch
Time-to-click anomaly
Multiple clicks
Multiple accounts from same redeemer IP

Example suspicious pattern:

Magic link requested from Singapore
Redeemed 20 seconds later from another country

C. Email channel compromise

Review:

Mailbox compromise indicators
Forwarding rules
OAuth grants to mailbox
Legacy email protocols
Email session theft
Email security gateway behavior
Links exposed in logs
Links visible in notification previews

D. Application logging leaks

Search for magic links in:

Application logs
Reverse proxy logs
Email provider logs
Analytics tools
Crash reports
Browser referrer headers
Support tickets
SIEM events

Magic links should not appear in logs.

E. Session handling after redemption

Review:

Were sessions created immediately?
Was step-up required for risky redemption?
Were old sessions revoked after recovery-style login?
Were session cookies secure, HttpOnly, SameSite?
Were refresh tokens rotated?

F. Post-login abuse

Look for:

Passwordless login followed by email change
New MFA/passkey added
API token created
OAuth app authorized
Data exported
Payment details changed
Admin actions

G. Immediate remediation

Actions:

Invalidate outstanding magic links
Revoke suspicious sessions
Force re-authentication for affected users
Disable magic links temporarily if needed
Shorten link TTL
Make links single-use
Bind link redemption to original browser/session
Add risk-based step-up
Notify affected users
Hunt for mailbox compromise

H. Longer-term redesign

Consider moving from magic links to:

Passkeys
Device-bound passwordless
Email link + trusted device confirmation
Email link + step-up for high-risk accounts
Federated login with phishing-resistant MFA

Model answer

After a magic-link replay incident, I would review whether links were single-use, short-lived, hashed server-side, invalidated after redemption, and bound to the requesting session or device. I would compare request and redemption IP/device, look for multiple clicks, email compromise, forwarding rules, logging leaks, and scanner behavior. I would revoke suspicious sessions, invalidate outstanding links, add risk-based step-up, and redesign toward passkeys or device-bound passwordless for higher-risk users.

Key Takeaways

  • Passwordless is a category of login designs, not a single security level.
  • Replacing passwords with weak channels such as email links can improve UX more than security.
  • Passkeys and strong device-bound authenticators are the highest-value passwordless options.
  • Recovery, enrollment, and session controls remain central.
  • The best passwordless systems reduce both phishing exposure and operational reset burden.
