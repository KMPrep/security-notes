Passwords - Shared Secret Authentication

Category: Authentication
Common Standards And Guidance: NIST SP 800-63B, OWASP ASVS, OWASP Authentication Cheat Sheet

Overview

Passwords are the most common knowledge-based authentication factor and remain the default primary credential in most systems. A password proves control of a shared secret known by the user and verifier, usually through server-side hash verification rather than plaintext comparison. Passwords are simple to deploy and understand, but they are highly vulnerable to reuse, phishing, guessing, stuffing, and weak recovery flows. Security depends less on the secret alone and more on storage, rate limiting, enrollment, reset, and complementary controls such as MFA.

How It Works

Core Mechanics

The basic password lifecycle involves:

Stage Purpose
Enrollment User chooses or receives an initial secret
Storage Server stores a slow salted password hash
Verification Submitted password is hashed and compared
Rotation or reset Secret is changed when needed
Recovery Access is restored if the user loses the password

Key security properties:

Property Why it matters
Salting Prevents precomputed rainbow table reuse
Slow hashing Raises attacker cost for offline cracking
Rate limiting Reduces online guessing speed
Breach monitoring Detects reused or compromised passwords
MFA pairing Reduces single-factor compromise impact

Step-by-Step Flow

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Credential stuffing Screen against breached passwords and require MFA Auth platform
2 Offline cracking Use Argon2id, bcrypt, or scrypt with strong parameters Password storage layer
3 Brute-force login Apply rate limits, IP heuristics, and risk checks Sign-in endpoint
4 Phishing Add phishing-resistant MFA or move toward passkeys Identity architecture
5 Weak reset flow Secure recovery with strong identity proofing Recovery workflow
6 Poor secret hygiene Keep passwords out of logs and support secret scanning Dev and ops tooling
7 Overly strict composition rules Prefer length and blocklists over complex forced patterns Password policy
8 Password spraying Monitor distributed failures and protect high-value accounts Detection and response

Top 3 Critical Mitigations

  1. Use modern slow password hashing: Fast hashes turn a database breach into a fast cracking exercise. Password verifiers should use memory-hard or deliberately slow algorithms with current cost parameters.
  2. Harden reset and recovery: Attackers often bypass login hardening by abusing the reset path. Recovery should have equal or higher assurance than the normal sign-in flow.
  3. Pair passwords with stronger factors: Passwords alone remain highly phishable and reusable. MFA or passkey migration materially reduces takeover risk.

Password Security Model Answers — Deep Explanation with Examples

1. Why should password verifiers store salted slow hashes instead of encrypted passwords?

A password verifier should store something that lets the system verify a password without needing to recover the original password.

That is why systems should store:

salted slow password hash

not:

encrypted password

Encrypted password problem

Encryption is reversible.

password + encryption key → ciphertext
ciphertext + decryption key → original password

If attackers steal the database and the encryption key, they can recover every password.

Database leak + encryption key leak
        ↓
All passwords decrypted
        ↓
Users compromised everywhere they reused passwords

This creates a high-value key management problem.

Salted slow hash model

Hashing is one-way.

password + salt → slow password hash

To verify login:

User enters password
        ↓
System applies same salt + hash function
        ↓
Compares result with stored hash

The system never needs to know the original password.

Why salt matters

A salt is a random value generated per password.

It prevents attackers from using precomputed lookup tables and ensures two users with the same password have different stored hashes.

Example:

User A password: Password123
Salt A: random1
Hash A: hash(Password123 + random1)

User B password: Password123
Salt B: random2
Hash B: hash(Password123 + random2)

Stored hashes are different even though the password is the same.

Why slow hashing matters

Normal hashes like SHA-256 are too fast.

Attackers with GPUs can try huge numbers of guesses quickly.

Password hashing algorithms are intentionally expensive:

Argon2id
bcrypt
scrypt
PBKDF2

They slow down offline cracking after a database leak.

Model answer

Password verifiers should store salted slow hashes because the service only needs to verify the password, not recover it. Encrypted passwords are reversible, so compromise of the database and encryption key exposes all original passwords. Salted slow hashes make each password unique and make offline cracking expensive. The goal is to reduce damage after a verifier database breach.

2. What is the operational difference between credential stuffing and brute-force guessing?

Both involve login attempts, but the attacker’s input source and detection patterns differ.

Credential stuffing

Credential stuffing uses known leaked username/password pairs from previous breaches.

Example:

Attacker has:
alice@example.com : Summer2021!
bob@example.com   : Liverpool123

They try those credentials against your service.

Why it works:

Users reuse passwords across sites.

Operational pattern:

Many accounts
Few attempts per account
High success rate compared with random guessing
Distributed IPs/proxies
Usernames are real
Passwords are from breach data

Brute-force guessing

Brute force means guessing passwords without necessarily having valid leaked pairs.

Example:

Try many passwords against one account
or
Try common passwords across many accounts

Subtypes:

Online brute force: many guesses against login endpoint
Password spraying: few common passwords across many accounts
Offline brute force: cracking stolen password hashes

Operational pattern:

Many attempts
Often low success rate
May target one account or many accounts
May trigger lockouts

Difference table

Area Credential Stuffing Brute Force
Input Leaked credentials Guessed passwords
Targeting Many real accounts One or many accounts
Attempts per account Usually low Can be high
Success reason Password reuse Weak passwords
Detection Reuse patterns, breached creds, bot traffic Attempt volume, guessing patterns
Defense MFA, breached-password detection, bot mitigation Rate limits, lockouts, MFA, password policy

Model answer

Credential stuffing uses leaked username/password pairs from other breaches and relies on password reuse. Brute-force guessing tries to guess passwords directly. Operationally, credential stuffing often has low attempts per account across many accounts and uses distributed infrastructure, while brute force may show many guesses against one account or common passwords sprayed across many accounts. The controls overlap, but credential stuffing especially needs MFA, breached-password checks, bot detection, and anomaly detection.

3. Why do composition rules often fail to improve real password security?

Composition rules are rules like:

Must include uppercase
Must include lowercase
Must include number
Must include symbol
Must be changed every 90 days

They often fail because users behave predictably.

User behavior problem

If forced to include a capital letter and number, users often do this:

password → Password1!
summer   → Summer2024!
company  → Company@123

Attackers know these patterns.

So the search space does not improve as much as policy designers think.

Rotation problem

Periodic rotation often causes predictable changes:

Winter2024!
Spring2024!
Summer2024!
Autumn2024!

Users may also write passwords down or reuse simple variants.

Complexity vs length

A short complex password may be weaker than a long memorable passphrase.

Weak-ish: P@ssw0rd1!
Stronger: correct horse battery river mango

Length and unpredictability usually matter more than arbitrary complexity.

Better password policy

Better controls:

Minimum length
Allow long passphrases
Block breached passwords
Block common passwords
Block context-specific passwords
Use MFA
Use rate limiting
Use password managers
Do not force periodic rotation unless compromise is suspected

Model answer

Composition rules often fail because users satisfy them in predictable ways, such as capitalizing the first letter, adding a number at the end, or using seasonal rotations. These patterns are built into modern cracking tools. A better approach is to require sufficient length, allow passphrases, block breached/common passwords, use MFA, and avoid forced rotation unless there is evidence of compromise.

4. How would you choose between Argon2id, bcrypt, and scrypt in a new system?

For a new system, I would generally prefer Argon2id if the platform supports it well.

Argon2id

Argon2id is designed to be memory-hard and resistant to GPU/ASIC cracking.

Strengths:

Modern password hashing winner
Memory-hard
Good protection against GPU cracking
Tunable memory, time, and parallelism
Hybrid design balances side-channel and GPU resistance

Best for:

New systems
Security-sensitive applications
Systems where modern library support exists

bcrypt

bcrypt is older but still widely used and battle-tested.

Strengths:

Mature
Widely supported
Easy to operate
Adaptive cost factor

Limitations:

Not memory-hard like Argon2id/scrypt
Password length handling can be tricky in some implementations
Less resistant to specialized cracking than Argon2id

Best for:

Legacy compatibility
Platforms where Argon2id is not available
Systems needing proven operational simplicity

scrypt

scrypt is also memory-hard.

Strengths:

Memory-hard
Good GPU resistance
Mature enough in many libraries

Limitations:

Parameter tuning can be more complex
Less commonly recommended than Argon2id for new password storage

Best for:

Systems where scrypt is already standardized
Platforms with good scrypt implementation

Parameter selection

Regardless of algorithm:

Use unique random salt per password
Tune cost to your latency budget
Use strong library implementation
Store algorithm and parameters with hash
Plan for future rehashing
Use pepper only if you can protect it well

Example stored verifier format:

algorithm = argon2id
memory = 64 MiB
iterations = 3
parallelism = 1
salt = random
hash = output

Model answer

For a new system, I would choose Argon2id if supported because it is modern, memory-hard, and designed to resist GPU cracking. I would use bcrypt when I need broad compatibility or mature legacy support. I would use scrypt where memory hardness is desired and the platform has good scrypt support. The exact parameters should be tuned to the service’s latency and capacity, and the system should support rehashing over time as hardware improves.

5. Why is recovery usually the weakest part of a password-based system?

Recovery is often weaker because it bypasses the normal password check.

Normal login:

User knows password
User passes MFA
Login succeeds

Recovery path:

Forgot password
Click email link
Reset password
Login succeeds

If recovery is weak, the attacker does not need the password.

Common weak recovery mechanisms

Email-only reset
SMS OTP reset
Security questions
Helpdesk reset based on public information
Weak identity proofing
Long-lived reset links
Reusable reset tokens
No session revocation after reset

Example attack

Attacker compromises user's email
        ↓
Requests password reset
        ↓
Receives reset link
        ↓
Changes password
        ↓
Takes over account

Or enterprise version:

Attacker social-engineers helpdesk
        ↓
Helpdesk resets password
        ↓
Attacker enrolls MFA
        ↓
Account takeover

Strong recovery controls

Short-lived single-use reset tokens
Risk-based recovery checks
Notify previous trusted channels
Require MFA where possible
Revoke sessions after reset
Protect helpdesk workflows
Use stronger proofing for admins
Delay sensitive actions after recovery
Monitor recovery abuse

Model answer

Recovery is usually the weakest part because it is an alternate path that can bypass the password and sometimes MFA. If recovery relies on email, SMS, security questions, or weak helpdesk proofing, attackers target recovery instead of guessing the password. Recovery assurance must match account risk, and successful recovery should revoke old sessions and credentials.

6. How would you detect password spraying at enterprise scale?

Password spraying is when attackers try a small number of common passwords against many accounts.

Example:

Try "Spring2026!" against 10,000 users.
Then try "Welcome2026!" later.

This avoids per-account lockout because each account gets only one or two attempts.

Detection signals

Look for:

Many usernames targeted
Same password or pattern attempted
Low attempts per account
High number of failures from same IP/ASN/device
Failures across many accounts in short time
Attempts just below lockout threshold
Attempts against disabled/dormant accounts
Attempts from unusual geographies
Attempts against admin accounts

Enterprise-scale correlation

You need to aggregate by:

Source IP
ASN
User agent
Device fingerprint
Target tenant/domain
Time window
Failure reason
Username count
Geo location
Authentication protocol

Useful detection logic

If one source IP attempts login to 100+ users in 10 minutes
AND failure rate is high
AND attempts per user are low
THEN likely password spraying

Another:

If many accounts receive exactly one failed attempt
AND attempts use legacy protocol
AND source ASN is unusual
THEN likely spray

Important: legacy protocols

Password spraying often targets protocols that do not support modern MFA well.

Examples:

IMAP
POP3
SMTP AUTH
Basic authentication
Legacy VPN
Old federation endpoints

Response

Block source IP/ASN if appropriate
Apply risk-based MFA
Disable legacy authentication
Force password reset only for successful compromised accounts
Notify SOC
Hunt for successful logins after failed spray
Check mailbox rules/OAuth grants
Increase monitoring for targeted users

Model answer

I would detect password spraying by correlating failed login attempts across many accounts, especially where each account has only one or two failures but the source, ASN, user agent, or protocol is common. I would pay special attention to legacy protocols, dormant accounts, admin accounts, and successes after broad failures. Response would include blocking sources, disabling legacy auth, enforcing MFA, and investigating any successful logins from the spray campaign.

7. What are the tradeoffs between aggressive lockout and availability?

Aggressive lockout can reduce online guessing but can also create denial-of-service risk.

Benefit of lockout

Attacker cannot try unlimited guesses against one account.

Example:

Lock account after 5 failed attempts.

This protects against targeted brute force.

Availability risk

Attackers can intentionally lock out many users.

Attacker tries wrong password 5 times for every employee
        ↓
Thousands of accounts locked
        ↓
Business disruption
        ↓
Helpdesk overwhelmed

This is especially dangerous for:

Hospitals
Call centers
Trading systems
Critical operations
Emergency access accounts

Better alternatives

Instead of hard lockout only, use adaptive controls:

Progressive delays
Risk-based challenges
Temporary soft lock by source
Device/IP rate limits
MFA step-up
Bot detection
Credential stuffing detection
User notification
Protect against username enumeration

Lockout design by account type

Account Type Lockout Strategy
Normal users Soft lock / progressive delay
Admins Stricter monitoring + phishing-resistant MFA
Service accounts Do not use interactive login; alert on attempts
Break-glass Strong controls, avoid easy lockout DoS
High-risk attempts Source throttling before account lockout

Model answer

Aggressive lockout reduces brute-force guessing but creates denial-of-service risk because attackers can lock many users out intentionally. A better design uses rate limiting, progressive delays, risk-based challenges, source throttling, MFA, and targeted lockouts. Lockout policy should differ for normal users, privileged users, service accounts, and break-glass accounts.

8. How would you migrate a large system away from password-only auth?

I would migrate in phases, with risk-based prioritization.

Phase 1: Inventory and risk segmentation

Identify:

All login surfaces
User populations
Privileged users
APIs and service accounts
Legacy clients
Recovery flows
MFA capability
Apps that cannot support modern auth

Segment:

Admins
Employees
Customers
Contractors
Developers
Service accounts
High-risk users

Phase 2: Strengthen passwords immediately

Before migration completes:

Block breached passwords
Increase minimum length
Remove bad composition rules
Rate-limit login
Detect stuffing/spraying
Disable legacy authentication
Improve password reset security

Phase 3: Add MFA

Rollout order:

Admins first
High-risk users
Employees
Customers
Remaining users

Prefer stronger factors:

Passkeys
FIDO2 hardware keys
Number matching push
TOTP as fallback
Avoid SMS except temporary exception

Phase 4: Move to passwordless where possible

Target:

Passkeys for consumers
FIDO2/security keys for admins
Device-bound credentials for managed workforce
Federated SSO for enterprise apps

Phase 5: Reduce password dependency

Make passwordless default
Disable password login for enrolled users
Keep recovery strong
Monitor fallback usage
Remove SMS fallback
Retire legacy auth

Phase 6: Governance and monitoring

Track:

MFA enrollment
Passwordless enrollment
Failed login rates
Helpdesk volume
Recovery abuse
Account takeover rate
Fallback usage
Legacy protocol usage
Admin compliance

Model answer

I would migrate away from password-only auth by first inventorying login surfaces and segmenting users by risk. I would immediately improve password security with breached-password blocking, rate limiting, and legacy-auth removal. Then I would roll out MFA starting with admins and high-risk users, prefer phishing-resistant factors, introduce passwordless such as passkeys, and gradually disable password-only login and weak fallback methods. Recovery must be upgraded in parallel.

9. Why are password breach blocklists more useful than forcing periodic rotation?

Breach blocklists prevent users from choosing passwords that attackers already know.

Periodic rotation often makes users choose predictable variants.

Periodic rotation problem

Example:

Password before: Summer2025!
Password after: Autumn2025!
Next: Winter2025!

Attackers know this pattern.

Forced rotation can also cause:

Weaker passwords
Password reuse
More helpdesk calls
Users writing passwords down
Predictable suffix changes

Breach blocklist benefit

A breach blocklist blocks passwords known to be compromised or commonly used.

Examples:

Password123
Company@123
Welcome2026!
Summer2024!
qwerty123

This directly prevents high-risk password choices.

Better policy

Check password against breached-password corpus
Require reasonable length
Allow long passphrases
Do not force arbitrary complexity
Do not force periodic rotation without compromise
Require reset after suspected breach
Use MFA/passkeys

When rotation is useful

Rotation is useful when:

Password is suspected compromised
Credential appears in breach
User fell for phishing
Admin password exposed
Hash database leaked
Privileged access policy requires emergency reset

Not simply because 90 days passed.

Model answer

Breach blocklists are more useful because they stop users from choosing passwords attackers already have or commonly guess. Periodic rotation often causes predictable password changes and user fatigue without meaningfully improving security. I would require password changes after suspected compromise, but otherwise focus on breached-password blocking, length, MFA, rate limiting, and passwordless migration.

10. What would you review first after a leaked password-hash incident?

After a leaked password-hash incident, the first priority is to determine whether the hashes are crackable and what immediate containment is needed.

First review: hashing scheme

Ask:

Which algorithm was used?
Were passwords salted?
Was the salt unique per user?
Was the algorithm slow and memory-hard?
What parameters/cost factor were used?
Was there a pepper?
Was the pepper compromised?
Were any passwords encrypted instead of hashed?

Severity differs greatly:

Argon2id with strong parameters + unique salts → lower immediate crack risk
Unsalted MD5/SHA1 → severe emergency
bcrypt low cost → significant risk
Encrypted passwords + leaked key → critical

Second review: exposed data

Determine:

Which users were affected?
Were admins included?
Were password reset tokens exposed?
Were MFA seeds exposed?
Were recovery codes exposed?
Were emails/usernames exposed?
Was source code/config leaked?
Was pepper/KMS key exposed?

Third review: signs of cracking/use

Look for:

Credential stuffing against your service
Login attempts from new sources
Successful logins after leak
Password reuse against other systems
Admin login anomalies
OAuth grants/API tokens created
Recovery changes

Immediate response

Depending on hash strength:

Force password reset for affected users
Prioritize admins/high-risk users
Revoke sessions and refresh tokens
Rotate exposed peppers/secrets
Disable weak legacy auth
Increase monitoring
Notify users/regulators if required
Block known cracked passwords

If hashes were weak

Emergency path:

Invalidate all affected passwords
Force reset
Revoke sessions
Require MFA step-up
Hunt for account takeover
Monitor credential stuffing

If hashes were strong

Still respond:

Notify and monitor
Force reset for privileged/high-risk accounts
Opportunistically rehash on next login
Consider staged password reset based on risk

Long-term fixes

Upgrade to Argon2id/bcrypt/scrypt
Tune parameters
Add breach password checks
Implement MFA/passkeys
Improve database access controls
Reduce data exposure
Improve secrets management
Add detection for hash export
Review backup/log access

Model answer

First, I would review the password hashing scheme: algorithm, unique salts, cost parameters, memory hardness, and whether any pepper or encryption key was compromised. That determines how fast attackers can crack the hashes. Then I would identify affected users, especially admins, check whether MFA seeds/recovery codes/tokens were exposed, revoke sessions if needed, force resets based on risk, and monitor for credential stuffing or account takeover. Long term, I would upgrade hashing, add breached-password checks, enforce MFA/passkeys, and improve data access controls.

Key Takeaways

  • Passwords are still common but are a weak primary factor on their own.
  • Storage, rate limiting, reset, and phishing resistance matter more than composition trivia.
  • Slow salted hashing is mandatory for breach resilience.
  • Recovery paths need equal or stronger protections than sign-in.
  • Passwords should be paired with MFA or gradually replaced with stronger authenticators.
