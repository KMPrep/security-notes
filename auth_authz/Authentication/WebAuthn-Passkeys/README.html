WebAuthn And Passkeys - Phishing-Resistant Authentication

Category: Authentication
Primary Standards: WebAuthn, CTAP, FIDO2

Overview

WebAuthn is a web authentication standard that uses public-key cryptography and authenticators such as platform secure enclaves or hardware security keys. Passkeys are synchronized or device-bound credentials built on the same model and designed to replace passwords with a phishing-resistant user experience. Instead of sharing a secret with the server, the authenticator creates a key pair and the server stores only the public key. This removes whole classes of password theft and drastically improves resistance to phishing and credential reuse.

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Weak fallback path Remove or harden fallback authentication paths Identity architecture
2 Recovery abuse Apply strong identity proofing and audit to recovery Recovery workflow
3 Mis-scoped origins Validate RP ID and origin strictly WebAuthn verifier
4 Over-trusting attestation Treat attestation as optional assurance input, not blanket trust Registration policy
5 Sync trust misunderstanding Document whether passkeys are synced or device-bound Threat model and product design
6 Poor device lifecycle handling Provide revocation, device inventory, and alerts Identity operations
7 Usability failures Offer multiple phishing-resistant authenticators per user Enrollment design
8 Session over-trust Pair strong auth with secure session management and step-up policy App security layer

Top 3 Critical Mitigations

  1. Harden fallback and recovery: The strongest authenticator is irrelevant if support or fallback flows let an attacker reset the account with weak proof.
  2. Validate RP ID and origin correctly: WebAuthn security depends on origin binding. Implementation errors here can neutralize the phishing resistance you are trying to gain.
  3. Support multiple strong authenticators: Users need resilience for device loss. Backup hardware keys or additional registered authenticators reduce pressure to create weak recovery exceptions.

Passkeys / WebAuthn Model Answers

1. Why are passkeys considered phishing-resistant while passwords plus OTP are not?

Passkeys are phishing-resistant because they do not rely on a reusable secret that the user can type into a fake website.

With password + OTP:

User visits fake login page
        ↓
User types password
        ↓
User types OTP
        ↓
Attacker relays both to real site
        ↓
Login succeeds

The password and OTP are both portable secrets. If the user can type them, the attacker can capture and replay them.

With passkeys:

User visits website
        ↓
Website sends cryptographic challenge
        ↓
Authenticator signs challenge with private key
        ↓
Website verifies signature with public key

The private key stays on the user’s authenticator. The website stores only a public key. FIDO describes passkeys as cryptographic key-pair credentials designed for phishing-resistant sign-in, and WebAuthn credentials are scoped to a relying party so they can only be used by origins belonging to that relying party. (FIDO Alliance)

The critical difference:

Password + OTP = user reveals reusable values
Passkey = device proves possession without revealing the secret

Model answer

Passkeys are phishing-resistant because the user never types a reusable secret into the website. The authenticator signs an origin-bound challenge using a private key that does not leave the device. A fake website cannot use a passkey registered for the real website because the WebAuthn credential is scoped to the relying party. Passwords and OTPs are not phishing-resistant because attackers can capture or relay them in real time.

2. What is the difference between WebAuthn registration and authentication ceremonies?

WebAuthn has two main ceremonies:

Registration = create and register a new credential
Authentication = use an existing credential to sign in

The WebAuthn specification explicitly treats registration and authentication as “ceremonies,” meaning they involve both protocol messages and human/user interaction. (W3C)

Registration ceremony

Registration happens when the user adds a passkey or security key to an account.

Flow:

User signs in or proves identity
        ↓
Relying party asks browser to create credential
        ↓
Authenticator creates public/private key pair
        ↓
Private key stays in authenticator
        ↓
Public key and credential ID go to server
        ↓
Server stores public key for future logins

The server stores:

User ID
Credential ID
Public key
Relying party ID
Authenticator metadata if available
Signature counter / backup status if available

Authentication ceremony

Authentication happens when the user signs in using an existing passkey.

Flow:

User starts login
        ↓
Server sends random challenge
        ↓
Browser asks authenticator to sign challenge
        ↓
User performs biometric/PIN/touch
        ↓
Authenticator signs challenge
        ↓
Server verifies signature using stored public key
        ↓
Login succeeds

Key difference

Area Registration Authentication
Purpose Create credential Use credential
Key pair Generated Reused
Server receives Public key + credential ID Signed challenge
Private key Created and kept private Used internally to sign
Risk Fake enrollment, weak authenticator, account takeover during setup Replay, stolen device, weak session handling

Model answer

Registration is the ceremony where the authenticator creates a new key pair for the relying party and the server stores the public key and credential ID. Authentication is the ceremony where the server sends a challenge and the authenticator signs it with the private key. Registration establishes trust; authentication proves possession of the previously registered private key.

3. Why does origin binding matter more than just storing a public key?

Storing a public key proves only that the server can verify signatures from a matching private key.

But phishing resistance requires more than public-key cryptography. It requires that the credential cannot be used on the wrong website.

That is where origin binding and relying party scoping matter.

Example:

Real site:
login.company.com

Phishing site:
login-company.example.com

If authentication were only “sign this challenge with the private key,” a fake website could ask the authenticator to sign something and relay it to the real site.

WebAuthn prevents this by binding credentials to a relying party. The W3C WebAuthn specification says public key credentials are scoped to a WebAuthn Relying Party, and after creation, the credential can only be accessed by origins belonging to that relying party. (W3C)

Why this matters

A passkey for:

company.com

should not work for:

company-login.example.net

Even if the phishing page looks identical.

Model answer

A public key alone proves that some private key signed the challenge. Origin binding ensures that the credential can only be used for the legitimate relying party. This is what makes WebAuthn phishing-resistant. Without origin binding, attackers could build fake sites that trick users into signing challenges and relay them to the real service.

4. What trust assumptions change when passkeys are synced across devices?

Synced passkeys improve usability and recovery because a user can create a passkey on one device and use it on another device through a cloud keychain or password manager.

Examples:

Apple iCloud Keychain
Google Password Manager
Microsoft account
1Password
Bitwarden
Dashlane

FIDO notes that passkeys can be either synced across devices or bound to a particular device. (FIDO Alliance)

What changes

With device-bound hardware keys:

Private key stays on one physical authenticator
Recovery requires backup authenticator or formal recovery

With synced passkeys:

Private key material is protected and synchronized through a platform/cloud account
Recovery depends partly on that ecosystem

New trust assumptions

You now trust:

Cloud keychain security
Platform account recovery
Device enrollment process
Password manager security
Account recovery process of Apple/Google/Microsoft/password manager
Sync encryption model
User’s other devices

Security impact

Synced passkeys are still phishing-resistant for login, but their assurance level depends on how hard it is for an attacker to add a device or recover the platform account.

Example risk:

Enterprise app requires passkey
User’s passkey syncs through personal cloud account
Attacker compromises cloud account recovery
Attacker gains access to synced passkey

For standard users, synced passkeys may be a strong usability/security improvement. For privileged admins, many enterprises prefer hardware-bound or managed device-bound authenticators.

Model answer

Synced passkeys remain phishing-resistant at the login ceremony, but the trust boundary expands. The relying party is now also depending on the cloud keychain or password manager, its account recovery process, device enrollment controls, and sync protection. For normal users this may be acceptable and highly usable. For privileged users, enterprises may require device-bound or hardware-backed authenticators to reduce dependence on consumer sync recovery.

5. How should an enterprise think about attestation and authenticator policy?

Attestation helps a relying party understand what kind of authenticator created the credential.

During FIDO registration, the authenticator creates a unique key pair for the relying party; attestation can provide extra information about the authenticator model or provenance. (FIDO Alliance)

Why enterprises care

A consumer app may only care that the user has a valid passkey.

An enterprise may need to enforce:

Only company-approved security keys
Only hardware-backed authenticators
Only FIPS-certified devices
Only managed platform authenticators
No unknown personal authenticators for admins
User verification required
Resident/discoverable credentials allowed or disallowed

Microsoft Entra ID, for example, supports enforcing attestation for FIDO2 security keys in authentication method policy. (Microsoft Learn)

Policy examples

For standard workforce users:

Allow platform passkeys
Allow synced passkeys if risk accepted
Require user verification
Block weak fallback

For privileged admins:

Require hardware security keys
Require attestation
Require PIN/user verification
Require at least two keys
Disallow SMS/email recovery
Monitor all registration events

For regulated environments:

Allow only certified hardware authenticators
Require enterprise attestation
Inventory keys
Tie authenticators to asset/user lifecycle

Privacy tradeoff

Attestation can create privacy and interoperability concerns because it reveals information about the authenticator. For consumer services, broad attestation requirements can reduce adoption. For enterprises, stronger authenticator governance may justify that tradeoff.

Model answer

Enterprises should use authenticator policy to match account risk. For normal users, broad passkey support may maximize adoption. For privileged users, the organization may require attested, hardware-backed, phishing-resistant authenticators with user verification. Attestation helps verify authenticator type and provenance, but it introduces privacy, compatibility, and operational tradeoffs, so it should be applied mainly where assurance requirements justify it.

6. Why is fallback design the hardest part of a passkey rollout?

Fallback is hard because it can silently destroy the security benefit of passkeys.

Example:

Normal login:
Passkey

Fallback:
SMS OTP or helpdesk reset

In this case, attackers do not need to break passkeys. They attack the fallback.

Common fallback paths

Password fallback
SMS OTP fallback
Email magic-link fallback
Helpdesk recovery
Backup codes
Device replacement workflow
Temporary access pass

Why fallback is difficult

You must balance:

User lockout risk
Device loss
Authenticator migration
Accessibility
BYOD support
Helpdesk capacity
Phishing resistance
Admin/high-value account security

Bad fallback design

Passkey login required
But user can click “lost device”
Receive SMS OTP
Register new passkey

This is vulnerable to SIM swap.

Better fallback design

For standard users:

Use backup passkey or trusted device
Use backup codes
Use email + risk checks
Notify old devices/channels
Revoke old sessions after recovery

For admins:

Require backup hardware key
Require manager/security approval
Use temporary access pass
Require re-enrollment of hardware key
Revoke all sessions
Audit recovery
No SMS fallback

Model answer

Fallback design is the hardest part because fallback becomes the attacker’s preferred path. If passkey login is strong but fallback uses SMS, email-only recovery, or weak helpdesk proofing, the real assurance level is the fallback. A good rollout must design backup authenticators, recovery codes, temporary access, user notifications, session revocation, and stricter recovery for admins before enforcing passkeys.

7. How would you design recovery for users who lose all authenticators?

If a user loses all authenticators, they have lost the normal proof of possession.

At that point, recovery must become identity re-proofing, not just “send a weaker login code.”

Standard user recovery

Possible controls:

Verified trusted device if one still exists
Backup codes
Previously registered recovery contact
High-confidence email plus risk checks
Government ID verification for high-risk consumer accounts
Manager approval for enterprise accounts
Temporary access pass
Cooling-off period for sensitive changes

Enterprise recovery flow

User reports lost authenticators
        ↓
Helpdesk opens recovery case
        ↓
Verify user through approved channel
        ↓
Manager approval required
        ↓
Issue short-lived temporary access pass
        ↓
User registers new passkey/security key
        ↓
Old authenticators revoked
        ↓
All sessions revoked
        ↓
Recovery event reviewed and logged

Admin recovery flow

For admins:

No self-service weak fallback
No SMS
No email-only reset
Require security approval
Require manager approval
Possibly require in-person/video verification
Issue time-limited privileged recovery
Require hardware key registration
Review recent account activity

Post-recovery controls

Immediately revoke:

Old sessions
Refresh tokens
Old authenticators
Recovery links
Backup codes if exposed
OAuth grants if suspicious
API tokens if suspicious
Remembered devices

Model answer

If users lose all authenticators, recovery should use high-assurance re-proofing proportional to account risk. For enterprise users, that means helpdesk workflow, verified channels, manager approval, temporary access passes, and forced re-enrollment. For admins, add security approval and possibly in-person verification. After recovery, revoke old authenticators, sessions, refresh tokens, and suspicious grants.

8. What are the tradeoffs between platform authenticators and roaming security keys?

Platform authenticators

Platform authenticators are built into devices.

Examples:

Windows Hello
Apple Face ID / Touch ID passkeys
Android passkeys
macOS Touch ID

Benefits:

Very convenient
Good adoption
Biometric/PIN user verification
No extra hardware for most users
Great for consumer and general workforce rollout

Risks/tradeoffs:

Tied to device or cloud sync model
Recovery depends on device/platform ecosystem
Device loss or replacement needs planning
Assurance varies by platform
Harder to enforce exact hardware model unless managed

Roaming security keys

Roaming authenticators are external keys.

Examples:

YubiKey
Google Titan Security Key
Feitian key
Smart-card-like FIDO2 keys

Benefits:

Strong device-bound assurance
Portable across machines
Good for admins/shared workstations
Private keys usually non-exportable
Can be governed with attestation

Risks/tradeoffs:

Users can lose them
Procurement and logistics required
Need backup keys
Less seamless on mobile depending on NFC/USB/Bluetooth
Higher support burden

Comparison table

Area Platform Authenticator Roaming Security Key
Usability Very high Medium
Deployment cost Low if devices support it Higher due to hardware
Admin assurance Medium to high High
Recovery Easier with sync Requires backup/recovery
Device-bound control Varies Strong
Best for Broad workforce/consumers Admins, developers, high-risk users

Model answer

Platform authenticators are highly usable and ideal for broad rollout because users can authenticate with built-in biometrics or device PINs. Roaming security keys provide stronger device-bound assurance and portability across machines, making them better for administrators and high-risk users. The tradeoff is adoption and recovery: platform authenticators are easier, while roaming keys require procurement, backup keys, and stronger lifecycle management.

9. How would you migrate a password-heavy service to passkeys at scale?

I would use a phased migration rather than forcing everyone immediately.

Phase 1: Prepare the platform

Implement WebAuthn registration and authentication
Support multiple authenticators per account
Store credential IDs and public keys
Support user verification policy
Support device naming and credential management
Build recovery workflow
Build telemetry and fraud detection

Phase 2: Start as optional

After password login, prompt:

“Add a passkey for faster, safer sign-in.”

Good timing:

After successful MFA
After account security checkup
During onboarding
When user is on compatible device/browser

Phase 3: Use passkeys as MFA first

For existing password users:

Password + passkey

Then gradually move to:

Passkey-first
Password fallback

Then for high-confidence users:

Passkey-only with strong recovery

Phase 4: Risk-based enforcement

Prioritize:

Admins
High-value accounts
Users attacked by credential stuffing
Users with breached passwords
Employees
Frequent users

Phase 5: Reduce weak fallback

Once adoption is high:

Disable SMS fallback for high-risk users
Require passkey for sensitive actions
Require passkey for new devices
Make password fallback step-up/risk-based
Eventually remove password for eligible users

Phase 6: Monitor migration

Track:

Registration success rate
Authentication success rate
Fallback usage
Recovery volume
Device/browser compatibility
Authenticator types
Failed ceremonies
Account takeover rate
Helpdesk tickets
User abandonment

Model answer

I would migrate by first adding WebAuthn support and strong recovery, then offering passkey enrollment after successful login. I would start with optional enrollment, then use passkeys as MFA, then make passkey-first login the default, and finally remove password fallback for eligible or high-risk users. I would prioritize admins and high-risk accounts, support multiple authenticators, monitor adoption and failures, and reduce weak recovery paths before enforcing passkeys broadly.

10. What telemetry would help detect abuse in a WebAuthn deployment?

WebAuthn abuse detection should cover registration, authentication, recovery, device changes, and post-login behavior.

Registration telemetry

Collect:

User ID
Credential ID
Authenticator type
Attestation result
AAGUID if available
User verification used
Resident/discoverable credential status
Backup eligibility / backup state if available
Device/browser
IP/ASN/geo
Registration after recovery?
Registration after password reset?

Suspicious examples:

New authenticator registered immediately after account recovery
Admin registers unknown authenticator type
Many accounts register authenticators from same IP/device
Authenticator added from unusual country

Authentication telemetry

Collect:

Credential ID used
User verification result
Authenticator sign count if available
Device/browser
IP/ASN/geo
Failed assertions
Origin/RP ID failures
Authentication method used
Session created

Suspicious examples:

Passkey login from impossible travel location
Passkey used after suspicious recovery
Repeated failed WebAuthn assertions
Unexpected authenticator for privileged account

Recovery and fallback telemetry

Collect:

Fallback used instead of passkey
Lost authenticator claim
Helpdesk reset
Temporary access pass issued
Old credentials removed
New credentials added
Sessions revoked or not

High-risk pattern:

Recovery completed
        ↓
New passkey registered
        ↓
Old passkeys removed
        ↓
Data export/API token created

Post-authentication telemetry

Collect:

Password changed
Recovery email/phone changed
MFA/passkey removed
OAuth grant added
API token created
Sensitive data accessed
Admin role assumed
Large export performed

Model answer

I would collect WebAuthn registration and authentication telemetry: credential ID, authenticator type, attestation result, user verification status, device/browser, IP/ASN/geo, failed assertions, and RP/origin errors. I would heavily monitor recovery and fallback events, especially new authenticator registration after recovery, authenticator removal, unusual admin authenticator types, and sensitive actions immediately after passkey changes.

Key Takeaways

  • WebAuthn and passkeys replace shared secrets with origin-bound public-key authentication.
  • They are among the strongest mainstream defenses against phishing and credential reuse.
  • Recovery, fallback, and device lifecycle design are the main implementation risks.
  • Server-side private secret exposure is reduced because only public keys are stored.
  • Strong authenticators still require strong session and authorization controls after login.
