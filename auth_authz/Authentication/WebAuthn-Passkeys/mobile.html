WebAuthn / FIDO2 Threat Modeling Report

Mobile Android Environment

1. Executive Summary

This report provides a threat model for WebAuthn/FIDO2/passkey authentication in a mobile Android environment. The target environment includes:

  • Android end-user devices
  • Google Chrome on Android
  • Native Android mobile apps using Credential Manager
  • Android WebView where applicable
  • Google Password Manager or third-party credential providers
  • Android screen lock, biometrics, PIN, or pattern
  • External FIDO2 security keys over supported transports
  • Relying Party web application and backend server

WebAuthn defines a browser/platform API for creating and using scoped public-key credentials. A public-key credential is created for a relying party and can only be accessed by origins belonging to that relying party, which is the foundation of WebAuthn phishing resistance. (W3C)

On Android, passkeys are commonly exposed through Credential Manager, which supports passkeys, passwords, and federated sign-in through a unified API. (Android Developers) For native Android apps, the app must be associated with the relying-party domain using an OS-provided association mechanism such as Digital Asset Links. (Google for Developers)

A key Android-specific design point is that passkeys may be stored in a credential provider such as Google Password Manager and may sync across devices. Google states that passkeys are securely encrypted on-device before syncing, and Android 14 or later can also use compatible third-party password managers. (Google for Developers) Therefore, in Android threat modeling, you must distinguish between:

  • Credential binding to the RP and user account
  • App-to-domain binding using Digital Asset Links
  • Device compliance binding through MDM / Android Enterprise / Play Integrity / Conditional Access
  • Authenticator assurance through UV, attestation, and credential metadata

4. WebAuthn vs FIDO2 in Android Context

FIDO2 is the broader standard family that includes WebAuthn and CTAP2. WebAuthn is the web-facing API, while CTAP2 defines communication between a client/platform and authenticators, including roaming authenticators over transports such as USB, NFC, or BLE. (FIDO Alliance)

10. Native Android App Binding with Digital Asset Links

For Android native apps, the RP must not treat any app claiming to represent the brand as trusted. The app must be bound to the RP server domain using Digital Asset Links. Google’s passkey guidance states that a passkey mobile application must be bound to the RP server domain using an OS-provided association mechanism such as Digital Asset Links. (Google for Developers)

Digital Asset Links Security Requirements

Control Purpose
Host assetlinks.json on HTTPS Prevent tampering
Serve with correct content type Ensure platform verification
No redirects for .well-known file Avoid association failure or ambiguity
Include correct package name Bind only the intended app
Include correct SHA-256 signing certificate fingerprint Bind to the real signed app
Maintain release/debug signing separation Prevent test app from being trusted in production
Monitor assetlinks changes Detect unauthorized domain-app association changes

The SHA-256 certificate fingerprint is the cryptographic identity commonly used by API providers alongside the Android package name, and Google Play Console exposes SHA-256 fingerprints for app signing and upload certificates. (Android Developers)

13.2 What WebAuthn Does Not Automatically Prove

WebAuthn/passkey registration does not automatically prove:

  • Exact Android device serial number
  • Corporate asset ID
  • MDM compliance status
  • Whether device is rooted
  • Whether the app is genuine
  • Whether the phone is the same physical device as last login
  • Whether passkey has synced to another device
  • Whether the credential provider is enterprise-approved

13.3 How to Add Device/App Assurance

Requirement Control
Only genuine app can access backend Play Integrity API
Only managed Android devices Android Enterprise / MDM compliance
Only approved app package/signing cert Digital Asset Links + app signing controls
Only approved credential providers Enterprise mobile policy where available
Detect rooted/tampered/emulated devices Play Integrity / MDM / mobile threat defense
Higher assurance for admins Hardware security key, device compliance, no BYOD

Google describes the Play Integrity API as helping servers check that requests come from the genuine app installed by Google Play and running on a genuine certified Android device, enabling responses to risky interactions from tampered apps, untrustworthy devices, or emulated environments. (Android Developers)

27. Minimum RP Server Validation

For every Android passkey authentication assertion, the RP should verify:

1. Challenge matches the server-issued challenge.
2. Challenge is single-use.
3. Challenge is short-lived.
4. clientDataJSON.type is expected.
5. Origin / app identity is expected.
6. RP ID hash equals SHA-256(expected RP ID).
7. Credential ID exists.
8. Credential ID belongs to the expected user.
9. Signature verifies using the stored public key.
10. UP flag is present if required.
11. UV flag is present if user verification is required.
12. COSE algorithm is approved.
13. Sign counter is evaluated where meaningful.
14. Backup eligibility / backup state is evaluated for policy decisions.
15. Device/app risk signals are evaluated where required.
16. Session ID/token is rotated after successful authentication.
17. Authentication event is logged.

33. Practical Server-Side Validation Pseudocode

function verifyAndroidPasskeyAssertion(assertion, expectedUser, expectedChallenge, expectedRpId, allowedOrigins, policy):

    clientData = parseClientDataJSON(assertion.clientDataJSON)
    authData = parseAuthenticatorData(assertion.authenticatorData)

    require clientData.type == "webauthn.get"
    require clientData.challenge == expectedChallenge
    require challengeNotExpired(expectedChallenge)
    require challengeNotUsedBefore(expectedChallenge)

    require clientData.origin in allowedOrigins

    expectedRpIdHash = SHA256(expectedRpId)
    require authData.rpIdHash == expectedRpIdHash

    credential = credentialRepository.find(assertion.credentialId)
    require credential != null
    require credential.userId == expectedUser.id
    require credential.status == "ACTIVE"

    require verifySignature(
        publicKey = credential.publicKey,
        signedData = assertion.authenticatorData + SHA256(assertion.clientDataJSON),
        signature = assertion.signature
    )

    require authData.flags.UP == true

    if policy.requiresUserVerification:
        require authData.flags.UV == true

    if policy.disallowsBackedUpCredentialsForAdmins:
        require authData.flags.BE == false or userIsNotAdmin(expectedUser)

    if credential.signCountSupported:
        if authData.signCount < credential.previousSignCount:
            raiseRiskEvent("Possible cloned credential")
        update credential.previousSignCount

    if policy.requiresMobileIntegrity:
        integrityResult = verifyPlayIntegrityToken(assertion.integrityToken)
        require integrityResult.appIsGenuine == true
        require integrityResult.deviceMeetsPolicy == true

    markChallengeUsed(expectedChallenge)

    rotateSession()
    createAuthenticatedSession(expectedUser)

    logEvent("PASSKEY_LOGIN_SUCCESS", expectedUser, credential, authData.flags)

36. Conclusion

For Android, the WebAuthn cryptography remains strong, but the threat model expands beyond the browser. The main Android-specific risks are:

  1. Fake or repacked mobile apps
  2. Missing or misconfigured Digital Asset Links
  3. Synced passkeys used from unmanaged devices
  4. Unapproved credential providers
  5. Rooted or malware-infected devices
  6. Overlay and accessibility abuse
  7. WebView session exposure
  8. Weak recovery through email/SMS/helpdesk
  9. Weak mobile token storage
  10. Ignoring the UV flag on the RP server

The recommended design is:

Use Credential Manager for Android passkeys, enforce Digital Asset Links for native apps, require user verification, validate all WebAuthn ceremony fields server-side, protect mobile sessions, evaluate app/device integrity, and apply stricter authenticator, recovery, and device-compliance rules for privileged users.
