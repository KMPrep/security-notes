WebAuthn / FIDO2 Threat Modeling Report

Windows End-User Environment with Google Chrome Browser

1. Executive Summary

This report provides a security threat model for a WebAuthn/FIDO2 deployment where end users authenticate from Windows endpoints using Google Chrome. The main components are:

  • Browser / Client: Google Chrome on Windows
  • Operating system layer: Windows WebAuthn platform APIs
  • Authenticator: Windows Hello, Windows Hello for Business, platform passkeys, or external FIDO2 security keys
  • Relying Party: Web application and backend server
  • Identity layer: Optional IdP, SSO, Conditional Access, or enterprise IAM

WebAuthn is a W3C API that allows web applications to create and use scoped public-key credentials. These credentials are bound to a relying party and are only usable by origins belonging to that relying party, which is a core reason WebAuthn is phishing-resistant. (W3C)

FIDO2 is the broader standard family that includes WebAuthn and CTAP2. CTAP2 enables a browser or platform to communicate with external authenticators such as FIDO security keys, mobile devices, and other authenticators over transports such as USB, NFC, or BLE. (FIDO Alliance)

In Windows environments, WebAuthn can use Windows Hello, Windows Hello for Business, or FIDO2 security keys. Microsoft describes Windows Hello for Business as using a device-specific credential combined with a PIN or biometric gesture. (Microsoft Learn)

10. User Presence vs User Verification

WebAuthn authenticator data contains flags including UP, UV, BE, and BS. The WebAuthn Level 3 specification describes backup eligibility and backup state using BE and BS flags, and the same authenticator data structure includes the flags used by the RP to evaluate authenticator behavior. (W3C)

Concept Flag Meaning Example
User Presence UP Someone interacted with the authenticator Touching a USB security key
User Verification UV Authenticator locally verified the user Windows Hello PIN, face, fingerprint, FIDO2 PIN

10.1 User Presence

User Presence means:

A human interacted with the authenticator during the ceremony.

Examples:

  • Touching a FIDO2 security key
  • Tapping an NFC security key
  • Confirming a platform authenticator prompt

User Presence does not prove the correct user was verified. It only proves someone was physically present.

10.2 User Verification

User Verification means:

The authenticator locally verified the user before using the credential.

Examples:

  • Windows Hello PIN
  • Windows Hello biometric
  • FIDO2 security key PIN
  • Security key biometric verification

For passwordless and privileged access, the RP should require:

{
  "userVerification": "required"
}

And the server must verify that the UV flag is present in the returned authenticatorData.

11. Does Registration Bind to a Device?

11.1 What WebAuthn Registration Binds

WebAuthn registration primarily creates a credential binding, not a full enterprise device binding.

The RP normally stores:

Data Purpose
Credential ID Identifies the registered credential
Public key Verifies future signatures
User handle Binds credential to account
RP ID / RP ID hash Binds credential to relying party
Sign counter Helps detect cloned authenticators where supported
AAGUID Identifies authenticator model/type
Attestation data Optional authenticator assurance evidence
Transports USB, NFC, BLE, internal, hybrid, where available
Backup flags Whether credential is backup eligible or backed up

11.2 What WebAuthn Does Not Automatically Provide

WebAuthn registration does not normally prove:

  • Windows hostname
  • Laptop serial number
  • Exact corporate asset ID
  • TPM endorsement key identity
  • MAC address
  • Intune device ID
  • Whether device is currently compliant
  • Whether this is the same physical laptop as last time

23. Minimum Server-Side WebAuthn Validation

For every authentication assertion, the RP should verify:

1. Challenge matches server-issued challenge.
2. Challenge is single-use.
3. Challenge is not expired.
4. clientDataJSON.type is expected.
5. clientDataJSON.origin is in approved origin allowlist.
6. authenticatorData.rpIdHash equals SHA-256(expected RP ID).
7. Credential ID exists and belongs to the expected user.
8. Signature verifies using the stored public key.
9. UP flag is present if user presence is required.
10. UV flag is present if user verification is required.
11. COSE algorithm is approved.
12. Sign counter is evaluated where meaningful.
13. Session ID is rotated after successful login.
14. Authentication event is logged.

RR-08 Subdomain takeover affects RP ID trust Low High High DNS hygiene and subdomain scanning
RR-09 Lost authenticator causes lockout Medium Medium Medium Backup authenticator and recovery process
RR-10 Excessive attestation data collection Medium Medium Medium Data minimization

33. Practical Server-Side Validation Pseudocode

function verifyWebAuthnAssertion(assertion, expectedUser, expectedChallenge, expectedRpId, allowedOrigins):

    clientData = parseClientDataJSON(assertion.clientDataJSON)
    authenticatorData = parseAuthenticatorData(assertion.authenticatorData)

    require clientData.type == "webauthn.get"
    require clientData.challenge == expectedChallenge
    require challengeNotExpired(expectedChallenge)
    require challengeNotUsedBefore(expectedChallenge)

    require clientData.origin in allowedOrigins

    expectedRpIdHash = SHA256(expectedRpId)
    require authenticatorData.rpIdHash == expectedRpIdHash

    credential = credentialRepository.find(assertion.credentialId)
    require credential is not null
    require credential.userId == expectedUser.id
    require credential.status == "ACTIVE"

    require verifySignature(
        publicKey = credential.publicKey,
        signedData = assertion.authenticatorData + SHA256(assertion.clientDataJSON),
        signature = assertion.signature
    )

    require authenticatorData.flags.UP == true

    if policy.requiresUserVerification:
        require authenticatorData.flags.UV == true

    if credential.signCountSupported:
        require authenticatorData.signCount >= credential.previousSignCount
        update credential.previousSignCount

    markChallengeUsed(expectedChallenge)

    rotateSessionId()
    createAuthenticatedSession(expectedUser)

    logAuthenticationSuccess(expectedUser, credential, authenticatorData.flags)

36. Conclusion

WebAuthn/FIDO2 provides strong phishing-resistant authentication, but the overall system is only as strong as the surrounding controls.

The highest-risk areas are usually not the WebAuthn cryptography itself. They are:

  1. Weak account recovery
  2. Weak passkey registration controls
  3. Ignoring the UV flag
  4. Malicious browser extensions
  5. Compromised Windows endpoints
  6. Weak session management after successful login
  7. Over-permissive fallback authentication
  8. No special controls for privileged administrators

For your environment, the recommended model is:

Use WebAuthn with user verification required, validate all server-side ceremony fields, restrict privileged users to approved authenticators on managed Windows devices, govern Chrome extensions, secure the session layer, and protect credential lifecycle and recovery flows with step-up authentication and monitoring.
