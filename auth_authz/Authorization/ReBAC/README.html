ReBAC - Relationship-Based Access Control

Category: Authorization
Common References: Zanzibar-style authorization models, graph-based permission systems

Overview

Relationship-based access control, or ReBAC, grants access based on relationships between principals and resources rather than only on roles or attributes. Typical rules look like user is editor of document, user is member of group that owns project, or user can view item because they are friend of owner. ReBAC is a strong fit for collaboration products, multi-tenant data platforms, document sharing, and graph-shaped domains where access propagates through membership and ownership links. The main challenges are graph complexity, consistency, and making effective permissions understandable.

How It Works

Core Mechanics

ReBAC models authorization as tuples or edges in a graph:

Element Example
Subject user:alice
Relation viewer, editor, member, owner
Object doc:quarterly-plan

Common concepts:

Concept Why it matters
Direct relationship User has direct permission on object
Derived relationship Access flows through group, parent, or sharing chain
Recursive evaluation Permissions may require traversing multiple edges
Consistency model Tuple writes and reads may not be instantly aligned

Step-by-Step Flow

sequenceDiagram
    participant U as User
    participant E as Enforcement Point
    participant G as Relationship Store
    U->>E: Request action on object
    E->>G: Check relation graph for permission
    G->>G: Traverse direct and inherited relations
    G-->>E: Permit or deny
    E-->>U: Enforce decision

flowchart TD
    A[Access request] --> B[Identify subject, relation, and object]
    B --> C[Look for direct tuple match]
    C --> D{Direct relation found?}
    D -- Yes --> E[Permit]
    D -- No --> F[Traverse groups, parents, or shares]
    F --> G{Derived relation found?}
    G -- Yes --> E
    G -- No --> H[Deny and log]

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Graph mis-modeling Relationships are defined too broadly Cross-tenant or unintended sharing High
2 Stale tuples Revocations do not propagate quickly Persistent unauthorized access High
3 Unbounded traversal Recursive checks become too expensive Latency spikes or DoS Med
4 Hidden inheritance paths Access comes from indirect relationships reviewers miss Surprise privilege grants Med
5 Consistency gaps Write and read paths disagree temporarily Flapping or unsafe decisions Med
6 Tenant boundary mistakes Shared groups or parents bridge tenants Data exposure High
7 Poor explainability Users and admins cannot tell why access exists Audit and support pain Med
8 Bulk-sharing abuse One relationship change exposes many resources Large blast radius Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Graph mis-modeling Keep schema simple and threat-model derived relationships Authorization design
2 Stale tuples Prioritize fast revocation and bounded cache lifetime Runtime architecture
3 Unbounded traversal Set traversal depth and complexity limits Decision engine
4 Hidden inheritance paths Build explain APIs for effective relation resolution Review and support tooling
5 Consistency gaps Make critical writes and reads use clear consistency guarantees Platform design
6 Tenant boundary mistakes Encode tenant isolation explicitly in relationship schema Multi-tenant model
7 Poor explainability Log resolved relation chains for decisions Audit tooling
8 Bulk-sharing abuse Alert on high-impact sharing changes Detection and governance

Top 3 Critical Mitigations

  1. Design the relation schema carefully: ReBAC power comes from graph expressiveness, but bad schema design can create invisible privilege propagation and tenant isolation failures.
  2. Make revocation and consistency explicit: Relationship stores that are eventually consistent need clear operational guidance so security-critical revocations are not assumed instantaneous.
  3. Support explainability for effective access: Users and admins must be able to see the chain of relationships that granted access, or the system will be too opaque to trust.

Key Takeaways

  • ReBAC models authorization as relationships rather than only roles or attributes.
  • It is powerful for collaboration, sharing, and graph-shaped product domains.
  • Schema design, revocation behavior, and explainability are the critical security concerns.
  • Tenant isolation must be explicit in the relationship model.
  • ReBAC often pairs well with RBAC or ABAC rather than replacing them completely.
