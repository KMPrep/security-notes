Kerberos - Ticket-Based Network Authentication

Category: Federation and network authentication
Primary Standards: RFC 4120 and related extensions

Overview

Kerberos is a ticket-based authentication protocol designed for mutual authentication across untrusted networks using symmetric cryptography and a trusted key distribution center. It is the backbone of Active Directory domain authentication and is widely used for enterprise workstation logon, service authentication, and single sign-on inside corporate environments. Instead of repeatedly sending passwords to services, clients obtain tickets and present them to services that trust the Kerberos infrastructure. Time synchronization, key management, and service principal hygiene are central to Kerberos security.

How It Works

Core Mechanics

Key components:

Component Role
Client User or host requesting authentication
KDC Trusted authority made of the AS and TGS
Authentication Server Issues ticket-granting tickets after initial auth
Ticket Granting Server Issues service tickets
Service Accepts Kerberos service tickets

Important artifacts:

Artifact Purpose
TGT Ticket-granting ticket used to request service tickets
Service ticket Ticket for a specific target service
Session key Shared key used between parties for an auth step
Service principal Identity bound to a service account and key

Kerberos depends on clocks being reasonably synchronized because tickets have defined validity windows. Much of Kerberos abuse in real environments involves weak service account keys, over-privileged principals, and ticket theft after endpoint compromise.

Step-by-Step Flow

User sign-in is step 0 in practice. The user enters a password, smart card PIN, or other credential locally on the client, and the client uses that secret to prove identity to the Authentication Server. The password itself is not sent to every service and, in normal Kerberos flows, is not sent in plaintext to the KDC.

sequenceDiagram
    participant C as Client
    participant AS as Authentication Server
    participant TGS as Ticket Server
    participant S as Service
    C->>C: User signs in locally
    C->>AS: AS-REQ: client principal, realm, nonce, etypes, pre-auth data
    AS-->>C: AS-REP: TGT, client-TGS session key, flags, lifetime
    C->>TGS: TGS-REQ: TGT, authenticator, target SPN, nonce, options
    TGS-->>C: TGS-REP: service ticket, client-service session key, flags, lifetime
    C->>S: AP-REQ: service ticket and authenticator
    S-->>C: AP-REP: optional mutual auth proof

flowchart TD
    A[User signs in locally with password smart card or other factor] --> B[Client derives or unlocks key material]
    B --> C[AS-REQ to AS: principal realm nonce etypes pre-auth]
    C --> D[AS-REP to client: TGT and client-TGS session key]
    D --> E[TGS-REQ to TGS: TGT authenticator target SPN]
    E --> F[TGS-REP to client: service ticket and client-service session key]
    F --> G[AP-REQ to service: service ticket and authenticator]
    G --> H[Service validates ticket time and session data]
    H --> I[Optional AP-REP and access decision]

Data passed at each stage:

Step Exchange What the client sends What comes back
0 Local sign-in Username plus password, smart card PIN, or other factor entered on the device Local key material is derived or unlocked for Kerberos use
1 AS-REQ Client principal, realm, requested TGS service (krbtgt/REALM), nonce, supported encryption types, and pre-auth data such as an encrypted timestamp No service access yet; this is the request for an initial TGT
2 AS-REP None; this is the KDC response to the AS-REQ TGT, client-TGS session key, ticket flags, validity times, and other reply metadata
3 TGS-REQ TGT, authenticator encrypted with the client-TGS session key, requested service principal name (SPN), nonce, and ticket options No direct service access yet; this is the request for a service ticket
4 TGS-REP None; this is the KDC response to the TGS-REQ Service ticket, client-service session key, ticket flags, validity times, and target service identity
5 AP-REQ Service ticket plus an authenticator proving possession of the client-service session key Service decides whether to trust the ticket and grant access
6 AP-REP optional None unless mutual auth is required by the protocol or application Server proof that it also holds the client-service session key

Why the original user sign-in looked missing: the earlier sequence diagram started at the first network-visible Kerberos exchange with the KDC. The actual user sign-in happens locally before that exchange and is what lets the client generate or unlock the secret needed for Kerberos pre-auth.

Message Payloads and Ticket Structure

This section covers the actual fields sent in each Kerberos exchange, the internal structure of a TGT and service ticket, and the layout of the authenticator. Understanding these fields is essential for protocol-level debugging, security analysis, and attack detection.

AS-REQ — Client to Authentication Server

The AS-REQ is the first network message in Kerberos. The client sends it to the AS to request a TGT.

AS-REQ {
  pvno           = 5                         -- Kerberos protocol version
  msg-type       = KRB_AS_REQ (10)
  padata         = [                         -- pre-authentication data
    PA-ENC-TIMESTAMP {                       -- encrypted with client long-term key
      patimestamp  = <current UTC time>      -- replay prevention
      pausec       = <microseconds>
    }
    PA-PAC-REQUEST {                         -- request PAC inclusion (Windows AD)
      include-pac = TRUE
    }
  ]
  req-body {
    kdc-options    = <flags>                 -- e.g. forwardable, renewable, canonicalize
    cname          = PrincipalName {         -- identity requesting the TGT
      name-type    = NT-PRINCIPAL (1)
      name-string  = ["alice"]
    }
    realm          = "CORP.EXAMPLE.COM"      -- client's realm
    sname          = PrincipalName {         -- target: always krbtgt/REALM
      name-type    = NT-SRV-INST (2)
      name-string  = ["krbtgt", "CORP.EXAMPLE.COM"]
    }
    till           = <requested expiry>      -- desired ticket lifetime
    rtime          = <renewable until>       -- desired renewable limit
    nonce          = <random 32-bit integer> -- ties request to response
    etype          = [18, 17, 23]            -- supported enctype priority list
                                             -- 18=AES256, 17=AES128, 23=RC4
    addresses      = [                       -- optional: restrict ticket to host
      HostAddress { addr-type=2, address=<IPv4> }
    ]
  }
}

Key points:

Field Security Relevance
PA-ENC-TIMESTAMP Encrypted with the client's long-term key (derived from password); proves identity without sending the password; requires clock sync
kdc-options FORWARDABLE and RENEWABLE flags affect how the issued TGT can be reused; disabling FORWARDABLE limits delegation attacks
etype Attacker-controlled tools may request only RC4 (type 23) to get tickets that are cheaper to crack offline
nonce Prevents response replay; client verifies its own nonce in the AS-REP
addresses Restricting tickets to specific IP addresses limits ticket theft but is often omitted for compatibility

If pre-authentication is disabled on the account (UF_DONT_REQUIRE_PREAUTH), the padata field is absent. The KDC responds anyway, enabling AS-REP Roasting.

AS-REP — Authentication Server to Client

The AS-REP carries the TGT back to the client. Part of it is encrypted with the client's long-term key; the TGT itself is encrypted with the krbtgt key and is opaque to the client.

AS-REP {
  pvno     = 5
  msg-type = KRB_AS_REP (11)
  crealm   = "CORP.EXAMPLE.COM"
  cname    = PrincipalName { name-string = ["alice"] }

  ticket   = Ticket {                        -- the TGT; opaque to the client
    tkt-vno  = 5
    realm    = "CORP.EXAMPLE.COM"
    sname    = PrincipalName {
      name-string = ["krbtgt", "CORP.EXAMPLE.COM"]
    }
    enc-part = EncryptedData {               -- encrypted with krbtgt's long-term key
      etype   = 18                           -- AES256
      kvno    = 2                            -- key version of krbtgt
      cipher  = <encrypted EncTicketPart>    -- see TGT internal structure below
    }
  }

  enc-part = EncryptedData {                 -- encrypted with client's long-term key
    etype  = 18
    cipher = <encrypted EncASRepPart> {
      key         = EncryptionKey {          -- client-TGS session key
        keytype   = 18
        keyvalue  = <random session key>
      }
      last-req    = [{ lr-type=0, lr-value=<last logon time> }]
      nonce       = <matches nonce from AS-REQ>  -- proof this is fresh
      key-expiration = <password expiry>
      flags       = { forwardable, renewable, initial, pre-authent }
      authtime    = <time of initial authentication>
      starttime   = <ticket valid from>
      endtime     = <ticket valid until>     -- typically 10 hours in AD
      renew-till  = <maximum renewable until> -- typically 7 days in AD
      srealm      = "CORP.EXAMPLE.COM"
      sname       = ["krbtgt", "CORP.EXAMPLE.COM"]
    }
  }
}

The client decrypts enc-part using its long-term key. The decrypted EncASRepPart contains the client-TGS session key the client needs for subsequent TGS-REQ operations. The TGT itself stays encrypted with the krbtgt key — the client cannot read or modify it.

TGT — Internal Structure (EncTicketPart)

The TGT is the blob inside AS-REP.ticket.enc-part. Only the KDC (using the krbtgt key) can read or forge it. When the client presents the TGT in a TGS-REQ, the TGS decrypts it to extract the session key and client identity.

EncTicketPart {                              -- encrypted with krbtgt key
  flags       = { forwardable, renewable, initial, pre-authent }
  key         = EncryptionKey {             -- client-TGS session key (same one given to client)
    keytype   = 18
    keyvalue  = <random session key>
  }
  crealm      = "CORP.EXAMPLE.COM"
  cname       = PrincipalName { name-string = ["alice"] }
  transited   = TransitedEncoding { ... }  -- realm traversal path for cross-realm
  authtime    = <time of authentication>
  starttime   = <valid from>
  endtime     = <valid until>
  renew-till  = <maximum renewal limit>
  caddr       = [ <client addresses if restricted> ]
  authorization-data = [                   -- contains the PAC in AD environments
    AuthorizationData {
      ad-type = AD-WIN2K-PAC (128)
      ad-data = <PAC blob>                 -- see PAC structure below
    }
  ]
}

PAC (Privilege Attribute Certificate): Windows-specific extension inside authorization-data. The PAC carries the user's SIDs, group memberships, account flags, and logon information. It is signed by both the KDC service key and the krbtgt key. Services that validate the PAC can enforce fine-grained authorization from AD group data without another LDAP lookup. Forged TGTs with manipulated PAC data are the basis of attacks like MS14-068.

PAC {
  LogonInfo {
    LogonTime, LogoffTime, PasswordLastSet
    EffectiveName = "alice"
    FullName, LogonScript, HomeDirectory
    UserId    = 1105             -- RID of the user
    PrimaryGroupId = 513         -- Domain Users
    GroupIds = [512, 519, ...]   -- Domain Admins, Enterprise Admins, etc.
    UserFlags, UserSessionKey
    LogonServer, LogonDomainName
    LogonDomainId = <domain SID>
    UserAccountControl
  }
  ServerChecksum  -- signed with service/krbtgt key
  KDCChecksum     -- signed with krbtgt key
  ClientInfo      -- client name and authentication time
  UPNDNSInfo      -- UPN and DNS domain name
}

TGS-REQ — Client to Ticket Granting Server

The TGS-REQ presents the TGT and an authenticator to request a service ticket for a specific SPN.

TGS-REQ {
  pvno     = 5
  msg-type = KRB_TGS_REQ (12)

  padata = [
    AP-REQ {                                 -- the TGT presentation
      pvno        = 5
      msg-type    = KRB_AP_REQ (14)
      ap-options  = {}
      ticket      = <TGT from AS-REP>        -- encrypted with krbtgt key; TGS decrypts it
      authenticator = EncryptedData {        -- encrypted with client-TGS session key
        etype  = 18
        cipher = <Authenticator> {
          authenticator-vno = 5
          crealm  = "CORP.EXAMPLE.COM"
          cname   = ["alice"]
          cusec   = <microseconds>
          ctime   = <current timestamp>      -- must be within skew of TGS clock
          subkey  = <optional sub-session key>
          seq-number = <optional replay prevention>
          authorization-data = [...]         -- optional extra authz data
        }
      }
    }
  ]

  req-body {
    kdc-options   = <flags>                  -- e.g. renewable, forwardable, S4U2Self
    realm         = "CORP.EXAMPLE.COM"
    sname         = PrincipalName {          -- the target service SPN
      name-type   = NT-SRV-INST (2)
      name-string = ["MSSQLSvc", "sql01.corp.example.com:1433"]
    }
    till          = <requested expiry>
    nonce         = <random 32-bit integer>
    etype         = [18, 17, 23]
    enc-authorization-data = <optional>      -- additional authz passed through
    additional-tickets     = [...]           -- used for U2U or delegation
  }
}

Key points:

Field Security Relevance
AP-REQ inside padata Carries the TGT and proves the client holds the client-TGS session key via the authenticator
authenticator.ctime Timestamp prevents replay; the TGS rejects authenticators older than the clock skew window or already seen
sname Specifies the exact SPN; the TGS looks up which account owns that SPN to encrypt the service ticket
kdc-options S4U2SELF flag enables constrained delegation impersonation; FORWARDABLE on the resulting ticket enables further delegation
etype Attackers explicitly request RC4 (23) here during Kerberoasting to get tickets cheaper to crack

TGS-REP — Ticket Granting Server to Client

The TGS-REP returns the service ticket and the client-service session key.

TGS-REP {
  pvno     = 5
  msg-type = KRB_TGS_REP (13)
  crealm   = "CORP.EXAMPLE.COM"
  cname    = PrincipalName { name-string = ["alice"] }

  ticket   = Ticket {                        -- the service ticket; opaque to the client
    tkt-vno  = 5
    realm    = "CORP.EXAMPLE.COM"
    sname    = PrincipalName {
      name-string = ["MSSQLSvc", "sql01.corp.example.com:1433"]
    }
    enc-part = EncryptedData {               -- encrypted with the SERVICE ACCOUNT's long-term key
      etype  = 18                            -- AES256 (or 23=RC4 if downgraded)
      kvno   = 3                             -- key version of the service account
      cipher = <encrypted EncTicketPart>     -- same structure as TGT's EncTicketPart
                                             -- but encrypted with service key, not krbtgt
    }
  }

  enc-part = EncryptedData {                 -- encrypted with client-TGS session key
    etype  = 18
    cipher = <encrypted EncTGSRepPart> {
      key         = EncryptionKey {          -- client-SERVICE session key
        keytype   = 18
        keyvalue  = <new random session key>
      }
      last-req    = [...]
      nonce       = <matches nonce from TGS-REQ>
      flags       = { forwardable, renewable, ... }
      authtime    = <original authentication time>
      starttime   = <valid from>
      endtime     = <valid until>            -- typically shorter than TGT
      renew-till  = <max renewal>
      srealm      = "CORP.EXAMPLE.COM"
      sname       = ["MSSQLSvc", "sql01.corp.example.com:1433"]
    }
  }
}

Critical distinction: The service ticket enc-part is encrypted with the service account's long-term key, not the krbtgt key. This is why Kerberoasting works — an attacker who captures the TGS-REP.ticket.enc-part has a blob they can crack offline using the service account's password as the key. Neither the KDC nor the service is involved in that offline process.

Kerberoasting target = TGS-REP.ticket.enc-part
                       (encrypted with service account key)
Crack loop: hash(candidate_password) → try to decrypt → valid Kerberos structure? → found

Service Ticket — Internal Structure

Once at the service, the service decrypts the ticket with its own long-term key to extract the embedded session key and client identity. The structure is identical to EncTicketPart in the TGT, but encrypted with the service key.

EncTicketPart {                              -- encrypted with SERVICE ACCOUNT's long-term key
  flags       = { forwardable, renewable, ... }
  key         = EncryptionKey {             -- client-service session key
    keytype   = 18
    keyvalue  = <session key>              -- same value given to client in TGS-REP enc-part
  }
  crealm      = "CORP.EXAMPLE.COM"
  cname       = PrincipalName { name-string = ["alice"] }
  transited   = TransitedEncoding { ... }
  authtime    = <original authentication time>
  starttime   = <valid from>
  endtime     = <valid until>
  renew-till  = <renewal limit>
  caddr       = [...]
  authorization-data = [
    AuthorizationData {
      ad-type = AD-WIN2K-PAC (128)
      ad-data = <PAC>                      -- copied from TGT; service uses PAC for authz
    }
  ]
}

The service validates:

  1. It can decrypt the ticket with its own key (confirms it is the intended recipient).
  2. endtime has not passed (freshness check).
  3. ctime in the AP-REQ authenticator is within clock skew.
  4. The authenticator has not been seen before (replay prevention via short-lived replay cache).
  5. PAC signatures are valid (optional but recommended; prevents PAC forgery like MS14-068).

AP-REQ Authenticator — Client Proof of Ticket Possession

The authenticator is sent alongside every ticket presentation (both TGS-REQ and AP-REQ). It proves the sender holds the session key without revealing it.

Authenticator {                              -- encrypted with the relevant session key
  authenticator-vno = 5
  crealm      = "CORP.EXAMPLE.COM"
  cname       = PrincipalName { name-string = ["alice"] }
  cksum       = Checksum { ... }            -- optional checksum of AP-REQ
  cusec       = <microseconds>
  ctime       = <current timestamp>         -- must be within ~5 min of receiver's clock
  subkey      = EncryptionKey { ... }       -- optional: override session key for this connection
  seq-number  = <integer>                   -- optional: used to detect out-of-order/replayed messages
  authorization-data = [...]               -- optional additional authz claims
}

Why the authenticator matters for security:

Property Effect
Encrypted with session key Only a holder of the session key can produce a valid authenticator
ctime timestamp Binds the authenticator to a narrow time window; receiver checks it is recent
One-time use Servers cache recent authenticators to reject replays within the skew window
subkey Can establish a per-connection key, improving forward secrecy within a session
No password material The original password or long-term key is never present

Technique Deep-Dives

1. Pass-the-Ticket (PtT)

How it works: An attacker with local admin or SYSTEM on a Windows host dumps Kerberos tickets from lsass.exe memory using tools like Mimikatz (sekurlsa::tickets /export) or Rubeus (dump). The stolen ticket (TGT or service ticket) is injected into another session (kerberos::ptt ticket.kirbi) and used to authenticate as the ticket's owner without ever knowing the password.

Real example: In many ransomware intrusions, operators gain initial access, escalate to local admin, dump a Domain Admin's TGT from a shared jump server, and use it to move laterally to domain controllers — all without triggering a new Kerberos authentication event detectable as a login.

Key distinguisher from Pass-the-Hash: PtH sends an NTLM hash; PtT sends a valid Kerberos ticket. PtT is harder to detect because the ticket itself is legitimate — it was issued by the KDC.

2. Kerberoasting

How it works: Any authenticated domain user can request a service ticket for any SPN. The ticket's encrypted portion uses the service account's password hash as the key. An attacker requests tickets for accounts with SPNs registered (GetUserSPNs.py or Rubeus kerberoast), captures the encrypted blob, and cracks it offline with hashcat.

Real example: A developer account in a flat AD environment has MSSQLSvc/dbserver.corp.local:1433 registered. The service account password is Winter2022!. The attacker requests the ticket, extracts it, and cracks the hash in minutes with a wordlist attack. Now they have credentials for a service account that may have DBA rights.

Why it's operational, not protocol: The KDC does exactly what it should — issue tickets. The weakness is the low-entropy password on the service account. Managed Service Accounts (MSAs) or Group Managed Service Accounts (gMSAs) eliminate this by using 240-bit random, auto-rotated passwords.

How offline cracking works: The service ticket's encrypted section uses the service account's NTLM hash as the key and contains predictable Kerberos structure, so the cracker can verify a guess instantly without touching the network.

for each candidate password in wordlist/ruleset:
    1. hash the candidate → NTLM hash
    2. attempt to decrypt the captured blob with that hash
    3. check if the result has valid Kerberos structure
    4. if yes → password found (no lockout, no KDC log)
    5. if no  → next candidate

RC4-HMAC (type 23) tickets crack at ~1 billion guesses/sec on a modern GPU. Attackers explicitly request RC4 tickets even when AES is available because RC4 is ~10x faster to crack. AES256 tickets (hashcat mode 19700) crack at ~100 million/sec — slower but still feasible against weak passwords. The fix is not a stronger cipher — it is a high-entropy password that makes the search space infeasible regardless of speed.

Password Crack time estimate (GPU cluster, RC4)
Winter2022! Seconds — common wordlist + rules
P@ssw0rd1 Minutes — common substitution patterns
CorrectHorseBatteryStaple Hours to days — long but dictionary-based
gMSA auto-generated (240-bit random) Effectively impossible

The same cracking loop applies to AS-REP Roasting, with the AS-REP encrypted section as the captured blob and the user account's hash as the key.

3. Golden Ticket

How it works: If an attacker obtains the krbtgt account's NTLM hash (e.g., from a DCSync attack via lsadump::dcsync /user:krbtgt in Mimikatz), they can forge arbitrary TGTs offline. Because TGTs are signed with krbtgt's key, forged tickets are indistinguishable from real ones by any domain service.

Real example: After Mimikatz was publicly released, the NotPetya and subsequent nation-state campaigns routinely used DCSync to extract krbtgt and forge golden tickets with 10-year lifetimes, embedding themselves into domains even after password resets of normal accounts.

Persistence implication: Resetting user passwords does not invalidate golden tickets. The only remediation is rotating the krbtgt password twice (because Kerberos accepts tickets encrypted by either the current or previous krbtgt key for a window to avoid service disruption). Even then, outstanding forged tickets with extended lifetimes may remain valid until they expire.

4. Silver Ticket

How it works: Similar to golden tickets but scoped to a single service. The attacker needs only the service account's NTLM hash (not the krbtgt hash). They forge a service ticket directly, bypassing the KDC entirely — no TGS request is made, so there is no KDC log entry for this access.

Real example: An attacker compromises svc-backup (which runs a backup agent with local admin on all servers). They forge a silver ticket for CIFS/fileserver.corp.local, authenticating as a Domain Admin to the file server's SMB service — with no KDC-side log of the access.

Detection gap: Because the KDC is not involved, event logs on domain controllers show nothing. Detection requires monitoring on the target service itself (Kerberos decryption errors, unusual PAC fields, or behavioral anomaly detection).

5. AS-REP Roasting

How it works: If a user account has UF_DONT_REQUIRE_PREAUTH set (pre-authentication disabled), anyone on the network can send an AS-REQ for that account without credentials. The KDC replies with an AS-REP containing a portion encrypted with the account's password hash. The attacker captures this and cracks it offline.

Real example: A legacy application required pre-auth to be disabled on a service account because the app's Kerberos client library didn't support it. An attacker with only network access (no domain account required) harvests AS-REPs with impacket-GetNPUsers and cracks several service accounts.

Distinguisher from Kerberoasting: Kerberoasting requires a valid domain account to request service tickets; AS-REP roasting requires no credentials at all if pre-auth is disabled.

6. SPN Misconfiguration

How it works: Each SPN must map to exactly one account. Duplicate or stale SPNs cause authentication failures. More critically, if an SPN is registered on an attacker-controlled account (via setspn -A), they can intercept service authentication attempts for that SPN or use the SPN to make the account Kerberoastable.

Real example: An admin decommissions a SQL server but forgets to remove the MSSQLSvc SPN from the old service account. A new SQL server registers the same SPN on a different account, causing a conflict. Users intermittently authenticate to the wrong service. In a deliberate attack scenario, a low-privileged account is made Kerberoastable by registering a high-value SPN on it.

8. Unconstrained Delegation Abuse

How it works: If a computer or service account has unconstrained delegation enabled (TrustedForDelegation flag), when any user authenticates to it, their full TGT is forwarded to and cached by that service. An attacker who compromises the service can harvest those cached TGTs.

Real example (Printer Bug / SpoolSample): An attacker coerces a Domain Controller's machine account to authenticate to a server with unconstrained delegation using the MS-RPRN RpcRemoteFindFirstPrinterChangeNotification call. The DC's machine account TGT is delivered and cached on the attacker's server. The attacker injects it and DCsyncs the entire domain — a full domain compromise from a single host with unconstrained delegation.

10. Resource-Based Constrained Delegation (RBCD) Abuse

How it works: RBCD lets a resource (e.g., a computer object) specify which accounts can delegate to it, controlled via msDS-AllowedToActOnBehalfOfOtherIdentity. If an attacker can write to this attribute on a computer object (e.g., because they have GenericWrite over it), they create a new machine account, set it in the attribute, then use S4U2Self and S4U2Proxy to impersonate any user to that computer — typically gaining local admin.

Real example: In many AD privilege escalation chains (popularized by the "Kerberos RBCD" attack path), a low-privileged user has GenericWrite on a workstation object. They create a new machine account (domain users can create up to 10 by default via ms-DS-MachineAccountQuota), configure RBCD, and impersonate the local admin to compromise the workstation — without ever touching the KDC logs in an obvious way.

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Pass-the-ticket Harden endpoints and protect credential material in memory Host and EDR layer
2 Kerberoasting Use strong managed service account secrets and rotate them Identity operations
3 Golden or forged tickets Protect KDC secrets and monitor privileged identity changes Domain controllers and IAM ops
4 SPN misconfiguration Maintain accurate SPN registration and review collisions Directory administration
5 Clock skew Enforce reliable time synchronization Infrastructure layer
6 Weak delegation controls Restrict and audit constrained delegation Directory and service config
7 Over-privileged service accounts Reduce service account privileges and scope IAM design
8 Legacy crypto support Disable weak Kerberos encryption types Domain policy

Top 3 Critical Mitigations

  1. Protect ticket-bearing endpoints: Many Kerberos attacks start after endpoint compromise, not from protocol failure. Endpoint hardening and credential theft detection matter as much as KDC configuration.
  2. Use strong, rotated service credentials: Kerberoasting targets service accounts with weak secrets. Managed service accounts and long random keys materially reduce that risk.
  3. Constrain delegation and privilege: Delegation is powerful but dangerous. Constrained delegation and least privilege are necessary to stop services from becoming broad deputy identities.

Kerberos Model Answers — Deep Security Explanation

1. How does Kerberos avoid sending the user's password to each service?

Kerberos avoids sending the user’s password to every service by using a trusted third party, the Key Distribution Center, or KDC.

In Active Directory, the KDC runs on the Domain Controller.

Instead of this flow:

User password → Web Server
User password → File Server
User password → Database Server

Kerberos uses this flow:

User proves identity once to KDC
KDC gives user tickets
User presents tickets to services
Services trust the KDC

The user’s password is only used locally to derive a cryptographic key. The password itself is not sent to each service.

Simplified Kerberos Login Flow

User / Client                  Domain Controller / KDC                  Service
     |                                   |                                  |
     | 1. AS-REQ: I am Alice             |                                  |
     |---------------------------------->|                                  |
     |                                   |                                  |
     | 2. AS-REP: Here is a TGT          |                                  |
     |<----------------------------------|                                  |
     |                                   |                                  |
     | 3. TGS-REQ: I need access to CIFS |                                  |
     |---------------------------------->|                                  |
     |                                   |                                  |
     | 4. TGS-REP: Service ticket        |                                  |
     |<----------------------------------|                                  |
     |                                   |                                  |
     | 5. Present service ticket         |                                  |
     |-------------------------------------------------------------------->|
     |                                   |                                  |
     | 6. Access granted if valid        |                                  |

The key idea is:

Kerberos uses tickets, not passwords, to authenticate users to services.

The password is used only to decrypt the initial response from the KDC. After that, authentication relies on tickets and session keys.

2. What is the difference between a TGT and a service ticket?

A TGT and a service ticket are both Kerberos tickets, but they serve different purposes.

TGT — Ticket Granting Ticket

A TGT is used to request access to other services.

It is issued by the Authentication Service, or AS, part of the KDC.

The TGT is not meant for file servers, web servers, or databases. It is meant for the Ticket Granting Service, or TGS.

Think of the TGT as:

“I have already logged in. Please let me request access to services.”

Example:

Alice logs into her Windows workstation.
The Domain Controller issues Alice a TGT.
Alice can now request service tickets without typing her password again.

Service Ticket

A service ticket is used to access a specific service.

For example:

cifs/fileserver01.company.com
HTTP/app.company.com
MSSQLSvc/db01.company.com
LDAP/dc01.company.com

Think of the service ticket as:

“Alice is allowed to access this specific service.”

Comparison Table

Area TGT Service Ticket
Full name Ticket Granting Ticket Service Ticket
Used for Requesting service tickets Accessing a specific service
Presented to Ticket Granting Service Target application/service
Issued by Authentication Service Ticket Granting Service
Scope Broad Service-specific
Example krbtgt/domain.com cifs/fileserver01
If stolen Attacker can request many service tickets Attacker can access one service
AD account involved krbtgt account Service account or computer account

Simple Analogy

TGT = Passport
Service Ticket = Visa for one country/service

A passport proves who you are generally.
A visa allows entry into a specific country.

Similarly:

TGT proves you authenticated to the domain.
Service ticket proves you can access a specific service.

3. Why is time synchronization security-critical in Kerberos?

Time synchronization is security-critical because Kerberos uses timestamps to prevent replay attacks.

A replay attack happens when an attacker captures a valid authentication message and reuses it later.

Example:

Alice authenticates to a file server.
Attacker captures Alice’s Kerberos authenticator.
Attacker tries to replay it later to impersonate Alice.

Kerberos prevents this by including a timestamp inside the authenticator.

Authenticator = {
  client name,
  timestamp,
  session information
}

The service checks:

Is this timestamp close to the current time?
Has this authenticator already been seen?

If the timestamp is too old or too far in the future, the request is rejected.

Typical Kerberos Clock Skew

In many AD environments, the default allowed clock skew is around 5 minutes.

So if the client and server clocks differ too much:

Client time: 10:00 AM
Server time: 10:10 AM
Difference: 10 minutes
Result: Kerberos failure

Why This Matters for Security

Without strict time validation, an attacker could reuse captured authentication data.

Captured valid Kerberos request
        ↓
Replay later
        ↓
Gain unauthorized access

Why This Matters Operationally

Time drift can break authentication across the enterprise.

Affected systems may include:

ndows logon
File share access
SQL Server authentication
LDAP binds
Web SSO using Kerberos
Domain Controller replication
Service account authentication

Model Answer

Kerberos depends on synchronized time because it uses timestamps inside authenticators to limit the lifetime of authentication messages. This prevents replay attacks. If clocks drift too much, legitimate authentication fails; if time validation is too loose, replay risk increases.

4. What is Kerberoasting, and why is it mainly an operational weakness?

Kerberoasting is an attack where an authenticated domain user requests Kerberos service tickets for services running under domain user accounts, then cracks those tickets offline to recover the service account password.

Kerberoasting Flow

.Attacker has normal domain user account
        ↓
Queries AD for accounts with SPNs
        ↓
Requests service ticket for target SPN
        ↓
KDC returns ticket encrypted with service account key
        ↓
Attacker extracts ticket
        ↓
Offline password cracking
        ↓
Service account password recovered

Why This Works

In Kerberos, a service ticket is encrypted using the secret key of the service account.

For a service running as a domain user:

Service account password → cryptographic key
Service ticket encrypted using that key

If the service account has a weak password, the attacker can brute-force or dictionary-crack the ticket offline.

Example

Suppose there is a SQL service account:

Account: svc-sql-prod
SPN: MSSQLSvc/sql-prod.company.com:1433
Password: Winter2024!

An attacker can request a ticket for:

MSSQLSvc/sql-prod.company.com:1433

Then crack the ticket offline. If the password is weak, the attacker gets:

svc-sql-prod : Winter2024!

Now the attacker may use that account to access SQL Server or move laterally.

Why It Is Mainly an Operational Weakness

Kerberoasting is not usually a flaw in the Kerberos protocol itself.

It is usually caused by poor operational practices:

Weakness Why It Helps Kerberoasting
Weak service account passwords Easier offline cracking
Long-lived service accounts Passwords stay valid for years
Overprivileged service accounts Cracked password gives high access
User-based service accounts Easier to roast than machine accounts/gMSA
RC4 enabled Easier/faster cracking than modern AES
Poor SPN hygiene Many roastable accounts exposed

The protocol allows users to request service tickets. That behavior is normal. The weakness is that many environments use weak, static, overprivileged service account passwords.

Model Answer

Kerberoasting abuses normal Kerberos behavior. Any authenticated user can request a service ticket for an SPN. The ticket is encrypted with the service account’s key, so the attacker can crack it offline. It is mainly an operational weakness because the attack succeeds when service accounts have weak passwords, excessive privileges, old encryption types, and poor lifecycle management.

5. How does delegation create confused deputy risk in Active Directory environments?

Delegation allows one service to act on behalf of a user when accessing another service.

This is useful for multi-tier applications.

Example:

User → Web App → Database

The web app may need to access the database as the user.

Without delegation:

Database sees only the web app identity.

With delegation:

Database sees the original user identity.

Where the Risk Comes From

A confused deputy problem happens when a privileged service is tricked into using its authority on behalf of an attacker.

In AD delegation, the “deputy” is the delegated service.

Example:

Attacker authenticates to WebApp
        ↓
WebApp has delegation rights
        ↓
WebApp obtains delegated access to backend services
        ↓
Attacker abuses WebApp to reach resources they should not access

Types of Delegation

Type Risk Level Explanation
Unconstrained delegation Very high Service can impersonate users to many services
Constrained delegation Medium Service can impersonate users only to listed services
Resource-based constrained delegation Medium to high Backend resource controls who can delegate to it
Protocol transition High if misconfigured Service can obtain Kerberos tickets without user using Kerberos directly

Unconstrained Delegation Risk

With unconstrained delegation, when a user authenticates to a service, the user’s TGT may be made available to that service.

If the service is compromised:

Compromised delegated server
        ↓
Collect user TGTs
        ↓
Wait for admin to connect
        ↓
Steal admin TGT
        ↓
Move laterally as admin

This is especially dangerous if Domain Admins connect to systems trusted for unconstrained delegation.

Constrained Delegation Risk

Constrained delegation is safer but still risky.

Example:

WebApp can delegate to:
- MSSQLSvc/db01.company.com

If WebApp is compromised, the attacker may impersonate users to that SQL service.

Confused Deputy Example

Alice is a normal user.
Admin Bob has access to sensitive database rows.

WebApp is trusted for delegation to SQL Server.

Attacker compromises WebApp.
When Bob logs into WebApp, attacker abuses delegated credentials.
WebApp becomes the deputy that performs access on Bob's behalf.

Model Answer

Delegation creates confused deputy risk because a trusted service can be tricked or compromised into using another user’s authority against backend services. In AD, this is especially dangerous with unconstrained delegation, protocol transition, and overly broad constrained delegation. If the delegated service is compromised, attackers may impersonate users to downstream services.

6. What conditions make pass-the-ticket attacks possible?

A pass-the-ticket attack happens when an attacker steals a Kerberos ticket and reuses it without knowing the user’s password.

The attacker does not need the plaintext password.

They only need a valid ticket.

Basic Flow

Attacker compromises workstation
        ↓
Extracts Kerberos tickets from memory
        ↓
Injects ticket into attacker session
        ↓
Accesses services as victim

Conditions That Make Pass-the-Ticket Possible

1. Ticket Exists in Memory

Kerberos tickets are cached on the client machine after login.

On Windows, tickets may be available through LSASS.

User logs in
        ↓
TGT and service tickets cached
        ↓
Attacker with local admin/SYSTEM can extract them

2. Attacker Has Local Admin or SYSTEM Access

To dump tickets, the attacker usually needs elevated access on the machine.

Common paths:

Local admin compromise
Credential dumping
Malware running as SYSTEM
EDR bypass
LSASS memory access

3. Ticket Is Still Valid

Kerberos tickets have lifetimes.

If a ticket is expired, it cannot be used unless renewable and still within renewal period.

Example:

TGT lifetime: 10 hours
Renewable lifetime: 7 days

The attacker benefits if the ticket is still valid.

4. Target Service Accepts Kerberos

Pass-the-ticket only works against services that accept Kerberos authentication.

Examples:

CIFS/file shares
LDAP
MSSQL
WinRM
HTTP with Kerberos
Remote service management

5. Network Access Exists

The attacker must be able to reach the target service.

Stolen ticket + no network path = limited usefulness

6. No Strong Credential Protections

The attack is easier when protections are missing, such as:

Credential Guard disabled
LSASS not protected
Excessive local admin rights
Weak endpoint monitoring
No tiering model
Admins logging into low-trust machines

7. Privileged Users Log Into Compromisable Machines

If admins log into standard workstations or servers, their tickets may be exposed.

Example:

Domain Admin logs into infected helpdesk server
        ↓
Admin TGT cached
        ↓
Attacker steals TGT
        ↓
Pass-the-ticket as Domain Admin

Model Answer

Pass-the-ticket is possible when attackers can obtain valid Kerberos tickets from a compromised host, usually by gaining local admin or SYSTEM access and extracting tickets from memory. The attack requires valid, unexpired tickets, network access to Kerberos-enabled services, and insufficient credential protections such as weak endpoint isolation, lack of Credential Guard, and poor admin tiering.

7. How would you detect forged-ticket activity in a large enterprise?

Forged-ticket detection is difficult because forged tickets may appear valid to services.

The best approach is to combine Domain Controller logs, endpoint logs, service logs, and behavioral analytics.

Types of Forged Tickets

Ticket Type Description
Golden Ticket Forged TGT using the krbtgt account hash
Silver Ticket Forged service ticket using a service account hash
Diamond Ticket Legitimate TGT modified or forged with valid structure
Sapphire-style / advanced variants Abuse of PAC or delegation paths depending on environment

Detection Strategy

A. Detect Golden Ticket Indicators

Golden Tickets are forged TGTs. They are created using the krbtgt secret.

Suspicious signs:

TGT used but no matching AS-REQ seen
Unusual ticket lifetime
Impossible user behavior
Old disabled user authenticating
User SID does not match expected attributes
Unusual group membership in PAC
Authentication from unexpected host

Relevant Windows Events

Event ID Meaning
4768 Kerberos Authentication Ticket requested
4769 Kerberos Service Ticket requested
4770 Kerberos Service Ticket renewed
4624 Successful logon
4672 Special privileges assigned
4627 Group membership information
4738 User account changed
4765 / 4766 SID history changes
5136 Directory object modified

Golden Ticket detection often looks for service access without normal prior authentication patterns.

Example logic:

If service ticket activity exists
AND no recent AS-REQ/TGT request exists for that user
AND source host is unusual
THEN investigate possible forged TGT

B. Detect Silver Ticket Indicators

Silver Tickets are forged service tickets. They are encrypted with the service account key and are presented directly to the service.

A key detection challenge:

Silver Tickets may not generate normal Domain Controller TGS request logs.

Why?

Because the attacker may forge the service ticket offline and present it directly to the target service.

Suspicious signs:

Service logon occurred
No matching 4769 TGS request on Domain Controller
Unexpected user accessing specific service
Unusual source host
Service account password recently exposed
PAC validation failures

Example:

File server shows Alice accessed ADMIN$.
Domain Controller has no recent 4769 for Alice requesting cifs/fileserver.
This mismatch is suspicious.

C. Detect Abnormal Ticket Properties

Look for tickets with:

Very long lifetimes
Unusual renewal periods
Weak encryption types
Unexpected client addresses
Unexpected realm/domain
Malformed or unusual PAC data
Privileged group membership not matching AD

Example:

Normal ticket lifetime: 10 hours
Observed ticket lifetime: 10 years
Possible forged ticket

D. Monitor Privileged Group Claims

Forged tickets may include elevated group memberships.

Example:

User: bob.contractor
Actual AD groups:
- Domain Users

Ticket PAC groups:
- Domain Admins
- Enterprise Admins

That mismatch is highly suspicious.

E. Correlate Endpoint and DC Logs

In a large enterprise, detection must be correlation-based.

Useful correlations:

4768 → 4769 → 4624
TGT request → Service ticket request → Service logon

Suspicious pattern:

4624 service logon exists
But no matching 4769 exists

Or:

4769 from host never used by that user before

F. Watch for Kerberos Tooling Behavior

Tools used in attacks may create identifiable patterns.

Examples:

Mimikatz
Rubeus
Impacket
Kekeo

Suspicious activity:

Kerberos ticket injection
Unusual klist behavior
LSASS access
Abnormal Kerberos encryption downgrade
Large number of TGS requests
Requests for many SPNs

G. Monitor Encryption Types

Downgrade to RC4 can be suspicious in modern environments.

Example:

AES normally used
Sudden RC4 service ticket requests appear

This may indicate Kerberoasting or compatibility abuse.

H. Use Honeytokens

Create decoy accounts or SPNs that should never be used.

Example:

Account: svc-fake-backup
SPN: MSSQLSvc/old-backup.company.com
Password: strong and monitored

Any ticket request or login using that identity is suspicious.

Detection Summary Table

Detection Area What to Look For
DC logs 4768, 4769, 4770 anomalies
Service logs Logons without matching DC ticket requests
Endpoint logs LSASS access, ticket injection
AD data PAC group mismatch, disabled users
Network Unusual Kerberos flows
Encryption RC4 or weak encryption usage
Behavior Impossible travel, unusual hosts
Honeytokens Any access to decoy SPNs/accounts

Model Answer

I would detect forged-ticket activity by correlating Domain Controller Kerberos events, endpoint telemetry, and service logons. Golden Tickets often show abnormal ticket lifetimes, unusual group claims, or service access without normal authentication patterns. Silver Tickets are harder because they may bypass DC TGS logging, so I would compare service logons against 4769 events. I would also monitor encryption downgrades, privileged PAC claims, LSASS access, unusual SPN requests, and use honeytoken service accounts.

8. Why do service principal names matter operationally and security-wise?

A Service Principal Name, or SPN, maps a service instance to the account under which that service runs.

Example:

HTTP/app.company.com
MSSQLSvc/sql01.company.com:1433
CIFS/fileserver01.company.com
LDAP/dc01.company.com

Kerberos uses SPNs to know which account’s key should encrypt the service ticket.

Operational Importance

When a client wants to access a service, it requests a ticket for the SPN.

Example:

Client wants SQL access:
MSSQLSvc/sql01.company.com:1433

The KDC checks AD:

Which account owns this SPN?

Then it encrypts the service ticket using that account’s key.

If the SPN is missing or wrong, Kerberos authentication fails.

Common Operational Problems

Problem Result
Missing SPN Kerberos fails; fallback to NTLM may happen
Duplicate SPN KDC does not know which account owns service
Wrong SPN owner Service cannot decrypt ticket
Old/stale SPN Tickets issued to unused or risky accounts
SPN on user account May become Kerberoastable
Misconfigured HTTP SPN Web SSO fails

Example: Missing SPN

Application URL:
https://app.company.com

Expected SPN:
HTTP/app.company.com

But SPN is missing.

Result:

Kerberos authentication fails.
Client may fall back to NTLM.

This is both an availability and security issue.

Security Importance

SPNs matter because accounts with SPNs are Kerberoastable.

Any authenticated user can request a service ticket for an SPN.

If the SPN is tied to a weak service account:

SPN: MSSQLSvc/sql01.company.com
Account: svc-sql
Weak password: Company@123

An attacker can request a ticket and crack it offline.

Duplicate SPN Risk

Duplicate SPNs can cause authentication failures and may lead to confusing troubleshooting.

Example:

HTTP/app.company.com assigned to:
- svc-web-prod
- svc-old-test

The KDC may fail to issue the right ticket.

SPNs and Privilege

SPNs on highly privileged accounts are dangerous.

Example:

Account: domain-admin-svc
SPN: HTTP/legacyapp.company.com
Privileges: Domain Admin

If this account is Kerberoasted and cracked, compromise impact is severe.

Model Answer

SPNs are operationally important because Kerberos uses them to map a requested service to the account key used to encrypt the ticket. Missing, duplicate, or incorrect SPNs break authentication or cause fallback to weaker protocols. Security-wise, SPNs expose accounts to Kerberoasting, so service accounts with SPNs must have strong passwords, minimal privileges, modern encryption, and good lifecycle management.

10. What are the highest-priority hardening steps for a Kerberos-based domain?

The highest-priority hardening steps focus on reducing credential theft, limiting ticket abuse, protecting privileged accounts, and improving service account hygiene.

Priority 1: Protect the krbtgt Account

The krbtgt account is used to sign/encrypt Kerberos TGTs.

If its secret is stolen, attackers can create Golden Tickets.

Hardening actions:

Rotate krbtgt password periodically
Rotate twice after suspected compromise
Monitor access to Domain Controllers
Protect DC backups
Limit who can replicate directory secrets
Monitor DCSync attempts

Why rotate twice?

Because Kerberos keeps current and previous keys for compatibility. A full reset usually requires two rotations with proper spacing.

Priority 2: Enforce Strong Service Account Hygiene

Service accounts are major Kerberoasting targets.

Hardening actions:

Use gMSA where possible
Use long random passwords
Avoid human-managed service account passwords
Remove unnecessary SPNs
Remove stale service accounts
Prevent service accounts from being Domain Admins
Disable interactive logon for service accounts
Use AES instead of RC4

Good service account password example:

Random 120-character managed password via gMSA

Bad service account password example:

Spring2024!
Company@123
PasswordNeverExpires123

Priority 3: Disable or Minimize RC4

RC4-based Kerberos tickets are much easier to crack than AES-based tickets.

Hardening actions:

Disable RC4 where possible
Require AES128/AES256
Update legacy systems that require RC4
Monitor for RC4 ticket requests

Detection example:

Alert when service ticket encryption type = RC4
especially for sensitive service accounts

Priority 4: Reduce Delegation Risk

Delegation misconfiguration is a common AD escalation path.

Hardening actions:

Eliminate unconstrained delegation
Use constrained delegation only where required
Prefer resource-based constrained delegation with controls
Mark privileged users as "Account is sensitive and cannot be delegated"
Put admins in Protected Users group where appropriate
Monitor delegation configuration changes

High-risk setting:

Trust this computer for delegation to any service

This should be rare and heavily monitored.

Priority 5: Implement Administrative Tiering

Prevent privileged credentials from touching low-trust machines.

Example model:

Tier 0: Domain Controllers, identity systems
Tier 1: Servers
Tier 2: Workstations

Rules:

Domain Admins log only into Tier 0 systems
Server admins do not log into workstations with privileged accounts
Workstation compromise should not expose domain admin tickets

This directly reduces pass-the-ticket and credential theft risk.

Priority 6: Enable Credential Protections

Hardening actions:

Enable Windows Defender Credential Guard
Enable LSASS protection
Block credential dumping tools
Restrict local admin rights
Use EDR monitoring
Enable Attack Surface Reduction rules

Goal:

Make ticket extraction from endpoints much harder

Priority 7: Monitor Kerberos Events at Scale

Collect and analyze:

4768 - TGT requested
4769 - Service ticket requested
4770 - Ticket renewed
4771 - Pre-authentication failed
4776 - NTLM authentication
4624 - Successful logon
4672 - Special privileges

Useful alerts:

Many TGS requests by one user
RC4 tickets requested
Service tickets requested for sensitive SPNs
Authentication from unusual hosts
Service logon without matching TGS request
Disabled user authenticating
Long ticket lifetime
Admin account used from workstation

Priority 8: Reduce NTLM Fallback

Kerberos failures often cause fallback to NTLM.

NTLM increases credential relay and pass-the-hash risk.

Hardening actions:

Fix SPN issues
Audit NTLM usage
Disable NTLM where possible
Use SMB signing
Use LDAP signing and channel binding

Priority 9: Secure Domain Controllers

Domain Controllers are the heart of Kerberos.

Hardening actions:

Restrict interactive logon
Patch aggressively
Monitor replication permissions
Protect backups
Use secure admin workstations
Limit installed software
Enable advanced audit logging
Monitor suspicious LSASS access

Priority 10: Use Honeytokens and Deception

Create monitored decoy accounts and SPNs.

Example:

svc-sql-legacy
MSSQLSvc/legacy-db.company.com

No legitimate system should use it.

Alert on:

TGS request for honey SPN
Logon using honey account
LDAP query targeting honey account

Group Managed Service Accounts (gMSA) — Deep Dive

The Problem with Regular Service Accounts

Normal service accounts used by Windows services (IIS, SQL Server, scheduled tasks) have passwords that are:

  • Human-managed: weak, shared across multiple services, rarely rotated
  • Stored in LSA secrets or the registry on the host — extractable with admin privileges
  • Tied to SPNs — any domain user can request a service ticket encrypted with the account's NTLM hash
  • Frequently set to PasswordNeverExpires for "operational convenience"

This makes them the primary Kerberoasting target and a frequent pivot point in intrusions.

What gMSA Is

Group Managed Service Accounts (introduced in Windows Server 2012) are Active Directory objects that:

  • Have their passwords generated and rotated automatically by the domain
  • Allow multiple authorized hosts to retrieve and use the same account (unlike standalone MSA, which is bound to one host)
  • Never require a human to know, set, or manage the password
  • Automatically register SPNs when deployed to authorized hosts
  • Can be used by Windows services, IIS app pools, and scheduled tasks

How gMSA Works — Mechanics

Step 1 — KDS Root Key Setup

Before any gMSA can be created, a Key Distribution Service (KDS) Root Key must exist in the forest:

# Create KDS Root Key (takes 10 hours to replicate by default)
Add-KdsRootKey -EffectiveImmediately   # dev/lab only; use default in prod

The KDS Root Key is a 256-bit secret stored in AD under CN=Master Root Keys,CN=Group Key Distribution Service,CN=Services,CN=Configuration,DC=.... It replicates to all Domain Controllers. No administrator ever sees its raw value after creation.

Step 2 — gMSA Account Creation

New-ADServiceAccount -Name "svc-webapp" `
    -DNSHostName "svc-webapp.corp.local" `
    -PrincipalsAllowedToRetrieveManagedPassword "WebServers-Group"

Step 3 — Password Derivation (no central storage)

The gMSA password is never stored anywhere. It is derived on demand using a deterministic PRF:

gMSA_password = PRF(
    KDS_Root_Key,
    gMSA_account_SID,
    current_time_interval          # floor(unix_time / interval_in_seconds)
)

Any DC can derive the same password independently from the same inputs. This means:

  • No single-point-of-failure password store
  • No replication of the plaintext password
  • The DC returns the derived password to an authorized requester via a DPAPI-protected LDAP blob

Step 4 — Host Retrieval Flow

sequenceDiagram
    participant H as Authorized Host
    participant DC as Domain Controller
    participant KDS as KDS Root Key in AD
    H->>DC: LDAP read of msDS-ManagedPassword for svc-webapp
    DC->>KDS: Derive password from root key + SID + time interval
    KDS-->>DC: Derived 256-bit password material
    DC-->>H: DPAPI-encrypted blob containing password
    H->>H: Decrypt blob using machine credentials
    H->>H: Configure service or app pool with derived password

The host repeats this retrieval automatically when the password interval rolls over — no admin action needed.

Step 5 — Auto-Rotation

Every 30 days (configurable), the time interval increments. The next time an authorized host starts the service or calls Install-ADServiceAccount, it fetches the new password. Old password remains valid for one overlap period to prevent service interruption.

How gMSA Stops Kerberos Attacks

Kerberoasting — Fully Mitigated

Kerberoasting works by cracking the service account's NTLM hash offline. The NTLM hash is derived from the password using MD4:

NTLM_hash = MD4(UTF-16LE(password))
Account type Password entropy Crack feasibility
svc-sql with Winter2022! ~40 bits Seconds on GPU
svc-sql with 25-char random ~148 bits Years
gMSA auto-generated 240 bits Computationally infeasible — beyond all hardware

A 240-bit random password means the attacker would need to try $2^{240}$ candidates — the GPU cluster that cracks normal passwords at 1 billion hashes/second would need longer than the age of the universe.

No password policy, rotation schedule, or cipher upgrade achieves this — only gMSA gives infeasibility by construction.

AS-REP Roasting — Fully Mitigated

AS-REP Roasting requires the DONT_REQUIRE_PREAUTH flag on the account. gMSA accounts:

  • Always require Kerberos pre-authentication
  • Cannot have DONT_REQUIRE_PREAUTH set
  • No human can interactively sign in to a gMSA account, so there is no reason to disable pre-auth

Password Spraying and Credential Stuffing — Fully Mitigated

There is no human-known password to spray or stuff. The password is 240 bits of randomness — it cannot appear in any wordlist or be guessed.

Silver Ticket — Partially Mitigated (Blast Radius Reduced)

A Silver Ticket requires the service account's NTLM hash to forge a service ticket without contacting the KDC.

With a regular service account:

  • Hash extracted from LSA secrets → valid indefinitely until someone manually changes the password

With gMSA:

  • Hash is still in LSASS on hosts that have fetched the password → still extractable by a local admin/SYSTEM attacker
  • BUT: the password rotates every 30 days → any forged Silver Ticket is limited to the current rotation window
  • After rotation, the old hash is invalid and previously forged tickets become unusable

gMSA does not prevent Silver Ticket attacks on a compromised host. It limits the persistence window.

Pass-the-Hash — Same Partial Mitigation

NTLM hashes of gMSA passwords are in LSASS on authorized hosts. An attacker with SYSTEM can still extract them. The mitigation is the same: 30-day rotation windows limit the value of a stolen hash. Credential Guard (VBS-protected LSASS) is the complementary control to prevent extraction.

What gMSA Does NOT Stop

Attack Why gMSA Doesn't Help
Golden Ticket Depends on krbtgt, not the service account
Pass-the-Ticket Ticket theft from LSASS is unrelated to password management
Unconstrained delegation abuse A gMSA can still be configured with unconstrained delegation
DCSync Domain replication rights are unrelated to gMSA
Local host compromise If the host is owned, attacker can call Get-ADServiceAccountPassword

Final Summary

Kerberos is designed so users do not send passwords to every service. Instead, users authenticate to a trusted KDC and receive tickets. The TGT allows the user to request service tickets, while service tickets allow access to specific services. Kerberos depends heavily on time synchronization to prevent replay attacks.

Most real-world Kerberos attacks are not pure protocol breaks. They usually exploit operational weaknesses such as weak service account passwords, bad SPN hygiene, excessive delegation, RC4 support, local admin sprawl, and poor privileged access management.

For enterprise defense, I would prioritize protecting the krbtgt account, eliminating weak service accounts, disabling RC4, reducing delegation, implementing admin tiering, enabling credential protections, monitoring Kerberos events, and correlating service logons with KDC activity.

Key Takeaways

  • Kerberos is a ticket-based mutual authentication system built around a trusted KDC.
  • Endpoint compromise, weak service accounts, and delegation misuse cause many real-world Kerberos incidents.
  • TGTs, service tickets, SPNs, and time synchronization are the core concepts to understand.
  • Strong service credential hygiene and privileged identity protection are essential controls.
  • Kerberos is extremely effective inside managed enterprise environments but operationally demanding.
