OAuth 2.0 - Authorization Framework

Category: Federation and delegated authorization
Primary Standards: RFC 6749, RFC 6750, RFC 6819, RFC 7636, RFC 8707, RFC 9126, RFC 9449

Overview

OAuth 2.0 is a delegated authorization framework that lets a client obtain limited access to a protected resource on behalf of a user or system. It separates the client from the user's credentials by issuing tokens through an authorization server. OAuth 2.0 is widely used for third-party API access, mobile and browser sign-in flows, backend service delegation, and enterprise application integration. The framework is flexible, but that flexibility means security depends heavily on choosing the right grant types, token handling, and client protections.

How It Works

Core Mechanics

The main OAuth actors are:

Actor Role
Resource Owner The user or system that owns the data or capability
Client The app requesting delegated access
Authorization Server Authenticates the user and issues tokens
Resource Server API that validates tokens and serves protected data

Common artifacts:

Artifact Purpose
Authorization code Short-lived code exchanged for tokens
Access token Credential presented to the resource server
Refresh token Long-lived token used to obtain new access tokens
Scope Limits the actions or data the token can access
Redirect URI Where the authorization server returns the user agent

Modern best practice centers on the authorization code flow with PKCE. Public clients such as mobile apps and SPAs should not rely on a client secret as their main protection because the secret cannot be kept confidential.

Step-by-Step Flow

sequenceDiagram
    participant U as User
    participant C as Client
    participant A as Auth Server
    participant R as Resource Server
    U->>C: Start action that needs delegated access
    C->>A: Authorization request with PKCE and redirect URI
    A-->>U: Authenticate and collect consent
    A-->>C: Redirect with authorization code
    C->>A: Exchange code and PKCE verifier for tokens
    A-->>C: Access token and optional refresh token
    C->>R: API request with access token
    R-->>C: Protected resource response

flowchart TD
    A[Client wants API access] --> B[Send user to authorization server]
    B --> C[User authenticates and consents]
    C --> D[Authorization server returns code]
    D --> E[Client exchanges code for token]
    E --> F[Client calls resource server]
    F --> G[Resource server validates token and scope]
    G --> H[Allow or deny request]

OAuth 2.0 Grant Types (Flows)

OAuth 2.0 defines several grant types, each designed for a different client and deployment scenario. Choosing the wrong one is a common source of security vulnerabilities.

Flow Selection Guide

flowchart TD
    A[Who is the initiating party?] --> B[A human user]
    A --> C[A machine or service]
    B --> D[Can the client store a secret securely?]
    D --> E[Yes - server-side app] --> F[Authorization Code]
    D --> G[No - SPA or mobile] --> H[Authorization Code + PKCE]
    B --> I[Device with no browser?] --> J[Device Authorization Flow]
    C --> K[Acting on its own behalf?] --> L[Client Credentials]
    C --> M[Acting on behalf of a user?] --> N[Token Exchange - RFC 8693]

1. Authorization Code Flow

RFC 6749 Section 4.1

The user is redirected to the authorization server, authenticates, and consents. The auth server redirects back with a short-lived code. The client exchanges the code for tokens via a back-channel request using its client secret.

sequenceDiagram
    participant U as User
    participant C as Client (server-side)
    participant A as Auth Server
    U->>C: Initiate login
    C->>A: Redirect with response_type=code, redirect_uri, state
    A-->>U: Login and consent screen
    A-->>C: Redirect with authorization code
    C->>A: POST /token with code, client_secret
    A-->>C: Access token + refresh token

Use cases: Traditional web apps with a secure server-side backend (e.g., Node.js, Django, Spring Boot apps calling third-party APIs).

Pros Cons
Tokens never touch the browser front channel Requires a server to hold the client secret
Back-channel token exchange is harder to intercept Redirect round-trip adds latency
Refresh tokens enable long sessions without re-auth Client secret must be rotated if leaked
Well-understood, widely implemented Not suitable for purely public clients

2. Authorization Code Flow with PKCE

RFC 7636

Same as authorization code flow but replaces the client secret with a cryptographic challenge-verifier pair generated by the client. The client generates a code_verifier, hashes it to a code_challenge, and sends the challenge with the authorization request. The verifier is sent during token exchange. The auth server checks that the verifier matches the earlier challenge.

sequenceDiagram
    participant U as User
    participant C as Public Client (SPA or mobile)
    participant A as Auth Server
    C->>C: Generate code_verifier and code_challenge
    C->>A: Redirect with code_challenge, code_challenge_method=S256
    A-->>U: Login and consent
    A-->>C: Authorization code
    C->>A: POST /token with code + code_verifier
    A->>A: Verify hash(code_verifier) == code_challenge
    A-->>C: Access token + optional refresh token

Use cases: SPAs, mobile apps, CLI tools, desktop apps — any client that cannot securely store a client secret.

Pros Cons
Secure for public clients without a secret Client must implement the verifier generation correctly
Prevents code interception attacks Auth server must support PKCE
Recommended by OAuth 2.1 for all clients Adds one extra step of state management
Works alongside confidential clients too Still needs careful redirect URI validation

3. Client Credentials Flow

RFC 6749 Section 4.4

No user is involved. The client authenticates directly to the authorization server using its own credentials (client ID and secret, or a certificate/assertion) and receives an access token scoped to its own identity.

sequenceDiagram
    participant S as Service / Daemon
    participant A as Auth Server
    participant R as Resource Server
    S->>A: POST /token with client_id, client_secret, grant_type=client_credentials
    A-->>S: Access token (scoped to service identity)
    S->>R: API call with access token
    R-->>S: Response

Use cases: Backend services, daemons, cron jobs, microservice-to-microservice calls, CI/CD pipelines accessing APIs.

Pros Cons
Simple — no user redirect or browser required Secret must be securely stored and rotated
Suitable for fully automated M2M flows No user context in the token
Short-lived tokens reduce risk if secret leaks Client secret compromise = full service impersonation
Works well with mTLS or private key JWT for stronger auth No refresh token; must re-authenticate each cycle

4. Device Authorization Flow (Device Code)

RFC 8628

For devices that cannot open a browser or accept text input (smart TVs, IoT devices, game consoles, CLI tools). The device gets a device code from the auth server and displays a URL plus a short user code. The user visits the URL on a separate device, authenticates, and enters the code. The device polls the auth server until approval is granted.

sequenceDiagram
    participant D as Device or CLI
    participant A as Auth Server
    participant U as User (separate browser)
    D->>A: POST /device_authorization with client_id and scope
    A-->>D: device_code, user_code, verification_uri
    D-->>U: Display verification_uri and user_code
    U->>A: Visit URL, authenticate, enter user_code
    D->>A: Poll POST /token with device_code (interval)
    A-->>D: Access token when user approves

Use cases: Smart TV apps, IoT devices, CLI tools like gcloud auth login, developer toolchains.

Pros Cons
Works on input-constrained and headless devices Polling creates load on the auth server
User authenticates on a trusted secondary device User code can be phished if verification URI is spoofed
No browser required on the device itself Short expiry window for user code requires timely action
Good UX for input-limited scenarios Slightly more complex implementation

5. Refresh Token Flow

RFC 6749 Section 6

Not an initial grant type, but a continuation mechanism. When an access token expires, the client sends the refresh token to the auth server to get a new access token without involving the user again.

sequenceDiagram
    participant C as Client
    participant A as Auth Server
    participant R as Resource Server
    C->>R: API call with expired access token
    R-->>C: 401 Unauthorized
    C->>A: POST /token with grant_type=refresh_token
    A-->>C: New access token (and optionally new refresh token)
    C->>R: Retry API call with new access token
    R-->>C: Response

Use cases: Any long-lived session where re-prompting the user for credentials would be disruptive. Used alongside authorization code flows.

Pros Cons
Enables long sessions without re-authentication Refresh tokens are high-value targets; theft = persistent access
Supports silent re-auth in background Must be rotated and invalidated on use (refresh token rotation)
Access token expiry windows can be kept short Revocation requires coordination with the auth server
Reduces user friction Single-use refresh tokens add implementation complexity

6. Implicit Flow (Deprecated)

RFC 6749 Section 4.2 — NOT recommended for new systems

Access tokens were returned directly in the URL fragment after the authorization redirect. There was no back-channel token exchange. Deprecated because tokens in the URL are exposed in browser history, referrer headers, and server logs.

Use cases (historical): Browser-only JavaScript apps before PKCE existed.

Pros Cons
Simple — no server required Access token in URL fragment leaks to history and logs
Fast — one round trip No refresh token issued
No PKCE equivalent; cannot prevent interception
Deprecated by OAuth 2.0 Security BCP and OAuth 2.1

Do not use. Replace with authorization code + PKCE for all public clients.

7. Resource Owner Password Credentials (ROPC) (Deprecated)

RFC 6749 Section 4.3 — NOT recommended for new systems

The user's username and password are collected directly by the client and sent to the authorization server. This breaks the fundamental OAuth trust model by re-introducing direct credential handling by the client.

Use cases (historical): First-party apps migrating from legacy auth systems. Sometimes still seen in enterprise internal tooling.

Pros Cons
Works without browser redirects Client sees the user's raw credentials
One step — no redirect round trip Bypasses MFA since credentials go direct to client
Phishing risk: users get trained to enter passwords into apps
Deprecated in OAuth 2.1; no path forward for public clients

Do not use. No acceptable replacement within ROPC; use authorization code + PKCE.

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Authorization code interception Missing PKCE or weak redirect handling Token theft and account access High
2 Token replay Bearer tokens are stolen from browser, logs, or storage Unauthorized API access High
3 Redirect URI abuse Overly broad or weakly validated redirect URIs Authorization code leakage High
4 Excessive scopes Client asks for more than it needs Privilege expansion Med
5 Refresh token compromise Long-lived token stored insecurely Persistent account compromise Med
6 Confused deputy Client uses token for wrong resource or audience Cross-service authorization bypass Med
7 Implicit or deprecated flow use Legacy browser patterns keep tokens in front channel Easier exfiltration Med
8 Weak client identity Public client treated like confidential client Broken trust assumptions Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Authorization code interception Require PKCE for all code flows Client and auth server
2 Token replay Use short-lived access tokens and sender-constrained tokens where possible Client, auth server, resource server
3 Redirect URI abuse Enforce exact redirect URI matching Authorization server
4 Excessive scopes Use least-privilege scopes and consent review Client and auth server
5 Refresh token compromise Rotate refresh tokens and bind them to client behavior Auth server
6 Confused deputy Validate audience, issuer, and resource indicators Resource server
7 Implicit or deprecated flow use Standardize on authorization code with PKCE Platform design
8 Weak client identity Distinguish public and confidential clients correctly Auth server and client registration

Top 3 Critical Mitigations

  1. PKCE everywhere: PKCE prevents intercepted authorization codes from being exchanged by an attacker because the attacker lacks the original verifier. It is essential for public clients and increasingly expected for all code flows.
  2. Exact redirect URI validation: Redirect handling is one of the highest-risk parts of OAuth. Exact matching is stricter than prefix or pattern matching and removes many code-leak paths.
  3. Short-lived and well-scoped tokens: Small scopes and short expiry windows reduce the damage from bearer token theft. The trade-off is more token refresh activity and more operational complexity.

1. OAuth 2.0 is authorization, not authentication

OAuth 2.0 answers this question:

“Can this client application access this protected resource on behalf of this user, with these permissions?”

It does not directly answer:

“Who is the user, and how was the user authenticated?”

That distinction matters.

In OAuth 2.0, the core artifact is the access token. The access token is meant for the resource server, not necessarily for the client to learn user identity. OAuth 2.0 defines roles like resource owner, client, authorization server, and resource server, and defines how access tokens are issued and used. It does not standardize user identity claims, login session semantics, identity proofing, or user profile claims. That is why OpenID Connect was built on top of OAuth 2.0 to provide authentication and identity through an ID token. OAuth 2.0 itself is the authorization framework. (IETF Datatracker)

A strong answer:

OAuth 2.0 delegates authorization. OpenID Connect performs authentication. If an application uses an OAuth access token as proof that “the user is logged in,” it is mixing protocol layers and may create security bugs.

Example:

A third-party calendar app asks for permission to read my Google Calendar.

OAuth answers:

Can app calendar-helper.example access user Maha's calendar.read scope?

It does not inherently answer:

Is this really Maha?
What assurance level was used?
Was MFA used?
What is Maha's verified email?

That identity layer comes from OpenID Connect, enterprise IdP claims, session state, and authentication policy.

2. Why the industry moved away from implicit flow to authorization code + PKCE

The old implicit flow returned the access token directly through the browser redirect. Historically, this was used for JavaScript single-page apps because browsers could not safely hold a client secret.

Simplified implicit flow:

Browser client → Authorization Server
Authorization Server → Browser redirect with access_token
Browser client uses access_token directly

The problem is that the access token is exposed to the browser environment. It can leak through browser history, malicious JavaScript, extensions, referrer behavior, logs, compromised redirect pages, or debugging tools.

Authorization code + PKCE changed the model.

Modern flow:

Client generates code_verifier
Client derives code_challenge
Client sends user to authorization endpoint with code_challenge
Authorization server returns short-lived authorization code
Client exchanges code + code_verifier at token endpoint
Authorization server verifies code_verifier matches original challenge
Access token is returned from token endpoint

PKCE prevents an attacker who steals only the authorization code from redeeming it, because the attacker does not have the original code_verifier. OAuth 2.0 Security Best Current Practice, now RFC 9700, deprecates less secure modes and recommends stronger patterns based on practical security experience since the original OAuth 2.0 specification. (IETF Datatracker)

Phrasing:

The ecosystem moved away from implicit because browsers became hostile execution environments. Authorization code + PKCE reduces front-channel token exposure and binds the authorization code to the client instance that initiated the flow.

Attack example:

1. User logs into legitimate SPA.
2. Authorization server redirects with access_token in URL fragment.
3. Malicious browser extension reads the URL.
4. Extension exfiltrates token.
5. Attacker calls API as the user.

With authorization code + PKCE:

1. Attacker steals authorization code.
2. Attacker tries to redeem it.
3. Token endpoint asks for code_verifier.
4. Attacker does not have it.
5. Token exchange fails.

3. Why redirect URI validation is security-critical

Redirect URI validation is one of the highest-risk parts of OAuth.

The authorization server eventually sends an authorization code or token to the client’s redirect URI. If an attacker can manipulate that redirect URI, the authorization server may send the code or token to the attacker.

RFC 6749 requires the authorization server to ensure the redirect URI used to obtain the authorization code is identical to the one used during token exchange, and requires public clients to register redirect URIs. (IETF Datatracker)

Bad validation examples:

Allowed:
https://app.example.com/callback

Vulnerable accepted redirect:
https://app.example.com.evil.com/callback
https://evil.com/callback?next=https://app.example.com
https://app.example.com/callback/../../evil
https://app.example.com/callback?redirect=https://evil.com

Strong validation:

Exact match:
https://app.example.com/oauth/callback

Reject:
wildcards
substring matching
open redirects
unregistered subdomains
HTTP except localhost dev cases
dynamic redirect_uri from request parameters

Answer:

Redirect URI validation is security-critical because OAuth uses browser redirects as a delivery channel for authorization grants. If redirect validation is weak, the authorization server becomes a token/code delivery service for the attacker.

Real attack:

Attacker registers malicious client or abuses weak redirect matching:
https://trusted-client.example.com/redirect?next=https://evil.example

Authorization server accepts it because domain starts with trusted-client.example.com.

User authorizes access.

Authorization code is sent to trusted-client open redirect.

Open redirect forwards code to evil.example.

Attacker redeems code and gets access token.

Controls:

- Exact redirect URI matching
- No wildcard redirects for sensitive clients
- No open redirect endpoints as OAuth callbacks
- Per-client redirect registration
- Bind authorization code to client_id and redirect_uri
- Short-lived authorization codes
- PKCE for public clients
- Use state/nonce to prevent CSRF and mix-up attacks

4. Scopes are not the same as real authorization decisions

Scopes are coarse permissions granted by the authorization server.

Example:

scope = "calendar.read"
scope = "payments.write"
scope = "profile.email"

But scopes do not fully determine whether the request should succeed.

The resource server still needs to make contextual authorization decisions:

Does this token belong to this user?
Can this user access this specific object?
Can this client act on this tenant?
Is the operation allowed under policy?
Is the account suspended?
Is step-up authentication required?
Is the transaction amount too high?

Example:

A token has:

scope: documents.read
sub: user_123
client_id: thirdparty_app_456
tenant_id: acme

Request:

GET /tenants/beta/documents/doc_999
Authorization: Bearer eyJ...

The resource server must not say:

Token has documents.read, allow.

It must check:

Is user_123 a member of tenant beta?
Does user_123 have access to doc_999?
Is client_id thirdparty_app_456 allowed for tenant beta?
Was this token issued for this API audience?

Phrasing:

Scopes are authorization hints or delegated permission boundaries. They are necessary but not sufficient. The resource server owns the final authorization decision because it understands object ownership, tenant boundaries, business rules, and risk context.

Common weak answer:

“If the token has the scope, allow the request.”

Better answer:

“Scope is one input. Final authorization requires subject, audience, tenant, resource ownership, policy, operation, environment, and risk checks.”

5. Bearer tokens are risky by default

A bearer token means:

Whoever possesses the token can use it.

There is no built-in proof that the caller is the legitimate client. RFC 6750 describes bearer token usage and states the core risk clearly: any party in possession of the bearer token can use it to access protected resources. (self-issued.info)

Bearer token risks:

- Token stolen from browser storage
- Token leaked in logs
- Token leaked through URL/query string
- Token captured by malicious mobile app instrumentation
- Token copied from developer machine
- Token exposed in crash reports
- Token replayed from another IP/device
- Token used against wrong audience if validation is weak

Example:

GET /api/payments
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

If an attacker obtains that string, they can replay it unless additional controls exist.

Controls:

- Short-lived access tokens
- Never put tokens in URLs
- Use TLS everywhere
- Store tokens securely
- Use httpOnly, Secure, SameSite cookies where appropriate
- Validate audience, issuer, expiry, signature
- Use sender-constrained tokens where possible
- DPoP or mTLS for high-risk clients
- Token replay detection
- Minimize scopes

DPoP is one modern mechanism for sender-constraining OAuth tokens using proof of possession, helping detect replay attacks involving access and refresh tokens. (RFC Editor)

6. Refresh token rotation for mobile and browser clients

Refresh tokens are dangerous because they are long-lived. If stolen, they can mint new access tokens.

A good design uses refresh token rotation:

1. Client receives refresh_token_1.
2. Client uses refresh_token_1 to get new access token.
3. Authorization server returns access_token_2 and refresh_token_2.
4. refresh_token_1 is invalidated.
5. If refresh_token_1 is used again, detect reuse.
6. Reuse suggests token theft.
7. Revoke the token family and force re-authentication.

Token family model:

family_id: fam_abc123

refresh_token_1 → used → replaced by refresh_token_2
refresh_token_2 → used → replaced by refresh_token_3
refresh_token_1 reused later → suspicious reuse → revoke entire family

Mobile client design

For mobile:

- Use authorization code + PKCE
- Store refresh token in platform secure storage
  - iOS Keychain
  - Android Keystore-backed storage
- Rotate refresh token on every use
- Bind token to client instance where possible
- Use device identifiers carefully; avoid privacy-invasive fingerprinting
- Detect impossible travel and abnormal token use
- Consider DPoP or mTLS for high-risk enterprise mobile apps
- Require re-authentication after long inactivity or risk event

Mobile nuance:

Mobile apps are public clients. A client secret embedded in the app is not a real secret. Reverse engineering can extract it. So PKCE and runtime controls matter more than static client secrets.

Browser client design

For browser apps:

Preferred pattern for high-security apps:

Browser → Backend-for-Frontend → Authorization Server
Browser stores session cookie
Backend stores refresh token
Browser never sees refresh token

This is often called a BFF pattern.

For pure SPA:

- Authorization code + PKCE
- Avoid long-lived refresh tokens in localStorage
- Prefer in-memory access token storage
- Use refresh token rotation if refresh tokens are issued
- Use httpOnly Secure SameSite cookies where architecture allows
- Strong CSP
- XSS prevention
- Token reuse detection
- Short idle timeout

Critical point:

Refresh token rotation does not prevent the initial theft. It limits persistence and gives the authorization server a way to detect replay.

Attack scenario:

1. XSS steals refresh_token_5 from browser.
2. Legitimate browser uses refresh_token_5 and receives refresh_token_6.
3. Attacker later uses refresh_token_5.
4. Authorization server detects reuse.
5. Server revokes token family.
6. User must re-authenticate.

7. Confused deputy problem in OAuth-style systems

The confused deputy problem occurs when a more privileged component is tricked into using its authority on behalf of the wrong party.

In OAuth, this can happen when the authorization server, client, or resource server confuses:

Who is the user?
Who is the client?
Which resource server is the token intended for?
Which tenant does this request belong to?
Which redirect flow initiated this response?

Example: OAuth mix-up/confused deputy

1. Client supports multiple identity providers.
2. Attacker starts flow with malicious authorization server.
3. Client receives code and sends it to the wrong token endpoint.
4. Client or authorization server confuses issuer/client context.
5. Attacker gains token or causes login/session confusion.

Another example: API confused deputy

1. Third-party app has access to user A's files.
2. Attacker causes app to call API for user B's file.
3. Resource server only checks app-level scope, not user/resource binding.
4. App becomes a deputy that misuses its access.

Controls:

- Validate issuer
- Validate audience
- Bind authorization code to client_id and redirect_uri
- Use state and nonce
- Use PKCE
- Validate tenant context
- Enforce object-level authorization
- Avoid shared callback confusion across IdPs
- Use explicit resource indicators where needed
- Do not rely only on client_id or scope

Phrasing:

In OAuth systems, confused deputy bugs usually come from losing binding between subject, client, issuer, audience, redirect URI, tenant, and resource. Strong OAuth implementations preserve those bindings across the whole flow.

8. How a resource server should validate an access token

Before trusting an access token, the resource server should validate both token integrity and authorization semantics.

For JWT/self-contained access tokens

Checklist:

1. Extract token only from approved location
   - Usually Authorization: Bearer
   - Avoid query string tokens

2. Verify signature
   - Use trusted JWKS from expected issuer
   - Pin allowed algorithms
   - Reject alg=none
   - Do not allow algorithm confusion

3. Validate issuer
   - iss == expected authorization server

4. Validate audience
   - aud includes this API
   - Prevent token replay across services

5. Validate expiration
   - exp not expired
   - nbf not in future
   - iat reasonable

6. Validate client/application
   - azp/client_id if relevant
   - Is this client allowed to call this API?

7. Validate subject and tenant
   - sub present
   - tenant/org claim trusted
   - tenant matches requested resource

8. Validate scopes/permissions
   - Required scope present
   - Scope sufficient for operation

9. Apply resource-level authorization
   - Object ownership
   - RBAC/ABAC/policy checks

10. Check revocation/risk where needed
   - Token version
   - Session status
   - User disabled
   - Client disabled
   - High-risk transaction controls

Example:

GET /v1/tenants/acme/payments/pay_123
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

Resource server checks:

iss == https://auth.example.com
aud == payments-api
exp > now
scope contains payments.read
tenant_id == acme
sub has access to pay_123
client_id is allowed for payments API
token not revoked or session not terminated

Bad validation:

Signature valid → allow.

Good validation:

Signature valid + issuer valid + audience valid + scope valid + tenant valid + object authorization valid + risk acceptable.

9. Token introspection vs self-contained tokens

There are two common access token models.

Self-contained token

Usually JWT.

Resource server validates token locally.

Pros:

- Fast
- No network call per request
- Good for high-scale APIs
- Works well in distributed systems

Cons:

- Harder to revoke immediately
- Claims may become stale
- Larger token size
- Key rotation complexity
- Every resource server must validate correctly

Introspection token

Usually opaque token.

Resource server asks authorization server:
"Is this token active, and what does it represent?"

OAuth token introspection is standardized in RFC 7662, while JWT access token profiling is covered separately in RFC 9068. (Wikipedia)

Pros:

- Centralized validation
- Easier immediate revocation
- Smaller tokens
- Claims not exposed to clients
- Good for high-risk environments

Cons:

- Adds latency
- Adds dependency on authorization server availability
- Requires caching strategy
- Can become a bottleneck

Use introspection when:

- Immediate revocation matters
- Tokens are opaque
- Resource servers are not trusted to validate JWTs correctly
- Authorization state changes frequently
- Regulatory or enterprise controls require central policy checks
- You want to avoid exposing claims to clients

Use self-contained JWTs when:

- Very high scale
- Low latency needed
- Short token lifetime acceptable
- Revocation latency is tolerable
- Resource servers have mature validation libraries and controls

Hybrid model:

- Short-lived JWT access tokens
- Long-lived refresh tokens managed centrally
- Introspection for high-risk APIs
- Event-driven revocation for major events
- Token version claim for coarse invalidation

10. Hardening OAuth for thousands of third-party integrations

At Google-scale or large-enterprise scale, the main challenge is not just protocol correctness. It is ecosystem governance.

You need to assume:

- Some third-party apps are poorly built
- Some clients will leak secrets
- Some redirect URIs will be misconfigured
- Some integrations will request excessive scopes
- Some partners will get compromised
- Some users will consent without understanding
- Some tokens will leak into logs or CI systems

A strong hardening design

A. Client registration governance

- Verified developer accounts
- App ownership verification
- Domain ownership verification
- Production vs test apps separated
- Risk-tiered client approval
- Security review for sensitive scopes
- App attestation for mobile where appropriate
- No shared clients across unrelated apps

B. Redirect URI controls

- Exact redirect URI matching
- HTTPS only except localhost development
- No wildcards for production clients
- No open redirect callback endpoints
- Limit number of redirect URIs per client
- Monitor redirect URI changes
- Require re-review for risky redirect changes

C. Secure grant types

- Authorization code + PKCE as default
- No implicit flow
- No resource owner password credentials grant
- Client credentials only for machine-to-machine use
- Device code flow only with anti-phishing controls
- mTLS or private_key_jwt for confidential high-risk clients

RFC 9700 captures current OAuth security best practices and deprecates modes considered less secure or insecure based on modern deployment experience. (IETF Datatracker)

D. Token design

Access tokens:
- Short-lived
- Audience-restricted
- Scope-minimized
- Issuer-bound
- Tenant-aware
- Sender-constrained for high-risk clients

Refresh tokens:
- Rotation
- Reuse detection
- Family revocation
- Device/client binding
- Idle and absolute lifetime
- Risk-based reauthentication

E. Scope governance

- Fine-grained scopes
- Sensitive scopes require review
- Incremental authorization
- Least privilege defaults
- Separate read/write/admin scopes
- User-friendly consent screen
- Admin controls for enterprise tenants

Example:

Bad scopes:

scope: full_access

Better scopes:

scope:
  drive.metadata.read
  drive.file.read
  drive.file.write
  drive.admin.audit.read

F. Resource server enforcement

- Validate iss, aud, exp, nbf, client_id, tenant_id
- Enforce object-level authorization
- Do not trust scopes alone
- Central authorization library or sidecar
- Consistent policy decision points
- Deny by default

G. Monitoring and abuse detection

Signals:

- Refresh token reuse
- Token use from impossible geography
- Sudden scope expansion
- High-volume API calls from new client
- Many users consenting to new risky app
- Redirect URI changes followed by token spikes
- Calls from unexpected ASN/cloud provider
- Token used against wrong resource server
- Excessive 401/403 patterns
- Suspicious consent phishing campaigns

H. Partner and third-party risk controls

- App verification
- Security questionnaire for sensitive scopes
- Penetration test evidence for high-risk integrations
- Breach notification obligations
- Rate limits and quotas
- Per-client kill switch
- Per-scope kill switch
- Per-tenant allow/deny lists

I. User and admin controls

- Consent transparency
- Show app name, publisher, domain, scopes
- Allow users to revoke app access
- Enterprise admin approval workflow
- Tenant-level app allowlisting
- Risk warnings for unverified apps

J. Incident response readiness

- Revoke one token
- Revoke token family
- Revoke all tokens for client
- Disable client
- Disable scope
- Disable redirect URI
- Notify affected users/admins
- Audit all token exchanges and API calls

Phrasing:

For thousands of third-party integrations, OAuth hardening becomes a control-plane problem. I would build strong defaults, strict redirect validation, PKCE everywhere, sensitive-scope review, short-lived audience-bound tokens, refresh token rotation, resource-server authorization enforcement, abuse detection, and fast revocation mechanisms.

Senior-level summary answer

OAuth 2.0 is an authorization delegation framework, not an authentication protocol, because its primary output is an access token for resource access, not a standardized identity assertion. The industry moved from implicit flow to authorization code + PKCE because implicit exposed tokens in the browser front channel, while PKCE protects authorization code redemption for public clients. Redirect URI validation is critical because weak validation lets attackers receive codes or tokens. Scopes define delegated permission boundaries, but resource servers must still enforce real authorization using subject, audience, tenant, object ownership, and policy. Bearer tokens are risky because possession equals access, so they need short lifetimes, strict validation, secure storage, and ideally sender-constraining for high-risk use cases. At scale, OAuth security becomes an ecosystem governance problem involving client registration, redirect validation, scope governance, token lifecycle, resource-server enforcement, monitoring, and rapid revocation.

Key Takeaways

  • OAuth 2.0 delegates access without exposing the user's primary credentials to the client.
  • The safest mainstream pattern is authorization code flow with PKCE.
  • Redirect URI validation, token handling, and scope design are the main security control points.
  • OAuth tokens are not automatically proof of identity; they are credentials for delegated access.
  • Resource servers still need local authorization logic even when scopes exist.
