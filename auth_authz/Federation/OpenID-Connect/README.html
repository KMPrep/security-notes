OpenID Connect - Identity Layer On OAuth 2.0

Category: Federation and authentication
Primary Standards: OpenID Connect Core 1.0, Discovery 1.0, Session Management 1.0, RFC 6749, RFC 7636

Overview

OpenID Connect, or OIDC, is an identity layer built on top of OAuth 2.0. It adds authentication semantics, an ID token, standard user info retrieval, issuer metadata discovery, and defined claims so a client can verify who the user is. OIDC is widely used for web and mobile sign-in, enterprise SSO, and consumer identity federation. It is the standard answer when a system needs both delegated access and federated user authentication in modern web architectures.

How It Works

Core Mechanics

OIDC adds these concepts on top of OAuth:

Artifact Purpose
ID token Signed token that conveys authentication event details and user identity claims
openid scope Signals that the request is an OIDC request
UserInfo endpoint Optional endpoint for additional claims
Discovery document Publishes issuer metadata, endpoints, and key locations
Nonce Defends against replay of front-channel authentication responses

Trust boundaries matter:

Component Trust role
OpenID Provider Authenticates the user and issues identity artifacts
Relying Party Consumes ID token and establishes local session
Browser or app Carries redirects and sometimes tokens

The relying party must validate the ID token carefully, including issuer, audience, expiry, and nonce where relevant. OIDC is often misused when implementers treat any JWT from any issuer as a valid sign-in proof.

Step-by-Step Flow

sequenceDiagram
    participant U as User
    participant RP as Relying Party
    participant OP as OpenID Provider
    U->>RP: Start sign-in
    RP->>OP: Auth request with openid scope and nonce
    OP-->>U: Authenticate user
    OP-->>RP: Redirect with authorization code
    RP->>OP: Exchange code for tokens
    OP-->>RP: ID token and access token
    RP->>RP: Validate ID token claims and signature
    RP-->>U: Establish local session

flowchart TD
    A[Receive ID token] --> B[Validate signature with issuer keys]
    B --> C[Check issuer and audience]
    C --> D[Check expiry and issued-at window]
    D --> E[Check nonce if front-channel flow used]
    E --> F[Map claims to local account]
    F --> G[Create local session]

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Accepting untrusted issuer Weak issuer validation Account takeover via token substitution High
2 ID token replay Nonce or session binding missing Session hijack Med
3 Confusing access token and ID token Token type semantics are misunderstood Improper trust and broken auth logic High
4 Claim over-trust App trusts email or role claims without policy checks Authorization bypass Med
5 Discovery abuse Dynamic issuer or key selection is unsafe Signing key confusion Med
6 Weak logout assumptions Local session remains after upstream logout Stale sessions Med
7 Front-channel leakage Tokens or codes leak through browser artifacts Credential disclosure Med
8 Federation misconfiguration Tenant or audience boundaries are too broad Cross-tenant access Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Accepting untrusted issuer Pin trusted issuers and validate issuer exactly Relying party
2 ID token replay Use nonce and bind auth response to the user session Relying party
3 Confusing token types Keep separate validation paths for ID and access tokens Client and backend
4 Claim over-trust Treat claims as inputs to policy, not the entire policy Application authorization layer
5 Discovery abuse Restrict dynamic registration and metadata trust Federation configuration
6 Weak logout assumptions Manage local session invalidation explicitly Application session layer
7 Front-channel leakage Use code flow with PKCE and avoid exposing tokens in browser URLs Client design
8 Federation misconfiguration Enforce tenant, audience, and client registration boundaries Identity platform

Top 3 Critical Mitigations

  1. Strict issuer and audience validation: The relying party must know exactly which provider it trusts and for which client ID the ID token was minted. This is the core protection against token substitution.
  2. Code flow with PKCE plus nonce: PKCE protects the code exchange while nonce protects the front-channel authentication response against replay. Together they cover the most common browser and app deployment risks.
  3. Separate authentication from authorization: An ID token proves an authentication event, not complete authorization. Applications still need local policy enforcement for roles, entitlements, and tenant boundaries.

OIDC Model Answers — Senior Security Engineer Style

0. Mental Model: OAuth 2.0 vs OIDC

OAuth 2.0 is primarily an authorization framework: it lets a client obtain an access token to call protected resources on behalf of a user or itself. It does not, by itself, standardize user login identity, user profile claims, or an authentication token for the client. RFC 6749 describes OAuth as introducing an authorization layer and issuing access tokens with scope, lifetime, and access attributes. ([IETF Datatracker][1])

OpenID Connect adds an identity layer on top of OAuth 2.0. Its key addition is the ID token, a JWT containing claims about the authentication event and the end user. The OIDC core spec says the ID token is the primary extension OIDC makes to OAuth 2.0 to enable end-user authentication. (OpenID Foundation)

flowchart LR
    User[End User]
    RP[Relying Party / Client App]
    OP[OpenID Provider / IdP]
    API[Resource Server / API]

    User --> RP
    RP -->|OIDC Auth Request| OP
    OP -->|ID Token + Access Token| RP
    RP -->|Access Token| API

    IDT[ID Token: Who authenticated?]
    AT[Access Token: What API access is allowed?]

    OP -.issues.-> IDT
    OP -.issues.-> AT

1. What does OIDC add that plain OAuth 2.0 does not provide?

Core answer

OIDC adds standardized authentication on top of OAuth 2.0. Plain OAuth tells a client how to get an access token, but it does not reliably answer:

  • Who is the user?
  • Which identity provider authenticated the user?
  • When did authentication happen?
  • Which client was the token intended for?
  • How can the client verify the authentication assertion?
  • Where can the client retrieve standard user profile claims?

OIDC answers these by adding:

OIDC Addition What It Provides
openid scope Signals that the client is requesting an OIDC authentication flow.
ID token Signed JWT containing authentication and identity claims.
Standard claims iss, sub, aud, exp, iat, auth_time, acr, amr, email, email_verified, etc.
UserInfo endpoint Standard endpoint for retrieving claims about the authenticated user.
Discovery metadata Standard .well-known/openid-configuration endpoint for endpoints, supported algorithms, scopes, and key locations.
JWKS endpoint Public keys used to validate signed tokens.
Session/logout specs Optional mechanisms for RP-initiated logout, front-channel logout, back-channel logout, and session management.

OIDC’s abstract flow is: RP sends authentication request to OP, OP authenticates user, OP returns ID token and usually an access token, and the RP can call UserInfo with the access token to retrieve claims. (OpenID Foundation)

Real-world example

OAuth-only example:

“Allow this calendar app to read my Google Calendar.”

The client needs authorization to access an API.

OIDC example:

“Sign in to this enterprise app using Google Workspace.”

The app needs to know who the user is, which IdP authenticated them, whether the token was meant for this app, and whether the assertion is trustworthy.

In brief

OAuth answers “Can this client access this resource?” OIDC answers “Who authenticated, through which issuer, for which client, and with what identity claims?”

2. Why is an ID token not a replacement for a session policy engine?

Core answer

An ID token is an authentication assertion, not a full runtime authorization or session-control system.

An ID token can tell the application:

  • The user authenticated.
  • The issuer was a trusted IdP.
  • The token was intended for this client.
  • The token has not expired.
  • Some authentication context may have been asserted.

But a session policy engine decides things like:

  • Is this device still compliant?
  • Has the user been disabled after login?
  • Has the user changed role?
  • Is the request coming from a risky country?
  • Is the action high risk and requires step-up MFA?
  • Has the session exceeded inactivity timeout?
  • Has the token been revoked?
  • Is this API action allowed under current business policy?
  • Has DLP/risk/UEBA flagged the user?

The OIDC spec explicitly notes that ID token expiration is unrelated to the lifetime of the authenticated session between the RP and OP. (OpenID Foundation)

Bad design

flowchart TD
    A[User logs in] --> B[App receives ID token]
    B --> C[App stores token]
    C --> D[Every request allowed until exp]
    D --> E[No device/risk/entitlement check]

Problem: if the user is disabled, device becomes compromised, or admin removes entitlements, the app may continue granting access until token/session expiry.

Better design

flowchart TD
    A[OIDC Login] --> B[Validate ID Token]
    B --> C[Create App Session]
    C --> D[Session Policy Engine]
    D --> E{Request Context}
    E -->|Low Risk| F[Allow]
    E -->|High Risk| G[Step-up MFA]
    E -->|User Disabled / Role Removed| H[Deny + Invalidate Session]

Strong answer

An ID token should be used to bootstrap a local application session, not replace session governance. After login, the application should issue its own secure session cookie and continuously evaluate internal policy.

3. What claims must a relying party validate before trusting an ID token?

Core answer

A relying party must validate both the cryptographic integrity and the semantic meaning of the ID token.

The OIDC validation rules require the RP to validate issuer, audience, signature/key source, expiry, nonce when used, and related claims such as azp, auth_time, and acr where applicable. (OpenID Foundation)

Validation checklist

Check Why It Matters
Signature Confirms token was issued by the IdP and not modified.
iss Must exactly match the trusted issuer. Prevents accepting tokens from rogue IdPs.
aud Must include this RP’s client_id. Prevents token substitution from another client.
azp If present, verify authorized party, especially when token has multiple audiences.
exp Token must not be expired.
iat Reject tokens issued too far in the past or future.
nonce Must match the nonce sent in the auth request if one was used. Prevents replay/mix-up attacks.
auth_time Useful when enforcing recent login or step-up.
acr / amr Validate assurance level or MFA method when required.
sub Stable subject identifier within issuer. Use with iss as the user key.
email_verified Only means the IdP claims email verification; do not treat it as strong identity proof.
alg Must be an expected algorithm. Do not accept none or algorithm confusion.
kid + JWKS Key must come from the trusted issuer’s JWKS endpoint.

The OIDC spec states that iss, sub, aud, exp, and iat are required ID token claims. (OpenID Foundation) It also says iss and sub together are the only stable identifier an RP can rely on for an end user; claims like email, phone number, preferred username, and name must not be used as unique identifiers. (OpenID Foundation)

Common pitfall

Do not say:

“We decode the JWT and read the email.”

Correct answer:

“We verify the signature, issuer, audience, expiry, nonce, algorithm, and key source first. Then we map iss + sub to an internal user record. Email is an attribute, not the principal key.”

4. Why are nonce and state both needed in many browser sign-in flows?

Core answer

state and nonce protect different parts of the flow.

Parameter Protects Main Threat
state Authorization response callback CSRF, login CSRF, wrong browser session binding
nonce ID token Replay, token injection, mix-up involving ID token

OAuth’s state parameter is an opaque value used to maintain state between request and callback and should be used to prevent CSRF. ([IETF Datatracker][1]) RFC 6749 describes how CSRF against the client redirect URI can let an attacker inject their own authorization code or access token into a victim’s client session; the client should use a non-guessable binding value delivered through state. ([IETF Datatracker][1])

OIDC’s nonce is checked inside the returned ID token. If a nonce was sent in the authentication request, the ID token must contain the same nonce, and the client should check it for replay attacks. (OpenID Foundation)

Attack example: missing state

sequenceDiagram
    participant Attacker
    participant VictimBrowser
    participant RP as Relying Party
    participant OP as IdP

    Attacker->>OP: Start login as attacker
    OP-->>Attacker: Authorization code for attacker account
    Attacker->>VictimBrowser: Trick victim into visiting RP callback with attacker's code
    VictimBrowser->>RP: /callback?code=attacker_code
    RP->>OP: Exchange code
    OP-->>RP: Tokens for attacker
    RP-->>VictimBrowser: Victim now linked to attacker's account

state prevents this by binding the callback to the browser session that initiated the login.

Attack example: missing nonce

An attacker who obtains or replays an old ID token may try to inject it into a browser flow. nonce binds the ID token to a specific authentication request.

Strong answer

Use both:

  • state = “Is this callback for the login transaction I started?”
  • nonce = “Is this ID token for the authentication request I started?”

5. How can a discovery endpoint become a trust-boundary problem?

Core answer

OIDC Discovery is convenient, but it turns metadata into a security control plane.

The discovery document tells the RP where to send users, where to exchange codes, where to fetch UserInfo, and which JWKS URI contains signing keys. Discovery metadata includes issuer, authorization_endpoint, token_endpoint, userinfo_endpoint, and jwks_uri; the JWKS contains keys the RP uses to validate OP signatures. (OpenID Foundation)

If an attacker can influence discovery, they may redirect the RP to attacker-controlled:

  • Authorization endpoint
  • Token endpoint
  • JWKS endpoint
  • UserInfo endpoint
  • Logout endpoint
  • Dynamic registration endpoint

Dangerous design

flowchart TD
    A[User enters email domain] --> B[RP discovers issuer dynamically]
    B --> C[RP fetches discovery document]
    C --> D[RP trusts endpoints and JWKS blindly]
    D --> E[Attacker-controlled IdP signs token]
    E --> F[RP accepts fake login]

Trust-boundary risks

Risk Example
Issuer confusion RP accepts iss from discovery without comparing expected issuer.
JWKS injection Discovery points jwks_uri to attacker keys.
SSRF RP fetches discovery/JWKS from internal or link-local addresses.
Tenant confusion company.com.evil.com or path-based issuer trick.
Algorithm downgrade Discovery advertises weak or unexpected algorithms.
Dynamic registration abuse Attacker registers a malicious client or redirect URI.
Cache poisoning RP caches attacker metadata for a legitimate issuer.

The discovery spec requires the issuer value returned in configuration to be identical to the issuer URL used to retrieve the configuration, and this must match the iss claim in ID tokens. If validation fails, operations requiring that information must be aborted. (OpenID Foundation)

Secure design

  • Prefer preconfigured issuer allowlists for enterprise apps.
  • Pin expected issuer per tenant.
  • Validate discovery over TLS.
  • Block private IPs and metadata service IPs for metadata/JWKS fetches.
  • Validate issuer exactly.
  • Cache JWKS safely with TTL and key rotation handling.
  • Reject unexpected algorithms.
  • Separate dev/test/prod IdPs.
  • Log issuer, client ID, kid, JWKS URI, and metadata version.

6. What are the risks of dynamic multi-tenant OIDC sign-in?

Core answer

Dynamic multi-tenant OIDC sign-in means the RP accepts many issuers or discovers issuers based on user input, email domain, tenant name, or an iss parameter. This is powerful but risky because authentication becomes a federation supply-chain problem.

Key risks

Risk Explanation
Rogue tenant onboarding Attacker registers their IdP and gets accepted as a valid enterprise tenant.
Email-domain confusion App assumes user@company.com from any IdP proves company membership.
Issuer mix-up RP sends request to one issuer but accepts token from another.
Account takeover through linking Existing local account gets linked to attacker-controlled federated identity.
Weak IdP assurance Some tenants may allow password-only auth while enterprise app expects phishing-resistant MFA.
Claim injection Attacker-controlled IdP sends groups: admin.
Over-trusting email email_verified=true from an untrusted issuer is not enterprise identity assurance.
Discovery poisoning Dynamic discovery returns attacker-controlled endpoints and keys.
Open redirect abuse Dynamic redirect handling or target_link_uri mishandling turns the RP into a redirector.

OIDC supports third-party initiated login where an iss parameter can tell the RP which OP to use; the same section warns that target link URIs must be verified to prevent open redirect behavior. (OpenID Foundation)

Secure multi-tenant model

flowchart TD
    A[Login Request] --> B[Resolve Tenant]
    B --> C{Tenant Registered?}
    C -->|No| D[Deny]
    C -->|Yes| E[Load Tenant OIDC Config]
    E --> F[Expected Issuer + Client ID + JWKS]
    F --> G[Perform OIDC Flow]
    G --> H[Validate ID Token]
    H --> I[Map iss+sub to Internal User]
    I --> J[Evaluate Entitlements]
    J --> K[Create Session]

Controls

  • Tenant onboarding must be admin-approved.
  • Each tenant has explicit issuer, client ID, redirect URI, allowed algorithms, expected acr, and claim mapping.
  • Do not auto-trust email domains.
  • Do not auto-create privileged accounts.
  • Require verified domain ownership for tenant admins.
  • Map external claims to internal entitlements through a controlled policy layer.
  • Treat every IdP as a separate trust domain.
  • Keep per-tenant logs and audit trails.

In brief

Dynamic OIDC is not just “login with any issuer.” It is delegated trust. Every issuer becomes a security dependency.

7. How would you design logout and session invalidation in an OIDC-based app?

Core answer

Logout in OIDC has multiple layers:

  1. Local application session logout
  2. Token revocation / refresh token invalidation
  3. IdP session logout
  4. Federated logout across other relying parties
  5. Back-channel invalidation for server-side sessions

RP-Initiated Logout lets an RP redirect the user to the OP logout endpoint, typically discovered via end_session_endpoint; the spec defines parameters such as id_token_hint, logout_hint, post_logout_redirect_uri, and state. ([OpenID Foundation][4])

Logout design

sequenceDiagram
    participant User
    participant App as RP App
    participant SessionDB
    participant IdP as OpenID Provider
    participant APIs

    User->>App: Click Logout
    App->>SessionDB: Revoke local session
    App->>APIs: Revoke refresh/access tokens if supported
    App->>IdP: Redirect to end_session_endpoint with id_token_hint
    IdP-->>User: Clear IdP session / prompt
    IdP-->>App: Redirect back with logout state
    App-->>User: Logged out page

What I would implement

Local app logout

  • Delete server-side session.
  • Clear session cookie with correct domain/path.
  • Rotate CSRF tokens.
  • Clear browser storage if SPA.
  • Invalidate refresh token references.
  • Add session ID to denylist until natural expiry if using self-contained session tokens.

IdP logout

  • Use RP-initiated logout where supported.
  • Include id_token_hint.
  • Validate post_logout_redirect_uri.
  • Use state to protect logout callback integrity.

Enterprise invalidation

  • Maintain central session store.
  • Support admin-triggered global logout.
  • Subscribe to IdP events: user disabled, password reset, MFA reset, device compromised, group removed.
  • On high-risk event, invalidate all app sessions for that subject.
  • Short-lived access tokens; refresh tokens with rotation and replay detection.

Back-channel logout

For high-security enterprise systems, prefer back-channel logout because it does not rely solely on browser redirects or iframes.

Important caveat

Logout from one RP does not always mean the user is logged out everywhere. Some providers support only RP-initiated logout, some support front-channel logout, some support back-channel logout, and some do not provide full single logout semantics.

Strong answer

I separate “user clicked logout” from “session is no longer valid.” Logout is a user action; invalidation is a security control. The app must be able to invalidate sessions even if the browser never completes the logout redirect.

8. Why is email verification not equivalent to strong identity assurance?

Core answer

email_verified=true means the issuer claims that the user controlled that email address at some point according to the issuer’s policy. It does not prove:

  • The person is an employee.
  • The person still controls the mailbox.
  • The mailbox was strongly verified.
  • The account was created by HR.
  • MFA was used.
  • The domain belongs to the tenant.
  • The user is authorized for the application.
  • The identity proofing reached a required assurance level.

OIDC explicitly warns that email, phone_number, preferred_username, and name must not be used as unique identifiers; only iss + sub is guaranteed stable for a user. It also notes that email may be reused or change over time. (OpenID Foundation)

Attack scenarios

Scenario Why Email Verification Fails
Personal Gmail used for enterprise app email_verified=true does not prove employment.
Recycled email address New owner gets old address and may inherit account mapping if app keys by email.
Rogue IdP Attacker’s IdP asserts email_verified=true for ceo@company.com.
Compromised mailbox Attacker can pass email verification but is not the legitimate user.
Weak tenant policy Tenant permits self-service accounts without HR approval.

Stronger assurance signals

Use a combination of:

  • Trusted issuer
  • iss + sub
  • Tenant allowlist
  • Domain ownership verification
  • HR authoritative source
  • SCIM-provisioned account
  • Group/role from enterprise directory
  • acr / amr for MFA or phishing-resistant auth
  • Device posture
  • Risk score
  • Just-in-time access approval for sensitive roles

In brief

Email is a contact attribute. It is not an identity root of trust.

9. How should an enterprise app map OIDC claims to internal entitlements safely?

Core answer

The app should not directly trust external OIDC claims as internal authorization decisions. Instead, it should use OIDC claims as inputs to an internal entitlement system.

Bad pattern:

if token.groups contains "admin":
    grant admin

Better pattern:

principal = lookup_user(issuer, subject)
tenant = validate_tenant(issuer)
entitlements = internal_policy_engine.resolve(principal, tenant, app, environment)

Safe mapping architecture

flowchart TD
    A[Validated ID Token] --> B[Extract iss + sub]
    B --> C[Find Internal Principal]
    C --> D[Check Tenant Trust Policy]
    D --> E[Fetch Authoritative Entitlements]
    E --> F[Policy Engine]
    F --> G{Allowed?}
    G -->|Yes| H[Issue App Session]
    G -->|No| I[Deny]

Recommended model

Input Claim Safe Use
iss Select tenant trust configuration.
sub Stable external principal key.
email Display/contact only, unless verified through internal process.
email_verified Weak signal, not authorization.
groups Input to mapping table, not direct permission.
roles Input to policy, not direct admin grant unless issuer is highly trusted.
acr Enforce assurance level for sensitive actions.
amr Check whether MFA or phishing-resistant auth was used.
auth_time Enforce recent authentication.

Controls

  • Use iss + sub as the external identity key.
  • Maintain internal user ID separate from external subject.
  • Use SCIM or HR feed as authoritative source for employee lifecycle.
  • Implement explicit claim-to-role mapping per tenant.
  • Require approval for privileged role mapping.
  • Deny by default for unknown groups.
  • Normalize claim values carefully.
  • Do not let customers self-assert global admin roles.
  • Add drift detection between IdP groups and internal permissions.
  • Log all entitlement decisions.

Example

External claim:

{
  "iss": "https://idp.customer-a.example",
  "sub": "00u123456",
  "email": "alice@customer-a.com",
  "groups": ["finance-approver"]
}

Internal mapping:

tenant: customer-a
trusted_issuer: https://idp.customer-a.example
claim_mappings:
  groups:
    finance-approver:
      internal_role: invoice_approver
      max_scope: customer-a
      requires_mfa_acr: phishing_resistant

The external group does not directly grant access. It maps to a controlled internal role under a tenant-specific policy.

10. What would you audit if you suspected a federation misconfiguration incident?

Core answer

I would audit the entire federation trust chain: issuer configuration, client registration, redirect URIs, token validation, claim mapping, session creation, entitlement assignment, and post-login activity.

Investigation flow

flowchart TD
    A[Suspected Federation Incident] --> B[Identify Affected Tenant / Issuer]
    B --> C[Review OIDC Client Config]
    C --> D[Review Token Validation Logs]
    D --> E[Review Claim Mapping]
    E --> F[Review Session Creation]
    F --> G[Review Entitlement Changes]
    G --> H[Review User Activity]
    H --> I[Containment]
    I --> J[Root Cause + Control Fix]

Audit checklist

1. IdP and issuer trust

  • Which issuers are trusted?
  • Was dynamic discovery enabled?
  • Were any new issuers added recently?
  • Does configured issuer exactly match token iss?
  • Was metadata fetched from the expected .well-known/openid-configuration location?
  • Did jwks_uri change?
  • Were unexpected signing keys or kids observed?

OIDC Discovery metadata includes endpoint and key-location information, including jwks_uri; therefore metadata changes are security-relevant. (OpenID Foundation)

2. Client registration

  • Client ID
  • Client secret rotation history
  • Redirect URIs
  • Post-logout redirect URIs
  • Allowed response types
  • Allowed grant types
  • Allowed signing algorithms
  • PKCE requirement
  • Public vs confidential client classification

3. Redirect URI abuse

  • Any wildcard redirect URIs?
  • Any open redirects in callback paths?
  • Any unregistered redirect URI accepted?
  • Any callback endpoint serving third-party JavaScript?
  • Any suspicious state mismatch errors?

OAuth requires redirect URI comparison against registered values when a redirect URI is included, and weak redirect URI registration can enable open redirector abuse. ([IETF Datatracker][1])

4. Token validation

Audit whether the RP validated:

  • Signature
  • iss
  • aud
  • azp
  • exp
  • iat
  • nonce
  • auth_time
  • acr
  • Expected algorithm
  • Correct JWKS source

5. Nonce and state

Look for:

  • Missing state
  • Reused state
  • State not bound to browser session
  • Missing nonce
  • Replayed nonce
  • Nonce not checked in ID token
  • High rate of failed callback validations

6. Claim mapping

  • Were users keyed by email instead of iss + sub?
  • Were external groups mapped directly to admin roles?
  • Were tenant-specific roles applied globally?
  • Were unverified or weak IdPs allowed to assert privileged claims?
  • Were new claim mappings added recently?

7. Account linking

  • Did a federated identity link to an existing local account?
  • Was linking based only on email?
  • Was step-up required before linking?
  • Was the user notified?
  • Were privileged accounts linked recently?

8. Session and authorization

  • Which sessions were created from suspicious tokens?
  • Were sessions invalidated after detection?
  • Were refresh tokens issued?
  • Were refresh tokens rotated?
  • Did any suspicious session survive user disablement?

9. User activity after login

  • Admin role changes
  • API key creation
  • OAuth consent grants
  • Data exports
  • New service accounts
  • New MFA devices
  • Password or recovery changes
  • Privileged configuration changes
  • Unusual IP/device/geography

10. Containment

Immediate actions:

  • Disable affected OIDC client.
  • Remove rogue issuer or tenant.
  • Rotate client secret/private key.
  • Invalidate sessions created during incident window.
  • Revoke refresh tokens.
  • Pin issuer and JWKS configuration.
  • Disable dynamic tenant onboarding temporarily.
  • Review admin accounts and entitlement mappings.
  • Add detection rules for repeated state/nonce failures and unexpected issuer/kid.

Incident examples

Misconfiguration 1: RP accepts any issuer

Attacker hosts their own IdP and signs an ID token for admin@example.com. RP validates the JWT signature against attacker JWKS because it blindly trusts discovery. Result: unauthorized login.

Misconfiguration 2: RP keys users by email

A user’s old email is reassigned. New mailbox owner logs in and gets mapped to the old account. Result: account takeover.

Misconfiguration 3: RP trusts external groups directly

A customer tenant sends:

"groups": ["global-admin"]

The SaaS app directly grants global admin. Result: cross-tenant privilege escalation.

Final Summary

One-line answers

Question Strong Model Answer
What does OIDC add? Authentication, ID token, standard claims, UserInfo, discovery, JWKS, and session/logout extensions on top of OAuth.
Why ID token is not session policy? It proves an authentication event; it does not continuously enforce device, risk, entitlement, revocation, or step-up policy.
What claims validate? Signature, issuer, audience, azp, expiry, iat, nonce, auth_time, acr/amr, algorithm, and key source.
Why nonce and state? state protects the callback from CSRF; nonce binds the ID token to the auth request and prevents replay.
Discovery trust issue? Discovery controls endpoints and keys; malicious metadata can redirect trust to attacker infrastructure.
Dynamic multi-tenant risk? Every issuer becomes a trust dependency; rogue tenants, claim injection, issuer mix-up, and unsafe account linking are major risks.
Logout design? Clear local session, revoke tokens, use RP-initiated/logout specs, support admin invalidation, and prefer back-channel for high assurance.
Email verification? Email is an attribute, not strong identity proof or authorization.
Claims to entitlements? Map iss + sub to internal principal and evaluate internal policy; never directly trust external roles/groups.
Federation incident audit? Audit issuer trust, discovery, JWKS, redirect URIs, token validation, nonce/state, claim mappings, sessions, and privileged activity.

Key Takeaways

  • OIDC is the modern identity layer for federated sign-in on top of OAuth 2.0.
  • The ID token is an authentication artifact and must be validated strictly.
  • Code flow with PKCE and nonce is the standard secure pattern.
  • Federation claims are useful inputs, but authorization still belongs in local policy.
  • Discovery, issuer trust, and tenant boundaries are major implementation risk areas.
