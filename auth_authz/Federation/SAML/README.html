SAML - Security Assertion Markup Language

Category: Federation and enterprise SSO
Primary Standards: SAML 2.0 Core, Bindings, Profiles

Overview

SAML is an XML-based federation standard used primarily for enterprise single sign-on between an identity provider and a service provider. It predates OIDC in mainstream enterprise federation and remains common in legacy SaaS integrations, workforce identity, and internal enterprise systems. SAML uses signed assertions that carry authentication and attribute statements, often delivered through browser redirects and form posts. Its flexibility and age mean interoperability is broad, but secure implementation can be error-prone.

How It Works

Core Mechanics

Primary actors and artifacts:

Item Purpose
Identity Provider Authenticates the user and issues SAML assertions
Service Provider Consumes assertions and creates a local session
Assertion XML statement about authentication and attributes
ACS URL Assertion Consumer Service endpoint at the service provider
Metadata XML describing endpoints, certificates, and identifiers

Common security properties:

Property Why it matters
XML signature validation Ensures assertions come from the trusted IdP
Audience restriction Limits where an assertion is valid
Recipient and destination checks Ensures the assertion was sent to the right place
Time window validation Limits replay and stale assertion use

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 XML signature wrapping Parser validates one node but app trusts another Authentication bypass High
2 Assertion replay Time window or one-time use tracking is weak Session hijack Med
3 Misconfigured audience or ACS SP accepts assertions for the wrong target Cross-app sign-in abuse Med
4 Weak XML parsing XXE or parser bugs are enabled Data exposure or SSRF Med
5 Over-trusted attributes App treats assertion attributes as authorization truth Privilege escalation Med
6 IdP certificate confusion Wrong signing certificate or rollover handling Token acceptance failure or spoofing risk Med
7 Incomplete logout assumptions SP session persists after IdP logout Session inconsistency Med
8 Metadata trust failure Metadata is imported unsafely or stale Broken federation trust Low

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 XML signature wrapping Use hardened SAML libraries and strict node reference checks SP implementation
2 Assertion replay Enforce narrow validity windows and replay detection SP session layer
3 Misconfigured audience or ACS Validate exact ACS, audience, and recipient fields SP configuration
4 Weak XML parsing Disable unsafe XML parser features such as XXE Parsing layer
5 Over-trusted attributes Separate authentication attributes from authorization policy Application authorization layer
6 IdP certificate confusion Manage signing certificate pinning and rollover carefully Federation configuration
7 Incomplete logout assumptions Handle local logout independently of federation events App session layer
8 Metadata trust failure Treat metadata as a controlled trust input Federation operations

Top 3 Critical Mitigations

  1. Use mature SAML libraries: Hand-rolling XML signature handling is where many catastrophic SAML bugs begin. Mature libraries handle canonicalization and node reference validation correctly.
  2. Validate the full assertion context: Signature alone is not enough. Audience, recipient, destination, issuer, and time windows all need exact checks.
  3. Do not equate assertion attributes with authorization truth: Group and role claims from the IdP are useful, but applications still need local privilege models and validation.

Key Takeaways

  • SAML is still a major federation technology in workforce identity and enterprise SSO.
  • The biggest risks are XML handling, assertion validation mistakes, and over-trusting attributes.
  • Signed assertions must also be checked for audience, destination, time window, and issuer.
  • Local sessions and authorization policy still need their own control plane.
  • Mature libraries and conservative federation configuration are essential.
