SCIM - System For Cross-Domain Identity Management

Category: Federation and identity lifecycle
Primary Standards: RFC 7643, RFC 7644

Overview

SCIM is a standard for provisioning and lifecycle management of identities, groups, and related attributes across organizational boundaries. It is commonly used to automate user and group creation, updates, deactivation, and entitlement synchronization from an identity provider into SaaS applications. SCIM is not an authentication protocol; it is an identity data management and provisioning protocol. Its security importance comes from the fact that provisioning accuracy directly shapes downstream access and deprovisioning speed.

How It Works

Core Mechanics

SCIM defines standard resource types such as User and Group, a REST-style API, schema extension points, and patch semantics for updates. An identity source system acts as the provisioning client and a SaaS or downstream platform acts as the SCIM service provider.

SCIM is built on top of HTTP and JSON. The service provider exposes a RESTful API and the client (provisioning agent) pushes lifecycle changes to it. There is no federation token or authentication assertion involved — SCIM is purely about identity data synchronization.

Resource Purpose
User Represents an identity with core attributes and status
Group Represents collections used for membership and entitlements
Patch operations Modify selected fields without replacing full objects
Active flag Commonly used to drive deprovisioning or suspension
ServiceProviderConfig Describes supported SCIM features and auth methods
ResourceTypes Lists the resource endpoints the service provider exposes
Schemas Describes attribute names, types, and mutability

API Endpoints and HTTP Methods

SCIM maps identity operations to standard HTTP verbs on resource URIs:

Operation HTTP Method Example URI Semantics
Create user POST /Users Create a new user resource
Read user GET /Users/{id} Retrieve a user by stable SCIM ID
List / search users GET /Users?filter=userName eq "alice" Filter users with SCIM filter syntax
Full replace PUT /Users/{id} Replace all attributes on a user
Partial update PATCH /Users/{id} Apply add, replace, or remove ops only
Delete user DELETE /Users/{id} Permanently delete the resource
Bulk operations POST /Bulk Execute multiple operations atomically
Service discovery GET /ServiceProviderConfig Query what the provider supports

SCIM User Resource Schema

A SCIM User is a JSON document. Core attributes are defined in RFC 7643:

{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
  "id": "2819c223-7f76-453a-919d-413861904646",
  "externalId": "701984",
  "userName": "alice@example.com",
  "name": {
    "formatted": "Alice Smith",
    "familyName": "Smith",
    "givenName": "Alice"
  },
  "emails": [{ "value": "alice@example.com", "primary": true }],
  "active": true,
  "groups": [{ "value": "e9e30dba-f08f-4109-8486-d5c6a331660a" }],
  "meta": {
    "resourceType": "User",
    "created": "2024-01-10T00:00:00Z",
    "lastModified": "2024-06-01T12:00:00Z",
    "location": "https://example.com/v2/Users/2819c223-..."
  }
}

The id field is assigned by the service provider and is the stable identifier used for all subsequent operations. The externalId is assigned by the provisioning client to correlate back to its own directory. Mixing up id vs externalId ownership is a common integration mistake.

PATCH Operations in Detail

SCIM PATCH uses JSON Patch-like semantics but with SCIM-specific operation objects. A PATCH body contains an array of Operations:

{
  "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
  "Operations": [
    { "op": "replace", "path": "active", "value": false },
    {
      "op": "add",
      "path": "emails",
      "value": [{ "value": "alice2@example.com", "primary": true }]
    },
    { "op": "remove", "path": "phoneNumbers[type eq \"work\"]" }
  ]
}
Operation Meaning
add Add a new attribute value or extend a multi-valued list
replace Overwrite a specific attribute with the given value
remove Delete a specific attribute or matching multi-value member

PATCH is preferred over PUT for incremental updates because PUT replaces all attributes, which can accidentally erase attributes set by other systems.

Deprovisioning: Active Flag vs. Deletion

SCIM distinguishes between deactivation and deletion:

  • Deactivation: PATCH or PUT sets "active": false. The user record remains. The service provider is expected to revoke access but retain data.
  • Deletion: DELETE on the user URI. The record is removed from the service provider. Some providers soft-delete and some hard-delete.

Most enterprise deprovisioning flows use active: false rather than DELETE to preserve audit history and allow reactivation. Failing to honor active: false in the service provider implementation is a common source of stale access.

Group Provisioning and Membership

A SCIM Group resource holds display name and a list of member references:

{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:Group"],
  "id": "e9e30dba-f08f-4109-8486-d5c6a331660a",
  "displayName": "Engineering",
  "members": [
    {
      "value": "2819c223-7f76-453a-919d-413861904646",
      "$ref": "/Users/2819c223-..."
    },
    {
      "value": "c75ad752-64ae-4823-840d-f2217d630a9d",
      "$ref": "/Users/c75ad752-..."
    }
  ]
}

Group membership is the most common vector for entitlement drift. If the provisioning system sends only delta updates and a retry is missed, a user may remain in a group they were removed from in the source directory.

SCIM and SAML / OIDC Relationship

SCIM is complementary to, not a replacement for, authentication federation protocols:

flowchart LR
    A[HR System or IDP] -->|SCIM provisioning| B[SaaS App]
    A -->|SAML assertion or OIDC token| B
    B --> C[Access granted based on provisioned role and active flag]
  • SAML / OIDC handle authentication — asserting who the user is at login time.
  • SCIM handles provisioning — ensuring the user account and entitlements exist before or during onboarding.
  • JIT provisioning via SAML can substitute for SCIM in simple deployments but lacks the deprovisioning and group sync capabilities that SCIM provides.

Authentication for SCIM Endpoints

RFC 7644 does not mandate a specific auth mechanism but recommends OAuth 2.0 bearer tokens. Common approaches in practice:

Method Strength Risk
Long-lived bearer token Low Token leakage gives full provisioning access indefinitely
OAuth 2.0 client credentials High Scoped access, token expiry, rotation possible
Mutual TLS High Strong client identity but complex to operate
Basic auth (legacy) Very low Credentials in every request, often stored in config files

SCIM tokens effectively grant admin-level write access to identity records. They must be stored in a secrets manager, rotated regularly, and scoped to least-privilege provisioning operations only.

Joiner-Mover-Leaver Lifecycle via SCIM

SCIM is the primary mechanism for automating the JML lifecycle in enterprise environments:

flowchart TD
    J[New hire created in HR] --> JP[SCIM POST /Users - create account and assign groups]
    M[Role change in HR] --> MP[SCIM PATCH /Users - update department, title, groups]
    L[Termination in HR] --> LP[SCIM PATCH /Users - set active false]
    LP --> LD[SCIM DELETE /Users - optional hard delete after retention period]
Event SCIM Operation Critical Timing
Joiner POST /Users, PATCH /Groups to add member Must complete before first login attempt
Mover PATCH /Users attributes, group membership Should complete same-day as role change
Leaver PATCH active=false, remove group members Must complete within minutes of offboard

Bulk Operations

The /Bulk endpoint allows batching multiple create, update, and delete operations in a single HTTP request. Each operation in the batch has a bulkId for correlation and a method field:

{
  "schemas": ["urn:ietf:params:scim:api:messages:2.0:BulkRequest"],
  "Operations": [
    { "method": "POST", "path": "/Users", "bulkId": "qwerty", "data": { ... } },
    { "method": "PATCH", "path": "/Users/abc123", "bulkId": "ytrewq", "data": { ... } }
  ]
}

Partial failures in bulk are a common source of inconsistency. The service provider may process some operations and fail others. Provisioning clients must handle partial responses and reconcile state.

Schema Extensions

SCIM allows enterprise-specific attributes via schema extensions. The extension namespace is appended to the schemas array and the attributes are namespaced:

{
  "schemas": [
    "urn:ietf:params:scim:schemas:core:2.0:User",
    "urn:ietf:params:scim:schemas:extension:enterprise:2.0:User"
  ],
  "urn:ietf:params:scim:schemas:extension:enterprise:2.0:User": {
    "employeeNumber": "701984",
    "costCenter": "4130",
    "organization": "Universal Studios",
    "division": "Theme Park",
    "department": "Tour Operations",
    "manager": { "value": "26118915-6090-4610-87e4-49d8ca9f808d" }
  }
}

Custom extension attributes are a common source of mapping errors when the source directory uses different field names or value formats.

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Delayed deprovisioning SCIM sync fails or runs too slowly Stale access remains High
2 Schema mapping errors Source attributes map to wrong target fields Misprovisioned access or identity corruption High
3 Group sync drift Membership changes do not propagate correctly Over- or under-entitlement High
4 Weak SCIM authentication Provisioning API token is poorly protected Mass account manipulation High
5 Partial update logic flaws Patch semantics applied incorrectly Unexpected attribute loss or persistence Med
6 Idempotency problems Retries create duplicates or inconsistent state Operational instability Med
7 Soft-delete confusion Deactivation is mistaken for hard delete or vice versa Retained access or data loss Med
8 Missing audit trails Teams cannot prove what provisioning changed and when Slow incident response Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Delayed deprovisioning Monitor sync health and prioritize offboarding paths Provisioning operations
2 Schema mapping errors Test mappings and keep attribute contracts versioned Identity integration layer
3 Group sync drift Reconcile memberships regularly and alert on divergence Provisioning and governance
4 Weak SCIM authentication Protect SCIM credentials like privileged admin secrets Integration security
5 Partial update logic flaws Validate patch behavior and edge cases with contract tests SCIM implementation
6 Idempotency problems Use stable identifiers and retry-safe write semantics API design
7 Soft-delete confusion Define deactivation and deletion semantics clearly Lifecycle policy
8 Missing audit trails Log provisioning actor, object, diff, and outcome Audit and response

How Okta SCIM Provisioning Works (Reference Architecture)

Okta is the most common SCIM provisioning client in enterprise SaaS environments. Its provisioning flow:

flowchart TD
    A[HR System such as Workday] -->|Employee record created or changed| B[Okta Universal Directory]
    B -->|Attribute mapping| C[Okta Provisioning Engine]
    C -->|SCIM POST /Users| D[SaaS App such as Salesforce or Slack]
    C -->|SCIM PATCH /Groups| D
    D -->|200 OK with canonical SCIM ID| C
    C --> E[Okta stores external ID for correlation]
    F[HR termination event] --> B
    B -->|active=false trigger| C
    C -->|SCIM PATCH active false| D

Key Okta provisioning features relevant to security:

  • Push Groups: Okta pushes group membership changes to the app; groups drive entitlements
  • Import: Okta can read existing accounts from the app and link them to directory identities (prevents orphaned accounts)
  • On-demand provisioning: Provision a specific user immediately without waiting for the sync cycle
  • Provisioning policies: Control which users are in scope and what attributes are pushed
  • Error surfacing: Failed provisioning tasks are visible in the Okta admin console and can alert on deprovisioning failures

GitHub Enterprise Managed Users (EMU) as a SCIM Example

GitHub EMU is a strict SCIM-only deployment model where all user accounts are provisioned and controlled entirely through the IDP. No self-signup is permitted:

  • Users are created via SCIM and cannot change their own usernames or email addresses
  • Deactivating a user via SCIM immediately suspends their GitHub access
  • Group membership in the IDP maps directly to GitHub team membership and repo access
  • If the SCIM provisioning breaks, users cannot recover their own accounts — dependency on IDP health is absolute

This makes EMU one of the highest-integrity SCIM deployments in terms of access control, but also one of the most operationally fragile.

Key Takeaways

  • SCIM automates identity and group lifecycle changes across systems.
  • It is a provisioning protocol, not an authentication protocol.
  • The biggest risks are stale deprovisioning, mapping errors, and weak protection of SCIM integration credentials.
  • Provisioning quality directly affects authorization quality downstream.
  • Offboarding and group synchronization deserve the highest operational scrutiny.
  • Okta, Microsoft Entra, and Ping are the dominant SCIM clients; Salesforce, Slack, GitHub, and Snowflake are common SCIM service providers.
  • IGA platforms like SailPoint sit above SCIM and add governance, certification, and SoD controls that raw SCIM lacks.
