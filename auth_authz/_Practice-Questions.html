Authentication — Google Senior Security Engineer Model Answer

Authentication is not just “login.” It is a system of controls that determines:

Who is the user?
How strongly was that user verified?
Is this still the same user during the session?
Can the user recover access safely?
What happens when credentials, devices, sessions, or recovery paths are compromised?

A senior answer should emphasize that authentication assurance is only as strong as the weakest path into the account: primary login, MFA, session handling, federation, recovery, helpdesk, device trust, and fallback.

1. MFA as a control family, phishing-resistance, and step-up design

Why MFA is a control family, not a single technology

MFA means using more than one factor from different categories:

Something you know       → password, PIN
Something you have       → phone, hardware key, authenticator app, smart card
Something you are        → biometrics
Something you are using  → managed device, device certificate
Somewhere you are        → network/location/risk signal

But not all MFA gives the same security.

For example:

MFA Type Example Phishing-resistant? Main Weakness
SMS OTP 6-digit SMS code No SIM swap, phishing, malware
Email OTP Code sent to email No Email compromise
TOTP Google Authenticator, Microsoft Authenticator code No User can type code into phishing site
Push approval “Approve login?” Usually no Push fatigue, consent phishing
Number matching push Match number shown on login page Better, but not fully phishing-proof Real-time phishing proxy
FIDO2/WebAuthn/passkeys Hardware key or platform passkey Yes, when correctly implemented Recovery/fallback complexity
Smart card / certificate auth Enterprise cert-based login Strong Device lifecycle and recovery complexity

So MFA is a control family because it includes different technologies with different threat models, usability tradeoffs, failure modes, operational costs, and recovery implications.

Senior-level phrasing:

MFA is not a binary checkbox. SMS OTP, TOTP, push, FIDO2, device certificates, and smart cards all sit at different assurance levels. The right design depends on threat model, user population, account value, recovery process, and attacker capability.

Why FIDO2/passkeys are fundamentally different from OTP and push

OTP and push are usually shared-secret or approval-based mechanisms.

A phishing site can proxy them.

Example OTP phishing:

1. User visits fake-login.example.
2. Fake site asks for username and password.
3. Fake site forwards credentials to real login.example.
4. Real site asks for OTP.
5. Fake site asks user for OTP.
6. User enters OTP.
7. Attacker forwards OTP to real site.
8. Attacker logs in.

The OTP is valid because it is not cryptographically bound to the real website origin.

FIDO2/WebAuthn/passkeys work differently.

During registration:

Relying Party: login.example.com
Authenticator creates:
  private key → stored in authenticator
  public key  → registered with login.example.com

During login:

1. Server sends challenge.
2. Browser includes origin: https://login.example.com
3. Authenticator signs challenge only for that relying party ID.
4. Server verifies signature using registered public key.

A phishing site cannot simply ask the user for a reusable code.

If the user is on:

https://evil-login.example

the authenticator will not produce a valid signature for:

https://login.example.com

That is the key difference.

Senior phrasing:

OTP proves the user knows a code. FIDO2 proves the user possesses a private key and that the authentication ceremony happened for the legitimate origin. That origin binding is what gives phishing resistance.

How push fatigue undermines MFA

Push MFA often fails because it asks the user to make a security decision under pressure.

Attack pattern:

1. Attacker obtains password from breach or phishing.
2. Attacker repeatedly attempts login.
3. User receives many push notifications:
   "Approve login?"
4. User becomes annoyed, confused, or assumes it is a system issue.
5. User taps Approve.
6. Attacker gains access.

This is called push fatigue or MFA bombing.

Push fatigue is dangerous because technically MFA was enabled, but operationally the system trained the user to approve unexpected prompts.

Controls:

- Number matching
- Show location, device, and application context
- Limit repeated push attempts
- Require re-authentication after multiple denies
- Alert security team on repeated push failures
- Prefer FIDO2/passkeys for high-risk users
- Add impossible travel and risk scoring
- Educate users: never approve unexpected prompts

But even number matching is not equivalent to FIDO2. A real-time phishing proxy can show the number from the legitimate site to the victim and ask the user to approve.

When should step-up authentication be triggered?

Step-up authentication should not happen only at login. It should happen when the user attempts an action that increases risk.

Examples:

- Changing password
- Changing MFA settings
- Adding a new authenticator
- Viewing recovery codes
- Changing recovery email or phone
- Adding a new device
- Creating API keys
- Downloading sensitive data
- Accessing admin console
- Performing financial transaction
- Changing bank account details
- Disabling audit logging
- Creating privileged IAM role
- Accessing production secrets
- Accessing from new geography/device
- Session risk score increases

Example design:

Normal login:
  Password + passkey or enterprise SSO

Low-risk action:
  Continue existing session

High-risk action:
  Require recent strong authentication

Very high-risk action:
  Require phishing-resistant MFA + device trust + approval workflow

Important concept: authentication freshness.

A user may have logged in 8 hours ago, but that should not be enough to add a new MFA device or export production secrets.

Senior phrasing:

Step-up should be tied to risk and privilege escalation, not just time. I care whether the user is about to do something irreversible, sensitive, privileged, or inconsistent with the existing session context.

MFA design for privileged admins vs standard users

Standard users

Goal: good security with low friction.

- Prefer passkeys or app-based MFA
- Avoid SMS where possible
- Use risk-based step-up
- Allow multiple registered authenticators
- Provide safe recovery
- Strong session monitoring

Example:

Standard employee:
  Login with SSO + passkey
  Step-up for payroll, HR data, or sensitive downloads

Privileged administrators

Goal: high assurance, low tolerance for compromise.

- Require phishing-resistant MFA
- Hardware security key or enterprise-managed passkey
- No SMS fallback
- Privileged access workstation or managed device
- Device posture checks
- Just-in-time privilege elevation
- Separate admin account from daily account
- Shorter session lifetimes
- Step-up for every privileged action
- Approval workflow for destructive operations
- Break-glass accounts tightly controlled
- Logging and alerting on admin auth events

Example:

Cloud production admin:
  Separate admin identity
  Hardware security key required
  Login only from managed device
  Privilege granted just-in-time for 1 hour
  Step-up required before IAM policy change
  All admin actions logged to immutable audit store

Senior phrasing:

For privileged admins, MFA is not enough. I would combine phishing-resistant MFA, device trust, least privilege, just-in-time access, session controls, and high-fidelity monitoring.

2. Password systems: hashing, detection, and migration

Why store salted slow hashes instead of encrypted passwords?

Password systems should not store recoverable passwords.

Bad design:

password → encrypt with key → store ciphertext

Problem: if the encryption key is stolen, all passwords can be decrypted.

Correct design:

password + unique salt → slow password hash → store hash and salt

A password hash is one-way. During login:

1. User enters password.
2. System retrieves salt.
3. System hashes entered password with same parameters.
4. System compares result with stored hash.

Why salt matters:

- Prevents identical passwords from having identical hashes
- Defeats precomputed rainbow tables
- Forces attacker to crack each password separately

Why slow hashing matters:

- Passwords have low entropy
- Attackers use GPUs/ASICs
- Slow/memory-hard functions increase cracking cost

Senior phrasing:

We hash passwords because the server never needs to recover the original password. Encryption preserves recoverability; password verification should not.

Argon2id vs bcrypt vs scrypt

Argon2id

Best modern default for new systems.

Strengths:

- Memory-hard
- Designed to resist GPU cracking
- Hybrid mode balances side-channel and GPU resistance
- Tunable memory, iterations, parallelism

Use when:

- Building a new system
- Library support is mature
- You can tune memory and CPU cost

bcrypt

Still acceptable and widely deployed.

Strengths:

- Battle-tested
- Broad library support
- Simple operational model

Weaknesses:

- Not memory-hard in the same way
- Password length handling can be tricky
- Older implementations may truncate long passwords

Use when:

- Legacy compatibility matters
- Argon2id support is unavailable

scrypt

Strong memory-hard option.

Strengths:

- Memory-hard
- Better GPU resistance than simple CPU-only hashes

Weaknesses:

- Operational tuning can be more complex
- Less commonly used than bcrypt in some stacks

My choice for new system:

Primary: Argon2id
Fallback acceptable: bcrypt with strong cost
Alternative: scrypt if platform support is mature

Important operational point:

The right algorithm is not enough. Parameters must be tuned so login remains acceptable for users but expensive for attackers.

Example:

Target server-side hash time:
  100–500 ms depending on risk and scale

Also consider:
  memory cost
  CPU capacity
  login peak traffic
  denial-of-service risk
  background rehash migration

Credential stuffing vs password spraying

These are different attacks and require different detections.

Credential stuffing

Attacker has username/password pairs from breaches.

Input:
  alice@example.com : Summer2024!
  bob@example.com   : Password123
  charlie@example.com : qwerty

Attack:
  Try known password for each account.

Operational pattern:

- Many usernames
- One or few attempts per username
- Distributed IPs
- High volume
- Uses breached credential lists

Controls:

- Breach password detection
- Bot detection
- Rate limits by IP, account, ASN, device, credential pair
- MFA
- Passwordless/passkeys
- Login anomaly detection

Password spraying

Attacker tries common passwords across many accounts.

Input:
  Password: Welcome2026!

Attack:
  Try Welcome2026! against thousands of users.

Operational pattern:

- Same password across many accounts
- Low and slow attempts
- Avoids account lockout
- Often targets enterprise SSO/VPN/email

Controls:

- Detect same password failure across many users
- Tenant-wide spray detection
- Smart lockout
- MFA
- Block common passwords
- Conditional access

Senior phrasing:

Credential stuffing is breached credential replay. Password spraying is low-rate guessing using common passwords across many accounts. If you only rate-limit per account, spraying may bypass you. If you only rate-limit per IP, distributed stuffing may bypass you.

Why breach blocklists beat forced periodic rotation

Forced periodic rotation often causes users to choose predictable passwords:

PasswordJan2026!
PasswordFeb2026!
PasswordMar2026!

It may reduce theoretical exposure time but creates poor usability and predictable behavior.

Breach blocklists are better because they target passwords known to be compromised or commonly guessed.

Controls:

- Check new passwords against breached/common password lists
- Block organization name, username, and simple variants
- Check password at registration and reset
- Trigger reset only when compromise is suspected
- Encourage passphrases or password managers

Senior phrasing:

Forced rotation often improves policy compliance but not real security. Blocking known-compromised passwords directly reduces the attacker’s success set.

Migration from password-only to stronger authentication

A large migration must balance security, adoption, recovery, and user support.

Phase 1: Measure and prepare

- Inventory user populations
- Identify privileged/high-risk users
- Understand current login paths
- Measure password reset volume
- Identify legacy clients
- Build MFA/passkey enrollment flows
- Build recovery process before rollout

Phase 2: Improve password baseline

- Salted slow hashes
- Breach password blocklist
- Credential stuffing detection
- Rate limits
- Bot protection
- Login risk scoring

Phase 3: Introduce MFA

- Start with admins
- Then employees
- Then high-risk users
- Then general population

Phase 4: Move to phishing-resistant MFA

- Encourage passkeys
- Require FIDO2 for admins
- Deprecate SMS
- Allow multiple authenticators
- Provide recovery codes or managed recovery

Phase 5: Passwordless or password-optional

- Passkey-first login
- Password as fallback initially
- Later remove password for eligible users
- Monitor fallback abuse

Phase 6: Enforce

- Deadlines by user risk tier
- Admin enforcement first
- Exceptions with expiration
- Continuous monitoring

Migration principle:

Do not remove the old path until the new path and recovery path are both secure.

3. Session management: lifecycle, binding, and global logout

Session fixation vs session hijacking

Session fixation

The attacker forces or tricks the victim into using a session ID chosen or known by the attacker.

Attack:

1. Attacker obtains session ID: SID=abc123.
2. Attacker sends victim a link using SID=abc123.
3. Victim logs in.
4. Application keeps same SID after login.
5. Attacker uses SID=abc123 as authenticated session.

Root problem:

Session ID did not rotate after authentication.

Session hijacking

The attacker steals an existing valid session ID.

Attack:

1. Victim logs in normally.
2. Session cookie is stolen through XSS, malware, proxy, logs, or network issue.
3. Attacker replays session cookie.
4. Server treats attacker as victim.

Root problem:

Session secret was exposed or replayable.

Senior phrasing:

Fixation is about attacker choosing or knowing the session before login. Hijacking is about stealing the session after login.

Why rotate session identifiers after authentication

Before authentication, the session may be low-trust:

anonymous cart
CSRF token
pre-login state

After authentication, the session becomes high-value.

If the same ID is retained, pre-auth compromise becomes post-auth compromise.

Correct behavior:

Anonymous session: SID_A
User authenticates successfully
Server creates new authenticated session: SID_B
SID_A is invalidated

Also rotate after:

- MFA completion
- Privilege elevation
- Password change
- Account recovery
- Role change
- Suspicious activity

Server-stored sessions vs signed token sessions

Server-stored sessions

Cookie contains random session ID:

Cookie: SID=random_256_bit_value

Server stores session state:

SID → user_id, auth_time, MFA level, device, expiry, roles

Pros:

- Easy revocation
- Easy global logout
- Session state can change immediately
- Small cookie
- Good control for high-security apps

Cons:

- Requires centralized/session store
- Scaling and availability considerations

Signed token sessions

Cookie or header contains signed token, often JWT:

JWT:
  sub=user123
  exp=...
  roles=...
  mfa=true

Pros:

- Stateless validation
- Good for distributed systems
- Reduced central dependency

Cons:

- Revocation is harder
- Claims can become stale
- Logout is difficult unless you add server-side state
- Token theft remains dangerous until expiry

Senior phrasing:

Stateless tokens are rarely fully stateless once you need logout, revocation, risk response, or role changes. Most serious systems reintroduce state through revocation lists, token versioning, or short lifetimes.

Cookie attributes: HttpOnly, Secure, SameSite

HttpOnly

Set-Cookie: SID=abc; HttpOnly

Protects against:

- JavaScript reading cookie via document.cookie

Does not protect against:

- XSS making authenticated requests
- CSRF by itself
- Malware/browser compromise

Important nuance:

HttpOnly reduces token theft via XSS, but XSS can still act as the user inside the browser.

Secure

Set-Cookie: SID=abc; Secure

Protects against:

- Cookie being sent over plain HTTP

Requires HTTPS.

Does not protect against:

- XSS
- CSRF
- Server-side compromise

SameSite

Set-Cookie: SID=abc; SameSite=Lax
Set-Cookie: SID=abc; SameSite=Strict
Set-Cookie: SID=abc; SameSite=None; Secure

Protects against:

- Some cross-site request forgery attacks

The problem it solves:

Classic CSRF attack:
  1. You are logged into bank.com — session cookie is stored.
  2. You visit evil.com.
  3. evil.com has a hidden form that POSTs to bank.com/transfer.
  4. Your browser automatically attaches the bank.com session cookie.
  5. bank.com sees a valid authenticated request.

SameSite breaks step 4 by restricting when the browser sends the cookie.

Modes:

Strict:
  Cookie sent only same-site.
  Strongest, may break flows.

  Example:
    User on evil.com clicks a link to bank.com/dashboard
    → Browser navigates to bank.com
    → Cookie NOT sent
    → User lands logged out

  Best for: high-security apps (banking, admin consoles) where
  you can accept that inbound links from email or search start
  a new unauthenticated session.

Lax (browser default since ~2020):
  Good default.
  Cookie withheld on many cross-site subrequests,
  but sent on top-level navigations.

  Example:
    User on evil.com clicks a link → GET bank.com/dashboard
    → Cookie IS sent (top-level nav, safe method)

    evil.com auto-submits a hidden POST form → bank.com/transfer
    → Cookie NOT sent

  Practical sweet spot: blocks most CSRF, does not break
  "login via link in email" flows.

None:
  Cookie sent cross-site.
  Must use Secure.
  Needed for some federated login/embedded flows.

  Example:
    evil.com iframe loads bank.com/widget
    → Cookie IS sent

  Required for: OAuth flows, embedded widgets, payment iframes,
  federated SSO where cookie must survive cross-origin navigation.

What SameSite does NOT protect against:

Threat                              Protected?  Why
CSRF via cross-site POST            Yes         Cookie withheld (Strict/Lax)
CSRF via top-level GET navigation   No          Cookie sent with Lax
XSS                                 No          Attacker runs same-origin
Session hijacking via network       No          That is Secure's job
Subdomain cookie theft              No          Same-site includes subdomains

Key nuance — "same-site" is not "same-origin":

SameSite uses eTLD+1 (registrable domain), not full origin.

Same-site:
  app.example.com  →  api.example.com   (same eTLD+1)

Cross-site:
  app.example.com  →  evil.com
  app.example.com  →  app.example.co.uk (different eTLD+1)

Consequence:
  SameSite=Lax does not protect against attacks launched from
  a sibling subdomain you do not control (e.g., a user-controlled
  *.example.com subdomain). CSRF tokens remain important for that.

Why Lax alone is not enough for high-security apps:

- A GET request that causes a state change (logout, password reset trigger)
  is still sent with Lax.
- SameSite=Strict may break needed UX.
- CSRF tokens remain important for defense-in-depth.

Cookie best practice:

Set-Cookie: __Host-session=<random>;
  Path=/;
  Secure;
  HttpOnly;
  SameSite=Lax

For highly sensitive apps, consider:

- SameSite=Strict where possible
- CSRF tokens
- Re-auth for sensitive actions
- Origin/Referer validation

Global logout across federated services

Global logout is difficult because many services may have their own sessions.

Example architecture:

Identity Provider
  ├── App A session
  ├── App B session
  ├── App C session
  └── API token sessions

When user logs out or account is compromised, you need to terminate:

- IdP session
- Application sessions
- Refresh tokens
- Access token families
- Device sessions
- API keys where appropriate
- Remembered devices

Design options:

Central session store

All services check central session state.
Logout marks session revoked.

Strong revocation, but adds dependency.

Short-lived access tokens + refresh token revocation

Access tokens expire quickly.
Refresh token revoked immediately.

Good balance for APIs.

Event-driven logout

User logout/account reset emits event:
  SESSION_REVOKED
  TOKEN_FAMILY_REVOKED
  USER_RISK_CHANGED

Services consume event and clear local sessions.

Useful at scale.

Token versioning

User record has:

session_version = 17

Token contains:

session_version = 16

Resource server rejects old version.

Back-channel logout

Federated services receive server-to-server logout notification.

Senior design:

- Revoke IdP session
- Revoke refresh tokens/token families
- Rotate session version
- Send logout event to relying services
- Short TTL access tokens
- Require re-authentication on next request
- Clear browser cookies where possible
- Audit logout propagation

Why strong login factors do not remove session controls

Even if login uses hardware keys, the session can still be stolen.

Attack:

1. User logs in with FIDO2 hardware key.
2. Application issues session cookie.
3. XSS or malware steals/uses session.
4. Attacker bypasses login completely.

Authentication strength protects the login ceremony. Session controls protect everything after login.

Controls:

- Secure cookies
- Session rotation
- Short idle and absolute timeout
- Device binding where appropriate
- Re-auth for sensitive actions
- Session anomaly detection
- Global logout
- XSS prevention

Senior phrasing:

Authentication proves the user at a point in time. Session management preserves that assurance over time. A strong front door is not enough if session tokens are weak, long-lived, or unrevocable.

4. Account recovery as the real assurance floor

Why recovery defines the real assurance floor

Attackers do not need to beat your strongest login path. They need to find the weakest way into the account.

Example:

Primary login:
  Password + hardware key

Recovery:
  Call helpdesk, answer weak questions, reset MFA

Actual assurance:
  Helpdesk process, not hardware key

If recovery can bypass FIDO2 using a phone call and weak identity proofing, then the system is not truly hardware-key assured.

Senior phrasing:

Account recovery is the backdoor you intentionally build. Its assurance must match the value of the account.

How weak recovery defeats hardware-bound authentication

Attack:

1. Admin account requires hardware security key.
2. Attacker cannot phish key.
3. Attacker calls helpdesk.
4. Claims phone was lost.
5. Helpdesk resets MFA after weak verification.
6. Attacker enrolls new authenticator.
7. Attacker logs in as admin.

The hardware key did not fail. The recovery process failed.

Hardening helpdesk-driven recovery against social engineering

Controls:

- Strong identity verification
- Call-back to pre-registered number
- Manager approval for privileged users
- Two-person approval for admins
- No recovery based only on caller-provided information
- No SMS-only verification for high-risk accounts
- Require government ID only where appropriate and privacy-reviewed
- Use prior trusted device approval where possible
- Cooling-off period before full privilege restoration
- Notify user through multiple existing channels
- Record and audit recovery interactions
- Train helpdesk on social engineering patterns
- Scripted verification process
- Risk scoring for recovery attempts

For privileged accounts:

- Helpdesk cannot reset alone
- Require security team approval
- Require manager approval
- Require proof from existing trusted device or in-person verification
- Temporary limited-access account may be issued
- Privileged access restored separately after review

Why recovery policy should differ for privileged and high-value accounts

Not all accounts have the same blast radius.

Standard user compromise:

- Email
- Personal data
- Some SaaS access

Privileged admin compromise:

- IAM changes
- Production access
- Secrets access
- Logging disabled
- Data exfiltration
- Persistence creation

High-value accounts:

- Executives
- Finance approvers
- HR admins
- Cloud admins
- Source code admins
- CI/CD administrators
- Security administrators

Different policy:

Account Type Recovery Requirement
Standard user Self-service plus risk checks
Employee SSO recovery plus managed device
Admin Multi-party approval
Break-glass Offline/sealed process
Executive Dedicated high-risk workflow

What to revoke after successful recovery

After recovery, assume previous factors may be compromised.

Immediately revoke:

- Existing sessions
- Refresh tokens
- Remembered devices
- Existing recovery codes
- App passwords
- API keys where relevant
- OAuth grants if suspicious
- Old MFA authenticators if lost/compromised
- Password reset tokens
- Pending login challenges

Also require:

- New MFA enrollment
- Password reset if password may be compromised
- Recent activity review
- User notification
- Risk monitoring period

Senior phrasing:

Recovery is not complete when the user gets back in. It is complete when old access paths are invalidated and the account returns to a known-good state.

Redesign after SIM-swap-driven account takeover campaign

Problem:

Attackers port victims' phone numbers.
They receive SMS OTP or password reset links.
They reset account credentials.
They take over accounts.

Redesign:

1. Remove SMS as a recovery factor for high-risk accounts.
2. Treat phone-number change as high-risk.
3. Add cooldown for recovery phone changes.
4. Notify old and new channels.
5. Require existing trusted device or passkey for recovery.
6. Use recovery codes stored offline.
7. Require stronger identity proofing for account reset.
8. Revoke sessions and tokens after recovery.
9. Detect carrier-change signals where available.
10. Add fraud analytics:
    - recent SIM change
    - new device
    - new IP/ASN
    - password reset followed by payout/account change
11. Step-up before financial changes after recovery.
12. Temporarily restrict high-risk actions post-recovery.

Example post-recovery restriction:

For 24–72 hours after recovery:
  - Cannot change payout account
  - Cannot add new MFA without extra approval
  - Cannot export all data
  - Cannot create API keys

5. WebAuthn and passkeys: ceremonies, origin binding, and fallback

WebAuthn registration ceremony

Actors:

User
Browser
Authenticator
Relying Party server

Registration flow:

1. User starts passkey/security key registration.
2. Server sends creation options:
   - challenge
   - relying party ID
   - user ID
   - allowed algorithms
   - authenticator preferences
   - attestation preference

3. Browser calls navigator.credentials.create().
4. Authenticator verifies user presence/user verification.
5. Authenticator creates key pair:
   - private key stays in authenticator
   - public key returned to server

6. Authenticator signs attestation data.
7. Browser returns credential to server.
8. Server verifies:
   - challenge
   - origin
   - RP ID
   - attestation if required
   - algorithm
9. Server stores:
   - credential ID
   - public key
   - sign counter if applicable
   - user handle

Simplified:

Server stores public key.
Authenticator keeps private key.

WebAuthn authentication ceremony

Authentication flow:

1. User enters username or uses discoverable credential.
2. Server sends challenge.
3. Browser calls navigator.credentials.get().
4. Browser includes origin and RP ID.
5. Authenticator verifies user presence or biometrics/PIN.
6. Authenticator signs challenge + authenticator data.
7. Server verifies signature with stored public key.
8. Server validates:
   - challenge
   - origin
   - RP ID hash
   - user verification flag
   - signature counter where applicable
9. Login succeeds.

Key property:

Private key never leaves authenticator.
Challenge prevents replay.
Origin binding prevents phishing.

Why origin binding prevents phishing

OTP phishing:

User can type OTP into any website.

WebAuthn:

Authenticator signs for a specific relying party ID.

Example:

Legitimate site:

https://accounts.example.com
RP ID: example.com

Phishing site:

https://accounts.example-login.com
RP ID: example-login.com

The authenticator will not produce a valid assertion for example.com while the user is on example-login.com.

Senior phrasing:

OTP depends on user judgment. WebAuthn depends on cryptographic origin binding enforced by the browser and authenticator.

Platform-bound vs roaming hardware authenticators

Platform-bound authenticator

Examples:

- Windows Hello
- macOS Touch ID
- iPhone Face ID passkey
- Android device passkey

Pros:

- Very usable
- Built into device
- Supports biometrics/PIN
- Good adoption

Cons:

- Device loss/replacement creates recovery need
- May be synced depending on ecosystem
- Enterprise control varies by platform

Roaming hardware authenticator

Examples:

- YubiKey
- Titan Security Key
- Smart card

Pros:

- Strong separation from endpoint
- Portable across devices
- Excellent for administrators
- Easier to reason about as hardware possession factor

Cons:

- Users lose keys
- Enrollment logistics
- Backup key required
- More support overhead

Recommended design:

Standard users:
  Platform passkeys + safe recovery

Privileged users:
  Hardware security key + backup hardware key
  Possibly platform passkey only as secondary/limited factor

Synced passkeys and changed trust assumptions

Passkeys may be synced across devices using cloud accounts such as Apple, Google, or Microsoft ecosystems.

This improves usability and recovery, but changes the trust model.

With device-bound key:

Compromise requires access to that physical device/authenticator.

With synced passkey:

Compromise may involve:
  - cloud account compromise
  - device ecosystem recovery weakness
  - malicious enrollment into sync account
  - platform provider trust

This does not make synced passkeys “bad.” It means the relying party must understand the assurance level.

For consumer apps:

Synced passkeys are often a major improvement over passwords and OTP.

For privileged enterprise admin:

May require hardware-bound, enterprise-managed authenticators.

Senior phrasing:

Synced passkeys improve adoption and reduce lockout, but they move some trust into the platform account and its recovery process. For admin access, I would be more conservative.

Attestation and authenticator policy

Attestation can tell the relying party something about the authenticator model or manufacturer.

Enterprise may care about:

- Is this a hardware-backed authenticator?
- Is it enterprise-approved?
- Is user verification supported?
- Is the key exportable?
- Is it FIPS-certified or policy-approved?
- Is it synced or device-bound?

Tradeoff:

Strict attestation:
  + Strong control
  - User friction
  - Privacy concerns
  - Device compatibility issues

Loose attestation:
  + Better adoption
  - Less assurance over authenticator type

Enterprise design:

Standard workforce:
  Allow broad platform authenticators

Privileged users:
  Require approved security keys
  Require user verification
  Require backup key
  Track credential inventory

Hardest part of passkey-first fallback and recovery

The hardest part is avoiding a weaker fallback that destroys the assurance of passkeys.

Bad design:

Primary login:
  Passkey

Fallback:
  SMS OTP reset

Actual security:
  SMS OTP

Better design:

- Multiple passkeys registered
- Backup hardware key
- Recovery codes
- Managed device recovery
- Admin-assisted recovery with strong verification
- Cooling-off period for sensitive changes
- Revoke old sessions and credentials after recovery

Senior phrasing:

Passkey-first systems are easy to make secure at the front door and hard to make secure at the side door. Recovery design is the hardest part.

6. Passwordless authentication: magic links, device certificates, and assurance

Why passwordless is not automatically phishing-resistant

Passwordless only means no password is used.

It does not automatically mean phishing-resistant.

Examples:

Method Passwordless? Phishing-resistant?
Email magic link Yes Usually no
SMS OTP login Yes No
TOTP-only login Yes No
Push-only login Yes Usually no
Passkey Yes Yes
Device certificate + user verification Yes Strong, depending design
Federated SSO Passwordless to app Depends on IdP auth

Senior phrasing:

Passwordless removes passwords. Phishing resistance requires origin-bound or sender-constrained cryptographic proof. Those are different properties.

Magic links vs passkeys vs device certificates

Magic links

Flow:

1. User enters email.
2. App emails login link.
3. User clicks link.
4. App creates session.

Pros:

- Very easy
- No password storage
- Good for low-risk consumer apps

Risks:

- Email account becomes primary factor
- Email forwarding rules can steal links
- Mailbox compromise equals account compromise
- Link leakage through logs or browser history
- Phishing can trick users into clicking attacker-controlled links
- Possession-only unless combined with device verification

Use for:

- Low-risk applications
- Account bootstrap
- Recovery with additional controls

Avoid as sole factor for:

- Admin access
- Financial systems
- Sensitive enterprise apps

Passkeys

Pros:

- Phishing-resistant
- No shared secret
- Good usability

- Biometric/PIN local verification

Risks:

- Recovery complexity
- Synced passkey trust model
- Enterprise policy complexity

Use for:

- Consumer login
- Workforce login
- High-value accounts with proper recovery

Device certificates

Flow:

Device has private key/certificate.
Server verifies certificate during TLS or application auth.

Pros:

- Strong device identity
- Great for managed enterprise environments

- Useful for zero trust access

- Device possession alone may not prove user presence

Best design:

Device certificate + user authentication + device posture

Access to internal admin portal requires:
  - Managed device certificate

Trust assumptions when email becomes primary login factor

If magic links are primary login, then email security becomes account security.

You now depend on:

- User's email provider security
- Email account MFA
- Email recovery process
- Mailbox forwarding rules
- Mail client security
- Link confidentiality
- Phishing resistance of email flow

Attack:

Possession-only means:

Problem:

Example:

Login method: Sign in with Google/Microsoft/Okta

  Passwordless

  Maybe password + MFA

Senior phrasing:

Federation externalizes authentication. It may remove passwords from the application, but it does not automatically remove passwords from the identity ecosystem. The app inherits the IdP’s assurance, recovery, MFA, session, and compromise model.

Controls for relying app:

- Require specific IdP assurance level
- Check authentication context/class reference

Real-world attack scenarios

Scenario 1: Push fatigue against employee VPN

Context:
  Employee password leaked from third-party breach.

Attack:
  Attacker attempts VPN login repeatedly.
  Employee receives many push prompts.
  Employee approves one.

Impact:
  Attacker obtains internal access.

- Number matching
- Push rate limits
- Alert on repeated denied prompts
- Phishing-resistant MFA for VPN
- Impossible travel detection
- User education

Senior point:

Push is better than password-only, but it is still human-mediated approval. For high-risk access, use FIDO2 or certificate-backed access.

Scenario 2: Password spraying against enterprise SSO

Context:
  Attacker tries "Welcome2026!" against 20,000 employee accounts.

Attack:
  Low rate per account.
  Distributed infrastructure.
  Avoids account lockout.

Impact:
  One weak password succeeds.

- Tenant-wide spray detection
- Common password blocklist
- Smart lockout
- MFA
- Risk-based conditional access
- Disable legacy auth protocols

Account-level lockout alone is not sufficient because spraying distributes attempts across users.

Scenario 3: Session theft after FIDO login

- CSP
- Re-auth for dangerous actions
- Session binding
- Short admin sessions
- Audit sensitive changes

Senior point:

Phishing-resistant login does not eliminate session-layer attacks.

Scenario 4: Helpdesk recovery bypass

Context:
  Cloud admin uses hardware security key.

  Attacker calls helpdesk claiming lost key.
  Helpdesk resets MFA using weak verification.

  Admin account takeover.

- Two-person approval
- Manager/security approval
- Callback to trusted channel
- Cooling-off period
- Revoke old sessions
- High-risk recovery audit

Recovery defines the assurance floor.

Scenario 5: Magic-link account takeover via email compromise

Context:
  SaaS app uses email magic links.

Attack:
  Attacker compromises mailbox.
  Requests login link.
  Clicks link.
  Gains SaaS access.

  Passwordless login fails because email became the single factor.

- Email security monitoring
- Passkeys for high-risk users

Senior point:

Passwordless does not imply phishing-resistant or high assurance.

Authentication design principles for senior-level discussions

1. Match control strength to risk

Standard user:
  Passkey or app MFA

Privileged admin:
  Hardware key + managed device + JIT access

Break-glass:
  Offline, monitored, heavily restricted

2. Avoid weak fallback paths

Strong primary auth + weak recovery = weak system

3. Treat sessions as credentials

Session cookies and refresh tokens are bearer credentials.
Protect, rotate, revoke, and monitor them.

4. Prefer phishing-resistant authentication for high-risk access

FIDO2/WebAuthn/passkeys
Hardware security keys
Smart cards
Device certificates with user verification

5. Design for lifecycle

Enrollment

Device change
Factor change
Recovery
Logout
Revocation
Incident response

6. Monitor authentication behavior

Useful signals:

- Failed login spikes
- Password spraying patterns
- Credential stuffing patterns
- MFA fatigue patterns
- New device + new location
- Recovery request followed by sensitive change
- MFA reset followed by admin action
- Impossible travel
- Suspicious session reuse

Strong closing answer

A mature authentication system is not just password plus MFA. It is a layered assurance system. MFA should be viewed as a family of controls, with FIDO2/passkeys providing fundamentally stronger phishing resistance because authentication is cryptographically bound to the legitimate origin. Password systems should use salted slow hashes and focus on breach resistance rather than periodic rotation. Sessions must be treated as credentials, rotated after authentication, protected with secure cookies, and revocable across distributed systems. Account recovery is the real assurance floor because attackers will bypass strong primary authentication through weak fallback paths. Passwordless improves usability, but only some passwordless methods, such as WebAuthn/passkeys, are truly phishing-resistant. For privileged users, I would combine phishing-resistant MFA, managed device trust, short sessions, step-up authentication, just-in-time access, strong recovery controls, and high-fidelity monitoring.

13. OAuth 2.0: authorization framework, flows, scopes, and token lifecycle

Why is OAuth 2.0 an authorization framework and not an authentication protocol, why did the ecosystem move from the implicit flow to authorization code with PKCE, and why is redirect URI validation security-critical? How do scopes at the authorization server relate to actual authorization decisions at the resource server, how would you design refresh token rotation for mobile and browser clients, and how would you harden an OAuth deployment used by thousands of third-party integrations?

1. Core answer

OAuth 2.0 is an authorization framework because its primary purpose is to let a client application obtain limited delegated access to a protected resource. It answers:

“Can this app access this resource, on behalf of this user, with these permissions?”

It does not reliably answer:

“Who is the user, how was the user authenticated, and what identity claims can I trust?”

That identity problem is handled by OpenID Connect, which adds an ID token and authentication semantics on top of OAuth 2.0. The OAuth 2.0 specification itself describes OAuth as a framework that enables a third-party application to obtain limited access to an HTTP service. (IETF Datatracker)

A simple real-world example:

A user authorizes calendar-sync.example.com to read their Google Calendar. OAuth gives the app an access token with permission like calendar.readonly. That does not mean the app has authenticated the user for login to its own application. It only means the app has been granted delegated access to a resource.

2. Why OAuth 2.0 is not authentication

The common mistake is this:

“I received an OAuth access token, therefore the user is logged in.”

That is unsafe.

An OAuth access token is meant for the resource server, not necessarily for the client application. The token may be opaque, audience-bound to an API, scoped to a resource, and not designed to convey user identity to the client. OAuth does not standardize a user profile, session, authentication time, authentication method, or identity assurance level.

Bad pattern

Client app receives access_token
Client app calls /userinfo-like endpoint or decodes token
Client app treats token subject as logged-in user

This can fail because:

Problem Why it matters
Token audience may be for another API Client may accept a token not meant for it
Access token may not contain identity claims It is an authorization artifact, not identity proof
No authentication context Client may not know MFA, login freshness, or assurance level
Token substitution is possible Attacker may inject a valid token issued for a different client/resource

Correct approach

Use OpenID Connect for authentication. OIDC adds:

Artifact Purpose
id_token Identity assertion for the client
aud Confirms the token is intended for this client
iss Confirms trusted identity provider
nonce Binds authentication response to the client request
auth_time, acr, amr Login freshness and authentication method context

in practical terms:

OAuth delegates access. OIDC authenticates the user. Confusing the two creates token substitution, confused deputy, and broken login vulnerabilities.

3. Why the ecosystem moved from implicit flow to authorization code with PKCE

The old implicit flow was designed for browser-based clients when browsers had limited ability to securely perform back-channel token exchanges. In implicit flow, the authorization server returns the access token directly through the browser redirect.

User Browser
   |
   | 1. Login and consent
   v
Authorization Server
   |
   | 2. Redirect with access_token in URL fragment
   v
Single Page App

Example:

https://app.example.com/callback#access_token=eyJ...

That design exposes tokens to browser-side risks:

Risk Explanation
Token appears in front-channel Token passes through browser-controlled surfaces
JavaScript exposure XSS can steal token
Browser history / extensions Tokens may leak to browser components
No back-channel client verification Token is delivered directly to a public client
Harder sender binding Bearer token can be replayed if stolen

Modern guidance has moved away from implicit flow. OAuth 2.0 Security Best Current Practice deprecates less secure modes, and OAuth 2.1 omits implicit grant and requires PKCE for authorization code flow. (IETF Datatracker)

4. Why authorization code with PKCE is safer

With authorization code + PKCE, the browser receives only a short-lived authorization code. The actual token exchange happens separately, and the client must prove it started the flow using a secret-like one-time value called the code_verifier.

Flow

Client generates:
  code_verifier = random high-entropy string
  code_challenge = BASE64URL(SHA256(code_verifier))

Step 1: Authorization request

Client -> Authorization Server:
  response_type=code
  client_id=mobile-app-123
  redirect_uri=com.example.app:/callback
  scope=profile calendar.read
  state=random_csrf_value
  code_challenge=abc123
  code_challenge_method=S256

Step 2: User authenticates and consents

Authorization Server -> Browser/App:
  redirect_uri?code=AUTH_CODE&state=random_csrf_value

Step 3: Token exchange

Client -> Token Endpoint:
  grant_type=authorization_code
  code=AUTH_CODE
  redirect_uri=com.example.app:/callback
  code_verifier=original_random_value

Authorization Server:
  SHA256(code_verifier) == original code_challenge?
  If yes, issue tokens.

PKCE was created to mitigate authorization code interception attacks for public clients. The attacker may steal the authorization code, but without the original code_verifier, the code is useless. (IETF Datatracker)

Real-world example

Imagine a mobile banking app uses OAuth to connect to an investment portfolio API.

Without PKCE:

Attacker app registers same custom URI scheme:
  com.bank.app:/callback

Victim authenticates.
Authorization code is delivered to attacker app.
Attacker exchanges code for tokens.

With PKCE:

Attacker steals code.
Attacker does not know code_verifier.
Token endpoint rejects exchange.

That is the main security improvement: the app that starts the flow must be the app that completes the flow.

5. Why redirect URI validation is security-critical

Redirect URI validation is one of the most important OAuth controls because the authorization server sends sensitive authorization results to the redirect URI.

Depending on the flow, that result may be:

Flow Redirect contains
Implicit Access token
Authorization code Authorization code
OIDC hybrid Code, ID token, or token
Error flow Error details and state

If redirect URI validation is weak, the attacker can cause the authorization server to send codes or tokens to an attacker-controlled location.

Vulnerable example

Registered redirect URI:

https://client.example.com/callback

Authorization request sent by attacker:

https://auth.example.com/authorize?
  client_id=trusted-client
  &redirect_uri=https://evil.example.net/callback
  &response_type=code

If the authorization server accepts this, it will send the victim’s authorization code to the attacker:

https://evil.example.net/callback?code=AUTH_CODE

OAuth 2.1 guidance requires exact redirect URI matching, and security best practice strongly discourages unsafe patterns. (IETF Datatracker)

Dangerous redirect validation patterns

Bad pattern Why dangerous
Prefix matching https://client.example.com.evil.com may pass
Wildcards https://*.example.com/callback may include compromised subdomains
Open redirect chaining Valid redirect points to /redirect?url=https://evil.com
Case/encoding confusion %2e, punycode, double encoding bypasses
Allowing query-controlled redirect Attacker controls final destination
Custom scheme collision Mobile apps can claim same private URI scheme

Strong validation

Registered:
  https://app.example.com/oauth/callback

Accepted:
  https://app.example.com/oauth/callback

Rejected:
  https://app.example.com/oauth/callback/
  https://app.example.com/oauth/callback?next=https://evil.com
  https://app.example.com.evil.com/oauth/callback
  https://evil.com/callback

For native apps, app-claimed HTTPS redirect URIs or loopback redirects are generally safer than private custom schemes, because private-use URI schemes may be claimable by multiple apps depending on the OS. RFC 8252 discusses redirect URI patterns and security considerations for native apps. (RFC Editor)

6. How scopes relate to actual authorization decisions

Scopes are often misunderstood.

A scope is a permission label granted by the authorization server. It says what the client is allowed to ask for or what category of access was delegated.

But the resource server still has to make the real authorization decision.

Example

Token contains:

{
  "iss": "https://auth.example.com",
  "sub": "user-123",
  "aud": "https://api.payments.example.com",
  "client_id": "expense-app",
  "scope": "payments.read payments.create",
  "tenant_id": "tenant-a"
}

API request:

POST /payments HTTP/1.1
Host: api.payments.example.com
Authorization: Bearer eyJ...
Content-Type: application/json

{
  "fromAccount": "acct-999",
  "amount": 50000,
  "currency": "SGD"
}

The resource server should not simply say:

scope contains payments.create -> allow

It must also check:

Check Example
Token issuer Is this from trusted authorization server?
Token audience Is token meant for this API?
Scope Does token permit payment creation?
Subject entitlement Can user-123 create payments?
Tenant boundary Does user belong to tenant-a?
Resource ownership Can user access acct-999?
Transaction policy Is amount within limit?
Step-up requirement Does this action require MFA?
Client trust Is this third-party app allowed for high-risk actions?

Better resource server decision

Allow only if:
  token.iss == trusted issuer
  token.aud == payments API
  token.scope contains payments.create
  user has role PaymentInitiator
  account belongs to user's tenant
  amount <= user's approval limit
  risk score acceptable
  recent MFA present for high-risk payment

In plain terms:

Scopes are coarse-grained delegated permissions. Resource servers must combine scopes with local policy, resource ownership, tenant boundaries, risk signals, and business authorization.

7. Token lifecycle: access tokens, refresh tokens, and rotation

A robust OAuth deployment needs a clear lifecycle for each token type.

Token Lifetime Stored where Purpose
Authorization code Seconds to minutes Temporary server-side state Exchange for tokens
Access token Short-lived, often minutes Client memory or secure storage API access
Refresh token Longer-lived Strongly protected storage Obtain new access tokens
ID token Short-lived Client Authentication assertion in OIDC

OAuth 2.1 guidance says refresh tokens for public clients should be sender-constrained or one-time use. (OAuth Community Site)

8. Designing refresh token rotation for mobile clients

Mobile apps are public clients. They cannot safely keep a client secret because secrets embedded in mobile binaries can be extracted.

Recommended mobile design

Use:

  1. Authorization code + PKCE
  2. System browser, not embedded webview
  3. Short-lived access tokens
  4. Refresh token rotation
  5. Secure OS storage
  6. Device binding where possible
  7. Reuse detection

Rotation model

Every time the app uses a refresh token, the authorization server returns:

{
  "access_token": "new-access-token",
  "expires_in": 900,
  "refresh_token": "new-refresh-token"
}

The old refresh token is invalidated.

RT1 -> used -> issue RT2, invalidate RT1
RT2 -> used -> issue RT3, invalidate RT2
RT3 -> used -> issue RT4, invalidate RT3

Reuse detection

If an old refresh token appears again:

RT2 already used.
Someone presents RT2 again.
This means token theft or race condition.

The authorization server should:

Action Reason
Revoke the active refresh token family Prevent attacker continuing chain
Revoke related access tokens where possible Reduce active exposure
Require re-authentication Recover trust
Notify user / client Security visibility
Log high-severity event Detection and investigation

Mobile storage

Platform Storage
iOS Keychain, optionally Secure Enclave-backed keys
Android Keystore-backed encrypted storage
Cross-platform Avoid plain SQLite, shared preferences, logs, crash dumps

Mobile race condition handling

Mobile apps can create accidental duplicate refreshes due to poor network conditions.

Example:

App sends refresh request with RT5.
Network times out.
App retries with RT5.
Server already issued RT6.

If strict reuse detection immediately revokes the family, legitimate users suffer.

A practical design includes a short grace window:

If RT5 is reused within 5-15 seconds from same device binding and same client instance:
  return same RT6 or tolerate once

If RT5 is reused later, from different IP/device/fingerprint:
  revoke token family

Mobile refresh token table

refresh_token_id
token_family_id
client_id
user_id
device_id
status: active | used | revoked
issued_at
used_at
rotated_to_token_id
last_ip
last_user_agent
cnf/device_binding_thumbprint
risk_score

Browser clients are more dangerous because JavaScript is exposed to XSS.

For SPAs, I would avoid long-lived refresh tokens in JavaScript-accessible storage.

Better browser architecture: BFF pattern

Use a Backend-for-Frontend.

Browser
  |
  | Secure, HttpOnly, SameSite cookie
  v
BFF Server
  |
  | Stores refresh token server-side
  v
Authorization Server
  |
  v
Resource APIs

Cookie settings

Set-Cookie: session=abc;
  HttpOnly;
  Secure;
  SameSite=Lax;
  Path=/;

For higher-risk apps:

CSRF token for state-changing requests
Content Security Policy
No refresh tokens in localStorage/sessionStorage

If SPA must receive refresh tokens

Then use:

Control Purpose

Short absolute lifetime Prevent indefinite abuse
In-memory access token Reduce persistence
No localStorage Reduce XSS token theft
Strong CSP Reduce XSS probability
DPoP / sender-constrained tokens where supported Reduce replay value

But for a high-risk Google-scale environment, I would strongly prefer BFF or token handler architecture for browser clients.

10. Attack scenario: OAuth authorization code theft through weak redirect validation

Scenario

A SaaS company allows thousands of third-party apps to integrate with its API.

A trusted third-party app has this registered redirect URI:

https://partner.example.com/oauth/callback

The authorization server uses prefix matching:

redirect_uri startsWith "https://partner.example.com"

The attacker registers:

https://partner.example.com.evil.net/oauth/callback

Victim clicks:

https://auth.saas.com/authorize?
  client_id=trusted-partner
  &response_type=code
  &scope=files.read
  &redirect_uri=https://partner.example.com.evil.net/oauth/callback

Authorization server accepts the redirect URI and sends the code to attacker.

If PKCE is missing or misconfigured, attacker exchanges the code for tokens.

Impact

Attacker gains files.read access to victim's data through trusted client context.

For third-party integrations:

Low-risk read-only Rotation + inactivity expiry
Sensitive user data Rotation + reuse detection + app review

Suspicious clients No refresh token or very short lifetime

Recommended lifecycle:

Access token TTL: 5-15 minutes
Refresh token idle timeout: 30-90 days
Refresh token absolute lifetime: risk-based
Rotation: every use
Reuse detection: revoke token family
User/admin revocation: immediate
Client compromise: revoke all client token families

E. Consent and scope hardening

Consent screens should make risk understandable.

Bad consent:

[Allow]

Good consent:

ExpenseBot wants to:
- Read your Drive file names
- Read files you explicitly select
- Create calendar events

It will not be able to:
- Delete files
- Read all emails
- Change your password

Scope strategy:

Principle Example
Least privilege calendar.events.readonly instead of calendar.full_access
Granular scopes Separate read, write, delete, admin

Do not let each API implement OAuth validation differently.

Use shared libraries or gateway enforcement for:

But avoid putting all business authorization only in the gateway.

Good split:

Layer Responsibility

Resource service Tenant, ownership, business policy

15. Closing answer

OAuth 2.0 is authorization, not authentication, because it grants delegated access to resources; it does not by itself prove user identity to the client. For authentication, use OpenID Connect.

The industry moved from implicit flow to authorization code with PKCE because implicit flow exposed bearer tokens through the browser front channel. Authorization code with PKCE keeps tokens out of the redirect response and binds the code exchange to the client that initiated the flow.

Redirect URI validation is security-critical because the authorization server sends authorization codes or tokens to that URI. Weak validation turns OAuth into a token exfiltration mechanism.

Scopes are necessary but not sufficient. The authorization server grants scopes, but the resource server must still enforce issuer, audience, tenant, ownership, user entitlement, client trust, and business policy.

For refresh tokens, I would use rotation, reuse detection, short-lived access tokens, secure client storage, and sender-constrained tokens where possible. For browsers, I would prefer a BFF pattern so refresh tokens never enter JavaScript.

For thousands of third-party integrations, I would harden the ecosystem with strict client registration, verified redirect URIs, sensitive scope review, app risk tiers, PKCE everywhere, disabled implicit flow, token telemetry, anomaly detection, revocation workflows, and consistent resource-server authorization libraries.
