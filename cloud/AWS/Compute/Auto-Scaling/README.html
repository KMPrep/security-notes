Auto Scaling — AWS Auto Scaling

Category: Compute
AWS Service Type: Elasticity and Capacity Management
Key Docs: Auto Scaling User Guide, EC2 Auto Scaling

Overview

AWS Auto Scaling automatically adjusts compute capacity to maintain performance and minimize cost. It operates across multiple services: EC2 Auto Scaling Groups (ASGs) for EC2 capacity, Application Auto Scaling for ECS services, DynamoDB, Aurora, and Lambda concurrency, and AWS Auto Scaling plans for coordinated multi-service scaling. EC2 ASGs are the foundational mechanism — they maintain a desired, minimum, and maximum count of instances, replace unhealthy ones, and span multiple Availability Zones. Scaling policies include target tracking (e.g., maintain 50% CPU), step scaling, and scheduled scaling. Predictive scaling uses ML to forecast demand.

How It Works

Core Mechanics

Concept Description

Launch template Instance configuration blueprint: AMI, instance type, security group, IAM role
Scaling policy Rules for when and how to scale; target tracking, step, or scheduled
Target tracking Maintains a metric at a target value; adjusts capacity automatically
Cooldown period Pause between scale actions to let metrics stabilize
Health check EC2 or ELB-based check; failed instances are replaced automatically
Mixed instances policy Combines multiple instance types and Spot/On-Demand ratios for cost/resilience
Lifecycle hooks Custom actions during instance launch or termination (e.g., drain connections)

Top 3 Critical Mitigations

  1. Use launch templates, not launch configurations: Launch configurations are legacy. Launch templates support versioning, mixed instances, and latest AMI resolution. Always reference AMI by SSM parameter or latest approved version.
  2. Span ASGs across at least 3 Availability Zones: Single-AZ designs create a complete blast radius that an AZ event can trigger. Multi-AZ with AZ rebalancing ensures capacity is redistributed after AZ recovery.
  3. Use lifecycle hooks for graceful drain: Instance termination without a lifecycle hook can cut active connections. Use termination hooks to let the load balancer drain connections completely before the instance stops.

AWS Auto Scaling Model Answers — Detailed with Examples

1. What is the difference between target tracking, step scaling, and scheduled scaling policies?

Amazon EC2 Auto Scaling supports different policy types depending on whether you want scaling to be metric-driven, threshold-driven, or time-driven.

A. Target tracking scaling

Target tracking is the simplest and most commonly recommended dynamic scaling method.

You define a target value, and Auto Scaling adjusts capacity to keep the metric close to that target. AWS describes target tracking as automatically scaling the Auto Scaling group based on a target metric value. (AWS Documentation)

Example:

Keep average CPU utilization around 50%.

If CPU goes above 50%, Auto Scaling adds instances.
If CPU goes far below 50%, Auto Scaling removes instances.

Example policy:

Metric: Average CPU utilization
Target value: 50%

Good for:

Web applications
API services
Worker fleets
Steady metric-to-capacity relationships

Mental model:

Target tracking = "Keep this metric near this value."

Example:

If 4 instances produce 80% CPU,
Auto Scaling may add more instances until average CPU drops closer to 50%.

B. Step scaling

Step scaling uses CloudWatch alarms and predefined scaling steps. AWS documents that step scaling policies scale capacity in predefined increments based on CloudWatch alarm breaches. (AWS Documentation)

Example:

If CPU > 60%, add 1 instance.
If CPU > 75%, add 2 instances.
If CPU > 90%, add 4 instances.

Step scaling gives more control than target tracking.

Example step policy:

Alarm Condition Scaling Action
CPU 60%–75% Add 1 instance
CPU 75%–90% Add 2 instances
CPU > 90% Add 4 instances

Good for:

Applications with known scaling behavior
Queue workers
Batch processors
Systems where scaling impact is not linear

Mental model:

Step scaling = "If the metric crosses this threshold, scale by this amount."

C. Scheduled scaling

Scheduled scaling changes capacity at known times. AWS says scheduled actions tell EC2 Auto Scaling to scale at specific times, optionally changing desired, minimum, and maximum capacity. (AWS Documentation)

Example:

Every weekday at 8:00 AM, scale out to 20 instances.
Every weekday at 8:00 PM, scale in to 4 instances.

Good for predictable traffic:

Office-hour business apps
Market open / close workloads
Monthly payroll processing
Known campaign launches
Exam registration windows

Mental model:

Scheduled scaling = "Scale at this known time."

2. How does Auto Scaling handle an Availability Zone that becomes impaired?

An Auto Scaling Group, or ASG, is designed to maintain desired capacity across enabled Availability Zones. AWS says Auto Scaling monitors instance health and replaces terminated or impaired instances to maintain desired capacity. (AWS Documentation)

In normal operation, Auto Scaling tries to keep instances balanced across Availability Zones. If launching in one AZ fails, Auto Scaling can attempt to launch instances in another healthy AZ. AWS describes this as “balanced best effort.” (AWS Documentation)

Normal multi-AZ ASG

ASG Desired Capacity: 6

AZ-A: 2 instances
AZ-B: 2 instances
AZ-C: 2 instances

If AZ-B becomes impaired

Auto Scaling may not be able to launch or maintain healthy instances in AZ-B.

Possible result:

AZ-A: 3 instances
AZ-B: impaired
AZ-C: 3 instances

The group may become temporarily imbalanced to preserve capacity.

Important modern feature: zonal shift / AZ impairment behavior

AWS now supports an Availability Zone impairment policy for Auto Scaling groups integrated with ARC zonal shift. The policy can specify whether unhealthy instances in the impaired zone should be replaced or ignored during an active zonal shift. (AWS Documentation)

Two behaviors:

Behavior Meaning
Replace unhealthy Replace unhealthy instances, including in the impaired zone
Ignore unhealthy Do not replace unhealthy instances in the impaired zone during active zonal shift

Why this matters

If an AZ is impaired, blindly replacing instances inside that same AZ can be wasteful or harmful.

Example:

AZ-B networking is impaired.
Instances in AZ-B fail health checks.
ASG keeps replacing instances in AZ-B.
New instances also fail.

Better behavior may be:

Shift traffic away from AZ-B.
Ignore unhealthy instances in AZ-B temporarily.
Launch or serve capacity from healthy AZs.

Model answer

Auto Scaling tries to maintain desired capacity and balance instances across Availability Zones. If one AZ is impaired or launches fail there, Auto Scaling can launch capacity in healthy AZs on a best-effort basis. With ARC zonal shift integration, you can also configure how the ASG handles unhealthy instances in an impaired zone, such as replacing them or ignoring them during the impairment. The design goal is to preserve application capacity without continuously recycling instances in a broken zone.

3. How would you design an ASG that scales on a custom application metric rather than CPU?

CPU is not always the best scaling signal.

For example, a payment API may have low CPU but high request latency. A worker fleet may have low CPU but a growing queue backlog.

Example problem

Application: Image processing workers
Metric that matters: Number of pending jobs per instance
CPU: Not reliable

Better metric:

QueueBacklogPerInstance = PendingMessages / InServiceInstances

Architecture

Application / Worker
        |
        | publish custom metric
        v
Amazon CloudWatch
        |
        | alarm or target tracking metric
        v
EC2 Auto Scaling Policy
        |
        v
Auto Scaling Group

Step-by-step design

Step 1: Choose the right custom metric

Good custom metrics:

RequestsPerTarget
QueueBacklogPerInstance
ActiveConnectionsPerInstance
P95Latency
JobsWaiting
KafkaConsumerLag
CheckoutFailuresPerMinute

Bad custom metrics:

Total requests without normalizing by capacity
Average CPU when CPU is not the bottleneck
Memory if memory does not correlate with demand
Business metrics that lag too much

Step 2: Publish the metric to CloudWatch

Example metric:

Namespace: CustomApp/Workers
MetricName: QueueBacklogPerInstance
Dimensions:
  AutoScalingGroupName = image-worker-asg
Period: 60 seconds

Application or a sidecar publishes:

Pending queue messages / healthy worker instances

Step 3: Create scaling policy

You can use:

Target tracking with customized metric
Step scaling with CloudWatch alarms
Predictive scaling with custom metrics for suitable patterns

AWS supports customized metrics in predictive scaling too, provided the metrics are published to CloudWatch with the needed data frequency. (AWS Documentation)

Example target tracking policy

Metric: QueueBacklogPerInstance
Target value: 100

Meaning:
Try to keep each worker processing around 100 queued jobs.

If backlog grows:

QueueBacklogPerInstance = 300
Target = 100
Action = scale out

If backlog falls:

QueueBacklogPerInstance = 20
Target = 100
Action = scale in

Key design considerations

Area Recommendation
Metric frequency Publish every 1 minute if possible
Metric quality Use a metric that scales inversely with capacity
Cooldowns Prevent overreaction
Scale-in protection Avoid killing busy workers
Lifecycle hooks Drain work before termination
Alarms Alert when backlog grows despite scale-out
Max capacity Set based on downstream limits

Model answer

I would first identify the metric that best represents application pressure, such as queue backlog per instance, request latency, or active sessions per instance. Then I would publish that metric to CloudWatch with the ASG name as a dimension. I would configure a target tracking or step scaling policy using that custom metric. For worker workloads, I would usually scale on backlog per instance and combine it with lifecycle hooks so scale-in does not terminate instances that are still processing jobs.

4. What is the purpose of a warm pool, and when would you use one?

A warm pool is a pool of pre-initialized EC2 instances associated with an Auto Scaling Group.

The purpose is to reduce scale-out latency.

Instead of launching a brand-new instance from scratch during traffic spikes, Auto Scaling can move a pre-warmed instance into service.

Normal scale-out

Traffic spike
   ↓
Launch EC2 instance
   ↓
Boot OS
   ↓
Install agents
   ↓
Download application
   ↓
Load configuration
   ↓
Warm cache / JVM / runtime
   ↓
Register with load balancer
   ↓
Serve traffic

This may take several minutes.

With warm pool

Traffic spike
   ↓
Take instance from warm pool
   ↓
Finalize startup
   ↓
Register with load balancer
   ↓
Serve traffic

Much faster.

When to use warm pools

Use warm pools when instance startup is slow or expensive.

Examples:

Java applications with long JVM warm-up
ML inference servers loading large models
Windows servers with long boot times
Apps requiring heavy bootstrap scripts
Game servers
Security agents that take time to initialize
Large container image pulls

Example

ML inference service
Model size: 20 GB
Cold start time: 12 minutes
Traffic spike tolerance: 2 minutes

A warm pool helps because instances can pre-load the model before being needed.

Tradeoffs

Benefit Cost
Faster scale-out You may pay for warm capacity
Better spike handling More operational complexity
Reduced cold starts Need patching/update strategy for warm instances
Predictable readiness Warm pool sizing must be tuned

Model answer

A warm pool reduces scale-out latency by keeping pre-initialized instances ready before they are needed. I would use it for workloads with slow bootstrapping, such as Java applications, Windows workloads, ML inference services, or systems that need to download large artifacts before serving traffic. The tradeoff is cost and operational complexity, because warm instances must still be managed, patched, and sized appropriately.

5. How do lifecycle hooks enable zero-downtime scale-in operations?

Lifecycle hooks allow you to pause an instance during launch or termination so custom actions can run before the ASG continues.

For zero-downtime scale-in, the important hook is the termination lifecycle hook.

Problem without lifecycle hook

ASG decides to terminate instance
        ↓
Instance is killed
        ↓
Active requests/jobs are interrupted
        ↓
Users see errors or jobs fail

With lifecycle hook

ASG decides to terminate instance
        ↓
Lifecycle hook pauses termination
        ↓
Instance enters Terminating:Wait
        ↓
App drains traffic or finishes jobs
        ↓
Instance deregisters from load balancer
        ↓
Hook completes
        ↓
Instance terminates safely

Example: Web application behind ALB

Scale-in flow:

1. ASG selects instance for termination.
2. Lifecycle hook triggers.
3. Instance is marked as draining in the load balancer.
4. New requests stop going to that instance.
5. Existing requests complete.
6. App flushes logs and metrics.
7. Hook sends CompleteLifecycleAction.
8. ASG terminates the instance.

Example: Queue worker

1. ASG selects worker for termination.
2. Lifecycle hook triggers.
3. Worker stops accepting new jobs.
4. Worker completes current job.
5. Worker checkpoints progress.
6. Worker sends lifecycle completion signal.
7. ASG terminates instance.

Supporting controls

Control Purpose
ALB deregistration delay Allows in-flight requests to finish
Lifecycle hook timeout Gives app time to drain
Instance scale-in protection Prevents killing critical busy workers
SQS visibility timeout Prevents job loss
Health checks Ensures only ready instances serve traffic

Model answer

Lifecycle hooks enable zero-downtime scale-in by pausing instance termination and giving the application time to drain. For a web service, the instance can deregister from the load balancer and finish in-flight requests. For workers, it can stop accepting new jobs, finish current processing, checkpoint state, and then signal Auto Scaling to continue termination. This prevents request drops and job corruption during scale-in.

6. What are the tradeoffs of mixed instance policies with Spot and On-Demand?

Mixed instance policies allow an ASG to launch multiple instance types and purchase options, such as On-Demand and Spot.

This improves flexibility and cost efficiency.

Example policy

Base capacity: 4 On-Demand instances
Additional capacity:
  70% Spot
  30% On-Demand

Instance types:
  m7i.large
  m7a.large
  m6i.large
  c7i.large

Benefits

Benefit Explanation
Lower cost Spot can be much cheaper
Better capacity availability Multiple instance types improve placement options
Resilience Less dependency on one instance family
Flexible scaling Can use capacity-optimized allocation
Good for stateless workloads Easy to replace interrupted instances

Risks

Risk Explanation
Spot interruption Instances can be reclaimed
Capacity shortages Spot capacity may not be available
Application compatibility Different instance types may behave differently
Performance variance CPU/network performance may differ
Operational complexity More testing and observability needed
Bad for stateful workloads Interruption may cause data loss if not designed properly

Best practices

Use On-Demand base capacity for minimum reliability.
Use Spot for burst or fault-tolerant capacity.
Use multiple instance families and sizes.
Use capacity-optimized Spot allocation.
Keep workload stateless where possible.
Use lifecycle hooks and graceful shutdown.
Checkpoint long-running jobs.
Avoid using only the cheapest Spot pool.

Example good use case

Image processing workers
Stateless API replicas
CI/CD runners
Batch jobs
Data processing workers

Example bad use case

Single-node database
Stateful legacy application
Non-checkpointed long-running transaction processor
Critical control-plane service

Model answer

Mixed instance policies reduce cost and improve capacity flexibility by combining Spot and On-Demand across multiple instance types. The tradeoff is interruption risk, performance variability, and operational complexity. I would use On-Demand for baseline capacity and Spot for elastic or fault-tolerant capacity. The workload must handle interruption through stateless design, checkpointing, graceful shutdown, and lifecycle hooks.

7. How does predictive scaling work, and what are its limitations?

Predictive scaling uses historical traffic patterns to forecast future demand and proactively increase ASG capacity. AWS says predictive scaling analyzes historical load data to detect daily or weekly patterns and forecast future capacity needs. (AWS Documentation)

How it works

Historical load metrics
        ↓
Pattern detection
        ↓
Forecast future demand
        ↓
Schedule capacity ahead of time
        ↓
Instances are ready before traffic arrives

Example

An e-commerce app has predictable weekday traffic:

9:00 AM: traffic rises
12:00 PM: lunch peak
6:00 PM: evening drop

Predictive scaling can forecast:

At 8:45 AM, increase capacity from 10 to 25.
At 6:30 PM, reduce capacity from 25 to 8.

Good use cases

Daily business-hour traffic
Weekly payroll processing
Known recurring batch workloads
Retail sites with recurring peak periods
Streaming apps with regular evening demand

Limitations

Limitation Explanation
Needs historical data New workloads may not have enough history
Works best with patterns Random traffic is hard to predict
Not enough for sudden spikes Viral events or attacks may exceed forecast
Metric choice matters Bad metrics produce bad forecasts
Downstream bottlenecks Scaling EC2 does not scale databases automatically
Launch time still matters Prediction must happen early enough
Can overprovision Forecast may be higher than actual demand
Needs dynamic scaling too Predictive should usually be paired with target tracking

Design recommendation

Use predictive scaling plus target tracking:

Predictive scaling = handles expected demand
Target tracking = handles unexpected changes

Model answer

Predictive scaling forecasts future capacity from historical load patterns and scales out before demand arrives. It is useful for workloads with recurring daily or weekly patterns. Its limitations are that it needs historical data, performs poorly with random or one-off spikes, depends on good metrics, and can overprovision if the forecast is wrong. I would usually combine predictive scaling with target tracking so expected peaks are handled proactively and unexpected spikes are handled reactively.

8. How would you detect and remediate an ASG that is thrashing between scale-out and scale-in?

Thrashing means the ASG repeatedly scales out and scales in within a short period.

Example:

10:00 scale out from 4 to 8
10:05 scale in from 8 to 4
10:10 scale out from 4 to 8
10:15 scale in from 8 to 4

This causes instability, cost waste, cold starts, and user impact.

Common causes

Cause Explanation
Metric too sensitive Small changes trigger scaling
Cooldown too short ASG reacts before system stabilizes
Bad target value Target is too aggressive
Scale-in too aggressive Removes capacity too quickly
Startup delay ignored New instances not ready before metric changes
Wrong metric CPU does not represent actual load
Load imbalance Some instances hot, others idle
App warm-up time New instances initially slow
Conflicting policies Multiple policies fight each other
Scheduled + dynamic conflict Schedule scales in while target tracking scales out

Detection signals

Monitor:

GroupDesiredCapacity
GroupInServiceInstances
ScalingActivity events
CloudWatch alarm history
CPU / latency / request count
ALB TargetResponseTime
ALB 5XX count
Instance warm-up duration
Instance launch/termination count

Symptoms:

Frequent scaling activities
Capacity oscillation
Repeated instance launches and terminations
CloudWatch alarms flipping OK/ALARM
Latency spikes during scale-in
Application errors after termination

Example detection query logic

If ASG scaling activities > N times in 30 minutes
AND desired capacity repeatedly increases and decreases
THEN flag ASG thrashing

Remediation approach

A. Increase cooldown or instance warm-up

If instances take 5 minutes to become useful, do not evaluate too aggressively.

App warm-up time: 7 minutes
Set instance warm-up: around 7–10 minutes

B. Use target tracking instead of complex step policies

Target tracking behaves more like a thermostat and is often simpler.

Target CPU: 50%
Disable conflicting step policies unless needed

C. Make scale-in conservative

Scale out fast, scale in slow.

Example:

Scale out: add 4 instances quickly
Scale in: remove 1 instance every 10–15 minutes

D. Use better metrics

For web apps:

Use ALB RequestCountPerTarget or latency
not only CPU

For workers:

Use QueueBacklogPerInstance
not total queue size alone

E. Add hysteresis

Avoid scaling in immediately after scaling out.

Example:

Scale out when CPU > 70%
Scale in when CPU < 35%

This gap prevents oscillation.

F. Check app/load balancer behavior

Sometimes the ASG is fine, but traffic is not evenly distributed.

Check:

ALB target health
Connection draining
Sticky sessions
Long-lived WebSocket connections
Uneven shard allocation

Model answer

I would detect thrashing by looking at ASG scaling activity history, desired capacity oscillation, CloudWatch alarm state changes, and application latency/error spikes. The likely causes are overly sensitive metrics, short cooldowns, aggressive scale-in, wrong metrics, app warm-up delay, or conflicting policies. I would remediate by increasing cooldown or instance warm-up, using target tracking, making scale-in conservative, choosing a better normalized metric, adding hysteresis, and removing conflicting scaling policies.

9. How do you ensure new instances launched by an ASG always use a patched, approved AMI?

This is both an operations and security governance question.

The goal is:

Every new instance launched by ASG must come from an approved, patched, tested image.

Recommended design

Golden AMI pipeline
        ↓
Security scanning
        ↓
Approval
        ↓
Update Launch Template
        ↓
ASG instance refresh
        ↓
Monitoring and rollback

Step 1: Build golden AMIs

Use a controlled image pipeline.

Example tools:

EC2 Image Builder
Packer
AWS Systems Manager Automation
CI/CD pipeline

AMI contents:

Latest OS patches
EDR/security agent
CloudWatch agent
SSM agent
CIS hardening baseline

Step 2: Scan and validate

Before approval:

Vulnerability scan
Malware scan

Package inventory check

Compliance tagging

Step 3: Store approved AMI ID in a controlled location

Example:

SSM Parameter Store:
/golden-ami/prod/web/latest

Or use versioned launch templates.

Step 4: Use Launch Templates, not old Launch Configurations

ASG should reference a launch template version.

Example:

Launch template: web-prod-lt
Version: 42
AMI: ami-0abc123approved

Avoid:

Manual AMI entry by engineer
Unversioned AMI usage
Old AMIs in stale launch templates

Step 5: Enforce with IAM/SCP/policy controls

Prevent unapproved AMIs.

Controls:

IAM condition keys
Service Control Policies
Tag-based controls
Allowed AMI owner restrictions
CI/CD-only permission to update launch templates
Deny RunInstances with unapproved AMIs

Step 6: Use ASG instance refresh

When a new AMI is approved:

Update launch template
Start instance refresh
Replace old instances gradually
Monitor health
Rollback if needed

Step 7: Continuously detect drift

Monitor:

Running instances with non-approved AMIs
Instances older than patch SLA
Launch template using old AMI
ASG not refreshed after AMI update

Model answer

I would use a golden AMI pipeline where AMIs are built, patched, scanned, tested, and approved before use. The ASG would use a versioned launch template referencing only approved AMIs, ideally with the latest approved AMI stored in SSM Parameter Store or managed through CI/CD. I would enforce this with IAM or SCP controls that deny launching unapproved AMIs, and I would use ASG instance refresh to roll the patched AMI across the fleet safely.

10. How would you control costs for an ASG-backed workload with highly erratic traffic patterns?

Highly erratic traffic is difficult because you need to balance:

Cost efficiency
Availability
Latency
Cold-start risk
Spot interruption risk
Downstream capacity

Example workload

Traffic pattern:
Normal: 5 instances
Random spikes: 80 instances
Idle overnight: 1 instance
Spike duration: 10–20 minutes

Cost control strategy

A. Use target tracking with a good metric

Use a metric that tracks real demand.

For web:

ALB RequestCountPerTarget
TargetResponseTime
CPU if CPU-bound

For workers:

QueueBacklogPerInstance
KafkaConsumerLagPerInstance

This avoids over-scaling on misleading metrics.

B. Set min, desired, and max carefully

Minimum capacity: enough for baseline availability
Desired capacity: normal load
Maximum capacity: cost and dependency guardrail

Example:

Min: 2
Desired: 4
Max: 50

The max prevents runaway scaling.

C. Use mixed instances with Spot for burst

Design:

Base capacity: On-Demand
Burst capacity: Spot

Example:

Minimum 4 On-Demand instances
Additional capacity 70% Spot, 30% On-Demand

This keeps baseline reliable while reducing cost during spikes.

D. Use warm pools only where justified

Warm pools reduce latency but can add cost.

Use only if:

Cold start is too slow
Traffic spikes are user-facing
Startup time causes SLA breach

Do not use warm pools for cheap, fast-starting stateless apps unless needed.

E. Use scale-in conservatively but not too slowly

For erratic traffic, aggressive scale-in can cause thrashing.

Better:

Scale out quickly
Scale in gradually

But do not leave excess capacity running for hours.

F. Use scheduled scaling if there are partial patterns

Even erratic workloads may have some known baseline.

Example:

Weekdays 8 AM–8 PM: min 10
Night: min 2
Weekend: min 3

G. Protect downstream systems

Do not let ASG scale beyond what the database, cache, or third-party API can handle.

Example:

Max ASG capacity should respect:
DB connection limit
Redis throughput
API rate limits
License limits

Otherwise, scaling out EC2 can increase failure and cost.

H. Use queue-based buffering

For erratic workloads, queues are powerful.

Architecture:

Traffic / Jobs
      ↓
Queue
      ↓
ASG worker fleet
      ↓
Backend processing

Benefits:

Absorbs spikes
Allows gradual processing
Reduces need for instant huge capacity
Protects backend systems

I. Use rightsizing and Graviton where possible

Cost options:

Use smaller instance types if app scales horizontally
Use Graviton if compatible
Use Compute Savings Plans for baseline
Use Spot for burst
Use On-Demand only where reliability requires it

J. Detect runaway scaling

Set alerts:

Desired capacity near max
Cost anomaly detection
High scale-out frequency
Low utilization after scale-out
Spot interruption surge
Error rate increasing despite scale-out

Cost-optimized design example

Baseline:
  4 On-Demand instances covered by Savings Plan

Burst:
  Mixed instance policy
  70% Spot
  30% On-Demand

Scaling metric:
  RequestCountPerTarget or QueueBacklogPerInstance

Protection:
  Max capacity = 60
  Scale-in cooldown = 10 minutes
  Lifecycle hooks for graceful termination
  Queue buffering for spikes
  Cost anomaly alerts

Model answer

For highly erratic traffic, I would use a good demand-based target tracking metric, set clear min and max capacity guardrails, and use mixed instance policies with On-Demand for baseline and Spot for burst. I would add queue buffering where possible so spikes do not require instant overprovisioning. I would scale out quickly but scale in carefully to avoid thrashing. I would also use Savings Plans for predictable baseline capacity, Spot for elastic capacity, warm pools only when cold starts hurt SLA, and cost anomaly alerts to detect runaway scaling.

Key Takeaways

  • ASGs replace unhealthy instances automatically and are the foundation of self-healing EC2 architectures.
  • Multi-AZ distribution is essential; never run a production ASG in a single AZ.
  • Launch templates supersede launch configurations and support versioning and mixed instance policies.
  • Lifecycle hooks allow graceful drain, configuration registration, and signal-based readiness before or after scaling events.
  • Scaling policy selection matters: target tracking is self-managing; step scaling gives finer control; scheduled scaling handles known patterns.
