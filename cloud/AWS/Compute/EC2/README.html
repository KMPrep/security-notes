EC2 — Elastic Compute Cloud

Category: Compute
AWS Service Type: Infrastructure as a Service (IaaS)
Key Docs: EC2 User Guide, EC2 Instance Types

Overview

EC2 provides resizable virtual machines in the cloud, giving full control over the operating system, networking, storage, and runtime environment. It is the foundational compute primitive in AWS and underpins many higher-level services. EC2 instances run inside a VPC and can be grouped behind load balancers, scaled with Auto Scaling Groups, and attached to EBS volumes or instance store. Instance types span general purpose, compute optimized, memory optimized, storage optimized, and accelerated computing families. Pricing models include On-Demand, Reserved Instances, Savings Plans, Spot Instances, and Dedicated Hosts.

How It Works

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 SSRF via IMDS v1 Metadata endpoint has no auth; apps can be tricked to query it Attacker finds an SSRF vulnerability in a web app running on the EC2 instance. They craft a request that causes the server to fetch http://169.254.169.254/latest/meta-data/iam/security-credentials/. IMDSv1 returns temp AWS credentials with no token required. Attacker uses those credentials from their own machine to move laterally across AWS services. Credential theft from instance role High
2 Overly permissive IAM role Instance profile grants excessive AWS permissions A developer attaches AdministratorAccess to an instance profile for convenience. An attacker who gains RCE on the instance (via any vuln) calls aws sts get-caller-identity, confirms admin role, then enumerates S3 buckets, exfiltrates data, creates backdoor IAM users, or pivots to other accounts via assume-role chains. Lateral movement to other AWS services High
3 Public SSH port exposure Port 22 open to 0.0.0.0/0 in security group Scanner (e.g., Shodan, Masscan) discovers port 22 open. Attacker attempts credential stuffing with leaked username/password lists or exploits an SSH daemon CVE. If the instance uses password auth or a weak key, attacker gains a shell and can access instance metadata, attached volumes, and internal network resources. Brute-force and exploit surface High
4 Hardcoded secrets in user data Bootstrap scripts stored in plaintext metadata endpoint A developer hardcodes a database password or API key in the user data bootstrap script. Any principal with ec2:DescribeInstanceAttribute on the instance (or SSRF access to IMDS) can retrieve the full user data in plaintext. The secret is then used to access the target database or third-party service directly. Secrets exposed to anyone who can call IMDS Med
5 Unpatched AMI or OS No patching pipeline; stale base images An instance is launched from a 6-month-old AMI. A known CVE (e.g., a kernel privilege escalation or a remote code execution in a listening service) exists but was not patched. Attacker scans the instance, identifies the vulnerable service version, runs a public exploit, and achieves root-level access without valid credentials. Known CVE exploitation High
6 Snapshot or AMI sharing EBS snapshots or AMIs shared publicly by mistake A developer runs aws ec2 modify-snapshot-attribute --attribute createVolumePermission --operation-type add --group-names all while testing, then forgets to revert it. An external researcher or attacker finds the public snapshot via automated enumeration, attaches it to their own EC2 instance, mounts it, and reads database files, config files, or keys stored on the volume. Data exposure Med
7 Spot interruption handling absent App not designed for sudden termination A Spot Instance running a stateful transaction processor receives a 2-minute interruption notice from AWS. The application has no SIGTERM handler and does not checkpoint state. AWS terminates the instance; in-flight writes are lost, the database is left in a partially committed state, and downstream consumers encounter corrupt records. Data loss and availability failure Med
8 Security group misconfiguration Overly broad egress rules allow data exfiltration paths Security group egress is left as 0.0.0.0/0 on all ports. An attacker who gains a foothold (via RCE or malicious code) installs a reverse shell or data exfiltration tool. The unrestricted egress allows the compromised instance to beacon out to a C2 server, exfiltrate S3 data, or scan internal subnets without triggering any network-level block. Data exfiltration or C2 communication Med
9 No CloudWatch monitoring CPU, disk, memory not alarmed An attacker installs a crypto miner after gaining access. CPU spikes to 100% but no alarm exists, so no alert fires. The compromise persists for weeks undetected. Alternatively, disk fills up due to log growth, causing the application to crash silently with no on-call notification. Silent failures and undetected intrusion Med
10 Instance store data loss Instance store is ephemeral and lost on stop or terminate A developer stores a processed data cache or partial ML training checkpoint on the instance store (NVMe SSD) for performance. The instance is stopped for maintenance or terminated by an ASG scale-in event. All instance store data is immediately and permanently wiped. There is no warning, no snapshot, and no recovery path. Unintended data loss Med

3 Public SSH port exposure Restrict SG to known CIDRs or replace SSH with SSM Session Manager Security group rules
4 Hardcoded secrets in user data Reference Secrets Manager or Parameter Store from user data at launch Secrets management
5 Unpatched AMI or OS Use Systems Manager Patch Manager and golden AMI pipelines Patch management
6 Snapshot or AMI sharing Audit snapshot permissions with Config rules; disable public sharing AWS Config and S3 Block
7 Spot interruption handling absent Handle SIGTERM in app; use mixed instance types and on-demand fallback Application architecture
8 Security group misconfiguration Use restrictive egress rules and VPC Flow Logs for traffic auditing Network controls
9 No CloudWatch monitoring Deploy CloudWatch agent; set alarms on key metrics and log patterns Observability
10 Instance store data loss Document ephemeral nature; persist critical data to EBS or S3 Storage architecture

Top 3 Critical Mitigations

  1. Enforce IMDSv2: IMDSv1 is exploitable via SSRF from any web application running on the instance. IMDSv2 requires a PUT-based token exchange that is not followable by simple SSRF redirects. Enforce it with an account-level EC2 default setting.
  2. Replace SSH with SSM Session Manager: Eliminates the need for inbound port 22, removes key management overhead, and provides full session audit logging to CloudWatch or S3.
  3. Attach least-privilege instance profiles: Instance roles are frequently over-scoped. Use IAM Access Analyzer and condition keys to scope permissions to the specific resources and actions each workload actually needs.

EC2 Security Model Answers — Detailed with Examples

1. What is the difference between IMDSv1 and IMDSv2, and why does IMDSv2 resist SSRF attacks?

What is IMDS?

IMDS means Instance Metadata Service.

Every EC2 instance can query a special link-local address:

http://169.254.169.254

This service provides metadata such as:

Instance ID
AMI ID
Region
Availability Zone
IAM role name
Temporary credentials for the instance profile
User data

This is useful for applications running on EC2 because they can retrieve temporary AWS credentials without storing static access keys.

IMDSv1

IMDSv1 is simple request/response.

Example:

curl http://169.254.169.254/latest/meta-data/iam/security-credentials/

Then:

curl http://169.254.169.254/latest/meta-data/iam/security-credentials/MyInstanceRole

Problem:

Any process on the instance that can make HTTP GET requests to 169.254.169.254 can retrieve metadata.

This becomes dangerous if the application has an SSRF vulnerability.

What is SSRF?

SSRF means Server-Side Request Forgery.

It happens when an attacker tricks a server-side application into making HTTP requests chosen by the attacker.

Example vulnerable endpoint:

https://app.example.com/fetch?url=http://example.com/image.png

Attacker changes URL to:

https://app.example.com/fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/

If the app blindly fetches that URL, the attacker may retrieve EC2 role credentials.

IMDSv2

IMDSv2 uses session-oriented requests. First, the application must send a PUT request to obtain a session token. That token must then be included in later metadata requests. The token can last from one second to six hours. (AWS Documentation)

Example:

TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

Then use the token:

curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/

Why IMDSv2 resists SSRF better

Many SSRF vulnerabilities only allow simple GET requests.

IMDSv2 requires:

1. HTTP PUT request
2. Custom request header
3. Session token
4. Token reuse in later GET request

So a basic SSRF primitive like this:

Server, please fetch this URL:
http://169.254.169.254/latest/meta-data/iam/security-credentials/

will fail when IMDSv2 is required because the metadata service expects a valid token. AWS allows an instance to be configured so IMDSv2 is required and IMDSv1 calls are rejected. (AWS Documentation)

Model answer

IMDSv1 allows unauthenticated HTTP GET requests from the instance to retrieve metadata. IMDSv2 requires a session token obtained through a PUT request with a specific header, and that token must be supplied on later metadata requests. This makes IMDSv2 much more resistant to SSRF because many SSRF bugs cannot issue arbitrary PUT requests with custom headers and then reuse the returned token. IMDSv2 does not fix SSRF itself, but it greatly reduces the chance that SSRF can be used to steal instance profile credentials.

3. When would you choose Spot Instances over On-Demand and how do you handle interruptions?

On-Demand

On-Demand is best when you need reliable capacity without interruption.

Use On-Demand for:

Critical production services
Stateful workloads
Databases
Control planes
Latency-sensitive systems
Systems that cannot tolerate interruption

AWS describes On-Demand as pay-as-you-go compute capacity with no upfront payment or long-term commitment. (Amazon Web Services, Inc.)

Spot Instances

Spot Instances are unused EC2 capacity offered at a discount, but AWS can interrupt them when capacity is needed.

Use Spot for:

Batch processing
CI/CD workers
Image/video processing
Stateless web/API replicas
Data processing
ML training workers with checkpointing
Queue consumers
Fault-tolerant workloads

Avoid Spot for:

Single-node databases
Stateful applications without replication
Critical systems that cannot restart
Long-running jobs without checkpointing

Handling Spot interruptions

AWS provides a Spot interruption notice, usually a two-minute warning before interruption. (AWS Documentation)

Design for interruption:

Checkpoint progress
Drain connections
Stop accepting new jobs
Save state to S3/EBS/database
Use SQS visibility timeout correctly
Use Auto Scaling mixed instance policy
Use multiple instance types
Use capacity-optimized allocation
Use Capacity Rebalancing

Auto Scaling Capacity Rebalancing can replace Spot Instances that receive rebalance recommendations before interruption. (AWS Documentation)

Example

Workload: Image processing queue
Baseline: 4 On-Demand instances
Burst: 20 Spot instances
Queue: SQS
Storage: S3

When interruption notice arrives:

1. Worker stops accepting new image jobs.
2. Current job is completed or checkpointed.
3. Output is saved to S3.
4. Message is deleted only after successful processing.
5. ASG replaces capacity using another Spot pool or On-Demand fallback.

Model answer

I would choose Spot for stateless, fault-tolerant, time-flexible, or checkpointable workloads where cost reduction matters more than guaranteed capacity. I would use On-Demand for critical baseline capacity. To handle interruptions, I would watch the two-minute interruption notice, checkpoint work, drain traffic, use queues, diversify instance types and AZs, and enable Capacity Rebalancing in Auto Scaling.

4. How does an instance profile differ from an IAM user, and what are the security implications?

IAM user

An IAM user is a long-lived identity.

It may have:

Username
Password
Long-term access key
Secret access key
Attached policies

Risk:

Static credentials can leak.
Leaked access keys may remain valid until rotated or disabled.

Instance profile

An instance profile is a container that lets an EC2 instance assume an IAM role.

The application running on EC2 does not need hardcoded AWS keys.

Flow:

EC2 instance
    ↓
Instance profile
    ↓
IAM role
    ↓
Temporary credentials from metadata service
    ↓
AWS API calls

The temporary credentials are retrieved from IMDS.

Security implications

Instance profiles are safer than IAM users for EC2 workloads because they use temporary credentials.

But they must be tightly controlled.

Risks:

Overprivileged instance role
SSRF stealing credentials through IMDSv1
Malware using role permissions
Lateral movement from compromised EC2
Privilege escalation through overly broad IAM permissions

Bad role policy:

{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*"
}

Better role policy:

{
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject"],
  "Resource": "arn:aws:s3:::company-app-prod-uploads/*"
}

Model answer

An IAM user is a long-lived human or workload identity with static credentials, while an EC2 instance profile attaches an IAM role to an instance and provides temporary credentials through the metadata service. Instance profiles are preferred for EC2 because they avoid hardcoded keys, but the role must be least-privileged and IMDSv2 should be required to reduce SSRF-based credential theft.

5. What are the tradeoffs between instance store and EBS for stateful workloads?

Instance store

Instance store is temporary block storage physically attached to the host. AWS says instance store is ideal for temporary data such as buffers, caches, scratch data, and replicated temporary data. (AWS Documentation)

Important point:

Instance store data does not survive instance termination.

AWS specifically warns not to rely on instance store for valuable long-term data unless you copy it to persistent storage such as EBS, S3, or EFS. (AWS Documentation)

EBS

EBS is network-attached block storage. AWS says EBS volumes persist independently from the running life of an EC2 instance and can be used for data that requires frequent updates, such as databases or system drives. (AWS Documentation)

Comparison

Area Instance Store EBS
Persistence Ephemeral Persistent
Survives instance stop No Yes
Survives instance termination No, unless data copied elsewhere Can, depending on delete-on-termination setting
Performance Very high local I/O Configurable IOPS/throughput
Durability Tied to host Designed for persistent storage
Snapshot support No direct EBS-style snapshots Yes
Best use Cache, scratch, temp data Databases, boot volumes, durable state
Failure impact Host loss can mean data loss Volume can be retained/recovered

Stateful workload design

For stateful workloads, prefer EBS unless the application replicates data elsewhere.

Use instance store for:

Redis cache with replication
Spark temporary shuffle data
Temporary ML training scratch space
Search index replicas
Local temp files

Use EBS for:

Databases
File systems requiring persistence
Application state
Boot/root volumes
Transaction logs

Model answer

Instance store gives very fast local temporary storage but loses data when the instance is terminated or the underlying host fails. EBS is persistent block storage that survives independently of the instance lifecycle and supports snapshots and recovery. For stateful workloads, I would use EBS unless the application already replicates state and can tolerate losing the local disk.

Detection

I would combine cloud-native telemetry, host telemetry, and network telemetry.

Signals:

Command-and-control traffic
Unusual DNS resolvers
DNS over HTTPS / DNS over TLS anomalies
High outbound traffic
Port scanning
Crypto mining behavior
Unusual process execution
Unexpected binaries

Amazon GuardDuty has EC2 finding types for suspicious behavior such as communication with unusual public DNS resolvers, DNS over HTTPS or DNS over TLS anomalies, and communication with IPs or domains from threat lists. (AWS Documentation)

GuardDuty Runtime Monitoring can also generate findings based on operating-system-level behavior from EC2 instances. (AWS Documentation)

Response plan

Do not immediately terminate if forensics are needed.

Contain the instance:

Block outbound internet
Detach from load balancer
Remove from Auto Scaling Group carefully
Disable IAM role if abused
Add NACL deny if needed

Quarantine SG example:

Inbound: only from forensic jump host
Outbound: deny all except logging/forensics endpoints

Step 2: Preserve evidence

Capture:

EBS snapshot
Memory dump if tooling exists
Instance metadata
CloudTrail events
VPC Flow Logs
GuardDuty finding details
SSM command history
Process list
Network connections
DNS logs

Step 3: Investigate

Questions:

How did it get compromised?
Was a public port exposed?
Was there a vulnerable application?
Were credentials stolen from IMDS?
Was the IAM role abused?
Did the attacker move laterally?
Did the instance scan or attack others?

Step 4: Eradicate and recover

Usually rebuild instead of clean in place.

Terminate compromised instance after evidence collection

Patch vulnerable software
Rotate credentials
Rotate instance profile permissions if exposed
Invalidate tokens
Block malicious indicators

Step 5: Prevent recurrence

Controls:

Require IMDSv2
Least-privilege IAM role
No public SSH/RDP
Patch management
EDR/runtime monitoring
Restrictive egress
VPC endpoints
GuardDuty and Security Hub
AWS Inspector
Centralized logging

Model answer

I would detect botnet behavior using GuardDuty, VPC Flow Logs, Route 53 Resolver DNS logs, CloudTrail, EDR/runtime telemetry, and application logs. I would look for C2 traffic, threat-listed IPs, unusual DNS, scanning, crypto mining, and abnormal outbound traffic. Response would start with quarantine, evidence preservation, IAM credential review, malware analysis, rebuild from a clean AMI, credential rotation, and hardening such as IMDSv2, patching, least privilege, and egress restrictions.

7. What controls prevent an EC2 instance from reaching the internet unexpectedly?

Preventing unexpected internet access requires controls at multiple layers.

A. Subnet route tables

A route table controls where subnet traffic goes. (AWS Documentation)

To prevent direct internet access:

Do not attach 0.0.0.0/0 route to an Internet Gateway.
Do not attach ::/0 route to an Internet Gateway.

A public subnet typically has a default route to an Internet Gateway. (AWS Documentation)

B. No public IP

Even if a subnet is public, an instance needs a public IPv4 or IPv6 route/addressing path to be reachable directly from the internet.

Controls:

Disable auto-assign public IP on subnet
Deny AssociateAddress
Deny RunInstances with public IP
Use private subnets

C. Security groups

Security groups act as virtual firewalls controlling inbound and outbound traffic for associated resources. (AWS Documentation)

Default SG outbound is often permissive, so restrict egress:

Allow outbound only to required destinations
Use prefix lists
Allow only VPC endpoints
Block 0.0.0.0/0 where not needed

D. Network ACLs

NACLs can provide subnet-level allow/deny controls.

Use them for coarse guardrails, but do not rely on them alone.

E. NAT Gateway control

A NAT Gateway allows instances in private subnets to initiate outbound internet traffic while preventing unsolicited inbound connections. (AWS Documentation)

If you do not want internet egress:

Do not add 0.0.0.0/0 route to NAT Gateway.

If you need controlled egress:

Route through firewall / inspection VPC
Use AWS Network Firewall
Use egress proxy
Use domain allowlisting

F. VPC endpoints

Use VPC endpoints to reach AWS services privately.

Example:

S3 Gateway Endpoint
DynamoDB Gateway Endpoint
Interface endpoints for SSM, EC2, CloudWatch, ECR, Secrets Manager

This avoids sending AWS API traffic over the public internet.

G. SCP and IAM controls

Prevent creation of internet paths:

Deny Internet Gateway attach
Deny route table default route to IGW
Deny public IP assignment
Deny Elastic IP association
Deny security group egress 0.0.0.0/0 for restricted accounts

Model answer

To prevent unexpected internet access, I would place instances in private subnets, remove default routes to Internet Gateways or NAT Gateways unless required, disable public IP assignment, restrict outbound security group rules, use NACLs as subnet guardrails, and route required egress through a firewall or proxy. For AWS API access, I would use VPC endpoints. At the organization level, SCPs and AWS Config would prevent public IPs, IGW routes, and overly broad egress.

8. How do placement groups affect fault tolerance and performance?

Placement groups control how EC2 instances are placed on underlying hardware.

AWS supports placement groups to cluster instances into low-latency groups or spread them across hardware to reduce correlated failure risk. (AWS Documentation)

There are three main types:

Cluster
Spread
Partition

A. Cluster placement group

Places instances close together in one AZ.

Best for performance:

Low-latency networking
High throughput
HPC
Distributed training
High-performance databases

Tradeoff:

Higher correlated failure risk because instances are close together.
Usually single-AZ.

Example:

10-node HPC cluster needing very low latency.

B. Spread placement group

Spreads instances across distinct underlying hardware.

Best for fault tolerance:

Small number of critical instances
Domain controllers
Quorum nodes
Critical monitoring nodes

Tradeoff:

Limited scale compared with cluster placement.
Less optimized for ultra-low latency.

Example:

3 critical control-plane nodes should not share the same physical rack.

C. Partition placement group

Divides instances into partitions. Each partition has separate hardware.

Best for large distributed systems:

Kafka
Cassandra
Hadoop
HBase
Distributed databases

Tradeoff:

You must map replicas intelligently across partitions.
More operational planning.

Example:

Kafka brokers spread across partitions so replicas are not on the same failure domain.

Comparison table

Placement Group Optimizes For Fault Tolerance Performance Use Case
Cluster Low latency/high throughput Lower Highest HPC, ML training
Spread Isolation Highest Normal Critical small fleets
Partition Large distributed fault domains High Good Kafka, Cassandra

Model answer

Cluster placement groups improve performance by placing instances close together for low latency and high throughput, but they increase correlated failure risk. Spread placement groups improve fault tolerance by placing instances on distinct hardware, but they are less suitable for large fleets or ultra-low-latency needs. Partition placement groups are a middle ground for distributed systems like Kafka or Cassandra, where replicas should be spread across separate failure domains.

9. Describe the data flow when a Lambda function calls the EC2 API versus when an EC2 instance calls other AWS APIs.

This question tests understanding of AWS identity, network path, and credential source.

A. Lambda function calls EC2 API

Flow:

Lambda function code
      ↓
AWS SDK
      ↓
Lambda execution role credentials
      ↓
AWS STS-managed temporary credentials
      ↓
EC2 public API endpoint or VPC endpoint
      ↓
EC2 control plane
      ↓
Action performed, if IAM allows

Lambda is a serverless compute service that runs code without managing servers. (AWS Documentation)

The Lambda function uses its execution role.

Example permission:

{
  "Effect": "Allow",
  "Action": "ec2:DescribeInstances",
  "Resource": "*"
}

If the Lambda is configured inside a VPC and needs private access to AWS APIs, use an EC2 interface VPC endpoint or NAT path depending on architecture.

B. EC2 instance calls AWS APIs

Flow:

Application on EC2
      ↓
AWS SDK credential provider chain
      ↓
IMDSv2
      ↓
Instance profile role temporary credentials
      ↓
AWS API endpoint or VPC endpoint
      ↓
AWS service control plane
      ↓
Action performed, if IAM allows

The key difference:

Lambda gets credentials from its execution role.
EC2 gets credentials from its instance profile through IMDS.

Security comparison

Area Lambda calling EC2 API EC2 calling AWS APIs
Runtime identity Lambda execution role EC2 instance profile role
Credential source Lambda service environment IMDS on the instance
Main risk Overprivileged function role IMDS credential theft / overprivileged role
Network path AWS API endpoint / VPC endpoint AWS API endpoint / VPC endpoint
Hardening Least privilege, resource constraints IMDSv2, least privilege, egress control

Example

Lambda stops an EC2 instance:

Lambda has permission:

EC2 uploads object to S3:

Model answer

When Lambda calls the EC2 API, the AWS SDK signs the request using temporary credentials from the Lambda execution role, then sends it to the EC2 API endpoint or VPC endpoint. When an EC2 instance calls AWS APIs, the application gets temporary credentials from the instance profile through IMDS, preferably IMDSv2, and uses those credentials to sign API requests. In both cases IAM authorizes the action, but the credential source and threat model differ.

10. How would you enforce that all EC2 instances in an org use IMDSv2, use approved AMIs, and have no public IPs?

This is an organization-wide governance problem.

The best answer is layered enforcement:

Prevent → Detect → Remediate

Requirement 1: Enforce IMDSv2

Prevent

Use EC2 account-level IMDSv2 enforcement per Region. AWS supports enforcing IMDSv2 at the account level so instances can launch only if configured to require IMDSv2. (AWS Documentation)

Also use SCP/IAM conditions to deny launches where:

ec2:MetadataHttpTokens != required

Expected launch setting:

HttpTokens = required
HttpEndpoint = enabled
HttpPutResponseHopLimit = 1 or controlled value

AWS notes that IMDSv2 requires token-based calls, while IMDSv1 does not. (AWS Documentation)

Detect

Use AWS Config rule or custom Config rule:

ec2-imdsv2-check

Remediate

Use SSM Automation or Lambda:

ModifyInstanceMetadataOptions → HttpTokens=required

AWS documents that existing instances can be modified to require IMDSv2. (AWS Documentation)

Requirement 2: Enforce approved AMIs

Prevent

Use:

Allowed AMIs
SCP deny RunInstances if AMI is not approved
IAM conditions based on AMI owner/tag
Launch templates controlled by CI/CD
AWS Service Catalog

AWS Allowed AMIs can restrict AMI discovery and use according to criteria such as trusted providers. (AWS Documentation)

Also enable block public access for AMIs to prevent accidental public sharing of organization AMIs. (AWS Documentation)

Approved AMI criteria:

Owner = central security account
Tag: Approved=true
Tag: PatchBaseline=prod
Signed/validated by pipeline
Not older than patch SLA

Detect

AWS Config custom rule:

Instance image ID must be in approved AMI list.

Remediate

Mark noncompliant instance
Isolate or terminate after exception process
Trigger ASG instance refresh with approved AMI
Create ticket to owning team

Requirement 3: Enforce no public IPs

Prevent

Use SCP/IAM to deny:

RunInstances with AssociatePublicIpAddress=true
AssociateAddress
AllocateAddress in restricted accounts
ModifySubnetAttribute auto-assign public IP=true
Create default route to IGW in restricted subnets

Set subnet defaults:

Auto-assign public IPv4: disabled
Use private subnets for workloads

Detect

AWS Config managed/custom rules:

ec2-instance-no-public-ip
subnet-auto-assign-public-ip-disabled
restricted-ssh/rdp

Also inspect:

PublicIpAddress field
Elastic IP associations
Route tables with IGW default routes
Security groups exposing inbound internet

Remediate

Disassociate public IP/EIP where possible
Stop/replace instance in private subnet
Remove from public subnet
Open incident/ticket
Quarantine if unexpected exposure is high risk

Example SCP-style logic

{
  "Effect": "Deny",
  "Action": "ec2:RunInstances",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "ec2:MetadataHttpTokens": "required"
    }
  }
}

Conceptual public IP deny:

{
  "Effect": "Deny",
  "Action": "ec2:RunInstances",
  "Resource": "*",
  "Condition": {
    "Bool": {
      "ec2:AssociatePublicIpAddress": "true"
    }
  }
}

Conceptual AMI enforcement:

{
  "Effect": "Deny",
  "Action": "ec2:RunInstances",
  "Resource": "arn:aws:ec2:*::image/ami-*",
  "Condition": {
    "StringNotEquals": {
      "ec2:Owner": "123456789012"
    }
  }
}

Exact condition keys and policy structure should be validated against the organization’s AWS partition, launch patterns, and supported EC2 condition keys before production rollout.

Model answer

I would enforce this with layered preventive and detective controls. For IMDSv2, I would enable EC2 account-level IMDSv2 enforcement and add SCP/IAM denies for launches where metadata tokens are not required. For approved AMIs, I would use a golden AMI pipeline, Allowed AMIs, launch templates, and SCPs restricting AMI owners or tags. For no public IPs, I would disable subnet auto-assign public IP and deny public IP association through SCPs. AWS Config, Security Hub, EventBridge, Lambda, and SSM Automation would detect and remediate drift across all accounts.

Final Summary Table

Question Strong Answer
IMDSv1 vs IMDSv2 IMDSv1 is simple GET metadata access; IMDSv2 requires token-based session requests, reducing SSRF credential theft risk.
EC2 launch pipeline Golden AMI, IaC checks, approved launch templates, SCPs, AWS Config, SSM, Inspector, GuardDuty.
Spot vs On-Demand Use Spot for fault-tolerant/stateless/checkpointable workloads; handle interruption with two-minute notice, checkpointing, and Capacity Rebalancing.
Instance profile vs IAM user Instance profile gives EC2 temporary role credentials; IAM users often use long-lived credentials. Prefer roles with least privilege.
Instance store vs EBS Instance store is ephemeral local storage; EBS is persistent network block storage suitable for durable state.
Botnet EC2 Detect via GuardDuty, Flow Logs, DNS, EDR; quarantine, preserve evidence, rebuild, rotate credentials.
Prevent internet access Private subnets, no public IP, no IGW/NAT routes unless approved, restrictive SG egress, VPC endpoints, SCPs.
Placement groups Cluster optimizes performance, spread optimizes isolation, partition supports distributed fault domains.
Lambda vs EC2 AWS API flow Lambda uses execution role; EC2 uses instance profile via IMDS; both call AWS APIs with IAM authorization.
Org-wide enforcement SCPs, account defaults, Allowed AMIs, Config rules, Security Hub, EventBridge remediation.

Key Takeaways

  • EC2 is foundational IaaS; most AWS services build on or integrate with it.
  • IMDSv2 and SSM Session Manager together eliminate the two biggest EC2 attack vectors.
  • Instance profiles replace static credentials — never embed AWS keys in AMIs or user data.
  • Security groups are stateful; VPC NACLs are stateless — use both for defense in depth.
  • Patch management, AMI pipelines, and Config rules are required for production workloads.
