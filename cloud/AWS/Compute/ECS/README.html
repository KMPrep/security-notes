ECS — Elastic Container Service

Category: Compute
AWS Service Type: Container Orchestration (managed)
Key Docs: ECS Developer Guide, ECS Best Practices

Overview

ECS is AWS's managed container orchestration service that runs Docker containers on either EC2 instances (EC2 launch type) or serverless infrastructure (Fargate launch type). It uses task definitions to describe containers, resource requirements, networking, and IAM roles. Services maintain desired task counts, integrate with Application Load Balancers, and support rolling deployments and circuit breaker patterns. ECS is tightly integrated with IAM, CloudWatch, ECR, Secrets Manager, and Service Connect for service mesh capabilities. It is a lighter-weight alternative to EKS for teams that don't need Kubernetes primitives.

How It Works

Core Mechanics

Concept Description
Cluster Logical grouping of tasks and services running on EC2 or Fargate infrastructure
Task definition Blueprint for a task: container image, CPU/memory, ports, env vars, volumes
Task Running instantiation of a task definition
Service Desired-count controller that maintains N running tasks and handles restarts
Task role IAM role that containers in the task assume for AWS API access
Task execution role IAM role used by ECS agent to pull images and retrieve secrets at launch
Service Connect Built-in service discovery and mTLS mesh between ECS services
Capacity provider Links ECS to Auto Scaling Groups or Fargate; manages placement and scaling

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Overly broad task IAM role Scope task role to minimum actions; use resource ARNs and conditions IAM policy design
2 Plain-text secrets in env vars Reference Secrets Manager or SSM in task definition using valueFrom Task definition design
3 Public ECR image or unverified pull Use private ECR; enable ECR image scanning on push Container registry policy
4 No container resource limits Set CPU and memory hard and soft limits in task definition Task definition design
5 Task execution role over-scoping Scope execution role to specific ECR repo ARNs and secret ARNs IAM policy design
6 Host networking mode in EC2 launch type Use awsvpc networking mode; block IMDS from containers via iptables Network configuration
7 No health check configured Define container and load balancer health checks; use ECS circuit breaker Service configuration
8 Immutable tag not enforced Use image digest or immutable tags; enforce via CI/CD pipeline Deployment pipeline
9 Container escape Never run privileged containers; use read-only file system where possible Task definition security
10 No log routing Configure awslogs driver in task definition to send to CloudWatch Logs Observability

Top 3 Critical Mitigations

  1. Use valueFrom for secrets in task definitions: Never put raw secret values in the environment block. Use valueFrom to reference Secrets Manager ARNs or SSM paths — ECS retrieves them at launch using the execution role.
  2. Scope task roles and execution roles separately: Task role = what the running container can do. Execution role = what ECS can do on behalf of the task at launch time. They have different trust and different minimum scopes.
  3. Use awsvpc networking mode and block IMDS from containers: In awsvpc mode, each task gets its own ENI and IP. Combined with iptables rules blocking 169.254.169.254, containers cannot steal the EC2 instance's role credentials.

1. What is the difference between the task role and the task execution role in ECS?

In ECS, there are two different IAM roles that people often confuse:

Task role            = permissions for your application container
Task execution role  = permissions for ECS/Fargate agent to prepare and run the task

AWS describes the task role as the IAM role your application code inside the container uses to access AWS services, while the task execution role lets ECS use AWS services on your behalf, such as pulling images from ECR and sending logs to CloudWatch Logs. (AWS Documentation)

Simple example

Assume your container runs a payment API.

The application needs to read from DynamoDB:

Container application → DynamoDB

That permission belongs in the task role.

But ECS also needs to pull the container image and send logs:

ECS/Fargate agent → ECR
ECS/Fargate agent → CloudWatch Logs

Those permissions belong in the task execution role.

Comparison table

Area Task Role Task Execution Role
Used by Your application code ECS agent / Fargate infrastructure
Purpose App calls AWS APIs ECS prepares and runs task
Example permission s3:GetObject, dynamodb:Query ecr:GetAuthorizationToken, logs:PutLogEvents
Exposed to container app? Yes, via task credentials No, normally used by ECS runtime
Security concern App compromise can abuse this role Overprivileged execution role can expose secrets/log/image access
Defined in Task definition Task definition

Example

Task role

{
  "Effect": "Allow",
  "Action": ["dynamodb:GetItem", "dynamodb:Query"],
  "Resource": "arn:aws:dynamodb:ap-southeast-1:111122223333:table/Orders"
}

This allows the application to query the Orders table.

Task execution role

{
  "Effect": "Allow",
  "Action": [
    "ecr:GetAuthorizationToken",
    "ecr:BatchGetImage",
    "logs:CreateLogStream",
    "logs:PutLogEvents"
  ],
  "Resource": "*"
}

This allows ECS to pull images and write logs.

Model answer

The ECS task role is for the application running inside the container. It controls what AWS APIs the application can call, such as S3, DynamoDB, SQS, or Secrets Manager. The task execution role is for ECS itself, allowing the ECS agent or Fargate runtime to pull images from ECR, fetch configured secrets, and send logs to CloudWatch. Security-wise, the task role should be least-privileged per application, while the execution role should only contain permissions needed to start and operate the task.

2. How does awsvpc networking mode improve security compared to bridge mode?

awsvpc mode gives each ECS task its own elastic network interface, private IP address, and security group controls. AWS states that awsvpc provides greater security by allowing security groups and network monitoring at a more granular task level. (AWS Documentation)

Bridge mode

In bridge mode, containers share the EC2 host networking stack through Docker’s bridge network.

EC2 instance
 ├── Container A
 ├── Container B
 └── Docker bridge network

AWS describes bridge mode as using a virtual network bridge between the host and the container, with host-to-container port mappings. (AWS Documentation)

Security challenges:

Multiple tasks share the same EC2 host network boundary
Security group applies at EC2 instance level
Port mapping complexity
Harder to isolate task-to-task traffic
Greater risk from misconfigured host networking

awsvpc mode

In awsvpc mode:

Task A → ENI A → Security Group A
Task B → ENI B → Security Group B
Task C → ENI C → Security Group C

Security benefits:

Each task gets its own network identity
Each service can have its own security group
Traffic can be controlled service-to-service
VPC Flow Logs can observe task-level traffic
No shared host port mapping problem
Better alignment with zero-trust network segmentation

AWS recommends awsvpc unless you have a specific need for another network mode, and notes that with awsvpc, each service can have a unique security group allowing inbound traffic only from specific upstream services. (AWS Documentation)

Example

For a three-tier app:

ALB → frontend-service → orders-service → database

You can enforce:

frontend-service SG allows inbound only from ALB SG
orders-service SG allows inbound only from frontend-service SG
database SG allows inbound only from orders-service SG

This is much cleaner than relying on host-level security groups and dynamic port mappings.

Model answer

awsvpc improves security because every ECS task gets its own ENI, private IP, and security group. This gives task-level network isolation and allows precise service-to-service access control. Bridge mode shares the EC2 host network through Docker bridge and typically applies security controls at the instance level, making isolation and observability weaker. For production ECS workloads, especially microservices, awsvpc is usually the preferred security model.

3. How would you design a zero-downtime rolling deployment for an ECS service?

A zero-downtime rolling deployment means replacing old tasks with new tasks while keeping enough healthy capacity available.

ECS rolling deployments use minimumHealthyPercent and maximumPercent to control how many tasks must remain healthy and how many extra tasks can run during deployment. AWS documents that minimumHealthyPercent is the lower limit of running tasks during deployment, and maximumPercent controls how many tasks can run above desired count. (AWS Documentation)

Example service

Service desired count: 10
minimumHealthyPercent: 100
maximumPercent: 200

This means:

At least 10 healthy tasks must remain running.
ECS can temporarily run up to 20 tasks during deployment.

Rolling deployment flow

1. Register new task definition revision.
2. ECS starts new tasks using the new revision.
3. New tasks pass container health checks.
4. New tasks pass ALB target group health checks.
5. ECS stops old tasks gradually.
6. Deployment completes when all running tasks use the new revision.

Design checklist

Control Why it matters
minimumHealthyPercent = 100 Prevents reducing below current desired capacity
maximumPercent = 200 Allows new tasks to start before old tasks stop
ALB health checks Ensures traffic only goes to ready tasks
Container health checks Detects broken application startup
Graceful shutdown Allows old tasks to finish in-flight requests
Deregistration delay Allows ALB connections to drain
Deployment circuit breaker Rolls back failed deployments
Autoscaling buffer Prevents deployment from fighting traffic spikes

AWS notes that during rolling deployments, ECS replaces unhealthy tasks while maintaining minimumHealthyPercent, and when maximumPercent allows it, the scheduler launches replacement tasks before stopping unhealthy ones. (AWS Documentation)

Example

Current version: v1
New version: v2
Desired count: 4
minimumHealthyPercent: 100
maximumPercent: 200

Deployment:

Start 4 new v2 tasks
Wait until v2 tasks are healthy
Drain 4 old v1 tasks
Complete deployment

Model answer

I would use ECS rolling deployment with an ALB, strong health checks, minimumHealthyPercent at 100%, and maximumPercent high enough to start new tasks before stopping old ones. I would configure container health checks, ALB target health checks, graceful shutdown, deregistration delay, and deployment circuit breaker rollback. This ensures traffic only reaches healthy new tasks and old tasks are drained safely.

4. What are the tradeoffs between EC2 launch type and Fargate for a latency-sensitive workload?

ECS can run tasks on EC2 launch type or Fargate.

EC2 launch type = you manage the compute fleet
Fargate = AWS manages the compute infrastructure

EC2 launch type

With EC2 launch type, you control the underlying EC2 instances.

Advantages:

More control over instance family
Can use placement groups
Can use specialized hardware
Can tune kernel/OS/network settings
Can pre-warm capacity
Can use reserved instances/savings plans/spot fleets
Potentially better for very latency-sensitive systems

Disadvantages:

You manage patching of EC2 hosts
You manage cluster capacity
You manage AMI lifecycle
You manage bin-packing and scaling
Larger operational burden

Fargate

With Fargate, you do not manage EC2 hosts.

Advantages:

No host management
Strong task isolation
Simpler scaling
Good security boundary
No EC2 patching burden
Fast operational onboarding

Disadvantages:

Less control over host-level tuning
Less suitable for specialized networking/performance tuning
Cold start may matter for very latency-sensitive workloads
Potentially higher cost for steady high utilization
Some workloads need more control than Fargate provides

Latency-sensitive decision

For a normal API:

Fargate is often good enough and simpler.

For ultra-low-latency or high-throughput systems:

EC2 may be better because you can choose instance types, placement groups, kernel tuning, ENA behavior, capacity pre-warming, and host-level optimizations.

Model answer

For latency-sensitive workloads, Fargate gives simpler operations and stronger managed isolation, but less control over the underlying host and performance tuning. EC2 launch type gives more control over instance type, placement, AMI, networking, capacity pre-warming, and kernel/OS tuning, which can matter for low-latency or high-throughput systems. I would choose Fargate for most standard APIs, but EC2 for workloads requiring deterministic performance, specialized hardware, placement groups, or deep runtime tuning.

5. How does ECS Service Connect differ from an external service mesh like Envoy or Istio?

ECS Service Connect is an AWS-managed way to provide service discovery, service-to-service connectivity, and traffic observability for ECS. AWS says Service Connect creates endpoints for ECS services and configures tasks to connect to those endpoints. (AWS Documentation)

ECS Service Connect

Service Connect gives you:

Simple service discovery
Consistent service names
ECS-native configuration
Traffic telemetry
Managed integration with ECS deployments
Less operational overhead

AWS describes Service Connect as needing minimal configuration: define named ports in the task definition, turn on Service Connect in the service, and choose a namespace. (AWS Documentation)

Example:

orders-service
payments-service
inventory-service

Your application can call:

http://payments

instead of managing raw IPs or custom service discovery.

External service mesh: Envoy/Istio/App Mesh-style model

A full mesh usually gives deeper controls:

Advanced traffic routing
mTLS policy control
Circuit breaking
Retries/timeouts
Fault injection
Canary rules
Authorization policies
Cross-platform workloads
Multi-cluster mesh
Rich policy ecosystem

But it comes with complexity:

Sidecar lifecycle management
Control plane operations
Certificate management
Debugging complexity
Performance overhead
Policy sprawl
Version compatibility

Service Connect uses AWS-managed components and integrates with ECS. AWS also provides Service Connect blue/green traffic testing support with header-based routing, such as the default x-amzn-ecs-blue-green-test header for HTTP-based protocols. (AWS Documentation)

Comparison table

Area ECS Service Connect External Mesh like Istio
Integration ECS-native Platform-neutral, often Kubernetes-first
Complexity Lower Higher
Service discovery Built-in Built-in
Traffic policy depth Moderate Very rich
mTLS/custom policy More managed/limited Deeply configurable
Operations AWS-managed experience You operate mesh/control plane
Best for ECS microservices needing simple connectivity Complex multi-platform/multi-cluster policy needs

Model answer

ECS Service Connect is an ECS-native service connectivity layer that simplifies service discovery, routing, and observability with minimal configuration. A full service mesh like Istio or a custom Envoy-based mesh provides more advanced traffic policies, mTLS controls, retries, circuit breaking, authorization, and multi-platform support, but adds significant operational complexity. I would use Service Connect for ECS-native microservices unless I need deep mesh features across Kubernetes, ECS, on-prem, or multiple clusters.

6. What controls prevent one ECS task from stealing credentials from another?

The main goal is to prevent credential cross-access between tasks.

Main controls

A. Use task roles, not host instance roles

Each ECS task should have its own least-privileged task role.

Bad model:

All containers rely on EC2 instance profile.
Compromise of one task may expose broad host role credentials.

Better model:

orders-task-role
payments-task-role
inventory-task-role

Each task gets only the permissions it needs.

AWS recommends task IAM roles for application AWS API permissions, while the container instance role is for EC2 instances to register with the ECS cluster. (AWS Documentation)

B. Use awsvpc mode

awsvpc gives each task a separate ENI and task-level network control. This reduces cross-task network exposure compared with shared bridge/host-style networking. (AWS Documentation)

C. Block access to EC2 instance metadata from containers on EC2 launch type

For EC2-backed ECS, containers should not be able to steal the EC2 host instance profile credentials.

AWS documents using iptables to prevent containers using bridge mode from accessing credential information supplied to the EC2 instance profile, while still allowing task role credentials. (AWS Documentation)

Important protection:

Block container access to:
169.254.169.254

Where possible, require IMDSv2 and restrict metadata access.

D. Least privilege per task

Do not reuse one broad role for all services.

Bad:

ecs-app-role:
  s3:*
  dynamodb:*
  sqs:*
  secretsmanager:*

Better:

orders-task-role:
  dynamodb:Query on Orders table only

payments-task-role:
  secretsmanager:GetSecretValue only for payment secret

inventory-task-role:
  sqs:ReceiveMessage only for inventory queue

E. Secrets isolation

Only inject secrets into the task that needs them.

Avoid:

Common secret bundle for every service

Prefer:

Per-service secret ARNs
Per-service KMS permissions
Per-service task roles

F. Runtime hardening

Controls:

Run as non-root
Read-only root filesystem
Drop Linux capabilities
No privileged containers
No Docker socket mount
No host PID/network namespace unless required
Use AppArmor/SELinux where available
Patch base images
Scan images

Model answer

To prevent one ECS task from stealing another task’s credentials, I would use separate least-privileged task roles, avoid broad EC2 instance profile permissions, use awsvpc networking for task isolation, block container access to EC2 instance metadata on EC2 launch type, and ensure secrets are injected only into tasks that need them. I would also harden containers by avoiding privileged mode, Docker socket mounts, root users, and unnecessary Linux capabilities.

7. How would you implement blue/green deployments for ECS using CodeDeploy?

Blue/green deployment means running the old and new versions side by side, shifting traffic only after the new version passes validation.

Blue  = current production version
Green = new version

AWS supports ECS blue/green deployments using CodeDeploy, and ECS also has infrastructure roles for load balancers for blue/green deployments. (AWS Documentation)

Architecture

Users
  |
  v
Application Load Balancer
  |
  +--> Blue target group  → ECS service v1
  |
  +--> Green target group → ECS service v2

Implementation steps

Step 1: ECS service with CodeDeploy deployment controller

Configure ECS service to use CodeDeploy deployment type.

Deployment controller: CODE_DEPLOY

Step 2: Use two target groups

Blue target group: current production tasks
Green target group: replacement tasks

Step 3: CodeDeploy AppSpec

The AppSpec defines:

New task definition
Container name
Container port
Target group routing
Lifecycle hooks

Step 4: Traffic shifting strategy

Options:

All-at-once
Canary
Linear

Example:

Shift 10% traffic to green
Wait 10 minutes
If healthy, shift remaining 90%

Step 5: Validation

Before full cutover:

Run smoke tests
Check ALB target health
Check 5xx rate
Check latency
Check business metrics
Check logs/errors

Step 6: Rollback

Configure CloudWatch alarms:

ALB 5xx increased
TargetResponseTime high
Task stopped unexpectedly
Error rate above threshold
CPU/memory abnormal

If alarms trigger:

CodeDeploy shifts traffic back to blue
Green deployment is stopped

Model answer

I would configure the ECS service with the CodeDeploy deployment controller, use an ALB with separate blue and green target groups, and define the new task definition in the CodeDeploy AppSpec. CodeDeploy would create the green task set, run validation hooks, shift traffic using canary or linear deployment, and monitor CloudWatch alarms. If health checks or alarms fail, CodeDeploy rolls traffic back to the blue target group.

8. How would you design a multi-region ECS architecture with failover?

A multi-region ECS architecture should separate:

Traffic failover
Application deployment
Data replication
Secrets/configuration
Observability
Runbooks

Active-passive design

Users
  |
  v
Route 53 / Global Accelerator
  |
  +--> Region A active
  |       ALB → ECS service → RDS/DynamoDB/Cache
  |
  +--> Region B standby
          ALB → ECS service → replicated data

Use when:

Lower cost
Simpler consistency model
Disaster recovery focus

Tradeoff:

Failover time may be higher
Standby region may be cold/warm

Active-active design

Users
  |
  v
Route 53 latency routing / AWS Global Accelerator
  |
  +--> Region A ECS service
  |
  +--> Region B ECS service

Use when:

Low latency globally
High availability
Regional isolation
Large user base across geographies

Tradeoff:

Data consistency is harder
Conflict resolution may be required
Higher cost
More operational complexity

Key design elements

Layer Design
DNS/traffic Route 53 health checks, failover routing, latency routing, or Global Accelerator
Compute ECS services deployed in both regions
Images ECR replication across regions
Secrets Secrets Manager replication or per-region secrets
Config SSM Parameter Store/AppConfig per region
Data DynamoDB Global Tables, Aurora Global Database, RDS cross-region replica, S3 CRR
Networking Separate VPCs, regional ALBs, private subnets
Observability Centralized logs, metrics, traces, synthetic checks
Deployment CI/CD deploys to both regions with promotion gates
Failover Tested runbooks and automated health-based routing

Important point

ECS failover is easy compared to data failover.

The hard part is:

Where is the source of truth data?
Can both regions write?
What happens during split brain?
How are secrets and config synchronized?

Model answer

I would deploy the ECS service, ALB, ECR images, secrets, and configuration into at least two regions. For traffic failover, I would use Route 53 failover/latency routing or AWS Global Accelerator. For data, I would choose DynamoDB Global Tables, Aurora Global Database, RDS cross-region replicas, or S3 replication depending on the workload. Active-passive is simpler and cheaper; active-active gives better latency and availability but introduces data consistency and conflict-resolution complexity. I would continuously test failover with synthetic checks and game days.

9. How do you handle secret rotation for secrets referenced in ECS task definitions?

ECS can reference secrets from Secrets Manager or SSM Parameter Store in task definitions. AWS recommends storing sensitive data in Secrets Manager or encrypted SSM Parameter Store parameters; Secrets Manager supports automatic rotation, random secret generation, and cross-account sharing. (AWS Documentation)

Important behavior

If a secret is injected into a container as an environment variable, the container receives the secret value at task start.

That means:

Rotating the secret does not automatically update already-running containers.

You usually need to restart or redeploy tasks.

AWS documents that ECS can inject Secrets Manager values into containers using the secrets container definition parameter. (AWS Documentation)

Recommended design

A. Store secrets in Secrets Manager

Example:

/prod/payments/db-password
/prod/orders/api-key

Enable automatic rotation where supported. Secrets Manager rotation updates credentials in the secret and the target database or service. (AWS Documentation)

B. Grant access only to the correct task role/execution role

Depending on how the secret is consumed:

Injected by ECS at startup → task execution role needs permission
Fetched by app at runtime → task role needs permission

C. Restart tasks after rotation

Options:

EventBridge rule on Secrets Manager rotation success
Trigger ECS force new deployment
Run rolling deployment
Replace tasks gradually

Example:

aws ecs update-service \
  --cluster prod \
  --service payments \
  --force-new-deployment

D. Prefer runtime retrieval for high-frequency rotation

For some applications, instead of environment variables:

App fetches secret from Secrets Manager at runtime
Caches secret for short TTL
Refreshes when authentication fails or TTL expires

This avoids waiting for task replacement, but adds complexity.

E. Use dual-secret rotation when possible

For databases:

Old password still works temporarily
New password is added
Application refreshes
Old password is removed

This prevents downtime.

Model answer

For ECS secrets, I would store them in Secrets Manager, enable automatic rotation where possible, and give only the required ECS role permission to read them. If secrets are injected as environment variables, running tasks do not automatically see the new value, so I would trigger a rolling force-new-deployment after successful rotation. For high-frequency or zero-downtime rotation, I would make the application fetch secrets at runtime with caching and support dual credentials during the rotation window.

10. What are the scaling limits and failure modes of ECS services under sudden traffic spikes?

Sudden spikes can break ECS workloads even when autoscaling is configured correctly.

The failure usually occurs because scaling has dependencies:

Metric detection → scaling decision → task placement → image pull → app startup → health check → traffic serving

Each step can fail or be too slow.

Common scaling limits

A. Metric delay

Autoscaling is not instant.

Traffic spike starts
CloudWatch metric updates
Alarm/policy triggers
ECS desired count increases
New tasks start

If traffic rises faster than this loop, users may see errors before new tasks are ready.

B. Image pull time

Large images slow down scale-out.

2 GB container image
Cold pull from ECR
Slow startup

Mitigation:

Smaller images
Layer caching on EC2
Fargate image optimization
Pre-warmed EC2 capacity
Avoid unnecessary packages

C. Capacity shortage

For EC2 launch type:

ECS wants more tasks
But cluster has no free CPU/memory/ENI capacity
Cluster autoscaling must add EC2 instances
EC2 instances must boot and join cluster

For Fargate:

Regional/account service quotas
Fargate capacity availability
Subnet IP exhaustion

D. ENI/IP exhaustion

With awsvpc, every task needs networking resources.

Failure mode:

Subnet has no available IPs
Tasks remain PENDING
Service cannot scale

Mitigation:

Large enough subnets
Multiple subnets/AZs
Monitor available IP addresses
Plan CIDR capacity

E. Load balancer bottlenecks

ALB/NLB usually scales well, but issues can occur:

Target registration delay
Health check delay
Slow start not configured
Connection draining behavior
Sticky sessions overload some tasks

F. Downstream bottlenecks

Scaling ECS does not automatically scale:

RDS connections
Redis throughput
Third-party API rate limits
Kafka partitions
DynamoDB capacity if not on-demand/properly autoscaled

Example:

ECS scales from 20 tasks to 200 tasks.
Each task opens 20 DB connections.
RDS suddenly receives 4,000 connections.
Database fails.

G. Bad autoscaling metric

CPU may not reflect actual bottleneck.

Examples:

High latency but low CPU due to DB waits
High queue backlog but low CPU due to external API throttling
Memory pressure but CPU-based scaling only

H. Deployment/scaling interaction

During deployment:

Rolling deployment already consumes extra capacity
Traffic spike also triggers scale-out
Cluster runs out of capacity

I. Startup health check failure

Under pressure, new tasks may fail health checks:

App starts slowly
Health check timeout too short
Task killed before ready
ECS restart loop

Remediation design

Failure Mode Mitigation
Slow scale-out Predictive/scheduled scaling, pre-warming, buffer capacity
Large images Slim images, optimize layers
EC2 capacity shortage Capacity providers, cluster autoscaling, mixed instance types
Fargate quota Increase quotas, monitor usage
IP exhaustion Larger CIDRs, multiple subnets, IP monitoring
DB overload Connection pooling, RDS Proxy, rate limits
Queue spikes SQS/Kafka buffering
Bad metrics Use request count per target, latency, backlog per task
Health check failures Startup probes, realistic grace periods
Thrashing Cooldowns, scale-in stabilization

Strong scaling architecture

Users
  |
  v
CloudFront / WAF
  |
  v
ALB
  |
  v
ECS service with target tracking
  |
  +--> Queue for async work
  |
  +--> RDS Proxy / cache / rate-limited downstream

Model answer

ECS services can fail under sudden spikes because autoscaling is not instantaneous. Common failure modes include CloudWatch metric delay, task startup delay, large image pulls, lack of EC2/Fargate capacity, subnet IP exhaustion, ENI limits, load balancer health check delays, and downstream bottlenecks such as database connection exhaustion. I would mitigate this with good scaling metrics, buffer capacity, predictive or scheduled scaling for known events, small images, capacity providers, larger subnets, queue-based buffering, RDS Proxy, realistic health checks, and service quota monitoring.

Task role vs execution role Task role is for app AWS API access; execution role is for ECS runtime operations like ECR image pull, logs, and startup secrets.
awsvpc vs bridge awsvpc gives task-level ENI, IP, security groups, and monitoring; bridge shares host networking through Docker bridge.
Zero-downtime rolling deployment Use ALB, health checks, minimumHealthyPercent, maximumPercent, graceful shutdown, deregistration delay, and rollback.
EC2 vs Fargate Fargate is simpler and isolated; EC2 gives more tuning/control for latency-sensitive workloads.
Service Connect vs mesh Service Connect is ECS-native and simpler; Istio/Envoy mesh gives deeper cross-platform traffic/security policy at higher complexity.
Prevent credential stealing Use per-task roles, awsvpc, block host metadata access, least privilege, secret isolation, and container hardening.
Blue/green with CodeDeploy Use blue/green target groups, CodeDeploy controller, AppSpec, traffic shifting, validation hooks, and rollback alarms.
Multi-region ECS Use Route 53/Global Accelerator, regional ECS/ALB, replicated ECR/secrets/config/data, and tested failover.
Secret rotation Rotate in Secrets Manager, trigger ECS redeploy for env-var secrets, or fetch secrets dynamically with caching.
Scaling failure modes Watch metric delay, image pull time, capacity shortage, IP exhaustion, health checks, and downstream bottlenecks.

Key Takeaways

  • ECS separates concerns: task role governs runtime AWS access; execution role governs launch-time image and secret retrieval.
  • Use awsvpc networking for network isolation between tasks and to block IMDS abuse.
  • Fargate removes the burden of host management but adds cost; EC2 launch type gives more control and cost efficiency at scale.
  • Secrets belong in Secrets Manager, referenced via valueFrom, never in the environment block.
  • ECS circuit breaker and health checks are essential to prevent bad deploys from silently degrading service.
