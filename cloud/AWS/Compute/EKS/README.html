EKS — Elastic Kubernetes Service

Category: Compute
AWS Service Type: Managed Kubernetes (CaaS)
Key Docs: EKS User Guide, EKS Best Practices

Overview

EKS is a managed Kubernetes control plane service that removes the overhead of running etcd, kube-apiserver, kube-controller-manager, and kube-scheduler. AWS manages high availability and patching of the control plane; customers manage worker nodes (EC2 node groups, Fargate profiles, or Karpenter-managed nodes). EKS is API-compatible with upstream Kubernetes so standard tooling — kubectl, Helm, Flux, ArgoCD — works without modification. IAM Roles for Service Accounts (IRSA) and EKS Pod Identity are the primary mechanisms for giving pods access to AWS services. EKS is the standard choice for teams that require Kubernetes ecosystem compatibility, multi-cloud portability, or large-scale container orchestration.

How It Works

Core Mechanics

Fargate profile Serverless node model; AWS runs pods on Firecracker micro-VMs

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Public API server endpoint Enable private endpoint; restrict public to specific CIDR or disable Cluster networking
2 Overly broad aws-auth or access entries Grant minimum Kubernetes RBAC roles; avoid mapping to system:masters Access control
3 Node instance role abuse Use IRSA or Pod Identity; remove AWS permissions from EC2 node role IAM design
4 No network policies Deploy Calico or use VPC CNI network policy; default-deny namespaces Kubernetes network policy
5 Privileged pod or host path mount Enforce OPA/Gatekeeper or Kyverno policies; use PodSecurity standards Admission control
6 Outdated Kubernetes version Upgrade clusters within EKS supported version window Patch management
7 etcd exposed or unencrypted Enable envelope encryption of secrets at rest using KMS key EKS secrets encryption
8 No audit logging for API server Enable EKS control plane logging; send to CloudWatch Logs Observability
9 IRSA OIDC trust misconfiguration Scope OIDC condition to specific namespace and service account name IAM trust policy design
10 Unrestricted egress from pods Use network policies to restrict pod egress; route via NAT with logging Network controls

Top 3 Critical Mitigations

  1. Restrict API server access: The Kubernetes API is the blast radius center. Disable or restrict the public endpoint, require VPN or bastion access, and audit all API calls via EKS control plane logging.
  2. Use IRSA or Pod Identity instead of node roles: Node-level IAM roles are reachable by any pod via IMDS. IRSA scopes credentials to a specific service account in a specific namespace, dramatically reducing lateral movement surface.
  3. Enforce pod security with admission controllers: Use Kyverno or OPA/Gatekeeper to block privileged containers, host path mounts, and root containers. PodSecurity admission (built into k8s 1.25+) is a baseline enforcement layer.

Amazon EKS Model Answers — Security + Architecture Deep Dive

1. What is the difference between IRSA and Pod Identity, and which should you prefer for new deployments?

Both IRSA and EKS Pod Identity solve the same high-level problem:

How does a pod get AWS permissions without storing AWS access keys?

The goal is to avoid this bad pattern:

Kubernetes Secret contains AWS_ACCESS_KEY_ID
Kubernetes Secret contains AWS_SECRET_ACCESS_KEY
Pod reads static credentials
Attacker steals secret
Credentials remain valid until manually rotated

Instead, both IRSA and Pod Identity provide temporary IAM credentials to pods.

IRSA: IAM Roles for Service Accounts

IRSA means IAM Roles for Service Accounts.

With IRSA, a Kubernetes service account is connected to an IAM role using an OIDC identity provider.

Flow:

Pod
  ↓
Kubernetes service account token
  ↓
OIDC provider
  ↓
AWS STS AssumeRoleWithWebIdentity
  ↓
Temporary AWS credentials
  ↓
AWS API call

Example:

Service account: payments-sa
Namespace: payments
IAM role: payments-s3-reader-role
Permission: s3:GetObject on payments bucket

The pod uses its projected service account token to call AWS STS and assume the IAM role.

AWS documents that IRSA uses AssumeRoleWithWebIdentity, while EKS Pod Identity uses a node-local agent and the EKS Auth service to provide credentials. (AWS Documentation)

EKS Pod Identity

EKS Pod Identity is a newer EKS-native way to assign IAM roles to pods.

Flow:

Pod
  ↓
EKS Pod Identity Agent on node
  ↓
EKS Auth service
  ↓
IAM role association
  ↓
Temporary AWS credentials
  ↓
AWS API call

You create an association like:

Cluster: prod-eks
Namespace: payments
ServiceAccount: payments-sa
IAM Role: payments-s3-reader-role

The pod then receives temporary credentials through the Pod Identity Agent.

AWS describes EKS Pod Identities as providing least-privilege access scoped to Kubernetes service accounts, with credentials issued through the EKS Auth service and the Pod Identity Agent running on each node. (AWS Documentation)

Key Difference

IRSA = Kubernetes OIDC token → STS AssumeRoleWithWebIdentity
Pod Identity = EKS association → EKS Auth service → Pod Identity Agent

Comparison Table

Area IRSA EKS Pod Identity
Credential mechanism OIDC + STS AssumeRoleWithWebIdentity EKS Auth service + Pod Identity Agent
Setup Requires OIDC provider and IAM trust policy per role Uses EKS Pod Identity associations
Operational complexity Higher Lower
IAM trust policy Cluster/OIDC-specific Reusable IAM role trust model across clusters is easier
Scalability Each pod/SDK may call STS EKS Pod Identity reduces repeated STS load by using node agent flow
Portability Works with Kubernetes OIDC model EKS-specific
Best fit Existing IRSA workloads, older clusters, portability needs New EKS-native deployments
Main dependency OIDC provider and STS EKS Auth service and Pod Identity Agent

AWS highlights scalability as a benefit of Pod Identity because credentials are assumed by the EKS Auth service and then issued by the Pod Identity Agent, reducing duplicated credential-assumption load from every pod SDK. (AWS Documentation)

Which should you prefer for new deployments?

For new EKS-native deployments, I would usually prefer EKS Pod Identity.

Why:

Simpler operations
Cleaner association model
Less IAM/OIDC trust policy management
Better scalability model
Purpose-built for EKS
Easier for platform teams to manage centrally

However, I would still use or keep IRSA when:

The workload already uses IRSA and is stable
The organization has mature IRSA automation
The workload needs compatibility with non-EKS Kubernetes patterns
The cluster/runtime does not support Pod Identity requirements
A vendor chart only supports IRSA-specific assumptions

Model answer

IRSA maps Kubernetes service accounts to IAM roles using an OIDC provider and STS AssumeRoleWithWebIdentity. EKS Pod Identity is newer and uses EKS Pod Identity associations, the EKS Auth service, and a node-local agent to provide temporary credentials to pods. For new EKS-native workloads, I would generally prefer EKS Pod Identity because it is simpler to operate, reduces OIDC trust policy complexity, and scales better operationally. I would keep IRSA for existing stable workloads or where Kubernetes/OIDC portability is required.

2. How does EKS handle multi-tenancy isolation between teams sharing one cluster?

EKS does not automatically make a single cluster a hard multi-tenant security boundary.

A shared EKS cluster can support multi-tenancy, but the strength of isolation depends on how you configure:

Kubernetes RBAC
Namespaces
IAM permissions
Pod security
Network policies
Node isolation
Admission control
Resource quotas
Secrets boundaries
Observability

The key point:

Kubernetes namespaces are an organizational boundary, not a full security boundary by themselves.

Basic multi-tenant model

Shared EKS Cluster
 ├── namespace: team-payments
 ├── namespace: team-risk
 ├── namespace: team-analytics
 └── namespace: platform-system

Each team gets:

Own namespace
Own service accounts
Own IAM roles
Own quotas
Own network policies
Own deployment permissions
Own secrets

Isolation Layers

A. Namespace isolation

Namespaces separate Kubernetes objects.

Example:

team-payments can deploy only in namespace payments
team-risk can deploy only in namespace risk

But namespace isolation alone does not stop:

Network access across namespaces
Cluster-wide RBAC mistakes
Privileged pods escaping node boundaries
Shared node attacks
Overbroad service account permissions

B. Kubernetes RBAC

Use RBAC to limit what each team can do.

Example:

Payments team:
- Can create deployments in payments namespace
- Can read logs in payments namespace
- Cannot read secrets in risk namespace
- Cannot create clusterrolebindings
- Cannot create privileged pods

Bad RBAC:

subjects:
  team-payments
role:
  cluster-admin

Better RBAC:

subjects:
  team-payments
role:
  namespace-developer
scope:
  namespace payments

C. IAM isolation

Each workload should use a separate role through IRSA or Pod Identity.

Example:

payments-api pod → payments-dynamodb-role
risk-api pod → risk-s3-read-role
analytics-job pod → analytics-athena-role

Do not share one broad IAM role across namespaces.

Bad:

all-pods-role:
  Action: *
  Resource: *

Good:

payments-task-role:
  dynamodb:Query only on Payments table

D. Network segmentation

By default, many Kubernetes networking setups allow broad pod-to-pod communication unless network policies are enforced.

Use:

Kubernetes NetworkPolicy
AWS VPC CNI Network Policy support or compatible network policy engine
Security Groups for Pods where appropriate
Ingress controls
Egress controls

Example goal:

payments namespace cannot directly call risk namespace
analytics namespace cannot call production database

E. Pod security controls

Use Kubernetes Pod Security Admission or policy engines to prevent dangerous pod specs.

Block:

privileged: true
hostNetwork: true
hostPID: true
hostPath mounts
running as root
adding dangerous Linux capabilities
mounting service account tokens unnecessarily

Require:

runAsNonRoot
readOnlyRootFilesystem
drop capabilities
seccomp profile
resource limits

F. Node isolation

For strong isolation, separate tenants by node groups.

Example:

payments workloads → payments node group
analytics workloads → analytics node group
platform workloads → platform node group

Use:

taints
tolerations
node selectors
affinity
dedicated managed node groups
Karpenter NodePools

Why this matters:

If two tenants share the same node, a container escape or kernel exploit can affect both.

For highly sensitive tenants, use separate clusters.

G. Resource quotas and limit ranges

Prevent one team from consuming the whole cluster.

Example:

payments namespace:
  max CPU: 100 cores
  max memory: 400 GiB
  max pods: 300

This prevents noisy-neighbor problems.

H. Secrets isolation

Use:

Namespace-scoped Kubernetes Secrets
External Secrets with scoped IAM
Secrets Manager per team
KMS keys per environment/team where needed
RBAC preventing cross-namespace secret reads

Never allow generic read access to all secrets.

Isolation Strength

Model Isolation Strength Use Case
Namespace only Weak Dev/test convenience
Namespace + RBAC + NetworkPolicy Medium Internal team sharing
Dedicated node groups per tenant Stronger Sensitive workloads
Dedicated clusters per tenant/team Strongest Regulated/high-risk workloads

Model answer

EKS multi-tenancy is built through layers. I would use namespaces for logical separation, RBAC for access control, Pod Identity or IRSA for per-workload IAM, network policies for namespace segmentation, Pod Security Admission to block privileged workloads, resource quotas to prevent noisy neighbors, and dedicated node groups for stronger isolation. For high-risk or regulated workloads, I would not rely on a shared cluster; I would use separate clusters or accounts because Kubernetes namespaces are not a hard security boundary.

3. How would you design a zero-downtime Kubernetes version upgrade for a production EKS cluster?

A zero-downtime EKS upgrade needs to handle:

Control plane upgrade
Add-on compatibility
Node upgrade
Workload availability
Rollback/contingency
Observability

Important EKS detail:

Upgrading the EKS control plane does not automatically upgrade managed node groups.

AWS states that managed node groups are not automatically upgraded when the control plane is updated, and recommends keeping node and control plane versions aligned. (AWS Documentation)

High-level upgrade flow

1. Pre-check cluster and workloads
2. Upgrade EKS control plane
3. Upgrade EKS add-ons
4. Create/update node groups
5. Drain old nodes gradually
6. Validate workloads
7. Remove old nodes

Step 1: Pre-upgrade checks

Check:

Deprecated Kubernetes APIs
EKS version support
Add-on compatibility
Ingress controller compatibility
CSI driver versions
VPC CNI version
CoreDNS version
kube-proxy version
PodDisruptionBudgets
Replica counts
StatefulSet disruption behavior
Cluster autoscaler/Karpenter compatibility
Admission webhook availability

A common upgrade failure is deprecated APIs.

Example:

Deployment still using old apiVersion
Ingress using deprecated apiVersion
Webhook using old admissionregistration API

Step 2: Ensure workload redundancy

Before upgrade:

At least 2+ replicas for stateless services
Pods spread across AZs
PodDisruptionBudgets configured
Readiness probes correct
Liveness probes not too aggressive
Horizontal Pod Autoscaler healthy
Cluster Autoscaler/Karpenter has capacity

Example PDB:

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: payments-api-pdb
spec:
  minAvailable: 80%
  selector:
    matchLabels:
      app: payments-api

This prevents too many pods being evicted at once.

Step 3: Upgrade control plane

Upgrade one minor version at a time.

Example:

1.28 → 1.29 → 1.30

Do not skip unsupported version paths.

During control plane upgrade, worker nodes continue running existing workloads, but Kubernetes API availability may be affected briefly or have changing behavior. Plan maintenance windows and freeze risky deployments.

Step 4: Upgrade managed add-ons

Update:

VPC CNI
CoreDNS
kube-proxy
EBS CSI driver
EFS CSI driver if used
CloudWatch/ADOT agents if needed

Why this matters:

Old VPC CNI can break pod networking behavior.
Old CoreDNS can cause service discovery issues.
Old CSI driver can affect volume attach/mount behavior.

Step 5: Upgrade nodes safely

For managed node groups, update the node group version or AMI version. AWS documents managed node group updates for newer Kubernetes/AMI versions. (AWS Documentation)

Recommended pattern:

Create new node group with new Kubernetes AMI
Cordon old nodes
Drain old nodes gradually
Respect PDBs
Validate workloads
Scale down old node group

This blue/green node-group pattern is safer than in-place changes.

Step 6: Stateful workload handling

For stateful workloads:

Check storage class behavior
Check EBS volume AZ affinity
Ensure one pod at a time disruption
Back up data
Test restore
Respect quorum

Example:

Kafka brokers should not all drain at once.
Database replicas must maintain quorum.

Step 7: Validation

Validate:

API server health
Node readiness
CoreDNS resolution
Ingress/ALB routing
Application latency
Error rate
Pod restarts
HPA behavior
CSI volume attach/mount
AWS API calls from pods
Security controls

Model answer

I would perform an EKS upgrade in phases: first run pre-checks for deprecated APIs, add-on compatibility, PDBs, and workload redundancy; then upgrade the control plane one minor version at a time; then upgrade managed add-ons such as VPC CNI, CoreDNS, kube-proxy, and CSI drivers; then roll nodes using a blue/green node group approach. I would cordon and drain old nodes gradually, respect PodDisruptionBudgets, validate service health and storage behavior, and keep rollback plans for workloads and node groups.

4. What are the failure modes of Karpenter versus managed node groups for node autoscaling?

Both Karpenter and managed node groups can provide node capacity, but they fail differently.

Managed Node Groups

Managed node groups are AWS-managed groups of EC2 instances backing EKS.

They are predictable and simple.

Failure modes:

ASG scaling too slow
Wrong instance type selected
Node group max size reached
Insufficient EC2 capacity
AZ capacity constraints
Launch template misconfiguration
AMI or bootstrap failure
Pods pending due to taints/labels mismatch
Slow scale-out during sudden spikes
Overprovisioning because groups are coarse

Managed node groups scale around pre-defined groups.

Example:

Node group: m6i.large only
Max size: 20

If the workload needs GPU or more memory, it may not schedule unless another group exists.

Karpenter

Karpenter provisions nodes dynamically based on unscheduled pods.

It can choose instance types, sizes, AZs, and capacity types more flexibly.

Failure modes:

Provisioner/NodePool misconfiguration
Overly restrictive requirements
No compatible instance types
Insufficient AWS capacity
IAM permission issues
Subnet/security group discovery tag problems
Disruption/consolidation evicts too aggressively
Spot interruption churn
DaemonSet overhead underestimated
Workload uses constraints that Karpenter cannot satisfy
Cost surprises from large instances

Example:

Pod requires:
  zone = ap-southeast-1a
  instance type = c7i.2xlarge
  capacity type = spot

But Spot capacity is unavailable in that AZ.
Pod stays pending.

Comparison Table

Area Managed Node Groups Karpenter
Scaling model ASG/node-group based Pod-driven dynamic provisioning
Flexibility Lower Higher
Operational simplicity Higher Medium
Bin packing Less efficient Better
Spot diversification More limited Stronger
Failure risk Slow/coarse capacity Misconfiguration/dynamic disruption
Best for Stable predictable workloads Dynamic, diverse, bursty workloads

Model answer

Managed node groups are simpler and predictable but can be slow and coarse because they scale predefined ASGs. They fail when max size is reached, capacity is unavailable, instance types are wrong, or node bootstrap fails. Karpenter is more flexible and efficient because it provisions nodes directly for pending pods, but it can fail through misconfigured NodePools, overly restrictive scheduling constraints, subnet/security group discovery issues, aggressive consolidation, Spot churn, or unexpected instance selection. I would use managed node groups for stable baseline capacity and Karpenter for dynamic burst capacity if the platform team can operate it well.

5. How does the VPC CNI plugin assign IP addresses, and what are the scaling limitations?

Amazon VPC CNI gives Kubernetes pods IP addresses from the VPC.

This means pods are first-class VPC citizens:

Pod IP = VPC IP
Pod can communicate with VPC resources directly
Security and routing integrate with VPC

Default mode: secondary IP mode

In default secondary IP mode:

Each EC2 worker node has ENIs
Each ENI has secondary private IP addresses
VPC CNI assigns those secondary IPs to pods

Flow:

EC2 node
 ├── Primary ENI
 │    ├── primary IP for node
 │    ├── secondary IP → pod A
 │    ├── secondary IP → pod B
 │    └── secondary IP → pod C
 └── Additional ENIs if needed
      ├── secondary IP → pod D
      └── secondary IP → pod E

AWS describes secondary IP mode as the default mode for the VPC CNI. (AWS Documentation)

Prefix delegation mode

With prefix delegation, the CNI assigns prefixes to ENIs instead of assigning individual secondary IPs.

AWS says prefix assignment can significantly increase the number of IP addresses available to pods and reduce EC2 API calls. (AWS Documentation)

Example:

Instead of attaching individual IPs:
10.0.1.10
10.0.1.11
10.0.1.12

Attach a prefix:
10.0.1.16/28

A /28 IPv4 prefix contains 16 addresses.

AWS documents that VPC CNI prefix mode for Linux assigns prefixes to EC2 network interfaces to increase pod density, and can use IPv4 or IPv6 prefixes depending on configuration. (AWS Documentation)

Scaling limitations

A. Subnet IP exhaustion

Because pods consume VPC IPs, you can run out of subnet CIDR space.

Example:

Subnet has 2,000 available IPs
Cluster needs 4,000 pods
Result: pods remain Pending

This is one of the biggest EKS scaling constraints.

B. ENI and IP limits per instance type

Each EC2 instance type supports a maximum number of ENIs and IPs per ENI.

Small instances have lower pod density.

Example concept:

t3.medium supports fewer ENIs/IPs than m7i.4xlarge

If you choose small instances, pod density may be limited by networking before CPU/memory.

C. EC2 API throttling

In secondary IP mode, rapid scale-out may require many EC2 API calls to attach ENIs and assign IP addresses.

Prefix delegation helps reduce this because assigning prefixes gives multiple IPs at once. (AWS Documentation)

D. Warm IP configuration

VPC CNI maintains warm IPs or prefixes for fast pod startup.

Misconfiguration can cause:

Too many unused IPs reserved
Too few warm IPs causing slow pod startup
Subnet exhaustion

E. Security Groups for Pods complexity

Security Groups for Pods provides granular network control, but can add ENI/resource constraints depending on configuration.

Mitigations

Use larger subnets
Use prefix delegation
Use IPv6 where appropriate
Use custom networking to place pods in separate CIDRs
Choose instance types with higher ENI/IP limits
Monitor available subnet IPs
Tune WARM_IP_TARGET / WARM_PREFIX_TARGET
Use Karpenter/node sizing carefully

Custom pod networking is one AWS-documented approach to help with IPv4 exhaustion in large EKS deployments. (Amazon Web Services, Inc.)

Model answer

The Amazon VPC CNI assigns VPC IP addresses directly to pods. In default secondary IP mode, it attaches secondary private IPs to node ENIs and assigns those IPs to pods. In prefix delegation mode, it attaches IP prefixes such as IPv4 /28 prefixes to ENIs, increasing pod density and reducing EC2 API pressure. The major scaling limits are subnet IP exhaustion, per-instance ENI/IP limits, EC2 API throttling during rapid scale-out, warm IP misconfiguration, and security-groups-per-pod constraints.

6. How would you detect and respond to a compromised pod that is performing AWS API calls?

This is a cloud + Kubernetes incident response question.

The key is to correlate:

AWS API activity
Pod identity
Kubernetes workload metadata
Node metadata
Network behavior
Runtime behavior

Detection Signals

A. CloudTrail

Look for suspicious API calls by the pod’s IAM role.

Examples:

Unusual S3 ListBucket/GetObject
Unexpected SecretsManager GetSecretValue
IAM privilege escalation attempts
KMS Decrypt spikes
STS AssumeRole calls
EC2 Describe/RunInstances attempts
Calls from unusual region
Calls outside normal service behavior

If using Pod Identity or IRSA, the API call should map back to the IAM role associated with the service account.

B. GuardDuty

GuardDuty can detect suspicious AWS API behavior, credential misuse, unusual network activity, and Kubernetes/EKS-related findings depending on enabled protections.

For a compromised pod, useful findings may include:

IAM role credentials used anomalously
Credential exfiltration indicators
Suspicious API discovery behavior
Crypto mining or C2 traffic
Kubernetes audit anomalies
Runtime findings if enabled

C. Kubernetes audit logs

Enable and review EKS control plane logs, especially:

api
audit
authenticator
controllerManager
scheduler

Look for:

Pod exec into containers
New privileged pods
Secret reads
Service account token access
RoleBinding/ClusterRoleBinding changes
DaemonSet creation
HostPath mounts

D. Runtime telemetry

Use tools such as:

GuardDuty Runtime Monitoring
Falco
eBPF-based runtime sensors
EDR for nodes
Container image scanning

Look for:

Shell spawned in container
curl/wget to metadata endpoints
Unexpected AWS CLI use
Credential file access
Suspicious outbound connections
Reverse shell behavior
Crypto miner process

E. VPC Flow Logs and DNS logs

Look for:

Unexpected public IP destinations
Known malicious IPs
Unusual DNS queries
Large data egress
Calls to AWS APIs outside VPC endpoints if unexpected

Response Plan

Step 1: Contain

Actions:

Scale deployment to zero
Delete compromised pod
Apply NetworkPolicy deny egress
Cordon and isolate node if node compromise is suspected
Block IAM role permissions temporarily
Disable or restrict Pod Identity/IRSA association
Quarantine namespace

Be careful:

Deleting the pod may destroy evidence.
If forensics matter, snapshot node volume or preserve logs first.

Step 2: Preserve evidence

Collect:

Pod spec
Container image digest
Kubernetes events
Container logs
Node logs
CloudTrail events for role
VPC Flow Logs
DNS logs
Runtime telemetry
Mounted secrets/configmaps
Service account details
IRSA/Pod Identity association

Step 3: Determine blast radius

Ask:

What IAM role did the pod have?
What AWS APIs were called?
Were secrets read?
Was data exfiltrated?
Were other roles assumed?
Were Kubernetes secrets accessed?
Was the node compromised?
Were other pods affected?

Step 4: Rotate and revoke

Actions:

Rotate application secrets
Rotate database credentials
Revoke or restrict IAM role session path
Remove unnecessary IAM permissions
Rotate external API keys
Invalidate app tokens

Temporary IAM credentials expire automatically, but you still need to reduce permissions and rotate secrets that may have been accessed.

Step 5: Eradicate and recover

Patch vulnerable app
Rebuild image from trusted source
Deploy fixed image digest
Add admission policy controls
Add egress NetworkPolicy
Reduce IAM permissions
Improve detection rules

Model answer

I would detect a compromised pod performing AWS API calls by correlating CloudTrail activity for the pod’s IAM role with Kubernetes audit logs, pod identity mappings, runtime telemetry, VPC Flow Logs, and DNS logs. I would look for unusual API calls, secret reads, privilege escalation attempts, region anomalies, and suspicious runtime behavior. Response would involve containing the workload, restricting the IAM role, preserving pod/node evidence, determining what AWS resources were accessed, rotating exposed secrets, redeploying from a trusted image, and tightening IAM, network, and admission controls.

7. What is the attack surface of the Kubernetes API server in EKS, and how do you reduce it?

The Kubernetes API server is the control plane entry point.

If compromised or misused, attackers can control the cluster.

They may be able to:

Create pods
Read secrets
Modify RBAC
Exec into containers
Create privileged DaemonSets
Mount host paths
Deploy crypto miners
Steal service account tokens

Main attack surface

A. Public endpoint exposure

EKS clusters can have public and/or private API endpoints.

Risk:

Public endpoint reachable from internet
Weak IAM/RBAC mapping
Stolen developer credentials
Brute-force or token abuse attempts

B. IAM authentication

EKS authentication integrates with AWS IAM.

Risk:

Overprivileged IAM principal mapped to cluster-admin
Stale IAM roles
Developer role has production cluster-admin
Compromised AWS credentials become cluster access

C. Kubernetes RBAC

Risk:

cluster-admin overuse
wildcard verbs/resources
Users can create RoleBindings to escalate
Service accounts can read secrets broadly

Bad RBAC:

verbs: ["*"]
resources: ["*"]

D. Admission webhooks

Model answer

The Kubernetes API server attack surface includes public endpoint exposure, IAM-to-Kubernetes authentication, RBAC misconfiguration, admission webhooks, secrets access, service account tokens, and dangerous pod creation permissions. I would reduce it by using private or CIDR-restricted endpoints, least-privileged IAM and RBAC, minimal cluster-admin access, strong admission policies, Pod Security Admission, restricted secret access, control plane audit logging, and service account token hardening.

8. How do you enforce network segmentation between namespaces in EKS?

Network segmentation means:

Pods in namespace A should not freely talk to pods in namespace B.

By default, Kubernetes networking is often permissive unless network policies are enforced.

Target model

Namespace: frontend
  Can call: payments API

Namespace: payments
  Can call: database
  Cannot call: risk namespace directly

Namespace: risk
  Can call: risk database
  Cannot receive traffic from frontend

Controls

A. Use Kubernetes NetworkPolicy

Start with default deny.

Example:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: payments
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress

This blocks all ingress and egress for pods in the payments namespace unless other policies allow it.

B. Allow only specific namespace-to-namespace traffic

Example:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-payments
  namespace: payments
spec:
  podSelector:
    matchLabels:
      app: payments-api
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              name: frontend
          podSelector:
            matchLabels:
              app: frontend-api
      ports:
        - protocol: TCP
          port: 8080

Meaning:

Only pods labeled app=frontend-api in namespace frontend can reach payments-api on port 8080.

C. Enforce egress policies

Ingress-only control is not enough.

Example:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: payments-egress
  namespace: payments
spec:
  podSelector:
    matchLabels:
      app: payments-api
  policyTypes:
    - Egress
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              name: database
      ports:
        - protocol: TCP
          port: 5432

D. Use a network policy engine

Kubernetes NetworkPolicy requires enforcement by the CNI/plugin.

Options include:

Amazon VPC CNI network policy support
Calico
Cilium
Other compatible policy engines

Without enforcement, NetworkPolicy objects may exist but do nothing.

E. Use Security Groups for Pods where appropriate

Security Groups for Pods can enforce AWS security group controls at pod level.

Good for:

Pod-to-RDS access
Pod-to-AWS service access
Sensitive workload segmentation

But it has scaling and operational tradeoffs, so do not blindly apply everywhere.

F. Admission control to require policies

Prevent new namespaces or workloads from being deployed without policies.

Use:

OPA Gatekeeper
Kyverno
ValidatingAdmissionPolicy
Platform templates

Example rules:

Every namespace must have default deny
Every app must define allowed ingress
No pod may use hostNetwork unless approved
No egress to 0.0.0.0/0 unless exception exists

Model answer

I would enforce namespace segmentation with Kubernetes NetworkPolicy using a default-deny model, then explicitly allow required namespace-to-namespace and pod-to-pod traffic. I would use an enforcing policy engine such as Amazon VPC CNI network policy support, Calico, or Cilium. For sensitive AWS resource access, I may use Security Groups for Pods. I would also use admission controls to ensure every namespace has default-deny policies and that broad egress or host networking requires approval.

9. What are the tradeoffs between running stateful workloads on EKS with EBS CSI versus EFS CSI?

EBS and EFS solve different storage problems.

EBS = block storage
EFS = shared file storage

EBS CSI

EBS provides block volumes.

Best for:

Databases
Kafka brokers
Elasticsearch/OpenSearch data nodes
Single-writer StatefulSets
Low-latency block storage
High IOPS workloads

Typical access mode:

ReadWriteOnce

Meaning:

One node can mount the volume for read/write at a time.

Important EBS constraint:

EBS volume is tied to an Availability Zone.
A pod using that volume must run in the same AZ.

This affects scheduling and failover.

EFS CSI

EFS provides shared NFS file storage.

Best for:

Shared content
User uploads
Shared configuration/assets
Multiple pods needing same files
Read-heavy shared workloads
Cross-AZ shared access

Typical access mode:

ReadWriteMany

Meaning:

Many pods across nodes can mount the same file system.

Comparison Table

Area EBS CSI EFS CSI
Storage type Block Network file system
Access mode Usually ReadWriteOnce ReadWriteMany
Performance Low-latency, high IOPS possible Shared NFS, higher latency
AZ behavior AZ-bound Regional/multi-AZ access
Best for Databases/stateful sets Shared files/content
Failover Must reschedule in same AZ unless restored/replicated Easier cross-AZ mount
Consistency Block device semantics NFS semantics
Cost model Volume size/type/IOPS Storage + throughput modes
Operational risk Volume attach/detach delays NFS performance/locking behavior

Failure modes

EBS failure modes

Pod stuck Pending due to AZ mismatch
Volume attach/detach delay
Multi-attach not supported for most patterns
Node failure delays recovery
Snapshot/restore needed for region recovery

EFS failure modes

Application not designed for NFS
File locking issues
Higher latency hurts databases
Throughput bottlenecks
Noisy neighbor shared file system behavior

Model answer

I would use EBS CSI for stateful workloads that need block storage, low latency, and high IOPS, such as databases or single-writer StatefulSets. The tradeoff is that EBS volumes are AZ-bound and typically ReadWriteOnce, so scheduling and failover require careful handling. I would use EFS CSI when multiple pods need shared ReadWriteMany file access across AZs, such as shared content or uploads. The tradeoff is higher latency and NFS semantics, which make EFS a poor fit for many database-style workloads.

10. How would you design an EKS multi-region active/active architecture with consistent state?

This is one of the hardest EKS architecture questions because Kubernetes multi-region compute is easier than multi-region state consistency.

The first thing I would say in practice:

Active/active EKS is primarily a data architecture problem, not just a Kubernetes problem.

High-level architecture

Users
  |
  v
Route 53 latency routing or AWS Global Accelerator
  |
  +-------------------+
  |                   |
  v                   v
Region A             Region B
EKS Cluster A        EKS Cluster B
ALB/NLB              ALB/NLB
App Pods             App Pods
  |                   |
  +------ Data Layer --+

Key design decisions

A. Traffic routing

Use:

AWS Global Accelerator
Route 53 latency-based routing
Route 53 weighted routing
Route 53 health checks
CloudFront for edge-heavy apps

Active/active means both regions receive traffic.

B. Cluster independence

Each region should have its own:

EKS cluster
VPC
Node groups/Karpenter
Ingress controller
ECR image replication
Secrets/config
Observability pipeline
IAM roles
KMS keys where required

Avoid one region depending on the other for core runtime.

C. Consistent state

This is the hard part.

Options depend on consistency needs.

Option 1: DynamoDB Global Tables

Best for:

Key-value workloads
High availability
Multi-region active/active writes
Low-latency regional access

Tradeoff:

Eventual consistency across regions
Conflict resolution model must be acceptable

Good example:

User preferences
Shopping cart
Session metadata
Feature flags

Option 2: Aurora Global Database

Best for:

Relational workloads
One primary write region
Fast cross-region reads
Disaster recovery

Tradeoff:

Not usually true multi-writer active/active in the simple case
Write failover is operationally sensitive

Good example:

Financial ledger where strong write ordering matters

For strict consistency, many systems choose active/passive writes even if compute is active/active.

Option 3: Event-driven state replication

Use:

Kafka/MSK replication
EventBridge global patterns
SQS/SNS with regional processing
Kinesis replication
Outbox pattern
CDC

Best for:

Eventual consistency
Microservices
Audit trails
Asynchronous workflows

Tradeoff:

Duplicate events
Ordering issues
Conflict resolution
Replay complexity

Option 4: S3 Cross-Region Replication

Best for:

Objects/files
Uploads
Static content
Data lake objects

Tradeoff:

Asynchronous replication
Conflict handling if both regions write same key

D. Conflict resolution

For active/active writes, define conflict rules.

Examples:

Last-write-wins
Version vector
Application-level merge
Region ownership per user/account
Single-writer partitioning
Idempotency keys

For financial/security-sensitive systems, avoid naive last-write-wins.

E. Secrets and config

Use:

Secrets Manager replication
SSM Parameter Store per region
AWS AppConfig
External Secrets Operator
GitOps

Do not let Region B depend on Region A to start pods.

F. Deployment model

Use GitOps or CI/CD:

Build image once
Replicate to regional ECR
Deploy same version to Region A and B
Validate with synthetic checks
Shift traffic gradually

G. Failure handling

Failure scenarios:

Region A total outage
Region A data layer degraded
Network partition between regions
One region has bad deployment
Replication lag increases
Global traffic manager sends users to wrong region

Controls:

Regional health checks
Cell-based architecture
Circuit breakers
Regional isolation
Automated traffic shift
Manual override
Runbooks
Game days

H. Observability

You need global and regional views:

Per-region error rate
Per-region latency
Replication lag
Data conflict count
Queue lag
Pod health
Node capacity
Ingress health
AWS API errors

Strong design pattern

For many enterprises, the safest design is:

Active/active compute
Region-local reads
Partitioned or single-writer data model
Asynchronous replication
Automated failover
Clear conflict strategy

Example:

Users in Asia → Region Singapore
Users in Europe → Region Frankfurt

Each user account has a home region for writes.
Other region can serve cached/read-only data.
Failover moves account write ownership only during incident.

This avoids uncontrolled multi-writer conflict.

Model answer

I would deploy independent EKS clusters in two or more regions, fronted by Route 53 latency routing or AWS Global Accelerator. Each region would have its own VPC, ingress, node capacity, ECR replica, secrets, configuration, and observability. The hard part is consistent state. For key-value active/active workloads, I would consider DynamoDB Global Tables with explicit conflict handling. For relational workloads requiring strong consistency, I would usually avoid true multi-writer active/active and use Aurora Global Database or a single-writer/partitioned-writer model. I would design idempotency, conflict resolution, replication lag monitoring, regional health checks, and tested failover runbooks.

Key Takeaways

  • EKS manages the control plane; you own nodes, networking, workload config, and RBAC.
  • IRSA and Pod Identity are the correct way to give pods AWS API access — never use node-level credentials.
  • Network policies are opt-in; without them, all pods communicate freely by default.
  • The API server endpoint and aws-auth ConfigMap (or access entries) are the two most common misconfConfiguration targets.
  • EKS requires ongoing operational effort: version upgrades, add-on updates, cluster scaling policy, and security posture management.
