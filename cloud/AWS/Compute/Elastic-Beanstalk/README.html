Elastic Beanstalk — AWS Elastic Beanstalk

Category: Compute
AWS Service Type: Platform as a Service (PaaS)
Key Docs: Elastic Beanstalk Developer Guide

Overview

Elastic Beanstalk is a fully managed PaaS that deploys and manages web applications and services without requiring deep infrastructure knowledge. Developers upload application code or Docker containers, and Beanstalk provisions EC2, Auto Scaling Groups, load balancers, CloudWatch alarms, and S3 for log storage. It supports Node.js, Python, Java, .NET, Ruby, PHP, Go, and Docker. While it hides infrastructure complexity, all provisioned resources are accessible and customizable via .ebextensions configuration files and Platform Hooks. Beanstalk is best suited for small to medium applications that benefit from fast deployment without requiring full Kubernetes or ECS operational expertise.

How It Works

Core Mechanics

Concept Description
Application Logical container for environments and versions
Environment Running deployment of one application version on provisioned infrastructure
Application version S3-stored ZIP of code or Docker config with a version label
Environment tier Web server tier (ALB-backed) or worker tier (SQS-backed for background jobs)
Platform Runtime stack: Python 3.12, Node.js 20, Docker, etc.
.ebextensions YAML config files embedded in the app bundle to customize infrastructure
Platform hooks Shell scripts run at lifecycle phases (prebuild, predeploy, postdeploy)
Managed updates Beanstalk can auto-apply platform and OS patches during a maintenance window
Saved configurations Environment snapshots for reuse or rollback

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Default EC2 instance role too broad Replace default profile with least-privilege custom instance profile IAM policy design
2 Environment variables used for secrets Fetch from Secrets Manager or SSM at startup via .ebextensions Secrets management
3 All-at-once deploy causes downtime Use rolling with additional batch, immutable, or blue/green deploys Deploy policy configuration
4 No managed platform updates Enable managed platform updates with maintenance window Beanstalk configuration
5 .ebextensions not reviewed Treat .ebextensions as infrastructure code; require peer review DevSecOps process
6 Environment not in VPC Launch environments in custom VPC with private subnets for EC2 Network configuration
7 Public load balancer with no WAF Attach AWS WAF to the ALB; enable rate limiting and managed rules WAF and ALB
8 No HTTPS on load balancer listener Configure HTTPS listener on ALB with ACM certificate; redirect HTTP Load balancer config
9 Log access not restricted Restrict S3 bucket policy for log storage; enable encryption at rest S3 bucket policy
10 Health reporting not monitored Configure CloudWatch alarms on environment health and alarm on degraded Observability

Top 3 Critical Mitigations

  1. Replace the default IAM instance profile: The aws-elasticbeanstalk-ec2-role default profile grants broad permissions. Create a minimal custom role with only the permissions your application code actually needs.
  2. Use immutable or blue/green deployments for production: All-at-once deploys drop traffic during the update window. Immutable launches new instances first, waits for health checks, then swaps the ASG. Blue/green deploys to a new environment then swaps DNS — fully reversible.
  3. Enable managed platform updates: Beanstalk can auto-apply minor and patch version updates during a configurable maintenance window. Without this, runtime CVEs accumulate silently.

1. What are the tradeoffs between all-at-once, rolling, immutable, and blue/green deployment policies?

Elastic Beanstalk supports multiple deployment styles. The main tradeoff is between:

Speed
Cost
Risk
Rollback safety
User impact

A. All-at-once deployment

All instances are updated at the same time.

Old version on all instances
        ↓
Stop app / deploy new version everywhere
        ↓
New version on all instances

Benefits

Fastest deployment
Lowest cost
Simple to operate
Good for dev/test environments

Risks

Causes downtime or brief service interruption
If deployment fails, all instances are affected
Rollback may take time
Not ideal for production

Example

Use all-at-once for:

Development environment
Internal test app
Non-critical admin tool

Avoid for:

Customer-facing payment API
Production trading system
High-availability web application

B. Rolling deployment

Elastic Beanstalk updates instances in batches.

Batch 1: update some instances
Batch 2: update next instances
Batch 3: update remaining instances

Benefits

Less downtime than all-at-once
Lower cost than immutable or blue/green
Limits blast radius to one batch at a time

Risks

Reduced capacity during deployment
Old and new versions run together temporarily
Bad if application versions are incompatible
Rollback is slower than blue/green

Example

If you have 10 instances and batch size is 2:

Update 2 instances
Keep 8 serving traffic
Then update next 2
Continue until all are updated

This works well when the app supports mixed versions.

C. Rolling with additional batch

Elastic Beanstalk launches an additional batch first, then performs rolling deployment.

Original capacity: 10
Add temporary batch: +2
Update old instances in batches
Maintain full capacity during deployment

Benefits

Maintains capacity better than normal rolling
Lower risk than rolling
Less expensive than full immutable

Risks

Costs more than normal rolling
Still has mixed old/new versions
Still not as clean as blue/green

Use this when production capacity must not drop during deployment.

D. Immutable deployment

Elastic Beanstalk launches a full set of new instances with the new version. If the new instances pass health checks, traffic moves to them and old instances are terminated. AWS documents that traffic-splitting deployments work similarly by launching a full set of new instances, sending a percentage of traffic to the new version, and rolling back if health checks fail. (AWS Documentation)

Old environment instances keep serving traffic
        ↓
New instances are launched separately
        ↓
New version is validated
        ↓
Traffic moves to new instances
        ↓
Old instances are removed

Benefits

Very safe
No in-place mutation of existing instances
Easy rollback before cutover
Avoids partially updated instances
Good for production

Risks

More expensive during deployment
Slower than rolling
Needs enough EC2 capacity

Example

Use immutable deployment for:

Production banking app
Security-sensitive workload
Critical API
Application with complex bootstrapping

E. Blue/green deployment

Blue/green uses two separate environments.

Blue environment  = current production
Green environment = new version

Traffic is shifted by swapping CNAMEs, changing Route 53, or changing load balancer routing.

Users → Blue
          ↓
Deploy and test Green
          ↓
Switch traffic
Users → Green

Benefits

Strong rollback story
Test green before exposing users
No in-place changes
Good for major releases
Good for database migration rehearsals

Risks

Highest cost during parallel run
More operational complexity
Need to handle environment config drift
Database compatibility is still hard

Example

Use blue/green for:

Major framework upgrade
Large database migration
High-risk release
Security-critical production system

Comparison Table

Policy Speed Cost Downtime Risk Rollback Best For
All-at-once Fastest Lowest Highest Weak Dev/test
Rolling Medium Low Medium Medium Normal production apps
Rolling with additional batch Medium Medium Lower Medium Production with capacity sensitivity
Immutable Slower Higher Low Strong Safer production deploys
Blue/green Medium Highest Lowest Strongest High-risk releases

Model answer

All-at-once is fastest and cheapest but has the highest downtime and blast-radius risk. Rolling reduces risk by updating batches, but temporarily runs mixed versions and may reduce capacity. Rolling with an additional batch preserves capacity better. Immutable launches a new fleet and cuts over only after health checks pass, making it safer but more expensive. Blue/green uses two separate environments, gives the best rollback and validation story, but costs more and requires stronger operational discipline.

2. How does Elastic Beanstalk relate to the underlying EC2 and Auto Scaling resources it provisions?

Elastic Beanstalk is a managed application platform, not a separate compute service.

Under the hood, it provisions normal AWS resources such as:

EC2 instances
Auto Scaling Group
Elastic Load Balancer
Security groups
IAM roles
CloudWatch alarms
Launch templates/configurations
S3 application version storage
Optional RDS if configured

Think of Elastic Beanstalk as an orchestration layer:

Developer uploads application
        ↓
Elastic Beanstalk environment
        ↓
CloudFormation-managed AWS resources
        ↓
EC2 + ASG + ELB + CloudWatch + IAM

.ebextensions can even create additional CloudFormation resources because Elastic Beanstalk uses CloudFormation to create environments. (AWS Documentation)

What Elastic Beanstalk manages for you

Elastic Beanstalk handles:

Provisioning infrastructure
Deploying application versions
Creating Auto Scaling resources
Creating load balancers
Health monitoring
Platform patching options
Log collection
Environment configuration
Rolling/immutable deployments

What you still own

You still own:

Application code
Security configuration
IAM permissions
VPC design
Secrets handling
Database migration
Scaling settings
Cost controls
Runtime dependencies
Patch/update risk validation

Example

When you create a load-balanced Beanstalk web environment:

Elastic Beanstalk creates:
- ALB or Classic Load Balancer
- Auto Scaling Group
- EC2 instances
- Security groups
- Instance profile
- CloudWatch health metrics

If you terminate the Beanstalk environment, those managed resources are usually deleted as part of the environment lifecycle.

Model answer

Elastic Beanstalk is a platform layer that provisions and manages underlying AWS resources such as EC2, Auto Scaling Groups, load balancers, security groups, IAM roles, and CloudWatch alarms. It simplifies deployment and lifecycle management, but those resources are still real AWS infrastructure. You can inspect and sometimes customize them, but you should avoid manually changing Beanstalk-managed resources outside Beanstalk because it can cause drift.

3. When would you choose Elastic Beanstalk over ECS or EKS?

I would choose Elastic Beanstalk when I want a simple platform-as-a-service experience for a traditional application and do not need full container orchestration control.

Choose Elastic Beanstalk when

Application is a traditional web app
Team wants simple deployment
Limited platform engineering capacity
You want EC2/ASG/ALB managed for you
You do not need Kubernetes-level control
You want quick migration from VM-style apps
You use supported platforms like Java, .NET, Node.js, Python, PHP, Ruby, Go, Docker

Example:

A Java Spring Boot app packaged as a JAR/WAR
A .NET web app
A Python Django app
A legacy monolith

Choose ECS when

You are container-first
You need multiple services and task definitions
You want better container-native deployments
You want Fargate
You need fine-grained task IAM roles
You need service discovery and container scaling
You want simpler operations than Kubernetes

Choose EKS when

You need Kubernetes APIs
You need portability across cloud/on-prem
You need advanced Kubernetes ecosystem tools
You have platform engineering maturity
You need CRDs/operators/service mesh/policy engines
You run many microservices with Kubernetes-native workflows

Comparison Table

Platform Best For Operational Complexity Control
Elastic Beanstalk Simple web apps, monoliths, quick PaaS Low Medium
ECS AWS-native container workloads Medium High
EKS Kubernetes-native platforms High Highest

Model answer

I would choose Elastic Beanstalk for straightforward web applications where the team wants AWS to manage most of the EC2, Auto Scaling, load balancing, deployment, and platform lifecycle. I would choose ECS for container-native services needing task-level control and Fargate. I would choose EKS when the organization needs Kubernetes APIs, portability, operators, CRDs, or advanced multi-service platform capabilities.

4. How do .ebextensions work, and what can they configure?

.ebextensions are configuration files bundled with your application source.

They live in:

.ebextensions/

Files must usually end with:

.config

Elastic Beanstalk configuration files are YAML or JSON files placed in a .ebextensions folder in the source bundle. They can configure the environment, customize EC2 instances, and create additional resources. (AWS Documentation)

What .ebextensions can configure

.ebextensions can configure:

Elastic Beanstalk option settings
Environment variables
Auto Scaling settings
Load balancer settings
Security groups
Instance packages
Files on EC2 instances
Linux users/groups
Commands during provisioning
Container commands during deployment
Services
Additional CloudFormation resources

AWS documents sections such as packages, sources, files, users, groups, commands, container_commands, and services for configuring launched instances. (AWS Documentation)

Example: install packages

packages:
  yum:
    jq: []
    git: []

Example: create a file

files:
  "/etc/myapp/config.json":
    mode: "000600"
    owner: root
    group: root
    content: |
      {
        "logLevel": "INFO"
      }

Example: environment option settings

option_settings:
  aws:elasticbeanstalk:application:environment:
    APP_ENV: production
    LOG_LEVEL: INFO

Example: Auto Scaling setting

option_settings:
  aws:autoscaling:asg:
    MinSize: 2
    MaxSize: 6

Important caution

.ebextensions are powerful, but risky.

Bad uses:

Hardcoding passwords
Embedding private keys
Making manual OS changes that are hard to reproduce
Complex scripts with no testing
Changing Beanstalk-managed resources unpredictably

Better pattern:

Keep infrastructure config version-controlled
Use secrets manager for secrets
Use CI/CD to validate .config syntax
Prefer platform hooks for deployment lifecycle logic
Avoid manual console drift

Model answer

.ebextensions are YAML or JSON .config files placed under the .ebextensions directory in the application source bundle. Elastic Beanstalk reads them during deployment to set environment options, customize EC2 instances, install packages, create files, run commands, manage services, and create additional CloudFormation resources. They are powerful but should not contain secrets or fragile manual scripts.

5. How would you secure environment variables and secrets in a Beanstalk application?

The main rule is:

Do not store sensitive secrets directly in source code or plain .ebextensions.

Risky pattern

option_settings:
  aws:elasticbeanstalk:application:environment:
    DB_PASSWORD: SuperSecret123
    STRIPE_KEY: sk_live_xxxxxx

Problems:

Secret is in source bundle
Secret may appear in deployment history
Secret may be visible to users with environment access
Rotation is hard
Auditability is weak

Better pattern

Use:

AWS Secrets Manager
SSM Parameter Store SecureString
KMS encryption
Least-privileged instance profile
Runtime retrieval by application

Recommended design

Application starts
        ↓
Reads secret ARN/name from environment variable
        ↓
Uses instance profile IAM role
        ↓
Calls Secrets Manager or SSM Parameter Store
        ↓
Caches secret in memory
        ↓
Uses secret to connect to database/API

Environment variable contains only a reference:

DB_SECRET_ARN=arn:aws:secretsmanager:ap-southeast-1:111122223333:secret:prod/db

Not the actual password.

IAM policy example

{
  "Effect": "Allow",
  "Action": ["secretsmanager:GetSecretValue"],
  "Resource": "arn:aws:secretsmanager:ap-southeast-1:111122223333:secret:prod/db-*"
}

If using KMS customer-managed keys:

{
  "Effect": "Allow",
  "Action": "kms:Decrypt",
  "Resource": "arn:aws:kms:ap-southeast-1:111122223333:key/abcd-1234"
}

Additional controls

Use IMDSv2 on EC2 instances
Use least-privileged instance profile
Encrypt EBS volumes
Restrict console/IAM access to Beanstalk configuration
Avoid logging environment variables
Rotate secrets regularly
Use separate secrets per environment
Use CloudTrail alerts for unusual GetSecretValue calls
Use VPC endpoints for Secrets Manager/SSM where possible

Secret rotation

For database credentials:

1. Rotate secret in Secrets Manager.
2. Update database password.
3. Application refreshes secret or restarts.
4. Monitor connection failures.

If the app only reads secrets at startup, trigger a rolling or immutable deployment after rotation.

Model answer

I would avoid putting secrets directly in Beanstalk environment variables or .ebextensions. Instead, I would store secrets in AWS Secrets Manager or SSM Parameter Store SecureString, expose only the secret name or ARN as an environment variable, and let the application retrieve the secret at runtime using the EC2 instance profile. I would enforce least privilege, KMS encryption, IMDSv2, secret rotation, audit CloudTrail access, and avoid logging secret values.

6. How do managed platform updates work, and what risks should you be aware of?

Elastic Beanstalk platforms include the operating system, runtime, web server, application server, agents, and Beanstalk components.

Examples:

Java platform
Node.js platform
Python platform
Docker platform
.NET platform

Managed platform updates let Elastic Beanstalk automatically update the environment to newer platform versions during a scheduled maintenance window. AWS states that managed updates can keep the application in service during the update with no reduction in capacity. (AWS Documentation)

What managed platform updates can update

Operating system patches
Runtime patch versions
Web/application server components
Elastic Beanstalk platform components
Security fixes
Bug fixes

How they usually work

Maintenance window arrives
        ↓
Elastic Beanstalk checks for platform update
        ↓
Applies update using configured update mechanism
        ↓
Health checks validate environment
        ↓
Environment continues running or rolls back/fails

Managed platform updates can use safer deployment mechanisms such as immutable updates depending on configuration.

Benefits

Reduces unpatched platform risk
Automates routine maintenance
Helps with security fixes
Reduces manual operational burden

Risks

Runtime compatibility issues
Application breaks after platform update
Custom .ebextensions scripts fail
Dependency/version changes
Agent or proxy behavior changes
Maintenance window surprises
Insufficient testing in lower environments
Rollback complexity

Example:

A Node.js minor runtime update changes dependency behavior.
The app starts but health checks fail.
Production update rolls back or degrades.

Best practices

Enable managed updates first in dev/stage
Use the same platform branch across environments
Run automated smoke tests after updates
Pin critical application dependencies
Avoid fragile OS-level customizations
Use immutable updates for safer replacement
Monitor health during maintenance windows
Keep rollback plan ready
Track platform retirement dates

AWS Config also has a managed rule to check whether Beanstalk managed platform updates are enabled. (AWS Documentation)

Model answer

Managed platform updates automatically apply Elastic Beanstalk platform updates during a configured maintenance window. They reduce patching burden and help keep OS/runtime components current. The risk is that platform updates can change runtime, proxy, agent, or OS behavior and break the application, especially if the app relies on fragile .ebextensions customization. I would test updates in lower environments first, use immutable updates, monitor health, and maintain rollback plans.

7. How would you implement zero-downtime deployments for a Beanstalk web application?

For zero downtime, I would avoid all-at-once deployments in production.

Recommended choices:

Rolling with additional batch
Immutable
Traffic splitting
Blue/green

Option 1: Rolling with additional batch

Good when cost matters but capacity must be preserved.

Original capacity: 6
Temporary capacity: 8
Update 2 at a time
Keep at least 6 serving

Option 2: Immutable deployment

Good default for safer production deployments.

Current instances keep serving traffic
New instances launch with new version
Health checks pass
Traffic shifts to new instances
Old instances terminate

This avoids mutating live instances.

Option 3: Traffic splitting

Good for canary-style validation.

Elastic Beanstalk traffic-splitting deployments launch a full new set of instances, send a specified percentage of traffic to the new version during an evaluation period, and roll back to the old version if health checks fail. (AWS Documentation)

Example:

Send 10% traffic to new version for 15 minutes
If healthy, send 100%
If unhealthy, send 0% and terminate new instances

Option 4: Blue/green

Best for high-risk releases.

Blue environment: current production
Green environment: new version
Test green privately
Swap CNAME or Route 53 target
Rollback by switching back

Technical checklist

Use load-balanced environment, not single instance
Have at least 2 instances across AZs
Use health checks that verify real readiness
Set connection draining/deregistration delay
Make app graceful on shutdown
Use backward-compatible database migrations
Avoid breaking old/new version compatibility
Use smoke tests before traffic shift
Monitor 5xx, latency, CPU, memory, environment health
Automate rollback

Database migration strategy

Zero-downtime deployment often fails because of database changes.

Use expand/contract:

1. Add new nullable column.
2. Deploy app that writes both old and new fields.
3. Backfill data.
4. Deploy app that reads new field.
5. Remove old field later.

Avoid:

Deploy app requiring a column that does not exist yet.
Drop a column still used by old version.

Model answer

For zero-downtime Beanstalk deployments, I would use a load-balanced environment with multiple instances, strong health checks, connection draining, and either immutable, traffic-splitting, rolling with additional batch, or blue/green deployment. For critical production releases, I prefer immutable or blue/green because they avoid mutating live instances. I would also make database migrations backward-compatible, validate with smoke tests, and monitor health metrics for automated rollback.

8. How does the worker tier differ from the web server tier, and when would you use it?

Elastic Beanstalk has two main environment tiers:

Web server tier
Worker tier

Web server tier

The web server tier handles HTTP/S requests from users or clients.

User
  ↓
Load Balancer
  ↓
Beanstalk web EC2 instances
  ↓
Application

Use for:

Web apps
REST APIs
Frontend/backend services
Customer-facing applications

Worker tier

The worker tier is for background processing.

Elastic Beanstalk worker environments use Amazon SQS and a daemon on each EC2 instance. The daemon reads messages from the queue and sends an HTTP POST request to the local application on port 80 with the message body. (AWS Documentation)

Web app
  ↓
SQS queue
  ↓
Beanstalk worker daemon
  ↓
HTTP POST to local worker app
  ↓
Background job runs

AWS also documents that worker environments provision resources such as an Auto Scaling group, EC2 instances, an IAM role, and an SQS queue if one is not already provided. (AWS Documentation)

When to use worker tier

Use worker tier for tasks that should not block the web request.

Examples:

Sending emails
Generating reports
Image/video processing
Payment reconciliation
PDF generation
Data import/export
Webhook retry processing
Long-running business workflows

Example

Bad web flow:

User uploads image
        ↓
Web server resizes image for 30 seconds
        ↓
User waits
        ↓
Request times out

Better flow:

User uploads image
        ↓
Web server stores file in S3
        ↓
Web server sends SQS message
        ↓
Worker processes image asynchronously
        ↓
User gets notification when done

Web vs Worker Table

Area Web Server Tier Worker Tier
Main purpose Serve HTTP requests Process background jobs
Input User/client traffic SQS messages
Fronted by Load balancer SQS + worker daemon
Best for APIs/web apps Async processing
Scaling metric Request/CPU/latency Queue depth/job backlog
Failure handling HTTP retry/load balancing SQS retry/dead-letter queue

Model answer

The web server tier serves synchronous HTTP/S traffic through a load balancer. The worker tier processes asynchronous background jobs from SQS. Beanstalk worker environments run a daemon that pulls SQS messages and posts them locally to the application. I would use the worker tier for long-running or retryable tasks such as email sending, report generation, image processing, and webhook handling.

10. How would you migrate a legacy Elastic Beanstalk application to ECS or EKS?

I would migrate gradually, not with a big bang.

The migration target depends on the organization:

ECS = simpler AWS-native container platform
EKS = Kubernetes-native platform with higher control and complexity

Step 1: Assess current Beanstalk app

Inventory:

Runtime and platform version
.ebextensions
Environment variables
Secrets
IAM instance profile permissions
Load balancer config
Auto Scaling rules
Health checks
Background jobs
Cron jobs
File system usage
Logs
Database dependencies
Deployment process
DNS/CNAME routing

Special attention:

Anything in .ebextensions must be converted.
Anything stored on local disk must be externalized.
Any instance-level dependency must be containerized or moved.

Step 2: Containerize the application

Create:

Dockerfile
Health check endpoint
Non-root runtime
Structured logging to stdout/stderr
Configuration through environment variables or secrets

Bad pattern:

App writes important data to local EC2 disk

Better:

App writes files to S3/EFS
App stores state in RDS/DynamoDB/ElastiCache

Step 3: Move configuration and secrets

Convert:

Beanstalk environment variables → ECS task env / Kubernetes ConfigMap
Beanstalk secrets → Secrets Manager / SSM Parameter Store
Instance profile → ECS task role or IRSA/Pod Identity

Step 4: Replace Beanstalk infrastructure mapping

Model answer

I would first inventory the Beanstalk environment, especially .ebextensions, environment variables, IAM instance profile, load balancer, scaling rules, worker tier, logs, and local disk dependencies. Then I would containerize the app, externalize state, move secrets to Secrets Manager, map IAM to ECS task roles or EKS Pod Identity/IRSA, and recreate infrastructure with IaC. I would run Beanstalk and ECS/EKS in parallel and gradually shift traffic using weighted DNS or ALB routing, while monitoring latency, errors, and business metrics. After stable cutover, I would decommission Beanstalk resources.

Key Takeaways

  • Beanstalk is PaaS — fast to start, but all AWS resources it creates are visible and customizable.
  • The default IAM instance profile is far too permissive for production; always replace it.
  • Deploy policy choice significantly impacts availability, rollback ability, and deployment speed.
  • Environment variables are not appropriate for secrets; use Secrets Manager retrieved at startup.
  • Managed updates and health monitoring are opt-in but essential for production environments.
