Fargate — AWS Fargate

Category: Compute
AWS Service Type: Serverless Container Runtime
Key Docs: Fargate User Guide, Fargate for EKS

Overview

Fargate is a serverless compute engine for containers that works with both ECS and EKS. It eliminates the need to provision, configure, or scale EC2 instances for container workloads. Each Fargate task or pod runs in its own dedicated kernel isolation boundary using lightweight micro-VMs, providing stronger workload isolation than shared EC2 hosts. AWS manages the underlying infrastructure, patching, and scaling. Fargate is priced by vCPU and memory consumed per second. It is ideal for unpredictable workloads, short-lived batch jobs, and teams that want to minimize operational overhead at the cost of higher per-unit cost compared to EC2.

How It Works

Core Mechanics

vCPU and memory configs Pre-defined combinations: 0.25 vCPU to 16 vCPU, 0.5 GB to 120 GB RAM

No SSH access to hosts No access to underlying infrastructure; SSM or Exec for container access

Top 3 Critical Mitigations

  1. Task role least privilege: Fargate tasks run their own IAM role and can call AWS APIs directly. Scope each task role carefully — credential theft from a Fargate task is equivalent to credential theft from a Lambda or EC2 instance profile.
  2. Use Secrets Manager valueFrom in task definitions: Instead of environment variables, reference secret ARNs in the task definition. ECS retrieves and injects secrets at launch using the execution role. The values never appear in CloudTrail or the console.
  3. Handle SIGTERM for Spot reclamation: Fargate Spot sends SIGTERM 120 seconds before reclamation. Apps that don't handle it gracefully lose in-flight work. Add a SIGTERM handler, drain connections, checkpoint state, and exit cleanly.

AWS Fargate Model Answers — Detailed with Examples

1. How does Fargate's isolation model differ from running containers on a shared EC2 node?

With EC2-backed ECS/EKS, multiple containers from different tasks or pods can run on the same EC2 worker node.

EC2 worker node
 ├── Container / task A
 ├── Container / task B
 ├── Container / task C
 └── Shared host kernel, runtime, network, disk, IAM instance role risk

With Fargate, AWS manages the underlying compute. You do not manage or access the worker node. Each task or pod gets a more isolated runtime boundary and its own allocated CPU, memory, network interface, and ephemeral storage.

Fargate task A → isolated managed runtime
Fargate task B → isolated managed runtime
Fargate task C → isolated managed runtime

Main difference

Area Fargate Shared EC2 Node
Host management AWS manages host You manage EC2 worker nodes
Patch responsibility AWS handles Fargate infrastructure You patch AMIs, runtime, kubelet/ECS agent
Isolation Stronger task-level managed isolation Multiple workloads share node/kernel
Host access No SSH/host access You can SSH/SSM into node
Privileged workloads Limited/not suitable Possible if allowed
DaemonSets/host agents Limited, especially on EKS Fargate Fully supported
Operational control Lower Higher
Security burden Lower host burden Higher node hardening burden

Why Fargate is safer operationally

On shared EC2 nodes, a compromised container may attempt to attack:

Host kernel
Container runtime
Other containers on same node
Node IAM role
Instance metadata service
Host-mounted volumes
Docker/containerd socket

On Fargate, you remove much of that host-level attack surface because you do not expose or operate the underlying node.

However, Fargate does not remove application risk. You still need:

Least-privileged task role
Secure container image
No secrets in image
Network egress controls
Runtime logging
Patch application dependencies

Model answer

Fargate provides a stronger managed isolation model because each task or pod runs without the customer managing or sharing an EC2 worker node. With EC2 node groups, multiple containers share the same host kernel, container runtime, instance profile, and node resources, so node compromise can affect co-located workloads. Fargate reduces host-management and cross-container risk, but gives less low-level control and still requires application, IAM, image, and network hardening.

2. When would you choose Fargate over EC2 node groups for an EKS workload?

For EKS, Fargate is useful when you want to run Kubernetes pods without managing worker nodes.

Choose Fargate when

You want no node management
You run stateless workloads
You have bursty or variable workloads
You want stronger workload isolation
You do not need privileged pods or DaemonSets
You want simpler patching responsibility
You want per-pod compute billing

Good examples:

Internal APIs
Small microservices
Event-driven workers
Batch jobs
Dev/test workloads
Low-ops production services
Tenant-isolated workloads

Choose EC2 node groups when

You need DaemonSets
You need privileged containers
You need GPUs
You need custom AMIs
You need host-level monitoring/security agents
You need high sustained utilization
You need storage/network tuning
You need specialized instance types
You need very predictable low latency

Good examples:

Service mesh with node agents
Security tools requiring host access
GPU inference/training
High-throughput data platforms
Large steady-state services
Stateful workloads needing node/storage tuning

EKS-specific limitation to remember

EKS Fargate does not behave like a normal worker-node environment. You cannot treat it as “managed EC2 nodes.” You lose many node-level controls. That matters if your platform relies on DaemonSets, host networking, node-local caching, custom CNI behavior, or privileged security agents.

Model answer

I would choose Fargate for EKS workloads when I want a serverless pod experience, stronger pod isolation, lower node-management burden, and the workload is stateless or simple. I would choose EC2 node groups when I need DaemonSets, privileged pods, GPUs, custom AMIs, node-level agents, specialized networking, high sustained utilization, or deep performance tuning. For many platforms, a hybrid model works well: Fargate for simple isolated workloads, EC2 node groups for platform-heavy or performance-sensitive workloads.

3. How does Fargate Spot work, and how do you design for its reclamation behavior?

Fargate Spot runs tasks on spare AWS Fargate capacity at a lower price. The tradeoff is that AWS can reclaim the capacity when it needs it back. AWS says Fargate Spot tasks can be interrupted with a two-minute notification when AWS needs the capacity back. (Amazon Web Services, Inc.)

Good use cases

Batch processing
Queue workers
Image/video processing
CI jobs
Stateless APIs with enough replicas
Fault-tolerant background jobs
Non-critical async tasks

Bad use cases

Single critical production task
Non-idempotent job without checkpointing
Long-running transaction that cannot stop
Stateful workload with local-only state
Latency-critical workload with no fallback capacity

Reclamation flow

Fargate Spot task running
        ↓
AWS needs capacity back
        ↓
Task receives interruption notice
        ↓
Application gets SIGTERM
        ↓
Application has limited time to shut down gracefully
        ↓
Task stops

Design for interruption

Your app should treat Spot interruption as normal, not exceptional.

Controls:

Handle SIGTERM
Stop accepting new work
Finish or checkpoint current work
Use idempotency keys
Use SQS/Kafka visibility/retry semantics
Save progress to durable storage
Keep baseline capacity on regular Fargate or EC2
Use ECS capacity providers with weighted mix
Configure stopTimeout appropriately

Example capacity provider strategy:

Base: 4 tasks on regular Fargate
Burst: additional tasks on Fargate Spot

For queue workers:

1. Worker receives SQS message.
2. Worker starts processing.
3. Spot interruption notice arrives.
4. Worker checkpoints progress or stops before deleting message.
5. Message becomes visible again.
6. Another worker retries safely.

Model answer

Fargate Spot uses spare Fargate capacity at a discount, but tasks can be reclaimed with a short interruption notice. I would use it only for stateless, retryable, checkpointable, or queue-based workloads. The design should handle SIGTERM, stop accepting new work, checkpoint or complete in-flight jobs, store state outside the task, use idempotency, and keep critical baseline capacity on regular Fargate or EC2.

4. What is the maximum ephemeral storage for a Fargate task, and what should you never store there?

For ECS on Fargate, tasks get 20 GiB of ephemeral storage by default, and this can be increased up to 200 GiB using the ephemeralStorage parameter in the task definition. (AWS Documentation)

Important point

It is called ephemeral because it is temporary.

Task starts → ephemeral storage exists
Task stops → ephemeral storage is gone

Never store this there

Do not store durable or irreplaceable data in Fargate ephemeral storage.

Avoid storing:

Database files
Customer uploads
Transaction logs that must survive
Audit logs as the only copy
Generated reports as the only copy
Payment evidence
Model training checkpoints as the only copy
Secrets
Private keys
Long-term application state

Acceptable uses

Temporary files
Scratch data
Intermediate processing files
Unpacked archives
Short-lived caches
Temporary image/video processing files
Build artifacts during task lifetime

Better durable destinations

Data Type Better Storage
Customer uploads S3
Database state RDS, DynamoDB, Aurora
Shared files EFS
Logs CloudWatch Logs, OpenSearch, S3
Secrets Secrets Manager / SSM Parameter Store
Checkpoints S3, EFS, DynamoDB

AWS also recommends encrypting Fargate ephemeral storage, either through AWS-managed encryption or customer-managed keys depending on configuration/platform support. (AWS Documentation)

Model answer

A Fargate task gets 20 GiB of ephemeral storage by default, configurable up to 200 GiB. I would use it only for temporary scratch data, caches, and intermediate files. I would never store durable state, customer uploads, audit logs as the only copy, database files, secrets, private keys, or anything that must survive task termination. Durable data belongs in S3, EFS, RDS, DynamoDB, or a logging system.

5. How do Fargate tasks authenticate to ECR and retrieve secrets from Secrets Manager?

There are two important IAM roles:

Task execution role = used by ECS/Fargate runtime
Task role           = used by your application code

AWS states that the task execution role lets the ECS container and Fargate agents make AWS API calls on your behalf, such as pulling images and writing logs. (AWS Documentation)

Image pull from ECR

Flow:

ECS/Fargate runtime
        ↓
Uses task execution role
        ↓
Authenticates to ECR
        ↓
Pulls container image
        ↓
Starts container

Typical execution role permissions include:

ecr:GetAuthorizationToken
ecr:BatchCheckLayerAvailability
ecr:GetDownloadUrlForLayer
ecr:BatchGetImage
logs:CreateLogStream
logs:PutLogEvents

Secrets injection from Secrets Manager

If the task definition references Secrets Manager secrets using the ECS secrets field, the secret is fetched during task startup and injected into the container. AWS documents that this pattern requires the task execution role to have additional Secrets Manager permissions. (AWS Documentation)

Flow:

ECS/Fargate runtime
        ↓
Uses task execution role
        ↓
Calls Secrets Manager
        ↓
Injects secret into container as env var
        ↓
Container starts

Runtime secret retrieval by application

If your application retrieves secrets itself during runtime:

Application code inside container
        ↓
Uses task role
        ↓
Calls Secrets Manager GetSecretValue
        ↓
Caches/uses secret

Security best practice

Use different roles:

Execution role:
  Pull image
  Write logs
  Fetch startup-injected secrets if needed

Task role:
  Only application-level AWS permissions

Bad pattern:

{
  "Action": "*",
  "Resource": "*"
}

Better pattern:

{
  "Effect": "Allow",
  "Action": "secretsmanager:GetSecretValue",
  "Resource": "arn:aws:secretsmanager:ap-southeast-1:111122223333:secret:prod/payments/db-*"
}

Model answer

Fargate uses the ECS task execution role to pull images from ECR, write logs, and retrieve secrets that are injected by ECS at startup. The application itself uses the task role for AWS API calls at runtime. If Secrets Manager values are referenced in the task definition, the execution role needs permission. If the application fetches secrets dynamically, the task role needs permission. Both roles should be least-privileged and separated.

6. What networking constraints apply to Fargate tasks, and why can't you use bridge networking mode?

Fargate tasks use awsvpc networking. AWS says awsvpc gives ECS tasks the same networking properties as EC2 instances, including their own network interface and VPC-level networking behavior. (AWS Documentation)

Fargate networking model

Fargate task
   ↓
Elastic Network Interface
   ↓
Private IP from subnet
   ↓
Security group
   ↓
VPC routing

Each task gets:

Own ENI
Own private IP
Security group attachment
Subnet placement
VPC routing behavior

Why not bridge mode?

Bridge mode depends on a shared Docker bridge running on an EC2 host.

EC2 host
 ├── docker0 bridge
 ├── container A
 ├── container B
 └── host port mappings

In Fargate, you do not manage or share the host. There is no customer-visible Docker bridge or host-level port mapping model. Therefore, Fargate uses awsvpc, not bridge networking. AWS’s ECS networking documentation recommends awsvpc unless another mode is specifically needed for EC2-backed tasks. (AWS Documentation)

Constraints to remember

Task must be placed in a subnet
Task consumes subnet IP capacity
Security groups apply at task level
No host/bridge networking
No host port sharing model
Private subnet tasks need NAT or VPC endpoints for outbound AWS/internet access
Public subnet tasks need public IP assignment if direct internet egress is required
ENI/IP capacity and account quotas matter

Practical example

For a private Fargate service pulling from ECR and writing logs:

Private subnet
No public IP
VPC endpoints for:
  ECR API
  ECR DKR
  CloudWatch Logs
  Secrets Manager
  S3 gateway endpoint for image layers if needed

Without NAT or required VPC endpoints, the task may fail to pull images, fetch secrets, or write logs.

Model answer

Fargate tasks use awsvpc networking, where each task gets its own ENI, private IP, subnet placement, and security group. Bridge mode is not supported because bridge networking depends on a shared EC2 host and Docker bridge, while Fargate abstracts away the host. The main constraints are subnet IP availability, security group design, VPC routing, NAT or VPC endpoints for outbound access, and no host-level port mapping.

7. How would you debug a Fargate task that exits immediately without logs?

This is a common real-world issue.

The key is to determine whether the task failed:

Before container started
During image pull
During secret retrieval
During app startup
After app started but before logs flushed

Step 1: Check ECS stopped reason

Look at:

ECS service events
Task stopped reason
Container exit code
Container reason

Common examples:

CannotPullContainerError
ResourceInitializationError
Essential container in task exited
OutOfMemoryError
CannotCreateContainerError
AccessDeniedException

Step 2: Check image pull path

If there are no logs, the container may never have started.

Check:

Does execution role have ECR permissions?
Can task reach ECR?
Private subnet has NAT or ECR VPC endpoints?
Image tag exists?
Image architecture matches Fargate runtime?

Example issue:

Task in private subnet
No NAT Gateway
No ECR/CloudWatch Logs VPC endpoints
Result: task cannot pull image or write logs

Step 3: Check CloudWatch Logs configuration

No logs may mean logging is misconfigured.

Check:

awslogs driver configured?
Log group exists or execution role can create it?
Correct region?
Execution role has logs:CreateLogStream and logs:PutLogEvents?
Application writes to stdout/stderr?

Step 4: Check secrets and environment variables

Startup can fail before logging if secrets cannot be retrieved.

Check:

Execution role can read injected Secrets Manager secret?
KMS decrypt permission exists?
Secret ARN and region are correct?
Secret version/stage exists?
VPC endpoint or NAT path to Secrets Manager exists?

Step 5: Check application command and entrypoint

Common container mistakes:

Wrong ENTRYPOINT
Wrong CMD
Missing executable permission
App exits after running one command
Missing config file
Missing required env var
App binds to wrong port
App listens on localhost only

Example:

CMD ["python", "manage.py", "migrate"]

This runs migration and exits immediately. For a service, that is wrong.

Step 6: Check CPU/memory and OOM

If the container exits quickly:

Exit code 137 often means killed/OOM
Exit code 1 usually app error
Exit code 0 means app completed successfully

Step 7: Reproduce locally

Run the exact image locally:

docker run --rm \
  -e APP_ENV=prod \
  -e DB_HOST=example \
  image:tag

Or run a debug variant:

Override command to sleep
Use ECS Exec if the task stays alive
Add early startup logging
Add a /health or /ready endpoint

Step 8: Add bootstrap logging

At the top of startup script:

echo "Starting app..."
echo "Checking env..."
echo "Checking config files..."

Do not print secrets.

Model answer

I would first check the ECS task stopped reason, container exit code, and service events. If there are no logs, I would suspect the container never started or logging failed. I would check ECR pull permissions, private subnet NAT/VPC endpoints, task execution role permissions, CloudWatch Logs config, Secrets Manager/KMS access, entrypoint/CMD errors, missing environment variables, and OOM kills. Then I would reproduce the image locally or run a debug task with a sleep command and add early startup logs.

8. What are the cost differences between Fargate and equivalent EC2 capacity, and when does EC2 win?

Fargate charges based on the resources requested by the task: vCPU, memory, OS/architecture, runtime duration, and additional ephemeral storage beyond the included amount. AWS’s pricing examples show vCPU-second, GB-second, and additional ephemeral storage pricing dimensions. (Amazon Web Services, Inc.)

EC2 charges for instances, regardless of how efficiently you pack containers onto them.

Fargate cost model

Pay per task requested vCPU
Pay per task requested memory
Pay for runtime duration
Pay extra for configured ephemeral storage above included amount
No direct node management cost

EC2 cost model

Pay for EC2 instances
Pay for EBS volumes
Pay whether capacity is fully used or idle
Can use Reserved Instances / Savings Plans / Spot
Can bin-pack many containers onto nodes

When Fargate wins

Low operational overhead matters
Workloads are spiky or intermittent
Utilization is unpredictable
Small teams
You want strong isolation
You do not want node patching
You run many small services with variable demand

Example:

A service runs 20 minutes every hour.
Fargate may be cheaper because you pay only while tasks run.

When EC2 wins

EC2 usually wins when workloads are steady and highly utilized.

High sustained CPU/memory usage
Large stable services
Many tasks can be packed efficiently
Need Spot diversification
Need Reserved Instance/Savings Plan optimization
Need GPUs or special instance types
Need host-level tuning

Example:

100 services run 24/7.
Average node utilization can be kept at 70–80%.
Platform team can operate nodes well.
EC2 may be cheaper than Fargate.

Simple mental model

Fargate = pay for simplicity and per-task isolation
EC2     = pay for raw capacity and optimize utilization yourself

Cost trap with Fargate

Fargate charges based on requested resources, not actual app usage.

If a task requests:

2 vCPU and 8 GB memory

but actually uses:

0.2 vCPU and 1 GB memory

you are overpaying.

Model answer

Fargate is priced per task based on requested vCPU, memory, runtime duration, and additional ephemeral storage, while EC2 is priced as instance capacity that you manage and bin-pack. Fargate often wins for bursty, intermittent, smaller, or low-ops workloads. EC2 wins for steady high-utilization workloads, large fleets, special hardware, aggressive Spot/RI/Savings Plan optimization, or workloads needing host-level tuning. The key cost question is whether you can keep EC2 capacity highly utilized.

10. What scaling limits apply to Fargate, and how do you plan for sudden demand spikes?

Fargate removes EC2 node management, but it does not remove all scaling limits.

Scaling dependency chain

Traffic spike
   ↓
Auto Scaling policy triggers
   ↓
ECS desired count increases
   ↓
Fargate capacity is requested
   ↓
Images are pulled
   ↓
Secrets/config fetched
   ↓
Task starts
   ↓
Health checks pass
   ↓
Load balancer sends traffic

Any step can become the bottleneck.

Common scaling limits and failure modes

A. Service quotas

Fargate has account and regional quotas. During a sudden spike, you may hit:

Running task quota
vCPU quota
Launch rate/throttling limits
Elastic network interface related limits
Load balancer target registration pressure
CloudWatch/logging pressure

Plan quota increases before big launches.

B. Subnet IP exhaustion

Every Fargate task needs an IP address.

Failure mode:

Desired count increases to 1,000
Subnets have only 300 free IPs
Tasks remain PENDING

Mitigation:

Use large private subnets
Spread across multiple AZs
Monitor available IPs
Plan CIDR ranges early
Use IPv6 where appropriate

C. Image pull bottleneck

Large images slow scale-out.

Mitigation:

Minimize image size
Use multi-stage builds
Remove unnecessary packages
Use regional ECR
Avoid huge startup downloads
Pin stable layers

D. Cold start and app warm-up

Even if Fargate starts the task, the app may not be ready.

Mitigation:

Readiness/health checks
Startup grace period
Preload only what is needed
Use caching carefully
Scheduled scaling before known peaks

E. Downstream bottlenecks

Scaling tasks can overload:

RDS connections
Redis
Kafka partitions
Third-party APIs
DynamoDB capacity
Internal APIs

Mitigation:

RDS Proxy
Connection pooling
Rate limiting
Queues
Backpressure
Circuit breakers
Bulkheads

F. NAT Gateway or VPC endpoint bottlenecks/design issues

If all tasks pull images, fetch secrets, or call AWS APIs through a constrained egress path, scaling can fail or become expensive.

Mitigation:

Use VPC endpoints for ECR, CloudWatch Logs, Secrets Manager, S3
Distribute across AZs
Avoid unnecessary internet egress

G. Load balancer and health check behavior

Failure mode:

Tasks start slowly
Health check is too aggressive
Tasks are killed or never receive traffic

Mitigation:

Correct health check path
Reasonable thresholds
Deregistration delay
Slow start if needed
Autoscaling based on ALB metrics

Planning pattern for spikes

1. Forecast peak demand.
2. Calculate required tasks.
3. Confirm Fargate quotas.
4. Confirm subnet IP capacity.
5. Confirm downstream capacity.
6. Optimize image size/startup time.
7. Use scheduled scaling for known events.
8. Use target tracking for unexpected growth.
9. Add queue buffering where possible.
10. Run load tests and game days.

CloudFront / WAF

      ↓

      ↓
Async work pushed to SQS
      ↓

Controls:
- Scheduled scale-out before campaign
- Target tracking on RequestCountPerTarget
- Worker scaling on QueueBacklogPerTask
- RDS Proxy for database
- VPC endpoints for AWS APIs
- Large private subnets
- Quota increases pre-approved

Model answer

Fargate scales without EC2 node management, but it is still limited by regional service quotas, subnet IP availability, task launch rate, image pull time, secrets/logging access, load balancer health checks, and downstream dependencies. For sudden spikes, I would pre-request quotas, use large subnets across AZs, optimize images, configure scheduled scaling for known events, use target tracking for dynamic demand, add queues and backpressure, use VPC endpoints, and load test the full dependency chain.

Key Takeaways

  • Fargate provides per-task kernel isolation, removing the shared-host attack surface common with EC2 node groups.
  • No SSH or host access is available; use ECS Exec or SSM for container-level debugging.
  • Fargate requires awsvpc networking mode — each task gets its own ENI and VPC identity.
  • Secrets must use valueFrom pattern; environment variables are insecure and less auditable.
  • Fargate Spot reduces cost significantly but requires your application to handle abrupt termination gracefully.
