Lambda — AWS Lambda

Category: Compute
AWS Service Type: Function as a Service (FaaS) / Serverless
Key Docs: Lambda Developer Guide, Lambda Operator Guide

Overview

Lambda runs code in response to events without requiring server provisioning or management. Functions are stateless, event-driven, and billed per invocation and duration in milliseconds. Lambda scales automatically from zero to thousands of concurrent executions and supports Node.js, Python, Java, Go, .NET, Ruby, and custom runtimes. It integrates with over 200 AWS services as event sources and is the primary building block of serverless architectures. Execution environments are ephemeral but may be reused across warm invocations via the execution context reuse pattern.

How It Works

Core Mechanics

Concept Description
Function Unit of code with a handler, runtime, memory, and timeout configuration
Execution role IAM role Lambda assumes — grants permissions to AWS APIs during execution
Event source mapping Polling-based trigger for SQS, DynamoDB Streams, Kinesis
Trigger Push-based event from S3, API Gateway, SNS, EventBridge, etc.
Concurrency Number of simultaneous executions; reserved and provisioned concurrency
Layers Shared libraries and dependencies bundled separately from function code
Destination Success or failure routing to SQS, SNS, EventBridge, or another Lambda
Execution environment Micro-VM (Firecracker) with /tmp ephemeral storage up to 10 GB
Cold start Latency to initialize a new execution environment; warmed by provisioned concurrency

1 Overly permissive execution role Scope role to minimum actions and specific resource ARNs IAM policy design

3 Secrets in environment variables Store secrets in Secrets Manager or SSM Parameter Store; load at runtime Secrets management

Top 3 Critical Mitigations

  1. Least-privilege execution roles: Lambda functions frequently access many AWS services. Scope each role to the specific actions and resource ARNs required. Use IAM Access Analyzer to identify over-permission.
  2. Never put secrets in environment variables: They appear in the console, IaC state files, and CloudTrail. Use Secrets Manager with automatic rotation and retrieve at runtime with caching (e.g., AWS Lambda Powertools or custom cache layer).
  3. Configure DLQs and failure destinations: Silent drop of failed async events is the most common cause of data loss in serverless pipelines. Every async Lambda must have a failure path to an observable, recoverable store.

1. How does Lambda’s execution model differ from a container or traditional VM runtime?

Direct answer

AWS Lambda is an event-driven, managed, short-lived compute runtime. You provide code and configuration; AWS manages the execution environment, scaling, patching of underlying infrastructure, placement, and lifecycle. A VM gives you an operating system to manage. A container gives you a packaged process runtime, usually scheduled onto a container host. Lambda gives you a function invocation model where compute is created, reused, frozen, and destroyed based on demand.

AWS Lambda runs code inside an isolated execution environment and automatically manages concurrency, scaling, and invocation handling. By default, an AWS account has a regional concurrency quota, and AWS recommends using function-level controls such as reserved concurrency for predictable scaling and throttling behavior. (AWS Documentation)

VM vs container vs Lambda

Area Traditional VM Container Lambda
Unit of deployment Full machine image Container image/process Function code/package or container image
Runtime ownership You manage OS, patches, daemons You manage image/runtime dependencies AWS manages infrastructure/runtime lifecycle
Scaling Scale instances Scale pods/tasks Scale invocations/concurrency
Startup VM boot can be slow Container start faster Cold start when new execution environment is created
Lifecycle Long-running Usually long-running, but disposable Short-lived invocation model; warm reuse possible
State Local state possible but risky Local ephemeral state possible /tmp and memory may persist between warm invocations, but not guaranteed
Security boundary VM hypervisor boundary Kernel/container boundary plus host controls Managed execution environment isolation plus IAM/resource boundaries
Operational model Server management Orchestrator management Event source + function configuration management

Real-world analogy

A VM is like renting an entire apartment. A container is like renting a room inside a managed building. Lambda is like booking a meeting room only when someone arrives. The room may already be warm and ready, or AWS may need to prepare a new room.

Important point

Lambda is not “just a small container.” It has container-like packaging options, but the operational model is different:

Event source → Lambda service → Execution environment → Handler invocation

The developer does not directly control the host, scheduling, process lifetime, or long-running daemon behavior.

2. What is cold start, when does it happen, and what are the tradeoffs of provisioned concurrency?

Direct answer

A cold start happens when Lambda needs to create a new execution environment before running your function. This includes preparing the runtime, loading your code, initializing dependencies, and running initialization code outside the handler.

It commonly happens when:

Trigger Explanation
First invocation after deployment No warm environment exists
Traffic spike Existing warm environments are insufficient
Function update New code/configuration requires new environments
Idle period Previous environments may have been reclaimed
Concurrency expansion More parallel environments are needed
VPC networking setup Additional networking setup can affect startup behavior

Cold start flow

Client/Event
   ↓
Lambda receives invoke
   ↓
No warm execution environment available
   ↓
Create execution environment
   ↓
Download/load code or image
   ↓
Initialize runtime
   ↓
Run global initialization code
   ↓
Invoke handler

Warm invocation flow

Client/Event
   ↓
Lambda receives invoke
   ↓
Warm environment available
   ↓
Invoke handler directly

Provisioned concurrency

Provisioned concurrency keeps a configured number of Lambda execution environments initialized and ready to respond. This reduces cold-start latency for latency-sensitive workloads.

Tradeoffs

Benefit Cost / Risk
Lower and predictable latency You pay for provisioned capacity even when idle
Better for APIs and user-facing flows Need capacity planning
Reduces cold-start variance Does not fix slow handler logic
Useful for Java/.NET/heavy dependency functions Overprovisioning wastes money
Can combine with autoscaling Misconfigured scaling can still throttle or cold start

Example

Suppose a fintech login API uses Lambda behind API Gateway. Most requests complete in 80 ms when warm, but cold starts take 1.5 seconds because the function loads a large Java framework and initializes database clients. During morning login spikes, users experience intermittent latency.

A good design would:

API Gateway
   ↓
Lambda alias with provisioned concurrency
   ↓
Secrets Manager / RDS Proxy / DynamoDB

Use provisioned concurrency for the production alias during business hours, keep initialization lightweight, avoid loading unnecessary dependencies, and monitor InitDuration.

3. How would you design a high-throughput event processing pipeline using Lambda and SQS?

Direct answer

I would design it as a buffered, horizontally scalable, idempotent event-processing pipeline using SQS as the shock absorber and Lambda as the stateless compute worker.

AWS Lambda can poll SQS and invoke functions with batches of messages. For standard SQS queues, Lambda can scale concurrent function instances for an event source mapping, with AWS documenting a default scaling capability up to 1,250 concurrent function instances per SQS event source mapping; newer SQS event source mapping provisioned mode can support faster scaling and higher concurrency for large workloads. (AWS Documentation)

Reference architecture

                 ┌─────────────────────┐
                 │ Event Producers      │
                 │ apps, services, logs │
                 └──────────┬──────────┘
                            │
                            v
                 ┌─────────────────────┐
                 │ SQS Standard Queue   │
                 │ durable buffer       │
                 └──────────┬──────────┘
                            │ batch poll
                            v
                 ┌─────────────────────┐
                 │ Lambda Workers       │
                 │ batch processing     │
                 └──────────┬──────────┘
                            │
             ┌──────────────┼──────────────┐
             v              v              v
       DynamoDB         OpenSearch        S3
       state/idempotency analytics        archive

Failures:
SQS → Lambda retry → visibility timeout → DLQ

Design principles

Concern Design choice
Throughput Use SQS standard queue, batching, parallel Lambda concurrency
Backpressure SQS absorbs spikes
Poison messages DLQ and partial batch failure
Duplicate processing Idempotency key
Downstream protection Reserved concurrency and max concurrency
Cost Batch size and memory tuning
Observability CloudWatch metrics, structured logs, tracing
Security Least privilege IAM, encrypted queues, VPC endpoints if needed

Key settings

Setting Recommendation
Batch size Tune based on payload and processing cost
Visibility timeout Greater than max Lambda processing time
Lambda timeout Short enough to fail fast, long enough for batch
Reserved concurrency Protect account and downstream systems
Maximum concurrency on event source mapping Prevent one queue from starving others
DLQ Required for poison messages
Partial batch response Avoid retrying successful messages
Idempotency store DynamoDB with conditional writes

Example: payment event processor

A payment platform receives card transaction events:

{
  "eventId": "evt-9a82",
  "transactionId": "txn-44192",
  "customerId": "cust-1001",
  "amount": 125.4,
  "currency": "SGD",
  "eventType": "PAYMENT_AUTHORIZED"
}

The Lambda must write to a ledger table exactly once.

Idempotency pattern:

For each message:
  1. Extract eventId
  2. Conditional write to DynamoDB:
       PK = eventId
       condition = attribute_not_exists(PK)
  3. If condition succeeds:
       process event
  4. If condition fails:
       treat as duplicate and acknowledge

This is critical because SQS and Lambda processing can be at-least-once, not exactly-once.

4. What are the security boundaries between Lambda execution environments?

Direct answer

Each Lambda execution environment is isolated from other execution environments. However, security should not rely only on the runtime boundary. In real systems, the most important boundaries are:

  1. AWS-managed compute isolation
  2. Function IAM execution role
  3. Resource policies
  4. Network boundaries
  5. Account and organizational boundaries
  6. Code/package supply-chain boundaries
  7. Tenant/application-level authorization

Important nuance

Two invocations of the same function may reuse the same warm execution environment sequentially. That means memory, /tmp, initialized clients, and global variables may persist across invocations. This is useful for performance but dangerous if developers accidentally store request-specific or tenant-specific secrets in global state.

Boundary model

AWS Account
 ├── Lambda Function A
 │    ├── Execution environment A1
 │    ├── Execution environment A2
 │    └── IAM Role A
 │
 ├── Lambda Function B
 │    ├── Execution environment B1
 │    └── IAM Role B
 │
 └── Shared resources
      ├── S3 buckets
      ├── DynamoDB tables
      ├── KMS keys
      └── Secrets Manager secrets

Security boundaries to enforce

Boundary Control
Function-to-function Separate IAM roles
Environment-to-environment Do not rely on local state for security
Tenant-to-tenant Strong app-layer authorization
Secret access Secrets Manager + IAM + KMS
Network access VPC/subnet/security group design
Deployment integrity Signed artifacts, CI/CD controls
Runtime behavior Logging, tracing, anomaly detection

Real-world failure

A multi-tenant SaaS Lambda function stores the current tenant ID in a global variable:

current_tenant = None

def handler(event, context):
    global current_tenant
    if event.get("tenantId"):
        current_tenant = event["tenantId"]

    return query_data(current_tenant)

On a warm invocation, if a malformed request does not include tenantId, the function may reuse the previous tenant’s value. That becomes a cross-tenant data leak.

Correct pattern:

def handler(event, context):
    tenant_id = validate_and_authorize(event)
    return query_data(tenant_id)

Never use warm-state reuse as a security assumption.

5. How would you debug a Lambda function that behaves differently on cold versus warm invocations?

Direct answer

I would compare the initialization path and the handler path, then look for state leakage, connection reuse issues, dependency initialization problems, time-sensitive code, and environment assumptions.

Cold vs warm difference

Cold start:
  global initialization runs
  handler runs

Warm invocation:
  global initialization does not rerun
  handler runs using existing memory/process state

Common root causes

Symptom Likely cause
First request slow Heavy init, dependency loading, VPC networking
Warm request uses old data Global variable cache
Cold request fails, warm succeeds Race condition in initialization
Warm request fails after idle time Stale DB connection
Different auth behavior Cached token not refreshed
Different tenant behavior Request state stored globally
Random failures Non-thread-safe or reused clients
File conflicts Reusing /tmp incorrectly

Debugging method

Step 1: Separate init logs from handler logs

import time

init_start = time.time()
print("INIT_START")

# Initialize clients here
# db_client = ...

print(f"INIT_DONE duration_ms={(time.time() - init_start) * 1000}")

def handler(event, context):
    start = time.time()
    print({
        "phase": "handler_start",
        "aws_request_id": context.aws_request_id,
        "remaining_ms": context.get_remaining_time_in_millis()
    })

    # process

    print({
        "phase": "handler_done",
        "duration_ms": (time.time() - start) * 1000
    })

Step 2: Check for unsafe global state

Bad:

user_id = None
permissions = None

Better:

def handler(event, context):
    user_id = extract_user(event)
    permissions = load_permissions(user_id)

Step 3: Check connection reuse

Global clients are fine, but must handle stale connections:

db = create_client()

def handler(event, context):
    try:
        return db.query(...)
    except StaleConnectionError:
        db.reconnect()
        return db.query(...)

Step 4: Force cold-start-like behavior

Ways to test:

Method Purpose
Publish new version Forces new environments
Change environment variable Triggers new environment
Use provisioned concurrency disabled/enabled comparison Compare init behavior
Load test concurrency spike Observe cold-start ratio
Add explicit init metrics Measure initialization cost

In short

“I treat cold/warm differences as a signal that the function has hidden state, unsafe initialization, stale dependencies, or bad lifecycle assumptions.”

6. Why are environment variables not a safe place for secrets in Lambda?

Direct answer

Lambda environment variables are useful for configuration, not ideal for secrets. AWS explicitly recommends using AWS Secrets Manager instead of environment variables for database credentials, API keys, and authorization tokens. Lambda environment variables can be encrypted at rest, but once available to the function runtime, they are accessible to code and to any attacker who compromises the function or overly broad IAM permissions. (AWS Documentation)

Why encryption is not enough

Environment variable encryption protects against some storage exposure risks, but it does not solve:

Risk Explanation
Runtime exposure Function code can read the variables
Log leakage Developers may accidentally print environment
IAM exposure Users with Lambda config access may retrieve values depending on permissions
No fine-grained rotation workflow Secrets rotation is harder
Blast radius Same secret may be copied across many functions
Deployment leakage Secrets may appear in IaC templates, build logs, Terraform state
Memory compromise If code execution is achieved, secrets are available

Bad pattern

DB_PASSWORD=SuperSecretPassword123
STRIPE_API_KEY=sk_live_xxxxxx
JWT_SIGNING_KEY=xxxxx

Better pattern

DB_SECRET_ARN=arn:aws:secretsmanager:ap-southeast-1:123456789012:secret:prod/db

At runtime:

Lambda execution role → Secrets Manager → KMS decrypt → secret value

Secure design

Lambda
  environment variables:
    DB_SECRET_ARN
    CONFIG_TABLE_NAME

Lambda IAM role:
  allow secretsmanager:GetSecretValue only for specific secret ARN
  allow kms:Decrypt only for specific KMS key with conditions

Secrets Manager:
  stores actual credential
  rotation enabled
  access logged via CloudTrail

Real-world attack scenario

A developer adds debug logging:

import os
print(os.environ)

Now production API keys are in CloudWatch Logs. Any principal with logs read access can retrieve them.

Mitigations

Control Why it helps
Secrets Manager / SSM Parameter Store Central secret management
KMS customer-managed keys Controlled decrypt access
IAM least privilege Limits secret access
Secret rotation Reduces lifetime of compromise
Log redaction Prevents accidental leakage
IaC scanning Prevents hardcoded secrets
CI/CD secret detection Stops exposure before deploy
Use env vars for secret references only Avoids direct secret leakage

7. How does Lambda handle retry behavior for synchronous versus asynchronous invocations?

Direct answer

For synchronous invocations, Lambda does not automatically retry the function on behalf of the caller. The caller receives the response or error and must decide whether to retry.

For asynchronous invocations, Lambda queues the event and returns success to the caller after accepting it. Then Lambda separately delivers the event to the function. By default, asynchronous function errors are retried twice, and retry behavior can be configured with max retry attempts and max event age. (AWS Documentation)

Invocation types

Invocation type Example Retry owner
Synchronous API Gateway, direct SDK RequestResponse Caller/client
Asynchronous S3 event, EventBridge, SNS Lambda service
Poll-based SQS, Kinesis, DynamoDB Streams Event source mapping/service integration

Synchronous example

Client → API Gateway → Lambda → Response/Error

If Lambda times out, the client gets an error. Lambda itself does not keep retrying automatically.

Asynchronous example

S3/EventBridge/SNS → Lambda async queue → Lambda function

If the function fails:

Attempt 1 → fail
Wait
Attempt 2 → fail
Wait
Attempt 3 → fail
Send to destination / DLQ / discard based on config

SQS retry model

For SQS, the message remains in the queue until successfully deleted. If Lambda fails or times out, the message becomes visible again after the visibility timeout and can be retried.

SQS message received
   ↓
Lambda invoked
   ↓
Success?
   ├── Yes → delete message
   └── No → visibility timeout expires → message retried

Security and reliability risk

Retries can create duplicate side effects.

Example:

Charge customer card
Function times out before recording success
SQS retries message
Customer is charged again

Mitigation:

Idempotency key = transactionId
Before charging:
  check idempotency table
After charging:
  persist result atomically

8. How would you design a Lambda-based workflow that processes 100k events per second?

Direct answer

At 100k events/sec, I would not design “one Lambda per event.” I would design a partitioned, batched, backpressure-aware event processing platform. Lambda can be part of the design, but the architecture must control concurrency, batching, downstream capacity, retries, idempotency, and cost.

Important sizing logic

If each Lambda invocation processes 100 messages per batch:

100,000 events/sec ÷ 100 messages/batch = 1,000 Lambda invokes/sec

If each invocation takes 1 second:

Required concurrency ≈ 1,000

If each invocation takes 5 seconds:

Required concurrency ≈ 5,000

So the design must calculate:

concurrency = arrival_rate × average_duration / batch_size

Architecture option A: SQS + Lambda

Producers
   ↓
SQS Standard Queue shards/logical queues
   ↓
Lambda event source mappings
   ↓
Processing
   ↓
DynamoDB / S3 / OpenSearch / downstream services
   ↓
DLQ + replay pipeline

For high scale, I may use multiple queues by partition key:

events-queue-00
events-queue-01
events-queue-02
...
events-queue-N

Architecture option B: Kinesis/MSK + Lambda

For ordered high-throughput streaming:

Producers
   ↓
Kinesis Data Streams / MSK
   ↓
Partition/shard consumers
   ↓
Lambda batch processors
   ↓
S3 data lake / DynamoDB / analytics

Use this when ordering, replay, shard-level processing, or streaming analytics matter.

Design checklist for 100k/sec

Area Design
Ingestion SQS standard, Kinesis, MSK, or EventBridge Pipes depending on semantics
Partitioning Partition by tenant/customer/entity
Batching Process many events per invocation
Idempotency Required
Backpressure Queue depth, age-of-oldest-message alarms
Downstream protection Reserved concurrency, max concurrency
Failure isolation Per-tenant queues or partitions for noisy tenants
Replay DLQ + redrive + event archive
Observability Metrics by tenant, queue, partition, error class
Security IAM least privilege, encryption, VPC endpoints if needed
Cost Memory tuning, batch size tuning, avoid excessive per-event calls

Example: security telemetry pipeline

A company processes endpoint telemetry from 2 million devices.

Endpoint Agents
   ↓
API Gateway / ALB / Kinesis Producer
   ↓
Kinesis Data Streams
   ↓
Lambda normalizer
   ↓
S3 raw archive
   ↓
OpenSearch detection index
   ↓
Security alerts

Failure mode to avoid

Bad design:

100k events/sec
   ↓
100k Lambda invokes/sec
   ↓
Each Lambda writes individually to RDS
   ↓
RDS connection exhaustion
   ↓
Retries
   ↓
Retry storm
   ↓
Total outage

Use DynamoDB/S3/Kinesis/OpenSearch bulk APIs
Use RDS Proxy if relational DB is unavoidable
Limit concurrency
Use DLQ and replay

9. What are the failure modes specific to Lambda functions inside a VPC?

Direct answer

Putting Lambda inside a VPC changes networking behavior. It is required when accessing private resources such as RDS, ElastiCache, or internal services, but it introduces risks around subnet capacity, routing, NAT, DNS, security groups, NACLs, VPC endpoints, and downstream connection exhaustion.

Common VPC-specific failure modes

Failure mode Cause Symptom
No internet access Lambda in private subnet without NAT Cannot call public APIs
NAT gateway bottleneck/cost All egress through NAT Latency, cost spike
Missing VPC endpoint Calls to AWS APIs go through NAT or fail Secrets/S3/DynamoDB errors
Security group misconfig Egress/ingress blocked Connection timeout
NACL misconfig Stateless network ACL blocks ephemeral ports Intermittent failures
DNS disabled/misconfigured VPC DNS support issue Cannot resolve hosts
Subnet IP exhaustion Too many ENIs/resources needing IPs Scaling failures
RDS connection exhaustion High Lambda concurrency DB max connections exceeded
Cross-AZ routing issues Poor subnet/resource placement Latency/failure
Cold-start/network setup latency VPC networking path overhead Higher init duration

Architecture issue

Lambda in private subnet
   ↓
Needs to call Secrets Manager
   ↓
No NAT gateway
No Secrets Manager VPC endpoint
   ↓
Function times out during secret retrieval

Better VPC design

Lambda private subnet
   ├── RDS via security group
   ├── Secrets Manager via interface VPC endpoint
   ├── S3 via gateway/interface endpoint
   ├── DynamoDB via gateway endpoint
   └── Internet egress via NAT only if needed

RDS-specific failure

Lambda can scale very fast. RDS cannot accept unlimited connections.

Bad:

1,000 concurrent Lambdas × 1 DB connection each = 1,000 DB connections

If RDS supports only 300 connections, the application fails.

Controls:

Control Purpose
RDS Proxy Connection pooling
Reserved concurrency Cap DB pressure
Batch processing Reduce calls
DynamoDB for high scale Avoid relational bottleneck
Circuit breakers Stop retry storms
Short timeouts Fail fast
Connection reuse Reduce overhead

  4. AWS Organizations SCPs

Terraform rule examples

Reject Lambda if:

Example IaC module pattern

Instead of allowing teams to create raw Lambda resources, provide a secure module:

module "secure_lambda" {
  source = "git::ssh://git.example.com/security/secure-lambda-module"

  function_name = "payment-processor"
  runtime       = "python3.12"
  handler       = "app.handler"

  source_path = "./src"

  environment = {
    CONFIG_TABLE = "payment-config"
    SECRET_ARN   = aws_secretsmanager_secret.db.arn
  }

  allowed_secret_arns = [
    aws_secretsmanager_secret.db.arn
  ]

  sqs_event_source = {
    queue_arn           = aws_sqs_queue.payment_events.arn
    batch_size          = 100
    max_concurrency     = 500
    partial_batch_failures = true
  }

  dlq_enabled = true

  tags = {
    Owner              = "payments-team"
    DataClassification = "confidential"
    Environment        = "prod"
  }
}

Summary

Lambda is powerful because it removes server management and gives fast event-driven scaling, but the security and reliability model changes. The biggest mistakes are treating Lambda like a normal server, assuming warm state is reliable, putting secrets in environment variables, ignoring retries, and allowing unbounded concurrency into fragile downstream systems.

For a production-grade Lambda system, I would focus on:

Stateless handler design
Idempotent processing
Controlled concurrency
Strong IAM boundaries
Secrets Manager instead of env secrets
DLQ and replay strategy
VPC endpoints and egress control
IaC guardrails
Observability from day one

The senior-level answer is not “Lambda scales automatically.” The senior answer is:

“Lambda scales automatically, but my job is to make sure it scales safely, predictably, securely, and without destroying downstream systems.”

Key Takeaways

  • Lambda is stateless by design; state must live in external stores like DynamoDB, S3, or ElastiCache.
  • Execution roles are the key security boundary — treat them with the same rigor as IAM user policies.
  • Cold starts are a real operational concern; provisioned concurrency trades cost for latency.
  • Async invocations require explicit failure handling; silent drops are the default without a DLQ.
  • Serverless reduces operational overhead but requires careful design around concurrency, timeout, and retry behavior.
