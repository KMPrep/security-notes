ECS + EC2 — AWS-only team, no Kubernetes expertise needed, cost-optimized long-running services at scale with reserved or Spot EC2.

ECS + Fargate — AWS-only team, batch jobs, event-driven workers, or unpredictable workloads where you want zero node management. Easiest entry point.

EKS + EC2 — Multi-cloud targets, Kubernetes ecosystem (Helm, ArgoCD, Istio, Kyverno, custom controllers), large teams with k8s expertise, complex scheduling requirements.

EKS + Fargate — Kubernetes API portability with serverless convenience. Good for dev/test namespaces. Limited (no DaemonSets, no privileged pods, no persistent volumes via EBS).

Key Limitations to Know

ECS

  • No Kubernetes API — custom controllers, CRDs, and Helm charts do not apply.
  • Service Connect is less flexible than a full service mesh.
  • Spot interruption for EC2 launch type requires ASG lifecycle hooks.

EKS

  • Version upgrade cadence is mandatory — AWS drops support for old versions.
  • aws-auth ConfigMap (legacy) is a single point of misconfiguration; prefer access entries.
  • VPC CNI IP exhaustion is a scaling bottleneck at large pod counts.
  • DaemonSet pods cannot run on Fargate profiles.

Fargate (ECS or EKS)

  • No DaemonSets (EKS Fargate).
  • No privileged containers.
  • No EBS persistent volumes from Fargate pods (EKS).
  • Cold start is slower than a pre-warmed EC2 node.
  • Fargate Spot can be reclaimed with 120-second notice — apps must handle SIGTERM.
  • Higher per-unit cost than equivalent EC2 for steady, predictable workloads.

1. 200 microservices on EKS and 20 services on ECS: consolidate or keep separate?

Direct answer

I would not consolidate just because “one platform is cleaner.” I would consolidate only if the operational, security, cost, and developer-experience benefits outweigh migration risk. EKS and ECS can both be good production platforms, but they optimize for different levels of control.

EKS is usually better when the organization needs Kubernetes-native extensibility, multi-cloud portability, custom controllers, service mesh, advanced scheduling, CRDs, admission control, and large platform engineering capabilities. ECS is usually better when teams want simpler AWS-native container orchestration with less Kubernetes operational overhead.

Decision framework

Factor Consolidate to EKS Consolidate to ECS Keep separate
Platform complexity Team already has strong Kubernetes platform maturity Team wants simpler AWS-native operations Different teams have different maturity
Workload count 200+ services already on EKS Small number of simple services Migration cost not justified
Compliance Need centralized Kubernetes admission control/policy Need simpler IAM/task-level model Different regulatory boundaries
Developer experience Teams use Helm, Kustomize, Argo CD, CRDs Teams use ECS services/task definitions Existing pipelines are stable
Portability Avoid AWS-specific orchestration lock-in AWS-native is acceptable Mixed strategy is intentional
Operations Platform team can run clusters well Smaller ops team Each platform has clear ownership
Security controls Need Kubernetes-native policy enforcement Need simpler service/task isolation Separate blast-radius domains
Cost Better bin-packing on EC2 node groups Fargate/ECS simplicity reduces ops cost Migration cost exceeds savings

What would drive consolidation?

I would consolidate if I see duplicated operational effort:

Two CI/CD systems
Two observability stacks
Two policy engines
Two incident playbooks
Two secrets patterns
Two IAM models
Two patching models
Two runtime baselines

If both teams are solving the same problems differently, consolidation reduces cognitive load and improves governance.

What would drive keeping them separate?

I would keep them separate if the ECS workloads are simple, stable, and low-risk, while the EKS estate supports complex platform requirements.

Example:

EKS:
  200 microservices
  service mesh
  Kubernetes network policies
  custom autoscalers
  admission control
  internal developer platform

ECS:
  20 simple backend services
  no Kubernetes dependency
  stable deployment model
  lower operational overhead

In that case, forcing ECS workloads into EKS may create more risk than value.

Senior answer

The decision is not “EKS is better than ECS” or “ECS is simpler than EKS.” The right answer is:

“I would consolidate control planes, policy, observability, identity, and deployment standards first. I would consolidate runtimes only when there is a clear business and risk reduction case.”

2. Why can’t you run DaemonSet pods on EKS Fargate, and what workloads does that rule out?

Direct answer

DaemonSets require a pod to run on every Kubernetes node. EKS Fargate abstracts away the worker nodes, so you do not manage or schedule pods onto stable EC2 nodes. AWS explicitly states that DaemonSets are not supported on EKS Fargate, and if a workload requires a daemon, AWS recommends reconfiguring it as a sidecar where possible. Privileged containers are also not supported on EKS Fargate. (AWS Documentation)

Why DaemonSets do not fit Fargate

DaemonSet model:

Kubernetes Node 1 → daemon pod
Kubernetes Node 2 → daemon pod
Kubernetes Node 3 → daemon pod

EKS Fargate model:

Pod → scheduled onto AWS-managed Fargate runtime
Pod → no customer-managed node lifecycle
Pod → no host-level access

DaemonSets assume there is a durable node surface to attach to. Fargate removes that surface from the customer.

Workloads ruled out or complicated on EKS Fargate

Workload type Why it is a problem on Fargate
Node-level log collectors Commonly deployed as DaemonSets
Host-based security agents Need host visibility
eBPF agents Need kernel-level access
Node exporters Need node metrics access
CNI plugins Need node networking integration
CSI node plugins Often require node-level privileges
Service mesh node agents May depend on DaemonSet behavior
Privileged containers Not supported on EKS Fargate
HostPath volume workloads Depend on node filesystem
Runtime security sensors Often need kernel/container runtime visibility

AWS Prescriptive Guidance also notes that Fargate nodes are not supported by Container Insights in the same way as EC2 nodes because the nodes are managed by AWS and do not support DaemonSets. (AWS Documentation)

Security implication

This is both a benefit and a limitation.

Benefit:

No host access
No privileged containers
Smaller host-management responsibility
Reduced node-level attack surface

Limitation:

Less ability to run deep runtime security
Less custom networking/security instrumentation
Less node-level observability

Example

If a security team mandates a DaemonSet-based runtime detection agent that uses eBPF to monitor syscalls, EKS Fargate is probably not suitable. I would use EKS managed EC2 node groups, Bottlerocket nodes, or another supported host model.

3. How does IRSA differ from the ECS task role mechanism?

Direct answer

Both IRSA and ECS task roles solve the same high-level problem: giving a workload AWS permissions without embedding static AWS keys. But the mechanism is different.

IRSA maps a Kubernetes service account to an IAM role using OIDC federation. ECS task roles map an ECS task definition/task to an IAM role, and ECS delivers credentials to containers through the ECS credentials mechanism. AWS describes IRSA as a way for pod containers to use AWS SDKs/CLI with IAM permissions, similar in goal to EC2 instance profiles. (AWS Documentation) AWS describes ECS task IAM roles as a way to associate IAM permissions with containers regardless of compute option, with ECS delivering credentials through a defined interface. (AWS Documentation)

Comparison

Area EKS IRSA ECS task role
Orchestrator Kubernetes/EKS ECS
Identity unit Kubernetes service account ECS task
Trust mechanism OIDC federation to STS ECS service/task credential delivery
Configuration object ServiceAccount annotation + IAM role trust policy Task definition taskRoleArn
Credential source Projected service account token exchanged via STS ECS credentials endpoint
Security concern Service account reuse, trust policy mis-scoping Task role over-permission, EC2 metadata exposure on EC2 launch type
Portability Kubernetes-native pattern AWS ECS-specific pattern

IRSA flow

Pod
 └── uses Kubernetes service account
      ↓
Projected service account token
      ↓
AWS STS AssumeRoleWithWebIdentity
      ↓
Temporary IAM credentials
      ↓
AWS API calls

ECS task role flow

ECS task
 └── taskRoleArn in task definition
      ↓
ECS agent / Fargate credential delivery
      ↓
Temporary IAM credentials exposed to container
      ↓
AWS API calls

Important security nuance

For ECS on Fargate, AWS states each task has its own isolation boundary and does not share the underlying kernel, CPU, memory, or ENI with another task. For ECS on EC2, AWS warns that containers are not a security boundary and that tasks on EC2 do not have the same task isolation as Fargate. (AWS Documentation)

Misconfiguration examples

Bad IRSA trust policy

Trusts too broadly:

{
  "Condition": {
    "StringLike": {
      "oidc.eks.region.amazonaws.com/id/CLUSTER:sub": "system:serviceaccount:*:*"
    }
  }
}

This may allow many service accounts to assume the role.

Better:

{
  "Condition": {
    "StringEquals": {
      "oidc.eks.region.amazonaws.com/id/CLUSTER:sub": "system:serviceaccount:payments:payment-api"
    }
  }
}

Bad ECS task role

One task role shared by all services:

ecsTaskExecutionRole/payment-platform-admin
  s3:*
  dynamodb:*
  secretsmanager:*

Better:

payment-api-task-role
  dynamodb:GetItem only on payment-config table
  secretsmanager:GetSecretValue only on payment-api secret

4. NoSQL write-heavy service needs consistent sub-10ms latency. Fargate or EC2 node groups?

Direct answer

For consistent sub-10ms latency, especially for write-heavy workloads, I would usually prefer EC2-backed node groups over Fargate, assuming the team can operate them well.

Fargate is excellent for operational simplicity and isolation, but EC2 node groups give more control over instance type, CPU architecture, networking, placement, kernel tuning, local caching, daemon-based observability, and predictable capacity.

Why EC2 node groups are usually better here

Requirement EC2 node groups Fargate
Consistent low latency More control Less host control
High write throughput Tune instance/network/storage Limited tuning
Daemon-based monitoring Supported DaemonSets not supported on EKS Fargate
Dedicated nodes Easy with taints/node groups Task/pod abstraction
Placement control More control Less direct control
Kernel/sysctl tuning More options Limited
Cost at sustained load Often better with reserved/savings plans Can be more expensive for steady high load
Operational simplicity More responsibility Easier

Important distinction

If the NoSQL service is managed, such as DynamoDB, then the application compute platform matters less than data access patterns, partition key design, SDK tuning, connection reuse, and regional placement.

If the NoSQL service is self-hosted, such as Cassandra, ScyllaDB, or Aerospike, I would strongly avoid Fargate and use EC2 node groups or dedicated EC2 because the database needs stable storage, predictable networking, placement control, and operational visibility.

Example

For a write-heavy fraud telemetry service:

API workers
   ↓
NoSQL write path
   ↓
P99 must be < 10 ms

I would use:

EKS managed node group
  instance family optimized for network/CPU
  dedicated node pool
  topology spread constraints
  pod anti-affinity
  reserved capacity
  tuned client connection pools
  local metrics agents
  strict autoscaling controls

For Fargate, I would use it only if the service is stateless and the database is managed externally, and if performance testing proves p99 is stable.

Answer

“For sub-10ms consistent write latency, I do not want a black-box compute runtime unless testing proves it. I want predictable capacity, placement, observability, and network behavior. That points me toward EC2 node groups for the latency-critical path.”

5. How would you enforce that no container in ECS or EKS runs as root in production?

Direct answer

I would enforce it at multiple layers: image build, IaC, admission control, runtime detection, and exception management. The control must prevent root containers before deployment, not just alert afterward.

For EKS

Use Kubernetes security context and admission control.

Required pod/container settings:

securityContext:
  runAsNonRoot: true
  runAsUser: 10001
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop:
      - ALL

Enforce using:

Pod Security Admission
OPA Gatekeeper / Kyverno
CI policy checks
Helm chart validation
Admission webhook
Runtime detection

Example Kyverno-style policy objective:

Deny pod if:
  spec.securityContext.runAsNonRoot != true
  container.securityContext.runAsUser == 0
  allowPrivilegeEscalation != false

For ECS

Use task definition controls.

Example ECS container definition:

{
  "name": "payment-api",
  "image": "123456789012.dkr.ecr.ap-southeast-1.amazonaws.com/payment-api:1.0.0",
  "user": "10001",
  "readonlyRootFilesystem": true,
  "linuxParameters": {
    "capabilities": {
      "drop": ["ALL"]
    }
  }
}

Enforce using:

Terraform/CDK policy-as-code
CI checks against task definitions
ECR image scanning
AWS Config custom rule
Deployment pipeline deny rules
Exception approval workflow

Image-level control

Dockerfile should not default to root:

FROM python:3.12-slim

RUN groupadd -g 10001 app && useradd -u 10001 -g app app
WORKDIR /app
COPY . /app
RUN chown -R app:app /app

USER 10001

CMD ["python", "app.py"]

Attack scenario

A container runs as root and has a vulnerable image-processing library. An attacker uploads a malicious file, gets remote code execution, and lands inside the container as root.

Attack path:

Upload malicious file
   ↓
Exploit parser vulnerability
   ↓
Code execution inside container
   ↓
Root inside container
   ↓
Attempt privilege escalation / container escape
   ↓
Access service account token / task credentials
   ↓
Access cloud resources

Controls

Layer Control
Build Dockerfile USER required
Registry Reject images with root default user
IaC Deny ECS task definitions without user
EKS admission Deny pods without runAsNonRoot
Runtime Alert on UID 0 process
Linux hardening Drop capabilities, no privilege escalation
Filesystem Read-only root filesystem
Secrets Least-privileged workload identity

In short

“I would not rely on developer discipline. I would make root containers technically undeployable in production unless an approved exception exists.”

6. What is VPC CNI IP exhaustion, and how do you architect around it at 5000+ pods?

Direct answer

In EKS with the Amazon VPC CNI, pods receive IP addresses from the VPC. IP exhaustion happens when subnets or node ENI/IP limits cannot supply enough pod IPs. At 5000+ pods, this becomes a real architecture constraint.

The Amazon VPC CNI assigns VPC IP addresses to Kubernetes pods, which is powerful because pods are first-class VPC network participants, but it means pod density depends on subnet CIDR capacity, ENI limits, and IP allocation strategy. AWS recommends features such as prefix delegation to increase pod density and reduce IP management pressure. (Amazon)

What happens during IP exhaustion?

Symptoms:

Pods stuck in Pending
FailedCreatePodSandBox
CNI failed to assign IP address
Node has CPU/memory but cannot accept more pods
Subnet free IPs near zero
Autoscaler adds nodes but subnet still lacks IPs

Why it happens

Each pod needs an IP
   ↓
Node has limited ENIs
   ↓
Each ENI has limited secondary IPs or prefixes
   ↓
Subnet has finite IP capacity
   ↓
At scale, pod scheduling fails

5000+ pod architecture

1. Use large enough VPC/subnet CIDRs

Bad:

/24 private subnet = about 251 usable IPs

For thousands of pods, this is too small.

Better:

Large private subnets per AZ
Separate subnet tiers:
  node subnets
  pod subnets
  database subnets
  endpoint subnets

2. Use prefix delegation

Prefix delegation allows assigning IP prefixes to ENIs rather than individual secondary IPs, increasing pod density and improving allocation efficiency.

3. Use custom networking where appropriate

Separate pod IP space from node IP space:

7. How would you handle secrets rotation such that running containers get updated credentials without restart?

Direct answer

The best answer is: do not inject long-lived secrets only as environment variables. Environment variables are loaded at process start and do not automatically update inside running containers. For no-restart rotation, the application must fetch secrets dynamically, watch a mounted secret file, or use a sidecar/agent that refreshes credentials and signals the app.

AWS Secrets Manager supports automatic rotation, where the secret and the backing database/service credential are updated. (AWS Documentation) The container still needs a runtime refresh pattern to use the new value.

Patterns

Pattern Restart needed? Notes
Env var injection Yes Simple but stale until restart
App fetches secret on demand No Best for many services
App caches secret with TTL No Good balance
Mounted secret file + watcher No if app reloads file Good for Kubernetes
Sidecar secret agent No if app integrates More moving parts
Dual credential rotation No downtime Best for databases
Short-lived IAM auth No static secret Preferred where possible

Recommended application pattern

App starts
   ↓
Fetch secret from Secrets Manager
   ↓
Cache in memory for short TTL
   ↓
Before using connection:
      if TTL expired, refresh secret
   ↓
If auth fails:
      refresh secret immediately
      recreate connection pool

Pseudo-code:

class RotatingSecretProvider:
    def __init__(self, secret_id, ttl_seconds=300):
        self.secret_id = secret_id
        self.ttl_seconds = ttl_seconds
        self.cached = None
        self.expires_at = 0

    def get_secret(self):
        now = time.time()
        if now > self.expires_at:
            self.cached = fetch_from_secrets_manager(self.secret_id)
            self.expires_at = now + self.ttl_seconds
        return self.cached

Database credential rotation without restart

Use dual-user rotation:

App currently uses user_a
   ↓
Rotate user_b password
   ↓
Switch secret to user_b
   ↓
Running containers refresh secret by TTL
   ↓
Existing user_a connections drain
   ↓
Rotate user_a later

This avoids breaking all active connections at once.

EKS option

Use Secrets Store CSI Driver or external secrets pattern to mount/update secrets, but the app must reload the file or refresh its connection pool. Mounted secret updates alone do not guarantee the application uses the new credential.

ECS option

For ECS, prefer app-level fetching from Secrets Manager or a sidecar/agent. If secrets are injected into task environment at startup, the running task will not magically get updated credentials.

Senior answer

“Secret rotation is not complete when the secret value changes in Secrets Manager. It is complete only when running workloads safely pick up the new credential and old credentials are retired without outage.”

8. A Fargate task exits with OOMKilled. Diagnostic steps.

Direct answer

OOMKilled means the container exceeded its memory limit and the runtime killed it. I would debug it as a resource sizing, memory behavior, traffic pattern, and deployment regression problem.

Diagnostic flow

Task stopped
   ↓
Check stopped reason / container exit reason
   ↓
Confirm OOMKilled
   ↓
Compare task memory vs container memory
   ↓
Check CloudWatch memory metrics/logs
   ↓
Correlate with deployment, traffic, payload size
   ↓
Reproduce with load test
   ↓
Fix memory leak, limit, batch size, or concurrency

“I would not just increase memory and walk away. I would treat OOMKilled as either a sizing miss, an unbounded memory pattern, a bad traffic assumption, or a deployment regression.”

9. Compare blast radius: compromised Fargate task vs compromised pod on shared EKS EC2 node

Direct answer

A compromised Fargate task generally has a smaller infrastructure blast radius because each Fargate task has its own isolation boundary and does not share the underlying kernel, CPU, memory, or ENI with another task. AWS explicitly states this for Fargate. (AWS Documentation)

A compromised pod on a shared EKS EC2 node may have a larger blast radius because multiple pods share the same node kernel, kubelet, container runtime, node IAM role, local network context, and sometimes host-mounted resources. Kubernetes isolation is strong when configured well, but containers are not equivalent to VM boundaries.

Comparison

Area Compromised Fargate task Compromised EKS pod on EC2 node
Kernel sharing Does not share kernel with another Fargate task Shares node kernel with other pods
Network interface Task has own ENI Pods share node networking substrate
Host access No customer host access Risk if privileged/hostPath/hostNetwork
Credential risk Task role credentials Service account token, possible node role exposure if misconfigured
Lateral movement Mostly app/network/IAM level App/network/IAM plus node-level paths
Runtime agents Less deep visibility More visibility possible with agents
Escape impact Lower cross-task host impact Higher if node escape succeeds
Operational control Less host tuning More control but more responsibility

10. Multi-cluster EKS active/passive failover with shared state

Direct answer

I would design this as a multi-region, active/passive EKS architecture with stateless services deployable in both clusters, state replicated through managed data services, DNS or global load-balancing failover, GitOps-based configuration consistency, and tested runbooks.

The hardest part is not starting a second EKS cluster. The hardest part is state consistency, identity, traffic failover, data replication lag, and safe promotion of the passive region.

Reference architecture

Final summary

The senior-level answer is:

“EKS gives me control and ecosystem power; ECS gives me AWS-native simplicity; Fargate gives me operational simplicity and stronger per-task isolation but less host-level control. For high-scale production, I would choose based on workload characteristics, latency, observability needs, security boundaries, and operational maturity — not based on service popularity.”

For this specific environment:

200 microservices on EKS:
  likely keep EKS as main platform if Kubernetes maturity exists

20 services on ECS:
  keep separate if simple/stable and low operational burden
  consolidate only if governance/ops duplication is painful

Fargate:
  good for stateless, moderate-complexity, isolation-friendly workloads

EC2 node groups:
  better for latency-sensitive, daemon-heavy, custom-networking, high-control workloads

Key Takeaways

  • Fargate is a launch type / runtime — not a separate orchestrator. It works under both ECS and EKS.
  • ECS is simpler and AWS-native; EKS gives Kubernetes portability at the cost of operational complexity.
  • Use Fargate when you want zero node management; use EC2 when you need cost efficiency, DaemonSets, or specific node configuration.
  • IAM identity for workloads: task role in ECS, IRSA or Pod Identity in EKS — both avoid node-level credential sharing.
  • For production: enforce awsvpc networking, secret injection via valueFrom/CSI, image scanning, and runtime threat detection (GuardDuty) regardless of which orchestrator you choose.
