DynamoDB — Amazon DynamoDB

Category: Database
AWS Service Type: Managed NoSQL Key-Value and Document Database
Key Docs: DynamoDB Developer Guide

Overview

DynamoDB is AWS's fully managed NoSQL database for high-scale, low-latency key-value and document workloads. It offers single-digit millisecond latency at virtually any scale, automatic multi-AZ replication, on-demand or provisioned capacity, point-in-time recovery, streams, TTL, and Global Tables for multi-region replication. Data modeling is access-pattern first: table design starts from query patterns, partition keys, sort keys, and secondary indexes rather than normalized schema. DynamoDB is widely used for session stores, shopping carts, event metadata, feature flags, rate limiting, and serverless application backends.

Top 3 Critical Mitigations

  1. Model for access patterns, not for normalization: Most DynamoDB problems are schema design problems. If the partition key is wrong, no amount of tuning fixes it cleanly.
  2. Protect writes with conditions: Conditional expressions prevent accidental overwrite of counters, state machines, and ownership fields.
  3. Enable PITR everywhere important: DynamoDB is resilient, but logical deletion and application bugs still happen. PITR turns a catastrophic table mistake into a recoverable event.

1. How do partition keys determine scale and performance in DynamoDB?

Direct answer

In DynamoDB, the partition key is the main scalability lever. It determines how items are distributed across logical and physical partitions. If your partition key spreads traffic evenly, DynamoDB can scale very well. If many reads or writes hit the same partition key, you create a hot partition, which causes throttling and high latency.

AWS states that the partition key portion of the primary key determines the logical partitions where data is stored, and poor distribution can create hot partitions that use capacity inefficiently and cause throttling. AWS also documents that each physical partition is designed to deliver up to 3,000 read capacity units per second and 1,000 write capacity units per second. (AWS Documentation)

Simple mental model

Think of DynamoDB as many storage workers.

Good key design:

PK=user#001 → partition A
PK=user#002 → partition B
PK=user#003 → partition C
PK=user#004 → partition D

Bad key design:

PK=2026-05-12 → partition A
PK=2026-05-12 → partition A
PK=2026-05-12 → partition A
PK=2026-05-12 → partition A

The second design sends too much traffic to one logical key.

Example: bad partition key

For an order system:

PK = orderDate
SK = orderId

If Black Friday traffic sends all writes to:

PK = 2026-11-27

then one partition key becomes extremely hot.

Better partition key

PK = customerId
SK = orderDate#orderId

or for very high write volume:

PK = customerId#shardNumber
SK = orderDate#orderId

What makes a good partition key?

Good property Why it matters
High cardinality Many unique values spread load
Access-pattern aligned Supports efficient Query
Even traffic distribution Avoids hot keys
Bounded item collection size Avoids huge partitions
Avoids time-only keys Time-based keys often become hot
Tenant-aware but not tenant-hot Important for SaaS

Answer

“In DynamoDB, you do not design tables first and queries later. You design partition keys from access patterns. The partition key determines whether DynamoDB behaves like a massively scalable distributed database or an expensive single hot shard.”

2. What is the difference between a GSI and an LSI?

Direct answer

A Global Secondary Index, or GSI, lets you query the table using a different partition key and optional sort key from the base table. It is stored in its own partition space and scales separately from the base table.

A Local Secondary Index, or LSI, uses the same partition key as the base table but a different sort key. It must be created when the table is created. AWS describes GSIs as indexes with partition and sort keys that can differ from the base table, while LSIs have the same partition key as the base table but a different sort key. (AWS Documentation)

Comparison

Feature GSI LSI
Partition key Can be different from base table Same as base table
Sort key Can be different Different from base table
Created after table creation? Yes No, must be created with table
Capacity Separate capacity in provisioned mode Shares table capacity
Storage Separate index partition space Local to base table partition
Consistency Eventually consistent reads only Can support strongly consistent reads
Scale Scales independently Bound to base partition key
Use case New access pattern across table Alternative ordering within same partition key

Example base table

Table: Orders

PK = CUSTOMER#123
SK = ORDER#2026-05-12#9001

LSI example

Query same customer’s orders by amount:

LSI:
PK = CUSTOMER#123
LSI_SK = AMOUNT#500

This is still within the same customer partition.

GSI example

Query all orders by status:

GSI1:
GSI1PK = STATUS#PENDING
GSI1SK = CREATED_AT#2026-05-12T10:00:00

This can query across all customers.

Important GSI risk

If your GSI key is low-cardinality, it can become hot.

Bad:

GSI1PK = STATUS#PENDING

If millions of items are pending, this becomes a hot GSI partition.

Better:

GSI1PK = STATUS#PENDING#SHARD#07
GSI1SK = CREATED_AT#...

Answer

“An LSI gives me another sort order within the same partition key. A GSI gives me a completely different access path, but it also creates another scaling and hot-partition surface.”

3. How would you design a DynamoDB schema for a multi-tenant SaaS application?

Direct answer

I would start with access patterns, tenant isolation requirements, scale profile, and noisy-neighbor risk. For most SaaS workloads, I would use a single-table design or a small number of tables with tenant-aware partition keys, careful item collections, and IAM/application-level authorization.

“For multi-tenant DynamoDB, every key and index must be tenant-aware unless there is a deliberate cross-tenant administrative access pattern. Otherwise, the GSI becomes a data-leakage and noisy-neighbor surface.”

4. Common causes of hot partitions and mitigations

Direct answer

Hot partitions happen when too much read or write traffic targets the same partition key or a small set of keys. They are usually caused by poor key distribution, time-based keys, celebrity tenants, low-cardinality attributes, or overloaded GSIs.

AWS notes that DynamoDB uses burst and adaptive capacity to help with spikes and uneven access patterns, but adaptive capacity is not a replacement for good key design. (AWS Documentation)

Common causes

Cause Bad example Why it is hot
Time-based key PK=2026-05-12 All current writes hit same key
Low-cardinality key PK=STATUS#OPEN Few possible values
Tenant-only key PK=TENANT#bigTenant Large tenant dominates
Celebrity entity PK=PRODUCT#iphone One item/entity gets huge traffic
Global counter PK=COUNTER#orders Every update same item
GSI low-cardinality key GSI1PK=REGION#SG Too many items per key
Monotonic sort pattern SK=timestamp under one hot PK All writes concentrated
Scan-heavy workload Full table/index scans Burns broad capacity inefficiently

Mitigations

Mitigation How it helps
High-cardinality partition key Spreads load
Write sharding Splits one logical key into N physical keys
Read sharding Distributes very hot reads
Random suffixes Spreads writes
Calculated suffixes Allows deterministic lookup
Time bucket + shard Avoids one time bucket getting too hot
DAX/cache Reduces repeated read pressure
GSI redesign Avoids hot GSI partitions
Pre-aggregation Avoids global counters
SQS/Kinesis buffering Smooths write bursts
Per-tenant throttling Controls noisy neighbors

Example: hot audit log table

Bad:

PK = TENANT#acme
SK = TIMESTAMP#2026-05-12T10:00:00Z

If Acme sends 50,000 audit events per second, this gets hot.

Better:

PK = TENANT#acme#DATE#2026-05-12#SHARD#07
SK = TIMESTAMP#2026-05-12T10:00:00Z#EVENT#abc

Read path:

Query SHARD#00
Query SHARD#01
...
Query SHARD#15
Merge by timestamp

Example: global counter

Bad:

PK = COUNTER#pageViews

Every request updates one item.

Better:

PK = COUNTER#pageViews#SHARD#00
PK = COUNTER#pageViews#SHARD#01
...
PK = COUNTER#pageViews#SHARD#99

Then aggregate periodically.

Answer

“A hot partition is not just a capacity problem. It is a data model problem. I first fix key design; then I use adaptive capacity, caching, and buffering as supporting controls.”

5. How do conditional writes support optimistic concurrency control?

Direct answer

Conditional writes let you update an item only if a condition is true. This enables optimistic concurrency control, where multiple clients can read the same item, but only one update succeeds if the item version has not changed.

Problem

Two users update the same record:

User A reads version = 7
User B reads version = 7

User A updates item to version = 8
User B updates item using stale version = 7

Without concurrency control, User B may overwrite User A’s update.

Conditional write pattern

Item:

{
  "PK": "ORDER#123",
  "SK": "META",
  "status": "PENDING",
  "version": 7
}

Update request:

SET status = 'PAID', version = 8
CONDITION version = 7

If the current version is still 7, update succeeds. If someone already changed it to 8, DynamoDB rejects the update.

Flow

Client reads item version 7
   ↓
Client sends update with condition version = 7
   ↓
DynamoDB checks current version
   ├── version is 7 → update succeeds
   └── version is not 7 → ConditionalCheckFailedException

Example pseudo-code

table.update_item(
    Key={
        "PK": "ORDER#123",
        "SK": "META"
    },
    UpdateExpression="SET #s = :new_status, version = version + :one",
    ConditionExpression="version = :expected_version",
    ExpressionAttributeNames={
        "#s": "status"
    },
    ExpressionAttributeValues={
        ":new_status": "PAID",
        ":expected_version": 7,
        ":one": 1
    }
)

Security and correctness example

For payment processing:

Order status = PENDING
Payment callback arrives
Refund callback arrives
Manual cancellation arrives

Conditional writes prevent invalid transitions:

Only set PAID if current status = PENDING
Only set REFUNDED if current status = PAID
Only set CANCELLED if current status = PENDING

Answer

“Conditional writes turn DynamoDB into a safe coordination point for item-level state transitions. They are essential for idempotency, optimistic locking, inventory updates, payment workflows, and avoiding lost updates.”

6. When should you use on-demand versus provisioned capacity?

Direct answer

Use on-demand capacity when traffic is unpredictable, spiky, new, or you want operational simplicity. Use provisioned capacity when traffic is predictable and you can manage capacity to reduce cost.

AWS describes on-demand mode as a serverless throughput option that automatically scales and uses pay-per-request pricing. AWS describes provisioned capacity as specifying read/write throughput upfront and paying based on provisioned capacity rather than actual consumed capacity. (AWS Documentation)

On-demand capacity

Best for:

New applications
Unknown traffic
Spiky workloads
Dev/test
Intermittent workloads
Operational simplicity
Fast launches

Pros:

No capacity planning
Handles spikes better
Pay per request
Simpler operations

Cons:

Can cost more at steady high volume
Less cost predictability for runaway workloads
Still needs good key design

AWS notes that new on-demand tables can sustain up to 4,000 writes/sec and 12,000 reads/sec, and on-demand can instantly accommodate up to double the previous peak traffic on a table. (AWS Documentation)

Provisioned capacity

Best for:

Predictable steady workloads
High sustained throughput
Cost optimization
Known traffic patterns
Reserved capacity use cases

Pros:

More predictable cost
Can use auto scaling
Can use reserved capacity
Better for known high traffic

Cons:

Requires capacity planning
Can throttle if under-provisioned
Can waste money if over-provisioned

Decision table

Scenario Recommendation
New SaaS launch On-demand
Unknown customer adoption On-demand
Traffic varies 100x by hour On-demand
Large predictable payment workload Provisioned + auto scaling
Constant high read/write volume Provisioned, possibly reserved capacity
One-time migration load On-demand or temporarily raised provisioned capacity
Strict cost governance Provisioned with alarms
Early-stage prototype On-demand

Answer

“On-demand optimizes for uncertainty and operational simplicity. Provisioned optimizes for predictable economics. Neither mode fixes a bad partition key.”

7. Tradeoffs of Global Tables for active-active systems

Direct answer

DynamoDB Global Tables replicate data across multiple AWS Regions and are useful for low-latency global access and regional resilience. The tradeoff is that active-active writes introduce conflict handling, replication lag, higher cost, operational complexity, and correctness challenges.

AWS describes Global Tables as multi-Region, multi-active replication for DynamoDB, and documentation notes that conflicts can occur when applications update the same item in different Regions at about the same time. (Amazon Web Services, Inc.)

Active-active architecture

Users in Singapore
   ↓
App in ap-southeast-1
   ↓
DynamoDB Global Table replica in ap-southeast-1
   ↕ replication

Users in US
   ↓
App in us-east-1
   ↓
DynamoDB Global Table replica in us-east-1

Benefits

Benefit Explanation
Low-latency local reads/writes Users access nearest Region
Regional resilience Another Region has replicated table
Active-active apps Multiple Regions can serve traffic
Managed replication Less custom replication logic
Operational simplicity versus DIY AWS manages replication mechanics

Tradeoffs

Tradeoff Why it matters
Conflict resolution Same item updated in two Regions
Replication lag Remote Region may not see write immediately
Cost Replicated writes, storage, streams/replication cost
Debugging complexity Harder to reason about data timeline
Idempotency required Retries and replication can duplicate effects
Global uniqueness Harder for counters/sequences
Compliance Data replicated across jurisdictions
Failback complexity Need careful regional recovery
Hot keys still hot Global tables do not fix poor keys

Conflict example

Region A writes:

{
  "PK": "USER#123",
  "email": "alice@example.com",
  "updatedAt": "10:00:01"
}

Region B writes at almost the same time:

{
  "PK": "USER#123",
  "email": "alice.new@example.com",
  "updatedAt": "10:00:02"
}

One update may win depending on conflict resolution. The application must be designed to tolerate this.

Better active-active design

Avoid multi-Region writes to the same item where possible.

Region-owned keys:
PK = REGION#ap-southeast-1#ORDER#123
PK = REGION#us-east-1#ORDER#456

or ownership model:

User profile writes only in user home Region.
Other Regions read replicated copy.

Workloads that fit Global Tables

Good fit:

User preferences
Shopping carts with conflict-aware design
Session metadata
Feature flags
Regional workload ownership
Low-latency read/write apps

Risky fit:

Bank ledger
Inventory counters
Strongly consistent global balances
Global unique sequence numbers
Financial state machines without conflict design

Answer

“Global Tables are excellent for regional resilience and low-latency access, but active-active does not mean strongly consistent global writes. I design item ownership, idempotency, and conflict semantics explicitly.”

8. Why is scan-heavy access usually a sign of poor DynamoDB design?

Direct answer

DynamoDB is optimized for key-value and query access using partition keys and indexes. A scan reads every item in a table or index and then filters results afterward. If your application frequently scans, it usually means the access pattern was not modeled into the primary key or secondary indexes.

Query vs scan

Query:
  Go directly to items with PK = X

Scan:
  Walk through many or all items
  Then filter

Example

Bad:

Scan table where status = 'OPEN'

This reads a large portion of the table.

Better:

GSI1PK = STATUS#OPEN
GSI1SK = CREATED_AT#...
Query GSI1 where GSI1PK = STATUS#OPEN

For multi-tenant SaaS:

GSI1PK = TENANT#t1#STATUS#OPEN

Why scan-heavy design is bad

Problem Explanation
Expensive Reads many unnecessary items
Slow Latency grows with table/index size
Unpredictable Performance worsens as data grows
Capacity waste Burns RCUs inefficiently
Hot operational impact Large scans can affect normal traffic
Pagination complexity Results may be incomplete without careful pagination
Poor user experience Queries become slower over time

When scans are acceptable

Scans are not always evil. They are okay for:

Small admin tables
Backfills
Data migration
Periodic offline jobs
Export jobs
Rare maintenance tasks
Parallel scan for controlled batch processing

But scans should not be in latency-sensitive application paths.

Answer

“A scan in a back-office migration job is fine. A scan in a customer-facing API is usually a data-modeling smell.”

9. How would you migrate a relational workload to DynamoDB safely?

Direct answer

I would not do a table-for-table migration. I would redesign around access patterns, validate correctness with dual writes or CDC, migrate incrementally, and cut over access pattern by access pattern.

Bad migration approach

MySQL customers table → DynamoDB Customers table
MySQL orders table    → DynamoDB Orders table
MySQL order_items     → DynamoDB OrderItems table

This usually fails because DynamoDB is not designed around joins and ad hoc queries.

Safe migration approach

Phase 1: Understand access patterns

Collect:

Top queries
Read/write volume
Latency SLOs
Transaction boundaries
Join patterns
Reporting needs
Consistency requirements
Cardinality
Hot entities
Tenant distribution

Example relational query:

SELECT *
FROM orders
WHERE customer_id = ?
ORDER BY created_at DESC
LIMIT 20;

DynamoDB design:

PK = CUSTOMER#123
SK = ORDER#2026-05-12T10:00:00#ORDER#9001

Phase 2: Classify workload

Workload type DynamoDB fit
Key-value lookup Excellent
High-scale event writes Excellent
Known query patterns Excellent
Complex joins Poor
Ad hoc analytics Poor
Multi-row transactions-heavy Careful
Reporting Use S3/OpenSearch/Redshift, not primary DynamoDB path

Phase 3: Design target schema

Use:

Single-table or small-table design
Tenant-aware keys
GSIs for known alternate access patterns
Sparse indexes where useful
TTL where appropriate
Streams for downstream processing
Conditional writes for correctness

Phase 4: Backfill

Relational DB snapshot
   ↓
Transform rows into DynamoDB items
   ↓
Bulk write with controlled throughput
   ↓
Validate counts and checksums

Phase 5: CDC / dual write

Use one of:

AWS DMS CDC
Application dual writes
Outbox pattern
Event stream from relational DB

Preferred for correctness:

Relational DB transaction
   ↓
Write business state
   ↓
Write outbox event
   ↓
CDC/event processor writes to DynamoDB

Phase 6: Shadow reads

Production request reads from relational DB
   ↓
Also read from DynamoDB in background
   ↓
Compare results
   ↓
Log mismatches

Phase 7: Gradual cutover

Cut over one access pattern at a time:

Customer profile lookup → DynamoDB
Recent orders lookup    → DynamoDB
Order detail lookup     → DynamoDB
Reporting               → remains analytics system

Phase 8: Rollback

Keep source of truth clear.

Before cutover:
  relational DB remains source of truth

During transition:
  decide whether DynamoDB is read model or source of truth

After cutover:
  DynamoDB may become source of truth for selected aggregates

Real-world example

An e-commerce app migrates order history reads to DynamoDB but keeps payment settlement in PostgreSQL.

Good split:

DynamoDB:
  customer order history
  order status lookup
  shopping cart
  event metadata

Relational DB:
  financial ledger
  settlement
  accounting reports

Answer

“A safe relational-to-DynamoDB migration is not a database migration; it is an access-pattern migration. I would migrate bounded, high-scale access patterns first and keep relational systems for joins, reporting, and transactional workflows that do not fit DynamoDB.”

10. What monitoring signals indicate a DynamoDB table is becoming unhealthy?

Direct answer

The best unhealthy signals are throttling, rising latency, hot keys/partitions, consumed capacity near limits, failed conditional writes where unexpected, replication lag for Global Tables, stream backlog, and error-rate changes.

Core health signals

Signal Why it matters
ThrottledRequests Table/index cannot serve requested capacity
ReadThrottleEvents / WriteThrottleEvents Read/write pressure issue
ConsumedReadCapacityUnits Read capacity usage trend
ConsumedWriteCapacityUnits Write capacity usage trend
ProvisionedReadCapacityUnits Capacity headroom
ProvisionedWriteCapacityUnits Capacity headroom
SuccessfulRequestLatency User-facing performance signal
SystemErrors AWS-side or service-side issue
UserErrors App misuse, bad conditions, validation issues
ConditionalCheckFailedRequests May be normal, but spikes can indicate concurrency conflict
GSI throttles Index hotness or under-provisioning
Account-level throttling Regional/account quota pressure
Global Table replication latency DR/active-active staleness risk
Stream iterator age Downstream consumers falling behind
DLQ depth Failed stream/event processing
Item size growth Approaching 400 KB item limit
Returned item count vs scanned count Scan/filter inefficiency
Hot key metrics/contributor insights Uneven access pattern

Unhealthy patterns

Pattern 1: table looks fine, GSI is throttling

Base table consumed WCU: normal
GSI WriteThrottleEvents: high
Application writes fail/throttle

Cause:

Every base table write also updates GSI.
GSI key is hot or under-provisioned.

Pattern 2: latency rises before throttling

SuccessfulRequestLatency p99 increasing
No big error spike yet

Cause:

Hot partition
Large item growth
Bad query pattern
Retries hiding real issue

Pattern 3: scan/filter ratio is bad

ScannedCount = 10,000
Count = 20

This means the app reads 10,000 items to return 20. That is a design smell.

Pattern 4: Global Table replication lag

Region A write succeeds
Region B sees stale data longer than expected

Risk:

Incorrect active-active behavior
Stale user state
Conflict amplification

Alarms I would create

ThrottledRequests > 0 for critical tables
GSI throttles > 0
p99 latency above SLO
Consumed capacity > 80% for provisioned tables
On-demand cost anomaly
ConditionalCheckFailedRequests sudden spike
Stream iterator age increasing
Global Table replication latency above threshold
SystemErrors above baseline
DLQ messages > 0

Answer

“For DynamoDB, I monitor not only table-level capacity but also GSI health, hot keys, latency, scan efficiency, stream lag, and conditional failure patterns. Many DynamoDB incidents are not table outages — they are bad access patterns becoming visible at scale.”

Attack tree: DynamoDB performance, correctness, or data-isolation failure

Goal: Cause DynamoDB-backed application failure or data leak

│   ├── Use low-cardinality partition key
│   ├── Use date/time-only partition key

│
├── 2. Break correctness
│   ├── No conditional writes

│   ├── App trusts tenant ID from request body
│   ├── IAM policy does not restrict leading keys

├── 4. Create cost explosion
│   ├── Scan-heavy API

Risk Control

Senior summary

The senior-level DynamoDB answer is:

“DynamoDB is extremely scalable when the access patterns are known and the partition keys distribute traffic. Most DynamoDB failures are not because DynamoDB cannot scale; they are because the data model concentrates traffic, relies on scans, ignores consistency semantics, or treats GSIs as free indexes.”

My production checklist would be:

Model access patterns first.
Use high-cardinality partition keys.
Make SaaS keys tenant-aware.
Avoid scan-heavy application paths.
Design GSIs as carefully as base tables.
Use conditional writes for concurrency and idempotency.
Choose on-demand for uncertainty and provisioned for predictable economics.
Treat Global Tables as eventually replicated active-active, not global strong consistency.
Migrate relational workloads access-pattern by access-pattern.
Monitor throttles, p99 latency, GSI health, hot keys, stream lag, and cost anomalies.

Key Takeaways

  • DynamoDB rewards explicit access-pattern-driven design and punishes generic relational habits.
  • Partition key quality determines whether the table scales smoothly.
  • Conditional writes and PITR are production defaults.
  • On-demand simplifies unpredictable traffic; provisioned can reduce cost for stable workloads.
  • IAM scoping, streams, and TTL behavior matter as much operationally as table schema.
