ElastiCache — Amazon ElastiCache

Category: Database
AWS Service Type: Managed In-Memory Data Store
Key Docs: ElastiCache User Guide

Overview

ElastiCache is AWS's managed in-memory caching service for Redis OSS, Valkey, and Memcached. It is used to reduce database load, accelerate reads, maintain ephemeral state, support rate limiting, leaderboards, pub/sub, and session management. Redis and Valkey provide persistence options, replication, cluster mode sharding, and richer data structures; Memcached is simpler and focused on distributed caching. ElastiCache handles patching, failover orchestration, backups for Redis/Valkey, parameter groups, and CloudWatch metrics. It is most valuable when low-latency repeated reads would otherwise overload upstream systems.

Top 3 Critical Mitigations

  1. Keep ElastiCache private and encrypted: A public cache is a direct data exposure path and, for Redis, often an easy write path for abuse.
  2. Never make cache the source of truth: Treat cache contents as disposable unless you have explicitly designed and backed up a stateful Redis use case.
  3. Design for cache stampede: The fastest way to melt a primary database is to let many clients miss the same hot key at once.

1. When would you choose Redis/Valkey over Memcached?

Direct answer

I would choose Redis/Valkey when I need richer data structures, persistence options, replication, high availability, pub/sub, streams, Lua scripting, sorted sets, atomic counters, distributed locks, or advanced cache/data-structure behavior.

I would choose Memcached when I need a very simple, horizontally scalable, volatile key-value cache with no persistence, simple object caching, and minimal operational complexity.

Amazon ElastiCache supports Valkey, Redis OSS, and Memcached. AWS positions Valkey/Redis OSS as engines with richer data structures and high-availability features, while Memcached is simpler and designed for straightforward object caching use cases. (AWS Documentation)

Redis/Valkey vs Memcached

Requirement Redis/Valkey Memcached
Simple object cache Good Excellent
Strings only Yes Yes
Hashes, lists, sets, sorted sets Yes No
Atomic counters Yes Limited/basic
TTL-based cache Yes Yes
Persistence options Yes No
Replication Yes No native Redis-like replication model
Automatic failover Yes in ElastiCache replication groups No equivalent primary/replica promotion pattern
Pub/Sub Yes No
Streams/queues Yes No
Lua scripting Yes No
Sorted leaderboard Yes No
Rate limiting Excellent Possible but weaker
Session store Common, but risky if only copy Possible, but less common
Distributed lock Possible with care Not ideal
Operational simplicity More complex Simpler

Real-world example

For a product catalog cache:

GET product:12345 → cached JSON blob

Memcached may be enough.

For a fintech API rate limiter:

INCR rate:user:123:2026-05-12T10:30
EXPIRE rate:user:123:2026-05-12T10:30 60

Redis/Valkey is a much better fit because you need atomic increments and TTL behavior.

Answer

“I choose Memcached when I need a disposable object cache. I choose Redis or Valkey when the cache becomes an active data-structure service with counters, sorted sets, replication, failover, streams, or richer consistency needs.”

2. How does cluster mode change key design and app behavior?

Direct answer

Cluster mode changes Redis/Valkey from a single logical shard into a partitioned keyspace across multiple shards. Keys are assigned to hash slots, and those slots are distributed across shards. This improves scale, but the application and client library must become cluster-aware.

ElastiCache Valkey/Redis OSS cluster mode enabled clusters partition data across shards and can scale to many shards; AWS documents cluster mode enabled clusters as supporting up to 500 shards. (AWS Documentation)

Non-cluster mode

Application
   ↓
Primary Redis/Valkey node
   ↓
All keys live in one shard

You scale mostly by:

larger node
read replicas

Cluster mode enabled

Application with cluster-aware client
   ↓
Cluster endpoint
   ↓
Shard 1: hash slots 0-5000
Shard 2: hash slots 5001-10000
Shard 3: hash slots 10001-16383

Why key design changes

In cluster mode, multi-key operations usually require keys to be in the same hash slot. Redis Cluster supports hash tags, where the portion inside {} determines the slot. Redis documentation explains that hash tags are used to ensure multiple keys are allocated to the same hash slot, which is important for multi-key operations. (Redis)

Example problem

Bad cluster-mode key design:

cart:user123
cart-total:user123
cart-lock:user123

These may land on different shards. A multi-key transaction or script may fail with cross-slot issues.

Better:

cart:{user123}
cart-total:{user123}
cart-lock:{user123}

Now all three keys use the same hash tag:

{user123}

and land in the same hash slot.

App behavior changes

Area Cluster mode impact
Client Must use cluster-aware client
Multi-key operations Keys must be in same slot
Lua scripts Usually need keys in same slot
Transactions More constrained
Pipelining Must route commands to correct nodes
Hot keys Still dangerous; one key maps to one shard
Scaling Add shards and reshard slots
Failover Shard-level promotion
Observability Monitor per-shard, not only cluster aggregate

Common failure

MGET user:1 user:2 user:3

If keys are in different slots, the client may fail unless it splits the request per shard. AWS re:Post notes that multi-key operations across different hash slots can produce CROSSSLOT errors in cluster mode. (Repost)

Answer

“Cluster mode improves horizontal scale, but it makes key design part of application architecture. I need to design hash tags deliberately for multi-key operations and avoid hot keys that concentrate traffic on one shard.”

3. What causes cache stampede and how do you prevent it?

Direct answer

A cache stampede happens when many clients request the same expired or missing key at the same time, causing all of them to hit the backend database or API simultaneously. The cache was supposed to protect the backend, but during a miss storm it amplifies load.

Simple example

Popular key: homepage:top-products
TTL: 5 minutes

At 10:00:00 key expires.
10,000 requests arrive.
All miss cache.
All query database.
Database overloads.

Stampede flow

Hot key expires
   ↓
Many concurrent cache misses
   ↓
All clients regenerate value
   ↓
Database/API overloaded
   ↓
Latency rises
   ↓
Retries increase
   ↓
Outage

Prevention techniques

Technique How it helps
TTL jitter Prevents many keys expiring at same time
Request coalescing Only one request rebuilds cache
Distributed lock One worker recomputes; others wait/use stale
Stale-while-revalidate Serve stale value while refreshing
Background refresh Refresh before expiry
Negative caching Cache “not found” responses briefly
Rate limiting Protects backend
Circuit breaker Stops retry storm
Hot key prewarming Loads known hot keys before traffic
Multi-level cache Local memory + Redis/Valkey

Example: lock-based protection

Client requests key
   ↓
Cache miss
   ↓
Try SET lock:key NX EX 10
   ├── Lock acquired → rebuild value → set cache → release lock
   └── Lock not acquired → wait briefly or serve stale value

Pseudo-code:

value = cache.get(key)
if value:
    return value

if cache.set(f"lock:{key}", "1", nx=True, ex=10):
    value = db.query(...)
    cache.set(key, value, ex=300 + random_jitter())
    cache.delete(f"lock:{key}")
    return value
else:
    stale = cache.get(f"stale:{key}")
    if stale:
        return stale
    sleep(50_ms)
    return cache.get(key) or fallback()

Senior security/reliability view

Cache stampede is a reliability failure and can become a security issue. An attacker can deliberately request cache-expensive keys or force cache misses to overload the database.

Attack pattern:

Attacker identifies expensive uncached endpoint
   ↓
Sends many requests with unique parameters
   ↓
Cache miss every time
   ↓
Database overloaded
   ↓
Service unavailable

Controls

Normalize cache keys
Limit unbounded query parameters
Use request-level rate limits
Apply negative caching
Set backend query budgets
Use circuit breakers
Monitor cache miss spikes

Answer

“A cache stampede is a cache-miss amplification problem. I prevent it with jitter, request coalescing, stale-while-revalidate, distributed locking, and backend protection.”

4. How would you run ElastiCache securely in a multi-tier VPC?

Direct answer

I would run ElastiCache in private subnets only, restrict access using security groups, enable encryption in transit and at rest where supported, require authentication/RBAC/IAM where available, prevent public exposure, and limit which application tiers can connect.

ElastiCache supports in-transit encryption for data moving between clients and cache endpoints, but AWS notes TLS has performance impact and should be benchmarked for the workload. (AWS Documentation) ElastiCache also supports at-rest encryption; AWS states at-rest encryption is always enabled on serverless caches. (AWS Documentation) ElastiCache supports authentication using IAM and Valkey/Redis OSS AUTH, plus RBAC authorization. (AWS Documentation)

Secure VPC architecture

   ↓
CloudFront / WAF / ALB
   ↓
Public subnets
   ↓
Application tier in private subnets
   ↓
ElastiCache subnet group in isolated/private subnets
   ↓
No direct internet route

Network controls

ElastiCache security group:
  inbound 6379 only from application security group
  no inbound from bastion by default
  no inbound from 0.0.0.0/0
  outbound minimal where possible

Application security group:
  egress 6379 to cache security group

Security baseline

Area Control
Network placement Private/isolated subnets
Inbound access Security group source = app SG only
Internet exposure No public access
Encryption in transit TLS enabled
Encryption at rest Enabled where supported/required
Authentication Redis AUTH, RBAC, or IAM auth where supported
Authorization Separate users/access strings

IAM authentication

ElastiCache supports IAM authentication for Valkey/Redis OSS version 7 or above, allowing IAM identities and least-privilege access models for cache users. (AWS Documentation)

Example multi-tier rule

Web tier:
  cannot access ElastiCache

API tier:
  can access ElastiCache on 6379

Worker tier:
  can access ElastiCache on 6379

  no direct cache access unless required

Secrets stored in app config

Shared admin Redis user across all services
No tenant-aware key design
No monitoring for unusual commands

Answer

“I do not treat ElastiCache as safe just because it is in a VPC. I restrict network paths, enforce authentication, encrypt traffic, scope users, rotate credentials, and ensure only the application tier can reach it.”

5. What are the risks of storing session state only in Redis?

Direct answer

Redis/Valkey can be a good session store, but storing session state only in Redis creates availability, durability, security, and operational risks. If Redis loses data, evicts keys, fails over incorrectly, or is compromised, users may be logged out or sessions may be hijacked.

Risks

Risk Explanation
Data loss Cache is not always a durable source of truth
Eviction Sessions can be removed under memory pressure
Failover gaps Recent writes may not be replicated before failover
Operational outage Cache outage becomes auth/session outage
Security compromise Session tokens can be stolen
No auditability Redis-only session state may be hard to reconstruct
Inconsistent TTL Bugs can create never-expiring sessions
Cross-region DR Sessions may not exist in failover region
Compliance Sensitive session data may be stored insecurely
Logout semantics Global logout can fail if keys are lost or replicated badly

Example failure

User logs in
Session stored only in Redis
Redis memory reaches maxmemory
Eviction policy removes session key
User is suddenly logged out

Worse:

Redis exposed to broad app subnet
Compromised app reads session:* keys
Attacker replays active sessions

Better session patterns

Pattern Use case
Stateless signed token Reduces server-side session dependency
Redis session store + durable refresh token DB Common web/mobile pattern
Short-lived access token + revocation list Balance scale and control
Redis as cache of session metadata Durable source remains elsewhere
Session encryption Protects sensitive fields
Hash session IDs before storing Reduces impact of key leakage
Per-session TTL and idle timeout Limits lifetime
Global logout list Supports revocation
Multi-AZ Redis with replicas Improves availability

Safer design

Browser/mobile
   ↓
Short-lived access token
   ↓
API validates token
   ↓
Redis stores session metadata/revocation/cache
   ↓
Durable store keeps refresh token/session record

Production rule

Do not store this in plaintext in Redis:

{
  "password": "...",
  "mfa_secret": "...",
  "full_card_number": "...",
  "raw_access_token": "..."
}

Store minimal metadata:

{
  "userId": "u123",
  "tenantId": "t456",
  "authLevel": "mfa",
  "createdAt": "...",
  "lastSeen": "..."
}

Answer

“Redis can hold session state, but I would not make a volatile cache the only security source of truth for high-risk sessions. I would design for eviction, failover, compromise, logout, and regional disaster recovery.”

6. How do failover and replica promotion work in ElastiCache?

Direct answer

For Valkey/Redis OSS replication groups, ElastiCache can use a primary node and one or more replicas. If the primary fails and Multi-AZ automatic failover is enabled, ElastiCache promotes a replica to primary and updates the endpoint so clients can reconnect to the new primary.

For cluster mode disabled, AWS documents that a Valkey/Redis OSS cluster has a single shard, and with replicas the replication group can support failover within that shard. Cluster mode enabled spreads data across multiple shards, each with its own primary and replicas. (AWS Documentation)

Cluster mode disabled

Replication group
  └── Shard 1
      ├── Primary
      ├── Replica 1
      └── Replica 2

Failover:

Primary fails
   ↓
ElastiCache detects failure
   ↓
Replica promoted
   ↓
Primary endpoint points to new primary
   ↓
Clients reconnect

Cluster mode enabled

Replication group
  ├── Shard 1: primary + replicas
  ├── Shard 2: primary + replicas
  └── Shard 3: primary + replicas

Failover is per shard:

Shard 2 primary fails
   ↓
Shard 2 replica promoted
   ↓
Slots for shard 2 served by new primary
   ↓
Cluster-aware clients update topology

Important app behavior

Applications must handle:

Connection drops
Reconnects
MOVED/ASK redirections in cluster mode
Transient write failures
Possible loss of very recent writes
Read replica lag

Client requirements

Mode Client behavior
Cluster mode disabled Reconnect to primary endpoint
Cluster mode enabled Use cluster-aware client, refresh slot map
Read replicas Handle stale reads
TLS/auth enabled Re-authenticate on reconnect

Real-world failure

A service uses a Redis client with infinite connection pooling and no reconnect backoff. During failover:

Primary fails
   ↓
Connections break
   ↓
Client retry storm
   ↓
New primary overloaded
   ↓
Application latency spikes

Better client design:

Short connect timeout
Exponential backoff with jitter
Pool health checks
Command timeout
Circuit breaker
Fail closed for auth/rate-limit decisions
Graceful degradation for optional caches

Answer

“ElastiCache failover promotes a replica, but successful failover is not only a server-side event. The application must reconnect, refresh topology, tolerate transient errors, and avoid retry storms.”

7. What eviction policy would you use for rate limiting versus general caching?

Direct answer

For general caching, I usually use an LRU/LFU-style eviction policy, depending on access patterns. For rate limiting, I prefer to avoid eviction entirely for active limiter keys by sizing memory correctly and using TTLs. If eviction must exist, I would be very careful because evicting rate-limit keys can create a security bypass.

General caching

Good choices:

allkeys-lru
allkeys-lfu
volatile-lru
volatile-ttl

Use case:

Cache product pages, user profiles, configuration, API responses

If a cached item is evicted, the system can rebuild it from the source of truth.

Rate limiting

Rate-limit keys are security controls:

rate:ip:203.0.113.10:minute
rate:user:u123:minute
rate:apiKey:k789:minute

If these keys are evicted, an abusive client may get a clean slate.

For rate limiting, prefer:

maxmemory-policy noeviction

or isolated Redis/Valkey cluster dedicated to rate limiting.

Why noeviction for rate limiting?

If memory is full:

INCR rate:user:attacker

should fail loudly, not silently evict another limiter key.

Security choice:

If limiter unavailable:
  fail closed for sensitive endpoints
  fail open only for low-risk user experience paths

Recommended separation

Cache cluster:
  maxmemory-policy = allkeys-lru or allkeys-lfu

Rate-limit cluster:
  maxmemory-policy = noeviction
  strict TTL on all keys
  bounded cardinality
  memory alarms

Rate-limiter key design

INCR rate:{tenantId}:{userId}:2026-05-12T10:31
EXPIRE key 60

Use hash tags carefully in cluster mode if atomic multi-key logic is needed.

Answer

“For general cache, eviction is expected. For rate limiting, eviction can become a security bypass. I would isolate rate limiting and prefer noeviction with strict TTLs and strong memory alarms.”

9. How do you size cache memory and node count for a read-heavy service?

Direct answer

I size cache based on working set size, item size distribution, overhead, TTL, expected hit ratio, read QPS, network throughput, CPU, replication needs, and failure headroom. I do not size only by raw data size.

Sizing inputs

Number of keys
Average item size
p95/p99 item size
TTL distribution
Read QPS
Write QPS
Hit ratio target
Peak traffic multiplier
Eviction tolerance
Replication requirement
Cluster mode or not
Availability target

Memory sizing formula

Approximate:

Required memory =
  working_set_data_size
+ Redis/Valkey object overhead
+ key overhead
+ fragmentation overhead
+ replication/failover buffer headroom
+ growth headroom

Practical rule:

Do not run cache at 95% memory.
Target maybe 60–75% steady-state depending on workload.

Example

50 million keys
Average value size = 500 bytes
Average key size = 50 bytes
Raw value data = 25 GB
Key + metadata overhead = significant
Fragmentation/headroom = 30–50%
Replica/failover/growth headroom = additional

A “25 GB dataset” may require 50+ GB of node memory depending on encoding, object overhead, fragmentation, and safety margin.

Node count sizing

For read-heavy workloads:

Total read QPS
÷ per-node sustainable read QPS
= number of read nodes

But also check:

Engine CPU
Network bandwidth
Connections
Hot keys
Shard distribution
Replica lag
p99 latency

Cluster mode disabled read scaling

1 primary
N replicas
Reads distributed to replicas
Writes go to primary

Good when:

data fits in one shard
read scale is main issue
multi-key operations are common

Cluster mode enabled scale

N shards
Each shard has primary + replicas
Data distributed by hash slot

Good when:

data too large for one shard
write throughput too high for one shard
need horizontal partitioning

Read-heavy architecture

Application
   ↓
Local in-process cache for ultra-hot small values
   ↓
ElastiCache read replicas / cluster mode
   ↓
Database fallback

Hot key issue

Even with many nodes:

key = product:iphone-launch

can overload one shard or one node.

Mitigate:

local cache
request coalescing
replicated hot-key copies
short payload
CDN if HTTP content
precompute

Answer

“I size cache using the working set and performance SLO, not total database size. Then I validate with load tests for memory, CPU, network, hit ratio, and p99 latency under failover conditions.”

10. What monitoring signals tell you a cache tier is becoming unhealthy?

Direct answer

The most important signals are hit ratio degradation, rising latency, evictions, memory pressure, CPU pressure, connection growth, replication lag, failover events, network saturation, command errors, and backend load increase.

AWS recommends monitoring key ElastiCache metrics with CloudWatch alarms, and for Valkey/Redis OSS specifically, AWS highlights EngineCPUUtilization as a precise metric because Redis/Valkey engine processing is largely single-threaded from the command execution perspective. (AWS Documentation)

│   ├── Force high-cardinality cache misses
│   ├── Flush hot keys
│   ├── Use very short TTLs
│   └── Create retry storm
│
├── 2. Exploit weak cache security
│   ├── Cache reachable from broad VPC
│   ├── No authentication
│   ├── TLS disabled for sensitive traffic
│   ├── Shared admin user
│   ├── Secrets stored in plaintext
│   └── Overbroad security groups
│
├── 3. Bypass rate limiting
│   ├── Rate-limit keys evicted
│   ├── Fail-open limiter during Redis outage
│   ├── Attacker creates unbounded keys
│   ├── Clock/window design bug
│   └── Cluster cross-slot bug breaks limiter
│
├── 4. Break session security
│   ├── Session state only in Redis
│   ├── Session keys stolen
│   ├── Eviction logs users out
│   ├── Failover loses recent session writes
│   └── No durable revocation state
│
├── 5. Cause cluster-mode failure
│   ├── Multi-key commands across slots
│   ├── Non-cluster-aware client
│   ├── Hot key overloads one shard
│   ├── Bad hash tag design
│   └── Resharding not tested
│
├── 6. Trigger memory exhaustion
│   ├── Missing TTLs
│   ├── Large values
│   ├── Unbounded key cardinality
│   ├── Bad maxmemory policy
│   └── Fragmentation grows
│
└── 7. Fail migration/failover
    ├── No client reconnect logic
    ├── Old scripts incompatible with cluster mode

    ├── No rollback path

Summary

The senior-level answer is:

“Redis/Valkey is not just a cache; it is often a distributed data-structure and coordination layer. That means I must design for key distribution, failover, memory pressure, eviction semantics, security boundaries, and application behavior during cache misses. Memcached is simpler and excellent for disposable object caching, but Redis/Valkey is the better choice when the cache needs richer behavior.”

My production checklist:

Choose Redis/Valkey for rich data structures, counters, rate limits, sessions, HA, and replication.
Choose Memcached for simple disposable object caching.
Use cluster mode only with cluster-aware clients and deliberate hash-tag key design.
Prevent stampedes with jitter, locks, request coalescing, and stale-while-revalidate.
Run ElastiCache in private subnets with TLS, AUTH/RBAC/IAM, and strict security groups.
Do not store critical session state only in a volatile cache without a durable recovery model.
Use noeviction for rate limiting; use LRU/LFU policies for general caching.
Migrate stateful Redis carefully; cache-only Redis can be rebuilt more easily.
Size by working set, overhead, CPU, network, hit ratio, and failover headroom.
Monitor hit ratio, p99 latency, EngineCPUUtilization, memory, evictions, connections, and replica lag.

Key Takeaways

  • ElastiCache reduces latency and protects upstream systems, but it is still a networked service that needs strong isolation.
  • Redis/Valkey adds features and state options; Memcached stays simpler and purely cache-oriented.
  • Cache stampede and wrong eviction policy are the most common application-level failures.
  • Backups matter if Redis/Valkey is used for more than ephemeral caching.
  • Secure subnet placement, TLS, and auth are the operational baseline.
