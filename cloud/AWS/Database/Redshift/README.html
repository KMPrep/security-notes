Redshift — Amazon Redshift

Category: Database
AWS Service Type: Managed Data Warehouse
Key Docs: Redshift Database Developer Guide

Overview

Redshift is AWS's managed petabyte-scale cloud data warehouse for analytical workloads. It is optimized for OLAP queries over large datasets rather than OLTP transactions. Redshift stores data in a columnar format, uses massively parallel processing (MPP), and integrates closely with S3 data lakes, Spectrum, Glue, IAM, Lake Formation, and BI tools. Clusters can be provisioned traditionally or run in Redshift Serverless. It is commonly used for enterprise reporting, dashboards, data marts, and cross-domain analytics where query latency in seconds is acceptable in exchange for high-scale scanning and aggregation performance.

Top 3 Critical Mitigations

  1. Keep Redshift private and federated: Analytical warehouses often contain the highest-value aggregated data in the company. Public exposure is rarely justified.
  2. Get table design right early: Sort keys, distribution keys, and partitioning shape both performance and cost more than instance size alone.
  3. Separate analytics from transactional traffic: Redshift is excellent for scans and aggregation, not for OLTP behavior.

1. How does Redshift differ from RDS and DynamoDB in workload design?

Direct answer

Redshift is an OLAP data warehouse. It is designed for analytical queries over large datasets: joins, aggregations, scans, reporting, BI dashboards, and historical analysis.

RDS is OLTP relational database service. It is designed for transactional applications: orders, payments, user records, inventory updates, and low-latency row-level reads/writes.

DynamoDB is a NoSQL key-value/document database. It is designed for high-scale predictable access patterns using partition keys, sort keys, and secondary indexes.

Mental model

RDS:
  "Process this order now."

DynamoDB:
  "Fetch/update this item by known key at massive scale."

Redshift:
  "Analyze 5 years of orders by region, product, campaign, and month."

Comparison

Area Redshift RDS DynamoDB
Workload type OLAP analytics OLTP transactions Key-value / document access
Query style Large scans, joins, aggregations Point lookups, transactions, joins Query by key/index
Data model Star/snowflake, denormalized analytics models Normalized relational model Access-pattern driven NoSQL
Storage format Columnar Row-oriented engine storage Partitioned key-value storage
Concurrency BI/ETL workload management App transaction concurrency Massive key-based concurrency
Latency target Seconds to minutes depending on query Milliseconds to low seconds Single-digit ms possible
Scaling concern Query throughput, scan volume, distribution, WLM CPU, locks, IOPS, connections Partition key distribution
Bad fit High-frequency row updates Huge analytical scans Ad hoc joins/scans

Example

An e-commerce company might use all three:

RDS:
  checkout transactions
  payment state
  inventory updates

DynamoDB:
  shopping cart
  user session
  product recommendation cache
  event metadata by key

Redshift:
  daily revenue reports
  fraud trend analysis
  marketing attribution
  seller performance dashboards

Senior answer

“Redshift is not a replacement for RDS or DynamoDB. It is where operational data goes after it has been shaped for analytics. The design mindset changes from transaction correctness and key access to scan efficiency, distribution, sort order, workload isolation, and cost per query.”

2. What do distribution keys and sort keys optimize?

Direct answer

Distribution keys optimize where rows are placed across Redshift compute nodes/slices. They are mainly about reducing data movement during joins and spreading workload evenly.

Sort keys optimize how data is physically ordered on disk. They are mainly about reducing the amount of data scanned using zone maps and improving range-filter query performance.

Amazon Redshift supports distribution styles such as AUTO, EVEN, KEY, and ALL; if no distribution style is specified, Redshift uses AUTO. (AWS Documentation) Redshift sort keys can be compound or interleaved, and AWS recommends AUTO sort keys where Redshift should choose the sort order automatically. (AWS Documentation)

Distribution key

Distribution decides:

Which node/slice owns this row?

Example:

fact_orders DISTKEY(customer_id)
dim_customers DISTKEY(customer_id)

Now rows for the same customer_id are colocated, making joins cheaper.

Bad distribution

fact_orders distributed by order_id
dim_customers distributed by customer_id

Join:

SELECT *
FROM fact_orders f
JOIN dim_customers c
  ON f.customer_id = c.customer_id;

Redshift may need to redistribute data across nodes before joining.

Sort key

Sort decides:

How rows are ordered on disk?

Example:

fact_orders SORTKEY(order_date)

Query:

SELECT sum(amount)
FROM fact_orders
WHERE order_date BETWEEN '2026-01-01' AND '2026-01-31';

Redshift can skip blocks that do not contain January data.

Distribution styles

Style Use case
AUTO Let Redshift choose and adjust
EVEN Default-like spread when no obvious join key
KEY Colocate large tables joined on same key
ALL Copy small dimension table to every node

Sort key choices

Sort key type Use case
Compound Common when queries filter by leading columns, such as date first
Interleaved Useful when multiple columns are filtered independently, but maintenance can be heavier
AUTO Good default unless you have clear evidence

Real-world example

For a sales warehouse:

fact_sales:
  DISTKEY(customer_id)
  SORTKEY(sale_date)

dim_customer:
  DISTKEY(customer_id)

dim_product:
  DISTSTYLE ALL if small

This optimizes:

customer joins
date-filtered reports
small dimension joins

In short

“Distribution keys reduce network movement. Sort keys reduce disk scanning. Bad distribution causes shuffles; bad sorting causes unnecessary scanning.”

3. When should you choose Redshift Serverless versus provisioned clusters?

Direct answer

Choose Redshift Serverless when you want minimal infrastructure management, variable or unpredictable analytics workloads, fast onboarding, dev/test analytics, or team-level workgroups without sizing clusters.

Choose provisioned Redshift clusters when you need predictable sustained workloads, mature workload management, reserved capacity economics, tighter sizing control, or stable enterprise BI/ETL patterns.

Amazon Redshift Serverless lets teams run and scale analytics without provisioning and managing a data warehouse. (AWS Documentation) Provisioned Redshift clusters consist of a leader node and compute nodes; for RA3 nodes, AWS lets you scale and pay for compute and managed storage independently. (AWS Documentation)

Decision table

Scenario Better choice
New analytics team Serverless
Spiky dashboard usage Serverless
Ad hoc data science queries Serverless
Dev/test warehouse Serverless
Departmental analytics Serverless
24/7 high-volume BI Provisioned
Predictable nightly ETL + BI Provisioned
Need reserved-capacity economics Provisioned
Strict WLM tuning maturity Provisioned
Existing large Redshift platform Provisioned or hybrid

Serverless benefits

No cluster sizing upfront
Auto-scales capacity
Good for intermittent usage
Good for experimentation
Fast team onboarding

Serverless risks

Cost surprise if queries are inefficient
Less direct node-level sizing control
Need workgroup/base capacity governance
VPC/IP requirements still matter
Poor SQL still costs money

AWS notes Redshift Serverless has VPC/subnet requirements, including free IP address requirements per subnet. (AWS Documentation)

Provisioned benefits

Predictable capacity
Stable performance profile
Reserved capacity options
Strong workload isolation patterns
Better for sustained high utilization
Mature operational tuning

Provisioned risks

Need sizing decisions
Idle capacity cost
Cluster maintenance decisions
Scaling requires planning

Answer

“Serverless is best when demand is variable and the priority is speed and low ops. Provisioned is best when usage is sustained, predictable, and cost/performance tuning matters. I would decide using query concurrency, daily processed data, utilization pattern, cost model, and governance needs.”

4. How does Spectrum change the boundary between warehouse and data lake?

Direct answer

Redshift Spectrum lets Redshift query structured and semi-structured data directly in Amazon S3 without loading all data into Redshift tables. That changes the boundary: Redshift no longer needs to own every analytical dataset physically. S3 becomes the durable data lake, while Redshift becomes the high-performance warehouse/query engine for curated, hot, or frequently joined data.

AWS describes Redshift Spectrum as querying and retrieving data from files in S3 without loading the data into Redshift tables; much of the processing occurs in the Spectrum layer while most data remains in S3. (AWS Documentation)

Traditional warehouse model

Source systems
   ↓ ETL
Redshift internal tables
   ↓
BI / analytics

Everything important is loaded into Redshift.

Lake + warehouse model

Source systems
   ↓
S3 data lake: raw, curated, archive data
   ↓
Glue Data Catalog external tables
   ↓
Redshift Spectrum queries S3
   ↓
Redshift internal tables for hot/high-value data
   ↓
BI / analytics

What Spectrum is good for

Querying cold historical data
Joining Redshift hot data with S3 lake data
Reducing warehouse storage cost
Exploring large raw datasets
Keeping data in open formats such as Parquet

What Spectrum is not ideal for

Very low-latency BI dashboards
High-concurrency small queries
Poorly partitioned S3 data
Small-file-heavy lakes
Uncompressed CSV at huge scale
Frequent updates/deletes

Example

Keep recent 90 days in Redshift:

redshift.fact_orders_recent

Keep older 7 years in S3:

s3://company-lake/orders/year=2020/month=01/...

Query both:

SELECT customer_id, sum(amount)
FROM (
  SELECT customer_id, amount FROM fact_orders_recent
  UNION ALL
  SELECT customer_id, amount FROM spectrum.orders_history
)
GROUP BY customer_id;

Answer

“Spectrum lets me keep cold or massive data in S3 and query it from Redshift. The warehouse becomes the performance layer; the lake becomes the durable, low-cost storage layer.”

5. Why is columnar storage well suited for OLAP?

Direct answer

Columnar storage is well suited for OLAP because analytical queries usually read a few columns across many rows. A columnar engine can scan only the needed columns, compress similar values efficiently, and perform vectorized aggregations more effectively.

Row-oriented storage

Good for OLTP:

Row:
customer_id | order_id | amount | status | created_at | address | phone | ...

If you fetch one order, row storage is efficient.

Columnar storage

Good for OLAP:

customer_id column
order_id column
amount column
status column
created_at column
...

If you run:

SELECT region, sum(amount)
FROM sales
WHERE sale_date >= '2026-01-01'
GROUP BY region;

the engine mostly needs:

region
amount
sale_date

It does not need to read:

shipping_address
phone_number
comments
metadata_json

Benefits

Benefit Why it helps OLAP
Fewer columns scanned Less I/O
Better compression Similar column values compress well
Faster aggregation Column batches are efficient
Better cache locality Only relevant data loaded
Works well with sort keys Skips blocks efficiently
Efficient for wide tables Avoids reading unused attributes

Real-world example

A table has 200 columns and 10 billion rows. A dashboard query needs only:

date
country
revenue
product_category

A row-store may touch much more data. A column-store scans only the needed columns.

Answer

“OLTP asks for a few rows with many columns. OLAP asks for many rows with a few columns. Columnar storage matches that access pattern.”

6. How would you secure cross-account data sharing in Redshift?

Direct answer

I would secure Redshift cross-account data sharing with explicit producer approval, consumer association, least-privilege datashares, object-level grants, account allowlists, IAM governance, encryption controls, audit logging, and data classification reviews.

Amazon Redshift data sharing lets you share live data across clusters, workgroups, AWS accounts, and Regions without manually copying data. (AWS Documentation) For cross-account data sharing, AWS requires a two-way handshake: the producer authorizes the consumer account, and the consumer account administrator associates the datashare. (AWS Documentation)

Secure sharing model

Producer account
  owns sensitive warehouse data
  creates datashare
  grants only required schemas/tables/views
  authorizes consumer account

Consumer account
  accepts/associates datashare
  creates database from datashare
  grants local users/groups access

Security controls

Control Purpose
Use views instead of raw tables Mask/filter sensitive data
Share minimum schemas/tables Least privilege
Account allowlist Prevent accidental external sharing
Data classification review Avoid sharing restricted data
Row/column-level controls Limit tenant/region/PII exposure
IAM policy on datashare operations Restrict who can authorize consumers
CloudTrail/audit logs Track share creation and changes
KMS/encryption governance Ensure data remains protected
Separate producer/consumer duties Avoid unilateral broad sharing
Periodic access review Remove stale consumers

Bad pattern

Share entire analytics schema to another AWS account.
Consumer account has broad BI access.
No masking views.
No review of PII columns.

Risk:

cross-account data leakage
regulatory violation
unbounded downstream sharing
hard-to-track consumers

Better pattern

Create schema: shared_finance_safe
Create views:
  v_monthly_revenue_no_pii
  v_region_sales_aggregated
  v_customer_metrics_masked

Share only those views.
Authorize only approved consumer accounts.

Attack scenario

Analyst in producer account creates datashare with raw customer table
   ↓
Consumer account associates datashare
   ↓
Consumer BI users see email, phone, tax ID
   ↓
Data leaves intended trust boundary

Controls:

Require data-owner approval
Use views with masking
Block raw PII datashares through policy/process
Monitor datashare changes

Answer

“Cross-account data sharing should not be treated like a convenient shortcut around data governance. I would share curated, approved, least-privileged views and treat the consumer account as a separate trust boundary.”

7. What signals indicate poor workload management configuration?

Direct answer

Poor Redshift workload management, or WLM, shows up as queueing, unpredictable query latency, ETL starving BI, BI starving ETL, high concurrency with low throughput, memory spills to disk, frequent query aborts, and poor isolation between workload classes.

Signals

Signal What it suggests
High queue wait time Not enough slots/capacity for workload
Short BI queries wait behind long ETL Poor queue isolation
ETL jobs fail during dashboard peak Resource contention
Queries spill to disk Insufficient memory per query or bad query plan
High concurrency, low throughput Too many queries competing
CPU high but progress low Inefficient queries or contention
Disk I/O high Spills, scans, poor sort/distribution
Frequent query timeouts Wrong queue/time limits
Unpredictable dashboard latency BI shares resources with heavy jobs
One user/team dominates cluster Missing user/group/query group routing
Concurrency scaling always active Baseline capacity or WLM wrong
Manual queue tuning constantly needed Workload profile not understood

Bad WLM design

Single shared queue:
  ETL jobs
  BI dashboards
  data science exploration
  admin queries
  ad hoc analyst queries

Result:

One bad query slows everyone.
Critical dashboards wait behind batch jobs.
ETL misses SLA.

Better WLM design

Queue 1: short BI dashboard queries
Queue 2: scheduled ETL
Queue 3: data science/ad hoc
Queue 4: admin/maintenance

Add:

query priorities
timeouts
user/group routing
concurrency scaling where useful
query monitoring rules

Real-world example

A CFO dashboard runs every morning at 9 AM. At the same time, nightly ETL overran and is still running. If both share a queue, the CFO dashboard waits 15 minutes.

Better:

Dashboard queue:
  short timeout
  high priority
  limited query complexity

ETL queue:
  controlled concurrency
  larger memory
  lower priority during business hours

Answer

“Poor WLM is visible when business-critical short queries queue behind heavy batch work. The fix is not simply more nodes; it is workload isolation, priorities, query rules, and sometimes separate clusters or serverless workgroups.”

8. How would you design a low-cost analytics architecture that mixes S3 and Redshift?

Direct answer

I would use S3 as the low-cost durable data lake, store data in partitioned columnar formats such as Parquet, use Glue Data Catalog for metadata, use Redshift Spectrum for cold or external data, and load only hot/curated/high-concurrency datasets into Redshift internal tables.

Reference architecture

Source systems
  ├── RDS/DynamoDB exports
  ├── application logs
  ├── SaaS data
  └── events
        ↓
S3 data lake
  ├── raw zone
  ├── cleaned zone
  └── curated zone
        ↓
Glue Data Catalog
        ↓
Redshift Spectrum for external S3 queries
        ↓
Redshift internal tables for hot BI datasets
        ↓
QuickSight / Tableau / analysts

Cost optimization principles

Design choice Cost benefit
Store raw/cold data in S3 Much cheaper than warehouse storage
Use Parquet/ORC Less data scanned
Partition by date/tenant/region Less S3 data scanned
Compress files Lower scan cost
Avoid small files Better query efficiency
Load only hot aggregates into Redshift Smaller warehouse
Use materialized views/summary tables Faster dashboards
Use Serverless for intermittent workloads Avoid idle cluster cost
Use provisioned RA3 for sustained workloads Better predictable economics
Lifecycle old data to cheaper S3 classes Lower storage cost

Hot vs cold split

Redshift internal tables:
  last 90 days
  dashboard aggregates
  high-concurrency BI tables
  curated dimensions

S3 + Spectrum:
  raw logs
  historical data older than 90 days
  exploratory datasets
  infrequently queried archives

Example

Security analytics platform:

S3:
  raw CloudTrail, VPC Flow Logs, EDR telemetry for 7 years

Redshift:
  last 30 days normalized alerts
  daily aggregates
  executive dashboards

Spectrum:
  incident investigation over historical S3 data

Performance rules

Use Parquet, not raw CSV where possible.
Partition S3 paths by high-value filters.
Avoid millions of tiny files.
Avoid SELECT * from external tables.
Push filters down.
Pre-aggregate repeated BI queries.

Answer

“Low-cost Redshift architecture means S3 stores the lake, Redshift stores the hot warehouse, Spectrum bridges the two, and physical design prevents expensive full scans.”

9. What are the risks of letting ETL and BI share the same queues?

Direct answer

If ETL and BI share the same Redshift queues, long-running batch jobs can starve dashboards, dashboard bursts can delay data loads, memory-heavy transformations can cause spills, and business users experience unpredictable performance.

Risk breakdown

Risk Example
BI latency spikes Dashboard waits behind large ETL query
ETL SLA miss Analyst queries consume queue slots
Memory contention Large joins spill to disk
Unfair resource usage One workload dominates
Retry amplification BI tool retries slow queries
Lock/contention issues ETL writes while BI reads
Poor incident isolation Hard to identify which workload caused degradation
Cost spikes Concurrency scaling or serverless capacity surges
Failed refreshes BI extracts timeout
Bad user experience Executives lose trust in dashboards

Example failure

06:00 - ETL starts rebuilding fact_sales
08:55 - BI users open dashboards
09:00 - CFO dashboard query enters same queue
09:10 - Dashboard still waiting
09:12 - BI tool retries queries
09:15 - Cluster load worsens

Better design

BI queue:
  short-running queries
  high priority
  timeout rules
  concurrency tuned for dashboard load

ETL queue:
  controlled concurrency
  larger memory
  scheduled windows
  lower priority during business hours

Ad hoc queue:
  lower priority
  strict timeout
  query limits

Stronger isolation options

Separate Redshift workgroups
Separate clusters
Data sharing from producer ETL cluster to consumer BI cluster
Materialized views for BI
Serverless workgroup for ad hoc/data science

Answer

“ETL and BI have opposite workload profiles. ETL wants throughput and memory. BI wants low queue time and predictable latency. Sharing one queue couples their failure modes.”

10. How would you migrate an on-prem data warehouse to Redshift incrementally?

Direct answer

I would migrate incrementally by domain and workload, not by copying everything at once. I would start with discovery, build a landing zone in S3, migrate historical data, set up CDC or scheduled incremental loads, validate data quality, run parallel reporting, then cut over subject areas one by one.

“A successful Redshift migration is not a lift-and-shift of tables. It is a workload migration: data domains, ETL pipelines, reports, security models, and performance expectations move incrementally with validation.”

Summary

The senior-level answer is:

“Redshift is an analytics platform, not an application database. Its success depends on modeling data for scans, joins, aggregations, workload isolation, and cost-efficient storage. RDS optimizes transactions, DynamoDB optimizes key-based access, and Redshift optimizes analytical throughput.”

My production checklist:

Use Redshift for OLAP, not OLTP.
Design distribution keys to reduce data movement.
Design sort keys to reduce scanning.
Use Serverless for variable/ad hoc workloads.
Use provisioned RA3 clusters for sustained predictable workloads.
Keep cold/raw data in S3 and query with Spectrum.
Load hot/curated data into Redshift internal tables.
Secure cross-account sharing with least-privileged curated views.
Separate ETL, BI, and ad hoc workloads.
Migrate on-prem warehouses incrementally by domain with parallel validation.

Key Takeaways

  • Redshift is for analytics, not transactions.
  • Data model and workload management dominate performance outcomes.
  • Spectrum extends analytics into S3, but scanned-byte cost must be governed.
  • Private networking, encryption, and IAM scoping are mandatory because warehouses concentrate sensitive data.
  • Serverless simplifies operations, while provisioned clusters offer more explicit control for steady workloads.
