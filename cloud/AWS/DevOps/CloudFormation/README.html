CloudFormation — AWS CloudFormation

Category: DevOps
AWS Service Type: Infrastructure as Code
Key Docs: CloudFormation User Guide

Overview

CloudFormation is AWS's native infrastructure-as-code service for modeling and provisioning AWS resources as declarative templates. Stacks create, update, and delete groups of resources together while tracking drift and dependency order. CloudFormation integrates with IAM, Change Sets, StackSets, SSM parameters, nested stacks, macros, and CI/CD systems. It is central to repeatable, reviewable AWS delivery because it converts console-driven infrastructure into versioned change control.

How It Works

Core Mechanics

Concept Description
Template YAML or JSON document declaring resources and configuration
Stack Live deployed instance of a template
Change Set Preview of stack modifications before execution
Stack policy Guardrail preventing updates to protected resources
StackSet Multi-account, multi-region deployment of a stack
Drift detection Check for out-of-band changes versus template state
Nested stack Reusable sub-template deployed as part of a parent stack
Deletion policy Resource behavior on stack deletion, such as retain or snapshot

What Can Go Wrong

2 No Change Set review Risky updates applied blindly Destructive or unexpected changes Med
3 Stack delete destroys data Retain or snapshot policies absent Data loss Med
4 Broad deployment role Pipeline can create anything everywhere Large blast radius on compromise High
5 Failed rollback traps stack Update failure leaves stack in bad state Operational delay Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Manual drift Restrict console mutation and run periodic drift detection Change governance
2 No Change Set review Require reviewed Change Sets for production updates Deployment safety
3 Stack delete destroys data Use DeletionPolicy and UpdateReplacePolicy intentionally Data protection
4 Broad deployment role Scope pipeline roles and use permission boundaries or SCPs IAM control
5 Failed rollback traps stack Test changes in lower envs and break large stacks into modules Release engineering
6 Template sprawl Use nested stacks or modules with clear ownership Template architecture
7 Drift ignored Alert on drift and reconcile before major updates Ops hygiene
8 Secrets in templates Reference Secrets Manager or SSM, not raw secret values Secret handling

Top 3 Critical Mitigations

  1. Keep infrastructure changes in code only: CloudFormation loses much of its value when console changes are allowed to accumulate.
  2. Use Change Sets and data-retention policies: The most painful failures are accidental destructive updates.
  3. Scope the deployment role tightly: IaC pipelines are privileged automation and should not default to account-admin power.

1. How does CloudFormation compare to Terraform in AWS-heavy environments?

Direct answer

In AWS-heavy environments, CloudFormation is the native AWS infrastructure-as-code service, while Terraform is a multi-cloud, provider-based IaC tool with a very large ecosystem and strong workflow ergonomics.

I would choose CloudFormation when the organization wants AWS-native integration, no external state backend, deep service support from AWS, StackSets, and native governance. I would choose Terraform when the organization needs multi-cloud support, broad third-party provider support, mature module ecosystems, and a consistent IaC workflow across AWS, SaaS, Kubernetes, GitHub, Datadog, etc.

Comparison

Area CloudFormation Terraform
Ownership AWS-native service HashiCorp/OpenTofu ecosystem
Best fit AWS-only or AWS-first platforms Multi-cloud / multi-provider platforms
State Managed by CloudFormation Terraform state file/backend
Language YAML/JSON HCL
Drift detection Native stack drift detection terraform plan detects many differences against state/provider
Change preview Change Sets Plan
Multi-account deployment StackSets Workspaces/modules/pipelines/provider aliases
Rollback Built-in stack rollback Depends on workflow and state
AWS new feature support Often good through native service Depends on provider support
Non-AWS resources Limited/custom resources Strong provider ecosystem
Governance AWS Config, StackSets, IAM, CloudTrail Policy-as-code, state controls, CI/CD
Operational risk Large stack rollback complexity State corruption/leakage risk

AWS-heavy decision

For a pure AWS enterprise:

CloudFormation is strong for:
  account baseline
  guardrails
  StackSets
  AWS-native governance
  service-managed deployments

Terraform is strong for:
  platform modules
  multi-provider workflows
  developer experience
  reusable patterns across AWS and non-AWS tools

Real-world example

A company may use both:

CloudFormation StackSets:
  organization-wide baseline
  IAM roles
  AWS Config
  GuardDuty
  Security Hub
  CloudTrail
  VPC endpoints

Terraform:
  application infrastructure
  EKS add-ons
  Datadog monitors
  GitHub repositories
  DNS across multiple providers

Answer

“I would not frame it as CloudFormation versus Terraform religiously. In AWS-heavy environments, CloudFormation is excellent for native AWS governance and StackSets, while Terraform is excellent for cross-platform orchestration and reusable modules. The main risk is not the tool; it is unmanaged drift, overprivileged deployment roles, weak review, and poor state/change governance.”

2. What are the risks of drift and how do you manage it?

Direct answer

Drift happens when the real infrastructure no longer matches the IaC template. This usually happens because someone changes resources manually through the console, CLI, SDK, or another automation path.

AWS CloudFormation drift detection identifies stack resources whose actual configuration differs from the expected template configuration. AWS also notes that out-of-band changes can complicate future stack updates or deletions. (AWS Documentation)

Why drift is dangerous

Drift type Risk
Security group opened manually Public exposure
IAM policy expanded manually Privilege escalation
RDS deletion protection disabled Data loss risk
Lambda env var changed manually Runtime inconsistency
S3 bucket policy changed Data leakage
Auto Scaling group changed Capacity surprise
KMS key policy changed Backup/restore failure
Manual hotfix not captured Next deployment overwrites it

Example incident

Production CloudFormation template:
  RDS publiclyAccessible = false

Manual console change:
  RDS security group allows 0.0.0.0/0 on 5432

Next deployment:
  CloudFormation update focuses on tags only.
  Team does not notice security group drift.
  Database remains exposed.

Drift management strategy

Prevent → Detect → Reconcile → Learn

Prevent drift

Control Purpose
Deny console changes for production Force IaC path
Use CI/CD deployment roles Centralize changes
SCPs Block risky out-of-band actions
Permission boundaries Limit human/admin changes
Break-glass workflow Allow emergency changes with audit
Tags/ownership Know who owns resources

Detect drift

Scheduled CloudFormation drift detection
AWS Config rules
CloudTrail alerts for manual changes
Security Hub findings
Periodic change review
Drift-aware change sets

CloudFormation now supports drift-aware change sets that compare templates with actual stack state and help bring drifted resources back into sync safely. (AWS Documentation)

Reconcile drift

Options:

1. Revert the manual change to match template.
2. Update the template to codify the intended change.
3. Import/adopt the resource if it is unmanaged.
4. Replace the resource if drift cannot be safely reconciled.

Answer

“Drift is dangerous because IaC stops being the source of truth. I manage it by preventing manual changes, detecting drift continuously, reconciling intentionally, and treating emergency console changes as incidents that must be codified or reverted.”

3. Why are Change Sets valuable for production deployments?

Direct answer

CloudFormation Change Sets let you preview what CloudFormation intends to change before executing an update. They are valuable in production because they show whether a change will add, modify, delete, or replace resources.

AWS says Change Sets allow you to preview how proposed stack changes might impact running resources, including changes to resource properties and attributes. (AWS Documentation)

Why this matters

A small-looking template change can cause a destructive replacement.

Example:

DBSubnetGroupName: old-subnet-group

changed to:

DBSubnetGroupName: new-subnet-group

Could require resource replacement depending on the resource and property.

Change Set review checklist

Before production execution, review:

Resources to be deleted
Resources to be replaced
IAM policy changes
Security group changes
KMS key changes
RDS/EFS/S3 changes
Custom resource behavior
Outputs being removed
Dependency ordering
Expected interruption level

Good production flow

Developer opens PR
   ↓
CI validates template
   ↓
CloudFormation creates Change Set
   ↓
Reviewer inspects blast radius
   ↓
Security/owner approval for high-risk changes
   ↓
Execute Change Set
   ↓
Monitor stack events and app health

Example

Change Set shows:

Modify: AWS::IAM::Policy
Replacement: False

Replace: AWS::RDS::DBInstance
Replacement: True

Delete: AWS::S3::Bucket

This should block the deployment until reviewed.

Senior answer

“A Change Set is production blast-radius visibility. It does not prove a change is safe, but it tells me when a harmless-looking diff may replace a database, delete a bucket, or widen an IAM policy.”

4. How would you structure stacks for a large multi-team platform?

Direct answer

I would structure stacks by lifecycle, ownership, blast radius, and dependency direction. I would avoid one giant stack for everything and also avoid thousands of tiny stacks without governance.

Good stack boundaries

Stack type Owned by Examples
Organization baseline Cloud/platform/security CloudTrail, Config, GuardDuty, Security Hub
Network foundation Network/platform VPC, subnets, route tables, endpoints
Shared security Security/platform KMS keys, IAM boundaries, logging buckets
Shared platform Platform EKS/ECS clusters, ALB, service mesh
Data foundation Data/platform RDS, DynamoDB, Redshift, S3 data buckets
App stack Product team Lambda, ECS service, queues, APIs
Monitoring stack SRE/platform alarms, dashboards, log groups
Per-environment stack Team/platform dev/test/prod isolation

Dependency direction

Organization baseline
   ↓
Network
   ↓
Shared security
   ↓
Platform services
   ↓
Application services
   ↓
Observability

Avoid circular dependencies like:

App stack exports value to network stack
Network stack exports value to app stack

Large platform example

account-baseline-stackset
network-prod-stack
security-kms-prod-stack
eks-platform-prod-stack
shared-observability-prod-stack
payment-api-prod-stack
order-api-prod-stack
reporting-prod-stack

Cross-stack references

Use carefully:

Exports/imports for stable platform values
SSM Parameter Store for loosely coupled references
CloudFormation parameters for explicit dependency injection

Avoid exporting values that change frequently, because it can make stacks hard to update.

Multi-team governance

Platform team owns base modules.
App teams consume approved nested stacks/modules.
Security team owns guardrail stacks.
CI/CD enforces templates, tags, and policies.

Answer

“I structure stacks so that a payment API deployment cannot accidentally replace a VPC, delete a KMS key, or change organization-wide logging. Stack boundaries should follow ownership and blast radius.”

5. What controls prevent CloudFormation from destroying persistent data?

Direct answer

Use DeletionPolicy, UpdateReplacePolicy, stack policies, deletion protection at the service level, IAM/SCP guardrails, backups, and production Change Set approval.

CloudFormation’s DeletionPolicy can preserve or back up a resource when the stack is deleted; if no DeletionPolicy is specified, CloudFormation deletes the resource by default. (AWS Documentation) Stack policies can prevent stack resources from being unintentionally updated or deleted during a stack update. (AWS Documentation)

Persistent data resources needing protection

RDS databases
DynamoDB tables
S3 buckets
EFS file systems
OpenSearch domains
Redshift clusters
KMS keys
EBS volumes
Backup vaults
Secrets

Important controls

1. DeletionPolicy

MyDatabase:
  Type: AWS::RDS::DBInstance
  DeletionPolicy: Snapshot
  UpdateReplacePolicy: Snapshot

Options commonly include:

Retain
Snapshot
Delete

Use:

Retain for S3 buckets, DynamoDB tables, KMS keys
Snapshot for RDS/Redshift/EBS where supported

2. UpdateReplacePolicy

Protects against replacement during update.

UpdateReplacePolicy: Retain

This matters because a resource can be replaced even if the stack itself is not deleted.

3. Stack policy

Deny updates/replacements/deletes to critical resources unless explicitly overridden.

Deny Update:Delete
Deny Update:Replace
on RDS, DynamoDB, KMS, S3

4. Service-level deletion protection

Examples:

RDS deletion protection
DynamoDB deletion protection
S3 bucket versioning/object lock where appropriate
Backup vault lock
KMS deletion window and restricted scheduling

5. CI/CD guardrails

Block production Change Sets that include:

Delete: AWS::RDS::DBInstance
Replace: AWS::DynamoDB::Table
Delete: AWS::KMS::Key
Delete: AWS::S3::Bucket

unless approved through exception workflow.

Answer

“For persistent data, I assume humans will eventually make a bad template change. I protect data with DeletionPolicy, UpdateReplacePolicy, stack policies, service deletion protection, backups, and CI/CD approval gates.”

6. How would you secure a CI/CD role that deploys infrastructure?

Direct answer

I would treat the CI/CD deployment role as a high-privilege production identity and secure it with least privilege, trust restrictions, permission boundaries, approval gates, session controls, logging, and separation between plan and apply.

Threat model

A compromised CI/CD role can:

Create admin IAM roles
Disable logging
Open security groups
Delete databases
Exfiltrate secrets
Modify KMS policies
Deploy malicious Lambda/container code
Create persistence across accounts

Secure deployment architecture

Developer PR
   ↓
CI validate/lint/test
   ↓
Plan / Change Set role - read/preview limited
   ↓
Approval
   ↓
Deploy role - scoped write permissions
   ↓
CloudFormation service role - constrained by boundary
   ↓
CloudTrail/Security Hub monitoring

Controls

Control Purpose
OIDC federation from CI provider Avoid long-lived CI keys
Restrict trust policy Only specific repo/branch/workflow can assume role
Separate dev/test/prod roles Prevent lateral blast radius
Separate plan and deploy roles Reduce accidental writes
Permission boundaries Prevent privilege escalation
CloudFormation service role Constrain stack operations
SCPs Block dangerous org-wide actions
Change Set approval Human review for prod
Session duration limits Reduce token abuse window
CloudTrail alerts Detect unusual deployment actions
Artifact signing Ensure trusted templates/packages
No secrets in CI logs Prevent credential leakage

Trust policy example idea

Only allow GitHub Actions from one repo and branch:

sub = repo:company/payment-api:ref:refs/heads/main
aud = sts.amazonaws.com

Critical IAM guardrails

Deny deployment role from:

iam:CreateUser
iam:CreateAccessKey
iam:PutRolePolicy with AdministratorAccess
kms:ScheduleKeyDeletion
cloudtrail:StopLogging
guardduty:DeleteDetector
config:StopConfigurationRecorder
organizations:LeaveOrganization

unless explicitly required and separately controlled.

Answer

“A CI/CD role is effectively a production administrator. I secure it with identity federation, narrow trust, least privilege, permission boundaries, Change Set approvals, CloudTrail monitoring, and explicit denial of security-control tampering.”

7. What are the failure modes of rollback in large stacks?

Direct answer

CloudFormation rollback can fail or become dangerous when stacks are large, resources are stateful, dependencies are complex, replacements are partially complete, custom resources fail, quotas are reached, or external systems are changed outside CloudFormation.

Common rollback failure modes

Failure mode Example
Stateful replacement RDS replacement cannot be safely rolled back
Deletion blocked S3 bucket not empty
Quota exhausted Rollback tries to recreate old resource but quota unavailable
Dependency mismatch New resource depends on partially deleted old resource
Custom resource failure Lambda-backed custom resource fails rollback
External side effects DNS, third-party SaaS, or DB migration already changed
Drift Actual state differs from template
Long-running updates Stack stuck in rollback state
IAM permission failure Rollback role lacks permission to restore old state
Name collision Old physical resource name unavailable
Data migration mismatch Schema migration not backward compatible

Example

Stack update replaces an RDS instance.
New DB is created.
App runs schema migration.
Deployment fails later on ALB config.
CloudFormation rolls back.
Old app points back to old DB.
But data/schema changed on new DB.
Rollback is not logically safe.

Large-stack risk

The bigger the stack, the more unrelated resources are coupled:

One IAM policy error
   ↓
Rollback affects ECS, ALB, RDS, Lambda, Route53
   ↓
Many teams impacted

Mitigations

Smaller stacks by lifecycle/blast radius
Change Sets
Stack policies
DeletionPolicy/UpdateReplacePolicy
Blue/green deployments
Backward-compatible DB migrations
Custom resource idempotency
Preflight quota checks
Manual approval for replacement/delete
Rollback playbooks

Senior answer

“CloudFormation rollback is infrastructure rollback, not always business rollback. If the deployment changes data, schema, traffic, or external systems, automatic rollback may not be sufficient or safe.”

8. How would you deploy standardized infrastructure across many accounts with StackSets?

Direct answer

I would use CloudFormation StackSets with AWS Organizations service-managed permissions to deploy standardized baselines across accounts and Regions. I would use delegated administrator accounts, target OUs, controlled concurrency, failure tolerance, drift detection, and versioned templates.

AWS Organizations integration allows CloudFormation StackSets to create, update, or delete stacks across multiple AWS accounts and Regions with a single operation using service-managed permissions and service-linked roles. (AWS Documentation) AWS also supports delegated administrator accounts for StackSets so a member account can create and manage StackSets with service-managed permissions. (AWS Documentation)

Good use cases for StackSets

CloudTrail organization trail support resources
AWS Config recorder/rules
GuardDuty/Security Hub enablement resources
IAM baseline roles
VPC baseline components
Logging buckets or forwarding
SCP-supporting roles
Security notification topics
Config aggregators

StackSet architecture

Management account or delegated admin
   ↓
StackSet definition
   ↓
Target OU: Production
   ↓
Accounts:
  account A / region ap-southeast-1
  account B / region ap-southeast-1
  account C / region us-east-1

Deployment strategy

1. Create StackSet in delegated admin account.
2. Use service-managed permissions.
3. Target OUs, not hand-maintained account lists where possible.
4. Start with dev/security sandbox OU.
5. Roll to non-prod OU.
6. Roll to prod OU with low concurrency and failure tolerance.
7. Enable automatic deployment for new accounts in OUs.
8. Run periodic drift detection.

CloudFormation StackSets support drift detection to determine whether stack instances differ from expected configuration. (AWS Documentation)

Operational controls

Control Purpose
Delegated admin Avoid overusing management account
OU targeting Scalable account selection
Parameter overrides Account/Region-specific values
Low concurrency in prod Reduce blast radius
Failure tolerance Stop bad rollout early
Versioned templates Repeatability
Drift detection Detect manual account changes
StackSet operation logging Audit
SCP guardrails Prevent bypass/tampering

Answer

“StackSets are ideal for organization-wide baselines, but they are also high-blast-radius deployment mechanisms. I would use delegated administration, OU targeting, staged rollout, low concurrency, failure tolerance, drift detection, and strong approval controls.”

9. How do you keep secrets out of templates and stack outputs?

Direct answer

Do not put secret values in templates, parameters, metadata, outputs, mappings, or CI logs. Use Secrets Manager, SSM Parameter Store secure strings, dynamic references, and application runtime retrieval.

CloudFormation dynamic references let templates reference values stored in services like Systems Manager Parameter Store or Secrets Manager. AWS says CloudFormation retrieves the value when needed and does not store the actual reference value. (AWS Documentation) AWS recommends storing credentials in Secrets Manager and using versionless dynamic references to support rotation workflows. (AWS Documentation)

Bad pattern

Parameters:
  DBPassword:
    Type: String
    Default: SuperSecret123

Outputs:
  DatabasePassword:
    Value: !Ref DBPassword

This leaks secrets through:

template repository
CloudFormation stack parameters
stack outputs
CI logs
change sets
console visibility

Better pattern

Resources:
  DBSecret:
    Type: AWS::SecretsManager::Secret
    Properties:
      Name: prod/payment/db
      GenerateSecretString:
        SecretStringTemplate: '{"username":"payment_app"}'
        GenerateStringKey: password
        PasswordLength: 32
        ExcludePunctuation: true

For resources that need the secret:

MasterUserPassword: "{{resolve:secretsmanager:prod/payment/db:SecretString:password}}"

Even better application pattern

CloudFormation stores only the secret ARN:

Environment:
  Variables:
    DB_SECRET_ARN: !Ref DBSecret

Application retrieves secret at runtime using its IAM role.

Rules

Never output secret values.
Output ARNs, names, or references only.
Use NoEcho for sensitive parameters, but do not rely on it alone.
Do not log rendered templates.
Do not commit generated templates with resolved values.
Avoid secrets in user data.
Avoid secrets in metadata.
Scan templates and CI logs for secrets.

Answer

“Secrets should be referenced by CloudFormation, not embedded in CloudFormation. The template should create or point to a secret, and applications should retrieve it through least-privileged IAM.”

10. How would you migrate a console-built environment into CloudFormation safely?

Direct answer

I would migrate in stages: inventory resources, classify ownership, create templates from actual state, import supported resources, validate with drift detection and Change Sets, then lock down manual changes. I would not try to import everything at once.

Migration phases

Phase 1: Discovery

Collect:

Accounts and Regions
VPCs/subnets/security groups
IAM roles/policies
RDS/DynamoDB/S3/KMS
ECS/EKS/Lambda/API Gateway
Route 53/CloudFront/ALB
Tags and owners
Dependencies
Manual scripts
Known exceptions

Tools:

AWS Config
CloudTrail
Resource Explorer
Tag Editor
Service consoles
Existing diagrams/runbooks

Phase 2: Classify resources

Class Action
Critical persistent data Import carefully with Retain policies
Stateless app resources Recreate or import
Shared network/security Import into foundation stacks
Unknown/orphaned Investigate before managing
Deprecated Retire, do not codify
External dependencies Document manually

Phase 3: Build templates

Start with low-risk resources:

security groups
IAM roles
log groups
alarms
SQS queues
Lambda configuration

Be more careful with:

RDS
S3
KMS
Route 53
CloudFront
production ALBs

Phase 4: Import resources

CloudFormation supports resource import for many resource types, but the template must match the existing resource configuration closely.

Safe import approach:

1. Create template matching existing resource.
2. Add DeletionPolicy: Retain for persistent resources.
3. Import resource into stack.
4. Run drift detection.
5. Create no-op Change Set.
6. Confirm CloudFormation sees no destructive update.

Phase 5: Validate

Drift detection clean
No-op Change Set clean
Application still healthy
Tags/outputs correct
Backups intact
Monitoring intact

Phase 6: Enforce IaC going forward

Deny manual production changes except break-glass.
Require PR + Change Set.
Schedule drift detection.
Monitor CloudTrail for console changes.

Example migration order

1. Observability resources
2. IAM app roles
3. Security groups
4. Stateless compute
5. Load balancers/listeners
6. Data resources last
7. Network foundations carefully or separately

Major risk

If the generated/imported template does not exactly match reality, the first “update” after import may unintentionally replace or modify resources.

Mitigation:

No-op Change Set after import
Stack policy
DeletionPolicy/UpdateReplacePolicy
Manual review of replacements
Production maintenance window for first update

Answer

“Migrating console-built infrastructure into CloudFormation is not just reverse engineering. It is taking ownership safely. I would import gradually, protect data resources, prove no-op updates, then make IaC the only allowed change path.”

Controls mapped to risks

Risk Control
CI/CD compromise OIDC, narrow trust, permission boundaries, approvals
Destructive update Change Sets, stack policies, DeletionPolicy, UpdateReplacePolicy

Summary

The senior-level answer is:

“CloudFormation gives AWS-native infrastructure governance, but production safety comes from how you use it: stack boundaries, Change Sets, drift detection, deletion protection, least-privileged deployment roles, and secret hygiene.”

My production checklist:

Use CloudFormation or Terraform consistently as source of truth.
Use Change Sets before production updates.
Run drift detection and reconcile intentionally.
Structure stacks by ownership, lifecycle, and blast radius.
Protect persistent data with DeletionPolicy and UpdateReplacePolicy.
Use stack policies for critical resources.
Secure CI/CD roles like production admin identities.
Use StackSets carefully for multi-account baselines.
Keep secrets in Secrets Manager or SSM, not templates or outputs.
Import console-built resources gradually and validate no-op updates.

Key Takeaways

  • CloudFormation turns infrastructure into a reviewable, repeatable delivery artifact.
  • Drift and overly broad deployment permissions are the main operational risks.
  • Change Sets and retention policies materially reduce destructive-update risk.
  • Stack design affects velocity and failure recovery.
  • IaC only works when it becomes the authoritative path for change.
