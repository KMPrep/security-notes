ECR — Amazon Elastic Container Registry

Category: DevOps
AWS Service Type: Container Image Registry
Key Docs: ECR User Guide

Overview

ECR is AWS's managed OCI-compatible container image registry for storing, versioning, and distributing container images. It integrates with ECS, EKS, Lambda container images, CodeBuild, Inspector, and lifecycle policies. ECR supports private repositories, public repositories, replication, immutable tags, image scanning, and fine-grained IAM access. It is a critical supply-chain component because every container deployment depends on the integrity and availability of images stored there.

2 No image scanning Vulnerable base images promoted unchecked Known CVEs in production High

4 Broad push permissions Untrusted pipeline or user can overwrite release image Malicious deployment path High
5 No lifecycle policy Registry accumulates stale images Cost growth and clutter Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Mutable tags in production Enable immutable tags and use release-version tags Registry governance
2 No image scanning Enable scanning and add gating in CI/CD Supply-chain security
3 Broad pull permissions Scope repo access by role and account purpose Access control
4 Broad push permissions Limit push to trusted build identities only CI/CD security
5 No lifecycle policy Apply lifecycle cleanup rules for old or untagged images Registry hygiene

7 Runtime pulls by tag not digest Deploy by digest in ECS/EKS/Lambda where possible Deployment correctness
8 Scan findings ignored Define remediation SLAs and block high-severity releases Vulnerability management

Top 3 Critical Mitigations

  1. Use immutable tags and preferably digests: Tags are for humans; digests are for correctness.
  2. Restrict push rights tightly: Whoever can push to ECR can influence production containers.
  3. Integrate scanning into release decisions: Visibility without release gating leaves vulnerable images in circulation.

1. Why are immutable tags and digests important in container supply chains?

Direct answer

Immutable tags and image digests protect the container supply chain from image substitution.

A container tag is a human-friendly pointer like:

payment-api:1.8.2
payment-api:prod
payment-api:latest

A digest is the cryptographic content address of the image:

payment-api@sha256:ab12cd34...

If tags are mutable, an attacker or careless user can overwrite the image behind a trusted tag. ECR supports image tag immutability so tags cannot be overwritten after they are pushed. AWS also added tag mutability exclusion modes, allowing repositories to make most tags immutable while allowing selected tag patterns to remain mutable where needed. (AWS Documentation)

Why mutable tags are dangerous

Bad pattern:

Deployment manifest:
  image: 123456789012.dkr.ecr.ap-southeast-1.amazonaws.com/payment-api:prod

Attack:

Attacker gets ECR push permission
   ↓
Pushes malicious image with tag prod
   ↓
Next deployment pulls payment-api:prod
   ↓
Malicious image runs in production

The deployment did not change, but the image behind the tag changed.

Why digests matter

A digest identifies the exact image content.

payment-api@sha256:7f3a...

If someone changes the tag later, the digest still points to the original image content.

Tag vs digest

Reference type Example Risk
Mutable tag payment-api:latest Can change silently
Version tag payment-api:1.8.2 Safer, but still mutable unless enforced
Immutable tag payment-api:1.8.2 Safer because tag cannot be overwritten
Digest payment-api@sha256:... Strongest deploy reference

Production rule

Use tags for humans:

payment-api:1.8.2
payment-api:git-abc123
payment-api:build-9821

Use digests for deployment:

image: payment-api@sha256:ab12cd34...

Answer

“Immutable tags prevent tag reuse. Digests prevent image substitution. In production, I want the deployment to reference the exact artifact that passed testing, not whatever a tag points to at deploy time.”

2. How would you secure an ECR repository used by many deployment platforms?

Direct answer

I would secure it with least-privileged IAM, repository policies, tag immutability, image scanning, artifact signing, network controls, encryption, lifecycle governance, replication rules, and audit logging.

Reference model

CI build platform
   ↓ push only to build repositories
ECR central artifact account
   ↓ replicated to target Regions/accounts
Deployment platforms
   ↓ pull only by digest
ECS / EKS / Lambda

Security controls

Area Control
Push access Only trusted build roles can push
Pull access Only deployment/runtime roles can pull
Tag mutability Immutable for release tags
Scanning Basic or enhanced scanning enabled
Signing Only signed images promoted
Encryption KMS-managed encryption where required
Network Use VPC endpoints/PrivateLink where applicable
Repository policy Cross-account pull/push rules
IAM policy Identity-level permission control
Lifecycle policy Clean old images safely
CloudTrail Audit push/delete/policy changes
EventBridge React to vulnerability findings
SCPs Prevent public or unmanaged repositories

Example repository separation

app/payment-api-build
  used by CI to push candidate images

app/payment-api-release
  only promotion pipeline can push signed release images

app/payment-api-prod
  production deployment pulls only approved digests

Push role should not deploy

BuildRole:
  ecr:PutImage
  ecr:UploadLayerPart
  ecr:CompleteLayerUpload

No:
  ecs:UpdateService
  eks:*
  cloudformation:UpdateStack

Deployment role should not push

ProdDeployRole:
  ecr:BatchGetImage
  ecr:GetDownloadUrlForLayer
  ecs:UpdateService

No:
  ecr:PutImage
  ecr:DeleteRepository
  ecr:BatchDeleteImage

Dangerous permissions

Be careful with:

ecr:PutImage
ecr:BatchDeleteImage
ecr:DeleteRepository
ecr:PutImageTagMutability
ecr:SetRepositoryPolicy
ecr:PutLifecyclePolicy
ecr:ReplicateImage

Senior answer

“For a shared ECR repository, I separate push, promote, and pull identities. Build systems can create artifacts, promotion systems can approve them, and production runtimes can only pull the exact approved digest.”

3. What should happen when ECR scanning finds a critical vulnerability?

Direct answer

A critical vulnerability finding should trigger an automated and human-visible vulnerability response workflow. The response depends on whether the image is not deployed, staged, or already running in production.

Amazon ECR image scanning identifies software vulnerabilities in container images. With enhanced scanning, Amazon Inspector scans ECR images and emits EventBridge events when findings are created, updated, or closed. (AWS Documentation)

Response workflow

Critical finding detected
   ↓
EventBridge rule fires
   ↓
Security automation enriches finding
   ↓
Classify image:
      not deployed?
      deployed in dev?
      deployed in prod?
      internet-facing?
      exploitable package path?
   ↓
Block promotion or trigger remediation
   ↓
Notify service owner
   ↓
Track SLA
   ↓
Verify fixed image

Policy by stage

Image state Action
Built but not promoted Block promotion
In staging Block production release
In production, not internet-facing Risk assess, patch within SLA
In production, internet-facing/exploitable Emergency patch or rollback
Base image vulnerability Rebuild dependent images
False positive Document exception with expiry

Automated controls

Prevent new production deployment of images with critical findings.
Open ticket automatically.
Notify owner by repository tag.
Map digest to running ECS/EKS workloads.
Rebuild using patched base image.
Run tests and sign new image.
Deploy patched digest.
Close finding only after verification.

Important nuance

Do not blindly kill production workloads only because a scanner says “critical.” First determine:

Is the vulnerable package actually present?
Is it reachable/exploitable?
Is there a fixed version?
Is the image running?
Is there compensating control?
Is rollback safer or riskier?

Example

Finding:

CVE critical in openssl library
Image: payment-api@sha256:abc...
Running in production ECS service
Internet-facing API

Response:

1. Block further deployments of sha256:abc.
2. Rebuild base image with patched openssl.
3. Run regression/security tests.
4. Sign image sha256:def.
5. Canary deploy.
6. Full rollout.
7. Confirm old digest no longer running.
8. Retain old image only if needed for forensic/rollback decision.

Answer

“Critical ECR findings should become deployment policy, not just dashboard noise. New vulnerable images should be blocked; running vulnerable images should be prioritized based on exposure, exploitability, and business risk.”

4. How would you design multi-region image distribution?

Direct answer

I would use ECR cross-Region replication for release images, deploy from the local Region’s ECR registry, and pin deployments to digests. This reduces pull latency, avoids cross-region dependency during outages, and supports regional disaster recovery.

Amazon ECR supports both cross-Region and cross-account replication. Cross-account replication requires destination registry permissions allowing replication from the source registry. (AWS Documentation)

Reference architecture

Build Region / Artifact Account
   ↓
ECR release repository
   ↓ replication
ap-southeast-1 ECR
us-east-1 ECR
eu-west-1 ECR
   ↓
Regional ECS/EKS/Lambda deployments pull local image

Design principles

Requirement Design
Regional resilience Replicate images to every runtime Region
Fast pulls Pull from local ECR
Artifact integrity Verify digest/signature after replication
Account isolation Central artifact account + target accounts
Least privilege Runtime roles pull only
DR readiness Pre-replicate before traffic failover
Cost control Replicate release images, not every dev image
Governance Same scanning/signing policy per Region

ECR replication limits to know

AWS documentation says a private registry replication configuration can contain up to 10 rules, up to 25 unique destinations across all rules, and 100 filters per rule. (AWS Documentation)

Promotion flow

1. Build image once.
2. Scan image.
3. Sign image.
4. Promote digest to release repo.
5. ECR replicates digest to target Regions/accounts.
6. Deployment checks digest exists in local Region.
7. Deploy local regional URI by digest.

Example

Source:
111111111111.dkr.ecr.ap-southeast-1.amazonaws.com/payment-api@sha256:abc

Replicated:
222222222222.dkr.ecr.us-east-1.amazonaws.com/payment-api@sha256:abc
333333333333.dkr.ecr.eu-west-1.amazonaws.com/payment-api@sha256:abc

Same digest, regional registry.

Answer

“For multi-region workloads, image availability is part of DR. I do not want a us-east-1 outage or cross-region network issue to prevent ap-southeast-1 from starting containers.”

5. Tradeoffs of pulling by tag versus digest

Direct answer

Pulling by tag is more human-friendly and operationally convenient. Pulling by digest is more secure and reproducible. In production, I prefer using tags for discovery and release metadata, but deployments should resolve and pin to digests.

Comparison

Area Pull by tag Pull by digest
Human readability High Low
Reproducibility Weak if tag mutable Strong
Rollback clarity Ambiguous if tag reused Exact
Supply-chain security Weaker Stronger
Developer convenience High Lower
Promotion workflow Easier Requires digest tracking
Auditability Tag can lie Digest is evidence
Kubernetes/ECS usage Common Recommended for strict prod

Bad production example

image: payment-api:prod

This does not prove what code is running.

Better production example

image: payment-api@sha256:ab12cd34...

Add labels/annotations for readability:

annotations:
  gitCommit: "abc123"
  releaseVersion: "1.8.2"
  imageTag: "payment-api:1.8.2"
  imageDigest: "sha256:ab12cd34..."

Practical compromise

Build:
  tag = git SHA and build number

Promote:
  immutable release tag = 1.8.2

Deploy:
  digest = sha256:...

Audit:
  map digest ↔ tag ↔ commit ↔ build ↔ approver

Answer

“Tags are names. Digests are identities. I use tags for humans and digests for machines that deploy to production.”

6. How do ECR repository policies differ from IAM identity policies?

Direct answer

An ECR repository policy is a resource-based policy attached to a specific ECR repository. It controls who can access that repository.

An IAM identity policy is attached to an IAM user, group, or role. It controls what that identity can do across AWS resources.

AWS describes ECR repository policies as a subset of IAM policies scoped to individual ECR repositories, while IAM policies are generally used to apply permissions for the whole ECR service or other AWS services. (AWS Documentation)

Comparison

Area ECR repository policy IAM identity policy
Attached to ECR repository User/role/group
Policy type Resource-based Identity-based
Best for Cross-account access to a repository Granting identity permissions
Scope Specific repository One or many resources/actions
Principal specified? Yes No principal in policy; attached to identity
Common use Allow another account to pull images Let CI role push to repos
Evaluation Combined with IAM/SCP/resource policies Combined with resource/SCP/session policies

Example: repository policy

Allow production account to pull from central artifact account:

{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::222222222222:role/ProdEcsTaskExecutionRole"
  },
  "Action": ["ecr:BatchGetImage", "ecr:GetDownloadUrlForLayer"]
}

Example: IAM identity policy

Allow CI role to push to one repository:

{
  "Effect": "Allow",
  "Action": [
    "ecr:PutImage",
    "ecr:InitiateLayerUpload",
    "ecr:UploadLayerPart",
    "ecr:CompleteLayerUpload",
    "ecr:BatchCheckLayerAvailability"
  ],
  "Resource": "arn:aws:ecr:ap-southeast-1:111111111111:repository/payment-api"
}

Answer

“IAM policy answers: what can this role do? Repository policy answers: who can access this repository? In cross-account ECR designs, you often need both.”

7. How would you migrate from Docker Hub to ECR safely?

Direct answer

I would migrate using inventory, image pinning, vulnerability scanning, provenance validation, controlled mirroring, digest comparison, staged rollout, and fallback. I would avoid simply changing every image from docker.io/foo:latest to ECR without knowing exactly what content is being mirrored.

Migration phases

Phase 1: Inventory

Find all Docker Hub references:

Dockerfiles
Helm charts
Kubernetes manifests
ECS task definitions
Lambda container configs
CI workflows
base images
developer scripts

Classify:

application images
base images
third-party images
public dependencies
build-only images

Phase 2: Freeze and pin

Bad:

FROM node:latest

Better:

FROM node:20.11.1@sha256:...

Pin the exact upstream image digest before mirroring.

Phase 3: Mirror to ECR

Pull Docker Hub image by digest
   ↓
Scan locally or in CI
   ↓
Tag with source metadata
   ↓
Push to ECR
   ↓
Enable immutable tags
   ↓
Scan in ECR
   ↓
Sign mirrored image or attach provenance

Phase 4: Validate

Check:

source digest
ECR digest
SBOM
vulnerability scan
license risk
runtime behavior
image architecture
entrypoint/cmd

Phase 5: Cutover gradually

dev → staging → canary production → full production

Phase 6: Prevent regression

Block direct Docker Hub pulls in production:

Admission policy for EKS
ECS task definition policy checks
CI checks
network egress controls where appropriate
approved base-image registry list

Example policy

Deny production deployment if image registry is not:
  123456789012.dkr.ecr.ap-southeast-1.amazonaws.com

Risks

Risk Mitigation
Mirroring malicious upstream image Pin digest and validate provenance
Tag drift Immutable tags and digest deploys
Missing architecture Mirror amd64/arm64 variants
License exposure SBOM/license scan
New vulnerabilities ECR/Inspector scanning
Broken runtime Staged deployment
Developers still use Docker Hub Policy enforcement

Answer

“Docker Hub to ECR migration is a supply-chain migration, not just a registry migration. I would pin, mirror, scan, sign, deploy by digest, and enforce approved registries.”

8. What lifecycle policy strategy keeps ECR clean without deleting needed rollback images?

Direct answer

I would use a lifecycle strategy based on promotion stage, age, count, and rollback requirements. Release images need longer retention than build-candidate images. Production digests currently deployed or recently deployed must be protected.

ECR lifecycle policies expire images based on rules such as tag status, tag prefixes, count, and age. Lifecycle policies are useful, but bad policies can delete rollback images.

Recommended repository layout

payment-api-build:
  short retention
  keep last 50 or 14 days

payment-api-release:
  medium retention
  keep all releases for 90-180 days

payment-api-prod:
  long retention
  keep currently deployed + last N production releases

Retention model

Image type Retention
Untagged build layers Short, for example 7–14 days
PR/test images Short
Main branch candidate images Medium
Release images Longer
Production images Longest
Known rollback images Explicitly protected
Images under incident investigation Legal/forensic hold

Safe lifecycle policy principles

Never delete by age alone for production repos.
Keep last N successful production digests.
Keep all currently running digests.
Keep images referenced by active ECS task definitions or Kubernetes deployments.
Use tag prefixes to separate build/release/prod.
Treat untagged images carefully because digest-only deployments may appear untagged.

Dangerous policy

Delete all images older than 30 days

Problem:

A stable service has not deployed in 45 days.
Rollback image deleted.
Incident occurs.
Rollback unavailable.

Better strategy

Build repo:
  expire untagged > 7 days
  keep last 100 ci-* images

Release repo:
  keep last 50 release-* images
  expire non-prod > 90 days

Prod repo:
  keep last 20 prod-* images
  protect deployed digests

Answer

“Lifecycle policies should reflect release risk. I aggressively clean build garbage but conservatively retain production rollback images and any digest currently or recently deployed.”

9. How would you audit who pushed a bad image into production?

Direct answer

I would correlate ECR CloudTrail events, CI/CD logs, source control commits, artifact metadata, image digest, deployment records, and runtime state. The goal is to trace from running container digest back to who built, pushed, approved, and deployed it.

Evidence chain

Running workload
   ↓
Image digest
   ↓
ECR image push event
   ↓
CI build execution
   ↓
Git commit / PR
   ↓
Approver
   ↓
Deployment pipeline execution
   ↓
CloudTrail AssumeRole and deploy events

What to collect

image URI and digest
ECR PutImage CloudTrail event

role session name
source IP / CI runner identity
build ID
commit SHA
branch/tag
PR reviewers
artifact signature/provenance
scan results at time of deploy
deployment pipeline execution ID
approver
ECS task definition / EKS deployment revision
time first deployed
time first detected bad

AWS events to inspect

Example investigation

Production ECS task runs:

ecr:PutImage by arn:aws:sts::111111111111:assumed-role/CIBuildRole/build-9821

If audit chain is weak

immutable logs

Answer

“I should be able to answer who pushed a bad image by starting from the running digest and walking backward through ECR, CI, Git, approval, and deployment logs. If I cannot do that, my supply-chain evidence is insufficient.”

10. How would you build provenance and signing around ECR-based releases?

Direct answer

I would build a release process where every production image has: source identity, build identity, SBOM, vulnerability scan result, signature, immutable digest, approval record, and deployment record.

Provenance model

Source commit
   ↓
Protected branch / reviewed PR
   ↓
Trusted build environment
   ↓
Image build
   ↓
SBOM generated
   ↓
Image scanned
   ↓
Provenance attestation generated
   ↓
Image signed
   ↓
Pushed to ECR release repository
   ↓
Deployment verifies signature and policy
   ↓
Production pulls by digest

Provenance data

Attach or store:

git repository
commit SHA
branch/tag
PR number
builder identity
build timestamp
build workflow ID
base image digest
dependency lockfile hash
SBOM
SAST/SCA result
ECR image digest
approver
deployment environment

Signing options

Common patterns:

AWS Signer for container image signing where adopted
Sigstore/cosign signing
Notary-style signing depending on platform
Admission controller verification in Kubernetes
Pipeline verification before ECS/Lambda deploy

EKS enforcement

Admission controller:
  verify image comes from approved ECR registry
  verify signature
  verify provenance predicate
  reject unsigned or untrusted images

ECS enforcement

ECS does not have Kubernetes-style admission control, so enforce earlier:

CI/CD deploy gate:
  resolve image digest
  verify signature
  verify provenance
  verify scan policy
  register task definition only if approved

Example release policy

Production image must satisfy:

image registry = approved ECR account
reference = digest, not mutable tag
signature = valid
builder = approved CI workflow
source branch = main or release/*
commit = reviewed PR
critical vulnerabilities = none unless approved exception
SBOM = present
base image = approved base image

Attack scenario prevented

Attacker pushes image directly to ECR
   ↓
Image has no trusted signature
   ↓
Deployment gate rejects it

or:

Attacker modifies GitHub workflow in feature branch
   ↓
Build provenance says untrusted workflow/ref
   ↓
Production policy rejects artifact

Answer

“Provenance tells me where the image came from. Signing tells me it was not altered. Policy enforcement makes both meaningful by refusing to deploy images that lack trusted evidence.”

Controls and mitigations

Risk Control
Tag overwrite Immutable tags, deploy by digest
Unauthorized push Least-privilege push roles, repository policies
Bad image promotion Scanning, signing, provenance, approval gates
Critical CVE EventBridge/Inspector workflow, block promotion
Cross-account misuse Repository policies scoped to specific roles/accounts
Multi-region outage ECR cross-region replication and local pulls
Docker Hub drift Pin digest, mirror, scan, sign, enforce approved registries

Unsigned image deploy Admission/deploy-time signature verification

Summary

The senior-level answer is:

“ECR is not just a place to store Docker images. It is part of the production trust chain. I need to know exactly what image is running, who built it, what source produced it, whether it was scanned, whether it was signed, who approved it, and whether it can be reproduced or rolled back.”

My production checklist:

Enable immutable tags for release repositories.
Deploy production workloads by digest.
Separate build, release, and production repositories or permissions.
Allow only trusted CI roles to push.
Allow runtime/deployment roles to pull only.
Enable ECR scanning and automate critical finding response.
Use ECR cross-region/cross-account replication for DR.
Use repository policies for cross-account access and IAM policies for identity permissions.
Migrate from Docker Hub by pinning, mirroring, scanning, and signing.
Use lifecycle policies that keep current and recent production rollback digests.
Audit from running digest back to ECR push, CI build, commit, approval, and deploy event.
Require provenance and signature verification before production deployment.

Key Takeaways

  • ECR is a core container supply-chain control point.
  • Mutable tags are one of the most common causes of deployment ambiguity.
  • Push permissions deserve the same rigor as production deploy permissions.
  • Scanning must influence release policy to matter.
  • Replication and lifecycle policies improve both resilience and hygiene.
