EventBridge — Amazon EventBridge

Category: Messaging
AWS Service Type: Event Bus and Event Routing
Key Docs: EventBridge User Guide

Overview

EventBridge is AWS's managed event bus for routing events from AWS services, SaaS applications, and custom applications to targets such as Lambda, SQS, SNS, Step Functions, ECS, API destinations, and cross-account buses. It matches events using structured JSON patterns rather than simple topic subscriptions. EventBridge is commonly used for decoupled event-driven architectures, control-plane automation, scheduled jobs, and cross-account integration patterns. It is especially valuable when many producers and consumers need flexible filtering without tight coupling.

How It Works

Core Mechanics

Concept Description
Event bus Channel receiving AWS, partner, or custom events
Rule Pattern match plus target action
Event pattern JSON matching criteria on source, detail-type, account, fields, etc.
Target Lambda, SQS, Step Functions, API destination, event bus, and more
Archive Retains events for replay later
Replay Re-sends archived events to rules for reprocessing
Schema registry Catalog of event shapes for code generation and governance
API destination Outbound delivery to external HTTP APIs with connection config

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Rule too broad Match on exact source, detail-type, and scoped fields Rule design
2 No DLQ or archive Enable archives and DLQs for important rules Reliability and recovery
3 Cross-account bus trust too broad Restrict event-bus policies to explicit accounts and conditions Access control
4 Event contract drift Version event schemas and test consumer compatibility Contract governance
5 Idempotency ignored Make targets idempotent and dedupe by event ID or business key Target design

7 Sensitive data in event detail Publish minimal payloads and fetch details from source systems when needed Data minimization
8 No replay controls Rehearse replay procedures and isolate destructive targets Operations

Top 3 Critical Mitigations

  1. Keep rules narrow: EventBridge's flexibility is powerful, but loose matching creates unintended automation.
  2. Design event consumers as idempotent: Retries and replays are normal recovery mechanisms.
  3. Use archives for important buses: Replaying events is one of EventBridge's most valuable operational features.

1. How does EventBridge differ from SNS in routing and filtering behavior?

Direct answer

SNS is primarily pub/sub fanout. A publisher sends a message to a topic, and SNS delivers it to subscribers.

EventBridge is primarily an event bus and routing service. Producers put events onto an event bus, and EventBridge rules evaluate event patterns to decide which targets should receive each event. EventBridge patterns can match event source, metadata, and values inside the event detail. AWS describes EventBridge event patterns as a way to select subsets of events based on source, metadata, and detail values. (AWS Documentation)

Mental model

SNS:
  Publisher → Topic → Subscribers

EventBridge:
  Producer → Event Bus → Rules with event patterns → Targets

Comparison

Area SNS EventBridge
Primary model Topic-based pub/sub Event bus + rule-based routing
Routing unit Topic Event pattern
Filtering Subscription filter policies Event patterns on event envelope and detail
Event schema thinking Optional More central
SaaS/AWS service events Less central Strong fit
Many producers/consumers Possible Very strong
Archive/replay Not native in same way Native archive/replay
Scheduled events Not core feature EventBridge Scheduler/rules
Cross-account event bus Strong feature Strong governance concern
Best for Fanout notifications Event-driven integration/routing

Example

SNS works well for:

OrderCreated topic
  ├── Email service
  ├── SMS service
  └── Fraud service

EventBridge works well for:

Event bus receives:
  source = "payments.service"
  detail-type = "PaymentAuthorized"
  detail.amount = 1500
  detail.currency = "SGD"

Rules:
  high-value-payment-rule → fraud target
  all-payment-events-rule → data lake target
  failed-payment-rule → operations target

Practical difference

In SNS, the publisher usually decides the topic:

Publish to arn:aws:sns:...:payment-events

In EventBridge, the event content drives routing:

{
  "source": "com.company.payments",
  "detail-type": "PaymentFailed",
  "detail": {
    "tenantId": "t-123",
    "amount": 1200,
    "reason": "INSUFFICIENT_FUNDS"
  }
}

Answer

“SNS is excellent when I know the fanout topic. EventBridge is better when I want a central event bus where many consumers independently subscribe using content-based routing rules.”

2. What are the risks of broad event patterns?

Direct answer

Broad event patterns match more events than intended. That can cause data leakage, unexpected automation, duplicate processing, cost spikes, event loops, noisy alerts, and security incidents.

AWS explicitly recommends specifying content filters to make EventBridge patterns precise. (AWS Documentation)

Example of broad pattern

Bad:

{
  "source": [{ "prefix": "" }]
}

or:

{
  "source": ["aws.ec2"]
}

This may match too many EC2 events.

Better:

{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped"],
    "instance-id": ["i-0123456789abcdef0"]
  }
}

Risks

Risk Example
Over-triggering Lambda runs for thousands of irrelevant events
Cost spike Broad rule invokes targets excessively
Data leakage Events from one team/account sent to wrong target
Unauthorized automation Rule remediates resources it should not touch
Event loop Automation emits event that matches same rule
Noisy detection Security alerts become useless
Privilege misuse Target role acts on unintended resources
Replay danger Replaying broad archives re-triggers too much

Real-world scenario

A security automation rule is intended to quarantine EC2 instances tagged:

SecurityAction = Quarantine

But the event pattern only matches:

{
  "source": ["aws.ec2"]
}

Now every EC2 state change can trigger quarantine logic. One noisy deployment could accidentally isolate production instances.

Controls

Match source + detail-type + account + region.
Match specific detail fields.
Use allowlisted event sources.
Require schema validation.
Use target-side authorization checks.
Use dry-run mode for new rules.
Use EventBridge sandbox bus before production.
Use rule ownership tags.
Use rule reviews for broad patterns.

Answer

“A broad event pattern is like a broad IAM policy: convenient, but dangerous. I want precise matching and target-side validation because event routing mistakes can become production automation mistakes.”

3. How would you version event contracts safely across many teams?

Direct answer

I would treat events as public APIs. That means versioned schemas, backward-compatible changes, clear ownership, consumer-driven contract tests, schema registry, deprecation policy, and staged rollout.

Event contract example

{
  "source": "com.company.orders",
  "detail-type": "OrderCreated.v1",
  "detail": {
    "eventId": "evt-123",
    "tenantId": "t-001",
    "orderId": "ord-999",
    "customerId": "cust-123",
    "amount": 125.5,
    "currency": "SGD",
    "createdAt": "2026-05-12T09:30:00Z"
  }
}

Safe versioning rules

Change Safe? Example
Add optional field Usually safe Add couponCode
Add new enum value Risky Consumer may not handle it
Rename field Breaking customerId → buyerId
Remove field Breaking Remove currency
Change type Breaking amount string → number
Change semantics Breaking createdAt becomes local time
Split event Breaking unless dual-published OrderCreated split into OrderReserved

Recommended strategy

Use additive changes for v1.
Do not break existing fields.
For breaking changes, create v2 event.
Dual-publish v1 and v2 during migration.
Track consumers.
Set deprecation date.
Remove old version only after consumer migration.

Versioning options

Option 1: Version in detail-type

OrderCreated.v1
OrderCreated.v2

Good for routing clarity.

Option 2: Version in detail

{
  "detail-type": "OrderCreated",
  "detail": {
    "schemaVersion": "1.0"
  }
}

Good when consumers inspect payload.

Often I prefer both:

detail-type = OrderCreated.v1
detail.schemaVersion = 1

Governance model

Producer owns event schema.
Platform owns registry and standards.
Consumers own compatibility tests.
Security owns data classification rules.

Contract testing

Producer pipeline:
  validate event against schema
  run consumer contract tests
  block breaking changes

Consumer pipeline:
  test against sample events
  verify unknown optional fields are ignored

Answer

“Events are APIs with many hidden consumers. I would make event changes backward-compatible by default and use explicit v2 events plus dual-publish migration for breaking changes.”

4. How do archives and replay change incident recovery options?

Direct answer

Archives and replay let you recover from missed, failed, or buggy event processing by replaying historical events back onto the same event bus. This changes incident response from “we lost the event” to “we can repair the consumer and replay the affected window.”

AWS states that EventBridge archives let you store events and replay, or resend, them later to the event bus that originally received them. (AWS Documentation) AWS also states that replays can be sent only to the same event bus that emitted the events, and you can replay to all rules or selected rules. (AWS Documentation)

Without archive/replay

Buggy consumer fails for 2 hours
   ↓
Events are dropped or sent to DLQ
   ↓
Manual reconstruction from logs/database
   ↓
Slow, risky recovery

With archive/replay

Buggy consumer fails for 2 hours
   ↓
Events are archived
   ↓
Fix consumer
   ↓
Replay events from 10:00–12:00
   ↓
Consumer rebuilds missing state

Use cases

Use case Benefit
Consumer bug Replay affected events after fix
New feature validation Replay production-like historical events to new rule
Data backfill Rebuild derived state
Incident recovery Recover missed downstream processing
Audit investigation Inspect historical event patterns
Rule misconfiguration Replay events to corrected target

Risks

Risk Explanation
Duplicate processing Replayed events may already be partially processed
Side effects Replaying payment/email events can duplicate actions
Broad replay Too many rules triggered
Data retention Archive may contain sensitive data
KMS/access risk Archive encryption and access must be controlled
Cost Storage and replay can cost money
Ordering assumptions Replay does not mean global transaction reconstruction

Replay-safe consumer design

Every event has eventId.
Consumer stores processed eventId.
Side effects are idempotent.
External calls use idempotency keys.
Replays can target selected rules.
Dangerous targets can be disabled during replay.

Example

Bad replay target:

OrderCreated replay → send customer email again

Better:

OrderCreated replay → rebuild analytics projection
Email target disabled or idempotent by notificationId

Answer

“Archive and replay turn EventBridge into a recovery tool, but only if consumers are idempotent. Replay without idempotency can turn one incident into duplicate payments, duplicate emails, or repeated remediation.”

5. What are the security concerns of cross-account event buses?

Direct answer

Cross-account event buses create a trust boundary. If policies are too broad, another account can send unexpected events, create rules, or influence automation in your account. The main concerns are spoofed events, overbroad resource policies, confused deputy problems, data leakage, event injection, and weak target permissions.

EventBridge event buses use resource-based policies to grant additional permissions such as PutEvents, PutRule, and PutTargets to another account, and IAM conditions can scope access by organization, tags, rule, or account. (AWS Documentation) AWS also notes that new cross-account event bus targets require IAM roles for event delivery to another account, and recommends roles so SCP organization boundaries apply. (AWS Documentation)

Risk model

Account A sends events
   ↓
Central event bus in Account B
   ↓
Rules trigger automation
   ↓
Automation modifies resources

If Account A is compromised, can it trigger dangerous automation in Account B?

Security concerns

Concern Example
Event spoofing Attacker sends fake SecurityFindingClosed
Broad PutEvents Any account can send to central bus
Broad rules Event from wrong account triggers action
Sensitive event leakage Events forwarded to unauthorized account
Confused deputy EventBridge target role abused across accounts
Poor source validation Consumer trusts detail.accountId blindly
Replay abuse Replaying sensitive events to wrong rules
No schema enforcement Malformed events crash consumers

Secure design

Central event bus policy:
  allow PutEvents only from approved accounts/OUs
  require specific event source prefix
  require principal org ID where appropriate
  deny unapproved event sources

Target-side validation

Every consumer should verify:

event.account
event.source
event.detail-type
schemaVersion
signature or trusted origin where needed
tenant/account mapping

Do not trust only:

detail.accountId
detail.tenantId

because event details are producer-controlled.

Example pattern

{
  "source": ["com.company.security"],
  "account": ["111111111111", "222222222222"],
  "detail-type": ["SecurityFindingCreated.v1"]
}

Answer

“A cross-account bus is not just integration; it is a trust relationship. I would scope who can publish, validate event origin, limit rules and targets, and make automation verify authorization before acting.”

6. How would you prevent event loops in an automation-heavy environment?

Direct answer

I would prevent loops through event design, rule filtering, loop-detection metadata, idempotency, state transition checks, event source separation, and circuit breakers.

What is an event loop?

Rule sees event A
   ↓
Target updates resource
   ↓
Update emits event B
   ↓
Rule matches event B
   ↓
Target updates resource again
   ↓
Loop continues

Example

S3 bucket policy changed
   ↓
Security automation remediates policy
   ↓
Remediation emits another policy changed event
   ↓
Rule matches again
   ↓
Automation runs repeatedly

Controls

Control How it helps
Precise event patterns Avoid matching remediation events
Add origin metadata Ignore events created by automation
Add correlation ID Detect repeated chain
Idempotent remediation No-op if already compliant
State checks Act only if resource is non-compliant
Max retry/dead-letter Avoid infinite target retries
Circuit breaker Disable rule after abnormal invocation rate
Separate buses Separate raw events from remediation events
Event age limits Drop stale recursive events
Rule monitoring Alert on invocation spikes

Event metadata

{
  "source": "com.company.security-automation",
  "detail-type": "S3BucketRemediated.v1",
  "detail": {
    "resourceId": "bucket-123",
    "correlationId": "corr-abc",
    "origin": "auto-remediator",
    "remediationAttempt": 1
  }
}

Rule filter

Do not match automation-originated events:

Match:
  source = aws.s3
  detail.eventName = PutBucketPolicy

Ignore:
  source = com.company.security-automation

Target-side idempotency

def handler(event):
    bucket = event["detail"]["bucketName"]

    current = get_bucket_policy(bucket)
    if is_compliant(current):
        return "no-op"

    apply_compliant_policy(bucket)

Circuit breaker

If rule invocations > 100 in 5 minutes:
  disable rule
  alert on-call
  stop automated mutation

Answer

“Event-loop prevention cannot rely only on EventBridge filtering. The target must be idempotent and state-aware, because automation should remediate only when the current state actually requires it.”

7. When should you use EventBridge versus direct service integration?

Direct answer

Use EventBridge when you need loose coupling, many consumers, content-based routing, cross-account routing, archive/replay, auditability, or future extensibility.

Use direct service integration when the flow is simple, synchronous or tightly coupled, latency-sensitive, or when the producer knows exactly one target.

Decision table

Requirement Better fit
One producer, one known target Direct integration
Many independent consumers EventBridge
Content-based routing EventBridge
Cross-account event routing EventBridge
Need archive/replay EventBridge
Low-latency synchronous result Direct integration
Request/response workflow Direct API/Step Functions
Fanout with simple topic SNS
Durable work queue SQS
Service orchestration Step Functions
Scheduled tasks at scale EventBridge Scheduler

Example: direct integration

API Gateway → Lambda

Good when:

client expects immediate response
one target
simple request/response

Example: EventBridge

OrderCreated event
   ↓
EventBridge bus
   ├── billing service
   ├── fraud service
   ├── notification service
   ├── analytics pipeline
   └── audit archive

Good when:

producer should not know all consumers
new consumers may be added later
events need replay or audit

Anti-pattern

Do not use EventBridge just to call a single Lambda if direct invocation is enough and there is no routing, replay, or decoupling benefit.

Answer

“EventBridge is for decoupled event routing. Direct integration is for simple known flows. I choose EventBridge when I want producer-consumer independence and operational recovery through archive/replay.”

8. How would you design an organization-wide control plane on EventBridge?

Direct answer

I would design a central, governed EventBridge architecture that collects security, compliance, and operational events from all accounts, routes them to specialized targets, and enforces strict event contracts, account allowlists, least-privilege bus policies, and automation guardrails.

Reference architecture

AWS Organization accounts
  ├── prod app accounts
  ├── dev/test accounts
  ├── security tooling accounts
  └── shared services accounts
        ↓
Account-local event buses/rules
        ↓ cross-account PutEvents
Central Security/Operations Event Bus
        ↓
Rules:
  high severity security finding → incident queue
  IAM change → security automation
  public exposure → remediation workflow
  config drift → compliance system
  cost anomaly → FinOps workflow
        ↓
Targets:
  SQS
  Lambda
  Step Functions
  Security Lake/S3
  SIEM
  ChatOps
  ticketing

Key design choices

Area Design
Account onboarding StackSets deploy local forwarding rules
Central bus policy Allow only org accounts/OUs and approved sources
Event contracts Standard schema for security/ops events
Target isolation SQS buffers before automation
Automation safety Human approval for destructive remediation
Replay Archives enabled for central bus
Observability Per-rule metrics and DLQs

Event schema

  "detail-type": "ControlPlaneFinding.v1",
  "account": "111111111111",
  "region": "ap-southeast-1",
  "detail": {
    "findingId": "finding-123",

    "controlId": "S3_PUBLIC_BLOCK",

Automation pattern

Avoid direct destructive action:

EventBridge rule
   ↓
SQS queue
   ↓

   ↓
Validate finding still true
   ↓

   ↓

Controls

Bus resource policy scoped to AWS Organization.

Circuit breakers on high-volume rules.

Answer

“An organization-wide EventBridge control plane is powerful but dangerous. I would design it as a governed automation fabric with precise event contracts, account-scoped trust, idempotent workflows, approval gates, DLQs, replay, and circuit breakers.”

9. What observability do you need for EventBridge event-driven systems?

Direct answer

You need observability across the entire event path: producer, bus, rule matching, target invocation, target processing, retries, DLQs, archive/replay, and business-level outcomes.

10. How would you migrate ad hoc cron jobs into EventBridge scheduled rules?

Direct answer

I would inventory cron jobs, classify criticality, convert each job into a managed EventBridge Scheduler schedule or rule, define explicit targets and IAM roles, add retries/DLQs, centralize logging, and migrate gradually with parallel runs.

AWS says scheduled rules are now a legacy EventBridge feature and recommends EventBridge Scheduler for centrally managed scheduled tasks at scale. EventBridge Scheduler supports recurring cron/rate schedules, one-time invocations, flexible time windows, retry limits, and maximum retention for failed invocations. (AWS Documentation) EventBridge Scheduler can schedule millions of tasks and invoke many AWS services/API operations. (AWS Documentation)

Answer

“Migrating cron to EventBridge is not just moving schedule syntax. It is turning unmanaged scripts into owned, observable, retry-aware, least-privileged, idempotent scheduled workflows.”

Summary

The senior-level answer is:

“EventBridge is a routing and automation fabric. Its power comes from decoupling producers and consumers, but the risks are broad patterns, spoofed events, loops, contract breakage, and unsafe replay. I would design it with precise routing, versioned contracts, idempotent consumers, cross-account trust boundaries, DLQs, archives, and strong observability.”

My production checklist:

Use EventBridge when you need decoupled content-based routing.
Use SNS for simple topic fanout.
Keep event patterns precise.
Treat event schemas as APIs.
Version events safely with backward compatibility and dual-publish for breaking changes.
Enable archives/replay for recovery, but make consumers idempotent.
Secure cross-account buses with scoped resource policies and target-side validation.
Prevent loops with origin metadata, no-op checks, and circuit breakers.
Use EventBridge Scheduler for cron migration at scale.
Observe producer, bus, rule, target, DLQ, replay, and business outcome.

Key Takeaways

  • EventBridge is a flexible routing fabric for event-driven AWS systems.
  • Rule specificity and event-contract discipline determine whether it stays reliable.
  • Archives and replay are major operational advantages.
  • Cross-account event trust needs the same rigor as any other API trust.
