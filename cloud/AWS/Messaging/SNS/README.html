SNS — Amazon Simple Notification Service

Category: Messaging
AWS Service Type: Pub/Sub Messaging and Fan-Out
Key Docs: SNS Developer Guide

Overview

SNS is AWS's managed pub/sub service for fan-out messaging to multiple subscribers. A single published message can be delivered to SQS queues, Lambda functions, HTTP endpoints, email, SMS, mobile push, and other subscribers. SNS is commonly used to broadcast events, alerts, notifications, and workflow triggers across many downstream systems. It complements SQS: SNS fans out; SQS buffers and decouples. Together they form a common eventing backbone in AWS architectures.

How It Works

Core Mechanics

Concept Description
Topic Named event channel to which publishers send messages
Subscription Delivery path from a topic to an endpoint
Filter policy Attribute-based rule deciding which subscriber receives which message
FIFO topic Ordered SNS topic with deduplication semantics
Raw message delivery Sends payload without SNS envelope to supported endpoints
Delivery retry Managed retry behavior for supported protocols
Delivery status logs Logs for endpoint delivery attempts
Topic policy Resource policy controlling publish and subscribe permissions

2 No DLQ on subscriptions Failed deliveries disappear or retry inadequately Event loss Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Broad topic policy Scope publish and subscribe rights tightly Topic access control
2 No DLQ on subscriptions Configure DLQs where supported and monitor failures Reliability
3 Filter policy mistakes Test attribute schemas and subscription behavior in staging Event contract governance
4 Sensitive data in fan-out payload Minimize payload data and segregate topics by trust boundary Data classification
5 HTTP subscriber weak auth Use signed webhook validation or route via more trusted services Subscriber security
6 FIFO assumptions on Standard topic Use FIFO topics/queues only where ordering is required Design correctness
7 No delivery observability Enable delivery status logging and alarms Monitoring
8 SMS/email misuse Keep notifications low sensitivity and user-safe Content governance

Top 3 Critical Mitigations

  1. Treat topics as trust boundaries: Publishing is equivalent to influencing downstream behavior.
  2. Do not fan out sensitive payloads casually: More subscribers usually means more exposure.
  3. Validate delivery, not just publish success: SNS accepted a message does not mean every subscriber handled it.

1. How do SNS and SQS complement each other?

Direct answer

SNS is for fan-out notification.
SQS is for durable buffering and worker decoupling.

Together, they create a strong event-notification architecture:

Producer
   ↓
SNS topic
   ↓ fan-out
SQS queue A → Consumer A
SQS queue B → Consumer B
SQS queue C → Consumer C

SNS lets one producer notify many subscribers. SQS lets each subscriber process messages independently at its own speed.

Why SNS alone is not always enough

SNS can deliver directly to Lambda, HTTP, email, mobile push, or SQS. But if the subscriber is slow, down, throttled, or needs controlled processing, direct delivery can become fragile.

SQS gives each subscriber:

buffering
retry isolation
backpressure
dead-letter queue
independent scaling
independent failure handling

Why SNS + SQS is powerful

Need SNS role SQS role
Fan-out to many consumers Publishes once to many Each consumer gets own queue
Subscriber isolation Sends copy to each subscriber One slow consumer does not block others
Retry handling Delivers to subscription Queue keeps messages until processed
Backpressure Not its main job Absorbs consumer slowness
Durable async processing Partial Strong
Independent scaling Topic fan-out Workers scale per queue
Failure isolation Per subscription Per queue + DLQ

Example: order platform

Order service publishes OrderCreated
   ↓
SNS topic: order-events
   ├── SQS billing-queue
   ├── SQS inventory-queue
   ├── SQS email-queue
   └── SQS analytics-queue

If email processing fails, billing and inventory still continue.

Answer

“SNS gives me broadcast; SQS gives me durable consumption. Together, they decouple producers from consumers and consumers from each other.”

2. What are the security risks of broad SNS topic policies?

Direct answer

Broad SNS topic policies can allow unintended publishers or subscribers. That can lead to event injection, data leakage, message spoofing, subscription hijacking, cross-account abuse, denial of service, and unauthorized fan-out.

Dangerous topic policy example

{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "sns:Publish",
  "Resource": "arn:aws:sns:ap-southeast-1:111111111111:payment-events"
}

This means anyone, depending on other policy conditions and account context, may be able to publish to the topic. For production, this is extremely dangerous.

Risks

Risk Example
Event injection Attacker publishes fake PaymentSucceeded
Data leakage Unauthorized account subscribes HTTP endpoint
DoS/cost spike Malicious publisher floods topic
Confused deputy AWS service publishes from unintended source
Cross-account abuse External account gets publish/subscribe rights
Message spoofing Consumers trust SNS message blindly
Privilege escalation Automation target acts on fake notification
Compliance breach PII sent to unauthorized subscription

Common mistakes

Principal = "*"
No aws:SourceArn condition
No aws:SourceAccount condition
Allow Subscribe broadly
Allow Publish from whole organization without event validation
No encryption/access control for sensitive topics
No subscription confirmation governance

Better policy pattern

For an S3 bucket publishing to SNS, scope by source:

{
  "Effect": "Allow",
  "Principal": {
    "Service": "s3.amazonaws.com"
  },
  "Action": "sns:Publish",
  "Resource": "arn:aws:sns:ap-southeast-1:111111111111:bucket-events",
  "Condition": {
    "StringEquals": {
      "aws:SourceAccount": "111111111111"
    },
    "ArnLike": {
      "aws:SourceArn": "arn:aws:s3:::trusted-prod-bucket"
    }
  }
}

Attack scenario

Goal: Trigger fraudulent fulfilment

Attacker obtains ability to publish to order-events topic
   ↓
Publishes fake OrderPaid message
   ↓
Fulfilment service receives message
   ↓
Ships goods without real payment

Controls

Least-privilege topic policies
Explicit publisher allowlist
Explicit subscriber allowlist
aws:SourceArn and aws:SourceAccount conditions
KMS encryption for sensitive topics
Message schema validation by consumers
Do not trust message body alone
CloudTrail alerts on topic policy changes
DLQs for delivery failures

Answer

“A broad SNS topic policy is like a broad API authorization policy. If consumers act on messages, unauthorized publish permission can become unauthorized business action.”

3. How would you design attribute-based filtering for many subscribers?

Direct answer

I would define a stable message attribute taxonomy, keep filtering attributes small and explicit, avoid embedding sensitive data in attributes, and make each subscriber own a precise filter policy.

SNS supports subscription filter policies based on message attributes or the message body. By default, filter policies apply to message attributes; SNS also supports filtering on the message body when FilterPolicyScope is set to MessageBody. (AWS Documentation)

Example event

{
  "eventType": "OrderCreated",
  "tenantTier": "enterprise",
  "region": "ap-southeast-1",
  "priority": "high",
  "schemaVersion": "1.0"
}

SNS message attributes:

eventType     = OrderCreated
tenantTier    = enterprise
region        = ap-southeast-1
priority      = high
schemaVersion = 1.0

Subscriber filter examples

Billing subscriber

{
  "eventType": ["OrderCreated", "OrderCancelled"]
}

Enterprise support subscriber

{
  "tenantTier": ["enterprise"],
  "priority": ["high"]
}

Singapore analytics subscriber

{
  "region": ["ap-southeast-1"]
}

Attribute taxonomy design

Attribute Purpose
eventType Business event classification
schemaVersion Contract version
tenantId or tenantTier Tenant routing, if appropriate
region Regional routing
environment dev/staging/prod separation
dataClass public/internal/confidential
priority Operational prioritization
sourceService Producer identification

Design rules

Use stable attribute names.
Avoid one-off subscriber-specific attributes.
Keep attributes low-cardinality where possible, but not too broad.
Do not put secrets or sensitive payloads in attributes.
Document the contract.
Validate attributes in publisher pipeline.
Test subscriber filter policies.

Common mistake

Bad:

Subscriber A wants: sendToBilling=true
Subscriber B wants: sendToFraud=true
Subscriber C wants: sendToEmail=true

This makes the publisher know too much about consumers.

Better:

eventType=OrderCreated
paymentStatus=AUTHORIZED
riskScore=HIGH

Subscribers decide what they care about.

Answer

“Attribute-based filtering should describe the event, not the subscribers. The producer publishes facts; subscribers express interest through filter policies.”

4. Why is payload minimization important in fan-out architectures?

Direct answer

Payload minimization limits data exposure, reduces cost, improves performance, and prevents accidental coupling. In fan-out systems, one message may go to many subscribers, so every unnecessary field is multiplied across many trust boundaries.

Bad event

{
  "eventType": "CustomerCreated",
  "customerId": "cust-123",
  "name": "Alice Tan",
  "email": "alice@example.com",
  "phone": "+65...",
  "nric": "S1234567A",
  "fullAddress": "...",
  "creditScore": 720,
  "internalRiskNotes": "..."
}

If this is broadcast to 15 subscribers, many systems receive sensitive data they do not need.

Better event

{
  "eventType": "CustomerCreated",
  "eventId": "evt-123",
  "customerId": "cust-123",
  "tenantId": "tenant-001",
  "createdAt": "2026-05-12T09:00:00Z",
  "dataRef": "customer/cust-123"
}

Subscribers that need more data fetch it through an authorized API.

Benefits

Benefit Explanation
Lower data leakage risk Less sensitive data distributed
Lower cost Smaller messages and less network/storage
Lower coupling Consumers do not depend on every producer field
Easier schema evolution Smaller contracts change less
Better compliance Less PII propagation
Safer logging Messages often appear in logs/DLQs
Safer replay Archived/retried messages contain less sensitive data
Easier access control Consumers fetch details through authorization checks

Security concern

Fan-out makes data exposure multiplicative:

1 producer
   ↓
1 topic
   ↓
20 subscribers
   ↓
20 places where sensitive data may be logged, stored, replayed, or leaked

Payload design pattern

Event = notification that something happened.
API/database = authorized place to fetch full details.

Example:

{
  "eventType": "InvoiceReady",
  "invoiceId": "inv-123",
  "tenantId": "t-001",
  "classification": "confidential"
}

The accounting service can fetch invoice details. The notification service may only need the invoice ID.

Answer

“In fan-out, every field becomes everyone’s data. Payload minimization is a security boundary, cost control, and contract design discipline.”

5. When should you use FIFO SNS topics?

Direct answer

Use FIFO SNS topics when subscribers require strict ordering and deduplication across fan-out paths, and when throughput limits and FIFO semantics fit the workload.

SNS FIFO topics support message ordering and deduplication, and are commonly paired with SQS FIFO queues for ordered fan-out. AWS describes SNS FIFO topics as supporting ordering and deduplication for applications that need messages delivered in strict order. (Amazon Web Services, Inc.)

Use FIFO SNS when

Order of events matters.
Duplicate processing is dangerous.
Multiple subscribers need same ordered stream.
You can define a good message group ID.
Throughput requirements fit FIFO limits.
Consumers can process per-group ordering.

Good use cases

Use case Why FIFO helps
Order lifecycle OrderCreated → OrderPaid → OrderShipped
Account balance events State transitions must be ordered
Inventory reservation Avoid out-of-order stock state
Workflow state Consumers need deterministic transitions
Compliance audit stream Ordered event history per entity

Message group design

FIFO ordering is scoped by message group.

Good:

MessageGroupId = orderId

or:

MessageGroupId = accountId

Bad:

MessageGroupId = global

A global message group serializes everything and destroys throughput.

Example

Topic: order-events.fifo

MessageGroupId = ORDER#123
DeduplicationId = eventId

ORDER#123:
  OrderCreated
  PaymentAuthorized
  OrderPacked
  OrderShipped

When not to use FIFO

Do not use FIFO if:

Ordering does not matter.
General fan-out is enough.
You need very high unconstrained throughput.
Consumers are naturally idempotent.
Approximate order is acceptable.

Answer

“I use FIFO SNS when ordering is a correctness requirement, not just a nice-to-have. The hardest part is choosing the right message group ID so ordering is preserved without creating a global bottleneck.”

6. How would you make SNS-to-HTTP integrations secure?

Direct answer

I would secure SNS-to-HTTP integrations with HTTPS, authenticated endpoints, SNS message signature verification, narrow topic policies, subscription confirmation governance, replay protection, idempotency, input validation, rate limiting, and DLQs/retry policies.

SNS supports HTTPS delivery and HTTP/S delivery retry policies can be customized with phases such as no-delay, pre-backoff, backoff, and post-backoff. (AWS Documentation)

Secure flow

SNS topic
   ↓ HTTPS POST
API endpoint
   ↓
Verify SNS signature
Validate topic ARN
Validate message type
Validate schema
Check timestamp/replay window
Process idempotently
Return 2xx only after durable acceptance

Important controls

Control Why
HTTPS only Protects message in transit
Verify SNS signature Confirms message came from SNS
Validate TopicArn Prevents accepting messages from wrong topic
Validate SigningCertURL domain Avoid certificate URL abuse
Subscription confirmation workflow Prevents unauthorized endpoint subscription
Idempotency key Handles retries safely
Replay protection Blocks old message reuse
Rate limiting Protects endpoint from floods
DLQ Captures undeliverable messages
Narrow topic policy Prevents unauthorized publishers
Minimal payload Reduces exposure

HTTP endpoint failure example

SNS sends PaymentSucceeded
   ↓
HTTP endpoint processes payment notification
   ↓
Endpoint times out before returning 200
   ↓
SNS retries
   ↓
Payment notification processed twice

Mitigation:

Use eventId as idempotency key.
Persist event before side effects.
Return 2xx only after durable acceptance.

Bad design

Public HTTP endpoint accepts any JSON payload
No SNS signature verification
No topic ARN validation
No idempotency

Attack:

Attacker sends fake SNS-looking HTTP POST
   ↓
Endpoint processes fake event

Answer

“For SNS-to-HTTP, the endpoint must treat every request as untrusted until it verifies the SNS signature, topic, schema, freshness, and idempotency.”

7. What are the failure modes of SNS delivery to Lambda or SQS?

Direct answer

SNS delivery to Lambda or SQS can fail due to permissions, throttling, unavailable targets, misconfigured resource policies, payload size limits, encryption/KMS issues, disabled targets, filter mismatches, or downstream processing failures.

For server-side errors involving AWS-managed endpoints such as SQS or Lambda, SNS can retry failed deliveries many times over an extended period; AWS documents retries up to 100,015 times over 23 days for those endpoint types. (AWS Documentation)

SNS → Lambda failure modes

Failure mode Example
Missing Lambda permission SNS not allowed to invoke function
Lambda throttling Concurrency limit reached
Lambda error Function throws exception
Timeout Function exceeds timeout
Payload issue Message too large or malformed for function
KMS/env failure Function cannot decrypt config/secrets
Bad filter policy Messages never delivered
Retry storm Function repeatedly fails
DLQ missing Failed deliveries hard to recover

SNS → SQS failure modes

Failure mode Example
Queue policy missing SQS does not allow SNS topic to send
KMS key issue Encrypted queue denies SNS/KMS use
Queue deleted Subscription target gone
Message size too large Delivery fails
FIFO mismatch Standard/FIFO compatibility issues
Bad filter policy Expected messages filtered out
DLQ missing Delivery failure visibility reduced
Consumer lag SNS delivery succeeds, but SQS backlog grows

Example SQS policy requirement

For SNS to send to SQS, the SQS queue policy must allow that specific topic:

{
  "Effect": "Allow",
  "Principal": {
    "Service": "sns.amazonaws.com"
  },
  "Action": "sqs:SendMessage",
  "Resource": "arn:aws:sqs:ap-southeast-1:111111111111:billing-queue",
  "Condition": {
    "ArnEquals": {
      "aws:SourceArn": "arn:aws:sns:ap-southeast-1:111111111111:order-events"
    }
  }
}

Operational rule

Always configure:

delivery status logging
DLQs where supported
CloudWatch alarms
target permission checks
filter policy tests
idempotent consumers

Answer

“SNS delivery failure is not always SNS failure. It may be target permissions, Lambda throttling, SQS queue policy, KMS denial, filter mismatch, or a consumer that cannot keep up.”

8. How would you design an event-notification system with clear trust boundaries?

Direct answer

I would separate producers, topics, subscribers, and processing queues by trust boundary, data classification, environment, and ownership. I would use minimal payloads, schema validation, least-privilege publish/subscribe policies, per-consumer queues, and explicit cross-account controls.

Reference architecture

Producer account/service
   ↓
SNS topic owned by event domain
   ↓
Subscription filter policies
   ↓
Per-consumer SQS queues
   ↓
Consumer-owned workers

Trust-boundary design

payments account
  topic: payment-events
  publishers: payment service only

consumer accounts
  billing queue
  fraud queue
  analytics queue

security account
  audit queue / archive pipeline

Key principles

Principle Design
Producer authentication Only approved roles can publish
Consumer isolation Each subscriber gets its own SQS queue
Payload minimization Event IDs and references, not full sensitive records
Authorization Consumers fetch details through authorized APIs
Filtering Filter by attributes, not subscriber-specific flags
Data classification Separate topics for sensitive vs non-sensitive events
Cross-account boundaries Explicit topic and queue policies
Replay/retry safety Idempotent consumers
Observability Per-subscription DLQs and metrics

Event example

{
  "eventId": "evt-123",
  "eventType": "PaymentAuthorized",
  "schemaVersion": "1.0",
  "tenantId": "tenant-001",
  "paymentId": "pay-999",
  "amountBucket": "100-500",
  "currency": "SGD",
  "dataClass": "confidential",
  "occurredAt": "2026-05-12T10:00:00Z"
}

Avoid:

{
  "cardNumber": "...",
  "cvv": "...",
  "customerNric": "...",
  "fullBankAccount": "..."
}

Attack tree: trust-boundary failure

Goal: Abuse event-notification system

├── Publish fake event
│   ├── Broad topic policy
│   ├── Compromised producer role
│   └── No consumer validation

├── Receive unauthorized event
│   ├── Broad subscribe permission
│   ├── HTTP endpoint added secretly
│   └── Cross-account policy too broad

├── Leak sensitive payload
│   ├── PII in event body
│   ├── DLQ/logs store full payload
│   └── Many subscribers receive unnecessary data

└── Trigger unsafe action
    ├── Consumer trusts event blindly
    ├── No idempotency
    └── No state verification

Answer

“Clear trust boundaries mean producers are authorized to publish, subscribers are authorized to receive, payloads contain only what they need, and consumers verify before acting.”

9. How do you monitor subscriber health in SNS-driven systems?

Direct answer

Monitor both SNS delivery health and downstream subscriber health. SNS may successfully deliver to SQS, while the actual consumer is failing and the queue backlog is growing.

Health signals

Layer Signals
SNS topic publish count, failed notifications, delivery latency
Subscription delivery failures, filter drops, DLQ messages
SQS subscriber queue depth, age of oldest message, DLQ depth
Lambda subscriber errors, throttles, duration, concurrency
HTTP subscriber 4xx/5xx, timeout, retry count
Consumer app processing success/failure, idempotency rejects
Business outcome expected downstream action completed
Security policy changes, new subscriptions, unusual publishers

Key alarms

SNS failed notifications > 0
SNS DLQ messages > 0
SQS ApproximateAgeOfOldestMessage rising
SQS queue depth above threshold
SQS DLQ depth > 0
Lambda errors/throttles > baseline
HTTP delivery failures spike
Subscriber has received no messages for unexpected duration
Filter policy changed
Topic policy changed
New subscription created

Important distinction

SNS success does not mean business success.

SNS → SQS delivery succeeded
   ↓
Consumer workers are down
   ↓
Queue backlog grows
   ↓
Business process is failing

Therefore monitor:

end-to-end event processing latency
time from publish to completed business action

Example dashboard

Topic: order-events
  publish rate
  failed notifications
  delivery attempts

Subscription: billing-queue
  messages delivered
  queue depth
  oldest message age
  billing worker errors
  billing event processing p95

Subscription: email-queue
  queue depth
  email provider errors
  DLQ count

Answer

“Subscriber health is not just whether SNS delivered. It is whether each subscriber accepted, processed, and completed the intended business action within its SLO.”

10. How would you migrate from direct service-to-service calls to SNS fan-out safely?

Direct answer

I would migrate incrementally using a strangler pattern: publish events alongside existing direct calls, add subscribers in shadow mode, validate outcomes, then gradually remove direct dependencies once consumers are reliable and idempotent.

Current state

Order service
   ├── calls billing service directly
   ├── calls inventory service directly
   ├── calls email service directly
   └── calls analytics service directly

Problems:

tight coupling
slow request path
failure cascades
hard to add consumers
producer knows too much

Target state

Order service
   ↓
SNS topic: order-events
   ├── SQS billing-queue
   ├── SQS inventory-queue
   ├── SQS email-queue
   └── SQS analytics-queue

Migration phases

Phase 1: Define event contract

{
  "eventId": "evt-123",
  "eventType": "OrderCreated",
  "schemaVersion": "1.0",
  "orderId": "ord-999",
  "tenantId": "tenant-001",
  "occurredAt": "2026-05-12T10:00:00Z"
}

Phase 2: Publish without changing behavior

Order service still calls downstream services directly.
Order service also publishes OrderCreated event.
Subscribers consume in shadow mode.

Phase 3: Validate

Compare:

direct call result
event subscriber result
processing latency
error rate
duplicate handling
missing events
business reconciliation

Phase 4: Move non-critical consumers first

Good first migrations:

analytics
email notifications
audit logging
search indexing

Avoid first:

payment settlement
inventory decrement
mission-critical synchronous decisions

Phase 5: Cut over critical consumers

For critical flows:

add idempotency
add DLQs
add replay/reconciliation
add compensating actions
monitor SLOs
use feature flags

Phase 6: Remove direct calls

Only after:

subscriber stable
DLQs clean
latency acceptable
reconciliation passes
rollback path ready

Safe rollout pattern

0% event-driven action, 100% direct
   ↓
event shadow mode
   ↓
10% event-driven for one consumer
   ↓
50%
   ↓
100%
   ↓
remove direct call

Common failure

Bad migration:

Remove direct billing call.
Publish event to SNS.
Billing consumer is not idempotent.
SNS retries after timeout.
Customer charged twice.

Better:

Billing event uses paymentId idempotency key.
Billing checks order state before charging.
Duplicate events are ignored.

Answer

“I would not replace direct calls with fan-out in one step. I would dual-publish, shadow consume, validate, migrate low-risk consumers first, and make every subscriber idempotent before relying on events.”

Controls and mitigations

Risk Controls
Unauthorized publish Least-privilege topic policy, SourceArn/SourceAccount
Unauthorized subscribe Explicit subscriber allowlist, subscription review
Data leakage Payload minimization, data classification, no secrets/PII in events
Bad filtering Stable attributes, filter tests, schema validation
Lambda/SQS delivery failure Permissions, DLQs, retry awareness, KMS policy checks
HTTP integration abuse HTTPS, SNS signature verification, topic validation
Duplicate processing Idempotency keys, processed-event store
Subscriber backlog Queue age alarms, DLQs, autoscaling
Unsafe fan-out migration Dual-publish, shadow mode, feature flags, staged cutover

Summary

The senior-level answer is:

“SNS is excellent for event fan-out, and SQS makes that fan-out durable and independently consumable. The main risks are broad topic policies, over-shared payloads, bad filters, duplicate processing, and weak subscriber monitoring.”

My production checklist:

Use SNS for fan-out and SQS per subscriber for buffering.
Use least-privileged topic and queue policies.
Use SourceArn and SourceAccount conditions.
Design stable message attributes for filtering.
Keep payloads minimal and avoid sensitive data.
Use FIFO SNS only when ordering and deduplication are correctness requirements.
Secure HTTP subscribers with HTTPS, signature verification, replay protection, and idempotency.
Configure DLQs and monitor delivery failures.
Monitor subscriber queues, Lambda errors, HTTP failures, and business outcomes.
Migrate from direct calls using dual-publish, shadow validation, and staged cutover.

Key Takeaways

  • SNS is for broadcast and fan-out, not durable buffering by itself.
  • Topic policies and payload design are the main security controls.
  • Delivery visibility matters because publish success is not end-to-end success.
  • SNS plus SQS is a strong default for scalable async pipelines.
  • Filter policies reduce noise but need disciplined event contracts.
