SQS — Amazon Simple Queue Service

Category: Messaging
AWS Service Type: Managed Message Queue
Key Docs: SQS Developer Guide

Overview

SQS is AWS's managed message queue for decoupling producers and consumers. It supports Standard queues for near-unlimited throughput with at-least-once delivery and best-effort ordering, and FIFO queues for strict ordering with exactly-once processing semantics within defined constraints. SQS integrates tightly with Lambda, ECS, EC2, Step Functions, SNS, EventBridge, and DLQ patterns. It is commonly used to absorb bursts, smooth asynchronous processing, isolate failures, and decouple request handling from background work.

How It Works

Core Mechanics

Concept Description
Standard queue High-throughput queue with at-least-once delivery
FIFO queue Ordered queue with message group semantics and deduplication
Visibility timeout Period after receipt when message is hidden from other consumers
DLQ Dead-letter queue for messages that exceed max receive count
Long polling Reduces empty receives and API cost
Retention period How long messages stay in queue before expiration
Message group FIFO ordering boundary enabling parallelism across groups
Redrive policy Rule linking source queue to DLQ

Top 3 Critical Mitigations

  1. Always add a DLQ: Poison messages are normal, not exceptional.
  2. Treat consumers as idempotent: At-least-once delivery means duplicates are part of the contract.
  3. Monitor age, not just count: Queue depth without age hides whether the backlog actually threatens user-facing SLOs.

1. How do Standard and FIFO queues differ operationally?

Direct answer

Standard SQS queues optimize for very high throughput and at-least-once delivery, but they do not guarantee strict ordering.
FIFO SQS queues optimize for ordered processing and deduplication, but they require careful message-group design and usually have more throughput constraints.

Standard vs FIFO

Area Standard SQS FIFO SQS
Ordering Best-effort ordering Strict order within a message group
Delivery At-least-once Exactly-once processing attempt within deduplication window, but consumers must still be idempotent
Throughput Very high Depends on message groups and FIFO throughput settings
Duplicate messages Possible Deduplication reduces duplicates
Message group ID Not required Required
Deduplication ID Not required Required unless content-based deduplication enabled
Best for High-scale async processing Ordered workflows and state transitions
Risk Duplicate/out-of-order processing Hot message groups bottleneck throughput

Standard queue example

Use Standard SQS for:

image processing
log ingestion
email sending
analytics events
webhook processing
batch jobs

If image 103 is processed before image 102, usually it does not matter.

FIFO queue example

Use FIFO SQS for:

payment state transitions
order lifecycle events
inventory reservation per SKU
account ledger events per account
workflow steps per entity

Example:

MessageGroupId = ORDER#123

1. OrderCreated
2. PaymentAuthorized
3. OrderPacked
4. OrderShipped

These must be processed in order for that order.

Answer

“I use Standard queues when throughput and loose ordering are acceptable. I use FIFO queues when ordering is a correctness requirement. The main FIFO design challenge is choosing the right message group ID so I preserve order without creating a bottleneck.”

2. Why does visibility timeout matter so much for correctness?

Direct answer

Visibility timeout controls how long a message stays hidden after a consumer receives it. If the consumer finishes and deletes the message before the timeout expires, processing succeeds. If the consumer fails, crashes, or does not delete the message, the message becomes visible again and another consumer can retry it.

This matters because a wrong visibility timeout can cause duplicate processing, stuck retries, slow recovery, or concurrent processing of the same message.

Flow

Message in queue
   ↓
Consumer receives message
   ↓
Message becomes invisible for visibility timeout
   ↓
Consumer processes message
   ↓
Consumer deletes message

If not deleted:

Visibility timeout expires
   ↓
Message becomes visible again
   ↓
Another consumer receives it

Why correctness depends on it

Timeout too short

Consumer needs 90 seconds to process.
Visibility timeout = 30 seconds.

Consumer A receives message.
After 30 seconds, message becomes visible.
Consumer B receives same message.
Both process same order.

Result:

duplicate email
duplicate payment
duplicate shipment
duplicate database update

Timeout too long

Consumer crashes after 5 seconds.
Visibility timeout = 30 minutes.

Result:

message is stuck invisible for 30 minutes
processing recovery is slow
queue backlog grows

Good design

Set visibility timeout based on:

maximum expected processing time
external API timeout
database timeout
batch size
retry behavior
failure recovery target

Common pattern:

visibility timeout > max processing time
but not so long that failure recovery is delayed

For long processing jobs, extend visibility timeout while processing:

Receive message
   ↓
Start work
   ↓
Periodically call ChangeMessageVisibility
   ↓
Delete message after success

Answer

“Visibility timeout is the lease on the message. If the lease is too short, two workers may process the same work. If it is too long, failed work takes too long to retry.”

3. How would you design an idempotent SQS consumer?

Direct answer

I would assume every SQS message can be delivered more than once. The consumer must safely handle duplicates by using an idempotency key, recording processing state, and making side effects conditional.

Idempotent consumer pattern

Receive message
   ↓
Extract idempotency key
   ↓
Check processed-event store
   ↓
If already processed:
      delete message
      return
   ↓
Acquire processing record conditionally
   ↓
Perform business action
   ↓
Mark processed
   ↓
Delete message

Example message

{
  "eventId": "evt-123",
  "orderId": "ord-999",
  "eventType": "OrderPaid",
  "amount": 125.5,
  "currency": "SGD"
}

Use:

idempotencyKey = eventId

or for business operations:

idempotencyKey = paymentId

DynamoDB idempotency table

Table: ProcessedEvents

PK = eventId
status = PROCESSING | COMPLETED | FAILED
expiresAt = TTL timestamp

Conditional write

PutItem eventId
ConditionExpression: attribute_not_exists(eventId)

If it succeeds, this consumer owns the processing.
If it fails, the message is duplicate or already being processed.

Payment example

Bad consumer:

Receive PaymentAuthorized
   ↓
Charge card
   ↓
Timeout before delete
   ↓
Message retried
   ↓
Charge card again

Good consumer:

Receive PaymentAuthorized
   ↓
Create idempotency record for paymentId
   ↓
Call payment provider with idempotency key
   ↓
Store result
   ↓
Delete message

Important rule

Idempotency must cover external side effects:

payments
emails
shipments
database writes
ticket creation
notifications

Consumer pseudo-code

def handle_message(msg):
    event_id = msg["eventId"]

    created = try_create_processing_record(event_id)

    if not created:
        return "duplicate-safe-delete"

    try:
        perform_business_action(msg)
        mark_completed(event_id)
        return "success-delete-message"
    except TemporaryError:
        mark_failed_or_leave_processing(event_id)
        raise
    except PermanentError:
        send_to_dlq_or_mark_permanent_failure(event_id)
        return "handled-delete-message"

Answer

“An SQS consumer is correct only when duplicate delivery does not create duplicate business effects. I design idempotency around the business operation, not just around message receipt.”

4. What metrics best indicate a queue-processing incident?

Direct answer

The strongest incident signals are age of oldest message, queue depth, DLQ depth, consumer error rate, consumer throttling, and processing latency. Queue depth alone is not enough; a queue can be deep but healthy if consumers are draining it fast.

Key SQS metrics

Signal Why it matters
ApproximateAgeOfOldestMessage Best backlog severity signal
ApproximateNumberOfMessagesVisible Ready backlog
ApproximateNumberOfMessagesNotVisible In-flight messages
NumberOfMessagesSent Producer rate
NumberOfMessagesReceived Consumer receive rate
NumberOfMessagesDeleted Successful processing rate
NumberOfEmptyReceives Polling inefficiency
DLQ message count Poison/failure signal
DLQ oldest message age Unresolved failed work
Lambda errors/throttles Consumer failure
ECS task CPU/memory Consumer capacity
Downstream DB/API latency Consumer bottleneck
Business completion rate True end-to-end health

Healthy vs unhealthy patterns

Healthy burst

MessagesVisible increases
MessagesDeleted also increases
OldestMessageAge remains low
DLQ remains zero

This means consumers are keeping up.

Processing incident

MessagesVisible increasing
OldestMessageAge increasing
MessagesDeleted flat
Consumer errors high

This means messages are not being processed successfully.

Downstream bottleneck

Consumer count high
In-flight messages high
DB latency high
OldestMessageAge rising

Scaling more consumers may make things worse.

Poison-message incident

Same message retried repeatedly
Receive count increases
DLQ count grows
Consumer errors spike

Best alarms

ApproximateAgeOfOldestMessage > SLO
DLQ ApproximateNumberOfMessagesVisible > 0
MessagesVisible increasing for N minutes
MessagesDeleted drops below expected baseline
Lambda throttles > 0
Lambda errors above baseline
ECS service desired != running
Consumer p95 processing duration increasing
Downstream dependency errors increasing

Answer

“For SQS, oldest message age tells me customer impact better than queue depth. Depth tells me amount of work; age tells me whether the system is falling behind.”

5. How would you scale consumers safely under burst traffic?

Direct answer

I would scale consumers based on backlog and processing latency, but with guardrails to protect downstream systems. The goal is not maximum concurrency; the goal is stable throughput without creating a retry storm or overwhelming dependencies.

Scaling design

Producer burst
   ↓
SQS absorbs burst
   ↓
Consumers scale out gradually
   ↓
Concurrency capped to protect DB/API
   ↓
DLQ handles poison messages

For Lambda consumers

Controls:

reserved concurrency
maximum concurrency on SQS event source mapping
batch size
batch window
visibility timeout
partial batch response
DLQ/redrive policy

Safe pattern:

Queue backlog increases
   ↓
Lambda concurrency increases
   ↓
Max concurrency cap prevents DB overload
   ↓
Failed messages are isolated

For ECS consumers

Controls:

ECS service autoscaling on queue depth per task
min/max task count
CPU/memory limits
connection pool limits
rate limits
graceful shutdown
backoff on downstream errors

Scaling metric:

backlog per task = visible messages / running consumer tasks

Example:

If backlog per task > 100 for 5 minutes:
  scale out

If backlog per task < 10 for 15 minutes:
  scale in

Downstream protection

Scaling consumers can harm dependencies:

SQS backlog
   ↓
Scale from 10 to 1000 consumers
   ↓
All hit RDS
   ↓
RDS connection exhaustion
   ↓
Errors
   ↓
Retries
   ↓
Bigger backlog

Controls:

max concurrency
connection pooling
rate limiting
circuit breakers
bulkheads per queue
backoff with jitter
reserved capacity for critical queues

Answer

“SQS lets me absorb bursts, but consumers must scale with a brake. I cap concurrency based on downstream capacity, not just queue size.”

6. What are the most common poison-message patterns?

Direct answer

A poison message is a message that repeatedly fails processing and blocks or slows progress. Common causes include malformed payloads, schema mismatches, missing referenced resources, permanent business-rule failures, oversized messages, bad encoding, and messages that trigger consumer bugs.

Common patterns

Poison pattern Example
Malformed JSON Consumer cannot parse body
Unknown schema version Producer sends v2, consumer supports v1
Missing required field No orderId
Invalid enum status = "PAYMENT_MAYBE"
Referenced resource missing orderId does not exist
Authorization issue event belongs to tenant consumer cannot access
Permanent external failure invalid email address
Consumer bug trigger specific payload causes null pointer
Oversized payload/reference payload too large or S3 object missing
Non-idempotent side effect retry creates duplicate work
FIFO head-of-line blocker one bad message blocks group

Poison-message flow

Message received
   ↓
Consumer fails
   ↓
Visibility timeout expires
   ↓
Message retried
   ↓
Consumer fails again
   ↓
After maxReceiveCount
   ↓
Message moves to DLQ

Controls

Schema validation at producer and consumer
Strict event contract versioning
maxReceiveCount redrive policy
DLQ with alarms
Partial batch failure for Lambda
Idempotency
Permanent vs transient error classification
Quarantine queue for invalid messages
Replay tooling after fix

Example

Consumer expects:

{
  "schemaVersion": "1.0",
  "orderId": "ord-123"
}

Producer sends:

{
  "schemaVersion": "2.0",
  "purchaseId": "ord-123"
}

The consumer fails every time. This is not a transient error; retrying will not help.

Good consumer behavior

If malformed/permanent:
  send to DLQ or mark as permanent failure
  do not retry forever

If transient:
  retry with backoff

If duplicate:
  safely delete

Answer

“A poison message is often not a system outage; it is bad input meeting a retry system. The fix is classification, validation, DLQ isolation, and safe replay.”

7. How do queue policies and IAM combine for SQS access?

Direct answer

SQS access is controlled by both IAM identity policies and SQS queue resource policies. IAM policies define what an identity can do. Queue policies define who can access a specific queue, including cross-account principals or AWS services.

Both are evaluated together with other controls like SCPs, permission boundaries, session policies, and KMS key policies if the queue is encrypted.

IAM identity policy

Attached to a role/user.

Example: consumer role can receive from one queue.

{
  "Effect": "Allow",
  "Action": [
    "sqs:ReceiveMessage",
    "sqs:DeleteMessage",
    "sqs:ChangeMessageVisibility",
    "sqs:GetQueueAttributes"
  ],
  "Resource": "arn:aws:sqs:ap-southeast-1:111111111111:orders-queue"
}

Queue resource policy

Attached to the queue.

Example: allow SNS topic to send to queue.

{
  "Effect": "Allow",
  "Principal": {
    "Service": "sns.amazonaws.com"
  },
  "Action": "sqs:SendMessage",
  "Resource": "arn:aws:sqs:ap-southeast-1:111111111111:orders-queue",
  "Condition": {
    "ArnEquals": {
      "aws:SourceArn": "arn:aws:sns:ap-southeast-1:111111111111:order-events"
    }
  }
}

Cross-account example

For Account A producer to send to Account B queue:

Producer role in Account A:
  IAM policy allows sqs:SendMessage to Account B queue

Queue in Account B:
  queue policy allows Account A producer role

Usually both sides are needed.

Encrypted queue consideration

If SQS uses a customer-managed KMS key, the principal or AWS service also needs appropriate KMS permissions.

sqs:SendMessage allowed
but kms:GenerateDataKey denied
   ↓
send fails

Common mistakes

IAM allows send but queue policy denies cross-account access
Queue policy allows SNS but missing SourceArn condition
Queue encrypted but KMS policy does not allow producer/service
Consumer has ReceiveMessage but not DeleteMessage
Producer has broad sqs:* on all queues

Answer

“IAM says what the caller is allowed to do. The queue policy says who the queue trusts. For cross-account or service integrations, you normally need both plus KMS permissions if encrypted.”

8. How would you connect SQS to Lambda versus ECS?

Direct answer

Use Lambda when processing is event-driven, short-lived, bursty, and easy to parallelize. Use ECS when processing is long-running, needs custom runtime control, heavy dependencies, predictable workers, connection pooling, or sustained throughput.

SQS to Lambda

Architecture:

SQS queue
   ↓ event source mapping
Lambda consumer
   ↓
Downstream services

Best for:

short tasks
bursty traffic
simple event processing
low operational overhead
automatic scaling

Important settings:

batch size
batch window
visibility timeout
function timeout
reserved concurrency
maximum concurrency
partial batch response
DLQ/redrive

Failure behavior:

Lambda receives batch
   ↓
If entire batch fails, messages become visible again
   ↓
With partial batch response, only failed messages retry

SQS to ECS

Architecture:

SQS queue
   ↓
ECS worker service polls queue
   ↓
Processes message
   ↓
Deletes message

Best for:

long-running tasks
CPU/memory-heavy processing
custom libraries
large files
steady high throughput
long polling control
database connection pooling
fine-grained backpressure

Autoscaling:

Scale ECS tasks based on:
  ApproximateNumberOfMessagesVisible / RunningTaskCount
  ApproximateAgeOfOldestMessage
  CPU/memory

Lambda vs ECS comparison

Area Lambda ECS
Ops overhead Lower Higher
Runtime duration Limited Long-running possible
Burst scaling Strong Requires service scaling
Custom environment Limited Strong
Connection pooling Harder at high concurrency Easier
Cost model Per invocation/duration Task runtime
Backpressure Concurrency caps Worker scaling/rate limits
Large processing Less ideal Better
Startup latency Cold starts possible Workers already running

Answer

“Lambda is excellent for short bursty SQS workloads. ECS is better when I need long-lived workers, heavy processing, custom runtime control, or careful downstream connection management.”

9. What are the risks of using FIFO for workloads with skewed message groups?

Direct answer

FIFO queues guarantee ordering within a message group. If one message group receives most of the traffic, that group becomes a bottleneck because messages in the same group are processed sequentially. This creates head-of-line blocking and poor throughput.

Example skew

Bad message group design:

MessageGroupId = tenantId

If one tenant is huge:

tenant-small-1: 10 msg/sec
tenant-small-2: 8 msg/sec
tenant-big: 10,000 msg/sec

The big tenant’s group becomes the bottleneck.

Head-of-line blocking

MessageGroupId = CUSTOMER#123

Message 1 fails repeatedly
   ↓
Message 2 waits
Message 3 waits
Message 4 waits

One poison message blocks the whole group.

Risks

Risk Explanation
Low throughput One hot group serializes processing
Head-of-line blocking Failed message blocks later messages
Poor scaling Adding consumers does not help hot group
DLQ delay Messages behind poison wait
Latency variance Small groups fast, hot groups slow
Bad tenant isolation One big tenant backlog grows
Operational surprise Queue depth rises despite many consumers

Better message group design

Choose group ID based on the smallest entity that truly needs ordering.

Instead of:

MessageGroupId = tenantId

Use:

MessageGroupId = orderId

or:

MessageGroupId = accountId

or sharded grouping if strict global tenant order is not required:

MessageGroupId = tenantId#shard-07

Tradeoff

Sharding message groups increases throughput, but weakens ordering.

tenantId
  strict tenant-wide order
  low throughput for hot tenant

tenantId#shard
  higher throughput
  order only within shard

Answer

“FIFO does not make everything ordered and scalable for free. Ordering is scoped by message group, and a bad message-group key can turn a distributed queue into a single-lane road.”

10. How would you drain and replay a DLQ safely?

Direct answer

I would treat DLQ replay as a controlled production operation. First classify why messages failed, fix the consumer or data issue, sample and validate messages, replay in small batches, monitor closely, and stop immediately if failures recur.

Safe DLQ replay process

DLQ has messages
   ↓
Stop automatic replay
   ↓
Classify failure reason
   ↓
Fix root cause
   ↓
Sample messages
   ↓
Validate schema and business safety
   ↓
Replay small batch
   ↓
Monitor target queue and consumers
   ↓
Gradually increase replay
   ↓
Archive or delete only after success

Step 1: Understand failure

Ask:

Why did messages land in DLQ?
Is the error transient or permanent?
Was the consumer fixed?
Are messages still valid?
Will replay cause duplicate side effects?
Are downstream systems ready?

Step 2: Categorize messages

Category Action
Transient failure Replay after dependency recovery
Consumer bug fixed Replay carefully
Bad schema Transform or discard with approval
Permanent business invalid Do not replay blindly
Duplicate side effect risk Verify idempotency first
Too old/stale Decide based on business rules

Step 3: Replay pattern

Do not replay everything at once.

Replay 10 messages
   ↓
Validate
   ↓
Replay 100
   ↓
Validate
   ↓
Replay 1,000
   ↓
Continue with rate limit

Step 4: Protect downstream

Controls:

rate-limited replay
idempotency checks
consumer concurrency cap
DLQ redrive allow policy
temporary dashboards
alarms on failure recurrence
pause switch

Example: email DLQ

If old email events failed yesterday:

Should they still be sent?
Will customers receive outdated notifications?
Should we suppress messages older than 24 hours?

Replay decision may be:

send password reset emails? No, expired.
send invoice ready emails? Maybe yes.
send shipment updates? Maybe if still relevant.

Example: payment DLQ

Never blindly replay payment messages.

Check paymentId idempotency.
Check current order state.
Check provider transaction state.
Then replay or reconcile manually.

Answer

“DLQ replay is not a button; it is an incident recovery workflow. I fix the cause, classify the messages, prove idempotency, replay slowly, and monitor business effects.”

Summary

The senior-level answer is:

“SQS gives durable asynchronous processing, but correctness comes from consumer design. I assume at-least-once delivery, design idempotent consumers, tune visibility timeout, isolate poison messages, scale with downstream protection, and treat DLQ replay as a controlled recovery operation.”

My production checklist:

Use Standard queues for high-throughput unordered workloads.
Use FIFO only when ordering is a correctness requirement.
Tune visibility timeout above expected processing time.
Make every consumer idempotent.
Use DLQs with alarms.
Monitor oldest message age, queue depth, in-flight count, delete rate, and DLQ depth.
Scale consumers based on backlog, but cap concurrency to protect dependencies.
Classify poison messages as transient or permanent.
Use least-privileged IAM, queue policies, and KMS policies.
Choose Lambda for short bursty work and ECS for long-running controlled workers.
Design FIFO message groups carefully to avoid hot groups.
Replay DLQs slowly and safely after fixing the root cause.

Key Takeaways

  • SQS is a durability and decoupling tool, not a guarantee of exactly-once processing for most workloads.
  • DLQs and idempotent consumers are the production baseline.
  • Visibility timeout must match actual processing behavior.
  • Queue age is one of the most important operational signals.
  • Access policies matter because queues are also injection surfaces.
