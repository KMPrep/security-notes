API Gateway — Amazon API Gateway

Category: Networking
AWS Service Type: Managed API Front Door
Key Docs: API Gateway Developer Guide

Overview

API Gateway is AWS's managed service for publishing, securing, throttling, and observing APIs. It supports REST APIs, HTTP APIs, and WebSocket APIs, integrating with Lambda, ECS, EC2, ALB, Step Functions, and private VPC backends. API Gateway handles authentication, request validation, usage plans, throttling, custom domains, caching, transformation, and detailed execution logging. It is commonly used as the public ingress layer for serverless applications and internal service APIs.

How It Works

Core Mechanics

Concept Description
REST API Feature-rich API type with request/response mapping, API keys, usage plans
HTTP API Lower-cost, lower-latency API type with simpler routing and auth
WebSocket API Stateful bidirectional API service for real-time apps
Authorizer JWT, Lambda, or IAM-based authentication and authorization layer
Stage Deployed environment such as dev, test, prod
Usage plan API key quota and throttling policy

What Can Go Wrong

3 No throttling Abuse or client bug floods backend Cost spike or outage High

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Missing auth on route Require authorizers by default in IaC and deployment checks API governance
2 Public backend bypass Restrict backend reachability to API Gateway path only Backend protection
3 No throttling Configure stage and usage-plan throttles plus WAF where needed Abuse protection
4 Overly permissive CORS Allow only trusted origins and avoid wildcard with credentials Browser security
5 Sensitive data in logs Redact or avoid body/header logging for sensitive APIs Logging hygiene
6 Lambda authorizer latency Cache auth decisions appropriately and simplify token validation Authorizer design
7 Stage config drift Manage stages, routes, and integrations entirely in IaC Change management
8 Cache used with auth variance Include auth context in cache key or disable cache for private APIs Cache design

Top 3 Critical Mitigations

  1. Make auth explicit per route: API Gateway is easy to deploy quickly, which also makes it easy to leave a route unprotected.
  2. Prevent backend bypass: API Gateway only adds value if clients cannot skip it and talk to the backend directly.
  3. Throttle early: API Gateway is one of the cheapest places to reject abusive or accidental traffic before it reaches more expensive backends.

1. How do REST APIs and HTTP APIs differ in API Gateway?

Direct answer

In Amazon API Gateway, REST APIs are the older, feature-rich API product, while HTTP APIs are the newer, simpler, lower-cost, lower-latency API product.

AWS’s own guidance is straightforward: choose REST APIs when you need API keys, per-client throttling, request validation, AWS WAF integration, or private API endpoints. Choose HTTP APIs when you do not need those extra REST API features and want a simpler, lower-cost API. (AWS Documentation)

Comparison

Area REST API HTTP API
Maturity Older, feature-rich Newer, simpler
Cost/latency Usually higher Usually lower
API keys Supported More limited
Usage plans Supported More limited
Request validation Supported More limited
Mapping templates More powerful Simpler
JWT authorizer Possible, but HTTP APIs have native JWT authorizers Strong native support
Lambda authorizer Supported Supported
IAM auth Supported Supported
Private integration Supported Supported through VPC Link
WAF integration Stronger native REST API support Often handled with CloudFront/WAF pattern
Best for Enterprise API management Simple modern HTTP workloads

Practical rule

Use REST API when:
  API keys, usage plans, request validation, complex mapping, mature API management are required.

Use HTTP API when:
  simple JWT/IAM/Lambda-authorized routing to Lambda, ALB, ECS, or backend services is enough.

Real-world example

For a public partner API:

Partner A gets 100 requests/sec.
Partner B gets 500 requests/sec.
Each partner has API keys, quotas, and usage plans.
Request schema must be validated.

I would lean toward REST API.

For a mobile app backend:

Mobile app sends JWT.
API Gateway validates JWT.
Requests go to Lambda or private ALB.

I would usually start with HTTP API.

Answer

“HTTP API is my default for simple APIs. REST API is my choice when I need mature API management controls such as usage plans, API keys, request validation, and stronger feature depth.”

2. Tradeoffs between JWT authorizers, IAM auth, and Lambda authorizers

Direct answer

Use JWT authorizers when clients use OIDC/OAuth-style tokens.
Use IAM auth when callers are AWS workloads or identities.
Use Lambda authorizers when authorization logic is custom, dynamic, or depends on external systems.

API Gateway HTTP APIs can validate JWTs and optionally enforce scopes before allowing requests. (AWS Documentation) Lambda authorizers let you implement custom authorization logic using request parameters or bearer tokens. (AWS Documentation)

Comparison

Auth type Best for Strengths Tradeoffs
JWT authorizer User/app auth with OIDC provider Fast, managed token validation, scopes Token claims can become stale
IAM auth AWS service-to-service access Strong AWS-native identity, SigV4, no shared secret Harder for external/mobile/browser clients
Lambda authorizer Custom auth logic Very flexible More latency, cost, failure mode

JWT authorizer

Good for:

Cognito
Auth0
Okta
Azure AD
OIDC provider
machine-to-machine OAuth clients

Flow:

Client sends JWT
   ↓
API Gateway validates issuer, audience, expiry, signature
   ↓
Optional scope check
   ↓
Request forwarded to backend

Strengths:

No custom auth code.
Low latency.
Good for standard OAuth/OIDC patterns.
Works well for route-level scope checks.

Risks:

Claims may be stale until token expiry.
Revocation is hard unless token lifetime is short.
Backend must still enforce object-level authorization.
Scopes are coarse.

IAM auth

Good for:

Lambda calling API Gateway
ECS task calling internal API
Cross-account AWS workloads
automation tools
backend services

Flow:

Caller signs request with AWS SigV4
   ↓
API Gateway verifies IAM permission
   ↓
Request forwarded

Strengths:

Strong AWS-native identity.
No bearer token leakage.
Works with IAM conditions.
Good for service-to-service APIs.

Tradeoffs:

Difficult for browsers and mobile apps.
Requires AWS credential management.
Not ideal for public third-party developers unless they are AWS-integrated.

Lambda authorizer

Good for:

Custom entitlement checks
Legacy tokens
API key plus tenant rules
Dynamic authorization from database
Fine-grained partner access
Custom deny logic

Flow:

Request hits API Gateway
   ↓
API Gateway invokes Lambda authorizer
   ↓
Authorizer returns allow/deny/context
   ↓
API Gateway forwards or rejects

Strengths:

Maximum flexibility.
Can query entitlement DB.
Can enforce custom tenant, plan, risk, or device posture logic.

Risks:

Adds latency.
Adds cost.
Can become availability bottleneck.
Authorizer bugs can deny all traffic or allow too much.
Caching mistakes can leak authorization across tenants.

Security warning

Authorization is not complete at API Gateway.

Bad design:

JWT contains tenantId=t1
API Gateway validates JWT
Backend trusts request body tenantId=t2

Better:

API Gateway validates token.
Backend derives tenant from trusted token/authorizer context.
Backend enforces object-level authorization.

Answer

“JWT authorizers are best for standard token validation, IAM is best for AWS-native service identity, and Lambda authorizers are best when authorization is custom. But the backend must still enforce business authorization.”

3. How would you front private VPC services with API Gateway safely?

Direct answer

I would use API Gateway with a private integration through VPC Link to a private ALB or NLB, keep backend services in private subnets, restrict security groups, authenticate at API Gateway, and prevent direct backend access.

API Gateway private integrations use VPC Link to connect API Gateway routes to private resources behind an ALB or NLB in a VPC. (AWS Documentation) AWS also describes VPC Link as an abstraction that lets API Gateway connect to private resources inside a VPC. (Amazon Web Services, Inc.)

Secure architecture

Internet
   ↓
CloudFront + WAF
   ↓
API Gateway
   ↓
Authorizer / throttling / validation
   ↓
VPC Link
   ↓
Private ALB or NLB
   ↓
ECS / EKS / EC2 services in private subnets
   ↓
Databases/internal dependencies

Security controls

Layer Control
Edge CloudFront, WAF, TLS
API Gateway AuthN/AuthZ, throttling, request validation
VPC Link Private path to backend
ALB/NLB Internal only
Security groups Allow traffic only from expected path
Backend Verify trusted headers/token/context
Network No public IPs for backend
Logs API Gateway access logs, ALB logs, app logs
Secrets No secrets in headers unless protected and validated

Critical rule

The backend must not be reachable directly from the internet.

Bad:

Client → API Gateway → private service
Client → public ALB → same private service

If the public ALB path bypasses API Gateway, attackers can skip:

authorizer
WAF
throttling
request validation
usage plan
audit controls

Better:

Only API Gateway/VPC Link can reach internal ALB/NLB.
Backend rejects requests missing trusted gateway context.

Backend protection

Even though API Gateway is the front door, the backend should still validate:

expected Host header
trusted source network/security group
JWT/authorizer context if forwarded
mTLS or signed internal header where appropriate
tenant authorization
request schema

Answer

“I front private VPC services with API Gateway through VPC Link and an internal ALB/NLB, but I also harden the backend so API Gateway is the only valid path.”

4. Why is backend bypass a serious architectural flaw?

Direct answer

Backend bypass means clients can reach the backend without going through the intended security control point. This is serious because all controls enforced at API Gateway become optional.

Example

Intended path:

Client
   ↓
API Gateway
   ├── JWT authorizer
   ├── WAF
   ├── throttling
   ├── request validation
   └── audit logs
   ↓
Backend service

Bypass path:

Client
   ↓
Public ALB / public IP / direct service URL
   ↓
Backend service

What breaks

Control bypassed Impact
Authentication Anonymous access possible
Authorization Tenant/object checks may be skipped
WAF Exploit payloads reach app
Throttling Brute force/DoS easier
API keys/usage plans Quotas bypassed
Request validation Malformed payloads reach backend
Logging Audit trail incomplete
CORS/origin controls Browser assumptions broken
Rate limits Abuse becomes cheaper

Real-world attack scenario

A fintech API uses API Gateway JWT authorizer. The backend ECS service is also reachable through a public ALB used during testing.

Attack:

Attacker finds public ALB DNS name.
   ↓
Sends requests directly to /transfer.
   ↓
Backend assumes API Gateway already authenticated user.
   ↓
Transfer endpoint processes unauthenticated request.

Controls

Make backend private.
Use internal ALB/NLB only.
Restrict security groups.
Block public IP assignment.
Require mTLS or signed internal identity for backend.
Backend revalidates authorization context.
Monitor for direct backend access.
Do not trust X-Forwarded-* headers from arbitrary clients.

Answer

“Backend bypass is serious because it turns centralized security into a suggestion. Every backend must either be unreachable directly or capable of enforcing the same security itself.”

5. How would you design multi-tenant throttling and quota enforcement?

Direct answer

I would enforce throttling at multiple layers: API Gateway for coarse enforcement, tenant-aware application logic for precise enforcement, and distributed counters for complex quotas. For REST APIs, usage plans and API keys are useful for per-client quotas, but they should not be the only security boundary.

API Gateway REST API usage plans can define throttling and quota limits for API keys across stages, and method-level throttles can be configured. AWS notes throttles and quotas are best-effort targets, not guaranteed hard ceilings. (AWS Documentation)

Multi-tenant throttling model

Global API limit
   ↓
Route-level limit
   ↓
Tenant-level limit
   ↓
User/API-key-level limit
   ↓
Operation-level limit

Example tiers

Tenant tier Rate Burst Monthly quota
Free 10 rps 20 100k/month
Business 100 rps 200 5M/month
Enterprise 1000 rps 2000 Custom

Architecture

Client
   ↓
API Gateway
  ├── coarse throttling
  ├── API key / usage plan where appropriate
  └── authorizer extracts tenant
   ↓
Backend rate-limit middleware
   ↓
Redis/Valkey or DynamoDB counter
   ↓
Business handler

Why app-level enforcement is needed

API Gateway usage plans are useful, but tenant identity often comes from JWT claims, organization membership, subscription state, or internal billing system.

Example:

same customer has 50 API keys
tenant-level quota must aggregate across all keys

That requires application-level or centralized rate-limiter logic.

Rate-limiter key design

rate:tenant:{tenantId}:route:{routeName}:minute:{timestamp}
quota:tenant:{tenantId}:month:{yyyymm}

Security design

Tenant ID must come from trusted token/authorizer context.
Never trust tenant ID in request body or query string.
Rate-limit before expensive backend processing.
Fail closed or degrade for high-risk endpoints if limiter is unavailable.
Use separate limits for login, password reset, exports, and write APIs.

Answer

“API Gateway gives me coarse and per-key throttling. A serious SaaS platform also needs tenant-aware quotas enforced using trusted tenant identity and a central counter.”

6. Security pitfalls of CORS in API services

Direct answer

CORS is a browser access-control mechanism. It is not API authentication, not CSRF protection by itself, and not a server-side authorization control. The main pitfalls are overly broad origins, unsafe credentialed CORS, poor origin matching, missing Vary: Origin, and trusting CORS as a security boundary.

Common pitfalls

Pitfall Why dangerous
Access-Control-Allow-Origin: * with sensitive API Any website can read non-credentialed responses
Reflecting arbitrary Origin Attacker origin gets allowed
Allowing credentials broadly Cookies/tokens can be exposed to malicious sites
Regex mistakes evil-example.com matches example.com
Allowing null origin Local files/sandboxed contexts risk
Missing Vary: Origin CDN cache may serve wrong CORS header
Overbroad methods/headers Increases attack surface
Thinking CORS blocks requests CORS blocks browser reads, not all sends
No CSRF protection State-changing cookie-auth requests remain risky

Dangerous pattern

Origin: https://evil-examplecorp.com

Server checks:
  if "examplecorp.com" in Origin:
      allow

This allows attacker-controlled domains.

Better origin validation

Allowed origins:
  https://app.examplecorp.com
  https://admin.examplecorp.com

Exact match only.
No substring matching.
No wildcard with credentials.

CORS and credentials

Bad:

Access-Control-Allow-Origin: https://evil.com
Access-Control-Allow-Credentials: true

If the browser sends cookies, attacker JavaScript may read sensitive responses.

Important point

CORS does not prevent non-browser clients.

curl
Postman
malware
server-side scripts

can call the API regardless of CORS.

Controls

Exact origin allowlist.
No wildcard for credentialed APIs.
Use Authorization headers instead of cookies where appropriate.
CSRF tokens/SameSite cookies for cookie-auth state changes.
Return Vary: Origin for dynamic CORS.
Keep CORS separate from authorization.
Log and review allowed origins.

Answer

“CORS protects browsers from reading cross-origin responses. It does not authenticate callers, authorize actions, or protect the server from non-browser requests.”

7. How would you debug 5xx errors from an API Gateway integration?

Direct answer

I would isolate whether the 5xx is generated by API Gateway, the integration, the network path, the authorizer, or the backend. Then I would correlate request IDs across API Gateway access logs, execution logs, ALB/Lambda logs, and application logs.

Debug flow

Client sees 5xx
   ↓
Check API Gateway access logs
   ↓
Identify status, integrationStatus, requestId, latency
   ↓
Check authorizer result
   ↓
Check integration latency/error
   ↓
Check backend logs using correlation ID
   ↓
Check network/VPC Link/ALB/Lambda metrics

Common API Gateway 5xx causes

Cause Symptom
Lambda timeout 504/5xx, high integration latency
Lambda exception 502 or integration error
Bad Lambda proxy response 502 malformed response
Backend ALB returns 5xx API Gateway passes/integration returns error
VPC Link issue Connection errors
TLS/backend cert issue Integration failure
Authorizer failure 5xx if authorizer errors unexpectedly
Mapping template error Gateway-side failure
Integration timeout 504
Payload too large Error depending on path
Backend unavailable 502/504
Throttling elsewhere Backend/Lambda throttles

Logs to enable

API Gateway access logs
API Gateway execution logs where appropriate
Lambda logs
ALB access logs
ECS/EKS app logs
X-Ray/tracing where possible
WAF logs
CloudWatch metrics

Useful access log fields

requestId
extendedRequestId

httpMethod
resourcePath

responseLength

integrationLatency
error.message
authorizer.error

Example

Access log:

status=502
integrationStatus=200
error="Malformed Lambda proxy response"

This means Lambda ran but returned a response API Gateway could not parse.

Expected Lambda proxy response:

{
  "statusCode": 200,
  "headers": {
    "Content-Type": "application/json"
  },

}

Answer

“For API Gateway 5xx, I start with access logs to separate gateway, integration, and backend failures. Then I follow the correlation ID into Lambda, ALB, ECS, or the private VPC path.”

8. When should you use API Gateway versus ALB for HTTP services?

Direct answer

Use API Gateway when you need API management: auth, usage plans, throttling, request validation, transformations, serverless integrations, partner APIs, and per-route control.

Use ALB when you need load balancing for HTTP services, especially ECS/EKS/EC2 services, with simpler routing and lower operational complexity for traditional web apps.

Comparison

Requirement API Gateway ALB
Public API management Strong Limited
JWT/Lambda/IAM authorizers Strong Limited/different
API keys/usage plans REST API supports No native equivalent
Request validation/transformation Stronger Limited
Lambda integration Strong Supported but less API-management focused
ECS/EKS/EC2 HTTP routing Good via VPC Link Very strong
Web app load balancing Possible Natural fit
Long-lived connections/websockets Depends on API type ALB often simpler
Cost at high volume Depends Often cheaper for simple HTTP
WAF Both can integrate in architecture Strong with ALB/CloudFront
Private services API Gateway via VPC Link Native inside VPC

Use API Gateway for

external developer APIs
mobile/backend APIs
serverless APIs
multi-tenant throttling/API keys
request validation
centralized authorization
API lifecycle/versioning

Use ALB for

web applications
microservices behind ECS/EKS/EC2
HTTP routing by host/path
simple auth at app layer
high-throughput internal services
container workloads

Common pattern

Use both:

Client
   ↓
API Gateway
   ↓
VPC Link
   ↓
Internal ALB
   ↓
ECS services

API Gateway handles API controls. ALB handles service load balancing.

Answer

“API Gateway is an API control plane. ALB is a load balancer. If I need API governance, I use API Gateway. If I need simple HTTP load balancing to services, I use ALB. For private microservices exposed as public APIs, I often use both.”

9. How would you manage API versioning and deprecation cleanly?

Direct answer

I would treat APIs as contracts. Versioning and deprecation need explicit compatibility rules, documentation, telemetry, consumer communication, parallel support windows, and controlled shutdown.

Versioning options

Method Example Notes
URI version /v1/orders Simple and visible
Header version Accept: application/vnd.company.v2+json Cleaner URLs, more complex
Stage version API Gateway stage v1, v2 Operationally convenient
Domain/path mapping api.example.com/v1 Good public API pattern

Safe evolution rules

Change Safe?
Add optional response field Usually safe
Add optional request field Usually safe
Remove field Breaking
Rename field Breaking
Change type Breaking
Change auth/scopes Breaking or high-risk
Change error semantics Often breaking
Add enum value Risky for strict clients

Deprecation lifecycle

1. Announce v1 deprecation.
2. Publish v2 documentation and migration guide.
3. Run v1 and v2 in parallel.
4. Add response headers warning clients.
5. Monitor v1 usage by client/tenant/API key.
6. Contact remaining consumers.
7. Block new v1 onboarding.
8. Reduce quota for v1 if appropriate.
9. Retire v1 after agreed date.

Useful headers

Deprecation: true
Sunset: Wed, 31 Dec 2026 23:59:59 GMT
Link: <https://developer.example.com/migrate-v2>; rel="deprecation"

API Gateway design

Custom domain:
  api.example.com

Base path mappings:
  /v1 → API/stage v1
  /v2 → API/stage v2

Security concern

Never break authorization silently.

Bad:

v2 changes tenant filtering logic.
Old clients accidentally see broader data.

Version changes must include:

authorization regression tests
schema compatibility tests
consumer contract tests
rate-limit plan
rollback plan

Answer

“Clean API deprecation is not deleting an endpoint. It is a managed contract migration with telemetry, communication, parallel support, and a clear sunset date.”

10. How would you design a globally distributed API edge with API Gateway and CloudFront?

Direct answer

I would use CloudFront as the global edge layer, WAF for edge protection, API Gateway as the regional API control point, regional private integrations to backends, and Route 53/CloudFront origin failover or multi-origin routing for resilience.

Reference architecture

Global clients
   ↓
CloudFront
   ├── AWS WAF
   ├── TLS/certificates
   ├── caching for safe GETs
   ├── geo/routing policies if needed
   └── origin failover
        ↓
Regional API Gateway origin
        ↓
Authorizer / throttling / validation
        ↓
VPC Link
        ↓
Private ALB/ECS/EKS/Lambda
        ↓
Regional data stores

Multi-region model

CloudFront
   ├── Origin 1: API Gateway ap-southeast-1
   ├── Origin 2: API Gateway us-east-1
   └── Origin 3: API Gateway eu-west-1

Design concerns

Area Design
TLS ACM cert at CloudFront
WAF Attach at CloudFront for global protection
Caching Cache only safe/idempotent GET responses
Auth Usually do not cache personalized responses unless carefully keyed
Origin headers Add origin verification header if used
Regional failover Use origin groups or DNS strategy
Data residency Route users to correct Region
Throttling Edge plus regional API Gateway limits
Observability CloudFront logs + API Gateway logs + backend logs
Backend bypass Backends private only
Versioning Path-based /v1, /v2
DDoS CloudFront, Shield, WAF, throttling

Caching warning

Do not cache this carelessly:

GET /me
Authorization: Bearer token

If cache key does not include authorization/session context, one user may receive another user’s response.

Better:

Cache public catalog:
  GET /products

Do not cache:
  GET /me
  GET /account/balance
  GET /orders

or use careful cache key design.

Origin protection

CloudFront can add a custom header to origin requests:

X-Origin-Verify: random-secret-reference

API Gateway/backend checks it.

But do not rely on this alone for high-risk auth. Use proper authentication and private backend design.

Global failover challenge

Failing over API compute is easier than failing over data.

You must decide:

active/active reads?
single-writer Region?
global database?
eventual consistency?
tenant home Region?
DR RTO/RPO?

Answer

“CloudFront gives global edge, WAF, TLS, caching, and DDoS absorption. API Gateway gives regional API control. The hard part is not routing traffic globally; it is preserving authorization, cache safety, backend privacy, and data consistency across Regions.”

Controls and mitigations

Risk Controls
Backend bypass Private backend, VPC Link, SG restrictions, backend validation
Broken auth JWT/IAM/Lambda authorizer plus backend object-level authorization
CORS issues Exact origin allowlist, no wildcard credentials, Vary: Origin

Summary

The senior-level answer is:

“API Gateway is not just a reverse proxy. It is an API control boundary for authentication, authorization integration, throttling, validation, routing, observability, and tenant isolation. But it only works if the backend cannot be bypassed and the application still enforces business authorization.”

My production checklist:

Use HTTP API for simple low-cost JWT/IAM/Lambda-authorized APIs.
Use REST API for usage plans, API keys, request validation, and heavier API management.
Choose JWT for OIDC clients, IAM for AWS workloads, Lambda authorizers for custom logic.
Front private VPC services through VPC Link and internal ALB/NLB.
Prevent backend bypass with private networking and backend validation.
Use tenant-aware throttling beyond API keys where needed.
Treat CORS as browser policy, not API security.
Debug 5xx using API Gateway access logs, integration status, backend logs, and correlation IDs.
Use API Gateway for API governance; ALB for straightforward HTTP load balancing.
Version APIs as contracts with telemetry and deprecation windows.
Use CloudFront + WAF + API Gateway for global edge, but handle caching and data-region safety carefully.

Key Takeaways

  • API Gateway is a security and governance layer, not just a request router.
  • Route-level auth, throttling, and backend isolation are the highest-value controls.
