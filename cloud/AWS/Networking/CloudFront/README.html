CloudFront — Amazon CloudFront

Category: Networking
AWS Service Type: CDN and Edge Delivery
Key Docs: CloudFront Developer Guide

Overview

CloudFront is AWS's global content delivery network that caches and serves content from edge locations close to users. It accelerates static and dynamic content, reduces origin load, terminates TLS globally, supports signed URLs and cookies, integrates with WAF and Shield, and can run edge logic through CloudFront Functions and Lambda@Edge. CloudFront commonly fronts S3 static sites, ALBs, API Gateway endpoints, and media streaming origins. Because it is a user-facing edge layer, its configuration strongly influences performance, abuse resistance, and origin protection.

How It Works

Core Mechanics

Concept Description
Distribution CloudFront configuration containing origins, behaviors, caching, and security
Edge location Global POP that serves cached content near end users
Origin Backend source such as S3, ALB, API Gateway, or custom server
Cache behavior Path-based routing plus cache key, headers, methods, and TTL rules
Origin Access Control Signed service-to-origin access for private S3 origins
Signed URL/cookie Application-controlled authorization to private content
Lambda@Edge Full edge compute executed on viewer or origin request/response
CloudFront Functions Lightweight JavaScript for header, URL, or simple request logic

2 Sensitive data cached Cache key omits auth headers or cookies One user's data served to another High
3 Signed URL misuse Long-lived or poorly scoped signed URLs Unauthorized content access Med

6 Lambda@Edge logic bug Edge code changes requests or headers incorrectly Global outage at edge layer Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Origin bypass Restrict origin to CloudFront only using SG, custom header, or OAC Origin protection
2 Sensitive data cached Design cache key to include auth context or bypass cache for private paths Cache behavior
3 Signed URL misuse Short expiries and resource scoping for signed URLs/cookies Content authorization
4 Cache poisoning Allowlist only required headers, cookies, and query strings Cache policy design
5 No OAC/OAI for S3 Use Origin Access Control and deny direct S3 access S3 origin security
6 Lambda@Edge logic bug Stage and test edge code before global rollout Deployment discipline
7 WAF not attached Attach AWS WAF with managed and custom rules Edge security
8 TLS or cert misconfig Use ACM cert in us-east-1 and validate aliases carefully TLS configuration

Top 3 Critical Mitigations

  1. Prevent origin bypass: If clients can hit the ALB or S3 origin directly, CloudFront adds performance but not meaningful security.
  2. Be precise about cache keys: The most dangerous CloudFront failures are correctness failures, where private data gets cached and replayed to the wrong users.
  3. Use OAC for S3 origins: Modern CloudFront-to-S3 access should rely on Origin Access Control with bucket policies that deny everyone else.

1. How does CloudFront improve both performance and security posture?

Direct answer

CloudFront improves performance by caching content closer to users at edge locations, reducing origin load, reducing latency, and absorbing traffic spikes.

CloudFront improves security by acting as a controlled edge layer in front of origins, integrating with AWS WAF/Shield, enforcing TLS, hiding origins, supporting signed URLs/cookies, applying geo restrictions, and allowing request normalization or filtering at the edge.

Performance benefits

User in Singapore
   ↓
CloudFront edge location near user
   ↓
Cache hit served from edge
   ↓
No round trip to origin in another Region

Security benefits

Internet
   ↓
CloudFront
   ├── TLS termination
   ├── AWS WAF
   ├── DDoS absorption with AWS Shield
   ├── signed URL/cookie validation
   ├── geo restrictions
   ├── header normalization
   └── origin shielding/protection
   ↓
Origin: S3 / ALB / API Gateway / custom origin

Performance and security together

CloudFront capability Performance value Security value
Edge caching Lower latency, less origin load Reduces attack load reaching origin
TLS at edge Faster secure connection near user Enforces HTTPS
AWS WAF integration Blocks malicious traffic early Reduces origin exposure
Origin shielding Fewer origin requests Smaller origin attack surface
Signed URLs/cookies Controlled premium access Prevents unauthorized content access
Geo controls Regional traffic optimization Blocks disallowed geographies
Edge functions Lightweight redirects/normalization Reject bad requests early

Real-world example

For a video learning platform:

Static assets:
  videos, thumbnails, JS, CSS
  cached heavily at edge

Security:
  signed URLs for paid videos
  WAF blocks scraping and SQLi/XSS probes
  origin only accepts CloudFront traffic

Answer

“CloudFront improves performance by moving delivery closer to users and reducing origin calls. It improves security by making the edge the enforcement point before requests reach expensive or sensitive origins.”

2. What causes private-response caching bugs in CDNs?

Direct answer

Private-response caching bugs happen when CloudFront caches a response that is specific to one user, tenant, session, or authorization context and then serves it to another user.

This is one of the most serious CDN misconfigurations because it can create cross-user or cross-tenant data leakage.

Common causes

Cause Example
Missing cache key variation Cache does not vary by Authorization or cookie
Caching authenticated responses /me or /account cached publicly
Wrong Cache-Control Backend returns public for private data
Missing Vary header Origin varies by header but CDN cache key does not
Forwarding too few headers Backend response depends on header not in cache key
Forwarding too many headers Cache fragmentation and confusing behavior
Query string ignored /invoice?id=1 and /invoice?id=2 treated same
Cookie ignored Session-specific response cached as shared
Multi-tenant host/path misuse Tenant A and Tenant B share cache key

Dangerous example

Request:

GET /api/me
Authorization: Bearer token-user-a

Origin response:

{
  "userId": "user-a",
  "email": "alice@example.com"
}

If CloudFront cache key ignores Authorization, another user can request:

GET /api/me
Authorization: Bearer token-user-b

and receive User A’s cached response.

Correct behavior

For private APIs:

Cache-Control: no-store

or at least:

Cache-Control: private, no-cache

For shared public assets:

Cache-Control: public, max-age=31536000, immutable

Safe cache design

Public static content:
  cache aggressively

Authenticated personalized API:
  do not cache at CloudFront

Semi-private content:
  use signed URLs/cookies and carefully designed cache keys

Answer

“Private-response caching bugs occur when the cache key does not include everything that makes the response unique, or when private responses are cached as if they were public.”

3. How do CloudFront Functions and Lambda@Edge differ?

Direct answer

CloudFront Functions are lightweight JavaScript functions for very high-scale, low-latency request/response manipulation at the edge. They are best for simple viewer request/response logic.

Lambda@Edge is more powerful. It supports Node.js/Python, can run on viewer and origin events, has longer execution time, and can handle heavier customization. AWS documents CloudFront Functions and Lambda@Edge as two edge compute options with different supported events, runtimes, and limits. (docs.aws.amazon.com)

Comparison

Area CloudFront Functions Lambda@Edge
Runtime Lightweight JavaScript Node.js or Python
Best for Simple header/URI logic Heavier edge logic
Events Viewer request/response Viewer and origin request/response
Latency Very low Higher
Cost Lower Higher
Network access No external network calls More capable, but still constrained
Request body access No More options depending on event
Use case Redirects, header normalization, simple auth checks Origin selection, complex auth, content modification

AWS’s Lambda@Edge documentation states Lambda@Edge functions are authored in US East (N. Virginia), us-east-1, and then replicated to CloudFront edge locations. (docs.aws.amazon.com)

CloudFront Functions examples

Redirect HTTP to HTTPS
Normalize URI case
Add security headers
Block bad user agents
Rewrite `/old-path` to `/new-path`
Simple signed token check

Lambda@Edge examples

Dynamic origin selection
Advanced authentication
Origin request rewriting
A/B testing with more logic
Custom response generation
More complex header/cookie processing

Answer

“CloudFront Functions are for tiny, fast, cheap logic at massive scale. Lambda@Edge is for heavier edge customization, especially when origin request/response handling is needed.”

4. How would you protect an ALB-backed application from origin bypass?

Direct answer

I would ensure users cannot access the ALB directly and must go through CloudFront. For ALB origins, common controls include custom origin headers validated at the ALB/application, AWS-managed CloudFront prefix lists in security groups, WAF at CloudFront, and strict backend authorization.

AWS documents a pattern where CloudFront adds a custom header to origin requests and the ALB forwards only requests containing that header; AWS warns that this relies on keeping the header name and value secret. (docs.aws.amazon.com) AWS also documents using custom headers to prevent users from bypassing CloudFront and accessing custom origins directly. (docs.aws.amazon.com)

Secure architecture

Internet
   ↓
CloudFront + WAF
   ↓
Origin request includes secret header
   ↓
ALB listener rule checks header
   ↓
Target group / ECS / EC2

Controls

Control Purpose
ALB security group allows CloudFront origin-facing IPs/prefix list Reduces direct internet access
Custom origin header Allows ALB/app to verify CloudFront path
ALB listener rule Reject requests without header
WAF at CloudFront Blocks attacks before ALB
Backend validation Do not blindly trust headers from public clients
No direct DNS advertised Avoid exposing ALB name
Monitor ALB direct traffic Detect bypass attempts
mTLS/origin auth where appropriate Stronger origin authentication

Example ALB rule

If header X-Origin-Verify equals secret-value:
  forward to target group

Else:
  return 403

Important warning

A secret custom header helps, but it is not perfect:

If header leaks, attacker can send it directly.
If ALB is open to all IPs, bypass risk remains.
If backend trusts X-Forwarded-* from client, spoofing risk remains.

Better defense-in-depth:

CloudFront prefix-list SG restriction
custom header validation
CloudFront WAF
backend authorization
monitoring

Answer

“Origin bypass defeats the edge security model. I would make the ALB accept only CloudFront-originated traffic and still make the backend enforce authorization.”

5. What are the risks of forwarding too many headers or query strings?

Direct answer

Forwarding too many headers, cookies, or query strings reduces cache efficiency, increases origin load, increases cost, and can create security or correctness bugs.

CloudFront cache keys can vary based on headers, cookies, and query strings. AWS recommends caching based only on the values that actually affect the origin response; for query strings, AWS explicitly says to configure caching only on the parameters needed to improve cache hit ratio and reduce origin load. (docs.aws.amazon.com) AWS also notes that headers like Date and User-Agent can have many possible values and are not recommended for cache key variation because they cause significantly more origin requests. (docs.aws.amazon.com)

Risks

Risk Explanation
Cache fragmentation Too many cache variants
Low hit ratio More requests go to origin
Higher latency Fewer edge hits
Higher origin cost ALB/API/backend sees more traffic
Header abuse Attackers vary headers to bypass cache
Query-string cache busting Random query params force misses
Data leakage Incorrect cache key can mix private/public content
Hard debugging Behavior varies by invisible headers
WAF/rate-limit bypass patterns Attackers create high-cardinality variants

Example

Bad cache policy:

Forward all headers
Forward all query strings
Forward all cookies

Effect:

GET /logo.png?random=abc
GET /logo.png?random=def
GET /logo.png?random=ghi

Each may become a different cache key, causing origin hits.

Better

For static assets:

Forward no cookies.
Forward no unnecessary headers.
Ignore query strings unless used for versioning.

For product images:

Cache only on:
  size
  format

Not:

utm_source
fbclid
sessionId
timestamp

Answer

“Every forwarded value can become part of the cache key or origin behavior. I forward only what the origin truly needs.”

6. How would you design signed URL access for premium content?

Direct answer

I would use CloudFront signed URLs or signed cookies to restrict access to premium content. The application authenticates the user, checks entitlement, then issues a short-lived signed URL or cookie for the specific content or path.

CloudFront signed URLs and signed cookies control who can access content; signed URLs are best when access is to individual files, while signed cookies are useful when access covers multiple restricted files. (docs.aws.amazon.com)

Architecture

User logs in
   ↓
Application checks subscription entitlement
   ↓
Application creates signed CloudFront URL
   ↓
User requests premium video/file through CloudFront
   ↓
CloudFront validates signature and policy
   ↓
If valid, fetches from private origin

Example

GET /premium/videos/course-123/lesson-4.mp4

Signed policy includes:

resource path
expiry time
optional IP restriction
key pair ID / key group
signature

Design choices

Use signed URL Use signed cookie
One file or one download Multiple files under path
Shareable short-lived link Session-like access
Document/video download Course with many chunks/assets

Security controls

Short expiration time.
Bind to path/resource.
Use key groups/trusted public keys.
Rotate signing keys.
Do not sign overly broad wildcards unless needed.
Keep origin private.
Use HTTPS only.
Avoid logging full signed URLs where possible.

Premium video example

For HLS video:

Signed cookie for:
/premium/course-123/*

Reason:

HLS video has many segment files.
One signed URL per segment is operationally painful.
Signed cookie covers the authorized path.

Attack scenario

If signed URL is valid for 7 days and not IP/path constrained:

Subscriber copies URL
   ↓
Posts to public forum
   ↓
Thousands of unauthorized users access content

Mitigation:

short TTL
path scope
optional IP binding where practical
abuse monitoring
token refresh through authenticated app

Answer

“For premium content, the app makes the authorization decision, and CloudFront enforces access using short-lived signed URLs or cookies. The origin must remain private.”

7. Why must ACM certificates for CloudFront live in us-east-1?

Direct answer

CloudFront is a global service, but ACM certificates used for viewer HTTPS on CloudFront must be requested or imported in US East (N. Virginia), us-east-1. AWS documents this as a CloudFront certificate requirement. (docs.aws.amazon.com)

Important nuance

There are two TLS legs:

Viewer → CloudFront
CloudFront → Origin

For viewer-facing HTTPS:

ACM certificate must be in us-east-1.

For origin HTTPS to an ALB:

The ALB certificate can be in the ALB’s Region.

AWS’s CloudFront certificate docs make this distinction: the viewer certificate must be in us-east-1, while a certificate used by an Elastic Load Balancer origin can be requested or imported in any AWS Region. (docs.aws.amazon.com)

Example

CloudFront alternate domain:
  api.example.com

Viewer certificate:
  ACM us-east-1

Origin:
  ALB in ap-southeast-1

Origin certificate:
  ACM ap-southeast-1

Answer

“CloudFront uses ACM certificates from us-east-1 for the viewer-facing global distribution. Regional origins such as ALBs use certificates in their own Regions.”

8. How would you debug a global outage caused by an edge configuration change?

Direct answer

I would treat it as a high-blast-radius edge incident: identify the exact distribution/config version changed, confirm symptoms by geography and behavior, roll back to the last known good config if needed, then analyze logs and propagation effects.

Debug flow

Global errors reported
   ↓
Check recent CloudFront distribution changes
   ↓
Identify config version / deployment ID / changed behavior
   ↓
Compare current config to last known good
   ↓
Check CloudFront metrics:
      4xx, 5xx, cache hit rate, origin latency
   ↓
Test from multiple Regions
   ↓
Check WAF logs / Function logs / Lambda@Edge logs
   ↓
Check origin health
   ↓
Rollback config if high impact
   ↓
Invalidate only if cache object issue

Common edge-change outage causes

Change Failure
Cache policy changed Private data cached or cache miss storm
Origin request policy changed Required header no longer forwarded
Function/Lambda@Edge deployed Bad rewrite/redirect/auth logic
WAF rule changed Legit traffic blocked
Origin path changed 404/502 errors
TLS policy/cert changed TLS failures
Allowed methods changed POST/OPTIONS broken
CORS headers changed Browser clients fail
Origin failover changed Traffic sent to wrong origin
Query forwarding changed Backend receives wrong request shape

Rollback principle

If user impact is high:

rollback first
debug second

CloudFront configuration changes can take time to propagate globally, so measure impact during propagation and do not make many rapid untracked changes.

What to inspect

CloudFront distribution config history from IaC/Git
CloudTrail UpdateDistribution events
CloudFront standard/real-time logs
WAF logs
CloudFront Function test/publish history
Lambda@Edge version association
Origin ALB/API Gateway logs
Synthetic checks from multiple geographies

Answer

“For global edge outages, I first establish whether the edge or origin is failing, then roll back the distribution configuration if blast radius is broad. Edge changes need change control because one bad cache or function rule can break the world.”

9. What metrics matter most for CloudFront health and abuse monitoring?

Direct answer

The most important metrics are request volume, 4xx/5xx error rates, cache hit ratio, origin latency, bytes downloaded/uploaded, WAF blocks, geographic anomalies, top URLs, top IPs, and unusual header/query variation.

Core health metrics

Metric/signal Why it matters
Requests Traffic baseline and spikes
4xx error rate Client/auth/WAF/routing problems
5xx error rate Origin or edge integration problems
Total error rate Overall health
Cache hit ratio Performance and origin protection
Origin latency Backend health
Origin error rate Origin failure
Bytes downloaded/uploaded Cost and abuse signal
WAF blocked requests Attack visibility
Top URLs Hot object or abuse target
Top IPs/ASNs/countries Bot/DoS analysis
Function validation errors Edge code issue
TLS errors Certificate/policy/client issues

Abuse indicators

Sudden request spike from one ASN/country.
High 403 from WAF.
High 404 for random paths.
Low cache hit ratio with random query strings.
Many requests for expensive uncached endpoints.
Large egress spike.
Credential stuffing patterns on login.
Bot user-agent concentration.

CloudFront logs

Use:

standard logs for broad analysis
real-time logs for high-speed incident response
WAF logs for blocked/allowed attack patterns
origin logs for backend correlation

Useful alerts

5xx error rate > threshold
4xx spike by path
cache hit ratio drops sharply
origin latency increases
origin request count spikes
WAF block rate spikes
egress bytes anomaly
requests by unknown country/ASN spike

Answer

“CloudFront health is not just 5xx. A cache hit ratio drop can be the first sign of a cache-busting attack or bad cache policy that will overload the origin.”

10. How would you front API traffic differently from static asset traffic in CloudFront?

Direct answer

I would use separate CloudFront behaviors, cache policies, origin request policies, WAF rules, and security controls for APIs versus static assets.

Static assets should be cached aggressively. API traffic should usually be cached minimally or not at all unless responses are clearly public and cache-safe.

Distribution design

CloudFront distribution
├── Behavior: /static/*
│   ├── Origin: S3
│   ├── Cache: long TTL
│   ├── Query strings: limited or none
│   └── Cookies: none
│
├── Behavior: /images/*
│   ├── Origin: S3/image service
│   ├── Cache: long/medium TTL
│   └── Query keys: width, format only
│
└── Behavior: /api/*
    ├── Origin: API Gateway / ALB
    ├── Cache: disabled or short TTL
    ├── Forward Authorization header if needed
    ├── Forward required headers only
    ├── WAF API rules
    └── Rate limiting / auth at API layer

Static asset behavior

Cache-Control: public, max-age=31536000, immutable
File names include content hash:
  app.abc123.js
  styles.def456.css

Security:

Origin Access Control for S3
No public S3 bucket access
Limited methods: GET/HEAD
No cookies
Minimal headers/query strings

API behavior

Cache-Control: no-store for personalized APIs
Allowed methods: GET/POST/PUT/PATCH/DELETE/OPTIONS as needed
Forward Authorization only where required
Forward specific headers only
Do not cache authenticated/private responses

Security:

WAF rules for API abuse
API Gateway authorizer
rate limiting/throttling
CORS exact allowlist
origin protection
correlation IDs

Example

Good:

/static/app.abc123.js
  cached for 1 year

/api/me
  not cached

/api/products?category=books
  maybe cached 30 seconds if public and safe

Bad:

Cache all /api/* for 5 minutes

This can leak private data or serve stale state-changing responses.

Answer

“Static content wants aggressive caching and minimal forwarding. API traffic wants authorization, careful cache control, precise forwarding, and usually no shared caching for personalized responses.”

Attack tree: CloudFront security or reliability failure

Goal: Leak data, bypass edge controls, overload origin, or cause global outage

├── 1. Private-response cache leak
│   ├── Authorization header not in cache key
│   ├── Cookies ignored for personalized page
│   ├── Backend marks private data public
│   ├── Missing Vary behavior
│   └── Tenant ID not part of cache key

├── 2. Origin bypass
│   ├── ALB public and accepts direct requests
│   ├── Custom header leaked
│   ├── Security group allows all internet
│   ├── Backend trusts spoofable headers
│   └── WAF only at CloudFront

├── 3. Cache-busting attack
│   ├── Random query strings
│   ├── High-cardinality headers
│   ├── Forward all cookies

│   └── Origin request storm

├── 4. Signed URL abuse
│   ├── Long expiration
│   ├── Broad wildcard resource
│   ├── URL leaked publicly
│   ├── Signing key not rotated
│   └── Origin remains public

├── 5. Edge code failure
│   ├── Bad CloudFront Function rewrite
│   ├── Lambda@Edge auth bug
│   ├── Infinite redirect
│   ├── Header removed accidentally
│   └── Global deployment without canary

├── 6. Certificate/TLS failure
│   ├── ACM certificate not in us-east-1
│   ├── Expired alternate domain cert

    ├── No real-time logs

Controls and mitigations

Risk Controls
Private cache leak No-store for private APIs, correct cache keys, test with multiple users
Origin bypass ALB SG restriction, custom origin header, backend validation
Cache-busting Forward only required headers/query/cookies
Premium content theft Short-lived signed URLs/cookies, private origin, key rotation
Edge-code outage Staged deployment, tests, rollback plan, monitoring
Certificate issue ACM in us-east-1 for viewer certs, expiry monitoring
API caching bug Separate /api/* behavior with caching disabled/default no-store
Static asset inefficiency Hashed filenames and long TTLs
Abuse WAF, rate limiting, bot controls, anomaly alerts
Debugging gap CloudFront logs, WAF logs, origin logs, correlation IDs

Summary

“CloudFront is both a performance layer and a security control point. The main design challenge is separating public cacheable content from private personalized responses, protecting the origin from bypass, and forwarding only the headers/query strings/cookies that are truly required.”

My production checklist:

Cache static assets aggressively with hashed filenames.
Do not cache authenticated/private API responses unless explicitly safe.
Use separate behaviors for static, images, APIs, and premium content.
Protect ALB/custom origins from direct access.
Use WAF and Shield at the edge.
Use CloudFront Functions for lightweight viewer logic.
Use Lambda@Edge for heavier viewer/origin logic.
Use signed URLs/cookies for premium content.
Keep viewer ACM certificates in us-east-1.
Monitor error rates, cache hit ratio, origin latency, WAF blocks, and egress anomalies.
Treat CloudFront config changes as high-blast-radius production changes.

Key Takeaways

  • CloudFront is a performance layer and a security boundary when origins are protected properly.
  • Cache-key design is the main correctness challenge.
  • OAC is the modern control for private S3 origins.
  • Edge code is powerful but globally risky if deployed carelessly.
  • CloudFront should usually be paired with WAF and explicit origin hardening.
