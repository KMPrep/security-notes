Direct Connect — AWS Direct Connect

Category: Networking
AWS Service Type: Dedicated Private Connectivity
Key Docs: Direct Connect User Guide

Overview

Direct Connect provides a dedicated private network link between on-premises environments and AWS, bypassing the public internet for predictable bandwidth, lower jitter, and consistent latency. It is commonly used for hybrid architectures, large data transfer, private access to VPCs, and regulated environments that want stable private connectivity. Direct Connect can terminate into a virtual interface for VPC private connectivity, public AWS services, or Transit Gateway connectivity, and often pairs with VPN for encrypted backup or additional path resilience.

How It Works

Core Mechanics

Concept Description
Dedicated connection Physical port provisioned directly to a customer or partner
Hosted connection Partner-provided carve-out on existing Direct Connect capacity
Private VIF Virtual interface to reach VPC resources via VGW or DXGW
Public VIF Virtual interface to reach public AWS services privately
Transit VIF Virtual interface to reach Transit Gateway through DX gateway
Direct Connect gateway Global construct linking DX to one or more VPCs or TGWs
BGP Routing protocol used to exchange prefixes over the link
MACsec Optional link-layer encryption on supported dedicated connections

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Single circuit dependency Use dual DX links in separate locations and devices Resilience architecture
2 No encrypted overlay Use IPsec VPN overlay or MACsec where supported Data protection
3 Bad BGP advertisements Apply prefix filters, max-prefix, and route review procedures Routing governance
4 Overlapping CIDRs Maintain central IPAM and approve hybrid address plans Address planning
5 No backup VPN Deploy Site-to-Site VPN as failover path Hybrid availability
6 Wrong VIF type Match VIF choice to actual connectivity requirement DX design
7 MACsec unsupported assumption Document whether encryption is MACsec, VPN, or neither Control validation
8 Insufficient route governance Segment on-prem to AWS paths via TGW route tables and filters Network segmentation

Top 3 Critical Mitigations

  1. Treat Direct Connect as resilient only when dual-homed: One circuit is a convenience link, not an enterprise-grade hybrid connectivity design.
  2. Add encryption deliberately: Private transport is not the same as encrypted transport. Use MACsec where available or overlay VPN where required.
  3. Govern BGP and address planning centrally: Hybrid outages are often routing mistakes, not hardware failures.

1. How does Direct Connect differ from Site-to-Site VPN?

Direct answer

AWS Direct Connect gives you a dedicated private network connection from your data center, office, or colocation environment into AWS. AWS Site-to-Site VPN creates encrypted IPsec tunnels over the public internet, or in some designs over Direct Connect.

AWS describes Direct Connect as linking your internal network to an AWS Direct Connect location over a standard Ethernet fiber-optic cable, bypassing internet service providers in your network path. AWS Site-to-Site VPN enables connectivity between your VPC and remote network through VPN connections. (AWS Documentation)

Simple comparison

Area Direct Connect Site-to-Site VPN
Transport Dedicated private circuit Encrypted tunnel, usually over internet
Encryption Not encrypted by default IPsec encrypted by default
Performance More predictable latency/bandwidth Internet-dependent variability
Provisioning time Longer, needs circuit/partner/colo Faster to set up
Cost model Port/hour, data transfer, partner/cross-connect costs VPN hourly + data transfer
Bandwidth Higher, dedicated options Lower and variable compared with DX
Best for Production hybrid, high throughput, stable latency Backup, quick setup, encrypted connectivity
Failure dependency DX location/provider/cross-connect Internet path and VPN endpoints
Operational complexity Higher Lower

Architecture view

Site-to-Site VPN

On-prem router/firewall
   ↓ IPsec over internet
AWS VPN endpoint
   ↓
VGW / Transit Gateway
   ↓
VPCs

Direct Connect

On-prem router
   ↓ private circuit / cross-connect
Direct Connect location
   ↓ virtual interface
AWS network
   ↓
VPC / Transit Gateway / public AWS services

Common production pattern

For critical hybrid connectivity, I usually do Direct Connect plus VPN backup:

Primary path:
  Direct Connect

Backup path:
  Site-to-Site VPN

Routing:
  BGP prefers Direct Connect
  VPN used during DX failure

AWS also documents combining Direct Connect and Site-to-Site VPN to get IPsec encryption with low latency and increased bandwidth from Direct Connect. (AWS Documentation)

Answer

“Direct Connect gives predictable private connectivity. VPN gives encrypted connectivity and faster setup. For production hybrid networks, I often use Direct Connect as the primary path and VPN as encrypted backup or overlay.”

2. What are the roles of private, public, and transit virtual interfaces?

Direct answer

A Direct Connect connection needs a virtual interface, or VIF, to reach a specific AWS routing domain.

AWS documents three main VIF types: public virtual interfaces to public AWS services, private virtual interfaces to VPC resources, and transit virtual interfaces to Transit Gateways through a Direct Connect gateway. (AWS Documentation)

VIF types

VIF type Connects to Use case
Private VIF VPC through VGW or DX gateway Private IP connectivity to VPC workloads
Public VIF Public AWS service endpoints S3, DynamoDB public endpoints, public AWS APIs
Transit VIF Direct Connect gateway associated with Transit Gateway Many VPCs/accounts through TGW hub

Private VIF

On-prem
   ↓ DX private VIF
Virtual Private Gateway or Direct Connect Gateway
   ↓
VPC private IPs

Use when:

You want private connectivity to EC2/RDS/EKS/ECS in a VPC.

Public VIF

On-prem
   ↓ DX public VIF
AWS public service endpoints
   ↓
S3 / DynamoDB / public AWS APIs

Use when:

You want to reach public AWS services over DX instead of internet.

Important: public VIF does not mean “internet”; it means AWS public IP prefixes over Direct Connect.

Transit VIF

On-prem
   ↓ DX transit VIF
Direct Connect Gateway
   ↓
Transit Gateway
   ↓
Many VPCs/accounts

AWS states that a transit VIF should be used to access one or more Amazon VPC Transit Gateways associated with Direct Connect gateways. (AWS Documentation)

Answer

“Private VIF is for VPC private connectivity, public VIF is for AWS public service endpoints, and transit VIF is for scalable multi-VPC connectivity through Transit Gateway.”

3. How would you design resilient hybrid connectivity using Direct Connect?

Direct answer

I would design for device, circuit, location, provider, Region, and routing resilience. A single Direct Connect circuit is not resilient enough for critical production workloads.

Resilient design

Data Center A router 1
   ↓ DX circuit 1
DX Location 1
   ↓
AWS

Data Center A router 2
   ↓ DX circuit 2
DX Location 2
   ↓
AWS

Optional backup:
Site-to-Site VPN over internet

Stronger design

On-prem DC1 ── DX location A ── AWS
On-prem DC1 ── DX location B ── AWS
On-prem DC2 ── DX location C ── AWS
Backup VPN over internet

Resilience dimensions

Failure Control
Router failure Dual customer routers
Cross-connect failure Multiple cross-connects
DX port failure Multiple DX connections
DX location failure Diverse DX locations
Provider failure Diverse providers where possible
AWS device maintenance Redundant BGP sessions
Region issue Multi-Region routing design
Misrouted prefix BGP filters and route governance
Encryption requirement VPN over DX or MACsec where supported

BGP design

Use eBGP on each VIF.
Advertise only approved prefixes.
Prefer primary DX using local preference/AS path.
Use VPN as lower-preference backup.
Monitor BGP session state.
Test failover regularly.

Example active/active

DX1 advertises 10.10.0.0/16
DX2 advertises 10.10.0.0/16

On-prem uses ECMP or local preference depending on architecture.
AWS receives redundant paths.

Answer

“Direct Connect resilience is not just two cables. I want diverse routers, circuits, locations, providers, BGP sessions, and a tested VPN fallback path.”

4. Why is BGP governance so important for hybrid links?

Direct answer

BGP controls what networks are reachable across the hybrid boundary. A bad BGP advertisement can blackhole traffic, leak routes, create asymmetric routing, override intended paths, or expose networks that should not be reachable.

Why BGP is high-risk

On-prem advertises routes to AWS.
AWS advertises routes to on-prem.
Routers choose paths based on BGP policy.
A mistake can affect many VPCs and data centers.

Common BGP failure modes

Failure Example
Route leak On-prem advertises internet default route incorrectly
Blackhole More-specific route points to dead path
Asymmetric routing Forward path DX, return path VPN
Over-advertisement Dev networks can reach prod VPCs
Prefix hijack internally Wrong team advertises another team’s CIDR
Route table explosion Too many prefixes hit limits
Backup not working AS path/local preference wrong
Failover loop DX and VPN preferences conflict

Governance controls

Prefix allowlists.
Max-prefix limits.
Route filtering inbound and outbound.
Approval workflow for new prefixes.
Separate prod/non-prod route domains where needed.
BGP communities/local-pref policy.
Monitoring for unexpected route changes.
Documented ASNs and ownership.
Regular failover testing.

Example route leak

Bad:

On-prem advertises 0.0.0.0/0 to AWS.

Impact:

VPC traffic tries to reach internet through on-prem.
Firewalls overload.
Applications fail.
Security inspection path changes unexpectedly.

Answer

“BGP is effectively the authorization system for network reachability. Without governance, one bad route can create an outage or a cross-environment exposure.”

5. When do you still need encryption over a private circuit?

Direct answer

Direct Connect is private, but it is not automatically encrypted end-to-end. You still need encryption when the data is sensitive, regulated, crosses third-party facilities, traverses partner networks, or when your security policy requires cryptographic protection independent of transport.

When encryption is needed

Scenario Why
Regulated data Compliance requires encryption in transit
Sensitive customer/financial data Defense-in-depth
Third-party colocation/provider path Private does not mean trusted
Multi-tenant network provider Reduce exposure risk
Zero-trust network model Network location is not trust
Cross-border traffic Data protection requirements
Public VIF to AWS public services Use TLS/IPsec
High-value admin traffic Protect credentials and control-plane traffic

Encryption options

TLS at application layer
IPsec VPN over Direct Connect
MACsec on supported dedicated DX connections
mTLS for service-to-service traffic
Database TLS
SSH/RDP over secure bastion/SSM

AWS documents a pattern combining Direct Connect with Site-to-Site VPN to get IPsec encryption plus the more consistent network experience of Direct Connect. (AWS Documentation)

Key principle

Private path reduces exposure.
Encryption protects confidentiality and integrity.
They are not the same control.

Answer

“Direct Connect gives private transport, not automatic end-to-end encryption. For sensitive or regulated traffic, I still use TLS, IPsec over DX, MACsec where available, or mTLS.”

6. How does a Direct Connect gateway simplify multi-VPC designs?

Direct answer

A Direct Connect gateway lets you use one Direct Connect connection/VIF architecture to reach multiple VPCs across accounts and Regions, instead of creating separate private VIFs for every VPC.

AWS documents that you can use a Direct Connect gateway to connect over a private VIF to one or more VPCs in the same or different accounts and Regions. (AWS Documentation)

Without DX gateway

DX connection
   ├── private VIF to VPC A
   ├── private VIF to VPC B
   ├── private VIF to VPC C
   └── private VIF to VPC D

This becomes hard to manage.

With DX gateway

On-prem
   ↓
DX connection
   ↓
Private VIF
   ↓
Direct Connect Gateway
   ├── VGW association to VPC A
   ├── VGW association to VPC B
   └── VGW association to VPC C

With Transit Gateway

On-prem
   ↓
DX transit VIF
   ↓
Direct Connect Gateway
   ↓
Transit Gateway
   ├── VPC A
   ├── VPC B
   ├── VPC C
   └── VPC D

AWS states a Direct Connect gateway supports communication between attached transit VIFs and associated transit gateways. (AWS Documentation)

Benefits

Fewer VIFs
Centralized hybrid connectivity
Multi-account support
Multi-Region support
Cleaner routing model
Easier onboarding of VPCs
Works well with Transit Gateway hub-and-spoke

Answer

“A Direct Connect gateway abstracts the DX connection from individual VPCs. Combined with Transit Gateway, it becomes the scalable pattern for many VPCs and accounts.”

7. What are the tradeoffs between hosted and dedicated connections?

Direct answer

A dedicated connection is a physical Ethernet connection associated with a single customer. A hosted connection is provisioned for you by an AWS Direct Connect Partner. AWS documents these as the two Direct Connect connection types. (AWS Documentation)

Comparison

Area Dedicated connection Hosted connection
Provisioning Requested from AWS Requested through DX Partner
Physical port Dedicated to customer Partner-provisioned
Bandwidth Larger standard port options Flexible/sub-rate options from partner
Control More direct Partner-dependent
Operational dependency AWS + colo/cross-connect AWS + partner + provider
Use case Enterprise, high throughput, strict control Faster onboarding, smaller bandwidth
Change speed Port speed fixed after request Partner may offer flexible changes
Commercial model AWS port + colo costs Partner contract/pricing

AWS notes that for dedicated connections, you request through the Direct Connect console, while hosted connections are requested through a Direct Connect Partner; for dedicated connections, port speed cannot be changed after the connection request, while hosted speeds may be partner-changeable. (AWS Documentation) AWS also notes you cannot request hosted connections directly through the console; a Direct Connect Partner creates/configures it and you accept it. (AWS Documentation)

Choose dedicated when

Need high bandwidth.
Need strict operational control.
Have colocation/cross-connect capability.
Need predictable enterprise connectivity.

Choose hosted when

Need smaller bandwidth.
Want partner-managed connectivity.
Need faster/simple onboarding.
Do not have direct DX location presence.

Answer

“Dedicated gives more control and is typical for large enterprise connectivity. Hosted is partner-delivered and often better for smaller or faster-to-provision connectivity, but adds partner dependency.”

8. How would you detect and troubleshoot asymmetric routing over DX?

Direct answer

Asymmetric routing happens when traffic from on-prem to AWS uses one path, but return traffic uses another. It can break stateful firewalls, NAT, inspection tools, and troubleshooting assumptions.

Example

On-prem → AWS:
  Direct Connect

AWS → on-prem:
  Site-to-Site VPN

Stateful firewall:
  sees only one side of flow
  drops return packets

Detection signals

Application timeouts but one-way packet counters increase.
Firewall logs show SYN without SYN-ACK or reverse.
VPC Flow Logs show traffic leaving different path.
BGP route tables show different best paths.
Traceroute differs by direction.
VPN tunnel metrics increase unexpectedly.
DX metrics look normal but app fails.

Troubleshooting steps

1. Identify source/destination IP pair.
2. Check on-prem routing table for AWS prefix.
3. Check AWS route tables/TGW route tables for on-prem prefix.
4. Check BGP advertised and received routes.
5. Look for more-specific prefixes.
6. Compare AS path/local preference/MED.
7. Check firewall/NAT state logs.
8. Check VPC Flow Logs and TGW Flow Logs if available.
9. Test with controlled traceroute/TCP probes.
10. Temporarily withdraw/prefer routes to confirm hypothesis.

Common causes

Cause Example
More-specific route /24 via VPN beats /16 via DX
Local preference mismatch On-prem prefers DX, AWS prefers VPN
Route propagation issue TGW table missing route
Firewall path insertion Only one direction inspected
NAT on one path Return path cannot match session
Different route domains Prod/non-prod route leak
Backup path accidentally preferred AS path prepending wrong

Controls

Consistent BGP policy.
Explicit primary/backup preferences.
Prefix filtering.
Route table segmentation.
No accidental more-specific advertisements.
Flow logs and route monitoring.
Failover testing.
Stateful firewall path design.

Answer

“Asymmetric routing is usually a route-policy problem made visible by stateful devices. I troubleshoot it by comparing BGP decisions and packet paths in both directions.”

9. How would you connect many accounts and VPCs to on-prem without creating a routing mess?

Direct answer

I would use a hub-and-spoke design with Transit Gateway, Direct Connect gateway, account vending standards, route table segmentation, prefix governance, and centralized network ownership.

AWS describes Transit Gateway as a network transit hub that interconnects VPCs and consolidates Amazon VPC routing configuration in one place. (AWS Documentation)

Scalable architecture

On-prem data centers
   ↓
Direct Connect
   ↓
Direct Connect Gateway
   ↓
Transit Gateway
   ├── Prod VPCs
   ├── Non-prod VPCs
   ├── Shared services VPC
   ├── Security inspection VPC
   └── Egress VPC

Multi-account pattern

Network account:
  owns Transit Gateway
  owns DX Gateway associations
  owns route tables

Application accounts:
  own VPCs
  attach to TGW using AWS RAM sharing

Route table segmentation

TGW route table: prod
  prod VPCs ↔ on-prem prod
  no direct non-prod access

TGW route table: non-prod
  non-prod VPCs ↔ on-prem non-prod

TGW route table: shared-services
  controlled shared access

TGW route table: inspection
  centralized firewall path

Governance

Central IPAM.
No overlapping CIDRs.
CIDR request/approval workflow.
Route advertisements approved.
Prefix summarization.
Separate prod/non-prod route domains.
AWS RAM for TGW sharing.
SCPs preventing rogue VPN/DX attachments.
Automated route validation.

Avoid this

Every account creates its own VPN.
Every VPC has custom peering.
Random CIDRs.
Manual route tables.
No central visibility.

Result:

routing loops
overlaps
unintended connectivity
hard troubleshooting
security bypass

Answer

“At scale, hybrid connectivity is a network product, not a per-team activity. I centralize it with Transit Gateway, Direct Connect gateway, IPAM, route segmentation, and route governance.”

10. What is the operational blast radius of overlapping CIDRs in hybrid networks?

Direct answer

Overlapping CIDRs can break routing, prevent connectivity, cause traffic to go to the wrong environment, block mergers/cloud onboarding, make firewall policy ambiguous, and force expensive renumbering or NAT workarounds.

Example

On-prem data center:
  10.20.0.0/16

AWS VPC A:
  10.20.0.0/16

Problem:

When a host sends traffic to 10.20.5.10,
is that on-prem or AWS?

Routing cannot distinguish identical prefixes.

Blast radius

Impact Explanation
VPC cannot connect to on-prem Same CIDR conflict
TGW attachment blocked/limited Overlaps break routing intent
Incorrect routing Traffic follows local route unexpectedly
Security policy ambiguity Firewall rules match multiple environments
DNS confusion Name resolves to unreachable/ambiguous IP
M&A integration pain New network overlaps existing ranges
Forced NAT Adds complexity and troubleshooting burden
Renumbering Expensive and risky
DR failure Secondary environment overlaps primary/on-prem
Logging ambiguity Same IP appears in multiple places

NAT workaround

You may use NAT to map one side:

AWS VPC real CIDR:
  10.20.0.0/16

Presented to on-prem as:
  172.20.0.0/16

But NAT creates issues:

harder observability
broken IP-based identity
complex firewall rules
application allowlist changes
bidirectional connection challenges

Prevention

Central IPAM.
Allocate AWS CIDRs from non-overlapping enterprise pools.
Require CIDR approval before VPC creation.
Reserve space for future Regions/accounts.
Detect overlaps automatically in account vending.
Use IPv6 strategy where appropriate.
Document ownership of every prefix.

Answer

“Overlapping CIDRs are a long-term operational tax. They turn simple routing into NAT, exceptions, and ambiguity. The best mitigation is centralized IPAM before teams create VPCs.”

Summary

“Direct Connect gives predictable private connectivity, but the hard parts are resilience, routing governance, encryption decisions, and multi-account scale. A hybrid network can fail not only because a circuit goes down, but because one wrong route changes reachability for hundreds of systems.”

My production checklist:

Use Direct Connect for predictable private hybrid connectivity.
Use Site-to-Site VPN for encrypted backup or fast setup.
Use private VIF for VPCs, public VIF for AWS public endpoints, transit VIF for TGW.
Build redundant DX paths across routers, circuits, providers, and locations.
Govern BGP with prefix filters, max-prefix, and approval workflows.
Encrypt sensitive traffic even over private circuits.
Use Direct Connect gateway and Transit Gateway for multi-VPC/account scale.
Choose hosted connections for partner-managed/flexible access; dedicated for enterprise control.
Detect asymmetric routing with BGP tables, flow logs, firewall logs, and path testing.
Use central IPAM to prevent overlapping CIDRs before they become permanent architecture debt.

Key Takeaways

  • Direct Connect improves consistency and bandwidth but still needs redundancy planning.
  • Private does not automatically mean encrypted.
  • BGP policy and IP planning are the main operational risks.
  • Transit Gateway plus DXGW is the scalable pattern for many VPCs.
  • VPN failover remains a common and useful companion control.
