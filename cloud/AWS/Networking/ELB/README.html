ELB — Elastic Load Balancing

Category: Networking
AWS Service Type: Managed Load Balancing
Key Docs: Elastic Load Balancing User Guide

Overview

Elastic Load Balancing distributes traffic across multiple targets and Availability Zones. AWS provides four variants: Application Load Balancer (ALB) for Layer 7 HTTP/HTTPS routing, Network Load Balancer (NLB) for high-performance Layer 4 TCP/UDP/TLS traffic, Gateway Load Balancer (GWLB) for inline virtual appliances, and the older Classic Load Balancer (legacy). ELB integrates with Auto Scaling, ECS, EKS, WAF, ACM, CloudWatch, and target-group health checks. It is one of the central traffic control services in AWS and often determines both availability and internet exposure posture.

How It Works

Core Mechanics

Concept Description
Listener Front-end protocol and port definition such as HTTPS:443
Target group Back-end pool of EC2, IPs, Lambda, or containers plus health checks
Health check Probe determining whether a target can receive traffic
ALB Layer 7 routing by host, path, header, method, or query
NLB Layer 4 pass-through with static IP support and low latency
Cross-zone load balancing Distributes traffic across AZs regardless of which node receives it
TLS termination Certificate attached at load balancer instead of application server
Deregistration delay Drain period before traffic stops to a removed target

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Internet-facing when internal intended Use internal scheme and private subnets deliberately LB placement
2 Health check too shallow Probe deeper app readiness path with key dependencies Health-check design
3 Deregistration delay too short Match drain time to connection behavior and deployment model Deployment tuning
4 No WAF on ALB Attach AWS WAF to internet-facing ALBs Edge security
5 Wrong idle timeout Tune timeout for SSE, uploads, WebSockets, or API patterns LB configuration
6 TLS policy weak Enforce modern TLS security policy and ACM-managed certs TLS controls
7 Target SG misdesign Reference LB security group in target SG ingress rules Network security
8 NLB chosen for L7 features Choose ALB when HTTP-aware routing is required Architecture selection

Top 3 Critical Mitigations

  1. Choose internal versus internet-facing explicitly: ELB exposure posture is set by a few deployment choices and is easy to get wrong during fast delivery.
  2. Make health checks represent real readiness: A shallow 200 OK is worse than no check if it convinces the load balancer to send traffic to broken targets.
  3. Use security-group referencing: Allow target ingress only from the load balancer's SG, not from the whole VPC or CIDR blocks.

1. How do ALB, NLB, and GWLB differ and when would you choose each?

Direct answer

ALB is for Layer 7 HTTP/HTTPS application routing.
NLB is for Layer 4 TCP/UDP/TLS high-performance network load balancing.
GWLB is for inserting network/security appliances transparently into traffic flows.

Comparison

Load balancer Layer Best for Example
ALB Layer 7 HTTP/HTTPS apps, path/host routing, containers, APIs /api/* to API service, /web/* to web service
NLB Layer 4 TCP/UDP/TLS, static IPs, very high throughput, low latency Private TCP service, gRPC/TLS passthrough, database proxy
GWLB Layer 3/4 appliance insertion Firewalls, IDS/IPS, packet inspection appliances Central inspection VPC using Palo Alto/Fortinet/Suricata appliances

ALB

Choose ALB when you need:

HTTP/HTTPS routing
host-based routing
path-based routing
header-based routing
redirects
OIDC/Cognito authentication
HTTP/2 or gRPC support
WebSocket support
container service routing
AWS WAF integration

Example:

app.example.com/api/*       → API target group
app.example.com/admin/*     → admin target group
static.example.com/images/* → image service

NLB

Choose NLB when you need:

TCP/UDP/TLS load balancing
very high throughput
very low latency
static IP addresses
preserve client source IP
TLS passthrough
PrivateLink endpoint service
non-HTTP protocols

Example:

internal-payments.example.local:443
   ↓
NLB
   ↓
ECS/EKS/EC2 TCP service

GWLB

Choose GWLB when you need:

transparent traffic inspection
centralized firewall appliance fleet
intrusion detection/prevention appliances
north-south or east-west traffic inspection
appliance autoscaling

Example:

VPC traffic
   ↓
GWLB endpoint
   ↓
GWLB
   ↓
Firewall appliance fleet
   ↓
Destination

Answer

“ALB understands HTTP. NLB understands connections. GWLB inserts appliances into network paths. I choose based on protocol layer and whether I need application routing, raw network performance, or security inspection.”

2. What makes a good health check endpoint for a production service?

Direct answer

A good health check endpoint tells the load balancer whether a target should receive traffic right now. It should be fast, reliable, dependency-aware enough to avoid sending traffic to broken instances, but not so strict that it causes cascading outages.

Types of health checks

Endpoint Purpose
/live Process is alive
/ready Service is ready to receive traffic
/health/deep Deeper dependency check, usually not for high-frequency LB checks

Good ALB health check

GET /ready

Should verify:

application started
server is accepting requests
critical local dependencies initialized
database connection pool available if required for every request
required config/secrets loaded
service is not in shutdown/draining mode

Should avoid:

slow queries
expensive dependency checks
calling many downstream services
checking optional dependencies
performing writes
requiring authentication unless specifically supported

Bad health check

/health calls:
  database
  Redis
  third-party payment API
  email provider
  analytics system
  downstream microservices

If the email provider fails, every app target becomes unhealthy even though most API routes could still work.

Better design

/readiness:
  checks only dependencies required to serve core traffic

/deep-health:
  used by monitoring, not load balancer

Example response

{
  "status": "ready",
  "version": "1.8.2",
  "build": "abc123",
  "dependencies": {
    "dbPool": "ready",
    "config": "loaded"
  }
}

Security caution

Do not expose sensitive health data publicly:

database hostname
internal IPs
secret names
dependency credentials
stack traces
version details if public and sensitive

Answer

“A load-balancer health check should answer one question: should this target receive traffic now? It should be fast, stable, and readiness-focused, not a full dependency audit.”

3. How does deregistration delay affect zero-downtime deployments?

Direct answer

Deregistration delay controls how long the load balancer allows existing in-flight requests to complete after a target is removed from service. It is critical for zero-downtime deployments because it prevents the load balancer from cutting off requests while the old task/pod/instance is shutting down.

Deployment flow

New version starts
   ↓
New target passes health check
   ↓
Old target starts deregistration
   ↓
Load balancer stops sending new requests to old target
   ↓
Existing requests drain during deregistration delay
   ↓
Old target exits

If deregistration delay is too short

Long request still running
   ↓
Target is terminated
   ↓
Client receives 502/connection reset

If deregistration delay is too long

Bad target remains draining too long
   ↓
Deployment slows
   ↓
Capacity remains tied up
   ↓
Rollback takes longer

Good deployment settings

Coordinate:

ALB/NLB deregistration delay
container termination grace period
application shutdown hook
ECS stop timeout / Kubernetes terminationGracePeriodSeconds
readiness endpoint behavior
connection keep-alive timeout

ECS/EKS example

Good shutdown behavior:

SIGTERM received
   ↓
App marks /ready = false
   ↓
Load balancer stops new traffic
   ↓
App finishes in-flight requests
   ↓
App closes listeners gracefully
   ↓
Container exits

Answer

“Zero downtime requires traffic draining and app graceful shutdown to agree. Deregistration delay protects in-flight requests while new targets take over.”

4. Why is SG-to-SG referencing preferable to broad CIDR rules for ELB targets?

Direct answer

Security-group-to-security-group referencing is preferable because it expresses identity-based network intent inside the VPC: “only traffic from this load balancer security group can reach these targets.” Broad CIDR rules express only IP ranges and often become over-permissive.

Bad target security group

Inbound:
  TCP 8080 from 10.0.0.0/16

Problem:

any workload in the VPC can hit the target directly
API Gateway/ALB/WAF path can be bypassed
compromised dev instance can scan prod service

Better target security group

ALB security group:
  inbound 443 from internet or CloudFront as appropriate

Target security group:
  inbound 8080 from ALB security group only

Benefits

Benefit Explanation
Least privilege Only ALB can reach target
Prevents backend bypass Other VPC instances cannot call service directly
Dynamic scaling safe No need to track target IPs
Cleaner intent Rules map to architecture
Lower lateral movement Compromised instance has less reach
Easier audit “ALB → service” is explicit

Architecture

Internet
   ↓
ALB SG
   ↓ allowed by SG reference
ECS/EKS/EC2 target SG

Answer

“CIDR rules say ‘anything in this network can connect.’ SG references say ‘only this specific tier can connect.’ For ELB targets, that is a much stronger security boundary.”

5. How would you front ECS or EKS services safely with ELB?

Direct answer

I would put the ELB as the controlled ingress point, keep workloads in private subnets, use security-group references, health checks, least-privileged listener rules, TLS, WAF where appropriate, and ensure services cannot be reached directly except through the intended load balancer path.

ECS with ALB

Internet / CloudFront
   ↓
Public ALB
   ↓
Target group
   ↓
ECS tasks in private subnets

Controls:

ECS tasks have no public IP.
Task SG allows app port only from ALB SG.
ALB listener uses HTTPS.
HTTP redirects to HTTPS.
ALB access logs enabled.
WAF attached if public.
Health checks use /ready.

EKS with ALB

Internet / CloudFront
   ↓
ALB created by AWS Load Balancer Controller
   ↓
Kubernetes Ingress
   ↓
EKS service/pods

Controls:

Ingress annotations reviewed.
Pods are private.
Security groups for pods where useful.
Kubernetes NetworkPolicies applied.
ALB SG restricted.
No NodePort exposed broadly.
mTLS/service mesh where required.

Internal service pattern

Internal ALB or NLB
   ↓
private ECS/EKS service

Use for:

service-to-service APIs
internal admin APIs
private backend services

Security checklist

No public task/pod IPs.
No direct NodePort exposure.
Target SG allows only ELB SG.
TLS at edge or end-to-end.
WAF for public HTTP workloads.
Access logs enabled.
Health checks are readiness-based.
Backend validates auth/tenant context.
Listener rules do not expose admin paths accidentally.

Answer

“For ECS/EKS, ELB should be the only ingress path. The service should be private, target security groups should trust only the load balancer, and Kubernetes/ECS should not accidentally expose parallel paths.”

6. Tradeoffs of TLS termination at load balancer versus end-to-end TLS

Direct answer

Terminating TLS at the load balancer simplifies certificate management, WAF/HTTP routing, and observability. End-to-end TLS protects traffic from the load balancer to the backend and is better for sensitive or zero-trust environments, but adds certificate, performance, and operational complexity.

TLS termination at ALB

Client
   ↓ HTTPS
ALB terminates TLS
   ↓ HTTP
Backend target

Benefits:

central certificate management
easy HTTP routing
WAF and header inspection
simpler backend services
lower backend TLS overhead

Risks:

traffic from ALB to target is plaintext
not ideal for sensitive workloads
insider/VPC lateral risk
compliance concerns

End-to-end TLS

Client
   ↓ HTTPS
ALB terminates or passes/re-encrypts
   ↓ HTTPS
Backend target

Benefits:

encrypts traffic inside VPC
supports zero-trust posture
better for regulated data
protects against internal network sniffing

Tradeoffs:

backend certificate lifecycle
trust store management
mTLS complexity if used
harder debugging
more CPU overhead
health check complexity

NLB TLS passthrough

Client
   ↓ TLS
NLB passes TCP/TLS
   ↓
Backend terminates TLS

Good when:

backend must see client TLS
custom TLS/mTLS required
non-HTTP protocol

Tradeoff:

load balancer cannot inspect HTTP content
less Layer 7 routing
WAF cannot inspect at NLB layer

Answer

“TLS at the load balancer is operationally simple and enables Layer 7 controls. End-to-end TLS is better when the internal network is not trusted or compliance requires encryption all the way to the service.”

7. How would you debug intermittent 502 or 503 errors from an ALB?

Direct answer

I would correlate ALB access logs, target health, target application logs, deployment events, connection behavior, and CloudWatch metrics. Intermittent 502/503 usually points to unhealthy targets, connection resets, target overload, bad deployments, or insufficient healthy capacity.

ALB 502 common causes

target closes connection unexpectedly
target returns malformed response
SSL handshake failure with HTTPS target
target application crashes
idle timeout mismatch
backend port wrong
security group/NACL intermittent issue

ALB 503 common causes

no healthy targets
target group empty
targets failing health checks
deployment removed old tasks before new tasks ready
capacity exhausted
listener rule forwards to unavailable target group

Debug flow

1. Check ALB metrics:
   HTTPCode_ELB_5XX_Count
   HTTPCode_Target_5XX_Count
   TargetConnectionErrorCount
   UnHealthyHostCount
   TargetResponseTime

2. Check target group health:
   which targets unhealthy?
   reason codes?

3. Check ALB access logs:
   elb_status_code
   target_status_code
   target_processing_time
   error_reason

4. Correlate with deployments:
   ECS rollout?
   EKS pod restart?
   ASG scale-in?
   config change?

5. Check app logs:
   crashes
   timeouts
   memory pressure
   connection resets

6. Check network:
   SG/NACL
   target port
   TLS settings

Example

Symptom:

503 only during deployment
UnHealthyHostCount spikes

Likely cause:

new tasks not ready before old tasks drained
health check too slow or too strict
minimum healthy percent too low
deregistration delay/termination mismatch

Fix:

readiness endpoint
increase deployment minimum healthy capacity
tune health check interval/threshold
graceful shutdown

Another example

Symptom:

502 with target_connection_error

Likely causes:

target closed connection
app crashed
container port wrong
TLS mismatch
idle timeout mismatch

Answer

“For ALB 502/503, I separate load-balancer-generated errors from target-generated errors, then correlate target health, deployments, access logs, and app logs.”

8. When is an internal NLB better than an internal ALB?

Direct answer

Use an internal NLB when you need Layer 4 load balancing, static private IPs, TCP/UDP/TLS passthrough, very low latency, high throughput, source IP preservation, or PrivateLink. Use an internal ALB when you need HTTP-aware routing.

Internal NLB is better for

TCP services
gRPC/TLS passthrough where ALB features are not desired
database-like protocols
message brokers
static IP requirements
PrivateLink endpoint service
source IP preservation
very high-throughput internal services
non-HTTP traffic

Internal ALB is better for

HTTP path routing
host routing
headers/rules
OIDC/Cognito authentication
HTTP health checks
WAF-style Layer 7 use cases
microservice HTTP ingress

Example NLB use case

On-prem service
   ↓ Direct Connect
internal NLB static IP
   ↓
ECS TCP service

Why NLB?

on-prem firewall allowlists static IPs
protocol is TCP, not HTTP
source IP preservation required

Example ALB use case

internal-api.company.local
   ↓
internal ALB
   ├── /orders → order service
   ├── /payments → payment service
   └── /users → user service

Why ALB?

HTTP routing by path/host
centralized internal API routing

Answer

“Internal NLB is better for network-level services, static IPs, PrivateLink, and non-HTTP protocols. Internal ALB is better when I need HTTP semantics and routing.”

9. How do you avoid cross-zone availability surprises with target placement?

Direct answer

I would ensure targets are deployed across all enabled Availability Zones, understand whether cross-zone load balancing is enabled, and align load balancer subnets, target groups, Auto Scaling/ECS/EKS placement, and health checks.

The problem

If the load balancer is enabled in three AZs but targets exist only in one AZ:

ALB/NLB in AZ-a, AZ-b, AZ-c
Targets only in AZ-a

Depending on load balancer type and cross-zone settings, traffic distribution, failure behavior, cost, and latency may surprise you.

Risks

Risk Explanation
AZ imbalance One AZ has all targets
Reduced resilience One AZ failure removes all capacity
Unexpected cross-AZ traffic Cost/latency increases
No healthy target in an AZ Zonal failure or routing issue
Scale-in removes last target in AZ Partial outage
Kubernetes/ECS placement skew Pods/tasks unevenly distributed
NLB cross-zone behavior misunderstood Traffic may not balance as expected

Controls

Enable load balancer in all intended AZs.
Run targets in each enabled AZ.
Use ECS spread placement across AZs.
Use EKS topology spread constraints.
Use Auto Scaling groups across AZs.
Set minimum healthy capacity per AZ.
Monitor healthy targets by AZ.
Understand cross-zone load balancing setting.
Test AZ failure.

ECS example

ECS service:
  desired count = 6
  placement strategy = spread across attribute:ecs.availability-zone

EKS example

topologySpreadConstraints:
  topologyKey: topology.kubernetes.io/zone
  whenUnsatisfiable: DoNotSchedule

Answer

“Multi-AZ load balancing only helps if healthy targets actually exist in multiple AZs. I monitor target health per AZ and design placement policies to avoid hidden single-AZ dependency.”

10. How would you design a public edge to private-service architecture using ELB and CloudFront?

Direct answer

I would use CloudFront as the global public edge, WAF for edge protection, an internet-facing ALB only if needed as the CloudFront origin, and private ECS/EKS/EC2 targets behind the ALB. For stronger origin protection, I would restrict ALB access to CloudFront-originated traffic and ensure workloads are not directly public.

Reference architecture

Users
   ↓
CloudFront
   ├── TLS
   ├── WAF
   ├── caching for safe content
   └── origin protection header
   ↓
Public ALB
   ├── HTTPS listener
   ├── validates CloudFront origin header
   └── forwards to target groups
   ↓
Private ECS/EKS/EC2 services
   ↓
Private databases/caches

Security controls

Layer Control
CloudFront TLS, WAF, geo/bot/rate controls
ALB HTTPS only, header validation, access logs
ALB security group Allow only CloudFront origin-facing IPs/prefix list where feasible
Targets Private subnets, SG allows only ALB SG
Backend Authz validation, no trust in spoofable headers
Static assets S3 origin with Origin Access Control
APIs No shared caching unless safe
Logs CloudFront, WAF, ALB, app logs correlated

Static + dynamic split

CloudFront behaviors:
  /static/* → S3 private bucket with OAC, long cache
  /api/*    → ALB/API origin, cache disabled or controlled
  /admin/*  → stricter WAF/auth, no cache

Origin bypass protection

CloudFront adds:
  X-Origin-Verify: secret-value

ALB listener:
  if header matches → forward
  else → 403

And:

target SG allows only ALB SG
ALB SG restricted as much as possible
backend not public

Answer

“The public edge should terminate at CloudFront and WAF, while compute stays private behind ALB. The key is preventing origin bypass and using different cache/security behavior for static assets versus dynamic APIs.”

Attack tree: ELB architecture failure or abuse

Goal: Bypass controls, cause outage, or reach private services directly

├── 1. Origin/backend bypass
│   ├── ALB public and accepts direct traffic
│   ├── Targets allow broad VPC CIDR
│   ├── ECS tasks have public IPs
│   ├── EKS NodePort exposed
│   └── Backend trusts spoofable headers

├── 2. Deployment outage
│   ├── Deregistration delay too short
│   ├── App lacks graceful shutdown
│   ├── Health check marks new targets healthy too early
│   ├── Old targets removed before new healthy
│   └── One AZ has no healthy targets

├── 3. Health check failure
│   ├── Health endpoint too shallow
│   ├── Health endpoint too deep
│   ├── Requires unavailable dependency
│   ├── Slow health check
│   └── Leaks sensitive data

├── 4. TLS weakness
│   ├── TLS terminates but backend plaintext violates policy
│   ├── Weak TLS policy
│   ├── Expired cert
│   ├── Origin cert mismatch
│   └── mTLS not used where required

├── 5. Routing/config error
│   ├── Listener rule sends admin path to wrong target
│   ├── Target port mismatch
│   ├── Wrong target type
│   ├── Host/path rule too broad
│   └── Weighted/canary rule misconfigured

├── 6. Cross-zone surprise
│   ├── Targets only in one AZ
│   ├── Cross-zone setting misunderstood
│   ├── Scale-in removes zonal capacity
│   ├── EKS pods skewed to one AZ
│   └── AZ failure drains all capacity

└── 7. Debugging blind spot
    ├── ALB logs disabled
    ├── No target health alarms
    ├── No correlation IDs
    ├── App logs not linked to request IDs
    └── No deployment event correlation

Controls and mitigations

Risk Controls
Backend bypass Private targets, SG-to-SG references, CloudFront origin protection
Direct ALB access CloudFront prefix restrictions, custom origin header, WAF
Bad health checks Readiness endpoint, fast checks, no sensitive data
Deployment 502/503 Graceful shutdown, deregistration delay, healthy capacity
TLS gap End-to-end TLS or mTLS where required
Broad CIDR access Target SG allows only ELB SG
Cross-AZ surprises Spread targets by AZ, monitor healthy targets per AZ
NLB/ALB mismatch Choose based on protocol and routing needs
Debugging gap ALB access logs, target health metrics, app correlation IDs
Container exposure No public tasks/pods, no broad NodePort access

Summary

“ALB, NLB, and GWLB solve different problems: application routing, network load balancing, and appliance insertion. The production risks are usually not the load balancer itself, but bad target exposure, weak health checks, unsafe deployments, TLS assumptions, and hidden single-AZ target placement.”

My production checklist:

Use ALB for HTTP/HTTPS routing.
Use NLB for TCP/UDP/TLS, static IPs, PrivateLink, and low-latency network services.
Use GWLB for firewall/IDS/IPS appliance insertion.
Use readiness-based health checks.
Coordinate deregistration delay with graceful shutdown.
Use SG-to-SG references instead of broad CIDRs.
Keep ECS/EKS targets private.
Choose TLS termination model based on compliance and trust boundary.
Debug 502/503 with ALB logs, target health, deployment events, and app logs.
Use internal NLB when Layer 4/static IP/PrivateLink matters.
Place targets across all enabled AZs.
Use CloudFront + WAF + ALB + private targets for public edge architectures.

Key Takeaways

  • ELB is often the public entry point and must be treated as a security-relevant control plane.
  • ALB is for HTTP-aware routing; NLB is for transport-layer performance and static IP needs.
  • Realistic health checks and drain settings are essential for stable rollouts.
  • WAF belongs on internet-facing ALBs.
  • Security-group design around the load balancer is a common source of both exposure and outages.
