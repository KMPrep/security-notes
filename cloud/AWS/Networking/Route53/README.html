Route 53 — Amazon Route 53

Category: Networking
AWS Service Type: DNS, Domain Registration, and Health-Checked Routing
Key Docs: Route 53 Developer Guide

Overview

Route 53 is AWS's globally distributed DNS and domain registration service. It hosts public and private DNS zones, supports weighted, latency-based, failover, geolocation, and multivalue answer routing, and integrates health checks directly into DNS decision-making. It is commonly used for internet-facing domains, internal service discovery within VPCs, and cross-region traffic steering. Because DNS is both a control plane and a trust anchor, Route 53 design affects availability, incident response, and takeover risk.

Mitigation Controls

# Failure Mitigation Layer / Where Applied

Top 3 Critical Mitigations

  1. Clean up dangling records aggressively: Route 53 often becomes the final takeover vector after a load balancer, CloudFront distribution, or S3 website endpoint is deleted.
  2. Put DNS in infrastructure as code: Ad hoc DNS changes are a major source of outages because the blast radius is immediate and global.
  3. Use realistic health checks: A 200 on /healthz is not enough if the application depends on a database, cache, or backend that is already failing.

1. How do Route 53 failover, weighted, and latency-based routing differ?

Direct answer

Failover routing is for active/passive availability.
Weighted routing is for controlled traffic splitting.
Latency-based routing is for sending users to the AWS Region that should give them the lowest latency.

AWS describes failover routing as routing traffic to a primary resource when healthy and to a secondary resource when the primary is unhealthy. Weighted routing lets you assign traffic percentages across multiple resources. Latency routing chooses among regional resources based on the Region that gives the user the lowest latency. (AWS Documentation)

Comparison

Routing policy Main purpose Example
Failover Active/passive DR Primary ALB in Singapore, secondary ALB in Tokyo
Weighted Traffic splitting/canary 90% to old ALB, 10% to new ALB
Latency-based User performance SG users to ap-southeast-1, US users to us-east-1
Weighted + health checks Controlled rollout with safety Send 5% to new service only if healthy
Latency + health checks Performance with failover Route to lowest-latency healthy Region

Failover routing

api.example.com
  ├── Primary: ALB in ap-southeast-1
  └── Secondary: ALB in ap-northeast-1

If primary health check passes:
  return primary

If primary health check fails:
  return secondary

Use when:

one Region is active
another Region is standby
you want simple active/passive DR

Weighted routing

api.example.com
  ├── ALB-old weight 90
  └── ALB-new weight 10

Use when:

canary release
blue/green migration
A/B testing
gradual traffic shifting
controlled migration

Latency-based routing

api.example.com
  ├── ap-southeast-1 endpoint
  ├── us-east-1 endpoint
  └── eu-west-1 endpoint

Route 53 chooses the Region with lowest expected latency for the resolver/user location.

Use when:

multi-region active/active service
users are globally distributed
data/app architecture supports regional traffic

Answer

“Failover answers ‘which endpoint is healthy?’ Weighted answers ‘how much traffic should each endpoint get?’ Latency-based answers ‘which Region should be fastest for this user?’”

2. What is a dangling DNS record and why does it matter in AWS specifically?

Direct answer

A dangling DNS record points to a resource that no longer exists or is no longer controlled by you. In AWS, this matters because many DNS records point to AWS-managed resources such as S3 website endpoints, CloudFront distributions, Elastic Beanstalk apps, API Gateway custom domains, ALBs, or released public IPs. If the target is deleted but DNS remains, an attacker may be able to claim the target and take over the subdomain.

AWS specifically documents dangling delegation records in Route 53: if a parent domain delegates to Route 53 name servers but the corresponding hosted zone does not exist or is misaligned, this can create a takeover-style risk. (AWS Documentation)

Example

DNS:
  app.example.com CNAME old-app.us-east-1.elasticbeanstalk.com

AWS:
  old Elastic Beanstalk environment deleted

Risk:

Attacker creates a compatible AWS resource using the same name/endpoint.
DNS still points there.
Users visiting app.example.com reach attacker-controlled content.

AWS-specific dangling patterns

DNS record Dangling risk
CNAME to S3 website endpoint Bucket deleted or never created
CNAME to CloudFront distribution Distribution deleted or alias not attached
CNAME to Elastic Beanstalk Environment deleted
CNAME to API Gateway custom domain Domain mapping removed
A record to released Elastic IP IP released and reallocated
NS delegation to Route 53 name servers Hosted zone missing or wrong
CNAME to ALB/NLB DNS name LB deleted

Why it is serious

Subdomain takeover can allow:

phishing on trusted domain
session/cookie abuse depending on domain scope
OAuth redirect URI abuse
brand damage
malware hosting
bypassing allowlists
email/domain trust abuse

Controls

Automated DNS inventory.
Validate every DNS target still exists.
Delete DNS before deleting target resources.
Use IaC lifecycle dependency controls.
Prevent wildcard DNS unless strongly justified.
Scan for CNAMEs to unclaimed AWS service endpoints.
Monitor NS delegation consistency.
Use domain ownership validation where supported.

Answer

“Dangling DNS is dangerous because DNS can outlive cloud resources. In AWS, deleted resources may leave behind names that attackers can sometimes reclaim.”

3. How would you design internal DNS for a multi-account AWS environment?

Direct answer

I would design internal DNS using Route 53 private hosted zones, Route 53 Resolver endpoints, centralized DNS governance, account/VPC association controls, and clear namespace ownership.

A Route 53 private hosted zone stores DNS records that Route 53 answers only for associated VPCs. AWS says you create a private hosted zone and associate it with one or more VPCs. (AWS Documentation)

Reference architecture

AWS Organization
│
├── Shared network account
│   ├── Private hosted zones
│   │   ├── corp.internal
│   │   ├── prod.aws.internal
│   │   └── shared.aws.internal
│   ├── Route 53 Resolver inbound endpoints
│   ├── Route 53 Resolver outbound endpoints
│   └── Resolver forwarding rules
│
├── Prod app accounts
│   └── VPCs associated with approved private hosted zones
│
├── Non-prod accounts
│   └── separate private zones / route domains
│
└── Security account
    └── query logs / monitoring

Namespace design

Namespace Purpose
prod.aws.internal Production AWS services
dev.aws.internal Development AWS services
shared.aws.internal Shared services
corp.internal Corporate/on-prem-integrated namespace
svc.region.env.company.internal Service discovery pattern

Important design rules

Separate prod and non-prod DNS zones.
Avoid overlapping private hosted zones unless deliberately designed.
Centralize ownership of shared namespaces.
Let app teams own delegated sub-zones where appropriate.
Use IaC for records.
Enable Route 53 Resolver query logging.
Control VPC associations.

Multi-account association pattern

Central DNS account owns private hosted zone.
Application VPCs in other accounts are associated with the zone.
Account vending process automatically attaches required zones.

Security controls

Only DNS platform role can modify shared zones.
Application teams can modify only their delegated service records.
CloudTrail alerts on zone/record changes.
Resolver query logs to security account.
No manual console edits in production.

Answer

“For multi-account AWS, internal DNS should be treated as shared infrastructure. I centralize common private zones, automate VPC associations, delegate ownership carefully, and log DNS queries for security visibility.”

4. What are the tradeoffs of low versus high TTL values?

Direct answer

DNS TTL controls how long resolvers cache a DNS answer. Low TTLs give faster change and failover behavior but increase DNS query volume and reduce caching efficiency. High TTLs reduce query load and improve cache stability but slow migrations, failovers, and recovery from mistakes.

Low TTL

Example:

TTL = 30 seconds

Benefits:

faster failover
faster migration
faster rollback
better for active deployment changes

Tradeoffs:

more DNS queries
higher resolver/load
more cost/noise
some clients/resolvers may not honor very low TTL exactly

High TTL

Example:

TTL = 3600 seconds

Benefits:

fewer DNS queries
better cache efficiency
more stable behavior
less authoritative DNS load

Tradeoffs:

slow cutover
slow correction of bad records
failover can be delayed by cached answers
harder incident recovery

Deployment pattern

Before migration:

Lower TTL from 3600 to 60 well before cutover.
Wait at least old TTL.
Perform weighted or failover change.
After stable, raise TTL again if appropriate.

Important nuance

TTL does not guarantee every client changes exactly at that time.

Some layers cache DNS independently:

browser
OS
JVM
corporate resolver
mobile carrier resolver
application connection pools

Answer

“Low TTL buys agility at the cost of cache efficiency. High TTL buys stability at the cost of slower failover and rollback. For migrations, lower the TTL before the change, not during the change.”

5. How do private hosted zones and Route 53 Resolver endpoints work together?

Direct answer

Private hosted zones provide private DNS records for VPCs. Route 53 Resolver endpoints connect DNS resolution between AWS VPCs and external networks such as on-prem data centers.

AWS says the VPC Resolver automatically answers DNS queries for local VPC domain names and records in private hosted zones. Resolver outbound endpoints forward DNS queries from VPCs to external DNS resolvers, and inbound endpoints allow external networks to query Route 53 Resolver for VPC/private hosted zone names. (AWS Documentation)

Private hosted zone

Private hosted zone:
  app.prod.aws.internal

Records:
  orders.app.prod.aws.internal → internal ALB
  db.app.prod.aws.internal     → RDS endpoint

Associated VPCs can resolve those names.

Resolver inbound endpoint

Use case:

On-prem server wants to resolve:
  orders.app.prod.aws.internal

Flow:

On-prem DNS
   ↓ forwards aws.internal to inbound Resolver endpoint IPs
Route 53 Resolver
   ↓
Private hosted zone answer

Resolver outbound endpoint

Use case:

EC2 instance wants to resolve:
  db.corp.internal

Flow:

EC2 instance
   ↓ VPC+2 Resolver
Resolver rule forwards corp.internal
   ↓ outbound Resolver endpoint
On-prem DNS
   ↓
Answer returned

Hybrid DNS architecture

AWS VPCs
  use VPC Resolver
  private hosted zones for AWS names
  outbound endpoint for on-prem names

On-prem DNS
  forwards aws.internal to inbound endpoint

Security controls

Security groups on Resolver endpoints.
Only approved on-prem DNS servers can query inbound endpoints.
Only required domains are forwarded outbound.
No broad recursive exposure.
Query logging enabled.
Separate prod/non-prod resolver rules where needed.

Answer

“Private hosted zones define private AWS DNS names. Resolver endpoints connect those private names to the rest of the enterprise DNS world.”

6. How would you secure a high-value corporate domain against registrar compromise?

Direct answer

I would secure the domain at multiple layers: registrar account hardening, registry lock/domain lock, DNSSEC, least-privileged DNS administration, change approval, monitoring, and incident recovery plans.

Route 53 supports locking a registered domain to prevent unauthorized transfer to another registrar. (AWS Documentation) Route 53 also supports DNSSEC signing and chain-of-trust configuration to protect against DNS spoofing/tampering. (AWS Documentation)

Domain protection controls

Layer Control
Registrar account MFA, hardware security keys, no shared accounts
Domain transfer Registrar lock / clientTransferProhibited
High-value domains Registry lock where supported
DNS integrity DNSSEC signing
Access control Least-privileged IAM/registrar roles
Change process Dual approval for NS/DS/MX changes
Monitoring Alert on NS, DS, MX, A/AAAA/CNAME changes
Recovery Registrar emergency contacts and runbook
Separation Registrar admin separate from DNS record admin
Email security SPF, DKIM, DMARC monitoring

High-risk DNS records

NS records
DS records
MX records
A/AAAA for apex/root
CNAME for login/auth/payment
CAA records
TXT verification records
OAuth redirect domains

Attack scenario

Attacker compromises registrar account.
   ↓
Changes nameservers for example.com.
   ↓
Controls DNS for login.example.com.
   ↓
Issues certificates or redirects users.
   ↓
Captures credentials/tokens.

Controls

Hardware MFA for registrar.
Domain/registry lock.
DNSSEC.
CAA records restricting certificate authorities.
Certificate transparency monitoring.
Alert on NS/DS changes.
Break-glass registrar recovery plan.

Answer

“A high-value domain is identity infrastructure. I protect the registrar account, the nameserver delegation, DNS records, certificate issuance path, and monitoring.”

7. What are the operational risks of manual DNS management at scale?

Direct answer

Manual DNS management creates drift, outages, stale records, takeover risk, inconsistent TTLs, unreviewed changes, and weak auditability. At scale, DNS must be managed like infrastructure-as-code.

Common risks

Risk Example
Stale records Old CNAME points to deleted ALB
Typo outage Wrong target for payment API
Inconsistent TTLs Migration behaves unpredictably
No approval Intern changes production MX
No rollback Cannot quickly restore previous records
No ownership Nobody knows who owns legacy.example.com
Dangling DNS Subdomain takeover risk
Split-horizon mismatch Public/internal DNS differ unexpectedly
Shadow IT Teams create unmanaged zones
Weak audit Hard to know who changed record

Better operating model

DNS records in Git.
Pull request review.
Automated validation.
CI checks for dangling targets.
Change windows for high-risk records.
Automated rollback.
Record ownership tags.
Scheduled stale-record scans.
CloudTrail alerts for manual changes.

Production DNS checklist

Every record has owner.
Every record has purpose.
Every record has environment.
Every external target is validated.
Every high-risk change has approval.
Every zone has logging and monitoring.

Answer

“Manual DNS does not scale because DNS is both availability infrastructure and security infrastructure. At scale, unmanaged DNS becomes a subdomain takeover and outage factory.”

8. How would you perform a zero-downtime migration between two ALBs using Route 53?

Direct answer

I would use weighted routing with health checks, lower the TTL before the migration, gradually shift traffic from the old ALB to the new ALB, monitor error rates and latency, and keep rollback simple by shifting weight back.

Migration flow

api.example.com
  ├── old-alb weight 100
  └── new-alb weight 0

Then:

100/0 → 95/5 → 90/10 → 75/25 → 50/50 → 0/100

Step-by-step

1. Create new ALB and target groups.
2. Validate TLS/cert/listeners/rules/WAF/security groups.
3. Add Route 53 weighted record for new ALB with weight 0.
4. Attach health checks if appropriate.
5. Lower DNS TTL ahead of cutover.
6. Send 1–5% traffic to new ALB.
7. Monitor:
   4xx/5xx
   latency
   target health
   logs
   business metrics
8. Increase weight gradually.
9. Roll back by setting new ALB weight to 0 if needed.
10. After stable, remove old ALB record after safety window.

Important caveats

DNS weighted routing is not exact per request. It is influenced by resolver caching.

A recursive resolver may cache one answer for many users.
Traffic shift is approximate.

Better if exact progressive delivery is needed

Use:

CloudFront weighted origin logic
ALB weighted target groups
application feature flags
service mesh traffic splitting

depending on architecture.

Answer

“For ALB migration, weighted Route 53 gives safe coarse traffic shifting. I lower TTL in advance, move traffic gradually, monitor business and technical metrics, and keep rollback as a DNS weight change.”

9. How do you detect and prevent subdomain takeover risk in a large AWS estate?

Direct answer

I would continuously inventory public DNS records, map every record to an owned cloud resource, detect dangling records, prevent deletion order mistakes, and enforce DNS/resource lifecycle through IaC.

Detection approach

1. Enumerate all public hosted zones and records.
2. Identify CNAME/A/AAAA/NS records pointing to cloud resources.
3. Validate target resource exists and is owned.
4. Detect released Elastic IPs.
5. Detect deleted S3 website buckets.
6. Detect removed CloudFront/API Gateway/Elastic Beanstalk targets.
7. Alert owners and auto-create tickets.

Common AWS takeover targets

S3 website endpoints
CloudFront alternate domain names
Elastic Beanstalk environments
API Gateway custom domains
ELB DNS names
released Elastic IPs
Route 53 delegated sub-zones
Amplify/App Runner style service domains

Prevention controls

Delete DNS records before deleting cloud resources.
Use IaC dependencies.
Require ownership verification before record creation.
Block CNAMEs to unapproved external domains.
Use automated stale-record scanner.
Tag DNS records with owner/service.
Monitor Route 53 changes.
Review NS delegations.
Use reusable delegation sets carefully.

Example scanner logic

Record:
  blog.example.com CNAME blog.s3-website-ap-southeast-1.amazonaws.com

Check:
  Does bucket blog exist?
  Is bucket in our account?
  Is website hosting configured?
  Is ownership verified?

If no:
  flag takeover risk

Answer

“Subdomain takeover prevention is lifecycle management. DNS records must be deleted or updated when the underlying AWS resource is deleted.”

10. How would you design DNS-based traffic steering for multi-region active/passive failover?

Direct answer

I would use Route 53 failover routing with health checks for active/passive, low-enough TTLs for recovery targets, application-level readiness checks, and a DR design that ensures the passive Region is actually ready to serve traffic.

Reference architecture

api.example.com
  ├── Primary failover record:
  │     ALB/API Gateway in ap-southeast-1
  │     health check: /ready
  │
  └── Secondary failover record:
        ALB/API Gateway in ap-northeast-1
        health check: /ready

Health check design

Bad:

/health returns 200 if web server is alive

Better:

/ready checks:
  app can serve traffic
  dependencies available
  database role is correct
  Region is allowed to accept writes
  critical config/secrets loaded

Active/passive failover flow

Primary Region healthy
   ↓
Route 53 returns primary endpoint

Primary health check fails
   ↓
Route 53 returns secondary endpoint

Primary recovers
   ↓
Depending on policy, traffic may return to primary

DR readiness

DNS failover works only if the passive Region has:

deployed application
valid TLS certs
current config/secrets
replicated container images
database replica/promoted writer plan
KMS permissions
WAF/security rules
capacity
runbooks

Split-brain control

For write services:

Only one Region should accept writes.

Use:

write-region flag
database promotion lock
manual approval for critical failover
application fencing

TTL selection

Low TTL, such as 30–60 seconds, helps faster failover.
But client/resolver caching may still delay complete shift.

Answer

“DNS failover is the traffic switch, not the whole DR strategy. The passive Region must be application-ready, data-ready, and authorized to take writes safely.”

Summary

“DNS is both availability infrastructure and security infrastructure. Route 53 routing policies can steer traffic, but the hard parts are lifecycle governance, domain protection, resolver design, safe migrations, and preventing stale records from becoming takeover paths.”

My production checklist:

Use failover routing for active/passive DR.
Use weighted routing for canaries and migrations.
Use latency routing for active regional performance.
Treat dangling DNS as subdomain takeover risk.

Key Takeaways

  • Route 53 is both a networking tool and a trust boundary.
  • TTL choice directly affects failover speed and migration safety.
  • Dangling records are a common AWS-specific takeover risk.
  • Private DNS needs explicit namespace ownership and hybrid planning.
  • DNS should be audited and versioned like application code.
