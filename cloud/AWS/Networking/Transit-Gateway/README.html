Transit Gateway — AWS Transit Gateway

Category: Networking
AWS Service Type: Hub-and-Spoke Network Transit
Key Docs: Transit Gateway Guide

Overview

Transit Gateway (TGW) is AWS's managed hub for connecting multiple VPCs, VPNs, and Direct Connect attachments without creating a full mesh of peering relationships. It centralizes routing and segmentation using route tables and attachments, making it the standard choice for large multi-account AWS networks and hybrid connectivity. TGW reduces operational complexity compared to many VPC peering links, but its centralized nature also makes route governance and blast radius control critical.

How It Works

Core Mechanics

Concept Description
Attachment Connection from TGW to VPC, VPN, Direct Connect gateway, or Connect peer
TGW route table Central route policy determining how attachments reach each other
Association Attachment linked to one route table for lookup
Propagation Attachment advertises routes into one or more TGW route tables
Appliance mode Preserves symmetric routing for inspection VPCs and appliances
Inter-region peering Connects TGWs across regions
RAM sharing Shares TGW across AWS accounts using Resource Access Manager
Route domain Practical segmentation model using multiple TGW route tables

Top 3 Critical Mitigations

  1. Do not build one flat transit network: TGW makes connectivity easy, which is exactly why segmentation discipline matters more once it exists.
  2. Govern propagation deliberately: Automatic propagation everywhere turns the hub into an accidental trust-merging device.
  3. Design for inspection symmetry: Service insertion without appliance mode and tested return paths creates intermittent failures that are hard to debug.

1. How does Transit Gateway compare to VPC peering at scale?

Direct answer

VPC peering is point-to-point connectivity.
Transit Gateway is hub-and-spoke connectivity.

VPC peering works well for a small number of VPCs. Transit Gateway is better when you have many VPCs, many accounts, on-prem connectivity, inspection requirements, segmentation, and centralized routing.

AWS describes Transit Gateway as a way to interconnect VPCs and on-premises networks through a central hub, simplifying routing compared with many individual connections. Transit Gateway also supports multiple route tables, attachments, and route propagation/association, which makes it more scalable for enterprise segmentation. (AWS Documentation)

VPC peering model

VPC A ─── VPC B
VPC A ─── VPC C
VPC A ─── VPC D
VPC B ─── VPC C
VPC B ─── VPC D
VPC C ─── VPC D

This becomes a mesh.

For N VPCs, full mesh peering requires:

N × (N - 1) / 2 peering connections

For 50 VPCs:

50 × 49 / 2 = 1,225 peering relationships

That is operationally painful.

Transit Gateway model

             ┌───────── VPC A
             │
VPC B ─── Transit Gateway ─── VPC C
             │
             └───────── VPC D

Each VPC attaches once to the Transit Gateway.

For 50 VPCs:

50 TGW attachments

instead of 1,225 peering relationships.

Comparison

Area VPC Peering Transit Gateway
Topology Point-to-point Hub-and-spoke
Scale Good for few VPCs Better for many VPCs/accounts
Transitive routing No Yes, through TGW route tables
Central inspection Hard Strong pattern
On-prem integration More complex Natural with VPN/DX/TGW
Route segmentation Manual per peering TGW route tables
Operational model Distributed Centralized
Cost Often cheaper at small scale Better manageability at scale
Blast radius Smaller per peering Larger if TGW poorly governed
Governance Harder as mesh grows Easier with central network account

When I would use VPC peering

Two or three VPCs.
Simple private connectivity.
No transitive routing required.
No centralized inspection required.
No large multi-account architecture.

When I would use Transit Gateway

Many VPCs.
Many AWS accounts.
Hybrid on-prem connectivity.
Centralized egress/inspection.
Prod/dev/shared-services segmentation.
Multi-region routing.
Need centralized route control.

Answer

“VPC peering is simple and cheap for small topologies. Transit Gateway is the enterprise routing hub when scale, segmentation, inspection, and hybrid connectivity matter.”

2. What is the difference between association and propagation in TGW route tables?

Direct answer

In Transit Gateway:

Association decides which route table an attachment uses to route traffic leaving that attachment.
Propagation decides which attachment routes are automatically inserted into a TGW route table.

AWS documentation states that each VPC or VPN attachment is associated with a single Transit Gateway route table, and that associated route table decides the next hop for traffic coming from that attachment. AWS also describes route propagation as adding routes from an attachment into a Transit Gateway route table. (AWS Documentation)

Simple mental model

Association = Which routing table do I consult?
Propagation = Which routes do I advertise into that table?

Example

Attachments:
  VPC-A
  VPC-B
  VPC-C
  On-prem VPN

Transit Gateway route tables:

prod-rt
dev-rt
shared-rt

If VPC-A is associated with prod-rt, then traffic from VPC-A uses prod-rt to decide where to go.

If VPC-B propagates into prod-rt, then prod-rt learns the CIDR route for VPC-B.

Association example

VPC-prod attachment → associated with prod-route-table

Meaning:

Traffic entering TGW from VPC-prod uses prod-route-table.

Propagation example

shared-services attachment → propagated into prod-route-table

Meaning:

prod-route-table learns a route to shared-services VPC.

Critical point

Association and propagation are independent.

You can have:

VPC A associated with route table X
VPC A propagated into route table Y

This is how you build segmentation.

Common mistake

A team attaches a VPC to TGW and enables default association/propagation everywhere.

Result:

Dev can reach prod.
Prod can reach dev.
Shared services become reachable from unintended VPCs.

Answer

“Association controls the route table used by traffic coming from an attachment. Propagation controls which attachments advertise routes into a route table. Most TGW security design is careful use of those two concepts.”

3. How would you segment prod, dev, and shared services using TGW?

Direct answer

I would use separate TGW route tables for prod, dev, and shared services, then explicitly control which attachments are associated and propagated into each route table.

Goal

Prod can reach shared services.
Dev can reach shared services.
Prod and dev cannot talk directly.
On-prem prod can reach prod.
On-prem dev can reach dev.
Security inspection can sit between zones where required.

Reference design

Transit Gateway
├── prod-rt
├── dev-rt
├── shared-rt
└── inspection-rt

Attachments

prod-app-vpc-1
prod-app-vpc-2
dev-app-vpc-1
dev-app-vpc-2
shared-services-vpc
security-inspection-vpc
onprem-prod-dx
onprem-dev-vpn

Route table associations

Attachment Associated TGW route table
Prod VPCs prod-rt
Dev VPCs dev-rt
Shared services VPC shared-rt or dedicated
Inspection VPC inspection-rt
On-prem prod prod-rt
On-prem dev dev-rt

Route propagations

Route table Propagated attachments
prod-rt prod VPCs, shared services, on-prem prod
dev-rt dev VPCs, shared services, on-prem dev
shared-rt shared services, selected prod/dev/on-prem if needed
inspection-rt inspection VPC, selected routes requiring inspection

Simple architecture

Prod VPCs ─────┐
               │
Dev VPCs ─── Transit Gateway ─── Shared Services
               │
On-prem ───────┘

But route tables enforce:

Prod ↔ Shared: allowed
Dev ↔ Shared: allowed
Prod ↔ Dev: denied

Stronger inspection pattern

Prod VPC
   ↓
TGW prod route table
   ↓
Inspection VPC
   ↓
TGW inspection route table
   ↓
Shared service / on-prem

Controls

Disable default association and default propagation.
Use explicit TGW route tables.
Use IaC for every association/propagation.
Use central network account ownership.
Use route review for every new attachment.
Use Reachability Analyzer / route-table tests.
Monitor TGW route table changes.

Answer

“I segment TGW with multiple route tables. Prod, dev, shared services, on-prem, and inspection zones should have explicit associations and propagations. I never rely on the default TGW route table for a large environment.”

4. When is appliance mode necessary?

Direct answer

Appliance mode is necessary when traffic must pass symmetrically through stateful network appliances, such as firewalls, IDS/IPS, or inspection devices, deployed behind a Transit Gateway attachment.

Without appliance mode, traffic may enter an appliance through one Availability Zone and return through another, which can break stateful inspection.

Why it matters

Stateful firewalls track connection state:

SYN
SYN-ACK
ACK
data packets

If one direction goes through firewall appliance A and the return path goes through firewall appliance B, appliance B may not have session state and may drop the traffic.

Example without appliance mode

Request:
VPC A → TGW → Firewall AZ-a → VPC B

Response:
VPC B → TGW → Firewall AZ-b → VPC A

Problem:

Firewall AZ-b did not see original SYN.
It drops return traffic.

With appliance mode

Transit Gateway maintains flow symmetry for traffic through the appliance VPC attachment.

Request:
VPC A → TGW → Firewall AZ-a → VPC B

Response:
VPC B → TGW → Firewall AZ-a → VPC A

Use appliance mode for

Centralized firewalls
IDS/IPS appliances
Stateful NAT appliances
Traffic inspection VPCs
Third-party security appliances
East-west inspection
North-south inspection via TGW

Not usually needed for

Simple VPC-to-VPC routing
Stateless services
Non-inspected traffic
Basic shared-services access

Answer

“Appliance mode is about preserving symmetric flow through stateful appliances. If a firewall must see both directions of a connection, appliance mode becomes critical.”

5. How would you connect on-prem, many VPCs, and inspection services without a full mesh?

Direct answer

I would use a centralized hub-and-spoke design with Transit Gateway, Direct Connect or VPN for on-prem, an inspection VPC for security appliances, and separate TGW route tables for segmentation.

Reference architecture

                  On-prem
                     │
              Direct Connect / VPN
                     │
              Direct Connect Gateway
                     │
              Transit Gateway
          ┌──────────┼──────────┐
          │          │          │
       Prod VPCs   Dev VPCs   Shared Services
          │          │          │
          └──── Inspection VPC ┘

Route pattern

For inspected traffic:

Spoke VPC
   ↓
TGW
   ↓
Inspection VPC firewall
   ↓
TGW
   ↓
Destination VPC / on-prem

Components

Component Purpose
Transit Gateway Central hub
Direct Connect Gateway Connect DX to TGW/VPC architecture
Site-to-Site VPN Backup or encrypted path
Inspection VPC Central firewall/IDS/IPS
TGW route tables Segmentation
AWS RAM Cross-account sharing
IPAM CIDR governance
IaC Repeatable onboarding

Why this avoids full mesh

Instead of:

VPC A peers with VPC B, C, D, E...
VPC B peers with C, D, E...
On-prem VPNs to each VPC...

You use:

Each VPC attaches once to TGW.
On-prem attaches once or redundantly to TGW/DX gateway.
Inspection is centralized.

Security controls

Separate TGW route tables by environment.
Default deny between environments.
Routes to sensitive destinations go through inspection.
No rogue VPNs or peering outside network account.
Central logging of flow logs/firewall logs.

Answer

“I avoid full mesh by making Transit Gateway the routing hub, Direct Connect/VPN the hybrid edge, and an inspection VPC the controlled choke point.”

6. What is the blast radius of a poorly governed TGW?

Direct answer

A poorly governed Transit Gateway can accidentally connect many VPCs, accounts, environments, and on-prem networks. Its blast radius is large because it sits at the center of network reachability.

What can go wrong

Misconfiguration Impact
Default route table used broadly Unintended connectivity
All attachments propagate everywhere Prod/dev/on-prem become reachable
Bad static route Blackholes many VPCs
Overbroad on-prem route Corporate networks exposed
Wrong inspection route Firewall bypass
Rogue attachment Unapproved VPC gains network reach
Overlapping CIDR Route ambiguity
Peering to another TGW Cross-region/account exposure
Appliance mode missing Stateful firewall failures
Poor route governance Outages during changes

Example incident

New dev VPC attached to TGW.
Default propagation enabled.
Dev CIDR propagates into prod route table.
Prod route table propagates into dev route table.
Dev workload can now reach prod database subnet.

Blast radius

Dev → Prod lateral movement
On-prem → unintended VPCs
Inspection bypass
Shared services exposure
DNS resolution confusion
Incident response complexity

Controls

Disable default association/propagation.
Central network account owns TGW.
Use AWS RAM with approval.
Use explicit route tables per segment.
Use IaC-only route changes.
Alert on new attachments and route changes.
Use SCPs to block rogue TGW/VPN/peering creation.
Run route reachability tests.

Answer

“A TGW is a high-blast-radius network control plane. Poor governance can turn isolated accounts into a flat enterprise network.”

7. How would you debug unexpected connectivity between two attached VPCs?

Direct answer

I would trace the path from source subnet route table to TGW attachment, then through the associated TGW route table, then to the destination attachment, and finally through destination subnet route tables, security groups, and NACLs.

Debug flow

1. Identify source IP, destination IP, port, protocol.
2. Check source subnet route table.
3. Confirm route points to TGW.
4. Identify source VPC TGW attachment.
5. Check which TGW route table source attachment is associated with.
6. Check routes in that TGW route table.
7. Identify destination attachment route.
8. Check whether destination VPC propagated into that route table.
9. Check destination subnet route table return path.
10. Check security groups and NACLs.
11. Check VPC Flow Logs/TGW Flow Logs/firewall logs.

Most likely causes

Wrong TGW route table association.
Unexpected propagation enabled.
Static route added manually.
Default route table used.
Shared-services route table too permissive.
Destination security group allows broad CIDR.
On-prem route leaking back.

Example

Unexpected:

Dev VPC can reach Prod VPC on TCP 5432.

Check:

Dev VPC subnet route:
  10.10.0.0/16 → TGW

Dev attachment associated TGW route table:
  dev-rt

dev-rt routes:
  10.10.0.0/16 → prod-vpc-attachment

Why?
  prod attachment was propagated into dev-rt.

Fix:

Remove prod propagation from dev-rt.
Add explicit deny model through route segmentation.
Tighten prod database SG.

Answer

“For unexpected TGW connectivity, I debug from the source subnet route, through TGW association and propagation, to the destination return path and security controls.”

8. Why do overlapping CIDRs become especially painful with TGW?

Direct answer

Transit Gateway is a central routing hub. If multiple attached VPCs have overlapping CIDRs, TGW cannot unambiguously route to both. Overlaps break reachability, prevent attachments from being useful, complicate route propagation, and often force NAT or renumbering.

Example

VPC A CIDR: 10.20.0.0/16
VPC B CIDR: 10.20.0.0/16
Both attached to same TGW

Problem:

Route 10.20.0.0/16 should point where?
VPC A or VPC B?

Why worse with TGW than isolated VPCs

In isolation, overlap may not hurt.

With TGW:

VPCs need to communicate.
On-prem needs to route to VPCs.
Shared services need to resolve/reach them.
Security inspection needs deterministic paths.

Overlaps make all of that ambiguous.

Operational blast radius

Impact Explanation
Routing ambiguity Same prefix appears multiple times
Attachment limitations Route propagation cannot solve duplicate prefixes
On-prem confusion Corporate routers cannot distinguish VPCs
DNS ambiguity Same internal names may map to overlapping IPs
NAT complexity Must translate one or both sides
Security policy confusion Firewall rules match wrong environment
M&A pain Acquired networks often overlap
DR issues Secondary Region overlaps primary/on-prem
Renumbering cost Application config, allowlists, certificates, docs

Mitigations

Central IPAM.
CIDR allocation before VPC creation.
Account vending with approved CIDR pools.
Separate prod/dev/shared ranges.
Overlap detection in CI.
IPv6 planning where applicable.
NAT only as exception, not default.

Answer

“TGW makes CIDR planning non-negotiable. Once VPCs become part of a routed hub, overlapping CIDRs turn into routing ambiguity and long-term operational debt.”

9. How would you structure cross-account TGW sharing safely?

Direct answer

I would put the Transit Gateway in a central network account, share it with application accounts using AWS Resource Access Manager, require approval for attachments, and manage route table association/propagation centrally through IaC.

AWS documents that Transit Gateways can be shared across accounts using AWS Resource Access Manager, including across an AWS Organization. (AWS Documentation)

Safe structure

Network account
  owns TGW
  owns TGW route tables
  owns DX/VPN attachments
  owns inspection architecture

Application accounts
  create VPC attachments to shared TGW
  cannot modify TGW route tables
  cannot self-propagate into prod

Cross-account model

AWS RAM share:
  TGW shared with selected OUs/accounts

App account:
  requests TGW VPC attachment

Network account:
  approves attachment
  associates it to correct TGW route table
  enables only approved propagations

Controls

Control Purpose
Central network account Clear ownership
AWS RAM OU-based sharing Scalable onboarding
Attachment approval Prevent rogue connectivity
Explicit route tables Segmentation
IaC route management Auditability
SCPs Prevent rogue TGWs/VPNs/peerings
IPAM enforcement Prevent overlaps
Tags Owner, env, data classification
CloudTrail alerts Detect attachment/route changes
Reachability tests Validate allowed/denied paths

Example onboarding workflow

Team requests VPC attachment.
   ↓
Automation checks:
  CIDR non-overlap
  account/environment
  required tags
  route intent
  security approval
   ↓
Network pipeline creates association/propagation.
   ↓
Connectivity tests run.

Answer

“Sharing the TGW does not mean sharing control. App accounts can attach, but the network account should govern associations, propagations, and route segmentation.”

10. How would you design multi-region TGW routing and failover?

Direct answer

I would deploy a Transit Gateway per Region, peer TGWs across Regions where needed, use Direct Connect/VPN per Region or through a controlled hybrid design, keep route domains segmented, and define explicit failover preferences using BGP and route tables.

AWS supports inter-Region Transit Gateway peering attachments, allowing traffic routing between Transit Gateways across Regions. (AWS Documentation)

Reference architecture

Region A
  TGW-A
  ├── prod VPCs
  ├── shared services
  ├── inspection VPC
  └── DX/VPN attachment

Region B
  TGW-B
  ├── DR/prod VPCs
  ├── shared services
  ├── inspection VPC
  └── DX/VPN attachment

TGW-A ↔ TGW-B peering

Design principles

Use one TGW per Region.
Peer TGWs only where required.
Do not create uncontrolled full mesh between Regions.
Keep prod/dev/shared route tables separate in every Region.
Use route summarization where possible.
Use BGP preference for primary/backup hybrid paths.
Test failover and failback.

Active/passive failover

Normal:

On-prem → DX → TGW-A → primary Region services

During Region A failure:

On-prem → DX/VPN → TGW-B → secondary Region services

or:

On-prem → TGW-A → TGW peering → TGW-B

depending on which parts remain healthy.

Routing controls

Primary Region routes preferred under normal conditions.
Secondary Region routes lower preference or inactive.
Health/failover automation updates routes only through approved pipeline.
Avoid simultaneous active writers unless app/data supports it.

Security/inspection

For multi-region, decide:

Inspect locally in each Region?
Backhaul traffic to one inspection Region?
Allow cross-region east-west?

Usually:

Inspect locally per Region for resilience and latency.

Failure modes

Failure Control
TGW peering route missing IaC and route validation
Backup Region not advertised BGP failover tests
Split-brain writes App/database write fencing
Inspection bypass Regional inspection route tables
Asymmetric return Consistent route policy
Cross-region cost surprise Monitor inter-region transfer
DR path untested Game days

Answer

“Multi-region TGW design is not just peering two hubs. It requires route preference, segmentation, inspection placement, hybrid failover, and application/data failover alignment.”

Attack tree: Transit Gateway misconfiguration or abuse

Goal: Create unintended connectivity, outage, inspection bypass, or hybrid route failure

Summary

“Transit Gateway is the enterprise network hub for AWS, but its power comes with blast radius. The security model is route-table design: association, propagation, segmentation, inspection, and governance.”

My production checklist:

Use VPC peering for small/simple point-to-point cases.
Use TGW for many VPCs, many accounts, hybrid, inspection, and segmentation.
Understand association as the route table used by an attachment.
Understand propagation as the route advertisement into a route table.
Create separate TGW route tables for prod, dev, shared, on-prem, and inspection.
Use appliance mode for stateful inspection appliances.
Connect on-prem through DX/VPN into TGW, not a VPC mesh.
Govern TGW centrally in a network account.
Share TGW with AWS RAM but keep route control centralized.
Prevent overlapping CIDRs with IPAM before VPC creation.
Use TGW peering for multi-region, but design route preference and failover carefully.

Key Takeaways

  • TGW is the scalable hub-and-spoke backbone for multi-VPC and hybrid AWS networking.
  • Its biggest risk is not performance but accidental over-connectivity.
  • Route-table design is the real security boundary.
  • Appliance mode matters for inline inspection patterns.
  • Enterprise IP planning and route governance are prerequisites for safe TGW growth.
