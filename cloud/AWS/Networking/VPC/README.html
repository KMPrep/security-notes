VPC — Amazon Virtual Private Cloud

Category: Networking
AWS Service Type: Virtual Network Isolation
Key Docs: VPC User Guide

Overview

A VPC is the fundamental network isolation boundary in AWS. It provides a logically isolated virtual network with your own IPv4 and IPv6 CIDR ranges, subnets, route tables, internet and NAT gateways, security groups, network ACLs, VPC endpoints, peering, and Transit Gateway attachments. Most AWS workloads run inside or connect to a VPC, making it the base layer for security segmentation, egress control, and private connectivity. Good VPC design affects blast radius, cost, observability, and long-term scalability across accounts and regions.

How It Works

Core Mechanics

Concept Description
CIDR block Address range assigned to the VPC
Subnet AZ-scoped slice of the VPC CIDR, public or private by routing
Route table Per-subnet or shared routes controlling next-hop behavior
Internet Gateway VPC attachment enabling internet ingress and egress for public subnets
NAT Gateway Managed egress path for private subnet outbound internet traffic
Security group Stateful allow-list firewall attached to ENIs
Network ACL Stateless subnet-level rule set evaluated on every packet
VPC endpoint Private path to AWS services without traversing public internet
Flow Logs Metadata logs for accepted or rejected network flows

1 Flat network design All workloads share one broad VPC and open SG rules Large blast radius and lateral movement High
2 Public subnet confusion Public IPs assigned to workloads that should be private Unintended internet exposure High
3 No egress control Private workloads route through NAT with unrestricted outbound access Data exfiltration and C2 risk High
4 Overreliance on SGs only No subnet-level or endpoint restrictions Weak defense in depth Med
5 Missing Flow Logs No network telemetry for investigations Blindspot during incident response Med
6 CIDR overlap across VPCs Future peering or TGW attachments impossible or complex Network redesign and routing limits Med
7 Single-AZ NAT design One NAT gateway serves whole VPC AZ failure causes private egress outage Med
8 Unused default VPC Teams deploy workloads in default VPC accidentally Poor segregation and policy drift Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Flat network design Segment by tier, environment, or account with least-privilege SG rules Network architecture
2 Public subnet confusion Disable auto-assign public IP except where explicitly required Subnet configuration
3 No egress control Use egress filtering, VPC endpoints, and proxy or firewall patterns Egress architecture
4 Overreliance on SGs only Combine SGs, NACLs where needed, and endpoint policies Defense in depth
5 Missing Flow Logs Enable VPC Flow Logs to CloudWatch or S3 organization-wide Observability
6 CIDR overlap across VPCs Reserve IP plan centrally and track in network design inventory Address planning
7 Single-AZ NAT design Deploy one NAT gateway per AZ and route privately within same AZ HA design
8 Unused default VPC Restrict or remove default VPC usage and enforce landing-zone standards Governance

Top 3 Critical Mitigations

  1. Design for segmentation early: Retrofitting subnet, SG, and account boundaries after growth is painful. Start with clear private/public separation and least-privilege connectivity.
  2. Prefer private access to AWS services through endpoints: S3, DynamoDB, STS, and interface endpoints reduce NAT dependence, lower exposure, and can enforce resource policy controls.
  3. Turn on Flow Logs everywhere: You will want network telemetry during the first incident, not after it. Flow Logs are the minimum baseline for understanding reachability and suspicious traffic.

1. How do security groups and network ACLs differ in evaluation and use cases?

Direct answer

Security groups are stateful firewalls attached to elastic network interfaces, such as EC2 instances, ENIs, Lambda-in-VPC ENIs, RDS ENIs, and load balancer ENIs.

Network ACLs are stateless subnet-level packet filters. They evaluate inbound and outbound traffic separately and require explicit rules for both directions. AWS documents that security groups are stateful, while network ACLs are stateless. (AWS Documentation)

Comparison

Area Security group Network ACL
Scope ENI/resource level Subnet level
State Stateful Stateless
Rules Allow rules only Allow and deny rules
Evaluation All rules evaluated together Ordered by rule number
Return traffic Automatically allowed Must be explicitly allowed
Best use Application-tier access control Coarse subnet guardrail
Common pattern ALB SG → app SG → DB SG Block known bad CIDRs or enforce subnet boundary

Example

Security group pattern:

ALB security group:
  inbound 443 from internet

App security group:
  inbound 8080 from ALB security group only

DB security group:
  inbound 5432 from app security group only

NACL pattern:

Public subnet NACL:
  allow 443 inbound
  allow ephemeral outbound
  deny known malicious CIDR

Database subnet NACL:
  allow DB port only from app subnet CIDR
  allow ephemeral return traffic

Common mistake

A team allows this on an app server:

Inbound TCP 8080 from 10.0.0.0/16

That means any workload in the VPC can reach the app directly. Better:

Inbound TCP 8080 from ALB security group only

Answer

“Security groups are my primary workload firewall. NACLs are coarse subnet guardrails. I use security groups for application intent and NACLs sparingly for broad network controls.”

2. Why is a subnet considered public or private?

Direct answer

A subnet is considered public if it has a route to an Internet Gateway and resources in that subnet can have public IP addresses reachable from the internet.

A subnet is considered private if it does not route directly to an Internet Gateway. Private subnets may still reach the internet through a NAT Gateway, NAT instance, proxy, firewall, or egress VPC.

Public subnet

Subnet route table:
  0.0.0.0/0 → Internet Gateway

Typical resources:

public ALB
NAT Gateway
bastion, if still used
public-facing edge components

Private subnet

Subnet route table:
  0.0.0.0/0 → NAT Gateway / firewall / no default route

Typical resources:

ECS tasks
EKS nodes/pods
EC2 app servers
RDS
ElastiCache
internal ALB/NLB

Important nuance

A route to an Internet Gateway alone is not enough. For inbound internet reachability, the instance also needs:

public IPv4 or IPv6 address
security group allowing traffic
NACL allowing traffic
route table path

Better naming

I prefer naming by purpose, not just public/private:

ingress-public-subnet-a
app-private-subnet-a
data-isolated-subnet-a
egress-inspection-subnet-a

Answer

“A public subnet has a direct internet route through an Internet Gateway. A private subnet does not. But true exposure also depends on public IPs, security groups, NACLs, and routing.”

3. How would you design a multi-account VPC strategy for a growing organization?

Direct answer

I would use a multi-account landing zone with centrally governed networking, standardized VPC patterns, IPAM-managed CIDRs, Transit Gateway for shared routing, VPC endpoints for AWS service access, and strict segmentation between production, non-production, shared services, and security accounts.

Account model

AWS Organization
├── Network account
│   ├── Transit Gateway
│   ├── Direct Connect/VPN
│   ├── central egress/inspection
│   └── shared DNS/Resolver
│
├── Security account
│   ├── GuardDuty/Security Hub
│   ├── centralized findings
│   └── forensic tooling
│
├── Log archive account
│   └── CloudTrail, VPC Flow Logs, DNS logs
│
├── Shared services account
│   ├── AD/DNS
│   ├── CI/CD runners
│   └── artifact services
│
├── Prod workload accounts
│
└── Non-prod workload accounts

VPC pattern per workload account

VPC
├── public ingress subnets, if needed
├── private app subnets
├── isolated data subnets
├── endpoint subnets
└── inspection/egress route if required

Key design choices

Area Strategy
CIDR Central IPAM, no overlaps
Routing TGW hub-and-spoke
Segmentation Separate TGW route tables for prod/dev/shared
Egress Centralized or per-VPC depending on security/cost
AWS services Gateway/interface VPC endpoints
DNS Private hosted zones + Resolver endpoints
Logs VPC Flow Logs centralized
Guardrails SCPs to block rogue IGW/VPN/TGW/peering
IaC Account vending creates standard VPC

AWS describes Transit Gateway as a hub-and-spoke service that consolidates VPC routing at regional scale. (AWS Documentation)

Answer

“A growing organization needs VPCs as a governed product: IPAM, account vending, standard subnet tiers, TGW connectivity, DNS, endpoints, flow logs, and egress controls built in from day one.”

4. What are the tradeoffs between peering, Transit Gateway, and PrivateLink?

Direct answer

Use VPC peering for simple point-to-point network connectivity.
Use Transit Gateway for scalable hub-and-spoke routing across many VPCs/accounts/on-prem.
Use PrivateLink when consumers need private access to a service without full network-level connectivity.

Comparison

Option Connectivity model Best for Key tradeoff
VPC peering VPC-to-VPC Few VPCs, simple routing No transitive routing; mesh grows poorly
Transit Gateway Hub-and-spoke routing Many VPCs/accounts/hybrid Central blast radius; route governance needed
PrivateLink Service-level private access Expose one service privately Consumer reaches service, not whole VPC

VPC peering

VPC A ↔ VPC B

Good:

two VPCs need private connectivity
simple architecture
low operational overhead

Bad at scale:

many VPCs become full mesh
route tables become hard
no transitive routing

Transit Gateway

VPC A ─┐
VPC B ─┼── TGW ── On-prem
VPC C ─┘

Good:

many VPCs
hybrid connectivity
inspection
route segmentation

AWS notes TGW enables VPCs and VPNs to attach and route traffic between them, and can work across AWS accounts via AWS RAM. (AWS Documentation)

PrivateLink

Consumer VPC
   ↓ interface endpoint
Provider service behind NLB

Good:

private service exposure
SaaS/provider-consumer model
avoid overlapping CIDR problems
avoid broad network reachability

Answer

“Peering connects networks. Transit Gateway builds a routed network hub. PrivateLink exposes a service without exposing the network.”

5. How do VPC endpoints change security and cost posture?

Direct answer

VPC endpoints allow private connectivity from your VPC to AWS services or endpoint services without sending traffic through the public internet. This improves security by reducing internet/NAT exposure and can reduce NAT Gateway cost, especially for heavy S3/DynamoDB access.

There are gateway endpoints for S3 and DynamoDB with no additional endpoint charge, and interface endpoints powered by PrivateLink that create ENIs in your VPC and require security group/NACL design. AWS documents that S3 gateway endpoints avoid internet/NAT paths and have no additional cost, while interface endpoints are needed for certain cross-VPC/on-prem/TGW scenarios and have additional cost. (AWS Documentation)

Security benefits

No internet gateway path needed for AWS service access.
No NAT Gateway exposure for supported services.
Endpoint policies can restrict service access.
Interface endpoint SGs can restrict who connects.
Traffic stays on AWS private network.

Cost benefits

For S3/DynamoDB gateway endpoints:

private subnet → S3 gateway endpoint → S3

instead of:

private subnet → NAT Gateway → public S3 endpoint

This can reduce NAT Gateway hourly/data processing cost. NAT Gateway has hourly and per-GB processing charges. (Amazon Web Services, Inc.)

Tradeoffs

Endpoint type Benefit Tradeoff
Gateway endpoint No extra charge; great for S3/DynamoDB Only S3/DynamoDB; route-table based
Interface endpoint Many AWS services; SG-controlled Hourly and data processing cost; per-AZ ENIs
Centralized endpoint VPC Fewer endpoints More routing/DNS complexity
Decentralized endpoints Simpler local access More endpoints and cost

Example endpoint policy

{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::company-prod-artifacts/*",
  "Condition": {
    "StringEquals": {
      "aws:PrincipalOrgID": "o-example"
    }
  }
}

Answer

“VPC endpoints reduce dependence on internet/NAT paths and give policy control near the network edge. But interface endpoints introduce cost, DNS, security group, and endpoint sprawl considerations.”

6. What mistakes in CIDR planning cause the most long-term pain?

Direct answer

The most painful mistakes are overlapping CIDRs, VPCs that are too small, no room for future AZs/subnets, inconsistent environment ranges, no IPAM, and ignoring Kubernetes/pod/service CIDR growth.

Common CIDR mistakes

Mistake Long-term pain
Overlapping VPC/on-prem CIDRs Hybrid routing breaks
Too-small VPC CIDR Cannot add subnets/workloads
No reserved growth space Readdressing later
Same ranges for prod/dev Confusing security and routing
No IPAM Teams allocate randomly
Ignoring EKS pod IPs VPC CNI IP exhaustion
Too many tiny subnet ranges AZ expansion becomes painful
Acquired company overlap M&A integration blocked
DR Region overlaps primary Failover routing breaks
IPv6 ignored completely Future modernization harder

Example bad plan

Every account gets:
  10.0.0.0/16

This works only while accounts are isolated. Once you add TGW, hybrid, shared services, or centralized inspection, it becomes a major problem.

Better plan

Enterprise CIDR allocation:
  10.0.0.0/8 owned centrally

Prod AWS:
  10.40.0.0/12

Non-prod AWS:
  10.56.0.0/12

Shared services:
  10.72.0.0/16

Region/account allocations:
  assigned through IPAM

Controls

Central IPAM.
CIDR approval workflow.
Overlap detection in account vending.
Separate prod/dev/shared ranges.
Reserve space per Region.
Plan for EKS pod density.
Track on-prem and cloud CIDRs together.

Answer

“CIDR mistakes are architecture debt. Once hybrid connectivity and TGW arrive, overlapping or tiny ranges become expensive renumbering or NAT projects.”

7. How would you design egress control for sensitive workloads?

Direct answer

I would centralize and restrict outbound traffic using private subnets, VPC endpoints, AWS Network Firewall or proxy controls, DNS filtering, NAT restrictions, and explicit allowlists. Sensitive workloads should not have unrestricted 0.0.0.0/0 internet egress.

AWS VPC security best practices include VPC Flow Logs, Network Access Analyzer, Network Firewall, GuardDuty, IAM, security groups, and NACLs as part of VPC security posture. (AWS Documentation)

Egress architecture

Sensitive workload subnet
   ↓
Route table:
  AWS services → VPC endpoints
  approved internet → egress firewall/proxy
  no direct IGW
   ↓
AWS Network Firewall / proxy / NAT
   ↓
Internet or SaaS endpoints

Control layers

Layer Control
Subnet routing No direct IGW
AWS service access VPC endpoints
Domain control DNS firewall / proxy allowlist
IP/port control Network Firewall
App control HTTP proxy with auth/logging
Identity Workload IAM role
Detection Flow Logs, firewall logs, GuardDuty
Data DLP/log review for exfil paths

Example policy

Sensitive app can access:

S3 via gateway endpoint
Secrets Manager via interface endpoint
CloudWatch Logs via interface endpoint
approved SaaS API via proxy

Cannot access:

arbitrary internet
unknown IPs
public S3 endpoint through NAT
crypto mining pools
paste sites

Common failure

Private subnet has 0.0.0.0/0 → NAT Gateway.
Security group outbound allows all.

This is “private” only for inbound. Outbound exfiltration is still broad.

Answer

“For sensitive workloads, private subnet is not enough. I need controlled egress: endpoints for AWS services, firewall/proxy for internet, DNS controls, and logging.”

8. Why is the default VPC problematic in serious environments?

Direct answer

The default VPC is designed for ease of first use, not enterprise segmentation. It comes with public subnets in each Availability Zone, an Internet Gateway, default routing, DNS settings, and a permissive default security group unless changed. AWS documents that a default VPC includes a public subnet in each AZ, an Internet Gateway, and DNS settings so you can immediately launch resources. (AWS Documentation)

Why problematic

Issue Risk
Public subnets by default Accidental public exposure
Internet Gateway attached Easier unintended internet path
Default security group Often misused or left permissive
No enterprise CIDR plan Overlap or routing issues later
No segmentation Apps/data/security mixed
Unmanaged resources Shadow IT
Inconsistent across accounts Governance pain
Not aligned to logging/endpoints/egress Weak controls

AWS notes every VPC has a default security group named default; default SG behavior is commonly risky if teams rely on it instead of explicit tier SGs. (AWS Documentation)

Enterprise approach

Disable/delete default VPCs where policy allows.
Use account vending to create standard VPCs.
Block public IP auto-assign except ingress subnets.
Use SCPs/Config to detect default VPC use.
Create explicit security groups per tier.

Answer

“The default VPC optimizes for convenience. Serious environments optimize for segmentation, least privilege, logging, endpoints, and egress control.”

9. How do you troubleshoot asymmetric routing or reachability failures in VPCs?

Direct answer

I troubleshoot from the source to the destination and back, checking route tables, security groups, NACLs, TGW/peering/endpoint paths, DNS resolution, and logs. Asymmetric routing usually appears when forward and return paths use different routes, especially with TGW, firewalls, NAT, VPN, or peering.

Debug flow

1. Confirm source IP, destination IP, port, protocol.
2. Resolve DNS and confirm target IP.
3. Check source subnet route table.
4. Check destination subnet route table return path.
5. Check security groups on source and destination.
6. Check NACL inbound/outbound both ways.
7. Check TGW/peering/endpoint route tables if involved.
8. Check firewall/NAT state if inspection path exists.
9. Check VPC Flow Logs.
10. Use Reachability Analyzer where possible.

Common causes

Symptom Likely cause
SYN leaves source but no response Return route missing or SG/NACL block
Works one direction only Asymmetric routing or stateful firewall issue
Works in one AZ only Subnet route/NACL/target placement issue
DNS resolves public IP Wrong resolver/private zone association
Timeout, no reject logs Route blackhole or NACL drop
Immediate reject SG or app listener refusal
Peering path fails Missing route on one side
TGW path fails Wrong association/propagation

VPC Flow Logs interpretation

ACCEPT but app fails:
  app/listener/TLS/auth problem

REJECT:
  SG/NACL issue

No logs:
  route/DNS/path issue before ENI

Answer

“Reachability debugging is bidirectional. I always check the forward path and the return path, because stateful security groups do not fix missing routes or stateless NACL mistakes.”

10. How would you migrate a flat VPC into segmented architecture with minimal downtime?

Direct answer

I would migrate incrementally: discover flows, design target subnets/security groups, introduce segmentation in parallel, move stateless services first, then data services, using blue/green, DNS cutover, and compatibility routes during transition.

Current flat VPC

One VPC
One or two broad subnets
Security groups allow VPC CIDR
Apps, databases, caches, admin tools mixed
0.0.0.0/0 egress through NAT

Target segmented VPC

VPC
├── public ingress subnets
├── private app subnets
├── isolated data subnets
├── endpoint subnets
├── inspection/egress subnets
└── management/shared subnets

Migration phases

Phase 1: Discovery

Collect:

current resources
traffic flows
security groups
route tables
NACLs
DNS names
dependencies
hardcoded IPs
stateful services
internet egress paths

Use:

VPC Flow Logs
Reachability Analyzer
CloudTrail
ALB logs
database connection logs
application configs

Phase 2: Build target segmentation

new subnets
new route tables
new SG-to-SG rules
VPC endpoints
egress firewall/proxy
private hosted zones
new ALB/NLB if needed

Phase 3: Move stateless services first

Deploy app version into new private app subnets.
Register new targets behind ALB.
Shift traffic gradually.
Monitor.

Phase 4: Move data services carefully

Create read replica or snapshot/restore where needed.
Use DNS aliases.
Use maintenance window if unavoidable.
Test rollback.

Phase 5: Tighten security groups

Move from:

allow from 10.0.0.0/16

to:

allow from specific SG only

Phase 6: Remove legacy paths

delete broad routes
remove public IPs
remove old SG rules
remove direct DB access
disable unused NAT paths
delete stale DNS

Minimal downtime pattern

Old flat service
   ↓
New segmented service runs in parallel
   ↓
Weighted ALB/Route 53 cutover
   ↓
Rollback by shifting traffic back

Common risk

Hardcoded IPs break migration.

Mitigation:

Move to DNS/service discovery first.
Use low TTLs during migration.

Answer

“I would not big-bang a flat VPC migration. I would discover flows, build segmented paths in parallel, move stateless workloads first, cut over with load balancers/DNS, and then tighten routes and security groups.”

Summary

“VPC design is not just subnet creation. It is the foundation for segmentation, routing, egress control, hybrid connectivity, and blast-radius reduction. Most serious VPC failures come from overbroad security groups, poor CIDR planning, unmanaged egress, default VPC usage, and weak route governance.”

My production checklist:

Use security groups as primary workload firewalls.
Use NACLs as coarse subnet-level guardrails.
Define subnet type by route table behavior.
Use multi-account VPC patterns with central IPAM and TGW governance.
Use peering for simple point-to-point, TGW for hub routing, PrivateLink for service exposure.
Use VPC endpoints to reduce NAT/internet dependency.
Plan CIDRs before account/VPC creation.
Control egress for sensitive workloads with endpoints, firewall/proxy, and DNS controls.
Avoid default VPCs for serious workloads.
Troubleshoot reachability bidirectionally.
Migrate flat VPCs gradually using discovery, parallel paths, DNS/load balancer cutover, and rollback.

Key Takeaways

  • VPC is the base network security boundary for AWS workloads.
  • Subnet routing determines public versus private behavior.
  • Segmentation, egress control, and endpoint strategy are more important than just allocating IP space.
  • Flow Logs and consistent IP planning prevent future blindspots and redesign pain.
  • The simplest VPC is not always the safest or most scalable VPC.
