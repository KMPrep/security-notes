KMS — AWS Key Management Service

Category: Security
AWS Service Type: Managed Cryptographic Key Management
Key Docs: KMS Developer Guide

Overview

KMS is AWS's managed key-management service for creating, storing, and using cryptographic keys. It integrates with most major AWS services, including S3, EBS, RDS, EFS, Secrets Manager, CloudTrail, and Lambda environment encryption. KMS supports symmetric and asymmetric keys, envelope encryption, grants, rotation, imported key material, and multi-region keys. Because encrypted data is only as recoverable as the KMS policy that controls its key, KMS sits at the center of both confidentiality and recoverability in AWS.

How It Works

Core Mechanics

Concept Description
CMK / KMS key Logical key object stored and protected by KMS HSM-backed service
Envelope encryption KMS encrypts a data key, which encrypts actual application data
Key policy Resource policy controlling who may use or administer the key
Grant Delegated short- or medium-lived permission for a service to use a key
Multi-Region key Related key material replicated across regions for consistent crypto workflows
Rotation Automatic annual rotation for supported symmetric keys
Decrypt Operation that turns encrypted data key into plaintext for client use
Imported key material Customer-provided key bytes loaded into KMS for special governance needs

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Key policy lockout Admins remove their own management path Encrypted data becomes hard or impossible to recover Med
2 Overly broad decrypt rights Many principals can call kms:Decrypt Sensitive data exposure across services High
3 Key deletion scheduled casually CMK set for deletion while still in use Permanent data loss after waiting period Med
4 Service-linked grants unchecked Hidden or stale grants remain active Unexpected key usage path Med
5 Separate keys not used One key protects unrelated high- and low-sensitivity data Larger blast radius Med
6 Cross-account key trust too broad External account can encrypt or decrypt unexpectedly Data sharing abuse Low
7 Region assumptions wrong Encrypted workloads fail during DR in another region Recovery failure Med
8 No key-usage logging review Unusual decrypt spikes go unnoticed Slow breach detection Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Key policy lockout Keep recovery admins in every key policy and review before changes KMS governance
2 Overly broad decrypt rights Scope decrypt to exact roles and use kms:ViaService and encryption context Key authorization
3 Key deletion scheduled casually Disable keys first, verify impact, and tightly control deletion permissions Key lifecycle
4 Service-linked grants unchecked Audit grants regularly and prefer minimal service permissions Grant governance
5 Separate keys not used Partition keys by data sensitivity, app, or environment Key architecture
6 Cross-account key trust too broad Constrain external principals and conditions explicitly Cross-account trust
7 Region assumptions wrong Use multi-region keys or explicit DR key strategy DR design
8 No key-usage logging review Monitor CloudTrail for Decrypt, ScheduleKeyDeletion, and policy changes Monitoring

Top 3 Critical Mitigations

  1. Treat key policy as both security and recovery policy: A bad KMS key policy can lock out attackers and defenders alike.
  2. Constrain decrypt with conditions: kms:Decrypt is often too broad unless tied to service usage, encryption context, or exact role paths.
  3. Protect against accidental deletion: Disabling is reversible; deletion is not once the waiting period expires.

Key Takeaways

  • KMS protects both confidentiality and recoverability.
  • Key policy mistakes are often more dangerous than crypto mistakes.
  • Decrypt permission should be tightly constrained and monitored.
  • Key deletion is a data-destruction event, not a routine cleanup action.
  • Multi-region and service-aware key design matter for resilience.
