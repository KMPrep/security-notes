WAF — AWS WAF

Category: Security
AWS Service Type: Web Application Firewall
Key Docs: AWS WAF Developer Guide

Overview

AWS WAF filters and monitors HTTP and HTTPS requests to protect web applications and APIs from common attack patterns. It integrates with CloudFront, Application Load Balancer, API Gateway, App Runner, and Cognito. WAF rules can match IP reputation, rate limits, request size, headers, SQL injection, XSS patterns, bot signals, and custom conditions. Managed rule groups accelerate baseline protection, while custom rules tailor enforcement to application behavior. WAF is most effective when treated as one control in a layered web-defense model rather than a substitute for secure code.

How It Works

Core Mechanics

Concept Description
Web ACL Top-level ruleset associated with one or more protected resources
Rule Match logic plus action such as allow, block, count, or challenge
Managed rule group Prebuilt vendor or AWS rule package for common threat classes
Rate-based rule Detects and limits excessive requests from a source
Label Metadata assigned by one rule for downstream rules to inspect
Scope-down statement Narrows when a managed or rate rule applies
CAPTCHA / Challenge Browser-interaction friction to reduce automated abuse
Logging Full WAF request decision logs delivered to Kinesis, S3, or CWL

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 WAF not attached Public app has no web ACL Common attacks reach app directly High
2 False positives block users Rule tuning too aggressive Legitimate traffic denied Med
3 Rule gaps App-specific abuse not covered by managed rules Attacks succeed despite WAF High
4 No logging Rule decisions cannot be reviewed or tuned Blind tuning and weak forensics Med
5 Rate limit too blunt Shared NATs or proxies penalized Legitimate user impact Med
6 Challenge not used Simple bots bypass static signatures Abuse continues Med
7 Origin bypass exists Attackers hit ALB or API directly around WAF layer Control bypass High
8 WAF treated as app-security substitute Secure coding gaps ignored Injection or auth flaws still exploitable High

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 WAF not attached Require web ACL association for all internet-facing HTTP services Security baseline
2 False positives block users Start in count mode and stage rule rollout with sampled logs Tuning process
3 Rule gaps Add app-specific rules and labels beyond managed defaults Custom protection
4 No logging Enable WAF logs and review top matches and misses regularly Monitoring
5 Rate limit too blunt Scope rate rules by path, header, or auth state where possible Rule design
6 Challenge not used Use challenge or CAPTCHA for bot-prone endpoints Abuse mitigation
7 Origin bypass exists Restrict origin access to only the WAF-fronted path Architecture hardening
8 WAF treated as app-security substitute Keep secure coding, auth, and patching programs in place Layered defense

Top 3 Critical Mitigations

  1. Ensure no origin bypass path exists: WAF only matters when all client traffic is forced through it.
  2. Use logs and count mode to tune: Untuned WAF causes self-inflicted outages.
  3. Add custom rules for app-specific abuse: Managed rules cover common patterns, not business-logic attacks.

Key Takeaways

  • WAF is a useful edge filter, not a replacement for secure applications.
  • Logging and staged rollout are essential for safe tuning.
  • Managed rules are baseline only; app-specific abuse needs custom detection.
  • Origin bypass invalidates much of the security value.
  • WAF is strongest when paired with CloudFront, Shield, and disciplined backend design.
