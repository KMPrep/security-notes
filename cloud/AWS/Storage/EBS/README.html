EBS — Elastic Block Store

Category: Storage
AWS Service Type: Block Storage
Key Docs: EBS User Guide

Overview

EBS provides persistent block storage volumes for use with EC2 instances. Unlike instance store (ephemeral), EBS volumes persist independently of the instance lifecycle and can be detached and reattached across instances in the same Availability Zone. Volumes appear as raw block devices to the OS and support standard file systems and database services. Volume types include gp3 (general purpose SSD), io2 Block Express (provisioned IOPS SSD), st1 (throughput optimized HDD), and sc1 (cold HDD). EBS snapshots are point-in-time backups stored durably in S3 and can be used to create new volumes or copy data across regions.

How It Works

Core Mechanics

Concept Description
Volume type gp3, io2, io1, st1, sc1 — each trades IOPS, throughput, and cost
IOPS Input/Output Operations Per Second; provisioned separately on io2/io1
Snapshot Incremental backup stored in S3; foundation for AMIs and cross-region copies
Encryption AES-256 at rest using KMS CMK or AWS-managed key; encrypts data in transit too
Multi-attach io2 volumes can attach to up to 16 Nitro instances simultaneously (cluster FS)
Fast Snapshot Restore Pre-warms snapshot data for immediate full-speed volume access after restore
EBS-optimized instances Dedicated network bandwidth between instance and EBS; reduces I/O contention
Throughput MB/s sustained; relevant for sequential workloads like log processing

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Unencrypted volume in production Encryption not required by default or policy Data at rest accessible if storage is reviewed High
2 Public snapshot sharing EBS snapshot shared with all accounts or public Any AWS account can create volume from snapshot High
3 No snapshot lifecycle policy Snapshots accumulate without cleanup Cost waste; old snapshots with sensitive data Med
4 Snapshot copied to wrong region Cross-region copy of unencrypted snapshot Data sovereignty violation Med
5 Wrong volume type for workload gp2/gp3 used for high-IOPS database workload I/O contention and latency spikes Med
6 Volume in single AZ EBS is AZ-specific; AZ failure unavailable to instance Outage until volume or instance is recovered Med
7 No deletion protection Volume deleted when instance terminates (DeleteOnTermination) Permanent data loss on instance termination Med
8 Over-provisioned IOPS io2 over-provisioned for workload that doesn't use it Unnecessary monthly cost Med
9 KMS key deletion CMK used for volume encryption is scheduled for deletion Volume becomes permanently inaccessible Low
10 Performance baseline assumption gp3 burst credits exhausted on t3 instance class Sudden IOPS drop and application slowdown Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Unencrypted volume in production Enable account-level default encryption; enforce via SCP Account and EC2 settings
2 Public snapshot sharing Block public AMIs policy; audit shared snapshots with Config rule Account policy and Config
3 No snapshot lifecycle policy Create DLM (Data Lifecycle Manager) policies for automated rotation DLM configuration
4 Snapshot copied to wrong region Use KMS key conditions in snapshot copy IAM policies IAM and KMS policy
5 Wrong volume type for workload Benchmark workload IOPS and throughput; right-size with performance tuning Volume type selection
6 Volume in single AZ Use EFS for multi-AZ shared storage; replicate critical data via snapshots Architecture design
7 No deletion protection Set DeleteOnTermination=false; use EC2 termination protection EC2 and EBS configuration
8 Over-provisioned IOPS Monitor CloudWatch VolumeReadOps and VolumeWriteOps; right-size Cost optimization
9 KMS key deletion Disable key instead of schedule deletion; set DeletionWindow to 30 days KMS key management
10 Performance baseline assumption gp3 has a fixed baseline of 3000 IOPS with no burst credits Volume type planning

Top 3 Critical Mitigations

  1. Enable account-level default encryption: Go to EC2 settings → EBS Encryption → enable default encryption with a specific KMS key. All new volumes and snapshots are encrypted automatically without relying on per-volume configuration.
  2. Audit and restrict snapshot sharing: EBS snapshots exposed as public or cross-account without intent are a significant data breach risk. Use AWS Config rule ebs-snapshot-public-restorable-check and block public sharing with a SCP.
  3. Use DLM for snapshot lifecycle management: AWS Data Lifecycle Manager automates snapshot creation, retention, and deletion. Without it, snapshots accumulate silently and represent both a cost issue and a data governance risk.

Key Takeaways

  • EBS is AZ-local; an AZ failure makes volumes in that AZ temporarily inaccessible.
  • gp3 is the default choice; io2 Block Express is for high-IOPS, mission-critical databases.
  • Enable account-level default encryption — it adds no latency and eliminates unencrypted volumes.
  • Snapshots are incremental, durable, and the foundation for backups, AMI creation, and cross-region migration.
  • DLM automates snapshot hygiene; without it, snapshots accumulate with cost and governance risk.
