EFS — Elastic File System

Category: Storage
AWS Service Type: Managed Network File System (NFS)
Key Docs: EFS User Guide

Overview

EFS provides a managed, scalable, shared NFS file system that can be mounted concurrently by thousands of EC2 instances, ECS tasks, EKS pods, and Lambda functions across multiple Availability Zones. Unlike EBS (which is AZ-local and single-attach by default), EFS is a regional service with automatic data replication across AZs. EFS uses the NFSv4 protocol and supports two storage classes: Standard and Infrequent Access (IA), with Intelligent Tiering moving data automatically. Performance modes include General Purpose (latency-sensitive) and Max I/O (higher throughput for parallel workloads). Throughput modes include Elastic (auto-scaling), Bursting (proportional to stored data), and Provisioned.

How It Works

Core Mechanics

Concept Description
File system Regional, multi-AZ NFS namespace; each file system has a DNS name
Mount target ENI in each AZ subnet through which instances mount the file system
Access point Application-specific entry point with enforced root directory and POSIX user
Performance mode General Purpose (default) or Max I/O for parallel-heavy workloads
Throughput mode Elastic (auto), Bursting (based on file system size), or Provisioned (fixed)
Encryption at rest KMS-based; enabled at creation; cannot be changed after creation
Encryption in transit TLS enforced when mounting with tls option via EFS mount helper
EFS Replication Built-in cross-region replication for DR with RPO near zero
IAM authorization NFS client identity combined with EFS resource policy for access control

# Failure / Attack Root Cause Impact Likelihood

4 No mount target per AZ Only one AZ has a mount target Outage if EC2 is in AZ without mount target Med
5 No EFS resource policy Any IAM principal in account can mount the file system Unauthorized access from other services Med
6 Cross-account mount without policy EFS shared cross-account without explicit allow Access denied or unintended access path Med
7 Burst credit exhaustion Large file system in Bursting mode with high sustained throughput I/O throttling when credits deplete Med

9 Replication not enabled No DR replication; single-region file system File system loss or prolonged outage on disaster Med
10 No backup policy EFS not included in AWS Backup plan Data loss with no restore point Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 No encryption in transit Use tls and iam mount options; enforce via EFS resource policy Mount configuration
2 No encryption at rest Enable KMS encryption at file system creation; cannot be added later File system creation
3 Overly permissive POSIX permissions Use EFS access points to enforce POSIX UID/GID and root path per app EFS access points
4 No mount target per AZ Create mount target in every AZ where EC2 or EKS workloads run Mount target configuration
5 No EFS resource policy Attach resource policy requiring IAM authentication for mount EFS resource policy
6 Cross-account mount without policy Explicitly grant cross-account access via EFS resource policy EFS resource policy
7 Burst credit exhaustion Switch to Elastic throughput mode for unpredictable workloads Throughput mode selection
8 No lifecycle policy configuration Enable EFS Intelligent Tiering or set explicit IA transition lifecycle EFS lifecycle policy
9 Replication not enabled Enable EFS Replication to a second region for DR recovery EFS Replication config
10 No backup policy Add EFS file system to AWS Backup plan with appropriate retention Backup configuration

Top 3 Critical Mitigations

  1. Always use TLS for in-transit encryption with IAM auth: Add -o tls,iam to mount commands or set it as the default in the EFS CSI driver config. Without TLS, NFS communication is plaintext over the VPC network — readable by anyone with network access or packet capture capability.
  2. Use Access Points to isolate application namespaces: EFS access points enforce a specific root directory, POSIX UID, and GID for the mounted client. This prevents one application from reading or modifying another application's files in the same file system.
  3. Enforce encryption at rest at creation time: EFS encryption at rest cannot be enabled after creation. Enforce it by default via an SCP (elasticfilesystem:CreateFileSystem with condition elasticfilesystem:Encrypted = true).

Key Takeaways

  • EFS is multi-AZ shared NFS — the correct choice when multiple instances or pods need to share the same files.
  • Encryption at rest must be enabled at creation; retroactively enabling it requires creating a new file system.
  • Access points provide per-application POSIX isolation within a shared file system namespace.
  • Elastic throughput mode removes burst credit management and is the safest default for production.
  • EFS latency is higher than EBS because it traverses two network hops — do not use EFS for high-IOPS databases.
