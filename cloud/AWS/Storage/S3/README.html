S3 — Simple Storage Service

Category: Storage
AWS Service Type: Object Storage
Key Docs: S3 User Guide, S3 Security Best Practices

Overview

S3 is AWS's foundational object storage service, providing virtually unlimited, highly durable (11 nines) storage for any type of data. Objects are stored in buckets and addressed by key. S3 supports static website hosting, event notifications, lifecycle policies, versioning, replication, and fine-grained access control via bucket policies, ACLs, and IAM. Storage classes span Standard, Intelligent-Tiering, Standard-IA, One Zone-IA, Glacier Instant Retrieval, Glacier Flexible Retrieval, and Glacier Deep Archive, enabling cost optimization based on access patterns. S3 is the most commonly used AWS service and appears as a building block in virtually every architecture.

How It Works

Core Mechanics

Concept Description
Bucket Globally unique namespace container for objects; region-specific
Object Key-value pair: key is the full path, value is the object data and metadata
Versioning Stores all versions of an object; enables recovery from overwrite or delete
Storage class Cost and access tradeoff: Standard to Glacier Deep Archive
Bucket policy Resource-based IAM policy controlling access from principals and services
Block Public Access Account or bucket-level switch that overrides ACLs and policies to block public
Server-side encryption SSE-S3, SSE-KMS, or SSE-C to encrypt objects at rest
Object Lock WORM immutability per-object or per-bucket; governance or compliance mode
S3 Transfer Acceleration Routes uploads through CloudFront edge network for faster upload from far clients
S3 Event Notifications Triggers to Lambda, SQS, SNS, or EventBridge on object create/delete events

# Failure Mitigation Layer / Where Applied
1 Public bucket exposure Enable Block Public Access at account level; audit with Config/Macie Account and bucket settings
2 Over-permissive bucket policy Deny s3:_ to _ unless required; use IAM and resource conditions Bucket policy and IAM design
3 No encryption at rest Enforce SSE-KMS via bucket default encryption and bucket policy deny Bucket encryption settings
4 Versioning disabled Enable versioning on all production buckets; use MFA Delete Bucket configuration
5 Unintended presigned URL lifetime Set maximum expiry; rotate credentials used to sign URLs Application code and IAM

8 Lifecycle policy misconfiguration Test lifecycle rules in non-prod; peer review all changes Lifecycle rule management
9 Replication to untrusted bucket Constrain replication destination account via role trust conditions IAM and replication config
10 No Object Lock for compliance data Enable Object Lock with compliance mode for retention-required data Bucket Object Lock settings

Top 3 Critical Mitigations

  1. Enable Block Public Access at the AWS account level: This is a single toggle that prevents any bucket or object in the account from being made public, regardless of bucket policy or ACL. It should be the default in every account and enforced via AWS Organizations SCP.
  2. Enforce SSE-KMS via bucket policy: Add a Deny statement for s3:PutObject when s3:x-amz-server-side-encryption is not aws:kms. This ensures all objects land encrypted and the key usage is auditable in CloudTrail.
  3. Enable versioning with MFA Delete: Versioning prevents data loss from accidental overwrites. MFA Delete requires MFA authentication before a version can be permanently deleted, protecting against both accidents and insider threats.

Key Takeaways

  • Block Public Access at the account level is the simplest and most effective single S3 security control.
  • Bucket policies are evaluated together with IAM policies; both must allow unless one is a deny.
  • Versioning protects against accidental deletion; Object Lock enforces immutability for compliance.
  • SSE-KMS gives per-key access control and full CloudTrail audit of key usage.
  • S3 is eventually consistent for overwrites of existing objects in some edge cases but strongly consistent for new object PUTs and DELETEs since 2020.
