Azure Databricks

Category: AI & Analytics
Azure Service Type: Managed Apache Spark Analytics
Key Docs: Azure Databricks Overview

Overview

Azure Databricks is a managed Apache Spark-based analytics platform deeply integrated with Azure for data engineering, data science, machine learning, and SQL analytics. It runs on customer Azure subscriptions (VNet-injected compute) or on Databricks-managed infrastructure. Key components include Workspaces (collaborative Spark environment), Clusters (Spark compute — all-purpose or job clusters), Delta Lake (ACID-transactional data format on Azure Data Lake), Unity Catalog (fine-grained data governance layer — tables, columns, rows), Jobs (scheduled or triggered Spark workloads), and Databricks Repos (Git integration for notebooks). Security features include VNet injection, Azure Private Link, Unity Catalog for data access control, credential passthrough (deprecated in favor of Managed Identity), and Secrets (backed by Azure Key Vault).

How It Works

Core Concepts

Concept Description
Workspace Azure Databricks environment — users, clusters, notebooks, jobs, and data
Cluster Spark compute (driver + workers); all-purpose (interactive) or job cluster (automated)
Delta Lake Open-source storage format with ACID transactions, versioning, schema enforcement
Unity Catalog Centralized data governance — catalog/schema/table hierarchy; column and row-level security
Metastore Unity Catalog's central metadata store — shared across workspaces
Cluster Policy Restrict cluster configuration options (instance types, autoscaling limits, Spark configs)
Secrets Databricks Secret Scope backed by Azure Key Vault — retrieve secrets in notebooks without hardcoding
VNet Injection Deploy Databricks compute into customer-managed VNet subnets
Managed Identity System-assigned MI on cluster node VMs for accessing ADLS Gen2 without storage keys
Data Exfiltration Protection Network egress controls to prevent data being written to external storage

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Storage access key hardcoded in notebook No Secret Scope usage A data engineer stores the Azure Data Lake Storage access key directly in a Databricks notebook cell for quick prototyping. The notebook is shared with a team via Databricks Workspace. The key appears in the notebook revision history and in job logs. A colleague exports the notebook (including the hardcoded key) to GitHub. The storage key provides read/write access to all containers in the storage account. An attacker exfiltrates 10 TB of customer data. Entire storage account compromised via hardcoded key High
2 No Unity Catalog — flat data access model Legacy table ACLs or no access control A Databricks workspace contains tables from multiple sensitive domains: HR (salaries, performance reviews), Finance (revenue, forecasts), Customer (PII, purchase history). Without Unity Catalog, data access is controlled via Databricks table ACLs (deprecated, per-cluster). Different user groups can query all tables if they have cluster access. Analysts from the marketing team query customer PII tables that they have no business need to access. Sensitive data accessible to unauthorized users High
3 All-purpose cluster shared by all users — credential exfiltration Shared cluster credential passthrough A large all-purpose cluster is shared across 50 data engineers for cost savings. The cluster uses Azure AD credential passthrough (deprecated), where each user's own Entra ID token is used for ADLS access. A malicious user with cluster access uses the %sh command to access the Spark driver process environment and extract another user's Azure AD token from memory. The token is used to access ADLS with the victim user's permissions. Token theft from shared cluster environment Med
4 No cluster policy — users deploy large expensive clusters No compute governance Without cluster policies, any Databricks user can create clusters with any instance type and no autoscaling limit. A junior data scientist accidentally creates a cluster with 50 Standard_E64s_v3 nodes (64 vCPU each, $10/hour/VM) for an exploratory analysis. The cluster runs for 8 hours before the user notices. The job could have run on a 4-node cluster. The accidental cost is $4,000. $4,000+ wasted on accidental over-sized cluster High
5 Data exfiltration via external write No egress network controls A Databricks cluster has internet egress enabled without restriction. A compromised notebook (via supply-chain attack on a Python library) writes processed data to an external AWS S3 bucket controlled by the attacker. The cluster's network configuration allows any outbound TCP connection. The data exfiltration is only discovered weeks later during an audit. 10 TB data exfiltration to external storage via egress Med
6 No VNet injection — compute in Databricks-managed VNet Default workspace configuration A Databricks workspace is deployed without VNet injection (default configuration). All cluster VMs run in a Databricks-managed VNet that is not controlled by the customer. The organization's security policy requires all compute to run within the customer-managed VNet for network segmentation and firewall policy. The workspace cannot be migrated post-deployment without recreation. Compute outside customer VNet — violates network policy High
7 Secrets in notebook outputs Secrets printed/logged accidentally A data engineer uses print(dbutils.secrets.get(scope="kv", key="db-password")) to debug a connection issue. The print statement is committed to the notebook. In Databricks, secret values are redacted in interactive output — but job run logs archive the output. The secret appears in job run logs, which are accessible to all users with Can View on the job. Secrets leaked via job run log output Med
8 No audit logging — workspace activity untracked Databricks audit logs not configured Databricks workspace audit logs (user logins, cluster creation, notebook edits, data access) are not exported to Log Analytics. An insider downloads 500 GB of sensitive customer data by running a job that writes data to their personal Azure Storage account. The activity is not detected because no SIEM visibility exists for Databricks. The data exfiltration is discovered during a UEBA alert on the user's Azure Storage account 3 weeks later. Insider data exfiltration with no Databricks audit trail High
9 Delta Lake no schema enforcement — data quality failure Delta Lake created without schema evolution controls A Delta Lake table is created with schema enforcement disabled. An upstream data pipeline changes a field from integer to string (for a country code field). The breaking schema change is written to the Delta table without error. Downstream analytics queries begin failing silently — numeric aggregations on the country code field produce NULL. The data quality issue is undetected for 2 weeks, corrupting 2 weeks of financial reporting. Silent data corruption from schema drift Med
10 Job cluster with admin-level Databricks permissions Over-privileged job service principal The service principal used to run Databricks jobs (via REST API or ADF) has Databricks workspace Admin role. An attacker who compromises the service principal's credentials (from a pipeline secret or Key Vault) can create new admin users, modify cluster policies, access all data, and delete workspace objects. Job execution only requires Databricks Can Manage Run permission on the target job. Full workspace takeover via over-privileged job SP Med

Mitigation Controls

# Failure Mitigation Layer
1 Hardcoded storage key Use Azure Key Vault-backed Secret Scope; access ADLS via Managed Identity (no key at all) Credential management
2 No Unity Catalog Enable Unity Catalog; define catalog/schema hierarchy; apply column masking and row filters Data governance
3 Shared cluster credential theft Use job clusters (ephemeral, per-user) instead of shared all-purpose clusters for sensitive workloads Cluster isolation
4 No cluster policy Create cluster policies restricting instance types, max workers, and auto-termination timeout Cluster governance
5 Data exfiltration via egress Enable data exfiltration protection (secure cluster connectivity); restrict egress to known destinations via VNet NSG Network egress control
6 No VNet injection Deploy Databricks with VNet injection at workspace creation; use Private Link for control plane VNet injection
7 Secrets in outputs Never print secrets; use dbutils.secrets.get() inline — values are masked in interactive output; remove print statements Secret handling
8 No audit logging Export Databricks audit logs to Log Analytics via diagnostic settings; create Sentinel analytics rules Audit logging
9 No schema enforcement Create Delta tables with CONSTRAINT schema enforcement; use Delta Lake MERGE with explicit schema validation Schema enforcement
10 Over-privileged job SP Grant job service principal minimum Databricks permissions (Can Manage Run on specific job); use managed identity where possible RBAC

Top 3 Critical Mitigations

  1. Unity Catalog for Fine-Grained Data Access Control: Without Unity Catalog, Databricks data access is flat — anyone with cluster access can read any table. Unity Catalog enables catalog/schema/table-level RBAC, column masking, and row-level security. It is the foundation of a compliant Databricks data governance model.
  2. Azure Key Vault-Backed Secret Scope + Managed Identity for Storage: All secrets used in notebooks and jobs must be retrieved from an AKV-backed Secret Scope at runtime. For ADLS access, use Managed Identity (storage credential) — this eliminates static storage keys entirely from the Databricks environment.
  3. VNet Injection + Egress Controls at Workspace Creation: Databricks workspace VNet injection cannot be changed after creation — deploy into a customer-managed VNet from the start. Add NSG rules restricting egress to prevent data exfiltration to external storage. Enable Databricks secure cluster connectivity to eliminate public cluster IP exposure.

Key Takeaways

  • Enable Unity Catalog before onboarding users — retrofitting data governance after data is in the workspace is very difficult.
  • Use Key Vault-backed Secret Scopes and Managed Identity storage credentials — never hardcode storage keys or connection strings in notebooks.
  • Deploy Databricks with VNet injection from the start — this cannot be changed post-deployment without recreating the workspace.
  • Cluster policies are essential for cost governance — without them, users can accidentally provision extremely expensive clusters.
  • Export Databricks audit logs to Log Analytics — workspace activity is a high-value audit trail for detecting insider threats and compliance violations.
