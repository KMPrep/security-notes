Azure Machine Learning

Category: AI & Analytics
Azure Service Type: ML Platform
Key Docs: Azure Machine Learning Overview

Overview

Azure Machine Learning (AML) is a managed end-to-end platform for building, training, deploying, and monitoring machine learning models. An AML Workspace is the top-level resource that ties together compute (compute clusters, compute instances, managed online endpoints), data (datastores, datasets), experiments and runs (tracked with MLflow), model registry, pipelines, and environments. AML supports both code-first workflows (Python SDK v2, CLI v2) and the Designer (drag-and-drop). Models are deployed to Managed Online Endpoints (real-time inference) or Batch Endpoints (large-scale scoring). Security considerations include VNet integration for compute, private workspace endpoints, managed identity for data access, customer-managed encryption keys, and role-based access control over workspace operations.

How It Works

Core Concepts

Concept Description
AML Workspace Central resource containing all AML assets — compute, data, models, experiments
Compute Cluster Autoscaling VM pool for training and batch inference jobs
Compute Instance Single-user VM for interactive notebook development
Managed Online Endpoint Managed real-time inference endpoint with auto-scaling, blue/green deployment, traffic split
MLflow Open-source tracking for experiments (metrics, parameters, artifacts) — integrated into AML
Model Registry Versioned model store with lineage, metadata, and deployment history
Datastore Abstraction over Azure Storage (Blob, ADLS) for training data — uses credential or identity-based access
Pipeline Multi-step workflow of ML tasks with reusable components
Environment Docker container definition (base image + conda/pip packages) for reproducible training
Managed Identity System or user-assigned identity for AML compute to access Storage, Key Vault, etc.

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Training data leaked via datastore misconfiguration Datastore with credential-based access and over-broad permissions An AML datastore is configured with a Storage Account access key for the entire storage account. All users with AML Workspace Reader access can list datastores and retrieve the connection string. A contractor accesses the datastore and downloads 2 TB of training data containing customer PII. The access key provides full read/write access to all blobs in the storage account — not just the training data container. Mass PII leak from over-broad datastore credentials High
2 Public workspace endpoint — compute metadata accessible AML Workspace public access not disabled An AML Workspace has a public endpoint. Compute instances are provisioned in a VNet but the workspace control plane is accessible from the internet. An attacker enumerates the public workspace URL (via Azure API), discovers the workspace exists, and probes compute instance metadata endpoints. The workspace control plane should be private but the public endpoint was never disabled during deployment. Workspace metadata exposed via public control plane Med
3 Model poisoning — training data tampered No data integrity controls An AML pipeline reads training data from an Azure Blob container with broad write permissions. A compromised storage account allows an attacker to modify training data (add backdoor trigger samples — adversarial ML). The model is retrained on the poisoned dataset. The deployed model behaves normally for most inputs but exhibits controlled misbehavior when a specific trigger pattern is present in inference requests. Model backdoor inserted via poisoned training data Med
4 No model versioning or lineage — untracked deployments Model registry not used Models are deployed directly to endpoints without using the AML model registry. Months later, a model exhibits bias and poor performance on a demographic group. The organization cannot determine which training dataset version was used, what hyperparameters were applied, who trained the model, or what evaluation metrics justified deployment. Auditors require model lineage for AI governance — it doesn't exist. No model lineage for AI governance and compliance High
5 Compute instance with public IP Exposed development VM AML Compute Instances (development VMs for data scientists) are provisioned with public IP addresses enabled. A data scientist uses a weak password for their AML account. An attacker brute-forces the account and gains access to the compute instance, which has access to all datastores (training data, models) and the AML workspace API. From the compute instance, the attacker can pivot to storage accounts and other workspace resources. Compute instance pivot point via exposed development VM Med
6 No drift monitoring — model performance degrades silently No production monitoring A fraud detection model is deployed to a Managed Online Endpoint. No data drift monitoring is configured. Over 6 months, the distribution of incoming transaction features shifts significantly (new payment methods, different amount distributions). The model's fraud detection accuracy degrades from 94% to 71%. The degradation goes unnoticed because there are no monitoring alerts on prediction confidence or data distribution shift. Degraded model performance undetected — fraud missed High
7 All workspace roles granted to all data scientists Over-broad workspace RBAC All members of a 50-person data science team are granted AzureML Data Scientist role plus Storage Blob Data Owner on all datastores. When a project ends, the team members' access is not revoked. Ex-team members retain access to sensitive training datasets and model artifacts for months after their project concludes. Data access persists beyond project lifecycle Med
8 Inference endpoint exposed publicly with no authentication Managed Online Endpoint key auth disabled A model is deployed to a Managed Online Endpoint. The deployment is configured without key-based or token-based authentication (auth_mode: anonymous). The endpoint URL is discoverable via the AML workspace API. Competitors discover the publicly accessible endpoint and query the model extensively, extracting approximate model behavior through repeated inference (model extraction attack). Model theft via unauthenticated public inference endpoint Med
9 Sensitive PII in training artifacts stored unencrypted Default encryption only Training job output artifacts — feature importance files, model explanation outputs — contain derived statistics about individual training records (partial PII). These artifacts are stored in the AML workspace default storage account. The storage account uses Microsoft-managed encryption keys, but the organization's compliance requirement mandates customer-managed keys (CMK) for all data containing PII derivatives. PII-derived artifacts not meeting CMK encryption requirement Med
10 GPU cluster left running after training — cost overrun No cluster auto-shutdown An AML compute cluster configured with 16 A100 GPUs (Standard_NC96ads_A100_v4, $30/hour/VM) is used for a one-time training run. The minimum node count is set to 2 instead of 0 (default idle scale-down). After training completes, 2 GPU VMs remain running permanently. The team does not notice. The monthly cost for 2 idle A100 VMs is $43,200. The waste continues for 3 months before a cost review discovers it. $130K+ wasted on idle GPU VMs from misconfigured min nodes High

Mitigation Controls

# Failure Mitigation Layer
1 Datastore credential leak Use identity-based datastore access (Managed Identity); grant Storage Blob Data Reader only on specific containers Identity-based access
2 Public workspace endpoint Enable private workspace (public_network_access: Disabled); deploy private endpoint for workspace Private workspace
3 Model poisoning Sign training datasets with checksums; restrict write access to training data containers; validate data before training Data integrity
4 No model lineage Register all models in AML model registry before deployment; track run lineage via MLflow Model governance
5 Compute instance with public IP Disable public IP on all compute instances; require SSH via Bastion or VPN Compute security
6 No drift monitoring Configure AML data drift monitoring; alert when input feature distribution exceeds threshold Model monitoring
7 Over-broad workspace RBAC Apply least-privilege AML roles (AzureML Data Scientist, AzureML Compute Operator); review access quarterly RBAC
8 Unauthenticated endpoint Enable key-based or AAD token authentication on all Managed Online Endpoints Endpoint auth
9 CMK requirement not met Configure workspace with Customer-Managed Keys; apply CMK to all associated storage accounts Encryption
10 GPU cluster min nodes > 0 Set compute cluster minimum node count to 0; configure idle shutdown time; use cost alerts Cost control

Top 3 Critical Mitigations

  1. Identity-Based Datastore Access via Managed Identity: Replace all credential-based datastores (access keys, SAS tokens) with identity-based access using Managed Identity. Grant the compute cluster's managed identity Storage Blob Data Reader scoped to the specific training data container — not the entire storage account.
  2. Private Workspace with No Public Endpoint: Disable the AML workspace public endpoint and deploy private endpoints for workspace, storage, Key Vault, and container registry. Compute clusters and instances should be VNet-injected. This prevents public internet enumeration and lateral movement from compromised developer machines.
  3. Register All Models in AML Model Registry with MLflow Lineage: Every model deployed to an endpoint must be registered in the AML model registry with training run lineage (data version, code version, hyperparameters, evaluation metrics). This is required for AI governance, compliance audits, and incident response when a deployed model behaves unexpectedly.

Key Takeaways

  • Use identity-based datastore access (Managed Identity) exclusively — credential-based datastores create shared key blast radius across all workspace users.
  • Disable the AML workspace public endpoint; deploy all components (workspace, storage, Key Vault, ACR) with private endpoints in a VNet.
  • Register all models in the AML model registry with full MLflow lineage — this is required for AI governance, compliance, and debugging production model failures.
  • Set compute cluster minimum nodes to 0 and configure idle shutdown — GPU VM costs can grow rapidly from misconfigured minimum node counts.
  • Monitor production models for data drift — model performance can degrade silently over months without distribution shift alerting.
