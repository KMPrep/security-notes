Azure Container Instances (ACI)

Category: Compute
Azure Service Type: Serverless Containers
Key Docs: ACI Overview

Overview

Azure Container Instances provides on-demand container execution without managing a cluster. Containers start in seconds and bill per second of CPU and memory consumption. ACI supports Linux and Windows containers, multi-container groups (sidecar pattern), VNet injection for private network access, managed identity for Azure resource access, persistent volume mounts via Azure Files, and GPU SKUs. It is suited for short-lived batch jobs, CI agents, event-driven processing, and dev/test workloads.

How It Works

Core Concepts

Concept Description
Container Group Top-level resource; one or more containers sharing network namespace and lifecycle
Restart Policy Always, OnFailure, or Never — controls container restart behavior
VNet Injection Deploy container group into a subnet for private connectivity
Managed Identity Assign system or user identity for passwordless access to Azure resources
Azure Files Mount Persistent SMB share mounted into containers for stateful workloads
Init Containers Run to completion before app containers start; useful for bootstrapping
Environment Variables Passed at deploy time; support secure (not logged) flag for secrets
DNS Name Label Optional public FQDN on <name>.<region>.azurecontainer.io

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Registry credentials in plain text ACR admin credentials used instead of managed identity A container group spec is deployed with imageRegistryCredentials using the ACR admin username and password. The ARM template is stored in a public repo. An attacker reads the credentials, pulls all container images, and examines them for secrets baked into layers. Image theft and embedded secret exposure Med
2 Sensitive env vars in deployment manifest Secrets passed as plaintext environment variables Connection strings and API keys are passed as regular environment variables (not secureValue). Azure Activity Logs record the full ARM deployment template including all env var values. Any principal with read access to activity logs retrieves all secrets. Secrets logged in Activity Log High
3 Public container group without auth Container exposes an unauthenticated port publicly A developer deploys an ACI with a public DNS label and binds the app to port 80 without any authentication. The container runs an admin dashboard or internal API. An external attacker discovers the azurecontainer.io FQDN and accesses the service directly. Unauthorized access to internal service Med
4 No VNet injection — flat network Container communicates with internet-accessible services Without VNet injection, ACI containers use a shared Azure-managed network. Outbound traffic to a database goes over the internet. An attacker who compromises the container can exfiltrate data to external hosts with no network-level restrictions. Data exfiltration path Med
5 Overpermissive managed identity Container identity has broad Azure RBAC A container job that only needs to write to one storage container has Contributor on the entire resource group. If the container is compromised (via a vulnerable dependency), the attacker calls IMDS to get the token and pivots to all resources in the group. Lateral movement via identity Med
6 Long-running container with no restart policy Transient failure leaves job in broken state A batch job container fails halfway through due to a transient network error. restartPolicy: Never prevents any retry. The job silently terminates without completing. Downstream systems that expect a completed artifact proceed with stale or missing data. Silent data pipeline failure Med
7 Image not pinned by digest Mutable tag pulled on each restart A container group references myimage:latest. The upstream image is rebuilt with a new dependency. On the next restart, ACI pulls the new layer set, which includes a malicious package introduced via a supply chain attack. Supply chain compromise on restart Med
8 No resource limits Container consumes unbounded CPU and memory A runaway container process (infinite loop, memory leak) consumes all allocated resources. Since ACI bills per CPU/memory-second, a runaway container can generate unexpected costs. If the container group is shared with a sidecar, the sidecar also starves of resources. Cost amplification and sidecar failure Med
9 Init container writes secrets to shared volume Secrets persisted on Azure Files or tmpfs accessible to all containers An init container fetches a secret from Key Vault and writes it to a shared volume for the main container. A sidecar container in the same group (monitoring, logging) also has access to the shared volume and inadvertently logs the secret file contents. Secrets leakage via shared volume Low
10 No logging or monitoring Container stdout not shipped to Log Analytics A container fails silently. Without log shipping to a Log Analytics workspace or Application Insights, there is no visibility into errors or security events. An attacker exfiltrating data via the container leaves no trace that is detectable after the ephemeral container is deleted. Undetected failure and forensic gap Med

Mitigation Controls

# Failure Mitigation Layer
1 Registry credentials in plain text Use managed identity for ACR pull; disable admin credentials on ACR Managed Identity
2 Secrets in env vars Use secureValue for sensitive env vars; pull secrets at runtime from Key Vault via CSI or init container Secret handling
3 Unauthenticated public access Remove DNS label; deploy into VNet; front with API Management or Application Gateway Network access control
4 No VNet injection Use VNet injection subnet; apply NSG to restrict egress VNet integration
5 Overpermissive identity Scope managed identity RBAC to specific resource (container, blob, queue) Azure RBAC
6 No restart policy Use restartPolicy: OnFailure for batch jobs; implement idempotent job logic Restart configuration
7 Mutable image tags Pin images by digest; rebuild and re-pin through a secure CI pipeline Image management
8 No resource limits Set explicit requests and limits for CPU and memory on every container Resource governance
9 Secrets on shared volume Pass secrets as injected env vars or via Key Vault SDK calls; avoid shared volume for secrets Secret design
10 No logging Configure Log Analytics workspace for container log streaming; enable diagnostic settings Observability

Key Takeaways

  • ACI is ideal for short-lived, on-demand container workloads without cluster management overhead.
  • VNet injection provides private network connectivity; required for any production workload.
  • Managed identity replaces registry credentials and secret env vars.
  • Always pin images by digest and restrict resource limits to prevent cost amplification.
  • Ship logs to Log Analytics — ephemeral containers otherwise leave no forensic trace.
