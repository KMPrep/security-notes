Azure Kubernetes Service (AKS)

Category: Compute
Azure Service Type: Managed Kubernetes
Key Docs: AKS Overview, AKS Best Practices

Overview

Azure Kubernetes Service (AKS) is a managed Kubernetes offering where Azure manages the control plane (API server, etcd, scheduler) at no cost. You pay only for the agent node VMs. AKS supports system and user node pools, Windows and Linux nodes, GPU nodes, spot node pools, and virtual node scale-out to Azure Container Instances. Integration points include Azure CNI/Kubenet networking, Azure Container Registry (ACR) pull-through, Microsoft Entra ID for RBAC, Azure Monitor and Container Insights for observability, and Azure Policy for admission control.

How It Works

Core Concepts

Concept Description
Control Plane Managed by Microsoft — API server, etcd, scheduler, controller manager
Node Pool Group of VMs (agent nodes) running workloads; can have separate OS, SKU, and taints
Kubenet vs Azure CNI Kubenet uses NAT; Azure CNI assigns VNet IPs directly to pods — required for private endpoints
Workload Identity Federated identity using OIDC — replaces aad-pod-identity; maps pod service accounts to Azure managed identities
Azure RBAC for Kubernetes Entra ID principals get Kubernetes RBAC via Azure role assignments, no kubeconfig credentials needed
KEDA Kubernetes Event-Driven Autoscaling — scales deployments based on queue depth, metrics, etc.
Cluster Autoscaler Adds or removes nodes automatically based on pending pod resource requests
Azure Policy Add-on OPA Gatekeeper-based admission controller enforcing policy on pod specs
Private Cluster API server has no public IP; accessible only via private endpoint inside VNet
Node Upgrade Cordon and drain strategy; can be configured for automatic minor version upgrades

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Privileged container escape Pod runs with privileged: true or mounts host filesystem A developer deploys a debug container with securityContext.privileged: true to diagnose a disk issue. An attacker who exploits an application vulnerability inside the container uses the privileged context to mount the host root filesystem (/proc/1/root), read node credentials, and escape to the underlying VM. From there they enumerate all pods and steal Kubernetes secrets and Azure credentials. Container escape to node and cluster High
2 Exposed Kubernetes dashboard Dashboard deployed with cluster-admin role A legacy cluster has the Kubernetes Dashboard deployed with a service account bound to cluster-admin. The dashboard is exposed via a LoadBalancer service. An attacker finds the public IP, accesses the dashboard without authentication, and uses it to deploy malicious pods, read all secrets, or escalate to node-level access. Full cluster takeover High
3 Overpermissioned workload identity Pod identity has Owner or Contributor on subscription A microservice's workload identity is assigned Contributor at subscription scope for convenience. An attacker who achieves RCE inside the pod calls the IMDS endpoint, obtains the Azure AD token, and uses it to enumerate the entire subscription, read Key Vault secrets, and create backdoor resources. Subscription-level compromise High
4 Unpatched node pool OS Node pool OS not updated; known kernel CVE present A node pool runs Ubuntu 20.04 and has not received OS updates in 6 months. A CVE for the kernel allows a container-to-host escape via a misconfigured cgroup or namespace bug. An attacker who gets code execution inside any container on the node exploits the kernel bug to break containment and access node-level secrets. Node compromise and lateral movement High
5 etcd exposed or unencrypted Control plane etcd not encrypted at rest AKS encrypts etcd by default, but a self-managed or improperly configured cluster stores Kubernetes secrets in plaintext in etcd. An attacker with access to the control plane (or a backup of etcd data) reads all Kubernetes Secrets including service account tokens and TLS certificates. All cluster secrets exposed Med
6 No network policy All pod-to-pod traffic unrestricted A multi-tenant cluster hosts workloads from different teams with no Kubernetes NetworkPolicy applied. An attacker who compromises one pod scans all other pod CIDRs across namespaces, reaches a database service, and extracts data — completely unconstrained by any network boundary. Lateral movement across namespaces High
7 Image pull from public registry without scanning Pods pull images directly from Docker Hub A deployment spec references a Docker Hub image without a digest pin. The image tag is mutable. A supply chain attacker poisons the upstream image tag, and the next pod restart or node replacement pulls the malicious image and executes arbitrary code inside the cluster. Supply chain compromise Med
8 API server not private Public API server accessible from internet The AKS API server has a public IP. An attacker discovers the endpoint and targets it with credential stuffing or exploits a Kubernetes API server vulnerability. If any default or weak service account token is exposed, the attacker can interact with the Kubernetes API. Unauthorized API access Med
9 Unrestricted secret access All pods can read all Kubernetes Secrets No RBAC restrictions on Kubernetes Secret reads. A compromised pod in namespace frontend mounts or reads Secrets from namespace payments, including database credentials and TLS private keys, enabling lateral movement across service boundaries. Cross-namespace secret theft Med
10 No pod disruption budget during upgrades Node drain terminates too many replicas simultaneously A deployment has 3 replicas. During a node pool upgrade, AKS drains two nodes simultaneously. With no PodDisruptionBudget, all 3 pods are evicted at once, causing a complete service outage during the maintenance window. Availability failure during upgrades Med

Mitigation Controls

# Failure Mitigation Layer
1 Privileged container Enforce restricted Pod Security Standards; use Azure Policy add-on to deny privileged containers Admission control
2 Exposed dashboard Remove Kubernetes Dashboard; use kubectl port-forward for local access only Dashboard hardening
3 Overpermissioned workload identity Use Workload Identity with least-privilege Azure RBAC scoped to specific resources Workload Identity + RBAC
4 Unpatched node OS Enable auto-upgrade for node OS patches; use AKS managed upgrades with maintenance windows Node upgrade policy
5 etcd plaintext secrets AKS encrypts etcd by default; enable encryption at rest with customer-managed keys Encryption at rest
6 No network policy Deploy Calico or Azure CNI Powered by Cilium; apply default-deny NetworkPolicy per namespace Network policy
7 Mutable image tags Pin images by digest (image@sha256:...); scan with Microsoft Defender for Containers or Trivy Image security
8 Public API server Enable private cluster mode; use authorizedIPRanges for emergency break-glass access Control plane hardening
9 Unrestricted secret access Apply Kubernetes RBAC with namespace-scoped roles; use Secrets Store CSI Driver with Key Vault RBAC + secrets management
10 No PodDisruptionBudget Define PDB for all production deployments with minAvailable or maxUnavailable Availability controls

Top 3 Critical Mitigations

  1. Pod Security Standards (restricted) + Azure Policy Add-on: Blocks privileged containers, host path mounts, and hostNetwork — the most common container escape vectors — at admission time.
  2. Workload Identity with scoped RBAC: Replaces node-level managed identity with per-pod federated identity, ensuring a compromised pod only has access to its own resources.
  3. Private cluster + NetworkPolicy default-deny: Removes the API server from the internet attack surface and isolates pod-to-pod communication to declared flows only.

Key Takeaways

  • AKS manages the control plane; you manage node pools, workloads, and RBAC.
  • Workload Identity (OIDC federation) replaces node-level managed identity for per-pod Azure auth.
  • Pod Security Standards and Azure Policy Add-on enforce container security at admission time.
  • NetworkPolicy default-deny is essential for multi-tenant clusters and lateral movement prevention.
  • Private clusters + authorized IP ranges eliminate the API server from the internet attack surface.
