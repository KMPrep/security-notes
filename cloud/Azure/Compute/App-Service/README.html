Azure App Service

Category: Compute
Azure Service Type: Platform as a Service (PaaS)
Key Docs: App Service Overview, App Service Plans

Overview

Azure App Service is a fully managed PaaS for hosting web applications, REST APIs, and mobile backends. It supports .NET, Node.js, Python, Java, PHP, and Ruby runtimes, as well as custom containers. App Service abstracts the underlying OS and infrastructure — Microsoft handles patching, load balancing, and availability. Features include deployment slots for blue-green deployments, autoscaling, custom domains with TLS, VNet integration, Easy Auth (built-in authentication), and WebJobs for background tasks.

How It Works

Core Concepts

Concept Description
App Service Plan Compute tier (Free, Shared, Basic, Standard, Premium, Isolated) defining vCPU, RAM, and scaling
Deployment Slot Staging environment with independent config; swap into production with zero downtime
Easy Auth Built-in Entra ID, Google, Facebook, GitHub, or Twitter authentication with no code changes
VNet Integration Outbound traffic from the app routes through a designated VNet subnet
Private Endpoint Inbound traffic to the app arrives only via a private IP inside a VNet
Managed Identity System or user identity for passwordless access to Key Vault, Storage, SQL, etc.
SCM / Kudu Advanced management site for logs, console, file browsing, and deployment triggers
App Settings Environment variables injected at runtime; Key Vault references supported
WebJobs Background tasks (triggered or continuous) running inside the same App Service plan

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 SCM / Kudu endpoint exposed publicly Kudu runs at <app>.scm.azurewebsites.net with default basic auth A developer leaves Kudu basic authentication enabled (username/password) on a production app. An attacker brute-forces the credentials or finds them in a leaked .publishsettings file. Through the Kudu console, the attacker gets an interactive OS shell, can read all app files, environment variables (including connection strings), and deploy arbitrary code. Full app compromise and secret exfiltration High
2 Secrets in app settings Connection strings visible via ARM API Any AAD principal with Microsoft.Web/sites/config/read can call the Azure API and retrieve all app settings, including database connection strings and API keys. A compromised CI/CD service principal or an over-privileged developer account retrieves all secrets in one API call. Credential disclosure High
3 Missing authentication on public web app No Easy Auth and no application-level auth An internal API app is deployed to App Service and intended only for internal use, but the app has no authentication. The public DNS name <app>.azurewebsites.net resolves globally. An external attacker discovers the hostname and accesses all API endpoints without any credentials, reading sensitive business data. Unauthorized data access High
4 Deployment slot swap with mismatched secrets Staging slot using different secrets than production Staging slot is configured with test database credentials. During a slot swap, Azure swaps the app content but the production connection string is a slot-sticky setting. After the swap, the new production code points to the test database, causing data integrity issues. Data written to wrong environment Med
5 Unrestricted outbound — data exfiltration No VNet integration and no outbound restriction A compromised app uses its outbound internet access to exfiltrate data to an attacker-controlled server. Without VNet integration and NSG egress rules, all outbound traffic is unrestricted and no network-level log captures the exfiltration path. Data exfiltration via compromised app Med
6 HTTPS not enforced App accessible over HTTP An App Service is deployed without the "HTTPS Only" setting. A man-in-the-middle attack on an unencrypted Wi-Fi network intercepts the HTTP authentication cookie from an authenticated user session, allowing session hijacking without any server-side compromise. Session hijacking via network interception Med
7 Old TLS or weak ciphers Minimum TLS version not enforced An App Service defaults to TLS 1.0. An attacker on the network path performs a BEAST or POODLE downgrade attack against a TLS 1.0 connection, recovering plaintext from the encrypted session. Decryption of in-transit data Med
8 IP restrictions bypass via direct origin call App behind Front Door but no origin restrictions Traffic is intended to flow only through Azure Front Door (which has WAF). However, App Service's IP access restriction is not configured. An attacker who discovers the azurewebsites.net origin hostname bypasses Front Door and WAF entirely by sending requests directly to the origin. WAF bypass, direct exploit path High
9 Unmanaged custom domain TLS expiry Custom domain certificate not auto-renewed An App Service uses a manually uploaded TLS certificate for a custom domain. The certificate expires without automated renewal. HTTPS connections start failing, and the site becomes inaccessible to users. If the expired domain is also used in strict HSTS environments, the site is entirely unreachable. Availability failure and user trust loss Med
10 WebJob running without resource limits Background job consumes all CPU/memory of the plan A WebJob performs a CPU-intensive import operation. Because it shares the App Service Plan with the web application, it consumes all available CPU, causing the main web app to become slow or unresponsive. There is no process isolation between the WebJob and the web app in the same plan. Application performance degradation Med

Mitigation Controls

# Failure Mitigation Layer
1 SCM endpoint exposed Disable basic auth for SCM; restrict SCM to known IPs or Entra ID-only access Kudu access control
2 Secrets in app settings Use Key Vault references (@Microsoft.KeyVault(...)) for all sensitive values Key Vault integration
3 Missing authentication Enable Easy Auth with Entra ID for internal apps; use API Management for external APIs Authentication layer
4 Slot swap secret mismatch Mark sensitive settings as slot-sticky so they do not swap with deployment slots Deployment slot config
5 Unrestricted outbound Enable VNet integration; apply NSG to the integration subnet with egress rules Network controls
6 HTTP access allowed Enable "HTTPS Only" in App Service TLS settings TLS configuration
7 Weak TLS Set minimum TLS version to 1.2 and restrict to strong cipher suites TLS configuration
8 Origin access bypass Set IP access restrictions to Azure Front Door service tags; use X-Azure-FDID header validation IP restrictions
9 TLS cert expiry Use App Service Managed Certificates (free, auto-renewed) for custom domains Certificate management
10 WebJob resource contention Run WebJobs on a separate App Service Plan; consider Azure Functions for background tasks Compute isolation

Top 3 Critical Mitigations

  1. Key Vault references for all app settings: Prevents credential exposure via ARM API or portal. The app retrieves secrets at runtime via its managed identity — no secret is ever stored in the App Service configuration.
  2. IP access restrictions + Front Door FDID validation: Locks the origin to only accept traffic from Azure Front Door, ensuring the WAF policy is always enforced and cannot be bypassed.
  3. Disable basic auth for SCM (Kudu): Removes the most commonly exploited credential-based access to App Service internals. Enforce Entra ID-only access to the SCM endpoint.

Key Takeaways

  • App Service is PaaS for web apps and APIs; Microsoft manages the platform, you manage the app and configuration.
  • Deployment slots enable zero-downtime blue-green deployments with instant swap and automatic rollback.
  • Easy Auth provides built-in Entra ID authentication with zero code changes — ideal for internal tools.
  • Key Vault references and managed identity eliminate all static credentials from app configuration.
  • Lock the origin with IP access restrictions to Front Door service tags to prevent WAF bypass.
