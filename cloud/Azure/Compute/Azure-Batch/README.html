Azure Batch

Category: Compute
Azure Service Type: Managed HPC and Parallel Compute
Key Docs: Azure Batch Overview

Overview

Azure Batch enables large-scale parallel and high-performance computing (HPC) workloads without managing cluster infrastructure. A Batch account contains pools of compute nodes (VMs), jobs, and tasks. Batch allocates and manages nodes, queues tasks, retries failures, and deallocates nodes when work is complete. It integrates with Azure Storage for input/output data, supports custom VM images, Docker containers, low-priority (Spot) nodes for cost reduction, and VNet injection for private network access. Common use cases include rendering, genomics, financial risk simulation, ETL, and CI workloads.

How It Works

Core Concepts

Concept Description
Batch Account Top-level resource; linked to a storage account for job I/O
Pool Group of compute nodes (VMs); can be fixed or auto-scale with formulas
Node VM in a pool running the Batch node agent; executes task commands
Job Container for related tasks; has a pool association
Task Single unit of work; runs a command line or container on a node
Start Task Runs on every node at pool join time; used for software installation
Job Manager Task Manages job lifecycle; can submit additional tasks dynamically
Multi-Instance Task Used for MPI workloads requiring multiple nodes per task
Certificate PEM/PFX cert installed on pool nodes for app authentication
Low-Priority Nodes Spot-priced VMs; can be preempted with short notice

Step-by-Step Flow

sequenceDiagram
    participant App as Batch Client App
    participant BatchSvc as Azure Batch Service
    participant Pool as Compute Pool
    participant Node as Compute Node
    participant Storage as Azure Storage
    App->>BatchSvc: Create pool with node config and start task
    BatchSvc->>Pool: Allocate VMs, install node agent
    Pool->>Node: Run start task (install dependencies)
    App->>BatchSvc: Submit job with tasks
    BatchSvc->>Node: Schedule and dispatch task
    Node->>Storage: Download input files
    Node->>Node: Execute task command
    Node->>Storage: Upload output files
    Node-->>BatchSvc: Task complete, exit code 0
    BatchSvc-->>App: Task result available

What Can Go Wrong

| # | Failure / Attack | Root Cause | Attack Scenario | Impact | Likelihood |
| --- | ------------------------------------------------------ | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------- | --- |
| 1 | Storage account key in pool environment | Credentials passed as task env vars | A pool is configured with a storage account key in a task environment variable for data access. A task on a compromised or shared node reads the environment variable and exfiltrates the storage key. The attacker uses the key to read or delete all blobs in the storage account. | Storage credential theft | High |
| 2 | No node isolation — multi-tenant pool | Shared pool processes data from multiple customers | An organization runs a single pool shared across teams. Task A from Team 1 writes temporary files to the node's local disk. Task B from Team 2, scheduled on the same node, reads the temporary directory and accesses Team 1's intermediate output data. | Cross-tenant data leakage | Med |
| 3 | Start task runs untrusted code | Start task pulls setup script from public URL | The pool's start task runs curl <public_url> | bash to install dependencies. An attacker compromises the URL or injects a malicious package. Every new node in the pool executes the attacker's code at startup, establishing persistence on all pool nodes. | Node-level RCE on pool join | Med |
| 4 | Low-priority node preemption — data loss | Spot VMs preempted mid-task without checkpointing | A long-running simulation runs on low-priority nodes. A node is preempted by Azure after 6 hours of computation. The task has no checkpointing mechanism. Batch marks the task as failed and retries from the beginning, wasting hours of compute time and potentially causing missed deadlines. | Compute waste and job failure | Med |
| 5 | Pool nodes accessible from internet | NSG allows unrestricted inbound on pool VNet | Pool nodes are deployed without VNet injection. Node IP addresses are reachable from the internet. An attacker scans the IP range, finds an open management port left by a start task, and establishes a connection to a pool node. | Unauthorized node access | Med |
| 6 | Overpermissive managed identity on nodes | Node identity has broad RBAC | The pool's managed identity is assigned Contributor on the subscription. A task running attacker-controlled code calls IMDS on the node to retrieve the identity token and uses it to provision new resources or exfiltrate data from other Azure services. | Subscription-level lateral movement | Med |
| 7 | No task output persistence | Task results only on ephemeral node disk | Task output is written to the local node disk only. The node is deallocated at job completion. All task output is permanently lost. A downstream consumer expects the data to be in storage but finds nothing. | Data loss from ephemeral storage | High |
| 8 | Auto-scale formula error — runaway scale-out | Formula bug triggers unbounded node allocation | An auto-scale formula has a logic error that evaluates to maxNodes continuously. Batch scales the pool to the subscription's vCPU quota ceiling, consuming all compute quota. Other workloads in the subscription fail to scale because no quota remains. Costs spike dramatically. | Quota exhaustion and cost spike | Med |
| 9 | No task retry — transient failure treated as permanent | maxTaskRetryCount = 0 on flaky operations | Tasks interact with an external API that occasionally returns HTTP 503. With no retry policy, all tasks that hit the transient error are immediately marked failed and the job fails. The root cause is never investigated because the one-shot task does not retry. | Job failure from transient error | Med |
| 10 | Certificates stored as Batch account certs | Legacy cert API stores private keys on all nodes | A TLS private key certificate is added to the Batch account using the deprecated certificate API. The cert is installed on every node in every pool in the account. A compromised node in any pool can read the private key, allowing TLS decryption or impersonation of the service. | Private key exposure across all nodes | Med |

Mitigation Controls

# Failure Mitigation Layer
1 Storage key in env vars Use managed identity for storage access; use SAS tokens with minimum permissions if keys required Identity + access
2 Shared pool data leakage Use dedicated pools per tenant or team; clean up node disk between tasks using pool task cleanup Pool isolation
3 Untrusted start task script Host start task scripts in private Azure Storage with managed identity access; validate with checksum Start task security
4 Low-priority preemption data loss Implement task checkpointing; use maxTaskRetryCount to rerun from last checkpoint; mix dedicated and low-priority Resilience design
5 Nodes internet-accessible Deploy pools into a VNet subnet; apply NSG with no inbound from internet Network controls
6 Overpermissive node identity Scope managed identity to specific storage containers and queues only Azure RBAC
7 No task output persistence Use Batch file conventions or explicit upload in task command to write outputs to Azure Storage Output design
8 Auto-scale formula bug Test formulas in staging pools; set maximum node count cap; monitor node count with alerts Auto-scale governance
9 No retry on transient error Set maxTaskRetryCount appropriately; implement exponential backoff in task code Retry policy
10 Legacy certificate API Migrate to Key Vault references or managed identity; avoid Batch certificate API entirely Certificate management

Key Takeaways

  • Azure Batch provides managed parallel compute for HPC and large-scale job workloads without cluster management.
  • Use managed identity for storage and downstream access — never store credentials in env vars or start tasks.
  • VNet injection and NSGs are essential to prevent pool nodes from being internet-accessible.
  • Implement checkpointing and retry policies for low-priority (Spot) nodes to handle preemption gracefully.
  • Always write task output to Azure Storage — node local disk is ephemeral and lost on deallocation.
