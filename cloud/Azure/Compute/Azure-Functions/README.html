Azure Functions

Category: Compute
Azure Service Type: Serverless / Function as a Service (FaaS)
Key Docs: Azure Functions Overview, Triggers and Bindings

Overview

Azure Functions is an event-driven serverless compute platform. Functions execute in response to triggers such as HTTP requests, timer schedules, queue messages, blob events, Cosmos DB change feed, Event Hubs, and many more. The runtime hosts function code and scales automatically — including to zero — based on incoming event rate. Functions run in a managed sandbox with configurable runtime isolation. Hosting plans include Consumption (true serverless), Flex Consumption (vNet-integrated serverless), Premium (pre-warmed workers with VNet), and Dedicated (App Service Plan for predictable load).

How It Works

Core Concepts

Concept Description
Trigger Event source that invokes a function (HTTP, timer, queue, blob, etc.)
Binding Declarative connection to input/output data sources without SDK boilerplate
Function App Unit of deployment; shares runtime, config, and identity
Hosting Plan Consumption, Flex Consumption, Premium, or Dedicated
Durable Functions Extension for stateful orchestrations, fan-out/fan-in, and async patterns
Managed Identity System or user identity for secretless access to downstream resources
KEDA Kubernetes-based event-driven autoscaling (used in Container Apps-hosted functions)
Cold Start Latency on first invocation after scale-to-zero in Consumption plan

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Unauthenticated HTTP trigger Function key not required; anonymous auth level A Function App exposes an HTTP-triggered function with AuthorizationLevel.Anonymous to simplify development. The developer forgets to change it before deployment. An external attacker discovers the endpoint via subdomain enumeration (*.azurewebsites.net) and directly invokes the function, triggering backend writes, data processing, or downstream API calls. Unauthorized business logic execution High
2 Function key leakage Keys stored in source code or logged A team stores the function host key (x-functions-key) in a .env file that gets committed to a public GitHub repository. Attackers harvesting GitHub secrets find the key and call the function endpoint directly, bypassing any upstream controls. Unauthorized API access High
3 Overly permissive managed identity Function identity has broad RBAC on storage, DB, or subscriptions A function identity is granted Storage Blob Data Contributor on the entire storage account. An attacker who can trigger arbitrary code execution (via a dependency confusion or code injection attack) runs curl with the IMDS token to list and download all blobs in the account. Data exfiltration Med
4 Dependency confusion or supply chain Third-party packages in requirements pulled at build time A Python function app pulls packages from PyPI without pinning exact versions. An attacker publishes a malicious package with a name that shadows an internal package. When the function app redeploys, it downloads and executes the malicious package, establishing a reverse shell or exfiltrating environment variables including connection strings. Supply chain RCE Med
5 Secrets in application settings Connection strings visible in ARM export Connection strings and API keys are stored as App Settings (environment variables). Any principal with Microsoft.Web/sites/config/read on the Function App can retrieve all settings via az functionapp config appsettings list. An over-privileged developer account or a compromised CI token exposes all secrets. Secrets disclosure High
6 Runaway invocations — cost amplification Poison queue message triggers infinite retry loop A malformed message on a Service Bus queue causes the function to throw an unhandled exception. The message is dead-lettered after max delivery attempts, but a bug in the error handler re-enqueues it. The function invokes thousands of times per minute, generating large compute costs and consuming downstream API quotas. Cost amplification, DoS of downstream APIs Med
7 No VNet integration — internet-reachable dependencies Premium plan without VNet triggers against public endpoints A function connects to an Azure SQL database over its public endpoint because VNet integration was not configured. An attacker who compromises the function's connection string can connect to the database from any internet host, bypassing any planned network isolation. Network isolation failure Med
8 Verbose error responses Unhandled exceptions return stack traces in HTTP response A function returns a full .NET or Python stack trace in a 500 response, including file paths, class names, and internal library versions. An attacker uses this information to fingerprint the runtime, identify vulnerable libraries, and target known CVEs. Information disclosure Med
9 Cold start latency SLA breach Consumption plan with cold starts in latency-sensitive path A payment processing function runs on a Consumption plan. After a period of inactivity, the first invocation experiences a 2-4 second cold start. A payment system with a 1-second SLA timeout marks the transaction as failed, triggering a double-payment retry. Business logic failure from latency Med
10 Logging of sensitive data Input payloads containing PII logged to Application Insights A function logs the full request body for debugging. The body contains credit card numbers or SSNs. Application Insights retains the data for 90 days and is accessible to anyone with Reader access on the workspace. PII/PCI data leakage into telemetry Med

Mitigation Controls

# Failure Mitigation Layer
1 Unauthenticated HTTP trigger Use AuthorizationLevel.Function minimum; front with API Management for auth policies Function auth level
2 Function key leakage Store function keys in Key Vault; rotate regularly; use Managed Identity instead of keys where possible Key management
3 Overly permissive managed identity Scope RBAC to specific container or resource; use user-assigned identity per function app Azure RBAC
4 Dependency confusion Pin all package versions; use private artifact feeds (Azure Artifacts); scan with Dependabot Supply chain controls
5 Secrets in app settings Reference Key Vault secrets using @Microsoft.KeyVault(...) syntax in app settings Key Vault reference
6 Runaway invocations Configure dead-letter queue; set max retry count; implement idempotency checks Queue configuration
7 No VNet integration Use Premium or Flex Consumption plan with VNet integration; use private endpoints for dependencies VNet integration
8 Verbose errors Catch exceptions globally; return generic 500 responses; log detail internally only Error handling
9 Cold start latency Use Premium plan with pre-warmed instances for latency-sensitive functions Hosting plan selection
10 PII in logs Sanitize inputs before logging; use Application Insights sampling; enable workspace data masking Telemetry controls

Top 3 Critical Mitigations

  1. Key Vault references for all secrets: Eliminates plaintext credentials in app settings. The function runtime fetches the secret at startup using its managed identity — no secret ever lives in ARM or the portal.
  2. Managed Identity instead of function keys: Use Entra ID-protected triggers (through API Management or EasyAuth) so that access is auditable, revocable, and tied to identity rather than a shared static key.
  3. VNet integration with private endpoints: Ensures all outbound connections from the function stay inside the Azure private network, eliminating the risk of public endpoint exposure for databases and storage.

Key Takeaways

  • Azure Functions is event-driven serverless; trigger types span HTTP, timer, queue, blob, Event Hubs, and more.
  • Hosting plan determines cold start behavior, VNet integration capability, and scaling characteristics.
  • Managed Identity + Key Vault references eliminate static secrets from all function configuration.
  • Durable Functions add stateful orchestration, long-running workflows, and human approval patterns.
  • Always front HTTP-triggered functions with API Management for consistent auth, rate-limiting, and observability.
