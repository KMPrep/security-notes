Azure Container Apps

Category: Compute
Azure Service Type: Serverless Containers (PaaS)
Key Docs: Container Apps Overview

Overview

Azure Container Apps is a serverless container platform built on Kubernetes and KEDA, abstracting away cluster management. It supports scale-to-zero, scale on HTTP, queue depth, custom metrics, or KEDA scalers. Container Apps natively integrates with Dapr (Distributed Application Runtime) for service invocation, pub/sub, state management, and secret stores. Each app runs in an environment (shared or dedicated) backed by an AKS cluster managed by Microsoft. It bridges the simplicity of ACI with the power of AKS, making it ideal for microservices, APIs, and event-driven workers.

How It Works

Core Concepts

Concept Description
Environment Logical isolation boundary; shared or dedicated networking; all apps share a VNet
Revision Immutable snapshot of an app version; traffic can be split between revisions
Ingress Built-in HTTP/HTTPS ingress with optional external or internal-only exposure
Dapr Sidecar runtime for service-to-service calls, pub/sub, state, and secret injection
KEDA Scaler Event-driven autoscaling — HTTP, Service Bus, Event Hubs, custom metrics
Managed Identity System or user identity for secretless downstream access
Secret References App-level secrets injected as env vars; Key Vault references supported
Workload Profile Dedicated or consumption profile; dedicated for memory/GPU-intensive workloads

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 External ingress on internal microservice Ingress set to external when internal-only intended A developer deploys a payment processing container app with ingress.external: true thinking it only affects internal routing. The app gets a public FQDN. An attacker enumerates *.azurecontainerapps.io subdomains and finds the endpoint, calling payment processing APIs directly without any API gateway or auth layer. Unauthorized access to internal business logic High
2 Secrets stored in revision metadata Secrets defined at app level visible in ARM Container App secrets are stored in the ARM resource definition and returned by az containerapp secret list. Any principal with Reader access on the resource can retrieve the secret names and reference them. If secrets are defined with values rather than Key Vault references, values are also exposed. Secrets disclosure via ARM API Med
3 Dapr state store misconfiguration State store component uses unencrypted or shared backend A Dapr state store component points to a Redis instance without TLS and with no access controls. Multiple container apps in the same environment share state key namespaces. App A can enumerate or overwrite App B's state if component scoping is not applied. Cross-app data tampering Med
4 No revision traffic control All traffic immediately routes to new revision on deploy A container app is deployed with a bug that breaks authentication. Because no traffic splitting is configured, 100% of traffic immediately routes to the broken revision. There is no canary rollout to detect the issue before full cutover. Full availability failure on bad deploy Med
5 KEDA scaler triggers uncontrolled scale-out Malicious messages flood queue and trigger massive scale An attacker sends 100,000 messages to an Azure Service Bus queue that a container app listens on. KEDA scales the app to maximum replicas. Each replica processes a message and calls a downstream paid external API. The cost amplification and downstream API quota exhaustion cause significant financial and operational damage. Cost amplification and downstream DoS Med
6 Overpermissive managed identity App identity has subscription-level access A container app that only needs to read from one Service Bus topic is assigned Contributor on the subscription. A compromised replica's IMDS token grants the attacker full subscription access to create, read, and delete any resource. Lateral movement via identity Med
7 No health probes on replicas Unhealthy replicas continue receiving traffic A container app replica enters a degraded state (connection pool exhausted, memory leak) but does not return HTTP 5xx. No liveness probe is configured. The ingress continues routing traffic to the degraded replica, causing slow responses and errors for a subset of users with no automatic remediation. Degraded availability, hard-to-diagnose errors Med
8 Shared environment — noisy neighbor Multiple apps in one environment compete for resources A Container Apps environment is shared across teams. A runaway app generates extreme CPU and network load, degrading the performance of all other apps in the environment. Without workload profile isolation, there is no hard resource boundary between apps. Performance degradation across apps Low
9 Container image from untrusted source Image pulled from public Docker Hub without scanning A container app spec references a third-party Docker Hub image without a digest pin. The upstream image is replaced by a malicious actor. On the next revision deployment or replica restart, the malicious image is pulled and executed in the environment. Supply chain compromise Med
10 Verbose Dapr tracing logs Dapr distributed tracing logs request payloads Dapr is configured with verbose tracing enabled in production. Request bodies containing PII or financial data are captured in the distributed trace and shipped to Application Insights, accessible to anyone with Reader access on the workspace. PII leakage into telemetry Low

Mitigation Controls

# Failure Mitigation Layer
1 External ingress on internal service Set ingress.external: false; use Dapr service invocation or API Management for cross-app calls Ingress configuration
2 Secrets in ARM Use Key Vault secret references instead of inline secret values Key Vault integration
3 Dapr state store misconfiguration Apply Dapr component scoping to restrict state store to specific app IDs; enable TLS on Redis Dapr component scoping
4 No traffic splitting Use revision traffic split (canary/blue-green) before full cutover Revision management
5 Uncontrolled KEDA scale Set maxReplicas on KEDA scaler; implement message validation and dead-lettering Scale limits
6 Overpermissive identity Scope managed identity to specific queue/topic/container; use user-assigned identity per app Azure RBAC
7 No health probes Configure liveness and readiness probes on all container apps Health probe config
8 Noisy neighbor Use dedicated workload profiles for critical apps; separate environments per team Environment isolation
9 Untrusted image Pin by digest; pull from private ACR with vulnerability scanning enabled Image security
10 Verbose Dapr tracing Set Dapr sampling rate appropriately; disable payload capture in production tracing Telemetry config

Key Takeaways

  • Container Apps is serverless Kubernetes with scale-to-zero, Dapr integration, and no cluster management.
  • Dapr provides service invocation, pub/sub, state, and secret store abstractions across microservices.
  • Set ingress to internal-only for all services not meant to be publicly accessible.
  • Use Key Vault references for all secrets and managed identity for all Azure resource access.
  • Traffic splitting between revisions enables safe canary deployments without full cutover risk.
