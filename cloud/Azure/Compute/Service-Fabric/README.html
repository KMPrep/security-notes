Azure Service Fabric

Category: Compute
Azure Service Type: Microservices and Container Orchestration
Key Docs: Service Fabric Overview

Overview

Azure Service Fabric is a distributed systems platform for deploying and managing microservices and containers at scale. Unlike AKS, Service Fabric natively supports stateful services through its reliable collections (Reliable Dictionary, Reliable Queue) without requiring an external database. It orchestrates both stateless and stateful services, manages service discovery, health monitoring, rolling upgrades, and failure handling. Service Fabric clusters can run on Azure, on-premises, or other clouds. Programming models include Reliable Services, Reliable Actors, Guest Executables, and containers.

How It Works

Core Concepts

Concept Description
Cluster Set of VMs (nodes) running the Service Fabric runtime
Node Type VM scale set group; primary node type runs system services
Reliable Services SDK-based stateless or stateful service with built-in state replication
Reliable Actors Virtual actor model; each actor has its own state and single-threaded execution
Service Manifest XML definition of service code, config, and resources
Application Manifest Aggregates service manifests; defines instance counts and health policies
Partition Unit of state distribution; stateful services partition state across replicas
Upgrade Domain Rolling upgrade grouping; ensures availability during app or cluster upgrades
Fault Domain Physical failure boundary (rack, datacenter) for replica placement
Service Fabric Explorer Web-based cluster management and health dashboard

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Service Fabric Explorer exposed publicly Management port 19080 accessible from internet A Service Fabric cluster is deployed with NSG allowing port 19080 (Service Fabric Explorer) and 19000 (client port) from 0.0.0.0/0. An attacker discovers the URL, accesses the management interface, and can deploy arbitrary application packages, delete services, or extract environment variables and certificates from running services. Full cluster management takeover High
2 Cluster certificate expired Client or cluster cert reaches expiry without rotation The cluster's primary certificate expires. Service Fabric nodes can no longer authenticate to each other. The cluster becomes unhealthy or split-brained, taking down all services. Recovery requires manual certificate rotation and cluster restart in the worst case. Full cluster outage High
3 Overprivileged RunAs policy Service runs as LocalSystem or with elevated OS privileges A service is deployed with RunAsPolicy set to LocalSystem for convenience. A code vulnerability in the service allows an attacker to execute arbitrary OS commands. Because the process runs as LocalSystem, the attacker has full node-level access including the ability to read all certificates, environment variables, and Service Fabric configuration files. Node-level compromise Med
4 No application package integrity check Malicious app package uploaded to image store Service Fabric deploys applications from a package uploaded to the cluster image store. If the CI pipeline is compromised, a malicious package is uploaded and deployed. Without image signing or digest verification, Service Fabric deploys the tampered binary across all nodes. Cluster-wide RCE via tampered deploy Med
5 Quorum loss in stateful service All secondary replicas fail leaving no quorum Three nodes in the primary node type fail simultaneously (zone outage). A stateful service with replication factor 3 loses quorum. The service is unable to accept writes or reads until quorum is restored. Data is not lost but the service is unavailable. Stateful service unavailability Med
6 Rolling upgrade misconfiguration Health check policy too lenient — bad upgrade propagates An upgrade is deployed with HealthCheckRetryTimeout: 00:00:30 (too short) and upgrade domains roll out too fast. A bug in the new version causes health check failures, but the upgrade continues before the failures are noticed. All upgrade domains receive the broken version before manual intervention. Full application outage from bad rollout Med
7 Secrets in service configuration Secrets in Settings.xml stored in plaintext Service configuration files (Settings.xml) contain database connection strings in plaintext. Any developer with deployment access to the cluster can read all configuration files via Service Fabric Explorer or by downloading the application package. Secrets disclosure to all cluster operators Med
8 Unbounded actor count — memory exhaustion Actor virtual instances not garbage collected A Reliable Actor service creates a new actor instance per user session. Inactive actors are never explicitly deleted and Service Fabric retains their state indefinitely. Over time, the actor count grows to millions, exhausting disk space and memory on the partition replica, causing service instability. Memory and disk exhaustion Med
9 No chaos testing Untested failure modes cause production surprises A stateful service has never been tested against node failures. A production node reboots unexpectedly. The failover to a secondary replica takes 30 seconds, during which the service returns errors. The SLA requires 99.99% uptime; this single event exceeds the monthly error budget. SLA violation from untested failover path Med
10 Stale DNS or naming cache Client caches stale endpoint after service move A stateless front-end service caches the address of a stateful back-end service. The stateful service's primary replica moves to a new node after a failover. The front-end continues sending requests to the old address, receiving connection refused errors for several minutes until the cache expires. Transient availability failure Med

Mitigation Controls

# Failure Mitigation Layer
1 Explorer exposed publicly Restrict NSG to management VNet CIDRs; use Azure Bastion for cluster access NSG + access control
2 Certificate expiry Automate certificate rotation with Key Vault; use secondary cert for zero-downtime rotation Certificate management
3 Overprivileged RunAs Run services as a low-privilege domain or local account; use RunAsPolicy with minimal OS rights Service security policy
4 No package integrity Sign application packages; validate checksums in the deployment pipeline Supply chain controls
5 Quorum loss Use Silver or Gold durability tier; deploy primary node type across Availability Zones Fault tolerance design
6 Upgrade misconfiguration Use health check stabilization with adequate timeouts; deploy upgrades through staging environment first Upgrade policy
7 Secrets in config Use Service Fabric secret store or Key Vault; encrypt values with cluster certificate Secrets management
8 Unbounded actors Implement actor garbage collection; use IActorGarbageCollect and set idle timeout Actor lifecycle management
9 No chaos testing Run Azure Chaos Studio or Service Fabric Testability API for fault injection Resilience testing
10 Stale naming cache Use Service Fabric ServicePartitionClient with retry and re-resolve logic Client resilience

Key Takeaways

  • Service Fabric provides native stateful microservices through Reliable Collections with automatic replication.
  • Certificate management is critical — expiry causes cluster outages; automate rotation via Key Vault.
  • Lock down management ports (19080, 19000) to private network access only — never expose to internet.
  • Use fault domains + upgrade domains aligned to Availability Zones for zone-resilient deployments.
  • Service Fabric is best suited for stateful services; AKS is preferred for stateless container workloads.
