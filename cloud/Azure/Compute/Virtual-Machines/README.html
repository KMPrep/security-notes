Azure Virtual Machines

Category: Compute
Azure Service Type: Infrastructure as a Service (IaaS)
Key Docs: VM Overview, VM Sizes

Overview

Azure Virtual Machines provide on-demand, scalable computing resources with full control over the operating system, configuration, and installed software. VMs are deployed inside a Virtual Network (VNet), attached to managed disks, and secured by Network Security Groups (NSGs). VM series span general purpose (Dsv5), compute optimized (Fsv2), memory optimized (Esv5), storage optimized (Lsv3), and GPU-accelerated (NVv4) families. Pricing models include Pay-as-you-go, Reserved Instances (1 or 3 year), Spot VMs, and Azure Hybrid Benefit for Windows/SQL licensing.

How It Works

Core Concepts

Concept Description
VM Image OS disk snapshot from Marketplace or custom gallery
VM Size vCPU, memory, temp storage, max disk, and NIC count
Managed Disk Azure-managed OS or data disk (Premium SSD, Standard SSD, Ultra)
NSG Stateful L4 firewall controlling inbound and outbound traffic
VM Extension Agent plugins for monitoring, DSC, custom script execution
Instance Metadata Service REST endpoint at 169.254.169.254 returning identity and config
Azure Bastion Browser-based SSH/RDP without public IP or open ports
Availability Set Fault domain and update domain grouping for HA within a region
VM Scale Set (VMSS) Auto-scaling group of identical VMs with load balancing
Spot VM Unused capacity at deep discount — evictable with 30s notice

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 IMDS credential theft via SSRF IMDSv1 returns managed identity tokens without auth Attacker finds an SSRF vulnerability in a web app on the VM. They send a crafted request to http://169.254.169.254/metadata/identity/oauth2/token without any token header. Azure returns an OAuth token scoped to the VM's managed identity. Attacker uses the token to access Azure Key Vault, Storage, or other resources from an external machine. Identity token theft, lateral movement High
2 Overly permissive managed identity Identity has Owner or Contributor at subscription scope A developer assigns Contributor on the subscription to a VM's managed identity for convenience. Any RCE on the VM yields full subscription-level Azure access. Attacker enumerates all resources, reads secrets from Key Vault, clones disks, or creates backdoor identities. Full subscription takeover High
3 Public SSH/RDP port open to internet NSG allows port 22 or 3389 from 0.0.0.0/0 Shodan-style scanner discovers the open port. Attacker runs credential stuffing or targets a known CVE in the SSH daemon or RDP service. Successful brute-force or exploit yields an OS shell or desktop session. Unauthorized access and RCE High
4 Unmanaged disk with public access Managed disk export URL not disabled A disk is temporarily exported for cloning. The shared access signature URL is not revoked and has no expiry. A misconfigured app logs the SAS URL. An external attacker with the URL downloads the full disk image, mounts it offline, and reads all data including credentials stored on the volume. Full disk data disclosure Med
5 Stale or unpatched VM image No patching pipeline; long-running VMs drift from baseline VM was deployed from a Marketplace image 12 months ago and never updated. A critical kernel privilege escalation CVE (e.g., Dirty Pipe, Looney Tunables) is published. An attacker with a low-privilege foothold (e.g., via a web app) exploits the kernel bug locally to gain root. Privilege escalation to root High
6 Custom script extension running untrusted code Extension scripts fetched from public or compromised URLs A VMSS launch configuration pulls a bootstrap script from a public blob storage URL without checking integrity. An attacker compromises the storage account or a public CDN host. All new VMs bootstrapped by the scale set execute the attacker's payload as root at launch time. RCE on all new VMSS instances Med
7 Secrets in VM user data or tags Tags and custom data visible to anyone with read access A developer stores a database connection string in a VM tag or custom data field for application config. Any AAD principal with Microsoft.Compute/virtualMachines/read on the resource group can retrieve this information via az vm show. Credential exposure across the org Med
8 No disk encryption Managed disks at rest without Azure Disk Encryption or CMK A VM is decommissioned but the OS disk is not deleted. The disk exists as an orphaned resource in the subscription. An attacker or rogue insider with Contributor access creates a new VM and attaches the old disk as a data disk, mounting it to read all filesystem contents. Data exposure from orphaned disks Med
9 VMSS autoscale misconfiguration Scale-in policy terminates wrong instances A stateful workload is deployed to a VMSS. The scale-in policy is set to delete the newest instances. A sudden traffic spike triggers scale-out followed by scale-in. The newest (active) instances are terminated while older (idle) instances persist, causing active sessions to drop and data to be lost from in-memory state. Application availability failure Med
10 No Azure Defender for Servers Missing EDR and threat detection on the VM A VM has no Microsoft Defender for Endpoint or Defender for Cloud agent. An attacker installs a fileless crypto miner using a reflective loader injected into a legitimate process. No behavioral detection fires. The compromise runs for weeks until a billing alert on elevated compute costs triggers investigation. Silent intrusion, undetected persistence High

Mitigation Controls

# Failure Mitigation Layer
1 IMDS credential theft Restrict outbound IMDS access from app process accounts; use workload identity where possible NSG egress + app config
2 Overly permissive managed identity Apply least-privilege Azure RBAC roles scoped to specific resource groups or resources Azure RBAC
3 Public SSH/RDP exposure Replace with Azure Bastion; restrict NSG to known IPs or use Just-in-Time VM access NSG + Defender for Cloud JIT
4 Disk export SAS URL exposed Set short SAS expiry; disable public disk access via disk network access policy Disk access policy
5 Unpatched VM Use Azure Update Manager or Automation Update Management; enforce patch compliance with Azure Policy Patch management
6 Untrusted script extension Store bootstrap scripts in private blob with managed identity access; validate integrity with hash Storage access control
7 Secrets in tags/user data Use Azure Key Vault or App Configuration; never store secrets in resource metadata Secrets management
8 Unencrypted disks Enable Azure Disk Encryption (ADE) or server-side encryption with customer-managed key (CMK) in Key Vault Disk encryption
9 VMSS scale-in misconfiguration Use Default or OldestVM scale-in policy for stateless workloads; externalize state to Redis or DB VMSS configuration
10 No endpoint detection Onboard all VMs to Defender for Cloud (Defender for Servers P2) for EDR and behavioral analytics Defender for Cloud

Top 3 Critical Mitigations

  1. Azure Bastion + JIT access: Eliminates public-facing SSH/RDP. JIT opens ports only for approved sessions with time limits, reducing the attack window to near zero.
  2. Managed Identity with scoped RBAC: Removes static credentials entirely. Scope the managed identity role to the minimum required resource and operation rather than subscription-wide.
  3. Azure Disk Encryption: Encrypts OS and data disks with keys stored in Key Vault. Protects against unauthorized disk attachment and offline data extraction from orphaned disks.

Key Takeaways

  • VMs are foundational IaaS in Azure; Managed Disks, NSGs, VNets, and Managed Identities are core primitives.
  • Azure Bastion and JIT access replace public SSH/RDP and are the recommended access patterns.
  • Managed identities eliminate static credentials; scope RBAC roles to the minimum required resource.
  • Azure Disk Encryption (ADE) and CMK-based SSE protect data at rest from orphaned disk attacks.
  • VMSS enables autoscaling; use Availability Zones for zone-redundant, highly available deployments.
