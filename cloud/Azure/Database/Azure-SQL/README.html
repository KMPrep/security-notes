Azure SQL Database

Category: Database
Azure Service Type: Managed Relational Database (PaaS)
Key Docs: Azure SQL Overview

Overview

Azure SQL Database is a fully managed, intelligent relational database service built on the SQL Server engine. Microsoft handles patching, backups, high availability, and infrastructure. It offers two primary deployment models: single database (per-database resources) and elastic pool (shared resource pool across multiple databases). Service tiers span DTU-based (Basic, Standard, Premium) and vCore-based (General Purpose, Business Critical, Hyperscale). Features include automatic tuning, threat detection, Transparent Data Encryption (TDE), Always Encrypted, Ledger tables, Advanced Threat Protection, and built-in HA via Always On availability groups.

How It Works

Core Concepts

Concept Description
Logical Server Admin namespace for databases; not a VM; holds firewall rules and Entra ID admin
Single Database Isolated database with dedicated resources
Elastic Pool Multiple databases sharing a pool of eDTUs or vCores for cost efficiency
Service Tier General Purpose (remote storage), Business Critical (local SSD + HA replicas), Hyperscale (multi-TB, tiered storage)
TDE Transparent Data Encryption — AES-256 encryption at rest, enabled by default
Always Encrypted Client-side column encryption; keys never leave client; server never sees plaintext
Active Geo-Replication Readable secondary in another region; manual failover
Auto-Failover Group Abstraction over geo-replication; automatic failover with DNS-based endpoint
Microsoft Entra Authentication Entra ID tokens for DB authentication instead of SQL logins
Azure Defender for SQL Advanced Threat Protection detecting SQL injection, anomalous access, and data exfiltration

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 SQL injection via unsanitized queries Dynamic SQL without parameterization A web application constructs SQL queries by concatenating user input. An attacker sends a payload like '; DROP TABLE users; -- or uses UNION-based injection to extract the contents of all tables. Azure Defender for SQL may detect the pattern, but if not enabled, the attack succeeds silently. Data exfiltration and destruction High
2 Firewall rule 0.0.0.0 - 255.255.255.255 Allow all Azure services checkbox ticked A developer enables "Allow Azure services and resources to access this server" in the firewall settings, which opens the database to all Azure public IPs — not just the specific resources intended. Any Azure-hosted service (including attacker VMs in Azure) can reach the database endpoint if they have valid credentials. Broader network attack surface High
3 SQL admin credentials in app config Connection string with sa or admin password stored in plaintext The database connection string including password is stored in an app settings file or environment variable. The file is committed to source control. Any developer with repo access retrieves the admin password and can log into the database with DBA-level privileges. Full database admin access High
4 No Entra ID admin — SQL auth only Only SQL logins used; no MFA, no auditability All database access uses SQL authentication (username/password). There is no MFA enforcement, no Entra ID Conditional Access policy, and no identity-based audit trail. A password brute-force attack against a weak SQL login succeeds without triggering any identity protection alert. Unauthorized DB access with no MFA High
5 Over-permissive database roles App service accounts with db_owner A connection string user is granted db_owner for simplicity. An attacker who exploits the web application can execute xp_cmdshell, create new users, read all tables, and modify schema. The principle of least privilege is violated — the app only needed SELECT on two tables. Maximum blast radius from app compromise High
6 Active Geo-Replication not configured No secondary for regional DR A production database has no geo-replication or failover group. The primary Azure region experiences an outage. The database is unavailable for hours. The RTO is breached because the recovery requires restore from geo-redundant backup, which takes significantly longer than a failover. RTO breach during regional outage High
7 No long-term retention 7-day point-in-time backup insufficient for compliance A regulated financial workload stores transaction records in Azure SQL. The default backup retention is 7 days. A compliance audit requires 7-year retention of all transaction data. Without long-term backup retention (LTR) configured, the organization cannot provide historical backups demanded by regulators. Compliance failure Med
8 Ledger table not enabled Audit trail for tamper detection not in place An insider modifies financial records directly in the database. Without Ledger tables enabled, standard query logs show the current state but not a cryptographically verifiable history of changes. The forensic investigation cannot prove which values were altered and by whom at a specific time. Insider tampering undetectable Med
9 Database diagnostic logs not sent to SIEM No audit logging to Log Analytics Failed login attempts, unusual query patterns, and bulk data exports are not logged anywhere outside the database. When a data breach is discovered, the forensics team finds no evidence of when the attack started, which tables were accessed, or how many records were extracted. Forensic gap on breach investigation Med
10 Hyperscale IOPS spike — cold pages from remote storage Hyperscale cold data read from remote storage A Hyperscale database has a workload that reads infrequently accessed pages. These pages are served from Azure remote storage with higher latency than local SSD. A sudden analytical query against cold data causes IOPS to spike and read latency to degrade, impacting concurrent OLTP transactions on the same database. Latency spike from cold page reads Med

Mitigation Controls

# Failure Mitigation Layer
1 SQL injection Use parameterized queries or ORMs; enable Azure Defender for SQL with SQL injection alerts Application + threat detection
2 Overly broad firewall Remove Allow Azure services; use private endpoint instead Firewall configuration
3 SQL credentials in config Use managed identity with Entra ID token auth; reference connection string from Key Vault Secrets management
4 SQL auth only Set Entra ID admin; disable or restrict SQL auth; enforce MFA via Conditional Access Authentication policy
5 Over-permissive roles Grant minimum required permissions (SELECT on specific tables); use schema-level grants RBAC in database
6 No geo-replication Configure active geo-replication or auto-failover group in secondary region HA and DR design
7 Short backup retention Configure long-term retention (LTR) policies for weekly/monthly/yearly backups Backup management
8 No tamper detection Enable Ledger tables for critical financial or audit tables Ledger / audit tables
9 No audit logging Enable database auditing to Log Analytics workspace; connect to Microsoft Sentinel Observability
10 Hyperscale cold page latency Use Named Replicas for analytical queries; offload to Synapse for cold analytics Architecture separation

Top 3 Critical Mitigations

  1. Private endpoint + Entra ID authentication: Removes the database from the internet attack surface and replaces passwords with token-based, MFA-enforceable identity.
  2. Managed identity for app connections: Eliminates connection strings from code and configuration. The app runtime fetches a short-lived token from Entra ID using its identity — no password ever exists.
  3. Azure Defender for SQL + auditing to Log Analytics: Provides behavioral threat detection for SQL injection, anomalous access, and data exfiltration with full audit trail for forensics.

Key Takeaways

  • Azure SQL is managed SQL Server PaaS — Microsoft handles HA, patching, and backups.
  • Use Entra ID authentication with managed identity — eliminate SQL passwords from application config.
  • Always Encrypted provides column-level encryption where keys never leave the client.
  • Configure auto-failover groups for automatic regional DR with transparent DNS failover.
  • Enable Azure Defender for SQL and audit logging to Log Analytics for threat detection and compliance.
