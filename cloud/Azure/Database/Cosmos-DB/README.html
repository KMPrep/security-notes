Azure Cosmos DB

Category: Database
Azure Service Type: Globally Distributed Multi-Model NoSQL
Key Docs: Cosmos DB Overview

Overview

Azure Cosmos DB is a globally distributed, multi-model database service designed for planet-scale applications. It offers turnkey global distribution, single-digit millisecond reads and writes at any scale, and five well-defined consistency levels (Strong, Bounded Staleness, Session, Consistent Prefix, Eventual). Cosmos DB supports multiple APIs: NoSQL (native), MongoDB, Cassandra, Gremlin (graph), and Table. It provides automatic and instantaneous scaling with serverless or provisioned throughput (RU/s). Integrated with Microsoft Entra ID, change feed, multi-region writes, and Synapse Link for analytics.

How It Works

Core Concepts

Concept Description
Request Unit (RU) Normalized cost unit; 1 RU = 1KB point read; writes cost more
Partition Key Field that distributes items across logical partitions; determines scalability
Consistency Level Strong, Bounded Staleness, Session, Consistent Prefix, Eventual — SLA vs latency tradeoff
Global Distribution Add or remove regions with a click; data replicated automatically
Multi-Region Write Write to any region; conflict resolution via LWW or custom merge
Change Feed Ordered, incremental log of all changes in a container; enables event-driven patterns
Synapse Link Analytical store for OLAP queries without impacting OLTP performance
Serverless Pay per RU consumed, no capacity management; for spiky or low-traffic workloads
Provisioned Throughput Reserved RU/s capacity; can autoscale within a configured max
TTL Item-level time-to-live for automatic expiration

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Primary key used by application Account master key embedded in connection string The Cosmos DB primary key is stored in an application's environment variable. Any developer with access to the app configuration retrieves the key and gains full read-write-delete access to every document in every container in the account — with no audit trail tied to their identity. Full account access and data manipulation High
2 Hot partition — throughput exhaustion Poor partition key choice causes uneven distribution A social media app uses userId as the partition key. A few celebrity users generate millions of events per minute, all hitting the same logical partition. That partition's RU/s limit is constantly exceeded, causing HTTP 429 (too many requests) errors for other users even though the total account throughput is not exhausted. Availability failure for all users due to hot partition High
3 Cross-partition queries — RU runaway Queries without partition key filter scan all partitions A developer writes a query SELECT * FROM c WHERE c.status = 'pending' with no partition key filter. Cosmos DB fans this query out to all physical partitions. For a 10 TB container with 500 partitions, this query consumes thousands of RUs, depleting the provisioned throughput and throttling all other operations. RU exhaustion and workload throttling High
4 No private endpoint — account accessible from internet Cosmos DB account reachable via public endpoint Cosmos DB endpoint is accessible globally by default. An attacker who obtains the primary key (via a leaked environment variable) can read, write, or delete all data from any internet connection. Without private endpoint, network controls cannot supplement identity controls. Remote data exfiltration with stolen key Med
5 Consistency level mismatch — stale reads App assumes strong consistency but uses Session An order processing app reads inventory counts from Cosmos DB with Eventual consistency to reduce latency. Two concurrent reads on different replicas return stale counts. Two orders are confirmed for the last unit of inventory, causing an oversell. Strong or Bounded Staleness consistency was needed for this business logic. Business logic error from stale reads Med
6 Change feed consumer failure — message reprocessing Consumer loses offset without checkpointing A change feed processor crashes and restarts without persisting its lease checkpoint. It begins reprocessing all changes from the beginning of the container. Downstream systems that are not idempotent process the same events multiple times, causing duplicate orders, duplicate payments, or duplicate notifications. Duplicate processing from change feed replay Med
7 Multi-region write conflict not resolved LWW resolves conflict silently, losing business data Two regions simultaneously update the same document (e.g., an account balance). Last-Write-Wins (LWW) conflict resolution discards one of the writes silently based on timestamp. A fund transfer that was decremented in one region is overwritten by a stale value from another region, effectively reversing the debit. Silent data loss from write conflict Med
8 TTL deletes data too aggressively TTL set too short for operational data A container stores session tokens with a 24-hour TTL. A bug in the provisioning pipeline sets the TTL to 24 seconds instead of 86400. All session tokens expire within seconds of creation. All authenticated users are immediately logged out and cannot re-authenticate. Authentication failure for all users Med
9 Synapse Link analytical query impact OLAP query via Synapse Link affects OLTP throughput Despite Synapse Link being designed to isolate analytical queries from OLTP, a misconfigured Synapse Spark job issues extremely high-frequency small reads against the analytical store. The increased read pressure causes latency spikes on the Cosmos DB account, affecting real-time application performance. OLTP latency degradation from analytics Low
10 RBAC not used — resource token for mobile clients Resource tokens expose partition-level write access A mobile app is given a resource token that allows write access to a container partition. An attacker reverse-engineers the mobile app, extracts the resource token generation logic, and generates tokens for other users' partitions. The attacker can write or delete documents in victim users' partitions. Cross-user data tampering via token forgery Med

Mitigation Controls

# Failure Mitigation Layer
1 Master key in config Use managed identity with Cosmos DB RBAC (Data Reader/Contributor roles); disable key auth Identity-based auth
2 Hot partition Design partition key for even distribution (composite key, hash suffix for hot items) Data modeling
3 Cross-partition fan-out Always include partition key in query filter; use indexing policy to optimize frequently queried fields Query design
4 No private endpoint Configure private endpoint; disable public network access Network controls
5 Stale reads Choose consistency level appropriate for the business requirement; document consistency per operation Consistency design
6 Change feed offset loss Use Cosmos DB Change Feed Processor library with persistent leases; implement idempotency in consumers Consumer design
7 Multi-region write conflict Use custom conflict resolution procedure for financial data; prefer single-region write for strong consistency Conflict resolution
8 TTL misconfiguration Validate TTL values in CI/CD deployment checks; use IaC for all container settings Configuration governance
9 Synapse Link impact Monitor Cosmos DB latency metrics when Synapse jobs run; throttle analytical query concurrency Observability
10 Resource token forgery Use Cosmos DB RBAC with Entra ID for client auth instead of resource tokens Authentication

Key Takeaways

  • Cosmos DB is globally distributed NoSQL with turnkey multi-region distribution and five consistency levels.
  • Partition key design is the most critical architectural decision — hot partitions cause throttling and availability issues.
  • Use managed identity with Cosmos DB RBAC instead of account keys or resource tokens.
  • Change feed enables event-driven architectures; implement idempotent consumers with persistent checkpoints.
  • Synapse Link enables HTAP by separating analytical queries from OLTP on the same data without ETL.
