Azure Database for PostgreSQL — Flexible Server

Category: Database
Azure Service Type: Managed Open-Source Relational Database (PaaS)
Key Docs: Azure PostgreSQL Flexible Server

Overview

Azure Database for PostgreSQL Flexible Server provides a fully managed PostgreSQL database service with fine-grained control over database configuration, maintenance windows, and zone-redundant high availability. It supports the latest PostgreSQL community releases (up to 16+). Features include built-in HA with standby in the same or different AZ, point-in-time restore, read replicas, VNet integration, Entra ID authentication, pgAudit extension for detailed audit logging, and Microsoft Defender for open-source relational databases for threat detection.

How It Works

Core Concepts

Concept Description
Flexible Server Deployment model with full control over config, maintenance windows, and networking
Compute Tiers Burstable (B-series), General Purpose, Memory Optimized
Zone-Redundant HA Synchronous standby in different AZ; automatic failover in 60-120 seconds
Same-Zone HA Standby in same AZ; faster failover for zones that don't require AZ separation
Read Replica Asynchronous read-only replica; can be in same or different region
Point-in-Time Restore Restore to any point within the retention window (1-35 days)
pgAudit PostgreSQL audit extension logging DDL, DML, roles, and object access
Entra ID Authentication Log in with Entra ID tokens instead of PostgreSQL passwords
Connection Pooling PgBouncer built-in; reduces connection overhead for high-concurrency workloads
Private Access VNet-integrated deployment with no public endpoint

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Postgres admin password in connection string Password stored in app config or environment The PostgreSQL admin user password is stored as a plaintext environment variable in a Kubernetes secret. A developer with kubectl get secret access retrieves it. The attacker connects to the database with full admin rights, reads all tables, creates backdoor users, and dumps the entire database. Full database admin access High
2 Public network access enabled Server accessible from internet on port 5432 PostgreSQL Flexible Server is deployed with public access enabled. Port 5432 is reachable from the internet. An attacker runs a password spray attack against the default postgres superuser or discovered application users. A weak password is cracked in minutes, yielding a database shell. Brute-force attack on DB endpoint High
3 No pgAudit extension No DDL/DML audit trail An insider with DBA access dumps a customer table and truncates it to cover the track. Without pgAudit enabled, the only evidence in server logs are generic connection records. The security team cannot prove which statements were executed, which tables were accessed, or when the delete occurred. Undetectable insider data tampering Med
4 Misconfigured pg_hba.conf equivalent Overly permissive host-based authentication The Flexible Server firewall rules include an entry for 0.0.0.0/0 (all IP addresses) with password authentication. This allows any internet host to attempt authentication against the database, making it vulnerable to credential stuffing and brute force without rate limiting or lockout. Open attack surface for credential attacks High
5 Connection pool exhaustion Too many connections without PgBouncer An application deploys multiple replicas, each opening up to 100 PostgreSQL connections. PostgreSQL has a max_connections = 100 limit. At 2 replicas, the limit is reached. New connections are rejected with "too many clients already". The application returns 500 errors until connections are freed. Application unavailability from connection exhaustion High
6 Read replica lag — stale reads Async replication allows replica lag under load A reporting application reads from a read replica. A batch import job writes a large dataset to the primary. Replication lag grows to 5 minutes. The reporting query returns counts that exclude the new data, generating an incorrect business report that is sent to executives before the discrepancy is discovered. Incorrect business decisions from stale data Med
7 Point-in-time restore RTO misunderstood PITR creates a new server, not an in-place restore An operator assumes that point-in-time restore replaces the existing server in-place. In reality, PITR creates a new server. The old server remains. The application connection string must be manually updated to point to the new server. Without planning for this, recovery time is extended by the reconfiguration effort. Extended RTO from operational misunderstanding Med
8 Long-running query blocks vacuum Idle transaction holds lock preventing autovacuum An application opens a transaction, runs a query, and then stalls (network issue, client crash) without committing or rolling back. PostgreSQL cannot vacuum dead tuples while the transaction holds a lock. Table bloat grows, query performance degrades, and eventually the txid_wraparound risk is reached, requiring emergency maintenance. Performance degradation and data availability risk Med
9 Extensions with security implications pg_stat_statements or dblink exposing internal data The dblink extension is installed and grants any database user the ability to connect to other PostgreSQL instances from within a query. A user with limited permissions uses dblink to connect to another internal PostgreSQL server using credentials they discovered in a readable config table, bypassing the network-level isolation. Lateral movement via database extension Med
10 Backup retention too short Default 7-day retention insufficient A compliance requirement mandates 30-day PITR capability. The database is configured with the default 7-day backup retention. A data corruption event is discovered 10 days after it occurred. The PITR window does not extend back far enough to restore to a clean state. Data recovery impossible beyond retention window Med

Mitigation Controls

# Failure Mitigation Layer
1 Password in config Use Entra ID authentication with managed identity; store admin password in Key Vault Authentication
2 Public access Deploy in private access mode (VNet integration); disable public endpoint Network controls
3 No pgAudit Enable pgAudit extension; ship logs to Log Analytics workspace Audit logging
4 Permissive firewall Use private access (VNet); if public, restrict firewall to known app IPs only Firewall rules
5 Connection exhaustion Enable built-in PgBouncer; set max_connections appropriately; use connection pooling in app Connection management
6 Stale replica reads Monitor replication lag metric; set alert threshold; add lag check before reporting queries Observability + app design
7 PITR RTO misunderstood Document PITR process; automate connection string update in DR runbooks DR runbook
8 Long-running idle transactions Set idle_in_transaction_session_timeout; monitor and alert on long-running transactions PostgreSQL config
9 Dangerous extensions Audit installed extensions; restrict extension installation to admin role; avoid dblink in production Extension governance
10 Short backup retention Configure backup retention to match compliance requirements (up to 35 days) Backup policy

Key Takeaways

  • Azure PostgreSQL Flexible Server provides managed PostgreSQL with zone-redundant HA and full config control.
  • Use private access (VNet integration) and Entra ID authentication — eliminate passwords and public endpoints.
  • Enable pgAudit for detailed DDL/DML audit logging and ship to Log Analytics for SIEM integration.
  • Configure built-in PgBouncer connection pooling for high-concurrency workloads to prevent connection exhaustion.
  • Set backup retention to meet compliance requirements; understand that PITR creates a new server instance.
