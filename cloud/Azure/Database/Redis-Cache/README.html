Azure Cache for Redis

Category: Database
Azure Service Type: In-Memory Cache and Data Store
Key Docs: Azure Cache for Redis Overview

Overview

Azure Cache for Redis provides a managed, in-memory data store based on Redis, enabling sub-millisecond latency caching, session storage, real-time leaderboards, pub/sub messaging, and distributed locks. Azure manages high availability, patching, and scaling. Tiers include Basic (single node, no HA), Standard (primary + replica, automatic failover), Premium (VNet, geo-replication, persistence, Redis Cluster), and Enterprise (Redis Enterprise engine with advanced features like Active-Active geo-distribution and RediSearch). Azure Cache for Redis Enterprise is the enterprise-grade offering with 99.99% SLA.

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Access key leakage Redis connection string stored in plaintext config A Redis connection string including the access key is stored in an App Service app setting (not a Key Vault reference). A principal with Microsoft.Web/sites/config/read access retrieves the key. With the key, they connect to Redis and read all cached session tokens, user data, and JWT tokens, enabling session hijacking at scale. Session token theft for all active users High
2 Unauthenticated Redis exposure Redis port 6379 accessible without TLS or auth A Redis instance is deployed with enableNonSslPort: true and without requirepass. An attacker who can reach the endpoint (e.g., on the same VNet due to flat networking) sends FLUSHALL and wipes all cached data, or reads all keys containing session tokens and user data. Complete cache data loss or mass session theft High
3 KEYS command causing latency spike KEYS * pattern scan on large cache An application or debug script runs KEYS user:* to enumerate cached user sessions. Redis executes KEYS as a blocking O(N) operation on the entire keyspace. A cache with 1 million keys stalls the Redis server for hundreds of milliseconds, during which all other clients experience timeouts. The application returns 500 errors while Redis processes the scan. Service-wide latency spike from blocking command High
4 No eviction policy — memory exhaustion MaxMemory reached without eviction configured A cache grows to maxmemory with maxmemory-policy noeviction. New write operations fail with OOM command not allowed. The application receives errors when attempting to store session tokens or cache database results, causing user session failures and increased database load. Cache write failure and application degradation High
5 Cache stampede — thundering herd Many concurrent cache misses hit DB simultaneously A hot key (e.g., the homepage product list) expires from cache. Simultaneously, 500 concurrent user requests all get a cache miss and send queries to the database. The database is overwhelmed with 500 identical queries. Response times spike to 30 seconds. Some database queries time out, causing errors for a portion of users. Database overload on cache expiry High
6 Session data stored with no TTL Sessions persist indefinitely in cache User session data is written to Redis without an expiry time. Over months, millions of stale session entries accumulate. The cache fills with data from users who have not been active in months. When maxmemory is reached, live session data is evicted while stale sessions remain (LRU policy may still evict recently written data before old data is properly aged). Memory waste and stale session accumulation Med
7 Geo-replication failover not tested Secondary region Redis not promoted in DR test A Redis instance has geo-replication configured. The DR plan states the secondary will be promoted on primary failure. The failover has never been tested. During a regional incident, the team discovers that promoting the secondary is a manual process requiring CLI commands, and the application connection string must be manually updated. RTO is exceeded. RTO breach from untested DR procedure Med
8 Sensitive data cached in plaintext PII or tokens stored without additional encryption A session cache stores full user profile objects including email addresses, phone numbers, and partial payment information in plaintext JSON values. An attacker who reads Redis keys (via key leakage or network sniffing) accesses PII for millions of users in a single SCAN operation. PII exposure via cache data Med
9 No Redis Cluster — single shard size limit Premium single shard limited to 120 GB A rapidly growing workload approaches the 120 GB single-shard limit for Premium Redis. The team has not enabled Redis Cluster (3 shards minimum). When the limit is reached, writes fail. Enabling Cluster mode requires redeployment of the cache instance, causing downtime and data loss during migration. Capacity-limited cache failure Med
10 Pub/sub message loss Redis pub/sub has no message persistence An event-driven application uses Redis pub/sub to distribute job notifications. If a subscriber is offline when a message is published, the message is lost — Redis pub/sub does not buffer messages for offline subscribers. The application processes no events for the offline period and does not detect the gap. Silent message loss for offline consumers Med

Mitigation Controls

# Failure Mitigation Layer
1 Key leakage Store Redis connection string in Key Vault; rotate keys regularly; use private endpoint to limit blast radius Secrets management
2 Unauthenticated exposure Disable non-SSL port; require TLS 1.2 minimum; place behind private endpoint or VNet Network + auth
3 KEYS command latency Use SCAN with cursor instead of KEYS; disable KEYS command via Redis command rename Command safety
4 No eviction policy Set maxmemory-policy allkeys-lru or volatile-lru; set maxmemory to 80% of available RAM Memory management
5 Cache stampede Use probabilistic early expiration; implement distributed lock (SETNX) for cache refresh Application pattern
6 No TTL on sessions Always set TTL on every SET operation; enforce TTL policy in cache wrapper library Application pattern
7 Geo-replication not tested Run DR failover drills quarterly; automate connection string update via Azure DNS or Traffic Manager DR testing
8 PII in cache Encrypt sensitive fields before caching; avoid caching full PII objects; use tokens as cache keys Data design
9 No cluster planning Enable Redis Cluster for large caches; plan cluster shard count based on data volume projections Capacity planning
10 Pub/sub message loss Use Azure Service Bus or Event Hubs for durable messaging; use Redis Streams for persistent pub/sub Architecture choice

Key Takeaways

  • Azure Cache for Redis provides sub-millisecond in-memory caching; use Standard+ tier for HA.
  • Always use TLS and disable non-SSL port; store access keys in Key Vault and use private endpoint.
  • Use SCAN instead of KEYS; set eviction policy and TTL on all cache entries.
  • Implement cache stampede protection for high-traffic, time-critical keys.
  • Redis pub/sub has no durability — use Service Bus or Redis Streams for reliable messaging.
