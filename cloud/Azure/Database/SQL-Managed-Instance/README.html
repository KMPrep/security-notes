Azure SQL Managed Instance

Category: Database
Azure Service Type: Managed SQL Server (Near-Full Compatibility PaaS)
Key Docs: SQL Managed Instance Overview

Overview

Azure SQL Managed Instance provides near-complete SQL Server compatibility in a fully managed PaaS deployment. Unlike Azure SQL Database, Managed Instance supports SQL Agent, cross-database queries, linked servers, Common Language Runtime (CLR), Service Broker, and other features that are unavailable in the single-database model. It is the primary lift-and-shift target for SQL Server workloads migrating to Azure. SQL MI is always deployed inside a VNet (no public endpoint by default), uses a dedicated subnet, and offers Business Critical and General Purpose service tiers.

How It Works

Core Concepts

Concept Description
VNet Deployment SQL MI is always deployed in a dedicated subnet within a VNet — no public endpoint by default
Business Critical Tier Local SSD storage, 3 additional readable HA replicas, higher resilience
General Purpose Tier Remote storage (Azure Premium), suitable for most workloads
SQL Agent Native SQL Server Agent for scheduling and automation jobs
Link Feature Near real-time replication between SQL MI and on-premises SQL Server for hybrid or migration
Failover Group Automatic failover to a secondary MI in another region
Distributed Transaction Cross-database and cross-instance MSDTC transactions
Managed Instance Pool Shared resource pool for multiple small SQL MIs
Maintenance Window Control scheduled maintenance to defined windows
Instance-Level Logins Supports Windows logins (Entra ID), SQL logins, and contained database users

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 SQL Agent job running malicious commands Insider with SQL Agent operator access creates malicious job A DBA with SQL Agent access creates an operating system job step that runs xp_cmdshell or executes a PowerShell script to exfiltrate database contents to an external location. SQL MI supports CLR and Agent — more attack surface than Azure SQL Database. Because this is an insider action via a legitimate account, it may not trigger any alert. Insider data exfiltration via SQL Agent Med
2 NSG misconfiguration breaks management Required management ports blocked SQL Managed Instance requires specific NSG rules allowing inbound TCP 9000, 9003, 1438, 1440, 1452 and outbound 80, 443, 12000 for Microsoft management traffic. An administrator applies a restrictive NSG that blocks these ports. SQL MI enters an unhealthy state, management operations fail, and the instance may stop accepting connections. Full SQL MI unavailability High
3 Route table misconfiguration User-defined routes break management traffic A UDR routes all traffic (0.0.0.0/0) through a network virtual appliance. Microsoft management traffic that must reach Azure infrastructure cannot route correctly through the NVA. SQL MI management operations fail, potentially causing the instance to become unresponsive. MI management failure and outage High
4 Linked server credentials exposed Linked server to on-premises SQL uses stored password A linked server connection to an on-premises SQL Server is configured with a stored password in the linked server security definition. Any sysadmin on the MI can read the stored credentials. A compromised DBA account retrieves the linked server password and connects to the on-premises SQL Server directly. Lateral movement to on-premises systems Med
5 Cross-database query bypasses row-level security Shared instance, multiple databases, no isolation Multiple customers' databases are hosted on the same Managed Instance. A developer with access to Database A writes a cross-database query (SELECT * FROM [DatabaseB].dbo.customers) and the SQL login has CONNECT permission to DatabaseB. Cross-database queries bypass database-level isolation expectations in a multi-tenant model. Cross-tenant data access High
6 Maintenance window causing unexpected downtime Scheduled maintenance hits during business hours SQL MI performs a planned maintenance event (patching, failover) during peak business hours because no maintenance window is configured. The failover causes a 20-30 second connection interruption. Applications without retry logic fail hard, causing user-visible errors. Availability interruption during peak hours Med
7 MI subnet CIDR too small /28 subnet fills with MI management IPs A SQL Managed Instance is deployed in a /28 subnet (14 usable IPs). SQL MI requires a large number of IPs for management (at minimum 32 IPs for a single MI). The deployment fails or the subnet exhausts IPs, preventing new deployments or management operations. Deployment failure from insufficient subnet size Med
8 No failover group — manual DR Business Critical MI without regional DR A Business Critical MI provides local HA via readable replicas, but there is no cross-region failover group. A full regional Azure outage makes the MI unavailable. Recovery requires restoring from geo-redundant backup in a secondary region — a process that can take hours for a large database. RTO breach during regional outage High
9 Excessive sysadmin grants Too many accounts with sysadmin role Multiple developers and DBAs have sysadmin on the MI. A sysadmin can do anything: read any database, create logins, modify Agent jobs, execute OS commands via CLR or xp_cmdshell. The blast radius of any compromised sysadmin account is complete MI takeover. Full MI compromise from any sysadmin account High
10 TDE key in service-managed key No customer-managed key for compliance Managed Instance uses service-managed TDE keys by default. A compliance requirement mandates BYOK with proof of key rotation and custody. Service-managed keys provide no customer visibility into key management. Compliance failure for regulated workloads Med

Mitigation Controls

# Failure Mitigation Layer
1 Malicious SQL Agent job Restrict SQL Agent operator role; audit all Agent job definitions; enable Advanced Threat Protection Privilege management
2 NSG blocks management Use Microsoft-provided NSG service tag SqlManagement; validate NSG before applying changes NSG configuration
3 UDR breaks management Use the MI-specific route table guidance; set next hop for MI management prefixes to Internet, not NVA Route table design
4 Linked server credentials Use Windows/Entra ID authentication for linked servers; avoid stored passwords Linked server auth
5 Cross-database query bypass Isolate tenants on separate MIs; use contained databases; apply explicit DENY on cross-DB permissions Database isolation
6 Unexpected maintenance window Configure a maintenance window aligned to non-peak hours; implement application retry logic Maintenance configuration
7 Subnet too small Use /27 or larger subnet for MI; plan for 2x current IPs to allow growth Network planning
8 No failover group Configure auto-failover group to secondary MI in DR region HA and DR
9 Excessive sysadmin Follow least-privilege; use dedicated service accounts per application; audit sysadmin membership RBAC
10 No CMK for TDE Configure TDE with customer-managed key in Key Vault; enable Key Vault purge protection Encryption

Key Takeaways

  • SQL Managed Instance provides near-complete SQL Server compatibility — SQL Agent, CLR, cross-DB queries, linked servers.
  • MI is always VNet-deployed; NSG and UDR configuration is critical — management port mistakes cause outages.
  • Use failover groups for cross-region DR; Business Critical tier provides local HA with readable replicas.
  • Minimize sysadmin grants; audit SQL Agent jobs; use Entra ID authentication where possible.
  • Plan subnet size carefully — SQL MI requires at least a /27 subnet with dedicated IPs for management.
