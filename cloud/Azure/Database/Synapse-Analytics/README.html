Azure Synapse Analytics

Category: Database / Analytics
Azure Service Type: Unified Analytics Platform
Key Docs: Synapse Analytics Overview

Overview

Azure Synapse Analytics is a unified analytics platform that integrates enterprise data warehousing, big data processing, and data integration. It combines a dedicated SQL pool (formerly Azure SQL Data Warehouse) for high-performance OLAP queries, a serverless SQL pool for on-demand querying of data lakes, an Apache Spark pool for large-scale data engineering and ML, and data integration pipelines (Azure Data Factory engine). Synapse Studio provides a single pane for development, monitoring, and management. It integrates deeply with ADLS Gen2, Cosmos DB Synapse Link, and Power BI.

How It Works

Core Concepts

Concept Description
Dedicated SQL Pool Massively parallel processing (MPP) engine; data distributed across 60 distributions
Serverless SQL Pool Pay-per-query SQL engine; reads ADLS Gen2 directly without loading data
Apache Spark Pool Managed Spark cluster; scales to zero; supports Python, Scala, SQL, .NET
Synapse Pipeline ADF-based data orchestration; copy, transform, and load pipelines
Synapse Link Direct analytical access to Cosmos DB without ETL
Managed Private Endpoints Private connectivity from Synapse to ADLS, SQL, Key Vault within the managed VNet
Workspace Managed Identity Synapse workspace identity for accessing storage and other resources
Linked Services Connection definitions for external data sources
SQL Login vs Entra ID Dedicated SQL pool supports both SQL authentication and Entra ID
Data Exfiltration Protection Managed VNet mode restricts outbound traffic to approved targets only

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Overpermissive workspace managed identity Workspace identity has Storage Blob Data Contributor on entire data lake A Synapse workspace managed identity is assigned Storage Blob Data Contributor on the entire ADLS Gen2 account. A developer with Synapse Contributor can run a pipeline that reads and writes to any path in the data lake, bypassing intended access boundaries between raw, curated, and serving zones. Data zone access bypass High
2 No data exfiltration protection Managed VNet not enabled; pipelines can reach external URLs A Synapse pipeline is compromised via a malicious linked service or injected code in a mapping data flow. Without data exfiltration protection (managed VNet), the pipeline can make HTTP calls to external attacker-controlled URLs, exfiltrating data by encoding it in query parameters or request bodies. Data exfiltration via pipeline Med
3 Dedicated SQL pool always running — cost bleed DWU billed when pool is not paused A dedicated SQL pool (2000 DWU) runs 24/7 during development and testing, even when not in use. At $0.96/DWU/hour, an idle 2000 DWU pool costs over $13,000/month. The team is unaware that dedicated pools must be explicitly paused and resumed. Unnecessary cost of idle compute High
4 SQL admin password used for dedicated pool SQL logins for all Synapse users All analysts log into the dedicated SQL pool with shared SQL logins. Passwords are shared over email. There is no MFA, no audit trail per-user, and no way to revoke access without changing the shared password. An ex-employee retains access because the password has not been rotated. Unauthorized access after employee departure High
5 Serverless SQL query scans full data lake OPENROWSET with no WHERE clause on petabyte-scale data An analyst writes SELECT * FROM OPENROWSET(BULK 'abfss://container@account.dfs.core.windows.net/**', FORMAT='PARQUET') with no filters. This query scans every Parquet file in the data lake. The data billed per TB queried for serverless SQL results in a multi-thousand-dollar query bill for a single accidental scan. Unexpected high cost from full scan Med
6 Spark pool auto-pause not configured Spark cluster idles after job completion A Spark pool is not configured with auto-pause. After a Spark job completes, the cluster idles at full capacity. If no other jobs are submitted for hours, the cluster incurs idle billing until someone manually stops it. For large clusters (64 vCores), idle billing can be thousands of dollars per day. Idle Spark cluster cost waste Med
7 Unrestricted column access in dedicated pool No row/column-level security A Synapse dedicated SQL pool contains a customer table with SSN and salary columns. No column-level security or dynamic data masking is applied. Any analyst with SELECT permission on the table can read all columns including sensitive fields. An analyst exports a query result with SSN to a personal laptop. PII exposure via unrestricted column access Med
8 Pipeline credential in plaintext linked service JDBC connection string with password in linked service A Synapse pipeline linked service to an on-premises database contains a connection string with a hardcoded password. Any Synapse contributor can view the linked service definition. The password is also exposed in pipeline activity run history logs. Credential exposure via pipeline config Med
9 Dedicated pool distribution key — data skew Poor distribution key causes hotspot A dedicated SQL pool table uses ROUND_ROBIN distribution. A frequently joined dimension table is replicated. A fact table uses HASH(order_date) where most orders are on weekdays. 5 of 60 distributions hold 80% of the data. Queries against this table are bottlenecked at the hot distributions. Query performance degradation from data skew Med
10 No workspace audit logging Synapse activity not shipped to SIEM User queries, pipeline executions, and data access events are not forwarded to a Log Analytics workspace. When a data breach investigation requires knowing which queries accessed a specific table in the past 30 days, there are no logs to analyze. Forensic gap on data access investigation Med

Mitigation Controls

# Failure Mitigation Layer
1 Overpermissive workspace identity Scope workspace managed identity RBAC to specific containers; use Synapse RBAC roles for user access RBAC design
2 No data exfiltration protection Enable managed VNet for the Synapse workspace; use managed private endpoints for all data sources Network controls
3 Idle dedicated pool cost Pause dedicated SQL pool during off-hours via Automation or Synapse schedule Cost governance
4 SQL auth only Enable Entra ID authentication; assign Synapse RBAC roles; disable SQL admin where possible Authentication
5 Full lake scan Enforce query cost controls via serverless SQL access controls; use views with predicate pushdown Query governance
6 Spark cluster idle Enable auto-pause on Spark pools (minimum 5 minutes); use serverless Spark for ad-hoc queries Cost governance
7 Unrestricted column access Apply dynamic data masking for sensitive columns; use column-level GRANT for analysts Column security
8 Credentials in linked services Use managed private endpoints with managed identity; store credentials in Key Vault linked service Secrets management
9 Data skew Design HASH distribution keys on high-cardinality join columns; use REPLICATE for small dimension tables Query optimization
10 No audit logging Enable Synapse workspace diagnostic settings to Log Analytics; use Microsoft Sentinel for SIEM Observability

Key Takeaways

  • Synapse Analytics unifies dedicated SQL (DW), serverless SQL (lake query), Spark, and pipelines in one platform.
  • Pause dedicated SQL pools when not in use — idle DWU billing is the most common cost waste.
  • Enable managed VNet and managed private endpoints to prevent data exfiltration via pipelines.
  • Use Entra ID authentication and Synapse RBAC; apply dynamic data masking for sensitive columns.
  • Distribution key design is critical for dedicated SQL pool performance — avoid data skew.
