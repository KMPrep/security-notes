Azure Resource Manager (ARM) and Bicep

Category: DevOps
Azure Service Type: Infrastructure as Code
Key Docs: Bicep Overview

Overview

Azure Resource Manager (ARM) is the deployment and management layer for all Azure resources. Every Azure API call — whether from the portal, CLI, SDK, or IaC — goes through ARM, which enforces RBAC, policies, and locks. ARM templates (JSON) and Bicep (a DSL that compiles to ARM JSON) are declarative IaC languages for defining and deploying Azure resource configurations. Bicep offers cleaner syntax, type safety, modularity, and better error messages than raw ARM JSON. Terraform (HashiCorp) is an alternative popular in multi-cloud environments, also deploying via the ARM API. IaC enables consistent, auditable, repeatable infrastructure with version-controlled configuration.

How It Works

Core Concepts

Concept Description
ARM Control plane for Azure; all resource CRUD operations go through ARM
ARM Template JSON declaration of resources and their properties
Bicep Strongly-typed DSL compiling to ARM JSON; modules, params, vars, outputs
Deployment Mode Complete (remove resources not in template) vs Incremental (add/update only)
Deployment Stack Manage a group of resources as a unit; enforce deny-assignments to prevent drift
Resource Group Logical container; deployment scope; lifecycle boundary
Management Group Hierarchy above subscriptions for policy and RBAC scope
What-If Preview changes a deployment will make before execution
Parameter File .bicepparam or parameters.json — externalize configurable values
Module Reusable Bicep file representing a resource pattern (e.g., vnet.bicep)

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Secrets in ARM/Bicep templates committed to git Parameters with sensitive values checked in A developer parameterizes a database admin password in a .bicepparam file and commits it to the repository. The file is in a private repo, but the company has an internal developer portal that mirrors repos. The password is in git history. Even after the file is removed, the secret persists in commits accessible to any internal developer. Credential exposure in source code history High
2 Complete deployment mode deletes unmanaged resources Mode not understood — deletes extra resources A team runs az deployment group create --mode Complete to a resource group. The template does not include a Key Vault and storage account that were manually created for debugging. Complete mode deletes all resources in the resource group not present in the template. The Key Vault and storage account are permanently deleted along with their data. Accidental data deletion via Complete mode High
3 No What-If before deployment — surprises in production Deploying without preview A Bicep change accidentally references a wrong subnet ID (copy-paste error). Without running what-if first, the deployment runs and modifies a load balancer to use the wrong subnet, disrupting traffic. The change takes 20 minutes to identify and roll back. Running what-if would have shown the subnet change before execution. Production disruption from unreviewed change Med
4 Overprivileged deployment identity Pipeline SP has Owner role on subscription The service principal used by the IaC pipeline has Owner role on the subscription. An attacker who compromises the pipeline (via supply chain attack on a pipeline template) can deploy arbitrary resources, modify RBAC assignments, and create new admin users using the Owner-level SP. Full subscription control from compromised pipeline identity High
5 Hardcoded resource IDs in templates Non-parameterized cross-resource references A Bicep template hardcodes the Key Vault resource ID from the production subscription. When the template is used to deploy a staging environment, the staging resources are linked to the production Key Vault. Staging applications read and potentially write production secrets. The environment boundary collapses. Cross-environment resource access from hardcoded references Med
6 No resource locks on critical infrastructure Production resources deletable via IaC or portal A production Azure SQL server, VNet, and Key Vault have no resource locks. A developer runs an outdated version of the IaC template (with different resource names). In Complete mode, the current resources are deleted. Resource locks (CanNotDelete) would have prevented the deletion even with Owner-level credentials. Critical resource deletion without locks High
7 Bicep modules from unverified external registries Using community modules without audit A Bicep template references a module from a public Bicep registry maintained by a community contributor. The module is updated to add a backdoor that grants Owner role to an attacker-controlled service principal. The next IaC deployment installs the backdoor. No version pinning or module audit exists. Supply chain attack via compromised Bicep module Med
8 Deployment drift — manual changes bypassing IaC Ad-hoc portal changes not in template A developer manually opens a storage account's public access setting via the portal during an incident. The IaC template has publicNetworkAccess: disabled. The next IaC deployment reverts the portal change (which is correct), but the drift window creates a security gap. More dangerously, if the IaC template is never re-applied, the drift persists. Security control drift from manual changes High
9 No Azure Policy blocking non-compliant deployments IaC deploys insecure configurations IaC templates deploy storage accounts with publicNetworkAccess: enabled to speed up development. No Azure Policy exists to block or audit this configuration. The insecure configuration is deployed to production because the policy gate does not exist. Data in the storage account is publicly readable. Insecure configuration deployed without policy gate High
10 Parameter file with environment-specific secrets in same repo Prod params alongside code Production parameter files (containing subscription IDs, resource names, and sometimes credentials) are stored in the same application repository as code. All developers with repo access can see production resource identifiers, subscription IDs, and any secrets that were mistakenly parameterized. Production topology disclosed to all developers Med

Mitigation Controls

# Failure Mitigation Layer
1 Secrets in param files Use @secure() decorator; link params to Key Vault references; never commit secret values Secret handling
2 Complete mode deletes Use Incremental mode by default; use Complete only explicitly with what-if review; add resource locks Deployment mode
3 No What-If Require what-if output as PR comment via pipeline; block merge until what-if passes review Change preview
4 Overprivileged SP Use Workload Identity Federation; grant Contributor on resource group scope, not Owner on subscription Least privilege
5 Hardcoded IDs Use Bicep existing resource references; parameterize all cross-environment references Parameterization
6 No resource locks Add CanNotDelete lock as part of IaC for production resource groups; enforce via Azure Policy Resource locks
7 Unverified modules Pin module references to specific version or digest; use private Bicep registry; audit modules before use Module supply chain
8 Manual drift Enable Azure Policy with DeployIfNotExists remediation; use Deployment Stacks with deny-assignments Drift prevention
9 No policy gate Deploy Azure Policy assignments for critical controls; deny non-compliant IaC via ARM deny effect Policy enforcement
10 Prod params in repo Store production parameter files in a separate secured repository or Key Vault Parameter isolation

Top 3 Critical Mitigations

  1. Never Commit Secrets — Use Key Vault References: Bicep's @secure() decorator and ARM KeyVault references allow parameters to be fetched from Key Vault at deployment time — never stored in code or parameter files.
  2. Resource Locks on Production Resources: CanNotDelete locks prevent accidental or malicious deletion even from Owner-level credentials. This is the last line of defense against IaC mistakes and rogue deployments.
  3. Azure Policy as the IaC Compliance Gate: Azure Policy with deny effects blocks non-compliant deployments at the ARM layer, regardless of how the deployment is triggered. This is environment-agnostic enforcement that works even for manual portal changes.

Key Takeaways

  • Bicep is the modern IaC choice for Azure-native deployments; it compiles to ARM JSON and has full feature parity.
  • Never use Complete deployment mode without what-if review — it deletes resources not in the template.
  • Use Key Vault references for all secrets in deployment parameters — never commit sensitive values to git.
  • Resource locks (CanNotDelete) on production resources prevent accidental deletion from IaC mistakes or human error.
  • Azure Policy with deny effects is the policy-as-code enforcement layer that complements IaC — policy blocks non-compliant deployments regardless of source.
