Azure Pipelines

Category: DevOps
Azure Service Type: CI/CD Platform
Key Docs: Azure Pipelines Overview

Overview

Azure Pipelines is a fully managed continuous integration and continuous delivery (CI/CD) service within Azure DevOps. It supports building, testing, and deploying code to any cloud, on-premises environment, or container platform. Pipelines are defined in YAML (azure-pipelines.yml) and support multi-stage deployments with approvals, gates, and environment protections. Microsoft-hosted agents provide ephemeral build environments (Windows, Ubuntu, macOS), while self-hosted agents offer custom tooling and VNet connectivity. Service connections securely link pipelines to Azure subscriptions, GitHub repositories, container registries, and third-party services using managed identities or service principals.

How It Works

Core Concepts

Concept Description
Pipeline YAML-defined CI/CD workflow
Stage Major phase (Build, Test, Deploy-Staging, Deploy-Prod)
Job Unit of work running on an agent; jobs in a stage can run in parallel
Step Individual task or script within a job
Agent Build runner — Microsoft-hosted (ephemeral) or self-hosted (persistent)
Service Connection Authenticated link to external service (Azure subscription, GitHub, ACR, etc.)
Environment Deployment target with history, approvals, and checks
Approval Gate Manual or automated check before a stage deploys to an environment
Variable Group Centralized set of variables and secrets linked from Azure Key Vault
Artifact Build outputs published between stages

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Service connection with Owner role Overprivileged pipeline identity A service connection is created with Owner role on the production subscription for convenience. A pull request from a fork adds a pipeline step that extracts the service connection token and exfiltrates it to an external URL. The attacker uses the token to gain Owner access to the production Azure subscription. Full subscription access from pipeline token theft High
2 Pull request triggers on forked repos PR from fork can access pipeline secrets A pipeline is configured to trigger on pull requests, including from public forks. The fork PR pipeline run has access to pipeline variables (including those marked as secret). A malicious contributor submits a PR with code that prints or exfiltrates the pipeline secret variables. Pipeline secrets exfiltrated via malicious PR High
3 Self-hosted agent runs as privileged user Agent process has admin/root access A self-hosted build agent runs as a local administrator on the host VM. A pipeline job executing malicious code (injected via a compromised dependency) has administrator access to the build host. The attacker installs persistence, accesses cached service connection tokens in the agent's work folder, and pivots to the Azure environment. Build agent compromise leading to Azure pivot High
4 No approval gates on production deployment Automated deploy to prod on every merge Every merge to the main branch automatically deploys to production without any approval gate or change management process. A developer accidentally merges a broken feature branch. The broken code deploys to production immediately, causing an outage. No human review occurs between merge and production deployment. Unreviewed code deployed to production High
5 Plaintext secrets in pipeline YAML Variables hardcoded in YAML A developer hardcodes a database password in the YAML pipeline file as an environment variable: DB_PASSWORD: "Pa$$w0rd". The YAML file is in the repository and accessible to all team members with repo access. The secret is visible in pipeline run logs. It is also in git history permanently, even after it is removed. Credential exposure in source code and logs High
6 No dependency vulnerability scanning Build pulls unscanned third-party packages The build pipeline runs npm install or pip install without scanning for known vulnerabilities. A transitive dependency introduces a critical vulnerability (e.g., Log4Shell equivalent). The vulnerable dependency is packaged into the production container image and deployed. No pipeline stage scans dependencies or container images for CVEs. Vulnerable dependencies shipped to production High
7 Retention of old pipeline artifacts Build artifacts stored indefinitely Build artifacts (compiled binaries, Docker layers, test reports) are retained indefinitely. After 18 months, hundreds of gigabytes of artifacts accumulate. Old artifacts for decommissioned services contain credentials that were valid at build time (now rotated but still in old artifact logs). An attacker with artifact download access reviews old build logs and finds credential patterns. Old credential patterns in retained artifacts Med
8 Pipeline-as-Code in external repo without review External YAML loaded without audit A pipeline template is loaded from an external public GitHub repository using template@external-repo. The external repo is compromised and the template is modified to exfiltrate tokens. The pipeline uses the compromised external template on the next build run and all pipeline secrets are exfiltrated. Supply chain attack via external pipeline template High
9 Environment without deployment protection All environments allow unrestricted access Pipeline environments (Staging, Production) have no required reviewers, branch checks, or deployment gates. Any developer can deploy directly to production by editing the pipeline YAML to skip stages. A developer deploys an experimental feature directly to production from a feature branch without review. Unauthorized production deployments Med
10 Agent pool shared across projects Isolation boundary between projects broken A single self-hosted agent pool is shared across multiple Azure DevOps projects. Project A has lower security requirements; Project B deploys to a regulated production environment. A malicious pipeline job in Project A leaves a persistent script on the shared agent. The next Project B pipeline job runs on the same agent and the script executes with access to Project B's service connection tokens. Cross-project secret theft via shared agent Med

Mitigation Controls

# Failure Mitigation Layer
1 Overprivileged service connection Use Workload Identity Federation for service connections; grant Contributor on resource group, not Owner on subscription RBAC scoping
2 Fork PR secrets access Disable secrets for fork PR builds; require manual comment trigger for external contributors Fork protection
3 Privileged agent user Run self-hosted agents as a low-privilege dedicated service account; use ephemeral agents where possible Agent hardening
4 No approval gates Add required reviewers on production environments; enforce branch policies (PR required for main) Deployment gates
5 Plaintext secrets Use Variable Groups linked to Azure Key Vault; mark variables as secret; never hardcode in YAML Secret management
6 No dependency scanning Add OWASP Dependency Check or Microsoft Defender for DevOps scanning step; scan container images with Trivy or Defender Supply chain security
7 Indefinite artifact retention Set retention policies (30 days for feature builds, 90 days for release builds); purge old runs Retention policy
8 External template without review Pin external templates to commit hash; review and fork templates used in production pipelines Template security
9 No environment protection Configure required reviewers and branch checks on all environments; lock production to main branch only Environment protection
10 Shared agent pool Use dedicated agent pools per project or security tier; enable agent isolation Agent isolation

Top 3 Critical Mitigations

  1. Workload Identity Federation for Service Connections: Traditional service connection credentials are static tokens that can be stolen. Workload Identity Federation uses short-lived OIDC tokens exchanged at runtime — no static secret to steal.
  2. Fork PR Build Secret Isolation: Public repository fork PRs should never have access to pipeline secrets. Disable secret variables for fork builds and require manual approval before trusted contributor pipelines run.
  3. Environment Required Reviewers + Branch Policy: Production deployments must require human approval. Combine environment required reviewers with branch protection (merge only via PR) to enforce a complete change control chain.

Key Takeaways

  • Use Workload Identity Federation for service connections — eliminates static credentials that can be stolen from build logs.
  • Fork PR builds should never access secrets — disable secret variable inheritance for fork-triggered builds.
  • Production environments require required reviewers and branch checks — automated deploy-on-merge to prod is a critical risk.
  • Scan dependencies and container images in the pipeline — supply chain attacks are the fastest-growing CI/CD threat vector.
  • Dedicated agent pools per security tier prevent cross-project secret theft via shared agent persistence.
