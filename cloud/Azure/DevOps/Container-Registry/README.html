Azure Container Registry

Category: DevOps
Azure Service Type: Private Container Registry
Key Docs: Container Registry Overview

Overview

Azure Container Registry (ACR) is a managed, private Docker and OCI container image registry built on Docker Registry v2. It stores and distributes container images, Helm charts, and OCI artifacts to Azure compute services (AKS, App Service, ACI, Azure Functions). ACR supports geo-replication (multi-region image replication), automated image build tasks (ACR Tasks), and vulnerability scanning via Microsoft Defender for Containers. Tiers: Basic (development), Standard (production), Premium (geo-replication, private endpoint, customer-managed keys, retention policies). ACR integrates with Entra ID for RBAC-based access — eliminating the need for static admin credentials.

How It Works

Core Concepts

Concept Description
Repository Named collection of related container images (e.g., myapp/api)
Tag Mutable label pointing to an image digest (e.g., latest, v1.2.3)
Digest Immutable SHA256 hash of the image manifest — the true image identifier
Geo-replication Replicate registry to multiple Azure regions; clients pull from nearest replica
ACR Tasks Automated build and test pipelines triggered by code commit or base image update
Webhooks Notifications on push or delete events (trigger AKS updates, CI pipelines)
Private Endpoint Restrict registry access to VNet private IP; disable public endpoint
Admin Account Built-in legacy admin user with static password — disabled by default in new registries
Token Repository-scoped token for granular access (CI/CD token with push to one repo only)
Defender for Containers Vulnerability scanning for pushed images; alerts in Defender for Cloud

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Admin account enabled — static credentials Legacy admin account used for CI/CD The ACR admin account is enabled for convenience to provide CI/CD credentials. The admin username and one of two passwords are embedded in the GitHub Actions workflow as secrets. A developer with access to GitHub repository settings copies the credentials. The admin password grants push and pull access to every repository in the registry. The ex-developer pushes a backdoored image to a production repository. Full registry access from leaked admin credential High
2 latest tag used in production deployments Mutable tag prevents deterministic deployments AKS deployment manifests reference myapp:latest. A CI pipeline pushes a new broken image tagged as latest. AKS new pods pull the latest version automatically on restart or rolling update. The broken image causes pod failures across the cluster. Rollback requires identifying and re-tagging the previous image, which may no longer be tagged. No immutable reference exists. Production failure from unintentional image update High
3 No image vulnerability scanning Unscanned images in production Container images are built and pushed to ACR without any vulnerability scanning step. A critical CVE (e.g., in the base image OS or a runtime library) exists in the production image for 4 months before the security team performs an ad-hoc scan. The vulnerability is actively exploited. Known CVE running in production undetected High
4 Public registry endpoint accessible No private endpoint, no firewall ACR is deployed without a private endpoint and no network firewall rules. The registry is accessible from any IP on the internet. An attacker who obtains a valid token (via pipeline secret leak or credential stuffing) can pull images from any location, inspecting application code and discovering environment variables baked into layers. Proprietary code exposure via image pull Med
5 Image with secrets baked into layers Secrets added in Dockerfile then deleted A Dockerfile uses COPY app.config /app/ where app.config contains a database password, then deletes it with RUN rm /app/app.config. The secret is deleted in a later layer, but Docker image layers are immutable. Any user who pulls the image can run docker history or inspect intermediate layers to recover the deleted secret. Secrets recoverable from image layer history High
6 No image retention policy — stale images accumulate Images never pruned ACR Premium supports retention policies. Without them, thousands of CI-built images accumulate over months, consuming significant storage and increasing costs. Old images with known CVEs remain in the registry. If a pipeline fails to tag a new release correctly, it may fall back to pulling an old vulnerable image. Storage cost and CVE exposure from stale images Med
7 Geo-replication pull from wrong region AKS in East US pulls from West Europe replica An AKS cluster in East US pulls images from the West Europe ACR replica due to a DNS misconfiguration. Cross-region image pulls add 200–400ms to pod startup times and consume expensive egress bandwidth. During West Europe maintenance, image pulls fail completely and pod scheduling in East US is blocked. Latency, cost, and availability from wrong replica Med
8 No content trust — unsigned images run Docker Content Trust disabled ACR supports image signing via Notary. No signing policy is enforced. An attacker who gains push access to the registry (via compromised CI token) pushes a backdoored image. Without content trust, AKS pulls and runs the unsigned, backdoored image. No policy exists to reject unsigned images. Backdoored image runs in production undetected Med
9 Overprivileged CI/CD token CI pipeline has AcrPush on all repositories The CI/CD service principal has AcrPush role at the registry level, giving it push access to every repository. A compromised CI pipeline for a low-security service can push images to the production service repository, overwriting valid images with malicious ones. Cross-service image poisoning via overprivileged push Med
10 ACR used without dedicated service accounts per application One identity pulls images for all services A single managed identity with AcrPull at the registry level is used by all AKS namespaces to pull images. If one namespace's workload is compromised and the attacker can enumerate IMDS for the identity token, they can pull any image in the registry — including images containing proprietary algorithms, configuration, or embedded keys from other services. Image enumeration and extraction from compromised workload Med

Mitigation Controls

# Failure Mitigation Layer
1 Admin account enabled Disable admin account; use Managed Identity with AcrPush/AcrPull RBAC roles Authentication
2 Mutable latest tag Reference images by digest in production manifests; use tags only for identification Image tagging
3 No vulnerability scanning Enable Defender for Containers; add Trivy scan step in CI pipeline; fail build on CRITICAL CVEs Image scanning
4 Public endpoint Enable private endpoint; disable public access; configure firewall for CI/CD agent IPs only Network access
5 Secrets in layers Never pass secrets as Dockerfile ARG or COPY; use multi-stage builds; scan with docker history check Layer hygiene
6 No retention policy Configure ACR Premium retention policy; set untagged manifest deletion after 30 days Retention policy
7 Wrong geo-replica Configure closest replica for each AKS cluster using --registry flag or DNS-based routing Geo-replication
8 Unsigned images Enable Notary v2 signing in CI pipeline; enforce signed image policy with AKS admission controller Content trust
9 Overprivileged push Use repository-scoped tokens for CI/CD; grant AcrPush only to specific repositories per pipeline Token scoping
10 Shared pull identity Assign per-namespace user-assigned managed identities with AcrPull scoped to specific repositories Identity isolation

Key Takeaways

  • Disable the admin account — use Managed Identity with AcrPush/AcrPull roles for all CI/CD and runtime access.
  • Reference images by digest in production manifests — mutable latest tags cause unpredictable deployments and broken rollbacks.
  • Scan images for CVEs in CI pipeline (Trivy or Defender) and fail builds on critical vulnerabilities.
  • Enable private endpoint and disable public access for production registries.
  • Use repository-scoped ACR tokens for CI/CD pipelines — a compromised pipeline should only push to its own repository.
