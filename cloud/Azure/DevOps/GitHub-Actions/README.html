GitHub Actions with Azure

Category: DevOps
Azure Service Type: CI/CD Workflows
Key Docs: GitHub Actions for Azure

Overview

GitHub Actions is GitHub's native CI/CD automation platform, executing workflows defined in YAML files (.github/workflows/) on events like push, pull_request, or schedule. For Azure deployments, GitHub Actions integrates with Azure services using the azure/login action — authenticating via either a service principal client secret or, preferably, OpenID Connect (OIDC) Workload Identity Federation. OIDC federation eliminates the need for stored Azure credentials in GitHub secrets by exchanging a short-lived GitHub OIDC token for an Azure access token at runtime. GitHub Actions runners can be GitHub-hosted (ephemeral) or self-hosted (for VNet connectivity and custom tooling).

How It Works

Core Concepts

Concept Description
Workflow YAML file triggered by events; contains jobs and steps
Job Runs on a runner; jobs can run in parallel or with dependencies
Runner GitHub-hosted (Ubuntu, Windows, macOS) or self-hosted
Action Reusable step unit — from GitHub Marketplace or custom
Secret Encrypted repository or organization-level variables for credentials
Environment Named deployment target with required reviewers and wait timers
OIDC Federation Exchange GitHub OIDC token for Azure access token — no static credentials
Workflow Permissions permissions: key scopes the GITHUB_TOKEN to minimum required scopes
Reusable Workflow Callable workflow template shared across repositories
Concurrency Prevents multiple deployments running simultaneously

Step-by-Step Flow — OIDC-Based Azure Deployment

sequenceDiagram
    participant GH as GitHub Actions Runner
    participant GHOIDC as GitHub OIDC Provider
    participant Entra as Entra ID
    participant Azure as Azure Resource Manager
    GH->>GHOIDC: Request OIDC token (repo, ref, environment claims)
    GHOIDC-->>GH: Short-lived JWT (audience: api://AzureADTokenExchange)
    GH->>Entra: Exchange JWT for Azure access token (federated credential)
    Entra->>Entra: Validate JWT issuer, subject, audience claim
    Entra-->>GH: Azure access token (1 hour)
    GH->>Azure: Deploy with az cli using access token
    Azure-->>GH: Deployment success
    Note over GH,Entra: No static credential stored — token expires with job

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Azure credentials stored as static GitHub secret Service principal client secret in repo secrets A service principal client secret is stored in a GitHub repository secret and used in the azure/login action. A developer with repo admin access exports all repository secrets. The attacker uses the client secret to authenticate as the service principal and gain access to the Azure subscription. The secret does not expire automatically. Azure access from leaked static credential High
2 Pull request workflows accessing secrets PR from fork triggers workflow with secret access A public repository has a workflow triggered on pull_request that uses Azure secrets. A malicious contributor opens a PR from a fork. The workflow runs and the fork code can reference and exfiltrate secrets via workflow run artifacts or environment outputs. Azure credential exfiltration via malicious PR High
3 Overly broad OIDC federation subject claim Federation allows any branch The OIDC federated credential is configured with subject: repo:org/repo:* (all refs). Any branch in the repository can trigger a workflow that exchanges for an Azure token. A developer working on an experimental branch with a malicious workflow (or a compromised dependency) can deploy to production from a non-main branch without approval. Unauthorized Azure access from non-main branch High
4 Workflow with write permissions to all resources No permissions: key set A workflow does not declare the permissions: key. The default GITHUB_TOKEN has contents: write, pull-requests: write, and other broad permissions. A compromised action from the Marketplace abuses the GITHUB_TOKEN to modify repository files, approve pull requests, and push commits. Repository compromise via broad GITHUB_TOKEN Med
5 Third-party action used without pinning to commit SHA Action version may change A workflow uses - uses: some-action@main or @v2. The action maintainer pushes a malicious update to the main branch or v2 tag. The next workflow run pulls the updated action, which exfiltrates Azure tokens or GITHUB_TOKEN from the runner environment. Supply chain attack via unpinned action High
6 Self-hosted runner shared across repositories Isolation boundary between repos broken An organization uses a self-hosted runner for Azure deployments (for VNet access). The runner is registered in an organization runner group accessible to multiple repositories. A low-trust repository's workflow job runs on the shared runner, executes code that reads .credentials files or environment variables from previous jobs left on the runner's disk, and gains access to Azure tokens cached from a previous high-trust repo's job. Cross-repo credential theft via shared runner High
7 No environment protection — deploys without approval Environment not configured with reviewers A GitHub environment named production exists but has no required reviewers configured. A workflow with a bug (or malicious change) automatically deploys broken code to production. No human review occurs before the production deployment executes. Unreviewed code deployed to production High
8 Workflow secrets logged in debug mode ACTIONS_STEP_DEBUG set to true A developer enables ACTIONS_STEP_DEBUG to troubleshoot a failing deployment. The debug output logs environment variables including the GitHub Actions token and any exported Azure access tokens. The workflow run log is accessible to all repository contributors and is retained for 90 days. Credential exposure in workflow run logs Med
9 Reusable workflow without trust boundaries Calling workflow inherits called workflow's secrets A reusable workflow is designed for deployment. An untrusted caller workflow (lower security context) passes secrets: inherit to the reusable workflow, inheriting all secrets including Azure credentials. The caller workflow is from a repository with lower access controls and compromises the Azure deployment pipeline. Secret inheritance abuse in reusable workflows Med
10 No concurrency controls — parallel deployments Multiple workflow runs deploy simultaneously A busy repository triggers multiple simultaneous deployments to the same environment (e.g., multiple PRs merged in quick succession). Without concurrency controls, two deployments run in parallel against the same infrastructure, causing state conflicts, resource deletion/recreation races, and potentially leaving infrastructure in an inconsistent state. Infrastructure state corruption from concurrent deployments Med

Mitigation Controls

# Failure Mitigation Layer
1 Static credential in secrets Migrate to OIDC federation (azure/login with client-id, tenant-id, subscription-id) OIDC federation
2 Fork PR secrets access Use pull_request_target carefully; require workflow approval for first-time fork contributors; disable secret access for fork builds Fork protection
3 Broad OIDC subject claim Scope federated credential to ref:refs/heads/main and specific environment claims Subject claim scoping
4 Broad GITHUB_TOKEN Always declare permissions: with minimum required scopes (e.g., contents: read) Token scoping
5 Unpinned actions Pin all third-party actions to commit SHA; use Dependabot for action version updates Action pinning
6 Shared self-hosted runner Use ephemeral self-hosted runners (container-based, one job per runner); dedicate runners per repository or security tier Runner isolation
7 No environment protection Configure required reviewers on all production environments; add wait timer Environment protection
8 Debug logging secrets Disable ACTIONS_STEP_DEBUG in production; never echo secrets; use ::add-mask:: for dynamic secrets Log hygiene
9 Secret inheritance abuse Avoid secrets: inherit; explicitly pass only required secrets to reusable workflows Reusable workflow security
10 Parallel deployments Use concurrency: group: ${{ github.workflow }}-${{ github.ref }}, cancel-in-progress: true Concurrency control

Top 3 Critical Mitigations

  1. OIDC Federation over Static Secrets: Replace service principal client secrets with OIDC federation. The GitHub OIDC token is exchanged at runtime for a short-lived Azure access token — no static credential to steal or rotate.
  2. Pin Third-Party Actions to Commit SHA: Action tags are mutable and can be changed to point to malicious code. Pin to uses: action/name@<commit-sha> and use Dependabot to receive trusted updates.
  3. Scope OIDC Subject Claims to Branch + Environment: An overly broad subject claim (repo:org/repo:*) allows any branch to deploy. Scope the federated credential to ref:refs/heads/main and specific deployment environments to prevent unauthorized deployments.

Key Takeaways

  • Use OIDC federation for all Azure deployments — eliminates static credentials that can be leaked from GitHub Secrets.
  • Pin all third-party actions to commit SHA — action tags are mutable and a target for supply chain attacks.
  • Scope OIDC federated credentials to specific branches and environments — never allow all refs (*).
  • Always declare permissions: with minimum scopes — the default GITHUB_TOKEN has broader permissions than most jobs need.
  • Configure required reviewers on production GitHub Environments — they are the human gate before any production deployment.
