Azure API Management

Category: Messaging / Integration
Azure Service Type: API Gateway
Key Docs: API Management Overview

Overview

Azure API Management (APIM) is a hybrid, multi-cloud API gateway platform that provides a consistent frontend for backend services regardless of their location (Azure, on-premises, other clouds). It acts as a reverse proxy, applying policies (authentication, rate limiting, transformation, caching, logging) to all API traffic. APIM consists of a Gateway (traffic handling), Management Plane (API/policy configuration), and Developer Portal (self-service API discovery and subscription). Tiers range from Consumption (serverless, per-call) to Premium (multi-region, VNet injection, zone redundancy). APIM is the standard pattern for exposing microservices, legacy backends, and third-party integrations through a unified, secured API surface.

How It Works

Core Concepts

Concept Description
API Logical grouping of operations exposed through the gateway
Product Bundle of APIs with usage terms; consumers subscribe to products
Subscription Key Ocp-Apim-Subscription-Key header — primary access credential for API products
Policy XML snippets applied inbound/outbound/backend/on-error (auth, rate limit, transform, cache)
Backend Target service APIM proxies to; can have retry, circuit breaker, load balancer
Named Value Key-value store for secrets and config values referenced in policies
Gateway Self-hosted or managed runtime that processes API requests
Developer Portal Customizable portal for API discovery, documentation, and self-service subscriptions
VNet Integration External or Internal mode injects gateway into VNet
Managed Identity APIM can use managed identity to authenticate to backends and Key Vault

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Subscription key exposed in client-side code Key embedded in JavaScript or mobile app A frontend developer embeds the APIM subscription key in JavaScript code for a public web application. The key appears in browser developer tools network requests. An attacker extracts the key and uses it to call the API directly at high volume, consuming the rate limit and generating charges. The subscription key also grants access to all APIs in the product, potentially including internal APIs. API abuse and data access via exposed key High
2 No backend authentication — API key stripping only APIM validates key but backend is open APIM validates subscription keys but passes requests to the backend without any authentication. The backend API is accessible directly via its URL if discovered. An attacker performs reconnaissance on Azure-hosted services, discovers the backend App Service URL, and calls the API directly, bypassing all APIM policies (rate limiting, JWT validation, IP filtering). APIM bypass via direct backend access High
3 Rate limiting policy misconfigured — wrong scope Rate limit per product, not per subscription A rate limit policy is configured at the product level (rate-limit instead of rate-limit-by-key). The limit allows 1,000 requests per minute per product. Multiple subscribers all share the same product counter. A single high-volume subscriber exhausts the 1,000/min limit. Other subscribers receive 429 errors even though they have not exceeded their individual expected usage. Rate limit shared across all subscribers Med
4 Secrets in policy XML — plaintext Backend API keys hardcoded in policy A developer hardcodes a backend API key directly in the APIM policy XML: <set-header name="X-Api-Key"><value>secret123</value></set-header>. Policy XML is visible to anyone with APIM configuration access (e.g., Azure Portal contributor). The secret is exposed to all administrators and is not rotated because no one manages it as a secret. Backend credentials exposed in policy configuration High
5 Self-hosted gateway — insecure default config Self-hosted gateway deployed without TLS or cert pinning A self-hosted APIM gateway is deployed on-premises for hybrid connectivity. The gateway container runs with default configuration and no TLS termination. Traffic between on-premises clients and the gateway is unencrypted. Additionally, the gateway configuration endpoint is reachable from the internal network without authentication, exposing API routing configuration. Plaintext API traffic on internal network Med
6 CORS policy too permissive * allowed origins The CORS policy is configured with <allowed-origins><origin>*</origin></allowed-origins>. Any website can make cross-origin AJAX calls to the API using the browser-stored subscription key. A malicious website visited by an authenticated user can trigger AJAX calls to the API and steal data from the responses using the user's session context. Cross-site request forgery via open CORS Med
7 No JWT validation — tokens not verified Missing validate-jwt policy APIs are protected only by subscription keys, not JWT tokens. Applications send user JWTs in the Authorization header, but the APIM gateway does not validate them (no validate-jwt policy). The backend also does not validate tokens. An attacker sends a crafted or expired JWT token. The API processes the request using the claims in the unverified token, performing actions as any user the attacker specifies in the token claims. Authorization bypass via unverified JWT High
8 Developer portal publicly exposes internal APIs All APIs added to developer portal without product control APIs are added to the developer portal without restricting them to internal products. External developers who self-register on the portal can subscribe to internal APIs intended only for internal services. The subscription self-service allows the external developer to obtain a valid key and call internal APIs that expose sensitive business data. Unauthorized access to internal APIs via portal Med
9 APIM deployed without VNet — public gateway External-facing gateway in shared tier APIM is deployed without VNet integration. The gateway has a public IP. All backend connections from APIM to backends also go over public endpoints. The management API is accessible publicly. An attacker attempts a brute force against the management API or looks for known vulnerabilities in the APIM management surface. Expanded attack surface from public management endpoint Med
10 No circuit breaker on backend — cascading failures APIM forwards requests to unhealthy backend A backend service is degraded and responding slowly (10+ second response times). APIM does not have a circuit breaker policy configured. All incoming requests queue and wait for the slow backend. APIM connection threads are exhausted. The gateway itself becomes unresponsive, taking down all APIs — not just those connected to the failing backend. Gateway-wide outage from one unhealthy backend High

Mitigation Controls

# Failure Mitigation Layer
1 Subscription key exposure Use OAuth 2.0 / JWT tokens for frontend auth instead of subscription keys; use keys only for backend-to-backend Authentication model
2 Unauthenticated backend Add managed identity or certificate auth in backend policy; restrict backend to APIM IP/VNet Backend auth
3 Wrong rate limit scope Use rate-limit-by-key with subscription-id counter key for per-subscriber limits Rate limiting
4 Secrets in policy Use Named Values backed by Key Vault for all secrets in policies; rotate via KV Named Values for secrets
5 Insecure self-hosted gateway Enable TLS on self-hosted gateway; authenticate gateway to APIM management plane with gateway key Self-hosted TLS
6 Open CORS Specify exact allowed origins; never use * for credentialed CORS requests CORS policy
7 No JWT validation Add validate-jwt policy to all APIs requiring user identity; validate issuer, audience, expiry JWT validation
8 Portal exposes internal APIs Create separate Internal and External products; restrict internal APIs to products requiring admin subscription Product access control
9 Public gateway Deploy APIM in Internal VNet mode for internal APIs; use External mode + Front Door WAF for public APIs VNet integration
10 No circuit breaker Configure backend circuit breaker policy (circuit-breaker); set failure threshold and recovery timeout Resilience policy

Top 3 Critical Mitigations

  1. Validate-JWT Policy on All Protected APIs: Subscription keys are long-lived and shareable. JWT validation at the gateway enforces user identity, expiry, audience, and scope — ensuring tokens from the correct issuer are required.
  2. Backend Authentication via Managed Identity: APIM should use managed identity to authenticate to backends. Backends should only accept traffic from APIM (via IP/VNet restriction), not from any caller with a subscription key.
  3. Named Values + Key Vault for All Secrets: Never hardcode secrets in policy XML. Named Values backed by Key Vault allow secrets to be rotated without policy changes and ensure access is audited.

Key Takeaways

  • APIM is a policy engine — authentication, rate limiting, transformation, and backend routing are all defined as composable XML policies.
  • Never bypass APIM to backends — lock backends to accept traffic only from APIM via VNet or IP restriction.
  • Use Named Values backed by Key Vault for all policy secrets — never hardcode credentials in policy XML.
  • validate-jwt is mandatory for any API with user identity; subscription keys alone are insufficient for user-level authorization.
  • Deploy APIM in VNet Internal mode for internal APIs; combine with Azure Front Door or Application Gateway for public exposure.
