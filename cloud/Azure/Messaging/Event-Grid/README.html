Azure Event Grid

Category: Messaging
Azure Service Type: Serverless Event Routing
Key Docs: Event Grid Overview

Overview

Azure Event Grid is a serverless, fully managed event routing service that enables reactive programming based on Azure resource events. It uses a push-based pub/sub model where publishers emit events and subscribers receive them via webhooks, Azure Functions, Event Hubs, Service Bus, or storage queues. Event Grid delivers events with at-least-once semantics and retries with exponential backoff for 24 hours. It natively integrates with Azure services (Blob Storage, Resource Manager, IoT Hub, Entra ID, etc.) as built-in event sources. Custom topics allow applications to publish their own events. Event Grid Domains support large-scale multi-tenant event routing.

How It Works

Core Concepts

Concept Description
Event Source Publisher of events: Azure service (Blob Storage, ARM) or custom application
Topic Endpoint where event sources send events
System Topic Built-in Azure service event source (e.g., Storage, IoT Hub, Resource Group)
Custom Topic User-defined topic for application events
Event Subscription Defines who receives which events from a topic; includes filter and endpoint
Event Handler Destination for events: webhook, Azure Function, Event Hubs, Service Bus, Storage Queue
Filter Subject, event type, or advanced attribute filters on subscriptions
Dead-Letter Events that cannot be delivered after 24h of retries go to configured storage blob
Event Grid Domain Namespace for thousands of topics — used for multi-tenant or large-scale scenarios
CloudEvents CNCF CloudEvents 1.0 schema support for interoperability

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Webhook endpoint impersonation — event injection No signature validation on webhook handler An Event Grid subscription delivers events to a webhook at https://api.contoso.com/events. The webhook handler does not validate the aeg-sas-key or aeg-event-key header. An attacker who discovers the webhook URL sends crafted POST requests with fake event payloads. The handler processes the fake events as legitimate Event Grid events, triggering incorrect business logic. Unauthorized event injection into processing pipeline High
2 No dead-letter configured — events silently lost Webhook returns 5xx, events expire A webhook endpoint is temporarily unavailable for 25 hours (longer than the 24h retry window). Event Grid retries delivery for 24 hours, but the events expire and are dropped. No dead-letter destination is configured. Events representing order status changes, user registrations, or security notifications are permanently lost. Silent event loss after 24-hour retry window High
3 Overpermissioned Event Grid subscription System topic subscription triggers dangerous operations An Event Grid subscription on a Resource Manager system topic triggers an Azure Function on all ResourceWriteSuccess events. The function was intended to log changes. A bug in the function logic causes it to try to delete resources matching certain criteria. Because the event is triggered on every ARM write, the function runs on every deployment and may inadvertently delete resources. Unintended resource deletion from overpermissive trigger Med
4 Event Grid topic key in application config Static key committed to source control The custom topic access key is stored in application config and committed to a private repository. A developer with repository access extracts the key and publishes fake events to the custom topic directly. The fake events trigger downstream Azure Functions and Service Bus processors, injecting fraudulent business events into the production pipeline. Event stream manipulation via leaked topic key High
5 No filter on subscription — event flood All event types delivered to handler An Event Grid subscription on a Blob Storage system topic has no filters. The handler is triggered on every blob event: BlobCreated, BlobDeleted, BlobTierChanged. A bulk storage operation creates 50,000 blobs, triggering 50,000 Function invocations simultaneously. The Functions consumption plan scales aggressively, causing unexpected Azure compute costs and downstream database connection exhaustion. Uncontrolled invocation storm from no filtering Med
6 Resource Group event subscription monitoring prod changes Too broad event scope An Event Grid subscription monitors an entire subscription-level Resource Manager system topic. Security team receives events for ALL changes in ALL resource groups, including dev and test. The alert system is flooded with development changes, masking real production security events. The team stops reviewing alerts due to noise. Alert fatigue from over-broad event scope Med
7 Webhook validation handshake not implemented New subscription fails validation Event Grid requires webhooks to complete a validation handshake (SubscriptionValidationEvent) when a subscription is created. A new webhook endpoint does not implement the handshake response. The Event Grid subscription cannot be created — the endpoint is considered invalid. The integration fails silently until developers investigate. Integration failure from missing webhook validation Med
8 Cross-tenant event delivery — data boundary violation Events delivered to external tenant webhook An Event Grid subscription is accidentally configured to deliver events to a webhook URL that belongs to a third-party or partner organization's endpoint. Storage access events, user change events, and resource events are now delivered outside the organization's tenant boundary. Data leakage via misdirected event delivery Med
9 Event Grid Domain key exposed — all tenants compromised Domain-level key allows cross-tenant event injection An Event Grid Domain is used for a multi-tenant SaaS application. The Domain-level access key is used for all publishers. A key leak allows any attacker to publish events to any topic within the domain, injecting fraudulent events for any tenant. The blast radius of the key leak is all tenants in the domain. All tenant event streams compromised via domain key Med
10 Retry storm overloads endpoint Endpoint slow — concurrent retries flood A webhook endpoint becomes slow under load (>30 seconds per request). Event Grid retries unacknowledged events. Retries and new events pile up against the already-slow endpoint, increasing load further. The endpoint reaches 100% CPU and starts returning 5xx. Event Grid retries all 5xx events, creating a feedback loop that keeps the endpoint degraded. Retry amplification causing endpoint overload Med

Mitigation Controls

# Failure Mitigation Layer
1 No webhook validation Validate aeg-sas-key header on every webhook request; use Event Grid-managed validation for Azure Functions Webhook security
2 No dead-letter Configure dead-letter blob container on all Event Grid subscriptions; monitor DLQ blob Dead-letter config
3 Overpermissive trigger Scope event subscriptions to specific event types and subjects; validate event content before acting Subscription filters
4 Topic key in config Use Managed Identity (Event Grid Data Sender role) for custom topic publishing Managed Identity
5 No filters Always add subject and event type filters to subscriptions; design idempotent handlers Filter configuration
6 Broad event scope Scope system topics to resource groups, not subscriptions; filter by resource type and event type Topic scope
7 Missing validation handshake Implement SubscriptionValidationEvent response in all webhook handlers Webhook implementation
8 Misdirected delivery Validate webhook endpoint domain before subscription creation; use Private Endpoints for internal handlers Endpoint validation
9 Domain key exposure Use per-topic SAS keys or Managed Identity for Event Grid Domain publishers Domain security
10 Retry storm Implement fast webhook acknowledgment (return 202 immediately, process async); use circuit breaker Async processing

Key Takeaways

  • Event Grid is for reactive event routing (push-based, low latency); Event Hubs is for high-throughput stream ingestion (pull-based).
  • Always validate webhook signatures — unchecked webhooks are vulnerable to event injection attacks.
  • Configure dead-letter destinations on all subscriptions — events silently expire after 24 hours without dead-letter.
  • Use Managed Identity for custom topic publishers — topic keys are long-lived credentials vulnerable to leakage.
  • Filter subscriptions to specific event types and subjects — unfiltered subscriptions cause invocation storms on bulk operations.
