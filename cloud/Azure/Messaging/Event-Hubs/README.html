Azure Event Hubs

Category: Messaging
Azure Service Type: High-Throughput Event Streaming
Key Docs: Event Hubs Overview

Overview

Azure Event Hubs is a fully managed, real-time event streaming platform capable of ingesting millions of events per second. It is designed for telemetry pipelines, log aggregation, clickstream analytics, IoT data ingestion, and fraud detection. Event Hubs is partition-based — each event hub has 1–32 partitions; events are ordered within a partition. Consumers use consumer groups to read independently at their own pace. Event Hubs supports the Apache Kafka protocol (port 9093), enabling Kafka clients to use Event Hubs without code changes. Event Hubs Capture automatically writes event data to Azure Blob Storage or Data Lake Gen2 for cold-path analytics.

How It Works

Core Concepts

Concept Description
Namespace Container for event hubs; unit of billing and network configuration
Event Hub Named stream within a namespace (analogous to Kafka topic)
Partition Ordered, immutable log; events are appended to and read from partitions
Consumer Group Named logical reader of an event hub; each CG maintains its own offset
Partition Key Determines partition assignment; events with the same key go to the same partition
Event Retention Events retained 1–90 days (Standard/Premium); reads can replay from any offset
Capture Automatically archive events to Blob Storage or Data Lake in Avro format
Kafka Endpoint Event Hubs exposes Kafka-compatible endpoint on port 9093
AMQP Native protocol for Event Hubs producers and consumers
Checkpoint Consumer stores its current offset so it can resume after restart

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 SAS key leaked — event injection Connection string in application config An Event Hubs producer SAS key is committed to source control. An attacker extracts the key and sends fraudulent events to the event hub. The consumer processes these synthetic events (e.g., fraudulent IoT sensor readings, fake transaction events) as real data. The fraud is only detected when the physical sensor values are compared against processed data in a manual audit. Data poisoning via fraudulent event injection High
2 Hot partition — all events to one partition Low-cardinality partition key All events use the same partition key (e.g., Environment=Production). All events go to a single partition. Event Hubs load is uneven — one partition processes all traffic while 7 others are idle. The single partition's throughput is limited, causing producers to receive EventHubsException: ServerBusy. Events are dropped or delayed under load. Throughput bottleneck and event loss from hot partition High
3 No checkpointing — events reprocessed on restart Consumer does not persist offset An Event Hubs consumer using the Event Processor Host does not call UpdateCheckpointAsync after processing each batch. The consumer crashes and restarts. On restart, it reads from the last checkpoint (or beginning of retention if no checkpoint). All events from the last restart point are reprocessed. Downstream systems receive duplicate data, causing double-counting in analytics dashboards. Duplicate event processing from missing checkpoint High
4 Retention period too short — events lost 1-day retention for debugging purposes An Event Hub is configured with 1-day event retention. A consumer falls behind due to a downstream service outage lasting 30 hours. When the consumer recovers, the events from the first 6 hours of the outage are beyond the retention window and permanently unavailable. The data gap is unrecoverable. Permanent event loss from short retention Med
5 No Capture — event data not persisted long-term Relying on retention window only An IoT platform stores 30 days of events in Event Hubs for reprocessing. Capture is not enabled. A new ML model needs to be trained on 6 months of historical data. Only 30 days are available. The team must rebuild the historical dataset from other sources. Event data that was not captured is permanently unrecoverable. Historical data unavailable for analytics Med
6 Consumer group exhausted Exceeded max concurrent consumer groups An Event Hub has the default 20 consumer group limit. Multiple new analytics pipelines (Stream Analytics jobs, Databricks streams, custom consumers) each need their own consumer group. The limit is reached. New consumer groups cannot be created. A new pipeline shares a consumer group with another, causing offset conflicts — one consumer's checkpoint affects the other. Consumer group contention and offset corruption Med
7 Namespace public endpoint not locked down No private endpoint or firewall Event Hubs Premium is deployed without a private endpoint. Producers and consumers connect over the public internet. An attacker on the same network performs traffic interception. Although AMQP uses TLS, an attacker who obtains a SAS token (from MITM of the token endpoint or log leakage) can connect from any IP. Unauthorized event stream access Med
8 Event ordering broken across partitions Consumers read multiple partitions concurrently An application requires strict event ordering across all events (not just per partition). Events are distributed across 8 partitions for throughput. A consumer reads all 8 partitions concurrently. Events arrive out of chronological order because partition 3 lags behind partition 1. The consumer processes events in partition read order, not emission time order, breaking the ordering assumption. Application logic failure from cross-partition ordering Med
9 Kafka client misconfiguration — offset reset auto.offset.reset=earliest on restart A Kafka client using the Event Hubs Kafka endpoint is configured with auto.offset.reset=earliest. When the consumer group has no committed offset (first run or after offset expiry), the consumer reads from the earliest available event. On a system with 30-day retention, this reprocesses 30 days of events. Downstream systems receive duplicates of weeks of historical data. Bulk duplicate processing from offset reset Med
10 Throughput unit (TU) exhaustion TU capacity not provisioned for peak An Event Hub namespace has 1 Throughput Unit (1 MB/s ingress, 2 MB/s egress). During a peak event like a flash sale, producers generate 5 MB/s of event data. The namespace is throttled at 1 MB/s. Events are rejected with ServerBusy errors. Producers must implement retry with exponential backoff or events are dropped. Event loss during traffic peaks from TU exhaustion Med

Mitigation Controls

# Failure Mitigation Layer
1 SAS key leaked Use Managed Identity (RBAC: Event Hubs Data Sender/Receiver) for all producers and consumers Managed Identity
2 Hot partition Use high-cardinality partition keys (e.g., DeviceId, UserId); or use null key for round-robin Partition design
3 No checkpointing Use Event Processor Client with UpdateCheckpointAsync after successful batch processing Checkpointing
4 Short retention Set retention to 7 days minimum; use Capture for long-term storage requirements Retention config
5 No Capture Enable Capture on all production event hubs to Blob Storage or Data Lake Gen2 Capture configuration
6 Consumer group exhaustion Request consumer group limit increase; plan consumer groups per pipeline at design time Consumer group planning
7 Public endpoint Enable private endpoint on Premium namespace; restrict public access via firewall Network security
8 Cross-partition ordering Design for per-partition ordering only; use single partition (at cost of throughput) if global order required Partition design
9 Kafka offset reset Set auto.offset.reset=latest in production; manage initial offset explicitly Kafka configuration
10 TU exhaustion Enable Auto-inflate for automatic TU scaling; set max TU to cap cost Throughput scaling

Key Takeaways

  • Event Hubs is designed for high-throughput event streaming (millions/sec); Service Bus is for reliable message delivery with ordering and DLQ.
  • Use high-cardinality partition keys (DeviceId, UserId) — low-cardinality keys create hot partitions.
  • Enable Capture for durable event storage; the retention window is not a backup.
  • Always checkpoint in consumer code — missing checkpoints cause bulk reprocessing after restarts.
  • Use Managed Identity (RBAC) instead of SAS keys — leaked SAS keys allow event injection and stream access.
