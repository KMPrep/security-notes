Azure Logic Apps

Category: Messaging / Integration
Azure Service Type: Low-Code Workflow Automation
Key Docs: Logic Apps Overview

Overview

Azure Logic Apps is a cloud-native integration platform that enables building automated workflows connecting Azure services, SaaS applications (Salesforce, Slack, Office 365), and on-premises systems through 400+ pre-built connectors. Logic Apps uses a visual workflow designer with triggers and actions. Standard tier (based on Azure Functions runtime) replaces Consumption tier for stateful and stateless workflows with VNet integration, private endpoints, and local execution for testing. Logic Apps is commonly used for Sentinel playbooks, Service Bus message processing, data transformation pipelines, and IT automation.

How It Works

Core Concepts

Concept Description
Trigger Event that starts a workflow (HTTP request, Service Bus message, timer, Event Grid)
Action Step in a workflow (HTTP call, condition, loop, connector action)
Connector Pre-built integration with a service (Office 365, Salesforce, SQL, Azure Storage)
Managed Connector Hosted by Microsoft; uses API connections with credentials
Built-in Connector Runs in-process with the Logic App runtime (Service Bus, HTTP, Azure Functions)
API Connection Stores authentication credentials for a managed connector
Workflow JSON-defined sequence of trigger + actions
Standard SKU App Service Plan-based; VNet integration, private endpoints, stateful + stateless
Consumption SKU Serverless, pay-per-execution; no VNet integration; simpler for lightweight automation
Run History Record of all executions including inputs and outputs

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Sensitive data logged in run history Inputs/outputs containing secrets stored in run history A Logic App workflow calls an HTTP action with an Authorization header containing a bearer token, and a SQL action with a database password as a parameter. Both inputs are stored in plain text in the Logic App run history. A developer with read access to the Logic App views run history and extracts bearer tokens and database credentials from logged payloads. Credential theft via run history High
2 API connection uses shared service account Connector authenticated with a shared account Multiple Logic App workflows share a single Office 365 API connection authenticated with a shared service account (automation@contoso.com). The service account has broad Mail.ReadWrite permissions. A developer who can modify Logic Apps creates a new workflow using the shared connection to exfiltrate emails from any mailbox. Unauthorized email access via shared API connection Med
3 HTTP trigger without authentication Logic App trigger URL is a public unauthenticated endpoint A Logic App HTTP trigger URL (a long random URL) is treated as implicit security by obscurity. An attacker who obtains the trigger URL (via log leakage, network capture, or source code exposure) can invoke the Logic App repeatedly, triggering business logic actions like provisioning users, sending emails, or modifying data. Unauthorized workflow execution High
4 No error handling — partial workflow failures Workflow actions fail silently A Logic App workflow creates a user, assigns a license, and sends a welcome email in sequence. The license assignment action fails, but the workflow has no error handling (Configure run after: Failed). The workflow completes successfully (from Logic Apps' perspective) because the failed action is not terminal. The user receives a welcome email but has no license and cannot access services. Silent partial failure in automated workflows High
5 Connector credentials not scoped Connector has admin-level permissions A Logic App using a SharePoint connector is authenticated with a Global Admin account. The connector is used to read a single SharePoint list. The Global Admin credential allows the Logic App (and any attacker who can modify it) to access all SharePoint sites, all data, and perform admin operations on the tenant. Over-privileged connector with blast radius Med
6 No resource lock — Logic App modified by attacker IAM misconfiguration allows unauthorized edits An attacker who compromises an Entra ID account with Contributor role on the Logic App resource modifies the Sentinel playbook. They change the revokeSignInSessions action to a no-op and add an action that sends incident details to an external webhook. Sentinel playbooks now exfiltrate incident data to the attacker's server while appearing to function normally. Playbook backdooring for data exfiltration Med
7 Consumption tier without VNet integration Connectors route over public internet A Logic App Consumption workflow connects to an on-premises SQL Server via the on-premises data gateway. The connection traverses the public internet. While TLS is used, the workflow also processes sensitive financial data. An outbound firewall rule is misconfigured, allowing plaintext connections through the gateway. Financial data is transmitted without encryption. Sensitive data exposure via unencrypted gateway Med
8 Runaway loop — infinite retry Loop condition bug causes infinite execution A Logic App workflow uses a Until loop to poll a status endpoint. A bug in the condition causes the loop to never terminate. The workflow runs indefinitely. In the Consumption tier, this generates thousands of action executions per hour, incurring large per-action charges. In Standard tier, the workflow thread hangs. Runaway execution cost and resource exhaustion Med
9 Trigger polling too frequent Timer trigger every 1 minute for low-frequency events A Logic App polls an external REST API every minute for new records. The API has a low update frequency (new records once per hour). Each poll generates an HTTP request and a Logic App run, even when no data is returned. This results in 1,440 executions per day in Consumption tier, with 99% returning empty results — all generating execution charges. Unnecessary cost from over-polling Low
10 Standard Logic App not in VNet Outbound connections from Standard LA use public IP A Standard Logic App (which supports VNet integration) is deployed without VNet integration. The Logic App connects to Azure SQL, Azure Storage, and Key Vault over public endpoints. The SQL and Key Vault firewalls must allow traffic from any IP, expanding the attack surface. Expanded network attack surface Med

Mitigation Controls

# Failure Mitigation Layer
1 Sensitive data in run history Enable Secure Inputs/Outputs in action settings to obfuscate sensitive data in run history Data protection
2 Shared API connection Create per-workflow API connections with Managed Identity where supported; use least-privilege app registrations API connection security
3 Unauthenticated HTTP trigger Enable Azure Active Directory authentication on HTTP triggers; use trigger URL only within secure pipelines Trigger security
4 No error handling Add Configure run after: Failed branches; implement compensation logic for partial failures Error handling
5 Overprivileged connector Use Managed Identity with RBAC for Azure service connectors; use minimal permissions for SaaS connectors Connector permissions
6 No resource lock Apply ReadOnly lock on Sentinel playbooks; require PIM elevation to modify Resource protection
7 No VNet Use Standard SKU with VNet integration and private endpoints to on-premises data gateway Network isolation
8 Runaway loop Set maximum iteration count on Until loops; enable workflow timeout Loop protection
9 Over-polling Use event-driven triggers (Event Grid, Service Bus) instead of polling; increase polling interval Trigger optimization
10 No VNet integration Enable VNet integration in Standard Logic Apps; use private endpoints for all connected services VNet integration

Key Takeaways

  • Enable Secure Inputs/Outputs on actions handling sensitive data — run history stores inputs/outputs in plain text by default.
  • Use Managed Identity for Azure service connectors — API connections with stored credentials are vulnerable to credential theft.
  • Always add error handling (Configure run after: Failed) — Logic Apps marks workflows as succeeded even if actions fail silently.
  • Standard SKU with VNet integration is required for production workflows connecting to private resources.
  • Sentinel playbooks (Logic Apps) are a security control — protect them with resource locks and audit modifications.
