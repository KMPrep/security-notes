Azure Service Bus

Category: Messaging
Azure Service Type: Enterprise Messaging
Key Docs: Service Bus Overview

Overview

Azure Service Bus is a fully managed enterprise message broker supporting queues (point-to-point) and topics/subscriptions (publish-subscribe). It provides guaranteed message delivery, FIFO ordering with message sessions, duplicate detection, dead-letter queues, scheduled delivery, and message deferral. Service Bus supports both AMQP 1.0 and HTTP protocols and is compliant with JMS 2.0. It is used for decoupling microservices, distributing workloads, and implementing reliable workflows. The Premium tier adds dedicated capacity, VNet integration, customer-managed keys, and private endpoints — required for production and regulated workloads.

How It Works

Core Concepts

Concept Description
Queue Point-to-point channel; each message consumed by exactly one receiver
Topic Pub/sub channel; multiple subscriptions each receive a copy of published messages
Subscription Named filter on a topic — subscribers receive messages matching their filter rules
Dead-Letter Queue DLQ — messages that cannot be processed after MaxDeliveryCount retries
Message Session Groups related messages by SessionId for ordered, exclusive processing
Duplicate Detection Idempotent deduplication window (up to 7 days) based on MessageId
Lock Duration Receiver holds a lock during processing; message redelivered if lock expires
Peek-Lock Receive and hold message; must Complete or Abandon before timeout
Scheduled Delivery Enqueue message to be delivered at a future time
Premium Tier Dedicated capacity, VNet integration, private endpoint, CMK

Step-by-Step Flow — Reliable Message Processing

sequenceDiagram
    participant Producer as Order Service (Producer)
    participant SB as Service Bus Queue
    participant Consumer as Payment Service (Consumer)
    participant DLQ as Dead-Letter Queue
    Producer->>SB: Send message (OrderId=123, MessageId=abc)
    SB->>SB: Duplicate check — MessageId abc seen? No
    SB-->>Producer: Accepted
    Consumer->>SB: Receive (Peek-Lock — 30s lock)
    SB-->>Consumer: Message (OrderId=123)
    Consumer->>Consumer: Process payment
    Consumer->>SB: Complete message (remove from queue)
    Note over Consumer,SB: On processing failure:
    Consumer->>SB: Abandon (delivery count + 1)
    SB->>SB: Redeliver up to MaxDeliveryCount (10)
    SB->>DLQ: Move to DLQ after 10 failures

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Shared access key in application code Connection string with Manage rights in config A developer embeds the Service Bus primary connection string (with Manage rights) in a .NET application config file. The config is committed to a public GitHub repository. An attacker extracts the connection string and gains full control of the namespace — creating queues, sending messages, receiving and deleting messages. The attacker can disrupt messaging and inject malicious messages. Full namespace access via leaked key High
2 Messages not deleted — poison message loop Message processor crashes before completing A consumer receives a message and begins processing. The processor crashes before calling Complete(). The lock expires and the message is redelivered. The consumer always crashes on this message. The message is delivered 10 times (MaxDeliveryCount) and moves to the DLQ — but if DLQ is not monitored, the original message is silently lost. Other messages in the queue queue behind it, causing processing delays. Message loss and queue backup from poison message High
3 No dead-letter queue monitoring DLQ messages accumulate unnoticed Messages fail processing and accumulate in the DLQ without alerting. A payment processing issue causes all financial messages to dead-letter. The DLQ grows to 10,000 messages over 24 hours. No Azure Monitor alert is configured for DLQ message count. The business discovers the issue only when customer complaints arrive about missing orders. Silent message loss from unmonitored DLQ High
4 Namespace accessible via public endpoint No VNet integration or private endpoint Service Bus Premium is deployed without a private endpoint. Applications are granted access via IP filtering, but a misconfigured IP rule set changes allows all IPs. Any internet host with a valid SAS key can connect. An attacker who obtains a SAS key (via credential scan) can connect from any location and read business-critical messages. Message exfiltration via public endpoint Med
5 SAS key rotation not automated Static keys never rotated Service Bus SAS keys are created at namespace setup and never rotated. The primary and secondary keys are shared with many teams. When an employee with access to the keys leaves the organization, the keys are not rotated. The former employee retains access to the messaging infrastructure indefinitely using the unrotated SAS key. Continued access by departed employee Med
6 No message TTL — stale messages processed Long message expiry on time-sensitive data An order queue has no message time-to-live (TTL) configured. A Service Bus outage causes messages to queue for 6 hours. When the consumer resumes, it processes 6-hour-old order messages. The orders were manually fulfilled by the support team during the outage. Processing the stale messages causes duplicate order fulfillment and double billing. Stale message processing after outage Med
7 Lock duration too short — double processing Lock expires before processing completes A consumer receives a message with a 30-second lock duration. The message processing involves a slow downstream API call that takes 45 seconds. The lock expires at 30 seconds and the message is redelivered to another consumer instance. Both instances complete processing — the downstream action is performed twice. No idempotency protection exists. Duplicate message processing from lock expiry High
8 No message ordering — concurrent consumers break sequence Multiple consumers without sessions An order processing workflow requires messages to be processed in sequence per customer (create → pay → ship). Multiple consumer instances process messages concurrently without message sessions. Message session-level FIFO ordering is not configured. Messages for the same customer are processed out of order, causing state corruption in the order management system. Out-of-order processing breaks business logic Med
9 Topic filter misconfiguration — messages misrouted Wrong SQL filter on subscription A topic has two subscriptions: premium-orders and standard-orders. The SQL filter on premium-orders is OrderType = 'Premium'. A developer changes the message property name from OrderType to order_type in a service update. The subscription filter no longer matches any messages — all premium orders fall through to standard-orders. Premium orders are processed as standard orders silently. Message misrouting from broken subscription filter Med
10 Namespace deletion without backup Dev automation accidentally targets prod An IaC pipeline designed to tear down a dev namespace uses the wrong resource group name and targets the production Service Bus namespace. All queues, topics, and in-flight messages are deleted. Service Bus does not have a backup or export feature — messages that have not been consumed are permanently lost. Permanent message loss from accidental deletion Med

Mitigation Controls

# Failure Mitigation Layer
1 SAS key in code Use Managed Identity for all Service Bus authentication; use RBAC roles (Service Bus Data Sender, Receiver) Managed Identity
2 Poison message loop Implement idempotent message handlers; set appropriate MaxDeliveryCount; monitor DLQ Error handling
3 Unmonitored DLQ Create Azure Monitor alert on DLQ active message count > 0; assign ownership for DLQ reprocessing DLQ monitoring
4 Public endpoint Enable private endpoint on Premium namespace; configure firewall to deny public access Network access
5 Static SAS keys Regenerate SAS keys on personnel change; better: migrate to Managed Identity Key rotation
6 No TTL Set message TTL to business-appropriate value (minutes, not hours); design consumers for idempotency Message TTL
7 Lock expiry Set lock duration > max expected processing time; implement RenewMessageLockAsync for long-running tasks Lock management
8 No ordering Use Message Sessions (SessionId = CustomerId) for ordered, exclusive per-entity processing Message sessions
9 Broken subscription filter Test subscription filters before deployment; add filter change monitoring in CI/CD Filter testing
10 Namespace deletion Apply resource locks (CanNotDelete) on production Service Bus namespaces Resource protection

Key Takeaways

  • Always use Managed Identity (RBAC) for Service Bus authentication — SAS keys are long-lived credentials vulnerable to leakage.
  • Monitor the DLQ — unmonitored DLQs cause silent message loss that manifests as business logic failures.
  • Use Message Sessions for ordered, exclusive per-entity processing in workflows that require sequence.
  • Set lock duration longer than max processing time and renew locks for long-running operations to prevent double processing.
  • Apply CanNotDelete resource locks on production namespaces — Service Bus has no backup/restore; deletion is permanent.
