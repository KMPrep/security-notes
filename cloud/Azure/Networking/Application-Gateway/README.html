Azure Application Gateway

Category: Networking
Azure Service Type: L7 Application Delivery Controller
Key Docs: Application Gateway Overview

Overview

Azure Application Gateway is a managed Layer 7 (HTTP/HTTPS) load balancer that provides URL-based routing, SSL/TLS termination, session affinity, WebSocket support, HTTP/2, and an integrated Web Application Firewall (WAF). It operates within a dedicated subnet in a VNet and can route to backends across VMs, VMSS, App Services, IP addresses, and FQDNs. WAF protects against OWASP Top 10 threats including SQL injection, XSS, and command injection. Application Gateway v2 supports autoscaling, zone redundancy, and private link, making it the preferred choice for regional application delivery.

How It Works

Core Concepts

Concept Description
Frontend IP Public or private IP listening for inbound HTTP/HTTPS traffic
Listener Binds a frontend IP, port, and protocol to a routing rule
Routing Rule Basic (all traffic to one pool) or Path-based (URL paths to different pools)
Backend Pool Target resources: VMs, VMSS, App Service, IP, FQDN
Backend HTTP Settings Defines protocol, port, session affinity, and probe for backend connections
Health Probe Custom or default HTTP/HTTPS probe to check backend health
SSL Termination Decrypts TLS at Application Gateway; re-encrypt optional for end-to-end TLS
WAF Policy OWASP CRS rule sets; custom rules; exclusions; detection or prevention mode
Rewrite Rules Modify HTTP headers and URL paths on request/response
Autoscaling v2 SKU auto-scales capacity units based on traffic load

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 WAF in Detection mode only Default or misconfigured WAF mode WAF is deployed in Detection mode, which logs but does not block malicious requests. An attacker runs SQL injection attacks against the application. The WAF logs every injection attempt but forwards them all to the backend. The backend has no prepared statements and the database is compromised. The detection logs go unreviewed. SQL injection succeeds despite WAF presence High
2 Direct backend access bypasses WAF Backend accessible without going through Application Gateway App Service is accessible via its default azurewebsites.net URL in addition to the custom domain routed through Application Gateway WAF. An attacker discovers the direct URL by examining an old DNS record or certificate transparency log. They send attack traffic directly to app.azurewebsites.net, bypassing the WAF entirely. WAF bypass via direct backend URL High
3 SSL termination without end-to-end TLS Traffic from App Gateway to backend is plaintext Application Gateway terminates SSL but sends plaintext HTTP to the backend over the internal VNet. An insider with network packet capture capability can intercept the unencrypted backend traffic. Compliance requirements (e.g., PCI DSS) require encryption in transit, including backend hops. Compliance violation; insider traffic exposure Med
4 WAF false positives blocking legitimate traffic WAF rule overly broad or no tuning done A healthcare application uses URLs like /patient/record?id=1 OR active=1. The WAF matches the OR keyword and blocks the request as a SQL injection attempt. No WAF tuning or exclusions are applied. Legitimate application features become inaccessible to users in Prevention mode, effectively causing a self-inflicted denial of service. Service degradation from WAF false positives Med
5 Application Gateway subnet too small /28 or /29 subnet exhausted by autoscaling Application Gateway v2 in autoscaling mode requires multiple IP addresses for each scaling unit. A /28 subnet (11 usable IPs minus reserved) runs out of IPs as the gateway scales out under load. Scaling operations fail silently. The gateway cannot add capacity, and requests start timing out under peak load. Scaling failure under load Med
6 No custom WAF rules for application-specific attacks Only OWASP CRS rules — no business logic protection The OWASP CRS protects against generic web attacks. An attacker performs a credential stuffing attack against the login endpoint, sending thousands of valid-format login attempts. No rate limiting or custom WAF rule blocks the attack. The CRS sees normal-looking POST requests and allows them all through. Credential stuffing succeeds despite WAF High
7 Health probe checks root path of all backends Generic probe not reflecting real health The default health probe sends GET / to all backend instances. A backend API server is running but its database connection pool is exhausted, causing /api/* to return 503 while GET / returns 200 (static HTML). The Application Gateway considers the backend healthy and continues routing API traffic to the degraded instance. Unhealthy backends flagged as healthy Med
8 Large file uploads consume all capacity units Unrestricted request body size Application Gateway WAF inspects request bodies up to 128 KB by default. An attacker sends thousands of large file uploads (e.g., 64 MB each) to the upload endpoint. Each request consumes many capacity units during inspection. The gateway exhausts available capacity, causing legitimate traffic to be queued or dropped. Resource exhaustion via large requests Med
9 Listener on port 80 with no HTTP-to-HTTPS redirect Cleartext traffic accepted Application Gateway is configured with listeners on both port 80 and 443. No redirect rule is configured on the port 80 listener. Users and automated tools accessing the site over HTTP receive pages and may submit credentials in plaintext. The absence of HSTS and forced HTTPS allows downgrade attacks. Credential theft via cleartext HTTP High
10 Certificate expiry takes down HTTPS listener No certificate rotation automation An SSL certificate uploaded to Application Gateway expires. The HTTPS listener stops accepting connections. The certificate was manually uploaded and no renewal alert was configured. Users see SSL errors; the site is inaccessible over HTTPS. Manual renewal and re-upload take several hours. HTTPS outage from certificate expiry Med

Mitigation Controls

# Failure Mitigation Layer
1 WAF detection only Set WAF policy to Prevention mode after initial tuning; alert on WAF blocks WAF mode
2 Backend bypass Restrict App Service to accept traffic only from Application Gateway IP using access restrictions or service tags Backend lockdown
3 No end-to-end TLS Configure backend HTTP settings with HTTPS and trusted root certificate for end-to-end TLS Transport security
4 WAF false positives Tune WAF rules; use per-URI exclusions; test in Detection mode before switching to Prevention WAF tuning
5 Small subnet Use /24 subnet for Application Gateway; account for autoscaling IP requirements Subnet sizing
6 No rate limiting Add custom WAF rules for rate limiting per source IP on sensitive endpoints Custom WAF rules
7 Generic health probe Create custom health probes targeting meaningful health check endpoints Health probe config
8 Large upload resource exhaustion Configure max request body size limit; use Azure Blob Storage with SAS for large uploads instead of routing through App Gateway Request size limits
9 Port 80 without redirect Add redirect rule from port 80 listener to HTTPS; set HSTS header via rewrite rules HTTPS enforcement
10 Certificate expiry Use Key Vault-integrated certificates with automatic rotation via managed identity Certificate lifecycle

Top 3 Critical Mitigations

  1. Prevention Mode + Backend Lockdown: WAF in Detection mode is not protection — it is logging. Switch to Prevention after tuning, and restrict backend access to App Gateway IPs only.
  2. End-to-End TLS: Terminate SSL at App Gateway for inspection, but re-encrypt backend traffic. Plaintext internal traffic violates compliance requirements and insider threat models.
  3. Custom WAF Rules for Business Logic: OWASP CRS covers generic attacks; add custom rules for rate limiting, geo-restriction, and application-specific attack patterns.

Key Takeaways

  • Always deploy WAF in Prevention mode after tuning; Detection mode provides no protection, only logging.
  • Lock down backends to accept traffic only from Application Gateway — WAF bypass is a critical risk.
  • Use Key Vault-integrated certificates for automatic SSL rotation to prevent expiry outages.
  • Path-based routing enables microservices-style backend decomposition behind a single entry point.
  • Application Gateway subnet should be /24 to accommodate autoscaling IP allocation.
