Azure DNS

Category: Networking
Azure Service Type: Managed DNS
Key Docs: Azure DNS Overview

Overview

Azure DNS provides managed hosting for DNS zones (public and private) on Microsoft's global anycast infrastructure. Public DNS zones host internet-resolvable records for custom domains. Private DNS zones provide name resolution within VNets without exposing records publicly and are critical for private endpoint resolution. Azure DNS is not a domain registrar — domains must be registered elsewhere and delegated via NS records. Azure Private DNS Resolver extends resolution to on-premises networks, bridging hybrid cloud DNS architectures. DNS is a critical infrastructure component; misconfiguration leads to service outages, data exfiltration, and phishing attacks.

How It Works

Core Concepts

Concept Description
Public DNS Zone Internet-resolvable zone hosted by Azure; delegates from domain registrar via NS records
Private DNS Zone VNet-scoped zone; resolved only within linked VNets
VNet Link Associates a private DNS zone with one or more VNets for resolution
Auto-registration Automatically creates DNS A records for VMs deployed in linked VNets
Private Endpoint DNS Private DNS zone (e.g., privatelink.blob.core.windows.net) maps PE to private IP
DNS Resolver Inbound Endpoint Receives DNS queries from on-premises networks forwarded to Azure
DNS Resolver Outbound Endpoint Forwards DNS queries from Azure to on-premises or external resolvers
Record Sets A, AAAA, CNAME, MX, TXT, NS, SOA, SRV, PTR records
Alias Record Azure-specific record type pointing to an Azure resource (LB, Front Door, Traffic Manager)
TTL Time-to-live in seconds; affects how long resolvers cache records

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Subdomain takeover — dangling CNAME CNAME points to deleted resource A CNAME record api.contoso.com points to contoso-api.azurewebsites.net. The App Service is deleted but the DNS CNAME record is not removed. An attacker registers the same App Service name and creates content at that URL. Users who visit api.contoso.com are served the attacker's content. The attacker can also obtain a TLS certificate for api.contoso.com via domain validation. Subdomain takeover — content injection and cert issuance High
2 Private DNS zone not linked to VNet Zone created but VNet link missing A private endpoint is created for Azure Key Vault. The private DNS zone privatelink.vaultcore.azure.net is set up but not linked to the app tier VNet. The app resolves the Key Vault FQDN to the public IP. Traffic goes over the internet and the Key Vault firewall blocks it (or allows it publicly, defeating the purpose of the private endpoint). Private endpoint traffic bypassed High
3 DNS hijacking via compromised resolver On-premises DNS server compromised An on-premises DNS server that Azure VNets use as a custom resolver is compromised by an attacker. The attacker modifies DNS responses to redirect Azure service FQDNs to attacker-controlled IPs. Azure VMs resolve vault.azure.net to an attacker IP, and applications send credentials and secrets to the attacker. Credential and secret theft via DNS poisoning High
4 Low TTL causing excessive DNS queries TTL set to 0 or very low for flexibility All DNS records are configured with TTL=0 to allow instant propagation of changes. Every application request triggers a DNS resolution. At scale, the DNS server receives millions of queries per minute. This dramatically increases latency (DNS round-trip per request) and creates a single point of failure if the DNS service is unavailable. Performance degradation and DNS as SPOF Med
5 Wildcard record exposes unexpected subdomains Wildcard A record in public zone A wildcard A record *.contoso.com is added for convenience during development. It remains in production. Any subdomain under contoso.com resolves to the wildcard IP, including typos and unintended subdomains. Phishing actors register secure-login.contoso.com — which resolves to the wildcard IP. If the IP handles requests, it may serve the application unintentionally. Unintended subdomain resolution Med
6 SPF/DKIM/DMARC missing — email spoofing No email security DNS records The contoso.com DNS zone has no SPF TXT record, no DKIM public key record, and no DMARC policy. Attackers spoof @contoso.com email addresses to send phishing campaigns. Email clients accept and display the spoofed emails, leading to credential theft and BEC attacks. Email spoofing and phishing High
7 Auto-registration leaks internal hostnames VNet auto-registration enabled on public zone Auto-registration is configured against a private DNS zone, but a developer accidentally links a public DNS zone instead. All VMs in the VNet automatically register A records into the public DNS zone, exposing internal hostnames and private IP addresses to internet-wide DNS enumeration. Internal topology disclosure via DNS Med
8 NS delegation misconfiguration — zone not authoritative NS records pointing to wrong servers When delegating app.contoso.com to Azure DNS, the parent zone NS records are not updated correctly (pointing to a mix of old and Azure nameservers). Some resolvers use the old servers (returning NXDOMAIN or stale records) while others use Azure (returning current records). Service is partially available; some users see failures based on which resolver they use. Intermittent DNS resolution failures Med
9 Alias record not cleaning up on resource deletion Resource deleted but alias record persists An alias record points to a Front Door endpoint. Front Door is deleted (resource decommission). The alias record now points to a deleted resource. Azure handles alias record cleanup, but if the record is a regular CNAME (not alias), the dangling CNAME creates a subdomain takeover risk. Dangling record after resource deletion Med
10 No DNS logging — exfiltration over DNS undetected DNS queries not monitored An attacker who compromises a VM inside the VNet uses DNS tunneling to exfiltrate data. They encode data in subdomains of an attacker-controlled domain (data-chunk.attacker.com) and resolve them repeatedly, sending data out through the DNS resolution path. No DNS query logging is enabled. The exfiltration runs undetected for weeks. Data exfiltration via DNS tunneling Med

Mitigation Controls

# Failure Mitigation Layer
1 Subdomain takeover Audit and remove dangling CNAME records when resources are deleted; use Azure DNS alias records DNS hygiene
2 VNet link missing Verify private DNS zone VNet links exist for all VNets requiring PE resolution; automate with IaC DNS zone configuration
3 Compromised DNS resolver Use Azure Private DNS Resolver instead of custom DNS VMs; enable DNSSEC where supported Resolver hardening
4 Low TTL exhaustion Set TTL ≥ 300 seconds for production records; use low TTL only during planned cutover windows TTL management
5 Wildcard records Avoid wildcard A records in public zones; use explicit records per subdomain Zone record hygiene
6 Missing email security records Add SPF, DKIM, and DMARC TXT records; set DMARC policy to p=quarantine or p=reject Email security
7 Auto-registration leakage Enable auto-registration only on private DNS zones; never on public zones Auto-registration config
8 NS delegation errors Verify NS delegation in parent zone matches Azure DNS nameservers exactly Zone delegation
9 Alias vs CNAME for Azure resources Use alias record sets for Azure resources (Front Door, LB, Traffic Manager) for safe cleanup Alias record use
10 No DNS logging Enable Azure DNS diagnostic logging to Log Analytics; alert on unusual query patterns and high-volume DNS subdomains DNS logging

Top 3 Critical Mitigations

  1. Audit Dangling CNAME Records: Subdomain takeover via dangling CNAMEs is easy to execute and difficult to detect. Regularly audit DNS zones for records pointing to deleted or unowned resources.
  2. Private DNS Zone VNet Links: Missing VNet links cause private endpoints to resolve to public IPs, defeating the purpose of private networking. Validate all private DNS zones have correct VNet links in IaC.
  3. SPF + DKIM + DMARC: Missing email security records allow any attacker to send emails spoofing your domain. Set DMARC policy to reject for all production domains.

Key Takeaways

  • Private DNS zones are essential for private endpoint name resolution; missing VNet links cause silent failures (traffic going to public IPs).
  • Dangling CNAME records enable subdomain takeovers — audit DNS zones regularly and use alias records for Azure resources.
  • Azure Private DNS Resolver is the preferred hybrid DNS solution, replacing custom DNS server VMs.
  • Always add SPF, DKIM, and DMARC records — missing email DNS security is exploited for phishing and BEC attacks.
  • Enable DNS diagnostic logging to detect DNS tunneling and data exfiltration patterns.
