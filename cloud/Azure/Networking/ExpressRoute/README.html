Azure ExpressRoute

Category: Networking
Azure Service Type: Dedicated Private Connectivity
Key Docs: ExpressRoute Overview

Overview

Azure ExpressRoute provides dedicated, private connectivity from on-premises networks to Azure datacenters through a connectivity provider or colocation facility. Unlike VPN, ExpressRoute traffic does not traverse the public internet — it travels over a private MPLS or Ethernet circuit. It offers higher bandwidth (50 Mbps to 100 Gbps), lower latency, and SLAs unavailable with internet-based VPN. ExpressRoute supports three peering types: Azure Private Peering (VNet access), Microsoft Peering (Microsoft 365 and Azure PaaS), and Global Reach (site-to-site through Microsoft backbone). Circuit redundancy with two BGP sessions to separate Microsoft Edge routers (MSEE) is mandatory for production.

How It Works

Core Concepts

Concept Description

Connectivity Provider Layer 2 or Layer 3 provider that establishes the physical circuit to Microsoft Edge
MSEE Microsoft Enterprise Edge — Microsoft's routers at peering locations
Azure Private Peering BGP peering for access to resources in Azure VNets (VM, AKS, private services)
Microsoft Peering BGP peering for Microsoft Online services (M365, Azure PaaS public endpoints)
Global Reach Connect two ExpressRoute circuits to route between on-premises sites via Microsoft backbone
ExpressRoute Gateway Gateway in GatewaySubnet that connects VNets to ExpressRoute circuits
UltraPerformance SKU 10 Gbps ExpressRoute Gateway for highest bandwidth workloads
FastPath Bypass ExpressRoute Gateway for data path — reduces latency and throughput limits
Zone-Redundant Gateway ErGw*AZ SKUs deployed across AZs for gateway redundancy

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Single BGP session — no redundancy One physical path, no failover An organization connects via a single ExpressRoute circuit with one BGP session to one MSEE. A fiber cut at the provider facility severs the circuit. All on-premises to Azure connectivity is lost. There is no fallback VPN. Recovery requires the provider to restore the physical circuit, which takes hours to days. Complete hybrid connectivity loss during circuit failure High
2 BGP route advertisement leaking RFC1918 to Microsoft Peering Incorrect routing configuration An engineer configures Microsoft Peering (intended for M365/PaaS) but accidentally advertises large private RFC1918 address blocks. Microsoft rejects these advertisements but the misconfiguration causes BGP session instability. During the instability, production traffic to Microsoft Online services is disrupted. Microsoft Peering disruption and service outage Med
3 ExpressRoute without backup VPN — provider maintenance No redundant path during planned maintenance The connectivity provider schedules maintenance and takes the ExpressRoute circuit down. No backup VPN exists. During the maintenance window (typically 1–4 hours), all on-premises to Azure traffic is interrupted. Batch jobs, database syncs, and file transfers fail. The maintenance was communicated but the organization had no contingency. Production outage during scheduled maintenance window High
4 Unencrypted traffic assumption ExpressRoute is not encrypted by default An organization assumes ExpressRoute traffic is encrypted because it travels over a private circuit. In reality, ExpressRoute does not encrypt traffic. A malicious insider at the connectivity provider or a colocation facility can intercept plaintext traffic on the provider MPLS network. Compliance frameworks (e.g., PCI DSS) require encryption in transit. Plaintext traffic exposure at provider layer Med
5 Global Reach enabling unintended inter-site traffic Global Reach creates full mesh Global Reach is enabled between three ExpressRoute circuits to allow site-to-site routing. This creates a full mesh: Site A can reach Site B, Site C can reach Site A, etc. A security incident at Site B allows an attacker to reach Site A's network over the ExpressRoute Global Reach mesh, even though Sites A and B should have no business connectivity. Unintended inter-site network access Med
6 ExpressRoute Gateway bottleneck All traffic through ERGW data path An ExpressRoute Gateway processes all VNet-to-on-premises traffic. At high bandwidth, the gateway becomes a bottleneck. A GW2 SKU supports 2 Gbps. A large overnight batch transfer of 5 Gbps from on-premises saturates the gateway, causing all other Azure-to-on-premises traffic to queue and drop. Gateway throughput bottleneck under heavy load Med
7 Private Peering advertising default route from Azure Azure advertises 0.0.0.0/0 to on-premises ExpressRoute private peering is misconfigured to advertise a default route from Azure to on-premises. On-premises routers accept the route and begin routing internet traffic through Azure. Internet-bound traffic from on-premises traverses the ExpressRoute circuit and Azure network, causing extreme latency and potential policy violations. On-premises internet traffic rerouted through Azure Med
8 BGP session timer misconfiguration — premature failover BGP holdtime too short BGP session timers are set with a very short holdtime (e.g., 3 seconds). A brief network blip causes the BGP holdtime to expire before connectivity is restored. BGP sessions drop and reconverge, causing a traffic outage during what was a transient event. Reconvergence with route advertisements takes 20–60 seconds. Spurious failover from short BGP timers Med
9 ExpressRoute circuit key leaked Service key exposed The ExpressRoute Service Key is embedded in a Terraform state file committed to a public GitHub repository. The service key allows the bearer to view circuit configuration. While it does not grant data plane access, the leaked key could allow an attacker to gather topology information (circuit bandwidth, peer AS, prefixes). Configuration data leakage Med
10 Provider SLA not meeting business RPO Provider circuit recovery time exceeds RPO The connectivity provider's SLA for circuit restoration is 24 hours. The business RPO for the application is 4 hours. During a major provider outage, the circuit is not restored within the business RPO. The organization has no documented fallback path. The gap between provider SLA and business SLA is not discovered until an incident occurs. Business continuity failure from SLA mismatch Med

Mitigation Controls

# Failure Mitigation Layer
1 Single circuit Provision two circuits from different providers or peering locations; use active-active BGP Redundancy
2 RFC1918 leak to Microsoft Peering Advertise only public IP prefixes on Microsoft Peering; validate with route filter BGP route policy
3 No backup VPN Configure site-to-site VPN as failover; use BGP route preferences to deprioritize VPN path Failover VPN
4 No encryption Enable MACsec (Layer 2) at colocation peering; or use IPsec over ExpressRoute at Layer 3 Encryption in transit
5 Global Reach mesh Scope Global Reach to specific circuit pairs; apply NSGs on VNets to control inter-site traffic Traffic policy
6 Gateway bottleneck Enable FastPath on Ultra Performance and ErGw3AZ SKUs; use appropriate gateway SKU Gateway sizing
7 Default route advertisement Use outbound route filters; validate that Azure is not advertising 0.0.0.0/0 on private peering BGP routing
8 Short BGP timers Use BGP holdtime ≥ 90 seconds; configure BFD for faster failure detection without false positives BGP configuration
9 Service key leakage Treat service key as a secret; store in Key Vault; rotate on exposure; audit Terraform state storage Secret management
10 Provider SLA mismatch Document provider SLA vs business RPO; require dual-provider for critical circuits; test failover regularly Business continuity

Top 3 Critical Mitigations

  1. Dual Circuit Redundancy: A single ExpressRoute circuit is a single point of failure. Deploy two circuits from different providers (or peering locations) for production connectivity. Test failover regularly.
  2. Encrypt ExpressRoute Traffic: ExpressRoute is private but not encrypted. For compliance-regulated data (PCI DSS, HIPAA), enable MACsec at colocation or IPsec over ExpressRoute to ensure encryption in transit.
  3. Backup VPN Failover: Always maintain a VPN Gateway as a lower-preference fallback path. Use BGP route preferences (lower AS path weight for VPN) to ensure ExpressRoute is preferred while VPN provides failover.

Key Takeaways

  • ExpressRoute is private but not encrypted — enable MACsec or IPsec for compliance workloads.
  • Two BGP sessions (to two MSEEs) are mandatory for redundancy; a single session means a single point of failure.
  • Always pair ExpressRoute with a VPN Gateway failover path; provider outages can take circuits down for hours.
  • FastPath bypasses the ExpressRoute Gateway for the data path — essential for high-bandwidth, low-latency scenarios.
  • Global Reach creates inter-site connectivity automatically; scope it carefully to avoid unintended cross-site network access.
