Azure Front Door

Category: Networking
Azure Service Type: Global CDN and L7 Load Balancer
Key Docs: Azure Front Door Overview

Overview

Azure Front Door is a globally distributed, anycast-based Layer 7 load balancer, CDN, and WAF running on Microsoft's global edge network (Points of Presence in 100+ locations). It accelerates application delivery by routing users to the nearest PoP, caching static content at the edge, and forwarding dynamic requests over Microsoft's optimized backbone to origins. Front Door Standard/Premium (unified offering) replaces Classic Front Door and Azure CDN from Microsoft. It provides global load balancing with health-based failover, WAF at the edge, Private Link origin connectivity, custom domains with managed TLS, and DDoS mitigation.

How It Works

Core Concepts

Concept Description
Endpoint Domain name (e.g., contoso.azurefd.net or custom domain) for client requests
Origin Group Set of origins (backends) for health-based routing and failover
Origin Backend: App Service, Azure Storage, Application Gateway, or any public IP/FQDN
Route Maps URL patterns to origin groups; defines caching and forwarding rules
WAF Policy DRS/OWASP rule sets at the edge; rate limiting; custom rules; geo-filtering
Health Probe HTTP/HTTPS probes to origin groups for failover detection
Caching Edge cache with TTL, query string handling, and cache purge
Private Link Origin Connect to origins (App Service, Storage, LB) without public IP via Private Link
Rules Engine Per-request routing, header manipulation, redirect, and cache behavior overrides
Anycast Client DNS resolves to nearest PoP; traffic enters Microsoft network at the edge

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Origin accessible without Front Door No IP allowlist on origin An origin App Service has no access restrictions and is accessible via its default azurewebsites.net URL. An attacker discovers the origin URL via certificate transparency logs or old DNS records. They send attack traffic directly to the origin, bypassing the Front Door WAF, DDoS protection, and edge caching. The origin is directly exposed to brute force and SQL injection. Full WAF and DDoS bypass High
2 WAF in Detection mode Default mode after initial deployment Front Door WAF is deployed in Detection mode, logging but not blocking attacks. An attacker runs a credential stuffing campaign against the login API endpoint. The WAF logs 50,000 login attempts but forwards them all to the origin. No alert triggers on the detection log. The attack succeeds in compromising multiple accounts. Attack succeeds despite WAF High
3 No geo-restriction — attacks from unused regions WAF has no geo-blocking The application serves only North American users. No geo-restriction rule exists. Attackers from Eastern Europe and Southeast Asia run automated scanning, credential stuffing, and API abuse campaigns. The attack traffic reaches origins and increases infrastructure costs and abuse logs. Exposure to global attack traffic Med
4 Health probe reveals sensitive paths Probe URL exposes internal endpoint The Front Door health probe is configured to check /health/detailed, which returns system information, version numbers, and dependency status. An attacker who discovers the health probe URL (via directory enumeration) gains intelligence about the application stack, including backend service names, versions, and internal hosts. Information leakage via health probe Med
5 Cache poisoning — malicious cache entry Incorrect cache key configuration Cache behavior includes URL query strings but not all varying headers. A response with a Set-Cookie header or varying Content-Type is cached and served to all users from the PoP cache. An attacker crafts a request that causes a malicious or incorrect response to be cached, which is then served to legitimate users from the edge. Cached malicious content served to users Med
6 DDoS exhausts origin capacity through cache bypass Attackers use unique query strings to bypass cache A cache hit serves from the edge. An attacker adds random query string parameters (?rand=<random>) to every request, causing every request to be a cache miss and forwarded to the origin. The origin is flooded with unique requests that cannot be served from cache. Front Door DDoS protection operates at the PoP level but requests are forwarded to the origin. Origin DDoS via cache bypass High
7 Managed TLS certificate not renewing Let's Encrypt or managed cert renewal failure Front Door manages TLS certificates for custom domains. A renewal fails due to a DNS validation error (CNAME not pointing correctly). The certificate expires. All HTTPS traffic to the custom domain fails with SSL errors. The application is inaccessible to all users. HTTPS outage from cert expiry Med
8 Session cookie not marked SameSite=None;Secure HTTPS session cookies require proper attributes Front Door terminates TLS and forwards requests. The origin sets session cookies without Secure and SameSite=None attributes. When the user's browser sees the cross-site cookie from the CDN domain, it may block or not send the cookie (SameSite policy). Users experience session loss and unexpected logouts. Session cookie loss from SameSite policy Med
9 Rules Engine misconfiguration — wrong origin group Path pattern overlap Two route rules exist: /api/* → API backend and /api/public/* → public backend. The more specific path is not evaluated first. Front Door routes /api/public/data to the API backend (which requires auth), returning 401 errors for public endpoints. The misconfiguration is not caught in testing because the same backend works. Routing errors from path rule overlap Med
10 No WAF rate limiting — API endpoint abused Origin API lacks throttling A public API endpoint /api/lookup has no rate limiting enforced by Front Door WAF or the origin. Automated bots query the API at thousands of requests per second for data scraping. The origin is overwhelmed, and the cost for the API calls increases significantly. Competitors scrape the data within hours. Data scraping and origin overload High

Mitigation Controls

# Failure Mitigation Layer
1 Direct origin access Restrict origin to Front Door service tag only; use Private Link origins in Premium SKU Origin lockdown
2 WAF detection only Switch WAF to Prevention mode after tuning; alert on detection logs in real time WAF mode
3 No geo-restriction Add WAF geo-filter rule for allowed countries; log but review blocked regions Geo-restriction
4 Health probe leakage Use a dedicated /healthz endpoint returning only 200 OK or 503; no diagnostic data Health probe hardening
5 Cache poisoning Define cache key rules explicitly; do not cache authenticated or user-specific responses Cache configuration
6 Cache bypass DDoS Normalize query strings in routing rules; use caching rules to ignore unknown parameters Cache bypass mitigation
7 TLS cert renewal failure Validate DNS CNAME points correctly; monitor certificate expiry with Azure Monitor Certificate monitoring
8 Cookie SameSite issue Set Secure, HttpOnly, and SameSite=None on cookies; configure via Rules Engine headers Cookie security
9 Rules Engine misconfiguration Test route rules thoroughly; use longer prefix match order; review with staging environment Route rule testing
10 No API rate limiting Add WAF rate limit custom rule per client IP per endpoint; implement origin-side throttling Rate limiting

Top 3 Critical Mitigations

  1. Origin Lockdown to Front Door Only: If origins are accessible without Front Door, all edge security is bypassed. Use service tag AzureFrontDoor.Backend in NSG/App Service access restrictions, and use Private Link origins for Premium.
  2. WAF Prevention Mode: Detection mode provides observability but no protection. Enable Prevention after testing to actually block attacks at the edge.
  3. Rate Limiting on API Endpoints: Without rate limits, any public endpoint is vulnerable to credential stuffing, data scraping, and resource exhaustion; configure WAF rate limiting rules for sensitive endpoints.

Key Takeaways

  • Front Door runs at the Microsoft edge (PoP network) — it provides global load balancing, CDN, WAF, and DDoS mitigation in one service.
  • Lock origins to Front Door only (service tag + Private Link); direct origin access is the most common Front Door security bypass.
  • WAF in Prevention mode is essential; rate limiting custom rules protect APIs from credential stuffing and scraping.
  • Private Link origins (Premium SKU) remove the need for public IPs on origins entirely.
  • Front Door manages TLS certificates — monitor DNS CNAME validity to avoid cert renewal failures.
