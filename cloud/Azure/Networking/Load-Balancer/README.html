Azure Load Balancer

Category: Networking
Azure Service Type: L4 Load Balancing
Key Docs: Azure Load Balancer Overview

Overview

Azure Load Balancer operates at Layer 4 (TCP/UDP) and distributes inbound traffic across healthy backend pool instances using hash-based and session persistence algorithms. It is offered in two SKUs: Standard (recommended) and Basic (legacy). Standard Load Balancer supports zone-redundant frontends, global load balancing, HTTPS health probes, availability zones, outbound rules, and integration with NSGs. It comes in two types: public (internet-facing) and internal (private VNet traffic). Azure Load Balancer is the foundation for VM scale set traffic distribution and is used by AKS for service-type LoadBalancer.

How It Works

Core Concepts

Concept Description
Frontend IP Public or private IP that receives incoming traffic
Backend Pool Set of VMs, VMSS instances, or NICs that receive distributed traffic
Health Probe TCP, HTTP, or HTTPS probe to determine backend instance health
Load Balancing Rule Maps frontend IP:port to backend pool:port using a defined algorithm
Outbound Rule Configures SNAT for outbound internet connectivity from backend instances
Session Persistence None (default), Client IP, or Client IP and Protocol for sticky sessions
Standard SKU Zone-redundant, supports NSG, HTTPS probe, outbound rules
HA Ports Rule that load-balances all TCP and UDP ports simultaneously (internal LB only)
Global Load Balancer Cross-region load balancing for Standard LBs in multiple regions

    FrontendIP->>LB: Match load balancing rule

    LB->>Backend1: Forward TCP connection
    Backend1-->>Client: TCP handshake + response
    Note over LB: Health probe failure on Backend2
    LB->>LB: Remove Backend2 from rotation
    Client->>FrontendIP: New TCP connection
    LB->>Backend1: Route (Backend2 unhealthy)

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Health probe too lenient — unhealthy backends serving traffic Long probe interval with high threshold A backend VM is under extreme memory pressure. It responds slowly but does not fail the health probe (e.g., a 30-second interval with 3 failure threshold means 90 seconds to detect failure). During this window, users experience slow or failed connections routed to the degraded VM. Degraded user experience from unhealthy backend Med
2 No NSG on Standard Load Balancer Standard LB requires explicit NSG allow rules A Standard Load Balancer is deployed with no NSG on the backend subnet. Unlike Basic LB, Standard LB does not allow traffic by default — it requires explicit NSG inbound rules. The developer expects traffic to flow but all connections are dropped silently. The application is completely unreachable. Application unavailability from missing NSG High
3 SNAT port exhaustion Insufficient SNAT ports for outbound connections Backend VMs behind an internal load balancer make outbound internet connections using default SNAT (Azure platform SNAT). A high-concurrency workload establishes thousands of simultaneous outbound connections. Default SNAT provides limited ports per VM. Port exhaustion causes new outbound connections to fail with ECONNREFUSED or ETIMEDOUT, while existing connections are unaffected. Outbound connectivity failure under load High
4 Single-zone frontend — AZ failure Frontend IP in a single Availability Zone A public load balancer is configured with a zone-pinned frontend IP (Zone 1 only). When Azure Zone 1 experiences an outage, the frontend IP is unavailable even if backend VMs exist in Zones 2 and 3. The service is completely unreachable from the internet during the zone outage. Full service outage on AZ failure Med
5 Session persistence breaks autoscaling Client IP persistence routes to one backend Session persistence (Client IP mode) is enabled on a load balancer serving a stateless web application. A single large client (e.g., a corporate NAT gateway behind one public IP) sends all its traffic to one backend instance. That instance is overloaded while others are idle. Autoscaling adds new instances but the persistent client does not route to them. Uneven load distribution despite autoscaling Med
6 HA Ports rule on internal LB with no traffic filtering All ports forwarded to all backends An internal load balancer uses HA Ports to route all TCP and UDP traffic from a network virtual appliance subnet to backend firewalls. A misconfiguration also includes a standard load balancing rule on port 22 (SSH), inadvertently making SSH reachable via the load balancer. An internal attacker can reach the management port of all backend VMs. Unexpected management port exposure Med
7 Idle timeout causing connection drops Long-lived connections exceed idle timeout A database client uses connection pooling and holds idle connections open for 10 minutes. Azure Load Balancer has a default idle timeout of 4 minutes. The load balancer silently drops idle connections at 4 minutes. The application receives broken pipe errors and must re-establish connections frequently, impacting performance. Unexpected connection resets for long-lived connections Med
8 No outbound rule — default SNAT with Basic LB Migrating from Basic to Standard LB breaks SNAT An application migrates from Basic Load Balancer (which automatically provides SNAT) to Standard Load Balancer (which does not automatically provide outbound connectivity). After migration, all outbound internet connections from backend VMs fail. Third-party API calls, DNS resolution to external servers, and NTP synchronization stop working. Outbound connectivity loss post-migration High
9 UDP load balancing — no connection tracking UDP is stateless; LB cannot track sessions A UDP-based gaming or media application uses Azure Load Balancer for UDP distribution. Since UDP is stateless, load balancer session persistence is not effective. Different UDP packets from the same source IP+port may route to different backends (unless Client IP+Protocol persistence is set). The application breaks because stateful game logic expects all packets from a session to reach the same server. Application session break from stateless UDP Med
10 Diagnostics not enabled No visibility into LB health and SNAT metrics SNAT ports are exhausting, but no diagnostic metrics are configured. The team observes intermittent outbound failures but cannot identify the root cause. Without UsedSNATPortCount and AllocatedSNATPortCount metrics in Azure Monitor, the diagnosis takes days of manual testing instead of minutes. Extended MTTR from lack of observability Med

Mitigation Controls

# Failure Mitigation Layer
1 Lenient health probe Set health probe interval to 5s with unhealthy threshold of 2 for faster detection Health probe config
2 No NSG Add explicit NSG inbound allow rules for load-balanced ports on backend subnet NSG configuration
3 SNAT exhaustion Configure explicit outbound rules with NAT Gateway; or use Azure NAT Gateway for outbound SNAT configuration
4 Zone-pinned frontend Use zone-redundant frontend IP (no zone pinning) for production load balancers Zone redundancy
5 Session persistence breaks scaling Use None (hash-based) persistence for stateless apps; design apps to be session-agnostic Session persistence design
6 HA Ports overexposure Use HA Ports only on NVA bypass subnets; do not combine with other rules on the same LB Rule design
7 Idle timeout drops Increase idle timeout to 30 minutes; implement TCP keepalive in application Idle timeout config
8 No outbound rule after migration Create explicit outbound rule or use NAT Gateway before removing Basic LB SNAT planning
9 UDP session stickiness Use Client IP + Protocol session persistence for UDP workloads Session persistence
10 No diagnostics Enable Load Balancer diagnostic metrics in Azure Monitor; alert on SNAT exhaustion Observability

Key Takeaways

  • Azure Load Balancer is L4 (TCP/UDP); Application Gateway and Front Door provide L7 (HTTP/HTTPS) routing.
  • Standard LB requires explicit NSG rules — no default allow; this is a common migration gotcha from Basic.
  • Configure outbound rules or NAT Gateway for reliable SNAT; default SNAT exhausts under load.
  • Use zone-redundant frontend IPs (no zone pinning) for production load balancers.
  • Enable Azure Monitor diagnostics — SNAT exhaustion is diagnosable in seconds with the right metrics.
