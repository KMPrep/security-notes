Azure Virtual Network (VNet)

Category: Networking
Azure Service Type: Network Isolation and Connectivity
Key Docs: VNet Overview

Overview

Azure Virtual Network (VNet) is the fundamental networking primitive in Azure, providing isolated network environments for Azure resources. VNets enable communication between Azure resources (VMs, App Services, AKS), connectivity to on-premises networks (via VPN or ExpressRoute), and security through Network Security Groups (NSGs) and Azure Firewall. VNets are scoped to a region; resources in different regions connect via global VNet peering. Subnets segment a VNet into logical network zones. Service endpoints and private endpoints integrate PaaS services into the VNet fabric.

How It Works

Core Concepts

Concept Description
Address Space CIDR block assigned to the VNet (e.g., 10.0.0.0/16)
Subnet Subdivision of the VNet address space; resources deployed into subnets
NSG Stateful L4 security group applied at subnet or NIC level
UDR User-defined route overriding Azure's default routing
VNet Peering Direct, low-latency connectivity between VNets (same or different regions)
Service Endpoint Extends VNet identity to Azure PaaS services; traffic stays on Azure backbone
Private Endpoint Private IP in the VNet pointing to a specific PaaS service resource
VNet Integration Allows App Service, Functions, ACI to route outbound traffic into a VNet
Bastion Managed SSH/RDP gateway that eliminates public IP requirement on VMs
Network Watcher Monitoring and diagnostics for VNet connectivity and flow logging

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Flat VNet with no subnet segmentation All resources in a single subnet with permissive NSG A VNet uses a single subnet for all workloads. An attacker who compromises a web tier VM has direct network access to database VMs, management hosts, and all other resources. There are no NSG rules restricting east-west traffic. The attacker scans the 10.0.0.0/16 and connects directly to the database on port 1433 without any barrier. Unrestricted lateral movement across all tiers High
2 NSG rule allows all inbound (*:*:Allow) Permissive fallback rule added for troubleshooting An operator adds source: Any, destination: Any, port: *, action: Allow to resolve a connectivity issue during troubleshooting and forgets to remove it. The NSG effectively becomes a pass-through. Any internet host reaching the subnet can attempt connections on any port against any VM. All VMs exposed to internet scanning and attacks High
3 VNet peering without NSG between peered networks Peered networks treated as trusted Two VNets are peered for application connectivity. No NSG rules exist between the peered subnets. A developer workstation network is peered to the production network for debugging. A malware infection on a developer laptop scans the production network over the peering connection and reaches production databases directly. Developer network to production lateral movement High
4 Overlapping CIDR blocks — peering fails Address spaces conflict between peered VNets A new VNet is provisioned with the same address space (10.0.0.0/16) as an existing VNet. VNet peering requires non-overlapping address spaces. The peering fails. Services that rely on inter-VNet connectivity are unable to communicate. Fixing requires re-addressing the VNet, requiring downtime for all resources. Connectivity failure requiring re-architecture Med
5 Service endpoint bypass of firewall rules Service endpoint traffic bypasses NVA Storage account firewall is configured to allow only VNet service endpoint traffic from subnet X. An attacker who gains access to a VM in subnet X can reach the storage account directly via the service endpoint, bypassing any NVA or Azure Firewall that sits in the path. Service endpoints do not route through the default route to the NVA. NVA/firewall bypass for PaaS service access Med
6 Public IP on every VM Resources have public IPs for convenience Every VM in the VNet has a public IP assigned for SSH access. Each IP is an attack surface for scanning, brute force, and exploitation. The attack surface scales with the number of VMs. Even with NSG protection, public IPs enable direct DDoS targeting and contribute to the organization's public attack surface. Expanded public attack surface High
7 No VNet Flow Logs No visibility into network traffic patterns An attacker who gains access to a VM performs east-west scanning and lateral movement across subnets over several days. No network flow logs are enabled. The security team has no visibility into which IPs are communicating, what ports are in use, or what volume of traffic is flowing between tiers. No forensic evidence of lateral movement Med
8 UDR misconfiguration — traffic blackholed UDR next-hop points to inactive NVA A UDR routes all traffic through an NVA for inspection. The NVA VM is stopped for maintenance. All traffic routed through the NVA is silently dropped. Applications in the affected subnet lose all connectivity. No alert exists for NVA availability. Complete connectivity failure for subnet High
9 CIDR too small — subnet exhaustion Subnet /28 fills up, blocking new deployments A subnet is provisioned as /28 (14 usable IPs, minus Azure reserved = 11). Resource deployments consume all available IPs. New VMs, private endpoints, and managed services cannot be added. Azure reserves 5 IPs per subnet; planning must account for this. Deployment failures from IP exhaustion Med
10 DNS resolution failure — private endpoint not resolving Private DNS zone not linked to VNet A private endpoint for Azure SQL is created. The private DNS zone privatelink.database.windows.net is not linked to the app tier VNet. The app resolves the SQL FQDN to the public IP, bypassing the private endpoint. Traffic goes over the internet rather than the private path. Private endpoint traffic bypassed Med

Mitigation Controls

# Failure Mitigation Layer
1 Flat VNet Design with tiered subnets (web, app, data, mgmt); apply NSG with explicit allow rules and deny-all default Subnet segmentation
2 Permissive NSG rule Use Azure Policy to detect and deny *:*:Allow rules; periodically audit NSG rules NSG governance
3 Peering without NSG Apply NSGs on each peered subnet; use Azure Firewall in hub VNet for cross-peering inspection Hub-spoke architecture
4 Overlapping CIDR Plan and document VNet CIDR ranges before provisioning; use IPAM tooling IP address management
5 Service endpoint bypass Use private endpoints instead of service endpoints for services requiring firewall control Private endpoint preference
6 Public IPs on VMs Remove public IPs; use Azure Bastion for SSH/RDP; use private endpoints for all service connectivity Public IP governance
7 No flow logs Enable NSG Flow Logs v2 (with Traffic Analytics) to Log Analytics; enable Network Watcher Network observability
8 UDR with inactive NVA Monitor NVA health; use zone-redundant NVA; set up health probes with automatic failover NVA high availability
9 Subnet IP exhaustion Use /24 or larger subnets; plan for 2x expected resource count Subnet sizing
10 Private DNS not linked Link private DNS zones to all VNets requiring private endpoint resolution DNS configuration

Key Takeaways

  • VNet is the foundation of Azure network isolation; always design with tiered subnets and explicit NSG rules.
  • Private endpoints are preferred over service endpoints — they route through the VNet and support Azure Firewall.
  • Hub-spoke topology with Azure Firewall centralizes egress inspection and east-west traffic control.
  • Enable NSG Flow Logs and Network Watcher for all production VNets — essential for threat detection and forensics.
  • Never assign public IPs to VMs; use Azure Bastion for management access.
