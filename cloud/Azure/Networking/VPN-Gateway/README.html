Azure VPN Gateway

Category: Networking
Azure Service Type: VPN Connectivity
Key Docs: VPN Gateway Overview

Overview

Azure VPN Gateway provides IPsec/IKE VPN tunnels connecting Azure VNets to on-premises networks (site-to-site), remote users (point-to-site), or other Azure VNets (VNet-to-VNet). It is deployed in the GatewaySubnet of a VNet and runs as a managed active-passive or active-active pair. SKUs range from Basic (legacy, no SLA) to VpnGw5AZ (zone-redundant, 10 Gbps, BGP). BGP (Border Gateway Protocol) enables dynamic routing and failover. Point-to-site (P2S) VPN supports IKEv2, OpenVPN, and SSTP and is commonly used for remote workforce access with Azure AD or certificate-based authentication.

How It Works

Core Concepts

Concept Description
GatewaySubnet Dedicated subnet (GatewaySubnet) required for VPN Gateway deployment
S2S VPN Site-to-site IPsec tunnel connecting on-premises VPN device to Azure VPN Gateway
P2S VPN Point-to-site — individual clients connect via IKEv2, OpenVPN, or SSTP
BGP Dynamic routing protocol for automatic route advertisement and failover
Active-Active Both VPN Gateway instances are active; higher availability and throughput
Active-Passive One instance active, one standby; 10–15 second failover for planned maintenance
Local Network Gateway Azure resource representing on-premises VPN device (IP, BGP ASN, address space)
Connection Links a VPN Gateway to a Local Network Gateway with a shared key or certificate
Zone-Redundant SKU VpnGw*AZ SKUs deployed across Availability Zones for zone-redundant VPN
VNet-to-VNet IPsec tunnel between two Azure VNets (alternative to VNet peering)

    VNet-->>VPNGW: Response
    VPNGW-->>OnPrem: Encrypted response
    Note over OnPrem,VPNGW: BGP keepalives maintain routing

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Weak PSK — brute force attack Simple shared key for S2S VPN A site-to-site VPN connection uses a simple shared key like Azure123!. An attacker who intercepts IKE negotiation packets (if UDP 500/4500 is not filtered) can perform offline brute force to recover the PSK. Once the PSK is known, the attacker can establish a rogue VPN connection from any IP and gain network-level access to the Azure VNet. Unauthorized VPN access from attacker Med
2 Overly broad address space advertised All RFC1918 space in address selectors The Local Network Gateway address space is configured as 0.0.0.0/0 or overly broad CIDR. When the VPN tunnel is established, all traffic from the Azure VNet is routed to the on-premises network. Azure-to-Azure PaaS traffic may be routed on-premises and broken. DNS queries, Windows Update, and Azure service traffic detour through the corporate network, causing failures and latency. Unintended traffic routing and connectivity failures Med
3 Active-passive failover during patching 10–15 second failover gap Azure VPN Gateway undergoes mandatory maintenance and fails over from the active to passive instance. The failover takes 10–15 seconds. BGP sessions drop, and on-premises routing tables take additional time to converge. During this window, all S2S VPN traffic is dropped. Applications with short TCP timeouts fail and must retry connections. 10–15 second connectivity disruption during maintenance Med
4 GatewaySubnet too small /29 subnet running out of IPs The GatewaySubnet is provisioned as /29 (3 usable IPs after Azure reserves). Azure requires at least 2 IPs for the VPN Gateway pair (active-active = 2). Future gateway upgrades or ExpressRoute co-existence require additional IPs. Migrating to a larger GatewaySubnet requires deleting and recreating the gateway, causing prolonged VPN downtime. Downtime during GatewaySubnet resize Med
5 P2S VPN — certificate not revoked No CRL or revocation of compromised certificates A remote employee's laptop with a P2S VPN certificate is stolen. The certificate is not revoked in the VPN Gateway revocation list. The attacker uses the stolen certificate to authenticate P2S VPN and gains access to the Azure VNet. Access continues until the certificate's natural expiry (up to 1–2 years). Unauthorized P2S VPN access from stolen device High
6 No MFA on P2S VPN — stolen credentials P2S with username/password only P2S VPN is configured with SSTP and RADIUS authentication using AD username/password. An attacker obtains an employee's credentials via phishing. They authenticate the P2S VPN tunnel using the stolen credentials and gain full VNet access. No MFA is required for VPN authentication. Network access from phished credentials High
7 VPN Gateway Basic SKU — no SLA Legacy Basic SKU used The VPN Gateway is deployed with the Basic SKU, which provides no SLA and is not recommended for production workloads. During an Azure platform event, the Basic SKU gateway becomes unavailable. The organization has no SLA recourse and no zone redundancy. The gateway also cannot be upgraded in-place to a higher SKU — a new gateway must be created. No availability SLA; gateway replacement required for upgrade Med
8 No monitoring of VPN tunnel state Tunnel drops unnoticed An S2S VPN tunnel goes down due to a DPDK or on-premises device issue. No Azure Monitor alert is configured for tunnel state. The on-premises network and Azure VNet are disconnected. Applications that depend on on-premises databases or domain controllers fail silently. The team discovers the outage only when a user reports application failures hours later. Extended outage from undetected tunnel drop High
9 IKE/IPsec using weak cipher suite Default or legacy cipher negotiation The on-premises VPN device negotiates IKE Phase 1 using DES or 3DES with MD5 authentication. Azure VPN Gateway allows these legacy ciphers if the on-premises device proposes them first. An attacker with long-term traffic capture can decrypt the VPN traffic offline using the weak algorithm. VPN traffic decrypted via weak cipher Med
10 Split tunneling leaks corporate traffic P2S VPN with split tunnel and no policy P2S VPN is configured with split tunneling, routing only Azure traffic through the VPN. A remote user accesses a corporate SharePoint instance (hosted in Azure) directly over the VPN but also accesses the internet directly. A DNS rebinding attack on a malicious website causes the user's browser to make requests to the corporate SharePoint. The corporate traffic is exposed to the attacker-controlled website. Data leakage through split tunnel Med

Mitigation Controls

# Failure Mitigation Layer
1 Weak PSK Use a long random PSK (≥ 32 characters); store in Key Vault; rotate periodically Authentication
2 Overly broad address space Define precise address prefixes in Local Network Gateway; avoid 0.0.0.0/0 Routing precision
3 Failover gap Use active-active mode with BGP for sub-second failover; implement TCP keepalive High availability
4 Small GatewaySubnet Provision GatewaySubnet as /27 minimum (/26 recommended for future expansion) Subnet sizing
5 No cert revocation Maintain CRL; upload revoked certificate thumbprints to VPN Gateway revocation list immediately on device loss Certificate revocation
6 No MFA on P2S Use OpenVPN with Azure AD authentication and Conditional Access MFA for P2S MFA for VPN
7 Basic SKU Use VpnGw2 or higher for production; VpnGw*AZ SKUs for zone-redundant VPN SKU selection
8 No monitoring Create Azure Monitor alert for TunnelIngressBytes and TunnelEgressBytes metrics; alert on zero bytes for 5 minutes VPN monitoring
9 Weak cipher suite Configure custom IPsec/IKE policies on Azure side; specify AES-256 + SHA-256/384 + DH Group 14/21 Cipher policy
10 Split tunnel risk Disable split tunneling for sensitive workloads; force all traffic through VPN (force tunnel) Traffic routing policy

Top 3 Critical Mitigations

  1. MFA on P2S VPN via Azure AD + Conditional Access: Password-only VPN authentication is one phished credential away from full network compromise. Azure AD-integrated P2S OpenVPN supports Conditional Access MFA.
  2. Active-Active with BGP: Active-passive mode has a 10–15 second failover gap. Active-active with BGP provides near-seamless failover and higher throughput for production VPN.
  3. Custom IPsec/IKE Policy: Do not accept legacy cipher proposals from on-premises devices. Explicitly configure AES-256/SHA-256 cipher suites on the Azure connection to prevent weak cipher negotiation.

Key Takeaways

  • Use active-active mode with BGP for production VPN — active-passive has a 10–15 second failover gap.
  • Enforce MFA for P2S VPN via Azure AD + Conditional Access; certificate or password alone is insufficient.
  • Provision GatewaySubnet as /27 minimum — resizing requires gateway recreation and downtime.
  • Configure explicit IPsec/IKE policies with AES-256 — never rely on default weak cipher negotiation.
  • Monitor tunnel state with Azure Monitor — silent tunnel drops cause extended outages without alerting.
