Azure Monitor

Category: Operations
Azure Service Type: Observability Platform
Key Docs: Azure Monitor Overview

Overview

Azure Monitor is the unified observability platform for Azure, collecting metrics, logs, and traces from Azure resources, applications, VMs, containers, and on-premises systems. It consists of several integrated components: Metrics (time-series numerical data), Log Analytics (log storage and KQL querying), Application Insights (APM for web applications), Azure Monitor Agent (AMA, replacing older MMA/OMS agents), Alerts (metric, log, activity log, and smart detection), Dashboards and Workbooks (visualization), and Autoscale. Azure Monitor is the data backbone for Defender for Cloud, Microsoft Sentinel, and Azure Service Health. Proper observability is a security requirement — you cannot detect or respond to incidents you cannot see.

How It Works

Core Concepts

Concept Description
Metrics Numerical time-series data collected every 1 minute; stored for 93 days
Logs Structured records in Log Analytics; queried with KQL; long-term retention
Application Insights APM SDK for web apps — request, dependency, exception, and performance tracking
Azure Monitor Agent Unified agent replacing MMA; collects logs and metrics from VMs
Data Collection Rule Defines what data to collect, transform, and where to send (AMA)
Action Group Notification and action set triggered by an alert (email, SMS, webhook, Logic App)
Alert Rule Triggers on metric threshold, log query result, activity log event, or resource health
Workbook Interactive parameterized reports combining metrics, logs, and visuals
Autoscale Rule-based or predictive scaling of resources based on metric thresholds
Diagnostic Settings Route resource platform logs and metrics to Log Analytics, Storage, or Event Hub

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Diagnostic settings not configured Resources don't emit logs VMs, Key Vaults, NSGs, and Azure SQL databases are deployed without diagnostic settings. No logs flow to Log Analytics. An attacker compromises a VM and performs lateral movement over 3 days. The security team has no visibility into authentication events, process execution, or network flows from any affected resource. Zero visibility — incident undetectable High
2 Alert rules not tested — silent failures Alerts created but never validated Alert rules are created for critical metrics (high error rate, CPU > 90%, disk full). The alert email goes to a distribution list that was decommissioned. The action group's webhook endpoint returns 404. When CPU spikes and an application fails, no alert fires. The issue is discovered 2 hours later via user complaints. Alerts exist on paper but never fire High
3 Log Analytics workspace retention too short Default 30 days used Log Analytics workspace retention is left at the default 30 days. An intrusion is discovered 45 days after it began. The earliest evidence of the initial access (Day 1 logs) has already been purged. The incident response team cannot establish how the attacker entered, what credentials were used, or what data was accessed. Forensic investigation blocked by log expiry High
4 Single workspace for all environments Dev and prod logs mixed A single Log Analytics workspace collects logs from all environments (dev, staging, prod). Developers have read access to the workspace for debugging. Production security logs (Key Vault access, admin authentication, Defender alerts) are visible to developers who have no business need to see them. An insider reads production authentication logs to plan an attack. Production security data accessible to unauthorized users Med
5 No resource health alerting Service health events go unnoticed An Azure region experiences a partial outage affecting Storage. No Azure Service Health alert is configured. The team discovers the outage only when applications start failing. An automated alert would have notified on-call engineers 20 minutes earlier, reducing customer impact by allowing proactive failover to the secondary region. Delayed incident response from no health alerting Med
6 Application Insights sampling too aggressive Important traces dropped Application Insights is configured with 10% sampling to reduce costs. A critical exception occurs in a payment processing function. Due to sampling, 90% of exception instances are not recorded. The error is reported by users but only a small fraction of traces are available for investigation. The root cause is difficult to determine from incomplete trace data. Incomplete diagnostics from over-sampling Med
7 No autoscale — resource exhaustion Manual scaling cannot keep up A web application does not have autoscale configured. A marketing campaign drives 10x normal traffic. The fixed number of App Service instances is overwhelmed. Response times degrade, then the service returns 503 errors. Manually scaling takes 10 minutes. Autoscale would have added instances proactively based on CPU or request metrics before saturation occurred. Service outage from traffic spike without autoscale High
8 Alert storm — too many low-quality alerts Poorly tuned alert thresholds Hundreds of alerts are configured on raw metrics without smoothing (e.g., alert when any single CPU reading exceeds 80%). During normal workload spikes, hundreds of alerts fire simultaneously. The on-call engineer receives a flood of notifications and begins ignoring them (alert fatigue). A real P1 incident alert is lost in the noise. Real alerts missed due to alert fatigue High
9 No cost alerts — surprise bill Monitor budget not configured Azure Monitor's Cost Management integration is not used. An Azure Cognitive Services API key is leaked in a public GitHub repository. Bots immediately begin making API calls against the key. Within 72 hours, API usage charges of $50,000 are incurred. No budget alert exists to notify the team when spending exceeds $1,000. Large unexpected cloud bill from credential leak High
10 Azure Monitor Agent not deployed on all VMs Coverage gaps in log collection Azure Monitor Agent collects security events (Windows Security Event logs, Linux syslog) from VMs. Several older VMs were provisioned before the AMA deployment standard was established. These VMs emit no logs to Log Analytics. An attacker who compromises one of these unmonitored VMs operates without any visibility from the monitoring platform. Unmonitored VMs create blind spots Med

Mitigation Controls

# Failure Mitigation Layer
1 No diagnostic settings Use Azure Policy with DeployIfNotExists to auto-provision diagnostic settings on all resources Diagnostic coverage
2 Untested alerts Test alerts monthly using the Test action group button; validate endpoint availability Alert testing
3 Short retention Set security table retention to 1–2 years; use Archive tier for older logs Log retention
4 Mixed workspace Use separate Log Analytics workspaces per environment; apply RBAC to restrict access Workspace isolation
5 No health alerting Configure Azure Service Health alerts for all subscriptions and critical service types Health alerting
6 Over-sampling Set Application Insights sampling to 100% for critical paths (payments, auth); use adaptive sampling on others Sampling configuration
7 No autoscale Configure autoscale rules on App Service, VMSS, and AKS node pools; test with load testing Autoscale
8 Alert storm Use dynamic thresholds or smoothed average (5-minute average CPU) instead of instantaneous metrics Alert tuning
9 No cost alerts Configure Azure Cost Management budget alerts at 80% and 100% of monthly budget per subscription Cost alerting
10 AMA coverage gaps Enforce AMA deployment via Azure Policy; audit VM coverage with Azure Monitor Insights Agent coverage

Key Takeaways

  • Diagnostic settings must be explicitly configured on each resource — use Azure Policy to enforce at scale.
  • Log retention of 30 days is insufficient for security investigations; set security tables to 1–2 years.
  • Test alert action groups regularly — silent alert failures are a common gap found in post-incident reviews.
  • Separate Log Analytics workspaces per environment and apply RBAC — production security logs should not be visible to all developers.
  • Alert tuning is ongoing work — use dynamic thresholds and aggregated metrics to reduce alert fatigue while maintaining detection quality.
