Azure Policy

Category: Operations / Governance
Azure Service Type: Cloud Governance
Key Docs: Azure Policy Overview

Overview

Azure Policy is a governance service that enforces organizational standards and assesses compliance at scale across Azure resources. Policies are JSON rules evaluated by Azure Resource Manager at resource creation and update time. They can Audit (report non-compliance), Deny (block non-compliant deployments), DeployIfNotExists (auto-remediate by deploying required configurations), Modify (change resource properties), and Append (add required properties). Policies are grouped into Initiatives (also called Policy Sets). Assignments scope policies to Management Groups, Subscriptions, or Resource Groups. Azure Policy is the primary mechanism for enforcing security baselines, compliance frameworks, and organizational guardrails across an Azure environment.

How It Works

Core Concepts

Concept Description
Policy Definition JSON rule with condition and effect (Audit, Deny, DINE, Modify, Append)
Initiative Group of related policies (e.g., PCI DSS Initiative = 200+ policies)
Assignment Apply a policy/initiative to a scope (MG, subscription, resource group)
Compliance State Compliant / Non-compliant / Exempt / Unknown — for each resource
Deny Effect Block resource creation or update if it violates the policy condition
DeployIfNotExists Deploy a companion resource when a target resource doesn't have the required config
Remediation Task Manually or automatically trigger DINE policies on existing non-compliant resources
Exemption Exclude specific resources or resource groups from a policy assignment
Built-In Policies Microsoft-maintained policies for common security and compliance controls
Regulatory Compliance Built-in initiatives for PCI DSS, HIPAA, NIST SP 800-53, CIS, ISO 27001

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Policies in Audit mode only — non-compliance never remediated Audit effect without enforcement plan All Azure Policy assignments use Audit effect to avoid breaking existing deployments. Non-compliant resources accumulate. A policy requiring private endpoints on storage accounts has been in Audit mode for 8 months. 150 storage accounts with public access remain non-compliant. The compliance dashboard shows 23% compliance — but nothing is actually enforced. An attacker accesses data from a publicly exposed storage account. Security controls never enforced despite policy High
2 Policy exemptions without expiry or justification Blanket exemptions for "legacy" resources Policy exemptions are created for production resource groups during initial deployment (to allow migration time). The exemptions have no expiry date and no documented justification. Two years later, no one knows why the exemptions exist. Legacy resources remain non-compliant and exempt. The organization fails a PCI DSS audit because exempted resources are in scope for cardholder data. Compliance gaps from stale exemptions High
3 No policy at Management Group scope Each subscription has independent policies Each Azure subscription has independent policy assignments. New subscriptions provisioned for new projects start with no policies. A new subscription for a development project is created by a developer. There are no policies requiring encryption, private endpoints, or MFA. Insecure configurations are deployed and go undetected until a quarterly review. New subscriptions start with no guardrails High
4 DINE policy with insufficient RBAC Remediation identity lacks permissions A DeployIfNotExists policy is configured to auto-deploy Azure Monitor diagnostic settings on all storage accounts. The policy's managed identity has only Reader role, not Monitoring Contributor. When a new storage account is created, the DINE policy attempts to deploy the diagnostic settings but fails with a permissions error. The remediation silently fails. The resource remains without diagnostic settings. Auto-remediation silently fails — resource left unmonitored Med
5 Deny policy blocks legitimate deployments Over-broad policy condition A Deny policy requires all VMs to have a specific tag (Environment: Production). A CI/CD pipeline deploys VMs for an integration testing environment. The policy blocks the deployment because the test VMs don't have the Production tag (correct behavior), but no Audit/exception path exists for non-production VMs. The pipeline fails and the team disables the policy assignment to unblock CI. Legitimate deployments blocked; policy disabled as workaround Med
6 Regulatory initiative not customized to actual scope PCI DSS initiative applied to all resources The PCI DSS regulatory initiative is assigned at the subscription scope. The subscription hosts both cardholder data environment (CDE) resources and non-CDE internal tools. PCI controls for CDE (e.g., network segmentation, encryption) are applied to internal tools that have no PCI scope. The policy generates thousands of non-compliance findings for resources out of scope. The real CDE compliance findings are buried in noise. Compliance findings diluted across non-scoped resources Med
7 No policy for resource naming convention Resources named inconsistently No policy enforces resource naming conventions. Resources are named inconsistently across teams (e.g., prod-db-01, database-production, SQLDB-PROD). Automated monitoring and alerting rules that match on resource names break when new resources are created with inconsistent names. Incident response is slowed because resource ownership is unclear from resource names alone. Operational confusion and monitoring gaps Med
8 Policy compliance data stale Policy evaluation lag Azure Policy evaluates resources on a 24-hour cycle (or on-demand trigger). A storage account is misconfigured at 9 AM. The policy compliance state does not update until the next evaluation cycle at 3 AM the next day. Security tools that rely on Policy compliance state for reporting show the resource as compliant for 18 hours. 18-hour compliance blindspot after misconfiguration Med
9 Custom policy with logic error — all resources non-compliant Policy condition incorrectly written A custom policy intended to require HTTPS on App Services has a logic error in the condition (wrong comparison operator). The policy marks all App Service plans as non-compliant regardless of HTTPS status. The compliance dashboard shows 0% compliance for the subscription. The team spends 2 days debugging before identifying the policy condition error. False non-compliance report causing confusion Med
10 No policy for allowed resource types Any resource type can be deployed No policy restricts what resource types can be deployed in subscriptions. A developer experiments with expensive or insecure resource types (e.g., HD Insight clusters, Azure OpenAI with no network restrictions). The accidental deployments generate unexpected costs and create unmanaged attack surfaces before discovery. Unexpected resource sprawl and cost Med

Mitigation Controls

# Failure Mitigation Layer
1 Audit-only policies Transition critical security policies to Deny; use Audit for new controls during hardening period Effect escalation
2 Stale exemptions Set exemption expiry dates; require justification; review exemptions quarterly Exemption governance
3 No MG-level policy Assign baseline security initiative at Management Group root; all subscriptions inherit MG scope
4 DINE remediation failing Grant policy managed identity Contributor on scope; test remediation tasks after assignment RBAC for DINE
5 Broad Deny blocks CI Use parameterized policies with environment exclusions; test policies in Audit mode in lower environments first Policy testing
6 Over-scoped regulatory initiative Scope PCI/HIPAA initiatives to specific resource groups containing in-scope resources Initiative scoping
7 No naming convention Implement Deny policy for resource name pattern (e.g., regex match); or use Append to enforce required tags Naming enforcement
8 Policy evaluation lag Trigger Start-AzPolicyComplianceScan after critical deployments; do not rely on compliance state for real-time security On-demand scan
9 Logic error in custom policy Test custom policy conditions in what-if mode; validate against test resources before assignment Policy testing
10 No allowed resource types Use Allowed resource types or Not allowed resource types built-in policy Resource type control

Top 3 Critical Mitigations

  1. Assign Baseline Initiative at Management Group Root: Policies assigned at the MG root apply to all current and future subscriptions automatically. This is the scalable way to ensure no subscription starts without security guardrails.
  2. Escalate Critical Audit Policies to Deny: Audit-only policies report problems but don't prevent them. Migrate the top 10 security-critical policies (public network access, encryption, MFA requirements) to Deny after confirming no legitimate workloads are blocked.
  3. DINE Policies with Correct RBAC for Auto-Remediation: DeployIfNotExists policies that auto-deploy diagnostic settings, network configs, or tags are only valuable if the remediation actually succeeds. Validate the policy managed identity has the required RBAC roles and test remediation tasks after assignment.

Key Takeaways

  • Assign security baseline initiatives at the Management Group root — every new subscription inherits guardrails automatically.
  • Audit-only policies are awareness tools; enforce critical security controls with Deny effects after validation.
  • DINE policies enable auto-remediation — configure policy managed identity with correct RBAC to ensure remediation succeeds.
  • Policy exemptions should have expiry dates and documented justifications — stale exemptions are a compliance and audit risk.
  • Azure Policy evaluation runs on a 24-hour cycle — do not rely on compliance state for real-time security decisions.
