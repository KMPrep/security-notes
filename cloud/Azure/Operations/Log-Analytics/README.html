Log Analytics

Category: Operations
Azure Service Type: Log Store and Query Engine
Key Docs: Log Analytics Overview

Overview

Azure Log Analytics is the log storage and query engine underlying Azure Monitor, Microsoft Sentinel, and Defender for Cloud. Log data from Azure resources, virtual machines, applications, and custom sources is ingested into a Log Analytics workspace and queried using KQL (Kusto Query Language). Workspaces are isolated data stores with their own retention policies, access controls, and pricing tiers. Log Analytics supports both workspace-context queries (all tables in a workspace) and resource-context queries (logs for a specific resource). Data is organized into tables (e.g., SecurityEvent, AzureActivity, Syslog, Heartbeat, ContainerLog). Data Collection Rules (DCRs) control what data is collected, where it goes, and apply transformations at ingestion time.

How It Works

Core Concepts

Concept Description
Workspace Isolated container for log data; has its own retention, access control, and pricing
Table Schema-defined log data store within a workspace (e.g., SecurityEvent, SignInLogs)
KQL Kusto Query Language — tabular query language for log analytics
Data Collection Rule (DCR) Defines what data to collect from agents or data sources, with optional transformations
Retention Per-workspace (30–730 days interactive) + Archive tier (up to 12 years, lower cost)
Workspace RBAC Reader / Contributor on workspace; resource-context RBAC via table-level permissions
Saved Queries Reusable KQL queries; can be pinned to dashboards or used in alert rules
Data Export Continuously export tables to Storage Account or Event Hub
Linked Services Connect workspace to Automation Account or Defender for Cloud
Workspace Design One workspace vs multiple — balance isolation, cost, query cross-workspace complexity

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Overly permissive workspace RBAC Log Analytics Contributor granted broadly Developers are given Log Analytics Contributor role on the production security workspace to allow query access. Contributor allows them to also delete saved queries, modify data collection rules, and export logs. A disgruntled developer modifies a DCR to stop collecting Security Event logs. The security team loses authentication log visibility for 3 weeks before the missing data is noticed. Log collection sabotaged by overprivileged user Med
2 Single workspace — no environment isolation All data in one workspace A single Log Analytics workspace receives logs from all environments (dev, staging, prod) and all teams. All data is visible to anyone with workspace Reader access. A junior developer with Reader access can query production security event logs, including failed authentication attempts for service accounts and privileged users. Sensitive operational data is over-shared. Production security logs visible to all workspace readers High
3 Short retention — forensic data unavailable Default 30-day retention A Log Analytics workspace is configured with 30-day retention. An incident response is triggered 45 days after the initial compromise. The earliest indicators — authentication logs from Day 1, network flow logs showing initial C2 communication — have been purged. The IR team cannot reconstruct the initial access vector. Forensic investigation blocked by data expiry High
4 Inaccurate KQL alert query — false negatives Complex KQL with logic error A Sentinel analytics rule uses KQL to detect brute-force attacks. The query has a where clause that accidentally filters out accounts with uppercase letters in the name. Service account names (which are all uppercase in the environment) are excluded from the alert. Brute force against service accounts goes undetected for weeks. Real attacks missed due to alert logic error Med
5 High ingestion cost — log everything No ingestion filtering All VerboseSQL diagnostic logs are enabled on Azure SQL databases. The workspace ingests 500 GB/day of verbose SQL query logs (all SELECT statements). The monthly Log Analytics bill is $80,000 — mostly for data the security team never queries. Proper DCR filtering would have reduced ingestion to 5 GB/day by collecting only failed login events and schema change logs. Excessive cost from unfiltered log ingestion High
6 No data export — no immutable backup Logs exist only in workspace No Log Analytics data export is configured. An insider with Log Analytics Contributor role deletes the workspace (accidentally or intentionally). All logs are permanently lost. There is no external archive (Storage Account or Event Hub) to recover from. The incident response for an ongoing investigation is blocked. Log loss from workspace deletion with no backup Med
7 Cross-workspace queries not used — siloed investigation Separate workspaces with no cross-query An enterprise has separate Log Analytics workspaces per subscription. Alerts in the Sentinel workspace require correlating SignInLogs (from Entra ID workspace) with SecurityEvent logs (from VM workspace). The analyst must manually run queries in each workspace and join results externally. This 4-hour manual process means the attacker has a 4-hour head start. Slow incident response due to siloed workspaces Med
8 Workspace region mismatch — high latency Workspace and resources in different regions VMs and Azure resources are in East US. The Log Analytics workspace is in West Europe. Cross-region log ingestion introduces 200–500ms additional latency per log batch. Log ingestion backlog builds up during traffic spikes. Alert rules fire up to 30 minutes late. Time-sensitive alerts for active incidents are delayed significantly. Alert latency from cross-region log routing Low
9 No table-level RBAC — all-or-nothing access Workspace-level RBAC only A Log Analytics workspace contains both SecurityEvent (sensitive auth logs) and ContainerLog (application logs). All developers need ContainerLog access for debugging. Workspace-level Reader grants access to all tables including SecurityEvent. Table-level RBAC (per-table custom role) is not configured. Security team members' authentication events are visible to the entire engineering team. Over-broad log access without table-level controls Med
10 DCR transformation misconfigured — data loss Ingestion-time transformation drops valid events A DCR ingestion-time transformation is written to filter noise. A KQL transformation error causes all events to be dropped instead of just filtered. Log ingestion silently stops for the affected DCR. The issue is discovered 2 days later when an alert notes zero events from target VMs — by which time 2 days of security event logs are permanently lost. Silent log loss from DCR transformation error Med

Mitigation Controls

# Failure Mitigation Layer
1 Overpermissive workspace RBAC Grant Read-only to security analysts; restrict Contributor to SIEM admin service accounts only RBAC
2 No workspace isolation Separate workspaces per environment and team; use cross-workspace queries for correlation Workspace design
3 Short retention Set security tables to 1–2 years interactive; archive older data for 7–12 years Retention policy
4 KQL query errors Test alert queries against historical data; use query simulation/validation before production Query validation
5 Excessive ingestion cost Use DCR ingestion filtering; configure basic logs tier for high-volume low-value tables Cost optimization
6 No backup of logs Configure data export to immutable Storage Account (WORM policy); export to Event Hub for SIEM Data export
7 Siloed workspaces Use cross-workspace KQL (workspace("workspace2").SecurityEvent); consider Sentinel as central correlation layer Cross-workspace queries
8 Region mismatch Deploy Log Analytics workspace in same region as primary workloads Region co-location
9 No table-level RBAC Configure table-level RBAC via custom roles with microsoft.operationalinsights/workspaces/tables/query/read on specific tables Table RBAC
10 DCR transformation error Test DCR transformations in staging; monitor ingestion volume dashboards for unexpected drops DCR validation

Top 3 Critical Mitigations

  1. Set Security Table Retention to 1–2 Years: Default 30-day retention is too short for most forensic investigations. Extend interactive retention for SecurityEvent, SignInLogs, and AzureActivity to at least 1 year. Use the Archive tier for older data at lower cost.
  2. Export Logs to Immutable Storage: Configure Data Export to a Storage Account with immutable blob storage (WORM) and soft delete disabled. This ensures logs survive workspace deletion, insider tampering, or ransomware targeting the Log Analytics workspace.
  3. Apply Workspace and Table-Level RBAC: Workspace Reader grants access to all tables — including sensitive security logs. Use table-level custom RBAC to grant developers access only to application log tables (ContainerLog, AppRequests) without exposing security event tables.

Key Takeaways

  • Default 30-day retention is insufficient — extend security table retention to 1–2 years and archive older data.
  • Always export logs to an immutable external store (WORM Storage) as a backup against workspace deletion.
  • Use workspace-level RBAC for broad access control; add table-level RBAC to prevent developers from reading sensitive security event tables.
  • DCR ingestion-time transformations reduce cost and noise, but must be tested carefully — a filter error can silently drop all data.
  • Cross-workspace KQL queries (workspace("name").TableName) enable correlation across siloed workspaces without merging them.
