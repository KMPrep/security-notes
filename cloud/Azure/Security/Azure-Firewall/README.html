Azure Firewall

Category: Security / Networking
Azure Service Type: Managed Network Firewall
Key Docs: Azure Firewall Overview

Overview

Azure Firewall is a managed, stateful network security service deployed in a VNet's AzureFirewallSubnet. It provides centralized egress filtering, east-west inspection, FQDN-based application rules, network rules (L3/L4), and NAT rules. Azure Firewall Premium adds signature-based IDPS (Intrusion Detection and Prevention System), TLS inspection (deep packet inspection for encrypted traffic), URL filtering, and web categories. It is the hub in a hub-spoke VNet topology, inspecting all traffic between spokes and egress to the internet via UDRs. Azure Firewall Manager centralizes policy management across multiple firewalls and subscriptions.

How It Works

Core Concepts

Concept Description
AzureFirewallSubnet Dedicated /26 subnet required for Azure Firewall deployment
Application Rule Allow/deny HTTP/HTTPS traffic based on FQDN, URL, or web category
Network Rule Allow/deny traffic based on IP address, port, and protocol (L3/L4)
DNAT Rule Destination NAT — maps public IP to private IP for inbound connections
IDPS (Premium) Signature-based intrusion detection and prevention engine
TLS Inspection (Premium) Decrypt, inspect, and re-encrypt HTTPS traffic for deep inspection
Threat Intelligence Block traffic to/from known malicious IPs and domains (Microsoft TI feed)
Firewall Policy Reusable policy object — base policy + child policies for hub-spoke
Forced Tunneling Route all internet-bound traffic from Azure through the firewall
SNAT Azure Firewall performs SNAT for outbound internet traffic

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 IDPS in Alert mode only — not blocking Default Premium IDPS mode is Alert Azure Firewall Premium IDPS is deployed but left in Alert mode (default). An attacker who has compromised a VM in a spoke VNet initiates a command-and-control callback to a known C2 server. IDPS detects the signature and logs an alert, but does not block the traffic. The C2 session establishes and the attacker exfiltrates data over the permitted egress path. C2 traffic permitted despite IDPS detection High
2 Missing UDR — spoke traffic bypasses firewall UDR not applied to spoke subnets Hub-spoke architecture is designed with Azure Firewall at the hub. A new spoke subnet is provisioned without the required UDR (0.0.0.0/0 → Firewall Private IP). Traffic from that subnet goes directly to the internet, bypassing all firewall rules, IDPS, and application rules. The subnet is effectively unprotected. Traffic egress bypasses firewall High
3 Overly permissive application rules Allow-all FQDN rule An application rule allows *.com FQDNs for a dev workload. This rule is propagated to the production firewall policy through a shared base policy. Production workloads can now egress to any .com domain, enabling data exfiltration to attacker-controlled domains, malware downloads, and command-and-control communications. Data exfiltration and malware via overpermissive egress High
4 No TLS inspection — encrypted C2 not detected HTTPS traffic not decrypted Azure Firewall Standard (no TLS inspection) or Premium without TLS inspection policy allows HTTPS traffic based on FQDN match only. An attacker uses DNS-over-HTTPS and HTTPS C2 to a legitimate-looking domain (updates.cdn-delivery.com). The firewall allows the connection (FQDN not blocked) and cannot inspect the encrypted payload for malicious signatures. Encrypted C2 bypasses firewall inspection High
5 AzureFirewallSubnet too small /27 or smaller subnet Azure Firewall requires a /26 subnet at minimum and is a zone-redundant service requiring multiple IPs. A /27 subnet (27 usable IPs) is provisioned. The firewall fails to deploy or cannot scale to additional instances due to IP exhaustion. The deployment fails mid-rollout, and the subnet must be resized, requiring the firewall to be deleted and recreated. Deployment failure and firewall downtime Med
6 Rule order misconfiguration — deny overridden Network rules evaluated before application rules Network rules are evaluated before application rules in Azure Firewall. An engineer adds a broad network rule Allow Any:Any:443 to resolve an application connectivity issue. This network rule overrides more specific application rule denies for blocked FQDNs. Traffic that should be blocked by application rules is permitted because the network rule matches first. Security rules bypassed by rule order Med
7 No threat intelligence blocking TI-based filtering in alert mode Azure Firewall threat intelligence filtering is set to Alert only (not Alert and Deny). A VM in a spoke VNet makes a connection to an IP address flagged in Microsoft's threat intelligence feed (known botnet C2). The alert is logged but not blocked. The connection succeeds and the VM receives malware instructions. Known-bad IP/domain connectivity permitted High
8 Firewall policy misconfiguration — base policy overrides Inherited base policy has permissive rules Azure Firewall Manager uses a base policy inherited by all child policies. A permissive Allow Any to Any rule in the base policy is inherited by all child policies. Child policies cannot override base policy allow rules with deny rules (allow rules in parent propagate). All firewalls across the organization inherit the permissive rule. Organization-wide firewall bypass from base policy Med
9 No firewall logs to Log Analytics Diagnostic settings not configured Azure Firewall does not have diagnostic settings enabled. An attacker exfiltrates data from multiple VMs over several days using allowed egress paths. No network flow logs exist. After the incident, the security team cannot determine what data left the network, which VMs were involved, or what external IPs communicated with the environment. No forensic evidence for incident response High
10 East-west traffic not inspected — lateral movement Spoke-to-spoke without firewall routing The hub-spoke UDRs route only internet-bound traffic through the firewall. Spoke-to-spoke traffic routes directly via VNet peering without passing through the firewall. An attacker who compromises a VM in Spoke 1 pivots laterally to Spoke 2's database subnet without any firewall inspection or logging. Undetected lateral movement between spokes Med

Mitigation Controls

# Failure Mitigation Layer
1 IDPS alert only Set IDPS mode to Alert and Deny for all traffic directions; review alerts regularly IDPS mode
2 Missing UDR Apply UDR to all spoke subnets via Azure Policy; validate with Network Watcher next-hop UDR governance
3 Permissive FQDN rules Define explicit allow-list of required FQDNs; use web category blocking for broad egress control Application rules
4 No TLS inspection Enable TLS inspection in Premium policy; install intermediate CA in trusted root of VMs TLS inspection
5 Small subnet Use /26 for AzureFirewallSubnet; document as mandatory in IaC templates Subnet sizing
6 Rule order issue Use application rules for FQDN-based control; avoid catch-all network rules; test with Firewall Policy Analyzer Rule design
7 No threat intelligence blocking Set threat intelligence mode to Alert and Deny; keep Microsoft TI feed enabled Threat intelligence
8 Base policy misconfiguration Review base policy rules carefully; test policy inheritance with Firewall Manager Policy Analyzer Policy governance
9 No firewall logs Enable diagnostic settings sending to Log Analytics; create workbook for egress analysis Firewall logging
10 No east-west inspection Use UDRs between spokes routing to firewall; enable inter-spoke inspection in policy East-west control

Top 3 Critical Mitigations

  1. IDPS and TI in Alert-and-Deny Mode: Alert mode is observability, not protection. Set both IDPS and Threat Intelligence to Alert and Deny — these blocks known-bad traffic automatically and reduce analyst noise.
  2. Enforce UDRs via Azure Policy: Every new spoke subnet without a UDR is a firewall bypass. Use Azure Policy to audit/enforce that all subnets have the required UDR pointing to the Azure Firewall private IP.
  3. Enable TLS Inspection: Modern C2 and exfiltration use HTTPS. Without TLS inspection, Azure Firewall Premium can only see the SNI hostname — the payload is opaque. TLS inspection enables deep IDPS analysis of encrypted traffic.

Key Takeaways

  • Azure Firewall is the hub egress control in hub-spoke VNet architectures; all spoke traffic must route through it via UDRs.
  • IDPS and Threat Intelligence must be in Alert and Deny mode — Alert-only is logging, not prevention.
  • TLS inspection is required for Azure Firewall Premium to inspect HTTPS traffic — most modern attacks use TLS.
  • East-west (spoke-to-spoke) traffic requires explicit UDRs between spokes; VNet peering bypasses the firewall by default.
  • Enable diagnostic logging to Log Analytics — firewall logs are the primary egress visibility data source.
