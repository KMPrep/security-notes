Azure DDoS Protection

Category: Security / Networking
Azure Service Type: Distributed Denial-of-Service Mitigation
Key Docs: DDoS Protection Overview

Overview

Azure DDoS Protection defends Azure VNet resources against volumetric, protocol, and application-layer DDoS attacks. It offers two tiers: Basic (free, automatically applied to all Azure infrastructure) and Standard/Network (paid, per-VNet, with adaptive tuning and SLA guarantees). Standard DDoS Protection uses machine learning-based adaptive traffic profiling, automatically mitigating attacks targeting public IPs in protected VNets. Azure DDoS Protection IP protects individual public IPs. For application-layer (L7) protection, DDoS Standard is combined with Azure Front Door WAF, Application Gateway WAF, or Azure Firewall. DDoS Standard provides attack telemetry, diagnostics, and post-attack reports.

How It Works

Core Concepts

Concept Description
DDoS Basic Free, always-on, protects Azure platform infrastructure; no tenant-level visibility
DDoS Network Protection Per-VNet plan; adaptive profiling, SLA, attack analytics, cost protection
DDoS IP Protection Per-public-IP plan; alternative to VNet plan for smaller deployments
Adaptive Tuning ML model learns baseline traffic; sets per-resource mitigation thresholds automatically
Mitigation Pipeline Monitors → detects anomaly → activates scrubbing → forwards clean traffic
Attack Metrics Real-time metrics: UnderDDoSAttack, PacketsDropped, BytesDropped in Azure Monitor
Attack Analytics Post-attack reports and real-time logs for attack vectors and magnitude
Cost Protection DDoS Standard covers Azure resource scaling costs incurred during an attack
Rapid Response DDoS Rapid Response (DRR) team engagement during active attacks (included with Standard)
L7 Complement DDoS Standard operates at L3/L4; L7 (HTTP flood) protection requires WAF (AFD/AppGW)

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 DDoS Basic only — no adaptive protection Using free tier for production workloads Production public IPs are protected only by DDoS Basic, which defends Azure infrastructure but does not provide adaptive per-resource profiling or tenant-level attack visibility. An attacker launches a volumetric UDP flood targeted at the application's public IP. Basic provides some protection at the infrastructure level, but the application-tier public IP is overwhelmed before Azure infrastructure-level mitigation triggers. The service is unavailable for 15–30 minutes. Application-tier outage from volumetric attack Med
2 No L7 DDoS protection — HTTP flood bypasses L3/L4 DDoS Standard only, no WAF DDoS Standard protects L3/L4 volumetric attacks but cannot inspect HTTP request content. An attacker launches a slow HTTP flood (e.g., Slowloris, HTTP GET flood) with legitimate-looking requests at low volume. Each request consumes application server threads and memory. DDoS Standard does not detect this attack (traffic is below volume thresholds). The web servers are overwhelmed by connection exhaustion. L7 application-layer DDoS succeeds High
3 Attack on unprotected public IP — VNet has DDoS Standard Public IP not under protected VNet A load balancer is moved to a new VNet that does not have DDoS Standard enabled. The original VNet had DDoS Standard, but the assumption was that standard protects all resources. DDoS protection is scoped to VNets — the new VNet and its public IPs are unprotected. An attack on the new load balancer public IP is not mitigated. Attack succeeds on unprotected VNet Med
4 Adaptive tuning not converged — false positive blocking New deployment with no traffic baseline A new service is deployed and DDoS Standard has not yet learned the traffic baseline (requires 30 days to fully tune). A legitimate marketing campaign drives a sudden spike in traffic. DDoS adaptive tuning triggers a false positive mitigation, blocking legitimate user traffic as attack traffic. The service appears down for 10–30 minutes during a high-visibility event. Legitimate traffic blocked by false positive Med
5 No monitoring or alerting — attack not detected DDoS metrics not in Azure Monitor DDoS Standard emits detailed attack metrics but no Azure Monitor alert is configured. An attack is actively mitigated by DDoS Standard, but the security team is not notified. The attack logs and analytics are generated but reviewed only during quarterly audits. A subsequent targeted attack exploits a pattern learned from the first attack. Unnoticed attack; delayed response and threat intelligence Med
6 Amplification attack exploiting Azure services UDP-based amplification from reflected sources An attacker spoofs the victim's IP and sends small requests to DNS or NTP amplifiers that reflect large responses to the victim's public IP. The reflected traffic appears to come from legitimate sources. DDoS Standard detects the volumetric anomaly and mitigates, but the brief peak traffic saturates the VNet before scrubbing activates. Brief saturation before mitigation activates Med
7 Cost exploitation — autoscaling costs during attack No DDoS Standard cost protection Azure services behind a load balancer autoscale during a DDoS attack to handle the increased load. Without DDoS Standard (which includes cost protection), the attack causes thousands of VM instances to spin up, generating a large Azure bill. The attack's goal is financial damage, not service disruption. Financial damage from autoscale-triggered cost Med
8 No DRR engagement during active attack Rapid Response team not contacted An active large-scale DDoS attack is partially mitigated by DDoS Standard but a residual attack vector is still causing degradation. The DDoS Rapid Response (DRR) team, available to DDoS Standard customers, could provide real-time analysis and custom mitigation. The team is not contacted because the SOC does not know the service exists. Degradation continues for hours. Prolonged outage from non-engagement of DRR Low
9 No post-attack forensics review Attack analytics not utilized After a DDoS attack is mitigated, the attack analytics report (attack type, vectors, source geography, peak BPS/PPS) is not reviewed. The organization misses intelligence about attack patterns and does not tune WAF rules or add geo-restrictions to address the attack vectors. The same attack type recurs two weeks later with the same impact. Missed intelligence leads to repeat attack Med
10 Single-region deployment without global distribution All public IPs in one Azure region A high-profile application has all public IPs in a single Azure region. A volumetric attack targeting that region's public IPs exhausts scrubbing capacity (extremely large attacks can overwhelm even DDoS Standard). Azure Front Door distributes traffic across multiple Azure edge PoPs, absorbing attacks globally. Without Front Door or Traffic Manager, the single region bears the full attack volume. Regional scrubbing capacity overwhelmed Med

Key Takeaways

  • DDoS Basic protects Azure infrastructure; DDoS Network Protection adds per-resource adaptive profiling and attack telemetry.
  • L7 attacks (HTTP flood, Slowloris) require WAF — DDoS Standard only protects L3/L4.
  • Enable UnderDDoSAttack metric alerts — DDoS Standard mitigates automatically but silent attacks miss security intelligence.
  • Azure Front Door provides global edge distribution, making volumetric attacks harder to sustain across 100+ PoPs.
  • DDoS Standard includes cost protection and DRR team access — document the DRR engagement process in runbooks.
