Microsoft Defender for Cloud

Category: Security
Azure Service Type: CSPM and CWPP
Key Docs: Defender for Cloud Overview

Overview

Microsoft Defender for Cloud (formerly Azure Security Center + Azure Defender) is a unified Cloud Security Posture Management (CSPM) and Cloud Workload Protection Platform (CWPP). CSPM continuously assesses Azure resources against security benchmarks (Microsoft Cloud Security Benchmark, CIS, NIST, PCI DSS) and provides a Secure Score with actionable recommendations. CWPP provides runtime threat detection for servers (Defender for Servers), containers (Defender for Containers), databases (Defender for SQL), and storage (Defender for Storage). Defender for Cloud spans Azure, AWS, and GCP through the multi-cloud connector, making it a single pane of glass for hybrid and multi-cloud security.

How It Works

Core Concepts

Concept Description
Secure Score Percentage score (0-100) reflecting security posture against benchmark controls
Recommendation Specific actionable item to improve a control (e.g., "Enable MFA for all accounts")
Security Control Group of related recommendations; fixing all recommendations in a control contributes to score
Defender Plan Enable-per-resource-type paid plans (Servers, Containers, SQL, Storage, etc.)
Alert Security incident detected by runtime threat detection (CWPP)
CSPM Agentless scan of resource configurations against security benchmark policies
CWPP Agent-based runtime protection for workloads (VMs, containers)
Regulatory Compliance Dashboard showing compliance with frameworks (PCI DSS, ISO 27001, SOC 2)
Attack Path Analysis Graph-based visualization of exploit paths through the environment
Cloud Security Graph Asset inventory and relationship graph powering attack path and risk findings

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Secure Score gaming without real improvement Closing recommendations without fixing root cause A team improves Secure Score by suppressing recommendations (marking as Not Applicable or Dismissed) without addressing the underlying misconfiguration. Public storage blobs, unencrypted disks, and missing MFA are suppressed. The Secure Score reaches 90%, but the actual security posture is unchanged. An attacker exploits the misconfigured storage to exfiltrate data. False sense of security; real vulnerabilities persist High
2 Defender plans not enabled for all resource types CWPP coverage gaps Defender for Servers is enabled, but Defender for SQL and Defender for Storage are disabled to save cost. A SQL injection attack on Azure SQL and an anomalous data download from Azure Blob Storage generate no security alerts. The attacker operates undetected for weeks because the relevant Defender plans are not active. Attacker activity undetected in uncovered resource types High
3 High-severity recommendations unaddressed for months No SLA or remediation workflow for recommendations Defender for Cloud has been recommending remediation for a critical vulnerability (e.g., "VMs should have an endpoint protection solution installed") for 8 months. No owner is assigned; no SLA exists. The vulnerability is known to the security team, but the application team has not prioritized it. The vulnerability is exploited by a ransomware group. Exploited known vulnerability from ignored recommendation High
4 Alert suppression rules masking real attacks Alert fatigue leading to over-suppression An analyst creates suppression rules to reduce noise from a noisy dev environment. The suppression rule is too broad and also suppresses alerts for the production subscription. A brute force attack on production VMs is suppressed. The attack succeeds in compromising a VM while the alert never reaches the analyst. Real attack suppressed by overly broad rule Med
5 No workflow automation — manual response only Alerts require manual triage and action Defender for Cloud generates a high-severity alert (Suspicious process executed). There is no Logic App playbook attached to auto-isolate the VM or notify the SOC team via Microsoft Teams. The analyst sees the alert the next morning (alert was generated at 11 PM). The attacker had 9 hours of uncontested access to the compromised VM. Extended dwell time from manual-only response High
6 Multi-cloud connectors not configured AWS and GCP resources not in Defender The organization runs workloads in Azure, AWS, and GCP. Defender for Cloud is configured for Azure only. AWS EC2 instances, S3 buckets, and GCP VMs have no CSPM or threat detection coverage. A compromised AWS IAM key is used to access production S3 data. No Defender alert is generated because the AWS connector is not configured. Zero visibility into non-Azure cloud workloads Med
7 Just-in-time VM access not enforced Management ports open permanently Defender for Cloud recommends enabling JIT access on all VMs with management ports (22, 3389) open. The recommendation has been open for 3 months. An external attacker scans for open port 3389 and brute forces an RDP-accessible VM. JIT access would have closed the port when not in active use, preventing the attack vector entirely. VM compromise via always-open management port High
8 Attack path analysis ignored No one reviews attack path findings Defender CSPM detects an attack path: Internet-facing VM → with overpermissioned Managed Identity → with access to Key Vault → that contains database master key. This high-priority attack path is visible in the portal but no one reviews the Attack Path Analysis dashboard. An attacker who compromises the internet-facing VM uses the managed identity to extract the Key Vault secrets and pivot to the database. Multi-hop attack path exploited unaddressed High
9 Regulatory compliance gaps not tracked Compliance framework dashboard not reviewed The organization is PCI DSS-compliant but Defender for Cloud's PCI DSS compliance dashboard shows 45 failing controls. The compliance team performs annual audits manually and is unaware of the real-time compliance dashboard. Failing controls (e.g., no audit logging on databases, no encryption of cardholder data at rest) represent real compliance violations that could result in PCI fines or audit failure. Regulatory non-compliance from untracked violations Med
10 Defender for Servers Plan 2 not used — no file integrity monitoring Plan 1 lacks FIM and Defender for Endpoint integration Defender for Servers Plan 1 is enabled, but Plan 2 is not. Plan 2 includes File Integrity Monitoring (FIM), Just-In-Time VM Access, and full Microsoft Defender for Endpoint integration. An attacker modifies critical system files on a Linux VM after gaining a foothold. Without FIM, the modification is not detected. File tampering undetected Med

Mitigation Controls

# Failure Mitigation Layer
1 Score gaming Audit suppressed/dismissed recommendations quarterly; require justification for dismissals Governance
2 Defender plan gaps Enable Defender plans for all resource types in all subscriptions; audit with Management Group policy Coverage
3 Recommendations unaddressed Assign owners to recommendations; set SLA by severity (Critical: 7 days, High: 30 days) Remediation SLA
4 Over-suppression Scope suppression rules to specific resources, not subscriptions; review suppression rules monthly Alert management
5 No workflow automation Create Logic App playbooks for high-severity alerts; auto-notify SOC via Teams/PagerDuty Automated response
6 No multi-cloud Configure AWS and GCP connectors in Defender for Cloud; enroll non-Azure resources Multi-cloud coverage
7 Open management ports Enable JIT access on all VMs; enforce via Defender recommendation + Azure Policy JIT access
8 Attack paths ignored Assign Attack Path Analysis review to weekly security review cadence Attack path review
9 Compliance not tracked Review compliance dashboard weekly; create Defender for Cloud workbook for PCI/ISO tracking Compliance monitoring
10 Plan 1 only Upgrade to Defender for Servers Plan 2 for FIM, JIT, and full MDE integration Defender plan

Top 3 Critical Mitigations

  1. Enable All Defender Plans: Security coverage gaps are blind spots. Enable Defender plans for Servers, SQL, Storage, Containers, and KeyVault across all subscriptions. The cost is justified by the detection capability.
  2. Assign Owners and SLAs to Recommendations: Recommendations without owners never get remediated. Automate assignment via workflow automation and track remediation velocity as a security KPI.
  3. Workflow Automation for High-Severity Alerts: Hours of uncontested attacker access are avoidable. Logic App playbooks can auto-isolate VMs, disable compromised identities, and page the SOC within minutes.

Key Takeaways

  • Defender for Cloud = CSPM (posture) + CWPP (runtime protection); both are needed for comprehensive coverage.
  • Enable all Defender plans — disabled plans create detection blind spots for that resource type.
  • Attack Path Analysis reveals multi-hop exploitation paths that individual recommendations miss.
  • Workflow automation (Logic Apps) enables auto-response to alerts, dramatically reducing dwell time.
  • Secure Score gaming (dismissing recommendations) creates false confidence; track remediation velocity as a real KPI.
