Microsoft Entra ID (Azure Active Directory)

Category: Security / Identity
Azure Service Type: Cloud Identity Platform
Key Docs: Entra ID Overview

Overview

Microsoft Entra ID (formerly Azure Active Directory) is Microsoft's cloud-based identity and access management platform. It provides authentication (who you are), authorization (what you can do), and federation (trust with external identity providers). Entra ID supports single sign-on (SSO) across thousands of SaaS applications, Conditional Access for risk-based authentication policies, Privileged Identity Management (PIM) for JIT privileged role elevation, and multi-factor authentication (MFA). It is the identity backbone of Azure, Microsoft 365, and third-party SaaS integrations. Identity attacks (credential theft, token hijacking, OAuth consent phishing) are the leading entry vector for cloud breaches.

How It Works

Core Concepts

Concept Description
Tenant Dedicated Entra ID instance for an organization
User Principal Human user identity — UPN, credentials, MFA methods, directory roles
Service Principal Non-human identity representing an application or workload in a tenant
Application Registration Represents an application; has client ID, client secret or certificate, redirect URIs
Managed Identity Azure-managed service principal; no credentials to manage
Conditional Access Policy-based authentication enforcement — MFA, device compliance, location, risk
PIM Privileged Identity Management — JIT role activation with approval workflow
Directory Role Entra ID built-in roles (Global Admin, User Admin, etc.) — tenant-wide scope
RBAC Azure RBAC roles — resource-level access (differs from directory roles)
OAuth 2.0 / OIDC Token-based protocols for delegation and federation; Entra ID as the authorization server

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Global Administrator compromise Overprivileged permanent GA assignment A global administrator has a permanent GA role assignment (not PIM-managed) and uses the same account for day-to-day email. A phishing email with an AiTM proxy attack steals the authenticated session token and bypasses MFA. The attacker now has Global Admin access to the entire Entra ID tenant and all Azure subscriptions. They create a new admin account for persistence and disable MFA for targeted users. Complete tenant takeover Critical
2 OAuth consent phishing Malicious app granted delegated permissions An attacker registers a malicious Azure app and sends users a phishing link disguised as a productivity tool. Users consent to Mail.ReadWrite and Files.ReadWrite.All permissions. The app now has delegated access to their mailbox and OneDrive, persisting even after the user changes their password (the access token refreshes independently). Persistent mailbox and file access High
3 Conditional Access not covering all apps CA policy with incomplete application scope Conditional Access requires MFA for Office 365 but no CA policy covers custom Azure applications. An attacker who steals a user's password can authenticate directly to the Azure application without MFA by using the token endpoint that is not in any CA policy scope. The custom app becomes the weakest authentication link. MFA bypass via uncovered application High
4 Legacy authentication protocols enabled Basic auth, POP3, IMAP, SMTP allowed An older Exchange or IMAP client is allowed to authenticate with legacy protocols that do not support MFA. An attacker uses a password spray attack against these protocols (which are not throttled by Entra ID's Smart Lockout the same way). The attacker compromises accounts by trying a single common password against thousands of users. Credential-based account compromise High
5 Refresh token not revoked after compromise No token revocation on incident response A user's device is compromised. The incident response team resets the user's password but does not revoke all refresh tokens. The attacker retains a valid long-lived refresh token (default lifetime: 90 days) and continues to obtain new access tokens silently. The user's account remains compromised despite the password reset. Continued access post-compromise High
6 Service principal with client secret — no expiry Permanent credentials on service principal An application registration has a client secret with no expiry date. The client secret is embedded in an application config file. The config file is accidentally committed to a public GitHub repository. The attacker extracts the secret and authenticates as the service principal, gaining access to all resources the SP can access — potentially production databases and Key Vault secrets. Application identity compromise High
7 PIM approval not required for Global Admin activation PIM configured without approval workflow PIM is enabled for Global Administrator roles, but no approver is required — users can self-activate the GA role instantly with justification only. A compromised admin account activates GA privileges without any human review. The attacker escalates to full tenant admin within seconds. Unchecked privilege escalation Med
8 Guest accounts with excessive permissions External users granted resource-level RBAC Guest accounts invited for collaboration are granted Contributor roles on production subscriptions. When the guest user leaves their organization, their account remains active in the host tenant. A former collaborator retains access to production resources indefinitely. No guest user lifecycle management exists. Stale external access to production Med
9 Sign-in risk and user risk not responded to Identity Protection signals not actioned Entra ID Identity Protection detects a high-risk sign-in (atypical travel, impossible travel, leaked credentials) but no Conditional Access policy is configured to block or require MFA for high-risk sign-ins. The risk flag is visible in the portal but generates no automated response. An attacker using leaked credentials from a breach list signs in without challenge. Successful sign-in despite risk signal High
10 Directory role assigned at tenant scope instead of AU Overly broad directory role scope A helpdesk administrator is assigned User Administrator at the full tenant scope to manage a specific department's user accounts. The role allows them to reset passwords for all users in the tenant, including global administrators. An attacker who compromises the helpdesk account uses password reset to take over privileged accounts. Privilege escalation via broad admin scope Med

Mitigation Controls

# Failure Mitigation Layer
1 GA compromise Use PIM for all privileged roles; require phishing-resistant MFA (FIDO2/WHfB) for GA; use dedicated cloud-only admin accounts Privileged access
2 OAuth consent phishing Enable admin consent requirements for all non-Microsoft apps; deploy Defender for Cloud Apps to detect suspicious consent App consent governance
3 Incomplete CA coverage Create a CA policy scoping to All cloud apps; use named exclusions only; audit with CA What If tool Conditional Access
4 Legacy auth Block legacy authentication via Conditional Access; monitor Legacy Authentication sign-in report Legacy auth blocking
5 Stale refresh tokens Use Revoke-AzureADUserAllRefreshTokens or revokeSignInSessions on incident response; set token lifetime policies Token revocation
6 Permanent client secrets Use certificates instead of secrets for service principals; set secret expiry; use Managed Identities where possible Secret rotation
7 PIM without approval Configure PIM with approver requirement for GA; set maximum activation duration to 1–2 hours PIM configuration
8 Stale guest accounts Implement Access Reviews for guest accounts; enforce guest expiry policy; use entitlement management Guest lifecycle
9 Risk not actioned Configure Conditional Access risk-based policy: block/MFA for high-risk sign-in; password change for high-risk user Identity Protection
10 Broad directory roles Scope User Administrator to Administrative Units for specific departments; apply least-privilege role assignments Role scoping

Top 3 Critical Mitigations

  1. Phishing-Resistant MFA for All Admins: Conventional TOTP MFA is defeatable with AiTM proxy attacks. Require FIDO2 security keys or Windows Hello for Business for all privileged accounts — these are bound to the origin and cannot be relayed.
  2. Block Legacy Authentication Immediately: Legacy auth protocols do not support MFA and are the primary vector for password spray attacks. A single Conditional Access policy blocking legacy auth eliminates this entire attack class.
  3. PIM with Approval for Global Admin: Permanent GA assignments are unacceptable. All privileged roles should be eligible-only in PIM with an approval workflow, justification requirement, and short max activation duration.

Key Takeaways

  • Entra ID is the identity foundation of Azure; identity attacks (credential theft, token hijacking) are the dominant cloud breach vector.
  • Block legacy authentication via Conditional Access — legacy protocols bypass MFA and are actively exploited.
  • Use PIM for all privileged roles; eliminate permanent GA assignments; require phishing-resistant MFA for admins.
  • OAuth consent phishing grants persistent delegated access; enforce admin consent for third-party apps.
  • Token revocation must be explicit on incident response — password reset alone does not invalidate existing refresh tokens.
