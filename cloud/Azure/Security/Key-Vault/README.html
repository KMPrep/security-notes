Azure Key Vault

Category: Security
Azure Service Type: Secrets, Keys, and Certificate Management
Key Docs: Key Vault Overview

Overview

Azure Key Vault is a managed cloud HSM-backed service for storing and accessing secrets (passwords, connection strings, API keys), cryptographic keys (RSA, EC), and certificates (X.509). It provides centralized, audited access control, eliminating hard-coded credentials in application code and configuration files. Key Vault offers two tiers: Standard (software-protected keys) and Premium (HSM-backed keys with FIPS 140-2 Level 2/3). For higher HSM requirements, Azure Managed HSM (Dedicated HSM) is available. Key Vault integrates with Azure services, Managed Identities, and supports event-driven notifications via Event Grid. Soft-delete and purge protection prevent accidental or malicious deletion of vault contents.

How It Works

Core Concepts

Concept Description
Vault Top-level container; each vault has a unique DNS name
Secret Opaque text value with versioning and expiry
Key Cryptographic key; operations (encrypt, decrypt, sign) performed in Key Vault — key never leaves
Certificate X.509 certificate with auto-renewal via integrated CAs (DigiCert, GlobalSign)
Soft Delete Deleted objects retained for 7–90 days; can be recovered
Purge Protection When enabled, deleted objects cannot be permanently deleted until retention period expires
RBAC Recommended access model — Entra ID roles (Key Vault Secrets Officer, Crypto User, etc.)
Access Policy Legacy per-principal permission model; object-level permissions
Private Endpoint Key Vault accessible only from VNet private IP; disables public endpoint
Managed HSM Dedicated, single-tenant HSM with FIPS 140-2 Level 3; no shared infrastructure

Step-by-Step Flow — Application Secret Retrieval via Managed Identity

    participant IMDS as Azure IMDS (169.254.169.254)
    participant Entra as Entra ID
    participant KV as Key Vault
    App->>IMDS: GET /metadata/identity/oauth2/token
    IMDS->>Entra: Exchange managed identity assertion for token
    Entra-->>IMDS: Access token (audience: vault.azure.net)
    IMDS-->>App: Access token
    App->>KV: GET /secrets/db-password with Bearer token
    KV->>KV: Validate token, check RBAC
    KV-->>App: Secret value (current version)
    App->>App: Use secret; do not log or cache long-term

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Key Vault accessible via public endpoint Firewall not enabled on vault A Key Vault is deployed without configuring the network firewall. The vault is accessible from any IP on the internet via its public endpoint. An attacker who obtains a valid bearer token (via SSRF, compromised managed identity, or phished credential) can access all secrets in the vault from anywhere on the internet, not just from within the VNet. Secrets accessible from internet High
2 Soft delete disabled — permanent deletion Default soft delete not enforced A Key Vault is provisioned before the Azure platform enforced soft delete (pre-2020). Soft delete is not enabled. A misconfigured IaC pipeline runs a destructive operation that deletes the vault. All secrets, keys, and certificates are immediately and permanently deleted. Applications fail immediately, and recovery requires restoring from backup or manual recreation. Permanent secret loss from accidental deletion Med
3 Overprivileged access policy — all permissions Legacy access policy grants all secret permissions An access policy is created with All Secret Permissions for an application service principal. The service principal only needs Get on a specific secret. If the service principal is compromised, the attacker can list, read, write, delete, and backup all secrets in the vault. The over-permission follows the compromised identity. Full vault access from compromised identity High
4 Shared vault for multiple environments One vault per organization A single Key Vault holds secrets for development, staging, and production environments. A developer with access to development secrets (for debugging) has the same vault access as production. The developer can read production database passwords and API keys. No environment boundary exists. Cross-environment secret leakage High
5 Secret expiry not enforced Secrets with no expiry date Database passwords and API keys stored in Key Vault have no expiry date set. A secret that is compromised (via log leakage, SSRF, or audit log analysis) remains valid indefinitely. The organization has no automated rotation policy. The compromised credential is exploited for months before detection. Long-lived secret enables prolonged exploitation High
6 No private endpoint — SSRF bypasses network rules Network firewall with IP allowlist but no PE Key Vault has a network firewall allowing traffic from the VNet subnet via service endpoint. However, an SSRF vulnerability in a web app allows an attacker to make requests from the app server (which is in the allowed subnet) to the Key Vault. Service endpoints do not prevent same-subnet SSRF; the attacker reads all secrets the app has access to. SSRF leading to Key Vault secret theft Med
7 Audit logging disabled No Key Vault diagnostic settings Key Vault has no diagnostic settings configured. An insider threat with legitimate access reads hundreds of secrets over several weeks in preparation for leaving the organization. No log exists of which secrets were accessed, when, or from what IP. The theft is only discovered after the insider leaks the data. Undetected secret exfiltration High
8 Certificate private key exported exportable flag set A Key Vault certificate is created with the private key marked as exportable. The service principal with Certificate Get permission can export the certificate with its private key using the GetSecret permission (certificates are stored as secrets when exportable). An attacker who compromises the service principal can export the private key, impersonate the certificate subject indefinitely. Certificate private key theft Med
9 Purge protection not enabled — ransom attack No purge protection on vault An attacker who compromises a Key Vault with Delete permissions soft-deletes all secrets. Without purge protection, they then purge all secrets immediately (permanent deletion). The organization has no offline backup of secrets. All applications that rely on Key Vault fail immediately. The attacker demands ransom for the secrets (which they exfiltrated before deletion). Ransomware attack on vault contents Med
10 Key rotation without application awareness Key rotated but app cached old key A cryptographic key is rotated in Key Vault. The application caches the key version in its configuration. After rotation, the app attempts to use the old key version, which is disabled. Decryption operations fail and data becomes temporarily inaccessible. No mechanism exists for the app to detect and reload the new key version. Application failure after key rotation Med

Mitigation Controls

# Failure Mitigation Layer
1 Public endpoint Enable Key Vault firewall; use private endpoint; disable public access Network access
2 No soft delete Enable soft delete (now default); enable purge protection for production vaults Deletion protection
3 Overprivileged access Use RBAC with minimum permissions (e.g., Key Vault Secrets User for Get only); avoid Access Policies RBAC least privilege
4 Shared vault Use separate Key Vaults per environment (dev/staging/prod); enforce with Azure Policy Vault isolation
5 No secret expiry Set expiry dates on all secrets; configure Event Grid + Logic App for rotation automation; use Key Vault rotation policy Secret lifecycle
6 SSRF via service endpoint Use private endpoint instead of service endpoint; enforce Managed Identity for all access Private endpoint
7 No audit logging Enable Key Vault diagnostic settings sending to Log Analytics; alert on bulk secret reads Audit logging
8 Exportable certificate Create certificates without the exportable flag; use non-exportable keys in HSM (Premium SKU) Certificate policy
9 No purge protection Enable enablePurgeProtection: true in IaC; set retention days to 90 for production vaults Purge protection
10 Key rotation awareness Use Key Vault event notifications (Event Grid) to trigger app reload on key rotation; use latest key version in app config Rotation integration

Top 3 Critical Mitigations

  1. Private Endpoint + Disable Public Access: Key Vault network firewall with IP/VNet rules still leaves a public endpoint. Private endpoint eliminates public accessibility entirely and is the correct approach for all production vaults.
  2. RBAC with Least Privilege + No Access Policies: Access Policies grant all or nothing; RBAC allows fine-grained assignments. Use Key Vault Secrets User (GetSecret only) not Secrets Officer for application identities.
  3. Purge Protection + Audit Logging: Enable purge protection to prevent ransom-deletion attacks. Enable diagnostic logging to detect insider threats and bulk secret access before exfiltration is complete.

Key Takeaways

  • Use private endpoints for Key Vault in production — network firewall alone still exposes a public DNS endpoint.
  • Enable purge protection on all production vaults — soft delete without purge protection can be bypassed.
  • Use RBAC with minimum permissions (Secrets User not Secrets Officer) for application identities.
  • Separate vaults per environment (dev/staging/prod) — shared vaults collapse environment boundaries.
  • Enable diagnostic logging for all vaults — insider threats and bulk reads are only detectable via access logs.
