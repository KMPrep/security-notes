Azure Managed Identity

Category: Security / Identity
Azure Service Type: Workload Identity
Key Docs: Managed Identity Overview

Overview

Azure Managed Identity provides Azure resources with an automatically managed identity in Entra ID, eliminating the need for developers to store and manage credentials. When a resource (VM, App Service, AKS pod, Function App) has a managed identity, it can authenticate to any Azure service that supports Entra ID authentication without any stored secret or certificate. Azure manages the credential lifecycle — creation, rotation, and deletion. There are two types: System-assigned (tied to the resource lifecycle, one-to-one) and User-assigned (standalone resource, reusable across multiple services). Managed Identity is the recommended identity pattern for all Azure workloads accessing Azure resources.

How It Works

Core Concepts

Concept Description
System-Assigned Identity created with the resource; deleted when resource is deleted; one per resource
User-Assigned Standalone identity resource; can be assigned to multiple Azure resources
IMDS Token Endpoint 169.254.169.254/metadata/identity/oauth2/token — local endpoint for token acquisition
Service Principal Managed Identity creates an SP in Entra ID; invisible credential managed by platform
RBAC Assignment Grant the managed identity RBAC roles to access Azure resources (Storage, Key Vault, SQL)
Workload Identity Federation AKS pods can federate with Managed Identity using OIDC token exchange (no IMDS required)
Token Audience Access token is scoped to a resource (e.g., https://vault.azure.net)
Cross-tenant User-assigned managed identities can only access resources in the same tenant

Step-by-Step Flow — Token Acquisition and Service Access

    participant App as App Service (Managed Identity)
    participant IMDS as Azure IMDS 169.254.169.254
    participant Entra as Entra ID
    participant KV as Azure Key Vault
    App->>IMDS: GET /token?resource=https://vault.azure.net
    IMDS->>Entra: Exchange managed identity credential for token
    Note over IMDS,Entra: Credential is platform-managed — no secret stored
    Entra-->>IMDS: Access token (1 hour, audience: vault.azure.net)
    IMDS-->>App: Return access token
    App->>KV: GET /secrets/db-password [Bearer ]
    KV->>KV: Validate token; check RBAC — identity has Secrets User role
    KV-->>App: Return secret value

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 SSRF to IMDS — token theft Application has SSRF vulnerability A web application has an SSRF vulnerability that allows an attacker to make requests to internal URLs. The attacker sends a crafted request that causes the server to fetch http://169.254.169.254/metadata/identity/oauth2/token?resource=https://management.azure.com. The IMDS returns a valid access token for the Azure Management API. The attacker uses the token to enumerate and access Azure resources the VM's managed identity can reach. Azure access token theft via SSRF High
2 Overprivileged managed identity Managed identity has Contributor on subscription A VM's system-assigned managed identity is given Contributor role on the entire subscription for convenience. An attacker who compromises the VM's application code can use the IMDS token to create new resources, modify existing configurations, delete security controls, and access all data in the subscription. The overpermission turns an application compromise into a full subscription takeover. Full subscription access from compromised VM High
3 User-assigned MI reused across environments Same MI in dev and prod A user-assigned managed identity is assigned to VMs in both development and production environments (to simplify RBAC management). A developer can obtain a token on the dev VM and use it to access production Key Vault or Storage because the identity has permissions in production. The identity bridge collapses the dev/prod security boundary. Dev environment pivots to production access High
4 Token cached after role removal Access token valid for 1 hour post-revocation A managed identity's RBAC role is removed immediately after a security incident to revoke access. However, the application has cached the access token issued 30 minutes earlier. For the remaining 30 minutes of the token's validity period, the application (or attacker) can still use the cached token to access the resource. Token revocation in Entra ID does not invalidate issued access tokens — only refresh tokens. 1-hour access window after RBAC removal Med
5 IMDS accessible to containers without metadata protection Shared IMDS in multi-tenant container environment In a multi-container environment (e.g., AKS without Workload Identity or with misconfigured network policies), all containers running on the same node can reach the IMDS endpoint. A compromised low-privilege container queries the IMDS and obtains the node's managed identity token. The node identity has broad RBAC permissions, enabling the compromised container to escalate privileges. Container escape via IMDS token theft High
6 System-assigned MI not deleted — orphaned permissions Resource deleted without RBAC cleanup A VM is deleted. The system-assigned managed identity is automatically deleted, but RBAC role assignments for that identity are not automatically cleaned up. Azure shows the RBAC assignments as "Identity not found" (unknown identities). This is a minor security issue — the identity is gone, so the assignments are harmless. But accumulation of orphaned RBAC assignments creates noise that obscures real access control. RBAC audit noise from orphaned assignments Low
7 Managed identity token passed to third-party service Token exfiltration from application An application obtains a managed identity token for Azure Storage and logs the full Authorization: Bearer <token> header for debugging purposes. The log file is accessible to the development team. A developer notices the token and uses it externally to access the storage account directly. Managed identity tokens are short-lived (1 hour) but can be used by anyone who has them. Access token leakage via log exfiltration Med
8 No managed identity — connection string in config Developer reverts to stored credentials A developer cannot figure out Managed Identity configuration within a sprint deadline and falls back to storing a storage account connection string in appsettings.json. The config is committed to source control. The connection string grants full storage account access. It is discovered by a secret scanning tool weeks later, but the storage data may have already been accessed. Credential leak via source code High
9 Cross-tenant resource access limitation misunderstood MI cannot access cross-tenant resources A developer assigns a user-assigned managed identity and expects it to authenticate to an Azure SQL database in a different Entra ID tenant (a partner organization). Managed identities cannot authenticate cross-tenant. The design assumption is wrong. The developer falls back to a shared service account password — introducing the credentials management problem that Managed Identity was supposed to solve. Incorrect architecture leading to credential use Med
10 AKS Workload Identity misconfiguration — wrong service account OIDC federation annotation mismatch Workload Identity for AKS requires annotating the Kubernetes service account with the managed identity's client ID and federating the service account OIDC token. A misconfiguration (wrong client ID annotation or wrong namespace) causes the token exchange to fail. The pod cannot authenticate to Key Vault or Storage. The application falls back to a mounted secret that was left as a debugging fallback, bypassing Workload Identity entirely. Secret fallback used instead of Workload Identity Med

Mitigation Controls

# Failure Mitigation Layer
1 SSRF to IMDS Implement IMDS v2 with session token requirement; apply Metadata: true header enforcement; fix SSRF in app code SSRF mitigation
2 Overprivileged MI Apply least-privilege RBAC — specific role on specific resource scope; never Contributor on subscription RBAC scoping
3 MI shared across environments Use separate user-assigned MIs per environment; enforce with Azure Policy Environment isolation
4 Cached token after revocation Plan for 1-hour revocation delay; for immediate access blocking, rotate the resource key/secret the MI accesses Token lifecycle
5 IMDS in containers Deploy AKS with Workload Identity (OIDC federation); use NetworkPolicy to block IMDS from pods that don't need it Container isolation
6 Orphaned RBAC Run periodic RBAC audits; use az role assignment list to identify "Unknown" identity assignments RBAC hygiene
7 Token in logs Never log Authorization headers; use structured logging frameworks that redact bearer tokens Log hygiene
8 Fallback to credentials Block connection strings in code via Defender for DevOps secret scanning and branch policies Secret scanning
9 Cross-tenant misunderstanding Document MI cross-tenant limitations; for cross-tenant access use service principal with certificate Architecture guidance
10 Workload Identity misconfiguration Test Workload Identity with kubectl exec and az login --federated-token; validate annotations AKS configuration

Top 3 Critical Mitigations

  1. Least-Privilege RBAC on Managed Identity: A managed identity with broad RBAC (Contributor, Owner) turns any SSRF or RCE vulnerability into a full subscription compromise. Assign the narrowest role on the narrowest scope.
  2. Protect Against IMDS SSRF: IMDS v2 requires a session token (PUT request first), making it harder to exploit via SSRF. Ensure applications use v2 and that SSRF vulnerabilities in application code are fixed.
  3. AKS Workload Identity over Pod-Level IMDS Access: In AKS, IMDS access is shared per node. Use Workload Identity (OIDC federation) so each pod gets its own federated identity without access to the node's IMDS.

Key Takeaways

  • Managed Identity eliminates stored credentials — always prefer it over connection strings, client secrets, and service account passwords.
  • Apply least-privilege RBAC to all managed identities — overprivileged identities turn application compromises into infrastructure takeovers.
  • IMDS SSRF is a real attack vector; use IMDSv2 (session token) and fix SSRF vulnerabilities in application code.
  • AKS Workload Identity provides per-pod identity via OIDC federation — use it instead of shared node IMDS access.
  • Managed identities cannot authenticate cross-tenant; use service principals with certificates for cross-tenant scenarios.
