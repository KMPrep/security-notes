Microsoft Sentinel

Category: Security
Azure Service Type: Cloud-Native SIEM/SOAR
Key Docs: Microsoft Sentinel Overview

Overview

Microsoft Sentinel is a cloud-native SIEM (Security Information and Event Management) and SOAR (Security Orchestration, Automation, and Response) platform built on Azure Log Analytics. It ingests security data from across an organization (Azure, on-premises, multi-cloud, SaaS), correlates events using built-in and custom analytics rules, detects threats using machine learning and threat intelligence, and enables automated response through playbooks (Logic Apps). Sentinel replaces on-premises SIEM appliances with a scalable, pay-per-GB cloud service. It integrates natively with Microsoft Defender products, Entra ID, and hundreds of third-party connectors.

How It Works

Core Concepts

Concept Description
Workspace Azure Log Analytics workspace — the underlying data store for Sentinel
Data Connector Integrations that ingest logs (Defender connectors, Syslog, CEF, REST API, custom)
Analytics Rule Scheduled, ML, Fusion, or Microsoft Security rule that generates incidents
Incident Aggregated group of related alerts; basis for SOC triage and investigation
Playbook Logic App triggered by alert or incident for automated response
Workbook Azure Monitor Workbook for dashboards and visualization of Sentinel data
Hunting Interactive KQL queries for proactive threat hunting across raw data
Entity First-class concept: User, Host, IP, URL, File — enriched across incidents
UEBA User and Entity Behavior Analytics — baseline behavior and anomaly scoring
Threat Intelligence STIX/TAXII feeds and Microsoft TI integrated with analytics rules

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Critical data source not connected Missing connector for a key system Entra ID Sign-In logs are not connected to Sentinel. A credential stuffing attack succeeds — 500 accounts are compromised. Sentinel has no visibility into the authentication events. The incident is discovered via password reset volume on the helpdesk ticketing system, 72 hours after it began. The delay allows the attacker to exfiltrate email from compromised mailboxes. Detection gap from missing data connector High
2 Log retention too short — forensics impossible Default 90-day retention An attacker compromises a VM and maintains persistence for 45 days before conducting the data exfiltration. Incident response begins on day 50. Log Analytics default retention is 90 days. The investigator needs to trace the attacker's activity from day 1 (day 45 before discovery) but those logs are within the 90-day window — barely. Had discovery been delayed 1 more month, the early-stage logs would be gone. Forensic data lost due to retention limits Med
3 Analytics rule tuned to near-zero — alert fatigue suppression Too many alerts causes over-suppression After deployment, Sentinel generates thousands of false positive alerts daily from overly broad rules. Analysts suppress most rules or raise thresholds so high that only obvious attacks generate alerts. The effective detection coverage drops to near zero. A sophisticated attacker using slow, low-volume attack techniques is never detected because the tuned thresholds are above their activity level. Sophisticated attacks undetected post over-tuning High
4 Playbook credentials hardcoded Automation account secrets in Logic App A Sentinel playbook uses a hardcoded client secret for an Entra ID app registration. The Logic App is visible to broad teams in Azure Portal. The client secret is extracted and used to perform unauthorized actions against Entra ID. Lateral escalation from exposed automation credentials Med
5 No incident ownership — incidents go unassigned No assignment automation or on-call process High-severity incidents are created but no assignment rule or on-call rotation exists. Incidents sit in the queue for hours. On weekends, no analyst monitors the queue. An attacker who triggers a high-severity incident on a Saturday evening has 40+ hours of uncontested access before the incident is noticed on Monday morning. Extended dwell time from unassigned incidents High
6 UEBA not enabled Behavior analytics not turned on UEBA is not enabled. An insider threat who legitimately has access to production data starts bulk-downloading files after being informed of upcoming termination. No baseline behavior model exists to detect the anomaly. The activity is only discovered during a post-termination audit, 30 days later. Insider threat undetected without behavioral baselines Med
7 Cross-workspace queries not configured — multi-workspace silos Separate Sentinel workspaces per BU Three business units each have their own Sentinel workspace. A sophisticated attacker moves laterally from BU1 (compromised in workspace 1) to BU3 (only in workspace 3). Each Sentinel instance sees only its own data. No cross-workspace correlation rule exists. The lateral movement pattern is only visible when analysts manually compare incidents from two separate Sentinel instances. Cross-environment lateral movement undetected Med
8 Log ingestion cost spike — uncontrolled data source High-volume connector without volume cap A new verbose Windows Security Event connector is enabled on all Windows VMs without filtering. Security Event ID 4688 (process creation) generates 10,000 events per second per VM across 200 VMs. The daily ingestion jumps from 50 GB to 5 TB. The Log Analytics workspace bill increases 100x, reaching thousands of dollars per day. Uncontrolled cost from verbose log ingestion Med
9 Threat intelligence not integrated TI feed enabled but not used in rules A STIX/TAXII threat intelligence feed is connected to Sentinel's Threat Intelligence table. However, no analytics rule is configured to correlate the TI feed with network or authentication logs. Known attacker IPs and malicious domains in the TI feed are never matched against incoming logs. Connections to known C2 IPs are ingested into Sentinel but never generate alerts. Threat intelligence data unused — no detections Med
10 No hunting queries for active incidents Reactive-only SIEM usage Sentinel is used reactively — analysts only respond to generated incidents. After a nation-state actor compromises the environment using techniques that do not trigger any existing analytics rules (TTPs are novel), the attacker is in the environment for 6 months. No hunting queries are run proactively during this period. Hunting would have surfaced the attacker's persistence mechanism. Persistent threat undetected without proactive hunting Med

Mitigation Controls

# Failure Mitigation Layer
1 Missing data connectors Audit connected sources; prioritize Entra ID, Defender XDR, Azure Activity, NSG Flow Logs, and custom app logs Data ingestion
2 Short retention Set Log Analytics retention to 1–2 years for security tables; use Archive tier for older data Log retention
3 Over-suppression Tune rules using allow-lists and entity exclusions — not threshold raising; measure detection coverage Alert tuning
4 Hardcoded playbook credentials Use Managed Identity for Logic App authentication; never store secrets in Logic App connectors Playbook security
5 Unassigned incidents Create automation rule to auto-assign by severity and source; implement 24/7 on-call rotation Incident management
6 No UEBA Enable UEBA in Sentinel; connect Entra ID and Active Directory as data sources Behavioral analytics
7 Multi-workspace silos Use cross-workspace queries or consolidate into a single Sentinel workspace; use Azure Lighthouse for MSSP Workspace architecture
8 Log cost spike Use Azure Monitor Data Collection Rules (DCR) to filter verbose events at source; set workspace daily caps Cost management
9 TI not used in rules Create Threat Intelligence Matching analytics rule (built-in template); enable Microsoft TI connector TI integration
10 No hunting Schedule weekly proactive hunting sessions; use Sentinel Community hunting queries from GitHub Proactive hunting

Top 3 Critical Mitigations

  1. Connect All Critical Data Sources First: Sentinel is only as good as what it can see. Prioritize Entra ID Sign-In logs, Defender XDR alerts, Azure Activity log, and key application logs before tuning rules.
  2. Automated Incident Assignment + Playbooks for High-Severity: An unassigned high-severity incident is as dangerous as no SIEM. Automation rules can assign, notify, and trigger containment playbooks within minutes.
  3. Managed Identity for Playbooks: Never use hardcoded credentials in Logic App playbooks. Use Managed Identity with RBAC to authorize automation — eliminates a category of credential theft from automation infrastructure.

Key Takeaways

  • Sentinel is built on Log Analytics — data connectors, KQL, and workspace architecture are foundational to effective deployment.
  • Analytics rules must be tuned with allow-lists and entity exclusions — not threshold raising — to maintain detection coverage.
  • UEBA requires connected data sources and time to baseline; plan for 2–4 weeks before meaningful anomaly detection.
  • Playbooks must use Managed Identity, not hardcoded credentials — automation infrastructure is itself an attack surface.
  • Proactive hunting is not optional — sophisticated actors use TTPs that never trigger existing rules; hunting finds them.
