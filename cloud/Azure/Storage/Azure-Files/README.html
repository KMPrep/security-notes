Azure Files

Category: Storage
Azure Service Type: Managed File Shares (SMB / NFS)
Key Docs: Azure Files Overview

Overview

Azure Files provides fully managed cloud file shares accessible via the industry-standard Server Message Block (SMB) and Network File System (NFS) protocols, enabling lift-and-shift of applications that depend on file share access. Shares can be mounted concurrently from Windows, Linux, and macOS, and from on-premises machines via Azure VPN or ExpressRoute. Azure File Sync extends Azure Files to on-premises Windows Server file servers, enabling cloud tiering and centralized backup. Performance tiers include Transaction Optimized, Hot, Cool (standard), and Premium (SSD-backed).

How It Works

Core Concepts

Concept Description
File Share SMB or NFS share inside a storage account; max 100 TiB per share
Azure File Sync Agent on Windows Server that syncs with Azure Files; supports cloud tiering
Storage Account Key Full admin access to all shares — equivalent to root credentials
Kerberos Authentication Entra ID Kerberos or on-premises AD DS auth for identity-based share access
Share-level RBAC Entra ID RBAC for coarse-grained share permissions
NTFS ACLs File and directory-level permissions within an SMB share
Private Endpoint Private IP for file share access; eliminates public endpoint exposure
Snapshots Point-in-time read-only copy of a file share
Soft Delete Recover deleted shares and files within retention period
Large File Shares Shares up to 100 TiB; requires specific storage account type

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Storage account key used for mount SMB mount uses account key instead of identity Applications mount the Azure File Share using the storage account key embedded in a mount credential script. The script is stored in plaintext on the VM or in a configuration management repository. Any user with read access to the script retrieves the key and gains full access to all shares in the account. Full storage account compromise High
2 SMB port 445 blocked by ISP Port 445 blocked on outbound internet connections On-premises machines attempt to mount Azure Files over SMB port 445. The corporate or ISP firewall blocks outbound port 445. The mount fails silently with "Access Denied" or connection timeout errors. The team cannot lift and shift the file server workload without VPN or ExpressRoute. Connectivity failure for on-premises access Med
3 No identity-based auth — NTFS ACLs not enforced Mount without AD auth bypasses file-level permissions Azure Files is mounted using a storage account key rather than AD DS or Entra ID Kerberos auth. Since key-based mounts give full access, NTFS ACLs on files and directories are not enforced. All users who can mount the share get full read and write access to all files, bypassing intended file-level security boundaries. File-level authorization bypass High
4 Unencrypted SMB in transit Encryption in transit not required A file share is configured to allow SMB 2.x connections without encryption. An attacker on the same network segment (or via a compromised network device) intercepts SMB traffic and captures file contents or NTLM authentication hashes, which can be relayed or cracked offline. Data interception and credential theft Med
5 Large file share snapshot accumulation Snapshots not managed — storage cost escalates A backup solution creates hourly snapshots of a 50 TiB file share without a retention policy. After 90 days, 2,160 snapshots have accumulated. Storage costs for snapshots exceed the cost of the primary data. More critically, the maximum 200 snapshot limit per share is reached and new backups fail silently. Backup failure and cost explosion Med
6 File share quota not set Uncontrolled share growth fills storage account No quota is set on the file share. An application writes large temporary files and fails to clean them up. The share grows to the storage account limit, causing all writes to fail, including writes from other applications using the same storage account. Storage exhaustion and multi-app failure Med
7 Azure File Sync conflict — split-brain Concurrent edits on multiple sync endpoints Two Windows Servers connected to the same Azure File Sync share edit the same file simultaneously. File Sync detects a conflict and creates a conflict copy. Over time, thousands of conflict files accumulate. Users cannot tell which copy is authoritative, leading to data inconsistency and user confusion. Data inconsistency from sync conflicts Med
8 No soft delete or snapshots Accidental deletion unrecoverable A user deletes a critical directory from a mounted share. No snapshots exist and soft delete is not enabled. The deletion is permanent. The organization has no recovery path and the data loss requires re-creation from scratch. Permanent data loss High
9 Public file share endpoint accessible No private endpoint; share reachable from internet Azure Files public endpoint is enabled. An attacker who obtains a storage account key (from a leaked secret or misconfigured environment) can access the file share from any internet location. With a private endpoint, access would be limited to the VNet even with valid credentials. Remote exfiltration with stolen key Med
10 File Sync agent not updated Outdated agent on Windows Server causes sync errors The Azure File Sync agent on a Windows Server is not updated. A new feature or protocol requirement in the cloud endpoint is incompatible with the old agent version. Sync silently fails. On-premises users continue to create and modify files that are not synced to Azure, causing the cloud copy to diverge and backup gaps to accumulate. Data drift and backup gap Med

Mitigation Controls

# Failure Mitigation Layer
1 Account key in mount script Use AD DS or Entra ID Kerberos identity-based authentication Identity-based auth
2 Port 445 blocked Use Azure VPN Gateway or ExpressRoute for on-premises connectivity Network configuration
3 No NTFS ACL enforcement Enable AD DS or Entra ID Kerberos; NTFS ACLs are only evaluated with identity-based mounts Authentication requirement
4 Unencrypted SMB Require SMB 3.0 with encryption; set Secure transfer required on storage account Encryption in transit
5 Snapshot accumulation Use Azure Backup for file share with retention policies; set snapshot count limit Backup management
6 No share quota Set file share quota commensurate with expected data volume Storage governance
7 Sync conflicts Design workflows to avoid concurrent edits; monitor conflict files via Azure Monitor Sync design
8 No recovery option Enable share soft delete with adequate retention; create daily snapshots via Azure Backup Data protection
9 Public endpoint Configure private endpoint; disable public endpoint for production shares Network controls
10 Stale File Sync agent Use Azure Update Manager or WSUS to keep File Sync agent updated Patch management

Key Takeaways

  • Azure Files provides managed SMB/NFS shares; enable identity-based auth (AD DS or Entra ID Kerberos) to enforce NTFS ACLs.
  • Require SMB 3.0 with encryption (Secure transfer required) to prevent in-transit interception.
  • Enable soft delete and Azure Backup snapshots for all production shares — key-only mounts bypass file-level ACLs.
  • Private endpoints are required for production; public endpoints expose shares to internet-based credential attacks.
  • Azure File Sync extends shares to on-premises servers with cloud tiering for cost optimization.
