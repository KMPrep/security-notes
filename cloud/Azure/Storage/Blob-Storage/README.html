Azure Blob Storage

Category: Storage
Azure Service Type: Object Storage
Key Docs: Blob Storage Overview

Overview

Azure Blob Storage is Microsoft's massively scalable object store for unstructured data — files, images, videos, backups, log archives, and machine learning datasets. A storage account contains containers (analogous to S3 buckets), which hold blobs. Three blob types exist: Block Blobs (general purpose), Append Blobs (log files), and Page Blobs (VM disks, random I/O). Access tiers (Hot, Cool, Cold, Archive) allow cost optimization by data access frequency. Features include lifecycle management policies, immutable blob storage (WORM), versioning, soft delete, static website hosting, and Azure CDN integration.

How It Works

Core Concepts

Concept Description
Storage Account Top-level resource; defines redundancy (LRS, ZRS, GRS, GZRS) and access tier
Container Namespace for blobs; access level: private, blob (anonymous read), or container
Blob Object stored in a container; up to 190.7 TiB for block blobs
Access Tier Hot (frequent), Cool (infrequent), Cold (rare), Archive (offline)
SAS Token Shared Access Signature — time-limited, scoped URL for delegated access
Stored Access Policy Named policy backing a SAS; revocable by deleting the policy
Lifecycle Management Rules to transition blobs between tiers or delete based on age
Immutable Storage WORM (time-based or legal hold) — blobs cannot be modified or deleted
Soft Delete Recovers deleted blobs within a retention period
Private Endpoint Inbound blob access via private IP; disables public endpoint

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Public blob container — anonymous read Container access level set to Blob or Container A developer creates a container to share files with a partner and sets the access level to "Container" (anonymous read + list). They forget to revert it. An internet scanner finds the publicly readable container at the predictable <account>.blob.core.windows.net/<container> URL. The attacker lists all blobs and downloads customer PII, financial records, or internal documents. Mass data exposure High
2 Storage account key exposure Access keys stored in code or leaked via git A developer hardcodes the storage account key in an application config file and commits it to a public GitHub repository. A GitHub Actions secret scanner flags it, but not before a threat actor using automated GitHub monitoring tools clones the key. The attacker uses az storage blob download-batch with the key to exfiltrate every blob in the account. Full account access — read, write, delete High
3 Overly permissive SAS token SAS with no IP restriction, long expiry, full permissions A SAS token is generated with rwdlac (all permissions) and a 1-year expiry and embedded in a mobile app binary. A researcher reverse-engineers the app, extracts the SAS URL, and uses it to enumerate all containers, read all blobs, and delete arbitrary objects. Since the SAS is not backed by a stored access policy, it cannot be revoked before expiry. Full storage access until expiry High
4 No immutable storage for compliance data Audit logs or compliance records can be deleted An audit logging system writes logs to blob storage without WORM protection. An insider with Storage Blob Data Contributor role deletes log files to cover a fraud investigation trail. Without immutability or legal hold, the deletion is permanent and unrecoverable after the soft-delete retention window. Evidence tampering, compliance failure Med
5 Lifecycle policy deletes data prematurely Misconfigured rule deletes data before retention period ends A lifecycle policy is created to delete blobs older than 30 days to reduce costs. A misconfigured rule pattern matches production data files (not just temp files) and deletes them after 30 days. The data loss is discovered months later when a business process attempts to access archived records. Unintended data deletion Med
6 Shared key auth not disabled Apps use storage key when managed identity should be used Disabling shared key auth forces all clients to use Entra ID (managed identity or service principal). If shared key auth remains enabled, a leaked storage key bypasses all RBAC controls — even if Entra ID was intended as the only auth path. RBAC bypass via key auth Med
7 No private endpoint — data exfiltration path Storage reachable from internet A compromised application on an Azure VM can exfiltrate data to an attacker-controlled storage account over the internet because no egress restrictions prevent it. Similarly, an attacker with valid credentials can download data from the internet rather than only from the VNet. Data exfiltration via internet Med
8 Cross-origin resource sharing (CORS) misconfiguration CORS allows all origins A static website hosted on blob storage has CORS configured as AllowedOrigins: *. An attacker crafts a malicious web page that makes authenticated cross-origin requests to the storage account from a victim's browser session, reading blobs the victim has access to (confused deputy). CORS-based data theft Med
9 No blob versioning — overwrites unrecoverable Container has no versioning enabled A ransomware attack or script error overwrites all blobs with encrypted or empty content. Without versioning, the original content cannot be recovered. With soft delete but no versioning, deleted blobs recover but overwritten blobs do not. Unrecoverable data overwrite High
10 Archive-tier blob access latency misunderstood Archive rehydration not accounted for in RTO A disaster recovery plan assumes all blobs are available on demand. Critical backup blobs are in the Archive tier. When recovery is triggered, the team discovers that Archive rehydration takes 1-15 hours depending on priority. The RTO is breached before data is accessible. RTO breach on disaster recovery Med

Mitigation Controls

# Failure Mitigation Layer
1 Public container Set all containers to private; disable anonymous access at storage account level Container access policy
2 Storage key exposure Use managed identity for all access; rotate keys via Key Vault; enable Defender for Storage Key management
3 Permissive SAS token Back SAS with Stored Access Policy (revocable); limit permissions, expiry, and IP range SAS policy
4 No immutable storage Enable time-based immutability or legal hold on compliance containers Immutable storage
5 Lifecycle policy misconfiguration Test lifecycle policies against non-production containers first; use prefix filters carefully Lifecycle management
6 Shared key auth Disable shared key authorization (allowSharedKeyAccess: false); enforce Entra ID only Authentication policy
7 No private endpoint Configure private endpoint and disable public network access Network controls
8 CORS misconfiguration Restrict AllowedOrigins to specific app domains; never use * for authenticated origins CORS policy
9 No versioning Enable blob versioning and soft delete; test recovery procedures Data protection
10 Archive tier RTO Keep DR-critical blobs in Cool/Cold tier; document rehydration time in RTO planning Tiering strategy

Top 3 Critical Mitigations

  1. Disable anonymous access at account level and set containers to private: Prevents the most common mass data exposure from misconfigured container ACLs with a single account-level setting.
  2. Managed identity + disable shared key auth: Eliminates all static credential risk. RBAC-controlled managed identity is the only authorized access path, and key compromise cannot bypass it.
  3. Blob versioning + immutable storage: Prevents ransomware overwrites and insider deletion. Versioning recovers overwrites; WORM prevents deletion of compliance records.

Key Takeaways

  • Blob Storage is Azure's primary object store; disable anonymous access and shared key auth for all production accounts.
  • Use managed identity with Storage Blob Data Reader/Contributor roles instead of storage account keys or SAS tokens.
  • Enable versioning + soft delete to survive ransomware and accidental deletion.
  • Immutable storage (WORM) satisfies compliance requirements for audit logs and financial records.
  • Private endpoints eliminate the public internet attack surface for storage access.
