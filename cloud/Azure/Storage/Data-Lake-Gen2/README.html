Azure Data Lake Storage Gen2

Category: Storage
Azure Service Type: Hierarchical Namespace Object Store
Key Docs: ADLS Gen2 Overview

Overview

Azure Data Lake Storage Gen2 (ADLS Gen2) combines Azure Blob Storage with a hierarchical namespace (HNS), enabling true directory semantics, atomic rename operations, and POSIX-compatible access control lists (ACLs). ADLS Gen2 is the foundation for Azure analytics workloads — Azure Synapse Analytics, Azure Databricks, Azure HDInsight, and Azure Data Factory all natively integrate with it. It supports fine-grained ACLs at the file and directory level layered on top of Azure RBAC. Enabling HNS on a storage account converts it to a Data Lake endpoint alongside the standard blob endpoint.

How It Works

Core Concepts

Concept Description
Hierarchical Namespace (HNS) True directory structure; atomic rename and delete; required for ADLS Gen2
Filesystem Top-level container in ADLS Gen2 (equivalent to container in Blob)
Directory First-class object when HNS is enabled; atomic operations at directory level
POSIX ACLs Named ACLs on files and directories for fine-grained owner/group/other permissions
Azure RBAC Coarse-grained role assignments at account, filesystem, directory, or file level
Service Principal / Managed Identity Identity-based access for Spark, Databricks, Synapse connectors
Shared Access Signature (SAS) Token-based delegated access scoped to path and operations
Storage Explorer GUI for navigating and managing ADLS Gen2 directory trees and ACLs
Soft Delete Recovers deleted blobs and versions within retention period
Lifecycle Management Tier data to Cool/Archive or delete based on last modified time

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Overly permissive filesystem ACL Root-level ACL grants read to all authenticated users An ADLS Gen2 filesystem has a default ACL that grants r-x to all service principals in the tenant for the root directory. A data scientist who should only see the curated zone also has read access to the raw zone, which contains unmasked PII and regulated financial data. Unauthorized access to sensitive raw data High
2 Storage account key used in Spark config Key embedded in Databricks or Synapse Spark config A Spark notebook hardcodes the storage account key for ADLS access as spark.conf.set("fs.azure.account.key...": "<key>"). The notebook is committed to a shared git repository. Any team member with repo access retrieves the key and can download the entire data lake. Full data lake exfiltration High
3 No zone separation — flat data lake All data zones share one filesystem with weak ACLs Raw, curated, and serving zone data is stored in a single filesystem with no ACL hierarchy enforcement. A BI analyst with access to the serving zone can navigate the directory tree and access raw data directories. The flat structure does not enforce data classification boundaries. Zone boundary bypass Med
4 ACL mask not set correctly Default ACL mask allows broader permissions than intended A directory default ACL has mask rwx but owner group is r-x. When a new file is created, the effective permission for the group is r-x AND mask = r-x, which appears correct. However, the mask is changed to rwx by a file copy operation, allowing group write access where none was intended. Unintended write access after file copy Med
5 Shared service principal used for all pipelines Single SP with Storage Blob Data Contributor on root All ETL pipelines use the same service principal. One pipeline is compromised via a SQL injection in a transformation step. The attacker uses the SP's credential to access the ADLS root, reading all zones and all datasets across every team. Full data lake access via single compromised SP High
6 No private endpoint — data accessible from internet ADLS public endpoint not disabled ADLS is accessible from the internet. An attacker with a leaked storage account key or compromised service principal credential can download all data from any location. Without a private endpoint, network-level restrictions cannot supplement identity-based controls. Data exfiltration from any internet source Med
7 Uncontrolled Spark cluster access to sensitive data Cluster runs with shared identity and shared storage mount A shared Databricks cluster has ADLS mounted at the cluster level, making all datasets accessible to all users running notebooks on the cluster. A user who should only see sales data can run dbutils.fs.ls on sensitive HR or finance directories. Cross-team data access via shared cluster mount High
8 No soft delete — bulk delete unrecoverable A pipeline bug deletes a full directory An ETL pipeline has a bug where the cleanup step deletes the wrong output directory. The directory contains a month of curated analytics data. Without soft delete enabled, the deletion is permanent. Reprocessing from raw data takes 72 hours, exceeding the SLA. Data loss, SLA breach Med
9 Sensitive data in file names or paths File paths contain PII An ingestion pipeline writes files with paths like /patients/<SSN>/lab_results.csv. These paths are stored in audit logs, query metadata, and Spark executor logs. An attacker or nosy employee who can read logs extracts SSNs from path patterns without needing to open any file. PII leakage via metadata Med
10 Retention policy removes data before analytics completes Lifecycle policy archives data too aggressively A lifecycle policy archives blobs older than 7 days to Cool tier. An analytics pipeline runs weekly and reads week-old data. Some files have been archived and the ReadBlobFromArchive call fails or incurs high latency, causing the analytics job to fail or exceed its SLA window. Pipeline failure from premature tiering Med

Mitigation Controls

# Failure Mitigation Layer
1 Overly permissive root ACL Apply principle of least privilege ACLs per zone; audit ACLs regularly with Storage Explorer or PowerShell POSIX ACL design
2 Account key in Spark config Use managed identity or service principal with OAuth; never use account key in Spark configs Identity-based access
3 Flat data lake Create separate filesystems per data zone; assign dedicated service principals per zone Data lake zone design
4 ACL mask error Understand mask semantics; test ACL inheritance with representative files; use Storage Explorer for ACL inspection ACL governance
5 Shared service principal Use dedicated managed identity or SP per pipeline with zone-scoped RBAC or ACL Identity isolation
6 No private endpoint Configure private endpoint; disable public network access on storage account Network controls
7 Shared cluster access Use cluster-level access controls; mount paths with read-only credentials; consider credential passthrough (AAD passthrough) Databricks access control
8 No soft delete Enable HNS soft delete with adequate retention period Data protection
9 PII in file paths Use surrogate keys in paths; separate PII identifiers from file path structure Data design
10 Lifecycle archives active data Align lifecycle policy timing to analytics pipeline schedules Lifecycle policy design

Key Takeaways

  • ADLS Gen2 is Blob Storage with hierarchical namespace — true directories with atomic operations and POSIX ACLs.
  • Use managed identity or service principal with OAuth; never use storage account keys in Spark or pipeline configs.
  • Design separate filesystems per data zone and assign dedicated identities per zone for strict data access boundaries.
  • Enable soft delete and align lifecycle policies to pipeline schedules to prevent accidental data loss.
  • Private endpoints eliminate internet-based data exfiltration paths for all storage interactions.
