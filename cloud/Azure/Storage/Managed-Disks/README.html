Azure Managed Disks

Category: Storage
Azure Service Type: Block Storage (IaaS)
Key Docs: Managed Disks Overview

Overview

Azure Managed Disks are block-level storage volumes managed by Azure for use with Azure Virtual Machines. Azure handles disk storage account management, redundancy, and allocation — you simply specify the disk type, size, and performance tier. Disk types include Ultra Disk (sub-millisecond latency, configurable IOPS), Premium SSD v2, Premium SSD, Standard SSD, and Standard HDD. Disks support OS and data roles, snapshots, incremental snapshots, shared disks (for clustered workloads), and encryption via platform-managed or customer-managed keys.

How It Works

Core Concepts

Concept Description
Disk Type Ultra, Premium SSD v2, Premium SSD, Standard SSD, Standard HDD — performance vs cost tradeoff
Managed Disk Azure manages the underlying storage account; you interact only with the disk resource
Snapshot Point-in-time copy of a disk; full or incremental
Disk Encryption Platform-managed keys (PMK) by default; customer-managed keys (CMK) via Key Vault
Disk Export / SAS URL Temporary URL to download a disk image; must be managed carefully
Shared Disk Single disk attached to multiple VMs simultaneously (for WSFC, Pacemaker clusters)
Disk Bursting Credit-based IOPS bursting for Premium SSD to handle transient spikes
Azure Disk Encryption (ADE) OS-level encryption using BitLocker (Windows) or dm-crypt (Linux) with Key Vault keys
Incremental Snapshot Captures only changes since the last snapshot; significantly reduces snapshot storage cost
Ultra Disk Dynamically configurable IOPS and throughput; zone-pinned; sub-millisecond latency

Step-by-Step Flow — Encrypted Disk Attach

sequenceDiagram
    participant ARM as Azure Resource Manager
    participant KV as Key Vault
    participant Disk as Managed Disk
    participant VM as Virtual Machine
    ARM->>KV: Wrap disk encryption key (DEK) with key encryption key (KEK)
    KV-->>ARM: Wrapped DEK
    ARM->>Disk: Provision disk with CMK encryption set
    ARM->>VM: Attach disk
    VM->>KV: Unwrap DEK using VM managed identity
    KV-->>VM: DEK
    VM->>Disk: Mount and use encrypted disk

What Can Go Wrong

# Failure / Attack Root Cause Attack Scenario Impact Likelihood
1 Orphaned disk attached to attacker VM Disk not deleted after VM decommission A VM is deleted but the OS disk is retained as an orphaned managed disk resource. An attacker or insider with Contributor access to the resource group creates a new VM, attaches the orphaned disk as a data disk, and mounts it in read-write mode. All data on the disk — including application data, credentials in config files, and OS user databases — is accessible. Full disk data disclosure High
2 SAS URL for disk export not revoked Disk access URL shared or logged An operator generates a SAS URL to export a disk for cloning and pastes it in a Slack channel. The URL has a 24-hour expiry but is in a public Slack workspace. An external party finds the URL and downloads the full disk image (potentially hundreds of GB) before expiry. Full disk data disclosure via URL leak Med
3 No encryption at rest — CMK Disks use platform-managed keys only; CMK not enforced A regulated workload stores PHI on disk. Platform-managed key encryption protects the physical medium but does not satisfy the "bring your own key" compliance requirement. The organization cannot demonstrate key custody, key rotation history, or immediate key revocation to auditors. Compliance failure for regulated data Med
4 CMK key deleted without disk migration Key Vault key deleted while disks still use CMK An operator deletes a Key Vault key that backs an active disk encryption set. All VMs using those disks immediately fail to start or read data because the DEK cannot be unwrapped. The entire workload is unavailable until the key is recovered from Key Vault's soft-deleted state. Full workload outage High
5 Ultra Disk zone pinning causes AZ failure blast radius Ultra Disk locked to single Availability Zone Ultra Disks must be deployed in the same Availability Zone as the attached VM. If the zone experiences an outage, both the VM and the Ultra Disk are affected simultaneously. There is no cross-zone failover option for Ultra Disk-backed stateful workloads. Full AZ outage = full workload outage Med
6 Disk bursting exhausted silently Burst credits depleted without application awareness A Premium SSD in burst mode handles a daily traffic spike. Burst credits are exhausted. The disk drops to its baseline IOPS, which is 20% of the burst peak. The application has no metric alerting on disk IOPS. Database query times degrade from 5ms to 200ms, causing slow UI responses without obvious root cause. Silent performance degradation Med
7 Shared disk — split-brain Shared disk used without proper cluster fencing A shared disk is attached to two Windows VMs for a failover cluster. The cluster fencing (quorum) mechanism fails. Both nodes believe they are active and write to the disk simultaneously. The file system is corrupted because both nodes modify the same blocks without coordination. Disk corruption and data loss Med
8 Disk snapshot exfiltration Snapshot shared publicly or with wrong subscription An operator creates a disk snapshot for a support ticket and shares it with an external party by setting --grant-access-level Read and not setting an expiry. The snapshot contains a full disk image with customer PII. The external party downloads and analyzes the disk contents beyond the intended scope. Customer data disclosure Med
9 Incremental snapshot chain broken Dependency on prior snapshot deleted Incremental snapshots depend on a parent snapshot chain. An operator deletes an intermediate snapshot in the chain to save cost. The dependent incremental snapshots become orphaned or invalid. Backup recovery requires falling back to a much older full snapshot, exceeding the RPO. Recovery point objective breach Med
10 ADE Key Vault access lost during failover Key Vault not accessible in DR region A DR plan uses Azure Site Recovery to replicate VMs to a secondary region. The Key Vault holding ADE keys is in the primary region and is not replicated or accessible cross-region. During a regional failure, recovered VMs in the secondary region cannot unwrap the DEK and fail to boot. DR recovery failure for encrypted VMs High

Mitigation Controls

# Failure Mitigation Layer
1 Orphaned disk Enforce Azure Policy to require disk deletion with VM; audit orphaned disk resources regularly Policy + governance
2 SAS URL leak Set minimum required expiry on SAS URLs; revoke by revoking disk access immediately after use Disk access management
3 No CMK Enforce CMK disk encryption sets via Azure Policy; use Key Vault with RBAC authorization Encryption policy
4 CMK key deleted Enable Key Vault soft-delete and purge protection; restrict key delete permission to break-glass accounts Key Vault configuration
5 Ultra Disk zone failure Use Premium SSD for cross-zone-compatible workloads; design stateful failover with geo-redundant storage Architecture design
6 Burst credits exhausted Monitor Disk IOPS Consumed Percentage metric; set alerts at 80% baseline IOPS Observability
7 Shared disk split-brain Use proper cluster fencing (SCSI PR); test failover; use Azure witness for quorum Cluster configuration
8 Snapshot exfiltration Set snapshot access expiry; do not share snapshots across subscriptions without privacy review Snapshot governance
9 Broken snapshot chain Document snapshot dependencies; use Azure Backup instead of manual snapshots for managed retention Backup management
10 ADE Key Vault not in DR Replicate Key Vault keys to DR region Key Vault; configure Site Recovery with DR-region Key Vault reference DR design

Key Takeaways

  • Managed Disks are Azure-managed block storage; use Premium SSD or Ultra Disk for performance-sensitive workloads.
  • Always delete disks when VMs are decommissioned — orphaned disks are a common source of data exposure.
  • CMK encryption with Key Vault + purge protection satisfies BYOK compliance requirements.
  • Incremental snapshots reduce backup cost significantly but require a valid snapshot chain for recovery.
  • Plan Key Vault DR replication for ADE to ensure VMs can boot after a regional failover.
