GKE Security — Google Kubernetes Engine Hardening Deep-Dive

Domain: Cloud / Managed Kubernetes (GCP Compute)
Frameworks: CIS GKE Benchmark, NSA/CISA Kubernetes Hardening Guide, Google Safe/Secure GKE guidance, Pod Security Standards, MITRE ATT&CK for Containers
Related notes: cloud/Kubernetes-Security (generic k8s — this note extends it for GKE), cloud/AWS/Compute/EKS, cloud/Azure/Compute/AKS, infra/Virtualization/Escape-Techniques

This note assumes the generic Kubernetes control-plane, RBAC, admission, NetworkPolicy, and secrets material in cloud/Kubernetes-Security. It does not re-teach RBAC verbs, the API-server pipeline, or Pod Security Standards from scratch. It focuses on the GCP-specific layer: the shared-responsibility split GKE gives you, Workload Identity Federation, private clusters and control-plane authorized networks, Shielded/Confidential nodes, GKE Sandbox (gVisor), Binary Authorization, Dataplane V2 network policy, and the metadata/node-service-account attack surface that is unique to running Kubernetes on Google Cloud.

1. Overview

GKE is Google Cloud's managed Kubernetes offering. Google runs and secures the control plane (API server, etcd, controller-manager, scheduler) inside a Google-owned project; you consume it through a regional or zonal endpoint. GKE ships in two operational modes — Standard (you manage node pools, machine types, and node-level config) and Autopilot (Google manages nodes and enforces a hardened baseline you cannot weaken). The security implication is that GKE moves the shared-responsibility line: the control plane is Google's problem, but node configuration, IAM, Workload Identity, network exposure, admission policy, and workload posture remain yours. Most real GKE breaches are not control-plane CVEs — they are an over-permissioned node service account plus a pod that can reach the GCE metadata server, converting one pod RCE into full project compromise.

At scale, GKE security is about closing the cloud pivot: a Kubernetes cluster is a soft target sitting inside a Google Cloud project, and the crown jewels are the IAM permissions the cluster's identity can reach. GKE gives you purpose-built controls for exactly this — Workload Identity Federation (per-pod GCP identity instead of node credentials), metadata concealment, private clusters, Shielded/Confidential nodes, Binary Authorization, and GKE Sandbox — but almost none are on by default in Standard mode. Autopilot flips many defaults to secure. A senior engineer must know which controls are default-on in each mode, which attack path each closes, and the operational trade-offs.

TL;DR
- Workload Identity Federation is the single most important GKE control — it binds a Kubernetes ServiceAccount (KSA) to a Google ServiceAccount (GSA) via GKE's OIDC/metadata layer, so pods get short-lived, per-workload GCP credentials instead of stealing the node's over-broad service account. Enabling it also flips the metadata server to a concealing GKE Metadata Server.
- The GKE cloud-pivot chain: pod RCE → reach 169.254.169.254 metadata → steal the node service account token (often Editor) → own the whole project. Break it with Workload Identity + Metadata Concealment + least-privilege node SA + NetworkPolicy egress deny.
- Autopilot > Standard for security posture: Autopilot enforces Shielded nodes, Workload Identity, no privileged pods, no host namespaces, and a restricted-like admission baseline by default; you trade node flexibility for a hardened floor.
- Private cluster + control-plane authorized networks removes the public API endpoint and restricts who can even reach the control plane.
- Defense in depth is layered on: Binary Authorization (only signed images run), GKE Sandbox/gVisor (untrusted workloads), Confidential GKE Nodes (memory encryption), node auto-upgrade (patch the escape CVEs), Dataplane V2 (eBPF NetworkPolicy + logging).

2. How It Works

2.1 Core Mechanics

GKE reshapes the standard Kubernetes trust model by taking ownership of the control plane and layering GCP identity and networking primitives underneath the cluster. The security-relevant objects:

GKE Concept What It Is Primary Security Concern
Autopilot mode Google manages nodes; hardened baseline enforced, per-pod billing Cannot run privileged/hostPath pods; secure-by-default floor
Standard mode You own node pools, machine types, node config Every hardening flag is opt-in; easy to misconfigure
Node service account GCE service account attached to worker VMs Over-permissioning (Editor) = project takeover on escape
Workload Identity Federation Maps KSA → GSA via GKE metadata + OIDC Correct binding removes need for node-SA credentials
GKE Metadata Server Per-node concealing proxy in front of GCE metadata Prevents pods reading the raw node SA token
Private cluster Nodes (and optionally control plane) have no public IP Reduces internet-facing attack surface
Control-plane authorized networks CIDR allow-list for who can reach the API endpoint Stops internet-wide API brute force / exploit
Shielded GKE Nodes Secure Boot + vTPM + integrity monitoring on node VMs Rootkit/boot-level tamper detection
Confidential GKE Nodes AMD SEV / Intel TDX memory encryption for node VMs Protects data-in-use from host/hypervisor
GKE Sandbox gVisor userspace kernel via RuntimeClass Contains container escape for untrusted workloads
Binary Authorization Admission gate requiring image attestations/signatures Blocks unsigned/untrusted images at deploy
Dataplane V2 Cilium/eBPF datapath; enforces NetworkPolicy + flow logs Enables policy + observability without kube-proxy
Node auto-upgrade Auto-patches node OS and kubelet to fix CVEs Unpatched nodes = escape/CVE exposure
GKE audit logs Admin Activity + Data Access via Cloud Audit Logs Detection source for API abuse

Two GKE-specific facts drive most of the threat model. First, every GKE node is a Compute Engine VM, so it exposes the GCE metadata server at http://metadata.google.internal (169.254.169.254), which by default returns the node's attached service-account access token. Any pod that can make an HTTP request to that address inherits the node's IAM — this is the crux of GKE's attack surface. Second, GKE historically shipped nodes with the default Compute Engine service account, which carries the project Editor role. On a cluster with default settings, a single pod RCE that can reach the metadata endpoint yields project-Editor credentials. Workload Identity plus Metadata Concealment plus a minimal-scope node SA is the canonical fix.

2.2 Concrete Configuration

A. A hardened Standard cluster in one gcloud command. Each flag closes a specific attack path (commented):

gcloud container clusters create prod-hardened \
  --project my-prod-project \
  --region us-central1 \
  --release-channel regular \                       # managed, tested auto-upgrades
  --enable-private-nodes \                           # nodes have no public IP
  --enable-private-endpoint \                        # control plane is private-only
  --master-ipv4-cidr 172.16.0.32/28 \               # control-plane peering range
  --enable-master-authorized-networks \             # restrict who can hit the API
  --master-authorized-networks 10.8.0.0/24 \        # e.g. bastion/CI CIDR only
  --workload-pool my-prod-project.svc.id.goog \     # ENABLE Workload Identity Federation
  --enable-shielded-nodes \                          # Secure Boot + vTPM + integrity
  --shielded-secure-boot \
  --shielded-integrity-monitoring \
  --enable-dataplane-v2 \                            # eBPF datapath (NetworkPolicy + logs)
  --enable-network-policy \                          # enforce NetworkPolicy objects
  --binauthz-evaluation-mode=PROJECT_SINGLETON_POLICY_ENFORCE \  # Binary Authorization
  --enable-intra-node-visibility \                  # log intra-node pod traffic
  --no-enable-legacy-authorization \                # RBAC only, no ABAC
  --metadata disable-legacy-endpoints=true \        # kill /0.1/ & /v1beta1/ metadata
  --service-account gke-node-minimal@my-prod-project.iam.gserviceaccount.com \  # least-priv node SA
  --enable-secret-manager \                          # Secret Manager CSI add-on
  --enable-master-global-access=false \
  --logging=SYSTEM,WORKLOAD --monitoring=SYSTEM

The key security flags: --workload-pool turns on Workload Identity (and switches nodes to the concealing GKE Metadata Server); --enable-private-nodes + --enable-private-endpoint + --enable-master-authorized-networks remove public exposure of both nodes and the API; --enable-shielded-nodes gives verified boot; --metadata disable-legacy-endpoints=true kills the v1beta1 metadata paths that ignored the required Metadata-Flavor: Google header (the old SSRF-friendly ones); and --service-account swaps the dangerous default Editor-scoped SA for a minimal one.

B. Create a least-privilege node service account (never reuse the default Compute Engine SA):

gcloud iam service-accounts create gke-node-minimal \
  --display-name "GKE minimal node SA"

# Only the three roles nodes genuinely need to function:
for R in roles/logging.logWriter roles/monitoring.metricWriter \
         roles/monitoring.viewer; do
  gcloud projects add-iam-policy-binding my-prod-project \
    --member "serviceAccount:gke-node-minimal@my-prod-project.iam.gserviceaccount.com" \
    --role "$R"
done
# Add roles/artifactregistry.reader ONLY if pulling from Artifact Registry.

C. Bind a Kubernetes ServiceAccount to a Google ServiceAccount (Workload Identity). This is the canonical example every senior candidate should be able to reproduce. The goal: pods using KSA analytics-ksa in namespace data can call BigQuery as GSA bq-reader, and nothing else in the cluster gets that permission.

# 1. Create the GSA that carries the ACTUAL app permissions (scoped tight)
gcloud iam service-accounts create bq-reader
gcloud projects add-iam-policy-binding my-prod-project \
  --member "serviceAccount:bq-reader@my-prod-project.iam.gserviceaccount.com" \
  --role roles/bigquery.dataViewer

# 2. Allow the specific KSA to impersonate the GSA via the workload pool.
#    The member string is the magic: it encodes cluster workload-pool + ns + KSA name.
gcloud iam service-accounts add-iam-policy-binding \
  bq-reader@my-prod-project.iam.gserviceaccount.com \
  --role roles/iam.workloadIdentityUser \
  --member "serviceAccount:my-prod-project.svc.id.goog[data/analytics-ksa]"

# 3. Create the KSA and annotate it with the target GSA.
apiVersion: v1
kind: ServiceAccount
metadata:
  name: analytics-ksa
  namespace: data
  annotations:
    iam.gke.io/gcp-service-account: bq-reader@my-prod-project.iam.gserviceaccount.com
---
# 4. Pods that reference the KSA now get short-lived bq-reader tokens from the
#    GKE Metadata Server. They CANNOT read the node SA or any other GSA.
apiVersion: apps/v1
kind: Deployment
metadata:
  name: analytics
  namespace: data
spec:
  template:
    spec:
      serviceAccountName: analytics-ksa       # binds the identity
      containers:
        - name: app
          image: us-docker.pkg.dev/my-prod-project/apps/analytics@sha256:abc123...
          securityContext:
            runAsNonRoot: true
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities: { drop: ["ALL"] }
            seccompProfile: { type: RuntimeDefault }

D. Metadata Concealment / block the raw metadata path (belt-and-suspenders even with Workload Identity). A NetworkPolicy egress deny to the metadata IP, plus for legacy clusters the --workload-metadata=GKE_METADATA node-pool config:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-metadata-egress
  namespace: data
spec:
  podSelector: {}
  policyTypes: ["Egress"]
  egress:
    - to:
        - ipBlock:
            cidr: 0.0.0.0/0
            except:
              - 169.254.169.254/32    # block GCE metadata server
    - to:                             # still allow kube-dns
        - namespaceSelector: {}
          podSelector:
            matchLabels: { k8s-app: kube-dns }
      ports:
        - { protocol: UDP, port: 53 }
        - { protocol: TCP, port: 53 }

E. GKE Sandbox (gVisor) for an untrusted workload via RuntimeClass on a sandboxed node pool:

gcloud container node-pools create sandbox-pool \
  --cluster prod-hardened --region us-central1 \
  --sandbox type=gvisor --machine-type e2-standard-4

apiVersion: v1
kind: Pod
metadata: { name: untrusted-plugin, namespace: tenant-x }
spec:
  runtimeClassName: gvisor        # syscalls intercepted in userspace
  containers:
    - name: plugin
      image: us-docker.pkg.dev/my-prod-project/apps/plugin@sha256:def456...

F. Binary Authorization policy — only images signed by the prod attestor may deploy, and GKE system images are exempted:

# gcloud container binauthz policy import policy.yaml
defaultAdmissionRule:
  evaluationMode: REQUIRE_ATTESTATION
  enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG
  requireAttestationsBy:
    - projects/my-prod-project/attestors/prod-attestor
globalPolicyEvaluationMode: ENABLE          # trust Google-managed system images
admissionWhitelistPatterns:
  - namePattern: us-docker.pkg.dev/my-prod-project/apps/*
clusterAdmissionRules:
  us-central1.prod-hardened:
    evaluationMode: REQUIRE_ATTESTATION
    enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG
    requireAttestationsBy:
      - projects/my-prod-project/attestors/prod-attestor

Case study 1 — The Shopify metadata SSRF (2018). A researcher found a server-side request forgery in a Shopify app running on GKE. Because the pod could reach the GCE metadata server and Workload Identity did not yet exist / was not enabled, the SSRF returned the node service account's access token and then Kubelet credentials, which were broad enough to pivot toward the cluster. The whole chain hinged on two GKE facts: nodes are GCE VMs with a metadata server, and the node SA was over-scoped. The controls that would have stopped it are precisely the ones this note centers on — Workload Identity + GKE Metadata Server concealment (so the raw node token is not returned to pods) and a least-privilege node SA (so even a leaked token is near-useless). A NetworkPolicy egress-deny to 169.254.169.254 is the third belt.

Case study 2 — Default-Editor node service account. For years the fastest path to owning a GCP project through GKE was: get code execution in any pod, curl the metadata server's service-accounts/default/token endpoint, and receive an OAuth token for the default Compute Engine SA — which holds project Editor. Editor can create service-account keys, modify firewall rules, read Cloud Storage, and spin up resources, so this is effective project takeover from a single pod. Google's own hardening guidance now tells you to create a custom node SA with only logging.logWriter, monitoring.metricWriter, monitoring.viewer, and (if needed) artifactregistry.reader, and to enable Workload Identity so pods never use the node SA at all. Autopilot enforces Workload Identity, which is why it is the safer default.

Case study 3 — "Leaky Vessels" runc escape (CVE-2024-21626, 2024). A file-descriptor leak in runc let a crafted image or WORKDIR escape the container into the host filesystem during runc exec/run. On a GKE Standard cluster with default runtime and no sandbox, a malicious image could break out to the node. The GKE-relevant mitigations layer up: node auto-upgrade on a release channel patched the runc version quickly; Binary Authorization would have prevented the untrusted image from ever deploying; and GKE Sandbox (gVisor) would have contained the whole class of runtime escapes by never exposing the host kernel/runtime to the workload in the first place.

Top 5 In Depth

1. Workload Identity Federation (the keystone control). Workload Identity is GKE's answer to "how does a pod call a Google API without a stolen node token or a static JSON key?" When you create a cluster with --workload-pool=PROJECT.svc.id.goog, GKE runs a per-node GKE Metadata Server that intercepts requests to the metadata endpoint. A pod requests a token; the metadata server identifies the pod's KSA (namespace + name), and — if an iam.workloadIdentityUser binding exists mapping that KSA to a GSA — federates the workload-pool identity into a short-lived access token for that GSA. Technically, the KSA's projected token is exchanged via GCP's Security Token Service for a GSA impersonation token. The critical security property: pods can only reach the GSA they are explicitly bound to, and the raw node service-account token is no longer returned to pods — the metadata server conceals it. This simultaneously kills the metadata-SSRF cloud-pivot and eliminates long-lived key files. Trade-offs: it requires the metadata server (small per-request latency), every workload needing GCP access must have a KSA→GSA binding (operational overhead), and legacy apps that hardcode GOOGLE_APPLICATION_CREDENTIALS to a key file must be migrated. Autopilot enforces it; on Standard you must set both the cluster --workload-pool and each node pool's --workload-metadata=GKE_METADATA.

2. Private clusters + control-plane authorized networks. Attack surface reduction is cheapest at the network edge. --enable-private-nodes removes public IPs from worker VMs (egress goes through Cloud NAT), so nodes cannot be scanned or hit directly from the internet. --enable-private-endpoint makes the Kubernetes API server reachable only over the internal VPC peering range (--master-ipv4-cidr), and --enable-master-authorized-networks adds an explicit CIDR allow-list of who may reach the control plane at all — typically just your bastion host and CI runner ranges. Together they mean an attacker on the internet cannot even initiate a TLS handshake with your API server, which neutralizes whole classes of API-exposure and unauthenticated-exploit risk (the crypto-mining-via-exposed-API incidents). Trade-offs: private endpoints complicate kubectl access (you need a bastion, Cloud VPN, IAP tunnel, or authorized-network entry), and you must provision Cloud NAT for image pulls and egress. A frequent misconfiguration is putting 0.0.0.0/0 in the authorized-networks list, which silently re-exposes the endpoint — audit for it.

3. Shielded and Confidential GKE Nodes. These protect the node VM itself. Shielded GKE Nodes enable UEFI Secure Boot (only signed bootloaders/kernels run), a virtual TPM (vTPM) for measured boot, and integrity monitoring that compares boot measurements against a known-good baseline and alerts on drift — defending against boot-level rootkits and kernel tampering that would otherwise survive reboots and evade in-guest tooling. Confidential GKE Nodes go further by encrypting node memory in use with AMD SEV (or Intel TDX on newer machine types), so even a compromised hypervisor or a malicious host operator cannot read the plaintext RAM of your pods — closing the data-in-use gap that at-rest and in-transit encryption leave open. Trade-offs: Shielded Nodes are essentially free and should always be on (Autopilot enforces them); Confidential Nodes carry a performance and machine-type cost and are reserved for genuinely sensitive workloads (regulated PII, keys, confidential compute requirements). Neither stops an application-layer compromise — they protect the platform beneath it.

4. GKE Sandbox (gVisor) for untrusted workloads. Standard containers share the host kernel through namespaces and cgroups, so a kernel or runtime CVE (CVE-2022-0492, CVE-2024-21626) can escape to the node. GKE Sandbox runs pods under gVisor, a userspace application kernel that intercepts syscalls and services them itself, presenting a tiny, hardened surface to the host kernel instead of the full Linux syscall table. You enable it per node pool (--sandbox type=gvisor) and opt workloads in via runtimeClassName: gvisor. This is the control you reach for when running genuinely untrusted or multi-tenant code — customer plugins, CI build jobs, code-execution sandboxes — where a shared-kernel escape is unacceptable. Trade-offs: gVisor adds syscall-interception overhead (noticeable for syscall- or IO-heavy workloads), has some compatibility gaps (a minority of syscalls/features are unsupported), and cannot host privileged or host-namespace pods. Use it selectively for the untrusted tier, not the whole cluster.

5. Binary Authorization + node auto-upgrade (supply chain + patch). Binary Authorization is a GKE admission gate that blocks any image lacking a valid attestation from a trusted attestor, so only images that passed your CI pipeline (built from trusted source, scanned, signed) can be deployed — enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG denies and logs violations, while globalPolicyEvaluationMode: ENABLE exempts Google-managed system images. Paired with Artifact Analysis vulnerability scanning and Cosign/Sigstore signing, it closes the "someone deployed a random Docker Hub image" path (ATT&CK T1610). Node auto-upgrade on a release channel handles the other half of supply chain — the platform CVEs — by automatically patching the node OS (Container-Optimized OS), kubelet, and container runtime so runc/kubelet escape bugs are fixed within days of disclosure. Trade-offs: Binary Authorization requires disciplined CI to produce attestations (break-glass annotations exist for emergencies but are logged), and auto-upgrade means nodes drain and reschedule on Google's timeline, so you need PodDisruptionBudgets and maintenance windows to avoid availability surprises.

6. Detection & Response

What to log / telemetry sources.
- Cloud Audit Logs — Admin Activity (always on: cluster create/update, IAM binding changes, firewall changes) and Data Access (must be explicitly enabled: container.googleapis.com data-access reads/writes, secret access). These are your primary GKE control-plane detection source.
- GKE Kubernetes audit logs — API-server audit events (who ran exec, created privileged pods, read secrets) delivered into Cloud Logging under k8s.io/....
- Security Command Center (SCC) — GKE Security Posture dashboard and Container Threat Detection surface misconfigurations (public endpoint, over-broad node SA) and runtime threats (reverse shell, added binary, suspicious kubectl exec).
- Dataplane V2 / Hubble flow logs — pod-to-pod and egress flows, essential for spotting metadata-endpoint access and lateral movement.
- VPC Flow Logs + Firewall Logs — node-level network telemetry, including attempts to reach 169.254.169.254.
- Falco / eBPF runtime sensors — syscall-level detections (shell in container, sensitive mount) for teams wanting beyond-SCC coverage.

Sample detection 1 — Cloud Logging / MQL: a node SA token being minted for an unexpected caller or metadata scraping. Log-based metric query for pods hitting the metadata IP (from Dataplane V2 flow logs):

resource.type="k8s_cluster"
logName=~"logs/networking.googleapis.com%2Fdataplane_flows"
jsonPayload.connection.dest_ip="169.254.169.254"
jsonPayload.connection.dest_port!=988   -- 988 is legit GKE metadata server
-- Any pod talking to 169.254.169.254 on non-standard ports = SSRF/exfil attempt

Sample detection 2 — Sigma rule: privileged pod or host-namespace pod created (Standard cluster escape prep).

title: GKE Privileged or HostNamespace Pod Created
logsource:
  product: gcp
  service: gke.audit
detection:
  selection:
    protoPayload.methodName: 'io.k8s.core.v1.pods.create'
  privileged:
    protoPayload.request.spec.containers.securityContext.privileged: true
  hostns:
    protoPayload.request.spec.hostPID: true
    protoPayload.request.spec.hostNetwork: true
  condition: selection and (privileged or hostns)
level: high
tags:
  - attack.privilege_escalation
  - attack.t1611

Sample detection 3 — KQL/log query: IAM binding granting workloadIdentityUser or new SA key created (persistence).

protoPayload.methodName=("SetIamPolicy" OR "google.iam.admin.v1.CreateServiceAccountKey")
protoPayload.serviceName="iam.googleapis.com"
-- Alert on service-account key creation (Workload Identity should mean ZERO keys)
-- and on new workloadIdentityUser bindings outside the CI pipeline principal.

Response playbook — GKE workload compromise / suspected cloud pivot.
1. Triage & scope. Confirm from GKE audit logs which pod/KSA/namespace is involved and whether any GSA impersonation or metadata access occurred. Identify the node SA and its IAM roles (blast radius).
2. Contain the pod. Cordon and drain the node (kubectl cordon / drain), or apply a deny-all NetworkPolicy to the namespace to cut egress (especially to 169.254.169.254 and the internet). Scale the affected Deployment to zero if safe.
3. Contain the identity. If a GSA token may be leaked, remove the iam.workloadIdentityUser binding and, for any static key, disable/delete the key. Rotate the node SA and, if it was over-broad, strip roles immediately. Revoke active OAuth tokens where possible.
4. Eradicate. Delete the compromised pods/images; if a malicious image ran, purge it from Artifact Registry and add it to the Binary Authorization denylist. Patch the exploited CVE by forcing node upgrade on the pool.
5. Recover. Redeploy from a known-good, attested image; re-enable network policy in enforce mode; confirm Workload Identity and metadata concealment are active. Restore workloads to healthy nodes.
6. Harden & post-incident. Enable any missing controls the incident exposed (Data Access audit logs, Binary Authorization, private endpoint, least-priv node SA, GKE Sandbox for the untrusted tier). Add SCC/Sigma detections for the observed technique. Review Terraform/IaC so the fix is codified, not manual.

7. Practice Questions

Tier 1 — Fundamentals

  1. What is Workload Identity in GKE and why does it matter?
    It federates a Kubernetes ServiceAccount to a Google ServiceAccount so pods get short-lived, per-workload GCP credentials instead of using the node's service account or a static JSON key. You bind them with an iam.workloadIdentityUser policy on the GSA (member PROJECT.svc.id.goog[namespace/ksa]) and a iam.gke.io/gcp-service-account annotation on the KSA. It matters because it eliminates the metadata-SSRF cloud-pivot and removes long-lived keys — it is the single most important GKE control. Signal: does the candidate know the flagship GKE identity mechanism, not just generic RBAC.

  2. How does an attacker turn a pod compromise into a GCP project takeover on a default GKE cluster?
    By reaching the GCE metadata server at 169.254.169.254 and requesting the node service account's token; on legacy defaults that SA is the Compute Engine default with project Editor, so the token grants near-full project control. The fix chain is Workload Identity (conceals the node token), a least-privilege node SA, and a NetworkPolicy egress-deny to the metadata IP. Signal: understands the GKE-specific cloud pivot, the crux of the whole topic.

  3. What are the security differences between Autopilot and Standard mode?
    Autopilot has Google manage nodes and enforces a hardened baseline you cannot weaken: Shielded Nodes, Workload Identity, no privileged pods, no host namespaces/hostPath, and a restricted-like admission floor. Standard gives you node-level control but every hardening flag is opt-in, so misconfiguration is easy. For most teams Autopilot is the safer default; you pick Standard when you need specific machine types, GPUs, custom kernels, or DaemonSets Autopilot restricts. Signal: knows the default-on vs opt-in distinction.

  4. What does a private GKE cluster protect against, and what are its parts?
    It removes public IPs from nodes (--enable-private-nodes) and can make the API endpoint private (--enable-private-endpoint), while --enable-master-authorized-networks restricts which CIDRs may reach the control plane. This shrinks the internet-facing attack surface — nodes cannot be scanned directly and the API cannot be reached from the open internet. You need Cloud NAT for egress and a bastion/VPN/IAP for kubectl. Signal: baseline GKE network-exposure literacy.

  5. What are Shielded GKE Nodes?
    Node VMs with UEFI Secure Boot (only signed boot components run), a vTPM for measured boot, and integrity monitoring that alerts when boot measurements drift from the baseline. They defend against boot-level rootkits and kernel tampering. They are effectively free, Autopilot enforces them, and you should always enable them on Standard with --enable-shielded-nodes. Signal: awareness of platform/boot integrity beyond the app layer.

Tier 2 — Design & Trade-offs

  1. Design a hardened GKE cluster for a regulated, multi-tenant SaaS. Walk through your choices.
    Start with Autopilot for the secure baseline, or Standard with the full hardening flag set if I need node control. Enable Workload Identity so each tenant workload has its own scoped GSA; private nodes + private endpoint + tight authorized networks to remove public exposure; Dataplane V2 with default-deny NetworkPolicy per tenant namespace (ingress and egress, blocking the metadata IP). Enforce Pod Security restricted plus Binary Authorization so only attested images run, and run untrusted tenant code under GKE Sandbox (gVisor). Add Confidential Nodes if the data-in-use requirement is regulatory. Wire Data Access audit logs, SCC Container Threat Detection, and Falco for runtime. The trade-offs: gVisor and Confidential Nodes cost performance, private endpoints complicate operator access, and Binary Authorization demands disciplined CI. Signal: can compose GKE controls into a coherent multi-tenant design with stated trade-offs.

  2. A pod is compromised via RCE. Trace how far the attacker gets and where each GKE control stops them.
    First they try the metadata server for cloud creds — Workload Identity's GKE Metadata Server conceals the node token, and a NetworkPolicy egress-deny to 169.254.169.254 blocks the request entirely. If they got a token anyway, a least-privilege node SA limits blast radius. They try to escape to the node — Pod Security restricted blocks privileged/hostPath, seccomp RuntimeDefault shrinks the syscall surface, and gVisor contains the escape class; node auto-upgrade has already patched runc CVEs. They try lateral movement — default-deny NetworkPolicy stops east-west traffic. Throughout, GKE audit logs + SCC detect the exec, privileged-pod attempt, and metadata access. Signal: holistic GKE attack-path reasoning — this is Diagram C in words.

  3. How do you give a pod BigQuery read access without any long-lived credentials, and how does it fail safe?
    Create a GSA with only roles/bigquery.dataViewer, bind the specific KSA to it with roles/iam.workloadIdentityUser on member PROJECT.svc.id.goog[ns/ksa], annotate the KSA with the GSA, and set serviceAccountName on the pod. The pod then gets short-lived BigQuery tokens from the GKE Metadata Server; no other pod gets that identity, and there is no key file to leak. It fails safe because the token is short-lived and scoped — if the pod is compromised, the attacker gets only dataViewer on that dataset, not project-wide access, and revoking the binding cuts it immediately. Signal: can reproduce the canonical Workload Identity binding and reason about least privilege.

  4. When would you use GKE Sandbox (gVisor), and what does it cost?
    For untrusted or multi-tenant workloads where a shared-kernel escape is unacceptable — customer plugins, CI build jobs, code-execution sandboxes. gVisor intercepts syscalls in a userspace kernel, so the host kernel sees a tiny surface, containing whole classes of runtime/kernel escape CVEs. The cost is syscall-interception overhead (bad for syscall/IO-heavy apps), some feature/compat gaps, and no support for privileged or host-namespace pods. You apply it selectively to the untrusted tier via a sandboxed node pool and runtimeClassName: gvisor, not cluster-wide. Signal: understands shared-kernel limits and when sandboxing is worth the tax.

  5. Compare securing pod-to-cloud identity across GKE, EKS, and AKS.
    All three use OIDC-based workload identity federation to avoid static keys: GKE Workload Identity maps KSA→GSA via the svc.id.goog pool and a concealing metadata server; EKS uses IAM Roles for Service Accounts (IRSA) or the newer EKS Pod Identity, mapping KSA→IAM role via the cluster OIDC provider; AKS uses Microsoft Entra Workload Identity, mapping KSA→managed identity/app registration via federated credentials. The common threat they all address is the metadata-server node-credential theft — GKE's Metadata Server, EKS's IMDSv2 hop-limit, and AKS's managed-identity model each blunt it. Signal: can translate the concept across clouds, a senior integration skill.

Tier 3 — Expert / Deep-Dive

  1. Explain exactly what happens under the hood when a Workload-Identity pod requests a token, including why the node SA is not returned.
    The pod calls the metadata endpoint (metadata.google.internal). On a Workload-Identity cluster this is intercepted by the per-node GKE Metadata Server, not the raw GCE metadata service. It reads the pod's projected KSA token (namespace + KSA name), checks that an iam.workloadIdentityUser binding exists mapping that KSA to a GSA, and exchanges the identity through GCP's Security Token Service for a short-lived impersonation token of the bound GSA. Crucially, the GKE Metadata Server does not proxy the service-accounts/default/token path back to pods, so the node SA token is concealed — that concealment is what breaks the classic SSRF pivot. If no binding exists, the pod gets no GCP credentials. Signal: deep understanding of the metadata-server concealment mechanism, not just "it uses OIDC."

  2. Which GKE defaults have historically been dangerous, and how do you audit a fleet for them?
    The default Compute Engine node SA carrying project Editor; legacy metadata endpoints that ignored the Metadata-Flavor header; Workload Identity off; public control-plane endpoints and 0.0.0.0/0 in authorized networks; legacy ABAC. Audit with SCC's GKE Security Posture dashboard and CIS GKE Benchmark scans, plus IAM policy analysis to find nodes still on the default SA, gcloud container clusters describe to check privateClusterConfig, masterAuthorizedNetworksConfig, workloadIdentityConfig, and shieldedNodes, and Cloud Asset Inventory / Policy queries across projects. Signal: knows the specific dangerous defaults and how to hunt them at fleet scale.

  3. How does Binary Authorization actually enforce, how do you handle break-glass, and where are its limits?
    It is a GKE admission controller: at pod admission it evaluates the image digest against a policy requiring attestations from named attestors; ENFORCED_BLOCK_AND_AUDIT_LOG denies and logs unattested images, and globalPolicyEvaluationMode: ENABLE exempts Google system images. Attestations are cryptographic signatures (Cosign/PKIX/PGP) produced in CI after the image passes source, build, and scan gates. Break-glass is the alpha.image-policy.k8s.io/break-glass: true annotation, which lets a deploy bypass the policy but is recorded in audit logs for review. Limits: it verifies provenance/signature, not runtime behavior — a signed-but-vulnerable image still runs, so pair it with Artifact Analysis scanning and runtime detection; and it gates by digest, so :latest tag discipline still matters. Signal: understands enforcement mechanics, the escape hatch, and that signing is not scanning.

  4. What is the difference between Shielded and Confidential GKE Nodes, and when is each justified?
    Shielded Nodes protect boot integrity — Secure Boot ensures only signed boot components load, the vTPM provides measured boot, and integrity monitoring alerts on drift — defending against rootkits/boot tampering; they are essentially free and should always be on. Confidential Nodes protect data in use by encrypting node memory with AMD SEV / Intel TDX, so a compromised hypervisor or malicious host operator cannot read pod RAM. Confidential Nodes carry performance and machine-type costs, so they are justified for regulated PII, key-handling, or contractual confidential-compute requirements — not blanket use. Neither protects against an application-layer compromise; they harden the platform beneath the workload. Signal: precise distinction between boot integrity and memory confidentiality with cost-aware judgment.

  5. Design defense-in-depth for a shared multi-tenant GKE cluster running untrusted customer code, and justify each layer against a specific attack.
    Namespace-per-tenant with default-deny NetworkPolicy (blocks lateral movement, T1046) and metadata egress deny (blocks cloud-cred theft, T1552.005). Each tenant workload gets its own scoped GSA via Workload Identity, so a compromise cannot reach another tenant's data or the node SA (T1078). GKE Sandbox/gVisor for the customer code so a runc/kernel escape (T1611, CVE-2024-21626) is contained; Pod Security restricted + seccomp RuntimeDefault as the second escape barrier. Binary Authorization ensures only attested platform images run (T1610), and node auto-upgrade patches escape CVEs. Confidential Nodes if tenants require memory isolation from the host. SCC Container Threat Detection + GKE audit + Falco for runtime detection, and per-tenant resource quotas to blunt noisy-neighbor/DoS. The layering is deliberate: even if gVisor has a gap, network policy and least-priv identity still bound the blast radius. Signal: senior-level, attack-mapped, layered design showing no single control is trusted alone.

9. Key Takeaways

  • Workload Identity Federation is the keystone GKE control — it binds each KSA to a scoped GSA, gives pods short-lived per-workload credentials, and (via the GKE Metadata Server) conceals the node service-account token, breaking the classic cloud pivot and eliminating static keys.
  • The GKE cloud-pivot chain is pod RCE → metadata server → node SA token → project takeover; break it with Workload Identity + a least-privilege custom node SA + a NetworkPolicy egress-deny to 169.254.169.254 + disable-legacy-endpoints=true.
  • Never run the default Compute Engine node SA (project Editor) — create a minimal SA with only logging/monitoring (and artifactregistry.reader if needed) and pass it via --service-account.
  • Autopilot is the safer default — it enforces Shielded Nodes, Workload Identity, no privileged/host-namespace pods, and a restricted admission floor you cannot weaken; choose Standard only when you need node-level control, and then set every hardening flag explicitly.
  • Remove public exposure with private nodes, a private control-plane endpoint, and tight master authorized networks (watch for the 0.0.0.0/0 misconfiguration).
  • Layer platform integrity: Shielded Nodes for verified boot (always on, free), Confidential Nodes for memory encryption when regulated, and node auto-upgrade on a release channel to patch runc/kubelet escape CVEs quickly.
  • Gate the supply chain and contain the untrusted tier: Binary Authorization so only attested images deploy, and GKE Sandbox (gVisor) for untrusted/multi-tenant code — signing is not scanning, so pair Binary Authorization with Artifact Analysis and runtime detection.
  • Enforce Dataplane V2 default-deny NetworkPolicy (ingress and egress) so a single pod compromise cannot pivot east-west or reach the metadata endpoint.
  • Turn on Data Access audit logs and SCC — Admin Activity logs are on by default but Data Access is not, and without it metadata scraping, secret reads, and GSA impersonation are invisible.
