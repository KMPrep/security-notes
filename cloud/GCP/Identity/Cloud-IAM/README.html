GCP Cloud IAM — The Authorization Control Plane

Cloud: Google Cloud Platform
Category: Identity, Authentication & Authorization
Key Docs: IAM Overview · Policy Reference · IAM Conditions / CEL · Deny Policies
ATT&CK: T1078.004 (Valid Accounts: Cloud), T1548 (Abuse Elevation Control), T1098 (Account Manipulation), T1550.001 (Application Access Token)

1. Overview

Cloud IAM is the single authorization decision point for almost every Google Cloud API call. When a principal calls storage.objects.get or compute.instances.setMetadata, the request is routed through IAM's policy evaluation engine, which answers one question: is this identity allowed to perform this permission on this resource, right now, under these conditions? IAM is not an identity provider — it does not authenticate humans (Google Identity / Workspace / Cloud Identity does) — it is the authorization plane that binds already-authenticated principals to roles on resources arranged in a hierarchy (organization → folder → project → resource). Because a single over-broad binding at the org node cascades to every project beneath it, IAM misconfiguration is the highest-leverage misconfiguration in GCP: it is consistently the root cause behind cloud breaches, and service-account privilege escalation is the most common lateral-movement path attackers use once they land in a project.

At scale the hard problems are not "how do I grant access" but "how do I prevent the accumulation of standing, unused, transitively-escalatable privilege across tens of thousands of principals and resources." That is where predefined roles, custom roles, IAM Conditions, deny policies, the policy analyzer, and the recommender come in.

TL;DR
- IAM answers authorization, not authentication; every request is evaluated against the union of allow-policies inherited down the hierarchy, minus any matching deny-policy — and IAM is additive with no allow-precedence over parents (a child cannot subtract a parent's grant; only a deny policy or Org Policy can).
- Deny always wins. Evaluation is: is there a matching deny rule? → deny. Otherwise, is there a matching allow binding (including inherited)? → allow. Otherwise → default deny.
- Primitive roles (owner/editor/viewer) are dangerous because they are thousands of permissions wide, cross-service, and owner includes setIamPolicy — meaning owner can re-grant itself anything. Prefer predefined, then custom.
- Service accounts are the crown jewels. iam.serviceAccounts.getAccessToken, getOpenIdToken, implicitDelegation, signBlob, and actAs are privilege-escalation primitives: they let a low-priv identity become a high-priv SA.
- allUsers / allAuthenticatedUsers are the public-access members; a single binding of allUsers to roles/storage.objectViewer makes a bucket world-readable. Detect and block with Org Policy iam.allowedPolicyMemberDomains (domain restricted sharing).

2. How It Works

2.1 Core Mechanics

IAM's model has three nouns and one verb. The nouns are principals (who), roles (a named bundle of permissions), and resources (what, arranged in a hierarchy). The verb is the binding that ties a principal to a role on a resource inside an allow policy (google.iam.v1.Policy). A separate object type, the deny policy, attaches deny rules that are evaluated before allow.

Concept Description
Principal (member) The identity in a binding: user:, group:, serviceAccount:, domain:, principal:/principalSet: (Workload Identity Federation), allUsers, allAuthenticatedUsers
Permission Fine-grained action string service.resource.verb, e.g. storage.objects.get, iam.serviceAccounts.getAccessToken. You never grant these directly
Role A named collection of permissions. Three kinds: primitive (owner/editor/viewer), predefined (Google-managed, e.g. roles/storage.admin), custom (org/project-defined)
Binding {role, members[], condition?} — the atomic grant inside an allow policy
Allow policy The Policy object attached to a resource; a list of bindings + an etag for optimistic concurrency
Deny policy A separate object (v2/policies) with deny rules that block permissions regardless of allow policy; evaluated first
IAM Condition A CEL expression on request/resource attributes gating a single binding or deny rule
Resource hierarchy Organization → Folder(s) → Project → Service resource. Policies inherit downward and are unioned
Basic vs granular Basic = primitive roles (legacy, wide). Granular = predefined + custom (least privilege)
etag Version tag on a policy read; setIamPolicy fails if etag is stale — prevents lost updates

Effective policy = union of allow bindings from the resource and every ancestor, evaluated after subtracting any matching deny rule. There is no concept of a child policy "narrowing" a parent's grant through the allow policy — inheritance is purely additive. The only ways to remove effective access a parent grants are (a) a deny policy, (b) an Organization Policy constraint (a different system, e.g. iam.disableServiceAccountKeyCreation), or (c) VPC Service Controls perimeters (network-context, orthogonal to IAM identity).

Roles are versioned bundles. roles/storage.objectAdmin maps to a concrete permission set that Google maintains; when Google adds a new permission to a service they can add it to predefined roles (which is convenient but means predefined roles silently widen over time — a reason security teams sometimes freeze critical grants to custom roles).

2.2 Concrete Configuration

Reading and understanding the effective policy on a project. Always start an investigation here:

# Full allow policy for a project (bindings + conditions + etag)
gcloud projects get-iam-policy prod-payments-01 --format=json

# Who has a specific dangerous role, across the project
gcloud projects get-iam-policy prod-payments-01 \
  --flatten="bindings[].members" \
  --filter="bindings.role:roles/owner" \
  --format="table(bindings.members)"

Granting a predefined role (least-privilege default). Grant the narrowest predefined role, not a primitive:

# GOOD: object-level read on ONE bucket, not project-wide storage.admin
gcloud storage buckets add-iam-policy-binding gs://pay-invoices-prod \
  --member="group:invoice-readers@example.com" \
  --role="roles/storage.objectViewer"

# BAD (do not do this): project-wide editor to a human
# gcloud projects add-iam-policy-binding prod-payments-01 \
#   --member="user:alice@example.com" --role="roles/editor"

A policy binding as JSON — this is what get-iam-policy returns and what you'd store in IaC. Note the condition block and the version: 3 (required whenever conditions are present):

{
  "version": 3,
  "etag": "BwYD1a2b3c4=",
  "bindings": [
    {
      "role": "roles/storage.objectViewer",
      "members": [
        "group:invoice-readers@example.com",
        "serviceAccount:reporting@prod-payments-01.iam.gserviceaccount.com"
      ]
    },
    {
      "role": "roles/bigquery.dataViewer",
      "members": ["user:analyst@example.com"],
      "condition": {
        "title": "biz-hours-corp-network-only",
        "description": "Analyst can read BQ only during business hours from corp IPs",
        "expression": "request.time.getHours('America/Los_Angeles') >= 8 && request.time.getHours('America/Los_Angeles') <= 18"
      }
    }
  ]
}

Applying a policy from a file (the read-modify-write pattern with etag protection):

gcloud projects get-iam-policy prod-payments-01 --format=json > policy.json
# edit policy.json ...
gcloud projects set-iam-policy prod-payments-01 policy.json
# If the etag is stale (someone else changed it), this fails loudly — good.

A real IAM Condition (CEL) gating access by resource name prefix and time — a common pattern for "read only the prod-* buckets, and only until an expiry date":

gcloud projects add-iam-policy-binding prod-payments-01 \
  --member="serviceAccount:etl@prod-payments-01.iam.gserviceaccount.com" \
  --role="roles/storage.objectViewer" \
  --condition='expression=resource.name.startsWith("projects/_/buckets/prod-") && request.time < timestamp("2026-12-31T00:00:00Z"),title=prod-buckets-until-eoy,description=ETL scoped read'

CEL attributes you can reference in IAM Conditions: resource.name, resource.type, resource.service, request.time, and for a subset of services request.auth.claims. Note the limit: conditions can inspect the target resource and the request, but not arbitrary resource tags or the caller's group membership — that scoping is coarser than AWS's condition keys.

A deny policy (the "even if allowed, block it" control). This blocks SA key creation org-wide except for a break-glass group — deny policies are attached to org/folder/project nodes via the iam.googleapis.com v2 API:

{
  "displayName": "Block SA key creation except break-glass",
  "rules": [
    {
      "denyRule": {
        "deniedPrincipals": ["principalSet://goog/public:all"],
        "exceptionPrincipals": ["principalSet://goog/group/breakglass-sa-admins@example.com"],
        "deniedPermissions": ["iam.googleapis.com/serviceAccountKeys.create"],
        "denialCondition": {
          "expression": "!resource.matchTag('12345/env', 'sandbox')"
        }
      }
    }
  ]
}

gcloud iam policies create-policy deny-sa-keys \
  --attachment-point="cloudresourcemanager.googleapis.com/organizations/1234567890" \
  --kind=denypolicies --policy-file=deny-sa-keys.json

Creating a custom role to replace an over-wide predefined grant. Define exactly the permissions a workload needs:

gcloud iam roles create invoiceProcessor \
  --project=prod-payments-01 \
  --title="Invoice Processor" \
  --description="Read invoices, write results, no admin" \
  --permissions=storage.objects.get,storage.objects.list,storage.objects.create,pubsub.topics.publish \
  --stage=GA

Service-account impersonation done right — grant roles/iam.serviceAccountTokenCreator on a specific SA (never project-wide), so the on-call SRE can mint short-lived tokens for the deploy SA without holding a key:

gcloud iam service-accounts add-iam-policy-binding \
  deploy@prod-payments-01.iam.gserviceaccount.com \
  --member="group:sre-oncall@example.com" \
  --role="roles/iam.serviceAccountTokenCreator"

# Use it — no exported key, ~1h token:
gcloud storage ls gs://pay-invoices-prod \
  --impersonate-service-account=deploy@prod-payments-01.iam.gserviceaccount.com

Workload Identity Federation — the modern keyless pattern that lets an external identity (GitHub Actions OIDC, AWS role, on-prem) act as a GCP SA without a downloaded key. The principalSet member ties a specific external subject to the SA:

gcloud iam service-accounts add-iam-policy-binding \
  ci-deployer@prod-payments-01.iam.gserviceaccount.com \
  --role="roles/iam.workloadIdentityUser" \
  --member="principalSet://iam.googleapis.com/projects/1234567890/locations/global/workloadIdentityPools/github-pool/attribute.repository/acme/payments"

Recommender & Policy Analyzer — the tools that find over-grant and answer "who can reach what":

# Role recommendations: unused permissions → suggested smaller role
gcloud recommender recommendations list \
  --project=prod-payments-01 \
  --recommender=google.iam.policy.Recommender \
  --location=global --format=json

# Policy Analyzer: "who can call storage.objects.get on this bucket?" (needs Policy Analyzer API)
gcloud asset analyze-iam-policy \
  --organization=1234567890 \
  --permissions=storage.objects.get \
  --resource-selector-full-resource-name="//storage.googleapis.com/pay-invoices-prod"

3. Real-World Incidents & CVEs

Incident / CVE Year Root Cause Lesson
GhostToken (CVE-class, Google Cloud) 2023 OAuth app tokens/SA grants survived project deletion in a "pending deletion" state, letting a malicious app stay hidden and re-attach Deleted ≠ revoked; audit lingering grants and tokens, not just live resources
Rhino Security Labs privesc research 2020–2024 Documented 30+ GCP privilege-escalation methods via impersonation, actAs, setIamPolicy, deployment-manager Fine-grained permissions like iam.serviceAccounts.getAccessToken are admin-equivalent
Vulnerable-by-Design SAs on GCE/GKE ongoing Default Compute Engine SA (PROJECT_NUMBER-compute@) granted roles/editor and attached to every VM A single SSRF/RCE on any VM yields project-wide editor via the metadata server
Google Groups → GCP grant confusion 2018 Externally-joinable Google Groups bound to IAM roles; anyone could join the group and inherit access Treat group membership as an access boundary; lock group join policies
Capital One (AWS, analog) 2019 SSRF → instance metadata → over-privileged role → S3 exfiltration of 100M records The GCP equivalent: SSRF → GCE metadata token → default editor SA. Same class, different cloud
Codecov supply-chain (analog) 2021 CI env leaked long-lived cloud creds harvested from build logs Kill static SA keys; use Workload Identity Federation with short-lived tokens
Public GCS buckets (recurring) 2017–present allUsers bound to objectViewer; buckets indexed by scanners Enforce iam.allowedPolicyMemberDomains; block public access at org level

Case study — the default Compute Engine SA as blast-radius amplifier. In a default GCP project, the Compute Engine default service account carries roles/editor and is attached to every GCE instance and GKE node unless you override it. Suppose an app on one VM has an SSRF vulnerability. An attacker who reaches the instance can call the metadata server (http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token) and retrieve an OAuth token for that SA. Because the SA is project editor, the token now grants write access to nearly every resource in the project — start VMs, read Cloud Storage, publish to Pub/Sub, edit firewall rules. This is precisely the shape of the Capital One breach translated to GCP. The control that stops it: create a purpose-built SA with only the roles the workload needs, set --service-account and --scopes explicitly on the instance, disable the default SA's editor grant, and block instance-metadata SSRF paths. Google's own hardening guidance now recommends disabling automatic role grants to default SAs via the iam.automaticIamGrantsForDefaultServiceAccounts Org Policy.

Case study — impersonation chain privilege escalation. A developer SA is granted roles/iam.serviceAccountTokenCreator at the project level (a common mistake — people scope it to the project rather than a single SA). That means the developer SA can mint tokens for any SA in the project, including a terraform@ SA that holds roles/owner. The escalation is two API calls: enumerate SAs (gcloud iam service-accounts list), then gcloud ... --impersonate-service-account=terraform@.... Now the low-priv developer effectively owns the project. The control: never grant serviceAccountTokenCreator/serviceAccountUser at project or folder scope; bind it on the specific target SA resource, and use IAM Conditions + deny policies to prevent broad impersonation. Policy Analyzer would surface "developer SA can obtain owner" before an attacker does.

Case study — public bucket exfiltration. A team fronts a static site with a GCS bucket and, to "make it work," binds allUsers to roles/storage.objectViewer on the bucket. Later, sensitive exports land in the same bucket. Because allUsers means unauthenticated internet, any scanner (GrayhatWarfare-style bucket enumerators) can list and pull objects. The control: the iam.allowedPolicyMemberDomains (Domain Restricted Sharing) org policy would have rejected the allUsers binding at write time; Public Access Prevention on the bucket blocks it at the storage layer; SCC surfaces existing public buckets.

Top 5 in Depth

1. Eliminate primitive roles and the default-SA editor grant. Primitive roles (roles/owner, roles/editor, roles/viewer) predate the granular role system; each spans thousands of permissions across every service, and owner uniquely includes resourcemanager.projects.setIamPolicy — so an owner can grant itself or anyone else any role, making it a self-perpetuating admin. The concrete fix is a layered one: (a) set the Org Policy constraints/iam.automaticIamGrantsForDefaultServiceAccounts to prevent new projects from auto-granting editor to the Compute/App Engine default SAs; (b) sweep existing projects with gcloud projects get-iam-policy ... --filter='bindings.role:roles/editor OR roles/owner' and replace with predefined roles; (c) attach purpose-built SAs to compute with explicit --service-account and --scopes. Trade-off: least privilege generates churn — you must discover the real permission set (use Recommender / audit-log analysis), and overly-tight roles cause runtime PERMISSION_DENIED errors that are hard to debug. Budget for an iterate-and-widen loop.

2. Scope impersonation permissions to the target SA, never the project. The permissions iam.serviceAccounts.getAccessToken, getOpenIdToken, signBlob, signJwt, implicitDelegation, and actAs (via roles/iam.serviceAccountUser / serviceAccountTokenCreator) are the primary GCP privilege-escalation primitives — Rhino Security Labs' research catalogs how each turns a low-priv identity into a high-priv one. The control is where you bind them: attach the role to the specific SA resource (gcloud iam service-accounts add-iam-policy-binding TARGET_SA ...), not to the project. Add an IAM Condition to limit the source, and back it with a deny policy that blocks iam.serviceAccounts.getAccessToken except for named principals. Trade-off: teams love project-level TokenCreator because it "just works" for CI; you'll get pushback, so pair the restriction with a self-service, audited impersonation path.

3. Kill static service-account keys; use Workload Identity Federation. Downloaded SA key JSON files are long-lived (they don't expire), frequently committed to git, and pasted into CI secrets — they are the single most common GCP credential-leak vector. The mitigation is to make them impossible: set the Org Policy constraints/iam.disableServiceAccountKeyCreation (and disableServiceAccountKeyUpload), then migrate every workload to short-lived tokens — Workload Identity Federation for external CI (GitHub Actions OIDC, AWS, on-prem), and Workload Identity for GKE pods. Federation trades a downloaded secret for a trust relationship: you pin the external issuer, audience, and subject via attribute mappings so only a specific repo/branch can impersonate. Trade-off: WIF setup is fiddly (attribute conditions, audience mismatches produce opaque 403s), and some legacy tools still expect a key file — for those, mint short-lived keys via signJwt rather than a permanent key.

4. Block public and cross-domain sharing with Org Policy. allUsers (any unauthenticated caller) and allAuthenticatedUsers (any Google account, including personal Gmail) are the two public principals; a single binding makes a resource world-accessible. The strongest control is constraints/iam.allowedPolicyMemberDomains (Domain Restricted Sharing), which restricts IAM bindings to principals in your allowed Cloud Identity customer IDs — an attempt to bind allUsers is rejected at write time with an error, which is far better than detecting it after the fact. Layer Storage's Public Access Prevention on top for defense-in-depth. Trade-off: DRS breaks legitimate cross-org sharing (partners, external auditors) — you'll need an explicit exception process and folder-level policy overrides, and it can surprise teams that rely on allAuthenticatedUsers for "internal-ish" sharing.

5. Run the IAM Recommender and Policy Analyzer as a continuous loop. Standing, unused privilege is the ambient risk; the Recommender uses 90 days of audit-log usage to propose smaller roles ("this SA has storage.admin but only ever called objects.get/list → recommend objectViewer"), and Policy Analyzer answers reachability questions ("who can call iam.serviceAccounts.getAccessToken on the terraform SA?") that reveal escalation paths before attackers find them. Operationalize it: export recommendations via gcloud recommender recommendations list, feed them into ticketing, and gate high-risk grants in CI with a Policy Analyzer check. Trade-off: recommendations are usage-based, so a permission used quarterly (e.g. DR failover) looks unused and may be wrongly removed — always human-review before applying, and tag known-rare permissions as exceptions.

6. Detection & Response

What to log / telemetry sources. Cloud IAM produces its highest-value signal in Cloud Audit Logs:
- Admin Activity logs (logName ending activity) — always on, non-disableable; capture SetIamPolicy, CreateServiceAccountKey, role creation, google.iam.admin.v1.*. This is your primary IAM-change tail.
- Data Access logs (logName ending data_access) — off by default except for BigQuery; must be enabled to see getIamPolicy/testIamPermissions reads and object-level access. Enable for sensitive services.
- protoPayload.serviceData.policyDelta.bindingDeltas — the field that shows exactly which member/role was added or removed on a SetIamPolicy. This is the field to alert on.
- protoPayload.authenticationInfo.principalEmail — who did it; watch for impersonation chains where principalEmail is a SA and serviceAccountDelegationInfo shows a chain.
- IAM Credentials API calls (GenerateAccessToken, GenerateIdToken, SignBlob) — impersonation events.

Sample detection 1 — dangerous role grant (Log Analytics / BigQuery SQL over exported audit logs):

SELECT
  timestamp,
  proto_payload.audit_log.authentication_info.principal_email AS actor,
  JSON_VALUE(binding_delta, '$.member')  AS granted_member,
  JSON_VALUE(binding_delta, '$.role')    AS granted_role,
  JSON_VALUE(binding_delta, '$.action')  AS action
FROM `org_logs.cloudaudit_googleapis_com_activity`,
  UNNEST(JSON_QUERY_ARRAY(
    proto_payload.audit_log.service_data,
    '$.policyDelta.bindingDeltas')) AS binding_delta
WHERE proto_payload.method_name = 'SetIamPolicy'
  AND JSON_VALUE(binding_delta, '$.action') = 'ADD'
  AND JSON_VALUE(binding_delta, '$.role') IN (
    'roles/owner', 'roles/editor',
    'roles/iam.securityAdmin', 'roles/iam.serviceAccountTokenCreator',
    'roles/iam.serviceAccountUser', 'roles/resourcemanager.organizationAdmin')
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 DAY)
ORDER BY timestamp DESC;

Sample detection 3 — service-account impersonation spike (KQL-style / Chronicle):

SELECT principal_email, target_service_account, COUNT(*) AS token_count
FROM iam_credentials_events
WHERE method_name = 'GenerateAccessToken'
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
GROUP BY principal_email, target_service_account
HAVING token_count > 50   -- tune per baseline
ORDER BY token_count DESC;

Response playbook (containment → eradication → recovery):
1. Confirm & scope. Pull the SetIamPolicy / GenerateAccessToken events; identify the principalEmail, the added binding(s), and the resource/hierarchy node. Determine blast radius with Policy Analyzer ("what can this principal now reach?").
2. Contain — revoke the grant. gcloud <resource> remove-iam-policy-binding the malicious binding (mind the etag race). For an active impersonation, disable the abused SA (gcloud iam service-accounts disable) — this immediately invalidates its ability to be impersonated for new tokens.
3. Contain — cut existing tokens. Outstanding OAuth tokens live up to ~1 hour and are not revoked by removing the binding. Disabling/deleting the SA and rotating any static keys forces re-auth. For humans, revoke sessions via Cloud Identity and reset credentials.
4. Eradicate — remove persistence. Hunt for attacker-created SAs, new keys (serviceAccountKeys.create), added bindings elsewhere in the hierarchy, and Workspace domain-wide-delegation changes. Check for GhostToken-style lingering OAuth grants.
5. Eradicate — apply the missing guardrail. Add the deny policy / Org Policy that should have prevented it (disableServiceAccountKeyCreation, allowedPolicyMemberDomains, scoped TokenCreator).
6. Recover & verify. Restore intended least-privilege bindings from IaC (source of truth), re-run Policy Analyzer to confirm the escalation path is gone, and re-enable services.
7. Post-incident. Backfill detections (alert on the exact binding delta seen), enable Data Access logs on the affected services if they were off, and add the escalation path to your continuous Policy Analyzer sweep.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is the difference between primitive, predefined, and custom roles, and why are primitive roles discouraged?
Primitive (basic) roles — owner, editor, viewer — predate granular IAM and each span thousands of permissions across every service. Predefined roles are Google-maintained, service-scoped bundles (e.g. roles/pubsub.publisher) that follow least privilege. Custom roles are org/project-defined sets of exactly the permissions you list. Primitive roles are discouraged because they grossly over-grant, and owner in particular includes setIamPolicy, letting the holder re-grant itself anything — so it's effectively unbounded admin. You use predefined by default and custom when no predefined role fits.
Signal: knows the role taxonomy and the specific danger of setIamPolicy in owner.

Q2. Walk me through how IAM decides whether to allow a request.
IAM first checks deny policies on the resource and its ancestors; if a deny rule matches the permission and its condition is true, the request is denied outright. Otherwise IAM gathers the union of allow bindings from the resource and every ancestor up to the org node, checks whether any binding grants the requested permission to the principal, and evaluates that binding's condition. If a matching allowed binding with a satisfied condition exists, allow; otherwise default deny. The key properties: deny wins, allow is additive up the hierarchy, and there's no default access.
Signal: understands deny-first, inheritance-as-union, default-deny.

Q3. What are allUsers and allAuthenticatedUsers?
They are the two special public principals. allUsers means anyone on the internet, unauthenticated — no Google account required. allAuthenticatedUsers means any principal authenticated with a Google account, which includes personal Gmail accounts, not just your org. Binding either to a role makes the resource public to that audience; a single allUsers + objectViewer binding makes a bucket world-readable. You block both org-wide with the Domain Restricted Sharing org policy.
Signal: knows allAuthenticatedUsers is not "internal" — it includes any Gmail.

Q4. How does policy inheritance work in the GCP resource hierarchy?
Resources form a tree: organization → folders → projects → service resources. An allow policy set at any node is inherited by all descendants, and effective permissions are the union of policies from the resource up to the org. So a roles/viewer grant at the org node gives that principal viewer on every project and resource beneath. Inheritance is purely additive — a child policy cannot subtract a parent's grant; only a deny policy, an Org Policy constraint, or VPC Service Controls can remove effective access.
Signal: knows inheritance is additive and children can't narrow parents via allow policy.

Q5. What is a service account and how does it differ from a user?
A service account is a non-human identity used by workloads (VMs, GKE pods, CI) to authenticate to GCP APIs; it is identified by an email like svc@project.iam.gserviceaccount.com and can be both a principal (it's granted roles) and a resource (you set IAM policy on it to control who can impersonate it). Unlike a user, it has no password/2SV and authenticates via keys or, preferably, short-lived tokens from impersonation or Workload Identity Federation. Its dual nature — identity you grant and resource others can impersonate — is exactly why SAs are the primary privilege-escalation target.
Signal: recognizes the principal/resource duality of SAs.

Tier 2 — System Design / Trade-off

Q6. Design an IAM model for a 200-project org with prod/non-prod separation and 500 engineers. How do you avoid privilege sprawl?
I'd anchor on the hierarchy: an org node, folders per environment (prod, non-prod, shared-services) and per business unit, and projects as the unit of workload isolation. Grants go to groups, never individual users, and at the lowest node that satisfies the need — team access at the folder, workload access at the project. Humans get predefined roles via groups synced from the HR/IdP system; workloads get purpose-built SAs with custom or narrow predefined roles. Guardrails: Org Policies to ban primitive roles, disable SA key creation, and enforce Domain Restricted Sharing; deny policies to backstop impersonation. I'd wire the IAM Recommender into a quarterly right-sizing loop and gate risky grants in CI with Policy Analyzer. Access to prod is time-bound via IAM Conditions or a just-in-time elevation tool, so standing prod privilege trends toward zero. All bindings live in Terraform as the source of truth so drift is detectable.
Signal: groups-not-users, lowest-node grants, guardrails, JIT, IaC as truth.

Q7. When would you use IAM Conditions versus deny policies versus Org Policy constraints? They seem to overlap.
They operate at different layers. IAM Conditions attach to a single allow binding and narrow it — "grant this only for prod- buckets during business hours"; they're additive-scope refinements. Deny policies are separate objects that block permissions for principals regardless of any allow — the right tool for "no one except break-glass can create SA keys," because deny wins over allow. Org Policy constraints are a different system entirely: they restrict resource configuration and metadata (e.g. iam.disableServiceAccountKeyCreation, iam.allowedPolicyMemberDomains, allowed regions) and are enforced at write time, so they prevent bad state from ever existing. Rule of thumb: use Conditions to scope legitimate grants, deny policies to carve exceptions into broad grants you can't easily remove, and Org Policy to enforce non-negotiable guardrails that reject bad configuration up front.
Signal: understands the three are complementary layers, not interchangeable.*

Q8. Your CI needs to deploy to GCP. Compare downloaded SA keys, Workload Identity Federation, and running CI on GCE with an attached SA.
Downloaded SA keys are the worst option: long-lived, non-expiring, easily leaked into logs or git, and a top breach vector — I'd disable their creation org-wide. Running CI on a GCE/GKE runner with an attached SA is good if the runner is inside my trust boundary — the workload gets short-lived metadata-server tokens, no secret to leak — but it requires me to host and secure runners, and a compromised runner still yields the SA. For external CI like GitHub Actions, Workload Identity Federation is best: GitHub's OIDC token is exchanged for a short-lived GCP token, with no stored secret, and I pin the trust to a specific repo and branch via attribute mappings so only that pipeline can impersonate the deploy SA. The trade-off is setup complexity and opaque 403s on audience/attribute mismatches. My default: WIF for external CI, attached SA for in-VPC runners, keys never.
Signal: keyless-first, understands WIF attribute pinning and the trust-boundary distinction.

Q9. How do you design least-privilege for a workload when you don't know the exact permissions it needs?
I start slightly broad but observable, then tighten. Grant a reasonable predefined role, deploy to a non-prod project, and run the workload through its real paths while collecting Data Access + Admin Activity audit logs. I then use the IAM Recommender (which analyzes ~90 days of usage) and log analysis to derive the actual permission set, and codify it as a custom role scoped to exactly those permissions. I iterate — over-tight roles surface as PERMISSION_DENIED errors I widen deliberately, one permission at a time, rather than reverting to editor. I explicitly tag rarely-used permissions (DR, quarterly jobs) as known exceptions so the recommender doesn't strip them. The output is a pinned custom role in Terraform, which also protects against predefined-role scope creep.
Signal: iterate-and-tighten, uses Recommender + logs, handles rare-permission false-negatives.

Q10. An SRE says project-level serviceAccountTokenCreator is fine because "it's just our team." Argue the other side and propose an alternative.
Project-level TokenCreator means the team can impersonate every SA in the project — including any owner/editor SA like a Terraform runner — so it's a latent full-owner escalation, and Policy Analyzer would flag exactly that reachability. It also erases attribution: actions appear as the target SA, not the human, complicating IR. The blast radius of one compromised team member is the whole project. The alternative: bind serviceAccountTokenCreator on the specific SAs they legitimately need to impersonate, add an IAM Condition limiting source (corp network / time), route elevation through an audited just-in-time flow, and add a deny policy backstop that blocks impersonation of the high-value SAs except for a named break-glass group. This preserves the SRE's workflow for the SAs that matter while removing the escalation path to owner.
Signal: names the escalation-to-owner path, attribution loss, and a scoped + deny-backstop fix.

Tier 3 — Expert / Deep-dive

Q11. Enumerate the GCP IAM privilege-escalation primitives and explain why each is admin-equivalent.
The core primitives, catalogued by Rhino Security Labs, cluster around service-account abuse and policy control. Impersonation: iam.serviceAccounts.getAccessToken (mint an OAuth token as a target SA), getOpenIdToken (mint an ID token), signBlob/signJwt (sign as the SA, enabling token forging), and implicitDelegation. Deployment: iam.serviceAccounts.actAs (via serviceAccountUser) lets you attach a powerful SA to compute/Cloud Functions/Cloud Run and run your code as it. Policy control: resourcemanager.projects.setIamPolicy (or roles/iam.securityAdmin) lets you bind yourself any role directly. Indirect vectors: cloudfunctions.functions.create + actAs, compute.instances.create + actAs, deploymentmanager.deployments.create, orgpolicy edits, and creating SA keys (serviceAccountKeys.create) for persistence. Each is admin-equivalent because it grants a path to act as, or become, a higher-privileged identity — the permission itself looks narrow, but its transitive reach is total. This is why you evaluate IAM by reachability (Policy Analyzer), not by role names.
Signal: names the actual permissions and articulates transitive reach vs nominal scope.

Q12. Deny policies and allow policies both exist. Walk through exact evaluation order and a case where a deny policy with a condition changes the outcome.
Evaluation is deny-first: IAM collects all deny policies attached to the resource and its ancestors, and for each deny rule checks whether the requested permission is in deniedPermissions, the principal is in deniedPrincipals and not in exceptionPrincipals, and the denialCondition (CEL) evaluates true. If any deny rule matches, the request is denied — no allow policy can override it. Only if no deny matches does IAM evaluate the union of allow bindings. Example: a developer SA has roles/editor (allow) but a deny policy blocks iam.serviceAccountKeys.create for principalSet://goog/public:all with a denialCondition of !resource.matchTag('env','sandbox'). In prod (no sandbox tag) the condition is true → key creation denied despite editor. In a sandbox project the tag makes the condition false → the deny doesn't apply → editor's allow lets key creation proceed. So the same principal and permission resolve differently based on the resource's tag evaluated in the denial condition.
Signal: precise deny-rule field semantics and condition-driven divergent outcomes.

Q13. How does IAM interact with VPC Service Controls, and why isn't IAM alone sufficient to prevent data exfiltration?
IAM answers "does this identity have permission on this resource," but it says nothing about network context — a valid principal with legitimate storage.objects.get can read a bucket from anywhere, including an attacker's machine using stolen credentials. VPC Service Controls adds an orthogonal, identity-independent boundary: a service perimeter around projects that blocks API access to protected services unless the request originates from inside the perimeter (or an allowed access level / ingress rule). So even a stolen token with full IAM rights is stopped at the perimeter if the request comes from outside. They compose: IAM decides who can do what, VPC-SC decides from where. Neither replaces the other — IAM without VPC-SC leaks under credential theft; VPC-SC without IAM has no notion of least privilege. For sensitive data I'd use both, plus context-aware access levels so even in-perimeter requests need device/identity posture.
Signal: articulates the identity vs network-context split and why both are needed against credential theft.

Q14. A compromised GCE VM's app has SSRF. Trace the full attack path to project-wide compromise and the layered controls that break each step.
The SSRF lets the attacker make the VM issue HTTP requests to the metadata server (http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token, requiring the Metadata-Flavor: Google header). That returns an OAuth token for the VM's attached SA. If that's the default Compute SA with roles/editor, the token now grants project-wide write: enumerate resources, read all buckets, create new SAs and keys for persistence, edit firewall rules, and pivot. Layered breaks: (1) at the SSRF layer, block egress to the metadata IP and require the metadata header (GKE Workload Identity / metadata concealment); (2) at the identity layer, don't attach the default editor SA — use a purpose-built SA with only needed roles and explicit scopes, and set the automaticIamGrants org policy; (3) at the escalation layer, deny policies on serviceAccountKeys.create and scoped impersonation prevent persistence and lateral SA abuse; (4) at the network layer, VPC-SC stops the stolen token from being used outside the perimeter; (5) at detection, alert on the metadata-token-then-anomalous-API pattern and on new SA-key creation. Each control independently shrinks blast radius; defense-in-depth means no single failure yields owner.
Signal: end-to-end chain plus a control at every layer, not just "use least privilege."

Q15. Predefined roles change over time and conditions have limits. What are the subtle IAM pitfalls a senior engineer must anticipate?
Several. (1) Predefined-role drift: Google adds permissions to predefined roles, so a grant you audited last year may be wider today — pin critical grants to custom roles. (2) Condition coverage gaps: an IAM Condition only constrains the binding it's on; if the same principal has a parallel un-conditioned binding (perhaps inherited from a folder), the condition is moot — you must ensure no un-scoped path exists, and use deny policies as a backstop. (3) Token TTL vs revocation: removing a binding doesn't kill outstanding ~1h OAuth tokens; you must disable/rotate the SA for immediate effect. (4) Impersonation attribution: actions run as the impersonated SA, and serviceAccountDelegationInfo chains can obscure the human origin — build detections on the delegation chain. (5) Group-based grants inherit group membership risk: an externally-joinable Google Group bound to a role means group-join = access. (6) Data Access logs are off by default, so getIamPolicy/testIamPermissions recon and object reads are invisible until enabled. (7) Deleted ≠ revoked (GhostToken): grants and OAuth tokens can linger past resource deletion. A senior engineer designs assuming each of these, and validates with continuous Policy Analyzer reachability sweeps rather than trusting role names.
Signal: depth beyond happy-path — drift, condition gaps, token TTL, attribution, logging defaults.

9. Key Takeaways

  • IAM is authorization, not authentication. It binds already-authenticated principals to roles on hierarchical resources; identity comes from Cloud Identity/Workspace.
  • Evaluation is deny-first, then additive union of inherited allow, then default-deny. A grant at the org node cascades to every descendant; only deny policies, Org Policy, and VPC-SC can subtract effective access.
  • Primitive roles are a trap — thousands of permissions wide, and owner includes setIamPolicy, making it self-perpetuating admin. Default to predefined, pin critical grants to custom roles.
  • Service accounts are the crown jewels and the top escalation target. getAccessToken, actAs, signBlob, and setIamPolicy are admin-equivalent primitives — scope them to a specific SA, never a project.
  • The default Compute SA with editor, attached to every VM, turns any SSRF into project-wide compromise. Disable auto-grants and use purpose-built SAs.
  • Public principals allUsers/allAuthenticatedUsers cause the classic public-bucket breach; block them at write time with Domain Restricted Sharing, not after the fact.
  • Kill static SA keys (disableServiceAccountKeyCreation) and move to Workload Identity Federation for short-lived, secret-less credentials.
  • Enforce guardrails as configuration, not vigilance: Org Policy constraints reject bad state up front; deny policies backstop broad grants; IAM Conditions scope legitimate ones.
  • Operate least privilege as a loop: IAM Recommender to right-size, Policy Analyzer for reachability, both wired into ticketing and CI.
  • Detect on the binding delta. Alert on SetIamPolicy adds of dangerous roles, public-principal bindings, and impersonation-token spikes in Cloud Audit Logs; enable Data Access logs on sensitive services.
