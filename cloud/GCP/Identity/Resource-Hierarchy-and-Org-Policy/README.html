GCP Resource Hierarchy & Organization Policy Service — The Top-Down Guardrail Layer

Domain: GCP / Identity & Governance
Provider docs: Cloud Resource Manager, Organization Policy Service, IAM
Analogues: AWS Service Control Policies (SCPs) + Resource Control Policies (RCPs); Azure Policy + Management Groups
Key constraints referenced: constraints/compute.vmExternalIpAccess, constraints/iam.disableServiceAccountKeyCreation, constraints/storage.uniformBucketLevelAccess, constraints/sql.restrictPublicIp, constraints/gcp.resourceLocations

1. Overview

Google Cloud's resource hierarchy is the spine of its governance model: every resource lives under a Project, projects roll up into Folders, and folders roll up into a single Organization node tied to a Cloud Identity / Workspace domain. IAM policies and Organization Policies both flow down this tree, so where you attach a control determines its blast radius. The Organization Policy Service is the guardrail layer that sits on top of IAM: it restricts what configurations are even allowed to exist on resources — regardless of who is asking — while IAM restricts who can do what. The two are complementary and both are needed: a Storage Admin with storage.buckets.setIamPolicy can technically make a bucket public, but an org policy enforcing constraints/storage.publicAccessPrevention makes that action fail no matter how much IAM power the principal holds.

This matters at scale because IAM alone cannot express "no VM in this company may ever have a public IP" or "no data may leave the EU." Those are invariants — properties that must hold across thousands of projects and hundreds of teams with delegated admin. Org Policy is how a central security team encodes invariants once at the Organization node and lets them inherit everywhere, closing the gap between "we told people not to" and "the platform won't let them." Getting this layer wrong is one of the most common root causes of GCP data exposure: public buckets, exportable service account keys, and internet-facing Cloud SQL instances are almost always missing-guardrail problems, not IAM-misgrant problems.

TL;DR
- The hierarchy is Organization → Folder → Project → resource; IAM and Org Policy both inherit downward and are additive/enforced as you descend.
- IAM answers "who can do what"; Org Policy answers "what can be configured at all." Org Policy is evaluated before the action succeeds and cannot be overridden by IAM permissions.
- Constraints come in two shapes: boolean (on/off, e.g. disableServiceAccountKeyCreation) and list (allow/deny values, e.g. resourceLocations = in:eu-locations).
- Inheritance semantics matter: list policies merge with parent by default but a child can set inheritFromParent: false to break the chain, or use denyAll/allowAll. Boolean policies are simply overridden by the most specific node.
- Custom constraints (CEL over resource fields) extend Org Policy beyond Google's predefined catalog — this is GCP's answer to AWS SCP Condition expressiveness.
- Detect drift by alerting on SetOrgPolicy / google.cloud.orgpolicy.v2.OrgPolicy.CreatePolicy in Cloud Audit Logs.

2. How It Works

2.1 Core mechanics

The resource hierarchy is managed by the Cloud Resource Manager API. There is exactly one Organization resource per Cloud Identity/Workspace account; it is created automatically the first time a user from that domain interacts with GCP. Below it you can nest up to 10 levels of folders (folders can contain folders and projects), and each project is a flat container for service resources (Compute instances, GCS buckets, BigQuery datasets, Cloud SQL instances, etc.). Two distinct policy systems attach to nodes of this tree:

  • IAM allow policies — bind principals (users, groups, service accounts) to roles at a node. A binding at the Organization or Folder level is inherited by all descendants and is purely additive: the effective permission set of a principal on a resource is the union of all bindings from that resource up to the org root. GCP IAM has no explicit deny at the binding level by default; you cannot "subtract" an inherited grant with a lower-level allow binding. (IAM Deny policies, a separate newer object, can subtract — more below.)
  • Organization Policies — attach a constraint with a policy (the enforcement configuration) at Org, Folder, or Project. These express restrictions on resource configuration. They are evaluated by the platform when a mutating API call is made and can block the call outright.

A constraint is a typed schema published by Google (or defined by you as a custom constraint). Each constraint has:

Concept Meaning Example
Constraint name Fully-qualified identifier constraints/compute.vmExternalIpAccess
Constraint type list or boolean list = allow/deny value sets; boolean = enforced true/false
Policy The applied configuration at a node enforced: true, or allowedValues: [...]
Inheritance Whether the node merges with parent inheritFromParent: true/false
Rules Conditional enforcement (CEL) condition: resource.matchTag(...)
allowAll / denyAll List-policy shortcuts deny every value / allow every value
Dry-run policy Audit-only shadow policy liveState vs dryRunSpec

Boolean constraints are the simplest: they are either enforced: true or enforced: false. Example: constraints/iam.disableServiceAccountKeyCreation. When enforced at a node, no descendant may create user-managed service account keys.

List constraints carry an ordered set of allowedValues and/or deniedValues, drawn from a constraint-specific value domain. Some list constraints support custom values (arbitrary strings, e.g. a project ID or region), others only accept predefined values published by Google (prefixed is: or referenced via value groups like in:us-locations). Example: constraints/gcp.resourceLocations with allowedValues: [in:eu-locations] forces new resources into EU regions/multi-regions.

Evaluation and override semantics. When the platform evaluates whether an action is allowed, it walks from the resource up to the org root and computes the effective policy:

  • Boolean: the most specific node that sets a policy wins. A project-level enforced: false will override an org-level enforced: true only if the project is permitted to override — and whether that's possible depends on whether a higher node used inheritFromParent semantics plus who holds orgpolicy.policyAdmin. There is no hard "no-override" flag on a plain boolean; the control is IAM on the Org Policy Admin role.
  • List: by default a child policy is merged with the resolved parent policy (inheritFromParent: true). Setting inheritFromParent: false replaces the parent's evaluated policy for that subtree. denyAll: true denies all values regardless of parent; allowAll: true allows all.
  • Conflict rule for lists: if the merged result both allows and denies the same value, deny wins.

This is the single most important subtlety for real systems: an org-level list policy is NOT automatically un-overridable. A folder admin who also holds roles/orgpolicy.policyAdmin on their folder can set inheritFromParent: false and allowAll: true to punch a hole in an org-wide restriction for their subtree. The only thing preventing that is not granting orgpolicy.policyAdmin below the org node (or scoping it very carefully). Treat Org Policy Admin like a root-of-trust role.

Org Policy vs IAM — the mental model. IAM is an allow-list of actors. Org Policy is a shape constraint on the resource. A request to create a VM with an external IP must pass both: the caller needs compute.instances.create (IAM) and the resulting resource config must not violate compute.vmExternalIpAccess (Org Policy). Org Policy is checked by the service's admission logic; a violation returns a 412/400-class error such as Constraint constraints/compute.vmExternalIpAccess violated for project .... Crucially, Org Policy applies to all principals including Organization Admins and even Google support paths that go through the API — it is not an IAM grant you can escalate around.

2.2 Concrete configuration

Inspect the hierarchy.

# Find your org ID
gcloud organizations list
# DISPLAY_NAME        ID              DIRECTORY_CUSTOMER_ID
# example.com         123456789012    C03xxxxxx

# List folders under the org
gcloud resource-manager folders list --organization=123456789012

# List projects under a folder
gcloud projects list --filter="parent.id=555555555555 AND parent.type=folder"

Boolean constraint — block external service account keys org-wide. User-managed SA keys are long-lived, exportable secrets and the #1 GCP credential-theft vector. Turn creation off at the org root:

gcloud resource-manager org-policies enable-enforce \
  constraints/iam.disableServiceAccountKeyCreation \
  --organization=123456789012

Or the newer gcloud org-policies (v2 API) using a YAML spec:

# disable-sa-keys.yaml
name: organizations/123456789012/policies/iam.disableServiceAccountKeyCreation
spec:
  rules:
  - enforce: true

gcloud org-policies set-policy disable-sa-keys.yaml

Boolean — deny public IPs on Compute. Even with disableServiceAccountKeyCreation, a VM with a public IP is directly attackable. Enforce at org, then (optionally) exempt one folder for a bastion subnet:

gcloud resource-manager org-policies enable-enforce \
  constraints/compute.vmExternalIpAccess \
  --organization=123456789012

Note compute.vmExternalIpAccess is actually a list constraint whose values are VM instance names — enforcing/denying all effectively blocks external IPs. The cleaner v2 form:

# deny-external-ip.yaml
name: organizations/123456789012/policies/compute.vmExternalIpAccess
spec:
  rules:
  - denyAll: true

List constraint — data residency (gcp.resourceLocations). Force all new resources into EU locations for a GDPR-scoped folder, using Google's eu-locations value group:

# eu-residency.yaml
name: folders/555555555555/policies/gcp.resourceLocations
spec:
  inheritFromParent: false
  rules:
  - values:
      allowedValues:
      - in:eu-locations

gcloud org-policies set-policy eu-residency.yaml

in:eu-locations is a value group that expands to europe-west1, europe-west4, the eu multi-region, etc. Value groups are maintained by Google so you don't have to enumerate regions. You can also use in:us-locations, or specific values like in:europe-west1-locations.

List — restrict Cloud SQL public IPs.

gcloud resource-manager org-policies enable-enforce \
  constraints/sql.restrictPublicIp \
  --organization=123456789012

This boolean-style SQL constraint forces all Cloud SQL instances to be private-IP only. Pair with constraints/sql.restrictAuthorizedNetworks to stop 0.0.0.0/0 authorized networks on any instance that does keep a public IP.

Boolean — enforce Uniform Bucket-Level Access (UBLA). Object ACLs are a legacy, per-object access mechanism that bypasses IAM and are a classic public-exposure vector. Force all buckets to IAM-only:

# ubla.yaml
name: organizations/123456789012/policies/storage.uniformBucketLevelAccess
spec:
  rules:
  - enforce: true

Complement with constraints/storage.publicAccessPrevention (enforced) to guarantee no bucket or object can be made public even by an IAM binding to allUsers/allAuthenticatedUsers.

Conditional (tag-based) enforcement. Org Policy v2 supports CEL conditions on Resource Manager tags, so you can enforce a constraint only where a tag is present — e.g. allow external IPs only on resources tagged network-tier=dmz:

# conditional-external-ip.yaml
name: organizations/123456789012/policies/compute.vmExternalIpAccess
spec:
  rules:
  - condition:
      expression: "resource.matchTag('123456789012/network-tier', 'dmz')"
      title: allow-external-ip-in-dmz
    allowAll: true
  - denyAll: true          # default rule: everything else denied

Dry-run mode lets you deploy a policy in audit-only mode to see what it would block, surfaced as orgpolicy audit-log violations without actually failing calls:

gcloud org-policies set-policy --update-mask=dryRunSpec eu-residency.yaml

Custom constraint (CEL). When Google has no predefined constraint for what you need, define your own over a resource type's fields. Example: forbid Compute instances that don't enable Shielded VM secure boot:

# custom-shieldedvm.yaml
name: organizations/123456789012/customConstraints/custom.requireSecureBoot
resourceTypes:
- compute.googleapis.com/Instance
methodTypes:
- CREATE
- UPDATE
condition: "resource.shieldedInstanceConfig.enableSecureBoot == true"
actionType: ALLOW
displayName: Require Shielded VM Secure Boot
description: All Compute instances must have secure boot enabled.

# 1. Register the custom constraint (defines the schema)
gcloud org-policies set-custom-constraint custom-shieldedvm.yaml

# 2. Then enforce it like any constraint
gcloud org-policies enable-enforce \
  custom.requireSecureBoot --organization=123456789012

The condition CEL expression is evaluated against the resource being created/updated; if actionType: ALLOW and the condition is false, the call is denied (and vice versa for DENY). This gives you SCP-Condition-like power inside GCP.

Read the effective, resolved policy for a resource (crucial for debugging inheritance):

gcloud org-policies describe compute.vmExternalIpAccess \
  --project=my-app-prod --effective

The --effective flag returns the computed result after walking the hierarchy — the actual truth the platform will enforce, not just the local node's policy.

Case study 1 — the exportable service account key. The most repeated GCP compromise pattern is not an exotic exploit; it is a service-account-key.json file that leaked. A developer runs gcloud iam service-accounts keys create key.json --iam-account=..., commits it to a repo or bakes it into a container, and the key — a non-expiring, non-rotating RSA credential — is later found by an attacker scraping public repos or a compromised CI system. Because the key can impersonate a service account that often holds roles/editor, the attacker gets broad, long-lived access with no MFA and no session expiry. The single highest-leverage guardrail is enforcing constraints/iam.disableServiceAccountKeyCreation at the organization node so these keys cannot be minted at all, pushing teams to Workload Identity Federation and short-lived tokens. Where a legacy system genuinely needs a key, you carve a narrow exception at a specific project and alert on it — you do not leave key creation globally on.

Case study 2 — the public bucket that "no one made public." GCS has two access models: modern IAM (bucket-level) and legacy object ACLs. A team enables fine-grained ACLs and, somewhere in an automation script, sets an object's ACL to grant allUsers:READER. IAM audits look clean because the exposure is at the object ACL layer, invisible to bucket-IAM review. Enforcing constraints/storage.uniformBucketLevelAccess disables object ACLs entirely so all access flows through auditable IAM, and constraints/storage.publicAccessPrevention blocks any binding to allUsers/allAuthenticatedUsers. Together they make "accidentally public" structurally impossible — the platform rejects the operation rather than trusting every engineer to never make a mistake.

Case study 3 — the folder-admin override. A central security team enforces compute.vmExternalIpAccess = denyAll at the org root, confident that no VM company-wide can get a public IP. Months later a pen-test finds internet-facing VMs in one product folder. Root cause: to move fast, platform-eng had granted the product team's lead roles/orgpolicy.policyAdmin on their folder. The lead set inheritFromParent: false + allowAll: true on that constraint for the folder, silently punching a hole in the org guardrail. The fix is organizational and technical: orgpolicy.policyAdmin is a root-of-trust role that should be held by a tiny central group at the org node only, and every SetOrgPolicy/OrgPolicy.UpdatePolicy event must be alerted on so a weakening is caught within minutes.

Top 5 in depth

1. iam.disableServiceAccountKeyCreation (org-wide). This boolean constraint disables the serviceAccounts.keys.create API path for all descendants, so gcloud iam service-accounts keys create fails with a constraint violation regardless of the caller's IAM. Technically it is enforced at the IAM admission layer for the SA-keys sub-API. The trade-off is that legacy integrations expecting a downloadable JSON key break — the correct migration is Workload Identity Federation (external OIDC/SAML identities exchange for short-lived GCP tokens) for CI/CD, and attached service accounts for compute. Its limit: it only stops new keys; you must separately inventory and revoke existing keys (query serviceAccountKeys and rotate/delete) and monitor keyType: USER_MANAGED. This is the single highest-ROI org policy in most environments because leaked keys are the dominant GCP credential-compromise vector.

2. storage.publicAccessPrevention + uniformBucketLevelAccess (org-wide). UBLA disables object-level ACLs so all access decisions route through bucket IAM — auditable, uniform, and reviewable. publicAccessPrevention then blocks any IAM binding or ACL granting allUsers/allAuthenticatedUsers, making public exposure structurally impossible even for a Storage Admin. Technically these are admission checks in the GCS control plane: a setIamPolicy adding allUsers returns 412 Precondition Failed. Trade-off: legitimately public assets (static websites, public datasets) must be served via an explicit exception (a dedicated folder with the constraint relaxed, ideally fronted by Cloud CDN/Load Balancer rather than raw public buckets). The limit is that these don't protect against an authenticated over-broad IAM grant (e.g. to a wide group) — that's an IAM-hygiene problem, not an org-policy one.

3. compute.vmExternalIpAccess (org-wide denyAll). Denies external IP assignment to Compute instances, forcing them behind Cloud NAT for egress and Load Balancers / IAP for ingress. As a list constraint keyed on instance identifiers, denyAll: true is the clean way to express "no VM anywhere." This shrinks the internet-facing attack surface dramatically and is a prerequisite for a defensible network posture. Trade-offs: bastions, NAT gateways, and some managed services genuinely need public IPs — handle these with a tag-conditioned rule (allowAll only where network-tier=dmz) rather than blanket exceptions, so the exception is explicit and auditable. Limit: it governs Compute Engine external IPs specifically; serverless ingress, Cloud SQL public IP, and GKE need their own constraints.

4. gcp.resourceLocations (regulated folder). This list constraint restricts the physical location of newly created resources to an allowed set — using Google-maintained value groups like in:eu-locations you get data-residency enforcement without enumerating every region. Technically the service checks the requested location/region/zone field against the effective allow set at create time and rejects out-of-region requests. Trade-offs: some global services (IAM, certain Cloud DNS, global load balancers) and some multi-region behaviors are not fully constrainable this way, so resourceLocations is necessary-but-not-sufficient for full sovereignty — pair it with VPC Service Controls and Assured Workloads for regulated estates. Limit: it applies to new resources; pre-existing out-of-region resources need separate remediation.

5. Guarding the guardrail — orgpolicy.policyAdmin scoping + SetOrgPolicy alerting. The most important control is meta: whoever can change org policy can remove every other control. Grant roles/orgpolicy.policyAdmin to a small central group at the organization node only, never at folders (so no team can override org invariants for their subtree), and gate changes behind change management. Technically, pair this with a log sink filtering for protoPayload.methodName="SetOrgPolicy" or the v2 google.cloud.orgpolicy.v2.OrgPolicy.CreatePolicy/UpdatePolicy/DeletePolicy methods, routed to BigQuery/Pub-Sub, with a real-time alert on any policy weakening. Trade-off: central control creates a bottleneck for teams needing exceptions — mitigate with a well-defined exception request path and dry-run testing. This closes the "folder-admin override" and "silent drift" gaps that defeat every other guardrail.

6. Detection & Response

What to log / telemetry sources

Org Policy changes and enforcement violations are captured in Cloud Audit Logs:

  • Admin Activity logs (always on, free): SetOrgPolicy, ClearOrgPolicy, and v2 methods google.cloud.orgpolicy.v2.OrgPolicy.CreatePolicy, UpdatePolicy, DeletePolicy, CreateCustomConstraint, DeleteCustomConstraint. Key fields: protoPayload.methodName, protoPayload.authenticationInfo.principalEmail, protoPayload.resourceName, protoPayload.request (the new policy body), protoPayload.serviceName="orgpolicy.googleapis.com" (or cloudresourcemanager.googleapis.com for legacy v1).
  • Data Access logs (must be explicitly enabled) for reads if you want to see who queried policies.
  • Enforcement violations: when a constraint blocks an action, the denied API call appears in that service's Admin Activity log with an error status (e.g. compute.instances.insert with status.code=9 FAILED_PRECONDITION and message referencing the constraint). Dry-run policy violations surface as orgpolicy audit entries without blocking.
  • Security Command Center (SCC) Premium/Enterprise generates findings like Public IP address, Public bucket ACL, SQL public IP, and Org policy misconfigurations via Security Health Analytics — a managed detection layer over these same constraints.

Best practice: create an aggregated log sink at the Organization level exporting to BigQuery and/or Pub-Sub so all child-project audit logs are centralized and immune to a single project's log config being tampered with.

Sample detections

Detection 1 — org policy weakened (BigQuery SQL over exported audit logs).

SELECT
  timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS actor,
  protopayload_auditlog.methodName AS method,
  protopayload_auditlog.resourceName AS target,
  protopayload_auditlog.requestJson AS request_body
FROM `security_logs.cloudaudit_googleapis_com_activity`
WHERE protopayload_auditlog.serviceName IN
      ('orgpolicy.googleapis.com','cloudresourcemanager.googleapis.com')
  AND protopayload_auditlog.methodName IN
      ('SetOrgPolicy','ClearOrgPolicy',
       'google.cloud.orgpolicy.v2.OrgPolicy.UpdatePolicy',
       'google.cloud.orgpolicy.v2.OrgPolicy.DeletePolicy')
  AND (REGEXP_CONTAINS(protopayload_auditlog.requestJson, r'"allowAll"\s*:\s*true')
       OR REGEXP_CONTAINS(protopayload_auditlog.requestJson, r'"enforce"\s*:\s*false')
       OR REGEXP_CONTAINS(protopayload_auditlog.requestJson, r'"inheritFromParent"\s*:\s*false'))
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
ORDER BY timestamp DESC;

                       "google.cloud.orgpolicy.v2.OrgPolicy.DeleteCustomConstraint")
| project TimeGenerated, Actor=PrincipalEmail, MethodName, ResourceName, RequestBody
| order by TimeGenerated desc

Response playbook

  1. Triage / validate. Confirm the SetOrgPolicy event is unauthorized: check principalEmail, source IP (callerIp), whether it correlates with an approved change ticket, and which constraint was weakened (public IP? SA keys? residency?).
  2. Contain. Immediately re-apply the correct policy from your source-of-truth IaC (Terraform/gcloud org-policies set-policy) to restore the guardrail. If the actor holds orgpolicy.policyAdmin, revoke that binding and any tokens (gcloud iam service-accounts keys list/disable; revoke OAuth sessions) to stop re-weakening.
  3. Assess blast radius. Enumerate resources created while the guardrail was down — e.g. list VMs with external IPs, buckets with public bindings, SA keys created, resources outside allowed regions — using Cloud Asset Inventory (gcloud asset search-all-resources) scoped to the window.
  4. Eradicate. Delete/quarantine the offending resources (detach public IPs, remove allUsers bindings, delete unauthorized SA keys, migrate out-of-region data). Rotate any credentials that were exposed while exposed.
  5. Recover. Re-enforce all affected constraints at the org node, verify with gcloud org-policies describe ... --effective on sample projects, and confirm SCC findings clear.
  6. Harden. Move Org Policy to IaC-only (deny interactive orgpolicy.policyAdmin, enforce changes through pipeline), enable Access Approval for policy changes, verify the org-level log sink cannot be disabled by project admins, and add the weakening-detection rule to standing alerts if it wasn't already the trigger.
  7. Post-incident. Document root cause (over-delegation? missing alert?), and add a preventive control: scope policyAdmin to org node only, add drift-detection in CI (gcloud org-policies describe diffed against desired state).

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is the GCP resource hierarchy and why does it matter for security?
The hierarchy is Organization → Folder → Project → resource, with a single Organization node bound to a Cloud Identity/Workspace domain. It matters because both IAM policies and Organization Policies inherit downward, so the node where you attach a control defines its blast radius. Attaching a guardrail at the Organization node makes it apply to every project automatically, which is how you enforce company-wide invariants without touching each project. It also gives you a clean unit for delegation and billing.
Why asked / signal: Confirms the candidate has the foundational mental model everything else builds on.

Q2. How is Organization Policy different from IAM?
IAM answers "who can do what" — it binds principals to roles and is additive. Org Policy answers "what configurations are allowed to exist" — it constrains the shape of resources regardless of caller. A create-VM call must pass both gates: the caller needs the IAM permission and the resulting resource must not violate any constraint. Org Policy applies even to Organization Admins, so it's a guardrail you can't IAM-escalate around.
Why asked / signal: This distinction is the crux of the topic; conflating the two is a red flag.

Q3. What's the difference between a boolean constraint and a list constraint?
A boolean constraint is simply enforced true/false — e.g. iam.disableServiceAccountKeyCreation either blocks SA-key creation or doesn't. A list constraint carries allowedValues/deniedValues drawn from a value domain — e.g. gcp.resourceLocations allowing in:eu-locations. List constraints support merge-with-parent inheritance and allowAll/denyAll shortcuts; booleans are just overridden by the most specific node.
Why asked / signal: Tests concrete familiarity with the constraint model, not just the concept.

Q4. Name three org policy constraints you'd enforce on day one and why.
iam.disableServiceAccountKeyCreation (leaked exportable keys are the top GCP credential-theft vector); storage.publicAccessPrevention + uniformBucketLevelAccess (public buckets are the top data-exposure vector); and compute.vmExternalIpAccess denyAll (shrinks internet attack surface). I'd add sql.restrictPublicIp and gcp.resourceLocations shortly after. Each turns a "please don't" policy into a platform-enforced invariant.
Why asked / signal: Reveals whether the candidate knows the actual high-ROI controls by name.

Q5. If IAM already lets me restrict who can make a bucket public, why do I need publicAccessPrevention?
Because IAM is a grant of capability to trusted principals, and a trusted Storage Admin can still make a mistake or be compromised. Org Policy removes the capability from the platform entirely, so even a principal with storage.buckets.setIamPolicy cannot bind allUsers. It converts a human-trust control into a structural one, and it also blocks the legacy object-ACL path that bucket IAM review doesn't even see.
Why asked / signal: Tests understanding of defense-in-depth and the limits of IAM.

Tier 2 — System design / trade-offs

Q6. Design an org-policy strategy for a company with strict EU data residency for one product line but global operations otherwise.
I'd structure the hierarchy so the regulated product lives in its own Folder and enforce gcp.resourceLocations = in:eu-locations at that folder with inheritFromParent: false so it's authoritative for that subtree, while leaving the rest of the org unconstrained (or constrained to broader regions). For genuinely global services that resourceLocations can't fully bound (IAM, some DNS), I'd layer VPC Service Controls perimeters and, for the strictest workloads, Assured Workloads to get sovereignty guarantees and personnel controls. I'd deploy the location constraint in dry-run first to find existing out-of-region resources, remediate them, then flip to enforcing. Detection: SCC findings + a log-based alert on resources created outside allowed regions. The key trade-off is that residency is necessary-but-not-sufficient — data can still leave via application-layer exfil, so VPC-SC and DLP complete the picture.
Why asked / signal: Tests ability to compose org policy with adjacent controls for a real compliance requirement.

Q7. A team needs external IPs for a small set of bastion hosts but you enforce no-external-IP org-wide. How do you handle it?
I'd keep the org-wide compute.vmExternalIpAccess denyAll and add a tag-conditioned rule: create a Resource Manager tag like network-tier=dmz, and a policy rule that allowAll only where resource.matchTag(...) is true, with the default rule remaining denyAll. This makes the exception explicit, auditable, and scoped to tagged resources rather than a whole folder. I'd tightly control who can apply that tag (resourcemanager.tagUser) and alert on tag bindings. The alternative — carving a permissive folder — is worse because it opens the exception to any resource in that subtree. Ideally bastions are replaced with IAP TCP forwarding so no public IP is needed at all, which I'd push as the strategic answer.
Why asked / signal: Tests conditional org policy knowledge and preference for least-privilege exceptions.

Q8. How would you prevent a folder admin from overriding your org-wide guardrails?
The override risk comes entirely from roles/orgpolicy.policyAdmin being granted below the org node. I'd grant that role to a small central team at the Organization node only and never at folders or projects, so no delegated admin can set inheritFromParent: false / allowAll on their subtree. I'd back that with a real-time alert on SetOrgPolicy/UpdatePolicy events filtering for weakening patterns, move policy management to IaC with change control, and enable Access Approval. For extra assurance I'd periodically diff gcloud org-policies describe --effective across a sample of projects against the desired state to catch drift. The role is a root-of-trust — I treat it like org-admin.
Why asked / signal: This is the classic guardrail-bypass gap; strong candidates name the exact role and the detection.

Q9. Compare GCP Org Policy to AWS SCPs. Where is each stronger?
Both are top-down, hierarchy-inherited guardrails that bound what's allowed regardless of identity-policy grants. SCPs are IAM-action-based deny/allow with rich Condition expressions and can filter by any request context key, so they're extremely expressive over API actions. GCP Org Policy is more resource-configuration-oriented — predefined constraints target specific resource properties (external IP, public access, location) declaratively, which is often simpler for the common cases, and custom constraints (CEL) now give SCP-like conditional power over resource fields. SCPs don't grant permissions (only bound them); GCP Org Policy similarly never grants. AWS added RCPs (resource control policies) to bound resource-based policies, closing a gap GCP addresses via publicAccessPrevention-type constraints. Net: SCPs win on action-level conditional expressiveness; GCP wins on out-of-the-box declarative guardrails for common misconfigurations.
Why asked / signal: Cross-cloud fluency; senior candidates should translate between providers.

Q10. You want to enforce a rule Google has no predefined constraint for — say, require Shielded VM secure boot. How?
I'd write a custom constraint with a CEL condition over the resource type. Define a YAML naming the resource type (compute.googleapis.com/Instance), method types (CREATE, UPDATE), actionType: ALLOW, and condition: resource.shieldedInstanceConfig.enableSecureBoot == true. Register it with gcloud org-policies set-custom-constraint, then enforce it like any constraint at the org node. Because actionType is ALLOW and the condition must be true, any instance without secure boot is rejected at admission. I'd deploy in dry-run first to find non-compliant existing instances, remediate, then enforce. Custom constraints are GCP's escape hatch when the predefined catalog doesn't cover a needed invariant.
Why asked / signal: Tests knowledge of the extensibility mechanism and safe rollout.

Tier 3 — Expert / deep-dive

Q11. Walk through exactly how a list-policy conflict resolves across three levels: org allows regions A,B; folder denies B; project allows A,B,C with inheritFromParent true.
Evaluation walks root-to-resource merging list policies. Org contributes allow {A,B}. Folder denies {B}; merged with parent that's allow {A,B} minus deny {B} → effectively allow {A}, deny {B}. Project has inheritFromParent: true, so its allow {A,B,C} merges with the resolved parent (allow {A}, deny {B}). The deny-wins rule means B stays denied even though the project tries to allow it; A is allowed; C — allowed by project, not mentioned above — becomes allowed. So the effective policy is allow {A, C}, deny {B}. If the project had set inheritFromParent: false, it would discard the parent result and allow {A,B,C} outright (B would come back). This is why deny wins and inheritFromParent are the two levers that decide everything.
Why asked / signal: Only candidates who truly understand merge semantics can trace this; separates memorizers from experts.

Q12. Org Policy is enforced at admission, but what about resources that already existed before you enforced a constraint? How do you handle the gap and detect it?
Org Policy governs mutating operations going forward — it does not retroactively delete or reconfigure resources that predate enforcement. So enforcing compute.vmExternalIpAccess today does nothing to a VM that already has a public IP. I close this with a two-phase approach: (1) inventory existing violations using Cloud Asset Inventory (gcloud asset search-all-resources) and SCC Security Health Analytics findings, which continuously flag public IPs, public buckets, public SQL, and out-of-region resources regardless of when created; (2) remediate those, then enforce so no new ones appear. For ongoing assurance I run scheduled asset queries and diff against policy. The mental model: Org Policy is preventive at the create/update boundary; asset inventory + SCC is the detective control for the installed base.
Why asked / signal: Tests understanding that guardrails aren't retroactive — a common blind spot.

Q13. How does Org Policy interact with VPC Service Controls, and why aren't they redundant?
They solve orthogonal problems. Org Policy constrains resource configuration — can this bucket be public, can this VM have an external IP, in which region can it live. VPC Service Controls creates a service perimeter that constrains data movement — it stops a compromised or over-permissioned identity from exfiltrating data from an in-perimeter service (e.g. GCS, BigQuery) to a project outside the perimeter, defeating stolen-credential exfil and many SSRF-style attacks even when IAM would otherwise allow it. A public-access-prevention org policy stops anonymous exposure; VPC-SC stops authenticated exfil across a trust boundary. A mature GCP security posture layers both plus IAM: Org Policy for config invariants, VPC-SC for data-egress boundaries, IAM for least-privilege. Missing either leaves a class of attack open.
Why asked / signal: Distinguishing config-guardrails from data-perimeters is senior-level GCP fluency.

Q14. An attacker compromises a principal with orgpolicy.policyAdmin at the org node. Detail their likely actions and your layered defenses.
With org-node policyAdmin, the attacker's highest-value move is to weaken guardrails before doing damage: enforce: false on disableServiceAccountKeyCreation (to mint exportable keys), allowAll on vmExternalIpAccess (to create internet-reachable pivots), disable publicAccessPrevention (to exfil via public buckets), and relax resourceLocations (to stage data in an attacker-friendly region). Each is a SetOrgPolicy event. Layered defenses: (1) least-privilege so almost no one holds this role and it's org-node-only; (2) real-time alerting on SetOrgPolicy/UpdatePolicy weakening patterns with auto-revert from IaC; (3) Access Approval + MFA/hardware keys on the admins so a stolen password alone can't act; (4) an org-level log sink the attacker can't disable per-project; (5) VPC-SC so even with guardrails down, data can't leave the perimeter without also breaching the perimeter config; (6) Cloud Asset Inventory to detect the resources they created. The theme: defense-in-depth so no single role compromise is game-over, and fast detect-and-revert.
Why asked / signal: Tests threat-modeling of the control plane itself — the guardrail that guards the guardrails.

Q15. Critique the statement "we enforced it at the org, so we're safe." Where does org policy fall short, and what compensating controls do you add?
It's a dangerous half-truth. Org policy falls short in several ways: (1) it's not retroactive — pre-existing violations persist; (2) it can be overridden by anyone holding policyAdmin below the org node, or via inheritFromParent: false/allowAll; (3) dry-run policies look enforced but don't block — an easy misconfiguration; (4) constraints have coverage gaps — a boolean covers Compute external IPs but not every service's public exposure, so you need multiple constraints and custom ones; (5) it governs configuration, not data movement or runtime behavior — an authenticated insider can still exfil within allowed config. Compensating controls: Cloud Asset Inventory + SCC for the installed base and drift, strict policyAdmin scoping + SetOrgPolicy alerting for override protection, verifying --effective and that spec (not just dryRunSpec) is set, VPC-SC for data egress, and IAM least-privilege + Deny policies for the actor dimension. "Safe" is the composition of these, not any single layer.
Why asked / signal: Senior signal is intellectual honesty about a control's limits and knowing the compensating stack.

9. Key Takeaways

  • The hierarchy Organization → Folder → Project → resource is the inheritance spine; where you attach IAM or Org Policy sets the blast radius, so put company-wide invariants at the Organization node.
  • IAM = who can do what; Org Policy = what can be configured at all. Both gates must pass; Org Policy binds even Organization Admins and can't be IAM-escalated around.
  • Constraints are boolean (enforced on/off) or list (allow/deny value sets, with allowAll/denyAll and merge-vs-replace inheritance). For lists, deny wins on conflict and inheritFromParent: false breaks the chain.
  • The five highest-ROI constraints: iam.disableServiceAccountKeyCreation, storage.publicAccessPrevention+uniformBucketLevelAccess, compute.vmExternalIpAccess, sql.restrictPublicIp, and gcp.resourceLocations.
  • Custom constraints (CEL) extend guardrails beyond Google's catalog — GCP's answer to SCP Condition expressiveness — and should be rolled out via dry-run first.
  • Org Policy is preventive and not retroactive: pair it with Cloud Asset Inventory + Security Command Center to find and remediate the pre-existing violating installed base.
  • The biggest bypass risk is over-delegating roles/orgpolicy.policyAdmin below the org node; scope it to a tiny central group at the org and alert on every SetOrgPolicy change.
  • Org Policy governs configuration; it does not stop authenticated data exfil — compose it with VPC Service Controls (data egress) and IAM least-privilege for a complete posture.
  • Use gcloud org-policies describe --effective to see the truth after inheritance resolves, and beware dryRunSpec being mistaken for an enforcing spec.
  • Manage all of this as IaC with change control, centralize audit logs in an org-level sink that project admins can't disable, and treat the guardrail layer itself as a monitored root-of-trust.
