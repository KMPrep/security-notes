GCP Service Accounts — Security Model, Hardening, and Detection

Domain: Cloud Security — GCP Identity & Access Management (IAM)
Scope: Service account types, credentials, impersonation, blast radius, org-policy hardening, detection engineering
MITRE ATT&CK: T1078.004 (Valid Accounts: Cloud Accounts), T1552.005 (Unsecured Credentials: Cloud Instance Metadata API), T1550.001 (Use Alternate Authentication Material: Application Access Token), T1098.001 (Account Manipulation: Additional Cloud Credentials)
Cross-refs: infra/Lateral-Movement/Cloud-Service-Accounts · Workload-Identity-Federation

1. Overview

A GCP service account (SA) is a non-human IAM principal — an identity that a workload (a Compute Engine VM, a GKE pod, a Cloud Function, a Cloud Run service, a CI job) uses to authenticate to Google APIs. It is simultaneously an identity (something you grant roles to, so it can be a member in an IAM policy) and a resource (something that lives inside a project, so it has its own IAM policy governing who may use or manage it). That dual nature is the single most important idea in this note: a service account can be a grantee of privilege and a target of privilege at the same time, and most real GCP compromises exploit the seam between the two.

At scale, service accounts are where cloud IAM actually gets dangerous. Humans log in interactively, hit MFA, and are watched by anomaly detection. Machines authenticate silently, thousands of times a minute, using credentials that are frequently long-lived, over-scoped, and reachable by any code that runs on the workload. When a defender reviews a GCP org, the service-account inventory — how many exist, what roles they hold, whether they have exported keys, and who can impersonate them — is the highest-signal artifact for estimating blast radius. Over-privileged service accounts are the most common cause of excessive lateral reach in GCP: an attacker who lands code execution on any resource inherits that resource's SA, and from there the reachable graph is exactly the union of that SA's roles plus every SA it can impersonate.

This note is written from the defender/reviewer seat. The goal is to explain the model precisely, enumerate the misconfigurations that weaken it, and give concrete hardening and detection you can apply and audit.

TL;DR
- A service account is both an identity (grantee of roles) and a resource (has its own IAM policy) — impersonation lives in that second policy, not the first.
- Blast radius = granted roles ∪ (SAs it can impersonate, transitively). Long-lived JSON keys extend that radius to "anywhere on the internet, indefinitely."
- The three impersonation permissions — iam.serviceAccounts.getAccessToken, iam.serviceAccounts.signJwt, and iam.serviceAccounts.actAs — are effectively "become this identity." Grant them narrowly, on the specific SA, ideally with IAM Conditions.
- Prefer keyless: attached SAs + the metadata server for workloads inside GCP, Workload Identity Federation for workloads outside GCP. Kill exported keys with org policy.
- Detect on iam.serviceAccounts.keys.create, GenerateAccessToken/SignJwt from unexpected callers/IPs, and any grant of an impersonation role — these map to ATT&CK T1078.004 and T1098.001.

2. How It Works (deep)

2.1 Core mechanics

A GCP service account is identified by an email of the form NAME@PROJECT_ID.iam.gserviceaccount.com (user-managed) or a Google-generated address for the managed/default kinds. Each SA also has a numeric unique ID and a stable OAuth2 client ID. It has no password and cannot log in interactively. It authenticates in one of two fundamentally different ways:

  1. By presenting a private key it holds — either a user-managed key you exported as JSON, or a Google-managed key that Google holds internally and never exposes (used transparently when a workload has an attached SA and asks the metadata server for a token). The SA signs a JWT assertion, exchanges it at Google's OAuth2 token endpoint, and receives a short-lived (~1 hour) access token.
  2. By being impersonated — another already-authenticated principal calls the IAM Credentials API and asks Google to mint a token for the SA. No key is involved; authorization is purely IAM (does the caller hold an impersonation permission on the target SA?).

Two policies, two questions. Understanding SAs requires holding two separate IAM evaluations in your head:

Question Which policy answers it Example permission
What can this SA do? The IAM policy on the resources the SA is a member of (project, bucket, etc.) storage.objects.get on a bucket
Who can act as / use / manage this SA? The IAM policy attached to the SA resource itself iam.serviceAccounts.actAs, .getAccessToken

The first defines the SA's blast radius. The second defines the SA's attack surface as a target. A common review mistake is auditing only the first.

The three service account types — and when each is used.

Type Email pattern Created by Default privilege Defender's stance
User-managed name@project.iam.gserviceaccount.com You None until you grant roles The right kind — one purpose-built SA per workload, least privilege
Default (Compute/App Engine) PROJECTNUMBER-compute@developer.gserviceaccount.com, PROJECT@appspot.gserviceaccount.com Auto-created when the API is enabled Historically Editor on the whole project Highest-risk default; strip roles or disable, block auto-grant

The default compute SA is the classic footgun. When the Compute Engine API is enabled, GCP creates PROJECTNUMBER-compute@developer.gserviceaccount.com and — under the legacy behavior — grants it the primitive Editor role on the project. Every VM launched without an explicit SA runs as this identity with --scopes often defaulting to broad access. Result: any code execution on any such VM = project-wide Editor. This is the single most common "how did they move so far" finding in GCP incident reviews.

2.2 Concrete configuration

Create a purpose-built, least-privilege SA (the right default):

# One SA per workload, named for its purpose
gcloud iam service-accounts create billing-exporter \
  --project=acme-prod \
  --display-name="Billing export job (read-only BQ)"

# Grant ONLY the narrow role it needs, and scope it to a single dataset
# (resource-level binding, not project-level)
bq add-iam-policy-binding \
  --member="serviceAccount:billing-exporter@acme-prod.iam.gserviceaccount.com" \
  --role="roles/bigquery.dataViewer" \
  acme-prod:billing_dataset

Attach the SA to a VM keylessly, with minimal cloud-platform scope:

gcloud compute instances create exporter-vm \
  --project=acme-prod --zone=us-central1-a \
  --service-account="billing-exporter@acme-prod.iam.gserviceaccount.com" \
  --scopes="https://www.googleapis.com/auth/cloud-platform" \
  --shielded-secure-boot --shielded-vtpm --shielded-integrity-monitoring
# No JSON key is ever created. The VM gets tokens from the metadata server.

How a workload gets its token (what your app SDK does under the hood):

# From inside the VM/pod — the metadata server mints a short-lived token
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
# {"access_token":"ya29.c.b0A...","expires_in":3599,"token_type":"Bearer"}

Application Default Credentials (ADC) automatically use this endpoint, so well-written code never touches a key file.

Impersonation done right — short-lived, scoped, no key exported. A CI pipeline or an operator uses --impersonate-service-account instead of downloading a key:

# The human/CI principal must hold roles/iam.serviceAccountTokenCreator on the target SA.
gcloud storage ls gs://acme-prod-artifacts \
  --impersonate-service-account=deployer@acme-prod.iam.gserviceaccount.com
# gcloud silently calls iamcredentials.generateAccessToken for you.

Scope the impersonation grant with an IAM Condition so a token-creator can only impersonate one SA, and (optionally) only during change windows:

gcloud iam service-accounts add-iam-policy-binding \
  deployer@acme-prod.iam.gserviceaccount.com \
  --member="group:release-engineers@acme.com" \
  --role="roles/iam.serviceAccountTokenCreator" \
  --condition='expression=resource.name == "projects/-/serviceAccounts/deployer@acme-prod.iam.gserviceaccount.com",title=only-deployer,description=Scope token creation to the deployer SA only'

Enforce the three highest-value org policies (kill keys, kill auto-Editor):

# 1. No one can create exportable user-managed keys
gcloud resource-manager org-policies enable-enforce \
  constraints/iam.disableServiceAccountKeyCreation --organization=123456789012

# 2. No one can upload their own public key material to an SA
gcloud resource-manager org-policies enable-enforce \
  constraints/iam.disableServiceAccountKeyUpload --organization=123456789012

# 3. Default SAs are NOT auto-granted Editor at project creation
gcloud resource-manager org-policies enable-enforce \
  constraints/iam.automaticIamGrantsForDefaultServiceAccounts --organization=123456789012

GKE Workload Identity — stop pods from inheriting the node SA:

# Cluster + node pool with Workload Identity and metadata concealment
gcloud container clusters create secure-cluster \
  --project=acme-prod --region=us-central1 \
  --workload-pool=acme-prod.svc.id.goog \
  --workload-metadata=GKE_METADATA

# Bind the K8s SA to a least-priv GCP SA; pods use THIS identity, not the node's.
apiVersion: v1
kind: ServiceAccount
metadata:
  name: billing-exporter
  namespace: batch
  annotations:
    iam.gke.io/gcp-service-account: billing-exporter@acme-prod.iam.gserviceaccount.com

# Allow the K8s SA to impersonate the GCP SA via the workloadIdentityUser role
gcloud iam service-accounts add-iam-policy-binding \
  billing-exporter@acme-prod.iam.gserviceaccount.com \
  --role="roles/iam.workloadIdentityUser" \
  --member="serviceAccount:acme-prod.svc.id.goog[batch/billing-exporter]"

With GKE_METADATA, the legacy 169.254.169.254 node-SA endpoint is concealed from pods, and each pod is scoped to exactly the GCP SA its K8s SA is bound to.

Case study 1 — Capital One (the metadata + over-privilege pattern). An attacker exploited a server-side request forgery (SSRF) flaw in a WAF-fronted application and used it to query the EC2 instance metadata service (IMDSv1), which returned temporary credentials for the instance's IAM role. That role had s3:ListBucket/s3:GetObject far more broadly than the app needed, so the credentials could enumerate and download ~100M records. The GCP-equivalent chain is identical: SSRF → metadata.google.internal/.../token → a token for an over-privileged attached SA. The control that stops it: (a) least-privilege on the attached SA so a stolen token reaches almost nothing, and (b) metadata hardening (IMDSv2 on AWS; GKE Workload Identity / metadata concealment and disable-legacy-endpoints=true on GCP) so SSRF can't trivially fetch the token.

Case study 2 — Default compute SA as Editor. A team spins up a VM to run a scheduled report and never sets --service-account or --scopes. The VM runs as the default compute SA, which the project inherited with roles/editor, and the default cloud-platform-adjacent scopes. A dependency in the report tool has an RCE. The attacker reads the metadata token and now holds Editor on the entire project — able to create new SAs, mint keys, modify firewall rules, and read every bucket. The control that stops it: enforce constraints/iam.automaticIamGrantsForDefaultServiceAccounts so new projects never auto-grant Editor, replace the default SA with a purpose-built least-privilege SA, and set --no-scopes/--no-service-account on workloads that don't need cloud access.

Case study 3 — Impersonation granted too broadly. During a review you find roles/iam.serviceAccountTokenCreator granted to allAuthenticatedUsers-adjacent groups at the project level rather than on a specific SA. Because a project-level token-creator grant applies to every SA in the project, any member of that group — or any workload running as an SA in that group — can mint a token for the project's most privileged SA and escalate. The control that stops it: never grant the impersonation roles at project/folder/org scope; bind them on the individual target SA resource, add an IAM Condition pinning resource.name, and alert on any new token-creator binding.

Top 5 in depth

1. Eliminate exportable keys with org policy (disableServiceAccountKeyCreation + disableServiceAccountKeyUpload).
A user-managed key is an asymmetric private key you can download as JSON. Unlike a metadata token, it does not expire and is valid from any network on earth. It is the single worst credential type in GCP because a leak is silent, durable, and global. The constraints/iam.disableServiceAccountKeyCreation list constraint blocks iam.serviceAccounts.keys.create org-wide; disableServiceAccountKeyUpload blocks the sibling move where an attacker (or well-meaning engineer) uploads their own public key to bind a key they control. Technically these are enforced at the Resource Manager policy layer before the IAM mutation commits, so they can't be bypassed by anyone lacking org-policy-admin. Enforce them at the organization node and use policy exceptions (dry-run first, then scoped enforce=false on a narrow folder) only for the rare legacy integration that genuinely can't use keyless auth. Trade-off: a handful of external systems (on-prem cron, third-party SaaS) may still need a key; the answer is Workload Identity Federation, not an exception — reserve exceptions for weeks, not forever, and alert on any key that exists.

2. Prefer keyless everywhere: attached SAs inside GCP, WIF outside.
For anything running on GCP (GCE, GKE, Cloud Run, Cloud Functions, Cloud Build), attach an SA and let ADC pull short-lived tokens from the metadata server — no key ever exists. For anything running off GCP (GitHub Actions, on-prem, another cloud), use Workload Identity Federation: the external platform's OIDC/SAML token is exchanged at Google's STS for a short-lived federated token, then optionally used to impersonate a GCP SA — again, no exported key. Technically WIF verifies the external IdP's signature and evaluates attribute conditions you define (e.g. only tokens whose assertion.repository == "acme/deploy" and ref == "refs/heads/main"), so trust is pinned to a specific external identity rather than a stealable secret. Trade-off: WIF requires careful attribute-condition design — a permissive mapping (attribute.actor unconstrained) re-opens the door — so the control's strength lives entirely in the condition. See the Workload-Identity-Federation note.

3. Scope impersonation to the individual SA with IAM Conditions, and treat the three impersonation permissions as admin.
iam.serviceAccounts.getAccessToken, iam.serviceAccounts.signJwt/signBlob, and iam.serviceAccounts.actAs all mean, effectively, "become this SA." getAccessToken mints a live OAuth2 token; signJwt lets the caller produce SA-signed assertions (usable to obtain tokens for downstream services or to forge identity to systems that trust the SA's signature); actAs is checked at deploy time — it lets a principal create a VM/function/job that runs as the target SA, so it's escalation-by-deployment. Because these are equivalent to holding the target SA's entire role set, bind the corresponding roles (serviceAccountTokenCreator, serviceAccountUser) on the specific SA resource, never at project/folder/org scope, and pin them with an IAM Condition on resource.name. Technically, granting serviceAccountTokenCreator at project scope silently confers it on every SA in the project — a frequent hidden escalation path. Trade-off: resource-scoped bindings are more numerous and slightly more work to manage; that verbosity is the least-privilege property you want, and IaC (Terraform) makes it maintainable.

4. Neutralize the default compute SA and metadata exposure.
Enforce constraints/iam.automaticIamGrantsForDefaultServiceAccounts so newly created projects never auto-grant roles/editor to default SAs. For existing projects, remove primitive roles from the default SA and replace it with purpose-built SAs; for workloads needing no cloud access, launch with --no-service-account or --no-scopes. On GKE, enable Workload Identity (--workload-pool + --workload-metadata=GKE_METADATA) so pods receive tokens for their bound GCP SA and the legacy 169.254.169.254 node-SA endpoint is concealed; on plain GCE, set disable-legacy-endpoints=true. This directly breaks the SSRF→metadata→token→lateral-movement chain that powered Capital One and every TeamTNT container campaign. Trade-off: Workload Identity requires per-workload SA bindings and a small migration effort; the payoff is that a single pod RCE no longer yields the node's (often broad) identity.

5. Least-privilege role design + continuous blast-radius analysis.
Blast radius equals granted roles ∪ impersonatable SAs. So the durable control is disciplined role design: one SA per workload, predefined or custom roles (never the primitive Owner/Editor/Viewer), resource-scoped bindings (a bucket, a dataset — not the project), and periodic review with IAM Recommender (surfaces unused permissions from 90-day usage) and Policy Analyzer (answers "who can access what" and expands impersonation chains). Technically, Recommender computes the delta between granted and used permissions and proposes a tighter role; Policy Analyzer traverses the IAM graph including actAs/token-creator edges so you can see transitive reach. Trade-off: aggressive least-privilege can break workloads when usage patterns change; run Recommender in observe mode, stage role tightening in non-prod, and keep a fast path to widen a role under change management.

6. Detection & Response

What to log / telemetry sources

Service-account abuse is visible almost entirely in Cloud Audit Logs, but two of the most important event classes are off by default:

Log stream Contains Default state Must-enable
Admin Activity audit logs iam.serviceAccounts.create, keys.create, SetIamPolicy, create/enable/disable SA Always on, free —
Data Access audit logs (IAM/credentials) GenerateAccessToken, SignJwt, SignBlob, GetIamPolicy Off by default Enable DATA_READ/ADMIN_READ for iam.googleapis.com + iamcredentials.googleapis.com
VPC Flow / firewall logs Egress to 169.254.169.254 from unexpected pods Off Enable for metadata-exfil detection
Access Transparency Google-staff access to your resources Off (needs enablement) Optional

Key fields to pivot on: protoPayload.methodName, protoPayload.authenticationInfo.principalEmail, protoPayload.requestMetadata.callerIp, protoPayload.resourceName (the target SA), and for token minting protoPayload.request.scope. Enable data-access logging for the credentials API explicitly:

# gcloud organizations get-iam-policy ... then set auditConfigs:
auditConfigs:
- service: iamcredentials.googleapis.com
  auditLogConfigs:
  - logType: DATA_READ
  - logType: ADMIN_READ
- service: iam.googleapis.com
  auditLogConfigs:
  - logType: DATA_READ
  - logType: ADMIN_READ

Sample detections

Detection 1 — New service-account key created (BigQuery over exported audit logs). Any key creation in a key-disabled org is either a policy gap or an attacker adding credentials (T1098.001).

-- Flag creation of user-managed SA keys
SELECT
  timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS actor,
  protopayload_auditlog.requestMetadata.callerIp          AS src_ip,
  protopayload_auditlog.resourceName                      AS target_sa
FROM `acme-logs.security.cloudaudit_googleapis_com_activity`
WHERE protopayload_auditlog.methodName =
      'google.iam.admin.v1.CreateServiceAccountKey'
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
ORDER BY timestamp DESC;

  selection:
    data.protoPayload.methodName:
      - 'GenerateAccessToken'
      - 'SignJwt'
    data.protoPayload.serviceName: 'iamcredentials.googleapis.com'
  filter_known:
    data.protoPayload.authenticationInfo.principalEmail:
      - 'deployer@acme-prod.iam.gserviceaccount.com'
      - 'release-ci@acme-prod.iam.gserviceaccount.com'
  condition: selection and not filter_known
level: high
tags:
  - attack.t1550.001
  - attack.t1078.004
falsepositives:
  - New legitimate CI runner not yet allow-listed

Detection 3 — Any new grant of an impersonation/escalation role (KQL-style, log-based metric). Alert whenever SetIamPolicy adds serviceAccountTokenCreator, serviceAccountUser, iam.serviceAccountKeyAdmin, or a primitive role.

SELECT timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS actor,
  protopayload_auditlog.resourceName AS target,
  binding.role AS granted_role,
  member
FROM `acme-logs.security.cloudaudit_googleapis_com_activity`,
  UNNEST(JSON_QUERY_ARRAY(protopayload_auditlog.servicedata_v1_iam.policyDelta.bindingDeltas)) AS bd,
  UNNEST([JSON_VALUE(bd.role)]) AS binding_role
WHERE protopayload_auditlog.methodName LIKE '%SetIamPolicy'
  AND JSON_VALUE(bd.action) = 'ADD'
  AND JSON_VALUE(bd.role) IN (
    'roles/iam.serviceAccountTokenCreator',
    'roles/iam.serviceAccountUser',
    'roles/iam.serviceAccountKeyAdmin',
    'roles/owner', 'roles/editor');

Detection 4 — Metadata endpoint reached from a pod that shouldn't (behavioral). With Workload Identity enabled, direct pod egress to 169.254.169.254 for the token path is anomalous; alert on VPC flow logs or a runtime sensor (Falco rule contact cloud metadata service).

Response playbook (SA compromise)

  1. Detect & scope — from the audit log, pin the affected SA(s), the actor principal, callerIp, and the time window. Pull all GenerateAccessToken/SignJwt/SetIamPolicy events for that SA and any SA it can impersonate (expand the chain with Policy Analyzer).
  2. Contain (cut the credential path) — disable the compromised SA (gcloud iam service-accounts disable), delete any user-managed keys (gcloud iam service-accounts keys delete), and remove offending IAM bindings (revoke the impersonation grant / self-added role). Because access tokens are short-lived (~1h), disabling the SA and revoking keys bounds the exposure quickly.
  3. Contain (stop lateral spread) — revoke any bindings the attacker added (new token-creator grants, new SA keys, new SAs). If a VM/pod was the entry point, isolate it (firewall deny egress) and snapshot for forensics before termination.
  4. Eradicate — remove attacker-created SAs/keys/functions, rotate any downstream secrets the SA could read (KMS-wrapped secrets, DB creds), and confirm no persistence via actAs-deployed resources (Cloud Functions/Run services running as high-priv SAs).
  5. Recover — re-create the workload with a purpose-built least-privilege SA, keyless (attached SA or WIF), and re-enable service only after least-privilege is verified with Recommender.
  6. Harden & learn — if key creation was possible, enforce disableServiceAccountKeyCreation; if impersonation was too broad, re-scope with resource-level bindings + IAM Conditions; enable Data Access logs if they were off; add the actor/IP to detections. Feed timeline into the incident review.

7. Practice Questions

Tier 1 — Fundamentals

1. What is a GCP service account and how is it different from a user account?
A service account is a non-human IAM principal that workloads use to authenticate to Google APIs, identified by an email like name@project.iam.gserviceaccount.com. Unlike a user, it has no password and cannot log in interactively; it authenticates by holding a private key or by being impersonated, and it receives short-lived OAuth2 tokens. Crucially it is both an identity (you grant it roles) and a resource (it has its own IAM policy governing who can use it). That dual nature is why service-account governance is subtler than user governance.
Why asked / signal: confirms the candidate knows machine identity is a distinct, dual-natured object — the basis for everything else.

2. Name the three service-account types and when each is used.
User-managed SAs are ones you create for a specific workload and grant least-privilege roles to — the right default. Default SAs (compute/App Engine) are auto-created and historically carried roles/editor on the whole project, which is the classic over-privilege footgun. Google-managed SAs (service agents, service-...@gcp-sa-...) are created and controlled by Google so a service can operate on your behalf; you leave them alone and never grant them extra roles.
Why asked / signal: tests whether they can spot the dangerous default and the "don't touch" managed identities.

3. Why are long-lived JSON service-account keys considered a liability?
They are asymmetric private keys that never expire and are valid from any network on earth, so a single leak — in a Git commit, Docker layer, or CI log — grants durable, global access with no MFA. They are silently harvested by bots scanning public repos within minutes. The defensive answer is to not create them: use attached SAs inside GCP and Workload Identity Federation outside, and enforce disableServiceAccountKeyCreation org-wide.
Why asked / signal: the candidate should instinctively treat exported keys as the worst credential class.

4. How does a VM or pod get credentials without a key file?
It uses an attached service account plus the metadata server. Application Default Credentials call http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token, which returns a short-lived (~1 hour) OAuth2 access token minted with a Google-managed key that is never exposed. No secret is stored on disk, and rotation is automatic.
Why asked / signal: checks understanding of keyless issuance and the metadata endpoint, which is also the SSRF target.

5. What does the actAs permission do and why does it matter?
iam.serviceAccounts.actAs lets a principal deploy a resource (VM, function, Cloud Run service, pipeline) that runs as a given SA — it's checked at deploy time. It matters because it lets a lower-privileged principal cause code to execute with a higher-privileged SA's identity, which is privilege escalation by deployment. You scope it by granting roles/iam.serviceAccountUser on the specific target SA, never at project level.
Why asked / signal: actAs is the most misunderstood escalation gate; interviewers want to see it flagged.

Tier 2 — System design / trade-offs

6. Design least-privilege service-account architecture for a microservices platform on GKE.
One GCP SA per service, bound via GKE Workload Identity to that service's Kubernetes SA in its own namespace, so pods never inherit the node SA. Each GCP SA gets only predefined/custom roles scoped to the exact resources it touches (a specific bucket, a specific topic), never primitive roles or project-wide bindings. Enable --workload-pool and --workload-metadata=GKE_METADATA to conceal the legacy metadata endpoint, and enforce disableServiceAccountKeyCreation so no service ships a key. Continuously prune with IAM Recommender and map impersonation edges with Policy Analyzer. The trade-off is more SAs and bindings to manage, which you absorb with Terraform and naming conventions — the payoff is that any single pod RCE is contained to one narrow identity.
Why asked / signal: tests holistic least-privilege design plus the GKE-specific metadata control.

7. When would you use a service-account key, WIF, or an attached SA — and how do you decide?
Decision tree by where the workload runs: inside GCP → attached SA + metadata (keyless, automatic); outside GCP (GitHub Actions, on-prem, another cloud) → Workload Identity Federation, which exchanges the external OIDC/SAML/AWS identity for a short-lived GCP token with no exported key; a JSON key only as a genuinely last resort for a legacy system that can't do OIDC, and even then time-boxed and alarmed. The deciding factor is: can the workload present a verifiable external identity? If yes, federate; a static key is never the right long-term answer. Trade-off: WIF requires careful attribute-condition design, which is where the security actually lives.
Why asked / signal: wants a crisp keyless-first framework rather than reflexive key usage.

8. serviceAccountTokenCreator was granted at project scope. What's the risk and how do you fix it?
At project scope the role applies to every SA in the project, so any member of that binding — or any workload running as one of those SAs — can mint a token for the project's most privileged SA and escalate. The fix is to remove the project-level binding and re-grant serviceAccountTokenCreator only on the specific SA that legitimately needs to be impersonated, pinned with an IAM Condition on resource.name, and ideally limited to a change window. Then alert on any future SetIamPolicy that adds this role at folder/project/org scope. Trade-off: many resource-scoped bindings instead of one broad one — but that verbosity is the least-privilege you want.
Why asked / signal: tests understanding that impersonation-role scope is a silent escalation surface.

9. How do you prevent SSRF from turning into cloud compromise on GCP?
Two layers. First, reduce what a stolen token is worth: least-privilege attached SAs so the metadata token reaches almost nothing, and per-workload identity via GKE Workload Identity so a pod can't get the broad node SA. Second, make the token harder to fetch: GKE_METADATA conceals the legacy endpoint, disable-legacy-endpoints=true on GCE blocks the v0.1/v1beta1 paths, and network policy denies pod egress to 169.254.169.254. Fix the SSRF itself (allow-list outbound, block link-local ranges). This is the exact chain that produced Capital One, so defense-in-depth across identity and metadata is the answer, not any single control.
Why asked / signal: connects an app-layer bug to cloud identity blast radius — a senior cross-domain signal.

10. You must let GitHub Actions deploy to GCP with no static secrets. How?
Configure Workload Identity Federation: create a workload identity pool and an OIDC provider trusting https://token.actions.githubusercontent.com, with an attribute mapping that pins assertion.repository, assertion.ref, and audience. Grant the pool's principal set roles/iam.workloadIdentityUser on a purpose-built deployer SA, and constrain with an attribute condition so only acme/deploy on refs/heads/main can federate. The Action requests an OIDC token, exchanges it at Google STS for a short-lived token, and impersonates the deployer SA — no key ever exists. Trade-off: the entire security rests on the attribute condition; a loose mapping (unconstrained actor/repository) lets any repo assume the identity, so review it like an admin grant.
Why asked / signal: tests real keyless-CI design and the WIF failure mode.

Tier 3 — Expert / deep-dive

11. Explain the difference between getAccessToken, signJwt, and signBlob, and why all three are escalation-equivalent.
getAccessToken has the IAM Credentials API mint a live OAuth2 access token for the target SA, immediately usable against Google APIs within the granted scopes. signJwt has the SA sign a JWT payload you supply — usable to obtain OIDC/ID tokens for the SA or to authenticate to any system that trusts the SA's signing key, effectively forging the SA's identity. signBlob signs arbitrary bytes with the SA's key, which can be composed into signed URLs or custom assertions. All three let the caller act with the SA's key/identity, so holding any of them on a privileged SA is equivalent to holding that SA's entire role set — which is why they must be granted at resource scope, to almost no one, and audited as admin-tier permissions.
Why asked / signal: separates candidates who memorized actAs from those who understand the full signing/impersonation surface.

12. Walk through a full lateral-movement chain from a compromised low-privilege VM to project owner, and the control that breaks each hop.
(1) RCE on a VM → read the attached SA token from metadata [break: least-privilege SA + metadata concealment]. (2) Enumerate IAM to find the SA holds serviceAccountTokenCreator on a higher SA → mint a token for it [break: resource-scoped impersonation + Condition, alert on GenerateAccessToken]. (3) The higher SA holds resourcemanager.projects.setIamPolicy → grant self roles/owner [break: no setIamPolicy on workload SAs; deny primitive roles; alert on SetIamPolicy adding owner]. Alternatively, at hop 2 use actAs to deploy a Cloud Function running as an admin SA [break: actAs Condition + review deploy permissions]. Each hop is an IAM edge; the defensive program is to map those edges with Policy Analyzer and cut the token-creator/actAs/setIamPolicy edges that connect low-trust workloads to high-privilege identities.
Why asked / signal: elite signal — the candidate reasons about the IAM graph and control placement per hop.

13. Which permissions do you treat as "escalation-equivalent" and audit like admin, beyond the obvious impersonation ones?
Beyond actAs, getAccessToken, signJwt/signBlob: resourcemanager.projects/folders/organizations.setIamPolicy (grant yourself anything), iam.roles.update (add permissions to a custom role you hold), iam.serviceAccounts.getOpenIdToken, iam.serviceAccountKeys.create (mint a durable key for an SA you can reach — T1098.001), and deploy permissions that combine with actAs (cloudfunctions.functions.create, run.services.create, deploymentmanager.deployments.create, compute.instances.create with --service-account). Rhino Security Labs catalogued roughly three dozen such permissions. Defensively I inventory who holds any of them, treat them as break-glass, and alert on their use.
Why asked / signal: tests breadth of the GCP privilege-escalation permission set, not just the headline permissions.

14. Data Access audit logs are off by default. What are you blind to, and what exactly do you enable?
Admin Activity logs (always on) capture mutations like key creation and SetIamPolicy, but the use of impersonation — GenerateAccessToken, SignJwt, SignBlob on iamcredentials.googleapis.com — is a Data Access event and is off by default, so runtime impersonation is invisible to your SIEM until you enable it. You enable DATA_READ/ADMIN_READ audit configs for iamcredentials.googleapis.com and iam.googleapis.com (org-wide via the IAM policy auditConfigs), route logs to a BigQuery/Chronicle sink, and build detections on unexpected principalEmail/callerIp for token minting. Without this, an attacker impersonating across SAs leaves no trace. Trade-off: Data Access logs can be high-volume, so scope to IAM/credentials services rather than enabling globally.
Why asked / signal: a real detection-engineering gap that trips up many teams — knowing it is a strong senior signal.

15. Critique the security of Workload Identity Federation attribute mappings — where does it fail?
WIF's strength is that it pins trust to a verifiable external identity, but that guarantee is only as good as the attribute condition. Common failures: mapping only google.subject without constraining attribute.repository/attribute.ref, which lets any repo (or any actor) in the trusted org federate as the SA; accepting a broad audience; or granting workloadIdentityUser to the whole pool's principal set (principalSet://.../*) instead of a specific subject. An attacker who controls any workflow in a permissively mapped org can then impersonate the deployer SA. The hardening is a tight attribute_condition (assertion.repository == 'acme/deploy' && assertion.ref == 'refs/heads/main'), least-privilege on the impersonated SA, short token TTLs, and audit-logging the STS exchanges. WIF removes the key liability but relocates the trust decision into the mapping — review it as carefully as an IAM binding.
Why asked / signal: expert-level — tests that the candidate knows keyless doesn't mean risk-free; the trust just moves.

9. Key Takeaways

  • A service account is both an identity (grantee of roles) and a resource (has its own IAM policy) — impersonation lives in the second policy, and auditing only the first misses the escalation surface.
  • Blast radius = granted roles ∪ transitively-impersonatable SAs. Over-privileged SAs are the top cause of excessive lateral reach in GCP; every workload RCE inherits its SA.
  • Long-lived JSON keys are the worst credential class — non-expiring, globally valid, silently harvested. Prefer keyless everywhere and enforce disableServiceAccountKeyCreation + disableServiceAccountKeyUpload.
  • Keyless decision: attached SA + metadata for workloads inside GCP; Workload Identity Federation for workloads outside GCP; static keys only as a time-boxed, alarmed last resort.
  • The three impersonation permissions (getAccessToken, signJwt/signBlob, actAs) are "become this SA." Grant them on the individual SA resource with IAM Conditions, never at project scope.
  • The default compute SA with Editor is a classic footgun; enforce automaticIamGrantsForDefaultServiceAccounts, strip primitive roles, and use --no-service-account/--no-scopes.
  • On GKE, Workload Identity (--workload-pool + GKE_METADATA) stops pods inheriting the node SA and breaks the SSRF→metadata→token chain that produced Capital One.
  • Data Access audit logs are off by default — enable them for iamcredentials.googleapis.com/iam.googleapis.com or you are blind to impersonation (GenerateAccessToken/SignJwt).
  • Detect on key creation, unexpected token minting, and new impersonation/escalation grants (T1078.004, T1098.001, T1550.001); map impersonation edges continuously with Policy Analyzer and prune with IAM Recommender.
  • WIF removes the key liability but relocates trust into the attribute mapping — a loose condition re-opens the door, so review mappings as carefully as an admin IAM binding.
