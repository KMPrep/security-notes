GCP Workload Identity Federation (WIF) and GKE Workload Identity — Keyless Authentication for External and In-Cluster Workloads

Category: Cloud Security — Identity and Access Management (GCP)
Applies To: Workload Identity Federation (external identities), GKE Workload Identity (in-cluster), Cloud IAM, STS token exchange
Primary Standards / Docs: OpenID Connect Core 1.0, RFC 8693 (OAuth 2.0 Token Exchange), RFC 7519 (JWT), Google Cloud IAM WIF docs, GitHub Actions OIDC spec
Cross-refs: auth_authz/Federation/OpenID-Connect/README.md · auth_authz/Infrastructure/Workload-Identity/README.md · infra/Lateral-Movement/Cloud-Service-Accounts/README.md · ai_security/AI-Supply-Chain/Poisoned-Dependencies-And-Finetunes/README.md

1. Overview

Workload Identity Federation (WIF) lets an external identity — a GitHub Actions job, an AWS EC2 role, an Azure VM managed identity, an on-prem OIDC/SAML workload — obtain short-lived Google Cloud credentials without ever holding a Google service account key. Instead of exporting a key.json and pasting it into a CI secret (a long-lived, exfiltratable, rarely-rotated bearer credential), the workload presents an OIDC ID token (or SAML assertion / AWS GetCallerIdentity signature) that it already holds from its home identity provider, and Google's Security Token Service (STS) exchanges it for a federated access token scoped by IAM policy. GKE Workload Identity is the sibling mechanism for in-cluster workloads: it binds a Kubernetes ServiceAccount (KSA) to a Google service account (GSA) so pods authenticate to Google APIs through the GKE metadata server with no key mounted.

This matters at scale because service account keys are the single most abused credential in cloud breaches — they leak into Git history, container layers, CI logs, and Slack, and they never expire on their own. WIF eliminates the key entirely. But it moves the entire trust decision into two configuration surfaces — the provider's attribute mapping + attribute condition (CEL) and the IAM principalSet binding — and a permissive setting on either one silently converts "keyless and safe" into "anyone on the public internet who can mint an OIDC token from the same issuer can impersonate your production deployer." Missing repository/branch pinning on a GitHub provider is a live, mass-exploited misconfiguration class, not a theoretical one.

TL;DR
- WIF trades a stored static key for a federated trust relationship: an external OIDC/SAML/AWS identity → STS token exchange → short-lived GCP token, gated by an attribute condition and an IAM binding.
- The attribute condition (CEL) is the critical security control. A GitHub provider with no condition trusts the entire token.actions.githubusercontent.com issuer — i.e., every repository on GitHub. You must pin repository_owner + repository (+ ref/environment for prod).
- Two independent gates must both be tight: (1) the provider decides which tokens are accepted; (2) the IAM principalSet:// binding decides which accepted identity may impersonate which GSA. A wildcard on either is game over.
- GKE Workload Identity binds KSA → GSA via roles/iam.workloadIdentityUser on the member serviceAccount:PROJECT.svc.id.goog[namespace/ksa]; the GKE metadata server issues per-pod tokens, no key file.
- Threats to name cold: over-broad attribute conditions (missing repo/branch pin), audience confusion, forgeable subject mapping (google.subject=assertion.actor), and principalSet://.../* wildcards.

2. How It Works

2.1 Core mechanics

WIF replaces "prove you hold Google's secret" with "prove you already hold a token from an issuer Google trusts, and that the claims in that token match a condition I defined." The building blocks:

Object Resource / Identifier Role
Workload Identity Pool projects/PROJECT_NUM/locations/global/workloadIdentityPools/POOL_ID Namespace/container for external identities; isolates one trust domain from another
Workload Identity Provider .../workloadIdentityPools/POOL_ID/providers/PROVIDER_ID Configures the trusted issuer (OIDC issuer URI + JWKS, AWS account, or SAML metadata)
Attribute mapping google.subject=assertion.sub, attribute.repository=assertion.repository, ... Copies claims from the external token into Google attributes usable in IAM and conditions
Attribute condition CEL expression, e.g. assertion.repository_owner == 'my-org' Gate: token is rejected unless the CEL evaluates true. The provider's security boundary
Allowed audiences --allowed-audiences (defaults to provider full resource name) The aud value the external token must carry; defends against token replay/confusion
Federated principal principal://iam.googleapis.com/.../subject/SUBJECT The identity Google assigns after a successful exchange (one subject)
Federated principalSet principalSet://iam.googleapis.com/.../attribute.repository/my-org/my-app A set of identities sharing an attribute value; used in IAM bindings
Google service account (GSA) deployer@PROJECT.iam.gserviceaccount.com Optional impersonation target; the federated identity assumes its permissions
roles/iam.workloadIdentityUser IAM role on the GSA Grants the WIF principal/principalSet the right to impersonate that GSA
STS sts.googleapis.com/v1/token Performs the RFC 8693 token exchange (subject token → federated access token)
IAM Credentials API iamcredentials.googleapis.com generateAccessToken Exchanges the federated token for a GSA access token (the impersonation step)

Decision logic during an exchange (STS side):

  1. Verify the subject token's signature against the provider's issuer keys (JWKS fetched from the OIDC discovery document for OIDC; AWS SigV4 verification for AWS; XML signature for SAML).
  2. Verify the issuer matches the provider's configured issuer-uri, and the token is unexpired (exp/iat/nbf).
  3. Verify the audience (aud) is in the provider's allowed audiences (default: the provider's full resource name //iam.googleapis.com/projects/.../providers/PROVIDER_ID).
  4. Apply the attribute mapping to produce google.subject and attribute.*.
  5. Evaluate the attribute condition (CEL) against the mapped attributes / raw assertion. If false → deny.
  6. Mint a federated access token whose principal is principal://.../subject/<google.subject> and which also carries the mapped attribute.* values.
  7. That token can (a) access resources on which the principal/principalSet was granted IAM directly, or (b) be exchanged via generateAccessToken to impersonate a GSA on which it holds roles/iam.workloadIdentityUser.

The two independent gates are the recurring theme: the provider (steps 1–5) decides which tokens exist at all; IAM (step 7) decides what those tokens can touch. A weakness in either is sufficient for compromise.

GKE Workload Identity is a specialization. The cluster is created with a workload pool PROJECT_ID.svc.id.goog. Each namespace/KSA pair becomes a first-class member serviceAccount:PROJECT_ID.svc.id.goog[NAMESPACE/KSA]. A pod running under that KSA calls the GKE metadata server (a per-node DaemonSet, gke-metadata-server, intercepting 169.254.169.254), which brokers a token for the bound GSA. There is no key, no node-wide credential, and pods that are not bound get no Google access — this closes the old "every pod inherits the node's GCE service account" lateral-movement path (see infra/Lateral-Movement/Cloud-Service-Accounts/README.md).

2.2 Concrete configuration

A worked GitHub Actions → GCP WIF setup (the canonical case)

# ---- Variables ---------------------------------------------------------
export PROJECT_ID="acme-prod"
export PROJECT_NUMBER="$(gcloud projects describe "$PROJECT_ID" --format='value(projectNumber)')"
export POOL_ID="github-pool"
export PROVIDER_ID="github-provider"
export SA_EMAIL="gh-deployer@${PROJECT_ID}.iam.gserviceaccount.com"
export REPO="acme/payments-service"      # owner/repo we intend to trust

# ---- 1. Create the identity pool (a trust namespace) -------------------
gcloud iam workload-identity-pools create "$POOL_ID" \
  --project="$PROJECT_ID" --location="global" \
  --display-name="GitHub Actions pool"

# ---- 2. Create the OIDC provider WITH mapping AND a strict condition ----
# The issuer-uri is GitHub's shared, multi-tenant OIDC issuer. Every repo
# on GitHub is signed by the same keys, so the attribute-condition is the
# ONLY thing standing between "our repo" and "the entire internet".
gcloud iam workload-identity-pools providers create-oidc "$PROVIDER_ID" \
  --project="$PROJECT_ID" --location="global" \
  --workload-identity-pool="$POOL_ID" \
  --display-name="GitHub provider" \
  --issuer-uri="https://token.actions.githubusercontent.com" \
  --attribute-mapping="google.subject=assertion.sub,attribute.repository=assertion.repository,attribute.repository_owner=assertion.repository_owner,attribute.ref=assertion.ref,attribute.workflow=assertion.workflow" \
  --attribute-condition="assertion.repository_owner == 'acme' && assertion.repository == 'acme/payments-service' && assertion.ref == 'refs/heads/main'"

# ---- 3. Grant impersonation to ONLY that repo (scoped principalSet) -----
# Bind the workloadIdentityUser role to the repository attribute, not the
# whole pool. Combined with the ref pin above, only main of this one repo
# can impersonate the deployer SA.
gcloud iam service-accounts add-iam-policy-binding "$SA_EMAIL" \
  --project="$PROJECT_ID" \
  --role="roles/iam.workloadIdentityUser" \
  --member="principalSet://iam.googleapis.com/projects/${PROJECT_NUMBER}/locations/global/workloadIdentityPools/${POOL_ID}/attribute.repository/${REPO}"

# ---- 4. Grant the deployer SA only what it needs (least privilege) ------
gcloud storage buckets add-iam-policy-binding gs://acme-prod-artifacts \
  --member="serviceAccount:${SA_EMAIL}" --role="roles/storage.objectAdmin"

The GitHub Actions workflow that consumes it — note id-token: write is what lets the job mint an OIDC token at all:

name: deploy
on:
  push:
    branches: [ main ]
permissions:
  id-token: write        # REQUIRED: grants the job an OIDC token
  contents: read
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - id: auth
        uses: google-github-actions/auth@v2
        with:
          project_id: acme-prod
          # Full provider resource name — this becomes the STS audience.
          workload_identity_provider: "projects/123456789012/locations/global/workloadIdentityPools/github-pool/providers/github-provider"
          service_account: "gh-deployer@acme-prod.iam.gserviceaccount.com"
      - run: gcloud storage cp ./build/app.tar gs://acme-prod-artifacts/

The raw token-exchange flow (what google-github-actions/auth does under the hood)

# (a) Inside the runner, request an OIDC ID token from GitHub. The audience
#     is set to the provider resource name so Google will accept it.
AUD="//iam.googleapis.com/projects/123456789012/locations/global/workloadIdentityPools/github-pool/providers/github-provider"
GH_JWT="$(curl -s -H "Authorization: bearer $ACTIONS_ID_TOKEN_REQUEST_TOKEN" \
  "$ACTIONS_ID_TOKEN_REQUEST_URL&audience=$AUD" | jq -r '.value')"

# (b) Exchange the GitHub JWT for a Google FEDERATED access token (RFC 8693).
FED_TOKEN="$(curl -s -X POST "https://sts.googleapis.com/v1/token" \
  -H "Content-Type: application/json" \
  -d "{
    \"audience\": \"$AUD\",
    \"grantType\": \"urn:ietf:params:oauth:grant-type:token-exchange\",
    \"requestedTokenType\": \"urn:ietf:params:oauth:token-type:access_token\",
    \"scope\": \"https://www.googleapis.com/auth/cloud-platform\",
    \"subjectTokenType\": \"urn:ietf:params:oauth:token-type:jwt\",
    \"subjectToken\": \"$GH_JWT\"
  }" | jq -r '.access_token')"

# (c) Impersonate the deployer GSA (needs roles/iam.workloadIdentityUser).
gcloud auth print-access-token >/dev/null   # illustrative
curl -s -X POST \
  "https://iamcredentials.googleapis.com/v1/projects/-/serviceAccounts/gh-deployer@acme-prod.iam.gserviceaccount.com:generateAccessToken" \
  -H "Authorization: Bearer $FED_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"scope":["https://www.googleapis.com/auth/cloud-platform"],"lifetime":"900s"}'
# -> { "accessToken": "ya29....", "expireTime": "..." }  (max 1h, here 15m)

The decoded GitHub OIDC token (GH_JWT) carries the claims that your condition and mapping rely on:

{
  "iss": "https://token.actions.githubusercontent.com",
  "aud": "//iam.googleapis.com/projects/123456789012/.../providers/github-provider",
  "sub": "repo:acme/payments-service:ref:refs/heads/main",
  "repository": "acme/payments-service",
  "repository_owner": "acme",
  "ref": "refs/heads/main",
  "ref_type": "branch",
  "environment": "production",
  "workflow": "deploy",
  "actor": "octocat",
  "job_workflow_ref": "acme/payments-service/.github/workflows/deploy.yml@refs/heads/main",
  "exp": 1751000000, "iat": 1750999000
}

GKE Workload Identity setup (in-cluster)

# 1. Enable Workload Identity on the cluster and node pool.
gcloud container clusters update prod-cluster --region=us-central1 \
  --workload-pool="acme-prod.svc.id.goog"
gcloud container node-pools update default-pool --cluster=prod-cluster \
  --region=us-central1 --workload-metadata=GKE_METADATA

# 2. Create the GSA the pods should act as, and grant it least privilege.
gcloud iam service-accounts create ledger-reader --project=acme-prod
gcloud projects add-iam-policy-binding acme-prod \
  --member="serviceAccount:ledger-reader@acme-prod.iam.gserviceaccount.com" \
  --role="roles/spanner.databaseReader"

# 3. Bind the KSA -> GSA (the member string is the magic identifier).
gcloud iam service-accounts add-iam-policy-binding \
  ledger-reader@acme-prod.iam.gserviceaccount.com \
  --role="roles/iam.workloadIdentityUser" \
  --member="serviceAccount:acme-prod.svc.id.goog[ledger/ledger-ksa]"

# 4. Annotate the KSA so the metadata server maps it to the GSA.
apiVersion: v1
kind: ServiceAccount
metadata:
  name: ledger-ksa
  namespace: ledger
  annotations:
    iam.gke.io/gcp-service-account: ledger-reader@acme-prod.iam.gserviceaccount.com
---
apiVersion: apps/v1
kind: Deployment
metadata: { name: ledger, namespace: ledger }
spec:
  template:
    spec:
      serviceAccountName: ledger-ksa   # pods now get GSA tokens, no key file
      containers:
        - name: app
          image: gcr.io/acme-prod/ledger:1.4.2

Inside any pod under ledger-ksa, a call to the metadata server returns a token for ledger-reader — and only the Spanner read scope it was granted:

curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
# {"access_token":"ya29...","expires_in":3599,"token_type":"Bearer"}

Case study 1 — the condition-less GitHub provider. A team wires up google-github-actions/auth following a blog post, creates an OIDC provider with --issuer-uri=https://token.actions.githubusercontent.com but omits --attribute-condition, and binds roles/iam.workloadIdentityUser on their prod deployer SA to principalSet://.../* (the whole pool). Because GitHub's issuer is shared across every repository on the planet and all tokens are signed by the same keys, any GitHub user can create a public repo, add a workflow with id-token: write, request an OIDC token with the victim's provider audience, and exchange it for the deployer SA. There is no repository check anywhere in the path. The fix is two lines: an attribute condition pinning repository_owner/repository, and a principalSet scoped to attribute.repository/acme/payments-service. Google now requires an attribute condition when creating a provider for a recognized shared issuer like GitHub precisely because this mistake was so common.

Case study 2 — audience confusion. A provider is created with --allowed-audiences widened to accept a generic value (or the workflow requests a default audience like https://github.com/acme that other unrelated tooling also uses). An OIDC token minted for a different purpose — say, an internal service that also trusts https://github.com/acme as an audience — becomes replayable against the WIF provider. The token's sub/repository claims are legitimate, so if the IAM binding is only scoped by repository_owner, a low-trust repo's token now impersonates a high-trust SA. The mitigation is to keep the audience equal to the provider's full resource name (the default) so a token is only ever valid for exactly one provider, and to pin the specific repository, not just the owner.

Case study 3 — GKE node-SA inheritance (pre-WIF). Before GKE Workload Identity, every pod on a node could reach 169.254.169.254 and pull the node's GCE default service account token. A single compromised pod (SSRF, RCE, or a malicious sidecar image) inherited the node SA's broad permissions and moved laterally across the project — the exact pattern in infra/Lateral-Movement/Cloud-Service-Accounts/README.md. Enabling Workload Identity (--workload-metadata=GKE_METADATA) makes the metadata server hand out only the bound GSA per KSA, and unbound pods get nothing. The control that would have stopped the lateral movement is precisely this per-pod identity plus metadata concealment.

Top 5 controls in depth

1. The attribute condition (CEL) — pin owner, repo, and ref. This is the provider's security boundary and the single most important control. Because a shared issuer signs tokens for everyone, the condition is the only place you assert "I trust this repository on this branch." Implement it as a conjunction of immutable claims:

--attribute-condition="assertion.repository_owner == 'acme' && \
  assertion.repository == 'acme/payments-service' && \
  assertion.ref == 'refs/heads/main'"

For production deploys prefer pinning assertion.environment == 'production' (a GitHub Environment with required reviewers) over a branch, because environments add human approval and cannot be reached from arbitrary PRs. Trade-off: conditions are brittle — a repo rename, a monorepo split, or a switch from branch to tag deploys breaks auth, and a logic slip (|| where you meant &&, or a typo in a claim name) silently widens trust with no error. Mitigate by code-reviewing the CEL and testing it against sample tokens before rollout. Limit: the condition can only reference claims the issuer actually sets and signs; you cannot invent a trust attribute GitHub doesn't emit.

2. Scoped principalSet IAM binding — never bind the whole pool. The second gate. Bind roles/iam.workloadIdentityUser to attribute.repository/<owner>/<repo>, not principalSet://.../* and not merely attribute.repository_owner/<org> (which admits every repo in the org, including low-trust ones). This gives defense in depth: even if a condition regresses, the IAM binding still limits which SA a token can assume. Trade-off: per-repo bindings multiply as you scale, so many teams template them in Terraform. Limit: it governs impersonation, not the token's existence — a broad provider still lets an attacker authenticate, they just can't reach a well-scoped SA.

3. Least-privilege on the impersonated GSA + actAs restriction. WIF is only as safe as the SA it fronts. A deployer SA with roles/owner turns any successful exchange into project takeover, and if it holds iam.serviceAccounts.actAs on higher SAs it enables impersonation chaining (threat 13). Grant the narrowest roles (roles/storage.objectAdmin on one bucket, roles/run.developer on one service), and constrain actAs with IAM Conditions or org policy. Use IAM Recommender to strip unused permissions. Trade-off: tight scoping causes deploy failures when pipelines grow; solve with additive, reviewed role grants rather than a blanket Editor. Limit: least privilege reduces blast radius but doesn't stop the authentication — pair it with controls 1 and 2.

4. GKE Workload Identity + metadata concealment. Enforce --workload-metadata=GKE_METADATA on every node pool so pods can only obtain the GSA bound to their KSA, and use NetworkPolicy/host firewall to block direct pod egress to 169.254.169.254. This eliminates the "every pod inherits the node SA" lateral-movement path and defeats SSRF-to-metadata (CVE-2020-8555 class). Implement one GSA per KSA per namespace and bind serviceAccount:PROJECT.svc.id.goog[ns/ksa] narrowly. Trade-off: legacy workloads that expected the node SA break and must be migrated to bound KSAs; the metadata server adds a small per-node component to operate. Limit: it protects Google-API access, not in-cluster secrets — combine with RBAC and Pod Security Standards.

5. Enable Data Access audit logs and disable SA keys org-wide. You cannot detect WIF abuse if the exchanges are invisible: turn on Data Access audit logs for sts.googleapis.com and iamcredentials.googleapis.com so every ExchangeToken and GenerateAccessToken is captured with its principalSubject and source IP. Simultaneously apply constraints/iam.disableServiceAccountKeyCreation to force teams onto WIF and eliminate the static-key attack surface entirely. Trade-off: Data Access logs add volume/cost (scope them to the relevant services), and the key-creation ban breaks any tool still expecting key.json until migrated. Limit: logging is detective, not preventive — it shortens dwell time but must be wired to alerting to matter.

6. Detection & Response

What to log / telemetry sources

  • Cloud Audit Logs — Data Access, service sts.googleapis.com, method google.identity.sts.v1.SecurityTokenService.ExchangeToken. Key fields: protoPayload.authenticationInfo.principalSubject (e.g. principal://.../subject/repo:acme/...), protoPayload.requestMetadata.callerIp, protoPayload.requestMetadata.callerSuppliedUserAgent, and the mapped attributes.
  • Cloud Audit Logs, service iamcredentials.googleapis.com, method GenerateAccessToken / GenerateIdToken — shows which GSA a federated principal impersonated.
  • Admin Activity logs for SetIamPolicy on service accounts and workloadIdentityPools/providers create/update — detect a condition being loosened or a /* binding appearing.
  • GKE: gke-metadata-server and node logs; Kubernetes audit logs for KSA usage.
  • Security Command Center findings for public/over-broad IAM and WIF misconfigurations.

Ensure Data Access logs are enabled (they are off by default):

# gcloud projects get-iam-policy / setIamPolicy auditConfigs snippet
auditConfigs:
  - service: sts.googleapis.com
    auditLogConfigs:
      - logType: DATA_READ
      - logType: DATA_WRITE
  - service: iamcredentials.googleapis.com
    auditLogConfigs:
      - logType: DATA_READ

Sample detections

Detection 1 — token exchange with an unexpected repository attribute (Cloud Logging / Log Analytics SQL). Flags any STS exchange whose mapped repository is not in an allowlist, or from an unusual IP:

SELECT
  timestamp,
  proto_payload.audit_log.authentication_info.principal_subject AS principal,
  proto_payload.audit_log.request_metadata.caller_ip AS src_ip,
  proto_payload.audit_log.request_metadata.caller_supplied_user_agent AS ua
FROM `acme-prod.global._AllLogs`
WHERE proto_payload.audit_log.service_name = 'sts.googleapis.com'
  AND proto_payload.audit_log.method_name LIKE '%ExchangeToken%'
  AND NOT REGEXP_CONTAINS(
        proto_payload.audit_log.authentication_info.principal_subject,
        r'subject/repo:acme/(payments-service|billing-service):')
ORDER BY timestamp DESC

Detection 2 — Sigma rule: WIF provider or binding loosened. Alerts when a provider is created/updated without a condition or an SA is bound to a wildcard principalSet:

title: GCP WIF Trust Relationship Weakened
id: 8f2a1c74-wif-loosened
status: experimental
logsource:
  product: gcp
  service: audit
detection:
  update_provider:
    protoPayload.methodName|contains:
      - 'workloadIdentityPools.providers.create'
      - 'workloadIdentityPools.providers.update'
      - 'workloadIdentityPools.providers.patch'
  wildcard_binding:
    protoPayload.methodName|endswith: '.setIamPolicy'
    protoPayload.request.policy.bindings|contains:
      - 'principalSet://iam.googleapis.com'
      - '/attribute.repository_owner/'   # owner-wide, no repo pin
  condition: update_provider or wildcard_binding
fields:
  - protoPayload.authenticationInfo.principalEmail
  - protoPayload.resourceName
level: high

(Operationally, pair the Sigma update_provider match with a follow-up gcloud iam workload-identity-pools providers describe check that the resulting attributeCondition is non-empty and references repository.)

Response playbook

  1. Contain — cut the trust. Disable the offending provider immediately: gcloud iam workload-identity-pools providers update-oidc PROVIDER_ID --workload-identity-pool=POOL_ID --location=global --disabled. If a specific SA is being impersonated maliciously, remove the workloadIdentityUser binding for the abused principalSet.
  2. Contain — revoke live tokens. Federated/SA tokens are short-lived (≤1h) but active ones persist; disable the impersonated GSA (gcloud iam service-accounts disable SA_EMAIL) to stop in-flight token use, accepting the outage.
  3. Scope the blast radius. From the principalSubject and GenerateAccessToken logs, enumerate which GSAs were assumed and what APIs those tokens touched (Data Access logs on the target services). Identify created/modified resources.
  4. Eradicate the entry. Determine how the trust was abused: condition-less provider, wildcard binding, forgeable subject, or a compromised legitimate repo (poisoned workflow / PPE). Fix the root — add/tighten the attribute condition, scope the principalSet, correct the subject mapping.
  5. Rotate downstream. Rotate any secrets, keys, or data the impersonated SA could reach; re-issue anything the attacker may have exfiltrated.
  6. Recover. Re-enable the provider/SA only after the condition and binding are corrected and reviewed. Confirm the fix with a negative test (a token from a non-allowlisted repo must be rejected).
  7. Harden & watch. Apply disableServiceAccountKeyCreation, add the SCC/Sigma detections, and add a config-scanning gate in Terraform CI that fails any provider lacking a repository-scoped condition or any /* binding.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What problem does Workload Identity Federation solve, and what does it replace?
It removes long-lived Google service account keys (key.json) — the most-leaked, rarely-rotated cloud credential — from external and CI workloads. Instead of storing a static Google secret, the workload presents an OIDC/SAML/AWS identity it already holds, and Google's STS exchanges it for a short-lived (≤1h) access token gated by IAM. This shrinks the attack surface (nothing to leak), makes credentials ephemeral, and centralizes the trust decision in provider conditions and IAM bindings.
Signal: does the candidate frame it as eliminating static keys, not just "a login method"?

Q2. Walk through the GitHub Actions → GCP token exchange end to end.
The job requests an OIDC ID token from token.actions.githubusercontent.com with the provider's resource name as the audience (id-token: write grants this). It POSTs that JWT to sts.googleapis.com/v1/token with grant_type=token-exchange. STS verifies the signature (issuer JWKS), issuer, and audience, applies the attribute mapping, and evaluates the attribute-condition CEL; on pass it returns a federated access token. Optionally the runner calls iamcredentials.generateAccessToken to impersonate a GSA (needs roles/iam.workloadIdentityUser), then calls Google APIs with that short-lived token.
Signal: mentions audience, condition, and the impersonation step — not a hand-wave.

Q3. What is the difference between an attribute mapping and an attribute condition?
The mapping copies claims from the external token into Google attributes (google.subject=assertion.sub, attribute.repository=assertion.repository) so they can be referenced in IAM and conditions. The condition is a CEL boolean that must evaluate true for the token to be accepted at all (assertion.repository == 'acme/payments-service'). Mapping decides what you can reference; the condition decides whether to trust the token. The condition is the security gate.
Signal: recognizes the condition — not the mapping — as the boundary.

Q4. How does GKE Workload Identity differ from WIF, and how is a KSA bound to a GSA?
GKE Workload Identity is for in-cluster pods rather than external identities. The cluster gets a workload pool PROJECT.svc.id.goog; you annotate a KSA with iam.gke.io/gcp-service-account and bind the GSA's roles/iam.workloadIdentityUser to the member serviceAccount:PROJECT.svc.id.goog[namespace/ksa]. The GKE metadata server then issues per-pod GSA tokens, so no key is mounted and unbound pods get no Google access.
Signal: knows the PROJECT.svc.id.goog[ns/ksa] member format and the metadata server's role.

Q5. Why is a service account key more dangerous than a WIF-issued token?
A key is a long-lived bearer secret: it works from anywhere, never expires on its own, and if it leaks into Git, a container layer, or CI logs it grants full SA access until someone notices and rotates it. A WIF token is short-lived (≤1h), scoped, and bound to a verified identity + condition, so a leak is time-boxed and the trust can be revoked centrally by disabling the provider.
Signal: articulates lifetime, portability, and revocation differences.

Tier 2 — System design / trade-offs

Q6. Design keyless CI/CD auth for 500 repos deploying to 30 GCP projects. How do you avoid trust sprawl?
Use one pool per environment (prod/staging) rather than per repo, with a single GitHub provider carrying an org-pinned condition (repository_owner == 'acme') as a coarse gate. Per-repo trust lives in the IAM layer: template principalSet://.../attribute.repository/<repo> bindings on each project's deployer SA via Terraform, and require ref/environment pinning for prod. Enforce disableServiceAccountKeyCreation org-wide and constraints/iam.workloadIdentityPoolProviders to allowlist issuers. Continuously scan for /* bindings and condition-less providers. This keeps the provider count small while making each repo's reach explicit, reviewable, and least-privilege. The trade-off is binding volume — solved with IaC and policy-as-code gates.
Signal: separates coarse provider gate from fine-grained IAM; thinks about governance at scale.

Q7. GitHub is a shared OIDC issuer. What are the security implications and how do you contain them?
Every repository's token is signed by the same GitHub keys, so the issuer/signature check proves "some GitHub repo," not "my repo." Without an attribute condition the provider trusts all of GitHub. Contain it by pinning repository_owner + repository (+ ref/environment) in the condition and scoping the IAM principalSet to the exact repo. Also keep the audience equal to the provider resource name to prevent cross-purpose token replay, and set id-token: write only on the specific job. Defense in depth means both gates must fail before an attacker wins.
Signal: understands shared-issuer risk and the two-gate containment.

Q8. Compare WIF impersonating a GSA vs. granting IAM directly to the federated principal. When would you choose each?
Direct grants (principalSet://... on a resource/role) remove the impersonation hop, so there's no intermediary SA to over-permission and one fewer credential exchange — cleaner blast radius and simpler audit. Impersonation is still needed when a tool or library only speaks "service account," when you want a stable identity decoupled from the changing external subject, or when you must chain to permissions only a GSA holds. Prefer direct grants for new designs; use impersonation for compatibility. Either way, keep the target least-privileged.
Signal: knows direct federation exists and reasons about when the SA hop is justified.

Q9. How do you prevent a pull request from a fork or a feature branch from deploying to production?
Don't pin only the repository — pin the trigger context. Require assertion.ref == 'refs/heads/main' or, better, assertion.environment == 'production' in the attribute condition, and gate the GitHub Environment with required reviewers so prod is unreachable from arbitrary branches/PRs. Set permissions: id-token: write only on the deploy job, and avoid pull_request_target in privileged workflows (PPE risk). Fork PRs already receive restricted tokens, but branch/environment pinning is what stops an insider or a poisoned feature branch.
Signal: cites ref/environment pinning and PPE, not just "restrict the repo."

Q10. Your team wants a 12-hour token lifetime for a long batch job. What are the trade-offs and alternatives?
Longer lifetime widens the window during which a stolen token is usable and increases the value of intercepting CI logs. generateAccessToken caps at 1h anyway, so a 12h job should re-mint tokens periodically from the still-valid federated identity (the auth library handles refresh) rather than hold one long token. If truly long-lived credentials are unavoidable, minimize scope with a Credential Access Boundary (downscoped token) so even a stolen token touches only the specific bucket/prefix. Prefer refresh + downscoping over a single long token.
Signal: knows the 1h cap, refresh pattern, and downscoping.

Tier 3 — Expert / deep-dive

Q11. Explain the audience-confusion attack against a WIF provider and how the default configuration prevents it.
STS checks the token's aud against the provider's allowed audiences. By default that audience is the provider's full resource name (//iam.googleapis.com/projects/.../providers/PROVIDER_ID), which is unique — so a token minted for provider A is useless against provider B or any non-Google system. Confusion arises when an operator widens --allowed-audiences to a generic value (e.g. https://github.com/acme) that other systems also accept; then a token issued for a different consumer, but carrying legitimate repository claims, becomes replayable against WIF. If the IAM binding is only owner-scoped, a low-trust repo's cross-purpose token now impersonates a high-trust SA. The defense is to leave the audience at the default and pin the exact repository, so a token is valid for exactly one provider and one trust.
Signal: connects aud semantics to replay and to why the default resource-name audience is safe.

Q12. Storm-0558 forged tokens after a signing key was stolen. What does that teach about WIF's trust model, and what residual risk exists?
It shows federation collapses if the issuer's signing key is compromised or if aud/iss validation is sloppy — attackers forged tokens that validated because of a scope/audience gap. For WIF the residual risks are: (a) a compromised external IdP (GitHub, your OIDC provider) could mint tokens with any claims, so your condition can't distinguish forged from real; (b) a custom OIDC issuer with an unpinned or attacker-reachable JWKS lets forged tokens in. Mitigate by trusting only well-run issuers, pinning --jwks-json for custom providers, keeping conditions tight so even a forged token needs specific claims, and — crucially — enabling audit logs so anomalous exchanges are detectable even when prevention fails.
Signal: understands that WIF inherits the IdP's key custody risk and reasons about detection as backstop.

Q13. Contrast GCP WIF, AWS sts:AssumeRoleWithWebIdentity, and Azure Workload Identity Federation at the mechanism level.
All three exchange an external OIDC token for cloud credentials, but the trust surface differs. AWS ties trust to an IAM role's trust policy (a JSON Condition on token.actions.githubusercontent.com:sub/aud) and returns temporary STS creds for that role. GCP separates concerns into a provider (issuer + attribute condition CEL) and an IAM binding (principalSet → workloadIdentityUser on a GSA), giving two gates. Azure federates an external identity onto an App Registration / User-Assigned Managed Identity via a federated credential matching issuer + subject + audience, then issues an AAD token. Practically: AWS's condition is StringLike on sub (wildcards are the classic footgun); GCP's is a CEL expression (richer but easy to widen); Azure matches exact subject/issuer/audience tuples. The universal failure is not pinning the repo/branch subject.
Signal: names the exact primitives per cloud and the shared sub-pinning failure.

Q14. A poisoned CI dependency runs in a job that has WIF configured. Trace the blast radius and the layered controls that cap it.
On import/install the malicious package (dependency-confusion / typosquat, see ai_security/AI-Supply-Chain/Poisoned-Dependencies-And-Finetunes/README.md) runs with the job's identity. If the job holds id-token: write, the attacker can mint the OIDC token and complete the exchange, obtaining whatever the deployer SA can do — so blast radius = the deployer SA's permissions for up to 1h. Layered caps: least-privilege on the deployer SA (control 5) bounds what the token reaches; scoped principalSet + condition ensure only the intended repo/branch even authenticates (so the poison must run in your trusted pipeline, e.g. via PPE, not any repo); id-token: write only on the deploy job denies the token to build/test jobs where untrusted deps run; and Data Access logs on STS/target services give detection. The strongest structural fix is separating untrusted build steps (no id-token) from a minimal, reviewed deploy step.
Signal: connects supply-chain execution to identity abuse and enumerates the cap at each layer.

Q15. How would you detect and respond to abuse of a condition-less WIF provider that's already in production?
Detect: query STS Data Access logs for ExchangeToken events whose principalSubject repository attribute is outside the allowlist, and alert on GenerateAccessToken for sensitive GSAs from unusual IPs/geos or unusual user agents; add a config-scan (SCC or a nightly gcloud ... providers describe) that fails any provider with an empty attributeCondition. Respond: disable the provider (--disabled) to cut trust immediately, disable the impersonated GSA to kill in-flight tokens, enumerate assumed SAs and touched APIs from logs, rotate anything reachable, then remediate by adding a repository/ref-scoped condition and a scoped principalSet, and re-enable only after a negative test proves non-allowlisted tokens are rejected. Finally, prevent recurrence with a Terraform policy gate that rejects condition-less providers and /* bindings.
Signal: gives concrete log fields, an immediate containment lever, and a preventive control — full IR loop.

9. Key Takeaways

  • WIF replaces exfiltratable static SA keys with short-lived federated tokens — but relocates the entire trust decision into the provider condition and the IAM binding, both of which must be tight.
  • Two gates, both mandatory: the provider decides which tokens are accepted (issuer + audience + CEL condition); IAM decides what an accepted identity can reach (principalSet scope + GSA permissions).
  • On a shared issuer like GitHub, a condition-less provider trusts every repository on the internet. Always pin repository_owner + repository + ref/environment.
  • Keep the audience at the default (provider resource name) to prevent cross-purpose token replay; map google.subject from an immutable claim (assertion.sub), never a mutable one like actor.
  • Scope the principalSet to the exact repo (attribute.repository/<owner>/<repo>), never /* or owner-wide; and keep the impersonated GSA least-privileged so a successful exchange isn't project takeover.
  • GKE Workload Identity binds KSA→GSA and, with GKE_METADATA enforced, kills the "every pod inherits the node SA" lateral-movement path — a major hardening win over the metadata-server default.
  • Enable Data Access audit logs for sts.googleapis.com and iamcredentials.googleapis.com — exchanges are invisible otherwise — and alert on unexpected repository attributes and wildcard binding changes.
  • Prefer direct IAM grants to the federated principal over the SA impersonation hop when possible; it removes an over-permissionable intermediary.
  • Force adoption org-wide with disableServiceAccountKeyCreation, and gate WIF config in Terraform CI to reject condition-less providers and /* bindings.
  • WIF inherits the external IdP's key-custody and validation risk (Storm-0558 lesson): trust only well-run issuers, pin JWKS for custom OIDC, and treat detection as the backstop when prevention fails.
