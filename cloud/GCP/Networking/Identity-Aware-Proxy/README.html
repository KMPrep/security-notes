GCP Identity-Aware Proxy (IAP) & BeyondCorp Enterprise — The Zero-Trust Access Layer

Category: Cloud Networking / Zero-Trust Network Access (ZTNA)
GCP Products: Identity-Aware Proxy (IAP), BeyondCorp Enterprise, Access Context Manager, IAP TCP forwarding, Chrome Enterprise Premium
Related notes: BeyondCorp / ZTNA · Tunneling · SSH · TLS
Standards anchors: NIST SP 800-207 (Zero Trust Architecture), BeyondCorp USENIX papers (2014–2018), RFC 7519 (JWT), RFC 7515 (JWS), RFC 6749 (OAuth 2.0)

1. Overview

Identity-Aware Proxy (IAP) is Google Cloud's front-door authorization layer: a globally distributed reverse proxy, running on the Google Front End (GFE) fabric, that intercepts requests to your applications and administrative endpoints and enforces per-request identity and context checks before the request ever reaches your backend. Instead of "you are on the corporate network, therefore you are trusted," IAP asks "who is this principal, what device and context are they coming from, and does IAM policy allow this principal to reach this resource, right now?" It is the productized, customer-facing form of the same access-proxy pattern Google built internally as BeyondCorp, and it is the enforcement point for BeyondCorp Enterprise (rebranded under Chrome Enterprise Premium).

IAP matters at scale because it collapses two historically separate problems — network reachability and application authorization — into one identity-centric control plane. A load balancer with IAP enabled will 401/redirect an unauthenticated request and 403 an unauthorized one at the edge, so your App Engine service, GKE Ingress, Compute Engine backend, or even an on-prem app (reached via a hybrid connector) never sees traffic from principals who lack roles/iap.httpsResourceAccessor. IAP TCP forwarding extends the same model to raw TCP — SSH and RDP to VMs with no public IP, no bastion host, and no VPN — by tunneling the connection through an IAM-gated Google-operated proxy. The load-bearing subtlety, and the part interviewers probe, is the signed IAP JWT: IAP asserts the verified identity in a cryptographically signed header, and a backend that does not verify that JWT (and pin its audience) can be bypassed by anyone who can route directly to it.

TL;DR
- IAP is a reverse proxy on the GFE that enforces IAM-based, per-request authZ (roles/iap.httpsResourceAccessor) before traffic hits the backend — access is a function of identity+context, not IP/network location. This is BeyondCorp productized.
- Context-Aware Access = IAP + Access Context Manager access levels (device posture, IP/CIDR, geo, encryption/screen-lock state) layered on top of the IAM allow decision. Both must pass.
- IAP TCP forwarding gives you SSH/RDP to private VMs with no bastion/VPN — the tunnel terminates at Google's proxy, gated by roles/iap.tunnelResourceAccessor.
- IAP injects a signed JWT (X-Goog-IAP-JWT-Assertion) with the verified identity. Backends must verify the signature and pin the aud, or a request that reaches the backend directly (SSRF, misconfigured LB, VPC exposure) trivially bypasses IAP. Defense-in-depth requires backends unreachable except through IAP.
- Trust model vs VPN: a VPN grants broad network reachability after one perimeter auth; IAP grants narrow, per-app, per-request, context-evaluated access with no implicit lateral movement.

2. How It Works

2.1 Core mechanics

IAP sits in the request path of a Google Cloud HTTP(S) External/Internal Load Balancer (for web apps) or acts as a TCP tunnel broker (for TCP forwarding). For web traffic, the flow is: client → GFE/Load Balancer → IAP authorization → backend service. IAP is enabled per backend service (or per App Engine service). When a request arrives, IAP checks for a valid session; if absent, it runs the user through Google's OAuth 2.0 / OIDC sign-in (or validates a service-account/OIDC token for programmatic clients), establishes an encrypted session cookie (GCP_IAAP_AUTH_TOKEN), then evaluates authorization.

Authorization is a two-gate decision. Gate 1 is IAM: the authenticated principal must hold roles/iap.httpsResourceAccessor (or a custom role with iap.webServiceVersions.accessViaIAP) on the IAP-protected resource. Gate 2 is Context-Aware Access: if the IAM binding carries an access-level condition (an IAM Condition referencing an Access Context Manager access level), IAP evaluates that access level — device posture, source IP range, geography, corp-managed status — against signals gathered at request time. Both gates must return allow. Only then does IAP forward the request to the backend, stamping it with a signed JWT assertion of the verified identity so the backend can (and must) confirm the request truly came through IAP.

The control plane is IAM policy + ACM access-level definitions (declarative, versioned, auditable via Cloud Audit Logs). The data plane is the GFE proxy fleet enforcing those policies on every request. Crucially, IAP is stateless with respect to your backend's trust: it does not open a network hole for you. If your backend is also reachable by a path that skips IAP, IAP has enforced nothing on that path — hence JWT verification is not optional hardening, it is the closing half of the control.

2.2 Concrete configuration

Enable IAP on a Compute Engine / GKE backend behind an HTTPS Load Balancer. IAP needs an OAuth brand (consent screen) and client, then it is toggled on the backend service.

# 1. Create the OAuth brand (consent screen) for the project — internal org apps
gcloud iap oauth-brands create \
  --application_title="Internal Admin Console" \
  --support_email="secops@example.com"

# 2. Create an OAuth client IAP will use to sign users in
gcloud iap oauth-clients create \
  projects/PROJECT_ID/brands/BRAND_ID \
  --display_name="iap-admin-console"
# -> returns CLIENT_ID and CLIENT_SECRET (used as the JWT audience later)

# 3. Turn IAP ON for the backend service fronting the app
gcloud iap web enable \
  --resource-type=backend-services \
  --service=admin-console-backend \
  --oauth2-client-id=CLIENT_ID \
  --oauth2-client-secret=CLIENT_SECRET

# 4. Grant a group web access THROUGH IAP (Gate 1: IAM)
gcloud iap web add-iam-policy-binding \
  --resource-type=backend-services \
  --service=admin-console-backend \
  --member="group:sre-oncall@example.com" \
  --role="roles/iap.httpsResourceAccessor"

Add Context-Aware Access (Gate 2) — require a corporate-managed device on a US IP range. First define the access level in Access Context Manager, then bind the IAM grant with a Condition referencing it.

# Define an access level: corp-managed device AND US geo AND corp CIDR
cat > corp_us.yaml <<'EOF'
- devicePolicy:
    requireCorpOwned: true
    requireScreenlock: true
    osConstraints:
    - osType: DESKTOP_MAC
      minimumVersion: "14.0.0"
    - osType: DESKTOP_CHROME_OS
  ipSubnetworks:
  - 203.0.113.0/24
  regions:
  - US
EOF

gcloud access-context-manager levels create corp_managed_us \
  --title="Corp Managed US" \
  --basic-level-spec=corp_us.yaml \
  --policy=ACCESS_POLICY_ID

# Bind IAM access ONLY when the request meets the access level (IAM Condition)
gcloud iap web add-iam-policy-binding \
  --resource-type=backend-services \
  --service=admin-console-backend \
  --member="group:sre-oncall@example.com" \
  --role="roles/iap.httpsResourceAccessor" \
  --condition='expression=request.auth.access_levels.contains("accessPolicies/ACCESS_POLICY_ID/accessLevels/corp_managed_us"),title=corp-us-only'

IAP TCP forwarding — SSH to a private VM with no bastion and no external IP. The gcloud compute ssh --tunnel-through-iap flag opens a tunnel from your localhost through the IAP proxy to the VM's port 22. Grant the tunnel role and a firewall rule allowing IAP's source range.

# Allow IAP's proxy range to reach SSH/RDP on private VMs (35.235.240.0/20 is IAP)
gcloud compute firewall-rules create allow-iap-ingress \
  --network=prod-vpc \
  --direction=INGRESS \
  --action=ALLOW \
  --rules=tcp:22,tcp:3389 \
  --source-ranges=35.235.240.0/20

# Grant an engineer the tunnel accessor role, scoped to one VM (resource-level IAM)
gcloud compute instances add-iam-policy-binding jump-free-vm \
  --zone=us-central1-a \
  --member="user:alice@example.com" \
  --role="roles/iap.tunnelResourceAccessor"

# SSH straight to the private VM — no public IP, no bastion
gcloud compute ssh jump-free-vm \
  --zone=us-central1-a \
  --tunnel-through-iap

# Or open a raw local tunnel (e.g. map local 3389 -> VM RDP for a Windows box)
gcloud compute start-iap-tunnel windows-vm 3389 \
  --local-host-port=localhost:3389 \
  --zone=us-central1-a

Backend JWT verification (the load-bearing control). Every request IAP forwards carries X-Goog-IAP-JWT-Assertion, an ES256-signed JWT. The backend must verify the signature against Google's IAP public keys, and pin the audience to its own resource identifier so a token minted for a different IAP-protected app cannot be replayed. The audience format is /projects/PROJECT_NUMBER/global/backendServices/SERVICE_ID (Compute/GKE) or /projects/PROJECT_NUMBER/apps/PROJECT_ID (App Engine).

from google.auth.transport import requests as g_requests
from google.oauth2 import id_token

# Pin this to YOUR backend service — mismatched aud => reject
EXPECTED_AUDIENCE = "/projects/1234567890/global/backendServices/9876543210"

def verify_iap_jwt(request):
    assertion = request.headers.get("X-Goog-IAP-JWT-Assertion")
    if not assertion:
        # No IAP header => request did NOT come through IAP => refuse
        abort(403, "missing IAP assertion")
    try:
        # Verifies ES256 signature against IAP keys AND checks aud/iss/exp
        payload = id_token.verify_token(
            assertion,
            g_requests.Request(),
            audience=EXPECTED_AUDIENCE,
            certs_url="https://www.gstatic.com/iap/verify/public_key",
        )
    except Exception as e:
        abort(403, f"invalid IAP assertion: {e}")
    # iss must be Google, sub is the stable user id, email is the identity
    assert payload["iss"] == "https://cloud.google.com/iap"
    return payload["email"], payload["sub"]

The commentary that matters: do not trust the plaintext X-Goog-Authenticated-User-Email / -Id headers on their own — those are legacy signed headers that any upstream (or an attacker who reaches the backend directly) can forge. The JWT is the only assertion with a verifiable signature and an audience binding, and pinning aud is what stops a valid token from app A being replayed against app B.

Name / ID Year Root cause Lesson

Okta support-system compromise 2023 Session/credential theft against a trusted identity broker IAP depends on the IdP; compromise of the identity plane bypasses posture gates — pair with device binding
CVE-2020-8555 (GKE SSRF) 2020 SSRF in Kubernetes let attacker reach internal endpoints Off-path reachability to IAP-protected backends is the recurring bypass vector

Case study 1 — Aurora and why IAP exists. In 2010, attackers phished a Google engineer, gained a foothold, and used the implicitly trusted corporate network — reachable via VPN — to move laterally toward source-control systems. The strategic response was not a bigger firewall but abolishing the trusted network: BeyondCorp moved every internal app behind an access proxy (the internal ancestor of IAP) that evaluates user identity and device state on every request. Had that model existed, the phished credential alone would not have granted network reachability to sensitive systems; each app would have demanded a managed, healthy device and an explicit per-app grant.

Case study 2 — The audience-confusion bypass. A common early IAP integration bug: a developer copies a JWT-verification snippet, validates the ES256 signature against Google's IAP keys, reads the email claim, and ships it — but omits the audience= argument. Because all IAP apps in the ecosystem are signed by the same Google keys, a token legitimately issued for a low-value app the attacker can access (its aud is app A) still has a valid signature, so it is accepted by a high-value app B that only checks the signature. The attacker replays their app-A token to app B and is authenticated as themselves — or, worse, the app B code trusts the asserted email for authZ. The fix is one line: pin aud to app B's backend-service identifier so the token's aud=A fails the check.

Case study 3 — SSRF as the universal IAP-bypass primitive. Capital One (2019) and the GKE SSRF class (CVE-2020-8555) share a shape: an attacker who can make the backend itself originate a request reaches endpoints the front-door control never sees. Applied to IAP: if the protected backend has a public IP, a permissive internal-VPC path, or is reachable via an SSRF gadget in a neighboring service, the IAP gate is simply not in the path. The mitigation is architectural — the backend must be reachable only through IAP (private backend, firewall restricted to the LB/IAP ranges) and must verify the IAP JWT, so even an off-path request is rejected for lacking a valid assertion.

Top 5 in depth

1. Make the backend reachable only through IAP (and verify the JWT). This is the single most important control because IAP enforces nothing on a path it does not sit in. Technically: give the backend no external IP, put it in a private subnet, and set the VPC firewall to allow ingress only from the load balancer's health-check and proxy ranges (and 35.235.240.0/20 for TCP forwarding). Then, in code, verify X-Goog-IAP-JWT-Assertion with the audience pinned. The two together are complementary: firewalling stops most off-path traffic, JWT verification catches the residual (SSRF, internal peers, future misconfigs). Trade-off: this makes local development and third-party integrations harder, pushing teams toward service-account OIDC tokens minted with the backend's audience — which is the correct pattern but adds friction.

2. Pin the JWT audience. IAP signs every assertion with Google's shared IAP keys, so signature validity proves only "some IAP issued this," not "this app issued this." Pinning aud to /projects/PROJECT_NUMBER/global/backendServices/SERVICE_ID is what makes the token app-specific. Implementation is the audience= argument in id_token.verify_token (or equivalent JOSE library check). The trade-off is operational: the audience string embeds the project number and backend-service ID, so it must be updated if the backend is recreated — bake it into config/secrets, not a hardcoded constant that silently drifts.

3. Least-privilege access levels with real device posture. IP allowlists alone are weak — a VPN or a compromised host on an allowed IP passes. A strong access level combines requireCorpOwned: true, requireScreenlock: true, an OS-version floor, a corp CIDR, and a geo constraint, all sourced from Endpoint Verification. Bind it via IAM Conditions so the grant is conditional on the access level. The trade-off is signal fidelity: posture is only as trustworthy as the agent reporting it, so treat device posture as a strong-but-not-infallible signal and pair it with phishing-resistant MFA rather than relying on either alone.

4. Least-privilege IAM for both web and tunnel roles. Grant httpsResourceAccessor to groups, never to allUsers/allAuthenticatedUsers, and scope tunnelResourceAccessor to individual instances (gcloud compute instances add-iam-policy-binding) rather than project-wide, so an engineer who needs SSH to one VM cannot tunnel to all of them. Combine with IAM Recommender to prune unused grants. Trade-off: fine-grained bindings multiply policy objects and management overhead; use groups + IaC (Terraform) to keep it tractable.

5. Tight firewall for TCP forwarding. IAP TCP forwarding only works if the VPC firewall permits IAP's source range 35.235.240.0/20 to the target ports — but teams often over-open to 0.0.0.0/0, which removes the IAP gate by allowing direct SSH from anywhere. The correct rule allows tcp:22/3389 only from 35.235.240.0/20, and the VM has no external IP, so the only way in is through the IAM-gated tunnel. Trade-off: emergency/console access paths must be planned (e.g. serial console, Access Approval) since you have intentionally removed the direct path.

6. Detection & Response

What to log / telemetry sources

  • Cloud Audit Logs — Data Access for IAP: enable DATA_READ on IAP; captures each authorized/denied access with principal, resource, access levels evaluated, and caller IP. Log name: cloudaudit.googleapis.com/data_access, service iap.googleapis.com.
  • IAP request logs / Load Balancer logs: HTTP status (401/403 patterns), user agent, jwt/identity fields, latency, backend served.
  • Access Context Manager evaluation results: which access level passed/failed (surfaced in the IAP audit log metadata).
  • Cloud Audit Logs — Admin Activity for SetIamPolicy on IAP resources and firewall changes (detect grant of iap.*Accessor, or firewall opening 22/3389).
  • IAP TCP tunnel logs: AuthorizeUser / tunnel establishment events with source principal, target instance, and port.
  • VPC Flow Logs: corroborate that backend ingress originates only from LB/IAP ranges — a flow to the backend from an unexpected source is a bypass signal.

Sample detections

KQL/Log Analytics-style — IAP TCP tunnel to sensitive VM from unusual principal:

GcpAuditLogs
| where ServiceName == "iap.googleapis.com"
| where MethodName has "AuthorizeUser" or MethodName has "tunnel"
| extend principal = tostring(AuthenticationInfo.principalEmail)
| extend target = tostring(Request.resource)
| where target has "instances/prod-" 
| summarize tunnels = count(), vms = make_set(target) by principal, bin(TimeGenerated, 1h)
| where tunnels > 5 or array_length(vms) > 3   // fan-out = possible lateral movement

Sigma — access-level failures spiking (posture/geo probing or misconfig):

SQL (BigQuery over exported audit logs) — IAM grant of IAP accessor to broad principal:

SELECT
  timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS actor,
  JSON_EXTRACT_SCALAR(binding, '$.role') AS role,
  member
FROM `project.dataset.cloudaudit_googleapis_com_activity`,
  UNNEST(JSON_EXTRACT_ARRAY(protopayload_auditlog.servicedata_v1_iam.policyDelta.bindingDeltas)) AS binding,
  UNNEST(JSON_EXTRACT_ARRAY(binding, '$.members')) AS member
WHERE protopayload_auditlog.methodName = 'SetIamPolicy'
  AND JSON_EXTRACT_SCALAR(binding, '$.role') LIKE '%iap.%Accessor%'
  AND member IN ('allUsers', 'allAuthenticatedUsers')   -- dangerous grants

Response playbook

  1. Detect & triage. Alert fires (broad IAM grant, direct-to-backend flow, or access-level denial spike). Confirm the principal, resource, and whether the access path bypassed IAP (check VPC Flow Logs source vs LB/IAP ranges).
  2. Contain (identity). Revoke the offending IAM binding: gcloud iap web remove-iam-policy-binding ... or remove tunnelResourceAccessor. Suspend the user/SA in Cloud Identity; rotate/disable any leaked SA key.
  3. Contain (network). If a bypass path exists, tighten the firewall to LB/IAP ranges immediately; remove any external IP on the backend. Kill active IAP sessions by rotating the OAuth client secret / forcing re-auth if session hijack is suspected.
  4. Eradicate. Fix the root cause: add/repair JWT verification with pinned audience; correct over-broad access levels; scope tunnel roles to specific instances; remove allAuthenticatedUsers grants.
  5. Recover. Re-enable access via least-privilege groups and conditional bindings; restore service. Validate with a synthetic request that the backend now rejects any request lacking a valid IAP JWT.
  6. Post-incident. Add detections for the specific gap; enforce the invariant in CI/IaC (policy-as-code test that backends have no external IP and firewalls only allow IAP ranges); review Access Approval logs for any break-glass usage during the window.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is IAP and what problem does it solve?
IAP is a reverse proxy on Google's Front End that enforces per-request authentication and authorization before traffic reaches your app or VM. It solves the problem of trusting network location: instead of "inside the VPN = trusted," access is a function of verified identity plus context, evaluated on every request. It is the customer-facing implementation of BeyondCorp. Concretely, it gates web apps (App Engine/Compute/GKE) via roles/iap.httpsResourceAccessor and TCP (SSH/RDP) via roles/iap.tunnelResourceAccessor.
Signal: Do they understand identity-centric vs perimeter-centric access.

Q2. How does IAP TCP forwarding let you SSH to a VM with no public IP or bastion?
gcloud compute ssh --tunnel-through-iap opens a tunnel from your localhost through Google's IAP proxy to the VM's port 22; the tunnel is authorized by IAM (tunnelResourceAccessor), and a firewall rule allows IAP's range 35.235.240.0/20 to reach the port. The VM needs no external IP because the connection originates from Google's proxy inside the fabric. This eliminates the bastion/VPN while keeping every connection identity-gated and audited.
Signal: Understands the tunnel-broker model and the firewall/role prerequisites.

Q3. What is the IAP JWT and why must backends verify it?
IAP injects X-Goog-IAP-JWT-Assertion, an ES256-signed JWT asserting the verified user's identity (email, sub) and audience. Backends must verify the signature against Google's IAP keys and pin the aud to their own backend-service ID; otherwise a request that reaches the backend off-path (SSRF, open firewall) is trusted with no proof it came through IAP, and a token minted for another app could be replayed. The plaintext X-Goog-Authenticated-User-* headers are not sufficient because they are forgeable.
Signal: Grasps that IAP is only enforced on the path it sits in, and that the JWT is the closing control.

Q4. What is Access Context Manager and how does it relate to IAP?
Access Context Manager defines reusable access levels — named predicates over device posture, source IP/CIDR, geography, and encryption/screen-lock state. IAP references these via IAM Conditions, so a grant becomes conditional: the principal must both hold the accessor role (Gate 1) and satisfy the access level (Gate 2). This is how IAP does Context-Aware Access, adding device/context signals to pure identity.
Signal: Knows the two-gate model and where context lives.

Q5. How does IAP differ from a traditional VPN?
A VPN authenticates once at the perimeter and grants broad network reachability, enabling lateral movement; IAP authenticates and authorizes each request to each app individually, with context re-evaluated, and grants no implicit reachability to anything else. VPN trusts the network; IAP trusts nothing by default and verifies per request. The blast radius of a stolen VPN credential is the whole internal network; of a stolen IAP session, one app under its posture constraints.
Signal: Can articulate the zero-trust trust-model difference crisply.

Tier 2 — System design / trade-offs

Q6. Design zero-trust access to 200 internal apps for a remote workforce on GCP.
Put each app behind an HTTPS Load Balancer with IAP enabled per backend service; keep backends private (no external IP), firewall ingress to LB/IAP ranges. Model access with Google Groups mapped to httpsResourceAccessor bindings, and layer Context-Aware Access levels (corp-managed device via Endpoint Verification, geo, CIDR) via IAM Conditions. Enforce JWT verification in every backend with pinned audiences, ideally via a shared middleware/sidecar so 200 teams don't each reimplement it. For admin access to VMs, use IAP TCP forwarding with per-instance tunnel roles. Centralize decisions in audit logs feeding a SIEM. The trade-offs: shared middleware reduces per-team error but becomes a critical dependency; group-based IAM scales but risks group sprawl, so pair with periodic access reviews and IAM Recommender.
Signal: Architecture at scale, reuse of controls, awareness of operational failure modes.

Q7. A team enabled IAP but security testing shows the app is still reachable directly. What happened and how do you fix it?
IAP enforces only in the request path; the backend almost certainly has an external IP or a permissive VPC/firewall path, so requests can hit it without traversing IAP. Fix: remove the external IP, restrict the firewall to LB/IAP ranges, and — critically — add JWT verification with a pinned audience so any off-path request (including SSRF from a peer) is rejected for lacking a valid assertion. Verify by sending a request directly to the backend IP and confirming a 403. This is defense-in-depth: firewalling handles the network path, JWT handles the residual.
Signal: Understands the number-one real-world IAP failure and the layered fix.

Q8. When would you choose IAP TCP forwarding vs a bastion host vs a full VPN for admin access?
IAP TCP forwarding is the default for GCP-native admin access: no bastion to patch, per-VM IAM gating, full audit, works from anywhere. A bastion adds a managed hop you must harden and log yourself, and centralizes risk; it's justified only for non-GCP targets or when you need session recording IAP doesn't natively give. A full VPN grants broad reachability and is the weakest zero-trust posture — appropriate only for legacy systems that can't sit behind IAP. I'd default to IAP TCP forwarding, use a hardened bastion or PAM tool where session recording/keystroke logging is mandated, and treat VPN as legacy to be retired.
Signal: Opinionated, trade-off-aware, understands operational cost of each.

Q9. How do you handle machine-to-machine / programmatic access to an IAP-protected service?
Clients authenticate with an OIDC ID token whose audience is the IAP OAuth client ID; a service account mints this token (ideally via Workload Identity Federation, no long-lived key) and sends it as a Bearer token, which IAP validates like a user session. The service account must hold httpsResourceAccessor. Avoid embedding SA JSON keys; prefer WIF or metadata-server-sourced tokens. The trade-off: WIF adds setup complexity but eliminates the leaked-key threat that plagues M2M integrations.
Signal: Knows programmatic IAP flow and the credential-hygiene angle.

Q10. Your access levels use IP allowlists. Why is that weak, and what would you add?
IP-only context is weak because anyone on an allowed IP — a VPN endpoint, a compromised host, a shared NAT — passes, and IPs are spoofable/borrowable; it re-creates a network-location trust model inside a zero-trust product. Add device posture (corp-owned, screen-lock, OS floor, disk encryption) from Endpoint Verification and pair with phishing-resistant MFA at the IdP, so the decision rests on who + what device + context, not just where. IP becomes one signal among several, not the gate.
Signal: Recognizes that IP allowlists undermine zero trust; knows the stronger signals.

Tier 3 — Expert / deep-dive

Q11. Walk through exactly how the audience-confusion bypass works and the minimal fix.
All IAP assertions are signed by the same Google IAP key set, so verifying only the signature proves "issued by IAP," not "issued for this app." An attacker with legitimate access to a low-value IAP app A obtains a valid JWT with aud = A. If high-value app B verifies the signature but not the audience, it accepts the app-A token, authenticating the attacker (as themselves — dangerous if B trusts the asserted email for authorization) or enabling replay. The minimal fix is to pass audience=<B's backend-service ID> to the verifier so aud=A fails. This generalizes: any shared-issuer JWT system must bind tokens to the specific resource via audience, or signature validity alone is a confused-deputy vector.
Signal: Deep JWT/OAuth security understanding; confused-deputy reasoning.

Q12. How would you enforce, in CI, that no IAP-protected backend can be reached off-path?
Encode the invariant as policy-as-code over Terraform plans (OPA/Conftest or gcloud/Forseti-style checks): assert that any instance/NEG behind an IAP-enabled backend service has no access_config external IP, that firewall rules for its ports allow only LB health-check ranges and 35.235.240.0/20, and that no rule opens 22/3389/backend ports to 0.0.0.0/0. Additionally, add a runtime canary that periodically curls the backend's internal IP directly and alerts unless it gets a 403 (proving JWT verification is live). The combination catches both drift in network exposure and a regression that disables JWT checks — the two ways the bypass reappears.
Signal: Preventive controls, policy-as-code, thinks about drift and regression.

Q13. IAP depends on the IdP. How do you reduce risk if the IdP (or Google sign-in session) is compromised, à la Okta 2023?
Layer controls the IdP compromise doesn't defeat: require hardware-backed, phishing-resistant MFA (FIDO2/passkeys) so stolen passwords/OTPs are useless; enforce device binding via Context-Aware Access so a valid session from an unmanaged device is still denied; shorten IAP session lifetimes and force re-auth on sensitive apps so a stolen session expires fast; and monitor for anomalous access-level pass/fail patterns and impossible-travel in audit logs. The strategy is defense-in-depth: identity, device, context, and time each independently constrain access, so no single plane's compromise grants entry.
Signal: Understands identity-plane risk and compensating device/context controls.

Q14. Compare IAP/BeyondCorp with AWS Verified Access and Entra Private Access at the enforcement-model level.
All three are ZTNA reverse proxies decoupling access from network location. IAP evaluates IAM + Access Context Manager access levels and asserts identity to backends via a signed JWT; its differentiator is deep integration with Google IAM and Chrome/Endpoint Verification posture. AWS Verified Access evaluates Cedar policies against trust data from identity and device-trust providers (CrowdStrike, Jamf) and fronts apps via an ALB, with logging to CloudTrail/OpenSearch. Entra Private Access (Global Secure Access) uses connectors to publish private apps with Conditional Access policies and Continuous Access Evaluation. Key differences: IAP and Verified Access are policy-engine-per-request proxies; Entra leans on Conditional Access + connectors and is strongest in Microsoft-centric estates. The shared invariant across all: the backend must be reachable only through the proxy and should verify the proxy's assertion.
Signal: Cross-cloud fluency; sees the common model and the real differences.

Q15. Design monitoring to detect an insider abusing IAP TCP forwarding for lateral movement.
Baseline each principal's normal tunnel behavior (which VMs, which ports, which hours) from IAP AuthorizeUser audit logs. Alert on fan-out (one principal tunneling to many distinct VMs in a short window), tunnels to sensitive tiers they don't own, off-hours access, or new principal→VM pairs. Correlate with VPC Flow Logs to catch post-tunnel scanning (a private VM suddenly initiating many internal connections after a tunnel opens = pivot). Enforce least privilege so the blast radius is small (per-instance tunnelResourceAccessor), and require Access Approval for the most sensitive VMs so a human review gates each tunnel. The detection (fan-out + flow correlation) plus the prevention (per-instance roles + Access Approval) together bound insider abuse.
Signal: Detection engineering + preventive design for the insider tunnel-abuse case.

9. Key Takeaways

  • IAP is BeyondCorp productized: a GFE reverse proxy that authorizes every request by identity + context, replacing network-location trust — access is granted per app, per request, not per network.
  • Authorization is a two-gate decision: IAM role (httpsResourceAccessor / tunnelResourceAccessor) and an Access Context Manager access level via IAM Condition; both must pass.
  • The signed IAP JWT is the load-bearing control — backends must verify its signature and pin the audience, or any off-path request (SSRF, open firewall, misrouted LB) bypasses IAP entirely. Never trust the plaintext X-Goog-Authenticated-User-* headers alone.
  • Defense-in-depth requires backends reachable only through IAP: no external IP, firewall restricted to LB/IAP ranges (35.235.240.0/20 for TCP forwarding). Firewalling + JWT verification are complementary, not redundant.
  • IAP TCP forwarding eliminates bastions and VPNs for SSH/RDP to private VMs, with per-instance IAM gating and full audit — default to it over bastions where session recording isn't mandated.
  • Context-Aware Access with device posture beats IP allowlists; IP-only levels re-introduce network-location trust and are defeated by VPNs/compromised hosts on allowed IPs.
  • Least privilege matters twice: scope web access to groups (never allAuthenticatedUsers) and tunnel access to specific instances, and prune with IAM Recommender.
  • The recurring real-world failure is off-path reachability (Aurora, Capital One SSRF, GKE SSRF class) — the fix is architectural (only-via-IAP) plus cryptographic (verify + pin aud).
  • IAP inherits IdP risk; pair it with phishing-resistant MFA and device binding so an identity-plane compromise (à la Okta 2023) doesn't grant access from an unmanaged device.
  • Detection rests on Cloud Audit Logs + VPC Flow Logs: watch for broad IAM grants to IAP roles, firewalls opening 22/3389 to the world, tunnel fan-out, and access-level denial spikes.
