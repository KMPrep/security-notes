GCP VPC Service Controls (VPC-SC)

Domain: Cloud Security — GCP Networking / Data-Exfiltration Prevention
Related services: Access Context Manager (ACM), Private Google Access, Cloud Audit Logs, Organization Policy, IAM
Provider docs: VPC Service Controls, Access Context Manager, Ingress/Egress rules
Defensive framing: blast-radius reduction and exfiltration prevention for study — not an offensive playbook.

1. Overview

VPC Service Controls (VPC-SC) draws a logical security perimeter around Google-managed API resources — BigQuery datasets, Cloud Storage buckets, Cloud KMS keys, Pub/Sub topics, Vertex AI, and dozens of other services — so that data cannot flow out of the perimeter or be read into it by a caller outside the boundary, even when the caller holds otherwise-valid IAM credentials. This is a distinctly different control from IAM: IAM answers "is this principal allowed to do this?"; VPC-SC answers "is this request coming from a context we trust, to a resource that is allowed to leave this boundary?" The two are evaluated as an AND — a request must pass both to succeed. That property is why VPC-SC is the canonical GCP answer to the practice question "a service-account key leaked to the public internet — how do you stop the attacker from gsutil cp-ing your entire data lake to their laptop?" With a correctly scoped perimeter, the stolen token is useless from outside the perimeter because the perimeter check fails before the data ever moves. VPC-SC matters at scale because in a large org, credential leakage is a when, not an if — keys land in GitHub, in CI logs, in SSRF-reachable metadata endpoints — and a perimeter converts "full data-lake exfiltration" into "authentication that goes nowhere." GCP is unusual among the hyperscalers in offering this as a first-class, org-wide, service-boundary primitive; AWS and Azure require you to assemble an approximation from several partial controls.

TL;DR
- VPC-SC is a data-exfiltration prevention perimeter around Google API resources (BigQuery, GCS, KMS, Pub/Sub, Vertex AI…), configured through Access Context Manager at the org level.
- It is orthogonal to IAM and evaluated as AND: a request needs IAM allow and to satisfy the perimeter. A stolen token used from outside the perimeter is denied — this is defense in depth against credential compromise (T1078 / T1552).
- Building blocks: service perimeters (a set of projects), restricted services (the protected APIs), access levels (IP/device/identity conditions), ingress/egress rules (fine-grained cross-perimeter allow), and bridges (controlled sharing).
- Dry-run mode lets you observe violations in audit logs before enforcing — always dry-run first, because VPC-SC is famous for breaking legitimate pipelines.
- Limits: it protects supported APIs only, does not stop exfil to a colluding project inside the same perimeter, and has documented bypass classes via non-restricted services — so it is a layer, not a silver bullet.

2. How It Works (go deep)

2.1 Core mechanics

VPC-SC is enforced by the Google API front end — the shared serving layer in front of *.googleapis.com. When a request arrives for a restricted service, before the target service runs its own IAM check, the front end evaluates whether the request satisfies the perimeter. The decision uses the origin of the request (its VPC network, source IP, or the identity/device asserted by Access Context Manager), the target resource (which project/perimeter it belongs to), and the configured ingress/egress rules and access levels.

The control plane lives in Access Context Manager (ACM), an org-scoped service. You create one access policy per organization (the container), and within it you define access levels and service perimeters. A perimeter is fundamentally a list of project numbers plus a list of restricted services plus rules. Because it keys on project number (immutable) rather than project ID (mutable), it survives project renames.

There are two configurations on every perimeter that you must internalize: the enforced config (status) and the dry-run config (spec). Requests are evaluated against both, but only the enforced config blocks; the dry-run config only logs what it would have done. This dual-config model is the single most important operational feature — it is how you roll out a perimeter without a production outage.

Decision logic (mental model): for a call to a restricted service, the request is allowed iff (a) IAM allows it, and (b) the perimeter allows it. The perimeter allows it if the caller and target are in the same perimeter, or an ingress rule matches (for inbound cross-boundary), or the caller is inside and an egress rule matches the outside target, or the caller satisfies an access level bound to the perimeter. If none match → deny, and a VPC-SC violation is written to Cloud Audit Logs with a violationReason.

2.2 Concrete configuration

Everything below assumes you have captured the org-scoped access-policy ID once:

# One access policy per org; capture its numeric ID for --policy on every later call
export ORG_ID=123456789012
export POLICY_ID=$(gcloud access-context-manager policies list \
  --organization="$ORG_ID" --format='value(name)')
echo "Policy: $POLICY_ID"

Create an access level that trusts only the corporate egress IP ranges (a "basic" level; conditions inside a block are AND, blocks are OR):

# corp_access.yaml — trust requests from corp NAT ranges only
- ipSubnetworks:
  - 203.0.113.0/24
  - 198.51.100.0/24

gcloud access-context-manager levels create corp_ip \
  --title="Corp egress IPs" \
  --basic-level-spec=corp_access.yaml \
  --policy="$POLICY_ID"

For stronger assurance, use a custom level with a CEL expression that also requires a corporate, encrypted, screen-locked device (device signals come from Endpoint Verification):

# device_level.yaml — CEL: encrypted managed device AND from a corp IP
expression: >-
  device.encryption_status == DeviceEncryptionStatus.ENCRYPTED &&
  device.is_corp_owned_device == true &&
  origin.ip in ["203.0.113.0/24"]

gcloud access-context-manager levels create corp_device \
  --title="Corp managed device" \
  --custom-level-spec=device_level.yaml \
  --policy="$POLICY_ID"

Create the service perimeter around the production data projects, protecting BigQuery, GCS, and KMS. Note the direct jump to --perimeter- flags and the association of the access level:

gcloud access-context-manager perimeters create prod_data_perimeter \
  --title="Prod data perimeter" \
  --resources=projects/700000000001,projects/700000000002 \
  --restricted-services=bigquery.googleapis.com,storage.googleapis.com,cloudkms.googleapis.com \
  --access-levels=corp_device \
  --policy="$POLICY_ID"

Lock the API surface reachable from inside the VPC (second lock — even an inside caller can only reach these APIs, blocking a compromised VM from pivoting to, say, compute.googleapis.com for lateral movement):

gcloud access-context-manager perimeters update prod_data_perimeter \
  --add-vpc-allowed-services=bigquery.googleapis.com,storage.googleapis.com \
  --enable-vpc-accessible-services \
  --policy="$POLICY_ID"

Ingress rule — allow a specific ETL service account, only from a corp IP, to read BigQuery in the perimeter (this is how you let a trusted external orchestrator in without opening the whole boundary):

# ingress.yaml
- ingressFrom:
    identities:
    - serviceAccount:etl-runner@analytics-shared.iam.gserviceaccount.com
    sources:
    - accessLevel: accessPolicies/POLICY_ID/accessLevels/corp_ip
  ingressTo:
    operations:
    - serviceName: bigquery.googleapis.com
      methodSelectors:
      - method: "*"
    resources:
    - projects/700000000001

Egress rule — allow inside callers to get objects from one specific external analytics project's GCS, nothing else. Egress rules are the exfiltration-shaped direction; keep them narrow, method-scoped, and resource-pinned:

# egress.yaml
- egressFrom:
    identities:
    - serviceAccount:reporter@700000000001.iam.gserviceaccount.com
  egressTo:
    operations:
    - serviceName: storage.googleapis.com
      methodSelectors:
      - method: google.storage.objects.get   # read only, no objects.create/list
    resources:
    - projects/900000000009

gcloud access-context-manager perimeters update prod_data_perimeter \
  --set-ingress-policies=ingress.yaml \
  --set-egress-policies=egress.yaml \
  --policy="$POLICY_ID"

Dry-run first (mandatory in production). Create the enforced config empty/loose, put the real config in the dry-run spec, watch audit logs for what would break, then promote:

# Stage the full config as a dry-run spec on an existing perimeter
gcloud access-context-manager perimeters dry-run update prod_data_perimeter \
  --add-restricted-services=bigquery.googleapis.com,storage.googleapis.com \
  --add-resources=projects/700000000001 \
  --policy="$POLICY_ID"

# ...observe VpcServiceControlAuditMetadata logs with dryRun:true for days...

# Promote dry-run spec to enforced once clean
gcloud access-context-manager perimeters dry-run enforce prod_data_perimeter \
  --policy="$POLICY_ID"

Perimeter bridge — let a shared logging project talk to prod without merging perimeters:

gcloud access-context-manager perimeters create prod_logging_bridge \
  --title="Prod to logging bridge" \
  --perimeter-type=bridge \
  --resources=projects/700000000001,projects/800000000008 \
  --policy="$POLICY_ID"

Client-side prerequisite — for private connectivity, VMs route Google APIs to the restricted VIP so traffic never traverses the public internet and only reaches VPC-SC-supported APIs:

# Route restricted.googleapis.com to the restricted VIP 199.36.153.4/30
gcloud dns managed-zones create restricted-apis \
  --dns-name="googleapis.com." --visibility=private \
  --networks=prod-vpc --description="VPC-SC restricted VIP"
# A record googleapis.com -> 199.36.153.4/30, plus CNAME *.googleapis.com -> restricted.googleapis.com

3. Real-World Incidents & CVEs

VPC-SC is a configuration control, so the relevant "CVEs" are (a) real breaches whose exfiltration phase a perimeter would have blunted, and (b) documented VPC-SC bypass classes from cloud-security research. Both are fair game.

Incident / finding Year Root cause Lesson for VPC-SC
Capital One breach 2019 SSRF → instance metadata creds → S3 exfil (~100M records) AWS had no perimeter; a GCP-style egress boundary would deny exfil to attacker context even with valid creds
Uber breach 2016 Hardcoded AWS keys in a private GitHub repo → S3 Credential leakage is inevitable; a perimeter makes leaked keys unusable from outside
Codecov supply-chain compromise 2021 Modified CI script exfiltrated env-var secrets from customer pipelines Leaked cloud tokens flowed to attacker infra; egress-scoped perimeter limits where inside creds can send data
Toyota / T-Connect data exposure 2022 Access key committed to public GitHub for ~5 years Long-lived leaked keys → perimeter converts the key into a dead end from external origins
Imperva breach 2019 Stolen AWS API key + exposed DB snapshot Exfil via valid credential from unexpected origin; perimeter blocks cross-boundary reads
VPC-SC bypass via non-restricted service 2020+ Data moved through a service not on the restricted list (e.g. a build/logging/telemetry sink) Add all data-capable services to restricted-services; unlisted services are exfil channels
VPC-SC bypass via log sinks / exports 2020+ Exporting data to a destination outside the perimeter through an allowed pipeline Treat exports, transfers, and sinks as egress; constrain destinations, not just services
Access-level IP spoofing / on-net attacker ongoing IP-only access levels trust anyone on the corp range Prefer device + identity conditions over raw IP; IP alone is weak

Case study 1 — Capital One (2019), the canonical "why a perimeter matters." An attacker exploited a Server-Side Request Forgery (T1190) in a misconfigured WAF, reached the EC2 instance metadata service (T1552.005), and lifted temporary IAM credentials. Those credentials were legitimately scoped to read S3, so the subsequent s3 sync of ~100 million records (T1530 → T1567) was allowed by IAM — nothing was "broken," the attacker simply used valid permissions from an unexpected place. In a GCP world with VPC-SC, the equivalent BigQuery/GCS read would be evaluated against the perimeter: the request originates from the attacker's context, not from inside the perimeter and not matching any access level, so it is denied before any data moves. This is the single cleanest illustration of IAM-allow but perimeter-deny, and exactly the scenario interviewers use to test whether you understand that VPC-SC and IAM are independent gates.

Case study 2 — the "non-restricted service" bypass class. Security researchers (and Google's own known limitations documentation) have repeatedly shown that VPC-SC only guards the services you list. If BigQuery and GCS are restricted but some other data-capable API — a build service, a data-transfer service, a logging export, an analytics pipeline — is not, an insider or a compromised principal inside the perimeter can route data through the unlisted service to an external destination. The data never crossed a restricted boundary, so no violation fires. The lesson is operational: perimeter completeness is a moving target; every time GCP adds a data-capable API you must evaluate whether it belongs on the restricted list, and you must treat exports/sinks/transfers as egress channels with pinned destinations. This is why "VPC-SC is a layer, not a wall" is the honest senior answer.

Case study 3 — dry-run that never got enforced. A very common real-world failure is organizational, not technical: teams create a perimeter in dry-run to avoid breaking pipelines, generate months of dryRun: true violation logs, and never promote to enforced because someone always fears an outage. The result is a perimeter that looks deployed on a dashboard but blocks nothing. The lesson: dry-run is a rollout phase, not a destination — you need an SLO ("promote within N weeks") and audit-log alerting that distinguishes dry-run from enforced violations, or you have paid all the complexity cost for none of the protection.

Top 5 in depth

1. Enforce the perimeter with device-and-identity access levels (mitigates the core credential-compromise threat). The whole value of VPC-SC collapses if the perimeter stays in dry-run or trusts a raw IP range. Technically, you enforce by promoting the dry-run spec to the enforced status (gcloud access-context-manager perimeters dry-run enforce), and you make the access level strong by using a custom CEL level that requires device.is_corp_owned_device == true && device.encryption_status == ENCRYPTED in addition to an IP condition, so a stolen token from an unmanaged host fails even if the attacker tunnels through a corp IP. The trade-off is real user friction: device signals depend on Endpoint Verification being deployed, and break-glass access for incident responders must be pre-planned or you lock yourself out during an outage.

2. Complete and continuously reviewed restricted-services list (mitigates the non-restricted-service bypass). A perimeter only guards listed APIs, so the control is only as strong as list completeness. Implement it by starting from Google's "supported services" list, restricting every data-capable service you use, and adding a quarterly review plus a CI check that fails if a newly-adopted data API is missing from the perimeter. Use --enable-vpc-accessible-services to also cap what inside callers can reach. The limit: services that VPC-SC does not support at all cannot be protected — you must handle those with data-governance (don't put crown-jewel data there) rather than perimeter config.

3. Tightly scoped egress rules (mitigates exfil-shaped flows). Egress is the dangerous direction. Each egress rule should pin an exact identity, an exact method (e.g. google.storage.objects.get, never *), and an exact destination project. Implement egress rules as version-controlled YAML applied via --set-egress-policies, and lint them in CI to reject wildcards and public destinations. The trade-off is operational overhead: legitimate cross-perimeter integrations need explicit rules, which slows delivery — but that friction is the point, because every egress rule is a sanctioned hole.

4. Dry-run-first rollout with promotion SLO (mitigates outages and the "dry-run forever" anti-pattern). The dual spec/status model means you can observe dryRun: true violations for weeks, fix the surprises (a forgotten CI service account, a cross-region job), then promote — all without downtime. Make it real with an alert that tracks the ratio of dry-run to enforced perimeters and a documented "promote within N weeks" SLO. Trade-off: the observation window is exactly when you have no protection, so keep it as short as the risk allows and prioritize your most sensitive perimeters.

5. ACM-as-code with least-privilege admin and mutation alerting (mitigates config drift and defense-impairment). VPC-SC's own weakness is that an over-privileged principal can delete or weaken the perimeter to enable theft (T1562). Manage the entire access policy — levels, perimeters, ingress/egress — in Terraform (google_access_context_manager_service_perimeter), gate changes behind PR review and policy-as-code, restrict accesscontextmanager.* admin roles to a tiny group with break-glass, and alert on any perimeter mutation in audit logs. Trade-off: IaC adds process weight and a learning curve, but hand-edited perimeters in a large org will drift and silently open holes.

6. Detection & Response

What to log / telemetry sources

  • Cloud Audit Logs — Policy Denied / VPC-SC violations. The authoritative signal is protoPayload.metadata.@type = type.googleapis.com/google.cloud.audit.VpcServiceControlAuditMetadata. Key fields: violationReason (e.g. NO_MATCHING_ACCESS_LEVEL, RESOURCES_NOT_IN_SAME_SERVICE_PERIMETER, EGRESS_VIOLATION, INGRESS_VIOLATION), dryRun (true/false), the servicePerimeterName, the caller identity (authenticationInfo.principalEmail), and requestMetadata.callerIp.
  • Data Access audit logs for BigQuery/GCS (data_access) to correlate what data was targeted around a violation.
  • Admin Activity logs for ACM mutations — SetIamPolicy, UpdateServicePerimeter, DeleteServicePerimeter, ReplaceAccessLevels — to catch someone weakening the boundary.
  • IAM impersonation events (GenerateAccessToken) to catch identity-borrowing into the perimeter.
  • Export all of the above via a log sink to BigQuery (for hunting) and to Chronicle/SIEM (for alerting). Ironically, pin that sink's destination so the sink itself is not an exfil channel.

Sample detections

Detection 1 — enforced VPC-SC violations (real blocks), Cloud Logging / Log Analytics filter:

logName:"cloudaudit.googleapis.com%2Fpolicy"
protoPayload.metadata.@type="type.googleapis.com/google.cloud.audit.VpcServiceControlAuditMetadata"
protoPayload.metadata.dryRun="false"
protoPayload.metadata.violationReason=("NO_MATCHING_ACCESS_LEVEL" OR "EGRESS_VIOLATION")

Detection 2 — burst of egress violations from one principal (BigQuery SQL over exported audit logs):

SELECT
  protopayload_auditlog.authenticationInfo.principalEmail AS principal,
  JSON_VALUE(protopayload_auditlog.metadataJson, '$.violationReason') AS reason,
  protopayload_auditlog.requestMetadata.callerIp AS src_ip,
  COUNT(*) AS violations
FROM `security_logs.cloudaudit_googleapis_com_policy_*`
WHERE JSON_VALUE(protopayload_auditlog.metadataJson, '$.dryRun') = 'false'
  AND JSON_VALUE(protopayload_auditlog.metadataJson, '$.violationReason') = 'EGRESS_VIOLATION'
  AND _PARTITIONTIME > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
GROUP BY principal, reason, src_ip
HAVING violations > 20
ORDER BY violations DESC

Detection 3 — Sigma-style rule for perimeter tampering (map to your SIEM):

title: GCP VPC-SC Perimeter Deleted or Weakened
logsource:
  product: gcp
  service: audit
detection:
  selection:
    protoPayload.methodName:
      - 'google.identity.accesscontextmanager.v1.AccessContextManager.DeleteServicePerimeter'

      - 'google.identity.accesscontextmanager.v1.AccessContextManager.ReplaceServicePerimeters'
  condition: selection
level: high

Response playbook (containment → eradication → recovery)

  1. Triage the violation. Pull the VpcServiceControlAuditMetadata — is dryRun false (real block) or true (staging)? Capture principalEmail, callerIp, violationReason, targeted resource. A cluster of enforced EGRESS_VIOLATION from one identity to one external project is an active exfil attempt.
  2. Contain the identity. Disable/rotate the implicated service-account key (gcloud iam service-accounts keys delete), revoke active OAuth grants, and remove the SA from sensitive roles. If a user, force re-auth and revoke sessions.
  3. Contain the path. If an egress or ingress rule was abused, remove or narrow it immediately (--set-egress-policies with the tightened file). If a bridge was the vector, drop the offending project from the bridge.
  4. Verify the perimeter is intact. Diff the live perimeter against the Terraform state; if DeleteServicePerimeter/UpdateServicePerimeter appears in Admin Activity around the incident, treat it as attacker-driven defense impairment and restore from IaC.
  5. Eradicate the leak source. Hunt for where the credential leaked (git secret scan, CI logs, SSRF-reachable metadata) and close it — the perimeter blocked the symptom, not the root cause.
  6. Assess exposure. Cross-reference Data Access logs for any successful reads by the identity before the block, to bound what actually left.
  7. Recover and harden. Re-enable clean credentials, strengthen the relevant access level (add device posture), close any non-restricted-service gap the attacker probed, and add/verify alerting for the observed violationReason.
  8. Post-incident. Feed the pattern back: was the perimeter in dry-run when it should have been enforced? Was an egress rule too broad? Update the promotion SLO and policy-as-code lints.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What problem does VPC Service Controls solve that IAM does not?
IAM decides whether a principal is authorized to perform an action; it says nothing about where the request came from or whether data may cross a boundary. VPC-SC adds a context- and boundary-aware gate: it prevents data in restricted services (BigQuery, GCS, KMS…) from being read or moved by a caller outside the perimeter, even when that caller holds valid IAM credentials. The two are evaluated as an AND, so a stolen but legitimately-scoped token is denied at the perimeter before any data moves. In short, VPC-SC is a data-exfiltration control; IAM is an access control.
Signal: Do they understand IAM-allow-but-perimeter-deny? This is the core mental model.

Q2. What are the main building blocks of a VPC-SC deployment?
An org-level access policy in Access Context Manager is the container. Inside it you define service perimeters (a set of project numbers plus a list of restricted services), access levels (named conditions on IP, device posture, identity, region), ingress and egress rules (fine-grained cross-boundary allows), and optionally bridges (controlled sharing between perimeters). Each perimeter also carries an enforced config and a dry-run config.
Signal: Can they name the pieces without hand-waving? Bonus for mentioning project number vs ID.

Q3. What is dry-run mode and why does it matter?
Every perimeter has a spec (dry-run) and a status (enforced) configuration; requests are evaluated against both, but only the enforced config blocks — the dry-run config merely logs what it would have denied, tagged dryRun: true. It matters because VPC-SC is notorious for breaking legitimate pipelines, so you stage the real config as dry-run, watch audit logs for surprises, fix them, and only then promote. The failure mode is leaving it in dry-run forever, which gives false comfort and zero protection.
Signal: Operational maturity — do they know how to roll this out without an outage?

Q4. If a service-account key leaks to a public GitHub repo, how does VPC-SC help?
If the key's project is inside an enforced perimeter and the attacker uses the key from their own host, the request to a restricted service originates from outside the perimeter and matches no access level or ingress rule, so it is denied before any data is read — the leaked key becomes a dead end. IAM would have allowed the read (the key is valid and scoped), which is exactly why the perimeter's independent check is the thing that saves you. You still must rotate the key and fix the leak; VPC-SC contains the blast radius, it doesn't fix the root cause.
Signal: Connecting a concrete leak to defense-in-depth.

Q5. What is the restricted VIP and why is it needed?
restricted.googleapis.com resolves to 199.36.153.4/30, a special virtual IP that routes Google API traffic privately (via Private Google Access) and reaches only VPC-SC-supported APIs. You point your private DNS for *.googleapis.com at it so that VMs without external IPs can still call Google APIs and cannot reach non-supported (potentially bypass-capable) endpoints. Using the ordinary public endpoint or private.googleapis.com would widen the reachable API surface.
Signal: Do they understand the network plumbing that makes private, perimeter-consistent access work?

Tier 2 — System design / trade-offs

Q6. Design a perimeter strategy for an org with a data-lake project, an analytics project, and an external partner integration.
I'd put the data-lake and analytics projects in one regular perimeter with bigquery, storage, and cloudkms restricted, since they share sensitive data and need low-friction internal access. The partner integration stays outside; I grant it in via a tightly scoped ingress rule — exact partner service account, sources gated by an access level (partner's IP plus, ideally, mutual identity), and ingressTo limited to the specific dataset and methods needed. Any outbound flow to the partner becomes an egress rule pinned to their destination project and read-only methods. I roll the whole thing out dry-run first, manage it in Terraform, and alert on both enforced and dry-run violations. I'd resist merging the partner into the perimeter or using a broad bridge, because that widens intra-perimeter trust.
Signal: Segmentation instinct, ingress/egress scoping, rollout discipline.

Q7. What are VPC-SC's biggest limitations, and how do you compensate?
Four big ones. (1) It only guards listed services — I keep the restricted list complete and reviewed, and treat exports/sinks as egress. (2) It does not stop exfil to a colluding project inside the same perimeter — I segment crown-jewel data into its own perimeter and lean on least-privilege IAM and DLP. (3) There are documented bypass classes via non-restricted or export-capable services — I pin destinations and monitor. (4) It adds operational friction that tempts teams to disable it — I back it with governance, break-glass, and support runbooks. The honest framing is "a strong layer with sharp edges," not a wall.
Signal: Intellectual honesty; a senior candidate volunteers weaknesses.

Q8. IP-based access levels vs device-and-identity levels — trade-offs?
IP-only levels are easy but weak: any attacker who lands on the corp network or tunnels through corp NAT is trusted, and IPs can be spoofed or shared. Device-and-identity levels (custom CEL requiring is_corp_owned_device, encryption, screen-lock, plus specific identities) give far higher assurance because a stolen token from an unmanaged laptop fails even from a corp IP — but they require Endpoint Verification deployment, add user friction, and complicate incident-responder access. I default to device+identity for crown-jewel perimeters and reserve IP-only for low-sensitivity or transitional cases, always with break-glass planned.
Signal: Understanding that "context" should be more than an IP.

Q9. How do you roll out VPC-SC across a large org without causing outages?
Inventory data-capable services and dependencies first, then deploy dry-run everywhere and mine dryRun: true violations to build an allowlist of legitimate ingress/egress (surprise service accounts, cross-region jobs, CI). Manage the config in Terraform so it's reviewable and reversible, alert on the dry-run signal, and promote perimeter-by-perimeter starting with the most sensitive, each behind a promotion SLO so nothing lingers un-enforced. Provide a self-service runbook so teams can request scoped ingress/egress rather than demanding the perimeter be turned off. The key is treating dry-run as a phase with an exit criterion, not a permanent state.
Signal: Change management at scale, not just the CLI.

Q10. When would you use a perimeter bridge vs an ingress/egress rule vs merging perimeters?
A bridge is for letting projects in different perimeters share resources bidirectionally with minimal rule-writing — good for a shared logging or tooling project that many perimeters need, but it widens trust, so membership must be minimal and reviewed. Ingress/egress rules are for directional, fine-grained access — a specific external identity reaching in, or an inside identity reaching one external destination — and they're my default because they're least-privilege. Merging projects into one perimeter is right only when they genuinely form one trust domain with shared sensitive data; over-merging destroys segmentation. I reach for ingress/egress first, bridges sparingly, merging deliberately.
Signal: Knowing three overlapping tools and picking by least privilege.

Tier 3 — Expert / deep-dive

Q11. Walk through exactly how a request to a restricted service is evaluated end-to-end.
The request hits the Google API front end for the target *.googleapis.com service. Before the service's own logic runs, the front end evaluates the perimeter: it determines the request origin (VPC network, source IP, or ACM-asserted identity/device) and the target resource's perimeter membership. The allow path is: caller and target in the same perimeter → OK; else if inbound and an ingress rule matches (identity + source + target service/method/resource) → OK; else if outbound and an egress rule matches the external target → OK; else if a bound access level is satisfied → OK; otherwise deny and emit a VpcServiceControlAuditMetadata violation with a violationReason. Only after the perimeter allows does IAM evaluate the principal's permissions. Both the enforced and dry-run configs are checked, but dry-run only logs. So the final result is perimeter-allow AND iam-allow.
Signal: Depth — the AND semantics, the ordering, the front-end enforcement point.

Q12. Describe a realistic VPC-SC bypass and how you'd defend against it.
The classic is the non-restricted-service pivot: BigQuery and GCS are restricted, but some other data-capable API — a build/CI service, a data-transfer service, a logging export, or an analytics endpoint — is not on the restricted list. A principal inside the perimeter routes sensitive data through that unlisted service to an external destination; because the data never crossed a restricted boundary, no violation fires. A related variant abuses an over-broad egress rule or a log sink whose destination lives outside the perimeter. Defenses: keep the restricted list exhaustive and review it whenever GCP GAs new data APIs; enable vpc-accessible-services to cap the inside surface; model every export/sink/transfer as an egress rule with a pinned destination and read-only methods; and monitor egress-shaped audit events. The meta-point is that perimeter completeness is a continuous program, not a one-time config.
Signal: Knows the real weakness class and the operational answer, not a toy.

Q13. How does VPC-SC interact with service-account impersonation and cross-project access?
Impersonation (iam.serviceAccounts.getAccessToken) lets principal A borrow SA B's identity; VPC-SC evaluates the request using the resulting identity and the request's origin context, so impersonation doesn't magically place the caller inside the perimeter — the origin still matters, and an impersonated call from outside still needs an access level or ingress rule. The dangerous combination is an attacker who can impersonate an SA and originate from a trusted context; then the perimeter alone won't save you, which is why you also restrict serviceAccountTokenCreator tightly and alert on GenerateAccessToken. For legitimate cross-project access within a perimeter, membership by project number means all listed projects are mutually reachable — a reason to segment sensitive projects rather than dumping everything into one large perimeter.
Signal: Understands the intersection of identity-borrowing and boundary context.

Q14. Compare achieving a VPC-SC-equivalent data perimeter on AWS. What's hard about it?
AWS has no single org-wide service-boundary primitive; you assemble an approximation. Service Control Policies (and, since 2024, Resource Control Policies) can add org-wide Deny conditions like aws:SourceVpc, aws:SourceVpce, aws:PrincipalOrgID; resource policies (S3 bucket policies, KMS key policies) enforce aws:SourceVpce; VPC endpoints / PrivateLink with endpoint policies keep traffic private and constrain reachable resources; and AWS publishes formal "data perimeter" guidance to stitch these together. The hard parts: it's many controls that must be consistent across every account and resource type, there's no clean "one boundary object" to reason about, per-service coverage and condition-key support vary, and it's easy to leave a gap (a bucket without the SourceVpce condition). GCP's VPC-SC gives you a single declarative perimeter object spanning services and projects, which is conceptually simpler even if operationally sharp.
Signal: Cross-cloud fluency and honesty about each model's rough edges.

Q15. Your perimeter is enforced but you see a burst of successful BigQuery reads from an unusual IP with no violations. What's your hypothesis and investigation?
No violation means the request satisfied the perimeter — so either the IP falls inside a trusted access level (e.g. an over-broad IP CIDR, or an attacker who reached a corp-trusted context), or the access happened via a legitimately-scoped ingress rule, or through a project inside the same perimeter (intra-perimeter trust). My first move is to pull the Data Access logs to confirm the principal, method, and datasets, then check which access level / ingress rule the caller matched and whether that IP range is too broad. I'd correlate with impersonation events (was an inside SA's token minted?) and with the identity's normal baseline. If it's an over-broad IP-only level, I tighten it to require device posture; if it's an inside compromised principal, the perimeter can't help and I pivot to IAM containment and DLP. The teaching point: absence of a violation doesn't mean absence of exfil — it means the exfil used a sanctioned path, and those paths are exactly what you must keep narrow.
Signal: Can they reason about the silent failure mode where the perimeter is working as configured but the configuration is too loose?

The optimized takeaway: VPC-SC is GCP-distinctive. On AWS and Azure you build a data perimeter out of several partial controls that must be kept mutually consistent across every account/resource, with no single "boundary" object and no native perimeter-level dry-run. That difference is exactly why VPC-SC comes up in GCP-focused interviews — being able to say "there is no clean AWS equivalent; you approximate it with SCPs/RCPs, resource policies, and PrivateLink, and the gaps are where exfil sneaks through" is a strong signal.

9. Key Takeaways

  • VPC-SC is a data-exfiltration perimeter, orthogonal to IAM and evaluated as AND — a stolen but valid credential used from outside the perimeter is denied before any data moves. This is the headline defense-in-depth story.
  • It guards only listed restricted services on listed project numbers. Completeness of the restricted-services list is the whole ballgame; unlisted data-capable services are exfil channels.
  • Dry-run (spec) vs enforced (status) is the operational core. Roll out dry-run, mine violations, then promote — but treat dry-run as a phase with an exit SLO, never a permanent state.
  • Egress is the dangerous direction. Pin egress rules to exact identities, methods, and destination projects; wildcards turn a control into a hole.
  • Access levels should assert more than an IP. Prefer device posture and identity via custom CEL levels; IP-only trust falls to any on-network attacker.
  • VPC-SC does not stop intra-perimeter exfil. A colluding or compromised principal inside the boundary can still move data around; segment crown-jewel data and layer IAM least-privilege and Cloud DLP.
  • The perimeter itself is an attack target. Over-privileged ACM admins can delete or weaken it (T1562); manage it as code, restrict admin, and alert on perimeter mutations.
  • Detection hinges on VpcServiceControlAuditMetadata. Alert on enforced and dry-run violations, distinguish violationReasons, and remember a successful read with no violation can still be exfil via a sanctioned path.
  • There is no clean AWS/Azure equivalent — those clouds assemble a data perimeter from SCPs/RCPs, resource policies, and Private Link, which makes VPC-SC a high-signal GCP topic.
  • VPC-SC is a strong layer, not a wall. Its honest value is blast-radius reduction against credential compromise; pair it with credential hygiene, DLP, and monitoring.
