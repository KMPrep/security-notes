GCP VPC Networking & Firewall Security

Category: Cloud Networking / Network Segmentation
GCP Services: VPC, Cloud Firewall (VPC firewall rules, hierarchical + network firewall policies), Cloud NAT, Private Google Access, Private Service Connect, Shared VPC, VPC Peering
Key Docs: VPC Overview · Cloud Firewall · Firewall Policies
Frameworks: MITRE ATT&CK (Cloud), NIST SP 800-207 (Zero Trust), CIS GCP Foundations Benchmark v3.0

1. Overview

A Google Cloud VPC is a global, software-defined virtual network that spans every region in a Google Cloud organization without any customer-managed routers, tunnels, or gateways. This is the single most important structural fact in a design review: unlike an AWS VPC (which is regional) or an Azure VNet (which is regional and peered per-region), one GCP VPC's subnets can live in us-central1, europe-west1, and asia-southeast1 simultaneously, and workloads in those subnets reach each other over Google's private backbone using internal IPs with no peering, no VPN, and no inter-region gateway. Subnets are the regional object; the VPC is global. Routing, firewalling, and DNS are all properties of the global network.

Security in a GCP VPC is enforced primarily by stateful firewall rules and, increasingly, firewall policies (hierarchical and network/global-network policies) evaluated per-VM-NIC — not per-subnet like AWS security groups tied to ENIs or Azure NSGs tied to subnets. Firewall targeting can key off network tags or, far more securely, service accounts, enabling identity-anchored microsegmentation. Because the network is global and the firewall is identity-aware, GCP pushes you toward a default-deny, least-privilege segmentation design where you delete the permissive defaults, deny egress by default, and open flows explicitly. Getting this wrong is how real breaches happen: an over-broad 0.0.0.0/0 ingress rule or a forgotten default-allow-ssh is the cloud equivalent of leaving the front door open.

TL;DR
- GCP VPC is global; subnets are regional. AWS VPC and Azure VNet are regional — GCP needs no peering to cross regions inside one VPC.
- Firewall rules are stateful and applied at the VM NIC, evaluated by priority (0–65535, lowest wins); return traffic is auto-allowed. Every VPC has implied allow-egress and implied deny-ingress.
- Target by service account, not just network tag — tags are unauthenticated metadata any editor can set; SAs require iam.serviceAccountUser and give real identity-based microsegmentation.
- Firewall policies (hierarchical at org/folder + network firewall policies) add goto_next, are inherited, and can't be overridden by project owners — this is how a security org enforces guardrails top-down.
- Design: default-deny egress + Private Google Access + Cloud NAT + org policy constraints/compute.vmExternalIpAccess (no public IPs). Detect with VPC Flow Logs + Firewall Rules Logging in BigQuery.

2. How It Works

2.1 Core Mechanics

The GCP network stack separates the global control plane (Andromeda SDN — programs flow rules onto every host) from the data plane (per-host virtual switch enforcing firewall and routing decisions in the hypervisor before packets ever hit a physical wire). A firewall decision is made on the sending and receiving host, not at a chokepoint appliance, which is why GCP firewalling scales horizontally and is stateful without a stateful appliance bottleneck.

Concept Description
VPC network Global resource; a namespace of subnets, routes, and firewall rules. Auto-mode or custom-mode.
Subnet Regional resource with a primary IPv4 range plus optional secondary ranges (for GKE alias IPs).
Auto-mode VPC Auto-creates a /20 subnet in every region from 10.128.0.0/9. Convenient, insecure default — avoid.
Custom-mode VPC You define every subnet/CIDR explicitly. The only defensible choice for production.
Firewall rule (VPC-level) Stateful L3/L4 rule: direction, priority, action (allow/deny), source/dest, protocol/port, target.
Firewall policy Container of rules attachable at org/folder (hierarchical) or to a network (network firewall policy); supports goto_next.
Priority Integer 0–65535; lower number = higher precedence. First match by priority wins; ties broken deny-over-allow.
Target Which VMs the rule applies to: all instances, network tags, or service accounts.
Network tag Arbitrary string label on a VM's NIC; unauthenticated; used for targeting and as route/firewall selector.
Implied rules Every VPC has an invisible allow-all-egress (priority 65535) and deny-all-ingress (priority 65535).
Route Global (dynamic/custom) or subnet routes; steer traffic; a route to 0.0.0.0/0 via default-internet-gateway = internet egress path.
Private Google Access Subnet flag letting VMs with no external IP reach Google APIs via private.googleapis.com / restricted.googleapis.com.
Cloud NAT Regional, managed, egress-only SNAT — no inbound; gives private VMs outbound internet without public IPs.
Shared VPC One host project owns the network; service projects attach and deploy into it. Central network control.
VPC Peering Non-transitive, RFC1918 connectivity between two VPCs; each keeps its own firewall/routes.
Private Service Connect (PSC) Consumer-side private IP endpoint mapping to a producer service (Google API or partner) — no peering, no range overlap.

Firewall evaluation order (critical to state correctly): For a given packet on a NIC, GCP evaluates hierarchical firewall policies (org → folder, top-down) first, then network firewall policies / global network firewall policies, then VPC firewall rules, then the implied rules — but within each level, evaluation is by priority. A policy rule can carry the special action goto_next, which means "stop evaluating this level and fall through to the next level down." An explicit allow or deny in a higher level is terminal. This layering is what lets a central security team set an org-wide "deny inbound RDP from the internet" that a project owner cannot override, while still delegating everything else.

Firewall rules are stateful: if ingress is allowed, the return path is automatically permitted regardless of egress rules, and vice versa — you do not create mirror rules like you would with AWS NACLs. Rules filter only IPv4/IPv6 L3–L4 (protocol + port + CIDR/tag/SA); there is no L7 in the base firewall (that's Cloud NGFW with intrusion prevention, or the WAF in Cloud Armor).

2.2 Concrete Configuration

Create a custom-mode VPC and a private subnet with flow logs and Private Google Access:

# Custom-mode VPC — no auto subnets, we define everything
gcloud compute networks create prod-vpc \
  --subnet-mode=custom \
  --bgp-routing-mode=global

# Regional subnet with Private Google Access + Flow Logs ON at high sampling
gcloud compute networks subnets create prod-app-usc1 \
  --network=prod-vpc \
  --region=us-central1 \
  --range=10.10.0.0/24 \
  --secondary-range=gke-pods=10.10.16.0/20,gke-svc=10.10.32.0/24 \
  --enable-private-ip-google-access \
  --enable-flow-logs \
  --logging-flow-sampling=0.5 \
  --logging-aggregation-interval=interval-5-sec \
  --logging-metadata=include-all

Delete the dangerous auto-created defaults (do this first on any new project):

# The 'default' network ships default-allow-ssh, -rdp, -icmp, -internal from 0.0.0.0/0-ish ranges
gcloud compute firewall-rules delete default-allow-ssh default-allow-rdp \
  default-allow-icmp default-allow-internal --quiet

Service-account-targeted ingress (identity microsegmentation) — allow only the LB health check + IAP range to reach the app tier's SA:

# Allow Google front-end health-check + IAP TCP-forwarding ranges to app VMs by SERVICE ACCOUNT
gcloud compute firewall-rules create allow-app-ingress-from-lb-iap \
  --network=prod-vpc \
  --direction=INGRESS \
  --action=ALLOW \
  --priority=1000 \
  --rules=tcp:8443 \
  --source-ranges=130.211.0.0/22,35.191.0.0/16,35.235.240.0/20 \
  --target-service-accounts=app-tier@prod-proj.iam.gserviceaccount.com \
  --enable-logging

Default-deny egress, then explicitly allow only Google APIs + the DB tier — the core of a zero-trust egress posture:

# 1) Highest-priority-number catch-all DENY egress (implied allow is 65535; we shadow it at 65534)
gcloud compute firewall-rules create deny-all-egress \
  --network=prod-vpc --direction=EGRESS --action=DENY \
  --priority=65534 --rules=all --destination-ranges=0.0.0.0/0 \
  --enable-logging

# 2) Allow egress to restricted.googleapis.com VIP only (used with Private Google Access + VPC-SC)
gcloud compute firewall-rules create allow-egress-google-restricted \
  --network=prod-vpc --direction=EGRESS --action=ALLOW \
  --priority=1000 --rules=tcp:443 \
  --destination-ranges=199.36.153.4/30 \
  --target-service-accounts=app-tier@prod-proj.iam.gserviceaccount.com

# 3) App tier may egress to DB tier on 5432 only, SA->SA
gcloud compute firewall-rules create allow-egress-app-to-db \
  --network=prod-vpc --direction=EGRESS --action=ALLOW \
  --priority=1000 --rules=tcp:5432 \
  --destination-ranges=10.10.2.0/24 \
  --target-service-accounts=app-tier@prod-proj.iam.gserviceaccount.com

Hierarchical firewall policy at the folder level — a guardrail project owners cannot remove:

# Create the policy under a folder, add a org-wide block on inbound RDP from the internet
gcloud compute firewall-policies create \
  --folder=123456789012 \
  --short-name=org-guardrails

POLICY_ID=$(gcloud compute firewall-policies list --folder=123456789012 \
  --format='value(id)')

gcloud compute firewall-policies rules create 1000 \
  --firewall-policy=$POLICY_ID \
  --direction=INGRESS --action=deny \
  --layer4-configs=tcp:3389 \
  --src-ip-ranges=0.0.0.0/0 \
  --enable-logging

# A goto_next rule lets everything else fall through to lower levels / VPC rules
gcloud compute firewall-policies rules create 2000 \
  --firewall-policy=$POLICY_ID \
  --direction=INGRESS --action=goto_next \
  --layer4-configs=all --src-ip-ranges=0.0.0.0/0

# Associate the policy with the folder so it's inherited by all descendant projects
gcloud compute firewall-policies associations create \
  --firewall-policy=$POLICY_ID --folder=123456789012

Cloud NAT for outbound-only internet from private VMs (no external IPs anywhere):

gcloud compute routers create prod-router \
  --network=prod-vpc --region=us-central1

gcloud compute routers nats create prod-nat \
  --router=prod-router --region=us-central1 \
  --nat-all-subnet-ip-ranges \
  --auto-allocate-nat-external-ips \
  --enable-logging --log-filter=ERRORS_ONLY

Org policy to forbid external IPs org-wide (prevents accidental public exposure):

# Deny external IPs on all VMs across the org — the single highest-value guardrail
gcloud resource-manager org-policies deny compute.vmExternalIpAccess all \
  --organization=987654321098

Private Service Connect endpoint to reach Google APIs by a private IP (no external IP, no NAT):

gcloud compute addresses create psc-google-apis \
  --global --purpose=PRIVATE_SERVICE_CONNECT \
  --addresses=10.10.250.10 --network=prod-vpc

gcloud compute forwarding-rules create psc-google-apis-fr \
  --global --network=prod-vpc \
  --address=psc-google-apis \
  --target-google-apis-bundle=all-apis

Case study 1 — The default-allow-internal east-west blast radius. An auto-mode VPC ships a firewall rule default-allow-internal permitting all protocols and ports from the entire 10.128.0.0/9 source range to every VM. A single compromised public-facing VM (say, an unpatched web server exploited via an application flaw, ATT&CK T1190) can then reach every other VM in every region on any port — Redis with no auth, unauthenticated Elasticsearch, internal admin panels. This is textbook lateral movement (T1021). The control that stops it is custom-mode VPC + service-account-scoped rules: app VMs can only reach db VMs on 5432, and nothing can reach the management subnet except from the bastion's SA. The blast radius collapses from "the whole company" to "one port on one tier."

Case study 2 — Metadata SSRF to token theft to exfil. An SSRF vulnerability in a GCP-hosted app lets an attacker coerce the server into requesting http://169.254.169.254/computeMetadata/v1/instance/service-accounts/default/token. If the VM runs with a broad service account and the legacy metadata endpoints are enabled, the attacker steals an OAuth token (T1552.005) and calls Google APIs as the workload. With a permissive egress firewall, they then exfiltrate the token/data to an external C2. Three layers each independently break this chain: (1) least-privilege SA limits what the token can do; (2) metadata concealment / GKE Workload Identity stops the token being reachable; (3) default-deny egress blocks the exfiltration call to the attacker's server because only restricted.googleapis.com and the DB tier are allowed outbound.

Case study 3 — CloudSQL on a public IP. A team, moving fast, provisions a Cloud SQL instance with a public IP and sets the authorized network to 0.0.0.0/0 "temporarily" for a migration. Internet-wide scanners (Shodan-class) find the open 5432/3306 within hours, and credential-stuffing or a weak password yields full DB access (T1190/T1078). The org-policy constraints/sql.restrictPublicIp plus constraints/compute.vmExternalIpAccess, combined with Private Service Connect for private DB connectivity, makes this misconfiguration structurally impossible — the API rejects the public-IP request at creation time.

Top 5 in Depth

1. Default-deny egress with an allowlist. GCP's implied allow-all-egress at priority 65535 means, out of the box, any VM can talk to any internet destination — the exact condition attackers need for exfiltration and C2. The fix is a deny-all-egress rule at priority 65534 (one better than the implied rule) scoped to destination-ranges=0.0.0.0/0, then a small set of higher-priority (lower-number) allow rules for exactly the destinations the workload needs: the restricted.googleapis.com VIP 199.36.153.4/30, your DB CIDR on 5432, and perhaps a package mirror. Technically this works because firewall evaluation is strict-priority and the first match wins; the deny sits just above the implied allow so it catches everything not explicitly permitted. Trade-off: it breaks anything that needs ad-hoc internet (OS package updates, third-party APIs, telemetry SDKs) until you enumerate and allow those flows — this is real operational work, and teams that skip it get paged. Pair it with a proxy or Secure Web Gateway if you need FQDN-based (not CIDR) egress control, since base firewall rules are L3/L4 only.

2. Service-account-based targeting instead of network tags. A network tag is just a string on a VM's NIC — anyone with compute.instances.setTags (often granted broadly via roles/compute.instanceAdmin) can add the tag that a firewall rule allow-lists, effectively granting themselves network access. Targeting rules by service account (--target-service-accounts / --source-service-accounts) ties the rule to an identity that requires roles/iam.serviceAccountUser to attach to a VM — a far more tightly held permission. This turns firewalling into identity-anchored microsegmentation aligned with zero-trust (NIST 800-207). Trade-off: a rule can target tags or SAs but not mix them in one rule; SA-based rules require disciplined SA-per-tier design and are harder to reason about in the console. In production I default to SAs for anything sensitive and reserve tags for coarse, non-security routing.

3. Hierarchical firewall policies as org guardrails. VPC firewall rules live in a project and can be edited by anyone with roles/compute.securityAdmin on that project — so a project owner can always undo them. Hierarchical firewall policies attach at the organization or folder node and are inherited by all descendants; a project owner cannot override or delete them. This is the mechanism a central security team uses to enforce non-negotiables ("no inbound RDP from the internet anywhere," "always allow the health-check ranges") while delegating everything else via goto_next. Technically, hierarchical rules are evaluated before any VPC-level rule, and their goto_next action is what preserves delegation. Trade-off: it's another evaluation layer to reason about; a too-aggressive hierarchical deny can break workloads org-wide and is slow to diagnose because project admins can't see why traffic is blocked without policy visibility. Always pair with Firewall Rules Logging on the policy rules.

4. No public IPs via org policy + Cloud NAT + Private Google Access. The org policy constraint constraints/compute.vmExternalIpAccess set to deny-all makes it impossible to attach an external IP to any VM in the org — removing the single most common exposure vector. Private VMs still need to (a) reach Google APIs and (b) sometimes reach the internet outbound. Private Google Access (a subnet flag) routes API calls to the private/restricted.googleapis.com VIPs over Google's backbone without any external IP, and Cloud NAT provides egress-only SNAT for genuine internet needs — crucially, Cloud NAT has no inbound path, so it can't be used for ingress attacks. Trade-off: you must plan NAT IP/port allocation (port exhaustion under high fan-out to few destinations is a real incident class) and use restricted.googleapis.com with VPC Service Controls for the strongest posture. This trio is the canonical GCP "private-by-default" network.

5. Flow Logs + Firewall Rules Logging exported to BigQuery. You cannot detect lateral movement or exfil you can't see, and both flow logs and firewall logging are OFF by default (for cost reasons). Enable VPC Flow Logs per-subnet with a sensible sampling rate (0.5 for sensitive subnets) and --enable-logging on every allow/deny rule you care about, then sink both to BigQuery via a log sink for SQL-based detection. Flow logs give you the 5-tuple, bytes, and RTT; firewall logging tells you which rule allowed/denied a connection and the target — invaluable for both detection and debugging "why is this blocked." Trade-off: logging volume and BigQuery cost can be significant on chatty subnets; tune sampling and the aggregation interval, and consider metadata EXCLUDE_ALL_METADATA where you only need the tuple. In practice the cost is trivial next to the cost of an undetected breach.

6. Detection & Response

What to log / telemetry sources

  • VPC Flow Logs — fields: connection.src_ip/dest_ip/src_port/dest_port/protocol, bytes_sent, packets_sent, rtt_msec, src_instance, dest_instance, src_vpc, reporter. Enabled per-subnet.
  • Firewall Rules Logging — fields: rule_details.reference (which rule), rule_details.action (ALLOW/DENY), rule_details.direction, connection.*, instance.*, target_service_accounts. Enabled per-rule via --enable-logging.
  • Cloud NAT logging — translations and (optionally) dropped/errors; the only view into private-VM outbound.
  • Cloud Audit Logs (Admin Activity) — compute.firewalls.insert/patch/delete, compute.networks.*, compute.instances.setTags, org-policy changes — the trail for config attacks.
  • Packet Mirroring / Cloud IDS / Cloud NGFW — for deep-packet L7 detection where the L3/L4 firewall is blind.

Sample detections

Detection 1 (BigQuery SQL) — internal host performing wide horizontal port scan (lateral movement, T1021):

-- Flow logs sink table: `project.vpc_flow.compute_googleapis_com_vpc_flows`
SELECT
  jsonPayload.connection.src_ip AS src,
  COUNT(DISTINCT jsonPayload.connection.dest_ip) AS distinct_dsts,
  COUNT(DISTINCT jsonPayload.connection.dest_port) AS distinct_ports,
  COUNT(*) AS flows
FROM `project.vpc_flow.compute_googleapis_com_vpc_flows`
WHERE timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
  AND STARTS_WITH(jsonPayload.connection.src_ip, '10.')
GROUP BY src
HAVING distinct_dsts > 50 AND distinct_ports > 20
ORDER BY distinct_dsts DESC;

Detection 2 (BigQuery SQL) — large egress to a non-Google external destination (exfil, T1048):

SELECT
  jsonPayload.connection.src_ip AS src,
  jsonPayload.connection.dest_ip AS dst,
  SUM(CAST(jsonPayload.bytes_sent AS INT64)) AS bytes_out
FROM `project.vpc_flow.compute_googleapis_com_vpc_flows`
WHERE timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
  AND jsonPayload.reporter = 'SRC'
  AND NOT STARTS_WITH(jsonPayload.connection.dest_ip, '10.')
  AND jsonPayload.connection.dest_ip NOT LIKE '199.36.153.%'  -- restricted VIP
GROUP BY src, dst
HAVING bytes_out > 500 * 1024 * 1024   -- >500 MB in one hour
ORDER BY bytes_out DESC;

    protoPayload.request.allowed|contains: 'tcp'
    protoPayload.request.allowed|contains: '22'
  condition: selection
level: high

Response playbook (containment → eradication → recovery)

  1. Triage & scope. From the alert, pull the offending src_instance / SA and the rule reference from Firewall Rules Logging. Confirm blast radius via a Flow Logs query for all flows from that host in the last 24h.
  2. Contain (network). Apply a high-priority (low-number, e.g. 100) deny-all ingress+egress firewall rule targeting the compromised VM's service account or a quarantine tag — this isolates it in seconds without deleting it (preserves forensics). For a config attack, revert the malicious firewall/route change and re-associate the hierarchical policy.
  3. Contain (identity). If a metadata SSRF/token theft is suspected, disable and re-key the VM's service account and revoke active OAuth tokens; rotate any downstream credentials the SA could reach.
  4. Eradicate. Snapshot disks for forensics, then terminate the VM and rebuild from a known-good image via the pipeline. Delete (not disable) any rogue firewall rules/routes; close the ingress vector (patch the app, remove public IP).
  5. Recover. Re-provision from IaC so segmentation is restored to the reviewed state; verify with Network Connectivity Tests that only intended flows work. Confirm default-deny egress and no-external-IP org policy are intact.
  6. Post-incident. Add the missed detection to BigQuery scheduled queries / Security Command Center; add a policy-as-code guardrail (OPA/Forseti) that would have rejected the misconfig in CI; run Firewall Insights to prune newly-discovered shadowed/over-broad rules.

7. Practice Questions

Tier 1 — Fundamentals

Q1. How does a GCP VPC differ structurally from an AWS VPC?
A GCP VPC is a global resource — its subnets are regional but all belong to one global network, so VMs in different regions communicate over internal IPs on Google's backbone with no peering or gateway. An AWS VPC is regional; spanning regions requires VPC peering, Transit Gateway, or a VPN. GCP routing and firewall rules are global network properties, whereas AWS scopes route tables and security groups within the regional VPC. This means a GCP segmentation design is naturally cross-region, and you must be deliberate that "one VPC" can mean a very large trust domain.
Signal: does the candidate know the global-vs-regional distinction cold — the #1 GCP networking fact.

Q2. What are GCP's implied firewall rules and why do they matter?
Every VPC has two invisible rules at priority 65535: implied allow-all-egress and implied deny-all-ingress. They mean a VM can reach anything outbound by default but nothing can reach it inbound unless you add an allow rule. This is why exfiltration is easy by default (you must add an explicit deny-egress to stop it) and why inbound is safe by default. You can't delete implied rules, only shadow them with higher-priority explicit rules.
Signal: understands the default posture and its exfil implication.

Q3. Firewall rules are stateful — what does that mean in practice?
If a connection is allowed in one direction, the return traffic is automatically permitted regardless of rules in the other direction; GCP tracks the connection in the host firewall. So you only write a rule for the initiating direction — you do not create mirror rules. This is the key difference from AWS NACLs, which are stateless and require explicit rules for both directions (including ephemeral return ports). GCP firewall rules behave like AWS security groups in this respect.
Signal: stateful vs stateless fluency; NACL contrast.

Q4. How are firewall rules evaluated when several match?
By priority, an integer 0–65535 where lower means higher precedence. The rule with the lowest priority number that matches wins and its action is terminal. If two rules of equal priority match, a deny beats an allow. Firewall policies add levels above VPC rules (hierarchical, then network policies) evaluated first, with a goto_next action to fall through. If nothing matches, the implied rules apply.
Signal: precise on the "lower number wins" and tie-break rule — commonly gotten backwards.

Q5. What is Private Google Access and when do you need it?
It's a per-subnet setting that lets VMs without an external IP reach Google APIs and services (Cloud Storage, BigQuery, etc.) using the private.googleapis.com or restricted.googleapis.com virtual IPs, routed over Google's internal network. You need it whenever you follow a no-public-IP design but workloads still call Google APIs — otherwise those calls would require an external IP or NAT. restricted.googleapis.com additionally keeps traffic inside a VPC Service Controls perimeter.
Signal: connects "no public IP" design to how APIs still get reached.

Tier 2 — System Design / Trade-off

Q6. Design network segmentation for a 3-tier app (web, app, db) on GCP with zero-trust east-west. Walk through it.
I'd use a custom-mode VPC with three regional subnets (or one per tier) and delete all default-* rules. Each tier runs under its own service account (web-tier, app-tier, db-tier). Firewall rules target SAs, not tags: ingress to web-tier only from the load-balancer/IAP ranges on 443; ingress to app-tier only from web-tier SA on the app port; ingress to db-tier only from app-tier SA on 5432. I'd add a default-deny egress and allow only restricted.googleapis.com, the DB CIDR, and a package mirror. No external IPs (org policy); Cloud NAT for any outbound; Private Google Access for APIs. Enforce cross-cutting denies (no inbound SSH/RDP from internet) via a hierarchical firewall policy at the folder. Turn on Flow Logs + Firewall Rules Logging to BigQuery. The result: a compromised web VM can reach app only on one port, app can reach db only on one port, and nothing can exfiltrate to the internet.
Signal: end-to-end zero-trust design using SAs, default-deny, and layered policy — the core senior competency.

Q7. Shared VPC vs VPC Peering vs Private Service Connect — when do you pick each?
Shared VPC centralizes network ownership: a host project owns subnets/firewall and service projects deploy into them — pick it when you want a central network team to control segmentation for many app teams in one org. VPC Peering connects two independent VPCs (possibly different orgs) with RFC1918 reachability but is non-transitive and each side keeps its own firewall — pick it for partner/mesh connectivity where each party retains autonomy, accepting that A-peered-to-B and B-peered-to-C does not let A reach C. Private Service Connect exposes a producer service (a Google API bundle, a managed service, or a partner SaaS) as a consumer-side private IP with no range-overlap constraints and no peering — pick it for consuming a service privately, especially across orgs or where CIDRs would collide. I default to Shared VPC for internal org structure and PSC for service consumption.
Signal: knows the three are different tools for different problems, and the non-transitivity gotcha.

Q8. How do you enforce "no VM may have a public IP" across an entire org, and what breaks?
Set the org policy constraints/compute.vmExternalIpAccess to deny-all at the organization node; it's a listConstraint, so you can allow-list specific VMs by exception if truly needed. What breaks: any VM that assumed internet egress via its own external IP, and inbound access patterns that relied on public IPs (replace with a load balancer + IAP for admin access). You backfill with Cloud NAT for egress and Private Google Access for API calls. You also want constraints/compute.skipDefaultNetworkCreation and constraints/sql.restrictPublicIp alongside it. The combination makes public exposure structurally impossible rather than relying on people to remember.
Signal: knows the exact constraint and the compensating controls needed so it doesn't break everything.

Q9. Why prefer service-account-based firewall targeting over network tags, and what's the catch?
Network tags are unauthenticated strings — anyone with compute.instances.setTags (bundled into common roles like instanceAdmin) can add the tag that a firewall rule allow-lists, self-granting access and silently defeating segmentation. Service accounts require roles/iam.serviceAccountUser to attach to a VM, a much more tightly controlled permission, so SA-based rules are true identity-based microsegmentation. The catch: a single firewall rule can target tags or SAs but not both, SA-per-tier requires disciplined identity design, and you must lock down serviceAccountUser for the model to hold. I use SAs for anything security-relevant and tags only for coarse routing.
Signal: understands the trust difference and the IAM permission that underpins it.

Q10. Base firewall rules are L3/L4 only. How do you get FQDN-based or L7 egress control?
The VPC firewall can't match on domain names or inspect application content — a deny-egress allowlist has to be by CIDR, which is brittle for SaaS endpoints on rotating IPs. To get FQDN/L7 control you use one of: Cloud NGFW Enterprise (with the intrusion-prevention/FQDN objects), a Secure Web Gateway / forward proxy (e.g. Squid or a managed SWG) that VMs route egress through and which enforces domain allowlists and does TLS inspection, or Cloud IDS / Packet Mirroring for detection (not blocking). I typically route egress through a proxy in a dedicated egress subnet, allowlist FQDNs there, and keep the base firewall as the coarse CIDR backstop. VPC Service Controls complements this by stopping API-level exfil to unauthorized projects.
Signal: knows the L3/L4 limitation and the concrete ways to add L7.

Tier 3 — Expert / Deep-dive

Q11. Trace exactly how a packet to a VM is evaluated across all firewall layers, including goto_next.
For an ingress packet at the VM's NIC, the host firewall evaluates in order: (1) Hierarchical firewall policies attached at org then folder, top-down, each level by priority; an allow/deny is terminal, goto_next drops to the next level. (2) Global/network firewall policies attached to the VPC, again by priority with the same terminal-vs-goto_next semantics. (3) VPC firewall rules in the project, by priority (lowest number wins, deny beats allow on ties). (4) If nothing matched, the implied rules (deny-ingress / allow-egress). The first terminal action anywhere in this chain decides. This is why a folder-level hierarchical deny can't be overridden by a project admin's VPC allow — the hierarchical layer is evaluated first and its deny is terminal before the VPC rule is ever consulted.
Signal: can articulate the full precedence chain and the security consequence of ordering.

Q12. A compromised VM is exfiltrating despite a default-deny egress rule. What are the likely bypasses and how do you close them?
Likely causes: (a) an allow-egress rule at a lower priority number shadows the deny — check for a broad allow all egress to 0.0.0.0/0 left from testing; fix the priority scheme and delete it. (b) Exfil is going to an allowed destination you trusted — e.g., data staged into an attacker-controlled Cloud Storage bucket reachable via the restricted.googleapis.com VIP; close with VPC Service Controls so only in-perimeter projects are reachable. (c) DNS exfil over an allowed resolver — inspect Cloud DNS logs and route DNS through a controlled resolver. (d) Egress via a peered VPC or NAT you didn't account for — audit routes and NAT logging. (e) The rule targets a tag/SA the VM doesn't have, so it never applied — verify target scoping. I'd confirm with Firewall Rules Logging which rule is actually matching the exfil flow, then tighten.
Signal: systematic reasoning about why a control isn't working — priority shadowing, trusted-destination abuse, VPC-SC, DNS.

Q13. How do VPC firewall rules interact with VPC Service Controls, and why do you need both?
They defend different planes. Firewall rules (and routes/NAT) control the network data plane — which IP:port can talk to which. VPC Service Controls creates a service perimeter around Google-managed APIs (Cloud Storage, BigQuery, etc.) so that even a valid credential can't read/write a resource from outside the perimeter — it stops API-level exfiltration that firewalls can't see because the traffic goes to Google's public API endpoints. A stolen SA token used from an attacker's laptop is blocked by VPC-SC (wrong perimeter) even though no firewall rule was involved. You need both: firewall stops network reachability; VPC-SC stops credential-based data exfil to unauthorized projects. Using restricted.googleapis.com with Private Google Access is what ties a private VM into the VPC-SC perimeter.
Signal: distinguishes network-plane vs API-plane controls — a nuanced GCP-specific insight.

Q14. Design detection for lateral movement in a large flat legacy VPC you've inherited and can't immediately re-architect.
First, turn on VPC Flow Logs on every subnet (start 0.5 sampling, include-all metadata) and Firewall Rules Logging on the permissive internal rules, sink to BigQuery. Build detections for the east-west patterns segmentation would normally prevent: (1) horizontal scans — one internal src hitting many distinct dst IPs/ports in a short window (the Detection 1 query). (2) new service talk — flows between tiers that have never communicated before, via a baseline table. (3) admin-port access — internal flows to 22/3389/6379/9200 from non-bastion sources. (4) beaconing — regular-interval small flows to a single external dst (NAT logs). In parallel, run Firewall Insights and Network Connectivity Tests to map what's actually reachable, then start carving the flat network with SA-targeted rules on the highest-value tiers first. Detection buys time while you segment.
Signal: pragmatic — detect first when you can't re-architect, with concrete queries and a segmentation roadmap.

Q15. Explain the metadata SSRF → token theft → exfil kill chain on GCP and every independent control that breaks it.
Chain: an app-layer SSRF (T1190) coerces a VM to fetch 169.254.169.254/computeMetadata/v1/instance/service-accounts/default/token; if legacy metadata endpoints are on, no Metadata-Flavor header is needed and the attacker gets an OAuth token (T1552.005); they call Google APIs as the SA and exfiltrate to their own project/bucket (T1048). Independent breaks: (1) Least-privilege SA — a token that can't read the sensitive bucket limits impact. (2) Metadata concealment / GKE Workload Identity / disable legacy metadata — the token is no longer trivially reachable via SSRF. (3) Default-deny egress — the exfil HTTP call to the attacker's server is blocked because only restricted.googleapis.com + DB are allowed out. (4) VPC Service Controls — even a valid token can't read the bucket from outside the perimeter. (5) Firewall to link-local — deny egress to 169.254.169.254 from workloads that never need metadata. Any one of these breaks the chain; defense-in-depth stacks them so no single miss is fatal.
Signal: connects app-layer, identity, and network controls into one defense-in-depth story — the senior "think in kill chains" signal.

9. Key Takeaways

  • Global VPC is the defining GCP fact: one VPC spans all regions with no peering; subnets are the only regional object. This makes "one VPC" a potentially huge trust domain — segment deliberately.
  • Firewall rules are stateful, per-NIC, priority-ordered (lowest number wins, deny wins ties), with implied allow-egress + deny-ingress. You must add an explicit deny-egress to stop exfil.
  • Delete the defaults. default-allow-ssh/rdp/internal/icmp and auto-mode VPCs are the top real-world exposure; always start custom-mode and remove them.
  • Target service accounts, not tags. Tags are unauthenticated metadata; SA-targeting is identity-anchored microsegmentation and needs iam.serviceAccountUser.
  • Hierarchical firewall policies are the org guardrail project owners can't override; use goto_next to enforce non-negotiables while delegating the rest.
  • Private-by-default = org policy no-external-IP + Cloud NAT (egress-only) + Private Google Access / PSC. This removes the biggest exposure class structurally.
  • Firewall (network plane) and VPC Service Controls (API plane) are complementary — you need both to stop network reach and credential-based API exfil.
  • Nothing is detected without logging: Flow Logs and Firewall Rules Logging are off by default — enable them and sink to BigQuery for lateral-movement and exfil detection.
  • Peering is non-transitive and not a trust boundary — firewall at both ends; use Shared VPC for central control and PSC for private service consumption.
  • Base firewall is L3/L4 only — for FQDN/L7 egress control add Cloud NGFW or a forward proxy; don't assume CIDR allowlists cover SaaS endpoints.
