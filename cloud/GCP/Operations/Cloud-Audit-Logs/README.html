GCP Cloud Audit Logs — The Authoritative Security Telemetry for Google Cloud

Category: Operations / Security Telemetry
GCP Service: Cloud Logging (Cloud Audit Logs)
Key Docs: Cloud Audit Logs overview, AuditLog proto, Logging query language
Cross-refs: Detection Engineering · IDS and SIEM · Anomaly Detection · AWS CloudTrail

1. Overview

Cloud Audit Logs are Google Cloud's canonical, tamper-evident record of "who did what, where, and when" across the platform. Every Google Cloud service that participates in audit logging emits structured LogEntry records whose protoPayload is an AuditLog message — a strongly-typed schema that captures the caller's identity, the API method invoked, the target resource, and the source network metadata. Because these logs are generated by the Google-operated control plane rather than by the customer's workload, they are the single most trustworthy signal source for detecting privilege escalation, data exfiltration, persistence, and account takeover in GCP. At scale — an organization with thousands of projects and billions of API calls per day — audit logs become the substrate for the entire detection program: they feed SIEMs, drive anomaly baselines, and anchor every incident-response timeline. If you get audit logging wrong (Data Access logs disabled, no org-level sink, short retention, a mutable sink bucket), you are effectively blind to the most dangerous class of attacks, and no amount of downstream detection engineering can recover the missing evidence.

TL;DR
- There are four audit log types: Admin Activity, Data Access, System Event, and Policy Denied. Admin Activity, System Event, and Policy Denied are always on and free; Data Access logs are OFF by default (except BigQuery) and must be explicitly enabled — this is the #1 audit-logging gap.
- The payload is protoPayload of type AuditLog; the four fields you must know cold are authenticationInfo.principalEmail (who), methodName (what), resourceName (target), and requestMetadata.callerIp (from where).
- Logs route through the Log Router to sinks; production requires an aggregated org-level sink to a separate, locked security project exporting to BigQuery (analytics), Cloud Storage (cheap immutable retention with bucket lock), and Pub/Sub (SIEM streaming).
- Admin Activity lands in the _Required bucket with 400-day fixed, non-configurable, non-deletable retention; everything else lands in _Default (30 days by default). Sink-owned buckets and Bucket Lock provide tamper-resistance.
- You cannot fully disable Admin Activity logging, but a sufficiently privileged attacker can disable future Data Access logs, tamper with sinks, or delete export buckets — so protect the Log Router config and export destinations with org policy and least privilege.

2. How It Works

2.1 Core Mechanics

Cloud Audit Logs are a special category of entries inside Cloud Logging. A GCP service (the "producer") calls the internal audit-logging API as a side effect of servicing a request; the resulting entry flows into the Log Router, which evaluates inclusion/exclusion filters and fans the entry out to one or more log buckets and sinks. The producer path is out-of-band from the API response — the caller's request succeeds or fails independently of whether the audit write completes, and the write is best-effort-durable on Google's side, not the customer's.

Concept Description
Admin Activity log Control-plane writes: config/metadata changes and API calls that create, modify, or delete resources (SetIamPolicy, compute.instances.insert). Always on, free, cannot be disabled.
Data Access log Reads of config/metadata and data-plane reads/writes of user data (storage.objects.get, bigquery.jobs.query). Off by default (except BigQuery), billable, high volume.
System Event log Google-initiated actions not driven by a user — e.g. live migration, automatic instance preemption. Always on, free.
Policy Denied log Records when a service denies access due to a security policy (VPC Service Controls, org policy, IAM). Always on; billable but excludable.
LogEntry The envelope: logName, resource, timestamp, severity, insertId, labels, and the payload.
protoPayload (AuditLog) The audit-specific payload with authenticationInfo, authorizationInfo, methodName, resourceName, requestMetadata, serviceName, request, response, status.
Log Router The routing engine that applies inclusion/exclusion filters and dispatches entries to buckets and sinks.
Log bucket Storage container for logs. Special buckets _Required and _Default exist per project/folder/org.
Sink A routing rule: filter + destination (a bucket, BigQuery dataset, Cloud Storage bucket, Pub/Sub topic, or another project's bucket).
Aggregated sink An org- or folder-level sink with --include-children that captures logs from all descendant projects in one place.
_Required bucket Holds Admin Activity, System Event, and (some) denied logs. 400-day retention, fixed, non-editable, no-cost.
_Default bucket Holds everything else routed by the default sink; 30-day retention by default, configurable 1–3650 days.

The decision logic for a single audited API call is: (1) the service determines the log category based on the operation; (2) if Data Access, it checks whether Data Access logging is enabled for that service/permission type (ADMIN_READ, DATA_READ, DATA_WRITE) in the effective IAM audit config; (3) the entry is constructed with the AuditLog payload; (4) the Log Router evaluates each sink's filter; (5) matching entries are written to the destination(s). Crucially, whether a Data Access entry is even generated is gated by the IAM auditConfigs block — if DATA_READ isn't enabled for storage.googleapis.com, a storage.objects.get produces no log at all, not a suppressed one.

2.2 Concrete Configuration

Enable Data Access logging org-wide (the single most important audit change). Data Access logs are off by default because they can be extremely high volume. You enable them via the IAM policy's auditConfigs. First read the current org policy, then patch it:

# Capture the current org-level IAM policy so you can diff and roll back
gcloud organizations get-iam-policy 123456789012 --format=json > org-policy.json

Add an auditConfigs block. ADMIN_READ is cheap and high-signal; DATA_READ/DATA_WRITE are expensive but essential for storage and BigQuery. A pragmatic production config enables all three for the crown-jewel services and ADMIN_READ everywhere:

{
  "auditConfigs": [
    {
      "service": "allServices",
      "auditLogConfigs": [
        { "logType": "ADMIN_READ" }
      ]
    },
    {
      "service": "storage.googleapis.com",
      "auditLogConfigs": [
        { "logType": "ADMIN_READ" },
        { "logType": "DATA_READ" },
        { "logType": "DATA_WRITE" }
      ]
    },
    {
      "service": "cloudkms.googleapis.com",
      "auditLogConfigs": [
        { "logType": "DATA_READ" },
        { "logType": "DATA_WRITE" }
      ]
    }
  ]
}

# Apply the merged policy back. Use etag from the GET to avoid a lost update.
gcloud organizations set-iam-policy 123456789012 org-policy.json

allServices with ADMIN_READ gives you read-of-config visibility everywhere at modest cost; carving out storage and cloudkms with DATA_READ/DATA_WRITE catches object exfiltration and key-material access. You can exempt specific high-noise principals per log type with exemptedMembers inside an auditLogConfig — use this sparingly, because an attacker who lands on an exempted service account becomes invisible.

Read audit logs with the Logging query language. The Logging query language is the filter syntax used by gcloud logging read, the Logs Explorer, and log-based metrics:

# All Admin Activity entries for a project in the last hour, newest first
gcloud logging read \
  'logName="projects/prod-app-42/logs/cloudaudit.googleapis.com%2Factivity"
   AND timestamp>="2026-07-06T00:00:00Z"' \
  --project=prod-app-42 --order=desc --limit=50 --format=json

The logName values are the four canonical streams (URL-encoded / as %2F):

Log type logName suffix (after cloudaudit.googleapis.com%2F)
Admin Activity activity
Data Access data_access
System Event system_event
Policy Denied policy

Create an aggregated org-level sink to BigQuery so every project's logs land in one analytics dataset:

gcloud logging sinks create org-audit-to-bq \
  bigquery.googleapis.com/projects/sec-logging-prod/datasets/org_audit_logs \
  --organization=123456789012 \
  --include-children \
  --log-filter='logName:"cloudaudit.googleapis.com"' \
  --use-partitioned-tables

--include-children is what makes it aggregated — it captures logs from every folder and project under the org node, not just the org resource itself. The command prints a writer identity (a service account like serviceAccount:o123456789012-...@gcp-sa-logging.iam.gserviceaccount.com); you must grant that identity write access on the destination, or the sink silently drops everything:

# Grant the sink's writer identity permission to write to the BigQuery dataset
bq add-iam-policy-binding \
  --member="serviceAccount:o123456789012-9f8e@gcp-sa-logging.iam.gserviceaccount.com" \
  --role="roles/bigquery.dataEditor" \
  sec-logging-prod:org_audit_logs

Create a locked, immutable Cloud Storage sink for long-term forensic retention:

# 1. Create a dedicated bucket in the security project with uniform access
gsutil mb -p sec-logging-prod -b on -l us gs://org-audit-archive-immutable

# 2. Apply a retention policy and LOCK it — after lock, objects cannot be deleted
#    or overwritten until the retention period elapses, even by project owners
gsutil retention set 2555d gs://org-audit-archive-immutable   # 7 years
gsutil retention lock gs://org-audit-archive-immutable

# 3. Point an aggregated sink at it
gcloud logging sinks create org-audit-to-gcs \
  storage.googleapis.com/org-audit-archive-immutable \
  --organization=123456789012 --include-children \
  --log-filter='logName:"cloudaudit.googleapis.com/activity" OR
                logName:"cloudaudit.googleapis.com/data_access"'

Bucket Lock is the technical control that converts "logs in a bucket" into "WORM (write-once-read-many) forensic evidence" — once locked, the retention period can only be increased, and no principal, regardless of IAM, can delete objects before it expires.

Stream to a SIEM via Pub/Sub (Splunk, Chronicle/Google SecOps, or a self-hosted stack described in IDS-SIEM):

gcloud pubsub topics create audit-siem-stream --project=sec-logging-prod

gcloud logging sinks create org-audit-to-siem \
  pubsub.googleapis.com/projects/sec-logging-prod/topics/audit-siem-stream \
  --organization=123456789012 --include-children \
  --log-filter='logName:"cloudaudit.googleapis.com" AND
                (protoPayload.methodName:"SetIamPolicy" OR
                 protoPayload.methodName:"keys.create" OR
                 severity>=NOTICE)'

Extend _Default retention so short-lived Data Access logs survive an intrusion investigation window:

gcloud logging buckets update _Default \
  --location=global --project=sec-logging-prod --retention-days=365

Case study 1 — The "Data Access off" blind spot (the GCP twin of Capital One). In the 2019 Capital One breach, an SSRF vulnerability let an attacker reach the instance metadata service, steal an IAM role's temporary credentials, and enumerate/exfiltrate ~100M records from S3. The forensic difficulty was compounded because S3 data events were not enabled in CloudTrail, so object-level reads were not recorded. The exact same failure mode exists in GCP: because Data Access logs are off by default, an attacker who compromises a workload's service account and runs storage.objects.get across a sensitive bucket generates zero audit entries. The Admin Activity log would show the initial IAM changes but not a single object read. The control that stops this from becoming an unbounded, unprovable incident is enabling DATA_READ on storage.googleapis.com before the breach — you cannot retroactively generate logs that were never written.

Case study 2 — GhostToken and the limits of console views (2023). Researchers at Astrix showed that a malicious OAuth application in Google Cloud could be put into a "pending deletion" state that hid it from the Google Account's app-management page while its refresh token remained valid, granting attackers durable, invisible access to a victim's Google services. The lesson for a detection engineer is that management-console inventories are not audit ground truth — they can be manipulated or lag reality. The authoritative record of the authorization grant lives in the audit logs (the SetIamPolicy, OAuth grant, and token-issuance events). A detection built on "diff the console app list" would have missed it; a detection built on "alert on any new third-party grant in the audit stream" would have caught the original authorization. Immutable, exported audit logs are the only source you can trust when the primary UI is compromised.

Case study 3 — Credential theft and anomalous callerIp (Legion, 2023). The Legion toolkit and similar credential harvesters scrape leaked .env files and CI logs for cloud API keys and service-account JSON, then abuse them from attacker-controlled infrastructure. Because a stolen GCP service-account key is a bearer credential, the only signal distinguishing legitimate from malicious use is context: requestMetadata.callerIp suddenly resolving to a hosting-provider ASN in a new country, principalEmail for a service account that normally only calls from within the VPC now calling public APIs, and a burst of keys.create for persistence. Enabling Data Access + ADMIN_READ and streaming to a SIEM with geo/ASN enrichment (see Anomaly Detection) turns an invisible key-abuse campaign into a same-hour alert.

Top 5 in depth

1. Enable Data Access logging for crown-jewel services. Technically, this is a change to the IAM policy's auditConfigs array at the org (or folder/project) node, specifying logType values ADMIN_READ, DATA_READ, and DATA_WRITE per service. Effective config is the union of org, folder, and project settings — a project cannot reduce what the org enables, only add. The concrete implementation is the set-iam-policy flow in §2.2. The trade-off is volume and cost: on a busy BigQuery or GCS estate, DATA_READ can generate terabytes of logs per day. The production answer is selective enablement — always-on ADMIN_READ for everything (cheap, high signal because it catches enumeration and config reads), full DATA_READ/WRITE only for Cloud Storage, KMS, Secret Manager, BigQuery, and Cloud SQL, and exclusion filters at the Log Router to drop known-benign high-frequency reads (e.g. a monitoring SA polling the same object). The limit: you must decide before an incident; there is no retroactive capture.

2. Aggregated org-level sink to a segregated security project. An aggregated sink with --include-children at the org node captures every current and future project automatically — the key property is that a rogue or shadow project spun up by an attacker is monitored the instant it emits a log, with no per-project onboarding. Implementation is the gcloud logging sinks create --organization ... --include-children command plus the writer-identity IAM grant on the destination. The destination lives in a separate, tightly-locked "security logging" project whose IAM is disjoint from workload projects, so compromising an app project does not grant access to the evidence. Trade-offs: the writer identity must be granted on the destination or logs silently drop; and centralizing all logs makes the security project a high-value target — protect it with VPC Service Controls and minimal human access.

3. Immutable retention with Bucket Lock (WORM). Bucket Lock applies a retention policy to a Cloud Storage bucket and then locks it, after which objects cannot be deleted or overwritten until the retention period elapses, and the period can only be lengthened — not even a project owner or org admin can bypass it. This converts exported audit logs into forensically-sound WORM evidence that survives an insider or a fully-compromised admin who tries to destroy history (threats #4 and #20). Implementation is gsutil retention set + gsutil retention lock. The trade-off is irreversibility: if you lock 7 years by mistake you pay for 7 years of storage with no early deletion; and locked buckets complicate legitimate data-lifecycle/GDPR erasure, so scope them to the audit archive only.

4. Detect-and-block on the logging pipeline itself (meta-audit). The most dangerous attacker move is to blind the defenders: disable Data Access logging (SetIamPolicy on auditConfigs), delete a sink, or narrow a filter. Because these are themselves control-plane operations, they generate Admin Activity logs that cannot be suppressed. The control is a high-priority, low-threshold detection on methodName values SetIamPolicy (when the request touches auditConfigs), google.logging.v2.ConfigServiceV2.DeleteSink, UpdateSink, and DeleteBucket, combined with least-privilege on roles/logging.admin and roles/iam.securityAdmin. Trade-off: SetIamPolicy is high-volume in some orgs, so you filter specifically for changes to the audit-config or sink resources rather than all IAM changes to keep signal-to-noise high.

5. Org policy guardrails that prevent the misconfiguration outright. Detection is reactive; org policy constraints are preventive. iam.disableServiceAccountKeyCreation blocks the persistence technique (#5) at the source, forcing workloads onto Workload Identity Federation / attached service accounts instead of exportable keys. storage.publicAccessPrevention makes it structurally impossible to bind a bucket to allUsers/allAuthenticatedUsers (#7). iam.allowedPolicyMemberDomains (domain-restricted sharing) stops grants to external identities. These are enforced at the org node and inherited, so a single well-placed constraint protects thousands of projects. Trade-off: they can break legitimate edge cases (a genuinely public asset, a partner integration), so they need documented exceptions via policy hierarchy — and they complement, never replace, the audit-log detections, because a policy can be changed by a sufficiently privileged attacker (which itself is audited).

6. Detection & Response

What to log / telemetry sources

  • Admin Activity (logName …/activity) — always the baseline; watch SetIamPolicy, *.create, *.delete, keys.create.
  • Data Access (…/data_access) — enable for GCS, BigQuery, KMS, Secret Manager, Cloud SQL; watch storage.objects.get, bigquery.jobs.query, google.cloud.secretmanager…AccessSecretVersion, cloudkms…Decrypt.
  • Policy Denied (…/policy) — VPC-SC and org-policy denials are a strong recon/exfil-attempt signal.
  • Key fields for every detection: protoPayload.authenticationInfo.principalEmail, protoPayload.methodName, protoPayload.resourceName, protoPayload.requestMetadata.callerIp, protoPayload.authorizationInfo[].granted, protoPayload.request, resource.labels.project_id, severity, timestamp, insertId (dedupe).

Sample detections

Detection 1 — New service-account key creation (persistence, T1098.001). Logging query language for a Logs Explorer saved search / log-based alert:

logName=~"cloudaudit.googleapis.com%2Factivity"
protoPayload.methodName="google.iam.admin.v1.CreateServiceAccountKey"
protoPayload.authorizationInfo.granted=true
-- Exclude your CI/CD provisioner service account to cut noise:
-protoPayload.authenticationInfo.principalEmail="terraform-cicd@sec-logging-prod.iam.gserviceaccount.com"

Equivalent SQL over the BigQuery export (partitioned tables), scheduled hourly:

SELECT
  timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail   AS actor,
  protopayload_auditlog.resourceName                        AS target_sa,
  protopayload_auditlog.requestMetadata.callerIp            AS src_ip,
  resource.labels.project_id                                AS project
FROM `sec-logging-prod.org_audit_logs.cloudaudit_googleapis_com_activity_*`
WHERE _TABLE_SUFFIX >= FORMAT_DATE('%Y%m%d', DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY))
  AND protopayload_auditlog.methodName = 'google.iam.admin.v1.CreateServiceAccountKey'
  AND protopayload_auditlog.authenticationInfo.principalEmail
        NOT IN ('terraform-cicd@sec-logging-prod.iam.gserviceaccount.com')
ORDER BY timestamp DESC;

Detection 2 — IAM policy change granting Owner (privilege escalation, T1098). The granted role lives inside the SetIamPolicy request bindings:

SELECT
  timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS actor,
  protopayload_auditlog.resourceName                      AS resource,
  binding.role                                            AS granted_role,
  member                                                  AS grantee,
  protopayload_auditlog.requestMetadata.callerIp          AS src_ip
FROM `sec-logging-prod.org_audit_logs.cloudaudit_googleapis_com_activity_*`,
  UNNEST(protopayload_auditlog.servicedata_v1_iam.policyDelta.bindingDeltas) AS binding,
  UNNEST([binding.member]) AS member
WHERE _TABLE_SUFFIX >= FORMAT_DATE('%Y%m%d', DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY))
  AND protopayload_auditlog.methodName LIKE '%SetIamPolicy'
  AND binding.action = 'ADD'
  AND binding.role IN ('roles/owner', 'roles/editor', 'roles/iam.securityAdmin')
ORDER BY timestamp DESC;

Detection 3 — Public bucket exposure (data exposure, T1530). Fire when a storage.setIamPolicy adds allUsers or allAuthenticatedUsers. Logging query language:

logName=~"cloudaudit.googleapis.com%2Factivity"
protoPayload.serviceName="storage.googleapis.com"
protoPayload.methodName="storage.setIamPolicy"
(protoPayload.request.policy.bindings.members="allUsers" OR
 protoPayload.request.policy.bindings.members="allAuthenticatedUsers")

Detection 4 — Audit-config tampering (defense evasion, T1562.008). The meta-detection that watches the watchers:

logName=~"cloudaudit.googleapis.com%2Factivity"
protoPayload.methodName="SetIamPolicy"
protoPayload.serviceData.policyDelta.auditConfigDeltas:*

Detection 5 — Sink deletion / narrowing.

logName=~"cloudaudit.googleapis.com%2Factivity"
(protoPayload.methodName="google.logging.v2.ConfigServiceV2.DeleteSink" OR
 protoPayload.methodName="google.logging.v2.ConfigServiceV2.UpdateSink")

Wire these into log-based alerting or a SIEM correlation layer as described in IDS-SIEM; enrich callerIp with ASN/geo and baseline per-principal behavior per Anomaly Detection.

Response playbook

  1. Triage / validate. Confirm the alert against raw audit entries by insertId (dedupe) and cross-check authenticationInfo.principalEmail, callerIp, and authorizationInfo.granted=true. Rule out known automation before escalating.
  2. Scope. Query the BigQuery export for all activity by the offending principalEmail and callerIp over the incident window — every methodName, every resourceName touched. Pull Data Access logs to enumerate what data was read.
  3. Contain. Disable the offending service-account key (gcloud iam service-accounts keys disable), revoke OAuth grants, and remove the malicious IAM binding. If a human identity, force session revocation and reset credentials. If logging itself was tampered, re-enable Data Access auditConfigs and recreate/repair the sink first — restore visibility before hunting.
  4. Preserve evidence. Snapshot relevant logs to the locked GCS archive; confirm Bucket Lock retention covers the window. Export the BigQuery result set of the actor's activity to an evidence dataset with restricted IAM.
  5. Eradicate. Remove attacker persistence — rogue keys, added members, malicious sinks pointing to external projects, scheduled functions/jobs created during the window.
  6. Recover. Rotate any credentials the attacker could have read (Secret Manager pulls, KMS decrypts). Rebuild affected workloads from known-good IaC.
  7. Post-incident. Add/tune detections for the observed technique, close the misconfiguration that enabled it (e.g. enable disableServiceAccountKeyCreation), and record the timeline sourced entirely from immutable audit logs.

7. Practice Questions

Tier 1 — Fundamentals

Q1. Name the four Cloud Audit Log types and which are on by default. Admin Activity, Data Access, System Event, and Policy Denied. Admin Activity, System Event, and Policy Denied are always on and free (Admin Activity cannot be disabled). Data Access logs are off by default — with the historical exception of BigQuery, which logs data access automatically — and must be explicitly enabled via IAM auditConfigs because they can be extremely high volume and are billable. Signal: knows the single biggest audit gap in GCP.

Q2. What is the payload type of an audit LogEntry and what are its most important fields? The protoPayload is an AuditLog message. The critical fields are authenticationInfo.principalEmail (who), methodName (what API), resourceName (target), and requestMetadata.callerIp (source IP); plus authorizationInfo[].granted (was it allowed), serviceName, and the request/response bodies for full context. Signal: can navigate a raw entry without hand-holding.

Q3. Where do Admin Activity logs go and how long are they kept? They route to the special _Required log bucket, which has a fixed 400-day retention that cannot be changed or shortened and incurs no charge; the logs also cannot be deleted before that. Everything else defaults to the _Default bucket at 30 days (configurable 1–3650). Signal: knows retention defaults matter for IR.

Q4. Why must you enable Data Access logs, and what's the trade-off? Without them, reads of user data — object gets, secret accesses, KMS decrypts, BigQuery queries — produce no audit entry at all, so data exfiltration is invisible (the Capital One failure mode). The trade-off is volume and cost, which can be terabytes/day; you manage it by enabling ADMIN_READ broadly but DATA_READ/WRITE only on crown-jewel services, plus exclusion filters. Signal: balances security and cost like an operator.

Q5. What is a log sink and what are common destinations? A sink is a Log Router rule pairing a filter with a destination. Destinations are log buckets, BigQuery datasets (analytics/hunting), Cloud Storage buckets (cheap, lockable long-term retention), Pub/Sub topics (streaming to SIEM), and other projects' buckets. Each sink has a writer identity that must be granted access on the destination. Signal: understands the routing model end to end.

Tier 2 — System design / trade-offs

Q6. Design an org-wide audit-logging architecture for a 3,000-project enterprise. Create a dedicated security logging project with IAM disjoint from workloads. At the org node, create an aggregated sink (--include-children) so every current and future project is captured automatically. Fan out to three destinations: BigQuery (partitioned tables) for hunting and scheduled detections, a Cloud Storage bucket with Bucket Lock (7-year WORM) for forensic retention, and Pub/Sub streaming to the SIEM for near-real-time alerting. Enable Data Access ADMIN_READ org-wide and DATA_READ/WRITE on GCS/KMS/Secret Manager/BigQuery/Cloud SQL. Protect the pipeline with org-policy constraints (disableServiceAccountKeyCreation, publicAccessPrevention) and VPC-SC around the security project. Add meta-detections on auditConfig/sink changes. Extend _Default retention to 365 days. Signal: production-grade centralization, immutability, and cost awareness.

Q7. How do you protect audit logs from a fully-compromised project owner? Segregate: the sink destinations live in a separate security project the workload owner has no IAM on. Make the archive immutable with Bucket Lock so even org admins cannot delete before retention expires. Use an aggregated org sink so the owner cannot stop export by deleting their project's sink. Add meta-detections on SetIamPolicy touching auditConfigs, DeleteSink, and DeleteBucket — these control-plane actions always generate un-suppressible Admin Activity logs. Enforce least privilege on logging.admin/iam.securityAdmin. The principle: the attacker can act, but every action leaves an immutable trace they cannot reach. Signal: threat-models the defender's own infrastructure.

Q8. Data Access logs are costing you a fortune. How do you cut cost without going blind? Keep ADMIN_READ everywhere (cheap, catches enumeration and config reads). Restrict DATA_READ/WRITE to genuine crown jewels rather than allServices. Add exclusion filters at the Log Router for known-benign high-frequency principals (monitoring/health-check service accounts hitting the same resources) using exemptedMembers sparingly or filter-based exclusion. Sample non-critical Data Access logs. Route only high-value method names to the expensive SIEM tier while sending the full firehose to cheap GCS. Never disable Data Access on storage/KMS/secrets. Signal: cost engineering without sacrificing the important signals.

Q9. Compare exporting to BigQuery vs Cloud Storage vs Pub/Sub — when each? BigQuery: structured, queryable, ideal for scheduled detections and interactive hunting; costs scale with query/storage and it's not WORM. Cloud Storage: cheapest at scale, supports Bucket Lock for immutable multi-year forensic retention, but not directly queryable (needs external tables/loads). Pub/Sub: real-time streaming for a SIEM/Chronicle with lowest detection latency, but it's transport, not storage. Production uses all three from the same aggregated sink — Pub/Sub for speed, BigQuery for analytics, GCS for immutable retention. Signal: understands the latency/cost/queryability triangle.

Q10. How do you make detections resilient to attacker attempts to blind you? Rely on the fact that logging-tampering actions are themselves control-plane operations that emit un-suppressible Admin Activity logs. Build low-threshold, high-priority meta-detections on auditConfig deltas, sink delete/update, and bucket delete. Stream to an external SIEM so evidence leaves GCP immediately (an attacker who later deletes exports can't recall what already left). Use immutable WORM archives. Enforce that Log Router config changes only happen through reviewed IaC, and alert on any out-of-band change. Defense in depth: prevention (org policy), immutability (Bucket Lock), and detection (meta-audit). Signal: assumes breach and defends the detection layer itself.

Tier 3 — Expert / deep-dive

Q11. An attacker with a stolen service-account key exfiltrates data from Cloud Storage but you see nothing in the logs. Diagnose. Almost certainly Data Access logs were never enabled for storage.googleapis.com, so storage.objects.get produced no entries — the objects reads are simply not recorded, and you cannot reconstruct them retroactively. Admin Activity would still show the key's creation and any IAM changes, and Policy Denied would show blocked attempts, but the successful reads are gone. Secondary possibilities: the principal is in exemptedMembers for DATA_READ, or a sink exclusion filter dropped them, or the SA acted in a project outside your (non-aggregated) sink coverage. The fix is preventive — enable DATA_READ on storage before the next incident — and the lesson is that missing logs cannot be generated after the fact. Signal: understands the default-off trap and its irreversibility.

Q12. Walk through exactly how you'd detect the GhostToken / OAuth-persistence class of attack. The console app inventory is untrustworthy because the app hides itself, so you must detect from the immutable audit stream at the moment of the authorization grant, not from state. Alert on new third-party OAuth grants and token issuance events in the audit logs, correlate principalEmail with unusual grantees, and watch for SetIamPolicy adding external or service identities. Because the malicious grant's authorization event is logged even though the app later hides, an audit-stream-based detection catches the original action. Reinforce with allowedPolicyMemberDomains to block external grantees and periodic reconciliation of granted-vs-expected third-party access sourced from logs, not the UI. Signal: knows logs beat console state when the UI is adversarial.

Q13. Explain the difference between request-body fields and requestMetadata/server fields for forensic trust. The protoPayload.request and response bodies reflect what the client sent and what the service returned — useful for context but partially client-influenced. requestMetadata (including callerIp, callerSuppliedUserAgent) and top-level LogEntry fields (timestamp, insertId, authenticationInfo.principalEmail) are populated by Google's control plane and are the trustworthy forensic anchors. For timelines, trust the server timestamp over any client-supplied time to avoid timestomp-style confusion (T1070.006). insertId is the dedupe key when the same entry is delivered multiple times. Signal: knows which fields survive an adversary's influence.

Q14. How does the effective audit configuration resolve across org, folder, and project, and how does an attacker exploit it? The effective auditConfigs is the union across the resource hierarchy — a project can add logging beyond what the org enables but cannot reduce it. exemptedMembers can suppress logging for specific principals per log type. An attacker with iam.setIamPolicy at a level they control tries either to add themselves/their SA to exemptedMembers (making their reads invisible) or, at the org level, to remove DATA_READ entirely. Both are themselves SetIamPolicy operations recorded in Admin Activity via auditConfigDeltas, so the meta-detection in §6 (Detection 4) catches them. The defense is least-privilege on setIamPolicy plus alerting on any audit-config delta. Signal: deep grasp of policy inheritance and its abuse.

Q15. Design detections for the full cloud-persistence kill chain in GCP audit logs. Chain the stages: (a) Initial access / credential use — callerIp from a new ASN/geo for a given principalEmail, baselined via anomaly detection. (b) Persistence — CreateServiceAccountKey, new IAM bindings to external members, new OAuth grants. (c) Privilege escalation — SetIamPolicy adding roles/owner/iam.securityAdmin, serviceAccounts.getAccessToken/signJwt (impersonation), actAs usage. (d) Defense evasion — audit-config deltas, sink deletion, log-bucket deletion. (e) Collection/exfil — AccessSecretVersion, cloudkms.Decrypt, storage.objects.get bursts, bigquery.jobs.query with large scans, storage.setIamPolicy adding allUsers. Correlate stages by principalEmail/callerIp in a windowed SIEM rule so a single actor traversing multiple stages escalates to a high-severity incident rather than five isolated low alerts. Feed all of it from the aggregated org sink into BigQuery + SIEM, with WORM archival for the eventual forensic timeline. Signal: thinks in kill chains and correlation, not isolated rules.

9. Key Takeaways

  • Data Access logs are off by default (except BigQuery) — enabling them via IAM auditConfigs is the single highest-impact audit change; without them, data exfiltration is invisible and unrecoverable after the fact.
  • The payload is protoPayload of type AuditLog; internalize authenticationInfo.principalEmail, methodName, resourceName, and requestMetadata.callerIp — they answer who/what/target/where for every detection.
  • Admin Activity, System Event, and Policy Denied are always on and free; Admin Activity cannot be disabled, which is why logging-tampering attempts are themselves reliably logged.
  • _Required gives a fixed 400-day floor for Admin Activity; _Default is a mutable 30-day bucket — extend it and add a WORM GCS archive for real IR coverage.
  • Production means an aggregated org sink (--include-children) into a segregated security project, fanning out to BigQuery (hunting), locked GCS (forensics), and Pub/Sub (SIEM).
  • Bucket Lock turns exported logs into WORM evidence that survives a compromised org admin or malicious insider.
  • Trust server-populated fields (requestMetadata, timestamp, insertId, principalEmail) over client-influenced request bodies during forensics.
  • Build meta-detections on audit-config deltas, sink delete/update, and bucket deletion — defending the detection pipeline is as important as the detections themselves.
  • Prevention beats detection where possible: org-policy constraints (disableServiceAccountKeyCreation, publicAccessPrevention, allowedPolicyMemberDomains) eliminate whole attack classes at the source.
  • Think in kill chains — correlate persistence, escalation, evasion, and exfil by principalEmail/callerIp so one actor traversing stages becomes one high-severity incident, not five ignored alerts.
