1. Overview

Cloud Logging and Cloud Monitoring are GCP's observability substrate, and for a security engineer they are the nervous system of cloud detection and response. Cloud Logging ingests every log — Admin Activity audit logs, Data Access logs, VPC Flow Logs, firewall logs, Cloud DNS logs, GKE audit logs, and arbitrary application logs — through a single write API, then a routing layer (the Log Router) decides, per log entry, which log buckets store it, which sinks forward it, and which entries are dropped by exclusion filters. Cloud Monitoring sits alongside, consuming time series (including log-based metrics distilled from those logs) and evaluating alerting policies that page a human or fire a webhook. At org scale this is where you centralize telemetry: an aggregated sink at the organization node fans every project's security-relevant logs into a central SIEM or a BigQuery lake, giving one chokepoint to detect Policy Denied spikes, firewall changes, and privilege escalation across thousands of projects.

The security stakes are high because logging is both your primary detection source and a first-class attack target. Adversaries who reach admin on a project routinely try to disable logging, delete sinks, or add exclusion filters to blind defenders (MITRE ATT&CK T1562.008 Impair Defenses: Disable or Modify Cloud Logs). Conversely, misconfigured logging leaks secrets and PII into logs that many principals can read, and unbounded ingestion turns into a five- or six-figure monthly bill that pressures teams to over-exclude exactly the logs a SOC needs. The senior engineer's job is to make security signal reliably reach detection, survive tampering, and stay affordable — all at once.

TL;DR
- The Log Router evaluates every entry against inclusion filters (sinks) and exclusion filters; exclusions apply before storage so an over-broad exclusion silently destroys evidence — and exclusions do not apply to _Required audit logs.
- Two sink types matter: the default _Default/_Required buckets (in-project retention) and user-defined sinks to BigQuery, Pub/Sub (→ SIEM), Cloud Storage, or another log bucket; an org-level aggregated sink centralizes everything and is the standard SecOps pattern.
- Log-based metrics turn a log filter into a Monitoring time series (counter or distribution); pair one with an alerting policy to detect events like a spike in PERMISSION_DENIED or any SetIamPolicy on the org.
- _Required audit logs are free, always-on, and immutable for 400 days; Data Access logs are off by default, can be huge, and are the ones most often (dangerously) excluded for cost.
- Guard three failure modes: secrets/PII in logs, retention/cost tradeoffs, and ensuring logs actually reach the SIEM (sink write-identity IAM, Pub/Sub backlog, dead-letter).

2. How It Works

2.1 Core mechanics

Logging is a write-once pipeline: an entry is created (by a service or the logging.write API), lands in the Log Router, and is matched against a set of sinks. Each sink has an inclusion filter (which entries it wants) and the router also holds a project/folder/org set of exclusion filters (which entries to drop before they hit any bucket the exclusion targets). A matched entry is written to the sink's destination. Monitoring is a separate service reading time series from a metric database; log-based metrics are the bridge that converts log matches into countable/measurable series.

Concept What it is Security relevance
Log entry Structured LogEntry proto: logName, resource, severity, timestamp, protoPayload/jsonPayload, labels Audit logs use protoPayload (AuditLog proto) — the field you filter on for who-did-what
Log Router Routing engine that evaluates every entry against sinks + exclusions The single chokepoint; tampering here blinds detection
Log bucket Regional storage container for logs (_Required, _Default, or custom) with a retention period _Required is immutable 400d; custom buckets can enable Log Analytics
Sink Named rule: inclusion filter → destination (bucket, BigQuery, Pub/Sub, GCS, or another project/org) Aggregated sinks centralize org-wide security logs
Exclusion filter Filter that drops matching entries before storage in a given bucket Over-exclusion = evidence destruction; cannot exclude _Required
Write identity Auto-created service account a sink uses to write to its destination Must be granted IAM on the destination or logs silently drop
Log-based metric Counter or distribution derived from a log filter Turns "firewall rule changed" into an alertable metric
Alerting policy Monitoring rule: condition over metric(s) → notification channels The paging layer for security events
Log Analytics BigQuery-backed SQL over an upgraded log bucket (no export cost) Ad-hoc threat hunting with SQL, no separate copy
Monitoring SLO Service-level objective over an SLI time series Repurposed for security signal freshness/coverage

Audit log tiers are the security backbone and behave differently:

Audit log type Default state Cost Can it be excluded/disabled?
Admin Activity Always on Free No — immutable, in _Required
System Event Always on Free No — in _Required
Policy Denied On when a service denies by policy Charged Yes (routable/excludable)
Data Access (ADMIN_READ) Off by default (except some free) Charged, high volume Yes — enable via IAM audit config
Data Access (DATA_READ/DATA_WRITE) Off by default Charged, very high volume Yes

The decision logic per entry is: (1) entry created; (2) Log Router matches inclusion filters of all sinks in scope; (3) for each destination, apply that scope's exclusion filters — but _Required ignores exclusions; (4) write to every surviving destination via the sink's write identity; (5) any log-based metric whose filter matches increments/records; (6) Monitoring evaluates alerting policies over those metrics.

2.2 Concrete configuration

Enable Data Access audit logs org-wide (the log most SOCs need and most orgs forget). Data Access logs are off by default; you turn them on in the IAM policy's auditConfigs. This is the single most impactful logging change for detection.

# Read current org IAM policy
gcloud organizations get-iam-policy 123456789012 --format=json > org-policy.json

Add an auditConfigs block enabling all three Data Access types for all services (allServices), then set it back:

{
  "auditConfigs": [
    {
      "service": "allServices",
      "auditLogConfigs": [
        { "logType": "ADMIN_READ" },
        { "logType": "DATA_READ" },
        { "logType": "DATA_WRITE" }
      ]
    }
  ]
}

gcloud organizations set-iam-policy 123456789012 org-policy.json

Commentary: allServices is a blunt hammer — DATA_READ on high-throughput services (BigQuery, GCS, Spanner) can dwarf everything else in volume and cost. In production I enable ADMIN_READ everywhere, but scope DATA_READ/DATA_WRITE to sensitive services (KMS, Secret Manager, IAM, sensitive BigQuery datasets) rather than allServices, and I route the noisy remainder to cheap Coldline GCS instead of BigQuery.

Create an organization-level aggregated sink to a central security project's BigQuery + a Pub/Sub topic feeding the SIEM. This is the canonical centralization pattern.

# Aggregated sink: every child project's audit + denied logs -> central BigQuery
gcloud logging sinks create org-audit-to-bq \
  bigquery.googleapis.com/projects/sec-central-01/datasets/org_audit \
  --organization=123456789012 \
  --include-children \
  --use-partitioned-tables \
  --log-filter='logName:"cloudaudit.googleapis.com" OR protoPayload.status.code!=0'

# Aggregated sink to Pub/Sub for streaming into Chronicle/Splunk/other SIEM
gcloud logging sinks create org-audit-to-siem \
  pubsub.googleapis.com/projects/sec-central-01/topics/siem-ingest \
  --organization=123456789012 \
  --include-children \
  --log-filter='logName:"cloudaudit.googleapis.com/activity" OR logName:"cloudaudit.googleapis.com/data_access" OR logName:"cloudaudit.googleapis.com/policy"'

Commentary: --include-children makes it aggregated — it captures logs from every folder and project under the org node without touching each project. The sink is created with an auto-generated write identity; you must grant it on the destination or logs drop silently:

# Discover the write identity, then grant it
SINK_SA=$(gcloud logging sinks describe org-audit-to-bq \
  --organization=123456789012 --format='value(writerIdentity)')

gcloud projects add-iam-policy-binding sec-central-01 \
  --member="$SINK_SA" --role="roles/bigquery.dataEditor"

# For the Pub/Sub sink
PS_SA=$(gcloud logging sinks describe org-audit-to-siem \
  --organization=123456789012 --format='value(writerIdentity)')
gcloud pubsub topics add-iam-policy-binding siem-ingest \
  --project=sec-central-01 \
  --member="$PS_SA" --role="roles/pubsub.publisher"

A log-based metric for Policy Denied events (privilege probing / recon). A counter metric with label extraction so you can slice by principal and method.

gcloud logging metrics create policy_denied_count \
  --description="Count of PERMISSION_DENIED audit events, labeled by principal and method" \
  --log-filter='protoPayload.status.code=7 OR protoPayload.@type="type.googleapis.com/google.cloud.audit.AuditLog" AND severity>=ERROR AND protoPayload.status.message:"PERMISSION_DENIED"'

For richer alerting, define it as JSON so you can attach label extractors (regex/field references into the log):

{
  "name": "policy_denied_count",
  "description": "PERMISSION_DENIED events by principal and method",
  "filter": "protoPayload.@type=\"type.googleapis.com/google.cloud.audit.AuditLog\" AND protoPayload.status.code=7",
  "metricDescriptor": {
    "metricKind": "DELTA",
    "valueType": "INT64",
    "labels": [
      { "key": "principal", "valueType": "STRING" },
      { "key": "method",    "valueType": "STRING" }
    ]
  },
  "labelExtractors": {
    "principal": "EXTRACT(protoPayload.authenticationInfo.principalEmail)",
    "method":    "EXTRACT(protoPayload.methodName)"
  }
}

gcloud logging metrics create policy_denied_count --config-from-file=policy_denied_metric.json

An alerting policy over that metric (spike in denials → page SecOps). Alerting policies are Monitoring resources; define one in JSON and create it with gcloud alpha monitoring policies create.

{
  "displayName": "SEC - Spike in Policy Denied events",
  "combiner": "OR",
  "conditions": [
    {
      "displayName": "PERMISSION_DENIED rate high",
      "conditionThreshold": {
        "filter": "metric.type=\"logging.googleapis.com/user/policy_denied_count\" resource.type=\"global\"",
        "aggregations": [
          {
            "alignmentPeriod": "300s",
            "perSeriesAligner": "ALIGN_RATE",
            "crossSeriesReducer": "REDUCE_SUM",
            "groupByFields": ["metric.label.principal"]
          }
        ],
        "comparison": "COMPARISON_GT",
        "thresholdValue": 20,
        "duration": "300s",
        "trigger": { "count": 1 }
      }
    }
  ],
  "alertStrategy": { "autoClose": "1800s" },
  "notificationChannels": [
    "projects/sec-central-01/notificationChannels/1122334455667788"
  ],
  "documentation": {
    "content": "Runbook: A principal exceeded 20 denials/5m. Check for credential misuse or recon. Pivot to Cloud-Audit-Logs note for authenticationInfo triage.",
    "mimeType": "text/markdown"
  }
}

gcloud alpha monitoring policies create --policy-from-file=policy_denied_alert.json

A log-based metric + alert for firewall/network changes (change-detection). Any mutation of firewall rules is high-signal.

gcloud logging metrics create firewall_change_count \
  --description="Firewall rule create/patch/delete events" \
  --log-filter='resource.type="gce_firewall_rule" AND (protoPayload.methodName:"compute.firewalls.insert" OR protoPayload.methodName:"compute.firewalls.patch" OR protoPayload.methodName:"compute.firewalls.delete")'

Alert on any occurrence (threshold > 0 over a short window) rather than a rate, because a single unexpected firewall change is worth a look.

Create an analytics-enabled log bucket for SQL threat hunting (no export duplication). Upgrading a bucket to Log Analytics gives you BigQuery SQL directly over stored logs.

gcloud logging buckets create sec-analytics \
  --location=us-central1 \
  --retention-days=400 \
  --enable-analytics \
  --project=sec-central-01

# Route audit logs into it with a sink whose destination is the bucket
gcloud logging sinks create to-analytics-bucket \
  logging.googleapis.com/projects/sec-central-01/locations/us-central1/buckets/sec-analytics \
  --project=sec-central-01 \
  --log-filter='logName:"cloudaudit.googleapis.com"'

A deliberately narrow exclusion filter (safe cost control). The goal is to shed known-benign noise without touching security signal. Never exclude by logName alone.

# SAFE: drop successful, high-volume health-check GETs on one load balancer,
# but keep anything non-2xx and never touch audit logs.
gcloud logging sinks update _Default \
  --add-exclusion=name=drop-lb-healthchecks,filter='resource.type="http_load_balancer" AND httpRequest.status<400 AND httpRequest.requestUrl:"/healthz"' \
  --project=app-prod-01

Commentary: The exclusion is scoped to _Default only, filters on a specific benign pattern, and explicitly keeps errors. An exclusion like filter='logName:"cloudaudit.googleapis.com/data_access"' would "save money" and destroy your Data Access trail — a self-inflicted detection gap that also fails CIS/PCI. Treat exclusion changes as security changes: require review and alert on them (see §6).

Case study 1 — The disabled sink (T1562.008). An attacker who phishes a project owner's session cookie lands with broad IAM. Their first defensive move is to delete or narrow the sink that ships logs to the SOC: gcloud logging sinks delete org-audit-to-siem. If that sink is defined at the project level, the deletion succeeds and the SOC goes dark for that project — but Admin Activity logs still land in the immutable _Required bucket, and the aggregated org-level sink (which the project owner cannot touch) still forwards them. The control that stops the blindout is (a) defining the centralizing sink at the organization node, where project-level compromise can't reach it, and (b) an alerting policy on google.logging.v2.ConfigServiceV2.DeleteSink / UpdateSink and on IAM auditConfig changes. This is the single most important architectural decision in the topic: centralize above the blast radius.

Case study 2 — Secrets in logs. A team logs full request bodies for debugging; those bodies include Authorization: Bearer tokens and, in one endpoint, plaintext SSNs. Because the app log group is readable by anyone with roles/logging.viewer on the project (dozens of engineers), a low-privilege insider or a compromised low-priv account can harvest live credentials straight from Logs Explorer — no privilege escalation needed. The fixes are layered: scrub at the source (never log auth headers/PII), apply Cloud DLP de-identification on the ingestion path or before writing, tighten logging.viewer via IAM, and route sensitive logs to a restricted bucket with Field-level access controls / restricted-fields and short retention. The lesson: logs inherit the sensitivity of what you put in them, but not automatically the access controls.

Case study 3 — The cost-driven over-exclusion. Finance flags a $40k/month Logging bill driven by BigQuery DATA_READ. An engineer "fixes" it by adding an exclusion logName:"cloudaudit.googleapis.com/data_access" on _Default. Three months later, IR needs to prove whether an exfiltrated dataset was actually read — and there's no record. The correct fix was surgical: exclude only high-volume successful reads on non-sensitive datasets, route the rest to Coldline GCS (cents/GB) instead of BigQuery, and keep KMS/Secret Manager/IAM DATA_READ at full fidelity. Cost optimization must be scalpel, not axe, and every exclusion is a potential evidence deletion.

Top 5 in depth

1. Centralize with an org-level aggregated sink above the blast radius. The mechanic: an aggregated sink at the organization (or folder) node with --include-children captures every descendant project's logs into a destination owned by a separate security project that workload owners have no IAM on. Because the sink lives on the org resource, a compromised project-owner cannot delete or edit it — they only control their own project's sinks. Implementation is the gcloud logging sinks create ... --organization=<id> --include-children shown in §2.2, with the write identity granted roles/bigquery.dataEditor / roles/pubsub.publisher on the central project. Trade-offs: aggregated sinks can route enormous volume, so you pay ingest/storage at the center; and you must design the inclusion filter carefully (audit + denied + network, not literally everything) or costs explode. Limit: it does not retroactively capture logs from before it existed, and it cannot pull the free _Required logs out of each project's immutable bucket — those stay in-project (still tamper-proof, just not centralized unless you also route them).

2. Enable Data Access audit logs deliberately and route the volume smartly. Data Access logs are the only record of reads — the "did they exfiltrate?" question. They're off by default because they're expensive. The control is enabling ADMIN_READ broadly and DATA_READ/DATA_WRITE on the crown-jewel services via auditConfigs (§2.2). Technically these are configured in the IAM policy, so changes to them are themselves audited — which lets you alert if someone turns them off. Trade-offs: on BigQuery/GCS/Spanner the volume can be 10–100x everything else; the mitigation is per-service scoping plus routing the bulk to Coldline GCS at ~$0.004/GB rather than into BigQuery Log Analytics. Limit: even with everything on, some managed-service internals aren't fully logged, and DATA_READ granularity is per-request, not per-row.

3. Narrow, reviewed exclusion filters treated as security changes. Exclusions are the legitimate cost lever, but they delete evidence before storage, so a bad one is invisible. The control is threefold: (a) exclusions must be surgical — match a specific benign pattern (successful health checks, a known chatty debug logger) and explicitly preserve errors and all audit logs; (b) every exclusion change goes through review like a firewall change; (c) an alerting policy fires on any google.logging.v2.ConfigServiceV2.UpdateSink/exclusion edit. Implementation is the scoped --add-exclusion in §2.2 plus a change-detection metric. Trade-off: surgical exclusions save less than blanket ones, so you complement them with cheaper destinations and shorter retention rather than deletion. Hard limit: _Required audit logs cannot be excluded at all — by design — which is exactly why they're your last line of forensic truth.

4. Log-based metric + alerting policy for high-signal security events. The mechanic converts a log filter into a Monitoring time series (counter/distribution) with optional label extractors, then an alerting policy evaluates a threshold/rate over a window and pages via notification channels. The canonical set: Policy Denied spikes (recon/priv-probe), any firewall change, any org/project-level SetIamPolicy, sink/audit-config edits, and service-account key creation. Implementation is the counter + JSON policy in §2.2. Trade-offs: metrics lose per-entry detail (you alert on the rate, then pivot to logs for the who), and high-cardinality label extractors (raw IPs, request IDs) blow up cost and can make the metric unusable — extract bounded labels (principal, methodName) only. Limit: log-based metrics have ingestion delay (seconds to a couple minutes), so they're detection, not prevention.

5. Redaction of secrets and PII with source scrubbing + Cloud DLP. Logs inherit the sensitivity of their payloads but not the access controls, so the primary control is don't log it: never log Authorization headers, cookies, request bodies with PII, or connection strings. Defense-in-depth adds Cloud DLP de-identification on the pipeline (inspect templates for US_SOCIAL_SECURITY_NUMBER, CREDIT_CARD_NUMBER, EMAIL_ADDRESS, then mask/tokenize), plus restricted log buckets with field-level access controls and tightened logging.viewer IAM. Trade-off: source scrubbing needs discipline and code review (regexes miss novel formats); DLP adds latency and cost and can over-redact useful signal. Limit: DLP is best-effort pattern matching — it will miss bespoke secret formats, so it complements but never replaces not-logging-secrets in the first place.

6. Detection & Response

What to log / telemetry sources

  • Admin Activity audit logs (logName ends /logs/cloudaudit.googleapis.com%2Factivity) — every mutating admin call; free, always on. Key fields: protoPayload.authenticationInfo.principalEmail, protoPayload.methodName, protoPayload.resourceName, protoPayload.requestMetadata.callerIp.
  • Data Access audit logs (.../data_access) — reads/writes to data; enable explicitly.
  • Policy Denied logs (.../policy) — VPC-SC and org-policy denials; protoPayload.status.code=7.
  • Log Router config changes — methodName in google.logging.v2.ConfigServiceV2.CreateSink/UpdateSink/DeleteSink and IAM SetIamPolicy touching auditConfigs.
  • VPC Flow Logs (resource.type="gce_subnetwork") and Firewall Rules Logging (resource.type="gce_firewall_rule") — network trail.
  • Sink error metrics (logging.googleapis.com/exports/error_count) and Pub/Sub subscription oldest_unacked_message_age — pipeline health to prove logs reach the SIEM.
  • GKE / Cloud Run audit + container logs, Cloud DNS query logs, service-account key events.

Sample detections

Detection 1 — Log-tampering (sink or audit-config disabled). Logs Explorer / sink filter (LQL):

logName:"cloudaudit.googleapis.com/activity"
AND (
  protoPayload.methodName="google.logging.v2.ConfigServiceV2.DeleteSink"
  OR protoPayload.methodName="google.logging.v2.ConfigServiceV2.UpdateSink"
  OR (protoPayload.methodName="SetIamPolicy" AND protoPayload.serviceData.policyDelta.auditConfigDeltas:*)
)
AND protoPayload.authenticationInfo.principalEmail!="known-logging-terraform-sa@sec-central-01.iam.gserviceaccount.com"

Detection 2 — Threat hunt over Log Analytics (BigQuery SQL): top principals by denials + new caller IPs.

SELECT
  proto_payload.audit_log.authentication_info.principal_email AS principal,
  proto_payload.audit_log.request_metadata.caller_ip          AS src_ip,
  proto_payload.audit_log.method_name                          AS method,
  COUNT(*)                                                     AS denials
FROM `sec-central-01.org_audit._AllLogs`
WHERE proto_payload.audit_log.status.code = 7            -- PERMISSION_DENIED
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
GROUP BY principal, src_ip, method
HAVING denials > 15
ORDER BY denials DESC;

Detection 3 — Sigma-style rule for firewall change (portable to SIEM):

title: GCP Firewall Rule Modified
logsource:
  product: gcp
  service: gcp.audit
detection:
  selection:
    data.protoPayload.methodName:
      - 'v1.compute.firewalls.insert'
      - 'v1.compute.firewalls.patch'
      - 'v1.compute.firewalls.delete'
  condition: selection
level: high
falsepositives:
  - Approved change windows via Terraform pipeline SA

Detection 4 — Ensure logs reach the SIEM (pipeline liveness alert). Alert when the Pub/Sub subscription feeding the SIEM has an oldest-unacked age over 10 minutes, or when sink error_count > 0:

metric.type="pubsub.googleapis.com/subscription/oldest_unacked_message_age"
resource.label.subscription_id="siem-ingest-sub"
# alert if value > 600 (seconds) for 5m  -> SIEM ingestion is stalled

Response playbook

  1. Triage / validate (0–15m). Confirm the alert isn't a known change (Terraform SA, approved window). Pull the triggering LogEntry: principalEmail, callerIp, methodName, resourceName, receiveTimestamp (use receive, not app, timestamp for ordering).
  2. Scope (contain intel). In Log Analytics, pivot on the principal and caller IP across all projects (the org sink makes this one query). Enumerate what else that identity did in the window; check for parallel sink/audit-config tampering (Detection 1).
  3. Contain. If an identity is compromised: disable the service-account key / revoke sessions, remove IAM bindings, and apply a deny policy. If logging was tampered with: recreate the sink from Terraform, confirm _Required still holds the immutable trail, and verify the org aggregated sink is intact.
  4. Preserve evidence. Snapshot relevant logs to a legal-hold GCS bucket (retention lock) — _Required is immutable but export a copy so exclusion changes/retention can't touch your working set.
  5. Eradicate. Remove attacker persistence (rogue firewall rules, SA keys, sinks that exfiltrate logs to attacker-controlled destinations), close the initial access vector.
  6. Recover. Restore correct firewall/IAM/logging config via IaC; re-enable any disabled Data Access audit configs; confirm SIEM ingestion caught up (unacked age back to baseline).
  7. Post-incident. Add/tune a log-based metric + alert for the exact technique seen; review exclusion filters for gaps the attacker exploited; update the runbook link embedded in the alerting policy documentation.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is the Log Router and what decides where a log entry goes?
The Log Router is the single engine every log entry passes through after creation. For each entry it evaluates all in-scope sinks' inclusion filters and the applicable exclusion filters, then writes the entry to each destination whose sink matched and whose exclusions didn't drop it. Destinations can be log buckets, BigQuery, Pub/Sub, GCS, or another project/org. Crucially, exclusions apply before storage, and _Required audit logs ignore exclusions entirely.
Signal: Do they understand routing is per-entry and exclusion-before-storage — the root of the over-exclusion risk.

Q2. Which GCP audit logs are on by default, and which must you enable?
Admin Activity and System Event logs are always on, free, and immutable in _Required. Policy Denied logs appear when a service denies by policy. Data Access logs (ADMIN_READ, DATA_READ, DATA_WRITE) are off by default and must be enabled in the IAM policy's auditConfigs because they can be high-volume and costly. For security you almost always want ADMIN_READ on, and DATA_READ/WRITE on sensitive services.
Signal: Knowing Data Access is off by default is table stakes for a cloud detection engineer.

Q3. What is a log-based metric and why use one?
A log-based metric converts a log filter into a Cloud Monitoring time series — either a counter (count of matching entries) or a distribution (over an extracted numeric value), optionally with label extractors. You use it to make log events alertable and dashboardable: e.g., count PERMISSION_DENIED events per principal so an alerting policy can page on a spike. It's the bridge from Logging to Monitoring.
Signal: Connects the two products and shows they know detection is built on metrics + logs, not just raw logs.

Q4. What's the difference between a log bucket and a sink?
A bucket is storage — a regional container with a retention period (_Required 400d immutable, _Default 30d, or custom). A sink is a routing rule (inclusion filter → destination); its destination can be a bucket, but also BigQuery/Pub/Sub/GCS. Every project has default _Required/_Default buckets and matching _Required/_Default sinks. You upgrade a bucket to enable Log Analytics for in-place SQL.
Signal: Separating storage from routing is fundamental; conflating them signals shallow knowledge.

Q5. Why can an exclusion filter be a security risk?
Because exclusions drop entries before they're stored, and no error surfaces. A cost-driven exclusion like dropping all data_access logs silently destroys your record of who read sensitive data — a self-inflicted detection gap and a compliance failure. Exclusions cannot touch _Required audit logs, but they can gut everything else. Treat exclusion changes as security changes and alert on them.
Signal: The single most important gotcha of the topic.

Tier 2 — System design / trade-offs

Q6. Design centralized security logging for a 2,000-project org.
Create an aggregated sink at the organization node with --include-children, routing audit + policy-denied + network logs into a dedicated security project that workload owners have no IAM on. Fan the destination into two: BigQuery (or an analytics-enabled log bucket) for hunting/retention, and a Pub/Sub topic for streaming into the SIEM (Chronicle/Splunk). Grant the sink's auto-created write identity the minimal role on each destination. Because the sink lives on the org node, a compromised project admin can't delete it, and immutable _Required logs remain per-project as a backstop. Add alerting on sink/audit-config changes, Pub/Sub backlog, and cost anomalies. Route noisy DATA_READ to Coldline GCS, keep crown-jewel services at full fidelity.
Signal: Org-node placement above the blast radius + cost-aware routing + pipeline health = senior thinking.

Q7. Logging costs are $60k/month. How do you cut it without losing detection?
First measure with the Logging usage metrics to find the top log names/resources (usually BigQuery/GCS DATA_READ). Then, in priority order: (1) scope DATA_READ off non-sensitive services rather than allServices; (2) route bulk logs to Coldline GCS (~$0.004/GB) instead of BigQuery; (3) apply surgical exclusions on high-volume benign patterns (successful health checks) that preserve errors and all audit logs; (4) shorten retention on non-security buckets while keeping 400d on the security bucket; (5) avoid high-cardinality log-based metric labels. Never blanket-exclude audit or error logs. Each change is reviewed like a security change with an alert on exclusion edits.
Signal: Scalpel-not-axe judgment; keeping detection and compliance intact while cutting cost.

Q8. How do you guarantee logs actually reach the SIEM?
The failure modes are (a) sink write identity missing IAM → silent drop, (b) Pub/Sub subscriber down/slow → backlog then age-out, (c) filter drift missing new log types. Controls: after creating any sink, grant the write identity on the destination and verify with the sink error_count metric (alert on > 0); use a Pub/Sub subscription with a dead-letter topic and alert on oldest_unacked_message_age > threshold; run a periodic synthetic "canary" log and confirm it lands in the SIEM end-to-end. Treat "logs reaching the SIEM" as an SLO with an error budget, not an assumption.
Signal: Understands the pipeline is a distributed system with its own reliability, not fire-and-forget.

Q9. How would you build SLOs for security signals, and what would they measure?
Repurpose Monitoring SLOs from availability to coverage and freshness of detection. Examples: ingestion freshness SLO — 99.9% of audit logs reach the SIEM within 5 minutes (SLI = Pub/Sub unacked age / end-to-end canary latency); detection coverage SLO — 100% of production projects are captured by the aggregated sink (SLI from an inventory check); alert MTTA SLO — 95% of sev-high security alerts acknowledged within 15 minutes. Each gets an error budget that, when burned, triggers a review. The point is to make "are we actually seeing everything, quickly?" a measured, alertable property rather than a hope.
Signal: Creative application of SRE constructs to security observability — rare and senior.

Q10. Trade-offs of Log Analytics (in-bucket SQL) vs exporting to BigQuery vs streaming to a SIEM?
Log Analytics gives BigQuery SQL directly over an upgraded log bucket with no export/duplication cost and 400d retention — great for ad-hoc hunting inside GCP, but it's GCP-only and not a real-time detection engine. Exporting to a separate BigQuery dataset lets you join with other data and control lifecycle/partitioning, at the cost of a duplicate copy and its storage. Streaming to Pub/Sub → SIEM gives near-real-time cross-cloud correlation and mature detection content, but adds pipeline complexity and per-GB SIEM licensing. In practice you use all three: Log Analytics for GCP-native hunting, Pub/Sub→SIEM for real-time detection, and cheap GCS/BigQuery for long-term retention.
Signal: No single right answer; the senior recognizes complementary roles and cost/latency trade-offs.

Tier 3 — Expert / deep-dive

Q11. An attacker with project-owner deletes your logging sink. Walk through what they can and cannot blind.
They can delete or narrow project-level sinks, add exclusions on non-_Required buckets, and disable Data Access audit configs going forward — so any detection depending solely on project-scoped export goes dark. They cannot touch _Required (Admin Activity/System Event stay immutable for 400d in-project) and cannot reach an org-level aggregated sink they have no IAM on, so those logs still flow to the SOC. Critically, the deletion itself is an Admin Activity event (DeleteSink) that lands in _Required and the org sink, which should trigger an alert. The architectural takeaway: put your centralizing sink above the blast radius, alert on config-plane tampering, and never let project-level export be your only copy.
Signal: Deep grasp of the trust boundaries and immutability guarantees — separates seniors from mid-level.

Q12. Design a detection for VPC Service Controls exfiltration attempts using logs.
VPC-SC denials surface as Policy Denied logs with protoPayload.status.code=7 and a metadata indicating a perimeter violation (vpcServiceControlsUniqueId, violationReason). Build a log-based metric counting these per principal and per perimeter, then an alerting policy on any occurrence for high-value perimeters plus a rate threshold to catch scanning. Enrich in Log Analytics by joining denied requests with the caller IP and service to distinguish a misconfigured legitimate service from a genuine exfil probe (external IP + sensitive service + repeated denials). Feed to the SIEM for correlation with identity signals. The subtlety: VPC-SC dry-run mode logs would-be denials without blocking, letting you baseline before enforcing.
Signal: Combines VPC-SC internals with log-based detection design and the dry-run rollout nuance.

Q13. How do you prevent secrets and PII from leaking through logs at org scale, defense-in-depth?
Layer it: (1) Prevent at source — lint/policy that blocks logging of Authorization headers, cookies, and known PII fields; structured logging so payloads are typed, not free-text. (2) Redact in pipeline — Cloud DLP de-identification with inspection templates for SSN/PAN/email that mask or tokenize before the entry is stored, run via a Dataflow template on the export path or at write time. (3) Contain access — least-privilege logging.viewer, restricted log buckets with field-level access controls, and separate buckets for sensitive logs with short retention. (4) Detect — a DLP job or scheduled Log Analytics query that scans stored logs for secret patterns and alerts, so leaks are caught even when prevention fails. No single layer is sufficient: DLP misses bespoke formats, so not-logging-it remains primary.
Signal: Shows layered thinking and honesty about each control's limits.

Q14. Explain the risks and correct design of log-based metric label extraction.
Label extractors pull fields (via EXTRACT() or regex) into metric labels so you can group/alert by them — but every distinct label-value combination is a new time series, so extracting a high-cardinality field (caller IP, request UUID, full resource name) can create millions of series, making the metric expensive, slow, and effectively unusable, and it can hit per-metric cardinality limits. The correct design extracts bounded dimensions — principalEmail (bounded by workforce), methodName, service, response code — and keeps unbounded values in the underlying logs for pivot after the alert fires. If you need per-IP analysis, do it in Log Analytics SQL, not as a metric label. This is the metrics-vs-logs division of labor: metrics for bounded aggregates and alerting, logs for high-cardinality investigation.
Signal: Cardinality is a classic footgun; understanding it shows real operational metric experience.

Q15. _Required audit logs are immutable but only kept 400 days and only in-project. How do you meet a 7-year compliance retention without trusting mutable copies?
Route a copy of the immutable logs, at write time, to two long-term destinations via the org aggregated sink: a Cloud Storage bucket with a retention policy lock (bucket-lock/WORM — even project owners cannot delete before the lock expires) for the compliance archive, and BigQuery for queryable history. Protect the GCS bucket with CMEK whose keys have a rotation policy and destruction protection, and enforce object versioning. Because the aggregated sink is at the org node and the GCS bucket is retention-locked, neither a compromised project admin nor an accidental lifecycle rule can shorten retention. Validate integrity periodically (hash manifests, or Log Analytics count reconciliation between _Required and the archive). Thus _Required gives short-term immutability, and retention-locked GCS gives long-term immutability — layered rather than trusting one store.
Signal: Distinguishes immutability from retention, knows bucket-lock/WORM, and designs for tamper-resistance end-to-end.

9. Key Takeaways

  • The Log Router is the chokepoint: every entry is matched against sink inclusion filters and exclusion filters, and exclusions apply before storage, which is why an over-broad exclusion silently destroys evidence.
  • _Required audit logs are your forensic backstop — free, always on, immutable for 400 days, and un-excludable — so architect around them, but pair with retention-locked GCS for long-term compliance.
  • Data Access logs are off by default; enabling ADMIN_READ broadly and DATA_READ/WRITE on sensitive services is the highest-leverage detection change most orgs skip.
  • Centralize with an org-level aggregated sink above the blast radius so a compromised project admin cannot delete the copy that feeds your SOC.
  • Log-based metric + alerting policy is the detection pattern: convert a filter (Policy Denied spike, firewall change, SetIamPolicy, sink deletion) into an alertable time series; alert on the rate, pivot to logs for the who.
  • Guard against silent drops: grant the sink's write identity IAM on its destination and alert on error_count and Pub/Sub backlog — otherwise logs vanish with no error.
  • Cost control must be a scalpel: scope services, route noise to Coldline GCS, use surgical exclusions that preserve errors and audit logs, and shorten retention only on non-security buckets.
  • Logs inherit payload sensitivity, not access controls: never log secrets/PII, add Cloud DLP redaction, and lock down logging.viewer.
  • Turn observability into SLOs: measure ingestion freshness, detection coverage, and alert MTTA with error budgets so "are we seeing everything, fast?" is measured, not assumed.
  • Trust logs over the console: build detections on audit logs and metrics, because UI can be hidden from (GhostToken) but the immutable audit trail cannot.
