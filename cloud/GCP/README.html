Google Cloud Platform — Security Area Overview and Mental Model

Cloud provider: Google Cloud Platform (GCP) / Google Cloud
Scope: Area entry note — security posture, resource hierarchy, identity model, core security services, and AWS/Azure translation
Frameworks / refs: Google SAIF, CIS GCP Foundations Benchmark v3.0, NIST SP 800-207 (Zero Trust), MITRE ATT&CK Cloud (IaaS) matrix, Google Cloud Architecture Framework — Security pillar

1. Overview

Google Cloud Platform is a public cloud whose security model is organized around one dominant idea: everything is a resource that lives in a strict parent-child hierarchy, and IAM policies attached to any node are inherited by everything below it. Where AWS thinks in accounts and Azure thinks in subscriptions and directories, GCP thinks in Organization → Folder → Project → resource, with a single global identity fabric (Google identities and service accounts) evaluated against allow policies and deny policies at each level. This is the mental model a senior engineer must hold in their head, because almost every GCP security question — blast radius, least privilege, data-exfiltration control, guardrails — resolves to "where in the hierarchy does the policy attach, and what inherits it?"

GCP matters at scale because it is the substrate for Google's own security-by-design engineering (Titan hardware root of trust, encryption-at-rest by default with Google-managed keys, BeyondCorp zero-trust access, ALTS service-to-service mutual auth) exposed to customers as products. For a Senior Security Engineer — and specifically for a data-protection role — the differentiators are VPC Service Controls (a data-exfiltration perimeter with no true AWS/Azure equivalent), Organization Policy (preventive guardrails distinct from IAM), Workload Identity Federation (keyless cross-cloud identity), and IAM Conditions (attribute-based constraints on bindings). Getting these four right covers most of what matters.

TL;DR
- GCP's core object is the resource hierarchy (Org → Folder → Project → resource); IAM allow policies inherit downward and are additive/union — you cannot subtract a grant at a lower level with an allow policy (use IAM deny policies or Org Policy for that).
- Identity is project-independent and global: human Google identities live in Cloud Identity / Workspace; workloads use service accounts; external and cross-cloud workloads use Workload Identity Federation to avoid long-lived service-account keys.
- Roles come in three flavors: primitive (Owner/Editor/Viewer — too broad, avoid), predefined (Google-curated, least-privilege per service), and custom (you compose permissions). Prefer predefined; use custom sparingly.
- VPC Service Controls draws a service perimeter around GCP APIs to stop data exfiltration even by validly-authenticated identities — the single most GCP-specific control and the one AWS/Azure lack a clean analog for.
- Organization Policy ≠ IAM: Org Policy constrains what configurations are allowed (preventive guardrails, GCP's answer to SCPs + Azure Policy); IAM constrains who can do what.

2. How It Works

2.1 Core mechanics

GCP security rests on four planes that a candidate should be able to separate cleanly: the resource hierarchy (organizes ownership and policy inheritance), the IAM policy engine (decides authorization), the identity providers (establish who/what the principal is), and the guardrail layer (Org Policy + VPC Service Controls + deny policies that bound what IAM can even permit).

The hierarchy has exactly four node types. The Organization is the root, tied 1:1 to a Cloud Identity or Google Workspace domain; it is created automatically the first time a user from that domain touches GCP. Folders are optional groupings (by department, environment, or business unit) and can nest. Projects are the primary unit of isolation, billing, quota, and API enablement — every resource belongs to exactly one project. Resources (a GCS bucket, a GCE instance, a BigQuery dataset) are the leaves.

An IAM allow policy is a collection of bindings, each mapping a role (a named bundle of permissions like roles/storage.objectViewer) to a set of members (principals) with an optional condition (a CEL expression on request/resource attributes). The effective permissions on any resource are the union of every binding on that resource plus every binding inherited from its project, folders, and organization. This is the load-bearing subtlety: allow policies only ever add permissions; a grant at the Org level cannot be revoked by any lower-level allow policy. To remove effective access you must use an IAM deny policy (evaluated before allow, an explicit deny wins) or prevent the resource/config from existing via Organization Policy.

The authorization decision for any API call proceeds in a specific order that is worth memorizing: (1) is the caller within any VPC Service Controls perimeter and does the call cross a perimeter boundary in a disallowed way? (2) does an IAM deny policy match? (3) does the union of allow bindings grant the required permission (evaluating any CEL conditions)? (4) does an Organization Policy constraint block the specific configuration being requested? Only if the call survives all four does it execute — and then Cloud Audit Logs records it.

2.2 Concrete configuration

Establish the hierarchy and grant at the right altitude. Grant broad, stable roles high (on a folder) and narrow roles low (on a resource).

# See where you are in the hierarchy
gcloud organizations list
gcloud resource-manager folders list --organization=123456789012
gcloud projects list --filter="parent.id=456 AND parent.type=folder"

# Create an environment folder and a project inside it
gcloud resource-manager folders create --display-name="prod" \
  --organization=123456789012
gcloud projects create acme-prod-data-471 \
  --folder=778899001122 \
  --labels=env=prod,data-classification=restricted

Grant a predefined role to a group (never to individual users for durable access) scoped to a single project — the effective access is this binding unioned with anything inherited from the folder and org:

# Predefined, least-privilege, granted to a group not a person
gcloud projects add-iam-policy-binding acme-prod-data-471 \
  --member="group:data-readers@acme.com" \
  --role="roles/bigquery.dataViewer"

Attach an IAM Condition so the binding only applies during a change window and only to one dataset — this is ABAC and a strong differentiator from classic AWS identity policy:

gcloud projects add-iam-policy-binding acme-prod-data-471 \
  --member="group:oncall-sre@acme.com" \
  --role="roles/bigquery.admin" \
  --condition='expression=request.time < timestamp("2026-07-08T00:00:00Z") && resource.name.startsWith("projects/acme-prod-data-471/datasets/pii"),title=oncall-window,description=temporary PII admin'

Set preventive guardrails with Organization Policy at the org node so every project below inherits them. These stop the two most common credential-exfil paths (long-lived SA keys and public buckets) before IAM is even consulted:

# Kill long-lived service-account keys org-wide (force WIF / impersonation instead)
gcloud org-policies set-policy - <<'EOF'
name: organizations/123456789012/policies/iam.disableServiceAccountKeyCreation
spec:
  rules:
  - enforce: true
EOF

# Enforce public-access prevention on all Cloud Storage buckets
gcloud org-policies set-policy - <<'EOF'
name: organizations/123456789012/policies/storage.publicAccessPrevention
spec:
  rules:
  - enforce: true
EOF

# Restrict which external identity providers can be federated (allowlist)
gcloud org-policies set-policy - <<'EOF'
name: organizations/123456789012/policies/iam.workloadIdentityPoolProviders
spec:
  rules:
  - values:
      allowedValues:
      - "https://token.actions.githubusercontent.com"
EOF

Replace SA keys with Workload Identity Federation so an external workload (here GitHub Actions) mints short-lived Google tokens with no downloadable secret — this is the modern keyless pattern and the correct answer to "how do you give CI access to GCP?":

gcloud iam workload-identity-pools create github-pool \
  --location=global --display-name="GitHub Actions"

gcloud iam workload-identity-pools providers create-oidc github-provider \
  --location=global --workload-identity-pool=github-pool \
  --issuer-uri="https://token.actions.githubusercontent.com" \
  --attribute-mapping="google.subject=assertion.sub,attribute.repository=assertion.repository" \
  --attribute-condition="assertion.repository=='acme/prod-deployer'"

# Let ONLY that repo impersonate the deploy SA — note principalSet, not a key
gcloud iam service-accounts add-iam-policy-binding \
  deployer@acme-prod-data-471.iam.gserviceaccount.com \
  --role="roles/iam.workloadIdentityUser" \
  --member="principalSet://iam.googleapis.com/projects/999/locations/global/workloadIdentityPools/github-pool/attribute.repository/acme/prod-deployer"

Draw a VPC Service Controls perimeter around the sensitive project so BigQuery/GCS data cannot be read out to a project outside the perimeter even with valid, admin-level credentials:

# Perimeter protecting Storage + BigQuery APIs for the restricted project
gcloud access-context-manager perimeters create acme_data_perimeter \
  --policy=987654321 --title="ACME restricted data" \
  --resources=projects/471 \
  --restricted-services=storage.googleapis.com,bigquery.googleapis.com \
  --perimeter-type=regular

Case study — Bad.Build (Cloud Build, 2023). Orca showed that Cloud Build's default service account (<project-number>@cloudbuild.gserviceaccount.com) was granted broad permissions including the ability to write logs, push to Artifact/Container Registry, and — critically — permissions that let a build step enumerate and impersonate other identities. An attacker who could influence a build (a poisoned dependency, a malicious PR triggering a build, or a supply-chain foothold) could pivot from "runs a build" to "acts as a powerful identity across the project," then push a backdoored image consumed downstream. The controls that stop it: disable automatic role grants to default service accounts (iam.automaticIamGrantsForDefaultServiceAccounts Org Policy), run builds under a dedicated minimal SA, gate deployable images with Binary Authorization, and perimeter the registry with VPC Service Controls so a stolen identity cannot pull images out of the org.

Case study — metadata SSRF token theft (T1552.005). GCE's metadata server at 169.254.169.254 serves a running instance's service-account access token. The modern endpoint requires the header Metadata-Flavor: Google, but legacy endpoints (/0.1/, /v1beta1/) did not, so any SSRF-able app on the box (a URL-fetching feature, an XXE, a vulnerable proxy) could be tricked into fetching the token and handing an attacker the instance's cloud identity — the GCP analog of the 2019 Capital One S3 breach on AWS. The fix set: disable legacy metadata (--metadata=disable-legacy-endpoints=true, now default), give instances minimal SAs with narrow scopes, prefer Workload Identity Federation over instance SAs where possible, and detect anomalous token use via Cloud Audit Logs and Security Command Center.

Case study — "Sign in with Google" domain reuse (2025). Truffle Security demonstrated that many SaaS apps using Google OAuth keyed accounts off the email address / hosted-domain (hd) claim. When a startup shut down and its domain lapsed, an attacker could re-register the domain, re-create the old employee email addresses in a new Workspace, and log in as those former employees to any SaaS still trusting the domain — inheriting access to HR, Slack, code, and cloud. The lesson is an identity-design one: trust the immutable sub claim, not mutable email/domain, and lifecycle-manage federated relationships when a domain changes hands.

Top 5 in depth

1. VPC Service Controls (the GCP-defining data-exfil control). VPC-SC draws a logical service perimeter around a set of projects and the GCP APIs they use (GCS, BigQuery, etc.). Once a project is inside a perimeter, API calls that would move data across the boundary are blocked at the API front-end even if the caller holds valid, fully-authorized IAM credentials — this is what makes it fundamentally different from IAM (which answers "may this identity do this?") and from a firewall (which filters IP traffic). Technically it evaluates the call's origin (source project, network via Private Google Access, and Access Context Manager access level such as corporate IP or managed device) against ingress and egress rules you define; anything not explicitly allowed across the boundary is denied. Implementation: define a regular perimeter listing resources (projects) and restricted-services, then add narrow ingress/egress rules for the few legitimate cross-perimeter flows. Trade-offs and limits: it is operationally heavy — CI pipelines, log exports, and cross-project analytics all break the first time and must be modeled as explicit rules; you should always run it in dry-run mode first and read vpcServiceControlsUniqueId violation logs before enforcing. It protects data going out via GCP APIs, not arbitrary application-layer exfil, so it complements — not replaces — DLP and egress firewalls.

2. Organization Policy (preventive guardrails). Org Policy constrains what configurations are permitted across the hierarchy, independent of who has IAM access — the answer to "how do I make sure no one, ever, in prod, can create a public bucket or an exportable key?" Constraints are either boolean (enforce: true) or list-based (allow/deny specific values), attach to Org/Folder/Project, and inherit downward like IAM. High-value baselines: iam.disableServiceAccountKeyCreation, storage.publicAccessPrevention, compute.requireOsLogin, compute.vmExternalIpAccess (deny public IPs), iam.allowedPolicyMemberDomains (block sharing outside your org — stops the accidental allUsers grant). Trade-off: because it is preventive and inherited, an over-broad constraint can break legitimate workloads at scale; use Policy Simulator / dry-run and scope exceptions at the folder/project level. It is GCP's unification of what AWS splits between SCPs and config rules, and what Azure calls Azure Policy.

3. Workload Identity Federation + no SA keys. The single most impactful credential-hygiene move is eliminating long-lived, exportable service-account JSON keys — historically the #1 GCP credential-leak vector (committed to Git, baked into images, exfiltrated from CI). WIF lets an external identity provider (GitHub OIDC, another cloud's IAM, an on-prem OIDC/SAML IdP) present a signed token that GCP's STS exchanges for a short-lived token allowed to impersonate a specific SA, gated by an attribute-condition (e.g. only assertion.repository=='acme/prod-deployer'). Combined with the Org Policy that disables key creation, there is simply no downloadable secret to steal. Trade-offs: federation config is fiddly, attribute mappings must be locked to immutable claims (never trust mutable email/domain — see the 2025 domain-reuse incident), and you must monitor the pool for over-broad principalSet:// grants.

4. IAM least privilege via predefined roles, Conditions, and Recommender. Effective IAM hygiene is three habits: (a) never use primitive roles (Owner/Editor/Viewer) for durable workload access — Editor alone is a near-admin blast radius; prefer predefined roles and only build custom roles when no predefined role fits; (b) attach IAM Conditions (CEL) to bound powerful grants by time, resource path, or access level — e.g. break-glass admin that auto-expires; (c) run IAM Recommender, which uses 90 days of usage to flag and right-size over-permissioned bindings. Implementation is gcloud projects add-iam-policy-binding ... --condition=... plus a recurring recommender review. Trade-off: conditions add cognitive load and custom roles drift as services add permissions, so they require ownership; the payoff is a materially smaller standing-privilege surface.

5. Cloud Audit Logs → Security Command Center / Chronicle. You cannot detect or investigate what you do not log. GCP emits Admin Activity logs always-on and free, but Data Access logs (who read which object/row) are off by default and must be explicitly enabled — a very common forensic blind spot. The production pattern: enable Data Access logging on sensitive services, create an aggregated log sink at the org level exporting to Chronicle/Google SecOps (for long-retention detection at scale) and/or BigQuery (for ad-hoc hunting), and turn on Security Command Center for continuous misconfiguration and threat findings (Event Threat Detection, anomalous IAM grants, exfil detection). Trade-off: Data Access logs on high-throughput services (GCS, BigQuery) generate enormous volume and cost, so scope them to sensitive projects and sample where appropriate.

6. Detection & Response

What to log / telemetry sources. The primary sources are Cloud Audit Logs in four streams: Admin Activity (control-plane writes, always on), Data Access (reads/writes to data, must enable), System Event (Google-initiated), and Policy Denied (calls blocked by VPC-SC/Org Policy — a high-signal exfil indicator). Key fields: protoPayload.authenticationInfo.principalEmail (who), protoPayload.methodName (what API), protoPayload.serviceName, protoPayload.resourceName, protoPayload.requestMetadata.callerIp, and for perimeter violations protoPayload.metadata.violationReason / vpcServiceControlsUniqueId. Enrich with Security Command Center findings, Access Transparency logs (Google staff access), and VPC Flow Logs / Firewall Logs for network context. Export everything via an aggregated sink to Chronicle (Google SecOps) for detection and to BigQuery for hunting.

Sample detection 1 — service-account key created (Sigma-style over GCP audit logs):

title: GCP Service Account Key Created
logsource:
  product: gcp
  service: gcp.audit

  selection:
    protoPayload.methodName:
      - google.iam.admin.v1.CreateServiceAccountKey

  - protoPayload.authenticationInfo.principalEmail

Sample detection 2 — VPC Service Controls egress violation (BigQuery over exported audit logs):

SELECT
  timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS principal,
  protopayload_auditlog.methodName AS method,
  protopayload_auditlog.resourceName AS resource,
  protopayload_auditlog.requestMetadata.callerIp AS caller_ip
FROM `acme-logs.security.cloudaudit_googleapis_com_policy`
WHERE protopayload_auditlog.status.code = 7            -- PERMISSION_DENIED
  AND protopayload_auditlog.metadata IS NOT NULL
  AND JSON_VALUE(protopayload_auditlog.metadata,
      '$.violationReason') LIKE '%SERVICE_PERIMETER%'
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
ORDER BY timestamp DESC;

Sample detection 3 — primitive Owner/Editor granted to a user (anomalous privesc):

SELECT timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS grantor,
  JSON_VALUE(binding, '$.role') AS role,
  JSON_VALUE(binding, '$.members') AS members
FROM `acme-logs.security.cloudaudit_googleapis_com_activity`,
  UNNEST(JSON_QUERY_ARRAY(protopayload_auditlog.request,
    '$.policy.bindings')) AS binding
WHERE protopayload_auditlog.methodName = 'SetIamPolicy'
  AND JSON_VALUE(binding, '$.role') IN ('roles/owner','roles/editor')
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR);

Response playbook — suspected credential / SA compromise with data-exfil signal:
1. Triage & scope (identify): In SCC/Chronicle pivot on principalEmail and callerIp; enumerate every methodName the identity called in the window and every project/resource touched. Determine if it is a human, a user SA, or a default SA.
2. Contain (disable identity): Disable the SA (gcloud iam service-accounts disable ...) or suspend the user in Cloud Identity; remove or condition the offending IAM bindings; if the identity is inside a VPC-SC perimeter, this already limited egress — verify no ingress/egress rule was abused.
3. Contain (kill tokens & keys): Delete any SA keys (gcloud iam service-accounts keys delete); note that already-issued OAuth access tokens live up to ~1 hour, so also revoke sessions and rotate anything the identity could read from Secret Manager.
4. Eradicate: Find and revert persistence — new SA keys, new IAM bindings, added WIF principalSet grants, modified Org Policy, altered log sinks or disabled Data Access logs. Rebuild any compromised compute/GKE nodes from a known-good image.
5. Recover: Restore least-privilege bindings, re-enable logging, re-enforce perimeter (exit dry-run), and rotate downstream credentials the attacker may have read.
6. Post-incident: Add a preventive Org Policy or deny policy that would have blocked the initial access (e.g. disable key creation), add the detection to Chronicle, and run Policy Analyzer to confirm no residual paths to the sensitive data remain.

7. Practice Questions

Tier 1 — Fundamentals

  1. Walk me through the GCP resource hierarchy and how IAM inheritance works. The hierarchy is Organization → Folder → Project → resource; every resource lives in exactly one project. An IAM allow policy attaches to any node, and a node inherits the union of all bindings from itself and every ancestor. The critical property is that allow policies are additive — a grant made at the org or folder level cannot be taken away by any lower-level allow policy; to remove effective access you use a deny policy or prevent the config with Org Policy. Signal: do they know inheritance is a union, not an override?

  2. What are the three role types and when do you use each? Primitive (Owner/Editor/Viewer) are legacy and coarse — Editor is nearly write-everything, so avoid for anything durable. Predefined roles are Google-maintained, least-privilege bundles scoped per service (e.g. roles/pubsub.subscriber) and are the default choice. Custom roles let you compose an exact permission set when no predefined role fits, at the cost of you maintaining them as services add permissions. Signal: least-privilege instinct and awareness of custom-role maintenance burden.

  3. What is a service account and how is it different from a human identity? A service account is a non-human identity used by workloads; it is unusual because it is both a principal and a resource — you grant roles to it (what it can do) and roles on it (who can impersonate or manage it). Humans authenticate via Google identities in Cloud Identity/Workspace; SAs authenticate via keys (avoid), impersonation, or the instance metadata server. Signal: the dual principal/resource nature of SAs.

  4. How is data encrypted at rest in GCP by default, and how do you take more control? All data at rest is encrypted by default with Google-managed keys, transparently, using envelope encryption. For more control you supply CMEK (customer-managed keys in Cloud KMS) so you own rotation and can revoke decrypt, or CSEK (customer-supplied keys) where Google never stores the key. CMEK is the common enterprise choice because it balances control with usability. Signal: knows default-on encryption and the CMEK/CSEK spectrum.

  5. What is the difference between Cloud IAM and Organization Policy? IAM answers "which principal may perform which action on which resource"; Organization Policy answers "which resource configurations are even allowed here" — it is a preventive guardrail independent of identity. Example: IAM decides who can create buckets; Org Policy storage.publicAccessPrevention guarantees none of those buckets can be public regardless of IAM. Signal: separates authorization from preventive guardrails.

Tier 2 — System Design & Trade-offs

  1. Design a landing zone for a company with prod, non-prod, and shared-services environments. I'd create an Organization tied to the Workspace domain, then folders for prod, nonprod, and shared, with projects under each (one per app/environment for blast-radius isolation). Broad, stable roles (security admin, network admin) bind at the org/folder level to groups, never individuals; app-team roles bind at the project level. I'd set a baseline Org Policy set at the org node (no SA keys, no public buckets, OS Login required, domain-restricted sharing), enable Data Access audit logs with an org-level aggregated sink to Chronicle and BigQuery, and enforce VPC Service Controls around the projects holding regulated data. CI/CD uses Workload Identity Federation, not keys. This gives inherited guardrails, clean delegation boundaries, and a data perimeter, while keeping per-project isolation. Signal: hierarchy design, guardrails-at-the-top, groups-not-users, perimeter thinking.

  2. A team needs BigQuery access for analysts but you must prevent them exfiltrating PII to a personal project. How? Grant analysts roles/bigquery.dataViewer scoped to the specific dataset (with an IAM Condition limiting it if needed), then wrap the data project in a VPC Service Controls perimeter so even a valid dataViewer cannot copy or export results to a project outside the perimeter. Add Sensitive Data Protection (DLP) to classify and optionally de-identify PII columns, enable Data Access logs so every query is attributable, and alert on Policy-Denied perimeter violations. IAM alone can't stop this — the analyst legitimately can read the data — which is exactly why VPC-SC exists. Signal: recognizes IAM cannot solve exfil-by-authorized-user; reaches for VPC-SC + DLP.

  3. How would you give a GitHub Actions pipeline access to deploy to GCP without long-lived keys? Configure Workload Identity Federation: create a workload identity pool with a GitHub OIDC provider, map immutable claims (assertion.sub, assertion.repository), and set an attribute-condition restricting to the exact repo. Grant the deploy SA roles/iam.workloadIdentityUser for that principalSet://, and enforce Org Policy iam.disableServiceAccountKeyCreation so no exportable key can exist at all. The pipeline exchanges its OIDC token for a short-lived impersonation token per run — nothing durable to steal. Signal: knows the keyless pattern and pins to immutable claims.

  4. When would you choose a custom role over a predefined role, and what's the risk? Choose custom when no predefined role matches the needed permission set — either predefined is too broad (violates least privilege) or you need permissions spanning a narrow slice of several services. The risk is maintenance drift: as Google adds permissions to services, predefined roles update automatically but your custom role does not, so it can silently under- or over-permission over time. I'd keep custom roles few, version-controlled, and reviewed, and lean on IAM Recommender to catch drift. Signal: understands the operational cost, not just the mechanic.

  5. Compare GCP's identity model to AWS IAM. What surprises AWS engineers? In AWS, identities (users/roles) live inside an account and cross-account access uses role assumption with trust policies. In GCP, identity is global and hierarchy is separate from identity — a Google identity or SA can be granted access across many projects and folders in one org, and access is granted by attaching bindings at hierarchy nodes rather than embedding policies in the identity. AWS engineers are surprised that (a) allow policies are purely additive with no per-account boundary unless you add Org Policy/VPC-SC, (b) there's no direct SCP-like "deny at account edge" without Org Policy or deny policies, and (c) service accounts are impersonated rather than assumed and can carry downloadable keys. Signal: can translate mental models, not just recite services.

Tier 3 — Expert / Deep-Dive

  1. Explain the full authorization decision path for a single GCP API call. The call is evaluated in order: first VPC Service Controls — is the caller inside the perimeter and does the call cross the boundary in a disallowed way (checked against ingress/egress rules and Access Context Manager access levels)? If it survives, IAM deny policies are evaluated — an explicit deny short-circuits and rejects. Then the union of allow bindings across the resource and all ancestors must include the required permission, with any attached CEL conditions evaluating true. Finally, if the request creates/modifies config, the relevant Organization Policy constraints must permit it. Only then does it execute, and Cloud Audit Logs records it. The key insight is that deny and VPC-SC are evaluated before allow, which is how you bound an otherwise-additive model. Signal: precise ordering and why deny/VPC-SC precede allow.

  2. How exactly does a metadata SSRF lead to cloud compromise on GCP, and how do you kill it end to end? A GCE instance runs as a service account; the metadata server at 169.254.169.254 will hand out that SA's access token. If an app on the instance can be coerced into making an attacker-controlled request (SSRF, XXE, open proxy), it can fetch the token — and legacy metadata endpoints (/0.1/, /v1beta1/) served it without the Metadata-Flavor: Google header, making exploitation trivial. With the token, the attacker acts as the instance's identity across the project (the GCP twin of the Capital One breach). End-to-end fix: enforce disable-legacy-endpoints=true (now default) so the header is mandatory, assign instances minimal SAs with tight scopes, prefer Workload Identity Federation so there's less standing instance identity, fix the SSRF, and detect anomalous token use (new caller IP for the SA) in Chronicle. Signal: connects app-layer bug → cloud identity → concrete GCP-specific fixes.

  3. Design a data-exfiltration defense-in-depth for a regulated PII dataset in BigQuery. Layers: (1) IAM least privilege — predefined dataViewer/dataEditor on the specific dataset, conditions to bound by time/access-level, no primitive roles; (2) VPC Service Controls perimeter around the data project restricting bigquery.googleapis.com and storage.googleapis.com, with narrow egress rules and dry-run-then-enforce; (3) CMEK in Cloud KMS so decrypt is revocable and key access is separately audited; (4) Sensitive Data Protection (DLP) to classify columns and de-identify/tokenize the highest-sensitivity fields so raw PII isn't even present in query paths; (5) Data Access audit logs exported to Chronicle with detections on bulk reads, cross-perimeter attempts (Policy-Denied), and anomalous principals; (6) Access Transparency to catch even Google-side access. The design assumes the attacker may hold valid credentials, so the perimeter and de-identification — not just IAM — are load-bearing. Signal: layered controls, insider/valid-credential threat model, GCP-specific primitives.

  4. Where does GCP's additive IAM model bite people, and how do you enforce a hard "no one below can do X"? It bites when engineers assume they can revoke an inherited grant by editing a lower-level policy — you cannot; allow policies only add. Someone with Editor at the folder retains Editor on every project below regardless of project-level bindings. To enforce a hard prohibition you have two tools: IAM deny policies (attach a deny rule for a permission/principal set at a hierarchy node — deny wins over any allow) and Organization Policy (prevent the configuration entirely, e.g. no SA keys, no public IPs). For "no one below this folder can create SA keys," an Org Policy constraint is cleaner than chasing bindings. The anti-pattern is trying to solve a guardrail problem with allow-policy edits. Signal: deep grasp of additive semantics and the deny/Org-Policy escape hatches.

  5. Critique VPC Service Controls: what does it protect, what does it not, and where does it break operationally? VPC-SC protects against data exfiltration via GCP APIs by an identity that is otherwise authorized — its unique value is stopping "valid credential copies data to an outside project." What it does not protect: application-layer exfil (data pulled through your app and out an arbitrary egress path), non-API data movement, or misuse within the perimeter; it's not a WAF and not an IAM replacement. Operationally it breaks the first time for anything crossing the boundary — CI/CD, cross-project analytics, log/monitoring exports, third-party SaaS integrations — so every legitimate flow must be modeled as an explicit ingress/egress rule, which is why you always run dry-run first and mine vpcServiceControlsUniqueId violations before enforcing. Its power is also its cost: a tight perimeter is high-assurance but high-maintenance. Signal: opinionated, knows the limits, has run it in anger.

key differences to emphasize:
- Project-centric model. GCP's project is the atomic unit of isolation, billing, quota, and API enablement — the closest analog to an AWS account or Azure subscription, but far cheaper to create, so teams often use many small projects for blast-radius isolation rather than one big shared account.
- IAM Conditions (CEL). Bindings can carry attribute constraints (time, resource path, request IP, access level), enabling time-boxed break-glass and path-scoped grants without minting new roles — a clean ABAC story.
- Primitive vs predefined vs custom roles. The primitive Owner/Editor/Viewer trio is a legacy over-grant trap (Editor ≈ write-everything); predefined roles are the least-privilege default; custom roles are the escape hatch you must maintain.
- VPC Service Controls as a data-exfiltration perimeter. This is the one control with no direct AWS/Azure equivalent: it blocks data movement across a service boundary via GCP APIs even for validly-authenticated identities, which IAM alone can never do. In a enterprise data-protection setting this is the single highest-leverage concept to nail.

10. Key Takeaways

  • The hierarchy is the model: Organization → Folder → Project → resource, and IAM allow policies inherit downward as a union — you cannot subtract an inherited grant with another allow policy.
  • To enforce a hard "no," reach for deny policies or Organization Policy, not more allow bindings — deny and VPC-SC are evaluated before allow, which is how you bound an additive model.
  • Kill long-lived service-account keys. Enforce iam.disableServiceAccountKeyCreation and use Workload Identity Federation with immutable-claim conditions; leaked SA keys are the #1 GCP credential-exfil vector.
  • Prefer predefined roles; avoid primitive Owner/Editor/Viewer; use custom roles sparingly and watch for permission drift with IAM Recommender.
  • VPC Service Controls is GCP's signature control — a data-exfiltration perimeter that stops even authorized identities from moving data out via GCP APIs; it has no clean AWS/Azure equivalent and is the highest-leverage concept for a enterprise data-protection setting.
  • Organization Policy ≠ IAM: Org Policy constrains what configurations are allowed (preventive guardrails, ≈ SCP + Azure Policy); IAM constrains who can do what.
  • Data Access audit logs are off by default — enable them on sensitive services and export via an aggregated sink to Chronicle/BigQuery, or you have a forensic blind spot.
  • Encryption at rest is on by default; step up to CMEK (Cloud KMS) for revocable, auditable, customer-owned keys on regulated data.
  • Service accounts are dual-natured (principal and resource); guard impersonation (serviceAccountTokenCreator) as carefully as you guard the roles the SA itself holds.
  • Design for blast radius: many small projects, groups (never individuals) for durable grants, guardrails at the top of the hierarchy, and a service perimeter around the crown-jewel data.
