GCP Binary Authorization — Deploy-Time Supply-Chain Admission Control

Domain: Cloud / Software Supply-Chain Security — Deploy-Time Enforcement
Platforms: GKE, Cloud Run, Anthos / GKE Enterprise, Cloud Service Mesh
Frameworks: SLSA v1.0, in-toto Attestation Framework, NIST SP 800-204D, CIS GKE Benchmark, MITRE ATT&CK for Containers (T1610, T1525)
Related notes: supply_chain/SLSA, supply_chain/Sigstore-And-Signing, supply_chain/SBOM, cloud/Kubernetes-Security, cloud/AWS/Security/KMS, cloud/AWS/DevOps/ECR

1. Overview

Binary Authorization (BinAuthz) is a Google Cloud service that enforces, at deploy time, that only container images meeting an organization's policy may run on GKE, Cloud Run, and GKE-on-prem/Anthos. It is a policy decision point wired into the Kubernetes admission chain (and Cloud Run's equivalent gate) that answers one question for every pod that tries to start: "Is this exact image digest permitted by our policy right now?" If the answer is no, the deployment is blocked (enforce mode) or logged (dry-run mode). Its purpose is to close the gap between "we built and signed this image in our hardened pipeline" and "only images that passed that pipeline are actually running in production."

The mechanism is deliberately simple and therefore hard to bypass: policy is expressed as an admission rule (a default rule plus optional per-cluster overrides), and the two enforcement modes that matter are allowlist by image digest (a static exception list) and require attestations (proof, in the form of cryptographically signed statements, that named quality gates were satisfied). Attestations are produced by attestors — named policy objects that hold a public key (PKIX or Cloud KMS-backed) — and each attestation is a signed claim bound to a specific image digest. Because the gate keys on the immutable SHA-256 digest, not a mutable tag like :latest, an attacker cannot swap a poisoned image behind a trusted tag and have it silently admitted.

At scale BinAuthz matters because it converts supply-chain hygiene from advisory (a scan report in a dashboard nobody reads) into a hard runtime control that fails closed. It is the deploy-time terminus of a chain that starts with SLSA provenance in the build system (supply_chain/SLSA) and Sigstore/cosign signatures (supply_chain/Sigstore-And-Signing): the build proves how an artifact was made, signing proves who vouched for it, and Binary Authorization is where that proof is checked before the workload runs.

TL;DR
- BinAuthz is a deploy-time admission gate for GKE/Cloud Run: it evaluates a policy (default admission rule + per-cluster rules) against every image and blocks or allows.
- The two enforcement primitives are allowlist by image digest and require attestations; attestations are signed statements bound to a digest, verified against an attestor's public key (PKIX or Cloud KMS).
- Always roll out in dry-run first (audit-only, enforcementMode: DRYRUN_AUDIT_LOG_ONLY) then flip to enforce (ENFORCED_BLOCK_AND_AUDIT_LOG); the #1 misconfig is a policy left at ALWAYS_ALLOW.
- Break-glass lets an operator bypass the gate with an annotation, but it is logged — abuse is detectable; Continuous Validation (CV) re-checks already-running pods against policy so drift is caught after admission.
- It composes with SLSA provenance and Sigstore: the attestor verifies "built by our trusted builder AND signed by our attestor," turning supply-chain claims into an enforceable boundary.

2. How It Works

2.1 Core Mechanics

Binary Authorization has three planes worth separating in your head. The policy plane is a single project-level (or org-level, via GKE Enterprise) policy resource plus a set of attestor resources. The enforcement plane is an admission webhook baked into GKE's control plane (and Cloud Run's deploy path) that GKE runs as a managed component — you don't deploy the webhook yourself, you enable the feature. The attestation plane is Container Analysis / Artifact Analysis, where attestations are stored as Occurrence objects attached to a Note that represents the attestor.

The policy is evaluated per image. The evaluation logic is: (1) if the image digest matches an entry in admissionWhitelistPatterns, admit it unconditionally (this is the global allowlist for system/infra images); (2) otherwise find the most specific rule — a clusterAdmissionRule keyed by cluster, an istioServiceIdentityAdmissionRule, or the defaultAdmissionRule; (3) apply that rule's evaluationMode (ALWAYS_ALLOW, ALWAYS_DENY, or REQUIRE_ATTESTATION); (4) if attestations are required, verify that a valid attestation exists for the digest, signed by every attestor listed in requireAttestationsBy, and that each signature validates against that attestor's public key.

A subtle but exam-worthy point: an attestation says nothing about what was checked — it only proves that the holder of the attestor's private key vouched for this digest. The semantics ("this image passed vuln scan," "this image has SLSA L3 provenance") live entirely in when and why your pipeline chooses to sign. BinAuthz enforces the signature; your pipeline enforces the meaning.

2.2 Concrete Configuration

Enable the API and inspect the default policy. A fresh project ships with a wide-open default (ALWAYS_ALLOW), which is why "enable BinAuthz" alone secures nothing.

gcloud services enable binaryauthorization.googleapis.com \
  containeranalysis.googleapis.com cloudkms.googleapis.com

# Export the live policy so you can diff and version it in Git
gcloud container binauthz policy export > /tmp/binauthz-policy.yaml
cat /tmp/binauthz-policy.yaml   # note: defaultAdmissionRule.evaluationMode: ALWAYS_ALLOW  <-- insecure default

Create a Cloud KMS asymmetric signing key for the attestor. Use an asymmetric-sign key (EC P-256 or RSA); the private key never leaves KMS and every signature is an IAM-authorized API call, which gives you a CloudTrail-equivalent audit trail of who signed what.

gcloud kms keyrings create binauthz --location global

gcloud kms keys create prod-attestor-key \
  --location global --keyring binauthz \
  --purpose asymmetric-signing \
  --default-algorithm ec-sign-p256-sha256

Create the Container Analysis Note and the attestor. The Note is the "authority"; IAM on it governs who may write attestations. The attestor references the Note and the KMS key version.

# 1. Create the Note (JSON body) via the Container Analysis API
cat > /tmp/note.json <<'EOF'
{ "attestation": { "hint": { "human_readable_name": "prod-quality-gate" } } }
EOF
curl -s -X POST \
  "https://containeranalysis.googleapis.com/v1/projects/${PROJECT}/notes/?noteId=prod-quality-gate" \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json" -d @/tmp/note.json

# 2. Create the attestor pointing at that Note
gcloud container binauthz attestors create prod-attestor \
  --attestation-authority-note=prod-quality-gate \
  --attestation-authority-note-project=${PROJECT}

# 3. Bind the KMS public key to the attestor
gcloud container binauthz attestors public-keys add \
  --attestor=prod-attestor \
  --keyversion-project=${PROJECT} \
  --keyversion-location=global --keyversion-keyring=binauthz \
  --keyversion-key=prod-attestor-key --keyversion=1

Write and import an enforcing policy. This policy requires a prod-attestor attestation on the production cluster, allows a sandbox cluster to run anything, and allowlists only the GKE system image path. Note globalPolicyEvaluationMode: ENABLE, which additionally trusts Google-signed system images so you don't have to enumerate them all.

# binauthz-policy.yaml  (gcloud container binauthz policy import binauthz-policy.yaml)
globalPolicyEvaluationMode: ENABLE
admissionWhitelistPatterns:
- namePattern: gcr.io/google-containers/*
- namePattern: gke.gcr.io/*
defaultAdmissionRule:
  evaluationMode: REQUIRE_ATTESTATION
  enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG
  requireAttestationsBy:
  - projects/PROJECT/attestors/prod-attestor
clusterAdmissionRules:
  us-central1-a.prod-cluster:
    evaluationMode: REQUIRE_ATTESTATION
    enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG
    requireAttestationsBy:
    - projects/PROJECT/attestors/prod-attestor
  us-central1-a.sandbox-cluster:
    evaluationMode: ALWAYS_ALLOW
    enforcementMode: DRYRUN_AUDIT_LOG_ONLY

Create an attestation for a specific digest. This is what a pipeline step does after all gates pass. Always resolve the digest first — never attest a tag.

# Resolve the immutable digest (BinAuthz only ever trusts digests)
DIGEST=$(gcloud container images describe \
  us-central1-docker.pkg.dev/${PROJECT}/apps/checkout:v1.4.2 \
  --format='get(image_summary.digest)')
IMAGE="us-central1-docker.pkg.dev/${PROJECT}/apps/checkout@${DIGEST}"

# Sign+attest in one step using the KMS-backed attestor
gcloud container binauthz attestations sign-and-create \
  --project=${PROJECT} \
  --artifact-url="${IMAGE}" \
  --attestor=prod-attestor \
  --attestor-project=${PROJECT} \
  --keyversion-project=${PROJECT} \
  --keyversion-location=global --keyversion-keyring=binauthz \
  --keyversion-key=prod-attestor-key --keyversion=1

Verify what BinAuthz would decide, before deploying. gcloud beta container binauthz and the dry-run logs let you test policy without shipping.

# List attestations already present for an image
gcloud container binauthz attestations list \
  --attestor=prod-attestor --attestor-project=${PROJECT}

Enforce it on the cluster. BinAuthz must be enabled at cluster-create (or via update on newer GKE), and the workload attempt is what triggers the webhook.

gcloud container clusters create prod-cluster \
  --zone us-central1-a --binauthz-evaluation-mode=PROJECT_SINGLETON_POLICY_ENFORCE

Cloud Build integration. A canonical pipeline builds, scans, checks provenance, then attests only if every gate passes. The critical property: the attestation service account is different from the build/deploy service account, so a compromised build step cannot both build a bad image and sign it (separation of duties — see 5, "Top 5").

# cloudbuild.yaml (excerpt)
steps:
- id: build
  name: gcr.io/cloud-builders/docker
  args: ['build','-t','us-central1-docker.pkg.dev/$PROJECT_ID/apps/checkout:$SHORT_SHA','.']
- id: push
  name: gcr.io/cloud-builders/docker
  args: ['push','us-central1-docker.pkg.dev/$PROJECT_ID/apps/checkout:$SHORT_SHA']
- id: scan-gate           # fail the build if CRITICAL vulns exist
  name: gcr.io/cloud-builders/gcloud
  entrypoint: bash
  args: ['-c','gcloud artifacts docker images scan ... && ./check_no_critical.sh']
- id: attest              # only runs if all prior steps succeeded
  name: gcr.io/cloud-builders/gcloud
  entrypoint: bash
  args:
  - -c
  - |
    DIGEST=$(gcloud container images describe us-central1-docker.pkg.dev/$PROJECT_ID/apps/checkout:$SHORT_SHA --format='get(image_summary.digest)')
    gcloud container binauthz attestations sign-and-create \
      --artifact-url="us-central1-docker.pkg.dev/$PROJECT_ID/apps/checkout@${DIGEST}" \
      --attestor=prod-attestor --attestor-project=$PROJECT_ID \
      --keyversion-project=$PROJECT_ID --keyversion-location=global \
      --keyversion-keyring=binauthz --keyversion-key=prod-attestor-key --keyversion=1

Break-glass. In a genuine incident where the gate itself is the blocker (e.g., attestor outage), an operator can annotate the pod to bypass — and that bypass is logged for later review.

apiVersion: v1
kind: Pod
metadata:
  name: hotfix-checkout
  annotations:
    alpha.image-policy.k8s.io/break-glass: "true"
spec:
  containers:
  - name: app
    image: us-central1-docker.pkg.dev/PROJECT/apps/checkout@sha256:abc...

Case study — SolarWinds and why "signed" is not "trustworthy." In SUNBURST, attackers compromised the SolarWinds Orion build environment and injected the backdoor during compilation. The resulting DLL was then signed with SolarWinds' legitimate code-signing certificate and distributed through the official update channel. Every downstream verification of the signature passed, because the signature was genuine — the build was subverted. The BinAuthz lesson is that an attestation is only as strong as the pipeline that decides to emit it. A BinAuthz attestor that signs any image the build produced would have "approved" a SUNBURST-equivalent artifact. The defense is to raise the bar for when you attest: require SLSA L3 provenance (hermetic, isolated, ephemeral builders where user build steps cannot reach the signing key), verify that provenance in a policy step, and only then create the BinAuthz attestation. BinAuthz is the enforcement muscle; SLSA is the assurance that the thing being enforced is meaningful.

Case study — Tesla cryptojacking and deploy-time gates. Attackers found a Kubernetes dashboard exposed without authentication in Tesla's AWS environment, obtained cloud credentials, and ran a Monero miner inside a pod. The initial access was an infrastructure misconfiguration, but the payload was an arbitrary attacker-chosen container image pulled from the internet. Had an enforced deploy-time image policy been present — the AWS analog would be OPA/Kyverno or ECR + admission; the GCP analog is exactly REQUIRE_ATTESTATION — the attacker's miner image would have carried no valid attestation from the org's attestor and would have been blocked at admission, converting a full compromise into a blocked deploy plus a high-signal audit event.

Case study — Codecov and separation of duties. Codecov's Bash Uploader script was modified by an attacker who exfiltrated environment variables (including CI secrets) from thousands of customer pipelines. This is the canonical argument for isolating the signing identity from the build steps. In a BinAuthz design, the service account that can call attestations sign-and-create (or kms AsymmetricSign on the attestor key) must not be the same identity that runs arbitrary, potentially-attacker-controlled build commands. If they are unified, a Codecov-style step that leaks or abuses the build identity can also mint attestations for malicious digests, and the gate becomes theater.

Top 5 in depth

1. Enforce REQUIRE_ATTESTATION with a hardened default rule. The single most important control is that the default admission rule — the one that applies to any cluster without an explicit override — requires attestation and blocks. Attackers and mistakes gravitate to the fallback path, so if the default is ALWAYS_ALLOW, a team that spins up a new cluster without a per-cluster rule silently has no gate. Concretely, set defaultAdmissionRule.evaluationMode: REQUIRE_ATTESTATION and enforcementMode: ENFORCED_BLOCK_AND_AUDIT_LOG, keep admissionWhitelistPatterns minimal, and set globalPolicyEvaluationMode: ENABLE so Google's own system images are trusted without you enumerating them. The trade-off: fail-closed enforcement can block legitimate emergency deploys, which is exactly what break-glass and a well-run dry-run rollout are for. The limit: BinAuthz only checks the digest is attested; it does not inspect the image contents, so pair it with vulnerability scanning as an attestation gate.

2. Cloud KMS-backed attestor keys with least-privilege IAM. Never hold the attestor private key as a raw file in CI. Create an asymmetric-signing key in Cloud KMS (ec-sign-p256-sha256), bind its public key version to the attestor, and grant only the pipeline's dedicated attestation service account the roles/cloudkms.signerVerifier role on that specific key. Every signature then becomes an IAM-authorized AsymmetricSign call logged in Cloud Audit Logs — you get non-repudiable evidence of who signed which digest and when. Rotate the key version and add the new public key to the attestor (BinAuthz supports multiple keys per attestor, so you can rotate without a flag day). The trade-off is latency and a hard dependency on KMS availability during signing; the limit is that KMS protects the key, not the decision to sign — a compromised pipeline with sign permission can still request signatures, which is why control #3 matters.

3. Separation of duties between build and signer identities. The Codecov and SolarWinds lessons converge here: the identity that runs arbitrary, potentially-attacker-influenced build steps must not be the identity that can create attestations. In practice, run the build under build-sa@ (no KMS sign, no attachOccurrence) and perform attestation in an isolated step or a separate pipeline under attestor-sa@ that only has cloudkms.signerVerifier on the attestor key and containeranalysis.occurrences.create on the Note. Ideally the attestation decision is gated by verifying independently-produced evidence (a SLSA provenance attestation, a scan result) rather than "the build said it's fine." The trade-off is pipeline complexity; the limit is that within a single Cloud Build execution, step isolation is weaker than a fully separate builder, so the strongest form uses a separate, hardened attestation service.

4. Dry-run-first rollout with audit-log tuning. Flipping straight to enforce in a large environment guarantees an outage: legacy images, third-party charts, and system pods will lack attestations. The disciplined path is to set the rule to REQUIRE_ATTESTATION but enforcementMode: DRYRUN_AUDIT_LOG_ONLY, then mine Cloud Logging for dry-run "would have blocked" entries over a representative window (a full release cycle), fix the pipeline and allowlist for each legitimate gap, and only flip to ENFORCED_BLOCK_AND_AUDIT_LOG once the would-block set is empty or fully understood. The trade-off is time-to-enforcement; the risk is a dry-run that never gets flipped (threat #2), so bind the rollout to a deadline and track "% images with valid attestation" as the exit metric.

5. Continuous Validation plus audit-log alerting. BinAuthz's admission check is a point-in-time decision; an image that was compliant at admission can later fall out of policy (its attestation revoked, a new CVE, policy tightened). Continuous Validation re-evaluates running pods against current policy on a schedule and emits log entries for violations, closing the time-of-check/time-of-use gap without killing the pod (so you decide the response). Wire this together with alerting on the BinAuthz audit logs: any ENFORCED_BLOCK deny and any break-glass annotation should page or ticket. The trade-off is CV is detective, not preventive — it tells you a non-compliant pod is running, it does not evict it — so pair CV findings with an automated or runbook-driven remediation.

6. Detection & Response

What to log / telemetry sources.
- Cloud Audit Logs — binaryauthorization.googleapis.com: policy read/update (UpdatePolicy), attestor create/update, and admission decisions surfaced as GKE events.
- GKE control-plane / admission events: imagepolicywebhook/BinAuthz deny messages in k8s.io audit and pod events (FailedCreate with a BinAuthz reason).
- Break-glass: pod events and audit entries carrying the alpha.image-policy.k8s.io/break-glass annotation.
- Continuous Validation: CV log entries (continuousValidation.googleapis.com / k8s_cluster resource) listing images that violate current policy.
- Cloud KMS audit logs: AsymmetricSign calls on the attestor key — the ground truth of who signed what.
- Container Analysis: Occurrence creation on the attestor Note — who wrote attestations.
- Security Command Center: BINARY_AUTHORIZATION_DISABLED / policy findings and Container Threat Detection.

Sample detections.

Detect a policy weakened to ALWAYS_ALLOW or a broadened allowlist (Cloud Logging / log-based metric query):

Response playbook (containment → eradication → recovery).
1. Triage: identify whether the event is a policy-weakening change, a break-glass use, an un-attested deploy attempt, or a KMS-signing anomaly. Pull the actor principal and source IP from the audit log.
2. Contain: if the attestor key or signer SA is implicated, immediately disable the KMS key version and revoke cloudkms.signerVerifier from the suspect identity; if a policy was weakened, re-import the known-good policy from IaC (gcloud container binauthz policy import known-good.yaml).
3. Contain (workload): for un-attested/break-glass pods that ran, cordon and delete the workload; snapshot the node/pod for forensics before deletion.
4. Eradicate: rotate the attestor key (add a new KMS key version + attestor public key, remove the old), invalidate attestations created during the compromise window (delete the malicious Occurrences), and rebuild affected images from a trusted commit through the hardened pipeline.
5. Recover: re-enable enforcement, re-attest clean images, confirm Continuous Validation reports zero violations, and verify no shadow clusters bypass the policy (org-policy audit).
6. Post-incident: add or tune detections for the exploited path, enforce policy-as-IaC review, and confirm separation-of-duties held (or fix it if it didn't).

7. Practice Questions

Tier 1 — Fundamentals

Q1. What problem does Binary Authorization solve, and at what point in the lifecycle?
It enforces, at deploy time, that only container images meeting policy may run on GKE/Cloud Run/Anthos. It closes the gap between building/signing an image in a trusted pipeline and actually running only those images in production. The gate lives in the Kubernetes admission chain (and Cloud Run's deploy path) and evaluates every pod's image against a policy, blocking or logging. It complements build-time controls (SLSA) and registry controls; it is the runtime enforcement terminus of the supply chain.
Why asked / signal: Confirms the candidate places controls correctly in the lifecycle (build vs deploy vs runtime).

Q2. Explain the policy model: default rule, cluster rules, and allowlist.
A project has one policy with a defaultAdmissionRule (fallback for any cluster), optional clusterAdmissionRules keyed by location.cluster, and admissionWhitelistPatterns (globs of exempt image paths). Evaluation checks the allowlist first (unconditional admit), then the most specific matching rule, then applies its evaluationMode (ALWAYS_ALLOW/ALWAYS_DENY/REQUIRE_ATTESTATION). The default rule is the security-critical one because new clusters inherit it. Over-broad allowlist globs are a common bypass.
Why asked / signal: Tests precise understanding of how a decision is reached, not just that a gate exists.

Q3. What is an attestor and what is an attestation?
An attestor is a named policy object holding a public key (PKIX or Cloud KMS-backed) and referencing a Container Analysis Note; it is the trust anchor for verification. An attestation is a signed statement bound to a specific image digest, stored as an Occurrence, proving the holder of the attestor's private key vouched for that digest. At admission, BinAuthz verifies a valid attestation exists for the digest signed by every required attestor. The signature proves who approved, not what was checked — the semantics live in when your pipeline chooses to sign.
Why asked / signal: Distinguishes the crypto/trust primitive from its business meaning.

Q4. Why does BinAuthz key on image digest instead of tag?
Tags are mutable — :latest or :v1 can be re-pointed to a different image after approval. A SHA-256 digest is immutable and content-addressed, so an attestation bound to a digest cannot be transferred to a swapped image. If you attested a tag, an attacker (or a routine re-push) could move the tag to a malicious image and inherit the approval. Always resolve and attest the digest in the pipeline.
Why asked / signal: Catches the classic tag-vs-digest supply-chain trap.

Q5. What's the difference between dry-run and enforced mode, and how do you roll out?
Dry-run (DRYRUN_AUDIT_LOG_ONLY) evaluates policy and logs "would have blocked" without stopping deploys; enforced (ENFORCED_BLOCK_AND_AUDIT_LOG) actually blocks. You roll out by first enabling REQUIRE_ATTESTATION in dry-run, mining audit logs for would-block events across a full release cycle, fixing pipeline/allowlist gaps, then flipping to enforce once the would-block set is understood. This avoids the outage of a hard cutover and de-risks legacy/third-party images.
Why asked / signal: Operational maturity — can they ship a control without breaking prod.

Tier 2 — System Design / Trade-offs

Q6. Design a supply-chain control that guarantees "only images built by our pipeline and signed by our attestor run in prod." Walk the components.
Build in Cloud Build under a scoped build-sa, push to Artifact Registry, and generate SLSA provenance. In a separate attestation step (or pipeline) under an isolated attestor-sa, verify the provenance and scan results, resolve the image digest, and call sign-and-create against a Cloud KMS-backed attestor. The BinAuthz policy sets defaultAdmissionRule and all prod clusterAdmissionRules to REQUIRE_ATTESTATION/ENFORCED, with a minimal allowlist and globalPolicyEvaluationMode: ENABLE. Grant cloudkms.signerVerifier on the attestor key only to attestor-sa, keep the Note's attachOccurrence least-privilege, enable Continuous Validation, and alert on denies and break-glass. The key design property is separation of duties: the identity that runs build commands cannot mint attestations. Manage the policy as IaC with PR review so it can't be silently weakened.
Why asked / signal: End-to-end design with SoD, least privilege, and enforcement — the core senior competency.

Q7. BinAuthz vs. an OPA/Gatekeeper or Kyverno admission policy for image control — when would you use which?
BinAuthz is a managed, purpose-built gate tightly integrated with GKE/Cloud Run, Container Analysis, and KMS; it excels at attestation-based "is this digest approved" decisions and is hard to misconfigure into being bypassable because Google runs the webhook. OPA/Gatekeeper or Kyverno are general policy engines: more flexible (arbitrary constraints on any resource field, cosign verification via Kyverno's verifyImages) but you own webhook availability, fail-open/closed behavior, and correctness. In practice many orgs run both: BinAuthz for the digest/attestation gate, and Kyverno/Gatekeeper for broader policy (no privileged pods, required labels, cosign keyless verification against a specific OIDC identity). The trade-off is managed simplicity vs. flexibility and operational burden.
Why asked / signal: Tests knowledge of the broader admission-control ecosystem and honest trade-offs.

Q8. How do you integrate Sigstore/cosign and SLSA provenance with BinAuthz? Do they overlap?
They compose. SLSA provenance (from the builder) proves how/from-what an image was built; cosign/Sigstore keyless signing proves who signed it with a transparency-logged, short-lived cert tied to an OIDC identity. BinAuthz is the enforcement point that checks an attestation exists before admission. You can wire cosign verification into the attestation decision (verify the image was cosign-signed by repo:org/app:ref:refs/heads/main and carries valid SLSA L3 provenance), and only then create the BinAuthz attestation — or use Kyverno verifyImages for cosign and BinAuthz for the KMS-attestation path. The overlap is that both are "signed statements about a digest"; the division of labor is SLSA=assurance of build integrity, Sigstore=identity-bound signing infra, BinAuthz=GCP-native deploy gate. (See supply_chain/SLSA and supply_chain/Sigstore-And-Signing.)
Why asked / signal: Whether the candidate sees the supply chain as one composed system, not disconnected tools.

Q9. How do you handle third-party / vendor images and Helm charts that you can't sign in your pipeline?
Options in order of preference: (1) re-attest — pull the vendor image, run it through your own verification (scan, provenance check where available, manual review for a base), and create your own attestation for its digest, so your gate approves it on your terms; (2) allowlist by exact digest for a small, reviewed set, never by broad glob; (3) mirror into your Artifact Registry and treat mirrored images as first-class pipeline inputs. Avoid open-ended namePattern globs that exempt whole registries. For Helm charts, template them and attest the resulting image digests; the chart itself is config, the images are what BinAuthz gates. Track exceptions with expiry and review.
Why asked / signal: Real-world messiness — most prod runs images you didn't build.

Q10. Fail-open vs fail-closed for the admission gate during a BinAuthz/KMS outage — what do you choose and why?
For a security control the default should be fail-closed: if the gate can't verify, don't admit. But a hard dependency on KMS/BinAuthz availability means an outage can halt all deploys, including emergency fixes. The reconciliation is: keep enforcement fail-closed, but provide a governed, logged break-glass path (the pod annotation) that a small set of operators can use during a genuine outage, with mandatory post-incident review and alerting on every use. This preserves the security invariant in steady state while giving a controlled escape hatch. Choosing fail-open globally would mean an attacker who can induce or await an outage runs arbitrary images unchecked.
Why asked / signal: The classic security-vs-availability trade-off with a concrete, defensible answer.

Tier 3 — Expert / Deep-dive

Q11. Walk through exactly how a signature is verified at admission, and what stops a replayed or transferred attestation.
The attestation payload is a canonical statement over the image digest (for the BinAuthz signing flow, a specific serialized structure), signed by the attestor's private key. At admission, BinAuthz fetches attestations (Occurrences) for the requested digest, and for each required attestor verifies the signature against the attestor's registered public key(s) and confirms the payload binds to this digest. Because the signature covers the digest, an attestation cannot be transferred to a different image (the digest wouldn't match) nor replayed onto another artifact. With Cloud KMS keys, verification uses the public key version bound to the attestor; supporting multiple keys enables rotation. What this does not prove is freshness or that the underlying build was hardened — those are handled by revocation/Continuous Validation and by your attestation policy, respectively.
Why asked / signal: Deep grasp of what the crypto binding does and does not guarantee.

Q12. Your attestor's KMS key is suspected compromised. Detail the blast radius and full response.
Blast radius: anyone with the private key (or AsymmetricSign on it) can mint attestations for any digest, so every prod cluster requiring that attestor would admit attacker images — this is a total bypass of the gate for the compromise window. Response: (1) disable the affected KMS key version immediately and revoke cloudkms.signerVerifier from all but essential identities; (2) enumerate attestations created in the window via Container Analysis + KMS AsymmetricSign audit logs, and identify any digests attested by an unexpected principal; (3) delete malicious Occurrences; (4) rotate — create a new KMS key version, add its public key to the attestor, remove the compromised one (BinAuthz's multi-key support avoids a flag day); (5) run Continuous Validation to find currently-running pods whose only approval came from the bad key; (6) cordon/delete and rebuild those from trusted source; (7) confirm separation of duties and IAM least-privilege on the key to prevent recurrence. The audit-log trail of AsymmetricSign calls is your ground truth for scoping.
Why asked / signal: Incident depth — key compromise is the highest-severity BinAuthz failure.

Q13. How does Continuous Validation differ from admission-time enforcement, and what gap does it close?
Admission enforcement is a point-in-time gate: it decides at pod-create and never re-checks. Continuous Validation periodically re-evaluates running pods against the current policy and emits log findings for violations. It closes the time-of-check/time-of-use gap: an image compliant at admission can later violate policy (attestation revoked, policy tightened, attestor removed, a newly-required attestor added), and without CV those pods keep running unnoticed until they restart. CV is detective, not preventive — it logs, it does not evict — so you wire its findings into alerting/remediation. It's essential for long-lived workloads that rarely reschedule, where admission-time approval could be months stale.
Why asked / signal: Understanding that admission control alone leaves a drift window.

Q14. An attacker has cluster-admin on a GKE cluster via a compromised RBAC binding. Does BinAuthz still help? What are its limits?
Yes, partially. BinAuthz's image gate is enforced by GKE's managed admission path, so even a cluster-admin cannot deploy an arbitrary un-attested image without either a valid attestation or a break-glass annotation — and break-glass is logged, producing high-signal detection. This is exactly the Tesla-cryptojacking mitigation: infra compromise no longer trivially yields arbitrary payload execution. Limits: (1) if the attacker can also reach the attestor key or signer SA, they can mint attestations; (2) they could use break-glass if they hold that permission (so restrict it); (3) BinAuthz gates images, not what an already-admitted, attested image does at runtime, nor exec-into-pod or host escape (see cloud/Kubernetes-Security); (4) if a cluster rule is ALWAYS_ALLOW, no help. So BinAuthz is one layer of defense-in-depth: it raises the cost of turning cluster compromise into malicious code execution, but must be paired with RBAC hardening, runtime detection, and node isolation.
Why asked / signal: Tests whether the candidate over-claims a single control; senior signal is knowing the boundary.

Q15. Design org-wide enforcement so no team can spin up an ungated cluster or weaken the policy, and prove it.
Use GCP Organization Policy to mandate BinAuthz enablement (and constrain cluster creation) across all projects, so a new GKE/Cloud Run resource inherits enforcement by default rather than opt-in. Manage the BinAuthz policy and attestors as Terraform, gated by CODEOWNERS/PR review, so any change is peer-reviewed and diffable — no manual policy import. Wire Security Command Center to surface BINARY_AUTHORIZATION_DISABLED and policy findings, and add log-based detections on UpdatePolicy events that weaken evaluationMode or broaden admissionWhitelistPatterns. Prove it continuously: a scheduled audit enumerates all clusters/Cloud Run services and asserts each maps to an enforcing rule (not ALWAYS_ALLOW, not covered only by a broad allowlist), and Continuous Validation confirms running workloads are attested. The combination — org policy (prevent), IaC review (govern), SCC + log detections (detect), CV + audit (verify) — gives both prevention and provable coverage.
Why asked / signal: Governance at scale — turning a per-project feature into an org invariant with evidence.

9. Key Takeaways

  • BinAuthz is a deploy-time admission gate: it evaluates a policy against every image and blocks (enforce) or logs (dry-run) — it is enforcement, not assurance.
  • The policy model is default rule + per-cluster rules + digest allowlist; the default rule is the security-critical fallback because new clusters inherit it, and ALWAYS_ALLOW there nullifies everything.
  • Attestations are signatures bound to an image digest, verified against an attestor's public key (prefer Cloud KMS/HSM); they prove who approved, not what was checked — the meaning lives in your pipeline's signing decision.
  • Always key on digest, never tag — a tag can be re-pointed to swap an approved image.
  • Roll out dry-run first, tune from audit logs, then flip to enforce; a perma-dry-run gate is theater.
  • Separation of duties between build and signer identities is the core defense against Codecov/SolarWinds-style pipeline compromise; scope KMS signerVerifier narrowly.
  • Break-glass is a governed, logged bypass — restrict who can use it and alert on every use.
  • Continuous Validation closes the time-of-check/time-of-use gap by re-checking running pods; it is detective, so wire it to remediation.
  • Compose with SLSA provenance (build integrity) and Sigstore/cosign (identity-bound signing) so the attestation you enforce is actually meaningful; enforce enablement org-wide via Org Policy + IaC review.
  • BinAuthz is one layer of defense-in-depth — it raises the cost of turning cluster or registry compromise into arbitrary code execution, but does not replace RBAC, runtime detection, or node isolation.
