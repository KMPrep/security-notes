GCP Cloud KMS — Key Management, Envelope Encryption, and Key Custody Models

Cloud: Google Cloud Platform — Security / Cryptography
Services: Cloud KMS, Cloud HSM, Cloud External Key Manager (EKM), Autokey
Related standards & refs: NIST SP 800-57 (Key Management), NIST SP 800-38F (key wrapping), FIPS 140-2 Level 3 (Cloud HSM), NIST SP 800-131A (algorithm transitions), MITRE ATT&CK T1486 (Data Encrypted for Impact), T1552.001 (Credentials in Files), T1078.004 (Cloud Accounts)

1. Overview

Cloud KMS is Google Cloud's managed cryptographic key service: it creates, stores, uses, rotates, and destroys cryptographic keys, and it exposes cryptographic operations (encrypt, decrypt, sign, verify, MAC, key-wrap) as API calls without ever exporting the raw key material to the caller. Its central architectural idea is that key material is a managed, access-controlled, audited resource that lives inside a hardware or software boundary, and that data is protected by envelope encryption — bulk data is encrypted locally with a fast symmetric data encryption key (DEK), and only the small DEK is sent to KMS to be wrapped by a key encryption key (KEK) that never leaves the service. This matters at scale because almost every managed GCP service (Cloud Storage, BigQuery, Persistent Disk, Cloud SQL, Pub/Sub, Secret Manager) delegates its encryption to KMS, so KMS is the single control point where an organization decides who can decrypt what, when, and prove it happened. Getting KMS right means data-at-rest encryption, regulatory key-custody requirements (BYOK/HYOK), and crypto-shredding all become policy rather than bespoke code. Getting it wrong means either a silent loss of confidentiality (over-broad cryptoKeyEncrypterDecrypter grants) or a self-inflicted denial of service (a destroyed key that shreds petabytes).

TL;DR
- KMS never gives you the key; it gives you operations on the key. Data plane calls are Encrypt/Decrypt/AsymmetricSign against a resource path, gated by IAM and logged in Cloud Audit Logs Data Access logs.
- The hierarchy is project → location → key ring → crypto key → crypto key version. IAM is enforced at key-ring or key level; key versions are the rotating unit that actually holds material.
- Envelope encryption: encrypt data with a local DEK, wrap the DEK with a KEK in KMS, store the wrapped DEK next to the ciphertext. Only unwrapping the DEK requires KMS, so throughput isn't bottlenecked on KMS.
- Custody spectrum: Google-managed (default) → CMEK (you own the KEK in KMS) → CSEK (you supply raw key bytes per request, few services) → EKM / HYOK (key material lives outside Google in a third-party HSM). Protection levels are SOFTWARE, HSM (FIPS 140-2 L3), EXTERNAL, EXTERNAL_VPC.
- Destruction is a two-step, reversible-for-a-window operation: DESTROY_SCHEDULED state for 24h–120d before material is irrecoverably gone. This window is both a safety net and a crypto-shred lever.

2. How It Works

2.1 Core mechanics

Cloud KMS is organized as a strict resource hierarchy. A key ring is a grouping/namespacing container bound to a single location (region like us-east1, dual-region like nam4, multi-region like us, or global). Inside a key ring live crypto keys; inside each crypto key live one or more crypto key versions. The key version is the object that actually holds (or references) cryptographic material; the crypto key is a logical pointer with a primary version that new symmetric encrypt operations use by default. Rotation is nothing more than creating a new version and repointing the primary — old versions stay enabled so existing ciphertext (which embeds the version that wrapped it) can still be decrypted.

Key rings and key versions are immutable and non-deletable by design in an important sense: you cannot delete a key ring, and you cannot delete a crypto key — you can only disable/destroy versions. This is deliberate, because deleting a key ring would orphan the audit history and could be weaponized. Locations are also immutable per key ring; a global key can be used from anywhere, while a regional key constrains where cryptographic operations can be served (relevant for data-residency).

Key purpose is fixed at creation and determines which operations are legal: ENCRYPT_DECRYPT (symmetric AES-256-GCM), ASYMMETRIC_SIGN (RSA 2048/3072/4096 PSS or PKCS#1, or EC P-256/P-384), ASYMMETRIC_DECRYPT (RSA-OAEP), and MAC (HMAC-SHA256). Symmetric keys are the workhorse: fast, used for envelope encryption and CMEK, and never exportable — you can only call Encrypt/Decrypt. Asymmetric keys expose a public key you can retrieve (GetPublicKey) and use offline for verify/encrypt, while the private half stays in KMS for AsymmetricSign/AsymmetricDecrypt — this is what powers code signing, JWT signing, and X.509 CA operations (Certificate Authority Service builds on this).

Protection level is orthogonal to purpose and pins where the material lives and what boundary protects it:

Protection level Where key material lives Assurance Latency/throughput Typical use
SOFTWARE Google-operated software crypto boundary FIPS 140-2 L1 Lowest latency, highest QPS Default CMEK, high-volume envelope KEKs
HSM (Cloud HSM) Google-operated FIPS 140-2 Level 3 HSM cluster Hardware-backed, tamper-resistant Slightly higher latency, quota-limited Regulated data, key never in software memory
EXTERNAL (EKM over internet) Third-party HSM/KMS outside Google You hold the material (HYOK) Network round-trip to partner; availability tied to partner Data residency / key-custody mandates
EXTERNAL_VPC (EKM over VPC) Third-party HSM reachable via private VPC HYOK, no public egress VPC path, better latency/isolation HYOK without internet exposure

Custody models (a separate axis again — this is the most-confused part):
- Google-managed encryption keys (GMEK) — the default. Google creates and rotates KEKs automatically; you never see them and there is no key ring in your project. Zero config, zero control, no separate audit surface for the key.
- CMEK (Customer-Managed Encryption Keys) — you create a KMS key (SOFTWARE/HSM/EXTERNAL) and point a service at it. The service's per-object DEKs are wrapped by your KEK. You control rotation, disable, destroy, and IAM — meaning you can revoke a whole service's ability to read its own data by disabling the key.
- CSEK (Customer-Supplied Encryption Keys) — you pass raw 256-bit key bytes (base64) on every request; Google uses them in memory to wrap the DEK and then discards them. Supported by a narrow set (Cloud Storage, Compute Engine disks/images). No KMS involved, no rotation service — you are the key server, and if you lose the bytes the data is gone.
- EKM / HYOK — CMEK where the KEK is EXTERNAL; the actual bits live in a partner HSM (Fortanix, Thales, Virtru, Futurex). Google calls out to unwrap. Pull the key at the partner and Google can no longer decrypt — the strongest "hold your own key" posture.

2.2 Concrete configuration

Create a key ring and a rotating symmetric key. Note rotation cadence and the algorithm.

# Key rings are location-bound and permanent — pick location deliberately (residency).
gcloud kms keyrings create app-secrets \
  --location=us-east1 \
  --project=acme-security-keys

# Symmetric ENCRYPT_DECRYPT key, HSM-backed, auto-rotate every 90 days,
# first rotation scheduled 90 days out.
gcloud kms keys create billing-cmek \
  --location=us-east1 \
  --keyring=app-secrets \
  --purpose=encryption \
  --protection-level=hsm \
  --default-algorithm=google-symmetric-encryption \
  --rotation-period=90d \
  --next-rotation-time="$(date -u -d '+90 days' +%Y-%m-%dT%H:%M:%SZ)" \
  --project=acme-security-keys

Direct encrypt/decrypt (for small blobs — DEKs, secrets < 64 KiB). The plaintext limit is 64 KiB precisely because KMS is meant to wrap DEKs, not bulk data.

# Encrypt a small secret. Output is base64 ciphertext bound to the key version used.
gcloud kms encrypt \
  --location=us-east1 --keyring=app-secrets --key=billing-cmek \
  --plaintext-file=./stripe-api-key.txt \
  --ciphertext-file=./stripe-api-key.enc \
  --additional-authenticated-data-file=./aad.bin   # AAD binds ciphertext to context

# Decrypt — caller needs cloudkms.cryptoKeyVersions.useToDecrypt
gcloud kms decrypt \
  --location=us-east1 --keyring=app-secrets --key=billing-cmek \
  --ciphertext-file=./stripe-api-key.enc \
  --plaintext-file=./decrypted.txt \
  --additional-authenticated-data-file=./aad.bin

The Additional Authenticated Data (AAD) is a critical, under-used control: it's not encrypted but it is authenticated, so decryption fails unless the exact same AAD is supplied. Bind AAD to context (e.g. tenant_id=42|table=cards) to prevent a stolen ciphertext from being decrypted in the wrong context — a cheap defense against ciphertext-swapping.

Least-privilege IAM — grant use, not admin. The single most important role split in KMS:

# App service account may ONLY encrypt/decrypt with this ONE key — not admin it.
gcloud kms keys add-iam-policy-binding billing-cmek \
  --location=us-east1 --keyring=app-secrets \
  --member="serviceAccount:billing-app@acme-prod.iam.gserviceaccount.com" \
  --role="roles/cloudkms.cryptoKeyEncrypterDecrypter" \
  --project=acme-security-keys

# Even tighter: decrypt-only (asymmetric consumers, or read-only services)
#   roles/cloudkms.cryptoKeyDecrypter
# Admins who manage keys but must NOT be able to decrypt data:
#   roles/cloudkms.admin   (create/rotate/destroy, but NO crypto ops)

That last line is the crux of separation of duties: roles/cloudkms.admin can rotate and destroy keys but cannot call Decrypt; roles/cloudkms.cryptoKeyEncrypterDecrypter can decrypt but cannot destroy. Never grant roles/owner on the keys project to app teams, and never grant roles/cloudkms.cryptoKeyEncrypterDecrypter at the project or key-ring level when a single key will do.

CMEK on a Cloud Storage bucket — the canonical CMEK example. The storage service agent (a Google-managed SA unique to your project) must be able to use your key, or every write fails.

# 1. Find/enable the Cloud Storage service agent and grant it use of the key.
SA="$(gsutil kms serviceaccount -p acme-prod)"   # service-<PROJECT_NUMBER>@gs-project-accounts.iam.gserviceaccount.com
gcloud kms keys add-iam-policy-binding billing-cmek \
  --location=us-east1 --keyring=app-secrets \
  --member="serviceAccount:${SA}" \
  --role="roles/cloudkms.cryptoKeyEncrypterDecrypter" \
  --project=acme-security-keys

# 2. Set the default CMEK on the bucket (must be same location as bucket).
gcloud storage buckets update gs://acme-billing-exports \
  --default-encryption-key=projects/acme-security-keys/locations/us-east1/keyRings/app-secrets/cryptoKeys/billing-cmek

# 3. Verify it stuck.
gcloud storage buckets describe gs://acme-billing-exports \
  --format="value(default_kms_key)"

Enforce CMEK org-wide so nobody can create a resource with default Google-managed keys. This is the control that turns CMEK from optional to mandatory:

# Deny resource creation unless CMEK is used, org-wide.
gcloud resource-manager org-policies enable-enforce \
  constraints/gcp.restrictNonCmekServices --organization=123456789012

# Restrict WHICH keys/projects may be used as CMEK (block foreign keys).
gcloud resource-manager org-policies set-policy cmek-projects-policy.yaml \
  --organization=123456789012
# cmek-projects-policy.yaml pins constraints/gcp.restrictCmekCryptoKeyProjects
# to under:projects/acme-security-keys so only the blessed keys project qualifies.

Rotation — automatic vs manual. Automatic rotation only makes sense for symmetric keys (old versions transparently decrypt old ciphertext). Asymmetric keys can't auto-rotate because verifiers hold the old public key.

# Set/confirm automatic rotation.
gcloud kms keys update billing-cmek \
  --location=us-east1 --keyring=app-secrets \
  --rotation-period=90d --next-rotation-time="2026-10-01T00:00:00Z"

# Manual rotation — create a new version and make it primary NOW (incident response).
gcloud kms keys versions create \
  --location=us-east1 --keyring=app-secrets --key=billing-cmek --primary

Destruction — the two-step, reversible-for-a-window flow:

# Schedule destruction. Version enters DESTROY_SCHEDULED for the key ring's
# destroy-scheduled duration (default 24h; configurable 24h–120d at key creation).
gcloud kms keys versions destroy 3 \
  --location=us-east1 --keyring=app-secrets --key=billing-cmek

# CHANGED YOUR MIND within the window? Restore it.
gcloud kms keys versions restore 3 \
  --location=us-east1 --keyring=app-secrets --key=billing-cmek

Set a long --destroy-scheduled-duration (e.g. --destroy-scheduled-duration=30d at key creation) so an accidental or malicious destroy is recoverable, but short enough that crypto-shredding still meets your data-deletion SLA.

Case study 1 — Capital One (2019) and "encryption at rest is not access control." The attacker exploited a misconfigured WAF (SSRF) to reach the instance metadata endpoint, stole the role credentials, and used them to list and read ~100M records from S3. Those objects were encrypted at rest with KMS. The encryption did nothing to stop the breach because the compromised role was authorized to decrypt — KMS transparently unwrapped the DEKs for it. The GCP-native lesson: CMEK/GMEK protects against stolen media / offline theft, not against a compromised principal with cryptoKeyDecrypter. The control that would have helped is least-privilege on the key (the role should not have had blanket decrypt across the whole dataset) plus VPC Service Controls to keep the metadata-derived credential from reaching the storage/KMS API from outside the perimeter, plus Key Access Justifications so an anomalous programmatic decrypt at scale could be denied or flagged.

Case study 2 — Storm-0558 (2023) and the skeleton signing key. Microsoft disclosed that threat actor Storm-0558 obtained a consumer (MSA) signing key and, combining it with a token-validation flaw, forged authentication tokens to access Exchange Online across many organizations. This is the asymmetric-key nightmare: a single private signing key, if extracted and long-lived, becomes a master forgery tool. The KMS-relevant takeaways map directly onto how you'd design a signing key on GCP: keep the private key in Cloud HSM (never exportable, FIPS 140-2 L3) so it cannot be extracted even by insiders; rotate signing keys on a schedule and support fast revocation; scope the key so it can only sign for its intended audience; and log every AsymmetricSign call as a Data Access event so mass or off-hours signing is detectable.

Case study 3 — Code Spaces (2014) and key/credential-driven DoS. An attacker gained control-plane access and, when the company resisted an extortion demand, deleted production data and the backups, ending the business. Translate to KMS: a principal with roles/cloudkms.admin (or owner) can schedule every key version for destruction. If the destroy-scheduled window is short and no restore happens, everything wrapped by those keys becomes permanently unreadable — crypto-shredding turned into a weapon. The controls: separate key-admin from data-plane, long destroy-scheduled durations, org-policy locks, and audit alerts on any DestroyCryptoKeyVersion.

Top 5 in depth

1. Least-privilege key IAM (roles/cloudkms.cryptoKeyEncrypterDecrypter at the key, per-SA). KMS IAM is evaluated additively down the hierarchy: a grant at the org, project, or key-ring level flows to every key beneath it. The single highest-leverage mistake is granting decrypt at a level above the individual key, which silently authorizes decryption of unrelated data. The technical control is to bind the predefined role roles/cloudkms.cryptoKeyEncrypterDecrypter — or the tighter cryptoKeyDecrypter/cryptoKeyEncrypter — directly on the specific cryptoKey resource, to a dedicated service account used by exactly one workload. Combine with IAM Conditions to add request-context constraints. Trade-off: fine-grained per-key IAM multiplies the number of bindings and can become unmanageable; mitigate with Terraform-managed bindings and periodic IAM Recommender review. Limit: IAM does not protect against a legitimately authorized but compromised principal (see Capital One) — that's what VPC-SC and Key Access Justifications are for.

2. CMEK enforcement + key-project pinning (org policy). Two org-policy constraints turn CMEK from best-effort into a guarantee. constraints/gcp.restrictNonCmekServices (list of service names) fails resource creation for listed services unless a CMEK key is specified — no more accidental Google-managed buckets. constraints/gcp.restrictCmekCryptoKeyProjects restricts which projects' keys are even accepted as CMEK, blocking an attacker (or a careless team) from pointing your production data at a key in a project they control (which would let them silently revoke or read). Implementation is a YAML policy at the org node. Trade-off: strict enforcement can break Terraform runs and new-service onboarding; roll out with dryRunSpec first and per-folder exceptions. Limit: it governs the KEK ownership, not day-to-day decrypt authorization.

3. Separation of duties between key admin and data plane. The predefined roles are deliberately split so that no single role both manages and uses keys. roles/cloudkms.admin can create, rotate, set IAM, and schedule destruction — but has no Encrypt/Decrypt permission. roles/cloudkms.cryptoKeyEncrypterDecrypter can perform crypto operations but cannot destroy or re-IAM the key. You enforce this by putting keys in a dedicated keys project whose owner/editor is held only by a small platform team, granting workloads only the crypto-op role on specific keys, and never granting basic roles/editor (which historically leaked KMS permissions). Trade-off: the key-admin team becomes a bottleneck for rotation/creation; solve with automation (Terraform) rather than broadening grants. Limit: an org-admin or a compromised platform SA still transcends this — layer Access Approval and audit alerting on top.

4. Rotation with a defined cryptoperiod. NIST SP 800-57 recommends bounded cryptoperiods so that any single key protects a limited quantity of data and time, capping blast radius from a future key compromise. For symmetric CMEK/envelope KEKs, automatic rotation (--rotation-period=90d) is transparent: a new primary version wraps new DEKs while old versions keep decrypting old ciphertext. Rotation does not re-encrypt existing data — that's a separate re-wrap job if you want to fully retire a version. For asymmetric keys, rotation must be manual and choreographed: publish the new public key to verifiers, cut over signing, then retire the old version, because auto-rotation would break every holder of the old public key. Trade-off: frequent rotation increases the number of live versions and audit noise; balance cadence against operational overhead (90 days is a common sweet spot). Limit: rotation alone doesn't help if you never disable/destroy compromised old versions.

5. Full Data Access audit logging + alerting. KMS Encrypt/Decrypt/AsymmetricSign are Data Access operations, and Data Access logging is off by default (only Admin Activity is always on). Without enabling it, key use is invisible — you can see who created a key but not who decrypted your customer PII. Enable DATA_READ and DATA_WRITE for the KMS API in the org's IAM audit config, sink logs to a BigQuery/Chronicle dataset in a separate logging project, and build alerts (below) on anomalous volume, off-hours use, new SAs, IAM changes, and destroy events. Trade-off: Data Access logs for high-QPS envelope workloads can be voluminous and costly — scope logging to sensitive keys, sample, or rely on DEK caching to cut call volume. Limit: logs are detective, not preventive; pair with VPC-SC and Key Access Justifications for prevention.

6. Detection & Response

What to log / telemetry sources. Enable Cloud Audit Logs → Data Access for cloudkms.googleapis.com (both DATA_READ and DATA_WRITE); Admin Activity logs are always on and capture create/update/setIamPolicy/destroy. Key fields to alert on in protoPayload:
- methodName — Decrypt, Encrypt, AsymmetricSign, DestroyCryptoKeyVersion, UpdateCryptoKeyVersion, SetIamPolicy, CreateCryptoKey.
- resourceName — the full key/version path (which key was used).
- authenticationInfo.principalEmail — who did it (watch for new/unexpected SAs).
- requestMetadata.callerIp / callerSuppliedUserAgent — source (watch for outside-perimeter IPs).
- authorizationInfo.granted — denied attempts are strong signal.
- Key Access Justifications keyAccessJustification reason codes (when EKM/KAJ enabled) — flag CUSTOMER_INITIATED_ACCESS anomalies or unexpected justification types.

Sample detection 1 — BigQuery/SQL: spike in Decrypt calls by a principal (mass-decrypt / exfil).

SELECT
  protopayload_auditlog.authenticationInfo.principalEmail AS principal,
  protopayload_auditlog.resourceName AS key_resource,
  COUNT(*) AS decrypt_calls,
  MIN(timestamp) AS first_seen,
  MAX(timestamp) AS last_seen
FROM `acme-logs.audit.cloudaudit_googleapis_com_data_access`
WHERE protopayload_auditlog.serviceName = 'cloudkms.googleapis.com'
  AND protopayload_auditlog.methodName LIKE '%Decrypt'
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
GROUP BY principal, key_resource
HAVING decrypt_calls > 1000        -- tune to per-principal baseline
ORDER BY decrypt_calls DESC;

Response playbook (containment → eradication → recovery):
1. Confirm & scope. Pull the offending principalEmail, resourceName, callerIp from audit logs; determine which key/versions and how many decrypt/sign calls occurred and over what window.
2. Contain — cut the identity, not the key first. Remove the compromised SA's KMS IAM binding (gcloud kms keys remove-iam-policy-binding …), disable the SA key, and revoke active tokens. Cutting the identity stops abuse without breaking legitimate consumers of the key.
3. Contain the key if needed. If the key itself is suspected compromised (e.g. EKM material or a signing key), disable the affected versions (versions disable) — reversible — before considering destruction. For signing keys, revoke the public key with verifiers and rotate immediately.
4. Eradicate. Rotate the key (new primary version), remove any attacker-added IAM bindings across the ring, and re-verify org policies (restrictNonCmekServices, key-project pinning) weren't relaxed. Hunt for persistence: re-enabled versions, new import jobs, altered rotation schedules.
5. Assess data exposure. Correlate Decrypt calls to the objects/DEKs they unwrapped; treat any data whose DEK was unwrapped by the attacker as potentially exfiltrated. Wrapped-DEK theft + later key access is the delayed-decryption risk — rotating and disabling old versions closes it.
6. Recover. Re-wrap/re-encrypt affected data under a fresh key version if required, restore any accidentally destroyed versions within the destroy-scheduled window (versions restore), and confirm services regained access. Validate CMEK is still enforced.
7. Post-incident. Tighten to per-key least privilege, shorten which principals hold decrypt, verify Data Access logging coverage, and add/adjust the detections above. Enable Key Access Justifications and Access Approval if not already, so future decrypts require an approved justification.

7. Practice Questions

Tier 1 — Fundamentals

Q1. Explain envelope encryption and why KMS uses it instead of encrypting data directly.
Envelope encryption encrypts bulk data locally with a randomly generated symmetric data encryption key (DEK), then encrypts (wraps) only that small DEK with a key encryption key (KEK) held in KMS; the wrapped DEK is stored alongside the ciphertext. KMS itself limits direct payloads to 64 KiB precisely because it's designed to wrap DEKs, not bulk data. This decouples throughput from KMS: you encrypt gigabytes locally at line speed and make only one small KMS call per object (or fewer, with DEK caching). It also enables cheap re-keying — rotating the KEK re-wraps DEKs without re-encrypting data — and lets a single KMS call revoke access to everything wrapped under a key.
Why asked / signal: This is the foundational KMS concept; a senior must explain the DEK/KEK split and the performance/blast-radius rationale without hesitation.

Q2. Walk through the Cloud KMS resource hierarchy.
Top to bottom: project → location → key ring → crypto key → crypto key version. The key ring is a location-bound namespacing container that cannot be deleted; the crypto key is a logical key with a fixed purpose and a primary version; the crypto key version holds the actual material and is the unit that rotates and can be destroyed. IAM is typically attached at the key-ring or individual-key level and inherits downward, so granting at the ring covers all keys inside it. Location is fixed at key-ring creation and drives data residency (global, regional, dual/multi-region).
Why asked / signal: Tests whether the candidate knows where IAM binds and what actually rotates — prerequisites for reasoning about least privilege.

Q3. Compare Google-managed keys, CMEK, and CSEK.
Google-managed (GMEK) is the zero-config default — Google creates and rotates the KEK, you have no key ring and no separate control. CMEK means you create and own the KEK inside KMS (SOFTWARE/HSM/EXTERNAL) and point a service at it, gaining control over rotation, IAM, disable, and destroy — including the power to revoke a service's ability to read its own data. CSEK means you pass raw 256-bit key bytes on every request for a narrow set of services (Cloud Storage, Compute disks); Google uses them in memory and discards them, so you are the key server and losing the bytes loses the data. The spectrum trades convenience for control.
Why asked / signal: These are constantly conflated; the answer shows the candidate understands the custody/control tradeoff, not just terminology.

Q4. What are the KMS protection levels and when would you pick each?
SOFTWARE (FIPS 140-2 L1, Google software boundary) is the low-latency, high-QPS default for envelope KEKs. HSM (Cloud HSM, FIPS 140-2 Level 3) keeps material in tamper-resistant hardware, never in software memory — pick it for regulated data or signing keys that must be non-extractable. EXTERNAL/EXTERNAL_VPC (EKM) keeps the actual key material in a third-party HSM outside Google (HYOK), for strict key-custody/residency mandates, at the cost of an external dependency and latency. Choose the weakest level that meets your compliance and threat model, because stronger levels cost latency, quota, and (for EKM) availability risk.
Why asked / signal: Tests that the candidate maps a compliance/threat requirement to a concrete protection level rather than defaulting to "always HSM."

Q5. What IAM roles govern KMS and why does the split matter?
roles/cloudkms.cryptoKeyEncrypterDecrypter allows encrypt+decrypt (with cryptoKeyEncrypter/cryptoKeyDecrypter as one-directional variants); roles/cloudkms.admin allows create/rotate/setIamPolicy/destroy but no crypto operations; roles/cloudkms.viewer is read-only metadata. The split enforces separation of duties: a key administrator can manage lifecycle without ever reading plaintext, and a workload can decrypt without being able to destroy or re-IAM the key. You should grant the crypto-op role on the specific key to a dedicated per-workload SA, and never grant roles/owner/editor on the keys project to app teams.
Why asked / signal: Least-privilege and SoD are the day-to-day core of key management; the exact role names signal hands-on experience.

Tier 2 — System design / trade-offs

Q6. Design CMEK for a multi-tenant SaaS storing regulated customer data across regions. How do you enforce it org-wide?
I'd stand up a dedicated keys project with tight owner held only by a platform team, and create regional key rings matching each data region for residency (never global for regulated data). Each tenant (or tenant tier) gets its own CMEK key so I can revoke or crypto-shred per tenant, HSM-protected for regulated classes. I'd grant each service's Google service agent cryptoKeyEncrypterDecrypter on only its key. To make it non-optional org-wide, I'd enforce constraints/gcp.restrictNonCmekServices (fail resource creation without CMEK) and constraints/gcp.restrictCmekCryptoKeyProjects (only the blessed keys project qualifies), plus a resourceLocations policy for residency. Rotation is automatic at 90 days on symmetric keys; Data Access logging is on and sinked to a separate logging project. Trade-offs: per-tenant keys multiply IAM bindings and rotation ops (solved with Terraform), and org-policy enforcement can break onboarding, so I roll out with dry-run first.
Why asked / signal: Tests end-to-end design: residency, per-tenant blast radius, enforcement mechanism, and operational reality.

Q7. Envelope encryption at 100k objects/sec — how do you avoid making KMS a bottleneck?
Never call KMS per byte or for bulk data; call it only to wrap/unwrap DEKs. The key optimization is DEK caching / key reuse with a bounded lifetime: generate a DEK, use it to encrypt many objects for a short window (bounded by object count and time to cap exposure), and wrap it once — this collapses KMS QPS by orders of magnitude. Cache decrypted DEKs briefly in memory (e.g. via Tink's KMS envelope AEAD with a cache) so repeated reads of the same object group don't re-hit KMS. Choose SOFTWARE protection for the KEK (higher QPS than HSM), request quota increases, and shard across keys/regions if needed. Trade-off: longer DEK reuse windows reduce KMS load but widen the blast radius if a DEK leaks, so I tune the cache TTL/usage cap against the data sensitivity.
Why asked / signal: Tests understanding that KMS is a control plane for keys, not a bulk crypto engine — and the caching-vs-blast-radius tradeoff.

Q8. When would you choose EKM/HYOK over Cloud HSM, and what new risks does it introduce?
I'd choose EKM (External Key Manager) when a regulatory or contractual mandate requires that key material physically reside outside the cloud provider and that we can render data unreadable to the provider by pulling the key — true "hold your own key." Cloud HSM keeps material in Google-operated hardware; EKM keeps it in a partner HSM (Fortanix, Thales, etc.) that Google calls out to for every unwrap. The new risks are availability (every decrypt now depends on the external partner and network path — an EKM outage means data is unreadable), latency (extra round-trip; mitigate with EXTERNAL_VPC), and operational complexity/two control planes. I'd pair it with Key Access Justifications so each unwrap carries an auditable reason the partner can approve or deny, and design a break-glass posture and multi-region EKM for resilience.
Why asked / signal: Tests grasp of the strongest custody model and honest reasoning about the availability tax it imposes.

Q9. A team wants to "delete customer data instantly for GDPR." How do you use KMS for crypto-shredding, and what are the pitfalls?
Crypto-shredding means the data stays encrypted at rest and you achieve "deletion" by destroying the key that can decrypt it — instant, verifiable un-readability without rewriting storage. The design: give each subject/tenant its own CMEK key (or a per-subject DEK wrapped by a tenant KEK), and on an erasure request schedule that key/version for destruction. Pitfalls: (1) granularity — if one key covers many subjects, you can't shred one without shredding all, so keying strategy must match your deletion unit; (2) the destroy-scheduled window means data isn't immediately gone (24h–120d), which you must reconcile with your SLA and prove in your DPA; (3) backups and replicas wrapped by the same key are also shredded (usually intended) — but copies encrypted under other keys are not; (4) it's a DoS weapon if misused, so destruction must be gated by SoD, org policy, and alerting. I'd document that crypto-shred = destroy key + confirm no other key can decrypt.
Why asked / signal: Tests whether the candidate can turn a compliance requirement into a concrete keying strategy and name the sharp edges (granularity, window, backups).

Q10. How do you rotate keys without breaking anything, for both symmetric and asymmetric keys?
For symmetric keys, rotation is transparent: create a new version and make it primary; new data wraps under the new version while old versions stay enabled to decrypt existing ciphertext (which embeds the version that wrapped it). No data re-encryption is required unless you want to retire an old version, which needs a separate re-wrap job before disabling it. Automatic rotation (--rotation-period=90d) is safe here. For asymmetric keys, rotation must be manual and choreographed because verifiers/holders have the old public key: publish the new public key to all verifiers, cut over signing to the new version, allow a dual-trust overlap, then disable the old version. Auto-rotating an asymmetric key would silently break every verifier. In both cases, disable-then-destroy compromised old versions rather than leaving them enabled.
Why asked / signal: The symmetric/asymmetric rotation asymmetry is a classic trap; a correct answer shows real operational depth.

Tier 3 — Expert / deep dive

Q11. Your KMS-encrypted data was breached even though everything was CMEK-encrypted. How is that possible, and what would have prevented it?
CMEK protects against offline/media theft and gives you a revocation lever, but it does not stop a legitimately authorized principal from decrypting — KMS transparently unwraps DEKs for any identity holding cryptoKeyDecrypter. This is exactly Capital One (2019): an SSRF stole a role's credentials, and that role was authorized to decrypt, so encryption at rest was irrelevant. Prevention is defense-in-depth beyond encryption: (1) least-privilege per-key IAM so a compromised SA can decrypt only its own narrow dataset; (2) VPC Service Controls so credentials exfiltrated to outside the perimeter can't call the KMS/Storage API at all; (3) Key Access Justifications + Access Approval so anomalous programmatic decrypt-at-scale requires an approved justification and can be denied; (4) Data Access audit logging with anomaly alerting to catch the mass-decrypt in progress. The point: encryption is a control over key custody, not over authorized access.
Why asked / signal: Separates candidates who think "encrypted = safe" from those who understand encryption's actual threat model and layer IAM/VPC-SC/KAJ on top.

Q12. Explain AES-GCM key commitment and why AAD matters in a KMS-backed scheme.
AES-GCM is an AEAD mode providing confidentiality and integrity, but standard GCM is not key-committing: a single ciphertext can be crafted to decrypt successfully under two different keys to two different plaintexts (the basis of CVE-2020-8897 in the AWS Encryption SDK and the broader "invisible salamanders" class). In multi-key or multi-tenant KMS designs this can be abused, so committing AEAD constructions or explicit key-commitment are the fix. Separately, AAD (additional authenticated data) is data that is authenticated but not encrypted — decryption fails unless the exact AAD is supplied. Binding AAD to context (tenant_id, object_id, purpose) prevents ciphertext-swapping: a stolen ciphertext can't be decrypted in a different tenant/context because the AAD won't match. So AAD is a cheap, powerful integrity-of-context control that every KMS-backed encrypt call should use.
Why asked / signal: Deep cryptographic literacy — knowing that AEAD has sharp edges (commitment) and that AAD is a design tool, not decoration.

Q13. Design detection for KMS key abuse. What do you log and what are your top alerts?
First, enable Data Access audit logs (DATA_READ/DATA_WRITE) for cloudkms.googleapis.com org-wide — they're off by default, so Decrypt/AsymmetricSign are otherwise invisible; Admin Activity logs (always on) cover create/destroy/setIamPolicy. Sink to a dedicated logging project (BigQuery/Chronicle). Top alerts: (1) SetIamPolicy on any key — silent grant of decrypt to an attacker SA is the stealthiest persistence; (2) DestroyCryptoKeyVersion — potential crypto-DoS; (3) decrypt volume anomaly per principal vs baseline — mass exfil; (4) new/unexpected principalEmail calling Decrypt, or calls from outside the VPC-SC perimeter / unusual callerIp; (5) UpdateCryptoKeyVersion to ENABLED on a previously disabled version — re-enable abuse; (6) denied ops (authorizationInfo.granted=false) spikes — probing. I'd also monitor Key Access Justification reason codes when EKM/KAJ is on. Pair detective controls with preventive ones (VPC-SC, KAJ) because logs alone don't stop the decrypt.
Why asked / signal: Detection engineering depth — knowing the default-off gotcha, the exact methodNames, and which events map to which attacker objective.

Q14. Compare BYOK import into Cloud KMS vs EKM. When is each appropriate?
BYOK (key import) brings your key material into Google's KMS: you create an import job that publishes a public wrapping key, you wrap your key locally per NIST SP 800-38F, upload it, and thereafter it lives and operates inside Google's HSM/software boundary like any other KMS key. Use it when policy says "we must generate keys ourselves / prove provenance" but you're fine with the material residing in Google after import. EKM/HYOK never imports the material — it stays permanently in a third-party HSM and Google calls out for each operation. Use EKM when the mandate is that Google must never possess the material and you must be able to revoke access by pulling the key externally. Trade-offs: BYOK gives you Google-native performance and availability but Google now holds the (imported) material; EKM preserves external custody at the cost of availability dependence and latency. Import wrapping must be handled carefully — a weak wrapping step exposes the key at import time.
Why asked / signal: Distinguishes "custody of generation" (BYOK) from "custody of storage" (HYOK) — a nuance many miss.

Q15. An org-admin's account is compromised and starts scheduling key destruction across the estate. Walk through detection, containment, and why your architecture limits the damage.
Detection fires on DestroyCryptoKeyVersion (Admin Activity log, always on) and on any SetIamPolicy/org-policy change — a burst of destroys by one principal is a high-severity alert. Containment: immediately revoke the compromised admin's sessions/keys and remove its roles, then restore every affected version within the destroy-scheduled window (gcloud kms keys versions restore) — this is exactly why I configure a long --destroy-scheduled-duration (e.g. 30 days) at key creation: destruction is reversible for a generous window, converting a would-be catastrophe into a recoverable event. Architectural limits on damage: (1) org-policy locks and SoD mean the app teams' SAs can't destroy at all; (2) key-admin and data-plane are separated so the blast radius of any one role is bounded; (3) Access Approval / KAJ can require external approval for sensitive operations; (4) keys live in a dedicated project with a very small owner set and mandatory MFA/hardware keys. Post-incident: rotate, re-verify org policies weren't relaxed, and hunt for persistence (re-enabled versions, new bindings). The design philosophy is that no single identity — even org-admin — should be able to cause irreversible loss in one action.
Why asked / signal: Tests incident response under a worst-case insider/admin-compromise scenario and whether the candidate designed reversibility and SoD in advance rather than reacting.

9. Key Takeaways

  • KMS gives you operations on keys, never the keys. Every use is an IAM-gated, audited API call; the security boundary is who can call Decrypt, enforced by IAM + VPC-SC + KAJ — not the ciphertext itself.
  • Envelope encryption (DEK wrapped by KEK) is the whole game: it decouples bulk-crypto throughput from KMS, enables cheap rekeying, and makes one call revoke access to everything under a key. Cache DEKs to survive high QPS.
  • Custody is a spectrum: GMEK (default) → CMEK (you own the KEK) → CSEK (you supply raw bytes) → EKM/HYOK (material outside Google). Each step trades convenience for control and, for EKM, availability.
  • Least privilege lives at the individual key. Bind cryptoKeyEncrypterDecrypter to one dedicated SA on one key; never at project/ring level; separate roles/cloudkms.admin (no decrypt) from data-plane roles.
  • Enforce CMEK with org policy (restrictNonCmekServices + restrictCmekCryptoKeyProjects) or it's merely aspirational — and pin keys to a blessed keys project.
  • Rotation differs by key type: symmetric auto-rotates transparently; asymmetric must be manually choreographed because verifiers hold the old public key.
  • Destruction is reversible for a window (DESTROY_SCHEDULED, 24h–120d). Set it long enough to survive accidents/attacks, short enough to meet crypto-shred SLAs, and alert on every destroy.
  • Crypto-shredding is real but sharp: it only works at the granularity of your keys, respects the destroy window, and shreds all copies under that key — and it's a DoS weapon if IAM/SoD are weak.
  • Data Access audit logs are OFF by default — enable them for KMS or key use is invisible. Alert on SetIamPolicy, Destroy, decrypt-volume anomalies, and denied ops.
  • Encryption is not access control (Capital One): a compromised authorized principal decrypts freely. Layer IAM, VPC-SC, Key Access Justifications, and detection on top.
