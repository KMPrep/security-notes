GCP Confidential Computing & Hardware Root of Trust

Domain: Cloud Security / Trusted Execution Environments (TEEs) / Data-in-Use Protection
GCP services: Confidential VMs, Shielded VMs, Confidential GKE Nodes, Confidential Space, Assured Workloads
Hardware: AMD SEV / SEV-ES / SEV-SNP, Intel TDX, Intel SGX, ARM CCA (context)
Cross-refs: infra/Side-Channel/Spectre-Meltdown/README.md, infra/Virtualization/Hypervisors/README.md, infra/Zero-Trust/BeyondCorp/README.md
Standards: NIST SP 800-193 (Platform Firmware Resiliency), IETF RATS RFC 9334, FIPS 140-3, TCG TPM 2.0

1. Overview

Data has three states — at rest (on disk), in transit (on the wire), and in use (loaded into RAM and CPU registers while being processed). The industry solved at-rest with disk/KMS encryption and in-transit with TLS, but for decades data-in-use sat in plaintext DRAM, fully visible to anything with sufficient privilege: the hypervisor, a compromised host kernel, a cloud operator with physical/console access, a co-tenant exploiting a side channel, or an attacker who dumped memory via a cold-boot or DMA attack. Confidential Computing closes that third gap by executing workloads inside a hardware-enforced Trusted Execution Environment (TEE) where memory is encrypted with a key held inside the CPU and never exposed to the host software stack.

On GCP this is delivered primarily through Confidential VMs (transparent, whole-VM memory encryption via AMD SEV/SEV-SNP or Intel TDX), layered on top of Shielded VM boot-integrity guarantees (secure boot, vTPM, measured boot, integrity monitoring). The defining property is a shift in the trust model: with Confidential Computing you no longer have to trust the hypervisor or the cloud provider's host software to preserve confidentiality of your running workload — you cryptographically reduce your Trusted Computing Base (TCB) to the CPU silicon, its microcode, and (for VM-based TEEs) your own guest OS. Remote attestation is what makes this trust actionable: before releasing secrets to a workload, a relying party verifies a hardware-signed quote proving the workload is genuinely running inside an authentic, up-to-date TEE.

TL;DR
- Confidential Computing protects data in use by encrypting RAM with a CPU-resident key; it complements — not replaces — at-rest and in-transit encryption.
- The threat model it targets is the privileged adversary: malicious/compromised hypervisor, a cloud insider, or a co-tenant — parties that TLS and KMS do nothing about.
- GCP Confidential VM = transparent full-VM memory encryption (AMD SEV-SNP / Intel TDX); Shielded VM = boot integrity (secure boot + vTPM + integrity monitoring). They are orthogonal and you usually want both.
- Attestation is the load-bearing control. A TEE without verified attestation gives false assurance — you must check the hardware signature, the measurement/PCR values, and the TCB/firmware version against a policy, then gate secret release on it.
- TEEs are not side-channel-proof. SGX and SEV have a long CVE history (Foreshadow, SGAxe, ÆPIC, CipherLeaks, transient-execution leaks). SEV-SNP/TDX add integrity + rollback protection but you still design assuming side channels exist.

2. How It Works

2.1 Core mechanics

A TEE provides three properties in hardware: confidentiality (memory encrypted, invisible to more-privileged software), integrity (memory can't be silently modified/replayed by the host), and attestability (the platform can produce a signed statement of what is running). Different silicon draws the isolation boundary in different places, which is the single most important concept to internalize.

Concept What it means GCP / hardware realization
TEE (Trusted Execution Environment) HW-isolated execution context with encrypted memory + attestation Confidential VM, Confidential GKE node, Confidential Space, SGX enclave
TCB (Trusted Computing Base) The set of components you must trust for the security property to hold VM-TEE: CPU + microcode + your guest OS. Enclave-TEE: CPU + microcode + enclave code only

VEK / per-VM key Ephemeral memory-encryption key bound to one guest, generated in the SoC AMD PSP (Platform Security Processor) generates & guards VEKs; never leaves silicon
Reverse Map Table (RMP) HW table enforcing that a physical page maps to exactly one owner SEV-SNP feature — stops hypervisor memory remapping/aliasing/replay

The isolation boundary — the key mental model. In a normal cloud VM the hypervisor sits above the guest and can read every guest page. Memory ballooning, live migration, introspection, and a compromised host all rely on that visibility. A confidential VM inverts one specific property: the memory-encryption engine tags every guest page with a per-VM key, so when the hypervisor reads a confidential guest's DRAM it sees ciphertext. The hypervisor can still schedule, deny service, and observe access patterns (it is not removed from the availability or metadata TCB), but it cannot read plaintext or (with SEV-SNP/TDX) tamper undetectably.

AMD SEV family — the evolution matters in practice:
- SEV (2017): Encrypts guest memory with a per-VM key. Register state on VMEXIT is not protected → hypervisor can read/modify guest registers.
- SEV-ES (Encrypted State, 2018): Adds encryption of the guest register state on world switch (VMSA is encrypted/integrity-checked). Closes the register-leak gap.
- SEV-SNP (Secure Nested Paging, 2020): Adds integrity via the Reverse Map Table (RMP). Defeats the hypervisor's ability to remap, alias, replay, or corrupt guest pages, and adds interrupt-injection protection and TCB versioning/rollback protection. This is the modern baseline and what you should insist on. GCP's N2D/C2D/C3D confidential offerings use SEV/SEV-SNP.

Intel TDX (Trust Domain Extensions): VM-based TEE analogous to SEV-SNP. Introduces a new CPU mode (SEAM) and the TDX Module (an Intel-signed reference TCB module) that mediates between the untrusted VMM and Trust Domains (TDs). Memory encrypted with MKTME keys; integrity + access control enforced by the TDX module and hardware. GCP offers TDX on C3 (Intel Sapphire Rapids) confidential VMs.

Intel SGX (Software Guard Extensions): Process-based. The app creates an enclave — encrypted pages (the EPC) that even ring-0 can't read. Smallest TCB but you must refactor the app into trusted/untrusted halves and cross the boundary with ECALLs/OCALLs. On GCP, SGX is not a general Confidential VM feature; the enclave model instead shows up in third-party/partner offerings and is more common on other clouds. SGX's brutal side-channel history (Foreshadow/L1TF, SGAxe, ÆPIC Leak) is the cautionary tale for the whole field.

Shielded VM — a separate, complementary control. Confidential VM protects runtime memory; Shielded VM protects boot integrity and answers "did this VM boot the software I expect, unmodified?" Three pillars:
1. Secure Boot — UEFI verifies each boot component's signature against a trusted certificate DB; unsigned/tampered bootloaders/kernels won't load. Blocks boot-level rootkits/bootkits.
2. vTPM (virtual TPM 2.0) — a virtualized, Google-attested TPM providing Measured Boot: firmware→bootloader→kernel hashes are extended into PCRs. Also provides sealed storage and a hardware-rooted identity for attestation.
3. Integrity Monitoring — Google compares the runtime boot measurements against a known-good integrity policy baseline; drift (e.g., a new kernel module, a modified bootloader) raises an integrity event you can alert on.

Confidential GKE Nodes: Confidential VM applied to Kubernetes worker nodes — every node's memory is encrypted so pod data-in-use is protected from the host and node metadata plane. Enabled per node pool (--enable-confidential-nodes). Note it protects the node's RAM; it does not by itself encrypt pod-to-pod traffic (use mTLS/service mesh) or protect against a compromised pod on the same node.

Confidential Space: A hardened, attestation-gated environment (built on Confidential VM + a minimal, measured OS image) for multi-party computation / data clean rooms — parties release their data to a workload only after attestation proves the exact container image and that no operator has shell access.

Assured Workloads: Not a TEE — a compliance and data-boundary control plane. It pins resources to a regulatory regime (FedRAMP High/Moderate, IL4/IL5, CJIS, ITAR, EU/regional data boundary, HIPAA) by enforcing data residency, personnel access controls (support-staff nationality/location), CMEK requirements, and product restrictions via Organization Policy constraints. It complements Confidential Computing: TEEs give technical insider-threat reduction; Assured Workloads gives administrative/sovereignty guarantees and provable compliance boundaries. In practice, keep them distinct.

2.2 Concrete configuration

Launch a Confidential VM (AMD SEV) — the canonical command:

gcloud compute instances create conf-vm-sev \
  --zone=us-central1-a \
  --machine-type=n2d-standard-4 \
  --confidential-compute-type=SEV \
  --maintenance-policy=TERMINATE \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud

Notes on each flag: --confidential-compute-type=SEV turns on transparent memory encryption. --maintenance-policy=TERMINATE is mandatory — confidential VMs historically could not be live-migrated (the per-VM key can't leave the SoC), so host maintenance stops the VM instead of migrating it; design for that availability trade-off. The machine family must support the TEE (n2d/c2d/c3d for AMD; c3 for Intel TDX).

Launch a Confidential VM with Intel TDX on C3:

gcloud compute instances create conf-vm-tdx \
  --zone=us-central1-a \
  --machine-type=c3-standard-4 \
  --confidential-compute-type=TDX \
  --maintenance-policy=TERMINATE \
  --min-cpu-platform="Intel Sapphire Rapids" \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud

Explicitly enable all Shielded VM options (Confidential VMs enable Shielded features by default, but be explicit for policy clarity):

gcloud compute instances create hardened-vm \
  --zone=us-central1-a \
  --machine-type=n2d-standard-4 \
  --confidential-compute-type=SEV_SNP \
  --maintenance-policy=TERMINATE \
  --shielded-secure-boot \
  --shielded-vtpm \
  --shielded-integrity-monitoring \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud

--shielded-secure-boot enforces UEFI signature checks; --shielded-vtpm provisions the measured-boot TPM; --shielded-integrity-monitoring streams boot-measurement comparisons to Cloud Logging/Monitoring. --confidential-compute-type=SEV_SNP selects the integrity-protected SNP variant where available.

Enforce it org-wide with Organization Policy — never rely on humans remembering flags:

# constraint: require Shielded VM for every new instance
name: organizations/123456789/policies/compute.requireShieldedVm
spec:
  rules:
    - enforce: true

gcloud resource-manager org-policies enable-enforce \
  compute.requireShieldedVm \
  --organization=123456789

Confidential GKE node pool:

gcloud container node-pools create conf-pool \
  --cluster=prod-cluster \
  --region=us-central1 \
  --machine-type=n2d-standard-8 \
  --enable-confidential-nodes \
  --shielded-secure-boot \
  --shielded-integrity-monitoring \
  --num-nodes=3

Or enforce cluster-wide at creation with --enable-confidential-nodes so every node pool inherits memory encryption.

Verify SEV is actually active from inside the guest (don't assume — verify):

# Kernel confirms memory encryption is on
dmesg | grep -i sev
# -> AMD Memory Encryption Features active: SEV SEV-ES SEV-SNP

# Confirm the guest sees the confidential flag
lscpu | grep -i sev

Assured Workloads folder for FedRAMP High:

  --next-rotation-time=2026-09-01T00:00:00Z \

Attestation-gated secret release (conceptual policy), the pattern that makes TEEs useful. In Confidential Space, an OIDC token containing the attestation claims is minted by the attestation service; you bind a Secret Manager / KMS key release to a Workload Identity Pool provider that only trusts that token when claims match:

# WIP attribute condition — release key ONLY to the exact attested image,
# on genuine SEV-SNP hardware, with no operator shell access
attributeCondition: >
  assertion.swname == 'CONFIDENTIAL_SPACE' &&
  assertion.submods.confidential_space.support_attributes.contains('STABLE') &&
  assertion.submods.container.image_digest ==
    'sha256:9b2c...verified-digest...e41' &&
  assertion.hwmodel == 'GCP_AMD_SEV' &&
  assertion.dbgstat == 'disabled-since-boot'

The last two lines are the crux: dbgstat == disabled-since-boot proves no debugger/console was ever attached (an operator can't peek), and the image_digest pin proves exactly which code will see the plaintext. Change the container, and the token no longer matches — secrets stay sealed.

3. Real-World Incidents & CVEs

TEEs have a rich vulnerability history — the signal is that you know these are real, exploited weaknesses, not that TEEs are useless.

Case study 1 — SGAxe and the attestation-key catastrophe (2020). SGX's whole value proposition is that a remote party can trust an enclave because the CPU signs a quote with a key only genuine Intel hardware holds. SGAxe chained transient-execution leakage to extract that attestation key material from the architectural enclave. Once an attacker holds the signing key, they can produce valid attestation quotes for fake enclaves — a relying party verifying signatures correctly would still be fooled. The lesson for a Senior engineer: attestation trust is only as strong as the secrecy of the platform's signing key and the freshness of its TCB. This is exactly why verifiers must check TCB recovery / security-version numbers (SVNs) and reject quotes from downlevel microcode — Intel revokes and re-issues provisioning certificates after such breaks. A verifier that ignores TCB version is exploitable even after a patch exists.

Case study 2 — SEV without integrity (SEVerity/SEVurity, 2020). Pre-SNP AMD SEV encrypted memory but did not authenticate it. A malicious hypervisor — precisely the adversary SEV claims to defend against — could move, replay, or splice ciphertext blocks. Researchers demonstrated injecting a chosen instruction sequence into an encrypted guest to gain code execution, fully defeating the confidentiality goal. The fix wasn't a patch; it was an architecture: SEV-SNP's Reverse Map Table cryptographically enforces single-owner, non-replayable page mappings. The takeaway you should state in practice: confidentiality without integrity is a false sense of security in the confidential-computing threat model, which is why you require SEV-SNP (or TDX), not plain SEV.

Case study 3 — CipherLeaks / ciphertext side channels (2021). Even SEV-SNP leaked information through a subtle channel: because memory encryption is deterministic per address (XTS-style tweak), an attacker observing ciphertext values in DRAM over time can correlate them to plaintext register spills on VMEXIT, recovering cryptographic key bits from constant-time libraries. AMD's mitigation (ciphertext hiding / VMSA changes) reduced but illustrates the enduring theme: TEEs stop reads of plaintext, not observation of side effects. You still deploy defense-in-depth (constant-time crypto, address-space randomization inside the guest) as if the host is watching, because for these channels, it effectively is.

Top 5 in depth

1) Verified remote attestation with policy (threats 1–4). This is the control — everything else assumes it. Mechanically: the workload asks the CPU root of trust for a signed report containing a hash of what booted (the launch measurement), the TEE type/mode, and a caller-supplied nonce. The relying party (or Google's Attestation Verification Service) validates the certificate chain up to the AMD VCEK/VLEK or Intel PCK root, confirms the nonce matches (anti-replay), checks the measurement against an allow-list, and enforces a minimum TCB security-version number so patched-but-downlevel platforms are rejected. Only then is a short-lived attestation token minted, and secret release (KMS/Secret Manager) is conditioned on that token's claims — including the exact container image_digest and dbgstat == disabled-since-boot. Implementation: Confidential Space + Workload Identity Pool attribute conditions (see §2.2). Trade-off/limit: it moves trust to the verifier and the measurement allow-list — a stale or overly permissive policy silently defeats the whole scheme, and you now own the operational burden of updating allow-lists on every legitimate image change.

2) Require SEV-SNP / TDX (integrity), not plain SEV (threat 5, 9). Confidentiality without integrity is exploitable (SEVerity). SEV-SNP's Reverse Map Table enforces that each physical page has exactly one validated owner and cannot be remapped, aliased, or replayed by the hypervisor; TDX's TDX-Module + hardware provide the equivalent. Implementation: --confidential-compute-type=SEV_SNP or =TDX, plus an org policy restricting instance creation to SNP/TDX-capable configurations. Trade-off: narrower machine-family/region availability and the no-live-migration constraint; SNP also adds some world-switch overhead versus plain SEV.

3) Enforce Shielded VM org-wide + wire integrity monitoring to detection (threats 12, 13). Secure Boot blocks unsigned bootloaders/kernels (bootkit defense); the vTPM measures the boot chain into PCRs; Integrity Monitoring diffs runtime measurements against a signed baseline and emits events on drift. The control is only real if (a) it's enforced, not opt-in — hence compute.requireShieldedVm at the org node — and (b) the integrity events are routed and alerted on. Implementation: org policy + a log sink filtering earlyBootReportEvent/lateBootReportEvent mismatches into Security Command Center or your SIEM. Trade-off: legitimate kernel/driver updates change measurements and generate baseline-update noise; you need a process to re-baseline after approved changes, or you'll train responders to ignore the alerts.

4) Pin the attested workload with Binary Authorization + digest pinning (threats 11, 16). Attestation proves code runs in a TEE; it does not prove the code is trustworthy — a backdoored image attests perfectly. You close this by only ever attesting/releasing to images that passed a signed CI/CD provenance check. Implementation: Binary Authorization admission policy requiring attestor signatures (SLSA provenance) for GKE, plus pinning the exact image_digest in the KMS/WIP release condition so a swapped image fails to obtain secrets. Trade-off: tight coupling between your build pipeline's signing keys and runtime trust — compromise of the signing attestor undermines both layers, so those keys belong in an HSM/KMS with strict separation of duties.

5) Assured Workloads for sovereignty/compliance boundaries (threat 17). Where TEEs give technical insider-threat reduction, Assured Workloads gives administrative and jurisdictional guarantees: it pins a resource folder to a compliance regime and enforces data residency, provider-personnel access constraints (location/nationality of support staff), CMEK, and product allow-lists via Organization Policy. Implementation: gcloud assured workloads create --compliance-regime=FEDRAMP_HIGH … then create all regulated resources inside that folder, with SCC monitoring for out-of-boundary resource creation. Trade-off: reduced product/region availability and higher operational friction; AW is a boundary and a policy set, not a magic wrapper — resources created outside the folder aren't covered, so you must monitor for bypass.

6. Detection & Response

What to log / telemetry sources

  • Shielded VM integrity events — Cloud Logging compute.googleapis.com with jsonPayload.@type = ShieldedVmIntegrityEvent; fields earlyBootReportEvent.policyEvaluationPassed, lateBootReportEvent.policyEvaluationPassed, and actualMeasurements vs policyMeasurements (PCR values).
  • Admin Activity audit logs — compute.instances.insert to catch VMs created without confidentialInstanceConfig or Shielded flags; compute.instances.setShieldedInstanceIntegrityPolicy (baseline resets — a re-baseline can hide tampering).
  • Attestation verification logs — from your verifier: failed signature chains, nonce mismatches, rejected TCB SVNs, unknown measurements.
  • KMS / Secret Manager access logs — secret-release decisions tied to attestation tokens; denials indicate policy mismatch or attack.
  • Assured Workloads / SCC findings — org-policy violations, resource-outside-boundary, personnel-access anomalies.
  • Guest-side — dmesg SEV/TDX status at boot; host maintenance TERMINATE events (availability signal).

Sample detections

Sigma-style — VM created without Confidential/Shielded protection in a regulated project:

title: Compute instance created without Confidential or Shielded VM
logsource:
  product: gcp
  service: cloudaudit
detection:
  selection:
    protoPayload.methodName: 'v1.compute.instances.insert'
  filter_conf:
    protoPayload.request.confidentialInstanceConfig.enableConfidentialCompute: true
  filter_shield:
    protoPayload.request.shieldedInstanceConfig.enableSecureBoot: true
  condition: selection and not (filter_conf and filter_shield)
level: high
tags:
  - attack.defense_evasion
  - attack.t1578   # Modify Cloud Compute Infrastructure

BigQuery / log-analytics SQL — Shielded VM integrity failures (boot tampering) in last 24h:

SELECT
  resource.labels.instance_id,
  timestamp,
  jsonPayload.lateBootReportEvent.policyEvaluationPassed AS late_pass,
  jsonPayload.earlyBootReportEvent.policyEvaluationPassed AS early_pass
FROM `my_project.compute_logs.cloudaudit_googleapis_com_system_event`
WHERE jsonPayload."@type" LIKE '%ShieldedVmIntegrityEvent%'
  AND (jsonPayload.lateBootReportEvent.policyEvaluationPassed = false
       OR jsonPayload.earlyBootReportEvent.policyEvaluationPassed = false)
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
ORDER BY timestamp DESC;

KQL-style — repeated attestation/secret-release denials from one workload (possible forged-quote or policy-mismatch probing):

SecretManagerAudit
| where MethodName == "AccessSecretVersion" and ResponseStatus == "PERMISSION_DENIED"
| where Reason has "attestation" or Reason has "attribute_condition"
| summarize denials = count() by CallerPrincipal, ImageDigest, bin(TimeGenerated, 10m)
| where denials > 5

Response playbook (containment → eradication → recovery)

  1. Triage / scope. For an integrity-monitoring alert, pull actualMeasurements vs policyMeasurements — determine whether it's an approved change (kernel update) or unexpected drift. For attestation-denial spikes, identify the workload principal and image digest.
  2. Contain. Isolate the instance: remove its service-account bindings, apply a deny-all firewall/network tag, and revoke any short-lived attestation tokens by rotating the KMS key / disabling the WIP provider so no further secrets release. For a node, cordon + drain the GKE node.
  3. Preserve evidence. Snapshot the boot disk (note: guest memory is encrypted and not host-dumpable — that's by design; rely on disk + logs). Export relevant audit, integrity, and attestation logs.
  4. Eradicate. If boot tampering is confirmed, do not "clean" — destroy and rebuild from a known-good, signed image; the whole point of measured boot is that you now know the current image is untrusted. Rotate secrets that could have been exposed.
  5. Root-cause the trust chain. Confirm the platform TCB version, whether SNP/TDX integrity was actually on, and whether the release policy pinned image digest + dbgstat. Tighten the attribute condition if it was too loose.
  6. Recover. Redeploy on SEV-SNP/TDX with enforced Shielded VM, re-baseline integrity policy from the new golden image, and confirm attestation succeeds end-to-end before restoring traffic.
  7. Hardening / post-incident. Push the missing org policy (compute.requireShieldedVm), add Binary Authorization if a rogue image was involved, and add/upgrade the detections above.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What problem does confidential computing solve that at-rest and in-transit encryption do not?
Answer: It protects data in use — data that is decrypted and sitting in RAM/registers while being processed. At-rest encryption protects disks and in-transit encryption (TLS) protects the wire, but between those two, data lives in plaintext in memory where a privileged adversary — the hypervisor, host kernel, a cloud operator, or a co-tenant via side channel — can read it. Confidential computing uses a hardware TEE to keep that in-use data encrypted in DRAM with a CPU-resident key, closing the third gap so the full lifecycle is covered.
Signal: Do they understand the three states of data and place the technology correctly.

Q2. Difference between a Confidential VM and a Shielded VM on GCP?
Answer: They're orthogonal. A Shielded VM protects boot integrity — secure boot (signature-checked bootloader/kernel), a vTPM providing measured boot, and integrity monitoring that alerts on drift; it answers "did this VM boot the software I expect?" A Confidential VM protects runtime memory by encrypting the guest's RAM with a per-VM key via AMD SEV-SNP or Intel TDX, so the hypervisor can't read data in use. You typically want both — Confidential VMs actually enable Shielded features by default — because one covers boot-time tampering and the other covers runtime memory disclosure.
Signal: Avoids conflating the two, which is a very common mistake.

Q3. What is remote attestation and why is it essential?
Answer: Attestation is the CPU producing a hardware-signed report proving that a genuine, up-to-date TEE is running specific measured code. It's essential because a TEE alone is worthless if you can't verify it — a relying party must confirm, before releasing secrets or data, that the workload really is inside authentic hardware running the expected code. Verification checks the signature up to the hardware root (AMD VCEK / Intel PCK), a fresh nonce for anti-replay, the boot measurement against an allow-list, and a minimum TCB version. Only then do you trust the environment.
Signal: Recognizes attestation, not encryption, as the trust-establishing mechanism.

Q4. Name the AMD SEV generations and what each added.
Answer: SEV (2017) encrypts guest memory with a per-VM key but leaves register state exposed on VMEXIT. SEV-ES (2018) adds encryption/integrity of the guest register state (the VMSA) on world switches. SEV-SNP (2020) adds full memory integrity via the Reverse Map Table — preventing the hypervisor from remapping, aliasing, or replaying pages — plus interrupt protection and TCB rollback protection. SNP is the modern baseline; plain SEV's lack of integrity was exploitable (SEVerity).
Signal: Knows the evolution and why integrity (SNP) matters, not just names.

Q5. Why must Confidential VMs use --maintenance-policy=TERMINATE, and what's the trade-off?
Answer: The per-VM memory-encryption key is generated inside the SoC and never leaves it, so the VM's encrypted state historically can't be live-migrated to another host. When Google needs to do host maintenance, the confidential VM is stopped (TERMINATE) rather than migrated. The trade-off is availability: you must architect for abrupt stops — use managed instance groups, graceful shutdown handling, and stateless or replicated designs — because you lose seamless live migration that regular VMs enjoy.
Signal: Understands the operational cost of the security guarantee.

Tier 2 — System design / trade-offs

Q6. Design a system where three companies jointly compute on their combined data without any party (or the cloud) seeing another's raw data.
Answer: Use Confidential Space on Confidential VMs (SEV-SNP/TDX). Publish the exact analytic container image and its digest; each party independently reviews it. The workload runs in a TEE and requests an attestation token proving the exact image_digest, genuine confidential hardware, and dbgstat == disabled-since-boot (no operator shell). Each party binds their data's decryption key in Cloud KMS to a Workload Identity Pool condition that only issues the key when the token's claims match. So each party's KMS releases their key only to the attested, unmodified, non-debuggable workload — not to each other and not to a Google operator. Outputs are aggregated results only. Add Binary Authorization so the image is provably from a trusted build. The residual risks are side channels and a malicious-but-attested image, so the image must be jointly reviewed and minimal.
Signal: Can compose attestation + KMS conditions + Confidential Space into a real MPC/clean-room design.

Q7. When would you choose SGX enclaves over Confidential VMs, and vice versa?
Answer: Choose SGX (process TEE) when you need the smallest possible TCB — only the enclave code is trusted, the OS/kernel/hypervisor are all excluded — for a small, security-critical routine like key handling or a licensing check, and you can afford to refactor the app into trusted/untrusted halves crossing ECALL/OCALL boundaries. Choose Confidential VMs (VM TEE) for lift-and-shift: you protect an entire unmodified workload/OS with zero code changes, at the cost of a much larger TCB (your whole guest OS is trusted, so a guest RCE defeats it). In practice for cloud data-in-use protection at scale, VM TEEs win on operability; SGX wins for tiny, high-assurance secrets with a hostile OS assumption — and SGX's side-channel history (Foreshadow, SGAxe) is a real cost.
Signal: Trades TCB size against operability and knows the programming-model cost.

Q8. How do you make attestation actually secure? What does a naive implementation get wrong?
Answer: A naive implementation "enables the TEE" and assumes it's protected, or verifies a quote's signature but nothing else. A correct verifier must: (1) validate the certificate chain to the genuine hardware root and honor revocation; (2) check a fresh nonce to stop replay of old quotes; (3) compare the measurement to an allow-list of expected images; (4) enforce a minimum TCB / security-version number so patched-but-downlevel platforms are rejected (this is what stops post-SGAxe forged quotes); and (5) gate secret release on all of the above plus workload-specific claims like image digest and debug status. Miss the nonce → replay; miss the TCB check → accept broken platforms; skip verification entirely → the TEE is theater.
Signal: Depth on the failure modes that make real deployments insecure.

Q9. Confidential GKE Nodes protect node memory. What does that NOT protect, and how do you close the gaps?
Answer: It encrypts the node's RAM against the host, but it does not: (1) encrypt pod-to-pod traffic — you still need mTLS / a service mesh (Istio) and NetworkPolicy; (2) protect a pod from another compromised pod on the same node — kernel namespaces still share the node, so a container escape or a noisy-neighbor pod is in scope, mitigated with strong pod security standards, gVisor/sandboxing, and pod-per-node placement for sensitive workloads; (3) protect against a malicious image — add Binary Authorization; or (4) protect secrets at the app layer — use Secret Manager with attestation-gated release. Confidential nodes are one layer; you compose it with mesh mTLS, admission control, and workload isolation.
Signal: Understands the precise scope boundary and defense-in-depth.

Q10. How does Assured Workloads relate to Confidential Computing — same thing?
Answer: No — they solve different problems and complement each other. Confidential Computing is a technical control: hardware memory encryption reducing the TCB so the hypervisor/insider can't read data in use. Assured Workloads is an administrative/sovereignty control plane: it pins a resource folder to a compliance regime (FedRAMP High, IL5, CJIS, EU data boundary) and enforces data residency, provider-personnel access restrictions (support-staff location/nationality), CMEK, and product allow-lists via Organization Policy. A regulated system often uses both: TEEs for provable technical insider-threat reduction, Assured Workloads for provable jurisdictional/compliance boundaries and auditability. They're distinct layers of the same defense-in-depth story.
Signal: Doesn't conflate technical and compliance controls.

Tier 3 — Expert / deep dive

Q11. Walk through the SEVerity attack and explain why SEV-SNP's Reverse Map Table stops it.
Answer: Plain AMD SEV encrypted guest memory but did not authenticate it — there was no integrity binding between a physical page, its contents, and its owner. A malicious hypervisor (the exact adversary SEV claims to defend against) could therefore remap, alias, or replay ciphertext pages, and because the encryption tweak was address-dependent and reusable, researchers spliced in a chosen ciphertext that decrypted to attacker-controlled instructions, achieving code execution inside the "protected" guest. SEV-SNP introduces the Reverse Map Table (RMP): a hardware-maintained structure recording, for every physical page, its single validated owner and guest-physical mapping. Any attempt by the hypervisor to remap, double-map, or replay a page that violates the RMP triggers a fault. This enforces single-ownership, non-aliasing, and anti-replay — turning confidentiality-only SEV into confidentiality plus integrity, which is why you require SNP.
Signal: Deep understanding that confidentiality without integrity is exploitable, at the mechanism level.

Q12. Are TEEs vulnerable to Spectre/Meltdown-class side channels? How do you reason about residual risk?
Answer: Yes. TEEs stop architectural reads of plaintext by privileged software, but transient-execution and microarchitectural side channels (Foreshadow/L1TF against SGX, Meltdown/Spectre broadly, CipherLeaks' ciphertext channel against SEV, ÆPIC's uninitialized-MMIO leak) can exfiltrate secrets without an architectural read — via cache timing, speculative loads, or observable ciphertext. The residual-risk reasoning: (1) the TEE reduces but doesn't eliminate the attack surface; (2) you require patched microcode and enforce a minimum TCB version in attestation so downlevel platforms are rejected; (3) you avoid SMT co-residency for the most sensitive workloads (core scheduling); (4) you keep the in-guest software constant-time and side-channel-hardened as if the host is observing, because for these channels it effectively is; and (5) you accept that TEEs raise the bar and shrink the TCB rather than providing absolute secrecy. See infra/Side-Channel/Spectre-Meltdown for the underlying transient-execution mechanics.
Signal: Nuanced — doesn't oversell TEEs, cross-links the attack class.

Q13. Design an attestation-gated secret-release flow and enumerate what each check defends against.
Answer: Flow: the TEE workload requests a hardware attestation report with a verifier-supplied nonce; Google's Attestation Verification Service validates the signature to the AMD/Intel root, checks the measurement and TCB SVN, and mints a short-lived OIDC attestation token with claims (hwmodel, image_digest, dbgstat, support_attributes). The workload presents that token to a Workload Identity Pool provider whose attributeCondition requires hwmodel == GCP_AMD_SEV, image_digest == <pinned sha256>, and dbgstat == disabled-since-boot; on match it federates to a service account allowed to AccessSecretVersion in Secret Manager / decrypt with a KMS key. Each check's purpose: signature-to-root defeats fake/software TEEs; nonce defeats quote replay; TCB SVN defeats downlevel/known-broken platforms (post-SGAxe); image_digest pin defeats swapped/rogue code (even if attested); dbgstat defeats an operator attaching a debugger to peek at memory; short TTL limits token theft. Miss any one and you have a corresponding bypass.
Signal: Can enumerate defenses per control — senior depth.

Q14. A malicious insider at the cloud provider wants your data. Precisely what does Confidential Computing stop, and what does it not?
Answer: Stops: reading your VM's plaintext memory via the hypervisor, host kernel, or console (they see ciphertext); tampering with/injecting into guest memory (with SNP/TDX integrity); and — with attestation + dbgstat gating — attaching a debugger to a Confidential Space workload to release secrets. It cryptographically removes host-software confidentiality from your TCB. Does not stop: an insider with the ability to deny service (they still schedule and can stop your VM), observe metadata and access patterns (timing, page-fault, I/O side channels), exploit a microarchitectural side channel on shared hardware, or subvert the hardware root of trust with physical fault injection (faulTPM-class, though at real physical/operational cost). It also does nothing if the workload itself is compromised (RCE inside the guest) or the image is malicious but attested. So it's a strong confidentiality/integrity control against privileged software adversaries, not an availability or side-channel panacea — pair it with attestation policy, Assured Workloads for personnel controls, and app-layer hardening.
Signal: Crisp threat-model boundary — knows exactly what's in and out of scope.

Q15. Compare AMD SEV-SNP, Intel TDX, and Intel SGX across TCB, integrity, migration, and use case — and state what you'd standardize on for a data-processing platform.
Answer: SGX is a process TEE: smallest TCB (only enclave code; OS excluded), strong isolation, but requires app refactoring, has a hard memory limit and heavy side-channel history (Foreshadow, SGAxe, ÆPIC); best for tiny high-assurance secrets. SEV-SNP is a VM TEE: whole-guest TCB, integrity via RMP, transparent lift-and-shift, no live migration, broad AMD availability on GCP N2D/C3D. TDX is Intel's VM TEE analog: TDX-Module-mediated TDs, MKTME encryption, integrity + access control, on GCP C3 Sapphire Rapids. For a general data-processing platform I'd standardize on VM-based TEEs (SEV-SNP or TDX) — they give data-in-use protection with zero app changes, integrated with Confidential Space for attestation-gated data sharing — and reserve SGX for a narrow enclave (e.g., a root-key custodian) only if the minimal-TCB requirement justifies the engineering cost. I'd require the integrity-protected variant (SNP/TDX, never plain SEV), enforce attestation with TCB-version policy, and architect around the no-live-migration constraint. Between SNP and TDX I'd let hardware availability/region and existing microcode-patch cadence decide, keeping the attestation-verification abstraction provider-agnostic.
Signal: Synthesizes the whole space into a defensible standardization decision with trade-offs.

9. Key Takeaways

  • Confidential Computing protects data in use by encrypting RAM with a CPU-resident per-VM key; it complements at-rest (disk/KMS) and in-transit (TLS) — the full three-state lifecycle.
  • The threat model is the privileged adversary: malicious/compromised hypervisor, cloud insider, co-tenant. It shrinks your TCB to CPU silicon + microcode (+ your guest OS for VM-TEEs).
  • Confidential VM ≠ Shielded VM. Confidential VM = runtime memory encryption (SEV-SNP/TDX). Shielded VM = boot integrity (secure boot + vTPM + integrity monitoring). Use both; enforce via org policy.
  • Require the integrity-protected variant — SEV-SNP or TDX, never plain SEV. Confidentiality without integrity was exploited (SEVerity); SNP's Reverse Map Table fixes it in hardware.
  • Attestation is the load-bearing control. Verify signature-to-hardware-root, a fresh nonce, the measurement allow-list, and a minimum TCB version — then gate secret release on the token (pin image_digest and dbgstat).
  • TEEs are not side-channel-proof. Foreshadow, SGAxe, ÆPIC, CipherLeaks, CacheWarp are real; keep guest crypto constant-time and assume the host observes side effects. Cross-ref Spectre/Meltdown.
  • A perfectly attested workload can still be malicious — pair attestation with Binary Authorization / signed provenance so you only trust code you built.
  • Confidential GKE encrypts node RAM but not pod-to-pod traffic or same-node pod isolation — add mTLS, NetworkPolicy, and sandboxing.
  • Assured Workloads is compliance/sovereignty, not a TEE — it enforces data residency and provider-personnel controls administratively; use it alongside Confidential Computing.
  • Confidential VMs trade away live migration (--maintenance-policy=TERMINATE) — design for abrupt host-maintenance stops with MIGs and stateless/replicated architecture.
