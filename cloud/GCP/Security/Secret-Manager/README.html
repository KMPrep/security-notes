GCP Secret Manager — Secrets, Rotation, and the Secrets-Management Problem

Cloud / Domain: GCP · Data protection · Identity & credential hygiene
Service: secretmanager.googleapis.com (regional endpoints: secretmanager.<region>.rep.googleapis.com)
Anchor frameworks/refs: NIST SP 800-57 (key mgmt), NIST SP 800-63B (secrets), CIS GCP Benchmark, MITRE ATT&CK T1555.006 / T1552 · Cross-refs: ../../../../supply_chain/README.md, ../../../../supply_chain/Dependency-Confusion/README.md, ../../../../supply_chain/Sigstore-And-Signing/README.md

1. Overview

Secret Manager is GCP's managed store for small blobs of sensitive data — database passwords, API keys, OAuth client secrets, TLS private keys, service-account keys you could not avoid minting — with a clean separation between the secret (a named container plus IAM and rotation policy), the version (an immutable, individually-addressable payload up to 64 KiB), and the access (a per-request, audited AccessSecretVersion call). It exists because the alternative — secrets pasted into source, baked into container images, dropped in environment variables committed to a repo, or emailed around — is the single most reliable way large organizations get breached. Real incidents keep proving the point: Uber (2016) lost 57M records because AWS keys sat in a private GitHub repo; Toyota (2022) left a database access key in a public repo for nearly five years; CircleCI (2023) had to force a global rotation of every customer secret after a laptop compromise stole session tokens. At scale, the hard problems are not "where do I put one password" but secret sprawl (thousands of secrets, nobody knows who can read what), secret zero (how does a workload prove identity to fetch its first secret without a secret), and detection (a valid credential read looks exactly like a legitimate one). Secret Manager solves storage, versioning, IAM-gated access, CMEK encryption, and rotation eventing; it does not by itself solve secret zero (that's Workload Identity) or rotation logic (you write that).

TL;DR
- A secret is a container (name + IAM + rotation + replication policy); a version is the immutable payload; you always resolve secrets/NAME/versions/{N|latest}.
- Access is gated by roles/secretmanager.secretAccessor and is best granted per-secret, not project-wide — project-level accessor = every service can read every secret.
- Secrets are always encrypted at rest; CMEK (user-managed replication + Cloud KMS key) gives you the key-destruction kill-switch and crypto-shredding.
- Rotation in GCP is event-only: Secret Manager publishes to Pub/Sub on a schedule; you run the Cloud Function that mints the new credential and adds a version. Unlike AWS, there is no managed rotation Lambda.
- The right way to fetch a secret from GKE/Cloud Run/GCE is a short-lived Workload Identity token, never a downloaded SA key — solving secret zero with the platform, not another secret.
- Enclose Secret Manager in a VPC Service Controls perimeter so a stolen credential cannot exfiltrate secrets to an attacker-controlled project.

2. How It Works

2.1 Core mechanics

Secret Manager is a global control plane over regionally-replicated ciphertext. The API is small and the object model is deliberately three-layered so that IAM, rotation, and audit attach at the right granularity.

Object / concept What it is Why it matters
Secret Named resource projects/P/secrets/NAME holding metadata: replication policy, rotation schedule, Pub/Sub topics, labels, CMEK key, IAM policy IAM and rotation bind here; the payload is not stored on the secret itself
Secret version Immutable payload .../secrets/NAME/versions/N; states ENABLED, DISABLED, DESTROYED Versions are how you rotate without breaking readers; destroy crypto-erases the payload but keeps the number
latest alias Resolves to the highest enabled version number Convenient but risky — pinning latest means a bad rotation instantly breaks all readers
Replication policy automatic (Google-managed multi-region) or user-managed (you pick regions) CMEK requires user-managed — you supply a KMS key per location
AccessSecretVersion The read RPC that returns the decrypted payload This is the only call that exposes plaintext; it is a Data Access audit log (off by default)
Rotation policy rotation-period + next-rotation-time + topics Fires a Pub/Sub message; does not change the secret value by itself
Secret Manager service agent service-<PROJECT_NUMBER>@gcp-sa-secretmanager.iam.gserviceaccount.com Needs KMS encrypt/decrypt on CMEK keys and pubsub.publisher on rotation topics
Regional secrets Secrets created against a regional endpoint, data pinned to one region Data-residency and lower-latency; distinct from global secrets with replication policy

Decision logic on a read. When a workload calls AccessSecretVersion, GCP evaluates, in order: (1) is the caller authenticated (valid Google identity / token)? (2) does IAM grant secretmanager.versions.access on that secret (via secretAccessor at the secret, project, folder, or org level, plus any conditions/deny policies)? (3) if a VPC-SC perimeter wraps the project, is the request originating from inside the perimeter (or an allowed access level)? (4) is the version ENABLED? (5) decrypt with the Google-managed or CMEK key (if CMEK, the KMS key must be ENABLED and the service agent must hold cryptoKeyDecrypter). Any failure short-circuits and is logged.

IAM roles you actually use.

Role Grants Use for
roles/secretmanager.secretAccessor versions.access (read payload) The only role a workload should get; grant per-secret
roles/secretmanager.viewer Read metadata, not payload Dashboards, inventory tooling
roles/secretmanager.secretVersionManager Add/enable/disable/destroy versions Rotation functions
roles/secretmanager.admin Full incl. setIamPolicy Break-glass / platform team only
roles/secretmanager.secretVersionAdder Add versions only CI that publishes new secret values

2.2 Concrete configuration

Create secrets (automatic vs user-managed replication) and add a version.

# Global secret, Google-managed multi-region replication (simplest default)
gcloud secrets create db-password \
  --replication-policy="automatic" \
  --labels="env=prod,team=payments"

# Data-residency: pin replicas to specific regions (required before CMEK)
gcloud secrets create db-password-eu \
  --replication-policy="user-managed" \
  --locations="europe-west1,europe-west4"

# Add the payload as a NEW immutable version. Never pass secrets on argv (shell history!)
printf 's3cr3t-value' | gcloud secrets versions add db-password --data-file=-

# Read it back — this is the audited AccessSecretVersion call
gcloud secrets versions access latest --secret="db-password"
gcloud secrets versions access 3 --secret="db-password"   # pin a specific version

Grant per-secret access to exactly one workload identity (least privilege).

# GOOD: bind accessor on the single secret, to the app's dedicated service account
gcloud secrets add-iam-policy-binding db-password \
  --member="serviceAccount:payments-api@my-proj.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"

# BAD (do not do this): project-level accessor = read EVERY secret in the project
# gcloud projects add-iam-policy-binding my-proj \
#   --member="serviceAccount:payments-api@my-proj.iam.gserviceaccount.com" \
#   --role="roles/secretmanager.secretAccessor"

IAM Condition to scope even a project-level grant by resource name (defense against blast-radius when you must grant broadly):

gcloud projects add-iam-policy-binding my-proj \
  --member="serviceAccount:ci-runner@my-proj.iam.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor" \
  --condition='expression=resource.name.startsWith("projects/_/secrets/ci-"),title=ci-secrets-only'

CMEK: user-managed replication + a Cloud KMS key per location. The Secret Manager service agent must be able to use the key.

# 1) Grant the Secret Manager service agent decrypt on the KMS key
gcloud kms keys add-iam-policy-binding payments-key \
  --location="us-central1" --keyring="prod-ring" \
  --member="serviceAccount:service-123456789012@gcp-sa-secretmanager.iam.gserviceaccount.com" \
  --role="roles/cloudkms.cryptoKeyEncrypterDecrypter"

# 2) Create the secret with CMEK (note: needs user-managed replication)
gcloud secrets create db-password-cmek \
  --replication-policy="user-managed" \
  --locations="us-central1" \
  --kms-key-name="projects/my-proj/locations/us-central1/keyRings/prod-ring/cryptoKeys/payments-key"

Version lifecycle (safe rotation hygiene). Disable before destroy so you can roll back; destroy performs cryptographic erasure of the payload.

gcloud secrets versions disable 2 --secret="db-password"    # readers of v2 now fail
gcloud secrets versions destroy 2 --secret="db-password"    # irreversible payload erase

Configure scheduled rotation eventing (this only notifies; the function does the work):

# Secret Manager service agent must be able to publish to the topic
gcloud pubsub topics add-iam-policy-binding secret-rotation \
  --member="serviceAccount:service-123456789012@gcp-sa-secretmanager.iam.gserviceaccount.com" \
  --role="roles/pubsub.publisher"

gcloud secrets update db-password \
  --add-topics="projects/my-proj/topics/secret-rotation" \
  --rotation-period="2592000s" \
  --next-rotation-time="2026-08-01T00:00:00Z"   # rotate every 30 days

Accessing a secret from a workload — Python client (short-lived token via ADC).

from google.cloud import secretmanager

client = secretmanager.SecretManagerServiceClient()
name = "projects/my-proj/secrets/db-password/versions/latest"
resp = client.access_secret_version(request={"name": name})
db_password = resp.payload.data.decode("utf-8")   # never log this
# Optionally verify integrity: resp.payload.data_crc32c

Cloud Run — inject a secret without any code change (mounted as env var or file):

# As an environment variable (pinned version recommended over :latest for prod)
gcloud run deploy payments-api --image=us-docker.pkg.dev/my-proj/app:v42 \
  --service-account="payments-api@my-proj.iam.gserviceaccount.com" \
  --set-secrets="DB_PASSWORD=db-password:latest"

# As a mounted file (avoids env-var leakage via /proc/<pid>/environ or crash dumps)
gcloud run deploy payments-api --image=us-docker.pkg.dev/my-proj/app:v42 \
  --set-secrets="/secrets/db/password=db-password:latest"

GKE — Workload Identity + the Secret Store CSI driver (mount, no key files):

# SecretProviderClass mounts db-password at /mnt/secrets-store/db-password
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: gcp-db-secret
spec:
  provider: gcp
  parameters:
    secrets: |
      - resourceName: "projects/my-proj/secrets/db-password/versions/latest"
        path: "db-password"

# The pod's KSA is bound to the GCP SA via Workload Identity (no downloaded key)
apiVersion: v1
kind: ServiceAccount
metadata:
  name: payments-ksa
  annotations:
    iam.gke.io/gcp-service-account: payments-api@my-proj.iam.gserviceaccount.com

Case study 1 — Toyota (2022): the five-year key. A subcontractor uploaded portions of the T-Connect source to a public GitHub repository, and embedded in it was an access key to a data server holding ~296,000 customer email addresses and management numbers. The key sat exposed from December 2017 to September 2022. Nothing about Secret Manager's storage would have helped if the key was still copied into source — but the control that closes this gap is (a) not minting a static key at all (use Workload Identity so there is no key to leak) and (b) secret scanning on every push (gitleaks/trufflehog) so the key never survives the first commit, plus automatic rotation so even a leaked value is short-lived. The failure was upstream of any vault: the credential was long-lived and copyable.

Case study 2 — CircleCI (2023): the secret aggregator. Malware on an engineer's laptop stole a valid, 2FA-backed SSO session token (T1539/T1078). Because CircleCI stores customers' pipeline secrets to inject into builds, the attacker could potentially read every customer's environment variables, project keys, and tokens. CircleCI's response was to rotate all customer secrets and advise customers to rotate everything they'd stored. The lesson for a Secret Manager design: (1) secrets that live in a third-party CI are only as safe as that CI's crown-jewel session tokens — prefer OIDC federation so CI gets a short-lived GCP token rather than a stored long-lived key; (2) design so a mass rotation is a scripted, low-effort event, not a multi-week scramble; (3) Data Access audit logs on AccessSecretVersion are what let you answer "which secrets did the attacker actually read?"

Case study 3 — Codecov (2021): env-var exfiltration in CI. Attackers altered Codecov's bash-uploader script so that, when run in customers' CI, it exfiltrated the entire environment — which in most pipelines includes injected secrets. This is why how you inject matters: a secret set as a build-wide environment variable is readable by any compromised step or dependency, whereas a file mount with tight file permissions, fetched just-in-time by a scoped identity, narrows the window. It also argues for keeping secrets out of CI entirely where possible — let the deployment target (Cloud Run/GKE) fetch its own secrets at runtime via Workload Identity, so CI never holds them.

Top 5 in depth

1) Per-secret secretAccessor, never project-level. The single most common misconfiguration is granting roles/secretmanager.secretAccessor on the project, which makes every principal with that binding able to read every secret in the project. IAM evaluates the role on the resource hierarchy (org → folder → project → secret), so a binding higher up is inherited downward. The correct pattern is gcloud secrets add-iam-policy-binding <secret> --member=<one SA> --role=secretAccessor on each secret, ideally one dedicated service account per workload. Where a broad grant is unavoidable (e.g., a CI runner), constrain it with an IAM Condition on resource.name.startsWith(...) so the grant only covers a naming prefix. Trade-off: per-secret bindings multiply the number of policies to manage — you offset this with Terraform modules and a naming convention, but it is real operational overhead versus one lazy project grant. The blast-radius reduction is worth it: a compromised workload reads its secrets, not the whole org's.

2) Workload Identity instead of downloaded SA keys (solving secret zero). A downloaded service-account JSON key is itself a long-lived, copyable, forgeable secret — the exact thing you're trying to eliminate — and it recreates the "secret zero" bootstrap problem. Instead, let the platform assert identity: GCE/Cloud Run/Cloud Functions get an attached service account and fetch short-lived OAuth tokens from the metadata server; GKE Workload Identity binds a Kubernetes ServiceAccount to a GCP SA so pods get tokens without any key file; and Workload Identity Federation lets external systems (GitHub Actions, on-prem, other clouds) exchange their native OIDC token for a short-lived GCP token — no stored GCP key at all. Enforce it with the org policy constraints/iam.disableServiceAccountKeyCreation. Trade-off: federation setup is fiddly (mapping claims, pool/provider config) and some legacy/off-cloud systems genuinely can't produce a usable OIDC token; for those, a short-TTL key with alerting is the fallback, not the default.

3) Rotation via Pub/Sub + Cloud Function — and know it's your job. GCP's rotation is event-only: setting --rotation-period and --add-topics makes Secret Manager publish a message with attribute eventType=SECRET_ROTATE on schedule, but it does not generate a new value. You subscribe a Cloud Function that (a) mints a new credential in the upstream system (DB user, API), (b) AddSecretVersion, (c) verifies the new version works, then (d) disables/destroys the old one. The classic robust pattern is two-set/AWS-style "AWSPENDING→AWSCURRENT" — create-and-test the new credential before cutting over, so a failed rotation never causes an outage. Trade-off: you own the correctness, retries, idempotency, and the "old version still valid during overlap" window; this is more work than AWS's managed rotation Lambdas but gives full control. Always alert if a secret's next-rotation-time is in the past (rotation silently stalled).

4) VPC Service Controls perimeter around Secret Manager. IAM answers "who can access"; VPC-SC answers "from where." Placing secretmanager.googleapis.com inside a service perimeter means that even a valid, correctly-authorized credential cannot be used to read secrets and copy them to a project outside the perimeter — it neutralizes the exfiltration step (T1537/TA0010) that follows most credential theft. You pair it with access levels (e.g., allow only from corporate IP ranges / trusted device posture) for human access. Trade-off: perimeters are notoriously operationally sharp — cross-project and cross-perimeter calls break in non-obvious ways, and you must maintain ingress/egress rules; roll out in dry-run mode first and read the vpcServiceControlsUniqueId denials in the audit log before enforcing.

5) Enable Data Access audit logging and alert on anomalies. By default GCP logs Secret Manager Admin Activity (create, setIamPolicy, destroy) but not AccessSecretVersion, because reads are DATA_READ logs that are off by default and can be high-volume. Without them you cannot answer "which secrets did the attacker read" after a breach — the exact question CircleCI had to answer. Turn them on via the project/org IAM audit config for secretmanager.googleapis.com, then build detections (§6) for volume spikes, access from new principals/regions, and reads immediately following an IAM change. Trade-off: DATA_READ volume and cost — a busy service reading latest every few seconds generates a lot of log lines; mitigate by caching secrets in-process with a sane TTL (which also reduces API calls) and by exporting logs to BigQuery/Chronicle for cheaper retention and querying.

6. Detection & Response

What to log / telemetry sources.
- Cloud Audit Logs — Admin Activity (always on): CreateSecret, AddSecretVersion, SetIamPolicy, DestroySecretVersion, UpdateSecret (rotation/topic changes). Field protoPayload.methodName, protoPayload.authenticationInfo.principalEmail.
- Cloud Audit Logs — Data Access (must enable): google.cloud.secretmanager.v1.SecretManagerService.AccessSecretVersion — the read events. Key fields: resourceName (which secret/version), principalEmail, requestMetadata.callerIp, requestMetadata.callerSuppliedUserAgent.
- VPC-SC violation logs: protoPayload.metadata.violationReason, vpcServiceControlsUniqueId for denied cross-perimeter reads.
- Cloud KMS logs: decrypt calls on the CMEK key (spike = mass read), and any DestroyCryptoKeyVersion.
- SCM / CI secret-scan findings: gitleaks/trufflehog/GitHub push-protection alerts.

Sample detection 1 — anomalous secret access (BigQuery/Log Analytics SQL on exported audit logs). Flags principals reading far more distinct secrets than their baseline, or reading from a new IP.

SELECT
  protopayload_auditlog.authenticationInfo.principalEmail AS principal,
  protopayload_auditlog.requestMetadata.callerIp          AS caller_ip,
  COUNT(DISTINCT protopayload_auditlog.resourceName)       AS distinct_secrets,
  COUNT(*)                                                 AS access_count
FROM `my-proj.security_logs.cloudaudit_googleapis_com_data_access`
WHERE protopayload_auditlog.methodName =
      'google.cloud.secretmanager.v1.SecretManagerService.AccessSecretVersion'
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
GROUP BY principal, caller_ip
HAVING distinct_secrets > 20          -- tune to your baseline
ORDER BY distinct_secrets DESC;

Sample detection 2 — Sigma rule: destroy or IAM-widen on secrets.

title: GCP Secret Manager Destroy or Broad IAM Grant
logsource:
  product: gcp
  service: gcp.audit
detection:
  destroy:
    protoPayload.methodName: 'google.cloud.secretmanager.v1.SecretManagerService.DestroySecretVersion'
  broad_grant:
    protoPayload.methodName: 'google.cloud.secretmanager.v1.SecretManagerService.SetIamPolicy'
    protoPayload.request.policy.bindings|contains: 'roles/secretmanager.secretAccessor'
    protoPayload.resourceName|contains: 'projects/'   # project-level, not per-secret
  condition: destroy or broad_grant
level: high

Sample detection 3 — CI secret-scan gate (gitleaks in a pre-merge job).

# .github/workflows/secret-scan.yml
- name: gitleaks
  uses: gitleaks/gitleaks-action@v2
  env:
    GITLEAKS_ENABLE_UPLOAD_ARTIFACT: "false"

Sample detection 4 — Cloud Logging alert on stalled rotation. Alert when UpdateSecret events stop for secrets that carry a rotation policy, or query the API for nextRotationTime < now on a schedule (a stalled rotation function is silent otherwise).

Response playbook (specific to a suspected secret compromise).
1. Confirm scope (identify): query Data Access logs for every AccessSecretVersion by the suspect principal/IP over the exposure window; list the exact resourceNames read. This is your "what did they get" list.
2. Contain — cut the identity: disable/rotate the compromised principal (SA key, session), remove its IAM bindings, and if a perimeter exists, tighten access levels. If exfil is suspected and VPC-SC was off, enable it now (dry-run then enforce).
3. Contain — cut the secret value: for each affected secret, AddSecretVersion with a fresh value in the upstream system, verify consumers pick it up, then disable (not yet destroy) the exposed version so you can roll back. For CMEK secrets under active abuse, disabling the KMS key is an instant blanket kill-switch (accept the outage).
4. Eradicate: rotate every credential the compromised path could reach, not just the confirmed reads (assume the log may be incomplete if Data Access logging was recently enabled). Remove any downloaded SA keys and replace with Workload Identity. Scrub the leaked value from source/CI history.
5. Recover: re-enable normal access, restore consumers on new versions, and only then destroy the exposed versions (crypto-erase). Confirm no consumer still pins the old version number.
6. Post-incident: add/verify per-secret IAM, rotation policy, Data Access logging, and secret scanning where they were missing; write a detection for the exact pattern observed; run a game-day mass-rotation drill.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What are the three layers of the Secret Manager object model and why does the split matter?
A secret is a named container holding metadata — IAM policy, rotation policy, replication/CMEK, labels. A secret version is the immutable payload (≤64 KiB), addressable by number, with states enabled/disabled/destroyed. Access is a per-request AccessSecretVersion call resolving secrets/NAME/versions/{N|latest}. The split matters because IAM and rotation attach to the container (stable identity) while the value changes over time as immutable versions — so you can rotate without changing who has access, and roll back by re-enabling an older version.
Signal: Do they know versions are immutable and that IAM binds at the secret, not the value.

Q2. Which IAM role does a workload need to read a secret, and what's the least-privilege way to grant it?
roles/secretmanager.secretAccessor, which grants secretmanager.versions.access. Least privilege means binding it on the individual secret to a dedicated service account for that workload — never roles/owner or accessor at the project level, because higher-level bindings are inherited by all secrets. If a broad grant is unavoidable, constrain with an IAM Condition on resource.name.
Signal: Per-secret vs project-level instinct; knowing the exact role name.

Q3. Is data in Secret Manager encrypted, and what does CMEK add?
Yes — all payloads are encrypted at rest by default with Google-managed keys. CMEK lets you supply your own Cloud KMS key (requires user-managed replication, one key per location), which adds: control over the key's lifecycle, the ability to crypto-shred by destroying the key, key-usage audit logs, and satisfying compliance requirements for customer-controlled keys. The Secret Manager service agent must hold cryptoKeyEncrypterDecrypter on the key.
Signal: Understands default encryption exists; CMEK is about control/kill-switch, not "encryption on vs off."

Q4. What is "secret zero" and how does GCP address it?
Secret zero is the bootstrap paradox: to fetch secrets you need a credential, but that credential is itself a secret that must be provisioned and protected. GCP addresses it with platform-issued identity — attached service accounts (metadata tokens), GKE Workload Identity, and Workload Identity Federation for external systems — so the workload proves identity via short-lived tokens without any stored key. The platform, not another secret, is the root of trust.
Signal: Recognizes the recursion and that Workload Identity is the answer, not "store it more carefully."

Q5. Why prefer mounting a secret as a file over an environment variable?
Environment variables are broadly readable — via /proc/<pid>/environ, crash dumps, printenv, child processes, and any compromised build/runtime step (Codecov, Log4Shell classes) — and often leak into logs. A file mount can have restrictive permissions, isn't inherited by child processes, and is less likely to be dumped. Secret Manager on Cloud Run/GKE supports both; files are the safer default for high-value secrets.
Signal: Knows env vars are a leak vector, not just a convenience.

Tier 2 — System design / trade-offs

Q6. Design secret access for a fleet of Cloud Run services so no service can read another's secrets and there are no key files.
Give each service its own dedicated service account (--service-account), so identity is per-service. Store each service's secrets with a naming convention and bind secretAccessor per secret to only that service's SA. Inject via --set-secrets as file mounts (not env) pinned to explicit versions for prod. No downloaded keys — Cloud Run's attached SA yields short-lived metadata tokens; enforce disableServiceAccountKeyCreation org-wide. Wrap the project in a VPC-SC perimeter so a compromised service can't exfiltrate to an outside project, and enable Data Access logging with a per-SA anomaly detection. The result: one service's compromise reads only its own secrets, and there's no static key to steal.
Signal: Ties identity-per-workload + per-secret IAM + no keys + perimeter + detection into a coherent design.

Q7. Walk through a safe automatic rotation for a database password with zero downtime.
Set --rotation-period and --add-topics so Secret Manager publishes SECRET_ROTATE to Pub/Sub. A Cloud Function (with only secretVersionManager, its own SA) subscribes and runs a create-then-cut-over flow: (1) create a new DB user/password (or alter a second credential) in the database; (2) AddSecretVersion with the new value; (3) actively test the new version connects; (4) only then have consumers move to it (they follow latest or you canary-promote); (5) once no connections use the old credential, disable then destroy the old version and drop the old DB user. The overlap window (both credentials valid) is what gives zero downtime. Alert if next-rotation-time goes stale. The key insight: GCP only notifies; correctness of the two-credential overlap is your code's job.
Signal: Knows rotation is event-only in GCP and describes create-test-cutover overlap, not a naive replace-in-place.

Q8. Compare Secret Manager to SSM Parameter Store / Azure Key Vault and when you'd choose each.
Secret Manager: purpose-built for secrets, per-secret IAM, versions, CMEK, Pub/Sub rotation eventing (you write logic). AWS splits it: Secrets Manager (native rotation Lambdas, cross-region replication, higher cost) vs SSM Parameter Store (cheaper, SecureString via KMS, good for config + low-churn secrets, no built-in rotation on standard tier). Azure Key Vault unifies secrets/keys/certs with Event Grid rotation events and managed-identity access. Choose Secret Manager on GCP; on AWS use Secrets Manager when you need managed rotation and Parameter Store for cheap config-grade secrets; Key Vault on Azure especially when you also need HSM-backed keys and cert management in one place.
Signal: Knows AWS's two-service split and the rotation-logic difference (managed vs DIY).

Q9. How would you detect that a compromised service account is bulk-reading secrets, and what limits your visibility?
You need Data Access audit logs (AccessSecretVersion) enabled — they're off by default. Then baseline each principal's normal access (distinct secrets, IP, user-agent, rate) and alert on deviations: sudden spike in distinct secrets read, access from a new IP/region, or reads right after an IAM SetIamPolicy change. The limits: if logging was only recently enabled you can't see historical reads; latest-polling services generate high volume that hides low-and-slow abuse; and cached secrets mean a stolen value can be used without any further Secret Manager call. So you complement with downstream detections (the credential's use in the DB/API) and VPC-SC to block exfil regardless.
Signal: Knows Data Access logs are opt-in and articulates the visibility gaps.

Q10. Terraform stores a secret's value in state. What's the exposure and how do you mitigate?
Terraform writes resource attributes — including a secret's plaintext payload if you set it via Terraform — into terraform.tfstate, so state becomes a secret trove readable by anyone with backend access. Mitigations: use an encrypted remote backend (GCS with CMEK) with tight IAM and locking; avoid putting the secret value in Terraform at all (create the secret container in Terraform but add versions out-of-band via a pipeline or gcloud); use ephemeral/write-only mechanisms where supported; and never output secrets in plan/apply logs. Treat any state file as sensitive and scan it.
Signal: Knows IaC state is a real secret-sprawl vector and gives concrete backend/ownership mitigations.

Tier 3 — Expert / deep-dive

Q11. An attacker has SSRF in a Cloud Run app whose SA has secretAccessor on several secrets. Trace the attack and the layered defenses.
SSRF lets the attacker make the app request the metadata server (http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token) to obtain a short-lived OAuth token for the attached SA (T1552.005), then call AccessSecretVersion for every secret that SA can read (T1555.006), then exfiltrate. Layered defenses: (1) fix the SSRF and validate/deny internal-address egress; (2) scope the SA minimally so the blast radius is only its own secrets (per-secret IAM); (3) VPC-SC so even with the token, reads can't be copied to an outside project; (4) Data Access logging + anomaly detection to catch the bulk read and the unusual caller pattern; (5) mount secrets as files with a cached in-process copy so the token isn't even needed for already-loaded secrets — but that doesn't help new reads, so (2)+(3) carry the weight. The lesson mirrors Capital One: identity/metadata is the new secret.
Signal: Connects SSRF→metadata→token→secrets and layers IAM + VPC-SC + detection rather than one control.

Q12. Explain crypto-shredding with CMEK and its failure modes as both a control and a risk.
With CMEK, destroying (or disabling) the Cloud KMS key renders every secret encrypted under it permanently unreadable — a fast, irreversible "delete everything" without touching each ciphertext, useful for data-subject deletion or emergency containment. Failure modes: (1) it's a DoS if triggered accidentally or maliciously by someone with KMS key admin — separate KMS key IAM from secret IAM and use scheduled-destroy delays so destruction is reversible for a window; (2) key rotation in KMS re-encrypts new writes but old versions stay decryptable under prior key versions, so shredding requires destroying the right key material; (3) if the KMS key is in a different project/region with its own availability, a KMS outage or misconfigured service-agent permission takes your secrets offline. The control's power (instant blanket invalidation) is exactly its risk.
Signal: Understands crypto-shredding mechanics and the availability/DoS dual-nature, plus IAM separation.

Q13. You must let GitHub Actions deploy to GCP and read deploy-time secrets without storing a GCP key in GitHub. How?
Use Workload Identity Federation: create a workload identity pool and an OIDC provider trusting GitHub's token issuer, with an attribute mapping/condition restricting to your specific repo and branch (e.g., assertion.repository == 'org/repo' and ref conditions) to prevent other repos from assuming the identity. GitHub Actions requests its native OIDC token, the google-github-actions/auth action exchanges it for a short-lived GCP access token for a designated service account, and that SA has narrowly-scoped secretAccessor on only the deploy secrets. No long-lived GCP key ever touches GitHub — eliminating the CircleCI-class "stored CI secret" risk. Pitfalls: an over-broad attribute condition (missing repo/branch pin) lets any repo impersonate the SA; and the SA still needs least-privilege IAM, since the token it yields is fully privileged for that SA.
Signal: Knows federation over stored keys, and specifically the attribute-condition pinning that prevents cross-repo abuse.

Q14. How do you run a mass-rotation drill, and why is CircleCI the motivating example?
CircleCI's 2023 breach forced customers to rotate every secret stored in it — those who had scripted, low-friction rotation recovered in hours; those who hadn't spent weeks. A drill: inventory every secret (labels/inventory tooling), ensure each has a rotation function or a documented rotation procedure, then execute a controlled rotation of a non-critical slice end-to-end — mint new, add version, verify consumers, disable old — measuring time-to-rotate and finding consumers that pin old versions or cache indefinitely. Track coverage (what % of secrets can be rotated automatically) as a metric. The goal is to make "rotate everything now" a scripted event, because you will eventually have to, and the failure mode is discovering at incident time that half your secrets have no rotation path.
Signal: Treats rotation as an operational capability to exercise, not a checkbox; cites the real incident.

Q15. VPC-SC protects Secret Manager but breaks a legitimate cross-project pipeline. How do you diagnose and fix without disabling the perimeter?
First read the denials: VPC-SC violations appear in audit logs with violationReason and a vpcServiceControlsUniqueId; find the exact resource, service, and calling project. The fix is not to drop the perimeter but to author precise ingress/egress rules or a perimeter bridge: an egress rule allowing the source project's identity to reach secretmanager.googleapis.com in the target, or an ingress rule permitting the specific SA/source. Roll changes in dry-run mode so you can see what would be denied before enforcing, and add access levels (device/IP) for human callers. If the two projects legitimately share secrets often, put both in the same perimeter or bridge them. The discipline is: perimeters fail closed and opaquely, so operate them log-first and rule-precise rather than loosening globally.
Signal: Knows VPC-SC diagnosis (violation logs, unique id), dry-run, and ingress/egress/bridge instead of disabling.

9. Key Takeaways

  • Grant secretAccessor per-secret to a dedicated SA, never at the project level — inherited broad grants turn one workload compromise into total secret exposure.
  • Kill the SA key file. Downloaded keys are secret zero reincarnated; use Workload Identity / Workload Identity Federation and enforce disableServiceAccountKeyCreation.
  • Rotation in GCP is event-only — Pub/Sub notifies, your Cloud Function mints the new value with a create-test-cutover overlap. AWS gives you managed rotation Lambdas; GCP makes you own the logic.
  • CMEK is a control and a kill-switch, not just "encryption" — it enables crypto-shredding but also a DoS if KMS key IAM isn't separated from secret IAM.
  • VPC Service Controls neutralizes exfiltration: IAM says who, VPC-SC says from where; wrap secretmanager.googleapis.com so a stolen token can't copy secrets out.
  • Data Access logs are off by default — without them you cannot answer "which secrets were read" post-breach; enable, baseline, and alert on anomalies.
  • How you inject matters: file mounts over env vars; pinned versions over latest in prod; cache in-process with a TTL to cut both cost and read exposure.
  • Assume some secrets are already leaked — the real world is secret sprawl (GitGuardian counts millions/year); design for fast, scripted mass rotation and run drills.
  • Secret scanning is upstream of the vault — gitleaks/trufflehog + push protection stop the Toyota-class "key in source" leak before a vault can matter; cross-ref the supply-chain notes.
  • Detection assumes valid credentials — a legitimate read and a stolen-token read look identical, so rely on volume/behavioral anomalies plus downstream credential-use signals, not just auth success.
