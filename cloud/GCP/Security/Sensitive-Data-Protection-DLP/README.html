1. Overview

Sensitive Data Protection (SDP) — the umbrella service that absorbed the API historically called Cloud DLP (dlp.googleapis.com, still dlp_v2) — is Google Cloud's managed service for discovering, classifying, de-identifying, and measuring re-identification risk on sensitive data. Where the repo's dlp/ notes describe DLP as an engineering discipline (classification → policy → enforcement → detection → tuning), SDP is the GCP-native primitive that implements the classification and transformation layers of that discipline for data at rest in BigQuery, Cloud Storage, Datastore/Firestore, and for arbitrary content inspected inline through an API.

It matters at scale because the alternative — hand-rolled regexes scattered across pipelines — silently rots: it misses new credential formats, produces no consistent likelihood signal, and offers no reversible tokenization or formal re-identification math. SDP centralizes 150+ maintained infoType detectors, a de-identification engine (masking, redaction, bucketing, date shifting, format-preserving encryption, KMS-wrapped deterministic tokenization), a risk analysis engine (k-anonymity, l-diversity, k-map, δ-presence), and Discovery — a continuous, org-wide data profiler that maps where your sensitive data actually lives without you writing a single job. The senior-engineer framing: SDP is not "a scanner," it is a data-governance control plane whose outputs feed Security Command Center, IAM/VPC-SC boundary decisions, and downstream analytics de-risking.

TL;DR
- SDP has three engines: inspection (find infoTypes + likelihood), de-identification (transform: mask/redact/bucket/date-shift/FPE/deterministic crypto tokenization), and risk analysis (k-anonymity, l-diversity, k-map, δ-presence).
- Content inspection API = synchronous, ≤0.5 MB/item, for inline/real-time; inspection jobs = async batch over BigQuery/GCS; Discovery = continuous org profiler that finds data you didn't know you had.
- Reversible tokenization is crypto-based (AES-SIV deterministic, FF1/FFX format-preserving) keyed by a KMS-wrapped DEK — the KMS key is the crown jewel; its compromise = mass re-identification.
- Built-in infoTypes (CREDIT_CARD_NUMBER, US_SOCIAL_SECURITY_NUMBER, EMAIL_ADDRESS) return a likelihood (VERY_UNLIKELY…VERY_LIKELY), not a boolean — tune the threshold per action.
- The two silent killers: incomplete infoType coverage (you don't detect what you didn't model) and includeQuote=true leaking the raw PII into findings storage (secondary exposure).

2. How It Works (deep)

2.1 Core mechanics

SDP is a regional/global managed API with no data-plane agent required for GCP-resident data (it reads through service-account IAM). Three logical engines share one config vocabulary — InspectConfig, DeidentifyConfig, RiskAnalysisJobConfig — and two reusable, versionable objects: inspect templates and de-identify templates (so policy is decoupled from job invocation).

Object / concept What it is Notes
infoType detector A named classifier for a data category (CREDIT_CARD_NUMBER, EMAIL_ADDRESS, US_SOCIAL_SECURITY_NUMBER) 150+ built-in; ML + pattern + checksum + context words
Custom infoType User-defined detector: regex, dictionary (word list), or storedType (large dictionary/LDM) Add detectionRules (hotword context) to raise/lower likelihood
Likelihood Confidence bucket per finding VERY_UNLIKELY, UNLIKELY, POSSIBLE, LIKELY, VERY_LIKELY

ContentItem Inline bytes/string/table for the content API ≤0.5 MB/request (inspectContent)

Action What to do with findings saveFindings (BQ), pubSub, publishToStackdriver, publishSummaryToCscc

Evaluation/decision logic for a finding. Each detector runs a pipeline: (1) pattern match (regex/format), (2) checksum/validation where applicable (Luhn for CREDIT_CARD_NUMBER, area/group validity for SSN), (3) ML/context scoring — proximity of hotwords ("SSN:", "card no.") and negative context raise/lower likelihood, (4) emit Finding{infoType, likelihood, location, quote?}. A finding survives to output only if its likelihood ≥ minLikelihood. This is why SDP output is probabilistic and must be tuned — same lesson as dlp/Content-Inspection (regex vs ML vs fingerprinting trade-offs) applied to a managed engine.

2.2 Concrete configuration (mandatory, substantial)

(a) Inline content inspection — the inspectContent request as JSON. Detects three built-ins plus one custom regex infoType with a hotword rule, at POSSIBLE minimum, and refuses to echo raw quotes (includeQuote:false).

{
  "item": {
    "value": "Cardholder Jane Roe, card 4532015112830366, SSN 078-05-1120, jane.roe@example.com, EMP# A1234567"
  },
  "inspectConfig": {
    "infoTypes": [
      { "name": "CREDIT_CARD_NUMBER" },
      { "name": "US_SOCIAL_SECURITY_NUMBER" },
      { "name": "EMAIL_ADDRESS" }
    ],
    "customInfoTypes": [
      {
        "infoType": { "name": "ACME_EMPLOYEE_ID" },
        "regex": { "pattern": "A[0-9]{7}" },
        "likelihood": "POSSIBLE"
      }
    ],
    "ruleSet": [
      {
        "infoTypes": [ { "name": "ACME_EMPLOYEE_ID" } ],
        "rules": [
          {
            "hotwordRule": {
              "hotwordRegex": { "pattern": "(?i)(emp#|employee id)" },
              "proximity": { "windowBefore": 20 },
              "likelihoodAdjustment": { "fixedLikelihood": "VERY_LIKELY" }
            }
          }
        ]
      }
    ],
    "minLikelihood": "POSSIBLE",
    "includeQuote": false,
    "limits": { "maxFindingsPerRequest": 100 }
  }
}

# Call it. Note the region-scoped parent; use a regional endpoint to keep data in-region.
gcloud auth print-access-token > /tmp/tok
curl -s -X POST \
  "https://dlp.googleapis.com/v2/projects/acme-prod/locations/us-central1/content:inspect" \
  -H "Authorization: Bearer $(cat /tmp/tok)" \
  -H "Content-Type: application/json" \
  -d @inspect-request.json

Commentary: ruleSet.hotwordRule promotes an otherwise-weak A[0-9]{7} regex to VERY_LIKELY only when "EMP#"/"employee id" sits within 20 chars — this is how you kill false positives on a naïve pattern. includeQuote:false is the single most important line for a config whose findings land in shared storage.

(b) A BigQuery inspection job with sampling and fan-out to Pub/Sub + SCC.

# job-config.yaml  (gcloud dlp jobs create --job-config=... expects the RPC shape)
inspectJob:
  storageConfig:
    bigQueryOptions:
      tableReference:
        projectId: acme-prod
        datasetId: crm
        tableId: customers
      rowsLimitPercent: 10          # sample 10% — a security/cost trade-off (see §4)
      sampleMethod: RANDOM_START
      identifyingFields:
        - name: customer_id         # carried into findings so you can locate the row
  inspectConfig:
    infoTypesFromTemplate: false
    infoTypes:
      - { name: CREDIT_CARD_NUMBER }
      - { name: US_SOCIAL_SECURITY_NUMBER }
      - { name: EMAIL_ADDRESS }
    minLikelihood: LIKELY
  actions:
    - saveFindings:
        outputConfig:
          table: { projectId: acme-sec, datasetId: dlp_findings, tableId: crm_scan }
    - pubSub:
        topic: projects/acme-sec/topics/dlp-findings
    - publishSummaryToCscc: {}      # summary only — never raw quotes to SCC

gcloud dlp jobs create \
  --project=acme-prod \
  --location=us-central1 \
  --job-config=job-config.yaml
# List / inspect status
gcloud dlp jobs list --project=acme-prod --location=us-central1 --filter="state=DONE"

Commentary: rowsLimitPercent:10 is the classic footgun — it bounds cost but guarantees you will miss sensitive values in the unsampled 90%. Use sampling for profiling, full scans (rowsLimitPercent omitted) for compliance attestation. publishSummaryToCscc sends counts/infoTypes, not the data — correct default.

(c) De-identify: mask + KMS-wrapped deterministic tokenization + date shift. This is the payload you POST to content:deidentify.

{
  "deidentifyConfig": {
    "recordTransformations": {
      "fieldTransformations": [
        {
          "fields": [ { "name": "ssn" } ],
          "primitiveTransformation": {
            "cryptoDeterministicConfig": {
              "cryptoKey": {
                "kmsWrapped": {
                  "wrappedKey": "CiQA... (base64 KMS-encrypted 256-bit DEK)",
                  "cryptoKeyName": "projects/acme-sec/locations/us-central1/keyRings/dlp/cryptoKeys/tok-ssn"
                }
              },
              "surrogateInfoType": { "name": "SSN_TOKEN" }
            }
          }
        },
        {
          "fields": [ { "name": "pan" } ],
          "primitiveTransformation": {
            "cryptoReplaceFfxFpeConfig": {
              "cryptoKey": { "kmsWrapped": {
                "wrappedKey": "CiQB...",
                "cryptoKeyName": "projects/acme-sec/locations/us-central1/keyRings/dlp/cryptoKeys/tok-pan"
              } },
              "commonAlphabetConfig": "NUMERIC",
              "surrogateInfoType": { "name": "PAN_TOKEN" }
            }
          }
        },
        {
          "fields": [ { "name": "card_last4" } ],
          "primitiveTransformation": {
            "characterMaskConfig": {
              "maskingCharacter": "#",
              "numberToMask": 0,
              "reverseOrder": false
            }
          }
        },
        {
          "fields": [ { "name": "dob" } ],
          "primitiveTransformation": {
            "dateShiftConfig": {
              "lowerBoundDays": -30,
              "upperBoundDays": 30,
              "context": { "name": "patient_id" },
              "cryptoKey": { "kmsWrapped": {
                "wrappedKey": "CiQC...",
                "cryptoKeyName": "projects/acme-sec/locations/us-central1/keyRings/dlp/cryptoKeys/dateshift"
              } }
            }
          }
        }
      ]
    }
  },
  "item": {
    "table": {
      "headers": [ {"name":"ssn"}, {"name":"pan"}, {"name":"card_last4"}, {"name":"dob"} ],
      "rows": [ { "values": [
        {"stringValue":"078-05-1120"},
        {"stringValue":"4532015112830366"},
        {"stringValue":"4111111111111234"},
        {"dateValue":{"year":1984,"month":7,"day":2}}
      ] } ]
    }
  }
}

Commentary on each transform:
- cryptoDeterministicConfig (AES-SIV) — same plaintext → same token every time, enabling JOINs/dedup on tokens without exposing SSNs. Reversible with the same KMS-wrapped key via reidentifyContent. Deterministic = supports analytics; it also leaks equality (frequency analysis on rare values), so pair with access controls.
- cryptoReplaceFfxFpeConfig (FF1/FFX FPE) — format-preserving: a 16-digit PAN tokenizes to another 16-digit string in the same alphabet, so it survives schema/NUMERIC constraints and legacy validation. Reversible. Weaker than SIV on very short domains (small message space).
- characterMaskConfig — irreversible display masking; here numberToMask:0 masks all chars. Use for last-4 shown in UIs.
- dateShiftConfig with context:patient_id — shifts each date by a per-entity random offset (deterministic given the same context + key), so a patient's events keep their relative spacing (day-30 fever still 30 days before discharge) while the absolute date is fake. Essential for longitudinal medical/HIPAA analytics.

(d) Generate a KMS-wrapped DEK (the production key path).

# 1) Create a 256-bit AES key locally, wrap it with a KMS symmetric key.
openssl rand 32 > dek.bin
gcloud kms encrypt \
  --project=acme-sec --location=us-central1 \
  --keyring=dlp --key=tok-ssn \
  --plaintext-file=dek.bin --ciphertext-file=dek.wrapped
base64 -w0 dek.wrapped     # -> paste into wrappedKey above
rm -f dek.bin              # never keep the raw DEK on disk

(e) Discovery — continuous org profiling, no per-job authoring.

# Create an org-level discovery config that profiles all BigQuery tables,
# writing data profiles to SCC and a Pub/Sub topic on high-sensitivity finds.
gcloud dlp discovery-configs create \
  --organization=123456789012 \
  --location=us \
  --discovery-config-from-file=discovery.yaml

# discovery.yaml
displayName: org-bq-profiler
orgConfig:
  location: { organizationId: "123456789012" }
targets:
  - bigQueryTarget:
      filter: { otherTables: {} }        # everything not explicitly excluded
      cadence:
        schemaModifiedCadence: { frequency: UPDATE_FREQUENCY_DAILY }
        tableModifiedCadence:  { frequency: UPDATE_FREQUENCY_DAILY }
actions:
  - exportData: { profileTable: { projectId: acme-sec, datasetId: profiles, tableId: bq } }
  - pubSubNotification:
      topic: projects/acme-sec/topics/dlp-profiles
      event: NEW_PROFILE
      pubsubCondition:
        expressions:
          conditions:
            - minimumSensitivityScore: HIGH_SENSITIVITY_SCORE

Commentary: Discovery is the answer to "we have 40,000 BigQuery tables and no idea which hold PII." It maintains TableDataProfile rows (sensitivity score, predicted infoTypes, freshness) and only pages you (minimumSensitivityScore: HIGH) when a new high-sensitivity table appears — the org-scale replacement for hand-authored jobs.

3. Real-World Incidents & CVEs

DLP/de-identification failures are usually process and coverage failures, not memory-corruption CVEs — the "vulnerability" is a missing or misconfigured control. The table mixes named breaches, a named DLP-relevant CVE class, and de-identification research.

Netflix Prize re-identification (Narayanan & Shmatikov) 2008 "Anonymized" ratings joined to IMDb aux data Anonymization ≠ removing names; use k-map / δ-presence vs aux population
Massachusetts GIC / Sweeney re-identification 1997 87% of US uniquely ID'd by ZIP+DOB+sex Quasi-identifiers matter → k-anonymity + generalization/bucketing
Capital One (Paige Thompson) 2019 SSRF → over-permissioned role read 100M records in S3 Scan buckets (Discovery) + tokenize at rest, least-priv SA
Strava heatmap 2018 Aggregated location "anonymous" data exposed bases Aggregation ≠ safe; measure re-id risk before publish
GitHub secret sprawl (industry-wide) ongoing API keys/PANs committed to repos & logs AUTH_TOKEN/custom infoType inspection in CI + log scrubbing
Optus (Australia) breach 2022 Unauthenticated API exposed passport/license IDs Discovery to find PII stores; de-identify non-prod copies

Case study 1 — Netflix Prize (why risk analysis exists). Netflix released 100M movie ratings with names stripped, believing that sufficient. Narayanan and Shmatikov showed that a handful of ratings + dates, cross-referenced against public IMDb reviews, uniquely re-identified users and exposed inferred political/sexual preferences. The failure is exactly what SDP's k-map and δ-presence measure: uniqueness of quasi-identifiers against an auxiliary population. Had they modeled (rating pattern, timestamp) as quasi-identifiers and required a minimum k against the moviegoing population, the release would have been blocked or forced further generalization (date bucketing).

Case study 2 — Capital One 2019. An SSRF on a misconfigured WAF let the attacker obtain temporary credentials for an over-permissioned role, which then read 100M+ customer records (including SSNs and linked bank accounts) from S3. Two SDP-shaped controls would have blunted it: Discovery continuously profiling buckets to flag "this bucket holds SSNs at HIGH sensitivity" (posture + tighter IAM/VPC-SC), and tokenization at rest so that even a full read returns FPE/SIV tokens, not raw SSNs — turning a catastrophic breach into a low-value token dump (attacker still needs the KMS key).

Case study 3 — PAN-in-logs (the recurring PCI finding). A payments service masks PANs in its UI but a debug logger serializes the full request object, writing 16-digit PANs into Cloud Logging, which fans out to a long-retention BigQuery sink. A QSA finds it during audit. The fix pattern: an inline SDP de-identify step on the log-ingestion pipeline (Dataflow → deidentifyContent with CREDIT_CARD_NUMBER → FPE), so PANs are tokenized before durable storage, and a scheduled inspection job on the logs BigQuery dataset that alarms if any CREDIT_CARD_NUMBER finding at LIKELY+ ever appears (regression detector).

Top 5 in depth

1) KMS-wrapped DEK + CMEK for tokenization (threat #4). The reversible transforms (cryptoDeterministicConfig, cryptoReplaceFfxFpeConfig, dateShiftConfig) are only as strong as their key. In production you never use unwrappedCryptoKey (raw key on the wire) — you use kmsWrapped: a 256-bit DEK encrypted by a Cloud KMS symmetric key. At transform time SDP calls kms.decrypt, holds the raw DEK in ephemeral memory, and discards it. Implementation: create the keyring/key, wrap the DEK (see §2.2(d)), grant the SDP service agent roles/cloudkms.cryptoKeyDecrypter on that one key only, enable key rotation, and route cloudkms.googleapis.com API access through VPC Service Controls so a stolen SA can't call KMS from outside the perimeter. Trade-off: rotation breaks determinism — tokens made under key v1 won't match tokens under v2, so schedule rotation with a re-tokenization plan, or keep the tokenization key long-lived but tightly controlled (and accept the concentrated risk). This is the single highest-value control and the single biggest blast radius.

2) Risk analysis before any external/analytics release (threats #7, #8). De-identification without measurement is theater — the Netflix and Sweeney cases prove uniqueness, not names, is the enemy. Before releasing a dataset, run a RISK_ANALYSIS_JOB declaring quasi-identifiers and sensitive fields:

riskJob:
  sourceTable: { projectId: acme-prod, datasetId: research, tableId: cohort }
  privacyMetric:
    kAnonymityConfig:
      quasiIds:
        - { field: { name: zip } }
        - { field: { name: birth_year } }
        - { field: { name: gender } }
  actions:
    - saveFindings: { outputConfig: { table: { projectId: acme-sec, datasetId: risk, tableId: cohort_k } } }

Read the smallest equivalence-class size (that's k); if k=1 you have unique rows → generalize (bucket ZIP to 3 digits, birth_year to decade) and re-run. Then run l-diversity to ensure each class has ≥ l distinct sensitive values (else attribute disclosure even at high k), and k-map / δ-presence against a population aux table for the realistic attacker who doesn't know who's in the dataset. Trade-off: every generalization destroys analytic utility; the job is to find the minimum generalization that meets your k/l target (NIST SP 800-188).

3) Custom infoTypes + Discovery to close coverage gaps (threats #1, #13). Built-ins don't know your internal employee-ID scheme, partner account numbers, or the new secret format your platform team shipped last quarter. Maintain a versioned custom infoType library (regex + dictionary + storedInfoType for big lists), each with hotword detectionRules to control likelihood, and a CI test corpus (positive + negative fixtures) so a regex change can't silently regress. Layer Discovery on top: it continuously profiles every table and predicts infoTypes even for data you never modeled, surfacing "unknown HIGH sensitivity" tables to SCC. Trade-off: aggressive custom regexes inflate false positives — the hotword/proximity rules and per-rule minLikelihood are how you keep precision acceptable; see dlp/Policy-Design on the <1% FP bar before moving from alert to block.

4) includeQuote=false + least-privilege findings storage (threat #3). The most common self-inflicted breach in DLP is the findings store itself becoming a PII repository because someone set includeQuote=true and pointed saveFindings at a broadly readable dataset. Default to includeQuote=false everywhere; only enable it in a dedicated investigation pipeline whose output BigQuery dataset has a tiny IAM allowlist, CMEK, and short retention. Route only summaries to SCC (publishSummaryToCscc), never quotes. Trade-off: without quotes, triage is harder (you get infoType + location + customer_id from identifyingFields, not the value) — acceptable, because the investigator can re-fetch the specific record under audit rather than duplicating PII by default.

5) IAM segregation + VPC Service Controls around SDP (threats #9, #10, #20). Split duties: roles/dlp.user for pipelines/SAs that only invoke inspect/deidentify; roles/dlp.admin (change-controlled, few humans) for template authoring; a separate principal set for reidentifyContent + KMS decrypt, ideally under dual-control. Put dlp.googleapis.com, bigquery.googleapis.com, and cloudkms.googleapis.com inside a VPC-SC perimeter so exfiltration of findings or tokens can't egress to another project, and turn on Cloud Audit Logs (Data Access) for DLP + KMS so every reidentifyContent and kms.decrypt is attributable and feeds UEBA. Trade-off: VPC-SC and split roles add friction (perimeter bridges, break-glass) — worth it because the alternative is one over-permissioned SA that reads every bucket and rewrites the very policy meant to protect it.

6. Detection & Response

What to log / telemetry sources

  • Cloud Audit Logs — Admin Activity & Data Access for dlp.googleapis.com: google.privacy.dlp.v2.DlpService.ReidentifyContent, .DeidentifyContent, .CreateDlpJob, .UpdateInspectTemplate, .CreateStoredInfoType. Fields: protoPayload.authenticationInfo.principalEmail, resourceName, request.
  • Cloud KMS audit logs: cloudkms.googleapis.com Decrypt on tokenization keys (spikes = mass re-identification attempts). Correlate principalEmail + resource (key name).
  • DLP findings BigQuery (saveFindings table): info_type, likelihood, location, job_name — trend infoType counts per source over time.
  • DLP job Pub/Sub notifications: job state, finding summary per scan.
  • SCC findings: Data profile (Discovery) + inspection summaries; sensitivity score deltas.
  • DlpJob state: alert on FAILED/partial — an incomplete scan is not a clean scan.

Sample detections

(a) KQL / log-query — spike in reidentifyContent or KMS decrypt on a token key (possible mass re-identification / insider abuse, maps §4 #4, #20).

// Cloud Audit Logs exported to a log analytics store
AuditLogs
| where ServiceName in ("dlp.googleapis.com","cloudkms.googleapis.com")
| where MethodName endswith "ReidentifyContent"
       or (MethodName endswith "Decrypt" and Resource contains "keyRings/dlp/cryptoKeys/tok-")
| summarize calls = count(), keys = dcount(Resource) by principal = AuthenticationInfo_PrincipalEmail, bin(TimeGenerated, 1h)
| where calls > 500          // baseline is single digits/hour for authorized reveal
| order by calls desc

(b) BigQuery SQL over the findings table — regression: PANs appearing in the logs dataset (maps §3 case 3, §4 #15).

SELECT
  job_name,
  location.container_name  AS source_table,
  COUNT(*)                 AS pan_findings,
  MAX(likelihood)          AS worst_likelihood
FROM `acme-sec.dlp_findings.logs_scan`
WHERE info_type.name = 'CREDIT_CARD_NUMBER'
  AND likelihood IN ('LIKELY','VERY_LIKELY')
  AND create_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
GROUP BY job_name, source_table
HAVING pan_findings > 0     -- any PAN in logs is a P1 PCI regression
ORDER BY pan_findings DESC;

(c) Sigma-style rule — inspect/deidentify template weakened (minLikelihood lowered or infoTypes removed; maps §4 #10).

title: SDP template minLikelihood downgraded
logsource:
  product: gcp
  service: dlp.googleapis.com
detection:
  selection:
    protoPayload.methodName: 'google.privacy.dlp.v2.DlpService.UpdateInspectTemplate'
  filter:
    protoPayload.request.inspectTemplate.inspectConfig.minLikelihood:
      - 'VERY_UNLIKELY'
      - 'UNLIKELY'
  condition: selection and filter
level: high

Response playbook (containment → eradication → recovery)

  1. Detect & scope. Alert fires (KMS decrypt spike / reidentify burst / PAN-in-logs). Pull the principalEmail, key/template names, and time window from audit logs. Determine: authorized batch job or anomalous principal?
  2. Contain — freeze the key path. If key compromise/insider suspected, remove cryptoKeyDecrypter from the suspect principal on the tokenization key (or disable the key version), cutting off both tokenize and re-identify. Revoke the SA's dlp.user/reidentifyContent bindings. VPC-SC: verify no perimeter bridge lets tokens/findings egress.
  3. Contain — stop the leak source. For PAN-in-logs: disable the offending log sink / debug logger, and insert the inline DLP scrub before re-enabling.
  4. Eradicate. Rotate the KMS tokenization key and re-tokenize affected data under the new key (retire old key version). Fix the weakened template (restore minLikelihood, restore infoTypes) and lock dlp.admin. Purge any findings rows that captured raw quotes; tighten that dataset's IAM.
  5. Assess re-identification blast radius. If the DEK is believed exposed, treat all deterministic/FPE tokens made with it as reversible by the adversary — this is a reportable event; scope which datasets used that key.
  6. Recover. Re-run full (non-sampled) inspection jobs on affected stores to confirm no residual raw PII; re-run risk analysis on any released dataset; restore normal IAM under change control.
  7. Post-incident. Add a detection for the specific gap (e.g., new Sigma rule), extend the custom infoType corpus, add the weakened-template alert to release gates, and feed the timeline to UEBA baselines (dlp/Insider-Threat).

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is an infoType and what is "likelihood"?
An infoType is a named detector for a category of sensitive data (CREDIT_CARD_NUMBER, US_SOCIAL_SECURITY_NUMBER, EMAIL_ADDRESS) — built-in (150+) or custom (regex/dictionary/stored). SDP doesn't return a boolean; each finding carries a likelihood (VERY_UNLIKELY → VERY_LIKELY) reflecting confidence from pattern match, checksum (Luhn for cards), and contextual hotwords. You set minLikelihood per config to trade precision vs recall — LIKELY/VERY_LIKELY for blocking, POSSIBLE for audit.
Signal: knows detection is probabilistic and must be tuned, not binary.

Q2. When do you use the content inspection API vs an inspection job vs Discovery?
Content API (inspectContent/deidentifyContent) is synchronous, ≤0.5 MB/item, for inline/real-time flows (scrub a log line, tokenize a request). Inspection jobs are async batch scans over BigQuery/GCS/Datastore with sampling and action fan-out. Discovery is the continuous org profiler that finds sensitive data across all tables without authoring per-job configs — the answer to "we don't even know where our PII is."
Signal: picks the right surface for scale and latency.

Q3. Name three de-identification techniques and whether each is reversible.
Masking (characterMask) → irreversible display protection. Bucketing/generalization and redaction → irreversible. Format-preserving encryption (cryptoReplaceFfxFpe) and deterministic crypto (cryptoDeterministic, AES-SIV) → reversible with the key (reidentifyContent). Date shifting → irreversible but consistent per entity via a keyed context. The key distinction interviewers want: reversible = tokenization/pseudonymization (still "personal data"); irreversible = anonymization.
Signal: distinguishes pseudonymization from anonymization.

Q4. Why is includeQuote=true dangerous?
It copies the raw matched value (the actual SSN/PAN) into your findings output — BigQuery table or logs. If that store is broadly readable, you've created a second PII repository, often less protected than the source. Default to false; only enable in an access-locked investigation pipeline with CMEK and short retention.
Signal: recognizes DLP tooling can itself become the breach.

Q5. What's the difference between a built-in and a custom infoType, and when do you need custom?
Built-ins cover universal formats (cards, SSNs, emails, cloud credentials). Custom infoTypes cover your data: internal employee IDs, partner account numbers, proprietary secret formats — defined via regex, a dictionary word list, or a storedInfoType for large lists. You need custom whenever your sensitive data doesn't match a public pattern; without it you have a coverage gap (§4 #1).
Signal: understands built-ins are necessary but never sufficient.

Tier 2 — System design / trade-offs

Q6. Design de-identification for a payments analytics warehouse that must JOIN on card and preserve card format for legacy validation.
Use two reversible transforms with KMS-wrapped keys. For the PAN, cryptoReplaceFfxFpeConfig (FF1) with commonAlphabetConfig: NUMERIC — token is 16 numeric digits, passes schema/length/Luhn-style validators, and is reversible for authorized chargeback flows. To JOIN across tables on the same card without exposing it, deterministic mode guarantees the same PAN → same token, so equality/dedup works on tokens. Store only tokens; put the KMS key behind VPC-SC with cryptoKeyDecrypter granted only to the SDP service agent and re-id restricted to a dual-control principal. Accept the trade-off that deterministic tokens leak equality (mitigate with ACLs and not exposing rare values), and that key rotation breaks token continuity — so plan a re-tokenization migration or keep the key long-lived but tightly held. Route all reidentifyContent and kms.decrypt to audit logs + UEBA.
Signal: balances utility (JOIN, format) against reversibility risk and key management.

Q7. Your BigQuery estate is 40,000 tables. How do you find and continuously monitor PII without authoring 40,000 jobs?
Deploy Discovery at the org/folder scope with a BigQuery target and daily schema/table-modified cadence. It profiles every table, predicts infoTypes, assigns a sensitivity score, and maintains TableDataProfile rows exported to BigQuery + SCC. Wire pubSubNotification with minimumSensitivityScore: HIGH so you're paged only when a new high-sensitivity table appears. Layer targeted full inspection jobs on the specific tables Discovery flags for attestation, and feed sensitivity into IAM/VPC-SC posture (tighten access to HIGH tables). This scales because you author policy once (the discovery config), not per table.
Signal: reaches for Discovery, not brute-force jobs.

Q8. Sampling vs full scan — when is each acceptable and what's the security cost?
Sampling (rowsLimitPercent, sampleMethod) bounds cost/latency on huge tables and is fine for profiling and drift detection where you want signal, not guarantees. It is not acceptable for compliance attestation, because unsampled rows can hold PII you now falsely believe absent (§4 #2). Rule: full scans (no rowsLimitPercent) for "prove this dataset is clean/tokenized" gates; sampling for continuous monitoring, and always document the sampling rate so nobody reads a partial result as complete. Also alarm on DlpJob FAILED/partial — an incomplete scan must not be treated as clean.
Signal: treats sampling as a security trade-off, not just cost.

Q9. How do you prove a dataset is safe to release externally?
Detection alone is insufficient — run risk analysis. Declare quasi-identifiers (ZIP, birth date, gender) and compute k-anonymity (smallest equivalence class ≥ target k); if k=1, generalize (bucket ZIP to 3 digits, DOB to year/decade via fixedSizeBucketing/dateShift) and re-run. Then l-diversity so each class has ≥ l distinct sensitive values (guards attribute disclosure at high k). Then k-map / δ-presence against a realistic population aux table for the attacker who doesn't know dataset membership. Iterate to the minimum generalization meeting your target (NIST SP 800-188). Only then release. This is exactly the control the Netflix Prize and Sweeney cases lacked.
Signal: knows anonymity is measured, and which metric guards which attack.

Q10. Tokenization key strategy: transient vs unwrapped vs KMS-wrapped, and rotation?
transient = a random per-job key (irreversible after the job — good for one-shot pseudonymization where you never re-identify). unwrapped = raw key on the wire (never in prod). kmsWrapped = the DEK encrypted by a Cloud KMS key — the production choice: SDP decrypts it just-in-time, the raw DEK never persists, and access is IAM/VPC-SC controlled. Rotation trade-off: rotating breaks deterministic-token continuity, so either keep the tokenization key long-lived under tight control (concentrated risk, low churn) or schedule rotation with a re-tokenization migration. Always grant cryptoKeyDecrypter on the single key, enable KMS audit logging, and separate re-identify permission from tokenize.
Signal: understands key lifecycle vs determinism tension.

Tier 3 — Expert / deep-dive

Q11. Deterministic tokenization leaks equality. When is that a real vulnerability and how do you mitigate?
AES-SIV deterministic tokens map identical plaintexts to identical ciphertexts by design — that's what enables JOINs. The leak: an attacker without the key can still do frequency analysis. If a column has low entropy or a skewed distribution (e.g., a rare diagnosis code appearing once, or a boolean-ish field), matching token frequencies to known population frequencies re-identifies values without ever decrypting. Mitigations: don't tokenize low-cardinality fields deterministically (bucket/generalize them instead); add a per-record context/salt where cross-record equality isn't needed; restrict access to the token table so frequency analysis isn't even possible; and for fields where equality-join isn't required, prefer non-deterministic encryption. It's the tokenization analogue of ECB-mode pattern leakage.
Signal: understands cryptographic subtleties of deterministic modes.

Q12. Contrast k-anonymity, l-diversity, k-map, and δ-presence — what attack does each address?
k-anonymity: every record is indistinguishable from ≥ k−1 others on quasi-identifiers — defeats unique re-identification but not attribute disclosure. l-diversity: each equivalence class holds ≥ l distinct sensitive values — defeats the homogeneity attack (all k people in a class have the same disease, so k didn't help). k-map: measures re-id risk against an external population table (how many people in the world share these quasi-IDs), which is realistic when the attacker doesn't know who's in your dataset — k-anonymity assumes the dataset is the population, k-map doesn't. δ-presence: bounds the probability an attacker can infer an individual's membership in the dataset (membership itself can be sensitive, e.g., a cancer-patient table). You typically compute all four and generalize until each meets target.
Signal: precise mapping of metric → threat model; senior-level privacy depth.

Q13. Walk through what happens end-to-end on a deidentifyContent FPE call, including the KMS interaction and where the risk concentrates.
The app POSTs the item + a DeidentifyConfig referencing a cryptoReplaceFfxFpeConfig whose key is kmsWrapped. SDP calls cloudkms.decrypt on the wrapped DEK; KMS checks the SDP service agent has cryptoKeyDecrypter, returns the raw 256-bit DEK, which SDP holds in ephemeral memory. SDP runs FF1/FFX over the field within the declared alphabet (NUMERIC here), producing a same-length token, tags it with surrogateInfoType (so re-scans recognize it), and returns tokens only. The raw DEK is discarded. Risk concentrates in three places: the KMS key (compromise = reversibility for everyone → mass re-id), the IAM binding letting who call decrypt/reidentify, and the perimeter — if VPC-SC doesn't fence KMS/DLP, a stolen SA can tokenize/re-identify from anywhere. That's why the KMS key + reidentifyContent permission are the crown jewels and why every decrypt is audited.
Signal: can trace the trust chain and name the single point of catastrophic failure.

Q14. A team says "we masked the data, so it's anonymous and GDPR doesn't apply." Where are they wrong?
Several ways. (1) If they used FPE/deterministic tokenization, it's reversible with the key → pseudonymization, not anonymization; still personal data under GDPR (Recital 26). (2) Even irreversible masking of direct identifiers leaves quasi-identifiers (ZIP+DOB+sex re-identify ~87% of Americans — Sweeney) → not anonymous without k-anonymity/generalization proven by risk analysis. (3) Deterministic tokens leak equality → frequency/linkage re-identification. (4) They didn't measure against auxiliary data (k-map/δ-presence — the Netflix failure). True anonymization requires irreversible transformation and a measured re-identification risk below threshold against a realistic attacker; masking a name column does neither.
Signal: knows the legal/technical line between pseudonymization and anonymization.

Q15. How would you architect SDP as an org-wide control plane feeding IAM, VPC-SC, and SCC, and what are the failure modes to design against?
Discovery runs at org scope, continuously classifying every BigQuery/GCS asset and emitting sensitivity-scored TableDataProfiles to SCC. High-sensitivity classifications drive posture: tighten IAM (remove broad readers), pull the resource into a VPC-SC perimeter, and require tokenization-at-rest before prod use. Inspection jobs on flagged stores attest cleanliness; inline de-identify (Dataflow) protects derivatives (logs, exports, non-prod copies). Everything routes to SCC → Pub/Sub → SIEM; reidentifyContent/KMS decrypt feed UEBA. Templates live in org, change-controlled, with a "template weakened" detection as a release gate. Failure modes to design against: incomplete infoType coverage (custom library + Discovery), the findings store becoming PII (includeQuote=false, locked dataset), KMS key over-exposure (VPC-SC + least-priv + rotation plan), sampling mistaken for full coverage (attestation = full scans), and un-inspected derivative channels (inline scrub on every pipeline that copies data). The through-line: SDP isn't a scanner, it's the classification substrate the rest of the security posture consumes.
Signal: systems thinking — integrates SDP with IAM/VPC-SC/SCC and names the org-scale failure modes.

9. Key Takeaways

  • SDP is three engines over one config vocabulary: inspection (infoTypes + likelihood), de-identification (mask/redact/bucket/date-shift/FPE/deterministic crypto), and risk analysis (k-anonymity, l-diversity, k-map, δ-presence).
  • It is the GCP-native implementation of the classification and transformation layers described in the repo's dlp/ notes — the enforcement primitive, not the whole DLP discipline (egress channels, endpoints, and policy lifecycle still live there).
  • Detection is probabilistic: tune minLikelihood and use hotword detectionRules to control false positives; built-in infoTypes are necessary but never sufficient — maintain a custom, CI-tested infoType library.
  • Reversible tokenization (FPE, AES-SIV) is only as strong as its KMS-wrapped key; that key + reidentifyContent permission are the crown jewels — least-privilege, VPC-SC, audit every decrypt, and plan for the rotation-vs-determinism trade-off.
  • Pseudonymization ≠ anonymization: reversible tokens are still personal data; true anonymization requires irreversible transforms and a measured re-identification risk (that's what k-anonymity/k-map/δ-presence are for — the lesson of Netflix Prize and Sweeney).
  • The findings store can become the breach: default includeQuote=false, lock the findings dataset, and send only summaries to SCC.
  • Deterministic tokenization leaks equality — don't deterministically tokenize low-entropy fields; bucket/generalize them and control token-table access.
  • Sampling is a security trade-off: fine for profiling, unacceptable for attestation; a partial/failed job is not a clean result.
  • Use Discovery to solve org-scale "where is our PII" without authoring thousands of jobs, and feed its sensitivity scores into IAM/VPC-SC posture and SCC.
  • Segregate duties (dlp.user vs dlp.admin vs re-identify), wrap the service in VPC-SC, and treat template weakening (minLikelihood downgrade) as an alertable, gate-worthy event.
