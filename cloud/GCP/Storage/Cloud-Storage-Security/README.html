GCP Cloud Storage (GCS) Security

Category: Cloud / GCP / Storage (Object Storage)
Defensive framing: study — how to reason about, harden, detect, and respond to object-storage exposure at scale.
Key Docs: Cloud Storage security, Uniform bucket-level access, Public access prevention, VPC Service Controls
Standards: NIST SP 800-53 (SC-8, SC-28, AC-3, AU-2), CIS GCP Foundations Benchmark v3.0 §5, MITRE ATT&CK Cloud Matrix (T1530, T1567.002, T1619, T1580)

1. Overview

Google Cloud Storage (GCS) is GCP's foundational object store: a globally-namespaced, strongly-consistent, exabyte-scale service that holds buckets containing objects addressed by key. It is the single most common source of cloud data-exposure incidents because it sits at the intersection of three failure-prone surfaces — identity (who can read/write), network reachability (can the public internet resolve storage.googleapis.com for this object), and encryption/retention (is the data protected at rest and tamper-resistant). Every large object-storage provider — S3, Azure Blob, GCS — has an identical structural weakness: the default anonymous identity (allUsers in GCP, "public" in S3/Azure) plus one permissive grant is enough to turn a private bucket into an internet-facing data feed. At scale, the security problem is not one bucket; it is the policy floor — proving that no bucket in any of ten thousand projects can ever become public, and proving it holds even when a developer with roles/storage.admin tries to override it.

GCS security in 2024+ is dominated by four pivots: Uniform Bucket-Level Access (UBLA) replacing legacy fine-grained ACLs, the Public Access Prevention (PAP) org-policy constraint that structurally forbids public grants, VPC Service Controls (VPC-SC) that draw a network-identity perimeter around the data so stolen credentials cannot exfiltrate outward, and CMEK + Bucket Lock for cryptographic control and ransomware/tamper resistance. A senior engineer must be able to reason about how these compose, where each fails, and how to detect the gap between "configured" and "enforced org-wide."

TL;DR
- Uniform bucket-level access (UBLA) disables per-object ACLs and makes IAM the single authorization model — always prefer it; fine-grained ACLs are the historical root cause of most accidental exposures because a public object ACL is invisible to bucket-level IAM review.
- Public Access Prevention is an org-policy constraint (storage.publicAccessPrevention = enforced) that structurally blocks any grant to allUsers/allAuthenticatedUsers, even by a project owner — set it at the org node, not per-bucket.
- Anonymous exposure in GCP = granting roles/storage.objectViewer (or legacy READER ACL) to the members allUsers or allAuthenticatedUsers; the latter is not "safe" — it means any Google account on earth.
- Signed URLs and signed policy documents are time-boxed, credential-derived capability tokens — a leaked signed URL is a bearer credential that bypasses IAM entirely until it expires; keep TTLs short and sign with a dedicated, low-privilege service account.
- Defense-in-depth for exfil: UBLA + PAP (identity floor) → VPC-SC perimeter (network floor) → CMEK (crypto control) → Object Versioning + Bucket Lock (recovery/immutability) → Data Access audit logs (detection).

2. How It Works

2.1 Core Mechanics

GCS has a clean split between the control plane (bucket/object metadata, IAM policy, org policy — served by the JSON/XML API and the resource manager) and the data plane (the actual bytes, served from storage.googleapis.com and Google's edge). Authorization is evaluated on every request against the union of IAM bindings and, if fine-grained mode is active, per-object ACLs. Org Policy sits above IAM: a DENY-style constraint like Public Access Prevention cannot be overridden by any IAM role, which is why it is the correct place to enforce non-negotiable invariants.

Concept Description
Bucket Globally-unique-named container; region/dual-region/multi-region located; holds objects and bucket-level config
Object Immutable blob addressed by key (name); has metadata, generation number, and optional custom metadata
IAM binding member → role grant at org/folder/project/bucket scope; the modern, preferred auth model
Fine-grained ACL Legacy per-object/per-bucket access list (OWNER/READER/WRITER); coexists with IAM unless UBLA is on
Uniform bucket-level access (UBLA) Bucket setting that disables ACLs entirely; IAM becomes the sole authorization source
allUsers Special IAM member = anyone on the internet, unauthenticated; a binding to it = public
allAuthenticatedUsers Special IAM member = anyone with any Google account; effectively public, not internal
Public Access Prevention (PAP) Bucket/org constraint that blocks binding allUsers/allAuthenticatedUsers; inherited or enforced
Org Policy constraint Guardrail evaluated above IAM (e.g. storage.publicAccessPrevention, storage.uniformBucketLevelAccess)
Signed URL Time-limited URL cryptographically signed by a key/SA; grants the encoded operation without IAM
Signed policy document Signed form-POST policy constraining browser uploads (key prefix, size, content-type, expiry)
CMEK Customer-Managed Encryption Key in Cloud KMS used to wrap the object's data-encryption key
CSEK Customer-Supplied Encryption Key — you provide raw key material per request; Google never stores it
Object Versioning Retains noncurrent generations on overwrite/delete; enables recovery from tamper/ransomware
Retention policy / Bucket Lock Minimum retention duration; Bucket Lock makes it immutable (WORM) even to project owners
VPC Service Controls Identity-aware network perimeter; blocks API access to GCS from outside the perimeter
Generation / metageneration Monotonic version identifiers used for optimistic concurrency and precondition guards

Authorization decision logic. For a data-plane request, GCS resolves: (1) is there a valid signed URL/token? If so, honor the encoded grant. (2) Otherwise authenticate the caller (OAuth token / SA). (3) Evaluate Org Policy guardrails (PAP etc.) — these can hard-deny. (4) Evaluate IAM at the bucket and, if fine-grained mode is on, ACLs at the object. (5) If VPC-SC applies, verify the request origin is inside the perimeter or covered by an ingress/egress rule. Access is granted only if authorization passes and no guardrail denies. The critical mental model: UBLA collapses steps (4)'s two sub-checks into one, removing the invisible ACL surface; PAP and VPC-SC are deny-overrides that no IAM grant can beat.

2.2 Concrete Configuration

Enforce Uniform Bucket-Level Access (disable ACLs). ACLs are the classic footgun: a single public object ACL will not show up when you review bucket IAM, so exposure hides. Turn UBLA on everywhere.

# Enable UBLA on an existing bucket (gcloud storage is the modern CLI; gsutil still works)
gcloud storage buckets update gs://acme-prod-pii \
  --uniform-bucket-level-access

# Verify it is on
gcloud storage buckets describe gs://acme-prod-pii \
  --format="value(uniform_bucket_level_access.enabled)"     # -> True

# gsutil equivalents (older muscle memory, still valid)
gsutil uniformbucketlevelaccess set on gs://acme-prod-pii
gsutil uniformbucketlevelaccess get gs://acme-prod-pii

Enforce UBLA org-wide via Org Policy so new buckets cannot be created with ACLs enabled:

# Require UBLA on every bucket in the org (boolean constraint)
cat > ubla-policy.yaml <<'EOF'
name: organizations/123456789012/policies/storage.uniformBucketLevelAccess
spec:
  rules:
  - enforce: true
EOF
gcloud org-policies set-policy ubla-policy.yaml

Apply Public Access Prevention (the single most important anti-leak control). Set the org-policy constraint at the organization node so it is inherited by every project, folder, and bucket. Even a user with roles/owner cannot then grant allUsers.

# Org-wide enforced PAP — blocks allUsers / allAuthenticatedUsers everywhere
cat > pap-policy.yaml <<'EOF'
name: organizations/123456789012/policies/storage.publicAccessPrevention
spec:
  rules:
  - enforce: true
EOF
gcloud org-policies set-policy pap-policy.yaml

# Per-bucket enforcement (belt-and-suspenders; the API field is publicAccessPrevention)
gcloud storage buckets update gs://acme-prod-pii \
  --public-access-prevention

# Confirm — should print "enforced"
gcloud storage buckets describe gs://acme-prod-pii \
  --format="value(public_access_prevention)"

With PAP enforced, this deliberate exposure attempt is rejected server-side:

# This FAILS with PAP enforced: "One or more users named in the policy do not
# belong to a permitted customer, ... public access prevention is enforced."
gcloud storage buckets add-iam-policy-binding gs://acme-prod-pii \
  --member="allUsers" \
  --role="roles/storage.objectViewer"

Least-privilege bucket IAM. Grant object-scoped roles to specific service accounts, never broad storage.admin to humans. Use conditions to constrain by object prefix where supported.

# App SA gets read-only on this bucket only
gcloud storage buckets add-iam-policy-binding gs://acme-prod-pii \
  --member="serviceAccount:ingest-svc@acme-prod.iam.gserviceaccount.com" \
  --role="roles/storage.objectViewer"

# Separate writer identity; note objectViewer != objectAdmin (no delete)
gcloud storage buckets add-iam-policy-binding gs://acme-prod-pii \
  --member="serviceAccount:writer-svc@acme-prod.iam.gserviceaccount.com" \
  --role="roles/storage.objectCreator"

CMEK — bind the bucket to a Cloud KMS key so key access (and thus data access) is independently controlled and audited, and revocation is possible.

# 1. Create a key ring + key
gcloud kms keyrings create acme-gcs --location us
gcloud kms keys create pii-cmek --location us --keyring acme-gcs --purpose encryption

# 2. Grant the GCS service agent permission to use the key
GCS_SA=$(gcloud storage service-agent --project=acme-prod)
gcloud kms keys add-iam-policy-binding pii-cmek \
  --location us --keyring acme-gcs \
  --member="serviceAccount:${GCS_SA}" \
  --role="roles/cloudkms.cryptoKeyEncrypterDecrypter"

# 3. Set the bucket default CMEK
gcloud storage buckets update gs://acme-prod-pii \
  --default-encryption-key=projects/acme-prod/locations/us/keyRings/acme-gcs/cryptoKeys/pii-cmek

Object Versioning + Retention + Bucket Lock (ransomware / tamper resistance). Versioning keeps noncurrent generations so an attacker's overwrite/delete is recoverable; a retention policy sets a minimum age before deletion; Bucket Lock makes that retention irreversible — WORM even against roles/owner.

# Keep every prior version on overwrite/delete
gcloud storage buckets update gs://acme-audit-logs --versioning

# Require objects be retained >= 2555 days (7y) before deletion is allowed
gcloud storage buckets update gs://acme-audit-logs \
  --retention-period=2555d

# LOCK it — irreversible; retention can now only be *extended*, never removed
gcloud storage buckets update gs://acme-audit-logs --lock-retention-period

Signed URL — time-boxed capability token. Sign with a dedicated low-privilege SA. Keep the TTL as short as the workflow tolerates. A leaked signed URL is a bearer token that bypasses IAM until expiry.

# 15-minute read URL, signed by a scoped SA (impersonation avoids downloading keys)
gcloud storage sign-url gs://acme-prod-pii/reports/q3.pdf \
  --duration=15m \
  --impersonate-service-account=url-signer@acme-prod.iam.gserviceaccount.com \
  --http-verb=GET

Enforce Data Access audit logging for GCS (it is NOT on by default — ADMIN_READ/DATA_READ/DATA_WRITE must be explicitly enabled):

# In the project/org IAM policy auditConfigs — logs object reads AND writes
auditConfigs:
- service: storage.googleapis.com
  auditLogConfigs:
  - logType: ADMIN_READ
  - logType: DATA_READ
  - logType: DATA_WRITE

Case study 1 — Anonymous read via legacy ACL (the classic). A team enables a static-asset bucket for a marketing microsite and, following a stale tutorial, runs gsutil acl ch -u AllUsers:R gs://acme-microsite-assets. Because the bucket used fine-grained ACLs (UBLA off), this grants anonymous read at the object level — invisible to anyone reviewing bucket IAM in the console. A scanner indexes the predictable bucket name within a day and mirrors the contents, which turn out to include an unrelated backups/ prefix a script had synced there. Control that stops it: UBLA (no object ACLs to set) plus PAP enforced at the org node (the AllUsers:R grant is rejected server-side). Detection would have come from a Data Access DATA_READ log spike from unauthenticated principals.

Case study 2 — SSRF → metadata → bucket exfil. An internet-facing GCE workload has a metadata-server SSRF. The attacker retrieves the VM's default service-account OAuth token from http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token (MITRE T1552.005), which happens to carry roles/storage.admin at the project. They enumerate buckets and begin bulk GET/storage.objects.list to an external host (T1530 → T1567.002). Controls that stop it: (a) least-privilege SA — the VM should have had objectViewer on one bucket, not project storage.admin; (b) VPC Service Controls — even with a valid token, egress of object bytes to a network outside the perimeter is denied; (c) org-policy blocking default-SA broad scopes. Detection: sudden high-volume storage.objects.get from a VM SA to novel object prefixes.

Case study 3 — Signed URL over-share. A backend mints signed download URLs with a 7-day TTL and posts them in a customer support ticketing tool. A support integration syncs ticket bodies to a third-party analytics SaaS. The URLs — bearer capabilities that bypass IAM — are now replayable by anyone who reads that analytics store, for a week. Controls that stop it: short TTL (minutes), signing with a dedicated low-privilege SA scoped to a single object, and treating signed URLs as secrets (never logged, never synced). Because signed-URL redemptions can appear as unauthenticated DATA_READ, correlation on the object key surfaces the abuse.

Top 5 in depth

1. Public Access Prevention (org-policy enforced). PAP is a boolean org-policy constraint evaluated above IAM. When set to enforced at the organization node, the GCS control plane rejects any attempt to add allUsers or allAuthenticatedUsers to a bucket or object policy — including by a project roles/owner — and removes public reachability from buckets that had it. Implementation is one policy at the org node (see §2.2) plus per-bucket --public-access-prevention for defense-in-depth. Why it is the single highest-leverage control: it converts "don't make buckets public" from a practice into an invariant that survives new projects, new engineers, copy-pasted tutorials, and Terraform drift. Trade-offs/limits: it breaks legitimately public buckets (static websites, public datasets) — those must live in a separate folder with PAP inherited/not-enforced and their own controls; PAP does not stop authenticated over-sharing (a broad grant to an internal-but-wrong group), so pair it with domain-restricted sharing and IAM review.

2. Uniform Bucket-Level Access (UBLA). UBLA disables per-object and per-bucket ACLs so IAM is the sole authorization model. Technically, once enabled the storage API refuses ACL operations and only evaluates IAM bindings. Why it matters: fine-grained ACLs are the stealthiest exposure vector — a single public object ACL never appears in a bucket-IAM review, so audits pass while data leaks. UBLA eliminates that entire class. Implementation: bucket setting plus the storage.uniformBucketLevelAccess org constraint so new buckets are born uniform. Trade-offs/limits: some legacy workflows depend on per-object ACLs (e.g. mixed-visibility objects in one bucket, or old integrations); the fix is architectural — split by sensitivity into separate buckets, each with bucket-level IAM. There is a 90-day window to revert after enabling, after which it is permanent — plan migrations.

3. VPC Service Controls perimeter. VPC-SC draws an identity-aware network perimeter around GCS (and other APIs) for a set of projects. Even with a valid stolen credential, an API call to read/list objects from outside the perimeter is denied, and egress of object data to destinations outside the perimeter is blocked unless an explicit egress rule allows it. Why it matters: it is the control that defeats the credential-theft-then-exfil chain (Capital One-style) that IAM alone cannot, because IAM says "this token may read" while VPC-SC says "but not to there / not from here." Implementation: define an access policy, a service perimeter listing projects and storage.googleapis.com, and ingress/egress rules for legitimate cross-perimeter flows. Trade-offs/limits: operationally heavy — breaks CI/CD, third-party SaaS integrations, and cross-project analytics until ingress/egress rules and access levels are modeled; start in dry-run mode and read the violation logs before enforcing.

4. CMEK + Cloud KMS. With CMEK, each object's data-encryption key is wrapped by a key you own in Cloud KMS; GCS must call KMS (as its service agent) to encrypt/decrypt. Why it matters: you gain independent authorization and audit over decryption (KMS access is separately logged and IAM-controlled), key rotation on your schedule, and crypto-shredding — disabling/destroying the key renders objects unrecoverable, a fast containment lever after compromise. Implementation: create key ring/key, grant the GCS service agent cryptoKeyEncrypterDecrypter, set the bucket default key (see §2.2). Trade-offs/limits: availability risk — if the key is disabled/destroyed or its region is unavailable, all dependent objects become unreadable (this is a self-DoS if mishandled); use rotation not deletion, keep keys in a resilient location, and separate duties so no single admin can both exfiltrate and destroy keys.

5. Object Versioning + Retention + Bucket Lock. Versioning retains a noncurrent generation on every overwrite/delete; a retention policy sets a minimum object age before deletion; Bucket Lock makes that retention irreversible (WORM), enforced even against roles/owner. Why it matters: this is the ransomware and evidence-tamper defense — an attacker who overwrites objects only creates noncurrent versions you can restore, and one who tries to delete audit logs is blocked by the locked retention. Implementation: --versioning, --retention-period, then --lock-retention-period (see §2.2). Trade-offs/limits: cost (you store every version — pair with lifecycle rules to age out old noncurrent versions), and Bucket Lock is genuinely irreversible so pick the retention duration carefully; versioning also means "deleted" secrets persist as old generations, so tighten read scope on versioned buckets.

6. Detection & Response

What to log / telemetry sources

  • Cloud Audit Logs — Admin Activity (always on): storage.setIamPermissions, storage.buckets.update, storage.buckets.create, ACL changes — catches config drift toward public.
  • Cloud Audit Logs — Data Access (must be explicitly enabled): storage.objects.get (DATA_READ), storage.objects.create/delete (DATA_WRITE), storage.objects.list — catches exfil and destruction. Key fields: protoPayload.authenticationInfo.principalEmail, protoPayload.requestMetadata.callerIp, resource.labels.bucket_name, protoPayload.resourceName, protoPayload.methodName.
  • VPC Service Controls violation logs: protoPayload.metadata.violationReason, securityPolicyInfo — perimeter denials indicate credential misuse from outside.
  • KMS logs: cloudkms.googleapis.com decrypt calls tie object access to key usage.
  • Org Policy audit: changes to storage.publicAccessPrevention / storage.uniformBucketLevelAccess.
  • Asset inventory / SCC: Security Command Center's "Public bucket" and "Bucket policy only disabled" findings.

Sample detections

SQL (BigQuery log sink) — buckets that became publicly readable:

-- Detect IAM setPolicy events granting allUsers/allAuthenticatedUsers on GCS
SELECT
  timestamp,
  protopayload_auditlog.authenticationInfo.principalEmail AS actor,
  resource.labels.bucket_name AS bucket,
  binding
FROM `acme-logs.cloudaudit_googleapis_com_activity`,
  UNNEST(protopayload_auditlog.servicedata_v1_iam.policyDelta.bindingDeltas) AS binding
WHERE protopayload_auditlog.methodName = 'storage.setIamPermissions'
  AND binding.action = 'ADD'
  AND binding.member IN ('allUsers', 'allAuthenticatedUsers')
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
ORDER BY timestamp DESC;

SQL — bulk read exfil from a single principal (possible token theft):

-- Spike of object GETs by one principal to many distinct objects in a window
SELECT
  protopayload_auditlog.authenticationInfo.principalEmail AS actor,
  protopayload_auditlog.requestMetadata.callerIp AS src_ip,
  resource.labels.bucket_name AS bucket,
  COUNT(*) AS gets,
  COUNT(DISTINCT protopayload_auditlog.resourceName) AS distinct_objects
FROM `acme-logs.cloudaudit_googleapis_com_data_access`
WHERE protopayload_auditlog.methodName = 'storage.objects.get'
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
GROUP BY actor, src_ip, bucket
HAVING distinct_objects > 1000
ORDER BY gets DESC;

for b in $(gcloud storage buckets list --format="value(name)"); do
  if gcloud storage buckets get-iam-policy gs://$b --format=json \

Response playbook (containment → eradication → recovery)

  1. Confirm exposure & scope. From the setIamPermissions log, identify the bucket, the actor who added the public binding, and the time window. Pull Data Access DATA_READ for that bucket to enumerate which objects were fetched and by which callerIp.
  2. Contain — remove public access immediately. gcloud storage buckets remove-iam-policy-binding gs://<bucket> --member=allUsers --role=roles/storage.objectViewer (and allAuthenticatedUsers), then set --public-access-prevention on the bucket. If exposure is broad, enforce PAP at the org node to stop lateral repeats.
  3. Contain credentials (if token theft suspected). Disable/rotate the implicated service account keys, revoke active OAuth tokens, and — if data was CMEK-encrypted — consider disabling the KMS key version to crypto-shred access while you investigate (weigh availability impact).
  4. Eradicate the root cause. Enable UBLA on the bucket; add the missing Org Policies (PAP, UBLA, domain-restricted sharing); tighten the offending SA to object-scoped roles; fix the SSRF/leaked-HMAC/long-TTL signed URL that enabled it.
  5. Assess data impact. Cross-reference fetched object keys against data-classification tags to determine what PII/secret content left; rotate any credentials/secrets that were in exposed objects.
  6. Recover integrity/availability. If objects were overwritten/deleted, restore from noncurrent versions (gcloud storage cp gs://bucket/obj#<generation> gs://bucket/obj) or from the locked-retention copy. Re-enable the KMS key once contained.
  7. Detect-forward & harden. Deploy the setIamPermissions and bulk-read detections above; add SCC "Public bucket" finding alerts; put PAP/UBLA into the org's IaC baseline so the fix is enforced, not manual. Run a post-incident review focused on why the guardrail was absent.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What exactly makes a GCS bucket "public," and what are the two special members involved?
A bucket is public when its IAM policy (or, in fine-grained mode, an object/bucket ACL) grants a read role to one of two special members: allUsers (anyone on the internet, unauthenticated) or allAuthenticatedUsers (anyone with any Google account). Both are effectively public — allAuthenticatedUsers is a common trap because engineers assume it means "my org's users," when it actually means any Google account worldwide. The fix is Public Access Prevention, which blocks binding either member.
Signal: knows the precise member semantics, not just "there's a public toggle."

Q2. What is Uniform Bucket-Level Access and why is it strongly preferred over fine-grained ACLs?
UBLA disables per-object/per-bucket ACLs so IAM is the sole authorization model. It is preferred because ACLs are the stealthiest exposure path: a public object ACL never shows up in a bucket-IAM review, so audits pass while data leaks. UBLA collapses authorization to one place you can reason about, review, and enforce with Org Policy. Fine-grained ACLs exist mainly for legacy interop and mixed-visibility patterns better solved by splitting buckets.
Signal: understands the invisibility-of-ACLs problem, the real reason UBLA matters.

Q3. How do signed URLs work and what is their core security risk?
A signed URL encodes an operation (e.g. GET this object), an expiry, and a cryptographic signature produced by a key/service account. GCS validates the signature and expiry and serves the request without evaluating IAM for the caller — the URL itself is the capability. The core risk is that a leaked signed URL is a bearer token: anyone who obtains it can perform the encoded action until it expires. Mitigate with short TTLs, a dedicated low-privilege signer SA, single-object scope, and never logging/sharing the URL.
Signal: grasps "bearer capability that bypasses IAM," not just "temporary link."

Q4. What does CMEK give you that Google-default encryption does not?
GCS always encrypts at rest; CMEK changes who controls the key. With a Customer-Managed Encryption Key in Cloud KMS, you get independent IAM control and audit over decryption, key rotation on your schedule, and crypto-shredding — disabling/destroying the key makes objects unrecoverable, a fast containment lever. The trade-off is availability risk: mishandling the key (destroy, revoke, region outage) makes your own data unreadable.
Signal: distinguishes "encrypted" from "you control the key," and knows the availability downside.

Q5. How would you make a bucket resistant to ransomware/tamper?
Enable Object Versioning so overwrites/deletes only create recoverable noncurrent generations; set a retention policy for a minimum object age; then apply Bucket Lock to make that retention irreversible (WORM) even against a project owner. Pair with lifecycle rules to age out old noncurrent versions for cost, and tight IAM so few identities can write. This means an attacker's destructive actions are recoverable and audit logs cannot be deleted.
Signal: knows versioning ≠ immutability; Bucket Lock is the WORM guarantee.

Tier 2 — System design / trade-offs

Q6. Design an org-wide guarantee that no bucket in 10,000 projects can ever become public. What are the pieces and the failure modes?
Set storage.publicAccessPrevention = enforced and storage.uniformBucketLevelAccess = enforce as Org Policy at the organization node so every current and future project inherits them — this makes "no public buckets" an invariant above IAM that even project owners cannot override. Layer domain-restricted sharing (iam.allowedPolicyMemberDomains) to stop authenticated over-sharing to external accounts. Back it with detection (setIamPermissions alerting, SCC public-bucket findings) as a safety net. Failure modes: legitimately-public buckets (static sites, public datasets) need a carved-out folder with PAP not enforced — a common exception path that must itself be tightly governed; org-policy exceptions/overrides at folder level can silently create gaps, so audit the policy hierarchy; and PAP does not stop over-broad internal grants, hence the domain restriction and IAM review still matter.
Signal: reaches for Org Policy as an invariant, not per-bucket toggles, and names the exception governance problem.

Q7. A workload needs public downloads for some assets and strictly-private storage for user PII in the same platform. How do you architect it?
Split by sensitivity into separate buckets in separate folders with different policy postures: a public-assets bucket in a folder where PAP is not enforced but with tight write IAM, CDN in front, and no PII ever written there; and a user-pii bucket in a folder with PAP enforced, UBLA on, CMEK, VPC-SC perimeter, versioning + Bucket Lock, and Data Access logging. For time-limited user access to private objects, mint short-TTL signed URLs from a dedicated signer SA rather than making anything public. Enforce the separation with Org Policy at the folder level and CI checks so no pipeline can write PII into the public bucket, and use data-classification tags to detect misroutes. The key trade-off is operational: two postures cost more to run but eliminate the "one bucket, mixed visibility" footgun that ACLs invite.
Signal: separates by bucket/folder posture instead of trying to mix visibility in one bucket.

Q8. When would you deploy VPC Service Controls around GCS, and what breaks?
Deploy VPC-SC when the threat you care about is credential theft leading to exfiltration — the case IAM alone cannot stop, because IAM authorizes the token while VPC-SC constrains where the API call may originate and where data may egress. It is essential for regulated/PII data at scale. What breaks: CI/CD runners, third-party SaaS integrations, cross-project analytics, and developer laptops all sit outside the perimeter and get denied until you model access levels and ingress/egress rules for each legitimate flow. The right rollout is dry-run mode first, read the violation logs to enumerate real dependencies, add narrowly-scoped rules, then enforce — a big-bang enforce will cause an outage.
Signal: knows VPC-SC defeats the exfil chain IAM cannot, and respects the operational cost/dry-run discipline.

Q9. How do you decide between short-lived signed URLs, IAM-based access, and IAP for serving objects to users?
Use IAM-based access when the consumer is a service/SA you control — it is auditable, revocable, and needs no bearer tokens. Use signed URLs for unauthenticated or third-party end users who need time-boxed access to specific objects (e.g. a customer downloading their export) — accept that it is a bearer capability and keep TTLs minimal. Use Identity-Aware Proxy in front of a backend bucket when you want per-request identity, group-based authorization, and context-aware access (device, location) for a web-served asset without minting URLs. The trade-off axis is: IAM = strongest audit but requires the consumer to have an identity; signed URLs = frictionless but bearer-token risk; IAP = identity + context but adds a proxy and only fits HTTP-served flows.
Signal: matches mechanism to consumer identity model and articulates the audit-vs-friction trade-off.

Q10. How would you roll out CMEK to an existing fleet of buckets without an availability incident?
CMEK only applies to newly-written objects, so set the bucket default key and then rewrite existing objects (copy in place) to bring them under CMEK, staged per bucket. Before flipping anything, ensure the GCS service agent has cryptoKeyEncrypterDecrypter on the key, put keys in a resilient location matching the bucket's, and set a rotation schedule (rotate, never destroy, live keys). Separate duties so no single identity can both read objects and destroy the key, and add KMS-decrypt monitoring. The dominant risk is availability: a disabled/destroyed key or a region mismatch renders objects unreadable, so treat key lifecycle as a change-controlled, staged operation with rollback (re-enable key) rather than a bulk flip.
Signal: knows CMEK is write-forward, and centers key-availability/duty-separation risk.

Tier 3 — Expert / deep-dive

Q11. Walk through the full SSRF-to-exfil chain against GCS and every control that breaks a link.
An internet-facing workload has an SSRF; the attacker hits the metadata server (metadata.google.internal) to steal the VM's default-SA OAuth token (T1552.005), then uses it to list and bulk-GET objects (T1530) and stream them to an external host (T1567.002). Break the chain at each link: fix the SSRF and require the Metadata-Flavor header / disable legacy metadata endpoints to make token theft harder; give the VM a minimal SA with object-scoped read on one bucket instead of project storage.admin so a stolen token is low-value; enforce VPC Service Controls so even a valid token cannot read from outside the perimeter or egress data outward; enable Data Access logs to detect the bulk-read spike from a novel IP; and use CMEK so you can crypto-shred if needed. The most robust single control is VPC-SC because it neutralizes the token's value regardless of how it was stolen.
Signal: end-to-end kill-chain thinking with a named "strongest link-breaker."

Q12. Compare UBLA-migration risk and the irreversibility windows for UBLA and Bucket Lock. How do you sequence them safely?
Enabling UBLA is reversible for 90 days, after which it becomes permanent; enabling it can break workflows that rely on per-object ACLs, so you inventory ACL usage (audit logs / storage.objects ACL reads) and migrate mixed-visibility buckets by splitting before flipping, validating within the 90-day window. Bucket Lock is immediately and permanently irreversible — once locked, retention can only be extended, never removed or shortened — so it must come after you are certain of the retention duration and have modeled cost of storing all versions. Sequence: first UBLA (reversible safety net), validate access still works, then set retention and only lock it once the compliance requirement and duration are signed off. Never lock a bucket you are still experimenting with, and keep a separate un-locked staging bucket for validation.
Signal: precise on the two different irreversibility semantics and safe ordering.

Q13. How do signed policy documents differ from signed URLs, and what's the abuse if you get them wrong?
A signed URL authorizes a single, fully-specified operation (usually a GET/PUT to one object); a signed policy document authorizes a browser form-POST upload and constrains it with conditions — allowed key prefix, content-length range, content-type, and an expiry. The abuse when done wrong: if the policy omits a key-prefix constraint, an attacker can upload to arbitrary object names (overwrite other users' objects or plant malicious content in a served path); without a content-length cap they can exhaust storage or upload huge payloads; without content-type limits they can smuggle executable/HTML content into a bucket fronted by a website. The correct posture is tight conditions (exact prefix per user, size ceiling, content-type allowlist, minutes-long expiry) and a low-privilege signer, treating the signed policy as a scoped capability just like a signed URL.
Signal: knows the upload-side capability and its specific misconfig abuses.

Q14. Design detection for "insider slowly exfiltrating via signed URLs" — which is deliberately stealthy against normal alerts.
Signed-URL redemptions can appear as unauthenticated Data Access DATA_READ, so per-principal read-volume alerts miss them. Detect on the generation side and the redemption side together: alert when a single service account (the signer) mints an anomalous number of signed URLs, or mints them for objects/prefixes outside its historical pattern (Admin Activity / app logs); and correlate redemptions by object key and callerIp diversity — many distinct external IPs fetching many distinct objects signed by one SA in a short window is the signal. Baseline per-SA signing rates and per-bucket external-IP fan-out, and flag deviations. Also constrain the blast radius so detection has teeth: dedicated signer SA per use-case, short TTLs (so a stolen batch expires), and object-scoped signing so mass-minting is visible as scope-widening. The hard part is that a patient insider staying under thresholds requires behavioral baselining, not static rules.
Signal: recognizes signed URLs evade principal-based read alerts and pivots to signing-side + fan-out behavioral detection.

Q15. GCS, S3, and Azure Blob all suffer public-bucket leaks. Argue which provider's default controls best prevent it in 2024+ and what a provider-agnostic policy floor looks like.
All three converged on similar controls: GCS has Public Access Prevention + UBLA as Org Policy; AWS has S3 Block Public Access (now on by default for new buckets since 2023) + object-ownership "bucket owner enforced" that disables ACLs; Azure has "allow blob anonymous access = disabled" at the storage-account level + no-ACL model by default. AWS arguably has the strongest default today because Block Public Access is enabled by default account-wide and BPA overrides any policy/ACL, whereas GCS PAP and Azure's account setting are strong but must be deliberately enforced org-wide (GCS ships PAP as inherited/unset by default). The provider-agnostic floor is identical in shape: (1) disable ACLs so IAM/RBAC is the single model; (2) a top-of-hierarchy guardrail that structurally forbids anonymous grants and cannot be overridden by resource owners; (3) an identity-aware egress perimeter (VPC-SC / S3 access points + VPC endpoints + SCPs / Azure private endpoints + firewall) so stolen credentials cannot exfiltrate; (4) customer-managed keys for crypto control; (5) versioning/object-lock for tamper resistance; and (6) mandatory data-access logging. The lesson from a decade of leaks across all three is that the control that works is a non-overridable invariant at the org root, not per-bucket hygiene.
Signal: cross-provider fluency plus the meta-point that org-root invariants beat per-resource practice.

9. Key Takeaways

  • Public Access Prevention enforced at the org node is the single highest-leverage control — it turns "no public buckets" into an invariant above IAM that survives new projects and human error.
  • Always enable Uniform Bucket-Level Access. Fine-grained ACLs hide object-level public grants from bucket-IAM review; UBLA removes that entire invisible exposure class.
  • allAuthenticatedUsers is not "internal" — it means any Google account on earth; treat it as public and block it with PAP.
  • Signed URLs and signed policy documents are bearer capabilities that bypass IAM. Short TTLs, a dedicated low-privilege signer SA, tight object/prefix scope, and never logging them.
  • VPC Service Controls is the control that defeats the credential-theft-then-exfil chain IAM alone cannot; roll it out in dry-run first because it breaks CI/CD and SaaS integrations.
  • CMEK gives you key control, audit, rotation, and crypto-shredding — but mishandling the key is a self-inflicted availability outage; rotate, never prematurely destroy.
  • Versioning is recovery; Bucket Lock is immutability. Together they give ransomware resistance and tamper-proof audit logs — and Bucket Lock is irreversible, so choose retention carefully.
  • Data Access audit logs are OFF by default — without ADMIN_READ/DATA_READ/DATA_WRITE you have no exfil forensics; enable them org-wide.
  • Least-privilege service accounts (object-scoped roles, not project storage.admin) shrink the blast radius of any stolen token.
  • The cross-provider lesson from a decade of leaks: a non-overridable guardrail at the org root beats per-bucket hygiene every time; the same shape (disable ACLs → block-public invariant → egress perimeter → CMK → WORM → logging) applies to S3 and Azure Blob.
