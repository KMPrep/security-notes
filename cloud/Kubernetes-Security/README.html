Kubernetes Security — Cluster Hardening Deep-Dive

Domain: Cloud / Container Orchestration Security
Frameworks: CIS Kubernetes Benchmark, NSA/CISA Kubernetes Hardening Guide, Pod Security Standards, MITRE ATT&CK for Containers
Related notes: infra/Virtualization/Escape-Techniques, cloud/AWS/Compute/EKS

1. Overview

Kubernetes is a distributed control system that schedules containers across a fleet of nodes, reconciling declared state stored in etcd against observed state through a set of controllers. Its power is also its attack surface: a single misconfigured RBAC binding, an over-privileged pod, or an exposed API endpoint can convert a workload compromise into full cluster and cloud-account takeover. Kubernetes ships "open by default" in many respects — no network segmentation between pods, permissive default service accounts, and no mandatory workload sandboxing — so security is something you must deliberately layer on rather than something you inherit.

Hardening a cluster means defending every layer of the 4C's model — Cloud, Cluster, Container, Code — because each outer layer is the trust foundation of the one inside it. A perfectly written application (Code) running in a locked-down container is still exposed if the Cluster grants it cluster-admin, and a hardened cluster is worthless if the Cloud account it runs in lets any node assume an administrative IAM role. This note consolidates the control-plane, authorization, admission, network, secrets, isolation, supply-chain, and runtime controls that a Senior Security Engineer is expected to reason about end to end, and maps the common attack paths — container escape, over-permissive RBAC, exposed dashboard/API, and SSRF to the cloud metadata endpoint — to the specific Kubernetes mechanisms that stop them.

2. How It Works

2.1 Architecture and Security Control Points

A Kubernetes cluster is split into a control plane and a set of worker nodes. The control plane is the security-critical core:

Component Role Primary Security Concern
kube-apiserver Single front door; all reads/writes go through it AuthN, AuthZ (RBAC), admission control, TLS, anonymous-auth
etcd Key-value store holding all cluster state and Secrets Encryption at rest, mutual TLS, network isolation
kube-controller-manager Runs reconciliation loops Holds powerful credentials; token/cert issuance
kube-scheduler Assigns pods to nodes Can be abused to place pods on target nodes
cloud-controller-manager Bridges to cloud APIs Node IAM role scope; metadata access
kubelet Node agent that runs pods Exposed kubelet API, anonymous auth, node authorization
kube-proxy / CNI Pod networking and service routing Network policy enforcement, default-allow traffic
container runtime containerd/CRI-O runs containers Container escape, runtime isolation

Every request to change or read cluster state passes through a fixed pipeline in the API server, and each stage is a security control point:

  1. Transport — TLS termination; reject anonymous/unauthenticated traffic.
  2. Authentication — establish who (X.509 client certs, OIDC tokens, service-account JWTs, or cloud IAM webhooks such as EKS access entries / IRSA).
  3. Authorization — decide whether the identity may perform the verb on the resource. RBAC is the dominant mode; Node and Webhook authorizers also exist.
  4. Admission control — mutating then validating admission webhooks that can rewrite or reject the object (Pod Security, OPA Gatekeeper, Kyverno, image-signature verification).
  5. Persistence — validated object written to etcd (ideally encrypted at rest).

This layered pipeline is why defense-in-depth works: authorization stops identities that shouldn't act, and admission stops even authorized identities from creating dangerous objects (e.g., a privileged pod).

Deeper Context — Top 3 Controls

1. RBAC Least-Privilege

RBAC is the authorization backbone, and it is where most real-world Kubernetes compromises escalate. The golden rule is that identities — especially pod service accounts — should hold the minimum verbs on the minimum resources in the minimum scope. Prefer namespaced Role/RoleBinding over cluster-wide ClusterRole/ClusterRoleBinding; never bind cluster-admin to a workload; and never use wildcards (verbs: ["*"], resources: ["*"]). Several rights are effectively "escalate-to-admin" and must be tightly held: create pods (mount any secret, run privileged), create pods/exec, create secrets, bind/escalate on roles, impersonate, create tokenrequests, and access to nodes/proxy. A subtle trap is the default service account: every pod without an explicit SA gets the namespace default, and if that SA (or the token) is broad, every pod inherits it — so set automountServiceAccountToken: false by default and grant tokens deliberately. The trade-off is operational friction: least-privilege requires knowing what each workload actually needs, which is why teams pair it with kubectl auth can-i --list, audit-log mining, and tools like rbac-lookup/rakkess to detect drift.

2. Pod Security and Admission Control

Authorization decides who can act; admission control decides what objects are allowed to exist — and it is the layer that stops even an authorized user from creating a dangerous pod. Pod Security Standards define three profiles: privileged (no restrictions), baseline (blocks the most obvious escalation vectors — hostNetwork, hostPID, privileged, most hostPath), and restricted (adds must-run-as-non-root, drop ALL capabilities, seccomp RuntimeDefault, no privilege escalation, read-only where possible). The built-in Pod Security Admission controller enforces a chosen level per namespace via labels (pod-security.kubernetes.io/enforce: restricted) with enforce/audit/warn modes for staged rollout. PSA is intentionally simple and namespace-scoped; when you need cross-cutting or field-level rules — image registries, required labels, mandatory resource limits, blocking :latest tags — you reach for a general policy engine. OPA Gatekeeper uses Rego and ConstraintTemplates and is highly expressive; Kyverno uses native YAML policies, is Kubernetes-idiomatic, and also does mutation, generation, and image-signature verification. Both are validating admission webhooks (and Kyverno additionally a mutating one). The key architectural point: mutating webhooks run first and can inject defaults (e.g., add a seccomp profile), then validating webhooks reject violations — set failurePolicy: Fail for security-critical policies so a webhook outage fails closed, but scope carefully to avoid locking yourself out of the cluster.

3. Network Policy Default-Deny

By default, Kubernetes networking is completely flat: every pod can reach every other pod and any external endpoint, so a single compromised workload can scan and pivot across the whole cluster and out to the cloud metadata service. NetworkPolicy objects restore segmentation, but they are additive allow-lists that only take effect once a policy selects a pod — and they only work if the CNI enforces them (Calico, Cilium, Weave; the AWS VPC CNI needs the network-policy add-on). The correct pattern is default-deny then explicit-allow: apply a policy that selects all pods with empty ingress and egress rules to deny everything, then add narrow policies allowing only the flows each workload needs. Egress default-deny is especially valuable for security because it blocks two of the worst attack paths at once — data exfiltration and SSRF/pivot to 169.254.169.254 — while ingress default-deny stops east-west lateral movement. The trade-off is that policies are label-driven and can silently break connectivity, so teams roll them out per namespace with observability (Cilium Hubble, flow logs) and treat DNS egress (to kube-dns) as an explicit allow that is easy to forget. For richer needs — L7 rules, identity-aware policy, cluster-wide policies — Cilium's CiliumNetworkPolicy extends the native primitive.

5. Practice Questions

Tier 1 — Fundamentals

  1. What are the 4C's of cloud-native security? Cloud, Cluster, Container, Code — nested trust layers where each outer layer secures the inner one, so a weakness in an outer layer (e.g., an over-scoped node IAM role) undermines everything inside it. Asked to confirm the candidate thinks in defense-in-depth rather than a single control.

  2. Walk through what happens when a request hits the API server. TLS → authentication (who) → authorization/RBAC (allowed?) → mutating then validating admission (should this object exist / rewrite it) → persist to etcd. Asked because every control point maps to a class of mitigation, and this pipeline is the mental model for the whole topic.

  3. How are Kubernetes Secrets protected by default, and why is that insufficient? They are only base64-encoded in etcd, readable by anyone with get secrets RBAC or etcd access. You must enable encryption-at-rest with a KMS provider and ideally externalize to Vault or a cloud secrets manager. Asked to catch the common myth that Secrets are encrypted.

  4. What is RBAC and how do Role and ClusterRole differ? RBAC binds subjects to sets of verbs on resources; Role/RoleBinding are namespace-scoped while ClusterRole/ClusterRoleBinding are cluster-wide. Prefer namespaced roles for least privilege. Asked to verify baseline authorization literacy.

  5. What does a default-deny NetworkPolicy do and why start there? It selects all pods and denies all ingress/egress, forcing you to explicitly allow only required flows — closing lateral movement and metadata-SSRF paths. Asked because "flat by default" is the single biggest network mistake in Kubernetes.

Tier 2 — Design and Trade-offs

  1. Design admission control for a multi-tenant cluster that must block privileged pods, enforce signed images, and require resource limits. Layer Pod Security Admission (restricted) for the pod-level baseline, add Kyverno or Gatekeeper for image-registry allow-lists, signature verification, and required limits; run mutating policies to inject safe defaults, validating policies with failurePolicy: Fail for security rules, staged via audit/warn first. Asked to test whether the candidate can compose built-in and policy-engine controls rather than reaching for one tool.

  2. A pod is compromised via RCE. Trace how far an attacker gets and where you'd place controls to stop them. Network policy (contain lateral movement), disabled token automount and least-privilege RBAC (limit API abuse), Pod Security restricted (block privileged escalation), metadata egress block + IRSA (stop cloud pivot), Falco (detect). Asked to see holistic attack-path reasoning — this is Diagram B in words.

  3. How do you stop the SSRF-to-metadata credential theft path in Kubernetes? NetworkPolicy egress deny to 169.254.169.254, IMDSv2 with hop-limit 1 so pods can't reach it, and per-pod cloud identity (IRSA/Pod Identity/Workload Identity) so the node role is minimal and not what pods use. Asked because this is the classic cloud-account takeover chain and connects Kubernetes to cloud IAM.

  4. Compare OPA Gatekeeper and Kyverno. Gatekeeper uses Rego and is very expressive but has a learning curve; Kyverno uses native YAML, is Kubernetes-idiomatic, and additionally supports mutation, generation, and image-signature verification. Both are validating webhooks. Asked to test practical tooling judgment and awareness of mutation vs validation.

  5. How would you give a pod access to a cloud API without long-lived credentials? Use workload identity federation — IRSA or EKS Pod Identity on AWS, Workload Identity on GKE — which maps a Kubernetes service account to a scoped cloud IAM role via OIDC, issuing short-lived tokens; never bake static keys into images or the node role. Asked to connect cluster identity to cloud identity, a Senior-level integration point (see cloud/AWS/Compute/EKS).

Tier 3 — Expert and Deep-Dive

  1. Which RBAC permissions are effectively cluster-admin even though they look narrow? create pods (mount any secret, run privileged, escape), pods/exec, create secrets/get secrets, bind and escalate on roles, impersonate, nodes/proxy, and create tokenrequests. Asked to test deep RBAC threat modeling beyond "don't grant ".*

  2. Explain container escape vectors and which Kubernetes controls address each. Privileged containers and CAP_SYS_ADMIN, hostPath mounts of the docker socket or /, hostPID/hostNetwork, and kernel CVEs. Controls: Pod Security restricted (drop capabilities, no privileged, no hostPath), seccomp RuntimeDefault, AppArmor/SELinux, and stronger isolation via RuntimeClass (gVisor, Kata). Asked to bridge to infra/Virtualization/Escape-Techniques and test kernel-level awareness.

  3. When would you use a sandboxed runtime (gVisor/Kata) via RuntimeClass, and what's the cost? For untrusted or multi-tenant workloads where a shared-kernel escape is unacceptable; gVisor intercepts syscalls in userspace and Kata runs micro-VMs, both adding a strong boundary at the cost of syscall/IO performance and some compatibility gaps. Asked to test understanding that namespaces/cgroups share a kernel and when that's not enough.

  4. How do you make admission webhooks fail safe without locking yourself out of the cluster? Set failurePolicy: Fail for security policies but scope them tightly with namespaceSelector/objectSelector, exclude kube-system and the webhook's own namespace, run the webhook HA, and keep a break-glass path; a fail-closed webhook that also gates its own control plane can brick the cluster. Asked because this is a subtle production failure mode Seniors are expected to have hit.

  5. Design end-to-end supply-chain integrity for images running in the cluster. Sign images with Cosign/Sigstore in CI, store provenance/SBOM, scan for CVEs pre-deploy, restrict to trusted registries, and enforce at admission with Kyverno verifyImages or Connaisseur so only signed, policy-compliant images run; pair with runtime detection (Falco) for post-admission drift. Asked to test whether the candidate closes the loop from build to admission to runtime, not just "scan images."

6. Key Takeaways

  • Kubernetes is open by default — flat networking, permissive service accounts, no mandatory sandboxing — so security is layered on deliberately across the 4C's (Cloud, Cluster, Container, Code).
  • The API-server pipeline (authn → RBAC → admission → etcd) is the mental model: authorization controls who acts, admission controls what objects exist, and both are needed.
  • The three highest-leverage controls are RBAC least-privilege (namespaced, no wildcards, no default-SA automount), Pod Security/admission enforcement (restricted + Gatekeeper/Kyverno), and default-deny NetworkPolicy (especially egress).
  • The signature cloud-takeover chain is pod RCE → over-permissive RBAC or privileged pod → container escape → SSRF to metadata → node IAM credentials; break it with per-pod cloud identity (IRSA/Pod Identity) and a metadata egress block.
  • Secrets are only base64 by default — enable etcd encryption-at-rest and externalize to Vault or a cloud secrets manager, and mount as files, not env vars.
  • Verify the supply chain at admission (signed images via Cosign + Kyverno) and detect at runtime (Falco/eBPF + audit logging) so build-time and run-time integrity both hold.
  • Fail security admission webhooks closed but scope them carefully — a poorly bounded failurePolicy: Fail can lock the whole cluster out.
