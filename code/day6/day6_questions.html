Day 6: Timed Practice Simulation

Google Security Engineer practice questions

Focus: Shift from learning to fluency. Simulate real time pressure.
Complexity: Mixed — 1 easy, 2 medium, 2 hard. Timed strictly.
Format: Structure every problem the same way before writing a single line of code.

Security framing: Google security engineering discussions test both algorithmic fluency and
your ability to think about adversarial inputs, edge cases in security data, and operational
realism. You will be expected to explain threat models, not just return correct values.

PRACTICE STRUCTURE (enforce for every problem)

Before coding, spend 3–5 minutes on:

  1. Restate: Rephrase the problem in your own words.
  2. Assumptions: List what you assume about inputs (sorted? null? ASCII only?).
  3. Pseudocode: Write steps in comments before any real code.
  4. Code: Implement cleanly from pseudocode.
  5. Test: Run 2–3 hand-made examples including an edge case.
  6. Complexity: State T(n) and S(n) explicitly.

EASY — 20 minutes

Q1. First Repeated Privileged Command
Given a list of shell commands run by a user, return the first command that appears more
than once. Return None if all commands are unique.

  • Input: ["ls","sudo su","whoami","sudo su","id"] → "sudo su"
  • Input: ["ls","cat","pwd"] → None
  • Constraints: Case-sensitive. Order of first recurrence matters.
  • Follow-up: If the list has 100M entries, how does your approach change?
  • Expected complexity: O(n) time, O(n) space.

MEDIUM — 35–40 minutes

Q2. Sliding Window: Rate Limit Enforcement
You are building a rate limiter for an API endpoint. Given a list of request timestamps
(in seconds, sorted ascending) for a single user and a window size W (seconds), return
the total number of requests that would be rate-limited (i.e., more than K requests within
any W-second window).

A request is rate-limited if, at its arrival time, there are already K or more requests
within the preceding W seconds (inclusive of current).

  • Input: timestamps=[1,1,2,3,4,5,6,6,6,7], W=3, K=3
  • Output: count of rate-limited requests
  • Write out which specific timestamps get rate-limited.
  • Complexity: O(n) sliding window.
  • Follow-up: How would your design change if multiple users share a global rate limit?

Q3. ACL Rule Conflict Detector
You have a list of firewall ACL rules. Each rule is:
{"priority": int, "src": str, "dst": str, "port": int, "action": "ALLOW"|"DENY"}

Rules are evaluated in priority order (lowest number = highest priority). A conflict exists
when two rules have:

  • Same src, dst, and port
  • Different action

Find all conflicting pairs. For each pair, report which rule wins (lower priority number).

  • Input:
    python rules = [ {"priority": 10, "src": "10.0.0.0/8", "dst": "any", "port": 443, "action": "ALLOW"}, {"priority": 20, "src": "10.0.0.0/8", "dst": "any", "port": 443, "action": "DENY"}, {"priority": 5, "src": "192.168.1.0/24", "dst": "any", "port": 80, "action": "DENY"}, {"priority": 30, "src": "192.168.1.0/24", "dst": "any", "port": 80, "action": "ALLOW"}, ]
  • Output: List of conflict pairs with winning rule.
  • Complexity: O(n²) acceptable for small rulesets. Discuss how to improve to O(n).

HARD — 45–50 minutes

Q4. Threat Event Correlation Engine (Hard)
You have two sorted lists of security events from different sensors:

# (timestamp, event_type, source_ip)
sensor_a = [(100, "port_scan", "1.1.1.1"), (200, "login_fail", "2.2.2.2"), ...]
sensor_b = [(110, "dns_query", "1.1.1.1"), (210, "exec_shell", "2.2.2.2"), ...]

A correlated threat occurs when the same source_ip appears in both sensors within a
configurable time window T (in seconds). Return all correlated pairs along with the
time delta.

Requirements:

  • Handle each IP's events independently.
  • If a sensor_a event correlates with multiple sensor_b events, report all pairs.
  • Sort output by sensor_a timestamp ascending.
  • Complexity: Aim for O(n log n) or better. Explain your approach.
  • Follow-up: How would your solution scale to 1 billion events per day?

Q5. Minimal Covering Policy
You have a set of N users and M permissions. You also have a set of roles, where each
role grants a subset of permissions. Find the minimum number of roles that, combined,
cover ALL required permissions for a given user's access request.

This is a variant of the Set Cover problem.

  • Input:
    python required = {"read", "write", "exec", "admin"} roles = { "viewer": {"read"}, "editor": {"read", "write"}, "developer": {"read", "write", "exec"}, "admin": {"read", "write", "exec", "admin"}, }
  • Output: ["admin"] (one role covers all)
  • Harder input: required = {"read","write","exec","audit","deploy"} with no single covering role.
  • Greedy approach: at each step, pick the role that covers the most uncovered permissions.
  • Explain: Is the greedy solution always optimal? (No — Set Cover is NP-hard. Greedy gives
    approximately optimal. Note this explicitly.)
  • Complexity: O(M _ N) per greedy step, O(M _ N²) total. Acceptable for small M, N.

SELF-REVIEW CHECKLIST (complete after every timed set)

Answer honestly:

Syntax

  • [ ] Did you look up any syntax mid-problem? (list comprehension, dict methods, regex flags)
  • [ ] Did you forget .items(), .values(), or .keys() at any point?
  • [ ] Did you accidentally use 0-indexing incorrectly?

Logic

  • [ ] Did your initial pseudocode lead to correct code, or did you have to restart?
  • [ ] Were there any off-by-one errors in loops or binary search?
  • [ ] Did you handle the empty input / single element case?

Edge Cases

  • [ ] Did you test with: empty input, single element, all same elements, maximum input?
  • [ ] For security problems: did you consider adversarial input (malformed log, null values)?

Complexity

  • [ ] Did you state T(n) and S(n) before being asked?
  • [ ] Could you explain why, not just what the complexity is?

Problem Selection

  • [ ] Did you use the right data structure instinctively, or did you have to think it through?
  • [ ] Which problems caused the most hesitation? (target these in Day 7)

Study Notes for Day 6

What Google interviewers watch for in Security Engineering roles:

  • You reason about adversarial inputs (what if someone sends malformed data?)
  • You mention scale explicitly (100M entries vs 100 entries)
  • You offer tradeoffs (heap vs sort, greedy vs exact)
  • You stay calm when stuck — verbalize your thinking out loud
  • You name complexity correctly and don't claim O(n) for O(n²) code

Tactics if you get stuck:

  1. Try a brute force approach first and state its complexity.
  2. Ask: "can I trade memory for speed here?"
  3. Ask: "is the input sorted, bounded, or constrained in any way I haven't used?"
  4. Return to the pseudocode — don't write code when confused.
