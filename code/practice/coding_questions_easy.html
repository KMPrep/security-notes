Google Security Engineer Coding Questions — Easy Set

Overview

This file contains 70 easy Google Security Engineer coding practice questions.

The focus is not only algorithm correctness. You should also practise explaining:

  • what security problem the code helps solve
  • what logs or data are required
  • what edge cases matter
  • what false positives or false negatives may exist
  • how the solution changes at production scale

Clarifying Questions to Ask

Ask these before coding. Google interviewers usually care about whether you clarify assumptions.

Input / Output

  1. What is the exact input format?
  2. Is the input already parsed, or should I parse raw log lines?
  3. Should the function return a list, dictionary, count, boolean, or structured objects?
  4. Should output ordering be deterministic?
  5. How should ties be handled?
  6. Should timestamps be strings or datetime objects?
  7. Are timestamps guaranteed to be in UTC?

Data Quality

  1. Can log lines be malformed?
  2. Can fields be missing?
  3. Can there be duplicate event IDs?
  4. Should we ignore invalid records or raise errors?
  5. Are usernames, domains, and IP addresses case-sensitive?
  6. Can input contain empty strings or null values?

Scale

  1. How large can the input be?
  2. Can the full dataset fit in memory?
  3. Is this batch processing or streaming?
  4. Do we need exact results or approximate results are acceptable?
  5. Do we need to support distributed processing?

Security Context

  1. What threat are we trying to detect?
  2. What is the expected false-positive tolerance?
  3. Are we building detection, prevention, enrichment, or reporting logic?
  4. Should the output include evidence explaining why something was flagged?
  5. Should we include severity or confidence score?
  6. What logs would prove the detection works?

Edge Cases

  1. What should happen for empty input?
  2. What should happen for malformed timestamps?
  3. What if an IP appears in IPv6 format?
  4. What if a field contains encoded data?
  5. What if logs arrive out of order?
  6. What if the same user appears with multiple identifiers?

Expected Easy-Level Approach

For easy questions, interviewers expect:

  • clear variable names
  • correct use of dictionaries, sets, and lists
  • simple parsing
  • O(n) thinking where possible
  • basic edge case handling
  • readable code over clever code

Easy-Level Edge Cases

  • empty list
  • single item
  • duplicate items
  • mixed uppercase/lowercase strings
  • malformed log line
  • missing field
  • unexpected whitespace
  • invalid IP-like strings
  • case-sensitive versus case-insensitive comparison

Easy-Level Practice Target

Try to solve each question in 10–15 minutes.

For each solution, explain:

  1. what data structure you use
  2. why it is correct
  3. time complexity
  4. space complexity
  5. one security relevance point

Clarifying questions to ask:

  • What is the exact input format?

  • What should the output format be?
  • Are results expected to be sorted?

  • Choose a simple data structure first.

Problem:
Detect users with more than N failed logins.

Clarifying questions:
- Are timestamps sorted?
- Is the threshold per user or per IP?
- Should failed attempts be counted over all time or a rolling window?
- Should malformed records be skipped?

Approach:
Use a dictionary keyed by username. Increment count for LOGIN_FAILED records.
Return users whose count exceeds N.

Complexity:
O(n) time, O(u) space where u is the number of unique users.

Security meaning:
This detects brute force or credential stuffing patterns but may need tuning to reduce false positives.
