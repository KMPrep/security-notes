Asymmetric Encryption

Category: Cryptography — Public Key Cryptography

Overview

Asymmetric encryption uses a key pair: a public key (freely shareable) and a private key (kept secret). Data encrypted with the public key can only be decrypted by the corresponding private key, and vice versa. This solves the key distribution problem that symmetric encryption cannot address.

Asymmetric encryption is slower than symmetric encryption by 100–1000x, which is why it is used to establish a shared secret or exchange a symmetric key, not to encrypt bulk data.

RSA (Rivest–Shamir–Adleman)

RSA is the most widely deployed asymmetric encryption algorithm. It is based on the mathematical difficulty of factoring the product of two large prime numbers.

Key Generation

  1. Choose two large primes p and q
  2. Compute n = p × q (the modulus)
  3. Compute Euler's totient φ(n) = (p-1)(q-1)
  4. Choose public exponent e (commonly 65537) where gcd(e, φ(n)) = 1
  5. Compute private exponent d where (d × e) mod φ(n) = 1
  6. Public key: (n, e) — Private key: (n, d)

Encryption and Decryption

$$\text{Encrypt: } C = M^e \mod n$$
$$\text{Decrypt: } M = C^d \mod n$$

RSA Padding

Raw RSA (textbook RSA) is insecure. Padding schemes are required:

Padding Standard Use
PKCS#1 v1.5 Legacy Vulnerable to Bleichenbacher oracle attack; avoid in new designs
OAEP (Optimal Asymmetric Encryption Padding) PKCS#1 v2 Current standard for RSA encryption
PSS (Probabilistic Signature Scheme) PKCS#1 v2 Current standard for RSA signatures

RSA Weaknesses — In Depth

RSA has structural weaknesses in its math, its padding, its implementation, and its operational use. Each category is distinct.

1. Textbook RSA Is Deterministic and Malleable

Raw RSA with no padding is called textbook RSA. Given the same message M and the same public key, it always produces the same ciphertext C = M^e mod n. This breaks security in several ways:

  • Deterministic: An attacker can encrypt a guessed plaintext with the public key and compare it to observed ciphertext. No brute force needed for small message spaces (e.g., yes/no, credit card numbers).
  • Multiplicative malleability: If you have two ciphertexts $C_1 = M_1^e$ and $C_2 = M_2^e$, then $C_1 \cdot C_2 \mod n = (M_1 M_2)^e \mod n$. An attacker can construct a ciphertext for a product of two messages without knowing either plaintext.
  • No integrity: Textbook RSA has no authentication. Any attacker can encrypt arbitrary values with the public key.

Fix: Always use a padding scheme. OAEP for encryption, PSS for signatures.

2. Bleichenbacher Attack (PKCS#1 v1.5 Padding Oracle)

This is one of the most important practical attacks in cryptography. Discovered by Daniel Bleichenbacher in 1998 and still found in production systems today.

Setup: PKCS#1 v1.5 padding for RSA encryption prepends a fixed structure: 0x00 0x02 <random non-zero bytes> 0x00 <message>. The server decrypts and checks this structure.

The vulnerability: If the server returns a different error (or behaves differently in timing) depending on whether the decrypted padding is valid, this is an oracle — it leaks one bit of information per query.

How the attack works:

1. Attacker intercepts ciphertext C (e.g., a TLS PreMasterSecret)
2. Attacker crafts C' = C * s^e mod n for chosen s
3. Server decrypts C' and returns "bad padding" or "good padding"
4. From the response, attacker learns whether (M * s) falls in a valid padding range
5. Repeat ~1 million times, narrowing down the range of possible M
6. Eventually recover M entirely

Why it is serious: It requires only a decryption oracle — any system that returns padding error vs. non-padding error is exploitable. TLS servers that return distinct bad_record_mac vs decryption_failed alerts were directly vulnerable.

Real-world impact: ROBOT attack (2017) found Bleichenbacher-vulnerable servers at Facebook, PayPal, and many others — 19 years after the original paper.

Fix: Use OAEP padding. Implement constant-time decryption that returns the same error regardless of where decryption fails.

3. Timing Side-Channel Attack

RSA decryption computes $M = C^d \mod n$ using modular exponentiation. Naive implementations process bits of d one at a time: a 1 bit does a multiply-and-square; a 0 bit does only a square. This means:

  • Operations that process 1 bits take slightly longer than 0 bits.
  • By measuring thousands of decryption times, an attacker can statistically reconstruct bits of the private exponent d.

Paul Kocher's 1996 paper demonstrated this on real hardware. It is a remote attack if the attacker can measure network round-trip times precisely enough.

Fix: Use RSA blinding — multiply the ciphertext by a random value before decryption, then unblind the result. This makes timing independent of the private key bits.

4. Small Private Exponent Attack (Wiener's Attack)

If the private exponent d is chosen to be small (to make decryption faster), Wiener's theorem shows that if:

$$d < \frac{1}{3} n^{1/4}$$

then d can be recovered efficiently using continued fractions — no factoring required.

Why this happens: Developers sometimes deliberately choose a small d for performance. This completely breaks RSA security.

Fix: Never choose d for performance reasons. Let the math determine d from a properly chosen e.

5. Small Public Exponent Attack (Coppersmith / Håstad's Broadcast Attack)

If the same message M is encrypted to k different recipients, all using the same small exponent e, an attacker can recover M using the Chinese Remainder Theorem (CRT):

  • For e = 3: collect 3 ciphertexts $C_1, C_2, C_3$ encrypted under different public moduli but same e = 3
  • CRT gives $C = M^3$ over the integers (without mod)
  • Take the integer cube root to recover M

No factoring. No brute force. Pure algebra.

Fix: Use e = 65537 (large enough to prevent this). Always use OAEP padding, which adds randomness and prevents identical plaintexts from producing related ciphertexts.

6. Common Modulus / Shared Factor Attack

If two RSA keys share a prime factor (due to a weak or compromised RNG), the shared factor can be extracted with the Euclidean GCD algorithm — a near-instant computation.

Real-world case: A 2012 study of public RSA keys on the internet found that ~0.2% of keys shared a prime factor with at least one other key, primarily because of low-entropy random number generators on embedded devices (routers, firewalls, smart cards) generating keys at boot before enough entropy was available.

Fix: Use a cryptographically secure RNG. For embedded systems, ensure sufficient entropy before generating keys (use a hardware RNG or wait for entropy pool to fill).

7. RSA-CRT Fault Attack (Bellcore Attack)

RSA decryption in practice often uses the Chinese Remainder Theorem to speed up computation — instead of computing $C^d \mod n$, compute $C^{d_p} \mod p$ and $C^{d_q} \mod q$ separately, then combine.

If a hardware fault (bit flip, voltage glitch) corrupts one of these computations:

  • The attacker gets one correct half and one faulty half
  • From the faulty signature, they can compute $\gcd(\text{faulty sig} - \text{correct sig}, n) = p$
  • This factors n and exposes the private key

Real-world relevance: This is the basis for smartcard attacks. An attacker with physical access can inject faults and extract the private key.

Fix: Verify each signature or decryption result before returning it. Detect fault injection via hardware sensors or software checks.

8. Quantum Threat — Shor's Algorithm

Shor's algorithm, running on a sufficiently large quantum computer, can factor n in polynomial time, which breaks RSA entirely. This is not a current threat — the required quantum hardware (thousands of stable logical qubits) does not exist yet — but:

  • Data encrypted today with RSA can be harvested and decrypted later once quantum hardware matures ("harvest now, decrypt later")
  • NIST standardized post-quantum key encapsulation mechanisms (CRYSTALS-Kyber / ML-KEM) and signatures (CRYSTALS-Dilithium / ML-DSA) in 2024 as replacements

ECC — Elliptic Curve Cryptography

ECC uses the algebraic structure of elliptic curves over finite fields. It provides equivalent security to RSA with much smaller keys, making it preferred for TLS, mobile, and embedded systems.

Key Exchange — Diffie-Hellman and ECDH

The Parameters: What Is p and g?

Before any exchange can happen, both parties must agree on two public parameters:

Parameter Name What it is
p The modulus A very large prime number (typically 2048–4096 bits). It defines the finite field over which all math is done.
g The generator (base) A small number (commonly 2 or 5) that, when raised to successive powers mod p, cycles through many different values in the field.

These parameters are not secret. They are published in the protocol spec, negotiated in the TLS handshake, or agreed in advance. Their security property is that the group they define is large enough that the discrete logarithm problem is computationally infeasible.

The Core Math: Modular Exponentiation

g^a mod p means: raise g to the power a, then take the remainder when dividing by p.

Concrete small example (using tiny numbers to illustrate the concept — real values are thousands of bits):

$$p = 23, \quad g = 5$$

  • Alice picks private a = 6
  • Alice computes $A = 5^6 \mod 23 = 15625 \mod 23 = 8$
  • Bob picks private b = 15
  • Bob computes $B = 5^{15} \mod 23 = 30517578125 \mod 23 = 19$

Alice sends 8 to Bob. Bob sends 19 to Alice. These are public.

Now both compute the shared secret:

  • Alice: $B^a \mod p = 19^6 \mod 23 = 2$
  • Bob: $A^b \mod p = 8^{15} \mod 23 = 2$

Both arrive at 2. An eavesdropper who sees 8 and 19 cannot compute 2 without solving: "what power of 5, mod 23, gives 8?" — the discrete logarithm problem. With a 2048-bit p, this is computationally infeasible.

Why Modular Arithmetic Makes This Work

The key insight is the commutativity of exponents:

$$(g^a)^b \mod p = (g^b)^a \mod p = g^{ab} \mod p$$

Alice and Bob can each apply the other's public value to their own private exponent and land on the same number — the shared secret — without ever transmitting the private exponents.

sequenceDiagram
    participant A as Alice
    participant B as Bob
    Note over A,B: Both know public params p and g
    A->>A: Pick private a, compute A = g^a mod p
    B->>B: Pick private b, compute B = g^b mod p
    A->>B: Send public value A = g^a mod p
    B->>A: Send public value B = g^b mod p
    A->>A: Compute shared secret = B^a mod p = g^ab mod p
    B->>B: Compute shared secret = A^b mod p = g^ab mod p
    Note over A,B: Same shared secret. Private exponents never sent.

An eavesdropper sees g, p, A = g^a mod p, and B = g^b mod p — but cannot compute g^ab mod p without solving the discrete log problem.

DH vs DHE — What Does Ephemeral Mean?

Variant Key Lifecycle Forward Secrecy Usage
DH (static) Server reuses the same DH private key across all sessions No — long-term key compromise decrypts all recorded sessions Obsolete
DHE (ephemeral) Server generates a new DH key pair for every TLS handshake Yes Used in TLS 1.2
ECDHE (ephemeral ECC) Same as DHE but on an elliptic curve Yes Default in TLS 1.3

Static DH is the dangerous version. If an attacker records encrypted traffic today and later steals the server's long-term DH private key, they can retroactively decrypt all of it. This is the "harvest now, decrypt later" threat.

Ephemeral means the private exponent a or b is generated fresh for each session and immediately discarded after the handshake. Even if the server is fully compromised later, past sessions cannot be decrypted.

ECDH and ECDHE

ECDH replaces modular exponentiation over a finite field with scalar multiplication on an elliptic curve. The structure is identical — only the math changes:

DH concept ECDH equivalent
Prime modulus p Elliptic curve parameters (a named curve like P-256)
Generator g Generator point G on the curve
Private key a Private scalar a (a random integer)
Public value g^a mod p Public point A = a × G (scalar multiplication)
Shared secret g^ab mod p Shared point a × B = b × A = ab × G

Scalar multiplication on a curve is easy to compute in the forward direction but computationally infeasible to reverse (the elliptic curve discrete log problem).

ECDHE is just ECDH where both sides generate a fresh key pair per session — the E stands for ephemeral. This is what TLS 1.3 uses exclusively. Static (non-ephemeral) DH is not permitted in TLS 1.3.

ECDHE in TLS 1.3 — Step by Step

sequenceDiagram
    participant C as Client
    participant S as Server
    C->>S: ClientHello + key_share (ephemeral public point A = a*G)
    S->>C: ServerHello + key_share (ephemeral public point B = b*G) + Certificate
    C->>C: Compute shared secret = a * B = a*b*G
    S->>S: Compute shared secret = b * A = a*b*G
    Note over C,S: Shared secret used to derive symmetric session keys
    C->>S: Finished (authenticated with derived key)
    S->>C: Finished

Both ephemeral private keys a and b are discarded after the handshake. The certificate is used to authenticate the server's identity but is not used to encrypt anything.

DHE vs ECDHE: Which to Use and Why

Dimension DHE ECDHE

In practice, nearly all modern TLS deployments use ECDHE with Curve25519 (X25519) or P-256.

Forward Secrecy Recap

flowchart TD
    A[Attacker records encrypted TLS session today] --> B{Key exchange type?}
    B -->|Static DH or RSA key exchange| C[Attacker steals long-term private key later]
    C --> D[Attacker decrypts all recorded sessions]
    B -->|DHE or ECDHE ephemeral| E[Ephemeral private key was discarded after handshake]
    E --> F[Past sessions cannot be decrypted even with long-term key]

Forward secrecy is a critical property for protecting data at rest or in transit logs. TLS 1.3 mandates it by requiring ephemeral key exchange in all cipher suites.

How Asymmetric Encryption Is Used in TLS

TLS does not encrypt session data with asymmetric encryption. It uses asymmetric cryptography only to:

  1. Authenticate the server (via certificate and digital signature)
  2. Establish the session key (via ECDHE key exchange)

Then symmetric encryption (AES-GCM) encrypts the actual data. This hybrid approach gets the key distribution benefit of asymmetric crypto and the speed of symmetric crypto.

From ECDHE Shared Secret to TLS Session Keys

No — the ECDHE shared secret is not directly used as a symmetric key. It is raw key material that is fed into a Key Derivation Function (KDF) to produce multiple distinct session keys. This distinction is critical.

Why Not Use the Shared Secret Directly?

  • The ECDHE output (ab × G) is a point on a curve. Its x-coordinate is used as raw bytes, but it has non-uniform distribution — not suitable as a key directly.
  • A single TLS session needs multiple independent keys: one for client→server encryption, one for server→client encryption, and integrity keys. You can't use the same key for both directions.
  • The shared secret has no built-in binding to the session transcript. A KDF mixes in the handshake hash to ensure the keys are unique to this specific session.

TLS 1.3 Key Schedule

TLS 1.3 uses HKDF (HMAC-based Key Derivation Function) in a defined sequence. The ECDHE shared secret is one input alongside handshake transcript hashes:

The handshake transcript hash is mixed in at each stage, which means:

  • Keys are bound to the exact sequence of messages exchanged.
  • Any tampering with the handshake (e.g., downgrade attack) produces different keys and the Finished message check fails.

The One-Line Answer

The ECDHE shared secret is raw key material. HKDF stretches and separates it into multiple symmetric session keys — one per direction, per phase — each bound to the handshake transcript so they are unique to this session.

TLS Cipher Suites

A cipher suite is a named combination of algorithms that defines every cryptographic operation in a TLS session. When a client and server negotiate TLS, they agree on a single cipher suite to use for that connection.

Anatomy of a TLS 1.2 Cipher Suite Name

TLS _ ECDHE_RSA _ WITH _ AES_128_GCM _ SHA256
 |       |    |           |      |  |      |
 |       |    |           |      |  |      +-- Hash for PRF / HMAC
 |       |    |           |      |  +--------- GCM = AEAD mode (no separate MAC)
 |       |    |           |      +------------ 128-bit key
 |       |    |           +------------------- Symmetric cipher
 |       |    +------------------------------- Authentication (server cert key type)
 |       +------------------------------------ Key exchange algorithm
 +-------------------------------------------- Protocol

Each component does a different job during the handshake and data transfer.

Example: TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256

This is one of the most commonly negotiated cipher suites in TLS 1.2. Each component operates at a different phase of the connection and provides a different security property.

Component 1 — ECDHE — Key Exchange

When it runs: During the TLS handshake, before any application data is sent.

Security property it provides: Confidentiality of the session keys + forward secrecy.

What it does in detail:

The problem it solves is: how do two parties who have never met agree on a secret key over a channel that anyone can observe? ECDHE solves this using the elliptic curve discrete log problem.

  1. Client generates a random private scalar a and computes public point A = a × G
  2. Server generates a random private scalar b and computes public point B = b × G
  3. They exchange A and B publicly
  4. Client computes a × B = ab × G. Server computes b × A = ab × G. Both get the same point.
  5. Both immediately discard a and b after computing the shared secret.

The shared secret ab × G is never transmitted. An eavesdropper only sees A and B, and reversing scalar multiplication on an elliptic curve is computationally infeasible.

What breaks without it: Without a key exchange, there is no shared secret and no session keys can be derived. The session cannot be encrypted.

What "ephemeral" (the E) specifically adds: If ECDHE were replaced with static ECDH (no E), the server would reuse the same b forever. Anyone who later obtains b (e.g., via server compromise or legal compulsion) could compute ab × G for every past session and decrypt all recorded traffic. The ephemeral variant throws away b after each handshake, so past sessions are safe even if the server is fully compromised later.

Alternatives in TLS 1.2:
| Option | Forward Secrecy | Notes |
|--------|----------------|-------|
| ECDHE | Yes | Preferred |
| DHE | Yes | Larger keys, slower |
| RSA key exchange (static) | No | Server's RSA private key encrypts the premaster secret — stealing it decrypts everything |

RSA key exchange (TLS_RSA_WITH_*) is prohibited in TLS 1.3 for exactly this reason.

Component 2 — RSA — Authentication

When it runs: During the TLS handshake, after the key exchange parameters are sent.

Security property it provides: Server authentication — prevents a man-in-the-middle from substituting their own ECDHE public point.

The problem it solves:

ECDHE alone is vulnerable to MITM. Without authentication, nothing stops an attacker from intercepting A from the client, performing their own ECDHE with both sides, and silently relaying traffic. The client and server each think they have a secure channel to each other, but both channels terminate at the attacker.

RSA authentication closes this gap:

  1. Server sends its certificate: contains the server's RSA public key, its identity (domain name), and a CA's signature binding the two together.
  2. Server signs the ECDHE public point B (and a nonce) with its RSA private key. This is the ServerKeyExchange message.
  3. Client verifies the signature using the public key in the certificate.
  4. Client verifies the certificate was signed by a trusted CA.

An attacker cannot forge this signature because they do not have the server's RSA private key. The client is now confident it is talking to the legitimate server — and that the ECDHE point B was not substituted in transit.

What breaks without it: Without authentication, ECDHE is MITM-able. The session would be encrypted but the encryption endpoint is the attacker, not the intended server. Confidentiality against passive eavesdroppers is preserved; confidentiality against an active MITM is not.

What RSA does NOT do in this suite: RSA does not encrypt the session keys or any application data. The RSA in the suite name refers exclusively to the key type on the server's certificate and the signature scheme used for authentication.

Alternatives to RSA for authentication:
| Auth Algorithm | Certificate Type | Notes |
|----------------|-----------------|-------|
| RSA | RSA cert | Most common; large signatures |
| ECDSA | EC cert | Smaller, faster signatures; common with P-256 or P-384 |
| Ed25519 | EdDSA cert | Fastest, safest signature scheme; growing adoption |
| Anonymous (no auth) | None | Never use in production — trivially MITM-able |

Component 3 — AES_128_GCM — Symmetric Encryption (AEAD)

When it runs: After the handshake. Every byte of application data (HTTP request, response, etc.) is processed by AES-128-GCM.

Security properties it provides: Confidentiality (nobody can read the data) + Integrity (nobody can silently modify the data) + Authenticity (data came from the peer who holds the session key).

Breaking it down:

AES_128_GCM is three things combined:

Part What it is Role
AES Advanced Encryption Standard block cipher The core encryption primitive
128 Key size in bits 128-bit key = $2^{128}$ brute-force operations to break
GCM Galois/Counter Mode The mode of operation that makes AES into a stream cipher and adds authentication

How GCM works conceptually:

GCM runs AES in counter mode (CTR) to produce a keystream, which is XOR'd with the plaintext to produce ciphertext — this is the encryption. In parallel, it computes a Galois field multiplication over the ciphertext to produce a 128-bit authentication tag (essentially a MAC). The tag is appended to the ciphertext.

On decryption, the receiver recomputes the tag and compares it to the received tag. If anyone has modified even a single bit of the ciphertext in transit, the tags will not match and the record is rejected. This makes GCM an AEAD — Authenticated Encryption with Associated Data.

The "Associated Data" part means GCM also authenticates metadata (like the TLS record header) that is sent in the clear — ensuring an attacker cannot tamper with unencrypted framing either.

What data goes into the authentication tag?

The tag is computed over three inputs, not just the ciphertext:

Input What it is Encrypted? Why it is included
Ciphertext The encrypted payload bytes Yes (it is the ciphertext, not plaintext) Detects any modification to the encrypted data
Additional Authenticated Data (AAD) Unencrypted metadata sent alongside No — sent in clear Detects tampering with headers/framing that must stay readable
Length fields Bit-lengths of AAD and ciphertext Encoded internally Prevents length-extension and truncation attacks

All three are fed into the GCM GHASH function together with the authentication key (derived from the session key). The result is the 128-bit tag.

In TLS, the AAD is the TLS record header — specifically:

AAD = sequence_number (8 bytes) || content_type (1 byte) || version (2 bytes) || plaintext_length (2 bytes)

The sequence number is critical — it means every TLS record is tagged with its position. An attacker cannot reorder or replay individual records without the tag check failing.

Is the authentication tag itself sent in clear?

Yes — the tag is appended to the ciphertext and transmitted as-is. It is not encrypted.

TLS Record on the wire:
┌──────────────────────────┬──────────────────────────────┬───────────────┐
│  TLS Record Header (AAD) │  Ciphertext (AES-CTR output) │  Auth Tag     │
│  content_type, version,  │  encrypted application data  │  128 bits     │
│  length — sent in clear  │                              │  sent in clear│
└──────────────────────────┴──────────────────────────────┴───────────────┘

This is intentional and safe:

  • The tag is a MAC output — it reveals nothing about the plaintext even to someone who sees it. An attacker cannot reverse it to learn the key or the message.
  • The tag must be readable by the receiver before decryption to allow verify-then-decrypt ordering. If the tag were encrypted, the receiver would have to decrypt first, which re-introduces the vulnerability of returning plaintext from corrupted data.
  • Without the key, the tag is computationally unforgeable — an attacker cannot compute a valid tag for a modified ciphertext even knowing the original tag.

What an attacker can and cannot do with the visible tag:

Attacker action Result
Read the tag Learns nothing — it is indistinguishable from random bytes without the key
Modify the ciphertext and keep the original tag Tag check fails on decryption — record rejected
Try to forge a valid tag for modified ciphertext Computationally infeasible — requires the AES session key
Truncate or drop records Sequence number in AAD breaks — subsequent records fail tag check
Replay a previously valid record Sequence number in AAD is unique per record — tag check fails

What breaks without GCM's authentication (using a non-AEAD mode like CBC without MAC):

Without built-in integrity, the cipher only provides confidentiality. An attacker can:

  • Flip specific bits in the ciphertext to predictably flip bits in the decrypted plaintext (bit-flipping attack)
  • Perform padding oracle attacks (POODLE, BEAST on CBC-mode TLS)

This is why CBC cipher suites in TLS 1.2 require a separate HMAC, and why TLS 1.3 mandates AEAD only.

Why AES-128 not AES-256:

AES-128 provides 128 bits of security — brute forcing requires $2^{128}$ operations, which is physically impossible with classical computers for the foreseeable future. AES-256 provides a larger margin but is ~40% slower. For most deployments AES-128 is the right choice; AES-256 is used in environments with higher long-term data sensitivity requirements.

Alternative AEAD ciphers:
| Cipher | Notes |
|--------|-------|
| AES-256-GCM | Higher security margin; slower |
| ChaCha20-Poly1305 | Software-optimised; faster than AES on devices without hardware AES acceleration (mobile, IoT) |
| AES-128-CCM | Less common; used in constrained environments |

Component 4 — SHA256 — Hash for PRF and Handshake Integrity

When it runs: During the handshake only. It is not involved in encrypting application data.

Security properties it provides: Key derivation correctness and handshake transcript integrity.

Two distinct jobs in TLS 1.2:

Job A — Pseudorandom Function (PRF) for key derivation:

After ECDHE produces the raw shared secret, TLS 1.2 uses a PRF to expand it into the master secret and then into the actual session keys. The PRF is built on HMAC-SHA256:

PreMaster Secret (from ECDHE)
        |
        v
   PRF(HMAC-SHA256) + client_random + server_random
        |
        v
   Master Secret (48 bytes)
        |
        v
   PRF(HMAC-SHA256) + "key expansion" label
        |
        v
   Key block → split into:
     client write key   (AES-128 key, 16 bytes)
     server write key   (AES-128 key, 16 bytes)
     client write IV    (GCM nonce, 4 bytes)
     server write IV    (GCM nonce, 4 bytes)

The client and server randoms (sent in ClientHello and ServerHello in plaintext) ensure the derived keys are unique per session even if the same ECDHE shared secret were reused.

Job B — Finished message MAC:

At the end of the handshake, both sides compute:

Finished = PRF(master_secret, "finished", SHA256(all_handshake_messages))

This is a MAC over the entire handshake transcript. If a MITM modified any handshake message (e.g., downgraded the cipher suite, swapped the ECDHE point), the transcript hash would differ and the Finished check would fail, terminating the connection.

This is the mechanism that detects handshake tampering. Without it, an active attacker could manipulate negotiation messages and neither side would notice.

What SHA256 is NOT doing here: It is not hashing the application data stream. Application data integrity comes from AES-GCM's authentication tag, not SHA256.

Why the hash algorithm in the suite matters:

Older suites used SHA1 in the PRF. SHA1 collision attacks (SHAttered, 2017) mean a crafted certificate pair could theoretically produce the same transcript hash, undermining the Finished check. SHA256 has no known practical collision attacks and provides a 256-bit output — the current standard.

TLS 1.3 Cipher Suites — Simpler by Design

TLS 1.3 removed the key exchange and authentication components from the cipher suite name entirely. Those are negotiated separately. A TLS 1.3 cipher suite only specifies the AEAD cipher and the hash for HKDF:

TLS 1.3 Cipher Suite AEAD Cipher Hash for HKDF
TLS_AES_128_GCM_SHA256 AES-128-GCM SHA-256
TLS_AES_256_GCM_SHA384 AES-256-GCM SHA-384
TLS_CHACHA20_POLY1305_SHA256 ChaCha20-Poly1305 SHA-256

Key exchange is always ECDHE (X25519 or P-256). Authentication is always via the certificate. Neither is part of the suite name. This eliminates the entire class of downgrade attacks that existed in TLS 1.2 where an attacker could negotiate a weaker key exchange algorithm.

What Makes a Cipher Suite "Good" or "Weak"

Property Good Weak / Avoid
Key exchange ECDHE, DHE (ephemeral) RSA key exchange (static), EXPORT DH
Authentication ECDSA, RSA-2048+ Anonymous (no auth), DSA-1024
Symmetric cipher AES-128-GCM, AES-256-GCM, ChaCha20-Poly1305 RC4, 3DES, NULL cipher
MAC / integrity GCM (built-in AEAD), HMAC-SHA256 MD5, SHA1 HMAC, NULL MAC
Forward secrecy Any suite with ECDHE or DHE Any suite without ephemeral exchange

Cipher suites containing EXPORT, NULL, anon, RC4, DES, or MD5 are universally broken and negotiating them is a misconfiguration that tools like testssl.sh and sslyze will flag.
