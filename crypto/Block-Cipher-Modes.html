Block Cipher Modes of Operation

Category: Cryptography
Parent: crypto/README.md

Overview

A block cipher (like AES) always encrypts exactly one fixed-size block — 128 bits for AES. To encrypt real data, which is rarely exactly 128 bits and often contains structure, you need a mode of operation that defines how blocks relate to each other and whether the output proves integrity.

The mode choice determines:

  • whether identical plaintext blocks leak structure in ciphertext
  • whether an IV or nonce is required and how critical uniqueness is
  • whether integrity and authentication are built in
  • what attack surface exists (padding oracles, nonce reuse, malleability)

ECB — Electronic Codebook

How It Works

Every 128-bit block is encrypted independently with the same key. No chaining, no IV.

Plaintext:  [Block 1] [Block 2] [Block 3]
                ↓         ↓         ↓
Ciphertext: [Enc(B1)] [Enc(B2)] [Enc(B3)]

The Fatal Flaw

Identical plaintext blocks produce identical ciphertext blocks. If data has repetition (file headers, protocol fields, pixel data), that structure leaks through. The classic demonstration is encrypting a bitmap image in ECB mode — you can still see the outline of the original image in the ciphertext because uniform-color regions encrypt to repeating blocks.

Properties

Property ECB
IV / nonce required No
Integrity No
Parallelizable Yes
Leaks plaintext pattern Yes

Use: Never for structured or repeating real data.

CBC — Cipher Block Chaining

How It Works

Each plaintext block is XOR'd with the previous ciphertext block before encryption. A random IV seeds the first block.

IV → XOR(P1) → Encrypt → [C1]
               [C1] → XOR(P2) → Encrypt → [C2]
               [C2] → XOR(P3) → Encrypt → [C3]

Decryption reverses this: each ciphertext block is decrypted and then XOR'd with the previous ciphertext block.

Why It Was Popular

Identical plaintext blocks produce different ciphertext because each block depends on all previous ciphertext. No structure leakage like ECB.

Why It Is Dangerous Now — Padding Oracles

CBC requires plaintext to be a multiple of the block size, so padding must be added (PKCS#7 is common). If a server:

  1. Decrypts a submitted ciphertext, and
  2. Returns different observable errors for "bad padding" vs "wrong MAC"

then an attacker can submit modified ciphertexts and, one byte at a time, recover the plaintext without the key. This is the padding oracle attack.

Real-world examples:

  • POODLE — SSL 3.0 CBC padding oracle
  • BEAST — TLS 1.0 predictable IV chaining
  • Lucky13 — timing side-channel on CBC-HMAC in TLS

See CBC-Padding-Oracle-Attack.md for a full byte-level walkthrough, working Python implementation, and real-world CVE analysis.

IV must be random and unpredictable. Reusing an IV with the same key exposes plaintext relationships. Using a predictable IV (as TLS 1.0 did, where the last block of one record became the IV for the next) enables chosen-plaintext attacks.

Properties

Property CBC
IV / nonce required Yes (random)
Integrity No — needs MAC
Parallelizable Decrypt only
Leaks plaintext pattern No

Use: Legacy TLS 1.0/1.1, legacy disk encryption. Avoid for new designs.

CTR — Counter Mode

How It Works

CTR turns a block cipher into a stream cipher. A counter (nonce + incrementing number) is encrypted to produce a keystream, which is XOR'd with the plaintext. No padding needed.

Encrypt(Nonce || Counter=0) → Keystream_0 → XOR Plaintext_0 → Ciphertext_0
Encrypt(Nonce || Counter=1) → Keystream_1 → XOR Plaintext_1 → Ciphertext_1
Encrypt(Nonce || Counter=2) → Keystream_2 → XOR Plaintext_2 → Ciphertext_2

Each keystream block is independent — blocks can be encrypted or decrypted in parallel, and you can seek into the middle of the stream.

The Critical Failure Mode — Nonce Reuse (Two-Time Pad)

If two different messages are encrypted with the same key and the same nonce, the keystreams are identical. XOR-ing the two ciphertexts cancels the keystream:

$$C_1 \oplus C_2 = (P_1 \oplus KS) \oplus (P_2 \oplus KS) = P_1 \oplus P_2$$

An attacker who observes both ciphertexts gets the XOR of the two plaintexts, which is often enough to recover both via frequency analysis or known prefixes. This is called the two-time pad attack and breaks WEP Wi-Fi encryption and many home-grown systems.

CTR also provides no integrity — ciphertext is bitwise malleable. Flipping a bit in Ciphertext_i flips the corresponding bit in Plaintext_i after decryption.

Properties

Property CTR
IV / nonce required Yes (unique)
Integrity No — needs MAC
Parallelizable Yes
Leaks plaintext pattern No

Use: Building block for protocol design. In isolation, prefer GCM which adds integrity on top.

GCM — Galois/Counter Mode

How It Works

GCM = CTR mode for encryption + Galois field MAC (GHASH) for authentication, composed into one AEAD (Authenticated Encryption with Associated Data) construction. Both operations run concurrently over the data.

flowchart TD
    A[Plaintext] --> B[CTR encryption]
    B --> C[Ciphertext]
    C --> D[GHASH authentication]
    E[Additional Authenticated Data] --> D
    D --> F[Authentication Tag 128-bit]

Associated data (AD): Data you want to authenticate but not encrypt — for example a packet header, message metadata, or a version field. It is transmitted in plaintext but any modification is detected by the tag.

What AEAD Means in Practice

A decryption call only returns plaintext if the authentication tag verifies successfully. There is no partial output and no decryption of modified ciphertext. This eliminates the class of attacks that required a separate MAC step to be glued on correctly — a common source of bugs in encrypt-then-MAC or MAC-then-encrypt compositions.

The Catastrophic Nonce Reuse Problem

GCM's authentication key H is computed as:

$$H = \text{Encrypt}(K, 0^{128})$$

If you reuse a nonce under the same key, an attacker who observes two (ciphertext, tag) pairs can:

  1. Algebraically recover H from the two authentication polynomials.
  2. With H in hand, forge authentication tags for arbitrary messages.

This is significantly worse than CTR nonce reuse — it breaks both confidentiality and authentication simultaneously. A single nonce reuse makes every past and future message under that key forgeable.

Safe nonce strategies:

  • Random 96-bit nonces (safe up to ~2^32 messages before birthday bound matters; 2^48 provides comfortable margin)
  • Monotonic counter per key, never reset on restart
  • Key rotation before nonce space is approached
  • Use AES-GCM-SIV for misuse resistance (nonce reuse degrades to CTR-level confidentiality loss but does not break authentication)

Properties

Property GCM
IV / nonce required Yes (unique — critical)
Integrity Yes (built-in tag)
Parallelizable Yes
Leaks plaintext pattern No
Authenticated data Yes (associated data)

Use: TLS 1.3, HTTPS, QUIC, disk encryption with integrity, general-purpose AEAD. The default choice for new systems.

XTS — XEX-based Tweaked CodeBook with Ciphertext Stealing

How It Works

XTS is designed specifically for disk and storage sector encryption. Each sector is encrypted independently using a "tweak" derived from the sector index so that the same plaintext in different sectors produces different ciphertext.

Two keys are used: Key1 for encrypting the sector data and Key2 for deriving a per-block tweak from the sector/block index:

Tweak_i = Encrypt(Key2, sector_number) << i   (shifted per block within the sector)
XTS_block = Decrypt(Key1, Ciphertext_block XOR Tweak_i) XOR Tweak_i

Ciphertext Stealing (the "CTS" part) handles sectors that are not a multiple of the block size without padding.

Why XTS Exists

Full-disk encryption has a unique constraint that makes AEAD unsuitable:

  • Disk sectors are fixed-size — there is no room to store an authentication tag alongside encrypted data.
  • Sectors must be readable and writable randomly and independently.
  • A length-preserving cipher is required.

XTS satisfies these constraints. It provides semantic security per sector (same plaintext in different sectors produces different ciphertext, unlike ECB).

What XTS Does NOT Provide

  • No integrity or authentication — an attacker with physical access who can modify ciphertext on disk can flip bits in a predictable way (malleability).
  • Not suitable for network messages — if you need integrity, use GCM.
  • Not suitable for short messages — designed for 512- to 4096-byte sectors.

Use: FileVault (macOS), BitLocker (Windows), dm-crypt (Linux), hardware storage encryption. Never for network protocol messages.

Key Attack Scenarios

Mode Attack Precondition What the Attacker Gets
ECB Pattern analysis Structured or repeating plaintext Plaintext structure visible in ciphertext
CBC Padding oracle Server reveals padding error Full plaintext recovery, byte by byte
CBC IV prediction IV is predictable or reused Chosen-plaintext advantage
CTR Two-time pad Nonce reused across two messages XOR of both plaintexts, often recoverable
GCM Nonce reuse Same nonce used twice under same key Authentication key H recovered; tags forgeable
XTS Sector bit flip Physical or logical write access Controlled plaintext modification

Practical Decision Rules

  1. New system, general purpose: Use AES-GCM (with careful nonce discipline) or ChaCha20-Poly1305.
  2. No AES hardware acceleration: Prefer ChaCha20-Poly1305 (software-efficient).
  3. Nonce reuse is a real risk in your design: Consider AES-GCM-SIV for misuse resistance.
  4. Full-disk or storage-layer encryption: Use XTS; it is the only appropriate mode for fixed-size sectors.
  5. Legacy interop: Understand CBC padding oracle risk; always pair with a MAC (HMAC-SHA256 in encrypt-then-MAC order, not MAC-then-encrypt).
  6. Never: Use ECB for anything except testing primitives in isolation.

Key Takeaways

  • ECB leaks structure — never use it for real data.
  • CBC was standard for years but is now legacy. Padding oracles and IV misuse make it dangerous in network protocols.
  • CTR is a clean stream cipher mode but has no integrity and is destroyed by nonce reuse.
  • GCM adds authentication on top of CTR in a single efficient pass. It is the modern default but demands strict nonce uniqueness.
  • XTS is purpose-built for disk encryption. It is length-preserving and has no integrity tag by design.
  • The safest path: use a high-level library (like AWS Encryption SDK, libsodium, or TLS 1.3) that chooses and manages modes for you.
