CBC Padding Oracle Attack

Category: Cryptography
Parent: Block-Cipher-Modes.md

What Is a Padding Oracle?

A padding oracle is any system behaviour that leaks whether the padding on a decrypted ciphertext is valid. The leak does not have to be an explicit error message. It can be:

  • A different HTTP status code (200 vs 403 vs 500)
  • A timing difference between a decryption error and a padding error
  • A different error message body
  • A redirect vs an inline error
  • A database exception vs an application exception

The attacker does not need the key. They only need a way to ask: "Does this ciphertext decrypt to something with valid PKCS#7 padding?" That single bit of information — valid or invalid — is enough to recover the entire plaintext, one byte at a time.

Prerequisites: How CBC Decryption Works

               C0 (IV)     C1 (block 1)    C2 (block 2)
                 │               │               │
                 │          ┌────▼────┐     ┌────▼────┐
                 │          │ Decrypt │     │ Decrypt │
                 │          └────┬────┘     └────┬────┘
                 │               │ D1            │ D2
                 └──────XOR──────┘               │
                        │                C1 ─── XOR
                        P1                       │
                                                 P2

The critical formula for block i is:

$$P_i = \text{Decrypt}(C_i) \oplus C_{i-1}$$

Let D_i = Decrypt(C_i) be the intermediate value after the block cipher decryption but before the XOR with the previous ciphertext block. We know C_i and C_{i-1}, but we do not know D_i directly.

PKCS#7 Padding

PKCS#7 padding fills the last block to a multiple of the block size (16 bytes for AES). The padding byte value equals the number of padding bytes added.

Plaintext length 11 bytes → 5 bytes of padding (value 0x05):
  48 65 6c 6c 6f 20 57 6f 72 6c 64 05 05 05 05 05
  H  e  l  l  o     W  o  r  l  d  ↑  ↑  ↑  ↑  ↑
                                   padding = 5×\x05

Plaintext length 15 bytes → 1 byte of padding (value 0x01):
  48 65 6c 6c 6f 20 57 6f 72 6c 64 21 21 21 21 01
                                               ↑
                                            padding = 1×\x01

Plaintext exactly 16 bytes → full extra block of 0x10:
  48 65 6c 6c 6f 20 57 6f 72 6c 64 20 57 6f 72 6c   ← original 16 bytes
  10 10 10 10 10 10 10 10 10 10 10 10 10 10 10 10   ← padding block (16×\x10)

A valid padding check means: the last byte value is n, and the preceding n-1 bytes are all also n, and 1 ≤ n ≤ block_size.

How the Attack Works — Step by Step

We have a ciphertext C = C0 || C1 (IV + one ciphertext block) encrypting one block of plaintext P1.

We want to recover P1 without the key.

Goal: Recover One Byte of D1

We manufacture a fake ciphertext C' = C'_prev || C1 where we control C'_prev byte by byte. We send this to the oracle. The oracle decrypts it as:

$$P' = \text{Decrypt}(C_1) \oplus C'{\text{prev}} = D_1 \oplus C'$$}

The oracle tells us: is the padding of P' valid?

Step 1: Recover the Last Byte (targeting \x01 padding)

We want P'[15] = 0x01 (single-byte valid padding).

$$P'[15] = D_1[15] \oplus C'_{\text{prev}}[15] = 0x01$$

Therefore: $D_1[15] = 0x01 \oplus C'_{\text{prev}}[15]$

We iterate C'_prev[15] through all 256 values (0x00 to 0xFF), keeping bytes 0–14 as zero (or any fixed value) and submit each to the oracle.

Exactly one value of C'_prev[15] will cause the oracle to return "valid padding" — meaning P'[15] = 0x01. We now know D_1[15].

Then the actual plaintext byte is:

$$P_1[15] = D_1[15] \oplus C_0[15]$$

Step 2: Recover the Second-to-Last Byte (targeting \x02\x02 padding)

We now want P'[14] = P'[15] = 0x02.

  • Fix C'_prev[15] = D_1[15] \oplus 0x02 (we know D_1[15], so this is easy)
  • Iterate C'_prev[14] through 0–255

When the oracle returns valid, $D_1[14] \oplus C'{\text{prev}}[14] = 0x02$, so: $D_1[14] = C'[14] \oplus 0x02$}

Then: $P_1[14] = D_1[14] \oplus C_0[14]$

Repeat for All 16 Bytes

Each byte costs at most 256 oracle queries. 16 bytes × 256 queries = at most 4096 requests to decrypt one 16-byte block. In practice, the average is ~128 queries per byte because the correct value is found on average halfway through.

Full Python Implementation

from Crypto.Cipher import AES
from Crypto.Util.Padding import pad, unpad
import os

# ─── Simulated server-side oracle ────────────────────────────────────────────

SECRET_KEY = os.urandom(16)  # 128-bit AES key — attacker does NOT know this

def encrypt(plaintext: bytes) -> tuple[bytes, bytes]:
    """Encrypt plaintext. Returns (iv, ciphertext)."""
    iv = os.urandom(16)
    cipher = AES.new(SECRET_KEY, AES.MODE_CBC, iv)
    ct = cipher.encrypt(pad(plaintext, 16))
    return iv, ct

def padding_oracle(iv: bytes, ciphertext: bytes) -> bool:
    """
    The oracle. Returns True if decrypted ciphertext has valid PKCS#7 padding.
    In a real attack this is a remote server call — the attacker observes
    only True/False (valid/invalid padding).
    """
    try:
        cipher = AES.new(SECRET_KEY, AES.MODE_CBC, iv)
        unpad(cipher.decrypt(ciphertext), 16)
        return True
    except ValueError:
        return False


# ─── Attacker code ────────────────────────────────────────────────────────────

def recover_intermediate_block(oracle, prev_block: bytes, target_block: bytes) -> bytes:
    """
    Given a prev_block and target_block, recover D = Decrypt(target_block)
    (the intermediate value before XOR with prev_block).
    Uses the padding oracle to brute-force each byte from last to first.
    """
    block_size = 16
    D = bytearray(block_size)  # intermediate decrypted bytes we are recovering

    for byte_pos in range(block_size - 1, -1, -1):
        padding_byte = block_size - byte_pos  # target padding value (1, 2, 3, ...)

        # Build a controlled previous block.
        # Bytes after byte_pos are set so they produce 'padding_byte' when XOR'd with D.
        # Bytes before byte_pos (including byte_pos itself) are iterated.
        crafted = bytearray(block_size)
        for k in range(byte_pos + 1, block_size):
            crafted[k] = D[k] ^ padding_byte

        found = False
        for guess in range(256):
            crafted[byte_pos] = guess
            if oracle(bytes(crafted), target_block):
                # Verify it is not a false positive caused by accidental multi-byte padding.
                # Flip the byte before byte_pos (if one exists) and re-check.
                if byte_pos > 0:
                    verify = bytearray(crafted)
                    verify[byte_pos - 1] ^= 0x01
                    if not oracle(bytes(verify), target_block):
                        continue  # false positive — actual padding was longer; keep searching
                D[byte_pos] = guess ^ padding_byte
                found = True
                break

        if not found:
            raise RuntimeError(f"Could not find byte at position {byte_pos}")

    return bytes(D)


def padding_oracle_decrypt(oracle, iv: bytes, ciphertext: bytes) -> bytes:
    """
    Decrypt all blocks of ciphertext using only the padding oracle.
    The attacker provides iv and ciphertext (both are transmitted in plaintext
    in most real protocols — they are not secret).
    """
    block_size = 16
    assert len(ciphertext) % block_size == 0

    # Split ciphertext into blocks
    blocks = [ciphertext[i:i+block_size] for i in range(0, len(ciphertext), block_size)]
    prev_blocks = [iv] + blocks[:-1]  # IV precedes first block; each block precedes next

    plaintext = b""
    for i, (prev, curr) in enumerate(zip(prev_blocks, blocks)):
        print(f"  [*] Recovering block {i+1}/{len(blocks)}...")
        D = recover_intermediate_block(oracle, prev, curr)
        # P_i = D_i XOR C_{i-1}   (XOR intermediate with actual previous block)
        P = bytes(d ^ p for d, p in zip(D, prev))
        plaintext += P

    # Remove PKCS#7 padding from the final plaintext
    pad_len = plaintext[-1]
    return plaintext[:-pad_len]


# ─── Demo ─────────────────────────────────────────────────────────────────────

if __name__ == "__main__":
    secret_message = b"Attack at dawn!!"   # 16 bytes — exactly one block

    print("=== Padding Oracle Attack Demo ===\n")
    print(f"Original plaintext : {secret_message}")

    iv, ciphertext = encrypt(secret_message)
    print(f"IV (hex)           : {iv.hex()}")
    print(f"Ciphertext (hex)   : {ciphertext.hex()}")
    print(f"\nAttacker knows: IV and ciphertext. Does NOT know the key.")
    print(f"Attacker queries the oracle (at most 256 × 16 = 4096 times)...\n")

    recovered = padding_oracle_decrypt(padding_oracle, iv, ciphertext)
    print(f"\nRecovered plaintext: {recovered}")
    assert recovered == secret_message, "Attack failed!"
    print("\n[+] Attack succeeded. Full plaintext recovered without the key.")

Byte-Level Walkthrough — Recovering the Last Byte

Suppose the target block decrypts to intermediate bytes D[0..15] (unknown) and the actual plaintext last byte is P[15] = 0x21 (!).

Real values (attacker does NOT know):
  D[15]     = 0x11   (intermediate decrypt output)
  C_prev[15]= 0x30   (known — it's the previous ciphertext block or IV)
  P[15]     = D[15] XOR C_prev[15] = 0x11 XOR 0x30 = 0x21 = '!'  ✓

Attacker manufactures crafted_prev with all zeros except the last byte:
  crafted_prev = 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 [guess]

  guess=0x00 → oracle sees P'[15] = D[15] XOR 0x00 = 0x11  invalid padding ✗
  guess=0x01 → P'[15] = 0x11 XOR 0x01 = 0x10              invalid padding ✗
  ...
  guess=0x10 → P'[15] = 0x11 XOR 0x10 = 0x01              VALID padding \x01 ✓

  Oracle returns True for guess=0x10.
  D[15] = guess XOR 0x01 = 0x10 XOR 0x01 = 0x11
  P[15] = D[15] XOR C_prev[15] = 0x11 XOR 0x30 = 0x21 = '!'

The attacker just recovered the last plaintext byte with at most 256 requests.

False Positive Edge Case

When targeting \x01 padding at position 15, there is a subtle false positive risk: the other bytes in the crafted block (positions 0–14 are zeros) might accidentally produce a longer valid padding pattern.

For example, if by coincidence bytes 14 and 15 of the decrypted block both happen to produce \x02\x02, the oracle returns True — but not because we found the \x01 target. This is why the implementation flips a bit at position byte_pos - 1 and re-checks:

if byte_pos > 0:
    verify = bytearray(crafted)
    verify[byte_pos - 1] ^= 0x01   # disturb the byte before our target
    if not oracle(bytes(verify), target_block):
        continue  # was a multi-byte false positive — keep searching

If disturbing the preceding byte breaks validity, the padding must have been longer than \x01 — it was accidental multi-byte padding, not the target.

Real-World Attack Examples

POODLE (2014) — SSL 3.0

SSL 3.0 used CBC with a specific padding check: only the last byte had to be valid (the rest could be anything). This made the oracle easier to exploit.

Attack surface: Any SSL 3.0 HTTPS connection where the attacker could inject chosen plaintext (e.g., browser making requests to attacker-controlled domain). POODLE forced browsers to downgrade to SSL 3.0 and then exploited the CBC padding oracle to steal session cookies.

Fix: Disable SSL 3.0 entirely.

BEAST (2011) — TLS 1.0

TLS 1.0 used the last block of the previous TLS record as the IV for the next record — making IVs predictable. This enabled a chosen-plaintext attack within CBC mode without needing a padding oracle per se.

Fix: Upgrade to TLS 1.1+ which fixed IV handling; or use RC4 as a workaround at the time (since deprecated).

Lucky13 (2013) — Timing Side-Channel

Even with constant-time MAC checking, the MAC computation in CBC-HMAC took slightly different amounts of CPU time depending on how much padding was present. This timing difference (13 clock cycles on average — hence the name) was measurable over a network.

Fix: Constant-time MAC verification; switch to AEAD cipher suites.

ASP.NET Padding Oracle (2010)

ASP.NET's ValidateRequest returned HTTP 500 for padding errors and HTTP 200 (or a different 500 with different body) for MAC errors. This exposed every ASP.NET web application's ViewState and encrypted session data to the attack.

Why The Attack Is So Powerful

Property Implication
Requires only 256 oracle queries per byte For a 16-byte block: at most 4,096 requests. Typically ~2,048 on average.
Works on any CBC-encrypted block Not just the last — any block can be targeted by presenting (prev_block, target_block)
Scales to arbitrary ciphertext length 100-block ciphertext costs ~200,000 oracle queries — a few minutes.
Key is never needed The math only requires knowledge of the ciphertext and access to the oracle.
Works over HTTPS The oracle is a remote server; TLS wrapping the outer connection does not help because the decryption oracle is running inside the server.

Mitigations

Mitigation How It Prevents the Attack
Use GCM or another AEAD Authentication tag is checked before any decryption output is returned. Forge one byte of ciphertext → authentication fails before padding is ever evaluated.
Encrypt-then-MAC Apply HMAC over the ciphertext before decryption. Reject any ciphertext that fails the MAC check without attempting to decrypt. The padding check is never reached for tampered ciphertext.
Constant-time padding verification Does not prevent the oracle — it only closes the timing channel. Still leaks through different response codes or bodies.
Uniform error responses Return the same error regardless of whether it was a padding error or MAC error. Reduces information leakage but does not eliminate timing channels.
TLS 1.3 Removed all CBC cipher suites. Only AEAD (AES-GCM, ChaCha20-Poly1305) allowed.

The only fully correct fix is not using unauthenticated CBC. Uniform errors and constant-time checks reduce the attack surface but do not eliminate it — Lucky13 demonstrated that even well-implemented TLS stacks leaked enough timing information.

Mac-Then-Encrypt vs Encrypt-Then-Mac

The ordering matters enormously:

MAC-then-Encrypt (dangerous with CBC):
  1. Compute MAC over plaintext
  2. Append MAC to plaintext
  3. Encrypt (plaintext || MAC) with CBC
  → Oracle can still see padding validity before MAC is checked.
     Attacker submits modified ciphertext, padding is checked first.

Encrypt-then-MAC (safe):
  1. Encrypt plaintext with CBC
  2. Compute MAC over (IV || ciphertext)
  3. Send (IV || ciphertext || MAC)
  → Server checks MAC first. Any tampered ciphertext fails immediately.
     Padding is never evaluated for forge requests.

TLS 1.0 and 1.1 used MAC-then-Encrypt — this is why the attack was broadly applicable. TLS 1.2 still allowed CBC but switched to a more careful verify-then-decrypt flow. TLS 1.3 removed CBC entirely.

Key Takeaways

  • A padding oracle is any observable difference in server behaviour between "bad padding" and "bad MAC/other error" — not just an explicit error message.
  • The attack requires no knowledge of the key. It only needs ciphertext and oracle access.
  • Each byte costs at most 256 oracle queries. A full 16-byte block costs at most 4,096 requests.
  • The correct fix is AEAD (GCM, ChaCha20-Poly1305), not better error handling or constant-time code.
  • MAC-then-Encrypt is fundamentally broken for CBC. Encrypt-then-MAC is safe because forgeries are rejected before padding is ever evaluated.
  • TLS 1.3 resolved this permanently by removing all CBC cipher suites.
