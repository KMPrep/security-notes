Entropy and PRNGs

Category: Cryptography — Randomness, Entropy Sources, PRNG Security

Overview

Cryptographic randomness is the foundation of nearly every security primitive: key generation, nonces, session tokens, salts, challenge values, and padding. Weak randomness is one of the most common and catastrophic real-world cryptographic failures. An attacker who can predict or bias the random number generator can:

  • Predict generated keys and break encryption
  • Predict session tokens and hijack sessions
  • Recover ECDSA private keys from biased nonces (see Asymmetric Encryption)
  • Forge cryptographic values

True Randomness vs Pseudo-Randomness

Type Source Predictable? Use
True RNG (TRNG) Physical entropy: thermal noise, hardware interrupts, mouse movement No Seeding PRNGs; hardware-generated keys
PRNG (Pseudo-RNG) Deterministic algorithm seeded from entropy pool Yes, if seed is known General-purpose random values
CSPRNG (Cryptographically Secure PRNG) PRNG designed to be unpredictable even with partial state knowledge No (if correctly seeded) All cryptographic purposes

Critical rule: Never use a non-cryptographic PRNG (e.g., rand(), Math.random(), Python's random module) for any security purpose. These are designed for statistical distribution, not unpredictability.

Entropy

Entropy is the measure of unpredictability in a system. Shannon entropy H of a source with n outcomes with probability $p_i$:

$$H = -\sum_{i=1}^{n} p_i \log_2 p_i$$

In practice, "bits of entropy" describes how hard it is to guess a random value. A 256-bit key generated from a CSPRNG with a well-seeded entropy pool has 256 bits of entropy — brute force requires $2^{256}$ guesses.

Entropy Sources in Operating Systems

Source Type Entropy Quality
Hardware RNG (RDRAND on x86, RDSEED) Hardware High; fast
Interrupt timing (keyboard, mouse, network) Software-collected Moderate; slow to accumulate
Disk seek timing Software-collected Low on SSDs
CPU execution timing Software-collected Moderate
/dev/random (Linux) Blocking entropy pool Blocks until sufficient entropy accumulated
/dev/urandom (Linux) Non-blocking CSPRNG Never blocks; safe for most use; preferred
getrandom() syscall (Linux 3.17+) CSPRNG Blocks only at boot until initial seed; preferred
CryptGenRandom (Windows) CSPRNG Windows equivalent; safe
BCryptGenRandom (Windows) CSPRNG Modern Windows API; preferred

/dev/random vs /dev/urandom

This is a historically contentious distinction. Modern guidance (since Linux 4.8 and cryptographic community consensus):

  • Use /dev/urandom or getrandom() for all cryptographic purposes. The CSPRNG is forward-secure once initially seeded.
  • /dev/random blocking behaviour was based on an overly conservative model. Once the CSPRNG is seeded, additional blocking provides no measurable security benefit.
  • Exception: Very early boot (before the entropy pool is seeded) — getrandom() blocks correctly here; use it instead of /dev/urandom to avoid the boot-time edge case.

Entropy Starvation

Entropy starvation occurs when a system cannot accumulate sufficient entropy to seed its PRNG, producing predictable or weak random values.

Common Scenarios

Scenario Risk
Virtual machines at boot VMs have no physical hardware entropy sources; all VMs cloned from the same image start with identical entropy state
Containers with no hardware RNG access Container may share host entropy pool; may starve
Embedded IoT devices Few interrupt sources; low-entropy hardware
Cloud instances (early history) Some hypervisors provided no hardware entropy to guests

Entropy Starvation Attacks

Debian OpenSSL RNG bug (2008): A Debian maintainer removed two lines of OpenSSL code that fed process ID and other data into the entropy pool, thinking they were "uninitialized memory" causing Valgrind warnings. This reduced the entropy of generated keys to only the process ID — about 15 bits. Every SSH key, TLS certificate, and OpenVPN key generated on Debian/Ubuntu from 2006–2008 was predictable. Entire key space could be enumerated in hours.

Implications: ~32,000 possible 2048-bit RSA keys. All affected keys had to be revoked and regenerated.

Mitigations for Entropy Starvation

Mitigation How
Hardware RNG (RDRAND/RDSEED) Intel/AMD CPUs provide hardware entropy; modern kernels use it
Virtio-RNG / VirtIO entropy device Hypervisor feeds hardware entropy to guest VMs
haveged daemon Generates entropy from CPU performance counter jitter
/dev/hwrng Direct hardware RNG access
Cloud provider managed keys AWS KMS, GCP KMS use HSM-backed entropy
Early-boot blocking getrandom() blocks until seed is ready; prevents weak early-boot keys

CSPRNGs in Practice

Linux Kernel CSPRNG

Linux uses ChaCha20 as its CSPRNG (since kernel 4.8, replacing ARC4). The entropy pool is seeded from hardware sources and OS events. Once seeded, /dev/urandom output is forward-secret — even if an attacker learns the current PRNG state, they cannot recover past output.

Language-Level CSPRNGs

Language Secure API Insecure (do not use for crypto)
Python secrets module, os.urandom() random module
Java SecureRandom Random
JavaScript (Node.js) crypto.randomBytes() Math.random()
Go crypto/rand math/rand
C/C++ getrandom(), BCryptGenRandom rand(), srand()
Ruby SecureRandom rand

UUID Generation

UUIDs (version 4) are generated from random bytes. They require a CSPRNG to be unpredictable. A UUID generated from Math.random() in JavaScript is not safe as a session token or secret identifier. Use crypto.randomUUID() (Web Crypto API) or equivalent.

Nonces and Initialization Vectors

A nonce ("number used once") is a random or pseudo-random value used once per operation to ensure that identical inputs produce different ciphertext. IVs in CBC mode and nonces in GCM/ChaCha20 serve this purpose.

Requirements:

  • Must never be reused with the same key
  • GCM and ChaCha20: 96-bit nonce; reuse is catastrophic
  • CBC: 128-bit random IV; must be unpredictable

Nonce generation strategies:

Strategy Safety Notes
Random (CSPRNG) Safe if nonce space is large enough Risk: birthday collision at $2^{n/2}$ operations
Counter Safe if counter is managed correctly Requires persistent state; dangerous at restart
Random + counter Hybrid; best of both Use 64-bit random prefix + 32-bit counter
Deterministic (e.g., SIV) Nonce-misuse resistant Use when you cannot guarantee nonce uniqueness

For AES-GCM with 96-bit random nonces: collision probability exceeds $2^{-32}$ (unacceptable) after $2^{32}$ operations. Best practice: rekey after $2^{32}$ nonces.
