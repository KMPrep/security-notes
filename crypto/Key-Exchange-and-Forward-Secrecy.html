Key Exchange: DH, ECDH, and Forward Secrecy

Category: Cryptography
Parent: crypto/README.md

Overview

Key exchange solves a specific problem: two parties who have never met need to establish a shared secret over a channel an attacker can observe. The attacker can read every message in transit. They should never be able to compute the shared secret.

This sounds paradoxical. The math of Diffie-Hellman makes it possible.

Diffie-Hellman (DH)

The Core Problem It Solves

Before DH (published 1976), establishing a shared secret required a pre-existing secure channel — you had to physically hand over a key ahead of time. DH allows two strangers to derive a shared secret over a fully public, compromised network.

How It Works — The Math

DH operates in a finite cyclic group. The commonly described version uses modular exponentiation.

Public parameters (known to everyone, including attackers):

  • p — a large prime number
  • g — a generator (small integer, typically 2 or 5)

The exchange:

Alice                               Bob
─────────────────────────────────────────────────────
Pick secret: a (private)            Pick secret: b (private)
Compute: A = g^a mod p              Compute: B = g^b mod p
          ────────── A ──────────→
          ←────────── B ─────────
Compute: S = B^a mod p              Compute: S = A^b mod p

Both arrive at the same shared secret S because:

$$S = B^a \mod p = (g^b)^a \mod p = g^{ab} \mod p = (g^a)^b \mod p = A^b \mod p$$

What the attacker sees: p, g, A, and B — all public.
What the attacker must solve: Given g, p, A = g^a mod p, find a. This is the discrete logarithm problem — believed to be computationally infeasible for large enough primes.

Concrete Small Example (for illustration only — never use these tiny values)

Public: p = 23, g = 5

Alice picks a = 6              Bob picks b = 15
A = 5^6 mod 23 = 8             B = 5^15 mod 23 = 19

Alice sends A=8   ──────────→  Bob receives 8
Bob sends   B=19  ←──────────  Alice receives 19

Alice: S = 19^6 mod 23 = 2
Bob:   S = 8^15 mod 23 = 2   ✓ same shared secret

An attacker who sees p=23, g=5, A=8, B=19 would need to solve: "what power of 5, mod 23, equals 8?" — the discrete log. For 23 this is trivially brute-forceable; for a 2048-bit prime it is not.

Real-World DH Key Sizes

Security Level DH Prime Size Equivalent ECC Size
80-bit (broken) 1024-bit 160-bit
112-bit (legacy) 2048-bit 224-bit
128-bit (current baseline) 3072-bit 256-bit
256-bit (long-term) 15360-bit 521-bit

This size disparity is the primary reason ECDH is preferred — equivalent security with dramatically smaller keys.

Elliptic Curve Diffie-Hellman (ECDH)

Same Goal, Different Math

ECDH achieves the same key exchange outcome, but replaces the finite-field discrete log with the elliptic curve discrete log problem (ECDLP), which is believed to be harder per bit. This means much smaller keys for equivalent security.

What an Elliptic Curve Is

An elliptic curve over a finite field is the set of integer-coordinate points satisfying:

$$y^2 \equiv x^3 + ax + b \pmod{p}$$

Points on the curve form a group. You can add two points together geometrically (draw a line through them, find the third intersection, reflect over the x-axis). This "point addition" can be repeated — multiplying a point G by a scalar k means adding G to itself k times.

The trap-door: Given G and Q = k * G, find k. This is the elliptic curve discrete log problem — computationally infeasible for well-chosen curves.

The ECDH Exchange

Public parameters: A standard named curve (e.g., P-256, X25519) and a base point G on that curve.

Alice                               Bob
─────────────────────────────────────────────────────
Pick secret: a (private scalar)     Pick secret: b (private scalar)
Compute: A = a * G (point on curve) Compute: B = b * G (point on curve)
            ───── A ─────→
            ←───── B ──────
Compute: S = a * B                  Compute: S = b * A

Both arrive at the same point:

$$S = a \cdot B = a \cdot (b \cdot G) = ab \cdot G = b \cdot (a \cdot G) = b \cdot A$$

The x-coordinate of point S is used as the shared secret (the y-coordinate is discarded).

Common Named Curves

Curve Key Size Used In Notes
P-256 256-bit TLS, ECDSA, FIDO2 NIST standard; widely supported
P-384 384-bit High-assurance TLS, government More conservative margin
X25519 255-bit TLS 1.3, Signal, SSH, WireGuard Designed for high security and implementation safety
X448 448-bit TLS 1.3 (optional), high-value Higher security margin, slower

X25519 is the preferred modern choice — it was designed to resist common implementation pitfalls (timing attacks, invalid curve attacks) that plague NIST curves.

DH vs ECDH — Side-by-Side

Dimension DH (Finite Field) ECDH (Elliptic Curve)
Security basis Discrete logarithm mod prime Elliptic curve discrete log
Key size for 128-bit ~3072-bit prime ~256-bit scalar
Performance Slow for large primes Fast, efficient in software and hardware
Implementation safety Weak groups, small subgroup attacks Curve choice matters; X25519 is safer
Modern protocol use Rare in new designs TLS 1.3, WireGuard, Signal

The MITM Problem: Why Bare DH Is Not Enough

DH and ECDH establish a shared secret — but they do not tell you who you are talking to. Without authentication, a man-in-the-middle can silently intercept both sides.

The Attack

Alice           Mallory (attacker)           Bob
─────────────────────────────────────────────────────
A = g^a mod p  ──→  Mallory intercepts A
                    Mallory computes M1 = g^m1 mod p
                    Sends M1 to Bob ──────────────→
                                              B = g^b mod p
                ←── Bob sends B, Mallory intercepts
                    Mallory computes M2 = g^m2 mod p
                    ←── sends M2 to Alice

Alice computes S1 = M2^a   }  Alice and Mallory share S1
Mallory computes S1 = A^m2 }  Bob and Mallory share S2
                    Mallory computes S2 = B^m1 }
                    Bob computes S2 = M1^b     }

Alice and Bob each think they have a shared secret with each other. They actually each have a separate shared secret with Mallory. Mallory decrypts Alice's traffic, reads it, re-encrypts it for Bob, and vice versa — a completely transparent relay.

How This Is Fixed in Practice

Authentication must be added on top of the key exchange:

Mechanism How It Authenticates Used In
Certificates (PKI) Server signs its DH/ECDH public value with a cert-backed private key TLS (HTTPS)
Pre-shared keys Both sides know a shared secret before exchange; used to verify WireGuard, IKEv2 PSK
Public key pinning Client pins a known server key or cert, rejects others SSH host keys, HPKP
Identity-based trust Separate identity layer (e.g. Signal's safety numbers, OOB verification) Signal, E2E messengers

In TLS:

  1. Server presents a certificate (issued by a trusted CA).
  2. Server signs the ephemeral DH/ECDH parameters with the private key from that certificate.
  3. Client verifies the signature against the certificate chain.
  4. Only then is the key exchange trusted.

Forward Secrecy

What It Means

Forward secrecy (also called perfect forward secrecy, PFS) means that compromise of a long-term secret after the fact should not allow decryption of past traffic.

Without forward secrecy:

2024: Traffic captured and stored by attacker
2026: Server's long-term private key is stolen
      → Attacker decrypts all 2024 traffic retroactively

With forward secrecy:

2024: Traffic captured and stored
2026: Long-term private key stolen
      → Attacker cannot decrypt 2024 traffic
        because session keys came from ephemeral material
        that was never stored and is now gone

How It Works — Ephemeral Key Exchange

The key insight is to derive session keys from fresh ephemeral key exchange material, not directly from the long-term private key.

sequenceDiagram
    participant Client
    participant Server
    Note over Client,Server: Server has long-term key pair (cert + private key)
    Client->>Server: ClientHello (supported ciphers)
    Server->>Client: Certificate (contains server's long-term public key)
    Server->>Client: Ephemeral ECDH public value (e_s), signed with long-term key
    Client->>Server: Ephemeral ECDH public value (e_c)
    Note over Client,Server: Both derive session key from e_c and e_s
    Note over Client,Server: Long-term key was used only to sign, not to derive session key
    Note over Client,Server: e_s and e_c are discarded after handshake
    Client->>Server: Encrypted traffic (protected by session key)

If the server's long-term private key is later stolen:

  • The attacker can impersonate the server going forward.
  • The attacker cannot recompute the session keys for past connections, because e_s (the ephemeral private scalar) was deleted immediately after the handshake.

Ephemeral vs Static DH in TLS

Cipher Suite Type Key Exchange Forward Secrecy
TLS_RSA_WITH_AES_* Static RSA (no DH at all) No
TLS_DHE_RSA_WITH_AES_* Ephemeral DH (DHE) Yes
TLS_ECDHE_RSA_WITH_AES_* Ephemeral ECDH (ECDHE) Yes
TLS 1.3 (all suites) Always ephemeral ECDH/DH Yes (mandatory)

TLS 1.3 removed all non-forward-secret cipher suites. The RSA key exchange (TLS_RSA_*) that was common in TLS 1.2 is gone because it used the server's static RSA key to directly encrypt the session key — meaning stealing the long-term key decrypts all recorded past sessions.

Concrete Forward Secrecy Example — TLS 1.3

Client                                     Server
────────────────────────────────────────────────────────────────────
client_random + key_share(X25519 e_c)  ───→
                                       server_random + key_share(X25519 e_s)
                                       + Certificate + CertificateVerify ←───

Both sides compute:
  shared_secret = ECDH(e_c, e_s)
  handshake_secret = HKDF-Extract(shared_secret, early_secret)
  session_keys    = HKDF-Expand(handshake_secret, ...)

e_c private scalar discarded by client after handshake
e_s private scalar discarded by server after handshake

What the server's long-term private key is used for: only to sign the CertificateVerify message — proving identity. It never touches the session key derivation.

Why Forward Secrecy Matters at Scale

For services handling millions of users, credential compromise is not hypothetical:

  • Keys are rotated during incidents.
  • Devices are lost or seized.
  • Insiders exfiltrate key material.
  • Governments or actors capture network traffic at scale and store it for future decryption ("harvest now, decrypt later").
  • Breaches sometimes go undetected for months — attackers may have recorded significant traffic before discovery.

Without forward secrecy, a single long-term key compromise retroactively exposes all past sessions that key was used to protect.

With forward secrecy, each session's keys are derived from ephemeral material that is destroyed after use. The blast radius of a key compromise shrinks to: sessions happening at the moment of compromise, not years of historical traffic.

flowchart LR
    subgraph Without Forward Secrecy
        A[Long-term key stolen] --> B[All past sessions decryptable]
        B --> C[Years of traffic exposed]
    end
    subgraph With Forward Secrecy
        D[Long-term key stolen] --> E[Attacker can impersonate server going forward]
        D --> F[Past sessions: ephemeral keys gone]
        F --> G[Past traffic NOT decryptable]
    end

Putting It All Together — What a Secure Key Exchange Requires

flowchart TD
    A[Need shared session key] --> B[Key exchange algorithm]
    B -->|Modern| C[ECDH with X25519 or P-256]
    B -->|Legacy| D[DH with 2048-bit+ prime]
    C --> E[Authentication layer]
    D --> E
    E -->|PKI| F[Peer signs ephemeral key with cert-backed private key]
    E -->|PSK| G[Pre-shared secret verified both ways]
    F --> H[Session key derivation via HKDF]
    G --> H
    H --> I[Ephemeral private values destroyed]
    I --> J[Forward secrecy: past traffic safe even if long-term key later stolen]

A complete design needs all three:

  1. Key exchange (DH / ECDH) — derive shared secret without sending it.
  2. Authentication (signatures / certificates / PSK) — verify you are talking to the right party.
  3. Ephemeral material (DHE / ECDHE) — generate fresh key exchange values per session and discard them.

Missing any one of them creates a specific attack surface:

  • No key exchange → key sent directly; passive attacker recovers it.
  • No authentication → MITM performs separate exchanges with both sides.
  • No ephemerality → long-term key compromise decrypts all historical traffic.

What Can Go Wrong

# Failure Root Cause Impact
1 Static RSA key exchange (no DHE) Session key derived from long-term key Past traffic decryptable on key theft
2 Unauthenticated DH No certificate or PSK to verify peer MITM silently relays and reads all traffic
3 Weak DH group (1024-bit or lower) Small prime is within modern compute budget Precomputed discrete log (Logjam attack)

Key Takeaways

  • DH and ECDH let two parties derive a shared secret over a public channel by relying on the computational hardness of discrete logarithm problems.
  • ECDH provides equivalent security to DH with dramatically smaller keys and faster computation.
  • Bare DH or ECDH is not enough — authentication must be layered on top or MITM is trivially possible.
  • Forward secrecy requires using fresh ephemeral key exchange values per session and deleting them after use. The long-term key signs, but never derives, session keys.
  • TLS 1.3 mandates forward secrecy — all cipher suites require ephemeral key exchange.
  • The blast radius of a long-term key compromise is bounded to active sessions, not years of archived traffic, when forward secrecy is enforced.
