Message Authentication Codes — MAC and HMAC

Category: Cryptography — Integrity, Authenticity, Message Authentication

Overview

A Message Authentication Code (MAC) is a short value appended to a message that proves the message has not been tampered with and was produced by someone who knows the secret key. Unlike a hash (which proves integrity but not authenticity), a MAC requires a shared secret, making it an authenticity primitive.

MACs answer: "Did someone who knows our shared key produce this message, and has it been modified since?"

MAC vs Hash vs Signature

Primitive Key Required Proves Integrity Proves Authenticity Non-Repudiation Speed
Hash (SHA-256) No Yes No No Fast
MAC (HMAC) Symmetric key Yes Yes (shared key) No — both parties can generate Very fast
Digital Signature (RSA/ECDSA) Asymmetric private key Yes Yes (only signer can generate) Yes Slow

When to use a MAC vs a signature:

  • MAC: Both parties share a secret (e.g., server-side session tokens, API request signing between known services)
  • Signature: One party signs and anyone can verify (e.g., JWT RS256, software release signing, TLS certificates)

HMAC — Hash-based Message Authentication Code

HMAC is the standard construction for building a MAC from a cryptographic hash function (RFC 2104).

$$\text{HMAC}(K, m) = H\bigl((K' \oplus \text{opad}) \,|\, H((K' \oplus \text{ipad}) \,|\, m)\bigr)$$

Where:

  • $K'$ is the key padded or truncated to the hash block size
  • $\text{ipad}$ = 0x36 repeated to block length
  • $\text{opad}$ = 0x5C repeated to block length
  • $H$ is the hash function (SHA-256, SHA-384, SHA-512)

In practice: HMAC wraps the hash with two keyed rounds that prevent the length extension attack (see below).

Common HMAC Variants

Variant Output Size Recommended Use
HMAC-SHA256 256 bits Current standard; use this
HMAC-SHA384 384 bits Higher security; used in TLS 1.3
HMAC-SHA512 512 bits Maximum HMAC security
HMAC-SHA1 160 bits Legacy; avoid in new systems
HMAC-MD5 128 bits Broken; do not use

Length Extension Attack

A length extension attack exploits a weakness in Merkle–Damgård hash constructions (MD5, SHA-1, SHA-256) where an attacker can append data to a hashed message and compute a valid hash without knowing the secret key, if the MAC is constructed naively as Hash(secret || message).

Why Naive MAC Construction Fails

If MAC = SHA256(key || message), the SHA-256 state after processing key || message is fully determined by the output hash. An attacker can:

  1. Take the known hash value
  2. Initialise SHA-256 from that state
  3. Append attacker-controlled data and compute a valid new hash
  4. The server accepts Hash(key || message || padding || attacker_data) as valid
flowchart LR
    K[key] --> H1[SHA-256 internal state]
    M[message] --> H1
    H1 -->|output = internal state| OUT[known MAC]
    OUT --> H2[SHA-256 continue from state]
    ATK[attacker data] --> H2
    H2 --> FORGED[valid forged MAC]

HMAC Prevents Length Extension

HMAC's double-hash construction means the inner hash output is re-hashed with a different key pad. An attacker cannot use the outer hash output to continue the inner computation. The attack requires knowledge of the inner hash output, which is never revealed.

Affected constructions: H(key || message), H(message || key) — both vulnerable.
Safe constructions: HMAC, BLAKE2/3 (tree-based, not Merkle–Damgård).

Timing Attacks on MAC Verification

MAC comparison must be done in constant time. Standard string comparison (==, strcmp) short-circuits — it returns as soon as it finds a differing byte. This leaks how many bytes of the MAC are correct, enabling an attacker to forge MACs byte by byte with ~256 queries per byte instead of $2^{256}$.

Constant-Time Comparison

# WRONG — timing oracle
if computed_mac == received_mac:
    ...

# CORRECT — constant time
import hmac
if hmac.compare_digest(computed_mac, received_mac):
    ...

hmac.compare_digest (Python), crypto.timingSafeEqual (Node.js), MessageDigest.isEqual (Java) — always use these for MAC comparison.

Authenticated Encryption and AEAD

Modern systems prefer AEAD (Authenticated Encryption with Associated Data) ciphers like AES-GCM and ChaCha20-Poly1305 over separate MAC + encrypt schemes. AEAD provides both encryption and authentication in one primitive, eliminating composition errors.

Encrypt-then-MAC vs MAC-then-Encrypt vs Encrypt-and-MAC

Scheme Construction Security
Encrypt-then-MAC Encrypt first, MAC the ciphertext Secure; prevents padding oracle attacks
MAC-then-Encrypt MAC the plaintext, then encrypt both Vulnerable to padding oracle attacks (TLS pre-1.3)
Encrypt-and-MAC MAC the plaintext, encrypt separately MAC reveals plaintext patterns; generally insecure

Always use Encrypt-then-MAC if building manually. Better: use AEAD and avoid the question entirely.

TLS history: TLS 1.0/1.1/1.2 with CBC used MAC-then-Encrypt, enabling POODLE and BEAST. TLS 1.3 mandates AEAD only (no CBC).

Example — Sending "Transfer $100 to Bob" with AES-CBC + HMAC-SHA256

Encrypt-then-MAC (correct):

plaintext  = "Transfer $100 to Bob"
ciphertext = AES-CBC(key_enc, IV, plaintext)
mac        = HMAC(key_mac, ciphertext)
send       → IV || ciphertext || mac

On receipt: verify MAC first → reject immediately if invalid → only decrypt on success. The server never attempts decryption on tampered data, so a padding oracle has no surface to probe.

MAC-then-Encrypt (TLS pre-1.3 CBC — vulnerable):

plaintext  = "Transfer $100 to Bob"
mac        = HMAC(key_mac, plaintext)
ciphertext = AES-CBC(key_enc, IV, plaintext || mac)
send       → IV || ciphertext

On receipt: decrypt first → check padding → then verify MAC. The padding check happens before the MAC check, leaking a timing/error oracle. An attacker can submit modified ciphertexts and observe whether the server returns a padding error vs. a MAC error to recover plaintext byte by byte (~256 queries per byte). This is the POODLE / BEAST attack surface.

flowchart LR
    CT[modified ciphertext] --> SRV[server decrypts]
    SRV -->|padding invalid| PE[padding error leaked]
    SRV -->|padding valid| MC[MAC check]
    PE --> ATK[attacker learns plaintext byte]
    MC --> ATK

Encrypt-and-MAC (SSH-style — generally insecure):

plaintext  = "Transfer $100 to Bob"
ciphertext = AES-CBC(key_enc, IV, plaintext)
mac        = HMAC(key_mac, plaintext)          ← MAC over plaintext, not ciphertext
send       → IV || ciphertext || mac

The MAC is computed over the plaintext. Two identical messages produce identical MAC tags, leaking that the plaintexts match. More critically, an attacker can modify the ciphertext — the MAC still verifies because it covers the original plaintext, not the ciphertext. No ciphertext integrity is provided.

HMAC in Practice

JWT Signing (HS256)

JSON Web Tokens signed with HS256 use HMAC-SHA256:

HMAC-SHA256(base64url(header) + "." + base64url(payload), secret_key)

Security requirements:

  • The secret key must have sufficient entropy (256+ bits for HS256)
  • The key must not be predictable — a weak secret key enables brute-force forgery
  • JWT alg:none attack: if the server accepts unsigned tokens, the MAC is bypassed entirely
  • Algorithm confusion: if the server accepts both RS256 and HS256, an attacker can take the RSA public key (which is public) and use it as the HS256 HMAC secret

API Request Signing (AWS SigV4)

AWS SigV4 uses HMAC-SHA256 to sign API requests:

StringToSign = Algorithm + \n + RequestDateTime + \n + CredentialScope + \n + HashedCanonicalRequest
Signature = HMAC(SigningKey, StringToSign)

The signing key is derived from the secret access key, date, region, and service via a chain of HMAC operations. This limits key reuse across regions and services.

Cookie Integrity (Flask / Django)

Web frameworks use HMAC to sign session cookies, preventing client-side tampering:

# Flask uses itsdangerous library:
# cookie = base64(data) + "." + HMAC-SHA1(secret_key, base64(data))

If the SECRET_KEY is weak, guessable, or the default ("changeme"), an attacker can forge any session cookie.

Poly1305

Poly1305 is a one-time MAC used in ChaCha20-Poly1305. It is a Wegman–Carter MAC based on polynomial evaluation over a prime field:

$$\text{MAC} = \left(\sum_{i=1}^{n} c_i \cdot r^i\right) \mod (2^{130} - 5) + s$$

Properties:

  • One-time: the r and s values must never be reused with different messages
  • Extremely fast in software (no hardware acceleration needed)
  • Provides 128-bit authentication security
  • The key (r, s) is derived fresh from ChaCha20 for each message using the nonce, so single-use is guaranteed

CMAC and GMAC

MAC Based On Use
CMAC (Cipher-based MAC) AES Used where AES hardware is available but hash hardware is not
GMAC AES-GCM auth component Used when you want authentication without encryption (uncommon)
HMAC Hash function General purpose; most widely used
Poly1305 Polynomial eval Used with ChaCha20; extremely fast
