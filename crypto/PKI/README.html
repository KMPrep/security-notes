Public Key Infrastructure (PKI)

Category: Cryptography — Certificate Management, Trust Systems

Overview

Public Key Infrastructure (PKI) is the set of roles, policies, hardware, software, and procedures needed to create, manage, distribute, use, store, and revoke digital certificates and manage public-key encryption. PKI solves the problem of trust: how do you know that a public key actually belongs to who you think it does?

Without PKI, an attacker can perform a person-in-the-middle attack by substituting their own public key. PKI binds public keys to identities via digital certificates signed by a trusted Certificate Authority (CA).

Digital Certificates

A digital certificate (X.509) contains:

Field Description
Subject The entity the certificate identifies (e.g., CN=example.com, O=Example Inc)
Subject Public Key The public key being certified
Issuer The CA that signed this certificate
Validity Period Not Before / Not After timestamps
Serial Number Unique per CA; used for revocation
Subject Alternative Names (SANs) Additional hostnames or IPs this cert covers
Key Usage / Extended Key Usage Restricts what the cert can be used for (TLS, code signing, email)
Signature CA's digital signature over all the above fields

Certificate chain: An end-entity certificate is signed by an intermediate CA, which is signed by a root CA. Browsers trust root CAs stored in the OS/browser root store.

flowchart TD
    Root[Root CA cert - self-signed, in OS trust store]
    Root --> Int[Intermediate CA cert - signed by Root]
    Int --> EE[End-Entity cert - signed by Intermediate]
    EE --> Server[example.com TLS certificate]

Certificate Authorities

Type Description
Root CA Self-signed; highest trust; kept offline (air-gapped) in HSM
Intermediate CA Signs end-entity certificates on behalf of Root; online
Public CA Commercially operated; trusted by all major OS/browsers (DigiCert, Let's Encrypt, Sectigo)
Private/Internal CA Run by an organisation for internal services; not trusted by public browsers
Cross-signing One CA signs another CA's certificate to extend trust

Root Stores

Root stores are curated lists of trusted root CA certificates maintained by OS/browser vendors:

Vendor Root Store
Mozilla NSS (Network Security Services) — used by Firefox
Microsoft Windows Certificate Store
Apple macOS/iOS Keychain
Google ChromeOS trust store; Android uses Mozilla's store

To be included in a root store, CAs must pass annual audits (WebTrust, ETSI) and adhere to the CA/Browser Forum Baseline Requirements.

DigiNotar Compromise (2011)

DigiNotar was a Dutch CA that was compromised by Iranian attackers. The attackers issued fraudulent certificates for Google, Mozilla, CIA, Mossad, and others. These certs enabled MITM attacks against ~300,000 Iranian Google users. DigiNotar was removed from all root stores within days and went bankrupt shortly after.

Lessons:

  • A compromised CA can attack any domain, not just the CA's own
  • Speed of response matters — revocation must be fast
  • Certificate Transparency (CT) logs allow detection of misissued certificates

Certificate Transparency (CT)

The Problem CT Solves

Any CA in a browser's root store can issue a certificate for any domain. A compromised or rogue CA (like DigiNotar) could issue google.com certificates silently, and no one would know until a MITM attack was discovered. CT makes certificate issuance publicly auditable and tamper-evident.

Architecture

CT is built around three components:

1. CT Logs
Append-only, cryptographically verifiable logs maintained by independent operators (Google, Cloudflare, DigiCert, etc.). Once a certificate is appended, it cannot be removed or altered without detection. They use a Merkle hash tree structure.

2. Certificate Authorities
Before (or just after) issuing a certificate, the CA submits it to one or more CT logs and receives a Signed Certificate Timestamp (SCT) in return — a cryptographic receipt proving the cert was logged.

3. Monitors and Auditors

  • Monitors watch logs for certificates of interest (e.g., any cert issued for yourcompany.com)
  • Auditors verify that SCTs embedded in live certs actually appear in the logs they claim to
sequenceDiagram
    participant CA
    participant CTLog as CT Log
    participant Browser
    participant Monitor

    CA->>CTLog: Submit pre-certificate
    CTLog-->>CA: Signed Certificate Timestamp (SCT)
    CA->>CA: Embed SCT in final certificate
    CA-->>Browser: Certificate with SCT
    Browser->>Browser: Verify SCT signature and timestamp
    CTLog->>Monitor: Streams all new entries
    Monitor->>Monitor: Alert if cert for your domain appears

Merkle Tree Internals

CT logs use a Merkle Hash Tree (binary hash tree). Each certificate is a leaf node. Parent nodes are hashes of their children. The root hash (Signed Tree Head / STH) is signed periodically by the log.

Parent = H(LeftChild || RightChild)

Why this matters:

  • Proving a certificate is in the log requires only O(log n) hashes (a Merkle inclusion proof), not the entire log
  • Proving the log has not been tampered with requires comparing STHs and verifying consistency proofs
  • Any retrospective modification of a log entry changes the root hash, which cannot match the signed STH

Signed Certificate Timestamp (SCT)

An SCT is a promise from the log that it will include the certificate within a Maximum Merge Delay (MMD) — typically 24 hours. It contains:

Field Description
log_id Identifies which CT log
timestamp When the log received it
signature Log's ECDSA signature over the above + cert hash

SCTs can be delivered three ways:

  1. Embedded in the certificate (most common — CA embeds before issuance)
  2. TLS extension (signed_certificate_timestamp during handshake)
  3. OCSP stapling (SCT delivered via stapled OCSP response)

Browser Enforcement

Chrome has required SCTs since April 2018. The policy:

  • Certificates issued by public CAs must have valid SCTs from at least 2 independent CT logs
  • Number of required SCTs scales with certificate lifetime (longer-lived = more SCTs required)
  • A certificate without valid SCTs gets a hard ERR_CERTIFICATE_TRANSPARENCY_REQUIRED error — not a soft warning

Safari (Apple) enforces CT similarly. Firefox delegates to OS cert validation, which varies.

What CT Does and Does Not Provide

CT Provides CT Does NOT Provide
Auditability — all certs are public Revocation — CT doesn't invalidate bad certs
Detection — you can find certs you didn't authorize Prevention — a rogue CA can still issue first, get logged second
Transparency — anyone can search crt.sh Real-time blocking — the MMD window is up to 24 hours
Tamper-evidence — logs are append-only Merkle trees Privacy — certificate SANs are public (internal hostnames leak)

Practical Attack Scenarios

Split-view attack: An attacker who controls a log could show an auditor a different tree than what was shown to a browser. Mitigated by requiring multiple independent logs and by gossip protocols where browsers share STHs they've seen.

CA compromise + CT: Even if a rogue CA issues a cert and logs it, the cert now shows up in public CT feeds. Your domain monitor alerts within minutes. You can then request the cert be added to browser revocation lists. CT doesn't prevent the attack but dramatically reduces the window for undetected abuse.

Internal hostname leakage: If your org uses private CAs for internal services but those CAs are also publicly trusted, certificate SANs (e.g., vpn.internal.corp.com) become visible in CT logs. Mitigation: use a private CA not in public root stores for internal certs.

Useful Tools

Tool Purpose
crt.sh Search CT logs by domain
certspotter Open-source CT monitor for your domains
Google Transparency Report Log health dashboard
openssl x509 -text View SCT extension in a DER-encoded cert

Summary

CT is a detection and accountability mechanism, not a prevention one. It shifts the security model from "trust every CA blindly" to "every issuance is public and auditable." Combined with short-lived certificates and domain monitoring, it significantly reduces the blast radius of a rogue or compromised CA.

Certificate Revocation

Certificates may need to be revoked before expiry if the private key is compromised, the entity changes, or misissued.

Method How It Works Problems
CRL (Certificate Revocation List) CA publishes list of revoked serial numbers; client downloads periodically Large; stale; clients often skip checks
OCSP (Online Certificate Status Protocol) Client queries CA in real time for cert status Privacy leak; performance; OCSP responder availability
OCSP Stapling Server periodically fetches and caches OCSP response; serves it to clients Solves privacy and performance issues; still server-dependent
CRLSets / OneCRL Browser vendors push short lists of revoked high-profile certs Incomplete; best-effort; not a full CRL
Short-lived certs Certs expire in days/hours; revocation becomes irrelevant Used by Let's Encrypt (90-day); being pushed shorter

Revocation is widely considered broken. Most browsers soft-fail OCSP checks (if the OCSP responder is unreachable, the certificate is accepted). This means a compromised certificate from an OCSP-blocked CA is still accepted.

Certificate Types

1. By TLS Validation Level

These differ in how much the CA verifies before issuing.

Domain Validated (DV)

  • CA verifies only that the applicant controls the domain — nothing about who they are
  • Proof via: serve a file at /.well-known/acme-challenge/ (HTTP-01), or add a DNS TXT record (DNS-01)
  • Issued in minutes; fully automated (Let's Encrypt, ZeroSSL)
  • Browser shows a padlock — identical to OV/EV visually since ~2019
  • Attack surface: Anyone who can briefly control a domain (DNS hijack, compromised registrar) can get a DV cert. A typosquat domain like paypa1.com gets the same padlock as paypal.com

Organisation Validated (OV)

  • CA verifies domain control plus confirms the legal existence of the organisation via business registry, phone verification, etc.
  • Takes hours to days; manual vetting involved
  • Organisation name is embedded in the cert's Subject field (O=Example Corp)
  • Visible in cert details but not prominently in browser UI anymore
  • Common for corporate and government sites

Extended Validation (EV)

  • Most rigorous: domain + organisation + legal entity + physical address + authorised representative verified
  • Used to show a green address bar with the company name — removed by Chrome (2019) and Safari (2019); now just a padlock
  • EV certs are still issued and the data is in the cert, but browsers no longer surface it prominently
  • Security value today: Mostly in the cert details and for compliance/audit trails, not user-visible trust

2. By Scope / Coverage

Single-Domain Certificate

  • Covers exactly one FQDN: www.example.com
  • Does not cover example.com or mail.example.com unless explicitly listed as a SAN

Wildcard Certificate (*.example.com)

  • Covers all immediate subdomains: www.example.com, api.example.com, mail.example.com
  • Does not cover the apex (example.com) or nested subdomains (a.b.example.com)
  • One private key shared across all subdomains — if any server is compromised, all subdomains are affected
  • Revocation revokes the cert for all subdomains at once

Multi-SAN / UCC Certificate

  • A single cert with multiple Subject Alternative Names: example.com, example.net, api.example.org
  • Common for CDNs, cloud providers, Microsoft Exchange (Unified Communications Certificate)
  • More efficient than managing dozens of individual certs
  • Revocation takes down all SANs simultaneously

3. By Use Case / Key Usage Extension

The Key Usage and Extended Key Usage (EKU) fields in X.509 restrict what a certificate can be used for. A TLS cert cannot sign code; a code-signing cert cannot authenticate a TLS server.

TLS / SSL Server Certificate

  • EKU: serverAuth (OID 1.3.6.1.5.5.7.3.1)
  • Authenticates the server to the client during TLS handshake
  • Most common cert type; what browsers verify when you visit HTTPS sites

Client Certificate (Mutual TLS / mTLS)

  • EKU: clientAuth (OID 1.3.6.1.5.5.7.3.2)
  • Authenticates the client to the server — used in zero-trust architectures, VPNs, API authentication
  • Subject contains user or device identity
  • Much stronger than password auth — private key never leaves the device (especially in TPM/HSM)
  • Used in: enterprise PKI (smartcards), Kubernetes service mesh (Istio), Google BeyondCorp

Code Signing Certificate

  • EKU: codeSigning (OID 1.3.6.1.5.5.7.3.3)
  • Signs executables, scripts, drivers, packages so the OS/browser can verify authenticity and integrity
  • Windows: Authenticode signatures (.exe, .dll, .ps1)
  • macOS: Apple Developer ID certificates
  • Java: JAR signing
  • Timestamping: Code signatures include a trusted timestamp so they remain valid even after the cert expires
  • EV Code Signing: Required for Windows kernel-mode drivers; bypasses SmartScreen reputation warnings immediately

Email / S/MIME Certificate

  • EKU: emailProtection (OID 1.3.6.1.5.5.7.3.4)
  • Enables two things:
  • Signing emails: recipient can verify the email wasn't tampered with and came from the keyholder
  • Encrypting emails: sender uses recipient's public key to encrypt; only recipient can decrypt
  • Subject includes email address in rfc822Name SAN field
  • Issued by CAs like Sectigo (free personal), or enterprise PKI

Document Signing Certificate

  • EKU: 1.2.840.113583.1.1.5 (Adobe), or standard digitalSignature key usage
  • Signs PDFs and Word documents to prove authenticity and detect post-signature modifications
  • Adobe Acrobat and Microsoft Office verify these
  • Used in legal, healthcare, financial workflows

OCSP Responder Certificate

  • EKU: OCSPSigning (OID 1.3.6.1.5.5.7.3.9)
  • Issued by the CA to its OCSP responder service so it can sign revocation responses on behalf of the CA
  • Has id-pkix-ocsp-nocheck extension so clients don't recursively check the OCSP cert's own revocation status

4. By Position in the Chain

Root CA Certificate

  • Self-signed (Issuer == Subject)
  • BasicConstraints: CA:TRUE; pathLenConstraint may be set
  • Kept offline / air-gapped in HSM
  • Trusted via inclusion in OS/browser root stores
  • If compromised, the entire PKI hierarchy is compromised

Intermediate CA Certificate

  • Signed by the Root CA (or another Intermediate)
  • BasicConstraints: CA:TRUE
  • Online; used to sign end-entity certificates day-to-day
  • Isolates the Root from operational risk — if the Intermediate is compromised, only it needs to be revoked, not the Root
  • Most public CAs use multiple Intermediates for different geographies or certificate types

End-Entity Certificate (Leaf Certificate)

  • BasicConstraints: CA:FALSE — cannot sign other certificates
  • All TLS server, client, code signing, and email certs are end-entity certificates
  • Positioned at the bottom of the chain

5. Special-Purpose Certificate Types

Self-Signed Certificate

  • No CA involvement; entity signs its own certificate
  • Not trusted by browsers/OS by default — must be manually imported into trust stores
  • Used for: internal development, internal microservices, IoT devices, testing
  • Risk: No third-party attestation of identity; MITM undetectable without out-of-band verification (certificate pinning)

Cross-Certificate

  • One Root CA signs another Root CA's certificate to extend trust
  • Used when migrating root stores or building trust bridges between separate PKI hierarchies
  • Let's Encrypt used cross-signing with IdenTrust's DST Root X3 to be trusted by older Android devices that didn't yet have ISRG Root X1

Attribute Certificate (AC)

  • Does not contain a public key — carries authorisation attributes (roles, clearances, group memberships)
  • Linked to a PKC (Public Key Certificate) by the holder's identity
  • Defined in RFC 5755; used in some enterprise and government PKI systems
  • Think of it as a signed "permissions token" separate from the identity cert

Let's Encrypt and ACME

Let's Encrypt is a free, automated, open CA that issues DV certificates. The ACME protocol automates certificate issuance and renewal:

  1. Client proves domain control (HTTP-01: serve a file at /.well-known/acme-challenge/, or DNS-01: create a TXT record)
  2. CA issues 90-day certificate
  3. Client renews automatically (certbot, acme.sh, cloud provider integrations)

Security note: 90-day expiry forces regular renewal and limits the damage window of a compromised key.

HTTP Public Key Pinning (HPKP) — Deprecated

HPKP allowed sites to specify which CA or public key should be trusted for their domain via an HTTP response header. Deprecated by Chrome in 2018 because:

  • Misconfiguration could permanently lock users out of a site
  • Attacker who compromised the site could pin their own key
  • No easy recovery mechanism

Replacement: CT monitoring + short-lived certificates.

HSM — Hardware Security Module

An HSM is a tamper-resistant hardware device that generates, stores, and uses cryptographic keys without ever exposing them in plaintext.

Use Case Why HSM
Root CA private key Never leaves hardware; air-gapped; physical tamper evidence
TLS certificate private key Key operations done in HSM; key never exported
Code signing key Protected signing pipeline
Payment HSM (PCI HSS) PIN encryption; card processing

Certificate chains, CA trust, CT, revocation, and real-world failures

A strong answer should show that you understand both sides:

TLS certificate validation is not only cryptography.
It is also a global trust-management problem.

Browsers trust many root CAs. Any trusted CA, if compromised or careless, can potentially issue a certificate for a domain it does not control. That is why modern controls include Certificate Transparency, CA audits, browser root programs, shorter certificate lifetimes, HSTS, CAA, and strict revocation/incident response processes.

1. What is a certificate chain?

A certificate chain is the sequence of certificates that connects a website’s certificate back to a root CA trusted by the browser or operating system.

Typical chain:

Leaf / end-entity certificate
  Subject: www.example.com
  Public key: website public key
  Signed by: Intermediate CA

        ↓

Intermediate CA certificate
  Subject: Example Intermediate CA
  Public key: intermediate CA public key
  Signed by: Root CA

        ↓

Root CA certificate
  Subject: Trusted Root CA
  Public key: root CA public key
  Self-signed
  Already trusted by browser/OS trust store

Example:

www.bank.com certificate
        signed by
Bank TLS Intermediate CA
        signed by
Global Trusted Root CA
        trusted by browser

The browser does not automatically trust the website certificate. It trusts a root CA in its trust store, then verifies that every certificate in the chain is valid and properly signed.

2. How does a browser verify a certificate chain?

When a browser connects to:

https://www.example.com

the server sends its leaf certificate and usually one or more intermediate certificates.

The browser performs several checks.

A. Chain-building

The browser builds a path:

Leaf certificate → Intermediate CA → Root CA

It checks whether the final root is trusted by the browser or OS trust store.

If the chain cannot be built to a trusted root, validation fails.

B. Signature verification

Each certificate is digitally signed by its issuer.

The browser verifies:

Leaf certificate signature verifies using intermediate CA public key.
Intermediate certificate signature verifies using root CA public key.
Root CA is trusted locally.

This proves that a trusted CA path issued the website certificate.

C. Validity period

The browser checks:

notBefore <= current time <= notAfter

Expired or not-yet-valid certificates fail.

D. Hostname validation

The browser verifies that the requested hostname matches the certificate’s Subject Alternative Name.

Example:

Requested host:
www.example.com

Certificate SAN:
DNS:www.example.com
DNS:example.com

Result:
valid

Invalid example:

Requested host:
www.paypal.com

Certificate SAN:
DNS:paypa1.com

Result:
invalid for paypal.com

The certificate may be valid, but not for that hostname.

E. Key usage and extended key usage

The browser checks that the certificate is allowed for TLS server authentication.

Relevant fields:

Key Usage:
  digitalSignature
  keyEncipherment depending on algorithm/protocol

Extended Key Usage:
  serverAuth

A certificate issued for code signing should not be accepted for HTTPS server authentication.

F. Basic constraints

The browser checks which certificates are allowed to act as CAs.

Important field:

CA:TRUE

A leaf certificate must not be able to issue other certificates.

Intermediate/root CA certificates need appropriate CA constraints.

G. Name constraints, policy constraints, path length

For some CA hierarchies, browsers check constraints such as:

This intermediate can only issue for example.org
This intermediate cannot issue below more than N levels

These are important for private PKI and delegated sub-CAs.

H. Revocation checks

The browser may check whether a certificate has been revoked using:

CRL
OCSP
OCSP stapling
Browser-maintained revocation lists

But, as discussed later, browser revocation checking is historically unreliable.

I. Certificate Transparency

For publicly trusted TLS certificates, browsers such as Chrome require Certificate Transparency evidence, usually in the form of Signed Certificate Timestamps. CT logs are public, append-only ledgers where certificates are recorded and can be monitored. (certificate.transparency.dev)

Browser validation summary

1. Build chain to trusted root.
2. Verify signatures.
3. Check certificate validity period.
4. Check hostname.
5. Check key usage and extended key usage.
6. Check CA/basic constraints.
7. Check revocation where applicable.
8. Check Certificate Transparency requirements.
9. Enforce browser policies such as distrust, weak algorithm bans, HSTS.

Phrasing:

A certificate chain proves delegation of trust from a root CA to an intermediate CA to the website certificate. The browser validates the cryptographic chain and the semantic constraints: hostname, validity, usage, CA constraints, revocation, CT, and browser policy.

3. DigiNotar incident: what happened?

DigiNotar was a Dutch certificate authority that was compromised in 2011. Attackers gained access to its CA infrastructure and fraudulently issued hundreds of certificates, including certificates for major domains such as Google, Mozilla, Skype, and others. Some fraudulent certificates were reportedly used in man-in-the-middle attacks against Iranian Gmail users. Mozilla ultimately removed trust in DigiNotar from Mozilla software, describing it as a complete removal rather than a temporary suspension. (Mozilla Blog)

The most important lesson:

A single compromised trusted CA can undermine TLS for unrelated domains.

Why? Because browsers trust many public roots. If any one of those trusted CAs can issue a certificate for:

*.google.com

then an attacker with network position can potentially impersonate Google to victims unless browsers detect and block it.

Simplified attack:

1. Attacker compromises DigiNotar CA.
2. Attacker issues fraudulent certificate for *.google.com.
3. Attacker positions themselves on victim network or ISP path.
4. Victim visits https://mail.google.com.
5. Attacker presents fraudulent certificate.
6. Browser accepts it because DigiNotar was trusted.
7. Attacker performs TLS man-in-the-middle.

Impact:

- Loss of trust in DigiNotar
- Browser vendors removed trust
- Government and customer impact
- DigiNotar eventually collapsed commercially

4. What controls would have prevented or limited DigiNotar?

No single control is sufficient. A mature design needs prevention, detection, containment, and ecosystem response.

A. Strong CA infrastructure security

Controls:

- Hardware Security Modules for CA private keys
- Strong network segmentation
- Offline root CA
- Tightly controlled intermediate CAs
- Multi-party approval for certificate issuance
- Strong admin MFA
- Privileged access management
- Tamper-evident logging
- Continuous monitoring
- Regular independent audits

If CA signing keys or issuance systems are weakly protected, the entire trust model fails.

B. Certificate Transparency

CT would not necessarily prevent the initial issuance, but it helps detect misissued certificates quickly because certificates must be logged in public append-only logs. Domain owners and monitors can watch logs for certificates issued for their domains. (certificate.transparency.dev)

Detection example:

Monitor sees:
  Certificate for *.google.com
  Issuer: DigiNotar

Google did not request this.
Alert triggers.
Browser vendors respond.

C. CA Authorization / CAA DNS records

CAA allows a domain owner to specify which CAs are authorized to issue certificates for the domain.

Example:

example.com. CAA 0 issue "letsencrypt.org"
example.com. CAA 0 issuewild ";"

This tells CAs:

Only Let's Encrypt may issue for example.com.
No CA may issue wildcard certificates.

CAA is not a browser-side enforcement mechanism. It is a CA-side issuance control. A malicious or compromised CA can ignore it, but compliant CAs must check it.

D. Browser root program enforcement

Browser vendors can:

- Remove trust in bad CAs
- Partially distrust CAs
- Require incident reports
- Require audits
- Ban weak algorithms
- Enforce CT
- Enforce certificate lifetime limits

Mozilla’s full removal of DigiNotar trust is an example of ecosystem-level containment. (Mozilla Blog)

E. HSTS and preloading

HSTS does not stop a fraudulent valid certificate, but it prevents SSL-stripping and accidental HTTP downgrade.

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Useful against:

http://bank.com → attacker prevents HTTPS redirect

Not sufficient against:

attacker has valid certificate for bank.com

F. Certificate pinning — carefully

Historically, static pinning in browsers or applications could detect unexpected certificate chains.

But web-scale HTTP Public Key Pinning was dangerous and deprecated because misconfiguration or hostile pinning could create denial-of-service risks. Chrome’s deprecation notice explicitly cited low adoption and risks of denial of service and hostile pinning. (Chrome for Developers)

Modern approach:

- Use CT monitoring
- Use CAA
- Use HSTS
- Use short-lived certificates
- Use app-level pinning only in controlled mobile/native apps where recovery is designed

5. What is Certificate Transparency?

Certificate Transparency is a public logging system for TLS certificates.

Core idea:

Every publicly trusted certificate should be visible.

CT uses public, append-only logs. Certificate authorities submit certificates or precertificates to logs. The logs return Signed Certificate Timestamps, or SCTs, which prove the certificate has been submitted for logging. Browsers can require certificates to carry enough valid SCTs before trusting them. CT logs are append-only and publicly auditable. (certificate.transparency.dev)

How CT works

1. CA prepares certificate or precertificate.
2. CA submits it to CT logs.
3. CT log returns SCT.
4. Certificate or TLS handshake includes SCT.
5. Browser verifies SCT.
6. Domain owners monitor logs.
7. Misissued certificates can be detected.

Visual:

CA ── submits certificate ──> CT Log
CA <── SCT ───────────────── CT Log

CA issues certificate with SCT

Browser ── connects to site ──> Server
Browser receives certificate + SCT
Browser validates certificate chain and CT evidence

Domain owner monitors CT logs for unexpected certificates

6. How does CT help detect misissued certificates?

Without CT:

A CA could issue a certificate for your domain.
You might never know unless you observe it in the wild.

With CT:

The certificate appears in public logs.
You can monitor logs for your domains.
Unexpected issuance triggers alert.

Example:

Domain owner: paypal.com

CT monitor detects:
  Certificate for:
    www.paypal.com
  Issuer:
    Unknown CA
  Requested by:
    Not PayPal

Response:
  Investigate
  Contact CA
  Revoke certificate
  Notify browsers if needed

Important nuance:

CT improves detection and accountability. It does not by itself stop a compromised CA from issuing a certificate. Browser enforcement of SCTs makes silent misissuance much harder.

7. Why is OCSP revocation checking considered unreliable?

OCSP lets a browser ask the CA:

Is this certificate still valid, or has it been revoked?

But in practice, browser revocation checking has several problems.

A. Soft-fail behavior

If the OCSP responder is unavailable, slow, blocked, or unreachable, many browsers historically continue the connection.

Why?

Hard-fail would break many websites whenever OCSP infrastructure has issues.

Attack implication:

An attacker with a revoked certificate may block OCSP traffic.
Browser cannot check revocation.
Browser soft-fails and continues.

B. Privacy leakage

Traditional OCSP can reveal browsing behavior to the CA.

Browser asks CA:
  Is certificate for sensitive-site.example valid?

CA learns:
  User may be visiting sensitive-site.example.

C. Performance and reliability

OCSP adds network latency and depends on CA infrastructure availability.

Problems:

- Slow OCSP responders
- Network failures
- Captive portals
- Enterprise proxies
- Firewalls blocking OCSP

D. Operational inconsistency

Different browsers and platforms handle revocation differently.

Some use:

- OCSP
- CRLs
- CRLSets
- OneCRL
- platform APIs
- short-lived certificates

This means revocation behavior is not always uniform.

8. Alternatives to traditional OCSP

A. OCSP stapling

The server fetches OCSP response from the CA and “staples” it into the TLS handshake.

Benefits:

- Better privacy
- Better performance
- Browser does not need to contact CA directly

Weakness:

- If not mandatory, attacker/server may omit staple.

B. OCSP Must-Staple

Certificate includes an extension requiring a valid stapled OCSP response.

Benefit:

Browser should fail if staple is missing.

Weakness:

Operational risk: if stapling breaks, site can become unavailable.

C. CRLs and browser-curated revocation lists

Browsers may distribute compact revocation lists.

Examples conceptually:

- Browser vendor-maintained emergency revocation list
- OneCRL-like mechanisms
- CRLSet-like mechanisms

Benefits:

- Fast emergency blocking
- No per-site privacy leak

Weakness:

- Usually not complete for all revoked certificates

D. Short-lived certificates

If certificates are valid only for a short time, revocation becomes less critical.

Example:

Certificate lifetime: days instead of months

Benefits:

- Smaller exposure window
- Less dependency on revocation infrastructure

Weakness:

- Requires reliable automation
- Not instant revocation

E. Automated certificate management

Use ACME-style automation to rotate certificates frequently and safely.

This does not replace revocation entirely, but it reduces operational failures and supports shorter certificate lifetimes.

9. DV, OV, and EV certificates

DV — Domain Validation

The CA validates control over the domain.

Example checks:

- DNS challenge
- HTTP challenge
- Email to domain contact

DV proves:

The requester controls the domain.

DV does not prove:

The requester is a legitimate business.
The site is trustworthy.
The domain is not deceptive.

OV — Organization Validation

The CA validates domain control plus some organization identity information.

OV proves more about the organization than DV, but browsers generally do not show strong user-facing indicators for OV.

EV — Extended Validation

EV requires stricter organization identity verification under CA/Browser Forum EV Guidelines. EV certificates are issued only after the entity passes a stricter authentication procedure. (CA/Browser Forum)

Historically, browsers displayed EV with a company name in the address bar.

Example old browser UI:

[PayPal, Inc.] https://www.paypal.com

Modern browsers largely reduced or removed prominent EV UI because users did not reliably understand or use it for security decisions.

Does EV provide meaningful security?

Senior answer:

EV provides stronger identity vetting of the certificate subscriber, but it does not provide strong user-facing phishing protection in practice.

Why?

- Users often do not notice EV indicators.
- Browsers reduced EV UI.
- Attackers can register lookalike legal names.
- EV does not stop phishing on different domains.
- EV does not improve TLS encryption strength.
- EV does not prevent malware, XSS, or credential theft.

Example:

paypal.com with EV:
  Stronger organization identity binding.

paypa1.com with DV:
  Still gets a valid HTTPS lock.
  Many users may not notice the difference.

So EV is useful for some identity/compliance contexts, but not a reliable anti-phishing control by itself.

10. Why was HPKP deprecated? What replaced it?

HPKP, HTTP Public Key Pinning, allowed a website to tell browsers:

For this domain, only accept certificates chaining to these public keys for the next N days.

Example header:

Public-Key-Pins:
  pin-sha256="base64==";
  pin-sha256="backupbase64==";
  max-age=5184000;
  includeSubDomains

The goal was to protect against CA misissuance.

But HPKP was dangerous.

Problems

- Misconfiguration could brick a site for users.
- If the pinned key was lost, users could not connect.
- Hostile pinning could be used as a denial-of-service attack.
- Operational complexity was high.
- Adoption was low.

Chrome deprecated HPKP, citing low adoption and the risks of denial of service and hostile pinning. (Chrome for Developers)

What replaced HPKP?

No single feature fully replaced HPKP. Instead, the ecosystem moved to layered controls:

- Certificate Transparency
- CT monitoring
- CAA DNS records
- HSTS preload
- Shorter certificate lifetimes
- Better CA/B Forum baseline requirements
- Browser root program enforcement
- Safer application-level pinning in controlled native/mobile apps

Senior phrasing:

HPKP tried to push CA-misissuance defense to website operators through a brittle browser cache mechanism. It was replaced in practice by CT-based ecosystem monitoring, CAA issuance constraints, HSTS, shorter lifetimes, and stronger browser root governance.

11. Attacker obtains a valid DV certificate for paypa1.com

This is important.

The certificate is not misissued.

paypa1.com

is a different domain from:

paypal.com

A CA can correctly issue a DV certificate for paypa1.com if the attacker controls that domain.

What attack does this enable?

It enables a user-facing phishing attack with a valid HTTPS lock.

Attack:

1. Attacker registers paypa1.com.
2. Attacker gets valid DV certificate for paypa1.com.
3. Attacker builds a PayPal lookalike page.
4. Victim sees HTTPS lock.
5. Victim thinks the site is safe.
6. Victim enters credentials or payment details.

User-facing problem:

The browser lock means encrypted connection to paypa1.com.
It does not mean paypa1.com is PayPal.

Senior phrasing:

HTTPS authenticates the domain you reached. It does not tell users that the domain is the brand they intended.

Defenses

A. User education is not enough

Users are bad at detecting lookalike domains.

Do not rely primarily on:

“Check the address bar carefully.”

B. Brand monitoring and takedown

- Monitor domain registrations
- Monitor CT logs for lookalike names
- Detect homographs and typosquats
- Takedown phishing infrastructure

C. FIDO2/WebAuthn/passkeys

This is one of the strongest defenses.

If PayPal uses WebAuthn/passkeys for login, a passkey registered for:

paypal.com

will not authenticate to:

paypa1.com

The phishing site cannot get a valid WebAuthn assertion for the real domain.

D. Password managers

Password managers help because they only autofill credentials on the correct domain.

Stored login:
paypal.com

Current site:
paypa1.com

Result:
No autofill

This is not perfect, but it is useful.

E. HSTS preload

Protects real paypal.com from SSL stripping, not from paypa1.com.

Useful but not sufficient for lookalike domains.

F. Email authentication

For phishing delivery:

- SPF
- DKIM
- DMARC with reject policy
- BIMI where appropriate

These reduce spoofed brand email but do not stop all lookalike-domain phishing.

G. Browser safe browsing and reputation

Browsers and security products can block known phishing domains.

H. Defensive domain registration

Register obvious typo domains where appropriate.

But this does not scale to all variants.

I. Transaction risk controls

Even if credentials are phished:

- Step-up authentication
- FIDO2 for sensitive actions
- Device binding
- Anomaly detection
- New payee delays
- Withdrawal velocity limits

12. When would you use a private/internal CA instead of a public CA?

Use a private CA when the identity you need to certify is internal, not public web identity.

Examples:

- Internal service-to-service mTLS
- Kubernetes workload identities
- SPIFFE/SPIRE identities
- Internal APIs
- Internal databases
- Corporate Wi-Fi certificates
- VPN/device certificates
- Enterprise user certificates
- Code signing inside enterprise
- Lab/dev/test environments
- IoT device identity

Why not use public CA for everything?

Public CAs are designed for public DNS names and browser trust.

They are not ideal for:

- Internal-only names
- Short-lived workload certs
- Non-public identities
- Machine identity at large scale
- Custom issuance policies
- Internal revocation and lifecycle control

Private CA advantages

- You control issuance policy
- You control certificate lifetime
- You can issue for internal names
- You can integrate with IAM/MDM/Kubernetes
- You can enforce workload identity
- You can revoke/rotate internally
- You avoid exposing internal hostnames to public CT logs

Important CT point:

Public certificates generally appear in CT logs. You may not want internal service names visible publicly.

Bad idea:

internal-payments-prod-01.corp.example.com

appearing in public CT logs.

Private CA risks

Private CA becomes a sensitive trust root.

You must protect it seriously:

- Offline root CA
- Online intermediate CAs
- HSM-backed keys
- Strict issuance controls
- Short-lived certs
- Automated rotation
- Revocation strategy
- Audit logs
- Least-privilege CA operators
- Separate roots/intermediates by environment

Example enterprise design:

Offline Root CA
   ├── Workforce Device Intermediate CA
   ├── Kubernetes Workload Intermediate CA
   ├── Database mTLS Intermediate CA
   └── IoT Device Intermediate CA

Do not use the same intermediate for everything. Compromise blast radius matters.

Summary

A certificate chain is a delegation path from a website certificate through one or more intermediate CAs to a trusted root. Browsers verify signatures, hostname, validity period, key usage, CA constraints, revocation signals, Certificate Transparency evidence, and browser policy. DigiNotar showed that compromise of one trusted CA can break TLS for unrelated domains; controls include stronger CA security, CT, CAA, HSTS, short-lived certificates, browser root enforcement, and rapid incident response. Certificate Transparency makes certificate issuance publicly auditable and helps domain owners detect misissuance, though it is mainly detection and accountability rather than pure prevention. OCSP is unreliable because of soft-fail behavior, privacy leakage, latency, and inconsistent browser handling; alternatives include OCSP stapling, must-staple, browser revocation lists, and shorter certificate lifetimes. DV proves domain control, OV and EV add organization validation, but EV does not provide strong practical anti-phishing protection. HPKP was deprecated because misconfiguration and hostile pinning could cause denial of service; CT, CAA, HSTS, shorter lifetimes, and browser root governance replaced it in practice. A valid DV certificate for paypa1.com enables convincing HTTPS phishing, not TLS misissuance; the best defenses include passkeys/WebAuthn, password managers, phishing detection, brand monitoring, DMARC, and transaction risk controls. For internal systems, use private CAs when you need internal machine/workload/device identity and lifecycle control, but protect the private CA like critical security infrastructure.
