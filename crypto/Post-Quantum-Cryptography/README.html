Post-Quantum Cryptography

Category: Cryptography / Applied Crypto Engineering
Key standards / frameworks: NIST FIPS 203 (ML-KEM), FIPS 204 (ML-DSA), FIPS 205 (SLH-DSA), NIST SP 800-208 (stateful hash-based signatures), NSA CNSA 2.0, RFC 9420-adjacent TLS 1.3 hybrid drafts (X25519MLKEM768), CNSSP 15

Overview

Post-quantum cryptography (PQC) is the family of public-key algorithms believed to resist attack by a large-scale fault-tolerant quantum computer. The threat is concrete and asymmetric: a cryptographically relevant quantum computer running Shor's algorithm factors RSA moduli and solves the elliptic-curve discrete log problem in polynomial time, which breaks RSA, DSA, Diffie-Hellman, ECDSA, and ECDH outright. Grover's algorithm gives only a quadratic speedup against unstructured search, so symmetric primitives are weakened rather than broken and are remediated simply by doubling key length. AES-256 and SHA-384 remain acceptable at the quantum-128-bit security level.

The operationally urgent dimension is harvest-now-decrypt-later. An adversary who records encrypted traffic today can archive it and decrypt it once a quantum computer becomes available. Any secret that must stay confidential for ten or twenty years is therefore already exposed even though no quantum computer exists yet. This shifts PQC from a future problem to a present migration deadline.

In August 2024 NIST finalized the first three PQC standards. ML-KEM (FIPS 203, derived from CRYSTALS-Kyber) is a lattice-based key encapsulation mechanism. ML-DSA (FIPS 204, from CRYSTALS-Dilithium) and SLH-DSA (FIPS 205, from SPHINCS+) are digital signature schemes. These are built on Module-LWE, Module-SIS, and hash-based constructions rather than integer factorization.

How It Works

The Quantum Threat Model

Shor's algorithm reduces factoring and discrete log to period-finding, which a quantum computer solves efficiently using the quantum Fourier transform. This is an exponential speedup and it is fatal for classical public-key crypto. Grover's algorithm only reduces a brute-force key search from 2^n to roughly 2^(n/2), which is why symmetric strength is preserved by choosing AES-256 instead of AES-128. The practical rule of thumb: double symmetric key sizes and hash output lengths, but replace asymmetric algorithms entirely.

ML-KEM Key Encapsulation

ML-KEM is a KEM, not a key-agreement protocol in the Diffie-Hellman sense. The recipient publishes a public key; the sender runs Encapsulate to produce a ciphertext plus a shared secret; the recipient runs Decapsulate on the ciphertext to recover the same shared secret. Security rests on the hardness of Module Learning With Errors. Parameter sets ML-KEM-512, 768, and 1024 target NIST security categories 1, 3, and 5. The Fujisaki-Okamoto transform provides IND-CCA2 security.

Hybrid Key Exchange and Crypto-Agility

Because lattice assumptions are younger than RSA, deployments combine a classical and a post-quantum KEM so the session stays secure if either one holds. The dominant TLS 1.3 construction is X25519MLKEM768, which concatenates the X25519 ECDH output with the ML-KEM-768 shared secret and feeds both into the key-derivation function. The same hybrid pattern is landing in OpenSSH via sntrup761x25519 and now mlkem768x25519. Crypto-agility is the design property that lets a system swap algorithms, key sizes, and negotiated parameters without re-architecting the protocol, achieved through algorithm identifiers, versioned negotiation, and abstracted key stores.

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 Harvest-now-decrypt-later on archived traffic Long-lived secrets protected only by RSA or ECDH Retroactive decryption once Shor-capable hardware exists High for high-value data
2 Naive rip-and-replace instead of hybrid Trusting immature lattice assumptions alone Total loss of confidentiality if ML-KEM is later broken Medium
3 Side-channel leakage in ML-KEM decapsulation Non-constant-time rejection sampling or NTT Secret-key recovery via timing or power analysis Medium
4 SLH-DSA misuse as stateful scheme Confusing stateless SPHINCS+ with LMS or XMSS one-time keys Catastrophic signature forgery from key reuse Low but severe
5 Handshake fragmentation and MTU failures Large ML-KEM keys exceeding packet and buffer limits Broken connections and middlebox drops Medium
6 Frozen crypto with no agility Hardcoded algorithm identifiers and key sizes Cannot patch a broken primitive without a full rebuild High in legacy systems
7 Downgrade to classical-only negotiation Attacker strips PQC ciphersuites during handshake Silent loss of quantum resistance Medium

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Harvest-now-decrypt-later Deploy hybrid KEM in TLS 1.3 and SSH for all long-lived channels first Transport layer
2 Immature-assumption risk Always combine X25519 with ML-KEM-768 rather than PQC alone Key-exchange design
3 Side-channel leakage Use constant-time reference implementations and validated FIPS modules Library and hardware
4 Stateful signature misuse Prefer stateless SLH-DSA or enforce strict state tracking for XMSS and LMS per SP 800-208 Signature subsystem
5 Handshake size growth Tune record sizes and test middleboxes for larger keys and signatures Network and protocol
6 No crypto-agility Abstract algorithms behind identifiers and a central crypto inventory Architecture and governance
7 Downgrade attacks Authenticate negotiated parameters in the transcript and disable classical-only fallback Protocol negotiation

Key Takeaways

  • Shor's algorithm breaks RSA, DSA, ECDSA, and ECDH; Grover only halves symmetric strength, so double symmetric keys and hashes but replace all asymmetric crypto.
  • Harvest-now-decrypt-later makes PQC migration urgent today for any secret with a multi-year confidentiality lifetime.
  • The finalized NIST standards are ML-KEM in FIPS 203 for key encapsulation, ML-DSA in FIPS 204 and SLH-DSA in FIPS 205 for signatures.
  • Hybrid key exchange such as X25519MLKEM768 stays secure if either the classical or the post-quantum component holds, hedging against young lattice assumptions.
  • PQC costs larger keys, signatures, and handshakes, driving MTU, buffer, and latency tradeoffs that must be tested in production paths.
  • Crypto-agility, achieved through algorithm identifiers and abstracted key stores, lets you swap primitives without re-architecting protocols.
  • A sound migration begins with a full cryptographic inventory, prioritizes long-lived secrets, and rolls out hybrids in TLS 1.3 and SSH before retiring classical-only modes.
