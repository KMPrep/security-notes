Protocol Attacks — POODLE, BEAST, CRIME, BREACH, HEARTBLEED

Category: Cryptography — TLS/SSL Attack History

Overview

These five attacks represent landmark vulnerabilities in the TLS/SSL protocol stack. Each exploited a different weakness — padding, compression, version negotiation, or implementation bugs — and each drove significant changes to how TLS is deployed. Understanding them demonstrates both technical depth in cryptography and the real-world consequences of protocol design flaws.

BEAST (2011) — Browser Exploit Against SSL/TLS

CVE: Not assigned as a single CVE; affects TLS 1.0
Target: TLS 1.0 using CBC mode (e.g., HTTPS with AES-CBC)
Attack type: Chosen-plaintext attack against CBC mode's IV chaining

How It Works

In TLS 1.0, CBC mode uses the last ciphertext block of one record as the IV for the next record. This is predictable. An attacker with a MITM position and the ability to inject JavaScript into the victim's browser can:

  1. Observe ciphertext blocks sent to a target site (e.g., HTTPS session cookie)
  2. Craft requests to move the known IV into a position where they can make predictions about the unknown secret cookie block
  3. Iteratively recover the secret byte by byte

Root cause: Predictable IV in CBC mode across records violates the IND-CPA security requirement.

Fix

  • Upgrade to TLS 1.1+ (uses random IV per record)
  • Use RC4 (mitigated BEAST temporarily; RC4 later broken completely)
  • TLS 1.3 removed CBC cipher suites entirely

CRIME (2012) — Compression Ratio Info-leak Made Easy

CVE: CVE-2012-4929
Target: TLS with HTTP compression (DEFLATE)
Attack type: Compression oracle

How It Works

When HTTP request compression (gzip/DEFLATE) is enabled in TLS, the compression algorithm compresses the request including the secret cookie before encryption. Compression works by replacing repeated strings with back-references. Therefore:

  • If the attacker's injected guess string matches part of the cookie, the compressed (and thus encrypted) payload will be shorter
  • The attacker iterates through guesses and observes ciphertext length to learn the cookie value
Request 1: Cookie: session=A [guessed] → compressed length 100 bytes
Request 2: Cookie: session=s [actual match] → compressed length 97 bytes (shorter!)

Root cause: Mixing secret data (cookies) with attacker-controlled data (injected string) under compression before encryption leaks information about the secret.

Fix

  • Disable TLS compression (now disabled by default in all modern TLS implementations)
  • Modern TLS 1.3 has no compression

BREACH (2013) — Browser Reconnaissance and Exfiltration via Adaptive Compression of Hypertext

CVE: CVE-2013-3587
Target: HTTP response compression (gzip) — distinct from CRIME (which targeted request compression)
Attack type: Compression oracle on HTTP responses

How BREACH Differs from CRIME

CRIME BREACH
Compressed data HTTP request HTTP response
Fix available Yes (disable TLS compression) Harder — disabling HTTP gzip hurts performance
Status Fixed Still a risk; mitigations required

Mitigations for BREACH

  • Disable HTTP compression for responses containing secrets (session tokens, CSRF tokens)
  • Separate secrets from user-controlled content in responses
  • Use per-request random CSRF token padding to defeat length oracle
  • Length-hide: pad responses to fixed lengths

POODLE (2014) — Padding Oracle On Downgraded Legacy Encryption

CVE: CVE-2014-3566
Target: SSLv3 using CBC mode padding
Attack type: Padding oracle + protocol downgrade

How It Works

SSLv3 CBC padding verification has a weakness: the server checks only the last byte of the padding, not the entire padding block. This makes it a padding oracle — the server's different behavior for valid vs invalid padding reveals information about the plaintext.

The attack also requires a protocol downgrade: the attacker forces the browser to fall back to SSLv3 by making TLS handshakes fail. Before TLS Fallback SCSV (Signalling Cipher Suite Value) was widely deployed, browsers would retry with an older protocol version if the handshake failed.

Attack flow:

  1. Attacker intercepts HTTPS connection and forces SSLv3 downgrade
  2. Attacker can make the victim's browser send arbitrary requests (via JavaScript)
  3. Attacker submits manipulated ciphertext blocks
  4. Server's padding error vs success reveals one byte of plaintext per attempt
  5. After ~256 attempts per byte, attacker recovers the session cookie

Fix

  • Disable SSLv3 entirely (it has no secure use case)
  • Deploy TLS_FALLBACK_SCSV to prevent downgrade
  • TLS 1.3 has no downgrade path to SSLv3/TLS 1.0/1.1

POODLE for TLS (2014)

A related finding showed that some TLS implementations (F5, A10 load balancers) had the same padding validation bug in their TLS CBC implementations — not limited to SSLv3. This was sometimes called "POODLE for TLS."

HEARTBLEED (2014) — OpenSSL TLS Heartbeat Extension Buffer Over-Read

CVE: CVE-2014-0160
Target: OpenSSL 1.0.1 through 1.0.1f
Attack type: Memory over-read (implementation bug, not protocol flaw)

How It Works

The TLS Heartbeat extension (RFC 6520) allows one side to send a heartbeat request: "Here is a message of length N, please send it back." The receiving side should echo back exactly N bytes of the message. In the vulnerable OpenSSL implementation:

  1. Attacker sends: "Here is a 1-byte message, but tell me it's 64KB long"
  2. OpenSSL trusts the claimed length parameter without validating it
  3. OpenSSL reads 64KB from process memory starting at the message and sends it back
  4. This 64KB can contain: private keys, session keys, plaintext passwords, and other secrets from adjacent memory
Attacker request: [1 byte of data] claimed length = 65535
Server response:  [1 byte data] + [65534 bytes of raw server memory]

What was leaked: Private keys, session tokens, passwords, any data that happened to be in OpenSSL process memory. Exploiting it repeatedly could dump large sections of process memory over time.

Impact and Scale

Heartbleed was one of the most serious vulnerabilities ever found. It affected:

  • ~17% of all HTTPS websites at disclosure
  • Apache and Nginx web servers on Ubuntu, Debian, many Linux distributions
  • OpenVPN, stunnel, and other TLS-using daemons
  • Network appliances (Cisco, Juniper, F5)

Because the attack left no log trace, there was no way to determine post-hoc whether any specific server had been exploited.

Fix

  1. Upgrade OpenSSL to 1.0.1g or later
  2. Revoke and reissue all TLS certificates — private keys may have been leaked
  3. Rotate all session secrets, API keys, and passwords
  4. Revoke active sessions

Why This Is Not a TLS Protocol Flaw

Unlike BEAST/CRIME/BREACH/POODLE, Heartbleed was an implementation bug in OpenSSL — missing bounds checking in the heartbeat handler. The TLS protocol spec was not the problem; the code was.

Code Simulations (Educational)

These simulations use a toy XOR block cipher (same CBC structural properties as AES, no external dependencies). They demonstrate the core vulnerability in each attack — not production-ready exploit code.

BEAST — Predictable IV enables block matching

# BEAST: TLS 1.0 reuses the last ciphertext block as the next IV.
# An attacker who can observe ciphertexts AND inject chosen plaintexts
# (via malicious JS in the victim's browser) can recover secret bytes.

import os

BLOCK = 8  # 8-byte blocks; real TLS uses AES-128 (16 bytes)

def xor(a, b):
    return bytes(x ^ y for x, y in zip(a, b))

def encrypt_block(key, data):
    """Toy block cipher — XOR with key. Real attacks use AES."""
    return xor(key, data)

def cbc_encrypt(key, iv, plaintext_block):
    """CBC: C = E_k(P XOR IV). Returns ciphertext block."""
    return encrypt_block(key, xor(plaintext_block, iv))

# ── Victim setup ──────────────────────────────────────────────────────────────
key    = os.urandom(BLOCK)
secret = ord('Z')                             # the one unknown byte to recover
target = bytes(BLOCK - 1) + bytes([secret])  # 7 known zeros + secret byte

IV_1     = os.urandom(BLOCK)
C_target = cbc_encrypt(key, IV_1, target)

# TLS 1.0 FLAW: next record's IV = C_target (visible to attacker on the wire!)
IV_2 = C_target

# ── BEAST attack ──────────────────────────────────────────────────────────────
# We want:  cbc_encrypt(key, IV_2, crafted) == C_target
# This means:  crafted XOR IV_2  ==  target XOR IV_1
# So:  crafted[i] = target[i] XOR IV_1[i] XOR IV_2[i]
#   For i < BLOCK-1, target[i]=0, so crafted[i] = IV_1[i] XOR IV_2[i]
#   For i = BLOCK-1: we guess g. Match when g XOR IV_2[-1] == secret XOR IV_1[-1]

for g in range(256):
    crafted = bytearray(xor(IV_1, IV_2))  # cancel known prefix
    crafted[-1] = g                        # replace last byte with guess

    if cbc_encrypt(key, IV_2, bytes(crafted)) == C_target:
        recovered = g ^ IV_2[-1] ^ IV_1[-1]
        print(f"BEAST recovered: byte={recovered}  char='{chr(recovered)}'")
        assert recovered == secret
        break

# ── Why TLS 1.1+ is immune ────────────────────────────────────────────────────
# IV is fresh random per record — attacker cannot see it before encryption,
# so they cannot craft a block that XORs to a known target. Block matching fails.

Key insight: With predictable IV (IV_2 = C_target, visible on the wire), the attacker crafts a block that produces the same ciphertext as the secret block. TLS 1.1+ breaks this by using a fresh random IV per record.

POODLE — Weak padding check is a decryption oracle

# POODLE: SSLv3 checks ONLY the last padding byte. An attacker who can
# modify ciphertext blocks and observe the server's accept/reject response
# can brute-force secret bytes one at a time (~256 requests per byte).

import os

BLOCK = 8

def xor(a, b):
    return bytes(x ^ y for x, y in zip(a, b))

def encrypt_block(key, data): return xor(key, data)
def decrypt_block(key, data): return xor(key, data)  # XOR is its own inverse

# ── Server-side padding checkers ──────────────────────────────────────────────
def ssv3_padding_valid(decrypted):
    """SSLv3 flaw: only reads the last byte as padding length. Never checks
    whether intermediate padding bytes actually match that value."""
    return decrypted[-1] == 0x00  # accepts any block whose last byte == 0

def tls1x_padding_valid(decrypted):
    """TLS 1.x fix: ALL padding bytes must equal the padding length value."""
    pad_len = decrypted[-1]
    if pad_len >= BLOCK:
        return False
    return all(b == pad_len for b in decrypted[-(pad_len + 1):])

def oracle(key, C_prev_modified, C_target, strict=False):
    decrypted = xor(decrypt_block(key, C_target), C_prev_modified)
    return tls1x_padding_valid(decrypted) if strict else ssv3_padding_valid(decrypted)

# ── Victim setup ──────────────────────────────────────────────────────────────
key       = os.urandom(BLOCK)
secret    = ord('P')
plaintext = bytes(BLOCK - 1) + bytes([secret])  # secret in last byte position
C_prev    = os.urandom(BLOCK)
C_target  = encrypt_block(key, xor(plaintext, C_prev))  # CBC encryption

# ── POODLE attack ─────────────────────────────────────────────────────────────
# Attacker modifies only the last byte of C_prev and watches for oracle acceptance.
#
# decrypted[-1] = D_k(C_target)[-1] XOR C_modified[-1]
#               = (plaintext[-1] XOR C_prev[-1]) XOR g
# Oracle says True when this equals 0x00:
#   secret XOR C_prev[-1] XOR g == 0  →  g = secret XOR C_prev[-1]
# Recovery: secret = g XOR C_prev[-1]

print("=== POODLE (SSLv3 — last-byte oracle) ===")
for g in range(256):
    C_modified = C_prev[:-1] + bytes([g])   # tamper only the last byte
    if oracle(key, C_modified, C_target, strict=False):
        recovered = g ^ C_prev[-1]
        print(f"Recovered: byte={recovered}  char='{chr(recovered)}'")
        assert recovered == secret
        break

print()
print("=== TLS 1.x strict padding — POODLE blocked ===")
hits = [g for g in range(256)
        if oracle(key, C_prev[:-1] + bytes([g]), C_target, strict=True)]
print(f"Oracle accepted {len(hits)} of 256 guesses (no reliable oracle)")

Key insight: SSLv3's one-byte padding check lets the attacker flip a single ciphertext byte until the server accepts — revealing the secret through ~256 requests per byte. TLS 1.x's strict all-bytes check eliminates this oracle.
