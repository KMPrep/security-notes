Cryptography

Category: Cryptography, secure systems design, and production security engineering

Overview

Cryptography is the part of security engineering that lets systems preserve secrecy, prove integrity, authenticate peers, and reduce the damage of untrusted networks or untrusted storage. At a conceptual level, people often describe cryptography as algorithms and formulas. At production scale, the harder problem is almost always key management, protocol composition, lifecycle hygiene, and making safe defaults hold across millions of clients, services, devices, and operators.

For a large internet company, the real questions are not just "is AES secure" or "what is a signature". The real questions are:

  • where keys are generated, stored, rotated, and revoked
  • which trust assumptions the protocol makes about clients, servers, and intermediaries
  • whether compromise of one component reveals past or future traffic
  • whether integrity and authenticity are enforced separately or accidentally assumed
  • whether engineers can operate the system safely under failure, latency, and incident conditions

Strong cryptographic primitives can still fail in practice if the wrong primitive is used for the wrong job, if key custody is weak, if randomness is weak, or if the surrounding protocol leaks secrets through metadata, replay, downgrade, or misuse.

Encryption vs Encoding vs Hashing vs Obfuscation vs Signing

These terms are frequently confused in practice and in real systems. They solve different problems.

Concept Primary Goal Reversible Uses Secret Key Typical Output Example Uses Important Warning
Encryption Secrecy / confidentiality Yes, with the right key Usually yes Ciphertext TLS record protection, disk encryption, encrypted database fields Encryption does not automatically provide integrity unless paired with authentication
Encoding Compatibility / representation Yes No Alternate representation Base64, URL encoding, UTF-8, ASCII Encoding is not a security control
Hashing Integrity fingerprinting / indexing No, by design No Fixed-length digest File fingerprints, content addressing, internal deduplication Plain hashes do not prove authenticity
Obfuscation Hindrance / raising effort Usually yes or recoverable No, or weakly Altered but still usable form Minified code, packed malware, renamed symbols Obfuscation slows analysis; it does not create strong security guarantees
Signing Authenticity, integrity, non-repudiation within a trust model Verification is public; signing requires private key Yes, private key for signer Signature blob plus signed data Software release signing, JWT/PKI signatures, document signing Signatures prove origin only relative to key trust and correct verification

Encryption

Encryption is for secrecy. It transforms plaintext into ciphertext so an observer without the proper key cannot read the protected data.

Two major families matter operationally:

Type Key Model Strengths Weaknesses Common Uses
Symmetric encryption One shared secret key Fast, efficient, ideal for large data volumes Key distribution is hard Bulk traffic encryption, disk encryption, database field encryption
Asymmetric encryption Public key and private key pair Simplifies trust bootstrapping and distribution Slower, larger keys or more expensive math Key exchange, signatures, certificate-based identity

Important examples:

  • RSA: classic asymmetric primitive; widely used historically for encryption and signatures, but modern deployments prefer it mostly for signatures and legacy compatibility.
  • ECC: asymmetric family based on elliptic curves; smaller keys and strong security properties for signatures and key exchange.
  • AES: dominant symmetric block cipher in modern systems.
  • ChaCha20: modern symmetric stream cipher often paired with Poly1305, especially useful where AES hardware acceleration is not available.
  • Salsa20: predecessor to ChaCha; important historically and conceptually, less common in mainstream protocols now.

Operationally, most real protocols use asymmetric cryptography only long enough to authenticate peers and establish shared keys, then switch to symmetric encryption for the data path because symmetric crypto is much cheaper.

Encoding

Encoding is for compatibility, transport, storage, or interoperability. It transforms data into a format another system can parse or carry.

Examples:

  • URL encoding for reserved characters in URLs
  • Base64 for binary-to-text representation
  • ASCII or UTF-8 encoding for character representation

Encoding is reversible without secrets. If a system stores secrets in Base64 and calls them "encrypted", that is a design failure, not a crypto design.

Hashing

Hashing is for creating a fixed-size digest from arbitrary input. A cryptographic hash function should be deterministic, fast to compute, hard to invert, and hard to find collisions for.

Examples:

  • MD5: broken for collision resistance; still seen in legacy fingerprints or non-security deduping.
  • SHA-1: no longer acceptable for collision-sensitive security uses.
  • SHA-256: common secure baseline.
  • BLAKE2 and BLAKE3: modern designs with strong performance and good engineering properties.

Security properties to know:

  • Preimage resistance: given a hash, it should be infeasible to find an input that matches it.
  • Second-preimage resistance: given one input, it should be infeasible to find a different input with the same hash.
  • Collision resistance: it should be infeasible to find any two different inputs with the same hash.

Hashes are useful for integrity checking, indexing, fingerprinting malware samples, content-addressable storage, and cache validation. They do not prove who produced the data. For that you need a keyed mechanism such as HMAC or a signature.

Obfuscation

Obfuscation is for hindrance, not for strong cryptographic security. It changes the representation of software or data to make analysis less convenient while preserving behavior.

Examples:

  • stripping symbols from binaries
  • removing whitespace from code
  • wrapping payloads in Base64 or layers of encoding
  • custom packers in malware

Obfuscation may slow down reverse engineering or reduce accidental exposure, but it should never be treated as equivalent to encryption or signing.

Signing

Signing is for authenticity and integrity. A signer computes a digest of the data and uses a private key to create a signature. Anyone with the corresponding public key can verify that the signature was produced by the holder of the private key and that the signed data has not changed.

Common uses:

  • package and software release signing
  • signed firmware updates
  • certificate issuance in PKI
  • token signatures in systems such as JWT
  • document signing workflows

Signing answers a different question than encryption:

  • encryption asks: who can read this?
  • signing asks: who approved or produced this?

Many systems need both.

Threat Models And Attack Models

Cryptography must be evaluated under explicit attacker models. Without a threat model, the primitive choice is often meaningless.

Attack Model What The Attacker Gets Why It Matters
Ciphertext-only attack Only encrypted outputs Basic baseline for confidentiality
Known-plaintext attack Some plaintext and matching ciphertext Realistic in many file and protocol formats
Chosen-plaintext attack Attacker can choose plaintexts and observe ciphertexts Critical for modern encryption definitions and web-facing protocols
Chosen-ciphertext attack Attacker can feed ciphertexts to a decryption oracle or equivalent behavior Important for adaptive protocol attacks and padding oracles
Replay attack Attacker reuses valid old messages Relevant for auth protocols and APIs
Downgrade attack Attacker forces weaker algorithms or modes Common failure mode in protocol negotiation
Side-channel attack Attacker infers secrets from timing, power, cache, faults, or error behavior Strong math can still fail under bad implementation

Production-grade protocol design assumes active attackers, not just passive eavesdroppers. That means defending against tampering, replay, downgrade, identity confusion, and implementation leakage, not just hiding bytes on the wire.

Public Key Infrastructure (PKI)

PKI is the operational system for establishing trust around public keys. The math is only one piece. The larger system includes identity proofing, certificate issuance, certificate chains, revocation, trust stores, rotation, audit, and governance.

What PKI Does

PKI binds a public key to an identity through a digitally signed certificate. That identity might be a DNS name, service name, user, device, workload, or code-signing entity.

Core components:

Component Role
Root CA Trust anchor that signs intermediate CAs
Intermediate CA Delegated issuer used to reduce root-key exposure
Certificate Signed statement binding an identity to a public key
Registration / issuance workflow Proves requester identity before cert issuance
Trust store Set of roots or anchors a client accepts
Revocation / status system Indicates when a certificate should no longer be trusted
Policy and audit Constrains issuance behavior and supports incident response

Why PKI Is Operationally Hard

At scale, the hard parts are:

  • proving identity correctly before issuance
  • preventing over-issuance or misissuance
  • rotating certificates without outages
  • distributing trust anchors consistently
  • revoking compromised keys fast enough to matter
  • keeping private keys inside HSMs or strong boundary controls

This is why many large companies invest heavily in private PKI, short-lived certificates, certificate transparency monitoring, automated renewal, and clear ownership of issuance policy.

Key Exchange: DH, ECDH, And Why It Matters

Key exchange solves the problem of establishing a shared secret over an untrusted channel. See Key-Exchange-and-Forward-Secrecy.md for a detailed breakdown with concrete examples, attack walkthroughs, and the full forward secrecy analysis.

Diffie-Hellman (DH)

Classic Diffie-Hellman allows two parties to derive the same shared secret without sending that secret directly over the wire. An eavesdropper sees public values but should not be able to compute the shared secret efficiently.

Elliptic Curve Diffie-Hellman (ECDH)

ECDH provides the same high-level goal using elliptic curve math. It is generally preferred in modern systems because it provides strong security with smaller keys and lower computational cost than many legacy finite-field DH configurations.

Important Practical Point

Bare DH or ECDH does not authenticate the peer by itself. Without authentication, a man-in-the-middle can perform separate key exchanges with each side and relay traffic. This is why key exchange is typically combined with signatures, certificates, pre-shared trust, or an authenticated key exchange protocol such as TLS.

Forward Secrecy

Forward secrecy means compromise of a long-term secret later should not reveal past session contents.

This matters enormously for high-value systems. If a server's long-term private key is stolen next month, an attacker should not be able to decrypt traffic they captured last year.

Forward secrecy is typically achieved by using ephemeral key exchange values per session, usually ephemeral ECDH in modern protocols. The long-term key authenticates the session setup, but the traffic keys are derived from fresh ephemeral material.

Why It Matters At Scale

For services handling millions of users, credential compromise is not hypothetical. Keys are rotated, devices are lost, insiders make mistakes, and logs or packet captures may be retained. Forward secrecy shrinks the blast radius of inevitable future compromise.

End-To-End Encryption For Messaging

End-to-end encryption means only the communicating endpoints can decrypt message content. Servers may route, queue, or fan out messages, but they should not be able to read plaintext message bodies.

Core Idea

Messaging systems often use an asymmetric setup phase to authenticate or bootstrap trust, then derive symmetric keys for message encryption. Modern secure messengers go further and derive fresh keys frequently, often per session step or per message chain, so key compromise has limited retrospective value.

Double Ratchet Mindset

A mature design uses evolving keys, not one long-lived shared session key. The practical goals are:

  • forward secrecy for past messages
  • post-compromise recovery for future messages after a device recovers or refreshes keys
  • limiting damage if one message key leaks

The high-level pattern is:

  1. Use long-term identity keys to establish trust.
  2. Use ephemeral key exchange to derive initial shared secrets.
  3. Derive message keys from a ratcheting key schedule.
  4. Delete old message keys when no longer needed.

In real systems, the hard part is not only the cipher. It is device verification, multi-device synchronization, recovery workflows, offline delivery, metadata exposure, abuse handling, and safe key backup policy.

Ciphers

The user's source list uses the spelling "cyphers", but the correct security term is "ciphers".

Block Ciphers vs Stream Ciphers

Type How It Works Strengths Weaknesses Common Examples
Block cipher Encrypts fixed-size blocks of data Mature, flexible, widely accelerated in hardware Needs a secure mode of operation; misuse can be catastrophic AES
Stream cipher Generates a keystream combined with plaintext incrementally Efficient, low latency, good for streams and variable-length data Nonce reuse is often fatal ChaCha20, Salsa20

Block Cipher Modes Of Operation

A block cipher alone is not enough. The mode determines how blocks are chained and whether integrity is provided. See Block-Cipher-Modes.md for a detailed breakdown of each mode, attack scenarios, and decision rules.

Mode Properties Major Risk
ECB Simple but leaks structure because identical plaintext blocks encrypt the same way Not suitable for most real data
CBC Historically common; requires IV handling and separate integrity protection Padding-oracle and misuse risk
CTR Turns a block cipher into a stream-like mode Nonce reuse breaks confidentiality
GCM AEAD mode providing confidentiality and integrity Nonce reuse can be catastrophic
XTS Common for disk encryption of sectors Not intended for general message authentication

AES-GCM

AES-GCM is a widely used AEAD mode: authenticated encryption with associated data. It provides:

  • confidentiality for encrypted fields
  • integrity for ciphertext and additional authenticated data
  • high performance on modern hardware with AES acceleration

Associated data can be authenticated without being encrypted, which is useful for protocol headers or metadata fields that must remain visible but tamper-resistant.

Important warning: GCM is powerful but unforgiving. Nonce reuse under the same key can destroy security. Large distributed systems therefore need robust nonce generation, key rotation, or library abstractions that eliminate nonce management footguns.

ChaCha20-Poly1305

In many modern systems, ChaCha20-Poly1305 is the practical peer to AES-GCM. It performs well in software and is valuable on mobile devices or CPUs without strong AES acceleration. Engineers should understand both, because real-world protocol stacks often negotiate between them.

Integrity And Authenticity Primitives

Confidentiality is only one part of secure design. Systems must also detect tampering and validate origin.

Hash Functions

Hash functions are foundational but easy to misuse.

Good uses:

  • identifying files or artifacts
  • fingerprinting malware samples
  • content addressing
  • caching and deduplication
  • constructing higher-level primitives such as HMAC, HKDF, Merkle trees, or signatures

Bad uses:

  • storing passwords without a password hashing function
  • assuming a bare hash proves authenticity
  • relying on MD5 or SHA-1 where collision resistance matters

Message Authentication Codes (MACs)

MACs provide integrity and authenticity for parties that share a secret key. If both sides know the MAC key, they can verify that a message came from someone holding the shared secret and was not modified in transit.

Important distinction:

  • a MAC is symmetric authenticity
  • a digital signature is asymmetric authenticity

HMAC

HMAC is a keyed-hash MAC built from a cryptographic hash function. It remains a robust and widely used primitive because it is simple, well studied, and less fragile than many custom constructions.

Common uses:

  • API request signing schemes
  • integrity protection for tokens or session artifacts
  • key derivation building blocks
  • validating webhook payloads

AEAD And Composition

Modern systems often prefer AEAD constructions such as AES-GCM or ChaCha20-Poly1305 because they combine confidentiality and integrity in one properly designed primitive. That reduces the risk of teams encrypting data but forgetting to authenticate it, which is a common and dangerous mistake.

Entropy, Randomness, And PRNGs

Randomness quality is one of the most operationally underestimated parts of cryptography. A mathematically strong cipher fails if the keys, nonces, or tokens are predictable.

PRNG vs CSPRNG

Term Meaning Security Relevance
PRNG Pseudorandom number generator May be statistically useful but not necessarily safe against prediction
CSPRNG Cryptographically secure pseudorandom number generator Designed so outputs are infeasible to predict without internal state

Security-sensitive systems should use the operating system CSPRNG or a well-audited library abstraction, not ad hoc application randomness.

Entropy Sources

Operating systems collect entropy from sources such as:

  • hardware noise and device interrupts
  • timing variation from system activity
  • dedicated hardware random generators where available
  • boot-time measurements and platform-specific sources
  • environmental noise mixed into kernel entropy pools

Entropy Buffer Draining

If a system consumes randomness faster than it accumulates or reseeds it safely, early-boot or low-entropy environments can become dangerous. This matters in:

  • virtual machines cloned from similar images
  • embedded devices with limited noise sources
  • container-heavy bursts at startup
  • appliances that generate many keys immediately after boot

Good operating systems and modern libraries hide much of this risk, but infrastructure teams still need to think about entropy health in image build pipelines, VM boot storms, hardware fleets, and cryptographic services.

Mitigation Controls

# Failure Mitigation Where It Applies
1 Encoding confusion Enforce clear data-classification and crypto APIs Application and platform design
2 Missing integrity Default to AEAD or encrypt-then-MAC patterns Libraries and protocol design
3 Weak hashes Migrate to SHA-256, SHA-384, BLAKE2, or BLAKE3 as appropriate Platform standards
4 Nonce reuse Use safe libraries, per-key nonce discipline, automated testing Crypto libraries and service code
5 Weak randomness Use OS CSPRNG and verify entropy health in special environments Host and platform
6 PKI over-issuance Strong issuance policy, approvals, CT monitoring, short-lived certs PKI operations
7 Key exposure HSMs, KMS boundaries, least privilege, rotation, audit logging Secrets and key management
8 MITM on key exchange Authenticated key exchange with certificates, signatures, or pinned trust Protocol and transport
9 No forward secrecy Prefer ephemeral DH/ECDH handshakes Protocol configuration
10 Custom crypto Standardize audited libraries and ban ad hoc primitives Engineering governance
11 Wrong password storage Use Argon2id, scrypt, or bcrypt with sound parameters Authentication systems
12 Obfuscation over-trust Treat obfuscation only as a delaying layer Risk and design review

Engineering Guidance For Systems Serving Millions

The most mature cryptography posture is usually boring by design:

  1. Use standard, audited libraries instead of custom constructions.
  2. Keep cryptography decisions behind safe platform APIs so every team does not reinvent nonce, IV, or key handling.
  3. Centralize key management in HSM, KMS, or similarly controlled systems.
  4. Prefer short-lived credentials and automated rotation over long-lived secrets.
  5. Encrypt data in transit and at rest, but also enforce integrity, identity, and authorization separately.
  6. Design for compromise: assume some keys, hosts, or clients will eventually fail.
  7. Make crypto observability real: key usage logs, issuance logs, expiry alerts, error-rate monitoring, and revocation workflows.
  8. Treat protocol agility as governed change, not arbitrary negotiation freedom.

Key Takeaways

  • Encryption is for secrecy; encoding is for representation; hashing is for fingerprints; obfuscation is for hindrance; signing is for authenticity.
  • Symmetric cryptography is the fast data-plane workhorse; asymmetric cryptography is primarily for trust establishment and signatures.
  • Bare confidentiality is not enough. Integrity and authenticity must be explicit.
  • PKI is mostly an operational trust system, not just a certificate file format.
  • Forward secrecy dramatically reduces retrospective damage after key compromise.
  • AES-GCM and ChaCha20-Poly1305 are modern practical building blocks, but both fail if misused.
  • Entropy and secure randomness are foundational. Predictable keys ruin everything above them.
  • At large scale, the biggest crypto failures usually come from key handling, unsafe composition, bad defaults, and weak operational discipline rather than broken math.
