Symmetric Encryption

Category: Cryptography — Symmetric Ciphers, AES, Stream Ciphers

Overview

Symmetric encryption uses the same key for both encryption and decryption. It is fast and efficient for bulk data encryption. The core challenge is key distribution — how do both parties get the same key without an attacker intercepting it? In practice, symmetric keys are typically established via asymmetric key exchange (e.g., ECDHE in TLS).

AES — Advanced Encryption Standard

AES is the gold standard symmetric cipher. It replaced DES in 2001 and is used everywhere: TLS, disk encryption, file encryption, VPNs.

AES Parameters

Variant Key Size Rounds Security Level
AES-128 128 bits 10 ~128 bits — strong
AES-192 192 bits 12 ~192 bits
AES-256 256 bits 14 ~256 bits — highest

AES operates on 128-bit blocks regardless of key size. Each round applies: SubBytes, ShiftRows, MixColumns, AddRoundKey (final round omits MixColumns).

Block Cipher Modes

The mode determines how AES is applied to data longer than one block. See also Block Cipher Modes.

Mode IV Required Parallel Encrypt Parallel Decrypt Authentication Use
ECB No Yes Yes No Never use — identical blocks produce identical ciphertext
CBC Yes No Yes No Legacy; padding oracle attacks (POODLE, BEAST)
CTR Yes (nonce) Yes Yes No Stream-cipher mode; good performance
GCM Yes (nonce) Yes Yes Yes (AEAD) Current standard for TLS 1.3, disk encryption
CCM Yes No No Yes (AEAD) IoT, embedded
SIV Yes No No Yes (AEAD) Nonce-misuse resistant

GCM (Galois/Counter Mode) is the most important to know:

  • Combines CTR mode encryption with GHASH authentication
  • Produces ciphertext + authentication tag (128 bits)
  • Nonce must never be reused — reuse of the same nonce with the same key is catastrophic (reveals keystream XOR and authentication key)
  • Used in TLS 1.3, SSH, WireGuard, LUKS
flowchart LR
    P[Plaintext] --> CTR[CTR Encryption with Key+Nonce]
    CTR --> C[Ciphertext]
    C --> GH[GHASH with Auth Key]
    AAD[Additional Authenticated Data] --> GH
    GH --> T[Auth Tag]

ChaCha20-Poly1305

ChaCha20-Poly1305 is a stream cipher + MAC combination developed by Daniel Bernstein. It is the main alternative to AES-GCM.

Property AES-GCM ChaCha20-Poly1305
Performance on hardware with AES-NI Very fast Slower
Performance on hardware without AES-NI Slower Faster
Side-channel resistance Requires AES-NI for resistance Designed to be constant-time
Nonce reuse Catastrophic Catastrophic
Standard use TLS 1.3, QUIC TLS 1.3, WireGuard, Signal

Both are AEAD (Authenticated Encryption with Associated Data) ciphers and are considered equivalent in security. ChaCha20 is preferred on mobile and low-power devices without hardware AES support.

DES and 3DES (Legacy)

Cipher Key Size Block Size Status
DES 56 bits 64 bits Broken — brute-forceable in hours
3DES (Triple DES) 112 effective bits 64 bits Deprecated; sweet32 attack on 64-bit block size

3DES was deprecated by NIST in 2017 and disallowed in TLS 1.3. Do not use either in new systems.

Stream Ciphers

A stream cipher generates a pseudo-random keystream and XORs it with the plaintext byte by byte.

Property Block Cipher (AES) Stream Cipher
Unit of operation Fixed blocks (128 bits) Individual bytes or bits
Use case File encryption, TLS record Real-time audio/video, embedded
IV/nonce Required per block Required per stream

RC4 — once ubiquitous in SSL/WEP, now completely broken. Never use.

ChaCha20 — modern, secure stream cipher; designed to be fast in software.

Critical Stream Cipher Rule

Never reuse a keystream. If the same key and nonce produce the same keystream, XORing two ciphertexts with the same keystream yields the XOR of the two plaintexts, which is trivially breakable.

$$C_1 \oplus C_2 = P_1 \oplus P_2$$

This is why WEP was broken — it frequently reused IVs (24-bit IV space = ~16 million values; birthday paradox means collisions after ~4000 packets).

Key Derivation Functions (KDFs)

A KDF derives one or more cryptographic keys from a master secret (e.g., a password or shared secret from DH).

KDF Purpose Notes
HKDF Derive keys from a high-entropy input (e.g., DH shared secret) Used in TLS 1.3 key schedule
PBKDF2 Derive keys from passwords Tunable iteration count; slower than bcrypt
bcrypt Password hashing Memory-moderate; time-adaptive; widely used
scrypt Password hashing Memory-hard; resists GPU cracking
Argon2 Password hashing Winner of Password Hashing Competition; memory-hard; current recommendation

Why KDFs for passwords? Raw passwords have low entropy. A KDF adds iterations (work factor) to make brute force expensive. A cryptographic hash (MD5, SHA256) is too fast — GPUs can compute billions/second.

1. Why is ECB mode never acceptable for encrypting data?

ECB = Electronic Codebook. It encrypts each plaintext block independently:

C1 = AES_K(P1)
C2 = AES_K(P2)
C3 = AES_K(P3)

So if two plaintext blocks are the same, their ciphertext blocks are also the same. NIST describes ECB as assigning a fixed ciphertext block to each plaintext block for a given key. That means ECB leaks structure, repetition, and equality patterns. (NIST Publications)

Concrete attack: encrypted role token cut-and-paste

Assume an application encrypts this token using AES-ECB:

uid=1001&role=user&expiry=2026-12-31

Because ECB encrypts independent 16-byte blocks, the ciphertext is like:

C1 = Enc("uid=1001&role=")
C2 = Enc("user&expiry=...")

Now suppose the attacker can register or submit controlled input that causes the system to encrypt a block containing:

admin&expiry=...

The attacker copies the ciphertext block for admin and replaces the ciphertext block for user.

Final forged ciphertext:

C1 || C_admin

When the server decrypts it, it sees:

uid=1001&role=admin&expiry=2026-12-31

The attacker did not break AES. They exploited ECB’s lack of chaining, randomness, and authentication.

Another practical example

If a database encrypts repeated values with ECB:

diagnosis=HIV_POSITIVE
diagnosis=HIV_POSITIVE
diagnosis=DIABETES
diagnosis=HIV_POSITIVE

all identical diagnosis blocks produce identical ciphertext blocks. An attacker who sees the database can infer which rows have the same hidden value, even without knowing the key.

Answer: ECB is unacceptable because it provides confidentiality only for isolated random-looking blocks, not for real structured data. It leaks patterns and enables block rearrangement attacks. Use an AEAD mode like AES-GCM or ChaCha20-Poly1305 instead.

2. What does AEAD stand for and why is it important?

AEAD = Authenticated Encryption with Associated Data.

It gives you three protections together:

Confidentiality: plaintext is encrypted.
Integrity: ciphertext cannot be modified undetected.
Authenticity: receiver knows the ciphertext was produced by someone with the key.

The “associated data” part means some fields are not encrypted, but are still integrity-protected.

Example:

Encrypted:
  payment_amount=1000
  beneficiary_account=123456

AAD:
  protocol_version=1
  tenant_id=abc
  request_id=xyz
  content_type=payment

The AAD remains visible, but if an attacker changes tenant_id, protocol_version, or content_type, tag verification fails.

NIST describes GCM as an authenticated encryption mode with associated data. It protects the authenticity of both plaintext and AAD, while only the plaintext is encrypted. NIST gives protocol metadata such as addresses, ports, sequence numbers, and version numbers as examples of AAD. (NIST Publications)

3. How does AES-GCM provide AEAD?

AES-GCM combines two mechanisms:

AES-CTR encryption  +  GHASH authentication

AES-CTR part

AES-GCM uses AES in counter mode to generate a keystream:

ciphertext = plaintext XOR keystream

This provides confidentiality.

GHASH part

GCM computes an authentication value over:

AAD || ciphertext || lengths

Then it produces an authentication tag.

On decryption, the receiver recomputes the tag. If the tag does not match, decryption must fail and the plaintext must not be trusted. NIST defines GCM authenticated decryption as returning either the plaintext or FAIL depending on tag verification. (NIST Publications)

So AES-GCM protects both:

Encrypted data: confidential + authenticated
AAD: visible + authenticated

4. What happens if you reuse a GCM nonce with the same key?

It is catastrophic.

AES-GCM relies on the rule:

same key + same nonce must never happen

NIST says GCM IVs must satisfy a uniqueness requirement, and that even one repeated IV with the same key may make the implementation vulnerable to forgery attacks. NIST also says this requirement is almost as important as secrecy of the key. (NIST Publications)

Why confidentiality breaks

GCM uses counter mode. If the same key and nonce are reused, the same keystream is reused.

For two messages:

C1 = P1 XOR S
C2 = P2 XOR S

where S is the reused keystream.

An attacker can compute:

C1 XOR C2 = P1 XOR P2

If the attacker knows or can guess one plaintext, they recover the other:

P2 = C1 XOR C2 XOR P1

Concrete attack

Suppose message 1 is predictable:

P1 = "GET /profile HTTP/1.1..."

and message 2 contains a secret:

P2 = "Authorization: Bearer eyJ..."

If both were encrypted with the same AES-GCM key and nonce, the attacker can use the known HTTP request structure from P1 to recover corresponding bytes of P2.

Why authentication also breaks

GCM authentication uses a GHASH value derived from the AES key. With nonce reuse, the attacker gets algebraic relationships between messages and tags. With enough structure, they can forge valid tags for modified ciphertexts.

Answer: GCM nonce reuse is not a minor bug. It can reveal plaintext relationships, recover unknown plaintext when one plaintext is known or guessable, and enable authentication tag forgery.

Design rule: Use a 96-bit nonce, generated by a crash-safe counter or a construction that guarantees uniqueness. NIST recommends restricting GCM IV support to 96 bits for interoperability, efficiency, and simplicity. (NIST Publications)

5. Compare AES-GCM and ChaCha20-Poly1305

Area AES-GCM ChaCha20-Poly1305
Type AEAD mode using AES + GHASH AEAD using ChaCha20 stream cipher + Poly1305 MAC
Best platform Servers, cloud, CPUs with AES-NI or ARM crypto extensions Mobile, embedded, software-only environments
Performance Very fast with hardware AES acceleration Often faster when AES hardware is absent
Side-channel risk Bad AES table implementations can leak timing/cache information Designed to be easier to implement in constant time
Compliance Common in FIPS/NIST environments Common in TLS/IETF ecosystems, but may not satisfy FIPS requirements
Nonce requirement Nonce must be unique per key Nonce must also be unique per key
Tag Usually 128-bit tag 128-bit Poly1305 tag

RFC 8439 states that ChaCha20 is considerably faster than AES in software-only implementations on platforms without specialized AES hardware, and that ChaCha20 is designed to avoid timing attacks. ([IETF Datatracker][3])

When I would prefer AES-GCM

Use AES-GCM when:

- You have hardware AES acceleration.
- You are in cloud/server-side workloads.
- You need FIPS-friendly cryptography.
- Your crypto library has a mature, constant-time AES-GCM implementation.

Example: encrypting application payloads in a backend service running on modern Intel/AMD/ARM servers.

When I would prefer ChaCha20-Poly1305

Use ChaCha20-Poly1305 when:

- You run on mobile, IoT, embedded, or older CPUs.
- AES acceleration may not exist.
- You want simpler constant-time software behavior.
- You are using modern TLS stacks that negotiate it.

Example: mobile app traffic where some devices lack fast AES hardware.

Important: ChaCha20-Poly1305 also fails badly on nonce reuse. RFC 8439 states that repeating a nonce repeats both the Poly1305 one-time key and keystream, revealing XOR of plaintexts. ([IETF Datatracker][3])

6. Why is DES broken? Why is 3DES also no longer acceptable?

DES is broken because the key is too small

DES uses a 56-bit effective key. That was acceptable decades ago, but it is now brute-forceable. NIST’s DES standard was withdrawn in 2005 because it no longer provided the security needed for federal information. (NIST Computer Security Resource Center)

Brute force for DES is conceptually simple:

for every possible 56-bit key:
    decrypt ciphertext
    check if plaintext looks valid

Modern hardware can search this keyspace far too cheaply compared with modern security expectations.

3DES tried to extend DES life

3DES applies DES three times:

C = DES_K3( DES^-1_K2( DES_K1(P) ) )

This increased resistance to simple brute force, but it still has major problems:

- It is slow.
- It has a 64-bit block size.
- It is vulnerable to birthday-bound issues when large data volumes are encrypted.
- It has been deprecated/disallowed by modern standards.

The 64-bit block size is especially painful. With enough encrypted traffic under the same key, block collisions become realistic. This is the basis of the SWEET32 class of attacks against 64-bit block ciphers such as 3DES in protocols like TLS and OpenVPN. (sweet32.info)

NIST announced withdrawal of the Triple Data Encryption Algorithm recommendation effective January 1, 2024, and had already urged users to migrate from TDEA/3DES to AES. (NIST Computer Security Resource Center)

Answer: DES is dead because 56-bit keys are brute-forceable. 3DES is dead because it is slow, has only a 64-bit block size, suffers birthday-bound limits, and is no longer acceptable for modern encryption. Use AES-GCM, AES-CTR with strong MAC, AES-SIV/GCM-SIV for misuse resistance, or ChaCha20-Poly1305 depending on the environment.

7. What is a Key Derivation Function and why do you need one when storing passwords?

A Key Derivation Function, or KDF, turns input key material into a derived key or verifier.

For password storage, the input is usually:

password + salt + cost parameters

and the output is:

password hash / verifier

NIST says stored passwords must be salted and hashed using a suitable password hashing scheme, and that these schemes take a password, salt, and cost factor to make each attacker guess more expensive. (NIST Pages)

Why this is needed

User passwords are low entropy. Humans choose passwords like:

Password123!
Welcome@2026
Singapore123

If an attacker steals the password database, they perform an offline cracking attack:

guess password -> hash guess -> compare with stolen hash

A password KDF makes every guess expensive.

Good password storage uses:

unique salt per user
slow/adaptive cost
memory hardness where possible
versioned parameters
optional pepper stored outside the database

The salt prevents rainbow-table reuse and makes identical passwords produce different stored hashes. The cost factor slows down each guess. Memory-hard functions make GPU/ASIC cracking more expensive.

8. Why is bcrypt preferred over SHA-256 for password hashing?

SHA-256 is designed to be fast. That is good for file integrity, signatures, and general hashing. It is bad for password storage.

OWASP explicitly says fast hashes such as SHA-256 are not suitable for password storage because they allow attackers to perform large numbers of guesses quickly. OWASP recommends slow password hashing algorithms such as Argon2id, bcrypt, scrypt, or PBKDF2 depending on environment and requirements. (OWASP Cheat Sheet Series)

SHA-256 password storage problem

Bad design:

stored_hash = SHA256(password)

Even with a salt:

stored_hash = SHA256(salt || password)

it is still too fast. Attackers can test huge numbers of guesses per second using GPUs.

bcrypt advantage

bcrypt is better because it is:

- slow by design
- salted
- adaptive via work factor
- widely supported
- expensive to brute force compared with SHA-256

Example stored bcrypt hash:

$2b$12$KIXxX9QnQxE2fH5GkYQYQe...

This encodes:

algorithm = bcrypt
cost = 12
salt = embedded
hash = embedded

So the application can increase the cost over time as hardware improves.

Important modern nuance

For a new system today, I would usually prefer:

1. Argon2id
2. scrypt
3. bcrypt if Argon2id/scrypt are not available or for legacy compatibility
4. PBKDF2-HMAC-SHA-256 when FIPS requirements apply

OWASP currently recommends Argon2id as the preferred modern choice and says bcrypt should mainly be used where Argon2id or scrypt are not available, with an appropriate work factor. (OWASP Cheat Sheet Series)

Answer: bcrypt is preferred over SHA-256 because password hashing is not about producing a digest quickly; it is about making offline guessing economically painful. SHA-256 is too fast. bcrypt adds salt and tunable cost. For greenfield systems, Argon2id is usually the stronger modern recommendation.
