Anomaly and Behaviour Detection

Category: Detection Engineering — Behavioural Analytics, UEBA, ML-Based Detection

Overview

Anomaly-based detection identifies threats not by matching known signatures but by recognising deviations from an established baseline of normal behaviour. It is the primary approach for detecting:

  • Zero-day exploits (no signature exists yet)
  • Insider threats (attacker uses legitimate credentials and tools)
  • Slow-and-low attacks (designed to stay under signature thresholds)
  • Living-off-the-land (LOLBin) attacks (using built-in OS tools to avoid AV)

The fundamental tradeoff: anomaly detection catches what signatures miss, but it generates more false positives because "normal" is inherently variable.

Baselines and Profiles

Anomaly detection requires a baseline — a model of what normal looks like for a user, host, service, or network.

What Gets Baselined

Entity Baseline Metrics Example Anomaly
User Login hours, source IPs, applications used, data volume Login at 3 AM from new country
Host Network peers, process list, CPU/memory usage Host suddenly scanning 1000 IPs
Service account Specific APIs called, source IPs, call frequency Service account suddenly calling admin API
Network segment Traffic volume, protocol mix, destination IPs Internal host sending 5 GB to external IP
Database Query patterns, rows returned, calling user DBA running SELECT * on PII table at midnight

Baseline Challenges

Challenge Description
Seasonality Traffic patterns differ by day-of-week, time of year
User role changes Legitimate changes in job function shift behaviour
New systems No history to build baseline
Shared accounts Multiple people using one service account confounds baseline
Attacker baseline poisoning Slow attacker gradually shifts their behaviour into the baseline

UEBA — User and Entity Behaviour Analytics

UEBA platforms apply statistical and machine learning models to detect anomalous behaviour across users, hosts, and services. They aggregate weak signals into a risk score per entity, reducing alert volume.

flowchart TD
    A[Raw Events from SIEM] --> B[Feature Extraction]
    B --> C[Baseline Models per Entity]
    C --> D[Anomaly Scoring]
    D --> E[Risk Score per User/Host]
    E --> F[Alert on High-Risk Entities]
    F --> G[SOC Triage]

Statistical Anomaly Detection Methods

Z-Score / Standard Deviation

Flag events where a metric is more than N standard deviations from the mean.

z = (observed_value - mean) / standard_deviation
Alert if |z| > 3  (3-sigma rule)

Good for: Volume anomalies (bytes transferred, login count, API call rate)
Bad for: Non-normal distributions; seasonal data without adjustment

Isolation Forest

A machine learning algorithm that identifies anomalies as points that are "isolated" in feature space with fewer random splits. Does not require labelled data (unsupervised).

Good for: High-dimensional feature spaces; mixed numerical features
Used in: Fraud detection, network intrusion detection

DBSCAN Clustering

Density-based spatial clustering. Points in low-density regions are classified as outliers.

Good for: Detecting unusual event clusters (login location clusters, C2 IP clusters)

Markov Chain / Sequence Models

Model sequences of events (e.g., normal user: login → read email → access CRM → logout). Flag sequences that deviate from expected transitions.

Good for: Detecting account takeover (unusual navigation path), API abuse

Time-Series Forecasting (ARIMA, Prophet)

Forecast expected metric values over time. Alert when observed value diverges significantly from forecast.

Good for: Network traffic volume, API call rates, login counts by hour

Behaviour-Based Detection Rules

Even without full ML, behaviour-based logic in a SIEM can catch many threats:

Impossible Travel

Two successful logins for the same user
where time_delta < 60 minutes
and haversine_distance(ip1, ip2) > 1000 km

Credential Stuffing (many users from same IP)

auth_failure events
group by src_ip, 15-minute window
where distinct_users > 50

Port Scanning

TCP SYN packets
from same src_ip, 10-second window
where distinct_dst_ports > 20
and SYN_ACK response count < 5

DNS Beaconing (Periodic C2)

DNS queries
group by src_ip, queried_domain, 1-hour window
where query_count > 10
and stddev(inter_query_interval) < 10 seconds

LOLBin Execution Chain (Sysmon)

Process creation events
where parent_process = "winword.exe"
and child_process in ("cmd.exe", "powershell.exe", "wscript.exe", "mshta.exe")

Analyst Fatigue and Alert Quality

The biggest operational problem in detection is alert fatigue — too many alerts, most of them false positives, causing analysts to start ignoring or batch-closing without investigation.

Metric Target Notes
False positive rate < 20% Higher rates cause analysts to distrust alerts
Alert-to-ticket ratio > 50% of alerts should warrant a ticket Low ratio means tuning is needed
Mean time to triage < 30 minutes for P1 Measure and track over time
Alert suppression rate Track carefully High suppression may mean hiding real signals

Improving Alert Quality

  1. Triage playbooks — define exactly what to check for each alert type
  2. Risk score aggregation — combine multiple low-confidence signals into one high-confidence alert per entity
  3. Context enrichment — automatically attach asset criticality, user role, recent activity history
  4. Allowlists — exclude known-good automation accounts, scanners, IT tools
  5. Feedback loops — analysts close tickets with "true positive / false positive"; model uses feedback to retrain
