Detection Engineering — Discipline, Lifecycle, and Detection-as-Code

Category: Security Operations, Threat Detection, Detection Engineering
Related notes: MITRE ATT&CK · Sigma and Detection-as-Code · Threat Hunting · IDS and SIEM · YARA and Snort · Anomaly Detection
Anchors: MITRE ATT&CK, Pyramid of Pain (Bianco 2013), Sigma, SIGMA HQ, NIST SP 800-53 (SI/AU families), Google/Chronicle SecOps, Palantir Alerting & Detection Strategy (ADS) framework

1. Overview

Detection engineering is the discipline of treating detections — the rules, analytics, and models that surface attacker activity — as engineered software products with a lifecycle, tests, version control, and measurable quality, rather than as a pile of vendor signatures nobody owns. The purpose is to reduce dwell time (Mandiant M-Trends reported a global median of 10 days in 2023, down from 78 days in 2018 — but internally-discovered intrusions still lag), and to do it without drowning analysts in noise. A detection engineer's real product is not "an alert" — it is a high-precision signal plus the context and response runbook that lets a human act on it in minutes. At scale, where a single environment generates tens of billions of events per day, the binding constraint is almost never "can we write a rule," it is "can we write a rule that is precise enough to survive contact with production and cheap enough to run continuously." The discipline exists because ad-hoc detection does not scale: rules rot, coverage drifts, and false positives silently train analysts to ignore the console.

The modern practice fuses three ideas: a repeatable lifecycle (hypothesis → develop → test → deploy → tune → retire), detection-as-code (rules live in Git, get peer-reviewed, and are validated by CI before they ever fire in production), and coverage measurement against a shared adversary model — almost always MITRE ATT&CK. Underlying all of it is the Pyramid of Pain: detections anchored to attacker behaviors (TTPs) cost the adversary far more to evade than detections anchored to hashes or IPs, so a mature program deliberately shifts its detection portfolio up the pyramid.

TL;DR:
- Detection engineering = detections as software: version-controlled, peer-reviewed, CI-tested, measured. The unit of value is a precise, actionable signal, not raw alert count.
- The lifecycle is hypothesis → develop → test → deploy → tune → retire. Retirement is a first-class step; dead rules are technical debt.
- The Pyramid of Pain says TTP/behavior detections beat IOC detections because they cost the adversary the most to bypass. Shift left-to-right up the pyramid.
- Signal quality is precision (of alerts that fired, how many were real) and recall (of real attacks, how many we caught). Alert fatigue is a precision failure; coverage gaps are a recall failure.
- Measure coverage against MITRE ATT&CK, and health with MTTD, precision/FP rate, alert volume, and time-to-tune. If you can't measure it, you can't defend the budget for it.

2. How It Works

2.1 Core mechanics

Detection engineering has a control plane (how detections are authored, reviewed, versioned, and governed) and a data plane (the pipelines that carry telemetry into an engine that evaluates detection logic and emits alerts). The core objects:

Object What it is Why it matters
Detection (rule/analytic) Logic that maps a telemetry pattern to an alert (Sigma, KQL, SPL, YARA-L, EQL, SQL) The atomic unit of the discipline; owned, versioned, tested
Hypothesis A testable statement: "an adversary doing X leaves observable Y in log Z" Anchors development to a real threat, not a vibe
Detection strategy (ADS) Palantir's structured spec: goal, categorization, technical context, blind spots, FP model, validation, priority, response Forces you to document what you can't see, not just what you can
Telemetry source The log/event feed a detection reads (Sysmon, CloudTrail, EDR, Zeek) No source, no detection — coverage is bounded by data availability
Coverage map Detections mapped to ATT&CK techniques with a confidence score Turns "are we secure?" into a heatmap you can reason about
Signal quality metrics Precision, recall, FP rate, alert volume per rule The feedback loop that keeps the portfolio healthy
Enrichment/context Asset owner, user role, geo, threat intel, prior alerts Converts a raw match into a triageable alert
Response runbook The documented triage/containment steps a firing rule triggers An alert with no runbook is noise by construction

Evaluation/decision logic. A telemetry event flows through: collection → normalization (map raw fields to a schema like OCSF or ECS) → detection evaluation (the engine runs each enabled rule against the event/stream) → enrichment → alert deduplication/aggregation → routing to a queue with a severity. The detection engineer owns everything from "what schema fields exist" to "what the analyst sees." Two failure axes govern quality: a detection that fires on benign activity is a false positive (hurts precision, drives fatigue); a real attack that no rule catches is a false negative (hurts recall, is a coverage gap). You cannot maximize both simultaneously — tuning is the deliberate management of that trade-off per rule, per environment.

The Pyramid of Pain (David Bianco, 2013). From bottom (trivial for attacker to change) to top (painful):

Level Indicator Attacker cost to change Detection durability
1 Hash values (MD5/SHA256) Trivial — recompile / flip a byte Days or less
2 IP addresses Easy — rotate infra, use cloud/CDN Days–weeks
3 Domain names Simple — register new, use DGA Weeks
4 Network/host artifacts Annoying — change tooling defaults Weeks–months
5 Tools Challenging — re-tool or rebuild Months
6 TTPs (behaviors) Tough — change how they operate Months–years

The strategic implication: a detection for "process whoami.exe spawned by w3wp.exe within 5s of an inbound web request" (a behavior — webshell recon, T1505.003 + T1033) survives re-compilation, IP rotation, and domain churn. A detection for sha256 == abc123... dies the moment the attacker recompiles. A mature portfolio still keeps IOC detections (they are cheap and catch commodity threats) but invests its engineering effort at levels 4–6.

2.2 Concrete configuration

A Sigma rule (portable detection-as-code). Sigma is a YAML-based, backend-agnostic detection format; a converter (sigma-cli / pySigma) compiles it to Splunk SPL, Elastic KQL, Chronicle YARA-L, Sentinel KQL, etc. This is the lingua franca of detection-as-code because one rule targets many SIEMs.

title: Suspicious Whoami Execution From Web Server Process
id: 8f2e1a90-4c7b-4e2a-b1d3-9c0f5a2e7b11
status: experimental
description: Detects reconnaissance via whoami spawned by an IIS worker process,
  a common post-exploitation step after webshell deployment.
references:
  - https://attack.mitre.org/techniques/T1033/
author: detection-eng@corp
date: 2026/07/07
tags:
  - attack.discovery
  - attack.t1033
  - attack.persistence
  - attack.t1505.003
logsource:
  category: process_creation
  product: windows
detection:
  selection:
    ParentImage|endswith: '\w3wp.exe'
    Image|endswith: '\whoami.exe'
  condition: selection
falsepositives:
  - Legitimate admin diagnostics on the web tier (rare; scope to prod web fleet)
level: high

Commentary: logsource pins this to Sysmon/Windows process_creation (Event ID 1). selection is a behavioral pattern (parent-child relationship), not an IOC — it lives near the top of the Pyramid of Pain. The tags are the ATT&CK mapping that feeds coverage measurement. falsepositives and level are metadata the pipeline consumes for routing and tuning.

Compile and test it in CI. The rule never reaches production unraveled by hand:

# 1. Lint every rule's schema/structure (fails the build on malformed YAML)
sigma check rules/

# 2. Compile to the target backend to prove it converts cleanly
sigma convert -t splunk -p sysmon rules/whoami_from_w3wp.yml
# -> index=* source="WinEventLog:*Sysmon*" EventCode=1
#    ParentImage="*\\w3wp.exe" Image="*\\whoami.exe"

# 3. Unit-test against known-true and known-false event fixtures
sigma test rules/whoami_from_w3wp.yml \
  --true  tests/positive/webshell_whoami.json \
  --false tests/negative/admin_whoami_workstation.json

A positive test fixture (must alert) and a negative fixture (must NOT alert) pin the rule's behavior so a future edit can't silently break it:

// tests/positive/webshell_whoami.json  -> EXPECT: match
{ "EventID": 1, "Image": "C:\\Windows\\System32\\whoami.exe",
  "ParentImage": "C:\\Windows\\System32\\inetsrv\\w3wp.exe",
  "CommandLine": "whoami /all", "User": "IIS APPPOOL\\DefaultAppPool" }

// tests/negative/admin_whoami_workstation.json  -> EXPECT: no match
{ "EventID": 1, "Image": "C:\\Windows\\System32\\whoami.exe",
  "ParentImage": "C:\\Windows\\System32\\cmd.exe",
  "CommandLine": "whoami", "User": "CORP\\jsmith-admin" }

A GitHub Actions detection-as-code pipeline — the control plane that enforces peer review + CI before deploy:

name: detections-ci
on:
  pull_request:
    paths: ['rules/**']
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install pySigma toolchain
        run: pip install sigma-cli pysigma-backend-splunk pysigma-backend-elasticsearch
      - name: Lint rule schema
        run: sigma check rules/
      - name: Compile to all target backends
        run: |
          for b in splunk elasticsearch; do
            sigma convert -t $b -p sysmon rules/ -o /tmp/out_$b.txt
          done
      - name: Run unit tests (positive + negative fixtures)
        run: python tools/run_detection_tests.py --rules rules/ --tests tests/
      - name: ATT&CK coverage diff (block if a technique loses its last rule)
        run: python tools/coverage_report.py --fail-on-regression

Branch protection requires ≥1 approving review from a detection engineer and a green detections-ci before merge. On merge to main, a deploy job pushes the compiled rules to the SIEM via API (Splunk savedsearches REST, Sentinel alertRules ARM, Chronicle YARA-L API). This is the whole point of detection-as-code: every rule change is a reviewable, tested, revertable diff with an audit trail — not a click in a vendor console at 2 a.m.

Coverage measurement config — mapping the portfolio to ATT&CK and emitting a Navigator layer:

# tools/coverage_report.py (sketch)
import glob, yaml, json
technique_hits = {}
for path in glob.glob("rules/**/*.yml", recursive=True):
    rule = yaml.safe_load(open(path))
    for tag in rule.get("tags", []):
        if tag.startswith("attack.t"):
            tid = tag.split(".")[1].upper()            # t1033 -> T1033
            technique_hits.setdefault(tid, []).append(rule["title"])
# Emit a MITRE ATT&CK Navigator layer for the heatmap
layer = {"name": "Detection Coverage", "domain": "enterprise-attack",
         "techniques": [{"techniqueID": t, "score": min(len(v), 5),
                         "comment": f"{len(v)} rule(s)"} for t, v in technique_hits.items()]}
json.dump(layer, open("coverage_layer.json", "w"), indent=2)

Case study — Target (2013). Attackers pivoted from an HVAC vendor's credentials into the payment network and installed POS-scraping malware. Target's FireEye deployment did generate alerts on the malware's staging behavior. The failure was purely operational: the alerts routed to a team that did not act, and there was no enforced runbook tying "this specific alert" to "isolate and investigate." The control that would have stopped it is not a smarter rule — it's the detection-engineering discipline that says every deployed detection ships with a documented response playbook and an owner, and that alert-to-action MTTR is a tracked metric. A detection nobody responds to has zero value.

Case study — SolarWinds/SUNBURST (2020). The implant was signed, used legitimate update channels, and delayed its beaconing to evade sandboxes. IOC-based detection was hopeless: there were no known-bad hashes or domains for a novel supply-chain implant. What could have caught it was behavioral/anomaly detection anchored to the Pyramid of Pain's upper levels — e.g., "a host in the monitoring tier making first-seen outbound connections to a newly-registered domain," or "an application spawning processes it has never spawned in 90 days of baseline." This incident is the canonical argument for investing detection-engineering effort in TTP/anomaly analytics rather than IOC feeds.

Case study — Log4Shell (2021). When CVE-2021-44228 dropped, every SOC on earth needed detections for ${jndi:ldap://...} in HTTP headers, plus post-exploitation behavioral rules (Java process spawning curl/bash). Teams with detection-as-code wrote a Sigma rule, added positive/negative test fixtures, ran CI, got a peer review, and deployed to every SIEM backend within the hour — with confidence it wouldn't melt the console. Teams hand-editing vendor consoles shipped slower and broke things. Log4Shell was the industry's clearest demonstration that detection velocity is a defensive capability, and detection-as-code is how you get it.

Top 5 in depth

1. Detection-as-code with mandatory CI and peer review (mitigates #3, #8, #10, #15, #17). This is the load-bearing control of the entire discipline. Every detection lives in Git; every change is a pull request; a CI pipeline lints the schema, compiles the rule to each target SIEM backend, and runs positive/negative unit-test fixtures; branch protection requires a peer review from another detection engineer before merge; and a deploy job pushes compiled rules to production SIEMs via API. Technically, the magic is that the tests pin behavior: a positive fixture proves the rule still catches the attack, a negative fixture proves it hasn't started firing on benign activity — so a schema-field rename that would silently break a hand-authored rule instead fails the build. Trade-off: it is real engineering overhead (you need a test corpus, backend converters, and pipeline maintenance), and it slows down the "I'll just tweak it in the console" hotfix. In a true 0-day scramble you may need an emergency fast-path (deploy first, PR-within-24h), which must itself be governed or it becomes the norm. But the payoff — reviewable, testable, revertable, auditable detections — is non-negotiable at scale.

2. Coverage measurement against MITRE ATT&CK (mitigates #2, #4). You map every detection to the ATT&CK technique(s) it covers (via Sigma tags or an equivalent field), aggregate into a Navigator heatmap scored by rule count and confidence, and gate CI so that removing the last rule for a technique fails the build. Technically this converts an unanswerable question ("are we covered?") into a visual, diffable artifact and lets leadership prioritize by threat-actor relevance (e.g., overlay the techniques of actors known to target your sector via ATT&CK's group data). It also exposes over-reliance on IOC detections — you'll see broad red across behavioral techniques. Trade-off: coverage is easy to game — one weak rule "covers" a technique on the map but may have terrible recall. The map measures presence, not efficacy; you must pair it with validation (atomic red-team tests, purple-team exercises) to confirm the rules actually fire. Treat green as "we have something," not "we're safe."

3. Precision/recall SLOs and FP-budget-driven tuning (mitigates #1, #9, #11). Assign each rule a measurable target: e.g., precision ≥ 0.7 for high-severity rules, and a false-positive budget (say, ≤ 5 FPs/week). Instrument analyst dispositions (true positive / false positive / benign true) so precision is computed automatically, and auto-quarantine any rule that blows its FP budget until it's tuned. Technically, tuning means adding scope (restrict to prod web fleet), context (only alert if the identity is non-admin), allowlists (known scanners), or aggregation (alert on the Nth occurrence, not the 1st). This directly attacks alert fatigue — the highest-frequency, most damaging failure mode. Trade-off: aggressive precision tuning trades recall — every allowlist and threshold is a potential blind spot an attacker can hide inside. The right answer is per-rule and per-severity: a critical-asset ransomware canary can tolerate low precision because a miss is catastrophic; a broad discovery rule cannot. Tuning is judgment, encoded as reviewable diffs, never a one-time event.

4. Behavioral detection anchored to the Pyramid of Pain (mitigates #4, #6, #13). Deliberately invest engineering effort at Bianco levels 4–6: parent-child process relationships, command-line semantics, sequence-of-actions correlation, and anomaly baselines — rather than hashes/IPs. Technically, this means writing detections like "office document spawns a scripting host that spawns a network utility" (a phishing execution chain) instead of "sha256 == X." Such rules survive re-compilation, IP rotation, and domain churn, raising attacker cost by months. Trade-off: behavioral rules are harder to write, more prone to false positives (benign automation looks like attacker automation), and demand richer telemetry (you need command lines and process lineage, i.e., Sysmon/EDR, not just firewall logs). You keep cheap IOC rules for commodity threats and spend scarce engineering time on the durable behavioral layer — a portfolio, not a religion.

5. Complete telemetry onboarding with parser tests (mitigates #5, #7, #20). A detection can only see what the pipeline collects and correctly parses. The control is a formal data-source onboarding process: verify the source is shipped in real time, off-host, and to immutable storage (so log-clearing, T1070, can't erase evidence and is itself detected); write parser unit tests so a vendor log-format change doesn't silently null out fields your rules depend on; and enforce a coverage inventory of "which techniques are even possible to detect given current data." Technically, this is where OCSF/ECS normalization pays off — rules written against a stable schema survive source changes. Trade-off: telemetry is expensive (ingest, storage, and query cost scale with volume), so you must tier data (hot/warm/cold), sample where safe, and accept that you cannot log everything everywhere. The engineering discipline is choosing which sources unlock the most ATT&CK coverage per dollar.

KQL (Microsoft Sentinel) — mass file rename indicative of ransomware (T1486), a low-precision-but-critical canary:

DeviceFileEvents
| where Timestamp > ago(10m)
| where ActionType == "FileRenamed"
| where FileName endswith ".locked" or FileName endswith ".encrypted"
| summarize renamed = count(), sample = make_set(FolderPath, 5)
    by DeviceId, InitiatingProcessAccountName, bin(Timestamp, 5m)
| where renamed > 200
| project Timestamp, DeviceId, InitiatingProcessAccountName, renamed, sample

SQL (data-warehouse detection over normalized events) — detection health / rule-rot monitor:

-- Rules that have not matched anything in 90 days: retire or investigate broken parsing
SELECT r.rule_id, r.title, r.att_ck_technique, MAX(a.fired_at) AS last_fired
FROM detection_rules r
LEFT JOIN alerts a ON a.rule_id = r.rule_id
WHERE r.status = 'production'
GROUP BY r.rule_id, r.title, r.att_ck_technique
HAVING MAX(a.fired_at) < CURRENT_DATE - INTERVAL '90' DAY
    OR MAX(a.fired_at) IS NULL
ORDER BY last_fired NULLS FIRST;

Response playbook (detection-engineering-specific)

This playbook is for the detection engineer when a new true positive or a detection-quality incident occurs — it complements, but is distinct from, the SOC's incident-response runbook.

  1. Triage the firing — confirm the alert is a true positive using enrichment (asset owner, identity, prior alerts); if benign, disposition it and note the FP for tuning.
  2. Contain (hand off, don't freelance) — for a real intrusion, immediately hand the enriched alert + ATT&CK context to incident response; the detection engineer's job is signal quality, not owning the IR.
  3. Assess coverage gap — ask: did we catch this at the earliest possible technique in the kill chain, or only late (e.g., at impact/ransomware)? Log the earliest missed technique as a coverage gap.
  4. Author the new/improved detection — write a rule for the earlier technique with positive/negative fixtures, open a PR, get review, let CI validate.
  5. Eradicate the blind spot — deploy the new rule; run an atomic test (e.g., Atomic Red Team) to prove it fires end-to-end in production.
  6. Recover / tune the noisy neighbor — if the incident surfaced alert fatigue (real signal was buried), retune the offending high-volume rules and re-check precision SLOs.
  7. Retire the dead — if the incident revealed a rule that never fired due to broken parsing, either fix the parser (with a parser test) or retire the rule so the coverage map is honest.
  8. Post-incident metric update — record MTTD for this detection, update the coverage map, and add the technique to the purple-team backlog for periodic re-validation.

7. Practice Questions

Tier 1 — Fundamentals

  1. What is detection engineering and how is it different from just running a SIEM?
    Detection engineering treats detections as engineered software: they live in version control, get peer-reviewed, are validated by CI tests, and are measured for quality — versus a SIEM full of unowned vendor rules nobody tunes. The product is a precise, actionable signal plus its response runbook, not raw alert volume. Running a SIEM is buying a tool; detection engineering is the discipline that makes the tool produce trustworthy signal. Signal: does the candidate see detection as an ongoing engineering practice, not a product purchase.

  2. Explain the detection lifecycle.
    Hypothesis (a testable "attacker doing X leaves observable Y") → develop (write the rule) → test (positive/negative fixtures, ideally atomic red-team validation) → deploy (via reviewed, CI-gated pipeline) → tune (manage precision/recall in the real environment) → retire (remove stale or superseded rules). Retirement is a first-class step because dead rules are technical debt that pollute the coverage map and waste compute. Signal: candidate knows retirement exists and why it matters.

  3. What is the Pyramid of Pain and why does it matter?
    Bianco's model ranks indicators by how painful they are for an attacker to change: hashes (trivial) → IPs → domains → host/network artifacts → tools → TTPs (hardest). It matters because detections anchored to TTPs/behaviors survive re-tooling and infra rotation, so they raise attacker cost by months, while hash/IP detections die on the next recompile. A mature program keeps cheap IOC rules but spends engineering effort high on the pyramid. Signal: understands why TTP detection beats IOC detection.

  4. What's the difference between precision and recall in detection, and which failure does each cause?
    Precision = of the alerts that fired, how many were real (low precision → false positives → alert fatigue). Recall = of the real attacks, how many we caught (low recall → false negatives → coverage gaps / missed intrusions). You can't maximize both at once, so tuning is a deliberate per-rule trade-off. Signal: can reason about the core quality trade-off quantitatively.

  5. What is detection-as-code?
    Detections written in a version-controlled, machine-readable format (e.g., Sigma), changed only via pull requests, validated by CI (lint, compile-to-backend, unit tests against event fixtures), peer-reviewed, and deployed to the SIEM through an automated pipeline. It gives you reviewable, testable, revertable, auditable detections — the same rigor as application code. Signal: knows the Git/CI/review workflow, not just the phrase.

Tier 2 — System design and trade-offs

  1. Design a detection-as-code pipeline for an org with three different SIEMs.
    Author rules once in Sigma (backend-agnostic). CI runs sigma check for schema, then sigma convert to all three backends (Splunk, Elastic, Chronicle) to prove clean compilation, then unit tests against positive/negative fixtures. Branch protection requires a peer review + green CI. On merge, a deploy job pushes compiled rules to each SIEM via its API and runs a synthetic event to confirm the rule fires post-deploy. A coverage job diffs the ATT&CK map and blocks last-rule regressions. Meta-telemetry monitors deploy success and rule "last matched." The key design choice is a single source format to avoid maintaining three divergent rule sets. Signal: understands abstraction (Sigma), testing, and multi-backend deployment.

  2. Your SOC is drowning in alerts. Walk through how you'd fix it.
    First measure: per-rule alert volume and precision (instrument analyst dispositions). Rank rules by (volume × 1/precision) to find the worst offenders. For each, tune with scope, context enrichment, allowlists, or aggregation (alert on Nth occurrence). Set precision SLOs and an FP budget per severity, and auto-quarantine rules that blow the budget until fixed. Simultaneously improve routing: enrich with asset criticality and identity so high-value alerts surface above noise. The trap to avoid is blanket-suppressing — every allowlist is a potential blind spot, so tune per-rule with the recall cost in mind. Signal: methodical, metric-driven, aware of the precision/recall trade-off.

  3. How would you measure and communicate detection coverage to leadership?
    Map every detection to ATT&CK techniques, render a Navigator heatmap scored by rule count and confidence, and overlay the techniques of threat actors relevant to our sector (from ATT&CK group data) so priorities are threat-informed. Be explicit that the map measures presence, not efficacy — pair it with purple-team/atomic validation results showing which rules actually fire. Report trend over time (techniques covered, gaps closed) plus health metrics (MTTD, precision). This turns "are we secure?" into a diffable, prioritizable artifact leadership can fund against. Signal: knows coverage maps can be gamed and must be paired with validation.

  4. When would you deliberately deploy a low-precision detection?
    When the cost of a miss vastly exceeds the cost of false positives — e.g., a ransomware mass-encryption canary or a domain-controller DCSync (T1003.006) detection on a crown-jewel asset. There, low recall is unacceptable, so you tolerate noise and manage it with tight scoping to critical assets and strong enrichment for fast triage. Conversely, a broad discovery rule across the whole fleet must be high-precision or it'll bury the SOC. The decision is per-rule and severity-driven, not a global policy. Signal: understands that precision targets are contextual, not universal.

  5. How do you decide what telemetry to onboard first with a limited budget?
    Optimize ATT&CK-coverage-per-dollar: process/EDR telemetry (Sysmon Event 1, command lines, process lineage) unlocks the most execution/injection/persistence techniques and enables durable behavioral detection, so it's usually first. Identity/SSO logs are high-leverage for credential and cloud-account attacks with modest volume. Then cloud audit logs, then network (Zeek). Tier data hot/warm/cold to control cost, and maintain an inventory of which techniques are even possible to detect given current sources so gaps are explicit. Signal: connects telemetry choices to coverage and cost, not "log everything."

Tier 3 — Expert / deep-dive

  1. Attackers increasingly use LOLBins and signed native tools. How does that change your detection strategy?
    Name-based and hash-based rules are useless against living-off-the-land, so you shift to behavioral analytics: parent-child anomalies (w3wp.exe spawning whoami), command-line semantics (encoded PowerShell, suspicious flags to certutil/bitsadmin), and sequence correlation (recon → credential access → lateral movement within a window). You baseline what each process/host normally does and alert on first-seen behavior. This demands richer telemetry (command lines, module loads) and tolerates more FPs, so it must be paired with strong enrichment and precision management. The strategic frame is the Pyramid of Pain — force the attacker up to changing TTPs. Signal: connects evasion trends to behavioral detection and telemetry requirements.

  2. How do you validate that a detection actually works, beyond unit tests?
    Unit tests prove logic against fixtures but not end-to-end pipeline reality. Add atomic testing (Atomic Red Team) that executes the real technique in a controlled host and confirms the alert fires through collection → normalization → evaluation → queue. Run purple-team exercises where red executes a full chain and blue measures which techniques were detected, at what stage, and with what MTTD. Track detection efficacy over time and feed misses back as coverage gaps. Validation closes the gap between "the rule compiles" and "the rule catches the adversary in production." Signal: distinguishes unit correctness from end-to-end efficacy; knows atomic/purple methods.

  3. Design a detection maturity model and place a typical org on it.
    Level 0: ad-hoc, vendor defaults, no ownership. Level 1: rules owned but hand-edited in consoles, no tests, IOC-heavy. Level 2: detection-as-code — Git, peer review, CI unit tests, Sigma abstraction. Level 3: coverage measured against ATT&CK, precision/recall SLOs, meta-telemetry on rule health. Level 4: continuous validation (atomic/purple), threat-informed prioritization, automated tuning, retirement discipline, MTTD tracked and improving. Most orgs sit between 1 and 2 — they have rules but no tests, no coverage map, and no retirement process. The jump from 1→2 (detection-as-code) yields the biggest reliability gain. Signal: has a structured maturity mental model and honest calibration.

  4. What metrics prove a detection program is effective, and how can each be gamed?
    MTTD (median time to detect) — gamed by only counting detections you already have. Precision/FP rate — gamed by suppressing hard-to-tune rules (raises precision, hides recall loss). Alert volume — gamed by muting rather than fixing. Coverage (% ATT&CK) — gamed by weak "checkbox" rules. True/false positive counts — gamed by mis-dispositioning. The defense is triangulation: pair coverage with validation results, pair precision with recall estimates from purple-team, and track trends not point values. No single metric is trustworthy alone; a mature dashboard shows the tension between them. Signal: sophisticated, skeptical view of metrics and Goodhart's law.

  5. How do you handle a 0-day like Log4Shell with detection-as-code without breaking production?
    Immediately write a rule for the exploitation signature (${jndi: in headers/params) and post-exploitation behavior (Java/web process spawning shells or network utilities — more durable). Add positive fixtures from the public PoC and negative fixtures from your own benign traffic so you don't nuke the SOC. Run CI to compile to all backends and validate; get an expedited peer review; deploy via the pipeline with a post-deploy synthetic test. Use a governed emergency fast-path (deploy-now, PR-within-24h) if truly urgent, but never let it become the default. The whole value proposition of detection-as-code is shipping a tested org-wide detection in minutes, not a panicked console edit that overwhelms analysts. Signal: velocity + safety together; understands durable vs. brittle detection under time pressure.

9. Key Takeaways

  • Detection engineering is detections-as-software: version-controlled, peer-reviewed, CI-tested, measured, and owned — the unit of value is a precise, actionable signal plus its runbook, not alert count.
  • The lifecycle is hypothesis → develop → test → deploy → tune → retire; retirement and tuning are first-class, because stale and noisy rules are the most common real-world failure.
  • The Pyramid of Pain is the strategic compass: invest engineering effort in behavioral/TTP detections (Bianco 4–6) that raise attacker cost by months, while keeping cheap IOC rules for commodity threats.
  • Signal quality = precision (fatigue) vs. recall (blind spots); you tune the trade-off per rule and per severity, encoded as reviewable diffs, never a one-time event.
  • Detection-as-code with CI + peer review is the load-bearing control — positive/negative test fixtures pin behavior so schema changes fail the build instead of silently breaking rules.
  • Coverage measurement against MITRE ATT&CK turns "are we secure?" into a diffable heatmap — but the map measures presence, not efficacy, so pair it with atomic/purple-team validation.
  • You can only detect what you collect: telemetry onboarding (with parser tests and immutable off-host storage) bounds the entire program; optimize coverage-per-dollar, don't "log everything."
  • Metrics (MTTD, precision/FP rate, alert volume, coverage, time-to-tune) are all gameable — triangulate them and track trends, per Goodhart's law.
  • Real incidents (Target, SolarWinds, Log4Shell) show the pattern: detecting isn't enough (need runbooks + response), IOC-only loses to novel TTPs, and detection velocity is a defensive capability.
  • See sibling notes: MITRE ATT&CK for the adversary model, Sigma and Detection-as-Code for the rule language and pipeline mechanics, and Threat Hunting for the hypothesis-generation front end of the lifecycle.
