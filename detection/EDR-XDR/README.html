EDR and XDR — Endpoint Detection and Response

Category: Detection Engineering — Endpoint Telemetry, Behavioral Detection, Response Automation
Related notes: Detection Engineering · IDS and SIEM · Anomaly and Behaviour Detection · YARA and Snort Rules
Frameworks referenced: MITRE ATT&CK (Enterprise), MITRE Engenuity ATT&CK Evaluations, MITRE D3FEND, NIST SP 800-61r2, Sysmon config schema

1. Overview

Endpoint Detection and Response (EDR) is a class of security tooling that instruments hosts — laptops, servers, VMs, containers — with a sensor that continuously streams high-fidelity behavioral telemetry (process creation, image loads, network connections, registry and file changes, script content) to a backend where detection logic evaluates it, and that can take active response on the host (kill a process, quarantine a file, isolate the machine from the network). EDR exists because signature-based antivirus (AV) fundamentally loses to attackers who recompile malware, use fileless techniques, or abuse legitimate binaries already on the host. The center of gravity for modern intrusion detection has shifted from the network perimeter to the endpoint, because that is where code actually executes, where credentials live in memory, and where the richest, least-forgeable evidence of attacker behavior can be observed. XDR (Extended Detection and Response) is the natural extension: it correlates that endpoint telemetry with network, cloud, email, and identity signals into a single incident graph, so that a suspicious login, a phishing email, an EDR process alert, and an anomalous cloud API call get stitched into one story instead of four disconnected alerts.

At scale you are not buying a product and forgetting it. You are running fleets of hundreds of thousands to millions of endpoints, ingesting tens of billions of process events per day, and the hard problems are telemetry cost, sensor performance overhead, detection precision at that volume, and building detection content that survives attacker evasion. The you want to convey to know that you understand where the telemetry comes from at the OS level, why behavioral detection beats signatures, how attackers try to blind the sensor, and how defenders regain visibility.

TL;DR
- EDR = a host sensor that streams process/file/registry/network/script telemetry to a backend that does behavioral detection and can respond (kill/quarantine/isolate). It replaces the "detect known-bad file" model of AV with "detect malicious behavior sequences."
- On Windows the richest sources are the kernel process-creation callbacks (PsSetCreateProcessNotifyRoutineEx), ETW (including Microsoft-Windows-Threat-Intelligence), and AMSI for script content; on Linux it is eBPF (and legacy auditd), on macOS the EndpointSecurity framework.
- Behavioral detection (process ancestry, command-line patterns, ATT&CK TTPs) is durable; signatures/hashes sit at the bottom of the Pyramid of Pain and are trivially evaded.
- Living-off-the-land (LOLBins) and EDR unhooking/blinding are the two evasion categories that matter most — the defensive answer is telemetry the attacker cannot easily remove: kernel callbacks, ETW-TI, and tamper-protected sensors.
- XDR correlates endpoint + network + identity + cloud + email into one incident graph, raising detection confidence and cutting mean-time-to-respond.

2. How It Works

2.1 Core mechanics

An EDR deployment has three planes. The sensor/agent runs on every endpoint and is responsible for collecting telemetry, applying local (often ML) prevention models, enforcing response actions, and surviving tampering. The control plane is the cloud backend that stores telemetry, runs cloud-side detection analytics, distributes detection content and policy, and exposes the console/API. The data plane is the telemetry stream itself plus the response channel back down to the host. The critical design tension is that the sensor must see everything an attacker might do while consuming near-zero CPU and never bugchecking the kernel — because a security agent that blue-screens a million machines is a bigger outage than most attacks.

The heart of an EDR sensor is a set of OS-provided instrumentation hooks. On Windows, the sensor registers kernel callbacks so the OS notifies it synchronously whenever a process is created, a thread is created, an image (DLL/EXE) is loaded, a handle to a process is opened, or the registry is modified. This is fundamentally different from old AV, which scanned files at rest. EDR watches actions as they happen and reconstructs the process tree — the parent/child/grandparent ancestry — which is the single most valuable primitive in endpoint detection.

Evaluation/decision logic. A raw event arriving at the sensor is enriched (resolve the parent process, hash the image, tag the signer), evaluated against on-sensor prevention models (block-if-malicious), then forwarded to the cloud where streaming analytics evaluate it against detection rules and cross-host correlation. A detection can fire from a single event (e.g. vssadmin delete shadows) or from a sequence across time and processes (e.g. credential access followed by lateral movement followed by data staging). When a detection crosses a confidence/severity threshold, it either auto-responds (policy-driven) or raises an alert for a SOC analyst who can trigger response from the console.

2.2 Concrete configuration

Sysmon (System Monitor from Microsoft Sysinternals) is the canonical free way to understand the shape of EDR telemetry — commercial EDRs collect a superset of the same event types via their own kernel driver. A hardened Sysmon config (SwiftOnSecurity / Olaf Hartong sysmon-modular style) shows exactly what an EDR watches. Here is a focused excerpt that captures the highest-value events:

<Sysmon schemaversion="4.90">
  <EventFiltering>
    <!-- Event ID 1: Process creation — the backbone of ancestry detection -->
    <RuleGroup groupRelation="or">
      <ProcessCreate onmatch="include">
        <!-- Office apps spawning shells = classic macro/phishing execution (T1059) -->
        <ParentImage condition="end with">winword.exe</ParentImage>
        <ParentImage condition="end with">excel.exe</ParentImage>
        <ParentImage condition="end with">outlook.exe</ParentImage>
        <!-- LOLBins frequently abused for execution/download -->
        <Image condition="end with">powershell.exe</Image>
        <Image condition="end with">mshta.exe</Image>
        <Image condition="end with">rundll32.exe</Image>
        <Image condition="end with">regsvr32.exe</Image>
        <Image condition="end with">wmic.exe</Image>
      </ProcessCreate>
    </RuleGroup>

    <!-- Event ID 7: Image/DLL load — catches DLL search-order hijack, unsigned loads -->
    <RuleGroup groupRelation="or">
      <ImageLoad onmatch="include">
        <Signed condition="is">false</Signed>
        <ImageLoaded condition="end with">amsi.dll</ImageLoaded>
      </ImageLoad>
    </RuleGroup>

    <!-- Event ID 8: CreateRemoteThread — process injection (T1055) -->
    <CreateRemoteThread onmatch="include">
      <SourceImage condition="is not">C:\Windows\System32\wbem\WmiPrvSE.exe</SourceImage>
    </CreateRemoteThread>

    <!-- Event ID 10: ProcessAccess — LSASS handle opens = credential dumping (T1003.001) -->
    <RuleGroup groupRelation="and">
      <ProcessAccess onmatch="include">
        <TargetImage condition="end with">lsass.exe</TargetImage>
        <GrantedAccess condition="is">0x1410</GrantedAccess>
      </ProcessAccess>
    </RuleGroup>

    <!-- Event ID 13: Registry value set — persistence via Run keys (T1547.001) -->
    <RegistryEvent onmatch="include">
      <TargetObject condition="contains">\CurrentVersion\Run</TargetObject>
      <TargetObject condition="contains">\CurrentVersion\RunOnce</TargetObject>
    </RegistryEvent>

    <!-- Event ID 22: DNS query — C2/DGA hunting -->
    <DnsQuery onmatch="exclude">
      <QueryName condition="end with">.microsoft.com</QueryName>
    </DnsQuery>
  </EventFiltering>
</Sysmon>

Every commercial EDR ships an equivalent set of collectors. The point of the config above is the include/exclude discipline: you cannot afford to ship every event from every host, so you filter to the events that carry detection value and exclude known-benign high-volume noise. Event IDs 1 (ProcessCreate), 7 (ImageLoad), 8 (CreateRemoteThread), 10 (ProcessAccess — critically, LSASS handle opens), 11 (FileCreate), 13 (RegistryEvent), and 22 (DnsQuery) are the workhorses.

On Linux, the modern collection path is eBPF — a sandboxed VM in the kernel that lets a sensor attach programs to tracepoints, kprobes, and LSM hooks without shipping a fragile out-of-tree kernel module. Tools like Falco, Tetragon, and the eBPF-based sensors inside CrowdStrike Falcon and SentinelOne use it. A Falco rule expressing "a shell was spawned inside a container" (a strong container-escape / RCE signal) looks like:

- rule: Terminal shell in container
  desc: A shell was used as the entrypoint/exec target in a container
  condition: >
    spawned_process and container
    and shell_procs and proc.tty != 0
    and container_entrypoint
    and not user_expected_terminal_shell_in_container_conditions
  output: >
    A shell was spawned in a container (user=%user.name container=%container.name
    shell=%proc.name parent=%proc.pname cmdline=%proc.cmdline)
  priority: NOTICE
  tags: [container, shell, mitre_execution, T1059]

The conceptual eBPF program a sensor attaches to catch every process exec is a tracepoint/sched/sched_process_exec (or security_bprt_check LSM) hook that emits pid, ppid, uid, comm, argv, filename into a ring buffer for user space — the Linux equivalent of Sysmon Event ID 1. Legacy Linux hosts fall back to the kernel audit subsystem (auditd) with rules like:

# Catch execve syscalls (process creation) — the auditd equivalent of Sysmon EID 1
-a always,exit -F arch=b64 -S execve -k proc_exec
# Watch for reads of the shadow file (credential access)
-w /etc/shadow -p r -k cred_access
# Watch for changes to sshd config (persistence / defense evasion)
-w /etc/ssh/sshd_config -p wa -k sshd_config_change

auditd is heavy and easy to overload; eBPF is the strategic direction because it is lower-overhead, richer, and harder for an attacker to silence than a userland auditd daemon.

Response actions via API — most EDRs expose containment through their console and a REST API. Conceptually (Microsoft Defender for Endpoint machine-isolation call shape):

POST https://api.securitycenter.microsoft.com/api/machines/{machineId}/isolate
Authorization: Bearer {token}
Content-Type: application/json

{ "Comment": "IR-4471 confirmed Cobalt Strike beacon", "IsolationType": "Full" }

Full isolation drops all network connectivity except the EDR's own management channel — the host is contained but still controllable, which is exactly what you want during eradication.

Case study — SolarWinds SUNBURST (2020). Attackers compromised the SolarWinds Orion build pipeline and shipped a digitally signed, trojanized SolarWinds.Orion.Core.BusinessLayer.dll. Because the DLL was legitimately signed and delivered through the trusted update mechanism, every file-signature and reputation control passed it. The malware slept up to two weeks, then resolved DGA subdomains under avsvmcloud.com for C2. The only durable detection surfaces were behavioral and correlative: the Orion process making anomalous outbound DNS/HTTP to newly seen infrastructure, and later hands-on-keyboard activity. This is the textbook argument for XDR — endpoint alone might see the odd child process, network alone sees the odd beacon, but correlating "trusted app + novel C2 + lateral movement" is what surfaces the intrusion. FireEye discovered it in part by noticing an unexpected second device enrolled for MFA — an identity signal — which is precisely the cross-domain stitch XDR is built to make.

Case study — CrowdStrike channel file 291 (July 2024). Not an attack, but the most important EDR incident of the decade for a senior engineer to reason about. A routine "Rapid Response Content" update (a detection-content channel file, not a sensor code update) contained a mismatch that caused the sensor's content interpreter — running in the kernel-adjacent sensor — to perform an out-of-bounds read and bugcheck the host, blue-screening roughly 8.5 million Windows machines worldwide and grounding airlines and hospitals. The lesson for anyone deploying EDR at scale: the sensor's privilege (needed to be un-evadable) is exactly what makes it a fleet-wide availability risk. You must treat detection-content pushes with the same canary/staged-rollout rigor as code, and demand that the vendor supports gradual deployment rings. Microsoft's post-incident push toward moving sensor logic out of the kernel (user-mode sensors with a thin kernel component) is a direct response.

Case study — NotPetya (2017). After initial access via a trojanized M.E.Doc update, NotPetya used an embedded Mimikatz-like module to scrape credentials from LSASS memory, then propagated with PsExec and WMIC to remote hosts and via EternalBlue. Every one of those steps is a canonical EDR behavioral detection: a non-security process opening a 0x1410 handle to lsass.exe (Sysmon EID 10), psexesvc.exe service creation, and wmic /node: remote execution. Signature AV that only knew the ransomware payload was too late; behavioral EDR watching credential access and lateral movement would fire before mass encryption.

Top 5 in depth

1. Behavioral / process-ancestry detection (counters LOLBins, fileless, injection). The technical core is reconstructing the full process tree from kernel process-creation events and writing detections against relationships and attributes rather than file identity. winword.exe spawning powershell.exe -enc <base64> is anomalous regardless of what the PowerShell binary hashes to, because Word legitimately never needs a shell. Implementation is a streaming rule over the process-create stream keyed on ParentImage, Image, and CommandLine (see the sample rule in §6). This is the single most durable EDR capability because it sits at the TTP level of the Pyramid of Pain — attackers can change binaries and hashes cheaply but cannot easily change the fact that their execution chain looks nothing like normal user behavior. Trade-off: ancestry rules generate false positives from legitimate admin/IT automation and require environment-specific tuning and allowlists; PPID spoofing (T1134.004) can also fool naive implementations, so you must use the sensor's kernel-truth lineage rather than the attacker-reportable parent PID.

2. Tamper protection + kernel-truth telemetry (counters unhooking, ETW patching, sensor kill). Because the whole value proposition is that the attacker cannot blind the sensor, the sensor must collect from sources the attacker cannot remove from user land. Kernel process-creation/image-load callbacks and Microsoft's protected Microsoft-Windows-Threat-Intelligence (ETW-TI) provider — which only Protected-Process-Light-signed anti-malware processes can consume — see events even when a process has patched its own ntdll hooks or disabled ordinary ETW. Tamper protection is kernel-enforced: attempts to stop the service, unload the driver, or delete the agent are blocked and alerted. Trade-off: this requires the sensor to run at high privilege (kernel or PPL), which is exactly the availability risk the CrowdStrike 2024 outage crystallized — more power to resist evasion means more blast radius when the sensor itself misbehaves. The mitigation is staged content rollout and moving parsing logic out of the kernel.

3. Credential-access hardening + detection (counters LSASS dumping). LSASS is the crown jewel because it caches credentials and Kerberos tickets. Two layers: prevent — enable Windows Defender Credential Guard (isolates LSASS secrets in a VBS/hypervisor-protected process) and run LSASS as a Protected Process Light (RunAsPPL) so ordinary processes cannot open a read handle; detect — alert on any non-allowlisted process opening a handle to lsass.exe with access mask containing 0x0010 (VM_READ), e.g. the 0x1410 mask used by Mimikatz sekurlsa::logonpasswords. Implementation is Sysmon EID 10 / EDR ProcessAccess telemetry filtered to TargetImage end with lsass.exe. Trade-off: legitimate tools (some AV, Windows Error Reporting, procdump for debugging) also open LSASS, so you need a maintained allowlist, and Credential Guard has compatibility constraints with some drivers and older auth protocols.

4. Application control / allowlisting (counters LOLBins, unsigned/unknown binaries). WDAC (Windows Defender Application Control) and AppLocker shift from "block known-bad" to "only run known-good," which structurally defeats novel malware and constrains LOLBin abuse (you can, for example, block mshta.exe and wscript.exe entirely if the business does not need them). WDAC policy is code-integrity enforced and can be signed so even an admin cannot trivially disable it. Trade-off: allowlisting is operationally expensive — it breaks legitimate software updates and developer workflows, requires a managed exception process, and is realistically deployable in "audit" mode first for months before enforcement. It is a control you phase in on high-value fixed-function servers (domain controllers, jump hosts) before general fleets.

5. XDR correlation + risk-based scoring (counters alert fatigue, encrypted C2, coverage gaps). A single medium-severity EDR alert (unusual PowerShell) is often not worth waking an analyst; the same alert correlated with a risky new sign-in for that user, a phishing email delivered 20 minutes earlier, and an anomalous cloud API call becomes a high-confidence incident. XDR ingests endpoint, identity, email, network, and cloud signals into a shared entity/incident graph and scores by aggregate risk, collapsing dozens of alerts into one investigable incident and enabling automated response only when confidence is high. Implementation is entity resolution (map the same user/host/IP across sources) plus a correlation engine (SIEM/XDR analytics, e.g. Microsoft Sentinel fusion, Chronicle). Trade-off: correlation quality depends entirely on data normalization and clock/identity hygiene across sources; poor entity resolution produces either missed stitches or wrong ones, and cross-domain data centralization concentrates a lot of sensitive telemetry that itself becomes a target.

Detection 1 — Office application spawning a command interpreter (Sigma, maps to T1059 / T1566 macro execution).

title: Office Application Spawning Command Shell or Script Host
id: 7b1f0a2e-office-spawn-shell
status: stable
description: Detects Word/Excel/Outlook spawning cmd, powershell, wscript, mshta, or rundll32 - classic phishing macro execution
logsource:
  category: process_creation
  product: windows
detection:
  selection_parent:
    ParentImage|endswith:
      - '\winword.exe'
      - '\excel.exe'
      - '\powerpnt.exe'
      - '\outlook.exe'
  selection_child:
    Image|endswith:
      - '\cmd.exe'
      - '\powershell.exe'
      - '\pwsh.exe'
      - '\wscript.exe'
      - '\cscript.exe'
      - '\mshta.exe'
      - '\rundll32.exe'
  filter_known_good:
    CommandLine|contains:
      - '\Program Files\CorpAddin\update.ps1'   # tune per environment
  condition: selection_parent and selection_child and not filter_known_good

  - Approved Office add-ins or IT automation launching scripts

DeviceEvents
| where ActionType == "OpenProcessApiCall"
| where InitiatingProcessFileName !in~ ("MsMpEng.exe","csrss.exe","wininit.exe","lsass.exe","procexp64.exe")
| extend Target = tostring(parse_json(AdditionalFields).TargetFileName)
| where Target endswith "lsass.exe"
| extend Access = tostring(parse_json(AdditionalFields).DesiredAccess)
// 0x1010 / 0x1410 include PROCESS_VM_READ (0x10) - required to scrape credentials
| where Access has_any ("0x1010","0x1410","0x143a","0x1438")
| project Timestamp, DeviceName, InitiatingProcessFileName,
          InitiatingProcessCommandLine, Target, Access, AccountName
| sort by Timestamp desc

Detection 3 — Sensor/telemetry gap (SQL-style, catches evasion via §4 #10/#11). A blind spot is itself a detection — alert when a host that normally reports stops, which flags sensor kills and tampering:

SELECT host_id, MAX(event_time) AS last_seen
FROM edr_telemetry
WHERE event_time > NOW() - INTERVAL '24 hours'
GROUP BY host_id
HAVING MAX(event_time) < NOW() - INTERVAL '45 minutes'   -- expected heartbeat window
   AND host_id IN (SELECT host_id FROM managed_assets WHERE status = 'active');

Response playbook

  1. Validate & scope (triage). Confirm the alert is a true positive; pull the full process tree, command lines, network connections, and signed status for the host and the timeframe. Identify the user, initial access vector, and whether ancestry indicates hands-on-keyboard.
  2. Contain — isolate the host. Trigger EDR network isolation (Full) so the host keeps only the management channel. This stops lateral movement and C2 while preserving the machine for forensics. Do this before eradication, not after.
  3. Contain — kill and quarantine. Terminate the malicious process tree and quarantine dropped files/binaries. Capture a memory image first if the threat is fileless, because killing the process destroys memory-only evidence.
  4. Contain — identity. If credential access (LSASS) occurred, treat all creds on the host as compromised: force-reset the user and any service accounts, revoke sessions/tokens, and rotate Kerberos (twice for krbtgt if a DC was touched). This is where XDR identity correlation is essential.
  5. Eradicate. Remove persistence (Run keys, scheduled tasks, services, WMI subscriptions), remove the implant, and confirm no secondary footholds on correlated hosts surfaced by the XDR incident graph.
  6. Recover. Restore isolation only after the host is verified clean or reimaged; reimage is the safe default for confirmed intrusions. Restore data from known-good backups if encryption/tampering occurred.
  7. Post-incident / detection engineering. Convert the observed TTPs into new/updated detections, add environment-specific allowlist tuning, feed IOCs to threat intel, and run a purple-team replay to confirm the new detection fires. Update the runbook. (See Detection Engineering for the tuning loop.)

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is the difference between traditional antivirus and EDR?
Traditional AV is primarily signature- and reputation-based: it scans files at rest or on execution and blocks known-bad hashes or byte patterns. EDR continuously collects behavioral telemetry (process creation, image loads, network, registry, script content), reconstructs process ancestry, and detects malicious behavior sequences that map to attacker TTPs — then it can respond (kill, quarantine, isolate) and retains telemetry for hunting and forensics. AV asks "is this file known-bad?"; EDR asks "does this activity look like an attack?" EDR is durable against recompiled and fileless malware that trivially defeats signatures.
Why asked / signal: Baseline understanding that detection has moved from artifacts to behavior.

Q2. Name the highest-value telemetry sources an EDR collects on Windows and what each catches.
Process creation (parent/child ancestry, command line — catches LOLBins and phishing execution), image/DLL loads (unsigned loads, DLL hijack, AMSI tamper), CreateRemoteThread and memory allocation (process injection), process-handle access to LSASS (credential dumping), registry writes to Run keys and services (persistence), DNS queries (C2/DGA), plus ETW — especially the protected Threat-Intelligence provider — and AMSI for de-obfuscated script content. Sysmon exposes the same event families for free (EIDs 1, 7, 8, 10, 13, 22).
Why asked / signal: Do you know where the data actually comes from, not just product names.

Q3. What is process ancestry and why is it central to endpoint detection?
Process ancestry is the parent→child→grandchild lineage of every running process, reconstructed from kernel process-creation events. It matters because individual binaries are often benign (PowerShell, cmd, rundll32 are all signed OS tools) but their lineage betrays intent — Word spawning PowerShell that spawns a network connection is malicious regardless of file hashes. Ancestry detection sits at the TTP level of the Pyramid of Pain, so it is durable against attackers who cheaply change hashes and file names.
Why asked / signal: Understanding of the single most important EDR primitive.

Q4. What is "living off the land" and why does it challenge detection?
Living off the land (LOTL) means the attacker achieves objectives using tools already present and trusted on the host — signed Windows binaries (LOLBins like rundll32, mshta, regsvr32, certutil), PowerShell, WMI, and legitimate admin/RMM tools. It challenges detection because there is no malware file to signature and the binaries are trusted, so signature and allowlist controls pass them. The defensive answer is behavioral: detect anomalous usage (unusual parent, encoded arguments, unusual network egress) and, where possible, block unneeded LOLBins via application control.
Why asked / signal: Recognizes the dominant modern evasion class.

Q5. What response actions can an EDR take, and in what order during an incident?
Core actions: network-isolate the host (keeping only the EDR management channel), kill a process or process tree, quarantine a file, and block a hash/indicator across the fleet. Order matters: isolate first to stop lateral movement and C2, capture memory if the threat is fileless, then kill/quarantine, then address identity (reset compromised credentials). Isolation must be reversible and audited, and response should be RBAC- and MFA-gated because mass isolation can itself be a denial of service.
Why asked / signal: Practical containment reasoning, not just detection.

Tier 2 — System design / trade-offs

Q6. Design the telemetry pipeline for an EDR fleet of one million endpoints. What are the hard trade-offs?
Each endpoint runs a sensor that collects, enriches, and filters events locally, then ships them over a backpressure-aware channel to a regional ingest tier, into a streaming detection layer (rules + correlation) fronting a hot analytical store (days) and a cheaper cold store (months) for hunting/forensics. The dominant trade-offs are cost vs visibility (you cannot ship every event from every host, so you filter aggressively and risk blind spots), and sensor overhead vs fidelity (more hooks = more CPU and more risk of destabilizing the host). You need per-host sampling awareness, gap detection (a host that stops reporting is an alert), schema versioning, and graceful degradation under load. I would keep raw process-create events near-complete (they are the backbone) and sample high-volume low-value events. Detection content must be staged/canaried like code — the CrowdStrike 2024 outage proved content pushes are as dangerous as code pushes.
Why asked / signal: Scale reasoning, cost/visibility trade-offs, and the availability risk of the sensor.

Q7. Kernel-mode vs user-mode sensors — what are the trade-offs and where is the industry going?
Kernel-mode (driver + callbacks/minifilter) sees events user-mode malware cannot easily hide, is much harder to unhook, and can enforce tamper protection — but a bug bugchecks the whole OS (fleet-wide availability risk) and requires signed-driver trust. User-mode (inline API hooks) captures rich arguments and is safe to crash, but is removable by the very process it monitors (unhooking, direct syscalls). Best practice is a hybrid: authoritative event capture via kernel callbacks + ETW-TI for un-evadable ground truth, with user-mode components for enrichment. After 2024, Microsoft is pushing anti-malware vendors toward a model with a thin kernel component and most logic in user mode, to cut the blast radius while keeping kernel-level visibility for the un-evadable events.
Why asked / signal: Deep understanding of the core architectural tension and current industry direction.

Q8. What is the difference between EDR and XDR, and when does XDR actually add value?
EDR is endpoint-scoped: telemetry, detection, and response on hosts. XDR extends the same detect-correlate-respond loop across domains — endpoint + network + identity + cloud + email — into a shared entity/incident graph. XDR adds value specifically when signals are individually low-confidence but jointly damning: an unusual PowerShell alone may not warrant response, but combined with a risky new sign-in, a phishing email, and an anomalous cloud API call it becomes a high-confidence incident. It also cuts alert fatigue by collapsing many alerts into one incident and enables safer automation because confidence is higher. The catch is that XDR is only as good as entity resolution and data normalization across sources; poor identity/clock hygiene produces missed or wrong correlations.
Why asked / signal: Can articulate the correlation value prop and its dependency on data quality.

Q9. How would you tune an EDR deployment to minimize false positives without creating blind spots?
Start from risk-based, environment-specific tuning rather than blanket suppression. Baseline what "normal" looks like per role (developer workstations legitimately spawn shells; a finance user's Word spawning PowerShell does not), scope allowlists tightly (specific signed binary + path + parent, not "ignore all PowerShell"), and prefer risk scoring/correlation over single-rule alerting so one noisy rule does not page anyone alone. Measure precision per detection and track it over time; retire or fix low-precision rules. Critically, distinguish suppression from blindness — every allowlist entry is documented, time-bounded where possible, and reviewed, and you add gap detection so a suppressed or dead sensor still triggers an alert. Purple-team replays validate that tuning did not silence a real detection.
Why asked / signal: Detection-ops maturity; understands the precision/coverage balance.

Q10. macOS, Windows, and Linux — how does EDR telemetry collection differ across them, and why does it matter?
Windows offers the richest native instrumentation: kernel process/image/registry callbacks, ETW (including the protected Threat-Intelligence provider), and AMSI for script content. Linux has no single equivalent; modern sensors use eBPF (attaching to tracepoints/kprobes/LSM hooks) for low-overhead, rich, tamper-resistant collection, falling back to the audit subsystem on old kernels — the challenge is kernel-version fragmentation and ephemeral containers. macOS deliberately restricts what security tools can do: Apple deprecated kernel extensions in favor of the user-space EndpointSecurity framework and Network Extensions, which is more stable but gives vendors less latitude and different event coverage. It matters because your detection content and coverage assumptions must be OS-specific — a Windows LSASS rule has no Linux analog, and container/cloud workloads need eBPF-based runtime sensors entirely.
Why asked / signal: Cross-platform depth; awareness of OS-imposed constraints.

Tier 3 — Expert / deep dive

Q11. An attacker unhooks your EDR's user-mode hooks and uses direct syscalls. How do you still detect them?
User-mode inline hooks in ntdll are removable by any process that can write its own memory (unhooking) and are bypassed entirely by direct/indirect syscalls that invoke syscall without going through the hooked stubs. The defensive answer is telemetry the attacker cannot reach from user land: kernel process/thread/image callbacks (PsSetCreateProcessNotifyRoutineEx, etc.) still fire, and Microsoft's ETW Threat-Intelligence provider emits kernel-side events for memory allocation/protection changes, remote thread creation, and other injection primitives that direct syscalls still trigger. You also detect the act of unhooking (a process writing to ntdll's .text section, or reloading a clean copy of ntdll from disk) and anomalous call stacks that originate from unbacked/RWX memory. Defense in depth means no single removable hook is your only sensor.
Why asked / signal: Deep evasion/counter-evasion knowledge — separates senior from staff.

Q12. Walk through detecting a Cobalt Strike beacon on an endpoint using EDR telemetry, without a payload signature.
Cobalt Strike's default behaviors leave endpoint tells even when the payload is obfuscated. On process behavior: beacons commonly inject into or spawn a sacrificial process and use named-pipe C2 (\\.\pipe\msagent_*, \postex_*) for peer-to-peer — named-pipe creation with default-ish patterns is a strong signal. Memory: the beacon runs from RWX or unbacked memory, so a call stack originating outside any loaded module (detectable via ETW-TI/thread-start telemetry) is suspicious. Network: default beacons sleep with jitter and beacon to CDN-fronted HTTPS; the endpoint sees periodic outbound from an unusual process (e.g. rundll32 with no arguments making network connections — a classic anomaly). Ancestry: spawnto defaults like rundll32.exe/dllhost.exe spawned by an unexpected parent with no command line. You correlate these — injection + named pipe + periodic egress + weird ancestry — rather than relying on one, which is exactly why behavioral EDR and XDR beat signatures here.
Why asked / signal: Concrete, tool-specific behavioral detection reasoning.

Q13. Critique the security-vs-availability trade-off exposed by the July 2024 CrowdStrike outage. How would you architect around it?
The outage came from a Rapid Response content update (not sensor code) that the kernel-adjacent content interpreter mishandled, causing an out-of-bounds read and a bugcheck across ~8.5M hosts. The root tension: the sensor must run at high privilege to be un-evadable, but that same privilege makes any defect a fleet-wide availability event — the security control becomes a bigger outage source than the threats it stops. Architecturally I would (a) treat detection content as code: schema-validate and fuzz content, canary it through deployment rings, never push globally at once; (b) demand vendor-supported staged rollout and let customers control ring assignment; (c) move as much parsing/interpretation as possible out of the kernel into a crash-safe user-mode component, keeping only minimal, heavily verified logic kernel-side (the direction Microsoft is now driving); and (d) maintain out-of-band recovery for bootloop scenarios. The meta-lesson: measure and rehearse the availability failure mode of your security tooling, not just its detection efficacy.
Why asked / signal: Systems maturity — treating security tooling as an availability risk and reasoning about blast radius.

Q14. How do you detect an attacker disabling or blinding your EDR, and why is "detecting the blindness" hard?
Blinding takes several forms: stopping the service (T1562.001), unloading/tampering the driver, BYOVD to kill the sensor from kernel, disabling ETW providers, or patching AMSI. You detect it with (a) a heartbeat/gap model — a managed host that stops reporting within its expected window is alerted even though the "signal" is absence of data; (b) tamper-protection events (kernel-enforced blocks on service stop / driver unload); (c) driver-load telemetry cross-referenced against Microsoft's vulnerable-driver blocklist for BYOVD; and (d) detecting the commands themselves (Set-MpPreference -DisableRealtimeMonitoring, sc stop, fltmc unload). It is hard because absence-of-data detection is inherently noisy — laptops legitimately sleep, go offline, or roam — so you must distinguish benign disconnection from malicious silence using context (was there suspicious activity just before the host went dark?), and because a sufficiently privileged attacker who reaches kernel can race the sensor's own reporting.
Why asked / signal: Understanding that a blind spot is itself a detection surface, and the difficulty of absence-based detection.

Q15. Design a detection-engineering program on top of EDR data for a large org. What is the lifecycle and how do you keep content durable?
Anchor everything to MITRE ATT&CK for coverage mapping — track which techniques you have detections for and where the gaps are. The lifecycle: (1) hypothesis from threat intel / red-team / recent incidents, expressed as an ATT&CK technique; (2) build the detection against EDR telemetry, preferring behavioral/TTP-level logic over IOCs for durability; (3) validate with atomic tests (Atomic Red Team) and purple-team replays in a controlled environment to confirm it fires and measure precision; (4) deploy with a documented response runbook — no detection ships without a response procedure; (5) tune continuously, tracking precision per rule and retiring noisy ones; (6) maintain — re-test after sensor/OS updates because telemetry schemas change. Durability comes from writing detections at the TTP level (process ancestry, behavior sequences) rather than hashes/domains, versioning detections in a code repo with CI that runs atomic tests, and correlating via XDR so single weak signals combine. Measure success by dwell-time reduction and detection coverage, not alert count. (Cross-reference Detection Engineering.)
Why asked / signal: Can operationalize EDR data into a durable, measurable program — the staff-level differentiator.

9. Key Takeaways

  • EDR moved detection from artifacts (hashes, signatures) to behavior (process ancestry, TTP sequences); the latter is durable because it sits at the top of the Pyramid of Pain.
  • The value of EDR is un-evadable visibility — which is why kernel callbacks and ETW-TI (which malware cannot remove from user land) matter more than removable user-mode hooks.
  • Process ancestry + command-line telemetry is the single highest-value primitive; most high-fidelity detections (Office-spawns-shell, LSASS access, injection) are expressible against it.
  • Living-off-the-land and sensor blinding (unhooking, ETW patching, BYOVD) are the two evasion classes that matter most; the counters are behavioral detection, tamper protection, application control, and detecting the blindness itself.
  • The sensor's power is also its risk: kernel privilege makes it un-evadable but turns any defect into a fleet-wide outage — the CrowdStrike 2024 event proves content must be staged like code.
  • Telemetry differs sharply by OS: Windows (ETW/AMSI/callbacks), Linux (eBPF, then auditd), macOS (EndpointSecurity); detection content and coverage assumptions must be OS-specific.
  • XDR adds value by correlating individually weak signals (endpoint + identity + email + cloud + network) into high-confidence incidents, cutting alert fatigue and enabling safer automation — but only if entity resolution is good.
  • A blind spot is a detection surface: a managed host that goes silent should page just like a malicious process.
  • Response is a sequence — isolate first (stop lateral movement/C2), capture memory for fileless threats, then kill/quarantine, then rotate compromised credentials.
  • Every detection needs a response runbook and measured precision; detection engineering on EDR data is a continuous, ATT&CK-mapped, code-versioned discipline, not a product you buy.
