Honeypots and Canary Tokens

Category: Detection Engineering — Deception Technology

Overview

Honeypots and canary tokens are deception-based detection techniques. They work by placing fake assets — systems, credentials, files, tokens — that have no legitimate use. Any interaction with them is a high-confidence signal of malicious or unauthorized activity, because there is no benign reason to touch them.

Deception technology has an extremely low false positive rate compared to signature or anomaly detection. The tradeoff is coverage: honeypots only catch attackers who touch the decoy, not all attackers.

Honeypots

A honeypot is a system, service, or resource that mimics a real target but has no legitimate production use. Its sole purpose is to detect, log, and study unauthorized access.

Honeypot Types by Interaction Level

Type Description Use Case
Low-interaction Emulates a small set of services; limited attacker capability Network scanning detection, basic threat intel
Medium-interaction More realistic service emulation; some command execution Malware capture, credential harvesting detection
High-interaction Real OS/service; attacker can do nearly anything Detailed attacker behavior study; high risk of pivot
Pure honeypot Real system fully monitored; no emulation Research; rarely used in production

Honeypot Deployment Locations

flowchart TD
    Internet --> Perimeter[Perimeter Honeypot]
    Perimeter --> DMZ[DMZ Honeypot]
    DMZ --> Internal[Internal Network Honeypot]
    Internal --> Endpoint[Endpoint Honeypot File/Creds]
    Internal --> Database[Honey Database Table]
Location What It Detects
Internet-facing External scanning, exploitation attempts, credential stuffing
DMZ Pivot attempts from compromised edge systems
Internal network Lateral movement after initial compromise
Endpoint (honey files/creds) Attacker enumerating file shares or stealing credentials
Database (honey table) SQL injection, unauthorized data access, insider threat

What Honeypots Capture

  • Credentials: Attackers often try default or common passwords; captured creds reveal attacker tooling
  • Exploits: Packets and payloads used in exploitation attempts
  • Malware: Droppers and payloads attackers deploy once they get shell access
  • TTPs: Commands run, files accessed, lateral movement attempts
  • Source IPs: Useful for threat intel (though often VPN/TOR exit nodes)

Honeypot Risks

Risk Mitigation
Attacker pivots from honeypot to real systems Strong network isolation; honeypot on separate VLAN or air-gapped
Honeypot fingerprinting Attacker detects it's a honeypot and avoids it; change banner strings and response timing
Legal issues with capturing attacker data Consult legal; do not actively interfere with attacker systems
High-interaction honeypots used as attack launchpad Monitor outbound connections; rate-limit or block egress

Canary Tokens

Canary tokens are lightweight, zero-maintenance tripwires that trigger an alert when accessed. They require no infrastructure — a token is a URL, file, or credential that, when used, calls home to a monitoring service.

Types of Canary Tokens

Token Type How It Works Alert Trigger
URL token A URL embedded in a doc or web page HTTP GET request when loaded
DNS token A domain that resolves to a monitoring server DNS query triggers alert with source IP
Document token (Word/PDF) A Word or PDF with embedded tracking pixel Opened on any connected machine
AWS key token Fake AWS access key ID + secret Alert fires when key is used to call AWS API
Azure key token Fake Azure service principal credentials Alert fires on Azure auth attempt
Kubeconfig token Fake Kubernetes config with token Alert fires on API server call
Windows directory token Folder with tracking attribute Alert fires when folder is browsed
SQL canary table/row Row with embedded URL in a BLOB Alert fires when data is read by an exfil tool
SSH key token Fake private key that alerts on use Alert fires on SSH authentication attempt

Canarytokens.org

The simplest way to deploy canary tokens. Steps:

  1. Go to canarytokens.org (or self-host with the open-source project)
  2. Select token type, enter alert email
  3. Receive a unique token (URL, key, file, etc.)
  4. Place it in the environment (file share, email inbox, password manager, etc.)
  5. Receive email/webhook alert when token is triggered, including source IP and user agent

Placement Strategy

Asset Token Type What It Detects
File server share Word doc with URL token Attacker browsing network shares
Password manager / vault Fake AWS key or login credential Attacker reading stored secrets
Source code repo AWS key in comments Developer pushing real secrets; exfil of repo
Database SQL row with DNS token URL Data exfiltration queries
Email inbox URL token in email signature Attacker reading email
Employee workstation PDF with URL token (named "salary_2026.pdf") Attacker hunting valuable-looking files
Admin account Fake admin credential that alerts on use Credential theft and reuse

Honey Credentials

Honey credentials are fake usernames and passwords planted in likely credential-harvesting locations. They are useless for real authentication but alert when used.

Deployment locations:

  • Plain text files on file shares (e.g., passwords_backup.txt)
  • Browser saved passwords (low-value account with tracking)
  • Configuration files on developer machines
  • Source code history (detect accidental secret commits)

Deception at Scale

At large organisations, deception can be systematized:

Technique Scale Benefit
Auto-generate honey tokens in every new S3 bucket Immediate alert if bucket is misconfigured and accessed
Honey accounts in every Active Directory OU Detect AD enumeration and lateral movement across the org
Honey rows in every database table Detect data exfiltration from any database
Honey DNS entries Detect internal reconnaissance
Distributed honeypots per network segment Detect lateral movement within internal network
