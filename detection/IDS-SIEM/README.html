IDS and SIEM

Category: Detection Engineering — Intrusion Detection, Log Management

Intrusion Detection Systems (IDS)

An IDS monitors network traffic or host activity and raises alerts when it detects patterns associated with attacks. It is passive — it detects but does not block. An Intrusion Prevention System (IPS) extends IDS with inline blocking capability.

IDS Classification

Axis Type A Type B
Placement NIDS — Network IDS (monitors traffic on a network segment) HIDS — Host IDS (monitors activity on a single host)
Detection method Signature-based — matches known attack patterns Anomaly/behaviour-based — detects deviations from baseline

Signature-Based vs Anomaly-Based

Property Signature-Based Anomaly/Behaviour-Based
Detection of known attacks Excellent May miss if not trained on them
Detection of zero-days Poor — signature must exist first Good — detects deviation from normal
False positive rate Low when signatures are tuned Higher — normal variation triggers alerts
Maintenance Ongoing signature updates required Baseline must be kept current
Evasion Fragmentation, encoding, polymorphism Slow attacks that blend with baseline
Example tools Snort, Suricata Darktrace, ML-based UEBA tools

Common NIDS Tools

Tool Type Notes
Snort NIDS, signature-based Open source; rule language is widely used industry standard
Suricata NIDS, signature + anomaly Multi-threaded; supports same rule format as Snort; preferred for high throughput
Zeek (Bro) Network analysis framework Generates structured logs from traffic; not a classic IDS but widely used for detection
OSSEC HIDS Open source; monitors log files, file integrity, rootkit detection

HIDS Capabilities

Host-based IDS monitors:

  • File integrity monitoring (FIM): detects changes to system files, binaries, configuration
  • Log file monitoring: watches system and application logs for suspicious entries
  • Process monitoring: detects unexpected process execution
  • Rootkit detection: looks for hidden processes, open ports, and kernel anomalies
  • Registry monitoring (Windows): detects persistence via registry run keys

SIEM — Security Information and Event Management

A SIEM is the central hub of security operations. It:

  1. Collects logs from all sources (network, host, cloud, application)
  2. Normalises events into a common schema
  3. Correlates events across sources to identify attack patterns
  4. Stores events for compliance and forensic investigation
  5. Alerts on rules and anomalies
  6. Provides search and investigation capabilities
flowchart LR
    A[Network Devices] --> SIEM
    B[Servers and Endpoints] --> SIEM
    C[Cloud Platforms] --> SIEM

    SIEM --> E[Correlation Engine]
    E --> F[Alerts to SOC]

SIEM Use Cases

Use Case How SIEM Enables It
Brute force detection Correlate N failed logins from same IP within T seconds
Lateral movement Auth success after multiple auth failures across different hosts
Data exfiltration Large outbound transfer to new external destination
Privilege escalation User account added to admin group followed by sensitive file access
Impossible travel Auth from New York, then auth from Tokyo 30 minutes later
C2 beaconing Repeated DNS queries to same domain at regular intervals
New admin account created Alert on any addition to Domain Admins or root group

SIEM Correlation Rules — Writing Effective Rules

A correlation rule should express: "If I see X events of type A from the same source within time window T, create an alert."

Example Splunk SPL rule for brute force:

index=auth action=failure
| stats count by src_ip, user
| where count > 10
| eval alert_reason="Brute force: " + tostring(count) + " failures"

Example rule for impossible travel:

index=auth action=success
| sort by user, _time
| streamstats current=f last(_time) as prev_time last(src_ip) as prev_ip by user
| eval time_diff = _time - prev_time
| where time_diff < 3600 AND src_ip != prev_ip
| iplocation src_ip
| iplocation prev_ip prefix=prev_
| eval distance = haversine(lat, lon, prev_lat, prev_lon)
| where distance > 1000

SIEM Tuning

Alert fatigue is the biggest threat to an effective SOC. Tuning principles:

Technique Purpose
Allowlisting known-good Exclude IT automation accounts, known scanner IPs
Adjusting thresholds Raise count thresholds where baseline volume is higher
Suppression windows Suppress repeat alerts for same entity within short window
Risk scoring Aggregate multiple low-signal events into a risk score per entity
Asset criticality weighting Same event on DC = P1; same event on test host = P3
