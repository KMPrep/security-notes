1. Overview

MITRE ATT&CK (Adversarial Tactics, Techniques, and Common Knowledge) is a curated, evidence-based knowledge base of real-world adversary behaviour, organised as a matrix of tactics (the adversary's goal) and techniques/sub-techniques (how they achieve it). It was first published by MITRE in 2013 and grew out of the FMX (Fort Meade Experiment) research programme, where MITRE researchers needed a common vocabulary to describe what red teams actually did so blue teams could measure whether they detected it. Unlike a vulnerability database (CVE) or a compliance checklist, ATT&CK describes behaviour — the durable, hard-to-change part of an attack that sits at the top of David Bianco's Pyramid of Pain.

For a defender, ATT&CK matters because it converts an unbounded problem ("detect all attacks") into a bounded, measurable one ("cover these ~200 techniques, here is my coverage, here are my gaps"). It is the connective tissue between four disciplines that otherwise speak different languages: threat intelligence (which groups do what), detection engineering (what rules catch what), red/purple teaming (what we actually tested), and risk/leadership reporting (a heatmap a VP can read). At scale, where you may ingest tens of billions of events a day across cloud, endpoint, identity, and SaaS, ATT&CK is the shared index that lets you answer "are we covered for what this threat actor does?" without hand-waving.

TL;DR
- ATT&CK is a behaviour taxonomy: Tactics (why, 14 in Enterprise) → Techniques (how, ~200) → Sub-techniques (specific how) → Procedures (the exact in-the-wild implementation). Techniques are Txxxx, sub-techniques Txxxx.001, tactics TA00xx.
- It is descriptive, not prescriptive and not sequential — it is a matrix of options, not an ordered chain like the Kill Chain.
- Defenders use it to build a detection coverage heatmap (via ATT&CK Navigator), find gaps, prioritise by threat intel, and drive threat-informed defense.
- Each technique page ships the three things a defender needs: Data Sources (what to log), Detection guidance, and Mitigations (Mxxxx).
- Use ATT&CK for TTP-level detection/coverage, the Kill Chain for high-level phase narrative, and the Diamond Model for per-intrusion analysis and pivoting. They compose; they do not compete.

2. How It Works (deep)

2.1 Core mechanics

ATT&CK is a directed knowledge graph. The object model has a small number of node types and a lot of edges; understanding the object model is what separates someone who "has heard of ATT&CK" from someone who can operationalise it. The canonical machine-readable form is STIX 2.1 (attack-stix-data on GitHub), which is what tools like Navigator, DeTT&CT, and SIEM enrichment pipelines consume.

Object ATT&CK term ID format Meaning Example
Tactic x-mitre-tactic TA0006 The adversary's tactical goal (a matrix column) Credential Access
Technique attack-pattern T1003 A general method to achieve a tactic OS Credential Dumping
Sub-technique attack-pattern T1003.001 A specific implementation of a technique LSASS Memory
Procedure (in-text sighting) n/a The exact real-world instance "APT29 used comsvcs.dll MiniDump on LSASS"
Group intrusion-set G0016 A tracked adversary cluster APT29 / Cozy Bear
Software malware/tool S0154 Malware or dual-use tool Cobalt Strike
Mitigation course-of-action M1040 A preventive control Behavior Prevention on Endpoint
Data Source / Component x-mitre-data-source DS0009 Telemetry that reveals the technique Process → Process Creation
Campaign campaign C0015 A time-bound set of activity SolarWinds Compromise
Detection (D3FEND) external Dxxxx Defensive countermeasure ontology Process Spawn Analysis

Two subtleties interviewers probe:

  • A technique can serve multiple tactics. T1078 Valid Accounts appears under Initial Access, Persistence, Privilege Escalation, and Defense Evasion. This is why ATT&CK is a matrix, not a tree — a single behaviour maps to multiple columns. When you build a coverage map you must decide whether you count the technique once or per-tactic; DeTT&CT counts per-technique, Navigator lets you score per (technique, tactic) cell.
  • Sub-technique vs procedure. A sub-technique is still a category ("LSASS Memory"). A procedure is the concrete instantiation ("Mimikatz sekurlsa::logonpasswords" vs "comsvcs.dll MiniDump" vs "ProcDump -ma lsass.exe"). Detections often live at the procedure level (the specific command line), while coverage is measured at the technique/sub-technique level. Confusing these two produces overstated coverage claims.

The matrices partition the knowledge base by technology domain, because the relevant behaviours differ:

Matrix Domain Tactics of note Why separate
Enterprise Windows, macOS, Linux, Cloud, Network, Containers, SaaS, Identity 14 tactics, Recon → Impact The flagship; platform sub-matrices below live under it
Cloud (IaaS/SaaS/Identity/Office) AWS, Azure, GCP, M365, Google Workspace, Entra ID No Execution-on-host emphasis; API-driven Cloud attacks are API/identity, not binary-on-disk
Containers Docker, Kubernetes Adds T1610 Deploy Container, T1611 Escape to Host Orchestrator + runtime specific
Mobile Android, iOS Distinct tactics (e.g., no "Boot autostart" the same way) MDM, app sandbox, different telemetry
ICS Operational technology Inhibit Response Function, Impair Process Control Safety/physical-impact goals

The three deliverables on every technique page are the defender's contract with ATT&CK: Data Sources (updated in the ATT&CK Data Sources / DS-object model since v9), Detection prose, and Mitigations. If you internalise nothing else, internalise the loop: pick technique → read its Data Sources → confirm you collect that telemetry → write/verify a detection → map a mitigation → colour the cell.

2.2 Concrete configuration

a) Fetch and query ATT&CK programmatically (STIX + Python). Never hand-copy technique lists; pull them from the source of truth.

# pip install mitreattack-python
from mitreattack.stixmapper import MitreAttackData

attack = MitreAttackData("enterprise-attack.json")  # from mitre/cti or attack-stix-data

# All techniques used by APT29 (G0016) — drives an intel-led coverage map
g = attack.get_groups_by_alias("APT29")[0]
techniques = attack.get_techniques_used_by_group(g["id"])
for t in techniques:
    tech = t["object"]
    print(tech["external_references"][0]["external_id"], tech["name"])

# Data sources that reveal T1003.001 LSASS Memory — tells you what to collect
lsass = attack.get_object_by_attack_id("T1003.001", "attack-pattern")
print(lsass.get("x_mitre_data_sources"))
# -> ['Process: OS API Execution', 'Process: Process Access',
#     'Command: Command Execution', 'Process: Process Creation']

c) Score coverage with DeTT&CT (data-source-driven, YAML). DeTT&CT (developed by Rabobank's CDC) scores your data source quality first, then derives technique visibility, which is more honest than colouring cells by gut feel.

# techniques-administration.yaml (excerpt) — visibility & detection per technique
techniques:
  - technique_id: T1003.001
    technique_name: OS Credential Dumping - LSASS Memory
    detection:
      - applicable_to: [all]
        location: [SIEM - Sigma rule cred_dump_lsass_access]
        score_logbook:
          - date: 2026-06-15
            score: 4          # 0-5; 4 = good, validated
    visibility:
      - applicable_to: [windows servers]
        score_logbook:
          - date: 2026-06-15
            score: 3          # driven by Sysmon config event 10 enabled

d) Map a fired alert back to ATT&CK in your SIEM (KQL, Microsoft Sentinel). Every detection should carry its technique ID as metadata so coverage is a query, not a spreadsheet.

// Sentinel: coverage rollup — how many distinct techniques did our analytics fire on last 30d?
SecurityAlert
| where TimeGenerated > ago(30d)
| mv-expand Tactic = todynamic(Tactics), Tech = todynamic(Techniques)
| summarize Alerts = count(), Rules = dcount(AlertName) by tostring(Tech), tostring(Tactic)
| order by Alerts desc

e) Tag Sigma rules with ATT&CK so the mapping is native. Sigma's tags field is the community-standard place for this; a converter (sigma-cli / pySigma) preserves it into Splunk/Sentinel/Elastic.

title: Credential Dumping via LSASS Process Access
logsource: { product: windows, category: process_access }
detection:
  selection:
    TargetImage|endswith: '\lsass.exe'
    GrantedAccess: ['0x1010','0x1410','0x1438','0x143a','0x1fffff']
  condition: selection
tags:
  - attack.credential_access
  - attack.t1003.001
level: high

3. Real-World Incidents & CVEs

ATT&CK's value is that these disparate incidents map to a shared, comparable set of techniques — you can overlay them in Navigator to see which behaviours recur and therefore deserve investment.

Incident / Report Year Key ATT&CK techniques Root cause / lesson
SolarWinds / SUNBURST (Campaign C0024) 2020 T1195.002 Supply Chain, T1078.004 Valid Accounts (cloud), T1550.001 App Access Token, T1606.002 Forge SAML (Golden SAML) Trusted build pipeline poisoned; identity trust (SAML signing key) abused. Detection of forged tokens, not malware, was the gap.
Colonial Pipeline (DarkSide) 2021 T1078 Valid Accounts (legacy VPN, no MFA), T1486 Data Encrypted for Impact A single reused VPN credential with no MFA. ATT&CK T1078 + M1032 MFA would have broken the chain at initial access.
Uber breach (Lapsus$-style) 2022 T1621 MFA Request Generation (push bombing), T1078, T1552.001 Creds in Files (hardcoded admin in PowerShell script), T1098 Account Manipulation MFA fatigue + hardcoded secret in a network share. Session/identity abuse, not exploits.
SolarWinds-adjacent: HAFNIUM / ProxyLogon (CVE-2021-26855 chain) 2021 T1190 Exploit Public-Facing App, T1505.003 Web Shell, T1003 Cred Dumping Unpatched Exchange SSRF→RCE; web shell persistence. T1190 + T1505.003 detections were the fast catch.
Log4Shell (CVE-2021-44228) 2021 T1190 Exploit Public-Facing App, T1059 Execution, T1105 Ingress Tool Transfer JNDI lookup RCE. Egress filtering (M1037) and outbound LDAP/RMI detection mattered more than the CVE patch race.
Capital One (SSRF → IMDS) 2019 T1190, T1552.005 Cloud Instance Metadata API, T1530 Data from Cloud Storage WAF SSRF reached IMDSv1; over-permissioned role read S3. IMDSv2 (M) + least privilege = the fix.
Scattered Spider (UNC3944) intrusions 2023-24 T1566 Phishing, T1621 MFA push bombing, T1199 Trusted Relationship (help-desk social eng), T1078 Help-desk-driven SIM swap / MFA reset. Human process is a "technique surface" too.

Case study — SolarWinds and the limits of malware-centric detection. SUNBURST was a backdoor smuggled into a signed, trusted SolarWinds Orion update (T1195.002). Because the binary was legitimately signed and the code went dormant for ~2 weeks, endpoint AV never flagged it. The decisive attacker moves were in identity: they stole the on-prem SAML token-signing certificate and forged SAML assertions (T1606.002 Golden SAML) to mint tokens for arbitrary cloud identities (T1078.004), then used those to read mail via OAuth app tokens (T1550.001). The lesson ATT&CK crystallised: a mature SOC must have coverage in the Cloud and Identity sub-matrices — anomalous SAML issuance, new service-principal credentials, and impossible-token-audience — not just endpoint EDR. FireEye caught it via a new MFA device registration anomaly on an employee account, i.e. an identity-tier detection.

Case study — Capital One and cloud technique coverage. An SSRF flaw in a WAF (T1190) was used to reach the EC2 Instance Metadata Service v1 (T1552.005) and retrieve temporary IAM role credentials, which had excessive S3 permissions (over 100M records, T1530). No host malware, no C2 beacon — the entire intrusion is invisible to endpoint-only ATT&CK coverage. Mapping it to the Cloud matrix shows exactly the three cells you must cover: SSRF/exploit at the edge, IMDS credential theft, and bulk cloud storage read. IMDSv2 (session-token-required metadata) directly mitigates T1552.005 — a control you can only "see" as relevant if you thought in ATT&CK terms.

Top 5 in depth

1) Phishing-resistant MFA + Conditional Access (mitigates T1078, T1621, T1550). T1078 Valid Accounts is the single highest-value defensive investment because legitimate credentials generate low signal — the login "looks normal." Technically, FIDO2/WebAuthn binds the authentication to an origin and a hardware authenticator, so a reverse-proxy phish (Evilginx) that relays a password + OTP fails: the passkey signs a challenge scoped to the real relying-party ID, and the attacker's proxy origin cannot satisfy it. Layer Conditional Access on top: require compliant device AND managed network OR step-up for sensitive scopes. Implementation (Entra):

# Conditional Access: block legacy auth, require phishing-resistant MFA for admins
az rest --method PATCH --uri "https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies/{id}" \
  --body '{"grantControls":{"operator":"AND","authenticationStrength":{"id":"<phishing-resistant-id>"}}}'

Trade-off: passkey rollout has device/logistics cost, and you must still cover the post-auth case (stolen session cookie, T1539) with token binding and short TTLs — MFA does not help once a valid session exists.

2) Data-source-driven coverage scoring (mitigates coverage theatre, threats 1/2/12). The defining failure of ATT&CK programs is colouring cells green by aspiration. The fix is to invert the dependency: a technique cannot be "covered" unless the Data Sources its ATT&CK page lists are actually collected and healthy. DeTT&CT formalises this — you score data-source quality (device completeness, timeliness, retention, consistency), and technique visibility is derived, not asserted. Concretely, you run a nightly job that checks: does Sysmon EventID 10 still arrive from ≥95% of Windows servers? If a GPO change silently drops it, the LSASS-access detection (T1003.001) degrades and the cell should auto-flip to red. Trade-off: this is real engineering (telemetry health monitoring, canary events) and cannot be bought as a product; but without it, your heatmap is fiction.

3) Egress control + C2 detection (mitigates T1071, T1105, T1041). Almost every intrusion eventually needs to talk out (C2) or move tools in. Default-deny egress with an explicit allowlist collapses the attacker's options and turns C2 into a loud event. Where full allowlisting is impractical, detect on the channel: JA3/JA3S TLS fingerprinting to spot beaconing implants, DNS query-length/entropy analytics for DNS tunnelling (T1071.004), and Beacon jitter analysis (periodicity with variance). Implementation sketch (Zeek + Sigma over conn.log/ssl.log): alert on long-lived, low-byte, regular-interval TLS to newly-seen SNI. Trade-off: TLS inspection has privacy/perf/pinning costs, and cloud-native C2 over legitimate SaaS (T1102 Web Service — GitHub, Slack, S3) blends with allowed traffic, so egress control must be paired with SaaS-side logging.

4) Least privilege + short-lived cloud credentials (mitigates T1552, T1078.004, T1530). Cloud breaches are usually identity breaches: an over-permissioned role reachable via a stolen or metadata-harvested token. Two controls compound: (a) eliminate long-lived static keys in favour of workload identity federation / STS with minutes-long TTL, so a stolen token expires before it's useful; (b) scope permissions with IAM conditions, SCPs (AWS) / Org Policies (GCP) / management-group policy (Azure) so even a valid token can't read all buckets. IMDSv2 specifically kills the Capital One class (T1552.005) by requiring a PUT-obtained session token with a hop limit. Trade-off: least privilege is an ongoing grind (permission creep is entropy), and workload federation adds setup complexity — but it removes the "one leaked key = game over" failure mode.

5) Purple-team validation with Atomic Red Team / CALDERA (mitigates threats 1/13/15). A detection you have never fired is a hypothesis, not a control. Atomic Red Team provides per-technique, per-procedure test cases (Invoke-AtomicTest T1003.001) that safely emit the exact telemetry a real attacker would, so you can confirm the rule fires, measure detection latency, and then colour the cell. CALDERA (also MITRE) chains techniques into full adversary emulation plans (e.g., an APT29 profile) to test detection across a whole kill chain, not one cell at a time. Wire this into CI: on every detection-rule change, run the mapped atomic test in a range and fail the pipeline if the alert doesn't fire. Trade-off: you need a safe range/canary fleet and careful blast-radius control (some atomics are destructive), and atomic tests exercise a procedure, so passing one doesn't mean every procedure of that technique is covered.

Response playbook (identity-driven cloud intrusion, techniques T1078.004 → T1552 → T1098)

  1. Triage & scope (map to ATT&CK first). Pull the alert's technique IDs; in the SIEM, pivot on the userIdentity.arn/principal and sourceIPAddress to enumerate every technique the same actor touched. Build a mini-Navigator layer of this incident to see how far along the tactics they got.
  2. Contain identity. Disable the compromised principal, revoke active sessions/tokens (Okta session revoke, aws iam update-access-key --status Inactive, Entra revokeSignInSessions). Killing the session is what stops T1550/T1539 reuse; a password reset alone does not.
  3. Contain blast radius. Quarantine any new access keys (T1098), remove attacker-created IAM users/roles/service principals, and apply a temporary deny SCP to affected accounts if lateral spread is suspected.
  4. Eradicate persistence. Hunt for the other ATT&CK cells: new OAuth app consents (T1550.001), forwarding rules (T1114), added MFA devices (T1556), and IAM policy attachments. Attackers plant multiple persistence mechanisms.
  5. Preserve evidence. Snapshot CloudTrail/audit logs to immutable storage (defeats T1070 anti-forensics); export the Diamond Model artefacts (adversary IP/infra, capability = tooling, victim, infrastructure) for pivoting.
  6. Recover. Rotate all credentials in blast radius (assume T1552 harvest occurred), re-enable principals with least-privilege + phishing-resistant MFA, restore any manipulated resources.
  7. Post-incident → close the ATT&CK gaps. For every technique the adversary used that you didn't detect, file a detection-engineering ticket and add the cell to the coverage backlog; validate the new rule with an Atomic test before marking it green. This is the loop that turns an incident into permanent coverage.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is the difference between a tactic, technique, sub-technique, and procedure in ATT&CK?
A tactic is the adversary's goal (the "why"), e.g. Credential Access (TA0006) — it's a matrix column. A technique is a general method to achieve that goal (the "how"), e.g. OS Credential Dumping (T1003). A sub-technique is a specific implementation of a technique, e.g. LSASS Memory (T1003.001). A procedure is the exact, in-the-wild instance — e.g. "Mimikatz sekurlsa::logonpasswords" or "comsvcs.dll MiniDump on lsass". Coverage is measured at technique/sub-technique level; detections often live at the procedure level.
Why asked / signal: Confirms you actually understand the object model, not just the buzzword.

Q2. Is ATT&CK sequential like the Cyber Kill Chain?
No. The Kill Chain is a linear 7-phase sequence an attacker walks through; ATT&CK is a matrix of options — a menu, not an ordered path. Adversaries skip tactics, repeat them, and run them in parallel. The columns are roughly ordered left-to-right by typical progression, but nothing forces an adversary to move that way; cloud attacks often jump straight from Initial Access to Impact with no on-host Persistence.
Why asked / signal: A very common misconception; separates rote memorisers from practitioners.

Q3. Why is a technique like T1078 Valid Accounts hard to detect, and what makes it worse than malware?
Because it uses legitimate credentials, the activity looks like a normal user — there's no malicious binary, no signature, and the login succeeds. Malware has artefacts (hashes, files, injected memory) high on the Pyramid of Pain's cheap-to-detect end; valid-account abuse sits at the behaviour/TTP end where you must reason about context (impossible travel, new device, anomalous privilege use). That's why it appears under four tactics and why identity telemetry, not endpoint AV, is the right lens.
Why asked / signal: Tests whether you think in behaviour vs indicators.

Q4. What three things does every ATT&CK technique page give a defender?
Data Sources (what telemetry reveals it), Detection guidance (how to spot it), and Mitigations (Mxxxx, how to prevent it). The defensive loop is: confirm you collect the Data Source, write a detection, apply a mitigation, then validate. If you can't collect the data source, you can't honestly claim coverage.
Why asked / signal: Checks you know how to operationalise, not just browse, ATT&CK.

Q5. What is ATT&CK Navigator and what is a "layer"?
Navigator is a web tool for visualising and annotating the ATT&CK matrix; a layer is a JSON file that assigns scores/colours/comments to (technique, tactic) cells. Defenders use layers to draw coverage heatmaps, overlay a threat group's techniques, and diff current vs target state. Layers are generated from your detection inventory — you don't colour cells by hand in production.
Why asked / signal: Confirms hands-on familiarity with the tooling.

Tier 2 — System design / trade-offs

Q6. Design a detection-coverage program for a cloud-first company using ATT&CK. How do you avoid "coverage theatre"?
Start by scoping the right matrices — Cloud (IaaS/SaaS/Identity/Office) and Containers, not just Enterprise-Windows, because our attacks are API/identity-driven. Invert the dependency so a cell can only be green if its Data Sources are collected and healthy: I'd use DeTT&CT to score data-source quality first and derive technique visibility, with a nightly telemetry-health job that auto-flips cells red on drift. Prioritise with threat intel — build Navigator layers for the groups that target us (e.g., Scattered Spider, APT29) and cover their techniques first rather than boiling the ocean. Every green cell must be backed by a validated detection: an Atomic Red Team test that actually fired the rule, wired into CI. Report coverage as validated-technique count with FP/latency metrics, never as "cells coloured." The anti-theatre discipline is: no data source → no coverage; no passing atomic test → no green.
Why asked / signal: Senior signal — do you build measurable, honest programs or vanity heatmaps?

Q7. When would you use ATT&CK vs the Kill Chain vs the Diamond Model? Do they compete?
They compose. The Kill Chain is best for executive-level narrative and "where can we break the chain" — coarse, linear, 7 phases. ATT&CK is best for TTP-level detection engineering and coverage measurement — granular, non-linear, ~200 techniques. The Diamond Model is best for analysing a single intrusion: it structures the four core features (adversary, capability, infrastructure, victim) and enables pivoting — from a malicious IP (infrastructure) to other victims, from a capability to an adversary. In practice: use Kill Chain to frame the story for leadership, ATT&CK to map every observed technique and drive detections, and Diamond during IR to pivot across the four vertices and enrich attribution. A mature workflow overlays all three — map ATT&CK techniques onto Kill Chain phases while populating a Diamond per intrusion.
Why asked / signal: Tests conceptual maturity and knowing frameworks are complementary tools.

Q8. How do you prioritise which techniques to build detections for first, given you can't cover all 200+?
Threat-informed prioritisation, not alphabetical. First, use CTI to identify the adversaries and campaigns that actually target our sector/tech stack, pull their ATT&CK techniques (from ATT&CK Groups, CTID Sightings, or vendor intel), and overlay them in Navigator — techniques used by many relevant groups rise to the top. Second, weight by our own architecture (a cloud-first shop weights T1078.004/T1552.005 over on-prem Kerberoasting). Third, weight by "chokepoint" value — techniques that appear across many kill chains (Valid Accounts, Command Execution, Ingress Tool Transfer) give the most coverage per rule. Fourth, factor feasibility: do we already collect the data source? Cheap-to-cover, high-frequency techniques win. MITRE's own "Top ATT&CK Techniques" methodology and CTID's Sightings data formalise this.
Why asked / signal: Real programs are resource-constrained; you want to convey a defensible prioritisation method.

Q9. Your leadership asks "what's our detection coverage percentage?" Why is a single number dangerous, and what would you report instead?
A single "% of techniques covered" is misleading because it treats all techniques as equal (Kerberoasting ≠ Data Encrypted for Impact in impact), counts a technique with one weak heuristic the same as one with a validated high-fidelity rule, ignores sub-technique/procedure granularity, and says nothing about telemetry health or false-positive burden. Instead I'd report: a Navigator heatmap weighted by relevance to our threat model, split into validated vs heuristic vs gap; coverage of the specific groups that target us; mean detection latency and FP rate per tier; and a trend line of validated cells over time. The honest metric is "of the techniques the adversaries who target us actually use, X% have a purple-team-validated detection."
Why asked / signal: Metrics literacy and resisting vanity numbers — a staff-level concern.

Q10. How do you keep ATT&CK-mapped detections from drifting into uselessness over time?
Two failure axes: telemetry drift and adversary drift. For telemetry, run continuous data-source health monitoring (canary events that should always fire; alert if a Sysmon EID or CloudTrail event type stops arriving) so a green cell can't silently rot. For detections, wire Atomic Red Team / CALDERA into CI so every rule is re-validated on change and on a schedule — if the atomic no longer fires the alert, the pipeline fails. For adversary drift, subscribe to ATT&CK version releases and re-map when techniques are added/deprecated/renumbered (pin one version org-wide to avoid join skew). Finally, track FP rate per rule and retire or retune rules that page without value — an alert nobody trusts is a negative-coverage cell.
Why asked / signal: Operational maturity; detections are a living system, not a one-time deliverable.

Tier 3 — Expert / deep-dive

Q11. Walk through mapping the SolarWinds intrusion to ATT&CK and explain which detection layer actually caught it and why endpoint AV didn't.
Initial compromise was T1195.002 Supply Chain (trojanised, signed Orion update — SUNBURST), which AV missed because the binary was legitimately signed and dormant ~2 weeks (defense evasion by design). The pivotal moves were identity: theft of the SAML token-signing certificate enabled T1606.002 Golden SAML, forging assertions to mint tokens for arbitrary identities (T1078.004), then using OAuth application tokens (T1550.001) to read mail via Graph. None of that touches endpoints in a way AV inspects — it's control-plane and identity behaviour. FireEye's decisive catch was an identity-tier anomaly: a new MFA device registered to an employee account. The lesson for coverage: if your Navigator map is all Enterprise-Windows and empty in the Identity/Cloud sub-matrices, you are structurally blind to this class. You need SAML-issuance monitoring, new service-principal credential detection, and impossible-token-audience analytics.
Why asked / signal: Deep incident fluency + the ability to connect a real breach to concrete coverage gaps.

Q12. Take T1552 Unsecured Credentials. Walk through data sources, a detection idea, and a mitigation across its sub-techniques.
T1552 has several sub-techniques with different telemetry: .001 Credentials in Files (Data Source: File Access, Command Execution) — detect via secret-scanning in CI (gitleaks/trufflehog) plus host FIM alerting on reads of .aws/credentials, .env, id_rsa; mitigate with secret managers and pre-commit hooks (M1041 no plaintext creds). .004 Private Keys (File Access) — alert on mass reads of *.pem/*.key by non-standard processes; mitigate with hardware-backed keys/HSMs. .005 Cloud Instance Metadata API (Network Traffic, Process) — the Capital One pattern: detect any process other than cloud-init/agents hitting 169.254.169.254/.../iam, and in cloud logs, use of role credentials from an IP that differs from the instance; mitigate decisively with IMDSv2 (session-token + hop limit) which breaks SSRF-to-metadata. .006 Group Policy Preferences — detect Groups.xml reads with the known AES key; mitigate by removing cpassword. Cross-cutting: short-lived tokens shrink the value of any harvested credential. The point is that "covering T1552" is meaningless without enumerating sub-techniques — each has its own data source and control.
Why asked / signal: Tests whether you can go from a technique ID to real, sub-technique-specific detection engineering — exactly the day job.

Q13. How would you build an automated, closed-loop system that keeps a coverage heatmap accurate at scale?
Treat it as a data pipeline with four joined datasets, all keyed on ATT&CK IDs. (1) Detection inventory: every SIEM rule tagged with attack.txxxx (enforced at rule-review time via Sigma tags / detection-as-code in Git). (2) Telemetry health: a service that verifies each ATT&CK Data Source is arriving from ≥N% of assets, emitting a per-data-source freshness score. (3) Validation results: CALDERA/Atomic runs on a canary fleet, emitting pass/fail + latency per (technique, procedure). (4) Threat priority: a weekly-refreshed layer of techniques used by relevant groups (from ATT&CK STIX + CTI). A job joins these: a cell is green only if it has a rule AND its data source is healthy AND its latest atomic passed; yellow if rule-but-unvalidated; red otherwise; and the whole map is weighted by threat priority. Output is a generated Navigator layer plus alerts when a previously-green cell regresses (data source dropped or atomic failed). This makes coverage a computed, self-healing artefact rather than a quarterly slide — the SolarWinds lesson (drift kills coverage) baked into engineering.
Why asked / signal: Staff-level systems thinking; can you engineer a program, not just describe one?

Q14. A red team says they achieved objective without a single detection firing, yet your heatmap is 80% green. Reconcile this.
Several classic gaps explain it. (a) Coverage theatre: cells green by assertion, not validated — the 80% is fictional. (b) Procedure vs technique: rules cover one procedure of a technique (e.g., Mimikatz for T1003.001) but the red team used another (comsvcs MiniDump), so the cell is "covered" yet blind. (c) Matrix scope: green in Enterprise-Windows but the attack lived in Cloud/Identity where you had no map. (d) Telemetry drift: the data source silently stopped arriving; the rule can't fire. (e) Sequencing/low-and-slow: each step is individually below threshold; you lack correlation across cells. (f) Anti-forensics (T1070): they cleared logs before detection latency elapsed. The fix is procedure-level validation (Atomic/CALDERA), telemetry-health gating, cross-technique correlation (not just per-cell rules), and honest scoring. This scenario is exactly why validated-coverage, not coloured-coverage, is the only metric that survives contact with a real adversary.
Why asked / signal: Elite signal — the gap between reported and real coverage is where breaches happen; can you diagnose it?

Q15. How do ATT&CK, D3FEND, and CVE/CWE relate, and how would you use them together to argue for a specific security investment?
ATT&CK is offensive behaviour (what adversaries do); D3FEND (also MITRE) is the defensive countermeasure ontology (what controls exist and what they act on) with a mapping back to ATT&CK techniques; CAPEC bridges to application-attack patterns; CWE is weakness types and CVE is specific vulnerabilities. To argue an investment, I chain them: a relevant threat group uses T1190 Exploit Public-Facing App (ATT&CK) → realised through CWE-918 SSRF / a specific CVE (e.g., Log4Shell CVE-2021-44228) → D3FEND proposes countermeasures (e.g., Outbound Traffic Filtering, Message Analysis) → I map those to concrete controls (egress allowlist, WAF, IMDSv2) and to my coverage gap. The business case becomes: "group X targets us, uses T1190/T1552.005, we have a red gap there, here are two mitigations D3FEND recommends and their cost, and here's the residual risk." Tying the vulnerability (CVE/CWE), the behaviour (ATT&CK), and the control (D3FEND) together turns a vague ask into a threat-informed, defensible investment.
Why asked / signal: Tests breadth across the MITRE ecosystem and the ability to translate frameworks into funded decisions.

9. Key Takeaways

  • ATT&CK is behaviour, not indicators. It lives at the TTP tip of the Pyramid of Pain — the hardest layer for adversaries to change and therefore the most durable to detect on.
  • It's a matrix, not a chain. Tactics are goals (columns), techniques are methods, sub-techniques are specific methods, procedures are exact instances. One technique can serve several tactics.
  • The defender's loop is Data Source → Detection → Mitigation → Validate. A cell you can't collect telemetry for is not covered, no matter its colour.
  • Coverage theatre is the #1 program failure. Score only after purple-team validation; gate cells on data-source health; count validated techniques, never coloured cells.
  • Scope the right matrices. Cloud-first orgs must map Cloud, Identity, SaaS, and Containers — endpoint-only coverage was structurally blind to SolarWinds and Capital One.
  • Prioritise with threat intel. Cover the techniques the adversaries who actually target you use first; chokepoint techniques (T1078, T1059, T1105) give the most coverage per rule.
  • ATT&CK, Kill Chain, and Diamond Model compose. Kill Chain for narrative, ATT&CK for TTP detection/coverage, Diamond for per-intrusion pivoting and attribution.
  • Validate with Atomic Red Team / CALDERA. An unfired detection is a hypothesis; wire adversary emulation into CI and re-validate on drift.
  • Map every alert and every incident to technique IDs. That single discipline turns coverage into a query and turns each incident into permanent, backlogged coverage.
  • Tie it to the wider MITRE ecosystem (D3FEND countermeasures, CAPEC/CWE/CVE) to turn a red cell into a funded, defensible security investment.
