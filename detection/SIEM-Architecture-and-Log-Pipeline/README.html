SIEM Architecture and the Security Log Pipeline at Scale

1. Overview

A SIEM (Security Information and Event Management) is the system that turns raw, heterogeneous log exhaust from thousands of sources into normalized, enriched, queryable security events, then runs detection logic over that stream to produce alerts. At scale the interesting engineering problem is not the correlation engine — it is the pipeline in front of it: collecting from agents, syslog, and cloud-native sinks; parsing and normalizing to a common data model; enriching with asset/identity/threat-intel context; and tiering storage across hot/warm/cold so that ingest cost does not become unbounded. A modern platform ingests tens of terabytes to petabytes per day, and roughly 60–80% of every SIEM engineering conversation is about what you can afford to ingest, index, and retain versus what you need to detect on.

The industry has bifurcated into two architectures. Traditional SIEMs (Splunk, IBM QRadar, Elastic) index everything into their own store and charge on ingest volume (Splunk) or EPS/FlexPoints (QRadar). Cloud-native SIEMs (Google Chronicle/SecOps, Microsoft Sentinel, Amazon Security Lake + Athena/OpenSearch) decouple storage from compute, normalize to a fixed schema (UDM, ASIM, OCSF), and price on ingest + query compute. The current best-practice pattern — "SIEM + data lake + detection-as-code" — routes cheap-but-voluminous logs to a lake (S3/GCS/BigQuery) and routes high-signal logs to the SIEM, with detections managed as version-controlled code (Sigma, YARA-L, Sentinel .yaml) in CI/CD.

TL;DR
- A SIEM is a pipeline, not a product: collect → parse/normalize → enrich → store (tiered) → detect → alert → respond. The hard parts are normalization drift and ingest economics, not the correlation engine.
- Normalization to a common data model (OCSF/ECS/UDM) is what makes cross-source detections possible — you write one rule against src_endpoint.ip, not five per-vendor rules.
- Garbage-in-garbage-out is the #1 failure mode: missing logs, broken parsers, and silent source outages cause more misses than bad detection logic. Monitor log-source health as a first-class SLO.
- Storage tiering + a data lake is how you control cost: hot index for 30–90 days of high-signal data, cold/lake object storage for 1+ years of everything at ~1/10th the cost.
- Detection-as-code (rules in Git, tested in CI, deployed via pipeline) is the difference between a SOC that scales and one that rots.

2. How It Works (deep)

2.1 Core mechanics

The pipeline is a directed flow of stages, each with distinct failure modes and cost characteristics. Think of it as an ETL/streaming system whose payload happens to be security telemetry.

Stage Function Key objects / concepts Primary failure mode
Collection Get events off the source Agents, forwarders, syslog, API pollers, cloud sinks Silent source outage, agent death, dropped UDP
Transport / buffering Reliable delivery, backpressure Kafka/Pub/Sub, disk queues, TLS Backpressure → drop; unencrypted transit
Parsing Raw bytes → structured fields Grok, regex, JSON/CEF/LEEF decoders Parsing drift on format change
Normalization Structured fields → common schema OCSF, ECS, UDM, ASIM field mapping Wrong field mapping; unmapped events
Enrichment Add context Asset DB, identity/CMDB, GeoIP, threat intel Stale enrichment; join latency
Routing Decide destination + tier Ingest-time filters, index routing Over-ingest of low-value data
Storage Persist for search/retention Hot index, warm, cold/lake object store Cost blowout; retention gaps
Detection execution Run rules/analytics Streaming rules, scheduled search, UEBA Rule silently disabled; latency
Alerting / response Notify + orchestrate Case mgmt, SOAR playbooks Alert fatigue, no runbook

Control plane vs data plane. The data plane is the high-throughput event flow (Kafka topics, forwarders, indexers). The control plane is configuration: parser definitions, normalization mappings, detection rules, routing policy, retention policy. A mature platform manages the control plane as code (Git) and treats parser/mapping changes with the same rigor as application deploys, because a bad parser push can blind detections silently.

Push vs pull collection. Agents and syslog push; cloud APIs are often pulled (poll CloudTrail/S3, GCP Pub/Sub sinks, Graph API for M365). Cloud-native sinks (AWS Kinesis Firehose → S3, GCP Log Router → Pub/Sub, Azure Diagnostic Settings → Event Hub) are the scalable path because they push structured JSON with delivery guarantees rather than relying on you to poll.

Schema-on-write vs schema-on-read. Traditional SIEMs and cloud-native SIEMs largely normalize at ingest (schema-on-write): parse and map before storage, fast query, but reprocessing history requires re-ingest. Data lakes favor schema-on-read: store raw, apply schema at query time (Athena/BigQuery views), cheap storage and reparse-friendly, but slower/more expensive queries. The modern pattern uses both.

2.2 Concrete configuration

(a) Cloud-native sink — GCP Log Router to Pub/Sub → data lake + SIEM. This is the canonical GCP export path; the Pub/Sub topic fans out to both BigQuery (lake) and Chronicle (SIEM).

# Create an aggregated org-level sink for all Data Access + Admin Activity audit logs
gcloud logging sinks create secops-audit-sink \
  pubsub.googleapis.com/projects/sec-logging-prj/topics/audit-events \
  --organization=123456789012 \
  --include-children \
  --log-filter='logName:"cloudaudit.googleapis.com" AND
    (protoPayload.methodName:"SetIamPolicy" OR
     resource.type="gcs_bucket" OR
     protoPayload.serviceName="storage.googleapis.com")'

# Grant the sink's writer identity publish rights (returned service account)
gcloud pubsub topics add-iam-policy-binding audit-events \
  --member="serviceAccount:o123456789012-987654@gcp-sa-logging.iam.gserviceaccount.com" \
  --role="roles/pubsub.publisher"

Commentary: the --log-filter is your ingest-time cost control — Data Access logs are enormous, so you route only IAM changes and GCS access, not every read. --include-children makes it org-wide. The Pub/Sub topic decouples producers from consumers so a SIEM outage does not lose logs (Pub/Sub retains for up to 7 days).

(b) AWS Security Lake — OCSF-normalized lake. Security Lake auto-normalizes CloudTrail, VPC Flow, Route 53, and findings into OCSF Parquet in your S3, partitioned by source/region/date.

aws securitylake create-data-lake \
  --configurations '[{"region":"us-east-1",
     "encryptionConfiguration":{"kmsKeyId":"alias/security-lake"},
     "lifecycleConfiguration":{
        "transitions":[{"storageClass":"STANDARD_IA","days":30},
                       {"storageClass":"GLACIER","days":180}],
        "expiration":{"days":730}}}]' \
  --meta-store-manager-role-arn arn:aws:iam::111122223333:role/AmazonSecurityLakeMetaStoreManager

# Register CloudTrail management events as an OCSF source
aws securitylake create-aws-log-source \
  --sources '[{"regions":["us-east-1"],
     "sourceName":"CLOUD_TRAIL_MGMT","sourceVersion":"2.0"}]'

Commentary: the lifecycleConfiguration is your storage tiering — hot in STANDARD, warm in STANDARD_IA at 30 days, cold in GLACIER at 180, expire at 2 years. Because the output is OCSF Parquet, Athena/OpenSearch/Splunk/any consumer reads one schema.

(c) Splunk Universal Forwarder + ingest-time filtering (traditional SIEM, license = ingest). Because Splunk bills on GB/day indexed, you drop noise at the forwarder/heavy-forwarder.

# inputs.conf — collect Windows Security + Sysmon
[WinEventLog://Security]
disabled = 0
index = wineventlog
renderXml = true

# props.conf + transforms.conf — DROP verbose 4662/5156 noise before indexing
# props.conf
[WinEventLog:Security]
TRANSFORMS-drop = drop_noise_4662

# transforms.conf
[drop_noise_4662]
REGEX = EventCode=(4662|5156|4663)
DEST_KEY = queue
FORMAT = nullQueue

Commentary: nullQueue routing is the classic Splunk cost lever — you can cut 30–50% of Windows volume by dropping high-volume, low-detection-value events before they count against license. The trade-off: if a future detection needs 4663 (object access), you have no history. Decide with a data-value review, not by gut.

(d) Sentinel — data collection rule (DCR) with KQL transform at ingest. Sentinel's DCRs let you transform/filter in the pipeline and route to the cheap Auxiliary/Basic Logs tier vs Analytics tier.

{
  "properties": {
    "dataFlows": [{
      "streams": ["Microsoft-SecurityEvent"],
      "destinations": ["sentinelWorkspace"],
      "transformKql": "source | where EventID in (4624,4625,4672,4688,4720,4732) | project TimeGenerated, Computer, EventID, Account, IpAddress",
      "outputStream": "Microsoft-SecurityEvent"
    }]
  }
}

Commentary: transformKql filters to detection-relevant EventIDs and projects only needed columns — reducing GB ingested into the expensive Analytics tier. Route the rest to Basic Logs (cheaper, query-limited, 8-day interactive retention) or archive.

(e) Detection-as-code — Sigma rule compiled to backend. One vendor-neutral rule, compiled to Splunk SPL, Sentinel KQL, or Chronicle YARA-L via sigma convert.

title: Suspicious GCP Service Account Key Creation
id: 7c8f2e10-3a4b-4c9d-9e21-abc123def456
status: experimental
logsource:
  product: gcp
  service: gcp.audit
detection:
  selection:
    data.protoPayload.methodName: google.iam.admin.v1.CreateServiceAccountKey
  filter_ci:
    data.protoPayload.authenticationInfo.principalEmail|endswith: '@terraform-ci.iam.gserviceaccount.com'
  condition: selection and not filter_ci
level: high
tags:
  - attack.persistence
  - attack.t1098.001   # Account Manipulation: Additional Cloud Credentials

Commentary: this rule lives in Git, is unit-tested against sample events in CI, then sigma convert -t chronicle (or -t splunk) emits the deployable artifact. The filter_ci excludes the known CI service account — an allowlist expressed as code and reviewed in PR, not a click in a console that no one can audit later.

Normalization example: raw log → normalized fields

A raw GCP Cloud Audit Log (SetIamPolicy granting roles/owner) arrives as nested JSON. The pipeline maps it into a common data model. Below, the same event is shown as raw, then as ECS, then as Chronicle UDM.

Raw (GCP audit log, abbreviated):

{
  "protoPayload": {
    "@type": "type.googleapis.com/google.cloud.audit.AuditLog",
    "authenticationInfo": {"principalEmail": "eve@contoso.com"},
    "requestMetadata": {"callerIp": "203.0.113.44",
      "callerSuppliedUserAgent": "google-cloud-sdk gcloud/458.0.1"},
    "methodName": "SetIamPolicy",
    "serviceName": "cloudresourcemanager.googleapis.com",
    "resourceName": "projects/prod-payments",
    "serviceData": {"policyDelta": {"bindingDeltas":
      [{"action": "ADD", "role": "roles/owner",
        "member": "user:mallory@gmail.com"}]}}
  },
  "severity": "NOTICE",
  "timestamp": "2026-07-06T14:22:07Z"
}

Normalized to ECS (Elastic Common Schema):

ECS field Value
@timestamp 2026-07-06T14:22:07Z
event.category iam
event.action SetIamPolicy
event.outcome success
cloud.provider gcp
cloud.project.id prod-payments
user.email eve@contoso.com
source.ip 203.0.113.44
user_agent.original google-cloud-sdk gcloud/458.0.1
gcp.audit.iam.role_added (custom) roles/owner
related.user [eve@contoso.com, mallory@gmail.com]

Normalized to Chronicle UDM (event fragment):

metadata.event_type = "USER_RESOURCE_UPDATE_PERMISSIONS"
metadata.product_name = "Google Cloud Platform"
principal.user.email_addresses = "eve@contoso.com"
principal.ip = "203.0.113.44"
target.resource.name = "projects/prod-payments"
target.user.email_addresses = "mallory@gmail.com"
security_result.summary = "Added roles/owner to external user"

The payoff: a single detection ("external Gmail user granted roles/owner on a prod project") is now writable against target.user.email_addresses in UDM or user.email in ECS — regardless of whether the source was GCP, AWS AttachUserPolicy, or Azure role assignment, because all three normalize to the same category (iam / permission update).

3. Real-World Incidents & CVEs

Name Year Root cause Lesson for the pipeline
Log4Shell (CVE-2021-44228) 2021 JNDI lookup in Log4j; malicious strings in log messages Log ingestion/parsing layers themselves were vulnerable — a SIEM parser can be an attack surface. Sanitize/validate on parse.
SolarWinds SUNBURST 2020 Supply-chain implant; C2 disguised as Orion telemetry Attackers evade by blending with normal telemetry; needed baseline/UEBA, not just IOC match. Log integrity + anomaly detection matter.
Capital One (SSRF) 2019 SSRF → IMDS creds → S3 exfil The CloudTrail/VPC-Flow evidence existed but detections/alerting were absent. Having logs ≠ detecting.
Splunk RCE CVE-2023-46214 2023 Unsafe XSLT in Splunk allowed RCE via crafted upload Your SIEM is a crown-jewel target; patch and segment it.
Sentinel/M365 "logging gap" (Storm-0558) 2023 Forged tokens; many orgs lacked the (then-paid) MailItemsAccessed logs Missing logs = invisible breach. Log availability is a security control; don't let licensing gate forensic logs.
Elastic/Kibana CVE-2019-7609 2019 Timelion prototype pollution → RCE SIEM UI components are exploitable; restrict access.
Chronicle/Mandiant UNC-attributed dwell time ongoing Median dwell time historically weeks Justifies the whole pipeline: faster normalize+detect = lower dwell.

Case study 1 — Capital One (2019). An SSRF flaw in a WAF let the attacker reach the EC2 Instance Metadata Service (IMDSv1), retrieve the instance role's temporary credentials, and use them to ListBuckets and sync ~100M records from S3. Crucially, the telemetry to catch this existed: CloudTrail recorded the anomalous s3:ListBuckets and mass GetObject from an unusual role, and VPC Flow Logs recorded the exfil volume. What was missing was the detection — no rule flagged a role suddenly enumerating buckets it never touched, and no baseline of per-role S3 access existed. The pipeline control that would have stopped it: enrichment tying the credential to its normal identity/asset behavior, plus a UEBA/threshold detection on "role accessing new bucket set + high egress." (See also ../../infra/Lateral-Movement/Cloud-Service-Accounts/README.md.)

Case study 2 — Storm-0558 / M365 (2023). A stolen Microsoft consumer signing key was used to forge Azure AD tokens and access Exchange Online mailboxes. Many victims could not confirm scope because the MailItemsAccessed audit event was behind a premium (E5/Purview) license — the log simply did not exist for them. This is the purest form of garbage-in via missing logs: your detections and forensics are bounded by what you collect. The lesson drove Microsoft to make more audit logs free by default in 2023–2024, and it is the strongest argument for treating log availability as a first-class security requirement, not a cost line to trim.

Case study 3 — SUNBURST (2020). The implant beaconed to C2 through domains and traffic patterns crafted to resemble legitimate Orion Improvement Program telemetry, and it checked for security tooling before activating. Signature/IOC detection failed because the IOCs were novel and short-lived. What could have helped: normalized network + process telemetry feeding behavioral baselines (new outbound domains from a server that historically only talked to update servers; unusual parent-child process trees), i.e., detection at the top of the Pyramid of Pain (TTPs), which the pipeline must be able to feed with enriched, long-retained data.

Top 5 in depth

1. Log-source health monitoring ("detect the silence"). The most damaging failures are silent — an agent stops, a firewall rotates a syslog target, a cloud sink loses IAM permission — and nobody notices until an incident needs those logs. The control: for every source, compute a rolling baseline of events-per-second/day and alert when volume drops below a floor (e.g., <20% of 7-day median for >N minutes). In Splunk this is a metadata/tstats search on latest(_time) per host/sourcetype; in Sentinel a scheduled KQL over Heartbeat/Usage; in Chronicle the Data Ingestion dashboards. Trade-off: sources with legitimately bursty/low volume (a DR site) generate false "silence" alerts, so per-source baselines and business-hours awareness are required. This is arguably the single highest-ROI detection you can build because it protects every other detection.

2. Normalization to a common data model + parser contract tests. Normalization is what lets one rule cover many sources, and parser drift is what silently breaks it. Implement mapping to OCSF/ECS/UDM as versioned config, and in CI run each parser against a corpus of real sample events asserting that key fields (src_ip, user.email, event.action) are populated and typed correctly. When a vendor changes format, the contract test fails in CI before the change blinds production. Trade-off: schema-on-write normalization is rigid — reprocessing history after a mapping fix requires re-ingest or lake reparse; mitigate by keeping raw payloads in the lake so you can replay. Measure a normalization coverage % (events mapped to a real event_type vs "generic") as an ongoing quality SLO.

3. Tiered storage + data-lake routing by value. Ingest cost is the constraint that quietly forces every other bad decision (short retention, dropped sources). Split logs into high-signal (auth, IAM, EDR, DNS, proxy) → hot SIEM index for 30–90 days, and high-volume/low-signal (verbose flow, debug, full packet) → cheap object lake (S3/GCS/BigQuery) as Parquet for 1–2 years, queried on demand (Athena/BigQuery/| from datamodel). Implementation: ingest-time routing (GCP Log Router filters, Sentinel Basic vs Analytics tiers, Cribl/logstash pipelines). Trade-off: lake queries are slower and schema-on-read, so a live investigation may need to "rehydrate" cold data into the SIEM — acceptable because incidents are rare relative to steady-state ingest. This pattern typically cuts SIEM license/ingest cost 40–70% while increasing retention.

4. Detection-as-code with CI/CD and backtesting. Rules stored in a console are unversioned, untested, and rot when schemas change. Manage detections as code: Sigma (portable) or native (KQL/.yaml, YARA-L) in Git; PRs run schema lint, unit tests against sample events, and a backtest against 30–90 days of real data to estimate false-positive rate before merge; a pipeline deploys to staging then production. This gives you auditability (who changed what, why), rollback, peer review of allowlists, and a coverage dashboard mapped to ATT&CK. Trade-off: upfront tooling/CI investment and a culture shift for analysts used to clicking; the payoff is a SOC whose detection quality compounds instead of decays. Tools: Sigma + sigma-cli, Panther, Chronicle rules API, Sentinel az sentinel/Bicep, Splunk Security Content/detection-testing.

5. Immutable, off-host, encrypted log storage. T1070 (Indicator Removal) is defeated by getting logs off the host in real time to a store the attacker cannot reach or alter. Controls: stream to a buffered bus immediately (so on-host deletion is too late), store in WORM/immutable buckets (S3 Object Lock, GCS retention policy/bucket lock), and TLS/mTLS all transport. Cloud audit logs help here because Admin Activity logs are Google/AWS-controlled and can't be disabled by a tenant admin. Trade-off: immutability + long retention raises storage cost and complicates GDPR "right to erasure" (mitigate with tokenization/field-level redaction at ingest). This control is what makes your logs trustworthy evidence, not just operational data.

6. Detection & Response

What to log / telemetry sources

Detecting attacks on the pipeline itself, plus feeding detections, requires:

Source Key fields Detection use
SIEM ingest metrics sourcetype, host, events/sec, latest(_time) Source silence, backpressure, cost spikes
Cloud audit logs methodName, principalEmail, callerIp, resourceName IAM changes, log-config tampering
Log-router/sink config changes sink filter edits, disabled sinks Attacker disabling telemetry (T1562.008)
Parser/detection deploy audit rule id, author, diff, deploy time Unauthorized rule disable/change
Agent/forwarder health agent status, last checkin Silent agent death
SIEM access logs user, query, export, admin action Insider misuse of the SIEM itself
Windows/Sysmon, DNS, proxy, EDR (per §IDS-SIEM note) Core attacker-activity detections

Specifically monitor "defense evasion via the pipeline": DeleteSink/UpdateSink (GCP), StopLogging/DeleteTrail/PutEventSelectors (AWS CloudTrail), diagnostic-setting deletion (Azure), and disabling of detection rules.

(b) SQL (data-lake / BigQuery over normalized audit logs) — log-source went silent:

-- Alert when a source that normally reports has produced no events in 60 min
WITH baseline AS (
  SELECT source_host,
         APPROX_QUANTILES(events_per_hour, 100)[OFFSET(50)] AS median_eph
  FROM `sec.log_volume_hourly`
  WHERE event_hour BETWEEN TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
                       AND TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR)
  GROUP BY source_host
),
current_vol AS (
  SELECT source_host, COUNTIF(TRUE) AS eph
  FROM `sec.normalized_events`
  WHERE event_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 60 MINUTE)

Response playbook (pipeline-integrity incident)

  1. Detect & validate. Confirm the trigger (silence alert, CloudTrail-disabled, rule-disabled). Rule out a legitimate change ticket. Snapshot the current pipeline config.
  2. Contain. If an attacker disabled logging/sinks: re-enable the trail/sink immediately, or fail over to the redundant sink you maintain in a separate account/project. Revoke the credential that made the change. Freeze detection-rule config changes.
  3. Preserve evidence. Pull the audit trail of who changed the config from the cloud provider's own immutable Admin Activity log (attacker usually can't alter that). Export to the immutable/WORM store.
  4. Eradicate. Remove attacker access (rotate keys, revoke sessions/tokens), close the initial vector (SSRF, exposed key), and delete any malicious IAM bindings.
  5. Assess blind window. Determine the time range the pipeline was blind. Backfill from any surviving source (cloud-side logs, lake, redundant collector). Explicitly document what cannot be recovered.
  6. Recover. Restore normal routing/retention, re-run affected detections over the backfilled window, verify normalization coverage returned to baseline.
  7. Post-incident hardening. Add/repair the "detect silence" and "logging-disabled" detections, make the disabled config immutable where possible, add alerting on the specific change API, and run a purple-team test that the blind spot is now covered.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What are the stages of a security log pipeline, end to end?
Collection (agents, syslog, cloud sinks) → transport/buffering (Kafka/Pub/Sub) → parsing (raw to structured) → normalization (structured to a common data model like OCSF/ECS/UDM) → enrichment (asset, identity, GeoIP, threat intel) → routing (by value/tier) → storage (hot/warm/cold + lake) → detection execution → alerting/response. The hard engineering is in normalization consistency and ingest economics, not the correlation engine.
Signal: do they see SIEM as a pipeline vs a magic box.

Q2. Why normalize logs to a common data model? Name some models.
Because attackers cross sources: one intrusion touches VPN, endpoint, cloud IAM, and proxy. Normalization maps every source's fields into one schema (user.email, source.ip, event.action) so you write one detection instead of N vendor-specific ones, and analysts learn one field set. Models: OCSF (open, vendor-neutral, backed by AWS/Splunk), ECS (Elastic), Chronicle UDM (Google), ASIM (Sentinel). Without it, cross-source correlation is brittle and rules multiply.
Signal: understands normalization as the enabler of correlation.

Q3. What is the difference between a SIEM and a data lake, and why use both?
A SIEM optimizes for fast, indexed search and real-time detection but is expensive per GB; a data lake (S3/GCS/BigQuery) optimizes for cheap, durable, schema-on-read storage at scale but slower/costlier per query. You route high-signal logs to the SIEM for detection and everything to the lake for retention/forensics, federating queries when an investigation needs cold data. This controls cost while extending retention.
Signal: grasps cost/scale tradeoff, not "SIEM = everything."

Q4. What does "garbage in, garbage out" mean here, and what's the most common form?
Detections and forensics are bounded by data quality and completeness. The most common failures: missing logs (source never onboarded), silent source outages (agent died, nobody noticed), and parsing drift (vendor changed format, fields go null). All three cause silent misses — the rule runs but sees nothing. That's why log-source health monitoring is foundational.
Signal: prioritizes data quality over fancy analytics.

Q5. What is detection-as-code and why does it matter?
Managing detection rules as version-controlled code (Sigma/KQL/YARA-L in Git) with CI tests, backtesting, peer review, and pipelined deploys — instead of clicking rules into a console. It brings auditability, rollback, review of allowlists, and coverage dashboards, and prevents rule rot when schemas change. It's the practice that lets detection quality compound rather than decay.
Signal: engineering maturity, treats detections like software.

Tier 2 — System design / trade-offs

Q6. Design a log pipeline for an org generating 50 TB/day. How do you keep it affordable without going blind?
Start by classifying sources by detection value: auth/IAM/EDR/DNS/proxy are high-signal (maybe 5–10 TB) and go to the hot SIEM index for 30–90 days; verbose VPC flow, debug, and full-packet (the other 40 TB) go to a Parquet data lake for 1–2 years, queried on demand. Put a buffered bus (Kafka/Pub/Sub) in front so spikes don't drop events, and do ingest-time filtering/redaction (drop known-noise EventIDs, redact secrets) before the expensive tier. Normalize once to OCSF/UDM so both SIEM and lake share a schema and detections are portable. Add log-source health monitoring and a normalization-coverage SLO so cost cuts don't silently create blind spots. The result: high-signal real-time detection + long cheap retention, typically cutting SIEM cost 40–70%.
Signal: balances cost, coverage, and reliability with concrete tiering.

Q7. Traditional SIEM (Splunk) vs cloud-native (Chronicle/Sentinel/Security Lake) — when do you pick which?
Traditional SIEMs give a mature query language (SPL), huge app ecosystem, and deployment flexibility, but bill on ingest volume, which punishes scale and pressures teams to under-collect. Cloud-native SIEMs decouple storage/compute, fix a normalization schema (UDM/ASIM/OCSF), scale to petabytes, and often price on flat ingest or query compute — better economics at large volume and less infra to run, but less flexible and cloud-lock-in. I'd pick Chronicle/SecOps for very high volume + Google ecosystem, Sentinel for Microsoft/M365-heavy shops (native AAD/Defender integration), Security Lake+OCSF when I want an open lake feeding multiple tools, and keep Splunk where deep SPL/app investment and on-prem/regulated data dominate. Many mature orgs run a hybrid: cloud-native/lake for volume, Splunk for specific high-value analytics.
Signal: nuanced, ecosystem-aware, not a fanboy answer.

Q8. How do you handle timestamps and ordering across thousands of sources?
Distinguish event-time (when it happened at the source) from ingest-time (when the pipeline received it) and store both; correlation windows must key on event-time or clock skew and delayed batches break them. Enforce NTP everywhere, normalize all timestamps to UTC at parse, and handle late-arriving data with watermarking (like stream processing) so a detection window can tolerate bounded lateness. For sources with unreliable clocks, flag skew and prefer ingest-time with a documented caveat. Without this, "N failures in 60s" rules produce both false negatives (spread across skewed windows) and false positives.
Signal: understands distributed-systems reality of logging.

Q9. Where do you enrich, and what are the trade-offs of ingest-time vs query-time enrichment?
Enrichment adds asset criticality, identity/CMDB context, GeoIP, and threat-intel matches. Ingest-time enrichment bakes context into the stored event (fast queries, stable point-in-time context — good for "was this host a prod DB at the time"), but bloats storage and can't be corrected retroactively. Query-time enrichment joins against current lookup tables (always fresh, cheap storage, retroactively correctable), but slower queries and loses historical accuracy if the asset changed. I default to ingest-time for slow-changing high-value context (asset criticality, data classification) and query-time for volatile data (current threat intel, current owner). Always TTL enrichment sources so stale CMDB doesn't silently misclassify severity.
Signal: depth on a subtle, real design choice.

Q10. How do you prevent detection coverage from silently rotting as sources and schemas change?
Three loops: (1) parser contract tests in CI that assert key fields populate against real sample events, so a vendor format change fails the build, not production; (2) detection-as-code with backtesting so every rule is versioned, tested, and its FP rate measured before deploy, with alerts on rules that stop firing entirely; (3) coverage dashboards mapping active detections and available log sources to ATT&CK (DeTT&CT/attack-navigator) to surface gaps. Add log-source health monitoring so a dead source is caught in minutes. Together these turn silent decay into loud, actionable signals.
Signal: systemic thinking about maintainability, not one-off rules.

Tier 3 — Expert / deep-dive

Q11. An attacker with cloud admin creds disables your log sink for 3 hours, then re-enables it. Walk through detection, blast-radius assessment, and hardening.
Detection: the DeleteSink/UpdateSink/StopLogging API call is itself recorded in the provider's Admin Activity audit log, which a tenant admin cannot disable — a high-severity detection should fire on it; independently, the log-source-silence monitor should fire within minutes of volume dropping. Blast radius: the 3-hour window is a blind spot for tenant-controlled logs, but provider-side Admin Activity logs and any redundant sink (in a separate security project/account) survive — pull those to reconstruct what happened, and backfill from the lake if it fed from a different path. Rotate the abused credential, revoke sessions, and remove malicious IAM. Hardening: maintain a redundant sink in an isolated account the workload identity can't touch; make the sink config immutable/locked; alert specifically on logging-config-change APIs; and run a purple-team test confirming both the API-change detection and the silence detection now fire. Document precisely what data is unrecoverable.
Signal: understands the meta-game of logs as an attack surface and provider-side immutability.

Q12. Compare OCSF, ECS, and Chronicle UDM. Why did OCSF emerge and what problem does it solve that ECS didn't?
ECS is Elastic's schema — excellent and widely used, but governed by one vendor and centered on the Elastic Stack, so cross-vendor adoption meant depending on a competitor's roadmap. Chronicle UDM is Google-specific, purpose-built for Chronicle's detection engine (YARA-L operates over UDM). OCSF (launched 2022, backed by AWS, Splunk, IBM, CrowdStrike, and many others) emerged to be a vendor-neutral, governed-in-the-open taxonomy with a formal class/attribute model and event categories, so any producer/consumer can interoperate without picking a vendor's side — Amazon Security Lake outputs OCSF Parquet natively. The practical difference: ECS/UDM optimize for their own platforms; OCSF optimizes for a multi-tool ecosystem and a data-lake-centric world. In practice you may normalize to OCSF in the lake and map into UDM/ASIM at the SIEM. All three share the same core idea — categories, normalized field names, typed values — the politics of governance is the differentiator.
Signal: current-events awareness + understands schema governance economics.

Q13. Design detection execution for both streaming (low-latency) and batch (retrospective) over the same data. What's the architecture?
Use a Lambda/Kappa-style split. Streaming path: events on the bus (Kafka/Pub/Sub) are evaluated by a low-latency engine (Flink, Chronicle's real-time rules, Sentinel near-real-time rules) for detections needing minutes-level response (impossible travel, C2 beaconing) — stateful windowing handles "N in T." Batch/retrospective path: normalized data lands in the lake, and scheduled queries (BigQuery, Athena, Splunk scheduled search) run heavier analytics, ML/UEBA baselines, and retro-hunts when new IOCs/TTPs emerge (e.g., re-scan 90 days after a new CVE). Critical design points: both paths consume the same normalized schema so a rule is portable; you keep raw + normalized in the lake to replay history after a mapping fix; and you reconcile so a streaming detection and its batch equivalent don't double-alert. Retrospective execution is what turns fresh threat intel into detection over historical data — the streaming path alone can't catch what you didn't yet know to look for.
Signal: mature architecture thinking, knows retro-hunt matters.

Q14. Your SIEM ingest bill just doubled with no obvious cause. How do you diagnose and remediate without creating blind spots?
Diagnose by volume attribution: break ingest down by source/sourcetype/index over time (| tstats count by sourcetype / Sentinel Usage table / Chronicle ingestion metrics) to find the delta — common causes are a newly-verbose source (debug logging left on after a deploy), a misconfigured duplicate/looping sink, a chatty new app, or a security event storm (which might itself be an incident). Correlate the spike start with change tickets and deploys. Remediate by value, not by blindly cutting: if it's debug noise, turn off debug at the source or drop it at ingest; if it's a loop, fix the fan-out; if it's genuinely useful high-volume data, route it to the cheap lake tier instead of the hot index rather than dropping it. Before any nullQueue/DCR drop, do a data-value review (does any detection use these fields?) and sample-retain for a period. Guardrail: put budget/anomaly alerts on ingest volume itself so this is caught in hours, and never let a cost panic silently delete a security-relevant source.
Signal: operational maturity — cost control without coverage loss.

Q15. How would you architect the pipeline to defend the SIEM itself as a crown-jewel target, given Log4Shell and Splunk/Elastic CVEs?
Treat the SIEM as tier-0 infrastructure. Attack surface reduction: keep it in a segmented network with strict ingress/egress (it should talk to sources and analysts, not the internet); patch on an aggressive SLA and subscribe to vendor advisories (Log4Shell taught us the parsing layer is exploitable via crafted log content, so sanitize/validate fields on parse and disable dangerous features like JNDI lookups). Identity: least-privilege service accounts for collectors, MFA + SSO + RBAC for analysts, and separate admin from read/search roles. Data protection: TLS/mTLS all transport, encrypt at rest, and store logs immutably (WORM) off-host so a SIEM compromise can't erase the evidence of it. Detection on the SIEM itself: log and alert on admin actions, rule/parser changes, bulk exports, and anomalous queries (insider misuse). Resilience: a redundant collection path in a separate trust domain so compromising the primary SIEM doesn't blind you. The principle: the system that watches everything must itself be watched, patched, segmented, and unable to destroy its own evidence.
Signal: thinks adversarially about the defensive infrastructure itself.

9. Key Takeaways

  • SIEM = pipeline, not product. Value and failure both live in collection → normalize → enrich → tier → detect. Interviewers reward candidates who reason about the pipeline, not the console.
  • Normalization to a common data model (OCSF/ECS/UDM/ASIM) is the multiplier — it turns N vendor-specific rules into one portable detection and makes cross-source correlation possible.
  • Garbage-in dominates missed detections: missing logs, silent source outages, and parsing drift cause more misses than weak rule logic. Monitor log-source health as a first-class SLO.
  • Cost is the real constraint. Tiered storage + a data lake (hot SIEM for high-signal, cheap Parquet lake for everything, long retention) is how you avoid the "cut logs to save money → go blind" trap.
  • Detection-as-code is the maturity line. Rules in Git, tested and backtested in CI, deployed via pipeline — this is what makes detection quality compound instead of rot.
  • Logs are an attack surface and an attacker target. T1562/T1070: adversaries disable sinks and clear logs. Provider-side immutable audit logs, redundant sinks, and WORM off-host storage are what preserve evidence.
  • Having logs ≠ detecting (Capital One): telemetry without detection logic and baselines is just storage cost.
  • Missing logs = invisible breach (Storm-0558): treat log availability as a security requirement, not a line-item to trim.
  • Traditional vs cloud-native is an economics + ecosystem choice, not a quality ranking; large-volume, cloud-centric orgs increasingly favor cloud-native + lake + OCSF, often hybridized with Splunk for deep analytics.
  • Distinguish event-time from ingest-time and normalize to UTC — correlation windows silently break on clock skew and late data otherwise.
