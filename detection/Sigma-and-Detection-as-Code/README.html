Sigma and Detection-as-Code

Category: Detection Engineering — Portable Detection Content, CI/CD for Detections
Related notes: detection/Detection-Engineering/README.md, detection/MITRE-ATTACK/README.md, detection/IDS-SIEM/README.md, detection/Anomaly-Detection/README.md
Anchors: Sigma specification (SigmaHQ), pySigma, sigma-cli, MITRE ATT&CK, the Alerting & Detection Strategy (ADS) framework, Palantir/Elastic detection-rules repos

1. Overview

Sigma is a vendor-neutral, YAML-based signature format for describing detections against log data. It plays the same role for SIEM log analytics that Snort/Suricata rules play for network traffic and YARA plays for files: one portable, human-readable rule that a compiler (sigma-cli/pySigma) translates into whatever backend query language your platform speaks — Splunk SPL, Elastic Query DSL / Lucene / EQL, Microsoft Sentinel/Defender KQL, Google Chronicle YARA-L, Elasticsearch ES|QL, Loki LogQL, and dozens more. Detection-as-Code (DaC) is the operating model wrapped around that format: detections are treated as source code — version-controlled in Git, peer-reviewed via pull requests, unit-tested against known log samples, linted, validated, and deployed to production through CI/CD pipelines rather than click-ops in a SIEM console.

At scale this matters because a detection estate is not 50 rules — it is thousands, spanning cloud audit logs (CloudTrail, GCP Cloud Audit Logs, Azure Activity), Kubernetes audit, EDR telemetry, identity providers, and application logs, across multiple SIEMs acquired through acquisitions and regional data-residency constraints. Without a portable format and a code pipeline, detections rot: they are undocumented, untested, tied to one vendor, and silently break when a log schema changes. Sigma + DaC turns detection engineering into a software-engineering discipline with the same guarantees — reproducibility, testability, review, and rollback.

TL;DR
- Sigma is a generic detection language (logsource + detection + condition); a converter (pySigma/sigma-cli) plus a pipeline maps abstract fields to a concrete backend query. The pipeline is where the real portability work happens.
- Detection-as-Code = detections in Git, PR-reviewed, unit-tested against saved log samples, CI-validated, and deployed by pipeline — not hand-edited in the SIEM.
- Write behavior-based, robust rules mapped to ATT&CK techniques; avoid brittle single-IOC rules (hashes, IPs, exact command strings) that an attacker trivially mutates.
- Test the detection, not the attack: replay a known-malicious log sample (true positive) and a known-benign sample (true negative) and assert the rule fires exactly on the malicious one. This is validation of the detection logic, distinct from running offensive tooling.
- Manage the rule lifecycle: idea → ADS/design doc → draft → test → PR review → staged deploy → tune (FP feedback) → measure (coverage, MTTD) → deprecate.

2. How It Works

2.1 Core mechanics

A Sigma rule is a YAML document with a fixed set of top-level keys. The heart is the detection block: a set of named search identifiers (each a map or list of field/value matchers) combined by a boolean condition expression. The converter walks that AST, applies a processing pipeline to translate Sigma's abstract taxonomy (field names, log sources, value modifiers) into the target's real schema, and emits a backend query string.

Concept What it is Notes
title / id Human name + stable UUID id is a UUID4; never reuse it — it is the rule's identity across renames
status experimental / test / stable / deprecated / unsupported Drives what a pipeline promotes to prod
logsource product / category / service triple Abstract selector (e.g. product: aws, service: cloudtrail) resolved by the pipeline to real indices/tables
detection Named search identifiers + condition Each identifier is field:value matchers combined with AND (within a map) and OR (across a list)
condition Boolean expression over identifiers Supports and/or/not, 1 of selection*, all of them, | count() by ... aggregations, near
Field modifiers |contains, |startswith, |endswith, |re, |base64, |cidr, |all, |gte, |lt, |expand, |cased Suffix on a field key that changes match semantics — the portability lever
fields Columns to surface in the alert Analyst context, not part of the match logic
falsepositives Known benign triggers Documentation for the triage analyst; also feeds tuning
level informational→low→medium→high→critical Severity; often mapped to SOAR routing
tags attack.tXXXX, attack.<tactic>, cve.YYYY.N ATT&CK/CVE mapping; enables coverage reporting
Pipeline pySigma ProcessingPipeline Ordered transformations: field mappings, logsource→index, value transforms, error/placeholder handling
Backend pySigma plugin Emits the final query (e.g. pySigma-backend-splunk)
Correlation rules Sigma "meta-rules" Newer spec: count, value_count, temporal, temporal_ordered across referenced base rules

Match semantics you must know cold:
- Inside a single search-identifier map, all key: value pairs are AND-ed.
- A list of values for one key is OR-ed (Image|endswith: ['\a.exe', '\b.exe'] = ends with a.exe OR b.exe).
- A search identifier expressed as a list of maps OR-s the maps.
- all of selection* / 1 of selection* wildcard-match identifier names.
- Values are case-insensitive by default and support * (any) and ? (single char) wildcards; use |cased and |re when you need exactness.

The three-stage conversion is the mental model to carry into a design review:

flowchart LR
  A[Sigma YAML rule] --> B[pySigma parser builds AST]
  B --> C[Processing pipeline maps fields and logsource]
  C --> D[Backend emits query]
  D --> E[Splunk SPL]
  D --> F[Elastic KQL or EQL]
  D --> G[Chronicle YARA-L]
  D --> H[Sentinel KQL]

2.2 Concrete configuration

Install the toolchain. sigma-cli is the CLI wrapper around pySigma; backends and pipelines install as plugins.

# sigma-cli + pySigma
pipx install sigma-cli
sigma plugin list                         # see available backends/pipelines
sigma plugin install splunk               # pySigma-backend-splunk + splunk pipelines
sigma plugin install elasticsearch        # Elastic (Lucene/DSL/EQL/ES|QL)
sigma plugin install cortexxdr            # example vendor backend

# Convert one rule to Splunk SPL
sigma convert -t splunk -p splunk_windows rules/cloud/aws_iam_backdoor_role.yml

# Convert a whole folder to Elastic Query String, apply the ECS pipeline,
# and wrap results as importable detection-rule NDJSON
sigma convert -t elasticsearch -f siem_rule_ndjson -p ecs_windows \
  -o out/elastic_rules.ndjson rules/

# List which pipelines a backend supports
sigma convert -t splunk --without-pipeline rules/x.yml   # no field mapping (debug)

A processing pipeline in YAML — this is what makes Sigma portable. It rewrites Sigma's abstract logsource and field names into your real schema. Example: map the AWS CloudTrail taxonomy onto a Splunk index and sourcetype and rename fields.

# pipelines/aws_cloudtrail_splunk.yml
name: aws-cloudtrail-splunk
priority: 20
transformations:
  - id: cloudtrail_index
    type: add_condition
    conditions:
      index: cloudtrail
      sourcetype: aws:cloudtrail
    rule_conditions:
      - type: logsource
        product: aws
        service: cloudtrail
  - id: field_rename
    type: field_name_mapping
    mapping:
      eventName: eventName          # already ECS-ish; explicit for clarity
      userIdentity.arn: userIdentity.arn
      requestParameters.roleName: requestParameters.roleName
    rule_conditions:
      - type: logsource
        product: aws
        service: cloudtrail
  - id: fail_unsupported
    type: detection_item_failure
    message: "Field not mapped for CloudTrail on Splunk"
    rule_conditions:
      - type: logsource
        product: aws
        service: cloudtrail
    field_name_conditions:
      - type: include_fields
        fields: []                  # any unmapped field errors out loudly

Rule example 1 — Suspicious cloud IAM change (AWS CloudTrail). Detects an identity attaching the AdministratorAccess managed policy or an inline admin policy to a principal — a classic privilege-escalation / backdoor step (ATT&CK T1098 Account Manipulation, T1078 Valid Accounts).

title: AWS IAM AdministratorAccess Attached To Principal
id: 8f2b1c4e-6d3a-4a71-9d2e-1f4c7a9b0c31
status: test
description: >
  Detects attachment of the AWS-managed AdministratorAccess policy, or creation of
  an inline policy granting Action=* Resource=*, to any IAM user, group, or role.
  Frequently used for privilege escalation and persistence after credential theft.
references:
  - https://attack.mitre.org/techniques/T1098/
author: detection-eng
date: 2026/07/07
tags:
  - attack.persistence
  - attack.privilege-escalation
  - attack.t1098
  - attack.t1078
logsource:
  product: aws
  service: cloudtrail
detection:
  managed_admin:
    eventSource: iam.amazonaws.com
    eventName:
      - AttachUserPolicy
      - AttachRolePolicy
      - AttachGroupPolicy
    requestParameters.policyArn|endswith: ':policy/AdministratorAccess'
  inline_star:
    eventSource: iam.amazonaws.com
    eventName:
      - PutUserPolicy
      - PutRolePolicy
      - PutGroupPolicy
    requestParameters.policyDocument|contains|all:
      - '"Effect":"Allow"'
      - '"Action":"*"'
      - '"Resource":"*"'
  success:
    errorCode: null            # no error code field => call succeeded
  condition: (managed_admin or inline_star) and success
fields:
  - userIdentity.arn
  - eventName
  - requestParameters.policyArn
  - sourceIPAddress
  - awsRegion
falsepositives:
  - Break-glass admin provisioning by IAM automation (allowlist the CI role ARN)
  - Sanctioned Terraform/CloudFormation runs (correlate with pipeline principal)
level: high

Converting Rule 1 to a backend query. Running sigma convert -t splunk -p aws_cloudtrail_splunk rules/cloud/aws_iam_admin_attached.yml yields Splunk SPL similar to:

index=cloudtrail sourcetype=aws:cloudtrail
(
  (eventSource="iam.amazonaws.com"
    (eventName="AttachUserPolicy" OR eventName="AttachRolePolicy" OR eventName="AttachGroupPolicy")
    "requestParameters.policyArn"="*:policy/AdministratorAccess")
  OR
  (eventSource="iam.amazonaws.com"
    (eventName="PutUserPolicy" OR eventName="PutRolePolicy" OR eventName="PutGroupPolicy")
    "requestParameters.policyDocument"="*\"Effect\":\"Allow\"*"
    "requestParameters.policyDocument"="*\"Action\":\"*\"*"
    "requestParameters.policyDocument"="*\"Resource\":\"*\"*")
)
NOT errorCode=*
| table userIdentity.arn eventName requestParameters.policyArn sourceIPAddress awsRegion

The same rule, targeting Elastic with -t lucene -p ecs_cloudtrail, becomes a Lucene/KQL query using ECS field names (aws.cloudtrail.event_name, aws.cloudtrail.user_identity.arn) — that field remapping is done entirely by the pipeline, not by editing the rule. For Chronicle, -t chronicle emits YARA-L 2.0 with events: matchers and a condition: block. The rule author never touches vendor syntax; the pipeline owns the translation.

Case study — Capital One (2019). An SSRF flaw let an attacker reach the EC2 instance metadata service, retrieve temporary credentials for the WAF's IAM role, and use them to enumerate and download ~100M records from S3. The credentials were legitimate (T1078), so no signature-IOC would fire. The detectable behavior was contextual: role credentials minted for an in-VPC workload were suddenly presented from an external IP with a burst of ListBuckets and thousands of GetObject calls. A Sigma rule on CloudTrail keying on "instance-role credential used from a sourceIPAddress outside AWS ranges" plus a correlation on S3 read volume would have surfaced it within minutes. The lesson that drives DaC: the durable detection is a behavior mapped to ATT&CK, version-controlled and tested, not an IP block.

Case study — Log4Shell (2021). Within hours of disclosure, SigmaHQ and vendors published Sigma rules matching JNDI patterns (${jndi:ldap://…}) in web logs and process-creation events for suspicious Java child processes. Because they were Sigma, a Splunk shop, an Elastic shop, and a Sentinel shop all deployed the same reviewed logic by running their converter — no re-authoring in three dialects. This is the portability dividend: incident response speed scales with the number of backends your one rule compiles to. The follow-up lesson was equally important — the naive contains jndi:ldap rule was quickly evaded by obfuscation (${${lower:j}ndi:…}), which is exactly why rule quality (regex normalization, decode modifiers) and a tuning loop matter.

Case study — SolarWinds (2020). SUNBURST dwelled for months using signed updates and valid credentials, deliberately avoiding known IOCs. Detection ultimately came from anomalous identity behavior — new federation trust relationships, tokens issued with unusual claims, service principals granted new permissions. These are precisely the events that make durable Sigma rules on cloud audit and identity logs (T1098, T1550.001). The incident retired the industry's faith in IOC feeds as a primary defense and elevated behavior-based, ATT&CK-mapped, code-managed detections.

Top 5 in depth

1. Detect at the TTP level, not the IOC level. The single highest-leverage control is where on the Pyramid of Pain you anchor the rule. Hashes, IPs, and domains are cheap for an attacker to rotate; TTPs (the actual technique — e.g. "attach admin policy," "create SA key," "sudo to shell as a service account") are expensive to change because they are how the attack fundamentally works. Concretely, Rule 1 keys on the CloudTrail eventName + the semantic shape of the policy (Action:* Resource:*), not on a specific role name or IP. Trade-off: TTP rules are broader and can be noisier, so they demand good filters and a tuning loop. IOC rules still have a place — but as time-boxed IR artifacts with an expiry, not the backbone of the estate. Interviewers want to hear you make this distinction explicitly.

2. Robust matching via modifiers and normalization. A rule that matches the exact string powershell -enc SQBFAF... is defeated by casing, whitespace, or a different base64 blob. Robustness comes from Sigma's modifiers: |contains|all to require several independent substrings, |base64offset|contains to catch encoded payloads at any alignment, |re for a normalized regex, and |cased only where exactness is genuinely needed. The pipeline can also insert value transforms (lowercasing, unescaping) before matching. Trade-off: regexes are slower and risk catastrophic backtracking on large fields, so lint for unbounded quantifiers and prefer anchored, bounded patterns. The goal is a rule that survives trivial mutation while staying cheap to evaluate at ingest volume.

3. Two-sided testing in CI (validate the detection, not the attack). Every rule ships with at least one true-positive log sample (a real, sanitized malicious event that MUST match) and one true-negative sample (a benign look-alike that MUST NOT match). CI converts the rule and runs it against both, asserting exactly one hit and zero hits respectively. This is fundamentally different from running offensive tooling: we are not launching an attack, we are replaying a recorded log line to prove the compiled query behaves. Where atomic-red-team-style thinking applies is as the source of representative telemetry — a technique is executed once in a lab, the resulting logs are captured, sanitized, and committed as fixtures. Trade-off: maintaining representative samples is real work and they drift from prod schemas; mitigate by seeding fixtures from sanitized production logs and re-validating on a schedule. This control is what turns "I think this rule works" into a reproducible guarantee.

4. Pipeline-owned portability with fail-loud on unmapped fields. The processing pipeline is the contract between abstract Sigma and concrete backends. Its most important safety property is failing loudly: if a rule references a field the pipeline doesn't know how to map, a detection_item_failure transform must error the build rather than silently emitting an empty or wrong query — the worst outcome, a rule that looks deployed but matches nothing. Implement per-logsource pipelines (CloudTrail→Splunk, auditd→Elastic ECS) with explicit field_name_mapping and an index/table resolver. Trade-off: pipelines are code too and need their own tests; a bug in a shared pipeline breaks every rule that uses it, so version and review pipelines with the same rigor as rules.

5. Git as the single source of truth with drift reconciliation. Detections live in Git and reach production only through the pipeline; nobody edits rules in the SIEM console. A scheduled reconciliation job diffs what's deployed against what's in main and alerts on drift — catching both well-meaning click-ops and malicious tampering (T1562.001). Combine with RBAC so few humans can mutate deployed detections directly. Trade-off: this slows down the "quick fix in prod at 2am" instinct; the mitigation is a fast, well-tested emergency-merge path (short-lived branch, expedited review) so the pipeline is never the reason an incident response stalls. Reproducibility and auditability are worth the discipline at scale.

What to log / telemetry sources

Detection content is only as good as its inputs. Key sources and the fields Sigma rules key on:

Instrument the pipeline as a first-class telemetry source: every deploy, every CI test pass/fail, and every alert's triage outcome (TP/FP/benign) must be logged so you can compute precision, recall proxies, and MTTD per rule.

Detection A — meta-detection: someone disabled or deleted a detection rule in the SIEM (tamper detection). Sigma over Splunk audit logs:

Detection C — the true-positive/true-negative CI assertion (pytest-style harness). This is how you test the detection, not the attack:

# tests/test_iam_admin_rule.py
import json, subprocess, re

def compile_rule(path, target="splunk", pipeline="aws_cloudtrail_splunk"):
    out = subprocess.check_output(
        ["sigma", "convert", "-t", target, "-p", pipeline, path], text=True)
    return out.strip()

def matches(query, sample):
    # minimal evaluator: check eventName + policyArn semantics against one log line
    ev = json.loads(sample)
    return (ev["eventSource"] == "iam.amazonaws.com"
            and ev["eventName"] in ("AttachUserPolicy","AttachRolePolicy","AttachGroupPolicy")
            and ev["requestParameters"]["policyArn"].endswith(":policy/AdministratorAccess")
            and "errorCode" not in ev)

def test_true_positive():
    q = compile_rule("rules/cloud/aws_iam_admin_attached.yml")
    tp = open("tests/samples/attach_admin_success.json").read()
    assert matches(q, tp), "rule must fire on the known-malicious sample"

def test_true_negative():
    q = compile_rule("rules/cloud/aws_iam_admin_attached.yml")
    tn = open("tests/samples/attach_readonly_success.json").read()
    assert not matches(q, tn), "rule must stay silent on benign ReadOnlyAccess attach"

In a real harness the "evaluator" is the SIEM itself in a container (e.g. a Splunk/Elastic test instance) or a purpose-built log-replay engine; the assertion contract is identical — exactly one hit on the TP fixture, zero on the TN fixture. This validates detection logic and packaging; it deliberately does not execute any attack tooling.

Response playbook

When a Sigma-driven alert fires (using Rule 1 — IAM admin attached — as the worked example):

  1. Triage / validate (minutes 0–10). Confirm the alert is a true positive: pull the raw CloudTrail record, check userIdentity.arn, sourceIPAddress, and whether the principal is the sanctioned automation role. Tag the alert outcome (feeds the precision metric).
  2. Contain. If confirmed malicious, immediately detach the offending policy and attach a deny-all boundary; if the actor is a compromised principal, disable its access keys / revoke its sessions (aws iam update-access-key --status Inactive, or revoke the role's active sessions via an AWSRevokeOlderSessions inline policy).
  3. Scope. Query CloudTrail for all actions by that principal and from that sourceIPAddress in the surrounding window; look for lateral movement (assumed roles, new keys, data access). Pivot to related Sigma rules (Rule 2 key-creation, S3 mass-read) via shared entities.
  4. Eradicate. Remove any persistence the actor created (new IAM users, access keys, trust policies, Lambda backdoors), rotate exposed credentials, and invalidate leaked tokens.
  5. Recover. Restore least-privilege policies, re-enable legitimate automation, confirm no residual access.
  6. Detection feedback (close the loop). File the outcome back into the detection repo: if it was a FP, add a filter/allowlist and a new true-negative fixture; if TP, verify the rule's coverage and consider a correlation rule for the broader kill chain. Update the rule's falsepositives and, if needed, level. Every incident should leave the detection content measurably better.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What are the required components of a Sigma rule and how does condition combine them?
A Sigma rule needs at minimum a title, a logsource (product/category/service selector), and a detection block containing one or more named search identifiers plus a condition. Each search identifier is a map of field:value matchers; pairs within a map are AND-ed and a list of values for one key is OR-ed. The condition is a boolean expression over those identifier names (and, or, not, 1 of them, all of selection*, aggregations). id (a UUID), level, and tags are strongly recommended for lineage, severity, and ATT&CK mapping.
Why asked / signal: Confirms the candidate can actually read and write the format, not just name-drop it.

Q2. How does one Sigma rule end up as a Splunk query and an Elastic query?
pySigma parses the YAML into an abstract syntax tree, then a processing pipeline maps Sigma's abstract taxonomy — logsource → index/table, and abstract field names → the backend's real schema (e.g. ECS) — after which a backend plugin serializes the AST into the target query language. sigma-cli is the CLI front-end. The rule author writes once; the pipeline+backend own all vendor-specific translation, which is the source of Sigma's portability.
Why asked / signal: Tests understanding that portability lives in the pipeline, not in the rule text.

Q3. What is Detection-as-Code and why adopt it?
Detection-as-Code treats detections as software: stored in Git, peer-reviewed via pull requests, unit-tested against known log samples, linted, and deployed through CI/CD instead of hand-editing in a SIEM console. Benefits are reproducibility, versioning/rollback, testability, review, and auditability — plus portability when the content is Sigma. It converts detection engineering from tribal click-ops into an engineering discipline with measurable quality.
Why asked / signal: Baseline maturity check; separates tool users from engineers.

Q4. What are field modifiers and give three examples.
Modifiers are suffixes on a field key that change match semantics: |contains (substring), |startswith/|endswith (anchored substring), |re (regex), |base64offset|contains (match base64-encoded value at any alignment), |cidr (IP in range), |all (require all listed values), |gte/|lt (numeric compare), |cased (case-sensitive). They are the primary tool for writing robust matches that survive attacker mutation.
Why asked / signal: Practical fluency; modifiers are where rule quality is won or lost.

Q5. Why prefer ATT&CK-mapped behavioral rules over IOC lists?
Because of the Pyramid of Pain: hashes, IPs, and domains are trivial for an attacker to rotate, while TTPs are expensive to change since they are how the attack fundamentally operates. Behavioral rules mapped to ATT&CK techniques stay valid across campaigns and let you build coverage heatmaps; IOC rules are still useful but belong as time-boxed IR artifacts, not the backbone. SolarWinds and Capital One both evaded IOC-based defenses but exhibited detectable behaviors.
Why asked / signal: Core detection-engineering philosophy.

Tier 2 — System design / trade-off

Q6. Design a detection-content pipeline for a company with Splunk in one region and Elastic in another.
Author all detections as Sigma in a single Git monorepo. On PR, CI runs: schema lint (sigma check), unique-UUID and required-tags lint, secret scanning, then conversion to both Splunk SPL and Elastic queries using region-specific pipelines, and finally replays each rule's true-positive and true-negative fixtures against ephemeral Splunk/Elastic test containers, asserting exactly-one and zero matches. On merge, a CD job deploys converted content to each region's staging SIEM in shadow mode; a reconciliation job diffs deployed vs. main to catch drift. Promotion to prod is gated on a false-positive budget observed in staging. Metrics (precision from alert-outcome tags, coverage from ATT&CK tags) feed a dashboard. The key design point is that one reviewed rule compiles to both backends, so logic never diverges between regions.
Why asked / signal: Tests end-to-end pipeline design and the portability payoff at scale.

Q7. How do you handle false positives systematically rather than case-by-case?
Build a closed feedback loop: every alert is triaged and tagged with an outcome (TP/FP/benign) in the SOAR, which feeds a per-rule precision metric. FPs above a threshold trigger a tuning task: add a filter/allowlist to the rule's condition, capture the benign event as a new true-negative fixture (so the fix is regression-tested), update the falsepositives field, and possibly lower level. Deploy the tuned rule through the same pipeline. Systemically, gate new rules on a staging false-positive budget before prod, and prefer correlation/aggregation over per-event rules for volumetric patterns. The discipline is that tuning changes are code changes, reviewed and tested, never silent console edits.
Why asked / signal: FP management is the daily reality of detection engineering; ad-hoc answers reveal inexperience.

Q8. When would you NOT use Sigma, and what are its limitations?
Sigma is a log-matching language; it is a poor fit when detection needs stateful stream processing, ML/statistical baselining, graph analytics, or rich joins that its condition grammar can't express — there you drop to native SPL/KQL/YARA-L or a streaming platform. Correlation support exists but is newer and less expressive than a full SIEM's. Some backends lack feature parity (aggregations, regex flavors), so a rule may convert lossily. Sigma also doesn't cover network-packet detection (that's Snort/Suricata) or file detection (YARA). The mature answer: use Sigma for the large portable middle of log-based detections and accept native content for the long tail that needs backend-specific power, keeping both in the same Git/DaC workflow.
Why asked / signal: Senior candidates know a tool's edges; juniors treat it as universal.

Q9. How do you measure whether your detection estate is any good?
Track multiple signals: coverage (ATT&CK technique heatmap derived from rule tags, weighted by data-source availability so you don't claim coverage for un-onboarded logs), precision (TP / (TP+FP) from alert-outcome tags per rule), volume/noise (alerts per rule per day), MTTD for validated incidents, and rule health (last-fired timestamp, conversion errors, failing tests). Combine coverage with threat intel to prioritize gaps against relevant adversaries. Beware vanity metrics: rule count is meaningless; a heatmap that ignores data availability lies. The north star is durable, high-precision coverage of techniques that matter to your threat model.
Why asked / signal: Tests metric literacy and skepticism toward vanity numbers.

Q10. Where do you draw the line between a signature-IOC rule and a behavioral rule in production?
Behavioral, ATT&CK-mapped rules form the durable backbone because they survive attacker mutation; IOC rules (specific hashes, IPs, domains) are legitimate but should be time-boxed IR artifacts with an expiry and a reference to the incident/intel that justified them, so the estate doesn't accumulate dead IOCs. Operationally I'd keep IOC content in a separate managed feed with TTLs and automated retirement, while behavioral rules live as reviewed Sigma with tests. The trade-off is freshness vs. durability: IOCs give precise, low-FP hits now but decay fast; behaviors are broader and need tuning but last. A good program runs both, clearly separated, not a pile of stale IP rules masquerading as coverage.
Why asked / signal: Nuance on the Pyramid of Pain in a production context.

Tier 3 — Expert / deep-dive

Q11. Walk through exactly what a pySigma processing pipeline does and why fail-loud matters.
A pipeline is an ordered list of transformations, each optionally gated by rule_conditions/field_name_conditions. Typical transforms: field_name_mapping (abstract→real field names), add_condition (inject index/sourcetype or a static filter for a logsource), value transforms (base64, lowercasing), placeholder resolution, and detection_item_failure/rule_failure to error on unsupported constructs. Order and priority matter because later transforms operate on the output of earlier ones. Fail-loud is critical: the single worst outcome is a rule that converts to an empty or wrong query and looks deployed while matching nothing — a silent blind spot. So an unmapped field must raise a build error, not pass through. Pipelines are code and get their own tests, because a shared-pipeline bug breaks every rule that depends on it.
Why asked / signal: Deep internals; only hands-on practitioners answer this well.

Q12. How do you test a detection without running attack tooling in production?
Separate the technique execution (done once, in an isolated lab) from detection validation (done continuously, in CI). In the lab you may run a controlled, atomic technique to generate authentic telemetry, then capture, sanitize, and commit the resulting log lines as fixtures. Detection validation replays those fixtures — pure recorded log data — through the compiled rule (ideally against an ephemeral SIEM container) and asserts exactly one match on the true-positive fixture and zero on the true-negative. No offensive tooling runs in CI or prod; you are exercising the query, not the attack. This gives regression safety: when a rule or pipeline changes, the fixtures re-prove behavior. Fixtures are seeded from sanitized prod logs to stay representative, and re-validated on a schedule to catch schema drift.
Why asked / signal: Directly probes the defensive test philosophy and the discipline boundary.

Q13. Explain Sigma correlation rules and when a single-event rule is insufficient.
Single-event rules fire on one log line; many attacks are only visible as patterns — password spray (many failures across accounts from one IP), impossible travel, or beaconing. Sigma's correlation spec defines meta-rules that reference base rules and apply count, value_count, temporal (several base rules within a window), or temporal_ordered (in sequence) with group-by and a timespan. Example: a base rule matches a failed console login, and a value_count correlation fires when ≥10 distinct usernames fail from one sourceIPAddress in 10 minutes. The trade-offs: correlation needs correct time normalization (UTC at ingest) and windows are stateful and heavier to evaluate; backend support varies, so a correlation rule may convert to native stateful search (e.g. Splunk stats ... by) or not at all. Use correlation for volumetric/sequential signal, single-event for discrete high-fidelity actions.
Why asked / signal: Tests the frontier of the spec and stateful detection reasoning.

Q14. An attacker with SIEM console access starts disabling detections. How does your architecture resist and detect this?
Defense in depth. Prevention: strict RBAC so almost no human can mutate deployed detections directly; deployment happens only via a service account through the pipeline, and Git is the source of truth. Detection: a meta-rule on the SIEM's own audit log fires when a saved search/correlation rule is edited, disabled, or deleted by anyone other than the deployment service account (see Detection A). Reconciliation: a scheduled job diffs live SIEM content against main and alerts on any drift, so even a silent tamper surfaces within the job interval. Recovery: because content is in Git, re-deploying the correct state is a pipeline run, not a rebuild from memory. This maps to ATT&CK T1562.001 (Impair Defenses) and the principle that your detection pipeline is itself a target that needs monitoring and integrity controls.
Why asked / signal: Tests thinking about the detection system as an attack surface.

Q15. You must roll out a Log4Shell-class emergency detection across three SIEMs in under an hour, safely. How?
Author one Sigma rule capturing the robust behavioral signal — not just contains jndi:ldap, which is trivially obfuscated, but a normalized regex plus decode modifiers and a companion process-creation rule for suspicious Java child processes. Fast-path it through an expedited pipeline: quick lint, convert to all three backends, and validate against a true-positive fixture built from a captured exploit log line and a true-negative from benign traffic — this proves the rule fires and won't storm the SOC. Deploy to production in shadow/alert-only mode across all three simultaneously via the pipeline, watching alert volume for a few minutes before enabling any automated response. The expedited path still runs the essential gates (convert + two-sided test), just with an accelerated single-reviewer approval and a tracked emergency ticket. The whole point of DaC + Sigma is that "one reviewed rule to three backends in an hour, tested" is a routine pipeline run, not a heroics exercise — and the shadow-mode step is what keeps "fast" from becoming "self-inflicted DoS on the SOC."
Why asked / signal: Synthesizes portability, testing, safe rollout, and incident tempo under pressure.

9. Key Takeaways

  • Sigma = one portable rule; the pipeline = the portability. The logsource+detection+condition structure is abstract; pySigma pipelines and backends do the vendor translation. Know the match semantics (map=AND, list=OR) cold.
  • Detection-as-Code is the operating model: Git, PR review, unit tests, lint, CI/CD, and drift reconciliation. Never click-ops production detections.
  • Anchor detections at the TTP level (Pyramid of Pain) and map every rule to ATT&CK via tags; reserve IOC rules as time-boxed IR artifacts.
  • Write robust matchers with modifiers (|contains|all, |base64offset, |re) so rules survive trivial attacker mutation; lint regexes against catastrophic backtracking.
  • Test the detection, not the attack: replay a sanitized true-positive and a true-negative fixture through the compiled query and assert exactly-one and zero matches. Execute techniques only in a lab to generate fixtures.
  • Fail loud in pipelines: an unmapped field must error the build; a silently empty query is the worst possible outcome.
  • Use correlation rules for volumetric/sequential signal (password spray, beaconing) that single-event rules miss; normalize time to UTC at ingest.
  • Measure with real metrics — coverage weighted by data availability, per-rule precision from alert-outcome tags, MTTD, rule health — not rule count.
  • The detection pipeline is itself an attack surface (T1562.001): RBAC it, monitor SIEM audit logs for rule tampering, and reconcile Git vs. prod.
  • Close the loop: every incident and every false positive should leave the detection content measurably better via a new fixture and a reviewed change.
