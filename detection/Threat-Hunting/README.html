Threat Hunting

Category: Detection Engineering — Proactive, Hypothesis-Driven Adversary Discovery
Frameworks/refs: MITRE ATT&CK, Sqrrl/TaHiTI Hunt Loop, Hunting Maturity Model (HMM), MITRE Cyber Analytics Repository (CAR), Pyramid of Pain, PEAK Threat Hunting Framework
Cross-references: detection/Detection-Engineering · detection/MITRE-ATTACK · detection/IDS-SIEM · detection/Anomaly-Detection

1. Overview

Threat hunting is the proactive, hypothesis-driven search through telemetry for adversary activity that has already evaded automated detection. It assumes breach: instead of waiting for an alert to fire, a human analyst forms a testable hypothesis about how an attacker might operate in this environment, queries raw data to prove or disprove it, and either finds evil or produces evidence of absence. It is fundamentally a detection-gap discovery discipline — every good hunt either surfaces a live intrusion or exposes a blind spot that should become a new automated detection. At scale, where you ingest petabytes of EDR, DNS, proxy, auth, and cloud audit logs daily, hunting is the mechanism that keeps mean-time-to-detect (MTTD) from silently drifting upward as attacker tradecraft evolves faster than your rule set.

Hunting is not alert triage (that's reactive SOC work), it's not incident response (that starts once you've confirmed a compromise), and it's not running a vendor's out-of-the-box detections. Those are downstream of, or orthogonal to, hunting. The defining property is that you start from a question the SIEM cannot already answer with an alert.

TL;DR
- Hunting is assume-breach: you look for what got past the tools, driven by a falsifiable hypothesis, not an alert.
- The output of a hunt is a decision: found evil → IR; found nothing → a new detection or an accepted-gap finding. A hunt that produces neither was a waste.
- Three modes: hypothesis-driven (ATT&CK/intel), IOC/intel-driven (retro-hunt for known indicators), baseline/anomaly (stack-count outliers). Mature teams do all three.
- Aim high on the Pyramid of Pain — hunt for TTPs and behaviors, not hashes/IPs, because behaviors are expensive for the adversary to change.
- Measure hunts by detections shipped, dwell-time reduction, and coverage of ATT&CK — not "number of hunts run."

2. How It Works (deep)

2.1 Core mechanics

Hunting operationalizes a loop borrowed from intelligence analysis: hypothesis → data → analysis → conclusion → codify. The engine that makes it repeatable is a small set of interacting concepts.

Concept What it is Why it matters
Hypothesis A falsifiable statement: "An adversary is using rundll32.exe to proxy-execute a malicious DLL from a user-writable path." Bounds the hunt; without it you are just staring at logs. Must be testable with data you have.
Hunting Maturity Model (HMM) HMM0 (rely on alerts only) → HMM4 (automated, continuous hunting). Ryan Stillions/Sqrrl model. Sets realistic expectations; you cannot hunt at HMM3 on HMM1 data.
The Hunt Loop Create hypothesis → investigate via tools/data → uncover patterns/TTPs → inform & enrich analytics. Sqrrl/TaHiTI. The repeatable process that turns a one-off insight into a durable detection.
Pyramid of Pain Hashes < IPs < domains < network/host artifacts < tools < TTPs — ascending order of adversary cost to change. Hunts targeting the top (TTPs) impose lasting cost; IOC hunts are cheap for the attacker to defeat.
ATT&CK as hypothesis generator A matrix of ~200 techniques mapped to tactics (TA00xx) with data-source requirements. Systematizes "what should I hunt next?" and gives you coverage bookkeeping.
Data source / analytic The raw telemetry (e.g., Sysmon EventID 1 process creation) and the query that interrogates it (MITRE CAR). The hypothesis is worthless if the required data source isn't collected and retained.
Stack counting (frequency analysis) Aggregate a field, sort by count; the long tail of rare values is where evil hides. The workhorse technique for "unknown-unknowns" — no signature required.
Enrichment & pivoting Add context (GeoIP, WHOIS, ASN, prevalence, reputation) and traverse relationships (this hash → these hosts → this parent). Turns a weak signal into an investigative lead; cuts false positives.

The maturity progression matters in practice because it forces you to be honest about prerequisites:

flowchart LR
  HMM0[HMM0 Initial rely on alerts] --> HMM1[HMM1 Minimal IOC search]
  HMM1 --> HMM2[HMM2 Procedural apply others hunts]
  HMM2 --> HMM3[HMM3 Innovative create new analytics]
  HMM3 --> HMM4[HMM4 Leading automate the hunt]
  HMM0 -.data quality.-> Data[Centralized retained telemetry]
  Data -.enables.-> HMM2

Three hunting modes — a senior engineer must distinguish them and know when each applies:

Mode Trigger / input Strengths Weaknesses Example
Hypothesis-driven (TTP) An ATT&CK technique or threat model Finds novel activity; high on Pyramid of Pain Requires skill and rich data "Hunt for LSASS credential access via handle to lsass.exe" (T1003.001)
IOC / intel-driven (retro-hunt) Fresh threat intel: hashes, domains, JA3, IPs Fast, cheap, high-confidence when it hits Low on pyramid; attacker rotates indicators trivially Sweep 90 days of DNS for a newly published C2 domain
Baseline / anomaly-driven Statistical model of "normal" Catches unknown-unknowns; no prior intel needed Noisy; needs good baselines and tuning Stack-count parent/child process pairs; investigate the rare tail

2.2 Concrete configuration & queries

A hunt program is only as good as the telemetry it can query. First, make sure the data source exists. On Windows endpoints, hunting depends on Sysmon with a curated config (the SwiftOnSecurity or Olaf Hartong sysmon-modular config is the de-facto standard) — you cannot hunt process lineage without EventID 1:

<!-- sysmon config excerpt: capture process creation + network + DNS -->
<Sysmon schemaversion="4.90">
  <EventFiltering>
    <RuleGroup name="proc-create" groupRelation="or">
      <ProcessCreate onmatch="exclude">
        <!-- exclude only high-volume known-good; include everything else -->
        <Image condition="is">C:\Windows\System32\SearchIndexer.exe</Image>
      </ProcessCreate>
    </RuleGroup>
    <RuleGroup name="dns" groupRelation="or">
      <DnsQuery onmatch="include"/>   <!-- EventID 22, critical for C2 hunts -->
    </RuleGroup>
  </EventFiltering>
</Sysmon>

Hunt example A — rare parent/child process relationships (stack counting). The premise: winword.exe, excel.exe, outlook.exe almost never legitimately spawn a shell or LOLBin. This is the classic macro/phishing-payload behavior (T1204.002 User Execution, T1059 Command and Scripting Interpreter). In Splunk over Sysmon EventID 1:

index=sysmon EventCode=1
| eval parent=lower(mvindex(split(ParentImage,"\\"),-1))
| eval child=lower(mvindex(split(Image,"\\"),-1))
| search parent IN ("winword.exe","excel.exe","powerpnt.exe","outlook.exe","mshta.exe")
| stats count dc(Computer) as hosts values(CommandLine) as cmds by parent child
| sort + count
| where hosts < 5              /* rare across the fleet = suspicious */

The sort + count puts the rarest pairs at the top — the long tail is your hunting ground. winword.exe → cmd.exe on three hosts out of 40,000 is a lead; winword.exe → splwow64.exe on 39,000 hosts is printing. Equivalent in Microsoft Sentinel / Defender KQL:

DeviceProcessEvents
| where InitiatingProcessFileName in~ ("winword.exe","excel.exe","powerpnt.exe","outlook.exe","mshta.exe")
| summarize hosts = dcount(DeviceId), cmds = make_set(ProcessCommandLine, 20), first=min(Timestamp)
    by InitiatingProcessFileName, FileName
| where hosts < 5
| order by hosts asc

Hunt example B — beaconing detection via connection-interval analysis. C2 implants (Cobalt Strike, Sliver, Mythic) "call home" on a periodic interval, often with jitter. The tell is a low variance in inter-arrival time for many connections to the same destination. Hunt in proxy/firewall/Zeek conn.log:

-- BigQuery / Presto over proxy or Zeek conn logs
WITH ordered AS (
  SELECT src_ip, dst_host, ts,
         TIMESTAMP_DIFF(ts,
           LAG(ts) OVER (PARTITION BY src_ip, dst_host ORDER BY ts),
           SECOND) AS delta
  FROM proxy_logs
  WHERE ts > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
)
SELECT src_ip, dst_host,
       COUNT(*)                    AS conns,
       AVG(delta)                  AS mean_interval,
       STDDEV(delta)               AS jitter,
       STDDEV(delta)/NULLIF(AVG(delta),0) AS coeff_var
FROM ordered
WHERE delta IS NOT NULL
GROUP BY src_ip, dst_host
HAVING conns > 30                  -- enough beacons to be periodic
   AND coeff_var < 0.15            -- very regular = machine, not human
   AND mean_interval BETWEEN 30 AND 3600
ORDER BY coeff_var ASC;

A coefficient of variation < 0.15 with dozens of connections to a single low-reputation host is a strong beaconing signal. Enrich dst_host with domain age (newly registered domains, NRDs), ASN reputation, and whether it's in your DNS baseline. This is the analytic behind RITA (Real Intelligence Threat Analytics) and Zeek-based beacon hunting.

Hunt example C — anomalous service-account token usage in cloud audit logs. Service accounts (SAs) are non-human identities with predictable behavior; deviation is high-signal. Threat: T1078.004 Valid Accounts: Cloud Accounts / stolen SA key exfiltrated and used from a novel location. In GCP, query Cloud Audit Logs (Data Access + Admin Activity):

-- GCP Log Analytics / BigQuery export of Cloud Audit Logs
SELECT
  protopayload_auditlog.authenticationInfo.principalEmail AS principal,
  protopayload_auditlog.requestMetadata.callerIp          AS caller_ip,
  protopayload_auditlog.methodName                        AS method,
  COUNT(*)                                                 AS calls
FROM `myorg.logs.cloudaudit_googleapis_com_data_access`
WHERE timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  AND principalEmail LIKE '%.gserviceaccount.com'          -- service accounts only
  AND caller_ip NOT IN (SELECT ip FROM `myorg.baseline.sa_known_ips`)
GROUP BY principal, caller_ip, method
ORDER BY calls DESC;

The hunt hypothesis: "A service account key has been stolen and is being used from an IP the SA has never called from." The baseline table sa_known_ips is itself a hunt artifact — you build it by profiling 30 days of normal SA behavior. Extend the analytic with impossible-travel logic and new-method detection (an SA that normally only reads GCS suddenly calling SetIamPolicy is privilege escalation, T1098). AWS equivalent hunts CloudTrail for AssumeRole / GetSessionToken from anomalous sourceIPAddress and userAgent; Azure hunts the sign-in logs and AuditLogs for managed-identity anomalies.

3. Real-World Incidents & CVEs

Threat hunting's value is clearest in breaches defined by long dwell time — cases where the intrusion sat undetected because it produced no alert, and only proactive search (or, tragically, its absence) determined the outcome.

Incident Year Root cause / tradecraft Dwell time Hunting lesson
SolarWinds / SUNBURST (UNC2452) 2020 Supply-chain trojan in Orion DLL; DGA C2 to avsvmcloud.com; blended with legit traffic ~9+ months Beaconing/DNS baseline hunts and "trusted software doing new things" hypotheses were the realistic detection path; signatures didn't exist.
Target POS breach 2013 HVAC vendor creds → network → BlackPOS RAM scraper; FireEye alerted but ignored ~3 weeks Alerts without a hunting/triage culture are worthless; hunting is a mindset, not just tooling.
APT29 / "Cozy Bear" (various) 2015–2021 LOLBins, WMI, low-and-slow, minimal malware Months High-Pyramid-of-Pain TTP hunts (WMI event subscription T1546.003, rundll32 proxy exec) are the only durable countermeasure.
NotPetya spread 2017 EternalBlue (CVE-2017-0144) + Mimikatz + PsExec lateral movement Fast, but pre-position undetected Lateral-movement hunts (anomalous SMB admin-share writes, psexesvc service creation) shrink pre-detonation dwell.
Log4Shell exploitation (CVE-2021-44228) 2021 JNDI lookup RCE; mass exploitation then quiet persistence Variable Retro-hunt (IOC-driven) of proxy/WAF logs for ${jndi:ldap:// and outbound LDAP from app servers found post-patch survivors.
Colonial Pipeline (DarkSide) 2021 Leaked VPN cred, no MFA ~1 week+ Auth-log hunts for VPN logins from new geographies / impossible travel on service and remote-access accounts.
Volt Typhoon (living-off-the-land) 2023–24 Chinese state actor, near-zero malware, LOTL on edge devices, valid accounts Years Only behavioral hunting (unusual netsh, wmic, ntdsutil, proxy chains) finds this class; IOC hunting is useless.

Case study — SolarWinds / SUNBURST. The implant was a signed, trusted Orion component that delayed ~12 days before beaconing to a DGA-generated subdomain of avsvmcloud.com, using DNS as its first C2 channel and mimicking Orion Improvement Program traffic. No AV or EDR signature existed. The realistic hunting hypotheses that would have surfaced it: (1) "Does any host show periodic DNS to a newly-registered/algorithmically-named domain?" — a DNS-baseline + NRD-enrichment hunt; (2) "Is trusted software (SolarWinds) initiating outbound connections to destinations it has never contacted?" — a behavioral-baseline hunt on parent process reputation. FireEye ultimately found it during an internal investigation triggered by a suspicious MFA device enrollment — itself a form of hunting. The control that shrinks this to weeks instead of months is continuous baseline-anomaly hunting on DNS and egress, not more signatures.

Case study — Volt Typhoon. CISA/Microsoft (2023) documented a PRC actor maintaining access to US critical-infrastructure networks for years using almost exclusively living-off-the-land binaries (LOLBins) and valid stolen credentials — wmic, netsh, PowerShell, ntdsutil for NTDS.dit extraction (T1003.003), and compromised SOHO routers as proxies. Because there was essentially no malware to signature, purely automated detection failed. The activity is only discoverable by TTP hunting: hypotheses like "an account is running ntdsutil or vssadmin to snapshot the domain controller," or "administrative commands are arriving via an unusual proxy chain." This incident is the canonical modern argument for hunting: as adversaries climb the Pyramid of Pain, human hypothesis-driven search becomes the primary, not supplementary, detection mechanism.

Case study — Log4Shell retro-hunt. After CVE-2021-44228 dropped (Dec 2021), even fully-patched orgs needed to answer "were we exploited before we patched?" This is a textbook IOC/intel-driven retro-hunt: sweep WAF/proxy logs for ${jndi:ldap://, ${jndi:rmi://, and obfuscated variants (${${lower:j}ndi:...}), then correlate any hit with outbound LDAP/RMI from the corresponding app server (the actual exploitation confirmation, since the string alone is just a scan). The behavioral half — an internet-facing Java app suddenly making outbound LDAP — is what separated real compromise from background scanning noise.

Top 5 controls in depth

1. Hunt→detection pipeline (the single most important control). A hunt that finds a gap but doesn't produce a durable artifact is a leak in the bucket — you'll rediscover the same gap next quarter. The control is an explicit, SLA-bound process: every hunt with a "detection gap found" outcome creates a ticket in the detection-engineering backlog with the query, the data source, the expected true-positive shape, and a tuning plan. Technically, this means the hunt query (Splunk SPL, KQL, SQL) is refactored into a scheduled analytic with alert thresholds, deduplication, and enrichment, then tracked against ATT&CK coverage. Cross-reference detection/Detection-Engineering — hunting is the R&D funnel that feeds detection engineering's production line. Trade-off: not every hunt should be automated — high-FP anomaly hunts may stay periodic-manual because a scheduled rule would drown the SOC. Judgment call: automate when the true-positive shape is stable and the FP rate is tunable below the SOC's tolerance.

2. Data-source coverage mapped to ATT&CK data components. You cannot hunt a technique whose required telemetry you don't collect — hunt #7 is the silent killer. The control is a coverage matrix: for each ATT&CK technique in your threat model, list its ATT&CK data components (e.g., T1003.001 needs "Process: OS API Execution" and "Process: Process Access"), then verify that source is ingested and retained. Tools like DeTT&CT and the ATT&CK Navigator visualize this. Implementation: run Sysmon with a modular config (EID 1, 3, 7, 8, 10, 11, 22), ship Windows Security 4688/4624/4625/4768/4769/4698/1102, enable GCP Data Access audit logs / AWS CloudTrail data events / Azure diagnostic settings, and centralize DNS + proxy. Trade-off: full data-access logging in cloud is expensive (CloudTrail data events and GCP Data Access logs can be your largest log source) — you sample or scope to sensitive resources, accepting a partial blind spot.

3. Baseline-anomaly analytics with enrichment (stack counting done right). The workhorse. Frequency analysis surfaces the rare tail, but raw rarity is noisy — the control is to pair it with enrichment and peer-group baselining so "rare" means "rare for this population." Technically: compute dcount(host) per artifact, join against prevalence tables, GeoIP/ASN, NRD lists, and known-good allowlists built from your own environment. For service accounts, baseline per-identity (each SA is its own peer group of one). Trade-off: baselines drift; a static allowlist becomes an attacker's hiding spot (they mimic baseline). Mitigate with rolling windows and periodic re-baselining, and never let the allowlist grow unbounded without review.

4. TTP hunting to climb the Pyramid of Pain. The strategic control against LOLBin/valid-account adversaries (Volt Typhoon class). Instead of hunting hashes and IPs (which the attacker rotates for free), hunt behaviors that are expensive to change: rundll32/regsvr32/mshta proxy execution, WMI persistence, LSASS access, abnormal Kerberos. Implementation is a prioritized hunt roadmap driven by ATT&CK — pick tactics/techniques relevant to your threat model (per the ATT&CK cross-reference), write a hunt per technique, and track coverage. Trade-off: TTP hunts demand richer data and more analyst skill than IOC sweeps, and produce more false positives, so they scale worse per-hunt — but they impose far higher adversary cost, so the ROI on durable security is much higher.

5. Tiered retention enabling retro-hunting. When fresh intel arrives (a new C2 domain, a new IOC set), you must be able to answer "were we ever hit?" — which requires that historical logs still exist. The control is a tiered retention strategy: hot/searchable for ~90 days, warm/cold (object storage, queryable via a data lake like BigQuery/Athena/Snowflake) for 1+ year. Retro-hunting Log4Shell required exactly this. Implementation: route SIEM data to cheap object storage with a query engine, and keep the high-value, low-volume sources (auth, cloud audit, DNS) longer than high-volume ones. Trade-off: cost vs. lookback — full-fidelity multi-year retention of EDR data is prohibitively expensive at scale, so you retain summaries/aggregates for the long tail and full fidelity only for the highest-signal sources.

KQL detection — anomalous service-account activity from a new IP (codifies Hunt C, Sentinel/GCP-export flavor):

let lookback = 30d;
let known =
    CloudAuditLogs
    | where TimeGenerated > ago(lookback) and TimeGenerated < ago(1d)
    | where PrincipalEmail endswith ".gserviceaccount.com"
    | summarize by PrincipalEmail, CallerIp;
CloudAuditLogs
| where TimeGenerated > ago(1d)
| where PrincipalEmail endswith ".gserviceaccount.com"
| join kind=leftanti known on PrincipalEmail, CallerIp   // IP never seen for this SA
| summarize calls=count(), methods=make_set(MethodName, 20), first=min(TimeGenerated)
    by PrincipalEmail, CallerIp
| extend SensitiveMethod = methods has_any ("SetIamPolicy","CreateServiceAccountKey","storage.objects.get")
| order by calls desc

Response playbook (hunt confirms a compromise)

Once a hunt crosses from "lead" to "confirmed adversary," you pivot into IR. The sequence, specific to hunt-originated findings:

  1. Preserve, don't tip off. The adversary doesn't know you found them via hunting — a huge advantage. Before acting, snapshot volatile evidence (memory, current process list, network connections) and expand the hunt to map full scope. Premature blocking burns your visibility.
  2. Scope via pivoting. From the initial indicator, pivot across all data sources: same hash → other hosts; same C2 → other beaconing sources; same SA key → all resources it touched. Build the complete blast radius before containing.
  3. Contain. Isolate affected endpoints (EDR network-isolate), revoke the compromised credential/SA key, block C2 at proxy/DNS, disable OAuth grants/tokens. Contain the whole scope simultaneously to prevent the adversary re-establishing.
  4. Eradicate. Remove persistence (scheduled tasks, WMI subscriptions, run keys, malicious SAs/keys), reimage where warranted, rotate all credentials in blast radius (assume Kerberos/token compromise → consider krbtgt double-reset for AD).
  5. Recover. Restore from known-good, re-enable services, monitor with hunt-derived detections specifically watching for re-entry using the same TTPs.
  6. Codify & retro. Turn the successful hunt into a permanent detection, retro-hunt the full retention window for earlier instances, and update the hunt library and ATT&CK coverage map. This closes the loop back to §2.3.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is threat hunting and how is it different from alerting/SOC triage?
Threat hunting is the proactive, hypothesis-driven search through telemetry for adversary activity that has already evaded automated detection — it operates on an assume-breach premise. SOC triage is reactive: it processes alerts that automated rules generated. The key distinction is the starting point: a hunt begins with a falsifiable hypothesis the SIEM cannot already answer with an alert, whereas triage begins with an alert. Hunting also has a distinct output contract — it must produce either an incident escalation or a new detection/documented gap, whereas triage produces a disposition (true/false positive).
Signal: does the candidate understand hunting is proactive and gap-finding, not just "advanced alerting."

Q2. Explain the three hunting modes.
Hypothesis/TTP-driven hunting starts from an ATT&CK technique or threat model and searches for the behavior (e.g., LSASS handle access). IOC/intel-driven hunting (retro-hunting) takes fresh indicators — hashes, domains, JA3 — and sweeps historical logs for matches; it's fast and high-confidence but low on the Pyramid of Pain. Baseline/anomaly-driven hunting builds a statistical model of "normal" and investigates outliers via stack counting, catching unknown-unknowns with no prior intel. Mature programs run all three; over-reliance on IOC hunting alone is a maturity anti-pattern because attackers rotate indicators trivially.
Signal: can they distinguish the modes and know each one's trade-offs.

Q3. What is the Pyramid of Pain and why does it matter for hunting?
The Pyramid of Pain (David Bianco) ranks indicator types by how costly they are for an adversary to change: hashes (trivial) < IP addresses < domain names < network/host artifacts < tools < TTPs (very costly). It matters because hunts targeting the top of the pyramid — behaviors and techniques — impose lasting cost on the attacker, since changing how they operate is expensive, while hunts for hashes/IPs are defeated the moment the attacker recompiles or rotates infrastructure. A senior hunter deliberately biases toward TTP hunts for durable defense.
Signal: strategic thinking about durable vs. ephemeral detection.

Q4. What data sources are essential for an effective hunting program?
At minimum: EDR/Sysmon (process creation with parent lineage and command line, image loads, LSASS access, DNS), DNS logs, egress proxy/firewall logs, authentication/IdP logs, and cloud audit logs (CloudTrail, GCP Cloud Audit, Azure). The critical insight is that hunting is data-bound — you cannot hunt a technique whose required ATT&CK data component you don't collect and retain. So a coverage audit mapping techniques to data sources is a prerequisite, and retention (hot ~90 days, cold 1+ year) governs your retro-hunt reach.
Signal: understands the data-first dependency, not just tool names.

Q5. What is stack counting and why is it useful?
Stack counting (frequency analysis) aggregates a telemetry field, counts occurrences, and sorts to expose the distribution. The value is in the long tail: rare values — a parent/child process pair seen on 3 of 40,000 hosts, an unusual user-agent, a rare DLL load path — are statistically anomalous and often malicious, and it requires no signature or prior intel. It's the workhorse for unknown-unknowns. The refinement is to pair rarity with enrichment (prevalence, GeoIP, reputation) so "rare" means "rare for this peer group," reducing false positives.
Signal: knows the core analytic technique and its noise pitfall.

Tier 2 — System design / trade-offs

Q6. Design a hunting program for a company with 50,000 endpoints and multi-cloud. Where do you start?
Start with data-source coverage, not hunts: audit which ATT&CK data components you collect and retain, because un-collected telemetry makes techniques un-huntable. Deploy Sysmon with a modular config, centralize DNS/proxy/auth, and enable cloud audit logging (accepting that full data-access logging is your biggest cost, so scope it to sensitive resources). Next, build a prioritized hunt roadmap from ATT&CK filtered by your threat model — bias toward TTP hunts high on the Pyramid of Pain. Establish the hunt→detection pipeline so every gap becomes a tracked detection ticket with an SLA. Instrument metrics that measure outcomes (detections shipped, dwell-time reduction, ATT&CK coverage) not activity (hunts run). Staff with people who can both write queries and reason about adversary behavior, and invest in baselines/enrichment infrastructure so anomaly hunts aren't drowned in false positives. Iterate: hunts feed detections, detections free hunters to climb the pyramid.
Signal: prioritizes data and process over flashy analytics; thinks about cost and outcomes.

Q7. How do you decide whether a successful hunt should become an automated detection?
Automate when the true-positive shape is stable and the false-positive rate can be tuned below the SOC's tolerance; keep it as a periodic manual hunt otherwise. Concretely, a rare-parent/child LOLBin hunt with a crisp, low-FP signature (Office spawning cmd.exe) is an easy automation candidate. A broad anomaly hunt — say, "any statistically rare outbound destination" — might generate hundreds of daily leads that require human judgment and enrichment; scheduling it as an alert would cause alert fatigue and get muted, which is worse than not automating. The decision hinges on FP tunability, the availability of enrichment to raise precision, the severity of the true positive (high-severity, low-frequency events justify more FP tolerance), and maintenance cost (baselines drift). The governing principle: automation should free hunters to climb the pyramid, not flood the SOC.
Signal: understands the automation/FP trade-off and alert fatigue.

Q8. How would you hunt for C2 beaconing in an all-HTTPS environment where you can't inspect payloads?
Pivot from content to metadata and timing. The strongest signal is temporal: aggregate connections per source/destination pair and compute inter-arrival intervals; low variance (coefficient of variation < ~0.15) across many connections indicates machine-driven periodicity — a beacon. Combine with metadata: JA3/JA3S TLS fingerprints to identify known implant stacks, SNI and certificate anomalies (self-signed, newly issued, mismatched), destination reputation, domain age (NRDs), and ASN. Enrich the destination — a regular beacon to a 3-day-old domain on a bulletproof-hosting ASN is high-confidence. Account for jitter (attackers randomize intervals) by using distributional statistics rather than exact-period matching, and filter legitimate periodic traffic (software update checks, telemetry) via baselining. This is precisely how RITA/Zeek beacon analysis works.
Signal: knows encrypted-traffic hunting relies on timing/metadata, not payload.

Q9. Walk through hunting for a compromised cloud service-account key.
Hypothesis: a service-account key has been exfiltrated and is being used from infrastructure the SA has never legitimately called from (T1078.004). Service accounts are ideal hunt targets because their behavior is narrow and predictable. Build a 30-day baseline per SA of caller IPs, user agents, geolocations, and the specific API methods it invokes. Then hunt for deviations: a leftanti join against the known-IP baseline surfaces calls from novel IPs; layer impossible-travel logic and a "new sensitive method" check (an SA that only reads GCS suddenly calling SetIamPolicy or CreateServiceAccountKey is privilege escalation/persistence, T1098). Enrich the novel IP with ASN/geo/reputation and correlate with whether a key was recently created or downloaded. In AWS the analog hunts CloudTrail AssumeRole/GetSessionToken by anomalous sourceIPAddress/userAgent; in Azure, managed-identity sign-in anomalies. The baseline table itself is a durable hunt artifact you maintain.
Signal: cloud-native hunting, per-identity baselining, understands SA threat model.

Q10. How do you measure the success of a hunting program? What are good and bad metrics?
Good metrics measure outcomes and coverage: number of confirmed incidents found by hunting, new detections shipped per hunt cycle (the hunt→detection conversion rate), reduction in mean-time-to-detect / dwell time, ATT&CK technique coverage gained, and data-source coverage improvements. Bad metrics measure raw activity: "number of hunts run" or "hours spent hunting" incentivize busywork over impact, and "number of alerts generated" rewards noise. A subtle but important metric is the negative-result value — a hunt that confidently proves absence of a technique is a legitimate success (it converts an unknown into a known-good and documents coverage), so a program that only counts "evil found" will undervalue rigorous hunts. Track findings-to-detection SLA to ensure the pipeline doesn't leak.
Signal: outcome-oriented; understands negative results have value; avoids vanity metrics.

Tier 3 — Expert / deep-dive

Q11. How does ATT&CK generate hunt hypotheses, and what's the limitation of ATT&CK-driven hunting?
ATT&CK provides a structured library of ~200 techniques mapped to tactics, each annotated with data components required to observe it and known procedures. This systematizes hypothesis generation: filter the matrix to techniques relevant to your threat model, and each becomes a hunt ("hunt for T1546.003 WMI event-subscription persistence"), with the data-component annotation telling you exactly what telemetry you need — enabling coverage bookkeeping via Navigator/DeTT&CT. The limitation is that ATT&CK is a catalog of known techniques — it's inherently backward-looking and doesn't describe novel tradecraft, environment-specific attack paths, or the chaining of techniques into a campaign. Over-indexing on ATT&CK can produce a checklist mentality that misses genuinely new behavior; it should anchor but not bound your hypotheses. Baseline-anomaly hunting complements it by catching what ATT&CK doesn't yet name (e.g., pre-Volt-Typhoon LOTL patterns).
Signal: uses ATT&CK properly but sees its backward-looking blind spot.

Q12. Design a beaconing detection that's robust to jitter and long sleep intervals. What defeats naive interval analysis?
Naive interval analysis (exact-period matching) is defeated by jitter (attacker randomizes ±30–50% around the base interval) and by long sleeps (a 24-hour beacon looks like normal daily traffic). A robust design uses distributional statistics over a long window: compute the set of inter-arrival deltas per src/dst pair, then measure regularity via coefficient of variation, or better, use a Fourier/autocorrelation approach to detect a dominant frequency even under jitter, and Median Absolute Deviation (MAD) for outlier-resistant spread. To handle long sleeps, extend the window (multi-day) and lower the connection-count threshold accordingly. Complement timing with size regularity (beacons often have consistent request/response byte sizes) and connection-count-to-time-span ratios. Defeats/evasions to acknowledge: high-jitter + low-and-slow (Volt Typhoon style) can fall below any statistical threshold, and domain-fronting/CDN-shared destinations blur per-destination aggregation — so pair timing analysis with destination reputation and metadata, and accept that a determined low-and-slow C2 may only be findable via TTP hunts on the host side (the implant's persistence/execution), not its network beacon.
Signal: deep understanding of beaconing math and its evasion limits.

Q13. An APT is using only living-off-the-land binaries and valid accounts (Volt Typhoon profile). How do you hunt when there's no malware to signature?
This is the hardest and most important hunting problem, because IOC and signature approaches are structurally useless — there is no malware and the credentials are legitimate. You hunt behavior and context. First, command-line intent: LOLBins like wmic, netsh, ntdsutil, vssadmin, certutil, rundll32 are legitimate but rare on most hosts and rarely used in the specific patterns attackers need — e.g., ntdsutil ... create full or vssadmin create shadow (NTDS.dit theft, T1003.003), netsh portproxy (tunneling), certutil -urlcache -f (download). Stack-count these command lines and investigate the rare tail. Second, context anomaly: a valid admin account is suspicious when it authenticates from a new source, at an unusual time, via an unusual path (e.g., admin commands arriving through a proxy chain from an edge device). Third, sequence/graph: individually-benign actions form a malicious chain — recon → credential access → lateral movement — so hunt for the sequence across hosts, not single events. Fourth, infrastructure: Volt Typhoon proxied through compromised SOHO routers, so hunt for internal admin traffic sourced from unexpected network segments. The meta-point: against LOTL, human hypothesis-driven TTP hunting becomes the primary detection mechanism, and your data richness (full command-line capture, auth context, network metadata) is the binding constraint.
Signal: elite — understands why LOTL breaks automation and how behavioral/contextual/graph hunting responds.

Q14. How do you keep baselines from becoming a place for attackers to hide, and handle baseline drift at scale?
Baselines are double-edged: an attacker who studies or slowly shifts "normal" can operate inside the allowlist (baseline poisoning), and static allowlists ossify into blind spots. Mitigations: (1) use rolling windows and periodic re-baselining rather than frozen allowlists, so the baseline reflects recent reality but you also diff changes to the baseline itself — a sudden shift in "normal" is its own signal. (2) Cap allowlist growth and require review for additions, since an ever-growing allowlist is an ever-growing hiding spot. (3) Baseline at the right granularity — per-identity/per-peer-group, not global — so an attacker must match a specific narrow profile, not just blend into aggregate noise. (4) Hunt the baseline-building process for poisoning: was a "new normal" established during a plausible compromise window? (5) Keep some hunts baseline-independent (pure TTP signatures) so an adversary can't defeat your entire program by manipulating statistics. At scale, drift is managed with automated re-baselining pipelines, versioned baselines (so you can diff and roll back), and alerting on baseline volatility. The philosophical point: never let any single detection philosophy — signature, anomaly, or baseline — be your only layer.
Signal: recognizes baseline poisoning and drift as adversarial problems, defends in depth.

Q15. Describe the end-to-end lifecycle of turning a one-off successful hunt into durable, automated, continuously-improving detection — including the failure modes at each stage.
The lifecycle: (1) Hunt produces a confirmed finding + a query. Failure mode: the query is a one-off notebook cell that's never captured — knowledge evaporates. Fix: a versioned hunt library (ABLE/notebook format) recording hypothesis, data source, query, and result. (2) Codify: refactor the ad-hoc query into a scheduled analytic with thresholds, deduplication, and enrichment, mapped to an ATT&CK technique. Failure mode: the raw hunt query is too FP-heavy to schedule as-is and gets shipped anyway, causing alert fatigue; or it's never prioritized in the backlog and rots. Fix: an SLA-bound hunt→detection pipeline and an explicit "automate vs. keep-manual" decision gate based on FP tunability. (3) Deploy & tune: run in monitor mode, measure precision/recall, tune. Failure mode: no feedback loop, so the rule silently degrades as the environment changes (a new legitimate tool starts triggering it). Fix: periodic detection health review and drift monitoring. (4) Retro-hunt: sweep the full retention window for earlier instances of the newly-understood technique. Failure mode: insufficient retention means you can't look back far enough — the intrusion predates your logs. Fix: tiered retention. (5) Coverage & iteration: update the ATT&CK coverage map, which reveals the next gap to hunt, closing the loop. Failure mode: measuring activity (hunts run) instead of outcomes (coverage gained, dwell-time reduced) so the program optimizes the wrong thing. The through-line: hunting is the R&D funnel and detection engineering is the production line (cross-ref); the value is realized only when the pipeline between them has no leaks.
Signal: staff-level systems thinking — sees the whole pipeline, its failure modes, and the metrics that keep it honest.

9. Key Takeaways

  • Hunting is assume-breach and gap-finding. It starts from a falsifiable hypothesis the SIEM can't already answer, not from an alert. If you're processing alerts, that's triage, not hunting.
  • Every hunt must produce a decision: escalate to IR, ship a detection, or document an accepted gap. A hunt that produces none was wasted — the hunt→detection pipeline is the highest-value control.
  • Climb the Pyramid of Pain. Hunt TTPs and behaviors, not hashes and IPs; behaviors are expensive for adversaries to change and give durable defense. IOC-only hunting is a maturity anti-pattern.
  • Hunting is data-bound. You cannot hunt a technique whose ATT&CK data component you don't collect and retain. Coverage audits and tiered retention are prerequisites, not afterthoughts.
  • Three modes, use all three: hypothesis/TTP-driven, IOC/intel-driven (retro-hunt), and baseline/anomaly (stack counting). Each has distinct strengths; mature programs blend them.
  • Stack counting finds unknown-unknowns — the rare tail is where evil hides — but raw rarity is noisy; pair it with enrichment and peer-group baselining so "rare" means "rare for this population."
  • Against LOTL/valid-account adversaries (Volt Typhoon), human TTP hunting is the primary detection, because there's no malware to signature and credentials are legitimate. Behavioral, contextual, and graph-based hunting is the only answer.
  • Measure outcomes, not activity: detections shipped, dwell-time reduction, ATT&CK coverage — never "number of hunts run." Negative results (proving absence) are legitimate wins.
  • Beaconing hunts rely on timing and metadata, not payload — coefficient of variation on inter-arrival intervals plus JA3/reputation/domain-age enrichment defeats even encrypted C2, within limits.
  • Baselines are adversarial artifacts — guard against poisoning and drift with rolling windows, capped allowlists, per-identity granularity, and baseline-independent TTP hunts as a backstop.
