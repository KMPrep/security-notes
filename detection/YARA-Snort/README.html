YARA and Snort Rules

Category: Detection Engineering — Signature Writing

Overview

YARA and Snort are the two most widely used signature languages in detection engineering. YARA detects malware in files and memory. Snort detects threats in network traffic. Both require analysts to express observable patterns of malicious behavior in a structured syntax, and both require ongoing maintenance as threats evolve.

YARA

YARA (Yet Another Recursive Acronym) is a tool for identifying and classifying malware samples based on text or binary patterns. A YARA rule describes what to look for in a file or memory region. It is widely used in malware triage pipelines, antivirus engines, sandbox analysis platforms (VirusTotal, Hybrid Analysis), and EDR products.

YARA Rule Structure

rule RuleName
{
    meta:
        author = "analyst"
        description = "Detects X malware family"
        date = "2026-01-01"
        hash = "abc123..."

    strings:
        $str1 = "malicious_string" ascii
        $str2 = "another_string" wide
        $hex1 = { 4D 5A 90 00 03 00 }
        $re1  = /C2_domain_[0-9]{4}\.com/ nocase

    condition:
        uint16(0) == 0x5A4D and           // MZ header check
        filesize < 500KB and
        any of ($str*) or $hex1
}

YARA String Types

Type Syntax Use Case
Plain string "value" Exact ASCII match
Wide string "value" wide UTF-16LE (common in Windows binaries)
Case-insensitive "value" nocase Domain names, function names
Hex bytes { 4D 5A ?? 00 } Byte sequences; ?? is wildcard
Regex /pattern/ Variable-length patterns like DGA domains
XOR "value" xor Match string XOR'd with any single byte

YARA Condition Expressions

Expression Meaning
all of ($str*) All strings matching $str* must be found
any of ($str*) At least one string matching $str* must be found
2 of ($str*) Exactly 2 of the matching strings must be found
uint16(0) == 0x5A4D First 2 bytes are MZ (PE header check)
filesize < 1MB Limit rule to small files
#str1 > 5 String $str1 appears more than 5 times
@str1 < @str2 String $str1 appears before $str2 in the file

Writing Durable YARA Rules

The goal is to write rules that detect a malware family across samples, not just one specific binary.

Good signal sources:

  • Unique function names or strings embedded in malware
  • Hardcoded C2 URLs or beacon intervals
  • Unusual byte sequences in the PE header or packer stubs
  • Configuration block format (e.g., encrypted config with known magic bytes)
  • Specific import combinations (e.g., VirtualAlloc + WriteProcessMemory + CreateRemoteThread)

Avoid:

  • Common strings present in benign software
  • Very short patterns (high false positive rate)
  • Rules that match entire file hashes (use hash databases for that; YARA is for families)

Snort / Suricata Rules

Snort and Suricata share a common rule language for detecting malicious network traffic patterns. Snort is the original; Suricata is the modern multi-threaded replacement that supports the same rule syntax.

Snort Rule Structure

action  protocol  src_ip  src_port  direction  dst_ip  dst_port  (options)

Example:

alert tcp any any -> $HOME_NET 445 (
    msg:"SMB EternalBlue Exploit Attempt";
    flow:to_server,established;
    content:"|00 00 00 85 ff|";
    content:"SMBr";
    depth:4;
    offset:4;
    content:"|73 00 65 00|";
    sid:1000001;
    rev:1;
)

Rule Actions

Action Effect
alert Generate alert and log the packet
log Log the packet without alerting
pass Ignore the packet
drop Drop the packet and generate alert (IPS mode)
reject Drop and send TCP RST / ICMP Unreachable

Rule Options — Most Important

Option Purpose Example
msg Human-readable alert message msg:"Possible port scan";
content Match bytes or string in payload content:"cmd.exe";
nocase Case-insensitive content match content:"GET"; nocase;
pcre Perl-compatible regex match pcre:"/[Pp]assword/";
flow Match on connection state flow:established,to_server;
flags Match TCP flags flags:S; (SYN only)
threshold Rate limiting for alert frequency threshold:type both, track by_src, count 10, seconds 60;
sid Unique rule identifier sid:1000042;
rev Rule revision number rev:3;
classtype Attack classification classtype:trojan-activity;
reference Link to CVE or external info reference:cve,2017-0144;
metadata Key-value metadata metadata:affected_product Windows;

Example Rules

Detect DNS tunnelling (long subdomain):

alert dns any any -> any 53 (
    msg:"Possible DNS Tunnelling - Long Subdomain";
    dns.query;
    pcre:"/^[a-z0-9]{30,}/i";
    threshold:type both, track by_src, count 5, seconds 60;
    sid:1000100;
    rev:1;
)

Detect port scan (many SYN to different ports):

alert tcp any any -> $HOME_NET any (
    msg:"Possible Port Scan";
    flags:S,12;
    threshold:type both, track by_src, count 20, seconds 10;
    sid:1000200;
    rev:1;
)

Detect reverse shell over HTTP (base64 encoded command):

alert http any any -> any any (
    msg:"Possible Reverse Shell via HTTP - base64 payload";
    http.uri;
    pcre:"/[A-Za-z0-9+\/]{30,}={0,2}/";
    flow:established,to_server;
    sid:1000300;
    rev:1;
)

Suricata Enhancements Over Snort

Feature Suricata Advantage
Performance Multi-threaded; handles 10Gbps+ links
TLS/JA3 inspection Built-in JA3/JA3S fingerprinting for TLS detection
File extraction Extract and hash transferred files from HTTP/FTP/SMB
EVE JSON logging Structured JSON output ideal for SIEM ingestion
Lua scripting Custom detection logic beyond rule syntax
Protocol detection Auto-detects protocol regardless of port number

JA3 Fingerprinting (Suricata)

JA3 is a method of fingerprinting TLS client hellos based on the combination of TLS version, cipher suites, extensions, elliptic curves, and elliptic curve formats. Different malware families use characteristic TLS configurations that produce consistent JA3 hashes, enabling detection even over encrypted traffic.

alert tls any any -> any any (
    msg:"Cobalt Strike Beacon TLS Fingerprint";
    ja3.hash; content:"72a589da586844d7f0818ce684948eea";
    sid:1000400;
    rev:1;
)

YARA vs Snort Comparison

Property YARA Snort/Suricata
Inspection target Files, memory Network packets
Use in IR Scanning endpoints and memory dumps Live traffic monitoring
Update frequency Updated per malware campaign Updated per network threat intel
Evasion Packing, encoding, polymorphism Encryption, protocol misuse, fragmentation
False positive risk Medium — tune string specificity Medium — tune thresholds and payloads
