CI/CD Pipeline Hardening

Category: Software Supply Chain Security
Key standards / frameworks: OWASP Top 10 CI/CD Security Risks, SLSA v1.0, NIST SP 800-204D, CIS Software Supply Chain Benchmarks, GitHub and GitLab hardening guides

Overview

The CI/CD pipeline is privileged infrastructure: it holds cloud credentials and signing keys and it executes arbitrary code on every commit. That makes it a prime target, because compromising the build lets an attacker inject malicious code into artifacts that are then signed as trusted and distributed to every downstream consumer. The SolarWinds attack implanted the SUNBURST backdoor by tampering with the Orion build process, and the Codecov breach altered the bash uploader script to exfiltrate environment secrets from thousands of downstream pipelines.

Hardening the pipeline means applying least privilege, isolation, and integrity verification to the machinery itself. The dominant risk class is Poisoned Pipeline Execution (PPE), where an attacker gets the pipeline to run their code with the pipeline's privileges. The controls below shrink both the likelihood of that execution and the blast radius if it happens.

How It Works

Poisoned Pipeline Execution and untrusted PR builds

PPE comes in two flavours. Direct PPE modifies the pipeline definition itself (a workflow file in a branch) to run malicious steps. Indirect PPE injects commands through files the pipeline consumes, such as a Makefile, test script, or build config, without touching the workflow. The classic trigger is an untrusted pull request from a fork: if the workflow runs on pull_request_target with repository secrets and checks out the attacker's code, the fork's code executes with write privileges and secret access. Untrusted PR builds must run in a low-privilege, secret-free context with manual approval before any privileged step.

Least-privilege tokens and OIDC federation

Pipelines historically stored long-lived cloud keys as CI variables, giving a broad standing blast radius. OIDC federation replaces them: the CI provider issues a short-lived, signed identity token describing the workflow, repository, and branch, and the cloud provider exchanges it for temporary credentials scoped by a trust policy. There is no static secret to steal, and access is bounded to specific repos and refs. Workflow tokens themselves should default to read-only and be granted specific scopes only where needed.

Pinning, isolation, and protected branches

Third-party GitHub Actions run with your pipeline's privileges, so a compromised or retagged action is a direct supply-chain risk. Pin every action to a full commit SHA rather than a mutable tag like @v4, because a tag can be force-moved to malicious code. Run builds on ephemeral, isolated runners that are destroyed after each job so nothing persists between builds. Protect the default branch with required reviews, status checks, signed commits, and restrictions on who can push, so the pipeline definition cannot be silently altered.

Artifact signing and provenance

Integrity is proven at the end with signing. Sigstore and its cosign tool produce keyless signatures tied to an OIDC identity, and SLSA provenance records how and from what an artifact was built. Consumers verify the signature and provenance before deploying, so a tampered artifact fails verification. This is the enforcement layer that would have flagged a SolarWinds-style implant, and it ties into supply_chain.

Key Takeaways

  • The pipeline runs untrusted code with production privileges; treat it as tier-zero infrastructure.
  • Poisoned Pipeline Execution is the core risk; the trigger is usually an untrusted pull request that gains secrets and write access.
  • Replace long-lived cloud keys with OIDC federation so there is no static credential to steal and access is scoped to repo and branch.
  • Pin third-party actions to a full commit SHA; a mutable tag can be force-moved to malicious code.
  • Use ephemeral, isolated runners and protected branches so nothing persists and no unreviewed step lands.
  • Sign artifacts with cosign and emit SLSA provenance, then verify before deploy; this is what catches SolarWinds-style tampering.
  • SolarWinds and Codecov prove attackers target the build system itself, so integrity verification is not optional.
