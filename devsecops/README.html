DevSecOps & CI/CD Pipeline Security

Category: Application & Software Supply Chain Security
Key standards / frameworks: OWASP DevSecOps Guideline, OWASP Top 10 CI/CD Security Risks, NIST SP 800-218 (SSDF), NIST SP 800-204D, SLSA, CIS Software Supply Chain Benchmarks

Overview

DevSecOps is the practice of embedding security controls directly into the software delivery lifecycle rather than bolting them on before release. The organising idea is shift-left: move detection of vulnerabilities, secrets, and misconfigurations as early and as cheaply as possible, ideally to the developer's IDE and the pull request, where a fix costs minutes instead of an incident bridge. Security becomes a set of automated gates in the pipeline instead of a manual review at the end.

Modern delivery pipelines are themselves a high-value attack surface. A CI/CD system holds cloud credentials, signing keys, and write access to production artifacts, and it executes untrusted code on every commit. The Codecov bash-uploader compromise, the SolarWinds Orion build-server implant, and a steady stream of poisoned npm and PyPI packages all show that attackers increasingly target the build system rather than the running application, because a single poisoned build fans out to every downstream consumer.

This section treats the pipeline as a system to be threat-modelled. The four child notes cover the detective controls that run inside it and the hardening controls that protect it: static, dynamic, and interactive analysis; software composition analysis; secret scanning; and pipeline hardening itself.

How It Works

Security gates across the SDLC

Each stage of the pipeline hosts a different class of control. In the IDE and pre-commit hooks, secret scanners and linters catch problems before code leaves the laptop. At pull-request time, SAST and SCA run against the diff and block merge on new high-severity findings. During the build, dependencies are resolved from lockfiles and artifacts are signed. In pre-production, DAST and IAST exercise the running application. In production, runtime protection and continuous SCA against fresh CVE feeds close the loop. The goal is defence in depth so that no single missed finding reaches customers.

The pipeline as an attack surface

A commit triggers a runner that clones the repository, executes build scripts, pulls dependencies, and pushes signed artifacts to a registry using privileged tokens. Every one of those steps is a trust boundary. Untrusted pull-request code can run on a runner that holds secrets. A typosquatted dependency can execute an install hook. An over-privileged token can be exfiltrated and used against production. The diagram traces the flow and the highest-value targets.

How the controls fit together

SAST reads source code for insecure patterns without running it. SCA inventories third-party and transitive dependencies and matches them against vulnerability databases. Secret scanning looks for committed credentials across code and git history. DAST and IAST test the deployed application from the outside and the inside. Pipeline hardening protects the machinery itself with least-privilege tokens, OIDC federation, pinned actions, and artifact signing. Together they map onto the broader software supply chain program covered under supply_chain, which adds SLSA provenance, SBOM generation, and Sigstore-based verification.

Key Takeaways

  • Shift-left works because the cost of remediating a vulnerability rises by orders of magnitude as it moves from IDE to production; automated gates make early detection the path of least resistance.
  • Treat the CI/CD system as production infrastructure: it holds cloud credentials and signing keys and runs untrusted code on every commit.
  • No single scanner is sufficient. SAST, DAST, IAST, and SCA have complementary blind spots and should be layered across pipeline stages.
  • Detective controls find problems; hardening controls stop the pipeline itself from becoming the exploit. You need both.
  • Real incidents (SolarWinds, Codecov, dependency-confusion campaigns) targeted the build system, not the app, validating supply-chain-first threat models.
  • DevSecOps is a program, not a tool purchase: policy-as-code, break-the-build gates, and rotation runbooks matter as much as the scanners.
