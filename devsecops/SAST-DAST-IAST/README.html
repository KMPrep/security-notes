SAST, DAST & IAST

Category: Application Security Testing
Key standards / frameworks: OWASP ASVS, OWASP WSTG, OWASP Top 10, NIST SP 800-218 (SSDF PW.7 / PW.8), CWE, MITRE SARIF

Overview

Application security testing splits into three complementary techniques defined by how much they know about the target. SAST (Static Application Security Testing) reads source code or bytecode without executing it. DAST (Dynamic Application Security Testing) attacks a running instance from the outside with no knowledge of internals. IAST (Interactive Application Security Testing) instruments the running application from the inside, correlating real requests with the code paths they touch.

Each technique sees a different slice of reality, which is why mature programs run more than one. SAST finds injection sinks and hardcoded weaknesses early but drowns teams in false positives. DAST finds real, exploitable issues but only on code paths it manages to reach. IAST bridges the two by watching from inside during functional tests. RASP (Runtime Application Self-Protection) extends the IAST instrumentation idea into production to block attacks live rather than merely report them.

How It Works

Static analysis (SAST)

SAST parses code into an abstract syntax tree and control/data-flow graphs, then runs taint analysis to trace untrusted input from a source (an HTTP parameter) to a dangerous sink (a SQL query or shell exec). Because it never runs the code it can scan a partial diff at pull-request time and cover paths that tests never exercise, but it has no runtime context and cannot know whether a flagged path is reachable. Tooling includes Semgrep (fast, rule-based, great for custom org policies), GitHub CodeQL (semantic query engine, deep taint tracking), SonarQube (quality plus security), and commercial engines like Checkmarx and Fortify. Results are commonly emitted as SARIF for pipeline ingestion.

Dynamic analysis (DAST)

DAST spiders a deployed application and fires malicious payloads at its endpoints, observing responses to infer vulnerabilities such as reflected XSS, SQL injection, and misconfigured headers. It is language-agnostic and produces low false positives because it demonstrates real behaviour, but it needs a running environment, struggles with authentication and single-page apps, and cannot point to the offending line of code. OWASP ZAP (open source, scriptable, automatable in CI) and Burp Suite (the manual pentester standard, with an enterprise scanner) are the dominant tools.

Interactive analysis (IAST) and RASP

IAST places an agent inside the application (a JVM agent, a .NET profiler, a Node hook) so that while functional or DAST tests drive traffic, the agent sees the exact tainted data flow through real code, yielding low false positives and precise file-and-line location. Its blind spot is coverage: it only reports on code the tests actually execute. RASP uses the same in-app instrumentation but in production, where it can block a live exploit attempt instead of just logging it.

Managing false positives

The single biggest reason AppSec programs fail is alert fatigue. Effective teams baseline existing findings so only new issues on the diff block a merge, tune rulesets to the stack, suppress with expiring justifications rather than blanket ignores, and deduplicate across tools by CWE and location. Reachability analysis and correlation between SAST and IAST sharply cut noise by confirming whether a static finding is actually exercised at runtime.

Key Takeaways

  • SAST is early and broad but noisy; run it on the diff at pull-request time and baseline aggressively.
  • DAST proves real exploitability with low false positives but needs a running app and authenticated coverage to be useful.
  • IAST combines DAST accuracy with SAST code-level precision, at the cost of depending on test coverage.
  • Correlating SAST and IAST is the most effective way to cut triage noise through reachability confirmation.
  • No automated tool finds business-logic flaws; keep manual penetration testing in the program with tools like Burp Suite.
  • Gates must break the build on new criticals; advisory-only scanning trains teams to ignore security.
  • RASP extends IAST instrumentation into production to block attacks live rather than only report them.
