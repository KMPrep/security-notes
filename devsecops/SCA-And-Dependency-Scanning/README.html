Software Composition Analysis & Dependency Scanning

Category: Software Supply Chain Security
Key standards / frameworks: OWASP Top 10 A06 Vulnerable and Outdated Components, NIST SP 800-218 (SSDF PW.4), CycloneDX, SPDX, OSV schema, CVSS, EPSS

Overview

Modern applications are mostly other people's code. A typical Node or Java service ships tens of thousands of transitive dependencies, and the majority of an application's attack surface lives in that third-party tree rather than in first-party code. Software Composition Analysis (SCA) inventories every declared and transitive component, matches it against vulnerability feeds, and flags known-vulnerable, unmaintained, or license-incompatible packages.

The stakes were made concrete by Log4Shell (CVE-2021-44228), where a single logging library exposed millions of Java applications to remote code execution, often through dependencies developers did not know they had. SCA answers the two questions that incident forced on everyone: do I use this component, and where. Combined with an SBOM, it turns a frantic manual audit into a database query.

How It Works

Dependency resolution and lockfiles

Package managers resolve a declared dependency into a full transitive graph. A lockfile (package-lock.json, yarn.lock, Cargo.lock, poetry.lock, go.sum) pins every resolved package to an exact version and cryptographic hash so builds are reproducible and an attacker cannot silently swap a dependency for a malicious version. SCA reads the lockfile to get the true, complete component list rather than trusting the loose ranges in the manifest.

Vulnerability feeds and scoring

Once the component list is known, SCA matches it against advisory databases. The NVD provides CVE records with CVSS scores; the GitHub Advisory Database curates ecosystem-specific advisories; and OSV (Open Source Vulnerabilities) offers a machine-readable, version-range-precise feed built for automation. Because raw CVSS over-prioritises, teams increasingly layer EPSS (exploitation probability) and reachability analysis to focus on the small fraction of vulnerabilities that are both exploitable and actually reached by their code.

flowchart TD
  A[Manifest and lockfile] --> B[Resolve full dependency graph]
  B --> C[Generate SBOM]
  C --> D[Match against OSV and NVD and GitHub Advisory]
  D --> E[Score with CVSS and EPSS]
  E --> F[Reachability and prioritisation]
  F --> G[Open fix pull request]
  G --> H[Break build on unfixed critical]

Automated remediation and confusion attacks

Dependabot and Renovate watch the graph and open pull requests that bump vulnerable packages, ideally with the lockfile hash updated in the same commit. Beyond known CVEs, SCA and registry configuration also defend against dependency confusion, where an attacker publishes a public package with the same name as an internal one at a higher version so the resolver pulls the malicious public copy. Scoped registries, namespace reservation, and pinned internal sources prevent this. See supply_chain for SBOM formats and provenance.

Tooling landscape

Snyk and GitHub Dependabot cover source-level dependency scanning; Trivy and Grype scan both dependencies and container image layers against OSV and vendor feeds; and Dependency-Track ingests CycloneDX SBOMs to track exposure continuously across a portfolio. Container scanning matters because a vulnerable OS package in the base image is as dangerous as a vulnerable application library.

Key Takeaways

  • Most of your attack surface is transitive third-party code; you cannot secure what you have not inventoried.
  • Lockfiles with cryptographic hashes are the foundation of reproducible, tamper-resistant builds.
  • OSV gives version-precise, machine-readable advisories built for automation; prefer it over scraping NVD alone.
  • Prioritise with EPSS and reachability, not raw CVSS, or the backlog becomes noise and nothing gets fixed.
  • Dependency confusion and typosquatting are namespace attacks; defend with scoped registries and publisher verification.
  • Automate remediation with Dependabot or Renovate so fixes ship as routine pull requests, not emergencies.
  • Scan container image layers, not just app dependencies; a vulnerable base image is a production exposure.
