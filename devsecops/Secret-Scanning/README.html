Secret Scanning & Credential Hygiene

Category: Application & Supply Chain Security
Key standards / frameworks: OWASP Top 10 A07 Identification and Authentication Failures, CWE-798 Hardcoded Credentials, NIST SP 800-218 (SSDF PW.9), CIS Controls, cloud provider secrets guidance

Overview

Secrets committed to source code are one of the most common and most damaging supply-chain exposures. API keys, database passwords, cloud access keys, and signing tokens end up in code, config files, notebooks, and CI logs, and once pushed to a shared or public repository they must be treated as compromised. Automated bots scrape GitHub for freshly committed AWS keys within seconds and use them to spin up crypto-mining fleets.

The defining property of a git-committed secret is that deleting it does not remove it: git history retains every prior blob, so a secret removed in a later commit is still trivially recoverable from the object store, forks, clones, and mirrors. This is why the only correct response to a leaked secret is to rotate it, not merely delete the line. Secret scanning combines pattern and entropy detection with pre-commit hooks and platform push protection to catch secrets before and after they land.

How It Works

Detection: patterns versus entropy

Two complementary techniques find secrets. Pattern (signature) detection matches known formats with regular expressions and structural checks, for example an AWS key prefix or a Stripe live-key shape, giving high precision for recognisable credentials. Entropy detection measures the randomness of a string using Shannon entropy to catch high-randomness blobs that match no known pattern, at the cost of more false positives. The strongest scanners add validation, actively testing a candidate credential against its provider API to confirm it is live before alerting.

Where scanning runs

Defence is layered across the developer workflow. Pre-commit hooks (detect-secrets, gitleaks) block a secret on the developer's machine before it is even committed. Server-side push protection (GitHub, GitLab) rejects a push containing a recognised secret so it never reaches the remote at all. Continuous repository and history scanning (TruffleHog, gitleaks) sweeps existing branches and full history to catch what earlier layers missed and to find secrets committed before scanning was adopted.

Rotate, do not just delete

When a secret is found, the incident sequence is fixed: revoke and rotate the credential at the provider first, then purge it from history if required, then investigate for abuse using access logs. Purging history with git filter-repo or BFG rewrites SHAs and breaks every clone and fork, and it still does not undo the exposure window, so rotation is always primary. Assume any secret that touched a remote has been harvested.

Moving secrets out of code

The durable fix is to stop putting secrets in code. Runtime secrets belong in a dedicated manager such as HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, or GCP Secret Manager, injected at deploy time and rotated automatically. For CI-to-cloud authentication, OIDC federation eliminates long-lived keys entirely by exchanging a short-lived signed workflow token for temporary cloud credentials, so there is no static secret to leak. This connects directly to CI/CD Pipeline Hardening.

Key Takeaways

  • A secret pushed to a remote is compromised; automated scrapers weaponise leaked cloud keys within seconds.
  • Deleting a secret does not remove it from git history, forks, or clones; rotation at the provider is the only real fix.
  • Combine pattern detection for known formats with entropy detection for custom tokens, and validate to cut noise.
  • Layer defences: pre-commit hooks, push protection, and continuous history scanning each catch what the others miss.
  • gitleaks, TruffleHog, and detect-secrets are the workhorses; wire them into hooks and CI, not just ad hoc scans.
  • Move runtime secrets into Vault or a cloud secrets manager and inject them at deploy time.
  • Use OIDC federation so CI never holds long-lived cloud keys, eliminating the highest-value static secret.
