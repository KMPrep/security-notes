CASB — Cloud Access Security Broker

Category: DLP, Cloud Security, Network Security
Parent: dlp/README.md

Overview

A Cloud Access Security Broker (CASB) is a security enforcement point between users and cloud services. It provides visibility into, and control over, data moving to and from cloud applications — both sanctioned (IT-approved) and unsanctioned (shadow IT).

CASB solves a specific gap that traditional DLP tools miss: employees directly accessing cloud services from a managed device without any traffic passing through a corporate network gateway. When a user opens a browser and logs into Dropbox directly over HTTPS, a traditional proxy can block the domain — but CASB can enforce fine-grained controls inside sanctioned apps and detect unsanctioned apps in use.

From a DLP perspective, CASB is the cloud-channel enforcement layer: the mechanism that enforces data movement policies for everything that flows through SaaS, PaaS, and IaaS APIs.

CASB Architecture Modes

CASB operates in three modes. Most enterprise deployments use a combination.

Mode 1: API Mode (Out-of-Band)

flowchart LR
    User[User] --> SaaS[Sanctioned SaaS App - e.g. Google Drive]
    SaaS --> CASB_API[CASB reads events via SaaS API]
    CASB_API --> Policy[Policy evaluation]
    Policy -->|Violation| Alert[Alert or retroactive quarantine]
    Policy -->|No violation| Log[Audit log]
  • How it works: CASB connects to the SaaS provider via admin API (e.g., Google Workspace Admin SDK, Microsoft Graph). It reads audit logs and file events after they occur.
  • Strengths: No traffic routing required; no latency impact; works on unmanaged devices.
  • Weaknesses: Retrospective — cannot block in real time. Best for detection and quarantine after the fact.
  • Use case: Discovering sensitive files shared externally in Google Drive; revoking sharing links after the fact.

Mode 2: Proxy / Forward Proxy Mode (Inline)

sequenceDiagram
    participant U as User
    participant E as Endpoint Agent
    participant C as CASB Inline Proxy
    participant S as Cloud Service
    U->>E: Upload sensitive file to Dropbox
    E->>C: Route traffic through CASB proxy
    C->>C: Inspect content - CONFIDENTIAL label found
    C->>C: Policy: Dropbox = unsanctioned - BLOCK
    C-->>U: 403 Forbidden
    Note over C,S: Request never reaches Dropbox
  • How it works: A PAC file or endpoint agent routes all SaaS traffic through the CASB proxy. The proxy performs TLS inspection and content inspection in real time.
  • Strengths: Real-time blocking; full content inspection; visibility into all SaaS including unsanctioned apps.
  • Weaknesses: Requires managed device to trust proxy certificate; adds latency; may break certificate-pinned apps.
  • Use case: Block upload of any CONFIDENTIAL file to personal cloud storage.

Mode 3: Reverse Proxy Mode (Session Proxy / ZTNA)

  • How it works: Access to sanctioned SaaS is routed through the CASB reverse proxy (acting as an access proxy, similar to BeyondCorp). The user never accesses the SaaS app directly.
  • Strengths: Works for unmanaged devices (BYOD, contractors); can enforce data controls without endpoint agent.
  • Weaknesses: Only covers sanctioned apps with configured reverse proxy; browser-rendered apps may break.
  • Use case: Enforce DLP on contractors using personal devices accessing Salesforce — block download of customer records.

CASB as DLP Enforcement Point

CASB applies DLP policies at the cloud application layer:

Control CASB capability
Block upload of sensitive files Inspect upload content; block if label = RESTRICTED
Block download to unmanaged device Device posture check; block download if device is unmanaged
Block external sharing Detect when files are shared outside the corporate tenant
Block unsanctioned app upload Block all upload to non-approved cloud storage domains
Quarantine exposed files Via API mode, remove public sharing links on sensitive files
Watermark downloads Add user identity watermark to downloaded sensitive files
Block copy-paste from SaaS In session proxy mode, intercept clipboard events in browser

Shadow IT Discovery

CASB passively discovers cloud applications in use by analyzing web proxy or firewall logs.

flowchart TD
    PL[Proxy or firewall logs] --> CASB[CASB Shadow IT analyzer]
    CASB --> Catalog[Match domains against cloud app catalog - 20000+ apps]
    Catalog --> Risk[Assign risk score per app]
    Risk --> Report[Shadow IT report: sanctioned vs unsanctioned vs high-risk]
    Report --> IT[IT team: sanction, block, or educate]

Typical findings:

  • 500–2000 cloud apps in use per 1000 employees
  • 20–40% of those are high-risk (consumer file sharing, personal cloud storage)
  • Many contain sensitive data uploaded without DLP controls

Data Governance in CASB

CASB integrates with the data classification system:

Event CASB action
User uploads file labeled CONFIDENTIAL to Google Drive Allow; log event with file identity
User attempts to share CONFIDENTIAL file outside corporate domain Block sharing; alert user and security team
User uploads file labeled RESTRICTED to any external service Block; create security incident
User downloads RESTRICTED file to unmanaged device Block; require managed device
User grants OAuth app broad access to Drive Alert; require admin approval

CASB for OAuth App Governance

OAuth app over-authorization is a significant DLP vector. An employee clicks "Allow" on a third-party OAuth app requesting drive.readonly or even drive (full read/write). That app now has persistent access to all the employee's Drive files.

CASB can:

  • Enumerate all third-party OAuth apps connected to the corporate tenant
  • Display data permissions each app has requested
  • Block apps above a risk threshold from being authorized
  • Revoke existing OAuth tokens for non-compliant apps
flowchart LR
    U[User grants OAuth app access] --> CASB[CASB OAuth governance policy]
    CASB --> R{App risk score}
    R -->|Low risk| A[Allow; log]
    R -->|Medium risk| AP[Require admin approval]
    R -->|High risk| BL[Block authorization; educate user]

What Can Go Wrong

Failure Root Cause Impact
BYOD devices bypass CASB proxy No endpoint agent; no proxy PAC file on personal device CASB has no visibility into BYOD activity
Certificate pinning breaks proxy App refuses proxy certificate CASB blind to that app's traffic; exfiltration possible
API mode latency means post-incident response only File already shared before CASB revokes Data may have been downloaded before revocation
OAuth app granted broad permissions before CASB governance enabled Legacy token with drive scope persists Rogue app has persistent full Drive access
CASB policy excludes executives Exec carve-outs create exploitable blind spots Insider or compromised exec account bypasses DLP
Shadow IT apps used on personal devices No proxy visibility on personal device Entire shadow IT landscape invisible

Mitigation Controls

  • Require managed device enrollment before any sanctioned SaaS access is permitted.
  • Run OAuth token governance on day one; revoke all apps that were granted broad scope before review.
  • Do not create blanket executive carve-outs; treat them as a risk acceptance decision with documentation.
  • Combine inline proxy mode (for real-time blocking) with API mode (for retrospective deep inspection).
  • Enable SCC integration so CASB findings appear in the same console as Cloud DLP and GuardDuty/GKE findings.

Key Takeaways

  • CASB is the cloud-channel DLP enforcement layer — it fills the gap between on-prem DLP and cloud SaaS.
  • API mode is retrospective; proxy mode is real-time. Both are needed.
  • Shadow IT discovery is a key CASB function — most organizations have 10× more SaaS apps in use than IT knows about.
  • OAuth app governance is a critical DLP control that CASB enables natively.
  • BYOD is an inherent gap in CASB proxy coverage; residual risk must be documented.
