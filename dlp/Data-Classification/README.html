Data Classification

Category: DLP, Data Governance, Data Protection
Parent: dlp/README.md

Overview

Data classification is the process of assigning a sensitivity label to data based on its content, context, and risk if disclosed. It is the foundation of every DLP control. Without classification, the DLP policy engine has no basis to decide what is sensitive and what is not.

Classification is not a one-time exercise. Data is created continuously — files, emails, database records, code commits, API payloads — and classification must be applied at creation time, updated when content changes, and enforced at every movement decision.

At Google scale, classification must be largely automated. Manual classification of petabytes of data is infeasible. But automated classification makes mistakes, and mistakes in both directions carry significant cost: over-classification breaks workflows; under-classification lets sensitive data move freely.

Classification Tiers

The standard enterprise model uses 4 tiers. Google internally uses similar tiers.

Tier Label Who Can Access Movement Policy Examples
Public PUBLIC Anyone, including anonymous No restrictions Press releases, product documentation, public APIs
Internal INTERNAL All employees and contractors Internal systems only; no personal accounts Internal wikis, team meeting notes, non-sensitive config
Confidential CONFIDENTIAL Specific teams or roles Restricted sharing; no external forwarding without approval Customer data, financial data, HR records, product roadmaps
Restricted RESTRICTED Named individuals or service accounts No external movement; logging required on all access Source code for core systems, M&A data, security keys, personal health information

Why 4 Tiers

More than 4 tiers creates confusion and misclassification. Fewer than 4 collapses important distinctions. The critical boundary is between CONFIDENTIAL and RESTRICTED — the latter implies that even internal circulation is controlled, not just external sharing.

How Classification Works

    A[Data created or modified] --> B{Classification method}
    B --> |Creator labels manually| C[Manual label applied]
    B --> |Auto-classification engine scans| D{Content matches rule?}
    D --> |Yes| E[Label applied based on highest match]
    D --> |No| F[Default label applied - usually INTERNAL]
    C --> G[Label stored as metadata or file property]
    E --> G
    F --> G
    G --> H[DLP policy engine reads label on movement event]
    H --> I{Policy allows movement?}
    I --> |Yes| J[Allow - emit audit log]
    I --> |No| K[Block or alert - create incident]

Auto-Classification Methods

Method How It Works Strengths Weaknesses
Keyword/regex matching Scan content for patterns: SSN (\d{3}-\d{2}-\d{4}), credit card (Luhn-valid 16-digit), email address Fast, deterministic, easy to audit High false positive rate on test data, sample data, code comments
Named entity recognition (NER) ML model identifies people, organizations, locations, dates in context Lower false positives than regex alone Requires training data; may miss domain-specific terminology
Document fingerprinting Hash known sensitive documents; match partial content against known-sensitive hash bank Very accurate for known documents; catches reformatted copies Only works for known content; does not detect novel sensitive data
Exact data matching (EDM) Compare against a structured dataset (e.g., a table of customer SSNs) Zero false positives for known data Requires maintaining and securing the reference dataset; performance-intensive
ML classifier Trained model classifies documents by sensitivity tier Handles complex context; adapts to language patterns Opaque; difficult to audit; requires ongoing training; can drift
Metadata / schema inference Classify based on file location, owner, filename pattern, database schema Low latency; no content inspection needed Does not catch mislabeled or misplaced files

Label Application: Where Labels Live

Labels must persist across format transformations. A PDF exported from a CONFIDENTIAL Google Doc must inherit the CONFIDENTIAL label.

Storage type Where the label lives
Files (Office, PDF, Google Docs) File metadata field; embedded in document properties; visual watermark
Email Message header; subject tag ([CONFIDENTIAL]); MIME metadata
Database records Column-level sensitivity tag; row-level label field
Cloud storage objects Object metadata key-value tag (sensitivity: RESTRICTED)
Code repository File-level classification comment; repo-level policy
API payloads Field-level classification in schema (JSON Schema x-sensitivity)

Failure Modes of Auto-Classification

Failure Root Cause Impact
Test data misclassified Code or QA data contains realistic-looking SSNs or credit card patterns False positives block developer workflows; alert fatigue
Context ignored Regex fires on the word "password" in security training material Analyst reviews are wasted on benign events
Label stripping User copy-pastes content into a new unlabeled document Label lost; DLP policy no longer applies to the copy
Image-embedded content Sensitive data in a screenshot PNG is not detected by text scanners Exfiltration via image upload bypasses DLP
Encryption masking content Encrypted ZIP or document sent before DLP can inspect Content inspection impossible without decryption
Stale labels File was PUBLIC; content was updated to include CONFIDENTIAL data; label not re-evaluated File treated as PUBLIC; no DLP enforcement
Classification drift Bulk auto-classify job ran once; new files added since are unclassified Growing pool of unlabeled sensitive data

Policy Binding: From Label to Enforcement

Classification without enforcement is a compliance theater exercise. Labels must be bound to enforceable policies.

flowchart LR
    L[Label: CONFIDENTIAL] --> P[DLP Policy: no external email]
    P --> E[Email gateway enforces: block outbound attachment]
    P --> F[CASB enforces: block upload to personal cloud storage]
    P --> G[Endpoint agent enforces: block USB copy]
    P --> H[Audit log: all access to CONFIDENTIAL files logged]

Data Residency and Classification

For regulated data (GDPR personal data, HIPAA PHI, PCI cardholder data), classification must also track residency — where the data is stored and whether movement crosses a jurisdictional boundary.

Regulation Data type Residency requirement DLP implication
GDPR EU personal data Cannot be transferred outside EEA without adequacy decision or SCCs DLP must block cross-region transfers without legal basis
HIPAA Protected Health Information (PHI) Must remain in covered entity or business associate environments DLP must block uploads to non-BAA cloud services
PCI DSS Cardholder data (CHD) Must stay within PCI-compliant scope DLP must block CHD from leaving the cardholder data environment
CCPA California consumer personal info Notice and opt-out requirements DLP should alert on bulk consumer data movements

What Can Go Wrong

Failure Consequence
Users manually downgrade labels to bypass DLP Sensitive data moves without enforcement
Auto-classifier trained on biased data Systematically misclassifies entire file types
No default label for unclassified data Unlabeled data has no DLP enforcement
Labels not persisted across format changes Copy-paste and export bypass classification
Classification engine compromised Attacker can re-label RESTRICTED files as PUBLIC

Mitigation Controls

  • Classify at creation time; never leave data unclassified — apply a default tier.
  • Protect the classification engine and label store with separate admin access controls.
  • Monitor for bulk label downgrades as a detection signal (insider threat indicator).
  • Use document fingerprinting alongside regex to catch reformatted or partially copied content.
  • Test classification accuracy quarterly with a labeled evaluation dataset.

Key Takeaways

  • Classification is the foundation; everything downstream depends on label accuracy.
  • Auto-classification must be continuously evaluated — it will drift and make mistakes.
  • Labels must persist through format transformations, copy-paste, and export.
  • Default to the most restrictive label when classification is uncertain.
  • Protecting the classification engine itself is a security requirement.
