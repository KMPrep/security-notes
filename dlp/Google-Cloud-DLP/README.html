Google Cloud DLP API (Sensitive Data Protection)

Category: DLP, Google Cloud, Data Protection
Parent: dlp/README.md

Overview

Google Cloud DLP — now branded Sensitive Data Protection — is a managed service that discovers, classifies, de-identifies, and protects sensitive data across Google Cloud storage systems and custom data streams. It is the primary cloud-native DLP tool in the Google ecosystem and directly relevant to the data-protection role.

Unlike endpoint or gateway DLP (which intercepts data in motion), Cloud DLP primarily works on data at rest — scanning Cloud Storage buckets, BigQuery tables, Datastore kinds, Firestore collections, and Cloud Spanner databases. It can also be called programmatically to inspect content on demand (e.g., in a data pipeline or API request handler).

What Cloud DLP Does

    A[Sensitive Data in Cloud Storage / BigQuery / Datastore] --> B[Cloud DLP Inspection Job]
    B --> C{Info type match found?}

Core Concepts

Info Types

An info type is a built-in or custom pattern definition for a category of sensitive data. Cloud DLP ships with 150+ built-in info types.

Info Type Name What It Detects
PERSON_NAME Human names using ML and gazetteer
EMAIL_ADDRESS Email addresses
PHONE_NUMBER Phone numbers (international formats)
US_SOCIAL_SECURITY_NUMBER SSN in multiple formats
CREDIT_CARD_NUMBER 13–19 digit sequences passing Luhn check
IBAN_CODE International bank account numbers
IP_ADDRESS IPv4 and IPv6 addresses
DATE_OF_BIRTH Dates that appear in a DOB context
PASSPORT Passport numbers (by country pattern)
MEDICAL_RECORD_NUMBER Healthcare MRN patterns
AUTH_TOKEN OAuth tokens, API keys, session tokens
GCP_CREDENTIALS GCP service account keys
GITHUB_ACCESS_TOKEN GitHub personal access tokens
ENCRYPTION_KEY Private key blocks (-----BEGIN...)
US_DRIVERS_LICENSE_NUMBER State-specific DL formats

Custom info types: You can define your own using:

  • Regex patterns with optional context rules
  • Dictionary of exact match terms
  • Stored info type (a large dataset of exact values, similar to EDM)

Likelihood Scores

Cloud DLP does not return binary match/no-match. It returns a likelihood score representing confidence in the finding.

Score Meaning Example
VERY_UNLIKELY Pattern matches but extremely low confidence 9-digit number in non-SSN context
UNLIKELY Pattern matches; some evidence against Possible SSN with no surrounding context
POSSIBLE Reasonable chance the data is sensitive SSN-like pattern with adjacent name
LIKELY High confidence SSN with adjacent name, date, and address
VERY_LIKELY Very high confidence SSN with explicit label in structured record

Policy implication: Set the minimum likelihood threshold per rule. Use LIKELY or VERY_LIKELY for blocking decisions; use POSSIBLE for audit and review.

Inspection Requests

An inspection request defines:

  • Storage target — what to scan (Cloud Storage bucket, BigQuery table, Datastore kind, custom content)
  • Inspect config — which info types to look for, minimum likelihood threshold, byte limits per item
  • Action — what to do with findings (save to BigQuery, publish to Pub/Sub, report to SCC, de-identify)
# Example Cloud DLP inspection request structure (Python SDK)
from google.cloud import dlp_v2

dlp = dlp_v2.DlpServiceClient()

item = dlp_v2.ContentItem(
    value="Customer name: John Doe, SSN: 123-45-6789"
)

inspect_config = dlp_v2.InspectConfig(
    info_types=[
        dlp_v2.InfoType(name="PERSON_NAME"),
        dlp_v2.InfoType(name="US_SOCIAL_SECURITY_NUMBER"),
    ],
    min_likelihood=dlp_v2.Likelihood.LIKELY,
    limits=dlp_v2.InspectConfig.FindingLimits(max_findings_per_request=100),
    include_quote=True,
)

response = dlp.inspect_content(
    request={
        "parent": "projects/my-project",
        "inspect_config": inspect_config,
        "item": item,
    }
)

for finding in response.result.findings:
    print(f"Found {finding.info_type.name} at likelihood {finding.likelihood}")

De-identification Transforms

Cloud DLP can not only detect sensitive data but also transform it to remove or obfuscate the sensitive content. This is critical for:

  • Anonymizing PII before loading into analytics pipelines
  • Redacting sensitive data from logs before sharing with third parties
  • Tokenizing data for secure storage with the ability to re-identify later
Transform What it does Reversible Use case
Redaction Replace matched text with [REDACTED] or a surrogate No Log sharing, document sanitization
Replacement Replace matched text with a fixed string No Uniform placeholder: [EMAIL]
Masking Replace characters with a mask character (*) No Display in UI: ***-**-6789
Bucketing Map a numeric value to a bucket No Age 32 → 30-40 for analytics
Date shifting Shift all dates by a random constant offset No (constant per dataset) Preserve date ordering without real dates
Format-preserving encryption (FPE) Encrypt while preserving format (FPE-AES/FF1) Yes (with key) Tokenization — reversible with key
Pseudonymization Replace with a deterministic hash or keyed token Yes (with key) Link records across systems without exposing PII
Crypto replace with FPE AES-SIV or AES-FF1 to produce a format-preserving encrypted token Yes (with key) Tokenize credit cards while keeping format

    PII[John Doe, SSN: 123-45-6789] --> DLP[Cloud DLP de-identify]
    DLP -->|Redaction| R[Redacted, SSN: REDACTED]
    DLP -->|Masking| M[John Doe, SSN: ***-**-6789]
    DLP -->|FPE tokenization| T[John Doe, SSN: 978-32-1045]
    T -->|Re-identify with key| PII

Storage Scanning: Inspection Jobs

Cloud DLP inspection jobs run against storage systems:

Storage system How Cloud DLP scans it
Cloud Storage Triggers scan on new files via Pub/Sub notification; or scheduled full scan
BigQuery Samples rows (configurable count); can scan all rows with larger limits
Datastore / Firestore Scans entity kinds; configurable field list
Cloud Spanner Via BigQuery export or direct API (preview)
Custom content Inline API call with content bytes or string

Sampling: For very large datasets, scanning every row is expensive. Cloud DLP supports sampling strategies: scan top N rows, scan random N%, or scan in a date range. Sampling introduces the risk of missing sensitive data in unsampled rows — this is a known limitation.

Security Command Center Integration

Cloud DLP findings can be published to Security Command Center (SCC) as security findings. This enables:

  • Single pane of glass for DLP findings alongside vulnerability findings and threat detections
  • Alerting via SCC notification channels to Pub/Sub, then to PagerDuty / SIEM
  • Automatic classification of assets (e.g., "this Cloud Storage bucket contains PII")
flowchart LR
    Bucket[Cloud Storage Bucket] --> DLP[Cloud DLP Inspection Job]
    DLP -->|Finding: CREDIT_CARD_NUMBER| SCC[Security Command Center]
    SCC --> Finding[Finding: Bucket contains PII]
    SCC --> Alert[Pub/Sub notification]
    Alert --> SIEM[SIEM / PagerDuty]
    Finding --> Posture[Asset posture: high-sensitivity]

Hybrid Inspection (On-Premises and Multi-Cloud)

Cloud DLP can also inspect data that is not in GCP, using the hybrid inspection feature. A local agent sends data to the Cloud DLP service for inspection; findings are returned and can be logged locally or in GCP.

Use case: inspect data from on-prem databases or AWS S3 buckets using Cloud DLP info types without migrating data to GCP.

Access Control for Cloud DLP

Role Permissions Use case
roles/dlp.admin Create and delete inspect templates, de-identify templates, jobs DLP platform team
roles/dlp.jobsEditor Create and run inspection jobs Security analyst or pipeline service account
roles/dlp.jobsReader Read job status and findings Read-only dashboard
roles/dlp.reader Read templates and configs Audit role

Least privilege for Cloud DLP:

  • Data pipelines that call Cloud DLP for inline inspection should have roles/dlp.user only.
  • Service accounts scanning production buckets should have roles/storage.objectViewer on specific buckets, not project-wide storage admin.
  • Findings that contain sensitive data (e.g., include_quote=true was set) should be stored in a tightly access-controlled BigQuery dataset.

What Can Go Wrong

Failure Root Cause Impact
Sampling misses sensitive rows Sampling rate too low for dataset size False confidence; sensitive data undetected
include_quote=true in findings log Finding log stores the actual sensitive value Secondary exposure — findings log now contains PII
Overly broad service account Pipeline SA has storage admin instead of object viewer Compromised pipeline can read all buckets
No re-scan on data update Inspection job ran once; bucket receives new sensitive files New sensitive files undetected
Custom info types not maintained Regexes become stale; new credential formats not added Credentials committed to repos not detected
Findings not routed to SIEM SCC findings not forwarded Security team never sees DLP findings
De-identification key not rotated FPE key used for years without rotation Stale key increases risk if key leaks

Mitigation Controls

  • Never set include_quote=true in findings destined for a broadly readable log; use it only in access-controlled investigation pipelines.
  • Trigger re-inspection on Cloud Storage via Pub/Sub notifications on object create events.
  • Use Org Policy constraints to require Cloud DLP inspection before sensitive buckets can be read by new principals.
  • Store de-identification keys in Cloud KMS; rotate on schedule; log all key usage.
  • Route all DLP findings to Cloud Logging + SCC + SIEM; alert on Critical and High likelihood findings.
  • Maintain a custom info type library for internal credential formats (internal API keys, internal token patterns).

Key Takeaways

  • Cloud DLP is the primary DLP tool in the GCP ecosystem — know it at API depth for the data-protection role.
  • Info types + likelihood thresholds are the core tuning knobs; set them per use case.
  • De-identification is not just about detection — it enables safe data sharing and analytics.
  • FPE tokenization is reversible with the key; use Cloud KMS to protect de-identification keys.
  • The include_quote flag is a critical security setting — misuse creates secondary PII exposure.
  • Sampling is a performance tradeoff with a security cost; document the sampling rate and its implications.
