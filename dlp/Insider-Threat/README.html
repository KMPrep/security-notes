Insider Threat and DLP

Category: DLP, Insider Threat, UEBA
Parent: dlp/README.md

Overview

Insider threat is the category of risk where a person with legitimate access to systems and data misuses that access to cause harm — whether through data exfiltration, sabotage, fraud, or espionage. DLP plays a central role in the insider threat detection and prevention framework, but it is not sufficient alone.

The critical distinction: external attacker DLP is about stopping data from leaving after a breach. Insider threat DLP is about detecting authorized users doing unauthorized things with data they are allowed to access.

An insider who can legitimately open every file in the repository can also exfiltrate all of it — DLP cannot block the read. DLP must detect anomalous patterns in how the authorized user moves or aggregates that data.

Insider Threat Taxonomy

Type Description DLP-relevant behaviors
Malicious insider Employee or contractor intentionally exfiltrating data for personal gain, competitor hire, or espionage Bulk download, USB copy, email to personal account, staging files before departure
Negligent insider Employee causing data loss through carelessness — misaddressed email, wrong sharing settings, public S3 bucket Accidental external share, emailing wrong recipient, misconfigured cloud storage
Compromised insider Employee account taken over by external attacker (phishing, credential theft, AiTM) Unusual login location, impossible travel, bulk download after credential compromise
Privileged insider IT admin, DLP admin, or security analyst who abuses privileged access Modifying DLP policy exclusions, deleting incident records, exporting raw classification data

The Insider Threat Kill Chain

flowchart LR
    A[Trigger event: resignation, termination notice, financial stress, grievance] --> B[Reconnaissance: discover what data is accessible]
    B --> C[Collection: aggregate sensitive files in staging location]
    C --> D[Exfiltration: move data via email, USB, cloud sync, print]
    D --> E[Cover tracks: delete local copies, clear logs if possible]

Key observation: The kill chain has multiple phases where detection is possible before exfiltration completes:

  • Reconnaissance: unusual access to directories or files the user rarely touches
  • Collection (staging): bulk file copy to desktop, unusual folder creation, large download of files
  • Exfiltration: covered by channel-specific DLP controls
  • Cover tracks: deletion of large numbers of files; attempts to clear browser history

UEBA — User and Entity Behavior Analytics

UEBA is the detection layer that catches insider threat activity that traditional DLP thresholds miss. It establishes a behavioral baseline for each user and alerts on statistically significant deviations.

flowchart TD
    L1[User activity logs - file access, email, web, auth] --> B[Baseline engine]
    B --> Normal[Normal behavior profile per user: typical hours, volumes, destinations]
    L1 --> Current[Current event stream]
    Current --> Anomaly{Deviation from baseline?}
    Normal --> Anomaly
    Anomaly -->|No significant deviation| Log[Log only]
    Anomaly -->|Significant deviation| Risk[Risk score elevated]
    Risk --> Alert[Alert to SOC or manager]
    Risk --> Action[Trigger enhanced DLP - move to block mode for this user]

UEBA Signals for Insider Threat

Signal Baseline Anomaly
File download volume User downloads ~50 files/day User downloads 5,000 files on a Monday
Access time User active 9am–6pm weekdays User active at 2am on Saturday
Data destination User emails internally only User starts emailing to personal Gmail
Cloud app usage User uses Google Drive only User accesses Dropbox, Box for first time
USB activity No USB activity in 180 days USB inserted; 2GB of files copied
Search queries User searches work topics User searches "how to copy files without IT detection"
Printing volume User prints ~5 pages/week User prints 400 pages in one day
Access scope User accesses own team's files User accesses finance, HR, and legal in same session
Authentication User authenticates from London User authenticates from Singapore 2 hours later (impossible travel)

High-Risk Lifecycle Events

Certain organizational events significantly elevate insider threat probability. DLP controls should be automatically tightened during these periods.

Lifecycle event Risk elevation DLP response
Resignation submitted High Move user to enhanced monitoring; USB blocked; external email requires approval
Termination notice Critical Immediate enhanced monitoring; disable external email; restrict to read-only if possible
Performance improvement plan (PIP) Medium-high Enhanced monitoring; alert on unusual access patterns
Disciplinary action Medium-high Enhanced monitoring
Demotion or role change Medium Review access rights; remove access no longer needed
Failed promotion Medium Monitor for elevated access attempts and unusual data aggregation
M&A / acquisition news High Employees aware of undisclosed material events may exfiltrate before announcement
Layoff announcement High Fleet-wide increase in USB and cloud sync activity is typical

Implementation: HR system triggers lifecycle event; event is sent to identity governance system; DLP policy for the affected user is automatically updated (JML integration). See also: auth_authz/Governance/Joiner-Mover-Leaver/.

Privileged Insider: The DLP Admin Problem

A DLP admin can:

  • Add themselves to a policy exclusion list
  • Lower the sensitivity threshold for their user account
  • Delete findings and incident records
  • Export the raw fingerprint database
  • Access classification metadata to identify the most sensitive files

Mitigations:

  • Require two-person integrity (2PI) for DLP policy changes — no single admin can modify their own monitoring profile
  • Log all DLP admin actions to a tamper-evident log that DLP admins cannot access
  • Alert on DLP policy changes (especially exclusion list modifications) as a security event
  • Restrict access to DLP incident records to a separate investigation team, not the DLP ops team
  • Use separation of duties: DLP configuration team ≠ DLP incident response team ≠ DLP audit team

DLP vs UEBA: Complementary, Not Redundant

Capability DLP UEBA
Detect a file containing PII being emailed externally Yes — content inspection triggers rule Depends on volume anomaly
Detect 1 small file exfiltrated per day for 30 days No — below volume threshold Yes — deviation from baseline over time
Detect user accessing files they don't normally access No Yes — scope anomaly
Block exfiltration in real time Yes — inline enforcement No — detection only
Detect impossible travel after credential compromise No Yes
Detect DLP policy being modified by an insider admin No Yes — admin behavior anomaly

The right model: UEBA elevates the risk score of a user → DLP moves that user from "monitor" mode to "block" mode → exfiltration attempt is blocked in real time.

Investigation Workflow

When a DLP or UEBA alert fires for a suspected insider:

flowchart TD
    A[Alert: user X - high risk score + DLP trigger] --> B[Triage: false positive check]
    B -->|False positive| C[Document and suppress rule]
    B -->|Credible| D[Preserve evidence: logs, file access, DLP findings]
    D --> E[Notify HR, Legal, and manager per policy]
    E --> F[Determine scope: what was accessed and when]
    F --> G[Interview employee if legally permitted]
    G --> H{Data confirmed exfiltrated?}
    H -->|Yes| I[Revoke access; legal escalation; breach assessment]
    H -->|No| J[Document; return to baseline monitoring]
    I --> K[Regulatory notification if PII breach confirmed]

Legal and HR process: In most jurisdictions, accessing employee communications and files for investigation purposes requires documented legal basis. Work with Legal and HR before accessing personal email content, even on company devices. Jurisdiction matters — GDPR imposes restrictions on employee monitoring in the EU.

What Can Go Wrong

Failure Root Cause Impact
No UEBA baseline established UEBA deployed but no historical data for baseline All users appear anomalous; alert storm
DLP not integrated with HR lifecycle events Manual process; HR does not notify security Resigning employee gets 2 weeks of unrestricted access
Overly aggressive monitoring All employees treated as suspects Employee morale, legal liability, union issues
Privileged admin not monitored DLP admin excluded from monitoring Privileged insider has unchecked access
Investigation without legal sign-off Security team accesses employee files without process Legal liability; evidence inadmissible
Alert fatigue from UEBA Too many false positives Analysts stop reviewing; real insider missed

Mitigation Controls

  • Integrate DLP with HR lifecycle systems for automatic access tightening on key events.
  • Establish a UEBA behavioral baseline for every user before using it for detection.
  • Treat privileged DLP admins as a separate risk tier with mandatory 2PI for policy changes.
  • Use the JML (Joiner-Mover-Leaver) process to revoke access immediately on termination.
  • Document the investigation process with Legal before any insider investigation begins.
  • Use honeytokens — fake credentials and files — as tripwires in sensitive directories.

Key Takeaways

  • Insider threat DLP detects authorized users misusing authorized access — not unauthorized access.
  • UEBA is essential for catching low-and-slow exfiltration that volume thresholds miss.
  • High-risk lifecycle events (resignations, terminations, PIPs) must trigger enhanced DLP controls automatically.
  • Privileged insiders (DLP admins) are a distinct and severe threat — separation of duties is required.
  • Legal and HR must be involved before acting on insider threat evidence in most jurisdictions.
