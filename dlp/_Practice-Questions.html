DLP — Practice Questions

Category: DLP, Data Protection, Security Engineering
Parent: dlp/README.md

Overview

These 15 questions cover the full DLP domain from data classification through Google Cloud DLP API, enforcement channels, CASB, insider threat, and policy design. They are written at the Google senior security engineer bar: each question expects a structured answer that covers architecture, failure modes, detection, and scale.

For each question: answer aloud, timed (5–8 minutes per question), no notes.

Questions

1. DLP Architecture and Classification

Question:

"Design a DLP system from scratch for a 100,000-employee company. Where does classification happen, where is policy enforced, and what are the three most likely places the system will fail?"

What a strong answer covers:

  • Classification layer: auto-classification engine using regex + NER + document fingerprinting; applied at creation time; default to INTERNAL if unclassified
  • Enforcement layers: email gateway, web proxy with TLS inspection, endpoint agent, CASB for cloud SaaS
  • Policy engine: label + context + user/device context evaluated per movement event
  • Three failure modes: encrypted channels bypass content inspection; BYOD devices bypass endpoint agent; label stripping when content is copy-pasted into a new unlabeled document

2. Data Classification Failure Modes

Question:

"What are the failure modes of auto-classification? How does each one produce either false positives or false negatives, and what mitigations exist for each?"

What a strong answer covers:

  • False positives: test data with realistic PII patterns; security documentation; sample datasets → mitigate with context rules, proximity matching, exclusion for dev/test environments
  • False negatives: label stripping (copy-paste), image-embedded content (no OCR), encrypted documents, novel sensitive data formats not in info type library
  • Classification drift: bulk classification job runs once but new files arrive unclassified
  • Manual label downgrade by users: treat bulk downgrade events as UEBA signals

3. Encrypted Channels and DLP Blind Spots

Question:

"A sophisticated insider knows DLP is deployed. What techniques can they use to bypass it, and for each technique, what is the most effective control — or is it an accepted residual risk?"

What a strong answer covers:

  • Encrypt before sending (password-protected ZIP): policy is to block all password-protected containers; or alert on them; or require justification
  • Steganography: no scalable technical control; residual risk; behavioral detection is the fallback
  • Personal device / phone hotspot: unmanaged device bypasses all endpoint controls; mitigate with network access control (NAC), ZTNA requiring managed device
  • DNS tunneling: low bandwidth but undetected; mitigate with DNS query volume and entropy anomaly detection
  • Photo of screen with personal phone: no technical control; physical controls (no-phone zones for high-sensitivity work); residual risk accepted

4. Google Cloud DLP API

Question:

"Walk me through how you would use the Google Cloud DLP API to detect and de-identify PII in a BigQuery dataset that is being used for analytics. What are the security tradeoffs of each de-identification method?"

What a strong answer covers:

  • Inspection job against BigQuery: configure info types (PERSON_NAME, EMAIL_ADDRESS, US_SOCIAL_SECURITY_NUMBER), set likelihood threshold LIKELY
  • Findings published to Cloud Logging and SCC
  • De-identification options: redaction (irreversible, safe for sharing), masking, date shifting, FPE tokenization (reversible with Cloud KMS key)
  • Tradeoff: FPE preserves analytical utility (format-preserving); redaction destroys analytical value; both have their place
  • Security: never set include_quote=true in findings sent to broadly readable logs — findings log would itself contain PII
  • Sampling risk: BigQuery sampling misses sensitive rows in unsampled portion; document the sampling rate

5. CASB Architecture and Modes

Question:

"A user on a managed laptop accesses corporate Google Drive via a browser and uploads a file containing customer PII to their personal Dropbox account. Walk me through how each CASB mode would or would not catch this, and which mode you would choose."

What a strong answer covers:

  • API mode: Dropbox personal account — no Dropbox API connection to corporate; blind. Google Drive API mode would have caught the original access. Cannot block Dropbox upload.
  • Proxy / forward proxy mode: routes browser HTTPS through inline proxy → TLS inspection → inspect upload body → detect PII → check destination (Dropbox personal = unsanctioned) → block
  • Reverse proxy mode: only covers sanctioned apps; Dropbox personal is not in reverse proxy config → blind
  • Best answer: deploy forward proxy mode with endpoint agent installing trust certificate; additionally have API mode for Google Drive to catch retroactive sharing
  • Device must be managed for proxy certificate to be trusted

6. Insider Threat: Departing Employee

Question:

"A sales manager submits their resignation. They have access to the entire customer database and all open deal files. What DLP and non-DLP controls do you activate, and how quickly?"

What a strong answer covers:

  • Immediate (same day as resignation): integrate HR event with identity governance; reduce cloud sync client permissions on sensitive repositories; enable enhanced monitoring for this user
  • Within 24 hours: USB blocked; external email requires business justification; manager notified of enhanced policy
  • UEBA: establish what the user's normal data access looks like; alert on any anomalous bulk download or file movement
  • Non-DLP: reduce access scope to only what is needed for the notice period; enable session recording on privileged system access
  • Day of departure: immediate access revocation; device return + wipe verification; audit log review of final 30 days of activity
  • Legal caveat: process must be documented and legally reviewed; employee must not be informed of individual monitoring activities without legal guidance

7. DLP Policy False Positives

Question:

"You deploy a new DLP rule to detect source code being sent to personal email. On day one, 2,000 alerts fire and most of them are from your own security engineering team sending CVE reports and security research. How do you handle this?"

What a strong answer covers:

  • Move the rule immediately from block to audit mode (if it was in block mode) to stop the workflow disruption
  • Analyze the false positive pattern: security team sending external security content is a distinct use case
  • Create a scoped exception: security team + approved external security domains (NVD, CVE.mitre.org, vendor PSIRT)
  • Do not create a blanket security team exclusion — scope it to the specific activity
  • Document the exception with business justification and expiry review date
  • Tune the rule: add context (source code patterns must appear alongside customer data or proprietary function names, not just generic code)
  • Set a target: < 1% FP rate before re-enabling block mode

8. DLP for Secrets and Credentials

Question:

"A developer commits an AWS access key to a public GitHub repository. How should DLP have prevented this, and how should the organization respond now that it has happened?"

What a strong answer covers:

  • Prevention: pre-commit hook with secret scanner (git-secrets, detect-secrets, Gitleaks) catching AKIA[0-9A-Z]{16} pattern before push; CI/CD pipeline scan as second layer; GitHub Advanced Security secret scanning as third layer
  • Why DLP alone is insufficient: developer tools often bypass endpoint DLP agent; pre-commit hooks are the right enforcement point for code repos
  • Response: immediately revoke the AWS access key via IAM (not just delete the commit — GitHub history preserves it); scan CloudTrail for any use of that key between commit and revocation; determine if any unauthorized access occurred; rotate related credentials
  • Google Cloud DLP has a built-in GCP_CREDENTIALS and AUTH_TOKEN info type — deploy inspection jobs on Cloud Storage to catch accidentally uploaded key files

9. Insider Threat: Privileged DLP Admin

Question:

"Your DLP admin decides to exfiltrate sensitive data. They know the DLP system better than anyone. What controls prevent this, and what is your residual risk?"

What a strong answer covers:

  • The admin can modify their own exclusion list, lower likelihood thresholds, delete incident records
  • Controls: two-person integrity (2PI) for all DLP policy changes; DLP admin actions logged to a separate immutable audit store that DLP admins cannot access; DLP admin cannot modify their own monitoring profile
  • Alert on: bulk exclusion additions, threshold changes, deletion of findings, DLP policy export
  • Separation of duties: DLP ops team ≠ DLP investigation team ≠ DLP audit team
  • Residual risk: a highly sophisticated DLP admin who understands the audit pipeline could theoretically construct a blind spot gradually over time via tiny policy increments — detect via: periodic red team exercises against DLP coverage, anomaly on DLP alert volume trends

10. DLP at Google Scale

Question:

"Google processes billions of events per day. How do you design a DLP inspection pipeline that performs content inspection on every outbound email and web upload without adding unacceptable latency?"

What a strong answer covers:

  • Inline path: lightweight signal extraction only (regex, fingerprint hash lookup) — sub-50ms; async deep inspection runs in parallel
  • Decision: if inline signal is conclusive (RESTRICTED label present, known fingerprint match, private key pattern) → block immediately
  • Decision: if inline signal is uncertain (possible PII, new file type) → allow with async inspection queued; retroactive quarantine or alert if async confirms
  • Architecture: use a message queue (Pub/Sub) to decouple inline enforcement from deep inspection
  • Tiered processing: fast path for known-safe files (hash match against known-clean corpus); slow path for unknown files
  • Scale: horizontal scaling of inspection workers; circuit breakers to prevent DLP latency from becoming an availability issue for email/web
  • Sampling for at-rest scanning: accept that 100% inspection of all data at rest is cost-prohibitive; risk-based sampling weighted toward high-sensitivity buckets

11. VPC Service Controls and Data Exfiltration

Question:

"How do VPC Service Controls in GCP prevent data exfiltration, and how does this complement Cloud DLP?"

What a strong answer covers:

  • VPC Service Controls define a security perimeter around GCP resources; API requests that attempt to move data outside the perimeter are denied even if the caller is authenticated and authorized
  • Example: a compromised service account cannot call storage.objects.copy to move data from a protected BigQuery dataset to an external Cloud Storage bucket, even with valid credentials
  • This is an authorization-level control, not a content inspection control — it blocks movement by perimeter, not by content
  • Complements Cloud DLP: VPC-SC prevents bulk exfiltration via API even by authorized identities; Cloud DLP detects sensitive content in data that is within the perimeter
  • Together: VPC-SC reduces blast radius of a compromised credential; Cloud DLP identifies which data is most sensitive and where it lives
  • Limitation: VPC-SC does not inspect content; an attacker inside the perimeter with authorized access can still read data; Cloud DLP is needed to identify what they are looking at

12. DLP for Regulated Data (GDPR/HIPAA/PCI)

Question:

"A company processes EU personal data, US health records, and credit card data. How do you design DLP to meet the different requirements of GDPR, HIPAA, and PCI DSS simultaneously?"

What a strong answer covers:

  • GDPR: classify all EU personal data; restrict cross-border transfers to EEA or countries with adequacy decision; DLP blocks transfer of EU PII to regions without a legal basis; 72-hour breach notification requirement means DLP incident queue must be monitored continuously
  • HIPAA: PHI must stay within BAA-covered environments; DLP blocks any PHI from going to services without a BAA in place; minimum necessary principle (DLP helps enforce access scope)
  • PCI DSS: cardholder data environment (CDE) scope; DLP must ensure CHD does not leave CDE boundary; requirement 3 (protect stored CHD) and requirement 4 (encrypt CHD in transit) both have DLP implications
  • Different sensitivity labels per regulation: PII-EU, PHI, CHD — each with different policies and enforcement
  • Single DLP platform should support multiple simultaneous policy sets; a file can carry multiple labels

13. Behavioral Detection for DNS Exfiltration

Question:

"An advanced persistent threat is exfiltrating data from your network via DNS tunneling — encoding data in subdomain labels of an attacker-controlled domain. How do you detect this with DLP and other controls?"

What a strong answer covers:

  • DNS exfiltration cannot be detected by content-based DLP (the data is encoded in DNS queries, not HTTP POST bodies)
  • Detection relies on behavioral anomalies: abnormally high DNS query volume per host, unusually long subdomain labels (normal hostnames are short), high entropy in subdomain labels (random-looking = encoded data), repeat queries to the same registered domain with varying subdomains
  • Security controls: DNS resolver logs to SIEM; anomaly detection rule on query frequency per host; entropy filter on subdomain labels; block or alert on newly registered domains (registrars selling domains used within hours is a C2 signal)
  • Rate limiting: even if content cannot be inspected, the bandwidth available via DNS tunneling is low (a few KB/s) — detecting large volumes is feasible
  • Segmentation: endpoints that should not be making direct DNS queries to the internet (they should use an internal recursive resolver) should be blocked from doing so at the network level
  • This is a gap in content-based DLP; must be in the detection/network monitoring layer

14. DLP Integration with Security Incident Response

Question:

"A DLP alert fires: a user sent 500 files labeled RESTRICTED to their personal Gmail. How do you run the incident from the moment the alert fires to closure?"

What a strong answer covers:

  • Immediately: assess severity — 500 RESTRICTED files = P0/P1; escalate to incident commander; do not alert the user yet
  • Preservation: pull DLP logs, email gateway logs, CASB logs; preserve the sending event before it ages out of short-retention logs; capture which files, timestamps, recipient
  • Contain: disable the user's ability to send external email pending investigation; do not lock the account yet (avoid tipping off)
  • Scope: was this 500 files in one send (bulk exfiltration) or 500 accumulated sends over time (ongoing low-and-slow)?
  • Notify: HR, Legal, employee's manager, CISO
  • Legal escalation: if files contain regulated data (PII, PHI, CHD) — regulatory breach notification timeline starts
  • Parallel track: determine if the personal Gmail account received the files; if the email provider has a legal process for removal
  • Lessons learned: was DLP in alert mode (not block mode) for this user or for this channel? Why? Change to block mode.

15. DLP vs Zero Trust

Question:

"BeyondCorp ensures only authorized users on trusted devices can access corporate resources. If Zero Trust is deployed correctly, is DLP still necessary? Why or why not?"

What a strong answer covers:

  • BeyondCorp controls access — who can reach which resource from which device. It does not control movement — what an authorized user can do with data once they have legitimate access.
  • A user with a managed, fully-compliant device and strong authentication is authorized to access the customer database. BeyondCorp says yes. DLP must still evaluate: can this user email 50,000 customer records to their personal Gmail?
  • Zero Trust and DLP are orthogonal: Zero Trust prevents unauthorized access; DLP prevents unauthorized data movement by authorized users.
  • Insider threat: BeyondCorp is specifically about device and identity trust. It cannot model "this user is about to resign and is bulk-downloading files before leaving." That is a behavioral signal that UEBA and DLP detect.
  • The strong answer: Zero Trust reduces the attack surface and limits blast radius after credential compromise. DLP is still required because authorization and data movement policy are different dimensions of data protection. Both are necessary; neither replaces the other.
