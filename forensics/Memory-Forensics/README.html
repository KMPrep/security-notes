Memory Forensics

Category: Digital Forensics — RAM Analysis, Volatile Evidence

Overview

Memory forensics analyses the contents of a system's RAM to recover running processes, loaded modules, network connections, decrypted data, and attacker artefacts that never touch disk. It is one of the most powerful forensic techniques because:

  • Malware increasingly runs entirely in memory (fileless malware)
  • Encryption keys, plaintext passwords, and decrypted data exist only in RAM
  • Process injection and hollowing are only visible in memory
  • Rootkits that hide from disk forensics are visible in memory

The key constraint: RAM is volatile — it is lost on shutdown. Memory acquisition must happen on a live, running system.

Memory Smear

Memory is constantly changing while being read. The dump will have some inconsistency — a process might be partially captured mid-write. This is called memory smear. It is unavoidable with live acquisition. Work around it by acquiring quickly and treating artefacts as probabilistic, not definitive.

Virtual vs Physical Memory

Concept Description
Physical memory Actual RAM chips; addressed by physical addresses
Virtual memory Each process sees its own address space; the OS translates virtual → physical via page tables
Page file / swap OS writes memory pages to disk when RAM is full; forensically important — may contain artefacts after RAM is cleared
ASLR Randomises virtual addresses each boot; complicates analysis of absolute addresses

A memory dump contains physical memory. Forensic tools translate physical addresses to virtual addresses per-process using the page directory base.

Volatility Framework

Volatility is the standard open-source memory forensics framework. It parses a memory dump and provides plugins to extract artefacts.

Basic usage:

# Identify the OS profile (Volatility 2)
volatility -f memory.dmp imageinfo

# List running processes
volatility -f memory.dmp --profile=Win10x64_19041 pslist

# Show process tree
volatility -f memory.dmp --profile=Win10x64_19041 pstree

# List network connections
volatility -f memory.dmp --profile=Win10x64_19041 netscan

# Dump a process's executable
volatility -f memory.dmp --profile=Win10x64_19041 procdump -p 1234 -D /output/

# Scan for injected code
volatility -f memory.dmp --profile=Win10x64_19041 malfind

# Extract command line arguments
volatility -f memory.dmp --profile=Win10x64_19041 cmdline

# List loaded DLLs per process
volatility -f memory.dmp --profile=Win10x64_19041 dlllist -p 1234

# Scan for YARA matches
volatility -f memory.dmp --profile=Win10x64_19041 yarascan -y rules.yar

Volatility 3 (newer) uses symbol tables instead of profiles and has improved Windows 10/11 support.

Process Injection Detection

Process injection is when malware executes code inside another process's address space. The injected code runs under the target process's identity, evading process-based detection.

Common Injection Techniques

Technique How It Works Volatility Detection
Classic injection VirtualAllocEx + WriteProcessMemory + CreateRemoteThread malfind finds RWX region with PE header
Process hollowing Spawn suspended process, replace memory with malware malfind; PE headers at unexpected addresses
DLL injection Write DLL path to target; LoadLibrary via remote thread dlllist shows unexpected DLL paths
Reflective DLL injection DLL loads itself without touching disk No disk artefact; malfind may catch PE in RWX region
APC injection Queue an APC to run shellcode in target thread Harder to detect; check thread start addresses
Process doppelgänging Uses NTFS transactions to execute without creating real file pstree anomalies; transaction log forensics

malfind Explained

malfind looks for VAD (Virtual Address Descriptor) entries that are:

  • Marked PAGE_EXECUTE_READWRITE (RWX) — executable and writable, unusual for legitimate code
  • Contain MZ (PE header) at the start — suggests injected executable

Example output:

Process: svchost.exe PID: 1234
VAD: 0x00400000 - 0x00410000  RWX PRIVATE
MZ header found at 0x00400000
Dump: injected_1234.dmp

What Stays in Memory That Never Hits Disk

Artefact Why It Is In Memory Only
Decrypted ransomware keys Key is in RAM while encryption is running
Browser session tokens In-memory HTTP state before written to cookie store
Cleartext credentials Lsass process holds credentials for SSO (Mimikatz target)
PowerShell IEX payloads Invoke-Expression downloads and executes without file write
Reflective DLLs Never touch disk; load directly into memory
Cobalt Strike beacon Shellcode runs entirely in injected memory region
Registry keys loaded but not flushed Pending registry writes

Mimikatz and LSASS

Mimikatz is a credential extraction tool that reads NTLM hashes, Kerberos tickets, and plaintext passwords from the LSASS (Local Security Authority Subsystem Service) process memory.

Why it works: Windows caches credentials in LSASS for SSO. Before Windows 8.1 / Server 2012 R2, WDigest authentication stored cleartext passwords in LSASS memory.

Mitigations:

  • Enable Credential Guard (virtualises LSASS in a separate trust domain)
  • Disable WDigest (HKLM\SYSTEM\CurrentControlSet\Control\SecurityProviders\WDigest\UseLogonCredential = 0)
  • Enable Protected Process Light (PPL) for LSASS
  • Run Sysmon rule: alert on any process accessing lsass.exe memory

Detection in memory dump: Dump LSASS (procdump -p <lsass_pid>) and analyse offline with Mimikatz — avoids triggering EDR on the live system.

GRR and Remote Memory Forensics

Google Rapid Response (GRR) is an agent-based remote forensics and incident response framework developed at Google. It allows:

  • Live memory acquisition from remote hosts without physical access
  • Remote file collection and timeline analysis
  • Fleet-wide IOC hunting (scan thousands of hosts simultaneously)
  • Hunt for specific file hashes, registry keys, running processes

Particularly valuable at scale: instead of shipping 1,000 memory dumps, GRR runs Volatility plugins remotely and returns only the results.
