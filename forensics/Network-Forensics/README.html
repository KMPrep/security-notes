Network Forensics

Category: Digital Forensics — Traffic Analysis, Protocol Reconstruction

Overview

Network forensics analyses captured network traffic and metadata to reconstruct attacker activity, identify data exfiltration, trace command-and-control communications, and establish timelines. Unlike disk or memory forensics, network evidence is highly transient — traffic that is not captured at the time of the attack is lost forever unless logging infrastructure was in place.

The two modes of network forensics are:

  • Full packet capture (PCAP): Records every packet in its entirety. Complete but storage-intensive.
  • Flow data (NetFlow/IPFIX): Records metadata (src/dst IP, port, bytes, duration) without payload. Lightweight; suitable for long-term retention at scale.

NetFlow and IPFIX

NetFlow records one record per conversation (flow), not per packet. Dramatically reduces storage.

Flow record fields:

  • Source IP, destination IP
  • Source port, destination port
  • Protocol (TCP/UDP/ICMP)
  • Bytes transferred, packet count
  • Flow start time, end time
  • TCP flags

What NetFlow can detect (without payload):

Pattern Detection Logic
Port scanning Many connections to same dst IP, different dst ports, within short window
Data exfiltration Large outbound byte count to unfamiliar external IP
C2 beaconing Regular small connections to same external IP every N seconds
Lateral movement Many SMB/RDP/SSH connections from one internal host to many others
DNS tunnelling High-frequency DNS flows with unusual byte counts
DDoS Massive inbound connection count to one dst IP

Sampling rate caveat: Large networks use sampled NetFlow (e.g., 1-in-1000 packets) to reduce overhead. At scale this means low-volume attacks may be missed entirely in flow data.

DNS Forensics

DNS logs are among the most valuable network forensic sources because:

  • Almost all malware uses DNS for C2 lookup at some point
  • DNS is rarely blocked internally
  • DNS logs show domain resolution history even if the connection payload is encrypted

Passive DNS

Passive DNS is a database built by storing all observed DNS queries and responses over time. It allows you to:

  • Look up what IP a domain resolved to at a specific time
  • Find all domains that have resolved to a given IP
  • Identify domain clusters registered by the same attacker
  • Reconstruct historical C2 infrastructure

Tools: SecurityTrails, PassiveTotal (RiskIQ), Farsight DNSDB, Zeek dns.log

DGA Detection

Domain Generation Algorithms (DGAs) produce pseudo-random domain names for C2. The malware generates the same list as the attacker (who pre-registers one domain from the list). Detection:

Method How
Entropy scoring DGA domains have high character entropy; legitimate domains are lower
N-gram analysis DGA domains fail dictionary-based n-gram analysis
Query volume DGA produces many NXDOMAIN responses as most generated domains aren't registered
ML classifier Train on known DGA families; classify new queries in real time

Zeek dns.log example fields: ts, uid, id.orig_h, query, qtype_name, answers, rcode_name

DNS Exfiltration

Data encoded in subdomain labels of DNS queries:

3a8f92bc11e04d5a.attacker.com
[encoded data].[C2 domain]

Detection signals:

  • Subdomain length > 30 characters
  • High query rate to same second-level domain
  • Low TTL responses from C2 domain
  • NXDOMAIN responses with unusually long query names

HTTP/HTTPS Forensics

HTTP Artefacts

From a full PCAP or web proxy log, extract:

  • URLs requested (GET/POST paths)
  • HTTP headers (User-Agent, Referer, Cookie, Authorization)
  • POST body data (may contain exfiltrated data)
  • Response content and content type

Suspicious signals:

  • Unusual User-Agent strings (tool names, random strings, outdated browser versions)
  • POST to unusual paths with binary or base64-encoded body
  • HTTP response with executable content type (application/octet-stream)
  • Large outbound POST requests (data staging for exfil)
  • Consistent request intervals (C2 beacon)

TLS / Encrypted Traffic Analysis

When traffic is TLS-encrypted, payload analysis requires TLS interception (proxy with re-signing certificate). Without decryption:

Technique What It Reveals
JA3 fingerprinting TLS client hello cipher/extension combination identifies tool or malware family
JA3S fingerprinting Server hello fingerprint; correlate client + server pairs
Certificate inspection Examine server certificate CN, issuer, validity period, SANs
Traffic metadata Byte counts, timing, packet sizes reveal communication patterns even without decryption
SNI (Server Name Indication) Reveals destination domain even in encrypted traffic

Certificate Red Flags

Indicator Significance
Self-signed certificate Unusual for legitimate services; common in attacker C2
Certificate issued hours before use Freshly registered domain + cert = likely malicious
CN does not match SNI Possible misconfigured C2
Let's Encrypt cert on unusual domain Easy to get; used by both legitimate sites and attackers
Weak cipher suite negotiated Old client or deliberately downgraded

Reconstructing Sessions from PCAP

Wireshark / NetworkMiner can reconstruct:

  • TCP streams (Follow > TCP Stream in Wireshark): Reassembles full conversation
  • HTTP sessions: Extracts request/response pairs, cookies, form data
  • Files transferred: Extract files from HTTP, FTP, SMB, SMTP
  • Credentials: FTP, Telnet, HTTP Basic Auth in cleartext

Tshark (CLI Wireshark) examples:

# Extract all HTTP requests
tshark -r capture.pcap -Y "http.request" -T fields -e http.host -e http.request.uri

# Extract all DNS queries
tshark -r capture.pcap -Y "dns.flags.response == 0" -T fields -e dns.qry.name

# Extract all files via SMB
tshark -r capture.pcap --export-objects smb,/output/

# Find all unique destination IPs
tshark -r capture.pcap -T fields -e ip.dst | sort -u
