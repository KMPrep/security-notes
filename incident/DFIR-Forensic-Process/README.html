DFIR — The Digital Forensics & Incident Response Process

Category: Incident Response, Digital Forensics, Evidence Handling
Standards & refs: NIST SP 800-86 (forensics into IR), NIST SP 800-61r2 (IR lifecycle), ISO/IEC 27037 (evidence identification/collection/preservation), RFC 3227 (evidence collection & archiving), SWGDE best practices
Cross-refs: forensics/README.md · forensics/Disk-Forensics · forensics/Memory-Forensics · forensics/Network-Forensics · incident/IR-Lifecycle-NIST · incident/README.md

1. Overview

Digital Forensics and Incident Response (DFIR) is the discipline of answering what happened, how, when, by whom, and what did they touch — with evidence that is technically sound and, where necessary, legally defensible. The "forensics" half is the rigorous acquisition and analysis of digital artifacts; the "IR" half is the operational loop of containing and evicting an adversary. They run in parallel: forensics feeds the incident commander the facts that drive containment decisions, while IR constrains how much time forensics gets before a host is reimaged. At scale the discipline is inverted from the "one seized laptop" model — you rarely image 4 TB over a write-blocker; you snapshot an EBS volume via API, pull EDR triage packages from thousands of hosts, and query centralized logs that the endpoint itself can never tamper with.

The non-negotiable core is evidence integrity. Every artifact is hashed at the moment of acquisition, worked on only as a verified copy, and tracked through an unbroken chain of custody. Get this wrong and your beautiful timeline is inadmissible and, worse, untrustworthy for your own root-cause analysis. This note covers the end-to-end process: order of volatility, acquisition (disk imaging, memory capture, cloud snapshots), integrity/hashing, the analysis workflow, the artifact sources that matter on Linux and Windows, memory forensics with Volatility, super-timelining with plaso, and how all of it bends in the cloud.

TL;DR
- Order of volatility drives collection order: CPU/RAM/network state before disk before backups — RFC 3227. Capture memory before you pull the plug.
- Never touch the original. Image it, hash it (SHA-256), verify the copy matches, analyze the copy. Work read-only via write blocker or ro loopback.
- Chain of custody is a document, not a vibe: who/what/when/where for every transfer, with hashes at each hop. No CoC → no admissibility.
- Windows artifacts to name cold: Event Logs (EVTX), Prefetch, Amcache/Shimcache, $MFT, $UsnJrnl, registry hives (SYSTEM/SOFTWARE/NTUSER). Linux: auditd, journald, /var/log/{auth,secure}, bash history, /proc.
- Cloud forensics ≠ disk forensics: you snapshot volumes and pull API/control-plane logs (CloudTrail); compute is ephemeral so acquire before terminate and treat logs as the primary evidence.

2. How It Works (go deep)

2.1 Core mechanics

DFIR is a pipeline with strict ordering constraints. Each stage has a control-plane concern (was this legally/procedurally sound?) and a data-plane concern (did I capture the bits without altering them?).

Concept What it is Why it matters
Order of Volatility Rank evidence by how fast it disappears (RFC 3227) Dictates what you collect first; RAM is gone at shutdown
Forensic acquisition Bit-for-bit copy of a source (disk, RAM, volume) Analysis must run on a copy, never the original
Write blocker Hardware/software that permits reads but drops writes to evidence Prevents the act of imaging from modifying the source
Hashing / integrity Cryptographic digest (MD5+SHA-256) computed at acquisition Proves the copy == original and detects later tampering
Chain of custody (CoC) Auditable log of every handler, action, transfer, timestamp Admissibility + your own trust in the evidence
Image format Container for the acquired bits (raw dd, E01, AFF4) E01 embeds hashes/metadata/compression; raw is dumbest & most portable
Triage vs full acquisition Fast collection of high-value artifacts vs full disk Scale forces triage first (KAPE/Velociraptor), full image if needed
Timeline / super-timeline Unified time-ordered view of all artifacts (plaso) Turns thousands of scattered artifacts into a narrative
Anti-forensics Timestomping, log clearing, LOLBins, in-memory-only malware Why you corroborate across independent artifact sources
Provenance / weight How much a court/analyst trusts an artifact Central logs > endpoint-local logs the attacker could edit

The single most important operating principle: corroborate across independent sources. An attacker can timestomp a binary's $STANDARD_INFORMATION timestamps, but the $FILE_NAME timestamps in the MFT and the $UsnJrnl change journal tell a different story; Prefetch and Amcache independently prove the binary ran. No single artifact is trusted alone.

flowchart TD
    A[Incident declared] --> B[Identify evidence sources]
    B --> C[Collect by order of volatility]
    C --> D[Acquire memory then disk then logs]
    D --> E[Hash at acquisition]
    E --> F[Verify copy matches original]
    F --> G[Analyze copy only]
    G --> H[Build timeline and correlate]
    H --> I[Report findings feed IR decisions]
    I --> J[Preserve evidence for legal]

2.2 Concrete configuration

Everything below is copy-pasteable. Realistic device names, real flags.

Order-of-volatility collection (RFC 3227) — the live-response first moves. On a running Linux host you suspect is compromised, collect volatile state before you image disk, and stream it off-box so you never write to the evidence:

# Send everything to a collection host over netcat; nothing lands on the suspect disk.
# Timestamp + hostname so the artifacts are self-describing.
date -u +"%Y-%m-%dT%H:%M:%SZ"; hostname
cat /proc/version /etc/os-release        # kernel + distro for symbol/profile selection
ss -tupan                                # sockets w/ PID (replaces netstat)
ps auxwwe                                # full process list w/ env
ls -la /proc/*/exe 2>/dev/null           # exe paths incl. deleted-but-running binaries
lsof -nP                                 # open files/sockets, deleted files still mapped
cat /proc/mounts; dmesg --time-format iso
for p in /proc/[0-9]*; do echo "== $p =="; cat $p/cmdline | tr '\0' ' '; echo; done

Memory acquisition (Linux) with LiME, streamed off-host so the capture never touches the evidence disk:

# Build/insert the LiME kernel module; format=lime keeps metadata; write over TCP to collector
# On the SUSPECT host:
insmod ./lime.ko "path=tcp:4444 format=lime"
# On the COLLECTOR host, capture the stream and hash it immediately:
nc -l -p 4444 > web01-mem-2026-07-07.lime
sha256sum web01-mem-2026-07-07.lime | tee web01-mem-2026-07-07.lime.sha256

Memory acquisition (Windows) with WinPmem / Magnet RAM Capture — produce an AFF4 that self-verifies:

# WinPmem writes an AFF4 container with embedded metadata; -o output, default is full physical memory
.\winpmem_mini_x64_rc2.exe DC01-mem-2026-07-07.aff4
Get-FileHash -Algorithm SHA256 DC01-mem-2026-07-07.aff4 | Format-List

Disk imaging with a write blocker. Physical seizure path: source drive → hardware write blocker (Tableau/WiebeTech) → analyst workstation. Then image to E01 with ewfacquire (libewf), which embeds case metadata and computes hashes as it reads:

# Confirm the source is mounted read-only / behind a blocker; identify by serial not just /dev node.
lsblk -o NAME,SIZE,SERIAL,RO /dev/sdb        # RO column must be 1 if using SW blocker
blockdev --setro /dev/sdb                    # software fallback if no HW blocker

# Image to Expert Witness Format (E01): compressed, chunk-hashed, case metadata embedded.
ewfacquire -t /evidence/laptop-hr-2026-07-07 \
  -C CASE-2026-0043 -E "unauthorized access" -e "J. Analyst" \
  -d sha256 -c fast /dev/sdb
# ewfverify re-reads and confirms stored vs recomputed hash.
ewfverify /evidence/laptop-hr-2026-07-07.E01

Raw dd/dcfldd alternative when you need a plain image (note dcfldd hashes inline, plain dd does not):

# dcfldd = dd + inline hashing + progress; hashwindow=0 hashes the whole stream.
dcfldd if=/dev/sdb of=/evidence/laptop-hr.dd hash=sha256 \
  hashlog=/evidence/laptop-hr.sha256 bs=4M conv=noerror,sync status=on

Integrity verification before analysis — mount the copy read-only so analysis cannot alter it:

# Verify the working copy equals the acquisition hash BEFORE touching it.
sha256sum -c /evidence/laptop-hr.sha256
# Mount read-only, no atime updates, no execution of device files.
mount -o ro,noatime,noexec,nodev,loop,offset=$((2048*512)) laptop-hr.dd /mnt/evidence

Triage collection at scale (Windows) with KAPE — grab the artifacts that matter in minutes instead of imaging 1 TB:

kape.exe --tsource C: --tdest E:\triage\WKS-4471 --target KapeTriage \
  --mdest E:\triage\WKS-4471\modules --module EvtxECmd,PrefetchParser,RECmd_Batch \
  --gui
# KapeTriage pulls $MFT, $J, EVTX, registry hives, Prefetch, Amcache, SRUM, browser history, etc.

Cloud volume snapshot (AWS) — the disk-image equivalent. Snapshot the attached EBS volume, then attach a copy to an isolated forensic instance:

# 1. Snapshot the compromised instance's root volume (point-in-time, immutable).
aws ec2 create-snapshot --volume-id vol-0a1b2c3d --description "IR-2026-0043 web01 root" \
  --tag-specifications 'ResourceType=snapshot,Tags=[{Key=incident,Value=IR-2026-0043}]'
# 2. Create a volume from the snapshot in the forensic AZ and attach to the analysis box.
aws ec2 create-volume --snapshot-id snap-0f9e8d7c --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=volume,Tags=[{Key=incident,Value=IR-2026-0043}]'
aws ec2 attach-volume --volume-id vol-0NEW --instance-id i-0FORENSIC --device /dev/sdf
# 3. Pull the authoritative control-plane evidence: CloudTrail is the API "who did what".
aws cloudtrail lookup-events --lookup-attributes AttributeKey=ResourceName,AttributeValue=i-0web01 \
  --start-time 2026-07-01T00:00:00Z --end-time 2026-07-07T00:00:00Z

Case study — Capital One (2019), cloud-native forensics. An attacker exploited an SSRF flaw in a WAF, reached the EC2 Instance Metadata Service (IMDSv1), retrieved the instance role's temporary credentials, and used them to list and exfiltrate ~100M records from S3. There was almost nothing to image on a traditional "disk" — the decisive evidence lived in CloudTrail: AssumeRole/GetCallerIdentity from the WAF role, followed by ListBuckets and thousands of GetObject calls from an unusual source IP and user agent. The controls that would have stopped it: enforcing IMDSv2 (session-token-bound, blocks SSRF credential theft), scoping the role's S3 permissions, and GuardDuty anomaly alerting on credential-exfil patterns. Forensically, the lesson is that in cloud the control plane log is the crime scene, and it must be delivered to an account the workload identity cannot alter.

Case study — NotPetya (2017), anti-forensics. NotPetya encrypted the MFT and overwrote the MBR, then rebooted — deliberately destroying the very NTFS structures a disk examiner relies on. Endpoint disk forensics was largely a dead end. What survived and told the story: memory captures taken from still-running hosts (showing the perfc.dat DLL, rundll32 execution, and the Mimikatz-derived credential theft feeding PsExec/WMI lateral movement), plus network captures and central logs showing SMB (EternalBlue, MS17-010) spread. The lesson that shaped modern DFIR: capture volatile memory early and ship logs off-box in real time, because a competent adversary will destroy on-disk evidence.

Case study — SolarWinds SUNBURST (2020), timeline across systems. The malicious code was injected into the SolarWinds Orion build pipeline, so the trusted, signed DLL on every victim endpoint looked legitimate. Endpoint artifacts alone (a signed DLL, benign-looking process) did not scream "compromise." The break came from timeline correlation: DNS logs showing the DGA-generated avsvmcloud.com beacons, tied to the exact Orion process, tied to the build artifact's provenance. This is the archetypal case for super-timelining (plaso/Timesketch) and for treating build systems as first-class forensic sources.

Top 5 in depth

1. Memory-first live response (mitigates RAM loss, in-memory malware, FDE). Because RAM sits at the top of the order of volatility, the single highest-leverage move is capturing it before anything else — before you run diagnostic commands, before containment reboots, and before an SSD's controller starts zeroing TRIMmed blocks. Technically you insert a minimal collector (LiME on Linux via a kernel module, WinPmem/Magnet on Windows) and stream the image off-host (format=lime path=tcp:4444) so nothing lands on the evidence disk. Memory is also your only shot against full-disk-encrypted, powered-on hosts: the DEK lives in RAM, and tools like bulk_extractor or Volatility can recover BitLocker/LUKS keys from the capture. Trade-off/limit: any live tool is an observer — inserting a kernel module perturbs memory slightly, and on a rootkit-controlled host the acquisition path itself could be subverted. You document the tool, version, and exact time, and accept that "forensically pure" is aspirational for volatile capture.

2. Write blocker + verified copy + hashing (mitigates alteration, admissibility). The bedrock. A hardware write blocker sits inline on the SATA/USB/NVMe path and physically drops write commands while passing reads, so the mere act of imaging cannot modify the source; the software fallback is blockdev --setro /dev/sdb, but hardware is preferred because it's outside the OS's control. You image to E01 with ewfacquire, which hashes each chunk and the whole stream, then ewfverify re-reads and confirms. You compute SHA-256 (MD5 is retained only for cross-tool compatibility and historical CoC, never for integrity trust given collision attacks). All analysis runs against a copy mounted ro,noatime,noexec,nodev. Trade-off/limit: write blockers don't exist for cloud volumes — there the equivalent is an immutable snapshot — and imaging large modern disks over a blocker is slow, pushing teams toward triage collection for the first pass.

3. Off-host, append-only central logging (mitigates log clearing, retention gaps, editable logs). The most reliable defense against anti-forensics is to make the evidence live somewhere the attacker who owns the host cannot reach. Ship auditd/journald/EVTX/Sysmon to a SIEM (Splunk, Chronicle, Elastic) and CloudTrail to a locked, versioned S3 bucket in a separate logging account with an Object Lock (WORM) policy. Now even a wevtutil cl Security or journalctl --rotate --vacuum-time=1s on the endpoint only erases the local copy; the authoritative record already left the building. This is what let investigators reconstruct NotPetya and Capital One despite on-host destruction. Trade-off/limit: cost and volume — full-fidelity logging at fleet scale is expensive, so teams tier retention (hot 30–90 days, cold 1+ year) and must resist the temptation to shorten security-log retention below expected dwell times.

4. Super-timeline correlation across independent sources (mitigates timestomping, log clearing, anti-forensics). No single artifact is trusted; the method is triangulation. log2timeline/plaso ingests dozens of artifact parsers (MFT, EVTX, registry, Prefetch, browser, syslog) into one storage file, and psort renders a time-ordered super-timeline you load into Timesketch to pivot on a window. Timestomping defeats a naive "sort by mtime" but not this: the attacker changes $STANDARD_INFORMATION times, yet $FILE_NAME MFT times, $UsnJrnl sequence, Prefetch last-run, and Amcache first-seen all disagree with the forged value, exposing the manipulation. Trade-off/limit: super-timelines are enormous and noisy (millions of rows), require UTC normalization to survive clock skew, and demand a strong hypothesis to be usable — you filter to a time window and pivot, you don't read it top to bottom.

5. Cloud IR automation: snapshot-and-isolate before terminate (mitigates ephemerality). Cloud compute is disposable — auto-scaling or an attacker can TerminateInstances and vaporize both disk and RAM. The control is automation that, on an IR tag or GuardDuty finding, (a) enables termination protection, (b) create-snapshot on every attached volume, (c) captures memory via SSM if the agent is alive, (d) moves the instance to a quarantine security group that only permits the forensic subnet, and (e) preserves the temporary role credentials for revocation. The snapshot is your immutable "disk image," attached to an isolated forensic instance for analysis. Trade-off/limit: RAM capture depends on a live agent (you can't hardware-image a hypervisor guest you don't own), and snapshotting doesn't help if the workload is a short-lived Lambda/Fargate task — for those, the logs are the only evidence, which loops back to control #3.

6. Detection & Response

What to log / telemetry sources

Forensics is only as good as the telemetry you preserved before the incident. Ensure these are collected and centralized:

  • Linux: auditd (execve, connect, path watches on /etc/passwd, /etc/shadow, sudoers), systemd-journald (with Storage=persistent), /var/log/auth.log or /var/log/secure (SSH/sudo), /var/log/syslog, shell history (~/.bash_history, HISTTIMEFORMAT for timestamps), /proc/<pid>/{exe,maps,cmdline} live, wtmp/btmp/lastlog (logins), package manager logs (dpkg.log, yum.log).
  • Windows: Security EVTX (4624/4625 logon, 4688 process creation with command line, 4672 special privileges, 4720/4726 account changes, 4698 scheduled task, 1102 log cleared), System/Application EVTX, Sysmon (1 process, 3 network, 7 image load, 8 CreateRemoteThread, 11 file create, 13 registry), Prefetch (C:\Windows\Prefetch\*.pf), Amcache (Amcache.hve) & Shimcache (SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache), $MFT, $UsnJrnl:$J, SRUM (SRUDB.dat), registry run keys / services / ShellBags / UserAssist, PowerShell Operational (4104 script block).
  • Cloud: CloudTrail (management + data events), VPC Flow Logs, GuardDuty/Detective, S3 access logs, IAM Access Analyzer, Config snapshots (resource state over time), GCP Cloud Audit Logs / Azure Activity + Sign-in logs.
  • Network: NetFlow/IPFIX, DNS query logs, full PCAP at chokepoints, Zeek connection logs, proxy/egress logs.

KQL (Microsoft Sentinel / Defender) — suspicious credential-dumping precursor via LSASS access (T1003.001):

DeviceEvents
| where Timestamp > ago(7d)
| where ActionType == "OpenProcessApiCall"
| where FileName =~ "lsass.exe" or InitiatingProcessCommandLine has "lsass"
| where InitiatingProcessFileName !in~ ("MsMpEng.exe","taskmgr.exe","procexp64.exe")
| where InitiatingProcessTokenElevation == "TokenElevationTypeFull"

SQL over CloudTrail (Athena) — mass S3 object reads from a single principal, the Capital One pattern:

SELECT useridentity.arn AS principal,
       sourceipaddress,
       COUNT(*) AS get_count,
       MIN(eventtime) AS first_seen,
       MAX(eventtime) AS last_seen
FROM cloudtrail_logs
WHERE eventsource = 's3.amazonaws.com'
  AND eventname   = 'GetObject'
  AND eventtime BETWEEN '2026-07-01T00:00:00Z' AND '2026-07-07T00:00:00Z'
GROUP BY useridentity.arn, sourceipaddress
HAVING COUNT(*) > 10000
ORDER BY get_count DESC;

Response playbook (forensic-aware containment → eradication → recovery)

  1. Declare & assign roles. IC, forensics lead, scribe. Open the incident doc; start the CoC log. Classify severity (see incident/README.md).
  2. Freeze the scene without destroying it. For a running host: capture memory first (LiME/WinPmem), then volatile state (ss, ps, lsof), streaming off-host. Do not reboot or run the host's own untrusted binaries.
  3. Contain without losing evidence. Network-isolate (quarantine VLAN / cloud security group), revoke credentials and rotate keys, but snapshot/image before terminating any cloud instance. Enable termination protection.
  4. Acquire persistent evidence. Write-blocked disk image (or EBS snapshot → forensic volume). Hash at acquisition; verify the copy. Pull central logs + CloudTrail for the full dwell window.
  5. Analyze on copies. Build a super-timeline (plaso → Timesketch); run Volatility on the memory image (pslist/psscan diff, malfind, netscan); establish initial access vector, persistence, lateral movement, and exfil scope.
  6. Scope the blast radius. Sweep the fleet for the same IOCs (hashes, C2 IPs, registry keys, scheduled tasks) via EDR/KAPE. Do not eradicate one foothold until you've mapped all of them.
  7. Eradicate. Reimage from known-good baseline (never trust AV-cleaned hosts), remove persistence, patch the exploited vulnerability everywhere, rotate all potentially exposed credentials.
  8. Recover & verify. Restore from clean backups (confirm backups aren't also compromised), elevate monitoring for 30+ days, watch for re-entry.
  9. Preserve for legal & post-mortem. Seal evidence with intact CoC; write the blameless post-incident review with a timeline, root cause, and detection-gap analysis (see incident/IR-Lifecycle-NIST).

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is the order of volatility and why does it dictate collection order?
The order of volatility (RFC 3227) ranks evidence by how quickly it's lost: CPU registers/cache → RAM and running process state → network connections and ARP/routing tables → running processes/open files → disk → remote logs → archived backups. You collect most-volatile first because RAM and network state vanish at shutdown or timeout, whereas disk persists. Practically that means capturing memory before you pull the plug or run intrusive commands, then volatile OS state, then imaging disk. Violating the order — booting or rebooting the host first — permanently destroys the highest-value volatile evidence.
Signal: do they instinctively protect volatile evidence, or jump straight to disk imaging?

Q2. Why do you hash evidence, and which algorithm(s) do you use?
Hashing produces a cryptographic fingerprint at acquisition that proves the working copy is bit-identical to the original and detects any later tampering. You compute the hash immediately after acquisition, record it in the chain of custody, and re-verify before every analysis session. Use SHA-256 for integrity trust; MD5 is kept only for legacy tool compatibility and cross-checking, never as the sole integrity proof, because MD5 collisions are trivially producible. If sha256sum -c fails, the evidence is compromised and you stop.
Signal: knows MD5 is deprecated for trust and why; treats hashing as mandatory, not optional.

Q3. What is chain of custody and what must it record?
Chain of custody is the auditable, chronological record of everyone who handled a piece of evidence and every action taken on it. It records who collected it (when, where, with what tool), the acquisition hash, every transfer of custody (name, date/time, reason), and where/how it's stored under access control. Its purpose is twofold: legal admissibility (a break can get evidence excluded) and your own confidence that the artifact wasn't altered. It applies even to internal investigations, not just those headed to court.
Signal: understands CoC is a process discipline, not paperwork theater.

Q4. Why must you never analyze the original evidence, and how do you enforce that?
Analysis operations — mounting, indexing, even the OS updating access times — modify data, and any modification of the original is irreversible and taints the evidence. You enforce it by imaging behind a write blocker, hashing, and working only on a verified copy mounted read-only (mount -o ro,noatime,noexec). The original is sealed and stored; the copy is disposable and re-derivable from the sealed original if needed. Hardware write blockers are preferred over software because they're outside the OS's control.
Signal: knows the write-blocker + read-only-copy discipline cold.

Q5. Name the key Windows artifacts that prove a program executed, even if the file is deleted.
Execution-evidence artifacts include Prefetch (.pf files with run count and last-8 run timestamps), Amcache.hve (first-seen time, SHA-1 of the binary), Shimcache/AppCompatCache (path + presence, from the registry), SRUM (network/resource usage per app), and Security EVTX 4688 (process creation with command line if audited). Because several of these persist after the executable is deleted, they let you prove execution and recover the original path/hash. You corroborate across them since each has different fidelity and anti-forensic resistance.
Signal: names multiple independent execution artifacts, not just "the event log."

Tier 2 — System design / trade-offs

Q6. Design an evidence-preservation pipeline for a 50,000-host fleet where you can't image every disk.
I'd invert the "seize the laptop" model toward telemetry-first forensics. Continuously ship Sysmon/auditd/EVTX and cloud audit logs to a central SIEM and to an immutable, WORM-locked archive in a separate logging account, with tiered retention (hot 90 days, cold 12+ months) sized to expected dwell time. For endpoints, deploy an EDR/Velociraptor agent so I can pull targeted triage packages (KAPE-style: $MFT, $J, EVTX, registry, Prefetch, Amcache) in minutes rather than imaging TBs. Full disk/memory imaging is reserved for the handful of hosts where deep analysis is warranted, triggered by an IR tag. In cloud, automation snapshots volumes and captures memory via SSM on tag, with termination protection. The design principle is that at scale the central, tamper-resistant log is the primary evidence and full imaging is the exception, driven by scoping from the timeline.
Signal: scales beyond single-host forensics; separates telemetry from targeted deep-dive.

Q7. How does forensic acquisition change between an on-prem physical server and an ephemeral cloud workload?
On-prem you have a physical disk you can put behind a write blocker and a running machine whose RAM you can capture with a local agent — classic disk+memory imaging with hardware integrity guarantees. In cloud, there's no physical disk to blocker: the equivalent of imaging is an immutable EBS/PD snapshot taken via API, which you turn into a volume attached to an isolated forensic instance. Memory capture requires a live agent (SSM avml) because you don't own the hypervisor. Crucially, compute is ephemeral — auto-scaling or TerminateInstances can destroy disk and RAM in seconds — so the control-plane logs (CloudTrail/Cloud Audit Logs) become the authoritative, and sometimes only, evidence. For serverless (Lambda/Fargate) there's nothing to snapshot at all, so logs are everything. The order of volatility still holds, but "acquire before terminate" and log immutability dominate.
Signal: understands ephemerality, snapshot-as-image, and control-plane-log-as-crime-scene.

Q8. An attacker cleared the Windows Security log and timestomped their tools. How do you still build a reliable timeline?
I lean on the principle that no artifact stands alone. Log clearing itself leaves an event (1102/104) and, more importantly, the central copy already shipped to the SIEM is untouched — attacker action on the endpoint doesn't reach it. For timestomping, $STANDARD_INFORMATION times are attacker-writable but $FILE_NAME times in the MFT, the $UsnJrnl:$J change-journal sequence, and Prefetch/Amcache first-seen/last-run are far harder to forge consistently; disagreement between them exposes the manipulation and often reveals the real time. I'd build a plaso super-timeline that fuses MFT, $J, EVTX, registry, Prefetch, SRUM, and browser artifacts, normalize to UTC, load it into Timesketch, and pivot on the suspected window. Memory (if captured) and network/DNS logs provide independent corroboration the endpoint attacker never controlled.
Signal: knows anti-forensic countermeasures and multi-source triangulation, not just "read the event log."

Q9. Walk through choosing between full disk imaging, triage collection, and live-response only. What are the trade-offs?
Full imaging gives you everything (unallocated space, deleted files, slack) but is slow, storage-heavy, and often impossible at scale or in cloud — best for a small number of high-value or legally significant hosts. Triage collection (KAPE/Velociraptor) grabs the high-signal artifacts in minutes and scales to many hosts, at the cost of missing deleted/unallocated data — my default first pass for scoping. Live-response-only (memory + volatile state) is mandatory when the host can't be taken down, when data is FDE-encrypted and only decryptable in RAM, or for fileless malware, but it's the least "pure" because the collection tools perturb the system. In practice I chain them: live memory capture first, triage for breadth, then full imaging only where the timeline shows I need deleted-data recovery or court-grade completeness.
Signal: reasons about speed/completeness/scale trade-offs rather than dogmatically imaging everything.

Q10. Where should security logs live so an attacker who fully owns a host cannot destroy your evidence?
Off the host, in an append-only sink the host's identity cannot modify. Concretely: stream endpoint telemetry to a central SIEM in real time, and for cloud send CloudTrail to a versioned S3 bucket with Object Lock (WORM) in a separate logging account whose access the workload role has no path to. The threat model is that root on the box can wevtutil cl, truncate journald, or rm local logs — but those actions can't reach data that already left the machine. This is exactly what preserved the evidence in NotPetya (wiped endpoints) and Capital One (API-driven S3 exfil). Retention must exceed realistic dwell time, so I'd tier hot/cold storage and monitor for gaps or log-source silence, which is itself a detection.
Signal: threat-models the log pipeline against a root-level adversary; knows WORM/separate-account patterns.

Tier 3 — Expert / deep-dive

Q11. Explain how you'd use Volatility to confirm process injection in a memory image, plugin by plugin.
I start by establishing ground truth on the process list: windows.pslist walks the linked EPROCESS list (what the OS advertises) while windows.psscan carves _EPROCESS pool tags directly from memory — diffing them exposes processes unlinked to hide (DKOM/rootkit). For injection specifically, windows.malfind flags memory regions that are private, committed, and RWX with PAGE_EXECUTE_READWRITE and MZ headers or shellcode with no backing file — the signature of reflective DLL/shellcode injection. I confirm scope with windows.dlllist and windows.handles (unexpected handles to LSASS suggest credential access), tie the process to its C2 with windows.netscan (sockets → PID → remote IP/port), then windows.dumpfiles/windows.memmap --dump to extract the injected region and run yarascan with C2/family signatures. On Vol3 I rely on automatic symbol resolution from the kernel's PDB rather than hand-picking a profile as in Vol2. The chain — pslist/psscan diff → malfind → handles → netscan → dump+yara — turns "something's off" into an evidenced injection with attribution.
Signal: fluent, specific Volatility workflow with the reasoning behind each plugin.

Q12. How do plaso/log2timeline and psort actually build a super-timeline, and what pitfalls make it lie to you?
log2timeline.py runs a battery of parsers (winreg, mft, usnjrnl, winevtx, prefetch, sqlite for browsers, syslog, etc.) against a source (image or triage dir), extracting every timestamped event into a Plaso storage file with each event tagged by source, type, and parser. psort.py then sorts, deduplicates, filters (by time window, regex, tags), and outputs to CSV, Elasticsearch, or Timesketch for interactive pivoting. Pitfalls: (1) timezone/clock skew — if you don't normalize everything to UTC and account for host clock drift, cross-source ordering is wrong; you pass -z and verify NTP. (2) Timestamp semantics — a single file has MACB times from both $STANDARD_INFORMATION and $FILE_NAME, and confusing "modified" with "MFT-changed" misleads you; the super-timeline shows all of them, which is the point. (3) Volume and noise — millions of rows mean you must start from a hypothesis and a bounded window, not read top-down. (4) Parser gaps — anything without a parser (proprietary app logs) is invisible unless you add one. The strength is fusing timestomp-resistant sources so forged $SI times stick out against $FN/$J/Prefetch.
Signal: understands the tool internals and the analytic failure modes, not just the command.

Q13. A powered-off laptop uses BitLocker full-disk encryption. Walk through your acquisition options.
A dead-box image of a BitLocker volume yields ciphertext, so I need a key or a live capture. Options in order of preference: (1) If the machine is still running when I arrive, do live acquisition — capture memory (the FVEK/DEK resides in RAM and tools like bulk_extractor/Volatility's bitlocker plugins can recover it) and image the mounted, decrypted volume live. (2) Obtain the BitLocker recovery key — from the user, from Active Directory/Azure AD escrow (msFVE-RecoveryInformation), or an MDM/Intune escrow — then image the raw encrypted volume and decrypt offline with dislocker/libbde or unlock in the analysis environment. (3) A TPM-only protector with no PIN means the disk auto-unlocks at boot to that TPM, which occasionally enables a controlled boot capture, but that's risky and modifies state. If none are available, the ciphertext image is preserved anyway in case a key surfaces later. The general lesson: FDE shifts acquisition strategy toward memory-first and key escrow, and it's a strong argument for capturing RAM before ever powering down a live host.
Signal: knows FDE forces live capture / key escrow and can name the concrete recovery paths.

Q14. How do SSDs (TRIM, wear leveling, controller GC) undermine classic disk forensics, and what changes?
Traditional recovery of deleted files relies on the OS only marking blocks free while the data lingers until overwritten. On SSDs, when a file is deleted the OS issues a TRIM command telling the controller those LBAs are free, and the controller's garbage collector may then zero or discard those NAND pages autonomously and asynchronously — often within seconds to minutes — so the data is gone even though nothing "wrote over" it from the OS's view. Wear leveling also means the controller remaps LBAs to different physical cells, so the logical block you image may not reflect what was physically there, and over-provisioned spare area is invisible to normal imaging. Practical consequences: deleted-file recovery is unreliable-to-impossible, slack/unallocated analysis is weakened, and two images of the same "idle" SSD can differ. What changes: image as early as possible (every minute of power-on lets GC run), prefer live capture over dead-box when data may still be in RAM/cache, document that TRIM was active, and lean harder on artifacts that don't depend on unallocated recovery (MFT, $J, logs, memory). Encryption-at-rest on SSDs (SED/OPAL) further means a "secure erase" is instantaneous by rotating the media key.
Signal: deep media-level understanding; adjusts methodology rather than assuming HDD recovery.

Q15. Design cloud-native forensics for a Kubernetes workload where a pod was compromised, spanning acquisition, integrity, and scope.
Containers are ephemeral and share the node kernel, so my strategy blends node-level and control-plane forensics. Acquire: immediately kubectl cordon the node and kubectl label/taint to prevent rescheduling, then avoid kubectl delete pod (which destroys evidence). Capture the pod's process state and memory from the node (the container is just namespaced processes — LiME/avml on the node, scoped by cgroup/PID), snapshot the node's EBS/PD volume, and copy the container's writable layer and any mounted volumes. For the image itself, pull the exact image digest from the registry and diff against the trusted build. Integrity: hash the volume snapshot and captured layers; the container image digest is itself an integrity anchor. Control-plane evidence: the Kubernetes audit log (who called the API — exec, create, RBAC changes), the container runtime logs (containerd/CRI), the cloud audit log (IAM/IMDS abuse), and admission-controller logs are primary, since a compromised pod that stole the node's service-account token will show up there. Scope: check whether the attacker broke out to the node (privileged container, hostPath mount, kernel exploit), pivoted via the pod's mounted service-account token to the API server, or reached cloud metadata for node IAM creds; sweep other pods from the same image for the same IOCs. The through-line is that ephemerality + shared kernel mean you forensicate the node and control plane, not just "the container," and you must move before the scheduler or attacker reaps the pod.
Signal: synthesizes container internals, ephemerality, K8s audit logging, and breakout scoping into a coherent plan.

9. Key Takeaways

  • Collect by order of volatility: memory and network state before disk before backups. Capturing RAM before shutdown is the single highest-leverage forensic move — RFC 3227.
  • Integrity is the whole game: hash at acquisition (SHA-256), verify the copy equals the original, and analyze only read-only copies. No hash and no chain of custody means no trust and no admissibility.
  • Never touch the original — write blocker (or immutable snapshot in cloud), verified duplicate, mount -o ro,noatime,noexec.
  • Corroborate across independent sources. Timestomping and log clearing defeat naive single-artifact analysis but not $FN+$UsnJrnl+Prefetch+Amcache+central-log triangulation.
  • Name your artifacts cold: Windows — EVTX, Prefetch, Amcache/Shimcache, $MFT, $UsnJrnl, registry hives, SRUM, Sysmon; Linux — auditd, journald, auth.log/secure, bash history, /proc.
  • Memory forensics beats disk for fileless malware, injected code, encryption keys, and rootkits — Volatility's pslist/psscan diff → malfind → netscan → dump is the workhorse chain.
  • Timelines turn artifacts into narrative: plaso → psort → Timesketch, normalized to UTC, pivoted on a bounded window.
  • Cloud forensics is different: snapshot volumes (immutable images), acquire before terminating ephemeral compute, and treat control-plane logs (CloudTrail) as the authoritative — sometimes only — evidence.
  • Ship logs off-host to WORM storage in a separate account so a root-level attacker can't erase their own trail; retention must exceed realistic dwell time.
  • Forensics serves IR in real time — scope from the timeline before you eradicate, and never reimage before evidence is preserved (see incident/README.md).
