Site Isolation

Category: Browser Security — Process Isolation
Applies To: Google Chrome, Chromium-based browsers, multi-tenant web environments
Chrome Feature: Shipped broadly in Chrome 67 (2018); Full Site Isolation

Overview

Site Isolation is a security architecture in Chrome that places each website in its own dedicated renderer process, isolating it from renderer processes handling other sites. Before Site Isolation, different tabs or iframes from different sites could share a renderer process. If an attacker compromised the renderer process via a bug in JavaScript/HTML handling, they could read memory from other sites in the same process — including session cookies, passwords, and banking data.

Site Isolation was accelerated after Spectre (2018) revealed that JavaScript in a renderer process could read memory across the entire process's address space using CPU speculative execution and cache timing. If multiple sites shared one process, Spectre allowed cross-site data theft from JavaScript. Site Isolation ensures that evil.com cannot read data from bank.com's process because they are in separate processes with separate address spaces.

At Google, every Chrome installation for billions of users runs Site Isolation. The engineering challenge was implementing it without breaking compatibility (iframes, cross-origin communication, shared state) and without degrading performance unacceptably (each process has a cost).

How It Works

Definition of "Site" for Isolation Purposes

Site Isolation uses "site" — not "origin" — as the isolation boundary:

Concept Definition Example
Origin scheme + host + port (exact match) https://sub.example.com:443
Site scheme + eTLD+1 (registrable domain) https://example.com
eTLD+1 Effective TLD (public suffix) + one level example.com (eTLD=com, +1=example)

https://sub1.example.com and https://sub2.example.com are the same site (both example.com) but different origins. Site Isolation puts them in the same process. CORB and SameSite cookies use this boundary; COEP/COOP use origin.

What Happens to iframes?

Before Site Isolation: an evil.com page with an embedded <iframe src="bank.com"> could be processed in the same renderer process.

With Site Isolation: the iframe for bank.com is placed in its own renderer process, separate from the evil.com parent page's process.

This is called Out-of-Process iframes (OOPIF) and required a major rewrite of Chrome's rendering architecture.

How Site Isolation Defends Against Spectre

Attack Step Without Site Isolation With Site Isolation
evil.com loads in same process as bank.com Yes — different tabs or iframes could share a process No — separate processes with separate address spaces
Spectre gadget reads process memory Can read bank.com session cookies, passwords from shared memory Cannot — bank.com memory is in a separate process
Cross-origin fetch/XHR response Response body enters process memory; Spectre can read it CORB strips response body before it enters process memory

CORB — Cross-Origin Read Blocking

Even with OOPIF, a renderer process fetches resources from many origins. CORB ensures that certain sensitive cross-origin responses never enter the process memory:

CORB Protected Type Condition Action
HTML text/html response to XHR/fetch that wouldn't normally be loaded Strip body; deliver empty response
JSON application/json in cross-origin XHR Strip body
XML text/xml in cross-origin XHR Strip body
Exceptions Resources expected to be loaded cross-origin (images, scripts, CSS) Allow through normally

CORB is implemented in the Network Service process, which means the sensitive bytes never enter the renderer process even if the renderer is compromised.

COEP, COOP, CORP — The SharedArrayBuffer Family

After Spectre, Chrome also restricted access to SharedArrayBuffer and high-resolution timers (performance.now()) because they enable precise cache timing attacks. These are re-enabled only for pages that opt into full isolation:

Header Meaning Enables
Cross-Origin-Opener-Policy: same-origin No cross-origin window handles to this page Isolation from window.opener
Cross-Origin-Embedder-Policy: require-corp Only allow cross-origin resources that opt in with CORP Full process isolation
Cross-Origin-Resource-Policy: same-site\|same-origin\|cross-origin Control which origins can embed this resource Per-resource sharing opt-in

A page with both COOP and COEP gets document.crossOriginIsolated = true and can use SharedArrayBuffer.

Chrome's Multi-Process Architecture

    A[Browser Process - privileged - controls UI and navigation] --> B[Renderer processes - sandbox - one per site]
    A --> C[GPU Process - graphics rendering]
    A --> D[Network Service - fetches resources - has cookies and cache]
    A --> E[Plugin Process - legacy NPAPI]
    B --> F[Each renderer sandboxed - no file system - no network direct access]
    D --> G[CORB applied here before data reaches renderer]

Chrome's sandbox for renderer processes:
- No direct file system access
- No direct network access (must go through Network Service)
- No raw socket access
- Limited system calls (seccomp-BPF filter on Linux; AppContainer on Windows)

A Spectre attack in a compromised renderer can only read that renderer's address space — and CORB ensures sensitive cross-site data never enters it.

Mitigation Controls

# Threat Mitigation Layer
1 Same-process Spectre Site Isolation (enabled by default in Chrome) Browser architecture
2 Cross-origin data in renderer CORB (enabled by default with Site Isolation) Network Service

Key Takeaways

  • Site Isolation places each website in its own renderer process, preventing Spectre-based cross-site memory reads.
  • Spectre is the driving reason Site Isolation was accelerated — JavaScript in a renderer could use CPU cache timing to read the entire process address space.
  • CORB strips sensitive cross-origin responses (HTML, JSON, XML) before they enter a renderer process.
  • COEP + COOP headers are required for crossOriginIsolated mode, which re-enables SharedArrayBuffer and high-resolution timers.
  • Chrome's sandbox on renderer processes (seccomp-BPF, AppContainer) ensures even a compromised renderer has limited OS access.
  • Out-of-Process iframes (OOPIF) was the hardest engineering challenge in deploying Site Isolation.
