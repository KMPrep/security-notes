Cloud Service Accounts — Lateral Movement and Privilege Escalation

Category: Cloud Security — Identity and Access Management
Applies To: GCP Service Accounts, AWS IAM Roles, Azure Managed Identities, Kubernetes Service Accounts

Overview

Cloud service accounts are machine identities used by workloads, applications, and automated systems to authenticate to cloud APIs. Unlike human user accounts, service accounts are often long-lived, over-privileged, and their credentials (keys, tokens) are frequently embedded in code, config files, environment variables, or accessible from the cloud metadata service. Abusing service accounts is the primary method for lateral movement and privilege escalation in cloud environments.

At Google scale, every GCP resource (GCE VM, Cloud Run instance, Cloud Function, GKE pod) runs as a service account. Attackers who gain code execution on any of these resources immediately inherit the service account's permissions. If the service account is over-privileged — common in environments where convenience was prioritized over least-privilege — the attacker can access storage, databases, other projects, and potentially escalate to organization-level admin.

How Cloud Service Accounts Work

GCP Service Accounts

Concept Description
Service Account A Google-managed identity for non-human principals (name@project.iam.gserviceaccount.com)
Service Account Key Long-lived JSON key file that can authenticate as the SA from anywhere (dangerous)
Workload Identity Short-lived OIDC token issued via GKE/Cloud Run; federated to GCP IAM without a key file
Metadata Server VM/Container inherits SA at runtime; token available from metadata.google.internal
IAM Role Set of permissions bound to the SA for specific resources
actAs permission Allows impersonation of another SA — the key privilege escalation permission

GCP metadata server token retrieval:

curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
# Returns: {"access_token":"ya29.xxx","expires_in":3599,"token_type":"Bearer"}

This token is a short-lived OAuth2 bearer token valid for the SA bound to the resource. Any code execution on a GCE VM, GKE pod, or Cloud Run instance can retrieve it without any additional credentials.

AWS IAM Roles (Equivalent)

# AWS EC2 Instance Metadata Service (IMDSv1)
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
# Returns IAM role name; then:
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRoleName
# Returns: AccessKeyId, SecretAccessKey, Token, Expiration

IMDSv2 requires a session-oriented PUT request first, significantly reducing SSRF-based credential theft.

Critical GCP IAM Permissions for Escalation

Permission Why It's Dangerous
iam.serviceAccounts.actAs Impersonate any SA you have this permission on; inherit all that SA's permissions
iam.serviceAccounts.getAccessToken Generate an OAuth2 token for any SA you have this on
iam.serviceAccounts.implicitDelegation Use SA in delegation chains
resourcemanager.projects.setIamPolicy Grant any principal any role on the project including Owner
resourcemanager.organizations.setIamPolicy Same but at org level
cloudfunctions.functions.create + run.services.create Deploy function/service running as high-priv SA
deploymentmanager.deployments.create Deploy infrastructure including IAM bindings
iam.roles.update Update a custom role that your SA has → add permissions to that role

GCP Service Account Impersonation

With iam.serviceAccounts.actAs on a high-privileged SA, an attacker can:

# Generate access token for the target SA
curl -X POST "https://iamcredentials.googleapis.com/v1/projects/-/serviceAccounts/target-sa@project.iam.gserviceaccount.com:generateAccessToken" \
  -H "Authorization: Bearer $CURRENT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"scope":["https://www.googleapis.com/auth/cloud-platform"]}'

# Use the new token to access resources as target-sa
curl "https://storage.googleapis.com/storage/v1/b?project=target-project" \
  -H "Authorization: Bearer $TARGET_SA_TOKEN"

Service Account Key Files — The Long-Lived Credential Problem

SA key files (JSON) are long-lived credentials that never expire unless explicitly deleted. They can authenticate from any machine worldwide.

Common ways SA keys are leaked:

Leak Vector Description
GitHub/GitLab commits Developer commits key JSON file; bots scan GitHub for service_account.json patterns
Docker image layers Key embedded in build step; visible in image history
CI/CD environment variables Key stored as CI secret; leaked via verbose build logs
Application config files Key in config.yaml or .env committed to version control
GCS buckets (public or over-shared) Key file stored in a bucket with incorrect ACLs
Cloud Function environment variables SA key stored as env var; visible to anyone who can describe the function

Best practice: Never create SA key files. Use Workload Identity instead.

Workload Identity vs SA Keys

Property SA Key File Workload Identity
Lifespan Years (until deleted) 1 hour (auto-refreshed)
Scope of exposure Usable from anywhere on the internet Scoped to a specific K8s SA + namespace
Rotation Manual Automatic
Revocation Requires deleting the key Removing Workload Identity binding
Audit trail Minimal Full Cloud Audit Log trace with namespace/pod identity
Risk if leaked Full SA access from attacker's machine Limited; token expires quickly

What Can Go Wrong

# Attack Root Cause Impact Likelihood
1 SA key file in GitHub repo Developer committed key file Full SA access from attacker machine High
2 SSRF to metadata service Web app SSRF + metadata endpoint accessible Node SA token stolen → lateral movement Med-High
3 Over-privileged SA on GKE node All pods on node inherit node SA's permissions Any compromised pod → node-level GCP access High
4 actAs escalation chain SA has actAs on higher-privileged SA Hop to project admin Med
5 SA key file in Docker image Key baked into image layer; pushed to public registry Full SA access to anyone who pulls the image Med
6 Default SA with Editor role GCP default SA for Compute has Editor by default Any code exec → project-wide editor access Med-High

Mitigation Controls

# Threat Mitigation Where
1 SA key leakage Disable/avoid SA key creation; enforce via org policy constraints/iam.disableServiceAccountKeyCreation GCP Org Policy
2 Over-privileged SA Least-privilege SA per service; regularly audit with Policy Analyzer IAM governance
3 Default Compute SA with Editor Disable default SA or change from Editor role to no roles; create purpose-built SAs IAM
4 Metadata service SA token theft Use Workload Identity on GKE; block metadata from pods via NetworkPolicy egress K8s + networking
5 actAs misuse Never grant actAs on admin SAs to lower-trust workloads; monitor with Cloud Audit Logs IAM audit
6 Cross-project SA escalation Limit cross-project SA roles; use VPC Service Controls to bound API access by perimeter VPC-SC + IAM
7 Detect leaked keys Use Secret Scanner in GitHub; enable Cloud Audit Log alerts on SA token use from unexpected IPs SIEM + detection

Key Takeaways

  • Service accounts are the primary lateral movement vehicle in cloud environments — every resource has one and they're poorly governed.
  • The GCP metadata service gives every VM/pod/function access to its SA token with zero additional authentication.
  • SA key files are the most dangerous credential type — long-lived, globally valid, frequently leaked in code and Docker images.
  • Workload Identity replaces SA keys with short-lived, scope-limited OIDC tokens — mandatory in any secure GKE deployment.
  • The actAs permission creates privilege escalation chains; audit all actAs bindings across a project.
  • GCP default Compute Service Account with Editor role is a common misconfiguration that turns any compromised VM into a project-level compromise.
