Lateral Movement and Privilege Escalation

Category: Infrastructure Security — Attack Techniques
MITRE ATT&CK: TA0008 (Lateral Movement), TA0004 (Privilege Escalation)
Applies To: Cloud environments (GCP, AWS, Azure), Kubernetes clusters, on-premises infrastructure, hybrid environments

Overview

Lateral movement is the set of techniques attackers use to progressively move through a network or infrastructure environment after initial access, spreading to additional hosts, services, or accounts with the goal of reaching high-value targets (databases, key vaults, admin accounts, billing data). Privilege escalation is the process of gaining higher permissions than initially available — from a low-privilege compromised container to cluster-admin, or from a constrained GCP service account to project owner.

At Google scale, lateral movement is the attacker's critical path. Initial access is rarely on the highest-value target — attackers gain a foothold (compromised container, phished employee, exploited public-facing service) and then must navigate to where the sensitive data lives. Defense-in-depth, least-privilege IAM, and microsegmentation all exist specifically to increase the cost of each lateral movement step.

Lateral Movement Techniques

On-Premises and VM Environments

Technique How It Works MITRE ID
Pass-the-Hash Use NTLM password hash directly without cracking to authenticate T1550.002
Pass-the-Ticket Steal Kerberos TGT/service ticket and use it on another system T1550.003
Overpass-the-Hash Convert NTLM hash to Kerberos ticket (TGT); use Kerberos where NTLM blocked T1550.002

Step-by-step K8s lateral movement from a compromised pod:

# Step 1: Check what's available in the environment
env | grep -i kube
cat /var/run/secrets/kubernetes.io/serviceaccount/token
cat /var/run/secrets/kubernetes.io/serviceaccount/namespace

# Step 2: Talk to the API server
export TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
export NS=$(cat /var/run/secrets/kubernetes.io/serviceaccount/namespace)
curl -k -H "Authorization: Bearer $TOKEN" https://kubernetes.default.svc/api/v1/namespaces/$NS/pods

# Step 3: Check permissions
curl -k -H "Authorization: Bearer $TOKEN" https://kubernetes.default.svc/apis/authorization.k8s.io/v1/selfsubjectaccessreviews \
  -H "Content-Type: application/json" -d '{"apiVersion":"authorization.k8s.io/v1","kind":"SelfSubjectAccessReview","spec":{"resourceAttributes":{"namespace":"default","verb":"create","resource":"pods"}}}'

# Step 4: Create privileged pod and exec into it
kubectl --token=$TOKEN exec -it victim-pod -- /bin/sh

Privilege Escalation Techniques

Linux Privilege Escalation

Technique Description Detection
SUID binary exploitation Find setuid root binaries; exploit to run as root find / -perm -4000 -type f 2>/dev/null
Sudo misconfiguration sudo -l; find commands that can be run as root without password /etc/sudoers audit
Writable cron jobs Cron running script owned/writable by attacker; add malicious commands File permission audit
PATH hijacking Create malicious binary matching a command called in root-owned script Monitor PATH in root scripts
Writable /etc/passwd Append root-equivalent entry with no password File integrity monitoring
Kernel exploits dirty_cow, dirty_pipe; exploit kernel to get root Track kernel version + patch status
Docker socket access User in docker group → create privileged container → root Audit docker group membership
Capabilities exploit Process with dangerous cap (CAP_SYS_ADMIN) uses it to gain root Audit capabilities per process

Attack Chain Example: Container to Cloud Admin

sequenceDiagram
    participant A as Attacker
    participant P as Compromised Pod
    participant K as K8s API Server
    participant M as GCP Metadata Service
    participant I as GCP IAM

    A->>P: Initial access via RCE in web app
    P->>P: Read /var/run/secrets/.../token
    P->>K: Query API - list pods, secrets
    K->>P: return Secret: database-credentials
    P->>M: GET metadata/v1/instance/service-accounts/default/token
    M->>P: return GCP SA access token (node SA)
    P->>I: listServiceAccounts - find over-privileged SAs
    I->>P: return SA: data-pipeline@project.iam.gserviceaccount.com with Storage Admin
    P->>I: generateAccessToken for data-pipeline SA (if actAs permission)
    I->>P: return access token for data-pipeline SA
    P->>A: Exfiltrate token - now has Storage Admin access

What Can Go Wrong

# Attack Root Cause Impact Likelihood
1 Kubernetes SA token used for cluster-admin SA over-privileged; automount not disabled Full cluster compromise High
2 Cloud SA escalated via actAs SA granted actAs on higher-privileged SA Project-wide data access Med-High
3 Pass-the-Hash on Windows domain NTLM credential in memory; no Kerberos enforcement Domain lateral movement Med
4 Sudo misconfiguration → root NOPASSWD entry for dangerous command Local root escalation High
5 SUID binary exploitation Writable SUID binary or exploitable SUID binary Root on host Med
6 Kubernetes RBAC wildcard * verb on * resource in role Cluster-admin equivalent High (misconfig)
7 etcd unauthenticated access No TLS client auth on etcd; network reachable All secrets readable + writable Med (misconfiguration)

Mitigation Controls

# Threat Mitigation Where
1 SA token misuse Disable automounting; use Workload Identity; scope SA to minimum permissions K8s + IAM
2 Over-privileged cloud SAs Audit and enforce least-privilege; no SA should have iam.serviceAccounts.actAs on privileged SAs Cloud IAM
3 Pass-the-Hash Enforce Kerberos; disable NTLM where possible; use Credential Guard (Windows) AD / Windows
4 Sudo misconfiguration Add NOPASSWD only for specific safe commands; regular sudoers audit Linux config
5 Kubernetes RBAC wildcard Regular audit (kubectl get clusterrolebindings); no wildcard roles in prod RBAC governance
6 East-west movement NetworkPolicy default-deny; service mesh mTLS (Istio/Anthos Service Mesh) CNI + Service mesh
7 etcd access TLS client certs; network isolation; encrypt at rest etcd config
8 Cloud metadata service Block from pods via NetworkPolicy; use Workload Identity instead Network + IAM
9 Command and control Egress filtering; detect unexpected outbound connections using VPC Flow Logs Network monitoring

Key Takeaways

  • Initial access is rarely on the final target; lateral movement is how attackers get to valuable data.
  • Kubernetes RBAC is a primary lateral movement vector — SA tokens are mounted by default and SA permissions are often over-scoped.
  • Cloud IAM permissions like actAs, setIamPolicy, and generateAccessToken are force multipliers for privilege escalation.
  • Least privilege is the most impactful control: minimize each SA, pod, and IAM member's permissions to only what is specifically needed.
  • Network microsegmentation and default-deny NetworkPolicy limit the blast radius once an attacker has a foothold.
  • Audit logs (Cloud Audit Logs, K8s audit, VPC Flow Logs) are essential for detection — baseline normal behavior and alert on deviations.
