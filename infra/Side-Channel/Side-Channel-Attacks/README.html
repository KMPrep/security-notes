Side-Channel Attacks

Category: Hardware and Software Security — Information Leakage
Applies To: Any computing environment: hardware, OS, hypervisors, crypto libraries, cloud platforms, browsers

Overview

A side-channel attack extracts secret information not by directly breaking an algorithm or exploiting a software vulnerability, but by measuring physical properties or observable side effects of a computation — timing, power consumption, electromagnetic emissions, acoustic noise, or CPU cache behavior. Side-channel attacks are particularly insidious because they leave the cryptographic algorithm mathematically intact; the weakness lies in the implementation or the physical environment.

At Google scale, side-channel attacks are a critical threat to any shared infrastructure. In a cloud environment, multiple tenants share CPUs, memory buses, caches, and DRAM. CPU cache timing attacks (the basis of Spectre/Meltdown) allow one tenant to read another tenant's memory. These attacks drove fundamental changes to CPU microarchitecture, browser JavaScript engines, and cloud hypervisor design.

Timing Attacks — The Foundation

The most widespread side-channel class. The computation time leaks information about the secret.

RSA Timing Attack (Kocher 1996)

The Montgomery modular multiplication algorithm used in RSA takes different time depending on the intermediate values during exponentiation. By measuring decryption time for many crafted ciphertexts, an attacker can statistically recover the private key.

Root cause: The mod_exp algorithm has a data-dependent branch — it only performs a multiply step when a bit is 1, not 0. The time difference is on the order of nanoseconds but with thousands of measurements, statistical signal extraction is feasible.

Padding Oracle Attack (POODLE, BEAST)

A padding oracle leaks timing or error response differences about whether CBC padding is valid.

sequenceDiagram
    participant A as Attacker
    participant S as Server (padding oracle)

    A->>S: Send crafted ciphertext block
    S->>S: Decrypt; check padding
    S->>A: 200 OK (valid padding) or 500 error (invalid padding)
    A->>A: Bit of information leaked about plaintext
    A->>S: Repeat 256 times per byte (on average ~128)
    A->>A: Recover full plaintext block

Timing Attack on String Comparison

# UNSAFE: early-exit comparison leaks timing information
def check_token_unsafe(provided, expected):
    return provided == expected  # Returns faster if prefix matches
    # An attacker can determine when the first N characters match
    # by measuring response time — recovers secret character by character

# SAFE: constant-time comparison using HMAC
import hmac
def check_token_safe(provided, expected):
    return hmac.compare_digest(provided, expected)
    # Always takes the same time regardless of where strings differ

Cache-Based Attacks

The CPU cache is a shared resource. Observing which cache lines are evicted during another process's computation reveals information about that computation.

Cache Hierarchy

Level Shared With Latency Size
L1 cache Only this core/thread ~4 cycles 32–64 KB
L2 cache Only this core ~12 cycles 256 KB–1 MB
L3 cache (LLC) All cores on one physical CPU ~30–40 cycles 4–32 MB
DRAM All sockets ~200–400 cycles GBs

The critical shared resource: The L3 (Last Level Cache) is shared across all cores on a physical CPU. In a VM/container environment, all co-tenant processes (VMs, containers) on the same physical CPU share the L3 cache. This is the attack surface for Flush+Reload, Prime+Probe, and Spectre.

Flush+Reload

  1. Attacker shares data with victim (via shared memory, shared library)
  2. Attacker flushes a cache line using CLFLUSH instruction
  3. Victim accesses the cache line (e.g., accessing an AES S-box entry that depends on key)
  4. Attacker measures time to reload the cache line: fast = victim loaded it (1), slow = victim didn't load it (0)
  5. Attacker recovers AES key bits one by one
flowchart LR
    A[Attacker: CLFLUSH target address] --> B[Cache line evicted to DRAM]
    B --> C[Victim process executes crypto with secret key]
    C --> D{Does victim access target address?}
    D -- Yes access depends on key bit = 1 --> E[Cache line loaded into L3]
    D -- No key bit = 0 --> F[Cache line remains in DRAM]
    A --> G[Attacker: time to read target address]
    E --> H[Fast reload ~5ns - key bit was 1]
    F --> I[Slow reload ~50ns - key bit was 0]

Prime+Probe

Works even without shared memory (usable across VMs sharing a physical CPU):
1. Attacker fills a cache set with its own data (Prime)
2. Victim executes secret-dependent computation
3. Attacker measures its own data access time (Probe)
4. If slow: victim evicted attacker's data from that cache set (cache set was accessed by victim)
5. Attacker maps which cache sets were accessed → recovers secret access pattern

Power Analysis (SPA/DPA)

Primarily a hardware/embedded attack against smartcards, HSMs, and IoT devices.

Type Description
Simple Power Analysis (SPA) Single power trace reveals algorithm structure and key operations
Differential Power Analysis (DPA) Statistical analysis of many power traces; isolates key-dependent power variation
Correlation Power Analysis (CPA) Correlates power traces with hypothetical power model for each key guess

Target: Any device that performs symmetric (AES, 3DES) or asymmetric (RSA, ECC) operations under secret key, and whose power consumption is measurable.

Defense: Algorithmic noise injection, randomized execution order, masking operations (XOR intermediate values with random mask), hardware countermeasures in crypto accelerators.

Rowhammer — DRAM Fault Injection

Rowhammer is a side-channel/fault injection hybrid. Repeatedly accessing the same DRAM row causes capacitor charge to leak into adjacent rows, flipping bits:

flowchart TD
    A[Attacker hammers rows above and below target row] --> B[Repeated DRAM accesses cause charge leakage]
    B --> C[Bit flip occurs in target row]
    C --> D{What is in target row?}
    D --> E[Page table entry] --> F[Set bit to 1 = grant write permission to attacker]
    D --> G[Kernel memory] --> H[Corrupt kernel data structure]
    D --> I[Adjacent VM memory in hypervisor] --> J[Cross-VM memory corruption]
    style F fill:#ff6b6b,color:#fff
    style J fill:#ff6b6b,color:#fff

Real-world exploitation: Google Project Zero (2015) used rowhammer to escalate from a sandboxed Chrome renderer process to kernel privileges by flipping a page table bit. This was a complete sandbox escape.

Defenses: ECC DRAM (corrects single-bit errors); LPDDR4 Target Row Refresh (TRR); MADV_DONTDUMP on attacker-accessible pages; cloud providers use ECC RAM.

Microarchitectural Data Sampling (MDS)

After Spectre/Meltdown, researchers discovered additional CPU internal buffer leaks:

Attack Buffer Targeted Data Leaked
RIDL (Rogue In-flight Data Load) Line Fill Buffers Data from other processes/hyperthreads
Fallout Store Buffers Recent store data from other processes
ZombieLoad Fill Buffers Data from other VMs via hyperthreading
TAA (TSX Async Abort) TSX internal buffers Arbitrary process data

Defense for cloud: Disable hyperthreading on hypervisor hosts, or ensure VMs are not co-scheduled with other tenants on the same physical thread. AWS and GCP provide dedicated single-tenant instances for workloads requiring isolation from hyperthreading attacks.

Mitigation Controls

# Threat Mitigation Layer
1 Timing attacks on crypto Use constant-time crypto libraries (BoringSSL, libsodium); hmac.compare_digest() Application code
2 Cache-based key extraction AES-NI (hardware AES eliminates table lookups); constant-time crypto CPU/library
3 Rowhammer ECC DRAM; LPDDR4 TRR; isolate attacker-controlled memory regions Hardware
4 Cross-VM cache attack Disable hyperthreading for security-critical VMs; single-tenant dedicated hosts Cloud config
5 Spectre in browser Site Isolation; timer jitter; SharedArrayBuffer gating Browser + server headers
6 MDS attacks Flush CPU buffers on VM context switches (MDS_NO or VERW instruction) Microcode/OS
7 Power analysis on embedded Algorithmic masking; hardware noise injection; power filter capacitors Hardware design

Key Takeaways

  • Side-channel attacks extract secrets from observable physical properties — timing, cache state, power — not from breaking the algorithm.
  • Timing vulnerabilities in string comparison and non-constant-time crypto are common implementation bugs; always use constant-time primitives.
  • The L3 cache is shared across all cores on a physical CPU, making it a side-channel attack surface in all multi-tenant environments.
  • Rowhammer demonstrated that hardware faults induced by software can lead to privilege escalation.
  • MDS vulnerabilities in Intel CPUs (2019) showed that CPU internal buffers could leak data across hyperthread and VM boundaries.
  • Defense requires a combination of constant-time code, hardware countermeasures (AES-NI, ECC), OS mitigations (VERW, KPTI), and cloud isolation (dedicated hosts, SMT disable).
