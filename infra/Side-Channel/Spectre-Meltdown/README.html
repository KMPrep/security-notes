Spectre and Meltdown

Category: Hardware Security — CPU Speculative Execution Vulnerabilities
CVE: Spectre: CVE-2017-5753 (v1), CVE-2017-5715 (v2); Meltdown: CVE-2017-5754
Disclosed: January 3, 2018 — one of the most impactful security disclosures in computing history
Discovered by: Google Project Zero (Jann Horn), Cyberus Technology, Graz University of Technology

Overview

Spectre and Meltdown are two classes of CPU vulnerability that exploit the speculative execution and out-of-order execution features found in virtually every modern processor. These features are critical performance optimizations — modern CPUs are 5–10x faster than they would be without them — but they leak information through the CPU cache timing side channel.

Meltdown breaks the boundary between user-space processes and kernel memory, allowing a process to read kernel memory (including other processes' private data, passwords, and keys) without any kernel vulnerability.

Spectre breaks the isolation between processes, allowing a malicious process to trick another process into speculatively executing instructions that expose the victim's memory via cache timing.

Google's response to Spectre/Meltdown was the largest coordinated hardware/software response in computing history — involving Intel, AMD, ARM, Microsoft, Apple, Google, Linux kernel developers, VMware, Xen, and all major cloud providers, all working under an embargo for months before the January 2018 public disclosure.

Background: Speculative Execution

Modern CPUs achieve performance by:

Feature Description Performance Gain
Out-of-order execution Execute instructions before preceding instructions complete ~20–30% IPC improvement
Branch prediction Predict which branch will be taken; execute speculatively Avoids ~15 cycle pipeline flush per misprediction
Speculative execution Execute instructions down a predicted path before the prediction is confirmed Combined: modern CPUs run 3–4x faster
Memory access pipelining Begin memory access before knowing if it's allowed Overlaps memory latency with computation

The security violation: when CPU rolls back a speculative execution because the prediction was wrong, it does not roll back the cache state. The speculatively executed instructions may have loaded secret data into the cache. An attacker can then use cache timing (measure access time to specific memory locations) to determine what was loaded — even if the instructions were never "officially" executed.

Meltdown — CVE-2017-5754

What Meltdown Does

Meltdown allows a user-space process to read arbitrary kernel memory, bypassing the hardware privilege check enforced by page table permission bits.

How It Works

sequenceDiagram
    participant CPU as CPU (out-of-order)
    participant ATK as Attacker Process
    participant KM as Kernel Memory

    ATK->>CPU: Load instruction: read kernel_addr (this will fault)
    CPU->>CPU: Out-of-order: speculatively execute BEFORE checking permissions
    CPU->>KM: CPU begins memory read from kernel_addr
    KM->>CPU: Return kernel secret byte (e.g., value = 42)
    CPU->>CPU: Speculative execution continues: use byte 42 as array index
    CPU->>CPU: Access probe_array[42 * 4096] - loads this cache line
    CPU->>CPU: Permission check catches: FAULT - user not allowed to read kernel
    CPU->>CPU: Rollback: discard register values
    CPU->>CPU: Cache NOT rolled back: probe_array[42*4096] is still in cache
    ATK->>CPU: Measure access time for each probe_array[i*4096] i in 0..255
    CPU->>ATK: probe_array[42*4096] is fast (cache hit)
    ATK->>ATK: Conclude: kernel byte was 42

The core code (pseudocode):

// This generates a segfault, but speculatively executed before fault detected
uint8_t secret = *(uint8_t *)kernel_address;  // FAULTS - but CPU speculatively executed it

// This is never officially executed BUT the cache was modified by speculative execution
// Attacker measures which element is in cache
cache_probe(probe_array + secret * PAGE_SIZE);

Meltdown Impact

Target readable Example
All kernel memory from user space OS secrets, other process data in kernel buffers
Physical memory map On some kernels: read arbitrary physical memory
VM memory (via hypervisor kernel) On guests without KPTI: hypervisor and other guest data

Spectre — CVE-2017-5753 (v1) and CVE-2017-5715 (v2)

Spectre is more subtle and more difficult to fix than Meltdown. It tricks the victim into speculatively executing code using the victim's own memory.

Spectre v1 — Bounds Check Bypass

// Victim code with bounds check
if (x < array1_size) {           // Bounds check - looks safe
    y = array2[array1[x] * 256]; // Only executed if x is in bounds
}

Attack:
1. Attacker trains the CPU branch predictor to predict that x < array1_size will be true
2. Attacker then calls victim code with an out-of-bounds x that points to a secret
3. CPU branch predictor predicts "true" → speculatively executes the body
4. array1[x] is the secret byte (out-of-bounds memory)
5. CPU speculatively accesses array2[secret * 256] — loads that cache line
6. Branch prediction resolves as false → rollback; but cache remains modified
7. Attacker measures access time to array2[i * 256] for each i → finds which entry is cached → recovers the secret byte

Spectre v2 — Branch Target Injection

Controls which indirect branch target the CPU speculatively jumps to:
1. Attacker poisons the Branch Target Buffer (BTB) — a CPU hardware table mapping indirect branch addresses to their predicted targets
2. When victim process executes an indirect branch, CPU mispredicts and jumps to attacker-chosen target in the victim's address space
3. Victim's code speculatively executes a "gadget" that loads victim's secret into cache
4. Attacker reads cache state

Mitigations

Meltdown Mitigation: KPTI (Kernel Page Table Isolation)

Before KPTI, the kernel's virtual address mappings were present in the user-space page table (but marked not accessible). This allowed Meltdown's speculative read. KPTI removes kernel mappings from the user-space page table entirely.

State User-space page table Kernel-space page table
Before KPTI Contains kernel mappings (marked inaccessible) Full kernel mappings
After KPTI Contains only user-space mappings (no kernel addresses) Full kernel mappings

Performance cost of KPTI: Every syscall requires a TLB flush (changing CR3 register). This caused 5–30% performance regression for syscall-heavy workloads (databases, storage systems). Google mitigated this by using PCID (Process Context ID) to allow TLB entries to coexist for different address spaces.

Spectre v1 Mitigation: LFENCE and Compiler Barriers

// Compiler inserts serializing instruction
if (x < array1_size) {
    LFENCE;  // Serialize: stops speculative execution past this point
    y = array2[array1[x] * 256];
}

Also: index masking — ensure index cannot be out of bounds even speculatively.

Spectre v2 Mitigation: Retpoline

Retpoline is a compiler technique that replaces indirect branches with a loop that the CPU's branch predictor cannot learn to speculate through:

# Instead of: jmp *rax  (indirect branch - BTB poisonable)
# Retpoline:
call set_up_target
capture_spec:
    pause
    jmp capture_spec  ; CPU speculatively loops here
set_up_target:
    mov [rsp], rax    ; Overwrite return address
    ret               ; CPU speculatively uses BTB; actually uses rsp

The ret is mispredicted, but the CPU speculates into the infinite pause+jmp loop, which does nothing. When the CPU figures out the actual target (from rsp), it jumps there without having leaked anything.

Full Mitigation Summary

Attack Mitigation Component Performance Cost
Meltdown KPTI OS kernel 5–30% on syscall-heavy workloads
Spectre v1 Compiler barriers (LFENCE), index masking Compiler / code ~1–3%
Spectre v2 (indirect branch) Retpoline; IBRS (Indirect Branch Restricted Speculation) Compiler + microcode 2–15%
Spectre v2 across hyperthreads STIBP (Single Thread Indirect Branch Predictor isolation) Microcode + OS ~1–2%
Store-to-load forwarding SSBD (Speculative Store Bypass Disable) Microcode ~2–8%
MDS (RIDL, Fallout, ZombieLoad) VERW instruction on privilege transitions; disable HT Microcode + OS 5–20% with HT disabled
Browser Spectre gadgets Site Isolation; timer resolution reduction; COOP+COEP Browser Memory: ~10% more RAM

The Coordinated Disclosure

Timeline of the world's largest coordinated vulnerability disclosure:

Date Event
June 2017 Jann Horn (Google Project Zero) discovers both Spectre and Meltdown
July 2017 Android team at Google independently discovers Spectre v2
July–August 2017 Intel notified; independent researchers also discover the bugs
Late 2017 KPTI (called KAISER) quietly merged into Linux kernel; spotted by security researchers
January 3, 2018 Coordinated public disclosure — one day early due to leaks
Several months later Patches rolled out across billions of devices

Key Takeaways

  • Spectre and Meltdown exploit CPU speculative execution — a performance optimization present in all modern processors since the late 1990s.
  • Meltdown breaks the kernel/user-space boundary; KPTI mitigated it with measurable performance cost.
  • Spectre tricks a process into speculatively executing code that leaks the process's own secrets via cache timing.
  • Retpoline defeats Spectre v2; LFENCE/compiler barriers mitigate v1 gadgets; mitigation is incomplete — new variants continue.
  • The Jan 2018 disclosure was the largest coordinated security response in computing history: CPUs, OSes, hypervisors, browsers, cloud providers, all changed simultaneously.
  • Site Isolation in Chrome was directly triggered by Spectre — cross-site JavaScript Spectre attacks required putting sites in separate processes.
