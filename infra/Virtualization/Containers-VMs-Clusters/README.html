Containers, VMs, and Clusters

Category: Infrastructure Security — Virtualization and Orchestration
Applies To: Docker, Kubernetes, Google Kubernetes Engine (GKE), AWS EKS, VM fleets, hybrid environments

Overview

Containers, VMs, and clusters are the three primary compute abstractions in modern cloud infrastructure. Each provides a different level of isolation, performance tradeoff, and security model. Understanding the differences — and their failure modes — is essential for designing secure, scalable infrastructure.

At Google scale, Borg (the predecessor to Kubernetes) runs containers for nearly all of Google's internal workloads. Google's threat model for containers assumes a compromised container must not be able to affect other containers on the same host or access another tenant's data. This is why Google runs GKE with hardened defaults (Container-Optimized OS, gVisor sandbox, Workload Identity) that significantly exceed standard Docker security.

Virtual Machines — Deep Dive

How VMs isolate workloads:

  1. Each VM has its own kernel — kernel exploits within the VM cannot affect the hypervisor or other VMs
  2. Memory is isolated at the EPT level — no shared physical pages between VMs (unless KSM is enabled)
  3. Network is isolated through virtual switches (vSwitch, OVS) — VMs communicate via emulated NICs
  4. Storage is isolated through virtual disks (qcow2, raw) — each VM sees only its own virtual block device

VM Security Properties:

Property Behavior
Kernel isolation Each VM has independent kernel; kernel CVE in guest VM does not affect others
Memory isolation EPT maps each VM to separate physical pages
Hypervisor attack surface VM escape requires exploiting hypervisor emulation layer
Boot integrity Can use UEFI Secure Boot within VM
Filesystem Separate block device per VM

The critical security difference: All containers share the host kernel. A kernel exploit (such as a kernel privilege escalation CVE) running inside a container can break out to the host. There is no hypervisor boundary.

Kubernetes — Cluster Architecture

Kubernetes orchestrates containers at scale. A cluster consists of:

Component Role Security Relevance
API Server Central control plane; all cluster state flows through it Primary attack target; must not be internet-exposed
etcd Key-value store for all cluster state (secrets, configs) Contains all Kubernetes Secrets; must be encrypted at rest
kubelet Node agent running on each worker node If compromised, attacker controls all pods on the node
kube-proxy Network proxy; implements Service networking Can intercept pod-to-pod traffic if compromised
Container Runtime runC, containerd, CRI-O Runs actual containers; interface between K8s and namespaces
Service Account Pod identity; JWT mounted in /var/run/secrets Token exposed by default; used by attackers for API Server access

Kubernetes Default Security Risks

Default Behavior Risk
Service account token automounted Every pod gets a service account JWT at /var/run/secrets/kubernetes.io/serviceaccount/token
API server accessible from pods Default network policy allows pod → API server traffic
Pods run as root by default runAsNonRoot: false by default; UID 0 in container → potential host root via escape
Privileged containers allowed privileged: true grants full host access including devices and capabilities
hostPID: true / hostNetwork: true accessible Pods can share host PID namespace or host network
No network policies by default Pods can communicate with any other pod in the cluster

Mitigation Controls

# Threat Mitigation Where Applied
1 Kernel CVE exploitation Use gVisor (runsc) as container runtime on GKE; separate syscall surface Container runtime
2 Privileged containers PodSecurityAdmission: Restricted profile; deny privileged, hostPID, hostNetwork Kubernetes admission controller
3 Service account token abuse Disable automounting; use Workload Identity (GKE) for cloud IAM binding instead of SA tokens K8s pod spec + IAM
4 Container running as root runAsNonRoot: true, runAsUser: 1000 in securityContext Pod manifest
5 API server exposure Private GKE clusters; firewall rules; restrict kubectl to VPN or bastion GKE network config
6 etcd not encrypted Enable etcd encryption at rest; TLS mutual auth for etcd access etcd config
7 Lateral movement Kubernetes NetworkPolicy to deny all ingress/egress by default; allow only necessary flows Network policy
8 Malicious images Binary Authorization (GKE): only signed images from known registries Admission controller
9 Over-privileged capabilities Drop all Linux capabilities (drop: ["ALL"]); add back only needed ones securityContext

Key Takeaways

  • Containers share the host kernel — a kernel exploit breaks out of the container; VMs have a full kernel boundary.
  • Kubernetes clusters have many default-insecure behaviors (automounted tokens, no NetworkPolicy, no PodSecurityAdmission).
  • Use Restricted PodSecurityAdmission, Workload Identity, and NetworkPolicy as your K8s security baseline.
  • etcd contains all cluster secrets — protect it with TLS, encryption at rest, and strict network access control.
  • gVisor and Kata Containers add a second isolation layer below containers, making kernel CVE exploitation much harder.
  • Defense in depth is required: image signing, runtime scanning, network policy, admission control, and IAM minimization.
