Hyperjacking

Category: Infrastructure Security — Advanced Hypervisor Attacks
MITRE ATT&CK: T1564.006 (Hypervisor Rootkit)
Applies To: Any hypervisor-based environment: cloud, on-premises virtualization, enterprise data centers

Overview

Hyperjacking is the installation of a malicious hypervisor (or a sub-hypervisor) beneath an existing operating system or hypervisor, creating an invisible software layer that intercepts and controls all hardware interactions. The attack vector is sometimes called a "Virtual Machine Based Rootkit" (VMBR). Unlike a traditional rootkit that runs within a guest OS, a hyperjacking attack runs below the OS, making it nearly invisible to standard detection tools running in the OS or VM.

At cloud scale, a successful hyperjacking attack against a hypervisor host gives the attacker:
- Read/write access to every VM's memory on that host
- The ability to intercept all I/O: network traffic, disk reads/writes, encryption keys flowing through memory
- Persistence that survives OS reinstalls and VM terminations
- A hidden vantage point from which to pivot to other hosts via the management plane

The most well-known proof-of-concept hyperjacking tool is BluePill (created by Joanna Rutkowska, 2006), which used AMD-V SVM to trap a running Windows system into a VM without it noticing. While largely theoretical at production cloud scale, hyperjacking represents the most severe possible virtualization threat.

How It Works

Attack Vectors

Attack Vector Description Prerequisites
Sub-hypervisor installation Attacker installs a rogue hypervisor beneath existing OS using CPU virtualization extensions Kernel-level (ring-0) code execution on host
Hypervisor replacement Compromise existing hypervisor binary or boot process; replace with malicious version Physical access or firmware-level access
Firmware-level implant UEFI/BMC firmware compromise inserts hypervisor before OS boots Physical access or firmware update mechanism exploit
Live migration attack Intercept and modify VM during live migration between hosts MITM on management network
Management plane compromise Attack vCenter, KVM management APIs, or cloud control plane Network access to management interfaces

BluePill Technique (Theoretical / PoC)

  1. Attacker achieves kernel-level code execution on a running Windows system
  2. Code loads AMD-V SVM module (VMX(Intel) or SVM(AMD) must be available)
  3. Code creates a VM control block (VMCB) that describes the current system state
  4. Code executes VMRUN instruction — hardware transfers control to the rogue hypervisor
  5. The original OS is now running as a Virtual Machine under the rogue hypervisor
  6. The OS has no way to detect this transition (the hypervisor controls CPUID responses)

What a Compromised Hypervisor Can Do

Capability Description
Memory read Read any VM's RAM, extracting encryption keys, passwords, session tokens
Memory write Inject code into running VMs — backdoor guest OSes
Network intercept Clone all network packets from all VMs; man-in-the-middle encrypted traffic at the point before encryption
Disk intercept Intercept reads/writes to virtual disks; exfiltrate or modify data
Keylogger Intercept keyboard input via virtual keyboard controller
Screen capture Capture VM display framebuffer
Bypass VM security tools Any security tool running in a VM is in a controlled environment; the hypervisor can hide events from it
Persistence Survives guest OS reboots, reinstalls, kernel updates

Mitigation Controls

# Threat Mitigation Where Applied
1 Sub-hypervisor installation Enforce Secure Boot; sign all hypervisor binaries; use TPM attestation Host firmware / boot chain
2 Management plane exposure Network-segment management interfaces; no internet access to vCenter/API; MFA Network architecture
3 Live migration attacks Encrypt vMotion / live migration traffic with TLS; verify destination host identity Hypervisor config
4 UEFI/BMC firmware attacks Enable UEFI Secure Boot; update BMC firmware; use vendor signing Hardware / firmware
5 Memory content exposure AMD SEV / Intel TDX: encrypt VM memory with per-VM keys inaccessible to hypervisor Hardware
6 Rogue module loading Kernel module signing; enforce with IMA (Linux Integrity Measurement Architecture) Host OS
7 Detection Runtime integrity monitoring via TPM TrustChain; compare running kernel hash to known good IDS / attestation system
8 Lateral movement via mgmt plane Least-privilege API keys for hypervisor management; audit all API calls IAM / audit

Top 3 Critical Mitigations

  1. Hardware attestation with TPM + Secure Boot — At each boot, Secure Boot verifies the bootloader and hypervisor binary signature. The TPM records measurements at each boot stage. Any modification to the hypervisor binary changes the PCR measurements, which remote attestation services can detect. Google's infrastructure uses a similar chain-of-trust for all GCP hosts.
  2. Segregate the hypervisor management plane — vCenter, KVM management APIs, IPMI, and BMC interfaces must be on an isolated management VLAN inaccessible from the internet or even from normal internal networks. Management access requires a dedicated jump host with strong authentication.
  3. Confidential Computing (AMD SEV / Intel TDX) — Even if the hypervisor is compromised, these hardware features encrypt each VM's memory with keys that are held in the CPU, never accessible to the hypervisor. This limits what an attacker controlling the hypervisor can extract from running VMs.

Key Takeaways

  • Hyperjacking installs a malicious hypervisor below the OS, making detection from within the OS impossible.
  • Realistic attack paths are via management plane compromise (vCenter CVEs), not raw sub-hypervisor installation.
  • UEFI Secure Boot + TPM attestation + kernel module signing creates a chain of trust that can detect hypervisor tampering.
  • AMD SEV/Intel TDX encrypts VM memory at the hardware level — even a compromised hypervisor cannot read VM RAM.
  • Management plane network segmentation is the most impactful preventive control.
  • Nation-state adversaries and APT groups have demonstrated hypervisor-level attacks (e.g., Equation Group firmware implants).
