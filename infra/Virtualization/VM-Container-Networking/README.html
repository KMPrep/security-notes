Network Connections from VMs and Containers

Category: Infrastructure Security — Virtual Networking
Applies To: Docker networking, Kubernetes CNI, virtual switch (OVS/VPC), cloud VPC networking

Overview

Understanding how network traffic flows from VMs and containers is essential for implementing correct network security controls. Virtual networking adds multiple abstraction layers between a container/VM and the physical network — each layer introduces potential attack surface, misconfigurations, and lateral movement opportunities. At Google scale, a compromised container must not be able to read traffic from another pod or reach services it has no business contacting. This is enforced through Kubernetes NetworkPolicy, VPC firewall rules, eBPF-based dataplane enforcement (Cilium), and GCP's VPC Flow Logs.

VM Networking Architecture

How a VM's Network Packet Reaches the Physical NIC

flowchart TD
    A[VM App - sends packet] --> B[Guest Network Stack]
    B --> C[Virtual NIC - emulated e1000 or virtio-net]
    C --> D[Hypervisor Network Backend - TAP device]
    D --> E[Virtual Switch - Linux bridge or OVS]
    E --> F[Physical NIC - via bonding or directly]
    F --> G[Physical Network]

Two VM networking models:

Model Description Performance Security
Software bridge (Linux bridge) VMs connected via kernel bridge; all traffic passes through hypervisor Moderate Bridge can be sniffed by root on host
OVS (Open vSwitch) Programmable virtual switch with OpenFlow rules Good Flow rules enforce isolation
SR-IOV (Single Root I/O Virtualization) VM gets direct access to virtual NIC function on physical NIC Near-native Bypasses hypervisor for data path
DPDK Bypass kernel entirely for NIC access Very high Requires careful isolation

VLAN-Based VM Isolation

Each VM or tenant is assigned a VLAN. The virtual switch tags outbound packets and enforces that VMs can only receive traffic on their VLAN.

VM A (VLAN 100) <-> vSwitch <-> VM B (VLAN 100)  → Can communicate
VM A (VLAN 100) <-> vSwitch <-> VM C (VLAN 200)  → Blocked by VLAN tagging

Container Networking Architecture

Docker Networking Modes

Mode How It Works Security Implications
bridge (default) Container gets a veth pair; one end in container, one in docker0 bridge All containers on same bridge can communicate; no isolation by default
host Container shares host network namespace Container opens ports directly on host; no isolation; not recommended
none No network interface; fully isolated Maximum network isolation
overlay Multi-host networking; uses VXLAN tunnels between hosts Encrypt VXLAN with IPsec for multi-host scenarios
macvlan Container gets a real MAC address on the physical network Bypasses docker0 bridge; appears as a physical host

Docker bridge mode — packet flow:

flowchart TD
    A[Container 1 - 172.17.0.2] --> B[veth0 in container]
    B --> C[veth1 on host - docker0 bridge]
    D[Container 2 - 172.17.0.3] --> E[veth2 in container]
    E --> F[veth3 on host - docker0 bridge]
    C --> G[docker0 bridge - 172.17.0.1]
    F --> G
    G --> H[host iptables/NAT rules]
    H --> I[Physical NIC - eth0]
    I --> J[External network]
    style G fill:#ff6b6b,color:#fff

By default, all containers on the same docker0 bridge can reach each other. This is the "flat network" problem — no micro-segmentation by default.

Kubernetes Networking

The Kubernetes Networking Model (CNI Contract)

Kubernetes requires:
1. Every pod gets its own unique IP address
2. All pods can communicate with all other pods without NAT (flat network by default)
3. All nodes can communicate with all pods
4. Pods see their own IP address as their IP (no NAT on pod-to-pod traffic)

This "flat network" model means network segmentation must be explicitly applied via NetworkPolicy.

CNI Plugins and Their Security Models

CNI Plugin Dataplane NetworkPolicy Encryption Notes
Calico iptables or eBPF Full NetworkPolicy support WireGuard optional Most deployed; strong security
Cilium eBPF Full + extended L7 policies WireGuard optional Best performance; eBPF-native; used by GKE Dataplane V2
Flannel VXLAN / UDP No NetworkPolicy by default None Simple; not suitable for production security
Weave Net VXLAN NetworkPolicy support Sleeve encryption Easier setup; moderate performance
AWS VPC CNI AWS VPC routing AWS Security Groups for pods VPC-level Native AWS integration; SG-per-pod

Kubernetes NetworkPolicy — The Security Baseline

Without NetworkPolicy, all pod-to-pod traffic is allowed. With NetworkPolicy, you create explicit allow rules:

# Default deny all ingress to a namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}  # Applies to all pods in namespace
  policyTypes:
  - Ingress

---
# Allow only frontend -> backend traffic
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080

Network Attack Scenarios

1. ARP Spoofing in a VM/Container Environment

On a shared Layer-2 virtual network, a malicious container can send gratuitous ARP packets claiming it has the IP of another container or gateway. Other containers update their ARP cache and route traffic through the attacker (ARP spoofing man-in-the-middle).

Prevention: Enable arp_ignore and arp_filter on hosts; use CNI plugins that prevent ARP spoofing (Cilium native routing mode eliminates L2 and ARP entirely).

2. Container to Metadata Service via Host Network

If a container is on the same network as the host (or has hostNetwork: true), it can reach the cloud metadata service (169.254.254.169 on GCP, 169.254.169.254 on AWS):

# Inside a hostNetwork container on GCP
curl -H "Metadata-Flavor: Google" http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token
# Returns the node's GCP service account token

Prevention: Block metadata IP in NetworkPolicy or with iptables on the host.

3. DNS Spoofing / CoreDNS Poisoning

If an attacker compromises a pod and can reach CoreDNS, they may be able to inject DNS responses. Kubernetes uses CoreDNS for cluster-internal DNS. A compromised pod can poison another pod's DNS, redirecting service traffic to the attacker.

Prevention: NetworkPolicy restricting CoreDNS access; deploy Cilium L7 DNS policy to validate DNS responses.

4. Lateral Movement via East-West Traffic

In a flat cluster network with no NetworkPolicy, a compromised pod in one namespace can reach any pod in any other namespace:

# Scan entire cluster from compromised pod (no NetworkPolicy)
for i in $(seq 1 254); do curl -s http://10.0.${i}.0/; done
# Can reach all pods across all namespaces

What Can Go Wrong

# Attack Root Cause Impact Likelihood

Mitigation Controls

# Threat Mitigation Where
1 Flat network / lateral movement Default-deny NetworkPolicy in all namespaces Kubernetes NetworkPolicy
2 Metadata service access Block 169.254.169.254 and metadata.google.internal in NetworkPolicy egress rules CNI / host iptables
3 ARP spoofing Use Cilium in native routing mode (no L2); or enable ARP filtering on vSwitch CNI / host config
4 VXLAN interception Enable WireGuard encryption in Cilium or Calico for pod-to-pod traffic CNI config
5 NodePort exposure Use ClusterIP or private LoadBalancer; audit all NodePort services K8s service spec
6 hostNetwork pods Deny in PodSecurityAdmission unless required by system DaemonSets Admission control
7 DNS spoofing Deploy Cilium L7 DNS policy; lock CoreDNS access to pods via NetworkPolicy CNI + CoreDNS
8 Promiscuous NIC access Drop CAP_NET_RAW; deny NET_ADMIN in pod security; monitor with Falco securityContext

Key Takeaways

  • Default Kubernetes networking is a flat network — all pods can reach all pods without any policies applied.
  • NetworkPolicy default-deny-all is the most critical security baseline for any production cluster.
  • The cloud metadata service (169.254.169.254) must always be blocked from container egress.
  • Cilium with eBPF provides the best combination of performance, security, and observability for production K8s networking.
  • ARP spoofing is a real threat in container environments with L2 virtual networks; Cilium native routing eliminates it.
  • Encrypt pod-to-pod traffic (WireGuard in CNI) for any sensitive workload — east-west traffic is often unencrypted by default.
