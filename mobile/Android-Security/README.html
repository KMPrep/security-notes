Android Security

Category: Mobile Security — Android Platform Security Model

Overview

Android is built on a Linux kernel with a security model layered on top: UID-per-app isolation, SELinux mandatory access control, an application sandbox, and a permission system that controls access to sensitive data and hardware. Understanding the Android security model is prerequisite to understanding how Android malware operates, how pentesting an Android app works, and how Google defends its platform at scale.

Android Security Architecture

flowchart TD
    APP[Android App APK] --> SANDBOX[App Sandbox\nLinux UID per app\nSELinux process label]
    SANDBOX --> ART[ART Runtime\nDalvik bytecode DEX\nAOT JIT compilation]
    ART --> FW[Android Framework\nJava system services\nActivity Package managers]
    FW --> BINDER[Binder IPC\nCross-process communication\nPermission checked at crossing]
    BINDER --> NATIVE[Native Services\nmediaserver surfaceflinger\nkeystore]
    NATIVE --> KERNEL[Linux Kernel\nSELinux mandatory access control\ncgroup seccomp]
    KERNEL --> HW[Hardware\nTrustZone TEE\nStrongBox HSM Titan M]

App Sandbox

Every Android app runs as a unique Linux user ID (UID) — typically in the range 10000–99999. Apps from the same developer can share a UID if they have the same signing certificate and declare android:sharedUserId (deprecated in API 29+).

What the sandbox isolates:

  • File system: each app has private storage at /data/data/<package>/ readable only by its UID
  • Memory: separate process; one app cannot read another's memory without a permission or exploit
  • Binder IPC: all cross-process calls go through Binder, which checks caller UID against required permissions

What the sandbox does NOT isolate:

  • External storage (/sdcard): historically world-readable; now requires READ_EXTERNAL_STORAGE permission
  • Intents: exported activities/services are accessible to all apps unless guarded by permissions
  • Shared content providers if improperly configured

SELinux on Android

Android uses SELinux in enforcing mode (since Android 5.0). Every process and file has a type label. Access is only permitted if an explicit policy rule allows it.

Component SELinux Label
Third-party app untrusted_app
System app platform_app or specific system domain
init process init
zygote zygote
/data/data/<pkg>/ Per-app file type derived from package UID

Why it matters: Even if an app exploits a buffer overflow in a native library, SELinux limits what that process can do. A breakout to kernel root is needed to escape SELinux entirely — a significantly harder bar.

Binder IPC

Binder is Android's inter-process communication (IPC) mechanism. All cross-process calls — including calls to system services, camera, location, telephony — go through Binder. The Binder kernel driver verifies the caller's UID and checks against the system's permission database.

Binder security properties:

  • Caller UID and PID are unforgeable — filled in by the kernel at the crossing point
  • System services use Binder.getCallingUid() and checkCallingPermission() to authorise callers
  • No way for an app to spoof another app's UID in a Binder call

Attack surface: Vulnerabilities in Binder interfaces of system services (e.g., in mediaserver, telephony, SystemServer) have historically led to privilege escalation. Stagefright (2015) was a Binder-reachable vulnerability in libstagefright triggered by parsing media files.

Dangerous Permission Groups (Android 12+)

Android groups related dangerous permissions. Granting one in a group typically grants the rest in that group:

Group Permissions
Location ACCESS_FINE_LOCATION, ACCESS_COARSE_LOCATION, ACCESS_BACKGROUND_LOCATION
Contacts READ_CONTACTS, WRITE_CONTACTS, GET_ACCOUNTS
Camera CAMERA
Microphone RECORD_AUDIO
Storage READ_EXTERNAL_STORAGE, WRITE_EXTERNAL_STORAGE, MANAGE_EXTERNAL_STORAGE
Phone READ_PHONE_STATE, CALL_PHONE, READ_CALL_LOG
SMS SEND_SMS, READ_SMS, RECEIVE_SMS

Background Location

Since Android 10, ACCESS_BACKGROUND_LOCATION is a separate, additional dangerous permission requiring explicit user grant. Many stalkerware/spyware apps request this.

APK Internals

An APK is a ZIP file. Understanding its structure is fundamental to Android reversing and security assessment.

app.apk
├── AndroidManifest.xml    # Binary XML: permissions, activities, services, receivers
├── classes.dex            # Dalvik bytecode (DEX format)
├── classes2.dex           # Multidex overflow
├── res/                   # Resources (layouts, drawables, strings)
├── assets/                # Raw asset files (sometimes encrypted payloads)
├── lib/                   # Native .so libraries by ABI (armeabi-v7a, arm64-v8a, x86)
├── META-INF/              # Signing certificates and manifest hashes
│   ├── CERT.RSA           # App signing certificate
│   └── MANIFEST.MF        # Hash manifest
└── resources.arsc         # Compiled resource table

AndroidManifest.xml — Security-Critical Fields

Element Security Significance
<uses-permission> Every requested permission — check for overpermissioning
android:exported="true" Activity/service/receiver accessible to other apps — attack surface
android:debuggable="true" Should never be true in production; allows ADB shell in app context
android:allowBackup="true" App data backed up via ADB; sensitive data leaks
android:networkSecurityConfig References NSC file; controls cleartext and cert pinning
<provider android:grantUriPermissions> Content provider URI access grants

Android Keystore

The Android Keystore System stores cryptographic keys in hardware-backed secure storage (TEE or StrongBox). Keys are not extractable — private key material never leaves the secure hardware.

Key usage:

// Generate a key in hardware-backed keystore
KeyGenerator kg = KeyGenerator.getInstance(KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore");
kg.init(new KeyGenParameterSpec.Builder("mykey",
    KeyProperties.PURPOSE_ENCRYPT | KeyProperties.PURPOSE_DECRYPT)
    .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
    .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
    .setUserAuthenticationRequired(true)    // requires biometric/PIN on use
    .build());

StrongBox: On supported devices (Pixel 3+, Samsung with Titan M), keys reside in a dedicated security chip isolated from the main CPU. Even a full kernel compromise cannot extract StrongBox keys.

Common mistake: Storing encryption keys in SharedPreferences, hardcoding them in the APK, or using the deprecated KeyStore with software-backed storage.

Network Security Configuration (NSC)

NSC (Android 7.0+) controls TLS behaviour per-domain via an XML file referenced in AndroidManifest.xml.

<!-- res/xml/network_security_config.xml -->
<network-security-config>
    <domain-config>
        <domain includeSubdomains="true">api.example.com</domain>
        <pin-set>
            <pin digest="SHA-256">base64encodedpublickeypin==</pin>
            <pin digest="SHA-256">backuppin==</pin>
        </pin-set>
    </domain-config>
    <base-config cleartextTrafficPermitted="false" />
</network-security-config>

Security implications:

  • cleartextTrafficPermitted="false" blocks all HTTP (plain text) connections — should be production default
  • Pin-set enables certificate pinning — MITM requires pin bypass
  • cleartextTrafficPermitted="true" in production is a finding (OWASP M5)

ADB (Android Debug Bridge)

ADB is the primary developer and forensic tool for Android:

adb devices                              # list connected devices
adb shell                                # interactive shell (as app UID or shell)
adb pull /data/data/com.app/databases/   # extract app database (requires root or backup)
adb logcat                               # live log output (often includes sensitive data)
adb backup -apk -all -shared backup.ab  # logical backup
adb shell dumpsys package com.app       # dump package info including permissions

# Reverse TCP proxy for intercepting app traffic
adb reverse tcp:8080 tcp:8080           # forward device port to host

Security risk: ADB must be explicitly enabled (Developer Options). On a debug build (debuggable=true), ADB connects without device unlock. On release builds, ADB requires device trust approval.

USB Debugging for pentest: Enable ADB on the test device. Set up Burp Suite as proxy, install Burp CA cert. Bypass NSC by patching manifest or using Frida to hook TLS.

WebView Security

WebView vulnerabilities have historically been severe:

  • addJavascriptInterface + reflection: any Java method callable from JavaScript if API < 17
  • setAllowFileAccess(true) (default): file:// URIs can access /data/data/ — UXSS
  • setJavaScriptEnabled(true) with remote content: XSS in loaded page executes in app context

Android Malware Techniques

Technique Description
Repackaging Legitimate app + malicious code; distributed outside Play Store
Dynamic code loading Load DEX at runtime from remote server to evade static analysis
Reflection Call sensitive APIs via reflection to bypass static analysis
Accessibility Service abuse Grants broad UI access; used for keylogging, click automation, overlay attacks
Notification listener Reads all notifications from all apps (banking OTPs)
Device admin abuse Request device admin rights; becomes impossible to uninstall without factory reset
SMS interception Read SMS to steal OTPs; requires READ_SMS or RECEIVE_SMS
Overlay attack Draw over banking/authentication app; capture credentials
