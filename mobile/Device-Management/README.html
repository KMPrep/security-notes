Mobile Device Management (MDM) and Microsoft Intune

Category: Mobile Security — Enterprise Device Management, Zero Trust, Endpoint Compliance

Overview

Mobile Device Management (MDM) is the technology layer that allows enterprises to configure, secure, monitor, and remotely manage mobile devices at scale. Microsoft Intune (part of Microsoft Endpoint Manager / Microsoft Intune suite) is the dominant MDM/MAM solution in enterprise environments, deeply integrated with Azure Active Directory (Entra ID), Conditional Access, and the Microsoft 365 ecosystem.

Understanding MDM is essential for security engineering because it is a primary control plane for enforcing Zero Trust on endpoints: a device that is not enrolled, not compliant, or not trusted cannot access corporate resources — regardless of whether credentials are valid.

MDM vs MAM

MDM (Mobile Device Management) MAM (Mobile Application Management)
What is managed Entire device Specific managed apps only
Enrollment Device fully enrolled; MDM certificate installed App-level enrollment (Intune App SDK or App Wrapping Tool)
Scope of control Device-wide policies, passcode, full wipe Per-app data protection, clipboard, backup, copy-paste
Best for Corporate-owned devices; COPE BYOD where full device management is invasive
Remote wipe Full device wipe possible App data wipe only ("selective wipe")
Employee privacy Lower — IT can push anything to device Higher — IT sees only app-level compliance
Typical scenario Issued laptop/phone Employee personal phone accessing work email

MAM without MDM: Intune supports MAM policies enforced at the app level (via Intune App SDK) without requiring full device enrollment. An employee can access work email on their personal iPhone through the Outlook app with Intune MAM policies applied to that app only — IT cannot see or manage anything else on the device.

Push notification channels:

  • iOS/macOS: Apple Push Notification service (APNs) — Intune pushes commands to devices via APNs
  • Android: Firebase Cloud Messaging (FCM) / Google Mobile Services (GMS)
  • Windows: Windows Notification Service (WNS) / WMI

Device Enrollment Methods

Android Enrollment

Method Scenario Management Level
Android Enterprise Work Profile (BYOD) Personal device; work profile created Work profile only
Android Enterprise Fully Managed (COBO) Corporate device; factory reset + QR or NFC enrollment Full device
Android Enterprise COPE Corporate device with personal profile allowed Full device + personal profile allowed
Android Enterprise Dedicated (Kiosk) Single-purpose device (POS, scanner) Full device; locked to specific apps
Device Admin (legacy) Older Android; being deprecated Device-wide; less secure than Enterprise

Android Zero-Touch Enrollment: Corporate Android devices can be pre-configured to auto-enroll in Intune on first boot using Google's Zero-Touch Enrollment portal. No manual user action needed — device arrives at employee's desk and enrolls automatically.

iOS/iPadOS Enrollment

Method Scenario Management Level
User Enrollment (BYOD) Personal iPhone; user-initiated via Company Portal Managed Apple Account + managed apps; no full device control
Device Enrollment (BYOD/COPE) User enrolls personal or corp device via Company Portal MDM profile installed; managed apps; limited supervision
Automated Device Enrollment (ADE/DEP) Corporate device via Apple Business Manager Fully supervised; cannot remove MDM profile without wipe
Apple Configurator 2 Manual setup for devices not in ABM Supervised; physical access required

Apple Push Notification Certificate: Intune requires an APNs certificate (renewed annually) issued by Apple to push MDM commands to iOS devices. If this certificate expires, enrolled iOS devices stop receiving commands. Renewal is a critical operational task.

Conditional Access Integration

Conditional Access (Entra ID) consumes Intune compliance status as a signal. Policy: "Only allow access to Exchange Online from devices that are Intune-enrolled AND compliant AND not high-risk (Defender signal)."

flowchart TD
    USER[User attempts access\nto Exchange Online] --> CA[Conditional Access evaluates]
    CA --> CHECK1{Is device enrolled\nin Intune?}
    CHECK1 -->|No| BLOCK1[Block: require enrollment]
    CHECK1 -->|Yes| CHECK2{Is device compliant?}
    CHECK2 -->|No| BLOCK2[Block: show compliance remediation]
    CHECK2 -->|Yes| CHECK3{Is device risk low?\nDefender MTD signal}
    CHECK3 -->|High risk| BLOCK3[Block: investigate threat]
    CHECK3 -->|Low risk| ALLOW[Allow access]

Named Locations and Device Filters: CA policies can be scoped by location (trusted IP ranges), device platform (iOS only, Android only), device state (domain-joined, compliant, hybrid-joined), and app (require managed client app like Outlook).

Intune App Protection Policies (MAM)

App Protection Policies (APP) enforce data protection at the app level — critical for BYOD and for protecting data in managed apps.

Supported Apps

App Protection Policies work with apps that either:

  1. Integrate the Intune App SDK (Microsoft apps: Outlook, Teams, OneDrive, Word, Edge, etc. — all support this natively)
  2. Are wrapped with the Intune App Wrapping Tool (for line-of-business apps without SDK access)

Intune Security Baselines

Intune provides pre-built Security Baselines — curated collections of recommended settings based on Microsoft security guidance, CIS Benchmarks, and STIG standards. Baselines can be applied to device groups and override individual policy settings.

Available baselines:

  • Microsoft Security Baseline (Windows 10/11) — hardening settings: Defender, BitLocker, Firewall, UAC, IE/Edge settings
  • Microsoft Edge Baseline — browser security settings
  • Microsoft Defender for Endpoint Baseline — EDR-specific hardening
  • Microsoft 365 Apps for Enterprise Baseline — Office macro, add-in, and feature restrictions
  • Windows 365 Baseline — Cloud PC hardening
