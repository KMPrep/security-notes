Mobile Security — Android and iOS

Category: Mobile Security — Platform Security, Application Security, Attack Surface

Overview

Mobile security spans two distinct platform security models (Android and iOS), a shared application vulnerability surface (OWASP Mobile Top 10), and unique attack vectors that differ significantly from web and desktop. Both platforms have hardened substantially since 2010, but their open/closed philosophies, permission models, and trust anchors differ in ways that matter for attack and defence.

Android vs iOS Security Philosophy

Property Android iOS
Source model Open source (AOSP); OEM-customised Closed source (Darwin kernel open, rest proprietary)
App distribution Google Play + third-party APKs (sideloading) App Store only; notarisation required; sideloading limited
Kernel Linux + Android kernel patches XNU (Mach + BSD hybrid)
Isolation Linux UID-per-app + SELinux mandatory access control BSD sandbox (Seatbelt) + entitlement-gated capabilities
Privilege model Runtime permissions (dangerous) + install-time Usage permission prompts + entitlements
Root/jailbreak exposure Root access common (OEM unlocking); more fragmented attack surface Jailbreak via kernel exploits; tightly controlled; harder
Secure hardware Titan M (Pixel), vendor TEEs, StrongBox Secure Enclave (T2/A-series); Touch/Face ID co-processor
Update fragmentation High (OEM delays, carrier delays, EOL devices) Low (Apple controls all updates; ~95% on current iOS within year)
Malware ecosystem Significant (Play Store malware; sideloading risk) Rare (enterprise abuse; state-actor jailbreak chains)

Shared Attack Surface

Despite different architectures, Android and iOS share a common application-layer attack surface:

flowchart TD
    APP[Mobile App] --> NET[Network Layer\nTLS certificate pinning\nAPI authentication]
    APP --> STORAGE[Local Storage\nSQLite databases\nSharedPreferences Keychain\nPlaintext secrets]
    APP --> IPC[IPC\nIntents Binder Scheme URLs\nDeep links URL schemes]
    APP --> AUTH[Authentication\nBiometrics PIN\nOAuth tokens session storage]
    APP --> CODE[Code\nReverse engineering\nTampering repackaging\nRuntime hooking]
    APP --> THIRD[Third-party SDKs\nAd analytics SDKs\nData leakage]

OWASP Mobile Top 10 (2024)

Rank Risk Quick Summary
M1 Improper Credential Usage Hardcoded credentials; insecure credential storage
M2 Inadequate Supply Chain Security Third-party SDK vulnerabilities; dependency confusion
M3 Insecure Authentication / Authorization Broken auth flows; improper session handling
M4 Insufficient Input / Output Validation Client-side injection; data exfiltration via unvalidated outputs
M5 Insecure Communication Missing or bypassable TLS; weak cipher suites; no cert pinning
M6 Inadequate Privacy Controls Excessive data collection; PII in logs, crash reports, third-party SDKs
M7 Insufficient Binary Protections No obfuscation; debug builds; repackageable; no integrity checks
M8 Security Misconfiguration Debug flags; overpermissioning; default credentials; cloud backend misconfig
M9 Insecure Data Storage Cleartext in SQLite, SharedPreferences, external storage, logs
M10 Insufficient Cryptography Weak algorithms; hardcoded keys; custom crypto; predictable IVs

Practice Questions

  • Compare the Android and iOS permission models. How does runtime permission handling differ from entitlement-based capabilities?
  • Why is update fragmentation a significant security risk for Android? How does Apple's approach differ?
  • What does a Secure Enclave provide that a standard ARM TrustZone TEE does not?
  • Why is a mobile application with certificate pinning harder to intercept? What tools bypass pinning and how?
  • An Android app stores OAuth refresh tokens in SharedPreferences. What are the risks and what should it do instead?
  • How does sideloading on Android differ from the iOS notarisation requirement? What security tradeoffs does each represent?
