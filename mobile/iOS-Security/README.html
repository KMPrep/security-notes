iOS Security

Category: Mobile Security — iOS Platform Security Model

Overview

iOS has one of the most layered security architectures of any consumer operating system. Apple controls the hardware, silicon, operating system, bootloader, compiler, and app distribution channel end-to-end. This vertical integration allows security controls that are not possible on Android's more fragmented stack. Understanding iOS security is critical for mobile penetration testing, forensics, and designing corporate mobile policies.

iOS Security Architecture — Layers

flowchart TD
    HW[Hardware\nApple Silicon A-series M-series\nSecure Enclave Processor SEP] --> BOOT[Secure Boot Chain\nBoot ROM AP firmware iBoot\nCryptographic signature verification]
    BOOT --> KERN[XNU Kernel\nMach BSD hybrid\nMemory protection entitlements]
    KERN --> SAND[Sandbox Seatbelt\nPer-app MAC policy\nEntitlement-gated syscalls]
    SAND --> APP[App Layer\nSigned code\nRuntime permission prompts]
    APP --> KEYCHAIN[Keychain\nSecure Enclave backed secrets\nData Protection class per item]

Secure Boot Chain

Every stage of iOS boot is cryptographically signed by Apple. A device will not boot if any stage fails verification.

Stage What It Does What It Checks
Boot ROM (iBoot ROM) Hardcoded in silicon; runs first; unmodifiable Verifies LLB/iBoot signature with Apple root cert
iBoot Second-stage bootloader Verifies iOS kernel signature
iOS Kernel XNU kernel loads Kernel Integrity Protection (KIP); KTRR on A10+
iOS Full OS startup Code signing of all processes

Significance: Modifying any bootloader stage requires an Apple-signed update or a silicon-level vulnerability. Jailbreaks that survive reboots (untethered) require a vulnerability in iBoot or earlier.

Secure Enclave (SEP)

The Secure Enclave Processor is a separate ARM co-processor embedded in Apple Silicon (A7+). It has its own boot ROM, encrypted memory, and OS (sepOS). It is isolated from the main Application Processor (AP) — even a fully compromised AP cannot extract keys from the SEP.

What the SEP manages:

Function Detail
Biometric matching Touch ID fingerprint match; Face ID depth map match performed inside SEP
Key derivation Device UID (unique per device, fused at factory; not readable by any software) used for key derivation
Data Protection keys Per-file class encryption keys wrapped by UID-derived key
Apple Pay Payment credential storage and authorisation
Passcode attempts Enforces exponentially increasing delays; limits brute force to ~10 attempts before data wipe
Secure boot Verifies sepOS integrity

Device UID: A 256-bit AES key fused into the silicon at manufacture. Never exposed to software. Used as the root of the key hierarchy for Data Protection. Even Apple cannot decrypt your device without the UID.

Data Protection — File Encryption

iOS uses a per-file encryption scheme called Data Protection. Every file is encrypted with a per-file key, which is wrapped by a class key, which is ultimately derived from the device UID and (for higher protection classes) the user passcode.

Class Name When Key Is Available
NSFileProtectionComplete Class A Only while device is unlocked
NSFileProtectionCompleteUnlessOpen Class B While file is open OR device is unlocked
NSFileProtectionCompleteUntilFirstUserAuthentication Class C After first unlock post-reboot (until next reboot)
NSFileProtectionNone Class D Always available (no passcode key wrapping)

Forensic implication (BFU vs AFU):

  • BFU (Before First Unlock): Only Class D files are accessible. Most app data is Class C or higher — inaccessible.
  • AFU (After First Unlock): Class C keys are derived. Most app data becomes accessible. Law enforcement targets AFU state.

Developer best practice: Use NSFileProtectionComplete (Class A) for sensitive data. Class C (default) is fine for most data but leaks more in BFU forensics.

Code Signing and Entitlements

All code running on a non-jailbroken iOS device must be signed by Apple (App Store or enterprise MDM certificate). The kernel verifies code signature pages at load time.

Entitlements are key-value properties embedded in a signed binary that grant specific capabilities. The entitlement is checked against the provisioning profile at install time and at runtime when the capability is exercised.

Entitlement Grants
com.apple.developer.healthkit Access to HealthKit data
com.apple.security.network.client Network client socket access
keychain-access-groups Specific Keychain access groups
com.apple.developer.nfc.readersession.formats NFC reading
com.apple.private.* Private Apple entitlements — not available to third-party apps

Attack surface: An app cannot claim an entitlement it was not granted. Privilege escalation via entitlement forgery requires a codesigning bypass vulnerability.

iOS App Sandbox

Every iOS app runs in a BSD Seatbelt sandbox. The sandbox policy is based on entitlements. By default, an app can only access:

  • Its own container directory (/var/mobile/Containers/Data/Application/<UUID>/)
  • Resources it was granted via entitlements
  • APIs exposed by the OS within its entitlements

App containers:

App Container/
├── Documents/          # User-visible files; backed up
├── Library/
│   ├── Application Support/   # App data; backed up
│   ├── Caches/                # Ephemeral; not backed up
│   └── Preferences/           # NSUserDefaults; backed up
└── tmp/                # Temporary; not backed up

No world-readable storage: Unlike Android's external storage, iOS apps cannot write to or read from other apps' containers without explicit sharing mechanisms (shared Keychain groups, App Groups, or Sharing Extensions).

Keychain

The iOS Keychain is the secure secret store. Items are encrypted and optionally bound to:

  • Device: Cannot be restored to another device
  • Passcode: Accessible only when device is unlocked (tied to Data Protection class)
  • Biometric: Accessible only after biometric (or passcode) verification via LAContext
// Store a secret accessible only while device is unlocked
let query: [String: Any] = [
    kSecClass as String: kSecClassGenericPassword,
    kSecAttrAccount as String: "apiToken",
    kSecValueData as String: tokenData,
    kSecAttrAccessible as String: kSecAttrAccessibleWhenUnlocked  // Class A
]
SecItemAdd(query as CFDictionary, nil)

Keychain sharing: Apps with the same keychain-access-groups entitlement and same team ID can share Keychain items — useful for SSO between apps in a suite.

Common mistake: Using kSecAttrAccessibleAlways (Class D) for tokens — accessible even while device is locked, included in unencrypted backups.

iCloud and Backup Security

Backup Type Encryption Where
iTunes/Finder backup (unencrypted) Not encrypted Local Mac/PC — highly sensitive
iTunes/Finder backup (encrypted) User password + device UID Local Mac/PC — safer
iCloud backup AES-256; Apple holds key Apple servers — subpoena risk
iCloud Advanced Data Protection (E2E) E2E encrypted; Apple has no key Apple servers — forensically dark

iCloud Advanced Data Protection (ADP): Introduced in iOS 16.2. When enabled, iCloud backups, Photos, iCloud Drive, Messages in iCloud, and more are end-to-end encrypted. Apple cannot provide the data to law enforcement even with a valid order. Requires setting up recovery contacts or a recovery key.

Jailbreaks

A jailbreak exploits one or more iOS vulnerabilities to:

  1. Bypass code signing (run unsigned code)
  2. Gain root/kernel access
  3. Install Cydia (package manager) or equivalent
Type Persistence Requirements
Tethered Requires re-jailbreak on every reboot via USB Older; rare
Semi-tethered Boots normally but unjailbroken until exploit is re-run on device Common
Untethered Survives reboot; jailbreak fully persistent Hardest; requires iBoot or earlier vuln
Rootless Does not modify root filesystem; uses writable areas Modern trend (Dopamine, palera1n)

Notable jailbreaks:

  • checkm8: Hardware (BootROM) exploit affecting A5–A11 chips. Unpatchable by software update. Used by Checkra1n.
  • palera1n: checkm8-based; targets A9–A11; rootless by default.

Security implications: A jailbroken device has:

  • Bypassed code signing — arbitrary code runs
  • Bypassed sandbox — apps read each other's data
  • Potential SSH server (Dropbear) exposed at default password
  • All Data Protection keys accessible (user data decryptable)
