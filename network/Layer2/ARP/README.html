ARP — Address Resolution Protocol

OSI Layer: 2-3 boundary (typically treated as Layer 2)
RFC: RFC 826

Overview

ARP maps an IPv4 address on the local subnet to a Layer 2 MAC address so frames can be delivered on the local segment. When a host wants to send an IPv4 packet to a neighbor or default gateway, it must first know the target MAC address. ARP fills that gap by issuing a broadcast request and receiving a unicast reply from the host that owns the IP. Because ARP is unauthenticated and local-link scoped, it is operationally simple but security-sensitive.

How It Works

Core Mechanics

If a host does not already have an IP-to-MAC mapping in its ARP cache, it broadcasts an ARP request asking which device owns a specific IPv4 address. The host with that address replies with its MAC address, and the sender stores the mapping in its ARP cache for future use. This cache ages out over time or is replaced when entries change.

ARP works only within the local broadcast domain. For traffic to remote networks, the host usually resolves the MAC address of its default gateway rather than the ultimate destination. Gratuitous ARP messages can also update caches or announce changes, which is operationally useful for failover but also creates room for spoofing abuse.

Top 3 Critical Mitigations

1. Use Switch-Level ARP Protections

ARP is vulnerable mainly because hosts trust replies by default. Features such as Dynamic ARP Inspection and DHCP snooping integration let switches validate claims before forwarding them. The trade-off is configuration complexity, but it is one of the strongest practical defenses.

2. Reduce Broadcast-Domain Blast Radius

The smaller the local segment, the smaller the population exposed to ARP abuse. Good segmentation, VLAN boundaries, and access-layer policy make ARP problems more containable. The trade-off is more network design work, but it helps both operations and security.

3. Treat Local-Link Protocols as Security-Relevant

Teams often focus on routed or internet-facing protocols and ignore local-link trust. ARP proves that local assumptions can still be exploited for interception and lateral movement. Monitoring and access control on local segments matter.

Practice Questions

Tier 1 — Fundamentals

  1. What does ARP do? It maps an IPv4 address to a MAC address on the local network. This is asked to confirm basic function.
  2. Why is ARP needed for local IPv4 delivery? Ethernet frames need MAC addresses, while the application and IP stack reason in IP addresses. This is asked to test layering understanding.
  3. Does ARP cross routers? No, it is local to the broadcast domain. This is asked to test scope awareness.
  4. What is an ARP cache? It is a temporary local table of IP-to-MAC mappings. This is asked to test operational familiarity.
  5. Why is ARP spoofing possible? ARP has no built-in authentication. This is asked to connect protocol simplicity to attack surface.

Tier 2 — System Design and Trade-offs

  1. How would you defend a campus network against ARP poisoning? I would use DAI, DHCP snooping, segmentation, and access controls on the edge. This is asked to test practical defensive design.
  2. Why can gratuitous ARP be both useful and risky? It helps failover and cache updates, but it can also be abused to poison caches. This is asked to test nuanced reasoning.
  3. How would you troubleshoot intermittent ARP-related connectivity failures? I would inspect caches, duplicate IPs, switch telemetry, and failover behavior. This is asked to test debugging structure.
  4. What changes in IPv6 instead of ARP? IPv6 uses Neighbor Discovery Protocol rather than ARP. This is asked to test stack evolution knowledge.
  5. Why is ARP still worth learning in cloud-heavy environments? Many virtualized networks and access layers still emulate or depend on ARP-like behavior. This is asked to test practical relevance.

Tier 3 — Expert and Deep-Dive

  1. Why are local-link protocols often underestimated in threat models? They look simple and internal, but they directly influence routing and trust on the local segment. This is asked to test threat-model maturity.
  2. How would you detect ARP poisoning at scale? Correlate unexpected IP-to-MAC changes, switch validation events, and unusual gateway claim behavior. This is asked to test detection design.
  3. What are the operational downsides of static ARP everywhere? It reduces flexibility, increases drift risk, and does not scale operationally. This is asked to test practical judgment.
  4. How does ARP illustrate the difference between protocol correctness and secure deployment? It works exactly as designed but still enables abuse because trust was not authenticated. This is asked to test deeper protocol reasoning.
  5. What does mature ARP hygiene look like? Validated edge switching, small broadcast domains, endpoint trust controls, and anomaly monitoring. This is asked to test operational design depth.

Key Takeaways

  • ARP maps IPv4 addresses to MAC addresses on the local segment.
  • It is local-only and fundamental to Ethernet-based IPv4 delivery.
  • Lack of authentication makes spoofing a major risk.
  • Switch-level protections are among the best mitigations.
  • Smaller broadcast domains reduce blast radius.
  • ARP problems are both operational and security issues.
  • Local-link protocols deserve real threat modeling.
