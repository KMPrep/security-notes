BGP — Border Gateway Protocol

OSI Layer: 3 — Network
RFC: RFC 4271, RFC 7454, RFC 9234

Overview

BGP is the interdomain routing protocol that distributes reachability information between autonomous systems. It does not choose paths based only on shortest distance; instead, it applies policy using attributes such as AS path, local preference, MED, and communities. BGP is the control plane that helps hold the global internet together and also powers many enterprise and cloud edge connections. Because BGP trusts propagated routing information unless operators add controls, it is both operationally critical and vulnerable to high-impact mistakes and abuse.

How It Works

Core Mechanics

BGP speakers establish a TCP session, usually on port 179, and exchange OPEN, KEEPALIVE, UPDATE, and NOTIFICATION messages. An UPDATE advertises or withdraws prefixes along with attributes that influence path selection. The receiving router stores candidate paths in the BGP table and selects a best path according to local policy.

iBGP is used within an AS to distribute externally learned routes, while eBGP is used between different ASes. Route selection is policy-driven, which is why BGP is so useful for traffic engineering and so dangerous when policy is wrong. Features such as prefix filtering, route validation, RPKI, max-prefix limits, and route reflectors exist to keep the system scalable and safer, but the protocol’s baseline trust model is still relatively weak.

Top 3 Critical Mitigations

1. Filter Routes Aggressively and Deliberately

Many major BGP incidents are not protocol exploits so much as policy failures. Prefix filters, AS-path expectations, and tightly scoped import/export rules are the first line of defense. The trade-off is maintenance overhead, but weak filtering is too dangerous at internet edge boundaries.

2. Use RPKI and Origin Validation

RPKI does not solve every BGP security problem, but it materially improves defense against invalid origin announcements. It raises the cost of simple prefix hijacking and helps operators reject obviously bad route advertisements. The trade-off is deployment and operational complexity, but the security value is real.

3. Monitor Control-Plane Changes Continuously

BGP problems can propagate far faster than humans can reason about them from scratch. Continuous visibility into route changes, prefix reachability, and peer health is essential for rapid response. The trade-off is tooling and telemetry investment, but it is fundamental for production internet operations.

Practice Questions

Tier 1 — Fundamentals

  1. What problem does BGP solve? It exchanges reachability information between autonomous systems. This is asked to test core purpose.
  2. Why is BGP called policy-based routing? Path choice depends on operator-defined attributes and preferences, not just shortest distance. This is asked to test conceptual understanding.
  3. What is the difference between iBGP and eBGP? iBGP operates within an AS, while eBGP operates between ASes. This is asked to test basic topology knowledge.
  4. Why does BGP use TCP? It uses TCP for reliable ordered session transport on port 179. This is asked to test protocol layering knowledge.
  5. What is a prefix hijack? It is when an AS advertises a prefix it does not legitimately control. This is asked to test security awareness.

Tier 2 — System Design and Trade-offs

  1. How would you design safe BGP peering for a cloud edge? I would use strict filters, max-prefix limits, monitoring, and validated route policy for each peer. This is asked to test operational design.
  2. Why are route leaks so common? BGP is flexible and policy-rich, so human error in export or import rules has outsized effects. This is asked to test realism about operations.
  3. How would you investigate sudden global reachability loss for your prefix? Check local announcements, RPKI status, peer sessions, looking glass outputs, and route collectors. This is asked to test troubleshooting depth.
  4. What are the trade-offs of multi-homing with BGP? It improves resilience and traffic control but adds policy complexity and operational burden. This is asked to test systems trade-offs.
  5. Why are communities useful and dangerous at the same time? They help express routing policy compactly but can cause surprises if interpreted inconsistently. This is asked to test nuanced policy thinking.

Tier 3 — Expert and Deep-Dive

  1. Why does the internet still depend on a protocol with such a weak native trust model? BGP is globally interoperable and entrenched, so the ecosystem layered operational controls around it rather than replacing it outright. This is asked to test ecosystem-level reasoning.
  2. How would you explain the difference between origin validation and full path validation? Origin validation checks who is allowed to originate a prefix, while full path validation would also validate every AS in the path. This is asked to test deeper routing-security knowledge.
  3. What are the hardest BGP problems in hyperscale or carrier networks? Policy complexity, convergence speed, route scale, and misconfiguration blast radius are among the hardest. This is asked to test production depth.
  4. How would you threat-model a company’s BGP edge? I would model peers, prefix ownership, route policy, control-plane access, and monitoring as critical trust boundaries. This is asked to test structured security thinking.
  5. What does mature BGP hygiene look like? Tight filters, validated origins, route monitoring, diverse peers, and tested incident procedures. This is asked to test platform maturity.

Key Takeaways

  • BGP is the policy-driven routing protocol between autonomous systems.
  • It is operationally essential and security-sensitive.
  • Most major failures come from policy mistakes and weak trust controls.
  • Filtering and RPKI are two of the most important defenses.
  • Monitoring route changes is critical for fast response.
  • Multi-homing improves resilience but increases complexity.
  • BGP is a control-plane protocol where small mistakes can have huge blast radius.
