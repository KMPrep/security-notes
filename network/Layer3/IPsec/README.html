IPsec - Internet Protocol Security

OSI Layer: 3 - Network
RFC: RFC 4301, RFC 4302, RFC 4303, RFC 7296

Overview

IPsec is a network-layer security suite that protects IP packets with authentication, integrity, anti-replay protection, and usually confidentiality. It is widely used for site-to-site VPNs, remote-access VPNs, cloud tunnel connectivity, and secure host-to-host traffic. Because it operates below the application layer, IPsec can protect many upper-layer protocols without changing the applications themselves.

In practice, modern deployments rely mostly on ESP with IKEv2. AH exists in the standard but is uncommon in production because it does not provide confidentiality and interacts poorly with NAT. Tunnel mode is the dominant mode for VPN gateways, while transport mode is more common for direct host protection.

Core Mechanics

IPsec has three major building blocks:

Component Purpose
IKEv2 Negotiates peers, authenticates them, and establishes keys
ESP Encrypts payloads and provides integrity/authentication
Security Associations (SAs) Define algorithms, keys, selectors, and lifetime

Operational concepts:

Concept Why it matters
Transport mode Protects the IP payload while keeping the original IP header
Tunnel mode Encapsulates the full original packet inside a new IP packet
SPI Identifies the correct inbound security association
Anti-replay window Rejects duplicate or replayed packets
NAT-T Encapsulates IPsec traffic in UDP to survive NAT devices

Step-by-Step Flow

1. IKEv2 control-plane setup

sequenceDiagram
	participant A as Gateway A
	participant B as Gateway B
	A->>B: IKE_SA_INIT proposals and nonce
	B-->>A: IKE_SA_INIT selected proposal and nonce
	A->>B: IKE_AUTH identity, auth data, child SA request
	B-->>A: IKE_AUTH identity, auth data, child SA response
	A->>B: Child SA established

2. ESP tunnel-mode packet protection

flowchart TD
	A[Original IP packet] --> B[Match traffic selector policy]
	B --> C[Apply ESP using outbound SA]
	C --> D[Encrypt payload and add ESP headers]
	D --> E[Wrap in new outer IP header]
	E --> F[Send through network]
	F --> G[Peer decrypts and validates packet]

3. Inbound IPsec processing

flowchart TD
	A[Receive protected packet] --> B[Locate SA from SPI]
	B --> C[Check sequence number and anti-replay window]
	C --> D[Verify integrity and decrypt ESP payload]
	D --> E[Check traffic selectors and lifetime]
	E --> F[Deliver recovered packet]

What Can Go Wrong

Failure mode What happens Security impact
Weak IKE policy Legacy ciphers or weak DH groups are accepted Downgrade and weaker confidentiality
PSK reuse Same shared secret appears across many tunnels Broad compromise if one secret leaks
Rekey failures SAs expire or flap during renewal Intermittent outages and blind spots
Mis-scoped selectors Wrong traffic is encrypted or bypasses tunnel Data exposure or routing confusion
NAT-T edge cases Middleboxes break or mis-handle encapsulated traffic Tunnel instability and packet loss
Operational blind spots Encrypted transport hides application details Reduced inspection and harder incident response

Mitigation Controls

Control How it helps
Use IKEv2 with strong cipher suites Removes legacy negotiation paths
Prefer certificate-based auth for scale Avoids weak PSK hygiene and simplifies rotation
Enforce PFS and sane SA lifetimes Limits blast radius of key compromise
Monitor tunnel health and rekey events Detects instability before it becomes an outage
Define explicit traffic selectors Prevents accidental bypass or over-broad encryption
Preserve metadata visibility where needed Maintain routing, logging, and security telemetry around the tunnel

Top 3 Critical Mitigations

  1. Standardize on IKEv2 with strong algorithms and remove legacy negotiation options.
  2. Use certificates or uniquely rotated PSKs with PFS-enabled child SAs.
  3. Continuously monitor tunnel establishment, rekey behavior, selector scope, and traffic drift.

Key Takeaways

  • IPsec secures IP traffic below the application layer, making it powerful for VPNs and transparent host protection.
  • Real-world reliability depends as much on IKE policy, selectors, NAT traversal, and monitoring as on cryptography.
  • Strong IKEv2 configuration, disciplined authentication, and tunnel observability are the critical controls.
