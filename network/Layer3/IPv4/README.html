IPv4 — Internet Protocol version 4

OSI Layer: 3 — Network
RFC: RFC 791, RFC 950, RFC 1812, RFC 4632

Overview

IPv4 is the core network-layer protocol that provides addressing and packet delivery across interconnected networks. It uses 32-bit addresses, typically written in dotted-decimal form, and relies on routers to forward packets toward their destinations based on longest-prefix-match routing. IPv4 is connectionless and best-effort, meaning it does not guarantee delivery, ordering, or integrity beyond a header checksum. Despite address exhaustion and the existence of IPv6, IPv4 remains deeply embedded in enterprise and internet infrastructure.

How It Works

Core Mechanics

Each IPv4 packet carries source and destination addresses plus metadata such as TTL, protocol number, fragmentation fields, and Type of Service or DSCP-related bits. Routers decrement TTL at each hop to prevent loops from persisting forever. The protocol field identifies the payload type, such as TCP, UDP, or ICMP.

IPv4 addressing is heavily tied to subnetting and routing policy. Hosts determine whether a destination is local or remote using their subnet mask, then either ARP for a local destination or send the packet to a default gateway. Historically IPv4 supported fragmentation in transit, but operationally fragmentation often causes reliability and security issues. Widespread use of NAT extended IPv4’s lifetime by allowing many private addresses to share fewer public ones, but that introduced its own complexity and statefulness.

Top 3 Critical Mitigations

1. Enforce Anti-Spoofing at Network Boundaries

IPv4 makes source spoofing easy unless the network explicitly blocks it. Ingress and egress filtering reduce abuse such as reflection attacks and false attribution. The trade-off is stricter policy management, but it is foundational internet hygiene.

2. Treat NAT as a Compensating Convenience, Not a Security Model

NAT extended IPv4’s lifespan, but it also complicates protocols, troubleshooting, and trust assumptions. It should not be confused with strong security, and systems should be designed with explicit identity and access controls above it. The trade-off is architectural complexity, but that is reality in IPv4-heavy environments.

3. Invest in IP Address Management and Routing Discipline

Many IPv4 incidents are configuration and allocation failures rather than protocol failures. Good IPAM, route review, and prefix hygiene reduce outages and make future transitions easier. The trade-off is more operational rigor, but the payoff is stability.

Practice Questions

Tier 1 — Fundamentals

  1. What does IPv4 provide? Addressing and best-effort packet delivery across routed networks. This is asked to confirm network-layer understanding.
  2. What does TTL do in IPv4? It limits packet lifetime and helps stop infinite routing loops. This is asked to test header knowledge.
  3. Why does IPv4 need subnet masks? They determine which destinations are local versus remote. This is asked to test addressing fundamentals.
  4. What problem did NAT help delay? It delayed public IPv4 address exhaustion. This is asked to connect protocol limits to operational responses.
  5. Does IPv4 guarantee delivery? No, it is a best-effort packet protocol. This is asked to test correct layering assumptions.

Tier 2 — System Design and Trade-offs

  1. How would you design an IPv4 address plan for a large enterprise? I would reserve growth space, avoid overlap, document ownership, and align subnets to routing and security boundaries. This is asked to test operational design.
  2. Why can NAT break applications? Some applications assume end-to-end address stability or embed address information that translation interferes with. This is asked to test practical networking knowledge.
  3. How would you investigate intermittent IPv4 reachability? Check routing, ARP, MTU, NAT state, ACLs, and TTL-related path behavior. This is asked to test troubleshooting structure.
  4. Why is source IP a weak security signal? It can be spoofed or shared behind translation, and it does not represent strong identity. This is asked to test security reasoning.
  5. How would you prioritize IPv6 transition in an IPv4-heavy organization? Start where scale, overlap, or public growth pressure is highest while keeping IPv4 stable. This is asked to test modernization judgment.

Tier 3 — Expert and Deep-Dive

  1. Why does IPv4 still dominate even after exhaustion? NAT, inertia, compatibility, and huge installed infrastructure delayed a clean transition. This is asked to test ecosystem realism.
  2. What are the most dangerous operational illusions around IPv4? Believing NAT is security, trusting source IP too much, and underestimating address-plan debt are common ones. This is asked to test mature judgment.
  3. How would you explain fragmentation problems to an application engineer? Large packets can be dropped or mishandled in the path, so application behavior may fail mysteriously unless MTU assumptions are sound. This is asked to test cross-layer communication.
  4. Why is IPAM often a security problem and not just a networking problem? Bad address ownership and overlap create blind spots, policy mistakes, and incident confusion. This is asked to test operational security perspective.
  5. What does good IPv4 hygiene look like at scale? Anti-spoofing, disciplined IPAM, clean routing policy, documented NAT behavior, and minimized overlap. This is asked to test platform-level depth.

Key Takeaways

  • IPv4 is the enduring best-effort network-layer protocol of today’s internet.
  • Addressing, subnetting, and routing are central to its behavior.
  • NAT extended IPv4 life but introduced complexity.
  • Anti-spoofing and route hygiene are critical controls.
  • MTU and fragmentation issues remain important operational pitfalls.
  • Source IP is not strong identity.
  • Strong IPAM and routing discipline matter as much as protocol knowledge.
