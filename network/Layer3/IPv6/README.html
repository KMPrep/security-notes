IPv6 — Internet Protocol version 6

OSI Layer: 3 — Network
RFC: RFC 8200, RFC 4861, RFC 4862

Overview

IPv6 is the modern successor to IPv4, designed to provide vastly larger address space, cleaner header processing, and better support for large-scale routed networks. It uses 128-bit addresses, a fixed base header, extension headers, and a different local-link discovery model built around ICMPv6 and Neighbor Discovery. IPv6 is intended to restore true end-to-end addressing without dependence on ubiquitous NAT. In practice, operating IPv6 securely still requires explicit routing, filtering, and identity design rather than assuming the larger space solves everything automatically.

How It Works

Core Mechanics

An IPv6 packet carries source and destination addresses, traffic class, flow label, hop limit, next-header information, and payload length. Routers forward packets based on prefix routes, decrementing hop limit at each step. Unlike IPv4, routers do not fragment packets in transit; endpoints are expected to handle packet sizing appropriately, often with Path MTU Discovery.

Addressing in IPv6 is more expressive and hierarchical than IPv4. Hosts may receive addresses via SLAAC, DHCPv6, or static configuration, and link-local addresses are fundamental to local communication. Neighbor Discovery replaces ARP and uses ICMPv6 messages for router discovery, neighbor resolution, and reachability checks. The protocol design removes some IPv4-era complexity while adding new operational dependencies around ICMPv6, addressing policy, and transition coexistence.

Top 3 Critical Mitigations

1. Treat IPv6 as Real Production Traffic, Not an Optional Side Path

One of the biggest IPv6 risks is that organizations enable it without applying the same security and operational rigor they apply to IPv4. Firewalls, logging, IAM assumptions, and monitoring must all include IPv6 deliberately. The trade-off is more policy work, but the alternative is a quiet blind spot.

2. Protect the Local Control Plane

IPv6 depends heavily on ICMPv6, router advertisements, and Neighbor Discovery for normal operation. That makes local-link protections such as RA Guard and correct ICMPv6 policy important operational and security controls. The trade-off is configuration care, but without it hosts can be redirected or broken easily.

3. Plan Addressing and Dual-Stack Transition Explicitly

IPv6 transition failures are often design and policy failures rather than protocol failures. Clear prefix planning, consistent DNS, and documented coexistence strategy reduce confusion and split-brain behavior. The trade-off is upfront planning effort, but it prevents years of operational drift.

Practice Questions

Tier 1 — Fundamentals

  1. Why was IPv6 created? To overcome IPv4 address limitations and improve long-term internet-scale addressing and routing. This is asked to test basic motivation.
  2. What replaces ARP in IPv6? Neighbor Discovery using ICMPv6 messages. This is asked to test protocol evolution knowledge.
  3. Does IPv6 eliminate the need for routing policy and filtering? No, it removes some scarcity problems but not security or operational discipline. This is asked to test realism.
  4. Why is ICMPv6 more critical than many teams expect? It supports essential control functions such as neighbor discovery and PMTUD. This is asked to test operational understanding.
  5. What is a link-local address used for? Local communication and control on the same segment. This is asked to test basic stack knowledge.

Tier 2 — System Design and Trade-offs

  1. How would you design an enterprise IPv6 rollout? I would plan addressing hierarchy, security policy parity, DNS behavior, and dual-stack transition stages. This is asked to test migration thinking.
  2. What are the trade-offs between SLAAC and DHCPv6? SLAAC is simpler for autonomous addressing, while DHCPv6 can give more central control depending on the environment. This is asked to test design choices.
  3. How would you troubleshoot a dual-stack service that works on IPv4 but not IPv6? Check AAAA records, routing, ICMPv6 policy, neighbor discovery, and client path behavior. This is asked to test practical debugging.
  4. Why can IPv6 accidentally expand exposure? Services may listen on IPv6 with weaker filtering or monitoring than on IPv4. This is asked to test security policy maturity.
  5. What are the hardest operational parts of dual-stack? Two address families, two sets of policies, and inconsistent client path selection create complexity. This is asked to test real-world depth.

Tier 3 — Expert and Deep-Dive

  1. Why is IPv6 not automatically more secure than IPv4? Security depends on policy, identity, and operations, not just address size or protocol generation. This is asked to test nuanced reasoning.
  2. How would you explain the risk of overblocking ICMPv6 to a security team? It can break core network functions rather than just remove diagnostics. This is asked to test cross-team communication.
  3. What are the most common IPv6 blind spots in enterprises? Logging gaps, firewall policy asymmetry, and weak local-link protections are common. This is asked to test operational security perspective.
  4. How would you think about identity in an IPv6 world without pervasive NAT? Identity should still come from authenticated higher layers, not just from source addresses. This is asked to test architectural judgment.
  5. What does mature IPv6 readiness look like? Strong addressing design, parity with IPv4 security controls, reliable DNS, and tested dual-stack operations. This is asked to test platform maturity.

Key Takeaways

  • IPv6 provides large-scale addressing and cleaner packet handling.
  • It depends heavily on ICMPv6 and neighbor discovery for normal operation.
  • IPv6 security requires explicit policy, not assumptions.
  • Dual-stack is often the hardest operational phase.
  • Router advertisements and NDP need local-link protections.
  • DNS and reachability parity matter for real deployments.
  • IPv6 should be treated as a first-class production protocol.
