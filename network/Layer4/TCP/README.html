TCP — Transmission Control Protocol

OSI Layer: 4 — Transport
RFC: RFC 9293, RFC 5681, RFC 6298

Overview

TCP is a connection-oriented transport protocol designed to provide reliable, ordered byte-stream delivery between endpoints. It establishes state on both sides, tracks sequence and acknowledgment numbers, retransmits missing data, and adjusts sending behavior using flow and congestion control. This makes TCP suitable for applications where correctness and in-order delivery matter more than minimal latency. HTTP, HTTPS, SSH, SMTP, and many file transfer protocols depend on TCP for these guarantees.

How It Works

Core Mechanics

TCP begins with a three-way handshake that synchronizes sequence numbers and confirms that both peers are ready to exchange data. Once established, each side sends segments with sequence numbers and acknowledges received bytes from the other side. Sliding windows let each peer advertise how much more data it can receive, which provides flow control.

Reliability comes from retransmission timers, duplicate acknowledgment handling, fast retransmit behavior, and conservative timeout calculation. Congestion control is separate from pure reliability: TCP grows or shrinks its sending rate based on perceived network conditions so it does not overwhelm the path. Connection teardown normally uses a FIN-based close handshake, though resets can terminate the session abruptly.

Top 3 Critical Mitigations

1. Protect the Handshake Path

TCP servers can be forced to allocate state before a client proves it can complete the handshake. SYN cookies, backlog tuning, and upstream filtering reduce the impact of this asymmetry. The trade-off is that aggressive defenses can affect uncommon TCP features, but protecting availability is usually the priority.

2. Respect Congestion and Backpressure

TCP already tries to behave fairly on shared networks, but applications can still overload queues and peers. Good systems align write rates with what receivers and networks can sustain. The trade-off is lower raw burst throughput, but much better stability.

3. Use Encryption Above TCP

TCP gives transport reliability, not confidentiality or authenticity. Sensitive traffic should run over TLS or another secure channel so session integrity does not depend only on path trust. The trade-off is handshake and CPU cost, but modern systems usually accept that easily.

Practice Questions

Tier 1 — Fundamentals

  1. Why does TCP need a handshake? It establishes shared state and sequence number expectations before data is exchanged. This is asked to confirm protocol basics.
  2. What does TCP guarantee that UDP does not? Reliable, ordered byte-stream delivery with retransmission and flow control. This is asked to test transport-layer understanding.
  3. What is an acknowledgment in TCP? It tells the sender which bytes have been received so far. This is asked to test sequence-space literacy.
  4. What is the difference between flow control and congestion control? Flow control protects the receiver, while congestion control protects the network path. This is asked to test conceptual precision.
  5. How does TCP close a connection cleanly? Normally with FIN and ACK exchange from both sides. This is asked because teardown matters in real systems.

Tier 2 — System Design and Trade-offs

  1. Why can HTTP still feel slow over TCP even with keep-alive? Loss and in-order semantics can stall application progress, especially before multiplexing or when one stream blocks others. This is asked to connect transport behavior to app latency.
  2. How would you defend a TCP service against SYN floods? Use SYN cookies, tune backlog, rate-limit at the edge, and monitor handshake failure ratios. This is asked to test defensive depth.
  3. What kinds of applications are a bad fit for TCP? Those that prefer timeliness over perfect ordering, such as real-time media or some gaming traffic. This is asked to test protocol selection judgment.
  4. How would you troubleshoot intermittent TCP stalls? Check RTT, loss, retransmits, MSS, PMTUD behavior, and middlebox state. This is asked to test debugging process.
  5. Why is TCP throughput on long fat networks tricky? Window sizing, RTT, and loss recovery all influence how much data can stay in flight. This is asked to test performance reasoning.

Tier 3 — Expert and Deep-Dive

  1. Why is TCP a byte stream and not a message protocol? It abstracts a continuous ordered stream, leaving message boundaries to higher layers. This is asked to test correctness about APIs and wire semantics.
  2. How does TCP become a liability for multiplexed modern applications? Head-of-line blocking and shared loss recovery can stall unrelated logical streams. This is asked to set up comparison with QUIC and HTTP/3.
  3. What are the hardest operational problems in large TCP fleets? Handshake abuse, long-tail latency under loss, idle state management, and inconsistent middlebox behavior are all common. This is asked to test production maturity.
  4. How would you explain why retransmission improves correctness but can worsen user experience? It hides loss from the application but increases delay when timeliness matters more than perfect recovery. This is asked to test trade-off clarity.
  5. What does TCP not solve for applications? It does not provide encryption, authentication, business-level idempotency, or message framing. This is asked to separate transport from application responsibilities.

Key Takeaways

  • TCP is a reliable ordered byte-stream transport.
  • Handshake, acknowledgments, and windows define its core behavior.
  • Flow control and congestion control solve different problems.
  • TCP can suffer under loss because ordering is strict.
  • Availability defenses often focus on handshake and state exhaustion.
  • Sensitive traffic needs TLS or equivalent on top of TCP.
  • Good application design must account for TCP latency and buffering behavior.
