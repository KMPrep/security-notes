RPC — Remote Procedure Call

OSI Layer: 5 — Session (also spans Layers 6 and 7 depending on implementation)
RFC: RFC 5531 (ONC RPC), RFC 1057

Overview

RPC is a communication model and protocol family that lets a client invoke operations on a remote system as though they were local function calls. It hides serialization, transport, and reply handling behind a procedure-oriented abstraction. Many distributed systems, enterprise middleware stacks, and operating-system services use RPC ideas even when they do not use a single universal wire format. Because RPC can make remote work look deceptively local, it is powerful but operationally and security-wise easy to misuse.

How It Works

Core Mechanics

An RPC client constructs a request that names a remote procedure and includes serialized arguments. The request is transported to the server, which deserializes the call, dispatches it to the correct handler, executes the procedure, and serializes the result or error back to the client. The client runtime then turns that response into something that looks like a normal return value or exception.

RPC frameworks differ in transport choice, schema definition, discovery, and security model. Some use HTTP-based transports, some use custom transports, and some support synchronous or streaming behavior. But the core pattern remains the same: hide networking behind a call interface. That abstraction is productive, but it can also obscure latency, partial failure, retry behavior, and authorization boundaries if engineers forget that the work is remote.

Top 3 Critical Mitigations

1. Make Remote Failure Explicit in Design

RPC is dangerous when developers forget that a remote procedure call is still a network operation. Timeouts, retries, idempotency, and partial failure handling must be designed explicitly. The trade-off is more upfront complexity, but the result is far more reliable systems.

2. Enforce Per-Method Authentication and Authorization

RPC often exposes powerful internal operations. Strong per-method authorization prevents the common mistake of treating internal network reachability as sufficient trust. The trade-off is more policy management, but it sharply reduces lateral-movement risk.

3. Use Safe Serialization and Version Governance

Serialization formats and schema evolution are frequent sources of both security bugs and reliability problems. Strict decoding, schema validation, and compatibility policy keep clients and servers from drifting into unsafe or broken states. The trade-off is slower uncontrolled change, but better long-term stability.

Practice Questions

Tier 1 — Fundamentals

  1. What problem does RPC try to solve? It lets remote operations look like local procedure calls to simplify distributed programming. This is asked to confirm conceptual understanding.
  2. Why can RPC be misleading to developers? It hides network latency and failure behind a familiar function-call interface. This is asked because that misunderstanding causes real outages.
  3. What does serialization do in RPC? It turns arguments and return values into a transferable format. This is asked to test core mechanics.
  4. What is the difference between a local function call and an RPC call? The RPC call crosses a network and can fail for distributed-systems reasons even if code is correct. This is asked to test precision.
  5. Why is authorization important for RPC methods? Methods often expose powerful internal actions and state changes. This is asked to connect API surface to security impact.

Tier 2 — System Design and Trade-offs

  1. How would you design an RPC API to be safe under retries? Make operations idempotent where possible, define retry semantics explicitly, and use request identifiers for de-duplication. This is asked to test distributed-systems thinking.
  2. What makes RPC attractive compared with raw message passing? It gives a cleaner programming model, strong schemas in some frameworks, and consistent tooling. This is asked to test trade-off reasoning.
  3. How would you investigate a high-latency RPC path? I would check deadlines, fan-out depth, serialization cost, downstream saturation, and network behavior using traces and logs. This is asked to test debugging maturity.
  4. Why can internal RPC traffic still need strong security? Internal networks are not inherently safe, and compromised services can abuse trusted east-west calls. This is asked to test zero-trust thinking.
  5. What is the main operational challenge of versioning RPC services? Keeping clients and servers compatible while changing schemas and behavior safely. This is asked to test lifecycle awareness.

Tier 3 — Expert and Deep-Dive

  1. Why is “make it look local” both the power and the danger of RPC? It improves developer ergonomics but obscures network cost, retries, and trust boundaries. This is asked to test architectural judgment.
  2. How would you explain the relationship between RPC and microservice failure amplification? Small call chains can become deep dependency trees where one slow or failing service degrades many others. This is asked to test systems reasoning.
  3. What are the main security risks of rich serialization formats in RPC? Unsafe deserialization, parser complexity, and payload abuse can turn convenience into an exploit surface. This is asked to test secure-design thinking.
  4. When is RPC the wrong abstraction? It is a poor fit when asynchronous workflows, durable messaging, or explicit event models are more appropriate than synchronous request-response calls. This is asked to test tool-selection maturity.
  5. What does a mature RPC platform look like? Strong identity, mTLS, tracing, schema governance, clear retry contracts, and explicit deadlines are part of the baseline. This is asked to test platform-level depth.

Key Takeaways

  • RPC hides network communication behind a procedure-call abstraction.
  • That abstraction is useful but can obscure latency and failure.
  • Authorization and serialization safety are critical.
  • Retry semantics and idempotency must be designed deliberately.
  • Internal RPC should not be assumed safe just because it is internal.
  • Tracing and schema governance are major operational needs.
  • Mature RPC systems make remote cost and trust visible again.
