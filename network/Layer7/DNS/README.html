DNS — Domain Name System

OSI Layer: 7 — Application
RFC: RFC 1034, RFC 1035, RFC 4033-4035 (DNSSEC), RFC 7766 (DNS over TCP)

Overview

DNS maps human-readable names to data that hosts and applications need, most commonly IP addresses. It is a distributed and hierarchical system built around recursive resolvers, root servers, TLD servers, and authoritative name servers. Most lookups use UDP port 53 for low latency, with TCP used for large answers, retries, and zone transfers. DNS also carries operational metadata such as MX, TXT, NS, PTR, AAAA, and CNAME records, which makes it both foundational and security-sensitive.

How It Works

Core Mechanics

A client usually asks a recursive resolver for a name such as www.example.com. If the resolver does not already have a cached answer, it performs iterative queries: first to a root server, then to the relevant TLD server, and finally to the authoritative server for the zone. The resolver caches the final answer for the duration allowed by the record TTL so later lookups can be answered faster.

DNS messages contain a header, question section, answer section, authority section, and additional records. The header includes flags such as recursion desired and recursion available, plus an ID used to match responses to requests. Record types are central to DNS behavior: A and AAAA map names to IPv4 and IPv6 addresses, MX routes mail, NS identifies authoritative servers, PTR supports reverse lookup, and TXT often carries policy or ownership data. DNSSEC adds signatures and chains of trust so resolvers can validate that answers were not forged.

DNSSEC — How It Works

DNSSEC (DNS Security Extensions) adds data origin authentication and integrity protection to DNS. It does not encrypt queries or provide confidentiality — it only guarantees that the answer came from the legitimate zone and was not tampered with in transit.

Key Concepts

Concept Description
Zone Signing Key (ZSK) A key pair used to sign individual resource record sets (RRSETs) in the zone. Rotated frequently (often monthly).
Key Signing Key (KSK) A key pair used only to sign the DNSKEY record set. Rotated less often. Its hash is published in the parent zone as a DS record.
DNSKEY Published in the zone itself; holds the public ZSK and KSK so resolvers can verify signatures.
RRSIG A cryptographic signature over a specific RRSET (e.g., the A records for www.example.com). Attached alongside the records in responses.
DS (Delegation Signer) A hash of the child zone's KSK, published in the parent zone (e.g., .com). This is how the chain of trust crosses zone boundaries.
NSEC / NSEC3 Records that prove a name or type does not exist, enabling authenticated denial of existence. NSEC3 hashes names to prevent zone enumeration.

Chain of Trust

The chain of trust starts at the root zone, which is signed and whose trust anchor is pre-configured in validating resolvers. Each zone's KSK is hashed into a DS record in the parent zone, creating a verifiable chain from root → TLD → domain.

flowchart TD
    A[Root zone trust anchor] --> B[Root DNSKEY]
    B --> C[Root signs .com DS]
    C --> D[.com DNSKEY]
    D --> E[.com signs example.com DS]
    E --> F[example.com DNSKEY]
    F --> G[example.com RRSIG signs A records]
    G --> H[Resolver validates answer]

Validation Flow

When a validating resolver receives a DNSSEC-signed response:

  1. Retrieve the DNSKEY records for the zone.
  2. Verify the DNSKEY RRSET is signed by a key whose hash matches the DS record in the parent zone.
  3. Use the ZSK (from DNSKEY) to verify the RRSIG over the answer records.
  4. If all signatures validate and the chain leads back to a trusted root anchor, set the AD (Authenticated Data) flag in the response to the client.
  5. If any signature is missing, expired, or invalid, return SERVFAIL.
sequenceDiagram
    participant R as Validating Resolver
    participant Auth as Authoritative Server
    participant Parent as Parent Zone
    R->>Auth: Query A record + DO flag
    Auth-->>R: A record + RRSIG + DNSKEY
    R->>Parent: Query DS record for zone
    Parent-->>R: DS record + RRSIG
    R->>R: Verify DNSKEY hash matches DS
    R->>R: Verify RRSIG over A record with ZSK
    R->>R: Chain valid -> set AD flag

Key Rollover

Keys must be rotated periodically. The standard approach is a pre-publish rollover:

  1. Publish the new key in DNSKEY alongside the old key (both active).
  2. Wait for TTL to expire so all caches have the new key.
  3. Switch signing to the new key.
  4. Remove the old key after another TTL window.

For KSK rollovers, the parent zone's DS record must also be updated, which requires coordination with the registrar or parent zone operator. Mistakes here — such as removing the old DS before caches expire — cause validation failures and outages.

What DNSSEC Does and Does Not Solve

Does solve Does not solve
Data origin authentication Query privacy (use DoT / DoH for that)
Integrity of records in transit Availability / DDoS protection
Cache poisoning with forged answers Registrar-level hijacking (key material can still be replaced)
Authenticated denial of existence Confidentiality of zone contents (NSEC leaks names; NSEC3 reduces but does not eliminate this)

Top 3 Critical Mitigations

1. Harden Recursive Resolvers

Resolvers sit in the trust path for almost every connection, so they should validate answers, randomize request metadata, and avoid open-recursion behavior. This reduces poisoning and abuse risks while keeping resolver infrastructure from becoming an amplification tool. The trade-off is stricter behavior can expose broken upstream configurations faster, but that is preferable to silently serving unsafe answers.

2. Control DNS Change Management

Many damaging DNS incidents are operational rather than purely protocol-level. Registrar lock, MFA, monitored changes, short emergency rollback paths, and inventory-aware cleanup reduce hijacking and dangling-record risk. The trade-off is more process overhead, but DNS is too high-impact to treat casually.

3. Use DNSSEC Carefully

DNSSEC adds authenticity to DNS data by signing records and validating chains of trust. It materially reduces forged-answer risk, especially for authoritative zones that matter to identity or routing of critical services. The trade-off is operational complexity, key rollover discipline, and the possibility of outages when signing is mismanaged.

Practice Questions

Tier 1 — Fundamentals

  1. What problem does DNS solve? DNS maps names to data such as IP addresses so users and software do not need to work directly with raw addresses. This is asked to confirm basic protocol purpose.
  2. Why does DNS mostly use UDP but sometimes TCP? UDP is lightweight for normal lookups, while TCP is used for large answers, retries, and zone transfers. This is asked to test transport trade-off knowledge.
  3. What is the difference between recursive and authoritative DNS? Recursive resolvers fetch and cache answers for clients, while authoritative servers publish the source-of-truth data for a zone. This is asked to verify role separation.
  4. What do A, AAAA, MX, and CNAME records do? They map names to IPv4, IPv6, mail routing, and aliases respectively. This is asked to test practical DNS literacy.
  5. What does TTL control? TTL controls how long answers may be cached before they should be refreshed. This is asked because TTL choices affect both reliability and rollout speed.

Tier 2 — System Design and Trade-offs

  1. How would you design DNS for a globally distributed application? I would use resilient authoritative providers, health-aware routing where appropriate, careful TTL choices, and monitored registrar controls. This is asked to see whether the candidate understands DNS as production infrastructure.
  2. When can low TTLs help and when can they hurt? Low TTLs help during failover and fast cutovers but increase query volume and sensitivity to resolver behavior. This is asked to probe operational judgment.
  3. How do you defend against DNS amplification abuse? Disable open recursion, limit who can query, rate-limit responses, and observe unusual query patterns. This is asked to test protocol abuse awareness.
  4. What can go wrong with split-horizon DNS? Internal and external views can drift, causing clients to resolve different or broken destinations. This is asked to test complexity management in enterprise environments.
  5. How would you investigate a DNS outage that affects only some users? I would check resolver path, cache state, TTL expiry, authoritative answers, EDNS size issues, and regional network differences. This is asked to test troubleshooting structure.

Tier 3 — Expert and Deep-Dive

  1. Why is DNS both a control plane and an attack surface? It controls where clients go next, so bad answers can redirect large portions of traffic. This is asked to test architectural framing.
  2. What does DNSSEC solve and what does it not solve? It authenticates DNS data but does not hide queries or guarantee service availability. This is asked to separate authenticity from privacy and availability.
  3. How would you detect subdomain takeover risk at scale? Inventory DNS records, compare them to active cloud resources, and alert on dangling references. This is asked to test cloud operational maturity.
  4. What are the hardest production problems in recursive resolver fleets? Abuse resistance, cache efficiency, privacy policy, and bad upstream behavior all compete with low-latency answering. This is asked to test systems thinking.
  5. How would you explain why DNS incidents often look like application incidents first? Applications fail where name resolution feeds them incorrect or missing destinations, so DNS faults surface as higher-layer symptoms. This is asked to test cross-layer reasoning.

Key Takeaways

  • DNS is a distributed control system for names, not just a phone book.
  • Recursive and authoritative servers have different trust and operational roles.
  • Cache behavior and TTL choices are core to reliability.
  • DNS is a major attack surface for hijack, reflection, and poisoning.
  • DNSSEC adds authenticity, not privacy.
  • Registrar and operational controls matter as much as protocol details.
  • Good DNS design reduces both outage blast radius and security risk.
