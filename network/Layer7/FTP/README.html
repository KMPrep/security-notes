FTP — File Transfer Protocol

OSI Layer: 7 — Application
RFC: RFC 959, RFC 2428, RFC 3659

Overview

FTP is a classic application-layer protocol for transferring files between a client and server. Its defining feature is the use of separate channels for control and data, which makes it operationally awkward compared with newer secure alternatives. FTP supports directory listing, upload, download, rename, delete, and transfer-mode negotiation, and historically powered a large amount of software distribution and internal file exchange. Because FTP sends credentials and content in cleartext unless wrapped in TLS as FTPS, it is generally unsuitable for untrusted networks.

How It Works

Core Mechanics

An FTP session begins with a TCP control connection, typically to port 21. The client authenticates with commands such as USER and PASS, and then sends further commands like LIST, RETR, STOR, CWD, and DELE. Server responses use numeric reply codes that indicate status, success, or failure.

Actual file contents do not travel over the control connection. Instead, FTP creates a separate data connection for each transfer or directory listing. In active mode the server connects back to the client, which often fails through NATs and firewalls. In passive mode the server advertises a port and the client initiates the data connection, which is much easier to operate in modern networks. ASCII and binary transfer modes also matter, especially for legacy systems that distinguish text translation from raw byte transfer.

Top 3 Critical Mitigations

1. Migrate Off Plain FTP

The strongest mitigation for FTP risk is to stop using plain FTP for sensitive environments. SFTP or FTPS preserves file-transfer capability while adding encryption and better modern controls. The main trade-off is compatibility with legacy tooling, but the security improvement is substantial.

2. Use Passive Mode with Narrow Port Ranges

Passive mode avoids server-to-client callbacks that break in NATed and firewalled networks. If FTP must remain, controlling the passive port range keeps firewall exposure manageable. The trade-off is a small amount of operational coordination between the FTP server and firewalls.

3. Treat FTP Servers as High-Risk Legacy Infrastructure

FTP servers should be isolated, tightly monitored, and treated as exception systems. They often sit at the boundary between users and sensitive files, and they frequently persist long after their original justification has expired. Strong logging and segmentation reduce the blast radius while migration is underway.

Practice Questions

Tier 1 — Fundamentals

  1. Why does FTP use two connections? One carries commands and replies, and the other carries file or listing data. This is asked to test protocol structure knowledge.
  2. What is the difference between active and passive FTP? In active mode the server connects back to the client, while in passive mode the client connects to a server-advertised port. This is asked because it explains most real-world FTP problems.
  3. Why is plain FTP insecure? Credentials and file contents are sent without encryption. This is asked to confirm basic security understanding.
  4. What is the control port for FTP? Port 21 is the standard control port. This is asked to test operational familiarity.
  5. Why has FTP declined in modern environments? Secure alternatives are easier to defend and operate. This is asked to connect protocol design to present-day usage.

Tier 2 — System Design and Trade-offs

  1. How would you run an unavoidable legacy FTP service safely? I would segment it, force passive mode, narrow passive ports, restrict clients, log heavily, and define a migration plan. This is asked to test pragmatic risk management.
  2. Why is passive FTP easier behind NAT? The client initiates both TCP connections, so stateful devices can track them more naturally. This is asked to test network-path reasoning.
  3. How would you detect abuse on an internet-facing FTP server? I would monitor auth failures, unusual downloads, anonymous access, command patterns, and data volume spikes. This is asked to test detection thinking.
  4. When would FTPS be chosen instead of SFTP? Usually when an organization must preserve FTP-style workflows or vendor compatibility. This is asked to test protocol-family distinction.
  5. What operational problems come from FTP’s dual-channel design? Firewall complexity, NAT issues, monitoring confusion, and broader attack surface are the main ones. This is asked to test systems comprehension.

Tier 3 — Expert and Deep-Dive

  1. Why is FTP a good example of protocol design aging badly? Its assumptions about trust and network topology do not match modern encrypted, NAT-heavy environments. This is asked to test architectural judgment.
  2. How would you phase out FTP in a large enterprise? Inventory dependencies, replace high-risk external uses first, add compensating controls, and provide migration tooling to SFTP or managed transfer services. This is asked to test program-level thinking.
  3. What security properties does FTPS add and what complexity does it preserve? It adds TLS confidentiality and authentication but keeps the multi-channel FTP operating model. This is asked to test nuanced comparison.
  4. Why are file-transfer logs often more important than teams expect? They support incident reconstruction, data-loss detection, and nonrepudiation in regulated workflows. This is asked to test operational maturity.
  5. What does FTP not solve for modern data exchange? It does not provide safe identity lifecycle, integrity semantics, or easy internet-era operability. This is asked to separate legacy transport from modern platform needs.

Key Takeaways

  • FTP separates control and data channels.
  • Passive mode is usually the only workable mode in modern networks.
  • Plain FTP exposes credentials and file contents.
  • Legacy FTP should be isolated and monitored if it cannot yet be removed.
  • Secure alternatives such as SFTP or FTPS are preferred.
  • FTP’s dual-channel design is the source of much of its complexity.
  • Migration planning is often the right long-term control.
