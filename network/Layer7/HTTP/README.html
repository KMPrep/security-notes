HTTP — HyperText Transfer Protocol

OSI Layer: 7 — Application
RFC: RFC 9110 (HTTP semantics), RFC 9111 (caching), RFC 9112 (HTTP/1.1), RFC 9113 (HTTP/2), RFC 9114 (HTTP/3)

Overview

HTTP is the application-layer protocol used to transfer documents, APIs, and media across the web. It follows a request-response model in which a client sends a method, target resource, headers, and optional body, and the server returns a status code, headers, and optional body. HTTP is stateless at the protocol layer, so higher-level session continuity is handled through cookies, bearer tokens, or application-managed state. Modern deployments commonly use HTTP/1.1 for compatibility, HTTP/2 for multiplexed streams over one connection, and HTTP/3 over QUIC for lower latency and better resilience on lossy networks.

How It Works

Core Mechanics

HTTP defines semantics for methods such as GET, POST, PUT, PATCH, DELETE, HEAD, and OPTIONS, along with status codes that indicate whether the server fulfilled the request. A typical request contains a request line or pseudo-headers, regular headers, and optionally a message body; the response contains status metadata, headers, and a body. Intermediaries such as reverse proxies, load balancers, CDNs, and caches may terminate connections, rewrite headers, enforce policy, or serve cached content before the request reaches the origin.

HTTP/1.1 is text-based and typically uses persistent TCP connections with one outstanding request per connection unless pipelining is used. HTTP/2 introduces binary framing, stream multiplexing, HPACK header compression, and flow control so multiple streams can share one connection efficiently. HTTP/3 preserves HTTP semantics but transports frames over QUIC instead of TCP, which removes transport-layer head-of-line blocking and improves migration across network changes.

Common request fields include the method, scheme, authority, path, headers such as Host, Accept, Content-Type, Authorization, Cache-Control, and Cookie, and an optional body. Common response fields include the status code, Content-Type, Content-Length, ETag, Set-Cookie, Cache-Control, and Location. Caching, conditional requests, content negotiation, range requests, authentication challenges, and compression are all layered onto the same message model.

Top 3 Critical Mitigations

1. Enforce HTTPS Everywhere

HTTPS protects confidentiality and integrity for HTTP messages by layering HTTP over TLS or QUIC with TLS-equivalent cryptographic guarantees. HSTS reduces downgrade risk by instructing browsers to refuse cleartext access after the first trusted HTTPS visit. The main trade-offs are certificate lifecycle management, operational complexity at the edge, and the need to inspect traffic through controlled termination points. Relevant standards include RFC 9110, RFC 9114, and browser HSTS behavior defined by RFC 6797.

2. Normalize Parsing at Proxies and Origins

Request smuggling occurs when one hop interprets framing differently from another, especially around Content-Length and Transfer-Encoding. The mitigation is to ensure edge proxies normalize or reject ambiguous requests and that origin servers follow the same parsing expectations as the proxy. This can break legacy clients that rely on non-compliant behavior, but the security payoff is high because smuggling can lead to cross-tenant cache poisoning or auth bypass. Vendor guidance from major CDNs and reverse proxies often includes strict parsing modes specifically for this reason.

3. Apply Strong Browser-Side and Server-Side Web Controls

HTTP is the delivery channel for many browser attacks, so controls such as CSP, SameSite cookies, CSRF tokens, strict CORS policies, and secure cookie attributes are essential. These mechanisms reduce the exploitability of XSS, CSRF, and cross-origin data exposure even when application code has minor defects. The trade-off is that these protections require consistent policy management and can break older client behavior if rolled out carelessly. In practice, mature teams combine these controls with automated security tests and staged deployments.

Practice Questions

Tier 1 — Fundamentals

  1. What makes HTTP stateless, and how do real systems maintain user sessions anyway?
    HTTP does not require the server to retain per-client protocol state between requests; each request is self-contained from the protocol perspective. Real systems layer session continuity on top using cookies, server-side session stores, JWTs, opaque bearer tokens, or signed session identifiers. This is asked to verify that the candidate distinguishes protocol semantics from application architecture.

  2. What is the difference between GET and POST, and why does idempotency matter?
    GET is intended for safe retrieval and should not mutate server state, while POST submits data that commonly changes state. Idempotency matters because retries, intermediaries, and distributed failures can cause duplicate delivery, so operations such as PUT or idempotency-key-backed POST are easier to reason about under failure. This is asked to see whether the candidate understands semantics, retries, and API correctness.

  3. Why was HTTP/2 introduced if HTTP/1.1 already had keep-alive?
    Keep-alive reduced connection setup overhead, but HTTP/1.1 still suffered from inefficient request concurrency patterns and application-layer head-of-line issues. HTTP/2 adds multiplexing, binary framing, header compression, and per-stream flow control so many requests can share a single connection more efficiently. This is asked to test whether the candidate can explain performance bottlenecks beyond memorizing version names.

  4. What is the purpose of status codes, and which classes should an engineer know cold?
    Status codes communicate standardized outcome categories: informational 1xx, success 2xx, redirection 3xx, client error 4xx, and server error 5xx. Engineers should know the operational meaning of common codes such as 200, 201, 301, 302, 304, 400, 401, 403, 404, 409, 429, and 500. This is asked because good engineers use status codes as part of system contracts and observability.

  5. How do cookies differ from authorization headers for authentication?
    Cookies are automatically attached by browsers based on scope rules and can carry session identifiers, while authorization headers are typically attached explicitly by the client and often carry bearer tokens. Cookies are more exposed to CSRF if not protected with SameSite and anti-CSRF controls, while bearer headers are more exposed to storage and exfiltration issues in client applications. This is asked to assess browser-security literacy and practical auth design judgment.

Tier 2 — System Design and Trade-offs

  1. How would you design a high-scale HTTP API to survive retries without duplicate side effects?
    I would define clear idempotency semantics at the API boundary, usually via idempotency keys for unsafe operations and naturally idempotent verbs where possible. The server would persist the first successful or terminal result keyed by client identity plus idempotency key, and replay that result on retries. I would also design request bodies and response schemas to make duplicate detection cheap and safe under partial failures. This is asked to evaluate distributed-systems thinking, not just HTTP trivia.

  2. When would HTTP caching materially improve a system, and when can it hurt you?
    Caching helps most when responses are expensive to compute, broadly reusable, and tolerant of bounded staleness, such as static assets, metadata, and public API reads. It hurts when cache keys are incomplete, user-specific data leaks into shared caches, or invalidation is poorly designed. Strong validators such as ETag and explicit Cache-Control directives are essential to make caching predictable. This is asked to test whether a candidate understands performance, correctness, and multi-layer architecture together.

  3. Why can a reverse proxy create security bugs even if the origin application is correct?
    The proxy may terminate TLS, rewrite headers, normalize paths, cache responses, and enforce auth or rate limits before traffic reaches the origin. If the proxy and origin disagree on path normalization, framing, trust boundaries, or header semantics, an attacker can exploit the mismatch through smuggling, auth bypass, or cache poisoning. A secure origin is not sufficient if the enforcement point in front of it behaves differently. This is asked because senior engineers must reason about whole request paths, not isolated services.

  4. How would you compare HTTP/2 and HTTP/3 for a mobile client fleet?
    HTTP/2 improves utilization over TCP through multiplexing, but packet loss can still create transport-level head-of-line blocking across streams. HTTP/3 runs over QUIC on UDP, which improves recovery from packet loss and supports connection migration when devices switch networks. The trade-offs include operational maturity, load-balancer support, observability changes, and occasional middlebox interference with UDP. This is asked to probe version-level understanding tied to user impact and production trade-offs.

  5. How would you rate-limit HTTP traffic fairly without harming tail latency too much?
    I would choose the limiting dimension carefully, such as user, API key, tenant, IP, or method-path class, then apply token bucket or leaky bucket logic at the edge. I would separate burst tolerance from sustained throughput and give critical control-plane operations different budgets from bulk data paths. For fairness, I would couple rate limiting with good error signaling like 429 plus retry hints and measure the effect on p95 and p99 latency. This is asked to test policy design, not just naming a standard algorithm.

Tier 3 — Expert and Deep-Dive

  1. Explain request smuggling in terms of parser differentials and describe how you would prove a platform is not vulnerable.
    Request smuggling arises when two hops in the request chain disagree on where one request ends and the next begins, often due to conflicting handling of Content-Length, Transfer-Encoding, duplicate headers, or malformed chunking. An attacker crafts a message that the front end interprets one way and the back end interprets another way, causing hidden prefix requests, cache poisoning, or credential confusion across users. To prove resistance, I would inventory all parsing hops, enable strict RFC-compliant normalization at the edge, test with known ambiguous payload classes, and ensure origin behavior matches the edge under fuzzing and regression tests. This is asked because it reveals whether the candidate understands protocol exploitation at the boundary between specs and implementations.

  2. What semantic guarantees does HTTP actually provide, and which guarantees are often incorrectly assumed by application teams?
    HTTP provides message semantics, method definitions, metadata exchange, and a framework for intermediaries, but it does not guarantee exactly-once execution, transactionality, ordering across multiple requests, or durable session state. Application teams often assume that one request maps cleanly to one business action, that retries are harmless, or that a 200 response means downstream side effects fully committed. In reality, load balancers, retries, timeouts, and partial failures make those assumptions unsafe unless the application explicitly enforces them. This is asked to separate engineers who know the wire protocol from those who can build reliable systems on top of it.

  3. How would you design observability for a large HTTP platform where proxies, services, and clients all contribute latency?
    I would break end-to-end timing into phases such as DNS, connect, TLS, request upload, origin queueing, application processing, backend calls, and response download. Correlation IDs, distributed tracing, structured logs, and standardized response headers would allow us to connect client-side telemetry with edge and service-side spans. I would also track protocol-version mix, cache hit rate, retry rate, connection reuse, status-code families, and per-route saturation signals. This is asked because senior engineers must isolate user-visible latency to concrete layers rather than treating HTTP as a single black box.

  4. Why are browser-facing HTTP systems fundamentally different from service-to-service HTTP systems from a security perspective?
    Browser-facing systems inherit the browser execution model, origin policy, cookie behavior, CORS, mixed-content rules, caching interactions, and user-driven navigation patterns. Service-to-service systems usually control both ends of the connection and can use mutual TLS, narrower trust boundaries, and simpler client behavior. As a result, browser threats like XSS and CSRF dominate one environment, while identity propagation, replay resistance, and policy drift often dominate the other. This is asked to see whether the candidate can apply protocol knowledge differently across trust models.

  5. If you had to explain why HTTP remains viable despite being old and text-oriented, what would your architecture argument be?
    HTTP survives because its semantics are stable, extensible, and widely interoperable across clients, servers, proxies, CDNs, browsers, and developer tooling. The ecosystem has repeatedly improved the transport beneath the semantics, moving from HTTP/1.1 over TCP to HTTP/2 framing and HTTP/3 over QUIC without discarding the application contract that made the web scale. It also aligns well with cache hierarchies, observability patterns, authentication gateways, and incremental deployment. This is asked because strong candidates can explain not just how a protocol works, but why it persists as an architectural substrate.

Key Takeaways

  • HTTP is stateless at the protocol level, but real applications layer state through cookies, tokens, and server-side session management.
  • HTTP semantics stay mostly constant across versions; the biggest changes between HTTP/1.1, HTTP/2, and HTTP/3 are transport efficiency and connection behavior.
  • Intermediaries are part of the protocol reality, so caches, CDNs, proxies, and load balancers must be included in both design and threat modeling.
  • Security failures in HTTP are often caused by semantic mismatches, unsafe browser behavior, or proxy-origin inconsistencies rather than the method line alone.
  • Caching and idempotency are core to correctness and scale, not optional optimizations.
  • Browser-facing HTTP requires explicit protection against XSS, CSRF, cookie abuse, mixed content, and CORS misconfiguration.
  • Senior engineers should be able to explain HTTP at the levels of wire behavior, distributed-system trade-offs, and production security controls.
