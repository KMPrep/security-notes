HTTPS — HTTP Secure

OSI Layer: 7 — Application (secured via TLS at Layer 6)
RFC: RFC 9110 (HTTP semantics), RFC 8446 (TLS 1.3), RFC 6797 (HSTS), RFC 6962 (Certificate Transparency)

Overview

HTTPS is HTTP carried over a cryptographically protected TLS session, giving web traffic confidentiality, integrity, and authenticated server identity. Before any HTTP request or response is exchanged, the client and server perform a TLS handshake to negotiate cryptographic parameters, validate certificates, and derive shared session keys. In modern deployments, HTTPS is the default transport for websites, APIs, authentication flows, and browser-based applications because plain HTTP is vulnerable to interception and tampering. HTTP semantics stay the same under HTTPS; the major difference is that the underlying transport is protected against eavesdropping, manipulation, and many active network attacks.

How It Works

Core Mechanics

HTTPS combines two layers: HTTP for application semantics and TLS for transport security. The client begins by connecting to the server, then sends a TLS ClientHello advertising supported protocol versions, cipher suites, extensions such as SNI and ALPN, and key share information. The server replies with a ServerHello, selects cryptographic parameters, sends its certificate chain, proves possession of the private key, and completes key agreement so both sides can derive symmetric session keys.

After the TLS handshake completes, all HTTP messages are encrypted and integrity-protected inside the TLS session. The client still sends familiar HTTP methods such as GET, POST, PUT, and DELETE, and the server still returns standard HTTP status codes and headers. In practice, ALPN determines whether the application protocol carried inside TLS is HTTP/1.1 or HTTP/2; HTTP/3 uses QUIC rather than TLS over TCP, but it preserves the same overall goal of protected HTTP transport.

Key HTTPS building blocks include X.509 certificates, certificate chains, public key cryptography for authentication, ephemeral key exchange for forward secrecy, symmetric encryption for bulk data, and MAC or AEAD modes for integrity. Browser trust stores, OCSP or stapling behavior, Certificate Transparency logs, and hostname verification are all part of the real-world trust model. Operationally, HTTPS also depends on correct certificate issuance, rotation, expiration monitoring, and secure TLS configuration at the edge.

TLS 1.2 vs TLS 1.3

TLS 1.2 (RFC 5246, 2008) and TLS 1.3 (RFC 8446, 2018) are the two versions in active production use. TLS 1.3 is the preferred modern baseline; TLS 1.2 remains in wide deployment but requires careful cipher suite hardening.

TLS 1.2

  • Handshake: 2 round trips before application data flows (full handshake), 1 round trip with session resumption.
  • Key exchange: RSA key transport or Diffie-Hellman (static or ephemeral). Only ECDHE provides forward secrecy; RSA key exchange does not.
  • Cipher suites: Negotiated across many combinations including legacy weak options (RC4, 3DES, export ciphers, MD5/SHA-1 MACs). Must be explicitly restricted in configuration.
  • MAC construction: Separate MAC-then-Encrypt model for older suites; vulnerable to BEAST and POODLE without mitigation. AEAD suites (AES-GCM, ChaCha20-Poly1305) are available and should be mandatory.
  • Renegotiation: Supported; historically exploited (CVE-2009-3555). Secure renegotiation extension (RFC 5746) is required.
  • 0-RTT: Not supported natively.
  • Recommended config: Disable TLS 1.0 and 1.1, restrict to ECDHE cipher suites with AEAD (e.g. TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256), disable RSA key exchange.

TLS 1.3

  • Handshake: 1 round trip (1-RTT) for a full handshake. 0-RTT (early data) available for resumption.
  • Key exchange: Only ephemeral key exchange (ECDHE or DHE) is allowed. Forward secrecy is mandatory, not optional.
  • Cipher suites: Reduced and fixed set — only AEAD with strong hash. No negotiable MAC algorithm; no static RSA. Three standard suites: TLS_AES_128_GCM_SHA256, TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256.
  • Removed features: No renegotiation, no compression, no RC4/3DES/export ciphers, no MD5/SHA-1 in the handshake, no static RSA key exchange.
  • Encrypted handshake: Certificate and most handshake messages are encrypted after the first flight, reducing metadata exposure compared to TLS 1.2.
  • 0-RTT trade-off: Early data allows replay attacks because there is no server-side freshness guarantee. Must be used only for replay-safe operations (idempotent requests) and with application-level replay mitigations.
  • Performance: Lower latency due to 1-RTT handshake and optimized key schedule (HKDF-based).

Key Exchange — Complete Deep-Dive

Key exchange solves one fundamental problem: two strangers on a public network need to agree on a shared secret without ever transmitting that secret. Everything else — encryption, integrity, authentication — depends on solving this first.

There are three mechanisms used in TLS: RSA key transport, DHE, and ECDHE. Understanding why RSA was abandoned and why ECDHE won requires understanding what each actually does.

Mechanism 1 — RSA Key Transport

The core idea: The client picks the secret and locks it in a box using the server's public key. Only the server's private key can open it.

The problem it solves:
  "I want to send you a secret. I'll encrypt it with your public key.
   Only you can decrypt it. Now we both know the secret."

Step by step:
  1. Client generates a random 48-byte value called the premaster_secret
  2. Client encrypts it with the server's RSA public key (from the certificate)
  3. Client sends the encrypted blob in ClientKeyExchange
  4. Server decrypts it using its RSA private key
  5. Both sides now have premaster_secret
  6. Both derive session keys from: premaster_secret + client_random + server_random

What the attacker sees on the wire:
  ┌──────────────────────────────────────┐
  │ ClientKeyExchange: [encrypted blob]  │  ← looks random, harmless now
  └──────────────────────────────────────┘

The trap:
  The server's RSA private key is LONG-LIVED — same key for years.
  An attacker who records the traffic today can wait.
  The day they get the private key (breach, court order, insider leak),
  they decrypt every past session they recorded.

  Record now → Decrypt later  (HARVEST NOW, DECRYPT LATER)

Why RSA key transport was removed from TLS 1.3: It has no forward secrecy. The session key's security is permanently coupled to the long-term private key. This is not a theoretical risk — intelligence agencies and nation-states are known to record TLS traffic at scale for future decryption.

Mechanism 2 — DHE (Diffie-Hellman Ephemeral)

The mental model — paint mixing analogy (never forget this):

Imagine mixing paint colors. Anyone can mix colors together,
but it is VERY hard to un-mix paint to figure out what was added.

1. Alice and Bob publicly agree on a base color: YELLOW (this is public)
2. Alice picks a SECRET color: RED (never shared)
3. Bob picks a SECRET color: BLUE (never shared)
4. Alice mixes: YELLOW + RED = ORANGE → sends ORANGE to Bob (public)
5. Bob mixes: YELLOW + BLUE = GREEN → sends GREEN to Alice (public)
6. Alice mixes: GREEN + RED = BROWN (her secret + Bob's public mix)
7. Bob mixes: ORANGE + BLUE = BROWN (his secret + Alice's public mix)
8. Both get BROWN — the shared secret!

Eve sees: YELLOW, ORANGE, GREEN — but cannot figure out BROWN
because she does not know RED or BLUE and cannot un-mix paint.

The actual math (discrete logarithm problem):

Public parameters (everyone knows these):
  p = a large prime number (e.g. 2048-bit)
  g = a generator (small number, typically 2 or 5)

Server picks private value:  a  (random, secret, ephemeral)
Server computes public value: A = g^a mod p  → sends A to client

Client picks private value:  b  (random, secret, ephemeral)
Client computes public value: B = g^b mod p  → sends B to server

Server computes shared secret: S = B^a mod p = (g^b)^a mod p = g^(ab) mod p
Client computes shared secret: S = A^b mod p = (g^a)^b mod p = g^(ab) mod p

Both arrive at the same S = g^(ab) mod p

Attacker sees A and B but to find a or b they must solve:
  "Given g^a mod p, find a"  ← this is the Discrete Logarithm Problem
  Computationally infeasible for large p (2048-bit or more)

Why "Ephemeral" is the key word:

Static DH:   server uses the SAME a every time → attacker who records
             enough traffic can eventually solve for a → breaks past sessions

Ephemeral DH: server generates a FRESH random a for EVERY handshake
              After the handshake: a is destroyed, A is thrown away
              Even if the server's certificate private key is stolen,
              there is no a to recover → past sessions stay secret

This property is called FORWARD SECRECY (or Perfect Forward Secrecy, PFS)

Diagram — DHE Key Exchange

sequenceDiagram
    participant C as Client
    participant S as Server
    S->>S: Generate ephemeral a, compute A = g^a mod p
    C->>S: ClientHello (random_C)
    S-->>C: ServerHello + Certificate (for auth only, not key transport)
    S-->>C: ServerKeyExchange (p, g, A, signed with cert private key)
    C->>C: Generate ephemeral b, compute B = g^b mod p
    C->>S: ClientKeyExchange (B)
    C->>C: Compute S = A^b mod p
    S->>S: Compute S = B^a mod p
    note over C,S: Both have S = g^(ab) mod p — derived from ephemeral values only
    S->>S: Discard a (ephemeral — gone forever)
    C->>C: Discard b (ephemeral — gone forever)
    C->>S: Finished (encrypted with keys derived from S)
    S-->>C: Finished

DHE weakness: For security, p must be at least 2048 bits. DHE over finite fields is CPU-heavy — the modular exponentiations on large numbers are expensive. This led to ECDHE.

Mechanism 3 — ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)

The core insight: ECDHE is DHE but the underlying hard problem is on an elliptic curve rather than modular arithmetic. The same paint-mixing logic applies, but the math is done in a completely different algebraic structure that gives the same security with much smaller key sizes.

What is an elliptic curve?

An elliptic curve is defined by: y² = x³ + ax + b  (over a finite field)

It looks like a smooth curve with a special "point at infinity" that acts as zero.

Key operation: POINT MULTIPLICATION
  Given a point G (the generator, public) and a scalar k (your private key):
  P = k × G  (add G to itself k times using the curve's addition rule)

  This is easy to compute: O(log k) steps
  This is hard to reverse: given P and G, find k
  ← this is the Elliptic Curve Discrete Logarithm Problem (ECDLP)

ECDHE in practice:

Public: curve parameters + base point G (standardized, e.g. P-256, X25519)

Server picks private key: a  (random scalar, ephemeral)
Server computes public key: A = a × G  → sends A in ServerKeyExchange

Client picks private key:  b  (random scalar, ephemeral)
Client computes public key: B = b × G  → sends B in ClientKeyExchange

Server computes shared secret: S = a × B = a × (b × G) = (ab) × G
Client computes shared secret: S = b × A = b × (a × G) = (ab) × G

Both get the same point S on the curve → x-coordinate becomes the shared secret

Attacker sees A = a×G and B = b×G but cannot recover a or b (ECDLP)

Why ECDHE beats DHE on performance:

DHE with 2048-bit p  ≈  equivalent security to  ECDHE with 224-bit curve
DHE with 3072-bit p  ≈  equivalent security to  ECDHE with 256-bit curve (P-256)

Smaller numbers = faster exponentiation = lower CPU = lower latency
This is why TLS 1.3 uses ECDHE by default.

Named curves used in TLS:

Curve Also known as Key size Security level Notes
P-256 secp256r1 256-bit ~128-bit NIST curve, universal support
P-384 secp384r1 384-bit ~192-bit NIST curve, used in high-security
X25519 Curve25519 255-bit ~128-bit Non-NIST, faster, preferred in 1.3
X448 Curve448 448-bit ~224-bit High security, less common

X25519 is the preferred curve in TLS 1.3 because it is faster than P-256, has no known backdoor concerns unlike NIST curves, and its implementation is naturally resistant to timing side-channel attacks.

AEAD — Authenticated Encryption with Associated Data

Once the key exchange is done, both sides have a shared secret. That secret is used to derive bulk encryption keys — the keys that protect every byte of the actual HTTP request and response. In modern TLS those keys are always AEAD keys.

Why AEAD Exists — The Problem with Separate MAC + Encrypt

Before AEAD, TLS used two separate operations: a symmetric cipher for confidentiality and a MAC (Message Authentication Code) for integrity. Combining them correctly turns out to be surprisingly hard. There are three orderings, and two of them are broken:

Option 1 — Encrypt-then-MAC (EtM):
  ciphertext = Encrypt(plaintext, key)
  tag        = MAC(ciphertext, mac_key)
  → Send: ciphertext + tag

  Receiver checks tag BEFORE decrypting. If tag is wrong, discard immediately.
  ✓ Secure — the attacker cannot tamper with the ciphertext undetected.
  ✓ Used in IPsec. Required for correct TLS 1.2 CBC mode (RFC 7366).

Option 2 — MAC-then-Encrypt (MtE):  ← what TLS 1.2 CBC used by default
  tag        = MAC(plaintext, mac_key)
  ciphertext = Encrypt(plaintext + tag, key)
  → Send: ciphertext

  Receiver must DECRYPT FIRST, then verify the MAC.
  ✗ Dangerous — the decryption happens before verification, leaking padding behavior.
  ✗ This is the root of POODLE (SSLv3), BEAST (TLS 1.0), Lucky13 (TLS 1.2 CBC).

Option 3 — Encrypt-and-MAC (E&M):
  ciphertext = Encrypt(plaintext, key)
  tag        = MAC(plaintext, mac_key)   ← MAC over plaintext, not ciphertext
  → Send: ciphertext + tag

  ✗ Dangerous — the MAC can leak plaintext structure.
  ✗ Used in SSH (and was found to leak data in certain modes).

The lesson: Composing primitives by hand is error-prone. AEAD packages encryption + authentication into a single, atomic primitive with a formally proven security model.

What AEAD Does

AEAD takes four inputs and produces two outputs:

Inputs:
  key           — the symmetric key (derived from the TLS key schedule)
  nonce         — a unique number used once per message (also called IV)
  plaintext     — the data to encrypt (HTTP request/response body + headers)
  AAD           — Additional Associated Data (data to authenticate but NOT encrypt)
                  In TLS: the record header (content type, version, length)

Outputs:
  ciphertext    — encrypted plaintext (same length as plaintext)
  auth tag      — a short fixed-size fingerprint (e.g. 16 bytes for GCM)
                  covers BOTH the ciphertext AND the AAD

Decryption:
  Receiver recomputes auth tag over (ciphertext + AAD)
  If tag does not match → reject immediately, do not touch the ciphertext
  If tag matches → decrypt and deliver plaintext

The AAD part is what makes it "with Associated Data." In TLS the record header (content type, protocol version, record length) is not encrypted but IS authenticated. This means an attacker cannot strip, duplicate, reorder, or truncate TLS records without the receiver detecting it — even though the header bytes are visible in plaintext.

TLS 1.3 record wire format (AES-GCM):

┌──────────────────────────────────────────────────────────────────────┐
│ TLS Record Header (5 bytes) — PLAINTEXT, included in AAD             │
│  content_type (1B) | legacy_version (2B) | record_length (2B)        │
├──────────────────────────────────────────────────────────────────────┤
│ Ciphertext (variable) — ENCRYPTED                                    │
│  = AES-GCM(key, nonce, plaintext, aad)                               │
├──────────────────────────────────────────────────────────────────────┤
│ Auth Tag (16 bytes) — appended to ciphertext                         │
│  = GHASH over (ciphertext + AAD), authenticated by GCM               │
└──────────────────────────────────────────────────────────────────────┘

The Two AEAD Algorithms in TLS

AES-GCM (AES in Galois/Counter Mode)

AES-GCM combines two operations:

  • CTR mode (Counter Mode) for encryption
  • GHASH (Galois field multiplication) for the authentication tag
Encryption side:
  Counter blocks: CTR0, CTR1, CTR2, ...  (nonce || counter, incremented)
  Each counter block is AES-encrypted: E(key, CTRi) = keystream_block_i
  plaintext_block_i XOR keystream_block_i = ciphertext_block_i

  → CTR mode turns AES (a block cipher) into a stream cipher.
  → Each block is independent — parallelizable.

Authentication side (GHASH):
  H = AES(key, 0)   ← a fixed hash subkey
  Tag is computed as a polynomial over GF(2^128) covering:
    AAD blocks + ciphertext blocks + length fields
  → Tag = GHASH(H, AAD, ciphertext) XOR AES(key, CTR0)

What hardware acceleration means:
  Modern CPUs have AES-NI instructions (AES round in 1 CPU cycle)
  and PCLMULQDQ for carry-less multiplication (GHASH).
  → AES-GCM runs at near memory-bandwidth speed on modern hardware.
  → This is why AES-GCM-SHA256 is the default TLS 1.3 cipher suite.

Critical nonce rule for AES-GCM: The nonce MUST be unique for every single record encrypted under the same key. If you ever reuse a (key, nonce) pair:

ciphertext1 = plaintext1 XOR keystream
ciphertext2 = plaintext2 XOR keystream  ← same keystream!

ciphertext1 XOR ciphertext2 = plaintext1 XOR plaintext2

XORing two ciphertexts with the same keystream gives you the XOR of the two plaintexts — from which both can be recovered with basic frequency analysis. This is called nonce reuse and it completely destroys confidentiality and also breaks the auth tag. In TLS, the nonce is constructed from the record sequence number (which increments), preventing this automatically.

ChaCha20-Poly1305

ChaCha20-Poly1305 pairs two algorithms:

  • ChaCha20 — a stream cipher (no block structure, pure addition/rotation/XOR)
  • Poly1305 — a fast polynomial MAC (one-time MAC over GF(2^130 - 5))
ChaCha20 state:
  16 × 32-bit words arranged in a 4×4 matrix
  Initialized with: "expa" "nd 3" "2-by" "te k"  (constant)
                    + 256-bit key (8 words)
                    + 64-bit counter + 96-bit nonce (4 words)

  20 rounds of "quarter-round" operations (Add, Rotate, XOR — ARX)
  Each round mixes all 16 words → output is 64 bytes of keystream
  No lookup tables → no timing side-channels (unlike AES without hardware)

Poly1305 MAC:
  One-time key derived from the first ChaCha20 block (counter=0)
  Treats the message as a polynomial evaluated at a secret point r
  over the prime field 2^130 - 5
  → 16-byte tag

Why ChaCha20-Poly1305 is preferred on mobile/IoT:
  No AES-NI hardware needed — pure software is fast
  Constant-time by construction — immune to cache-timing attacks
  No nonce reuse risk from GCM's finicky nonce construction
  Designed by Daniel Bernstein (djb) — open, well-audited, no NIST curve concerns

AEAD in Context — How a TLS Record is Protected

Application sends: GET /api/user HTTP/1.1\r\nHost: example.com\r\n\r\n

TLS 1.3 record layer does:

  1. Append content type byte to plaintext (inner content type = 0x17 for Application Data)
  2. Construct nonce = write_iv XOR (sequence_number padded to iv_length)
     → sequence_number increments per record — never reused
  3. Build AAD = TLS record header (5 bytes, unencrypted)
  4. Run AEAD:
       (ciphertext, tag) = AES-GCM-128(write_key, nonce, plaintext+type, AAD)
  5. Send: [header][ciphertext+tag]

Receiver does:
  1. Extract header → AAD
  2. Compute nonce from sequence number
  3. Run AEAD decrypt:
       if tag mismatch → send fatal alert, close connection immediately
       if tag OK → plaintext delivered to application

BEAST (TLS 1.0 CBC) Predictable CBC IV leaks plaintext via chosen-plaintext Yes — CTR/GCM has no CBC block chaining
Lucky13 Timing difference in MAC verification after decryption Yes — tag checked before any data touches
Bit-flipping ciphertext Flip bits in ciphertext to tamper with plaintext Yes — auth tag covers ciphertext; mismatch
Record reordering / replay Reorder or duplicate TLS records Yes — nonce = seq number; replay fails tag
Truncation attack Drop the last records of a stream Yes — close_notify alert required; AAD len
Nonce reuse Reuse (key, nonce) — breaks everything No — implementation must guarantee uniqueness

The One-Line Memory Hook

AEAD = "one primitive that gives you confidentiality + integrity + authenticity
        in a single atomic operation, with no way to peel them apart"

AES-GCM  = "AES for speed on hardware — but nonce discipline is critical"
ChaCha20 = "ARX for speed without hardware — safe on any device, timing-clean"
Auth tag  = "if even one bit of ciphertext or header is tampered, the tag fails
             and the record is silently dropped before the app ever sees it"

Session Resumption (TLS 1.2)

Resumption avoids a full 2-RTT handshake on reconnection. Two mechanisms exist:

Session ID — server stores session state; client sends the ID on reconnect.
Session Tickets — server encrypts session state in a ticket and sends it to the client; client presents the ticket on reconnect; server decrypts and resumes (stateless).

Diagram F — TLS 1.2 Session Resumption

sequenceDiagram
    participant C as Client
    participant S as Server
    note over C,S: Initial full handshake
    C->>S: ClientHello
    S-->>C: ServerHello + SessionTicket
    S-->>C: Finished
    C->>S: Finished
    note over C,S: Resumed session (1-RTT)
    C->>S: ClientHello + SessionTicket
    S-->>C: ServerHello (resumed)
    S-->>C: Finished
    C->>S: Finished
    C->>S: Encrypted HTTP request
    S-->>C: Encrypted HTTP response

Security note: Session ticket keys must be rotated frequently. A leaked ticket encryption key lets an attacker decrypt resumed sessions and breaks forward secrecy for those sessions.

TLS 1.3 Deep-Dive

HKDF Key Schedule

TLS 1.3 replaces the TLS 1.2 PRF with HKDF (HMAC-based Extract-and-Expand Key Derivation Function, RFC 5869). Keys for different purposes are derived from separate stages, so a compromise of one key does not cascade.

0 (zero-input)
  │
  ▼
Extract → Early Secret
  │         └── early_data keys (0-RTT traffic keys, binder keys)
  │
  ▼ + (EC)DHE shared secret
Extract → Handshake Secret
  │         └── client_handshake_traffic_secret
  │         └── server_handshake_traffic_secret
  │
  ▼ + 0 (no more secrets)
Extract → Master Secret
            └── client_application_traffic_secret
            └── server_application_traffic_secret
            └── exporter_master_secret
            └── resumption_master_secret

Each arrow is an HKDF-Extract or HKDF-Expand step using the current secret and a context label derived from a hash of the entire handshake transcript. This means handshake and application keys are cryptographically separate.

Encrypted Extensions

In TLS 1.2, extensions such as ALPN responses and supported groups appear in plaintext ServerHello. In TLS 1.3, the server sends an EncryptedExtensions message immediately after ServerHello and under the handshake encryption key, hiding:

  • ALPN-selected protocol (e.g. h2)
  • Server-accepted groups
  • Record size limits
  • Any custom extensions

This reduces metadata leakage to passive observers who can only see the ClientHello and the bare ServerHello key share.

0-RTT (Early Data)

0-RTT lets the client send application data in the very first flight alongside ClientHello, using keys derived from a previous session's resumption master secret.

Diagram G — TLS 1.3 0-RTT Resumption

sequenceDiagram
    participant C as Client
    participant S as Server
    note over C,S: Previous session established resumption_master_secret
    C->>S: ClientHello + early_data (0-RTT request)
    S-->>C: ServerHello + EncryptedExtensions
    S-->>C: Finished (encrypted)
    C->>S: EndOfEarlyData
    C->>S: Finished
    S-->>C: Encrypted HTTP response

Replay risk: The server has no guarantee of request freshness for early data. A network attacker can capture and re-send the 0-RTT flight. Mitigations:

  • Use 0-RTT only for idempotent operations (GET, read-only APIs).
  • Server-side replay detection using single-use tokens or timestamps.
  • anti_replay extension for per-ticket use tracking.

TLS 1.3 Cipher Suite Detail

TLS_AES_128_GCM_SHA256
 │    └── AEAD: AES-128-GCM
 └── Protocol: TLS 1.3

Key exchange and authentication are no longer encoded in the suite name —
they are always ephemeral (EC)DHE for key exchange, and the auth algorithm
is determined by the certificate type (RSA or ECDSA).

Certificate Chain and Trust Model

Chain Structure

A TLS certificate chain has three layers. The server presents the end-entity certificate and intermediate(s); the root is in the client trust store and never sent over the wire.

Root CA (self-signed, in trust store)
  └── Intermediate CA (signed by Root)
        └── End-Entity Certificate (signed by Intermediate, sent by server)
              ├── Subject: example.com
              ├── SANs: example.com, www.example.com
              ├── Public key
              ├── Validity: notBefore / notAfter
              └── Extensions: keyUsage, extKeyUsage, OCSP URL, CT SCTs

Diagram H — Certificate Chain Validation

flowchart TD
    A[Client receives chain from server] --> B[Parse end-entity cert]
    B --> C[Match hostname to SAN or CN]
    C --> D{Hostname match?}
    D -->|No| Z[Abort: hostname mismatch]
    D -->|Yes| E[Walk chain to trusted root]
    E --> F{Root in trust store?}
    F -->|No| Y[Abort: untrusted chain]
    F -->|Yes| G[Check each cert validity period]
    G --> H{All valid?}
    H -->|No| X[Abort: expired cert]
    H -->|Yes| I[Check revocation via OCSP or CRL]
    I --> J{Revoked?}
    J -->|Yes| W[Abort: revoked cert]
    J -->|No| K[Check CT SCTs if required]
    K --> L[Trust established]

Hostname Validation

Hostname validation checks the certificate's Subject Alternative Names (SANs) against the requested hostname:

  • Exact match: example.com matches SAN example.com
  • Wildcard: *.example.com matches api.example.com but not sub.api.example.com
  • The legacy CN field is deprecated for hostname matching (RFC 2818)

Common failures:

Failure Example Impact
CN-only cert, no SANs Old internal certs Browsers and modern clients reject
Wildcard too broad *.com Rejected by browsers
Wrong SAN Cert for internal.corp served to api.example.com Hostname mismatch error
Disabled validation curl -k, custom HTTP client Full MITM exposure

Certificate Transparency (CT)

CT requires CAs to log every issued certificate to public, append-only Merkle tree logs. Browsers enforce that end-entity certs carry Signed Certificate Timestamps (SCTs) proving the cert was logged.

CA issues cert → Submits to CT log → Log returns SCT
Server includes SCTs in TLS handshake (via extension or OCSP staple)
Browser verifies SCT signatures → confirms cert was publicly logged
Domain owner monitors CT logs → detects unexpected issuance

Why it matters: CT does not prevent misissuance, but it makes it detectable. Any unauthorized cert for example.com will appear in public logs, allowing the domain owner to revoke it quickly.

OCSP Stapling

OCSP stapling lets the server attach a pre-fetched, CA-signed revocation response to the TLS handshake instead of requiring the client to make a live OCSP request:

Without stapling:  Client → OCSP responder (real-time, leaks hostname to CA)
With stapling:     Server fetches OCSP response periodically → staples to handshake

Benefits: faster handshake, no privacy leak to CA, more reliable (no dependency on CA uptime during TLS).

HSTS and Downgrade Attacks

How SSL Stripping Works

SSL stripping (Moxie Marlinspike, 2009) exploits the fact that users often type bare domain names or click http:// links. A network attacker positioned as MITM intercepts the HTTP request before the browser upgrades to HTTPS:

Diagram I — SSL Stripping Attack

sequenceDiagram
    participant U as User Browser
    participant A as Attacker MITM
    participant S as Server
    U->>A: HTTP GET http://bank.com/
    A->>S: HTTPS GET https://bank.com/
    S-->>A: HTTPS 200 OK (secure page)
    A-->>U: HTTP 200 OK (downgraded page, links rewritten to http)
    note over U: User sees no HTTPS, attacker sees all traffic
    U->>A: HTTP POST credentials
    A->>S: HTTPS POST credentials

HSTS — How It Defends

The Strict-Transport-Security response header instructs browsers to always use HTTPS for the domain for the duration of max-age:

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Directive Meaning
max-age=N Enforce HTTPS for N seconds from last seen header
includeSubDomains Apply to all subdomains (e.g. api.example.com)
preload Signal intent to be included in browser HSTS preload list

After the browser stores an HSTS entry, any attempt to load http://example.com is internally rewritten to https://example.com — the HTTP request never leaves the browser, so the attacker has no interception opportunity.

Diagram J — HSTS Protection Flow

flowchart TD
    A[User types example.com] --> B{HSTS entry in browser?}
    B -->|No| C[HTTP request sent]
    C --> D[Server 301 redirect to HTTPS]
    D --> E[HTTPS connection]
    E --> F[Browser stores HSTS header]
    B -->|Yes| G[Browser rewrites to HTTPS internally]
    G --> E

HSTS Preload

The HSTS preload list is a hardcoded list of domains shipped with browsers. Sites on the list get HSTS enforcement from the very first visit, eliminating the TOFU (trust on first use) window.

  • Submit at: https://hstspreload.org
  • Requires max-age >= 31536000, includeSubDomains, and preload directive
  • Caution: Removal takes months to propagate; any HTTPS breakage for a preloaded domain is hard to recover from

Downgrade Attack Variants

Attack Mechanism Mitigation
SSL Stripping MITM rewrites HTTPS links to HTTP HSTS, HSTS preload

POODLE (CVE-2014-3566) Forced SSL 3.0 fallback, padding oracle Disable SSL 3.0; TLS_FALLBACK_SCSV

TLS Termination Patterns

TLS termination determines where the encrypted channel ends and how traffic flows internally. Each pattern has different security, performance, and operational trade-offs.

Pattern 1 — Edge Termination (TLS Offload)

The most common pattern for web services. TLS is terminated at a load balancer, CDN, or API gateway. Internal traffic between the load balancer and backend is plain HTTP or re-encrypted separately.

Diagram K — Edge Termination

flowchart LR
    C[Client] -->|HTTPS| LB[Load Balancer / CDN]
    LB -->|HTTP internal| B1[Backend 1]
    LB -->|HTTP internal| B2[Backend 2]
    LB -->|HTTP internal| B3[Backend 3]

Pros: Centralized certificate management, offloads TLS CPU from backends, enables HTTP/2 multiplexing, caching, and inspection.
Cons: Internal traffic is plaintext — trust the internal network. Headers like X-Forwarded-For and X-Forwarded-Proto can be spoofed if not sanitized. Lateral movement within the VPC exposes backend data.

Pattern 2 — End-to-End Encryption (Re-encryption)

TLS is terminated at the edge and a new TLS session is established to the backend. The edge issues its own internal certificate or trusts an internal CA.

Diagram L — End-to-End Re-encryption

flowchart LR
    C[Client] -->|HTTPS TLS1| LB[Load Balancer]
    LB -->|HTTPS TLS2| B1[Backend 1]
    LB -->|HTTPS TLS2| B2[Backend 2]

Pros: Protects traffic even if the internal network is compromised. Meets compliance requirements (PCI-DSS, HIPAA) that mandate encryption in transit end to end.
Cons: More CPU overhead. Internal certificate lifecycle must be managed. Backend must handle TLS.

Pattern 3 — Mutual TLS (mTLS)

Both the client and server present certificates, so the connection is mutually authenticated. Common in service-to-service communication (microservices, zero-trust networks).

Diagram M — mTLS Handshake

sequenceDiagram
    participant C as Client Service
    participant S as Server Service
    C->>S: ClientHello
    S-->>C: ServerHello + Server Certificate
    S-->>C: CertificateRequest
    C->>S: Client Certificate
    C->>S: CertificateVerify (signed with client private key)
    C->>S: Finished
    S-->>C: Finished
    note over C,S: Both peers authenticated
    C->>S: Encrypted request
    S-->>C: Encrypted response

Pros: Strong peer authentication without API keys or tokens in headers. Eliminates spoofed X-Client-ID or similar header abuse. Natural fit for service mesh (Istio, Linkerd).
Cons: Certificate provisioning and rotation for every service identity. Certificate revocation in dynamic environments requires short-lived certs or a fast revocation path.

Pattern 4 — TLS Passthrough

The load balancer forwards the raw TLS stream to the backend without decrypting it. The backend terminates TLS itself.

Diagram N — TLS Passthrough

flowchart LR
    C[Client] -->|HTTPS raw stream| LB[Load Balancer L4]
    LB -->|HTTPS raw stream| B1[Backend 1]

Pros: End-to-end encryption to the origin. Certificate and private key stay on the backend only.
Cons: Load balancer cannot inspect, route by HTTP headers, or cache. SNI-based routing is possible at L4 but no L7 visibility.

Top 3 Critical Mitigations

1. Automate Certificate Lifecycle Management

The most common HTTPS failure in production is not cryptographic breakage but certificate expiration or bad certificate rollout. Automated issuance, renewal, staged deployment, and alerting reduce the operational risk of outages and trust failures. The main trade-off is that automation itself must be secured because compromised issuance pipelines can mint trusted certificates for attackers. In practice, teams pair automation with inventory management, renewal testing, and tight control of ACME or CA credentials.

2. Enforce Modern TLS and HSTS

Modern TLS configuration means disabling obsolete protocol versions and weak ciphers, preferring TLS 1.3 where possible, and using secure defaults at the edge. HSTS closes an important downgrade window by telling browsers to refuse future HTTP access for the site after a successful HTTPS visit. The trade-off is that HSTS can make recovery harder if certificate deployment breaks, especially when preload is used, so rollout has to be deliberate. Together, these controls substantially reduce downgrade and SSL stripping exposure.

3. Protect Keys and Validate Trust End to End

HTTPS is only as strong as the private keys and trust decisions behind it. Keys should be stored in HSM-backed or similarly hardened systems, access should be tightly controlled, and rotation plans should be rehearsed. On the client side, strict certificate and hostname validation prevents trust bypasses that would silently defeat TLS. The trade-off is operational complexity across heterogeneous clients, proxies, and service meshes, but without this discipline HTTPS can become security theater rather than real protection.

Practice Questions

Tier 1 — Fundamentals

  1. What does HTTPS add on top of HTTP?
    HTTPS adds transport security through TLS, giving confidentiality, integrity, and authenticated server identity while preserving standard HTTP semantics. The application still uses methods, headers, status codes, and bodies in the same way, but they are protected from passive eavesdropping and many active tampering attacks. This is asked to confirm the candidate understands that HTTPS is not a separate application protocol so much as HTTP over a secure channel.

  2. Why does certificate validation matter in addition to encryption?
    Encryption alone does not prove who is on the other end of the connection; without certificate validation, an attacker could establish an encrypted channel and still impersonate the server. Certificate validation binds the server identity to a chain trusted by the client and checked against hostname expectations. This is asked because many candidates can describe encryption but miss the authentication side of TLS.

  3. What problem does HSTS solve?
    HSTS tells compliant browsers to always use HTTPS for a domain after it has been seen securely, which reduces downgrade and SSL stripping risks. Without HSTS, a user can still be tricked into making an initial HTTP request that is redirected or intercepted. This is asked to check whether the candidate understands practical browser trust behavior, not just TLS handshake theory.

  4. Why is forward secrecy valuable in HTTPS?
    Forward secrecy ensures that if a server's long-term private key is compromised later, previously captured traffic still cannot be decrypted because session keys were derived ephemerally. This materially limits the blast radius of key compromise for past traffic. This is asked to distinguish shallow TLS familiarity from real cryptographic threat modeling.

  5. What is mixed content and why is it dangerous?
    Mixed content happens when a secure page loads some resources over insecure HTTP, which can let attackers tamper with scripts, styles, or media. This undermines the security guarantees users think they have from the lock icon. This is asked because browser-facing HTTPS is about the full content graph, not just the main HTML document.

Tier 2 — System Design and Trade-offs

  1. How would you design HTTPS termination for a global service with many edge locations?
    I would centralize certificate policy while distributing termination to edge points such as CDNs or regional load balancers close to users. Certificate issuance and renewal would be automated, keys would ideally live in hardware-backed systems, and internal hops would be re-encrypted or mutually authenticated rather than assumed safe. I would also standardize TLS policy, observability, and rollback across regions. This is asked to test whether the candidate can scale trust management, not just explain TLS packets.

  2. What are the trade-offs between terminating TLS at the edge versus end to end at the origin?
    Edge termination improves latency, offloads CPU, simplifies caching, and gives a common place for policy enforcement. End-to-end protection to the origin reduces trust in internal networks and limits the damage of compromised intermediaries. The real trade-off is between operational simplicity and trust minimization, and mature platforms often do both by terminating at the edge and re-encrypting internally. This is asked to evaluate architectural judgment across security and performance.

  3. How would you detect and respond to certificate misissuance for your domain?
    I would monitor Certificate Transparency logs continuously, alert on unexpected issuance patterns, and maintain a fast incident path to revoke and replace certificates if something suspicious appears. I would also use CA restrictions and scoped issuance policies where possible. The goal is to detect impersonation opportunities quickly before they become active MITM incidents. This is asked because strong candidates know HTTPS security extends into PKI operations.

  4. Why can HTTPS still leave users exposed even when the handshake is technically correct?
    A correct handshake does not prevent XSS, CSRF, bad CORS policy, weak session handling, or insecure logic in the application itself. It also does not help if the browser trusts a malicious enterprise root, the user accepts warnings, or the page loads insecure subresources. HTTPS protects the channel, not the entire application stack. This is asked to see whether the candidate can separate transport security from application security.

  5. How would you optimize HTTPS performance without weakening security?
    I would use TLS 1.3, session resumption, OCSP stapling, CDN edge termination, HTTP/2 or HTTP/3 where appropriate, and careful certificate-chain optimization. I would also monitor handshake latency, cache hit rates, and CPU cost at termination points. The goal is to reduce repeated handshake overhead and network round trips while keeping strong ciphers and validation intact. This is asked to test whether the candidate can balance security and latency in production.

Tier 3 — Expert and Deep-Dive

  1. If the network path is encrypted, why do enterprises still deploy TLS interception, and what risks does that create?
    Enterprises intercept TLS to inspect content for malware, data exfiltration, policy violations, and regulated traffic patterns that would otherwise be opaque. They usually do this by installing a trusted enterprise root CA on managed endpoints and dynamically re-signing certificates at the proxy. The risks are substantial: the proxy becomes a high-value trust anchor, private traffic may be exposed internally, certificate validation assumptions change, and some applications break or lose end-to-end assurances. This is asked because senior engineers must reason about the difference between nominal encryption and actual trust boundaries.

  2. How would you reason about the security of a service that uses HTTPS externally but plain HTTP internally inside a cloud VPC?
    I would treat that as a risk decision, not automatically safe by default. Internal networks are broader attack surfaces than many teams assume, and plaintext internal hops can expose credentials, session cookies, or internal API data to compromised workloads, misrouted traffic, or logging mistakes. The right answer depends on environment maturity, segmentation, identity, and threat model, but the secure default is to re-encrypt or use mutual authentication internally as well. This is asked to test whether the candidate can challenge convenient but weak assumptions about trusted networks.

  3. What does Certificate Transparency change about the HTTPS trust model?
    Certificate Transparency adds public, append-only visibility into certificate issuance so domain owners and browsers can detect suspicious or unauthorized certificates. It does not replace CAs, but it makes CA behavior auditable and therefore less opaque. This shifts the model from pure trust in issuers toward trust plus verification and monitoring. This is asked because strong candidates understand how operational controls reshape protocol trust in practice.

  4. Why is hostname validation just as important as chain validation?
    A certificate can chain to a trusted root and still be wrong for the site the client intended to reach. Hostname validation checks whether the certificate's subject alternative names actually match the requested domain, which is what prevents a valid certificate for one domain from impersonating another. Without hostname checks, a large part of PKI becomes meaningless. This is asked to catch candidates who know certificate chains but not endpoint identity verification.

  5. How would you explain the difference between HTTPS, TLS, and HTTP/3 to a senior engineer designing a new platform?
    TLS is the cryptographic protocol that authenticates peers and protects data in transit. HTTPS is HTTP carried over a protected channel, traditionally TLS over TCP, while HTTP/3 is the newest major HTTP mapping that runs over QUIC on UDP and integrates modern transport behavior with strong security. The architectural point is that application semantics, cryptographic protection, and transport behavior are related but distinct design layers. This is asked because platform engineers need to reason clearly across those layers without conflating them.

Key Takeaways

  • HTTPS is HTTP over a cryptographically protected channel, not a different web application protocol with new semantics.
  • Certificate validation, hostname checks, and trust-store hygiene are as important as encryption strength.
  • HSTS materially reduces downgrade and SSL stripping risk, especially for browser-facing applications.
  • The most common real-world HTTPS outages come from certificate lifecycle failures, not broken cryptography.
  • Edge termination improves performance, but trust boundaries after termination still need protection and explicit design.
  • HTTPS protects the transport channel, but application-layer attacks such as XSS, CSRF, and broken auth still remain.
  • Senior engineers should be able to reason about HTTPS across cryptography, PKI operations, browser behavior, and platform architecture.
