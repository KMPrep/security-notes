RDP — Remote Desktop Protocol

OSI Layer: 7 — Application
Standard: Microsoft proprietary, influenced by T.128 family concepts

Overview

RDP is a remote desktop protocol that lets a client interact with a graphical session on a remote system. Rather than transferring only files or terminal text, it sends display updates, keyboard and mouse input, clipboard data, and optionally redirected devices between client and server. RDP is heavily used for Windows administration, remote support, VDI, and jump-host workflows. Because it exposes interactive login capability and powerful post-auth access, it is one of the most heavily targeted enterprise remote-access surfaces.

How It Works

Core Mechanics

An RDP client connects to the server, negotiates capabilities such as graphics, channels, and security settings, and then authenticates. Modern deployments commonly use Network Level Authentication so credentials are validated before the full desktop session is created. Once the session is established, the server sends encoded display updates while the client sends input events back.

RDP can use TCP and in some cases UDP for performance-related channels. Features such as clipboard sharing, printer mapping, drive mapping, smart card redirection, and audio redirection make the protocol very usable, but they also expand the trust boundary. In secure environments, the protocol should be exposed only behind strong access controls such as VPNs, bastions, or zero-trust brokers.

Top 3 Critical Mitigations

1. Never Expose RDP Directly Without Strong Access Controls

Directly exposed RDP is one of the most common enterprise footholds for brute force, credential-stuffing, and exploit attempts. Moving RDP behind VPNs, brokers, or zero-trust gateways dramatically reduces attack surface. The trade-off is extra access infrastructure, but the security benefit is obvious.

2. Treat Administrative Endpoints as Part of the RDP Threat Model

The security of RDP depends not just on the server but also on the client workstation and credential path. A compromised admin laptop can turn a legitimate RDP workflow into a privileged intrusion vector. Managed admin workstations and strong credential hygiene are therefore core controls.

3. Minimize Session Features and Administrative Privilege

Redirection, broad admin rights, and unrestricted target access all increase blast radius. A hardened RDP baseline should disable unnecessary features and constrain who can reach which systems. The trade-off is some usability loss, but it materially reduces risk.

Practice Questions

Tier 1 — Fundamentals

  1. What is RDP used for? It provides remote graphical desktop access and interaction. This is asked to confirm basic purpose.
  2. Why is RDP considered a major attack surface? It grants interactive remote access and is commonly targeted for brute force and exploitation. This is asked to test security awareness.
  3. What does Network Level Authentication improve? It authenticates the user before creating the full remote session. This is asked to test protocol-hardening knowledge.
  4. What port is RDP commonly associated with? Port 3389. This is asked for operational familiarity.
  5. Why can clipboard or drive redirection be risky? They can move sensitive data or malware across the session boundary. This is asked to test feature-level risk awareness.

Tier 2 — System Design and Trade-offs

  1. How would you design secure RDP access for administrators? Use a VPN or access broker, MFA, managed admin devices, logging, and least-privilege authorization. This is asked to test practical access design.
  2. Why is direct internet exposure of RDP so dangerous? It invites brute-force attempts and exposes the service to pre-auth vulnerability exploitation. This is asked to test defensive judgment.
  3. How would you investigate suspicious RDP logons? Inspect source paths, auth events, session history, endpoint posture, and subsequent lateral activity. This is asked to test incident-handling structure.
  4. What are the trade-offs of disabling redirection features? Security improves, but usability for support and workflows may decrease. This is asked to test balanced engineering reasoning.
  5. How would you reduce ransomware risk related to RDP? Reduce exposure, harden identities, segment targets, and monitor admin-session behavior. This is asked to test threat-focused design.

Tier 3 — Expert and Deep-Dive

  1. Why is RDP security really an identity and endpoint problem as much as a protocol problem? The protocol can be encrypted and still be dangerous if the identities or admin workstations are weak. This is asked to test layered thinking.
  2. How would you threat-model a bastion-based RDP environment? I would treat the bastion, identity provider, admin endpoints, and downstream hosts as critical trust boundaries. This is asked to test structured security reasoning.
  3. What are the hardest operational problems in large RDP estates? Session exposure, patch cadence, privilege control, endpoint trust, and auditability are the hardest recurring themes. This is asked to test production maturity.
  4. Why can NLA be necessary but still insufficient? It reduces pre-session exposure but does not solve weak credentials, overbroad access, or compromised admin devices. This is asked to test nuanced security judgment.
  5. What does mature RDP hygiene look like? No direct exposure, strong identity controls, hardened clients and bastions, minimized redirection, and good telemetry. This is asked to test platform-level maturity.

Key Takeaways

  • RDP is remote graphical access and a major administrative trust boundary.
  • Direct exposure of RDP is a high-risk design choice.
  • NLA is important but not sufficient by itself.
  • Admin workstation security is part of RDP security.
  • Redirection features expand the trust boundary.
  • RDP is a common lateral-movement and ransomware path.
  • Mature RDP deployments rely on strong access architecture, not just protocol settings.
