SFTP — SSH File Transfer Protocol

OSI Layer: 7 — Application
Standard: SSH File Transfer Protocol draft specifications, commonly implemented via SSH2

Overview

SFTP is a file-transfer protocol that runs over an SSH channel and provides encrypted, authenticated file operations over a single connection. Despite the similar name, it is not FTP with SSH added; it is a distinct binary protocol with its own commands and server behavior. SFTP supports upload, download, directory listing, file metadata changes, rename, delete, and other remote filesystem operations. Because it reuses SSH’s secure transport and single-port model, it is far easier to deploy safely than plain FTP.

How It Works

Core Mechanics

An SFTP session begins with an SSH connection, usually to port 22. After SSH transport encryption and authentication are established, the client requests the SFTP subsystem. Once accepted, the client and server exchange SFTP protocol messages over that SSH channel.

SFTP operations are request-response oriented and carry message IDs so multiple operations can be outstanding at once. Common operations include open, read, write, close, stat, setstat, opendir, readdir, rename, and remove. Because everything runs inside SSH, confidentiality, integrity, host authentication, and user authentication are inherited from the SSH session. Access controls therefore depend both on the SFTP server implementation and on underlying operating-system permissions.

Top 3 Critical Mitigations

1. Treat SFTP as SSH Security Plus File Workflow Security

The transport is only one part of the problem. Even with strong SSH cryptography, weak account design, overbroad directories, or unsafe ingestion pipelines can make SFTP dangerous. The best control set combines SSH hardening with workflow-aware validation and auditing.

2. Manage Keys and Host Trust Properly

SFTP is often automated, which means private keys end up in scripts, jobs, and service accounts. Centralized key management, rotation, and strict host-key validation reduce both credential theft and man-in-the-middle risk. The trade-off is more lifecycle management, but that is necessary for long-term safety.

3. Enforce Per-User Least Privilege

SFTP frequently sits at the boundary of important data exchange processes. Directory jails, per-user permissions, quotas, and strong attribution reduce the damage from compromised accounts or operational mistakes. The trade-off is more access-management work, but it pays off during incidents and audits.

Practice Questions

Tier 1 — Fundamentals

  1. Is SFTP the same thing as FTP over SSH? No, it is a distinct protocol carried inside SSH. This is asked because many candidates conflate the names.
  2. Why is SFTP easier to operate than FTP? It uses one encrypted connection instead of separate control and data channels. This is asked to test deployment understanding.
  3. What default port does SFTP typically use? Port 22 through SSH. This is asked for operational familiarity.
  4. What security properties does SFTP inherit from SSH? Confidentiality, integrity, host authentication, and user authentication. This is asked to connect layers correctly.
  5. Why are host keys important in SFTP? They let the client verify it is talking to the right server. This is asked to test MITM awareness.

Tier 2 — System Design and Trade-offs

  1. How would you design a secure automated SFTP ingestion pipeline? I would use dedicated service identities, strict host verification, isolated landing zones, malware scanning, and downstream validation. This is asked to test practical secure design.
  2. When might FTPS be chosen instead of SFTP? Usually when compatibility with existing FTP tooling or partner workflows dominates. This is asked to test nuanced protocol selection.
  3. How would you investigate an SFTP transfer failure? Check SSH negotiation, authentication, subsystem startup, filesystem permissions, quotas, and logs on both ends. This is asked to test troubleshooting structure.
  4. What risks come from sharing one SFTP account across many jobs? Poor attribution, excessive privilege, and large blast radius from one compromised credential. This is asked to test access-design maturity.
  5. How would you isolate customer uploads on a multi-tenant SFTP server? Per-tenant identities, separate directories or storage backends, quotas, and strong audit trails are the baseline. This is asked to test multi-tenant thinking.

Tier 3 — Expert and Deep-Dive

  1. Why can a secure transfer protocol still be dangerous in production pipelines? Because transport security does not validate the meaning or safety of the transferred data. This is asked to test end-to-end reasoning.
  2. What are the hardest enterprise problems in SFTP operations? Key lifecycle, automation identity sprawl, partner onboarding, auditability, and exception handling are the big ones. This is asked to test platform experience.
  3. How would you explain the difference between transport trust and content trust? SSH can protect the path, but you still need checks that the file itself is expected, complete, and safe to process. This is asked to test layered security thinking.
  4. What would make you replace SFTP with an API-based approach? Need for stronger workflow semantics, eventing, validation, and application-level authorization often pushes teams toward APIs. This is asked to test modernization judgment.
  5. Why are host-key prompts dangerous in automation? Teams often disable validation to avoid friction, which silently removes server authentication. This is asked to test realism about operational shortcuts.

Key Takeaways

  • SFTP is a separate protocol carried inside SSH.
  • It is much easier to secure and operate than plain FTP.
  • Host keys and key management are central security controls.
  • File-transfer security also depends on filesystem permissions and workflow validation.
  • Shared accounts and secret sprawl are common real-world problems.
  • One encrypted channel simplifies deployment and monitoring.
  • Strong transport does not remove the need for content validation.
