SMTP — Simple Mail Transfer Protocol

OSI Layer: 7 — Application
RFC: RFC 5321 (SMTP), RFC 6409 (submission), RFC 3207 (STARTTLS), RFC 4954 (AUTH)

Overview

SMTP is the standard protocol for sending and relaying email. It is used both by clients submitting outbound mail to their provider and by mail servers relaying mail toward destination domains. SMTP is a push protocol: the sender opens the connection and transfers a message to the next hop. Because SMTP was designed in a more trusting era, modern email security depends heavily on surrounding controls such as TLS, authentication, SPF, DKIM, and DMARC.

How It Works

Core Mechanics

An SMTP session starts with a TCP connection, typically on port 25 for relay, 587 for authenticated submission, or 465 for implicit TLS submission. The server sends a greeting banner, and the client responds with EHLO or HELO. A normal message transaction then uses MAIL FROM, one or more RCPT TO commands, and DATA followed by the message body and a terminating dot line.

SMTP uses reply codes to signal success, temporary failure, or permanent failure. Message delivery is store-and-forward rather than guaranteed end-to-end in one session. If a destination server is unavailable, the sending MTA queues and retries later. Extensions such as STARTTLS, AUTH, and SIZE add security and operational capabilities, but interoperability across the email ecosystem means SMTP remains full of backward-compatibility concerns.

Top 3 Critical Mitigations

1. Enforce Sender Authentication and Policy

SMTP alone does not prove that the sender domain is legitimate. SPF, DKIM, and DMARC add the controls needed to validate sending infrastructure and message integrity. The trade-off is operational complexity and the risk of mail disruption when policies are rolled out carelessly.

2. Lock Down Submission and Relay Behavior

Mail servers should relay only for explicitly trusted paths and authenticated users. Open-relay behavior is one of the fastest ways to become an abuse source and get blocklisted. The trade-off is tighter client and application requirements, but the risk reduction is large.

3. Treat Email as a Security Boundary, Not Just a Messaging Tool

Email is a delivery channel for phishing, malware, impersonation, and business-process abuse. Strong monitoring, filtering, and authentication controls are essential because SMTP is a high-volume human-facing attack surface. The trade-off is more layered infrastructure, but that is standard for serious environments.

Practice Questions

Tier 1 — Fundamentals

  1. What is SMTP used for? Sending and relaying email between clients and mail servers or between mail servers. This is asked to confirm protocol purpose.
  2. Why is SMTP called a push protocol? The sender initiates message delivery to the next hop. This is asked to test workflow understanding.
  3. What are the core SMTP commands in a basic transaction? EHLO, MAIL FROM, RCPT TO, and DATA. This is asked to test wire-level familiarity.
  4. Why is SMTP insecure by itself? It does not natively provide strong sender authentication or mandatory encryption. This is asked to connect protocol design to modern risk.
  5. What ports are commonly associated with SMTP? 25 for relay, 587 for submission, and 465 for implicit TLS submission. This is asked for operational knowledge.

Tier 2 — System Design and Trade-offs

  1. How would you secure an outbound mail submission service? I would require authenticated submission, MFA where relevant, TLS, rate controls, and abuse monitoring. This is asked to test practical service design.
  2. Why are SPF, DKIM, and DMARC separate rather than one thing? They solve related but distinct problems around sender authorization, message signing, and receiver policy. This is asked to test ecosystem understanding.
  3. How would you diagnose delayed outbound email? I would inspect queue depth, remote response codes, DNS for MX resolution, TLS negotiation, and reputation or blocklist issues. This is asked to test troubleshooting structure.
  4. Why is opportunistic TLS not the same as strict transport security? Opportunistic TLS can silently fall back when encryption fails. This is asked to test nuance around transport guarantees.
  5. How would you prevent one compromised mailbox from becoming a spam cannon? Use rate limits, anomaly detection, MFA, and fast automated containment. This is asked to test anti-abuse thinking.

Tier 3 — Expert and Deep-Dive

  1. Why is email security so distributed and operationally fragile? The protocol is federated, backward-compatible, and spans many independent domains and trust decisions. This is asked to test ecosystem-level thinking.
  2. How would you explain why SMTP is still widely used despite its age and flaws? It is deeply interoperable and globally deployed, so the ecosystem evolved compensating controls around it instead of replacing it outright. This is asked to test architectural perspective.
  3. What are the hardest production problems in large mail platforms? Abuse prevention, sender reputation, queue management, authentication policy drift, and safe content handling are among the hardest. This is asked to test real-world maturity.
  4. What is the difference between successful SMTP acceptance and successful message delivery? Acceptance means one hop took responsibility; it does not guarantee the message reached the user inbox. This is asked to test operational precision.
  5. How would you threat-model a company’s outbound email channel? I would treat it as both a brand-trust channel and a high-risk exfiltration and phishing surface. This is asked to test security framing beyond protocol syntax.

Key Takeaways

  • SMTP is the core protocol for sending and relaying mail.
  • Modern email security depends on controls around SMTP, not SMTP alone.
  • Sender authentication and relay policy are critical.
  • Opportunistic TLS is useful but not a complete security guarantee.
  • Queue behavior and DNS correctness strongly affect delivery.
  • Mail systems are high-value abuse targets.
  • Email is as much a security platform problem as a messaging problem.
