SSH — Secure Shell

OSI Layer: 7 — Application
RFC: RFC 4251-4254, RFC 5656

Overview

SSH is a secure remote-access and secure transport protocol used for shell access, command execution, tunneling, and secure file-transfer subsystems such as SFTP. It runs most commonly over TCP port 22 and replaces older plaintext remote-access protocols such as Telnet and rlogin. SSH provides confidentiality, integrity, host authentication, and user authentication, typically using a combination of asymmetric and symmetric cryptography. In modern infrastructure, SSH is both an essential administrative tool and a sensitive security boundary.

How It Works

Core Mechanics

An SSH connection begins with algorithm negotiation and key exchange between client and server. The server proves its identity through a host key, and the client verifies that key against a known trust store or administrator-approved fingerprint. After the transport is encrypted, the user authenticates using methods such as public keys, certificates, passwords, or keyboard-interactive flows.

Once authenticated, SSH multiplexes one or more channels across the secure connection. These channels can carry an interactive shell, a remote command, port forwarding, agent forwarding, or a subsystem such as SFTP. Because SSH is flexible, secure deployment depends not only on cryptographic choices but also on how features such as forwarding, root login, agent use, and key distribution are managed.

Top 3 Critical Mitigations

1. Control Identity and Key Lifecycle

Most SSH risk in production comes from identity sprawl rather than cryptographic weakness. Unmanaged keys, shared accounts, and long-lived automation credentials create silent persistence paths. Central inventory, rotation, expiry, and certificate-based or centrally approved access models reduce that risk.

2. Treat Host-Key Verification as Non-Negotiable

Host keys are how clients know they are connecting to the intended server. If users routinely accept changed or unknown keys without validation, SSH loses an essential security property. Good operations make host trust easy to manage so people do not bypass it.

3. Restrict SSH Features to What Is Actually Needed

SSH is powerful, which means it can also bypass intended controls through forwarding, subsystems, and automation. A hardened baseline disables risky features unless they are explicitly justified. The trade-off is reduced flexibility, but the benefit is a smaller attack surface.

Practice Questions

Tier 1 — Fundamentals

  1. What does SSH replace and why? It replaces plaintext remote-access tools such as Telnet by adding encryption and authentication. This is asked to confirm core purpose.
  2. What is the role of the SSH host key? It authenticates the server to the client. This is asked to test MITM understanding.
  3. Why is public-key auth usually preferred over passwords? It is harder to brute force and works better for controlled automation. This is asked to test basic security judgment.
  4. What else can SSH do besides open a shell? It can run commands, forward ports, and carry subsystems such as SFTP. This is asked to test protocol breadth.
  5. Why is port 22 security-sensitive? It is a common administrative entry point into systems. This is asked for operational awareness.

Tier 2 — System Design and Trade-offs

  1. How would you secure SSH access for a production fleet? Use bastions or centralized access, key or certificate-based auth, no direct root login, strict logging, and limited forwarding. This is asked to test practical architecture.
  2. What are the trade-offs of using SSH bastions? They improve control and visibility but create concentrated trust and operational dependency. This is asked to test systems trade-off reasoning.
  3. How would you investigate suspicious SSH usage? I would examine auth logs, source paths, key fingerprints, forwarded channels, and related command history or session recordings. This is asked to test detection thinking.
  4. Why can SSH tunneling be both useful and dangerous? It securely carries traffic across boundaries but can also bypass monitoring or segmentation. This is asked to test balanced reasoning.
  5. How would you manage SSH for automated jobs at scale? Use short-lived credentials, centralized authorization, and clear ownership rather than static private keys copied everywhere. This is asked to test automation maturity.

Tier 3 — Expert and Deep-Dive

  1. What are the hardest SSH problems in large organizations? Identity lifecycle, key sprawl, bastion design, host trust, and exception-driven forwarding are among the hardest. This is asked to test production experience.
  2. Why is SSH often a lateral-movement protocol as much as an admin tool? Once credentials or keys are stolen, it provides a clean way to move between systems. This is asked to test attacker-minded reasoning.
  3. How would you compare OpenSSH certificates to raw authorized keys operationally? Certificates centralize trust and expiry better, while raw keys are simple but sprawl quickly. This is asked to test nuanced access-control thinking.
  4. Why can perfect cryptography coexist with weak SSH security? Identity handling, trust bypasses, and feature misuse can undermine the deployment even when the algorithms are strong. This is asked to separate crypto from operations.
  5. What does a good SSH control plane look like? Centralized identity, auditable trust, restricted features, strong logging, and minimal standing access. This is asked to test architectural clarity.

Key Takeaways

  • SSH is both a secure remote shell and a general secure transport mechanism.
  • Host-key validation is essential, not optional friction.
  • Key lifecycle and account design dominate real-world SSH risk.
  • Port forwarding and agent forwarding should be tightly controlled.
  • Bastions help but must themselves be hardened.
  • Strong crypto does not compensate for poor identity hygiene.
  • SSH is a major administrative asset and a major lateral-movement surface.
