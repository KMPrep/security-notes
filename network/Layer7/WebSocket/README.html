WebSocket

OSI Layer: 7 — Application
RFC: RFC 6455

Overview

WebSocket is a persistent, full-duplex application protocol that allows a client and server to exchange messages over a single long-lived connection. It starts with an HTTP/1.1 upgrade request and, after the server accepts with 101 Switching Protocols, the connection switches from HTTP semantics to WebSocket framing. This avoids the overhead of repeated polling or repeated HTTP request-response cycles for real-time workloads such as chat, multiplayer games, collaboration tools, market feeds, and live dashboards. WebSocket commonly uses port 80 for ws and port 443 for wss, with wss adding TLS protection similar to HTTPS.

How It Works

Core Mechanics

The connection begins as an HTTP/1.1 request containing headers such as Upgrade: websocket, Connection: Upgrade, Sec-WebSocket-Key, and Sec-WebSocket-Version. If the server is willing to switch protocols, it replies with 101 Switching Protocols and returns a Sec-WebSocket-Accept value derived from the client key plus a fixed GUID. After the handshake, both peers exchange WebSocket frames instead of HTTP messages.

WebSocket frames include an opcode, payload length, masking information, and the payload itself. Browsers must mask client-to-server frames, while server-to-client frames are not masked. Control frames such as ping, pong, and close manage liveness and shutdown, while data frames carry text or binary application messages. Fragmentation is allowed, so a logical message can span multiple frames.

The protocol is transport-neutral at the application layer but is almost always carried over TCP. Because the connection stays open, the application must manage authentication lifetime, message ordering assumptions, backpressure, idle timeouts, reconnection logic, and per-connection resource usage. In production, WebSocket is often terminated at load balancers or reverse proxies, which means timeout configuration and sticky-session or fan-out architecture decisions matter.

Top 3 Critical Mitigations

1. Enforce Resource Controls Per Connection

WebSocket’s biggest operational risk is that open connections consume memory, file descriptors, buffers, and scheduler attention even when traffic is low. Per-client caps, queue bounds, frame-size limits, and idle connection eviction stop a small set of clients from degrading the whole service. The trade-off is that aggressive limits can disconnect legitimate bursty users, so thresholds need to be based on observed traffic rather than guesses. In production, these controls usually belong at both the edge and the application layer.

2. Treat Authentication as a Connection Lifecycle Problem

A WebSocket connection may live far longer than the token or cookie used to establish it, so auth cannot be treated as a one-time check only at upgrade time. Strong designs bind identity to the connection, revalidate for long sessions when needed, and apply authorization checks to each message type or logical channel. The trade-off is additional complexity in reconnect and token refresh flows, but without it long-lived sessions become a quiet privilege retention bug.

3. Design for Backpressure and Failure Recovery

Real-time systems fail badly when producers can write faster than consumers can handle messages. Bounded queues, explicit drop behavior, room-level fan-out controls, and reconnect-aware clients help the system degrade predictably under stress instead of collapsing. The trade-off is that some messages may be dropped or coalesced, so product requirements must define which updates are lossy and which require durable recovery. This is one of the clearest lines between toy WebSocket systems and production-grade ones.

Practice Questions

Tier 1 — Fundamentals

  1. How does a WebSocket connection start if it is not pure HTTP after setup?
    It begins as an HTTP/1.1 upgrade request, and the server replies with 101 Switching Protocols if it accepts the transition. After that point, the connection no longer carries regular HTTP messages and instead carries WebSocket frames. This is asked to check whether the candidate understands the handshake boundary and the relationship between HTTP and WebSocket.

  2. Why is WebSocket better than polling for some workloads?
    Polling wastes bandwidth and server work because clients repeatedly ask for updates even when nothing changed. WebSocket keeps one connection open so the server can push data as events happen, which reduces latency and protocol overhead for real-time use cases. This is asked to verify the candidate can justify protocol choice from first principles.

  3. What is the purpose of ping and pong frames?
    Ping and pong frames are control frames used to check liveness and keep intermediaries or applications from treating a connection as idle forever. They help detect dead peers and support timely cleanup of stale sessions. This is asked because many candidates know WebSocket is persistent but forget that persistence requires active liveness management.

  4. Why are browser client frames masked?
    Browser clients mask frames to prevent certain classes of proxy cache poisoning and cross-protocol confusion attacks involving crafted wire bytes. The server must unmask payloads before processing them. This is asked to test whether the candidate knows WebSocket framing details beyond the basic handshake.

  5. What is the difference between ws and wss?
    ws is unencrypted WebSocket over plain TCP, while wss is WebSocket carried over TLS and is the appropriate default for production internet-facing systems. wss protects confidentiality and integrity similarly to HTTPS. This is asked because transport security should be an automatic part of protocol reasoning.

Tier 2 — System Design and Trade-offs

  1. How would you scale a WebSocket chat service to millions of concurrent clients?
    I would separate connection termination from message distribution, using horizontally scaled gateway nodes to hold sockets and a pub-sub or event backbone to distribute room events. Presence, subscription state, and fan-out targets would need to be partitioned by shard key such as room or tenant. I would also define resource budgets per connection and per room so hotspots do not collapse a whole region. This is asked to see whether the candidate can move from protocol mechanics to scalable architecture.

  2. What makes load balancing WebSocket harder than ordinary stateless HTTP?
    WebSocket connections are long-lived, so balancing happens at connection establishment time and may remain skewed for long periods. Stateful fan-out, subscription ownership, and reconnection storms can create imbalances that simple per-request round robin does not solve. This is asked to test whether the candidate understands how long-lived state changes the architecture.

  3. How would you handle token expiry on a connection that stays open for hours?
    I would decide explicitly whether the connection must be re-authenticated, refreshed in-band, or torn down and re-established when the token expires. Critical systems should avoid assuming that initial authentication is valid forever just because the TCP socket remains alive. The exact design depends on user experience and threat model, but silent indefinite trust is usually the wrong answer. This is asked because session lifetime bugs are common in real-time systems.

  4. When would you choose SSE or gRPC streaming instead of WebSocket?
    SSE is simpler for one-way server-to-client updates in browser environments and works well through ordinary HTTP infrastructure. gRPC streaming is attractive when both endpoints are controlled services and you want strong schema tooling and RPC semantics. WebSocket is best when true bidirectional browser-server messaging is required. This is asked to see whether the candidate can choose the right tool rather than force everything into WebSocket.

  5. How would you defend a WebSocket service against connection floods and message floods at the same time?
    I would split the problem into admission control for new connections and runtime controls for message throughput after admission. At the edge I would use IP, user, and token-based rate limits plus handshake validation; inside the service I would enforce queue bounds, frame-size limits, and room-level publish budgets. This is asked to evaluate whether the candidate can separate phases of the protocol and defend each appropriately.

Tier 3 — Expert and Deep-Dive

  1. Why is backpressure often the hardest real engineering problem in WebSocket systems?
    The protocol itself allows fast producers and slow consumers to stay connected while buffers quietly accumulate, which turns memory and latency into failure surfaces. Without explicit policies, a single slow client can either force unbounded buffering or cause shared fan-out work to lag behind real time. Production systems need clear decisions about dropping, coalescing, replaying, or disconnecting under pressure. This is asked because it reveals whether the candidate understands operational mechanics rather than just API usage.

  2. How would you design message delivery semantics for a financial feed or collaborative editor over WebSocket?
    I would make delivery guarantees explicit in the application protocol because WebSocket itself does not provide domain-level exactly-once or replay semantics. For a financial feed, sequence numbers and gap detection are essential; for a collaborative editor, operation IDs, versioning, and conflict resolution matter more than raw transport persistence. The design must specify what the client does after reconnect, missed messages, or out-of-order application events. This is asked to assess whether the candidate can build a reliable protocol on top of a raw bidirectional channel.

  3. What trust boundaries change when a browser opens a WebSocket to your backend?
    The browser becomes a long-lived bidirectional endpoint rather than a short request initiator, which expands the surface for auth drift, origin confusion, CSRF-like upgrade abuse, and client-driven resource retention. Intermediaries such as CDNs and load balancers may support the upgrade differently from regular HTTP and may apply different timeout behavior. This is asked to see whether the candidate can reason about the whole path, not just the application code.

  4. If your WebSocket fleet suffers a region-wide reconnect storm, what should your architecture do?
    It should shed work predictably instead of allowing all clients to reconnect immediately and overwhelm the control plane. Techniques include jittered exponential backoff on clients, admission throttles, warm standby capacity, and separating connection acceptance from expensive subscription restoration. This is asked because the hardest failures in real-time systems often happen during recovery, not steady state.

  5. What are the limits of WebSocket as a protocol abstraction?
    WebSocket gives you a bidirectional frame transport, not a complete messaging system with delivery guarantees, durable replay, schema enforcement, authorization semantics, or backpressure policy. All of those concerns must be designed by the application or platform built on top of it. This is asked because experienced engineers know where the protocol stops and where system design begins.

Key Takeaways

  • WebSocket starts as an HTTP upgrade but becomes a separate framed protocol after the server sends 101 Switching Protocols.
  • It is a strong fit for low-latency bidirectional communication, but it pushes more state-management responsibility onto the application.
  • wss should be the default for production because long-lived plaintext connections are easy to intercept and tamper with.
  • Real production risks are usually resource exhaustion, auth drift, timeout mismatch, and backpressure failure rather than handshake syntax.
  • WebSocket itself does not define business-level delivery guarantees, replay semantics, or reconnection behavior.
  • Load balancing and scaling are harder than stateless HTTP because connections are long-lived and often tied to fan-out state.
  • Strong WebSocket systems are designed as protocols and platforms, not just as open sockets with JSON messages.
