Tunneling — Encapsulation, VPNs, and Covert Channels

OSI Layer: Cross-layer (tunnels exist at L2, L3, L5, and L7)
Related RFCs/specs: RFC 2784 (GRE), RFC 4301/4303 (IPsec/ESP), RFC 2637 (PPTP), RFC 8446 (TLS 1.3), RFC 9000 (QUIC), WireGuard whitepaper
Related notes: VPN · IPsec · GRE · PPTP · SSH · BeyondCorp / ZTNA

Overview

Tunneling is encapsulating one network protocol inside another so that packets can traverse a network that would otherwise not carry, permit, or protect them. The original packet (the payload or passenger protocol) is wrapped in a new header (the delivery protocol) at one endpoint, routed across the intermediate network, and unwrapped at the far endpoint. Tunneling is the mechanism underneath most VPNs, and it is equally a favorite technique of attackers, who tunnel command-and-control (C2) and data exfiltration inside protocols firewalls are forced to allow (DNS, HTTPS). The same primitive — "hide traffic X inside traffic Y" — is defensive or offensive depending on who controls the endpoints, which is exactly why it shows up on both sides of a security engineering.

Because encapsulation can happen at any layer, tunnels are best understood by three questions: what is encapsulated, what carries it, and what (if anything) protects it. A tunnel that encapsulates but does not authenticate or encrypt (GRE, plain IP-in-IP) provides connectivity, not security; security comes from pairing encapsulation with cryptography (IPsec, TLS, WireGuard).

How It Works

Core Mechanics

Every tunnel does the same three things: encapsulate (prepend a delivery header to the original packet), transport (route the outer packet by its new header across the underlay), and decapsulate (strip the outer header and inject the original packet into the far network). The endpoints share tunnel state — addresses, keys, selectors, and sequence numbers — so both sides agree on what is inside the wrapper.

Two orthogonal design axes matter for real systems:

  • Tunnel mode vs transport mode (IPsec framing, but the idea generalizes). Tunnel mode wraps the entire original IP packet in a new IP header — used gateway-to-gateway (site-to-site VPN), and it hides the original addresses. Transport mode protects only the payload, leaving the original IP header — used host-to-host, lower overhead, but exposes endpoints.
  • Full tunnel vs split tunnel (remote-access VPN routing). Full tunnel sends all client traffic through the tunnel (maximum visibility/control, higher latency and load). Split tunnel sends only selected routes through the tunnel and the rest direct (better performance, but a misconfigured split leaks sensitive traffic outside the protected path).

Deeper context — top 3 controls

1. Zero-trust authorization behind the tunnel (defeats over-trust + lateral movement). The single most important mindset shift: tunnel access is transport, not authorization. A remote-access VPN that dumps users onto a flat internal network means one stolen laptop = broad reach. BeyondCorp/ZTNA inverts this — every request is authorized on identity + device posture + context regardless of network location, so being "inside the tunnel" grants nothing by itself. Pair with microsegmentation so a compromised endpoint can't pivot. Trade-off: requires a mature identity/device-inventory backbone to enforce per-request policy.

2. DNS and egress inspection (defeats covert tunneling/exfil). DNS tunneling and TLS-based C2 succeed because those protocols are always allowed and rarely inspected. Controls: force all DNS through logged, policy-enforcing resolvers; alert on anomalous query length, entropy, subdomain cardinality, and TXT-record volume per domain; apply egress allowlists and TLS fingerprinting (JA3/JA4) to spot non-browser clients. Trade-off: encryption (DoH/DoT and TLS 1.3 ESNI/ECH) increasingly blinds on-path inspection, pushing detection to the endpoint and to behavioral/volumetric signals rather than payload content.

3. Strong tunnel crypto + disciplined key management (defeats weak/legacy tunnels and key compromise). Standardize on IKEv2/ESP, WireGuard, or TLS 1.3; ban PPTP and MS-CHAPv2 outright (broken). Use per-peer keys (never a shared PSK across many tunnels), short rekey intervals, and gate tunnel setup on device posture + MFA. Monitor tunnel establishment, rekey events, and selector scope for silent degradation. Trade-off: modern crypto plus posture checks add operational overhead and can break older clients — usually the right price.

Practice Questions

Tier 1 — Fundamentals

  1. What is tunneling, and why is encapsulation the core idea?
    Answer: Tunneling wraps one protocol's packet inside another so it can cross a network that wouldn't otherwise carry or permit it; the passenger packet is encapsulated at one end and decapsulated at the other. Signal: Do they separate encapsulation (connectivity) from encryption (security)?

  2. Difference between tunnel mode and transport mode in IPsec?
    Answer: Tunnel mode encrypts and wraps the entire original IP packet in a new IP header (gateway-to-gateway, hides internal addresses); transport mode protects only the payload and keeps the original IP header (host-to-host, less overhead). Signal: Precise packet-structure understanding.

  3. Full tunnel vs split tunnel — trade-offs?
    Answer: Full tunnel routes all traffic through the VPN (max visibility/control, more latency/load); split tunnel sends only selected routes through it (better performance, but misconfiguration can leak sensitive traffic outside inspection). Signal: Recognizes the security-vs-performance tension.

  4. Name three tunneling protocols and whether each provides security.
    Answer: GRE — encapsulation only, no security (needs IPsec); IPsec ESP — auth + encryption; SSH port forwarding — strong (SSH-protected). Signal: Knows encapsulation ≠ protection.

  5. Why is DNS a popular tunneling channel for attackers?
    Answer: DNS is almost always allowed outbound and rarely deeply inspected, so data can be encoded into query names and answers to exfiltrate data or carry C2 past firewalls. Signal: Connects protocol ubiquity to abuse.

Tier 2 — System Design & Trade-offs

  1. Design secure remote access for a global workforce — VPN or ZTNA, and why?
    Answer: Prefer ZTNA/BeyondCorp: authorize each request on identity + device posture + context rather than trusting network location, avoiding the flat-network blast radius of classic VPNs; if a VPN is required, use IKEv2/WireGuard, full tunnel or tightly scoped split routes, pushed DNS, and microsegmentation behind it. Signal: Treats tunnel access as transport, not trust.

  2. How would you detect DNS tunneling at scale?
    Answer: Baseline normal DNS and alert on anomalies — long/high-entropy labels, high subdomain cardinality per domain, unusual TXT/NULL record volume, and query rates from single hosts; centralize resolvers with logging and sinkhole known-bad domains. Signal: Behavioral/volumetric thinking over signatures.

  3. A VPN gateway works but users report broken connections to some sites. What's your first hypothesis?
    Answer: MTU/MSS problem — the tunnel's outer header reduces usable MTU, and blocked ICMP breaks PMTUD, black-holing larger packets; fix with MSS clamping / correct tunnel MTU and allow PMTUD. Signal: Knows the classic tunnel operational failure.

  4. Why pair GRE with IPsec instead of using either alone?
    Answer: GRE flexibly encapsulates arbitrary L3 (including multicast/routing protocols) but offers no confidentiality or authentication; IPsec adds crypto but tunnel-mode ESP doesn't natively carry multicast/routing well — GRE-over-IPsec gets both flexibility and protection. Signal: Understands each tool's gap.

  5. TLS 1.3 with ECH is rolling out. What does it do to your tunnel-inspection strategy?
    Answer: Payload and even SNI become opaque on-path, so content inspection degrades; shift detection to endpoints, TLS fingerprinting (JA3/JA4), destination reputation, and behavioral/volumetric signals, and decrypt only at controlled, lawful proxy points. Signal: Adapts controls to encryption trends.

Tier 3 — Expert & Deep-Dive

  1. Walk through how you'd exfiltrate data from a segmented network with only DNS allowed, and how a defender counters each step.
    Answer: Encode data into subdomain labels queried against an attacker-controlled authoritative server, chunking and base32-encoding to fit label limits, with answers carrying commands; defender counters via egress resolver choke points, per-domain query anomaly detection, response-size/entropy alerts, and blocking authoritative lookups to unknown domains. Signal: Both offense and layered defense.

  2. Why is "reaching the tunnel endpoint" a dangerous definition of trust, and how does zero trust reframe it?
    Answer: Network position is easy to reach (stolen device, compromised endpoint, malicious insider) yet classic designs equate it with authorization, enabling lateral movement; zero trust authorizes each request on verifiable identity + device state + context independent of location, so tunnel membership grants no implicit access. Signal: Architectural maturity.

  3. How does NAT break IPsec, and how does NAT-T solve it?
    Answer: NAT rewrites IP/port fields, but ESP has no ports and AH authenticates the IP header, so NAT invalidates integrity checks and can't demux multiple tunnels; NAT-T encapsulates ESP in UDP (port 4500) so NAT devices can translate and track it. Signal: Deep protocol knowledge.

  4. Compare WireGuard and IPsec/IKEv2 for a new deployment.
    Answer: WireGuard is lean (small codebase, modern fixed crypto, UDP, fast handshakes, roaming-friendly) but opinionated and stateless-IP-per-peer; IPsec/IKEv2 is feature-rich, standardized, interoperable, and policy-flexible but complex and larger attack surface — choose WireGuard for simplicity/perf, IKEv2 for interop/enterprise policy. Signal: Nuanced modern trade-offs.

  5. In a multi-tenant cloud using VXLAN overlays, what's the tunneling risk and mitigation?
    Answer: VXLAN encapsulation itself is unauthenticated, so VNI spoofing or fabric access could let traffic hop tenants; mitigate by securing/authenticating the underlay and control plane (EVPN with controls), restricting who can inject VTEP traffic, and not relying on VNI alone for isolation. Signal: Applies tunneling security to cloud fabrics.

Key Takeaways

  • Tunneling = encapsulate → transport → decapsulate; it's cross-layer and the primitive behind both VPNs and covert C2.
  • Encapsulation is not protection — GRE/IP-in-IP/VXLAN give connectivity; IPsec/TLS/WireGuard/SSH add the crypto.
  • Tunnel mode wraps the whole packet (gateway-to-gateway); transport mode protects only payload (host-to-host).
  • Split tunnels leak when routes/DNS aren't scoped; full tunnels trade performance for visibility.
  • Attackers tunnel through what's always allowed — DNS and HTTPS; detect with behavioral/volumetric signals and egress control, not just signatures.
  • Tunnel access is transport, not authorization — put zero-trust authZ and segmentation behind every tunnel.
  • Ban legacy tunnels (PPTP/MS-CHAPv2); standardize on IKEv2/ESP, WireGuard, TLS 1.3 with per-peer keys and posture checks.
