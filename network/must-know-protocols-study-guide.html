Must-Know Network Protocols — Study Guide

Goal: Know what an attacker can do, why the protocol is vulnerable, and what the mitigation is.
Study method: Read your notes → explain attack aloud from memory → answer the mock questions at the bottom.

Priority Tiers

Tier Protocols Why
P1 TLS, TCP, HTTP, DNS, SSH, IPsec Core of every security review; expect deep questions
P2 QUIC, BGP, SMTP/SPF/DKIM/DMARC, LDAP, ARP, DHCP, SMB, RDP Common attack surface; expect scenario questions
P3 SNMP, IPv4/IPv6, VLAN, MACsec, WebSocket Know the security angle; unlikely to go deep

P1 Protocols

TLS/SSL

Notes: Layer6/TLS-SSL/

What to know:

Concept Key point
TLS 1.3 handshake ClientHello → ServerHello + cert → key confirmation → encrypted data. 1-RTT by default
Cipher suites TLS 1.3 removes RSA key exchange; all suites use ephemeral keys (ECDHE) → mandatory forward secrecy
Certificate validation Chain to trusted root, hostname match, validity period, OCSP/CRL revocation check, CT logs
SNI Server name in plaintext ClientHello — leaks hostname even over TLS; ECH (Encrypted ClientHello) mitigates
Session resumption Session tickets (server-encrypted state) vs session IDs; ticket key rotation matters for FS
Downgrade attacks POODLE (forces SSLv3), BEAST (CBC IV), DROWN (SSLv2 cross-protocol) — all mitigated by TLS 1.3
Certificate pinning Hard-codes expected cert/SPKI; prevents rogue CA issuance; brittle for rotation
HSTS Forces HTTPS; includeSubDomains + preload matters; first-visit TOFU gap
mTLS Both sides present certificates; used for service-to-service auth

Attack to explain end-to-end: SSL stripping → attacker intercepts HTTP → rewrites to HTTP before client upgrades → fixed by HSTS preload.

Mock questions:

  • "Why did TLS 1.3 remove RSA key exchange?" (no forward secrecy — compromise of server private key decrypts past sessions)
  • "What does an attacker get from a TLS padding oracle attack?" (CBC IV leak → iterative plaintext recovery block by block)
  • "How does certificate transparency help?" (CT logs let domain owners detect mis-issued certs by monitoring)

TCP

Notes: Layer4/TCP/

What to know:

Concept Key point
3-way handshake SYN → SYN-ACK → ACK; connection state tracked on both sides
SYN flood Attacker sends many SYNs without completing; exhausts half-open connection table
SYN cookies Server encodes state into the SYN-ACK sequence number; no table entry until ACK arrives
RST injection Attacker in path can send spoofed RST to tear down sessions; ISN randomization makes off-path harder
Sequence prediction Weak ISN (Initial Sequence Number) randomness historically allowed blind injection
Head-of-line blocking In-order delivery stalls stream if a segment is lost; QUIC solves this per-stream
TIME_WAIT Prevents old duplicate segments from confusing new connections; can be exhausted under load

Mock questions:

  • "How do SYN cookies mitigate a SYN flood?" (encode SYN-ACK seq as hash of src/dst/timestamp; only allocate state on valid ACK)
  • "Why does TCP sequence prediction matter for session hijacking?" (allows injecting segments without being in the path)

HTTP

Notes: Layer7/HTTP/

What to know:

Concept Key point
Request smuggling Front-end and back-end disagree on where a request ends (CL.TE or TE.CL); lets attacker prefix requests to other users
CORS Browser blocks cross-origin reads; server allows via Access-Control-Allow-Origin; * with credentials is invalid but misconfigured origins are dangerous
Security headers CSP, HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy — know what each blocks
Cache poisoning Poisoning a shared cache with attacker-controlled response affects all users; unkeyed headers are the vector
HTTP/2 rapid reset CVE-2023-44487; attacker sends HEADERS + RST_STREAM in a loop; generates server work with no client cost; DDoS
Cookies HttpOnly blocks JS read; Secure requires TLS; SameSite=Strict/Lax mitigates CSRF; __Host- prefix locks to exact origin
Status codes misused 200 for auth failure leaks existence; 302 with sensitive data in URL leaks via Referer

Mock questions:

  • "Explain HTTP request smuggling and what it lets an attacker do." (ambiguous body delimiter parsing between proxy and backend; prepend requests to other users' sessions, bypass auth)
  • "What is the difference between SameSite=Strict and SameSite=Lax?" (Strict: no cookie on any cross-site navigation; Lax: cookie on top-level GET navigations but not subresource loads)
  • "Why is Access-Control-Allow-Origin: * with Access-Control-Allow-Credentials: true invalid?" (browser blocks it; wildcard + credentials would let any origin read authenticated responses)

DNS

Notes: Layer7/DNS/

What to know:

Concept Key point
Recursive resolution Client → recursive resolver → root → TLD → authoritative; cached for TTL duration
DNS cache poisoning Attacker races spoofed response with matching transaction ID and source port to resolver; injects malicious A record
Kaminsky attack Floods many random subdomains to force fresh lookups; increases poisoning probability dramatically
DNSSEC Signs records with zone key; chain of trust from root to zone; prevents poisoning but not DDoS; complex to operate
DNS-over-HTTPS (DoH) / DNS-over-TLS (DoT) Encrypts DNS traffic; prevents ISP/on-path interception and manipulation; C2 can tunnel over DoH to evade firewalls
Zone transfer AXFR gives full zone contents; should be restricted to secondary nameservers only; leaks all hostnames
DNS exfiltration Encodes data in subdomain labels; bypasses many egress controls; detection via entropy of queried names
TTL abuse Attacker sets very low TTL for fast IP rotation; complicates blocklisting
Subdomain takeover CNAME pointing to decommissioned cloud resource; attacker claims the resource and serves content under victim's subdomain

Mock questions:

  • "How does the Kaminsky DNS poisoning attack work?" (query random subdomains forcing new lookups; race spoofed replies with brute-forced transaction IDs before legitimate reply)
  • "What does DNS exfiltration look like in a PCAP?" (high volume of queries for random-looking subdomains of one domain; unusually long labels; no repeated queries)
  • "Why doesn't DNSSEC protect against DDoS?" (larger response sizes due to signatures can amplify reflection attacks; doesn't address resolver availability)

SSH

Notes: Layer7/SSH/

What to know:

Concept Key point
Host key verification Client checks server's public key against known_hosts; TOFU on first connect; ignoring warnings is MITM risk
Authentication methods Public key (preferred), password, certificate (CA-signed keys for scalable trust), keyboard-interactive
Key distribution at scale Per-user authorized_keys files don't scale; SSH CA (signed certificates with principals) is better
Agent forwarding Forwards SSH agent to remote host; remote host can use the agent to authenticate as you to other hosts; pivot risk
Port forwarding Local (-L), remote (-R), dynamic (-D SOCKS); can bypass firewalls or tunnel malicious traffic
Authorized_keys misconfig command=, no-port-forwarding, no-agent-forwarding restrict what a key can do
Weak ciphers Old servers may offer diffie-hellman-group1-sha1 (Logjam-vulnerable); audit with nmap --script ssh2-enum-algos
Rootkit / trojanized sshd Malware replaces sshd to capture passwords; detect with file integrity monitoring

Mock questions:

  • "Why is SSH agent forwarding risky and when should you disable it?" (remote host's root can use socket to authenticate as you to third hosts; disable unless explicitly needed)
  • "How would you design SSH access at Google scale without per-host authorized_keys?" (SSH CA: central CA signs short-lived user certs with principals; hosts trust CA; revocation via cert expiry)
  • "What does ProxyJump do and why is it safer than agent forwarding?" (connects through a bastion without forwarding the agent; bastion only proxies TCP, not your key material)

IPsec

Notes: Layer3/IPsec/

What to know:

Concept Key point
AH vs ESP AH authenticates but does not encrypt (includes IP header); ESP encrypts and authenticates payload; almost always use ESP
Tunnel vs transport mode Tunnel: wraps entire original IP packet (used for VPN gateways); Transport: protects only payload (used between hosts)
IKEv2 Key exchange and SA negotiation; supports EAP, MOBIKE (mobility), certificate or PSK auth
SA (Security Association) Unidirectional; each direction of communication has its own SA with SPI, algorithm, and key
PSK risks Pre-shared keys are prone to brute-force offline if IKE aggressive mode is used; prefer certificates
IPsec vs TLS IPsec is network layer (protects all IP traffic transparently); TLS is application layer (per-protocol)
IKE aggressive mode Sends hash of PSK in third message before identity is hidden; allows offline cracking

Mock questions:

  • "When would you choose IPsec over TLS for securing communications?" (site-to-site VPN, protecting all traffic including non-HTTP; when you control both network endpoints)
  • "What is the difference between tunnel mode and transport mode?" (tunnel: new IP header wraps full original packet — gateway-to-gateway; transport: only original payload protected — host-to-host)

P2 Protocols

QUIC

Notes: Layer4/QUIC/

Concept Key point
TLS 1.3 built in QUIC integrates TLS 1.3 at the transport layer; no separate handshake; 1-RTT on first connect, 0-RTT on resume
0-RTT replay risk 0-RTT data can be replayed by a network attacker; servers must treat 0-RTT as non-idempotent data only
Connection migration Uses connection IDs instead of 4-tuple; survives IP change (wifi → cell); also makes tracking by IP harder
UDP-based Firewalls that block UDP break QUIC; clients fall back to TCP/TLS; middlebox ossification concern
Stream multiplexing Independent streams; loss on one stream doesn't block others — fixes TCP head-of-line blocking
Google relevance QUIC originated at Google (gQUIC) and became IETF QUIC / HTTP/3; Google properties use it heavily

Mock question: "What is the security risk of QUIC's 0-RTT resumption and how is it mitigated?" (replay attacks; servers must use single-use tokens and not accept 0-RTT for non-idempotent operations)

BGP

Notes: Layer3/BGP/

Concept Key point
Prefix hijack AS advertises more-specific prefix it does not own; traffic diverted (intercept or blackhole)
Route leak Internal routes exported to internet; traffic routed through unexpected AS
RPKI Cryptographic ROAs (Route Origin Authorizations) bind prefix to authorized AS; ISPs drop invalid origins
BGP session security MD5 authentication on TCP session (weak); TTL security (GTSM) prevents remote spoofing
AS path filtering Filter on ingress: only accept routes you expect; max-prefix limits prevent table explosion
Real incidents Pakistan Telecom 2008 (YouTube blackhole), Rostelecom 2020 (financial prefixes hijacked)

Mock question: "What is RPKI and what attack does it prevent?" (signs prefix-to-AS ownership; prevents unauthorized AS from advertising a prefix and hijacking traffic)

SMTP / Email Security (SPF, DKIM, DMARC)

Notes: Layer7/SMTP/

Concept Key point
SPF TXT record lists IPs authorized to send for a domain; receiving server checks envelope MAIL FROM
DKIM Sender signs message headers with private key; receiver validates with public key in DNS
DMARC Policy that ties SPF and DKIM to the From: header domain; p=reject drops failing mail; enables reporting
SPF alone is not enough SPF checks envelope sender, not From: header; display-name spoofing still works without DMARC
SMTP relay abuse Open relays allow anyone to send through; used for spam and phishing at scale
STARTTLS Upgrades SMTP to TLS; can be stripped by on-path attacker unless DANE or MTA-STS enforces it

Mock question: "If a domain has SPF and DKIM but no DMARC, what can an attacker still do?" (spoof the From: header to display the legitimate domain while using a different envelope sender that passes SPF/DKIM)

LDAP / LDAPS

Notes: Layer7/LDAP/, Layer7/LDAPS/

Concept Key point
LDAP injection Unsanitized input in filter strings; analogous to SQLi; escape special chars ()=*\NUL
Anonymous bind Many servers allow unauthenticated reads; can enumerate users, groups, password policies
LDAPS vs StartTLS LDAPS (port 636) is TLS from start; StartTLS on port 389 can be downgraded by stripping the upgrade
Null base DN Some servers return root DSE without auth; reveals schema and naming contexts
AD-specific LDAP against Active Directory; userAccountControl, group membership, and Kerberos preauth flags are readable

Mock question: "How would you prevent LDAP injection?" (allowlist valid chars or use library with parameterized LDAP filters; never concatenate user input into filter strings)

ARP

Notes: Layer2/ARP/

Concept Key point
ARP spoofing Attacker broadcasts gratuitous ARP replies mapping victim IP to attacker MAC; hosts update cache without verification
MITM via ARP Attacker poisons both the victim and the gateway; traffic flows through attacker's host
ARP cache poisoning persistence Caches have short TTLs; attacker must keep sending; detectable by monitoring MAC changes for an IP
Mitigations Dynamic ARP Inspection (DAI) on managed switches; static ARP entries for critical hosts; 802.1X
Gratuitous ARP Sent on IP change or boot; no request needed; hosts accept them unconditionally

Mock question: "How does ARP spoofing enable a MITM attack on a switched network?" (poison both victim and gateway ARP caches to map each other's IPs to attacker's MAC; all traffic passes through attacker)

DHCP

Notes: Layer7/DHCP/

Concept Key point
Rogue DHCP server Attacker responds faster than legitimate server; assigns attacker-controlled DNS/gateway
DHCP starvation Flood DISCOVER with random MACs; exhaust pool; then offer rogue server
DHCP snooping Switch feature; only allows DHCP offers on designated trusted ports
No authentication DHCP has no built-in client or server authentication; relies entirely on network-level trust

SMB / CIFS

Notes: Layer7/SMB-CIFS/

Concept Key point
EternalBlue (MS17-010) SMBv1 buffer overflow; used by WannaCry; all SMBv1 should be disabled
NTLM relay SMB authentication challenge can be relayed to another service; attacker authenticates as victim
SMB signing Cryptographically signs packets; prevents relay and tampering; should be required everywhere
Pass-the-Hash NTLM hash captured from memory can authenticate without cracking; mitigated by Kerberos + Protected Users group
Anonymous access Null sessions allow share enumeration; disable null session access in policy

Mock question: "How does NTLM relay work and what mitigation breaks it?" (capture NTLM challenge-response, forward to target before expiry; mitigated by SMB signing — signed sessions can't be relayed because attacker can't sign)

RDP

Notes: Layer7/RDP/

Concept Key point
NLA (Network Level Authentication) Requires auth before full RDP session; prevents unauthenticated memory exposure
BlueKeep (CVE-2019-0708) Pre-auth RCE in RDP on unpatched Windows; wormable; patch or disable RDP
Credential exposure RDP transmits creds; captured in memory; target of credential theft tools
RDP over internet Exposed RDP (port 3389) is heavily brute-forced; restrict to VPN or jump host only
Clipboard and drive redirection Can exfiltrate files or paste malicious content; disable in high-risk sessions

P3 Protocols — Security Angles Only

Protocol Key security facts
SNMP v1/v2c Community strings are cleartext passwords; public/private defaults still common; v3 has auth and encryption; SNMP read = full device config disclosure
IPv4/IPv6 IP spoofing enables reflection/amplification DDoS; ingress filtering (BCP38) blocks spoofed sources; IPv6 link-local addresses bypass perimeter; DHCPv6/SLAAC rogue router attacks
VLAN VLAN hopping: double-tagging attack tricks switch into forwarding to another VLAN; mitigate by not using native VLAN = 1, disabling auto-trunking
MACsec IEEE 802.1AE; encrypts/authenticates Layer 2 frames; protects against ARP spoofing and MITM at the switch level; requires managed hardware
WebSocket Persistent bidirectional channel after HTTP upgrade; same-origin policy does NOT apply; CSRF-like origin bypass possible; secure with Origin header validation and per-socket auth tokens
NTP Monlist command amplifies 200x; widely abused for DDoS reflection; mitigate with access controls, restrict to trusted clients
Telnet Plaintext; completely replaced by SSH; any remaining Telnet in production is a finding
FTP Active mode FTP can be abused via PORT command to scan or attack third parties; FTPS or SFTP instead

OSI Layer Quick-Reference

Layer 7 — Application  : HTTP, DNS, SSH, SMTP, LDAP, RDP, SMB, SNMP, FTP
Layer 6 — Presentation : TLS/SSL, MIME, JSON/XML encoding
Layer 5 — Session      : (less security-relevant; NetBIOS, RPC)
Layer 4 — Transport    : TCP, UDP, QUIC
Layer 3 — Network      : IP, IPsec, BGP, OSPF, ICMP
Layer 2 — Data Link    : Ethernet, ARP, VLAN, MACsec, STP
Layer 1 — Physical     : (not a security focus)

study notes: When asked about an attack, anchor it to the layer where it occurs and explain why that layer lacks authentication or integrity controls.

Attack Pattern Cheat Sheet

Attack Protocol(s) Root cause Quick mitigation
SYN flood TCP Stateful connection setup, no auth SYN cookies, edge rate limiting
SSL stripping TLS/HTTP HTTP first-visit before upgrade HSTS preload
Certificate MITM TLS Rogue CA or misconfigured validation Certificate pinning, CT monitoring
DNS cache poisoning DNS Unauthenticated UDP responses DNSSEC, 0x20 encoding, DoT/DoH
DNS exfiltration DNS DNS queries bypass egress controls DNS firewall, anomaly detection on query entropy
BGP hijack BGP No built-in prefix ownership proof RPKI ROA validation
ARP spoofing → MITM ARP Stateless, no auth Dynamic ARP Inspection, MACsec
NTLM relay SMB/LDAP Challenge forwarded before expiry SMB signing required, Extended Protection for Auth
Request smuggling HTTP CL/TE ambiguity between proxy and origin Normalize at edge; reject ambiguous requests
Rogue DHCP DHCP No server authentication DHCP snooping on managed switches
SSH agent forwarding pivot SSH Agent socket accessible on remote host Disable forwarding; use ProxyJump instead
VLAN hopping VLAN/Ethernet Double-tagging native VLAN Disable native VLAN 1, disable auto-trunking
0-RTT replay QUIC/TLS 1.3 Resumption tickets replayable Single-use tokens; restrict 0-RTT to safe operations

Mock practice questions — Network Security

  1. Walk me through the TLS 1.3 handshake. Where does authentication happen, and what breaks if you skip hostname validation?
  2. An attacker sends 10 million SYN packets per second at your server. What happens, and how do SYN cookies help?
  3. Explain DNS cache poisoning. How does DNSSEC prevent it, and why is DNSSEC not universally deployed?
  4. Your threat model flags outbound DNS to random subdomains. What protocol are you worried about and how do you detect and block it?
  5. What is BGP prefix hijacking? Name a real-world example and explain how RPKI would have prevented it.
  6. How does NTLM relay work, and what two controls independently break it?
  7. You find RDP exposed on port 3389 on a production server. What are the top three risks and what do you do?
  8. Explain HTTP request smuggling. What conditions are required, and what can an attacker achieve?
  9. A developer says "we use TLS so the connection is secure." What are three things that can still go wrong?
  10. Compare DoT, DoH, and DNSSEC — what each protects against and what it does not protect against.
