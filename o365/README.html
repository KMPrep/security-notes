Microsoft 365 (O365) Security

Category: Cloud Productivity & SaaS Security
Key standards / frameworks: Microsoft Cloud Security Benchmark (MCSB), CIS Microsoft 365 Foundations Benchmark, NIST SP 800-53, MITRE ATT&CK (Enterprise + Cloud matrices), Zero Trust (NIST SP 800-207)

Overview

Microsoft 365 is a Software-as-a-Service suite in which Microsoft operates the physical infrastructure, the hypervisor, the operating systems, and the application code for Exchange Online, SharePoint Online, OneDrive, Teams, and the Office apps. Under the SaaS shared-responsibility model the customer never patches a server, but the customer remains fully responsible for identity, data, device, and configuration — the four areas where nearly every real-world M365 breach occurs. The defender's job in M365 is therefore configuration and identity engineering, not infrastructure hardening.

The security model orbits a single control plane: Entra ID (formerly Azure AD), the identity provider that authenticates every user, service principal, and workload accessing any Microsoft cloud resource. Because a valid Entra token is the universal key to mail, files, chat, and admin APIs, identity compromise — via password spray, adversary-in-the-middle phishing, token theft, or consent phishing — is the dominant attack path. Controls such as Conditional Access, MFA, and Identity Protection are the load-bearing walls of the whole tenant.

The tenant is the trust boundary. It is a dedicated Entra directory instance that owns all users, groups, licenses, and policy. Cross-tenant access (B2B guests, cross-tenant sync, multi-tenant OAuth apps) deliberately punches holes through that boundary and is a frequent source of lateral movement and data exfiltration. Licensing shapes the defensive posture materially: many high-value controls — risk-based Conditional Access, Defender for Office 365 Plan 2, Purview auto-labeling, Insider Risk Management — require E5 or standalone add-ons, while E3 tenants must lean on baseline anti-malware, Security Defaults or basic Conditional Access, and manual investigation.

How It Works

The tenant as a trust boundary

Every request to an M365 workload is a token-authorized API call. A user (or an app on the user's behalf) authenticates to Entra ID, receives an access token scoped to a resource and a set of permissions, and presents that token to the workload. Conditional Access sits inline at token issuance, evaluating signals and deciding allow, block, or step-up. This means the security perimeter is not a network — it is the token issuance decision. The diagram below shows the principal control and data planes inside the tenant boundary.

  User[User or Workload] --> Entra[Entra ID Identity Plane]
  Entra --> CA[Conditional Access Decision]
  CA --> Token[Scoped Access Token]
  Token --> Exchange[Exchange Online Mail]
  Token --> SharePoint[SharePoint and OneDrive]
  Token --> Teams[Teams Collaboration]
  Token --> Graph[Microsoft Graph and OAuth Apps]
  Defender[Defender XDR and Purview] --> Entra

The four attack surfaces

The M365 attack surface decomposes into four planes that map cleanly to defensive product areas. Identity covers authentication, MFA, and Conditional Access. Email covers phishing, business email compromise, malicious rules, and mail-flow abuse, defended by Exchange Online Protection and Defender for Office 365. Collaboration covers SharePoint, OneDrive, and Teams oversharing, anonymous links, and guest access. Apps and OAuth covers consent phishing and over-privileged service principals accessing Graph. A mature program instruments all four and correlates them in Defender XDR.

Licensing impact on posture

E3 delivers Entra ID P1 (Conditional Access without risk signals), Exchange Online Protection, and baseline Purview. E5 adds Entra ID P2 (Identity Protection risk-based policies, Privileged Identity Management), Defender for Office 365 Plan 2 (automated investigation, Threat Explorer, attack simulation), Defender for Cloud Apps, and advanced Purview (auto-labeling, Insider Risk, Communication Compliance). Knowing which controls a tenant can actually deploy is the first step in any M365 assessment.

Key Takeaways

  • M365 is SaaS: Microsoft owns the infrastructure, but the customer owns identity, data, device, and configuration — where breaches actually happen.
  • Entra ID is the universal control plane; a valid token is the key to everything, so identity compromise is the dominant attack path.
  • The tenant is the trust boundary; guest access, cross-tenant sync, and multi-tenant OAuth apps intentionally weaken it and need explicit governance.
  • Conditional Access is enforced at token issuance — the real security perimeter is the authentication decision, not the network.
  • The attack surface splits into identity, email, collaboration, and apps/OAuth; instrument and correlate all four in Defender XDR.
  • Licensing (E3 vs E5) determines which controls exist; risk-based CA, Defender P2, and advanced Purview are E5 or add-on features.
  • Defense-in-depth still matters even in SaaS: authentication catches credential attacks, Defender catches payloads authentication misses.
