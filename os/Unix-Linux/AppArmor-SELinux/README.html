AppArmor and SELinux — Mandatory Access Control on Linux

Category: Unix/Linux Security, MAC Systems
Parent: os/Unix-Linux/README.md

Overview

AppArmor and SELinux are the two dominant Linux Mandatory Access Control (MAC) systems. Both enforce security policy on top of traditional Unix DAC (discretionary access control), limiting what processes can do even when they run as root or have broad POSIX privileges.

MAC matters because DAC is inherently incomplete: a compromised root process can read any file, open any network connection, and modify any configuration. MAC policies define what specific processes are allowed to do regardless of their UID — reducing the blast radius when a service is exploited.

SELinux vs AppArmor: Core Differences

Dimension SELinux AppArmor
Policy basis Labels on every file, process, socket Path-based profiles per application
Model Type Enforcement (TE) + RBAC + MLS Per-application access control profiles
Complexity High — requires labeling everything Lower — path-based; easier to write
Distribution RHEL/CentOS/Fedora, Android Ubuntu/Debian, SUSE
Audit trail audit.log via auditd /var/log/syslog or kern.log
Granularity Very fine-grained; domain transitions, object classes Moderate; file paths, network access, capabilities
Default stance Deny unless explicitly allowed Deny unless allowed; complain mode available
Profile generation audit2allow, sepolicy generate aa-genprof, aa-logprof

SELinux Deep Dive

Type Enforcement

Every process has a domain (type for a process). Every file/socket/etc. has a type. The policy defines which domain can access which type via which operation class.

Subject (process domain) + Object (file/socket type) + Access vector (read/write/exec) → Allow or Deny

Example policy rule:

allow httpd_t httpd_config_t:file { read open getattr };

This says: the httpd_t domain (Apache) can read files of type httpd_config_t (Apache config files).

SELinux Contexts

Every labeled object has a context in the form user:role:type:level:

# View process context
ps -eZ | grep httpd
# system_u:system_r:httpd_t:s0 1234 ? 00:00:01 httpd

# View file context
ls -Z /etc/passwd
# system_u:object_r:passwd_file_t:s0 /etc/passwd

# View your current context
id -Z
# unconfined_u:unconfined_r:unconfined_t:s0-s0:c0.c1023

SELinux Modes

Mode Behavior
Enforcing Policy violations are blocked and logged
Permissive Policy violations are logged but not blocked — for policy development
Disabled SELinux not active (requires reboot to re-enable; relabeling needed) 
# Check current mode
getenforce

# Switch to permissive without reboot (temporary)
setenforce 0

# Permanently set in /etc/selinux/config
SELINUX=enforcing

# Never set SELINUX=disabled in production — requires full relabel to re-enable

Common SELinux Operations

# Check recent denials
ausearch -m AVC -ts recent
# or
journalctl -t audit | grep AVC

# Generate a policy module from denials (use with caution — understand what you're allowing)
audit2allow -M mymodule < /var/log/audit/audit.log
semodule -i mymodule.pp

# Relabel entire filesystem (needed after disabling/re-enabling)
touch /.autorelabel && reboot

# Check if a file has the right context
matchpathcon -V /etc/nginx/nginx.conf

# Fix wrong file context
restorecon -v /etc/nginx/nginx.conf

# Booleans: toggle broad policy switches
getsebool -a | grep http
setsebool -P httpd_can_network_connect on

SELinux Boolean: Common Security-Relevant Settings

Boolean Default What enabling it allows
httpd_can_network_connect off Apache to make outbound TCP connections
httpd_can_connect_ftp off Apache to connect to FTP servers
httpd_execmem off Apache to execute code in writable memory
selinuxuser_execstack off User processes to use executable stack
allow_execmem off All processes to execute from heap/stack
ftp_home_dir off FTP to access user home dirs
ssh_chroot_rw_homedirs off SSH to write to chroot home dirs

Security note: Setting httpd_execmem=on or allow_execmem=on defeats DEP/NX for that domain. Never enable unless absolutely necessary and understand the risk.

SELinux MLS (Multi-Level Security)

MLS extends type enforcement with sensitivity levels (like SECRET, CONFIDENTIAL) and compartments. It is used in high-assurance and government environments. Most production Linux deployments use the targeted policy (MLS disabled; only specific daemons are confined).

AppArmor Deep Dive

Profile Structure

An AppArmor profile defines what an executable can access:

/usr/sbin/nginx {
  # Include common networking rules
  #include <abstractions/base>
  #include <abstractions/nameservice>

  # Capability grants
  capability net_bind_service,
  capability setuid,
  capability setgid,

  # File access rules
  /etc/nginx/** r,                       # Read nginx config
  /var/log/nginx/** w,                   # Write to log dir
  /var/www/html/** r,                    # Serve web content
  /run/nginx.pid rw,                     # PID file
  /usr/sbin/nginx mr,                    # Map-read own binary

  # Network access
  network tcp,                           # TCP connections

  # Deny everything else implicitly (default deny)
}

AppArmor Modes

Mode Behavior
Enforce Profile violations are blocked and logged
Complain Violations are logged but not blocked — for profile development
Disabled Profile not active for that application 
# Check status of all profiles
aa-status

# Load a profile in enforce mode
apparmor_parser -r /etc/apparmor.d/usr.sbin.nginx

# Put a profile in complain mode (for testing)
aa-complain /etc/apparmor.d/usr.sbin.nginx

# Put a profile back to enforce mode
aa-enforce /etc/apparmor.d/usr.sbin.nginx

# Generate a new profile interactively (run app, capture denials, build profile)
aa-genprof /usr/sbin/nginx

# Update profile based on recent denials in complain mode
aa-logprof

AppArmor in Docker and Kubernetes

Docker automatically applies the docker-default AppArmor profile to all containers. The profile blocks:

  • mount — filesystem mounting
  • umount — unmounting
  • pivot_root — change root
  • /proc/sys/kernel/sysrq write — SysRq key
  • ptrace — cross-process inspection
# Check if a container has AppArmor profile applied
docker inspect --format='{{.HostConfig.SecurityOpt}}' <container>
# [apparmor=docker-default]

# Run container with custom profile
docker run --security-opt apparmor=my-custom-profile myimage

# Run container with no AppArmor (dangerous)
docker run --security-opt apparmor=unconfined myimage

In Kubernetes, AppArmor is configured via annotations or the appArmorProfile field:

# Kubernetes 1.30+ native field
spec:
  securityContext:
    appArmorProfile:
      type: RuntimeDefault

# Or via annotation (older K8s)
metadata:
  annotations:
    container.apparmor.security.beta.kubernetes.io/my-container: runtime/default

Comparing MAC to Other Controls

flowchart TD
    REQUEST[Process tries to access a resource] --> DAC{DAC check\nUID/GID/permissions}
    DAC -->|Denied| DENY1[Access denied - no MAC check needed]
    DAC -->|Allowed| MAC{MAC check\nSELinux or AppArmor}
    MAC -->|Denied| DENY2[Access denied - logged as AVC denial]
    MAC -->|Allowed| ACCESS[Access granted]

MAC is evaluated after DAC. Both must pass for access to be granted.

What Can Go Wrong

Failure Root Cause Impact
SELinux set to permissive permanently Operations team disabled enforce to fix denials Full DAC-only protection; MAC provides no containment
Wide permissive process domain Everything runs in unconfined_t MAC applies to no real processes
audit2allow blindly applied Policy generated without reviewing what it allows Policy opens holes silently
AppArmor profile in complain mode Never promoted to enforce after testing Violations logged but not blocked
No profile for new services New binary has no AppArmor profile No MAC enforcement
allow_execmem boolean enabled Quick fix for JIT-compiled app All processes in that domain can execute from heap

Mitigation Controls

  • Never disable SELinux; use permissive mode for development only; enforce in production.
  • Audit booleans quarterly — enabled booleans expand the attack surface.
  • Use restorecon and matchpathcon in deployment automation to ensure correct file contexts.
  • Apply AppArmor profiles in complain mode first; promote to enforce after testing with real workloads.
  • Review audit2allow-generated policy before applying — each rule is a potential bypass.
  • In Kubernetes, set AppArmorProfile: RuntimeDefault in Pod Security Standards enforcement.

Key Takeaways

  • SELinux and AppArmor both implement MAC — complementing DAC, not replacing it.
  • SELinux is label-based and more granular; AppArmor is path-based and easier to operate.
  • Disabling SELinux (not even permissive — disabled) requires reboot and relabeling; it destroys the protection layer.
  • Docker/Kubernetes apply default MAC profiles; verify these are not overridden with unconfined.
  • A misconfigured MAC policy that permits too much is better than no MAC — start with defaults and tighten.
