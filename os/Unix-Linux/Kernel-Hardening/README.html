Linux Kernel Hardening

Category: Unix/Linux Security, System Hardening
Parent: os/Unix-Linux/README.md

Overview

Kernel hardening is the set of compile-time, boot-time, and runtime controls that reduce the Linux kernel's attack surface and limit the impact of kernel vulnerabilities. At production scale it is managed through configuration management (Ansible, Puppet, Chef) and enforced via CIS benchmarks, STIGs, or custom baselines — not applied manually per host.

The threat model: an attacker has either a local shell or code execution in a container/sandbox and wants to escalate to kernel/host privilege. Kernel hardening adds layers between that foothold and a full system compromise.

sysctl Hardening — Key Parameters

/etc/sysctl.d/*.conf files are the primary place to set kernel parameters persistently.

Network Hardening

# Prevent IP spoofing — validate source addresses on incoming packets
net.ipv4.conf.all.rp_filter = 1
net.ipv4.conf.default.rp_filter = 1

# Ignore ICMP broadcast requests (Smurf attack mitigation)
net.ipv4.icmp_echo_ignore_broadcasts = 1

# Ignore bogus ICMP error responses
net.ipv4.icmp_ignore_bogus_error_responses = 1

# Disable IP forwarding unless this host is a router
net.ipv4.ip_forward = 0
net.ipv6.conf.all.forwarding = 0

# Disable source routing
net.ipv4.conf.all.accept_source_route = 0
net.ipv6.conf.all.accept_source_route = 0

# Disable ICMP redirects (prevent routing manipulation)
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.all.send_redirects = 0
net.ipv6.conf.all.accept_redirects = 0

# Enable TCP SYN cookies (SYN flood mitigation)
net.ipv4.tcp_syncookies = 1

# Protect against TIME_WAIT assassination
net.ipv4.tcp_rfc1337 = 1

Memory and Exploitation Hardening

# Restrict /proc/sysrq-trigger (prevent attacker from triggering SysRq)
kernel.sysrq = 0

# Restrict access to kernel pointers in /proc (prevents ASLR bypass via /proc/kallsyms)
kernel.kptr_restrict = 2

# Restrict dmesg to root (kernel messages may contain addresses or sensitive info)
kernel.dmesg_restrict = 1

# Enable kernel address space layout randomisation
kernel.randomize_va_space = 2   # 0=off, 1=shared libs+stack, 2=also heap

# Restrict ptrace to parent process only (prevents process injection by non-parent)
kernel.yama.ptrace_scope = 1
# Values: 0=all processes, 1=parent only, 2=only with CAP_SYS_PTRACE, 3=disabled

# Disable unprivileged user namespaces (reduces kernel attack surface)
kernel.unprivileged_userns_clone = 0   # Ubuntu/Debian
user.max_user_namespaces = 0           # RHEL/CentOS alternative

# Restrict loading of kernel modules to root only (default on most distros)
kernel.modules_disabled = 1   # Set to 1 to permanently prevent new module loading

# Restrict BPF to privileged users
kernel.unprivileged_bpf_disabled = 1

# Enable JIT hardening for BPF (randomizes BPF JIT-compiled code addresses)
net.core.bpf_jit_harden = 2

# Panic on kernel oops (prevents exploitation of partial system states)
kernel.panic_on_oops = 1

File System Hardening

# Prevent hard links to files the user doesn't own (symlink attack prevention)
fs.protected_hardlinks = 1
fs.protected_symlinks = 1

# Restrict FIFO opening in sticky/world-writable dirs
fs.protected_fifos = 2

# Restrict regular file opening in sticky/world-writable dirs
fs.protected_regular = 2

# Restrict core dump creation (prevent memory disclosure via crash)
fs.suid_dumpable = 0

Boot Parameters (GRUB / Kernel Cmdline)

Add these to /etc/default/grub in GRUB_CMDLINE_LINUX:

# Enable kernel page table isolation (KPTI — Meltdown mitigation)
pti=on

# Enable retpoline (Spectre v2 mitigation)
spectre_v2=on

# Disable SMT if you can accept the performance hit (Spectre/L1TF mitigation)
nosmt

# Randomize slab freelists (heap feng shui hardening)
slab_nomerge

# Enable zeroing of memory on free (prevents use-after-free info leak)
init_on_free=1

# Enable zeroing of memory on allocation
init_on_alloc=1

# Enable kernel page sanitization
page_poison=1

# Randomize kernel stack offset per syscall
randomize_kstack_offset=on

# Disable vsyscall page (historical; prevents return-to-vsyscall attacks)
vsyscall=none

# Lockdown mode: integrity (or confidentiality) — restricts kernel modification
lockdown=integrity

# Prevent loading of unsigned modules (requires Secure Boot on UEFI)
module.sig_enforce=1

Secure Boot and Module Signing

Secure Boot verifies that the bootloader and kernel are signed with trusted keys. Module signing extends this to kernel modules.

flowchart LR
    UEFI[UEFI firmware] -->|Verify with UEFI db keys| BL[Bootloader - GRUB]
    BL -->|Verify with shim keys| KERNEL[Linux kernel]
    KERNEL -->|Verify with kernel module signing key| MOD[Kernel modules - .ko]

# Check if Secure Boot is active
mokutil --sb-state

# Check if module signing is enforced
cat /proc/sys/kernel/modules_disabled
grep MODULE_SIG /boot/config-$(uname -r)

# List loaded modules
lsmod

# Check if a specific module is signed
modinfo <module_name> | grep sig

Attack surface without Secure Boot: If Secure Boot is disabled, an attacker with physical access or boot parameter manipulation can boot a different kernel or load unsigned malicious modules.

Kernel Lockdown Mode

Linux 5.4+ ships with kernel lockdown LSM. It prevents even root from modifying the running kernel:

Lockdown Level What it blocks
integrity Loading unsigned modules, direct /dev/mem and /dev/kmem access, hibernation, PCI BAR access, ACPI table overwrite, kexec of unsigned images
confidentiality Everything in integrity + extracting kernel memory via perf, restricting hibernation images from reading kernel memory 
# Check lockdown mode
cat /sys/kernel/security/lockdown

# Set via boot parameter: lockdown=integrity
# Cannot be changed at runtime once set to confidentiality

auditd — Kernel Audit Framework

The Linux audit system logs security-relevant kernel events. It is the primary source of forensic and detection data on Linux systems.

Key Audit Rules

# /etc/audit/rules.d/hardening.rules

# Audit writes to sensitive files
-w /etc/passwd -p wa -k identity
-w /etc/shadow -p wa -k identity
-w /etc/sudoers -p wa -k sudo_changes
-w /etc/sudoers.d -p wa -k sudo_changes

# Audit privilege escalation syscalls
-a always,exit -F arch=b64 -S setuid -S setgid -S setreuid -S setregid -k priv_esc
-a always,exit -F arch=b64 -S ptrace -k ptrace

# Audit module loading/unloading
-w /sbin/insmod -p x -k modules
-w /sbin/rmmod -p x -k modules
-w /sbin/modprobe -p x -k modules
-a always,exit -F arch=b64 -S init_module -S finit_module -S delete_module -k modules

# Audit cron changes
-w /etc/cron.d -p wa -k cron
-w /etc/crontab -p wa -k cron
-w /var/spool/cron -p wa -k cron

# Audit SSH configuration changes
-w /etc/ssh/sshd_config -p wa -k sshd_config

# Audit network configuration changes
-a always,exit -F arch=b64 -S sethostname -S setdomainname -k network_changes
-w /etc/hosts -p wa -k hosts
-w /etc/network -p wa -k network

# Audit capability usage
-a always,exit -F arch=b64 -S capset -k capabilities

# Log all executions (high volume — use selectively)
-a always,exit -F arch=b64 -S execve -k exec

# Make audit rules immutable (cannot be changed until reboot)
-e 2

# Query audit logs
ausearch -k identity -ts today
ausearch -k sudo_changes -ts recent
ausearch -m AVC -ts recent        # SELinux denials

# Generate reports
aureport --auth --summary
aureport --failed

Filesystem Hardening: Mount Options

# /etc/fstab hardening

# /tmp: noexec prevents executing binaries; nosuid prevents SUID bits; nodev prevents device files
/dev/sda3  /tmp  ext4  defaults,nodev,nosuid,noexec  0 2

# /var: nodev,nosuid
/dev/sda4  /var  ext4  defaults,nodev,nosuid  0 2

# /home: nodev,nosuid (noexec may break compiled scripts in user home dirs)
/dev/sda5  /home  ext4  defaults,nodev,nosuid  0 2

# Shared memory: tmpfs with noexec,nosuid,nodev
tmpfs  /dev/shm  tmpfs  defaults,nodev,nosuid,noexec  0 0

CIS Benchmark Coverage

The CIS Benchmark for Linux (RHEL, Ubuntu, Debian versions available) is the industry standard hardening guide. Key sections:

CIS Control Area Key settings
Filesystem Configuration nodev/nosuid/noexec mounts; AIDE setup
Software Updates Automatic security updates enabled
Secure Boot Settings GRUB password; Secure Boot
Process Hardening sysctl ASLR, core dump restriction, ptrace_scope
Network Configuration All sysctl network params above
Logging and Auditing auditd rules; log retention
Access, Authentication, Authorization sudo configuration; password policies; SSH hardening
Warning Banners /etc/motd, /etc/issue.net
PAM pam_pwquality, pam_faillock, pam_unix

What Can Go Wrong

Failure Root Cause Impact
ASLR disabled (randomize_va_space=0) Performance testing, never re-enabled Exploits reliably chain with fixed addresses
ptrace_scope=0 Debug tools require it; never re-secured Any process can ptrace any other; process injection trivial
dmesg_restrict=0 Monitoring tools use dmesg Kernel addresses exposed; ASLR bypass
Audit rules not loaded after reboot Rules in wrong location or not loaded by auditd No forensic trail for privesc events
Modules not signed + Secure Boot off Legacy compatibility Rootkit installation undetected
/tmp without noexec Default mount options Attacker can stage and execute payloads from /tmp

Key Takeaways

  • kernel.randomize_va_space=2, kernel.yama.ptrace_scope=1, kernel.kptr_restrict=2, and kernel.dmesg_restrict=1 are the four highest-impact sysctl settings.
  • Kernel lockdown mode (Linux 5.4+) prevents even root from modifying the running kernel.
  • auditd rules must be loaded and rules should end with -e 2 (immutable) in production.
  • CIS Benchmarks provide a scored, auditable baseline — use OpenSCAP to validate continuously.
  • Mount options (noexec, nosuid, nodev) on /tmp, /var, /home are cheap and effective friction.
