Linux Namespaces

Category: Unix/Linux Security, Container Isolation
Parent: os/Unix-Linux/README.md

Overview

Linux namespaces are the kernel mechanism that provides isolation between groups of processes. They are the foundational technology behind containers (Docker, containerd, Podman) and some sandboxing implementations (Chromium, Flatpak). Each namespace wraps a specific system resource and gives processes inside the namespace their own isolated view of that resource.

For security engineering, namespaces matter in two directions:

  1. As an isolation primitive: containers depend on namespaces to isolate filesystem, network, PIDs, and users from each other and from the host.
  2. As an attack surface: namespaces can be escaped or misconfigured in ways that allow a process inside a namespace to affect resources outside it.

The 8 Namespace Types

Namespace Flag What It Isolates Security relevance
Mount (mnt) CLONE_NEWNS Filesystem mount points; each namespace has its own mount table Container filesystem isolation; host mounts leaked into container = escape path
PID CLONE_NEWPID Process ID space; PIDs restart from 1 inside Without PID ns, container can see and signal host processes
Network (net) CLONE_NEWNET Network interfaces, routing, sockets, iptables Each container gets its own virtual NIC; without net ns, containers share host network
UTS CLONE_NEWUTS Hostname and domain name Low security relevance; cosmetic isolation
IPC CLONE_NEWIPC System V IPC, POSIX message queues Without IPC ns, container can read/write host IPC objects
User (user) CLONE_NEWUSER UID/GID mappings — UID 0 inside maps to unprivileged UID on host Allows rootless containers; can be abused to gain capabilities
Cgroup CLONE_NEWCGROUP cgroup v2 filesystem view Without cgroup ns, container can see host cgroup hierarchy
Time CLONE_NEWTIME System clock offsets (Linux 5.6+) Low security relevance; protects against clock manipulation

How Namespaces Work

flowchart TD
    HOST[Host kernel] --> NS1[Container A]
    HOST --> NS2[Container B]
    NS1 --> PID1A[PID ns: 1=nginx 2=worker]
    NS1 --> NET1A[Net ns: eth0=172.17.0.2]
    NS1 --> MNT1A[Mnt ns: / = overlay on host]
    NS2 --> PID1B[PID ns: 1=app 2=helper]
    NS2 --> NET1B[Net ns: eth0=172.17.0.3]
    NS2 --> MNT1B[Mnt ns: / = overlay on host]

On the host, the container processes appear with their real PIDs (e.g., 4271, 4272). Inside the PID namespace, they see 1, 2.

Namespace Inspection Commands

# List all namespaces for a process
ls -la /proc/<pid>/ns/

# View namespace IDs for all processes
lsns

# Enter a specific namespace of a running container
nsenter -t <pid> --mount --pid --net -- /bin/bash

# Create a new network namespace and run a command in it
ip netns add testns
ip netns exec testns ip link list
ip netns del testns

# Show your own namespaces
ls -la /proc/self/ns/

User Namespaces: Power and Danger

The user namespace is the most security-relevant. It maps UIDs inside the namespace to different UIDs on the host:

Inside namespace: UID 0 (root)
     ↕ mapping: uid_map says 0 → 1000
Outside (host): UID 1000 (unprivileged user)

This enables rootless containers — a container process runs as UID 0 inside but maps to an unprivileged UID on the host. If the container is escaped, the attacker lands as a normal user, not root.

User Namespace Attack Surface

User namespaces also grant the creating process a full set of capabilities within the new namespace. This has historically been a kernel attack surface:

CVE Mechanism Impact
CVE-2022-0847 (Dirty Pipe) pipe page flag manipulation Any user could overwrite read-only files including SUID binaries
CVE-2021-3493 Ubuntu-specific user ns + overlayfs Unprivileged user gains kernel code execution
CVE-2019-5736 (runc) /proc/self/exe race via procfs Container process overwrites host runc binary
CVE-2016-5195 (Dirty COW) MAP_PRIVATE + ptrace race in copy-on-write Write to read-only mapped files; overwrite SUID binaries

Many Linux distributions restrict unprivileged user namespace creation:

# Restrict user namespaces (Ubuntu 23.10+, some RHEL configs)
sysctl -w kernel.unprivileged_userns_clone=0

# Check current setting
cat /proc/sys/kernel/unprivileged_userns_clone

Mount Namespace: Container Filesystem Isolation

The mount namespace gives each container its own mount table. The container's / is a separate overlay filesystem stacked on top of container image layers.

Dangerous Mount Configurations

Misconfiguration Risk
-v /:/host — mount host root into container Container has full read/write access to host filesystem
-v /proc:/host-proc — mount host /proc Container can read all host process memory, environ, cmdline
-v /etc:/etc — mount host /etc Container can modify /etc/passwd, /etc/shadow, sudoers
-v /var/run/docker.sock:/var/run/docker.sock Container can create new privileged containers → full host compromise

The Docker Socket Escape

Mounting the Docker socket into a container is a trivially exploitable misconfiguration:

# Inside container with Docker socket mounted
docker run -it --privileged -v /:/host ubuntu chroot /host
# Now you have root on the host

Network Namespace: Container Network Isolation

Each container gets its own network namespace with virtual Ethernet pair (veth) connecting to the host bridge.

flowchart LR
    CONT[Container net ns\neth0=172.17.0.2] <-->|veth pair| BRIDGE[docker0 bridge\n172.17.0.1]
    BRIDGE --> HOST[Host network ns\neth0=192.168.1.10]

Network Namespace Attack Scenarios

Scenario Risk
--network=host Container shares host network stack; can bind host ports; can sniff host traffic
CAP_NET_ADMIN + net ns Container can modify its own routing, arp cache, iptables
Unfiltered inter-container communication Container A can reach container B directly on bridge network

PID Namespace and Isolation Gaps

Without a PID namespace, container processes appear in the host's PID list. With a PID namespace, container processes have isolated PIDs — but the host can still see and kill them by their host PID.

Security Implications

  • No PID ns → container process can kill -9 other user's processes (if same UID), read /proc/<host-pid>/environ for sensitive data
  • With PID ns → container process 1 is its own init; container cannot signal host processes; /proc is largely isolated
  • Shared PID ns (--pid=host) → container can ptrace host processes if CAP_SYS_PTRACE is held

Namespace Escape Techniques

1. Exploiting a privileged sibling container

If you can communicate with a privileged container (one with CAP_SYS_ADMIN or host mounts), you may be able to trigger it to run commands for you.

2. nsenter from a privileged context

From the host or from a container with sufficient privileges:

# Enter all namespaces of a target container
nsenter -t <container-pid> --all -- /bin/bash

3. Host PID namespace shared

If --pid=host is set, the container can see host processes. With CAP_SYS_PTRACE, it can inject into them.

4. User namespace + kernel exploit chain

Unpriv user creates user namespace → gains full capabilities within that ns → exploits kernel bug reachable from that privileged position → escapes to real root.

Hardening: Namespace Configuration Checklist

Control How Why
Never use --network=host Omit flag; use bridge networking Prevents host network stack access
Never use --pid=host Omit flag Prevents host process visibility
Never mount /var/run/docker.sock Audit docker run and Compose files Prevents trivial container escape
Never mount host root or sensitive dirs Audit volume mounts Prevents host filesystem access
Restrict unprivileged user ns creation sysctl kernel.unprivileged_userns_clone=0 Reduces kernel attack surface
Use user namespaces for rootless containers Run Docker or Podman in rootless mode Reduces blast radius of container escape
Audit namespace sharing in Kubernetes hostPID: false, hostNetwork: false, hostIPC: false in pod spec Prevents namespace sharing misconfigs at cluster level

Kubernetes Namespace Security Context

# Secure pod spec - deny all dangerous host namespace sharing
spec:
  hostPID: false
  hostNetwork: false
  hostIPC: false
  containers:
    - name: app
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        runAsNonRoot: true
        runAsUser: 1000
        capabilities:
          drop: ["ALL"]

What Can Go Wrong

Failure Root Cause Impact
Docker socket mounted into container Convenience for CI pipelines Trivial full host compromise
--privileged container No namespace/cap restrictions Full host access; same as root on host
--network=host + sensitive service Network isolation not applied Container can bind host ports, sniff traffic
User namespace exploit chain Unpatched kernel + unprivileged user ns Unprivileged user to host root
Host /proc mounted in container Debug shortcut Container reads all host process secrets

Key Takeaways

  • Namespaces are the isolation primitive for containers — not a security boundary by themselves.
  • Container security depends on correct namespace configuration + seccomp + capabilities + non-root user.
  • The Docker socket mount is the most common and most dangerous misconfiguration.
  • User namespaces expand attack surface for kernel bugs — restrict unprivileged creation in production.
  • --privileged negates nearly all namespace-based isolation — never use in production.
