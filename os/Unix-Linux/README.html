Unix And Linux Security Foundations

Category: Unix, Linux, and access control internals

Overview

Unix and Linux systems underpin the majority of modern cloud workloads, containers, CI runners, appliances, and network services. Understanding the kernel, userspace, filesystem permissions, and Linux-specific policy systems is essential because many "application security" incidents are really failures of host isolation, filesystem hygiene, or identity integration.

At production scale, Linux hardening is about consistency. One well-hardened host helps little if ten thousand similar hosts drift into inconsistent package versions, weak /tmp handling, stale SSH keys, or permissive LDAP-driven sudo rules.

Kernel, Userspace, And Permissions

The kernel is the privileged core of the operating system. It manages process scheduling, memory, filesystems, devices, networking, and access-control checks. Userspace contains ordinary processes such as shells, daemons, web servers, database clients, and utilities that rely on the kernel through system calls.

Why The Boundary Matters

Boundary Security Meaning
Kernel vs userspace A kernel bug usually defeats most local isolation
Process vs process File descriptors, namespaces, ptrace rules, and credentials decide what one process can inspect or change
User vs root Root can usually reconfigure the host unless MAC or hardware-backed protections restrict it
Namespace vs host Container isolation is meaningful only if kernel and runtime boundaries hold

Classic Unix permissions are built around:

  • owner
  • group
  • other
  • read, write, execute bits
  • special bits such as setuid, setgid, and sticky bit

These controls are simple and powerful, but they are discretionary: owners and privileged users can often reassign or relax them.

MAC Vs DAC

DAC (Discretionary Access Control) means the owner of an object or a privileged actor can decide access. Traditional Unix file permissions and many ACL systems are DAC.

MAC (Mandatory Access Control) means access is governed by centrally defined policy that ordinary users, and often even many privileged processes, cannot arbitrarily bypass.

Practical Difference

Model Strength Weakness
DAC Flexible and easy to understand A compromised privileged process can often change its own protections
MAC Stronger containment and policy consistency Harder to operate; misconfiguration can break workloads

For large infrastructure, MAC is valuable because it limits how far a compromised service can roam even when the service already has high Unix privileges relative to its own files.

SELinux

SELinux is a Linux MAC system that labels subjects and objects and evaluates access against policy. Instead of only asking "does this UID own this file?" SELinux also asks "is a process in this domain allowed to perform this class of action on an object with this type?"

Why SELinux Matters

  • It limits blast radius after process compromise.
  • It prevents many accidental cross-service accesses.
  • It makes policy explicit rather than relying only on filesystem ownership.

Core SELinux Concepts

Concept Meaning
Context / label Security identity attached to process or object
Domain Type associated with a running process
Type enforcement Policy that defines which domains can access which types
Enforcing mode Policy violations are blocked
Permissive mode Violations are logged but not blocked

Common Operational Mistakes

Mistake Result
Turning SELinux off instead of fixing labels Permanent loss of an important containment layer
Running too much in one domain One compromise reaches many resources
Ignoring audit logs Teams miss policy violations until production outage or incident

/proc

/proc is a virtual filesystem exposing kernel and process state. It is invaluable for diagnostics and also a rich source of sensitive information.

Why /proc Matters

Path Example Security Relevance
/proc/<pid>/cmdline Leaks process arguments, sometimes including secrets
/proc/<pid>/environ Leaks environment variables and tokens
/proc/<pid>/maps Helps attackers understand memory layout
/proc/net/* Reveals socket and network state
/proc/sys/* Exposes and controls kernel tunables

The lesson is operational: do not place secrets in command-line arguments or broadly readable environment variables if you can avoid it. Many incident-response and forensic workflows depend on /proc, but so do attackers after local compromise.

/tmp

/tmp is a world-writable temporary directory on most Unix-like systems. It is useful and dangerous for the same reason: many processes can write there.

Why /tmp Creates Risk

  • temporary files may accidentally contain credentials, dumps, or unencrypted sensitive data
  • race conditions and symlink attacks can occur if privileged code handles temp files unsafely
  • operators and malware alike use /tmp as a staging area because it is easy to write to

Hardening Guidance

Control Why It Helps
Use secure temp-file APIs Avoid predictable names and symlink races
Mount with noexec where feasible Raises the bar for simple staging and execution
Clean temp files aggressively Reduces forensic residue and accidental data exposure
Separate temp areas for sensitive services Limits cross-process abuse

noexec is useful but not sufficient. Attackers can still execute interpreters that read files from /tmp, use memory-only loaders, or copy payloads elsewhere. Treat it as friction, not a complete defense.

/shadow

/etc/shadow stores password hashes and related metadata for local Unix accounts. It is usually readable only by root or tightly controlled components.

Why /shadow Is Sensitive

Risk Why It Matters
Hash theft Offline cracking can recover weak passwords without further host interaction
Legacy hash formats Weak hash schemes shorten cracking time dramatically
Shared local accounts Same password across hosts multiplies blast radius

Defensive Priorities

  • prefer strong password hashing schemes such as yescrypt, scrypt, bcrypt, or strong SHA-512-based configurations as supported
  • eliminate unnecessary local passworded accounts on fleet systems
  • favor centralized authentication and strong MFA-backed admin workflows over shared local credentials

LDAP - Lightweight Directory Access Protocol

LDAP is a protocol for querying and modifying directory information such as users, groups, organizational units, and service records. In Unix and Linux environments, LDAP commonly integrates through NSS, PAM, and tools such as SSSD so that one central identity source can serve many systems.

See the broader directory overview in Directory Services.

Why LDAP Matters On Unix And Linux

Use Operational Meaning
Centralized account lookup Users can authenticate consistently across many systems
Group-based authorization Access can be managed without editing each host locally
Sudo and policy integration Central directory data may influence privileged operations
Service discovery and metadata Apps may rely on directory attributes for behavior

Key Risks

Risk Description
Plain LDAP without transport protection Credentials and directory queries can be exposed
Over-trust in directory groups Bad group data becomes bad authorization everywhere
Offline cache risk Cached credentials on endpoints can help attackers after host compromise
Directory outage Authentication dependency can become availability dependency

Engineering Guidance

Use LDAPS or StartTLS appropriately, keep directory privilege separate from host-root privilege where possible, and be careful about which directory attributes can influence sudo, SSH access, or application authorization. A central directory is convenient, but it also becomes a high-value dependency.

What Can Go Wrong At Scale

Failure Root Cause Impact
One container escape becomes node compromise Weak kernel hardening or dangerous capabilities Multi-tenant isolation failure
Secrets leak from /proc or temp files Apps place secrets in process args, env vars, or temp storage Credential theft and lateral movement
SELinux disabled fleet-wide for convenience Policy friction not addressed properly Loss of containment across many services
LDAP misconfiguration breaks auth everywhere Central dependency with poor change controls Large availability incident

Key Takeaways

  • Kernel, userspace, identity, and filesystem boundaries are the core security model of Unix and Linux.
  • DAC is necessary but often not sufficient; MAC systems such as SELinux improve containment.
  • /proc, /tmp, and /etc/shadow are operationally useful and security-sensitive at the same time.
  • LDAP brings central identity convenience but also concentrates risk and failure modes.
  • Fleet consistency and policy hygiene matter as much as any one hardening knob.
