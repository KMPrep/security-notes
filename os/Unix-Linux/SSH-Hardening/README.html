SSH Hardening

Category: Unix/Linux Security, Remote Access
Parent: os/Unix-Linux/README.md

Overview

SSH (Secure Shell) is the primary remote administration protocol for Linux systems. It is also one of the most targeted attack surfaces on internet-facing hosts. SSH hardening has two goals:

  1. Reduce authentication attack surface — prevent brute force, credential spraying, and weak-key authentication.
  2. Minimize post-authentication privilege — limit what an authenticated user can do and where they can go.

At scale, SSH hardening should be enforced via configuration management (Ansible, Puppet) with automated compliance scanning, not applied manually per host.

sshd_config: Critical Security Settings

# /etc/ssh/sshd_config

# --- Authentication ---

# Disable root login entirely
PermitRootLogin no
# If root login is needed for recovery: PermitRootLogin prohibit-password (key only, no password)

# Disable password authentication — require key-based auth
PasswordAuthentication no
ChallengeResponseAuthentication no
KbdInteractiveAuthentication no    # Also disables keyboard-interactive methods

# Disable empty passwords (extra safety)
PermitEmptyPasswords no

# Limit authentication attempts per connection
MaxAuthTries 3

# Disconnect if no successful auth within 60 seconds
LoginGraceTime 60

# Maximum number of concurrent unauthenticated connections
MaxStartups 10:30:60    # start:rate:full — throttle new connections

# --- Access Control ---

# Allow only specific users or groups
AllowUsers alice bob deploy-user
AllowGroups sshusers

# Deny specific users
DenyUsers guest nobody

# Restrict to specific source IPs (if AllowUsers is insufficient)
# This is done per-user via Match block (see below)

# --- Network ---

# Listen on non-default port (security through obscurity — reduces scan noise)
Port 2222   # or keep 22; matter of policy

# Bind to specific interface only (if multi-homed)
ListenAddress 10.0.1.5

# Disable protocol 1 (already default off, but explicit)
Protocol 2

# Disable GSSAPI authentication (reduces attack surface if not using Kerberos)
GSSAPIAuthentication no

# --- Ciphers and Algorithms ---
# Restrict to strong algorithms only (remove legacy ciphers)
Ciphers aes256-gcm@openssh.com,aes128-gcm@openssh.com,aes256-ctr,aes192-ctr,aes128-ctr
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com,hmac-sha2-512,hmac-sha2-256
KexAlgorithms curve25519-sha256@libssh.org,ecdh-sha2-nistp521,ecdh-sha2-nistp384,ecdh-sha2-nistp256,diffie-hellman-group16-sha512,diffie-hellman-group18-sha512
HostKeyAlgorithms ssh-ed25519,rsa-sha2-512,rsa-sha2-256

# --- Forwarding (restrict to reduce pivot paths) ---
AllowAgentForwarding no    # Disable SSH agent forwarding (lateral movement risk)
AllowTcpForwarding no      # Disable TCP tunneling
GatewayPorts no            # Prevent remote forwarding binding to 0.0.0.0
X11Forwarding no           # Disable X11 forwarding

# --- Session Hardening ---
# Disconnect idle sessions after 5 minutes
ClientAliveInterval 300
ClientAliveCountMax 2
# Client will be disconnected after 300 * 2 = 600 seconds of inactivity

# Disable TCP keepalives (use ClientAlive instead)
TCPKeepAlive no

# Log verbosity (INFO is baseline; VERBOSE adds key fingerprints)
LogLevel VERBOSE

# Disable user environment file (~/.ssh/environment) — can be used to set LD_PRELOAD
PermitUserEnvironment no

# Disable .rhosts and /etc/hosts.equiv (legacy trust mechanisms)
IgnoreRhosts yes
HostbasedAuthentication no

# --- Subsystem restriction ---
# Restrict SFTP to a chroot (for file transfer accounts)
Subsystem sftp /usr/lib/openssh/sftp-server

Match Blocks: Per-User and Per-IP Restrictions

# Allow a deploy account only from a specific IP range
Match User deploy-user Address 10.0.1.0/24
    AllowTcpForwarding no
    AllowAgentForwarding no
    ForceCommand /usr/local/bin/deploy-hook.sh
    PermitTTY no

# Allow jump host to use TCP forwarding (bastion)
Match Group bastion-users
    AllowTcpForwarding yes
    AllowAgentForwarding no
    PermitTTY no

# Chroot SFTP users to their home directory
Match Group sftp-only
    ForceCommand internal-sftp
    ChrootDirectory /home/%u
    PermitTTY no
    AllowAgentForwarding no
    AllowTcpForwarding no

SSH Key Management

Generating Strong Keys

# Ed25519 — best for new deployments (compact, fast, strong)
ssh-keygen -t ed25519 -C "user@host-$(date +%Y-%m-%d)"

# RSA 4096 — still acceptable; needed for compatibility with some systems
ssh-keygen -t rsa -b 4096 -C "user@host"

# Never use: DSA, ECDSA with P-256 (if you suspect NIST curves), RSA < 2048

authorized_keys Restrictions

Each line in ~/.ssh/authorized_keys can carry per-key restrictions:

# Restrict to specific source IPs and specific command
from="10.0.1.100",command="/usr/local/bin/backup.sh",no-agent-forwarding,no-port-forwarding,no-pty,no-X11-forwarding ssh-ed25519 AAAA... backup-key-2025

# Restrict a monitoring key — no shell, no forwarding
command="/usr/local/bin/monitor.sh",no-agent-forwarding,no-port-forwarding,no-pty ssh-ed25519 AAAA... monitor-2025

Key Rotation Policy

Key type Rotation frequency Enforcement
User SSH keys Annually or on role change Managed via configuration management or PAM
Host keys On OS reinstall; immediately after suspected compromise ssh-keygen -A regenerates; update known_hosts
Bastion/jump host keys Annually SSH certificate authority simplifies this
Service account keys Short-lived preferred (certificate-based) Use SSH CAs with short TTL

SSH Certificate Authority (CA) Model

SSH certificates solve the key management problem at scale. Instead of distributing host keys to every client's known_hosts and distributing user keys to every server's authorized_keys, a CA signs certificates that servers and clients trust.

flowchart LR
    CA[SSH Certificate Authority] -->|Sign user cert with TTL| UCERT[User certificate - valid 8 hours]
    CA -->|Sign host cert| HCERT[Host certificate - valid 1 year]
    U[User] -->|Present UCERT| S[Server]
    S -->|Verify UCERT against CA pub key| OK[Auth allowed]
    U -->|Check HCERT against CA pub key| TRUST[Host trusted - no known_hosts needed]

# Create a CA key pair (stored securely in HSM or offline)
ssh-keygen -t ed25519 -f /etc/ssh/ca_key -C "company-ssh-ca"

# Sign a user's public key with 8-hour TTL
ssh-keygen -s /etc/ssh/ca_key \
    -I "alice@host-2025-06-05" \
    -n "alice,servers-prod" \
    -V +8h \
    ~/.ssh/id_ed25519.pub
# Outputs: ~/.ssh/id_ed25519-cert.pub

# Sign a host key
ssh-keygen -s /etc/ssh/ca_key \
    -I "web-01.example.com" \
    -h \
    -V +52w \
    /etc/ssh/ssh_host_ed25519_key.pub

# sshd_config: trust certificates signed by the CA
TrustedUserCAKeys /etc/ssh/ca_user_key.pub

# Client ~/.ssh/known_hosts: trust host certs signed by the CA
@cert-authority *.example.com ssh-ed25519 AAAA...ca-public-key

Benefits:

  • Certificates expire automatically — compromised keys self-expire
  • No per-server authorized_keys management
  • Audit who accessed what via certificate principal logging
  • Immediate revocation via RevokedKeys list or KRL (Key Revocation List)

Brute Force and Scanning Defense

fail2ban

fail2ban monitors SSH log files and bans IPs that exceed failed authentication attempts:

# /etc/fail2ban/jail.local
[sshd]
enabled = true
port = ssh
logpath = %(sshd_log)s
backend = %(sshd_backend)s
maxretry = 3
findtime = 300
bantime = 3600    # 1 hour; use -1 for permanent ban

UFW / iptables Rate Limiting

# Rate limit SSH connections using iptables
iptables -A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW -m recent --set
iptables -A INPUT -p tcp --dport 22 -m conntrack --ctstate NEW -m recent --update --seconds 60 --hitcount 4 -j DROP

# UFW rate limiting (simpler)
ufw limit ssh

Port Knocking

Port knocking hides SSH behind a firewall rule that only opens after a secret sequence of connection attempts to closed ports. Reduces automated scan noise but is security through obscurity — not a primary control.

SSH Audit: Checking Configuration Quality

# ssh-audit: comprehensive cipher and configuration audit
ssh-audit localhost

# Test specific cipher/MAC/kex support
nmap --script ssh2-enum-algos -p 22 <target>

# Check what algorithms the server offers
ssh -vvv user@host 2>&1 | grep -E "(kex|cipher|mac|host key)"

# Verify no root login
grep PermitRootLogin /etc/ssh/sshd_config

# Verify password auth disabled
grep PasswordAuthentication /etc/ssh/sshd_config

SSH as a Lateral Movement Vector

Once on a compromised host, attackers use SSH aggressively to pivot:

Technique How Detection
Agent hijacking If agent forwarding is enabled, any root user on the hop host can use the agent socket to authenticate as the user to other hosts SSH_AUTH_SOCK in process environment; AuthorizedKeysFile modifications
Stolen private keys Read ~/.ssh/id_* on compromised host File access audit on .ssh/ files
SSH ProxyJump / tunneling Use compromised host as a jump host Unusual port forwarding; internal network connections from SSH session
Known hosts pivoting Read ~/.ssh/known_hosts to discover reachable systems known_hosts file enumeration
Authorized keys scraping Read /home/*/.ssh/authorized_keys across fleet Cross-correlate who can auth where

Hardening against lateral movement:

  • AllowAgentForwarding no in sshd_config
  • AllowTcpForwarding no unless bastion-specific
  • Read-protect ~/.ssh/ directories (chmod 700)
  • Use SSH certificate-based auth with short TTLs instead of persistent keys

What Can Go Wrong

Failure Root Cause Impact
PasswordAuthentication yes on internet-facing host Default or required for old clients Brute force / credential spray attacks
PermitRootLogin yes Operational convenience Direct root compromise from SSH attack
AllowAgentForwarding yes without need Default on many systems Agent hijacking enables lateral movement
Weak host key algorithm (DSA) Old config not updated Forged host identity; MITM
No key rotation No process for key lifecycle Stale compromised keys remain active indefinitely
No MaxAuthTries limit Default is 6 Brute force easier per connection

Key Takeaways

  • PasswordAuthentication no + PermitRootLogin no are the two non-negotiable baseline settings.
  • SSH certificate authorities eliminate the authorized_keys management problem at scale and enable short-lived credentials.
  • AllowAgentForwarding no prevents SSH agent hijacking — a major lateral movement vector.
  • fail2ban or iptables rate limiting is standard on internet-facing SSH ports.
  • Run ssh-audit against all exposed SSH services as part of the security posture review.
