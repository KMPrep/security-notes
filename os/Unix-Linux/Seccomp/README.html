Seccomp — Secure Computing Mode

Category: Unix/Linux Security, Sandboxing
Parent: os/Unix-Linux/README.md

Overview

Seccomp (Secure Computing Mode) is a Linux kernel feature that restricts which system calls a process can make. It is one of the most effective sandboxing primitives available to application developers and container runtimes because it reduces the kernel attack surface exposed to a potentially compromised process.

The threat model is: a process is compromised and the attacker now controls its execution. Seccomp limits what that attacker can do by denying access to dangerous syscalls — even if the process is running as root inside a container.

Seccomp Modes

Mode What it does
SECCOMP_MODE_STRICT Only read, write, _exit, sigreturn are allowed; any other syscall kills the process
SECCOMP_MODE_FILTER (BPF) A BPF (Berkeley Packet Filter) program is loaded to evaluate each syscall; can allow, deny, log, or return an error

Mode 2 (filter) is what all modern sandboxing uses. Mode 1 is too restrictive for any real application.

How Seccomp-BPF Works

flowchart TD
    P[Process makes syscall] --> K[Kernel: is seccomp filter loaded?]
    K -->|Yes| BPF[Run BPF filter program]
    BPF --> R{Filter verdict}
    R -->|ALLOW| Exec[Execute syscall normally]
    R -->|ERRNO| Err[Return -EPERM or custom errno to process]
    R -->|KILL| Kill[Kill process immediately with SIGSYS]
    R -->|TRACE| Trace[Notify tracer via ptrace]
    R -->|LOG| Log[Allow but log the syscall]
    K -->|No| Exec

The BPF program evaluates the syscall number and, optionally, syscall arguments. Filters are composed as a list of BPF instructions that run in order until a terminal verdict is reached.

Writing a Seccomp Filter with libseccomp

#include <seccomp.h>
#include <unistd.h>

int main() {
    // Create a new filter — default action: kill process
    scmp_filter_ctx ctx = seccomp_init(SCMP_ACT_KILL);

    // Allow only the syscalls the process needs
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(read),   0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(write),  0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(exit),   0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(brk),    0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(mmap),   0);
    seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(fstat),  0);

    // Load the filter into the kernel
    seccomp_load(ctx);
    seccomp_release(ctx);

    // Process is now restricted to the allowlisted syscalls
    return 0;
}

Docker Default Seccomp Profile

Docker ships a default seccomp profile that blocks ~44 syscalls out of ~300+ available. Key blocked syscalls:

Blocked Syscall Why Dangerous
ptrace Attach to any process; read/write memory
kexec_load Load a new kernel
create_module Load kernel modules
init_module Load kernel modules (newer API)
delete_module Unload kernel modules
mount Mount filesystems (needed for escape techniques)
unshare Create new namespaces
clone with CLONE_NEWUSER Create user namespaces
perf_event_open Performance monitoring; Spectre exploitation path
bpf Load BPF programs; can be used to escape filters
pivot_root Change root filesystem
acct Enable/disable process accounting

Check active profile:

docker run --security-opt seccomp=default alpine cat /proc/1/status | grep Seccomp
# Seccomp: 2 (means BPF filter active)

Kubernetes Seccomp

In Kubernetes, seccomp profiles are applied at the pod or container level:

# Apply the runtime default seccomp profile (Docker/containerd default)
spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault   # recommended baseline

# Apply a custom profile from a node-local file
spec:
  securityContext:
    seccompProfile:
      type: Localhost
      localhostProfile: profiles/my-app.json

# Unconfined (no seccomp) — avoid in production
spec:
  securityContext:
    seccompProfile:
      type: Unconfined

Pod Security Standards (Kubernetes 1.25+):

  • Restricted profile requires RuntimeDefault or Localhost seccomp
  • Baseline profile allows Unconfined but recommends RuntimeDefault

Syscall Categories to Harden Against

Category Dangerous Syscalls Why
Kernel module loading init_module, finit_module, delete_module Load rootkit/backdoor into kernel
Namespace creation unshare, clone with ns flags Create user ns to gain caps; pivot namespaces
Memory execution mprotect with PROT_EXEC Mark heap/stack as executable for shellcode
Process injection ptrace, process_vm_writev Inject code into other processes
Filesystem evasion mount, pivot_root, chroot Change root; mount host paths
Network raw access socket(AF_PACKET) Packet sniffing; ARP spoofing
Kernel debug perf_event_open, kexec_load Side-channel; kernel replacement

Seccomp Bypass Techniques

Technique Description
Syscall number mismatch 32-bit syscall numbers differ from 64-bit; if filter checks 64-bit numbers only, use int 0x80 (32-bit ABI) with different numbers
Syscall argument bypass Filter checks syscall number but not args; use a permitted syscall with crafted arguments to achieve denied effect
BPF filter race Race between seccomp filter installation and syscall execution in multithreaded programs
New syscall not in filter Kernel adds new syscalls (e.g., io_uring, landlock); old filters don't cover them
io_uring escape io_uring submission queue ring can perform file operations that bypass per-syscall seccomp filters

Auditing Seccomp Coverage

# Run a process under strace to capture all syscalls it makes
strace -f -e trace=all ./myapp 2>&1 | awk '{print $1}' | sort -u

# Use seccomp-tools to analyze BPF programs
seccomp-tools dump ./myapp

# Check if a container's process has seccomp active
cat /proc/<pid>/status | grep Seccomp
# 0 = no filter, 1 = strict mode, 2 = filter (BPF) mode

What Can Go Wrong

Failure Root Cause Impact
No seccomp profile applied Default Unconfined in Kubernetes All syscalls available; no kernel surface reduction
Overly permissive profile All syscalls allowed except obvious ones Dangerous syscalls (ptrace, mount) still reachable
32-bit syscall bypass Filter only checks 64-bit numbers Attacker uses int 0x80 to call denied syscalls
New syscall not in filter io_uring or landlock added after profile written Bypass via new unfiltered syscall
Profile applied in log-only mode SCMP_ACT_LOG instead of SCMP_ACT_KILL Violations logged but not blocked

Key Takeaways

  • Seccomp reduces the kernel attack surface available to a compromised process — essential for containers.
  • Docker's default profile blocks ~44 dangerous syscalls; it is a good baseline but not sufficient for high-security workloads.
  • Apply RuntimeDefault seccomp in all Kubernetes workloads; enforce with Pod Security Admission.
  • Audit actual syscall usage before writing a restrictive allowlist — strace and seccomp-tools are the right tools.
  • io_uring is a known seccomp bypass path; consider disabling or restricting it for high-security containers.
