Linux Security — Practice Questions

Category: Unix/Linux Security, Exploitation, Hardening
Parent: os/Unix-Linux/README.md

Overview

These 20 questions cover the full Linux security domain: exploitation techniques, privilege escalation, container isolation, kernel hardening, MAC systems, capabilities, namespaces, and SSH. They are written at the Google senior security engineer bar — each question expects a structured, technically precise answer that addresses the mechanism, attack, defense, and scale implications.

For each question: answer aloud, timed (5–8 minutes), no notes.

Questions

1. Linux Privilege Escalation: SUID and Sudo

Question:

"You land a shell as www-data on a Linux server. Walk me through your privilege escalation enumeration process from first command to root shell — covering at least four distinct techniques."

What a strong answer covers:

  • id, sudo -l, find / -perm -4000 as the first three commands
  • sudo misconfiguration: any GTFOBins binary (vim, find, python3) with NOPASSWD = trivial root
  • SUID binaries: look for non-standard SUID binaries (everything not in /bin, /usr/bin default); exploit via -p flag or by injecting through relative PATH calls
  • File capabilities: getcap -r / — cap_setuid+ep on python3 or similar is equivalent to SUID root
  • Cron jobs: pspy to watch cron execution; writable script called by root cron → append command
  • Kernel CVEs: uname -r → linux-exploit-suggester → PwnKit, Dirty Pipe, Dirty COW based on version

2. Linux Capabilities

Question:

"A container is running with CAP_SYS_ADMIN. What are the three most dangerous things an attacker inside that container can do, and how would you detect and prevent this configuration?"

What a strong answer covers:

  • Escape via cgroup release_agent: mount a cgroup with release_agent pointing to a host-side command → triggers on container process exit → code runs as root on host
  • Namespace operations: unshare, nsenter, pivot_root — all require CAP_SYS_ADMIN; can be used to escape mount namespace
  • Raw device access and mount: can mount host filesystems, bind mount /etc into container, then modify /etc/passwd
  • Detection: docker inspect or Kubernetes securityContext.capabilities; alert on containers with CAP_SYS_ADMIN or --privileged in CI/CD pipeline scan
  • Prevention: Pod Security Admission restricted profile; OPA/Gatekeeper policy blocking CAP_SYS_ADMIN; drop ALL capabilities, add only what's needed

3. Namespaces and Container Escape

Question:

"What is the Docker socket escape technique? What makes it so dangerous, and what does a defense-in-depth response look like?"

What a strong answer covers:

  • The Docker socket (/var/run/docker.sock) gives whoever can write to it full control over the Docker daemon
  • Inside a container with the socket mounted: docker run -it -v /:/host ubuntu chroot /host → you are now root on the host filesystem
  • Why it's so dangerous: it's a common misconfiguration in CI/CD (Jenkins, GitLab runners) because the container "needs to build Docker images"
  • Defense layer 1: never mount the Docker socket; use kaniko, Buildah, or Tekton for in-cluster builds without Docker socket access
  • Defense layer 2: if socket mounting is unavoidable, use docker:dind (Docker-in-Docker) as a sidecar — isolates the inner Docker daemon
  • Defense layer 3: seccomp + AppArmor + capabilities even if socket is present (defense in depth)
  • Detection: audit docker inspect for Binds containing docker.sock; Falco rule on container creation from inside a container

4. Seccomp

Question:

"What is seccomp-BPF and how does it reduce the kernel attack surface? What are the two main bypass techniques, and how do you verify it is applied correctly to a container?"

What a strong answer covers:

  • Seccomp-BPF loads a BPF program into the kernel that evaluates every syscall before execution; verdict: ALLOW, ERRNO, KILL, LOG
  • Reduces attack surface: dangerous syscalls (ptrace, mount, init_module, kexec) become unreachable even from root inside the container
  • Bypass 1: 32-bit syscall numbers via int 0x80 — if the BPF filter checks 64-bit numbers only, 32-bit syscall table is not covered; modern filters should set SECCOMP_FILTER_FLAG_LOG and deny arch=x86_32
  • Bypass 2: io_uring — post-filter I/O ring can perform file operations not caught by per-syscall filters on some kernel versions
  • Verification: cat /proc/<pid>/status | grep Seccomp (value 2 = BPF active); use seccomp-tools dump ./binary to inspect filter; check Kubernetes pod spec has seccompProfile: RuntimeDefault

5. SELinux in Production

Question:

"An operations team wants to set SELinux to permissive because a new service is generating AVC denials. What is the security impact, and what is the right engineering response?"

What a strong answer covers:

  • Permissive mode logs violations but does not enforce — the entire MAC protection layer becomes advisory only
  • Impact: a compromised service can now read any file it can reach with DAC (e.g., /etc/shadow if readable), make network connections, write to log files of other services
  • Never set the entire system to permissive; instead: set the specific domain to permissive (semanage permissive -a httpd_t)
  • Diagnose: ausearch -m AVC -ts recent | audit2why to understand what the service needs
  • Fix: generate a targeted policy module with audit2allow and review it before applying; fix file labels with restorecon if the issue is mislabeled files; set booleans if a legitimate use case exists
  • Production standard: enforcing mode always; permissive only for individual domains during development; never SELINUX=disabled (requires full relabel to re-enable; breaks audit trail)

6. Kernel Hardening: ptrace_scope

Question:

"What is kernel.yama.ptrace_scope and why does each value level represent a different security posture? What is the recommended production setting?"

What a strong answer covers:

  • ptrace is the syscall that allows one process to inspect and control another — used by debuggers, strace, gdb
  • ptrace_scope=0: any process can ptrace any other process of the same UID — easy cross-process injection
  • ptrace_scope=1 (recommended): a process can only ptrace its own children (parent-child relationship required)
  • ptrace_scope=2: only processes with CAP_SYS_PTRACE can ptrace
  • ptrace_scope=3: ptrace completely disabled; even root cannot attach
  • Attack with scope=0: attacker with any user shell can inject shellcode into any browser process, daemon, or privileged helper running as the same UID
  • Production recommendation: scope=1 minimum; scope=2 on high-security or shared systems
  • Container relevance: even with PID namespace isolation, if --pid=host is set or kernel doesn't enforce ns boundaries for ptrace, scope=1 is critical

7. AppArmor vs SELinux

Question:

"Compare AppArmor and SELinux. When would you choose one over the other, and what is the hardest operational challenge with each?"

What a strong answer covers:

  • SELinux: label-based (subjects and objects both labeled); type enforcement; every file/process/socket has a type; policy says domain X can access type Y via operation Z; very granular; used on RHEL/CentOS/Fedora/Android
  • AppArmor: path-based profiles per application; easier to write and understand; profile says /usr/sbin/nginx can read /etc/nginx/, write /var/log/nginx/; used on Ubuntu/Debian
  • SELinux hardest operational challenge: relabeling — if you disable SELinux (not even permissive, but disabled) and then re-enable, entire filesystem must be relabeled; also, any file in wrong location has wrong type and generates AVC
  • AppArmor hardest challenge: path-based means new file locations or moved binaries need profile updates; also symlinks can be followed outside profile-defined paths
  • Choice: use what the distribution ships with by default — changing MAC systems is high operational cost; on RHEL use SELinux; on Ubuntu use AppArmor; both are better than neither
  • Container context: both systems can be applied to containers (Docker + AppArmor by default; Kubernetes supports both via annotations/securityContext)

8. Dirty Pipe (CVE-2022-0847)

Question:

"Explain the Dirty Pipe vulnerability — what is the root cause, what can an attacker do with it, and what made it particularly severe for container environments?"

What a strong answer covers:

  • Root cause: Linux pipe infrastructure maintains a reference to memory pages; a flag (PIPE_BUF_FLAG_CAN_MERGE) was not correctly reset when a splice'd page was reused via pipe write, allowing a write to overwrite the page cache of a read-only file
  • What an attacker can do: any user (UID != 0) can overwrite the content of any readable file that is open for reading, even if the file is read-only — including SUID binaries, /etc/passwd, /etc/shadow
  • Classic exploit: overwrite the password hash in /etc/passwd or overwrite an SUID binary with a shell payload → instant root
  • Kernel versions affected: 5.8 through 5.16.11, 5.15.25, 5.10.102
  • Container severity: containers run many services as root inside the container but rely on the read-only root filesystem (readOnlyRootFilesystem: true) as a security control; Dirty Pipe allowed writing to those read-only files even from within the container, enabling overwriting the container runtime binary (runc) on the host
  • Fix: upgrade kernel; patch was released within days; demonstrates importance of kernel patch cadence

9. SSH Agent Forwarding Risk

Question:

"Your team uses SSH agent forwarding to access production servers through a bastion host. What is the specific attack, who can carry it out, and how do you redesign the architecture to eliminate it?"

What a strong answer covers:

  • With agent forwarding enabled, the SSH agent socket is forwarded to the bastion; the socket lives in /tmp/ssh-XXXXXX/agent.<pid> on the bastion
  • Any root user (or user with access to the socket) on the bastion can use SSH_AUTH_SOCK=/tmp/ssh-.../agent.X ssh user@prod-server — the agent handles the private key operation without the key ever leaving the original client
  • Attack: bastion is compromised, or another user on the bastion has root → they walk laterally to any server the forwarded agent has access to — all without knowing the key or passphrase
  • Redesign: disable AllowAgentForwarding in sshd_config on the bastion; instead, use an SSH jump host (-J flag) which does not forward the agent socket — the client's SSH client makes a direct connection through the bastion without the bastion ever seeing the agent
  • Better: SSH certificate authority with short-lived user certificates (8-hour TTL); bastion validates the cert; prod servers validate the cert; no persistent keys on any server
  • Best: BeyondCorp/ZTNA model — no bastion; context-aware access proxy validates device + identity + cert before proxying the connection

10. Container Escape via Kernel Exploit

Question:

"A container is running with the minimum possible privileges — non-root UID, dropped all capabilities, seccomp + AppArmor applied, read-only root filesystem. Can it still be escaped? How?"

What a strong answer covers:

  • Yes, via a kernel vulnerability — namespaces, capabilities, seccomp, and AppArmor are all kernel features; a bug in the kernel can bypass them all
  • Examples: CVE-2022-0492 (cgroup escape via release_agent even without CAP_SYS_ADMIN on some kernel configurations), CVE-2021-22555 (netfilter heap overflow → root), runc CVE-2019-5736 (overwrite runc binary via /proc/self/exe)
  • The key insight: container isolation is userspace + kernel policy — if the kernel policy can be circumvented by a kernel bug, the entire isolation stack fails
  • Mitigations that help reduce this risk: kernel patching (fastest fix after CVE disclosure); Seccomp blocks many syscall paths that kernel exploits use; gVisor/Kata Containers (adds a second kernel boundary — guest kernel + hypervisor between container and host); user namespaces map container root to host UID 65534
  • For highest-security workloads: use Kata Containers or gVisor; accept that standard container isolation is not equivalent to VM isolation
  • Detection: Falco rules monitoring for privilege escalation syscalls; CAP_SYS_ADMIN usage inside a container; /proc/sys writes

11. Format String Vulnerabilities

Question:

"Explain what a format string vulnerability is, what primitives it gives an attacker, and give a concrete example of how it can bypass ASLR."

What a strong answer covers:

  • Vulnerability: user-controlled input passed directly as the format argument to printf — printf(user_input) instead of printf("%s", user_input)
  • Read primitive: %p prints a pointer from the stack; %s reads a string from a stack address; can leak libc base address, stack canary, heap pointer
  • Write primitive: %n writes the count of printed characters to the next pointer argument; by controlling padding with %<N>x, attacker can write any 4-byte value to any address
  • ASLR bypass: leak a libc pointer via %<n>$p (positional format argument) → compute libc base = leaked_addr - known_offset → compute system() = libc_base + system_offset → overwrite GOT entry for an often-called function with system() via %n
  • Modern defenses: Full RELRO (GOT read-only — blocks GOT overwrite); FORTIFY_SOURCE detects some format string misuse at compile time; the real fix is always printf("%s", input) — never trust user input as a format string

12. ASLR and Information Leaks

Question:

"ASLR is supposed to prevent exploitation by randomizing memory layout. Describe three techniques attackers use to defeat ASLR, and for each, identify which mitigation specifically addresses it."

What a strong answer covers:

  • Technique 1 — Format string / OOB read leak: read a libc pointer from memory (format string, UAF, OOB) → compute base from known offset → bypass ASLR entirely. Mitigation: Full RELRO; FORTIFY_SOURCE; eliminate the read primitive at the vulnerability layer.
  • Technique 2 — Partial overwrite: ASLR randomizes the upper bytes of addresses but the lower 12 bits are fixed (page alignment); in 32-bit, entropy is low (~8 bits); attacker can brute-force. Mitigation: 64-bit with high ASLR entropy (randomize_va_space=2); 32-bit ASLR is effectively broken.
  • Technique 3 — Heap spray: spray the heap with so many copies of the payload that statistically one lands under a wild pointer, regardless of ASLR. Mitigation: heap guard pages; JIT hardening in browsers; init_on_alloc=1 to zero fresh pages.
  • Key point for real systems: ASLR is a probabilistic defense, not a guarantee; its value is to force the attacker to chain an info leak, increasing exploit complexity and detection opportunity.

13. Seccomp + Capabilities Defense in Depth

Question:

"Design the complete security posture for a container running a Node.js web application that handles user-uploaded files. What is the minimum-privilege configuration?"

What a strong answer covers:

  • Run as non-root: runAsUser: 1000, runAsNonRoot: true
  • Drop ALL capabilities: capabilities: drop: ["ALL"] — Node.js needs no capabilities
  • Read-only root filesystem: readOnlyRootFilesystem: true; mount a writable emptyDir only for /tmp and /var/uploads
  • Seccomp: RuntimeDefault or a custom profile allowing only: read, write, open, close, stat, mmap, brk, exit, futex, socket, connect, accept, send/recv, poll, epoll — block ptrace, mount, init_module, clone with ns flags
  • AppArmor: profile restricting file access to only /app/**, /tmp/**, /var/uploads/**; deny /etc, /proc/sys, /sys
  • Network policy: deny egress to metadata service (169.254.169.254); allow only to specific API backends
  • Pod Security Standard: restricted (enforces most of the above automatically)
  • File upload security (bonus): virus scan uploaded files before processing; quarantine in isolated pod; use ClamAV or Cloud DLP API to inspect content
  • Monitoring: Falco rules alerting on unexpected subprocess spawn (Node.js shouldn't fork shells), unexpected file access, or network connections to unusual destinations

14. Linux auditd for Detection

Question:

"How would you configure auditd on a fleet of Linux servers to detect privilege escalation attempts in real time? What are the five most important audit rules, and how do you avoid generating so much noise that analysts stop reviewing?"

What a strong answer covers:

  • Rule 1: -w /etc/sudoers -p wa -k sudo_change — any write to sudoers is a P1 security event
  • Rule 2: -a always,exit -F arch=b64 -S setuid -S setresuid -k privesc — capture all UID transitions
  • Rule 3: -w /etc/passwd -p wa -k identity and /etc/shadow — detect user database manipulation
  • Rule 4: -a always,exit -F arch=b64 -S init_module -S finit_module -k kernel_modules — detect rootkit loading
  • Rule 5: -a always,exit -F arch=b64 -S ptrace -k ptrace_use — detect cross-process inspection
  • Immutable rules: end with -e 2 — prevents an attacker with root from clearing audit rules
  • Noise reduction: do not log execve for all processes (extremely high volume); scope execve rules to specific directories or users; use ausearch --key + SIEM correlation instead of raw log review; set up alerts on high-priority keys (sudo_change, kernel_modules) that page SOC; low-priority keys feed into SIEM for weekly review

15. SSH Certificate Authority Design

Question:

"Your organization has 500 Linux servers and 1000 engineers. Describe the SSH access architecture you would design. How do you handle emergency access when the CA is unavailable?"

What a strong answer covers:

  • SSH CA architecture: offline root CA → online signing CA; user presents CSR; signing CA issues short-lived certificate (8-hour TTL, principal = username); server trusts TrustedUserCAKeys /etc/ssh/ca_user_pub.key
  • No persistent authorized_keys on servers; no long-lived SSH keys distributed to servers
  • Certificate contains: principal (username + POSIX role), valid-after, valid-before, critical options (source address restriction, force-command)
  • MFA integration: signing CA requires MFA (WebAuthn) before issuing certificate; ties SSH access to identity provider
  • Revocation: KRL (Key Revocation List) distributed to all servers; immediate revocation when employee leaves
  • Emergency access: breakglass accounts on each host with a long-lived key stored in a PAM vault (HashiCorp Vault); the key is checked out, checked in, rotated after each use; all breakglass access logged to tamper-evident store
  • Scale: certificate distribution is pull-based (servers fetch CA public key from S3/GCS at startup); no per-server configuration for new users
  • Audit: every certificate issuance logged to SIEM; every SSH login logs sshd: Accepted publickey for alice from X.X.X.X cert-id alice@host-2025-06-05 serial=12345 — full traceability
