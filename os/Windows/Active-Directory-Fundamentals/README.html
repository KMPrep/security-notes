Active Directory Fundamentals for Security

Category: Windows / Identity & Directory Services
Key standards / frameworks: RFC 4120 (Kerberos v5), RFC 4511 (LDAP v3), MS-ADTS, MS-DRSR, NIST SP 800-63, MITRE ATT&CK (TA0006 Credential Access, TA0004 Privilege Escalation)

Overview

Active Directory (AD) is Microsoft's directory service and the identity backbone of the vast majority of enterprise Windows estates. It stores users, computers, groups, and policy in a replicated, hierarchical database and issues the credentials that grant access across the domain. Because a single authentication to a domain controller can unlock file shares, applications, mail, and remote systems, AD implements a domain of trust that underpins single sign-on.

That same domain of trust is why AD is the crown-jewel attack surface. Compromise of the directory, and specifically of the krbtgt account or a Domain Admin, gives an adversary the ability to mint their own credentials for any principal and to persist below the level most defenders monitor. Understanding the structure, the trust boundaries, and the ticketing protocol is the prerequisite for both attacking and defending it.

How It Works

Logical Structure

A forest is the top-level security boundary and the unit of trust. Inside a forest, one or more domains form the replication and administrative boundary; domains sharing a contiguous DNS namespace form a tree, and multiple trees can coexist in one forest. Within a domain, Organizational Units (OUs) provide a container hierarchy for delegation and for linking Group Policy. Trusts (parent-child, tree-root, external, forest, and shortcut) let principals in one domain authenticate to resources in another. A critical security point: the forest, not the domain, is the true security boundary, because forest trusts and SID history can be abused across domain lines.

Physical and Service Components

Domain Controllers (DCs) host the directory database (NTDS.dit), run the Kerberos Key Distribution Center, and answer LDAP queries. SYSVOL is a replicated share on every DC that holds Group Policy Objects and logon scripts. A Global Catalog DC holds a partial, forest-wide read-only replica used for cross-domain lookups and universal group membership. Three authentication and lookup roles matter most: LDAP (directory reads and writes over port 389 or 636), Kerberos (the preferred ticket-based authentication over port 88), and NTLM (the legacy challenge-response fallback).

Identity Primitives

Every principal has a Security Identifier (SID) of the form S-1-5-21-<domain>-<RID>, where the RID is the per-domain relative identifier. Well-known RIDs matter: RID 500 is the built-in Administrator, and RID 512 is Domain Admins. SID history lets a migrated account carry its old SIDs in its token, a legitimate migration feature that is also an attacker persistence primitive. A Service Principal Name (SPN) maps a service instance to the account that runs it, which is what lets a client request a Kerberos ticket for a specific service. The krbtgt account is the KDC service account whose password hash encrypts every Ticket Granting Ticket in the domain.

Privileged Groups

The most sensitive built-in groups are Domain Admins (full control of a domain), Enterprise Admins (full control across every domain in the forest), and Schema Admins (able to modify the directory schema forest-wide). Membership in any of these is effectively Tier 0 and should be tightly controlled.

Detection & Telemetry

The richest AD telemetry comes from domain controller security logs. Kerberos authentication (TGT issuance) is Event ID 4768 and service ticket requests are Event ID 4769; watch for RC4 (encryption type 0x17) requests and high-volume 4769 events indicating Kerberoasting. Directory replication requests (Event ID 4662 with the DS-Replication-Get-Changes control access right, and DRSUAPI traffic) reveal DCSync. Group membership changes to Tier 0 groups appear as Event IDs 4728/4732/4756. Forwarding DC logs to a SIEM and baselining normal ticket and replication patterns is essential.

Key Takeaways

  • The forest, not the domain, is the real security boundary; plan trust and isolation accordingly.
  • Kerberos issues a TGT encrypted with the krbtgt hash, then service tickets encrypted with each service account hash; both hashes are prime targets.
  • SPNs, SIDs, RIDs, and SID history are identity primitives that attackers weaponize for roasting, forgery, and persistence.
  • Domain Admins, Enterprise Admins, and Schema Admins are Tier 0; membership must be minimal and governed.
  • Domain controllers, NTDS.dit, SYSVOL, and the krbtgt account are the crown jewels; access to any is close to game over.
  • Kerberos Event IDs 4768 and 4769 plus replication auditing are the backbone of AD attack detection.
  • Single sign-on convenience is exactly what makes AD compromise so catastrophic; defense in depth around identity is non-negotiable.
