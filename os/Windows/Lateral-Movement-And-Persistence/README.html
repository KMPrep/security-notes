Lateral Movement & Domain Persistence

Category: Windows / Lateral Movement & Persistence
Key standards / frameworks: MITRE ATT&CK TA0008 (Lateral Movement), TA0003 (Persistence), T1021 (Remote Services), T1550 (Use Alternate Authentication Material), T1207 (Rogue Domain Controller)

Overview

After initial credential access, an adversary spreads across the estate and then entrenches so that a single remediation does not evict them. Lateral movement in Windows overwhelmingly reuses native remote-administration channels, because blending in with legitimate admin traffic evades detection. Persistence in Active Directory ranges from stolen credentials and forged tickets to durable directory-object backdoors that survive password resets and even DC rebuilds if the directory itself is not cleaned.

The defender's leverage is that most of these techniques ride on identifiable services and directory changes, so instrumenting remote-execution channels and auditing sensitive ACLs turns stealthy activity into detectable events.

How It Works

Lateral Movement Channels

PsExec creates a service over SMB (ADMIN$) and runs a payload as SYSTEM. WMI (via Win32_Process Create over DCOM) and WinRM (PowerShell Remoting over WS-Management) execute commands remotely with valid credentials. DCOM objects such as MMC application classes offer additional execution paths. These channels combine with alternate authentication material: Pass-the-Ticket injects a stolen or forged Kerberos ticket into the current session, and token impersonation reuses the token of another logged-on user for their access without their password.

Domain Persistence

Golden and Silver Tickets provide forged, long-lived access as described in the Kerberos notes. Directory-resident persistence is more durable: abusing AdminSDHolder (whose ACL is stamped onto protected groups every 60 minutes by the SDProp process) plants a backdoor ACE that reappears even after removal. Granting an attacker-controlled account DCSync rights (the two replication extended rights) gives ongoing hash extraction. GPO abuse pushes malicious scheduled tasks or scripts to every linked machine. The Skeleton Key attack patches LSASS on a DC to accept a master password for any account, and SID history injection adds a privileged SID to a controlled account's token.

Detection & Telemetry

Lateral movement leaves consistent traces: service creation from PsExec appears as Event ID 7045 (System log) and 4697, while remote logons show as Event ID 4624 LogonType 3 and network authentication with 4648 (explicit credentials). WMI and WinRM execution surface in Sysmon and in the WinRM/WSMan operational logs. Persistence detection focuses on the directory: alert on ACL modifications to AdminSDHolder and on the two replication extended rights being granted, and on Event ID 4662 for GetNCChanges from non-DC accounts. GPO changes appear as Event IDs 5136/5137 (directory object modification) and in SYSVOL file monitoring. SID history additions and Skeleton Key deployment (an anomalous DC configuration or downgrade to RC4) warrant dedicated Tier 0 alerting.

Key Takeaways

  • Attackers prefer native channels (PsExec, WMI, WinRM, DCOM) because they blend with legitimate administration.
  • Pass-the-Ticket and token impersonation reuse existing authentication material, so credential hygiene and tiering matter more than password strength alone.
  • AdminSDHolder plus SDProp creates self-healing persistence; the backdoor ACE returns hourly unless the template itself is cleaned.
  • Granting DCSync rights and injecting SID history are quiet, durable persistence that survive user password resets.
  • GPO write access is a domain-wide code-execution primitive and must be treated as Tier 0.
  • Skeleton Key and rogue-DC techniques target the DC itself; LSA protection and Tier 0 isolation are the countermeasures.
  • Detection hinges on instrumenting remote-execution services (7045, 4697, 4624/4648) and auditing sensitive directory ACLs (4662, 5136).
