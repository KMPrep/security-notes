NTLM, Pass-the-Hash & Relay Attacks

Category: Windows / Legacy Authentication Abuse
Key standards / frameworks: MS-NLMP (NTLM), MS-NRPC, MS-EFSRPC, MS-RPRN, MITRE ATT&CK T1550.002 (Pass the Hash), T1557.001 (LLMNR/NBT-NS Poisoning and Relay)

Overview

NTLM is Microsoft's legacy challenge-response authentication protocol, still present in nearly every AD environment as a fallback. Its core weakness is that the client's response is derived directly from the NT hash of the password, and the protocol by default provides no binding between the authentication and the channel it travels over. This lets attackers replay hashes, relay authentication to other services, and coerce machines into authenticating on demand.

Unlike password cracking, pass-the-hash and relay do not require recovering plaintext; the hash or the live authentication itself is the currency. Combined with name-resolution poisoning and coercion, NTLM abuse is one of the most reliable paths from a foothold to privilege escalation and domain compromise.

How It Works

NTLM Challenge-Response

Authentication is a three-message exchange: the client sends a NEGOTIATE, the server replies with a CHALLENGE containing a random nonce, and the client returns an AUTHENTICATE message containing a response computed from the nonce and the user's NT hash. The server (or the DC via Netlogon) validates the response. Because the NT hash alone produces the response, possession of the hash is equivalent to possession of the password for NTLM purposes, which is the basis of Pass-the-Hash.

Poisoning and Coercion

When Windows fails to resolve a name via DNS, it falls back to LLMNR, NBT-NS, and mDNS broadcasts. Tools like Responder answer these broadcasts, claiming to be the requested host and capturing the victim's NTLM authentication. Attackers can also actively coerce authentication: PetitPotam abuses MS-EFSRPC to force a machine (including a DC) to authenticate to an attacker, and the PrinterBug / SpoolSample abuses the MS-RPRN print spooler for the same effect.

NTLM Relay

Instead of cracking the captured authentication, the attacker relays it in real time to another service using tools like ntlmrelayx. If SMB signing is not required, they authenticate to file servers or execute code. Relaying to LDAP on a DC can grant RBCD or DCSync rights; relaying coerced DC authentication to AD CS web enrollment (ESC8) yields a certificate for the DC, which is then used to obtain a TGT. Signing and channel binding are what break these chains.

Detection & Telemetry

NTLM authentication events appear as Event ID 4624 (successful logon, LogonType 3 with Authentication Package NTLM) and 4776 (NTLM credential validation on the DC). A spike in 4776 events referencing many source workstations, or logons where the NTLM hash is used from an unexpected host, indicates pass-the-hash. Responder activity can be detected with LLMNR/NBT-NS honeypot listeners and by monitoring for unusual name-resolution responses. AD CS enrollment abuse surfaces in the CA audit log (certificate issued to an unexpected requester) and Event ID 4886/4887. Coercion tools generate RPC calls to EFSRPC and spoolss endpoints that network sensors can flag.

Key Takeaways

  • NTLM derives its response from the NT hash, so the hash is as good as the password; pass-the-hash needs no cracking.
  • LLMNR, NBT-NS, and mDNS fallback let Responder harvest authentication passively; disabling them removes a major attack surface.
  • Relay converts captured authentication into live access; SMB and LDAP signing plus Extended Protection break the relay chain.
  • Coercion (PetitPotam, PrinterBug) forces even domain controllers to authenticate on demand, feeding relay attacks.
  • Relaying to AD CS web enrollment (ESC8) is a reliable path to a DC certificate and full domain compromise.
  • Unique local admin passwords via LAPS stop lateral pass-the-hash across identically configured machines.
  • Event IDs 4624 (LogonType 3) and 4776, plus CA audit logs, are core NTLM and relay telemetry.
