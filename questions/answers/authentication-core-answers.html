Chunk 1 - Questions 1-10

1. Design a secure authentication system for a global enterprise application.

A secure enterprise authentication system should centralize identity while keeping enforcement close to the application. I would start by clarifying whether users are employees, contractors, customers, service accounts, or admins, because each population needs a different assurance level and recovery model.

At a high level, the design has these layers:

Layer Responsibility
Identity provider User lifecycle, MFA, federation, device posture, risk signals
Authentication service Login orchestration, factor enrollment, token issuance, recovery
Application gateway TLS, bot controls, abuse throttling, coarse policy checks
Application services Session validation, authorization, step-up checks, audit events
Security telemetry Detection, investigation, risk scoring, compliance evidence 
flowchart TD
    A[User or workload] --> B[Global edge and bot controls]
    B --> C[Identity provider]
    C --> D[Risk engine]
    D --> E{Assurance sufficient?}
    E -- No --> F[Step up authentication]
    E -- Yes --> G[Issue tokens and session]
    F --> G
    G --> H[Application services]
    H --> I[Authorization checks]
    H --> J[Audit and detection pipeline]

For employees, I would normally federate to a corporate IdP, require phishing-resistant MFA for admins and sensitive roles, support device posture checks for managed devices, and use risk-based step-up for suspicious events. For customers, I would support passkeys, strong password storage, bot protection, adaptive rate limits, account recovery controls, and fraud-aware monitoring. For workloads, I would avoid static credentials and use short-lived workload identity such as AWS IAM roles, AWS STS, OIDC federation, SPIFFE-style identities, or service mesh certificates.

Critical controls:

  • Passwords stored with Argon2id, bcrypt, or scrypt, never plain SHA-256.
  • MFA policy based on account risk: passkeys or hardware keys for admins, TOTP or number-matching push as transitional controls for lower-risk users, SMS only as a low-assurance fallback when unavoidable.
  • Session tokens generated with cryptographic randomness, stored in secure cookies for browsers, rotated after login and privilege changes, and revoked after compromise.
  • Recovery treated as a privileged workflow with cooldowns, notifications, strong proofing, and post-recovery cleanup.
  • Logs designed for investigation: success and failure events, factor enrollment, recovery, session creation, token revocation, impossible travel, rate limiting, and admin changes.

Tradeoffs matter. Very strict MFA and short sessions improve security but can hurt availability for global users, mobile users, and incident responders. A senior design balances this by using risk tiers, backup authenticators, break-glass procedures, and strong observability rather than one universal rule.

AWS example: for a workforce app, use IAM Identity Center or an external IdP with SAML or OIDC, enforce hardware-backed MFA for administrators, use AWS WAF and CloudFront for edge abuse controls, centralize logs in a separate security account, and alert on unusual identity and session events through Security Hub, GuardDuty, CloudTrail, and SIEM correlation.

2. Why is FIDO2/WebAuthn more phishing-resistant than OTP-based MFA?

OTP-based MFA asks the user to type or approve something that can be captured or relayed. FIDO2/WebAuthn uses public-key cryptography and origin binding, so the authenticator proves possession of a private key without revealing a reusable secret.

With OTP phishing, the attack can be real time:

1. Victim enters username and password into a fake page.
2. Attacker forwards them to the real service.
3. Real service asks for OTP.
4. Fake page asks victim for OTP.
5. Attacker forwards OTP before it expires.
6. Login succeeds.

With WebAuthn, the relying party sends a challenge. The browser and authenticator include the relying party ID and origin in the ceremony. The authenticator signs only for the legitimate relying party. A passkey registered for login.company.com will not create a valid signature for login-company.example.net.

sequenceDiagram
    participant U as User
    participant B as Browser
    participant RP as RelyingParty
    participant AU as Authenticator
    U->>B: Start login
    B->>RP: Request challenge
    RP-->>B: Challenge and relying party ID
    B->>AU: Ask for signature for origin
    AU-->>B: Signed challenge
    B->>RP: Credential ID and signature
    RP->>RP: Verify signature and origin
    RP-->>U: Login success

The key distinction is that OTP proves the user knows a short-lived code, while WebAuthn proves possession of a private key bound to the real origin. A phishing page can ask for an OTP, but it cannot make the authenticator produce a valid signature for a different relying party.

Residual risks still exist. If account recovery falls back to SMS, helpdesk reset, or weak email recovery, the attacker may bypass WebAuthn entirely. Synced passkeys also introduce trust in the platform account and cloud keychain recovery process. For privileged enterprise accounts, I would often require hardware-backed or managed-device-bound authenticators and a hardened recovery workflow.

3. Compare SMS OTP, TOTP, push MFA, hardware security keys, and passkeys.

These methods all satisfy some version of MFA, but they offer very different assurance levels.

Factor Strength Main benefit Main weakness Best fit
SMS OTP Low Familiar and easy to deploy SIM swap, phishing, telecom dependence Low-risk fallback only
TOTP Medium Offline code generation, better than SMS Still phishable and manually typed Transitional enterprise MFA
Push MFA Medium Easy user experience Push fatigue, consent phishing Workforce login with number matching
Hardware key Very high Origin-bound and hardware protected Cost, logistics, recovery Admins and high-risk users
Passkey High Phishing-resistant and user friendly Recovery and sync trust assumptions Broad passwordless rollout

SMS OTP is the weakest because the phone number is not a strong security boundary. Attackers can phish the code, perform SIM swap, exploit SS7 weaknesses, or compromise the phone. TOTP avoids telecom dependence but still creates a code that the user can type into a phishing site.

Push MFA improves usability, but simple approve or deny prompts are vulnerable to MFA fatigue. Number matching and contextual prompts help, but a real-time phishing proxy can still show the victim the same number and ask them to approve. Push should be treated as better than passwords alone, not as phishing-resistant.

Hardware security keys and passkeys are qualitatively different because they use origin-bound public-key authentication. Hardware keys are often preferred for administrators because private keys are non-exportable and tied to a physical authenticator. Passkeys are easier to scale because platform authenticators and synced credentials improve usability, but the enterprise must understand whether passkeys are device-bound or synced through a consumer cloud account.

Example enterprise policy:

  • Admins: FIDO2 hardware key or managed platform passkey, backup key required.
  • Standard employees: passkeys preferred, number-matching push allowed during migration.
  • Contractors: federated SSO plus conditional access and device or risk checks.
  • Break-glass accounts: hardware keys stored in controlled process, monitored continuously.
  • SMS: disabled for privileged access and allowed only as temporary recovery for low-risk accounts.

4. How would you prevent MFA fatigue attacks?

MFA fatigue happens when an attacker who already has the password repeatedly triggers push notifications until the user approves one. The defense is to reduce prompt spam, improve user intent, and move high-risk populations to phishing-resistant authentication.

flowchart TD
    A[Attacker has password] --> B[Repeated login attempts]
    B --> C[Push prompts sent]
    C --> D{User approves?}
    D -- Yes --> E[Account takeover]
    D -- No --> F[More prompts]
    F --> C
    C --> G[Rate limit and suppress prompts]
    G --> H[Alert user and security team]

Controls I would implement:

  • Number matching so users cannot blindly tap approve.
  • Clear context in the prompt: application, approximate location, device, browser, and time.
  • Push rate limits per user, per source, per device, and per risk level.
  • Suppression after repeated denials or no responses.
  • A visible Report fraud action that creates a high-priority security event.
  • Risk-based blocking for impossible travel, anonymizing proxies, new devices, or breached-password signals.
  • User notification through a trusted channel after suspicious push activity.
  • Phishing-resistant MFA for admins, finance users, production operators, and other high-value groups.

Operationally, I would also track metrics: push prompts per successful login, denial rates, repeated prompts, time to approve, fraud reports, source IP diversity, and accounts with password success followed by MFA failure. These signals help detect credential stuffing where MFA is the only thing preventing compromise.

The residual risk is that number matching is still not equivalent to WebAuthn. A phishing proxy can relay the number displayed by the real login page to the victim. For high-risk accounts, the better answer is to remove push as a valid factor and require FIDO2 or passkeys.

5. What are the common failure modes of push-based MFA?

Push MFA commonly fails because it depends on user judgment during a stressful or confusing moment. It is convenient, but convenience can become the attack surface.

Common failure modes:

Failure mode Description Control
Push fatigue Repeated prompts pressure the user into approving Rate limit, suppress, alert, prefer FIDO2
Blind approval User taps approve without reading context Number matching and richer context
Phishing proxy Attacker relays login and push challenge in real time Origin-bound MFA
Device compromise Malware or malicious app abuses the enrolled device Device health checks and app integrity
Weak enrollment Attacker enrolls their own phone after password compromise Step-up and alerts for factor enrollment
Weak recovery Helpdesk resets MFA with low proof Harden recovery workflow
Prompt ambiguity User cannot tell which app or action triggered the prompt Show app, tenant, location, action, and risk
Notification overload Too many legitimate prompts train poor behavior Reduce prompt frequency and use session policy

A realistic example: an attacker obtains an employee password from a breach and attempts login every few minutes. The employee receives prompts while working, assumes a background corporate service is reauthenticating, and approves. The attacker now has a valid session even though MFA was technically enabled.

For design review, I would ask whether push is used for initial login only or also for high-risk actions. A vague Approve sign-in prompt is weaker than Approve access to Payroll Admin from Chrome on Windows near Seattle. For sensitive actions, push should include transaction details or be replaced by stronger authentication.

6. How would you design authentication for both managed and BYOD devices?

Managed and BYOD devices should not receive the same trust level. The design should separate user assurance from device assurance, then combine both in a conditional access policy.

For managed devices, the enterprise can enforce stronger posture:

  • Device inventory and ownership.
  • Disk encryption and screen lock.
  • Endpoint detection and response.
  • Patch level and OS version.
  • Managed certificates or device-bound keys.
  • Browser and application control.
  • Remote wipe and revocation.

For BYOD, the enterprise usually has less control and must avoid assuming the device is trustworthy. I would allow BYOD for lower-risk workflows through browser-based access, but require stronger user authentication, shorter sessions, limited data download, and step-up for sensitive actions.

Policy example:

Access scenario Authentication design
Managed laptop to normal app SSO plus passkey or device certificate, standard session
Managed laptop to admin app Phishing-resistant MFA, device compliance, step-up, PAM approval
BYOD to normal app SSO plus strong MFA, limited session, no persistent trust by default
BYOD to sensitive data Block, virtualize, or require managed browser and DLP controls
Unknown device Require step-up and restrict access until risk is understood

AWS example: use an enterprise IdP integrated with AWS IAM Identity Center, require phishing-resistant MFA for privileged permission sets, and use Verified Access or a zero trust access layer for internal applications. For BYOD, use conditional access and application-level restrictions rather than granting broad network trust.

The important senior point is that device trust is not binary. A managed device can still be compromised, and a BYOD device can still belong to the legitimate user. Device posture is a risk signal, not a substitute for strong user authentication and authorization.

7. How would you secure account recovery for high-value enterprise accounts?

For high-value accounts, recovery should be treated like a privileged access workflow because it can transfer account ownership. If recovery is weaker than login, attackers will bypass login and attack recovery.

flowchart TD
    A[Recovery request] --> B[Classify account risk]
    B --> C{High value account?}
    C -- No --> D[Standard recovery policy]
    C -- Yes --> E[Strong identity proofing]
    E --> F[Manager and security approval]
    F --> G[Issue temporary access]
    G --> H[Reset authenticators]
    H --> I[Revoke old sessions and tokens]
    I --> J[Notify user and audit]

Controls for admins, executives, finance users, production operators, and security engineers:

  • Pre-enrolled backup FIDO2 key or recovery key stored safely.
  • No SMS-only recovery.
  • Helpdesk scripts that prevent social engineering and urgency bypass.
  • Approval separation: helpdesk cannot unilaterally recover a privileged account.
  • Manager or sponsor approval plus security approval for high-value accounts.
  • Temporary access passes that are short-lived, single-use, and logged.
  • Mandatory re-enrollment of strong factors after recovery.
  • Revocation of existing sessions, refresh tokens, remembered devices, OAuth grants, and app passwords.
  • Cooling-off period before sensitive actions where practical.
  • Out-of-band notification to the prior trusted channels.

Example: if an AWS administrator loses a hardware key, I would not let the helpdesk reset MFA based only on employee ID and a phone call. I would require identity proofing, manager approval, security approval, issuance of a short-lived temporary access method, registration of a new hardware key, and monitoring of all actions for that account.

The tradeoff is availability. Emergency access must exist, but it should be designed before the incident through break-glass accounts, sealed hardware keys, documented approvals, and continuous monitoring.

8. Why is account recovery often weaker than login?

Account recovery is often weaker because organizations optimize it for availability and support cost while attackers optimize for the easiest path into the account. Normal login may require password, passkey, device posture, risk checks, and MFA. Recovery may require only email access, SMS, personal information, or a helpdesk call.

Common reasons recovery becomes weak:

  • Recovery channels were enrolled long ago and are no longer trustworthy.
  • Helpdesk agents are pressured by urgency, hierarchy, or executive impersonation.
  • Knowledge-based questions rely on public or breached information.
  • SMS and email are easier to compromise than the primary authenticator.
  • Backup codes are downloaded, screenshotted, or stored in email.
  • Post-recovery cleanup is incomplete, leaving attacker sessions active.
  • High-value accounts use the same recovery workflow as low-risk users.

Example failure:

Login policy: password plus FIDO2 hardware key
Recovery policy: call helpdesk and verify date of birth
Real assurance: date of birth and social engineering

This is why I describe recovery as the real assurance floor. A strong login ceremony is only as strong as the easiest alternate path that can replace its factors.

The fix is to make recovery risk-based. Normal users can have self-service options with trusted devices, backup codes, and delays. Privileged users need stronger proof, approval separation, limited temporary access, and heavy audit. Recovery should always trigger notifications and cleanup of sessions, tokens, remembered devices, and enrolled factors.

9. What risks remain after a user successfully completes MFA?

MFA reduces the risk of credential theft, but it does not prove that every later action is safe. After MFA, the system still has to protect sessions, authorization, devices, data, and recovery paths.

Remaining risks:

  • Session hijacking through stolen cookies, malware, XSS, browser extensions, or token leakage.
  • Authorization bugs such as IDOR, tenant isolation failure, or privilege escalation.
  • Compromised endpoint where the attacker acts through the legitimate user session.
  • OAuth consent abuse where a malicious app gets persistent access.
  • Recovery or factor enrollment abuse after login.
  • Insider misuse by a legitimate authenticated user.
  • Weak re-authentication for sensitive actions.
  • Long-lived refresh tokens or remembered devices that outlive the original risk decision.
  • Social engineering after login, such as tricking the user into approving a high-risk action.

Example: an administrator logs in with a hardware key. If malware on the admin workstation steals an active session cookie, or if the application allows the admin to create a permanent API key without step-up, MFA did not fully protect the environment.

Controls after MFA:

  • Secure cookies with Secure, HttpOnly, and SameSite attributes.
  • Session rotation after login and privilege changes.
  • Idle and absolute timeouts.
  • Step-up authentication for sensitive actions.
  • Device posture checks and endpoint telemetry.
  • Strong authorization at object, tenant, and action level.
  • Detection for impossible travel, new device use, token replay, and unusual data access.
  • Token revocation and global logout capability.

The senior answer is: MFA is an important authentication control, not a complete account security model.

10. How would you detect credential stuffing at global scale?

Credential stuffing uses leaked username and password pairs from other breaches. At global scale, the attacker usually distributes attempts across many IPs and tries a small number of passwords per account, so simple per-IP lockout is not enough.

Signals I would collect:

Signal Why it helps
Failure rate by account Detect targeted attempts against known users
Failure rate by IP and ASN Detect infrastructure used for stuffing
Password hash comparison pattern Identify reuse of known breached credentials without logging passwords
User agent and device fingerprints Detect automation clusters
Geo velocity Identify impossible or suspicious location changes
Success after failures Identify likely account takeover
MFA challenge spikes Detect password success blocked by MFA
New session attributes Identify suspicious first successful login

Detection strategy:

  • Build baselines by region, tenant, app, account type, ASN, and time of day.
  • Apply adaptive throttling to suspicious sources and usernames.
  • Use bot defenses such as proof-of-work, CAPTCHA only when useful, JavaScript challenges, and device fingerprinting.
  • Screen submitted passwords against breached-password intelligence without storing or logging the raw password.
  • Alert when many accounts see one or two failures from a common infrastructure pattern.
  • Alert on successful login after distributed failures, especially if followed by MFA reset, email change, API key creation, or data export.

AWS example: place authentication endpoints behind CloudFront and AWS WAF, use rate-based rules and bot controls at the edge, stream application login events to Kinesis or a SIEM, and correlate with GuardDuty, CloudTrail, and IdP logs. The app must still do account-aware throttling because edge controls alone may not see distributed low-rate abuse.

The response should be graduated: block obvious automation, step up suspicious logins, require password reset for confirmed reused credentials, revoke suspicious sessions, and notify users without revealing whether an account exists.

Chunk 2 - Questions 11-20

11. How would you detect password spraying?

Password spraying is different from brute force against one account. The attacker tries a small number of common passwords across many accounts to avoid per-account lockouts.

Example pattern:

Password tried: Spring2026!
Targets: thousands of users
Attempts per user: one or two
Source: distributed IPs or cloud hosts
Goal: find accounts with weak common passwords

Detection has to look across the population, not only at one user or one IP.

Useful signals:

Signal Why it matters
Many usernames with one failure each Classic spraying shape
Same password candidate behavior Common-password campaign, if safely detectable
Failures spread across many IPs Distributed attacker infrastructure
Same user agent or automation fingerprint Bot cluster clue
Login attempts against disabled or stale accounts Username list abuse
MFA prompts after password success Password guessed but MFA blocked takeover
Attempts timed just below lockout threshold Evasion behavior

I would aggregate by time window, tenant, source ASN, user population, app, and authentication result. A useful rule is not simply five failures for one user; it is more like one or two failures across hundreds of accounts sharing infrastructure or timing patterns.

Controls and response:

  • Use adaptive throttling by source, account population, and campaign fingerprint.
  • Step up or block risky attempts before password validation when possible.
  • Require password reset for accounts that show password success followed by MFA failure.
  • Disable or protect stale accounts.
  • Alert identity operations when privileged or high-risk groups are targeted.
  • Avoid hard lockouts that let attackers create denial of service.

AWS-style implementation: stream IdP logs and application auth logs into a centralized security lake, aggregate with Athena, OpenSearch, Splunk, or a SIEM, and alert on campaign-level features. Edge tools such as AWS WAF help, but app-level identity telemetry is required because spraying is often low and slow.

12. How would you design risk-based authentication?

Risk-based authentication uses context to decide whether the normal login path is enough or whether the user should be blocked, stepped up, or allowed with limited privileges. The goal is not to make authentication unpredictable; it is to apply friction when risk increases.

flowchart TD
    A[Login attempt] --> B[Collect signals]
    B --> C[Score risk]
    C --> D{Risk level}
    D -- Low --> E[Allow standard login]
    D -- Medium --> F[Require step up]
    D -- High --> G[Block or hold]
    F --> H[Create high assurance session]
    E --> I[Create normal session]
    G --> J[Alert and investigate]

Inputs should include:

  • User history: usual geographies, devices, login hours, successful factors.
  • Device posture: managed or unmanaged, EDR health, OS version, jailbreak or root signals.
  • Network context: IP reputation, ASN, VPN, Tor, cloud hosting provider, impossible travel.
  • Account sensitivity: admin, executive, finance, production access, data access scope.
  • Behavior: login velocity, failed attempts, MFA denials, new factor enrollment, session anomalies.
  • Threat intelligence: breached credentials, active campaigns, known malicious infrastructure.

Outputs should be explicit:

Risk decision Example action
Allow Known device, normal location, strong factor
Step up New device, unusual location, sensitive action
Limit BYOD device gets read-only or no download
Block Known malicious IP or impossible travel with weak factor
Investigate Password success followed by suspicious MFA pattern

The design must be explainable and privacy-aware. Risk engines can create user friction and false positives, especially for travelers, VPN users, remote workers, and people with changing schedules. I would maintain allowlisted high-confidence signals, safe fallback paths, and security-reviewable policy definitions.

13. What signals would you use to step up authentication?

Step-up authentication should trigger when the current session assurance is no longer enough for the requested action or context. The trigger can be contextual, behavioral, or action-based.

Action-based triggers:

  • Change password.
  • Add, remove, or reset MFA.
  • View recovery codes.
  • Add a new trusted device.
  • Create API keys or personal access tokens.
  • Export sensitive data.
  • Change payroll, bank, or billing information.
  • Assume admin role or access production systems.
  • Disable logs, change security policy, or modify IAM roles.

Contextual triggers:

  • New device or browser.
  • Unmanaged or unhealthy device.
  • New country, impossible travel, or risky ASN.
  • Anonymous proxy, Tor, or cloud hosting source.
  • Recently changed recovery channel.
  • Recent credential stuffing or password spraying indicators.
  • Long session age or idle session near timeout.

Behavioral triggers:

  • Sudden download spike.
  • Unusual access to sensitive datasets.
  • Abnormal sequence of admin actions.
  • Repeated denied authorizations.
  • OAuth consent grant to a new application.
  • Privilege escalation attempt.

The step-up factor should match the risk. For a mild anomaly, a passkey prompt may be enough. For a privileged production action, require phishing-resistant MFA, device compliance, just-in-time approval, and a fresh high-assurance session. For confirmed compromise indicators, do not step up; block and investigate.

Example: a user logged in yesterday with TOTP from a known device. Today they try to create an AWS access key from a new country. I would require fresh phishing-resistant MFA, validate device posture, alert on the behavior, and possibly require approval if the account is privileged.

14. How would you protect against session hijacking?

Session hijacking occurs when an attacker obtains a valid session token and reuses it. Since the user has already completed authentication, hijacking can bypass passwords and MFA.

flowchart TD
    A[User completes MFA] --> B[Session token issued]
    B --> C[Token stored in browser]
    C --> D{Token leaks?}
    D -- No --> E[Normal requests]
    D -- Yes --> F[Attacker replays token]
    F --> G{Session controls detect risk?}
    G -- Yes --> H[Revoke and step up]
    G -- No --> I[Account misuse]

Protection has to cover prevention, detection, and response.

Prevention:

  • Use TLS everywhere and HSTS for browser apps.
  • Store browser sessions in cookies with Secure, HttpOnly, and appropriate SameSite settings.
  • Avoid putting tokens in URLs, logs, local storage, crash reports, or referrers.
  • Defend against XSS with output encoding, CSP, safe frameworks, and dependency hygiene.
  • Rotate session identifiers after login, MFA, recovery, and privilege elevation.
  • Use short idle and absolute lifetimes based on risk.
  • Bind sessions carefully to device or risk context where it improves security without breaking users.

Detection:

  • Same session used from two distant geographies.
  • New device or user agent for an existing session.
  • Token replay after password reset or logout.
  • Sensitive action immediately after session attribute change.
  • Session use from known proxy or malware infrastructure.

Response:

  • Revoke the session and related refresh tokens.
  • Step up or force reauthentication for the user.
  • Invalidate remembered devices if compromise is suspected.
  • Preserve evidence: session ID hash, IPs, device data, timestamps, and actions.

Residual risk: if the endpoint is fully compromised, the attacker may act through the legitimate browser rather than replaying the token elsewhere. That requires endpoint detection, transaction confirmation, and authorization controls beyond basic session security.

15. What makes a session token secure?

A secure session token is unpredictable, protected in storage and transport, scoped to the right context, time-limited, rotatable, and revocable.

Core properties:

Property Requirement
Entropy Generated by a cryptographically secure random source
Unpredictability No sequential IDs, timestamps, usernames, or guessable structure
Confidentiality Not exposed in URLs, logs, analytics, or client scripts
Integrity Signed or server-side validated so it cannot be modified
Scope Bound to application, tenant, audience, and privilege level
Lifetime Idle and absolute expiration based on risk
Rotation Reissued after login, MFA, recovery, and privilege change
Revocation Can be invalidated after logout or compromise

For browser applications, I usually prefer a secure, opaque, server-side session ID in a cookie for high-risk apps because revocation and server-side state are straightforward. JWT-style sessions can work, but they are harder to revoke immediately unless paired with short lifetimes, token versioning, introspection, or a revocation store.

Cookie settings matter:

Secure      -> send only over HTTPS
HttpOnly    -> not readable by JavaScript
SameSite    -> reduce CSRF risk
Path        -> constrain where cookie is sent
Domain      -> avoid overly broad subdomain exposure

Example bad token:

session=alice-1700000000-admin

Example better model:

session=256-bit random opaque value
server record contains user, auth time, assurance, expiry, device, risk, and revocation state

The token itself should not be the only security control. The server must validate policy on every meaningful request.

16. How should session tokens be generated, stored, rotated, and revoked?

I would manage session tokens as a lifecycle, not as a one-time login artifact.

stateDiagram-v2
    [*] --> Created
    Created --> Active
    Active --> Rotated
    Rotated --> Active
    Active --> Expired
    Active --> Revoked
    Expired --> [*]
    Revoked --> [*]

Generation:

  • Use a cryptographically secure random number generator.
  • Use at least 128 bits of entropy, commonly 256 bits for opaque tokens.
  • Never include usernames, roles, timestamps, or predictable counters in the token value.
  • Create a fresh token after authentication and key assurance changes.

Storage:

  • Browser apps: store in Secure, HttpOnly, SameSite cookies.
  • Mobile apps: store in platform secure storage such as Keychain or Keystore.
  • APIs: prefer short-lived access tokens and carefully protected refresh tokens.
  • Server side: store only a hashed token value if feasible, plus metadata such as user, tenant, auth time, assurance, expiry, and revocation state.

Rotation:

  • Rotate after login, MFA completion, account recovery, password reset, role elevation, tenant switch, and suspicious risk changes.
  • Rotate refresh tokens on every use and detect reuse as possible theft.
  • Avoid letting a low-assurance session silently become a high-assurance admin session.

Revocation:

  • Revoke on logout, password change, MFA reset, suspected compromise, user disablement, device loss, and admin action.
  • Maintain a central session store or revocation index for distributed systems.
  • Push revocation events to services or require introspection for high-risk operations.

Tradeoff: fully stateless tokens scale well but are difficult to revoke immediately. Stateful sessions or hybrid designs are often better for enterprise security because incident response needs fast invalidation.

17. How would you handle logout in a distributed authentication system?

Logout is hard in distributed systems because many services may have local sessions, access tokens, refresh tokens, device cookies, and cached authorization state. A good design defines what logout means and propagates that decision consistently.

sequenceDiagram
    participant U as User
    participant A as App
    participant I as IdentityProvider
    participant S as SessionStore
    participant V as Services
    U->>A: Click logout
    A->>I: End identity session
    I->>S: Revoke central session
    S-->>V: Publish revocation event
    A-->>U: Clear browser cookies
    V->>S: Reject revoked session

Design choices:

  • Local logout clears the application session only.
  • Global logout clears the IdP session and all relying-party sessions where possible.
  • Security logout after compromise revokes sessions, refresh tokens, remembered devices, and possibly app grants.

Implementation controls:

  • Maintain a server-side session registry per user and device.
  • Revoke refresh tokens and rotate token family identifiers.
  • Publish revocation events through a reliable event bus.
  • Make high-risk services check revocation state before sensitive actions.
  • Clear cookies on the client, but do not rely only on client deletion.
  • Keep access tokens short-lived if immediate revocation is not guaranteed.
  • Provide an admin and user-facing sign out of all devices option.

Important nuance: OIDC and SAML logout across multiple apps is imperfect. Browser redirects, third-party cookie blocking, mobile apps, and cached tokens can leave residual sessions. For sensitive environments, central revocation and short token lifetimes matter more than relying on front-channel logout alone.

18. How would you invalidate all sessions after suspected compromise?

After suspected compromise, the goal is to remove attacker access without destroying evidence. I would treat it as an incident-response action with clear scope.

Steps:

  1. Identify the subject: user, tenant, device, session family, application, or whole population.
  2. Preserve relevant evidence: login events, session IDs hashed, IPs, user agents, device IDs, token issue times, factor events, and actions taken.
  3. Revoke active sessions in the central session store.
  4. Revoke refresh tokens and remembered-device tokens.
  5. Invalidate OAuth grants, API keys, app passwords, and personal access tokens if in scope.
  6. Reset password or require passkey re-enrollment if the credential path is suspect.
  7. Require phishing-resistant reauthentication before issuing new sessions.
  8. Notify the user and security operations.
  9. Monitor for replay attempts against revoked sessions.

For high-risk compromise, also remove trusted devices, rotate recovery codes, review recovery email and phone, and inspect recent privilege changes. If the user is an AWS administrator, I would review CloudTrail events, revoke console sessions where applicable, disable or rotate access keys, inspect recent IAM changes, and require a fresh hardware-backed sign-in.

Key engineering requirement: session invalidation must not depend on every service receiving a best-effort message. Critical services need a way to check authoritative revocation state or honor short-lived tokens. Otherwise, logout becomes eventual and attackers can continue acting during the propagation window.

19. What is the difference between authentication assurance and identity assurance?

Authentication assurance and identity assurance are related but different.

Authentication assurance asks: how confident are we that this session is controlled by the same entity that enrolled the credential?

Identity assurance asks: how confident are we that the account represents the real-world person, employee, customer, contractor, or workload it claims to represent?

Concept Question answered Example evidence
Authentication assurance Is the returning user proving control of enrolled factors? Password, passkey, FIDO2 key, device certificate, MFA
Identity assurance Was the account bound to the correct real identity? HR record, government ID proofing, manager sponsorship, KYC, device issuance

Example: a passkey login can have high authentication assurance because the user proves possession of an origin-bound private key. But if the account was originally created using an unverified email address, identity assurance may be low. Conversely, an employee account created from HR records may have strong identity proofing, but if it allows SMS-only login, authentication assurance may be weak.

In enterprise systems, identity assurance often comes from HR onboarding, contractor sponsorship, background checks, device issuance, and directory lifecycle processes. Authentication assurance comes from the current login and session controls.

Senior framing: do not confuse strong login with known person. For data protection and privileged access, I need both: the account must be bound to the right identity, and the current session must be strongly authenticated.

20. How would you design authentication for privileged administrators?

Privileged administrators need a higher-assurance authentication model because compromise can lead to tenant-wide data access, production outages, logging disablement, or persistence.

Core design:

  • Separate admin accounts from daily productivity accounts.
  • Require phishing-resistant MFA such as FIDO2 hardware keys or managed platform passkeys.
  • Require at least two registered strong authenticators to avoid unsafe recovery pressure.
  • Use privileged access workstations or hardened managed devices for admin access.
  • Enforce device compliance and block unmanaged devices for admin consoles.
  • Use just-in-time privilege elevation rather than standing admin rights.
  • Require approval or ticket binding for sensitive role activation.
  • Rotate sessions and require step-up for critical actions.
  • Monitor and alert on admin login, role activation, policy change, logging changes, and credential creation.
  • Maintain tightly governed break-glass accounts with hardware keys and continuous alerting.

Example AWS admin model:

Control AWS-style implementation
Central identity IAM Identity Center federated to enterprise IdP
Strong MFA FIDO2 key required for admin permission sets
JIT access Time-limited assignment to privileged permission sets
Device trust Admin access only from managed device or secured access path
Audit CloudTrail to separate log archive account with alerts
Break-glass Minimal root or emergency roles, hardware key, monitored use

Recovery must be stricter than normal user recovery. A helpdesk-only reset for a cloud admin is not acceptable. Require manager and security approval, temporary access, and mandatory post-recovery review.

The residual risk is that a strongly authenticated admin can still make a dangerous authorized change. That is why authentication must be paired with least privilege, separation of duties, change control, and detection.

21. How would you design authentication for service accounts?

Service accounts authenticate non-human actors such as applications, CI jobs, batch workers, bots, and automation. The main design goal is to avoid long-lived shared secrets and give each workload a narrowly scoped identity.

flowchart TD
    A[Workload starts] --> B[Platform verifies workload context]
    B --> C[Issue short lived credential]
    C --> D[Workload calls service]
    D --> E[Service validates identity]
    E --> F[Authorization policy decision]
    F --> G[Audit event]

Design principles:

  • One service account per workload, environment, and trust boundary.
  • Short-lived credentials issued by a trusted platform instead of static keys.
  • Least privilege policies scoped to exact resources and actions.
  • Strong provenance: credential issuance should depend on where the workload is running, not only possession of a file.
  • No shared service accounts across teams or applications.
  • No interactive login for service accounts unless explicitly required and controlled.
  • Separate development, staging, and production identities.
  • Central inventory with owner, purpose, permissions, last used time, and rotation posture.
  • Logs for credential issuance, token exchange, API calls, policy changes, and failed access.

AWS example: prefer IAM roles for EC2, ECS task roles, EKS IRSA or Pod Identity, Lambda execution roles, and OIDC federation for CI/CD instead of long-lived IAM access keys. If external workloads need AWS access, use IAM Roles Anywhere, OIDC federation, or STS AssumeRole with external ID and tight conditions. Avoid storing access keys in GitHub, container images, user laptops, or CI variables unless there is no better option.

Common mistakes:

  • One powerful automation account used by many systems.
  • Static keys with no owner and no rotation.
  • Service account credentials checked into code or images.
  • Human users using service accounts to bypass MFA.
  • Policies like *:* because the workload broke once during deployment.

The senior answer is that service account authentication is really workload identity plus governance. The credential should be short-lived, attributable, scoped, and issued only to the expected workload.

22. How would you design authentication for machine-to-machine workloads?

Machine-to-machine authentication should prove both workload identity and workload provenance. A target service should know not just this token is valid, but this token was issued to the expected service running in the expected environment.

Common patterns:

Pattern Best use Risk
mTLS with workload certificates Service mesh, internal APIs Certificate lifecycle complexity
Cloud workload identity Cloud-native services Misconfigured metadata or role policy
OAuth2 client credentials API-to-API access Client secret leakage if static
SPIFFE or SPIRE Multi-platform workload identity Operational maturity required
Static API keys Legacy integrations Long-lived leak and weak attribution

Recommended design:

  • Each workload gets a unique identity.
  • Credentials are short-lived and automatically rotated.
  • Issuance is based on attested runtime context such as cluster, namespace, service account, instance identity, or signed deployment identity.
  • Downstream services validate issuer, audience, expiry, subject, and authorization policy.
  • Use mTLS or signed tokens depending on protocol and platform needs.
  • Log every credential issuance and every privileged service call.
  • Fail closed when identity cannot be verified.

Example flow:

sequenceDiagram
    participant W as Workload
    participant P as Platform
    participant I as Issuer
    participant S as Service
    W->>P: Start with workload metadata
    P->>I: Attest workload context

    S->>S: Validate issuer audience and policy
    S-->>W: Allow or deny

AWS example: a GitHub Actions workflow can use OIDC federation to assume an AWS role without storing an access key. The role trust policy should restrict repository, branch, audience, and subject claims. An EKS workload can use IRSA or Pod Identity so a Kubernetes service account maps to a narrowly scoped IAM role.

The biggest residual risk is over-permission. Strong authentication of a machine identity is not enough if that identity can read all secrets, write all buckets, or assume broad roles.

23. How would you securely store passwords?

Passwords should be stored as salted, slow password hashes, not plaintext and not reversible encrypted values. The verifier only needs to check whether a submitted password matches; it should not need to recover the original password.

sequenceDiagram
    participant U as User
    participant A as App
    participant D as Database
    U->>A: Submit password during registration
    A->>A: Generate unique salt
    A->>A: Apply slow password hash
    A->>D: Store salt hash algorithm and parameters
    U->>A: Submit password during login
    A->>D: Fetch verifier record
    A->>A: Recompute hash with stored parameters
    A-->>U: Allow or deny

Storage requirements:

  • Use Argon2id, bcrypt, or scrypt with current cost parameters. PBKDF2 can be acceptable in some compliance contexts when configured with high iteration count, but it is less ideal against GPU attackers.
  • Generate a unique random salt per password.
  • Store algorithm, salt, cost parameters, and hash together so the system can migrate later.
  • Consider a pepper stored outside the database, such as in KMS or HSM-backed secret storage.
  • Use constant-time comparison to avoid timing leaks.
  • Block known compromised passwords at registration and reset.
  • Never log passwords, password hashes, reset tokens, or raw authentication inputs.
  • Protect the password database with strict access controls and monitoring.

Example record shape:

user_id: 12345
algorithm: argon2id
memory: 65536
iterations: 3
parallelism: 1
salt: random per user salt
hash: derived password verifier
version: 2

If the password verifier database is stolen, the attacker should be forced into expensive offline guessing. The design goal is damage reduction after breach, not pretending the database can never leak.

24. Compare bcrypt, scrypt, Argon2, and PBKDF2.

All four are password hashing or key derivation functions, but they differ in age, memory hardness, deployment support, and resistance to specialized cracking hardware.

Algorithm Strengths Weaknesses Typical recommendation
Argon2id Modern, memory-hard, strong GPU resistance Library and compliance support varies Best default for new systems when available
scrypt Memory-hard, strong against GPUs and ASICs Parameter tuning can be operationally tricky Good option when Argon2id is unavailable
bcrypt Widely supported, battle-tested, intentionally slow Not memory-hard, password length quirks Acceptable and common legacy-safe choice
PBKDF2 Standardized and compliance-friendly CPU-hard only, weaker against GPUs Use only with high iterations when required

Argon2 has variants. Argon2id is generally preferred for password storage because it balances side-channel resistance and GPU cracking resistance. scrypt is also memory-hard, which raises attacker cost by requiring both computation and memory. bcrypt remains acceptable when configured with a strong cost factor, but it has limitations such as a practical password input length limit in many implementations. PBKDF2 is older and widely approved in compliance environments, but because it is not memory-hard, attackers can scale guesses efficiently on GPUs.

Selection process:

  • New product: Argon2id if mature libraries and platform support exist.
  • Existing bcrypt system: keep bcrypt with strong cost and migrate opportunistically if there is a reason.
  • Compliance-bound environment: PBKDF2 may be required, but tune iterations aggressively.
  • High-risk consumer or enterprise app: combine strong hashing with breached-password checks, MFA, and rate limiting.

Important point: no password hashing algorithm saves users who choose breached passwords and face online guessing. Slow hashing primarily protects against offline cracking after a database leak.

25. Why is SHA-256 not enough for password storage?

SHA-256 is designed to be fast. That is useful for integrity checks and many cryptographic constructions, but it is the wrong property for password storage. If attackers steal a password hash database, they can try guesses extremely quickly using GPUs or specialized hardware.

Bad model:

hash = SHA256(password)

Problems:

  • Fast hashing enables massive offline guessing.
  • Unsalted SHA-256 lets attackers use precomputed tables.
  • Users often choose predictable or reused passwords.
  • Two users with the same password get the same hash if no salt is used.
  • A plain hash has no adjustable work factor for future hardware improvements.

Better model:

hash = Argon2id(password, unique_salt, memory_cost, time_cost)

Slow password hashing changes the economics. If one SHA-256 guess is extremely cheap, attackers can test billions of guesses. If each Argon2id guess requires meaningful memory and time, the attacker has to spend far more money and time per guess.

SHA-256 can appear inside a larger approved construction, but by itself it is not a password storage scheme. The right way to frame it is: password hashing needs salts, tunable cost, and ideally memory hardness. SHA-256 gives none of those by default.

26. What is salting and why is it required?

A salt is a unique random value generated for each password and stored alongside the password hash. It does not need to be secret. Its job is to make each stored password verifier unique even when users choose the same password.

Without salts:

Alice password: Password123 -> hash X
Bob password:   Password123 -> hash X

With salts:

Alice password plus salt A -> hash X1
Bob password plus salt B   -> hash X2

Why salts are required:

  • Prevent precomputed rainbow table attacks.
  • Prevent attackers from cracking one hash and instantly identifying all users with the same password.
  • Make every password guess specific to one user's salt.
  • Hide password reuse patterns within the same database.

Salt requirements:

  • Unique per password.
  • Random enough to avoid collisions.
  • Generated by a secure random source.
  • Stored with the hash.
  • Never reused globally as a system-wide secret.

Common mistake: confusing salt with pepper. A salt is public and per-user. A pepper is secret and system-wide or versioned. Salting is mandatory even if peppering is used.

Model answer: salting does not make a weak password strong, and it does not slow down a single password guess by itself. It prevents attackers from amortizing work across many accounts or using precomputed tables. It must be paired with slow hashing.

27. What is peppering and where should the pepper be stored?

A pepper is a secret value added to the password hashing process and stored separately from the password database. Unlike a salt, a pepper is intended to remain secret.

Common model:

stored_hash = Argon2id(password plus pepper, per_user_salt, parameters)

or:

stored_hash = HMAC(pepper, password_hash)

The pepper should be stored outside the password database, ideally in a dedicated secret manager, HSM, or KMS-backed environment with strict access controls. In AWS, this could be AWS Secrets Manager, AWS KMS, or CloudHSM depending on assurance requirements and architecture.

Pepper benefits:

  • If only the database leaks, attackers still lack the pepper.
  • It creates an additional secret boundary separate from database access.
  • It can slow or block offline cracking when application secrets are not compromised.

Pepper risks and tradeoffs:

  • If the pepper is lost, password verification may fail for all users protected by it.
  • Rotation is hard because users' passwords are not known.
  • Application servers need controlled access to the pepper.
  • If the app and database are both compromised, the pepper may also be exposed.

Design recommendation:

  • Use salting and slow hashing first; peppering is an additional defense.
  • Store peppers in a secret manager or HSM, not in source code or the same database.
  • Version peppers so new hashes can use a new pepper while old hashes still verify.
  • Plan rotation through password reset, login-time rehash, or HMAC wrapping strategies.

Pepper is useful, but it should never be used as an excuse to use weak hashing or poor database protection.

28. How would you migrate users from weak password hashes to strong password hashes?

Password hash migration should preserve user experience while gradually replacing weak verifiers. Since the system does not know users' plaintext passwords, migration usually happens when users log in or reset passwords.

flowchart TD
    A[User logs in] --> B[Load stored hash metadata]
    B --> C{Old hash type?}
    C -- Yes --> D[Verify with legacy algorithm]
    D --> E{Password valid?}
    E -- Yes --> F[Rehash with strong algorithm]
    F --> G[Store new verifier]
    G --> H[Create session]
    E -- No --> I[Deny login]
    C -- No --> J[Verify with current algorithm]
    J --> H

Migration plan:

  • Add hash metadata: algorithm, version, salt, cost parameters, and pepper version if used.
  • Implement verification for legacy and new hashes.
  • On successful login with a legacy hash, rehash the submitted password with the new algorithm and parameters.
  • For inactive accounts, force password reset after a defined period.
  • For high-risk accounts, require reset sooner and pair with MFA review.
  • Monitor migration progress and legacy hash population.
  • Keep rollback safe but avoid indefinite support for weak hashes.

Security controls during migration:

  • Do not lower login protections for legacy users.
  • Apply breached-password screening during reset or password change.
  • Increase monitoring for accounts still using weak hashes.
  • Avoid bulk password resets unless risk justifies the support and phishing impact.
  • Communicate carefully to avoid training users to trust phishing-like reset emails.

Example: if an old system uses unsalted SHA-1, add Argon2id support. When Alice successfully logs in, verify SHA-1 once, then immediately replace the stored verifier with Argon2id using a unique salt and current parameters. Alice does not need to know migration happened.

29. How would you prevent username enumeration in login flows?

Username enumeration happens when attackers can determine whether an account exists based on error messages, timing, status codes, recovery behavior, or side channels.

Controls:

  • Return generic login errors such as Invalid username or password.
  • Keep response timing similar for existing and non-existing users.
  • Use the same HTTP status pattern for authentication failures.
  • Do not reveal whether MFA is enrolled until after primary credential verification.
  • For password reset, say If an account exists, we sent instructions.
  • Avoid different rate-limit behavior that reveals valid accounts.
  • Avoid account-specific images, names, or prompts before authentication unless risk is accepted.
  • Monitor enumeration attempts across login, signup, password reset, and MFA recovery.

Common side channels:

Side channel Example leak
Error text No such user versus Wrong password
Timing Existing user triggers slow password hash, missing user returns fast
MFA prompt Valid username proceeds to MFA page
Reset flow Only valid users receive different UI
Signup Email already registered reveals account
Rate limit Valid accounts get different lockout behavior

Timing mitigation can use fake password hashing for unknown users or a consistent verification path. Be careful not to create excessive CPU load for attackers; combine timing consistency with rate limiting and bot controls.

Tradeoff: some consumer products intentionally reveal account existence during signup for usability. In enterprise and high-risk applications, I would prioritize privacy and anti-abuse by using generic flows and behind-the-scenes notifications.

30. What error messages should a login system return?

Login errors should help legitimate users recover without giving attackers account validity, credential validity, MFA enrollment, or policy details.

Good default:

We could not sign you in with those credentials.

For password reset:

If an account exists for that address, we sent recovery instructions.

For locked or throttled attempts:

We cannot complete this sign-in right now. Try again later or use account recovery.

Avoid messages like:

No account found for this email.
Password is wrong.
MFA not configured for this user.
Your account is disabled because of suspicious login from 1.2.3.4.
This admin account requires hardware key.

Internally, the system should log precise reason codes for security and support:

External message Internal reason code
Could not sign in Unknown user
Could not sign in Wrong password
Could not sign in Disabled account
Could not sign in MFA failed
Cannot complete now Rate limited
Cannot complete now High-risk source blocked

The key is separation between user-facing ambiguity and internal observability. Security teams need detailed logs; attackers should not receive an oracle.

There are exceptions. After a user is already strongly authenticated, it can be appropriate to show more specific guidance. For example, an enrolled user inside a recovery workflow may need to know that a hardware key is required. But pre-authentication errors should remain generic.

Chunk 4 - Questions 31-40

31. How would you rate-limit login attempts without locking out legitimate users?

Login rate limiting should slow attackers without giving them an easy denial-of-service tool against real users. A naive lockout such as five failures locks the account can be abused by attackers to lock out executives or whole departments.

Better design uses layered, adaptive throttling:

flowchart TD
    A[Login attempt] --> B[Check account risk]
    B --> C[Check source risk]
    C --> D[Check campaign signals]
    D --> E{Risk decision}
    E -- Low --> F[Allow normal attempt]
    E -- Medium --> G[Add delay or challenge]
    E -- High --> H[Temporarily block source]
    E -- Account risk --> I[Require step up after password]

Rate-limit dimensions:

  • Per account: prevent online brute force.
  • Per IP and subnet: block noisy sources.
  • Per ASN or cloud provider: detect distributed infrastructure.
  • Per device fingerprint: catch automation behind rotating IPs.
  • Per tenant or organization: detect campaigns against one enterprise.
  • Per password-spray pattern: detect many accounts with low attempts each.
  • Per credential pair reputation: identify breached-credential stuffing.

User-friendly controls:

  • Progressive delays instead of hard lockout.
  • Step-up after suspicious password success.
  • Allow account owner recovery through strong proof.
  • Notify users of suspicious attempts without confirming account existence to attackers.
  • Avoid requiring password reset for every failed attempt campaign.

Example policy:

Situation Response
One user mistypes password three times Small delay, no lockout
Hundreds of users receive one failure each Campaign throttling, source blocking
Password success from risky IP Require phishing-resistant MFA or block
Repeated failures for admin account Alert security and require step-up

The senior design point is that rate limiting belongs both at the edge and inside the identity layer. AWS WAF can slow source traffic, but the application must understand accounts, tenants, and authentication outcomes.

32. How would you design bot protection for authentication endpoints?

Authentication endpoints attract credential stuffing, password spraying, fake account creation, OTP abuse, MFA bombing, and enumeration. Bot protection should combine edge controls, application telemetry, and adaptive challenges.

Controls:

  • Edge rate limiting by IP, subnet, ASN, geography, and reputation.
  • Bot fingerprinting using device, browser, TLS, and behavior signals.
  • Credential stuffing detection using leaked credential intelligence.
  • Progressive challenges such as CAPTCHA only when risk justifies it.
  • Proof-of-work or JavaScript challenges for suspicious automation.
  • Account-aware throttling after password failures or MFA prompt abuse.
  • Separate protections for login, signup, password reset, MFA enrollment, and recovery.
  • Honey accounts or canary credentials to detect active stuffing tools.
  • Strong observability for success after suspicious failures.
flowchart TD
    A[Auth request] --> B[Edge reputation checks]
    B --> C[Bot fingerprinting]
    C --> D[Account and tenant risk]
    D --> E{Action}
    E -- Allow --> F[Continue auth]
    E -- Challenge --> G[Require adaptive challenge]
    E -- Throttle --> H[Delay request]
    E -- Block --> I[Reject and log]

CAPTCHA should not be the only defense. Modern attackers use CAPTCHA-solving services, residential proxies, and real browsers. Also, aggressive CAPTCHA can hurt accessibility and legitimate users. I would use it as one possible friction control, not the primary security boundary.

AWS example: put login endpoints behind CloudFront and AWS WAF Bot Control, use rate-based rules, label suspicious traffic, and pass signals to the application. Then use app-level detections for account, tenant, and credential behavior that the edge cannot see.

A mature design also protects downstream cost. Attackers can use auth endpoints to trigger SMS OTP costs, email floods, or push fatigue. Add quotas and anomaly detection for factor delivery, not just password attempts.

33. How would you detect impossible travel logins?

Impossible travel detection identifies logins where the same user appears from locations that cannot be physically connected within the time between events.

Basic calculation:

distance between locations / time between logins = required speed
if required speed is unrealistic, raise risk

Useful data:

  • Timestamp in UTC.
  • Source IP and geolocation.
  • User ID and tenant.
  • Device ID and browser.
  • Authentication factor strength.
  • Network type such as corporate VPN, mobile carrier, proxy, or cloud provider.
  • Whether the session was already active or newly authenticated.

Detection example:

Event Detail
09:00 UTC Alice logs in from New York with managed laptop
09:20 UTC Alice logs in from Singapore with unknown device
Decision Impossible travel, require step-up or block

False positives are common:

  • VPN egress changes location.
  • Mobile carrier geolocation is inaccurate.
  • Corporate proxy centralizes traffic.
  • Cloud-hosted browsers or remote desktops are used.
  • IP geolocation databases are stale.
  • User legitimately uses a travel VPN or privacy tool.

Because of false positives, impossible travel is often a risk signal rather than a standalone proof of compromise. I would combine it with device novelty, MFA strength, IP reputation, recent failures, user behavior, and action sensitivity.

Response:

  • Low confidence: step up authentication and log.
  • Medium confidence: require phishing-resistant MFA and notify user.
  • High confidence: block session, revoke tokens, and investigate.
  • Privileged accounts: alert immediately and require security review.

For AWS admins, an impossible-travel console login followed by role assumption or IAM policy change should be treated as high severity.

34. What would you log for authentication events?

Authentication logs should support detection, incident response, forensics, abuse analysis, and compliance. They should be detailed enough to reconstruct decisions without storing secrets.

flowchart TD
    A[Authentication event] --> B[Normalize fields]
    B --> C[Security log stream]
    C --> D[Detection rules]
    C --> E[Investigation store]
    C --> F[Compliance evidence]

Core fields:

  • Event type: login success, login failure, logout, MFA challenge, MFA failure, factor enrollment, recovery, password reset, token refresh, session revocation.
  • User and tenant identifiers, preferably stable internal IDs.
  • Timestamp in UTC with reliable clock source.
  • Source IP, ASN, geolocation, and network reputation.
  • User agent, device ID, platform, browser, and app version.
  • Authentication method and assurance level.
  • MFA factor type, challenge result, and failure reason category.
  • Session ID hash or token family ID, not raw token.
  • Risk score, policy decision, and reason codes.
  • Application, client ID, redirect URI, audience, and scopes where relevant.
  • Admin or support actor for manual recovery or overrides.
  • Correlation ID for tracing across services.

Important security events:

  • Password success followed by MFA failure.
  • MFA reset or new factor enrollment.
  • Recovery channel change.
  • New device trust granted.
  • Login from unusual location.
  • High-risk action after login.
  • Admin role activation.
  • Token refresh reuse.
  • Failed attempts against many accounts.

AWS-style logging: send authentication events to a central log account or security lake, protect with immutability controls where available, restrict access, and correlate with CloudTrail, VPC Flow Logs, WAF logs, endpoint telemetry, and SaaS IdP logs.

35. What should not be logged during authentication?

Authentication systems must never turn logs into a credential leak. Logs are widely accessed by support, developers, SIEM pipelines, and incident responders, so they need minimization.

Do not log:

  • Plaintext passwords.
  • Password hashes or salts unless there is an exceptional secured debugging workflow.
  • OTP codes, backup codes, recovery codes, or magic links.
  • Session tokens, refresh tokens, access tokens, ID tokens, authorization codes, or cookies.
  • Full security answers or identity proofing documents.
  • Private keys, client secrets, API keys, or service account keys.
  • Full PII when a stable internal identifier is enough.
  • Complete device fingerprints if they create unnecessary privacy risk.
  • Raw biometric data.
  • Full credit card, government ID, or sensitive HR data.

Safer alternatives:

Sensitive value Safer log value
Session token Token hash or session ID hash
Email address User ID or partially masked email
IP address Full IP for security logs, aggregated or masked for analytics
OAuth token Token family ID or grant ID
Password failure reason Internal category without raw password

Debug logging is a common failure mode. A developer temporarily logs request bodies to troubleshoot login and accidentally captures passwords or tokens. Authentication endpoints should have explicit redaction middleware and tests that fail if sensitive fields enter logs.

I would also separate security logs from product analytics. Security logs may need precise IPs and device signals; analytics often does not. Access to both should be governed and audited.

36. How would you protect authentication logs from tampering?

Authentication logs are only useful if investigators can trust them. Attackers often try to delete or alter logs after account compromise.

Controls:

  • Write logs to an append-only or immutable destination.
  • Send logs out of the application account or production account quickly.
  • Use a separate security account or tenant controlled by the security team.
  • Restrict who can modify log pipelines, retention, and encryption keys.
  • Alert on logging disablement, pipeline failure, retention changes, and access policy changes.
  • Use integrity checks, sequence numbers, signatures, or hash chaining for high-assurance logs.
  • Replicate logs across regions or storage classes where required.
  • Monitor gaps in expected event volume.
  • Keep administrative access to logs separate from application administrators.

AWS example:

  • Send CloudTrail and application auth logs to a dedicated log archive account.
  • Use S3 Object Lock for immutability where appropriate.
  • Encrypt with KMS keys controlled by the security account.
  • Use CloudWatch or EventBridge alerts for StopLogging, trail deletion, bucket policy changes, and KMS key changes.
  • Restrict write paths so applications can write logs but not delete or rewrite them.
flowchart TD
    A[Auth service] --> B[Log collector]
    B --> C[Security account]
    C --> D[Immutable storage]
    C --> E[SIEM detection]
    F[Application admin] --> G{Can delete logs?}
    G -- No --> D

Tamper resistance also includes operational monitoring. If the login system suddenly produces zero failures during peak traffic, that may indicate a broken pipeline or attacker interference.

37. How would you design secure authentication for mobile apps?

Mobile authentication must account for untrusted networks, device loss, malware, rooted devices, app tampering, and token storage. The mobile app should not be treated as a confidential client if it cannot protect secrets from the user or device.

Design:

  • Use OAuth2 Authorization Code Flow with PKCE for user sign-in.
  • Prefer system browser or trusted web authentication session over embedded web views.
  • Use passkeys or platform authenticators where supported.
  • Store tokens in Keychain or Android Keystore, not plain preferences or files.
  • Use short-lived access tokens and refresh token rotation.
  • Bind refresh tokens to device instance when feasible.
  • Detect jailbreak, root, tampering, emulator, and debugging signals as risk signals.
  • Require biometric or device PIN to unlock local app secrets for sensitive apps.
  • Use certificate pinning carefully only when operationally justified.
  • Support remote logout and token revocation for lost devices.
  • Avoid logging tokens or PII in crash reports.

Mobile-specific threats:

Threat Control
Lost phone Local device lock, secure storage, remote revocation
Malware App integrity checks, least token scope, anomaly detection
Phishing System browser, passkeys, origin-bound auth
Token theft Secure storage, rotation, binding, short lifetimes
Reverse engineering Do not embed long-lived secrets in app

Important point: client secrets in mobile apps are not truly secret because the app binary can be inspected. Public clients should use PKCE and server-side validation rather than relying on embedded secrets.

38. How would you design secure authentication for browser-based apps?

Browser-based authentication must handle XSS, CSRF, token storage, phishing, third-party scripts, and session theft. I would design differently for server-rendered apps and single-page applications, but the principles are similar.

Controls:

  • Use Authorization Code Flow with PKCE for OIDC login.
  • Prefer backend-for-frontend patterns for high-risk apps so tokens stay server-side.
  • Store session identifiers in Secure, HttpOnly, SameSite cookies.
  • Avoid storing long-lived tokens in local storage.
  • Rotate session after login, MFA, and privilege changes.
  • Use CSRF protections for cookie-authenticated state-changing requests.
  • Use strong CSP and XSS prevention because XSS can act as the user even with HttpOnly cookies.
  • Set strict redirect URI validation for OIDC flows.
  • Validate issuer, audience, nonce, state, expiry, and signature for tokens.
  • Require step-up authentication for sensitive browser actions.

Browser token storage tradeoff:

Storage Benefit Risk
HttpOnly cookie Not readable by JavaScript Needs CSRF protection
Memory only Reduced persistence Lost on refresh, XSS can still use live app
Local storage Easy for SPAs Readable by XSS, risky for long-lived tokens

For enterprise apps, I prefer server-managed sessions and a backend-for-frontend when possible. It reduces token exposure in the browser and gives the server better control over revocation, step-up, and authorization.

Example: after OIDC login, the backend exchanges the authorization code, stores tokens server-side, and sends the browser only an opaque session cookie. The SPA calls the backend, not cloud APIs directly with long-lived bearer tokens.

39. How would you handle remembered devices securely?

Remembered devices reduce MFA friction by recognizing a device after successful authentication. They are useful but dangerous if they become a silent MFA bypass forever.

Secure design:

  • Issue a high-entropy remembered-device token only after strong MFA.
  • Store it in a secure, HttpOnly cookie for browsers or secure storage for mobile.
  • Bind it to user, tenant, device, browser, and risk context.
  • Store only a hashed version server-side.
  • Give it an expiration and rotate it periodically.
  • Revoke it on password reset, MFA reset, recovery, suspicious login, device loss, or user request.
  • Show remembered devices in account settings.
  • Notify the user when a new device is remembered.
  • Do not allow remembered device status to authorize sensitive actions by itself.

Risk-based policy:

Situation Remembered device behavior
Managed compliant laptop Longer trust window may be acceptable
BYOD browser Shorter trust window and more step-up
Admin account Avoid or heavily restrict remembered device bypass
New location or risky source Ignore remembered status and require step-up
After recovery Clear all remembered devices

Common mistake: implementing remembered devices as a permanent cookie that bypasses MFA for months with no server-side revocation. That cookie becomes a long-lived bearer credential.

For high-risk accounts, I would either disable remembered devices or treat them only as one signal. A remembered device can reduce prompts, but it should not override strong risk signals or action-based step-up.

40. What are the security risks of 'keep me signed in'?

Keep me signed in extends session lifetime. It improves usability but increases the value and duration of stolen sessions, refresh tokens, and remembered device cookies.

Risks:

  • Stolen device keeps access for a long time.
  • Malware or browser extensions steal persistent tokens.
  • Shared or public computers remain authenticated.
  • User forgets active sessions across devices.
  • Refresh token theft gives durable access.
  • Password reset may not revoke persistent sessions if poorly designed.
  • Insider or family member uses an unlocked remembered browser.
  • Long sessions weaken the practical value of MFA freshness.

Controls:

  • Do not offer persistent sessions on shared, unmanaged, or high-risk devices.
  • Make persistent sessions shorter for privileged accounts.
  • Use refresh token rotation and reuse detection.
  • Require step-up for sensitive actions regardless of persistent login.
  • Let users view and revoke active sessions.
  • Revoke persistent sessions after password change, MFA change, recovery, or compromise.
  • Use device-bound tokens where feasible.
  • Show clear UI when enabling persistent sign-in.
  • Use idle and absolute timeouts, not infinite sessions.

Example policy:

Account type Keep signed in policy
Consumer low risk Allowed with device management and revocation
Enterprise standard user Allowed on managed or low-risk devices
Contractor Shorter lifetime and more frequent step-up
Administrator Disabled or very short with strong step-up

The senior answer is that persistent sign-in is not automatically bad, but it converts authentication from a momentary proof into a longer-lived bearer credential. That requires token rotation, revocation, device awareness, and sensitive-action reauthentication.

Chunk 5 - Questions 41-50

41. What are the security risks of biometric authentication?

Biometrics are useful for local user verification, but they are not magic identity proof. In most modern authentication systems, biometrics unlock a local private key or device credential; the biometric itself should not be sent to the server.

Security risks:

  • Biometrics are hard or impossible to rotate if compromised.
  • False accepts and false rejects exist, even if rates are low.
  • Presentation attacks can target weak sensors with photos, masks, recordings, or lifted fingerprints.
  • Device compromise can bypass local biometric checks.
  • Legal and privacy risks are higher because biometric data is sensitive personal data.
  • Coercion risk differs from passwords because a user's face or finger can be used under pressure.
  • Biometric enrollment may be weak if the device or identity proofing process is weak.
  • Users may over-trust biometrics and ignore session or recovery risks.

Good design:

  • Keep biometric templates local in secure hardware where possible.
  • Use biometrics to unlock a device-bound private key, not as a server-side shared secret.
  • Provide fallback to device PIN or strong recovery without weakening assurance too much.
  • Require stronger controls for high-risk actions and privileged users.
  • Treat biometric success as local user verification, not proof of real-world identity.
  • Avoid storing raw biometric data in application servers.

Example: Face ID unlocking a passkey is strong because the server receives a WebAuthn signature, not a face scan. The biometric gates use of a private key stored on the device. If the application instead uploads face images and compares them server-side, it creates a privacy and breach risk.

production-ready phrase: biometrics are best used as a local unlock factor for cryptographic credentials. They are convenient and can improve assurance, but they require careful privacy, recovery, and device security design.

42. How do passkeys change the enterprise authentication model?

Passkeys move enterprise authentication away from shared secrets and toward user-friendly public-key authentication. The server stores a public key, while the user's authenticator holds the private key and signs origin-bound challenges.

flowchart TD
    A[Password era] --> B[Shared secret stored as verifier]
    B --> C[Phishing and reuse risk]
    A --> D[Passkey era]
    D --> E[Public key registered]
    E --> F[Origin bound challenge signing]
    F --> G[Reduced phishing and stuffing risk]

What changes:

  • Credential stuffing becomes much less useful because there is no reusable password.
  • Phishing resistance improves because credentials are bound to the relying party origin.
  • User experience improves because sign-in can use platform biometrics or PIN.
  • Password reset volume may decrease.
  • Recovery and device lifecycle become central design problems.
  • Enterprises must decide whether synced passkeys are acceptable for each risk tier.
  • Attestation and authenticator policy matter more for privileged populations.

Enterprise policy example:

Population Passkey policy
Standard employees Platform passkeys allowed, managed devices preferred
Privileged admins Hardware-backed or managed device-bound passkeys required
Contractors Federated SSO plus conditional access, passkeys where supported
Break-glass accounts Hardware keys with controlled storage and monitoring
BYOD users Passkeys allowed but paired with risk-based access limits

Passkeys do not eliminate the need for session security, authorization, logging, and recovery. If recovery falls back to SMS or a weak helpdesk call, the enterprise still has a weak account-takeover path. If malware steals an active session after passkey login, post-auth controls still matter.

AWS-style implication: workforce users can authenticate to the corporate IdP with passkeys, then federate into AWS IAM Identity Center. The passkey improves the upstream authentication assurance, while AWS permission sets, session duration, SCPs, and CloudTrail still control and monitor what the user can do.

43. What is the role of device binding in authentication?

Device binding ties authentication or session use to a specific device or device-held key. It helps distinguish the legitimate user's enrolled environment from an attacker who only has a password or token.

Forms of device binding:

  • Device-bound passkeys or hardware security keys.
  • Client certificates or mTLS.
  • Mobile app instance keys stored in Keychain or Keystore.
  • Managed device certificates from MDM.
  • Remembered-device tokens with server-side state.
  • Token binding or proof-of-possession tokens where supported.

Benefits:

  • Reduces value of stolen passwords.
  • Can reduce value of stolen tokens if proof of possession is required.
  • Supports conditional access based on managed device posture.
  • Helps detect new-device risk.
  • Improves assurance for privileged workflows.

Risks and tradeoffs:

  • Device loss creates recovery pressure.
  • BYOD support becomes harder.
  • Re-imaging, browser changes, and hardware replacement can break access.
  • Overly strict binding can lock out legitimate users.
  • Binding to weak identifiers such as user agent or IP can create false confidence.
  • A compromised device can still act as the user.

Good design treats device binding as one layer. For high-risk access, I would combine device-bound credentials, user verification, device compliance, strong session management, and authorization. For lower-risk access, device binding may be a remembered-device signal rather than a hard requirement.

Example: an admin portal accepts only sessions created from a managed laptop with a device certificate and FIDO2 key. If the same password is used from an unmanaged laptop, the system blocks or requires a recovery-approved path.

44. How would you prevent phishing proxy attacks?

Phishing proxy attacks relay the victim's interaction to the real site in real time. They can capture passwords, OTPs, and sometimes session cookies. Traditional MFA like TOTP can fail because the attacker uses the code immediately.

sequenceDiagram
    participant U as User
    participant P as Proxy
    participant R as RealSite
    U->>P: Enter password and OTP
    P->>R: Relay credentials
    R-->>P: Issue session
    P-->>U: Show normal page
    P->>P: Capture session cookie

Best controls:

  • Use phishing-resistant authentication: FIDO2, WebAuthn, passkeys, hardware keys, smart cards, or device certificates.
  • Enforce strict origin binding through WebAuthn.
  • Reduce session cookie theft with secure cookies, short lifetimes, and anomaly detection.
  • Require step-up for high-risk actions after login.
  • Detect suspicious reverse proxy infrastructure and impossible travel.
  • Use transaction binding for sensitive actions where the user verifies specific action details.
  • Educate users, but do not rely on education as the main control.
  • Harden OAuth redirect URI validation and use state and nonce correctly.

Number matching push is better than blind push, but it does not fully prevent phishing proxies. The proxy can display the real number to the victim and ask them to enter it. TOTP has the same problem because the code is portable.

WebAuthn changes the attack because the credential is scoped to the legitimate relying party. If the victim is on evil-login.example, the authenticator will not produce a valid signature for login.company.com.

Residual risk: a phishing proxy may still steal a session cookie after successful authentication if the user signs into the real site through the proxy. Strong session protections and anomaly detection still matter, but the strongest mitigation is to prevent the proxy from completing authentication in the first place.

45. How would you protect high-risk actions after login?

High-risk actions need their own protection because initial login may be old, low-assurance, or performed before risk increased. I would classify actions by impact and require fresh assurance for sensitive operations.

High-risk actions:

  • Change password or recovery channel.
  • Add or remove MFA factors.
  • View recovery codes.
  • Create API keys or OAuth apps.
  • Export sensitive data.
  • Change payroll, payment, or billing details.
  • Assume admin roles.
  • Access production secrets.
  • Disable logging or security controls.
  • Change IAM policy or tenant configuration.
flowchart TD
    A[Authenticated session] --> B[Request high risk action]
    B --> C[Check session assurance]
    C --> D{Fresh and strong?}
    D -- Yes --> E[Authorize action]
    D -- No --> F[Require step up]
    F --> G{Step up passed?}
    G -- Yes --> E
    G -- No --> H[Deny and log]

Controls:

  • Fresh reauthentication within a short window.
  • Phishing-resistant factor for privileged actions.
  • Transaction details in the prompt, not generic approval.
  • Risk checks at action time, not only login time.
  • Authorization checks for object, tenant, and role.
  • Approval workflows for irreversible or high-blast-radius changes.
  • Delays or cooling-off periods for recovery and payout changes.
  • Notifications to existing trusted channels.
  • Detailed audit events and alerts for sensitive changes.

Example: creating an AWS access key or changing a KMS key policy should require recent strong authentication, authorization to perform the action, logging, and alerting. For especially sensitive roles, it may also require just-in-time approval.

46. How would you design re-authentication for sensitive actions?

Re-authentication should prove fresh user intent and adequate assurance for the specific action. It should not be a generic nuisance prompt that users blindly approve.

Design elements:

  • Define action risk tiers.
  • Track authentication time, factor type, assurance level, and session risk.
  • Require re-auth when assurance is stale or insufficient.
  • Use phishing-resistant factors for high-risk actions.
  • Display action-specific context to the user.
  • Rotate or upgrade the session after successful step-up.
  • Log the re-auth event and bind it to the action.
  • Deny or escalate if risk is too high rather than endlessly prompting.

Freshness examples:

Action Re-auth requirement
View normal dashboard Existing session ok
Change password Fresh MFA within minutes
Add new MFA factor Fresh phishing-resistant auth plus notification
Create production admin token Fresh phishing-resistant auth plus approval
Export regulated data Fresh auth, authorization check, audit event

Transaction binding improves quality. Instead of asking Approve action, show Approve adding security key ending 1234 or Approve export of customer dataset. This reduces confused approval and creates better audit evidence.

Re-authentication should also handle failure safely. Multiple failed step-up attempts should raise risk, suppress repeated prompts, and potentially revoke the session. For suspected compromise, the right action is containment, not another MFA prompt.

47. How would you handle authentication for contractors and temporary workers?

Contractors and temporary workers need strong authentication plus tighter lifecycle controls because their access is time-bound, sponsor-dependent, and often more variable than employee access.

Design:

  • Use a named identity for each contractor, not shared accounts.
  • Require sponsor or manager ownership.
  • Set start and end dates at provisioning time.
  • Federate from a trusted partner IdP when appropriate, but enforce local risk policy.
  • Require MFA, ideally phishing-resistant for sensitive access.
  • Restrict access by role, project, tenant, device posture, and time window.
  • Use just-in-time access for privileged or sensitive resources.
  • Shorter session lifetimes than employees where appropriate.
  • Block or restrict unmanaged BYOD for sensitive data.
  • Review access more frequently.
  • Disable access automatically at contract end.

Contractor-specific risks:

Risk Control
Forgotten access after contract ends Automatic expiry and joiner mover leaver process
Shared vendor accounts Named accounts and accountability
Weak partner MFA Enforce local MFA or conditional access
BYOD data exposure Managed access path, DLP, download restrictions
Sponsor drift Periodic recertification by sponsor

AWS example: contractor access to cloud accounts should use federated identities with time-bound permission sets, not long-lived IAM users. Require sponsor approval, short session duration, CloudTrail monitoring, and automatic removal when the HR or vendor record expires.

The key point is lifecycle. Contractor authentication is not just login strength; it is identity proofing, sponsorship, access expiry, device policy, and deprovisioning.

48. How would you deprovision authentication access when employees leave?

Deprovisioning should be event-driven from the authoritative HR or identity source and should revoke both login ability and existing authenticated state. Disabling future login is not enough if sessions, tokens, and service credentials remain active.

flowchart TD
    A[Termination event] --> B[Disable primary identity]
    B --> C[Revoke sessions and refresh tokens]
    C --> D[Remove MFA and remembered devices]
    D --> E[Revoke app grants and API keys]
    E --> F[Remove group and role memberships]
    F --> G[Trigger SaaS and cloud deprovisioning]
    G --> H[Audit completion]

Steps:

  • Receive termination or transfer event from HRIS.
  • Disable IdP account at the effective time.
  • Revoke active sessions and refresh tokens.
  • Remove remembered devices and device trust.
  • Revoke personal access tokens, API keys, app passwords, SSH keys, and OAuth grants.
  • Remove group memberships and privileged roles.
  • Transfer ownership of shared resources.
  • Disable or rotate credentials the user could access, especially for admins.
  • Trigger SCIM or automated SaaS deprovisioning.
  • Verify completion with reconciliation reports.

Special cases:

  • Involuntary termination may require immediate access revocation before notification.
  • Privileged users need review of recent activity and possible secret rotation.
  • Contractors should expire automatically without relying on manual cleanup.
  • Service accounts owned by the employee must be transferred, not blindly deleted.

AWS example: remove the user from IAM Identity Center groups, revoke active SSO sessions where supported, review CloudTrail for recent activity, remove access keys if any exist, rotate shared secrets the user could access, and verify permission set assignments are gone across accounts.

Metrics: time to disable, time to revoke sessions, orphaned accounts found, failed downstream deprovisioning tasks, and privileged access remaining after termination.

I would test the system across design, implementation, abuse cases, and operations. Authentication reviews fail when they only test the happy-path login screen.

Review areas:

  • Architecture and trust boundaries.
  • Credential storage.
  • MFA enrollment and enforcement.
  • Recovery and reset flows.
  • Session lifecycle.
  • Token validation and revocation.
  • Rate limiting and bot protection.
  • Username enumeration.
  • Logging and monitoring.
  • Admin and support workflows.
  • Mobile and browser storage.
  • Deprovisioning and lifecycle.

50. What are the top authentication mistakes you would look for in a design review?

Top mistakes:

  1. Treating MFA as a checkbox while allowing SMS, TOTP, or push for high-risk admin access.
  2. Strong login but weak account recovery.
  3. Passwords stored with fast hashes, reused salts, missing metadata, or reversible encryption.
  4. No protection against credential stuffing, spraying, or username enumeration.
  5. Session tokens stored in unsafe places or never rotated after login and privilege changes.
  6. Logout that clears only the browser but does not revoke server-side sessions or refresh tokens.
  7. Long-lived bearer tokens with no revocation strategy.
  8. Weak factor enrollment controls that let attackers add their own MFA.
  9. No step-up authentication for sensitive actions.
  10. Overly broad service accounts and static machine credentials.
  11. Authentication logs that either miss critical events or contain secrets.
  12. Admin accounts using the same controls as normal users.
  13. BYOD and managed devices treated as equivalent.
  14. Contractor access with no expiry or sponsor accountability.
  15. Deprovisioning that disables login but leaves sessions, tokens, API keys, and SaaS access active.

Senior design-review framing:

  • What is the strongest login path?
  • What is the weakest recovery path?
  • What happens after login if the session is stolen?
  • Can we revoke access quickly during an incident?
  • Are admins and service accounts treated as high blast radius identities?
  • Do logs prove what happened without leaking credentials?
  • Does the system fail safely under abuse?

The most important mistake is usually not one missing control. It is inconsistency: strong authentication in one path and weak bypasses in recovery, sessions, service accounts, or support operations. Attackers follow the weakest path, so the design has to cover the full identity lifecycle.
