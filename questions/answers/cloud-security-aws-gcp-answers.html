How To Use These Answers

Use each answer as a scaffold, not a script. A strong senior cloud security answer should reason from identity, workload credentials, metadata services, control-plane auditability, data-plane exposure, blast radius, and AWS-to-GCP mental model mapping.

Chunk 1 - Questions 1-10

1. An EC2 instance is compromised. What can the attacker do next?

After compromising an EC2 instance, the attacker usually tries to understand identity, network reachability, local secrets, and data access.

Likely next steps:

  • Query the Instance Metadata Service for instance profile credentials.
  • Enumerate AWS permissions with sts:GetCallerIdentity and allowed API calls.
  • Access S3 buckets, DynamoDB tables, queues, secrets, or KMS keys permitted to the role.
  • Inspect local files, environment variables, application config, container metadata, and logs.
  • Scan reachable VPC services such as databases, Redis, Elasticsearch, or internal APIs.
  • Pivot to other instances using SSH keys, tokens, or service credentials found locally.
  • Use the instance for outbound attacks, crypto mining, spam, or command and control.
  • Disable or tamper with local agents if the instance role permits it or host access allows it.
flowchart TD
	A[Compromised EC2] --> B[Local secrets]
	A --> C[Instance profile]
	A --> D[VPC reachability]
	C --> E[AWS APIs]
	D --> F[Private services]
	B --> G[Databases and SaaS]

Senior answer: the blast radius is primarily the union of the instance role, network path, local secrets, and attached data volumes.

2. A GCE VM is compromised. What can the attacker do next?

After compromising a GCE VM, the attacker follows the same pattern but with GCP primitives.

Likely next steps:

  • Query the GCE metadata server for service account tokens.
  • Identify the attached service account and OAuth scopes.
  • Call Google Cloud APIs permitted by IAM and scopes.
  • Access Cloud Storage, Secret Manager, BigQuery, Pub/Sub, GKE, or Compute APIs if allowed.
  • Read local application credentials or mounted volumes.
  • Scan VPC-reachable internal services.
  • Abuse workload identity or Kubernetes credentials if the VM participates in GKE workflows.
  • Persist through startup scripts, service account keys, or project-level access if available.

Important nuance:

  • GCP authorization for VM API access depends on both IAM permissions and legacy access scopes for many metadata-issued tokens.
  • Modern best practice is narrow service accounts and least-privilege IAM, with scopes set conservatively or cloud-platform only when IAM is tight.

Senior answer: in GCP, the attached service account is the main crown-jewel path. A default compute service account with broad project permissions can turn one VM compromise into project compromise.

3. How do you analyze cloud workload blast radius?

I analyze blast radius by asking what the compromised workload can read, write, assume, reach, decrypt, modify, and hide.

Dimensions:

  • Identity: instance profile, service account, workload identity, pod role, API keys.
  • Permissions: effective IAM, resource policies, bucket policies, key policies, organization policies.
  • Network: inbound paths, outbound egress, private service access, peering, shared VPCs.
  • Data: storage buckets, databases, queues, logs, snapshots, volumes, backups.
  • Secrets: environment variables, local config, secrets manager access, CI tokens.
  • Crypto: KMS decrypt, key admin, signing permissions.
  • Control plane: ability to create users, change IAM, disable logging, create keys.
  • Detection resistance: ability to stop agents, alter logs, or evade egress monitoring.
flowchart TD
	A[Workload] --> B[Identity permissions]
	A --> C[Network reachability]
	A --> D[Local secrets]
	A --> E[Data stores]
	B --> F[Effective blast radius]
	C --> F
	D --> F
	E --> F

Senior answer: blast radius is not just IAM. A read-only role with network access to an unauthenticated internal database can still be catastrophic.

4. How would you contain a compromised cloud instance?

Containment should preserve evidence while cutting off attacker control and preventing further cloud API abuse.

Immediate actions:

  • Snapshot disk and memory if tooling supports it.
  • Preserve instance metadata, tags, role, security groups, VPC flow logs, and cloud API logs.
  • Isolate network access using a quarantine security group or firewall policy.
  • Revoke or remove the attached instance profile or service account access.
  • Rotate secrets that were present on the host or reachable by the role.
  • Block suspicious egress destinations at network edge.
  • Disable compromised access keys or tokens discovered on disk.
  • Avoid terminating the instance until evidence is captured unless active harm demands it.

AWS examples:

  • Replace security group with quarantine group.
  • Detach or restrict IAM role.
  • Snapshot EBS volumes.
  • Review CloudTrail for role activity.

GCP examples:

  • Apply restrictive VPC firewall rules or network tags.
  • Remove service account permissions or detach service account where feasible.
  • Snapshot persistent disks.
  • Review Cloud Audit Logs for service account activity.

Senior answer: containment is a race between attacker persistence and evidence preservation. Isolate first, then scope credentials and data access.

5. What is the role of instance profiles or service accounts in cloud compromise?

Instance profiles in AWS and service accounts in GCP provide workload identity. They are intended to remove hard-coded credentials, but if a workload is compromised, the attacker can often use the same identity.

Why they matter:

  • They issue temporary credentials or tokens through metadata services.
  • They define cloud API permissions available to the workload.
  • They often grant access to storage, secrets, queues, logs, and KMS.
  • They can become privilege-escalation paths if over-permissioned.

Comparison:

Concept AWS GCP
Workload identity IAM role via instance profile Attached service account
Token source IMDS Metadata server
Permission model IAM policies, resource policies, SCPs, boundaries IAM allow policies, deny policies, org policies, scopes
Common mistake Broad role on many instances Default service account with broad project roles

Senior answer: workload identity is safer than static keys, but it makes least privilege mandatory. A compromised workload inherits the role's effective access.

6. How would you secure cloud metadata services?

Metadata services should be protected because they expose workload identity tokens and sensitive instance details.

Controls:

  • Require AWS IMDSv2 for EC2.
  • Set low metadata hop limit, usually 1, unless container networking requires more.
  • Block metadata access from containers that do not need it.
  • Use firewall, iptables, eBPF, or sidecar controls where appropriate.
  • Use narrow instance profiles or service accounts.
  • Avoid placing secrets in user data, startup scripts, or custom metadata.
  • Disable legacy metadata endpoints where available.
  • Monitor metadata credential use from unusual paths.
  • Prevent SSRF from reaching metadata IPs.

Important addresses:

  • AWS EC2 metadata: 169.254.169.254.
  • GCP metadata: metadata.google.internal and 169.254.169.254 with required metadata headers.

Senior answer: metadata hardening is necessary, but it is not enough. The attached identity must still be least privilege.

7. Compare AWS IMDSv1, IMDSv2, and GCP metadata protections.

Feature AWS IMDSv1 AWS IMDSv2 GCP metadata server
Request style Simple HTTP GET Session token required Header required
SSRF resistance Weak Stronger Stronger than no-header systems
Token or header None X-aws-ec2-metadata-token Metadata-Flavor: Google
Hop control Limited Supports hop limit Network and platform controls
Best practice Disable Require Use least privilege and metadata concealment where applicable

IMDSv2 improves security because the caller first obtains a session token with an HTTP PUT, then includes the token in metadata requests. Many SSRF primitives cannot send arbitrary methods or headers, so the metadata attack is harder.

GCP requires a metadata flavor header, which blocks simple SSRF cases that cannot add headers. However, if the SSRF primitive allows arbitrary headers, or if the attacker already has code execution, metadata access remains possible.

Senior answer: metadata protections reduce SSRF exploitability. They do not protect against full host compromise, so role design is still decisive.

8. How would you prevent SSRF from reaching metadata services?

SSRF-to-metadata prevention should happen at multiple layers.

Application controls:

  • Use strict URL allowlists for outbound fetch features.
  • Resolve DNS and validate final IP before connecting.
  • Block link-local, loopback, private, multicast, and reserved ranges.
  • Re-check after redirects and DNS rebinding windows.
  • Use dedicated egress proxy with policy enforcement.
  • Do not let users control headers or methods for generic fetchers.

Infrastructure controls:

  • Require IMDSv2 on AWS.
  • Set metadata hop limit to 1 where possible.
  • Deny metadata access from containers that do not need it.
  • Use firewall rules to block metadata IP from app processes without need.
  • Attach least-privilege roles.

Detection:

  • Alert on unusual metadata access patterns.
  • Alert on cloud API calls from roles that do not normally use them.
  • Watch for SSRF probes in URL parameters.

Senior answer: metadata SSRF defense is strongest when the application cannot fetch arbitrary hosts and the infrastructure makes metadata tokens hard to retrieve.

9. Explain AWS IAM policy evaluation.

AWS IAM evaluation combines identity policies, resource policies, permissions boundaries, session policies, service control policies, and explicit denies.

High-level logic:

flowchart TD
	A[Request] --> B[Authenticate principal]
	B --> C[Collect applicable policies]
	C --> D{Explicit deny?}
	D -- Yes --> E[Deny]
	D -- No --> F{Allowed by policy?}
	F -- No --> E
	F -- Yes --> G{Limited by boundary or SCP?}
	G -- Yes --> E
	G -- No --> H[Allow]

Important rules:

  • Default is deny.
  • Explicit deny overrides allow.
  • Identity policies grant permissions to principals.
  • Resource policies grant access to resources and can allow cross-account access.
  • Permissions boundaries limit maximum permissions for a principal.
  • SCPs limit maximum permissions in an AWS Organization account.
  • Session policies can further reduce assumed-role permissions.
  • Conditions constrain when a statement applies.

Senior answer: AWS IAM is a set-intersection and override system. You need to evaluate both grants and ceilings, and explicit deny always wins.

10. Explain GCP IAM policy evaluation at a high level.

GCP IAM evaluation determines whether a principal has a permission on a resource, usually through roles bound at the organization, folder, project, or resource level.

High-level model:

  • Principals include users, groups, service accounts, workload identities, and domains.
  • Roles contain permissions.
  • IAM allow policies bind principals to roles on resources.
  • Inheritance flows from organization to folder to project to resource.
  • Deny policies can block permissions even when allow policies exist.
  • Conditions can limit bindings by attributes such as time, resource, or request context.
  • Organization policies constrain configuration, not general API permissions.
  • For some VM access patterns, legacy OAuth scopes can further limit API calls from metadata-issued tokens.
flowchart TD
	A[Principal request] --> B[Resource hierarchy]
	B --> C[Inherited allow bindings]
	C --> D[Deny policies]
	D --> E[Conditions and constraints]
	E --> F{Permission effective?}
	F -- Yes --> G[Allow]
	F -- No --> H[Deny]

Senior answer: GCP IAM is resource-hierarchy centric. A broad project-level binding can grant access to many services, so project boundaries and service account design matter deeply.

Chunk 2 - Questions 11-20

11. Compare AWS IAM roles and GCP service accounts.

AWS IAM roles and GCP service accounts both provide identities for workloads and humans to assume or use, but the mental models differ.

Area AWS IAM role GCP service account
Primary use Assumable identity Resource identity and principal
Workload binding EC2 instance profile, ECS task role, Lambda execution role, IRSA Attached to VM, Cloud Run, Cloud Functions, GKE workload identity
Assumption STS AssumeRole based on trust policy Service account impersonation or attached identity
Permissions Identity policies and resource policies IAM roles bound to service account or resource hierarchy
Cross-boundary access Cross-account role assumption Service account impersonation, project IAM bindings

Common mistakes:

  • AWS: overly broad trust policy or role permissions.
  • GCP: granting broad project roles to default service accounts.
  • Both: using workload identity as a shared administrative identity.

Senior answer: in AWS, study who can assume the role and what the role can do. In GCP, study who can impersonate the service account and what that service account can access.

12. What are common cloud IAM privilege escalation paths?

Privilege escalation paths often involve creating, modifying, passing, or impersonating identities.

AWS examples:

  • iam:PassRole to a service that can run code.
  • sts:AssumeRole into a more privileged role.
  • Creating access keys for an existing user.
  • Attaching policies to a user, group, or role.
  • Updating a role trust policy.
  • Launching EC2 or Lambda with a privileged role.
  • Modifying resource policies on S3, KMS, SQS, SNS, or secrets.

GCP examples:

  • iam.serviceAccounts.actAs to run workloads as a privileged service account.
  • iam.serviceAccounts.getAccessToken to impersonate service accounts.
  • Creating service account keys.
  • Granting IAM roles on projects or resources.
  • Deploying Cloud Functions, Cloud Run, GCE, or GKE workloads with privileged identities.
  • Modifying Cloud KMS or storage IAM.

Senior answer: privilege escalation usually hides in meta-permissions: permissions to change who can act, what identity code runs as, or what policies apply.

13. How would you detect cloud privilege escalation?

I would detect privilege escalation by monitoring identity mutation, role assumption, key creation, and unusual deployment activity.

AWS signals:

  • AttachUserPolicy, AttachRolePolicy, PutRolePolicy.
  • UpdateAssumeRolePolicy.
  • CreateAccessKey.
  • CreatePolicyVersion and SetDefaultPolicyVersion.
  • PassRole followed by EC2, Lambda, ECS, or Glue creation.
  • AssumeRole into sensitive roles from unusual principals.

GCP signals:

  • SetIamPolicy on projects, folders, service accounts, buckets, or keys.
  • GenerateAccessToken or service account impersonation.
  • Service account key creation.
  • Deployment of compute resources using sensitive service accounts.
  • IAM changes involving Owner, Editor, Service Account Token Creator, or Service Account User.

Detection design:

  • Alert on sensitive permission grants.
  • Correlate IAM change then data access.
  • Baseline normal role assumption paths.
  • Route high-risk IAM changes to review.
  • Keep cloud audit logs in a separate security account or project.

Senior answer: do not only alert on final admin actions. Alert on the stepping stones that make admin actions possible.

14. How would you design least privilege for cloud workloads?

Least privilege starts from the workload's required actions and narrows identity, data, network, and time.

Process:

  • Identify workload purpose and data flows.
  • Create one identity per workload or bounded service.
  • Grant only required actions on specific resources.
  • Use conditions for source, time, resource tags, or service account constraints where practical.
  • Separate read, write, admin, and break-glass paths.
  • Avoid shared roles across unrelated applications.
  • Use short-lived credentials through metadata or workload federation.
  • Review unused permissions with access analyzer, IAM recommender, or policy simulation.
  • Add tests for denied sensitive actions.

Example:

Bad Better
Workload has s3:* on all buckets Workload has s3:GetObject on one prefix
VM uses project Editor service account VM uses custom service account with exact API roles
CI has permanent admin key CI uses OIDC federation with deploy-only role

Senior answer: least privilege is a lifecycle. Start narrow, observe actual access, remove unused permissions, and prevent privilege creep during incidents and launches.

15. How would you handle cross-account or cross-project access?

Cross-boundary access should be explicit, scoped, logged, and owned by both sides.

AWS design:

  • Use role assumption into the target account.
  • Trust specific source principals, not entire accounts unless necessary.
  • Use external ID for third-party access.
  • Restrict with conditions such as organization ID, source ARN, or MFA where appropriate.
  • Use separate roles for read, deploy, incident response, and break-glass.
  • Log AssumeRole in both source and target accounts.

GCP design:

  • Use service account impersonation or workload identity federation.
  • Bind principals to roles at the narrowest resource level.
  • Avoid broad project-level Owner or Editor bindings.
  • Use conditions and organization policies.
  • Log service account impersonation and IAM changes.

Senior answer: cross-account and cross-project access is a trust contract. The target environment must know exactly who can enter, why, and with what ceiling.

16. What is a cloud trust policy and how can it be abused?

In AWS, a role trust policy defines who can assume the role and under what conditions. The equivalent concept in GCP is who can impersonate or attach a service account.

Abuse cases:

  • Trusting an entire external account instead of a specific principal.
  • Missing external ID for third-party access.
  • Allowing broad federated identities through weak claims.
  • Trusting any workload in a Kubernetes cluster or CI provider.
  • Allowing service account impersonation to broad groups.
  • Missing conditions on source project, repository, branch, environment, or audience.

Example risk:

Role trust allows account 111122223333.
Any principal in that account with sts:AssumeRole can enter if no narrower condition exists.

Senior answer: a trust policy is the front door to an identity. A narrow permission policy does not help if a powerful role is assumable by the wrong principals.

17. What is confused deputy in cloud IAM?

Confused deputy occurs when a trusted service or third party is tricked into using its privileges on behalf of the wrong customer or resource.

Cloud example:

  • A SaaS vendor has access to customer AWS accounts.
  • Customer A can trigger the vendor integration.
  • Without an external ID or tenant binding, Customer A may trick the vendor into accessing Customer B's role.

AWS mitigations:

  • External ID in role trust policy for third-party access.
  • aws:SourceArn and aws:SourceAccount for service integrations.
  • Tight resource policies.

GCP mitigations:

  • Bind workload identity federation to exact issuer, audience, subject, and attribute conditions.
  • Use tenant-specific service accounts or resource bindings.
  • Validate project and resource ownership before acting.

Senior answer: confused deputy is authorization failure across delegation. The caller, tenant, and target resource must be bound together.

18. How would you prevent confused deputy in cloud integrations?

Prevention requires binding the delegated action to the correct tenant and resource.

Controls:

  • Use tenant-specific external IDs or equivalent unguessable binding values.
  • Restrict trust to exact principals and claims.
  • Validate SourceArn, SourceAccount, audience, subject, repository, branch, and environment claims.
  • Store integration ownership metadata and check it before every action.
  • Separate roles or service accounts per tenant for high-risk integrations.
  • Use least-privilege permissions on delegated roles.
  • Log tenant ID, integration ID, assumed role, target resource, and request ID.
  • Alert when one tenant integration attempts to access another tenant's resources.
flowchart TD
	A[Tenant request] --> B[Lookup integration owner]
	B --> C[Validate trust binding]
	C --> D[Assume delegated identity]
	D --> E[Act only on owned resources]

Senior answer: do not let possession of a role ARN or service account name be enough. The integration must prove tenant ownership.

19. How would you secure CI/CD access to cloud resources?

CI/CD should use short-lived, scoped, auditable access tied to repository, branch, workflow, and environment.

Controls:

  • Prefer OIDC workload identity federation over long-lived static keys.
  • Bind cloud roles to exact repository, branch, tag, workflow, and environment claims.
  • Separate build, test, deploy, and release identities.
  • Require approvals for production deploy roles.
  • Prevent pull requests from untrusted forks from receiving privileged tokens.
  • Protect secrets and restrict who can modify pipelines.
  • Sign artifacts and verify provenance.
  • Use least-privilege deploy roles.
  • Log role assumption and deployment actions.
  • Rotate or revoke legacy keys.

AWS examples:

  • GitHub Actions OIDC to assume a deploy role with conditions on repository and branch.

GCP examples:

  • Workload Identity Federation pool with attribute conditions and service account impersonation.

Senior answer: the CI system is a production identity provider. Treat pipeline configuration changes like privileged code changes.

20. How would you prevent long-lived cloud credentials?

Preventing long-lived credentials means making temporary identity the default and static keys exceptional.

Controls:

  • Use instance profiles, service accounts, and workload identity federation.
  • Disable creation of user access keys where possible.
  • Use organization policies or SCPs to restrict key creation.
  • Detect and alert on new access key or service account key creation.
  • Store remaining secrets in a secrets manager with rotation.
  • Set expiration and owner metadata for exceptions.
  • Scan code, container images, tickets, and logs for leaked keys.
  • Rotate existing static credentials into short-lived patterns.
  • Use break-glass credentials with strong controls and monitoring.

Migration pattern:

  1. Inventory static keys.
  2. Map key usage to workload owner.
  3. Replace with role, service account, or federation.
  4. Monitor for successful temporary access.
  5. Disable and delete static key.

Senior answer: long-lived keys are breach multipliers. The best key rotation strategy is to remove the key class from normal workflows.

Chunk 3 - Questions 21-30

21. What are workload identity federation risks?

Workload identity federation lets an external identity provider exchange assertions for cloud credentials. It removes static keys, but bad trust conditions can grant cloud access to the wrong workload.

Risks:

  • Trusting a broad issuer without tight subject or audience checks.
  • Allowing any repository, branch, or workflow from a CI provider.
  • Accepting tokens from pull requests or untrusted forks.
  • Missing environment approval for production roles.
  • Reusing one federated role across unrelated workloads.
  • Weak mapping between external claims and cloud permissions.
  • No logging of which external identity assumed the cloud role.
  • Token replay within validity windows.

Controls:

  • Validate issuer, audience, subject, repository, branch, workflow, and environment.
  • Use separate roles or service accounts by environment.
  • Require production approvals.
  • Keep token lifetimes short.
  • Log external claims with cloud audit events.
  • Review trust policies like code.

Senior answer: federation removes stored keys, but shifts risk into trust-policy correctness.

22. How would you design cloud secrets management?

Cloud secrets management should centralize storage, access control, rotation, audit, and delivery.

Design:

  • Store secrets in AWS Secrets Manager, SSM Parameter Store with KMS, GCP Secret Manager, or an approved equivalent.
  • Encrypt secrets with managed or customer-managed keys based on sensitivity.
  • Grant read access to workload identities, not humans by default.
  • Avoid secrets in environment variables for highly sensitive workloads where process inspection is a concern.
  • Use runtime retrieval with caching and short TTLs.
  • Separate secrets by environment and tenant where needed.
  • Enable versioning and rotation.
  • Log reads, writes, deletes, and permission changes.
  • Scan code and images for accidental secret commits.
flowchart TD
	A[Workload identity] --> B[Secrets manager]
	B --> C[KMS protected secret]
	B --> D[Audit log]
	A --> E[Runtime cache]

Senior answer: secrets management is not just encrypted storage. It is identity-scoped retrieval, rotation, audit, and leak response.

23. How would you rotate cloud secrets without downtime?

Zero-downtime rotation requires supporting old and new secrets during a controlled transition.

Pattern:

  1. Create new secret version.
  2. Update dependent service or credential provider to accept both old and new.
  3. Deploy workloads that read the new version.
  4. Verify traffic succeeds with the new secret.
  5. Revoke old secret.
  6. Monitor for clients still using the old secret.

Examples:

  • Database password rotation with dual users or alternating credentials.
  • API key rotation with overlapping active keys.
  • TLS certificate rotation with staged trust bundles.
  • OAuth client secret rotation with both secrets accepted temporarily.

Controls:

  • Automate rotation where possible.
  • Use version labels such as current and previous.
  • Track owners and dependencies.
  • Alert on use of deprecated versions.
  • Test rollback path.

Senior answer: rotation fails when systems assume one secret exists globally. Design clients and servers to tolerate overlap.

24. How would you design KMS usage for sensitive data?

I would use KMS for key protection, access control, auditability, and envelope encryption.

Design:

  • Use envelope encryption for large data.
  • Keep data encryption keys close to the data but encrypted by KMS keys.
  • Use customer-managed keys for high-sensitivity data requiring control, rotation, or audit.
  • Separate keys by environment, data class, tenant, or blast-radius boundary where justified.
  • Restrict decrypt permissions to specific workload identities.
  • Use encryption context or additional authenticated data to bind ciphertext to purpose.
  • Log decrypt and key administration events.
  • Use key rotation for cryptoperiod management.
flowchart TD
	A[Application] --> B[Generate data key]
	B --> C[KMS key]
	B --> D[Plaintext data key]
	D --> E[Encrypt data]
	E --> F[Store ciphertext and encrypted key]

Senior answer: KMS protects keys and gives policy and audit leverage. It does not replace data access control.

25. What are common KMS key policy mistakes?

Common mistakes:

  • Granting decrypt to broad roles or entire projects.
  • Giving application roles key administration permissions.
  • Letting root, owner, or project editor identities administer sensitive keys without guardrails.
  • Using one key for unrelated data classes or environments.
  • Missing separation between encrypt-only and decrypt permissions.
  • No conditions on encryption context, resource, or caller.
  • Allowing key deletion without approval and waiting period controls.
  • Not logging or alerting on key policy changes.
  • Assuming encryption prevents users with decrypt permission from reading data.

AWS-specific examples:

  • KMS key policy allows * principals through broad account delegation.
  • Secrets Manager secret policy and KMS policy are not reviewed together.

GCP-specific examples:

  • Broad Cloud KMS CryptoKey Decrypter role at project level.
  • Key admin and decrypter assigned to the same operational group.

Senior answer: the most dangerous KMS mistake is treating key admin, decrypt, and data admin as the same role.

26. How would you handle KMS key compromise?

First clarify what compromise means: unauthorized decrypt use, key policy takeover, key material exposure, or suspected misuse by a privileged principal. Managed KMS usually prevents raw key extraction, so many incidents are access-control compromise rather than key-material compromise.

Steps:

  • Preserve KMS audit logs and related cloud API logs.
  • Identify affected key, data class, principals, and time window.
  • Disable or restrict key use if it will not cause unacceptable outage.
  • Remove unauthorized grants, bindings, and key policy entries.
  • Rotate key or create replacement key.
  • Re-encrypt data when key material or decrypt access is no longer trusted.
  • Rotate dependent secrets protected by the key.
  • Review all decrypt events during exposure window.
  • Notify data owners and legal/privacy teams if sensitive data may have been exposed.

Senior answer: KMS incidents are often data-access incidents. The key event tells you who could decrypt; the impact analysis must determine what data they actually reached.

27. When would you use customer-managed keys?

Customer-managed keys are useful when the organization needs stronger control than provider-managed default encryption.

Use cases:

  • Regulated data requiring customer-controlled rotation or access review.
  • Separation of duties between data administrators and key administrators.
  • Per-tenant or per-environment key boundaries.
  • Audit requirements for decrypt operations.
  • Need to disable access quickly by disabling a key.
  • Cross-account or cross-project encrypted data access control.
  • Bring-your-own-key or external key manager requirements.

Tradeoffs:

  • More operational complexity.
  • Risk of outages from bad key policy or disabled keys.
  • Need for key lifecycle management.
  • More review burden for grants and bindings.

Senior answer: use customer-managed keys when key policy and audit are part of the security requirement, not just because encryption sounds stronger.

28. When would you use application-level encryption in cloud?

Application-level encryption is appropriate when cloud storage or database-level encryption does not provide sufficient separation from platform, database administrators, or broad service permissions.

Use cases:

  • Highly sensitive fields such as credentials, tokens, SSNs, or health data.
  • Multi-tenant systems requiring tenant-level cryptographic isolation.
  • Data stored in shared analytics platforms.
  • Protection from accidental broad database reads.
  • Client-side encryption requirements.
  • Need for field-level access control and selective disclosure.

Tradeoffs:

  • Harder search, indexing, and analytics.
  • More key management complexity.
  • Application bugs can still leak plaintext.
  • Rotation and migration require careful design.

Senior answer: application-level encryption is for reducing plaintext exposure to infrastructure and database layers. It should be targeted to specific sensitive fields, not blindly applied everywhere.

29. How would you design tamper-resistant cloud audit logging?

Tamper-resistant logging requires centralization, separation of duties, immutable storage, and monitoring for log pipeline changes.

Design:

  • Enable organization-level CloudTrail or GCP Audit Logs coverage.
  • Send logs to a dedicated security account or project.
  • Restrict write, read, and admin permissions separately.
  • Use immutable storage controls such as S3 Object Lock where appropriate.
  • Enable bucket versioning and retention policies.
  • Encrypt logs with keys administered by security.
  • Monitor changes to logging configuration, sinks, trails, buckets, and KMS keys.
  • Forward logs to SIEM or detection platform.
  • Test that new accounts, projects, regions, and services are covered.
flowchart TD
	A[Cloud accounts] --> B[Audit log service]
	B --> C[Security log account]
	C --> D[Immutable storage]
	C --> E[SIEM]
	F[Security admin] --> C

Senior answer: audit logs are only trustworthy if attackers who compromise a workload or app account cannot edit or delete the evidence.

30. Why should audit logs live in a separate account or project?

Audit logs should live in a separate account or project to prevent the same compromise from both causing harm and deleting evidence.

Benefits:

  • Separation of duties between workload admins and security log admins.
  • Reduced risk that compromised app roles can tamper with logs.
  • Centralized retention and legal hold.
  • Easier organization-wide detection.
  • Independent KMS keys and storage policies.
  • Better incident response when production accounts are unstable.

Failure mode avoided:

Attacker gains admin in app account.
Attacker disables CloudTrail and deletes local log bucket.
Central copy in security account remains intact.

Senior answer: logs stored inside the compromised environment are useful, but not sufficient. The authoritative copy should be outside the blast radius.

Chunk 4 - Questions 31-40

31. How would you detect attempts to disable logging?

I would monitor control-plane events that modify trails, sinks, buckets, retention, encryption, and permissions.

AWS signals:

  • StopLogging.
  • DeleteTrail.
  • UpdateTrail.
  • PutEventSelectors reducing coverage.
  • S3 log bucket policy changes.
  • S3 Object Lock or retention changes.
  • KMS key disable or policy changes for log encryption keys.
  • GuardDuty disable or detector changes.

GCP signals:

  • Logging sink deletion or modification.
  • Audit log exemption or exclusion changes.
  • Project, folder, or organization IAM changes affecting logging.
  • Bucket retention policy changes.
  • CMEK key disable or IAM changes for log buckets.
  • Security Command Center service disablement.

Detection design:

  • Alert in near real time.
  • Route alerts outside the affected account or project.
  • Require change tickets or approvals for logging configuration changes.
  • Correlate logging changes with recent privilege escalation.

Senior answer: logging disablement should be treated as high-severity attacker behavior until proven to be planned maintenance.

32. What CloudTrail events are most important for IAM abuse?

Important CloudTrail events for IAM abuse include identity creation, policy changes, role assumption, access key creation, and permission boundaries being weakened.

High-value events:

  • CreateUser, CreateRole, CreatePolicy.
  • AttachUserPolicy, AttachRolePolicy, PutUserPolicy, PutRolePolicy.
  • CreatePolicyVersion, SetDefaultPolicyVersion.
  • UpdateAssumeRolePolicy.
  • CreateAccessKey.
  • DeleteAccessKey following suspicious activity.
  • AssumeRole, especially into admin or break-glass roles.
  • PassRole followed by service creation.
  • PutRolePermissionsBoundary and DeleteRolePermissionsBoundary.
  • AddUserToGroup for privileged groups.

Context to log and alert on:

  • Source principal.
  • Target identity.
  • New permissions.
  • Source IP and user agent.
  • MFA presence.
  • Role session name.
  • Account and organization context.

Senior answer: focus on events that create new power, change trust, mint credentials, or allow code to run as a privileged role.

33. What GCP audit logs are most important for IAM abuse?

GCP IAM abuse detection relies heavily on Admin Activity audit logs and service account events.

High-value events:

  • SetIamPolicy on organization, folder, project, service account, bucket, key, or dataset.
  • Service account key creation.
  • Service account impersonation and access token generation.
  • Role creation or update for custom roles.
  • Project ownership or billing-related privilege changes.
  • Cloud Functions, Cloud Run, GCE, or GKE deployment using privileged service accounts.
  • Organization policy changes that weaken guardrails.
  • KMS IAM or key state changes.
  • Logging sink and audit configuration changes.

Sensitive roles to watch:

  • Owner.
  • Editor.
  • Service Account Admin.
  • Service Account User.
  • Service Account Token Creator.
  • Project IAM Admin.
  • Organization Admin.
  • KMS Admin or Decrypter.

Senior answer: GCP escalation often turns on service account impersonation and broad inherited project roles. Those are the events to make loud.

34. Compare GuardDuty, Security Command Center, and SIEM detections.

Tool Scope Strength Limitation
GuardDuty AWS managed threat detection Fast AWS-native findings from CloudTrail, VPC DNS, EKS, S3 signals AWS-focused and finding-driven
Security Command Center GCP security posture and threat findings GCP-native misconfig and threat visibility GCP-focused and tier-dependent
SIEM Cross-platform correlation Custom detections across clouds, identity, endpoint, SaaS Requires engineering, tuning, and data quality

How I use them together:

  • Native tools provide high-signal cloud-specific detections.
  • SIEM correlates across AWS, GCP, IdP, endpoint, CI/CD, and SaaS.
  • CSPM identifies misconfiguration before exploitation.
  • Incident response needs raw logs, not only findings.

Senior answer: managed detections are accelerators, not replacements for cloud audit strategy and custom correlation.

35. How would you investigate suspicious cloud API calls?

I would identify the principal, source, intent, permissions used, and downstream impact.

Investigation steps:

  • Confirm event source: CloudTrail, Cloud Audit Logs, SIEM, GuardDuty, SCC, or CSPM.
  • Identify principal, role session, service account, or federated identity.
  • Check source IP, user agent, region, device, and network path.
  • Review preceding events for login, role assumption, key creation, IAM change, or CI job.
  • Determine whether the action succeeded or failed.
  • Enumerate resources touched.
  • Correlate with data access, KMS decrypts, storage reads, and network egress.
  • Check whether credentials are still valid.
  • Contain by revoking token, disabling key, removing role binding, or isolating workload.

Questions:

  • Was this expected automation?
  • Was MFA or approved federation present?
  • Did the principal recently gain new permissions?
  • Were logs or detections disabled around the same time?

Senior answer: suspicious API calls are identity incidents until proven otherwise. Start from the principal and trace outward.

36. How would you design alerting for cloud data exfiltration?

Exfiltration alerting should combine data access, identity risk, volume anomalies, and destination signals.

Signals:

  • Large S3, Cloud Storage, BigQuery, or database reads.
  • Unusual object listing followed by bulk gets.
  • KMS decrypt spike.
  • Access from unusual country, ASN, or user agent.
  • New access key or service account key followed by data access.
  • Public bucket or signed URL creation.
  • Snapshot, export, or backup creation.
  • High egress to unfamiliar destinations.
  • Downloads outside normal working hours or by unusual principal.

Design:

  • Build baselines per workload, tenant, and data store.
  • Add high-fidelity rules for sensitive datasets.
  • Use DLP or classification tags to weight severity.
  • Correlate control-plane and data-plane logs.
  • Alert on impossible or rare paths, such as new key then bulk export.

Senior answer: exfiltration detection improves dramatically when data classification is connected to identity and egress telemetry.

37. How would you detect public storage bucket exposure?

Detection should continuously check policy, ACLs, public access settings, and actual reachability.

AWS checks:

  • S3 Block Public Access state.
  • Bucket policy with public principals.
  • Object ACLs where ACLs are still used.
  • Access points and cross-account grants.
  • CloudFront origins exposing private buckets accidentally.
  • AWS Config, Access Analyzer, Security Hub, or CSPM findings.

GCP checks:

  • Cloud Storage IAM bindings granting allUsers or allAuthenticatedUsers.
  • Uniform bucket-level access state.
  • Public object ACLs where applicable.
  • SCC and Cloud Asset Inventory findings.

Validation:

  • Confirm whether public listing or object read works.
  • Identify sensitive data classification.
  • Review who changed policy and when.
  • Alert data owner and security.

Senior answer: do not stop at policy inspection. Verify effective public reachability and classify what was exposed.

38. How would you prevent accidental public exposure of cloud storage?

Prevention should use default-deny guardrails, ownership review, and automated remediation.

Controls:

  • Enable S3 Block Public Access at account and bucket levels.
  • Use GCP organization policies to restrict public access where available.
  • Prefer uniform bucket-level access in GCP.
  • Disable legacy ACLs where possible.
  • Require infrastructure-as-code review for bucket policies.
  • Use least-privilege service identities for writes.
  • Tag buckets with owner, data class, and exposure intent.
  • Run CSPM and policy-as-code checks in CI.
  • Auto-remediate unintended public grants.
  • Use signed URLs or CDN with explicit controls for intended sharing.

Senior answer: public buckets should be impossible by default and explicit by exception.

39. How would you secure serverless functions?

Serverless security focuses on event sources, execution identity, secrets, dependencies, network egress, and observability.

Controls:

  • One execution role or service account per function or bounded function group.
  • Minimal permissions on queues, storage, databases, and secrets.
  • Validate event input from S3, Pub/Sub, HTTP, queues, and schedulers.
  • Store secrets in a secrets manager, not environment variables when avoidable.
  • Restrict outbound network where possible.
  • Put private functions behind authenticated gateways.
  • Use signed or authenticated event sources.
  • Patch dependencies and scan packages.
  • Set concurrency, timeout, and memory limits.
  • Log request IDs, event source, auth context, and downstream calls.

Senior answer: serverless reduces server management, but it does not remove identity, input, dependency, or data-flow risk.

40. What are Lambda or Cloud Functions security risks?

Risks:

  • Over-permissioned execution role or service account.
  • Secrets in environment variables or logs.
  • Unauthenticated HTTP triggers.
  • Event injection from writable buckets, queues, or topics.
  • Dependency vulnerabilities.
  • Insecure deserialization or command execution in handlers.
  • Excessive concurrency causing cost or downstream denial of service.
  • Cold-start packages containing vulnerable or malicious code.
  • Shared layers or build artifacts controlled by too many teams.
  • Public egress to attacker infrastructure.

AWS examples:

  • Lambda execution role can read all S3 buckets.
  • Function triggered by S3 processes attacker-controlled file with vulnerable parser.

GCP examples:

  • Cloud Function runs as default service account with broad project permissions.
  • Public HTTP trigger lacks IAM or application authentication.

Senior answer: the common serverless failure is treating small functions as low-risk. A tiny function with a broad role can be a large breach path.

Chunk 5 - Questions 41-50

41. How would you secure Kubernetes in cloud?

Cloud Kubernetes security spans cluster control plane, node identity, pod identity, network, admission, secrets, images, and observability.

Controls:

  • Use private clusters or restrict API server access.
  • Enforce strong authentication and least-privilege RBAC.
  • Separate cluster-admin from application teams.
  • Use workload identity instead of node-wide cloud permissions.
  • Restrict privileged pods, host networking, host PID, hostPath, and unsafe capabilities.
  • Enforce admission policies with Pod Security Admission, OPA Gatekeeper, Kyverno, or equivalent.
  • Scan images and sign artifacts.
  • Use network policies for east-west traffic.
  • Store secrets in cloud secret managers or encrypt Kubernetes secrets with KMS.
  • Patch nodes and control plane.
  • Enable audit logs and runtime detection.

Senior answer: secure cloud Kubernetes by preventing pod compromise from becoming node, cluster, or cloud-account compromise.

42. What are EKS/GKE privilege escalation risks?

Privilege escalation in EKS and GKE often crosses from Kubernetes permissions to cloud permissions or from pod to node.

EKS risks:

  • Pods reaching node IMDS and stealing node role credentials.
  • Over-permissive IRSA roles.
  • Misconfigured aws-auth granting cluster admin.
  • Privileged pods accessing host filesystem or kubelet credentials.
  • Broad node instance role.

GKE risks:

  • Workloads using broad node service account.
  • Missing Workload Identity controls.
  • Overbroad Kubernetes RBAC.
  • Privileged pods accessing node credentials.
  • Legacy metadata endpoints or insufficient metadata concealment.

Controls:

  • Use IRSA in EKS and Workload Identity in GKE.
  • Block pod access to node metadata unless required.
  • Restrict privileged workloads.
  • Keep node roles minimal.
  • Audit RBAC and cloud IAM together.

Senior answer: Kubernetes privilege escalation is not only cluster admin risk. It is often cloud identity theft through the node or workload identity layer.

43. How would you secure cloud network boundaries?

Cloud network boundaries should reduce reachability while preserving required service paths.

Controls:

  • Segment by account, project, VPC, subnet, environment, and data sensitivity.
  • Use security groups, firewall rules, NACLs, and network policies intentionally.
  • Keep management planes private or tightly restricted.
  • Use private endpoints for cloud services where possible.
  • Restrict inbound access through load balancers, identity-aware proxies, VPN, or zero-trust access.
  • Restrict east-west access between services.
  • Monitor VPC flow logs and firewall logs.
  • Avoid broad peering that flattens trust boundaries.
  • Use DNS controls to prevent unexpected service resolution.

	A[Internet] --> B[Edge controls]
	B --> C[Public services]
	C --> D[Private application tier]
	D --> E[Data tier]
	F[Admin access] --> G[Identity aware access]
	G --> D

Senior answer: cloud segmentation is about explicit reachability. If a workload does not need to connect to a service, the network should not allow it.

44. How would you design egress controls for cloud workloads?

Egress controls limit where workloads can send data and make exfiltration harder to hide.

Design:

  • Route outbound traffic through controlled NAT, proxy, or egress gateway.
  • Maintain destination allowlists for high-risk workloads.
  • Use private endpoints for cloud APIs and storage.
  • Block direct internet egress for sensitive subnets.
  • Inspect DNS and HTTP where privacy and policy allow.
  • Log destination IP, domain, workload identity, bytes, and decision.
  • Apply different egress policies by environment and data class.
  • Alert on new destinations, high volume, and rare geographies.

Cloud examples:

  • AWS: NAT gateway plus egress proxy, VPC endpoints, security group egress, route tables.
  • GCP: Cloud NAT, Private Google Access, firewall rules, VPC Service Controls where applicable.

Senior answer: egress controls are not only network hygiene. They are one of the last chances to catch or slow data theft.

45. What is CSPM and what problems does it solve?

CSPM, or Cloud Security Posture Management, continuously identifies cloud misconfigurations, risky permissions, exposure, and compliance drift.

Problems it solves:

  • Public storage exposure.
  • Overly broad IAM.
  • Missing encryption or logging.
  • Insecure network paths.
  • Unpatched or exposed workloads.
  • Risky Kubernetes configurations.
  • Drift from approved baselines.
  • Compliance evidence collection.

What CSPM does not solve alone:

  • Runtime exploitation.
  • Application-layer authorization bugs.
  • Business logic abuse.
  • Incident response without logs.
  • Ownership and remediation accountability.

Senior answer: CSPM turns cloud configuration into continuous security inventory. Its value depends on prioritization and remediation ownership.

46. How would you prioritize cloud misconfigurations?

I prioritize by exploitability, asset sensitivity, internet exposure, privilege, blast radius, and evidence of active abuse.

Priority factors:

  • Publicly reachable and exploitable.
  • Grants admin or privilege escalation path.
  • Protects regulated or sensitive data.
  • Affects production or shared platform services.
  • Has known exploit or active threat intelligence.
  • Breaks logging, detection, or backup controls.
  • Exists across many accounts or projects.
  • Has compensating controls or not.

Example ranking:

Finding Priority
Public bucket with customer PII Critical
Admin role assumable by external account Critical
Missing tag on dev VM Low
Security group allows SSH from internet on production High
Logging disabled in prod account Critical

Senior answer: prioritize paths to breach, data exposure, and detection failure. Compliance-only severity often needs security-context correction.

47. What are cloud security anti-patterns in large organizations?

Common anti-patterns:

  • One shared admin role for many teams.
  • Default service accounts with broad permissions.
  • Static access keys in CI/CD.
  • Flat networks connected by broad peering.
  • Public buckets handled only through manual review.
  • Logs stored only inside workload accounts.
  • Security exceptions with no owner or expiration.
  • One production account or project containing unrelated systems.
  • KMS keys administered by the same teams that administer data.
  • CSPM findings with no remediation workflow.
  • Treating AWS and GCP as identical instead of mapping concepts carefully.

Senior answer: cloud security fails at scale when ownership, identity boundaries, and guardrails are unclear.

48. How would you map AWS security controls to GCP equivalents?

I map by security function rather than by service name.

Threat detection GuardDuty, Security Hub Security Command Center
Key management KMS, CloudHSM Cloud KMS, Cloud HSM, External Key Manager

Senior answer: exact service parity is less important than preserving the control objective: identity, least privilege, logging, segmentation, encryption, and detection.

49. What cloud evidence would you collect during an incident?

Evidence depends on the incident, but I collect identity, control-plane, data-plane, network, workload, and configuration artifacts.

Evidence:

  • CloudTrail or Cloud Audit Logs.
  • IAM policy and role or service account history.
  • STS or service account impersonation events.
  • Access key and service account key creation events.
  • KMS decrypt and key admin events.
  • Storage access logs and object version history.
  • VPC flow logs, DNS logs, firewall logs, load balancer logs.
  • Instance or disk snapshots.
  • Container and Kubernetes audit logs.
  • Function invocation logs.
  • CI/CD logs and artifact provenance.
  • CSPM snapshots and asset inventory.
  • GuardDuty, SCC, SIEM, EDR, and WAF findings.

Senior answer: collect evidence before containment destroys volatile context where possible, but do not delay containment when active harm is ongoing.

50. What questions would you ask during a cloud architecture security review?

Review questions:

  1. What are the account, project, VPC, and tenant boundaries?
  2. Which identities can administer, deploy, assume, or impersonate workload identities?
  3. What can each workload identity read, write, decrypt, and modify?
  4. How is metadata service access protected?
  5. Where are secrets stored, rotated, and audited?
  6. Which data stores contain sensitive or regulated data?
  7. Are audit logs centralized outside the workload blast radius?
  8. What detections cover IAM abuse, data exfiltration, public exposure, and logging tampering?
  9. What network paths exist for inbound, east-west, and egress traffic?
  10. What are the top privilege escalation paths from one compromised workload?

Senior closing: a strong cloud review follows identity and data. If we know who can act, what they can reach, what they can decrypt, and what gets logged, we can reason about real risk.
