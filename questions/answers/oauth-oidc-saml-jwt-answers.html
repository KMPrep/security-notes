Chunk 1 - Questions 1-10

1. Explain OAuth2 Authorization Code Flow with PKCE step by step.

Authorization Code Flow with PKCE is the modern default for browser, mobile, desktop, and SPA-style clients because it keeps tokens out of the front channel and protects the authorization code from interception.

sequenceDiagram
    participant U as User
    participant C as Client
    participant A as AuthServer
    participant R as ResourceServer
    C->>C: Create verifier and challenge
    U->>C: Start sign in
    C->>A: Auth request with challenge and state
    A-->>U: Authenticate and consent
    A-->>C: Redirect with code and state
    C->>A: Exchange code with verifier
    A->>A: Verify challenge
    A-->>C: Access token and maybe ID token
    C->>R: Call API with access token
    R-->>C: API response

Step by step:

  1. The client creates a high-entropy code_verifier.
  2. The client derives a code_challenge, normally BASE64URL(SHA256(code_verifier)).
  3. The client redirects the user to the authorization server with response_type=code, client_id, exact redirect_uri, scope, state, code_challenge, and code_challenge_method=S256.
  4. The authorization server authenticates the user and obtains consent or applies policy.
  5. The authorization server redirects back to the registered redirect URI with an authorization code and the original state.
  6. The client validates state to ensure the response belongs to the session that initiated the flow.
  7. The client sends the code and original code_verifier to the token endpoint.
  8. The authorization server hashes the verifier and compares it with the original challenge.
  9. If valid, the authorization server issues tokens.
  10. The client uses the access token to call the resource server.

PKCE protects the code exchange. state protects the redirect response from CSRF or mix-up. OIDC adds nonce to bind the ID token to the authentication request.

Senior design point: PKCE is necessary but not sufficient. You still need exact redirect URI validation, issuer validation, token validation, secure token storage, short token lifetimes, and revocation behavior.

2. Why is PKCE required for public clients?

Public clients cannot keep a client secret confidential. A mobile app, desktop app, CLI, or browser SPA ships code to an environment the user or attacker can inspect. If the security of the flow depends on a static client secret embedded in that app, the secret is not really secret.

PKCE solves a specific problem: authorization code interception. If an attacker steals the authorization code from the redirect, they still cannot exchange it unless they know the one-time code_verifier generated by the legitimate client.

Without PKCE:

Attacker steals authorization code
Attacker sends code to token endpoint
Authorization server issues tokens

With PKCE:

Attacker steals authorization code
Attacker lacks code_verifier
Token endpoint rejects exchange

This is especially important for mobile custom URL schemes, browser redirects, malicious apps registered for similar schemes, intercepted logs, browser history, and compromised front-channel paths.

Best practices:

  • Use S256, not plain challenge method.
  • Generate a fresh verifier for each authorization request.
  • Bind the authorization code to client ID, redirect URI, and challenge.
  • Reject code reuse.
  • Use PKCE for confidential clients too where possible, because it adds defense in depth.

PKCE does not authenticate the user and does not authorize API access by itself. It protects the authorization code exchange so the right client instance can redeem the code.

3. What problem does the OAuth2 state parameter solve?

The OAuth2 state parameter protects the client redirect flow by binding the authorization response to the browser session that initiated it. Its main purpose is CSRF protection for OAuth redirects, and it also helps prevent mix-up and response injection when implemented carefully.

Attack without state:

1. Attacker starts OAuth flow for attacker account.
2. Authorization server redirects with attacker's code.
3. Attacker tricks victim browser into sending that code to client callback.
4. Client links victim session to attacker authorization result.

Correct design:

  • Client creates a high-entropy random state.
  • Client stores it server-side or in a protected browser-bound transaction store.
  • Client sends state in the authorization request.
  • Client receives state at the redirect URI.
  • Client accepts the response only if the value matches the pending transaction.
  • Client expires the state after use.
flowchart TD
    A[Start auth request] --> B[Create random state]
    B --> C[Store state with browser session]
    C --> D[Send state to auth server]
    D --> E[Receive redirect]
    E --> F{State matches?}
    F -- Yes --> G[Exchange code]
    F -- No --> H[Deny and log]

Common mistakes:

  • Static state value.
  • Predictable state value.
  • State stored in a place an attacker can modify.
  • State not checked on callback.
  • State used to carry sensitive data in cleartext.
  • State accepted more than once.

production-ready phrasing: state answers, Is this authorization response for the login transaction my client started?

4. What problem does the OIDC nonce parameter solve?

The OIDC nonce parameter binds an ID token to a specific authentication request. It prevents replay or injection of an ID token from a different authentication transaction, especially in browser-visible flows.

The client sends a fresh random nonce in the authentication request. The OpenID Provider includes that nonce in the ID token. The relying party validates that the nonce in the ID token matches the nonce stored for the original session.

sequenceDiagram
    participant RP as RelyingParty
    participant OP as OpenIDProvider
    RP->>RP: Generate nonce
    RP->>OP: Auth request with nonce
    OP-->>RP: ID token containing nonce
    RP->>RP: Validate nonce matches session

Without nonce, an attacker may be able to replay a previously issued ID token into a vulnerable client flow. This is especially relevant in implicit and hybrid flows, and it remains a useful binding control in OIDC designs.

Nonce validation is part of ID token validation, along with signature, issuer, audience, expiry, issued-at, authorized party, and authentication-time checks.

production-ready phrasing: nonce answers, Is this ID token the one produced for my authentication request?

5. Why are state and nonce not interchangeable?

state and nonce bind different artifacts at different protocol layers.

Parameter Protocol role Protects Validated by
state OAuth redirect transaction binding Authorization response and client session OAuth client
nonce OIDC ID token binding ID token replay or injection OIDC relying party

state is returned as a redirect parameter. It tells the client whether the callback corresponds to an authorization request it started. It is primarily a CSRF and transaction-binding control.

nonce is embedded in the ID token by the OpenID Provider. It tells the relying party whether that ID token was minted for the authentication request associated with this client session.

Using only state may leave ID token replay risks. Using only nonce may leave redirect CSRF risks. In OIDC browser flows, a robust client often uses both.

Example:

state protects the code callback
nonce protects the ID token

Senior answer: they are complementary. state binds the front-channel response to the client transaction. nonce binds the identity assertion to the authentication request.

6. What does OIDC add on top of OAuth2?

OAuth2 is an authorization framework. It lets a client obtain an access token to call an API. It does not, by itself, standardize how a client authenticates a user or receives a trustworthy user identity assertion.

OIDC adds an identity layer on top of OAuth2.

Key additions:

OIDC addition What it provides
openid scope Signals an OIDC authentication request
ID token Signed assertion about the authentication event and user
Standard claims iss, sub, aud, exp, iat, auth_time, amr, acr
Nonce Replay protection for ID token responses
UserInfo endpoint Standard way to fetch additional user claims
Discovery metadata Standard endpoint and key metadata publication
JWKS Public keys for token signature validation
Logout specs Optional session and logout coordination mechanisms

OAuth example: Allow this app to read my calendar.

OIDC example: Sign me into this SaaS app using my enterprise identity provider.

The relying party must validate the ID token and then usually create its own local session. The ID token proves an authentication event; it is not a replacement for application authorization.

7. Why should an ID token not be used as an API access token?

An ID token is intended for the client or relying party. It answers who authenticated and under which issuer and client context. An access token is intended for a resource server or API. It answers what API access was granted.

Using an ID token as an API access token breaks token audience and semantics.

Problems:

  • The aud of an ID token is normally the client ID, not the API.
  • ID tokens may contain identity claims but not API scopes or permissions.
  • APIs that accept ID tokens may accept tokens minted for the wrong client.
  • It encourages token substitution attacks.
  • It confuses authentication with authorization.
  • It may expose unnecessary identity claims to APIs.

Correct model:

flowchart TD
    A[User signs in] --> B[Client validates ID token]
    B --> C[Client creates session]
    A --> D[Client receives access token]
    D --> E[API validates access token]
    E --> F[API checks scopes and policy]

API validation should expect an access token with issuer, audience, expiry, scope, subject, client, and authorization context appropriate for that API. If the API receives an ID token, it should reject it because it was not the intended audience.

Phrasing: ID tokens are for the relying party; access tokens are for resource servers. Mixing them creates confused trust boundaries.

8. What claims must be validated before trusting an ID token?

A relying party must validate both cryptographic integrity and semantic claims before trusting an ID token.

Validation checklist:

Check Why it matters
Signature Confirms token was issued by the trusted provider
alg Prevents accepting unsafe or unexpected algorithms
kid Selects the right trusted key without trusting attacker input blindly
iss Must exactly match the configured trusted issuer
aud Must include the relying party client ID
azp Must match expected client when required, especially with multiple audiences
exp Reject expired tokens
nbf Reject tokens not yet valid
iat Reject tokens too old or from impossible future
nonce Must match stored request nonce when used
auth_time Required when enforcing recent authentication
acr and amr Useful for assurance and MFA policy decisions
sub Stable user identifier for account mapping

Do not use email as the primary stable account key unless the system has explicitly designed for email changes and issuer-specific identity. Prefer iss plus sub as the stable federated identity key.

Also validate the token was obtained through the expected flow and client registration. If dynamic discovery is used, the issuer and metadata must be pinned to a trusted tenant or allowlist.

Residual risk: a valid ID token says the provider authenticated the user. It does not automatically grant application permissions. Role and entitlement mapping still need local policy controls.

9. What claims must be validated before trusting a JWT access token?

A JWT access token must be validated by the resource server that receives it. The resource server should not accept any signed JWT just because the signature verifies under some known key.

Validation checklist:

Check Why it matters
Signature Confirms token integrity and issuer control
Expected algorithm Prevents algorithm confusion
Trusted issuer iss Prevents accepting tokens from untrusted providers
Audience aud Ensures token was minted for this API
Expiry exp Prevents stale token use
Not before nbf Prevents early use
Issued at iat Supports age and clock-skew checks
Subject sub Identifies user or workload
Client ID or authorized party Identifies calling app where relevant
Scope or permissions Defines requested API access
Tenant or organization Enforces tenant boundary
Token type Distinguishes access token from ID token or refresh token
Token ID jti Supports replay tracking or revocation where used

Access token validation should be API-specific. A token for billing-api should not work at admin-api. A token issued by a customer IdP should not work in an employee-only service unless that issuer is explicitly trusted for that audience.

For microservices, I would publish a shared validation library or sidecar pattern that pins trusted issuers, audiences, algorithms, JWKS locations, cache TTLs, and error behavior. That avoids every team inventing subtly different validation logic.

Important: claims are inputs to authorization, not the whole authorization decision. The API still needs object-level, tenant-level, and action-level checks.

10. What is the difference between issuer, audience, subject, and authorized party claims?

These claims answer different trust questions.

Claim Meaning Security question
iss Issuer that minted the token Do I trust this identity provider or authorization server?
aud Intended audience Was this token meant for me?
sub Subject of the token Who or what does this token represent?
azp Authorized party Which client was authorized to use this token?

Example OIDC ID token:

iss = https://idp.company.com
aud = payroll-app-client-id
sub = 00u123456789
azp = payroll-app-client-id

The relying party should trust it only if the issuer exactly matches the configured IdP and the audience matches that relying party's client ID.

Example API access token:

iss = https://auth.company.com
aud = https://api.company.com/invoices
sub = user:alice
azp = client:expense-app

The invoices API should validate that the issuer is trusted and the audience is the invoices API. It can then use sub, client identity, scopes, and local policy to decide what Alice can do.

Common failure: accepting a token with a valid signature but wrong audience. That lets a token minted for one service be replayed to another service.

Chunk 2 - Questions 11-20

11. What is the JWT alg:none vulnerability?

The alg:none vulnerability happens when a JWT verifier accepts a token that claims it has no signature algorithm. In a vulnerable implementation, an attacker can remove the signature, set the header to alg:none, modify claims such as sub or admin, and the server accepts it as valid.

Bad verifier logic:

Read JWT header
Use algorithm from token header
If alg is none, skip signature verification
Trust payload claims

Why it is dangerous:

  • The attacker controls the token header and payload.
  • The server lets attacker-controlled metadata decide verification behavior.
  • Any claim can be forged if signature verification is skipped.
  • It turns a signed token format into unsigned client-side JSON.

Correct controls:

  • Pin expected algorithms in server configuration.
  • Reject none unless the application has an extremely specific and isolated reason, which normal auth systems should not.
  • Do not let the token header decide whether verification is required.
  • Use mature JWT libraries and configure them strictly.
  • Validate issuer, audience, expiry, subject, and scopes after signature verification.
  • Test with unsigned and tampered tokens in CI.

Model answer: alg:none is an algorithm confusion bug where the verifier trusts the token's untrusted header and accepts an unsigned token. The fix is to require signature verification with explicitly allowed algorithms and keys.

12. Explain JWT key confusion attacks.

JWT key confusion attacks happen when the verifier accepts a token signed with an unexpected algorithm or key type. The classic version is confusing asymmetric and symmetric signing.

Example:

Expected design:
    Issuer signs with RS256 private key
    Verifier checks with RS256 public key

Vulnerable verifier:
    Accepts HS256 from token header
    Uses configured RSA public key as HMAC secret

Attacker:
    Signs forged token with HS256 using public key as secret
    Server accepts forged token

Root cause: the verifier lets attacker-controlled token metadata influence algorithm and key selection.

Controls:

  • Pin accepted algorithms per issuer and client.
  • Do not mix symmetric and asymmetric algorithms for the same token validation path.
  • Bind each kid to an expected issuer, algorithm, and key type.
  • Reject tokens with unexpected alg, typ, or key type.
  • Do not fetch keys from attacker-controlled URLs in token headers.
  • Keep separate validation code paths for ID tokens and access tokens.
flowchart TD
        A[Receive JWT] --> B[Read trusted issuer config]
        B --> C[Allowed algorithms and keys]
        C --> D{Header matches config?}
        D -- Yes --> E[Verify signature]
        D -- No --> F[Reject token]

Senior phrasing: JWT headers are routing hints, not policy. The verifier should use trusted configuration to decide algorithms and keys, then check whether the token is consistent with that configuration.

13. How can a weak JWKS implementation be exploited?

JWKS endpoints publish public keys used to verify JWT signatures. A weak implementation can be exploited if the verifier trusts the wrong JWKS, caches keys unsafely, accepts attacker-controlled key IDs, or fails to bind keys to issuers.

Common JWKS failure modes:

Failure Exploit
Dynamic JWKS URL from token header Attacker points verifier to attacker key set
No issuer binding Key from one IdP validates token from another context
Weak kid handling Path traversal, injection, or wrong key selected
Duplicate kid across issuers Confused key selection
Excessive cache lifetime Compromised or revoked key remains trusted too long
No cache or rate control Attacker causes JWKS fetch storm
Accepting unknown keys silently Attacker adds or rotates to malicious keys

Secure design:

  • Configure trusted issuers and JWKS URIs out of band.
  • Fetch JWKS only from pinned issuer metadata.
  • Bind key IDs to issuer, algorithm, and key type.
  • Cache keys with bounded TTL and refresh safely on unknown kid.
  • Rate-limit JWKS fetching to avoid DoS.
  • Reject tokens when key lookup is ambiguous.
  • Monitor key rotations and failed signature validation spikes.

Example attack: a token header includes jku=https://evil.example/jwks.json. A vulnerable verifier fetches that key set, verifies the attacker's forged token, and grants access. A secure verifier ignores attacker-supplied key URLs and uses only configured issuer metadata.

14. What risks exist when accepting tokens from multiple identity providers?

Accepting tokens from multiple IdPs expands the trust boundary. Each issuer has its own key lifecycle, claims, assurance level, tenant model, account recovery process, and operational maturity. If the relying service treats all tokens as equivalent, weaker providers can become the path into stronger environments.

Risks:

  • Issuer confusion: accepting a token from the wrong IdP.
  • Subject collision: same sub value means different users across issuers.
  • Audience confusion: token for one app or API accepted by another.
  • Claim schema mismatch: roles, groups, and email verified mean different things.
  • Assurance mismatch: one IdP uses phishing-resistant MFA, another uses passwords only.
  • Tenant confusion: multi-tenant IdP tokens accepted without tenant allowlisting.
  • Key confusion: JWKS or kid values not scoped by issuer.
  • Account linking abuse: attacker links an external identity to a victim local account.

Controls:

  • Maintain an explicit allowlist of trusted issuers.
  • Store federated identity as (issuer, subject), not just sub or email.
  • Validate audience per relying party or API.
  • Enforce tenant allowlists for multi-tenant IdPs.
  • Normalize claims through a controlled mapping layer.
  • Treat assurance claims as policy inputs, not blindly trusted truth.
  • Require local authorization checks after authentication.
  • Log issuer, subject, tenant, client, and mapping decision.

AWS-style example: if a SaaS platform accepts customer IdPs and employee IdPs, customer tokens should never be valid for employee admin APIs. Separate issuer trust stores, audiences, tenants, and authorization paths should enforce that boundary.

15. How would you securely implement token validation in a microservice platform?

In a microservice platform, token validation must be consistent. If every service writes custom validation logic, one service will eventually accept the wrong issuer, skip audience validation, or mishandle key rotation.

Recommended architecture:

flowchart TD
        A[Request with token] --> B[Gateway or service sidecar]
        B --> C[Shared validation library]
        C --> D[Trusted issuer config]
        C --> E[JWKS cache]
        C --> F{Token valid?}
        F -- Yes --> G[Pass verified identity context]
        F -- No --> H[Reject and log]
        G --> I[Service authorization]

Validation requirements:

  • Pin trusted issuers and allowed algorithms.
  • Validate signature with keys bound to issuer and algorithm.
  • Validate audience for the specific service or API.
  • Validate expiry, not-before, issued-at, and clock skew.
  • Validate token type and intended use.
  • Validate scopes, permissions, tenant, and client where relevant.
  • Keep ID-token and access-token validation paths separate.
  • Cache JWKS safely with bounded TTL and refresh controls.
  • Fail closed for sensitive endpoints if validation infrastructure is unavailable.

Enforcement design:

  • Gateway can perform coarse validation and reject obvious bad tokens.
  • Each service must still enforce service-specific authorization.
  • Internal identity context passed from gateway to services must be protected, for example with mTLS and signed internal headers.
  • For east-west calls, use workload identity and service-to-service authorization rather than forwarding user bearer tokens everywhere.

Detection:

  • Log token issuer, audience, subject, client, scopes, validation outcome, and reason codes.
  • Alert on invalid signature spikes, unknown issuers, wrong audience, expired token bursts, and unknown kid storms.

16. How would you design token revocation in a distributed system?

Token revocation is a control-plane problem. The system must identify what is revoked, record that state, and ensure resource servers honor it quickly enough for the risk level.

Revocation targets:

  • One access token.
  • One refresh token.
  • A refresh token family.
  • One user session.
  • All sessions for a user.
  • All tokens for a client application.
  • All tokens signed by a compromised key.
flowchart TD
        A[Revocation event] --> B[Identify token or session scope]
        B --> C[Record revoked state]
        C --> D[Publish revocation event]
        D --> E[Services update cache]
        E --> F[Future token use denied]

Design choices:

  • Use short-lived access tokens so access naturally expires quickly.
  • Keep refresh tokens centrally revocable.
  • Use token family IDs for refresh token rotation and family invalidation.
  • For high-risk APIs, use introspection or revocation checks.
  • For lower-risk APIs, rely on short lifetimes plus fast refresh revocation.
  • Propagate revocation events through reliable messaging.
  • Keep cache TTLs short for security-sensitive decisions.

Tradeoff: fully stateless JWTs scale well but are hard to revoke immediately. Opaque tokens or introspected tokens are easier to revoke but add latency and dependency on the authorization server. Many enterprise designs use short-lived JWT access tokens plus centrally managed refresh tokens.

17. What are the tradeoffs between short-lived tokens and revocation lists?

Short-lived tokens and revocation lists solve overlapping problems with different costs.

Approach Strength Weakness
Short-lived token Limits exposure window without runtime lookup Cannot stop misuse until expiry
Revocation list Can stop specific tokens before expiry Requires state, lookup, cache, and propagation
Introspection Central real-time status check Adds latency and availability dependency
Hybrid Balances scale and response More moving parts

Short-lived access tokens are simple and scalable. If a token leaks, it becomes useless soon. But during the lifetime window, an attacker can still use it unless the API checks revocation state.

Revocation lists help incident response and logout. They let the system invalidate a token, user, session, or token family immediately. But they create operational complexity: every resource server must check the list or receive updates, caches can go stale, and large revocation sets can affect performance.

Practical design:

  • Access tokens: short-lived, audience-specific, minimal claims.
  • Refresh tokens: longer-lived but rotated, centrally stored or revocable.
  • High-risk APIs: check revocation or introspect.
  • Low-risk APIs: use short lifetimes and rely on refresh-token revocation.
  • Incidents: revoke refresh token families and force re-authentication.

Senior answer: short token lifetimes reduce dependence on perfect revocation; revocation lists reduce exposure during the token lifetime. Use both based on risk.

18. How would you handle refresh tokens securely?

Refresh tokens are high-value credentials because they can mint new access tokens. I would treat them closer to long-lived sessions than ordinary API tokens.

Controls:

  • Issue refresh tokens only to clients that need long-lived access.
  • Store them securely: server-side for web apps, Keychain or Keystore for mobile, never browser local storage for high-risk apps.
  • Rotate refresh tokens on every use.
  • Detect reuse of an old refresh token as possible theft.
  • Bind refresh tokens to client, device, or sender-constrained proof where possible.
  • Scope refresh tokens to audience, client, tenant, and session.
  • Set absolute lifetime and idle lifetime.
  • Revoke on logout, password change, MFA reset, account recovery, device loss, and suspicious activity.
  • Log issuance, rotation, reuse, revocation, and anomalous use.
flowchart TD
        A[Client uses refresh token] --> B[Auth server validates token]
        B --> C{Token is current?}
        C -- Yes --> D[Issue new access token]
        D --> E[Rotate refresh token]
        C -- No --> F[Detect reuse]
        F --> G[Revoke token family]

Browser guidance: for high-risk browser apps, prefer a backend-for-frontend pattern where the refresh token stays server-side and the browser receives only a secure session cookie. For SPAs, be very cautious with long-lived refresh tokens in script-accessible storage.

19. What is refresh token rotation and what threat does it reduce?

Refresh token rotation means the authorization server issues a new refresh token each time the client uses the old one, then invalidates the old token. Each refresh token becomes single-use.

Normal rotation:

Client has RT1
Client uses RT1
Server returns access token and RT2
Server invalidates RT1
Client stores RT2

Theft scenario:

Attacker steals RT1
Client already used RT1 and received RT2
Attacker tries RT1
Server sees reuse of old token
Server revokes token family and forces reauth

Threat reduced: persistent replay after refresh-token theft. Rotation does not prevent initial theft, but it improves detection and containment. If both attacker and legitimate client race to use the same token, reuse detection can identify compromise.

Design details:

  • Track token family or session lineage.
  • Allow small grace windows only if necessary for network retries.
  • Treat reuse as high-risk and revoke the family.
  • Log old token ID, new token ID, client, device, IP, and session.
  • Avoid issuing non-rotating refresh tokens to public clients.

Rotation must be paired with secure storage. A client that leaks every new token immediately is still compromised, but rotation gives the server a chance to detect suspicious reuse.

20. How would you detect refresh token theft?

Refresh token theft is detected by looking for impossible or inconsistent use of a long-lived renewal credential.

High-signal detections:

  • Reuse of an already rotated refresh token.
  • Same refresh token family used from two devices or geographies.
  • Refresh from a new ASN, proxy, or country after stable history.
  • Refresh after user logout, password change, MFA reset, or device revocation.
  • Refresh attempts for disabled users or deleted clients.
  • Access token minting volume increases suddenly.
  • Refresh token used by a different client instance than expected.
  • Refresh followed by suspicious API activity such as data export or privilege changes.

Response:

  • Revoke the refresh token family.
  • Revoke derived access tokens where feasible.
  • Mark the user session high risk.
  • Require full reauthentication with strong MFA.
  • Notify the user or tenant admin.
  • Preserve evidence for investigation.
  • Inspect recent OAuth grants, API activity, and factor changes.

Useful logs:

Field Purpose
Token family ID Links rotations to one session lineage
Client ID Identifies app using token
Subject User or workload identity
Device ID Detects device mismatch
IP and ASN Detects network anomaly
Rotation result Identifies reuse or invalid token
Issued token ID Supports follow-on revocation

Senior answer: refresh token reuse after rotation is one of the cleanest theft signals, but you also need anomaly detection for geography, device, client, and post-refresh behavior.

Chunk 3 - Questions 21-30

21. Compare opaque tokens and self-contained JWTs.

Opaque tokens and self-contained JWTs represent two different token design tradeoffs.

Token type How it works Strength Weakness
Opaque token Random reference value looked up by server Easy revocation and minimal data exposure Requires introspection or database lookup
Self-contained JWT Signed token carrying claims Fast local validation and low lookup cost Harder revocation and claim freshness

Opaque token example:

access_token = 8PrandomReferenceValue...
API calls authorization server introspection or local token store

JWT example:

header.payload.signature
API validates signature and claims locally

Opaque tokens are attractive when revocation, privacy, and centralized policy are important. The resource server learns token status from the authorization server or a token store. If the token leaks, it reveals no claims by itself.

JWTs are attractive when many services need to validate tokens at high scale without a central lookup on every request. But JWTs can become dangerous if they are long-lived, overstuffed with claims, accepted by the wrong audience, or used without revocation strategy.

Practical hybrid: use short-lived JWT access tokens for high-volume APIs and opaque or centrally stored refresh tokens for session continuity and revocation. For very sensitive APIs, add introspection or a revocation check even if the access token is a JWT.

22. When would you prefer opaque tokens over JWTs?

I would prefer opaque tokens when centralized control, revocation, and privacy matter more than fully stateless validation.

Good use cases for opaque tokens:

  • High-risk APIs where immediate revocation is required.
  • Consumer systems where token payload privacy matters.
  • Environments with frequent entitlement changes.
  • Systems with complex dynamic policy that cannot fit safely in token claims.
  • Logout-sensitive applications.
  • Regulatory contexts where minimizing data in bearer tokens matters.
  • Internal platforms where introspection latency is acceptable.

Why opaque tokens help:

  • Token leaks reveal little because the token is just a reference.
  • The authorization server can revoke or change token status centrally.
  • Resource servers can get fresh policy context during introspection.
  • Claims are not cached in a long-lived bearer object.

Tradeoffs:

  • Introspection adds latency.
  • The authorization server or token introspection service becomes a runtime dependency.
  • Caching is needed for scale but introduces freshness tradeoffs.
  • Multi-region availability must be designed carefully.

Example: for an admin API that changes IAM policy or accesses customer data, I might prefer opaque access tokens with introspection, or at least JWTs with very short lifetimes plus revocation checks. For high-volume low-risk read APIs, short-lived JWTs may be more efficient.

23. How would you design service-to-service authorization using OAuth2?

For service-to-service authorization, OAuth2 commonly uses client credentials, token exchange, or workload identity federation. The key is that the token represents the calling workload and is scoped to the target API, not a broad shared secret.

sequenceDiagram
    participant S as ServiceA
    participant A as AuthServer
    participant R as ServiceB
    S->>A: Authenticate workload
    A-->>S: Access token for ServiceB
    S->>R: API request with token
    R->>R: Validate issuer audience and scope
    R-->>S: Response

Design principles:

  • Give each service a unique identity.
  • Prefer mTLS, private key JWT, workload identity, or platform-issued credentials over static client secrets.
  • Issue short-lived access tokens with audience set to the target service.
  • Scope tokens to specific actions and resources.
  • Validate issuer, audience, expiry, subject, client, and scopes at the resource service.
  • Use least privilege per service and environment.
  • Log actor service, target service, scopes, and decision outcome.
  • Do not use one shared backend-service credential for many services.

AWS example: an EKS workload can use OIDC federation or Pod Identity to obtain AWS credentials without static keys. For internal APIs, a service mesh can provide mTLS workload identity, while an authorization server issues audience-specific tokens for application-layer authorization.

Important distinction: authentication proves which service is calling. Authorization decides whether that service can perform the specific action on the target resource.

24. What is OAuth2 client credentials flow and when is it appropriate?

Client credentials flow is an OAuth2 grant where a machine client authenticates to the authorization server and receives an access token for its own identity. No human user is involved.

Flow:

sequenceDiagram
    participant C as ClientService
    participant A as AuthServer
    participant R as ResourceAPI
    C->>A: Token request with client authentication
    A-->>C: Access token
    C->>R: API call with access token
    R->>R: Validate token and scopes
    R-->>C: API response

Appropriate use cases:

  • Backend service calling another backend service.
  • Daemon or batch job accessing an API.
  • CI/CD automation calling deployment APIs.
  • Monitoring or security tooling collecting telemetry.
  • Machine-owned integration where no user delegation is involved.

Not appropriate when:

  • The action is on behalf of a user and needs user context.
  • The client cannot protect its credential and there is no stronger alternative.
  • The system needs user consent semantics.
  • The token would give broad access across many unrelated services.

Security controls:

  • Use strong client authentication: mTLS, private key JWT, workload identity, or rotated secrets.
  • Scope tokens narrowly by audience and permission.
  • Keep access tokens short-lived.
  • Separate identities by service and environment.
  • Monitor unusual token issuance and API calls.

Senior answer: client credentials is for machine acting as itself, not for a machine impersonating a user.

25. What are the risks of OAuth2 implicit flow?

Implicit flow returns tokens directly through the browser front channel. It was used historically for browser-only JavaScript apps, but it is no longer recommended for new systems because Authorization Code Flow with PKCE is safer.

Risks:

  • Access tokens appear in the browser front channel.
  • Tokens can leak through browser history, debugging tools, extensions, logs, or referrers.
  • There is no back-channel code exchange protected by PKCE.
  • Token substitution and replay risks are higher.
  • SPAs may store tokens in local storage, increasing XSS impact.
  • Refresh token handling was historically limited or unsafe.

Better design:

  • Use Authorization Code Flow with PKCE.
  • Prefer backend-for-frontend for high-risk browser apps.
  • Keep tokens out of URLs and browser-readable storage where possible.
  • Use secure cookies for local sessions.
  • Apply CSP and XSS defenses.

Migration example: a SPA that receives access_token in a URL fragment should migrate to code with PKCE. The SPA receives only a code, then exchanges it with a verifier. For high-value enterprise apps, the backend can perform the exchange and store tokens server-side.

Model answer: implicit flow exposes bearer tokens in places modern designs avoid. Code plus PKCE gives public clients a safer way to get tokens without relying on a client secret.

26. What is device authorization flow and where is it useful?

Device Authorization Flow is designed for devices or tools with limited input capability. The device asks the authorization server for a device code and user code. The user completes authentication on a separate browser-capable device.

sequenceDiagram
    participant D as Device
    participant A as AuthServer
    participant U as User
    D->>A: Request device code
    A-->>D: Device code and user code
    D-->>U: Show URL and user code
    U->>A: Authenticate and enter code
    D->>A: Poll token endpoint
    A-->>D: Tokens after approval

Useful for:

  • Smart TVs.
  • CLI tools.
  • IoT devices with limited keyboards.
  • Developer tools.
  • Printers or shared appliances.
  • Headless systems where the user can approve separately.

Security controls:

  • Short expiration for user code and device code.
  • Polling interval enforcement to avoid load abuse.
  • Clear display of application and device context during approval.
  • Phishing-resistant login at the browser step where possible.
  • User must confirm the device or app name before approving.
  • Limit scopes for device grants.
  • Revoke device grants when device is lost or retired.

Risk: attackers can phish the user code or trick a user into authorizing an attacker's device. That is why the approval page should show strong context: app name, device name, location if available, requested scopes, and warning for unexpected codes.

27. What are the security risks of redirect URI misconfiguration?

Redirect URI validation is one of the most important OAuth controls. If the authorization server sends codes or tokens to an attacker-controlled URI, the attacker can steal them.

Common misconfigurations:

Misconfiguration Risk
Wildcard redirect URIs Attacker registers or controls matching subdomain
Prefix matching https://good.com.evil.com bypasses weak checks
Open redirect at registered URI Code is forwarded to attacker site
Allowing HTTP Code can be intercepted on network
Custom scheme collision Malicious mobile app captures code
Missing exact path check Code sent to unintended endpoint
User-controlled redirect parameter Attacker controls final destination

Example:

Registered redirect: https://app.example.com/callback
Vulnerable validation: startsWith https://app.example.com
Attacker uses: https://app.example.com.evil.net/callback

Controls:

  • Require exact redirect URI matching for confidential web clients.
  • Avoid wildcards, especially for production clients.
  • Require HTTPS except for documented loopback native-app cases.
  • For mobile, use claimed HTTPS links or platform app links where possible.
  • Do not allow open redirects in registered redirect endpoints.
  • Bind authorization code to client ID and redirect URI.
  • Review redirect URI changes as security-sensitive configuration.

28. How would you validate redirect URIs safely?

Safe redirect URI validation should be strict, deterministic, and based on pre-registered values.

Recommended approach:

  • Store allowed redirect URIs during client registration.
  • Require exact string match after normal parsing and canonicalization.
  • Require HTTPS for web clients.
  • Reject fragments in redirect URIs.
  • Reject userinfo components such as user@host.
  • Reject ambiguous hostnames, encoded delimiters, and control characters.
  • Do not use suffix, prefix, or loose regex matching.
  • Bind the authorization code to the exact redirect URI used.
  • Re-check the same redirect URI at token exchange.
flowchart TD
    A[Auth request] --> B[Parse redirect URI]
    B --> C[Reject unsafe URI forms]
    C --> D{Exact registered match?}
    D -- Yes --> E[Continue auth flow]
    D -- No --> F[Reject request]

Native app nuance: loopback redirects such as http://127.0.0.1:{port}/callback can be acceptable for native apps when paired with PKCE and correct local handling. Custom URL schemes are riskier because another app may claim the same scheme.

Operationally, redirect URI changes should be audited and require owner approval. A malicious or mistaken redirect URI is effectively a token exfiltration route.

29. What is authorization code interception?

Authorization code interception occurs when an attacker obtains the authorization code before the legitimate client exchanges it for tokens.

Examples:

  • Malicious mobile app registers the same custom URI scheme.
  • Browser extension reads redirect URL.
  • Reverse proxy or debug log captures callback URLs.
  • Open redirect forwards code to attacker.
  • Localhost callback is intercepted by another process.
  • Network or device malware observes redirect.

Without PKCE, a stolen code may be enough to obtain tokens. With PKCE, the attacker also needs the code verifier, which the legitimate client generated and did not send through the authorization redirect.

flowchart TD
    A[Authorization code issued] --> B{Code intercepted?}
    B -- No --> C[Client exchanges code]
    B -- Yes --> D[Attacker tries token exchange]
    D --> E{Has verifier?}
    E -- Yes --> F[Tokens issued]
    E -- No --> G[Exchange denied]

Controls:

  • Use Authorization Code Flow with PKCE.
  • Use exact redirect URI validation.
  • Prefer claimed HTTPS redirects for mobile when available.
  • Reject code reuse.
  • Keep authorization codes short-lived.
  • Bind code to client ID and redirect URI.
  • Monitor failed token exchanges caused by PKCE mismatch.

30. What is token substitution?

Token substitution is when an attacker presents a valid token in the wrong context and a relying party accepts it. The token may be valid cryptographically but invalid semantically for the service, client, issuer, tenant, or token type.

Examples:

  • ID token accepted as an API access token.
  • Access token for API A accepted by API B.
  • Token from untrusted issuer accepted by a relying party.
  • Token for attacker's client accepted by victim client.
  • Token from one tenant accepted in another tenant.
  • Old token with stale claims accepted after privilege removal.

Root cause: incomplete validation and confused token semantics.

Controls:

  • Validate issuer exactly.
  • Validate audience exactly.
  • Validate token type and intended use.
  • Validate authorized party or client ID where relevant.
  • Validate tenant and account mapping.
  • Keep ID token and access token validation paths separate.
  • Enforce local authorization after token validation.
  • Use state and nonce to bind protocol responses.

Model answer: token substitution is not about forging a token; it is about replaying a real token where it was not meant to be trusted. Correct audience, issuer, token type, client, and tenant validation are the core defenses.

Chunk 4 - Questions 31-40

31. What is consent phishing?

Consent phishing is an attack where a malicious OAuth application tricks a user or admin into granting permissions that let the attacker access data or perform actions without stealing the user's password.

Attack flow:

sequenceDiagram
    participant U as User
    participant M as MaliciousApp
    participant I as IdP
    participant A as API
    U->>M: Click fake productivity app
    M->>I: Request OAuth consent
    I-->>U: Show consent screen
    U->>I: Approve requested scopes
    I-->>M: Authorization code and tokens
    M->>A: Access user data with token

Why it is dangerous:

  • MFA may not stop it because the user willingly authorizes the app.
  • Access can persist through refresh tokens.
  • The app can access email, files, contacts, calendars, or cloud resources depending on scopes.
  • The user may not understand the requested permissions.
  • Admin consent can grant access across many users.

Controls:

  • Restrict user consent for high-risk scopes.
  • Require admin approval for sensitive permissions.
  • Verify publisher identity for third-party apps.
  • Show clear consent screens with app, publisher, scopes, and tenant impact.
  • Monitor new OAuth app grants, especially high-risk scopes.
  • Alert on unusual consent from executives, admins, and sensitive teams.
  • Revoke suspicious grants and refresh tokens quickly.
  • Educate users, but rely on policy and detection rather than education alone.

Senior answer: consent phishing abuses delegated authorization, not just authentication. The control plane must govern which apps can receive which scopes from which users.

32. How would you protect users from malicious OAuth applications?

Protection requires prevention, approval governance, user-visible clarity, and detection.

Preventive controls:

  • Maintain an application allowlist for sensitive environments.
  • Block user consent for high-risk scopes such as mail read, file read, offline access, directory read, and admin APIs.
  • Require admin review for new apps and sensitive scopes.
  • Verify publisher identity and app ownership.
  • Limit multi-tenant app access unless explicitly approved.
  • Use least-privilege scopes and incremental consent.
  • Restrict refresh token issuance for untrusted apps.
  • Apply tenant-level policies for consumer apps and unmanaged publishers.

Detection controls:

  • New OAuth grant with high-risk scopes.
  • Consent by privileged user or executive.
  • Consent from unusual location or device.
  • App quickly accesses large amounts of data after grant.
  • Many users grant same new app in a short period.
  • App requests permissions unrelated to its stated function.
  • App has low reputation, new publisher, or suspicious redirect URIs.

Response:

  • Revoke app grants and refresh tokens.
  • Disable or block the app tenant-wide.
  • Review accessed data and API calls.
  • Notify impacted users and tenant admins.
  • Add detections for similar app IDs, publishers, redirect URIs, and scopes.

AWS-style analogy: just as IAM role trust policies should restrict who can assume a role, OAuth app consent policies should restrict which apps can receive delegated access and under what conditions.

33. What is the difference between delegated authorization and authentication?

Authentication proves who the user or workload is. Delegated authorization grants a client limited permission to act on behalf of a resource owner.

Concept Question answered Example
Authentication Who authenticated? Alice signs into an app using OIDC
Delegated authorization What can this client access for Alice? Calendar app can read Alice's calendar

OAuth2 is mainly delegated authorization. OIDC adds authentication on top.

Example:

Alice authenticates to Google.
Google asks whether ExpenseApp can read calendar events.
Alice consents.
ExpenseApp receives an access token with calendar scope.

Alice's authentication does not mean ExpenseApp can access everything Alice can access. The access token should be scoped to specific APIs, actions, and lifetimes.

Common mistake: using OAuth access token issuance as proof of login without OIDC. Another mistake is treating authentication as authorization, for example granting admin access because the user authenticated through a trusted IdP. The app still needs local role, tenant, and entitlement checks.

Model answer: authentication identifies the subject; delegated authorization constrains what a client may do on that subject's behalf.

34. Compare OAuth2, OIDC, and SAML for enterprise SSO.

OAuth2, OIDC, and SAML are related but solve different problems.

Technology Primary role Token format Common use
OAuth2 Delegated authorization Access tokens, opaque or JWT API access and consent
OIDC Authentication layer on OAuth2 ID token plus OAuth tokens Modern web and mobile SSO
SAML Enterprise federation and SSO XML assertions Workforce SaaS and legacy enterprise SSO

OIDC is usually preferred for new web, mobile, and API-centric systems because it fits modern application architectures, JSON, REST, mobile clients, and discovery. SAML remains common for enterprise SaaS because many workforce identity systems and vendors support it deeply.

SAML strengths:

  • Mature enterprise SSO support.
  • Broad SaaS compatibility.
  • Strong metadata and certificate-based federation model.
  • Works well for browser-based workforce SSO.

SAML weaknesses:

  • XML signature handling is complex.
  • Assertion validation mistakes can be severe.
  • Mobile and API scenarios are less natural.
  • Logout and attribute mapping are operationally tricky.

OIDC strengths:

  • Modern token format and libraries.
  • Better fit for mobile and APIs.
  • Supports standardized discovery and UserInfo.
  • Works naturally with OAuth delegated authorization.

OIDC weaknesses:

  • Token validation mistakes are common.
  • Discovery and multi-tenant issuer handling can become trust-boundary issues.
  • Claims are often over-trusted for authorization.

Senior answer: use OIDC for new application SSO and API ecosystems where possible, SAML for enterprise SaaS compatibility, and OAuth2 when the problem is delegated API authorization rather than sign-in.

35. How does SAML assertion validation work?

A SAML service provider receives a SAML response at its Assertion Consumer Service endpoint and must validate the assertion before creating a local session.

flowchart TD
    A[Receive SAML response] --> B[Parse XML safely]
    B --> C[Find signed assertion]
    C --> D[Validate signature]
    D --> E[Check issuer and certificate]
    E --> F[Check audience and recipient]
    F --> G[Check time window]
    G --> H[Map subject and attributes]
    H --> I[Create local session]

Validation checklist:

  • Use a hardened SAML library, not custom XML signature code.
  • Disable unsafe XML parser features such as external entity expansion.
  • Validate the XML signature against the trusted IdP signing certificate.
  • Ensure the application consumes the same signed assertion that was validated.
  • Validate issuer.
  • Validate audience restriction matches the service provider entity ID.
  • Validate recipient and destination match the ACS endpoint.
  • Validate NotBefore and NotOnOrAfter with tight clock skew.
  • Validate InResponseTo for SP-initiated flows where applicable.
  • Enforce one-time use or replay detection for assertion IDs.
  • Map subject and attributes through controlled local policy.

Signature alone is not enough. A validly signed assertion for a different service, audience, or time window should still be rejected.

36. What is XML signature wrapping in SAML?

XML signature wrapping is an attack where the attacker manipulates the XML structure so the service provider validates the signature on one element but uses a different attacker-controlled element for authentication or attributes.

Simplified idea:

Signed assertion says: user is Alice
Attacker inserts unsigned assertion says: user is Admin
Vulnerable parser validates signed Alice assertion
Application reads unsigned Admin assertion

Root causes:

  • XML signatures reference nodes by ID and canonicalized structure.
  • Application code may select nodes by position or name instead of the signed reference.
  • Parser and validation library disagree about which node is trusted.
  • Developers hand-roll SAML parsing or use unsafe library defaults.

Impact:

  • Authentication bypass.
  • Privilege escalation through forged attributes.
  • Tenant confusion if attacker changes audience or subject fields.

Senior answer: XML signature wrapping is dangerous because signature valid does not necessarily mean the assertion my application used was signed. The app must consume exactly the signed assertion verified by the SAML library.

37. How would you prevent XML signature wrapping attacks?

The best prevention is to use mature SAML libraries and strict validation that binds the consumed assertion to the verified signature reference.

Controls:

  • Do not hand-roll XML signature validation.
  • Disable dangerous XML parser features such as XXE.
  • Require signed assertions or signed responses according to policy.
  • Resolve the signed element by validated ID reference.
  • Reject duplicate IDs and ambiguous references.
  • Ensure application code reads claims only from the validated signed assertion.
  • Validate issuer, audience, recipient, destination, and time window after signature validation.
  • Add tests with known signature wrapping payloads.
  • Keep SAML libraries patched.

Good implementation pattern:

SAML library validates response
Library returns trusted assertion object
Application maps claims only from that object
Application never reparses raw XML for identity fields

Operationally, SAML SP security review should include malicious XML test cases, duplicate assertion IDs, extra unsigned assertions, nested assertions, and altered attributes. If the service provider accepts any of those, the validation path is unsafe.

38. What are common SAML misconfigurations?

Common SAML misconfigurations include:

Misconfiguration Risk
Missing audience validation Assertion for one SP accepted by another
Missing recipient or destination check Assertion delivered to wrong ACS accepted
Weak time validation Replay of old assertions
No InResponseTo check Login CSRF or unsolicited assertion abuse
IdP-initiated flow over-trusted Harder transaction binding and replay control
Unsigned assertions accepted Assertion forgery if response signature is mishandled
Certificate rollover mishandled Outage or acceptance of wrong signer
Over-trusted attributes Privilege escalation through group or role claims
No replay detection Assertion reuse creates sessions
Unsafe XML parser XXE, parser confusion, or signature wrapping

Secure configuration:

  • Require signed assertions from trusted IdP certificate.
  • Validate audience, issuer, destination, recipient, and time bounds.
  • Use narrow validity windows.
  • Track assertion IDs for replay prevention.
  • Treat IdP metadata as controlled configuration.
  • Separate SAML authentication from application authorization.
  • Test certificate rollover before production.

Example: accepting any signed assertion from a trusted IdP is not enough. The assertion must be for this SP, this ACS endpoint, this time window, and this expected flow.

39. How would you securely map SAML or OIDC claims to internal roles?

Claim mapping is authorization-sensitive. A SAML or OIDC claim should be treated as an input to local policy, not as automatic application authority.

Secure mapping design:

  • Use stable identity keys: (issuer, subject) for OIDC and trusted NameID or persistent subject for SAML.
  • Maintain an explicit mapping from external groups or claims to internal roles.
  • Scope mappings by tenant, application, environment, and issuer.
  • Require admin review for privileged role mappings.
  • Do not trust user-controlled claims.
  • Do not map raw email domain to admin access.
  • Validate group claim source and freshness.
  • Prefer local entitlements for high-risk permissions.
  • Log claim values used for mapping and resulting internal role.
flowchart TD
    A[Federated assertion] --> B[Validate issuer and token]
    B --> C[Extract trusted claims]
    C --> D[Apply mapping policy]
    D --> E{Role allowed?}
    E -- Yes --> F[Create local session with role]
    E -- No --> G[Deny or default role]

Common mistakes:

  • Trusting email_verified as enterprise employment proof.
  • Treating any groups=admin claim as admin access without issuer scoping.
  • Failing open when group claims are missing.
  • Letting customers map themselves into provider-admin roles.
  • Not reviewing effective permissions after group sync changes.

For privileged roles, I would require local approval, JIT activation, and audit even if the IdP sends an admin group claim.

40. Why is email verification not enough for strong identity assurance?

Email verification proves control of an email address at a point in time. It does not prove the real-world identity, employment status, authorization, or long-term account ownership of the person using it.

Limitations:

  • Email accounts can be compromised.
  • Email addresses can be recycled or reassigned.
  • Personal email does not prove corporate affiliation.
  • A verified email does not prove MFA strength.
  • Email domains can be spoofed in weak flows if issuer validation is wrong.
  • A user may control alice@example.com but not be the employee Alice the system intends.
  • Email recovery processes may be weaker than the application.

For strong identity assurance, use stronger evidence:

  • Enterprise HR or directory lifecycle.
  • Trusted IdP with tenant validation.
  • Government ID or KYC where appropriate.
  • Manager or sponsor approval.
  • Device issuance or managed identity proofing.
  • Contract or vendor onboarding records.
  • Phishing-resistant authentication for returning sessions.

Example: a SaaS app that grants admin access to anyone with a verified @company.com email is unsafe if accounts can be self-registered or if the domain is not tied to a controlled enterprise tenant. A better design federates with the company IdP, validates issuer and tenant, maps stable subject to a local account, and applies explicit role assignment.

Senior phrasing: email verification is a contact-channel proof, not strong identity proof.

Chunk 5 - Questions 41-50

41. How would you design logout across OIDC, SAML, and local sessions?

Federated logout is difficult because there are multiple sessions: the identity provider session, the relying party or service provider session, application cookies, access tokens, refresh tokens, and sometimes sessions in downstream services.

flowchart TD
    A[User clicks logout] --> B[End local app session]
    B --> C[Revoke refresh tokens]
    C --> D[Clear browser cookies]
    D --> E[Notify IdP where supported]
    E --> F[Notify downstream services]
    F --> G[Log logout result]

Design principles:

  • Define local logout, global logout, and security revocation separately.
  • Local logout must invalidate the app session even if IdP logout fails.
  • Revoke refresh tokens and token families for the app.
  • Clear browser cookies, but do not rely only on client-side deletion.
  • For OIDC, use RP-initiated logout, front-channel logout, or back-channel logout where supported.
  • For SAML, support Single Logout only if both IdP and SP implementations are reliable; otherwise treat it as best effort.
  • For high-risk apps, use server-side sessions and central revocation.
  • Make sign out everywhere revoke all sessions, refresh tokens, remembered devices, and app grants where intended.

Tradeoff: OIDC and SAML logout protocols are not uniformly reliable across browsers, mobile apps, SaaS providers, and third-party cookie restrictions. Security-sensitive logout should rely on local session invalidation and token revocation rather than assuming federation logout completed everywhere.

42. What are the risks of dynamic OIDC discovery?

OIDC discovery lets a relying party fetch issuer metadata from a standard discovery document. It is useful for configuration and key rotation, but dynamic discovery becomes risky when user input controls the issuer or metadata location.

Risks:

  • Attacker supplies a malicious issuer URL.
  • Relying party fetches attacker-controlled discovery metadata.
  • JWKS URI points to attacker keys.
  • Client accepts tokens from an untrusted issuer.
  • SSRF through discovery URL fetching.
  • Metadata changes unexpectedly and weak clients trust them immediately.
  • Multi-tenant issuer confusion if tenant allowlisting is missing.
  • Algorithm or endpoint metadata weakens validation.

Controls:

  • Pin trusted issuers or tenant IDs.
  • Fetch discovery only from allowlisted issuer domains.
  • Validate issuer in metadata matches configured issuer exactly.
  • Pin client registrations to expected issuer.
  • Cache metadata safely and monitor changes.
  • Restrict outbound network access for metadata fetchers.
  • Do not accept JWKS URLs from token headers.
  • Treat discovery as trusted configuration, not arbitrary runtime input.

Dynamic discovery is safe when it automates retrieval from already trusted issuers. It is dangerous when it decides which issuers are trusted.

43. How can OIDC discovery become a trust-boundary issue?

OIDC discovery crosses a trust boundary because metadata tells the relying party where to send users, where to exchange codes, where to fetch keys, and which issuer value to expect. If an attacker controls discovery input, they can influence core trust decisions.

Example vulnerable flow:

User enters email domain
App derives issuer from domain
App fetches discovery from attacker-controlled domain
App accepts attacker's JWKS
Attacker signs token for victim email
App creates session

The problem is not discovery itself; it is using discovery to establish trust without a prior trust relationship.

Secure trust model:

  • Tenant admin registers IdP metadata through an authenticated admin flow.
  • System verifies domain or tenant ownership.
  • Security review approves issuer and endpoints.
  • Runtime login only uses pre-approved issuer configuration.
  • Token validation requires exact iss, expected audience, and trusted keys.
flowchart TD
    A[Admin configures issuer] --> B[Verify tenant ownership]
    B --> C[Approve metadata]
    C --> D[Runtime login]
    D --> E[Use pinned issuer config]
    E --> F[Validate token]

Senior answer: discovery should retrieve metadata for a trusted issuer, not decide that an issuer is trusted.

44. How would you threat-model a multi-tenant OIDC login platform?

I would start by identifying tenants, identity providers, relying-party clients, users, administrators, token endpoints, redirect URIs, metadata stores, and claim mapping logic. Multi-tenant federation risk often comes from one tenant's configuration bleeding into another tenant.

Assets:

  • Tenant identity configuration.
  • Client secrets and redirect URIs.
  • Authorization codes, ID tokens, access tokens, refresh tokens.
  • User-to-tenant mappings.
  • Claim-to-role mappings.
  • JWKS and discovery metadata.
  • Admin consent and app registration workflows.

Trust boundaries:

  • Browser to platform callback.
  • Tenant admin to federation configuration.
  • Platform to external IdP metadata.
  • Token validation service to JWKS cache.
  • External claims to internal roles.
  • Tenant data boundary.

Key threats:

  • Malicious tenant registers rogue IdP and tries cross-tenant login.
  • Redirect URI misconfiguration leaks codes.
  • Subject collision across issuers.
  • Email-domain based tenant routing is abused.
  • JWKS confusion across tenants.
  • Overbroad claim mapping grants admin roles.
  • Consent phishing through malicious apps.
  • Logout and revocation inconsistencies.

Controls:

  • Store identities as tenant plus issuer plus subject.
  • Pin issuer and JWKS per tenant.
  • Validate redirect URIs exactly per client.
  • Require tenant ownership proof before IdP configuration.
  • Scope claim mappings to tenant and issuer.
  • Deny by default when tenant routing is ambiguous.
  • Log all federation config changes.
  • Build test cases for cross-tenant token substitution.

45. How would you investigate a suspected federation misconfiguration incident?

I would preserve evidence first, then determine whether the issue is token validation, IdP configuration, claim mapping, redirect handling, or account linking.

Investigation steps:

  1. Scope affected tenants, clients, issuers, and time window.
  2. Preserve relevant logs and configuration snapshots.
  3. Identify suspicious login events, token issuances, and role mappings.
  4. Review recent federation configuration changes.
  5. Validate issuer, audience, redirect URI, JWKS, and claim mapping for affected clients.
  6. Look for token substitution across issuers, audiences, tenants, or token types.
  7. Review account linking events and new external identity bindings.
  8. Inspect OAuth consent grants and refresh token issuance.
  9. Revoke affected sessions, refresh tokens, grants, or client secrets if needed.
  10. Add detections and tests to prevent recurrence.

Evidence to collect:

  • Raw token headers and claims where safe, never secrets or full tokens in broad logs.
  • Token validation failure and success reason codes.
  • IdP metadata and JWKS versions.
  • Client redirect URI configuration.
  • SAML metadata or OIDC discovery snapshots.
  • Role mapping rules and group claim values.
  • Admin audit trail for federation changes.

Containment depends on failure type. Wrong audience acceptance may require patching API validators and revoking sessions. Malicious app consent may require grant revocation. SAML certificate confusion may require metadata rollback and assertion replay review.

46. What logs would you collect for OAuth/OIDC troubleshooting and security?

OAuth and OIDC logs should let you reconstruct the protocol transaction without leaking secrets.

Useful fields:

Event Important fields
Authorization request client ID, redirect URI, scopes, state present, PKCE present
Authorization response result, error code, issuer, client ID, redirect URI
Token exchange client ID, grant type, PKCE result, token type, scopes, subject
Token validation issuer, audience, algorithm, key ID, outcome, failure reason
Refresh token family ID, client, subject, rotation result, reuse signal
Consent app ID, publisher, scopes, user, admin consent flag
Discovery issuer, metadata version, JWKS URI, fetch result
Logout session ID hash, revocation scope, token family revoked

Do not log:

  • Authorization codes.
  • Access tokens.
  • Refresh tokens.
  • ID tokens.
  • Client secrets.
  • PKCE verifiers.
  • Full cookies.

Detection value:

  • PKCE mismatch spikes can indicate code interception attempts.
  • Unknown kid spikes can indicate key confusion or attack traffic.
  • Wrong audience failures can reveal token substitution attempts.
  • New high-risk consent grants can indicate consent phishing.
  • Refresh token reuse can indicate token theft.

AWS-style implementation: centralize IdP logs, application auth logs, API gateway logs, CloudTrail, and SaaS audit logs into a security account or security lake with tamper-resistant retention.

47. How would you detect anomalous OAuth consent grants?

Consent grants should be monitored like privileged access changes because they can create durable delegated access.

High-risk signals:

  • New app receives high-risk scopes.
  • offline_access or refresh-token capability granted unexpectedly.
  • Admin consent granted to a new multi-tenant app.
  • Consent by executive, finance, security, or admin user.
  • Many users consent to same app in short time.
  • App publisher is unverified, new, or low reputation.
  • Redirect URI domain is suspicious or newly registered.
  • App accesses large data volume shortly after grant.
  • App requests scopes unrelated to its stated purpose.

Detection model:

flowchart TD
    A[Consent event] --> B[Score app reputation]
    B --> C[Score scopes]
    C --> D[Score user risk]
    D --> E{High risk grant?}
    E -- Yes --> F[Alert and hold]
    E -- No --> G[Monitor usage]

Response:

  • Revoke suspicious grants.
  • Revoke refresh tokens issued to the app.
  • Disable or block the app tenant-wide.
  • Review API activity performed by the app.
  • Notify affected users and admins.
  • Add app ID, publisher, and redirect URI indicators to detections.

48. How would you prevent confused deputy issues in OAuth integrations?

Confused deputy issues occur when a service with legitimate authority is tricked into using that authority for the wrong caller, client, audience, or resource.

OAuth-specific examples:

  • API accepts token with wrong audience.
  • Client uses a token minted for another resource.
  • Authorization server does not bind authorization code to redirect URI and client.
  • Service accepts a delegated token but ignores tenant or subject.
  • Cloud integration lets customer-controlled input choose a target role without external ID or audience binding.

Controls:

  • Bind authorization codes to client ID, redirect URI, and PKCE challenge.
  • Bind access tokens to audience and scopes.
  • Validate issuer, subject, tenant, client, and audience at resource server.
  • Use resource indicators or audience-specific tokens.
  • Use state and nonce to bind protocol responses.
  • For cloud integrations, use external IDs, source ARN, source account, or equivalent conditions.
  • Log both actor and effective subject.
  • Avoid broad service identities shared across tenants.

AWS example: cross-account role assumption should use a trust policy with external ID or strong source conditions so one customer cannot trick a SaaS provider into using its AWS integration role against another customer's resources.

Senior answer: bind authority to the original actor, intended client, target audience, and resource. Do not let a token or service credential float across contexts.

49. How would you securely integrate third-party SaaS with enterprise SSO?

Secure SaaS SSO integration requires strong federation configuration, lifecycle automation, least-privilege claims, and monitoring.

Steps:

  • Choose OIDC or SAML based on SaaS support and enterprise standard.
  • Register the SaaS as an application in the enterprise IdP.
  • Use exact redirect URIs for OIDC or exact ACS and entity IDs for SAML.
  • Configure signing keys or certificates with planned rollover.
  • Assign users through groups, not broad all-user access by default.
  • Map claims minimally and explicitly.
  • Use SCIM for provisioning and deprovisioning if supported.
  • Require MFA or conditional access for the SaaS based on data sensitivity.
  • Limit admin roles and require JIT where possible.
  • Test login, logout, deprovisioning, role changes, and certificate rollover.
  • Centralize SaaS audit logs.

Security review questions:

  • Does the SaaS validate SAML audience, recipient, destination, and signature correctly?
  • Does OIDC validate issuer, audience, nonce, and redirect URI?
  • What happens when a user is disabled in the IdP?
  • Are local SaaS accounts still usable after federation is enabled?
  • Can users self-link external identities?
  • Are admin roles locally assigned or IdP-controlled?

AWS-style example: for AWS IAM Identity Center integrated with an external IdP, validate SAML metadata, require strong MFA upstream, map groups to permission sets deliberately, and monitor CloudTrail for role assumptions and identity-center assignment changes.

50. What are the top five token-handling mistakes you look for in code review?

Top five mistakes:

  1. Skipping semantic validation: Code verifies the signature but does not validate issuer, audience, expiry, token type, tenant, or scopes.
  2. Trusting attacker-controlled token headers: Code accepts alg:none, algorithm confusion, untrusted kid, jku, or dynamic JWKS from the token.
  3. Using the wrong token for the job: ID token used as API access token, access token used as session identity without local checks, or refresh token sent to resource servers.
  4. Unsafe storage and logging: Tokens stored in local storage, URLs, logs, analytics, crash reports, or support traces.
  5. No revocation or rotation strategy: Long-lived JWTs, non-rotating refresh tokens, no token family invalidation, and logout that does not revoke anything server-side.

Other serious findings:

  • Accepting tokens from multiple issuers without issuer-specific config.
  • Using email as the only identity key.
  • Missing nonce or state validation.
  • Over-trusting claims for authorization.
  • Failing open when JWKS fetch or introspection fails.
  • Excessive token lifetime.
  • Sensitive data embedded in signed but unencrypted JWT payloads.

Code review question I always ask: if this token were stolen, where could it be used, for how long, and how would we revoke it?
