How To Use These Answers

Use each answer as a scaffold, not a script. A strong senior web and API security answer should name the trust boundary, explain the browser or server behavior being abused, identify the enforcement point, cover exploit examples, and close with layered prevention, detection, and operational tradeoffs.

Chunk 1 - Questions 1-10

1. Explain SSRF using a cloud metadata compromise example.

SSRF, or Server-Side Request Forgery, happens when an attacker causes a server to make a request to a destination chosen by the attacker. The server becomes the attacker's proxy into networks and services the attacker cannot reach directly.

Cloud metadata compromise is the classic severe example. Suppose an image import feature accepts a URL and fetches it from an EC2 instance.

Attacker submits:
    http://169.254.169.254/latest/meta-data/iam/security-credentials/app-role

Vulnerable app fetches it from the EC2 instance.
Metadata service returns temporary AWS credentials for the instance role.
Attacker uses those credentials against AWS APIs.

sequenceDiagram
		participant Attacker as Attacker
		participant App as WebApp
		participant Metadata as Metadata
		participant Cloud as CloudAPI
		Attacker->>App: Submit metadata URL
		App->>Metadata: Fetch role credentials
		Metadata-->>App: Temporary credentials
		App-->>Attacker: Return response
		Attacker->>Cloud: Use stolen credentials

Impact depends on the workload role. If the instance role can read customer data from S3, decrypt with KMS, or assume other roles, SSRF becomes cloud account compromise.

Senior answer: SSRF is dangerous because it converts a simple URL-fetch bug into credential theft, internal service access, and cloud control-plane abuse.

2. How would you prevent SSRF in a cloud application?

I would prevent SSRF with application validation, network egress controls, cloud metadata hardening, and least-privilege workload identity.

Controls:

  • Avoid arbitrary URL fetches when a safer product design exists.
  • Allowlist exact schemes and destinations needed by the feature.
  • Resolve DNS and validate the final IP before request.
  • Re-validate after redirects and block redirects to internal destinations.
  • Block loopback, link-local, RFC 1918, multicast, and cloud metadata IP ranges.
  • Disable non-HTTP schemes such as file, gopher, and ftp unless truly required.
  • Use an outbound proxy that enforces destination policy.
  • Put URL-fetch workers in an isolated network with no internal-service access.
  • Enforce AWS IMDSv2, GCP metadata headers, and least-privilege service accounts.
  • Monitor unusual outbound requests from web services.
flowchart TD
		A[User URL] --> B[Parse and normalize]
		B --> C[Resolve destination]
		C --> D{Allowed external target?}
		D -- Yes --> E[Fetch through egress proxy]
		D -- No --> F[Reject and log]

Senior answer: SSRF prevention should not rely on a single regex. The destination must be controlled at the app, DNS, network, and cloud identity layers.

3. What is the difference between allowlisting and denylisting for SSRF defense?

Allowlisting defines what destinations are permitted. Denylisting defines what destinations are forbidden. For SSRF, allowlisting is safer because the space of dangerous destinations and encodings is huge.

Comparison:

Approach Example Strength Weakness
Allowlist Only fetch https://images.partner.com Narrow and auditable Requires knowing legitimate destinations
Denylist Block 169.254.169.254 and localhost Easier for broad import features Many bypasses and aliases

Denylist bypass examples:

  • 127.1 for localhost.
  • IPv6 loopback.
  • Decimal or hex encoded IPs.
  • DNS rebinding to internal IPs.
  • Redirect from allowed-looking domain to metadata service.
  • CNAME to private ranges.

Senior answer: use allowlists for high-risk server-side fetches. Denylists can be defense-in-depth, but they should not be the primary boundary.

4. How can URL parsing bugs lead to SSRF bypass?

URL parsing bugs happen when validation code and request code interpret the same URL differently. Attackers exploit differences in encoding, userinfo, redirects, DNS resolution, or library behavior.

Examples:

https://trusted.com@169.254.169.254/
https://trusted.com.evil.com/
http://2130706433/
http://[::ffff:169.254.169.254]/
http://trusted.com/redirect?to=http://metadata

Common mistakes:

  • Checking string prefix instead of parsed hostname.
  • Matching contains("trusted.com") instead of exact origin.
  • Validating before following redirects but not after.
  • Resolving DNS once for validation and again for the request.
  • Trusting file extensions or URL schemes after decoding inconsistently.
  • Handling punycode or mixed encodings incorrectly.

Controls:

  • Use a single well-tested URL parser.
  • Canonicalize before validation.
  • Validate scheme, host, port, and resolved IP.
  • Re-check every redirect target.
  • Pin DNS result or route through a policy-enforcing proxy.

Senior answer: SSRF validation must match the exact request that will be sent. Any gap between parser and fetcher becomes a bypass surface.

5. How would you secure access to cloud metadata services?

Cloud metadata services provide instance identity, configuration, and often temporary credentials. They must be treated as credential services, not harmless local endpoints.

AWS controls:

  • Require IMDSv2.
  • Set hop limit to 1 where possible.
  • Disable metadata access for instances that do not need it.
  • Use least-privilege instance profiles.
  • Monitor use of role credentials from unusual locations.
  • Block metadata access from containers that do not need it.

GCP controls:

  • Metadata requests require Metadata-Flavor: Google header.
  • Limit service account scopes and IAM permissions.
  • Disable legacy metadata endpoints where applicable.
  • Use dedicated service accounts per workload.
  • Monitor token use and metadata access anomalies.

Design principle:

If SSRF succeeds, metadata should still be hard to reach and low-value if reached.

Senior answer: metadata protection is layered: token or header requirements, network isolation, least-privilege workload identity, and detection of credential misuse.

6. What is XSS and what are the main types?

XSS, or Cross-Site Scripting, is an injection vulnerability where attacker-controlled script executes in a victim's browser under a trusted site's origin. The script inherits the page's same-origin privileges.

Main types:

Type Description Example
Stored XSS Payload is stored and shown to later users Malicious comment rendered to all viewers
Reflected XSS Payload is reflected in immediate response Search query echoed without encoding
DOM XSS Client-side JS reads attacker input and writes to dangerous sink location.hash assigned to innerHTML

Impact:

  • Session theft if cookies are not HttpOnly.
  • Actions as the victim.
  • Data exfiltration from page DOM.
  • Phishing inside a trusted origin.
  • CSRF-token theft.
  • Stored worm-style propagation.

Senior answer: XSS is not just alert(1). It is arbitrary script execution in a trusted browser context.

7. How would you prevent stored XSS?

Stored XSS prevention requires safe input handling, output encoding, sanitization for rich content, and defense-in-depth controls.

Controls:

  • Encode untrusted data on output based on context: HTML text, attribute, URL, JavaScript, or CSS.
  • Use framework auto-escaping and avoid bypass APIs.
  • Sanitize rich HTML with a proven allowlist sanitizer.
  • Store user content as data, not pre-rendered trusted HTML.
  • Reject dangerous file types such as active SVG where not needed.
  • Apply CSP to reduce exploit impact.
  • Set session cookies HttpOnly and Secure.
  • Review Markdown or WYSIWYG rendering pipelines carefully.
  • Test stored content through every rendering surface, including email and admin consoles.
flowchart TD
		A[User content] --> B[Validate and store as data]
		B --> C[Contextual output encoding]
		C --> D[Browser renders text]

Senior answer: the primary control is contextual output encoding. Sanitization is needed only when the product intentionally allows user-supplied HTML.

8. How would you prevent reflected XSS?

Reflected XSS happens when request data is immediately returned in a page without safe encoding. Prevention focuses on output encoding and safe templates.

Controls:

  • Use server-side templates with auto-escaping.
  • Encode every reflected value for its output context.
  • Avoid constructing HTML strings manually.
  • Validate inputs for expected type and length, but do not rely on validation alone.
  • Return JSON with correct Content-Type for API responses.
  • Avoid reflecting raw error messages or debug values.
  • Use CSP as defense-in-depth.
  • Add tests for common payloads in search, error, redirect, and profile pages.

Example:

Bad:  Search results for <raw query>
Good: Search results for encoded text

Senior answer: reflected XSS is fixed at the render boundary. The server must never confuse user input with markup or script.

9. How would you prevent DOM-based XSS?

DOM XSS prevention requires safe client-side coding because the server may never see the payload.

Controls:

  • Avoid dangerous sinks such as innerHTML, outerHTML, document.write, eval, and string-based timers.
  • Prefer textContent, safe DOM creation APIs, and framework binding.
  • Validate and parse URL parameters before use.
  • Sanitize only with proven client-side sanitizers when HTML is required.
  • Use Trusted Types where supported.
  • Lock down postMessage handlers with strict origin and schema checks.
  • Avoid javascript: URLs and unsafe dynamic script creation.
  • Use CSP to limit script sources.
flowchart TD
		A[URL or message input] --> B[Client JavaScript]
		B --> C{Safe sink?}
		C -- Yes --> D[Render as text]
		C -- No --> E[DOM XSS risk]

Senior answer: DOM XSS is a source-to-sink problem. Trace attacker-controlled sources to executable sinks and remove the unsafe sink.

10. How does Content Security Policy reduce XSS impact?

CSP reduces XSS impact by telling the browser which scripts, styles, images, frames, and connections are allowed. It is defense-in-depth, not a replacement for output encoding.

Useful CSP protections:

  • Restrict script execution to trusted sources.
  • Block inline scripts unless nonce or hash is present.
  • Restrict connect-src to approved API endpoints.
  • Restrict framing with frame-ancestors.
  • Block plugin content and risky object embeds.
  • Report violations for monitoring.

Example policy shape:

Content-Security-Policy:
    default-src 'self';
    script-src 'self' 'nonce-random';
    object-src 'none';
    base-uri 'self';
    frame-ancestors 'none';
    report-uri /csp-report

How it helps:

  • A reflected script tag without the right nonce will not execute.
  • Injected script cannot easily load attacker-hosted JavaScript.
  • Exfiltration may be limited by connect-src.

Senior answer: CSP turns many XSS bugs from code execution into blocked browser violations, but only if the policy avoids unsafe shortcuts.

Chunk 2 - Questions 11-20

11. What are common CSP mistakes?

Common CSP mistakes weaken the policy until it no longer meaningfully reduces XSS impact.

Mistakes:

  • Using script-src *.
  • Allowing 'unsafe-inline' for scripts.
  • Allowing 'unsafe-eval' without strong justification.
  • Trusting broad CDN domains that host user-controlled content.
  • Missing object-src 'none'.
  • Missing base-uri 'self', enabling base tag abuse.
  • Missing frame-ancestors, leaving clickjacking risk.
  • Only deploying Report-Only and never enforcing.
  • Not rotating or generating nonces per response.
  • Ignoring CSP violation reports.
  • Forgetting connect-src, letting injected code exfiltrate data.

Safer rollout:

  • Start in report-only mode.
  • Remove inline scripts or migrate to nonces and hashes.
  • Use exact origins.
  • Enforce on high-risk pages first.
  • Monitor violations and tighten iteratively.

Senior answer: CSP is easy to make decorative. A strong policy removes inline script execution and uses narrow source lists.

12. What is CSRF and when is an application vulnerable?

CSRF, or Cross-Site Request Forgery, tricks an authenticated user's browser into sending a state-changing request to a site where the user is already logged in. The attacker does not need to steal the cookie; the browser sends it automatically.

An application is vulnerable when:

  • It uses cookie-based authentication.
  • It performs state-changing actions without a CSRF token or equivalent protection.
  • Session cookies are sent cross-site.
  • The endpoint accepts simple requests or forms from other sites.
  • The server trusts the presence of a session cookie as proof of user intent.
sequenceDiagram
	participant User as User
	participant Evil as EvilSite
	participant App as App
	User->>App: Login and receive session cookie
	User->>Evil: Visit attacker page
	Evil-->>User: Auto submit form to App
	User->>App: Request with session cookie
	App->>App: Action processed if no CSRF check

Examples:

  • Change email.
  • Add payment method.
  • Transfer funds.
  • Grant OAuth consent.
  • Create admin user.

Senior answer: CSRF is a user-intent problem. Authentication proves who the browser is logged in as, not that the user meant to perform the action.

13. How does SameSite cookie behavior reduce CSRF risk?

SameSite controls whether a browser sends a cookie on cross-site requests. Because CSRF relies on automatic cookie sending, SameSite reduces the attack surface.

Behavior:

  • SameSite=Strict: cookie is not sent on cross-site navigations or subrequests.
  • SameSite=Lax: cookie is sent on same-site requests and safe top-level cross-site navigations, but not most cross-site POST requests.
  • SameSite=None: cookie is sent cross-site and must use Secure.

How it helps:

  • Hidden cross-site POST forms usually lose the session cookie under Lax or Strict.
  • Cross-site image or script subrequests do not carry Strict or Lax cookies in most CSRF-relevant cases.
  • The server sees no authenticated session, so the attack fails.

Limitations:

  • SameSite is not a substitute for CSRF tokens on critical state changes.
  • Some flows require cross-site cookies, such as embedded apps or certain federation patterns.
  • XSS can bypass CSRF defenses because injected script runs same-origin.

Senior answer: SameSite moves a major CSRF control into the browser, but high-risk operations should still require explicit anti-CSRF tokens or re-authentication.

14. Compare SameSite Strict, Lax, and None.

SameSite value Cookie sent cross-site? Good use Risk
Strict No High-risk session cookies and admin apps Can break login redirects and shared links
Lax Limited safe top-level navigations General session cookies Some top-level GET flows still include cookies
None Yes, with Secure required Embedded widgets, third-party iframes, some federation CSRF exposure if no other protection

Design guidance:

  • Use Lax as a practical default for many session cookies.
  • Use Strict for highly sensitive admin or security actions where UX permits.
  • Use None; Secure only when cross-site usage is truly required.
  • Pair None with CSRF tokens, origin checks, and careful CORS policy.

Senior answer: SameSite is not one-size-fits-all. Pick the value based on whether cross-site cookie sending is required by the product flow.

15. What makes a cookie secure?

A secure cookie is scoped narrowly, sent only over HTTPS, unavailable to JavaScript when it contains session state, protected from CSRF where possible, and backed by server-side session controls.

Controls:

  • Secure for all sensitive cookies.
  • HttpOnly for session and auth cookies.
  • SameSite=Lax or Strict where possible.
  • Omit Domain for host-only session cookies.
  • Use Path=/ deliberately, not as a false security boundary.
  • Use short lifetimes and server-side expiration.
  • Rotate session IDs after login and privilege elevation.
  • Use __Host- prefix for high-value session cookies.
  • Do not store sensitive tokens in localStorage when HttpOnly cookies are viable.
  • Pair cookies with CSRF and XSS defenses.

Example:

Set-Cookie: __Host-session=abc; Secure; HttpOnly; Path=/; SameSite=Lax; Max-Age=3600

Senior answer: cookie flags are necessary, but session security also needs server-side invalidation, rotation, anomaly detection, and good authentication design.

16. Explain HttpOnly, Secure, SameSite, Path, and Domain cookie attributes.

Attribute Purpose Security implication
HttpOnly Hides cookie from JavaScript Reduces XSS cookie theft
Secure Sends cookie only over HTTPS Prevents plaintext network exposure
SameSite Controls cross-site sending Reduces CSRF risk
Path Limits URL path where cookie is sent Mostly routing, not strong isolation
Domain Controls host or subdomain scope Overbroad domain exposes cookie to subdomains

Important nuances:

  • HttpOnly does not stop XSS from making same-origin requests as the user.
  • Secure does not help if the app also leaks tokens into logs or JavaScript.
  • Path does not prevent scripts on the same origin from reading non-HttpOnly cookies.
  • Omitting Domain creates a host-only cookie, which is usually safer for sessions.
  • SameSite=None requires Secure in modern browsers.

Senior answer: cookie attributes define browser behavior. Mis-scoping a session cookie can turn a subdomain compromise into account compromise.

17. What is CORS?

CORS, or Cross-Origin Resource Sharing, is a browser mechanism that lets a server declare which other origins may read its responses. It is a controlled relaxation of Same-Origin Policy.

Key point: CORS is enforced by browsers. The server sends headers, but the browser decides whether JavaScript can read the response.

sequenceDiagram
	participant JS as BrowserJS
	participant API as API
	JS->>API: Request with Origin header
	API-->>JS: Response with allow origin header
	JS->>JS: Browser exposes or blocks response

Important headers:

  • Origin.
  • Access-Control-Allow-Origin.
  • Access-Control-Allow-Credentials.
  • Access-Control-Allow-Methods.
  • Access-Control-Allow-Headers.
  • Vary: Origin.

Senior answer: CORS does not authenticate users and does not protect the server from receiving requests. It controls whether browser JavaScript can read cross-origin responses.

18. How can CORS misconfiguration expose sensitive data?

CORS misconfiguration can allow an attacker-controlled website to read authenticated API responses from a victim's browser.

Dangerous pattern:

Access-Control-Allow-Origin: https://evil.example
Access-Control-Allow-Credentials: true

If the victim is logged into the API and the server allows the attacker's origin with credentials, JavaScript on the attacker site can read sensitive data.

Common misconfigurations:

  • Reflecting any Origin header.
  • Using broad regex such as *.example.com while subdomains are takeover-prone.
  • Allowing null origin.
  • Missing Vary: Origin with CDN caching.
  • Allowing credentials for public wildcard-like policies.
  • Enabling CORS on internal admin APIs.

Senior answer: CORS bugs turn the victim's browser into an authenticated data retrieval tool for the attacker.

19. Why is wildcard CORS dangerous with credentials?

Browsers do not allow Access-Control-Allow-Origin: * together with credentialed requests. But teams often recreate the same danger by reflecting arbitrary origins while also setting Access-Control-Allow-Credentials: true.

Why credentials matter:

  • Cookies may be sent.
  • HTTP auth may be sent.
  • Client certificates or bearer headers may be used in some flows.
  • Authenticated API responses become readable to attacker JavaScript if origin is allowed.

Bad logical policy:

If request has any Origin, echo it back.
Set Access-Control-Allow-Credentials: true.

Safe policy:

Only exact approved origins are allowed.
Credentials enabled only for origins that require them.
Vary: Origin is set for dynamic responses.

Senior answer: the dangerous part is not only the literal *; it is any policy that lets arbitrary origins read credentialed responses.

20. How would you review a CORS policy?

I would review CORS by identifying which browser origins need access, whether credentials are required, and whether the server performs authorization independent of CORS.

Review checklist:

  • List every allowed origin with scheme, host, and port.
  • Verify exact string matching, not broad regex or substring matching.
  • Check whether credentials are enabled and truly required.
  • Ensure wildcard is not used for sensitive APIs.
  • Ensure Vary: Origin is set when origin is dynamic.
  • Reject null origin unless there is a carefully justified case.
  • Limit methods and headers to what the client needs.
  • Avoid CORS on backend-only internal services.
  • Confirm server-side authz still runs on every request.
  • Audit allowed subdomains for takeover risk.
flowchart TD
	A[Request Origin] --> B{Exact allowlist match?}
	B -- No --> C[No CORS approval]
	B -- Yes --> D{Credentials needed?}
	D -- Yes --> E[Allow exact origin and credentials]
	D -- No --> F[Allow exact origin only]

Senior answer: CORS review is origin trust review. Every allowed origin is effectively allowed to read selected browser-accessible API responses.

Chunk 3 - Questions 21-30

21. What is SQL injection?

SQL injection happens when user-controlled input is interpreted as SQL code rather than data. The attacker changes the structure or meaning of the database query.

Example vulnerable pattern:

query = "SELECT * FROM users WHERE name = '" + userInput + "'"

If userInput is admin' --, the attacker may comment out the rest of the query or change the condition.

Impacts:

  • Authentication bypass.
  • Data exfiltration.
  • Data modification or deletion.
  • Schema discovery.
  • In some databases, operating system command execution.
  • Denial of service through expensive queries or sleeps.

Senior answer: SQL injection is not a database bug. It is an application bug where the code mixes untrusted data with executable query syntax.

22. How do prepared statements prevent SQL injection?

Prepared statements separate SQL structure from parameter values. The database parses the query template first, then binds user input as data, not syntax.

Unsafe:

SELECT * FROM users WHERE email = 'user input here'

Safe shape:

SELECT * FROM users WHERE email = ?
bind parameter: user input here

Why this works:

  • User input cannot add new SQL keywords, operators, or comments.
  • Quotes inside input are treated as characters.
  • Query plan and data binding are distinct.

Important caveats:

  • Prepared statements do not automatically make dynamic table names or column names safe.
  • ORDER BY fields and sort direction need allowlists.
  • ORM raw query methods can reintroduce injection.
  • Stored procedures can still be vulnerable if they build dynamic SQL unsafely.

Senior answer: parameterization is the primary fix. Escaping is fragile and should not be the main defense.

23. What is second-order SQL injection?

Second-order SQL injection happens when malicious input is stored safely at first, then later retrieved and used unsafely in a new query.

Example:

Step 1: attacker registers username admin'--
Step 2: app stores the username as data
Step 3: admin workflow later builds SQL using stored username
Step 4: payload executes in the later query

Why it is tricky:

  • Initial input validation may appear to work.
  • The payload may sit dormant for days or months.
  • Security tests that only check immediate response miss it.
  • Developers may incorrectly trust data that came from their own database.

Controls:

  • Treat stored data as untrusted when reusing it in queries.
  • Parameterize every query, including internal workflows and migrations.
  • Avoid dynamic SQL for stored values.
  • Add tests around admin, reporting, and batch jobs.

Senior answer: once user input enters your system, it does not become trusted merely because it lives in your database.

24. What is NoSQL injection?

NoSQL injection happens when user input changes the meaning of a query to a non-SQL datastore such as MongoDB, Elasticsearch, Redis-like query systems, or document databases.

Examples:

  • Passing JSON operators such as $ne or $gt into a MongoDB query.
  • Injecting query DSL into Elasticsearch filters.
  • Supplying unvalidated object structures where a scalar was expected.
  • Modifying aggregation pipeline stages.

Example risk:

Expected login filter:
    {"user":"alice", "password":"secret"}

Injected filter:
    {"user":{"$ne":null}, "password":{"$ne":null}}

Controls:

  • Validate schema and types before query construction.
  • Reject unexpected operators in user-controlled JSON.
  • Use safe query builders and parameter binding where available.
  • Use allowlists for searchable fields and sort keys.
  • Apply least privilege at datastore level.
  • Limit result size and query complexity.

Senior answer: NoSQL injection is still injection. The syntax changed, but the root cause is untrusted input controlling query semantics.

25. How would you prevent command injection?

Command injection occurs when user input reaches an operating system shell or command interpreter.

Controls:

  • Avoid shelling out when a library API exists.
  • Do not pass user input to sh, cmd, bash, or powershell.
  • Use argument arrays rather than command strings.
  • Validate inputs against strict allowlists.
  • Escape only as a last line of defense and per platform.
  • Run processes with least privilege.
  • Use sandboxing, containers, seccomp, AppArmor, or similar controls for risky processors.
  • Set timeouts and resource limits.
  • Log command execution attempts without logging secrets.

Bad pattern:

convert " + filename + " output.png

Better pattern:

execute program: convert
arguments: validated input file, output file
no shell involved

Senior answer: the safest command injection fix is removing the shell boundary. Treat shell execution as a high-risk trust boundary.

26. What is path traversal?

Path traversal happens when attacker-controlled path input escapes the intended directory and accesses files elsewhere on the filesystem.

Examples:

../../../../etc/passwd
..\..\windows\system32\config
uploads/../../secrets.env

Impacts:

  • Reading secrets or configuration files.
  • Accessing other users' files.
  • Overwriting application files.
  • Triggering unsafe file rendering.
  • Escalating to code execution if uploads or templates are abused.

Controls:

  • Do not use raw user-supplied paths.
  • Map user IDs to server-side file IDs.
  • Canonicalize and verify path stays within allowed root.
  • Reject path separators where filenames are expected.
  • Use object storage keys with controlled prefixes.
  • Enforce authorization on the object, not only the path.

Senior answer: path traversal is an authorization and filesystem boundary bug. The path must never decide access by itself.

27. How would you safely handle file paths?

Safe file-path handling means translating user intent into a server-controlled object reference.

Design:

  • Store files with generated IDs, not user-provided names.
  • Keep original filename as metadata only.
  • Normalize paths before use.
  • Join paths using platform-safe APIs.
  • Resolve canonical path and verify it remains under the intended base directory.
  • Reject absolute paths, drive letters, path separators, null bytes, and traversal segments.
  • Enforce object-level authorization before read or write.
  • Use separate storage buckets or prefixes per tenant where appropriate.
flowchart TD
		A[User file request] --> B[Lookup file ID]
		B --> C[Authorize object]
		C --> D[Build server path]
		D --> E[Read from allowed storage]

Senior answer: users should select objects, not filesystem paths. The server owns path construction.

28. Threat-model a file upload feature.

For file upload, I would model the uploader, upload service, storage, processors, renderers, download path, and downstream consumers.

Assets:

  • User files.
  • Storage bucket.
  • Malware scanning pipeline.
  • Image or document processors.
  • Metadata database.
  • Download URLs and access tokens.

Threats:

  • Malware upload.
  • Stored XSS through SVG, HTML, or PDF rendering.
  • Path traversal in filename.
  • Zip bombs or decompression bombs.
  • Parser exploits in image or document libraries.
  • Uploading executable content served from trusted domain.
  • Cross-tenant file access.
  • Sensitive data exposure through public links.
  • Denial of service through huge files or many uploads.
flowchart TD
		A[Uploader] --> B[Upload API]
		B --> C[Quarantine storage]
		C --> D[Scanner]
		D --> E{Clean?}
		E -- Yes --> F[Release storage]
		E -- No --> G[Block and alert]

Senior answer: file upload is not a single endpoint risk. It is a pipeline risk across storage, parsing, scanning, rendering, and download authorization.

29. What controls would you apply to uploaded files?

Controls:

  • Authenticate uploader and enforce tenant quota.
  • Validate file size, type, extension, and magic bytes.
  • Store initially in quarantine.
  • Rename to generated object IDs.
  • Strip or normalize dangerous metadata.
  • Scan for malware and known bad content.
  • Use safe parsers and isolate processors.
  • Limit archive depth and decompressed size.
  • Serve downloads with safe Content-Type and Content-Disposition.
  • Store files outside the web root.
  • Enforce object-level authorization on download.
  • Log upload, scan result, release, download, and delete events.

Storage policy:

Stage Access
Quarantine Scanner only
Clean release Authorized users only
Malicious Security team only, no rendering

Senior answer: the most important pattern is quarantine then release. Do not immediately serve user-uploaded bytes from a trusted origin.

30. How would you prevent malware upload and unsafe file rendering?

Malware prevention and safe rendering require isolation because scanners can miss novel payloads and file parsers have vulnerabilities.

Controls:

  • Scan uploads with multiple engines or vendor tools where risk justifies it.
  • Use file-type allowlists, not only extension blocklists.
  • Quarantine files until scanning completes.
  • Run document conversion and thumbnail generation in sandboxed workers.
  • Patch parsers and image libraries quickly.
  • Disable active content such as macros where not needed.
  • Serve untrusted files from a separate domain or origin.
  • Force download with Content-Disposition: attachment for risky types.
  • Set X-Content-Type-Options: nosniff.
  • Convert risky formats to safer formats for preview.
  • Limit file size, archive expansion, and processing time.

Senior answer: assume some malicious files will pass scanning. Isolation and safe rendering reduce blast radius when detection fails.

Chunk 4 - Questions 31-40

31. What is IDOR?

IDOR, or Insecure Direct Object Reference, happens when an application exposes an object identifier and fails to verify that the authenticated user is allowed to access that object.

Example:

GET /invoice/1001  returns Alice's invoice
GET /invoice/1002  incorrectly returns Bob's invoice

Root cause:

  • Authentication exists.
  • Object-level authorization is missing or incomplete.
  • The server trusts the object ID supplied by the client.

Controls:

  • Check principal, action, object, tenant, and relationship on every object access.
  • Avoid relying on unguessable IDs as the only protection.
  • Scope database queries by tenant and owner.
  • Test horizontal and vertical authorization cases.
  • Log denied and unusual object access patterns.

Senior answer: IDOR is not fixed by using UUIDs. UUIDs reduce enumeration but do not prove authorization.

32. What is BOLA in API security?

BOLA, or Broken Object Level Authorization, is the API-focused name for the same class of problem as IDOR. It is especially common in APIs because clients directly request objects by ID.

Example:

GET /api/accounts/acct_123/statement

The server must verify that the caller can read acct_123, not only that the caller has a valid token.

Common API BOLA surfaces:

  • REST object endpoints.
  • GraphQL resolvers.
  • Bulk export endpoints.
  • Mobile backend APIs.
  • Nested resources such as /users/{userId}/devices/{deviceId}.
  • Admin support tooling.

Senior answer: BOLA is usually the highest-risk API bug because one missing check can expose many records through automation.

33. How would you detect BOLA in API logs?

BOLA detection looks for unusual object access patterns and authorization failures.

Signals:

  • One user accesses many object IDs sequentially.
  • High rate of 403 or 404 responses followed by successful access.
  • Access to objects across many tenants, accounts, or regions.
  • User accesses objects outside normal relationship graph.
  • API client requests IDs that are close together or enumerated.
  • Mobile or browser client calls undocumented object endpoints.
  • Sudden spike in response size or record count.
flowchart TD
		A[API logs] --> B[Object ID patterns]
		A --> C[User and tenant context]
		A --> D[Status code sequence]
		B --> E[BOLA risk score]
		C --> E
		D --> E

Required log fields:

  • Actor ID.
  • Tenant ID.
  • Object ID or safe hash.
  • Action.
  • Decision.
  • Status code.
  • Client ID.
  • Request route template.

Senior answer: detection is useful, but BOLA must be prevented in authorization logic. Logs help catch missed checks and active enumeration.

34. How would you prevent object-level authorization bugs?

I would make object authorization a standard service-layer pattern rather than leaving every controller to invent it.

Controls:

  • Central authorization library or policy engine.
  • Required checks at service or resolver layer, not just gateway.
  • Data access methods require principal and tenant context.
  • Query filters include tenant and ownership constraints.
  • Deny by default when object ownership is unknown.
  • Tests for cross-user, cross-tenant, and privilege-boundary access.
  • Code review checklist for every endpoint taking object IDs.
  • Authorization decision logs with policy version.

Bad pattern:

db.getObject(request.objectId)
return object

Better pattern:

object = db.getObjectScoped(principal, tenant, objectId)
authorize(principal, read, object)
return object

Senior answer: object authorization should be hard to forget. Put it in the path developers naturally use.

35. What is mass assignment?

Mass assignment happens when an API binds client-supplied fields directly to a server-side object, allowing the client to set fields they should not control.

Example:

{
  "displayName": "Alice",
  "isAdmin": true,
  "accountStatus": "approved"
}

If the server blindly applies all JSON fields to the user model, the attacker may grant themselves admin rights or bypass workflow state.

Common dangerous fields:

  • role.
  • isAdmin.
  • tenantId.
  • ownerId.
  • price.
  • creditLimit.
  • accountStatus.
  • emailVerified.

Senior answer: mass assignment is an input authorization bug. The client should not decide which fields are writable.

36. How would you prevent mass assignment in APIs?

Controls:

  • Use explicit request DTOs or schemas for each operation.
  • Allowlist writable fields per endpoint and role.
  • Ignore or reject unknown fields.
  • Keep server-managed fields separate from client-editable fields.
  • Validate tenant and owner from authenticated context, not request body.
  • Add tests that attempt to set privileged fields.
  • Review ORM auto-binding and framework model binding defaults.

Example safe update shape:

Allowed for profile update:
    displayName
    avatarUrl

Rejected:
    role
    tenantId
    emailVerified

Senior answer: never bind request JSON directly onto persistence models. Use operation-specific input models.

37. How would you design API authentication?

API authentication should identify the caller, support machine and user clients, use short-lived credentials, and provide revocation and audit.

Design options:

Client type Good authentication pattern
Browser SPA OIDC login, session or short-lived token via backend-for-frontend
Mobile app OIDC with PKCE, device posture where needed
Server-to-server OAuth client credentials, mTLS, workload identity
Third-party API Scoped OAuth app, signed requests, key rotation
Internal service mTLS plus workload identity

Controls:

  • Use TLS everywhere.
  • Prefer short-lived access tokens and refresh-token controls.
  • Scope tokens by audience and permissions.
  • Validate issuer, audience, signature, expiration, and key ID.
  • Avoid API keys as sole auth for sensitive APIs.
  • Store secrets in a secrets manager.
  • Log client ID, subject, token type, and auth result.

Senior answer: API authentication should answer who is calling and which client or workload is involved. A bearer token with no audience or scope discipline is fragile.

38. How would you design API authorization?

API authorization decides whether the authenticated caller can perform a specific action on a specific resource.

Design:

  • Define actions and resources explicitly.
  • Enforce object-level authorization in the service layer.
  • Enforce function-level authorization for admin and privileged APIs.
  • Enforce field-level authorization for sensitive response fields.
  • Scope decisions by tenant.
  • Use policy engine or shared library for consistency.
  • Log permit and deny decisions.
  • Test direct API calls, not only UI behavior.
flowchart TD
		A[API request] --> B[Authenticate caller]
		B --> C[Load resource context]
		C --> D[Authorize action]
		D --> E{Allowed?}
		E -- Yes --> F[Return scoped response]
		E -- No --> G[Deny and log]

Senior answer: API authorization is not one check. It includes function, object, tenant, field, and workflow authorization.

39. What is API rate limiting and how would you implement it?

API rate limiting controls how many requests or how much resource consumption a caller can use over time. It protects availability, cost, and business logic.

Dimensions:

  • Per IP.
  • Per user.
  • Per API key or client ID.
  • Per tenant.
  • Per endpoint.
  • Per operation cost.
  • Per token or session.

Algorithms:

  • Token bucket for burst plus sustained rate.
  • Leaky bucket for smoothing.
  • Fixed window for simple limits.
  • Sliding window for fairer enforcement.
  • Cost-based limits for expensive operations.

Implementation:

  • Enforce at edge for coarse limits.
  • Enforce at app layer for user, tenant, and business-context limits.
  • Return useful 429 responses with retry guidance.
  • Exempt carefully and expire exceptions.
  • Monitor abuse patterns and false positives.

Senior answer: rate limiting should protect scarce resources and sensitive workflows, not just count HTTP requests.

40. How would you handle abusive API clients without blocking legitimate users?

I would use graduated, context-aware controls instead of one global block rule.

Actions:

  • Identify caller by user, tenant, token, IP, device, and client ID.
  • Apply risk-based throttles by endpoint sensitivity.
  • Use soft limits before hard blocks where possible.
  • Challenge suspicious clients with CAPTCHA or step-up auth for human flows.
  • Reduce response detail or disable expensive operations temporarily.
  • Quarantine suspicious API keys and notify owners.
  • Provide self-service dashboards for legitimate high-volume clients.
  • Use allowlisted partner quotas with contracts and monitoring.
  • Keep appeal and support paths for false positives.

Examples:

Abuse Response
Credential stuffing Per-account and per-IP throttles, bot detection, MFA step-up
Scraping Cost-based limits, pagination caps, anomaly detection
Expensive queries Query complexity limits and timeouts
Token misuse Revoke token and rotate client secret

Senior answer: good abuse handling preserves legitimate traffic while increasing friction for risky behavior.

Chunk 5 - Questions 41-50

41. What logs would you collect for API security monitoring?

API logs should support authentication, authorization, abuse detection, incident response, and privacy-safe forensics.

Fields:

  • Timestamp and request ID.
  • Route template, not only raw URL.
  • HTTP method and status code.
  • Authenticated subject and client ID.
  • Tenant ID.
  • Source IP and trusted proxy chain.
  • User agent and device signal.
  • Authorization decision and policy version.
  • Object ID or safe hash.
  • Response size and latency.
  • Rate-limit decision.
  • Error category.
  • Token issuer and audience, not raw token.

Do not log:

  • Passwords.
  • Session cookies.
  • Authorization headers.
  • Raw access tokens.
  • Full PII payloads.
  • Payment data.

Senior answer: API logs should explain who did what to which resource and whether the system allowed it, without becoming a new sensitive-data leak.

42. What are common GraphQL security risks?

GraphQL changes API shape from fixed endpoints to flexible queries, which creates specific risks.

Risks:

  • Deeply nested query denial of service.
  • Expensive queries and resolver fan-out.
  • Batching attacks that bypass per-request rate limits.
  • Introspection exposing schema and hidden fields.
  • Resolver-level BOLA.
  • Excessive data exposure through fields the UI does not show.
  • N+1 query amplification.
  • Missing authorization on nested objects.
  • Error messages revealing internals.

Controls:

  • Query depth and complexity limits.
  • Persisted queries for high-risk APIs.
  • Field-level authorization.
  • Resolver-level object checks.
  • Rate limits by operation cost, not only HTTP request count.
  • Disable or restrict introspection in production where appropriate.
  • Timeout and pagination requirements.

Senior answer: GraphQL security is mostly authorization and resource-control discipline applied at field and resolver granularity.

43. How would you prevent deeply nested GraphQL query abuse?

Deep nesting can cause exponential resolver work and backend load. Prevention requires measuring query cost before execution.

Controls:

  • Maximum depth.
  • Maximum total complexity score.
  • Maximum list size and mandatory pagination.
  • Resolver timeouts.
  • Query allowlisting or persisted queries.
  • Limit batching.
  • Rate limit by query cost per user or token.
  • Cache safe repeated queries.
  • Monitor slow resolvers and backend fan-out.
flowchart TD
	A[GraphQL query] --> B[Parse]
	B --> C[Depth and cost analysis]
	C --> D{Within budget?}
	D -- Yes --> E[Execute]
	D -- No --> F[Reject]

Senior answer: request count is the wrong unit for GraphQL abuse. Complexity and resolver cost are the units that matter.

44. How would you prevent excessive data exposure in APIs?

Excessive data exposure happens when APIs return fields the caller does not need or should not see, even if the UI ignores them.

Controls:

  • Define response schemas per operation.
  • Use DTOs instead of returning database models.
  • Apply field-level authorization.
  • Redact sensitive fields by default.
  • Avoid generic includeAll=true or debug fields.
  • Enforce pagination and response-size limits.
  • Review mobile and GraphQL responses for hidden fields.
  • Test responses with lower-privilege users.

Examples of fields that should not leak:

  • Password hashes.
  • Internal risk scores.
  • Admin notes.
  • Billing provider IDs.
  • Tenant IDs not needed by caller.
  • Feature flags exposing upcoming launches.

Senior answer: the API response is the security boundary, not the UI. If the API returns it, an attacker can read it.

45. What is request smuggling?

Request smuggling exploits inconsistencies between front-end and back-end HTTP parsers. An attacker crafts a request that the proxy and backend disagree about, causing hidden or prefixed requests to be processed unexpectedly.

Common root causes:

  • Conflicting Content-Length and Transfer-Encoding headers.
  • Different handling of chunked encoding.
  • Proxy normalizes one way while backend parses another.
  • HTTP/2 to HTTP/1 conversion inconsistencies.

Impact:

  • Bypass front-end security controls.
  • Poison another user's request.
  • Access internal endpoints.
  • Cache poisoning.
  • Session hijacking in some conditions.

Controls:

  • Use well-maintained proxies and backends.
  • Normalize and reject ambiguous requests at the edge.
  • Avoid unsupported HTTP downgrade paths.
  • Patch request parsing components.
  • Run targeted request-smuggling tests against proxy/backend combinations.

Senior answer: request smuggling is a parser differential bug across trust boundaries. The fix is strict edge normalization and consistent protocol handling.

46. How would you investigate suspected web exploitation?

I would investigate by preserving evidence, scoping affected users and endpoints, identifying exploit path, and containing active abuse.

Steps:

  • Identify signal: WAF alert, error spike, suspicious API pattern, customer report, or EDR alert.
  • Preserve load balancer, WAF, application, API gateway, auth, and database logs.
  • Build a timeline by request ID, IP, user, route, and object.
  • Look for payloads: SSRF URLs, XSS markers, SQL syntax, path traversal, unusual GraphQL queries.
  • Determine whether exploit succeeded, not just whether payload was attempted.
  • Scope data accessed or modified.
  • Contain by disabling feature, blocking pattern, rotating secrets, or patching code.
  • Add tests and detections for recurrence.

Evidence:

  • Request and response metadata.
  • Application errors and stack traces.
  • Auth and session events.
  • Downstream cloud API calls for SSRF.
  • Database query logs where available.

Senior answer: attempted web exploitation is common noise; the investigation must prove impact and blast radius.

47. How would you design security testing for web applications?

I would combine automated testing, manual review, threat modeling, and production feedback.

Testing layers:

  • Secure code review for auth, authorization, injection, and deserialization paths.
  • SAST for dangerous APIs and framework misuse.
  • DAST for web-facing vulnerabilities.
  • Dependency and container scanning.
  • API schema testing for auth and field exposure.
  • Fuzzing parsers, upload handlers, and request boundaries.
  • Authorization tests for every role and tenant boundary.
  • Browser security tests for XSS, CSP, CORS, CSRF, and cookies.
  • Abuse testing for rate limits and business logic.
  • Bug bounty or red-team exercises for high-risk apps.

Senior answer: web security testing must include business logic and authorization. Scanners find some bugs, but they rarely understand tenant ownership or workflow abuse.

48. What is the role of WAF and what are its limitations?

A WAF, or Web Application Firewall, filters HTTP traffic for known attack patterns and can provide virtual patching and abuse controls. It is defense-in-depth, not the primary security model.

Useful for:

  • Blocking known exploit payloads.
  • Rate limiting and bot controls.
  • Emergency mitigation before code fix.
  • Protecting legacy apps temporarily.
  • Logging suspicious traffic.

Limitations:

  • Cannot reliably fix broken authorization.
  • Can be bypassed with encoding or parser differences.
  • May not understand GraphQL or custom business logic.
  • Can create false positives.
  • Does not replace secure coding, patching, or tests.
  • Cannot protect internal paths it does not see.

Senior answer: WAF is a seatbelt, not steering. You still need to fix the vulnerable code and design.

49. What are common web security anti-patterns?

Common anti-patterns:

  • Trusting the UI to enforce permissions.
  • Building SQL or shell commands with string concatenation.
  • Using denylist-only SSRF filters.
  • Returning full database models from APIs.
  • Storing tokens in localStorage when HttpOnly cookies are viable.
  • CORS reflecting arbitrary origins.
  • Uploading files directly to trusted web origins.
  • Relying on UUIDs instead of object authorization.
  • No rate limits on login, search, export, or GraphQL.
  • Debug endpoints or Swagger docs exposed publicly.
  • Treating WAF findings as proof the app is secure.
  • Missing logs for authz decisions.

Senior answer: most anti-patterns come from trusting the wrong layer: browser, gateway, UI, filename, object ID, or WAF.

50. What questions would you ask in a web/API security design review?

Top questions:

  1. What are the trust boundaries between browser, API, services, and data stores?
  2. How is authentication performed and how are tokens or cookies protected?
  3. Where is authorization enforced for functions, objects, tenants, and fields?
  4. Which inputs reach interpreters such as SQL, shell, templates, URLs, or file parsers?
  5. Does any feature fetch attacker-supplied URLs or process attacker-supplied files?
  6. What CORS, CSP, CSRF, and cookie policies apply?
  7. How are rate limits and abuse controls designed by user, tenant, IP, and client?
  8. What logs prove allowed and denied access decisions?
  9. What security tests cover direct API calls and cross-tenant access?
  10. What happens when a key dependency such as WAF, proxy, scanner, or IdP fails?

Senior closing: web/API security review should trace attacker-controlled data from entry point to sink and trace authenticated identity from login to every authorization decision.
