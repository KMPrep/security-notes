enterprise data protection Authentication Core - Cheatsheet

Source: questions/answers/authentication-core-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Design a secure authentication system for a global enterprise application.

  • Centralize identity in an IdP, enforce close to the app; classify populations first (employees, contractors, customers, service accounts, admins) - each needs its own assurance + recovery model.
  • Layered design: IdP (lifecycle, MFA, federation, risk) -> auth service (orchestration, token issuance, recovery) -> gateway (TLS, bot, abuse) -> services (session, authz, step-up, audit) -> telemetry (detection).
  • Passwords with Argon2id/bcrypt/scrypt, never plain SHA-256; MFA by risk tier (passkeys/hardware keys for admins, SMS only as low-assurance fallback).
  • Sessions: CSPRNG tokens, secure cookies, rotate after login/privilege change, revocable. Recovery = privileged workflow (cooldown, proofing, notification, cleanup).
  • Workloads: short-lived identity (IAM roles, STS, OIDC federation, SPIFFE), never static keys.
  • Tradeoff: strict MFA + short sessions hurt global/mobile/IR availability - balance with risk tiers, backup authenticators, break-glass, observability.

2. Why is FIDO2/WebAuthn more phishing-resistant than OTP-based MFA?

  • WebAuthn = public-key crypto + origin binding; authenticator signs a challenge only for the legitimate relying-party ID.
  • OTP is a short-lived but reusable/relayable secret the user types - a real-time phishing proxy captures and forwards it before expiry.
  • A passkey registered for login.company.com produces no valid signature for a look-alike domain.
  • Residual risk: weak recovery (SMS, helpdesk, email) bypasses WebAuthn entirely; synced passkeys add platform/cloud-keychain trust.
  • Privileged accounts: require hardware-backed or managed-device-bound authenticators + hardened recovery.

3. Compare SMS OTP, TOTP, push MFA, hardware keys, and passkeys.

  • SMS = weakest (SIM swap, SS7, phishing, telecom dependence) - fallback only.
  • TOTP = medium; no telecom dependence but still phishable (user types code into fake page).
  • Push = medium; easy UX but MFA-fatigue/consent-phishing - number matching helps, still not phishing-resistant.
  • Hardware key = very high; origin-bound, non-exportable private key - best for admins/break-glass.
  • Passkey = high; phishing-resistant + good UX, but understand device-bound vs synced and recovery trust.
  • Policy: admins = hardware/managed passkey + backup; SMS disabled for privileged access.

4. How would you prevent MFA fatigue attacks?

  • Attacker already has the password and spams push prompts until the user taps approve.
  • Number matching + rich context (app, location, device, time) so users can't blindly approve.
  • Rate-limit/suppress prompts after repeated denials; add a visible "Report fraud" action.
  • Risk-based blocking (impossible travel, anonymizing proxy, new device, breached password) + notify via trusted channel.
  • Track metrics: prompts per login, denial rate, password-success-then-MFA-failure.
  • Real fix for high-risk groups: remove push, require FIDO2/passkeys (number matching is still relay-able).

5. What are the common failure modes of push-based MFA?

  • Depends on user judgment under pressure: push fatigue, blind approval, prompt ambiguity, notification overload.
  • Phishing proxy relays the challenge in real time; device compromise abuses the enrolled phone.
  • Weak enrollment lets attacker add their own phone; weak helpdesk recovery resets MFA on low proof.
  • Controls: number matching, rich context, rate limit/suppress, harden enrollment + recovery, prefer FIDO2.
  • Vague "Approve sign-in" is weaker than "Approve access to Payroll Admin from Chrome on Windows near Seattle."

6. How would you design authentication for both managed and BYOD devices?

  • Separate user assurance from device assurance, then combine in conditional access.
  • Managed: enforce posture (inventory, disk encryption, EDR, patch level, device cert, remote wipe).
  • BYOD: assume untrusted - browser-only access, stronger user auth, short sessions, limited download, step-up for sensitive actions.
  • Unknown device = step-up + restrict until risk understood.
  • Device trust is not binary: managed devices get compromised, BYOD can be the real user - posture is a signal, not a substitute for strong auth/authz.

7. How would you secure account recovery for high-value enterprise accounts?

  • Treat recovery as a privileged workflow - it can transfer account ownership; if weaker than login, attackers attack recovery.
  • Pre-enrolled backup FIDO2/recovery key; no SMS-only recovery.
  • Approval separation: helpdesk cannot unilaterally recover a privileged account (manager + security approval).
  • Short-lived single-use temporary access passes; mandatory strong-factor re-enrollment after recovery.
  • Revoke old sessions, refresh tokens, remembered devices, OAuth grants, app passwords; out-of-band notification.
  • Design break-glass before the incident; availability is the tradeoff to manage.

8. Why is account recovery often weaker than login?

  • Orgs optimize recovery for availability/support cost; attackers optimize for the easiest path in.
  • Weak channels: stale enrolled email/SMS, KBA from breached data, urgency/exec-impersonation pressure on helpdesk.
  • Backup codes screenshotted/emailed; incomplete post-recovery cleanup leaves attacker sessions live.
  • "Recovery is the real assurance floor" - login is only as strong as the easiest alternate path that replaces its factors.
  • Fix: risk-based recovery (stronger proof + approval separation + temp access + audit for privileged), always notify + clean up.

9. What risks remain after a user successfully completes MFA?

  • MFA proves credential possession at login, not safety of every later action.
  • Session hijacking (stolen cookie, malware, XSS), authz bugs (IDOR, tenant isolation, privesc), endpoint compromise acting as the user.
  • OAuth consent abuse, post-login factor/recovery enrollment, insider misuse, long-lived refresh tokens.
  • Controls after MFA: secure cookies, session rotation, idle/absolute timeouts, step-up for sensitive actions, strong authz, detection, global revoke.
  • Phrase: "MFA is an authentication control, not a complete account security model."

10. How would you detect credential stuffing at global scale?

  • Attacker uses leaked user/pass pairs, distributed across many IPs, few passwords per account - per-IP lockout fails.
  • Signals: failure rate by account/IP/ASN, breached-credential match (without logging passwords), automation fingerprints, geo velocity, success-after-failures, MFA spikes.
  • Build baselines by region/tenant/app/account type; adaptive throttling + bot defenses; screen passwords vs breach intel.
  • Alert on success after distributed failures, especially if followed by MFA reset, email change, key creation, or export.
  • Graduated response: block automation, step up, force reset for confirmed reuse, revoke sessions, notify without revealing account existence.

Chunk 2 - Questions 11-20

11. How would you detect password spraying?

  • Few common passwords across many accounts (1-2 attempts each) to dodge per-account lockout - detect across the population, not one user/IP.
  • Signals: many usernames with one failure, failures across many IPs, shared automation fingerprint, attempts on stale accounts, attempts timed just below lockout.
  • Rule shape: "1-2 failures across hundreds of accounts sharing infra/timing," not "5 failures for one user."
  • Response: adaptive throttling, step-up/block before password validation, reset accounts with password-success-then-MFA-fail, protect stale accounts, avoid hard lockouts (DoS risk).
  • Implement at app/identity layer (low-and-slow); edge WAF alone misses it.

12. How would you design risk-based authentication?

  • Use context to decide: allow / step up / limit / block / investigate - apply friction only when risk rises.
  • Inputs: user history, device posture, network context (IP rep, VPN, Tor, impossible travel), account sensitivity, behavior, threat intel.
  • Outputs explicit and mapped to actions (e.g., new device + sensitive action = step up; malicious IP + weak factor = block).
  • Must be explainable + privacy-aware; allowlist high-confidence signals; safe fallback for travelers/VPN/remote users.

13. What signals would you use to step up authentication?

  • Trigger when current session assurance is insufficient for the action/context.
  • Action-based: password/MFA/recovery change, view recovery codes, create API keys, export data, change payroll/billing, assume admin, disable logs, modify IAM.
  • Contextual: new device/browser, unmanaged/unhealthy device, new country/impossible travel/risky ASN, recently changed recovery channel.
  • Behavioral: download spike, unusual data access, abnormal admin sequence, repeated denied authz, new OAuth grant.
  • Match factor to risk; for confirmed compromise, block + investigate instead of stepping up.

14. How would you protect against session hijacking?

  • Hijacked token bypasses password + MFA (user already authenticated).
  • Prevent: TLS + HSTS, Secure/HttpOnly/SameSite cookies, never tokens in URLs/logs/localStorage, XSS defense (CSP), rotate IDs after login/privilege, short lifetimes.
  • Detect: same session from two distant geos, new device/UA for existing session, replay after logout/reset, sensitive action after attribute change.
  • Respond: revoke session + refresh tokens, force reauth, invalidate remembered devices, preserve evidence.
  • Residual: full endpoint compromise = attacker acts through the live browser - needs EDR, transaction confirmation, authz.

15. What makes a session token secure?

  • Properties: high entropy (CSPRNG), unpredictable (no usernames/timestamps), confidential, integrity-protected, scoped, time-limited, rotatable, revocable.
  • Prefer opaque server-side session IDs for high-risk apps (easy revocation); JWT sessions are hard to revoke unless short-lived + versioned + introspection.
  • Cookie flags: Secure, HttpOnly, SameSite, Path, Domain.
  • Bad: session=alice-1700000000-admin. Good: 256-bit random + server record (user, auth time, assurance, expiry, device, revocation state).
  • Token isn't the only control - validate policy on every meaningful request.

16. How should session tokens be generated, stored, rotated, and revoked?

  • Lifecycle, not one-time artifact: Created -> Active -> Rotated -> Expired/Revoked.
  • Generate: CSPRNG, >=128 bits (often 256 for opaque), no predictable structure.
  • Store: browser = Secure/HttpOnly/SameSite cookie; mobile = Keychain/Keystore; server = hashed value + metadata.
  • Rotate: after login/MFA/recovery/reset/role-elevation/tenant-switch; rotate refresh tokens on every use, detect reuse as theft.
  • Revoke: on logout/password change/MFA reset/compromise/disablement; central session store or revocation index for distributed systems.
  • Tradeoff: stateless scales but is hard to revoke instantly - stateful/hybrid better for IR.

17. How would you handle logout in a distributed authentication system?

  • Define what logout means: local (app session) vs global (IdP + all RPs) vs security (revoke sessions + refresh tokens + remembered devices + grants).
  • Server-side session registry per user/device; revoke refresh tokens + rotate token-family IDs; publish revocation via reliable event bus.
  • High-risk services check revocation state before sensitive actions; clear cookies client-side but don't rely on it.
  • Keep access tokens short-lived if instant revocation isn't guaranteed.
  • OIDC/SAML front-channel logout is imperfect (3rd-party cookies, mobile, cached tokens) - central revocation + short lifetimes matter more.

18. How would you invalidate all sessions after suspected compromise?

  • IR action: remove attacker access WITHOUT destroying evidence.
  • Steps: identify subject -> preserve evidence (hashed session IDs, IPs, UAs, token issue times, actions) -> revoke sessions -> revoke refresh + remembered-device tokens -> invalidate OAuth grants/API keys/PATs -> reset credential -> require phishing-resistant reauth -> notify -> monitor replay.
  • AWS admin: review CloudTrail, revoke console sessions, disable/rotate access keys, inspect IAM changes, require hardware-backed sign-in.
  • Critical: revocation must not depend on best-effort messaging - services need authoritative revocation check or short-lived tokens, else logout is eventual.

19. What is the difference between authentication assurance and identity assurance?

  • Authentication assurance = confidence the session is controlled by whoever enrolled the credential (password, passkey, FIDO2, MFA).
  • Identity assurance = confidence the account maps to the correct real-world person/workload (HR record, ID proofing, sponsorship, KYC, device issuance).
  • High-auth passkey login can still have low identity assurance (account made from unverified email), and vice versa.
  • Phrase: don't confuse "strong login" with "known person" - data protection + privileged access need both.

20. How would you design authentication for privileged administrators?

  • Higher assurance: separate admin from daily accounts, phishing-resistant MFA (FIDO2/managed passkey), >=2 strong authenticators.
  • Privileged access workstations / hardened managed devices; block unmanaged devices for admin consoles.
  • Just-in-time elevation (no standing admin), approval/ticket binding, session rotation + step-up for critical actions.
  • Monitor admin login, role activation, policy/logging changes, credential creation; governed break-glass with hardware keys + alerting.
  • Recovery stricter than normal users (manager + security approval, temp access, post-recovery review). Pair auth with least privilege + SoD + change control + detection.

Chunk 3 - Questions 21-30

21. How would you design authentication for service accounts?

  • Goal: avoid long-lived shared secrets; give each workload a narrowly scoped identity.
  • One service account per workload/environment/trust boundary; least privilege; strong provenance (issuance depends on where workload runs, not just a file).
  • No shared accounts across teams; no interactive login; separate dev/staging/prod; central inventory (owner, purpose, perms, last used, rotation).
  • AWS: prefer IAM roles (EC2/ECS/EKS IRSA/Lambda), OIDC federation, IAM Roles Anywhere, STS AssumeRole w/ external ID - not static access keys.
  • Common mistakes: one powerful "automation" account, static unowned keys, creds in code/images, humans using SA to bypass MFA, *:* policies.

22. How would you design authentication for machine-to-machine workloads?

  • Prove identity AND provenance: "issued to the expected service in the expected environment."
  • Patterns: mTLS workload certs, cloud workload identity, OAuth2 client credentials, SPIFFE/SPIRE; avoid static API keys.
  • Each workload unique identity; short-lived auto-rotated creds; issuance from attested runtime context (cluster, namespace, instance identity).
  • Downstream validates issuer/audience/expiry/subject + authz; fail closed when identity unverifiable.
  • AWS: GitHub Actions OIDC -> AssumeRole (trust policy restricts repo/branch/audience/subject); EKS IRSA/Pod Identity. Biggest residual risk = over-permission.

23. How would you securely store passwords?

  • Salted slow hash; never plaintext or reversible encryption - verifier only checks match.
  • Argon2id/bcrypt/scrypt with current cost; PBKDF2 only w/ high iterations in compliance contexts.
  • Unique random salt per password; store algo + salt + params + hash together for migration; optional pepper in KMS/HSM.
  • Constant-time compare; block breached passwords at registration/reset; never log passwords/hashes/reset tokens.
  • Goal = damage reduction after breach (force expensive offline guessing), not pretending DB never leaks.

24. Compare bcrypt, scrypt, Argon2, and PBKDF2.

  • Argon2id: modern, memory-hard, best default when libraries mature.
  • scrypt: memory-hard, good when Argon2id unavailable; parameter tuning tricky.
  • bcrypt: battle-tested, intentionally slow, NOT memory-hard, ~72-byte input limit - acceptable legacy.
  • PBKDF2: standardized/compliance-friendly but CPU-hard only (weak vs GPU) - use high iterations.
  • No hash saves users from breached passwords under online guessing; slow hashing mainly protects offline cracking after DB leak.

25. Why is SHA-256 not enough for password storage?

  • SHA-256 is fast by design = wrong property; enables massive offline GPU/ASIC guessing.
  • Unsalted -> rainbow tables; identical passwords -> identical hashes; no tunable work factor.
  • Fix: Argon2id(password, unique_salt, memory_cost, time_cost) changes attacker economics.
  • SHA-256 may appear inside a larger approved construction but alone is not a password scheme (no salt, no cost, no memory hardness).

26. What is salting and why is it required?

  • Salt = unique random per-password value, stored with the hash, NOT secret.
  • Makes identical passwords produce different verifiers; defeats rainbow tables; forces per-user guessing; hides reuse patterns.
  • Requirements: unique per password, random, CSPRNG, stored with hash, never a global system-wide value.
  • Salt != pepper (pepper is secret + system-wide). Salting is mandatory even with pepper; must pair with slow hashing.

27. What is peppering and where should the pepper be stored?

  • Pepper = secret value added to hashing, stored SEPARATELY from the password DB (KMS/HSM/secret manager).
  • Benefit: DB-only leak still lacks the pepper; adds a secret boundary; can block offline cracking.
  • Risks: losing it breaks all verification; rotation hard (don't know plaintext); app + DB compromise exposes it too.
  • Recommendation: salting + slow hashing first; pepper is additional; version peppers for rotation. Never an excuse for weak hashing.

28. How would you migrate users from weak to strong password hashes?

  • Can't read plaintext - migrate at login or reset.
  • On successful legacy-hash login: verify with old algo, then rehash with Argon2id + unique salt + current params, store new verifier.
  • Add hash metadata (algo, version, salt, cost, pepper version); support inactive-account forced reset; reset high-risk accounts sooner.
  • Keep login protections during migration; apply breached-password screening; avoid bulk resets (phishing/training risk).
  • User shouldn't notice (e.g., SHA-1 -> Argon2id silently on next login).

29. How would you prevent username enumeration in login flows?

  • Generic errors ("Invalid username or password"); consistent timing + HTTP status for valid/invalid users.
  • Don't reveal MFA enrollment before primary credential check; reset = "If an account exists, we sent instructions."
  • Watch side channels: error text, timing (slow hash for real user vs fast for missing), MFA prompt, reset UI, signup "email already registered," rate-limit differences.
  • Timing mitigation: fake hashing for unknown users + rate limiting + bot controls.
  • Tradeoff: consumer signup may intentionally reveal existence; enterprise/high-risk prioritizes generic flows.

30. What error messages should a login system return?

  • External: ambiguous ("We could not sign you in"); internal: precise reason codes for security/support.
  • Reset: "If an account exists for that address, we sent recovery instructions."
  • Avoid oracles: "No account found," "Password is wrong," "MFA not configured," "blocked from IP x.x.x.x."
  • Separate user-facing ambiguity from internal observability; log detailed reasons, give attackers nothing.
  • Exception: after strong auth (inside recovery), more specific guidance is OK; pre-auth errors stay generic.

Chunk 4 - Questions 31-40

31. How would you rate-limit login attempts without locking out legitimate users?

  • Naive "5 failures = lockout" is a DoS tool against execs/departments - use layered adaptive throttling.
  • Dimensions: per account, per IP/subnet, per ASN/cloud, per device fingerprint, per tenant, per spray pattern, per credential reputation.
  • User-friendly: progressive delays not hard lockout, step-up after risky password success, owner recovery via strong proof, notify without confirming existence.
  • Rate limiting belongs BOTH at edge (WAF) and in the identity layer (knows accounts/tenants/outcomes).

32. How would you design bot protection for authentication endpoints?

  • Threats: credential stuffing, spraying, fake signup, OTP abuse, MFA bombing, enumeration.
  • Combine edge (rate limit by IP/ASN/geo/reputation, bot fingerprinting), app telemetry (account/tenant/credential behavior), adaptive challenges.
  • CAPTCHA is NOT the primary boundary (solving services, residential proxies, real browsers; hurts accessibility) - friction only.
  • Protect downstream cost: quotas + anomaly detection on SMS/email/push delivery, not just password attempts.
  • AWS: CloudFront + WAF Bot Control + rate rules, pass signals to app for deeper detection.

33. How would you detect impossible travel logins?

  • distance / time = required speed; unrealistic -> raise risk.
  • Data: UTC timestamp, source IP/geo, user/tenant, device, factor strength, network type, new vs existing session.
  • False positives common: VPN egress, mobile carrier geo, corporate proxy, cloud browsers, stale geo DBs - so it's a risk SIGNAL, not standalone proof.
  • Combine with device novelty, MFA strength, IP reputation, recent failures, action sensitivity.
  • Privileged: impossible-travel console login + role assumption/IAM change = high severity.

34. What would you log for authentication events?

  • Support detection, IR, forensics, abuse analysis, compliance - reconstruct decisions without storing secrets.
  • Core fields: event type, user/tenant ID, UTC time, source IP/ASN/geo/reputation, UA/device, auth method + assurance, MFA factor/result, session ID hash (not raw), risk score + policy decision + reason codes, app/client/scopes, admin actor, correlation ID.
  • Key security events: password-success-then-MFA-fail, MFA reset/enrollment, recovery channel change, new device trust, unusual location, admin role activation, refresh token reuse.
  • AWS: central log account / security lake, immutability, restricted access, correlate w/ CloudTrail/VPC Flow/WAF/EDR/IdP.

35. What should not be logged during authentication?

  • Never: plaintext passwords, hashes/salts, OTP/backup/recovery codes, magic links, session/refresh/access/ID tokens, auth codes, cookies, security answers, private keys/secrets/API keys, raw biometrics, full PII/PAN/gov ID.
  • Safer: token hash, user ID or masked email, internal reason category instead of raw password.
  • Common failure: temporary debug logging of request bodies captures passwords/tokens - need redaction middleware + tests that fail on sensitive fields.
  • Separate security logs from product analytics; govern + audit access to both.

36. How would you protect authentication logs from tampering?

  • Append-only/immutable destination; ship out of prod account fast to a separate security-team-owned account.
  • Restrict who can modify pipelines/retention/encryption keys; alert on logging disablement, pipeline failure, retention/policy changes.
  • Integrity: sequence numbers, signatures, hash chaining; replicate; monitor volume gaps.
  • AWS: CloudTrail + app logs -> dedicated log-archive account, S3 Object Lock, KMS keys owned by security, alert on StopLogging/trail deletion/bucket-policy/KMS changes; apps write but cannot delete/rewrite.

37. How would you design secure authentication for mobile apps?

  • Untrusted networks, device loss, malware, root/jailbreak, app tampering, token storage - mobile app is NOT a confidential client.
  • OAuth2 Authorization Code + PKCE; system browser not embedded webview; passkeys/platform authenticators where supported.
  • Tokens in Keychain/Keystore (not prefs/files); short-lived access + refresh rotation; bind refresh to device instance.
  • Detect root/jailbreak/tamper/emulator as risk signals; biometric/PIN to unlock secrets; remote logout/revocation; no tokens in crash reports.
  • Embedded client secrets aren't secret (binary inspectable) - public clients use PKCE + server-side validation.

38. How would you design secure authentication for browser-based apps?

  • Handle XSS, CSRF, token storage, phishing, 3rd-party scripts, session theft.
  • Authorization Code + PKCE; prefer Backend-for-Frontend so tokens stay server-side; session ID in Secure/HttpOnly/SameSite cookie.
  • Avoid long-lived tokens in localStorage (XSS-readable); rotate session after login/MFA/privilege; CSRF protection for cookie-auth state changes.
  • Strong CSP + XSS prevention (XSS acts as the user even with HttpOnly); strict redirect URI validation; validate iss/aud/nonce/state/exp/signature.
  • BFF: backend exchanges code, holds tokens, sends browser only an opaque session cookie.

39. How would you handle remembered devices securely?

  • Useful but dangerous if it becomes a permanent silent MFA bypass.
  • Issue high-entropy token only after strong MFA; Secure/HttpOnly cookie or secure storage; bind to user/tenant/device/browser/risk; store hashed server-side.
  • Expire + rotate; revoke on password/MFA reset, recovery, suspicious login, device loss, user request; show devices in settings; notify on new remembered device.
  • Don't let remembered status authorize sensitive actions; ignore it on new location/risky source.
  • Risk-based window: longer for managed compliant laptop, restricted/avoided for admins.

40. What are the security risks of 'keep me signed in'?

  • Extends session lifetime -> increases value + duration of stolen sessions/refresh tokens/remembered cookies.
  • Risks: stolen device long access, malware/extension token theft, shared/public computers, durable refresh-token theft, password reset not revoking persistent sessions, weakened MFA freshness.
  • Controls: disable on shared/unmanaged/high-risk devices, shorter for privileged, refresh rotation + reuse detection, step-up for sensitive actions, user-visible session revocation, revoke after password/MFA/recovery change.
  • Persistent sign-in converts auth from momentary proof into a longer-lived bearer credential - needs rotation, revocation, device awareness, sensitive-action reauth.

Chunk 5 - Questions 41-50

41. What are the security risks of biometric authentication?

  • Biometrics unlock a local device-bound key; the biometric itself should NOT be sent to the server.
  • Risks: can't rotate if compromised, false accept/reject, presentation attacks, device compromise bypass, privacy/legal sensitivity, coercion (face/finger usable under pressure), weak enrollment.
  • Good design: templates local in secure hardware, biometric gates a device-bound private key, fallback to PIN/strong recovery, no raw biometrics on app servers.
  • Phrase: biometrics = local unlock factor for cryptographic credentials (e.g., Face ID unlocking a passkey -> server gets a WebAuthn signature, not a face scan).

42. How do passkeys change the enterprise authentication model?

  • Move from shared secrets to public-key auth: server stores public key, authenticator holds private key + signs origin-bound challenges.
  • Credential stuffing becomes useless; phishing resistance improves; UX improves; recovery + device lifecycle become the central problems.
  • Decide synced vs device-bound per risk tier; attestation/authenticator policy matters for privileged users.
  • Doesn't remove session security/authz/logging/recovery needs - SMS recovery or stolen post-login session still bites.
  • AWS: passkey to corporate IdP -> federate into IAM Identity Center; permission sets/SCPs/CloudTrail still govern actions.

43. What is the role of device binding in authentication?

  • Ties auth/session to a specific device or device-held key; distinguishes legit enrolled environment from attacker with only a password/token.
  • Forms: device-bound passkeys/hardware keys, client certs/mTLS, mobile instance keys, MDM device certs, proof-of-possession tokens.
  • Benefits: reduces value of stolen passwords/tokens, supports conditional access, detects new-device risk.
  • Tradeoffs: device loss recovery pressure, harder BYOD, re-imaging breaks access, binding to weak identifiers (UA/IP) = false confidence; compromised device still acts as user.
  • Treat as one layer; combine with user verification, compliance, session management, authz.

44. How would you prevent phishing proxy attacks?

  • Proxy relays victim interaction to the real site in real time - captures password, OTP, sometimes session cookie; TOTP/number-matching push fail (codes/numbers are portable/relayable).
  • Best control: phishing-resistant auth (FIDO2/WebAuthn/passkeys/smart cards/device certs) - origin binding means no valid signature for evil-login.example.
  • Also: reduce cookie theft (secure cookies, short lifetimes, anomaly detection), step-up for high-risk actions, transaction binding, harden OAuth redirect/state/nonce.
  • Residual: proxy may still steal a session cookie after real-site auth - strongest mitigation is preventing the proxy from completing authentication at all.

45. How would you protect high-risk actions after login?

  • Initial login may be old/low-assurance/pre-risk - classify actions by impact, require fresh assurance for sensitive ops.
  • High-risk: password/recovery/MFA change, view recovery codes, create API keys/OAuth apps, export data, change payroll/billing, assume admin, access prod secrets, disable logging, change IAM.
  • Controls: fresh reauth in short window, phishing-resistant factor, transaction details in prompt (not generic), risk checks at action time, object/tenant/role authz, approval for irreversible changes, notifications, audit + alerts.
  • Example: creating an AWS access key / changing a KMS key policy = recent strong auth + authz + logging + alerting (JIT approval for sensitive roles).

46. How would you design re-authentication for sensitive actions?

  • Prove fresh intent + adequate assurance for the SPECIFIC action - not a generic nuisance prompt.
  • Track auth time/factor/assurance/session risk; require re-auth when stale or insufficient; phishing-resistant for high-risk.
  • Transaction binding: "Approve adding security key ending 1234" / "Approve export of customer dataset" - reduces confused approval, better audit.
  • Rotate/upgrade session after step-up; log + bind re-auth to the action.
  • On repeated step-up failure: raise risk, suppress prompts, possibly revoke - for suspected compromise, contain, don't re-prompt.

47. How would you handle authentication for contractors and temporary workers?

  • Strong auth + tighter lifecycle (time-bound, sponsor-dependent, variable).
  • Named identity per contractor (no shared accounts), sponsor/manager ownership, start/end dates at provisioning, federate from partner IdP but enforce local risk policy.
  • MFA (phishing-resistant for sensitive), restrict by role/project/tenant/device/time, JIT for privileged, shorter sessions, restrict BYOD for sensitive data, frequent recert, auto-disable at contract end.
  • AWS: federated identities with time-bound permission sets (not long-lived IAM users), sponsor approval, short sessions, CloudTrail, auto-removal on HR/vendor expiry.
  • Key point: lifecycle (proofing, sponsorship, expiry, device policy, deprovisioning), not just login strength.

48. How would you deprovision authentication access when employees leave?

  • Event-driven from authoritative HR/identity source; revoke BOTH future login AND existing authenticated state.
  • Steps: HRIS event -> disable IdP account -> revoke sessions + refresh tokens -> remove MFA + remembered devices -> revoke PATs/API keys/app passwords/SSH/OAuth grants -> remove group/role memberships -> transfer shared resources -> SCIM/SaaS deprovision -> reconcile.
  • Special cases: involuntary term = immediate revoke before notify; privileged = review activity + rotate secrets; contractors auto-expire; SA owned by employee must be transferred not deleted.
  • AWS: remove from Identity Center groups, revoke SSO sessions, review CloudTrail, remove access keys, rotate accessible shared secrets.
  • Metrics: time to disable, time to revoke sessions, orphaned accounts, failed downstream tasks, privileged access remaining.

49. How would you test an authentication system during a security review?

  • Don't just test the happy-path login screen - cover design, implementation, abuse cases, operations.
  • Areas: architecture/trust boundaries, credential storage, MFA enrollment/enforcement, recovery/reset, session lifecycle, token validation/revocation, rate limiting/bot protection, enumeration, logging/monitoring, admin/support workflows, mobile/browser storage, deprovisioning.
  • Sample tests: verify Argon2id params + unique salts, compare error/timing for valid/invalid users, MFA reset/enrollment abuse, session fixation/rotation/logout, weak helpdesk/email recovery bypass, token iss/aud/exp/nonce/state, spraying/stuffing/low-rate, log redaction.
  • Run tabletop scenarios: stolen password, stolen cookie, lost hardware key, compromised helpdesk, contractor term failure, OAuth token theft - verify prevent + detect + respond.

50. What are the top authentication mistakes you would look for in a design review?

  • MFA as a checkbox (SMS/TOTP/push for admins); strong login but weak recovery; fast-hash/reused-salt/reversible password storage.
  • No stuffing/spraying/enumeration protection; tokens in unsafe storage or never rotated; logout that clears browser but not server sessions/refresh tokens.
  • Long-lived bearer tokens with no revocation; weak factor enrollment (attacker adds own MFA); no step-up; overly broad service accounts/static creds.
  • Logs missing critical events or containing secrets; admins = normal users; BYOD = managed; contractors with no expiry; deprovisioning that leaves sessions/keys/SaaS active.
  • Core framing questions: strongest login path? weakest recovery path? what after a stolen session? can we revoke fast? are admins/SAs treated as high blast radius? do logs prove events without leaking secrets? does it fail safely under abuse?
  • The worst mistake is usually inconsistency - strong auth in one path, weak bypass in recovery/sessions/service accounts/support.
