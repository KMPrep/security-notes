enterprise data protection Authorization and IAM Models - Cheatsheet

Source: questions/answers/authorization-iam-models-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Explain the difference between authentication and authorization using an enterprise example.

  • AuthN = "Who are you?"; AuthZ = "What can you do?" - design and test separately.
  • Valid SSO session/JWT proves identity, NOT permission; app must still check role/team/region/ownership/sensitivity/action per request.
  • AWS: IAM Identity Center proves you're Alice; permission set + IAM policy + SCP + resource policy + conditions decide if she can s3:GetObject a specific object.

2. When does RBAC break down in a large organization?

  • Breaks when role count/scope/complexity outgrows simple bundles: role explosion, overbroad Admin/Manager, hidden inheritance, stale access, cross-tenant scope mistakes, toxic combinations, review fatigue.
  • Works when job functions are stable; struggles when access depends on dynamic context (classification, region, device, relationship, time, tenant, risk).
  • Mitigate: scoped (not global) roles, review effective permissions, add ABAC conditions, ReBAC for sharing, JIT for privileged, automate joiner-mover-leaver.
  • Phrase: RBAC fails from weak governance (ownership/scope/lifecycle), not because the model is bad.

3. Compare RBAC, ABAC, and ReBAC.

  • RBAC = principal has role; best for job functions/admin roles; risk = role explosion/overbroad.
  • ABAC = subject/resource/action/environment attributes; best for context-aware + data sensitivity; risk = bad attributes + complexity.
  • ReBAC = relationship graph; best for sharing/collaboration/hierarchy/ownership; risk = hidden inheritance + graph complexity.
  • Real systems combine: RBAC for admin, ReBAC for sharing, ABAC for classification/device posture.

4. What is Zanzibar-style authorization and why is it useful?

  • Relationship-based (Google Zanzibar): access as tuples like document:roadmap#viewer@user:alice; answers by traversing direct + inherited relations.
  • Good for collaboration: direct sharing, inherited perms, group membership, resource hierarchy, centralized checks, explainability.
  • Challenges: schema is security-critical, revocation freshness, recursive-check cost, cross-tenant edges must be impossible by design.
  • Threat-model the relation schema + consistency model carefully.

5. Design an authorization system for a document-sharing platform.

  • Entities: user, group, tenant, folder, document, link share, roles (owner/editor/commenter/viewer).
  • ReBAC for sharing + RBAC for tenant admin + ABAC for classification/device/region/external policy; enforce tenant boundary in every object key + tuple.
  • Object-level check on every access; folder inheritance with explainability; link shares with expiry/scope/external-domain controls; separate view/download/share/edit/delete/transfer.
  • Step-up for external share/bulk export/ownership transfer; DLP before external access; fast revocation.
  • Failure modes: guessed doc ID, surprise folder inheritance, public link bypass, leftover external user, search index leaks unopenable docs.

6. Design an authorization system for a multi-tenant SaaS application.

  • Top priority = tenant isolation: every request scoped to tenant; no user input selects another tenant's data without authz.
  • Derive tenant from trusted session/token claims (not arbitrary params); validate resource tenant from DB before access; tenant-scoped roles (not accidental global admin).
  • Enforce object-level authz in service code (not just gateway); add DB row-level controls as defense in depth; test cross-tenant systematically; alert on denied cross-tenant + unusual switching.
  • AWS: separate accounts for strong isolation, or strict tenant-aware IAM + data controls for pooled - app authz still checks tenant boundaries.

7. How would you model group membership and inherited permissions?

  • Explicit model: direct membership, nested groups, resource hierarchy, relation inheritance (folder viewer -> document viewer), tenant/project scope, deny/restriction rules.
  • Controls: limit nesting depth, detect cycles, explicit tenant boundaries in IDs, log effective-permission path, provide explain API, invalidate cache on membership change, review high-impact groups.
  • Failure modes: nested group over-grants, removing from one group leaves inherited access via another, cross-tenant group, cache keeps old membership.
  • Needs bounded traversal + clear scope + cycle protection + revocation freshness + human-readable explanations.

8. How would you prevent authorization bypass in REST APIs?

  • Root cause: authenticates user but doesn't check access to the specific object/action.
  • Enforce authz in service layer per endpoint; check ownership/tenant AFTER loading resource; never trust client IDs/tenant/roles/scopes; deny-by-default routes; separate authN middleware from authZ logic.
  • Validate action-level perms (read/write/delete/export/share); allowlist writable fields (mass-assignment); never rely on unguessable IDs alone; test horizontal + vertical privesc; log denies + enumeration.
  • Every endpoint answers: "Can this principal perform this action on this object in this tenant right now?"

9. How would you prevent authorization bypass in GraphQL APIs?

  • One query traverses many types/fields - gateway check insufficient if resolvers return nested objects with different perms.
  • Enforce authz in resolvers/shared data layer; object-level AND field-level checks; limit query depth/breadth/aliases/complexity; don't rely on operation name; batch loaders must not bypass per-object checks.
  • Classic leak: project { users { salary } } passes top-level project check but nested salary field lacks authz.
  • Enforce at object + field resolver level, not just endpoint.

10. What is IDOR/BOLA and how do you detect it?

  • IDOR/BOLA = change an identifier to access another object because server skips object-level authz (e.g., /accounts/1001 -> /accounts/1002).
  • Detect: ID-swap tests across users/tenants, object-level authz unit/integration tests, API fuzzing, logs of sequential 403/404, cross-tenant denies, success-after-many-denies, high deny:allow ratio.
  • Prevention > detection: central object-level checks, data APIs requiring principal+resource context, test every endpoint for horizontal + vertical privesc.

Chunk 2 - Questions 11-20

11. How would you design object-level authorization checks?

  • Core defense against IDOR + tenant isolation failure: can principal do action on THIS object?
  • Flow: authenticate -> load object by ID -> get owner+tenant -> evaluate policy -> allow/deny+log.
  • Inputs: principal, action, object (ID/owner/tenant/sensitivity/state), context (device/location/risk/auth freshness).
  • Mandatory in data/service layer; never trust object IDs without ownership/relation check; check tenant boundary before detailed perms; centralized helper; test A-vs-B and tenant-vs-tenant.
  • GET /documents/123 = "return if user has viewer+ on that doc in correct tenant," not "return if logged in."

12. How would you design field-level authorization checks?

  • Controls sensitive fields inside an accessible object (view profile but not salary/national ID/medical).
  • Classify fields, define per-field/group policy, enforce in serializers/resolvers/projection, don't fetch unauthorized fields, redact consistently, log highly-sensitive field access.
  • Failure modes: full object returned + UI hides client-side, GraphQL nested field unprotected, export ignores policy, search index has sensitive fields, logs/analytics capture them.
  • Must be server-side and consistent across views, APIs, exports, search, analytics, logs.

13. How would you design row-level authorization for sensitive datasets?

  • Controls which records a user sees (analytics, data lakes, CRM, HR, SaaS).
  • Each row has security metadata (tenant/owner/region/classification/project/legal-hold); policies combine subject+row attrs+action+context; enforce near query path; default deny on missing metadata; filter before returning.
  • Use DB row-level security or trusted query service; prevent raw-table bypass; column masking/tokenization; audit bulk + unusual queries; cache carefully (stale attrs over-permit).

14. How would you design delegated access?

  • One actor grants another limited authority (calendar share, support impersonation, OAuth consent, assistant).
  • Require explicit grantor/delegate/resource/action/scope/expiration + consent; least-privilege (not full account sharing); revocable by grantor + admin; audit actor AND effective subject; step-up for high-risk; restrict chaining.
  • Failure modes: over-broad delegate, never expires, logs only original user (hides delegate), re-delegation without approval, support impersonation bypasses consent/monitoring.

15. What is confused deputy and how do you prevent it?

  • Privileged service tricked into using its authority for a less-privileged actor (e.g., export service with broad read fetches a private object for a user who can't read it).
  • Root cause: deputy uses its own broad privilege without binding to caller's authority + intended resource.
  • Prevent: bind authority to caller/audience/target/action, pass caller identity + enforce downstream authz, scoped (not shared broad) service creds, validate caller can access target first, use external IDs/source conditions in cross-account.
  • AWS: SaaS assuming customer roles uses external IDs + tight trust policies so one customer can't trigger action in another's account.

16. How would you handle emergency break-glass access?

  • Emergency access when normal paths fail - must be rare, controlled, monitored, reviewed (not a shared wiki password).
  • Pre-created minimal accounts/roles, phishing-resistant MFA, controlled credential custody, approval/incident workflow, time-limited minimal-perm sessions, immediate alert, full recording, post-use review + rotation, periodic test.
  • AWS: root locked w/ hardware MFA, strict emergency admin role, CloudTrail/Security Hub alert, short session, every use = incident review.
  • Balance: too much friction = unusable in outage; too little = permanent backdoor. Aim for controlled availability + loud monitoring.

17. How would you audit privileged access?

  • Answer: who had access, why, when used, what they did, was it appropriate.
  • Capture: role assignments/changes, JIT requests/approvals/expiry, break-glass use, admin login/step-up, commands/API calls, resource + policy changes, credential creation, logging changes, session duration/device, ticket justification.
  • Centralize in tamper-resistant location protected FROM the audited admins; record actor + effective role; correlate actions w/ approvals; alert on sensitive/off-hours/unusual-location/denied.
  • AWS: CloudTrail -> separate log archive account; alert on IAM/trust changes, key creation, disabling CloudTrail, KMS policy changes, GuardDuty changes, root use.

18. How would you design just-in-time privileged access?

  • Remove standing admin; grant elevated perms only when needed, time-limited, with approval + audit.
  • Eligible users pre-approved; activation requires phishing-resistant reauth + ticket/reason/scope/duration; approval for high-risk; auto-expire; log all elevated actions; risk signals can block; louder/restricted emergency path.
  • AWS: time-limited permission sets via IAM Identity Center/PAM instead of permanent AdministratorAccess; CloudTrail records assumptions + calls.
  • Residual: JIT reduces standing privilege but not unsafe actions - still need least privilege + SoD + change control + detection.

19. What is least privilege and why is it hard to implement?

  • Minimum access for function, for minimum time.
  • Hard because orgs change constantly: team/project moves, new dependencies, emergency grants become permanent, broad-for-speed requests, complex cloud IAM, numerous unowned machine identities, rubber-stamp reviews, removal breaks prod.
  • Controls: scoped roles + resource-level perms, JIT for privileged, expiry on temp grants, measure actual usage, staged removal of unused perms, split service accounts, policy simulation, review effective perms not labels, alert on broad grants + escalation paths.
  • It's an operating model (telemetry + automation + ownership + review + safe removal), not a one-time policy.

20. How would you reduce standing privileges?

  • Standing privilege = perms available all the time even when unneeded; reducing lowers blast radius for theft + insider misuse.
  • Inventory privileged roles/groups/SAs/long-lived tokens, find unused perms from logs, replace permanent admin w/ JIT (approval+reason+scope+expiry), split broad roles, break-glass only for emergencies, permission boundaries/guardrails, remove dormant accounts/stale memberships, rotate/eliminate static SA keys, monitor activation.
  • Example: 40 engineers w/ permanent prod admin -> eligible for JIT (ticket+approval+MFA, expires 2h); routine tasks use narrower roles.
  • AWS: short-session permission sets, Access Analyzer, SCPs, permission boundaries, CloudTrail, unused-permission review.
  • Tradeoff: reduce standing access via fast/reliable/auditable elevation, not by making engineers route around security.

Chunk 3 - Questions 21-30

21. How would you detect excessive permissions?

  • Compare granted perms vs actual usage vs peer baselines vs risk.
  • Signals: granted-but-never-used, wildcard admin, access outside job/tenant/region/project, SAs with human-like breadth, many high-risk roles, toxic combos, direct (non-group) grants, dormant accounts w/ privileges.
  • AWS: IAM Access Analyzer (unused + external access), CloudTrail (actual usage), Access Advisor (service usage), alert on AdministratorAccess/wildcard/* resource/risky trust.
  • Focus on effective permissions + real usage, not role names (a "viewer" role can hold dangerous privileges).

22. How would you design permission reviews for thousands of employees?

  • Risk-based, evidence-driven, easy to understand - opaque role names = rubber-stamping.
  • Group access by app/role/resource/risk; show effective perms + last-used; prioritize privileged/sensitive/external/unusual; route to accountable owner (manager/app/data/resource); recommend removals; require justification to keep high-risk; auto-remove on deny/incomplete.
  • Reviewer must answer: who has access, what can they do, why, when last used, is it typical, what breaks if removed.
  • Anti-fatigue: don't review low-risk like privileged, highlight changes since last review, campaigns for high-risk, auto-remove after grace, measure accuracy + overdue.

23. How would you automate access recertification?

  • Connect HR data + IAM grants + usage telemetry + risk scoring + workflow.
  • Inputs: HR (dept/manager/status/contractor end), IAM (groups/roles/direct/SAs), usage (last login/API/privileged action), risk (admin/sensitive/prod/external), ownership.
  • Automate: expire temp access, remove terminated users, flag unused privileged, route high-risk to resource owner, ticket denied removals, verify removal completed.
  • Keep human approval for high-impact; automate evidence collection, routing, reminders, enforcement, audit records.

24. What evidence would prove that access reviews are effective?

  • Risk reduction, not completed forms.
  • Strong: % high-risk reviewed on time, % access removed/reduced, less unused privileged, fewer standing admin roles, fast deny-to-removal time, fewer stale/orphaned accounts, fewer toxic combos, fewer external/cross-tenant grants, reviewer understood effective perms.
  • Weak: "100% reviews completed" with no removals, instant approvals, role-names-only, denied access never removed.
  • Effectiveness = risk reduction + enforcement completion, not review completion rate.

25. How would you design separation of duties?

  • Prevent one person holding combos that enable fraud/abuse/control-bypass.
  • Examples: create vendor != approve payment; deploy code != approve own prod change; investigate logs != delete audit logs; manage compute != change log retention.
  • Identify toxic combos, encode in IAM/workflow/approval, detect existing violations via effective-permission analysis, prevent new at assignment time, independent approval for sensitive workflows, log + review exceptions.
  • Enforce at grant time AND transaction time (even if user holds both roles, workflow blocks self-approval).

26. How would you prevent privilege escalation through role chaining?

  • Escalation when one role can assume/grant another, reaching higher privilege (A->B->C admin; update IAM policy then self-grant admin; modify CI that deploys privileged role).
  • Model role assumption graph + detect paths to privileged; restrict who can create/update/pass/assume roles; permission boundaries; explicit trust policies + conditions; deny self-escalation; monitor PassRole/trust changes/role creation; policy simulation; separate role admin from role usage.
  • AWS: iam:PassRole is a common escalation path - restrict by resource/service/condition + alert.
  • Analyze effective permissions + paths, not individual role labels.

27. How would you secure admin roles in cloud platforms?

  • High blast radius -> strong auth + least privilege + JIT + guardrails + monitoring.
  • Federated SSO + phishing-resistant MFA, no long-lived IAM users/static keys, separate admin from daily accounts, JIT + short sessions, split admin by function (network/security/IAM/db/break-glass), org guardrails (SCPs), permission boundaries, protect logging/security from ordinary admins, alert + review.
  • AWS: Identity Center permission sets, SCPs to prevent disabling CloudTrail, CloudTrail in log archive account, GuardDuty/Security Hub, Access Analyzer, root locked w/ hardware MFA.
  • Alert: key creation, trust policy change, attach admin policy, disable logging, KMS policy change, make storage public, change SCPs.

28. How would you design policy-as-code for authorization?

  • Policy = versioned, testable, reviewable code - improves consistency/audit but bad rollout = broad over-permit/outage.
  • Version control, code review + owner approval, unit tests (allow/deny), regression tests for known bugs, policy simulation, staged rollout/canaries, versioned bundles + rollback, human-readable explanations, log policy version per decision.
  • Test: default deny, tenant boundary, object ownership, privileged actions, break-glass, stale attributes, conflicting rules, known abuse.
  • Policy is production code - same rigor as app code (policy bugs expose data instantly across services).

29. What is the difference between centralized and decentralized authorization?

  • Centralized = common policy engine (consistency/audit/governance, but latency + availability dependency + broad blast radius from policy bugs).
  • Decentralized = each service owns logic (local context/autonomy, but inconsistency + duplication + harder audit).
  • Hybrid in practice: central policy definitions/libraries + local enforcement + central decision service for complex/high-risk + bounded-TTL local cache + shared audit schema.
  • Key: distinguish PDP (decision) from PEP (enforcement) - enforcement always happens in the service receiving the request.
  • Centralize policy semantics/governance, keep enforcement near the resource, design for availability/freshness/local context.

30. When should authorization be enforced at the service layer versus the gateway layer?

  • Gateway (coarse): token validation, authN, basic scope/route, rate limiting, block unauthorized clients, coarse tenant routing.
  • Service (required): object/field/row-level, business workflow rules, tenant boundary vs actual resource state, contextual checks, transaction-specific policy.
  • Gateway lacks resource state - knows Alice has invoice:read but not whether invoice 123 is in her tenant.
  • Gateway = coarse admission control; service = final decision tied to resource state.

Chunk 4 - Questions 31-40

31. Why is gateway-only authorization insufficient?

  • Gateway lacks full resource/object/field/tenant/workflow context.
  • Knows: token valid, iss/aud correct, basic scope, route needs auth, request rate/risk. Can't know: object belongs to user's tenant, ownership/relationship, field sensitivity, workflow state, row legal-hold/region, toxic combos.
  • Also fails when services call each other internally, bypassing the public gateway.
  • Gateway = admission control; service-layer authz = real protection for object-level + business decisions.

32. How would you design authorization for internal microservices?

  • Need service identity AND user/workload authz context; internal network location != authorization.
  • Authenticate services (mTLS/workload identity/signed tokens), authorize by caller/target/action/environment, pass end-user context only when needed + protect from spoofing, audience-specific tokens, avoid forwarding broad user bearer tokens, central policy/shared lib, enforce in each owning service, log caller+subject+action+resource+decision.
  • Mistakes: trust internal headers w/o mTLS/signing, broad DB access for every service, lost actor context, one shared SA, gateway validates token but service skips object check.
  • Zero trust applies inside the platform too.

33. How would you prevent tenant isolation failures?

  • Make tenant context explicit, trusted, enforced at every layer.
  • Tenant ID on every resource; derive active tenant from trusted session/membership (not just input); validate resource tenant before access/modify; tenant-scoped roles; include tenant filter in search/cache/analytics/queues/background jobs; DB row-level security or separate DBs/accounts; avoid global IDs bypassing checks; log tenant per decision; test cross-tenant continuously.
  • Failure modes: cache key missing tenant, background worker by-ID without filter, search cross-tenant results, admin API trusts tenant param, shared group/edge bridges tenants.
  • Account separation alone doesn't solve app-level authz - pooled SaaS still needs tenant-aware policy.

34. How would you test tenant isolation?

  • Combine automated tests + negative cases + fuzzing + production detections.
  • Create tenants A/B w/ users/admins/groups/resources; auth as A, attempt B actions; swap object/tenant/group/project/doc IDs; test reads/writes/deletes/exports/search/reports/background jobs across APIs/GraphQL/mobile/admin/bulk; test cache + async + webhooks; verify denies logged.
  • Cases: A reads B invoice, A admin edits B user, A search returns B doc, A export includes B rows, A webhook triggers B action.
  • Runtime: denied cross-tenant attempts, one user touches many tenants, resource tenant != session tenant, cross-tenant cache hits, multi-tenant bulk export.
  • Verified across every read/write/cache/index/export/async path, not one test.

35. How would you design authorization logs?

  • Support forensics/debug/compliance/detection without leaking policy details.
  • Fields: timestamp+correlation ID, actor, effective subject (if delegated), tenant, resource type+ID/hash, action, decision (permit/deny/step-up/error), policy version, reason code/rule ID, enforcement point, auth assurance/session age, source IP/device/client/workload, risk score.
  • Don't log: raw tokens, secrets, full sensitive payloads, complete policy internals, excessive PII.
  • Log denies AND permits - denies are early signal for probing/IDOR/tenant attacks/escalation.

36. What authorization events should generate security alerts?

  • Not every deny pages, but high-signal: cross-tenant attempt, repeated object-level denies across many IDs, privileged role assignment/activation, new admin/owner on sensitive resource, policy changed to wildcard, public/external sharing on sensitive data, break-glass use, denied attempt to disable logs, SA accesses unusual resource, role chaining/PassRole escalation, sensitive field/dataset outside pattern, export by newly-privileged user.
  • Context matters: one deny = mistake; hundreds across sequential objects = enumeration; admin activation from new country at 3 AM = suspicious.
  • Use suppression + baselines + severity tiers (cross-tenant success / break-glass = loud).

37. How would you debug a denied access request safely?

  • Risky - error handling can leak policy/tenant/object existence/attributes.
  • Generic denial + request ID to user; detailed reason codes internally; support tooling requires authz to view details; show only what requester may know; explain API for admins/support w/ redaction; include policy version + attributes used; don't reveal which exact condition failed.
  • User: "You do not have access. Reference ID: abc123." Internal: actor/action/resource/decision=deny/reason=tenant_mismatch/policy_version.
  • Separate external ambiguity from internal explainability.

38. How would you avoid leaking sensitive policy details in error messages?

  • Errors must not be an oracle: avoid "Project 123 exists but you lack FinanceAdmin," "requires LegalHoldViewer," "bob is a valid admin."
  • Generic denial before user is authorized to know existence; same response for not-found and not-authorized where appropriate; request IDs for support; precise internal reason codes; more detail only to authorized admins.
  • Tradeoff: too little frustrates users -> role-aware messaging (tenant admin gets more than an unrelated user).
  • Error messages are attack surface: generic at untrusted boundaries, detailed in protected support/audit tooling.

39. How would you migrate from RBAC to ABAC?

  • Gradual; most mature systems combine RBAC structure + ABAC conditions (not a direct replacement).
  • Steps: inventory roles/perms/pain points -> identify attribute-needing policies (region/sensitivity/device/tenant/project/time) -> define trusted attribute sources+owners -> add attributes -> build PDP/PEP -> shadow mode + compare vs RBAC -> test allow/deny/missing/stale -> start low-risk reads -> enforce on high-value roles -> retire obsolete roles.
  • Key risk: ABAC is only as reliable as its attributes (stale/user-controlled dept/region/device/labels fail dangerously).

40. What attributes are risky to use in ABAC policies?

  • Risky = stale/user-controlled/ambiguous/spoofable/poorly-governed/too-broad: client-supplied dept/role/tenant, IP geo as sole location, email domain as employment proof, device name w/o attestation, self-declared classification, slow-sync group membership, job title as precise authz, time-of-day alone, network location as trust, owner-editable tags.
  • Safer: define attribute owner + source of truth, classify trusted/derived/untrusted, freshness requirements, protect high-impact tags from self-service edits, log attributes used, deny-by-default on missing critical attrs, test w/ missing/stale/malicious.
  • Example: resource.sensitivity=public is dangerous if any editor can flip restricted->public.
  • ABAC moves trust into attributes - attribute governance becomes authorization governance.

Chunk 5 - Questions 41-50

41. How would you protect policy decision points and policy enforcement points?

  • PDP evaluates policy; PEP intercepts + enforces - both security-critical.
  • PDP: strong PEP<->PDP auth, restrict who changes policy, versioned + signed bundles, tests/canaries/rollback, protected attribute sources, HA + safe failure, decision logging w/ version, rate limiting.
  • PEP: cannot be bypassed on protected paths, fails closed for sensitive ops, sends validated inputs, enforces obligations (masking/step-up), no client-supplied trusted attributes, logs permit+deny, consistent libraries.
  • Failure modes: PEP omitted from new endpoint, PDP down + fail open, client-controlled attributes, rollout permits all, PEP ignores obligations.
  • Protect PDP integrity + PEP completeness - a correct engine is useless if enforcement is bypassed.

42. How would you handle stale group membership data?

  • Stale membership = access after role change/termination/IR.
  • Freshness by group risk level, event-driven directory updates, short TTL for high-risk groups, force invalidation on termination/admin removal, authoritative check for sensitive actions, track membership version/timestamp, monitor sync lag, fail closed when freshness unknown.
  • Tiers: newsletter = eventually consistent OK; app viewer = minutes; prod admin = near real time/authoritative; terminated user = immediate.
  • Removing from prod-admins should invalidate sessions quickly - 24h sync is unacceptable for privileged.

43. How would you handle authorization cache invalidation?

  • Caching improves perf but preserves stale access - design around revocation risk.
  • Cache low-risk longer than high-risk; key includes principal+action+resource+tenant+policy-version+attributes; short TTL for sensitive; invalidate on membership/role/policy/tenant/sensitivity change; event-driven for critical; store decision version for rollout invalidation.
  • Failure modes: cache key omits tenant, cached permit survives role removal, policy update doesn't invalidate, sensitivity change but cached access remains, local cache ignores central revocation.
  • Cache only with explicit freshness + invalidation triggers + risk-based TTLs; revocation-critical needs authoritative checks or very short windows.

44. What are the risks of embedding authorization logic directly in application code?

  • OK for simple apps; at scale = inconsistent/duplicated/hard-to-audit policy.
  • Risks: services implement differently, new endpoints forget checks, policy changes need deploys everywhere, tests miss edge cases, authz mixed with UI/controller, not explainable to auditors, can't centrally review, devs copy vulnerable patterns.
  • Better: shared authz library, central policy engine, declarative policy-as-code, service-layer wrappers, data APIs requiring principal context, tests per endpoint/object action.
  • Tradeoff: total centralization adds latency/reliability risk - centralize semantics+tooling, keep enforcement near resource.
  • Risk isn't code itself - it's scattered policy with no consistency/ownership/tests/visibility.

45. How would you review authorization logic during code review?

  • Adversarial + object-focused: trace request -> data access, ask if correct principal can do exact action on exact resource.
  • Checklist: authN required? where is authz enforced? includes principal+action+resource+tenant+context? object ownership server-side? field/row policies? can client params influence trusted attrs? admin/normal paths separated? bulk/export/search/async covered? fails closed? denies logged safely? tests for unauthorized + cross-tenant?
  • High-risk: if user.is_admin w/o scope, server trusts client role=admin/tenant_id, UI hides buttons but API allows, filter by object ID not tenant ID, resolver returns nested sensitive fields, background job w/o principal context.
  • Key question: "What prevents user A from performing this action on user B's resource?"

46. What are common authorization anti-patterns?

  • Gateway-only for object-level, UI-only (hidden buttons), trusting client roles/tenant/user IDs, role names != effective perms, global admin for routine work, no deny-by-default.
  • No checks on exports/search/batch/webhooks, public links bypass tenant policy, long-lived caches w/o invalidation, over-trusting SAML/OIDC claims as final authz, shared broad SAs, no audit on permission changes, oracle error messages, tests cover only authorized success.
  • Anti-pattern: controller checks logged-in -> DAO fetches by ID -> returns, no tenant/object check. Fix: pass principal+action to service -> service loads + checks tenant/policy -> DAO returns after success -> log decision.

47. How would you design customer-managed roles safely?

  • Tenant admins define roles - valuable but dangerous if they bypass provider boundaries/hidden system perms.
  • Permission catalog (safe documented perms), separate provider-only from customer-assignable, tenant-scoped by default, prevent delegating perms admin lacks authority for, role templates, show effective perms before save, validate toxic combos, log create/update/assign/delete, simulation, version + rollback.
  • Failure modes: customer role grants provider admin, disables audit logs, self-grants broader perms, applies globally, hides dangerous inherited perms.
  • Need constrained permission universe + tenant scoping + explainability + validation + audit.

48. How would you design audit trails for delegated access?

  • Show BOTH actor who performed it AND subject/owner on whose behalf.
  • Fields: actor, effective subject, delegation grant ID, resource+action, tenant, scope, approval/consent source, reason/ticket, start+expiry, source IP/device/session hash, decision+policy version.
  • Never overwrite actor with effective subject; show delegated sessions distinctly; require reason for support impersonation; revocable; alert on broad/unusual delegated actions; let customers review support access; protect logs from delegates/audited admins.
  • Delegated access without actor-subject auditability is indistinguishable from account compromise.

49. How would you handle legal hold or compliance overrides in authorization?

  • Special paths that preserve/restrict/expose data for legal/regulatory reasons - must be explicit/narrow/audited/protected.
  • Model legal hold as resource state/policy attribute controlled by authorized legal/compliance roles; prevent deletion/modification even by owners; strong approval to apply/remove; separate hold admin from data access; log all hold changes + access under hold; apply retention consistently across primary/backups/exports/derived; explainable to auditors.
  • Risks: override becomes broad backdoor, hold blocks deletion but exports leak, owner-editable hold status, audit omits who applied hold, privacy-deletion mishandled.
  • Legal hold = policy exception (SoD + tight scope + strong evidence), not an authz free-for-all.

50. What are the top five authorization questions you would ask in a design review?

    1. What is the decision? (principal/action/resource/tenant/context per sensitive op).
    1. Where is it enforced? (name PEP, verify no bypass via API/batch/search/export/internal).
    1. What is the source of truth? (roles/relationships/attributes/policy version + ownership).
    1. How does revocation work? (cache TTLs/group sync/role removal/session invalidation/propagation).
    1. How do we prove + debug safely? (logs/explainability/audit/error messages/detection).
  • Also: cross-tenant prevention, A-can't-access-B's-object, no self-escalation, toxic combos, PDP-unavailable behavior, least-privilege SAs, field/row/export coverage, customer roles can't escape tenant, break-glass monitoring, bypass tests.
  • Failures live at edges: new endpoints, async jobs, nested fields, cached decisions, delegated flows, emergency exceptions - look there deliberately.
