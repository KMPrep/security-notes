enterprise data protection Cloud Security AWS to GCP - Cheatsheet

Source: questions/answers/cloud-security-aws-gcp-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. An EC2 instance is compromised. What can the attacker do next?

  • Query IMDS for instance-profile creds; enumerate perms (sts:GetCallerIdentity); access S3/DynamoDB/queues/secrets/KMS the role allows.
  • Inspect local files/env/config/logs; scan reachable VPC services (DBs, Redis, internal APIs); pivot via local SSH keys/tokens; abuse for mining/C2; tamper local agents.
  • Blast radius = union of instance role + network path + local secrets + attached data volumes.

2. A GCE VM is compromised. What can the attacker do next?

  • Query metadata server for SA tokens; identify attached SA + OAuth scopes; call GCP APIs allowed by IAM+scopes (GCS, Secret Manager, BigQuery, Pub/Sub, GKE).
  • GCP authz depends on BOTH IAM perms AND legacy access scopes for metadata-issued tokens.
  • Main crown-jewel path = attached service account; default compute SA with broad project perms turns one VM into project compromise.

3. How do you analyze cloud workload blast radius?

  • Ask what it can read/write/assume/reach/decrypt/modify/hide.
  • Dimensions: identity, effective IAM/resource policies, network reachability, data stores, secrets, KMS, control-plane (create users/change IAM/disable logging), detection resistance.
  • Blast radius is NOT just IAM - a read-only role + network access to an unauthenticated internal DB is still catastrophic.

4. How would you contain a compromised cloud instance?

  • Preserve evidence while cutting attacker control + cloud API abuse: snapshot disk/memory, preserve metadata/tags/role/SGs/flow logs/API logs.
  • Isolate via quarantine SG/firewall; revoke/detach instance profile or SA; rotate reachable secrets; block suspicious egress; disable found access keys; don't terminate until evidence captured (unless active harm).
  • AWS: quarantine SG, detach IAM role, snapshot EBS, review CloudTrail. GCP: restrictive firewall/tags, remove SA perms, snapshot disks, review Audit Logs.
  • Containment is a race between attacker persistence and evidence preservation - isolate first, then scope creds + data.

5. What is the role of instance profiles or service accounts in cloud compromise?

  • Workload identity removes hard-coded creds but a compromised workload inherits the same identity.
  • They issue temp creds via metadata, define API perms, often grant storage/secrets/queues/logs/KMS, can be escalation paths if over-permissioned.
  • AWS = IAM role via instance profile (IMDS); GCP = attached SA (metadata server). Common mistakes: broad role on many instances / default SA with broad project roles.
  • Safer than static keys but makes least privilege mandatory.

6. How would you secure cloud metadata services?

  • Require IMDSv2 on EC2; low hop limit (1); block metadata from containers that don't need it; firewall/iptables/eBPF/sidecar controls; narrow roles/SAs.
  • No secrets in user data/startup scripts/custom metadata; disable legacy endpoints; monitor credential use from unusual paths; prevent SSRF reaching metadata IPs.
  • Addresses: EC2 169.254.169.254; GCP metadata.google.internal + Metadata-Flavor: Google header.
  • Hardening necessary but not enough - attached identity must still be least privilege.

7. Compare AWS IMDSv1, IMDSv2, and GCP metadata protections.

  • IMDSv1 = simple GET, weak SSRF resistance -> disable. IMDSv2 = PUT for session token then X-aws-ec2-metadata-token, hop limit -> require.
  • GCP requires Metadata-Flavor: Google header, blocking simple header-less SSRF.
  • These reduce SSRF exploitability; don't protect against full host compromise - role design is decisive.

8. How would you prevent SSRF from reaching metadata services?

  • App: strict URL allowlists, resolve DNS + validate final IP before connect, block link-local/loopback/private/multicast/reserved, re-check after redirects/rebinding, dedicated egress proxy, don't let users control headers/methods.
  • Infra: IMDSv2, hop limit 1, deny metadata from containers, firewall metadata IP, least-privilege roles.
  • Detect: unusual metadata access, API calls from roles that don't normally call them, SSRF probes in URL params.
  • Strongest when app can't fetch arbitrary hosts AND infra makes metadata tokens hard to retrieve.

9. Explain AWS IAM policy evaluation.

  • Default deny; explicit deny always wins.
  • Combine identity policies (grants), resource policies (cross-account), permission boundaries (ceiling), session policies (reduce), SCPs (org ceiling), conditions.
  • Logic: explicit deny? -> deny; allowed by policy? -> no = deny; limited by boundary/SCP? -> yes = deny; else allow.
  • Set-intersection + override: evaluate both grants AND ceilings; explicit deny dominates.

10. Explain GCP IAM policy evaluation at a high level.

  • Resource-hierarchy centric: org -> folder -> project -> resource, allow bindings inherit downward.
  • Principals (users/groups/SAs/workload identities/domains) bound to roles; deny policies block even with allow; conditions limit by attributes; org policies constrain config (not API perms); legacy OAuth scopes can further limit VM tokens.
  • Broad project-level binding grants many services - project boundaries + SA design matter deeply.

Chunk 2 - Questions 11-20

11. Compare AWS IAM roles and GCP service accounts.

  • AWS role = assumable identity via STS AssumeRole (trust policy); GCP SA = resource identity + principal, used via attachment or impersonation.
  • AWS: study who can assume + what it can do. GCP: study who can impersonate + what it can access.
  • Mistakes: AWS broad trust/permissions; GCP broad project roles on default SAs; both using workload identity as shared admin.

12. What are common cloud IAM privilege escalation paths?

  • AWS: iam:PassRole to code-running service, sts:AssumeRole to privileged role, create access keys, attach policies, update trust policy, launch EC2/Lambda with privileged role, modify resource policies (S3/KMS/SQS/SNS/secrets).
  • GCP: iam.serviceAccounts.actAs, getAccessToken impersonation, create SA keys, grant IAM roles, deploy workloads with privileged identities, modify KMS/storage IAM.
  • Escalation hides in meta-permissions: power to change who can act, what identity code runs as, or which policies apply.

13. How would you detect cloud privilege escalation?

  • AWS signals: AttachUserPolicy/AttachRolePolicy/PutRolePolicy, UpdateAssumeRolePolicy, CreateAccessKey, CreatePolicyVersion/SetDefaultPolicyVersion, PassRole then EC2/Lambda/ECS, AssumeRole to sensitive roles.
  • GCP signals: SetIamPolicy, GenerateAccessToken/impersonation, SA key creation, deploy with sensitive SA, IAM changes for Owner/Editor/Token Creator/SA User.
  • Alert on grants, correlate IAM-change-then-data-access, baseline normal assumption paths, logs in separate security account.
  • Alert on the stepping stones, not only final admin actions.

14. How would you design least privilege for cloud workloads?

  • Start from required actions, narrow identity/data/network/time: one identity per workload, only required actions on specific resources, conditions (source/time/tags), separate read/write/admin/break-glass.
  • Avoid shared roles; short-lived creds via metadata/federation; review unused perms (Access Analyzer/IAM Recommender/simulation); test denied sensitive actions.
  • Bad->better: s3:* all buckets -> s3:GetObject one prefix; project Editor SA -> custom SA; permanent CI admin key -> OIDC deploy-only role.
  • Least privilege is a lifecycle: start narrow, observe, remove unused, prevent creep during incidents/launches.

15. How would you handle cross-account or cross-project access?

  • Explicit, scoped, logged, owned by both sides.
  • AWS: role assumption, trust specific principals (not whole accounts), external ID for 3rd party, conditions (org ID/source ARN/MFA), separate roles per purpose, log AssumeRole both sides.
  • GCP: SA impersonation or workload identity federation, narrowest-resource bindings, avoid broad Owner/Editor, conditions + org policies, log impersonation + IAM changes.
  • It's a trust contract: target must know exactly who can enter, why, and with what ceiling.

16. What is a cloud trust policy and how can it be abused?

  • AWS trust policy = who can assume a role + conditions; GCP equivalent = who can impersonate/attach an SA.
  • Abuse: trust whole external account, missing external ID, weak federated claims, trust any CI/cluster workload, broad impersonation, missing source/repo/branch/env/audience conditions.
  • A trust policy is the front door - a narrow permission policy doesn't help if a powerful role is assumable by the wrong principals.

17. What is confused deputy in cloud IAM?

  • Trusted service/3rd party tricked into using privileges for the wrong customer/resource (e.g., SaaS vendor with access to many customer accounts; A tricks vendor into reaching B's role without external ID/tenant binding).
  • AWS mitigations: external ID, aws:SourceArn/aws:SourceAccount, tight resource policies. GCP: bind federation to exact issuer/audience/subject/attributes, tenant-specific SAs, validate ownership.
  • Authorization failure across delegation - bind caller + tenant + target resource together.

18. How would you prevent confused deputy in cloud integrations?

  • Bind delegated action to correct tenant + resource: tenant-specific external IDs/unguessable binding, restrict trust to exact principals/claims, validate SourceArn/SourceAccount/audience/subject/repo/branch/env, store + check integration ownership before every action.
  • Separate roles/SAs per tenant for high-risk, least-privilege delegated roles, log tenant/integration/role/target/request IDs, alert on cross-tenant access attempts.
  • Possession of a role ARN/SA name must NOT be sufficient - integration must prove tenant ownership.

19. How would you secure CI/CD access to cloud resources?

  • Short-lived, scoped, auditable, tied to repo/branch/workflow/environment.
  • Prefer OIDC federation over static keys; bind roles to exact repo/branch/tag/workflow/env claims; separate build/test/deploy/release identities; approvals for prod; block untrusted-fork PRs from privileged tokens; protect secrets + pipeline config; sign artifacts + verify provenance; least-privilege deploy roles; log assumptions + deploys.
  • AWS: GitHub Actions OIDC w/ repo+branch conditions. GCP: Workload Identity Federation pool + attribute conditions + impersonation.
  • The CI system is a production IdP - treat pipeline config changes like privileged code changes.

20. How would you prevent long-lived cloud credentials?

  • Make temporary identity the default, static keys exceptional: instance profiles/SAs/federation, disable user access key creation, SCPs/org policies to restrict key creation, alert on new key creation, secrets manager + rotation for the rest, expiry + owner metadata for exceptions, scan code/images/tickets/logs.
  • Migration: inventory static keys -> map to owner -> replace w/ role/SA/federation -> monitor temp access -> disable + delete.
  • Long-lived keys are breach multipliers - best rotation is removing the key class from normal workflows.

Chunk 3 - Questions 21-30

21. What are workload identity federation risks?

  • Removes static keys but bad trust conditions grant access to the wrong workload.
  • Risks: broad issuer w/o subject/audience checks, any repo/branch/workflow, tokens from PRs/forks, missing prod approval, one role reused across workloads, weak claim-to-permission mapping, no logging of external identity, token replay.
  • Controls: validate issuer/audience/subject/repo/branch/workflow/env, separate roles per env, prod approvals, short token lifetimes, log external claims, review trust policies like code.
  • Federation removes stored keys but shifts risk into trust-policy correctness.

22. How would you design cloud secrets management?

  • Centralize storage + access control + rotation + audit + delivery (Secrets Manager / SSM+KMS / GCP Secret Manager).
  • Encrypt w/ managed or CMK by sensitivity; grant read to workload identities (not humans by default); avoid env vars for highly sensitive; runtime retrieval + short-TTL cache; separate by env/tenant; versioning + rotation; log reads/writes/deletes/perm changes; scan code/images.
  • Not just encrypted storage - identity-scoped retrieval + rotation + audit + leak response.

23. How would you rotate cloud secrets without downtime?

  • Support old + new during controlled transition: create new version -> update dependents to accept both -> deploy readers of new -> verify -> revoke old -> monitor stragglers.
  • Examples: DB dual users/alternating creds, overlapping API keys, staged TLS trust bundles, OAuth client secret overlap.
  • Automate, use current/previous labels, track owners/dependencies, alert on deprecated-version use, test rollback.
  • Rotation fails when systems assume one global secret - design clients + servers to tolerate overlap.

24. How would you design KMS usage for sensitive data?

  • Envelope encryption for large data: DEK near data, encrypted by KMS key; CMK for high-sensitivity needing control/rotation/audit.
  • Separate keys by env/data-class/tenant/blast-radius; restrict decrypt to specific workload identities; encryption context/AAD to bind ciphertext to purpose; log decrypt + key admin; rotate for cryptoperiod.
  • KMS protects keys + gives policy/audit leverage - it does NOT replace data access control.

25. What are common KMS key policy mistakes?

  • Decrypt to broad roles/whole projects; app roles with key admin; root/owner/editor administer sensitive keys w/o guardrails; one key for unrelated data classes/envs; no encrypt-only vs decrypt separation; no encryption-context/resource/caller conditions; key deletion w/o approval/waiting period; no alert on key policy changes; assuming encryption stops decrypt-authorized users.
  • AWS: KMS policy * via broad account delegation; Secrets Manager + KMS policies not reviewed together. GCP: broad project-level Decrypter; key admin + decrypter same group.
  • Most dangerous mistake: treating key admin, decrypt, and data admin as the same role.

26. How would you handle KMS key compromise?

  • Clarify type: unauthorized decrypt use vs policy takeover vs key-material exposure vs privileged misuse. Managed KMS prevents raw extraction, so most are access-control compromise.
  • Preserve KMS + API logs; identify key/data-class/principals/window; disable/restrict key if no unacceptable outage; remove unauthorized grants/bindings; rotate or replace key; re-encrypt when material/decrypt no longer trusted; rotate dependent secrets; review decrypt events in window; notify data owners + legal/privacy.
  • KMS incidents are often data-access incidents - key event tells who could decrypt; impact analysis tells what data they reached.

27. When would you use customer-managed keys?

  • When you need stronger control than provider-managed default: regulated data needing controlled rotation/review, SoD between data + key admins, per-tenant/per-env key boundaries, decrypt audit, fast access kill (disable key), cross-account encrypted access control, BYOK/external key manager.
  • Tradeoffs: more operational complexity, outage risk from bad/disabled key policy, lifecycle management, more review burden.
  • Use CMK when key policy + audit are part of the requirement, not just because encryption "sounds stronger."

28. When would you use application-level encryption in cloud?

  • When storage/DB-level encryption doesn't separate from platform/DBAs/broad service perms: credentials/tokens/SSNs/health data, tenant-level crypto isolation, shared analytics platforms, protection from accidental broad DB reads, client-side encryption, field-level access control.
  • Tradeoffs: harder search/index/analytics, more key complexity, app bugs still leak plaintext, careful rotation/migration.
  • Target specific sensitive fields, not blindly everywhere.

29. How would you design tamper-resistant cloud audit logging?

  • Centralize + separate duties + immutable storage + monitor pipeline changes.
  • Org-level CloudTrail/Audit Logs coverage -> dedicated security account/project; separate write/read/admin perms; S3 Object Lock + versioning + retention; encrypt w/ security-administered keys; monitor changes to trails/sinks/buckets/KMS; forward to SIEM; test coverage for new accounts/projects/regions/services.
  • Logs are trustworthy only if a workload/app-account compromise can't edit or delete the evidence.

30. Why should audit logs live in a separate account or project?

  • Prevent the same compromise from causing harm AND deleting evidence.
  • Benefits: SoD between workload + log admins, compromised app roles can't tamper, centralized retention/legal hold, org-wide detection, independent KMS/storage policies, IR works when prod is unstable.
  • Avoided failure: attacker gets app-account admin, disables CloudTrail, deletes local bucket - central copy survives.
  • The authoritative copy must be outside the blast radius.

Chunk 4 - Questions 31-40

31. How would you detect attempts to disable logging?

  • Monitor control-plane changes to trails/sinks/buckets/retention/encryption/permissions.
  • AWS: StopLogging, DeleteTrail, UpdateTrail, PutEventSelectors reducing coverage, log bucket policy changes, Object Lock/retention changes, KMS disable/policy changes, GuardDuty disable.
  • GCP: sink deletion/modification, audit exemption/exclusion changes, IAM changes affecting logging, bucket retention changes, CMEK disable, SCC disablement.
  • Alert near real time, route outside the affected account, require tickets/approvals, correlate w/ recent escalation.
  • Treat logging disablement as high-severity attacker behavior until proven planned maintenance.

32. What CloudTrail events are most important for IAM abuse?

  • CreateUser/CreateRole/CreatePolicy, Attach*Policy/Put*Policy, CreatePolicyVersion/SetDefaultPolicyVersion, UpdateAssumeRolePolicy, CreateAccessKey, DeleteAccessKey after suspicious activity, AssumeRole (admin/break-glass), PassRole then service creation, Put/DeleteRolePermissionsBoundary, AddUserToGroup for privileged groups.
  • Context: source principal, target identity, new permissions, source IP/UA, MFA presence, role session name, account/org.
  • Focus on events that create new power, change trust, mint credentials, or run code as a privileged role.

33. What GCP audit logs are most important for IAM abuse?

  • SetIamPolicy (org/folder/project/SA/bucket/key/dataset), SA key creation, impersonation + access token generation, custom role create/update, project ownership/billing privilege changes, deploy with privileged SA, org policy weakening, KMS IAM/state changes, logging sink/audit config changes.
  • Sensitive roles: Owner, Editor, SA Admin, SA User, SA Token Creator, Project IAM Admin, Org Admin, KMS Admin/Decrypter.
  • GCP escalation turns on SA impersonation + broad inherited project roles - make those loud.

34. Compare GuardDuty, Security Command Center, and SIEM detections.

  • GuardDuty = AWS-native fast findings (CloudTrail/VPC DNS/EKS/S3); AWS-focused, finding-driven.
  • SCC = GCP-native misconfig + threat findings; GCP-focused, tier-dependent.
  • SIEM = cross-platform correlation (AWS/GCP/IdP/endpoint/CI/SaaS); needs engineering + tuning + data quality.
  • Use together: native tools = high-signal cloud detections, SIEM correlates, CSPM finds misconfig pre-exploit, IR needs raw logs not just findings.
  • Managed detections accelerate; don't replace audit strategy + custom correlation.

35. How would you investigate suspicious cloud API calls?

  • Identify principal, source, intent, permissions used, downstream impact.
  • Confirm event source -> identify principal/session/SA/federated identity -> check source IP/UA/region/device -> review preceding login/assumption/key-creation/IAM-change/CI -> success or fail? -> enumerate resources -> correlate w/ data access/KMS decrypts/storage reads/egress -> check creds still valid -> contain (revoke/disable/remove binding/isolate).
  • Ask: expected automation? MFA/approved federation? recently gained perms? logs/detections disabled nearby?
  • Suspicious API calls are identity incidents until proven otherwise - start from the principal, trace outward.

36. How would you design alerting for cloud data exfiltration?

  • Combine data access + identity risk + volume anomaly + destination.
  • Signals: large S3/GCS/BigQuery/DB reads, listing then bulk gets, KMS decrypt spike, unusual country/ASN/UA, new key then data access, public bucket/signed URL creation, snapshot/export/backup, high egress to unfamiliar dest, off-hours/unusual-principal downloads.
  • Baseline per workload/tenant/datastore; high-fidelity rules for sensitive datasets; DLP/classification tags weight severity; correlate control-plane + data-plane.
  • Detection improves dramatically when data classification connects to identity + egress telemetry.

37. How would you detect public storage bucket exposure?

  • Continuously check policy + ACLs + public-access settings + actual reachability.
  • AWS: S3 Block Public Access state, public-principal bucket policy, object ACLs, access points/cross-account grants, CloudFront origins, Config/Access Analyzer/Security Hub/CSPM.
  • GCP: IAM allUsers/allAuthenticatedUsers, uniform bucket-level access state, public object ACLs, SCC/Asset Inventory.
  • Validate effective reachability, classify exposed data, review who changed policy + when, alert owner + security.
  • Don't stop at policy inspection - verify effective public reachability and classify what was exposed.

38. How would you prevent accidental public exposure of cloud storage?

  • Default-deny guardrails + ownership review + auto-remediation.
  • S3 Block Public Access (account + bucket), GCP org policy public-access prevention, prefer uniform bucket-level access, disable legacy ACLs, IaC review of bucket policies, least-privilege write identities, tag owner/data-class/exposure-intent, CSPM + policy-as-code in CI, auto-remediate, signed URLs/CDN for intended sharing.
  • Public buckets should be impossible by default and explicit by exception.

39. How would you secure serverless functions?

  • Focus on event sources, execution identity, secrets, dependencies, egress, observability.
  • One role/SA per function/group, minimal perms, validate event input (S3/Pub-Sub/HTTP/queues), secrets in manager not env vars, restrict egress, private functions behind authenticated gateways, signed/authenticated event sources, patch + scan deps, set concurrency/timeout/memory, log request ID/source/auth/downstream.
  • Serverless removes server management, not identity/input/dependency/data-flow risk.

40. What are Lambda or Cloud Functions security risks?

  • Over-permissioned role/SA, secrets in env vars/logs, unauthenticated HTTP triggers, event injection from writable buckets/queues/topics, dependency vulns, insecure deserialization/command exec, excessive concurrency (cost/DoS), malicious cold-start packages, shared layers, public egress to attacker infra.
  • AWS: Lambda role reads all S3; S3-triggered function parses attacker file w/ vulnerable parser. GCP: function runs as default SA w/ broad perms; public HTTP trigger lacks auth.
  • Common failure: treating small functions as low-risk - a tiny function with a broad role is a large breach path.

Chunk 5 - Questions 41-50

41. How would you secure Kubernetes in cloud?

  • Spans control plane, node identity, pod identity, network, admission, secrets, images, observability.
  • Private clusters/restricted API server, strong authN + least-privilege RBAC, separate cluster-admin from app teams, workload identity (not node-wide perms), restrict privileged pods/hostNetwork/hostPID/hostPath/capabilities, admission policies (PSA/Gatekeeper/Kyverno), scan + sign images, network policies, secrets in cloud manager or KMS-encrypted, patch nodes/control plane, audit logs + runtime detection.
  • Prevent pod compromise from becoming node/cluster/cloud-account compromise.

42. What are EKS/GKE privilege escalation risks?

  • Cross from K8s perms to cloud perms or pod to node.
  • EKS: pods reach node IMDS + steal node role, over-permissive IRSA, misconfigured aws-auth granting cluster-admin, privileged pods access host FS/kubelet creds, broad node role.
  • GKE: broad node SA, missing Workload Identity, overbroad RBAC, privileged pods access node creds, legacy metadata endpoints.
  • Controls: IRSA / Workload Identity, block pod->node metadata, restrict privileged workloads, minimal node roles, audit RBAC + cloud IAM together.
  • Often cloud identity theft through the node/workload identity layer, not just cluster-admin risk.

43. How would you secure cloud network boundaries?

  • Reduce reachability while preserving required paths.
  • Segment by account/project/VPC/subnet/env/sensitivity; intentional SGs/firewall/NACLs/network policies; private/restricted management planes; private endpoints; restrict inbound via LB/IAP/VPN/zero-trust; restrict east-west; monitor flow + firewall logs; avoid broad peering that flattens trust; DNS controls.
  • Segmentation is about explicit reachability - if a workload doesn't need a service, the network shouldn't allow it.

44. How would you design egress controls for cloud workloads?

  • Limit where workloads send data + make exfil harder to hide.
  • Route outbound via controlled NAT/proxy/egress gateway, destination allowlists for high-risk, private endpoints for cloud APIs, block direct internet egress for sensitive subnets, inspect DNS/HTTP where allowed, log dest/identity/bytes/decision, per-env/data-class policies, alert on new destinations/high volume/rare geos.
  • AWS: NAT + egress proxy, VPC endpoints, SG egress, route tables. GCP: Cloud NAT, Private Google Access, firewall, VPC Service Controls.
  • Egress controls are one of the last chances to catch or slow data theft.

45. What is CSPM and what problems does it solve?

  • Cloud Security Posture Management: continuously finds misconfig, risky perms, exposure, compliance drift.
  • Solves: public storage exposure, broad IAM, missing encryption/logging, insecure network paths, exposed workloads, risky K8s config, baseline drift, compliance evidence.
  • Does NOT solve alone: runtime exploitation, app-layer authz bugs, business logic abuse, IR without logs, remediation ownership.
  • Turns cloud config into continuous security inventory - value depends on prioritization + remediation ownership.

46. How would you prioritize cloud misconfigurations?

  • By exploitability, asset sensitivity, internet exposure, privilege, blast radius, active-abuse evidence.
  • Critical: public bucket w/ PII, admin role assumable by external account, logging disabled in prod. High: SSH from internet on prod. Low: missing tag on dev VM.
  • Prioritize paths to breach, data exposure, and detection failure - compliance-only severity often needs security-context correction.

47. What are cloud security anti-patterns in large organizations?

  • One shared admin role for many teams, default SAs with broad perms, static keys in CI/CD, flat networks via broad peering, public buckets via manual review only, logs only inside workload accounts, exceptions w/o owner/expiry, one prod account with unrelated systems, KMS keys administered by data admins, CSPM findings w/o remediation workflow, treating AWS+GCP as identical.
  • Cloud security fails at scale when ownership, identity boundaries, and guardrails are unclear.

48. How would you map AWS security controls to GCP equivalents?

  • Map by function, not service name: Org/accounts/OUs <-> Org/folders/projects; IAM roles/instance profiles/IRSA <-> SAs/Workload Identity; CloudTrail <-> Cloud Audit Logs; GuardDuty/Security Hub <-> SCC; KMS/CloudHSM <-> Cloud KMS/HSM/EKM; S3 Block Public Access <-> public access prevention; VPC/SG/NACL <-> VPC/firewall/hierarchical policies; VPC endpoints/PrivateLink <-> Private Service Connect/Private Google Access; Secrets Manager <-> Secret Manager; Config/Security Hub <-> SCC/Asset Inventory/Policy Analyzer.
  • Preserve the control objective (identity, least privilege, logging, segmentation, encryption, detection) over exact service parity.

49. What cloud evidence would you collect during an incident?

  • Identity + control-plane + data-plane + network + workload + config artifacts.
  • CloudTrail/Audit Logs, IAM/role/SA history, STS/impersonation events, key creation events, KMS decrypt/admin, storage access logs + object versions, VPC flow/DNS/firewall/LB logs, instance/disk snapshots, container/K8s audit logs, function invocation logs, CI/CD logs + provenance, CSPM snapshots/asset inventory, GuardDuty/SCC/SIEM/EDR/WAF findings.
  • Collect volatile evidence before containment where possible, but don't delay containment during active harm.

50. What questions would you ask during a cloud architecture security review?

  • Account/project/VPC/tenant boundaries? Who can administer/deploy/assume/impersonate workload identities? What can each identity read/write/decrypt/modify? How is metadata protected? Where are secrets stored/rotated/audited?
  • Which data stores hold sensitive/regulated data? Audit logs centralized outside blast radius? Detections for IAM abuse/exfil/public exposure/logging tampering? Inbound/east-west/egress paths? Top privilege escalation paths from one compromised workload?
  • A strong review follows identity and data: who can act, what they reach, what they decrypt, what gets logged.
