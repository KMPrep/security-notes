enterprise data protection Cryptography Essentials - Cheatsheet

Source: questions/answers/cryptography-essentials-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Why is ECB mode insecure?

  • Encrypts each block independently with same key -> identical plaintext blocks = identical ciphertext blocks.
  • Leaks structure/equality patterns, no IV/nonce, no integrity, deterministic.
  • Encryption must hide value AND structure - ECB hides value but leaks equality patterns.

2. Give a concrete example of what ECB leaks.

  • Classic: ECB-encrypted bitmap still shows the image outline (repeated color blocks -> repeated ciphertext).
  • Enterprise: repeated DB status values (VIP/APPROVED), equal salaries -> equal ciphertext, repeated form/JSON templates.
  • Once attacker learns one mapping (correlate one known record), all equal values across the dataset become visible - ECB becomes a codebook.

3. Explain CBC mode and its main security risks.

  • XOR each plaintext block with previous ciphertext block; random IV for first block so identical messages differ.
  • Better than ECB but risky: needs unpredictable IVs + padding, confidentiality only (no integrity), padding-oracle vulnerable, malleable without MAC, history of TLS failures.
  • Rule: don't design new systems with raw CBC - use AEAD (AES-GCM or ChaCha20-Poly1305).

4. What is a CBC padding oracle attack?

  • System leaks whether decrypted padding is valid (error/status/timing/behavior) -> 1-bit signal recovers plaintext byte-by-byte WITHOUT the key.
  • Conditions: CBC + padding check (PKCS#7) + attacker submits chosen ciphertexts + padding validity observable.
  • Real-world: POODLE, Lucky13 - strong AES didn't fail, the composition + error behavior failed.

5. How would you prevent padding oracle attacks?

  • Best: avoid CBC, use AEAD; verify auth tag BEFORE processing plaintext; encrypt-then-MAC if CBC unavoidable.
  • Uniform errors for all decryption failures, constant-enough timing, don't expose partial plaintext, disable legacy TLS/CBC suites, use mature libraries.
  • Prevention is a composition problem - confidentiality without authenticated integrity is fragile.

6. What does AEAD mean?

  • Authenticated Encryption with Associated Data: confidentiality for plaintext + integrity/auth for ciphertext AND associated data.
  • Associated data = not encrypted but authenticated (protocol version, message type, tenant ID, key ID, sequence number).
  • Decryption returns plaintext ONLY if tag verifies. Examples: AES-GCM, ChaCha20-Poly1305, AES-GCM-SIV (nonce-misuse-resistant).

7. Why is authentication as important as confidentiality in encryption?

  • Confidentiality hides; authentication proves not-modified + from right-key holder.
  • Without auth: bit-flipping, padding oracle, chosen-ciphertext, truncation/replay, version/tenant confusion, silent corruption accepted.
  • Stream-mode bit flip = predictable plaintext bit flip (admin=false tampering).
  • Encryption without auth = sealed box attackers can't read but CAN tamper with. Default to AEAD.

8. How does AES-GCM provide confidentiality and integrity?

  • AES counter mode produces keystream -> XOR plaintext = ciphertext; GHASH authenticates ciphertext + associated data -> tag; decrypt verifies tag before accepting.
  • Requirements: unique nonce per encryption per key (prefer 96-bit), rotate keys before nonce limits, never ignore tag failures, authenticate context (tenant/object/version) as AAD.
  • Secure when nonce uniqueness + tag verification are mandatory invariants.

9. What happens if an AES-GCM nonce is reused with the same key?

  • Catastrophic: reuses CTR keystream AND breaks authentication.
  • Confidentiality loss (compare ciphertexts, recover plaintext relationships) + integrity loss (forge valid tags).
  • Causes: random collision at high volume, counter reset on restart, multiple nodes independent counters, VM snapshot restore, hardcoded test nonces.
  • Prevent: library-managed nonces, durable monotonic counters, partition keys by service/shard, rotate before nonce exhaustion, misuse-resistant modes. Treat reuse as a cryptographic incident.

10. Compare AES-GCM and ChaCha20-Poly1305.

  • AES-GCM = AES-CTR + GHASH; very fast with AES-NI/hardware; servers/cloud.
  • ChaCha20-Poly1305 = ChaCha20 stream + Poly1305 MAC; fast in software without AES hardware; mobile/embedded/older CPUs.
  • Both: unique nonce per key required, nonce reuse severe, auth tag.
  • Both good defaults; choice depends on platform/performance/side-channel/library support. Let TLS negotiate.

Chunk 2 - Questions 11-20

11. When would you prefer ChaCha20-Poly1305 over AES-GCM?

  • When no fast constant-time AES hardware, or performance across diverse clients matters more than server CPU optimization.
  • Mobile/IoT/embedded/older CPUs, heterogeneous browser clients, AES software side-channel risk, protocols with safe negotiation (TLS).
  • ChaCha20 fast in software + easier constant-time than table-based AES; Poly1305 strong MAC.
  • Choose on real platform properties + library support, not "AES sounds enterprise / ChaCha sounds modern."

12. Why are DES and 3DES no longer acceptable?

  • DES = 56-bit key, brute-forceable. 3DES = slow, 64-bit block (birthday-bound risk at high volume), deprecated.
  • Legacy DES/3DES modes often lack authenticated encryption; compliance baselines reject them.
  • Migrate: inventory -> disable for low-risk/internal first -> measure breakage -> exception path w/ expiry -> TLS 1.2/1.3 modern AEAD -> re-encrypt stored data.
  • Weak ciphers = operational debt: inventory + migration + exception governance, not one config flag.

13. What is forward secrecy?

  • Past session traffic stays protected even if long-term private key is later compromised.
  • Achieved via ephemeral DH: each session uses fresh ephemeral material; session key not derivable from cert private key later.
  • Turns key compromise from history compromise into go-forward incident (assuming sessions/endpoints otherwise secure).

14. How does ephemeral Diffie-Hellman provide forward secrecy?

  • Fresh key-exchange values per session; long-term key authenticates handshake, session secret from ephemeral values discarded after use.
  • Passive observer sees only public values; later cert-key theft doesn't reveal erased session secrets.
  • Failure modes: reusing ephemeral values, weak randomness, unauthenticated DH (MitM), logging session keys, obsolete groups.

15. Compare RSA key exchange and ECDHE key exchange.

  • RSA: client encrypts premaster to server RSA key -> NO forward secrecy; cert-key theft decrypts recorded old sessions. Legacy.
  • ECDHE: both derive secret via ephemeral ECDH -> forward secrecy; cert key only authenticates. Modern standard.
  • RSA binds confidentiality to long-term key; ECDHE separates authentication from session secrecy.

16. What is the difference between encryption and signing?

  • Encryption = confidentiality ("who can read this?"); recipient key decrypts.
  • Signing = authenticity + integrity ("who produced/approved this?"); signer private key signs, public key verifies.
  • JWT signature doesn't hide claims - proves issuer. TLS: certs/signatures authenticate, then symmetric encryption for traffic.
  • Encryption w/o signing may hide tampered data; signing w/o encryption proves origin but exposes content - many systems need both.

17. What is the difference between hashing and encryption?

  • Hashing = one-way, no key (fingerprints, integrity, indexing, password KDFs). Encryption = reversible with key (confidentiality).
  • Plain hash doesn't prove authenticity (anyone can compute) - use HMAC or signature for that.
  • Need original back -> hashing wrong. Need only fingerprint -> encryption unnecessary/risky.

18. What is the difference between MAC and digital signature?

  • MAC = shared secret (verifier with key can also create valid value -> no non-repudiation); fast; service-to-service/API auth.
  • Signature = private signs / public verifies (verifier can't forge); non-repudiation; software signing/certs/public verification; slower.
  • Use MAC when all verifiers are trusted with signing capability; signatures when verification must be public/separated from signing.

19. Explain HMAC and why plain hashing is not enough.

  • hash(message) is computable by anyone; HMAC(secret_key, message) requires the secret.
  • Uses: webhook signatures, internal request signing, API auth, stored-metadata integrity, symmetric token signing.
  • Controls: SHA-256+, constant-time compare, key rotation + key IDs, include method/path/body-hash/timestamp/tenant/nonce.
  • Plain hash answers "did bytes change vs this digest?"; HMAC answers "did someone with the key authorize these bytes?"

20. What is length-extension attack and why does HMAC avoid it?

  • Affects Merkle-Damgard hashes (MD5/SHA-1/SHA-256) misused as hash(secret || message): attacker who knows that hash + secret length can compute valid hash for appended data without the secret.
  • HMAC uses nested inner/outer keyed pads - output isn't the internal hash state, so attackers can't continue hashing.
  • Never build your own keyed hash by concatenating secret + message - use library HMAC.

Chunk 3 - Questions 21-30

21. How should passwords be hashed?

  • Password-specific, slow, salted, memory-hard function - never raw SHA-256.
  • Argon2id (modern default), bcrypt (legacy), scrypt, PBKDF2 (high iterations, compliance).
  • Unique random salt per password, work factor tuned to hardware, optional pepper in KMS/HSM, versioned format, rehash on login when outdated, plus rate limiting + MFA (hashing doesn't stop online guessing).
  • Password storage slows offline cracking after DB theft - not a replacement for MFA/breach detection/abuse controls.

22. Why should passwords not be encrypted for storage?

  • Encryption is reversible - app (or attacker with the key) can recover every password.
  • Store Argon2id(password, salt, params), verify by re-hashing the attempt.
  • Reuse across services amplifies impact; app never needs the original; reset should issue a reset flow, not read a stored password.
  • Exception: credential vaulting to authenticate to legacy downstream (not user password storage) - needs strong key mgmt.

23. What is a key derivation function?

  • KDF derives one or more keys from input key material (secret/password/master key).
  • Uses: TLS traffic keys from handshake secret, encryption+MAC keys from master, storage key from passphrase, expand KMS data key.
  • Types: password KDF (Argon2id/bcrypt/scrypt/PBKDF2) for humans; extract-and-expand (HKDF) for high-entropy protocol secrets; KMS derivation for envelope encryption.
  • Separate keys by purpose, bind context, use salts/nonces, use standard library KDFs, don't reuse one raw secret for many algorithms.

24. How would you derive encryption keys from a passphrase?

  • Low entropy -> need password KDF with salt + strong work factors.
  • Unique random salt, Argon2id/scrypt/PBKDF2 strong params, store algo+param version in metadata, derive AEAD key (AES-GCM/ChaCha20-Poly1305), unique nonce, authenticate metadata as AAD, support upgrades.
  • Risks: weak passphrase still crackable offline, salt reuse, low cost = cheap brute force, lost passphrase = unrecoverable data.
  • Passphrase-derived key only as strong as passphrase + KDF params; for enterprise prefer random data keys protected by KMS.

25. What makes randomness cryptographically secure?

  • Unpredictable even after seeing many prior outputs; from a properly seeded CSPRNG, not a statistical PRNG.
  • Requirements: high-entropy seed, prediction resistance, safe after fork/snapshot/restart, no predictable seeds (time/PID/username), correct OS/language API.
  • Safe APIs: Linux getrandom()//dev/urandom, Windows BCryptGenRandom, Python secrets/os.urandom(), Node crypto.randomBytes(), Go crypto/rand, Java SecureRandom.
  • Secure when attackers can't predict AND operators can't accidentally make it deterministic.

26. What can go wrong with poor entropy?

  • Predictable keys, repeated AEAD nonces, guessable tokens/reset links, repeated ECDSA nonces (reveal private key), identical keys across cloned VMs, weak early-boot SSH/TLS keys.
  • Strong algorithm + weak generated key = effectively weak (enumerable keys).
  • Mitigate: OS CSPRNG, ensure early-boot randomness, provide entropy to VMs/containers, avoid custom RNG, detect duplicate keys/repeated nonces, KMS/HSM generation.
  • Poor entropy is a root cause - turns brute-force problems into lookup-table problems.

27. How would you design secure key generation?

  • Trusted, auditable systems + approved algorithms + strong randomness.
  • Define purpose -> choose algo+size -> generate via CSPRNG/KMS/HSM -> bind metadata (owner/purpose/env/classification/rotation) -> store in KMS/HSM/secrets manager (not code/config) -> restrict usage -> log create/use/rotate/disable/delete -> test backup/recovery.
  • Failure modes: local generation pasted to config, same key across environments, one key for signing+encryption, no owner/rotation, generated before entropy ready.

28. How would you design key rotation for a large enterprise?

  • Separate wrapping keys / data keys / signing keys / certs / secrets (different semantics).
  • Inventory keys+owners+deps+data; add key IDs to ciphertexts/signatures; support multiple active decrypt/verify keys during transition; new encryptions use new key; lazily/batch re-encrypt old; monitor old-key decrypts; disable before delete; keep rollback.
  • Rotation is a compatibility + data-recovery project - never delete before proving nothing still needs decrypt.

29. What are the risks of manual key rotation?

  • Humans miss dependencies / wrong order: outage (can't decrypt old data), permanent loss (delete old key too soon), partial rollout, secrets leaked in tickets/chat/terminals, missed replicas/backups/regions/DR, inconsistent IAM updates, no audit trail, insecure rollback sharing.
  • Controls: automate workflows, KMS/HSM lifecycle, key IDs + versioned metadata, canaries + integration tests, stage disablement before deletion, alert on old-key use.
  • Manual rotation often increases risk during the moment you're trying to reduce it.

30. How would you design envelope encryption?

  • Data key encrypts data; key-encryption key (in KMS) encrypts the data key (KMS ops expensive/central, local encryption fast).
  • AEAD for local encryption, store encrypted data key with ciphertext, encryption context as AAD + KMS audit, discard plaintext data key after use, tight KMS decrypt perms, monitor decrypt spikes, rotate wrapping key without re-encrypting all payloads.
  • AWS: GenerateDataKey -> encrypt file locally -> store encrypted data key beside ciphertext in S3 -> Decrypt only when authorized.

Chunk 4 - Questions 31-40

31. What is the role of KMS in cloud security?

  • Managed creation/storage/use/policy/rotation/audit of keys - both confidentiality control AND operational control.
  • Protect material in HSM-backed boundary, enforce who can encrypt/decrypt/sign/admin, integrate with services, support envelope encryption, log usage, rotation/disablement, separation by account/region/env/app/sensitivity.
  • kms:Decrypt is often the real permission to sensitive data; encryption context binds decrypt to tenant/object/path; monitor decrypt spikes + policy changes.
  • KMS is a policy + audit boundary for key use - design its authz as carefully as IAM.

32. How would you protect master keys?

  • KEKs protect many data keys/signing ops - compromise exposes lots of data or enables impersonation.
  • Store in KMS/HSM (not memory/files), prevent export, least privilege per operation, separate admin from usage, MFA/privileged workflow for destructive actions, monitor decrypt/sign/policy/grant/disable/delete, separate keys by env/app/sensitivity/tenant, protect backups, deletion waiting periods + disable-before-delete.
  • Failure modes: one shared key for high+low sensitivity, broad admin can delete/re-policy, stale grants after decommission, untested recovery.

33. How would you handle key compromise?

  • Depends on key type + what it protected + whether past data is at risk.
  • Identify type -> determine exposure window/logs -> disable/revoke if safe -> generate replacement -> rotate services -> re-encrypt if confidentiality at risk -> re-sign/revoke artifacts if signing affected -> invalidate tokens/sessions if MAC/signing key -> fix root cause -> monitor abuse.
  • Impact: AEAD data key (that data exposed), KMS wrapping key (many data keys), TLS key w/o FS (recorded traffic), signing key (forge artifacts), HMAC key (forge messages/tokens).
  • New key doesn't solve it - understand what the old key authorized or protected.

34. What is certificate validation?

  • Client decides whether a cert should be trusted for the identity it claims.
  • Checks: chain to trusted root, valid signatures, within validity period, hostname matches SAN, key usage/EKU allow purpose, not revoked, algorithm/size meet policy, name/path-length constraints, CT for public web certs.
  • Binds public key to expected identity - skipping hostname validation or trusting any cert defeats TLS authentication.

35. What fields in a certificate must a TLS client validate?

  • SAN matches hostname (not legacy CN), validity period (Not Before/After), issuer+signature chain, Basic Constraints (CA flag), Key Usage, EKU (server auth), serial (revocation), public key algo/size, name constraints, revocation (CRL/OCSP).
  • A cert chaining to a root is not enough - must be valid for this name, this time, this purpose, this policy.

36. What is certificate pinning and when can it cause operational risk?

  • Client accepts only a specific cert/public-key/CA instead of the whole CA ecosystem.
  • Used to reduce rogue/compromised CA risk, protect high-risk mobile/APIs, enforce private PKI.
  • Operational risk: rotation breaks clients if pins not updated, hard emergency CA migration, lost key = outage if no backup pins, long-lived mobile clients keep old pins, leaf-pinning brittle, harder enterprise TLS interception debugging.
  • Safer: pin to public keys/small key set (not leaf), backup pins, realistic lifetimes, remote config + recovery plan, CT monitoring as complement.

37. What is certificate transparency?

  • Public append-only logging of certificates - makes issuance visible/auditable so domain owners detect misissuance.
  • CA submits precert -> log returns SCT -> final cert includes proof -> browsers require CT for public certs -> monitors watch logs.
  • Provides public visibility + faster misissuance detection + tamper-evident logs. Does NOT prevent all misissuance, encrypt DNS/TLS, or replace validation.
  • CT turns silent CA failure into detectable CA failure.

38. How would you detect a mis-issued certificate?

  • Monitor CT logs + cert inventories + DNS for company + lookalike domains.
  • Alert on unknown issuer/unexpected SAN/wildcard/unusual key; compare vs approved issuance; monitor CAA records; inventory public certs from scans; correlate with deployments; investigate external reports.
  • Respond: confirm authorized? -> CA revocation if misissued -> rotate keys if private-key exposure -> update CAA/CA account controls -> monitor phishing use -> document.
  • Key is knowing what legitimate issuance looks like - CT is noisy without an approved inventory + ownership model.

39. What is mutual TLS and when is it useful?

  • Both sides present + validate certs (normal web TLS authenticates only the server).
  • Useful: service-to-service in a mesh, workload identity in zero trust, admin/high-risk internal APIs, partner API (stronger than IP allowlists), device identity.
  • Needs private PKI/mesh identity, automated issuance/renewal, revocation or short-lived certs, identity-to-authz mapping, handshake-failure observability.
  • mTLS authenticates the workload - it does NOT replace application authorization (knowing client identity is step one).

40. What are common TLS misconfigurations?

  • SSL/TLS 1.0/1.1 enabled, weak/CBC suites, expired certs, missing/wrong SAN, clients skip hostname validation, broadly-trusted self-signed, TLS terminated at LB but plaintext internal hop, private keys on disk with broad access, no auto-renewal, no CT monitoring, downgrade/fallback, 0-RTT for replay-sensitive actions, no mTLS for sensitive internal APIs.
  • Remediate: modern TLS baseline, continuous scanning, automate issuance/renewal, re-encrypt internal hops or document trust boundaries, track exceptions w/ owners+expiry.
  • Most TLS incidents are operational, not mathematical - lifecycle + validation hygiene matter as much as cipher choice.

Chunk 5 - Questions 41-50

41. How would you disable weak cipher suites safely?

  • Inventory + staged rollout + compatibility measurement + exception governance.
  • Define baseline -> inventory endpoints/clients/LBs/SDKs/partners -> measure real weak-suite usage -> disable in canary -> roll out by tier/geo -> monitor handshake failures/latency/impact -> temporary exceptions w/ owners+expiry -> remove after upgrade.
  • Remove: SSL/old TLS, export-grade, RC4/DES/3DES/null, anonymous KX, weak DH groups, CBC where AEAD available.
  • Cipher retirement is a production migration - breaking legacy clients should be an explicit data-driven risk decision.

42. How would you store secrets securely?

  • Dedicated secrets manager / KMS-backed, not code/images/tickets/config.
  • Central manager + encryption at rest, IAM least privilege, short-lived dynamic creds, auto-rotation, audit logs, versioning+rollback, separate by env, no secrets in logs/dumps/telemetry, workload access via identity not static files.
  • AWS: Secrets Manager (DB creds), Parameter Store (config), KMS (encryption+audit).
  • Secure storage = access control + rotation + audit + minimizing plaintext lifetime, not only encryption.

43. What is secret zero in secrets management?

  • The initial credential/trust bootstrap a workload uses to obtain all other secrets - if weak, the secrets manager doesn't help.
  • Good bootstrap: cloud workload identity (instance role/SA), K8s audience-bound tokens, mTLS short-lived certs, hardware-backed device identity, CI OIDC federation.
  • Bad: static vault token in code, shared bootstrap secret, long-lived broad CI token, manual copy of root creds.
  • The first credential must come from a stronger identity boundary than the secrets it retrieves - else you only moved the hardcoded secret one layer.

44. How would you prevent secrets from leaking into source code?

  • Controls before commit, during push, after merge, across history: education, pre-commit scanning, server-side push protection, CI scanning (commits/PRs/artifacts), full-history scans, block high-confidence secrets, easy approved secret storage, restrict forks of sensitive repos, auto-rotate on confirmed leak.
  • The prevention path must be faster than the unsafe path - if getting a dev secret safely takes days, devs paste secrets into code.

45. How would you scan for exposed secrets?

  • Scan source/history/forks, CI logs/artifacts, container images/registries, cloud storage/drives, tickets/wikis/chat, public code/paste sites, logs/crash dumps.
  • Methods: regex for known formats, entropy checks, context keywords, provider-API validation when safe, allowlist test tokens.
  • Respond: confirm real -> revoke/rotate immediately -> identify exposure window/usage -> remove/rewrite (assume history copied) -> add prevention rule.
  • Deleting the commit isn't enough - once committed/published, treat as compromised and rotate.

46. How would you design crypto for data at rest?

  • Protect stored data if media/snapshots/backups/object stores accessed without authz - pair with access control + key management.
  • Classify data + scope, envelope encryption with KMS/HSM keys, AEAD/storage modes, separate keys by env/tenant/app/sensitivity, bind encryption context to object/tenant/dataset, encrypt backups/replicas/snapshots, log decrypt + key admin, plan rotation + DR, prevent plaintext in logs/caches.
  • Examples: S3 SSE-KMS + bucket policy, DB TDE + field-level for most sensitive columns, client-side envelope encryption for operator-opaque data.
  • At-rest encryption reduces blast radius, but "who can call decrypt" is the real question.

47. How would you design crypto for data in transit?

  • Protect from eavesdropping, tampering, endpoint impersonation.
  • TLS 1.3 (modern 1.2 where needed), correct cert validation, mTLS for service-to-service/high-risk client auth, automate issuance/renewal, disable weak protocols/suites, re-encrypt after LB termination unless internal trust documented, correct ALPN/SNI, monitor handshake failures + expiry, no custom transport crypto.
  • TLS protects the channel, not the application - authN/authz/replay/input validation still matter above it.

48. How would you design crypto for field-level sensitive data?

  • Protect specific sensitive fields even when broader DB/storage access exists (PII, payment, secrets, high-value IDs).
  • Envelope encryption or tokenization, dedicated encrypt/decrypt service/library, deterministic only when equality search needed (risk accepted), randomized AEAD otherwise, bind tenant+field+record-ID as AAD, separate keys by tenant/domain, control + audit decrypt, design for rotation/re-encryption.
  • Tradeoffs: breaks search/sort/join/analytics, deterministic leaks equality, app bugs log plaintext, key access becomes real authz boundary.
  • Most useful when DB access is broader than data need - must pair with clear decrypt authorization.

49. What are common mistakes developers make when using crypto libraries?

  • ECB/raw CBC for new designs, reused GCM nonces, ignoring tag failures, encryption without authentication, Math.random() for keys/tokens/nonces, SHA-256 for passwords, custom hash(secret || message) signing, skipping hostname validation, hardcoded keys, key reuse across purposes/envs, logging plaintext secrets, treating Base64 as encryption, custom crypto protocols instead of TLS/age/JOSE/KMS.
  • Most crypto bugs are misuse bugs - safest path is high-level libraries + safe defaults + review by people who know failure modes.

50. What questions would you ask before approving a custom cryptography design?

  • First challenge: is it needed at all? Most systems should use standard protocols + managed key services.
  • Ask: what threat model? why don't existing protocols solve it? which properties (confidentiality/integrity/authenticity/forward-secrecy/replay/non-repudiation)? primitives+modes? key gen/store/rotate/revoke/destroy? nonce generation+anti-repeat? what's authenticated as AAD? failure behavior? versioning? algorithm agility without downgrade? misuse detection logs? expert review? migration/recovery plan?
  • Default: standard reviewed protocol. Exception: custom only with explicit threat model + expert review + tests + operational ownership.
  • Custom crypto is rarely justified; even custom composition around standard primitives needs production-grade discipline.
