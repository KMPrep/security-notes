enterprise data protection Detection and Incident Response - Cheatsheet

Source: questions/answers/detection-incident-response-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Investigate an impossible-travel login alert.

  • Treat as compromise HYPOTHESIS not proof - could be VPN/proxy/cloud-IP/mobile-carrier/automation.
  • Build auth timeline: both logins' IPs, ASN, geo, device, user agent, session ID, MFA result, IdP risk, token issuance vs baseline (countries/hours/devices).
  • Inspect session quality (new device fingerprint, failed-then-success, MFA fatigue) + follow-on (mailbox rules, OAuth grants, exports, IAM changes, recovery-factor changes).
  • Contain on confidence: revoke sessions+refresh tokens, force reset, re-enroll MFA, kill risky OAuth, preserve logs. False positive -> tune with VPN/mobile context, don't suppress globally.

2. Investigate suspected DNS exfiltration.

  • Covert channel encoding data in subdomains (chunk1.secret.attacker.com) - distinguish from resolver/CDN/C2/real movement.
  • Profile domain: query volume/length/entropy, unique subdomain count, NXDOMAIN rate, TXT queries, TTLs, timing regularity, bypass of corporate resolver.
  • Pivot to source host process tree, parent, command line, EDR detections; for servers check user input reaching vulnerable component (blind exfil).
  • Contain: preserve DNS logs + endpoint snapshot, block domain/egress, force approved resolvers, isolate host, rotate exposed secrets. Long-term: resolver policy, length/entropy detection, egress filtering, canary tokens.

3. Investigate suspected IAM credential compromise.

  • Assume attacker already has valid API access - focus on SCOPE (which principal, which creds, what permissions, what touched, persistence?).
  • AWS: CloudTrail mgmt+data events, credential report, access-key-last-used, STS AssumeRole, source IP/user agent, API sequences, IAM/policy changes.
  • Attacker pattern = discovery (List/Get/Describe) -> privilege testing -> persistence (new keys/users/trust policy) -> data access.
  • Validate exposure path (GitHub/CI logs/endpoint/metadata/third-party) - rotating one key doesn't fix a leaking pipeline. Contain: disable key, revoke sessions, trim policies, rotate downstream, audit all IAM changes for backdoors.

4. Investigate a user downloading unusually large amounts of data.

  • Data exfil + insider-risk lens; volume alone != malicious (migration/analytics/legal/backup possible). Coordinate Legal/HR/Privacy.
  • Identify the data (repos/SaaS/DB/object store/PII) + access pattern (time, device, IP, auth strength, sensitivity labels, destination) vs role/peer baseline + recent events (resignation, litigation hold).
  • Cross-correlate: SaaS audit (downloads), endpoint DLP (USB/personal cloud/print/clipboard/archive), proxy (uploads), identity (new device), warehouse (SELECT * on sensitive tables).
  • Proportionate containment; don't tip off the user if insider investigation active (destroys evidence). Benign -> least privilege/JIT/export approvals/watermark/coaching.

5. Investigate a suspicious OAuth consent grant.

  • Dangerous = persistent API access WITHOUT password. Investigate app, consenting user, scopes, publisher, redirect URIs, tokens, post-consent API activity.
  • High-risk scopes: offline_access, mail read, mailbox settings, files read/write, calendar, directory read, app-only.
  • Watch tenant-wide admin consent (one compromised admin -> org-wide exposure); unverified publisher, lookalike name, newly registered app, consent right after phishing.
  • Contain: revoke grant + refresh tokens, block/delete app registration, review admin-consent policy, hunt data accessed pre-revocation. Prevent: verified publisher, admin approval for risky scopes, OAuth-phishing education.

6. Investigate a service account being used from an unusual location.

  • Service accounts don't travel/browse/MFA - unusual location suggests key leak, workload-identity misuse, metadata token theft, CI/CD compromise.
  • Identify credential type (static key = highest risk, copyable anywhere; short-lived token -> investigate minting workload); check source IP/ASN/region/API sequence vs normal project/namespace/data domain.
  • Map effective permissions + trust (read secrets? assume roles? KMS decrypt? export? change IAM? create creds?); look for discovery, denied calls, new persistence.
  • Contain by credential model: leaked static key -> disable + rotate downstream + find leak; workload token -> isolate workload, inspect metadata/pod-VM compromise. Long-term: short-lived creds, no downloadable keys, egress restrictions, source-bound policy.

7. Investigate an alert that CloudTrail or audit logging was disabled.

  • High severity - attackers blind defenders before escalation/data access/destruction. Treat as active compromise until disproven.
  • Preserve remaining: org-level logs, SIEM copies, provider event history, config snapshots, out-of-band logging.
  • Answer who/what credential/from where + what happened before+after. Events: StopLogging, DeleteTrail, PutEventSelectors, UpdateTrail, S3 log bucket policy, KMS key policy, sink deletion. Check single-region/account targeting.
  • Scope IAM changes if principal was privileged; use secondary sources (VPC flow, DNS, EDR, config history, billing). Contain: re-enable hardened, revoke principal, rotate creds, object-lock logs, multi-party approval. Mature = org-level logging workload admins can't disable.

8. Investigate suspected secret exposure in GitHub.

  • Incident even if removed fast - history/forks/clones/CI logs/caches/mirrors persist. Exposure starts at FIRST commit, not scanner-find time.
  • Triage: secret type/owner/scope/privileges/validity, repo visibility, branch, commit, author, PR, forks, Actions logs, artifacts. Check post-exposure use (cloud API/DB connection/token issuance).
  • Containment = ROTATION not deletion: revoke, replace via secret manager, update dependents, invalidate derived tokens, expand blast radius if it granted access to other secrets/KMS/prod.
  • Prevent: pre-commit scanning, push protection, CI redaction, short-lived creds, workload identity. Devs must know removing the line from latest commit isn't enough.

9. Investigate suspicious API enumeration behavior.

  • Recon phase before data theft/IDOR/cloud escalation. Pattern: high volume, many distinct object IDs, many 403/404, sequential IDs, repeated List/Describe, cross-tenant access.
  • Inspect identity+client (user/SA, auth method, token age, IP, user agent, API key, new/changed behavior) vs authorization boundaries; worse when errors differ for existing vs non-existing objects or 403s leak metadata.
  • Cloud = ListUsers/ListRoles/DescribeInstances/ListSecrets + many denied; SaaS/app = walking IDs, scraping search, changing tenant IDs, GraphQL introspection. Look for follow-on export/download/permission-change.
  • Contain: block token/client, adaptive rate limits, disable suspicious keys, add authz checks. Fix: object-level authz, tenant-aware constraints, consistent errors, detect by distinct-resource count not raw request count.

10. Investigate a suspected insider data leak.

  • Rigorous, discreet, coordinated with Legal/HR/Privacy. Don't over-collect employee data or tip off prematurely.
  • Timeline of access+movement: auth, device posture, repos, sensitivity labels, downloads, USB writes, print, archives, personal email/cloud, code cloning vs role/peer/HR events.
  • Evidence handling critical: preserve per policy/jurisdiction, need-to-know access, prefer metadata over content, privacy counsel guides boundaries.
  • Proportionate containment (access reduction, session revoke, device preservation, legal hold). Post: least privilege, JIT, classification, export approvals, watermarking, DLP, manager-driven reviews.

Chunk 2 - Questions 11-20

11. What makes a detection rule high fidelity?

  • Strong relationship between alert condition and real malicious/policy-violating behavior + enough context to decide quickly (what/who/why/evidence/next-action).
  • Combine multiple signals not one weak indicator ("new country" weak; "new country + new device + failed MFA + new mailbox rule + OAuth grant" strong).
  • Include entity context: asset criticality, role, SA baseline, data sensitivity, maintenance windows; prefer durable TTP logic over brittle IOCs.
  • Operational: clear owner, test cases, severity rationale, response playbook, suppression logic. No playbook = not production-ready.

12. What makes a detection rule noisy?

  • Fires often without producing useful decisions: broad conditions, missing context, thresholds ignoring baseline, no suppression, poor asset scoping, alerting on normal admin behavior ("any PowerShell execution").
  • Using a weak signal as if strong (foreign IP without VPN context, failed logins without distinct-user logic, download volume without role baseline).
  • Data quality issues: missing user IDs, NATed IPs, duplicate events, bad timestamps break correlation - verify input cleanliness before tuning logic.
  • Cost beyond analyst time: trains responders to distrust alerts, hides real attacks, encourages bulk-closing. Tune, downgrade to risk signal, or retire.

13. How would you reduce alert fatigue?

  • Treat detection as engineering system with measurement/ownership/lifecycle. Quantify first: volume by rule, FP rate, duplicate rate, MTTT, closure reasons - find the few noisy rules.
  • Improve signal: allowlist known automation, entity baselines, require multiple weak signals, weight by asset criticality, suppression windows, convert low-confidence to risk scores.
  • Improve workflow: every alert needs owner/severity/runbook/evidence-fields/auto-enrichment (role, device posture, geo, recent changes) so analysts skip repetitive lookups.
  • Detection review loop: retire never-true-positive rules, tune valuable-but-noisy with test data, shadow-mode new rules. Track alert QUALITY not just count.

14. How would you tune a detection without creating blind spots?

  • Document the objective first (threat technique, protected asset, attacker action, minimum trigger evidence) - else tuning removes core value.
  • Analyze FPs as CLUSTERS (one automation account/subnet/admin tool/maintenance window) -> targeted tuning safer than broad suppression (suppress signed backup binary on specific hosts during known schedule, still alert on unusual repos/new host).
  • Staged tuning: measure what a change would suppress in comparison window, replay known true positives / attack simulations, keep raw events searchable even when alerts suppressed.
  • Record why/evidence/residual-risk/approver/review-date; suppression should EXPIRE - permanent exceptions become blind spots.

15. How would you design detections for credential stuffing?

  • Many username-password attempts from leaked creds - look for distributed abuse not failures against one user.
  • Signals: high distinct-username count per IP/ASN, high failure rate across accounts, same user-agent/device fingerprint, failures-then-few-successes, hosting/proxy networks, credential-pair hash (no raw passwords).
  • Separate from spraying: stuffing = many user:pass combos, succeeds on reused passwords; spraying = few common passwords across many accounts. Response differs (bot mitigation + breached-password checks vs lockout strategy).
  • Controls: multi-key rate limits, bot detection, CAPTCHA/PoW, breached-password screening, MFA, step-up. Avoid IP-only hard blocks (NAT/mobile collateral damage).

16. How would you design detections for password spraying?

  • Low-and-slow: few common passwords across many accounts to avoid lockout. Core = high distinct target users, low failures per user, same source/ASN/user-agent/window.
  • Signals: one IP failing many usernames, regular intervals, off-hours, common/admin usernames, legacy protocols without MFA (IMAP/POP/SMTP AUTH/VPN/RDP), failures-then-success.
  • Use multiple grouping keys (if IP rotates -> group by user agent, TLS fingerprint, ASN, geo, app ID); tenant-wide spray = many users many countries same app/protocol.
  • Response: block/challenge source, disable legacy auth, force MFA/step-up for targets, check post-spray successes. Prevent: passwordless, banned-password lists, smart lockout, conditional access.

17. How would you design detections for token theft?

  • Harder than passwords - reusing issued session/OAuth refresh/API/STS bypasses MFA. Focus on token use violating expected binding (new IP/device, impossible travel, abnormal API sequence, use after reset).
  • User sessions: refresh redemption from new location, cookie reuse across devices, user-agent/TLS-fingerprint change, sensitive actions soon after new-env use. Cloud: STS used outside workload network or after workload terminated.
  • Best when tokens are context-bound (device-bound, cert-bound, workload identity, short lifetime, continuous access evaluation). Long-lived bearer tokens make every leak high-impact.
  • Respond: revoke sessions+refresh tokens, rotate API keys, inspect source endpoint/workload, hunt actions, check persistence (new OAuth/backup codes/recovery factors/keys/trust policy).

18. How would you design detections for session hijacking?

  • Unauthorized use of active session/cookie/token - attacker may not know password, may not trigger new login. Monitor session behavior across lifetime, not just auth.
  • Signals: session ID from two distant IPs, user-agent/device-fingerprint change mid-session, sudden privilege-sensitive actions, abnormal navigation, reuse after logout, concurrent use from different networks.
  • Look for theft paths (XSS, endpoint malware, insecure cookies, leaked HAR files, malicious extension); multiple users hijacked from same destination -> shared root cause not independent compromise.
  • Controls: secure cookie flags, short lifetime for sensitive apps, token rotation, device-posture binding, re-auth for sensitive actions, CSP/XSS prevention. Respond: kill session, invalidate tokens, inspect endpoint, review actions.

19. How would you design detections for cloud privilege escalation?

  • Focus on permission-CHANGING actions and suspicious CHAINS, not isolated API calls. Pattern: discovery -> find path -> persistence -> data access.
  • Signals: role-assumption anomalies, new access keys, policy attachment, trust-policy modification, permission-boundary removal, KMS policy change, logging disablement.
  • Model known paths: principal granting itself permissions, creating assumable role, updating function under privileged role, PassRole abuse, resource policy allowing external access, new IdP.
  • Context: CI role may update function code but not modify IAM trust at midnight from new IP. Respond: freeze principal, capture IAM state, revert changes, rotate, check persistence. Prevent: permission boundaries, SCPs, separation of duties, policy-as-code tests.

20. How would you design detections for DLP bypass.

  • Identify attempts to move data through channels/formats/transforms that avoid enforcement (compress, encrypt, rename, screenshot, chunk, encode, print, USB, personal cloud, unmanaged SaaS).
  • Start from sensitive-data inventory + approved egress paths; detect abnormal movement: archive creation after sensitive access, encrypted ZIP/7z, bulk download-then-upload, repeated DLP block then different channel.
  • Correlate endpoint (USB/archive) + CASB (SaaS sharing) + proxy (uploads) + warehouse (bulk export) + identity (new device) - single event benign, the SEQUENCE tells the story.
  • Tune with business context (engineers move test data, finance exports reports). Prevent: classification, tokenization, export approval, managed browsers, personal-cloud blocking, canary records.

Chunk 3 - Questions 21-30

21. What logs are needed for authentication investigations?

  • Need HOW the proofing decision was made, not just success: user ID, IP, geo, device ID, user agent, app, protocol, MFA method+result, conditional-access result, session/token ID, failure reason, correlation ID.
  • Sources: IdP, app login, VPN/ZTNA, MFA provider, password-reset/recovery, device posture, EDR, PAM; Windows DC logon success/failure/explicit-cred/privileged/lockout/group changes.
  • Must support timeline reconstruction (failed attempts -> reset -> MFA enroll -> success -> token -> SaaS access -> revocation); mismatched IDs/timezones cause wrong conclusions.
  • Retention matters - attacks found days/weeks later. Centralize, make tamper-resistant, normalize, searchable by user/device/IP/session/app/token.

22. What logs are needed for cloud IAM investigations?

  • Control plane audit + IdP + resource access + config history. AWS: CloudTrail mgmt (baseline, not enough) + data events for sensitive services, credential reports, access analyzer, config snapshots, KMS usage, flow/DNS logs, SIEM copies OUTSIDE the affected account.
  • Fields: principal ARN, assumed-role chain, source identity, session name, external ID, source IP, user agent, action, target resource, params, response code, denied perms, MFA context, region; for federation need upstream IdP user/workload.
  • Key categories: credential create/use, role assumption, policy attach, trust-policy change, permission-boundary change, org-policy change, logging change, KMS policy, secrets access, storage reads, snapshot sharing, compute creation - correlate the CHAIN.
  • Architecture must stop workload admin deleting own evidence: org trails, central log account, write-once storage, cross-account delivery, alert on logging failure.

23. What logs are needed for SaaS data exfiltration investigations?

  • App audit logs showing content access/export/sharing/permission change: file view/download/sync/share/external-invite/link-creation; source control clone/fetch/token-creation/Actions artifacts; CRM report export/bulk API.
  • Join to identity context (IdP sign-in, MFA, device posture, session ID, OAuth grants, role) - most SaaS breaches use valid creds; malice shows in unusual context (new device/OAuth client, impossible travel, excessive volume).
  • Network+endpoint fill gaps: SaaS says "downloaded", endpoint shows zip/USB/personal-cloud, proxy shows uploads, CASB shows external sharing.
  • Need object IDs + sensitivity labels + tenant/workspace ID + actor + delegated app - without them can't tell 500 marketing docs from regulated customer data.

24. What logs are needed for endpoint investigations?

  • Reconstruct process/file/network/identity/persistence: process name, command line, parent, GUID, user, hash, signer, file path, network connection, DNS, registry, service creation, scheduled task, driver load, script content.
  • Windows: EDR, Security events, Sysmon, PowerShell logs, Defender, service-control-manager, scheduled-task; Linux: auditd, syslog, auth logs, process accounting, FIM, systemd journal, cron, sudo, container runtime.
  • Support pivots: process tree from initial execution, network destinations, files written, credentials accessed, persistence created, lateral movement (Office->PowerShell->payload->scheduled task more meaningful than single event).
  • Need tamper detection - attackers clear logs/stop EDR/disable auditd. Alert on telemetry gaps, agent health, log clearing. Silent endpoint during investigation is itself a signal.

25. What logs are needed for DNS investigations?

  • Query logs + resolver logs + endpoint source attribution + flow logs + passive DNS/threat intel. Fields: query name/type, response code, response IPs, TTL, source host/user, resolver, allowed/blocked/sinkholed.
  • Exfil: query/label length, entropy, unique subdomain count, NXDOMAIN rate, TXT queries, periodicity. C2: DGA domains, newly registered, suspicious TLDs, low TTL, beacon intervals, poor reputation. Bypass: hosts using external resolvers directly.
  • Join to endpoint+network - resolver sees client host but NAT/forwarding/containers hide process; EDR/host DNS telemetry identifies process, flow logs show outbound to returned IPs.
  • Baseline per host role - long random subdomain from a recursive resolver is expected, same from a DB server is suspicious.

26. How would you correlate identity, endpoint, network, and cloud logs?

  • Build correlation around stable entities (user, device, session, token, SA, workload, IP, resource, process, data object) + timelines; each source sees only part of the incident.
  • Architecture: ingest to common schema -> enrich with context -> build entity timelines. Identity=who authenticated, endpoint=what executed, network=where traffic went, cloud=which APIs/resources, data=what accessed. Analyst pivots login -> endpoint -> processes -> role assumptions -> data reads.
  • Hard problems: identity resolution, time sync, duplicate events, missing fields (NAT shares IP, containers share nodes). Solve with device IDs, workload identities, session IDs, source-identity fields, mTLS identities, agent IDs, resource tags.
  • Risk-based alerting: weak signals accumulate into a case (new-IP login low; +suspicious process +secret access +export = high) without flooding analysts.

27. How would you use canary tokens?

  • Planted secrets/files/links/records/creds that should NEVER be used normally - any use = high-signal alert. Inverts detection: instead of guessing attacker behavior, create an object where any use is suspicious.
  • Examples: fake AWS keys (no privileges, monitored - first GetCallerIdentity triggers), URLs in documents, fake DB records, bogus SSH keys, honey files named like payroll exports, DNS canary domains.
  • Design: realistic enough to attract, safe enough to grant no real access; clear ownership, low FP, alert routing; placement on attacker paths (dev repos, vault decoys, file shares, data lakes, CI variables, admin workstations).
  • Playbook treats use as strong evidence but still scopes (account/device/process/source IP/adjacent files). High fidelity but root cause still needs investigation.

28. How would you use YARA or Sigma-style rules?

  • YARA = file/memory pattern detection (malware families via durable strings/bytes/PE structure/imports); Sigma = portable log-based behavioral detection (process creation, PowerShell flags, parent-child chains, cloud API sequences).
  • Good YARA combines conditions (file type+size+unique strings+hex+imports), not one hash/C2 domain (loader via VirtualAlloc+WriteProcessMemory+CreateRemoteThread+config marker). Use in triage, EDR scanning, memory forensics, email scanning, fleet hunts.
  • Good Sigma includes log source, fields, selection, FP notes, severity, ATT&CK mapping, test examples; portability = one idea -> Splunk/KQL/Elastic.
  • Both need lifecycle: owners, versioning, test cases, known FPs, deprecation; validate against benign corpora + malicious samples. Stale rule misses variants or creates noise.

29. How would you validate whether a detection works?

  • Three levels: logic correctness (fires for intended, not for benign), data availability (fields present/normalized/timely/retained), operational usefulness (context + clear playbook).
  • Methods: replay known incidents, adversary emulation, atomic tests, red/purple team, unit tests on sample logs, shadow-mode deployment (spray test = many users one IP + distributed ASN + normal office failures).
  • Test NEGATIVE cases too - catching the sim but firing on every deploy job isn't production-ready; FP testing includes admin tools, scanners, backups, CI/CD, vuln mgmt. Scoped exceptions not broad suppressions.
  • Validation continues post-deploy: volume, TP rate, FP reasons, MTTT, missed detections found by incidents. Detections decay - revalidate periodically.

30. What is the difference between detection coverage and detection quality?

  • Coverage = WHAT you can see (telemetry + logic for relevant tactics/techniques/assets/users/platforms/data flows). Quality = HOW WELL detections work (accurate, timely, actionable, understandable, routed right).
  • Broad coverage + poor quality = noisy rules analysts ignore; high quality in one area + blind elsewhere = gaps. Need both.
  • Coverage measured via ATT&CK mapping, data-source/asset inventory, threat-model alignment; quality via TP/FP rate, MTTT, enrichment completeness, response outcomes, analyst feedback.
  • Coverage maps aren't proof of security - "T1059 covered" is weak without telemetry, test cases, known limits, ownership, validation evidence. Maintain coverage map AND per-rule detection-health score.

Chunk 4 - Questions 31-40

31. How would you map detections to MITRE ATT&CK?

  • Connect each rule to the attacker behavior it observes (not after-the-fact tags). Start with threat modeling -> relevant tactics/techniques -> map telemetry + detections.
  • Document per detection: tactic, technique, data sources, logic, covered platforms, known bypasses, validation evidence (Office->PowerShell maps to command/scripting interpreter + phishing execution).
  • Mapping exposes gaps (many malware-execution rules but few for identity abuse/SaaS exfil/trust-policy changes/logging disablement). Don't make coverage a vanity metric - one noisy rule != technique well covered.
  • Use for planning/communication: prioritize high-risk techniques, align with red team, report to leadership, track over time. Link ATT&CK tags to test cases, telemetry health, rule quality, incident outcomes.

32. What is the difference between containment and eradication?

  • Containment stops it getting worse (isolate host); eradication removes footholds + root cause (remove malware/persistence/stolen creds, rebuild from clean baseline, patch vuln).
  • Containment is time-sensitive, may use incomplete info (revoke tokens, isolate, block C2, freeze IAM); risk = tipping off attacker before all footholds known. Mandatory immediate for active exfil/ransomware.
  • Eradication is deeper: reimage, remove persistence, patch, delete unauthorized accounts/keys, rotate creds, fix IAM weakness, validate no backdoor - requires root cause (rebuild server but leave exposed CI secret = attacker returns).
  • Recovery after eradication: restore service, validate normal behavior, elevated monitoring. Common mistake: calling containment "fixed" because the alert stopped.

33. What is the first action you would take in a suspected account compromise?

  • Preserve enough evidence to decide correctly while reducing immediate risk if confidence high. Verify alert, check if privileged/high-impact, look for active sessions; strong evidence -> revoke sessions+tokens immediately while preserving logs.
  • DON'T just change password in isolation - doesn't revoke refresh tokens/OAuth grants/app passwords/API keys/active sessions, and tips off attacker. Need session revocation + token invalidation + MFA review + OAuth/API containment.
  • Triage checklist: recent logins, MFA events, new device/location, password reset, recovery-factor changes, mailbox rules, OAuth consent, privileged actions, data access, endpoint alerts. Contact user via trusted channel UNLESS insider/sensitive investigation.
  • Then scope blast radius (what resources, persistence, secrets/code/PII/IAM/email, used to compromise other accounts?). First action opens a disciplined investigation, not closes it.

34. How would you preserve forensic evidence?

  • Maintain integrity, context, chain of custody. Collect VOLATILE first (processes, connections, memory, active sessions, temp files, instance metadata, live activity) then durable (logs, disk images, snapshots, audit trails, config).
  • Endpoint: EDR triage package, memory capture, disk/targeted image, process list, connections, persistence, registry hives, event logs, shell history, cron, scheduled tasks, browser artifacts, malware samples. Cloud: snapshot disks, preserve metadata, capture IAM state, copy logs to protected storage, tag to prevent deletion.
  • Integrity: record who/what/when/where/which-tool/storage-location, hash files+images, restrict storage separate from operational systems, avoid modifying originals.
  • Key tradeoff = evidence vs containment: active exfil -> contain first; attacker idle + strong visibility -> short collection window. Incident commander decides explicitly + documents.

35. How would you decide incident severity?

  • Based on impact, confidence, scope, sensitivity, attacker activity, business risk - NOT just alert type (suspicious login to test account low; same for prod admin with customer data = critical).
  • Factors: confirmed vs suspected, active attacker presence, data sensitivity, affected user/system volume, privilege level, regulatory exposure, availability impact, containment speed, public visibility, safety/financial systems. Consider blast radius (one endpoint -> org-wide control plane).
  • Practical model: P0 = active compromise + ongoing exfil/destruction/broad impact; P1 = confirmed high-value identity/prod; P2 = suspected, urgent investigation; P3 = policy violation/low-impact. Labels matter less than consistent escalation.
  • Revisit as facts change - start higher to mobilize then downgrade, or low-sev becomes P1 on finding persistence/data access/lateral movement. Commander owns + records rationale.

36. How would you communicate during a security incident?

  • Structured, timely, fact-based: primary incident channel, incident doc, decision log, update cadence by severity. Commander owns cadence; scribe records timeline/decisions/evidence/action items/owners/open questions.
  • Tailor audiences: responders need indicators/systems/owners/containment/hypotheses; execs need impact/risk/customer-exposure/next-update; Legal+Privacy need data categories/affected-users/jurisdictions/deadlines; Comms need approved external language.
  • Separate FACTS from HYPOTHESES - good update: what we know, what we don't, what changed, actions underway, decisions needed, next update time. Avoid speculation/blame/unsupported attribution.
  • External comms through Legal/Privacy/Comms - premature statements create legal risk, mislead customers, tip off attacker. Sensitive details (indicators, unpatched systems, employee investigations) stay need-to-know.

37. How would you run an incident bridge?

  • Create coordination not chaos. Name roles: incident commander, technical lead, scribe, comms lead, ops lead, forensics lead, service owners. Commander drives agenda/decisions; tech lead drives investigation (NOT same person in serious incidents).
  • Rhythm: start with severity/known facts/active risks/goals -> review workstreams (scoping, containment, forensics, customer impact, legal/privacy, comms, recovery). Every action item needs owner+deadline; scribe records decisions+timestamps; breakouts for side investigations.
  • Communication hygiene: state evidence source not just conclusions; avoid long 2-person debugging on the bridge; explicit decisions (contain now or observe, rotate now or after scoping, offline or isolate subset).
  • Handoff = written summary: status, severity, timeline, open risks, decisions, active workstreams, pending actions, next update. Good handoffs prevent incidents resetting every shift.

38. How would you handle legal and privacy considerations during investigation?

  • Involve Legal+Privacy early when personal/regulated data, employee monitoring, customer notification, law enforcement, contracts, or cross-border transfer involved. Security establishes facts; Legal/Privacy determine obligations + handling boundaries.
  • Privacy: data minimization, need-to-know, jurisdictional limits, employee-monitoring rules, retention, content-inspection necessity (metadata often enough; content review needs policy + legal approval).
  • Legal: privilege, regulatory timelines (e.g. GDPR 72h regulator notification for certain breaches - trigger is a legal determination, security supplies data categories/population/evidence/containment status), preservation, law enforcement, comms risk.
  • Documentation factual/timestamped/no-blame/no-speculation - use "breach"/"exfiltration" only when supported. Good notes help legal review + learning; sloppy notes create risk.

39. How would you contain a compromised service account?

  • Balance security vs availability. Identify what uses it (workloads, CI/CD, scheduled tasks, integrations, secrets, roles, downstream deps); determine active abuse + permissions.
  • Immediate options: disable exposed keys, revoke tokens, block source IPs, remove high-risk permissions, detach policies, disable account, isolate minting workload. Production -> coordinate with owners to rotate without outage; highly-privileged/active-abuse -> security may override availability.
  • Scope ALL actions during compromise window: data reads, secret access, KMS decrypts, role assumptions, IAM changes, deployments, storage/logging changes; check for new creds or modified trust policies for re-entry.
  • Long-term: remove static creds, workload identity/short-lived tokens, least privilege, restrict source networks, permission boundaries, anomaly monitoring, separate by workload+environment. One shared high-privilege SA = containment nightmare.

40. How would you contain a compromised endpoint?

  • First decision: isolate immediately vs brief evidence collection. Active exfil/malware-spread/ransomware/attacking-others -> isolate now (EDR containment, NAC, firewall, physical). Idle + high-value -> short collection window with commander approval only.
  • Containment: network isolation, kill malicious processes, disable compromised local accounts, revoke sessions, block C2, prevent lateral movement. Preserve EDR telemetry, memory, logs, process trees, persistence, attacker files. Don't just delete malware + return to service.
  • Scope: how compromised, what credentials exposed, what systems connected, lateral movement, sensitive data accessed. Endpoint compromise often becomes identity compromise (browsers, tokens, SSH keys, VPN sessions, password managers present).
  • Recovery: prefer REIMAGE over clean (especially confirmed malware/privileged), rotate exposed creds, patch root cause, restore from clean backups, monitor recurrence. Not contained if stolen tokens still valid elsewhere.

Chunk 5 - Questions 41-50

41. How would you contain a cloud workload compromise?

  • Identify workload type + control plane deps (VM/container/pod/serverless/managed service/CI runner/pipeline); determine if limited to runtime or includes cloud creds/IAM roles/secrets/images/pipelines/persistent infra.
  • Immediate: isolate network, remove from LBs, quarantine security group, block egress, revoke instance-profile/SA permissions, disable keys, pause pipeline, snapshot disk/container. K8s -> preserve pod specs, SA tokens, RBAC bindings, image digest, node info, API audit logs.
  • Scope metadata-service access, secret retrieval, KMS decrypts, storage access, role assumptions, lateral movement, outbound C2, IaC changes. Attackers use workload to reach metadata -> read SA tokens -> list resources -> pivot to data stores.
  • Recovery: rebuild from known-good image/artifact, patch entry point, rotate exposed secrets, reduce IAM, enforce egress, add detections. Immutable rebuild safer than trusting a modified runtime - don't just restart.

42. How would you recover from exposed secrets?

  • Assume the secret was COPIED - deleting from visible location isn't recovery. Identify type/scope/permissions/exposure-duration/storage-locations/derived-tokens/dependents; revoke or rotate via system of record (not manual config edits).
  • Sequence matters: prod DB password -> rotating without coordinating clients causes outage; highly-privileged key in public GitHub -> immediate disable despite service risk. Playbook supports emergency disable, planned rotation, dual-secret rollout, consumer validation.
  • Inspect use during exposure window (cloud API/DB connection/KMS/SaaS/CI logs for unfamiliar infra); if it granted access to other secrets/signing -> rotate downstream + invalidate derived artifacts/tokens.
  • Long-term: secret scanning, push protection, short-lived creds, workload identity, secret manager, CI redaction, least privilege, automated rotation. Best design improvement = make disclosure low-impact by reducing static broad secrets.

43. How would you verify that an attacker no longer has access?

  • Eviction is a HYPOTHESIS to test - not "the alerts stopped" (attackers go quiet). Verify all access paths closed, persistence removed, creds rotated, vulns patched, elevated monitoring shows no recurrence.
  • List every foothold: accounts, sessions, refresh tokens, OAuth grants, API/SSH keys, malware, scheduled tasks, services, startup items, cloud roles, trust policies, backdoor users, CI secrets, images, integrations - each needs explicit removal evidence.
  • Run targeted hunts for TTPs + indicators (IPs, domains, user agents, hashes, command lines, API sequences, resource names, data patterns); monitor failed attempts with revoked creds (confirms attacker trying to return).
  • Independent control checks: confirm old keys fail, EDR health post-reimage, diff IAM vs known-good, scan for vulnerable versions. Heightened monitoring for defined period (often 30 days for serious incidents).

44. How would you write a post-incident review?

  • Factual, blameless, action-oriented. Structure: executive summary, impact, timeline, root cause, contributing factors, detection/response analysis, what went well/poorly, action items with owners + due dates.
  • Timeline: first malicious activity, first detection, triage start, severity declaration, containment, eradication, recovery, customer/legal milestones, closure. Distinguish initial technical cause from systemic contributors (Log4j RCE vs missing SBOM + weak egress + over-privileged SA + insufficient logging).
  • Examine detection gaps + response friction: why did alert fire/fail? logs available? runbook help? roles clear? knew how to rotate creds? approvals slow? containment cause outage?
  • Action items specific + trackable: "improve logging" weak; "enable CloudTrail data events for prod buckets by July 15, owned by CloudSec" actionable. Only useful if leadership follows through.

45. What makes a postmortem useful?

  • It CHANGES future behavior, not just records the incident. Identifies real failure modes, explains why controls didn't prevent/detect sooner, produces prioritized improvements reducing recurrence/impact.
  • Honesty + specificity: avoid "engineer error" - explain why the system ALLOWED one action to expose prod data (missing guardrails, unclear ownership, no automated tests, dangerous defaults, insufficient review). Blameless != consequence-free; focuses on improving systems.
  • Include data (time to detect/contain, affected systems/records, alert sources, decision points, control failures) + tradeoffs (delayed containment to find all footholds can be reasonable if documented + bounded).
  • Final test = action-item quality: owner, deadline, priority, verification method. Ten vague recommendations + no completed fixes = theater; three high-impact completed changes = useful.

46. How would you convert incident lessons into preventive controls?

  • Map each root cause + contributing factor to a guardrail that blocks the same failure. Leaked static keys -> workload identity + secret scanning + push protection + short-lived tokens + remove long-lived keys. Excessive IAM -> permission boundaries + JIT + policy-as-code.
  • Control as close to source as possible: push protection + secretless deploy > training devs; org-level logging admins can't disable > asking admins not to disable; inventory+SBOM+auto-updates+patch-orchestration > asking teams to patch faster.
  • Prioritize by risk reduction + feasibility: some immediate (disable risky OAuth app), some need platform investment (egress filtering, central secret mgmt). Separate quick containment hardening from long-term architectural fixes.
  • Verify through tests: dummy secret confirms push protection blocks; policy simulation/red-team confirms escalation path closed. Controls aren't real until enforced + monitored.

47. How would you convert incident lessons into new detections?

  • Start from timeline, identify EARLIEST observable attacker actions; ask which signals were available, missing, or would give high-fidelity alert. Best detections come from first abnormal behavior, not final obvious impact.
  • Per detection define: threat behavior, data source, logic, severity, enrichment, response playbook, test cases, known FPs, owner (SA from new country + lists secrets = unusual-source-for-SA + sensitive-API-access; logging disabled -> alert on disablement + delivery failure all accounts/regions).
  • Create retrospective hunts: run new rule over historical data to find prior occurrences + estimate FPs; deploy shadow mode, tune, promote. Indicators become short-term hunts, TTP becomes durable detection.
  • Map to ATT&CK + coverage but quality > labels. Test with adversary emulation/synthetic event so team knows it would have caught the behavior.

48. What metrics would you track for incident response?

  • Speed: MTT detect/acknowledge/triage/contain/eradicate/recover; for serious incidents track first-malicious-activity to first-detection (DWELL TIME is the real failure).
  • Quality: TP rate, severity accuracy, escalation accuracy, reopened incidents, evidence completeness, runbook use, customer impact, regulatory-notification timeliness, % incidents with completed postmortem actions. Connect detection metrics to response metrics (high alert count != success).
  • Operational: on-call load, after-hours pages, role staffing, bridge duration, action-item aging, repeat incidents by root cause (same root cause repeating = not learning; postmortem actions open for months = documents not risk reduction).
  • Don't use metrics to PUNISH responders - fear -> hidden incidents + downgraded severity. Metrics should improve systems (telemetry, playbooks, escalation, preventive controls).

49. What are common incident-response mistakes?

  • Technical: acting before preserving evidence, waiting too long to contain active harm, confusing containment with eradication, failing to revoke tokens, relying only on password resets, cleaning malware instead of rebuilding, declaring closure because alerts stopped, focusing on first system while missing identity/cloud/SaaS persistence.
  • Communication: no incident commander, no scribe, unclear severity, conflicting channels, speculative executive updates, late Legal/Privacy involvement (poor coordination can create more risk than technical uncertainty).
  • Scoping: checking only one log source, ignoring denied API calls, forgetting data-plane logs, overlooking third-party integrations, missing regions/accounts/tenants/endpoints (attackers exploit org boundaries).
  • Post-incident: vague action items, blame-focused postmortems, no remediation owner, no validation fixes work. Most expensive incident = the one that repeats because it was documented but nothing changed.

50. What questions would you ask before declaring an incident closed?

  • Scope fully understood? Which users/systems/workloads/data-stores/credentials/third-parties affected? Earliest known attacker activity? Evidence for/against data access, exfiltration, persistence, lateral movement?
  • Containment + eradication complete? All compromised credentials/tokens/sessions/OAuth-grants/SSH-keys/API-keys/secrets revoked or rotated? Hosts rebuilt? Malicious files/tasks/services/cloud-roles/trust-policy-changes/backdoor-accounts removed? Root cause patched?
  • Recovery safe? Services from known-good baselines? Monitoring elevated? Old credentials fail? Recurrence detections in place? Legal/privacy/customer/exec obligations addressed? Evidence + timelines preserved?
  • Learning captured? Postmortem scheduled/completed? Action items specific/owned/tracked? Detection gap addressed? Closure = immediate incident controlled + path to reduce recurrence, NOT everyone's tired and wants the bridge to end.
