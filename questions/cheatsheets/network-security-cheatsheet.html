enterprise data protection Network Security - Cheatsheet

Source: questions/answers/network-security-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Trace a browser request to a service and identify security checks at every layer.

  • Path: Browser -> DNS -> TCP/QUIC -> TLS -> CDN/LB -> API gateway -> service -> datastore. Identify trust boundaries + where decisions are ENFORCED.
  • By layer: Browser (HSTS, cookie security, mixed content), DNS (resolver trust, DNSSEC), Transport (DDoS/rate), TLS (cert validation, version, cipher, SNI/ALPN), Edge (WAF, bot, geo), Gateway (authN, coarse authz, rate limit), Service (object authz, tenant checks), Data (row/field policy, encryption, audit).
  • Phrase: don't stop at "TLS is green" - the request crosses many control points, failure at any one becomes the real attack path.

2. Explain the TLS 1.3 handshake at a high level.

  • Establishes encrypted+authenticated channel with fewer round trips + fewer legacy options than 1.2.
  • Flow: ClientHello + key share -> ServerHello + key share -> server Certificate + Finished -> client Finished -> encrypted app data. Both derive shared secrets via ephemeral DH.
  • Finished messages prove both sides derived same keys + saw same handshake (transcript integrity).
  • Phrase: TLS 1.3 = faster setup + safer defaults, removes legacy negotiation paths that made 1.2 hard to secure.

3. How does TLS 1.3 differ from TLS 1.2?

  • 1-RTT new connections (vs more round trips), ephemeral key exchange ONLY (removed static RSA), forward secrecy baseline (vs optional), AEAD-only suites, more handshake encrypted, 0-RTT for resumption (replay risk).
  • Removed static RSA key exchange + weak ciphers -> reduces downgrade + misconfiguration risk.
  • Phrase: not just "1.2 with better ciphers" - it changes the handshake and default security posture.

4. What is forward secrecy in TLS?

  • Recorded past sessions stay protected even if server's long-term private key is later compromised; in 1.3 from ephemeral key exchange (cert key authenticates, per-session traffic keys from fresh ephemeral agreement).
  • Matters because attackers record traffic before compromising keys; keys leak via bugs/backups/logs/operator mistakes.
  • Failure modes: logging TLS session keys + unprotected logs, legacy static RSA, weak randomness, old protocol on termination devices.
  • Phrase: limits RETROSPECTIVE damage - doesn't protect traffic if the endpoint is compromised during the session.

5. How does certificate validation work in a browser?

  • Decides if cert is trusted for the requested domain: chain to trusted root, valid signatures, validity period, hostname matches SAN, key usage/EKU allow server auth, algorithm/size meet policy, CT for public certs, revocation, HSTS may force hard failure.
  • Phrase: browser isn't just checking a cert exists - it verifies identity, trust chain, time, purpose, and ecosystem policy.

6. What can go wrong with TLS termination at load balancers?

  • Termination changes trust boundaries - protected browser->LB, but post-termination path must be designed deliberately.
  • Risks: plaintext backend on untrusted internal network, LB becomes high-value key+plaintext point, spoofable X-Forwarded-For trusted, inconsistent security headers, backend logs expose sensitive data, cert rotation/SNI errors cause outages.
  • Controls: re-encrypt LB->backend, mTLS/mesh identity for backend hops, strip+set trusted forwarding headers at edge, HSM-backed keys, monitor expiry/handshake errors, document LB as trust boundary.
  • Phrase: termination isn't bad - the mistake is pretending the terminated internal path still has end-to-end TLS properties.

7. How would you secure service-to-service traffic inside a data center?

  • Workload identity + encryption in transit + least-privilege authz + segmentation + observability.
  • Controls: mTLS via private PKI/mesh, short-lived workload certs, identity tied to deployment (not IP), authz by service/method/tenant/env, segmentation, egress policies, telemetry for denied connections/unusual pairs, automated rotation.
  • AWS: private subnets, security groups referencing service groups, PrivateLink/VPC endpoints, mesh mTLS, VPC Flow Logs.
  • Phrase: internal traffic shouldn't be trusted just because it's internal - authenticate workloads + authorize actions.

8. Compare TLS, mTLS, IPsec, and MACsec.

  • TLS = transport/app, client-server channel (usually server-auth), web/APIs/DBs. mTLS = both sides auth with certs, service-to-service + partner APIs.
  • IPsec = network layer, IP packets host/gateway, site-to-site VPN/overlays (can hide traffic from inspection, selector complexity). MACsec = data link, Ethernet frames on local link, data center/campus.
  • Tradeoffs: TLS app-aware + widely supported; mTLS strong workload identity but cert lifecycle burden; IPsec protocol-agnostic but opaque; MACsec link-level but no app identity.
  • Phrase: choose layer by threat model - encrypting a link doesn't replace service authentication, mTLS doesn't replace segmentation.

9. When would you use mTLS instead of network allowlists?

  • mTLS when identity needs to be stronger + more portable than IP; allowlists brittle in dynamic cloud/container envs.
  • mTLS better when: workloads move across hosts/pods/VPCs/regions, IPs shared/ephemeral/NATed, partner needs crypto client identity, per-service identity + audit, zero-trust independent of network location.
  • Allowlists still useful for coarse exposure reduction but weak as only identity control.
  • Phrase: allowlists answer "where did this packet come from?", mTLS answers "which workload is this?" - the second matters more for sensitive traffic.

10. What are the operational challenges of mTLS?

  • Issuing certs to every workload, rotating before expiry without outages, revoking compromised identity, mapping cert->authz, debugging handshake failures, clock skew, trust roots across regions, legacy clients, cert sprawl, observability when encrypted.
  • Controls: automate issuance/renewal, short-lived certs, standardize naming, monitor cert age/handshake errors/trust-bundle drift, dev tooling, gradual rollout (permissive then enforcing).
  • Phrase: mTLS shifts work from firewall rules to identity + cert operations - good trade only if the platform owns the lifecycle.

Chunk 2 - Questions 11-20

11. How does QUIC reduce latency compared with TCP plus TLS?

  • Combines transport setup + TLS 1.3 crypto setup into ONE handshake over UDP (vs TCP handshake then TLS handshake).
  • Resumed connections can use 0-RTT for eligible early data; user-space transport evolves faster than kernel TCP; connection migration survives network changes; stream multiplexing avoids transport-level stalls.
  • Phrase: not faster because "UDP is magically faster" - faster because it redesigns connection setup, encryption, streams, and migration around modern web traffic.

12. Why does QUIC run over UDP?

  • New transport directly on IP would need OS/NAT/firewall/middlebox changes; UDP already traverses the internet so QUIC deploys in user space.
  • QUIC does NOT rely on UDP for reliability - provides own connection management, acks, retransmission, congestion control, stream multiplexing, TLS 1.3 integration.
  • Tradeoffs: some enterprises block/rate-limit UDP, TCP tools see less detail, LBs/firewalls need QUIC-aware support, amplification protection required.
  • Phrase: UDP is the deployable substrate; QUIC is the real transport.

13. What security tradeoffs come with QUIC 0-RTT?

  • Client sends early app data on resumed connection before handshake completes -> lower latency but REPLAY risk (attacker replays, server processes request more than once).
  • Unsafe: money transfer, password change, purchase, state-changing admin, any create/delete/modify. Safe: idempotent GETs, cacheable reads.
  • Controls: disable 0-RTT for state-changing endpoints, anti-replay windows + single-use tokens, idempotent handlers, mark early data before authz logic runs.
  • Phrase: 0-RTT is a performance feature with app security consequences - the transport can't know whether your business operation is safe to replay.

14. Why does QUIC avoid head-of-line blocking across streams?

  • TCP = one ordered byte stream, so one lost segment blocks ALL later bytes even for unrelated requests (transport-level HoL blocking).
  • QUIC multiplexes independent streams above UDP - loss on one stream delays only that stream, others continue.
  • Nuance: avoids HoL ACROSS streams, but within a single stream ordered delivery still matters; loss still affects congestion control.
  • Phrase: QUIC doesn't make packet loss disappear - it prevents one stream's loss from blocking unrelated streams.

15. Why is QUIC harder to inspect with traditional tools?

  • Encrypts most transport + app metadata, runs over UDP - breaks TCP-centric tools: no plaintext HTTP headers, encrypted transport params/stream details, connection IDs decouple identity from IP/port, TLS integrated not layered.
  • Tradeoff: better privacy + middlebox-tamper resistance vs less passive enterprise visibility.
  • Controls: edge logs, endpoint telemetry, app logs, QUIC-aware LBs, monitor DNS/SNI/destination IP/flow metadata/server request logs.
  • Phrase: QUIC shifts visibility from passive network inspection to endpoints + edges that terminate the connection.

16. How would you defend a QUIC service against amplification attacks?

  • UDP reflection risk if server sends more to spoofed victim than it received - defense = address validation + amplification limits.
  • Controls: limit bytes sent before validating client address, retry tokens for suspicious initials, validate source before large responses, rate limit by source/service/token, drop malformed/unsupported versions early, edge DDoS protection, monitor initial/retry rates + amplification ratio.
  • Phrase: server must prove the client can RECEIVE at the claimed address before sending much more data than it received.

17. What middleboxes can break QUIC?

  • Firewalls blocking UDP 443, NAT with short UDP idle timeouts, HTTP-over-TCP-only proxies, TLS inspection boxes, LBs without QUIC support, QoS throttling unknown UDP, gateways forcing TCP proxies, path MTU black holes.
  • Symptoms: intermittent failures, works on mobile not corporate network, frequent TCP fallback, poor perf from UDP throttling.
  • Phrase: QUIC improves modern traffic but the internet still has devices built on older assumptions - fallback + measurement are part of the design.

18. How should systems respond when QUIC is blocked?

  • Fall back gracefully to HTTPS over TCP+TLS WITHOUT weakening security: detect failure quickly, fall back to HTTP/2 or HTTP/1.1 over TLS, preserve TLS validation + app controls, avoid infinite retry loops, telemetry on where blocked, NEVER disable cert checks or plaintext fallback, careful alt-svc, test fallback regularly.
  • Metrics: QUIC success rate by network/region/ASN/client, fallback rate, latency difference, error/timeout patterns, UDP reachability trends.
  • Phrase: fallback should preserve security while accepting that UDP reachability isn't universal.

19. Explain DNS poisoning.

  • Resolver/client tricked into accepting forged DNS answer -> users connect to wrong IP. Attacker races/injects fake response -> resolver caches it -> future clients misdirected until TTL expires.
  • Risks: phishing with real-looking domains, interception, malware delivery, email misrouting via poisoned MX, outage via blackholed records.
  • Defenses: DNSSEC validation, source port + query ID entropy, resolver hardening/patching, restrict recursive access, monitor unexpected answers/TTL changes, TLS cert validation (wrong IP alone insufficient for HTTPS impersonation).
  • Phrase: DNS poisoning attacks name-to-address trust - TLS limits HTTPS damage but DNS remains critical for routing, email, service discovery.

20. How does DNSSEC help and what does it not solve?

  • Adds origin authentication + integrity - validating resolver verifies response signed by legitimate zone + unmodified; chain of trust root->zone; authenticated denial of existence.
  • Does NOT solve: query privacy (no encryption), registrar account compromise, compromised authoritative provider, malicious domain owners, BGP/routing attacks, TLS cert validation, availability/DDoS.
  • Phrase: DNSSEC gives INTEGRITY not confidentiality - pair with DoT/DoH for privacy and TLS for endpoint authentication.

Chunk 3 - Questions 21-30

21. How would you detect DNS tunneling?

  • Encodes data/command traffic inside queries+responses. Signals: very long subdomains, high-entropy labels, many unique subdomains under one domain, heavy TXT use, high query volume from one host, newly-registered/low-reputation domains, large/unusual response sizes, repeated NXDOMAIN, direct DNS to external resolvers.
  • Controls: force approved recursive resolvers, block direct outbound DNS except resolvers, central DNS log analysis, allowlist known high-volume internal domains, correlate with endpoint+proxy alerts.
  • Phrase: tunneling is usually visible in METADATA even when payload meaning isn't - look for shape and behavior, not just known-bad domains.

22. How would you investigate DNS exfiltration?

  • Steps: triage (domain/host/type/volume/time) -> pull resolver logs + raw samples -> check resolver bypass -> decode labels (Base32/Base64/hex) -> identify endpoint process -> correlate file access/archive/malware -> check domain registration/NS -> block/sinkhole -> preserve + contain.
  • Evidence: query names+timestamps, source device/user/process, resolver+firewall logs, domain registration + passive DNS, endpoint process tree + file activity.
  • Phrase: prove whether data ACTUALLY left, not only that DNS looked strange - endpoint correlation is usually decisive.

23. What is BGP hijacking?

  • An AS advertises IP prefixes it doesn't legitimately control; other networks accept the route + send traffic to the wrong AS.
  • Forms: origin hijack (originate someone's prefix), more-specific hijack (narrower prefix wins route selection), route leak (accidental re-advertisement between peers).
  • Impact: blackholing, interception, outage, credential/session exposure if higher layers fail, route instability.
  • Phrase: exploits the internet routing trust model - the route may be accepted even when the advertiser isn't the real owner.

24. How does RPKI reduce BGP hijacking risk?

  • Prefix owners publish signed ROAs stating which AS may originate a prefix; routers doing Route Origin Validation reject/de-prioritize invalid announcements.
  • Helps: detect invalid origin AS, reduce simple hijacks, crypto evidence for routing policy, hygiene across peers.
  • Limits: validates ORIGIN not full AS PATH, doesn't stop all route leaks, incorrect ROAs break your own reachability, adoption not universal.
  • Phrase: strong for origin validation but NOT complete path security - pair with prefix filters, peer policy, monitoring, IR.

25. What would you do if your company's prefix was hijacked?

  • Internet-edge incident: confirm via route collectors/looking glasses, identify hijacked prefix/origin AS/regions/impact, check own announcements + recent changes, verify RPKI ROA state, contact upstreams for filtering/correction, announce more-specifics if safe, coordinate with offending network, monitor for interception/cert/login anomalies, communicate impact, improve ROAs/filters/monitoring after.
  • Evidence: first/last seen times, invalid route paths, affected prefixes, upstream responses, service impact metrics.
  • Phrase: speed matters - response depends on upstream relationships + having route monitoring BEFORE the incident.

26. What is ARP spoofing?

  • Layer 2 attack: forged ARP messages associate attacker MAC with another host's IP (e.g. gateway) -> MitM interception, blackholing, session hijacking on unencrypted protocols, credential theft.
  • Defenses: Dynamic ARP Inspection on switches, DHCP snooping for trusted IP/MAC bindings, static ARP for critical systems, segmentation, endpoint ARP-table-change detection, TLS/SSH so L2 interception is less useful.
  • Phrase: local-network trust attack - strong network hygiene AND encrypted higher-layer protocols both matter.

27. How would you detect ARP spoofing in an enterprise network?

  • Signals: multiple MACs claiming same IP, unexpected gateway IP mapping changes, ARP reply storms, replies without requests, DAI violations, duplicate IP alerts, endpoint gateway ARP changes, sudden path changes/cert warnings.
  • Controls/telemetry: DHCP snooping + DAI, central switch logs, monitor endpoint network tables on managed devices, NAC for unknown devices, segment guest/corporate/server/management.
  • Phrase: detection is strongest at the SWITCH - it sees L2 behavior before endpoints only see symptoms.

28. What is VLAN hopping?

  • Device sends traffic into a VLAN it shouldn't access by abusing trunking/tagging/misconfig.
  • Forms: switch spoofing (negotiate trunk mode), double tagging (nested VLAN tags cross via native VLAN config), misconfigured access ports allowing unexpected tags.
  • Impact: bypass L2 segmentation, reach management/server VLANs from user network, lateral movement.
  • Phrase: usually a CONFIGURATION failure not magic - ports should be explicitly configured for their role, native VLANs shouldn't carry sensitive traffic.

29. How would you prevent VLAN hopping?

  • Switch-hardening + segmentation discipline: disable dynamic trunk negotiation on access ports, explicitly configure user ports as access, restrict allowed VLANs on trunks, use an unused native VLAN, keep users/sensitive systems off native VLAN, drop unexpected tagged frames, port security + NAC, separate management from user VLANs, monitor trunk config drift.
  • Phrase: prevention is mostly removing AUTOMATIC behavior and making trunk boundaries explicit.

30. What is network segmentation and why does it fail in practice?

  • Divides networks into zones to limit exposure + lateral movement. Fails when too coarse, poorly enforced, or misaligned to application reality.
  • Why it fails: flat networks + broad allows, any-to-any troubleshooting exceptions, shared admin networks reaching everything, legacy apps needing wide connectivity, rules not tied to ownership, security-group sprawl, no traffic observation before policy, ignoring identity/app-layer authz, temporary exceptions become permanent, no isolation testing.
  • Phrase: fails when diagrams LOOK segmented but effective reachability is broad - test actual paths, not intent.

Chunk 4 - Questions 31-40

31. How would you design segmentation using zero trust principles?

  • Network location alone isn't enough - each connection authenticated, authorized, limited, observable.
  • Principles: segment by sensitivity/env/tenant/function, identity-aware access for users+workloads, mTLS for service-to-service, least-privilege paths (not subnet trust), policy as code for firewall/SG/mesh rules, continuous posture verification, monitor denied + unusual allowed flows, test reachability regularly.
  • Phrase: zero trust segmentation isn't just smaller VLANs - it's identity + context + least privilege + continuous verification applied to network paths.

32. How would you segment production, development, and corporate networks?

  • Separate trust zones with tightly controlled access paths: separate accounts/VPCs/projects, production has NO direct inbound from corporate laptops, access prod via bastion/PAM/identity-aware proxy, dev can't reach prod DBs directly, shared services (identity/logging/CI) have explicit restricted paths, admin needs MFA+approval+logging, prod->dev data needs masking/synthetic.
  • Zones: Corporate (SaaS/admin via IAP), Development (dev services/CI/limited test data), Production (service graph only, restricted egress), Security tooling (read-only telemetry, controlled response).
  • Phrase: dangerous pattern = corporate laptop to production database - humans access prod through controlled workflows, not broad reachability.

33. How would you isolate high-value data stores?

  • Network isolation + strong identity + data-layer authz + encryption + close monitoring: private subnets/dedicated zones, inbound only from approved services/pipelines, SGs by service identity, TLS/mTLS, DB authN + least-privilege roles, row/column/tenant controls, KMS encryption + monitor decrypt, admin via PAM, detailed query/export/connection logging, block direct internet egress.
  • Phrase: network isolation is necessary but INSUFFICIENT - the store must also enforce identity, authorization, encryption, and audit.

34. How would you secure east-west traffic?

  • Internal service-to-service traffic = lateral movement path, needs authN + policy: mTLS, workload identity + authz, microsegmentation by tier, deny-by-default connectivity, egress restrictions per workload, detection for unusual service pairs, DNS/service-discovery controls, patch internal services, log internal API calls + flows.
  • Detection: web tier -> payroll DB unexpectedly, workstation scanning server subnets, build server -> prod secrets outside deployment window, sudden SMB/RDP fan-out.
  • Phrase: east-west security is about shrinking BLAST RADIUS after initial compromise - assume one workload is compromised, design so it can't roam.

35. What is lateral movement and how do network controls limit it?

  • Attacker moves from initial compromise to other internal systems. Paths: RDP/SSH/SMB/WinRM/DB ports/K8s API/metadata endpoints, stolen credential reuse, admin shares + remote exec, broad service accounts, flat user-to-server access.
  • Controls: segment user/server/prod/management/data zones, deny inbound management protocols by default, JIT admin, restrict service accounts to required hosts/ports, monitor scanning/auth-failures/new-pairs, EDR + identity detection alongside network.
  • Phrase: network controls don't eliminate credential theft but make stolen credentials much less useful by reducing reachable targets.

36. How would you detect command-and-control traffic?

  • Signals: known-malicious destinations, beaconing at regular intervals, low-volume long-lived connections, unusual user agents/TLS fingerprints, DGA domains, newly registered domains, servers bypassing proxies, unusual ports/protocols/countries, failed connections to many domains, DoH to unapproved resolvers.
  • Response: correlate with endpoint process+user, contain host, block destination + related infra, hunt pattern across environment.
  • Phrase: C2 detection is strongest when network signals are CORRELATED with endpoint process telemetry + DNS history.

37. What network logs are most valuable for security detection?

  • DNS resolver, proxy/SWG, firewall allow+deny, VPC Flow Logs/NetFlow, LB access, TLS termination, VPN/remote access, NAC, DHCP (IP->device), SG/firewall change logs, IDS/NDR alerts.
  • Value: DNS (tunneling/C2/malware domains/intent), flow logs (lateral movement/scanning/unusual pairs), proxy (web exfil/unsanctioned SaaS/user attribution), LB (external abuse), firewall denies (recon/drift).
  • Phrase: DNS + proxy + identity + flow logs TOGETHER are far more useful than any one log alone.

38. What are the limitations of NetFlow?

  • No payload content, limited app-layer context, NAT obscures original host, sampling misses low-volume, short flows aggregated/lost, encryption hides URLs/headers/commands, confusing directionality, often no process/user attribution, high cardinality = expensive storage.
  • Still useful for: beaconing, port scanning, unusual service pairs, data volume anomalies, rare-destination egress, baseline changes.
  • Phrase: NetFlow tells you communication HAPPENED not what was said - it's metadata, needs DNS+proxy+endpoint+identity context.

39. What are the limitations of packet capture in encrypted environments?

  • TLS/QUIC/SSH/IPsec hide payloads; QUIC also encrypts transport metadata; decryption keys usually unavailable + should be protected; captures collect sensitive data (privacy risk); high-volume expensive; point-in-time misses distributed activity; often lacks user/process context.
  • Complements: endpoint telemetry, app logs, TLS termination logs at controlled edges, DNS/flow metadata, structured audit, targeted capture with legal/privacy approval.
  • Phrase: in encrypted environments, packet capture shifts from content inspection to timing/handshake/metadata/troubleshooting unless captured at a trusted termination point.

40. How would you balance privacy with network monitoring?

  • Minimization + transparency + access controls + purpose limitation: metadata before content, defined legitimate purposes, avoid personal categories without legal basis, aggregation for broad analytics, restrict raw log access, redact/hash sensitive fields, retention by need + law, regional policies, audit analyst access, clear employee notices.
  • Phrase: privacy-respecting monitoring is still effective - detect security risk without turning every packet into broadly accessible surveillance data.

Chunk 5 - Questions 41-50

41. What is egress filtering and why is it important?

  • Controls outbound traffic to external destinations - many attacks need outbound for C2, exfiltration, malware download, credential abuse.
  • Why: limits C2, reduces exfil paths, prevents reaching arbitrary internet services, forces traffic through monitored proxies, enforces approved SaaS/API, reduces blast radius.
  • Controls: default-deny outbound for sensitive workloads, allow approved destinations by domain/IP/service, proxies/NAT/cloud firewalls/VPC endpoints, block direct DNS except approved resolvers, monitor deny logs + unusual allowed, review exceptions.
  • Phrase: ingress controls protect what can reach you; EGRESS controls limit what a compromised system can reach after it's inside.

42. How would you design outbound access controls for workloads?

  • Based on purpose + required dependencies + sensitivity: inventory outbound deps -> classify destinations (internal/cloud/SaaS/internet/partner) -> prefer private endpoints -> route internet egress through controlled NAT/proxy/firewall -> default-deny for sensitive prod -> service identity not just IP -> log allowed+denied -> review exceptions/unused rules.
  • AWS: VPC endpoints for S3/DynamoDB/STS/ECR/Secrets Manager, SGs + network firewalls, avoid unrestricted NAT egress for sensitive services, Route 53 Resolver logs + Flow Logs.
  • Phrase: outbound policy should be as intentional as inbound - a production service rarely needs the whole internet.

43. What is DNS over HTTPS and how does it affect enterprise visibility?

  • DoH sends DNS over HTTPS instead of traditional transport - improves privacy vs local observers but reduces enterprise DNS visibility/policy enforcement.
  • Benefits: protects queries from passive local observers, reduces manipulation by untrusted networks, uses HTTPS transport.
  • Challenges: clients bypass corporate resolvers, lost filtering/logging, malware hides C2 lookups, port-blocking insufficient, privacy vs monitoring conflict.
  • Controls: provide approved enterprise DoH/DoT resolvers, configure managed browsers/devices to use them, block known public DoH endpoints, monitor HTTPS to unapproved resolvers, limit DNS log access/retention. Phrase: DoH isn't inherently bad - avoid unmanaged resolver bypass while respecting privacy.

44. What is split tunneling and what are the risks?

  • Remote users send some traffic through corporate tunnel, other directly to internet - better performance + less gateway load but changes visibility/control.
  • Benefits: lower latency for internet/SaaS, reduced VPN bandwidth, better UX. Risks: internet traffic bypasses inspection, local network threats interact with corporate device, DNS leaks/bypasses policy, attackers use direct paths for C2, complex policy.
  • Controls: endpoint protection + SWG for direct traffic, force sensitive apps through controlled paths, device posture + identity-aware access, clear DNS handling, monitor direct-to-internet from managed devices.
  • Phrase: split tunneling is a tradeoff not a flaw - needs endpoint + cloud-delivered controls because the VPN is no longer the only chokepoint.

45. How would you secure remote access without traditional VPN?

  • ZTNA / identity-aware proxy instead of broad network-level VPN: strong/phishing-resistant MFA, verify device posture, authorize PER APPLICATION not per network, IAP/ZTNA connector for private apps, avoid exposing full subnets, log user/device/app/action/risk, step-up for sensitive actions, browser-based limited access for contractor/BYOD.
  • Benefits: smaller blast radius than VPN, better user+device context, easier per-app policy, reduced lateral movement.
  • Phrase: remote access should grant access to the APPLICATION, not to the network neighborhood around it.

46. How would you design network controls for cloud workloads?

  • Combine account/project boundaries + VPC design + subnet routing + SGs + endpoint policies + egress controls + flow telemetry: separate envs by account, private subnets for non-public workloads, internet entry behind LB+WAF+DDoS, SGs as least-privilege resource firewalls, network firewall/egress proxy, private endpoints, restrict metadata access, enable Flow+DNS logs, IaC + policy checks, review routing/peering/transit gateways.
  • AWS layout: public subnet (LB only), private app subnet, private data subnet, egress (NAT+firewall or VPC endpoints), telemetry (Flow Logs/DNS/LB logs/CloudTrail).
  • Phrase: cloud networks are programmable so drift + sprawl are the enemy - treat network policy as code + continuously verify reachability.

47. What is security group sprawl and how do you manage it?

  • Cloud firewall rules grow without ownership/purpose/review/cleanup -> effective access broader than intended.
  • Symptoms: duplicate/unused SGs, broad 0.0.0.0/0 to admin ports, rules referencing old CIDRs/deleted services, no owner/ticket, emergency rules never removed, cross-env references.
  • Management: require owner+purpose+expiration+env tags, IaC, auto-detect broad/unused rules, continuously review high-risk ports + internet exposure, service-to-service templates, remove stale groups after observation, reachability analysis.
  • Phrase: security groups are CODE - sprawl happens when treated like manual tickets rather than governed policy artifacts.

48. How would you review firewall rules at scale?

  • Data-driven + risk-based (manual line-by-line insufficient): normalize into source/dest/port/protocol/owner/purpose/env/last-used -> prioritize internet-exposed/admin/DB/cross-env -> identify unused/shadowed/duplicate/broad -> validate vs observed traffic -> owners recertify risky rules -> CI policy checks for new rules -> auto-expire temporary -> test reachability for critical boundaries.
  • Phrase: review EFFECTIVE access not just configured rules - routes + NAT + peering + SGs + firewalls combine into the real policy.

49. What are common network security anti-patterns?

  • Flat network + broad internal trust, VPN gives entire subnet access, no egress filtering, SGs allow any for troubleshooting + never tighten, prod/dev/corporate share trust, admin ports on internet, TLS terminates at edge + plaintext backend without risk acceptance, no DNS logging, no flow logs until after incident, IP allowlists as workload identity, unmanaged DoH bypassing resolvers, over-inspection violating privacy, no owner/expiration for exceptions.
  • Phrase: biggest anti-pattern = confusing network REACHABILITY with AUTHORIZATION - being able to connect should never mean being allowed to act.

50. What questions would you ask during a network security design review?

  • Trust boundaries + protected assets? Allowed inbound/outbound/east-west paths? How are users/devices/workloads authenticated? Where is TLS terminated + re-encrypted? What prevents lateral movement after one compromise? Egress paths + monitoring? DNS/proxy/firewall/LB/flow logs collected? SG rules owned/reviewed/expired? What happens when QUIC/DoH reduce visibility? Remote access scoped to apps not networks?
  • Additional: prod/dev/corporate separated? high-value stores isolated? BGP/DNS/cert dependencies monitored? cloud routes/peering/transit/endpoints reviewed? privacy safeguards defined?
  • Phrase: prove not only that traffic CAN flow, but that ONLY the right traffic can flow, for the right identity, to the right service, with useful evidence when it goes wrong.
