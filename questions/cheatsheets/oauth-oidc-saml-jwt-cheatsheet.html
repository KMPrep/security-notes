enterprise data protection OAuth2, OIDC, SAML, JWT - Cheatsheet

Source: questions/answers/oauth-oidc-saml-jwt-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Explain OAuth2 Authorization Code Flow with PKCE step by step.

  • Modern default for browser/mobile/desktop/SPA - keeps tokens out of front channel + protects code from interception.
  • Flow: client creates code_verifier -> derives code_challenge = BASE64URL(SHA256(verifier)) -> auth request with response_type=code + exact redirect_uri + state + challenge + S256 -> server authenticates+consents -> redirects with code + state -> client validates state -> exchanges code + verifier -> server hashes verifier vs challenge -> issues tokens.
  • PKCE protects code exchange, state protects redirect from CSRF/mix-up, OIDC nonce binds ID token to auth request.
  • Phrase: PKCE necessary but NOT sufficient - still need exact redirect URI validation, issuer/token validation, secure storage, short lifetimes, revocation.

2. Why is PKCE required for public clients?

  • Public clients (mobile/desktop/CLI/SPA) can't keep a secret - shipped code is inspectable, embedded secret isn't really secret.
  • PKCE solves authorization CODE INTERCEPTION: stolen code is useless without the one-time code_verifier the legit client generated.
  • Best practices: use S256 not plain, fresh verifier per request, bind code to client ID + redirect URI + challenge, reject code reuse, use PKCE for confidential clients too (defense in depth).
  • PKCE doesn't authenticate the user or authorize API access - it protects the code exchange so the right client instance redeems the code.

3. What problem does the OAuth2 state parameter solve?

  • Binds the authorization response to the browser session that initiated it - primarily CSRF protection for OAuth redirects (also helps mix-up/injection).
  • Design: high-entropy random state, stored server-side/browser-bound, sent in auth request, matched at callback, expired after use.
  • Mistakes: static/predictable state, attacker-modifiable storage, not checked on callback, carrying sensitive cleartext data, accepted more than once.
  • Phrase: state answers "Is this authorization response for the login transaction my client started?"

4. What problem does the OIDC nonce parameter solve?

  • Binds an ID token to a specific authentication request - prevents replay/injection of an ID token from a different transaction (esp. implicit/hybrid flows).
  • Client sends fresh random nonce -> OP includes it in ID token -> RP validates nonce matches stored session value.
  • Part of full ID token validation (signature, issuer, audience, expiry, iat, azp, auth_time).
  • Phrase: nonce answers "Is this ID token the one produced for my authentication request?"

5. Why are state and nonce not interchangeable?

  • Different artifacts at different layers: state = OAuth redirect transaction binding (validated by OAuth client, CSRF protection); nonce = OIDC ID token binding (validated by RP, replay/injection protection).
  • state returned as redirect param; nonce embedded in ID token by the OP.
  • Only state -> ID token replay risk; only nonce -> redirect CSRF risk. Robust OIDC browser flows use BOTH.
  • Phrase: complementary - state binds front-channel response to client transaction, nonce binds identity assertion to auth request.

6. What does OIDC add on top of OAuth2?

  • OAuth2 = authorization framework (get access token for API); doesn't standardize user authentication or trustworthy identity assertion.
  • OIDC adds identity layer: openid scope, ID token (signed auth-event assertion), standard claims (iss/sub/aud/exp/iat/auth_time/amr/acr), nonce, UserInfo endpoint, discovery metadata, JWKS, logout specs.
  • OAuth = "allow this app to read my calendar"; OIDC = "sign me in using my enterprise IdP."
  • RP validates ID token then usually creates its OWN local session - ID token proves authentication, not a replacement for app authorization.

7. Why should an ID token not be used as an API access token?

  • ID token is for the client/RP (who authenticated); access token is for the resource server (what access granted) - using ID token as access token breaks audience + semantics.
  • Problems: ID token aud = client ID not API, lacks API scopes, enables token substitution, confuses authN with authZ, exposes identity claims to APIs.
  • Correct: client validates ID token -> creates session; client gets access token -> API validates it -> checks scopes/policy. API should REJECT an ID token (wrong audience).
  • Phrase: ID tokens for the relying party; access tokens for resource servers - mixing creates confused trust boundaries.

8. What claims must be validated before trusting an ID token?

  • Crypto: signature, expected alg (prevent unsafe algos), kid (right trusted key, not blind attacker input).
  • Semantic: iss exact match, aud includes client ID, azp matches when multiple audiences, exp/nbf/iat time checks, nonce matches stored, auth_time for recent-auth enforcement, acr/amr for MFA policy, sub stable user key.
  • Use iss + sub as stable federated key, NOT email (unless explicitly designed for email changes).
  • Residual: valid ID token = provider authenticated user, does NOT auto-grant app permissions - role/entitlement mapping needs local policy.

9. What claims must be validated before trusting a JWT access token?

  • Resource server must validate - don't accept any signed JWT just because the signature verifies under some known key.
  • Checks: signature, expected algorithm (prevent confusion), trusted iss, aud for THIS API, exp/nbf/iat, sub, client ID/azp, scopes/permissions, tenant/org, token type (access vs ID vs refresh), jti for replay.
  • API-specific: token for billing-api must not work at admin-api; customer-IdP token must not work in employee service unless explicitly trusted.
  • Publish shared validation library/sidecar pinning issuers/audiences/algos/JWKS. Phrase: claims are INPUTS to authorization - still need object/tenant/action checks.

10. What is the difference between issuer, audience, subject, and authorized party claims?

  • iss = issuer that minted token ("do I trust this IdP?"); aud = intended audience ("was this token meant for me?"); sub = subject ("who/what does this represent?"); azp = authorized party ("which client was authorized?").
  • ID token: iss=IdP, aud=client ID, sub=user, azp=client. Access token: iss=auth server, aud=API URL, sub=user, azp=client.
  • Common failure: accepting valid signature but WRONG audience - lets a token minted for one service be replayed to another.

Chunk 2 - Questions 11-20

11. What is the JWT alg:none vulnerability?

  • Verifier accepts a token claiming no signature algorithm - attacker removes signature, sets alg:none, modifies claims (sub/admin), server accepts.
  • Dangerous because attacker controls header+payload, server lets attacker-controlled metadata decide verification behavior - turns signed format into unsigned client JSON.
  • Controls: pin expected algorithms in server config, reject none, don't let token header decide if verification is required, mature libraries configured strictly, validate iss/aud/exp AFTER signature, test unsigned/tampered tokens in CI.
  • Phrase: algorithm confusion bug where verifier trusts the untrusted header - require signature verification with explicitly allowed algorithms + keys.

12. Explain JWT key confusion attacks.

  • Verifier accepts a token signed with unexpected algorithm/key type. Classic: RS256 expected but verifier accepts HS256 using the RSA PUBLIC key as the HMAC secret -> attacker signs forged token with public key.
  • Root cause: verifier lets attacker-controlled token metadata influence algorithm + key selection.
  • Controls: pin accepted algorithms per issuer/client, don't mix symmetric+asymmetric in same path, bind each kid to expected issuer+algo+key type, reject unexpected alg/typ/key type, don't fetch keys from attacker URLs, separate ID/access token paths.
  • Phrase: JWT headers are routing HINTS not policy - verifier uses trusted config to decide algos+keys, then checks token consistency.

13. How can a weak JWKS implementation be exploited?

  • Failures: dynamic JWKS URL from token header (jku -> attacker keys), no issuer binding, weak kid handling (path traversal/injection), duplicate kid across issuers, excessive cache lifetime (revoked key still trusted), no rate control (fetch storm), silently accepting unknown keys.
  • Secure: configure trusted issuers + JWKS URIs out of band, fetch only from pinned issuer metadata, bind key IDs to issuer+algo+key type, bounded cache TTL + safe refresh on unknown kid, rate-limit fetching, reject ambiguous lookups, monitor rotations + validation failure spikes.
  • Attack example: jku=https://evil/jwks.json -> vulnerable verifier fetches + verifies forged token. Secure verifier ignores attacker URLs, uses only configured issuer metadata.

14. What risks exist when accepting tokens from multiple identity providers?

  • Each issuer = own key lifecycle, claims, assurance, tenant model, recovery process - weaker providers become path into stronger environments.
  • Risks: issuer confusion, subject collision (same sub = different users), audience confusion, claim schema mismatch, assurance mismatch (phishing-resistant MFA vs passwords), tenant confusion, key confusion, account linking abuse.
  • Controls: explicit issuer allowlist, store identity as (issuer, subject) not just sub/email, validate audience per RP/API, tenant allowlists, normalize claims through mapping layer, treat assurance claims as policy inputs, local authz after authN, log issuer/subject/tenant/client/mapping.
  • AWS: customer-IdP tokens never valid for employee admin APIs - separate trust stores, audiences, tenants, authz paths.

15. How would you securely implement token validation in a microservice platform?

  • Consistency critical - custom per-service validation eventually accepts wrong issuer/skips audience/mishandles key rotation.
  • Shared validation library/sidecar: pin issuers + allowed algorithms, validate signature with issuer-bound keys, validate audience per service, exp/nbf/iat + clock skew, token type, scopes/permissions/tenant/client, separate ID/access paths, safe JWKS cache, FAIL CLOSED for sensitive endpoints.
  • Gateway does coarse validation; each service still enforces service-specific authz; protect internal identity context (mTLS + signed headers); use workload identity for east-west not forwarded user bearer tokens.
  • Log issuer/audience/subject/client/scopes/outcome; alert on invalid-signature spikes, unknown issuers, wrong audience, unknown kid storms.

16. How would you design token revocation in a distributed system?

  • Control-plane problem: identify what's revoked, record state, ensure resource servers honor it fast enough for the risk.
  • Targets: one access token, one/family refresh token, one/all user sessions, all client tokens, all tokens signed by a compromised key.
  • Choices: short-lived access tokens (natural expiry), centrally revocable refresh tokens, token family IDs for rotation/family invalidation, introspection for high-risk APIs, short lifetimes + fast refresh revocation for low-risk, reliable messaging propagation, short cache TTLs.
  • Tradeoff: stateless JWTs scale but hard to revoke immediately; opaque/introspected easier to revoke but add latency. Common: short-lived JWT access + centrally managed refresh.

17. What are the tradeoffs between short-lived tokens and revocation lists?

  • Short-lived: limits exposure without runtime lookup, but can't stop misuse until expiry. Revocation list: stops specific tokens before expiry, but needs state/lookup/cache/propagation. Introspection: real-time central check but latency + availability dependency.
  • Practical: access tokens short-lived + audience-specific + minimal claims; refresh longer-lived but rotated + revocable; high-risk APIs check revocation/introspect; low-risk rely on short lifetimes + refresh revocation; incidents revoke refresh families + force reauth.
  • Phrase: short lifetimes reduce dependence on perfect revocation; revocation lists reduce exposure during the token lifetime - use both by risk.

18. How would you handle refresh tokens securely?

  • High-value (mint new access tokens) - treat closer to long-lived sessions: issue only to clients needing long-lived access, store securely (server-side web, Keychain/Keystore mobile, NEVER browser local storage for high-risk), rotate on every use, detect reuse as theft, bind to client/device/sender-constrained proof, scope to audience/client/tenant/session, absolute + idle lifetime, revoke on logout/password-change/MFA-reset/recovery/device-loss/suspicious, log issuance/rotation/reuse/revocation.
  • Browser: prefer backend-for-frontend (refresh stays server-side, browser gets secure session cookie); cautious with long-lived refresh in script-accessible storage.

19. What is refresh token rotation and what threat does it reduce?

  • Server issues a new refresh token each use + invalidates the old -> each refresh token single-use.
  • Theft scenario: attacker steals RT1, client already used RT1 + got RT2, attacker tries RT1 -> server sees reuse -> revokes token family + forces reauth.
  • Reduces PERSISTENT REPLAY after refresh-token theft (doesn't prevent initial theft but improves detection/containment).
  • Details: track token family/session lineage, minimal grace windows for retries, treat reuse as high-risk, log old/new token IDs + client/device/IP, don't issue non-rotating refresh to public clients. Must pair with secure storage.

20. How would you detect refresh token theft?

  • High-signal: reuse of already-rotated token, same family from two devices/geos, refresh from new ASN/proxy/country after stable history, refresh after logout/password-change/MFA-reset/device-revoke, refresh for disabled users/deleted clients, sudden minting volume increase, different client instance, refresh-then-suspicious-activity (export/privilege change).
  • Response: revoke token family, revoke derived access tokens, mark session high-risk, require full reauth + strong MFA, notify user/admin, preserve evidence, inspect recent grants/API/factor changes.
  • Logs: token family ID, client ID, subject, device ID, IP/ASN, rotation result, issued token ID. Phrase: reuse-after-rotation is one of the cleanest theft signals + need anomaly detection for geo/device/client/post-refresh behavior.

Chunk 3 - Questions 21-30

21. Compare opaque tokens and self-contained JWTs.

  • Opaque = random reference looked up by server: easy revocation + minimal exposure, but needs introspection/DB lookup. JWT = signed token carrying claims: fast local validation + low lookup cost, but harder revocation + claim freshness.
  • Opaque good for revocation/privacy/centralized policy (leak reveals no claims). JWT good for many services validating at scale without central lookup (dangerous if long-lived/overstuffed/wrong-audience/no-revocation).
  • Hybrid: short-lived JWT access for high-volume APIs + opaque/centrally-stored refresh for session+revocation; add introspection/revocation check for very sensitive APIs.

22. When would you prefer opaque tokens over JWTs?

  • When centralized control + revocation + privacy matter more than stateless validation: high-risk APIs needing immediate revocation, consumer payload privacy, frequent entitlement changes, complex dynamic policy, logout-sensitive apps, regulatory minimization, internal platforms tolerating introspection latency.
  • Why: leak reveals little (just a reference), central revoke/status change, fresh policy context during introspection, claims not cached in long-lived bearer.
  • Tradeoffs: introspection latency, auth-server runtime dependency, caching needed (freshness tradeoff), multi-region design.
  • Example: admin/IAM-policy/customer-data API -> opaque + introspection or very short JWT + revocation; high-volume low-risk reads -> short-lived JWT.

23. How would you design service-to-service authorization using OAuth2?

  • Client credentials / token exchange / workload identity federation - token represents the CALLING WORKLOAD scoped to the target API, not a broad shared secret.
  • Principles: unique identity per service, prefer mTLS/private-key-JWT/workload-identity/platform creds over static secrets, short-lived tokens with audience = target service, scope to actions+resources, validate iss/aud/exp/sub/client/scopes at resource service, least privilege per service+env, log actor/target/scopes/decision, NO shared backend-service credential.
  • AWS: EKS uses OIDC federation/Pod Identity (no static keys); mesh provides mTLS workload identity; auth server issues audience-specific tokens for app-layer authz.
  • Distinction: authN proves which service is calling; authZ decides whether it can perform the specific action on the target resource.

24. What is OAuth2 client credentials flow and when is it appropriate?

  • Machine client authenticates to auth server + gets access token for ITS OWN identity - no human user.
  • Appropriate: backend->backend, daemon/batch job, CI/CD calling deploy APIs, monitoring/security tooling, machine-owned integration.
  • NOT appropriate: action on behalf of a user (needs user context), client can't protect credential with no stronger alternative, needs user consent semantics, token gives broad cross-service access.
  • Controls: strong client auth (mTLS/private-key-JWT/workload-identity/rotated secrets), narrow audience+permission scope, short-lived, separate identities per service+env, monitor unusual issuance+calls.
  • Phrase: client credentials = machine acting AS ITSELF, not impersonating a user.

25. What are the risks of OAuth2 implicit flow?

  • Returns tokens directly through browser front channel - no longer recommended (Code + PKCE is safer).
  • Risks: access tokens in front channel, leak via history/devtools/extensions/logs/referrers, no PKCE-protected back-channel exchange, higher substitution/replay risk, SPA local-storage XSS impact, unsafe/limited refresh handling.
  • Better: Authorization Code + PKCE, backend-for-frontend for high-risk apps, keep tokens out of URLs+browser storage, secure cookies for local sessions, CSP + XSS defenses.
  • Migration: SPA receiving token in URL fragment -> migrate to code+PKCE (receives only code, exchanges with verifier); high-value apps do exchange server-side.

26. What is device authorization flow and where is it useful?

  • For limited-input devices: device gets device code + user code, user completes auth on a separate browser-capable device, device polls token endpoint.
  • Useful: smart TVs, CLI tools, IoT with limited keyboards, dev tools, printers/shared appliances, headless systems.
  • Controls: short expiration for user+device code, polling interval enforcement, clear app+device context at approval, phishing-resistant login, user confirms device/app name, limit scopes, revoke on device loss/retire.
  • Risk: phish the user code / trick user into authorizing attacker device -> approval page must show strong context (app, device, location, scopes, warning for unexpected codes).

27. What are the security risks of redirect URI misconfiguration?

  • If auth server sends codes/tokens to attacker URI, attacker steals them.
  • Misconfigs: wildcard URIs, prefix matching (good.com.evil.com), open redirect at registered URI, allowing HTTP, custom scheme collision, missing exact path check, user-controlled redirect param.
  • Example: registered https://app.example.com/callback, vuln validation startsWith, attacker uses https://app.example.com.evil.net/callback.
  • Controls: exact redirect URI matching for confidential web clients, avoid wildcards, require HTTPS (except loopback native), claimed HTTPS/app links for mobile, no open redirects, bind code to client ID + redirect URI, review redirect changes as security-sensitive.

28. How would you validate redirect URIs safely?

  • Strict + deterministic + pre-registered: store allowed URIs at registration, exact string match after canonicalization, require HTTPS for web, reject fragments + userinfo (user@host) + ambiguous hostnames/encoded delimiters/control chars, NO suffix/prefix/loose-regex matching, bind code to exact redirect URI, re-check same URI at token exchange.
  • Native nuance: loopback http://127.0.0.1:{port}/callback acceptable with PKCE + correct local handling; custom URL schemes riskier (another app may claim scheme).
  • Redirect URI changes should be audited + require owner approval - a malicious/mistaken redirect URI is a token exfiltration route.

29. What is authorization code interception?

  • Attacker obtains the authorization code before the legit client exchanges it.
  • Examples: malicious app registers same custom scheme, extension reads redirect URL, proxy/debug log captures callback, open redirect forwards code, localhost callback intercepted, network/device malware.
  • Without PKCE a stolen code may be enough; with PKCE attacker also needs the verifier (generated by legit client, not sent through redirect).
  • Controls: Code + PKCE, exact redirect URI validation, claimed HTTPS redirects for mobile, reject code reuse, short-lived codes, bind code to client ID + redirect URI, monitor failed exchanges from PKCE mismatch.

30. What is token substitution?

  • Attacker presents a valid token in the WRONG context and the RP accepts it - cryptographically valid but semantically invalid for service/client/issuer/tenant/type.
  • Examples: ID token as access token, access token for API A accepted by API B, untrusted-issuer token accepted, attacker-client token accepted by victim client, cross-tenant token accepted, stale-claims token after privilege removal.
  • Root cause: incomplete validation + confused token semantics.
  • Controls: validate iss/aud/token-type/azp/tenant exactly, separate ID/access paths, enforce local authz after validation, use state+nonce. Phrase: not forging - replaying a REAL token where it wasn't meant to be trusted.

Chunk 4 - Questions 31-40

31. What is consent phishing?

  • Malicious OAuth app tricks user/admin into granting permissions -> attacker accesses data WITHOUT stealing the password.
  • Dangerous because: MFA doesn't stop it (user willingly authorizes), persists via refresh tokens, accesses email/files/contacts/calendar/cloud per scopes, user misunderstands permissions, admin consent spans many users.
  • Controls: restrict user consent for high-risk scopes, require admin approval for sensitive perms, verify publisher identity, clear consent screens (app/publisher/scopes/tenant), monitor new grants (esp high-risk scopes), alert on exec/admin/sensitive-team consent, revoke suspicious grants+refresh tokens fast, education as backup not primary.
  • Phrase: abuses delegated AUTHORIZATION not just authentication - control plane must govern which apps get which scopes from which users.

32. How would you protect users from malicious OAuth applications?

  • Preventive: app allowlist for sensitive envs, block user consent for high-risk scopes (mail read/file read/offline access/directory read/admin APIs), admin review for new apps+sensitive scopes, verify publisher, limit multi-tenant access, least-privilege+incremental consent, restrict refresh tokens for untrusted apps, tenant policies for consumer/unmanaged apps.
  • Detection: new grant w/ high-risk scopes, privileged/exec consent, unusual location/device, rapid bulk data access post-grant, many users grant same new app, unrelated permissions requested, low reputation/new publisher/suspicious redirect URIs.
  • Response: revoke grants+refresh tokens, disable app tenant-wide, review accessed data+API calls, notify users/admins, add detections for similar app IDs/publishers/redirect URIs/scopes.
  • AWS analogy: like IAM role trust policies restrict who can assume - OAuth consent policies restrict which apps get delegated access + conditions.

33. What is the difference between delegated authorization and authentication?

  • AuthN proves who the user/workload is; delegated authZ grants a client limited permission to act ON BEHALF OF a resource owner.
  • AuthN: "who authenticated?" (Alice signs in via OIDC); delegated authZ: "what can this client access for Alice?" (calendar app reads Alice's calendar).
  • OAuth2 = mainly delegated authz; OIDC adds authentication on top.
  • Mistakes: using access-token issuance as proof of login without OIDC; treating authN as authZ (admin access just because authenticated via trusted IdP) - app still needs local role/tenant/entitlement checks.

34. Compare OAuth2, OIDC, and SAML for enterprise SSO.

  • OAuth2 = delegated authorization (access tokens opaque/JWT, API access+consent); OIDC = authN layer on OAuth2 (ID token + OAuth tokens, modern web/mobile SSO); SAML = enterprise federation/SSO (XML assertions, workforce SaaS + legacy).
  • SAML strengths: mature enterprise SSO, broad SaaS compat, certificate-based federation, browser workforce SSO. Weaknesses: complex XML signature handling, severe validation mistakes, awkward mobile/API, tricky logout+attribute mapping.
  • OIDC strengths: modern format+libraries, mobile/API fit, discovery+UserInfo, natural with OAuth delegated authz. Weaknesses: common validation mistakes, discovery/multi-tenant issuer trust issues, claims over-trusted for authz.
  • Phrase: OIDC for new app SSO + API ecosystems, SAML for enterprise SaaS compatibility, OAuth2 for delegated API authorization (not sign-in).

35. How does SAML assertion validation work?

  • SP receives SAML response at ACS endpoint, must validate before creating local session.
  • Checklist: hardened SAML library (not custom XML sig), disable unsafe parser features (XXE), validate signature vs trusted IdP cert, ensure app consumes the SAME signed assertion validated, validate issuer + audience (= SP entity ID) + recipient/destination (= ACS) + NotBefore/NotOnOrAfter (tight skew) + InResponseTo (SP-initiated), enforce one-time use/replay detection for assertion IDs, map subject+attributes through controlled local policy.
  • Phrase: signature alone is NOT enough - a validly signed assertion for a different service/audience/time window must still be rejected.

36. What is XML signature wrapping in SAML?

  • Attacker manipulates XML so SP validates the signature on one element but USES a different attacker-controlled element for authN/attributes.
  • Idea: signed assertion says Alice; attacker inserts unsigned assertion says Admin; vuln parser validates signed Alice but app reads unsigned Admin.
  • Root causes: signatures reference nodes by ID+canonicalization, app selects nodes by position/name instead of signed reference, parser+library disagree on trusted node, hand-rolled parsing/unsafe defaults.
  • Impact: authN bypass, privilege escalation via forged attributes, tenant confusion. Phrase: "signature valid" doesn't mean "the assertion my app USED was signed."

37. How would you prevent XML signature wrapping attacks?

  • Use mature SAML libraries + strict validation binding the consumed assertion to the verified signature reference.
  • Controls: don't hand-roll XML sig validation, disable XXE, require signed assertions/responses per policy, resolve signed element by validated ID reference, reject duplicate IDs/ambiguous references, app reads claims ONLY from validated signed assertion, validate iss/aud/recipient/destination/time AFTER signature, test with wrapping payloads, patch libraries.
  • Pattern: library validates response -> returns trusted assertion object -> app maps claims only from that object -> never reparses raw XML for identity fields.
  • SP review must include: malicious XML, duplicate IDs, extra unsigned assertions, nested assertions, altered attributes.

38. What are common SAML misconfigurations?

  • Missing audience validation (cross-SP acceptance), missing recipient/destination check, weak time validation (replay), no InResponseTo (login CSRF), over-trusted IdP-initiated flow, unsigned assertions accepted, mishandled cert rollover, over-trusted attributes (privilege escalation), no replay detection, unsafe XML parser (XXE/wrapping).
  • Secure: signed assertions from trusted cert, validate audience/issuer/destination/recipient/time, narrow validity windows, track assertion IDs for replay prevention, IdP metadata as controlled config, separate SAML authN from app authz, test cert rollover before prod.
  • Phrase: accepting any signed assertion from a trusted IdP is NOT enough - must be for this SP, this ACS, this time window, this expected flow.

39. How would you securely map SAML or OIDC claims to internal roles?

  • Claim = INPUT to local policy, not automatic application authority.
  • Design: stable keys (issuer, subject) for OIDC / trusted persistent NameID for SAML, explicit external-group->internal-role mapping, scope mappings by tenant/app/env/issuer, admin review for privileged mappings, don't trust user-controlled claims, don't map email domain->admin, validate group claim source+freshness, prefer local entitlements for high-risk, log claims used + resulting role.
  • Mistakes: trusting email_verified as employment proof, treating any groups=admin as admin without issuer scoping, failing open when group claims missing, letting customers map into provider-admin roles, not reviewing effective perms after group sync.
  • For privileged roles: require local approval + JIT activation + audit even if IdP sends admin group claim.

40. Why is email verification not enough for strong identity assurance?

  • Proves control of an email at a point in time - NOT real-world identity, employment, authorization, or long-term ownership.
  • Limits: email accounts compromised, addresses recycled/reassigned, personal email != corporate affiliation, doesn't prove MFA strength, domains spoofable with weak issuer validation, controlling alice@example.com != being employee Alice, weak email recovery.
  • Stronger evidence: HR/directory lifecycle, trusted IdP + tenant validation, government ID/KYC, manager/sponsor approval, device issuance/managed proofing, contract/vendor records, phishing-resistant auth for returning sessions.
  • Example: granting admin to anyone with verified @company.com is unsafe if self-registration possible / domain not tied to controlled tenant. Phrase: email verification is a CONTACT-CHANNEL proof, not strong identity proof.

Chunk 5 - Questions 41-50

41. How would you design logout across OIDC, SAML, and local sessions?

  • Multiple sessions: IdP session, RP/SP session, app cookies, access tokens, refresh tokens, downstream services.
  • Principles: define local vs global logout vs security revocation separately; local logout must work even if IdP logout fails; revoke refresh tokens + families; clear cookies but don't rely only on client-side deletion; OIDC RP-initiated/front-channel/back-channel logout where supported; SAML Single Logout only if both ends reliable (else best-effort); server-side sessions + central revocation for high-risk; "sign out everywhere" revokes all sessions/refresh tokens/remembered devices/grants.
  • Tradeoff: federation logout unreliable across browsers/mobile/SaaS/third-party-cookie restrictions - security-sensitive logout relies on local session invalidation + token revocation, not assuming federation completed.

42. What are the risks of dynamic OIDC discovery?

  • Useful for config + key rotation, but risky when user input controls the issuer/metadata location.
  • Risks: attacker-supplied issuer URL, fetch attacker-controlled metadata, JWKS URI -> attacker keys, accept untrusted-issuer tokens, SSRF via discovery fetch, unexpected metadata changes trusted immediately, multi-tenant issuer confusion, weakened validation via algo/endpoint metadata.
  • Controls: pin trusted issuers/tenant IDs, fetch discovery only from allowlisted domains, validate metadata iss matches configured exactly, pin client registrations to expected issuer, cache safely + monitor changes, restrict outbound network for fetchers, don't accept JWKS URLs from token headers, treat discovery as trusted config not runtime input.
  • Phrase: discovery is safe automating retrieval from ALREADY-trusted issuers; dangerous when it DECIDES which issuers are trusted.

43. How can OIDC discovery become a trust-boundary issue?

  • Metadata tells RP where to send users, exchange codes, fetch keys, which issuer to expect - attacker controlling discovery input influences core trust decisions.
  • Vuln flow: user enters email domain -> app derives issuer from domain -> fetches discovery from attacker domain -> accepts attacker JWKS -> attacker signs token for victim email -> session created.
  • Secure model: tenant admin registers IdP metadata via authenticated admin flow -> verify domain/tenant ownership -> security review approves issuer+endpoints -> runtime login uses only pre-approved config -> validation requires exact iss + expected audience + trusted keys.
  • Phrase: discovery should RETRIEVE metadata for a trusted issuer, not DECIDE that an issuer is trusted.

44. How would you threat-model a multi-tenant OIDC login platform?

  • Assets: tenant identity config, client secrets+redirect URIs, codes/ID/access/refresh tokens, user-tenant mappings, claim-role mappings, JWKS/discovery metadata, admin consent/app registration.
  • Trust boundaries: browser->callback, tenant-admin->federation config, platform->external IdP metadata, validation service->JWKS cache, external claims->internal roles, tenant data boundary.
  • Threats: malicious tenant registers rogue IdP + cross-tenant login, redirect URI leaks codes, subject collision across issuers, email-domain tenant routing abuse, JWKS confusion across tenants, overbroad claim mapping grants admin, consent phishing, logout/revocation inconsistencies.
  • Controls: identity = tenant+issuer+subject, pin issuer+JWKS per tenant, exact redirect URI per client, require tenant ownership proof before IdP config, scope claim mappings to tenant+issuer, deny by default on ambiguous routing, log all config changes, test cross-tenant token substitution.

45. How would you investigate a suspected federation misconfiguration incident?

  • Preserve evidence first, then determine if it's token validation / IdP config / claim mapping / redirect handling / account linking.
  • Steps: scope affected tenants/clients/issuers/time -> preserve logs+config snapshots -> identify suspicious logins/issuances/role mappings -> review recent federation config changes -> validate iss/aud/redirect/JWKS/claim mapping -> look for token substitution across issuers/audiences/tenants/types -> review account linking + new external bindings -> inspect consent grants + refresh issuance -> revoke affected sessions/tokens/grants/secrets -> add detections+tests.
  • Evidence: token headers/claims (never secrets/full tokens broadly), validation failure/success reason codes, IdP metadata+JWKS versions, redirect URI config, SAML/OIDC discovery snapshots, role mapping rules+group claims, admin audit trail.
  • Containment by failure type: wrong-audience -> patch validators + revoke sessions; malicious consent -> grant revocation; SAML cert confusion -> metadata rollback + assertion replay review.

46. What logs would you collect for OAuth/OIDC troubleshooting and security?

  • Reconstruct the transaction WITHOUT leaking secrets.
  • Fields: auth request (client/redirect/scopes/state-present/PKCE-present), auth response (result/error/issuer/client/redirect), token exchange (client/grant-type/PKCE-result/token-type/scopes/subject), token validation (iss/aud/algo/kid/outcome/failure-reason), refresh (family ID/client/subject/rotation/reuse), consent (app/publisher/scopes/user/admin-flag), discovery (issuer/version/JWKS-URI/result), logout (session hash/revocation scope/family revoked).
  • NEVER log: codes, access/refresh/ID tokens, client secrets, PKCE verifiers, full cookies.
  • Detection: PKCE mismatch spikes (code interception), unknown kid spikes (key confusion), wrong-audience failures (substitution), new high-risk consent (phishing), refresh reuse (theft). AWS: centralize IdP/app-auth/gateway/CloudTrail/SaaS logs into security lake + tamper-resistant retention.

47. How would you detect anomalous OAuth consent grants?

  • Monitor like privileged access changes - durable delegated access.
  • High-risk signals: new app w/ high-risk scopes, unexpected offline_access/refresh capability, admin consent to new multi-tenant app, consent by exec/finance/security/admin, many users same app short time, unverified/new/low-reputation publisher, suspicious/new redirect URI domain, large data access shortly after grant, scopes unrelated to stated purpose.
  • Model: score app reputation + scopes + user risk -> high-risk grant -> alert+hold; else monitor usage.
  • Response: revoke grants + refresh tokens, disable/block app tenant-wide, review API activity, notify users/admins, add app ID/publisher/redirect URI indicators.

48. How would you prevent confused deputy issues in OAuth integrations?

  • A service with legit authority tricked into using it for the wrong caller/client/audience/resource.
  • Examples: API accepts wrong-audience token, client uses token minted for another resource, auth server doesn't bind code to redirect+client, service accepts delegated token but ignores tenant/subject, cloud integration lets customer input choose target role without external ID/audience binding.
  • Controls: bind code to client ID+redirect+PKCE, bind access tokens to audience+scopes, validate iss/sub/tenant/client/aud at resource server, resource indicators/audience-specific tokens, state+nonce binding, cloud integrations use external IDs/source ARN/source account, log actor+effective subject, avoid broad shared service identities.
  • AWS: cross-account role assumption uses external ID/strong source conditions so one customer can't trick a SaaS integration role against another's resources. Phrase: bind authority to the original actor + intended client + target audience + resource.

49. How would you securely integrate third-party SaaS with enterprise SSO?

  • Steps: choose OIDC/SAML by SaaS support+standard, register SaaS as IdP app, exact redirect URIs (OIDC) / exact ACS+entity IDs (SAML), signing keys/certs with planned rollover, assign via groups (not all-user default), minimal explicit claim mapping, SCIM for provisioning/deprovisioning, MFA/conditional access by data sensitivity, limit admin roles + JIT, test login/logout/deprovisioning/role-changes/cert-rollover, centralize SaaS audit logs.
  • Review: does SaaS validate SAML audience/recipient/destination/signature? does OIDC validate iss/aud/nonce/redirect? what on IdP user disable? are local SaaS accounts still usable? can users self-link external identities? admin roles local or IdP-controlled?
  • AWS: IAM Identity Center + external IdP -> validate SAML metadata, strong upstream MFA, deliberate group->permission-set mapping, monitor CloudTrail for role assumptions + assignment changes.

50. What are the top five token-handling mistakes you look for in code review?

  • 1) Skipping SEMANTIC validation (verifies signature but not iss/aud/exp/type/tenant/scopes). 2) Trusting attacker-controlled HEADERS (alg:none, algo confusion, untrusted kid/jku/dynamic JWKS). 3) Wrong token for the job (ID token as access token, access token as session w/o local checks, refresh token sent to resource servers). 4) Unsafe storage/logging (local storage/URLs/logs/analytics/crash reports). 5) No revocation/rotation (long-lived JWTs, non-rotating refresh, no family invalidation, logout revokes nothing server-side).
  • Other findings: multi-issuer w/o issuer-specific config, email as only identity key, missing nonce/state, over-trusting claims for authz, failing open on JWKS/introspection failure, excessive lifetime, sensitive data in signed-but-unencrypted payloads.
  • Phrase: "if this token were stolen, where could it be used, for how long, and how would we revoke it?"
