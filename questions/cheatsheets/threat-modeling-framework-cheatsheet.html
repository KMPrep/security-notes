enterprise data protection Threat Modeling Framework - Cheatsheet

Source: questions/answers/threat-modeling-framework-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Threat-model an OAuth2 authorization server.

  • Scope: authenticates users, collects consent, issues codes/access/refresh/ID tokens, publishes discovery+JWKS. Assets: user identity+session, auth codes, tokens, client registrations+redirect URIs, signing keys+JWKS, consent+scopes.
  • Threats: code interception, open redirect/mix-up, weak client auth, token substitution/confused deputy, refresh replay, consent phishing, signing key compromise, overbroad scopes, discovery/JWKS poisoning.
  • Controls: PKCE for public clients, exact redirect URI match, strong client auth, state+nonce validation, audience+issuer validation, refresh rotation w/ reuse detection, key rotation+protection, consent clarity+scope minimization, audit grants/issuance/client changes/key events.
  • Phrase: most important boundary = client vs user vs authz server - prove token issued to RIGHT client, for RIGHT user, with RIGHT audience+scope.

2. Threat-model a login flow using STRIDE.

  • Components: browser, login service, identity store, MFA, session store, risk engine, logging. Apply STRIDE per flow: Spoofing (credential stuffing/fake IdP -> MFA+bot defense+phishing-resistant), Tampering (return URL/cookie -> signed state+secure cookies+allowlist), Repudiation (deny login -> auth event logs+request IDs), Info disclosure (password/token leak -> TLS+no secret logging+HttpOnly), DoS (lockout abuse -> risk-based throttling), EoP (session fixation/MFA bypass -> rotate session after auth+server-side MFA).
  • Requirements: rate limit by account/IP/device/ASN, secure password hashing, rotate session on login, bind MFA to server-side transaction, log success/failure/step-up/recovery/anomaly.
  • Phrase: STRIDE structures the conversation, but the OUTPUT is requirements - anti-automation, secure sessions, MFA integrity, login observability.

3. Threat-model a data export feature.

  • High-risk: packages many records for easy movement. Assets: exported data, job metadata, download link, authz policy, audit trail.
  • Threats: low-priv export, cross-tenant export, excessive fields, link leakage, insider bulk exfil, job bypassing row/field controls, long-lived files, missing audit.
  • Controls: explicit export permission, REUSE same authz filters as interactive views, field-level redaction, async jobs w/ approval for sensitive, short-lived single-use signed links, watermark/attribution, log requester/query/row-count/fields/tenant/download/expiration, alert on large/unusual exports.
  • Phrase: exports are data EXFILTRATION BY DESIGN - make allowed exfil auditable, scoped, revocable.

4. Threat-model a file upload service.

  • Crosses many boundaries: client->API->storage->scanners->preview->download. Threats: malware, stored XSS via active content, parser exploit (image/doc), zip/decompression bombs, cross-tenant access, path traversal in filenames, public exposure via links, metadata leakage.
  • Controls: authenticate uploader + authorize target workspace/tenant, quarantine first, server-side object names, validate type by content+extension, scan+sandbox processing, limit size/archive-depth/processing-time, serve risky files as attachment from separate origin, log upload/scan/release/download/delete.
  • Phrase: model the whole file LIFECYCLE, not just the upload endpoint.

5. Threat-model a cloud metadata service.

  • Local endpoint serving instance identity tokens+config. Main threat: untrusted code/SSRF reaching it. Assets: temp workload creds, SA/role identity, instance config+user data, request path.
  • Threats: SSRF reads creds, container escapes isolation, untrusted workload on same node accesses node identity, secrets in user data, overprivileged role/SA, metadata token reused for cloud API abuse.
  • Controls: IMDSv2 on AWS, GCP metadata headers + workload identity, low hop limits + block container access, no secrets in metadata/startup scripts, least-privilege workload identities, detect unusual credential use after metadata access.
  • Phrase: risk = REACHABILITY x PRIVILEGE - reducing either reduces blast radius.

6. Threat-model a secrets management platform.

  • Protects credentials/keys/passwords/certs/signing material - defend data access AND admin control. Assets: secret values, metadata, encryption keys, access policies, rotation jobs, audit logs.
  • Threats: unauthorized read, admin self-grants access, leak via logs/env vars, rotation breaks services, stale secrets valid after compromise, backup/replica exposes values, broad KMS decrypt.
  • Controls: least-privilege by workload identity, separate secret-admin/reader/KMS-admin, versioning+auto rotation, break-glass w/ approval+alerting, no raw secrets in logs, audit read/write/delete/policy/rotation, regular review+secret scanning.
  • Phrase: a high-value CONTROL PLANE - the model must include administrators + automation, not only external attackers.

7. Threat-model a DLP inspection pipeline.

  • Ingests, inspects, classifies, enforces/alerts - the pipeline processes sensitive data + must be protected. Assets: raw content, inspection results, classification rules, enforcement decisions, exception records, reviewer access.
  • Threats: sensitive content leaked via DLP logs/findings, malicious files exploit engines, false negatives allow exfil, false positives block business, insider abuses reviewer access, policy changes silently weaken enforcement, residency violations during inspection.
  • Controls: minimize+redact stored findings, sandbox parsers/scanners, version+review policy changes, privacy controls on reviewer workflows, SoD for authoring vs exception approval, log inspection/decision/reviewer-action/override, test rules against representative data.
  • Phrase: DLP pipelines are SENSITIVE-DATA PROCESSORS - protecting the detector is part of protecting the data.

8. Threat-model a multi-tenant SaaS application.

  • Core risk: cross-tenant access via identity/authz/isolation/jobs/logs/support tooling. Assets: tenant data+config, identity+roles, admin/support tools, billing+integration secrets, audit logs.
  • Threats: BOLA/IDOR across tenants, client-supplied trusted tenant ID, background job w/o tenant scope, shared cache returns other tenant data, support admin overreach, misconfigured storage prefix/query, logs expose tenant data.
  • Controls: server-derived tenant context, mandatory tenant filters in data layer, object+field-level authz, per-tenant encryption/storage where justified, support approval+JIT, cross-tenant CI tests, audit all admin+support.
  • Phrase: every path that reads data must CARRY tenant context - if tenant context is optional, cross-tenant bugs WILL appear.

9. Threat-model a data lake containing customer PII.

  • Concentrates PII for analytics/ML/reporting. Risks: unauthorized access, excessive retention, re-identification, uncontrolled downstream copies. Assets: raw PII, curated tables, catalog+labels, access grants, query logs, derived datasets+exports.
  • Threats: broad analyst access, public/cross-project exposure, weak de-identification, unapproved re-identifying joins, copies to less-secure workspaces, missing retention/deletion, broad KMS decrypt.
  • Controls: classify+tag, least-privilege+purpose-based access, row+column controls, tokenize/pseudonymize high-risk IDs, monitor queries+exports, retention+deletion workflows, audit logs OUTSIDE the lake project, approval for sensitive joins+external sharing.
  • Phrase: needs GOVERNANCE as much as encryption - track who can query, export, join, copy.

10. Threat-model a CI/CD pipeline.

  • Privileged path source->prod; compromise = cloud compromise or supply-chain incident. Assets: repo, build runners, pipeline config, secrets+deploy creds, artifacts+images, prod, signing keys+provenance.
  • Threats: malicious PR steals secrets, compromised dependency/plugin, config modified to deploy backdoor, leaked long-lived keys, artifact tampering between build+deploy, unprotected branch deploys to prod, runner escape/shared contamination.
  • Controls: OIDC federation w/ scoped roles, no privileged secrets for untrusted PRs, protected branches+required reviews, separate build+deploy identities, artifact signing+provenance, dependency pinning+scanning, ephemeral isolated runners, prod approvals, audit pipeline changes+role assumptions.
  • Phrase: treat the pipeline as a PRODUCTION CONTROL PLANE - it can change prod faster than a human admin.

Chunk 2 - Questions 11-20

11. What are trust boundaries and how do you identify them?

  • A place where data/identity/control/assumptions move between trust levels or ownership. Common: internet->edge, browser->backend, tenant->shared service, app->DB, service->service, cloud account/project, admin->data plane, CI/CD->prod, 3rd-party->internal.
  • Identify by asking: where does authentication change? where are authz decisions made? where does data cross network/process/tenant/account? who owns each component? where does untrusted input become trusted state? where are secrets/tokens presented?
  • Phrase: trust boundaries are where ASSUMPTIONS NEED PROOF - every boundary needs explicit auth, authz, validation, logging, failure behavior.

12. How would you draw a data flow diagram for a security review?

  • Keep simple enough for engineers+PMs to correct in real time. Elements: external entities (user/admin/partner/attacker client), processes (API/worker/auth/scanner/pipeline), data stores (DB/object storage/cache/log store/secrets), data flows (request/token/file/event/query/export), trust boundaries (internet/tenant/account/admin).
  • Process: start w/ user action/workflow -> draw components that receive/transform/store/forward -> label sensitive data+tokens -> mark trust boundaries -> walk for abuse cases+failure modes.
  • Phrase: the DFD is NOT art - it's a shared MAP for finding missing controls + wrong assumptions.

13. What assets would you identify first in a threat model?

  • Ask what the system must protect, what attackers want, what regulators/customers care about. Assets: customer data+PII, credentials/tokens/sessions/keys, authz policy+role assignments, source code+pipeline, financial/billing data, audit logs+findings, availability of critical workflows, AI model weights/prompts/training data, admin+support tools, 3rd-party trust relationships.
  • Prioritize by: sensitivity, business criticality, concentration of access, regulatory scope, blast radius.
  • Phrase: start w/ assets that cause CUSTOMER HARM, business disruption, legal exposure, or privilege escalation if compromised.

14. What actors would you identify in a threat model?

  • Humans + systems + orgs + automated agents. Actors: anonymous user, authenticated customer, tenant admin, employee, support engineer, SRE, SA/workload identity, CI/CD pipeline, 3rd-party integration, cloud managed service, insider, external attacker.
  • For each define: what they authenticate as, what permissions they SHOULD have, what resources reachable, what abuse they may attempt, what logs identify their actions.
  • Phrase: actors aren't only PEOPLE - workloads, pipelines, 3rd-party services often hold the MOST dangerous permissions.

15. How do you distinguish threats from vulnerabilities?

  • Threat = potential harmful event/attacker objective; Vulnerability = weakness making it easier/possible. Examples: read another tenant's invoices (threat) / missing object-level authz (vuln); insider exports PII (threat) / no export approval+anomaly detection (vuln); SSRF steals creds (threat) / metadata reachable + overprivileged role (vuln); SQL injection (threat) / string-concatenated query (vuln).
  • Distinction matters: threats = what can go wrong, vulns = why exposed, controls address the vuln + reduce threat likelihood/impact.
  • Phrase: a good finding connects BOTH - attacker objective + concrete weakness + impact.

16. How do you distinguish vulnerabilities from risks?

  • Vulnerability = weakness; Risk = likelihood + impact + exposure + business context. Example: public S3 policy (vuln) -> customer PII publicly downloadable causing breach+regulatory reporting (risk).
  • Risk depends on: asset sensitivity, exploitability, internet exposure, compensating controls, attacker motivation, business impact, detection+recovery.
  • Phrase: vulnerabilities are TECHNICAL FACTS - risk is the CONSEQUENCE of those facts in the system's real context.

17. How do you convert a threat into an engineering requirement?

  • Make control specific, testable, owned, tied to data flow. Example: threat = user can export another tenant's data; requirement = export service derives tenant ID from authenticated server-side context + applies tenant filter; test = Tenant A can't export Tenant B by modifying params; owner = export team.
  • Qualities: clear owner, specific control, testable acceptance criteria, severity+due date, detection/logging expectation.
  • Phrase: if engineers can't BUILD or TEST it, the threat model hasn't produced an engineering requirement yet.

18. How do you convert a threat into a detection requirement?

  • Ask what observable SIGNAL appears if the threat is attempted/succeeds. Example: threat = enumerate object IDs for cross-tenant data; detection = alert when one principal gets many authz denials across distinct object IDs/tenants in short window; logs = principal/tenant/object-ID-hash/decision/route/status/request-ID.
  • Fields: threat scenario, signal source, required log fields, detection logic, severity, expected FPs, response playbook, owner.
  • Phrase: detections fail when logs lack DECISION CONTEXT - design logging WHILE designing the control.

19. Compare STRIDE, PASTA, attack trees, and MITRE ATT&CK.

  • STRIDE: design reviews+DFDs, simple category coverage, can become checklist-like.
  • PASTA: risk-centric business analysis, connects technical->business impact, heavier process.
  • Attack trees: goal-driven attacker paths, shows chained steps+alternatives, needs threat expertise.
  • MITRE ATT&CK: detection+adversary behavior, maps tactics/techniques to telemetry, less focused on new product design.
  • Use: STRIDE first-pass review, attack trees for crown-jewel abuse paths, PASTA when business prioritization matters, ATT&CK for detection coverage.
  • Phrase: frameworks are LENSES - use the one that helps the team decide better, not the one that fills the most template fields.

20. When is STRIDE useful and when is it insufficient?

  • Useful: team has a DFD + needs systematic coverage; new service reviews; login/API/storage/pipeline flows; teaching teams to reason beyond confidentiality; ensuring logs+integrity aren't forgotten.
  • Insufficient: business abuse cases matter more than technical categories; complex fraud/insider/economic incentives; adversary behavior+detection coverage are the goal; broad ecosystem/supply-chain risk; team treats categories as checkboxes w/o scenarios.
  • Phrase: a good STARTING structure - insufficient when you need attacker goals, business impact, or operational detection depth.

Chunk 3 - Questions 21-30

21. When would you use attack trees?

  • When you need to reason from an attacker GOAL backward into possible paths. Use: account takeover, cross-tenant access, cloud privilege escalation, source-code exfil, DLP/export bypass, signing-key compromise.
  • Structure: root = goal (steal customer data) -> branches (compromise admin / exploit BOLA / abuse export) -> leaves (phish support user / enumerate object IDs / create large export).
  • Why: expose chained attacks, show alternate paths to same impact, prioritize controls that break MULTIPLE paths, easier for engineers than abstract labels.
  • Phrase: use when the question isn't what CATEGORY a threat is, but HOW an attacker would actually get there.

22. When would you use MITRE ATT&CK in design review?

  • When connecting product/infra choices to adversary behaviors + detection coverage. Use: cloud intrusion paths, identity abuse, credential access, persistence, lateral movement, exfiltration, detection engineering.
  • How: identify relevant techniques -> map to logs+detections -> find missing telemetry -> validate prevention+response exist -> avoid forcing irrelevant techniques.
  • Questions: credential access (workload tokens stolen/replayed?), persistence (attacker create keys/backdoor roles?), defense evasion (logging disabled?), exfiltration (data exported without alerting?).
  • Phrase: strongest AFTER architecture is understood - ask whether telemetry+controls cover REALISTIC adversary behavior.

23. How would you prioritize threats?

  • By impact + likelihood + exploitability + blast radius + exposure + control maturity. Factors: asset sensitivity, internet exposure, required attacker capability, ease of exploitation, privilege/data gained, tenants/users affected, existing preventive controls, detection+response, regulatory/customer impact, launch-blocker vs acceptable.
  • Ranking: Critical (plausible path to breach/escalation/safety), High (exploitable w/ meaningful impact), Medium (constrained/unusual conditions), Low (defense-in-depth gap, limited blast radius).
  • Phrase: prioritization helps teams decide what must CHANGE before launch vs what can be tracked w/ explicit residual risk.

24. How would you estimate likelihood and impact?

  • Likelihood = how plausible exploitation is; impact = consequence if it happens. Likelihood inputs: exposure (internet/tenant/insider/3rd-party), attacker prerequisites, known exploit patterns, control complexity+maturity, workflow frequency, existing detections+compensating. Impact inputs: data sensitivity+volume, privilege gained, tenants affected, availability, financial/legal exposure, reputational harm, recovery difficulty.
  • Phrase: argue both with EVIDENCE - avoid fake precision, use clear reasoning + document assumptions.

25. How would you handle low-likelihood but catastrophic threats?

  • Special handling because expected-value scoring hides existential risk. Examples: signing key compromise, KMS admin compromise, cross-tenant isolation break in core SaaS, deletion of all prod backups, CI/CD backdoor deploy.
  • Approach: identify if impact is irreversible/reportable, add strong preventive controls even at low likelihood, add INDEPENDENT detection+recovery, require executive/risk-owner sign-off for residual, test recovery w/ exercises, design blast-radius reduction (key separation/tenant isolation).
  • Phrase: catastrophic risks need guardrails + detection + recovery EVEN when hard to exploit - some failures can't be accepted casually.

26. How would you identify abuse cases?

  • How legitimate features get used maliciously/unintendedly. Process: start from high-value workflows (login/export/sharing/admin/billing/invite/API), ask what malicious user/insider/partner would try, ask how feature scales, ask what UI prevents but API allows, ask what incentives exist (fraud/scraping/spam/exfil), review historical incidents+bug bounty.
  • Examples: export for bulk PII exfil, invite flow for spam, free-tier API for scraping, support tooling to view VIP accounts, OAuth consent for phishing.
  • Phrase: abuse cases often don't violate CODE assumptions - they violate PRODUCT intent.

27. How would you threat-model insider risk?

  • Focus on legitimate access being misused/expanded. Actors: employee, contractor, support engineer, SRE, data scientist, admin, departing employee. Threats: bulk export, support access w/o need, secret/source-code exfil, cloud/prod privilege abuse, policy/log tampering, collusion w/ external attacker.
  • Controls: least-privilege+JIT, approval+ticket binding for sensitive, strong+immutable audit logs, SoD, DLP+anomaly detection, privacy-aware monitoring+investigation process, rapid offboarding+access review.
  • Phrase: NOT solved by trusting employees less - solved by designing access so legitimate work is possible AND misuse is visible, limited, reviewable.

28. How would you threat-model third-party integrations?

  • Introduce identity/data-sharing/availability/supply-chain risk. Questions: what data sent? what permissions received? can they call back into our APIs? how are tokens/webhooks/secrets protected? what if 3rd-party compromised? can one customer abuse to affect another? how to revoke quickly?
  • Controls: least-privilege OAuth scopes, tenant-bound tokens+external IDs, webhook signature validation+replay protection, data minimization+retention limits, separate SA/role per integration, monitor unusual 3rd-party access, contractual+incident-notification requirements.
  • Phrase: a DELEGATED TRUST BOUNDARY - the design must bind caller + tenant + scope + resource.

29. How would you threat-model admin functionality?

  • High risk: often bypasses normal user controls. Threats: admin account takeover, overbroad admin roles, support accessing customer data w/o need, admin action not logged, escalation read-only->write admin, admin APIs missing object/tenant checks, dangerous bulk/irreversible deletes.
  • Controls: strong+phishing-resistant MFA for admins, fine-grained roles, JIT elevation, approval for sensitive actions, break-glass w/ loud alerts, tamper-resistant audit logs, customer-data access justification, dual control for destructive, admin session isolation+short lifetimes.
  • Phrase: design admin tools like PRODUCTION CONTROL PLANES, not internal convenience pages.

30. How would you threat-model APIs?

  • Cover auth + authz + input handling + data exposure + abuse + observability. Auth: token type/issuer/audience/expiration/client identity. Authz: function/object/tenant/field/workflow checks. Input: injection/deserialization/file/URL/parser sinks. Data exposure: response schemas/pagination/exports/hidden fields. Abuse: rate limits/scraping/credential stuffing/expensive queries. Ops: logging/alerting/versioning/deprecation.
  • Flow: request -> authenticate -> authorize -> validate input -> business logic -> scoped response -> audit logs.
  • Phrase: direct API calls are the REAL interface - threat-modeling only the UI misses the paths attackers use.

Chunk 4 - Questions 31-40

31. How would you threat-model data flows across regions?

  • Attention to residency/replication/failover/access/encryption/visibility. Questions: what data crosses regions? regulated/residency-bound? which services auto-replicate? who accesses data per region? regional or global keys? what logs show cross-region transfer? what happens during failover?
  • Threats: PII replicated to unauthorized region, backup/analytics copy violates retention, failover bypasses access controls, logs in one region omit access from another, key in one region unexpectedly decrypts data in another.
  • Controls: classification+regional policy, explicit replication allowlists, regional KMS boundaries, cross-region transfer logging, tested failover, legal+privacy review for residency-sensitive.
  • Phrase: not only AVAILABILITY - it changes WHO can access data, WHERE data legally resides, what EVIDENCE exists during an incident.

32. How would you threat-model encryption key management?

  • Focus on who can create/use/rotate/disable/destroy/administer keys. Assets: key material, KMS policies+grants, encrypted data keys, rotation jobs, audit logs, break-glass access. Threats: broad decrypt, key admin self-grants decrypt, key deletion -> data loss, rotation fails -> breaks prod, key used across unrelated data classes, logs miss decrypt/policy changes, mishandled backup/exported key material.
  • Controls: separate key-admin vs data-decrypt, least-privilege decrypt grants, encryption-context/purpose binding, deletion protection+approval, rotation testing, audit alerts for decrypt spikes+policy changes, per-env+per-data-class keys.
  • Phrase: usually ACCESS-CONTROL risk + AVAILABILITY risk - losing control exposes data, losing keys destroys data.

33. How would you threat-model authorization logic?

  • Model around principal/action/resource/tenant/context/decision-point. Questions: who is the principal? what action? what resource? which tenant owns it? what context (device/time/region/approval)? where enforced? can clients bypass the path?
  • Threats: IDOR/BOLA, function-level bypass, field-level exposure, tenant ID spoofing, cached permission after revocation, admin/support overreach, background job skips authz.
  • Controls: central policy library/PDP, PEPs at API/service/resolver layers, deny by default, mandatory tenant context, authz decision logs, cross-role+cross-tenant tests.
  • Phrase: follow EVERY direct and indirect path to the resource - including jobs, exports, admin tools, APIs.

34. How would you threat-model logging and monitoring?

  • Model as SECURITY CONTROLS w/ their own assets+failure modes. Assets: audit logs, detection rules, alert routing, SIEM pipeline, retention config, sensitive log contents. Threats: missing logs for critical decisions, logs contain secrets/PII, attacker disables logging, logs in compromised account, alert to unmonitored queue, high-FP rule ignored, time-sync issues break timelines.
  • Controls: define required log fields during design, redact secrets+sensitive payloads, centralize outside workload blast radius, alert on logging changes, test detections w/ realistic events, document playbooks+owners, monitor pipeline health.
  • Phrase: a control that cannot be OBSERVED cannot be TRUSTED during an incident.

35. How would you identify missing detection controls?

  • Walk each high-priority threat + ask how we'd know it happened. Process: list top threats+abuse cases -> identify expected attacker actions -> map each to logs -> check logs contain principal/resource/decision/outcome -> define detection logic+threshold -> check alert routing+responder -> test w/ simulated events.
  • Examples: cross-tenant BOLA (no object/tenant ID in logs), insider export (no row count/reason logged), IAM escalation (no alert on policy changes), key misuse (no KMS decrypt anomaly detection).
  • Phrase: missing detection is often missing TELEMETRY, not missing SIEM rules.

36. How would you identify missing prevention controls?

  • Ask what must be true for an attacker path to succeed, check whether a control blocks that condition. Process: build attacker path/attack tree -> mark each required step -> identify current preventive controls -> find steps w/ no control or only detective -> verify controls enforceable (not UI-only) -> add bypass tests.
  • Examples: SSRF reaches metadata (egress allowlist/metadata blocking/IMDSv2/least-privilege role), user reads other tenant's file (server-side tenant check), CI deploys from untrusted branch (protected branches+OIDC claim conditions).
  • Phrase: prevention controls should BREAK attacker paths before impact, not merely record them afterward.

37. How would you identify missing recovery controls?

  • Answer how the org restores trust after a control fails. Questions: revoke tokens/sessions/keys/grants quickly? rotate secrets w/o outage? restore from clean backups? identify affected users/tenants? re-encrypt under new key? rebuild from known-good artifacts? tested playbook?
  • Examples: token theft (revocation+refresh rotation+session invalidation), KMS misuse (key disablement+re-encryption+decrypt review), bucket exposure (policy rollback+object access analysis+customer notification), CI compromise (revoke federation trust+rotate deploy creds+rebuild artifacts).
  • Phrase: recovery is part of DESIGN - if you can't recover, the residual risk is HIGHER than the prevention score suggests.

38. What makes a threat model actionable?

  • Produces concrete decisions + trackable work. Actionable outputs: clear scope+assumptions, prioritized threats w/ impact, specific engineering requirements, detection+logging requirements, owners+due dates, residual risks+approvers, tests/validation steps, follow-up issues tracked to closure.
  • Bad: "Risk: security issue with API." Good: "Export API can return cross-tenant records if tenant ID in body modified; requirement = derive tenant from authenticated context + cross-tenant test before launch; owner = export team."
  • Phrase: not done when threats are LISTED - done when DECISIONS are made and owners can act.

39. What makes a threat model too generic?

  • Could apply to any system + doesn't change engineering decisions. Warning signs: threats are only category labels, no assets named, no trust boundaries, no data flows, no tenant/role/resource context, vague controls ("use encryption"/"validate input"), no severity/owner/acceptance criteria, no connection to logs/tests.
  • Improve: name exact resource+attacker path, tie to a workflow, identify vulnerable assumption, add testable control, assign owner+due date.
  • Phrase: SPECIFICITY makes threat modeling useful - generic findings create the APPEARANCE of rigor without reducing risk.

40. How would you avoid turning threat modeling into a checklist exercise?

  • Center on real design decisions + highest-risk abuse paths. Practices: start with the team's architecture+data flow (not a blank template), ask engineers to explain assumptions+failure modes, use frameworks as prompts not deliverable, focus on pre-launch decisions, write findings in engineering language, prioritize a few meaningful risks, track actions to closure, revisit when architecture changes.
  • Facilitation: worst plausible misuse of this feature? what are we assuming another system enforces? what would we wish we'd logged during an incident? what can't be fixed after launch w/o migration?
  • Phrase: a checklist helps coverage, but CURIOSITY + design-specific reasoning find the real risks.

Chunk 5 - Questions 41-50

41. How would you run a 30-minute threat-modeling session?

  • Tightly scoped + outcome-oriented. Agenda: 0-5min define scope/goal/out-of-scope, 5-10min draw core data flow+trust boundaries, 10-20min identify top threats+abuse cases, 20-25min choose mitigations/open questions, 25-30min assign owners/severity/next steps.
  • Prep: ask for architecture diagram/API contract/data classification/launch timeline; invite engineer+product owner+security reviewer+ops owner; bring prior incidents+known bug classes.
  • Phrase: don't try to model EVERYTHING - find the top risks that can change design or launch readiness.

42. How would you run a threat-modeling session with product managers?

  • Connect security risks to user trust + abuse cases + launch tradeoffs + business impact. Approach: start from user workflows (not security taxonomy), ask what user harm is unacceptable, discuss abuse cases (fraud/spam/scraping/privacy/insider), translate technical->product decisions, clarify launch blockers vs follow-up hardening, discuss customer comms+privacy+compliance.
  • Questions: who should be allowed this action? what would surprise/harm a customer? what data is most sensitive here? what misuse scales fastest? what security friction is acceptable for this journey?
  • Phrase: PMs are essential because many threats are FEATURE-ABUSE or USER-HARM problems, not only technical bugs.

43. How would you challenge an engineering team's assumptions respectfully?

  • Evidence-seeking questions + make risk concrete without blaming. Techniques: restate assumption neutrally, ask what enforces it, ask how it fails, ask what test proves it, offer an attacker-path example, separate design critique from personal critique, acknowledge tradeoffs+constraints.
  • Example: assumption "only UI can call this endpoint" -> question "what prevents a direct API call w/ modified tenant ID?" -> test "add integration test calling as another tenant."
  • Phrase: respectful challenge = CURIOSITY + EVIDENCE - goal is to improve the design, not win an argument.

44. How would you document residual risk?

  • Risk remaining after accepted controls or intentional deferral. Fields: risk statement, affected assets+users, existing controls, missing controls, likelihood+impact, reason for acceptance/deferral, expiration/review date, risk owner+approver, compensating controls, monitoring/detection requirements.
  • Example: "export links valid 24h instead of 1h for enterprise compatibility; compensating = tenant-scoped+user-scoped+logged+revoked on password reset; owner = data platform PM; review next quarter."
  • Phrase: residual risk should be EXPLICIT and OWNED - hidden residual risk is just unmade security debt.

45. How would you track threat-model findings to closure?

  • Track in the same system engineering uses for delivery, w/ security metadata. Fields: title+description, threat scenario+data flow, severity+rationale, required fix/decision, owner team+lead, due date/launch gate, detection+test requirements, status+completion evidence, residual risk if not fully fixed.
  • Closure evidence: code/config change, test results, updated architecture diagram, detection rule/log field added, security review sign-off.
  • Phrase: threat modeling loses credibility if findings disappear into documents - closure tracking makes it part of engineering EXECUTION.

46. What metrics would you use for a threat-modeling program?

  • Measure coverage + quality + actionability + risk reduction, NOT session count. Useful: % high-risk launches threat-modeled before release, # high/critical findings found before launch, findings closed by SLA, % findings w/ tests/detections, repeat finding rate by category, time from review to actionable requirements, coverage of crown-jewel systems, # incidents linked to missed design risks, engineer satisfaction.
  • Bad metrics alone: raw # threat models, # pages written, # STRIDE boxes filled.
  • Phrase: measure whether threat modeling CHANGES designs and REDUCES incidents, not whether people attended meetings.

47. What are common threat-modeling mistakes?

  • Starting too late (architecture fixed), generic checklist w/o system context, ignoring authz+tenant isolation, forgetting operational controls (logging/recovery), treating encryption as universal fix, not including product/ops/data owners, not tracking findings to closure, over-focusing on exotic threats while missing common ones, failing to document assumptions, producing findings engineers can't test.
  • Phrase: the biggest mistake is mistaking DOCUMENTATION for risk reduction - the output must INFLUENCE the system.

48. What is an example of a good threat-model finding?

  • Specific + evidence-based + risk-ranked + actionable. Example: "Export API trusts tenantId from request body; threat = authenticated user modifies tenantId to export another tenant's records; impact = cross-tenant PII disclosure; requirement = derive tenant from authenticated session + enforce tenant filter; validation = integration test Tenant A can't export Tenant B; detection = log tenant ID/requester/row count/authz decision; severity = critical before launch; owner = export team."
  • Why good: names data flow, explains attacker action, states impact, gives fix+test+detection+owner+severity.
  • Phrase: a good finding tells the team EXACTLY what to change and HOW to know it's fixed.

49. What is an example of a weak threat-model finding?

  • Vague + unactionable + disconnected from design. Weak: "API security risk; recommendation: validate input and use encryption; severity: medium." Why weak: no asset named, no attacker path, no endpoint/data flow, no testable requirement, no owner, no severity reason, encryption may not address it.
  • Improved: "profile update endpoint accepts role field from client JSON; threat = mass assignment sets role=admin; requirement = allowlist writable fields + reject role/tenantId/emailVerified."
  • Phrase: weak findings create AMBIGUITY - ambiguity slows remediation and lets real risk survive.

50. What questions would you ask before approving a high-risk design?

  • (1) What assets/users harmed if design fails? (2) Main trust boundaries + who controls each side? (3) Top 3 abuse cases/attacker paths? (4) Which controls prevent those paths before impact? (5) Which logs/detections prove attempted/successful abuse? (6) Blast radius of one compromised user/tenant/workload/admin? (7) What secrets/keys/tokens/privileged identities involved? (8) What happens when a dependency/scanner/IdP/policy engine fails? (9) What residual risks remain + who accepts them? (10) What tests/runbooks/rollback plans exist before launch?
  • Phrase: approve a high-risk design ONLY when the team can explain the risk, demonstrate controls, show detection+recovery paths, and document residual risk with the right owner.
