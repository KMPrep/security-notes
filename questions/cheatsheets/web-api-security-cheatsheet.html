Source: questions/answers/web-api-security-answers.md
Purpose: Rapid-recall key points per question. Skim for quick recall; each bullet is a must-remember hook, not a full answer.

Chunk 1 - Questions 1-10

1. Explain SSRF using a cloud metadata compromise example.

  • SSRF = attacker makes server fetch attacker-chosen destination; server becomes a proxy into networks/services attacker can't reach. Classic: image-import URL feature on EC2.
  • Attack: submit http://169.254.169.254/latest/meta-data/iam/security-credentials/app-role -> app fetches -> metadata returns temp AWS creds -> attacker uses creds against AWS APIs.
  • Impact scales w/ workload role: if role reads S3/decrypts KMS/assumes roles -> SSRF becomes cloud account compromise.
  • Phrase: dangerous because it converts a simple URL-fetch bug into credential theft + internal service access + cloud control-plane abuse.

2. How would you prevent SSRF in a cloud application?

  • App validation + network egress controls + metadata hardening + least-privilege identity. Controls: avoid arbitrary fetches, allowlist exact schemes+destinations, resolve DNS+validate final IP before request, re-validate after redirects + block internal redirects, block loopback/link-local/RFC1918/multicast/metadata ranges, disable file/gopher/ftp, egress proxy enforcing policy, isolate URL-fetch workers (no internal access), IMDSv2 + GCP metadata headers + least-privilege SAs, monitor unusual outbound.
  • Phrase: don't rely on a single regex - control the destination at app + DNS + network + cloud identity layers.

3. What is the difference between allowlisting and denylisting for SSRF defense?

  • Allowlist = what's permitted (safer, narrow, auditable, requires knowing legit destinations); Denylist = what's forbidden (easier for broad import, MANY bypasses).
  • Denylist bypasses: 127.1, IPv6 loopback, decimal/hex IPs, DNS rebinding, redirect from allowed domain to metadata, CNAME to private ranges.
  • Phrase: use ALLOWLISTS for high-risk server-side fetches - denylists are defense-in-depth, not the primary boundary.

4. How can URL parsing bugs lead to SSRF bypass?

  • Validation code + request code interpret the same URL DIFFERENTLY. Examples: https://trusted.com@169.254.169.254/, https://trusted.com.evil.com/, http://2130706433/ (decimal), http://[::ffff:169.254.169.254]/, redirect param to metadata.
  • Mistakes: string-prefix check vs parsed hostname, contains("trusted.com") vs exact origin, validate before redirects but not after, resolve DNS twice (TOCTOU), inconsistent decoding/punycode.
  • Controls: single well-tested parser, canonicalize before validation, validate scheme/host/port/resolved-IP, re-check every redirect, pin DNS or route through policy proxy.
  • Phrase: validation must match the EXACT request that will be sent - any parser/fetcher gap becomes a bypass surface.

5. How would you secure access to cloud metadata services?

  • Treat as CREDENTIAL services, not harmless local endpoints. AWS: require IMDSv2, hop limit 1, disable where unneeded, least-privilege instance profiles, monitor role creds from unusual locations, block from containers that don't need it. GCP: require Metadata-Flavor: Google header, limit SA scopes+IAM, disable legacy endpoints, dedicated SA per workload, monitor token use+anomalies.
  • Design principle: if SSRF succeeds, metadata should STILL be hard to reach and low-value if reached.
  • Phrase: layered protection - token/header requirements + network isolation + least-privilege identity + detection of credential misuse.

6. What is XSS and what are the main types?

  • Injection where attacker script executes in victim's browser under a trusted origin, inheriting same-origin privileges. Types: Stored (payload stored, shown to later users - e.g. malicious comment), Reflected (echoed in immediate response - e.g. search query), DOM (client JS reads input + writes to dangerous sink - e.g. location.hash -> innerHTML).
  • Impact: session theft (if not HttpOnly), actions as victim, DOM data exfil, in-origin phishing, CSRF-token theft, stored worm propagation.
  • Phrase: XSS is not just alert(1) - it's arbitrary script execution in a TRUSTED browser context.

7. How would you prevent stored XSS?

  • Safe input handling + output encoding + sanitization for rich content + defense-in-depth. Controls: contextual output encoding (HTML text/attribute/URL/JS/CSS), framework auto-escaping (avoid bypass APIs), allowlist sanitizer for rich HTML, store content as DATA not pre-rendered HTML, reject active SVG, CSP, HttpOnly+Secure cookies, review Markdown/WYSIWYG pipelines, test through every rendering surface (email/admin consoles).
  • Phrase: PRIMARY control = contextual output encoding - sanitization only when product intentionally allows user-supplied HTML.

8. How would you prevent reflected XSS?

  • Output encoding + safe templates. Controls: server-side templates w/ auto-escaping, encode every reflected value for its context, avoid manual HTML strings, validate type+length (but don't rely on it alone), JSON w/ correct Content-Type for APIs, avoid reflecting raw error/debug values, CSP as defense-in-depth, test search/error/redirect/profile pages.
  • Phrase: fixed at the RENDER boundary - the server must never confuse user input with markup/script.

9. How would you prevent DOM-based XSS?

  • Safe client-side coding (server may never see payload). Controls: avoid dangerous sinks (innerHTML/outerHTML/document.write/eval/string timers), prefer textContent+safe DOM APIs+framework binding, validate+parse URL params, proven client sanitizers only when HTML required, Trusted Types, lock down postMessage w/ strict origin+schema, avoid javascript: URLs, CSP to limit script sources.
  • Phrase: a SOURCE-TO-SINK problem - trace attacker-controlled sources to executable sinks and REMOVE the unsafe sink.

10. How does Content Security Policy reduce XSS impact?

  • Tells browser which scripts/styles/images/frames/connections are allowed - defense-in-depth, NOT a replacement for output encoding. Protections: restrict script sources, block inline scripts unless nonce/hash, restrict connect-src to approved APIs, frame-ancestors against clickjacking, block plugins/objects, report violations.
  • Example: default-src 'self'; script-src 'self' 'nonce-random'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'.
  • Phrase: turns many XSS bugs from code execution into BLOCKED browser violations - but only if the policy avoids unsafe shortcuts.

Chunk 2 - Questions 11-20

11. What are common CSP mistakes?

  • script-src *, 'unsafe-inline' for scripts, 'unsafe-eval' w/o justification, trusting broad CDNs hosting user content, missing object-src 'none', missing base-uri 'self' (base tag abuse), missing frame-ancestors (clickjacking), only Report-Only never enforced, static/no nonces, ignoring violation reports, forgetting connect-src (exfil).
  • Safer rollout: start report-only, remove inline scripts/migrate to nonces+hashes, exact origins, enforce high-risk pages first, monitor+tighten iteratively.
  • Phrase: CSP is easy to make DECORATIVE - a strong policy removes inline script execution + uses narrow source lists.

12. What is CSRF and when is an application vulnerable?

  • Tricks an authenticated user's browser into sending a state-changing request; browser sends cookie automatically (no cookie theft needed). Vulnerable when: cookie-based auth, state-changing actions w/o CSRF token, session cookies sent cross-site, endpoint accepts cross-site forms, server trusts cookie presence as proof of intent.
  • Examples: change email, add payment method, transfer funds, grant OAuth consent, create admin user.
  • Phrase: a USER-INTENT problem - authentication proves who the browser is logged in as, NOT that the user MEANT to perform the action.

13. How does SameSite cookie behavior reduce CSRF risk?

  • Controls whether browser sends cookie on cross-site requests; CSRF relies on automatic sending. Behavior: Strict (not sent on cross-site nav/subrequests), Lax (sent on same-site + safe top-level cross-site nav, NOT most cross-site POST), None (sent cross-site, requires Secure).
  • Helps: hidden cross-site POST forms lose the cookie under Lax/Strict -> server sees no session -> attack fails. Limitations: not a substitute for CSRF tokens on critical changes, some flows need cross-site cookies, XSS bypasses CSRF (runs same-origin).
  • Phrase: moves a major CSRF control into the BROWSER, but high-risk ops should still require anti-CSRF tokens or re-auth.

14. Compare SameSite Strict, Lax, and None.

  • Strict: not sent cross-site; good for high-risk session/admin cookies; can break login redirects+shared links. Lax: limited safe top-level nav; general session cookies; some top-level GET still includes cookies. None: sent cross-site (Secure required); embedded widgets/iframes/federation; CSRF exposure if no other protection.
  • Guidance: Lax practical default, Strict for sensitive admin/security actions, None;Secure only when truly required (pair w/ CSRF tokens+origin checks+careful CORS).
  • Phrase: not one-size-fits-all - pick based on whether cross-site cookie sending is REQUIRED by the product flow.

15. What makes a cookie secure?

  • Narrow scope + HTTPS-only + JS-inaccessible (session) + CSRF-protected + server-side session controls. Controls: Secure, HttpOnly for session/auth, SameSite Lax/Strict, omit Domain (host-only), deliberate Path, short lifetimes + server-side expiration, rotate session ID after login+elevation, __Host- prefix, avoid localStorage when HttpOnly viable, pair w/ CSRF+XSS defenses.
  • Example: Set-Cookie: __Host-session=abc; Secure; HttpOnly; Path=/; SameSite=Lax; Max-Age=3600.
  • Phrase: cookie flags are necessary but session security also needs server-side invalidation + rotation + anomaly detection + good auth design.

16. Explain HttpOnly, Secure, SameSite, Path, and Domain cookie attributes.

  • HttpOnly (hides from JS -> reduces XSS theft), Secure (HTTPS-only -> prevents plaintext exposure), SameSite (cross-site sending -> reduces CSRF), Path (limits URL path -> mostly routing not isolation), Domain (host/subdomain scope -> overbroad exposes to subdomains).
  • Nuances: HttpOnly doesn't stop XSS making same-origin requests as user, Secure useless if tokens leak to logs/JS, Path doesn't stop same-origin scripts reading non-HttpOnly cookies, omitting Domain = safer host-only, SameSite=None requires Secure.
  • Phrase: attributes define browser behavior - mis-scoping a session cookie can turn a SUBDOMAIN compromise into ACCOUNT compromise.

17. What is CORS?

  • Browser mechanism letting a server declare which other origins may READ its responses - a controlled relaxation of Same-Origin Policy. Key: enforced by BROWSERS; server sends headers, browser decides if JS can read response.
  • Headers: Origin, Access-Control-Allow-Origin, Access-Control-Allow-Credentials, Access-Control-Allow-Methods/Headers, Vary: Origin.
  • Phrase: CORS does NOT authenticate users and does NOT stop the server from receiving requests - it controls whether browser JS can READ cross-origin responses.

18. How can CORS misconfiguration expose sensitive data?

  • Allows attacker-controlled site to READ authenticated API responses from victim's browser. Dangerous: Access-Control-Allow-Origin: https://evil.example + Allow-Credentials: true -> attacker JS reads victim's data.
  • Misconfigs: reflecting any Origin, broad regex *.example.com w/ takeover-prone subdomains, allowing null origin, missing Vary: Origin w/ CDN caching, credentials for wildcard-like policies, CORS on internal admin APIs.
  • Phrase: CORS bugs turn the victim's browser into an authenticated DATA RETRIEVAL TOOL for the attacker.

19. Why is wildcard CORS dangerous with credentials?

  • Browsers block ACAO: * + credentials, but teams recreate danger by REFLECTING arbitrary origins + Allow-Credentials: true. Credentials matter: cookies/HTTP-auth/client-certs/bearer may be sent, making authenticated responses readable to attacker JS.
  • Bad: echo any Origin + credentials true. Safe: only exact approved origins, credentials only where required, Vary: Origin for dynamic responses.
  • Phrase: the danger isn't only literal * - it's ANY policy that lets arbitrary origins read credentialed responses.

20. How would you review a CORS policy?

  • Identify which origins need access, whether credentials required, whether server authorizes independent of CORS. Checklist: list every allowed origin (scheme/host/port), exact string match (not regex/substring), credentials truly required?, no wildcard for sensitive APIs, Vary: Origin when dynamic, reject null, limit methods+headers, no CORS on internal services, server-side authz still runs every request, audit subdomains for takeover.
  • Phrase: CORS review is ORIGIN TRUST review - every allowed origin can effectively read selected browser-accessible API responses.

Chunk 3 - Questions 21-30

21. What is SQL injection?

  • User input interpreted as SQL CODE not data, changing query structure/meaning. Vulnerable: "SELECT * FROM users WHERE name = '" + input + "'"; input admin' -- comments out the rest.
  • Impacts: auth bypass, data exfil, modification/deletion, schema discovery, OS command exec (some DBs), DoS (expensive queries/sleeps).
  • Phrase: NOT a database bug - an application bug where code mixes untrusted data with executable query syntax.

22. How do prepared statements prevent SQL injection?

  • Separate SQL STRUCTURE from parameter VALUES; DB parses template first, then binds input as data. Safe: WHERE email = ? + bind. Works because: input can't add keywords/operators/comments, quotes treated as characters, plan+binding distinct.
  • Caveats: DON'T make dynamic table/column names safe, ORDER BY fields need allowlists, ORM raw query methods reintroduce injection, stored procedures vulnerable if they build dynamic SQL.
  • Phrase: parameterization is the PRIMARY fix - escaping is fragile and shouldn't be the main defense.

23. What is second-order SQL injection?

  • Malicious input stored SAFELY first, then later retrieved + used unsafely in a new query. Example: register admin'-- -> stored as data -> admin workflow builds SQL using stored username -> payload executes later.
  • Tricky: initial validation appears to work, payload dormant for days/months, tests checking only immediate response miss it, devs wrongly trust data from their own DB.
  • Controls: treat stored data as untrusted when reusing in queries, parameterize EVERY query (incl internal workflows+migrations), avoid dynamic SQL for stored values, test admin/reporting/batch jobs.
  • Phrase: once user input enters your system, it does NOT become trusted merely because it lives in your database.

24. What is NoSQL injection?

  • User input changes query meaning to a non-SQL datastore (MongoDB/Elasticsearch/Redis-like/document DBs). Examples: JSON operators $ne/$gt into Mongo, query DSL into Elasticsearch, object where scalar expected, modifying aggregation stages.
  • Risk: login {"user":{"$ne":null}, "password":{"$ne":null}} bypasses auth. Controls: validate schema+types before query, reject unexpected operators in user JSON, safe query builders+binding, allowlists for searchable fields+sort keys, least privilege at datastore, limit result size+complexity.
  • Phrase: still INJECTION - syntax changed, root cause is untrusted input controlling query SEMANTICS.

25. How would you prevent command injection?

  • User input reaching an OS shell/interpreter. Controls: avoid shelling out when a library API exists, don't pass input to sh/cmd/bash/powershell, use argument ARRAYS not command strings, strict allowlists, escape only as last line + per platform, least privilege, sandbox/containers/seccomp/AppArmor for risky processors, timeouts+resource limits, log attempts (not secrets).
  • Bad: "convert " + filename + " output.png". Better: execute convert w/ validated arg array, no shell.
  • Phrase: safest fix is REMOVING the shell boundary - treat shell execution as a high-risk trust boundary.

26. What is path traversal?

  • Attacker path input escapes intended directory to access files elsewhere. Examples: ../../../../etc/passwd, uploads/../../secrets.env. Impacts: read secrets/config, access other users' files, overwrite app files, unsafe rendering, escalate to code exec via uploads/templates.
  • Controls: no raw user paths, map user IDs to server-side file IDs, canonicalize+verify path stays within root, reject separators where filenames expected, object storage keys w/ controlled prefixes, authorize the OBJECT not the path.
  • Phrase: an AUTHORIZATION + filesystem boundary bug - the path must never decide access by itself.

27. How would you safely handle file paths?

  • Translate user intent into a server-controlled object reference. Design: store files w/ generated IDs (not user names), original filename as metadata only, normalize paths, join w/ platform-safe APIs, resolve canonical path + verify under base dir, reject absolute paths/drive letters/separators/null bytes/traversal, enforce object-level authz before read/write, separate buckets/prefixes per tenant.
  • Phrase: users should select OBJECTS, not filesystem paths - the server owns path construction.

28. Threat-model a file upload feature.

  • Model uploader/upload-service/storage/processors/renderers/download-path/consumers. Assets: user files, storage bucket, malware pipeline, image/doc processors, metadata DB, download URLs+tokens.
  • Threats: malware, stored XSS via SVG/HTML/PDF, path traversal in filename, zip/decompression bombs, parser exploits, executable served from trusted domain, cross-tenant access, public-link exposure, DoS via huge/many files.
  • Flow: uploader -> upload API -> quarantine -> scanner -> clean? -> release / block+alert.
  • Phrase: not a single endpoint risk - a PIPELINE risk across storage, parsing, scanning, rendering, download authorization.

29. What controls would you apply to uploaded files?

  • Authenticate uploader + tenant quota, validate size/type/extension/magic-bytes, quarantine first, rename to generated IDs, strip dangerous metadata, scan for malware, safe+isolated parsers, limit archive depth+decompressed size, safe Content-Type+Content-Disposition on download, store outside web root, object-level authz on download, log upload/scan/release/download/delete.
  • Storage policy: quarantine (scanner only), clean release (authorized users), malicious (security team only, no rendering).
  • Phrase: most important pattern = QUARANTINE THEN RELEASE - don't immediately serve user bytes from a trusted origin.

30. How would you prevent malware upload and unsafe file rendering?

  • Isolation, because scanners miss novel payloads + parsers have vulns. Controls: multi-engine scanning where justified, file-type allowlists (not extension blocklists), quarantine until scan completes, sandboxed conversion/thumbnail workers, patch parsers+image libs fast, disable macros/active content, serve untrusted files from separate origin, force Content-Disposition: attachment for risky types, X-Content-Type-Options: nosniff, convert risky formats to safer for preview, limit size+expansion+processing time.
  • Phrase: ASSUME some malicious files pass scanning - isolation + safe rendering reduce blast radius when detection fails.

Chunk 4 - Questions 31-40

31. What is IDOR?

  • App exposes an object ID + fails to verify the authenticated user can access THAT object. Example: GET /invoice/1002 returns another user's invoice. Root cause: auth exists, object-level authz missing/incomplete, server trusts client-supplied object ID.
  • Controls: check principal+action+object+tenant+relationship on every access, don't rely on unguessable IDs alone, scope queries by tenant+owner, test horizontal+vertical authz, log denied+unusual access.
  • Phrase: NOT fixed by UUIDs - UUIDs reduce enumeration but do NOT prove authorization.

32. What is BOLA in API security?

  • Broken Object Level Authorization = API-focused IDOR; common in APIs because clients request objects by ID directly. Example: GET /api/accounts/acct_123/statement - server must verify caller can read acct_123, not just have a valid token.
  • Surfaces: REST object endpoints, GraphQL resolvers, bulk export, mobile backends, nested resources /users/{u}/devices/{d}, admin tooling.
  • Phrase: usually the HIGHEST-RISK API bug - one missing check can expose MANY records through automation.

33. How would you detect BOLA in API logs?

  • Look for unusual object access patterns + authz failures. Signals: one user accesses many sequential IDs, high 403/404 then success, access across many tenants/accounts/regions, objects outside normal relationship graph, enumerated/close IDs, undocumented object endpoints, spike in response size/record count.
  • Required log fields: actor ID, tenant ID, object ID (or hash), action, decision, status code, client ID, route template.
  • Phrase: detection helps catch missed checks + active enumeration, but BOLA must be PREVENTED in authorization logic.

34. How would you prevent object-level authorization bugs?

  • Make object authz a standard service-layer PATTERN, not per-controller invention. Controls: central authz library/policy engine, required checks at service/resolver layer (not just gateway), data methods require principal+tenant context, query filters include tenant+ownership, deny by default when ownership unknown, cross-user/cross-tenant/privilege-boundary tests, code-review checklist for endpoints taking object IDs, decision logs w/ policy version.
  • Bad: db.getObject(request.objectId). Better: db.getObjectScoped(principal, tenant, objectId) + authorize(principal, read, object).
  • Phrase: object authz should be HARD TO FORGET - put it in the path developers naturally use.

35. What is mass assignment?

  • API binds client-supplied fields directly to a server object, letting client set fields they shouldn't control. Example: {"displayName":"Alice", "isAdmin":true} -> attacker grants self admin.
  • Dangerous fields: role, isAdmin, tenantId, ownerId, price, creditLimit, accountStatus, emailVerified.
  • Phrase: an INPUT AUTHORIZATION bug - the client should NOT decide which fields are writable.

36. How would you prevent mass assignment in APIs?

  • Explicit request DTOs/schemas per operation, allowlist writable fields per endpoint+role, ignore/reject unknown fields, keep server-managed fields separate from client-editable, validate tenant+owner from authenticated context (not body), tests attempting privileged fields, review ORM auto-binding defaults.
  • Safe update: allow displayName/avatarUrl; reject role/tenantId/emailVerified.
  • Phrase: NEVER bind request JSON directly onto persistence models - use operation-specific input models.

37. How would you design API authentication?

  • Identify caller, support machine+user clients, short-lived creds, revocation+audit. Patterns: browser SPA (OIDC + session/short token via BFF), mobile (OIDC+PKCE + device posture), server-to-server (client credentials/mTLS/workload identity), 3rd-party (scoped OAuth app/signed requests/key rotation), internal (mTLS+workload identity).
  • Controls: TLS everywhere, short-lived tokens+refresh controls, scope by audience+permissions, validate issuer/audience/signature/expiration/kid, avoid API keys as sole auth for sensitive APIs, secrets manager, log client-ID/subject/token-type/result.
  • Phrase: answer WHO is calling and WHICH client/workload - a bearer token with no audience/scope discipline is fragile.

38. How would you design API authorization?

  • Decide whether authenticated caller can perform a specific action on a specific resource. Design: explicit actions+resources, object-level authz in service layer, function-level for admin/privileged, field-level for sensitive response fields, scope by tenant, policy engine/shared library, log permit+deny, test direct API calls (not just UI).
  • Flow: request -> authenticate -> load resource context -> authorize action -> allowed? -> scoped response / deny+log.
  • Phrase: NOT one check - includes function + object + tenant + field + workflow authorization.

39. What is API rate limiting and how would you implement it?

  • Controls request/resource consumption over time; protects availability+cost+business logic. Dimensions: per IP/user/API-key/tenant/endpoint/operation-cost/token. Algorithms: token bucket (burst+sustained), leaky bucket (smoothing), fixed window (simple), sliding window (fairer), cost-based (expensive ops).
  • Implementation: edge for coarse limits, app layer for user/tenant/business-context, useful 429 w/ retry guidance, exempt carefully + expire exceptions, monitor abuse+FPs.
  • Phrase: should protect SCARCE RESOURCES + sensitive workflows, not just count HTTP requests.

40. How would you handle abusive API clients without blocking legitimate users?

  • Graduated, context-aware controls (not one global block). Actions: identify by user/tenant/token/IP/device/client-ID, risk-based throttles by endpoint sensitivity, soft limits before hard blocks, CAPTCHA/step-up for human flows, reduce response detail or disable expensive ops temporarily, quarantine suspicious keys+notify owners, self-service dashboards for high-volume clients, allowlisted partner quotas, appeal paths for FPs.
  • Examples: credential stuffing (per-account+per-IP throttle+bot detection+MFA), scraping (cost-based limits+pagination caps), expensive queries (complexity limits+timeouts), token misuse (revoke+rotate).
  • Phrase: good abuse handling preserves legitimate traffic while increasing FRICTION for risky behavior.

Chunk 5 - Questions 41-50

41. What logs would you collect for API security monitoring?

  • Support auth + authz + abuse detection + IR + privacy-safe forensics. Fields: timestamp+request-ID, route template (not raw URL), method+status, authenticated subject+client-ID, tenant ID, source IP+proxy chain, user agent+device, authz decision+policy version, object ID (or hash), response size+latency, rate-limit decision, error category, token issuer+audience (not raw token).
  • DON'T log: passwords, session cookies, Authorization headers, raw tokens, full PII payloads, payment data.
  • Phrase: explain WHO did WHAT to WHICH resource and whether allowed - without becoming a new sensitive-data leak.

42. What are common GraphQL security risks?

  • Flexible queries create specific risks: deeply nested query DoS, expensive queries+resolver fan-out, batching attacks bypassing per-request limits, introspection exposing schema+hidden fields, resolver-level BOLA, excessive data exposure, N+1 amplification, missing authz on nested objects, error messages revealing internals.
  • Controls: depth+complexity limits, persisted queries, field-level authz, resolver-level object checks, rate limits by operation COST, restrict introspection in prod, timeout+pagination requirements.
  • Phrase: mostly AUTHORIZATION + resource-control discipline applied at field + resolver granularity.

43. How would you prevent deeply nested GraphQL query abuse?

  • Measure query COST before execution. Controls: max depth, max total complexity score, max list size + mandatory pagination, resolver timeouts, query allowlisting/persisted queries, limit batching, rate limit by query cost per user/token, cache safe repeated queries, monitor slow resolvers+backend fan-out.
  • Flow: query -> parse -> depth+cost analysis -> within budget? -> execute / reject.
  • Phrase: request COUNT is the wrong unit for GraphQL abuse - complexity + resolver cost are the units that matter.

44. How would you prevent excessive data exposure in APIs?

  • Happens when APIs return fields the caller doesn't need/shouldn't see, even if UI ignores them. Controls: response schemas per operation, DTOs (not DB models), field-level authz, redact sensitive fields by default, avoid includeAll=true/debug fields, pagination+size limits, review mobile+GraphQL for hidden fields, test w/ lower-privilege users.
  • Should NOT leak: password hashes, internal risk scores, admin notes, billing provider IDs, unneeded tenant IDs, feature flags exposing launches.
  • Phrase: the API RESPONSE is the security boundary, not the UI - if the API returns it, an attacker can read it.

45. What is request smuggling?

  • Exploits inconsistencies between front-end+back-end HTTP parsers; crafted request the proxy+backend disagree about, causing hidden/prefixed requests. Root causes: conflicting Content-Length vs Transfer-Encoding, different chunked handling, proxy normalizes one way + backend parses another, HTTP/2->HTTP/1 conversion.
  • Impact: bypass front-end controls, poison another user's request, access internal endpoints, cache poisoning, session hijacking. Controls: well-maintained proxies+backends, normalize+reject ambiguous requests at edge, avoid unsupported downgrade paths, patch parsers, targeted smuggling tests.
  • Phrase: a PARSER DIFFERENTIAL bug across trust boundaries - fix = strict edge normalization + consistent protocol handling.

46. How would you investigate suspected web exploitation?

  • Preserve evidence + scope affected users/endpoints + identify exploit path + contain active abuse. Steps: identify signal (WAF/error spike/suspicious pattern/customer report/EDR) -> preserve LB/WAF/app/gateway/auth/DB logs -> timeline by request-ID/IP/user/route/object -> look for payloads (SSRF URLs/XSS/SQL/traversal/unusual GraphQL) -> determine if exploit SUCCEEDED (not just attempted) -> scope data accessed/modified -> contain (disable feature/block pattern/rotate secrets/patch) -> add tests+detections.
  • Evidence: request/response metadata, app errors+stack traces, auth+session events, downstream cloud API calls (SSRF), DB query logs.
  • Phrase: attempted exploitation is common noise - the investigation must prove IMPACT and blast radius.

47. How would you design security testing for web applications?

  • Combine automated + manual + threat modeling + production feedback. Layers: secure code review (auth/authz/injection/deserialization), SAST (dangerous APIs/framework misuse), DAST (web-facing), dependency+container scanning, API schema testing (auth+field exposure), fuzzing (parsers/uploads/request boundaries), authz tests per role+tenant boundary, browser tests (XSS/CSP/CORS/CSRF/cookies), abuse testing (rate limits/business logic), bug bounty/red-team for high-risk apps.
  • Phrase: must include BUSINESS LOGIC + authorization - scanners find some bugs but rarely understand tenant ownership or workflow abuse.

48. What is the role of WAF and what are its limitations?

  • Filters HTTP for known attack patterns; virtual patching + abuse controls; defense-in-depth, NOT the primary model. Useful: block known payloads, rate limiting+bot controls, emergency mitigation before code fix, protect legacy temporarily, log suspicious traffic.
  • Limitations: can't reliably fix broken authz, bypassed by encoding/parser differences, may not understand GraphQL/custom logic, false positives, doesn't replace secure coding/patching/tests, can't protect paths it doesn't see.
  • Phrase: WAF is a SEATBELT, not steering - you still need to fix the vulnerable code and design.

49. What are common web security anti-patterns?

  • Trusting the UI to enforce permissions, string-concatenated SQL/shell, denylist-only SSRF, returning full DB models from APIs, tokens in localStorage when HttpOnly viable, CORS reflecting arbitrary origins, uploads served from trusted origins, UUIDs instead of object authz, no rate limits on login/search/export/GraphQL, public debug/Swagger endpoints, treating WAF findings as proof of security, missing authz decision logs.
  • Phrase: most anti-patterns come from trusting the WRONG layer - browser, gateway, UI, filename, object ID, or WAF.

50. What questions would you ask in a web/API security design review?

  • (1) Trust boundaries between browser/API/services/data stores? (2) How is auth done + how are tokens/cookies protected? (3) Where is authz enforced for functions/objects/tenants/fields? (4) Which inputs reach interpreters (SQL/shell/templates/URLs/file parsers)? (5) Does any feature fetch attacker URLs or process attacker files? (6) What CORS/CSP/CSRF/cookie policies apply? (7) How are rate limits+abuse controls designed by user/tenant/IP/client? (8) What logs prove allowed+denied decisions? (9) What tests cover direct API calls + cross-tenant access? (10) What happens when WAF/proxy/scanner/IdP fails?
  • Phrase: trace attacker-controlled data from entry point to SINK, and trace authenticated identity from login to every AUTHORIZATION decision.
