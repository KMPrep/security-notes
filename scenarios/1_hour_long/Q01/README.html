Q01 — Centralized Security Logging Platform for a 200-Account AWS Organization

The Ask

"Design a centralized, tamper-evident security logging platform for an AWS Organization with 200 member accounts. It must ingest CloudTrail, VPC Flow Logs, DNS query logs, Config snapshots, and application logs. Security analysts need near-real-time search and a 7-year retention requirement for compliance."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Are all 200 accounts already in a single AWS Organization, and are SCPs actively enforced on the OU structure?
    Impact: If yes, I can rely on organization-wide SCPs to prevent log tampering and use organization-trail features. If not, I need a per-account enrollment pipeline and cannot rely on centralized guardrails yet.

  • Q: Is there an existing AWS Control Tower deployment with a designated Log Archive account and Security account, or are we building that structure from scratch?
    Impact: Control Tower gives me pre-built Log Archive and Audit accounts, an SCP guardrail library, and Account Factory. Building from scratch means I need to design that account structure myself and adds 4–6 weeks of baseline work.

  • Q: What is the expected daily log volume in GB across all 200 accounts?
    Impact: Below ~1 TB/day I can use a single Kinesis Data Firehose per log type. Above that I need multiple shards, partitioned S3 prefixes by account-id/region/date, and potentially Kinesis Data Streams in front of Firehose for fan-out buffering. Also determines OpenSearch cluster sizing and Firehose buffer tuning.

  • Q: Are there data residency or sovereignty constraints — for example, EU accounts whose logs cannot leave the EU?
    Impact: If yes, I need regional Log Archive sub-accounts per data residency zone, regional KMS CMKs, and S3 Cross-Region Replication only within the permitted regions. This significantly complicates the single-pane-of-glass search story.

  • Q: Does the 7-year compliance requirement mandate WORM immutability (cannot delete or overwrite), or is it a soft retention policy (delete allowed but must audit who did it)?
    Impact: True WORM requires S3 Object Lock in Compliance mode, which means even the account root cannot delete objects during the retention period. Governance mode gives delete flexibility with an audit trail. Compliance mode is harder to reverse if you misconfigure the retention period — ask the legal/compliance team before committing.

  • Q: Who are the security analysts and what search tooling do they expect — a SIEM they already use (Splunk, Sentinel), or are we building a cloud-native search experience?
    Impact: If there is an existing SIEM, I add a Kinesis Data Firehose → HTTPS endpoint or S3 export to feed it, and OpenSearch becomes secondary. If cloud-native, OpenSearch Service in the Security account becomes the primary analyst UI, with Athena for ad-hoc SQL queries on cold data.

  • Q: Is there a requirement for cross-account alerting from GuardDuty and Security Hub findings, or just log storage?
    Impact: If yes, I add GuardDuty organization-wide delegation to the Security account and EventBridge cross-account rules. This is architecturally separate from the logging pipeline but often conflated.

  • Q: What is the RTO/RPO if the Log Archive S3 bucket becomes unavailable? Do analysts need continued write (log delivery) or read (query) access during an outage?
    Impact: If write continuity is required (log delivery must not drop), I need Kinesis Data Firehose with a backup S3 bucket in a second region. If read continuity is required, S3 Cross-Region Replication with active CRR keeps a queryable replica.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. AWS Control Tower is deployed with three dedicated foundational accounts: Management, Log Archive, and Security (Audit). All 200 member accounts sit under Service or Workload OUs.
  2. Log volume is estimated at 500 GB/day across all accounts — within a single Kinesis Data Firehose deployment per log type, but requires explicit S3 prefix partitioning to keep Athena query performance reasonable.
  3. The compliance requirement is true WORM — S3 Object Lock in Compliance mode with a 7-year (2557-day) default retention period.
  4. Analysts use a cloud-native search stack: OpenSearch Service for hot logs (last 90 days, near-real-time) and Athena for SQL queries over cold logs (> 90 days, S3-based).
  5. No external data residency constraints — all logs may be centralized in us-east-1 primary with us-west-2 as the disaster-recovery replica.
  6. GuardDuty and Security Hub findings are in scope for aggregation — handled in the Security account, not in the logging pipeline itself, but fed into the same analyst search surface.
  7. Member accounts have no console or API access to the Log Archive account — access is zero-trust, enforced by SCP and Log Archive IAM resource policies.

Hop 1 — CloudTrail Organization Trail

  • The Management account enables a single Organization Trail via cloudtrail:CreateTrail with IsOrganizationTrail=true.
  • CloudTrail writes encrypted log files (SSE-KMS) directly to the Log Archive S3 bucket using a cross-account S3 bucket policy that grants s3:PutObject only to the cloudtrail.amazonaws.com service principal with condition aws:SourceArn matching the trail ARN.
  • Data is encrypted in transit (HTTPS) and at rest (KMS CMK). Log files arrive within ~15 minutes of the API call.
  • CloudTrail log file integrity validation is enabled: SHA-256 digest files are written hourly; tampering with any log file breaks the digest chain and is detectable.

Hop 2 — VPC Flow Logs

  • Each member account's VPC Flow Logs are published to CloudWatch Logs (managed by the member account) in the format ${account-id}/${region}/${vpc-id}/.
  • A CloudWatch Logs Subscription Filter in each member account forwards records to a Kinesis Data Firehose delivery stream configured to deliver to the Log Archive S3 bucket.
  • The Firehose delivery role in each member account assumes a cross-account IAM role in Log Archive via sts:AssumeRole to write to the centralized bucket.
  • Data is buffered (5 minutes / 128 MB, whichever comes first), gzip-compressed, and SSE-KMS encrypted on delivery.

Hop 3 — Route 53 Resolver Query Logs

  • Route 53 Resolver query logging is enabled per VPC via route53resolver:AssociateResolverQueryLogConfig.
  • The Resolver Query Log Config destination is set directly to the Log Archive S3 bucket ARN (Route 53 Resolver supports cross-account S3 delivery natively).
  • Logs arrive in JSON format with DNS query details, encrypted with the Log Archive KMS CMK.

Hop 4 — AWS Config Snapshots and Configuration History

  • An AWS Config Organization Aggregator in the Management account (config:PutConfigurationAggregator) collects configuration items from all accounts and regions.
  • Each member account's Config delivery channel is pointed at the Log Archive S3 bucket using a cross-account Config delivery role and bucket policy.
  • Config snapshots (daily) and configuration history (continuous) land in s3://log-archive-bucket/AWSLogs/{account-id}/Config/{region}/.

Hop 5 — Application Logs

  • Applications write to CloudWatch Logs log groups (structured JSON preferred, enforced via CloudWatch Logs subscription filter pattern).
  • A CloudWatch Logs Subscription Filter per log group sends matched records to Kinesis Data Firehose.
  • For 200 accounts, subscription filters are deployed via AWS Organizations EventBridge Rules (organization-wide event bus rules) that trigger a central Lambda function in Log Archive when new log groups are created (event: CreateLogGroup). The Lambda automatically attaches the subscription filter — no manual per-account work.
  • Firehose delivers to s3://log-archive-bucket/application/{account-id}/{region}/{year}/{month}/{day}/.

Hop 6 — Hot Path: Near-Real-Time Search

  • A second Kinesis Data Firehose stream (or a Kinesis Data Streams → Lambda fan-out) streams the last 90 days of high-value logs (CloudTrail, VPC Flow Logs, application) to Amazon OpenSearch Service in the Security account.
  • OpenSearch receives records in near-real-time (< 1 minute delivery). Index lifecycle management (ILM) policy rotates indices after 90 days and deletes from OpenSearch (cold data stays in S3).
  • Analysts authenticate to OpenSearch Dashboards via AWS IAM Identity Center (SSO) with SAML federation — no username/password.

Hop 7 — Cold Path: Ad-Hoc SQL Queries

  • AWS Glue Crawlers run daily to update the Glue Data Catalog with S3 partition metadata (year/month/day/account-id).
  • Amazon Athena queries the catalog directly over S3 using columnar formats (convert to Parquet via a daily Glue ETL job for cost efficiency).
  • Athena results are written to a separate query-results S3 bucket in the Security account (not the log archive bucket).

Hop 8 — Disaster Recovery Replica

  • S3 Cross-Region Replication (CRR) replicates all objects from the primary Log Archive bucket (us-east-1) to a replica bucket (us-west-2).
  • CRR is configured with ReplicaModifications: ENABLED and DeleteMarkerReplication: DISABLED (we don't replicate deletions, because Object Lock prevents them anyway — this is a defense-in-depth choice).
  • Object Lock retention settings are replicated to the destination bucket.

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Cross-account log delivery roles (member → Log Archive):

Role: arn:aws:iam::<log-archive-acct>:role/LogDeliveryFirehoseRole
Trust policy principal: arn:aws:iam::<member-acct>:role/FirehoseDeliveryRole
Condition: aws:PrincipalOrgID = o-xxxxxxxxxxxx
Allowed actions: s3:PutObject on arn:aws:s3:::log-archive-bucket/AWSLogs/${aws:PrincipalAccount}/*
  • The path-based condition ${aws:PrincipalAccount} ensures a compromised member account can only write to its own prefix, not overwrite another account's logs.

CloudTrail S3 bucket policy (cross-account write from CloudTrail service):

{
  "Effect": "Allow",
  "Principal": { "Service": "cloudtrail.amazonaws.com" },
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::log-archive-bucket/AWSLogs/*",
  "Condition": {
    "StringEquals": { "s3:x-amz-acl": "bucket-owner-full-control" },
    "ArnLike": {
      "aws:SourceArn": "arn:aws:cloudtrail:*:<mgmt-acct>:trail/org-trail"
    }
  }
}

Analyst read-only access:

  • IAM Identity Center permission sets grant AthenaReadOnly and OpenSearchRead to the Security account only.
  • Analysts have no write access to the Log Archive account whatsoever — not even via break-glass.
  • Log Archive account break-glass is a hardware-MFA-protected root login stored in a physical vault.

SCPs on Log Archive OU (deny-list guardrails):

{
  "Effect": "Deny",
  "Action": [
    "cloudtrail:StopLogging",
    "cloudtrail:DeleteTrail",
    "cloudtrail:UpdateTrail",
    "s3:DeleteBucket",
    "s3:DeleteBucketPolicy",
    "s3:PutBucketPolicy",
    "s3:DeleteObject",
    "s3:DeleteObjectVersion",
    "kms:ScheduleKeyDeletion",
    "kms:DisableKey"
  ],
  "Resource": "*"
}

This SCP applies to every principal in the Log Archive account, including its own account administrators. Even if an attacker achieves full Admin access in Log Archive, they cannot stop logging or delete objects.

SCPs on all member account OUs:

{
  "Effect": "Deny",
  "Action": ["cloudtrail:StopLogging", "cloudtrail:DeleteTrail"],
  "Resource": "arn:aws:cloudtrail:*:*:trail/org-trail"
}

Prevents member account admins from disabling the organization trail in their account.

Key hierarchy:

Deny external org access on every CMK:

condition {
  test     = "StringNotEquals"
  variable = "aws:PrincipalOrgID"
  values   = ["o-xxxxxxxxxxxx"]
}
# Exception for AWS service principals (BoolIfExists aws:PrincipalIsAWSService = false)

Encryption in transit:

  • All S3 bucket policies include Deny on aws:SecureTransport = false — rejects HTTP connections.
  • Kinesis Firehose → S3: enforced HTTPS by the service.
  • OpenSearch: EnforceHTTPS = true, TLSSecurityPolicy = Policy-Min-TLS-1-2-2019-07.
  • CloudWatch Logs → Firehose subscription: encrypted in transit by the AWS service boundary.

Envelope encryption:

  • Firehose uses a Firehose CMK to generate a data encryption key (DEK) per record batch. The DEK is stored encrypted alongside the batch in S3 using the S3 CMK — two-layer envelope encryption.

3. Network Isolation

Log Archive account VPC design:

  • OpenSearch Service is deployed in a private VPC in the Security account (not Log Archive) — three private subnets across three AZs.
  • No internet gateway in the Log Archive account VPC. All API calls to S3, KMS, Firehose, CloudWatch, and Config go through VPC Interface Endpoints with endpoint policies.

VPC endpoint policies (example for S3 in Log Archive):

{
  "Effect": "Allow",
  "Principal": "*",
  "Action": ["s3:PutObject", "s3:GetObject"],
  "Resource": "arn:aws:s3:::log-archive-bucket/*",
  "Condition": {
    "StringEquals": { "aws:PrincipalOrgID": "o-xxxxxxxxxxxx" }
  }
}

Security groups:

  • Firehose delivery endpoints: no inbound rules (Firehose is a managed push service, not pull). Outbound: HTTPS (443) to VPC endpoint only.
  • OpenSearch: inbound HTTPS (443) from Security analyst CIDR only (corporate network CIDR or VPN endpoint IP). No 0.0.0.0/0 ingress.
  • No NACLs needed beyond defaults for Firehose — complexity/benefit tradeoff favors Security Groups for stateful filtering.

AWS Network Firewall (optional for large-scale):

  • If the Security account OpenSearch VPC has a Transit Gateway attachment to corporate network, add Network Firewall in inspection mode for DNS-based domain filtering.

4. Logging and Audit Trail

CloudTrail configuration:

Trail setting Value Rationale
IsOrganizationTrail true Covers all 200 accounts automatically
IncludeManagementEvents true (Read + Write) Captures all control plane activity
DataEvents (S3) true on log-archive-bucket Detects GetObject, DeleteObject on the log store itself
DataEvents (Lambda) true on log delivery Lambdas Detects code execution against delivery infrastructure
InsightSelectors ApiCallRateInsight, ApiErrorRateInsight Detects anomalous API bursts (e.g., bulk GetObject exfiltration)
LogFileValidationEnabled true SHA-256 digest files enable tamper detection
CloudWatchLogsLogGroupArn Set to Security account CWL group Enables metric filters and real-time EventBridge rules

S3 server access logging: Enabled on the log archive bucket, writing meta-logs to a separate s3://log-archive-access-logs/ bucket (also Object Lock protected, 90-day retention).

Log integrity chain:

  1. CloudTrail writes log file → generates SHA-256 digest file hourly.
  2. Digest file references the previous digest file's hash → chain of custody.
  3. aws cloudtrail validate-logs CLI command validates the chain. Failure = tamper alert.
  4. Digest files are themselves stored in S3 Object Lock (Compliance mode, 7 years).

5. Least Privilege and Blast Radius

If a member account is fully compromised:

  • Attacker can write logs to their own S3 prefix (poisoned data), but cannot read other accounts' logs.
  • Attacker cannot delete any logs — Object Lock.
  • Attacker cannot disable the Organization Trail — SCP blocks cloudtrail:StopLogging and cloudtrail:DeleteTrail on the org trail ARN.
  • Attacker cannot access the KMS CMK — key policy denies usage outside Log Archive account except via kms:ViaService.
  • Blast radius: limited to log poisoning in their own account prefix.

If the Log Archive account administrator is compromised:

  • SCP blocks deletion of the S3 bucket, log objects, and stopping CloudTrail — even for account admins.
  • Object Lock (Compliance mode) prevents DeleteObject and DeleteObjectVersion even for root.
  • Attacker cannot disable the Organization Trail from Log Archive — it is owned by the Management account.
  • Blast radius: can read all logs (exfiltration risk — see Threat Model). Cannot destroy logs.

If the Security account (analyst account) is compromised:

  • Security account has read-only access to Log Archive (Athena queries, OpenSearch read). No write or delete permissions.
  • GuardDuty and Security Hub configuration could be tampered with — separate detection needed (Config rule change alerting via Management account).
  • Blast radius: can exfiltrate logs and disable detection. Cannot destroy logs.

Permission boundaries:

  • All cross-account roles in Log Archive have a permission boundary of LogArchiveDeliveryBoundary — maximum allowed actions are s3:PutObject on the specific prefix, kms:GenerateDataKey, kms:Decrypt. Even if a role's trust policy is widened, the boundary caps access.

Scale and Resilience (Phase 4 — 10 min)

Multi-Region Design

Region Role RPO RTO
us-east-1 Primary — all log ingestion, OpenSearch, Athena queries 0 (no data loss — write path) 15 min (Firehose retry exhaustion triggers backup bucket)
us-west-2 DR replica via S3 CRR < 15 min (CRR replication lag) 4 hours (Athena can query replica; OpenSearch rebuild takes longer)
  • S3 CRR replicates every new object within ~15 minutes. RTO for cold data queries: immediate (Athena in us-west-2 points to replica bucket).
  • OpenSearch DR: no active-active OpenSearch in us-west-2 at normal cost. In disaster, spin up a new OpenSearch cluster and reindex the last 90 days from S3. Estimated time: ~4 hours for 90 days × 500 GB/day = 45 TB reindex.
  • For regulatory mandates requiring continuous analyst access during regional failure: keep a warm OpenSearch cluster in us-west-2 with the last 7 days of data only (cost: ~30% of primary).

Threat Model and Operations (Phase 5 — 15 min)

Threat 1 — Insider Threat: Malicious Admin in a Member Account

Attack path:

  1. Member account admin calls cloudtrail:StopLogging on the org trail.
  2. Admin executes malicious action (data exfiltration, privilege escalation).
  3. Admin calls cloudtrail:StartLogging to resume — attempting to hide the activity window.

Preventive control: SCP on all member OUs denies cloudtrail:StopLogging and cloudtrail:DeleteTrail on the org trail ARN. The API call is rejected with AccessDenied before execution.

Detective control: CloudTrail records the denied StopLogging attempt. EventBridge rule on errorCode = AccessDenied + eventName = StopLogging → SNS → PagerDuty P1 alert within 2 minutes.

Response playbook (first 15 minutes):

  1. Isolate the member account — apply a Deny-All SCP to the specific account via organizations:AttachPolicy.
  2. Pull the CloudTrail event history for the member account for the last 24 hours using cloudtrail:LookupEvents.
  3. Identify any actions taken during any logging gap using Athena query on VPC Flow Logs (network activity doesn't require CloudTrail).
  4. Engage the account owner. Preserve all evidence — do not terminate instances or delete resources until forensic snapshots are taken.

Threat 2 — External Attacker Compromises Log Archive Account Root

Attack path:

  1. Attacker phishes the Log Archive account root email or finds credentials in a public repository.
  2. Attacker attempts to disable Object Lock or bulk-delete S3 objects to destroy evidence of their activity.
  3. Attacker attempts to delete the KMS CMK to make existing logs permanently inaccessible.

Preventive control:

  • Root account has hardware MFA (YubiKey) — cannot be phished without physical access.
  • S3 Object Lock Compliance mode: DeleteObject on locked objects returns AccessDenied even for root.
  • KMS ScheduleKeyDeletion has a minimum 7-day waiting period — cannot be instant. SCP blocks kms:ScheduleKeyDeletion and kms:DisableKey in Log Archive.
  • Root login triggers immediate CloudWatch alarm.

Detective control: CloudTrail ConsoleLogin with userIdentity.type = Root → EventBridge rule → SNS P0 → PagerDuty + automatic Slack alert to security team within 30 seconds.

Response playbook:

  1. Alert fires on root login — security team assembles within 5 minutes.
  2. Check CloudTrail for DeleteObject, ScheduleKeyDeletion, PutBucketPolicy calls.
  3. If compromise confirmed: contact AWS Support (abuse) to lock the account.
  4. Verify CRR replica in us-west-2 is intact using s3:ListObjectVersions on replica bucket.
  5. Rotate root credentials, enable new MFA device.

Threat 3 — Supply Chain / Third-Party Risk

Attack path:

  1. A third-party Lambda layer or container image used in the log delivery pipeline contains malicious code.
  2. Malicious code exfiltrates log data to an external endpoint, or modifies log records before delivery to S3.

Preventive control:

  • Log delivery Lambdas have VPC placement with no internet access — all outbound traffic goes through VPC endpoints only. Malicious exfiltration to an external endpoint is blocked at the network level.
  • Lambda functions use AWS-managed base layers only, pinned to specific versions. ECR private registry with image scanning (inspector:StartCisScan) on every push.
  • lambda:UpdateFunctionCode is denied for the delivery Lambda roles via SCP — code changes require a CD pipeline with human approval.

Detective control: GuardDuty Backdoor:Lambda/C&CActivity.B finding if Lambda attempts unexpected network calls. VPC Flow Logs show rejected outbound connections.

Response playbook:

  1. Isolate affected Lambda — set reserved concurrency to 0 (lambda:PutFunctionConcurrency = 0).
  2. Audit all log objects delivered in the last 24 hours using CloudTrail data events — verify SHA-256 hashes against a known-good baseline.
  3. Re-deploy Lambda from a clean artifact in the CD pipeline.

Threat 4 — Data Exfiltration (Log Data)

Attack path:

  1. Analyst with Security account access calls s3:GetObject in bulk on the Log Archive bucket via aws s3 sync.
  2. Logs containing sensitive application data (PII, credentials in error messages) are exfiltrated to a personal S3 bucket or external service.

Preventive control:

  • Athena is the query interface — analysts never have direct s3:GetObject on the raw log bucket. The bucket policy explicitly denies s3:GetObject to the Security account role; Athena queries via a service role with scoped access.
  • Amazon Macie enabled on the Log Archive bucket — detects PII in log objects and alerts if a new log group starts writing SSN/credit card data.
  • aws:RequestedRegion condition on bucket policy restricts all API calls to us-east-1 — prevents cross-region GetObject via a foreign endpoint.

Detective control: CloudTrail data events on the Log Archive bucket track every GetObject. CloudTrail Insights detects anomalous GetObject API rate (e.g., >10,000 calls/hour from a single principal). EventBridge rule → SNS → P1 alert.

Response playbook:

  1. Identify the IAM principal and session from CloudTrail GetObject events.
  2. Revoke the IAM Identity Center session via sso-admin:DeleteAccountAssignment.
  3. Scope the exfiltrated data — query CloudTrail for all GetObject calls in the past 24 hours from the principal.
  4. Invoke data breach response procedure if PII is confirmed exfiltrated (GDPR 72-hour notification clock starts).

Threat 5 — Privilege Escalation

Attack path:

  1. Attacker compromises a Firehose delivery role in a member account.
  2. Attacker uses sts:AssumeRole to escalate to the cross-account LogDeliveryFirehoseRole in Log Archive.
  3. From Log Archive, attacker attempts to escalate further using iam:PassRole or iam:AttachRolePolicy.

Preventive control:

  • LogDeliveryFirehoseRole has a permission boundary — maximum allowed actions are s3:PutObject on the specific account prefix and kms:GenerateDataKey. Cannot assume other roles or modify IAM.
  • iam:PassRole, iam:CreateRole, iam:AttachRolePolicy are not in the delivery role's permissions or boundary.
  • SCP on Log Archive denies iam:CreatePolicy, iam:AttachRolePolicy for all principals except the IamAdminRole.

Detective control: CloudTrail AssumeRole with a member account as source → EventBridge rule flags any AssumeRole into Log Archive that is not from the expected Firehose service principal → P1 alert.

Response playbook:

  1. Revoke all active sessions for the compromised role via sts:RevokeTemporarySecurityCredentials equivalent — apply an inline deny policy to the role.
  2. Review CloudTrail for all API calls made under the escalated role in the last 1 hour.
  3. Rotate the trust policy to restrict to a new, fresh delivery role ARN.

Threat 6 — Denial of Service (Log Pipeline)

Attack path:

  1. Attacker (or buggy application) generates a log flood — 100× normal volume — overwhelming Kinesis Firehose buffers and causing S3 throttling.
  2. Legitimate log delivery is delayed or dropped, creating a monitoring blind spot.

Preventive control:

  • Kinesis Data Firehose auto-scales to handle burst. Buffer size capped at 128 MB — large bursts are queued, not dropped.
  • CloudWatch Logs subscription filters have a FilterPattern — only matching events are forwarded, limiting the blast radius of a log flood from a single account.
  • S3 presigned URL delivery is not used — Firehose has its own retry logic (300-second retry duration, backup bucket for failed deliveries after retry exhaustion).

Detective control: CloudWatch metric DeliveryToS3.DataFreshness for Firehose — alarms if delivery latency exceeds 600 seconds. Separate alarm on BackupToS3.Success rate drops.

Response playbook:

  1. Identify the flooding log group/account from CWL IncomingBytes metric.
  2. Temporarily update the subscription filter pattern to a more restrictive filter for the flooding source.
  3. If an application bug: notify the account owner; they must fix the logging configuration.
  4. If a deliberate attack: isolate the account (Deny-All SCP on the specific account).

Common Mistakes to Avoid

  1. "Enable CloudTrail in every account individually."
    The correct answer is a single Organization Trail enabled from the Management account with IsOrganizationTrail=true. This automatically covers all current and future accounts. Per-account trails are unmanageable at 200 accounts and can be disabled by member admins (no SCP protection unless you add it per-trail, per-account).

  2. "Use S3 lifecycle to enforce retention."
    S3 lifecycle policies are retention helpers, not WORM controls. A lifecycle expiration rule can be deleted or modified by anyone with s3:PutLifecycleConfiguration. The only tamper-proof retention mechanism is S3 Object Lock in Compliance mode. Confusing lifecycle with immutability is a critical gap in a compliance-focused design.

  3. "Store logs in the same account as the workloads."
    Log data in the same account as a compromised workload can be tampered with before investigators arrive. The entire value of a centralized logging platform is the separation of duty — workload admins can never touch the Log Archive account due to SCPs and cross-account IAM design.

  4. "Encrypt with the default AWS-managed S3 key."
    AWS-managed keys (aws/s3) cannot be used for cross-account access control. You need a CMK with an explicit key policy listing exactly which service principals and cross-account roles can call kms:GenerateDataKey and kms:Decrypt. Without a CMK, you have no visibility into who decrypted your log data in CloudTrail.

  5. "Use a single OpenSearch cluster for everything."
    OpenSearch's index lifecycle management is essential here. Keeping 7 years of log data in a single "hot" OpenSearch cluster would cost ~$250,000/month. The correct pattern is: OpenSearch for hot data (last 90 days) + UltraWarm for warm data (90 days–1 year) + S3/Athena for cold data (1–7 years). Mention ILM in every answer involving OpenSearch.

  6. Forgetting the "meta-logging" loop.
    The log archive bucket itself must be logged. Enable S3 Server Access Logging on the log archive bucket to a separate access-log bucket. If you don't, an attacker who accesses or modifies the log archive leaves no evidence. The access-log bucket must also have Object Lock.

  7. Not mentioning log delivery gaps as a residual risk.
    Kinesis Firehose has a 60–900 second buffer. CloudTrail has a ~15-minute delivery lag. These gaps are documented, accepted risks — not design flaws. But failing to acknowledge them in practice signals you haven't operated these systems in production. Note them in the risk register with the acceptable latency thresholds.

Likely Follow-up Questions

Q1: How would you handle a new account being added to the organization? Does it automatically get logging coverage?

Yes — the Organization Trail covers all current and future accounts automatically. For VPC Flow Logs and Route 53 query logs, use an AWS Organizations EventBridge rule that triggers a Lambda when CreateAccount or MoveAccount events occur. The Lambda invokes ec2:CreateFlowLogs and route53resolver:CreateResolverQueryLogConfig in the new account via a cross-account role. For application logs, the same EventBridge → Lambda pattern attaches a CloudWatch Logs subscription filter to any new log groups. AWS Control Tower's Account Factory can bake these baseline controls into the account vending template.

Q2: The compliance team says we need to prove logs haven't been tampered with. How do you do that technically?

Three mechanisms in combination: (1) CloudTrail log file integrity validation — enable --enable-log-file-validation on the trail. CloudTrail generates an hourly SHA-256 digest file that chains to the previous digest. Run aws cloudtrail validate-logs against any time range; a broken chain proves tampering. (2) S3 Object Lock — even if someone could overwrite an object, Object Lock in Compliance mode rejects the write at the API level, so the original is preserved. (3) S3 Versioning — even for non-Object-Lock objects, all versions are retained. Present an auditor with the CloudTrail validation output, the S3 Object Lock configuration, and the Glacier inventory report showing every object's creation time and ETag.

Q3: A security analyst needs to query logs from 3 years ago. Walk me through the query path.

The analyst opens Athena in the Security account console (accessed via IAM Identity Center SSO). Athena queries the Glue Data Catalog, which has partitions pointing to s3://log-archive-bucket/AWSLogs/{account-id}/CloudTrail/{region}/{year}/{month}/{day}/. Since data older than 1 year is in S3 Glacier Instant Retrieval, the first SELECT on 3-year-old data triggers a Glacier Instant retrieval (millisecond latency — no restore needed for Instant Retrieval). The query runs, Athena charges $5/TB scanned. If the data is in Glacier Deep Archive (> 1 year if lifecycle is configured that way), the analyst must first initiate a restore (s3:RestoreObject), wait 12 hours for standard restore, then re-run the Athena query against the restored copy. Best practice: keep Instant Retrieval for year 1–2, Deep Archive only for years 3–7, to balance cost vs analyst wait time.

Q4: How do you ensure logs from a compromised member account aren't being poisoned (fake log entries injected)?

You can't fully prevent it — log poisoning (injecting false events) is a known limitation of centralized CloudTrail. A compromised account can make API calls that generate real CloudTrail events, including calls designed to obscure other activity. Mitigations: (1) CloudTrail Insights detects anomalous API call rates and error rates — a burst of unusual API calls stands out. (2) Correlation — cross-reference CloudTrail events against VPC Flow Logs (network layer) and application logs (application layer). Real attacks leave traces in all three; fabricated CloudTrail events alone without corresponding network activity are suspicious. (3) GuardDuty ML models are trained on normal account behavior; sudden behavioral changes (new API calls, unusual regions) generate findings even if CloudTrail looks clean. Document this as a residual risk: CloudTrail is a data-plane log, not a tamper-proof source of truth at the member account level.

Q5: The CISO asks why we need both OpenSearch AND Athena. Isn't that two search systems for the same data?

They serve different query patterns at fundamentally different cost points. OpenSearch is optimized for interactive, sub-second search — an analyst types a source IP and gets results in milliseconds. It indexes every field in every log record, which costs ~$3,200/month but makes exploration fast. Athena is optimized for batch analytical queries — "give me all S3 GetObject calls from this IAM role in the last 3 years" — scanned as SQL over compressed Parquet files. It costs $5/TB scanned with zero idle cost, but a complex query over 1 TB of data takes 30–90 seconds. Keeping both means you pay for OpenSearch only for the hot 90-day window, and Athena covers the long tail at near-zero cost. A single OpenSearch cluster for 7 years of data would cost approximately $250,000/month.

Q6: How would you handle a regulatory requirement that EU accounts' logs cannot leave the EU?

Introduce regional Log Archive sub-accounts per data residency zone — one log-archive-eu-west-1 account for EU workloads and one log-archive-us-east-1 account for US workloads. The Organization Trail is still single, but you configure separate S3 bucket destinations per region using the trail's s3BucketName parameter split by a prefix. EU KMS CMKs are created in eu-west-1 only. The S3 CRR policy for EU buckets restricts replication to eu-west-1 → eu-central-1 only — never crosses to us-east-1. OpenSearch for EU analysts is deployed in eu-west-1. The trade-off: you now have two analyst portals, two Athena workgroups, two KMS key hierarchies to manage. Document this as a design boundary. Use AWS Lake Formation to provide a unified data catalog view across both regions without physically moving data.

Q7: What happens if a GuardDuty finding fires on the Log Archive account itself?

GuardDuty organization administrator is the Security account — it receives findings from all accounts including Log Archive. A finding in Log Archive (e.g., UnauthorizedAccess:S3/TorIPCaller — Tor exit node accessing the log bucket) should be treated as a P0 incident immediately. Response: (1) EventBridge rule in the Security account monitors for aws.guardduty events with detail.accountId = <log-archive-account-id> and HIGH/CRITICAL severity → SNS → PagerDuty P0. (2) Automated response Lambda in Security account calls iam:PutRolePolicy to attach a Deny-All inline policy to any IAM role that was the detail.resource.accessKeyDetails.userArn. (3) Human review of full CloudTrail history for the Log Archive account for the 24 hours prior. The key point: Log Archive should have extremely low API call volume (mostly CloudTrail writes and scheduled Glue crawls) — any anomaly is highly suspicious.

Q8: Walk me through exactly what happens operationally on day 2,557 (end of the 7-year retention period) for an object locked in Compliance mode.

On the day the x-amz-object-lock-retain-until-date timestamp expires, the object automatically transitions out of the locked state — no action needed. The S3 Object Lock simply stops preventing deletion; it does not delete the object. The object now follows normal S3 lifecycle rules: if a lifecycle expiration rule exists for objects older than 2,557 days, S3 will delete the object on the next lifecycle run (daily). If no lifecycle expiration rule exists, the object persists indefinitely and continues incurring storage cost. Best practice: create a separate lifecycle rule for the "post-lock" tier that expires objects at days = 2558 — this ensures automatic cleanup the day after the lock expires, preventing unbounded cost growth. Test this in a lower-retention dev environment (e.g., 90-day lock + 91-day expiry) before relying on it for production 7-year objects.
