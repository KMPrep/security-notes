Q02 — Zero Trust Network Architecture for a Hybrid Enterprise

The Ask

"Design a Zero Trust network architecture on AWS for an enterprise with 10,000 employees,
50 on-premises applications, 100 cloud-native workloads, and contractors on BYOD devices.
All application access must be identity-aware, device-posture-checked, and micro-segmented.
No application should be reachable via VPN."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Is there an existing AWS Direct Connect or Site-to-Site VPN connecting on-premises data centers to AWS, and what is its bandwidth?
    Impact: If Direct Connect exists, I can route identity-broker traffic over private circuits. Without it, on-premises applications are accessed over the public internet, which means Verified Access endpoints must be public-facing, and I need tighter WAF rules and certificate pinning for the access proxy.

  • Q: Which Identity Provider (IdP) is in use — Okta, Azure AD / Entra ID, Google Workspace, or an on-premises AD FS?
    Impact: IAM Identity Center supports SAML 2.0 federation with all major IdPs. If Azure AD is the IdP, I use OIDC federation directly. The choice affects SSO token format, MFA claim propagation, and how device posture signals are surfaced to Verified Access trust providers.

  • Q: Are workloads primarily containerized (EKS) or VM-based (EC2), and is there a service mesh in place (Istio, App Mesh)?
    Impact: Containerized east-west traffic is best secured with VPC Lattice + mTLS at the sidecar level. VM-based traffic is secured with Security Groups + Network Firewall. The answer changes the micro-segmentation technology stack significantly.

  • Q: What is the contractor access pattern — browser-based web apps only, or do they need SSH/RDP to servers?
    Impact: Browser apps → AWS Verified Access with BYOD device trust. SSH/RDP → EC2 Instance Connect Endpoint or SSM Session Manager (no inbound ports opened). BYOD SSH is the hardest case and may require a bastion alternative with hardware-token MFA.

  • Q: Is this a greenfield Zero Trust design or migration from an existing Cisco AnyConnect / Palo Alto GlobalProtect VPN?
    Impact: Greenfield = clean slate, can enforce policy from day one. Migration = phased rollout, need a VPN-and-Verified-Access coexistence period, risk of policy gaps during transition. Migration adds 6–12 months to the timeline.

  • Q: What is the device management posture for employees — are all corporate devices enrolled in an MDM (Jamf, Intune, or AWS Verified Access device trust provider)?
    Impact: Device posture check is only as strong as MDM enrollment completeness. If 20% of devices are unmanaged, those users cannot satisfy the Cedar policy and will be blocked — I need an exception workflow and on-boarding path for unmanaged devices.

  • Q: What latency SLA do employees expect for application access? Is there a maximum acceptable added latency from the Zero Trust proxy layer?
    Impact: Verified Access adds ~2–5ms per request (TLS termination + policy evaluation). For latency-sensitive apps (trading, video conferencing), I may need to co-locate Verified Access endpoints in the same region as the application to minimize round-trip.

  • Q: Are there regulatory requirements (PCI-DSS, HIPAA, FedRAMP) that mandate specific network segmentation controls or audit logging of all access?
    Impact: PCI-DSS requires all CDE-bound access to be logged. Verified Access access logs satisfy this requirement. FedRAMP may require FIPS 140-2 endpoints. These constraints narrow the service choice or require additional configuration.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Corporate IdP is Azure AD (Entra ID) federated into AWS IAM Identity Center via OIDC.
  2. All corporate devices are enrolled in Microsoft Intune (MDM). Contractors use BYOD — device posture checked via Verified Access device trust provider (AWS-managed).
  3. Cloud workloads run on EKS (containerized) and EC2 (VM-based) across three VPCs: edge-vpc, app-vpc, data-vpc.
  4. On-premises applications are accessible via AWS Direct Connect (10 Gbps) — no public internet path for on-prem apps.
  5. The VPN is decommissioned in Phase 2 of the migration (90-day window). Verified Access is the sole access path by end state.
  6. All service-to-service traffic (cloud-native) uses VPC Lattice with IAM auth policies — no Security Group-based implicit trust.

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Verified Access trust chain:

  1. Employee opens browser → navigates to https://app.internal.company.com (Verified Access endpoint).
  2. Verified Access redirects to IAM Identity Center → federated to Azure AD → MFA prompt (TOTP or hardware key).
  3. IAM Identity Center returns OIDC token with claims: email, groups, mfa_authenticated=true.
  4. Intune device compliance signal sent to Verified Access device trust provider: managed=true, compliance_state=compliant, os_version=14.5.
  5. Cedar policy evaluated:
    cedar permit( principal, action == Action::"access", resource == Application::"payroll-app" ) when { context.identity.mfa_authenticated == true && context.device.managed == true && context.device.compliance_state == "compliant" && principal.groups.contains(Group::"payroll-team") };
  6. Verified Access proxies the request to the ALB in app-vpc. The ALB never has a public-facing rule — it only accepts traffic from the Verified Access service CIDR.

Contractor access (BYOD):

  • Verified Access device trust provider issues a device certificate via browser-based enrollment (no MDM required).
  • Cedar policy for contractors: managed=false is permitted only for contractor applications (lower-sensitivity apps). Production databases require managed=true.

IAM Identity Center permission sets:

  • Developers: PowerUserAccess minus iam:* actions, scoped to dev accounts only.
  • SREs: ReadOnlyAccess in prod; JIT AdministratorAccess via time-bound assignment (4-hour TTL) with manager approval.
  • Security team: custom SecurityAuditRole with read access across all accounts; no iam:PassRole.

2. Encryption (At Rest, In Transit, In Use)

In Transit:

  • All traffic between devices and Verified Access endpoints: TLS 1.3 (ECDHE key exchange, forward secrecy).
  • Verified Access to ALB: TLS 1.3 over VPC private network (HTTPS enforcement on ALB listener).
  • Service-to-service via VPC Lattice: mTLS with ACM Private CA-issued certificates (90-day rotation automated via ACM).
  • Direct Connect: MACsec encryption on the Direct Connect connection (hardware-level L2 encryption for on-prem traffic).
  • SSM Session Manager: TLS 1.2+ encrypted channel; sessions never traverse the public internet.

At Rest:

  • EBS volumes: encrypted with KMS CMK (aws:kms), per-account CMK with EnableKeyRotation = true.
  • RDS/Aurora: storage encrypted with CMK; automated backups encrypted with same CMK.
  • DynamoDB: SSE with CMK.
  • S3 (logs, artifacts): SSE-KMS with CMK; aws:SecureTransport enforced in bucket policy.

In Use (data plane):

  • Lambda and ECS task memory: application-level field encryption for PII fields using KMS GenerateDataKey (envelope encryption).

3. Network Isolation

VPC Design (3-tier):

  • edge-vpc (CIDR: 10.0.0.0/16): ALBs, Network Firewall, NAT Gateways. No direct compute.
  • app-vpc (CIDR: 10.1.0.0/16): ECS/EKS workloads in private subnets only. No public subnets.
  • data-vpc (CIDR: 10.2.0.0/16): RDS, DynamoDB, ElastiCache. Private subnets only. No internet gateway.

Transit Gateway route tables:

  • edge-rt: routes only to app-vpc; no route to data-vpc directly.
  • app-rt: routes to data-vpc and edge-vpc; no route to other spoke VPCs (east-west isolation).
  • data-rt: routes only to app-vpc. Data tier cannot initiate connections outbound.

Security Groups (micro-segmentation):

  • Every ECS task has its own Security Group; inbound rule source = specific SG of the upstream caller.
  • No 0.0.0.0/0 ingress rules anywhere. Verified by AWS Config rule no-unrestricted-ingress on all accounts.
  • Security Groups reference Security Group IDs (not CIDRs) to survive IP changes during scaling.

Network Firewall (stateful rules):

  • Block all non-HTTPS outbound (port 443 only allowed for known domain patterns via SNI inspection).
  • Suricata ET/Open ruleset enabled for IDS: detect port scans, lateral movement patterns, Cobalt Strike beacons.
  • Firewall logs: flow logs + alert logs → CloudWatch Logs → S3.

PrivateLink endpoints:

  • VPC interface endpoints in app-vpc and data-vpc for: com.amazonaws.*.ssm, com.amazonaws.*.kms, com.amazonaws.*.secretsmanager, com.amazonaws.*.ecr.api, com.amazonaws.*.ecr.dkr.
  • Endpoint policies restrict usage to specific IAM roles (not open to all principals in the account).

5. Least Privilege and Blast Radius

  • Blast radius by VPC: compromise of one VPC cannot reach others — Transit Gateway route tables enforce hard boundaries.
  • Blast radius by service account: EKS pods use Pod Identity (not node instance profile); each pod has a unique IAM role scoped to its own resources.
  • Blast radius by Verified Access endpoint: compromise of one application's Verified Access endpoint does not expose others — separate endpoint per application with distinct Cedar policies.
  • IAM Permission Boundaries: applied to all developer-created roles. Boundary policy caps maximum permissions to the specific account OU tier (dev, staging, prod).
  • SCP denials (cannot be overridden by account admins):
  • ec2:AuthorizeSecurityGroupIngress with 0.0.0.0/0 → denied.
  • ec2:CreateVpc without required tags → denied.
  • guardduty:DisableOrganizationAdminAccount → denied.

Scale and Resilience (Phase 4 — 10 min)

Verified Access scaling:

  • Managed service — auto-scales to handle thousands of concurrent sessions. No capacity planning needed.
  • Configure Verified Access endpoints in multiple AZs (Multi-AZ deployment within each region).
  • Multi-region: deploy Verified Access endpoints in us-east-1 and eu-west-1; Route 53 latency-based routing + health checks for failover. Verified Access configurations replicated via IaC (Terraform).

Transit Gateway scaling:

  • ECMP (Equal Cost Multi-Path) routing for Direct Connect: two 10 Gbps connections in Active/Active → 20 Gbps aggregate throughput.
  • Transit Gateway Connect (GRE tunnel over Direct Connect) for higher throughput attachments.
  • TGW route table isolation: adding new spoke VPCs does not automatically gain access to existing VPCs — explicit route table propagation required.

VPC Lattice scaling:

  • Service network handles 100+ microservices without per-service PrivateLink endpoints (eliminates ENI exhaustion risk).
  • Target group weight-based routing for A/B deployments; health checks at the VPC Lattice level.

IAM Identity Center:

  • Single IAM Identity Center instance per AWS Organization — regional failover is managed by AWS (HA across AZs).
  • Permission set changes are rolled out gradually via CI/CD (GitOps); no manual permission set changes.
  • SCIM provisioning from Azure AD ensures users de-provisioned from Entra ID are immediately removed from IAM Identity Center (no orphaned accounts).

Direct Connect resilience:

  • Two DX connections from two different DX locations (physical diversity) terminated to Transit Gateway.
  • Site-to-Site VPN as backup path for on-premises connectivity (used only as failover, not for user traffic).

Common Mistakes to Avoid

  1. "We'll use a VPN but make it Zero Trust." VPN grants network-level access; Zero Trust grants application-level access. They are architecturally incompatible. Verified Access is the replacement, not a VPN add-on.
  2. Forgetting device posture. Identity alone is insufficient for Zero Trust. You must check device health (MDM compliance, OS version, disk encryption status) on every request via the trust provider.
  3. Not explaining Cedar policy syntax. Interviewers expect you to write a Cedar policy, not just say "there's a policy engine."
  4. Conflating Zero Trust with Zero Trust Network Access (ZTNA). ZTNA is just one pillar. Full ZT covers identity, device, network, application, and data. Cover all five in your answer.
  5. Ignoring the migration path. A real enterprise has a VPN today. You must explain the coexistence period and how you decommission the VPN without user disruption.
  6. Using NACLs for micro-segmentation. NACLs are stateless and subnet-level. Security Groups are the right tool for east-west control. Using NACLs implies you misunderstand the AWS networking model.

Likely Follow-up Questions

Q1: How would you handle a contractor whose BYOD device fails the Intune compliance check mid-session?
Verified Access evaluates the trust provider signal on every request, not just at session start. When Intune marks the device as non-compliant, the next request from that device will fail the Cedar policy evaluation (device.compliance_state != "compliant") and be denied with a 403. The access log records the denial with the device ID. I'd configure a custom Verified Access error page that directs the contractor to the IT helpdesk portal.

Q2: What happens to Zero Trust for service-to-service calls that don't go through Verified Access?
East-west service calls go through VPC Lattice with IAM auth policies. Each service presents its Pod Identity IAM role credentials (SigV4 signed). The VPC Lattice service policy permits only specific source IAM roles. This is Zero Trust at the service mesh layer — no implicit trust based on being in the same VPC.

Q3: How do you prevent Transit Gateway from becoming a single point of failure for Zero Trust?
Transit Gateway is Multi-AZ by design within a region; AWS manages the HA. For multi-region, I deploy an independent Transit Gateway per region and connect them with TGW peering. If a regional TGW fails, Route 53 ARC fails traffic to the other region. The key insight is that TGW failure = network partition, not security breach — the policy enforcement layer (Verified Access, Security Groups) is separate from the routing layer.

Q4: How do you enforce Zero Trust for machine-to-machine API calls from on-premises servers?
On-premises servers cannot use IAM roles directly. I use IAM Roles Anywhere — an on-premises X.509 certificate issued by the corporate PKI is presented to IAM Roles Anywhere, which exchanges it for short-lived AWS credentials. These credentials are scoped to a specific IAM role with least-privilege. The certificate itself is the device identity assertion.

Q5: If Verified Access goes down, are all employees locked out?
Yes, for browser-based apps — Verified Access is the single path. Mitigation: Verified Access is a managed service with AWS SLA (99.99%). Additionally, configure multi-region Verified Access endpoints with Route 53 health checks so us-east-1 failure fails over to eu-west-1. For break-glass scenarios, I maintain a read-only break-glass IAM role in each account with direct console access, accessible only via hardware MFA from the security team's dedicated workstation.

Q6: How do you handle DNS for Zero Trust — employees expect to type app.internal.company.com, not an AWS endpoint URL?
Route 53 Private Hosted Zone for internal.company.com with an alias record pointing to the Verified Access endpoint DNS name. For on-premises employees, Route 53 Resolver outbound endpoints forward .internal.company.com queries from corporate DNS to Route 53. Employees see the familiar URL; resolution goes to Verified Access behind the scenes.

Q7: How would you phase the migration from VPN to Zero Trust over 90 days?
Phase 1 (Days 1–30): Deploy Verified Access alongside VPN. Onboard low-sensitivity apps first. Pilot with 200 volunteers. Phase 2 (Days 31–60): Migrate all browser-based apps to Verified Access. VPN remains for SSH/RDP and on-prem apps. Phase 3 (Days 61–90): Migrate SSH to EC2 Instance Connect Endpoint + SSM. On-prem apps via Verified Access reverse proxy. Phase 4 (Day 91): Decommission VPN. Comms campaign + helpdesk support surge.

Q8: What is the blast radius if a Verified Access endpoint is compromised via a vulnerability in the web application it proxies to?
Verified Access is a transparent proxy — it does not execute application code. The blast radius of a web application vulnerability is limited to that application's resources (its VPC, its IAM role, its database). Verified Access itself cannot be compromised via the proxied application. The application's IAM role should have least-privilege (no ability to create new IAM roles, no access to other applications' secrets).
