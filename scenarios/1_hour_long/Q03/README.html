Q03 — Secrets Management Platform for 500 Microservices

The Ask

"Design a secrets management platform on AWS for a SaaS company running 500 microservices
on EKS. Secrets include database passwords, third-party API keys, TLS private keys, and
internal service tokens. The platform must eliminate hardcoded secrets, enforce rotation,
and provide full audit trail. Dev teams must be able to self-service without opening
support tickets."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Are all 500 microservices in a single EKS cluster or distributed across multiple clusters, accounts, and regions?
    Impact: Single cluster = simpler IRSA/Pod Identity setup. Multi-cluster/multi-account = EKS Pod Identity Association must be configured per account; Secrets Manager replication needed to secondary regions; cross-account access requires resource-based policies on secrets.

  • Q: What databases are in use — RDS/Aurora only, or also external (MongoDB Atlas, Snowflake, Confluent)?
    Impact: RDS/Aurora has native Secrets Manager rotation via AWS-provided Lambda rotation functions (zero custom code). External systems require custom Lambda rotation functions. Third-party SaaS credentials (Stripe, Twilio) have no automated rotation — rotation must be manual or via the vendor's API.

  • Q: Is there an existing secrets solution that must be migrated — HashiCorp Vault, Kubernetes Secrets in plain etcd, .env files?
    Impact: HashiCorp Vault migration: map Vault paths to Secrets Manager ARNs; update all app configs. Plain Kubernetes Secrets: scan for existing secrets with kubectl get secrets -A, migrate values to Secrets Manager, delete the K8s Secret. .env files in Git: scan with GitGuardian/truffleHog, revoke and rotate all exposed secrets immediately.

  • Q: How do dev teams currently deploy — GitOps (ArgoCD/Flux) or direct kubectl / Helm?
    Impact: GitOps: Secrets must never be stored in Git repositories (even encrypted). Use External Secrets Operator to sync from Secrets Manager into K8s Secrets at deploy time. Direct kubectl: harder to enforce; must add pre-commit hooks and CI pipeline scanning.

  • Q: What is the rotation tolerance — can applications handle a short reconnect, or is strictly zero-downtime rotation required?
    Impact: Secrets Manager maintains AWSCURRENT and AWSPREVIOUS labels. Most ORMs support reconnection on auth failure. If an app cannot tolerate reconnection, it must implement a "rotation-aware" client that checks AWSPREVIOUS on failure before erroring out. This affects the Lambda rotation function design.

  • Q: Is there a multi-tenancy requirement — should team A be unable to enumerate team B's secrets?
    Impact: IAM resource conditions on secretsmanager:GetSecretValue and secretsmanager:ListSecrets. Use secret ARN prefix by team/service. Deny secretsmanager:ListSecrets globally; allow only for specific path prefixes matching the calling team's namespace.

  • Q: What is the compliance requirement — SOC 2, PCI-DSS, HIPAA? Does it require secrets never be in plaintext in any log or storage?
    Impact: PCI-DSS: card-related credentials must have rotation enforced, and rotation must be logged. HIPAA: access to PHI-related credentials must be logged with caller identity. This drives CloudTrail data events on Secrets Manager and alerts on unexpected GetSecretValue callers.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Microservices run on EKS with Managed Node Groups (not Fargate) across two accounts: dev and prod, in us-east-1 with DR in us-west-2.
  2. Databases: RDS Aurora (MySQL) for transactional data, DynamoDB for metadata. External: Stripe API keys, SendGrid API keys.
  3. No existing secrets manager — migrating from hardcoded environment variables in Kubernetes Secrets (base64, unencrypted in etcd).
  4. GitOps deployment via ArgoCD.
  5. Dev teams deploy via ArgoCD; they should be able to self-service create/update secrets in dev accounts without a ticket.
  6. Zero-downtime rotation required (apps must not fail during rotation).

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

EKS Pod Identity (per service isolation):
Every microservice gets a unique IAM role. The role trusts the EKS Pod Identity service:

{
  "Effect": "Allow",
  "Principal": { "Service": "pods.eks.amazonaws.com" },
  "Action": ["sts:AssumeRole", "sts:TagSession"],
  "Condition": {
    "StringEquals": {
      "aws:SourceAccount": "123456789012",
      "eks:cluster-name": "prod-cluster"
    }
  }
}

IAM policy (GetSecretValue scoped to own secret):

{
  "Effect": "Allow",
  "Action": "secretsmanager:GetSecretValue",
  "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/payments-service/*",
  "Condition": {
    "StringEquals": {
      "aws:sourceVpc": "vpc-0abc123"
    }
  }
}
  • Secret ARN prefix is prod/<service-name>/ — each service can only access its own secrets.
  • aws:sourceVpc condition: GetSecretValue is only permitted from within the VPC (via VPC endpoint). Calls from outside the VPC (e.g., leaked credentials) are denied.

Developer self-service model:

  • Dev account: IAM Identity Center permission set SecretsManagerDeveloper allows CreateSecret, UpdateSecret, PutSecretValue, DescribeSecret on arn:aws:secretsmanager:*:*:secret:dev/<requester-team>/*.
  • Prod account: no developer PutSecretValue permission. Prod secrets are created by Terraform (reviewed via PR). Rotation is automated.
  • DescribeSecret + ListSecrets is scoped to the developer's team prefix only — team A cannot enumerate team B's secrets.

2. Encryption (At Rest, In Transit, In Use)

Secrets Manager encryption:

  • All secrets encrypted with KMS CMK at rest. CMK per secret category:
  • prod-db-creds-cmk: used for all prod/*/db-password secrets
  • prod-api-keys-cmk: used for prod/*/api-key secrets
  • prod-tls-keys-cmk: used for prod/*/tls-private-key secrets
  • CMK key policy: kms:Decrypt granted to specific IAM role ARNs (rotation Lambda + service IAM roles). Key administration (scheduling deletion, disabling) restricted to break-glass security role only.
  • EnableKeyRotation = true on all CMKs (annual rotation of key material, existing encrypted data re-encrypted transparently).

Kubernetes etcd encryption:

  • EKS: enable --encryption-provider-config with KMS provider pointing to a dedicated CMK. ESO-synced K8s Secrets are encrypted at rest in etcd using KMS envelope encryption.
  • Without this: K8s Secrets are base64-encoded in etcd (not encrypted). Any etcd backup or API server compromise exposes all secrets.

In transit:

  • All Secrets Manager API calls use HTTPS/TLS 1.2+. VPC endpoint policy enforces aws:SecureTransport.
  • ESO ↔ Secrets Manager: traffic stays within the VPC via the interface endpoint (never traverses public internet).

3. Network Isolation

  • VPC Endpoints: com.amazonaws.us-east-1.secretsmanager interface endpoint in each VPC. Endpoint policy:
    json { "Effect": "Allow", "Principal": "*", "Action": "secretsmanager:GetSecretValue", "Resource": "arn:aws:secretsmanager:us-east-1:*:secret:prod/*", "Condition": { "StringEquals": { "aws:PrincipalOrgID": "o-xxxxxxxxxx" } } }
    Denies access to secrets outside the AWS Organization.
  • Lambda Rotation Function: deployed in a private subnet. Inbound: none. Outbound via NAT Gateway to reach Aurora (via RDS private endpoint) and Secrets Manager (via VPC endpoint). No internet access.
  • Security Groups: ESO controller pod SG allows only outbound 443 to Secrets Manager VPC endpoint ENI. No inbound rules.

4. Logging and Audit Trail

Event CloudTrail Action Alert Condition
Secret read secretsmanager:GetSecretValue Caller is not the expected service IAM role
Secret creation secretsmanager:CreateSecret Created in prod account by non-Terraform principal
Rotation failure SecretsManagerRotationFailed (CW metric) Any rotation failure → PagerDuty P1
Secret deletion secretsmanager:DeleteSecret Any deletion → P0 alert
KMS key access kms:Decrypt for CMK Access from unexpected principal

CloudTrail data events for Secrets Manager must be explicitly enabled (management events don't capture GetSecretValue). Enable in all accounts via Control Tower customization.

5. Least Privilege and Blast Radius

  • Per-service IAM role: if one pod is compromised, the attacker can only access that pod's secrets (e.g., prod/payments-service/*). Cannot access prod/auth-service/* or any other team's secrets.
  • No secretsmanager:ListSecrets globally: prevents an attacker from enumerating all secrets in the account and knowing which high-value secrets to target.
  • ESO refresh interval: ESO re-fetches secrets every 1 hour by default. After rotation, pods pick up the new value within 1 hour without a restart (configurable to 5 minutes for critical services).
  • Rotation Lambda: Lambda execution role has only secretsmanager:GetSecretValue, secretsmanager:PutSecretValue, secretsmanager:UpdateSecretVersionStage, rds:ModifyDBInstance for the specific DB identifier. Cannot list or access other secrets.

Scale and Resilience (Phase 4 — 10 min)

Multi-region replication:

  • Secrets Manager replication: prod secrets replicated to us-west-2 using aws_secretsmanager_secret.replica resource. The replica is a read-only copy — rotation writes to the primary, then replicates.
  • KMS: a CMK in us-west-2 must exist before replication (Secrets Manager encrypts the replica with the target region's CMK). Use KMS Multi-Region keys (mrk-*) to replicate key material itself, simplifying this.
  • ESO in us-west-2 cluster reads from the us-west-2 Secrets Manager endpoint (no cross-region latency).

Secrets Manager service limits:

  • Default: 10,000 secrets per account. At 500 microservices × ~5 secrets each = 2,500 secrets. Well within limits. Monitor with CloudWatch SecretsManagerSecretCount metric and alert at 8,000.
  • API rate limits: 1,500 API calls/second per region. 500 pods × 1 call/hour = 0.14 calls/second. No rate limit concern. ESO in-memory caching further reduces API calls.
  • Rotation concurrency: Lambda concurrency for rotation functions — reserve 10 concurrent executions for rotation to prevent throttling during bulk rotation.

ESO caching:

  • ESO caches secrets in memory; only calls Secrets Manager on refreshInterval expiry or pod restart.
  • Configure refreshInterval: 1h for most secrets; 5m for rapidly-rotated credentials.
  • ESO uses a ExternalSecret CRD that references SecretStore (cluster-scoped or namespace-scoped). Use cluster-scoped ClusterSecretStore for shared infra secrets; namespace-scoped for team secrets.

Cost model at scale:

  • 2,500 secrets × $0.40/month = $1,000/month base cost.
  • API calls: 2,500 secrets × 720 refreshes/month × $0.05/10,000 calls = negligible.
  • Total: ~$1,000–$1,200/month — justify vs. engineer time saved managing HashiCorp Vault clusters.

Common Mistakes to Avoid

  1. Recommending IRSA instead of Pod Identity for new clusters. EKS Pod Identity is the current AWS recommendation for new EKS clusters. IRSA requires annotation on the K8s service account and has more complex token exchange. Pod Identity has simpler IAM trust policy setup.
  2. Forgetting etcd encryption. ESO stores secrets as Kubernetes Secrets in etcd. Without KMS encryption at rest for etcd, your Secrets Manager migration just moves the problem into etcd.
  3. Not enabling CloudTrail data events for Secrets Manager. Management events do not capture GetSecretValue. Without data events, you have no audit trail of who accessed which secret.
  4. Scoping IAM to secretsmanager:* on * resource. Every service role must be scoped to its own ARN prefix. Wildcard resource on Secrets Manager means any compromised pod can read all secrets.
  5. Not addressing rotation failure. Rotation failures are silent unless you add a CloudWatch alarm on SecretsManagerRotationFailed. Without alerting, a failed rotation leaves a stale secret indefinitely.
  6. Ignoring the cost. 5,000 secrets at $0.40/month = $2,000/month. Interviewers want to hear that you've modeled this. Parameter Store (free for standard parameters) should be used for non-secret config values.

Likely Follow-up Questions

Q1: How do you handle secret rotation for a service that has 100 concurrent pods all using the same database credential?
During rotation, Secrets Manager updates the DB password to AWSPENDING, verifies it works, then promotes it to AWSCURRENT. The old password stays as AWSPREVIOUS for a configurable window (I set 30 minutes). During this window, ESO picks up the new value and the K8s Secret is updated. Pods that have already loaded the connection pool (with the old password) will start seeing auth failures, which triggers a reconnect. The app must implement reconnection-on-auth-failure logic — this is standard in most connection pool libraries (HikariCP, pgbouncer). After 30 minutes, AWSPREVIOUS is deleted and any pods that haven't reconnected will fail permanently (triggering a restart).

Q2: What do you do when Secrets Manager has a regional outage?
Secrets Manager is a regional service with very high availability, but for DR, I configure cross-region replication to us-west-2. ESO in the us-west-2 cluster reads from the local Secrets Manager endpoint. For the primary region during an outage: ESO's in-memory cache means pods don't call Secrets Manager on every request — cached values persist until the refresh interval. If the outage exceeds the refresh interval, pods that restart will fail to load secrets. Mitigation: set a longer refresh interval for stable secrets (e.g., 6 hours for TLS certs) and enable ESO's errorPolicy: Ignore to keep the stale K8s Secret rather than deleting it on fetch failure.

Q3: How do you prevent a developer from accidentally deleting a production secret?
Three controls: (1) IAM: developer permission set in prod account does not include secretsmanager:DeleteSecret. Only the Terraform pipeline role has it. (2) Secrets Manager resource policy: deny DeleteSecret for any principal not in the approved deletion role list. (3) Secrets Manager recovery window: RecoveryWindowInDays = 30 — even if deletion is triggered, the secret enters a PENDING_DELETION state and can be recovered for 30 days. CloudTrail alert on any DeleteSecret call triggers an immediate P0 ticket.

Q4: How would you detect if a developer has hardcoded a secret and pushed it to GitHub?
Three layers: (1) Pre-commit hook: GitGuardian or detect-secrets runs locally before every commit. Blocks commit if a secret pattern is detected. (2) CI pipeline: truffleHog or gitleaks scans every PR diff. PR fails if secrets are found. (3) Reactive: Amazon Macie with custom data identifiers (matching internal API key formats, JWT secrets, etc.) runs scheduled jobs on S3-hosted code archives and CodeCommit repos. All three layers together provide defense in depth — no single layer is sufficient alone.

Q5: How do you handle a secret that needs to be shared across multiple microservices (e.g., a shared internal signing key)?
I'd resist the urge to create a shared secret unless architecturally necessary. Instead, I'd evaluate whether each service truly needs the same key or whether they can use separate keys with the same trust anchor. If sharing is unavoidable (e.g., a JWT signing key used by both auth-service and gateway-service), I create a single Secrets Manager secret with a resource policy that allows GetSecretValue for both service IAM roles. This is visible in the resource policy as an explicit list of approved consumers — making it auditable. I'd also set a shorter rotation interval for shared secrets and add an extra CloudTrail alert whenever any consumer other than the two approved roles accesses it.

Q6: How would you migrate 500 microservices from hardcoded env vars to Secrets Manager without downtime?
Phase 1 (Week 1): Inventory all existing secrets using kubectl get secrets -A and code scanning. Catalog by team. Phase 2 (Weeks 2–4): Create secrets in Secrets Manager for each service. Deploy ESO. Add ExternalSecret CRD resources to ArgoCD repo. EKS pods continue using old K8s Secrets (no app code change). Phase 3 (Weeks 5–8): Flip ArgoCD to use ESO-managed K8s Secrets. Roll out service by service. Delete old manually-managed K8s Secrets. Phase 4: Enable rotation for all secrets. Enable rotation failure alerting. Block hardcoded env vars in CI.

Q7: What is your approach for TLS private key management — should they be in Secrets Manager?
TLS private keys are high-sensitivity secrets but have a specific lifecycle that makes cert-manager + ACM Private CA a better fit than Secrets Manager. I'd use cert-manager on EKS to request certificates from ACM Private CA. Certificates are stored in K8s Secrets (etcd KMS encrypted) and automatically rotated 30 days before expiry. Secrets Manager is better for credentials (DB passwords, API keys) than for PKI material. If the requirement is to keep all secrets in one place, Secrets Manager can store TLS keys, but you lose automatic certificate provisioning and rotation integration.

Q8: How do you handle secrets for batch/cron jobs that don't run continuously?
Batch jobs (AWS Batch, Kubernetes CronJob, Lambda) follow the same EKS Pod Identity or Lambda execution role pattern. For Lambda: the execution role has GetSecretValue scoped to the Lambda's secret ARN. Lambda fetches the secret at cold start and caches it for the function lifetime. For EKS CronJobs: the job pod gets a Pod Identity association; it fetches the secret at startup. The key difference from long-running services: batch jobs restart frequently, so they always fetch a fresh secret value — they're inherently rotation-compatible.
