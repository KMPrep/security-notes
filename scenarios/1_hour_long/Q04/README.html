Q04 — Multi-Account AWS Landing Zone with Security Guardrails

The Ask

"Design a secure AWS multi-account landing zone for a financial services company that
is migrating 200 applications from on-premises. The design must enforce compliance
(PCI-DSS, SOC 2), isolate environments (dev/staging/prod), prevent privilege escalation,
and give the security team centralized visibility without being a bottleneck for
development teams."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Is this a greenfield AWS Organization or are we migrating existing standalone AWS accounts into an Organization?
    Impact: Greenfield = start with Control Tower, design the OU hierarchy from scratch. Migrating existing accounts = use Control Tower Account Enrollment to bring accounts under governance, but existing IAM users/roles must be audited first and may conflict with SCPs applied on enrollment.

  • Q: How many business units are in scope, and does each need its own OU subtree with independent security configurations?
    Impact: If 5 BUs with different compliance requirements (one is PCI-DSS, one is just SOC 2), I need a nested OU structure where PCI-DSS SCPs apply only to the PCI OU, not all accounts. Flat OU structures with universal SCPs are simpler but may be too restrictive for some BUs.

  • Q: Is there an existing on-premises identity provider (Active Directory, Okta) that must be federated for AWS console access?
    Impact: If AD, I use IAM Identity Center with AD Connector or AWS Managed AD. If Okta, I use IAM Identity Center with SAML federation. This determines setup time (AD sync = days; SAML = hours) and the MFA enforcement mechanism.

  • Q: What is the developer autonomy model — do teams own individual accounts, or are multiple teams sharing accounts?
    Impact: One account per team/environment is the AWS best practice (Account Factory model). Shared accounts reduce cost but complicate blast radius isolation and IAM boundary design. For financial services, dedicated accounts per team-environment are strongly recommended.

  • Q: What are the specific PCI-DSS scope reduction requirements — which accounts are in-scope for PCI?
    Impact: Only accounts handling cardholder data (CHD) need PCI-DSS controls applied. I create a dedicated PCI OU with additional SCPs (e.g., block non-TLS 1.2 endpoints, restrict approved regions). Non-PCI accounts only need SOC 2 controls.

  • Q: What is the change management process for SCP modifications — does every SCP change require a change board approval?
    Impact: SCPs must be versioned in Git and deployed via CI/CD pipeline. A change board approval gate in the pipeline (e.g., Terraform plan reviewed by security team before apply) ensures governance without being a bottleneck.

  • Q: Are there existing AWS accounts with root access keys, non-federated IAM users, or legacy configurations that must be cleaned up as part of enrollment?
    Impact: Account enrollment into Control Tower applies guardrails immediately. If an account has root access keys or IAM users, the SCPs will not remove them — I need a remediation runbook to clean these up during enrollment.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Greenfield AWS Organization with Control Tower. Primary region: us-east-1. Secondary: us-west-2.
  2. Five business units: Payments (PCI-DSS), Healthcare (HIPAA), Analytics, Engineering, Corporate IT.
  3. Identity provider: Okta (SAML federation into IAM Identity Center).
  4. Developer autonomy model: dedicated AWS account per team per environment (dev, staging, prod).
  5. Estimated total accounts: 200 applications × 3 environments / ~5 apps per account = ~120 accounts initially, growing to 200+.
  6. Account provisioning: automated via Account Factory for Terraform (AFT) with baseline customizations.

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

IAM Identity Center permission set design (ABAC):

Permission Set: PaymentsTeam-Dev
  Policy: PowerUser minus iam:CreateRole (handled by boundary)
  Tags: Team=payments, Environment=dev
  Assignment: payments-dev account, payments-team AD group

Permission Set: SecurityTeam-ReadOnly
  Policy: SecurityAudit (AWS managed)
  Tags: Team=security, Environment=all
  Assignment: all accounts via Org-wide assignment

IAM Permission Boundaries (applied to all team-created roles):

{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "StringEquals": { "aws:RequestedRegion": ["us-east-1", "us-west-2"] },
    "StringNotEquals": {
      "aws:PrincipalArn": "arn:aws:iam::*:role/OrganizationAccountAccessRole"
    }
  }
}

The boundary denies actions outside approved regions and ensures no role can exceed the boundary itself.

Break-glass accounts:

  • Two named security engineers have break-glass IAM users (not SSO) in the Management account.
  • Hardware MFA required. Credentials stored in sealed physical envelope in the security team's safe.
  • CloudTrail alert triggers on any break-glass IAM user login.

2. Encryption (At Rest, In Transit, In Use)

  • KMS CMK per account: each workload account gets a CMK for general encryption (EBS, S3, RDS). Key managed by the Log Archive/Audit account (cross-account key policy grants usage to workload account).
  • S3 default encryption: Control Tower guardrail requires S3DefaultEncryption — Config rule enforces SSE-KMS on all S3 buckets.
  • TLS enforcement SCP: deny any API call that creates an unencrypted resource (e.g., rds:CreateDBInstance without StorageEncrypted=true; s3:CreateBucket without bucket encryption configuration).
  • In transit: aws:SecureTransport condition applied to all S3 bucket policies via AFT baseline module. ALB listeners terminate TLS 1.2+; HTTP redirects to HTTPS enforced.

3. SCPs — The Hardest Controls to Get Right

Preventive SCPs applied at Root / all OUs:

- Deny iam:CreateUser (force SSO; no local IAM users except break-glass)
- Deny ec2:AuthorizeSecurityGroupIngress with 0.0.0.0/0 to port 22 or 3389
- Deny cloudtrail:StopLogging, cloudtrail:DeleteTrail
- Deny s3:DeleteBucket on Log Archive buckets
- Deny guardduty:DeleteDetector, guardduty:DisassociateFromMasterAccount
- Deny securityhub:DisableSecurityHub
- Deny config:StopConfigurationRecorder, config:DeleteConfigurationRecorder
- Deny AWS service actions outside approved regions (eu-*, ap-*) unless explicitly approved
- Deny iam:CreatePolicyVersion with Effect=Allow Action=* Resource=*

PCI OU additional SCPs:

- Deny any API call creating resources without encryption (RDS, EBS, S3)
- Deny access from non-approved IP ranges (corporate and data center CIDRs only)
- Deny ec2:CreateVpc without VPC Flow Logs enabled (enforced via Config + EventBridge remediation)

SCP design principles:

  • Start with Deny SCPs at root; use Allow lists only where necessary.
  • Never apply Allow * SCPs — they are misunderstood (SCPs don't grant permissions; they only restrict).
  • Test every SCP in CountMode equivalent (use IAM simulator) before applying to production OU.
  • Store all SCPs in Git; apply via Terraform only. No manual SCP changes.

5. Logging and Audit Trail

  • CloudTrail Organization Trail: in Management account, trails all member accounts, writes to Log Archive S3 bucket with Object Lock (Compliance mode, 7-year retention).
  • Config Organization Aggregator: in Audit account; collects compliance data from all accounts. Single dashboard for all Config findings.
  • Security Hub Aggregation: Audit account is delegated admin. All accounts send findings to Security Hub in Audit account. Finding suppression requires documented justification.
  • VPC Flow Logs: required in all accounts by Config rule vpc-flow-logs-enabled. Logs to CloudWatch → Kinesis Firehose → Log Archive S3.
  • EventBridge cross-account event bus: workload accounts forward security events to Audit account's event bus → SIEM integration.

Scale and Resilience (Phase 4 — 10 min)

Account Factory for Terraform (AFT) at scale:

  • AFT runs on a dedicated CodePipeline in the Management account. New account provisioning is a Git commit (YAML manifest) → automatic account creation in ~20 minutes.
  • AFT customizations: two types — (1) account-level customizations applied to every account (baseline VPC, GuardDuty enrollment, Security Hub enrollment, Config recorder, CloudTrail) and (2) OU-level customizations (PCI accounts get extra Config rules, stricter SCP).
  • Drift detection: Config Organization Aggregator detects any drift from baseline. EventBridge rule triggers SSM Automation to remediate automatically for non-production accounts; creates SOAR ticket for prod accounts.

Scaling to 200+ accounts:

  • Config Aggregator handles 200+ accounts without performance degradation.
  • Security Hub supports 1,000+ member accounts per organization.
  • IAM Identity Center SCIM sync with Okta: user de-provisioning within 1 hour of HR system update.
  • Control Tower Account Factory: supports bulk account creation via API; AFT queues requests if CodePipeline is busy.

Permission set change management:

  • IAM Identity Center permission sets stored in Git.
  • Change requires PR with security team review.
  • CodePipeline applies changes with aws sso-admin update-permission-set.
  • Automated test: after permission set update, run IAM policy simulator to verify expected allow/deny behavior.

Trade-off Analysis — The Hard Choices

Decision 1: Control Tower vs. Custom Landing Zone (DIY)

Factor AWS Control Tower Custom Landing Zone
Time to production Days Months
Flexibility Opinionated; limited customization of foundational accounts Full control
Updates AWS manages Control Tower updates You manage everything
AFT integration Native Requires significant custom code
Recommendation Control Tower + AFT for most organizations. Custom only if Control Tower's opinionated structure conflicts with existing complex governance requirements

Decision 2: Shared Account vs. Dedicated Account per Team

Factor Shared Account Dedicated Account per Team
Cost Lower (fewer accounts) Higher (more accounts, each with baseline cost)
Blast radius Large — one account compromise affects all teams Small — limited to one team
IAM complexity Very high (namespacing, per-team IAM policies) Simple (each team owns their account)
Compliance Harder to prove per-team compliance controls Easier — one compliance scope per account
Recommendation Dedicated account per team per environment for financial services. Shared only for sandbox/dev cost optimization

Decision 3: Deny SCPs vs. Allow SCPs

Factor Deny SCPs Allow SCPs
Intent Block specific dangerous actions Whitelist allowed services
Risk May miss new AWS services (unknown unknowns) Service additions blocked until SCP updated
Operational overhead Lower — only add denies for new threats High — every new AWS service needs SCP update
PCI-DSS fit Allow SCPs easier to scope PCI perimeter
Recommendation Deny SCPs at root for dangerous cross-cutting actions. Allow SCPs only for PCI OU (restrict to approved services only)

Decision 4: Config Auto-Remediation vs. Manual Remediation

Factor Auto-Remediation Manual Remediation
Speed Seconds Hours to days
Risk May remediate intended configurations Human review catches false positives
Audit trail SSM Automation document logged Human action logged
Recommendation Auto-remediate for clear-cut non-compliance (public S3 bucket, unencrypted RDS in dev). Manual remediation for ambiguous findings or production accounts where auto-changes could cause outages

Common Mistakes to Avoid

  1. Confusing SCPs with IAM policies. SCPs don't grant permissions — they can only restrict what IAM policies can allow. An SCP Allow * does not grant any access. This is one of the most common AWS mistakes.
  2. Not having a Log Archive account. Storing CloudTrail logs in each member account defeats the purpose of centralized logging — a compromised member account can delete its own logs. Log Archive must be a separate, locked-down account.
  3. Applying SCPs without testing. A bad SCP deployed to 200 accounts can break all developer access instantly. Always test in a sandbox OU first, use the IAM policy simulator, and deploy gradually (one OU at a time).
  4. Forgetting the Management account is the crown jewel. The Management account controls all SCPs, all organizational settings. It must have the most restrictive access controls of any account in the organization. Many candidates spend time on workload account security and barely mention Management account protection.
  5. Not explaining developer autonomy. The question specifically says "without being a bottleneck." If your answer requires developers to file tickets for every IAM policy change, you've failed this requirement. AFT + IAM Identity Center permission sets + self-service in dev accounts = no ticket required.

Likely Follow-up Questions

Q1: How do you handle an SCP that accidentally blocks a critical production service?
SCPs take effect immediately on attachment. I maintain a break-glass procedure: the Management account admin (hardware MFA required) can temporarily detach the problematic SCP from the affected OU, restore service, then fix and reapply. All SCP changes are CloudTrail-logged. The fix must go through the normal Terraform PR process — no manual SCP changes in non-emergency situations. After the incident, I add a test case to the SCP testing suite to prevent regression.

Q2: How does Control Tower handle accounts that were created before the Organization existed?
Control Tower's Account Enrollment feature brings existing accounts under Control Tower governance. The process: (1) invite account to Organization, (2) account accepts, (3) run Control Tower's account enrollment script which applies guardrails, enables Config, creates the necessary IAM roles. The risk: existing IAM users/roles may conflict with SCPs. I'd run a pre-enrollment audit with IAM Access Analyzer and Config to identify conflicts, then remediate before enrollment.

Q3: What happens to the landing zone if the Management account is compromised?
The Management account controls SCPs and organizational settings. A compromised Management account means an attacker can remove SCPs, create new accounts outside the security perimeter, and disable GuardDuty org-wide. Mitigation: (1) 5-person access limit with hardware MFA, (2) CloudTrail Organization Trail writes to Log Archive (different account — attacker can't delete evidence), (3) AWS Organizations CloudTrail events generate P0 alerts, (4) Contact AWS Support to lock the Organization if Management account is confirmed compromised.

Q4: How do you enforce least privilege for 2,000 developers across 200+ accounts without manual IAM policy review for every deployment?
Three controls: (1) IAM Permission Boundaries on all developer-created roles — the boundary defines the maximum permissions available (e.g., can only interact with resources in their own account and region). (2) IAM Access Analyzer continuously reviews all roles and alerts on overly permissive policies. (3) AFT baseline module applies a standard IAM policy template per team-environment combination; developers customize within that template via PR. This means most developers never need to touch IAM directly.

Q5: How do you handle the Sandbox OU — what controls prevent sandbox accounts from being used for data exfiltration?
Sandbox accounts have strict outbound controls: (1) SCP denies s3:GetObject from any S3 bucket not in the sandbox OU. (2) SCP denies all Direct Connect and VPN gateway creation — no path to on-premises. (3) Config rule: no production data-classification tags allowed in sandbox resources. (4) Network: sandbox accounts have no Transit Gateway attachment — fully isolated network. (5) Data controls: Macie enabled in sandbox accounts to detect if PHI/PII accidentally appears in sandbox data.

Q6: How do you give the security team read-only access across 200 accounts without managing individual role assumptions?
IAM Identity Center + Organization-level permission set assignment. I create a SecurityAuditRole permission set (attached to the SecurityAudit AWS managed policy). Assign this permission set to the security team's AD group across all accounts in the organization via a single IAM Identity Center assignment. The security team clicks on any account in the Identity Center portal and assumes the read-only role with a single click — no manual role chaining, no trusts to configure per account.

Q7: What is your strategy for managing Config rules at scale — 200 accounts × 150 rules = 30,000 rule evaluations?
AWS Config Conformance Packs deployed from the Audit account via Organizations integration. A single conformance pack (e.g., OperationalBestPracticesForPCI-DSS) deploys 50+ Config rules to all 200 accounts simultaneously. Config aggregation in the Audit account collects all results. I don't manage individual rules per account — I manage conformance packs. For custom rules, I use Config Organization custom rules (Lambda-backed) deployed once from the Audit account. Rule evaluation is event-driven (on resource change) and periodic (every 24 hours), so it scales with the number of changes, not the number of accounts.

Q8: How do you handle a developer who needs temporary elevated access to debug a production issue?
JIT access via IAM Identity Center time-bound assignments. Workflow: (1) Developer submits request in Service Catalog → Step Functions approval workflow → manager approves in ServiceNow. (2) Lambda creates a time-bound IAM Identity Center assignment for the developer with an elevated permission set (e.g., PowerUserAccess in the affected account). (3) TTL: 4 hours. (4) EventBridge schedule revokes the assignment at TTL expiry. (5) All actions during the JIT session are logged to CloudTrail with the developer's identity. (6) Post-incident: security team reviews CloudTrail for any non-incident-related actions during the elevated session.
