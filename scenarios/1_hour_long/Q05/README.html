Q05 — Secure Data Lake for PII at Petabyte Scale

The Ask

"Design a secure data lake on AWS for a healthcare company that needs to store and
analyze petabytes of data containing PII and PHI (Protected Health Information).
The system must enforce column-level and row-level access controls, detect data
leakage, satisfy HIPAA requirements, and allow analysts to run queries without ever
accessing raw PII."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What is the data ingestion model — streaming (real-time from EHR systems), batch (nightly ETL from on-premises), or both?
    Impact: Streaming requires Kinesis Data Firehose + Glue Streaming ETL. Batch uses AWS Transfer Family or S3 multipart upload. Both paths need encryption in transit and PII detection before landing in the curated zone.

  • Q: Who are the data consumers — data scientists running Jupyter notebooks, analysts using SQL tools like Athena, or BI tools like QuickSight?
    Impact: This determines whether we use Lake Formation fine-grained access (Athena + EMR) or row-level security in Redshift. For Jupyter, we need SageMaker Studio + Glue DataBrew. For BI, we integrate QuickSight with Lake Formation column masks.

  • Q: What is the PII/PHI anonymization strategy — tokenization, pseudonymization, or differential privacy for analytics?
    Impact: Tokenization preserves format (e.g., SSN → same-length token) and is reversible by privileged services. Pseudonymization replaces with consistent fake values. Differential privacy adds noise and is irreversible. For HIPAA Safe Harbor, analysts can use de-identified data without a BAA.

  • Q: What are the access tiers — which teams need raw PHI vs. anonymized data vs. aggregate-only access?
    Impact: Lake Formation permission model: (1) PHI Admins → full raw data, (2) Analysts → column-masked SSN/DOB, row-filtered by their region, (3) Data Scientists → pseudonymized training datasets, (4) External partners → pre-computed aggregates only.

  • Q: Is there a requirement for data lineage tracking — proving exactly which records were accessed by whom for HIPAA audit?
    Impact: AWS Glue Data Catalog + S3 Object Access Logs + CloudTrail Data Events. Lake Formation audit logging provides fine-grained access logs at the column/row level. For lineage, integrate with Apache Atlas or use AWS Glue's built-in lineage tracking.

  • Q: What is the data residency requirement — US-only, or does PHI need to stay in specific regions?
    Impact: S3 Replication Rules + SCP to deny S3 PutObject to non-approved regions. Config rule s3-bucket-replication-enabled verifies cross-region replication isn't pointing to non-compliant regions.

  • Q: What is the expected query concurrency — 10 analysts or 1,000?
    Impact: 10 analysts → Athena (serverless, no capacity planning). 1,000 concurrent → Redshift Serverless with Lake Formation row-level security. Petabyte scale → separate storage (S3) from compute (Athena/Redshift), use columnar formats (Parquet + Snappy compression).

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Data sources: EHR system (HL7 FHIR API), claims processing system (batch CSV), IoT medical devices (streaming).
  2. Petabyte scale: ~5 PB total, growing 500 TB/year.
  3. Users: 50 PHI admins, 200 analysts, 20 data scientists, 5 external partners.
  4. Compliance: HIPAA, HITRUST, SOC 2. Business Associates Agreements (BAAs) signed with AWS.
  5. Regions: us-east-1 primary, us-west-2 DR. No cross-border data transfer.

Core Architecture — Data Flow Narrative

Data Sources → Ingestion Layer → Landing Zone → Processing → Curated Zones → Consumption

[EHR / Claims / IoT]
        ↓
[Kinesis Data Firehose / AWS Transfer Family SFTP]
        ↓  (encrypt in transit: HTTPS/TLS 1.2+)
[S3 Landing Zone — raw encrypted]
  Bucket: s3://datalake-landing-{account}
  KMS CMK: alias/datalake-landing
  No public access, no S3 ACLs
        ↓
[Amazon Macie — PII/PHI scan on landing]
  Triggers: SNS alert + Step Functions remediation if unexpected PII found in wrong bucket
        ↓
[AWS Glue ETL — PII detection + masking + cataloging]
  - Detect PII fields (SSN, MRN, DOB) using Glue built-in transforms
  - Tokenize SSN → call AWS KMS GenerateDataKey + DynamoDB token store
  - Write Parquet to curated zones
  - Update Glue Data Catalog (with Lake Formation column sensitivity tags)
        ↓
[S3 Curated Zones — 3 buckets]
  PHI Zone:     raw PHI, Parquet, KMS-CMK alias/datalake-phi
  Masked Zone:  SSN/DOB masked, analysts use this
  Aggregated:   pre-computed aggregates, partners use this
        ↓
[AWS Lake Formation — Table-level + Column-level + Row-level permissions]
  Column masks: SSN → show last 4 only for analysts
  Row filters: WHERE patient_state = analyst's assigned region
        ↓
[Athena / Redshift Serverless / SageMaker Studio / QuickSight]
  All queries go through Lake Formation; no direct S3 GetObject for analysts

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Lake Formation TBAC (Tag-Based Access Control):

LF-Tags:
  Sensitivity = PHI | PII | Public
  DataDomain  = Claims | EHR | Device
  Region      = northeast | southeast | midwest

Access grants by tag:
  PHI Admins role:         Sensitivity=PHI — Full access (SELECT, INSERT, ALTER)
  Analysts role:           Sensitivity=PII — SELECT only, column masks applied
  Data Scientists role:    Sensitivity=PII — SELECT on pseudonymized tables only
  External Partners role:  Sensitivity=Public — SELECT on aggregated tables only

IAM roles for data pipeline:

GlueETLRole:
  - s3:GetObject on landing bucket
  - s3:PutObject on curated buckets
  - kms:Decrypt (landing CMK), kms:GenerateDataKey (phi CMK, masked CMK)
  - glue:UpdateTable (catalog write)
  - dynamodb:PutItem (token store)

AthenaQueryRole:
  - lakeformation:GetDataAccess (required for column masking to work)
  - s3:GetObject on query results bucket
  - kms:Decrypt (masked CMK only — NOT phi CMK)
  - athena:StartQueryExecution

2. Encryption (At Rest, In Transit, In Use)

KMS CMK hierarchy:

alias/datalake-landing    → landing zone (short TTL, rotated annually)
alias/datalake-phi        → raw PHI zone (most restrictive key policy)
alias/datalake-masked     → masked/anonymized zone
alias/datalake-aggregate  → partner zone (less restrictive)
alias/datalake-tokens     → tokenization keys (used by token service only)

PHI CMK key policy (restrictive):

{
  "Effect": "Allow",
  "Principal": { "AWS": "arn:aws:iam::ACCOUNT:role/GlueETLRole" },
  "Action": ["kms:GenerateDataKey", "kms:Decrypt"],
  "Resource": "*",
  "Condition": {
    "StringEquals": { "kms:ViaService": "s3.us-east-1.amazonaws.com" }
  }
}

Analysts' Athena role has NO access to alias/datalake-phi CMK — they cannot decrypt raw PHI even if Lake Formation somehow granted access.

Tokenization design:

  • SSN 123-45-6789 → call TokenService Lambda → generate HMAC-SHA256 using KMS-protected key → store token=T8f2a..., ssn=encrypted(123-45-6789), created_at in DynamoDB token store.
  • Only the PHIAdminRole can call TokenService.detokenize(). Analysts only ever see tokens.

3. Network Isolation

  • VPC Endpoints for all AWS services: Glue, Athena, S3, KMS, Lake Formation, SageMaker — all via VPC Interface Endpoints. No internet traffic from data processing components.
  • S3 bucket policy with aws:sourceVpce condition: buckets only accept requests from the VPC endpoint (vpce-xxx). Direct internet access to PHI buckets is impossible.
  • Glue connection VPC: ETL jobs run inside a Glue VPC connection to the data lake VPC. No Glue public endpoint exposure.
  • Athena VPC access: analysts access Athena from within VPC; Athena results bucket restricted to internal VPC endpoint.
  • SageMaker Studio VPC-only mode: no direct internet access from notebooks; data access via VPC endpoint to Lake Formation/S3.

4. Logging and Audit Trail

  • CloudTrail Data Events: S3 data events enabled on all PHI zone buckets. Every GetObject, PutObject, DeleteObject logged with userIdentity, sourceIPAddress, requestParameters.
  • Lake Formation audit logs: every table-level SELECT, column access, and row filter applied is logged. Log group: /aws/lakeformation/audit.
  • Glue Data Quality: data quality rules check that PII fields are properly masked before write to masked zone. If a rule fails, the Glue job fails and triggers an SNS alert.
  • Macie findings: all findings sent to Security Hub and EventBridge. Automated response: if Macie finds unexpected PHI in the aggregated zone, Lambda quarantines the object (moves to quarantine bucket, removes from catalog).
  • S3 Access Logs: bucket-level access logs (distinct from CloudTrail) provide HTTP-level access audit, useful for forensic investigation.

5. Least Privilege and Blast Radius

  • Lake Formation, not bucket policies, for access control: analysts have zero direct IAM permissions on PHI S3 buckets. They get access only through Lake Formation, which enforces column masks and row filters before returning data.
  • Data Catalog separation: each OU has its own Glue Data Catalog. PHI tables are only cataloged in the PHI catalog; analysts can only see the masked catalog.
  • Cross-account access via Lake Formation: external partner accounts are granted access to specific Lake Formation databases/tables only. No S3 bucket policies granting partner accounts direct access.
  • Glue job isolation: each Glue job runs with its own IAM role scoped to the specific source and destination buckets for that job. No shared "super-ETL" roles.

6. Compliance Mapping (HIPAA)

HIPAA Safeguard Control
Technical — Access Control (§164.312a1) Lake Formation TBAC, IAM roles, no direct S3 access for analysts
Technical — Audit Controls (§164.312b) CloudTrail Data Events, Lake Formation audit logs
Technical — Integrity (§164.312c1) S3 Object Integrity (checksum), Glue Data Quality rules
Technical — Person Authentication (§164.312d) IAM Identity Center + MFA for all analyst console access
Technical — Transmission Security (§164.312e2ii) TLS 1.2+ on all Kinesis/Transfer Family ingestion; VPC Endpoints
Physical — Workstation Use (§164.310c) SageMaker Studio VPC-only; analysts cannot copy PHI to local machines
Administrative — Risk Analysis Macie continuous scanning; Security Hub HIPAA findings

Scale and Resilience (Phase 4 — 10 min)

Petabyte-scale storage optimization:

  • S3 Intelligent Tiering: moves objects not accessed for 30 days to Infrequent Access tier automatically, 90 days to Archive Instant. For PHI data accessed rarely, this reduces costs by 40%.
  • Parquet + Snappy compression: reduces storage 60-70% vs. raw CSV. Columnar format also means Athena scans only queried columns, reducing cost per query.
  • S3 Lifecycle Rules: move raw landing zone data to S3 Glacier Instant Retrieval after 90 days (raw data rarely re-processed after Glue ETL completes).

Athena query optimization:

  • Partitioning by year/month/day/patient_region: Athena partition pruning reduces scanned data by 90%+.
  • AWS Glue partition indexing: enables sub-second partition discovery on petabyte-scale catalogs.
  • Workgroup quotas: each analyst team has an Athena workgroup with data-scanned-per-query limits and data-scanned-per-day limits to prevent runaway queries and cost.

Cross-region DR:

  • S3 Cross-Region Replication to us-west-2. Replication rule scoped to PHI zone only. Destination bucket has same KMS CMK policy, same Lake Formation permissions.
  • KMS Multi-Region Keys: PHI CMK replicated to us-west-2 using KMS MRK API. Analysts in the DR region can query without key cross-region latency.
  • Glue Data Catalog: replicated manually (not automatic) — AFT customization module runs a nightly Lambda to sync catalog definitions to us-west-2.

Common Mistakes to Avoid

  1. Putting row-level security in application code instead of Lake Formation. Analysts query Athena directly or via Jupyter. There's no guaranteed application layer. Lake Formation row filters are enforced at the query engine level — they cannot be bypassed by clever query construction.
  2. Not explaining that IAM policies alone cannot do column masking. IAM can only control access at the table level. "I'll just give analysts an IAM policy that allows glue:GetTable but not the PHI columns" doesn't work — IAM doesn't understand column-level permissions. Lake Formation is the only AWS-native solution.
  3. Forgetting CloudTrail Data Events (they're off by default). Standard CloudTrail captures management events (create bucket, delete role). You must explicitly enable Data Events on PHI S3 buckets to get per-file access logging. Many candidates say "CloudTrail logs all S3 access" without knowing Data Events must be enabled separately.
  4. Not mentioning the BAA requirement. For HIPAA, AWS requires a signed Business Associate Agreement before you can store PHI on any AWS service. Mentioning this demonstrates operational awareness.
  5. Treating petabyte scale as just "use S3." At petabyte scale, partition design, columnar formats, and compression ratios are critical for both query performance and cost. An unpartitioned Parquet table at petabyte scale can cost $5,000 per Athena query scan.

Likely Follow-up Questions

Q1: How does Lake Formation column masking work technically — can an analyst still see the masked value?
Lake Formation column masking works at query execution time. When an analyst runs SELECT ssn FROM patients, Lake Formation intercepts the request via the GetDataAccess API call from Athena and returns a modified schema where the ssn column has a masking expression applied (e.g., CONCAT('***-**-', RIGHT(ssn, 4))). The analyst sees ***-**-6789. The raw value never leaves the encrypted S3 object — the masking happens in the Athena query engine after Lake Formation applies the transform. The analyst cannot reverse this without PHI Admin privileges.

Q2: How do you prevent a data scientist from training a model on raw PHI data?
Three controls: (1) Lake Formation only grants data scientists access to the pseudonymized catalog, not the PHI catalog. Even if they run code that constructs an Athena query against the PHI table, they'll receive an AccessDenied from Lake Formation. (2) SageMaker Studio is deployed in VPC-only mode — notebooks cannot access the PHI S3 bucket directly because the S3 bucket policy includes aws:sourceVpce condition pointing to the data lake VPC endpoint, not the SageMaker VPC. (3) SageMaker lifecycle scripts block installation of packages (like boto3 direct S3 access) that aren't on the approved list.

Q3: How does Macie know what counts as PHI in healthcare data vs. general PII?
Macie has built-in managed data identifiers for HIPAA-defined PHI including: full name + date + geographic data, health plan beneficiary numbers, medical record numbers, device identifiers, URLs, and IP addresses associated with individuals. For custom PHI (e.g., proprietary patient ID formats), I create Custom Data Identifiers in Macie using regex patterns. I also configure sensitivity scoring so that medical_record_number findings trigger Macie policy findings at CRITICAL severity vs. LOW for generic PII.

Q4: Describe how a new analyst gets access to query data — what's the end-to-end workflow?
(1) Analyst's manager submits a Lake Formation access request in Service Catalog, specifying data domain (e.g., Claims) and sensitivity level (PII-masked only). (2) Data governance team reviews and approves in ServiceNow. (3) Automated Lambda grants the analyst's IAM Identity Center permission set (mapped to their Athena IAM role) a Lake Formation SELECT permission with the appropriate column masks and row filters. (4) Analyst opens Athena in the AWS console, queries the masked claims table. Their session shows masked SSNs and only rows for their assigned region. Total time: 2 business days. No ticket to the infrastructure team required.

Q5: How do you handle the case where a HIPAA audit requires you to prove exactly who accessed which PHI records?
CloudTrail Data Events on PHI S3 buckets log every GetObject with the Athena query execution ID, principal ARN, timestamp, and S3 object key. Lake Formation audit logs provide the Glue catalog table, column list, and row filter applied. I correlate these two logs in Security Lake (using OCSF schema) to reconstruct: "Analyst X ran query Y at time T, which accessed rows in files F1, F2, F3 of the PHI table, and saw columns [patient_id, diagnosis_code, treatment_date]." This is the "accounting of disclosures" required by HIPAA §164.528.

Q6: What happens if a developer accidentally deploys a public S3 bucket in the data lake account?
The S3 Block Public Access setting is enforced at the account level (not just bucket level) as part of the AFT baseline customization — no bucket can be made public regardless of bucket policy. Additionally, Config rule s3-bucket-public-read-prohibited runs continuously and would immediately flag any violation. The SCP denies s3:PutBucketAcl with CannedACL=public-read. Multiple independent controls prevent this, so even if one fails, the others catch it.

Q7: How does cross-account data sharing work for the external partners?
Lake Formation Cross-Account Grants. The data lake account's Lake Formation admin grants the partner account's IAM principal SELECT on specific tables in the aggregated database (not PHI, not masked). The partner account's data analyst assumes a role that has been granted these Lake Formation permissions via Resource Links. All queries still execute in the data lake account's Athena — the partner never gets direct S3 access. CloudTrail logs all partner query activity with their account ID and role ARN.

Q8: How do you handle encryption key rotation for petabytes of already-encrypted PHI data?
AWS KMS key rotation (annual automatic rotation) updates the CMK's backing key material but does not re-encrypt existing S3 data. Existing encrypted objects remain readable because KMS stores all historical backing key versions and uses the correct version automatically. If I need to explicitly re-encrypt (e.g., after a suspected key compromise), I use S3 Batch Operations with a Batch Operations Copy job that specifies the new CMK — this creates new copies with the new key material. For 5 PB, this Batch Operations job takes approximately 6-12 hours. The old CMK is disabled (not deleted) until all objects are confirmed re-encrypted.
