Q06 — Runtime Security and Threat Detection for EKS at Scale

The Ask

"Design a comprehensive runtime security and threat detection platform for a company
running 50 EKS clusters across 10 AWS accounts, with 5,000 pods running at peak.
The system must detect container escapes, privilege escalation, cryptomining, lateral
movement between pods, and malicious network connections in real time. Compliance
requires SOC 2 and CIS Kubernetes Benchmark Level 2."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Are these EKS clusters on EC2 node groups, Fargate, or mixed?
    Impact: Fargate does not support daemonsets, so Falco (which runs as a daemonset) cannot be deployed. Fargate runtime security is limited to GuardDuty EKS Protection + VPC Flow Logs. For EC2 nodes, we have the full toolset: Falco daemonset, eBPF probes, GuardDuty Runtime Monitoring agent. This changes the architecture significantly.

  • Q: What is the current logging posture — do clusters have EKS control plane logging enabled (audit, authenticator, api, scheduler, controllerManager)?
    Impact: If not enabled, GuardDuty EKS Protection cannot detect IAM misuse in the Kubernetes API (it relies on EKS audit logs). Enabling control plane logs has no performance impact but does increase CloudWatch costs. This is non-negotiable for SOC 2.

  • Q: What is the expected false-positive tolerance — should the system auto-block detected threats, or just alert?
    Impact: Auto-block (e.g., isolate a pod via NetworkPolicy update when Falco detects a privilege escalation) risks disrupting production workloads if the rule fires incorrectly. Alert-only is safer initially. I recommend alert-only for 30 days during tuning, then enable auto-response for high-confidence rules.

  • Q: Is there an existing SIEM or alerting platform (Splunk, Datadog, PagerDuty)?
    Impact: Falco alerts and GuardDuty findings need to be forwarded. If Splunk exists, I route events via Kinesis Firehose → Splunk HEC. If no SIEM, Security Lake (OpenSearch) handles aggregation. This determines the integration architecture.

  • Q: What is the container image supply chain — do teams build their own images or use approved base images from a central registry?
    Impact: Central registry (ECR) enables Inspector v2 to scan all images continuously. If teams use Docker Hub or arbitrary base images, I need to enforce image admission via OPA Gatekeeper + ImagePolicyWebhook before containers are allowed to run.

  • Q: What Kubernetes admission control is currently in place — OPA Gatekeeper, Kyverno, or native Pod Security Admission?
    Impact: Pod Security Admission (PSA) is built into Kubernetes 1.25+ and is the simplest starting point (enforce restricted profile). Kyverno is more flexible for custom policies. If Kyverno is already deployed, I build on it; otherwise I recommend PSA for CIS Benchmark compliance and Kyverno for custom policies.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. 50 EKS clusters, 40 on EC2 managed node groups + 10 Fargate-only for batch workloads.
  2. Peak 5,000 pods; average 3 containers per pod = 15,000 containers.
  3. EKS version 1.29+; Pod Security Admission enabled; Kyverno deployed.
  4. Central ECR registry in the Tools account; images signed with AWS Signer.
  5. AlertManager → PagerDuty for on-call. No existing SIEM; will deploy Security Lake.
  6. EKS control plane logging enabled (audit, authenticator, api).

Core Architecture — Data Flow Narrative

Layer 1 — Prevention (Shift Left)
  ECR image push → Inspector v2 scan → Critical CVE found → block promotion to prod tag
  Git commit → Checkov IaC scan (Terraform, Helm charts) → PR gate
  Kubernetes admission → Kyverno + PSA (restricted) → block privileged containers
  AWS Signer → image signature verification → Kyverno ImagePolicyWebhook blocks unsigned images

Layer 2 — Runtime Detection (In-Cluster)
  Node-level:
    Falco daemonset (eBPF probe, no kernel module required on EKS 1.26+)
    → Rules: container_escape, privilege_escalation, cryptomining, shell_in_container
    → Alert → EventBridge → Step Functions SOAR
    → Alert → Security Lake (OCSF format)

  AWS-native:
    GuardDuty EKS Protection
    → EKS audit log analysis: UnauthorizedAPICall, Kubernetes IAM credential exfiltration
    → EKS Runtime Monitoring agent (EC2 nodes only): eBPF-based syscall monitoring
    → Findings → Security Hub → EventBridge

  Network:
    VPC Flow Logs → GuardDuty: unusual pod-to-pod lateral movement, C2 callbacks
    Kubernetes NetworkPolicy (Cilium): default-deny all; explicit allow per microservice
    Falco: unexpected outbound connections from containers

Layer 3 — Response (Automated)
  Falco alert (HIGH severity) → EventBridge → Lambda → update Kubernetes NetworkPolicy
    to isolate the pod (add label quarantine=true → Cilium network policy blocks all traffic)
  GuardDuty finding (EKS:CryptoCurrencyFound) → Step Functions:
    1. Capture pod metadata (kubectl describe)
    2. Preserve evidence (copy pod filesystem to S3)
    3. Terminate pod (kubectl delete pod)
    4. Notify PagerDuty P1

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

EKS Pod Identity (not IRSA for new clusters) for fine-grained AWS access:

# EKS Pod Identity association
apiVersion: eks.amazonaws.com/v1alpha1
kind: PodIdentityAssociation
metadata:
  name: payment-service-identity
spec:
  namespace: payments
  serviceAccountName: payment-service-sa
  roleArn: arn:aws:iam::ACCOUNT:role/payments-prod-payment-service

Pod Identity uses STS AssumeRoleWithWebIdentity per pod; no cluster-wide OIDC configuration required.

IAM roles for node groups:

NodeGroup IAM Role (least privilege):
  - ec2:DescribeInstances (self)
  - ecr:GetAuthorizationToken, ecr:BatchGetImage (ECR pull)
  - eks:DescribeCluster (self)
  - ssm:GetParameter (bootstrap configs only)
  NO s3:*, dynamodb:*, secretsmanager:* — all app access via Pod Identity

Kubernetes RBAC:

# Minimal RBAC for application pods
ClusterRole: read-only-pods
  rules: [{apiGroups: [""], resources: ["pods"], verbs: ["get", "list"]}]

# No ClusterAdmin bindings except for CI/CD service account
# EKS access entries: map IAM roles to Kubernetes groups

EKS Access Entries (replaces aws-auth ConfigMap, less vulnerable to misconfiguration):

IAM Role: SecurityTeam → Kubernetes group: security:readonly
IAM Role: DevTeam-Payments → Kubernetes group: dev:payments:edit (scoped to payments namespace)
IAM Role: CI/CD pipeline → Kubernetes group: cicd:deploy (can create Deployments, Services in prod namespaces)

2. Image Security (Supply Chain)

Image signing pipeline:

Developer pushes code → CodePipeline builds image → ECR push
→ AWS Signer signs image (cosign-compatible signature in ECR)
→ Inspector v2 scan (Critical CVE → pipeline fails, image not promoted)
→ On pass: image tagged :staging, :prod (only signed images)

Admission control (Kyverno policy):
  Rule: Require signatures
  - Image must have valid AWS Signer signature
  - Image must be from approved ECR registry
  - Image must not have Critical CVEs (Inspector finding attached to ECR tag)

ECR image vulnerability policy:

{
  "rules": [
    {
      "repositoryFilters": [
        { "filter": "prod/*", "filterType": "PREFIX_MATCH" }
      ],
      "severity": ["CRITICAL"],
      "action": { "type": "BLOCK" }
    }
  ]
}

This blocks docker pull of images with Critical vulnerabilities in prod repos.

3. Pod Security and Admission Control

Pod Security Admission (restricted profile) enforces:

  • No privileged containers
  • No hostPID, hostIPC, hostNetwork
  • No hostPath volumes (except approved PVCs)
  • Run as non-root user (UID > 1000)
  • Read-only root filesystem
  • Dropped capabilities: ALL; add back only needed: NET_BIND_SERVICE

Kyverno custom policies (CIS Benchmark L2):

# Deny latest tag (force pinned images)
- name: deny-latest-tag
  match: any pod with image tag "latest" → DENY

# Require resource limits
- name: require-resource-limits
  match: any pod without CPU/memory limits → DENY

# Deny NodePort services (force internal access via Ingress only)
- name: deny-nodeport
  match: Service.type = NodePort → DENY

4. Falco Runtime Rules

Key Falco rules deployed:

# Container escape via privileged exec
- rule: Privileged Shell in Container
  condition: spawned_process and container and shell_procs and proc.name in (sh, bash) and user.uid=0
  output: "Privileged shell in container (user=%user.name container=%container.name image=%container.image)"
  priority: CRITICAL

# Unexpected outbound connection (potential C2)
- rule: Unexpected Outbound Network
  condition: outbound and not fd.sip in (allowed_cidrs) and container
  output: "Unexpected outbound connection from container (ip=%fd.sip container=%container.name)"
  priority: HIGH

# Write to /etc/passwd equivalent
- rule: Write Below /etc
  condition: open_write and fd.name startswith /etc and container
  output: "Write to sensitive directory (file=%fd.name container=%container.name)"
  priority: HIGH

# Cryptomining detection
- rule: Crypto Miner Process
  condition: spawned_process and proc.name in (xmrig, minerd, cpuminer, kryptex)
  output: "Cryptominer process detected (proc=%proc.name container=%container.name)"
  priority: CRITICAL

Network segmentation with Cilium:

Scale and Resilience (Phase 4 — 10 min)

Falco at scale (5,000 pods across 50 clusters):

  • Falco deployed as DaemonSet via EKS addon. One Falco pod per node (not per pod — kernel-level monitoring covers all pods on the node).
  • Falco forwards events via falco-sidekick → Kinesis Data Streams → Security Lake (OCSF format) and EventBridge.
  • Rule tuning: Falco rules managed centrally in Git. Applied via Helm chart across all clusters. Each cluster runs the same baseline rules + OU-specific rules (dev clusters have permissive rules; prod clusters have strict rules).
  • High-cardinality Falco alert volume: at 5,000 pods, Falco can generate thousands of alerts/minute. Use Falco alert sampling (rate limiting per rule) and deduplication in falco-sidekick before forwarding to Security Lake.

GuardDuty EKS Runtime Monitoring:

  • EKS addon auto-updates the Runtime Monitoring agent. Agent is lightweight (< 1% CPU overhead per AWS measurement).
  • GuardDuty Runtime Monitoring covers 50 clusters from the central GuardDuty delegated admin account. Single pane of glass.
  • GuardDuty findings deduplication: suppress duplicate findings of same type per cluster per hour using Security Hub suppression rules.

Inspector v2 at scale:

  • Inspector v2 continuously monitors 50 ECR repositories. On new CVE publication, Inspector re-evaluates all affected images automatically without requiring a re-scan trigger.
  • SBOM export: Inspector generates SBOM in SPDX format for every image. Stored in S3 for audit purposes.

Common Mistakes to Avoid

  1. Relying on Kubernetes NetworkPolicy alone for east-west security. Standard Kubernetes NetworkPolicy is L3/L4 only (IP + port). It cannot enforce that a service only accepts legitimate HTTP GET requests (not POST to a sensitive endpoint). Cilium's L7 policy is required for true micro-segmentation.
  2. Not mentioning Fargate's limitation for Falco. Many candidates design a uniform runtime security platform and say "deploy Falco daemonset across all clusters" — but if 10 clusters are Fargate-only, Falco daemonsets won't run. GuardDuty EKS Runtime Monitoring supports Fargate.
  3. Treating image scanning as a one-time CI gate. Inspector v2 continuously re-evaluates images against new CVEs published after the image was scanned. A clean image today can have a Critical CVE tomorrow. The architecture must handle the "stale image" problem via Inspector's continuous re-evaluation + automated re-scan triggers.
  4. Forgetting EKS etcd encryption. By default, Kubernetes Secrets in etcd are base64-encoded (not encrypted). EKS supports envelope encryption of etcd using KMS. Without this, anyone with access to etcd storage can read all Secrets.

Likely Follow-up Questions

Q1: How does Falco's eBPF probe work without privileged containers?
Falco's eBPF driver runs at the kernel level, attached to kernel tracepoints (sys_enter/sys_exit hooks) using the Linux eBPF verifier. It does NOT require the Falco container to be privileged — the eBPF program is loaded into the kernel by the Falco init container (which does need brief elevated privileges to load the BPF program) and then the main Falco process reads events from the eBPF ring buffer as an unprivileged user-space process. On EKS 1.26+, the Amazon Linux 2023 kernel supports eBPF without kernel module compilation.

Q2: How do you handle zero-day container escapes that Falco doesn't have rules for?
Defense in depth: (1) PSA restricted ensures containers can't use CAP_SYS_ADMIN or hostPID — most kernel exploits require these. (2) Node isolation: each node runs pods from only one environment (prod vs. dev segregation); escape from a dev pod doesn't reach prod resources. (3) IMDSv2 + hop limit = 1 on EC2 nodes: even if an attacker escapes to the node, they can't impersonate node IAM role from inside a container. (4) Inspector v2 patches underlying node OS via SSM Patch Manager — reducing the kernel CVE surface.

Q3: How do you manage Falco rules across 50 clusters without drift?
Falco rules are stored in a Git repository and deployed as a ConfigMap via Helm chart. ArgoCD (GitOps) watches the Helm chart and applies updates to all 50 clusters automatically. Rule changes go through a PR process with security team review. Falco --dry-run mode is used in CI to validate rule syntax before merge. Cluster-specific overrides (e.g., dev clusters suppress some noisy rules) are in OU-specific Helm values files.

Q4: What happens if an attacker compromises a CI/CD pipeline and pushes a malicious image?
Multiple checkpoints: (1) ECR repository policy: PutImage only allowed from CodePipeline's IAM role, not from any arbitrary principal. (2) AWS Signer: only CodePipeline can call signer:StartSigningJob after build; the signing key is in an HSM-backed KMS key; a compromised pipeline that bypasses CodePipeline still can't sign. (3) Kyverno: at admission time, validates that the image signature is from the trusted AWS Signer key — not just any signature. (4) Inspector: even if a signed image passes, any Critical CVE blocks deployment to prod via ECR enhanced scanning.

Q5: How does the automated pod isolation avoid isolating the wrong pod in a high-density environment?
The isolation Lambda uses the exact pod UID (from GuardDuty/Falco finding metadata) + namespace + cluster ARN to target the specific pod. Kubernetes pod UIDs are globally unique within a cluster (UUID format). The Cilium network policy for quarantine uses a pod label (quarantine=true) as the selector — the Lambda patches the specific pod by UID, not by name or image, ensuring only the exact offending pod is isolated even if multiple pods share the same image or name prefix.

Q6: How do you achieve CIS Kubernetes Benchmark L2 compliance for etcd encryption?
When creating an EKS cluster, enable envelope encryption: aws eks create-cluster --encryption-config '[{"provider": {"keyArn": "arn:aws:kms:...CMK..."}, "resources": ["secrets"]}]'. This configures the Kubernetes API server to use a KMS provider plugin — every Secret is encrypted with a data encryption key (DEK), and the DEK is encrypted by the KMS CMK. The CMK's key policy restricts decryption to only the EKS cluster's IAM role. Even if etcd storage is compromised, Secrets are encrypted at the data level.

Q7: How do you prevent a developer from using kubectl exec to open a shell in a production pod?
Three controls: (1) IAM Identity Center permission set for prod clusters: developers only have view RBAC role (no pods/exec verb). The EKS access entry maps their IAM role to the view ClusterRole. (2) EKS audit log monitoring: GuardDuty and Falco detect kubectl exec events in the EKS audit log and alert immediately. (3) For legitimate debugging: developers submit a Service Catalog request for JIT exec access; Step Functions creates a time-bound EKS access entry with pods/exec permission for a specific pod/namespace, expiring after 2 hours. All exec sessions are recorded via EKS audit log.

Q8: How do you handle secrets rotation for pods that have long-running sessions using cached secrets?
External Secrets Operator (ESO) with rotation trigger: ESO syncs Secrets Manager/Parameter Store values into Kubernetes Secrets on a configurable interval (e.g., every 5 minutes). When a secret is rotated in Secrets Manager, ESO detects the version change and updates the Kubernetes Secret. For pods that need to pick up the new secret, I use Reloader (open-source): it watches for ConfigMap/Secret changes and triggers a rolling restart of Deployments that reference the updated secret. This ensures rotation completes within 10 minutes with no manual pod restarts.
