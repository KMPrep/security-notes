Q07 — Web Application Firewall and DDoS Protection for a Global SaaS

The Ask

"Design a comprehensive WAF and DDoS protection architecture for a global SaaS
platform serving 50 million users across North America, Europe, and Asia. The platform
has a REST API, a web front-end, and a mobile app backend. Requirements: sub-100ms
latency globally, protection against OWASP Top 10, volumetric DDoS attacks, bot
abuse, credential stuffing, and API abuse. Must pass PCI-DSS SAQ-A-EP."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Is the application already on CloudFront, or are we starting from an ALB/NLB origin?
    Impact: If not on CloudFront, I need to migrate traffic through CloudFront first. AWS WAF v2 must be attached to either CloudFront (global) or a regional ALB/API Gateway/AppSync. For 50M users globally with sub-100ms SLA, CloudFront is mandatory — WAF on regional ALB alone can't satisfy global latency requirements.

  • Q: What is the traffic profile — mostly authenticated API calls or a mix of unauthenticated browsing + API?
    Impact: Credential stuffing attacks target authenticated login endpoints. I need WAF rules specifically for login page rate limiting and Bot Control Smart Rule Group. For authenticated API calls, I focus on API abuse rules (token bucket rate limiting per API key, anomalous usage patterns).

  • Q: What is the expected peak legitimate traffic, and has the platform experienced DDoS attacks before?
    Impact: This determines Shield tier. Shield Standard (free) protects against L3/L4 volumetric attacks. Shield Advanced ($3,000/month) adds L7 application-layer DDoS protection, dedicated DRT (DDoS Response Team) access, and cost protection. For 50M users with SaaS revenue at risk, Shield Advanced is justified.

  • Q: What APIs are the most sensitive — payment APIs, auth endpoints, or data export APIs?
    Impact: PCI-DSS SAQ-A-EP scope: any API that handles cardholder data or redirects to payment processor. I need WAF rules specifically for these endpoints: rate limiting, geo-restriction if payments are only US-based, SQL injection rules tuned for payment data patterns.

  • Q: Is there a mobile SDK? Do mobile clients use certificate pinning?
    Impact: If mobile clients use certificate pinning, CloudFront must use a specific ACM certificate that the mobile SDK pins to. If the cert changes (rotation), the mobile app breaks. I need to manage certificate rotation carefully with the mobile team.

  • Q: What is the tolerance for false positives — can WAF rules occasionally block legitimate traffic?
    Impact: Aggressive WAF rules (e.g., AWS Managed Rules CommonRuleSet at BLOCK mode) can block legitimate traffic. I recommend starting all rules in COUNT mode for 2 weeks, analyze false positives, then move to BLOCK. High-impact rules (SQLi, XSS) in BLOCK mode from day 1; rate limiting rules in COUNT mode initially.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. CloudFront distribution already exists (or will be created). Origins: ALB in us-east-1 and eu-west-1 for failover.
  2. Shield Advanced enrolled for the CloudFront distribution and origin ALBs.
  3. WAF v2 attached to CloudFront (global) + separate WAF v2 on regional ALBs for defense-in-depth.
  4. Three traffic types: web front-end (HTTPS), REST API (HTTPS + JWT auth), mobile app (HTTPS + client cert).

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Who can modify WAF rules?

  • WAF rules managed via Firewall Manager policy in the Security account.
  • Only the Security team's IAM role can call wafv2:UpdateWebACL or fms:PutPolicy.
  • Developers cannot directly modify WAF rules in production. Emergency rule changes: Security team break-glass procedure (documented + reviewed post-incident).
  • All WAF changes are CloudTrail-logged: wafv2:UpdateWebACL generates an immutable audit event.

IP allow-listing for partners:

  • Trusted partner IPs added to WAF IP set via API (automated from partner onboarding workflow, not manual).
  • IP set changes are Terraform-managed and PR-reviewed.

2. TLS and Encryption

CloudFront TLS policy: TLSv1.2_2021
Supported cipher suites:

  • ECDHE-RSA-AES128-GCM-SHA256
  • ECDHE-RSA-AES256-GCM-SHA384
  • ECDHE-RSA-CHACHA20-POLY1305
    No TLS 1.0, TLS 1.1, RC4, 3DES — all disabled. PCI-DSS requires TLS 1.2+.

Certificate management:

  • CloudFront certificate: ACM in us-east-1 (mandatory for CloudFront).
  • ALB certificates: ACM in each region.
  • Auto-renewal: ACM handles renewal 60 days before expiration. CloudWatch alarm on DaysToExpiry < 30 for backup alert.
  • Mobile certificate pinning: a separate CloudFront distribution for mobile API with a dedicated pinned certificate. Certificate rotation requires coordinated mobile app update.

3. WAF Rule Deep Dive

Credential stuffing prevention:

Rule: LoginRateLimit
  Priority: 3
  Statement: RateBasedStatement
    Limit: 5
    EvaluationWindow: 300 (5 minutes)
    AggregationKey: IP + ForwardedIP (handles NAT)
    ScopeDownStatement: StringMatch uri = "/api/v1/auth/login"
  Action: BLOCK

Rule: LoginCaptcha (challenge before block)
  Priority: 2
  Statement: RateBasedStatement (threshold: 3 in 5 min)
  Action: CAPTCHA (force CAPTCHA challenge at threshold)

Bot Control configuration:

AWS Bot Control — Smart Rule Group (targeted mode):
  Endpoints: /api/auth/login, /api/checkout, /api/account
  Bot verified as legitimate (Googlebot, monitoring tools) → ALLOW
  Bot score > 75 (likely automated) → CAPTCHA challenge
  Bot score > 95 (highly confident automated) → BLOCK

Scraping protection:

Rule: ScrapingDetection
  Statement: ByteMatch — User-Agent contains empty string or common scraper signatures
  + RateBased: > 500 req/min per IP on /api/products/*
  Action: BLOCK

SQLi and XSS rules:

AWS Managed SQLiRuleSet:
  Inspects: query string, body, URI path, header values
  Sensitivity: HIGH (matches all SQLi patterns)
  Action: BLOCK (no exceptions — SQLi is never legitimate)

AWS Managed CommonRuleSet:
  Includes: Cross-site scripting (XSS) protection
  Inspects: body, query string
  Action: BLOCK
  Exception: /api/blog/content endpoint allows HTML body (tuned via scope-down statement)

4. Shield Advanced Integration

Proactive engagement:

  • Shield Advanced enabled on: CloudFront distribution ARN, origin ALB ARNs, Elastic IP addresses for NLBs.
  • Route 53 health check associated with CloudFront distribution: if health check fails during an attack, Shield DRT is automatically engaged.
  • DRT (DDoS Response Team) contacts: two 24/7 pager numbers for security team on-call.

Automatic application-layer (L7) mitigation:

Shield Advanced Automatic Application Layer DDoS Mitigation:
  - Enabled on CloudFront distribution
  - Action: BLOCK (not just count)
  - Shield automatically creates WAF rules during an attack to mitigate
    the specific attack traffic pattern without manual intervention

Shield Advanced cost protection:

  • During a DDoS attack, AWS credits data transfer and CloudFront costs caused by the attack traffic.
  • Protects against unexpected scale-out charges during a volumetric attack.

5. WAF Logging and Monitoring

WAF logging pipeline:

WAF → WAF Full Request Logs → Kinesis Data Firehose → S3 (Security Lake)
  - Log all requests (not just blocked) — essential for tuning and forensics
  - Log includes: rule match, action, IP, URI, headers, body (truncated to 64KB)
  - S3 lifecycle: 90 days in S3 Standard, 1 year in S3 Glacier

CloudWatch Metrics → Dashboard:
  - AllowedRequests, BlockedRequests, CountedRequests per rule
  - LoginRateLimit blocks per hour (spike = credential stuffing attempt)
  - Bot Control score distribution (how many bots at what confidence level)

Alerting:

CloudWatch Alarm: WAF BlockedRequests > 10,000/minute → SNS → PagerDuty P2
CloudWatch Alarm: Shield DDoS detected → SNS → PagerDuty P1
CloudWatch Alarm: OriginFailureRate > 5% → SNS → On-call (possible origin bypass attempt)

Scale and Resilience (Phase 4 — 10 min)

CloudFront absorbs DDoS volume:

  • CloudFront's anycast network (400+ POPs) absorbs volumetric attacks close to the attacker's geographic location. A 100 Gbps attack from Europe is absorbed at European POPs before reaching the origin.
  • AWS claims Shield Advanced can mitigate multi-Tbps attacks. No capacity planning required by the customer.

WAF scaling:

  • WAF v2 is fully managed and auto-scales. No WAF capacity limits to worry about at 50M users.
  • WAF rule evaluation adds ~1ms latency (AWS specification). For sub-100ms SLA, this is within budget.

Multi-region origin failover:

  • CloudFront origin groups: primary = ALB in us-east-1, secondary = ALB in eu-west-1.
  • Failover trigger: HTTP 5xx errors or health check failure (Route 53 health check integrated with CloudFront origin health check).
  • WAF rules deployed identically on both regional ALBs via Firewall Manager.

Firewall Manager at scale:

  • Single Firewall Manager policy covers all CloudFront distributions and ALBs across the organization.
  • When a new CloudFront distribution is created (e.g., new product launch), it automatically gets the baseline WAF rules applied within minutes via FMS policy.

Common Mistakes to Avoid

  1. Forgetting that WAF alone is not DDoS protection. WAF operates at L7 and can be overwhelmed by volumetric L3/L4 attacks before it even sees the traffic. Shield Advanced is the DDoS layer; WAF is the application layer. They are complementary.
  2. Not restricting the origin ALB to CloudFront IPs. If the ALB is publicly accessible, an attacker can simply bypass CloudFront (and WAF) by targeting the ALB directly. The ALB security group must only allow the CloudFront managed prefix list.
  3. Deploying WAF rules in Block mode immediately. AWS Managed Rules can have false positives. Starting in Count mode, analyzing logs for 2 weeks, then switching to Block mode prevents unexpected production outages.
  4. Using WAF as the only defense against SQL injection. WAF is a perimeter control. The application must also use parameterized queries / prepared statements. A WAF bypass (new SQLi pattern) combined with application-level vulnerability = breach.
  5. Not mentioning Firewall Manager for multi-account WAF coverage. A common architecture mistake is managing WAF rules per-account, per-resource. For a SaaS with many AWS accounts, Firewall Manager ensures consistent baseline rules automatically applied to new resources.

Likely Follow-up Questions

Q1: How does WAF Bot Control distinguish Googlebot from a fake Googlebot?
AWS Bot Control's Smart Rule Group uses multiple signals: (1) User-Agent string matching against known bot signatures, (2) IP reputation (known bot hosting providers), (3) Reverse DNS verification — legitimate Googlebot PTR records resolve to googlebot.com domain. A bot claiming to be Googlebot but coming from an IP that doesn't have googlebot.com PTR records is treated as impersonation. Additionally, Bot Control in targeted mode uses browser fingerprinting (JavaScript challenge) to distinguish real browsers from bots that don't execute JavaScript.

Q2: What happens during a Shield Advanced DDoS attack — walk me through the response flow?
Timeline: (1) Attack starts → Shield detects anomalous traffic pattern in <1 second. (2) Shield automatic mitigation activates: for volumetric L3/L4, traffic is scrubbed at AWS edge POPs close to the attack source. (3) For L7 attacks (HTTP flood), Shield Advanced automatically creates WAF rate-based rules to block attack traffic patterns. (4) If the associated Route 53 health check shows the origin is degraded, Shield DRT (DDoS Response Team) is automatically engaged and calls the security team's on-call number. (5) Post-attack: Shield provides a summary report with peak attack volume, attack vector, and mitigation applied.

Q3: How do you handle legitimate high-volume users that trip your WAF rate limits (e.g., a partner doing bulk API calls)?
Dedicated rate limit exemption for known partners: (1) Partner's static IP range is added to a WAF IP set labeled TrustedPartners. (2) The rate limiting rules include a scope-down statement that excludes the TrustedPartners IP set: only apply rate limiting if the source IP is NOT in TrustedPartners. (3) Alternatively, for partners using API keys, I use WAF's label-based matching: API Gateway authenticates the API key and adds a x-partner-tier: enterprise header; WAF rules scope-down to apply stricter limits only if the header is absent.

Q4: How do you protect against HTTP/2 Rapid Reset attacks (CVE-2023-44487)?
HTTP/2 Rapid Reset exploits the HTTP/2 stream cancellation mechanism to send and cancel streams rapidly, exhausting server resources. CloudFront automatically mitigates this because CloudFront's load balancer terminates HTTP/2 at the edge and proxies HTTP/1.1 to origins — the Rapid Reset attack hits CloudFront's fleet, not the origin. AWS patched CloudFront against CVE-2023-44487 within hours of disclosure in October 2023. For origins behind CloudFront, if using HTTP/2 directly to ALB, the ALB must also be patched — AWS ELB patched this automatically.

Q5: How does WAF handle encrypted traffic — can it inspect HTTPS bodies?
CloudFront terminates TLS at the edge PoP, decrypting the traffic. The WAF rules are applied to the decrypted HTTP request (including the decrypted body). The re-encrypted request is then forwarded to the origin over a new TLS connection. This means WAF can inspect all headers, query strings, URI paths, and request bodies (up to 8 KB by default, or 64 KB with extended inspection). The WAF never stores decrypted content — it inspects in memory and immediately discards.

Q6: How do you tune WAF rules to reduce false positives for a complex web application with rich HTML content?
Process: (1) Start all new rules in COUNT mode. (2) Deploy WAF full request logging to S3/Security Lake. (3) Run Athena queries to identify legitimate requests that are being COUNT-ed (not blocked) by specific rules — especially rules that inspect the request body (XSS rule often counts HTML editors that submit <script> tags legitimately). (4) For those specific endpoints, create a scope-down statement on the problematic rule: "apply XSS rule only if uri_path does NOT match /api/content-editor/*". (5) After 2 weeks with near-zero false positives in COUNT mode, switch to BLOCK mode. This process takes 2-4 weeks for complex applications.

Q7: What is your WAF rule prioritization strategy?
Rule evaluation stops at the first matching rule (by priority order). Strategy: (1) Lowest priority number (highest precedence) = IP Reputation list — cheap to evaluate, high confidence, eliminates known bad actors immediately. (2) Bot Control early — cheaper than deep rule inspection. (3) Rate limiting — stop floods before reaching signature-based rules. (4) AWS Managed Rules (SQLi, XSS, Common) — more expensive but accurate. (5) Custom rules at the end — least common, most specific. This ordering minimizes WAF processing cost (lower WCU consumption per request).

Q8: How do you handle geo-restriction for EU GDPR vs. PCI-DSS conflicting requirements?
Different requirements apply to different endpoints: (1) PCI-DSS payment endpoints: I may restrict to countries where the company operates (e.g., US + EU) using WAF geo-match rule — block payment API from jurisdictions where the company has no legal entity. (2) GDPR: not a geo-restriction requirement for WAF — GDPR requires data handling controls, not blocking EU users. (3) OFAC sanctions: WAF geo-block for sanctioned countries (Cuba, Iran, North Korea, Syria, Russia for some services) on all endpoints — this is a US regulatory requirement for all US-based companies. Rule: if country in [CU, IR, KP, SY] → BLOCK at high priority.
