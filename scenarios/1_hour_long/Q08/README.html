Q08 — Automated Vulnerability Management and Patch Compliance

The Ask

"Design an automated vulnerability management and patch compliance platform for a
company running 2,000 EC2 instances, 200 ECS tasks, 50 RDS instances, and 100 ECR
image repositories across 20 AWS accounts. The system must continuously discover
vulnerabilities, prioritize them by exploitability, automatically patch OS and
application dependencies, and demonstrate compliance to auditors within 72 hours
of a CVE disclosure."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What is the patching SLA by severity — how quickly must Critical CVEs be remediated?
    Impact: Most compliance frameworks define: Critical = 7 days, High = 30 days, Medium = 90 days. NIST CSF has no mandated SLA but recommends risk-based prioritization. PCI-DSS requires patching within 1 month for critical patches. This determines the urgency of automation vs. manual approval workflows.

  • Q: Are EC2 instances in Auto Scaling Groups (stateless) or are they stateful long-lived servers?
    Impact: Stateless ASG instances → patch by replacing with new AMI (immutable patching). Stateful servers (database servers, legacy apps) → in-place patching via SSM Patch Manager. These require different patching strategies. In-place patching requires maintenance windows.

  • Q: What OS types are in use — Amazon Linux 2, Amazon Linux 2023, Ubuntu, Windows Server, RHEL?
    Impact: SSM Patch Manager has different patch baselines per OS. Windows patching requires Windows Update integration. Ubuntu requires apt/dpkg. RHEL requires yum/dnf. Mixed OS environments need separate patch baselines and potentially different patch schedules.

  • Q: Is there an existing CMDB or asset inventory system the vulnerability data must integrate with?
    Impact: Inspector v2 outputs vulnerability findings as ASFF (Amazon Security Finding Format). If ServiceNow is the CMDB, I need an EventBridge → Lambda → ServiceNow API integration to create tickets for each finding. If no existing CMDB, Security Hub serves as the vulnerability tracker.

  • Q: What is the risk appetite for automated patching in production — can patches be auto-applied, or do they require change board approval?
    Impact: Conservative enterprises: auto-patch dev/staging; require change ticket for prod. Risk-accepting modern SaaS: auto-patch all tiers with 2-stage deployment (dev → staging → prod with automated rollback). This fundamentally changes the architecture.

  • Q: Do applications have known patch incompatibilities (e.g., specific Python version required)?
    Impact: Need application-specific patch exclusion lists. SSM Patch Manager supports RejectedPatches list per baseline. Inspector v2 findings can be suppressed (snoozed) for specific resources when a patch causes an incompatibility, with documented risk acceptance.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Mixed stateless ASG + stateful EC2. Majority (70%) are stateless.
  2. OS mix: Amazon Linux 2023 (60%), Ubuntu 22.04 (30%), Windows Server 2022 (10%).
  3. Patching SLA: Critical = 7 days, High = 30 days, Medium = 90 days.
  4. Auto-patch dev/staging; change-board-approved patching for prod (automated ticket creation, manual approval, automated execution).
  5. All instances managed via SSM Agent (required for SSM Patch Manager).

Core Architecture — Data Flow Narrative

Continuous Discovery → Prioritization → Ticketing → Patching → Verification → Audit

[Amazon Inspector v2 — delegated admin in Audit account]
  Coverage:
    - EC2 instances (OS CVEs + open ports)
    - ECR container images (OS + language dependency CVEs)
    - Lambda functions (package CVEs)
    - RDS (OS CVEs on underlying instances, Inspector DB coverage)

  Inspector v2 Integration:
    → EPSS score + CVSS score → severity normalized by actual exploitability
    → Findings sent to Security Hub (ASFF format)
    → EventBridge rule: CRITICAL finding → immediate notification

[AWS Security Hub — finding aggregation + compliance]
  → Inspector findings + Config rule violations in one dashboard
  → Compliance standard: CIS AWS v1.4
  → Custom Insight: "Critical EC2 CVEs older than 7 days" (breach of SLA)
  → All findings exported to Security Lake for trend analysis

[Ticketing Integration — EventBridge → Lambda → ServiceNow]
  Trigger: Inspector finding with severity CRITICAL or HIGH
  Lambda payload: {instance_id, CVE, CVSS, EPSS, affected_packages, suggested_patch}
  ServiceNow ticket:
    CRITICAL → P1 incident (7-day SLA enforcement)
    HIGH → P2 change request (30-day SLA)

[AWS Systems Manager — Patch Manager]
  Patch Baselines per OS:
    AmazonLinux2023-CriticalAndImportant: includes Critical + Important, excludes test packages
    Ubuntu2204-Standard: security-only patch baseline
    Windows2022-Standard: security updates + critical updates only

  Patch Groups (tags on EC2 instances):
    PatchGroup=Dev → maintenance window: Tuesday 10pm UTC (weekly, auto-approve)
    PatchGroup=Staging → maintenance window: Wednesday 10pm UTC (auto-approve)
    PatchGroup=Prod → maintenance window: Sunday 2am UTC (manual approval required)

  Patching workflow:
    1. SSM Run Command: scan-only (no patching) → generate patch compliance report
    2. Security Hub Custom Insight checks compliance percentage
    3. If auto-approve: SSM Patch Manager executes patching in maintenance window
    4. Post-patch: re-scan → verify CVE remediated → close ServiceNow ticket

[Stateless EC2 (ASG) — Immutable Patching]
  EC2 Image Builder pipeline (triggered monthly or on Critical CVE):
    1. Pull base AMI
    2. Apply OS patches (yum update -y --security)
    3. Inspector scan of pipeline instance
    4. If Critical CVE remains → fail pipeline, alert
    5. If clean → create new AMI
    6. Update ASG Launch Template to new AMI
    7. ASG rolling update (terminate old instances, launch patched ones)
    8. Rollback: Launch Template version rollback if health check fails

[Container Images — ECR Continuous Scanning]
  Inspector v2 ECR scanning:
    → Scans on image push (OS + language packages)
    → Re-evaluates on new CVE publication (no re-push needed)
    → Critical CVE on prod image → EventBridge → block tag promotion + create P1 ticket
    → Fix: update base image or dependency → rebuild + re-push → re-scan → verify clean

Security Deep Dive (Phase 3 — 15 min)

1. Inspector v2 — Exploitability-Driven Prioritization

EPSS (Exploit Prediction Scoring System) integration:

Inspector v2 enriches each CVE finding with:
  CVSS Score: technical severity (0-10)
  EPSS Score: probability the CVE will be exploited in the wild within 30 days (0-100%)
  Inspector Risk Score: normalized score combining CVSS + EPSS + network reachability

Prioritization example:
  CVE-2023-XXXX: CVSS 9.8, EPSS 2% → High priority (severe but rarely exploited)
  CVE-2023-YYYY: CVSS 7.1, EPSS 78% → CRITICAL priority (moderate CVSS but actively exploited)

Patching prioritization:
  Risk Score >= 90 (CRITICAL + high EPSS) → patch within 24 hours
  Risk Score 70-89 (HIGH) → patch within 7 days
  Risk Score 40-69 (MEDIUM) → patch within 30 days

Coverage across compute types:

EC2: Inspector reads SSM Inventory (packages, OS, network config) → no agent beyond SSM
ECS Tasks: Inspector scans ECR images that tasks are running
ECR Images: Inspector v2 enhanced scanning (OS + language packages)
Lambda: Inspector scans Lambda packages for known-vulnerable dependencies
RDS: Inspector v2 DB scanning (available for Aurora PostgreSQL, MySQL, MSSQL)

2. Patch Management Design

SSM Patch Manager Baseline (Amazon Linux 2023):

Patch Baseline Configuration:
  Name: AmazonLinux2023-SecurityCritical
  OS: AMAZON_LINUX_2023
  Approval Rules:
    Rule 1: severity IN [CRITICAL, IMPORTANT] → auto-approve after 0 days (immediate)
    Rule 2: severity IN [MEDIUM] → auto-approve after 7 days
  Rejected Patches: [python3-requests-2.28.0] (known incompatibility)
  Rejected Patches Compliance Level: INFORMATIONAL (don't fail compliance for intentional rejection)

Patch Groups (tag-based):
  EC2 tag: Patch Group = Dev|Staging|Prod
  Dev maintenance window: Tue 22:00 UTC, 2hr duration, 50% concurrency
  Staging: Wed 22:00 UTC, 20% concurrency
  Prod: Sun 02:00 UTC, 10% concurrency, Step Functions approval gate

Production patching approval workflow:

SSM Maintenance Window triggers (Prod) →
  1. Step Functions workflow starts
  2. Lambda: create ServiceNow change request, wait for approval (max 48h)
  3. On approval: Lambda initiates SSM Patch Manager scan-then-patch
  4. During patching: CloudWatch alarm threshold tightened (alert at 1% error rate vs normal 5%)
  5. Post-patch: SSM Compliance report generated
  6. Lambda: close ServiceNow change request, attach compliance report
  7. On failure: SNS P1 alert; rollback Lambda reverts Launch Template version

3. Immutable AMI Patching (EC2 Image Builder)

Image Builder pipeline:

Schedule: monthly + triggered by Inspector finding (Critical CVE in current AMI)
Pipeline:
  1. Source: official Amazon Linux 2023 AMI from AWS
  2. Build Phase:
     a. yum update -y --security (apply security patches)
     b. Install company baseline software (SSM Agent, CloudWatch Agent, CrowdStrike sensor)
     c. Harden: CIS Level 2 hardening script (disable root SSH, NTP config, audit rules)
  3. Test Phase:
     a. Inspector scan of running test instance
     b. Fail if Critical CVE remains
     c. Fail if CIS compliance score < 90%
  4. Distribute: AMI to all 20 member accounts via Resource Access Manager (RAM)
  5. Update: Lambda updates all ASG Launch Templates to new AMI
  6. ASG Rolling Update: terminates 10% of old instances at a time (zero downtime)

4. ECR Container Vulnerability Management

Pipeline enforcement:

ECR Enhanced Scanning (Inspector v2):
  - Scan on push: every image push triggers Inspector scan
  - Continuous: Inspector re-evaluates images when new CVEs published

Enforcement in CI/CD (CodePipeline gate):
  Post-build step: check Inspector findings for image
  Logic:
    if critical_count > 0: fail pipeline, block deployment
    if high_count > 5: require security team approval, auto-create P2 ticket
    if medium_count > 20: alert, allow deployment with risk acceptance

ECR Lifecycle Policy (clean up vulnerable old images):
  - Remove images with Critical CVEs older than 7 days if no active tasks use them
  - Prevents "ghost" vulnerable images accumulating in registry

5. Compliance Reporting for Auditors

72-hour auditor evidence package:

Trigger: auditor requests evidence for CVE-YYYY-XXXXX

Automated Lambda generates package:
  1. Inspector v2 findings report (all instances affected by CVE)
  2. SSM Patch Compliance report (patched instances with timestamp)
  3. Config rule compliance history (patch compliance over time)
  4. ServiceNow ticket trail (discovery → ticket creation → approval → patch → verification)
  5. Security Hub finding timeline (first detected → last updated → closed)
  6. AMI version history (before/after patching for ASG instances)

Package format: PDF + CSV
Delivery: S3 pre-signed URL (72-hour expiry), emailed to auditor

Scale and Resilience (Phase 4 — 10 min)

Inspector v2 at 20-account scale:

  • Inspector v2 Organization integration: delegated admin in Audit account sees all findings from all 20 member accounts. No per-account configuration needed — accounts enrolled via AWS Organizations auto-activation.
  • Finding volume: 2,000 EC2 instances × average 50 CVEs per instance = 100,000 findings. Inspector aggregates at the organization level; findings are searchable in Security Hub with filters.
  • Finding deduplication: Inspector deduplicates the same CVE on the same instance — updates the existing finding with severity/EPSS changes rather than creating duplicates.

SSM Patch Manager at scale:

  • Concurrent patching limit: SSM Run Command default concurrency is 50 targets. For 2,000 instances, use --max-concurrency 10% (200 concurrent) and --max-errors 5%.
  • Patching time: 2,000 instances at 20% concurrency = 10 batches. Each batch ~15 minutes = 2.5 hours total window.
  • Rate limiting: SSM API calls are subject to throttling. Use SSM Automation with exponential backoff retry for large-scale patching operations.

EC2 Image Builder AMI distribution:

  • AMI distributed to all 20 accounts via AWS RAM (Resource Access Manager). Single image build produces one AMI per region; shared to all accounts.
  • Image build time: approximately 20 minutes (build) + 10 minutes (test + Inspector scan) = 30 minutes from trigger to AMI ready.

Common Mistakes to Avoid

  1. Relying solely on CVSS scores for prioritization. CVSS measures technical severity but not real-world exploitability. CVE-2021-3156 (sudo Baron Samedit) had CVSS 7.8 but EPSS > 90% — it was being actively exploited. Inspector v2's combined risk score using EPSS is the right approach.
  2. Not differentiating between EC2 scanning and container image scanning. Many candidates describe "scanning all our systems with Inspector" without explaining that EC2 scanning uses SSM Inventory (agentless-ish), ECR scanning uses image layer analysis, and Lambda scanning uses package manifest analysis — three different mechanisms.
  3. Forgetting that Inspector v2 re-evaluates images automatically on new CVE publication. A common mistake is saying "we scan images on every push" — but what about a Critical CVE published 3 months after the image was scanned? Inspector v2 handles this automatically. This is a key selling point over traditional scanners.
  4. Not explaining the "coverage gap" problem. If an EC2 instance doesn't have SSM Agent, Inspector v2 can't scan it (Inspector EC2 scanning depends on SSM). You must detect and alert on instances not appearing in SSM Inventory. Config rules check for this.

Likely Follow-up Questions

Q1: How does Inspector v2 achieve agentless EC2 scanning?
Inspector v2 EC2 scanning is not fully agentless — it requires the SSM Agent to be running on the instance. Inspector reads the SSM Inventory data (list of installed packages, OS version, software inventory) collected by SSM Agent and matches this data against Inspector's CVE database. The Inspector service itself doesn't install a separate scanning agent; it leverages the SSM Agent already present. For instances without SSM Agent, Inspector coverage is zero — hence the Config rule monitoring SSM managed instance compliance.

Q2: What is your strategy for handling false positives from Inspector — where a CVE is flagged but the package is not actually exploitable in your environment?
Inspector v2 supports finding suppression with documented justification. Workflow: (1) Engineer identifies the finding as a false positive (e.g., the vulnerable code path is never reached, or the package version is patched with a backport not reflected in the version string). (2) Engineer creates a Jira ticket documenting the justification and mitigating controls. (3) Security team approves the suppression via a PR that adds the suppression rule to our Inspector suppression configuration (Terraform-managed). (4) Inspector suppression is applied; CloudTrail logs the BatchUpdateFindings suppression action. (5) Quarterly, all suppressions are reviewed — expired suppressions are removed. This ensures false positives are handled without bypassing the vulnerability management program.

Q3: How do you achieve zero downtime patching for a stateless web application?
EC2 Image Builder builds a new patched AMI. Once the AMI is tested and distributed, an automated Lambda updates the ASG Launch Template to the new AMI version. The ASG Instance Refresh feature is triggered with MinHealthyPercentage: 90% and CheckpointPercentages: [20, 50, 80, 100] — this replaces 10% of instances at a time. Each new instance comes up from the new AMI, passes health checks, and traffic is gradually shifted. Old instances are terminated in batches. The process takes approximately 20-30 minutes for a 100-instance ASG with no traffic interruption.

Q4: How do you handle a critical zero-day (e.g., Log4Shell) that affects 1,000 of your 2,000 EC2 instances before a patch is available?
When no patch exists (day-0 to day-1): (1) Inspector v2 detects affected instances immediately when AWS releases the finding. (2) Virtual patching via WAF: create a WAF rule blocking the Log4Shell JNDI lookup pattern (${jndi:) in request bodies and headers — this stops exploitation attempts while the real patch is prepared. (3) Network-level mitigation: GuardDuty ECS/EC2 Runtime Monitoring alerts on JNDI callback DNS lookups (indicator of exploitation attempt). (4) When patch/mitigation available: trigger emergency Image Builder pipeline; patch all 1,000 affected instances using priority patching outside normal maintenance windows. (5) Post-patch: verify with Inspector re-scan; close all affected findings.

Q5: How do you prove patch compliance to a PCI-DSS QSA?
PCI-DSS SAQ QSA wants evidence that: (1) All systems are inventoried (SSM Inventory report exported to CSV — complete list of all EC2 instances, OS versions, installed packages). (2) All Critical vulnerabilities patched within 1 month (Inspector finding timeline showing critical CVE found date → patch date < 30 days; exported from Security Lake via Athena query). (3) No Critical CVEs remain open beyond SLA (Security Hub Custom Insight: "Critical CVEs older than 30 days = 0" screenshot). (4) Patching process is documented and automated (ASG Instance Refresh logs, SSM Run Command execution records). The automated evidence package Lambda generates all of this in a 72-hour turnaround as a PDF with screenshots.

Q6: How does RDS vulnerability scanning work in Inspector v2?
Inspector v2 supports RDS database scanning for Aurora PostgreSQL, Aurora MySQL, RDS for MySQL, RDS for MSSQL, and RDS for Oracle. Inspector scans the underlying database engine version and configuration for known CVEs and security misconfigurations. However, Inspector does NOT scan the data inside the database (no SQL injection scanning or data-at-rest vulnerability scanning). For RDS OS-level vulnerabilities, the underlying EC2 is managed by AWS — AWS applies patches automatically during the RDS maintenance window. My role is to: (1) ensure the maintenance window is set (Config rule: rds-automatic-minor-version-upgrade-enabled), (2) monitor Inspector findings for RDS engine CVEs, (3) plan major version upgrades for end-of-life database versions.

Q7: How do you handle vulnerability management for Lambda functions?
Inspector v2 Lambda scanning works by analyzing the Lambda function's deployment package (zip or container image) for known-vulnerable packages in the function's dependencies (requirements.txt for Python, package.json for Node.js, pom.xml for Java). Inspector continuously re-evaluates Lambda findings when new CVEs are published. For remediation: (1) Update the vulnerable package in the function's requirements file. (2) Redeploy the Lambda function. (3) Inspector re-scans the new deployment package. (4) Finding closes automatically. Key challenge: Lambda functions are often owned by different teams — the vulnerability notification must go to the right team. I use Inspector's Lambda tag-based filtering to route findings to the correct team's ServiceNow queue using the function's Owner or Team tag.

Q8: What's your strategy for managing end-of-life OS versions (e.g., Amazon Linux 2 EOL in June 2025)?
EOL OS management is a vulnerability management problem: (1) Inspector v2 flags EOL OS versions as a finding (e.g., AMAZON_LINUX_2_EOL). (2) Security Hub custom insight: count of instances running EOL OS. (3) Migration plan: use EC2 Image Builder to build new AMIs on Amazon Linux 2023. For stateless instances: swap AMI in Launch Template and roll ASG. For stateful/legacy instances: lift-and-shift migration with application testing. (4) Enforcement SCP: after a defined cutoff date, SCP denies launch of EC2 instances using EOL AMIs. (5) Timeline: start migration 6 months before EOL; complete 1 month before EOL to allow for issues. Track progress in Security Hub dashboard.
