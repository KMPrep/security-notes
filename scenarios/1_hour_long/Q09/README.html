Q09 — Identity and Access Management Governance at Enterprise Scale

The Ask

"Design an IAM governance platform for a 5,000-person company running 50 AWS accounts.
The platform must enforce least privilege, detect and automatically remediate over-
privileged IAM identities, support just-in-time (JIT) access for privileged operations,
provide continuous compliance evidence, and integrate with HR onboarding/offboarding
processes. The company has SOC 2 and ISO 27001 requirements."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Is there an existing Identity Provider (IdP) — Okta, Azure AD, Ping Identity — or are users managed directly in IAM?
    Impact: IAM Identity Center (SSO) federated with an external IdP is the target architecture. If users are managed directly in IAM, there's a larger remediation effort to migrate to federated identities. Direct IAM users = higher risk (no MFA enforcement at IdP level, no centralized session control).

  • Q: What is the privilege escalation risk tolerance? Should JIT access be fully automated (approval-free for standard elevated actions) or require a manager/security-team approval?
    Impact: Fully automated JIT (self-service elevation with time-boxed permissions) moves fast but reduces oversight. Approval-gated JIT (Jira or ServiceNow approval) adds latency (hours) but provides a complete audit trail that satisfies SOC 2 CC6.3 (authorization control). For sensitive environments (prod, PCI, HIPAA), approval is typically required.

  • Q: Are there service accounts (non-human identities)? Batch jobs, CI/CD pipelines, Lambda functions?
    Impact: Non-human identities often accumulate the most privilege over time. They need separate governance: IAM Roles for EC2/Lambda (not access keys), GitHub Actions OIDC federation (not long-lived IAM users), key rotation enforcement for any remaining access keys.

  • Q: What is the offboarding timeline — how quickly must access be revoked when an employee leaves?
    Impact: SOC 2 CC6.2 requires timely access revocation. Best practice: IdP account disabled within 4 hours of HR termination → IAM Identity Center session invalidated → all active AWS console sessions terminated. This requires HR system → IdP → IAM Identity Center integration.

  • Q: Are there compliance requirements for access reviews — quarterly/annual certifications?
    Impact: SOC 2 CC6.3 requires periodic access reviews. ISO 27001 A.9.2.5 requires regular user access reviews. This means I need to generate access review reports, push them to managers in Jira/ServiceNow, track attestations, and auto-revoke uncertified access.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Identity Provider: Okta (SAML/OIDC federation to IAM Identity Center).
  2. 50 AWS accounts managed via AWS Organizations (Control Tower).
  3. Target: zero direct IAM users for humans; all access via IAM Identity Center Permission Sets.
  4. Non-human identities: EC2 instance profiles, Lambda execution roles, ECS task roles, GitHub Actions OIDC.
  5. JIT access: ServiceNow approval workflow for prod/PCI account access.

Core Architecture — Data Flow Narrative

Identity Lifecycle → Access Control → Continuous Monitoring → JIT Elevation → Access Review

[Okta — Source of Truth for Human Identities]
  HR system (Workday) → Okta SCIM provisioning (auto-provision on hire, de-provision on term)
  Okta groups map to IAM Identity Center Permission Sets:
    Engineers → read-only permission sets for dev/staging
    SREs → operational permission sets (no IAM changes)
    Security team → Security-Read permission set + JIT for Security-Admin
    No one has standing access to Production Admin or PCI accounts

[IAM Identity Center — Federated Access Management]
  Organization-level deployment (Management account)
  Permission Sets:
    ReadOnly: AdministratorAccess without IAM write, no resource creation
    Developer: PowerUserAccess minus IAM, minus production account access
    SRE-Operational: EC2+ECS+RDS read+describe+stop/start only
    SecurityAudit: SecurityAuditPolicy (read all, no write)
    Admin-JIT: AdministratorAccess — provisioned only on demand, expires in 8 hours

  Account assignment: Okta group → Permission Set → specific account list
  MFA: enforced at Okta level (FIDO2/Passkey required for console access)

[IAM Access Analyzer — Continuous Least Privilege Analysis]
  Organization-level analyzer (in Management account):
    → Identifies unused permissions across ALL 50 accounts
    → Generates IAM policy with only the permissions actually used in last 90 days
    → IAM Access Advisor: per-service, per-user last access timestamps

  Findings types:
    External access: S3 bucket publicly accessible → Critical
    Cross-account: Role accessible from unexpected account → High
    Unused permissions: Role granted S3:PutObject but last used 180+ days ago → Medium

  Auto-remediation:
    External access finding → Lambda: remove public access, notify owner
    Unused permission (>90 days): Lambda: generate scoped-down policy, create PR via Terraform
    Privilege creep: Lambda: create Jira task for permission review

[Permission Boundaries — Guardrail for Developer-Created Roles]
  Permission Boundary attached to all developer-created IAM roles:
    - Deny IAM:CreateRole, IAM:AttachRolePolicy without specific tag
    - Deny ability to create policies with * resource
    - Deny access to management account, security account, log archive account
    - Allow only actions within the developer's AWS account context

  SCPs (org-level):
    - Deny IAM:CreateUser (no direct IAM users)
    - Deny IAM:CreateAccessKey for non-service accounts
    - Deny removal of CloudTrail, GuardDuty
    - Deny access to us-gov regions

[JIT Access — Service Catalog + Step Functions]
  Request flow:
    1. Engineer requests prod admin access via internal portal (ServiceNow form)
    2. ServiceNow creates approval task for Engineering Manager + Security team
    3. On dual approval: Step Functions invokes Lambda
    4. Lambda: IAM Identity Center CreateAccountAssignment (Admin-JIT permission set → account → user)
    5. Assignment TTL: 8 hours. At expiration: Lambda DeleteAccountAssignment
    6. All actions logged: CloudTrail → Security Lake → Splunk
    7. Break-glass: automatic approval for P0 incidents (with mandatory post-incident review)

Key AWS Services Selected and Why

IAM Identity Center Centralized SSO across all 50 accounts with Permission Sets Manages account-level access from one place; Permission Sets are versionable (Terraform); replaces per-account IAM user management

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Permission Set design — principle of least privilege:

Permission Set: SRE-Operational
  AWS Managed: ReadOnlyAccess
  Inline additions:
    ec2:StartInstances, ec2:StopInstances, ec2:RebootInstances
    ecs:UpdateService (scale up/down only — no task definition changes)
    rds:RebootDBInstance, rds:StartDBInstance, rds:StopDBInstance
    ssm:StartSession (for SSM Session Manager only — no SSH)
  Explicit denies:
    iam:* (no IAM modifications)
    organizations:* (no org-level changes)
    cloudtrail:StopLogging, cloudtrail:DeleteTrail (no audit tampering)
    kms:ScheduleKeyDeletion, kms:DisableKey (no key destruction)

ABAC (Attribute-Based Access Control) for fine-grained resource access:

Use case: Engineers can only access resources tagged with their team
IAM Identity Center ABAC configuration:
  Session tag: PrincipalTag/Team = {Okta group attribute: team}
  IAM policy condition:
    "Condition": {
      "StringEquals": {
        "aws:ResourceTag/Team": "${aws:PrincipalTag/Team}"
      }
    }

Result: Frontend engineers can only access S3 buckets, DynamoDB tables, and Lambda
functions tagged Team=Frontend. No cross-team resource access without explicit grant.

2. Non-Human Identity Governance

IAM Roles for EC2 instances (instance profiles):
  No direct access key issuance for EC2 workloads
  SSM Systems Manager role: AmazonSSMManagedInstanceCore only (not PowerUserAccess)
  Application roles: scoped to specific S3 prefix, specific DynamoDB table ARN
  IMDSv2 required (SCP denies IMDSv1 EC2 launches)

Lambda execution roles:
  Per-function roles (not shared Lambda role)
  Resource-based policies with specific table/bucket ARNs
  Inspector v2 scans Lambda function packages for CVEs
  IAM Access Analyzer traces unused Lambda permissions

GitHub Actions OIDC federation:
  No IAM access keys stored in GitHub secrets
  OIDC trust policy:
    Principal: federated: token.actions.githubusercontent.com
    Condition: StringLike: token.actions.githubusercontent.com:sub: repo:org/repo-name:*
  Role session duration: 15 minutes (just enough for CI job)
  Separate deploy roles per environment (dev-deploy, staging-deploy, prod-deploy)

3. Offboarding Automation

HR System (Workday) termination event:
  1. Workday API → EventBridge → Lambda: disable Okta user account (SCIM API)
  2. Okta revokes all active sessions (SSO sessions, OAuth tokens)
  3. IAM Identity Center: user's account assignments are revoked within 1 hour
  4. Lambda: scan CloudTrail for any active sessions → force logout (revoke temporary creds)
  5. Lambda: check for any IAM access keys owned by user (should be zero) → if found, disable immediately
  6. Lambda: check for any IAM Identity Center groups where user was only member → notify team lead
  7. ServiceNow ticket created for 24-hour verification + manager confirmation
  8. Verification Lambda re-runs after 24h: confirm no active sessions, no access keys

Timeline: Full offboarding within 1 hour of HR event
SOC 2 evidence: CloudTrail shows termination of access within 1h (CC6.2 compliance)

4. Quarterly Access Reviews (Certification Campaigns)

Access Review workflow (quarterly, automated):
  1. Lambda queries IAM Identity Center: all account assignments
  2. Lambda queries IAM Access Advisor: last-accessed timestamps for each role
  3. Lambda generates review report per manager (which team members have access to which accounts)
  4. ServiceNow certification task created per manager (14-day deadline)
  5. Manager reviews: certify (keep) or revoke each assignment
  6. Auto-revoke: any assignment not certified within 14 days is automatically removed
  7. Post-campaign report: % certified, % auto-revoked, SLA compliance
  8. Athena query against Security Lake for "access never used" (last accessed = never in 90 days) → these are auto-revoked without manager review

SOC 2 CC6.3 evidence: certification campaign records, attestation timestamps, revocation records

5. Least Privilege Enforcement and Drift Detection

Continuous permission drift detection:
  IAM Access Analyzer + Lambda (daily):
    1. Query: all roles with FindingType=UnusedPermission, lastAccessedDate > 90 days
    2. For each role: generate least-privilege policy via Access Analyzer GeneratePolicy API
    3. Create Terraform PR in GitHub with scoped-down policy (human review required)
    4. If role has *:* permissions or excessive account-wide access: P1 alert, immediate review

SCP drift detection:
  AWS Config managed rule: scp-compliant (custom Lambda-backed rule)
    - Verifies no SCP overrides on member accounts
    - Verifies required SCPs (deny-root-login, deny-iam-user-creation) are attached
    - Security Hub finding if SCP drift detected

6. Compliance Mapping

Compliance Requirement Control
SOC 2 CC6.1 — Logical access restricted to authorized users IAM Identity Center + Okta MFA; SCPs prevent direct IAM users
SOC 2 CC6.2 — Remove access promptly Automated offboarding within 1 hour
SOC 2 CC6.3 — Access reviewed periodically Quarterly certification campaigns
ISO 27001 A.9.2.5 — Review user access rights Access review reports with manager attestation
ISO 27001 A.9.4.2 — Secure logon procedures MFA enforced; FIDO2/Passkey via Okta

Scale and Resilience (Phase 4 — 10 min)

IAM Identity Center at 50-account scale:

  • Permission Sets are Terraform-managed (gitops): PR → review → apply. Permission Set changes take ~30 seconds to propagate to all accounts.
  • Account assignments use Okta group membership as the source of truth. Adding a user to an Okta group → IAM Identity Center SCIM sync → account assignment within 10 minutes.
  • Max Permission Sets per organization: 2,000 (soft limit). With 50 accounts and ~20 permission set types, this is well within limits.

IAM Access Analyzer at organization scale:

  • Org-level analyzer covers all 50 accounts from a single analyzer. Finding volume: approximately 500-1,000 findings per run (unused permissions are common). Lambda processes findings in batches using SQS to avoid Lambda timeout.
  • Access Analyzer GeneratePolicy API is eventually consistent — allow up to 30 minutes for the generated policy to reflect recent CloudTrail activity.

JIT Access availability:

  • ServiceNow as approval system is a single point of failure. For P0 incidents: break-glass mechanism allows security team to directly invoke the JIT Lambda (bypasses ServiceNow approval). Break-glass usage requires post-incident review within 24 hours.

Trade-off Analysis — The Hard Choices

Decision 1: IAM Identity Center vs. Per-Account IAM Roles with Cross-Account Trust

Factor IAM Identity Center Per-account IAM cross-account trust
Centralized management Single place for all 50 accounts Must manage trust policies in every account
Session duration 1-12 hours (configurable) Configurable but harder to centralize
ABAC support Native session tags from IdP Possible but complex to orchestrate
JIT provisioning CreateAccountAssignment API Requires trust policy updates per account
Recommendation IAM Identity Center — mandatory for multi-account environments; per-account trust does not scale

Decision 2: Standing Admin Access vs. JIT-Only for Privileged Roles

Factor Standing Admin Access JIT-Only
Convenience High (always available) Low (requires approval process)
Blast radius High (compromised creds = instant admin) Low (no standing privilege to steal)
Audit trail Possible to miss routine access Every elevation is an explicit event
SOC 2 / ISO compliance Harder to demonstrate need-to-know Strong access control evidence
Recommendation JIT-only for all prod/PCI/HIPAA account admin access; standing read-only for troubleshooting

Decision 3: RBAC vs. ABAC for Resource Access

Factor RBAC (role-based) ABAC (attribute-based)
Simplicity Fewer roles, easier to understand Many fewer roles needed as attributes scale
Flexibility Role explosion as team/resource matrix grows Scales with tags without new roles
Implementation complexity Low Medium (requires consistent tagging)
Recommendation ABAC for resource-level access (team-scoped S3/DynamoDB/Lambda); RBAC for account-level access (Permission Sets)

Decision 4: Auto-Remediate vs. Alert-and-Review for Over-Privileged Roles

Factor Auto-remediate (scoped down immediately) Alert + human review
Risk of breaking apps High (unused permission might be needed for rare operation) None (human reviews before change)
Speed Fast (same day) Slow (depends on team responsiveness)
Recommendation Alert + human review via Terraform PR for production roles; auto-remediate only for roles with clear unused permissions for 180+ days

Common Mistakes to Avoid

  1. Conflating IAM users with IAM roles. Modern AWS architecture should have zero IAM users for humans. All human access should be via IAM Identity Center (SAML federation with IdP). IAM access keys should only exist for service accounts that cannot use roles, and even then they should be rotated via Secrets Manager.
  2. Not explaining Permission Boundaries vs. SCPs. Both are "guardrails" but at different levels. SCPs apply to the entire AWS account — they cannot be overridden by anyone in the account, including the account root. Permission Boundaries apply to a specific IAM principal within an account — they restrict what that principal can do, but do not apply to other principals.
  3. Forgetting OIDC federation for CI/CD. Many candidates describe "storing AWS access keys in GitHub secrets" which is an immediate security finding. The correct answer is GitHub Actions OIDC federation with short-lived STS credentials — no long-lived keys, no key rotation risk.
  4. Not covering non-human identity governance. IAM governance is often described only in terms of human users. But service roles (Lambda, EC2, ECS) often have more permissions than humans and are harder to deprovision. Inspector + Access Analyzer coverage of non-human identities is essential.

Likely Follow-up Questions

Q1: How do you prevent an engineer with IAM:PassRole permission from escalating to Administrator?
iam:PassRole combined with ec2:RunInstances or lambda:CreateFunction is a classic privilege escalation path: pass a high-privilege role to an EC2 instance or Lambda, then use that instance to assume the role. Prevention: (1) Restrict iam:PassRole to specific role ARNs only (resource-based condition iam:PassedToService limits which service can assume the passed role). (2) Permission Boundary on all developer-created roles prevents the boundary from being exceeded — even if they pass a role with AdministratorAccess, the boundary limits what the role can actually do. (3) IAM Access Analyzer alerts on iam:PassRole to sensitive roles.

Q2: How does IAM Access Analyzer generate least-privilege policies?
Access Analyzer's GeneratePolicy API analyzes CloudTrail logs for a specified IAM principal over a time window (up to 90 days). It identifies every API call made by that principal (service, action, resource ARN) and generates an IAM policy that grants exactly those permissions — no more. The generated policy uses specific resource ARNs where possible, not wildcards. However, the generated policy only reflects past behavior — if a new code path uses a new API that was never called in the 90-day window, the scoped policy will break it. Therefore, I treat the generated policy as a starting point for human review, not an automatically applied policy.

Q3: What happens when an IAM Identity Center session expires during an active console session?
When an IAM Identity Center session expires (configurable from 1 to 12 hours), the AWS console redirects the user to the IAM Identity Center login page. The temporary STS credentials issued at session start expire too, so any in-progress API calls will fail with ExpiredTokenException. For human console users, this is acceptable behavior — they re-authenticate via Okta (MFA required). For programmatic access via the AWS SDK (using IAM Identity Center's aws sso login), the SDK automatically refreshes the token if the refresh token is still valid. The session-level configuration (not the account-level token expiry) controls the console duration.

Q4: How do you govern Terraform/CDK infrastructure-as-code IAM role creation?
Two controls: (1) Terraform plan review: every IAM policy change goes through a PR review process. OPA (Open Policy Agent) or Checkov policy-as-code scans the Terraform plan for violations (star permissions, missing resource scoping, missing tags). The CI/CD gate blocks merges if violations are found. (2) Permission Boundaries enforced via SCP: an SCP requires that any new IAM role must have a specific Permission Boundary ARN attached. If the Terraform code creates a role without the boundary, the iam:CreateRole API call fails with AccessDenied. This means even if a developer's Terraform code tries to create an overly permissive role, the SCP prevents it from being created without the boundary.

Q5: How do you handle IAM governance for a newly acquired company being integrated into your AWS organization?
Acquisition IAM integration phases: (1) Discovery (week 1-2): IAM Access Analyzer scans all acquired accounts for external access findings, cross-account access, and overly permissive policies. IAM credential report exported to identify all users and access keys. (2) Quick wins (week 3-4): disable all IAM users with console access (they'll migrate to IdP federation); rotate all long-lived access keys; enable GuardDuty + CloudTrail. (3) Federation migration (month 2-3): provision acquired company's users in Okta; assign IAM Identity Center Permission Sets; deprecate IAM users. (4) Enforcement (month 4+): apply SCPs from parent organization (deny IAM user creation, enforce IMDSv2, etc.); run Access Analyzer to identify remaining privilege drift; quarterly access reviews.

Q6: What is the blast radius if an attacker compromises the IAM Identity Center admin account?
IAM Identity Center admin is the Management Account (or delegated admin account). Compromise of this account could allow the attacker to assign themselves AdministratorAccess Permission Sets to any/all of the 50 member accounts. Controls that limit blast radius: (1) Management account has almost no workloads — it's used only for org-level services (Control Tower, IAM Identity Center, consolidated billing). (2) SCPs cannot be modified by IAM Identity Center admin — SCPs are a separate AWS Organizations feature; modifying them requires organizations:UpdatePolicy which requires Management Account root or trusted delegated admin. (3) CloudTrail + Security Lake monitors for IAM Identity Center account assignment changes — alert on any assignment to production or security accounts. (4) Management account MFA enforced via Okta FIDO2 + hardware token — compromise requires physical device. (5) Break-glass response: rotate Management Account root credentials, disable IAM Identity Center federation, re-establish from known-good Terraform state.

Q7: How do you handle developers who accidentally commit IAM access keys to GitHub?
Automated detection + response: (1) GitHub secret scanning (native GitHub Advanced Security) detects AWS access key patterns in commits and alerts immediately. (2) AWS also has a partner integration with GitHub — AWS is notified by GitHub of exposed keys automatically (opt-in via AWS credential compromise response). (3) Response Lambda (triggered by AWS credential compromise EventBridge event): immediately disable the compromised access key via iam:UpdateAccessKey. Generate new access key (if programmatic access is truly needed) or migrate to OIDC. (4) Audit: query CloudTrail for all API calls made with the compromised access key from the moment of exposure to disablement. Determine if data was exfiltrated. (5) Root cause: if the developer needed access keys for CI/CD, migrate to OIDC federation. If for local dev, use aws sso login with IAM Identity Center.

Q8: How do you prove IAM governance compliance in a SOC 2 Type II audit?
SOC 2 Type II requires evidence of operating effectiveness over a 12-month period (not just point-in-time). Evidence package: (1) CC6.1 (logical access): screenshot of IAM Identity Center showing all users are federated (no direct IAM users); SCP showing iam:CreateUser denied; Okta MFA enforcement policy. (2) CC6.2 (access removal): 3 sample offboarding events — CloudTrail showing session termination within 1 hour of HR event. (3) CC6.3 (periodic review): 4 quarterly access review reports showing certification rates, auto-revocations, and completion within SLA. (4) Continuous monitoring evidence: Security Hub dashboard screenshots showing zero critical IAM findings over the 12 months; IAM Access Analyzer finding closure rates. All evidence is auto-generated by the quarterly evidence Lambda and stored in S3 (Object Lock) for the audit period.
