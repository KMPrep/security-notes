Q10 — Secure Serverless Architecture for a FinTech Payment API

The Ask

"Design a secure serverless architecture for a FinTech company's payment processing
API that handles 10,000 TPS at peak, stores cardholder data, must be PCI-DSS Level 1
compliant, and processes both card-present (POS terminal) and card-not-present (web
checkout) transactions. The system must achieve end-to-end encryption from the payment
terminal to the issuing bank."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Is the company a payment processor, payment gateway, or merchant? This changes the PCI-DSS scope significantly.
    Impact: A payment processor stores, processes, and transmits cardholder data (CHD) — full PCI-DSS Level 1 scope. A payment gateway routes transactions but may not store CHD. A merchant that uses point-to-point encryption (P2PE) may qualify for the simplified SAQ P2PE form. If P2PE is in scope, the primary system design goal is to ensure plaintext PANs (Primary Account Numbers) never reach the merchant's infrastructure.

  • Q: Is this a net-new payment system or migrating an existing on-premises payment processor to AWS?
    Impact: Net-new: can design with AWS Payment Cryptography (managed HSM service for payment key management) from the start. Migration: must maintain compatibility with existing HSM key blocks and host-level security modules.

  • Q: What card networks are supported — Visa, Mastercard, AMEX? Is 3DS (3D Secure) authentication required?
    Impact: 3DS requires integration with card network ACS (Access Control Server) and can significantly increase API design complexity. Visa and Mastercard have network-specific data formats (EMV, ISO 8583). AMEX has separate certification requirements.

  • Q: What is the target RTO/RPO for payment processing? Is active-active multi-region required?
    Impact: Payment systems typically require RPO = 0 (no lost transactions) and RTO < 30 seconds. This drives active-active multi-region design with DynamoDB Global Tables for transaction state. Active-passive would allow brief outages during failover.

  • Q: How are POS terminals connected — dedicated leased lines, TLS over internet, or payment network directly?
    Impact: POS terminals typically connect via TLS 1.3 over the internet to the payment gateway API. Some high-security environments use AWS Direct Connect from POS terminal networks. Terminal connectivity determines the entry point for the CDE (Cardholder Data Environment).

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Payment gateway role (not processor/acquirer): tokenize PAN at entry, route to processor network.
  2. P2PE: POS terminals encrypt PAN at the device; API Gateway receives encrypted payload only.
  3. PCI-DSS Level 1 required (>6M transactions/year).
  4. Serverless: Lambda for compute, API Gateway for entry, DynamoDB for token storage.
  5. Multi-region active-passive (us-east-1 primary, us-west-2 DR).

Core Architecture — Data Flow Narrative

PAN NEVER IN PLAINTEXT ON MERCHANT SYSTEMS
=========================================

[Card-Not-Present — Web Checkout]
Browser → Stripe.js / Braintree.js (client-side JS)
  → Client-side tokenization: PAN encrypted with RSA-2048 in browser
  → HTTPS (TLS 1.3) → API Gateway (edge-optimized, WAF attached)
  → API Gateway requests are validated (JWT authorizer from Cognito/OAuth2)
  → Encrypted token payload → Lambda Tokenization Function
  → Lambda: decrypt with AWS Payment Cryptography key
  → Lambda: generate format-preserving token (FPE) using AWS Payment Cryptography
  → Token stored in DynamoDB Tokens table (token → encrypted PAN reference)
  → Payment request forwarded to Acquirer network via PrivateLink

[Card-Present — POS Terminal]
POS Terminal (EMV chip reader):
  → Point-to-Point Encryption (P2PE): PAN encrypted at device with HSM-derived key
  → HTTPS/TLS 1.3 → API Gateway (Regional endpoint, restricted to terminal IP ranges via WAF)
  → Lambda: verify EMV cryptogram (AWS Payment Cryptography VerifyMac)
  → Lambda: decrypt PAN at HSM boundary using AWS Payment Cryptography
  → Lambda: generate token, store in DynamoDB
  → ISO 8583 message constructed → forwarded to acquirer network (via AWS Direct Connect to payment network)

[AWS Payment Cryptography — Key Management]
  Master Key (Zone Master Key — ZMK): stored in Payment Cryptography HSM
  Key hierarchy:
    ZMK → Terminal Master Key (TMK) per POS terminal
    ZMK → Data Encryption Key (DEK) for stored data
    ZMK → MAC Key for message authentication

  AWS Payment Cryptography operations:
    EncryptData: encrypt PAN with DEK
    DecryptData: decrypt incoming encrypted PAN
    GenerateMac / VerifyMac: EMV cryptogram validation
    TranslatePinData: PIN block translation (for PIN-based transactions)
    GenerateCardValidationData / VerifyCardValidationData: CVV/CVV2 validation

[Tokenization — Format Preserving Encryption (FPE)]
  Input: 16-digit PAN (4111 1111 1111 1111)
  Output: format-preserving token (4520 8823 9145 2198) — valid Luhn, same format
  Storage: DynamoDB token table (tokenization vault)
    Partition key: token value (FPE token)
    Attributes: encrypted_pan (KMS-encrypted), merchant_id, expiry, timestamp
    Encryption at rest: DynamoDB server-side encryption with KMS CMK (PCI-CMK)

  De-tokenization (only for settlement):
    Lambda Settlement: only role with DynamoDB:GetItem permission on token vault
    Lambda Settlement: calls Payment Cryptography DecryptData to recover PAN
    PAN sent directly to acquirer network via PrivateLink → never stored again

[Security Boundary — CDE Isolation]
  Lambda functions in VPC:
    - CDE VPC: only tokenization + settlement functions
    - Non-CDE VPC: merchant portal, reporting, webhooks
    - No peering between CDE VPC and non-CDE VPC
    - CDE Lambda → DynamoDB via VPC endpoint (no internet routing)
    - CDE Lambda → AWS Payment Cryptography via VPC endpoint
    - CDE Lambda → acquirer network via AWS Direct Connect (not public internet)

  Security groups:
    CDE Lambda SG: no inbound (Lambda, not a server)
    CDE Lambda SG: outbound only to DynamoDB endpoint, Payment Cryptography endpoint, Direct Connect

[API Layer]
  API Gateway (edge-optimized for CNP, regional for POS terminals):
    WAF v2 attached: OWASP CommonRuleSet, Rate limit 10,000/s per IP, JWT validation
    mTLS: POS terminals must present client certificate (ACM Private CA)
    Cognito User Pool authorizer: for merchant portal API calls
    Request validation: JSON schema validation built into API Gateway
    Access logging: all requests → CloudWatch Logs → Kinesis → Security Lake
    Throttling: 10,000 TPS burst, 5,000 TPS steady state (matching Lambda concurrency)

[DynamoDB — Transaction Storage]
  CDE VPC endpoint access only
  Encryption: SSE with KMS CMK (PCI-KMS key)
  Point-in-time recovery (PITR): enabled (RPO = 35 days)
  Streams: enabled for audit trail → Lambda → CloudWatch Logs
  IAM policy: resource-level ARN access (no * in resource)
  Global Tables (active-passive): us-east-1 → us-west-2 (write in us-east-1 only)

Security Deep Dive (Phase 3 — 15 min)

1. PCI-DSS CDE Isolation

Network isolation:

CDE VPC (us-east-1, us-west-2):
  Subnets: private only (no public subnets in CDE)
  Internet Gateway: NONE (no inbound/outbound internet from CDE)
  NAT Gateway: NONE (CDE functions don't need internet access)
  VPC Endpoints:
    - com.amazonaws.us-east-1.dynamodb (Gateway endpoint)
    - com.amazonaws.us-east-1.kms (Interface endpoint)
    - com.amazonaws.us-east-1.payment-cryptography (Interface endpoint)
    - com.amazonaws.us-east-1.secretsmanager (Interface endpoint)
    - com.amazonaws.us-east-1.logs (Interface endpoint — CloudWatch Logs)
  Direct Connect: to acquirer network (PCI-compliant carrier)

VPC Flow Logs: enabled, encrypted with KMS, retained 1 year
Security Groups: deny-all default, explicit allow only to endpoints

Scope reduction:

PAN flows only through:
  1. Payment Cryptography (decryption boundary) — managed HSM, not in VPC
  2. CDE Tokenization Lambda (in VPC) — only while tokenizing
  3. DynamoDB token vault — stores encrypted reference, not PAN
  4. CDE Settlement Lambda (in VPC) — only during settlement

PANs NEVER stored in:
  - CloudWatch Logs (DLQ from structured logging library strips PANs)
  - X-Ray traces (X-Ray sampling excludes payment-sensitive attributes)
  - API Gateway access logs (response body not logged)
  - S3 / Glacier backups
  - Developer laptops

2. Encryption (At Rest, In Transit, In Use)

In Transit:

All connections: TLS 1.3 minimum
  Browser → API Gateway: TLS 1.3 (enforced via CloudFront security policy)
  POS Terminal → API Gateway: TLS 1.3 + mTLS (terminal presents ACM Private CA cert)
  Lambda → DynamoDB: TLS via VPC endpoint
  Lambda → AWS Payment Cryptography: TLS 1.3 via FIPS endpoint
  Lambda → Acquirer network: TLS 1.3 over Direct Connect

Certificate pinning: POS terminals pin to ACM Private CA root

At Rest:

DynamoDB token vault:
  SSE: KMS CMK (pci-token-vault-key)
  Key rotation: annual automatic + on breach

KMS key hierarchy:
  pci-master-key (CMK): key policy restricts to CDE Lambda execution roles only
  ↓ encrypts
  pci-token-vault-key (CMK): encrypts DynamoDB token data
  pci-log-key (CMK): encrypts CloudWatch Logs in CDE
  pci-s3-key (CMK): encrypts S3 transaction receipts

Key access logging: every KMS Decrypt/Encrypt call logged to CloudTrail

In Use (P2PE):

PAN encrypted at POS terminal:
  Terminal has Terminal Master Key (TMK) derived from ZMK (in Payment Cryptography HSM)
  EMV chip: PAN + CVV + PIN block all encrypted before leaving terminal
  API receives only: encrypted data block + terminal serial number
  Payment Cryptography: decrypts with TMK, re-encrypts with internal DEK for tokenization
  Plaintext PAN exists only within Payment Cryptography HSM (never in Lambda runtime)

3. API Security

API Gateway security layers:
  1. WAF: IP reputation, OWASP rules, rate limit (100 req/min per IP for non-burst)
  2. mTLS: POS terminal client certificate from ACM Private CA (terminal registration)
  3. JWT authorizer (Lambda): validates Cognito JWT for CNP merchant requests
  4. Request validation: built-in API Gateway request schema validation
  5. Payload encryption: CNP requests have encrypted PAN (client-side tokenization JS)
  6. Response sanitization: API Gateway transforms responses to remove sensitive fields

Fraud prevention:
  Request fingerprinting: Lambda checks: device fingerprint, IP, velocity (DynamoDB DAX)
  Velocity rules:
    > 10 transactions/minute per card: flag + additional verification
    > 50 transactions/hour from one IP: auto-block IP (WAF dynamic IP block)
    New device + new card in < 5 min: 3DS challenge triggered

4. Logging and Audit

PCI-DSS Requirement 10: Audit logging

API Gateway: log request ID, timestamp, IP, HTTP method, path, response code
  NOT logged: request body (contains encrypted PAN), authorization headers

Lambda: structured JSON logging
  Included: transaction_id, token_id, merchant_id, timestamp, operation
  Explicitly excluded: PAN, CVV, PIN, full card number (structured logging library enforces)

CloudTrail:
  Management events: all API calls (IAM, KMS, DynamoDB admin)
  Data events:
    - S3:GetObject/PutObject on PCI S3 buckets
    - DynamoDB:GetItem/PutItem on token vault table
    - KMS:Decrypt/Encrypt calls

Log retention:
  CloudWatch Logs: 90 days (hot tier)
  Kinesis → S3 (Glacier after 90 days): 12 months (PCI-DSS Req 10 requires 1 year)
  S3 Object Lock (WORM): logs cannot be deleted or modified

Security Lake:
  All CDE logs normalized to OCSF
  QSA evidence package: Athena query for any access to token vault over past 12 months

5. Lambda Security Hardening

Lambda security configuration:
  VPC: all CDE functions in CDE VPC (private subnets, no internet)
  Execution role: least privilege (specific DynamoDB table ARN + Payment Cryptography ARN)
  Reserved concurrency: CDE functions have reserved concurrency to prevent noisy neighbor
  Environment variables: NO secrets in env vars; use Secrets Manager (via VPC endpoint)
  Code signing: Lambda code signing with AWS Signer (code must be signed by CDE deployment role)
  Dead Letter Queue: failed transactions → encrypted SQS queue → Lambda retry

Lambda resource policy:
  Principal: API Gateway source account only (no public Lambda URL)
  Condition: aws:SourceArn matching specific API Gateway ARN

Scale and Resilience (Phase 4 — 10 min)

Lambda at 10,000 TPS:

  • Default Lambda concurrency limit: 1,000 per account (per region). For 10,000 TPS: request limit increase to 10,000 concurrent Lambdas. Use provisioned concurrency for the tokenization function (eliminates cold-start latency for payment critical path).
  • Lambda concurrency = (requests per second × average execution duration). If tokenization Lambda runs in 20ms: 10,000 × 0.02 = 200 concurrent Lambdas at peak. Very manageable.

DynamoDB at 10,000 TPS:

  • DynamoDB on-demand mode: scales to 10,000+ TPS without capacity planning. For predictable load, provisioned capacity with Auto Scaling (20% buffer above peak) is more cost-effective.
  • DynamoDB Accelerator (DAX): for repeated reads of the same token (velocity checks), DAX cache reduces DynamoDB read load to near-zero. Cache TTL: 5 minutes for fraud velocity checks.

Active-Passive DR:

  • DynamoDB Global Tables replication to us-west-2 (< 1 second replication lag).
  • Lambda functions deployed in both regions via CodePipeline cross-region deployments.
  • Route 53 health check: DNS failover from us-east-1 to us-west-2 in < 60 seconds.
  • Payment Cryptography: available in both regions; key sync is not automatic — keys must be loaded separately in DR region (key ceremony required).

Trade-off Analysis — The Hard Choices

Decision 1: AWS Payment Cryptography vs. CloudHSM

Factor AWS Payment Cryptography CloudHSM
PCI-DSS compliance Purpose-built for payment (P2PE, EMV, ISO 8583) General HSM; PCI HSM for payment requires additional setup
Operations Fully managed (no HSM cluster management) You manage HSM cluster (patching, backups, HA)
Cost Per-call pricing (no idle cost) $1.60/hour per HSM instance (always on)
Supported payment operations Native: FPE, Pin Translation, EMV cryptograms, ZMK Custom PKCS#11 — must implement payment functions
Recommendation AWS Payment Cryptography for new payment systems; CloudHSM only if you need full PKCS#11 flexibility or existing key block format compatibility

Decision 2: FPE Tokenization vs. Random Token Tokenization

Factor Format-Preserving Encryption (FPE) Token Random Token
Format Looks like a real PAN (passes Luhn) Random string — doesn't look like PAN
Downstream compatibility Can be processed by legacy systems that validate Luhn May break legacy downstream systems
Security Slight information leakage (card BIN preserved) No information leakage
PCI scope FPE tokens are still in PCI scope if reversible Random tokens: system that holds mapping is CDE, but tokens themselves have no value
Recommendation FPE tokens for payment rail compatibility (legacy POS and acquirer systems expect PAN-format); random tokens preferred where compatibility allows

Decision 3: Lambda in VPC vs. Lambda without VPC

Factor Lambda in VPC Lambda without VPC
Cold start Higher (15-65ms additional due to ENI creation) Lower
VPC endpoint access Can reach DynamoDB/KMS via private endpoint Uses public endpoints
PCI compliance Required (CDE isolation demands VPC) Not PCI-compliant for CDE functions
Recommendation VPC required for all CDE functions; use provisioned concurrency to eliminate cold start penalty

Decision 4: Synchronous vs. Asynchronous Payment Processing

Factor Synchronous (customer waits for response) Asynchronous (immediate ack, async processing)
User experience Better (instant authorization result) Slight delay (show "processing")
Reliability More complex retry logic at API layer Easier retry via SQS
PCI scope Response must include sensitive-free data Queue must be encrypted (also CDE)
Recommendation Synchronous for authorization (card issuers expect real-time authorization responses < 2s); asynchronous for settlement (batch processing overnight)

Common Mistakes to Avoid

  1. Not distinguishing between card-present and card-not-present flows. These have fundamentally different security models. Card-present with P2PE means the PAN is never in plaintext on your systems. Card-not-present typically requires client-side tokenization (JavaScript library) before the PAN hits your API. Conflating these shows a lack of payment security depth.
  2. Forgetting about PCI-DSS scope reduction. The whole point of tokenization and P2PE is to minimize what's in CDE scope. If a system only ever sees tokens (never PANs), it may be out of scope. Candidates should explicitly say "PAN never enters these components" to demonstrate scope reduction thinking.
  3. Using CloudHSM when AWS Payment Cryptography is the right answer. CloudHSM is a general-purpose HSM. For payment-specific operations (EMV cryptogram verification, PIN block translation, ZMK key injection), AWS Payment Cryptography has native APIs. Using CloudHSM would require implementing these operations from scratch via PKCS#11.
  4. Not covering fraud prevention. Payment APIs are prime targets for carding attacks (automated card testing). Velocity checks, device fingerprinting, 3DS, and WAF rate limiting must be part of the design.

Likely Follow-up Questions

Q1: How does AWS Payment Cryptography differ from AWS KMS for payment use cases?
AWS KMS is a general-purpose key management service — it supports AES-256 encryption, RSA signing, and elliptic curve operations. It does not have payment-specific operations. AWS Payment Cryptography is purpose-built for the payments industry: it supports Format-Preserving Encryption (FPE/FF3-1) for tokenization, EMV card cryptogram generation and verification, PIN block generation and translation, MAC generation/verification (for ISO 8583 message authentication), and Zone Master Key (ZMK) injection workflows — all essential for PCI-DSS payment HSM compliance. Payment Cryptography is PCI PIN Security and PCI P2PE compliant, whereas CloudHSM requires additional custom implementation to achieve the same compliance level.

Q2: What happens if the AWS Payment Cryptography service experiences an outage?
AWS Payment Cryptography is a regional service. For business continuity: (1) Payment Cryptography is deployed in the DR region (us-west-2) with the same key hierarchy loaded (separate key ceremony required per region). (2) Route 53 health checks on the payment API endpoint; on failure, DNS failover to the DR region within 60 seconds. (3) DynamoDB Global Tables ensures the token vault is available in the DR region with near-real-time data. (4) The key implication: Payment Cryptography keys are NOT automatically replicated between regions — you must conduct a key ceremony in each region separately (load TMK, DEK, MAC keys into each region's HSM). This is a critical operational step that many forget to mention.

Q3: How do you prevent a developer from accidentally logging PANs in CloudWatch?
Defense in depth: (1) Lambda logging library: use a structured logging library (like aws-lambda-powertools) configured with a redaction list — any field named card_number, pan, cvv is automatically masked before logging. (2) Code review: PR checklist and automated lint rules flag any logging statement that references payment-sensitive variables. (3) CloudWatch Logs pattern detection: Lambda (triggered by CloudWatch Logs subscription filter) scans new log entries for Luhn-valid 16-digit numbers — if detected, alert immediately and expire the log stream. (4) Macie: S3 buckets containing exported logs are scanned by Macie for PAN patterns daily. Any detection triggers a P1 incident. (5) PCI QSA review: as part of annual PCI assessment, a QSA performs log review to confirm no CHD in logs.

Q4: How do you handle key rotation for Payment Cryptography keys?
Payment key rotation follows the ANSI X9.24 standard: Zone Master Keys (ZMK) must be rotated annually or on suspected compromise. Zone Encryption Keys (ZEK/DEK) are typically rotated per-transaction or per-session (derived keys). Terminal Master Keys (TMK) are rotated via the DUKPT (Derived Unique Key Per Transaction) mechanism — each transaction uses a unique derived key from the base TMK, so there's no need to physically rotate TMKs frequently (only when the terminal's key space is exhausted or on suspected compromise). AWS Payment Cryptography supports key rotation via UpdateAlias (point to new key version) and provides key management APIs compatible with TR-31 key blocks. Re-encryption of stored data: DynamoDB encrypted with the old DEK is re-encrypted in batches via a Lambda migration job that reads, decrypts with old key, and re-encrypts with new key.

Q5: How do you achieve zero-downtime deployment of a Lambda payment function change?
Lambda deployment strategy for payment functions: (1) Lambda aliases with weighted traffic routing: deploy new Lambda version, shift 10% traffic to new version via alias weighted routing. Monitor error rate and latency metrics. Gradually increase to 100% or rollback if errors spike. (2) CodeDeploy Lambda hook: BeforeAllowTraffic hook runs integration tests against the new Lambda version in the CDE environment. Only if tests pass does CodeDeploy proceed with traffic shifting. (3) Blue/Green API Gateway: two API Gateway stages (blue/current, green/new); Route 53 weighted routing between them. On successful validation, shift all traffic and decommission blue. (4) PCI implication: every Lambda deployment is a code change requiring change control documentation — CodeDeploy provides the audit trail.

Q6: How does the 3D Secure (3DS) flow integrate with your serverless architecture?
3DS authentication: (1) Card-not-present checkout triggers 3DS challenge if risk score threshold exceeded (new device + new card + high amount). (2) Lambda (3DS Requestor) sends AReq (Authentication Request) via HTTPS to the card network's Directory Server. (3) Directory Server routes to the card issuer's ACS (Access Control Server). (4) ACS sends challenge to cardholder browser (SMS OTP, biometric, or frictionless). (5) On success, ACS returns ARes with authentication value (CAVV/AAV). (6) Lambda includes CAVV in the authorization request to the acquirer. (7) This integration uses the EMVCo 3DS 2.x protocol. The sensitive 3DS data (device fingerprint, browser data) flows through API Gateway but does not contain PAN — 3DS uses a token for the device channel. PCI scope: 3DS components are PCI 3DS compliant (separate standard from PCI-DSS).

Q7: How do you handle a PAN data breach notification requirement?
PCI-DSS Req 12.10 requires a documented incident response plan. For a confirmed PAN breach: (1) Immediate containment (< 1 hour): revoke compromised Lambda execution role; rotate KMS CMK; block suspicious IPs at WAF; isolate the affected Lambda function (remove from API Gateway). (2) Notification timeline: notify acquiring bank and card networks (Visa, Mastercard) within 24 hours of confirmed breach. PCI-DSS requires notification within 72 hours. (3) Forensic investigation: preserve all CloudTrail logs, VPC Flow Logs, Lambda access logs with S3 Object Lock; engage PCI Forensic Investigator (PFI) — PCI-DSS requires a PFI for Level 1 breaches. (4) Cardholder notification: per applicable state laws and PCI-DSS; card networks may issue mass re-issuance of compromised cards. (5) Root cause: patch, pen test, re-assess PCI compliance before going back to production.

Q8: How do you handle PCI-DSS annual assessment (QSA on-site audit) for a serverless system?
PCI-DSS Level 1 requires an annual Report on Compliance (ROC) by a QSA. For serverless: (1) Scope evidence: network diagram showing CDE boundary (CDE VPC, no internet, VPC endpoints); demonstrate PANs flow through Payment Cryptography (managed HSM — PCI-validated) and never touch non-CDE infrastructure. (2) Key management: AWS Payment Cryptography has PCI PIN Security Program listing — QSA accepts this as meeting Req 3 HSM requirements. (3) Access control: demonstrate no developer has access to CDE — IAM Identity Center shows only Lambda execution roles + specific operational roles can access CDE VPC and DynamoDB token vault. (4) Logging: demonstrate 12 months of immutable CloudTrail logs (S3 Object Lock) and show the KMS Decrypt audit trail for any CHD access. (5) Penetration testing: AWS provides a penetration testing policy; engage a PCI-ASV for quarterly vulnerability scans and annual pen test. (6) Shared responsibility: AWS provides SOC 2 Type II reports and PCI AOC (Attestation of Compliance) for the underlying infrastructure; your QSA only needs to assess your application-level controls.
