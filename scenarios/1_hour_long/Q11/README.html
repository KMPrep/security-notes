Q11 — Secure Machine Learning Pipeline on AWS

The Ask

"Design a secure machine learning platform for a healthcare company that trains models
on HIPAA-regulated patient data. The platform must support 50 data scientists, prevent
data exfiltration during model training, protect model artifacts and intellectual
property, detect model tampering, and provide a compliant audit trail for FDA regulatory
submissions. Models are trained on a mix of structured EHR (Electronic Health Record)
data and unstructured clinical notes."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What is the sensitivity tier of the training data — de-identified, limited dataset, or fully identified PHI (Protected Health Information)?
    Impact: Fully identified PHI requires strictest controls (HIPAA §164.312 technical safeguards). De-identified data (per 18 Safe Harbor identifiers) is outside HIPAA scope but likely still sensitive IP. Limited datasets (dates retained, no direct identifiers) have intermediate controls. This determines whether SageMaker processing jobs need to run in a VPC-only mode and whether we need Macie scanning on the training data buckets.

  • Q: Are models being used for clinical decision support (FDA regulated 510(k) device) or internal analytics?
    Impact: FDA-regulated ML models (Software as a Medical Device — SaMD) require: complete model training reproducibility, version-controlled training datasets, change control for model updates, and full audit trail traceable to specific training runs. This drives Git-based model versioning, DVC (Data Version Control) integration, and SageMaker Experiments tracking.

  • Q: Do data scientists need access to raw PHI for exploratory analysis, or is PHI access limited to automated training pipelines?
    Impact: If data scientists access raw PHI: need SageMaker Studio with VPC-only mode + no direct S3 access to PHI buckets + session recordings. If PHI access is pipeline-only: data scientists work with de-identified or synthetic datasets; only automated SageMaker Training Jobs access PHI.

  • Q: What model types are expected — deep learning (GPU-intensive, long-running jobs) or traditional ML (CPU, short jobs)?
    Impact: Deep learning: multi-GPU training instances (p4d.24xlarge), distributed training (SageMaker data parallelism), potentially weeks-long training runs. Traditional ML: smaller instances, faster iteration. Both need the same security controls but resource isolation and cost management differ significantly.

  • Q: Is there an existing data lake (Lake Formation) or are data scientists using S3 buckets directly?
    Impact: Lake Formation with column-level and row-level security is the right architecture for controlled data access. Raw S3 buckets with bucket policies are harder to audit and don't provide fine-grained access. Lake Formation also provides data lineage (integration with Apache Atlas / AWS Glue data lineage).

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Training data: fully identified PHI (strictest controls).
  2. Models: clinical decision support (FDA 21 CFR Part 11 considerations for audit trail).
  3. Data scientists use SageMaker Studio (JupyterLab) — no access to raw PHI; PHI flows only through automated pipelines.
  4. Lake Formation manages PHI data access with column masking for data science exploration.
  5. SageMaker in VPC-only mode (no internet access during training).

Core Architecture — Data Flow Narrative

Data Ingestion → Data Governance Layer → Feature Engineering → Model Training → Model Registry → Deployment

[PHI Data Sources → S3 PHI Zone]
  EHR System (Epic) → AWS Transfer Family (SFTP, TLS 1.3) → S3 PHI Raw bucket
    S3 encryption: SSE-KMS (phi-raw-key)
    S3 Object Lock: WORM (regulatory retention)
    Macie: scans for PHI patterns (names, dates, MRNs, SSNs) on landing
    VPC endpoint access only (no internet S3 access)

[AWS Lake Formation — Data Governance]
  Lake Formation catalog governs all PHI data:
    Table: patient_records (EHR structured)
    Table: clinical_notes (unstructured text)

  Column-level security:
    - patient_name, dob, ssn, mrn: MASKED (data scientists see masked values)
    - diagnosis, labs, vitals, medications: readable by data scientists (de-identified)
    - raw_notes_text: readable only by NLP training pipeline (not data scientists)

  Row-level filter:
    - Data scientists in Research group: patients with research_consent = true only
    - Training pipeline: all patients (automated, not human)

  Data lineage:
    - Lake Formation + AWS Glue Data Catalog: every table → dataset version
    - FDA traceability: training run → dataset snapshot → model artifact

[SageMaker Studio — Data Scientist Workspace]
  Studio Domain: VPC-only mode
    - Private subnets only (no internet access)
    - VPC endpoints: S3 (gateway), SageMaker API, SageMaker Runtime, KMS, ECR
    - Studio execution role: access to de-identified Lake Formation tables only
    - No direct S3 PutObject to PHI buckets
    - EFS home directory: per-user isolated volume, encrypted with KMS

  Notebook code:
    - Queried via Athena (federated Lake Formation access) or SageMaker Processing Job
    - No local copy of PHI on EFS
    - SageMaker notebook instance isolation: dedicated execution role per user

[SageMaker Training Pipeline (PHI access)]
  SageMaker Pipelines (automated, no human access to PHI):
    Step 1: SageMaker Processing Job
      Input: S3 PHI bucket (via Lake Formation IAM role)
      Task: de-identify PHI (use Amazon Comprehend Medical to detect + mask PII)
      Output: de-identified features to S3 Features bucket
      VPC: CDE VPC, no internet

    Step 2: SageMaker Training Job
      Input: S3 Features bucket (de-identified)
      Instance type: ml.p4d.24xlarge (NVIDIA A100 GPUs)
      VPC: CDE VPC, no internet
      Network isolation mode: ENABLED (no outbound network from training container)
      Encryption: inter-node training traffic encrypted (TLS 1.3)
      KMS: training output volume encrypted with sagemaker-training-key
      Output: model artifact → S3 Model Artifacts bucket (SSE-KMS: sagemaker-model-key)

    Step 3: SageMaker Clarify (bias + explainability)
      Run bias detection + SHAP explainability report
      Output: report stored with model artifact (FDA explainability requirement)

    Step 4: SageMaker Model Monitor (baseline)
      Establish data quality baseline for production monitoring

    Step 5: SageMaker Model Registry
      Register model version with:
        - Training dataset version (S3 URI + SHA256 hash)
        - Training code version (Git commit hash)
        - Bias report, explainability report
        - Model accuracy metrics
        - Approval status: PendingManualApproval

[Model Approval Workflow]
  SageMaker Model Registry → EventBridge → Lambda → ServiceNow approval
  Approvers: Clinical Data Science Lead + Medical Informatics Officer
  On approval → SageMaker Pipeline: deploy model to Staging endpoint
  On staging validation pass → deploy to Production endpoint

[SageMaker Endpoints — Inference]
  VPC-only endpoints (data plane)
  Endpoint encryption: KMS (sagemaker-endpoint-key)
  Model Monitor: detect data drift, concept drift (alert if input distribution shifts)
  Access: API Gateway → Lambda → SageMaker Runtime (PrivateLink)
  Inference results: stored in S3 Inference Results (SSE-KMS, audit trail)

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

IAM roles in the ML platform:

sagemaker-studio-execution-role (per data scientist):
  Allow: athena:StartQueryExecution (Lake Formation tables, de-identified only)
  Allow: sagemaker:CreateTrainingJob (dev/staging only, not PHI training)
  Allow: s3:GetObject (s3://features-bucket/* only — no PHI bucket access)
  Deny: s3:GetObject, s3:ListBucket on PHI raw bucket
  Deny: sagemaker:CreateEndpointConfig on production accounts
  IAM condition: sagemaker:VpcSubnets required (enforce VPC mode)

sagemaker-phi-pipeline-role (automated pipeline only):
  Allow: s3:GetObject (s3://phi-raw-bucket/*)
  Allow: s3:PutObject (s3://features-bucket/*)
  Allow: comprehendmedical:DetectPHI
  Allow: lakeformation:GetDataAccess
  Deny: IAM actions, no ability to escalate

sagemaker-model-registry-role:
  Allow: sagemaker:CreateModelPackage, sagemaker:UpdateModelPackage
  Allow: sagemaker:DescribeModelPackageGroup
  Deny: sagemaker:CreateModel (model not deployed until approved)

SCP on ML accounts:
  Deny: sagemaker:CreateTrainingJob unless VpcConfig is specified (force VPC mode)
  Deny: sagemaker:CreateNotebookInstance (use Studio, not classic notebooks)

2. Encryption Strategy

KMS CMK hierarchy for ML platform:
  phi-raw-key: encrypts S3 PHI Raw bucket
    Key policy: phi-pipeline-role only (no data scientist access)

  features-key: encrypts S3 Features bucket (de-identified)
    Key policy: phi-pipeline-role, studio-roles, training-role

  sagemaker-training-key: encrypts training job output volumes + inter-node traffic
    Key policy: sagemaker-phi-pipeline-role only

  sagemaker-model-key: encrypts model artifacts in S3 Model Artifacts bucket
    Key policy: pipeline-role, model-registry-role, endpoint-role

  sagemaker-endpoint-key: encrypts endpoint model data volumes
    Key policy: endpoint-role, inference-gateway-role

Training job encryption:
  SageMaker Training Job:
    VolumeKmsKeyId: sagemaker-training-key
    EnableInterContainerTrafficEncryption: true (inter-GPU TLS)

3. Data Exfiltration Prevention

SageMaker Studio VPC-only controls:
  1. No internet gateway in ML VPC (training subnet has no IGW route)
  2. VPC endpoints for all required services (S3, SageMaker API, ECR, KMS, CloudWatch)
  3. NACLs: deny all outbound to 0.0.0.0/0 except VPC endpoints

SageMaker Training Job network isolation:
  EnableNetworkIsolation: true
    → Training container has NO network access at all
    → Cannot upload data to external endpoints
    → Input data provided via encrypted S3 channels before isolation

Custom ECR container policy:
  Only approved base images from internal ECR registry
  Image signing via AWS Signer: training containers must be signed
  SCP denies SageMaker Training Jobs with external ECR image URIs

S3 exfiltration prevention:
  S3 bucket policy: allow access only from VPC endpoint (aws:sourceVpce condition)
  S3 MFA Delete: enabled on PHI Raw bucket
  S3 Block Public Access: enabled org-wide (SCP enforces this)
  AWS Config rule: s3-bucket-public-read-prohibited

VPC S3 endpoint policy (restricts what data scientists can upload to):
  Allow s3:PutObject only to approved bucket ARNs (features, results)
  Deny s3:PutObject to any external buckets (including personal S3)

4. Model Artifact Integrity

Model artifact signing (anti-tampering):
  After training job completes:
    Lambda: calculate SHA256 hash of model.tar.gz
    Store hash in DynamoDB (model-registry-table) with training run metadata
    Store hash in SageMaker Model Package (additional metadata)

  Before deployment (Model Registry approval check):
    Lambda: recalculate SHA256 of model artifact
    Compare to stored hash → if mismatch: block deployment + P1 alert

  Before endpoint inference:
    SageMaker Model Monitor: validate model artifact hash at endpoint load time

S3 Object Lock on model artifacts:
  Governance mode: models cannot be deleted/overwritten for retention period
  Retention: minimum 10 years (FDA Medical Device regulations)

5. FDA 21 CFR Part 11 Audit Trail

Complete audit trail for each model version:

Training run metadata (SageMaker Experiments):
  - Training job ARN
  - Input dataset S3 URI + version
  - Input dataset SHA256 hash
  - Training code Git commit hash (GitHub)
  - Hyperparameters
  - Training start/end time, instance type
  - Output model artifact S3 URI + SHA256 hash
  - SageMaker Clarify: bias report S3 URI
  - SageMaker Clarify: explainability report S3 URI

Model Registry package:
  - Above training run metadata
  - Validation accuracy metrics
  - Approval record (who, when, justification)
  - Deployment history (staging → production timestamps)

CloudTrail data events:
  - Every GetObject on PHI raw bucket
  - Every PutObject on model artifacts bucket
  - KMS Decrypt calls for PHI key

All audit records:
  S3 Object Lock: WORM for 10 years
  CloudTrail: Organization trail → S3 + Security Lake
  Athena query: generate FDA audit package for any model version

Scale and Resilience (Phase 4 — 10 min)

50 data scientists:

  • SageMaker Studio Domain: single domain, 50 user profiles. Each profile has isolated EFS home directory (separate EFS access point, encrypted with per-user KMS key). Data scientists cannot read each other's notebooks.
  • SageMaker Studio execution roles: per-team (Frontend ML team, Clinical NLP team, etc.) — teams share roles but with different Lake Formation permissions per team group.

GPU training jobs:

  • SageMaker Training: p4d.24xlarge (8x A100 GPUs). For distributed training: SageMaker data parallelism library (SMDDP) splits dataset across multiple instances.
  • Training job quotas: 50 concurrent training jobs per account. Request quota increase for batch training campaigns.
  • Spot training: SageMaker managed spot training (60-70% cost reduction) with checkpointing enabled so jobs resume after spot interruption. Checkpoints encrypted with sagemaker-training-key.

Model serving:

  • SageMaker Endpoints: multi-model endpoint (host 100 model variants on single endpoint fleet) — cost-efficient for multiple clinical model versions.
  • Auto-scaling: SageMaker Application Auto Scaling based on InvocationsPerInstance metric.

Common Mistakes to Avoid

  1. Not distinguishing who can access PHI vs. what systems can. Many candidates say "we encrypt the PHI and give data scientists access with MFA." The HIPAA minimum necessary principle says data scientists should access the minimum data necessary — for most ML tasks, that's de-identified or column-masked data via Lake Formation, not raw PHI. Only automated pipelines should access raw PHI.
  2. Forgetting SageMaker network isolation mode. EnableNetworkIsolation: true on SageMaker Training Jobs prevents the training container from making outbound network connections. This is the primary control against a malicious training script exfiltrating data. Not mentioning this is a significant gap.
  3. Not addressing model artifact integrity. ML models can be tampered with after training (adversarial backdoor insertion). SHA256 hash verification at each pipeline stage, S3 Object Lock, and CloudTrail monitoring of model artifacts are essential controls that candidates often omit.
  4. Treating SageMaker Studio as automatically secure. By default, SageMaker Studio can be launched without VPC mode — this means notebooks have internet access and can download PHI to external services. VPC-only mode is NOT the default; it must be explicitly configured and enforced via SCP.

Likely Follow-up Questions

Q1: How does Amazon Comprehend Medical help with HIPAA compliance in clinical NLP?
Amazon Comprehend Medical is a HIPAA-eligible NLP service that can detect and extract PHI from clinical text — including patient names, dates, ages, medical record numbers, medications, diagnoses, and anatomical terms. For de-identification: the SageMaker Processing Job calls ComprehendMedical:DetectPHI on raw clinical notes, receives bounding boxes for all PHI entities, and replaces them with placeholder tokens (e.g., [PATIENT_NAME], [DATE]). The de-identified text is then stored in the Features S3 bucket for model training. This is more accurate than regex-based de-identification because Comprehend Medical understands clinical context (it correctly identifies "John Smith, 45M" as PHI while not flagging "Type 2 diabetes" as PHI).

Q2: How do you ensure model reproducibility for FDA submissions?
Reproducibility requires: (1) Exact training data: SageMaker Pipeline records the S3 URI and version ID of the training dataset. I also store the SHA256 hash of the training data snapshot (in DynamoDB Model Registry table) so the exact dataset can be re-retrieved. (2) Exact code: Git commit hash recorded in SageMaker Experiments. CodePipeline deploys only tagged releases (not branches) for production training runs. (3) Exact environment: Docker container image SHA digest (not tag, which is mutable) recorded. (4) Exact hyperparameters: SageMaker Experiments records all hyperparameters. (5) Deterministic training: for deep learning, set random seeds and use deterministic CUDA operations (may trade slight performance for reproducibility). With all of these recorded, the FDA reviewer can request a re-training run using the exact same dataset + code + environment and compare output model metrics.

Q3: How do you handle a data scientist who needs to debug a model failure that requires examining raw PHI?
Break-glass procedure for PHI access: (1) Data scientist submits a request explaining the medical necessity (specific patient case, debugging anomaly in model output). (2) HIPAA Privacy Officer + Clinical Lead approval (ServiceNow approval workflow). (3) On approval: time-limited SageMaker Studio profile with PHI access (8-hour TTL). All Studio session activity recorded (VPC Flow Logs, CloudTrail API calls). (4) Lake Formation audit: every query on PHI tables logged to CloudTrail. (5) After session: automatic PHI access revocation. Privacy Officer reviews the access report within 24 hours. (6) This is documented as a HIPAA minimum necessary authorization — the data scientist is accountable for using PHI only for the stated purpose.

Q4: How do you prevent a malicious training container from exfiltrating data through the model artifact?
Model artifacts (model.tar.gz) can be used as a covert channel — malicious training code could encode data into the model weights. Controls: (1) SageMaker network isolation (EnableNetworkIsolation: true): training container cannot reach external services during training — prevents direct exfiltration. (2) Model artifact size monitoring: CloudWatch alarm if model artifact is significantly larger than expected (baseline from previous versions). Large model files may indicate data hidden in weights. (3) Post-training scanning: SageMaker Processing Job runs a model inspection script that validates model architecture matches expected structure (layer count, parameter count). (4) S3 Object Lock on model artifacts: once stored, cannot be modified — prevents post-training insertion of data into the artifact. (5) Source code review: mandatory peer review for all training container code changes.

Q5: What's the security difference between SageMaker Classic Notebooks and SageMaker Studio?
SageMaker Classic Notebooks: each notebook is a managed EC2 instance, can have an internet-facing endpoint, and each instance runs independently. Classic Notebooks have known security gaps: internet access by default, can download data to the EBS volume, and there's no central user management. SageMaker Studio: centralized JupyterLab environment with a Domain that controls all user settings. Studio can be enforced to VPC-only mode at the Domain level, with a single SCP (Deny sagemaker:CreateNotebookInstance) preventing Classic Notebooks entirely. Studio uses EFS for shared storage (per-user isolated access points) and provides better audit logging via CloudTrail. For HIPAA environments: always use Studio, never Classic Notebooks, and enforce via SCP.

Q6: How does SageMaker Model Monitor detect if the input data distribution is shifting over time?
SageMaker Model Monitor works in three steps: (1) Baseline: after training, run a Baseline Processing Job on the training dataset to compute data statistics (mean, std dev, missing value %, feature distribution). This baseline is stored in S3. (2) Monitoring schedule: Model Monitor runs a monitoring job hourly (or daily) on recent inference requests. It samples a portion of real inference inputs and computes the same statistics. (3) Drift detection: Model Monitor compares current stats to baseline using configurable thresholds (e.g., KL-divergence for distribution shift, chi-square test for categorical features). If drift exceeds threshold, CloudWatch alarm fires → EventBridge → SNS → alert. For clinical ML specifically, I also set up a custom CloudWatch metric for model calibration score (predicted probability vs. actual outcomes from EHR feedback loop) — this detects concept drift even when input distribution appears stable.

Q7: How do you manage Python package dependencies in a VPC-only Studio environment?
VPC-only Studio cannot reach public PyPI, conda-forge, or GitHub. Solution: (1) AWS CodeArtifact as internal PyPI mirror. Security team reviews and approves package versions (scanned with Inspector v2 for known CVEs and Bandit for Python-specific security issues). (2) Approved package list: data scientists request new packages via Jira. Security review + functional review within 48 hours. On approval, package is mirrored to CodeArtifact. (3) SageMaker Studio custom lifecycle config: Studio kernels are configured to use CodeArtifact as the pip index (pip config set global.index-url https://<account>.d.codeartifact.us-east-1.amazonaws.com/pypi/ml-packages/simple/). (4) ECR base images: approved Docker base images in ECR (scanned by Inspector v2) for training containers. Data scientists extend only from approved base images.

Q8: How do you handle a model that produces unexpectedly biased predictions on a patient subgroup?
HIPAA and FDA both have patient equity concerns. Response: (1) SageMaker Clarify: post-training bias report already computes DPL (Demographic Parity Loss), WER (Weighted Error Rate) across patient subgroups (age group, gender, race/ethnicity). If any subgroup has error rate > 2x overall error rate, the model is flagged. (2) Model Registry approval gate: bias report is a required artifact. Approvers (Clinical Lead + Medical Informatics) must explicitly acknowledge the bias report before approving deployment. (3) Production monitoring: SageMaker Model Monitor includes custom bias monitoring (compare predictions for protected vs. non-protected groups in production). CloudWatch alarm if group disparity exceeds threshold. (4) Remediation: collect additional representative training data for the underperforming subgroup (may require targeted patient consent). Re-train with balanced dataset. Use techniques like SMOTE for synthetic minority oversampling or custom loss function weighting. (5) FDA notification: if the model is a SaMD device, unexpected bias may trigger FDA adverse event reporting (MDR) requirements.
