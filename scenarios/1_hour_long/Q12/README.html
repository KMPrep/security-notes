Q12 — Disaster Recovery with Security Controls Preserved

The Ask

"Design a disaster recovery architecture for a financial services company running a
core banking application across AWS us-east-1 (primary). The application must achieve
RTO of 15 minutes and RPO of 0 (zero data loss) when failing over to us-west-2. The
DR architecture must maintain full security control parity — the same WAF rules, IAM
policies, KMS keys, GuardDuty findings, and compliance posture — so that the DR region
is not a weaker attack surface than primary."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What constitutes a "disaster" that triggers failover — regional AWS outage only, or also application-level failures, data corruption, or ransomware?
    Impact: AWS regional outage: Region-level failover using Route 53 ARC (Application Recovery Controller). Application failure: health-check-based failover (Route 53 health checks + CloudWatch alarms). Ransomware: need clean-room recovery from immutable backups (S3 Object Lock), not failover (failover would propagate the ransomware). These require different DR runbooks.

  • Q: Is the core banking application stateless microservices, a monolithic application, or a mix?
    Impact: Stateless microservices: simpler DR — redeploy containers from ECR in DR region; no in-flight state. Stateful monolith: must replicate all session state, in-memory caches, and database state. Mix: need per-component DR strategy. For "zero data loss" (RPO=0), every stateful component needs synchronous replication or multi-region active setup.

  • Q: What databases are in use — RDS, Aurora, DynamoDB, or others?
    Impact: Aurora Global Database supports synchronous write forwarding from secondary region (sub-second RPO). RDS Multi-AZ does NOT support multi-region (only RDS read replicas, which have async replication → RPO > 0). DynamoDB Global Tables: sub-second async replication (not truly RPO=0). For RPO=0 in Aurora: Aurora Global Database with synchronous write forwarding.

  • Q: Does the DR plan need to comply with FFIEC (Federal Financial Institutions Examination Council) BCP/DR guidelines or SOX?
    Impact: FFIEC requires annual DR tests (not just documented plans), RTO/RPO objectives in BCPs, and evidence of testing. SOX requires DR test evidence as part of IT General Controls (ITGC). This drives the DR testing cadence and documentation requirements.

  • Q: Who initiates failover — fully automated, or human decision required?
    Impact: Fully automated failover is faster (Route 53 ARC automation) but risks false positives (failover during brief transient issues). Human-in-the-loop with pre-authorized runbook is safer for financial services but may miss 15-minute RTO. Common pattern: automated health checks trigger alert + 2-minute human confirmation window before automatic failover.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Core banking: microservices on ECS Fargate + Aurora Global Database.
  2. RPO = 0: Aurora Global Database synchronous write forwarding.
  3. RTO = 15 minutes: automated failover with Route 53 ARC.
  4. DR region: us-west-2 (all security services pre-deployed in warm standby).
  5. Security parity: all security controls are Terraform-managed and deployed identically to both regions.

Core Architecture — Data Flow Narrative

Primary (us-east-1) and DR (us-west-2) — Always-On Warm Standby

[SECURITY PARITY FRAMEWORK — Both regions identical]
Infrastructure-as-Code (Terraform):
  - All security controls defined as code (no manual config)
  - Single Terraform state, applied to both regions
  - GitOps: policy changes deployed to BOTH regions simultaneously
  - Diff check: Lambda nightly compares security configs between regions

[Application Layer]
  us-east-1 (Active):
    - ECS Fargate (2,000 tasks): core banking services
    - ALB + CloudFront: user-facing traffic
    - Aurora Global Database Cluster (writer)

  us-west-2 (Warm Standby):
    - ECS Fargate (200 tasks — minimum): services running at 10% capacity
    - ALB + CloudFront: DNS weight = 0 (no traffic in normal operations)
    - Aurora Global Database: read replica (synchronous write forwarding from primary)

  Route 53 ARC (Application Recovery Controller):
    - Readiness checks on us-west-2: verify ECS task count, Aurora replica lag, DNS routing
    - Routing controls: enable/disable us-west-2 in the Route 53 routing policy
    - Manual override: Security + Engineering leads can trigger via ARC console or API
    - Automated trigger: CloudWatch alarm (us-east-1 unhealthy for 5 minutes) → EventBridge → Lambda → ARC API: enable us-west-2

[DATABASE REPLICATION — RPO=0]
  Aurora Global Database:
    Cluster: core-banking-global-cluster
    Primary cluster: us-east-1 (writer + 2 readers)
    Secondary cluster: us-west-2 (reader replica)

    Synchronous write forwarding:
      Application writes to us-west-2 Aurora endpoint → write forwarded to us-east-1
      Transaction commits only after us-east-1 confirms
      Provides RPO=0 even for writes during partial regional degradation

    Replication lag: < 1 second (async physical replication)
    On DR failover: promote us-west-2 cluster to writer (< 1 minute via AWS console/API/Terraform)

    Note: Write forwarding adds ~30ms latency (us-east-1 to us-west-2 RTT). For DR drills,
    run full writes against us-west-2 to verify write path.

[DynamoDB Global Tables — Session State + Config]
  Tables: user-sessions, feature-flags, rate-limit-state
  Configuration: Global Tables (us-east-1 + us-west-2)
  Replication: async (< 1 second replication lag)
  RPO: < 1 second for DynamoDB (not true zero, but acceptable for session state)

  Note: If strict RPO=0 is required for session state, use Aurora with write forwarding
  instead of DynamoDB Global Tables.

[SECURITY CONTROLS — PRIMARY (us-east-1)]
  GuardDuty: enabled, delegated admin in Security account
  Security Hub: org-level aggregation, us-east-1 primary aggregation region
  WAF v2: attached to CloudFront (global) + ALB (regional us-east-1)
  KMS CMK: us-east-1 KMS keys (core-banking-key, aurora-key, s3-key)
  IAM Identity Center: Management account (global, not regional)
  CloudTrail: org-level trail → S3 us-east-1 (replicated to us-west-2)
  Config: org-level aggregator in us-east-1
  Macie: enabled on PHI S3 buckets in us-east-1

[SECURITY CONTROLS — DR (us-west-2) — IDENTICAL CONFIGURATION]
  GuardDuty: enabled in us-west-2 (same delegated admin account)
  Security Hub: cross-region finding replication to us-west-2
  WAF v2: same rules deployed to ALB in us-west-2 (Firewall Manager org policy)
  KMS Multi-Region Keys:
    core-banking-key: MRK with replica in us-west-2 (can decrypt us-east-1 encrypted data)
    aurora-key: MRK replica in us-west-2
    s3-key: MRK replica in us-west-2
  IAM Identity Center: global (same access in both regions)
  CloudTrail: same trail covers us-west-2 events (org trail is global)
  Config: us-west-2 recorder enabled (same conformance packs)
  Macie: enabled on DR S3 buckets in us-west-2

[KMS MULTI-REGION KEYS — CRITICAL FOR DR]
  Problem: us-east-1 KMS key cannot decrypt data in us-west-2 by default
  Solution: KMS Multi-Region Keys (MRK)
    - Create primary key in us-east-1
    - Replicate to us-west-2 (same key material, different ARN)
    - Aurora encrypted with MRK can be decrypted in us-west-2 post-failover
    - Same key policy (Terraform-managed) in both regions
    - CloudTrail: decrypt calls in us-west-2 logged to org trail

[FAILOVER RUNBOOK — RTO 15 MINUTES]
  T+0: CloudWatch alarm fires (us-east-1 ALB 5xx rate > 50% for 5 minutes)
  T+1min: EventBridge → Lambda → PagerDuty P1 alert to on-call team
  T+3min: On-call engineer confirms regional issue (or automated if pre-authorized)
  T+4min: Lambda triggers Route 53 ARC: EnableRoutingControl (us-west-2)
  T+5min: Route 53 DNS update propagates (TTL: 60 seconds)
  T+6min: Traffic shifting to us-west-2 (CloudFront + ALB)
  T+8min: ECS Fargate in us-west-2 scales from 200 → 2,000 tasks (Auto Scaling)
  T+10min: Aurora us-west-2 cluster promoted to writer (DBA executes or automated)
  T+12min: Application fully operational in us-west-2
  T+15min: Health checks green → RTO achieved

[SECURITY VERIFICATION POST-FAILOVER]
  Lambda "security parity check" runs automatically at T+15min:
    - Verify GuardDuty enabled in us-west-2
    - Verify WAF rules deployed (Firewall Manager status)
    - Verify KMS MRK accessible
    - Verify Config conformance pack compliant
    - Verify Security Hub aggregating us-west-2 findings
  If any check fails: alert on-call security engineer immediately

Security Deep Dive (Phase 3 — 15 min)

1. KMS Multi-Region Key Strategy

Problem: If Aurora is encrypted with a us-east-1 KMS key (key ARN: arn:aws:kms:us-east-1:...)
and the Aurora replica is in us-west-2, the Aurora data in us-west-2 cannot be decrypted
using the us-east-1 key (cross-region KMS calls are not supported).

Solution: KMS Multi-Region Keys (MRK)
  Primary key: arn:aws:kms:us-east-1:account:key/mrk-xxxxxxxx
  Replica key: arn:aws:kms:us-west-2:account:key/mrk-xxxxxxxx (same key material, same key ID prefix)

  Key policy (Terraform, identical in both regions):
    Allow: aurora.amazonaws.com to use the key
    Allow: core-banking-app IAM role to Decrypt (for app-layer decryption)
    Allow: security-audit-role to DescribeKey (not Decrypt)
    Deny: all other principals

  CloudTrail: every Decrypt in us-west-2 logged with the replica key ARN
  Note: MRK key material is THE SAME in both regions (not separate keys)

  Key rotation: annual (synchronizes across all replicas automatically)

2. GuardDuty Threat Detection During DR Events

Security operations during failover:
  GuardDuty findings from both us-east-1 and us-west-2 aggregate in Security account

  Known false positive during DR:
    GuardDuty may flag: "Unusual API calls from new region" when workloads shift
    Pre-suppress this pattern: create suppression rule for ECS task roles in us-west-2
    during DR drill (prevents alert fatigue during planned DR tests)

  Real threats during DR events:
    Threat actors know companies are distracted during DR events
    GuardDuty monitoring of DR region CRITICAL:
      - Credential abuse (attacker tries to use stolen creds while defenses are focused elsewhere)
      - S3 data staging (attacker knows DR region may have weaker monitoring)
      - Lateral movement (pivot from us-east-1 compromised to us-west-2)

    Response: Security team maintains separate DR incident queue during failover
    DR event does not pause GuardDuty or disable security monitoring

3. IAM and Identity Continuity

IAM Identity Center (global service):
  - IAM Identity Center is NOT regional — it operates from the Management account globally
  - During us-east-1 regional failure, users can still assume roles in us-west-2 via IdC
  - Permission Sets are applied to all accounts (both regions) — no regional config needed

Service-level IAM roles:
  - ECS Task roles are identical in both regions (Terraform-managed)
  - IAM policies reference ARNs — for cross-region resources, use the region-specific ARN
    or condition: aws:RequestedRegion to allow only from the intended region
  - Break-glass: Security account admin can assume emergency roles in any account

4. DR Region Security Hardening

DR region is a prime target for attackers:
  Reasons: often less monitored, security controls may be less mature,
  engineers focus on primary region.

DR region hardening:
  1. Config conformance packs: same conformance packs as primary
     - Verify: CIS AWS Benchmarks, NIST 800-53
     - Alert on non-compliance
  2. Security Hub: findings from us-west-2 aggregated to Security account
  3. GuardDuty: active (not standby) — findings generated even with minimal traffic
  4. Inspector v2: enabled in us-west-2 (even minimal ECS tasks scanned)
  5. IAM Access Analyzer: org-level (already covers us-west-2 automatically)
  6. Nightly compliance check: Lambda compares Config compliance % between us-east-1 and us-west-2
     Alert if us-west-2 compliance < us-east-1 - 5%
  7. Red team exercise: annual DR red team test specifically targeting the DR region
     during a simulated failover event

5. CloudTrail and Logging During DR

CloudTrail Organization Trail (covers all regions by default):
  - us-west-2 events automatically included in org trail
  - S3 bucket for CloudTrail logs: us-east-1 primary + us-west-2 replica bucket
    (S3 Cross-Region Replication: us-east-1 → us-west-2)
  - Object Lock: WORM on both buckets
  - If us-east-1 is down, CloudTrail can't write to us-east-1 S3 bucket
    → CloudTrail backup delivery: secondary S3 bucket in us-west-2 (configured at trail level)

Security Lake:
  - Multi-region Security Lake: logs from both regions stored in Security Lake
  - During DR: us-west-2 logs continue to flow to Security Lake
  - Athena queries on Security Lake work even if us-east-1 is down

Backup of security data:
  - Config snapshots: delivered to S3 in both regions (Config delivers to S3 in same region)
  - GuardDuty findings export: enabled, exported to S3 in both regions (same bucket policy)

Scale and Resilience (Phase 4 — 10 min)

Aurora Global Database performance:

  • Write forwarding latency: approximately 30-50ms for writes from us-west-2 → us-east-1 (inter-region RTT). For core banking, this is acceptable for low-frequency writes (transaction confirmations). For high-frequency writes (tick data, real-time trading), this may not be acceptable — consider DynamoDB Global Tables or active-active multi-region.
  • Failover time: Aurora Global Database failover (promote secondary to writer) takes 1-2 minutes. Adding to the total RTO timeline.

ECS Fargate scale-up in DR:

  • ECS Application Auto Scaling in us-west-2: minimum = 200 tasks (warm standby), maximum = 2,000 tasks.
  • Scale-up time from 200 → 2,000: approximately 5-7 minutes (new Fargate task launch time).
  • Pre-warmed capacity: Fargate has unlimited capacity in principle, but pre-warming with 200 tasks ensures fast scale-up (existing tasks can handle initial traffic while more scale up).

Route 53 TTL strategy:

  • Normal operations: Route 53 TTL = 300 seconds (5 minutes) for cost efficiency.
  • DR mode: reduce TTL to 60 seconds 24 hours before any planned maintenance that might require failover. Automate TTL reduction via Lambda on-call runbook.

Trade-off Analysis — The Hard Choices

Recommendation MRK required for RTO < 1 hour; single-region key + re-encrypt is acceptable only for multi-day RTO (cold DR)

Common Mistakes to Avoid

  1. Using RDS read replicas and claiming RPO=0. RDS cross-region read replicas use asynchronous replication — there is always a replication lag (seconds to minutes). This means some committed transactions in us-east-1 may not have replicated to us-west-2 at the moment of failover. RPO=0 requires Aurora Global Database with synchronous write forwarding.
  2. Forgetting KMS Multi-Region Keys. If Aurora is encrypted with a regional KMS key and fails over to another region, the data cannot be decrypted — the key is not available in the other region. MRK is the specific solution for this. Many candidates describe encryption but forget the cross-region key availability problem.
  3. Not mentioning the DR region as a security threat surface. DR regions are often neglected (lighter monitoring, delayed patching, stale configurations). Attackers specifically target DR infrastructure during or after a disaster, knowing the security team is overwhelmed. Explicitly stating that all security controls (GuardDuty, WAF, Config, Inspector) run identically in both regions demonstrates mature DR security thinking.
  4. Treating Route 53 failover as equivalent to Route 53 ARC. Simple Route 53 health checks + failover routing is not the same as Route 53 ARC. ARC provides readiness checks (verify the DR region is actually ready), routing controls (manually toggle regions with guardrails), and a control plane that continues to function during a regional failure. Regular Route 53 health checks may fail confusingly during a partial regional failure.

Likely Follow-up Questions

Q1: How does Aurora Global Database write forwarding work and what are its limitations?
Aurora Global Database write forwarding allows write requests sent to a secondary cluster's writer endpoint to be transparently forwarded to the primary cluster for processing. The secondary cluster's writer acts as a proxy: it receives the write, forwards it over the AWS global backbone to the primary, waits for the commit, and returns the result. This provides RPO=0 because all committed data exists in the primary before the secondary acknowledges. Limitations: (1) ~30-50ms additional latency for writes (inter-region round trip). (2) Maximum 1 secondary cluster can participate in write forwarding at a time. (3) Write forwarding is only available for Aurora MySQL and Aurora PostgreSQL. (4) During primary region failure, write forwarding stops — the secondary must be promoted to accept writes directly.

Q2: How long does it take to promote an Aurora Global Database secondary cluster to primary?
Aurora Global Database promotion is a two-step process: (1) "Detach and promote" — removes the secondary cluster from the global database and promotes it to a standalone primary cluster. This takes approximately 1-2 minutes. (2) After promotion, the application must update its connection string to point to the new primary cluster endpoint. If using Route 53 CNAME records for database endpoints, this can be done in seconds (low TTL). In the RTO timeline: Aurora promotion at T+10 minutes, Route 53 DNS update for DB endpoint at T+11 minutes — completing well within the 15-minute RTO target. Post-DR: when us-east-1 recovers, re-attach it as a secondary in the new Global Database configuration.

Q3: How do you ensure KMS key policies are identical in both regions?
Terraform manages KMS key policies as code. The same Terraform module creates both the primary key (us-east-1) and the replica (us-west-2) with the same key policy JSON. The Terraform module uses a for_each over regions and applies the same aws_kms_key_policy resource to each. Drift detection: a nightly Lambda function calls kms:GetKeyPolicy on both the primary and replica keys, compares the policy JSON (normalized for region-specific ARNs), and alerts if they diverge. CloudTrail: any kms:PutKeyPolicy call is logged and triggers a Security Hub finding (via Config rule) if the changed policy diverges from the Terraform-managed baseline.

Q4: What happens to in-flight financial transactions during failover?
In-flight transactions (initiated in us-east-1 but not yet committed when failover occurs): (1) Transactions that were mid-execution but not committed in Aurora are rolled back automatically when the primary cluster fails. Aurora's write-ahead logging ensures durability of committed transactions. (2) The application must handle SocketTimeoutException or connection errors during failover and retry with idempotency tokens. All payment API calls include an idempotency key (UUID) — duplicate retry with the same key returns the same result without double-processing. (3) Message queues (SQS): messages that were received but not processed in us-east-1 before failover remain in the queue (unprocessed) and will be picked up by us-west-2 consumers after failover. SQS is a global service (well, regional, but all consumers reconnect automatically). (4) Customers may see a brief "payment processing" spinner for 15-30 seconds — acceptable with clear UX messaging.

Q5: How do you test DR without impacting production?
DR testing methodology: (1) Annual full DR test (FFIEC requirement): failover production traffic to us-west-2 (done during low-traffic window, typically 2am Sunday). Operate for 2 hours in DR region, verify all functions, failback. This is the most realistic test but requires coordination. (2) Quarterly functional tests (non-traffic-impacting): spin up a separate test environment that mirrors production, execute the failover runbook on the test environment, measure RTO against the target. (3) Continuous DR readiness checks: Route 53 ARC readiness checks run continuously — they verify that us-west-2 ECS task count, Aurora replica lag, DNS weights, and security control statuses are within DR thresholds. These are passive checks (no traffic). (4) Chaos engineering: AWS FIS (Fault Injection Simulator) experiments targeting non-production environments — simulate AZ failure, RDS failover, Lambda throttling.

Q6: How does Security Hub maintain findings continuity during a regional failover?
Security Hub has an aggregation region (configured at org level). During failover: (1) Security Hub in us-west-2 continues to generate findings from the now-active workloads (GuardDuty, Inspector, Config all continue running). (2) These us-west-2 findings are replicated to the Security Hub aggregation region (separate account — usually the Security account, which is in neither primary nor DR region — it uses Security Hub's own cross-region aggregation). (3) If us-east-1 Security Hub is temporarily unavailable, us-east-1 findings queue locally and are delivered when the aggregation region is reachable. (4) Analysts in the Security account see all findings (both regions) in a single Security Hub dashboard without configuration changes.

Q7: What is Elastic Disaster Recovery (DRS) and when would you use it?
AWS Elastic Disaster Recovery (DRS) is a service for replicating on-premises servers or EC2 instances to AWS as a DR target. It installs an agent on the source server that continuously replicates OS, application data, and disk state to a staging area in the target AWS region. On DR trigger: DRS launches recovery instances from the latest replicated state within minutes. DRS is most useful for: (1) Legacy stateful applications that can't use Aurora Global Database (running on EC2 instances with local databases). (2) On-premises to AWS DR scenarios (lift the on-premises system to AWS during a datacenter failure). For the core banking scenario (ECS Fargate + Aurora), DRS is NOT needed because: ECS containers are stateless (redeploy from ECR), and Aurora Global Database handles data replication natively.

Q8: How do you handle compliance evidence continuity during and after DR?
Regulatory continuity: (1) CloudTrail: org-level trail delivers to S3 in both regions. During DR, us-east-1 events may have gaps if CloudTrail delivery fails. DR events in us-west-2 are fully logged. Post-DR: recovery Lambda fills gaps by pulling CloudTrail logs from us-east-1 once it recovers. (2) Config: Config in us-west-2 records configuration changes during DR. The Config aggregator in the Security account collects from both regions. (3) Audit Manager: configured to collect evidence from both regions (separate assessment for each region with the same control framework). (4) SOX ITGC: document the DR event as a change in ServiceNow: root cause, timeline, affected systems, RTO/RPO achieved. Evidence that security controls were maintained throughout (Terraform showing identical configs in both regions; Security Hub showing findings in both regions throughout). (5) Regulators: financial regulators may need to be notified of significant outages — have a pre-drafted notification template ready.
