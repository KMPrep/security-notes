Q13 — Secure API Gateway Platform for Internal Microservices

The Ask

"Design a secure internal API gateway platform for a company with 200 microservices
communicating with each other inside AWS. Services run on ECS Fargate and Lambda.
The platform must enforce zero-trust (no implicit trust between services), mTLS for
all service-to-service communication, JWT-based caller identity, fine-grained
authorization, certificate lifecycle automation, and full audit logging of every
service call."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Are services exclusively within AWS (same VPC/account) or do some span on-premises or multi-cloud?
    Impact: Same VPC/account: VPC Lattice + private ALB. Multi-VPC or multi-account: VPC Lattice service network (PrivateLink-backed) handles cross-account. On-premises: Direct Connect + PrivateLink. Multi-cloud: VPC Lattice does not cross to Azure/GCP — need service mesh (App Mesh, Istio) or API Gateway (Internet-facing with mTLS).

  • Q: What is the expected inter-service call volume (requests per second) and latency SLA?
    Impact: 10k RPS with < 10ms P99 latency requirement: inline sidecar (Envoy) adds ~1ms; VPC Lattice managed by AWS (no sidecar overhead). 100k+ RPS: VPC Lattice scales natively; App Mesh on ECS adds sidecar latency. High-throughput gRPC streaming: VPC Lattice HTTP2/gRPC support. This determines whether we use App Mesh sidecars or VPC Lattice (no sidecar).

  • Q: Is JWT-based identity a new requirement or do services currently use IAM roles for service-to-service auth?
    Impact: If services currently use IAM roles (ECS Task Roles): IAM Sigv4 request signing is already in place. Adding JWT: requires a service identity token issuer (Cognito, or Keycloak). Hybrid: IAM for AWS service calls (S3, DynamoDB), JWT for service-to-service calls. The question implies JWT is a requirement — need to decide on the token issuer and claims structure.

  • Q: Does each microservice have a distinct identity, or do groups of services share an identity?
    Impact: Distinct per-service identity: each service gets its own TLS certificate (ACM Private CA, CN = service-name.internal) and its own JWT client credential. This is true zero-trust (service-level identity, not just team-level). Shared identity per team: coarser-grained, easier to manage but wider blast radius if a credential is compromised.

  • Q: What authorization model is required — RBAC (service A can call service B), or ABAC (service A can call service B with resource X)?
    Impact: RBAC (service-to-service call permissions): VPC Lattice auth policies (aws-principal condition per service) or Envoy RBAC filter. ABAC (resource-level): requires claims-based evaluation (JWT claim: resource ownership → Envoy ext_authz → OPA policy). ABAC is more powerful but requires an external policy engine (OPA, Cedar).

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. 200 microservices on ECS Fargate + Lambda in a multi-account AWS organization.
  2. VPC Lattice as the service mesh (no sidecars, managed by AWS).
  3. ACM Private CA for TLS certificates (mTLS client + server certs per service).
  4. JWT tokens issued by IAM Roles Anywhere (services outside AWS) or directly signed by IAM (ECS tasks using STS AssumeRoleWithWebIdentity → JWT).
  5. OPA (Open Policy Agent) as the centralized authorization policy engine.

Core Architecture — Data Flow Narrative

[ZERO TRUST PRINCIPLE]
No implicit trust between services:
  - Every service-to-service call requires both mTLS and JWT
  - mTLS: proves WHAT is connecting (client certificate = service identity)
  - JWT: proves WHO is calling and WHAT they want (claims = caller service + requested action)
  - Network: VPC security groups restrict ports; no broad "all internal traffic" rules

[SERVICE IDENTITY — ACM Private CA]
  Private CA hierarchy:
    Root CA: internal-root-ca (offline, Hardware Security Module)
    ├── Intermediate CA: east-services-ca (us-east-1, ACM Private CA, online)
    └── Intermediate CA: west-services-ca (us-west-2, ACM Private CA, online)

  Certificate issuance (automated):
    ECS Fargate: at task startup, ECS exec calls ACM PCA to issue a client certificate
    Certificate subject: CN=payment-service,OU=ecs-tasks,O=Corp
    TTL: 24 hours (short-lived; automated rotation via ACME or ACM-PCA API)
    Lambda: at invocation start (cold start), Lambda function role calls PCA → certificate
    Certificate stored: in-memory only (not persisted to disk or S3)

    Rotation automation:
      ECS: ACM PCA → Application Load Balancer (server cert);
           Secrets Manager → Lambda rotation function → new client cert every 24h
      Lambda: certificate fetched on each cold start; warm invocations reuse cert (< 24h old)

[mTLS ENFORCEMENT — VPC Lattice]
  VPC Lattice Service Network:
    - Deployed in each VPC (service network attachment)
    - All 200 services registered as VPC Lattice services
    - VPC Lattice listener: HTTPS port 443, TLS mode = mTLS required
    - Client certificate validation: against ACM Private CA trust store
    - Server certificate: issued by ACM Private CA per service

  Service routing:
    payment-service.internal → VPC Lattice service ARN → target group → ECS tasks
    No direct IP-to-IP communication; all traffic through VPC Lattice

  VPC Lattice auth policy (pre-authorization, service-level):
    resource "aws_vpclattice_auth_policy" "payment_service" {
      Statement:
        Principal: { AWS: "arn:aws:iam::account:role/order-service-task-role" }
        Action: "vpc-lattice-svcs:Invoke"
        Condition: { "StringEquals": { "vpc-lattice-svcs:ServiceNetworkArn": "arn:..." } }
    }
    → Only services with an explicitly allowed IAM principal can invoke payment-service

[JWT IDENTITY — Token Issuance and Validation]
  Service tokens issued by internal token service:
    Token service: a dedicated ECS service (token-authority-service)
      Input: ECS task role credential (IAM SigV4 signed request proves identity)
      Output: signed JWT with claims:
        { "sub": "ecs:payment-service:task/xyz",
          "service": "payment-service",
          "account": "123456789012",
          "env": "production",
          "iat": <unix timestamp>,
          "exp": <iat + 900> (15 min TTL),
          "jti": "<uuid>" }
      Signing key: RSA 2048 (stored in KMS; token service signs via KMS GenerateDataKeyPair)
      Public JWKS endpoint: https://token-authority.internal/.well-known/jwks.json

  JWT validation at the receiver:
    Option A: Inline (ECS sidecar/application code): fetch JWKS from token authority → verify sig + exp + iss
    Option B: VPC Lattice JWT authorizer (native, no sidecar):
      VPC Lattice listener config: JWT authorizer → token-authority JWKS endpoint
      Validates: signature, expiry, iss, service claim
      If invalid: 403 before request reaches service

  JWT rotation:
    Services fetch a new JWT every 10 minutes (TTL is 15 min; 5 min overlap for clock skew)
    If token service is unavailable: circuit breaker (fail open with cached JWT, alert Security)

[AUTHORIZATION — OPA Policy Engine]
  OPA deployed as a sidecar (or centralized cluster):
    Input to OPA per request:
      { "source": "payment-service",
        "destination": "ledger-service",
        "action": "POST /transactions",
        "jwt_claims": { "service": "payment-service", "env": "production" },
        "resource": "account:12345" }

    OPA policy (Rego):
      allow {
        input.jwt_claims.service == "payment-service"
        input.destination == "ledger-service"
        input.action == "POST /transactions"
        input.resource != null
      }

    Policy store: OPA bundles in S3 (security team owns the bucket; S3:GetObject policy)
    Policy deployment: CI/CD pipeline commits OPA bundle → S3 → OPA sidecars pull new bundle
    Audit: every OPA decision logged to CloudWatch Logs (decision + input + result)

[AUDIT LOGGING — VPC LATTICE ACCESS LOGS]
  VPC Lattice access logs (enabled per service network):
    Delivery: CloudWatch Logs + S3 (Kinesis Firehose for high volume)
    Log record fields:
      - sourceIP, destService, HTTP method, path, status code
      - clientCertificate.subject (from mTLS handshake)
      - jwtClaims.service (from JWT validator)
      - requestId (UUID, traceable across services)

  AWS X-Ray tracing:
    ECS services: X-Ray sidecar daemon (or ADOT collector)
    Trace propagation: X-Ray trace ID passed in HTTP header (X-Amzn-Trace-Id)
    Service map: visualize all inter-service calls, latency, error rates
    Security: X-Ray trace IDs do not contain sensitive data (just request IDs)

[MUTUAL TLS + JWT — COMBINED SECURITY MODEL]
  mTLS provides: transport-layer identity (the calling process's certificate)
  JWT provides:  application-layer identity (the calling service + its claims)
  Together:
    mTLS = "this TLS connection comes from payment-service (cert CN=payment-service)"
    JWT = "the application making this call is payment-service in production environment"
    If mTLS cert is stolen but JWT token is not: request rejected (JWT invalid)
    If JWT is stolen but mTLS cert is not: request rejected (client cert doesn't match)
    Attacker needs BOTH the cert private key AND a valid unexpired JWT to impersonate a service

Service Role in Architecture Why this over alternatives
VPC Lattice Service mesh: service discovery, routing, mTLS enforcement, auth policy No sidecar required (unlike App Mesh/Istio); AWS-managed; native IAM auth policy; scales without Envoy tuning

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Service identity chain:
  ECS Task Role (IAM) → Token Authority (STS:AssumeRole verification) → JWT (app-layer identity)

  Service-to-service IAM flow:
    1. order-service ECS task: IAM Task Role = arn:aws:iam::account:role/order-service-task-role
    2. order-service calls token-authority with STS-signed request (proves IAM identity)
    3. token-authority issues JWT with claims: { "service": "order-service" }
    4. order-service calls payment-service with:
       - TLS with client cert (CN=order-service)
       - Authorization: Bearer <JWT>
    5. payment-service VPC Lattice listener:
       a. Validates client cert (against internal CA trust store)
       b. Validates JWT (against JWKS endpoint)
       c. If both valid: forward to payment-service ECS tasks
    6. payment-service application code also passes request to OPA sidecar for ABAC check

  No service can call another without:
    - A valid TLS certificate issued by the internal CA (proves network identity)
    - A valid JWT issued by the token authority (proves application identity)
    - Explicit allow in VPC Lattice auth policy (proves authorization)
    - OPA decision = allow (proves business-rule authorization)

2. Encryption (At Rest, In Transit, In Use)

In Transit:
  All service-to-service: mTLS 1.3 (VPC Lattice enforces TLS 1.3 minimum)
  Client certs: RSA 2048 or ECDSA P-256 (short-lived, 24h TTL)
  JWT signing: RS256 (RSA 2048 private key in KMS; token authority signs; recipients verify with public key)

At Rest:
  Certificate private keys: stored in Secrets Manager (ECS) or ephemeral memory (Lambda)
    Secrets Manager: KMS-encrypted; ECS task fetches at startup via Secrets Manager API
  OPA policy bundles: S3 (SSE-S3); S3 bucket policy: only OPA sidecars can read
  Access logs: S3 SSE-KMS; object lock (audit logs immutable)
  X-Ray traces: encrypted at rest in X-Ray service (AWS-managed)

Key management:
  Token authority signing key: KMS RSA key pair (GenerateDataKeyPair); private key never leaves KMS
  Token validation: public key served at JWKS endpoint; no client ever gets the private key
  Certificate CA keys: ACM Private CA (hardware-protected; not extractable)

3. Network Isolation

Network architecture:
  VPC segmentation:
    Security group per service: inbound = VPC Lattice managed prefix list only
    No direct service-to-service security group rules (no "allow all 10.0.0.0/8")
    VPC Lattice prefix list: managed by AWS; updated automatically as Lattice scales

  No public IPs: ECS tasks in private subnets only; VPC endpoints for all AWS services

  VPC Lattice service network:
    Service network associated with all VPCs (via VPC Lattice VPC association)
    Cross-account: resource shares via AWS RAM (Resource Access Manager)
    Service directory: all 200 services registered; consumers discover via DNS (service.lattice.aws)

  DNS:
    VPC Lattice services: automatically assigned DNS name (service-id.{region}.on.aws)
    Internal alias: CNAME in Route 53 Private Hosted Zone: payment-service.internal → {lattice DNS}
    No direct IP communication; all calls via VPC Lattice DNS-resolved endpoints

4. Logging and Audit Trail

VPC Lattice access logs (per service):
  Log retention: CloudWatch Logs (14 days), S3 (7 years for compliance)
  Fields logged: source service (from mTLS cert subject), dest service, method, path,
                 status code, latency, JWT subject claim, X-Ray trace ID

OPA decision log:
  Every allow/deny decision logged: input bundle + policy version + decision + timestamp
  Stored in CloudWatch Logs → Security Lake (OCSF normalization)
  Alerts: OPA deny rate > 5% for a service pair → GuardDuty custom finding

CloudTrail:
  ACM PCA certificate issuance (every IssueCertificate API call logged)
  Secrets Manager GetSecretValue calls (per service per region)
  VPC Lattice config changes (service policy changes)
  KMS GenerateDataKeyPair (token authority signing key access)

5. Least Privilege and Blast Radius

Blast radius per service:
  Each service has:
    - Unique IAM Task Role (no shared roles between services)
    - Unique TLS client certificate (CN = service-name)
    - Unique JWT claims (service name in "service" claim)
    - Unique VPC Lattice auth policy (explicitly named allowed callers)

  Compromise of one service:
    - Attacker gets that service's TLS cert + JWT
    - Can only call services explicitly allowed in VPC Lattice auth policies
    - Cannot impersonate another service (different cert CN, different IAM role)
    - All calls appear in access logs with service identity

  Token scope limitation:
    JWT TTL: 15 minutes (not 24 hours)
    OPA policy: JWT service claim must match the expected caller for each route
    If JWT is stolen: valid for max 15 minutes, then expires

  Certificate revocation:
    ACM Private CA: OCSP responder + CRL
    VPC Lattice: OCSP check on every TLS handshake (or CRL cached for 1 hour)
    On service compromise: call ACM PCA RevokeCertificate immediately
    + rotate that service's JWT signing capability (rotate token authority cache)

6. Compliance Mapping

Scale and Resilience (Phase 4 — 10 min)

VPC Lattice throughput: VPC Lattice is AWS-managed and scales automatically — no capacity planning for the mesh layer. Each VPC Lattice service supports millions of requests per minute. No sidecar patching or Envoy configuration management.

Token authority HA:

  • Token authority service: ECS Fargate with 3+ tasks across 3 AZs
  • Application Load Balancer (internal) in front of token authority
  • Caching: services cache their JWT for 10 minutes (reducing load on token authority)
  • Token authority down: services use cached JWT (valid for remaining TTL); circuit breaker prevents cascading failures

ACM PCA availability:

  • ACM PCA SLA: 99.9% availability
  • Certificate caching: issued cert cached in Secrets Manager; ECS tasks use cached cert even if PCA is temporarily unavailable
  • Short-lived certs (24h): re-issuance failure window is 24h before cert expiry → alert at 6h before expiry

OPA performance:

  • OPA decision latency: < 1ms (in-process or sidecar on same host)
  • Policy bundle size: < 1 MB (incremental bundle updates via S3)
  • OPA bundle polling: every 60 seconds (balance between policy staleness and S3 API cost)

Trade-off Analysis — The Hard Choices

Decision 1: VPC Lattice vs. App Mesh (Envoy Sidecar) Service Mesh

Factor VPC Lattice App Mesh (Envoy sidecar)
Operational overhead Low — AWS managed; no Envoy tuning High — manage Envoy config, upgrades, resource allocation per service
Latency Very low (network layer, no sidecar) Adds ~1ms per hop (sidecar on same host)
Feature richness Native AWS (auth policies, JWT auth, access logs) Full Envoy features (circuit breaker, retries, Lua filters, ext_authz)
OPA integration VPC Lattice + OPA sidecar: feasible App Mesh: Envoy ext_authz filter → OPA (native integration)
Recommendation VPC Lattice for pure AWS ECS/Lambda workloads; App Mesh (or Istio) for multi-cloud or when full Envoy control is required

Decision 2: JWT TTL — 15 Minutes vs. 1 Hour

Factor 15-Minute JWT TTL 1-Hour JWT TTL
Attack window if stolen 15 minutes 1 hour
Token authority load Higher (refresh every 10 min) Lower (refresh every 50 min)
Clock skew sensitivity Need 5-min grace period Need 5-min grace period (proportionally less sensitive)
Recommendation 15 minutes for zero-trust model: limits exposure; token authority is internal (HA, low latency) — cost of more refreshes is minimal

Decision 3: OPA Centralized Cluster vs. Per-Service Sidecar

Factor OPA Centralized Cluster OPA Sidecar per Service
Availability risk Single point if cluster fails (mitigate with HA + local cache) Distributed; no central failure
Policy consistency Immediate consistent policy push Bundle polling delay (up to 60s)
Latency Network call to central OPA (~2ms) In-process (~0.1ms)
Recommendation OPA sidecar for production (low latency, no SPOF); centralized OPA for development/test (simpler operations)

Decision 4: Certificate TTL — 24 Hours vs. 90 Days

Factor 24-Hour TLS Client Cert 90-Day TLS Client Cert
Revocation urgency Low: cert naturally expires in 24h High: must actively revoke to stop attacker
Certificate issuance overhead Higher (automated issuance every 24h) Lower (quarterly rotation)
Blast radius if CA compromised Low: attacker can only use cert for 24h Higher: certs valid for up to 90 days post-CA compromise
Recommendation 24-hour certs for client certificates (service identity); 90-day for ALB server certs (managed by ACM auto-rotation)

Common Mistakes to Avoid

  1. Conflating mTLS and JWT as equivalent controls. mTLS is transport-layer identity (the network process/certificate). JWT is application-layer identity (the logical service + claims). Both are needed: mTLS alone doesn't tell you what business operation is requested; JWT alone doesn't prove the network caller matches the JWT claim (need mTLS to bind them). A complete zero-trust service mesh uses both.
  2. Not describing certificate lifecycle automation. Saying "use mTLS with ACM Private CA" without explaining how 200 services rotate their certificates daily demonstrates incomplete design. The automation story (ECS exec at startup → PCA API → cert in Secrets Manager → task rotation via Lambda) is what proves production-readiness.
  3. Using API Gateway for internal service-to-service calls. AWS API Gateway (v1 or v2) is designed for north-south traffic (external clients → API). Using API Gateway for east-west (service-to-service) traffic adds unnecessary hops, NAT gateway costs, and public-facing attack surface. VPC Lattice is the correct service for internal service mesh.
  4. Forgetting the authorization layer beyond mTLS. mTLS authentication does not mean authorization. A service having a valid certificate only proves its identity — it does not authorize it to call any endpoint with any parameters. VPC Lattice auth policy (service-level) + OPA (resource-level ABAC) are required for a complete zero-trust model.

Likely Follow-up Questions

Q1: How does VPC Lattice handle cross-account service-to-service calls?
VPC Lattice supports cross-account service sharing via AWS Resource Access Manager (RAM). The service owner creates a VPC Lattice service and shares the service ARN via RAM to target accounts or the entire AWS organization. Consumer accounts create a VPC Lattice service network association and a VPC association in their own VPC. DNS resolution is handled via Route 53 Resolver rules — the consumer VPC can resolve the shared service's DNS name and route calls through their local VPC Lattice attachment. Auth policy on the service specifies which IAM principals (cross-account roles) are allowed to invoke it. All mTLS and JWT requirements still apply — cross-account calls look the same as same-account calls from the mesh's perspective.

Q2: How would you handle certificate revocation for a compromised service at scale?
Immediate response to a compromised service: (1) Call ACM PCA RevokeCertificate API — this updates the Certificate Revocation List (CRL) and OCSP responder within minutes. (2) VPC Lattice checks OCSP on every TLS handshake — all new connections from the revoked cert will be rejected within the OCSP cache TTL (typically 1 hour for CRL, near-real-time for OCSP). (3) For the 24h TTL design: even if OCSP is slow, the compromised cert expires in less than 24 hours. (4) Additionally: update the VPC Lattice auth policy to remove the compromised service's IAM Task Role — this provides a second layer of revocation. (5) Rotate the IAM Task Role credentials (ECS service restart deploys new task with new cert). The combination of short cert TTL + OCSP + auth policy update provides layered revocation.

Q3: How do you bootstrap trust — how does a new service get its first certificate?
The bootstrapping problem in zero-trust: to get a certificate, the service must prove its identity, but the certificate IS the identity. Solution: ECS Task Role is the bootstrap identity (issued by AWS IAM, cryptographically tied to the ECS task by IMDS v2). At service startup: (1) ECS task uses its Task Role to call ACM PCA IssueCertificate with a CSR (Certificate Signing Request). The ACM PCA permission policy allows only a specific Lambda enrollment function to issue certs — not the services directly. (2) A separate enrollment Lambda function: validates the requesting IAM role matches the service's expected role (Terraform-managed mapping: role ARN → service name), then calls PCA IssueCertificate on behalf of the service. (3) Cert is stored in Secrets Manager (KMS encrypted) with the task's role allowed to read it. This ensures: only an ECS task with the correct IAM Task Role can bootstrap a certificate for that service name.

Q4: How do you prevent a service from calling APIs it shouldn't — not just which services but which endpoints?
VPC Lattice auth policies operate at the service level (can service A call service B?). For endpoint-level authorization (can service A call POST /transactions but not DELETE /accounts?), OPA handles this. The OPA policy evaluates: input.jwt_claims.service + input.destination + input.action (HTTP method + path). OPA is deployed as a sidecar in the receiving service (payment-service). On every incoming request, payment-service's application code calls the local OPA HTTP API: POST http://localhost:8181/v1/data/payments/authz with the request context. OPA returns allow/deny within 1ms. Denied requests return HTTP 403 immediately. This provides path-level, method-level, and even resource-level authorization without modifying business logic beyond the single OPA call.

Q5: How do you monitor for lateral movement attempts between services?
VPC Lattice access logs are the primary source. Each log record contains: source_service (from mTLS cert subject) + destination_service + action (HTTP method + path). Baseline: for each service pair, establish a normal calling pattern (which endpoints, what frequency, what times of day). Anomaly detection via CloudWatch Metric Filters + CloudWatch Anomaly Detection: alert when service-A → service-B call volume spikes > 3x baseline, or when service A calls a service it has never called before (new_service_pair metric). Security Lake + Athena: query for service pairs that have never communicated before (zero historical calls → sudden calls). GuardDuty Network Behavior analysis: unusual internal traffic patterns. OPA denial spikes: if service A is attempting calls it's not authorized for, OPA denials spike — this is logged and alerted.

Q6: How does this architecture handle gRPC services (not just REST)?
VPC Lattice supports HTTP/2 natively, which is the transport layer for gRPC. gRPC service-to-service calls: (1) TLS: gRPC uses HTTP/2 over TLS; VPC Lattice terminates mTLS (mutual TLS handshake) and forwards to the target group. Client certificate validation works the same as HTTPS. (2) JWT: gRPC metadata carries the Authorization header (Bearer <JWT>); VPC Lattice JWT authorizer works on gRPC metadata. (3) Routing: VPC Lattice supports gRPC method-based routing (route to target group based on gRPC service name + method). (4) OPA: gRPC authorization uses the gRPC interceptor pattern — a gRPC unary + streaming interceptor calls local OPA before forwarding to the handler. Input to OPA includes the gRPC method path (e.g., /payments.PaymentService/ProcessPayment) and JWT claims.

Q7: How do you handle secret injection into services securely — database passwords, API keys?
Service secrets (database passwords, third-party API keys) are stored in AWS Secrets Manager, not environment variables. On ECS Fargate: task definition references Secrets Manager ARN in the secrets configuration — ECS injects the secret value as an environment variable at task launch (the secret value is fetched by ECS infrastructure, not the application). For dynamic rotation: Secrets Manager Lambda rotation function rotates the secret, and ECS tasks pick up the new value on their next restart. For Lambda: Lambda uses secretsmanager:GetSecretValue API call at runtime (not environment variable) — this allows hot rotation without Lambda redeployment. IAM policy: ECS Task Role allows secretsmanager:GetSecretValue only for the specific ARNs for that service's secrets (least privilege). Encryption: all Secrets Manager values encrypted with a per-service KMS CMK.

Q8: How do you prevent an insider from creating a rogue service that impersonates a legitimate service?
Insider threat — impersonation prevention: (1) ACM PCA enrollment Lambda validates that the IAM role requesting a certificate for service name payment-service is exactly the Terraform-managed role payment-service-task-role. An insider cannot create a new IAM role with that name (SCP: deny iam:CreateRole where role name matches *-task-role pattern without explicit IaC approval). (2) VPC Lattice auth policies are Terraform-managed and PR-reviewed: to add a new allowed caller, a PR to the IaC repo is required (Security team approves). (3) Any new ECS task in production must be deployed via CI/CD pipeline (no manual ECS task registration in production — SCP: deny ecs:RegisterTaskDefinition from non-CI/CD principals). (4) All IssueCertificate calls to ACM PCA are logged in CloudTrail — alert on certificate issuance for unexpected service names.
