Q14 — Cloud Security Posture Management (CSPM) Platform

The Ask

"Design a Cloud Security Posture Management (CSPM) platform for a company running
500 AWS accounts organized in AWS Organizations. The platform must continuously detect
misconfigured resources across all accounts, enforce compliance against CIS AWS
Benchmarks and PCI-DSS, auto-remediate low-risk misconfigurations, require human
approval for high-risk remediations, provide executive dashboards, and generate
audit-ready compliance reports."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Is there a primary compliance framework that drives business requirements — PCI-DSS, SOC 2, HIPAA, NIST 800-53, or ISO 27001?
    Impact: PCI-DSS and HIPAA have legal penalties for non-compliance — these findings must generate P1 alerts and have SLA-bound remediation. SOC 2 and ISO 27001 are audit-based — compliance report generation and evidence collection timelines matter. Multiple frameworks: Security Hub supports multiple standards simultaneously (CIS, PCI-DSS, FSBP — AWS Foundational Security Best Practices). Custom frameworks: AWS Audit Manager with custom control frameworks.

  • Q: Are there existing security tools (Prisma Cloud, Wiz, Dome9) that the team wants to integrate or replace?
    Impact: Native AWS tooling (Security Hub + Config + Audit Manager): lower cost, deep AWS integration, no third-party agent. Third-party CSPM (Wiz, Prisma): richer UI, multi-cloud, but adds cost. Integration: Security Hub accepts third-party findings via ASFF (Amazon Security Finding Format), so Wiz + Security Hub together is a common pattern. This design focuses on native AWS CSPM.

  • Q: What is the remediation approval process — who approves, and what systems (Jira, ServiceNow, Slack) must integrate?
    Impact: Remediation via ITSM (ServiceNow): Step Functions workflow creates change ticket, waits for approval callback. Slack-based approval: EventBridge → Lambda → Slack API → approval button → Lambda continues. Auto-remediation without approval: only for deterministic, low-risk fixes (e.g., enable S3 block public access, tag missing resources). Human approval: required for network changes (security group rules), IAM changes (detach policies), data storage changes.

  • Q: Should non-compliant accounts be preventively blocked from deploying resources, or only detected and alerted?
    Impact: Preventive: SCPs (Service Control Policies) enforced by AWS Organizations. Example: deny creating S3 buckets without Block Public Access. Preventive + detective = defense in depth. Detective only (Config rules): alert after resource is created. Guardrails Tower: AWS Control Tower's built-in preventive and detective guardrails. If they already have Control Tower, extend it; if not, implement SCPs directly.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. 500 accounts in AWS Organizations (OU structure: Production, Non-Production, Sandbox).
  2. Security Hub as the CSPM aggregation layer (org-level delegated admin in Security account).
  3. AWS Config as the compliance data plane (rules + conformance packs in all 500 accounts).
  4. Lambda-based auto-remediation for low-risk findings.
  5. Step Functions for human-approval workflows.
  6. QuickSight for executive dashboards; Audit Manager for compliance reports.

Core Architecture — Data Flow Narrative

[ACCOUNT STRUCTURE]
  Management Account:
    AWS Organizations root
    SCPs attached to OUs (preventive controls)
    Control Tower (optional, for guardrails automation)

  Security Account (dedicated):
    Security Hub delegated admin (aggregates all 500 accounts)
    GuardDuty delegated admin
    AWS Config aggregator (all accounts + all regions)
    AWS Audit Manager (compliance assessment)
    CSPM orchestration (Step Functions, EventBridge, Lambda)

  Member Accounts (498 accounts):
    Security Hub member (auto-enabled by Security Hub org settings)
    GuardDuty member
    AWS Config recorder (sends findings to aggregator in Security account)
    Config remediation SSM automations

  Log Archive Account:
    CloudTrail org-level trail → S3 log archive bucket (Object Lock WORM)
    Config configuration snapshots
    Security Hub findings export (S3)

[PREVENTIVE CONTROLS — AWS Organizations SCPs]
  OUs and SCP assignments:
    Production OU:
      - Deny s3:PutBucketAcl where Acl != private
      - Deny s3:DeleteBucketPolicy
      - Deny ec2:ModifyInstanceAttribute (disable IMDSv1 — require IMDSv2)
      - Deny cloudtrail:StopLogging
      - Deny config:StopConfigurationRecorder
      - Deny iam:CreateUser (require federated access)

    Non-Production OU: Relaxed SCPs (no deny iam:CreateUser; allow sandbox experimentation)
    Sandbox OU: Minimal SCPs (cost controls only: deny ec2:RunInstances if instance type not in allowlist)

  AWS Control Tower Guardrails:
    Mandatory: Disallow Public Access to S3 (SCP-based)
    Strongly Recommended: Disallow unrestricted root user access
    Elective: Detect changes to IAM roles and policies (Config rule)

[DETECTIVE CONTROLS — AWS Config Conformance Packs]
  Deployed to all 500 accounts (via CloudFormation StackSets or Control Tower customizations):

  Conformance Pack 1: CIS AWS Benchmarks v1.4
    cis-aws-benchmark-level-1: (50 rules — most critical)
      - iam-root-access-key-check
      - mfa-enabled-for-iam-console-access
      - access-keys-rotated (90 days)
      - cloudtrail-enabled
      - vpc-flow-logs-enabled
      - s3-bucket-public-read-prohibited
      - ec2-imdsv2-check (IMDS version requirement)
      - rds-instance-public-access-check

    cis-aws-benchmark-level-2: additional 30 rules
      - cloudwatch-alarm-action-check (alarm for root login, IAM changes, etc.)
      - vpc-default-security-group-closed
      - iam-no-inline-policy-check
      - iam-policy-no-statements-with-admin-access

  Conformance Pack 2: PCI-DSS v3.2.1
    pci-dss-operational-best-practices:
      - Req 1: alb-waf-enabled, ec2-security-group-attached-to-eni
      - Req 2: ec2-no-amazon-key-pair (require EC2 Instance Connect or SSM)
      - Req 3: dynamodb-table-encryption-enabled, rds-storage-encrypted, s3-bucket-server-side-encryption-enabled
      - Req 6: lambda-function-public-access-prohibited
      - Req 7: iam-policy-no-statements-with-admin-access
      - Req 8: iam-root-access-key-check, access-keys-rotated
      - Req 10: cloudtrail-s3-dataevents-enabled

[FINDING AGGREGATION — Security Hub]
  Security Hub Organization configuration (Security account as delegated admin):
    Auto-enable new accounts: yes
    Standards enabled: CIS AWS Benchmarks v1.4, PCI-DSS v3.2.1, FSBP
    Cross-region aggregation: us-east-1 aggregation region (all regions' findings aggregated here)

  Finding severity thresholds (ASFF format):
    CRITICAL (score 90-100): Exposed access key, public S3 bucket with sensitive data, root account compromise
    HIGH (score 70-89): No MFA on root, public RDS instance, unrestricted SG (0.0.0.0/0:22)
    MEDIUM (score 40-69): Access key > 90 days, disabled CloudTrail, no VPC flow logs
    LOW (score 1-39): Informational compliance deviations

  Security Hub Insights (custom views):
    - "PCI-DSS Critical findings > 30 days unresolved" (SLA breach indicator)
    - "Accounts with zero active security findings" (potential blind spot)
    - "Top 10 accounts by critical finding count" (prioritization)
    - "S3 public access findings across all accounts" (quick risk overview)

[AUTO-REMEDIATION — LOW RISK]
  Config rule → EventBridge rule → Lambda remediation function

  Auto-remediated (no approval):
    Config rule: s3-bucket-public-read-prohibited → violation detected
    EventBridge rule: matches config:ComplianceChangeNotification with ruleId = s3-bucket-public-read-prohibited and newEvaluationResult = NON_COMPLIANT
    Lambda: s3.put_bucket_public_access_block(Bucket=bucket_name, PublicAccessBlockConfiguration={all:True})

    Other auto-remediated rules:
      - ec2-imdsv2-check: modify_instance_metadata_options(HttpTokens=required)
      - rds-instance-deletion-protection-enabled: modify_db_instance(DeletionProtection=True)
      - s3-bucket-logging-enabled: put_bucket_logging(BucketLoggingStatus={enabled})
      - iam-user-unused-credentials-check (90 days): disable_access_key()

    Auto-remediation log:
      Lambda publishes to Security Hub: custom finding "REMEDIATION_APPLIED" with ResourceId + action
      All auto-remediation actions logged to CloudTrail
      Security Hub custom insight: "Auto-remediated findings in last 24h" (verify automation is working)

[HUMAN APPROVAL WORKFLOW — HIGH RISK]
  High-risk finding → EventBridge → Step Functions workflow:

  Trigger: Security Hub finding, Severity = CRITICAL or HIGH, WorkflowState = NEW
    AND Category IN [Software and Configuration Checks/Networking, IAM, Key Management]

  Step Functions state machine:
    State 1: CreateRemediation — Lambda analyzes finding, generates remediation plan
      Input: finding details (resource ARN, account ID, config rule, expected state)
      Output: human-readable remediation plan (JSON with proposed AWS API calls)

    State 2: NotifyApprover — Lambda creates ServiceNow change ticket + sends Slack notification
      ServiceNow: Change Request with remediation plan attached
      Slack: #security-remediation channel: "CRITICAL finding: public RDS in account X. Approve? [Yes] [No]"

    State 3: WaitForApproval — Step Functions callback (.waitForTaskToken)
      Timeout: 48 hours (SLA for critical findings)
      On approval: proceed to State 4
      On denial: close finding as "Suppressed with justification"; log to Audit Manager
      On timeout: escalate to CISO via PagerDuty; re-notify

    State 4: ExecuteRemediation — Lambda executes the pre-generated remediation plan
      Resource: security group — revoke ingress rule for 0.0.0.0/0:22
      Resource: IAM policy — detach overly permissive policy, attach scoped-down replacement

    State 5: Verify — Config re-evaluates the rule (Config:StartConfigRulesEvaluation)
      If still non-compliant: alert team; open Jira for manual investigation
      If compliant: update Security Hub finding WorkflowState = RESOLVED

[EXECUTIVE DASHBOARD — QuickSight]
  Data pipeline: Security Hub findings → S3 (export via EventBridge) → Glue ETL → QuickSight SPICE

  Dashboard views:
    1. Overall compliance posture: % compliant by framework (CIS, PCI-DSS) per OU
    2. Trend chart: critical finding count over last 90 days (is the security posture improving?)
    3. SLA breach indicator: findings open > SLA threshold (Critical: 24h, High: 7d, Medium: 30d)
    4. Top 10 violated Config rules across all accounts
    5. Remediation automation effectiveness: auto-remediated % vs manual %
    6. Account heat map: compliance score per account (green/yellow/red)

  Refresh: SPICE dataset refreshed every 4 hours (near-real-time for operational review)
  Access: IAM Identity Center SSO → QuickSight (Reader for executives, Author for security team)

[COMPLIANCE REPORTS — AWS AUDIT MANAGER]
  Assessment: one per compliance framework
    - CIS AWS Benchmarks v1.4 Assessment
    - PCI-DSS 3.2.1 Assessment
    Custom Assessment: SOX ITGC (custom control framework)

  Evidence collection (automated):
    Config rule evaluation results → Audit Manager evidence (triggered by Config compliance changes)
    CloudTrail events → Audit Manager evidence (IAM changes, S3 policy changes, etc.)
    Security Hub findings → Audit Manager evidence (via EventBridge + Lambda custom evidence upload)

  Report generation:
    Pre-audit: generate evidence report (PDF) per assessment
    Report includes: control ID, evidence sources, compliance status, exceptions with justifications

  Delegation: Audit Manager delegated admin in Security account

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Security Hub delegated admin:
  Designated from Management account: aws securityhub enable-organization-admin-account
  Security account: delegated admin for Security Hub, GuardDuty, Config, Audit Manager
  Security account IAM roles:
    - SecurityHub-ReadOnly: QuickSight data source, exec dashboard access
    - SecurityHub-Remediator: Lambda execution role for auto-remediation (assume role in member accounts)
    - SecurityHub-Admin: modify standards, create insights (security team only)

Cross-account remediation IAM:
  Remediation Lambda role (in Security account) assumes cross-account roles in member accounts
  Member account: arn:aws:iam::member:role/CSPMRemediationRole
    Trust policy: allows Security account Lambda role to assume
    Permission policy: only the specific remediation actions needed
    Example: s3:PutBucketPublicAccessBlock, ec2:ModifyInstanceMetadataOptions
    Deny: s3:DeleteBucket, ec2:TerminateInstances (prevent accidental destruction)

2. Encryption (At Rest, In Transit, In Use)

Findings data at rest:
  Security Hub: findings stored in AWS-managed storage (opaque to users; IAM-controlled access)
  Findings export to S3: SSE-KMS with cspm-findings-key (CMK)
  Audit Manager evidence: S3 SSE-KMS
  QuickSight SPICE: encrypted at rest (AWS-managed)

CloudTrail logs: S3 Object Lock WORM + SSE-KMS (cloudtrail-key CMK)
Config snapshots: S3 SSE-KMS (config-key CMK)

All API calls to CSPM services (Security Hub API, Config API) are TLS-encrypted in transit
No sensitive data stored in Lambda environment variables (role-based access to Secrets Manager)

3. Network Isolation

Security account network:
  Lambda functions for remediation: VPC + private subnets
  VPC endpoints for: Security Hub API, Config API, Secrets Manager, STS (no internet egress)

Remediation Lambda network:
  Cross-account AWS API calls go via VPC endpoints (not internet)
  STS AssumeRole call: VPC endpoint for STS
  S3 findings export: VPC gateway endpoint
  No inbound rules in Security account (no listener services)

4. Logging and Audit Trail

All CSPM actions logged:
  Security Hub finding state changes → EventBridge → CloudTrail
  Config rule evaluations → Config delivery channel → S3 + SNS
  Lambda remediation actions → CloudTrail (specific API calls) + Lambda execution log (CloudWatch)
  Step Functions execution history → CloudWatch Logs (approval workflow audit trail)
  Audit Manager evidence → S3 (tamper-proof; Object Lock)

CloudTrail Insights:
  Enable Insights on org-level trail: detect unusual API call volumes
  Alert: unusual spike in Config:PutEvaluations (may indicate rogue Config recorder)
  Alert: unusual Security Hub:BatchImportFindings (external tool injecting fake findings)

5. Least Privilege and Blast Radius

Remediation Lambda least privilege:
  Per-rule Lambda function with minimal IAM permissions
  Example: Lambda for s3-public-access only has s3:PutBucketPublicAccessBlock permission
  NOT: a single Lambda with full S3 admin (blast radius if compromised)

  IAM condition: Lambda role has condition: aws:ResourceTag/CSPMManaged = true
    Only modifies resources tagged by CSPM (not untagged/manually-created resources)

SCP guardrails on Security account:
  Deny: iam:CreateRole in Security account except from Management account (prevent privilege escalation)
  Deny: cloudtrail:StopLogging, config:StopConfigurationRecorder
  Deny: securityhub:DisableSecurityHub (no one can disable CSPM)

6. Compliance Mapping

Framework AWS Controls
CIS AWS Benchmarks v1.4 Security Hub CIS standard; Config CIS conformance pack
PCI-DSS 3.2.1 Security Hub PCI standard; Audit Manager PCI assessment
SOC 2 CC6/CC7 FSBP standard; custom Config rules; Audit Manager custom framework
NIST 800-53 Security Hub NIST 800-53 standard (available)

Scale and Resilience (Phase 4 — 10 min)

Config rule evaluation at 500 accounts:

  • Config rules deployed via CloudFormation StackSets (org-level, target all accounts/regions).
  • Rule evaluation triggers: resource creation/modification (change-triggered) + periodic (every 24 hours for time-sensitive rules like access-keys-rotated).
  • Config Aggregator: aggregates compliance data from all 500 accounts. Pagination-aware queries (Config API supports up to 100 accounts per request; Lambda pagination handles all 500).

Security Hub finding volume:

  • 500 accounts × 100 Config rules × 5 avg resources per rule = ~250,000 evaluations per day.
  • Security Hub: handles millions of findings. Performance: Security Hub aggregate API queries are paginated and rate-limited (100 findings per page; use filter on WorkflowState=NEW to reduce volume).
  • EventBridge rule: filter on Severity >= HIGH to reduce Lambda invocations (not every LOW finding triggers auto-remediation).

Auto-remediation throughput:

  • Lambda concurrency: set reserved concurrency = 50 for remediation Lambdas (prevent overloading member account API limits).
  • SQS dead-letter queue: failed remediation Lambda invocations → DLQ → CloudWatch alarm → Security Hub custom finding.

Trade-off Analysis — The Hard Choices

Decision 1: Native AWS CSPM vs. Third-Party (Wiz, Prisma Cloud)

Factor Native AWS (Security Hub + Config) Third-Party CSPM (Wiz, Prisma)
Cost Lower (Config + Security Hub fees) Higher (per-account/per-resource licensing)
AWS integration depth Very deep (real-time Config events) Deep but via API polling (latency)
Multi-cloud support AWS only AWS + Azure + GCP
UI/Dashboard quality Moderate (QuickSight custom dashboards) Excellent (purpose-built dashboards)
Recommendation Native AWS for AWS-only orgs; third-party CSPM for multi-cloud or when budget exists

Decision 2: Auto-Remediation — Config Remediation Actions vs. Lambda

Factor Config Remediation Actions (SSM Automation) Custom Lambda Remediation
Complexity Low — no code required; use pre-built SSM documents Higher — write and maintain Lambda code
Flexibility Limited to SSM automation documents Unlimited
Approval workflow Basic (no Step Functions integration) Full Step Functions integration
Recommendation SSM Automation for simple remediations (single API call); Lambda + Step Functions for complex workflows (approval, verification, rollback)

Decision 3: Security Hub Finding Aggregation — All Findings vs. Critical Only

Factor Aggregate All Findings Aggregate High + Critical Only
Visibility Complete compliance picture Focused on risk-relevant items
Finding volume Very high (millions/day at 500 accounts) Manageable (tens of thousands/day)
Storage/processing cost Higher Lower
Recommendation Aggregate all findings in Security Hub (Security Hub handles volume); filter in QuickSight (High+ for exec dashboard; all findings for security analyst view)

Decision 4: Conformance Pack Deployment — StackSets vs. Control Tower CfCT

Factor CloudFormation StackSets Control Tower CfCT (Customizations for Control Tower)
Auto-enroll new accounts Requires SCP + StackSets org deployment + EventBridge Lambda CfCT auto-deploys to new accounts automatically
Drift detection Manual (no built-in drift detection for StackSets) CfCT auto-remediates StackSet drift
Requires Control Tower No Yes
Recommendation Control Tower CfCT if already using Control Tower; StackSets with EventBridge auto-enrollment Lambda if not

Common Mistakes to Avoid

  1. Describing Config rules without conformance packs. Managing 100+ Config rules individually across 500 accounts is unmanageable. Conformance packs group rules into a deployable artifact. Via CloudFormation StackSets (or Control Tower CfCT), the entire CIS benchmark (50+ rules) can be deployed to all accounts with a single operation. Not knowing conformance packs signals limited real-world experience.
  2. Not describing the Security Hub delegated admin model. Many candidates say "enable Security Hub in each account." For 500 accounts, you need org-level Security Hub: Management account designates a delegated admin account (Security account), which auto-enables Security Hub in all member accounts and aggregates findings centrally. Without this, you'd need to log into 500 accounts individually.
  3. Conflating detection with prevention. Config rules DETECT misconfigurations after resources are created. SCPs PREVENT misconfigurations before they happen. A complete CSPM platform uses both layers. Candidates who only describe Config rules (detection) without SCPs (prevention) miss half of the architecture.
  4. Forgetting that CSPM must be tamper-proof. The CSPM platform itself is a target. An attacker who can disable GuardDuty, stop Config recorders, or mute Security Hub findings has effectively gone blind. The architecture must prevent this: SCPs that deny stopping Config, GuardDuty, or Security Hub; CloudWatch alarms on recorder status; and Security Hub finding: no-findings-for-24h is itself an alert.

Likely Follow-up Questions

Q1: How does Security Hub handle findings from non-AWS tools like third-party vulnerability scanners?
Security Hub accepts findings from any source via the BatchImportFindings API, as long as they conform to the Amazon Security Finding Format (ASFF). Third-party tools (Tenable, Qualys, Snyk) have native Security Hub integrations that format their findings as ASFF and push them to Security Hub. Custom tools: implement a Lambda that transforms findings from the tool's format into ASFF and calls BatchImportFindings. Once in Security Hub, findings from all sources are normalized — same severity levels, same workflow states, same filtering and dashboard views. This is why Security Hub is the correct aggregation layer for a CSPM platform: it's a finding bus, not just an AWS-native scanner.

Q2: How do you handle the SLA for critical findings — how do you ensure they're remediated in time?
SLA enforcement: (1) Security Hub finding state changes trigger EventBridge → Lambda → CloudWatch metric increment (FindingAge_CRITICAL). (2) CloudWatch Alarm: if FindingAge_CRITICAL > 24 hours without WorkflowState = RESOLVED → SNS → PagerDuty P1. (3) Security Hub Custom Insight: "Critical findings open > 24 hours" — a non-zero count here means SLA breach. (4) Audit Manager collects evidence of SLA compliance: how long findings were open before resolution. (5) Executive QuickSight dashboard: "SLA breach rate" widget shows % of critical findings resolved within SLA period, trending over time. For PCI-DSS: critical findings (Severity=CRITICAL for PCI-related rules) must be resolved within 1 day per PCI-DSS requirements. Step Functions workflow enforces the SLA with automatic escalation on timeout.

Q3: How do you prevent Config rule false positives from triggering unnecessary auto-remediations?
False positive prevention: (1) Suppression rules: Security Hub supports suppression rules — create a suppression rule for ConfigRuleName = rds-instance-public-access-check AND ResourceId = analytics-rds-instance (legitimate public RDS for reporting). This prevents the finding from reaching the remediation workflow. (2) Tagging-based bypass: remediation Lambda checks for tag CSPMExemption = approved:2024-12:justification. If tag exists, Lambda skips remediation and logs the exception to Audit Manager. (3) Dry-run mode: remediation Lambda first runs in "dry-run" mode (logs what it would do without doing it); only proceeds to live action after internal validation. (4) Config rule parameter tuning: Config rules have parameters (e.g., s3-bucket-logging-enabled has a targetBucket parameter to specify an acceptable logging destination). Tune parameters to match the organization's actual configuration before enabling auto-remediation.

Q4: How do you manage 500 accounts' Config conformance packs cost-effectively?
Config pricing: Config rules are charged per rule evaluation. For 500 accounts with 100 rules × 1000 resources = 50 million evaluations/day → significant cost. Cost optimization: (1) Use change-triggered rules where possible (only evaluate when resource changes, not periodically). (2) Consolidate rules into conformance packs (same unit cost as individual rules, but easier management). (3) Exclude non-production/sandbox accounts from expensive rules (use StackSet target OUs — only apply PCI-DSS conformance pack to Production OU). (4) Use AWS Config Advanced Query (SQL queries over Config data) instead of periodic Config rules for reporting-only checks. (5) Config aggregator is free — cost is only per-rule evaluation in member accounts.

Q5: How do you handle newly created accounts joining the organization?
New account auto-enrollment: (1) EventBridge rule in Management account: event type organizations:CreateAccountResult (status=SUCCEEDED) triggers a Lambda. (2) Lambda performs: (a) Security Hub: auto-enrollment is already configured (organization settings auto-enable). (b) GuardDuty: auto-enrollment configured. (c) Config: CloudFormation StackSets with ORGANIZATIONS deployment type automatically deploy to new accounts via auto-deployment setting. (d) SCPs: already applied at OU level — new account inherits OI SCPs on move to target OU. (3) Notification: Lambda sends Slack message "New account 123456789 enrolled in CSPM". (4) Audit trail: CloudTrail records the new account creation + all auto-enrollment API calls.

Q6: How do you generate PCI-DSS compliance evidence for auditors using Audit Manager?
Audit Manager PCI-DSS assessment: (1) Create an assessment in the Security account using the PCI DSS v3.2.1 framework. (2) AWS scope: all accounts in the PCI-DSS cardholder data environment (CDE) accounts — not all 500 accounts, only the ones that handle card data. (3) Evidence collection: Audit Manager automatically pulls Config rule evaluations (pass/fail for each PCI requirement mapped to a Config rule), CloudTrail events (IAM changes, S3 policy changes — evidence for PCI Req 7, 8, 10), and Security Hub findings (PCI standard findings). (4) Manual evidence: some PCI requirements (penetration testing Req 11.3, security awareness training Req 12.6) require manual evidence upload. Audit Manager provides a UI for uploading PDF evidence. (5) Report generation: at audit time, generate a PDF assessment report per section — each control shows: evidence sources, compliance status, exception justifications. The PDF is exportable to the auditor.

Q7: How do you handle multi-region compliance — are Config rules running in all regions?
Multi-region Config: Config recorder must be enabled in EVERY region where resources are deployed (and for some AWS services, even in unused regions if the service is global-region-aware). Config aggregator in the Security account aggregates from all accounts AND all regions. CloudFormation StackSets target region: specify all 8-10 AWS regions used by the organization. Config multi-region aggregation: the aggregator automatically pulls from all configured regions. Security Hub cross-region aggregation: designate a single aggregation region (us-east-1); Security Hub replicates findings from all other regions to the aggregation region. Compliance dashboards in QuickSight show a region filter — executives can see "which region has the most misconfigurations?"
