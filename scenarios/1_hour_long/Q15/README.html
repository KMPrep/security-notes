Q15 — Privileged Access Management (PAM) for AWS Infrastructure

The Ask

"Design a Privileged Access Management (PAM) solution for a financial services company
with 300 engineers who need temporary elevated access to AWS infrastructure. Requirements:
no standing privileged access, all sessions recorded, JIT (Just-In-Time) access with
automatic expiry, no SSH keys, break-glass procedures for emergencies, and full audit
trail for SOX compliance. The company runs 50 AWS accounts."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What types of privileged operations do engineers need — EC2 shell access, database access, Kubernetes admin, or AWS console/API access to modify infrastructure?
    Impact: EC2 shell access: SSM Session Manager (no SSH keys, all sessions logged to CloudWatch + S3). Database access: RDS IAM auth + Secrets Manager rotation + SSM port forwarding (no public DB endpoints). Kubernetes admin: EKS access via IAM + IRSA; kubectl sessions through AWS CloudShell or a bastion host with session recording. AWS console/API access: IAM Identity Center JIT permission sets.

  • Q: Is there an existing ITSM system (ServiceNow, Jira) for access request approval workflows?
    Impact: ServiceNow integration: Step Functions calls ServiceNow API to create a Change Request; ServiceNow approval triggers a callback webhook → Step Functions resumes. Slack-based approval: simpler but less audit-friendly for SOX. If no ITSM: use AWS native (Step Functions + SES email approval with JWT-signed approval links). SOX requires evidence of approval — ITSM creates better audit artifacts than Slack.

  • Q: What is the maximum duration for JIT sessions — hours, days?
    Impact: JIT session TTL drives the STS session duration. STS AssumeRole maximum: 12 hours (default), up to 36 hours (set MaxSessionDuration on the role). For SOX compliance: shorter TTL = less exposure (4-hour sessions are common). Emergency break-glass: 4-hour maximum with mandatory CISO notification.

  • Q: Are engineers currently using SSH keys for EC2 access?
    Impact: SSH key replacement is a project in itself. Remediation path: SCP (deny ec2:ImportKeyPair + deny ec2:CreateKeyPair for new instances); existing key pairs: rotate out (convert instances to SSM-managed only via SSM agent update + new launch templates). Some legacy AMIs may not have the SSM agent — need AMI baking plan.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. No SSH key-based access anywhere (enforced by SCP).
  2. All 50 accounts managed via IAM Identity Center (org-level, from Management account).
  3. SSM Session Manager for all EC2/ECS shell access.
  4. JIT access via Service Catalog + Step Functions (approval workflow).
  5. Break-glass: emergency roles in each account (separate from normal JIT, trigger PagerDuty P1).

Core Architecture — Data Flow Narrative

[PRINCIPLE: ZERO STANDING PRIVILEGE]
  All engineers start with minimal IAM Identity Center permission set:
    PermissionSet: ReadOnly
    Access: List/Describe APIs only across all 50 accounts
    No: write APIs, no: EC2 shell access, no: database access

  To elevate: engineer requests JIT access → approval → time-limited elevated role → auto-revoke

[JIT ACCESS WORKFLOW — Normal Operations]
  Step 1: Engineer submits access request
    Portal: Service Catalog self-service portal (or internal web app)
    Request form:
      - Target account (dropdown from all 50)
      - Access level: [Operator | SRE | DBA | SecurityAudit | Admin]
      - Target resource(s): EC2 instance IDs, RDS cluster ARN, S3 bucket ARN
      - Business justification (free text, required)
      - Duration: [1h | 2h | 4h | 8h] (max 8h, default 4h)
      - ServiceNow change ticket # (required for prod accounts)

    Lambda: validates request (account exists, access level in allowlist, duration within policy)
    Lambda: creates Step Functions workflow execution

  Step 2: Approval workflow (Step Functions)
    State 1: NotifyApprovers
      - Get approver list: IAM Identity Center: team lead + security reviewer
        (Okta group → IAM Identity Center → approver list Lambda)
      - Send ServiceNow change request (Category: JIT Access, Priority: Normal)
      - Send Slack notification to #access-requests: "Engineer A requests Operator access to Account B for 4h"
      - Include [Approve] and [Deny] buttons (signed JWT in URL, 15-min link expiry)

    State 2: WaitForApproval (waitForTaskToken, timeout: 30 minutes)
      - Approver clicks [Approve] → API Gateway → Lambda → send task success to Step Functions
      - Approver clicks [Deny] → API Gateway → Lambda → send task failure to Step Functions
      - Timeout (30 min): auto-deny; notify engineer

    State 3 (on approval): GrantAccess
      IAM Identity Center: CreateAccountAssignment
        Principal: engineer's IAM Identity Center user
        PermissionSet: Operator-JIT-PermissionSet
        Account: target account

      DynamoDB: write access record
        { userId, account, permissionSet, grantedAt, expiresAt, approvedBy, justification, changeTicket }

      Slack notification: "Access granted. Expires at 14:30 UTC. Session will be terminated automatically."

    State 4: WaitForExpiry (Step Functions Timer — duration from request)
      Wait: e.g., 4 hours (Step Functions timestamp wait)

    State 5: RevokeAccess
      IAM Identity Center: DeleteAccountAssignment
        (removes the engineer's JIT permission set assignment)
      DynamoDB: update record — revokedAt timestamp
      Slack notification: "JIT access for engineer A in account B has been revoked"

      Note: Existing active sessions in the account remain valid until the STS session token
      expires (max 8h). However, the engineer cannot start NEW sessions after revocation.
      For immediate session termination: SSM Session Manager can terminate active sessions
      via ssm:TerminateSession API.

[SSM SESSION MANAGER — EC2 SHELL ACCESS]
  No SSH keys (enforced by SCP):
    SCP policy:
      Deny: ec2:ImportKeyPair for all accounts
      Deny: ec2:CreateKeyPair for all accounts
      Deny: ec2:AuthorizeSecurityGroupIngress where port = 22 (no SSH allowed)

  SSM agent: required on all EC2 instances
    AMI baking: EC2 Image Builder with SSM agent installed + latest patch
    Existing instances: SSM State Manager association: AWS-UpdateSSMAgent + configure SSM

  Session Manager configuration (SSM Document: AWS-StartSSMSession):
    Session logging (mandatory):
      CloudWatch Logs: /aws/ssm/sessions/{account-id}/{instance-id} (14 days)
      S3: s3://sessions-archive/{account-id}/{date}/{session-id}.log (7 years, Object Lock WORM)

    Session timeout: 30 minutes of inactivity → auto-terminate

    Run-as: sessions run as ssm-user (not root)
      ec2-instance-connect: optional for specific break-glass scenarios

    IAM policy for engineer (in JIT Operator PermissionSet):
      Allow: ssm:StartSession on instances tagged JITAccessGroup = Operator
      Allow: ssm:TerminateSession (own sessions only)
      Deny: ssm:StartSession on instances tagged ProductionDB = true (DBAs only)

    Monitoring:
      CloudWatch Metric Filter: SSM session started → SNS notification to #audit-channel
      GuardDuty: unusual SSM session from unexpected IP or at unusual hours

[DATABASE ACCESS — NO DIRECT CONNECTION]
  Pattern: SSM Port Forwarding (no public DB endpoint)
    Engineer (local machine) → SSM Port Forward → private RDS endpoint
    Command: aws ssm start-session --target i-bastion --document-name StartPortForwardingSession
              --parameters portNumber=5432,localPortNumber=15432

    Alternative (preferred): RDS IAM Database Authentication
      - RDS: IAM authentication enabled
      - Engineer requests DBA permission set → gets sts:AssumeRole → generates
        rds:generateAuthToken → DB password is a short-lived IAM auth token (15 min)
      - No static database passwords; no Secrets Manager needed for user passwords
      - DB user mapped to IAM role via GRANT rds_iam TO db_user_role

    Database session recording:
      Enable PostgreSQL/MySQL query logs → CloudWatch Logs
      RDS Enhanced Monitoring + Performance Insights (query-level auditing)
      Macie: scan RDS snapshot exports for PII (optional)

[BREAK-GLASS ACCESS — EMERGENCY]
  Design principles:
    - Break-glass is for genuine emergencies ONLY (production outage, security incident)
    - No approval required (speed is critical)
    - But: mandatory notification to CISO + Security team (not just logging)
    - Short TTL: 4 hours maximum
    - Post-incident review required within 24 hours

  Break-glass IAM role (per account):
    Role name: BreakGlass-Admin-{account-id}
    Trust policy: allows only from Management account (break-glass execution Lambda)
    Permission: Administrator Access (necessary for break-glass scenarios)
    Condition: aws:PrincipalArg must match break-glass user identity

  Break-glass trigger:
    Engineer calls: aws sts assume-role --role-arn BreakGlass-Admin-{account}
                    --role-session-name "incident-{incident-id}"
    Lambda (EventBridge on AssumeRole with session-name prefix "incident-"):
      Immediate actions:
        1. Send PagerDuty P1 to CISO + Security team
        2. Send Slack message to #security-incidents + #executive-notifications
        3. Enable CloudTrail Insight (for unusual API call detection)
        4. Start GuardDuty Findings export in real-time
        5. DynamoDB: create break-glass event record (who, when, account, incident ID)

  Break-glass monitoring:
    CloudTrail: every API call during break-glass session tagged with session name
    GuardDuty: alert on break-glass role usage (custom EventBridge rule on AssumeRole for BreakGlass role)
    Post-incident: CloudTrail Athena query for all API calls in that session for SOX review

[AUDIT TRAIL — SOX COMPLIANCE]
  Every privileged action is captured:
    JIT request/approval: DynamoDB access records + Step Functions execution history
    SSM sessions: CloudWatch Logs + S3 WORM (session content)
    AWS API calls: CloudTrail (org-level, all regions, all accounts)
    Break-glass events: DynamoDB break-glass registry + real-time notifications

  SOX evidence package (automated, quarterly):
    Lambda queries DynamoDB: all privileged access in the period
    Lambda queries CloudTrail: all admin API calls by privileged roles
    Lambda generates report: Excel/CSV → S3 → notify Internal Audit team
    Audit Manager: custom ITGC framework collects JIT access evidence automatically

  Attestation (quarterly):
    Manager review: list of all JIT access by team member in last 90 days
    For each entry: manager confirms "access was appropriate" (Audit Manager workflow)
    Exceptions: escalate to CISO

Security Deep Dive (Phase 3 — 15 min)

1. JIT Permission Set Design

IAM Identity Center Permission Sets (JIT levels):

PermissionSet: Operator-JIT
  Duration: up to 8 hours
  Inline policies:
    - ssm:StartSession on tagged instances
    - ec2:Describe* (read-only EC2)
    - cloudwatch:GetMetricData, cloudwatch:GetLogEvents (read logs)
    - s3:GetObject on ops-logs-{account-id}
  Denial boundaries:
    - Deny: iam:*, sts:AssumeRole on sensitive roles, kms:Delete*, s3:DeleteBucket

PermissionSet: SRE-JIT
  Duration: up to 4 hours
  Inline policies: All Operator + ec2:RebootInstances, ec2:StopInstances
  Deny: ec2:TerminateInstances

PermissionSet: Admin-JIT (REQUIRES SECOND APPROVER)
  Duration: up to 2 hours (shorter for highest privilege)
  Policies: AdministratorAccess
  Additional controls:
    - Requires two approvers (Step Functions parallel approval)
    - Immediate notification to CISO
    - Enhanced CloudTrail Insight enabled for session duration
    - All API calls generate Security Hub custom finding for review

PermissionSet: SecurityAudit-JIT
  Duration: up to 8 hours
  Policies: SecurityAudit (AWS managed) + CloudTrail:GetTrailStatus + Config:Get*
  No write access

2. Encryption

SSM session content:
  In transit: TLS 1.3 (SSM uses HTTPS; all session traffic encrypted)
  At rest (S3): SSE-KMS with ssm-sessions-key (CMK)
  CloudWatch Logs: encrypted with sessions-cwl-key (CMK)

DynamoDB access records:
  SSE-KMS with jit-access-records-key (CMK)
  Key policy: allow DynamoDB + Audit Lambda; deny all others

Approval JWT (Slack links):
  JWT signed with jit-approval-key (KMS RS256)
  TTL: 15 minutes (approval link expires)
  One-time use (token recorded in DynamoDB; replay rejected)

Break-glass secrets:
  No pre-staged break-glass credentials
  Break-glass uses STS AssumeRole (IAM Identity Center or direct)
  No stored passwords or keys for break-glass

3. Network Isolation

SSM Session Manager:
  SSM agent to SSM endpoint: VPC endpoint for ssm + ssm-messages + ec2messages
  No port 22 required on instances
  No bastion host required (SSM IS the bastion)

  VPC endpoint policy (restrictive):
    Allow: ssm:StartSession from engineer role ONLY
    Deny: * from 0.0.0.0/0 (restricts endpoint to VPC traffic only)

RDS database access:
  RDS: private subnets only (no public access)
  SSM port forwarding: engineer → EC2 bastion (SSM) → RDS
  OR RDS IAM auth: no port forwarding needed (app uses IAM token)

4. Logging and Audit Trail

Complete privileged access audit trail:
  Layer 1: IAM Identity Center access logs (CreateAccountAssignment + DeleteAccountAssignment events)
  Layer 2: STS AssumeRole (every privilege assumption logged in CloudTrail)
  Layer 3: SSM session records (who, when, which instance, session duration, keystrokes)
  Layer 4: AWS API calls (every action taken during the session in CloudTrail)
  Layer 5: Application logs (if engineer accessed an application during the session)

CloudTrail Detective Rule:
  EventBridge rule: AssumeRole for Admin-JIT-PermissionSet role
    Action: send to Security Hub as finding (INFORMATIONAL, requires review)
    Alert: if AssumeRole is from unexpected sourceIP (engineer's normal CIDR range not matched)

Session Manager log retention:
  CloudWatch: 14 days (operational), S3: 7 years (SOX compliance)
  S3 Object Lock: WORM (session logs cannot be modified or deleted)

5. Least Privilege and Blast Radius

JIT blast radius minimization:
  Each JIT role has only necessary permissions for the task type
  Duration is limited (1-8 hours max)
  Approval reduces human error / insider misuse
  Automatic expiry: even if engineer forgets to "check out", access auto-revokes

  If engineer is compromised during a JIT session:
    Session duration: max 8 hours (attacker has a limited window)
    Session is logged: if attacker uses the session, CloudTrail captures every API call
    SSM session content: keystrokes recorded — forensic evidence of attacker commands
    Immediate response: security team can call ssm:TerminateSession remotely

Scale and Resilience (Phase 4 — 10 min)

IAM Identity Center throughput:

  • CreateAccountAssignment: IAM Identity Center API supports ~100 assignment changes per second. For 300 engineers, this is well within limits. Terraform manages static assignments; JIT Lambda only handles the dynamic ones.

Step Functions concurrency:

  • Step Functions: no hard limit on concurrent executions. At peak (Monday morning, all 300 engineers request access simultaneously): 300 concurrent Step Functions executions — well within limits (default: 1 million standard workflow executions).

SSM Session Manager scale:

  • SSM: no per-session rate limit for Session Manager. Large-scale parallel sessions are supported (tested at thousands of concurrent sessions).

DynamoDB access records:

  • Table: on-demand capacity (auto-scales for access log writes).
  • TTL: DynamoDB TTL attribute expiresAt + 1 year (keep for 1 year, then auto-delete old records). Audit queries need 7 years → export to S3 via DynamoDB streams + Lambda before TTL deletion.

Trade-off Analysis — The Hard Choices

Decision 1: IAM Identity Center JIT vs. HashiCorp Vault Dynamic Credentials

Factor IAM Identity Center JIT (permission set assignments) HashiCorp Vault (dynamic IAM credentials)
AWS integration Native, deep Via Vault AWS secrets engine (API-based)
Multi-cloud AWS only Multi-cloud (AWS, Azure, GCP)
Operational cost No additional infrastructure Vault cluster (HA) must be maintained
Audit CloudTrail + IdC logs Vault audit log + CloudTrail
Recommendation IAM Identity Center JIT for AWS-only; Vault for multi-cloud or when PAM policy uniformity across clouds is required

Decision 2: SSM Session Manager vs. CyberArk/BeyondTrust PAM for Session Recording

Factor SSM Session Manager CyberArk/BeyondTrust
Session recording S3 (text log, not video) Video recording + keylogging
Cost Included in SSM Significant licensing cost ($200+/user/year)
Vault integration Via Secrets Manager Built-in privileged account vault
Compliance Satisfies SOX/PCI audit requirements Rich compliance reports
Recommendation SSM Session Manager for EC2 access is sufficient for SOX; CyberArk is justified if video session recording is a regulatory requirement (some financial regulators)

Decision 3: Auto-Revoke (Step Functions timer) vs. Manual Checkout/Checkin

Factor Auto-Revoke on Timer Manual Checkin (engineer checks out when done)
Access overhang Eliminated (access always bounded by timer) Possible (engineers forget to check in)
User experience Engineer may lose access mid-task Engineer controls end time
SOX compliance Stronger (no standing access, even briefly) Weaker (access persists until checkin)
Recommendation Auto-revoke on timer is the correct design for zero standing privilege; allow engineers to request an extension before expiry (Step Functions: send expiry warning 15 minutes before → engineer can request extension with new approval)

Decision 4: Session Recording — Text Log vs. Video Recording

Factor Text Log (SSM default) Video Recording (screen capture)
Storage size Small (text, < 100KB/session) Large (video, GB/session)
Forensic value Command audit trail Full visual context
Analysis CloudWatch Insights queries on commands Requires manual video review
Recommendation Text log (SSM) is sufficient for most use cases and SOX compliance; video recording adds significant storage cost and is only justified for highly sensitive systems or regulatory mandates

Common Mistakes to Avoid

  1. Describing SSM Session Manager without session recording. SSM Session Manager alone provides shell access, but without configuring session logging (CloudWatch Logs + S3), it's not audit-ready. SOX requires evidence of what was done during privileged sessions. Always configure aws:cloudWatchLogGroupName and aws:s3BucketName in SSM Session preferences document, and enable S3 Object Lock on the session log bucket.
  2. Not describing automatic revocation. Saying "JIT access" without explaining the revocation mechanism is incomplete. The JIT workflow must auto-revoke: DeleteAccountAssignment via Step Functions timer, STS:RevokeTokens for immediate termination if needed. "JIT" without auto-revocation is just a fancy approval workflow — standing privilege accumulates over time.
  3. Conflating break-glass with JIT. Break-glass is NOT a faster JIT (no approval). Break-glass is for genuine production emergencies where waiting 30 minutes for approval would cause worse damage than the risk of abuse. Design them separately: JIT has approval workflow + duration selector; break-glass has no approval + shorter TTL + immediate CISO notification + mandatory post-incident review.
  4. Forgetting that STS session tokens outlive IAM Identity Center assignments. When DeleteAccountAssignment is called, the engineer can no longer START a new session (can't refresh STS tokens). But existing STS tokens from that assignment continue to work until they expire (up to 8 hours). For immediate termination, call STS:RevokeAccessTokens with the policy: DenyAll on the role session — this immediately invalidates all active sessions for that role.

Likely Follow-up Questions

Q1: How do you handle database access for a DBA who needs to run emergency queries?
Database access uses two patterns. For emergencies: (1) DBA requests DBA-JIT permission set (2-hour max for emergency, requiring manager approval). (2) DBA uses SSM Session Manager port forwarding: aws ssm start-session --document-name AWS-StartPortForwardingSessionToRemoteHost --parameters host=rds.endpoint,portNumber=5432,localPortNumber=15432. This creates an encrypted SSH tunnel through SSM (no public RDS endpoint, no security group rule for the DBA's IP needed). (3) DB connection uses RDS IAM authentication token: aws rds generate-db-auth-token (valid for 15 minutes). No static DB password exists. The DBA's query session is logged: RDS PostgreSQL has pg_query_settings_log which logs all queries to CloudWatch Logs. For non-emergency access: same JIT workflow but with longer approval SLA (4-8 hours for planned maintenance).

Q2: How do you handle MFA during JIT access?
MFA enforcement: (1) IAM Identity Center requires MFA for all users (configured in IdC settings: RequireMFA = true). (2) When an engineer authenticates to IdC, they complete MFA (TOTP or hardware key) before accessing any accounts. (3) For JIT Admin-JIT permission sets: add an additional MFA step by configuring the IAM role's trust policy with condition: aws:MultiFactorAuthPresent = true. This forces a fresh MFA challenge even if the engineer already has an active IdC session. (4) Hardware MFA (FIDO2/WebAuthn): recommended for break-glass and Admin-JIT users — phishing-resistant MFA is required for privileged access in high-security environments. (5) MFA evidence is captured in CloudTrail: mfa_authenticated = true in the requestContext field of the AssumeRoleWithSAML or AssumeRoleWithWebIdentity event.

Q3: How do you implement the quarterly access review for SOX compliance?
SOX access review process: (1) At the start of each quarter, Lambda generates a report from DynamoDB: all engineer → account → permission set assignments in the past 90 days. (2) For each assignment, the report shows: how many times the JIT role was used, what resources were accessed, and the approving manager. (3) Report is uploaded to AWS Audit Manager (custom ITGC control: "Privileged access review performed"). (4) Each manager receives an email with a link to their team's access report (pre-signed S3 URL, 14-day expiry). (5) Managers must attest each record as "Appropriate" or "Should be revoked" via an Audit Manager UI or a simple web form (API Gateway + Lambda + DynamoDB). (6) Deadline: 30 days from report generation. Non-responses are escalated to CISO. (7) Revocations: any "Should be revoked" entries trigger an automated workflow: review whether the JIT role needs restructuring (was the wrong permission set granted?).

Q4: How do you handle 300 engineers across multiple time zones — what if the approver is asleep?
Approval escalation design: (1) Primary approver: engineer's direct manager (15-minute timeout → next step). (2) First escalation: security operations center (24/7 team) receives notification if primary approver doesn't respond within 15 minutes. (3) Second escalation (admin-level only): CISO on-call (via PagerDuty). (4) For Operator-level access during off-hours: automated approval for specific, pre-approved use cases (e.g., on-call runbook task). Define in Service Catalog product: requests matching specific justification_template_id values can be auto-approved (runbook-approved access). (5) For Production emergency access: break-glass path (no approval, immediate notification). (6) Step Functions workflow timeout and escalation chain handles all of this — each state has a heartbeat timeout, and failure transitions trigger escalation notifications.

Q5: What happens if the IAM Identity Center service is temporarily unavailable during a production incident?
IAM Identity Center availability: IdC uses AWS-managed infrastructure (99.9% SLA). However, for break-glass (production incident), the design should not depend on IdC being available. Break-glass contingency: (1) Break-glass IAM roles exist in each account as native IAM roles (not IdC-managed). These roles can be assumed directly via STS:AssumeRole without IdC. (2) Break-glass user: a dedicated IAM user (break-glass-admin@company.aws) with MFA hardware key, stored in a physical safe (for true break-glass). Credentials are split: password in safe A, MFA device in safe B. (3) This IAM user's credentials are audited quarterly (not used, just verified to exist and be valid). (4) If both IdC AND STS are unavailable (very rare AWS regional outage): the DR region is the fallback — DR region has identical IAM roles deployed via Terraform.

Q6: How do you ensure JIT access to Kubernetes (EKS) clusters?
EKS privileged access via JIT: (1) EKS cluster access is controlled via aws-auth ConfigMap mapping IAM roles to Kubernetes RBAC roles. (2) JIT EKS access: JIT workflow creates a temporary IAM role binding in the aws-auth ConfigMap (Lambda calls kubectl edit configmap aws-auth via EKS API or a privileged Lambda). This adds the JIT role ARN → Kubernetes ClusterAdmin (or specific namespace admin) for the duration. (3) kubectl access: engineer uses aws eks update-kubeconfig with the JIT role, then kubectl commands. (4) Kubernetes API server audit logging: enabled in EKS (sends to CloudWatch Logs). Every kubectl command generates an audit log: who + what resource + what action. (5) At JIT expiry: Lambda removes the aws-auth binding. Existing kubectl sessions (kubeconfig with the JIT role's STS token) stop working when the STS token expires. (6) Falco (runtime security): deployed in EKS to detect unusual in-cluster activity (even after kubectl access is granted).

Q7: How do you handle Terraform/CloudFormation deployments — do they go through JIT access?
CI/CD pipeline access is separate from human JIT access: (1) Terraform/CloudFormation: deployed by a CI/CD pipeline (CodePipeline, GitHub Actions). The CI/CD system has a dedicated IAM role (terraform-deploy-role) with narrow permissions (only what Terraform needs to manage). This role is NOT a human role — it's a service role, assumed by the CI/CD system via OIDC (GitHub Actions OIDC → STS AssumeRoleWithWebIdentity). (2) This role is NOT JIT (it's always available for the CI/CD system) but it's also not given to humans. (3) SCPs: deny terraform-deploy-role from being used by human (by restricting it to the CI/CD system's OIDC provider). (4) Terraform plan: required for all changes; Terraform apply: triggered only after PR approval + plan review. (5) Drift detection: Config + Terraform Cloud state file comparison — alerts if production resources don't match Terraform state (someone bypassed the pipeline).
