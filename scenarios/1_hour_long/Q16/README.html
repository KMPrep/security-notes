Q16 — Data Encryption Strategy for a Multi-Tier Application

The Ask

"Design a comprehensive data encryption strategy for a multi-tier SaaS application
running on AWS that handles healthcare data (HIPAA), financial records (PCI-DSS),
and general user data. The application has a web tier (React), API tier (ECS Fargate),
processing tier (Lambda), and data tier (Aurora, S3, DynamoDB, ElastiCache Redis).
Design encryption at rest, in transit, and in use, plus key hierarchy, rotation strategy,
and key access controls."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Is the application multi-tenant (multiple customers' data in the same database) or single-tenant per customer?
    Impact: Multi-tenant: each tenant's data encrypted with a separate KMS data key (tenant isolation — one tenant's compromise does not expose another's data). Single-tenant (customer per account): per-account KMS CMK (simpler). Multi-tenant with shared database: must use application-layer encryption with per-tenant key identifiers in every row.

  • Q: Who manages the encryption keys — AWS managed, customer managed (BYOK/HYOK)?
    Impact: AWS managed keys (aws/rds, aws/s3): AWS controls key rotation, no key policy customization. Customer managed CMKs: full key policy control, manual/automatic rotation, CloudTrail auditing of every use. BYOK (Bring Your Own Key): customer generates key material outside AWS and imports into KMS. HYOK (Hold Your Own Key): key never leaves on-premises HSM (CloudHSM); AWS can only use key while on-premises HSM authorizes. For HIPAA/PCI-DSS: CMKs are the minimum; HYOK for highest assurance.

  • Q: Are there regulatory requirements for key custody and separation of duties on key management?
    Impact: HIPAA: no specific key custody requirement, but audit logging of key access is required. PCI-DSS Req 3.7: dual control and split knowledge for key custodians. This means no single person can fully manage a KMS CMK — key creation, key policy changes, and key deletion should require two IAM principals (key admin + key user are separate roles; key deletion requires CISO approval).

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Multi-tenant SaaS — per-tenant KMS data key for PHI/PCI data.
  2. CMKs (not AWS managed keys) for all regulated data.
  3. CloudHSM backing for the highest-assurance PCI cardholder data encryption.
  4. Separate key hierarchy for each data classification tier.
  5. No encryption keys in application code or environment variables.

Core Architecture — Data Flow Narrative

[DATA CLASSIFICATION AND KEY HIERARCHY]

Tier 1: PHI (HIPAA) — Highest sensitivity
  KMS CMK: phi-master-key
    Key policy: allow phi-data-processor Lambda role to GenerateDataKey, Decrypt
                allow phi-security-audit role to DescribeKey only
                allow key-admin role to manage (not use)
                deny: all other principals
    Annual rotation: enabled (KMS automatic)
    CloudTrail: every use of phi-master-key logged (GenerateDataKey, Decrypt, etc.)

  Envelope encryption pattern (PHI fields):
    phi-master-key → Generate AES-256 data key (in memory only)
    Data key encrypts: PHI fields (name, DOB, SSN, MRN, diagnosis)
    Encrypted data key stored with ciphertext (in DynamoDB attribute)
    Original data key discarded from memory

    Decryption:
      Lambda: KMS Decrypt(encrypted_data_key) → data key in memory → decrypt PHI field
      Data key lives in Lambda memory for < 60s (request duration)
      Lambda concurrency limit: prevents key material from accumulating in many Lambda instances

Tier 2: PCI Cardholder Data (CDE) — High sensitivity
  CloudHSM Cluster: dedicated HSM nodes (2 for HA)
    FIPS 140-2 Level 3 hardware
    PAN encryption: RSA 2048 (or AES-256-GCM via CloudHSM library)
    KMS custom key store backed by CloudHSM (hybrid: KMS API + CloudHSM key material)

  KMS CMK (custom key store): pci-cardholder-key
    Key material: stored in CloudHSM (never leaves hardware boundary)
    API: KMS API (standard SDKs work — no CloudHSM SDK required for app)
    Rotation: manual (CloudHSM custom key store does not support auto-rotation)

  PAN tokenization: FPE (Format-Preserving Encryption) or random token
    Token vault: DynamoDB table (token → encrypted PAN)
    Encrypted PAN: CloudHSM encrypt with pci-cardholder-key
    Token: opaque, 16-digit random (Luhn-valid for legacy system compatibility)

Tier 3: Financial Records — Medium-High sensitivity
  KMS CMK: financial-records-key
    Automatic rotation: enabled
    Used for: Aurora table-level encryption, S3 financial-records bucket, DynamoDB transactions table

Tier 4: General User Data — Medium sensitivity
  KMS CMK: user-data-key
    Automatic rotation: enabled
    Used for: S3 user uploads bucket, DynamoDB user-profile table, CloudWatch Logs

[ENCRYPTION AT REST — PER SERVICE]

Aurora PostgreSQL (all tiers):
  Storage encryption: Aurora cluster encrypted with service-specific CMK at launch
    PHI tables: aurora-phi-key (separate CMK per data tier)
    Financial tables: financial-records-key
    General tables: user-data-key
    Note: Aurora storage encryption encrypts the entire cluster, not individual tables
    For column-level PHI encryption: use application-layer encryption (pgcrypto extension)

  Application-layer column encryption (PHI fields in Aurora):
    PostgreSQL: pgcrypto extension (pgp_sym_encrypt function)
    Key: derived from KMS GenerateDataKey (AES-256)
    Function: INSERT INTO patients (name_encrypted, dob_encrypted)
              VALUES (pgp_sym_encrypt(name, kms_data_key), pgp_sym_encrypt(dob, kms_data_key));
    Encrypted columns cannot be searched without decryption (limitation: accept this for PHI)
    Non-searchable PHI: use DynamoDB + DAX for lookup-by-token patterns

S3 (all buckets):
  PHI bucket: SSE-KMS with phi-master-key (default server-side encryption)
    Bucket policy: deny requests without valid encryption header
      "Condition": { "StringNotEquals": { "s3:x-amz-server-side-encryption": "aws:kms" }}
      "Effect": "Deny" — rejects any request that doesn't specify KMS encryption
    Access: VPC endpoint only (block all public access)
    Object Lock: WORM (7 years for HIPAA medical records)

  PCI bucket (settlement files): SSE-KMS with pci-cardholder-key
    Versioning: enabled (accidental delete recovery)
    S3 Inventory + Macie: scan for PAN in plaintext (alert if found)

  General uploads: SSE-S3 (acceptable for non-regulated data; lower cost)
    For sensitive user files: SSE-KMS with user-data-key

DynamoDB:
  PHI table: SSE-KMS with phi-master-key
    But: field-level encryption applied BEFORE writing to DynamoDB (app-layer AES-256)
    Why: DynamoDB SSE only protects the physical storage layer (transparent to app)
         Field-level encryption protects against DynamoDB API-layer data exposure

  PCI token vault: SSE-KMS with pci-cardholder-key
    Encryption: PAN field encrypted before DynamoDB write
    Index: token (not PAN) used as the sort key

  Financial/general tables: SSE-KMS with respective CMKs

ElastiCache Redis:
  At rest: ElastiCache encryption at rest enabled (AWS-managed key)
    Custom: use Secrets Manager to store Redis AUTH token (rotated every 30 days)

  Application-layer: sensitive cached values serialized as JSON and encrypted before caching
    Redis key format: cache:{tenant_id}:{resource_type}:{resource_id}
    Value: AES-256-GCM encrypted (per-tenant data key from KMS)
    TTL: 15-30 minutes (short TTL limits window for cached encrypted data exposure)

  Note: cache a derived data key (not the CMK) in Lambda memory; refresh every 5 minutes

CloudWatch Logs (Lambda + ECS logs):
  Log group encryption: SSE-KMS with logs-key CMK
  CRITICAL: ensure no PHI/PCI data logged (PAN masking + PHI field masking in application)
  Config rule: custom rule to scan CloudWatch Logs for PAN patterns (Macie for S3 logs)

[ENCRYPTION IN TRANSIT — ALL TIERS]

Web tier (React → API):
  CloudFront: TLS 1.2 minimum, TLS 1.3 preferred (AES-256-GCM, ECDHE key exchange)
  HSTS: Strict-Transport-Security header (1 year, includeSubDomains, preload)
  ACM certificate: wildcard cert for *.app.company.com (auto-renewed by ACM)
  CSP: Content-Security-Policy header (prevents injection of external scripts)

API tier → Data tier:
  Aurora: require_secure_transport = ON (force TLS for all connections)
    Certificate: ACM Private CA (for aurora endpoints in private subnets)
    Connection: ssl_mode = verify-full (client verifies server cert — no MITM)

  DynamoDB: HTTPS only (no plain HTTP endpoint for DynamoDB — AWS-enforced)

  ElastiCache Redis: in-transit encryption enabled (Redis TLS mode = tls)

  Lambda → KMS: HTTPS only (all KMS API calls are TLS)

  Internal service-to-service (ECS): mTLS (see Q13 — VPC Lattice)

[ENCRYPTION IN USE — SENSITIVE OPERATIONS]

  Lambda — ephemeral compute:
    PHI decryption: data key is decrypted into Lambda execution environment memory
    Lambda memory is NOT persisted across invocations (ephemeral filesystem)
    Memory encryption: AWS Nitro hypervisor isolates Lambda execution environments
    No swap disk: Lambda execution environments do not use swap (data stays in RAM)

  Processing tier (EC2 Nitro):
    AWS Nitro System: hardware-enforced isolation of EC2 instances
    Memory encryption: some Nitro instances support AMD SME (Secure Memory Encryption)
    AWS confidential computing: AWS Nitro Enclaves for sensitive processing
      Nitro Enclave: isolated compute environment with no persistent storage
      Use case: PAN decryption for authorization (PAN never leaves enclave)
      KMS: Nitro Enclave attestation → KMS allows decrypt only from verified enclave

[KEY ROTATION STRATEGY]

CMK automatic rotation (KMS):
  All CMKs: annual rotation enabled (KMS rotates backing key material)
  Effect: new data uses new key version; old data can still be decrypted (KMS tracks versions)

  Rotation does NOT re-encrypt existing data (S3 objects, Aurora volumes still use old data key)
  To re-encrypt data: run a re-encryption job (Lambda reads → decrypts with old key → encrypts with new key → writes)
  Re-encryption schedule: run after each annual CMK rotation for PHI data

CloudHSM (PCI custom key store):
  Manual rotation required (no automatic rotation)
  Schedule: annual (PCI-DSS Req 3.6.4)
  Process: create new key version → update application key version config → re-encrypt token vault

Data key rotation (envelope encryption):
  KMS data keys: short-lived (generated per request, discarded after use)
  Cached data keys: refresh every 5 minutes (Lambda encryption SDK caching provider)

[KEY ACCESS CONTROLS AND SEPARATION OF DUTIES]

Key admin role (key management, not data access):
  iam:CreateKey, kms:PutKeyPolicy, kms:ScheduleKeyDeletion
  Cannot: kms:Decrypt, kms:GenerateDataKey (cannot USE the key, only manage it)
  Requires: MFA for all key management actions

Key user role (data access, not key management):
  kms:Decrypt, kms:GenerateDataKey, kms:DescribeKey
  Cannot: kms:PutKeyPolicy, kms:ScheduleKeyDeletion
  Requires: IP restriction (VPC endpoint condition)

Key deletion policy (PCI-DSS dual control):
  kms:ScheduleKeyDeletion requires TWO approvals:
    Primary: Key admin role calls ScheduleKeyDeletion (7-day minimum waiting period)
    Confirmation: CISO approves (Step Functions approval workflow, same as PAM)
  CloudTrail: ScheduleKeyDeletion → SNS P1 → PagerDuty notification

[MULTI-TENANT KEY ISOLATION]
  Per-tenant data keys (PHI):
    Master key (phi-master-key) encrypts per-tenant data keys
    Per-tenant data key: generated once at tenant onboarding, stored encrypted in DynamoDB
    tenant-key-registry table:
      tenantId (partition key) → encrypted_data_key (base64) + created_at + rotated_at

    Data access pattern:
      Request comes in with tenantId header
      Lambda: DynamoDB.GetItem(tenantId) → encrypted_data_key
      Lambda: KMS.Decrypt(encrypted_data_key) → data key (in memory)
      Lambda: decrypt(PHI_field, data_key)
      Guarantee: tenant A's data key ≠ tenant B's data key

    Tenant offboarding (key deletion):
      Delete tenant's encrypted_data_key from DynamoDB
      All PHI for that tenant becomes permanently irrecoverable (crypto shredding)
      No need to physically delete all S3 objects (they're encrypted with deleted key)

2. Encryption Context Usage

Encryption context: additional authenticated data (not secret; used for audit)
  KMS Encrypt call:
    EncryptionContext: { "DataClassification": "PHI", "TenantId": "tenant-123" }

  KMS Decrypt call must provide SAME context:
    If context doesn't match: KMS rejects decrypt → decryption fails

  Benefits:
    1. Audit: CloudTrail shows encryption context (what tenant's data was accessed)
    2. Access control: key policy can restrict by context (only decrypt PHI context)
    3. Binding: ciphertext can only be decrypted in same context (prevents reuse attack)

  Alert: if Decrypt is called with unexpected encryption context → Security Hub finding

3. Network Isolation

KMS VPC endpoint:
  All KMS API calls go through VPC endpoint (not internet)
  VPC endpoint policy:
    Allow: kms:* from phi-processor-lambda-role (within VPC only)
    Deny: * from outside VPC
  Effect: Lambda can only use KMS while inside the VPC — prevents out-of-band key access

CloudHSM:
  HSM nodes: private subnets, no internet-facing interfaces
  HSM security group: inbound 2223 from Lambda/ECS security groups only
  CloudHSM elastic network interface in customer VPC: direct network access to HSM

4. Logging and Audit Trail

CloudTrail KMS events:
  Every kms:GenerateDataKey → who, when, which key, encryption context, success/failure
  Every kms:Decrypt → same
  KMS API call logs cannot be disabled (CloudTrail cannot be stopped per SCP)

HIPAA key access audit (Lambda for automated review):
  Query CloudTrail: kms:Decrypt for phi-master-key over the last 30 days
  Unexpected principals accessing key → Security Hub finding
  Unusual volume of key operations → GuardDuty custom finding

Macie for data scanning:
  S3 PHI buckets: Macie scans for PII/PHI patterns in plaintext
  Alert: if Macie finds SSN, DOB, diagnosis terms in plaintext → P1 alert
  (Indicates encryption is not being applied at application layer)

5. Least Privilege and Blast Radius

Per-tier key isolation:
  Compromise of user-data-key: only general user data exposed; PHI keys unaffected
  Compromise of financial-records-key: financial records exposed; PHI/PCI keys unaffected

Per-tenant key isolation:
  Compromise of tenant-A's data key: only tenant A's PHI exposed
  Tenant B's data is encrypted with a different key (stored separately in DynamoDB)

Lambda ephemeral key access:
  Data keys in Lambda memory are NOT shared across concurrent invocations
  Lambda execution environment isolation: no shared memory between invocations
  Exception: Lambda warm invocations reuse the same execution environment
    → cached data key may persist for a few minutes
    → set cache TTL appropriately (5 minutes max for PHI data keys)

Scale and Resilience (Phase 4 — 10 min)

KMS throughput:

  • KMS API limits: 10,000 requests/second per region (GenerateDataKey + Decrypt combined).
  • Encryption SDK local cache: cache data keys for 60 seconds × 1,000 operations (reduces KMS calls by 99% for high-throughput services).
  • Cache design: per Lambda execution environment; cache TTL for PHI = 60 seconds; for general data = 300 seconds.

CloudHSM availability:

  • CloudHSM cluster HA: minimum 2 HSM nodes in different AZs (AWS recommends 3).
  • Failover: automatic (CloudHSM client handles it transparently).
  • Performance: each HSM node handles ~1,000 RSA operations/second; 3 nodes = 3,000 ops/second.
  • For high-volume PCI: FPE tokenization is done once at card-present; token is used thereafter (reduces CloudHSM call frequency).

Aurora encrypted storage:

  • Aurora encryption is transparent — no performance impact compared to unencrypted (encryption is done at storage layer, below the database engine).
  • Application-layer column encryption (pgcrypto): adds ~1ms per row read/write. Acceptable for PHI access patterns (typically low-volume, not scan operations).

Trade-off Analysis — The Hard Choices

Decision 1: KMS CMK vs. CloudHSM for PCI Cardholder Data

Factor KMS CMK CloudHSM (via KMS custom key store)
FIPS compliance FIPS 140-2 Level 2 FIPS 140-2 Level 3
PCI P2PE/PIN certification Not PCI P2PE certified PCI P2PE and PIN certified hardware
Cost ~$1/month/key + API fees ~$1.45/hour per HSM node (~$1,044/month per node)
Operational overhead Fully managed Must manage HSM cluster
Recommendation KMS CMK for most PCI-DSS scenarios (Req 3 encryption); CloudHSM when PCI P2PE/PIN security hardware certification is required

Decision 2: Server-Side Encryption (SSE-KMS) vs. Client-Side Encryption (CSE-KMS) for S3

Factor Server-Side (SSE-KMS) Client-Side (CSE-KMS)
Where encryption happens S3 service layer Application before sending to S3
S3 service exposure S3 sees plaintext during processing (S3 Select, Replication) S3 never sees plaintext
Application complexity None Requires Encryption SDK in app
Key management AWS manages key calls Application manages key calls
Recommendation SSE-KMS for general data (S3 operational features preserved); CSE-KMS for highest-sensitivity PHI (S3 never has plaintext — zero trust even toward S3 service layer)

Decision 3: Column-Level Encryption vs. Full-Table Encryption for Aurora

Factor Aurora Storage Encryption (full-table) Application-Layer Column Encryption (pgcrypto)
Granularity Entire cluster encrypted Individual columns
Query performance No impact (transparent) 1ms overhead per encrypted row; cannot index encrypted columns
Protection level Protects disk layer Protects application + DB API layer
Compliance strength HIPAA safe harbor (encrypted at rest) Stronger: even DB admin can't read PHI in plaintext
Recommendation Both: Aurora storage encryption (defense in depth) + column-level for specific PHI fields (name, DOB, SSN) — prevents DB admin from reading PHI directly

Decision 4: Key Per Tenant vs. Key Per Data Classification

Factor Key Per Tenant Key Per Data Classification
Blast radius Limited to one tenant's data All tenants' data of that type
KMS key count Grows with tenant count (costly at scale) Fixed count (5-10 keys total)
Tenant deletion Crypto shredding (delete tenant key) Cannot selectively delete one tenant's data
Recommendation Key per data classification for small-medium tenant count; Master key + per-tenant data key (envelope encryption, keys stored in DynamoDB) for large multi-tenant at scale

Common Mistakes to Avoid

  1. Saying "just use SSE-KMS on S3 and Aurora" without addressing application-layer encryption. SSE-KMS at the storage layer only protects against physical disk theft. Any authorized AWS API call (even by the wrong IAM principal) returns plaintext. For PHI/PCI, application-layer envelope encryption ensures that even if an IAM principal is compromised, they still can't decrypt PHI without being the specific Lambda role with the specific encryption context.
  2. Not explaining envelope encryption. Many candidates say "use KMS to encrypt data" without describing the key hierarchy. KMS cannot directly encrypt large data sets (KMS limit: 4 KB per Encrypt call). The correct pattern is envelope encryption: KMS generates a data key → data key encrypts the data → data key is discarded (or stored encrypted alongside the ciphertext). Not describing this shows you don't understand how KMS is actually used in production.
  3. Forgetting key rotation doesn't re-encrypt data. Annual KMS CMK rotation updates the backing key material for NEW encryptions but does NOT re-encrypt existing data. Old data is still encrypted with the old key version (KMS tracks which version to use for decryption). To actually rotate all existing ciphertext to the new key material, a separate re-encryption job must be run. Candidates who say "enable annual rotation and you're done" are wrong.
  4. Ignoring encryption in use. Most candidates cover at-rest and in-transit. Encryption in use (data while being processed in RAM) is the hardest and most valuable differentiator. AWS Nitro Enclaves provide hardware-isolated execution for sensitive processing (PAN decryption, PHI de-identification). Mentioning confidential computing / Nitro Enclaves is a strong signal of depth.

Likely Follow-up Questions

Q1: How do you handle key management for 10,000 tenants?
At 10,000 tenants, a KMS key per tenant is impractical ($10k+/month in key fees + management overhead). The correct pattern is master key + per-tenant data key: (1) One KMS CMK: phi-master-key (single key for the PHI classification). (2) At tenant onboarding: Lambda calls kms:GenerateDataKey → get a random AES-256 data key. Store the encrypted data key in DynamoDB (tenant-key-registry table, partitioned by tenantId). Discard the plaintext data key. (3) At data access: Lambda fetches the tenant's encrypted data key from DynamoDB → calls kms:Decrypt → gets plaintext data key in memory → decrypts tenant's PHI field → discards data key. (4) Crypto shredding at offboarding: delete the tenant's row in tenant-key-registry. All their PHI data is permanently inaccessible (no need to delete actual S3/DynamoDB records). This pattern scales to millions of tenants with a single KMS CMK.

Q2: What is the impact of KMS API throttling on application performance?
KMS default limits: 10,000 API calls per second (us-east-1; varies by region). For high-throughput applications: (1) Use the AWS Encryption SDK with local data key caching (LocalCryptoMaterialsCache). Cache parameters: max_age=60, max_messages_encrypted=1000. This allows 1,000 encryptions using the same data key (reusing it in memory), reducing KMS calls by 99%. (2) Connection pooling for KMS: use the KMS SDK with persistent HTTP connections (avoid creating a new boto3 client per Lambda invocation — create the client at module level, not inside the handler). (3) Throughput estimation: if Lambda serves 10,000 requests/second and each caches a data key for 60s × 1,000 messages, you need approximately 10 KMS calls/second — well within limits. (4) CloudWatch metric: ThrottlingExceptionCount for KMS API — alert if > 0.

Q3: How do you implement crypto shredding for GDPR "right to erasure"?
GDPR right to erasure using crypto shredding: (1) Per-user encrypted data key: at user registration, generate a user-specific AES-256 data key (via kms:GenerateDataKey). Store the encrypted data key in a DynamoDB table (user-key-registry). (2) All PII for that user is encrypted with their user-specific data key (not the CMK directly — envelope pattern). (3) On erasure request: delete the user's row from user-key-registry. The user's data key is now unrecoverable. All their PII data (in S3, DynamoDB, Aurora) is permanently indecipherable — effectively erased. (4) Physical data: the encrypted ciphertext can remain in databases (it's just random bytes without the key). This satisfies GDPR right to erasure without needing to locate and delete every data record. (5) Audit trail: log the erasure event in CloudTrail + Audit Manager (evidence for GDPR compliance).

Q4: How do you protect encryption keys in transit between Lambda and KMS?
All Lambda → KMS communication is over HTTPS (TLS 1.3). Additional protections: (1) KMS VPC endpoint: Lambda calls KMS via VPC endpoint (not via internet). VPC endpoint policy restricts access to specific Lambda role ARNs. Network traffic stays within AWS backbone (never leaves to internet). (2) DNS: KMS VPC endpoint hostname (kms.us-east-1.amazonaws.com) resolves to private IP within the VPC when endpoint is active. (3) Certificate pinning: Lambda SDK doesn't support cert pinning for KMS, but the VPC endpoint + private subnet architecture achieves the same goal (no MITM possible within AWS VPC). (4) IAM request signing: all KMS API calls are Sigv4-signed with the Lambda role credentials — a MITM cannot forge requests even if they intercept the TLS connection, because the signature would be invalid. (5) Encryption context: even if a KMS request is intercepted and replayed, the decrypted data is only useful with the matching ciphertext — the data key and ciphertext are inseparable.

Q5: How does Nitro Enclave-based encryption in use work for PAN decryption?
Nitro Enclave design for PAN decryption: (1) An EC2 Nitro instance runs a normal parent instance + an isolated enclave (enclave is a separate VM with no persistent disk, no network except via local VSOCK channel to parent instance). (2) KMS key policy for pci-cardholder-key: requires kms:RecipientAttestation condition: the PCR (Platform Configuration Register) hash of the enclave image must match an expected value. Only the exact, verified enclave binary can use this key. (3) Flow: encrypted PAN arrives at the parent instance → parent forwards to enclave via VSOCK → enclave requests kms:Decrypt via KMS Attestation API → KMS verifies PCR hash → decrypts → enclave performs authorization decision → returns ONLY the result (approved/denied, last 4 digits), never the plaintext PAN → back to parent via VSOCK. (4) If an attacker compromises the parent EC2 instance, they still cannot access the PAN — they'd need to compromise the enclave binary itself (which would change the PCR hash and be denied by KMS).

Q6: How do you audit which specific IAM principal accessed PHI and when?
PHI access audit using CloudTrail + encryption context: (1) Every kms:Decrypt for phi-master-key generates a CloudTrail event with: event time, IAM principal ARN (role + session name), source IP, encryption context (TenantId, DataClassification, ResourceId). (2) Athena query on CloudTrail logs: SELECT eventTime, userIdentity.arn, requestParameters.encryptionContext FROM cloudtrail_logs WHERE eventName = 'Decrypt' AND requestParameters.keyId = 'arn:aws:kms:...:key/phi-master-key' AND requestParameters.encryptionContext.TenantId = 'tenant-123'. This returns every PHI decryption for tenant-123, by whom, and when. (3) For application-level logging: Lambda logs the business operation (e.g., "physician Dr. Smith accessed patient record 12345") to CloudWatch Logs WITHOUT logging any PHI field values. This gives the business action audit trail. (4) HIPAA access logs + CloudTrail KMS events together provide the complete PHI access audit: CloudTrail = technical (what key was used), application logs = business context (why was it accessed).

Q7: How do you handle key management across multiple AWS regions for a global application?
Multi-region encryption strategy: (1) KMS Multi-Region Keys (MRK): create primary CMK in us-east-1, replicate to eu-west-1. Same key material in both regions — data encrypted in us-east-1 can be decrypted in eu-west-1 (needed for DR and data replication). (2) GDPR consideration: if EU users' PII must stay in EU, use region-specific CMKs (NOT MRK). Data encrypted in eu-west-1 with eu-only-key cannot be decrypted in us-east-1 (even by EU subsidiary's US parent). (3) Regional key policy: apply SCPs restricting eu-only-key usage to eu-west-1 region only (aws:RequestedRegion condition). (4) Cross-region backup: for DR, use S3 Cross-Region Replication. Encrypted S3 objects replicated to DR region must use MRK (to be decryptable). (5) Key rotation: when MRK primary rotates annually, all replicas receive the same new key material automatically. No separate rotation per region.

Q8: How would you detect if unencrypted sensitive data is being uploaded to S3 by a developer?
Preventive + detective approach: (1) S3 Bucket Policy (preventive): add condition to deny any PutObject request that doesn't specify x-amz-server-side-encryption: aws:kms header. Developers' uploads without encryption header are automatically rejected. (2) However: this only enforces S3-level encryption, not application-layer encryption. A developer could upload a plaintext file that happens to use SSE-KMS. (3) Macie (detective): run a continuous Macie discovery job on PHI S3 buckets. Macie detects SSN, DOB, medical record patterns in plaintext. If found: Macie generates a finding → EventBridge → Lambda → Security Hub HIGH finding + Slack P1 alert. (4) CloudTrail + Lambda (detective): monitor S3 PutObject events for files that don't use expected CMK key ID. Lambda: if x-amz-server-side-encryption-aws-kms-key-id ≠ expected phi-master-key ARN, trigger alert (may be using wrong key). (5) CI/CD guardrail: in test/staging, run a post-deployment script that calls Macie + inspects bucket for unencrypted data before promoting to production.
