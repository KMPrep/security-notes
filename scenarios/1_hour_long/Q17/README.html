Q17 — SIEM on AWS

The Ask

"Design a cloud-native SIEM (Security Information and Event Management) platform
on AWS for an enterprise with 500 AWS accounts, on-premises data centers, and 200
SaaS applications. The SIEM must ingest 500 GB/day of security logs, support real-time
threat detection with sub-5-minute alert latency, enable threat hunting with arbitrary
SQL queries over 2 years of historical data, and integrate with MITRE ATT&CK framework.
The platform must be scalable, cost-effective, and support Tier-1 SOC analyst workflows."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What is the current log source inventory? Which sources are structured (JSON), semi-structured (CEF/LEEF), and unstructured (syslog)?
    Impact: Structured sources (CloudTrail, VPC Flow Logs, GuardDuty) → Amazon Security Lake (native OCSF normalization). Semi-structured and custom sources → Kinesis Firehose + Lambda normalization to OCSF. Unstructured syslog → Amazon Kinesis Data Streams + Lambda parser. The mix determines the normalization pipeline complexity and cost.

  • Q: Is real-time correlation (correlate events across multiple log sources within a 60-second window) required, or is near-real-time (5-minute latency) sufficient?
    Impact: Sub-60-second correlation requires Kinesis Data Streams + Flink (Amazon Managed Service for Apache Flink) — stateful stream processing. 5-minute latency accepts EventBridge Pipes + Lambda-based rule evaluation. The stream processing path is significantly more expensive but enables complex multi-event correlation (e.g., detect port scan from IP X, followed by successful login from same IP within 5 minutes).

  • Q: What is the retention requirement for hot (queryable) vs. cold (archived) data?
    Impact: 2 years queryable = 500 GB/day × 730 days = 365 TB of Security Lake (S3 Parquet). Hot tier (recent 90 days): S3 Standard. Warm tier (90d-1yr): S3 Intelligent-Tiering. Cold tier (1-2yr): S3 Glacier Instant Retrieval (queryable via Athena, but with higher latency and cost). This directly impacts cost model.

  • Q: Are there compliance requirements that mandate log tamper-evidence and chain-of-custody?
    Impact: HIPAA, PCI-DSS, SOX all require log integrity. S3 Object Lock (WORM mode, 2 years) on the Security Lake bucket prevents deletion or modification by any principal, including root. CloudTrail log file integrity (SHA-256 hash of each log file → stored in separate digest files) provides chain-of-custody for AWS API logs.

  • Q: What SaaS applications need to ingest? What are their log delivery mechanisms?
    Impact: Most SaaS (Okta, Salesforce, Google Workspace) support webhook → Kinesis Data Firehose OR HTTPS endpoint. Some require polling (REST API pull). This determines whether a push-based or pull-based ingestion lambda pattern is needed for each SaaS connector.

  • Q: What is the expected alert volume and false positive rate target for Tier-1 SOC analysts?
    Impact: High false positive rate (> 10%) forces analysts to spend time validating false alarms instead of real threats. Use ML-based anomaly detection (GuardDuty, Amazon Detective behavior graphs) to reduce noise. Set alert cardinality limits per detection rule. Tier-1 queue SLA: < 30 minutes to triage each alert.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Amazon Security Lake as the central normalized log store (OCSF v1.1).
  2. Kinesis Data Firehose for log ingestion from custom/on-prem/SaaS sources.
  3. Amazon OpenSearch Service as the SIEM UI (dashboards, real-time search, alert rules).
  4. Amazon Managed Service for Apache Flink for stateful real-time correlation.
  5. Amazon Athena for threat hunting SQL queries across 2-year Security Lake history.
  6. Amazon Detective for graph-based investigation (related entities, timeline).

Core Architecture — Data Flow Narrative

[INGESTION LAYER]

AWS-Native Sources (auto-ingested by Security Lake):
  CloudTrail (management + data events) → Security Lake → OCSF Activity class
  VPC Flow Logs → Security Lake → OCSF Network Activity class
  Route 53 DNS Resolver logs → Security Lake → OCSF DNS Activity class
  Security Hub findings → Security Lake → OCSF Security Finding class
  EKS Audit Logs → Security Lake → OCSF API Activity class
  Lambda data events → Security Lake (via CloudTrail) → OCSF Activity class
  WAF access logs → Security Lake → OCSF HTTP Activity class

  Security Lake: creates separate AWS account (log archive account)
    Org-level configuration: all member accounts auto-contribute to central Security Lake
    Storage: S3 bucket in log archive account (encrypted with KMS CMK)
    Format: Apache Parquet (columnar, compressed — efficient for Athena)
    Partitioning: year/month/day/hour/account-id/region (Athena partition pruning)

On-Premises Sources (push via Kinesis Agent):
  Palo Alto NGFW (CEF format) → Kinesis Data Streams → Lambda normalizer → OCSF JSON → Security Lake custom source API
  CrowdStrike EDR (JSON) → Kinesis Data Firehose → S3 buffer → Lambda transformer → Security Lake
  Active Directory (Windows event logs) → Kinesis Agent (installed on Windows) → Kinesis Data Streams → Lambda parser → Security Lake
  Cisco ISE (RADIUS auth logs) → syslog → CloudWatch Logs Agent → Kinesis Firehose → Lambda → Security Lake

SaaS Sources (pull via scheduled Lambda):
  Okta: Lambda polls Okta System Log API (every 2 min) → normalize to OCSF IAM Activity → Security Lake
  Salesforce: Lambda polls Salesforce Event Monitoring API (hourly) → normalize → Security Lake
  Google Workspace: Lambda polls Admin SDK Reports API → normalize → Security Lake
  GitHub: Lambda polls GitHub Audit Log API → normalize → Security Lake

  SaaS ingestion rate control:
    Lambda: reserved concurrency = 10 per SaaS connector (prevent rate limit exhaustion)
    Exponential backoff on API rate limit responses (429)
    SQS Dead Letter Queue: failed normalizations for manual investigation

[NORMALIZATION — OCSF v1.1]

OCSF (Open Cybersecurity Schema Framework):
  All sources normalized to common event schema before storage
  Benefits: single SQL query syntax across all log sources (Athena)
  Key OCSF classes used:
    1001 - File System Activity (malware, data exfil)
    1003 - Network Activity (flow logs, firewall)
    1004 - DNS Activity (DNS tunneling, C2 detection)
    3001 - Account Change (IAM activity)
    3002 - Authentication (login events, MFA)
    6001 - Security Finding (GuardDuty, Inspector findings)

  Lambda normalizer architecture:
    Input: source-specific format (CEF, JSON, syslog, XML)
    Mapping: config-driven (JSON mapping file per source type, stored in S3)
    Output: OCSF JSON
    Validation: JSON schema validation (reject malformed events to DLQ)

[REAL-TIME DETECTION — KINESIS FLINK]

Kinesis Data Streams:
  All normalized events also stream to Kinesis Data Streams (in parallel with Security Lake write)
  Partition key: event source type (ensure ordering within source)
  Retention: 7 days (re-processing window for rule updates)

Amazon Managed Service for Apache Flink (real-time correlation):
  Flink app reads from Kinesis Data Streams
  Detection rules implemented as Flink SQL queries:

  Rule: Brute Force Detection
    SELECT source_ip, COUNT(*) as attempts, TUMBLE_END(event_time, INTERVAL '5' MINUTE)
    FROM authentication_events
    WHERE status = 'FAILED'
    GROUP BY source_ip, TUMBLE(event_time, INTERVAL '5' MINUTE)
    HAVING COUNT(*) > 10
    → Alert: brute force from source_ip

  Rule: Impossible Travel
    SELECT user_id, a1.source_country as country1, a2.source_country as country2,
           a1.event_time as t1, a2.event_time as t2,
           TIMESTAMPDIFF(MINUTE, a1.event_time, a2.event_time) as minutes_between
    FROM authentication_events a1
    JOIN authentication_events a2 ON a1.user_id = a2.user_id
    WHERE a1.status = 'SUCCESS' AND a2.status = 'SUCCESS'
      AND a1.source_country != a2.source_country
      AND TIMESTAMPDIFF(MINUTE, a1.event_time, a2.event_time) < 120
    → Alert: impossible travel for user_id

  Flink → Kinesis Firehose → OpenSearch (< 30 seconds ingestion latency)
  Flink → SNS → PagerDuty (P1/P2 alert notifications, < 60 seconds)
  Flink alert output → DynamoDB (alert state tracking, deduplication window = 1 hour)

[THREAT HUNTING — ATHENA + SECURITY LAKE]

Security Lake Athena integration:
  AWS Lake Formation: governs access to Security Lake Glue Data Catalog
  Table: s3://security-lake-bucket/ocsf/aws/v1/
  Partitioned by: year, month, day, accountid, region, source

  Analyst access:
    SOC Analyst role: Athena query permissions + Lake Formation SELECT on security tables
    Cannot: modify Security Lake data, delete partitions, write to S3
    Query quota: max 10 concurrent queries per analyst (cost control)

  Sample threat hunting queries:

  1. Find all lateral movement from compromised host:
    SELECT DISTINCT dst_endpoint.ip, dst_endpoint.port, COUNT(*) as connections
    FROM vpcflowlogs_ocsf
    WHERE src_endpoint.ip = '10.0.1.100'
      AND action = 'ALLOWED'
      AND time > current_timestamp - interval '24' hour
    GROUP BY dst_endpoint.ip, dst_endpoint.port
    ORDER BY connections DESC
    LIMIT 50;

  2. Detect DNS exfiltration (high entropy subdomains):
    SELECT query_name, COUNT(*) as requests, src_endpoint.ip
    FROM dns_ocsf
    WHERE LENGTH(query_name) > 50
      AND REGEXP_LIKE(query_name, '[0-9a-f]{20,}')  -- base64/hex subdomains
      AND time > current_timestamp - interval '1' hour
    GROUP BY query_name, src_endpoint.ip
    ORDER BY requests DESC;

  3. Find credential access events correlating with new access keys:
    SELECT a.time, a.src_endpoint.ip, a.user.name as user, b.time as key_create_time
    FROM cloudtrail_ocsf a
    JOIN cloudtrail_ocsf b ON a.user.name = b.user.name
    WHERE a.activity_name = 'AssumeRole'
      AND b.activity_name = 'CreateAccessKey'
      AND TIMESTAMPDIFF(MINUTE, b.time, a.time) < 5
    ORDER BY a.time DESC;

[SIEM UI — AMAZON OPENSEARCH SERVICE]

OpenSearch cluster (dedicated master nodes + data nodes):
  2 dedicated master nodes (m6g.large), 4 data nodes (r6g.2xlarge)
  Multi-AZ deployment (2 AZs)
  Encryption at rest: KMS CMK; in transit: TLS 1.3
  Fine-grained access control (FGAC):
    Security Hub admin: access to all indexes
    SOC Tier-1 analyst: access to alerts + last 30 days of events
    Threat hunter: access to all historical indexes (read-only)

  Index design:
    siem-alerts-YYYY.MM (alert queue for SOC workflow)
    siem-events-YYYY.MM (searchable events — 90 days on hot storage)
    → ILM policy: move siem-events-* to warm (UltraWarm) after 90 days; delete after 1 year
    (Security Lake S3 retains full 2-year history for Athena queries)

  OpenSearch Dashboards:
    Alert Queue: live Tier-1 SOC analyst queue (filter by severity, status, assignee)
    MITRE ATT&CK Heat Map: detection coverage mapped to ATT&CK tactics/techniques
    Account Risk Score: per-account risk score (GuardDuty findings + alert volume)
    Geographic Threat Map: source IPs by country (authentication failures, port scans)
    Detection Metrics: MTTD (mean time to detect), false positive rate per rule

[MITRE ATT&CK MAPPING]

Each detection rule tagged with MITRE ATT&CK technique:
  Brute Force (T1110) → rule in Flink: > 10 failed auth in 5 minutes
  Valid Accounts (T1078) → rule: successful login from new country (impossible travel)
  DNS Exfiltration (T1071.004) → rule: high-entropy subdomains > 50 chars
  Credential Dumping (T1003) → rule: lsass.exe memory access (CrowdStrike EDR event)
  Lateral Movement SMB (T1021.002) → rule: SMB connections from non-admin workstations

Tags propagated to OpenSearch alerts:
  alert.mitre_tactic = "TA0006"  (Credential Access)
  alert.mitre_technique = "T1110"  (Brute Force)

MITRE Coverage dashboard:
  Count of detection rules per technique → identifies blind spots
  Heatmap: technique coverage vs. technique prevalence in GuardDuty findings

[INCIDENT WORKFLOW]

EventBridge rule:
  Security Hub finding CRITICAL → EventBridge → Step Functions incident playbook
  OpenSearch alert (severity=HIGH) → Lambda → ServiceNow ticket + Slack alert

Alert deduplication (DynamoDB):
  Same rule + same source_ip + same user → deduplicate for 1 hour window
  Prevents alert storm: single IP triggering 1,000 brute force alerts → 1 ticket

Tier-1 analyst workflow:
  1. Alert appears in OpenSearch Dashboard queue
  2. Click alert → deep-link to Amazon Detective investigation
  3. Detective: shows related entities (IAM user, EC2 instance, IP, related CloudTrail events)
  4. Detective: behavior graph — is this user's behavior anomalous vs. 30-day baseline?
  5. Analyst: escalate to Tier-2 or close as false positive
  6. SOAR: if escalated, Step Functions playbook begins automated response

[COST OPTIMIZATION]

Security Lake storage cost:
  500 GB/day × 730 days = 365 TB
  Parquet compression: 4:1 ratio → 91 TB actual S3 storage
  S3 Standard (90 days): 91 TB × 0.30 = $7,280 (90 days)
  S3 Intelligent-Tiering (90d-1yr): lower per-GB cost as objects move to infrequent access
  S3 Glacier Instant Retrieval (1-2yr): $0.004/GB = ~$90/month for archive tier

Athena query cost:
  $5 per TB scanned. Partition pruning + Parquet: 99% scan reduction
  Typical threat hunt query (1 day, 1 account): scans ~0.5 GB = $0.0025 per query

OpenSearch UltraWarm (90d-1yr):
  Cost: 87% cheaper than hot storage
  Tradeoff: higher query latency for warm data (seconds vs. milliseconds)

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Security Lake account (log archive):
  Dedicated AWS account — only SIEM platform roles can access
  SCP: deny all CreateBucket, DeleteBucket, PutBucketPolicy from member accounts
  Security Lake data access:
    SOC Analyst role: Lake Formation SELECT on security_lake database (read-only)
    SIEM Admin role: manage Security Lake sources, subscribers
    Threat Hunter role: full read access + Athena saved queries

  No cross-account AssumeRole from production accounts to log archive
  Security Lake subscriber: OpenSearch pulls events via SQS notification (Security Lake sends SQS message when new object available; OpenSearch ingestion Lambda reads S3)

2. Encryption

Security Lake S3: SSE-KMS with siem-lake-key CMK
  KMS key policy: only Security Lake service + SOC roles can Decrypt

OpenSearch: encryption at rest (KMS CMK), TLS 1.3 for all connections
CloudWatch Logs (Lambda logs, Flink logs): encrypted with logs-key CMK
Kinesis Data Streams: server-side encryption (KMS CMK)
DynamoDB (alert deduplication): SSE-KMS

3. Network Isolation

All SIEM infrastructure: private subnets (no public IPs)
OpenSearch: VPC mode (not public endpoint)
Athena: VPC endpoint for S3 access
Security Lake S3: VPC endpoint (bucket policy denies access without VPC endpoint)
On-premises connectivity: Direct Connect (for log shipping from on-prem — not internet)
SaaS connectors: Lambda in private subnet → NAT Gateway → internet (for SaaS API polling)

4. Logging and Audit Trail

SIEM audit logs:
  Athena queries: logged in CloudTrail + Athena history (who queried what, when, data scanned)
  OpenSearch access: logged (every index read, dashboard access)
  Security Lake data access: CloudTrail S3 data events (every GetObject to log bucket)
  Alert lifecycle: DynamoDB (created, acknowledged, escalated, closed, false positive)

SOC analyst actions:
  Alert assigned to, closed as FP, escalated → all written to DynamoDB + CloudWatch Logs
  Quarterly analyst performance report: Athena query on alert DynamoDB table

5. Least Privilege and Blast Radius

SOC Tier-1 analyst cannot:
  Delete any log data (no S3 DeleteObject permission)
  Modify detection rules (Flink app code in CodeDeploy — separate SIEM Engineer role)
  Access production AWS accounts directly
  Disable GuardDuty or Security Hub (SCP prevents this)

Blast radius of Security Lake compromise:
  If Security Lake S3 bucket is accessed by attacker: they can read logs but NOT execute commands (no write access to production accounts)
  S3 Object Lock: even if compromised account attempts to delete logs, WORM prevents it for 2 years
  Encryption: logs encrypted at rest — exfiltrated data is ciphertext without kms:Decrypt

Scale and Resilience (Phase 4 — 10 min)

Ingestion throughput:

  • 500 GB/day = ~5.8 MB/second average; peak = 3-5× average = ~29 MB/second.
  • Kinesis Data Streams: 1 MB/second per shard. Need 29 shards minimum; provision 40 shards for headroom.
  • Kinesis Enhanced Fan-Out: Flink + OpenSearch Lambda both read from the same stream simultaneously (each gets their own 2 MB/second consumer throughput — no contention).

OpenSearch scaling:

  • UltraWarm: move indexes older than 90 days to UltraWarm (S3 + local cache). 87% cost savings vs. hot storage. Warm index query latency: 1-5 seconds (acceptable for investigation; real-time search on hot indexes < 100ms).
  • Auto-scaling: OpenSearch Service supports Auto-Tune (automated shard allocation + query optimization) and alerting on cluster health.

Flink auto-scaling:

  • Amazon Managed Apache Flink: serverless mode (auto-scales KPUs based on lag in Kinesis stream). No manual scaling required.

Cross-region DR:

  • Security Lake: S3 Cross-Region Replication to DR region (all logs replicated).
  • OpenSearch: cross-cluster replication (CCR) for alert indexes to DR region.
  • RTO: < 4 hours (OpenSearch cluster rebuild from Security Lake S3 in DR region).

Trade-off Analysis — The Hard Choices

Decision 1: Security Lake + Athena vs. Third-Party SIEM (Splunk, Sumo Logic)

Factor AWS-Native (Security Lake + OpenSearch + Athena) Third-Party SIEM (Splunk Enterprise)
Cost ~$30-50K/month at 500 GB/day $150K-$300K/month (Splunk licensing based on data volume)
AWS integration Native (no forwarder needed for CloudTrail, GuardDuty) Requires Splunk AWS Add-On (forwarder)
Query language ANSI SQL (Athena), Lucene (OpenSearch) SPL (Splunk Processing Language) — proprietary
Real-time correlation Flink (Flink SQL) Splunk ES correlation searches (slower, less scalable)
Recommendation AWS-native for AWS-heavy environments and cost sensitivity; Splunk if team has deep SPL expertise and needs advanced UEBA features

Decision 2: Amazon Managed Flink vs. Lambda for Real-Time Detection

Factor Apache Flink (Managed) Lambda-based Rules
Stateful correlation Yes (windowed joins, state management) Limited (must use DynamoDB for state)
Detection latency < 30 seconds 1-5 minutes (batch trigger)
Complexity High (Flink SQL knowledge required) Low (Python Lambda)
Cost KPU-based ($0.11/KPU-hour) Pay-per-invocation (can be cheaper for low-volume rules)
Recommendation Flink for complex multi-event correlation (impossible travel, kill chain); Lambda for simple single-event rules (GuardDuty finding → alert)

Decision 3: OpenSearch Hot vs. UltraWarm Storage for SIEM Events

Factor Hot Storage (EBS-backed) UltraWarm (S3-backed)
Query latency < 100ms 1-5 seconds
Cost 100% baseline 13% of hot storage cost
SOC workflow impact Real-time search during investigation Acceptable for historical analysis
Recommendation Hot: last 30 days (active investigations); UltraWarm: 30-90 days (recent hunts); Security Lake Athena: 90 days+ (threat hunting)

Decision 4: Alert-First vs. Data-First SOC Workflow

Factor Alert-First (SOC waits for alerts) Data-First (SOC proactively hunts with Athena)
False negative risk High (miss threats with no rule) Low (analyst can query any hypothesis)
Analyst skill requirement Low (follow runbook) High (SQL, threat intelligence knowledge)
Resource efficiency High (only work when alerted) Low (analysts spend time with no result)
Recommendation Hybrid: Tier-1 SOC works alert queue; Tier-2 threat hunters use Athena proactively. Separate team structures with different tooling access

Common Mistakes to Avoid

  1. Proposing only CloudWatch Logs as the SIEM store. CloudWatch is good for operational logs but expensive at 500 GB/day for 2 years (~$300K+/year in CloudWatch Logs Insights queries). Security Lake (S3 Parquet + Athena) is 10-50× cheaper for long-term log analytics. Candidates who describe "ship everything to CloudWatch" without mentioning Security Lake are not thinking about cost or scale.
  2. Not explaining OCSF normalization. Saying "collect all logs in S3" without describing how you query them together is incomplete. Without OCSF normalization, VPC Flow Logs and Palo Alto firewall logs have completely different schemas — you can't write a single Athena query that spans both. OCSF solves this: every source maps to the same network_activity class with the same src_endpoint.ip field. Candidates who skip normalization demonstrate lack of production SIEM experience.
  3. Forgetting Detection-as-Code (rules in version control). In large organizations, detection rules are code: they live in Git, have tests (unit tests on sample events), go through CI/CD with approval workflows, and are deployed atomically. Saying "I'll write Athena rules in the console" without mentioning version control shows lack of operational maturity.
  4. Not addressing alert fatigue. Designing a SIEM that generates 10,000 alerts/day with no deduplication, suppression, or prioritization makes the SOC non-functional. Candidates must mention: alert deduplication windows, severity scoring, ML-based noise reduction (GuardDuty anomaly scoring, Detective behavior baselines), and alert SLA targets per tier.

Likely Follow-up Questions

Q1: How does Amazon Security Lake handle multi-account log aggregation without manual setup in each account?
Security Lake uses AWS Organizations integration. When you enable Security Lake in the delegated admin account (typically the security/log archive account): (1) It automatically enrolls all member accounts via the Organizations API. (2) For each member account, Security Lake creates a CloudTrail, VPC Flow Logs, and Route53 configuration that forwards to a central S3 bucket in the log archive account (via S3 Cross-Account replication or direct delivery). (3) No need to log into each member account individually. (4) New accounts added to the Organization are automatically enrolled based on the root OU policy. The log archive account's Security Lake S3 bucket contains all logs from all 500 accounts in a single queryable location.

Q2: How do you ensure Athena queries are cost-controlled with 2 years of data?
Cost controls: (1) Athena Workgroups: create a soc-analysts workgroup with PerQueryDataScannedLimit = 1TB (query fails if it would scan > 1 TB — forces analysts to add partition filters). (2) Partition pruning: always require WHERE year=? AND month=? AND day=? in analyst queries (dashboards enforce this). A 1-day query scans ~125 GB (uncompressed); with Parquet compression (4:1), actual scan is ~31 GB = $0.16/query. (3) Query result caching: Athena result reuse caches identical queries for up to 7 days (same query by different analysts uses cached result — no additional cost). (4) Query review: weekly Athena cost report (who ran what, how much scanned) → identify inefficient queries → add indexes or create summary tables. (5) Resource-intensive queries: require approval (Tier-2 hunter submits hunting request → SIEM admin reviews before executing > 1 TB queries).

Q3: How would you implement MITRE ATT&CK-based detection coverage reporting?
Coverage report implementation: (1) Each Flink detection rule has a metadata tag: rule.mitre_technique = "T1110" and rule.mitre_tactic = "TA0006". (2) A DynamoDB table stores all detection rules: {rule_id, rule_name, mitre_technique, mitre_tactic, status: active/disabled, last_triggered, false_positive_rate}. (3) A weekly Lambda job: queries the rules DynamoDB table, maps each rule to its ATT&CK technique. Queries the MITRE ATT&CK TAXII server for the full technique list. Generates a coverage matrix: which techniques have rules vs. which are uncovered. (4) OpenSearch dashboard: ATT&CK navigator-style heatmap — green (covered + triggered), yellow (covered but never triggered — rule may be broken), red (not covered). (5) Coverage target: track percentage of Tier-1 techniques covered per quarter. Security roadmap: prioritize rule development for red (uncovered) techniques that appear in GuardDuty findings.

Q4: How do you handle log sources that can't push to Kinesis (e.g., legacy syslog)?
For legacy syslog sources: (1) On-premises: deploy CloudWatch Agent on syslog relay server. Configure /var/log/messages and /var/log/syslog as input. CloudWatch Agent sends to CloudWatch Logs (encrypted, authenticated). (2) CloudWatch Logs → Kinesis Firehose subscription filter: any new log event matching pattern → Firehose → Lambda transformer → OCSF JSON → Security Lake custom source API. (3) For network devices that can only send UDP syslog to a receiver: deploy a syslog collector EC2 (rsyslog/syslog-ng) in a private subnet. rsyslog: output to Kinesis Firehose via HTTPS (using rsyslog's HTTPS output module). (4) Custom format parsing: Lambda transformer uses regex-based parsing for legacy syslog formats (or Grok patterns for CEF/LEEF). Mapping configs stored in S3 (update without Lambda code deployment). (5) DLQ: unparseable events (unknown source format) → SQS DLQ → SIEM engineer reviews → adds new parsing rule.

Q5: How do you detect an insider threat from a SOC analyst using their privileged SIEM access?
SOC insider threat controls: (1) Athena query logging: every StartQueryExecution is in CloudTrail with the IAM principal and the SQL text. If an analyst runs SELECT * FROM security_lake WHERE year >= 2022 (full scan) — anomaly alert (too broad, too much data). (2) Analyst behavior baseline (Amazon Detective): add SOC analyst IAM roles to Detective monitoring. Detective shows typical query patterns per analyst. Deviation from baseline (new query types, unusual hours, bulk S3 downloads) → finding. (3) S3 data event logging: if analyst role calls GetObject directly on Security Lake bucket (bypassing Athena — possible bulk download attempt) → CloudTrail alert. (4) Workgroup limits: Athena results go to a dedicated results bucket. Analyst has no permission to move results to external locations. S3 endpoint policy: results bucket allows only GetObject to analyst role; no PutObject to other buckets. (5) Separation of duties: SOC analysts can read logs but cannot modify detection rules, disable alerts, or access production AWS accounts. These are separate roles with separate MFA requirements.

Q6: How would you integrate external threat intelligence (IOC feeds) into the SIEM?
Threat intelligence integration: (1) IOC ingestion Lambda: runs every 15 minutes, pulls from commercial threat intel feeds (Recorded Future, CrowdStrike Falcon X) and free feeds (CISA KEV, AlienVault OTX, MISP community). Feeds provide: malicious IPs, domains, file hashes, CVE indicators. (2) DynamoDB IOC table: {indicator_type: ip/domain/hash, value, source, confidence_score, first_seen, last_seen, ttl}. DynamoDB TTL: IOCs expire after 90 days unless refreshed. (3) Route 53 DNS Firewall: sync malicious domains from DynamoDB → Route 53 DNS Firewall custom block list (via Lambda → CreateFirewallDomainList API). New C2 domains blocked at DNS level within 15 minutes of feed update. (4) WAF IP sets: sync malicious IPs from DynamoDB → WAF IPSet (via Lambda → UpdateIPSet API). Attacker IPs blocked at CloudFront/ALB within 15 minutes. (5) Athena enrichment: join Security Lake events with DynamoDB IOC table (via Lambda pre-processing) → any Security Lake event with matching IP or domain tagged as threat_intel_match = true. OpenSearch alert: any matched event = CRITICAL severity, Tier-1 queue.

Q7: How does the SIEM detect a zero-day attack with no signature?
Zero-day detection strategy (behavior-based): (1) Amazon Detective behavior graphs: Detective establishes a 30-day rolling baseline for every IAM user, EC2 instance, and IP address. On day 31: an IAM user who has never called ec2:RunInstances suddenly launches 100 spot instances in an unusual region → Detective flags high-unusual behavior score. No signature needed — it's deviation from baseline. (2) GuardDuty ML anomaly detection: GuardDuty has built-in ML models for VPC Flow anomalies (new port access patterns), S3 access anomalies (new IAM principal accessing a bucket), and DNS anomalies (first-time C2 callback domain). These fire for zero-days if the behavior pattern is anomalous. (3) Flink rule: detect command-and-control (C2) beaconing (periodic, fixed-interval connection to external IP): SELECT src_ip, dst_ip, STDDEV(TIMESTAMPDIFF(SECOND, LAG(event_time) OVER ..., event_time)) as jitter FROM vpcflowlogs WHERE jitter < 5 — low jitter = automated beacon. (4) Acceptance: zero-days WILL get through some detection. Focus on reducing dwell time (mean time to detect via behavior analytics) rather than blocking all zero-days at the perimeter. Target MTTD < 24 hours via behavior-based detection.

Q8: How do you manage detection rules as code (rule lifecycle)?
Detection-as-Code lifecycle: (1) Rule development: SIEM engineers write Flink SQL detection rules in local environment. Test against sample events (unit tests with pytest-like framework; compare expected alert output vs. actual). (2) Version control: rules live in CodeCommit (or GitHub). PR review required: second engineer approves before merge to main. (3) CI/CD (CodePipeline): main branch push → CodeBuild (run unit tests + syntax validation) → deploy to staging SIEM (runs against 24h of historical Security Lake events) → compare alert output vs. expected (regression test). (4) Production promotion: manual approval gate (security lead signs off) → CodeDeploy pushes new Flink rule to production Kinesis Flink app. (5) Rule performance tracking: each rule logs to DynamoDB — trigger count, false positive rate (analyst marked FP), true positive rate (analyst escalated). Monthly review: disable rules with > 50% FP rate; tune thresholds. (6) Rollback: all Flink application versions kept in S3 (CodeDeploy artifact). If new rule causes alert storm, rollback to previous version in < 5 minutes.
