Q18 — Supply Chain Security for Software Delivery

The Ask

"Design a secure software supply chain for an enterprise delivering 50 microservices
using AWS CodePipeline. The platform must enforce code signing, container image
integrity, SBOM generation, dependency vulnerability scanning, provenance attestation,
and policy-as-code enforcement before any artifact reaches production. Address insider
threat, compromised build environment, and malicious dependency scenarios."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Are microservices built as container images (ECR) or Lambda deployment packages?
    Impact: ECR images: AWS Signer signs container images + ECR image signing verification policy (deny unsigned). Lambda packages: AWS Signer signs Lambda deployment packages (zip files). Different signing workflows and enforcement mechanisms for each.

  • Q: Is there an existing artifact registry (Artifactory, Nexus, npm private registry) or does the team only use public package repositories (npm, PyPI, Maven Central)?
    Impact: Existing Artifactory → AWS CodeArtifact can proxy from Artifactory (or replace it). If public registries only → CodeArtifact upstream proxy (pull packages through CodeArtifact, cache, scan). Direct pulls from public internet in builds → highest supply chain risk (no inspection layer).

  • Q: What languages/package managers are used across 50 microservices (npm, pip, Maven, Go modules)?
    Impact: SBOM generation tooling varies by language (Syft, cdxgen, or Inspector v2 support). Vulnerability matching database varies (npm: NSP, pip: OSV, Maven: NVD, Go: govulncheck). Inspector v2 supports all major package managers but coverage depth varies.

  • Q: What is the current build infrastructure — managed CodeBuild or self-hosted Jenkins?
    Impact: CodeBuild: AWS manages the build environment (fresh ephemeral instance per build — no persistent build agent compromise). Self-hosted Jenkins: persistent agents are a supply chain attack surface (poisoned agent cache). Migration recommendation: move to CodeBuild for ephemeral, auditable build environments.

  • Q: Is there a requirement for SLSA (Supply Chain Levels for Software Artifacts) framework compliance?
    Impact: SLSA Level 1: build provenance (who built it, when, from what source). SLSA Level 2: hosted build system (no self-hosted Jenkins). SLSA Level 3: hardened build (isolated environments, non-falsifiable provenance). SLSA Level 4: two-person review + hermetic builds. Target SLSA Level 3 for senior security roles-grade supply chain security.

  • Q: How quickly do vulnerabilities need to be patched after disclosure? What's the SLA for critical CVE remediation?
    Impact: If SLA = 24h for critical: need real-time vulnerability alerting (Inspector findings → PagerDuty P1). If SLA = 7 days: nightly scan + ticketing acceptable. The enforcement gate (block deployment until patched vs. alert only) depends on this SLA.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. All 50 microservices build to ECR container images.
  2. CodePipeline + CodeBuild (ephemeral, no persistent agents).
  3. EKS as the deployment target (Kubernetes admission control for policy enforcement).
  4. SLSA Level 3 compliance target.
  5. CodeArtifact as the dependency proxy for all package managers.

Core Architecture — Data Flow Narrative

[PIPELINE STAGE 1: SOURCE — CODE COMMIT CONTROLS]

Developer workstation → CodeCommit (or GitHub):
  Branch protection:
    Main branch: no direct push (PR required)
    PR: minimum 2 approvals + passing security checks before merge
    Signed commits required (GPG/SSH signed)

  Pre-commit hooks (developer's workstation):
    secret scanning: git-secrets / Gitleaks (block commit if AWS key, private key found in code)
    IaC lint: checkov (Terraform + CloudFormation rules)
    Dependency lock files: fail if package-lock.json / poetry.lock not committed

  AWS CodeCommit notifications:
    CodeCommit → EventBridge → Lambda → runs Gitleaks scan on new commits
    Alert if detected: SNS → Security Hub finding

[PIPELINE STAGE 2: BUILD — EPHEMERAL CODEBUILD]

CodeBuild build environment:
  Isolation:
    Fresh EC2 Nitro instance per build (no persistent state between builds)
    Build environment image: Amazon Linux 2023 (AWS managed, CIS-hardened)
    No network access to internet during build (VPC-only mode):
      Exception: CodeArtifact VPC endpoint + ECR VPC endpoint + S3 VPC endpoint
      All other outbound TCP: blocked (no package pulls from raw internet)

  CodeArtifact (dependency proxy):
    All package managers point to CodeArtifact domain:
      npm: registry = https://domain-account.d.codeartifact.region.amazonaws.com/npm/store/
      pip: index-url = https://domain-account.d.codeartifact.region.amazonaws.com/pypi/store/simple/
      Maven: repository = CodeArtifact endpoint

    CodeArtifact upstream proxying:
      npm public upstream: npmjs.com (CodeArtifact caches + scans packages)
      pip upstream: PyPI (CodeArtifact caches)

    CodeArtifact package origin controls:
      Block "dependency confusion" attacks:
        Setting: ALLOW_SPECIFIC_PACKAGE_ORIGINS
        Config: internal packages can ONLY come from internal namespace
        External packages can ONLY come from upstream proxy
        Prevents: attacker publishing typosquat package "my-company-internal-lib" on PyPI
                  → CodeArtifact rejects it (internal namespace can't come from upstream)

    CodeArtifact Upstream blocking:
      After package is cached: lock specific version (don't pull new version without approval)
      Package approval workflow: new package version → notification → manual approval → allowed in builds

  Build provenance (SLSA Level 3):
    CodeBuild: generates a signed provenance attestation for each build:
      provenance.json:
        { "builder": "arn:aws:codebuild:...",
          "buildId": "build-id",
          "sourceCommit": "git-sha",
          "sourceRepo": "codecommit-repo-arn",
          "buildStartTime": "ISO8601",
          "outputImage": "sha256:image-digest",
          "builderVersion": "codebuild-image-version" }

    Provenance signing: AWS Signer signs the provenance.json
    Provenance stored: S3 provenance bucket (immutable, Object Lock) + ECR image attestation (attached to image manifest)

[PIPELINE STAGE 3: SBOM GENERATION]

Amazon Inspector v2 (SBOM generation during build):
  Inspector scans the container image as it's built:
    Trigger: CodeBuild invokes Inspector StartCisScan + CreateFinding workflow
    Or: push to ECR staging repository → Inspector scans automatically

  SBOM output:
    Format: CycloneDX JSON (standard format, machine-readable)
    Content: all OS packages (rpm, deb) + application packages (pip, npm, maven) with versions
    Stored: S3 SBOM bucket (key: {image-sha256}/sbom.cdx.json)

  SBOM signing:
    AWS Signer signs the SBOM (attestation: "this SBOM was generated by Inspector from this image")
    Signature stored alongside SBOM in S3

  Vulnerability matching:
    Inspector: continuously re-evaluates stored SBOMs against new CVE/NVD data
    New CVE published → Inspector automatically generates a finding for all affected images
    Finding: { "imageSha": "sha256:...", "vulnerability": "CVE-2024-XXXX", "severity": "CRITICAL", "packageVersion": "openssl-3.0.7", "fixVersion": "3.0.8" }
    Critical finding → EventBridge → Lambda → PagerDuty P1 + Security Hub

[PIPELINE STAGE 4: CODE SIGNING — AWS SIGNER]

Container image signing:
  After build + Inspector scan passes:
  AWS Signer: signs ECR image (Notary v2 / Cosign-compatible OCI image signature)
    Command: aws signer start-signing-job --source-s3-key {image-tar} --destination-s3-key {sig-out}
    Or: Cosign with KMS backend: cosign sign --key awskms:///arn:aws:kms:...:key/signing-key {image}

  Signing key: KMS asymmetric key (ECC P-256 or RSA-3072)
    Key policy: only CodeBuild role can use the key for signing
    Key policy: audit role can verify (kms:Verify) but not sign

  ECR image signing:
    ECR stores OCI signature alongside image manifest
    Signature is cryptographically bound to the image digest (sha256:...)
    Cannot detach or reuse (each image has its own signature)

  ECR container image signing policy:
    ECR repository policy: require signature validation at pull time
    Kubernetes (EKS): admission webhook verifies signature before running container

Lambda deployment package signing:
  AWS Signer signing profile: CodeSigning-Lambda-Production
  CodeBuild: zip Lambda package → aws signer start-signing-job → signed zip
  Lambda function: code signing configuration { SigningProfileARN: "..." }
  Lambda: rejects deployment packages not signed by the approved profile
  Validation: "ENFORCE" policy (not just warn — actually blocks unsigned deploys)

[PIPELINE STAGE 5: POLICY ENFORCEMENT — EKS ADMISSION CONTROL]

Kyverno (policy-as-code engine, deployed in EKS):
  Kyverno ClusterPolicy: verify-image-signatures
    Rule: all container images MUST have a valid AWS Signer signature
    Verification key: Cosign public key from KMS signing key
    Effect: any deployment with unsigned image → REJECTED by admission webhook

  Kyverno ClusterPolicy: require-sbom-attestation
    Rule: image MUST have an attached SBOM attestation (signed CycloneDX)
    Check: if SBOM attestation missing → REJECTED

  Kyverno ClusterPolicy: block-critical-cve
    Rule: image with CRITICAL CVE (CVSS >= 9.0) that has an available fix → REJECTED
    Data source: Inspector findings synced to DynamoDB (image-sha → vulnerability list)
    Kyverno policy: checks DynamoDB via external data provider

  Kyverno ClusterPolicy: require-provenance
    Rule: image MUST have a signed SLSA provenance attestation
    Provenance must reference: expected CodeBuild builder ARN (prevent builds from rogue infrastructure)

[PIPELINE STAGE 6: DEPLOYMENT — PRODUCTION]

CodePipeline manual approval gate:
  After all automated checks pass:
    Checklist (automated):
      [ ] All critical/high CVEs fixed (Inspector < CRITICAL)
      [ ] SBOM attached and signed
      [ ] Provenance attestation present and signed
      [ ] Image signature verified

  Manual approval:
    Engineering lead reviews automated checklist + any warnings
    Only on approval: CodeDeploy promotes image to production ECR → EKS deployment

EKS enforcement (runtime):
  ImagePolicyWebhook (Kyverno): re-validates on every pod start (not just deploy)
  Falco (runtime threat detection): detects unexpected processes in containers
  EKS Pod Security Standards: restricted (no privileged containers, no host network, read-only root filesystem)
  ECR scan on push: enabled for production registry (re-scan daily)

[MONITORING AND ALERTING]

Inspector findings → EventBridge → Lambda routing:
  CRITICAL severity: PagerDuty P1 + Jira ticket (4h fix SLA)
  HIGH severity: Jira ticket (24h fix SLA)
  MEDIUM/LOW: weekly digest report

Security Hub:
  Inspector findings forwarded to Security Hub
  Custom insight: "container images with unpatched critical CVEs in production"

SBOM vulnerability dashboard:
  Athena queries on S3 SBOM bucket → QuickSight dashboard:
    Total packages across all images
    Images by vulnerability count
    Top 10 most vulnerable packages across fleet

Dependency drift alert:
  Weekly Lambda: compare current SBOMs vs. SBOMs from 7 days ago
  New packages added: alert security team (potential unauthorized dependency)
  Packages removed: log for audit (in case removed to hide malicious dependency)

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

CodeBuild role (build-service-role):
  Permissions:
    codeartifact:GetAuthorizationToken, codeartifact:ReadFromRepository (read-only dependency pull)
    ecr:PutImage, ecr:InitiateLayerUpload (push built image to staging ECR)
    signer:StartSigningJob (sign the image after build)
    inspector2:StartCisScan (trigger SBOM scan)
    s3:PutObject (upload SBOM + provenance to S3)
  Cannot:
    ecr:PutImage on production ECR (only staging ECR — promotion happens separately)
    kms:Decrypt (build service cannot decrypt production secrets)

CodeDeploy promotion role:
  Permissions:
    ecr:CopyImage (from staging ECR to production ECR)
    kms:Verify (verify signing certificate)
  Cannot:
    signer:StartSigningJob (cannot re-sign for production — signature must come from build)

Separation of duties:
  Developer: merge code → cannot approve their own PR
  Build service: builds + signs → cannot deploy to production
  Deployment service: deploys → cannot modify build artifacts

2. Encryption

Signing key (KMS asymmetric ECC P-256):
  Private key: never leaves KMS HSM
  Public key: distributed to Kyverno (as verification key), Inspector (for attestation verification)
  Key rotation: annual (manual for asymmetric keys — new key version; update Kyverno policy)

SBOM bucket: SSE-KMS
Provenance bucket: SSE-KMS + Object Lock (immutable provenance — cannot rewrite build history)
ECR: encrypted at rest (AES-256, AWS managed)

3. Network Isolation

CodeBuild VPC mode:
  No outbound internet during build
  All package pulls go through CodeArtifact VPC endpoint
  All image pushes go through ECR VPC endpoint
  Effect: attacker cannot use build environment to exfiltrate data or download malware

EKS nodes:
  Private subnets (no public IP)
  ECR pull: through ECR VPC endpoint (no internet egress for image pulls)

4. Logging and Audit Trail

CodeBuild build logs: CloudWatch Logs (every command, output, exit code)
AWS Signer: every signing job logged in CloudTrail (who triggered, which key, which artifact)
ECR: image push/pull events in CloudTrail
CodeArtifact: every package download in CloudTrail (which build, which package, which version)
Kyverno audit mode: every admission webhook decision logged (allowed/denied + policy that triggered)

5. Least Privilege and Blast Radius

Per-microservice signing keys:
  Option: one KMS signing key per microservice (blast radius = one service)
  Reality: 50 keys × $1/month = $50/month — acceptable
  Benefit: if signing key for service-A is compromised, attacker can only sign service-A images
           service-B signing key is unaffected

  Key policy per service: only service-A's CodeBuild role can use service-A-signing-key

Build isolation:
  Separate CodeBuild project per microservice (different IAM roles, different VPC)
  Compromise of build environment for service-A: cannot access service-B's secrets/keys

6. Compliance Mapping

Requirement Control
SLSA Level 3 Hosted build (CodeBuild managed), signed provenance, isolated build environment
NIST SSDF (Secure Software Development Framework) Code review, SAST, SBOM, vulnerability management, code signing
FedRAMP SA-12 — Supply Chain Risk Dependency origin control (CodeArtifact), SBOM, vulnerability scanning
PCI-DSS Req 6.3 — Security vulnerabilities Inspector v2 continuous CVE scanning; Critical patching SLA enforcement

Scale and Resilience (Phase 4 — 10 min)

Build parallelism:

  • CodePipeline supports parallel stages. 50 microservices: each has own CodePipeline. No shared build bottleneck.
  • CodeBuild concurrency: default 60 concurrent builds per account. Request increase to 200 (for large monorepo scenarios).

CodeArtifact caching:

  • Packages cached in CodeArtifact domain after first pull. Subsequent builds: no upstream call. Cache hit rate > 95% after steady-state.
  • Package availability: if PyPI/npm is down, builds still succeed (CodeArtifact has cached packages).

Inspector v2 scale:

  • Inspector continuously re-evaluates all ECR images as new CVEs are published. No build-triggered scan needed for ongoing vulnerability monitoring.
  • Inspector supports 1,000s of ECR repositories per account.

Trade-off Analysis — The Hard Choices

Decision 1: Enforce Blocking vs. Audit-Only for Unsigned Images in EKS

Factor Block (ENFORCE mode) Audit Only (WARN mode)
Security guarantee Unsigned images CANNOT run Unsigned images run but are logged
Risk of deployment failure High initially (legitimate images may lack signatures) None (builds continue)
Migration path Requires all 50 services to be signed before enforcement Can adopt gradually
Recommendation Start with audit mode for 30 days (identify all unsigned images); then switch to enforce mode with clear rollback plan. Never ship without a path to enforce.

Decision 2: One KMS Signing Key vs. Per-Service Signing Key

Factor Single Signing Key Per-Service Key
Blast radius All 50 services affected if key compromised Only 1 service affected
Key management cost Minimal ($1/month) $50/month
Operational overhead Simple 50 key policies to manage
Recommendation Per-service key for regulated microservices (payment, auth); shared key per environment (dev-key, staging-key, prod-key) for less critical services

Decision 3: CodeArtifact vs. Artifactory for Internal Artifact Proxy

Factor AWS CodeArtifact JFrog Artifactory
AWS integration Native (IAM auth, VPC endpoint, CodeBuild) Requires configuration
Package managers npm, pip, Maven, NuGet, Gradle, Swift All + Docker, Helm
Vulnerability scanning Limited (delegates to Inspector) Xray integrated
Cost Pay-per-package stored + request $2K-$10K+/month license
Recommendation CodeArtifact for AWS-native shops (no Docker images in Artifactory needed — use ECR); Artifactory for polyglot environments with existing investment

Decision 4: Inspector SBOM Generation vs. Third-Party SBOM Tool (Syft, cdxgen)

Factor Inspector v2 (AWS native) Syft / cdxgen (open source)
Continuous re-evaluation Yes (new CVEs auto-trigger findings) No (static generation at build time)
Format CycloneDX JSON, SPDX CycloneDX, SPDX, multiple
ECR integration Native (auto-scan on push) Manual (run in CI/CD step)
Cost $0.09 per ECR image/month + container scan Free (open source)
Recommendation Inspector v2 for production (continuous re-evaluation is critical — a CVE published 6 months after release still triggers a finding); Syft for local development and offline environments

Common Mistakes to Avoid

  1. Signing without enforcement. Many candidates describe AWS Signer and ECR image signing but forget the enforcement gate. Signing an image means nothing if Kubernetes doesn't verify the signature before running it. The enforcement chain is: AWS Signer signs → ECR stores OCI signature → Kyverno admission webhook verifies before pod admission. Without the Kyverno (or OPA Gatekeeper) policy, the signature is decorative.
  2. Not addressing dependency confusion. Dependency confusion is a critical supply chain attack vector (Alex Birsan's 2021 research). CodeArtifact package origin controls are the specific AWS mitigation. Candidates who say "use CodeArtifact" without explaining WHY (package origin control, not just caching) miss the key security property.
  3. Static SBOM vs. continuous re-evaluation. Generating a CycloneDX SBOM at build time is good, but its value rapidly decays (new CVEs are published daily). Inspector v2's continuous re-evaluation means you don't need to rebuild images to know if a previously-safe image is now vulnerable. This is the key differentiator vs. just running syft in CI/CD.
  4. Forgetting build environment security. Most candidates focus on the artifact (image) but not the build environment (CodeBuild). If a build server is compromised, the attacker can inject code BEFORE signing — making the signature meaningless. Ephemeral CodeBuild instances (fresh per build), VPC-only mode (no internet for C2 callbacks), and pinned build image digests address this.

Likely Follow-up Questions

Q1: How do you prevent a developer from using their personal AWS account to sign a malicious image with a valid key and push it to production ECR?
Defense-in-depth: (1) ECR signing policy: only specific signing profiles (signing keys) are trusted. Developer personal accounts don't have access to the production signing profile KMS key (key policy restricts to CodeBuild role ARN only). (2) ECR repository policy: only the CodeDeploy role can PutImage to production ECR (not developer IAM users). Developers can push to sandbox ECR only. (3) Kyverno provenance check: image must have a provenance attestation signed by CodeBuild (builder ARN = arn:aws:codebuild:...). A developer-signed image from a personal account would fail the provenance check (builder ARN is wrong). (4) CloudTrail: ECR PutImage to production from any principal that isn't CodeDeploy role → immediate alert.

Q2: How do you handle a zero-day in a base image (e.g., Ubuntu 22.04) that affects all 50 microservices?
Base image zero-day response: (1) Inspector v2 detects the CVE across all 50 ECR images within hours of CVE publication (continuous re-evaluation). EventBridge rule: Inspector CRITICAL finding → Lambda → create Jira tickets for each affected service. (2) Base image update strategy: all 50 Dockerfiles inherit from a common internal base image (company/ubuntu-22.04:latest). Only the internal base image needs updating. (3) Automated base image rebuild: CodePipeline trigger: Inspector finding for base image → triggers base image rebuild pipeline (pull latest Ubuntu 22.04 from Docker Hub → patch → push to internal ECR). (4) Downstream rebuild: after new base image is pushed, EventBridge: ECR new image event for base image → triggers CodePipeline for all 50 downstream services (parallel execution). All 50 services rebuild + re-sign + redeploy within the base image SLA. (5) Production deployment gate: Kyverno blocks images with CRITICAL unfixed CVEs → production cannot run vulnerable images after SLA expires.

Q3: What is SLSA provenance and why is it important?
SLSA (Supply Chain Levels for Software Artifacts) provenance is a signed, machine-verifiable statement about how an artifact was built: what source commit was used, which build system built it, what the build parameters were, and the output digest. It matters because: (1) Without provenance, you can't verify that a signed image was actually built from the commit you think it was. An attacker could replace the source, rebuild, and re-sign. (2) With SLSA Level 3 provenance (from a hosted build like CodeBuild): the provenance is signed by the build SYSTEM (not the developer), and the builder identity is hardware-attested. You can verify that image sha256:abc was built from commit deadbeef in CodeBuild project arn:aws:codebuild:... at a specific time. (3) Kyverno policy: check provenance attestation and verify the builder field matches an expected CodeBuild project ARN (not a rogue build environment). Any image built outside the approved pipeline fails this check.

Q4: How do you enforce that no hardcoded secrets make it into the container image?
Multi-layer secret detection: (1) Pre-commit hook (developer): Gitleaks scans changed files before git commit. Catches: AWS access keys, private keys, API tokens. Developer never commits the secret. (2) CodeBuild step: Gitleaks runs on the full source checkout as the first build step. Build fails if any secret pattern found in source files. (3) Inspector v2: container image scanning detects exposed credentials in image filesystem (file-level secret scanning inside the container image layers). (4) Dockerfile enforcement: Kyverno policy: Dockerfile must NOT contain ENV AWS_SECRET_ACCESS_KEY or ARG variables for secrets. Use Secrets Manager at runtime (not build time). (5) ECR image layer inspection: Lambda post-build check: inspect each image layer for secret patterns (using tools like trufflesecurity/trufflehog against the image tarball).

Q5: How would you implement a secure update mechanism for a published Lambda function?
Lambda secure update: (1) Lambda code signing configuration: Lambda function has a CodeSigning Config attached (SigningProfileArn: arn:aws:signer:...:signing-profile/Lambda-Production). This means Lambda REJECTS any deployment package not signed by that profile. (2) Build pipeline: CodeBuild builds the Lambda zip → AWS Signer signs the zip → signed zip pushed to S3 → Lambda UpdateFunctionCode from S3. (3) Lambda aliases and traffic shifting: new signed version deployed to $LATEST → CodeDeploy linear traffic shift (10% → 50% → 100% over 30 minutes). If CloudWatch alarm triggers during shift → automatic rollback to previous signed version. (4) Provenance: the same SLSA provenance approach (what commit produced this Lambda zip) is attached as S3 object metadata and verified before promotion. (5) Environment variables (secrets): never store secrets in Lambda environment variables. Use Secrets Manager extension (Lambda extension pulls secret at startup, caches in memory). Kyverno equivalent for Lambda: SCP denying lambda:CreateFunction if Environment.Variables contains patterns matching SECRET, KEY, TOKEN (limited but better than nothing).

Q6: How do you detect a malicious package that has already been cached in CodeArtifact?
Retrospective scanning for malicious cached packages: (1) Inspector v2: continuously scans all packages in CodeArtifact repositories for known CVEs and malware indicators. New threat intelligence → re-evaluation of cached packages. (2) CodeArtifact package integrity: when a package is first cached from upstream, its SHA-256 hash is stored. Periodic Lambda verification: re-hash cached packages and compare to stored hash (detect in-place tampering of cached packages). (3) SBOM comparison (weekly): run syft against all cached packages → compare current SBOM to the one from last week → any new transitive dependencies added by a malicious package update → alert. (4) Package behavior analysis: new packages in CodeArtifact → Lambda triggers pip-audit / npm audit → any package with known suspicious behaviors (post-install scripts that call home, network access in setup.py) → quarantine flag. (5) Emergency remediation: if malicious package found: CodeArtifact: DeletePackage API → removes from cache → all subsequent builds fail (forcing immediate acknowledgment and rebuild). Identify all previous builds that included the package (CodeArtifact download logs in CloudTrail) → trigger forced rebuild + re-deploy of affected services.

Q7: How do you manage SBOM for open source compliance (license scanning)?
Open-source license compliance via SBOM: (1) CycloneDX SBOM includes license information for each package (e.g., { "name": "react", "version": "18.2", "licenses": [{"license": {"id": "MIT"}}] }). (2) License policy engine: Lambda post-build step reads the CycloneDX SBOM, checks each package license against a policy: ALLOWED: MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause; WARN: LGPL-2.1 (may require source disclosure); BLOCK: GPL-2.0 (viral — requires releasing your source if distributed). (3) Policy enforcement: if BLOCK license found → CodeBuild fails with message: "GPL-2.0 package detected: replace with MIT-licensed alternative". (4) License inventory: all SBOMs stored in S3 → Athena query: list all unique licenses used across all 50 services → quarterly legal review export. (5) New license alert: if a package update changes its license (e.g., faker changed to non-commercial) → SBOM comparison Lambda detects license change → alert to engineering lead + legal.

Q8: How does container image layer caching interact with supply chain security?
Docker layer caching security implications: (1) Layer caching in CodeBuild: FROM ubuntu:22.04 is cached in the CodeBuild local Docker cache across builds. Risk: the cached layer may not include the latest Ubuntu security patches. Mitigation: docker pull ubuntu:22.04 at the start of every build (don't use cache for base image). (2) Base image pinning: instead of FROM ubuntu:22.04 (tag), use FROM ubuntu@sha256:abc... (digest). This guarantees the same base layer every time — but also means you must explicitly bump the digest to get security updates. Use a Renovate/Dependabot-equivalent tool to auto-PR base image digest updates when new patched digests are published. (3) ECR layer deduplication: ECR deduplicates layers across images. This means image push is faster (only new layers uploaded), but it also means that deleting one image doesn't remove shared layers. If a shared layer is found to contain malware, you need to delete ALL images that share it (use Inspector to identify affected images by layer digest). (4) Build cache poisoning: an attacker who can modify a cached layer in CodeBuild (e.g., through a previous build) could inject malicious content. Ephemeral CodeBuild environments (fresh EC2 per build) prevent this — there is no persistent Docker layer cache between builds.
