Q19 — DNS Security and Threat Intelligence Platform

The Ask

"Design a DNS security platform for a 200-account AWS Organization that prevents DNS-based
attacks (C2 callbacks, DNS exfiltration, domain generation algorithm traffic), integrates
external threat intelligence feeds, provides split-horizon DNS for internal vs. external
resolution, and enables real-time detection and automated response. Include DNSSEC for
your public-facing domains."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What is the current DNS architecture — centralized Route 53 Resolver, individual VPC resolvers, or on-premises DNS servers that forward to AWS?
    Impact: Centralized architecture (DNS firewall in shared-services VPC, hub-and-spoke with Transit Gateway) is the most effective — one DNS firewall policy covers all spoke VPCs. Distributed (individual VPC resolvers per account) requires DNS firewall policies per VPC (200 accounts × multiple VPCs = hundreds of policies to manage). Hybrid (on-prem forwards to Route 53): DNS Firewall only catches queries that Route 53 Resolver processes — on-prem DNS queries bypass it unless forwarded.

  • Q: Are there legitimate uses of DNS tunneling in your environment (VPN-over-DNS, DNS-based load balancing protocols)?
    Impact: If VPN-over-DNS or legitimate high-volume DNS is used (e.g., DNS-based service discovery in Consul), the entropy/subdomain-length detection rules need to whitelist those specific domain patterns to avoid false positives. This affects Flink/Lambda detection rule design.

  • Q: What threat intelligence sources does the organization currently subscribe to (commercial: Recorded Future, CrowdStrike; free: CISA, AlienVault OTX)?
    Impact: Commercial feeds provide higher-quality IOCs (lower false positive rate, faster time-to-indicator). Free feeds (CISA KEV, Emerging Threats): delayed but free. Architecture must support both. Integration frequency: commercial feeds = every 15 minutes; free feeds = every 1 hour.

  • Q: Are there internal private domains (corp.company.com, db.internal) that must ONLY resolve within the AWS network (split-horizon)?
    Impact: Split-horizon DNS: Route 53 private hosted zones for internal domains (visible only within VPC). Public hosted zones for external domains (public internet resolution). Prevents internal domain names from leaking to public DNS (an attacker can enumerate internal services via public DNS if split-horizon is not implemented).

  • Q: What is the automated response capability — can the platform automatically block domains (DNS firewall block response), or does it require human approval?
    Impact: For high-confidence malware/C2 domains (confidence ≥ 90% from commercial threat intel): auto-block via Route 53 DNS Firewall custom domain list (no human approval). For medium-confidence (50-89%): alert + auto-add to watchlist (log queries but don't block). For low-confidence: alert only. Automated blocking reduces response time from hours to < 15 minutes.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Hub-and-spoke DNS architecture: centralized Route 53 Resolver in shared-services account.
  2. Route 53 DNS Firewall for blocking/alerting on malicious domains.
  3. Kinesis Data Streams + Lambda for DNS query analysis and threat hunting.
  4. DNSSEC enabled on all public hosted zones.
  5. Automated IOC synchronization from threat intelligence feeds to DNS Firewall block lists.

Core Architecture — Data Flow Narrative

[DNS RESOLUTION ARCHITECTURE — SPLIT HORIZON]

External DNS (public internet):
  Route 53 Public Hosted Zone: company.com
    A/CNAME records for public services (www.company.com → CloudFront)
    DNSSEC enabled: KSK (Key Signing Key) + ZSK (Zone Signing Key) for company.com
    DNSSEC chain of trust: Route 53 DNSSEC → parent zone (registry) → ICANN root
    Effect: resolvers that support DNSSEC validation can verify responses haven't been tampered with

    KSK management:
      Route 53 generates and stores KSK in KMS (AWS-managed asymmetric key)
      KSK rotation: Route 60 days (short rotation = smaller window of KSK compromise)
      DS record: Route 53 provides DS (Delegation Signer) record → registrar publishes to TLD zone

    DNSSEC response:
      Queries for www.company.com → Route 53 signs response with ZSK
      RRSIG (Resource Record Signature) included in response
      Resolver verifies: RRSIG validates against ZSK public key

Internal DNS (VPC-private):
  Route 53 Private Hosted Zone: corp.company.com (VPC-associated)
    Resolves ONLY for VPCs associated with this hosted zone
    Records: db.corp.company.com → 10.0.1.50 (private RDS endpoint)
              api.corp.company.com → 10.0.2.100 (private ECS ALB)

    NOT visible to public internet:
      Private hosted zone does NOT respond to public queries
      External attacker cannot enumerate internal services via public DNS

  VPC DNS resolution (Route 53 Resolver):
    VPC setting: enableDnsHostnames = true, enableDnsSupport = true
    Default resolver: 169.254.169.253 (VPC+2 address)

[HUB-AND-SPOKE DNS ARCHITECTURE]

Shared Services Account (DNS hub):
  Route 53 Resolver Inbound Endpoint:
    2 ENIs in private subnets (2 AZs)
    IP: 10.10.0.10 and 10.10.0.11
    Purpose: on-premises → Route 53 (on-prem DNS servers forward to these IPs)

  Route 53 Resolver Outbound Endpoint:
    2 ENIs in private subnets (2 AZs)
    Forwarding rules:
      corp.company.com → on-premises DNS (10.1.0.53) [hybrid DNS]
      company.internal → Route 53 Private Hosted Zone (local resolution)
      * → Route 53 public resolver (all other queries to public DNS)

  Route 53 DNS Firewall (applied to shared-services VPC resolver):
    Rule groups (evaluated in priority order):

    Priority 1: BLOCK (high-confidence malicious domains)
      Rule group: malicious-domains-block
        List 1: AWS managed list - AWSManagedDomainsMalwareDomainList
        List 2: AWS managed list - AWSManagedDomainsAggregateThreatList
        List 3: Custom IOC list (from threat intel feeds, updated every 15 min)
        Action: BLOCK (return NXDOMAIN or custom IP → sinkhole)

    Priority 2: ALERT (suspicious — log and allow)
      Rule group: suspicious-domains-alert
        List 4: Newly-registered domains (< 30 days old, from threat intel feed)
        List 5: DGA (domain generation algorithm) pattern list (updated from detection)
        Action: ALERT (log the query — no blocking)

    Priority 3: ALLOW (explicit allowlist — bypass all blocks)
      Rule group: corporate-allowlist
        List 6: Approved business SaaS domains (salesforce.com, workday.com)
        Action: ALLOW (never block these domains even if they appear in threat lists)

    Priority 4: Default ALLOW
      All other queries: allow (DNS Firewall default = allow-only-blocked-domains)

  Transit Gateway (DNS query routing):
    Spoke VPCs forward DNS queries to shared-services VPC via Transit Gateway
    Route table: DNS query traffic (UDP/TCP 53) routes to shared-services VPC
    Centralization ensures: DNS Firewall in shared-services VPC covers ALL spoke VPCs

  Effect: workload in any of 200 spoke accounts → VPC DNS query → Transit Gateway
          → Shared-services Route 53 Resolver → DNS Firewall applied → response

Real-Time DNS Threat Detection (Lambda + EventBridge):

  Detection 1: DNS Exfiltration (high-entropy subdomains)
    Lambda trigger: Kinesis Data Streams (DNS logs)
    Rule: if Shannon entropy of subdomain > 3.5 bits/char AND subdomain length > 40
    Alert: potential DNS tunneling / data exfiltration
    False positive mitigation: whitelist domains with legitimate long subdomains (CloudFront)

    Entropy calculation:
      "aGVsbG8gd29ybGQ.exfil.badactor.com" → subdomain = "aGVsbG8gd29ybGQ" (base64 encoded)
      Shannon entropy: high (random-looking characters)

  Detection 2: DGA (Domain Generation Algorithm) traffic
    Rule: DNS query to a domain that matches DGA classification model
    Model: trained ML model (Lambda ML inference): features = n-gram analysis, consonant ratio, domain length
    Rule: DGA confidence > 80% → alert
    Auto-response: add domain to DNS Firewall custom block list (BLOCK immediately)

  Detection 3: Beacon detection (periodic DNS queries to same external domain)
    Rule: same EC2 instance queries the same external domain > 5 times in 5 minutes (fixed interval)
    Lambda: CloudWatch Metrics Insights query → if metric > threshold → alert + block

  Detection 4: C2 callback via legitimate domain (fast-flux)
    Rule: same domain name returns different IP addresses (TTL < 30 seconds, IP changes every query)
    Detection: Lambda compares DNS answer IPs for the same domain over 10 minutes
    Fast-flux indicator: 5+ different IPs for same domain in < 10 minutes → alert

[THREAT INTELLIGENCE INTEGRATION]

IOC ingestion pipeline:
  Commercial feeds (Recorded Future, CrowdStrike Falcon X):
    Lambda: polls REST API every 15 minutes
    Parses STIX 2.1 indicator objects (type: domain-name, pattern: like)
    Filters: malware: true, confidence: >= 70 (ignore low-confidence)

  Free feeds:
    CISA Known Malicious Domains: S3 sync (CISA posts to HTTPS endpoint, Lambda polls hourly)
    Emerging Threats DNS blocklist: Lambda polls hourly (text file format)

  DynamoDB IOC store:
    Table: dns-ioc-table (partition key: domain)
    Item: { domain, source, confidence, first_seen, last_seen, category: [malware, c2, phishing], ttl }
    DynamoDB TTL: expired IOCs removed after 90 days (unless refreshed by feed)

  DNS Firewall sync Lambda (every 15 min):
    Query DynamoDB: domains where last_seen > 15 min ago AND confidence >= 70
    Call: Route53Resolver.UpdateFirewallDomains (batch up to 1000 domains per API call)
    FirewallDomainList: custom-ioc-block-list (replaces previous batch)
    Log: how many domains added/removed to CloudWatch Metrics

  DNS Firewall limits:
    Domain list: up to 100,000 domains per list
    Lists per rule group: up to 25
    Total domains across all lists: ~2.5M (sufficient for most threat intel feeds)

[DNSSEC CONFIGURATION]

Route 53 DNSSEC setup:
  Step 1: Create KSK in Route 53 (backed by KMS asymmetric key)
    Key type: ECC P-256 (ECDSA, compact DNSSEC signatures)
    Key policy: Route 53 service can use key for signing (not any human)

  Step 2: Enable DNSSEC signing for company.com hosted zone
    Route 53: generates ZSK automatically (AWS manages ZSK internally)

  Step 3: Publish DS record to parent zone (registrar)
    Route 53 provides DS record value
    Registrar portal: add DS record to .com zone
    Chain of trust: ICANN root → .com zone (DS record) → company.com zone (KSK) → ZSK → records

  DNSSEC validation by resolvers:
    Recursive resolvers (Google 8.8.8.8, Cloudflare 1.1.1.1, Route 53 Resolver):
      Verify RRSIG signature → validate against ZSK
      Verify ZSK DS record → validate against KSK DS record in parent zone
      If validation fails: SERVFAIL response (resolver refuses to return unverified response)

  DNSSEC alarm:
    CloudWatch alarm: Route53/DNSSECInternalFailure → if KSK expires or signing fails → P1 alert
    KSK rotation reminder: EventBridge scheduled rule 30 days before KSK expiry

[SINKHOLE CONFIGURATION]

DNS Firewall BLOCK with custom IP (sinkhole):
  Instead of NXDOMAIN, return: 10.10.99.1 (sinkhole server)
  Benefit: client connects to sinkhole → connection is logged (with source IP, URL path, headers)
  vs. NXDOMAIN: client gets "domain not found" — no further connection attempt → less forensic data

  Sinkhole server (EC2 in security account):
    Accepts all TCP/UDP connections on all ports
    Logs: source IP, destination port, payload (first 1KB)
    CloudWatch Logs: all sinkhole connections → alert on first connection from new source IP
    Use case: malware executed on 10.0.1.55 → queries evil.com → BLOCKED (returns sinkhole IP)
              → malware connects to 10.10.99.1 → sinkhole logs → incident response triggered

5. Least Privilege and Blast Radius

DNS Firewall domain list limits:
  If IOC sync Lambda has a bug and adds all of company's internal domains to block list:
    Impact: all DNS in VPC breaks (blocked internal domains = application outage)
    Mitigation: allowlist rule (Priority 3) runs BEFORE block rules (Priority 1)
    Allowlist includes: all known internal domains, SaaS allowlist
    Result: allowlist always wins → internal domains never blocked even if in threat feed by mistake

Per-account DNS Firewall:
  Shared-services account controls centralized DNS Firewall for all spokes
  Member accounts: cannot modify Route 53 Resolver configuration (SCP denies)
  Blast radius of shared-services account compromise: all DNS control is there
  Mitigation: MFA required for all shared-services account access; SCPs prevent key changes

Scale and Resilience (Phase 4 — 10 min)

DNS Firewall throughput:

  • Route 53 Resolver handles millions of DNS queries per second across all VPCs. DNS Firewall rule evaluation adds < 1ms latency per query (negligible).
  • Custom domain list update (UpdateFirewallDomains): processes up to 100,000 domains — update completes in < 60 seconds.

High availability:

  • Route 53 Resolver inbound/outbound endpoints: 2 ENIs in 2 AZs (automatic AZ failover).
  • DNS Firewall rule association: applied at the VPC level — if one AZ's Resolver ENI fails, queries automatically route to the other AZ.

Query log scale:

  • 200 accounts × average 10 million queries/day = 2 billion queries/day.
  • DNS query log volume: ~500 bytes per log entry × 2B = ~1 TB/day.
  • Storage: S3 (Parquet compressed) = ~250 GB/day = ~90 TB/year.
  • Athena cost: partition by year/month/day/vpc-id — typical threat hunt query (1 day, 1 VPC) scans < 1 GB.

Trade-off Analysis — The Hard Choices

Decision 1: Route 53 DNS Firewall vs. Third-Party DNS Security (Umbrella, Cloudflare Gateway)

Factor Route 53 DNS Firewall Cisco Umbrella / Cloudflare Gateway
AWS integration Native (no agent, applies to Route 53 Resolver traffic) Requires DNS forwarding to external resolver
Data sovereignty DNS queries stay in AWS account (no third party sees your DNS) DNS queries sent to Umbrella/Cloudflare (data leaves your account)
Threat intelligence AWS managed lists + custom; requires external feed integration Best-in-class threat intel (Umbrella: world's largest DNS resolver dataset)
Cost Pay per query ($0.60/1M queries) + domain lists Per-seat or per-endpoint licensing ($5-30/user/month)
Recommendation Route 53 DNS Firewall for AWS-native + data privacy requirements; Umbrella/Cloudflare for superior threat intel quality and broader SaaS coverage

Decision 2: DNS Firewall BLOCK (NXDOMAIN) vs. BLOCK (Sinkhole IP)

Factor Block: NXDOMAIN Block: Sinkhole IP
Malware behavior Malware: "domain doesn't exist" → may try backup C2 domain Malware: "domain exists" → connects to sinkhole → captured
Forensic value Low (no connection attempt) High (connection shows malware is active on host)
Risk Malware may use DGA to find a working domain Slight resource cost for sinkhole server
Recommendation Sinkhole for known C2 domains (max forensic value); NXDOMAIN for phishing/malware distribution domains (deny the malware download before it executes)

Decision 3: Centralized DNS Firewall (Shared Services VPC) vs. Per-Account DNS Firewall

Factor Centralized (Hub-and-Spoke) Per-Account DNS Firewall
Management Single policy to update (1 rule group) 200 accounts × multiple VPCs = 400+ rule groups
Blast radius Shared services account = single point of failure Account compromise affects only that account's DNS
Latency Extra hop via Transit Gateway (< 1ms) Local resolution (minimal latency)
Recommendation Centralized for large organizations (200+ accounts): management overhead of per-account is prohibitive; Transit Gateway latency is negligible for DNS

Decision 4: Route 53 DNSSEC vs. Manual DNSSEC (Bind, Knot DNS)

Factor Route 53 Managed DNSSEC Self-managed DNSSEC (on-prem BIND)
Key management KMS-backed (AWS manages HSM security) Manual key generation (higher risk of key compromise)
ZSK rotation Automatic (Route 53 handles) Manual (missed rotation = expired ZSK = SERVFAIL for all resolvers)
KSK rotation Semi-automated (Route 53 wizard) Fully manual + registrar coordination required
Recommendation Route 53 DNSSEC (zero operational overhead for ZSK; KMS-backed KSK; alarm on signing failures). Self-managed only if already using on-prem BIND for primary zones.

Common Mistakes to Avoid

  1. Not distinguishing DNS Firewall from security groups or NACLs. DNS Firewall operates at the DNS resolution layer — it prevents the IP from ever being returned to the application, so the application never initiates a TCP connection to the malicious IP. A security group blocks the TCP connection AFTER the DNS resolution succeeds. DNS Firewall is earlier in the kill chain (disrupts C2 at the lookup phase), not a replacement for network-layer controls.
  2. Forgetting the allowlist priority in DNS Firewall. If you configure a block list without an allowlist, your IOC feed (which may contain false positives) will block legitimate business services. The allowlist rule MUST have the highest priority (lowest number) and be evaluated BEFORE block rules. Candidates who describe the architecture without mentioning rule priority ordering have not thought through the false positive scenario.
  3. Ignoring DNSSEC key rollover. Saying "enable DNSSEC" without addressing KSK rotation is incomplete. KSK expiry is the most common DNSSEC failure mode — if the KSK expires and isn't rotated, resolvers that perform DNSSEC validation return SERVFAIL for all queries to your domain (complete outage for DNSSEC-validating resolvers). Route 53 CloudWatch alarm for DNSSECInternalFailure is essential.
  4. Treating DNS security as only blocking. Advanced DNS threats (exfiltration) cannot be fully blocked by domain reputation lists because the attacker registers a new domain for each campaign. Detection (entropy analysis, beacon detection, DGA classification) and response (automated isolation, forensic logging) are as important as blocking. Candidates who focus only on DNS Firewall block lists are missing the detection dimension.

Likely Follow-up Questions

Q1: How does Route 53 DNS Firewall interact with AWS PrivateLink and VPC endpoints?
When a workload uses an AWS VPC endpoint (e.g., for S3, DynamoDB), DNS resolution for the service endpoint (e.g., s3.us-east-1.amazonaws.com) routes through Route 53 Resolver within the VPC. If the VPC endpoint is configured with private DNS (enabled by default), the endpoint hostname resolves to a private IP within the VPC. DNS Firewall applies to these queries too — if s3.amazonaws.com is accidentally added to a block list, S3 access from the VPC would fail. This is why the allowlist must include all *.amazonaws.com and *.aws.amazon.com domains at the highest priority. Additionally, Interface VPC endpoints create Route 53 private hosted zone entries for the service hostname — these entries bypass the outbound internet path entirely but still go through Route 53 Resolver and thus through DNS Firewall rules.

Q2: How do you detect DNS-based lateral movement inside the VPC?
Internal DNS lateral movement detection: (1) Route 53 Resolver Query Logs capture queries from any EC2 instance — including queries to internal hostnames (db.corp.company.com). If a workload that normally only queries its own service endpoint starts querying other services' DNS names (e.g., payment-service.corp.company.com), this is anomalous. (2) Detective baseline: Amazon Detective tracks DNS query patterns per EC2 instance. If web-server-1 queries db-server-3 for the first time, Detective flags unusual behavior (web server shouldn't be querying database servers directly by hostname). (3) Lambda rule on DNS logs: if src_ip = known web-tier subnet (10.0.1.0/24) AND queryName matches db-tier pattern (.db.corp.*), alert (web tier shouldn't query db tier by DNS — it should use a load balancer DNS name). (4) This doesn't replace network-level segmentation (security groups must also restrict actual connections), but DNS lateral movement detection gives early warning before a full connection attempt.

Q3: How would you handle a situation where a legitimate CDN (Akamai, CloudFront) domain appears in a threat intel feed due to malware hosted on a shared subdomain?
Domain fronting / shared CDN false positive handling: (1) Problem: d1234.cloudfront.net serves both a legitimate SaaS app AND (on a different subdomain) malware. The threat intel feed flags d1234.cloudfront.net. (2) Route 53 DNS Firewall: add cloudfront.net and akamaihd.net to the allowlist (Priority 3 — always allowed). Specific shared CDN hostnames should NEVER appear in the block list (too much collateral damage). (3) Correct approach: if the threat intel feed flags a CDN domain, the actual malicious content is on a specific path (URL), not at the DNS level. WAF is the right tool here — block the specific path (/evil-payload) at HTTP layer, not at DNS. (4) IOC feed curation: the IOC sync Lambda should have a "never-block" list: cloudfront.net, akamaihd.net, fastly.com, cloudflare.com, azureedge.net — these CDN domains should never be added to the DNS Firewall block list regardless of IOC feed content. Any IOC that matches these domains → log warning + skip (don't add to block list).

Q4: How do you implement DNS-based split tunneling for a corporate VPN?
Split-tunnel DNS design: (1) VPN clients (on-premises or remote workers) connect to corporate VPN (site-to-site via Direct Connect, or client VPN via AWS Client VPN). (2) Split tunnel: only corporate traffic routes through VPN; internet traffic goes directly from the endpoint. (3) DNS for split tunnel: VPN client DNS setting = Route 53 Resolver inbound endpoint IP (10.10.0.10). Corporate domains (corp.company.com, *.company.internal): Route 53 Resolver processes → resolves to private IPs → VPN client uses VPN tunnel for these connections. Non-corporate domains (google.com, github.com): Route 53 Resolver → public DNS → VPN client routes these directly to internet (not through VPN tunnel). (4) DNS Firewall still applies: even split-tunnel clients use the Route 53 Resolver inbound endpoint → DNS Firewall block rules apply → C2 domains blocked for remote workers. (5) Benefit vs. full tunnel: split tunnel reduces VPN bandwidth load (only corporate DNS traffic through VPN). Security tradeoff: internet-bound traffic from remote workers bypasses corporate DNS Firewall (accept this for users' personal browsing; deploy endpoint DNS protection for devices with CrowdStrike or similar).

Q5: How do you validate that DNSSEC is working correctly and hasn't been misconfigured?
DNSSEC validation testing: (1) CloudWatch alarm: Route 53 metric DNSSECInternalFailure → if this fires, DNSSEC signing is broken → immediate P1 (all DNSSEC-validating resolvers will return SERVFAIL for your domain). (2) External validator: dnsviz.net provides DNSSEC chain-of-trust visualization — shows KSK → ZSK → record validation path. Run this after any DNS change to verify chain is intact. (3) CLI test: dig +dnssec www.company.com @8.8.8.8 → response should include AD (Authenticated Data) flag → confirms Google's resolver validated DNSSEC. If AD flag is missing, DNSSEC chain is broken. (4) KSK expiry monitoring: Route 53 DNSSEC KSK has a validity period. EventBridge scheduled rule: 60 days before KSK expiry → alert to DNS team → initiate KSK rotation workflow (Route 53 console KSK rotation wizard). (5) DS record mismatch: if registrar's DS record doesn't match Route 53 KSK (happens if key was rotated but registrar DS record wasn't updated), all DNSSEC-validating resolvers will fail. Check: dig DS company.com @a.gtld-servers.net vs. dig DNSKEY company.com @ns-1234.awsdns-12.org — the KSK hash in DS record must match KSK hash in DNSKEY record.

Q6: How would you detect and respond to a DNS amplification attack originating from your infrastructure?
DNS amplification attack mitigation: (1) Detection: if your Route 53 Resolver is being used as an open resolver for DNS amplification (attacker sends queries with spoofed victim IP → your resolver sends large responses to victim), CloudWatch metrics show: sudden spike in outbound DNS response traffic volume, queries from external IPs to your Resolver. (2) Prevention: Route 53 Resolver inbound endpoints are NOT open resolvers — they only accept DNS queries from within the VPC (ENI IP range restrictions) or from Direct Connect/VPN IPs. Inbound endpoints are not reachable from the public internet (private subnets). So Route 53 Resolver cannot be used for DNS amplification by external attackers. (3) If you're running a public DNS resolver (not recommended): apply rate limiting on queries per source IP. AWS Shield Advanced: protects against volumetric attacks targeting Route 53 endpoints. (4) If amplification is originating FROM your EC2 instances (compromised instances participating in botnet DNS amplification): GuardDuty: UnauthorizedAccess:EC2/DNSAmplification finding → Step Functions playbook → isolate the EC2 instance → block UDP 53 outbound in security group.

Q7: How do you handle DNS for multi-region deployments with active-active failover?
Multi-region DNS design: (1) Route 53 latency-based routing or geolocation routing: api.company.com → us-east-1 ALB (for North America clients) or eu-west-1 ALB (for Europe clients). (2) Route 53 health checks: if us-east-1 ALB health check fails → Route 53 stops routing to us-east-1 → all traffic goes to eu-west-1 (automatic failover). Health check frequency: 10 seconds; failover threshold: 3 failures (30-second total failover time). (3) DNS TTL: set api.company.com TTL = 30 seconds (short TTL allows fast failover — DNS clients re-query within 30 seconds). Trade-off: short TTL = higher Route 53 query volume (cost). (4) Route 53 ARC (Application Recovery Controller): readiness checks validate that us-east-1 has sufficient capacity before receiving traffic. Routing controls: operator explicitly enables/disables routing to each region via ARC control panel. (5) DNSSEC in active-active: KSK is in a single Route 53 hosted zone (even if multi-region deployment). Route 53 hosted zone is global (not per-region) — DNSSEC configuration is centralized. Multi-region is a routing concern, not a DNSSEC concern.

Q8: How do you handle DNS for containers in EKS?
EKS DNS architecture: (1) CoreDNS: deployed as a Deployment in the kube-system namespace. Handles in-cluster DNS (service discovery: payment-service.default.svc.cluster.local). (2) CoreDNS forwarding: for external domains (not .cluster.local), CoreDNS forwards to Route 53 Resolver (VPC+2 address, e.g., 10.0.0.2). This means external DNS queries from pods go through Route 53 Resolver → DNS Firewall applies. (3) CoreDNS configuration (ConfigMap): forward . 10.0.0.2 — all non-cluster DNS queries go to VPC resolver. (4) DNS security in EKS: if a pod is compromised and attempts to query a C2 domain, the query goes CoreDNS → Route 53 Resolver → DNS Firewall (BLOCKED if domain is in block list). Pod source IP is the node IP in DNS logs (pods share node IP for outbound). To identify which pod made the query: cross-reference EKS container insights (pod IP → node IP) with DNS query timestamps. (5) Network policies (Calico/Cilium): can restrict which pods can query external DNS. Critical pods (payment service): restrict egress DNS to only *.amazonaws.com and *.company.com (Cilium DNS policy). Prevents compromised pod from querying attacker-controlled domains.
