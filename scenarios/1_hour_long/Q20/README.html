Q20 — Cloud Workload Protection Platform (CWPP) for EC2 and ECS

The Ask

"Design a Cloud Workload Protection Platform (CWPP) for an AWS environment running
2,000 EC2 instances and 500 ECS Fargate tasks across 50 accounts. The platform must
provide vulnerability management, runtime threat detection, OS hardening, patch
compliance, and behavioral anomaly detection. Design the agent deployment strategy,
data collection architecture, and response automation."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What OS types are in use (Amazon Linux 2, Amazon Linux 2023, Ubuntu, Windows Server)?
    Impact: Amazon Linux: AWS-native SSM agent pre-installed. Windows: SSM Agent available but additional configuration. Ubuntu: SSM agent must be installed via user data or AMI bake. Multi-OS: patch baselines differ by OS (Amazon Linux uses yum; Ubuntu uses apt; Windows uses WUA). Inspector v2 supports all major OS for vulnerability scanning.

  • Q: Are the EC2 instances stateless (launched from AMI, replaceable) or stateful (databases, legacy apps)?
    Impact: Stateless EC2: patch by replacing instances (terminate old → launch new patched AMI). Stateful EC2 (databases): in-place patching required (SSM Patch Manager maintenance window with pre-patching snapshot). This affects the patching strategy significantly.

  • Q: Are there compliance frameworks in scope (SOC 2, PCI-DSS CDE, HIPAA, FedRAMP)?
    Impact: SOC 2/PCI: require vulnerability scan evidence, patch within 30 days for HIGH, 7 days for CRITICAL. FedRAMP: NIST 800-53 controls including SI-2 (flaw remediation), SI-3 (malware protection), SI-4 (system monitoring). These requirements set the SLA targets for patch and remediation workflows.

  • Q: Is there an existing endpoint security tool in use (CrowdStrike, SentinelOne, Symantec)?
    Impact: If CrowdStrike is already deployed: CrowdStrike covers EDR/malware. AWS-native tools (GuardDuty EC2 Runtime, Inspector) complement CrowdStrike but don't replace it. Architecture would be hybrid (CrowdStrike agent + AWS native). If no existing EDR: evaluate CrowdStrike vs. GuardDuty Runtime Monitoring + Amazon Inspector (cost vs. coverage tradeoff).

  • Q: What is the instance lifecycle (long-running vs. auto-scaling groups)?
    Impact: Auto Scaling Groups: instances can be terminated at any time. Agent baking into AMI is mandatory (user data install too slow for scale-in/scale-out). Long-running instances: agents can be installed via SSM Distributor on existing instances. Lifecycle hooks: terminate hook allows quarantine of compromised instance before ASG terminates it.

  • Q: Are ECS Fargate tasks fully managed (no access to the underlying host)?
    Impact: ECS Fargate: no access to the underlying EC2 host (AWS manages it). Cannot install agents directly on Fargate tasks. Runtime protection for Fargate: GuardDuty ECS Runtime Monitoring (sidecars injected by ECS agent) — not a traditional agent. Inspector v2 scans the container image but not the running process. This changes the protection model for Fargate vs. EC2.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Amazon Linux 2023 primary (90% of fleet); Windows Server 2022 secondary.
  2. Mix of Auto Scaling Groups (stateless) and standalone instances (stateful databases).
  3. SOC 2 Type II and PCI-DSS in scope.
  4. No existing EDR — build AWS-native + CrowdStrike hybrid.
  5. Agents baked into golden AMI (not installed post-launch).

Core Architecture — Data Flow Narrative

[AGENT STRATEGY — GOLDEN AMI]

EC2 Image Builder pipeline:
  Base: amazon/al2023-ami-minimal (Amazon Linux 2023 minimal)

  Build components:
    Component 1: SSM Agent install/verify (pre-installed on AL2023; verify version)
    Component 2: AWS Systems Manager Inventory agent configuration
      /etc/amazon/ssm/ssm.json:
        { "Collect": { "Applications": "Enabled", "Files": "Enabled",
          "NetworkConfig": "Enabled", "Services": "Enabled", "WindowsRoles": "Disabled" } }
    Component 3: CrowdStrike Falcon sensor install
      Download from: S3 private bucket (not CrowdStrike cloud directly)
      falcon-sensor install --cid=<CID> --tags=environment:production,account:<accountId>
    Component 4: CIS Level 2 hardening (security-harden.sh):
      - Disable root login via SSH
      - Set SSH MaxAuthTries 3
      - Enable auditd (kernel audit framework)
      - Configure /etc/audit/rules.d/ for STIG-required audit rules
      - Set umask 027
      - Disable unnecessary services: bluetooth, cups, avahi-daemon
      - Enable firewalld (local firewall)
      - Set kernel parameters: net.ipv4.ip_forward=0 (no routing), net.ipv4.conf.all.rp_filter=1
    Component 5: IMDSv2 enforcement
      Configure: imds-token-required (tokens must be used — no v1 fallback)
    Component 6: CloudWatch Agent install + config (metrics + logs)

  AMI scanning (Inspector v2):
    EC2 Image Builder: after build, trigger Inspector v2 container image pipeline scan
    If CRITICAL vulnerabilities found: FAIL the AMI build
    If HIGH vulnerabilities found: WARN (notify security team, continue build with exception approval)

  AMI distribution:
    EC2 Image Builder distributes golden AMI to all 50 member accounts (cross-account sharing)
    AMI sharing policy: only approved AMI IDs can be used to launch instances (SCP enforcement)
    SCP: deny ec2:RunInstances if source AMI != latest approved golden AMI ARN list

[VULNERABILITY MANAGEMENT — INSPECTOR v2]

Amazon Inspector v2 (org-level, delegated admin in security account):
  Coverage:
    EC2: OS package vulnerabilities (CVE matching against OS package database)
         Network reachability findings (exposed ports, overly permissive security groups)
    ECS/ECR: container image vulnerabilities (matches Inspector EC2 findings with image layers)
    Lambda: function code vulnerabilities (language-specific package scanning)

  Finding types:
    os-package: CVE in OS package (e.g., OpenSSL, glibc) with installed version + fix version
    programming-language-package: CVE in application package (e.g., log4j, requests)
    network-reachability: port open + no security group restricting access

  Organization setup:
    Delegated admin: security account
    All 50 member accounts: auto-enrolled (Inspector enabled automatically via Organizations)
    No opt-out allowed (SCP denies inspector2:Disable for member accounts)

  Finding severity routing (EventBridge + Step Functions):
    CRITICAL (CVSS >= 9.0, fix available):
      SLA: 7 days
      Action: Jira P1 ticket (auto-created) + PagerDuty alert to responsible team
      If not patched in 7 days: escalate to director + block instance from receiving load balancer traffic
    HIGH (CVSS 7.0-8.9, fix available):
      SLA: 30 days
      Action: Jira P2 ticket
    MEDIUM/LOW:
      SLA: 90 days
      Action: weekly digest email to team

  Network reachability findings:
    Example: EC2 instance has port 22 (SSH) open to 0.0.0.0/0 → CRITICAL finding
    SCP enforcement: deny ec2:AuthorizeSecurityGroupIngress with CIDR 0.0.0.0/0 for port 22
    Inspector finding → EventBridge → Lambda → automatically REVOKE the permissive SG rule

  Suppression rules:
    Dev accounts: suppress LOW/MEDIUM findings (reduce noise)
    PCI accounts: suppress nothing (all findings must be tracked)

[OS PATCH MANAGEMENT — SSM PATCH MANAGER]

Patching strategy:

  Stateless EC2 (in Auto Scaling Groups):
    Patch by AMI replacement (not in-place patching):
    1. EC2 Image Builder: weekly scheduled pipeline (Sunday 2am)
    2. New golden AMI baked with latest patches
    3. Instance Refresh: ASG launches new instances from new AMI → health check passes → old instances terminated
    Timeline: AMI build (30 min) + instance refresh (2-4 hours for 2000 instances if max 20% at a time)

  Stateful EC2 (databases, legacy):
    SSM Patch Manager maintenance windows:
    - Production databases: Sunday 3am, 4-hour window
    - Dev: any day, 2-hour window

    Pre-patch: SSM Run Command → stop application → create EBS snapshot → verify snapshot complete
    Patch: SSM Patch Manager → AWS-RunPatchBaseline document
    Post-patch: verify application health → if unhealthy → SSM Run Command → reboot + retry

    Patch baselines:
      Amazon Linux 2023: AWS-AmazonLinux2023DefaultPatchBaseline
        Override: include CRITICAL security patches (auto-approved 0 days after release)
        Override: include HIGH security patches (auto-approved 3 days after release)
      Windows Server: AWS-WindowsServerDefaultPatchBaseline
        Override: include Critical/Security classification patches (7 days after release)

  Patch compliance reporting:
    SSM Inventory + Patch Manager: compliance data per instance (Compliant/NonCompliant)
    AWS Config rule: ec2-managedinstance-patch-compliance-status-check
    Security Hub: consolidates Config compliance findings
    Dashboard: QuickSight on SSM patch data (via Athena on S3 export)

  SCP enforcement:
    deny ssm:DeregisterManagedInstance (instances cannot opt out of SSM management)
    deny ec2:CreateSnapshot without tag "RequiresApproval=false" for non-scheduled snapshots

[RUNTIME THREAT DETECTION]

GuardDuty EC2 Runtime Monitoring:
  How it works:
    GuardDuty deploys a lightweight SSM-managed agent on EC2 (installed automatically via SSM)
    Agent: monitors system calls (syscalls), network connections, process spawning
    No kernel module required (user-space agent — safer than eBPF kernel module for prod)

  Detection capabilities:
    Malicious file execution: binary executed from /tmp, /dev/shm (write-then-execute pattern)
    Privilege escalation: unexpected use of sudo/su → root shell from non-admin process
    Reverse shell: unexpected outbound connection from a process that normally doesn't make network calls
    Container escape: process in container accessing host filesystem outside its mount namespace
    Credential theft: unexpected read of /proc/PID/mem or /etc/shadow

  GuardDuty ECS Runtime Monitoring (for Fargate):
    ECS task: GuardDuty agent runs as a sidecar container (injected automatically for enabled clusters)
    VPC endpoint for data collection (agent doesn't need internet access)
    Findings: same detection categories as EC2 (malicious file, reverse shell, etc.) but scoped to ECS task

  CrowdStrike Falcon (EDR layer):
    Agent: pre-installed in golden AMI
    Capabilities: behavioral IOC detection, exploit prevention, lateral movement detection
    Alert path: CrowdStrike → Falcon Connect webhook → Lambda → Security Hub (custom finding format)
    Integration: Security Hub custom product (CrowdStrike findings appear alongside GuardDuty findings)

  Alert correlation (Security Hub):
    GuardDuty Runtime finding (reverse shell from 10.0.1.55) + CrowdStrike finding (mimikatz process on 10.0.1.55)
    Same source IP + within 5 minutes = correlated incident → PagerDuty P1

[BEHAVIORAL ANOMALY DETECTION — AMAZON DETECTIVE]

Amazon Detective (behavior graphs):
  Linked to GuardDuty + CloudTrail + VPC Flow Logs

  Investigation workflow:
    GuardDuty finding: Reverse shell from ec2-10-0-1-55
    Detective: click "Investigate in Detective" from GuardDuty console
    Detective shows:
      - EC2 profile: what processes, users, and network connections are typical for this instance?
      - Baseline deviation: this instance normally only listens on port 443. Outbound connection on port 4444 is unusual.
      - Related entities: any IAM roles, API calls from this instance in the same timeframe?
      - Network graph: all connections from this IP in the last 24 hours

[EC2 HARDENING CONTROLS — DEFENSE-IN-DEPTH]

IMDSv2 enforcement:
  SCP: deny ec2:RunInstances if HttpTokens != "required"
  Existing instances: SSM Run Command fleet-wide: aws ec2 modify-instance-metadata-options --http-tokens required
  Purpose: prevents SSRF attacks from reaching IMDSv1 without token (SSRF cannot steal instance credentials)

No SSH keys:
  SCP: deny ec2:ImportKeyPair, deny ec2:CreateKeyPair, deny ec2:RunInstances with KeyName attribute
  Access: SSM Session Manager only (audited sessions, no port 22)
  Session log: CloudWatch Logs + S3 (30-day retention CWL, 7-year S3 Object Lock WORM for PCI)

Security Groups:
  No 0.0.0.0/0 ingress (SCP + Inspector network reachability finding auto-remediation)
  EC2 instance connect: disabled (SCP denies ec2-instance-connect:SendSSHPublicKey)
  Inbound from: only load balancer security group (ALB SG → app tier SG)

OS-level controls (built into AMI):
  auditd rules: monitor /etc/passwd writes, /etc/sudoers changes, setuid program execution
  auditd: forward audit logs to CloudWatch Logs (via CloudWatch Agent)
  AIDE (Advanced Intrusion Detection Environment): file integrity monitoring (daily scan)
    AIDE database: stored in S3 (initial baseline), compared daily
    Alert: if /usr/bin/*, /sbin/*, /lib/* change unexpectedly → GuardDuty custom finding via Lambda

[ECS FARGATE SPECIFIC CONTROLS]

No persistent host (Fargate — AWS manages the underlying EC2):
  Cannot install agents directly
  GuardDuty ECS Runtime: sidecar approach (no SSH, no exec)

Container hardening:
  Task definition: readonlyRootFilesystem: true
  Task definition: user: "1000" (non-root)
  Task definition: privileged: false
  Task definition: linuxParameters.capabilities.drop: ["ALL"] (drop all Linux capabilities)
  Task definition: linuxParameters.capabilities.add: ["NET_BIND_SERVICE"] (only if port < 1024 needed)

ECS Exec restrictions:
  SCP: deny ecs:ExecuteCommand in production (no interactive shell access to Fargate tasks)
  Exception: break-glass role with CloudTrail logging + approval workflow

Secrets (no env vars):
  Task definition: secrets[].valueFrom = "arn:aws:secretsmanager:..." (fetched at runtime)
  Not: environment[].value = "plaintext-secret" (Kyverno-like OPA Gatekeeper policy for ECS)

[PATCH COMPLIANCE AUTOMATION]

Compliance enforcement workflow:

  Day 0: Inspector finds CRITICAL CVE (CVE-2024-XXXX in openssl-3.0.7) on 150 EC2 instances

  Day 1:
    EventBridge: Inspector finding → SQS queue (batched by account)
    Lambda: creates Jira P1 tickets per team (group by owning team tag on EC2)
    PagerDuty: P1 alert to each team's on-call

  Day 7 (SLA deadline):
    Lambda: scheduled check → query Inspector API → which CRITICAL findings are still ACTIVE?
    For each ACTIVE finding past SLA:
      Step Functions: send escalation email to VP-Engineering
      Lambda: if ASG instance → trigger Instance Refresh (force replacement with patched AMI)
      Lambda: if stateful instance → create Jira escalation ticket + add to emergency maintenance window

  Day 14 (hard deadline for stateful instances):
    Lambda: if finding still ACTIVE → modify ALB target group → deregister the instance
    Result: instance stops receiving production traffic → team is forced to patch
    Alert: CloudWatch metric + PagerDuty P1 for the deregistered instance

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

SSM Patch Manager role (per member account):
  AmazonSSMManagedInstanceCore (AWS managed policy)
  s3:GetObject on patch artifacts bucket (patch binaries)
  Cannot: make API calls outside SSM (blast radius limited to OS patching)

Inspector delegated admin (security account):
  inspector2:ListFindings (read all findings from all accounts)
  Cannot: disable Inspector in member accounts (SCP prevents)

Break-glass ECS Exec:
  Separate break-glass role (requires MFA + approval)
  CloudTrail: every ecs:ExecuteCommand logged

2. Encryption

GuardDuty runtime agent data: encrypted in transit (HTTPS) to GuardDuty service endpoint
SSM Session Manager sessions: encrypted end-to-end (TLS) + optional KMS encryption for session data
SSM Patch Manager: downloads patches over HTTPS from AWS patch repository
CloudWatch Logs (audit logs, session logs): SSE-KMS
S3 (AIDE database, patch compliance exports): SSE-KMS

3. Network Isolation

GuardDuty Runtime Monitoring agent: uses SSM VPC endpoint (no internet)
Inspector v2 EC2 agent: uses Inspector VPC endpoint (no internet)
SSM agent: SSM VPC endpoints (ssm, ssmmessages, ec2messages) — no internet access required
CrowdStrike agent: needs internet access for cloud-based detection (or CrowdStrike proxy in VPC)
  Proxy option: CrowdStrike Falcon Proxy in shared-services VPC → all agents connect to proxy → proxy connects to CrowdStrike cloud
All agents in private subnets — no public IPs on EC2 instances

4. Logging and Audit Trail

SSM Session Manager: every session → CloudWatch Logs (session content encrypted) + S3
Inspector findings: CloudTrail (every CreateFinding, UpdateFinding)
Patch compliance: SSM Patch Manager → AWS Config → Security Hub
auditd: kernel audit events → CloudWatch Logs (forwarded by CWAgent) → Security Lake
CrowdStrike: detection events → Security Hub (custom product integration) → SIEM

5. Least Privilege and Blast Radius

Per-account Inspector:
  Findings visible to security account only (delegated admin)
  Member account teams: can view their own account's findings (via Inspector console with scoped IAM)

CrowdStrike CID isolation:
  Production CID separate from non-production CID
  Compromise of dev CrowdStrike API key: cannot see production endpoint detections

Runtime agent compromise:
  If GuardDuty/Inspector agent is somehow compromised:
    Agent runs as non-root (ssm-user or inspector-agent user)
    Agent cannot make IAM API calls (no instance profile permissions for agents)
    Blast radius: agent process — cannot escalate to full EC2 compromise via agent

Scale and Resilience (Phase 4 — 10 min)

SSM agent scale:

  • SSM supports up to 1,000,000 registered managed instances per account. 2,000 EC2 across 50 accounts = 40 average per account — trivial for SSM.
  • Patch Manager: maintenance windows support up to 1,000 targets. For 2,000 instances: use maintenance window with Rate Control = 20% (400 instances patched concurrently).

Inspector v2 scale:

  • Inspector continuously scans all EC2 instances without performance impact (scan is metadata-based, not agent-based for most CVE detection).
  • Network reachability findings: Inspector analyzes security group rules and route tables (no network traffic generated).

Golden AMI distribution:

  • EC2 Image Builder: cross-account sharing via Resource Share (AWS RAM). All 50 accounts automatically get the new AMI.
  • Instance Refresh: can run in parallel across all ASGs. Rate control: 20% of fleet at a time = 400 instances replaced simultaneously.
  • Total instance refresh time for 2,000 instances: ~4 hours (assuming 30-minute health check per batch).

Trade-off Analysis — The Hard Choices

Decision 1: GuardDuty Runtime Monitoring vs. CrowdStrike Falcon for Runtime Threat Detection

Factor GuardDuty Runtime Monitoring (AWS native) CrowdStrike Falcon
Deployment Auto via SSM (no AMI change required) Must be baked into AMI or installed via SSM Distributor
Threat coverage AWS-specific threats + common patterns Broader (ransomware, advanced APT, lateral movement, fileless attacks)
Cost Included in GuardDuty pricing (~$0.0025/vCPU/hour) Additional per-sensor licensing ($8-15/endpoint/month)
Tuning Limited (AWS-managed detection models) Extensive custom policy configuration
Recommendation Start with GuardDuty Runtime (free if GuardDuty already enabled); add CrowdStrike for PCI-CDE or high-value workloads requiring advanced EDR capabilities

Decision 2: In-Place Patching vs. AMI Replacement for EC2

Factor In-Place Patching (SSM Patch Manager) AMI Replacement (Instance Refresh)
Downtime Minimal (patch + reboot in maintenance window) Rolling replacement (no downtime with ASG)
Drift risk Instance state drifts from original AMI over time Each replacement starts fresh (no drift)
Stateful workloads Required (cannot replace a database) Not applicable
Speed Fast for individual instances Slow for full fleet (4+ hours for 2,000 instances)
Recommendation AMI replacement for stateless workloads (ASGs, ECS EC2 launch type); In-place patching for stateful (databases, legacy apps)

Decision 3: Inspector v2 vs. Qualys/Tenable for Vulnerability Management

Factor Amazon Inspector v2 Qualys/Tenable
AWS integration Native (no scanner network access, uses SSM metadata) Requires agent install or network scanner deployment
Cost ~$0.0015/instance/month per finding evaluation $5-15/IP/month
Coverage EC2, ECR, Lambda, ECS EC2, ECR, network (broader)
Network scan No network scanning (uses metadata) Active network scanning (can disrupt instances)
Recommendation Inspector v2 as the primary scanner (AWS-native, no performance impact); Qualys if compliance requires traditional agent-based scanner evidence or network vulnerability scanning

Decision 4: AIDE File Integrity Monitoring vs. AWS Systems Manager Change Notifications

Factor AIDE (OS-level FIM) SSM Change Notifications
Scope OS files (binaries, configs, libraries) SSM-managed resources (parameter changes, AMI changes)
Granularity File hash level (specific file changed) Higher-level (parameter store value changed)
Operational overhead Daily AIDE scan + baseline management EventBridge (managed by AWS)
Coverage Comprehensive OS-level Limited to SSM scope
Recommendation Both: AIDE for OS binary integrity (detect rootkit/binary replacement); SSM Change Notifications for infrastructure configuration drift

Common Mistakes to Avoid

  1. Forgetting that ECS Fargate has no agent model. Candidates often describe "install an agent on Fargate" — this is not possible. Fargate containers run in AWS-managed microVMs with no direct host access. Runtime protection for Fargate requires GuardDuty ECS Runtime Monitoring (sidecar injection) and container-native controls (readonlyRootFilesystem, capability dropping). Saying "deploy CrowdStrike on Fargate" is a red flag.
  2. Not addressing IMDSv2 enforcement. SSRF → IMDS credential theft is one of the most common EC2 attack vectors (Capital One breach was IMDSv1). Candidates who design a CWPP without mentioning IMDSv2 enforcement (SCP: HttpTokens=required) are missing a critical baseline control. This should be the FIRST SCP any org deploys.
  3. Treating vulnerability management as a one-time scan. Inspector v2 continuously re-evaluates existing instances as new CVEs are published. A vulnerability found today might not have a patch yet — but when the patch is released in 30 days, Inspector automatically updates the finding to include the fix version. Candidates who describe a "monthly scan" approach miss this continuous nature.
  4. Not distinguishing between CVE detection and runtime threat detection. These are two different capabilities: Inspector v2 detects that a vulnerable package is installed (static analysis). GuardDuty Runtime detects that the vulnerability is BEING EXPLOITED (dynamic behavioral detection — reverse shell launched, /tmp binary executed). Both are needed: static analysis catches the exposure; runtime detection catches the exploitation.

Likely Follow-up Questions

Q1: How do you handle a CrowdStrike agent update that breaks an application?
Rolling agent updates with canary testing: (1) CrowdStrike Falcon updates: use CrowdStrike N-1 policy (allow current version AND previous version). New agent version first deployed to: dev accounts (automatic), staging accounts (automatic, 48h after dev), production (manual promotion after staging validation). (2) Sensor groups in CrowdStrike: assign dev instances to "Auto-update" group; production to "Manual-approval" group. (3) Golden AMI: pin the CrowdStrike sensor version in the AMI build. When a new sensor breaks apps, roll back by reverting to the previous AMI version (EC2 Image Builder previous build artifact). (4) Canary: before triggering full Instance Refresh, update 5% of ASG instances (max unavailable = 5%) → run synthetic tests → if pass, proceed. If CrowdStrike update causes failures: stop Instance Refresh, revert ASG launch template to previous AMI version.

Q2: How do you manage GuardDuty Runtime Monitoring across 50 accounts?
Org-level GuardDuty Runtime Monitoring: (1) GuardDuty delegated admin (security account): enable EKS + EC2 + ECS Runtime Monitoring org-wide. All member accounts auto-enrolled. (2) Agent deployment: GuardDuty automatically installs the Runtime Monitoring agent on EC2 instances that have SSM agent installed (no manual action needed). For Fargate ECS: GuardDuty automatically injects the monitoring sidecar into ECS tasks (no task definition change required — GuardDuty manages the sidecar injection via ECS platform). (3) Finding deduplication: GuardDuty deduplicated findings per EC2 instance per day (no alert storm from repeated detections). (4) Coverage gaps: instances without SSM agent = no Runtime Monitoring coverage. Enforce via SCP: deny ec2:RunInstances if instance type not SSM-compatible. Use AWS Config rule: ec2-instance-managed-by-systems-manager — non-compliant instances alert Security Hub.

Q3: How do you detect if the SSM agent itself has been compromised?
SSM agent compromise detection: (1) GuardDuty Runtime Monitoring: if SSM agent is compromised to execute unexpected commands, Runtime Monitoring sees: process amazon-ssm-agent spawning unusual child processes (not expected SSM document commands). Alert: process ancestry anomaly. (2) SSM execution audit: all SSM Run Command and Session Manager sessions are logged in CloudTrail. Alert if: unexpected API call pattern (unusual document name, unusual principal), sessions opened by unexpected IAM principals. (3) AIDE: SSM agent binary (/usr/bin/amazon-ssm-agent) is in the AIDE monitored file set. If binary is replaced, AIDE detects the hash change. (4) SSM agent auto-update: configure SSM agent to auto-update to latest version (avoid running outdated agent with known vulnerabilities). Inspector v2 scans the SSM agent package itself for CVEs. (5) Memory protection: if an attacker injects code into the SSM agent process memory (shellcode injection), GuardDuty Runtime Monitoring detects: unexpected memory-mapped executable regions in the agent process.

Q4: How do you measure CWPP coverage (what percentage of workloads are protected)?
Coverage measurement: (1) SSM Fleet Manager: shows all managed instances (SSM agent running) vs. total EC2 instances in account. Coverage gap = EC2 instances NOT in SSM Fleet Manager. Config rule: ec2-instance-managed-by-systems-manager — compliance percentage = coverage. (2) Inspector v2: Inspector console → Coverage tab. Shows: EC2 instances with Inspector enabled / total EC2 instances. ECS tasks with Inspector enabled / total ECS tasks. Lambda functions with Inspector enabled / total Lambda functions. (3) GuardDuty Runtime: GuardDuty console → Protected resources. Shows: EC2 instances with Runtime Monitoring agent / total eligible EC2. ECS clusters with Runtime Monitoring enabled / total clusters. (4) Security Hub custom insight: "CWPP Coverage Gap": { "Filters": { "ResourceType": [{"Value": "AwsEc2Instance"}], "ProductFields": [{"Key": "GuardDuty.RuntimeMonitoring", "Value": "NOT_ENABLED"}] } } — shows unprotected instances. (5) Weekly dashboard (QuickSight): overall coverage percentage by account → target = 100%. Accounts below 95% coverage → alert to account team.

Q5: How would you handle a discovered rootkit on an EC2 instance?
Rootkit incident response: (1) Detection: AIDE reports unexpected changes to /usr/bin/*, /lib/* binaries. CrowdStrike or GuardDuty Runtime detects kernel-level activity (setuid program, unexpected syscalls). (2) Isolation (automated Step Functions playbook): modify instance SG → deny all inbound/outbound (quarantine). Add instance to "QUARANTINED" tag. Remove from ALB target group (no production traffic). Notify SOC P1. (3) Evidence preservation (before termination): EBS Snapshot: ec2:CreateSnapshot → forensic copy of root volume. Memory dump: SSM Run Command → install avml tool → capture /dev/mem to S3 (if attacker hasn't blocked SSM yet; if they have, rely on the EBS snapshot). VPC Flow Logs: preserve 30-day flow log history. (4) Investigation: forensic EC2: launch new instance in isolated forensic VPC, attach the snapshot as secondary volume. Analyze: what files were changed, what processes ran, what network connections were made. (5) Remedination: terminate the compromised instance (do NOT reboot — attacker likely has persistence in init scripts). Launch new instance from golden AMI in same subnet. Verify: new instance is clean (Inspector scan + CrowdStrike clean state). (6) Root cause analysis: how did the rootkit get installed? SSRF + IMDS credential theft? RCE vulnerability in application? Compromised admin SSH access? Fix the root cause before returning to production.

Q6: How do you ensure CWPP effectiveness for Lambda functions?
Lambda CWPP coverage: (1) Inspector v2 Lambda scanning: automatically scans Lambda function code packages for known CVEs in application dependencies (Python pip packages, Node.js npm packages, Java Maven dependencies). Triggered: on function deployment + continuously when new CVEs are published. (2) Lambda runtime security: Lambda execution environments run inside AWS Nitro (hardware isolation). No persistent filesystem between invocations. Read-only /var/task (function code). Write access: only /tmp (512MB). For high-security functions: use Lambda SnapStart with integrity verification (known-good state snapshot). (3) GuardDuty Lambda Protection: GuardDuty detects anomalous Lambda execution behavior. Finding: Lambda/MaliciousLambdaExecution — Lambda function making unexpected API calls (e.g., a data processing Lambda starting to call ec2:RunInstances). (4) Permissions audit: Lambda function roles — use IAM Access Analyzer to identify overly broad permissions. Lambda should have minimum permissions (no * in Action or Resource). (5) Dependency pinning: Lambda layers with pinned dependency versions. When Inspector flags a CVE: update the layer, redeploy functions that use the layer. Canary deployment: update 10% of functions first (Lambda alias weights) → verify no errors → promote to 100%.

Q7: How does AWS Compute Optimizer interact with CWPP for right-sizing?
Compute Optimizer and security: (1) Compute Optimizer makes CPU/memory right-sizing recommendations for EC2. From a security perspective: over-provisioned instances (too much CPU/memory relative to actual use) are a waste but not a direct security risk. HOWEVER: under-utilized instances that are still running may indicate zombie instances (forgotten workloads that haven't been patched). (2) Security team use of Compute Optimizer data: identify instances with > 60 days of low utilization → these are candidates for decommissioning. Decommissioning reduces attack surface (fewer unpatched instances). (3) Auto Scaling right-sizing: if Compute Optimizer recommends downsizing, this triggers Instance Refresh → new instances launched from latest golden AMI (fresh, patched). Unexpected benefit: right-sizing forces patching. (4) Inspector integration with instance health: Inspector vulnerability findings correlated with Compute Optimizer utilization → prioritize patching for high-utilization (high-value) instances first vs. low-utilization (can decommission instead of patching).

Q8: How do you handle EC2 instances that cannot be patched without an application outage?
Unpatcha-able instances (legacy / always-on): (1) Compensating controls (instead of patching): for instances that cannot be patched (COTS vendor hasn't released a patch, or patching requires extended downtime): Virtual patching via WAF: AWS WAF rules to block exploitation attempts for the specific CVE (block the HTTP payload pattern that exploits the vulnerability). Network segmentation: tighten security groups to minimize attack surface (reduce to minimum required inbound ports and source IPs). GuardDuty Runtime: ensure runtime detection is enabled (detect if exploitation occurs). (2) Formal exception process: create a security exception ticket: CVE ID, affected instance, reason cannot patch, compensating controls applied, risk acceptance sign-off (business owner + CISO), review date (30 days). (3) Isolation: place unpatched instances in a dedicated "legacy" VPC with strict outbound restrictions (cannot call AWS APIs → reduces blast radius if compromised). (4) Decommission timeline: for each unpatched instance, assign a decommission date (acceptable = end of fiscal year at most). Track in Security Hub custom finding with "AcceptedRisk" label. Review at each quarterly security review.
