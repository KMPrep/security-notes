Q21 — Secure Cross-Account Data Sharing Architecture

The Ask

"Design a secure cross-account data sharing architecture for a financial services company
that needs to share customer data between 10 business units (each in a separate AWS account),
with an external audit firm, and with three regulated data consumers (analytics team, risk
team, compliance team). Data contains PII and financial records. Design access control,
data minimization, audit logging, and revocation mechanisms."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What types of data access patterns are needed — bulk data export, real-time API queries, or read-only analytical queries?
    Impact: Bulk export → S3 pre-signed URLs or S3 Access Points with cross-account bucket policies. Real-time API → PrivateLink (API exposed via VPC endpoint, no data leaves). Analytical queries → Amazon Redshift Data Sharing or Lake Formation cross-account table access (query data without copying it).

  • Q: Should the data consumers receive a COPY of the data or query access to the source data?
    Impact: Copy: data leaves source account → harder to revoke. Query access (zero-copy): data stays in source account → revoking Lake Formation permission immediately denies access. For highly sensitive PII, prefer zero-copy (Lake Formation, Redshift Data Sharing) over physical copy.

  • Q: What are the data retention and sovereignty requirements? Can data cross specific geographic boundaries?
    Impact: EU users' PII → GDPR data residency: cannot move to US region. Data sharing must stay within EU region (S3 replicas only to EU regions, no cross-region sharing of EU PII). Financial records under specific jurisdiction may have similar restrictions.

  • Q: What is the sensitivity classification of the shared data — what fields contain PII vs. aggregate/anonymized data?
    Impact: PII fields (name, SSN, account number): field-level masking or tokenization before sharing. Aggregate data (total transactions per category, risk scores): can be shared broadly. Field-level access control is needed: analytics team sees aggregate + anonymized; compliance team sees PII with masking; audit firm sees full record for regulatory examination.

  • Q: How does the external audit firm connect — from their own AWS account, from corporate network, or via a portal?
    Impact: External party → AWS account: most secure (share via Lake Formation cross-account grant, auditor uses their AWS IAM credentials). External party → corporate network: PrivateLink with on-premises connectivity. External party → portal: build a web UI that calls AWS APIs on their behalf (no direct AWS console access). For regulated audit firms, separate AWS account with strict IAM is preferred.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Central data catalog account (data mesh governance) with domain accounts for each business unit.
  2. Lake Formation for zero-copy cross-account table access (analytics, risk, compliance teams).
  3. Redshift Data Sharing for analytics workloads requiring SQL queries.
  4. S3 Access Points for cross-account S3 object access with specific prefixes.
  5. AWS Clean Rooms for external audit firm (aggregated analytics without raw data exposure).

Core Architecture — Data Flow Narrative

[DATA MESH ARCHITECTURE]

Account structure:
  Governance Account (central):
    AWS Lake Formation: central data catalog (Glue Data Catalog)
    All other accounts register their S3 buckets as Lake Formation data locations
    Governance team: issues cross-account Lake Formation grants

  Domain Accounts (10 business units):
    BU01-Account: owns Customer_Profile data lake (S3 + Glue catalog)
    BU02-Account: owns Transaction_History data lake
    BU03-Account: owns Risk_Score data lake
    ...

  Consumer Accounts:
    Analytics-Account: analytics team (aggregated queries)
    Risk-Account: risk team (risk + transaction data)
    Compliance-Account: compliance team (PII data for regulatory review)
    AuditFirm-Account: external audit firm (restricted view)

[LAKE FORMATION CROSS-ACCOUNT DATA SHARING]

Step 1: Data owner registers data location (BU01-Account):
  BU01 Glue catalog: table "customer_profiles" (pointing to s3://bu01-datalake/customers/)
  BU01 registers Glue catalog resource in Governance Account's Lake Formation

Step 2: Governance Account (Lake Formation admin) issues cross-account grant:
  Resource link: create resource link "bu01.customer_profiles" in Analytics-Account
  Grant:
    Grantee: arn:aws:iam::analytics-account:role/analyst-role
    Table: bu01.customer_profiles
    Permissions: SELECT
    Column filter (data masking):
      - Analytics team: SELECT on columns [customer_id (tokenized), transaction_amount, category]
        EXCLUDE columns: [ssn, full_name, address, date_of_birth]  (PII excluded)
      - Compliance team: SELECT on all columns EXCEPT ssn
        ssn column: masked (obfuscated via Lake Formation column-level security)
      - Audit firm: SELECT on all columns (for regulatory examination)
        BUT: restricted to specific row filter (WHERE account_status = 'DISPUTED')

  Lake Formation resource policy:
    Prevents any consumer from doing s3:GetObject directly (bypass Lake Formation)
    S3 bucket owned by BU01: bucket policy DENY GetObject unless via Lake Formation service principal
    Lake Formation is the ONLY path to data

Data minimization in practice:
  Lake Formation column-level security:
    Analytics-Account analyst-role: cannot see SSN, full_name, address, DOB in any query
    Even SELECT * returns only permitted columns (Lake Formation filters at the catalog layer)

  Row-level security (RLS):
    Compliance-Account: only sees records for their jurisdiction (US customers → WHERE country = 'US')
    AuditFirm-Account: only sees records for disputed accounts (WHERE status = 'DISPUTED')

  Data cell filters:
    Lake Formation data cell filter: combine row filter + column filter in one policy
    audit-firm-filter:
      Column filter: all columns permitted
      Row filter: WHERE account_status = 'DISPUTED'
    Grant: audit-firm-role uses audit-firm-filter

[AMAZON REDSHIFT DATA SHARING]

Redshift Data Sharing (for analytics SQL workloads):
  Producer: BU01-Account Redshift cluster
    CREATE DATASHARE transaction_analytics;
    ADD TABLE transactions, aggregated_risk_scores TO DATASHARE transaction_analytics;
    GRANT USAGE ON DATASHARE transaction_analytics TO NAMESPACE '<analytics-cluster-id>';

  Consumer: Analytics-Account Redshift cluster
    CREATE DATABASE bu01_analytics FROM DATASHARE transaction_analytics OF NAMESPACE '<producer-namespace>';
    SELECT category, SUM(amount) FROM bu01_analytics.transactions GROUP BY category;

  Key properties:
    Zero-copy: data stays in BU01 Redshift cluster
    Consumer pays for their own compute (no data transfer cost)
    Revocation: REVOKE USAGE ON DATASHARE → Analytics-Account immediately loses access
    Encrypted: Redshift cluster encryption (KMS CMK) applies; data not decrypted for cross-account sharing

  Access controls:
    Analytics-Account: can query datashare but CANNOT WRITE to it (read-only)
    Analytics-Account: cannot CREATE TABLE AS SELECT into the shared database
    Consumer views: create views in local Analytics-Account database that join shared tables with local data

[S3 ACCESS POINTS FOR CROSS-ACCOUNT OBJECT ACCESS]

Use case: Risk-Account needs access to specific S3 prefix (raw transaction files)

S3 Access Point:
  Create access point in BU01-Account:
    aws s3control create-access-point
      --name risk-team-transactions
      --account-id bu01-account-id
      --bucket bu01-datalake
      --vpc-configuration VpcId=vpc-risk-account  (restrict to specific VPC)

  Access point policy:
    Allow: iam:PrincipalARN risk-account:role/risk-analyst-role
    Allow: s3:GetObject on /transactions/2024/* prefix only
    Deny: s3:GetObject on /transactions/*/pii-fields/ prefix (raw PII excluded)

  Bucket policy (BU01 bucket):
    Allow: access point ARN from BU01 account (delegate to access point policy)
    Deny: direct s3:GetObject from any principal except Lake Formation service (no bypass)

  Benefits vs. bucket policy:
    Isolated policy per consumer (risk team policy separate from compliance team policy)
    Revoke: delete the access point → risk team immediately loses access
    VPC restriction: access point only usable from specific VPC in Risk-Account (no internet access)

[AWS CLEAN ROOMS FOR EXTERNAL AUDIT FIRM]

Use case: Audit firm needs to validate data without seeing raw PII

AWS Clean Rooms:
  Collaboration: create Clean Rooms collaboration between BU01-Account and AuditFirm-Account

  Analysis rules (defined by BU01 as data owner):
    Aggregation rule:
      Allowed queries: GROUP BY + aggregate functions (SUM, COUNT, AVG) only
      Minimum group size: 5 (if fewer than 5 rows match a filter → result suppressed)
      Overlap analysis: enabled (audit firm can test if their customer list overlaps with BU01 PII)

    Custom analysis rule (Athena-based):
      Allowed SQL templates (pre-approved queries only):
        - "SELECT COUNT(*), SUM(disputed_amount), AVG(resolution_days) FROM disputes WHERE status = ?"
        - "SELECT jurisdiction, COUNT(*) as complaint_count FROM regulatory_reports WHERE year = ?"

    Clean Rooms prevents:
      Raw record export (no SELECT without aggregation)
      JOIN to non-approved tables
      Exfiltration via subquery tricks

  Audit firm analyst:
    Logs into AuditFirm-Account → accesses Clean Rooms collaboration
    Runs pre-approved SQL templates → sees only aggregated results
    No raw PII ever leaves BU01-Account

  Clean Rooms audit log:
    Every query by audit firm: logged in CloudTrail + Clean Rooms query log
    Query text + parameters + result summary (not raw results) → stored in governance account

[KMS KEY POLICY FOR CROSS-ACCOUNT DATA]

BU01 S3 data encrypted with bu01-data-key (KMS CMK in BU01-Account):
  Key policy: allow decryption ONLY to specific cross-account roles:
    Compliance-Account: compliance-role → kms:Decrypt (column filter applied by Lake Formation first)
    Risk-Account: risk-role → kms:Decrypt (access point policy restricts to specific prefix)
    AuditFirm-Account: clean-rooms-service-role → kms:Decrypt (Clean Rooms aggregates before returning)

  Key policy deny:
    Deny kms:Decrypt to * EXCEPT listed cross-account roles
    AuditFirm-Account: cannot decrypt directly — must use Clean Rooms workflow

  Cross-account KMS:
    BU01 key policy includes external account ARNs → those accounts' roles can use the key
    External account IAM policy must ALSO grant kms:Decrypt (both sides required for cross-account KMS)

[AUDIT LOGGING AND ACCESS TRAIL]

Lake Formation data access logging:
  Every SELECT query against a Lake Formation-managed table: CloudTrail event
    eventName: GetTable, BatchGetPartition, GetQueryResults
    Principal: who (cross-account role ARN), when, which table, which columns

  S3 data event logging:
    S3 object-level logging: every GetObject call → CloudTrail data event
    Who accessed: cross-account role (with Source-Account field)
    Object key: tells you which data file was read

  Lake Formation data access monitoring:
    Lambda: daily Athena query on CloudTrail → generate per-consumer access report
    Report: AuditFirm-Account accessed disputes_table 45 times yesterday (unusual spike) → alert

  Clean Rooms audit log:
    Every collaboration query logged with: query text, runtime, result row count
    Cannot log result data (would defeat the purpose of Clean Rooms)

  Centralized audit trail:
    All CloudTrail logs from BU01-Account + consumer accounts → Security Lake (governance account)
    Retention: 7 years for financial records (SOX requirement)
    Tamper-evident: S3 Object Lock WORM

[REVOCATION MECHANISMS]

Immediate revocation (< 5 minutes):
  Lake Formation: REVOKE SELECT on table from consumer-role
    → Consumer's next query returns: "permission denied" (no caching — Lake Formation checks on every query)

  S3 Access Point: DeleteAccessPoint
    → Consumer loses S3 access immediately

  Clean Rooms: remove account from collaboration
    → AuditFirm-Account cannot run any queries

  Redshift Data Sharing: REVOKE USAGE ON DATASHARE
    → Analytics-Account loses access to shared database immediately

  KMS key access revocation: update key policy to remove grantee
    → All subsequent decrypt attempts fail (< 5 seconds propagation)

  Break-glass mass revocation:
    Lambda function: "RevokeBU01Access"
    If BU01 data breach detected → one button:
      Revoke all Lake Formation grants for BU01 tables
      Delete all S3 Access Points for BU01 bucket
      Update KMS key policy to deny all external accounts
      Revoke all Redshift Data Sharing for BU01 cluster
    Effect: all external access to BU01 data revoked within 5 minutes

  Access review (quarterly):
    Lake Formation: list all grants (GetEffectiveLakeFormationPolicy)
    Access review Lambda: generates CSV of "who has access to what data" → sent to data owner for review
    Data owner: confirm or revoke each grant via UI

Key AWS Services Selected and Why

Service Role in Architecture Why this over alternatives
AWS Lake Formation Zero-copy cross-account table access with column/row filtering Data doesn't leave source account; immediate revocation; column-level PII masking; row-level security; cell filters
Amazon Redshift Data Sharing Zero-copy SQL queries across accounts for analytics Analytics team gets full SQL capability without data copy; built-in access control; KMS encryption preserved
S3 Access Points (cross-account) Granular S3 prefix access for specific consumers Per-consumer policy; VPC restriction; prefix-level access control; instant revocation by deleting access point
AWS Clean Rooms External audit firm access without raw data exposure Pre-approved SQL templates; minimum group size (k-anonymity); data never leaves source account; built for cross-company sharing
Lake Formation Tag-Based Access Control (LF-TBAC) Policy management at scale (50+ tables) Tag tables with Classification=PII, Domain=Finance; grant access by tag (not per-table)

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Data mesh IAM pattern:
  Data owner (BU01):
    lakeformation:GrantPermissions — grant access to consumers
    lakeformation:RevokePermissions — revoke access
    s3:PutObject on own bucket

  Data consumer (Analytics-Account analyst-role):
    glue:GetTable (via resource link)
    athena:StartQueryExecution + s3:PutObject on results bucket
    Cannot: s3:GetObject directly on BU01 bucket (bucket policy blocks direct access)
    Cannot: lakeformation:GrantPermissions (cannot grant to others what they don't own)

  Governance admin (Governance-Account):
    lakeformation:RegisterResource (register BU01 S3 as LF data location)
    lakeformation:GrantPermissions (centrally manages all cross-account grants)

  External audit firm:
    AuditFirm-Account: separate AWS account with strict SCPs
    SCP: deny all API calls EXCEPT cleanrooms:StartProtectedQuery, cleanrooms:GetCollaborationAnalysisTemplate
    Only Clean Rooms queries allowed — no S3, EC2, RDS, or other service access

2. Encryption

BU01 S3 data: SSE-KMS (bu01-data-key)
Cross-account KMS access:
  Consumer account's role → bu01-data-key in BU01 account
  Key policy: allow decrypt for specific cross-account role ARNs
  Consumer IAM policy: must also grant kms:Decrypt (both sides of cross-account KMS)

Lake Formation does NOT decrypt data for you:
  When Analytics-Account queries via Athena:
    Athena execution role needs kms:Decrypt for bu01-data-key
    Column filter applied at Lake Formation layer (before returning results)
    Athena sees only permitted columns in decrypted form

Redshift Data Sharing encryption:
  Data at rest: Redshift cluster KMS encryption preserved during sharing
  Data in transit: TLS between Redshift clusters

3. Network Isolation

S3 Access Points: VPC restriction
  Access point: VPCConfiguration.VpcId = vpc-risk-account
  Only requests from that specific VPC are authorized
  Risk team cannot access data from their laptop (must use VPN to corporate VPC)

Athena (for Lake Formation queries):
  Athena VPC endpoint: queries run within VPC (no internet)
  S3 results bucket: VPC endpoint for S3 (results don't traverse internet)

Redshift Data Sharing:
  PrivateLink: producer and consumer Redshift clusters communicate via PrivateLink (no public internet)

Clean Rooms:
  Clean Rooms service operates within AWS network (no external exposure)
  Audit firm AWS account → Clean Rooms → BU01 data — all within AWS backbone

4. Logging and Audit Trail

Cross-account data access:
  CloudTrail: every Glue/Athena call from Analytics-Account includes:
    Source account ID (Analytics-Account), role ARN, query timestamp
    Which Glue tables accessed (table ARN with account context)

  Lake Formation logging (in BU01 account):
    LF logs which principals accessed which tables/columns/rows
    exportable to S3 + Athena queryable

  Centralized audit (Governance Account):
    All accounts' CloudTrail → Security Lake (governance account)
    Athena query: "who accessed customer_profiles table in the last 30 days from any account?"

5. Least Privilege and Blast Radius

Column-level filtering:
  Analytics team: never sees SSN, address, DOB (even if they run SELECT *)
  Blast radius of Analytics-Account compromise: attacker can query aggregate+anonymized data
  Cannot reconstruct PII from permitted columns alone

Access point prefix restriction:
  Risk team access point: /transactions/2024/ prefix only
  Compromise of risk-analyst-role: attacker can download 2024 transactions but not customer PII directory

Clean Rooms minimum group size:
  Audit firm: aggregate query with fewer than 5 matching records → result suppressed
  Cannot use Clean Rooms to look up individual records (de-anonymization prevented)

6. Compliance Mapping

Requirement Control
GDPR Art. 25 — Data minimization Lake Formation column/row filters; Clean Rooms k-anonymity minimum group size
GDPR Art. 46 — Transfers to third parties Clean Rooms for external audit firm (no raw transfer); data stays in AWS account
PCI-DSS Req 7 — Access to cardholder data Lake Formation row/column filter (analytics team cannot see PAN); KMS key access restricted to authorized roles
SOX — Financial data audit trail 7-year CloudTrail retention; tamper-evident S3 Object Lock; Lake Formation access logs
HIPAA § 164.514 — De-identification Lake Formation column exclusion for PHI fields; tokenization of patient IDs

Scale and Resilience (Phase 4 — 10 min)

Lake Formation at scale:

  • Lake Formation tag-based access control (LF-TBAC): instead of per-table grants (50 tables × 3 consumer roles = 150 grant statements), assign tags: Classification=PII, Domain=Finance. Grant: consumer-role has SELECT on tables where Classification=PII AND Domain=Finance. Adding new tables with correct tags automatically extends access (no new grant needed).

Cross-account query performance:

  • Athena queries across account boundary (via resource link): same performance as same-account query (resource link is just a pointer, data stays in BU01's S3).
  • Redshift Data Sharing: query latency is same as same-cluster query (Redshift cluster handles the cross-account data sharing internally).

Resiliency:

  • If BU01-Account is temporarily unavailable: consumer accounts cannot query BU01 data (no local copy). Trade-off of zero-copy architecture.
  • For high-availability requirements: scheduled materialized views (copy critical aggregates to consumer account — with explicit approval and audit trail).

Trade-off Analysis — The Hard Choices

Decision 1: Zero-Copy (Lake Formation) vs. Physical Copy to Consumer Account

Factor Zero-Copy (Lake Formation) Physical Copy (S3 replication to consumer account)
Data sovereignty Data stays in source account Consumer account has independent copy
Revocation speed Instant (revoke grant) Must delete replica AND revoke access
Query availability Dependent on source account uptime Independent (works even if source is down)
Cost Source pays for S3 reads Consumer pays for storage + source pays for replication
Recommendation Zero-copy for PII and regulated data (cannot lose control once copy exists); Physical copy only for non-sensitive analytical data with performance SLA requirements

Decision 2: Lake Formation Column Filter vs. Data Masking at Application Layer

Factor Lake Formation Column Filter (schema-level) Application-Layer Masking (ETL job masks before sharing)
Enforcement point Catalog layer (before any query) Depends on ETL running correctly
Developer bypass risk Cannot bypass (enforced by Lake Formation) ETL bug could expose unmasked data
Performance No overhead ETL adds processing time
Flexibility Predefined column exclusions Can compute masked fields (hash SSN)
Recommendation Lake Formation column filter for access control; ETL masking for derived columns (e.g., share tokenized_ssn column that's computed from real SSN by ETL) — both together for defense-in-depth

Decision 3: AWS Clean Rooms vs. Custom Secure Computation Environment

Factor AWS Clean Rooms Custom (isolated EC2, network controls, manual audit)
Raw data exposure risk Zero (Clean Rooms never exposes raw records) Depends on implementation quality
Query flexibility Pre-approved templates only Full SQL
Operational overhead Fully managed High (maintain isolated environment, audit access)
Regulatory acceptance AWS service with built-in audit log Custom system requires more documentation
Recommendation AWS Clean Rooms for external parties (audit firm, regulators) where minimum raw data exposure is required; custom environment only if Clean Rooms query restrictions are too limiting

Common Mistakes to Avoid

  1. Proposing S3 bucket cross-account policies as the primary mechanism. Bucket policies for cross-account access work, but they bypass all fine-grained controls (no column/row filtering, no immediate revocation via permission change, no audit trail of specific data accessed). Lake Formation is the correct answer for structured data sharing with PII — it provides column/row filtering that bucket policies cannot.
  2. Not mentioning AWS Clean Rooms for external parties. External audit firms or regulators should NEVER get direct AWS console access or direct table access. Clean Rooms provides a purpose-built service for controlled cross-organization data analysis. Candidates who say "create an IAM user for the audit firm" show lack of depth.
  3. Forgetting that cross-account KMS requires BOTH sides. Cross-account KMS access requires: (1) key policy in the source account allows the grantee account/role AND (2) IAM policy in the grantee account allows kms:Decrypt. Both are required. Candidates who describe only the key policy without the IAM policy have an incomplete answer.
  4. Treating revocation as complex. Candidates sometimes describe revocation as a complex multi-step process. With Lake Formation: REVOKE immediately takes effect (no caching, no TTL). With S3 Access Points: delete the access point. These are single API calls. The simplicity of revocation is a KEY advantage of zero-copy architectures — make sure to highlight this.

Likely Follow-up Questions

Q1: How do you handle a scenario where the analytics team needs to join BU01 customer data with their own local data?
Cross-account join pattern with Lake Formation: (1) Analytics-Account creates a resource link to BU01's customer_profiles table (resource link appears as a local table in analytics-account's Glue catalog). (2) Analytics-Account has their own local enrichment table (e.g., campaign_metadata). (3) Athena query in Analytics-Account: SELECT a.customer_id, a.transaction_amount, b.campaign_name FROM bu01.customer_profiles a JOIN analytics.campaign_metadata b ON a.customer_id = b.customer_id. Athena handles the cross-account table join transparently. (4) Lake Formation column filter is enforced: even in the join, the customer_profiles table only returns permitted columns (no SSN, DOB) — the column filter applies at the query execution level. (5) Cost: Analytics-Account pays for Athena compute; BU01 pays for S3 read (cross-account reads billed to the requester account with requester-pays buckets, or the owner account). (6) Result: Analytics-Account can join their data with BU01's permitted columns — they never see BU01's PII columns.

Q2: How does Lake Formation handle row-level security for complex multi-condition filters?
Lake Formation data cell filters: (1) A data cell filter combines a column filter (which columns are visible) AND a row filter (which rows are visible) in a single named policy. (2) Row filter expression: any valid SQL WHERE clause: account_status = 'ACTIVE' AND customer_country = 'US' AND account_balance > 0. (3) Multiple filters per table: BU01 can create multiple data cell filters for customer_profiles: us-compliance-filter: rows WHERE country='US', columns: all except SSN. eu-compliance-filter: rows WHERE country='EU', columns: all except SSN. analytics-filter: rows WHERE account_status='ACTIVE', columns: customer_id (tokenized), amount, category ONLY. (4) Grant: assign each filter to the appropriate consumer role (EU compliance team uses eu-compliance-filter). (5) Limitation: row filter is evaluated at query time by Athena — for very large tables (billions of rows), row filter queries still scan the full table (partition pruning doesn't apply to LF row filters unless the row filter column is also a partition key). Optimization: design partition keys to align with row filter columns (e.g., partition by country so WHERE country='US' benefits from partition pruning).

Q3: How would you implement a data access request and approval workflow?
Self-service data access with approval: (1) Request portal: web UI (hosted in governance account) where consumers submit requests: "Analytics team requests SELECT on customer_profiles columns: customer_id, transaction_amount, category. Justification: quarterly spending analysis. Duration: 90 days." (2) Approval workflow: AWS Step Functions human approval task. Request stored in DynamoDB. Notification: SES email to BU01 data owner with "Approve/Deny" buttons (URL with JWT token). Data owner approves → Step Functions resumes. (3) Automated grant: on approval, Lambda calls lakeformation:GrantPermissions with the approved column filter. Grant includes: metadata tag ExpiryDate=2024-03-31 stored in DynamoDB access-request-table. (4) Time-bounded access: scheduled Lambda (daily): check DynamoDB for expired grants → call lakeformation:RevokePermissions for expired entries. (5) Audit: every grant + approval + revocation event stored in DynamoDB access-request-table (who requested, who approved, what was granted, when revoked). Athena query for quarterly SOC 2 evidence.

Q4: How do you prevent the Redshift Data Sharing from being used to reconstruct PII that Lake Formation would block?
Defense-in-depth for Redshift Data Sharing: (1) Data owner (BU01) controls WHAT is included in the datashare. Only include tables/views without PII columns in the datashare: ADD TABLE transactions_summary TO DATASHARE analytics_share (transactions_summary is a view that excludes PII columns). (2) Redshift column-level security (similar to Lake Formation): in the producer Redshift cluster, create a VIEW on top of the raw table with PII columns excluded: CREATE VIEW transactions_masked AS SELECT transaction_id, amount, category, merchant (NO SSN, account_number) FROM transactions;. Add this view (not the base table) to the datashare. (3) Row-level security in Redshift: apply a Redshift RLS policy to restrict which rows are visible: CREATE RLS POLICY consumer_policy USING (account_status = 'ACTIVE');. Attach to the view. Consumer query only returns active-status rows. (4) Cross-account audit: every query executed against the shared database in the consumer cluster is logged in the consumer Redshift audit log AND (via DataShare query logging) visible to the producer admin. Unusual queries (SELECT with no GROUP BY against large table) → alert.

Q5: How do you handle the scenario where a business unit refuses to participate in Lake Formation governance?
Non-participating data owners: (1) SCP enforcement: in the AWS Organizations landing zone, apply an SCP that requires all data publishing accounts to register their S3 data lake buckets with the central Lake Formation governance account within 90 days. Non-compliance → Config rule (custom) flags the account → Security Hub finding → escalation to VP. (2) Incentive: central governance offers benefits (self-service data access portal, automated consumer management, compliance-ready audit trail). Frame as service, not mandate. (3) For existing non-participating accounts: Glue crawlers from the governance account (with cross-account S3 read permission) can catalog the data without full LF participation. Partial governance is better than none. (4) CSPM: AWS Security Hub custom insight: "Data publishing accounts not registered with Lake Formation" — visible to CISO dashboard. Creates organizational pressure. (5) Data sharing without LF: if BU refuses LF governance, alternative is manual cross-account bucket policies + manual audit (much less secure). Security team can quantify the risk: without LF column filters, consumers could access PII → GDPR/PCI risk quantification.

Q6: How would you implement differential privacy for sensitive aggregate queries?
Differential privacy in Clean Rooms: (1) AWS Clean Rooms differential privacy (DP) feature: add DP noise to aggregate results. Configuration: epsilon = 1.0 (privacy budget per query — lower epsilon = more noise = more privacy). Budget: each analyst has a fixed privacy budget (e.g., epsilon = 10 per day). Each DP query consumes epsilon budget → analyst exhausts budget after sufficient queries. (2) Effect: COUNT of disputed accounts returns: actual 1,247 + Laplace noise (e.g., 1,249 ± 5). The noise prevents an attacker from using many closely related queries to deduce individual records. (3) Tradeoff: high noise level makes business analytics unreliable (can't distinguish 1,247 from 1,252). For business reporting: accept some privacy budget trade-off. For highest-sensitivity data (medical records, individual financial records): use strict epsilon. (4) Minimum group size + DP together: Clean Rooms can apply BOTH minimum group size suppression AND differential privacy noise — provides defense-in-depth against re-identification. (5) Privacy budget enforcement: Clean Rooms tracks epsilon budget per collaboration member. When analyst's daily budget is exhausted: further DP queries rejected (analyst must wait for budget reset the next day).

Q7: How do you handle data sharing for regulated data that cannot leave a specific region?
Data residency for cross-account sharing: (1) Lake Formation cross-account grants: by default, the data stays in the source account's S3 bucket (which is already in a specific region). Cross-account access means the consumer queries the data from the source region — the data doesn't move. Athena in the consumer account runs the query in the same region as the source S3 bucket (not the consumer's region). (2) S3 Access Points: VPC restriction ensures consumer accesses data from a VPC in the same region as the S3 bucket (if you require same-region access). Cross-region access to S3 is possible but can be blocked via S3 bucket policy: Condition: {"StringNotEquals": {"aws:RequestedRegion": "eu-west-1"}} → Deny. (3) Redshift Data Sharing: data sharing can only happen between Redshift clusters in the same AWS Region. Cross-region data sharing is not supported (as of 2024). This is an automatic data residency control for Redshift. (4) SCP enforcement: deny s3:ReplicateObject to non-EU regions for EU data buckets (prevents accidental cross-region replication of GDPR data). Deny glue:CreateTable in non-EU accounts for EU data sources (prevents cataloging EU data in non-EU accounts). (5) Audit: quarterly Macie classification jobs on all S3 buckets → verify EU PII is only in eu-west-1 bucket (no accidental copy to us-east-1).

Q8: How do you manage access for 500+ data assets at scale without individual grants?
Lake Formation Tag-Based Access Control (LF-TBAC): (1) Tag taxonomy: define LF tags for all data assets: Classification: PUBLIC, INTERNAL, CONFIDENTIAL, PII, PCI. Domain: Finance, Risk, CustomerService, Operations. Jurisdiction: US, EU, GLOBAL. (2) Tag all Glue tables at creation (automated via EventBridge: new Glue table → Lambda → lakeformation:AddLFTagsToResource). (3) Tag-based grants: instead of granting SELECT on customer_profiles + transactions + risk_scores individually: GRANT SELECT on tables where Classification IN (INTERNAL, CONFIDENTIAL) AND Domain = Finance TO analytics-role. Adding a new Finance domain table with Classification=INTERNAL automatically extends the analytics-role's access — no new grant required. (4) Grant hierarchy: LF-TBAC grants are additive — a role may receive access via multiple tag combinations. The most permissive applicable grant wins. Explicit DENY via a data cell filter overrides tag grants (PII columns still excluded). (5) Access review: quarterly Lambda: list all LF-TBAC grants + tagged tables → generate "effective access matrix" (which roles can access which tables via which tag grants). Send to data owners for review. Much simpler than reviewing 500 individual grants.
