Q23 — Secure Multi-Region Active-Active Architecture

The Ask

"Design a secure multi-region active-active architecture on AWS for a financial services
application processing 100,000 transactions per second across us-east-1 and eu-west-1.
Both regions must be fully operational simultaneously. Address data sovereignty (EU data
must not leave eu-west-1), encryption key management across regions, security control
parity, identity federation, and failure scenarios including a region-wide AWS outage."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: Does the application serve global users (routing by proximity) or is there strict data sovereignty that prevents cross-region routing?
    Impact: True active-active with geographic routing: US users → us-east-1, EU users → eu-west-1. This means EU data NEVER goes to us-east-1 (GDPR data residency). US data CAN serve from either region if needed for DR. Architecture must enforce data partition at the application layer, not just at the network layer.

  • Q: What is the RTO and RPO requirement for a full region failure?
    Impact: RTO=0, RPO=0: requires synchronous data replication (DynamoDB Global Tables, Aurora Global DB with synchronous replication). RTO=15 min, RPO=5 min: async replication (Aurora Global DB, S3 Cross-Region Replication) is sufficient. This drives the data layer design significantly.

  • Q: Is the database relational (Aurora) or NoSQL (DynamoDB), and how large is the data set?
    Impact: DynamoDB Global Tables: < 1ms cross-region replication, automatic conflict resolution. Aurora Global Database: < 1 second RPO, one primary per region. Both support active-active but with different conflict resolution models.

  • Q: For EU data: does the requirement apply only to PII, or to ALL data (including logs, metrics, encrypted blobs)?
    Impact: PII only → apply data residency controls only to customer_profile and transaction tables. Logs and metrics → typically exempt (aggregate, no personal data). Infrastructure data (AMI IDs, config) → can freely cross regions. Clarifying this reduces over-engineering.

  • Q: What identity provider is in use for workforce identity (Okta, Azure AD, internal IdP)?
    Impact: Workforce identity must work in both regions even if IdP is in one region. If IdP goes down during regional outage: engineers cannot log in (break-glass accounts needed). SAML/OIDC federation to both regions needs planning.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Application: microservices on EKS (active-active, serving transactions).
  2. US users → us-east-1, EU users → eu-west-1 (geographic routing with data residency enforcement).
  3. DynamoDB Global Tables for application state (conflict resolution: last-write-wins).
  4. Aurora Global Database for financial records (us-east-1 primary, eu-west-1 secondary).
  5. RPO=0 for transaction state, RPO=5min for financial records.
  6. KMS Multi-Region Keys for cross-region encryption.

Core Architecture — Data Flow Narrative

[GLOBAL TRAFFIC ROUTING — AWS GLOBAL ACCELERATOR]

AWS Global Accelerator (Anycast routing):
  Two accelerator endpoints: ALB in us-east-1 + ALB in eu-west-1
  Routing policy: GeographyBased (or latency-based)
    EU client IP (195.x.x.x/EU prefix) → eu-west-1 ALB
    US client IP (52.x.x.x/US prefix) → us-east-1 ALB
    APAC client IP → closest region (latency-based)

  Health checks:
    Global Accelerator: monitors both ALBs every 30 seconds
    If us-east-1 ALB health check fails:
      Global Accelerator automatically diverts ALL traffic to eu-west-1 (within ~30 seconds)
      No DNS TTL wait — anycast routing change is near-instant

  Route 53 ARC (Application Recovery Controller):
    Readiness checks: validates each region is ready to handle full load before failover
    Readiness check components:
      - EKS cluster: target # of nodes ready (>= minimum for full load)
      - DynamoDB Global Tables: replication status = ACTIVE
      - Aurora Global DB: secondary region lag < 1 second
      - WAF + Shield: active in target region
    Routing controls: manual/automated switch to redirect 100% traffic to one region
    Zonal shift: shift load away from impaired AZ within a region

[COMPUTE LAYER — EKS ACTIVE-ACTIVE]

EKS clusters (one per region):
  us-east-1 EKS: prod-us-east-1-cluster (3 AZs: us-east-1a, 1b, 1c)
  eu-west-1 EKS: prod-eu-west-1-cluster (3 AZs: eu-west-1a, 1b, 1c)

  Each cluster: same microservices, same Helm chart versions (GitOps: ArgoCD syncs both)
  EKS Version: same minor version (ensure consistency, no version drift between regions)

  Service mesh (AWS App Mesh or Istio):
    mTLS: all service-to-service calls encrypted and authenticated
    Traffic policies: EU cluster services MUST NOT make outbound calls to us-east-1 services
      (enforces data residency at the service mesh layer)

  EKS IRSA (IAM Roles for Service Accounts):
    Each microservice: dedicated IAM role per region
    eu-west-1 microservice IRSA role: can ONLY access DynamoDB Global Table in eu-west-1 endpoint
    Cannot accidentally write to us-east-1 DynamoDB endpoint (IAM condition: aws:RequestedRegion = eu-west-1)

  Security controls (applied identically in both regions):
    Kyverno policies: same version deployed via ArgoCD (no policy drift)
    Network policies (Calico): same policies, region-specific CIDR ranges
    Pod Security Standards: enforced namespace-level

[DATA LAYER — MULTI-REGION REPLICATION]

DynamoDB Global Tables (transaction state, session data):
  Table: customer_transactions
  Regions: us-east-1 + eu-west-1 (both active — writes accepted in either region)

  Data sovereignty enforcement:
    EU customers: partition key = eu_<customer_id> → DynamoDB item written to eu-west-1 region
    US customers: partition key = us_<customer_id> → DynamoDB item written to us-east-1 region

    Application layer enforcement:
      EU microservice: always writes to eu-west-1 DynamoDB endpoint (IRSA role: condition aws:RequestedRegion = eu-west-1)
      DynamoDB Global Tables: EU item written to eu-west-1 → replicated to us-east-1 copy
      BUT: replication is eventual — EU data will exist in us-east-1 table replica (GDPR problem!)

    Resolution — DynamoDB Global Tables with data residency:
      Option 1: Separate tables (NOT global tables):
        eu_customer_transactions table: ONLY in eu-west-1 (not replicated to us-east-1)
        us_customer_transactions table: ONLY in us-east-1
        Trade-off: no active-active for cross-region access (EU users cannot be served from US table if eu-west-1 fails)
        GDPR compliance: EU data NEVER in us-east-1

      Option 2: Global Tables with selective attribute encryption:
        EU PII fields: encrypted with eu-kms-key (KMS key that ONLY exists in eu-west-1)
        Replicated to us-east-1 but: encrypted ciphertext only — cannot decrypt in us-east-1
        AWS/US government cannot access plaintext EU PII even if they access us-east-1 (key is only in EU)
        Trade-off: EU data IS replicated (but encrypted with EU-only key)

  Aurora Global Database (financial records — compliance requirement: strong consistency):
    Primary: us-east-1 (writes go here for US records)
    Secondary: eu-west-1 (read-only replica, < 1 second replication lag)

    For EU financial records: Aurora cluster PRIMARY in eu-west-1 for EU writes
    Separate Aurora Global DB for EU:
      eu-primary → eu-west-1, eu-secondary → eu-west-1 (second AZ, NOT us-east-1)
    US Aurora Global DB:
      us-primary → us-east-1, us-DR-secondary → us-west-2 (DR only, not eu-west-1)

    Failover (Aurora Global DB):
      Manual failover: RTO ~1 minute (promote eu-secondary to primary)
      Automated: Route 53 ARC routing control + Lambda: trigger aurora:FailoverGlobalCluster
      RPO: < 1 second (Aurora Global DB lag SLA)

[KMS KEY MANAGEMENT — MULTI-REGION KEYS]

AWS KMS Multi-Region Keys:
  Problem: if us-east-1 application needs to decrypt data encrypted in eu-west-1, standard KMS CMK doesn't work cross-region

  Solution (where cross-region decryption is legitimately needed — e.g., DR):
    KMS Multi-Region Key: same key material in multiple regions (shared key ID prefix: mrk-)
    mrk-data-encryption-key:
      Primary key: us-east-1 (us-east-1 application can encrypt/decrypt)
      Replica key: eu-west-1 (eu-west-1 application can encrypt/decrypt with SAME key material)

    Data encrypted in us-east-1 with mrk-key → can be decrypted in eu-west-1 with replica
    Same KMS key material → ciphertext is compatible across regions

  EU data sovereignty with MRK:
    EU PII: encrypted with eu-only-key (regular CMK in eu-west-1, NOT multi-region)
      → Cannot decrypt in us-east-1 (key doesn't exist there)
      → GDPR: encrypted data in us-east-1 = pseudonymous (ICO guidance — encrypted data without key = not PII)

    US data (no residency restriction): encrypted with mrk-us-data-key (multi-region)
      → Can decrypt in either region for DR or active-active serving

  Key rotation:
    KMS automatic annual rotation: rotates key material independently per region
    Multi-region key rotation: primary key auto-rotation propagates to replica keys
    CloudTrail: every kms:Encrypt, kms:Decrypt logged per region

[SECURITY CONTROL PARITY — AWS FIREWALL MANAGER]

Problem: 50 microservices in 2 regions = security controls must be IDENTICAL

AWS Firewall Manager (org-level):
  WAF policies: defined ONCE in Firewall Manager → deployed identically to:
    CloudFront distributions (global)
    ALBs in us-east-1
    ALBs in eu-west-1
    API Gateways in both regions

  Rule groups:
    OWASP rule group (same version both regions)
    Custom IP block list (regional: eu WAF gets EU-specific threat intel, US WAF gets US threat intel)
    Rate limiting: 10,000 req/IP/minute (same rule, both regions)

  Security Group policies: Firewall Manager enforces same SG rules in both regions
    "No 0.0.0.0/0 ingress on port 22" — Firewall Manager auto-remediates non-compliant SGs

  Shield Advanced:
    DDoS protection: both regions
    Health-based detection: integrates with Route 53 ARC health checks
    DRT (DDoS Response Team): single support case covers both regions

  GuardDuty: org-level, delegated admin in security account
    Both regions automatically covered (GuardDuty is region-specific but org enrollment is global)

  AWS Config: same Config rules in both regions (via AWS Organizations)
    Config Conformance Pack: deployed identically (multi-region deployment via CloudFormation StackSets)
    Compliance drift: if eu-west-1 has different Config rule → Security Hub finding → remediate

  CloudFormation StackSets:
    All security infrastructure (WAF, GuardDuty, Config) managed by StackSets
    Single template → deployed to all accounts × all regions
    Version: same template = guarantee of parity

[IDENTITY FEDERATION — MULTI-REGION]

Workforce identity (Okta as IdP):
  Problem: if Okta has an outage → engineers cannot log in via SAML federation

  Solution:
    Primary: Okta → AWS IAM Identity Center (SSO)
    IAM Identity Center: multi-region (stored in one home region, accessible globally)
    Break-glass: separate break-glass IAM users in each region (no SAML)
      - Credentials stored in: physical safe (CEO office) + HashiCorp Vault (on-prem)
      - Use: ONLY if IAM Identity Center is unavailable
      - Quarterly rotation + test

  Cross-region SSO:
    IAM Identity Center home region: us-east-1
    If us-east-1 outage: IAM Identity Center portal unavailable for new logins
    Engineers with active SSO sessions: continue working (sessions valid for 8 hours)
    New logins during us-east-1 outage: use break-glass IAM users

  Service accounts (machine-to-machine):
    EKS IRSA: IAM roles per region (each cluster has its own role — no cross-region dependency)
    Secrets Manager: cross-region replication of service secrets (eu-west-1 replica of secrets)
    If us-east-1 Secrets Manager unavailable: eu-west-1 uses its own replicated secret copy
    Automatic replication: Secrets Manager multi-region replication (update primary → replica auto-updated)

[NETWORK SECURITY]

VPC per region (no peering between us-east-1 and eu-west-1 VPCs):
  us-east-1 VPC: 10.0.0.0/16
  eu-west-1 VPC: 10.1.0.0/16
  NO VPC peering: application services don't talk cross-region (data residency enforcement)

Transit Gateway (within region):
  Each region has its own TGW
  Attached: app VPC, data VPC, security VPC (VPC endpoints, GuardDuty agent traffic)

Private endpoints (no internet for AWS service calls):
  VPC endpoints for: DynamoDB, KMS, Secrets Manager, SSM, S3, Inspector, GuardDuty
  Both regions: identical set of VPC endpoints

Cross-region encryption (for legitimate data sync via AWS backbone):
  S3 Cross-Region Replication: S3 → KMS re-encryption at destination (uses destination region's MRK replica)
  DynamoDB Global Tables replication: uses DynamoDB service internal encryption (AWS-managed, not customer KMS)
    NOTE: for customer-managed key (CMK) on DynamoDB Global Tables: each region needs its own CMK (or MRK)

[REGION FAILOVER SECURITY CHECKLIST]

Before promoting eu-west-1 to sole active region (us-east-1 down):
  Route 53 ARC readiness check MUST PASS:
    ✅ eu-west-1 EKS: >= 200 nodes ready (minimum for 100K TPS)
    ✅ DynamoDB eu-west-1: replication status = ACTIVE, no pending writes from us-east-1
    ✅ Aurora eu-west-1: replication lag < 1 second (or Aurora failover already completed)
    ✅ WAF eu-west-1: rules are up to date (same version as us-east-1)
    ✅ KMS eu-west-1: MRK replica key operational
    ✅ Secrets Manager eu-west-1: all secrets replicated (last replication < 5 minutes)

  Security during failover:
    GuardDuty: continues operating independently in eu-west-1 (not dependent on us-east-1)
    Security Hub: eu-west-1 Security Hub still aggregates to security account eu-west-1 instance
    WAF: already active in eu-west-1 (no additional setup needed)

  Post-failover:
    Notify SOC: us-east-1 regional outage → eu-west-1 sole active region
    Monitor: GuardDuty findings spike? (attacker exploiting the failover?)
    Capacity: eu-west-1 must handle 200K TPS (both US + EU) → pre-warmed auto-scaling

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

IAM role per region:
  EKS IRSA roles: arn:aws:iam::123:role/payment-service-us-east-1
                  arn:aws:iam::123:role/payment-service-eu-west-1
  Each role: condition aws:RequestedRegion restricts API calls to own region

Break-glass accounts:
  One per region (local IAM user, no SAML dependency)
  Password: stored in hardware security device (not AWS Secrets Manager — single point of failure)
  MFA: hardware TOTP device
  Usage: CloudTrail alert on any usage of break-glass account → immediate SOC notification

2. Encryption

TLS everywhere:
  Global Accelerator → ALB: HTTPS (TLS 1.3)
  ALB → EKS: HTTPS (ALB certificate from ACM; same certificate ARN across regions not possible — separate ACM cert per region)
  EKS service mesh: mTLS (Istio/App Mesh — mutual TLS for every microservice call)

Cross-region encryption:
  US data (non-PII): KMS Multi-Region Key (same ciphertext usable in both regions for DR)
  EU PII: eu-only KMS CMK (eu-west-1 only, never replicated to us-east-1)

Secrets Manager replication:
  Secret replicated to eu-west-1 from us-east-1 (read replica)
  Replication uses KMS eu-west-1 key for re-encryption (secret encrypted with eu-west-1 CMK in eu-west-1)

3. Network Isolation

No VPC peering between regions:
  Microservices cannot directly call cross-region services
  Application layer enforces data residency (EU services only call EU endpoints)

AWS PrivateLink: does NOT extend cross-region (one VPC endpoint = one region)
  Each region: independent set of VPC endpoints (no sharing)

Outbound internet (for SaaS services like Stripe for payments):
  Each region: NAT Gateway in each AZ (no single-AZ dependency)
  Route: private subnet → NAT Gateway → Internet Gateway → external API

4. Logging and Audit Trail

CloudTrail multi-region configuration:
  Organization trail: captures all API activity in ALL regions
  S3 bucket: centralized in security account (us-east-1) with CRR to eu-west-1 (DR)
  CloudTrail log integrity: enabled (SHA-256 hash chain — tamper detection)

Security Lake: enabled in both regions
  eu-west-1 Security Lake: stores EU-origin logs (compliance requirement — EU logs in EU)
  us-east-1 Security Lake: stores US-origin logs
  Cross-region Security Lake aggregation: optional (for central SOC view)

5. Least Privilege and Blast Radius

Region-scoped IAM:
  IRSA role: condition aws:RequestedRegion = "eu-west-1"
  Blast radius of eu-west-1 compute compromise: cannot access us-east-1 data

Network-level blast radius:
  No VPC peering: eu-west-1 compromise cannot directly reach us-east-1 network
  Transit Gateway: eu-west-1 TGW is separate from us-east-1 TGW (no cross-region attachment)

Data isolation:
  EU PII encrypted with EU-only key: even if us-east-1 is compromised, attacker cannot decrypt EU PII

6. Compliance Mapping

Requirement Control
GDPR Art. 44 — Cross-border transfer restrictions EU-only KMS CMK for EU PII; DynamoDB separate tables for EU data (no replication to US)
PCI-DSS Req 1.3 — Network security controls Firewall Manager WAF + SG policies identical in both regions; no cardholder data crosses regions
NIST SP 800-53 CP-7 — Alternate processing site Route 53 ARC readiness checks; Global Accelerator failover; Aurora Global DB failover

Scale and Resilience (Phase 4 — 10 min)

100,000 TPS capacity planning:
- EKS: horizontal pod autoscaling (HPA). For 100K TPS: estimate ~500 pods per service per region. Pre-warm: scheduled scaling before peak (predictable load).
- DynamoDB Global Tables: on-demand capacity (auto-scales instantly). 100K TPS = 100K write capacity units/second per region.
- Global Accelerator: 100K TPS is trivial (Global Accelerator handles millions of RPS).
- Pre-warming: if us-east-1 fails and eu-west-1 must handle 200K TPS: auto-scaling pre-warmed to 150% of normal EU capacity at all times.

AZ failure (within region):
- EKS: pods spread across 3 AZs (topology spread constraints: maxSkew=1). AZ failure: 33% capacity reduction → auto-scaling replaces in remaining 2 AZs within 2 minutes.
- Aurora: Multi-AZ within each region (independent of Global DB). AZ failure: Aurora fails over to standby replica in another AZ (< 30 seconds).

Trade-off Analysis — The Hard Choices

Decision 1: Active-Active vs. Active-Passive Multi-Region

Factor Active-Active Active-Passive
RTO Near-zero (traffic already balanced) 1-5 minutes (failover time)
Cost 2x infrastructure cost (both regions fully provisioned) ~1.5x (standby region at reduced capacity)
Complexity High (data synchronization, conflict resolution, data residency) Lower (one-way replication, no write conflicts)
Data residency Complex (must enforce per-request routing) Simpler (EU region is EU only until failover)
Recommendation Active-active for 100K TPS financial app (zero downtime requirement); Active-passive for internal apps with < 99.99% SLA

Decision 2: DynamoDB Global Tables vs. Separate Regional Tables for EU Data

Factor DynamoDB Global Tables Separate Regional Tables
GDPR compliance Requires encryption workaround (EU-only key on replicated data) Clean (EU data never leaves eu-west-1)
DR (eu-west-1 failure) EU data readable from us-east-1 replica (but only if EU-only key accessible) EU data inaccessible during eu-west-1 outage
Operational complexity Lower (one table API, automatic replication) Higher (app must route to region-specific table)
Recommendation Separate regional tables for EU PII (cleanest GDPR compliance); Global Tables for non-PII application state

Decision 3: KMS Multi-Region Key vs. CloudHSM for Cross-Region Encryption

Factor KMS Multi-Region Key CloudHSM Cross-Region
Compliance FedRAMP, PCI-DSS (KMS = FIPS 140-2 Level 2) FIPS 140-2 Level 3 (higher)
Cost ~$1/key/month + usage $1.50/HSM/hour × 2 regions = $2,160/month
Management AWS managed (key replication automatic) Customer managed (HSM clusters, HA setup)
Portability AWS-specific (cannot extract key material) Standard PKCS#11 (key material exportable)
Recommendation KMS Multi-Region Keys for most workloads; CloudHSM only if PCI-DSS Level 3 evidence required or key exportability needed

Common Mistakes to Avoid

  1. Not addressing data sovereignty vs. Active-Active conflict. The inherent tension in this question is: active-active typically means data can be served from either region, but GDPR says EU data cannot leave eu-west-1. Candidates who describe a pure active-active architecture without addressing this contradiction are missing the core challenge. The correct answer acknowledges the conflict and proposes a solution (separate tables, EU-only CMK for encryption, IAM condition restricting EU writes to eu-west-1 only).
  2. Relying solely on Route 53 for instant failover. Route 53 DNS failover has 60-second health check TTL and client DNS caching delays the switch by minutes. For 100K TPS financial app: use Global Accelerator (anycast — failover in seconds, no DNS TTL). Route 53 is supplementary (for human-readable endpoint names).
  3. Ignoring IAM Identity Center failure during regional outage. IAM Identity Center is a regional service. If its home region (us-east-1) fails: engineers cannot log in via SSO. Without break-glass accounts pre-configured, the operations team is locked out of the remaining healthy region during a crisis. This is a critical design requirement.
  4. Not distinguishing KMS Multi-Region Keys from regular CMK replication. Standard KMS CMKs CANNOT encrypt/decrypt cross-region (each CMK exists in one region only). KMS Multi-Region Keys have the same key material in multiple regions — ciphertext encrypted in us-east-1 can be decrypted in eu-west-1. This is a specific feature candidates should name explicitly.

Likely Follow-up Questions

Q1: How do you handle Aurora database writes during a region failover?
Aurora Global Database failover: (1) Normal operation: us-east-1 primary accepts all writes (both US and EU writes for their respective clusters). eu-west-1 secondary: read-only replica (queries served locally, < 1 second lag). (2) us-east-1 failure: Aurora Global DB automated failover: Step 1: disconnect secondary from primary (promote eu-west-1 secondary to primary — ~1 minute). Step 2: application (EKS) Aurora connection: update connection string from primary endpoint to new regional writer endpoint. Implementation: application uses custom endpoint (Aurora custom endpoint wraps the current writer — endpoint doesn't change after failover). Step 3: all writes now go to eu-west-1 Aurora instance. (3) Data loss risk during failover: if primary fails with 0.8 second replication lag: up to 0.8 seconds of US transactions may be lost. Mitigation: Aurora Global DB forward replication log — can be replayed on the new primary to recover the last committed transactions (if not available, RPO = replication lag at time of failure). (4) Post-failover: when us-east-1 recovers: comes back as a secondary. New primary = eu-west-1. To return to original topology: manual failback (after validating data consistency). Monitor: CloudWatch metric AuroraGlobalDBReplicatedWriteIO confirms replication is active.

Q2: How do you ensure security patches are applied identically across both regions?
Multi-region patch parity via GitOps: (1) EC2 Image Builder pipeline: runs in both regions (separate pipeline per region, same Image Builder component documents stored in S3 CRR — identical components). When a new golden AMI is built (weekly), both us-east-1 and eu-west-1 pipelines trigger automatically. AMI distribution: EC2 Image Builder cross-region copy ensures AMI is available in both regions. (2) EKS container images: ECR with cross-region replication. When a new patched container image is pushed to us-east-1 ECR: ECR replication rule automatically copies to eu-west-1 ECR. ArgoCD in eu-west-1: detects new image digest → deploys updated pods (same GitOps process as us-east-1). (3) AWS managed services (Aurora, Lambda runtimes): AWS applies patches automatically — no customer action needed. Both regions receive updates on the same schedule. (4) Config rule parity: Config Conformance Pack deployed via StackSets to both regions. Same rules, same thresholds. Compliance drift: if eu-west-1 Config shows a compliant resource while us-east-1 shows non-compliant (or vice versa) → Security Hub finding → investigate why the same resource has different compliance in different regions. (5) Firewall Manager WAF rules: update WAF policy ONCE in Firewall Manager → deployed to BOTH regions' ALBs simultaneously (< 5 minute propagation). Zero possibility of WAF rule version drift between regions.

Q3: How do you test the regional failover process without impacting production?
Failover testing without production impact: (1) Route 53 ARC zonal shift: shift load away from one AZ (not the full region) → validate application handles reduced capacity. Non-destructive, reversible in seconds. Run monthly. (2) Chaos Engineering (non-production): in a staging environment (identical architecture): inject us-east-1 failure: terminate all EKS nodes in us-east-1 staging cluster. Validate: Global Accelerator routes all traffic to eu-west-1 staging within 30 seconds. DynamoDB: verify eu-west-1 can serve reads from its replica. Aurora: verify failover completes within target RTO. (3) Production read traffic test: temporarily route 1% of production READ traffic to eu-west-1 (using Global Accelerator endpoint weights: us-east-1=99, eu-west-1=1). Validate: eu-west-1 serves traffic correctly. Monitor: error rates, latency. This tests the data path without testing writes. (4) Aurora Global DB failover drill: during a planned maintenance window: manually initiate Aurora failover to eu-west-1. Measure: time from failover initiation to eu-west-1 accepting writes. Measure: transaction error rate during failover window. Rollback: failback to us-east-1 after test. (5) Runbook validation: at least quarterly, SOC team walks through the failover runbook step-by-step (without executing). Identify: any steps outdated (new services added?), any missing pre-checks, any ambiguous instructions.

Q6: How do you handle conflicting writes in DynamoDB Global Tables (active-active)?
DynamoDB Global Tables conflict resolution: (1) DynamoDB Global Tables conflict resolution algorithm: last-writer-wins (LWW) based on timestamp (DynamoDB uses write timestamps for conflict resolution). If two regions write the same item simultaneously: the write with the LATER timestamp wins. The earlier write is discarded (this is eventually applied to all regions). (2) Business logic implications: for financial transactions (e.g., deducting account balance): last-writer-wins can lead to incorrect balance. Example: account balance = $100. us-east-1 writes: balance = $75 (deducted $25) at T1. eu-west-1 writes: balance = $50 (deducted $50) at T2 > T1. Conflict: LWW = eu-west-1 wins (T2 later). Result: balance = $50. The $25 deduction from us-east-1 is LOST. Final balance should be $25 ($100 - $25 - $50). This is a PROBLEM for financial data. (3) Solution for financial transactions: use conditional writes (optimistic locking): DynamoDB condition expression: balance >= transactionAmount AND version = :expected_version. If conflict: conditional write fails → retry with fresh read. For idempotency: include transaction_id in the write → duplicate transaction detected (same transaction_id already exists → don't apply again). (4) Architecture recommendation: active-active writes for idempotent operations only (session state, UI preferences — conflicts don't matter). For financial records (non-idempotent): use SINGLE active region per customer partition (EU customers write to eu-west-1 only; US customers write to us-east-1 only). This is active-active for serving (reads from local region) but single-writer per partition (no write conflicts). (5) Aurora (relational): Aurora Global Database is NOT active-active for writes (only one primary per cluster). Write routing: all financial record writes → respective region's Aurora primary. No write conflict possible (single writer).
