Q24 — Compliance-as-Code for PCI-DSS on AWS

The Ask

"Design a Compliance-as-Code platform for a company that must maintain continuous
PCI-DSS v4.0 compliance across 30 AWS accounts. The platform must automate evidence
collection for QSA audits, enforce IaC security policies before deployment, detect
configuration drift in real time, and provide a compliance dashboard for the CISO.
The company runs Terraform for infrastructure and deploys via a CI/CD pipeline."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What scope of PCI-DSS applies — only the Cardholder Data Environment (CDE), or are all 30 accounts in scope?
    Impact: PCI-DSS applies to CDE accounts AND accounts that can impact CDE security. A typical org: 3-5 accounts in CDE scope + 10-15 "connected-to-CDE" accounts. All 30 accounts may have baseline requirements (PCI-DSS network segmentation, logging) but the strictest requirements apply only to CDE accounts. This drives the compliance enforcement strategy.

  • Q: Is there a QSA engaged for annual assessment, and what is their preferred evidence format?
    Impact: Some QSAs accept AWS Audit Manager automated evidence directly (native integration). Others require spreadsheet export or NIST-format control evidence. AWS Audit Manager generates evidence in JSON + PDF — most QSAs accept this. Understanding the QSA workflow prevents building custom evidence formats.

  • Q: What Terraform version and state backend is in use, and is there a central CI/CD platform (GitHub Actions, Jenkins, GitLab CI)?
    Impact: Terraform >= 1.x + GitHub Actions: use tfsec + Checkov + OPA Rego policies as GitHub Actions workflow steps. Policy failures block the PR. Terraform Cloud / Sentinel: Sentinel policies (HashiCorp-native policy framework) can enforce compliance. Jenkins: different pipeline hooks. The policy enforcement integration differs significantly.

  • Q: What is the organization's current approach to exceptions — do they have a formal exception process, and how are exceptions tracked?
    Impact: Every PCI-DSS compliance program needs an exception process (some deviations are business-justified). The compliance platform must: track exceptions (who approved, what risk is accepted, when it expires), exclude exceptions from compliance dashboards (else dashboard shows permanent red), and feed exceptions to the QSA as documented compensating controls.

  • Q: Is there a desire for preventive controls (block non-compliant deployments) or just detective controls (alert on drift)?
    Impact: Preventive (preferred): Terraform policy checks in CI/CD + SCPs. Detective: AWS Config + Security Hub. For PCI-DSS: BOTH are required (preventive reduces drift, detective catches manual changes). Starting with detective-only is a common phased approach (enable without breaking existing deployments first).

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Terraform + GitHub Actions for CI/CD.
  2. AWS Audit Manager for automated evidence collection.
  3. AWS Config + Security Hub for runtime compliance detection.
  4. OPA (Open Policy Agent) / Rego for IaC policy enforcement in CI/CD.
  5. Config Conformance Pack for per-account PCI-DSS baseline.

Core Architecture — Data Flow Narrative

[LAYER 1: PREVENTIVE — IaC POLICY ENFORCEMENT IN CI/CD]

Developer workflow:
  git push → GitHub PR → GitHub Actions pipeline triggers

GitHub Actions workflow:
  Step 1: tfsec (static analysis)
    tfsec --format=json --minimum-severity=HIGH terraform/
    Example finding: "aws_s3_bucket - Bucket does not have encryption enabled"
    GitHub Actions: if tfsec finds CRITICAL or HIGH → FAIL the PR (cannot merge)

  Step 2: Checkov (CIS/PCI-DSS rule set)
    checkov -d terraform/ --framework terraform --check CKV_AWS_* --compact --output json
    PCI-DSS specific checks:
      CKV_AWS_2 (ALB: HTTPS only, no HTTP listeners)
      CKV_AWS_34 (CloudFront: HTTPS only)
      CKV_AWS_130 (VPC: VPC flow logs enabled)
      CKV_AWS_18 (S3: access logging enabled)
      CKV_AWS_119 (DynamoDB: encryption enabled)
    Blocking: any PCI-DSS required check failure → FAIL PR

  Step 3: OPA/Conftest (custom organizational policies)
    conftest test terraform/ --policy policies/pci-dss/ --output json

    Custom PCI-DSS Rego policies:

    # Policy: all EC2 in CDE must use approved AMI
    package pci.ec2
    deny[msg] {
      resource := input.resource.aws_instance[name]
      not startswith(resource.ami, "ami-golden-")  # must use golden AMI
      msg := sprintf("EC2 instance %v must use approved golden AMI", [name])
    }

    # Policy: no security group allows 0.0.0.0/0 ingress in CDE accounts
    package pci.networking
    deny[msg] {
      resource := input.resource.aws_security_group_rule[name]
      resource.type == "ingress"
      resource.cidr_blocks[_] == "0.0.0.0/0"
      msg := sprintf("SG rule %v: 0.0.0.0/0 ingress is not PCI compliant", [name])
    }

    # Policy: all RDS instances must have deletion_protection = true
    package pci.rds
    deny[msg] {
      resource := input.resource.aws_db_instance[name]
      not resource.deletion_protection
      msg := sprintf("RDS %v must have deletion_protection=true", [name])
    }

    # Policy: CloudTrail log encryption must use KMS CMK (not SSE-S3)
    package pci.cloudtrail
    deny[msg] {
      resource := input.resource.aws_cloudtrail[name]
      not resource.kms_key_id
      msg := sprintf("CloudTrail %v must use KMS CMK for log encryption", [name])
    }

  Step 4: Terraform plan security scan
    terraform plan -out=plan.tfplan
    tfsec --scan-module-links --include-passed plan.tfplan

  Step 5: SBOM for Terraform providers
    providers.lock file: verify all Terraform providers match pinned hashes
    New provider version: requires explicit approval (supply chain security)

  Blocking vs. Warning policy:
    CRITICAL/required PCI controls → block PR
    Advisory controls → warning comment on PR (cannot block — informational)

  Exception handling in CI/CD:
    Terraform resource tag: pci_exception = "approved" + exception_id = "EXC-2024-001"
    OPA policy: if resource has pci_exception tag with valid exception ID in DynamoDB exceptions table → skip check
    Exception DynamoDB: { exceptionId, control, approvedBy, expiryDate, riskAcceptance }

AWS Config (all 30 accounts, Config Aggregator in audit account):

  Config rule deployment:
    AWS Organizations + CloudFormation StackSets:
      Deploy Config Conformance Pack "Operational Best Practices for PCI DSS 4" to all 30 accounts

  PCI-DSS Config rules (select key ones):

    pci-dss-4-cloudtrail-log-file-validation: log file integrity validation enabled
    pci-dss-4-encrypted-volumes: all EBS volumes encrypted
    pci-dss-4-rds-encryption-enabled: all RDS instances encrypted
    pci-dss-4-s3-server-side-encryption-enabled: all S3 buckets encrypted
    pci-dss-4-restricted-ssh: no SG allows 0.0.0.0/0 on port 22
    pci-dss-4-restricted-rdp: no SG allows 0.0.0.0/0 on port 3389
    pci-dss-4-vpc-flow-logs-enabled: VPC flow logs enabled in all VPCs
    pci-dss-4-mfa-enabled-for-iam-console-access: all IAM users with console access have MFA
    pci-dss-4-access-keys-rotated: IAM access keys rotated within 90 days
    pci-dss-4-password-policy: IAM password policy meets PCI requirements (8+ chars, complexity)

  Custom Config rules (Lambda-backed):
    pci-cde-only-approved-instance-types: CDE account EC2 must use approved instance types
      Lambda: check if instance type in approved_types list (no GPU instances in CDE — not needed + higher risk)
    pci-cardholder-data-not-in-logs: Macie integration — verify no PAN in CloudWatch Logs
    pci-no-root-account-activity: SNS alert if root account used

  Config Aggregator (audit account):
    Aggregates compliance data from all 30 accounts
    Athena-queryable: compliance snapshots for trend analysis

  Non-compliant resource auto-remediation:
    Config rule: restricted-ssh
    Remediation: SSM Automation document AWS-DisableIncomingSSHOnPort22
      → automatically modifies the SG to remove port 22 0.0.0.0/0 rule

    Config rule: s3-server-side-encryption-enabled
    Remediation: SSM Automation → put bucket encryption configuration

    NOT auto-remediated (manual review required):
      Any change that could break production (e.g., modifying SGs on production EKS nodes)
      Exception: only auto-remediate dev/test accounts (not production CDE accounts)

[LAYER 3: CONTINUOUS COMPLIANCE REPORTING — SECURITY HUB]

Security Hub (audit account, aggregated):
  PCI-DSS standard: Security Hub → Enable PCI DSS standard
    400+ controls across all PCI requirements
    Per-control status: PASSED/FAILED/NOT_AVAILABLE + compliance score

  Aggregation:
    All 30 member accounts forward Security Hub findings to audit account
    Cross-region aggregation: enabled (us-east-1 + eu-west-1 findings in one view)

  Custom Security Hub insight:
    CISO dashboard insight:
      "PCI-DSS Compliance Score by Account"
        Filter: ProductName = Security Hub, ComplianceStatus = FAILED
        Group by: AwsAccountId
        Metric: count of FAILED controls per account

    "PCI-DSS Critical Controls Failing"
        Filter: Severity = CRITICAL, ComplianceStatus = FAILED, ProductName = SecurityHub
        Grouped: by Control ID

  EventBridge: Security Hub FAILED finding → SQS → Lambda
    Lambda: check if this control is newly failed (was PASSED before)?
    If newly failed: create Jira ticket (tag: pci-drift) + Slack alert to #pci-compliance
    If persistently failed (acknowledged exception): update DynamoDB exception tracker

[LAYER 4: AUTOMATED EVIDENCE COLLECTION — AWS AUDIT MANAGER]

AWS Audit Manager:
  Framework: PCI DSS v4.0 (AWS-managed framework)
    400+ automated controls mapped to PCI requirements

  Assessment scope:
    Accounts: all 30 accounts in scope
    Services: EC2, RDS, S3, CloudTrail, IAM, VPC, KMS, ALB, GuardDuty, Config

  Evidence collection (automated):
    Config compliance snapshots: captured daily → stored in Audit Manager evidence folder
    CloudTrail events: specific API calls mapped to PCI controls (e.g., kms:EnableKey → Req 3.5 evidence)
    Security Hub findings: compliance assessment results → Audit Manager evidence
    AWS Config rule evaluations → evidence for each config-related PCI control

  Manual evidence collection:
    Penetration testing report: upload PDF → linked to PCI Req 11.4
    Network diagrams: upload → linked to PCI Req 1.1
    Approved compensating controls: upload risk acceptance → linked to specific requirement

  Evidence folder structure:
    PCI DSS v4.0 > Req 1 (Network) > Control 1.3.1 > Evidence:
      - aws_config_vpc_flow_logs_enabled_2024-01-15.json (automated)
      - aws_config_restricted_ssh_2024-01-15.json (automated)
    PCI DSS v4.0 > Req 3 (Protection of Stored Data) > Control 3.5.1 > Evidence:
      - aws_config_rds_encryption_enabled_2024-01-15.json (automated)
      - aws_config_encrypted_volumes_2024-01-15.json (automated)
      - kms_key_rotation_cloudtrail_2024-01-15.json (automated from CloudTrail)

  Assessment report generation (for QSA):
    Audit Manager: generate assessment report (PDF/CSV)
    Report includes: control ID, status (PASS/FAIL), evidence items, timestamp, resource ARN
    QSA: receives 90-day assessment window report automatically

  Evidence readiness:
    Audit Manager Insights dashboard: "Ready for audit" evidence percentage
    Target: >95% of controls have automated evidence (< 5% require manual collection)

[EXCEPTION MANAGEMENT WORKFLOW]

Exception request:
  Team submits: Jira ticket (pci-exception type)
    Fields: affected resource ARN, PCI control ID, business justification, compensating controls, expiry date

  Approval workflow:
    Step Functions: engineering-lead review (5 days) → CISO approval
    For CDE exceptions: also requires QSA acknowledgment (upload signed exception form)
    Approved: stored in DynamoDB exceptions-registry

  Exception suppression:
    Config rule: custom Lambda-backed rule: if resource ARN in exceptions-registry AND exception not expired → COMPLIANT
    Security Hub: findings suppression rule: suppress FAILED finding for specific resource if exception active
    CI/CD OPA: exception_id tag → skip corresponding policy check

  Exception tracking:
    Daily Lambda: scan DynamoDB exceptions-registry → find exceptions expiring in 14 days
    Email: exception owner + CISO: "PCI exception EXC-2024-001 expires in 14 days. Review and renew or remediate."
    Expired exceptions: immediately re-appear as FAILED in Security Hub and Config

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Compliance platform roles:
  audit-manager-role: AuditManagerAdmin (Audit Manager built-in) + read-only in all accounts
  config-aggregator-role: config:BatchGetAggregateResourceConfig (read-only)
  security-hub-aggregator-role: securityhub:GetFindings (read-only)

  Separation of duties (PCI-DSS Req 7):
    ci-cd-compliance-role: can read Config + Security Hub findings but CANNOT remediate
    compliance-remediation-role: can execute SSM Automation documents but CANNOT change policies
    compliance-admin-role: can modify Config rules + create exception workflows

  No developer access to compliance platform:
    Developers: can see their own account's compliance status (via Security Hub member account view)
    Developers: cannot modify Config rules or suppress Security Hub findings
    Modification of compliance rules: requires compliance-admin-role (restricted to compliance team)

2. Encryption

Audit Manager evidence: SSE-KMS (audit-evidence-key)
  Key policy: only Audit Manager service principal + CISO + QSA-reviewer-role can decrypt

Config snapshots: SSE-KMS on Config delivery S3 bucket
CloudTrail: SSE-KMS (already required by PCI-DSS Req 10.5)
Security Hub findings: stored encrypted within Security Hub service (AWS-managed)

IaC compliance:
  Terraform state: S3 bucket with SSE-KMS + DynamoDB state lock
  Terraform state read: only CI/CD role + infrastructure team (Terraform state contains resource IDs)

3. Network Isolation

Compliance monitoring traffic:
  Config Agent: uses VPC endpoint for Config (no internet)
  Audit Manager: AWS-managed service (no network configuration needed by customer)
  Security Hub: VPC endpoint available (uses it in CDE accounts)

GitHub Actions (CI/CD):
  GitHub Actions runners: AWS-hosted or self-hosted in private VPC
  Self-hosted runners (preferred for CDE): runners in compliance VPC (no internet, VPC endpoints for GitHub API)
  tfsec/Checkov/conftest: run in container in runner — no external calls during scan

4. Logging and Audit Trail

Compliance platform audit trail:
  Every Config rule change: CloudTrail (putConfigRule, deleteConfigRule)
  Every Security Hub control enablement/suppression: CloudTrail + Security Hub audit log
  Every exception approved in DynamoDB: Step Functions execution history + CloudTrail
  Every Audit Manager evidence collection: Audit Manager activity log

Immutable audit trail:
  CloudTrail → S3 Object Lock WORM (PCI-DSS requires log tamper protection)
  Audit Manager evidence folder: S3 Object Lock WORM (evidence cannot be deleted or modified)

QSA access to audit trail:
  QSA-reviewer-role: read-only access to Audit Manager assessment + evidence S3 bucket
  Time-bound: STS session credential for QSA (active only during assessment period)
  CloudTrail: logs QSA access (what evidence they accessed + when)

5. Least Privilege and Blast Radius

Compliance drift auto-remediation scope:
  ONLY auto-remediate: specifically safe, reversible actions (add bucket encryption, remove permissive SG rule)
  NEVER auto-remediate: any action that could cause production disruption
  Production accounts: remediation = human approval required (via Step Functions + Slack)
  Dev accounts: fully automated remediation

Compromise of compliance platform:
  If audit-manager-role is compromised: attacker has read-only access to compliance data
  Cannot: modify resources, delete evidence, change Config rules
  Impact: data exposure (compliance gaps visible to attacker) — not a system compromise

Trade-off Analysis — The Hard Choices

Decision 1: Preventive (Block in CI/CD) vs. Detective (Alert on Drift) — Which to Implement First?

Factor Preventive First Detective First
Existing non-compliant resources Cannot block new deployments to fix existing issues Reveals all existing issues immediately
Risk of false positives Blocks legitimate deployments if rules misconfigured Only generates alerts (doesn't disrupt)
Compliance maturity required High (requires tuned rules before enabling) Low (can enable immediately)
Recommendation Detective first for 60 days (discover all gaps, tune rules, reduce false positives) → then enable preventive (CI/CD block) after rules are validated. Run both in parallel permanently.

Decision 2: AWS Security Hub PCI Standard vs. Third-Party CSPM (Prisma Cloud, Wiz)

Factor Security Hub PCI Standard Third-Party CSPM (Wiz, Prisma Cloud)
PCI-DSS coverage 400+ controls (good coverage) Often more controls + CIS Benchmarks + custom
AWS integration Native (no API keys needed) Requires API/role integration
Multi-cloud AWS only AWS + Azure + GCP
Cost Included in Security Hub pricing $50K-$200K+/year
Recommendation Security Hub as primary (cost-effective, native) + Wiz or Prisma Cloud if multi-cloud compliance or advanced attack path analysis is needed

Decision 3: OPA/Rego vs. Sentinel vs. Checkov for Terraform Policy Enforcement

Factor OPA/Rego HashiCorp Sentinel Checkov
Open source Yes No (Terraform Cloud/Enterprise) Yes
Flexibility Very high (any policy) High (but Terraform-specific) Pre-built CIS/PCI rules
Learning curve High (Rego syntax) Medium Low
Custom policy support Full Full Limited (Python custom rules)
Recommendation Checkov for fast PCI-DSS coverage (pre-built rules); OPA/Rego for complex custom organizational policies; Sentinel only if already using Terraform Enterprise

Common Mistakes to Avoid

  1. Not distinguishing PCI-DSS "in scope" vs "connected to CDE." Treating all 30 accounts identically is expensive and unnecessary. CDE accounts (3-5): strictest controls, no auto-remediation exceptions. Connected accounts: medium controls. Out-of-scope accounts: baseline security only. Scoping is a PCI-DSS design decision — demonstrate awareness.
  2. Ignoring the exception management workflow. PCI-DSS in practice always has legitimate exceptions (vendor hardware that can't be patched, legacy systems). A compliance platform without exception management will show 100% red forever because not all findings are fixable immediately. The exception process (approval, documentation, expiry) is what makes compliance sustainable.
  3. Treating AWS Audit Manager as optional. Audit Manager is the KEY service for QSA audit readiness — it automates evidence collection into a QSA-ready format. Candidates who design a compliance platform using only Config + Security Hub and say "we'll manually collect evidence" are significantly understating the QSA audit preparation effort (weeks of manual evidence collection vs. automated).
  4. Forgetting that Security Hub PCI Standard and Config Conformance Packs overlap. Both should be used (Security Hub provides the 400+ control view + aggregation; Config provides the underlying compliance data + auto-remediation). They complement each other — Security Hub USES Config data for its control evaluations.

Likely Follow-up Questions

Q1: How do you handle a new PCI-DSS v4.0 requirement that has no existing AWS Config rule?
Custom Config rule development: (1) When a new PCI-DSS requirement doesn't have an AWS-managed Config rule: write a Lambda-backed custom Config rule. Config calls the Lambda on: resource creation/change, periodic evaluation (daily), or on-demand. (2) Example: PCI-DSS v4.0 Req 12.6.1 — Security awareness training program documentation (non-technical control, no AWS resource). Custom Config rule: Lambda checks: does the "security-training-current-year" SSM Parameter exist? (teams update this parameter after completing training). If parameter is missing or older than 1 year: NON_COMPLIANT. (3) Technical example: PCI-DSS Req 8.3.9 — Access tokens expire within 15 minutes for CDE. Custom Config rule: Lambda: call cognito:DescribeUserPoolClient for CDE user pools → check AccessTokenValidity <= 15 minutes. If any CDE user pool has longer token validity: NON_COMPLIANT. (4) Publish to AWS Config rule repository: contribute as Config Custom Rule to internal catalog (reusable across accounts). Deploy via StackSets. (5) Test the rule: create a known-compliant resource + known-non-compliant resource in a test account → verify rule evaluates correctly → enable in production.

Q2: How do you maintain compliance after a major AWS service update changes behavior?
Compliance resilience to AWS changes: (1) AWS managed Config rules: AWS maintains the rule logic. When AWS updates a managed rule (e.g., new PCI-DSS v4.0 requirements): the updated rule is deployed automatically. Watch for: AWS Config rule changelog (AWS publishes updates to managed rules). (2) Checkov/tfsec updates: pin specific versions in CI/CD: checkov==3.1.x. When updating to a new version: run in "warning mode" first (don't block PR) → review new findings → tune exceptions → enable blocking mode. (3) Security Hub PCI standard updates: new controls automatically appear when AWS adds them. New controls start as "NOT_AVAILABLE" → transition to PASSED/FAILED as resources are evaluated. Monitor: Security Hub controls summary weekly → identify any new controls in FAILED state. (4) Breaking changes: if an AWS service update (e.g., S3 default encryption behavior change) causes a previously-passing control to FAIL: Config immediately detects the change and marks resources as NON_COMPLIANT. Alert: Jira ticket auto-created. Investigation: is this a genuine compliance gap or a rule interpretation change? (5) Annual PCI-DSS standard review: before annual QSA assessment: run full Security Hub PCI standard assessment → review all FAILED controls → determine if any are new or changed since last year → brief CISO on scope changes.

Q3: How do you prove to a QSA that the compliance evidence hasn't been tampered with?
Evidence integrity and chain of custody: (1) Audit Manager evidence: stored in S3 with Object Lock (WORM mode, COMPLIANCE class). Neither the compliance team nor AWS can modify or delete evidence once stored. QSA can verify: S3 Object Lock retention configuration. (2) CloudTrail log integrity validation: CloudTrail generates SHA-256 digest files every hour. Each digest file contains hashes of all log files in that period. CloudTrail provides validate-logs CLI command: aws cloudtrail validate-logs --trail-arn <arn> --start-time <start>. QSA can run this command to verify no log files were modified. (3) Config configuration snapshots: Config stores JSON snapshots at each change + daily. S3 Object Lock: configuration snapshots immutable. SHA-256 hash of each snapshot stored in Config metadata. (4) Audit Manager assessment immutability: once an assessment report is generated, it's signed (Audit Manager generates a PDF with a unique assessment ID + generation timestamp). Any regenerated report would have a different timestamp — QSA can verify the original report timestamp matches the assessment period. (5) QSA access logs: CloudTrail logs every action taken by the QSA-reviewer-role during their assessment period. QSA receives the access log as part of the assessment package — this proves what they accessed (no hidden modifications). Q4: How do you handle Terraform modules that are shared across PCI-in-scope and out-of-scope accounts?

Q8: How do you conduct a scope reduction exercise to minimize PCI-DSS CDE surface area?
