Q25 — Security Lake and Threat Intelligence Integration Platform

The Ask

"Design a Security Lake and Threat Intelligence Integration Platform on AWS for an
organization with 50 accounts, 500TB of security log data per year, and a need to
correlate threat intelligence from 10 different feeds (commercial, open source, and
government ISAC) with internal security telemetry. The platform must enable SIEM queries
in under 30 seconds, support threat hunting, feed automatic IOC blocking to WAF and
GuardDuty, and retain data for 7 years with tiered storage."

Clarifying Questions I Would Ask First (Phase 1 — 5 min)

  • Q: What is the query engine of choice — Amazon OpenSearch (Elasticsearch-compatible), Athena + S3, or do you need SQL-based BI tooling?
    Impact: OpenSearch: sub-second queries, real-time ingestion, good for SOC alert triage (Kibana dashboards). Athena + S3: serverless, cheap for infrequent queries, 30-second query requirement is achievable with proper partitioning. Amazon Security Lake is Athena-native (OCSF format in S3). If < 30 second is the requirement and data volume is 500TB: Athena + proper partitioning + S3 Select can achieve this.

  • Q: What are the 10 threat intelligence feeds — are any feeds real-time streaming (like a commercial TI feed with webhooks) or all batch/scheduled updates?
    Impact: Real-time feeds (STIX/TAXII push): Lambda webhook receiver → immediate IOC extraction → WAF IP set update. Batch feeds (CSV/JSON downloaded daily): Lambda scheduled → S3 → Glue ETL → normalized IOC table. Mixed approach needed for different feed types.

  • Q: What is the current SIEM in use, and is the goal to replace it or augment it?
    Impact: Replace SIEM with Security Lake + OpenSearch: architectural shift (SOC team retraining required). Augment existing Splunk/QRadar: Security Lake as a long-term cold store + threat correlation layer, Splunk as the hot SOC tier. This dramatically affects architecture and cost.

  • Q: What is the organizational definition of "threat hunting" — ad-hoc analyst queries, scheduled hypothesis hunts, or automated hunt playbooks?
    Impact: Ad-hoc queries: analysts need a self-service query interface (Athena in AWS Console or SageMaker Studio for Jupyter-based hunting). Automated hunts: Step Functions + Lambda hunt playbooks running on schedule. Both: Amazon OpenSearch Security Analytics dashboards + Athena for deep historical analysis.

  • Q: Are there data residency requirements for the security logs themselves (EU logs must stay in EU)?
    Impact: Security Lake is regional — data stays in the S3 bucket of that region. Cross-region aggregation: Security Lake can aggregate to a rollup account, but EU log data will traverse to the aggregation region (GDPR concern). Solution: separate Security Lake deployments per jurisdiction with restricted cross-region aggregation.

Initial Architecture Design (Phase 2 — 15 min)

Design Assumptions

  1. Amazon Security Lake as the OCSF-normalized data lake.
  2. Athena for threat hunting (< 30 second queries with proper partitioning).
  3. Lambda-based threat intelligence ingestion pipeline.
  4. GuardDuty + WAF as automated blocking destinations for IOCs.
  5. 7-year retention with S3 Intelligent-Tiering + Glacier.

Core Architecture — Data Flow Narrative

[AMAZON SECURITY LAKE — OCSF DATA LAKE FOUNDATION]

Amazon Security Lake (org-level, subscriber account):
  Enabled in all 50 accounts → Security Lake collects:
    CloudTrail (management + data events): all API calls across 50 accounts
    VPC Flow Logs: network traffic metadata
    Route 53 DNS query logs: domain lookups
    AWS WAF logs: web request details
    Security Hub findings (normalized)
    GuardDuty findings (normalized)

  OCSF (Open Cybersecurity Schema Framework):
    All sources normalized to OCSF format
    Common schema: event_type, time, src_ip, dst_ip, user, resource, severity
    Benefit: query all log types with same SQL schema (no per-source query syntax)

  Storage (S3 Data Lake):
    Partition structure (optimized for query performance):
      s3://security-lake-123/aws/CloudTrail/us-east-1/year=2024/month=01/day=15/hour=14/
      Partition pruning: WHERE time_dt BETWEEN ... limits Athena scan to relevant partitions

    Format: Apache Parquet (columnar — 10x better query performance vs. JSON)
    Compression: Zstandard (better compression ratio than Snappy for log data)

  Data tiering (S3 Lifecycle):
    Hot (0-90 days): S3 Standard (frequent SOC queries)
    Warm (90-365 days): S3 Infrequent Access (monthly compliance reports)
    Cold (1-7 years): S3 Glacier Instant Retrieval (yearly compliance, regulatory)
    Delete: at 7 years + 1 day (S3 Object Lock minimum retention must be set to 7 years)

  S3 Object Lock (WORM):
    Compliance mode: 7-year retention (legal admissibility of security logs)
    Prevents: accidental deletion + malicious insider deletion of audit logs

  Subscriber model:
    OpenSearch subscriber: Security Lake → Kinesis Firehose → OpenSearch (hot tier for SOC)
    Athena subscriber: direct Athena queries on S3 (no additional data movement)
    SIEM subscriber (if Splunk): SQS subscription → Splunk SQS-based S3 import

[THREAT INTELLIGENCE INGESTION PIPELINE]

Feed types and ingestors:

  Type 1: STIX/TAXII commercial feeds (Recorded Future, CrowdStrike Intel):
    TAXII client Lambda (scheduled every 15 minutes):
      from taxii2client import Server
      server = Server("https://feed.recordedfuture.com/v1/", user=..., password=Secrets Manager)
      collection = server.api_roots[0].collections[0]
      stix_objects = [o for o in collection.get_objects()]

    Lambda: extract IOC from STIX objects:
      ip_indicators: { value: "185.x.x.x", type: "ip", confidence: 90, tlp: "white",
                       expiry: "2024-06-01", source: "RecordedFuture", last_seen: ISO8601 }
      domain_indicators: { value: "malware-c2.ru", type: "domain", confidence: 85 }
      hash_indicators: { value: "sha256:e3b0c4...", type: "file_hash", confidence: 95 }

    Output → DynamoDB IOC table (ioc_registry):
      Partition key: value (IP/domain/hash), Sort key: source
      TTL: expiry timestamp (auto-expire stale IOCs)

  Type 2: Government/ISAC feeds (FS-ISAC, DHS CISA):
    Lambda scheduled (daily): download CSV/JSON
    Lambda: parse + normalize to IOC schema
    Deduplication: DynamoDB conditional write (don't overwrite higher-confidence existing entry)

  Type 3: Open source feeds (abuse.ch, MISP, AlienVault OTX):
    Lambda: scheduled download of CSV/JSON indicator lists
    Rate limits: respect feed provider rate limits (exponential backoff)
    Filters: minimum confidence = 50% (don't ingest low-confidence indicators)

  IOC enrichment:
    Lambda: for each new IP IOC → Shodan/VirusTotal enrichment:
      shodan.host(ip): what services is this IP running?
      VT: any antivirus detections for this IP?
      Enrich: add ASN, geolocation, ISP, registrar to IOC record

  IOC normalization:
    DynamoDB IOC table structure:
    {
      ioc_value: "185.x.x.x",            // partition key
      ioc_type: "ip",                      // ip | domain | hash | url
      confidence: 90,                      // 0-100
      threat_category: ["C2", "malware"], // MITRE ATT&CK tactic/technique
      tlp: "white",                        // TLP marking (green/amber/red/white)
      sources: ["RecordedFuture", "FS-ISAC"],
      first_seen: "2024-01-10T00:00:00Z",
      last_seen: "2024-01-15T09:00:00Z",
      expiry_ttl: 1710000000,              // DynamoDB TTL (Unix timestamp)
      active: true
    }

[IOC TO BLOCKING — AUTOMATED ENFORCEMENT]

IOC Detection Lambda (triggered by DynamoDB Streams on IOC table):
  New IOC added or confidence updated → trigger blocking workflow

  WAF IP Set updates (for IP IOCs with confidence >= 80):
    aws wafv2 update-ip-set --name ThreatIntelBlock --scope REGIONAL
      --addresses ["185.x.x.x/32"]
    Update in ALL 50 accounts (Lambda: sts:AssumeRole per account → update WAF IP set)
    Firewall Manager: manages WAF centrally → update IP set ONCE → propagates to all accounts

    Rate limiting: WAF IP sets have max 10,000 IPs. If IOC feed adds 50K IPs/day:
      Priority: confidence >= 90 → WAF block
      Confidence 70-89 → WAF count (log but don't block) → review before promoting to block
      Confidence < 70 → no WAF action (log to Security Lake for threat hunting only)

  GuardDuty Threat Intelligence (for IP + domain IOCs):
    GuardDuty custom threat list (TXT file of IPs, one per line):
      Lambda: generate IP list from DynamoDB (confidence >= 80) → upload to S3
      GuardDuty: update-threat-intel-set with S3 file path
      GuardDuty: will generate findings for any network traffic to/from these IPs

    GuardDuty DNS threat list (domain IOCs):
      Lambda: generate domain list → upload to S3
      GuardDuty: update threat list (domains for DNS tunneling detection + custom block)

    Propagation: org-level GuardDuty admin (security account) → all member accounts inherit threat lists
    Update frequency: every 15 minutes (aligned with TAXII poll frequency)

  Route 53 DNS Firewall (for domain IOCs):
    Lambda: build domain block list from DynamoDB (domain IOCs, confidence >= 75)
    Route 53 Resolver Firewall rule group: BLOCK action + custom response IP (sinkhole)
    Deploy via Firewall Manager: same rule group in all VPCs across all 50 accounts
    Log: DNS Firewall blocks → CloudWatch Logs → Security Lake → SIEM alert

  Hash IOCs (for EC2/ECS malware detection):
    Lambda: export hash IOC list to S3
    GuardDuty: custom threat intelligence (file hash list) → GuardDuty Runtime Monitoring
    When GuardDuty Runtime detects execution of a blocked hash → finding immediately

[THREAT HUNTING — ATHENA ON SECURITY LAKE]

Query performance optimization (< 30 second target):

  Athena workgroup configuration:
    Result reuse: enabled (identical query within 1 hour reuses cached result)
    Engine version: Athena v3 (uses Apache Iceberg + improved Parquet reading)

  Partitioning strategy:
    Security Lake auto-creates Glue catalog tables with partitions by:
      aws_region, source_location_uri, event_hour, event_day, event_month, event_year
    Athena: WHERE event_day='2024-01-15' AND aws_region='us-east-1'
      Scans: only 2024/01/15/us-east-1 partitions (< 1% of total data)

  Iceberg tables (for time-travel and merge):
    Security Lake (newer versions): supports Iceberg table format
    Iceberg: OPTIMIZE command → compacts small Parquet files into larger ones (improves query speed)
    Schedule: Glue job runs OPTIMIZE every 6 hours → maintains query performance over time

  Query templates for common threat hunts:

  Hunt: Find all EC2 instances communicating with known C2 IPs (IOC correlation):
    SELECT DISTINCT src_ip, dst_ip, protocol, dstport, COUNT(*) as connection_count,
           src.aws_account_id, src.src_instance_id
    FROM security_lake_glue.vpc_flow_logs vfl
    JOIN threat_intel_db.ioc_table ioc ON vfl.dst_ip = ioc.ioc_value
    WHERE ioc.ioc_type = 'ip' AND ioc.confidence >= 80
      AND vfl.event_day = CURRENT_DATE
    GROUP BY src_ip, dst_ip, protocol, dstport, src.aws_account_id, src.src_instance_id
    HAVING connection_count > 10
    -- Expected: returns within 15-20 seconds (today's data only, partitioned)

  Hunt: Detect impossible travel (same IAM user from 2 geographies < 30 min apart):
    WITH events AS (
      SELECT user_identity.arn as user_arn,
             CAST(event_time AS timestamp) as ts,
             source_ip_address as ip,
             ip2country(source_ip_address) as country  -- custom Lambda UDF
      FROM security_lake_glue.cloudtrail_mgmt
      WHERE event_day >= DATE_ADD('day', -7, CURRENT_DATE)
    ),
    paired AS (
      SELECT a.user_arn, a.ts as ts1, b.ts as ts2, a.country as c1, b.country as c2,
             ABS(EXTRACT(MINUTE FROM (b.ts - a.ts))) as minute_gap
      FROM events a JOIN events b ON a.user_arn = b.user_arn
      WHERE a.ts < b.ts AND ABS(EXTRACT(MINUTE FROM (b.ts - a.ts))) < 30
        AND a.country != b.country
    )
    SELECT * FROM paired ORDER BY ts1 DESC LIMIT 100

  Hunt: Identify data exfiltration (unusually large S3 downloads):
    SELECT src_instance_id, dst_ip, SUM(bytes_transferred) as total_bytes,
           COUNT(*) as request_count
    FROM security_lake_glue.s3_data_events
    WHERE event_day = CURRENT_DATE
      AND api_action = 'GetObject'
    GROUP BY src_instance_id, dst_ip
    HAVING total_bytes > 1073741824  -- > 1 GB
    ORDER BY total_bytes DESC

  SageMaker notebooks (threat hunters):
    S3 Security Lake → SageMaker Studio (Jupyter + Spark)
    Analysts: write Python/SQL notebooks for hypothesis-driven threat hunting
    Scala Spark: for complex graph analysis (lateral movement detection)
    Export findings: write IOC matches to DynamoDB findings table → reviewed by SOC

[SIEM HOT TIER — OPENSEARCH SECURITY ANALYTICS]

Amazon OpenSearch (hot tier — last 30 days):

  Ingestion (Security Lake → OpenSearch):
    Kinesis Data Firehose: Security Lake S3 events → Firehose → OpenSearch
    OR: Security Lake subscriber (SQS) → Lambda → OpenSearch bulk API

  OpenSearch Security Analytics:
    SIEM rules: Sigma rule format (community rules + custom)
    Detection rules: automatically correlate events → generate alerts in OpenSearch

    Example detection rules:
      "EC2 Instance Querying IMDSv1":
        CloudTrail filter: ec2.amazonaws.com GetMetadata without X-aws-ec2-metadata-token header
      "GuardDuty + VPC Flow Log Correlation":
        GuardDuty finding for IP X → VPC flow log shows same IP active → correlated alert

  OpenSearch Dashboards:
    SOC dashboard: real-time event counts, top threat sources, geographic threat map
    Alert dashboard: unacknowledged alerts, alert severity distribution

  Retention: 30 days in OpenSearch (hot tier, expensive)
  After 30 days: data stays in Security Lake S3 (cold tier, queryable via Athena)

[7-YEAR RETENTION WITH TIERED STORAGE]

S3 Lifecycle policy (Security Lake bucket):
  0-90 days: S3 Standard → frequent queries (daily SOC use)
  91-365 days: S3 Standard-Infrequent Access → monthly compliance queries
  366-2555 days (7 years): S3 Glacier Instant Retrieval
    Glacier Instant: millisecond retrieval (unlike Glacier Flexible which takes hours)
    Good for: occasional 7-year audit queries (retrieve in < 1 second per object)

  Cost estimate (500TB/year):
    Year 1: 500TB S3 Standard = $11,500/month
    Year 2: 500TB IA + 500TB Standard = $16,000/month
    Years 3-7: S3 Glacier Instant grows — ~$2/TB/month = ~$4,000/month for older data
    Total 7-year cost: ~$600K (vs. $5M+ for on-prem storage equivalent)

  S3 Object Lock WORM (legal admissibility):
    Compliance mode: 7-year TTL
    After 7 years: compliance mode expires → delete automatically (lifecycle expiration)
    Cannot delete before 7 years: even security admin cannot override compliance mode

  Restore for investigation (Glacier):
    Glacier Instant Retrieval: no restore needed (access directly like S3 IA)
    Glacier Flexible (if used instead): restore via S3 Glacier restore API → takes 3-5 hours
    Investigation workflow: Athena queries can read Glacier Instant directly (no restore needed)

[CROSS-ACCOUNT SECURITY LAKE ARCHITECTURE]

Security Lake organization setup:
  Delegated administrator: security account (SecurityLakeAdmin role)
  All 50 member accounts: Security Lake enabled automatically (no per-account setup)

  Data storage:
    All accounts' logs → security account's Security Lake S3 bucket (centralized)
    OR: each account has its own Security Lake bucket → security account has read-only subscriber

  Choice: centralized S3 bucket (simpler) vs. per-account buckets (data sovereignty)
    EU accounts: prefer per-account Security Lake bucket in eu-west-1 (data stays in EU)
    US accounts: centralized bucket in us-east-1 for efficiency

Security Deep Dive (Phase 3 — 15 min)

1. Identity and Access Management

Security Lake access model:
  Security analysts (threat hunters):
    Athena workgroup: analysts-hunting-wg (per-analyst cost tracking)
    IAM policy: athena:StartQueryExecution on analysts-hunting-wg
    s3:GetObject on Security Lake S3 bucket (read-only)
    glue:GetTable, glue:GetDatabase (read schema)
    CANNOT: delete logs, modify S3 lifecycle, modify Security Lake configuration

  IOC platform admin:
    Can create/modify IOC ingestion Lambda functions
    Can update WAF IP sets via Firewall Manager
    Cannot: read Security Lake data directly (separation of duties — threat intel != log access)

  Threat intelligence feed credentials:
    TAXII API key: Secrets Manager
    Commercial TI API key: Secrets Manager per feed
    Service account: Least privilege — only call TAXII API, write to IOC DynamoDB table

2. Encryption

Security Lake S3: SSE-KMS (security-lake-key)
  Key policy: Security Lake service principal + security analysts + Athena execution role

DynamoDB IOC table: SSE-KMS (ioc-registry-key)
  IOC data sensitivity: TLP-RED indicators must be encrypted + access-controlled
  TLP-RED IOCs: only shared with specific teams (not full SOC)

OpenSearch: SSE-KMS + HTTPS for Kibana
CloudWatch Logs (Firehose, Lambda logs): SSE-KMS

3. Network Isolation

Security Lake: AWS-managed service (no VPC configuration)
  S3 VPC endpoint: Lambda/Athena access to Security Lake S3 via VPC endpoint (no internet)

Athena: VPC endpoint for Athena (query execution within VPC)
  Lambda threat hunting runner: in VPC private subnet + Athena VPC endpoint

OpenSearch: deployed in VPC (private endpoint only)
  No public OpenSearch endpoint (Kibana accessible only via VPN/IAM Identity Center proxy)

Threat intel feed ingestion Lambda: VPC + NAT Gateway (TAXII servers on internet)

4. Logging and Audit Trail

Who accessed Security Lake:
  S3 data events: every GetObject from Security Lake S3 → CloudTrail S3 data event
  Athena query history: every query execution → CloudTrail + Athena query history
  OpenSearch access: OpenSearch audit logs (every index search + query)

IOC changes:
  DynamoDB Streams: every IOC create/update/delete → Lambda → CloudTrail custom event
  WAF IP set changes: CloudTrail (wafv2:UpdateIPSet)
  GuardDuty threat list changes: CloudTrail (guardduty:UpdateThreatIntelSet)

7-year log retention:
  CloudTrail itself stored in Security Lake (recursive — CloudTrail logs about Security Lake access)
  All access audit logs: same S3 Object Lock WORM retention

5. Least Privilege and Blast Radius

IOC blocking automation:
  Lambda (IOC-to-WAF): only permission to update specific named WAF IP sets
  Cannot: modify WAF rules, delete WAF WebACLs, modify security groups
  Blast radius: attacker who compromises IOC Lambda can modify WAF IP sets
  Mitigation: WAF IP sets are additive (adding IPs to block list) — cannot delete existing rules via IP set
  Separate permissions: adding IPs vs. deleting IPs from WAF IP set

Security Lake reader:
  Analysts: read-only access to Security Lake S3 + Athena
  Cannot: modify lifecycle rules, disable log sources, modify OCSF schema
  Blast radius: analyst can see all security logs → design access based on team/account scope if needed

Scale and Resilience (Phase 4 — 10 min)

500TB/year data volume:

  • Security Lake + Parquet: Parquet columnar + Zstandard compression: typical 5-10x compression ratio on log data. 500TB raw → ~60-100TB stored. Cost: significant reduction.
  • Athena scan: with proper partitioning (year/month/day/hour), a query for "today's data" scans 1/365 of annual data = ~140GB. At $5/TB: $0.70 per query. Very cost-effective.

Athena < 30 second queries:

  • Partition pruning: mandatory (all queries must include date/region filters).
  • Iceberg OPTIMIZE (compact small files): essential — Security Lake writes many small files per hour. After compaction: 10x faster queries.
  • Athena result cache: repeated identical queries (daily compliance checks) served from cache (instant).

OpenSearch scaling:

  • 30-day hot data: ~40TB (after compression). OpenSearch: 3-node hot tier + 3-node warm tier. Total storage: ~45TB with UltraWarm nodes.
  • Kinesis Firehose → OpenSearch: auto-scaling (no fixed throughput limit).

Trade-off Analysis — The Hard Choices

Decision 1: Centralized vs. Per-Account Security Lake Storage

Factor Centralized Security Lake (security account) Per-Account Security Lake
Query performance All data in one place — single Athena query covers all 50 accounts Must query each account's bucket separately or use cross-account Glue catalog
Data sovereignty EU logs replicated to central bucket (GDPR concern) EU logs stay in EU region account
Operational complexity Simple (one S3 bucket, one Athena table) Complex (50 buckets, 50 Athena data sources)
Cost One storage cost center Distributed, harder to track
Recommendation Centralized for US accounts; Per-account for EU accounts (GDPR data residency). Hybrid approach.

Decision 2: OpenSearch (Hot Tier) vs. Pure Athena for All Queries

Factor OpenSearch Hot Tier (30 days) Athena-Only
Real-time query speed Sub-second 5-30 seconds
Cost $2,000-$5,000/month for OpenSearch cluster ~$0.005/query (Athena pay-per-query)
SIEM rule support Native Sigma rules in OpenSearch Security Analytics Manual Athena queries (no native SIEM rules)
Alert correlation Real-time (< 1 minute detection-to-alert) Scheduled (minimum 5-minute polling)
Recommendation OpenSearch for real-time SOC alerts (last 30 days) + Athena for historical threat hunting (30 days to 7 years). Both needed for full capability.

Decision 3: AWS-Managed GuardDuty Threat Lists vs. Commercial TI Platform for IOC Blocking

Factor GuardDuty Threat Lists (custom) Commercial TI Platform (Anomali, ThreatConnect)
Coverage Custom IP + domain lists (up to 6 per detector) Hundreds of curated feeds, correlation, analysis
Automation Lambda → GuardDuty update → finding Platform-native SOAR integration
Cost GuardDuty standard pricing $100K-$500K+/year
Context IP flagged, no context Full threat actor profile, attribution, TTPs
Recommendation GuardDuty custom lists for automated blocking (cost-effective); Commercial TI platform for high-value SOC analyst context and hunt support — if budget exists

Common Mistakes to Avoid

  1. Describing Security Lake as just "S3 with logs." Security Lake normalizes ALL log sources to OCSF format (common schema). This is the KEY architectural benefit — you can query CloudTrail + VPC Flow Logs + GuardDuty findings with the SAME SQL schema. Candidates who miss OCSF miss the point of Security Lake.
  2. Not addressing the WAF IP set scale limit. WAF IP sets have a limit of 10,000 IPs per IP set. Commercial threat intelligence feeds can include millions of IPs. You cannot block all of them via WAF. Solution: confidence thresholds (only block highest-confidence, most-active IOCs), WAF rate limiting for lower-confidence IPs, and GuardDuty for network-level detection of the broader list.
  3. Ignoring query performance optimization for 500TB dataset. Saying "just use Athena" without describing partitioning, Parquet format, Iceberg compaction, and result caching will fail the < 30 second SLA. Candidates must demonstrate knowledge of how columnar formats, partitioning, and file compaction enable sub-minute queries on petabyte-scale data.
  4. Not mentioning IOC expiry and stale indicator cleanup. Threat intelligence IOCs expire. An IP that was malicious in 2020 may now be used by a legitimate company. DynamoDB TTL (auto-delete expired IOCs) + confidence decay over time are essential for keeping the blocking lists accurate and avoiding false positives years later.

Likely Follow-up Questions

Q1: How do you correlate across different log source types (CloudTrail + VPC Flow Logs + GuardDuty) for a single incident?
OCSF-based correlation query: (1) The power of Security Lake OCSF is that all sources share common fields: src_endpoint.ip, time_dt, severity_id, user.uid. A single SQL query can JOIN across source tables using these common fields. (2) Example: GuardDuty finding says IP 185.x.x.x is a C2 server. Correlation: SELECT ct.user_identity, vfl.src_instance_id, dns.query, gd.finding_type FROM cloudtrail_events ct JOIN vpc_flow_logs vfl ON ct.source_ip_address = vfl.dst_ip JOIN guardduty_findings gd ON vfl.dst_ip = gd.service.remoteIpDetails.ipAddressV4 LEFT JOIN dns_logs dns ON vfl.src_ip = dns.src_ip WHERE gd.service.remoteIpDetails.ipAddressV4 = '185.x.x.x' AND ct.event_day = CURRENT_DATE → returns which IAM user from which EC2 instance made calls AND which DNS queries that instance made AND the GuardDuty finding context — all correlated on the same IP + timestamp. (3) Amazon Detective provides a visual graph of this correlation for specific GuardDuty findings (no SQL needed for standard investigation). (4) For complex temporal correlation: use Athena window functions: LAG() OVER (PARTITION BY src_ip ORDER BY time_dt) to find sequences of events from the same source.

Q2: How do you handle TLP (Traffic Light Protocol) restrictions for threat intelligence sharing?
TLP enforcement in the IOC platform: (1) TLP-WHITE: freely shareable (open source feeds, CISA advisories). Visible to all analysts. IOC table attribute tlp = "white". (2) TLP-GREEN: shareable within community. SOC team + senior analysts. IAM condition: StringEquals: { "dynamodb:LeadingKeys": ["tlp=green"] } — only roles with green-cleared attribute can query. (3) TLP-AMBER: limited distribution. Only teams directly involved in the incident. Store in separate DynamoDB table with restricted IAM. Auto-promote to GREEN after 90 days if no sensitivity objection. (4) TLP-RED: no further sharing. Store in separate encrypted S3 prefix + DynamoDB table. Access: only CISO + threat intel team leads (IAM: 3 specific role ARNs in the key policy). Never appears in WAF IP sets (automated blocking of TLP-RED IOCs could reveal the indicator). Manual blocking only. (5) Automated blocking rules: TLP-WHITE/GREEN → auto-block via WAF/GuardDuty (indicator can be "leaked" via network block = acceptable). TLP-AMBER → block but do NOT log the specific indicator value in CloudWatch (prevent log-based disclosure). TLP-RED → manual process only.

Q3: How do you measure the effectiveness of threat intelligence feeds?
TI feed effectiveness metrics: (1) Hit rate: for each feed source, how many IOCs resulted in actual GuardDuty findings or WAF blocks against real traffic? Feed hit rate = (IOCs that matched actual traffic) / (total IOCs from feed). Feeds with < 1% hit rate: possibly low-quality or irrelevant to your threat model. (2) False positive rate: how many WAF blocks were complaints from legitimate users? Track via WAF block logs → customer support tickets. Feeds causing > 0.1% false positive rate → review confidence threshold for that feed. (3) Detection timeliness: how long between a threat actor first seen in the wild and the feed including the IOC? Compare feed IOC first_seen timestamp vs. first GuardDuty finding for the same IOC in your environment. Fast feeds (< 24 hours): high value. Slow feeds (> 7 days): mainly useful for historical hunting, not real-time blocking. (4) Overlap analysis: what % of each feed's IOCs are already in another feed? High overlap = redundant feed (consider removing to reduce cost). AWS Clean Rooms: share IOC hashes (not values) between organizations to compute overlap without revealing sensitive TI. (5) Cost per detection: (feed subscription cost) / (confirmed true positive detections). Business case for renewing commercial feed subscriptions.

Q4: How do you operationalize threat hunting in a 24/7 SOC environment?
Structured threat hunting program: (1) Hypothesis-driven hunts: monthly hunt calendar. Each hunt has: hypothesis (e.g., "Are any EC2 instances exfiltrating data via DNS tunneling?"), data sources (DNS query logs in Security Lake), detection method (query for domains with high entropy + unusual subdomain length), success criteria (identify 0 or more instances matching the pattern). (2) SageMaker Studio notebooks: each hunt hypothesis = Jupyter notebook in a shared library. Analysts: clone notebook → customize parameters → run against Security Lake → document results. Reusable: next month's analyst runs the same notebook with updated date range. (3) Alert-driven escalation: GuardDuty + OpenSearch generates an alert → SOC L1 analyst: triage (5 minutes). If suspicious: escalate to L2 threat hunter. L2: opens Athena hunt query notebook → investigates 7-day historical context. L2: documents findings in the hunt notebook → creates Security Lake query for similar patterns. (4) Automated hunts (scheduled): Lambda: runs SQL hunt query against Security Lake every 6 hours → if results > threshold → creates GuardDuty custom finding → SOC alert. Example: "Any IAM user called > 100 distinct API calls in 1 hour?" (credential stuffing or reconnaissance indicator). (5) Hunt residuals: every hunt that finds something → new IOC added to DynamoDB → automatic blocking deployed. Every hunt that finds nothing → hunt query archived as "negative result" (evidence that we looked). Both outcomes documented for SOC maturity metrics.

Q5: How do you ensure Security Lake data completeness (no missing logs)?
Security Lake data completeness verification: (1) Coverage monitoring: Security Lake console → Sources tab. Shows: for each enabled log source (CloudTrail, VPC Flow Logs, etc.), whether data is flowing or status is ERROR. Alert: EventBridge rule on Security Lake source status change → Lambda → PagerDuty if any source goes to ERROR. (2) Volume anomaly detection: CloudWatch metric: Security Lake object count per day. If today's CloudTrail events = 50% of average (or 0): Lambda alert → "Security Lake may have a collection gap." Cause: could be CloudTrail disabled (Config rule detects), Security Lake configuration change, S3 permission issue. (3) Gap detection: daily Lambda: Athena query — count events per account per source for yesterday. Compare to 7-day average. If any account has < 70% expected volume → alert: "Account 123456789012 CloudTrail data gap detected." (4) Cross-validation: CloudTrail log file count (from the CloudTrail S3 delivery bucket) vs. Security Lake event count. Should be approximately equal. Discrepancy: Security Lake ingestion pipeline issue. (5) Backup path: if Security Lake has ingestion issue: CloudTrail S3 bucket is independent (logs delivered directly by CloudTrail). Security Lake failure doesn't mean logs are lost — they're in the original CloudTrail S3 bucket. Restoration: replay from CloudTrail S3 bucket to Security Lake using Glue ETL job.

Q6: How does Security Lake integrate with a third-party SIEM like Splunk?
Security Lake SIEM integration: (1) Subscriber model: Security Lake supports two subscriber types: Data Access (direct S3 access to Security Lake bucket) and Query Access (uses Lake Formation + Athena to query). Splunk: uses Data Access subscriber (SQS + S3). (2) Splunk SQS-based S3 input: Security Lake → SQS notification (new Parquet file landed) → Splunk Add-on for Amazon S3 polling SQS → downloads Parquet file from S3 → indexes into Splunk. Splunk transforms OCSF to CIM (Common Information Model) format during ingestion. (3) Near-real-time ingestion: Splunk reads SQS every 30 seconds → < 2 minute latency from event occurrence to Splunk index. Splunk: can search last 30 days at sub-second speed (hot/warm index). For historical: Splunk SmartStore (backs to S3 — but expensive for 500TB). Better: use Splunk for 30 days + Athena for 7-year historical. (4) OCSF to CIM mapping: Security Lake OCSF user.uid → Splunk CIM src_user. OCSF src_endpoint.ip → Splunk src_ip. Splunk Add-on for Security Lake handles this mapping automatically. (5) Federated search: Splunk can federate searches to Athena (via Splunk DB Connect JDBC + Athena JDBC driver). Analyst in Splunk: run a search that queries Security Lake Athena for 90-day data → results returned in Splunk UI. This avoids duplicating data in Splunk while still enabling single-pane-of-glass SOC experience.

Q7: How do you handle false positives from automated IOC blocking?
False positive management for IOC blocking: (1) Confidence threshold: don't block on confidence < 80. Confidence 50-79: GuardDuty detect-but-not-block (generates finding without network block). SOC can investigate the finding and manually promote to blocking if confirmed. (2) IP range validation (before WAF block): Lambda: check if IOC IP is in: AWS IP ranges (ip-ranges.json), Cloudflare/Akamai CDN ranges, Private IP ranges (RFC 1918), Well-known legitimate ASNs (Google, Microsoft, Amazon). If yes: override block → ALERT instead. Prevents blocking CDN nodes or AWS APIs. (3) Allowlist: operator-maintained allowlist in DynamoDB (allow_list table). If IOC IP is in allowlist: skip WAF block regardless of confidence. Common entries: payment gateway IPs, partner API IPs, CDN IPs. (4) Canary detection: before adding new IOC to WAF: run a dry-run query against Security Lake: "Does any production traffic flow to this IP?" If yes (> 100 connections in last 24h from production): HOLD for SOC review before blocking. Flag the IOC as "potential false positive candidate." (5) Unblock workflow: Step Functions + Slack: if WAF block is suspected false positive → [Unblock IP] button → Lambda: moves IP from block list to allowlist → WAF IP set updated → IOC in DynamoDB flagged as false-positive with reason. Feed provider notified: "Your indicator [IP] caused a false positive — please review."
