AWS Security System Design Questions — 25 × 1-Hour Sessions

Format: Each question is structured as a 1-hour session. The reviewer drives through
five phases: Clarify → Initial Design → Deep Dive → Scale & Resilience → Threat & Operations.
Prepare to whiteboard, talk through trade-offs, and defend every AWS service choice.

AWS Security Baselines Covered: IAM, KMS, CloudTrail, GuardDuty, Security Hub, WAF,
Shield, VPC, PrivateLink, S3, Secrets Manager, Certificate Manager, Config, Inspector,
Macie, Detective, Network Firewall, SCPs, Control Tower, Lake Formation, and more.

How to Use This Guide

PHASE BREAKDOWN PER QUESTION (60 minutes):
  Phase 1 — Clarify Requirements         (5 min)   Ask scoping questions before drawing anything
  Phase 2 — Initial Design               (15 min)  Core architecture, key AWS services, data flow
  Phase 3 — Security Deep Dive           (15 min)  IAM, encryption, network isolation, logging
  Phase 4 — Scale & Resilience           (10 min)  Multi-region, high availability, blast radius
  Phase 5 — Threat Model & Operations    (15 min)  Attack scenarios, detection, incident response

SELF-EVALUATION AFTER EACH SESSION:
  □ Did I name specific AWS services — not just "a log aggregator"?
  □ Did I cover encryption at rest AND in transit for every data store?
  □ Did I design IAM roles with least privilege — no wildcard policies?
  □ Did I address the blast radius if one AWS account is compromised?
  □ Did I include a detection/response layer — not just preventive controls?
  □ Did I think about cost and operational overhead of my design?

Question 1 — Centralized Security Logging Platform for a 200-Account AWS Organization

The Ask:

"Design a centralized, tamper-evident security logging platform for an AWS Organization
with 200 member accounts. It must ingest CloudTrail, VPC Flow Logs, DNS query logs,
Config snapshots, and application logs. Security analysts need near-real-time search
and a 7-year retention requirement for compliance."

Phase 1 — Clarify:

  • Are all accounts in a single AWS Organization with SCPs already enforced?
  • Is there a dedicated Security account or Log Archive account pattern already in use?
  • What is the expected log volume per day (GB/day)?
  • Are there data residency constraints (logs must stay in specific regions)?
  • Does the compliance requirement mandate immutability (WORM) or just retention?

Phase 2 — Initial Design (draw this):

  • AWS Organizations → AWS Control Tower with Log Archive account
  • Enable CloudTrail Organization Trail — writes to centralized S3 bucket in Log Archive account
  • Enable VPC Flow Logs → CloudWatch Logs → Kinesis Data Firehose → S3 (Log Archive)
  • Route 53 Resolver query logs → S3
  • AWS Config Organization Aggregator → Config delivery channel → S3
  • Application logs → CloudWatch Logs → Subscription Filter → Kinesis Data Firehose → S3
  • S3 Object Lock (Compliance Mode) for WORM enforcement
  • S3 Lifecycle policies: S3 Standard (0–90 days) → S3 Glacier Instant Retrieval (90 days–1 yr) → S3 Glacier Deep Archive (1–7 yr)

Phase 3 — Security Deep Dive:

  • S3 bucket policy: deny s3:DeleteObject, deny s3:PutBucketPolicy except from Log Archive account org SCP principal
  • KMS Customer Managed Key (CMK) in Log Archive account — key policy denies management from member accounts
  • S3 bucket: block all public access, enforce aws:SecureTransport in bucket policy
  • Kinesis Firehose delivery with server-side encryption using KMS CMK
  • IAM roles: member account log delivery role trusts only the Firehose delivery principal
  • GuardDuty enabled in all accounts, findings aggregated to Security account via GuardDuty administrator delegation
  • AWS Security Hub: organization-wide aggregation to Security account
  • CloudTrail log integrity validation enabled — SHA-256 digest files
  • S3 Access Logging enabled on the log archive bucket itself (meta-logging)
  • SCP on Log Archive account: deny cloudtrail:StopLogging, deny cloudtrail:DeleteTrail, deny s3:DeleteBucket

Phase 4 — Scale & Resilience:

  • Multi-region: replicate S3 log bucket to a second region using S3 Cross-Region Replication with Object Lock preserved
  • Kinesis Data Firehose: configure retry duration (300 seconds), backup bucket for failed deliveries
  • Athena + AWS Glue Data Catalog over S3 for ad-hoc analyst queries (no data movement, pay-per-query)
  • OpenSearch Service cluster (Security account) for near-real-time search — stream hot logs (last 90 days) from Kinesis
  • For 200 accounts scaling: use EventBridge organization rules to route events without per-account Lambda

Phase 5 — Threat Model & Operations:

  • Threat: malicious admin in a member account tries to delete CloudTrail → blocked by SCP
  • Threat: attacker compromises Log Archive account root → multi-person MFA for root, hardware key required
  • Threat: insider in Security account deletes S3 objects → S3 Object Lock Compliance mode prevents even root deletion
  • Detection: CloudTrail event DeleteTrail or StopLogging → EventBridge rule → SNS alert → PagerDuty
  • Detection: S3 server access log shows DELETE on log bucket → immediate P0 alert
  • Residual risk: Log delivery latency means a 5–15 minute gap — acceptable? Document in risk register.

AWS Services to Master for This Question:
AWS Organizations Control Tower CloudTrail Organization Trail S3 Object Lock KMS CMK
Kinesis Data Firehose VPC Flow Logs GuardDuty Security Hub Athena OpenSearch SCP

Question 2 — Zero Trust Network Architecture for a Hybrid Enterprise

The Ask:

"Design a Zero Trust network architecture on AWS for an enterprise with 10,000 employees,
50 on-premises applications, 100 cloud-native workloads, and contractors on BYOD devices.
All application access must be identity-aware, device-posture-checked, and micro-segmented.
No application should be reachable via VPN."

Phase 1 — Clarify:

  • Is there an existing AWS Direct Connect or Site-to-Site VPN to on-premises?
  • Which Identity Provider (Okta, Azure AD, Google Workspace)?
  • Are workloads containerized (EKS) or VM-based (EC2)?
  • What is the contractor access pattern — web apps only, or SSH/RDP to servers?
  • Is this a greenfield design or migration from existing VPN?

Phase 2 — Initial Design:

  • AWS IAM Identity Center (SSO) as the central identity broker, federated to corporate IdP
  • AWS Verified Access for browser-based application access (replaces VPN for web apps)
  • AWS Client VPN with mutual TLS + IAM Identity Center SAML for non-browser access (temporary)
  • AWS Transit Gateway connecting spoke VPCs — replace with VPC Lattice for service-to-service zero trust
  • AWS PrivateLink for all shared service access (no public endpoints)
  • Network Firewall at the perimeter of each application VPC
  • Security Groups as the primary east-west micro-segmentation control (not NACLs)

Phase 3 — Security Deep Dive:

  • AWS Verified Access: trust provider = IAM Identity Center (identity) + AWS Verified Access device trust (Cedar policy)
  • Cedar policies: permit(principal, action, resource) when { principal.mfa == true && device.compliance == "compliant" }
  • VPC design: separate VPCs per application tier (no shared VPCs) — Transit Gateway for controlled routing
  • Security Groups: no 0.0.0.0/0 ingress rules; source = Security Group ID of the calling service
  • IAM roles for EC2: instance profiles with least-privilege; no long-term access keys on instances
  • AWS Network Firewall: stateful rules blocking non-standard ports; Suricata-based IDS signatures enabled
  • PrivateLink for S3, DynamoDB, SSM, Secrets Manager, KMS (no traffic over public internet)
  • Route 53 Resolver private hosted zones — no split-horizon DNS public exposure

Phase 4 — Scale & Resilience:

  • Verified Access scales automatically; configure multiple Verified Access endpoints per region
  • Transit Gateway: enable Equal Cost Multi-Path (ECMP) routing for Direct Connect redundancy
  • VPC Lattice: service network for 100+ microservices eliminates per-service PrivateLink endpoints
  • Multi-region: replicate IAM Identity Center permission sets; use Route 53 health checks for failover
  • Blast radius: if one spoke VPC is compromised, Transit Gateway route tables isolate it without affecting others

Phase 5 — Threat Model & Operations:

  • Threat: stolen employee credential used from attacker's device → Verified Access device trust policy blocks non-compliant device
  • Threat: lateral movement from compromised EC2 → Security Group source restrictions limit reachable targets
  • Threat: attacker creates a rogue PrivateLink endpoint → endpoint policies restrict which principals can use the endpoint
  • Detection: VPC Flow Logs show unexpected cross-VPC traffic → GuardDuty network findings
  • Detection: Verified Access access logs show access from new geo/device → alert via Security Hub
  • Residual risk: contractors on BYOD — device trust relies on MDM enrollment; unmanaged devices have no attestation

AWS Services to Master:
Verified Access IAM Identity Center VPC Lattice Transit Gateway Network Firewall
PrivateLink Security Groups Client VPN GuardDuty Cedar Policy Language

Question 3 — Secrets Management Platform for 500 Microservices

The Ask:

"Design a secrets management platform on AWS for a SaaS company running 500 microservices
on EKS. Secrets include database passwords, third-party API keys, TLS private keys, and
internal service tokens. The platform must eliminate hardcoded secrets, enforce rotation,
and provide full audit trail. Dev teams must be able to self-service without opening
support tickets."

Phase 1 — Clarify:

  • Are microservices in a single EKS cluster or multi-cluster (multi-region, multi-account)?
  • What databases are in use (RDS, Aurora, DynamoDB, external)?
  • Is there an existing secrets solution (HashiCorp Vault, Kubernetes Secrets)?
  • How do dev teams currently deploy — GitOps (ArgoCD/Flux) or direct kubectl?
  • What is the rotation tolerance — can apps reconnect during rotation or is zero-downtime required?

Phase 2 — Initial Design:

  • AWS Secrets Manager as the primary secrets store
  • AWS Systems Manager Parameter Store for non-sensitive configuration (free tier for standard parameters)
  • EKS Pod Identity (not IRSA) for pod-level IAM role assignment → pods access Secrets Manager directly
  • External Secrets Operator (ESO) on EKS → syncs Secrets Manager secrets to Kubernetes Secrets
  • RDS/Aurora native integration with Secrets Manager for automatic password rotation (Lambda rotation function)
  • KMS CMK per secret category (database creds, API keys, TLS keys) — separate key policies per team

Phase 3 — Security Deep Dive:

  • EKS Pod Identity: each microservice has a unique IAM role — secretsmanager:GetSecretValue scoped to its own secret ARN prefix
  • IAM resource condition: secretsmanager:SecretId matches arn:aws:secretsmanager:region:account:secret:service-name/*
  • KMS key policy: key usage granted to specific IAM roles only; key administration denied to developer roles
  • Secrets Manager resource policy: deny GetSecretValue from outside the VPC (aws:sourceVpc condition)
  • Rotation: Lambda rotation function runs in a private subnet with a VPC endpoint for Secrets Manager and RDS
  • Versioning: Secrets Manager maintains AWSCURRENT and AWSPREVIOUS — no rotation-induced downtime
  • Dev self-service: IAM Identity Center permission set allows devs to call secretsmanager:CreateSecret in dev accounts only; prod secrets require PR-based Terraform with approval

Phase 4 — Scale & Resilience:

  • Secrets Manager replication: replicate secrets to secondary region for DR (KMS CMK must also exist in target region)
  • ESO caching: External Secrets Operator caches secrets in memory — reduces Secrets Manager API call volume
  • Secrets Manager has a 10,000 secrets limit per account — at 500 microservices with multiple secrets each, monitor usage
  • Rotation failure: CloudWatch alarm on SecretsManagerRotationFailed metric → PagerDuty
  • Cost: Secrets Manager charges $0.40/secret/month — model cost for 5,000 secrets at scale

Phase 5 — Threat Model & Operations:

  • Threat: developer hardcodes a secret in GitHub → AWS git-secrets or GitGuardian pre-commit hook; Macie scans S3/CodeCommit for secret patterns
  • Threat: compromised pod exfiltrates all secrets it has access to → IAM role scoped to single secret ARN; blast radius is one service
  • Threat: lateral movement via stolen IRSA/Pod Identity token → token is bound to pod's service account; short-lived (1 hour default)
  • Detection: CloudTrail GetSecretValue from unexpected IAM role or IP → EventBridge → Security Hub finding
  • Detection: Secrets Manager rotation failure event → CloudWatch alarm
  • Residual risk: ESO stores secrets as Kubernetes Secrets (base64, not encrypted) → enable KMS envelope encryption for etcd at rest on EKS

AWS Services to Master:
Secrets Manager EKS Pod Identity KMS CMK External Secrets Operator Parameter Store
RDS Rotation Lambda Macie CloudTrail EventBridge

Question 4 — Multi-Account AWS Landing Zone with Security Guardrails

The Ask:

"Design a secure AWS multi-account landing zone for a financial services company that
is migrating 200 applications from on-premises. The design must enforce compliance
(PCI-DSS, SOC 2), isolate environments (dev/staging/prod), prevent privilege escalation,
and give the security team centralized visibility without being a bottleneck for
development teams."

Phase 1 — Clarify:

  • Greenfield AWS Organization or migrating existing accounts into an Organization?
  • How many business units — should each get its own OU structure?
  • Is there an existing on-premises identity system (AD, Okta) to federate?
  • What compliance frameworks must be verifiably enforced at the infrastructure level?
  • What is the developer autonomy model — full account ownership or shared accounts?

Phase 2 — Initial Design (Account Vending Machine model):

  • AWS Control Tower as the landing zone orchestrator
  • Account hierarchy: Root → Security OU (Log Archive, Audit) → Infrastructure OU → Workload OUs (per BU: Dev, Staging, Prod)
  • Account Factory for Terraform (AFT) — automated account provisioning with baseline controls
  • IAM Identity Center: centralized permission sets, mapped to AD groups
  • AWS Config Organization Aggregator in Audit account
  • AWS Security Hub: delegated admin in Security account; all standards enabled (CIS, PCI-DSS, NIST)
  • GuardDuty: organization-wide with delegated admin in Security account
  • CloudTrail Organization Trail to Log Archive S3 bucket (Question 1 pattern)

Phase 3 — Security Deep Dive:

  • SCPs (top priority controls — these cannot be overridden by account admins):
  • Deny iam:CreateUser (force SSO) except in break-glass scenario
  • Deny all actions if aws:RequestedRegion not in approved regions list
  • Deny ec2:RunInstances with non-approved instance types (prevent crypto mining)
  • Deny cloudtrail:StopLogging, cloudtrail:DeleteTrail
  • Deny s3:DeleteBucket on log archive buckets
  • Deny leaving the AWS Organization (organizations:LeaveOrganization)
  • Require aws:MultiFactorAuthPresent for sensitive actions (MFA SCP)
  • AWS Config Rules:
  • restricted-ssh — no Security Groups allowing SSH from 0.0.0.0/0
  • s3-bucket-public-read-prohibited
  • encrypted-volumes — all EBS volumes must be encrypted
  • rds-storage-encrypted
  • iam-root-access-key-check — root must not have access keys
  • AWS Config Conformance Packs: PCI-DSS pack, CIS Level 2 pack
  • Delegated admin model: security team operates from Audit account, cannot modify workload account resources

Phase 4 — Scale & Resilience:

  • Account Factory for Terraform (AFT) customizations: apply baseline Terraform module to every new account (VPC defaults, CloudTrail, GuardDuty enrollment, Security Hub enrollment)
  • At 200+ accounts: use Config Organization Aggregator — single pane of glass, no per-account dashboards
  • Permission set changes via CI/CD pipeline (not manual) — IAM Identity Center permission sets versioned in Git
  • EventBridge cross-account event bus: member account security events → Security account bus → SIEM

Phase 5 — Threat Model & Operations:

  • Threat: developer escalates privileges in their prod account → SCP denies iam:AttachRolePolicy without condition on role path; IAM permission boundaries on all roles
  • Threat: account is compromised — blast radius limited to single account by design
  • Threat: Control Tower management account compromise → this account is the highest-privilege target; restrict access to 5 named individuals, hardware MFA, no programmatic access
  • Detection: AWS Config non-compliant resource → Security Hub finding → EventBridge → SOAR auto-remediation (e.g., revoke public S3 bucket)
  • Detection: IAM Access Analyzer: external access findings for any resource shared outside the organization
  • Residual risk: SCPs are account-level, not resource-level — a compromised account-admin can still cause damage within permitted SCP bounds

AWS Services to Master:
Control Tower Account Factory for Terraform SCP IAM Identity Center AWS Config
Security Hub GuardDuty IAM Permission Boundaries Access Analyzer EventBridge

Question 5 — Secure Data Lake for PII at Petabyte Scale

The Ask:

"Design a secure data lake on AWS for a healthcare company that processes 500 TB of
patient data (PHI/PII) per month. Multiple teams (data science, analytics, compliance)
need access to the data but with different granularity — compliance sees all, data science
sees de-identified data, analysts see aggregate only. The platform must satisfy HIPAA."

Phase 1 — Clarify:

  • What is the data ingestion source — on-premises EHR systems, IoT medical devices, SaaS apps?
  • Is the de-identification requirement statistical anonymization or tokenization?
  • What query engine do data science teams use (Athena, EMR, Redshift)?
  • What is the data freshness requirement (real-time vs batch)?
  • Are there cross-region replication requirements (DR)?

Phase 2 — Initial Design:

  • S3 as the data lake foundation (raw, processed, curated zones — separate S3 buckets per zone)
  • AWS Glue for ETL — ingestion, transformation, de-identification, and catalog
  • AWS Lake Formation as the fine-grained access control layer over S3 + Glue Catalog
  • Amazon Macie for continuous PII discovery and classification in S3
  • KMS CMK per data classification tier (PHI key, de-identified key, aggregate key)
  • Athena for serverless SQL queries with Lake Formation column-level and row-level security
  • AWS Glue DataBrew for data de-identification transformations

Phase 3 — Security Deep Dive:

  • Lake Formation permissions:
  • Compliance team: SELECT on all columns, all rows — granted via Lake Formation IAM role
  • Data science team: SELECT on de-identified table (separate Glue table pointing to processed zone) — column-level permissions exclude patient_id, dob, ssn
  • Analytics team: SELECT on aggregate views only — row-level filter applied via Lake Formation data filter
  • KMS: PHI data in raw S3 bucket encrypted with PHI CMK; only ingestion Glue job IAM role and compliance role can use PHI CMK
  • S3 bucket policies: deny direct S3 API access; all access must go through Athena + Lake Formation (deny s3:GetObject without Lake Formation resource link)
  • VPC endpoints for S3 and Glue — no PHI data traverses public internet
  • Glue job encryption: job bookmarks encrypted, temp storage encrypted, connection passwords in Secrets Manager
  • CloudTrail data events on S3 PHI bucket — every GetObject logged
  • Macie: scheduled discovery jobs on raw zone bucket; finding alerts to Security Hub

Phase 4 — Scale & Resilience:

  • S3 Intelligent Tiering for cost optimization on large datasets (500 TB/month ingestion adds up)
  • Glue job auto-scaling: G.2X worker type for large transforms; Glue Auto Scaling enabled
  • Athena workgroup: per-team workgroups with query result location in team-scoped S3, data usage limits
  • Multi-region: S3 Cross-Region Replication for DR; replicate CMK to DR region; Lake Formation permissions are regional — automate with IaC
  • At petabyte scale: S3 bucket with millions of objects — use S3 Inventory + Macie batch job instead of continuous scanning

Phase 5 — Threat Model & Operations:

  • Threat: data scientist queries raw PHI zone directly (bypassing Lake Formation) → S3 bucket policy denies all GetObject unless via Lake Formation service principal; tested with IAM policy simulator
  • Threat: Glue ETL job exfiltrates data to external S3 → Glue VPC network, no internet egress; SCP denies s3:PutObject to buckets outside the organization
  • Threat: Macie misconfiguration misses new PII fields → periodic manual review of Macie custom data identifiers; quarterly penetration test includes data exfiltration scenarios
  • Detection: Macie S3 object access anomaly (user accessing more objects than baseline) → finding → Security Hub → analyst triage
  • Detection: CloudTrail GetObject with unexpected IAM principal on PHI bucket → EventBridge → P0 alert
  • Residual risk: de-identification is not perfect — re-identification risk from quasi-identifiers remains; accepted with documented risk register entry signed by CISO

AWS Services to Master:
Lake Formation Macie AWS Glue S3 Object Lock KMS CMK Athena Workgroups
Glue DataBrew CloudTrail Data Events VPC Endpoints

Question 6 — Runtime Security and Threat Detection for EKS at Scale

The Ask:

"Design a runtime security and threat detection platform for a company running 500
microservices on EKS across 3 AWS regions. The platform must detect container escapes,
cryptomining, lateral movement, anomalous API calls, and data exfiltration in real time.
Mean time to detect (MTTD) target is under 5 minutes."

Phase 1 — Clarify:

  • Are EKS clusters on EC2 (Managed Node Groups) or Fargate?
  • Is there a multi-cluster service mesh (Istio, AWS App Mesh)?
  • What is the current security tooling (any existing SIEM, EDR, or Falco deployment)?
  • What is the MTTR target — detection only or also automated response?
  • Are Kubernetes API server audit logs currently enabled?

Phase 2 — Initial Design:

  • Amazon GuardDuty EKS Protection: EKS Audit Log Monitoring + EKS Runtime Monitoring (eBPF agent)
  • Amazon GuardDuty Malware Protection for EBS volumes
  • EKS control plane logging → CloudWatch Logs (api, audit, authenticator, controllerManager, scheduler)
  • Amazon Detective for investigation (integrated with GuardDuty findings)
  • AWS Security Hub: aggregate GuardDuty + Inspector findings
  • Amazon Inspector: continuous vulnerability scanning for container images in ECR
  • VPC Flow Logs for pod network traffic baseline
  • Falco (open-source) deployed as DaemonSet for custom runtime rules (complement to GuardDuty)

Phase 3 — Security Deep Dive:

  • GuardDuty EKS Audit Log findings (examples to memorize):
  • Execution:Kubernetes/ExecInKubernetes — kubectl exec into running pod
  • Privilege Escalation:Kubernetes/PrivilegedContainer — privileged pod launched
  • Persistence:Kubernetes/ContainerWithSensitiveMount — sensitive host path mounted
  • Discovery:Kubernetes/KubernetesAPICalledFromProxyAddress — API called via Tor/proxy
  • Impact:Kubernetes/CryptoCurrency:Runtime/BitcoinTool — crypto mining binary detected
  • GuardDuty Runtime agent: eBPF-based sensor on each node (no kernel module required); monitors syscalls
  • Kubernetes RBAC: no cluster-admin bindings for application service accounts; audit with kubectl get clusterrolebindings
  • Pod Security Standards (Baseline or Restricted) enforced via OPA Gatekeeper admission controller
  • Network Policies (Calico): default-deny all, explicit allow rules per service
  • ECR image scanning: Inspector v2 continuous scanning — block CRITICAL CVE images from deployment via admission webhook

Phase 4 — Scale & Resilience:

  • GuardDuty Runtime agent: managed DaemonSet updated automatically; no manual patching
  • At 500 microservices: GuardDuty finding volume can be high — prioritize HIGH and CRITICAL severity; use Security Hub suppression rules for known-safe patterns
  • EventBridge rule: GuardDuty HIGH/CRITICAL finding → Lambda → Slack alert + SOAR ticket
  • For MTTD < 5 min: GuardDuty near-real-time findings (typically 2–5 min latency from event to finding); test with a known-bad Docker image and measure
  • Multi-region: GuardDuty enabled independently per region; Security Hub cross-region aggregation

Phase 5 — Threat Model & Operations:

  • Threat: container escape via kernel exploit → GuardDuty Runtime agent detects host syscall from container namespace; PSA Restricted profile blocks hostPID, hostNetwork, privileged
  • Threat: cryptominer deployed via supply-chain compromised image → Inspector blocks image in ECR; GuardDuty BitcoinTool runtime finding if it runs
  • Threat: lateral movement via service account token → EKS Pod Identity (not static IRSA) limits token lifetime to 1 hour; network policy blocks pod-to-pod if not explicitly allowed
  • Threat: attacker disables GuardDuty in one account → SCP denies guardduty:DisassociateFromMasterAccount, guardduty:DeleteDetector
  • Detection: EKS audit log shows exec into production pod by non-approved user → EventBridge → P0 alert
  • Residual risk: Fargate pods — GuardDuty Runtime agent not available on Fargate; only EKS Audit Log Monitoring applies

AWS Services to Master:
GuardDuty EKS Protection GuardDuty Runtime Monitoring Amazon Inspector ECR
EKS Control Plane Logging Amazon Detective OPA Gatekeeper Network Policies
Pod Security Standards ECR Security Hub

Question 7 — Web Application Firewall and DDoS Protection for a Global SaaS

The Ask:

"Design a WAF and DDoS protection architecture for a SaaS product serving 50 million users
globally. The application runs on ALB + ECS Fargate. The threat model includes OWASP Top 10,
volumetric DDoS, credential stuffing, scraping bots, and L7 DDoS. The design must not
add more than 10ms latency to p99 requests."

Phase 1 — Clarify:

  • Is the application global (multi-region) or single-region behind CloudFront?
  • What is the current traffic profile — steady state RPS, peak RPS, typical request size?
  • Are there known bot patterns that are already identified (user agents, IPs, behaviors)?
  • Is the application API-only (JSON) or does it serve HTML?
  • What is the acceptable false-positive rate for blocking legitimate users?

Phase 2 — Initial Design:

  • Amazon CloudFront as the global CDN and first line of defense
  • AWS Shield Advanced: attached to CloudFront distribution and ALB
  • AWS WAF v2 on CloudFront (for global rules) and on ALB (for regional failover rules)
  • AWS WAF rule groups:
  • AWS Managed Rules: Core Rule Set (CRS), Known Bad Inputs, SQL Database, PHP, Linux
  • AWS Managed Bot Control rule group (targeted mode for low false positives)
  • AWS Managed Fraud Control (Account Takeover Prevention) for login endpoints
  • Custom rate-based rules: block IPs exceeding 2,000 requests per 5 minutes
  • Route 53 with health checks for multi-region failover

Phase 3 — Security Deep Dive:

  • WAF rule priority (lower number = evaluated first):
  • IP reputation list (block known malicious IPs — free)
  • Rate-based rule (burst protection)
  • Geo-blocking rule (block countries with zero legitimate users)
  • Bot Control managed rule group
  • Account Takeover Prevention rule (on /login, /auth paths only)
  • Core Rule Set (OWASP Top 10)
  • Custom application rules (e.g., block non-JSON content-type on API endpoints)
  • WAF logging: full request/response logging to S3 via Kinesis Firehose; WAF sampled requests in console for investigation
  • Shield Advanced: DRT (DDoS Response Team) access — IAM role granting DRT access to WAF and Shield configurations during attacks
  • Shield Advanced: proactive engagement — AWS proactively contacts you when an attack is detected against protected resources
  • CloudFront: X-Forwarded-For header forwarded to ALB for origin-level logging; Cache behavior for static assets reduces origin load during DDoS
  • ALB access logs to S3 — Athena queries for attack forensics

Phase 4 — Scale & Resilience:

  • CloudFront 450+ PoPs absorb volumetric attack before it reaches origin
  • WAF capacity units (WCU): plan WAF rule WCU consumption; limit is 5,000 WCU per WebACL by default
  • Shield Advanced cost model: $3,000/month base + data transfer out — ensure business case is approved
  • Multi-region: deploy identical WAF WebACL in secondary region; use Terraform for WAF-as-code (no manual rule management)
  • WAF rate-based rules aggregate by IP by default; also configure aggregation by forwarded-ip for cases behind proxies

Phase 5 — Threat Model & Operations:

  • Threat: credential stuffing on login endpoint → Account Takeover Prevention rule: velocity check on failed logins per IP/username; CAPTCHA challenge after threshold
  • Threat: L7 DDoS targeting a specific expensive API endpoint → custom rate-based rule scoped to that URL path; Shield Advanced DRT engagement
  • Threat: attacker rotates IPs to evade rate limiting → Bot Control behavioral analysis (not just IP-based); JA3 fingerprinting via CloudFront
  • Threat: WAF rule bypassed via obfuscated SQL injection → WAF CRS transformations (URL decode, HTML entity decode applied before rule matching); regular WAF penetration test
  • Detection: WAF BLOCK metric spike → CloudWatch alarm → PagerDuty
  • Detection: Shield Advanced attack notification → SNS → CISO notification
  • Residual risk: WAF false positives blocking legitimate users — measure BLOCK vs COUNT ratio; tune rules in COUNT mode before switching to BLOCK

AWS Services to Master:
AWS WAF v2 Shield Advanced CloudFront Managed Bot Control Account Takeover Prevention
Kinesis Firehose (WAF logs) Route 53 ALB Access Logs Athena

Question 8 — Automated Vulnerability Management and Patch Compliance

The Ask:

"Design an automated vulnerability management and patch compliance platform for an AWS
estate of 5,000 EC2 instances (Windows and Linux) and 300 container images in ECR, across
50 AWS accounts. SLA: critical CVEs patched within 24 hours, high CVEs within 7 days.
Automated patching must not cause production downtime."

Phase 1 — Clarify:

  • Are EC2 instances in Auto Scaling Groups (ASG) — can instances be terminated and replaced?
  • What is the change management process — is patching considered a change requiring approval?
  • Are container images built by internal teams or pulled from public registries?
  • What is the blast radius tolerance — can 1 instance per ASG be patched at a time?
  • Is there an existing CMDB that must be updated with patch status?

Phase 2 — Initial Design:

  • Amazon Inspector v2: continuous CVE scanning for EC2 (via SSM agent) and ECR container images
  • AWS Systems Manager Patch Manager: automated patching for EC2 instances
  • SSM Patch Baseline: custom baseline per OS (Amazon Linux 2, Ubuntu, Windows Server) with severity thresholds
  • SSM Maintenance Window: scheduled patching during low-traffic periods
  • SSM Run Command for immediate emergency patching
  • EventBridge: Inspector CRITICAL finding → Lambda → trigger immediate Patch Manager run on affected instance
  • AWS Security Hub: Inspector findings aggregated across 50 accounts

Phase 3 — Security Deep Dive:

  • Inspector v2 findings include: CVE ID, CVSS score, exploitability (EPSS score), affected package, fixed version
  • Prioritize by EPSS (Exploit Prediction Scoring System) not just CVSS — focus on exploitable CVEs first
  • Patch Baseline: CRITICAL severity → Auto-approve after 0 days (immediate); HIGH → auto-approve after 3 days

TTPs mapped to MITRE ATT&CK, detect multi-stage attack campaigns that span multiple
log sources, and provide a 72-hour incident reconstruction capability for IR teams."

Phase 1 — Clarify:

  • What external threat intelligence feeds are currently subscribed to (commercial TI, ISACs, OSINT)?
  • Are on-premises systems sending logs via Syslog, SNMP, or agent-based?
  • What is the volume of events per second at peak?
  • Is there an existing SIEM or is this greenfield?
  • What is the IR team's primary investigative tool (preferred query language)?

Phase 2 — Initial Design:

  • Amazon Security Lake: OCSF-normalized log store (foundation for all log sources)
  • CloudTrail, VPC Flow Logs, Route 53, EKS, Lambda, Security Hub → Security Lake (native sources, automatic)
  • SaaS logs (Okta, Salesforce, GitHub) → Firehose → Lambda (OCSF transform) → Security Lake custom source
  • On-premises logs → Kinesis Agent on log shipper → Kinesis Data Streams → Lambda (OCSF transform) → Security Lake
  • Amazon OpenSearch Service: Security Lake subscriber; streams logs for hot search and correlation
  • Amazon Detective: graph-based investigation across GuardDuty, CloudTrail, VPC Flow Logs
  • AWS Threat Intelligence: Security Hub ingests threat intel from AWS and third-party partners

Phase 3 — Security Deep Dive:

  • MITRE ATT&CK mapping: OpenSearch Detection rules tagged with ATT&CK technique IDs
  • TI enrichment pipeline: Lambda subscribes to STIX/TAXII TI feed → parses IOCs (IPs, domains, hashes) → writes to DynamoDB TI table
  • Real-time enrichment: EventBridge → Lambda intercepts Security Hub finding → queries DynamoDB TI table → adds ThreatIntelMatch tag to finding → enriched finding back to Security Hub
  • Multi-stage attack correlation: OpenSearch correlation rules — if InitialAccess finding AND CredentialAccess finding on same instance within 1 hour → generate HIGH campaign finding
  • Amazon Detective: automatically builds entity graph (EC2 instances, IAM roles, IPs, S3 buckets) and their relationships over rolling 1-year window; IR analyst queries "show me all activity by this IAM role in the last 72 hours"
  • Security Lake query for 72-hour IR reconstruction: Athena query over Security Lake S3 partitioned by account/region/date → reconstruct full timeline of API calls, network connections, DNS queries, container events

Phase 4 — Scale & Resilience:

  • Security Lake: S3 storage automatically scales; Glue partitioning ensures query performance as data grows
  • OpenSearch: dedicated master nodes + hot/warm/cold tiering (hot: 7 days, warm: 30 days, cold: 90 days via UltraWarm)
  • TI enrichment DynamoDB: on-demand capacity; TTL set on IOC records to auto-expire stale TI
  • At 50 accounts × multiple log sources: Security Lake handles ingestion; Glue Crawler maintains schema as new log types added
  • Subscription model: multiple subscribers (OpenSearch, Athena, third-party SIEM) can all read from Security Lake simultaneously

Phase 5 — Threat Model & Operations:

  • Threat: attacker moves slowly (low-and-slow APT) to evade per-event detection → Detective's entity timeline identifies anomalous behavioral shift over weeks; UEBA baseline comparison
  • Threat: attacker deletes CloudTrail to destroy evidence → SCP blocks DeleteTrail; Security Lake already has a copy of all logs before deletion event; S3 Object Lock prevents retroactive deletion
  • Threat: TI feed is poisoned (false IOCs added to block legitimate services) → TI enrichment Lambda validates IOC source reputation before writing; alert on any TI-triggered block of an AWS IP range (known legitimate)
  • Detection: Security Lake + OpenSearch: query for MITRE T1078 (Valid Accounts) technique across 5 or more accounts within 24 hours → APT lateral movement indicator
  • Detection: Detective investigation shows an IAM role assumed from a new ASN with no prior history → GuardDuty IAMUser/AnomalousBehavior + Detective behavioral baseline
  • Residual risk: log ingestion latency (5–15 min for some sources) means a fast-moving attacker could complete a phase before detection; mitigate with preventive controls (SCP, SCG, IMDSv2) to slow attacker
