High-risk credential theft, ticket-forging, and destructive actions are described at the technique level and paired with safe validation or detection commands rather than unrestricted weaponization.

2. Highest-Risk Design Decision

The most important design decision is trust direction.

Preferred architecture: Acquired forest trusts Your forest.

That means:

  • Your migration/admin identities can access explicitly approved acquired-company resources.
  • Acquired-company users cannot authenticate into your forest by default.
  • Selective Authentication limits even approved identities to specific resources.
  • SID filtering remains enabled so SIDHistory from the acquired forest cannot become authorization material in your forest.

A reverse or bidirectional trust changes the risk posture materially. With a flat acquired network, one phished acquired user can become the starting point for BloodHound path discovery, Kerberoasting, FSP abuse, and eventual Tier 0 compromise.

6. Threat-by-Threat Technical Review

T1 — Phishing an acquired-company user on a flat network

STRIDE category: Spoofing / Initial Access
Risk: High likelihood, High impact
Attacker objective: Obtain a domain foothold inside the acquired forest, then use the trust relationship or local lateral movement to reach privileged credentials.

How the attacker would exploit it

  • The attacker targets acquired-company users because the acquired network has 3,000 users and a flat network; one successful credential theft or endpoint compromise is enough to start broad internal discovery.
  • After initial access, the attacker enumerates domain information from the compromised host: logged-on users, domain controllers, domain trusts, local admin relationships, accessible shares, and service accounts exposed through LDAP.
  • Because the network is flat, the attacker does not need to bypass multiple internal segmentation layers before reaching peer workstations, application servers, or even domain-controller management paths.
  • The next goal is credential collection from endpoint memory, browser/session tokens, cached credentials, or exposed admin sessions. The attacker specifically wants accounts that can reach the acquired DC or can authenticate across the trust.
  • Once an acquired-domain account is established, the attacker tests whether cross-forest authentication is possible, whether selective authentication is enforced, and whether any Foreign Security Principal mappings exist in the acquiring forest.

Blue-team control review — what must stop the chain

  • Phishing-resistant MFA for accounts allowed to cross the trust, not just OTP or push MFA.
  • EDR controls that detect LSASS access, credential-dumping behaviors, suspicious PowerShell, remote service creation, and abnormal LDAP collection.
  • Network segmentation that prevents a random workstation in the acquired network from initiating arbitrary SMB, WinRM, RDP, LDAP, or Kerberos traffic to sensitive subnets.
  • Trust-direction validation: acquired users should not be able to authenticate to acquiring-forest resources unless this is explicitly intended and approved.
  • Baseline acquired-domain authentication two weeks before cutover, then alert on newly active accounts, unusual source hosts, and cross-forest attempts from user workstations.

Evidence and telemetry to validate controls

  • EDR process and memory-access telemetry
  • Windows 4624/4625 logon events
  • DC 4768/4769 Kerberos events
  • LDAP diagnostic logs
  • Network flow from acquired workstation ranges

T2 — Kerberoasting across the trust boundary

STRIDE category: Information Disclosure / Credential Access
Risk: Medium likelihood, High impact
Attacker objective: Request service tickets for SPN-backed service accounts, crack weak service-account passwords offline, then reuse the account for lateral movement or privilege escalation.

How the attacker would exploit it

  • The attacker needs any valid account that can reach a KDC and request service tickets. In a permissive trust or bidirectional trust, this may include acquired-domain users attempting to request TGS tickets for services in the acquiring forest.
  • The attacker enumerates SPNs and targets service accounts with weak or legacy encryption support. RC4-HMAC is especially dangerous because it maps to the NT hash and is far faster to crack offline than AES-based Kerberos material.
  • The KDC returns a TGS encrypted with the service account key. The attacker does not need to contact the target service; cracking happens offline and is invisible after the ticket is obtained.
  • If a password is recovered, the attacker authenticates as the service account, explores where it has local admin, database, deployment, or delegation rights, and may escalate toward Tier 0.
  • High-value service accounts often have broad permissions, old passwords, SPNs, and exceptions from modern controls, making them prime M&A trust targets.

Blue-team control review — what must stop the chain

  • Migrate SPN accounts to gMSA wherever supported so the password is long, random, and automatically rotated.
  • Disable RC4 and DES Kerberos. Enforce AES-only for both forests before the trust becomes operational.
  • Audit SPNs pre-cutover and remove orphaned or unnecessary SPNs. Treat new SPNs as drift that requires review.
  • Alert on Event 4769 bursts: one account requesting many unique SPNs in a short period, especially user accounts and cross-forest principals.
  • Monitor service-account interactive logons or unusual source hosts; service accounts should have tightly defined logon patterns.

Evidence and telemetry to validate controls

  • DC Event 4769
  • Ticket Encryption Type field
  • ServiceName / SPN field
  • AccountName and Client Address
  • Service-account logon events 4624

T3 — AS-REP roasting accounts with Kerberos pre-authentication disabled

STRIDE category: Information Disclosure / Credential Access
Risk: Medium likelihood, Medium-to-High impact
Attacker objective: Obtain password-derived Kerberos material for accounts that do not require pre-authentication, then crack it offline.

How the attacker would exploit it

  • The attacker searches for accounts with “Do not require Kerberos preauthentication” enabled. In legacy acquired AD environments, this may exist for old applications or misconfigured service accounts.
  • For such accounts, the attacker can request an AS-REP without proving knowledge of the password first. The returned material is encrypted with the account password-derived key.
  • The attacker performs offline password cracking. Weak, reused, or non-rotated passwords are the real exposure; the KDC only sees the request, not the cracking activity.
  • If the cracked account is a service account or privileged operator account, the attacker uses it to access systems, request more tickets, or move laterally.
  • The attack is especially useful before trust cutover because it can compromise acquired accounts that later receive migration or cross-forest permissions.

Blue-team control review — what must stop the chain

  • Zero accounts should have pre-authentication disabled unless there is a documented, time-bound exception.
  • Enforce AES-only and long random passwords, especially for service accounts that cannot immediately be converted to gMSA.
  • Alert on Event 4768 where pre-authentication type is absent or failure patterns suggest enumeration.
  • Review accounts with legacy flags before trust creation; block cutover until high-risk accounts are remediated.
  • Rotate passwords for any account discovered with pre-auth disabled after exposure.

Evidence and telemetry to validate controls

  • DC Event 4768
  • Pre-authentication type
  • Encryption type
  • Account flags from AD inventory
  • Password age and service-account inventory

T4 — BloodHound enumeration of Foreign Security Principal paths

STRIDE category: Elevation of Privilege
Risk: High likelihood if trust is permissive, Critical impact
Attacker objective: Discover a graph path where an acquired-forest identity is represented as an FSP in the acquiring forest and inherits privileged access through group membership, ACLs, delegation, or local admin rights.

How the attacker would exploit it

  • The attacker runs AD graph collection from a compromised domain-joined host. LDAP read operations reveal users, groups, ACLs, sessions, local admin relationships, delegation settings, SPNs, and trust relationships.
  • The attacker specifically searches for Foreign Security Principal objects in the acquiring forest that map to acquired users or groups.
  • A dangerous path exists if an FSP is a member of a privileged group, has local admin on servers, can modify a GPO, has delegated rights over an OU, or can reach Tier 0 through nested group memberships.
  • The attacker authenticates as the acquired principal and exercises the access exactly as AD intended. No exploit is needed; the misconfiguration is the vulnerability.
  • Once on an acquiring-forest server, the attacker steals cached credentials or tokens, pivots to higher-value systems, and attempts DCSync or GPO-based code execution.

Blue-team control review — what must stop the chain

  • Enumerate every FSP before trust cutover and review all group memberships and ACLs.
  • Run BloodHound defensively against both forests and close attack paths before enabling the trust.
  • Block FSP membership in privileged groups through policy-as-code and change approval.
  • Use selective authentication so possession of a valid trusted identity does not imply the right to authenticate to every resource.
  • Alert on bulk LDAP collection from non-DC hosts and on new FSP group membership changes.

Evidence and telemetry to validate controls

  • LDAP diagnostic logs
  • Event 4662 object access
  • AD group membership change events 4728/4732/4756
  • FSP object inventory
  • BloodHound differential reports

T5 — NTLM relay from the flat acquired network

STRIDE category: Spoofing / Credential Relay
Risk: High likelihood in legacy AD, High impact
Attacker objective: Coerce or capture NTLM authentication from a victim and relay it to a different service to authenticate without knowing the password.

How the attacker would exploit it

  • The attacker positions a host on the acquired flat network and waits for or induces NTLM authentication using name-resolution abuse, SMB interactions, printer/service coercion, or application fallback to NTLM.
  • Because NTLM lacks strong mutual authentication, the attacker relays the challenge-response to another service that accepts NTLM.
  • If SMB signing or channel binding is not required, the target service may accept the relayed authentication and create a session as the victim.
  • Relayed authentication can be used to access file shares, LDAP modify paths, web apps using Windows Integrated Authentication, or management interfaces.
  • If the victim is privileged or the relay target exposes sensitive operations, the attacker can create persistence, modify delegation settings, or obtain code execution.

Blue-team control review — what must stop the chain

  • Disable NTLMv1 immediately and restrict NTLM broadly across both forests; do not allow NTLM across the trust boundary.
  • Enforce SMB signing, LDAP signing, LDAP channel binding, and Extended Protection for Authentication on web services.
  • Disable LLMNR/NBNS/mDNS where not required and monitor for responder-style poisoning patterns.
  • Segment the acquired network so arbitrary workstation-to-workstation and workstation-to-server authentication cannot be coerced at scale.
  • Alert on NTLM authentication from unusual clients, especially from acquired subnets into acquiring resources.

Evidence and telemetry to validate controls

  • Netlogon logs
  • Windows 4624 authentication package
  • SMB/LDAP server logs
  • Network flow for 445/139/389/636
  • Name-resolution anomaly logs

T6 — SIDHistory injection when SID filtering is disabled

STRIDE category: Elevation of Privilege
Risk: Low likelihood if controls are correct, Critical impact if disabled
Attacker objective: Make an acquired-forest principal present privileged SIDs from the acquiring forest so authorization decisions treat it as an acquiring-forest privileged identity.

How the attacker would exploit it

  • The attacker first compromises administrative control in the trusted/acquired forest, or abuses a migration process that can write SIDHistory.
  • The attacker adds a high-value SID from the acquiring forest, such as a privileged group SID, into the SIDHistory attribute of an acquired-domain account.
  • If SID filtering is disabled on the forest trust, the cross-forest PAC may carry this SID into the acquiring forest.
  • When the user accesses resources in the acquiring forest, authorization checks may see the injected privileged SID and grant access.
  • This is catastrophic because it converts compromise of the weaker acquired forest into effective privilege in the acquiring forest without stealing acquiring-forest credentials.

Blue-team control review — what must stop the chain

  • SID filtering/quarantine must remain enabled on the trust, with no exception for migration convenience.
  • Monitor Event 4738 and directory changes for SIDHistory modifications, especially outside migration windows.
  • Require two-person approval for any migration operation capable of modifying SIDHistory.
  • Validate trust attributes continuously using IaC drift detection or scheduled compliance checks.
  • Prefer resource re-ACLing and controlled group mapping over preserving broad SIDHistory during M&A.

Evidence and telemetry to validate controls

  • Trust configuration audit
  • Event 4738 user changed
  • Directory change auditing
  • PAC/SID anomaly enrichment
  • Privileged group SID reference inventory

T7 — Trust key / TDO extraction via DCSync or NTDS.DIT compromise

STRIDE category: Spoofing / Credential Theft
Risk: Medium likelihood after acquired-domain compromise, Critical impact
Attacker objective: Steal the inter-realm trust secret from the Trusted Domain Object so referral tickets can be forged.

How the attacker would exploit it

  • The attacker escalates to domain-admin-equivalent privileges in either forest, commonly by credential theft, Kerberoasting, over-privileged admin paths, or DC compromise.
  • With replication privileges or direct DC access, the attacker extracts directory secrets, including trust material stored in the TDO and potentially krbtgt or service-account keys.
  • The trust key is valuable because it signs cross-forest referral tickets. Possession of the key weakens the boundary between forests even if user passwords are later rotated.
  • The attacker can then forge inter-realm tickets that appear to be issued as part of the trust relationship.
  • This attack is hard to detect after the key is stolen; prevention and immediate trust-key rotation are critical.

Blue-team control review — what must stop the chain

  • Treat any DC compromise in either forest as possible trust-key compromise.
  • Detect DCSync using Event 4662 replication rights from non-DC accounts or hosts.
  • Audit access to the TDO object and monitor unexpected reads/changes by non-DC or non-admin principals.
  • Limit replication permissions to DCs and explicitly known sync services such as Entra/AAD Connect, with SID-based allowlists.
  • Rekey or disable the trust after suspected exposure; do not rely only on endpoint cleanup.

Evidence and telemetry to validate controls

  • Event 4662 replication GUIDs
  • TDO object access audit
  • DC EDR telemetry
  • Directory replication allowlist
  • Trust change events

T8 — Forged inter-realm TGT / Trust Golden Ticket

STRIDE category: Spoofing + Elevation of Privilege
Risk: Low-to-Medium likelihood, Critical impact
Attacker objective: Use a stolen trust key to forge cross-forest referral tickets and access resources without normal KDC issuance from the trusted side.

How the attacker would exploit it

  • After stealing the inter-realm trust key, the attacker forges a referral ticket that claims an identity and authorization data acceptable across the forest trust.
  • The attacker may set ticket parameters that are unusual: unexpected lifetimes, encryption types, flags, client addresses, or SIDs.
  • Because the forged artifact is not produced through the expected KDC flow, classic authentication logs may be incomplete or inconsistent.
  • If SID filtering, selective authentication, and AES-only enforcement are weak, the forged referral may be accepted and converted into service access.
  • The attack can persist until the trust key is rotated or the trust is disabled.

Blue-team control review — what must stop the chain

  • Prevent key theft first: protect DCs, TDOs, and replication rights.
  • Enable SID filtering and selective authentication so forged authorization material has fewer opportunities to succeed.
  • Enforce AES-only and alert on RC4 in cross-forest tickets after enforcement date.
  • Use correlation: cross-realm referral events, service tickets, host logons, and missing expected precursor events.
  • Have a tested playbook to disable/rekey the trust rapidly when trust-key compromise is suspected.

Evidence and telemetry to validate controls

  • Event 4770 cross-realm referrals
  • Event 4769 service ticket requests
  • Ticket encryption type
  • Ticket lifetime and flags
  • Host Event 4624 correlation

T9 — Golden Ticket using the acquiring forest krbtgt hash

STRIDE category: Spoofing + Elevation of Privilege
Risk: Low likelihood until Tier 0 compromise, Critical impact
Attacker objective: Use the target forest krbtgt secret to mint arbitrary TGTs and impersonate any identity or group membership in the forest.

9. Suggested Detection Logic in Pseudocode

9.1 DCSync from non-DC

WHEN EventID = 4662
AND ObjectAccess contains any of:
  - Replicating Directory Changes
  - Replicating Directory Changes All
  - Replicating Directory Changes In Filtered Set
AND SubjectAccount NOT IN known_domain_controllers
AND SubjectAccount NOT IN approved_sync_accounts
THEN severity = Critical
ACTION = isolate source host, disable source account, start domain compromise playbook

9.2 Kerberoasting burst

WHEN EventID = 4769
GROUP BY AccountName, 10-minute window
COUNT_DISTINCT(ServiceName) > 5
AND AccountName NOT IN approved_orchestration_or_scanner_accounts
THEN severity = High
ACTION = disable account if user account; rotate exposed SPN account passwords if confirmed

9.3 RC4 in cross-forest Kerberos after enforcement

WHEN EventID = 4769
AND TicketEncryptionType = 0x17
AND AccountDomain or ServiceDomain indicates cross-forest path
AND timestamp > AES_ENFORCEMENT_DATE
THEN severity = Critical
ACTION = validate policy drift, investigate forged/referral-ticket possibility, review trust-key exposure

9.4 Silver Ticket suspicion

WHEN HostEventID = 4624
AND LogonType = 3
AND AuthenticationPackage = Kerberos
AND no matching DC EventID = 4769 exists
    for same AccountName + ServiceName + time window +/- 5 minutes
THEN severity = High
ACTION = isolate host, rotate service-account key, verify log coverage before closing

9.5 FSP privileged membership

WHEN AD group membership changes
AND AddedMember objectClass = foreignSecurityPrincipal
AND TargetGroup IN privileged_or_tier0_group_list
THEN severity = Critical
ACTION = remove membership, suspend change, require IAM/security review

10. Final Security Position

The trust itself is not the only risk; the real risk is combining:

  • a flat acquired network,
  • legacy authentication,
  • permissive or bidirectional trust direction,
  • stale Foreign Security Principal mappings,
  • weak service accounts,
  • and incomplete DC telemetry.

The strongest security position is to treat trust creation as a production Tier 0 change. It should be gated by automated validation, not by manual assurance. If any of the required gates fail, the trust should not go live.

Reviewer recommendation: Approve the trust only if it is one-way in the safer direction, SID filtering and selective authentication are enforced, legacy authentication is eliminated or blocked at the boundary, FSP attack paths are remediated, and all Tier 0 telemetry is flowing before cutover.
