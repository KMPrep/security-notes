Scenario 2 — Active Directory Domain Trust with Acquired Company (Post-M&A)

Context: Post-M&A, your organization creates a one-way forest trust between your AD forest and the acquired company's legacy AD. The acquired company has 3,000 users and a flat network.

0. KEY AD CONCEPTS — Foundations for This Scenario

This section covers the Active Directory building blocks you need before the rest of this document makes sense. Treat it as a reference layer — every term used in sections 1–8 is defined here.

0.1 What Is Active Directory?

Active Directory (AD) is Microsoft's directory service — a centralized identity and access management system for Windows environments. It stores information about users, computers, groups, and policies, and it controls who can authenticate and what they can access across a Windows network.

AD runs on Domain Controllers (DCs) — Windows Server machines that host the AD database and handle authentication requests. Every organization using AD has at least one DC; large organizations have many.

flowchart TD
    subgraph AD[Active Directory]
        DC1[Domain Controller 1]
        DC2[Domain Controller 2]
        NTDS[(NTDS.DIT - AD Database)]
        DC1 --- NTDS
        DC2 --- NTDS
    end
    User[User Workstation] -->|Authentication Request| DC1
    Server[App Server] -->|Resource Access| DC1

0.2 Core Structural Units

Domain

A domain is the basic administrative unit in AD. It is a logical grouping of users, computers, groups, and policies that share the same AD database. Every domain has a DNS name (e.g., corp.example.com) and is managed by one or more Domain Controllers.

  • Users and computers are members of a domain.
  • All authentication within a domain is handled by the domain's DCs.
  • A domain has its own security boundary — policies, passwords, and trusts are domain-scoped.

Tree

A tree is a collection of domains that share a contiguous DNS namespace. For example: example.com, corp.example.com, and sales.example.com form a tree. Domains in a tree automatically have two-way transitive trusts with each other — a user in corp.example.com can (subject to permissions) access resources in sales.example.com.

Forest

A forest is the top-level container in AD — a collection of one or more trees that share a common schema and Global Catalog, but not necessarily a contiguous namespace. The first domain created in a forest becomes the forest root domain.

The forest is the true security boundary in AD. Trusts between forests are not automatic — they must be explicitly configured and are a high-risk operation.

flowchart TD
    ForestRoot[Forest Root: example.com]
    Tree1Root[corp.example.com]
    Tree2Root[sales.example.com]
    Child1[eu.corp.example.com]
    Child2[us.corp.example.com]
    ForestRoot --- Tree1Root
    ForestRoot --- Tree2Root
    Tree1Root --- Child1
    Tree1Root --- Child2

Organizational Unit (OU)

An OU is a container inside a domain used to organize objects (users, computers, groups) for administrative delegation and Group Policy application. OUs do not have their own security boundary — they are organizational, not security, units.

Schema

The schema defines the structure of every object in AD — what attributes a user object has, what a computer object looks like, etc. The schema is forest-wide and shared across all domains in a forest. Schema changes are rare and irreversible.

Global Catalog (GC)

The Global Catalog is a partial replica of every object in the entire forest, stored on designated DCs (Global Catalog servers). It enables forest-wide searches and is required for cross-domain logon. The GC stores a subset of attributes for all objects — enough to locate and identify them without querying every domain separately.

0.4 The AD Database: NTDS.DIT

NTDS.DIT is the actual AD database file, stored at C:\Windows\NTDS\ntds.dit on every DC. It contains:

  • All user accounts and their password hashes (NT hashes and optionally LM hashes)
  • All group memberships
  • All computer accounts
  • All trust objects (TDOs)
  • SIDHistory for migrated accounts
  • Kerberos key material including the krbtgt account hash

If an attacker extracts NTDS.DIT (via DCSync, Volume Shadow Copy, or physical access), they obtain every password hash in the domain — equivalent to owning the entire domain. This is why DCs are the highest-value targets.

flowchart LR
    Attacker -->|DCSync or VSS| DC[Domain Controller]
    DC -->|Dumps| NTDS[NTDS.DIT]
    NTDS -->|Contains| Hashes[All NT Hashes]
    NTDS -->|Contains| krbtgt[krbtgt Hash]
    NTDS -->|Contains| TDO[Trust Keys]
    krbtgt -->|Enables| GoldenTicket[Golden Ticket Forgery]
    TDO -->|Enables| TrustGoldenTicket[Trust Golden Ticket Forgery]

0.5 Authentication: NTLM vs. Kerberos

AD supports two primary authentication protocols. Understanding both is critical because their weaknesses drive most AD attacks.

NTLM (NT LAN Manager)

NTLM is a challenge-response protocol. The client proves it knows the password by responding to a server challenge using the NT hash of the password — the password never travels on the wire, but the hash does (effectively).

sequenceDiagram
    participant C as Client
    participant S as Server
    participant DC as Domain Controller
    C->>S: Negotiate NTLM
    S->>C: Challenge (random nonce)
    C->>S: Response (NT hash + challenge)
    S->>DC: NetLogon verification request
    DC->>S: Accept/Deny
    S->>C: Access granted/denied

Why NTLM is dangerous:

  • The NT hash alone is enough to authenticate (Pass-the-Hash).
  • NTLM v1 uses 56-bit DES; the response can be cracked in seconds with rainbow tables.
  • NTLM is vulnerable to relay attacks — an attacker can capture a challenge-response from a victim and relay it to a different server.
  • NTLM does not provide mutual authentication — the client cannot verify the server's identity.

Kerberos

Kerberos is a ticket-based authentication protocol. Clients obtain a Ticket Granting Ticket (TGT) from the Key Distribution Center (KDC) at login, then use the TGT to request Service Tickets (TGS) for specific resources — without sending password material again.

sequenceDiagram
    participant C as Client
    participant KDC as KDC (Domain Controller)
    participant Svc as Service / Resource
    C->>KDC: AS-REQ (Authentication Service Request)
    KDC->>C: AS-REP (TGT - encrypted with krbtgt hash)
    C->>KDC: TGS-REQ (Request service ticket, presents TGT)
    KDC->>C: TGS-REP (Service Ticket - encrypted with service account hash)
    C->>Svc: AP-REQ (Presents Service Ticket)
    Svc->>C: AP-REP (Access granted)

Key Kerberos objects:

Object What It Is Attack If Stolen
krbtgt account Special AD account whose hash is used to encrypt and sign all TGTs in the domain Golden Ticket — forge any TGT for any user, valid for the ticket lifetime
TGT (Ticket Granting Ticket) Proof that the user authenticated; presented to KDC to request service tickets Pass-the-Ticket — steal and reuse an existing TGT
Service Ticket (TGS) Proof that the user is allowed to access a specific service; presented to the service Silver Ticket — forge a service ticket offline using the service account hash, bypassing the KDC entirely
SPN (Service Principal Name) Identifier that associates a service with the AD account running it Kerberoasting — request a TGS for any SPN, crack the service account password offline
KDC (Key Distribution Center) The DC role that issues TGTs and service tickets Compromise the DC → compromise the entire domain

0.7 Trusts — How They Work

A trust is a relationship between two domains or forests that allows users authenticated in one to access resources in the other. Trusts are the mechanism that makes cross-forest authentication possible — and also the mechanism that makes cross-forest attacks possible.

Trust Properties

Property Options Security Implication
Direction One-way or Two-way One-way: only users from the trusted side authenticate to the trusting side. Two-way: both sides authenticate to each other — doubles attack surface.
Transitivity Transitive or Non-transitive Transitive trusts chain: if A trusts B and B trusts C, A transitively trusts C. Forest trusts are non-transitive by default — A trusting B does not make A trust C.
Type Domain trust, Forest trust, External trust, Shortcut trust, Realm trust Forest trusts are the most powerful and most dangerous — they extend trust to the entire forest, not just one domain.
SID Filtering On (quarantine) or Off When ON, SIDHistory attributes from the trusted forest are stripped — prevents a compromised account from claiming SIDs it shouldn't have in your forest. Should ALWAYS be on for forest trusts.
Selective Authentication On or Off When ON, only principals explicitly granted "Allowed to Authenticate" on specific resources can use the trust. Without it, any user in the trusted forest can attempt to authenticate to any resource.

Trust Direction Intuition

flowchart LR
    subgraph TrustingForest[Trusting Forest - YOUR Forest]
        YourResources[Your Resources]
    end
    subgraph TrustedForest[Trusted Forest - ACQUIRED]
        AcqUsers[Acquired Users]
    end
    AcqUsers -->|Can authenticate to| YourResources
    TrustingForest -->|"Trusts"| TrustedForest

"Your forest trusts the acquired forest" means: acquired users can come in. This is the risky direction. Reverse it when possible.

The Inter-Realm Trust Key (TDO Secret)

When a trust is created, both DCs share a trust key — a secret stored in the Trusted Domain Object on both sides. This key signs the cross-forest referral tickets. If an attacker extracts this key from either DC, they can forge referral tickets without touching either KDC — this is the Trust Golden Ticket attack (also called Inter-Realm Ticket Forgery).

0.9 SIDHistory — What It Is and Why It Is Dangerous

When users are migrated from one domain to another, their old SID (Security Identifier — the unique identifier for every AD object) can be preserved in the SIDHistory attribute of their new account. This allows the new account to still access resources ACL'd to the old SID during the migration transition.

Why it is a security risk:

  • If SID filtering is disabled on a trust, a user in the trusted forest can have a SIDHistory value claiming to be a privileged SID from the trusting forest (e.g., Domain Admins SID).
  • The trusting forest's KDC will include that SID in the PAC (Privileged Attribute Certificate) of the ticket, granting the user elevated rights in the trusting forest.
  • An attacker who controls the trusted forest's DC can inject arbitrary SIDs into any account's SIDHistory.

Defense: SID filtering (quarantine mode) on forest trusts strips all SIDHistory values from cross-forest tickets. It must never be disabled.

0.10 Group Policy (GPO) — Security Relevance

Group Policy is the mechanism AD uses to enforce security settings, software deployments, and login scripts across all domain-joined machines. GPOs are linked to OUs, Sites, or the entire domain.

Security-critical GPO settings referenced in this document:

GPO Setting Purpose Where to Apply
Network security: LAN Manager authentication level = 5 Disable NTLM v1 and v2; require NTLMv2 minimum Domain-wide; all DCs and member servers
Kerberos encryption types = AES128 + AES256 only Disable RC4-HMAC and DES Kerberos Domain-wide
Audit Kerberos Service Ticket Operations = Success + Failure Generate Events 4769 for all TGS requests All DCs
Audit Kerberos Authentication Service = Success + Failure Generate Events 4768 for all AS requests All DCs
Audit Directory Service Access = Success + Failure Generate Event 4662 for AD object access All DCs
Credential Guard enabled Isolates LSASS secrets in a Hyper-V protected process; prevents Mimikatz from reading credentials All Tier 0 and Tier 1 machines

0.11 Replication — DCSync Attack Surface

All DCs in a domain replicate their copy of NTDS.DIT with each other using the Directory Replication Service (DRS). Any DC can request a replication pull from any other DC using the MS-DRSR protocol.

DCSync abuses this: an attacker with an account that has the Replicating Directory Changes All right can impersonate a DC and request a full replication pull — dumping all password hashes, krbtgt, and trust keys from the DC without ever logging into it directly.

Event 4662 with the replication rights 1131f0aa (Replicating Directory Changes) or 1131f0ad (Replicating Directory Changes All) from a non-DC host is the primary detection signal.

0.12 LSASS — The Credential Store in Memory

LSASS.exe (Local Security Authority Subsystem Service) is the Windows process that handles authentication. It keeps in memory:

  • NT hashes of recently logged-on users
  • Kerberos tickets (TGTs and service tickets)
  • NTLM tokens
  • Potentially plaintext credentials for accounts using WDigest (legacy)

Tools like Mimikatz read LSASS memory to extract these credentials — this is the primary lateral movement mechanism. Credential Guard (Windows 10+/Server 2016+) moves credential secrets into a Hyper-V isolated process that Mimikatz cannot read, even as SYSTEM.

0.13 BloodHound — Attack Path Visualization

BloodHound is an open-source tool that maps AD attack paths by collecting data via SharpHound (a .NET collector that runs LDAP queries against AD) and visualizing the graph of relationships between users, groups, computers, and ACLs.

BloodHound answers questions like:

  • "Which users have a path to Domain Admins?"
  • "Which Foreign Security Principals are in privileged groups?"
  • "Which service accounts have unconstrained delegation?"

In a post-M&A trust scenario, BloodHound is the first tool a sophisticated attacker runs after gaining a foothold — it maps all cross-forest attack paths through FSP group memberships. Running BloodHound yourself before the trust goes live is a required defensive step.

3. TRUST BOUNDARIES — Where does trust change?

3.1 Recommended Trust Architecture (One-Way: Acquired Trusts Your Forest)

The critical decision is trust direction. The recommended configuration is:

  • Acquired forest trusts Your forest (not the reverse)
  • Your admins can authenticate to acquired resources (for migration and management)
  • Acquired users cannot authenticate to your resources
  • Selective Authentication enabled — even your forest users can only access explicitly approved resources in the acquired forest
flowchart LR
    subgraph YourForest[Your Forest - Acquiring Org]
        YourAdmin[Your Admins]
        YourDC[Your DC - KDC]
        YourRes[Your Resources]
    end
    subgraph AcqForest[Acquired Forest - Legacy AD - Flat Network]
        AcqDC[Acquired DC - KDC]
        AcqUser[Acquired Users - 3000]
        AcqRes[Acquired Resources]
    end
    YourAdmin -->|AS-REQ| YourDC
    YourDC -->|Cross-forest referral - ALLOWED| AcqDC
    AcqDC -->|Selective auth check| AcqRes
    AcqUser -->|AS-REQ| AcqDC
    AcqDC -.->|Cross-forest referral - BLOCKED| YourDC
    AcqDC -.->|No access| YourRes
    YourDC --> SIEM[SIEM]
    AcqDC --> SIEM

3.2 Why Bidirectional Trust Is Dangerous

A bidirectional trust doubles the attack surface. Any compromise in the flat acquired network immediately becomes a vector into your forest.

flowchart TD
    AcqCompromised[Compromised Acq Account] -->|Bidirectional trust referral| YourDC[Your Forest DC]
    AcqCompromised -->|Kerberoast across trust| YourSPN[Your SPNs]
    AcqCompromised -->|BloodHound - find FSP memberships| FSP[FSP in Your Forest Admin Group]
    FSP -->|Group membership| YourDA[Your Domain Admin group]
    YourDA -->|DCSync| KrbtgtHash[Your krbtgt hash]
    KrbtgtHash -->|Forge| GoldenTicket[Golden Ticket - unlimited persistence]

With a bidirectional trust, the chain from "compromised acquired user on a flat network" to "Golden Ticket for your forest" can complete in under 24 hours by an advanced attacker.

3.3 Kerberos Cross-Forest Authentication Flow

Understanding the exact protocol flow is essential for knowing where to instrument detection.

sequenceDiagram
    participant C as Client in Your Forest
    participant YourKDC as Your DC KDC
    participant AcqKDC as Acquired DC KDC
    participant Svc as Acquired Resource
    C->>YourKDC: AS-REQ - get TGT for Your Forest
    YourKDC-->>C: TGT - Your Forest
    C->>YourKDC: TGS-REQ for resource in Acquired Forest
    YourKDC-->>C: Referral TGT - inter-realm - signed with trust key
    C->>AcqKDC: TGS-REQ with referral TGT
    AcqKDC-->>C: Service ticket with PAC - selective auth checked
    C->>Svc: AP-REQ
    Svc-->>C: AP-REP

The inter-realm trust key (shared between the two KDCs via the TDO — Trusted Domain Object) is the highest-value secret in this architecture. If an attacker extracts this key, they can forge cross-forest referral tickets without touching either DC.

6.2 Detection Rules

Rule 1 — Golden Ticket (Anomalous TGT Lifetime)

Alert when Event 4768 TGT lifetime exceeds 600 minutes (policy max)
OR when a TGS request (Event 4769) references an account
with no prior AS-REQ (Event 4768) within the past 12 hours.
  • Severity: Critical
  • MITRE: T1558.001
  • Source: DC Security Event Log
  • False positives: Service accounts with legitimately long session tokens (low volume — investigate each)
  • Tuning: Maintain a lookup table of accounts known to have extended ticket lifetimes; alert on anything not in the lookup

Rule 2 — Trust Golden Ticket (RC4 in Cross-Forest Ticket)

Alert when Event 4769 EncryptionType = 0x17 (RC4-HMAC)
occurs within 60 seconds of Event 4770 (cross-realm referral)
for the same account after enforcement date T0 (AES-only policy).
  • Severity: Critical
  • MITRE: T1558.001
  • Source: DC Security Event Log (Your Forest)
  • False positives: Near-zero after AES enforcement; legacy clients will fail authentication, not generate RC4 tickets through the trust
  • Tuning: Alert is effectively zero-FP after AES enforcement; run in audit mode for 2 weeks before enabling page

Rule 3 — Silver Ticket (Logon Without KDC Ticket)

Alert when Event 4624 (logon type 3) on any host occurs
for account X accessing service Y, but no Event 4769 for account X
targeting service Y appears on any DC within the prior 5 minutes.
  • Severity: High
  • MITRE: T1558.002
  • Source: Correlation of host Security Event Log + DC Security Event Log
  • False positives: Medium initially (log collection gaps, clock skew); requires complete DC log coverage
  • Tuning: Requires low-latency log ingestion (<30 seconds); clock synchronization enforced via NTP; accept some noise for first 30 days while tuning log completeness

Rule 4 — Kerberoasting Burst

Alert when a single account (AccountName) generates more than 5 unique
TGS-REQ events (Event 4769) for distinct ServiceNames within 10 minutes.
  • Severity: High
  • MITRE: T1558.003
  • Source: DC Security Event Log
  • False positives: Medium — legitimate service discovery tools, orchestration platforms; tune by allowlisting known orchestrators
  • Tuning: Maintain allowlist of CI/CD service accounts; alert on user accounts immediately without suppression

Rule 5 — DCSync from Non-DC (Critical)

Alert when Event 4662 is generated with AccessMask containing
Replicating Directory Changes All (1131f0ad) or Replicating Directory Changes (1131f0aa)
where the Subject Account is not a domain controller computer account.
  • Severity: Critical
  • MITRE: T1003.006
  • Source: DC Security Event Log (Object Access Audit)
  • False positives: Low — Azure AD Connect, legitimate replication tools are known; add to allowlist
  • Tuning: Allowlist sync accounts by SID; alert immediately for everything else

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and Policy-as-Code

At Google scale, AD forest trust configurations are not managed manually — they are managed as infrastructure code with automated validation gates.

  • Trust configuration as IaC: All trust parameters (SelectiveAuth, SIDFiltering, encryption types, delegation settings) are defined in Terraform or Puppet manifests. Drift detection runs continuously; any out-of-band change triggers an alert and automated revert.
  • Detection-as-code: Detection rules are expressed in YAML/DSL checked into version control. Changes go through code review, unit tests against synthetic log data, staging deployment with canary traffic before production rollout.
  • Pre-cutover automation gate: A check pipeline validates all prerequisites (NTLM v1 zero, AES enforced, all FSPs audited, BloodHound paths resolved, DCs forwarding logs) before trust establishment is permitted. Trust creation fails if any gate fails.
  • Continuous BloodHound validation: SharpHound collection runs nightly in a read-only service account context; new attack paths trigger an automated ticket to IAM for remediation within 24 hours.
  • Automated SPN auditing: Weekly job enumerates all SPNs, verifies gMSA coverage, flags any SPN account with a non-random password older than 30 days.

8.2 Fleet-Wide Impact

A misconfigured trust object or AD policy at Google scale affects millions of users and hundreds of thousands of services simultaneously.

ANSWER SUMMARY — 90-second version

The core security concern here is that a post-M&A forest trust creates a direct credential bridge between your mature, hardened AD forest and an organization with 3,000 users on a flat network — meaning a single phished employee in the acquired company is potentially one BloodHound run away from your Enterprise Admin group. My immediate design decision would be to establish a one-way trust where the acquired forest trusts your forest — not the reverse — so only your migration admins can reach acquired resources, and no acquired user can authenticate to anything in your environment. On top of that, I would enforce selective authentication so even your accounts are restricted to specifically approved resources.

The two most dangerous threats are: first, a BloodHound attack path through a Foreign Security Principal that was inadvertently placed in a privileged group in your forest — this is how a low-sophistication attacker goes from "domain user in a flat network" to "your domain admin" in under an hour. Second, trust Golden Ticket abuse — if the acquired DC is compromised and the inter-realm trust key is extracted, an attacker can forge cross-forest referral tickets indefinitely.

My top three controls before day-one cutover are: SID filtering stays on — no exceptions, no disabling for migration convenience; all service accounts with SPNs migrated to gMSA before trust goes live so Kerberoasting across the boundary yields uncrackable 240-bit hashes; and NTLM v1 is a hard blocker — the trust does not go live until NTLM v1 is zero across the acquired environment. For detection, I would instrument every DC in both forests to forward Event 4768, 4769, 4770, and 4662 to the SIEM with sub-60-second latency, with specific rules for RC4 encryption in cross-forest tickets and honeytoken accounts that would trigger a Critical alert if any ticket is ever issued for them.

The residual risk I would be honest about is this: if the acquired DC was already compromised before we established the trust, an attacker may already have the trust key. That is why pre-cutover forensic investigation of the acquired DC is a non-negotiable gate, not a nice-to-have.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: The acquired company uses NTLM v1. What is your immediate action?

NTLM v1 is a hard blocker for trust establishment — it is not a risk to accept or defer. My immediate action is to push an emergency GPO to the acquired forest setting Network security: LAN Manager authentication level to value 5 — "Send NTLMv2 response only; refuse LM and NTLM" — and simultaneously run a network-level block on NTLM traffic (TCP 445, TCP 139) crossing from the acquired network to my network at the firewall. I then audit every application that breaks as a result: each one gets a 90-day maximum exception window with compensating controls (network isolation, enhanced monitoring), and the trust establishment is blocked until the NTLM v1 count is zero. NTLM v1 uses 56-bit DES; those hashes can be cracked with rainbow tables in under a minute, and relay attacks against them are trivial on a flat network.

Follow-up 2: How do you model "time to breach" risk for this trust relationship?

I model it as a kill chain with probability-weighted stages: Stage 1 is initial access — on a flat network with 3,000 users, assuming a 0.5% phishing success rate, expected time to first compromise is under 7 days. Stage 2 is internal recon and BloodHound collection — on a flat network with no east-west controls, this takes 30 to 60 minutes after initial access. Stage 3 is cross-forest attack path exploitation — if FSP group memberships exist in your forest, this is 1–2 hours. Stage 4 is privilege escalation to your DA — another 30 minutes if the attack path exists. Total T2B with a bidirectional trust and no controls: roughly 8–10 days from zero. My target is to extend that to greater than 30 days through the controls I described, and make every stage noisily detectable. The flat network is the multiplier that makes every other assumption pessimistic — that is why network segmentation between the forests is the first thing I enforce, not the last.

Follow-up 3: What is the krbtgt double-rotation procedure and why does it matter for a Golden Ticket response?

The krbtgt account's password hash is what enables Golden Ticket forgery — an attacker with that hash can forge TGTs for any account with any group membership, valid for the maximum ticket lifetime (10 hours by default). A single rotation is insufficient because existing valid TGTs issued before the rotation are still valid until they expire. A double rotation with a 10-hour wait between rotations ensures all previously issued TGTs — whether legitimate or forged — are invalidated. The procedure is: rotate krbtgt once, wait the full maximum TGT lifetime (10 hours, or your MaxTicketAge policy value), then rotate again. After the second rotation no TGT signed with the old key can be used. The operational impact is that all Kerberos sessions are terminated during the rotation window — users will see authentication prompts; this is expected and necessary. At Google scale, this is a pre-planned SOAR playbook triggered automatically on a confirmed Golden Ticket detection, not a manual procedure.

COMMON MISTAKES TO AVOID

Mistake Why It Is Weak Stronger Answer
Saying "use a bidirectional trust for flexibility during migration" A bidirectional trust with a flat-network acquired company is catastrophically dangerous; "flexibility" is not a security justification. This is the answer that gets people fired post-breach. State clearly: one-way trust with the acquired forest trusting your forest; selective authentication on top; trust direction is a non-negotiable security decision, not a convenience decision.
Treating NTLM v1 as "we'll fix it eventually" NTLM v1 hashes are crackable in under a minute with rainbow tables. On a flat network, an attacker can capture and crack them passively. This is not a deferred risk — it is an active, immediate threat that makes every other control irrelevant. "NTLM v1 is a hard blocker. The trust does not go live until NTLM v1 count is zero. Period."
Saying "enable logging" for Golden Ticket detection without specifying the correlation logic Silver Tickets bypass the KDC entirely — there is no event on the DC when a Silver Ticket is used. Generic "enable logging" advice would miss Silver Tickets completely. Golden Ticket detection requires specific event field analysis (ticket lifetime, encryption type, presence of AS-REQ) not just event volume. Specify Event 4769 + encryption type field 0x17; specify correlation of logon events on hosts without prior KDC tickets; mention honeytoken accounts as zero-FP indicators.
Ignoring the BloodHound FSP problem Saying "segment the networks" addresses lateral movement but does not address the trust-based attack path where a legitimate cross-forest referral ticket leads to admin rights via FSP group membership. A manual control that nobody has time to run is not a control.
