Reviewer role: Senior Cybersecurity Lead, Microsoft-style security architecture review
Scenario: Enterprise decommissions on-prem Proofpoint appliance and migrates to Microsoft Defender for Office 365 Plan 2 for 30,000 mailboxes.

1. Review Objective

This enhanced version adds the technical depth that was missing in the earlier review. It covers:

  1. Attacker lens: How each threat would be exploited in practice, including the reconnaissance steps and abuse path.
  2. Authorized purple-team commands: Safe command examples to validate exposure, simulate benign control tests, and collect evidence.
  3. Blue-team control validation: Microsoft 365, Exchange Online, Entra ID, Defender for Office 365, and Sentinel checks that prove the control is working.
  4. Detection engineering: KQL-style detections and audit-log queries for each threat.
  5. Attack trees: Mermaid diagrams showing exactly where controls stop or contain the attack.

Important safety scope

This document intentionally includes safe validation commands rather than instructions to run real phishing infrastructure, steal session cookies, perform credential stuffing, exfiltrate mail, or bypass Defender scanning. For high-risk scenarios such as AiTM phishing, OAuth abuse, credential stuffing, malicious attachment delivery, and eDiscovery export abuse, the commands are framed as:

  • defensive exposure checks,
  • Microsoft-native simulation or audit workflows,
  • lab-only benign tests,
  • or detection queries.

Where a real attacker would use offensive tooling, this document describes the workflow and gives blue-team-safe ways to validate the same control boundary.

5. Threat-by-Threat Enhanced Technical Review

T1 — AiTM Phishing and Session-Cookie Theft

STRIDE: Spoofing / Credential Access / Session Hijacking
Risk: Critical for finance, executive, HR, IT admin users
Attacker objective: Trick the user into authenticating through an adversary-controlled reverse proxy, capture the post-MFA session cookie, and reuse it to access Exchange Online or Outlook Web Access.

Attacker technical workflow

A real attacker would typically:

  1. Register a lookalike domain or compromise a legitimate site.
  2. Send a migration-themed email: “Validate your Microsoft 365 mailbox after gateway migration.”
  3. Proxy the real Microsoft login flow and relay MFA challenge responses in real time.
  4. Capture the authenticated session cookie after MFA succeeds.
  5. Reuse the session from a new IP/ASN to open Outlook Web Access.
  6. Immediately create persistence: inbox forwarding rule, OAuth consent, mailbox delegate, or message deletion.

Safe authorized validation approach

Do not run AiTM kits in production. Use Microsoft Attack Simulation Training or a controlled phishing simulation platform that does not harvest passwords or tokens.

Validate that high-risk users are using phishing-resistant authentication:

# List Conditional Access policies and confirm Exchange Online is covered.
Get-MgIdentityConditionalAccessPolicy -All |
  Select-Object DisplayName, State, CreatedDateTime, ModifiedDateTime

# Review authentication methods policy.
Get-MgPolicyAuthenticationMethodPolicy | Format-List

# Check FIDO2-related settings where available through Graph module output.
Get-MgPolicyAuthenticationMethodPolicyAuthenticationMethodConfiguration |
  Select-Object Id, State

Check recent sign-ins for high-risk users where MFA was satisfied but the method was not phishing-resistant:

SigninLogs
| where TimeGenerated > ago(7d)
| where AppDisplayName has_any ("Office 365 Exchange Online", "Office 365")
| where UserPrincipalName in~ (dynamic(["<finance_user>@<domain>", "<exec_user>@<domain>"]))
| project TimeGenerated, UserPrincipalName, IPAddress, Location, AppDisplayName,
          ClientAppUsed, ConditionalAccessStatus, RiskLevelDuringSignIn,
          AuthenticationRequirement, AuthenticationDetails

Detect impossible or suspicious session reuse:

SigninLogs
| where TimeGenerated > ago(24h)
| where AppDisplayName has "Office 365"
| where ResultType == 0
| summarize Countries=dcount(Location), IPs=dcount(IPAddress), ASNs=dcount(AutonomousSystemNumber),
            FirstSeen=min(TimeGenerated), LastSeen=max(TimeGenerated),
            SampleIPs=make_set(IPAddress, 5)
  by UserPrincipalName, bin(TimeGenerated, 6h)
| where Countries > 1 or ASNs > 2 or IPs > 5

Blue-team validation commands

# Revoke sessions during a suspected AiTM event.
Revoke-MgUserSignInSession -UserId "<user>@<domain>"

# Search mailbox audit activity around the suspicious sign-in.
Search-UnifiedAuditLog -StartDate <start> -EndDate <end> \
  -UserIds "<user>@<domain>" \
  -Operations New-InboxRule,Set-InboxRule,Add-MailboxPermission,Set-Mailbox,UpdateInboxRules

T2 — Legacy Authentication MFA Bypass

STRIDE: Elevation of Privilege / Spoofing
Risk: Critical if any mailbox still allows legacy clients
Attacker objective: Use a leaked or guessed password against SMTP AUTH, POP3, IMAP, or Exchange ActiveSync Basic Auth paths that do not support modern MFA enforcement.

Attacker technical workflow

A real attacker would:

  1. Obtain credential pairs from breaches or phishing.
  2. Test them against Microsoft 365 legacy endpoints.
  3. Find accounts where SMTP AUTH or old clients are still enabled.
  4. Use the account to send internal phishing, enumerate mail, or stage BEC.

Safe exposure validation commands

Check tenant-wide SMTP AUTH state:

Get-TransportConfig | Format-List SmtpClientAuthenticationDisabled

Disable SMTP AUTH tenant-wide:

Set-TransportConfig -SmtpClientAuthenticationDisabled $true

Find mailboxes with per-mailbox SMTP AUTH explicitly enabled:

Get-CASMailbox -ResultSize Unlimited |
  Where-Object {$_.SmtpClientAuthenticationDisabled -eq $false} |
  Select-Object DisplayName, PrimarySmtpAddress, SmtpClientAuthenticationDisabled

Disable SMTP AUTH for a specific mailbox:

Set-CASMailbox -Identity "<user>@<domain>" -SmtpClientAuthenticationDisabled $true

Return mailbox to tenant default:

Set-CASMailbox -Identity "<user>@<domain>" -SmtpClientAuthenticationDisabled $null

Detection KQL

SigninLogs
| where TimeGenerated > ago(7d)
| where ClientAppUsed in~ ("IMAP4", "POP3", "SMTP", "SMTP AUTH", "Exchange ActiveSync")
   or ClientAppUsed has_any ("IMAP", "POP", "SMTP", "ActiveSync")
| project TimeGenerated, UserPrincipalName, IPAddress, Location, ClientAppUsed,
          AppDisplayName, ResultType, ResultDescription, ConditionalAccessStatus
| order by TimeGenerated desc

Alert after enforcement date:

let LegacyAuthDisableDate = datetime(2026-05-31);
SigninLogs
| where TimeGenerated > LegacyAuthDisableDate
| where ClientAppUsed has_any ("IMAP", "POP", "SMTP", "ActiveSync")
| summarize Attempts=count(), Users=dcount(UserPrincipalName), SampleUsers=make_set(UserPrincipalName, 10)
          by ClientAppUsed, IPAddress, bin(TimeGenerated, 1h)
| where Attempts > 0

Blue-team validation checklist

Check Expected result
Tenant SMTP AUTH SmtpClientAuthenticationDisabled = True
Per-mailbox SMTP AUTH No mailbox has $false unless in approved exception register
Legacy auth sign-ins Zero successful events after enforcement
Exceptions Dedicated relay accounts only, IP-restricted, no interactive mailbox use

T3 — Malicious OAuth App Consent

STRIDE: Elevation of Privilege / Persistence
Risk: High
Attacker objective: Obtain persistent API access to mailbox data through an app consent grant, often surviving password reset and MFA changes until the grant is revoked.

Attacker technical workflow

A real attacker would:

  1. Create or compromise an Entra application.
  2. Request high-value permissions such as Mail.Read, Mail.ReadWrite, Mail.ReadWrite.All, offline_access, or directory-write scopes.
  3. Trick a user or admin into granting consent.
  4. Use refresh tokens to access mail through Graph without repeated interactive sign-in.
  5. Hide under the appearance of a legitimate business integration.

Safe exposure review commands

List OAuth delegated permission grants:

Get-MgOauth2PermissionGrant -All |
  Select-Object Id, ClientId, ConsentType, PrincipalId, ResourceId, Scope

Find grants containing mail scopes:

Get-MgOauth2PermissionGrant -All |
  Where-Object {$_.Scope -match "Mail\.Read|Mail\.ReadWrite|offline_access|EWS"} |
  Select-Object Id, ClientId, ConsentType, PrincipalId, Scope

Map app IDs to service principal display names:

$grants = Get-MgOauth2PermissionGrant -All |
  Where-Object {$_.Scope -match "Mail\.Read|Mail\.ReadWrite|offline_access"}

foreach ($g in $grants) {
  $sp = Get-MgServicePrincipal -ServicePrincipalId $g.ClientId -ErrorAction SilentlyContinue
  [PSCustomObject]@{
    AppName = $sp.DisplayName
    AppId = $sp.AppId
    ConsentType = $g.ConsentType
    Scope = $g.Scope
    GrantId = $g.Id
  }
}

Review application permissions assigned to service principals:

Get-MgServicePrincipal -All |
  ForEach-Object {
    $sp = $_
    Get-MgServicePrincipalAppRoleAssignment -ServicePrincipalId $sp.Id -ErrorAction SilentlyContinue |
      Select-Object @{Name="ServicePrincipal";Expression={$sp.DisplayName}}, ResourceDisplayName, AppRoleId, CreatedDateTime
  }

Detection KQL

AuditLogs
| where TimeGenerated > ago(30d)
| where OperationName has_any ("Consent to application", "Add app role assignment", "Add delegated permission grant")
| mv-expand TargetResources
| extend TargetName=tostring(TargetResources.displayName)
| extend Modified=TargetResources.modifiedProperties
| where tostring(Modified) has_any ("Mail.Read", "Mail.ReadWrite", "offline_access", "Directory.ReadWrite.All", "Files.ReadWrite.All")
| project TimeGenerated, OperationName, InitiatedBy, TargetName, Modified, Result

Response commands

Use these only after confirming the app is unauthorized.

# Remove delegated permission grant by grant ID.
Remove-MgOauth2PermissionGrant -OAuth2PermissionGrantId "<grant_id>"

# Disable a suspicious service principal while investigating.
Update-MgServicePrincipal -ServicePrincipalId "<service_principal_id>" -AccountEnabled:$false

# Revoke user sessions if the consent followed account compromise.
Revoke-MgUserSignInSession -UserId "<user>@<domain>"

Control evidence required

Control Evidence
User consent disabled Entra consent settings show user consent not allowed or limited to verified low-risk apps
Admin consent workflow Requests require security review for mail/directory scopes
OAuth grants monitored SIEM alerts on sensitive scopes
Quarterly app review Inventory of all apps with mail scopes, owner, justification, last reviewed date

T4 — Inbox Forwarding or Redirect Rule After Account Takeover

STRIDE: Information Disclosure / Persistence
Risk: Critical for BEC
Attacker objective: Add mailbox rules that silently forward or redirect copies of messages to an external mailbox.

Attacker technical workflow

After gaining mailbox access, a BEC actor commonly:

  1. Searches for active payment, invoice, payroll, HR, or legal conversations.
  2. Creates a forwarding, redirect, or forward-as-attachment rule.
  3. Deletes rule notifications or hides evidence using mailbox rules.
  4. Monitors the mailbox externally until the right payment moment.
  5. Modifies bank account details or impersonates an approver.

Safe exposure review commands

Find existing forwarding settings on mailboxes:

Get-EXOMailbox -ResultSize Unlimited -Properties ForwardingSmtpAddress,ForwardingAddress,DeliverToMailboxAndForward |
  Where-Object {$_.ForwardingSmtpAddress -or $_.ForwardingAddress -or $_.DeliverToMailboxAndForward} |
  Select-Object DisplayName, UserPrincipalName, ForwardingSmtpAddress, ForwardingAddress, DeliverToMailboxAndForward

Find inbox rules with external forwarding/redirect actions:

Get-EXOMailbox -ResultSize Unlimited |
  ForEach-Object {
    $mbx = $_.UserPrincipalName
    Get-InboxRule -Mailbox $mbx -ErrorAction SilentlyContinue |
      Where-Object {$_.ForwardTo -or $_.RedirectTo -or $_.ForwardAsAttachmentTo} |
      Select-Object @{Name="Mailbox";Expression={$mbx}}, Name, Enabled, ForwardTo, RedirectTo, ForwardAsAttachmentTo
  }

Block automatic external forwarding at outbound spam policy layer:

Get-HostedOutboundSpamFilterPolicy |
  Select-Object Name, AutoForwardingMode

Set-HostedOutboundSpamFilterPolicy -Identity "Default" -AutoForwardingMode Off

Optional transport rule to reject external auto-forwarding:

New-TransportRule -Name "Block External Auto-Forwarding" \
  -MessageTypeMatches AutoForward \
  -SentToScope NotInOrganization \
  -RejectMessageReasonText "External auto-forwarding is not permitted."

Detection KQL

OfficeActivity
| where TimeGenerated > ago(14d)
| where Workload =~ "Exchange"
| where Operation in~ ("New-InboxRule", "Set-InboxRule", "Set-Mailbox")
| where Parameters has_any ("ForwardTo", "RedirectTo", "ForwardAsAttachmentTo", "ForwardingSmtpAddress")
| project TimeGenerated, UserId, Operation, ClientIP, Parameters, ResultStatus

Detect external destination domains:

OfficeActivity
| where TimeGenerated > ago(30d)
| where Workload =~ "Exchange"
| where Operation in~ ("New-InboxRule", "Set-InboxRule")
| where Parameters has_any ("ForwardTo", "RedirectTo", "ForwardAsAttachmentTo")
| extend Params=tostring(Parameters)
| where Params !has "@<domain>"
| project TimeGenerated, UserId, ClientIP, Operation, Params

Response commands

# Inspect a specific user's rules.
Get-InboxRule -Mailbox "<user>@<domain>" |
  Format-Table Name, Enabled, ForwardTo, RedirectTo, ForwardAsAttachmentTo

# Remove a suspicious rule.
Remove-InboxRule -Mailbox "<user>@<domain>" -Identity "<rule_name>" -Confirm:$false

# Clear mailbox-level forwarding.
Set-Mailbox -Identity "<user>@<domain>" -ForwardingSmtpAddress $null -ForwardingAddress $null -DeliverToMailboxAndForward $false

T5 — Migration Connector Allow-List Left Active

STRIDE: Spoofing / Defense Evasion
Risk: High during migration
Attacker objective: Abuse a stale inbound connector or trusted IP path created for Proofpoint to bypass the intended MDO inspection route.

Attacker technical workflow

A real attacker would:

  1. Identify that the enterprise previously used Proofpoint.
  2. Infer old relay IPs from DNS history, message headers, public documentation, or internet scanning.
  3. Attempt to send mail in a way that matches a trusted connector path or legacy allow-list.
  4. Benefit if Exchange Online treats messages from that path as trusted, internal, or already scanned.

Safe validation commands

Check current MX record:

Resolve-DnsName -Type MX <domain> | Sort-Object Preference

Linux/macOS:

dig MX <domain> +short

Review inbound connectors:

Get-InboundConnector |
  Format-List Name, Enabled, ConnectorType, SenderDomains, SenderIPAddresses,
              RequireTls, RestrictDomainsToIPAddresses, TreatMessagesAsInternal, CloudServicesMailEnabled

Find connectors referencing legacy Proofpoint IPs:

$OldProofpointIps = @("<old_proofpoint_ip>")
Get-InboundConnector | Where-Object {
  ($_.SenderIPAddresses | ForEach-Object {$_.ToString()}) -in $OldProofpointIps
} | Format-List *

Remove the legacy connector after cutover approval:

Remove-InboundConnector -Identity "Proofpoint-Legacy" -Confirm:$false

Lab-only test: send a benign canary email from an approved relay and verify it routes through the expected connector and receives MDO inspection headers.

swaks --server <tenant>.mail.protection.outlook.com \
  --from purpleteam-sender@<domain> \
  --to <test_mailbox> \
  --header "Subject: PURPLETEAM-CONNECTOR-ROUTE-TEST" \
  --body "Connector routing test. No credential, no attachment, no link."

Detection / evidence queries

Use Message Trace / Threat Explorer to identify messages from old IPs after the decommission date:

Get-MessageTrace -StartDate <start> -EndDate <end> -PageSize 5000 |
  Where-Object {$_.SenderAddress -like "*@*"} |
  Select-Object Received, SenderAddress, RecipientAddress, Subject, Status, MessageTraceId

In Defender advanced hunting, hunt for old-source IP patterns:

EmailEvents
| where Timestamp > ago(30d)
| where SenderIPv4 in ("<old_proofpoint_ip>")
| project Timestamp, NetworkMessageId, SenderFromAddress, RecipientEmailAddress,
          SenderIPv4, DeliveryAction, ThreatTypes, DetectionMethods, Subject

Control evidence required

Check Expected state
MX record Points to Exchange Online Protection only after cutover
Inbound connector No legacy Proofpoint connector remains enabled
Trusted IPs No stale Proofpoint IP allow-list in connectors or transport rules
Message trace Zero inbound mail from legacy connector/IP after decommission

T6 — DMARC p=none and Domain Spoofing During Migration

STRIDE: Spoofing
Risk: High if p=none remains too long
Attacker objective: Send email using the visible From: domain of the enterprise while DMARC is monitor-only or misconfigured.

Attacker technical workflow

A real attacker would:

  1. Check SPF, DKIM, and DMARC DNS records.
  2. Notice DMARC p=none or weak SPF ~all.
  3. Send spoofed messages that claim to be from executives or IT migration teams.
  4. Target employees, customers, or suppliers during the migration window.

Safe validation commands

Check SPF:

Resolve-DnsName -Type TXT <domain> |
  Select-Object -ExpandProperty Strings

Check DMARC:

Resolve-DnsName -Type TXT _dmarc.<domain> |
  Select-Object -ExpandProperty Strings

Check DKIM config in Exchange Online:

Get-DkimSigningConfig -Identity <domain> |
  Format-List Domain, Enabled, Selector1CNAME, Selector2CNAME, KeyCreationTime

Check DKIM DNS records:

dig TXT selector1._domainkey.<domain> +short
dig TXT selector2._domainkey.<domain> +short

Safe lab spoof test against your own test mailbox. This should be performed only from a non-authorized IP and only to a dedicated canary mailbox to verify DMARC handling.

swaks --server <tenant>.mail.protection.outlook.com \
  --from ceo@<domain> \

Safe CT monitoring script

Use this to monitor Certificate Transparency data for suspicious certificates. This is a defensive monitoring example.

# Defensive example: compare newly observed domains against your portfolio.
# Input: one domain per line in domains.txt, one observed domain per line in observed_ct_domains.txt
from difflib import SequenceMatcher
import idna

protected = [d.strip().lower() for d in open("domains.txt") if d.strip()]
observed = [d.strip().lower() for d in open("observed_ct_domains.txt") if d.strip()]

def similarity(a, b):
    return SequenceMatcher(None, a, b).ratio()

for obs in observed:
    try:
        decoded = idna.decode(obs)
    except Exception:
        decoded = obs
    for p in protected:
        score = similarity(decoded, p)
        if score >= 0.86 and decoded != p:
            print(f"Possible lookalike: {decoded} resembles {p} score={score:.2f}")

7. Incident Playbooks — Command-Level Response

7.1 Suspected AiTM / Mailbox Compromise

# 1. Revoke sessions.
Revoke-MgUserSignInSession -UserId "<user>@<domain>"

# 2. Temporarily block sign-in if confirmed compromise.
Update-MgUser -UserId "<user>@<domain>" -AccountEnabled:$false

# 3. Remove suspicious inbox rules.
Get-InboxRule -Mailbox "<user>@<domain>" |
  Where-Object {$_.ForwardTo -or $_.RedirectTo -or $_.ForwardAsAttachmentTo} |
  ForEach-Object { Remove-InboxRule -Mailbox "<user>@<domain>" -Identity $_.Name -Confirm:$false }

# 4. Clear mailbox-level forwarding.
Set-Mailbox -Identity "<user>@<domain>" -ForwardingSmtpAddress $null -ForwardingAddress $null -DeliverToMailboxAndForward $false

# 5. Review OAuth grants and mailbox permissions.
Search-UnifiedAuditLog -StartDate <start> -EndDate <end> -UserIds "<user>@<domain>" \
  -Operations New-InboxRule,Set-InboxRule,Add-MailboxPermission,ConsentToApplication,AddOAuth2PermissionGrant

7.2 Suspicious OAuth App

# Review grant.
Get-MgOauth2PermissionGrant -OAuth2PermissionGrantId "<grant_id>" | Format-List

# Revoke delegated grant.
Remove-MgOauth2PermissionGrant -OAuth2PermissionGrantId "<grant_id>"

# Disable service principal while investigating.
Update-MgServicePrincipal -ServicePrincipalId "<service_principal_id>" -AccountEnabled:$false

7.3 Stale Connector Discovered

# Export connector state for evidence.
Get-InboundConnector | Format-List * > .\InboundConnectors-Before.txt

# Disable first if immediate deletion requires change approval.
Set-InboundConnector -Identity "<connector_name>" -Enabled $false

# Remove after approval.
Remove-InboundConnector -Identity "<connector_name>" -Confirm:$false

# Confirm removal.
Get-InboundConnector | Format-List Name, Enabled, SenderIPAddresses

7.4 External Forwarding Rule Detected

# Remove mailbox rules and forwarding.
Remove-InboxRule -Mailbox "<user>@<domain>" -Identity "<rule_name>" -Confirm:$false
Set-Mailbox -Identity "<user>@<domain>" -ForwardingSmtpAddress $null -ForwardingAddress $null -DeliverToMailboxAndForward $false

# Enforce outbound forwarding block.
Set-HostedOutboundSpamFilterPolicy -Identity "Default" -AutoForwardingMode Off

# Scope historical exfiltration.
Search-UnifiedAuditLog -StartDate <start> -EndDate <end> -UserIds "<user>@<domain>" \
  -Operations New-InboxRule,Set-InboxRule,Send,MailItemsAccessed

9. Final Reviewer Position

The highest-risk migration failure is not simply that Proofpoint is being replaced. The risk comes from a short period where:

  • users are primed to trust migration-themed emails,
  • mail routing and connector behavior are changing,
  • DMARC may still be in monitoring mode,
  • legacy authentication exceptions may remain,
  • OAuth app governance may not be mature,
  • and audit baselines are not yet stable.

The migration should be approved only when the following are true:

  1. No stale Proofpoint connector or trusted IP path remains after cutover.
  2. SMTP AUTH and other legacy auth paths are disabled or restricted to approved, IP-bound relay identities.
  3. Finance, executive, HR, and admin users use phishing-resistant MFA.
  4. External auto-forwarding is blocked and inbox-rule alerts are real-time.
  5. DMARC progression has a named owner and target date for p=reject.
  6. Safe Links, Safe Attachments, ZAP, Threat Explorer, and UAL are operational before user migration.
  7. OAuth app consent is governed, monitored, and reviewed continuously.
  8. UAL and sign-in telemetry are retained long enough to scope slow-burn BEC.

Recommended decision: Approve migration only with these controls as cutover gates, not post-migration backlog items.
