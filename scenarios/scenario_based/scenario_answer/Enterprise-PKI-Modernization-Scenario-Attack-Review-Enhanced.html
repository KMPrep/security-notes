Enterprise PKI Modernization — Threat Exploitation and Blue-Team Control Review

Source file reviewed: Enterprise-PKI-Modernization-Scenario.md
Generated artifact: Enterprise-PKI-Modernization-Scenario-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Replacing a 15-year-old two-tier PKI (offline Root CA + single issuing CA) with an HSM-backed Root, two subordinate CAs, an OCSP responder, and an automated certificate lifecycle management platform for enterprise-wide TLS, code signing, and client authentication.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

The review focuses on attack mechanics, prerequisites, command-level purple-team validation examples, detection evidence, and control validation. Commands use placeholder values and are intended for an owned lab or approved internal purple-team exercise only. High-risk techniques are described at the technique level and paired with safe defensive validation commands.

2. Highest-Risk Design Decision

Whether certificate templates permit requestor-controlled Subject Alternative Names (SANs) without CA-side validation — the ADCS ESC1 pattern. If any certificate template allows the requesting account to specify arbitrary SANs (e.g., Subject: supplied by requestor) without the CA policy module validating that the requested SAN matches the requestor's authorized identity, then any account with enrollment rights on that template can request a certificate for Administrator@corp.com, DC01$@corp.com, or *.corp.com. This single misconfiguration collapses all Kerberos PKINIT, smart-card logon, and code-signing trust chains in the enterprise. The correct design requires that Subject and SAN fields are always controlled by the CA policy module using OID mapping from the requester's authenticated identity — never supplied verbatim by the requestor — and that every template change undergoes a two-person security review before deployment.

6. Threat-by-Threat Technical Review

T1 — Root CA Ceremony Compromise (Custodian Phishing)

STRIDE category: Spoofing, Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Compromise a Root CA key custodian's workstation before a scheduled ceremony to capture their smart card PIN via keylogger, then combine the captured PIN with physical theft or coercion to activate the Root CA and issue a rogue sub-CA certificate.

How the attacker would exploit it

  • Prerequisite: Attacker identifies the ceremony schedule (visible in organizational calendar, change management system, or inferred from sub-CA certificate validity expiry dates). Key custodians are enumerable via org charts, LinkedIn, and conference speaker bios.
  • Initial action: Spearphish targeting key custodians 2–4 weeks before the scheduled ceremony. Email with a fake "ceremony preparation document" containing a macro. When the custodian opens the document on their daily-use laptop, a keylogger is installed.
  • Core exploitation: Custodian practices their smart card PIN on the keylogger-infected laptop prior to the ceremony. Attacker captures the PIN. With the PIN, the physical smart card is the only remaining requirement. Attacker either: (a) physically steals the smart card from the custodian's office, or (b) coerces the custodian directly if the smart card is on their person.
  • Escalation: At the ceremony with one (or with collusion, two) compromised custodian(s), attacker has enough shards to activate the Root CA if M-of-N quorum is satisfied. Signs a rogue sub-CA certificate with the Root CA key — this rogue sub-CA is fully trusted by all relying parties.
  • Impact realization: Attacker has a fully trusted sub-CA that can issue TLS certificates for any domain, code-signing certificates for any publisher, and client auth certificates for any identity. All traffic to enterprise services can be intercepted; malware can be signed as if from the enterprise.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): M-of-N quorum of 3-of-5 — attacker must compromise the majority of custodians, each from different teams (Security, Legal, Compliance) and geographically separated. Collusion or simultaneous compromise of 3 separate teams is a very high bar.
  • Control 2 (Preventive): Custodians never practice or use their ceremony PIN on any networked device — the PIN is memorized only, never typed on a connected device. Smart cards are stored in a separate locked safe in a physically access-controlled room, not on person during daily work.
  • Control 3 (Preventive): Independent auditor present during all ceremony operations — any substitution of a custodian or deviation from ceremony script is flagged in real time and ceremony is halted.
  • Control 4 (Detective): HSM audit log reviewed immediately after each ceremony — log shows the exact operator card identities, operations performed, and timestamps. Any operation not in the ceremony script (e.g., signing an unexpected CSR) generates an immediate P0 alert.
  • Control 5 (Detective): Air-gapped workstation state verification — bootable OS image hash checked before and after ceremony; any modification to the OS image during ceremony indicates tampering attempt.

Evidence and telemetry to validate controls

  • HSM Audit Log: Hardware-signed log with sequence numbers — operator card identities, operation type (signCertificate, issueCRL), timestamp, success/failure. Any gap in sequence numbers indicates log tampering attempt.
  • Ceremony Transcript Hash: SHA-256 of signed ceremony transcript stored offline separately from ceremony materials — any discrepancy between the transcript and HSM log entries indicates falsification.
  • Video Recording Hash: Ceremony video recording hash published to an internal CT log immediately after ceremony — proves continuity of ceremony proceedings.
  • Custodian Smart Card Audit: After ceremony, smart card usage logs from HSM — only expected operator card IDs should appear; any unexpected card ID in the log means a rogue ceremony occurred.

T2 — HSM Supply Chain Tampering

STRIDE category: Tampering
Risk: Low likelihood, Critical impact
Attacker objective: Compromise the HSM hardware or initialization tooling before delivery to the enterprise so that the Root CA private key is exfiltrated to the attacker during or after key generation, while all audit logs and physical seals appear intact.

How the attacker would exploit it

  • Prerequisite: Nation-state actor or highly capable supply-chain attacker intercepts the HSM shipment or compromises the HSM vendor's firmware signing infrastructure. Target HSMs: Thales Luna Network HSM, nCipher nShield, or Entrust nShield — all used in enterprise PKI.
  • Initial action: Backdoored firmware installed on the HSM during manufacturing or shipping. The firmware appears to pass all validation checks (the FIPS certificate is for the base hardware, not the specific firmware build). Alternatively, a malicious initialization tool (fake CA setup wizard) is delivered alongside the HSM.
  • Core exploitation: When the enterprise generates the Root CA key on the compromised HSM, the firmware exfiltrates a copy of the private key via a covert channel during initialization — for example, encoding it in OCSP request padding, crafted X.509 extension values, or USB side-channel if the air-gap is not absolute.
  • Escalation: Attacker with the Root CA private key can generate a new sub-CA certificate at any time without needing to activate the enterprise's HSM. All certificates issued under the enterprise's PKI are now compromised.
  • Impact realization: Total PKI compromise with no ceremony forensic evidence — the enterprise's HSM audit log shows correct operations because the exfiltration occurred at the firmware level below the audit layer.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Verify HSM firmware cryptographic hash against vendor-published hashes on a separate, vendor-independent channel (e.g., vendor-signed verification tool downloaded separately from HSM delivery). Do not trust hash files delivered with the HSM.
  • Control 2 (Preventive): Purchase directly from authorized distributor with tamper-evident shipping and unbroken chain of custody documentation. Inspect tamper-evident seals before any use. Any broken seal = return unit; do not use.
  • Control 3 (Preventive): Require vendor firmware attestation — HSM vendors provide a firmware attestation certificate signed by the vendor's PKI that covers the specific firmware version. Verify attestation cert chain before ceremony.
  • Control 4 (Preventive): Key generation ceremony on an air-gapped workstation with no network interfaces — firmware exfiltration via network is blocked physically. USB ports blocked except for HSM itself. Faraday cage during ceremony.
  • Control 5 (Detective): Key consistency check — after generating the Root CA key in the HSM, immediately sign a test certificate. Verify the certificate chain validates correctly. Run this test on the air-gapped workstation before any actual PKI operations.

Evidence and telemetry to validate controls

  • Firmware Attestation Certificate: Verify chain VendorRootCA → VendorFirmwareCA → FirmwareAttestationCert matches the firmware hash loaded on the device. Check via hsm-fw-verify tool (vendor-specific).
  • Tamper-Evident Seal Inspection Log: Physical security log documenting seal numbers, inspection result, and inspector identity at each ceremony.
  • HSM Initialization Log: Time from key generation to first certificate operation — unexpected delays may indicate exfiltration activity. Document expected timing in ceremony script.
  • Vendor PSIRT Advisory Subscription: All known HSM firmware vulnerabilities tracked in organization's vulnerability management system; patch within ceremony cycle or replace unit.

T3 — ADCS ESC1: User-Controlled SAN Injection

STRIDE category: Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Request a certificate via ADCS with an arbitrary SAN value (e.g., upn=administrator@corp.com) from a misconfigured template that allows the requestor to supply the subject, enabling Kerberos PKINIT authentication as any domain user including Domain Admins.

How the attacker would exploit it

  • Prerequisite: Any domain user account (even a low-privilege service account) with enrollment rights on a misconfigured certificate template. The template must: allow subject to be supplied by the requestor (CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT), not require manager approval, have client authentication EKU (1.3.6.1.5.5.7.3.2), and be accessible to a broad group (Domain Users or Authenticated Users).
  • Initial action: Attacker enumerates certificate templates using Certify.exe find /vulnerable or certipy find -enabled to identify ESC1 templates. In the new PKI deployment, a template created for workstation auto-enrollment may be misconfigured during migration with CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT accidentally enabled.
  • Core exploitation: Attacker runs Certify.exe request /ca:<ca-name> /template:<vulnerable-template> /altname:administrator or certipy req -ca <ca> -template <template> -upn administrator@<domain>. The ADCS CA issues a certificate with upn=administrator@corp.com even though the requestor is a low-privilege account.
  • Escalation: Attacker uses the issued certificate with Rubeus.exe asktgt /user:administrator /certificate:<cert> /ptt to request a Kerberos TGT for the Administrator account via PKINIT. Immediately has Domain Admin rights.
  • Impact realization: Full domain compromise from any authenticated account with no credential theft required — only the certificate template misconfiguration is needed.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive — Critical): Disable CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT on every certificate template in the new PKI. The CA policy module should always derive the subject from the requester's authenticated identity (AD attribute or mTLS identity). Run Certify.exe find /vulnerable against the new CA after every template change.
  • Control 2 (Preventive): Enable CA Manager Approval on any template that includes Client Authentication EKU — requests require human approval before certificate issuance, breaking automated exploitation.
  • Control 3 (Preventive): Restrict template enrollment rights to specific security groups (not Authenticated Users or Domain Computers for privileged templates). Principle of least privilege applied at template ACL level.
  • Control 4 (Detective): CA Issuance Log alert: EventID 4886 (certificate issued) where the certificate's subjectAltName contains a UPN that does NOT match the requesting account's own UPN — this is the signature of ESC1 exploitation.
  • Control 5 (Detective): certipy find scan run weekly against all CAs — outputs vulnerable templates and misconfigurations. Results pushed to SIEM dashboard; any new vulnerable template = P0 alert.

Evidence and telemetry to validate controls

  • CA Issuance Log (Event 4886): Requester: svc-account@corp.com, SubjectAltName: upn=administrator@corp.com — mismatch between requester and SAN UPN is definitive ESC1 exploitation evidence.
  • CA Template Audit (Event 4899): Template modification events — CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT bit set on any template triggers P1 alert.
  • Kerberos TGT Request Log (DC Event 4768): PreAuthType = 16 (PKINIT) for a high-value account (Administrator, krbtgt backup, tier 0 accounts) from an unusual source IP or during off-hours.
  • certipy Weekly Scan Output: Vulnerability count per CA per week — target: zero vulnerable templates. Any non-zero count is remediation work.

T4 — ADCS ESC8: NTLM Relay to Web Enrollment

STRIDE category: Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Use an NTLM relay attack (e.g., via Responder + ntlmrelayx) to relay a domain computer's or user's NTLM authentication to the ADCS HTTP web enrollment endpoint, obtaining a certificate on behalf of the relayed identity.

How the attacker would exploit it

  • Prerequisite: ADCS HTTP enrollment endpoint (http://<CA>/certsrv/) is enabled and accepts NTLM authentication (not requiring HTTPS + EPA). Any domain user or computer that can be coerced into authenticating to an attacker-controlled server can be relayed.
  • Initial action: Attacker runs Responder.py -I <iface> to capture NTLM authentication from network broadcasts (LLMNR/NBT-NS poisoning) or uses coercion techniques: PetitPotam to coerce a domain controller to authenticate, PrinterBug to coerce any computer with spooler service.
  • Core exploitation: Attacker runs ntlmrelayx.py -t http://<CA>/certsrv/certfnsh.asp -smb2support --adcs --template <WebServer-template>. When a DC or high-privilege computer authenticates to Responder, the NTLM relay immediately requests a certificate from ADCS on behalf of that computer's identity — for example, DC01$@corp.com.
  • Escalation: Certificate for DC01$ is issued. Attacker uses Rubeus.exe asktgt /user:DC01$ /certificate:<cert> to get a Kerberos TGT for the domain controller computer account. With the DC's TGT, runs DCSync to extract all domain credentials.
  • Impact realization: Full domain compromise via a relay from any domain controller forced to authenticate via NTLM — no credentials required, only network adjacency.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive — Critical): Disable HTTP NTLM authentication on the ADCS web enrollment endpoint. Require HTTPS with Extended Protection for Authentication (EPA) and mutual TLS. Configure IIS: Windows Authentication → Advanced Settings → Enable Extended Protection = Required.
  • Control 2 (Preventive): Disable LLMNR and NBT-NS via GPO: Computer Configuration → Administrative Templates → Network → DNS Client → Turn off multicast name resolution = Enabled; disable NetBIOS over TCP/IP in DHCP options.
  • Control 3 (Preventive): Enable SMB signing on all domain computers (mandatory) via GPO: Microsoft network server: Digitally sign communications (always) = Enabled. SMB signing blocks NTLM relay over SMB channels.
  • Control 4 (Preventive): Disable the certsrv HTTP enrollment endpoint if automated ACME/EST enrollment is used exclusively. Fewer enrollment endpoints = smaller attack surface.
  • Control 5 (Detective): CA issuance log: certificate issued via CertEnrollWS or IIS web enrollment where the requester identity is a domain controller computer account or tier 0 account — DC accounts should not request certs via web enrollment (they use autoenrollment via RPC).

Evidence and telemetry to validate controls

  • IIS Access Log (CA Web Server): Requests to certsrv/certfnsh.asp with NTLM in Authorization header from IPs that are not the enrolled host's own IP — cross-machine NTLM web enrollment is the ESC8 indicator.
  • Windows Event 4625 (Logon Failure): NTLM relay attempts generate multiple 4625 events with LogonType=3 from unexpected sources attempting to reach the CA.
  • CA Issuance Log (Event 4886): Certificate issued to <DC-name>$ via HTTP web enrollment (rather than standard autoenrollment) — unexpected enrollment method for a DC.
  • Network Detection (IDS): LLMNR/NBT-NS responses from unexpected sources on the network — detect Responder via Suricata rule for LLMNR response from non-DNS-server IP.

STRIDE category: Spoofing, Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Steal a certificate lifecycle automation platform's service account credentials (from a CI/CD pipeline, application config file, or environment variable) and use them to request certificates for out-of-scope subjects — for example, a TLS certificate for a competitor's domain or a code-signing certificate for malware.

How the attacker would exploit it

  • Prerequisite: Attacker compromises a CI/CD pipeline or a container running a service that uses the automation platform's service account. Service account credentials are stored in plaintext in environment variables, .env files, or secrets managers with insufficient access controls.
  • Initial action: Attacker scans the build pipeline secrets (secrets scanning on the compromised build agent, or browses /proc/1/environ in a compromised container). Finds the automation platform API key or client certificate private key.

  • Control 4 (Detective): Automation platform API audit log alert: certificate request for a SAN outside the requesting service account's allowlist → immediate P1 alert (scope violation should never occur in normal operation).

T6 — PKI Admin Account Compromise

STRIDE category: Spoofing, Tampering, Repudiation
Risk: Medium likelihood, High impact
Attacker objective: Compromise a PKI admin account to modify certificate templates (expand issuance scope), add unauthorized enrollment agents, create new service accounts with broad issuance rights, or delete the CA audit log to hide prior rogue issuance.

How the attacker would exploit it

  • Prerequisite: Attacker identifies PKI admin accounts via AD enumeration (Get-ADGroupMember "CA Admins") or by querying the CA's ACL. PKI admins are high-value targets but fewer than 10 in most enterprises — spearphishing is targeted and effective.
  • Initial action: Spearphish PKI admin with a fake "PKI template renewal request" from a spoofed ITSM tool. Admin follows the link and provides TOTP code on a phishing proxy (Evilginx2). Attacker captures session cookie and TOTP code simultaneously.
  • Core exploitation: Attacker authenticates as PKI admin to the CA web management interface. Modifies a certificate template to add CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT and grant Authenticated Users enrollment rights — effectively enabling ESC1 for all domain users. Does this during off-hours to avoid detection.
  • Escalation: Any domain user can now exploit the newly misconfigured template (ESC1, T3). Alternatively, attacker deletes the CA audit log (Windows Event ID 1102) to cover prior operations.
  • Impact realization: PKI template modification is a force multiplier — a single admin account compromise enables domain-wide privilege escalation by all subsequent attackers who discover the misconfigured template.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): FIDO2 hardware key required for all PKI admin authentication — Evilginx2/AiTM phishing cannot capture a FIDO2 assertion because FIDO2 assertions are origin-bound to the real site.
  • Control 2 (Preventive): Template changes require a two-person review and an approved ITSM change ticket — even a compromised admin cannot make unilateral template changes without triggering the workflow.
  • Control 3 (Detective): Windows Security Event 4899 (Certificate Template Modified) alert — any template modification outside an approved change window is P1 alert with SOC response within 15 minutes.
  • Control 4 (Detective): Windows Security Event 1102 (Security Audit Log Cleared) on CA servers — immediate P0 alert; no legitimate operation clears the CA security log outside a documented log rotation procedure.
  • Control 5 (Response): On PKI admin off-hours authentication: automatic account suspension pending manager verification; any template change rolled back; full audit of CA operations since last verified clean state.

Evidence and telemetry to validate controls

  • CA Server Event Log (Event 4899): Template modification event with ObjectDN, SubjectName, TemplateOID, modifying account — every 4899 event must have an open ITSM change ticket; absent ticket = P1 alert.
  • CA Server Event Log (Event 1102): Security log cleared on any CA server → immediate P0 alert; treat as active incident.
  • IdP Sign-In Log: PKI admin authentication outside business hours or from new IP → P1 alert; CA operations in the 1 hour following any such alert must be reviewed.
  • AD ACL Change Log (Event 5136): Modification to ADCS ACLs or template objects in Active Directory → track all changes to CN=Certificate Templates,CN=Public Key Services,CN=Services,CN=Configuration.

T7 — CA Audit Log Tampering or Deletion

STRIDE category: Repudiation, Tampering
Risk: Low likelihood, High impact
Attacker objective: Delete or modify the CA audit log to cover prior rogue certificate issuance, making forensic investigation impossible and creating deniability for both attackers and potentially for insider threats.

How the attacker would exploit it

  • Prerequisite: Attacker has admin access to the CA server (via T6 or direct server compromise) or has database admin rights on the SQL database backing the CA issuance records. The CA audit log in ADCS is the Windows Security Event Log — clearable by anyone with SeSecurityPrivilege (local admins have this by default).
  • Initial action: Attacker runs wevtutil cl Security on the CA server to clear the Windows Security Event Log — this generates Event 1102 (log cleared) but then subsequent events are gone. If the attacker moves fast, the 1102 event may be the only evidence.
  • Core exploitation: All CA issuance events (4886), template modifications (4899), and admin operations (4672) are deleted. Any prior rogue certificate issuance is now forensically invisible from the CA server's local log.
  • Escalation: Attacker also targets the CA database — runs SQL DELETE FROM CertificateAuthority.dbo.IssuedCertificates WHERE SerialNumber = '<rogue-serial>' to remove the issuance record. Now neither the Windows event log nor the CA database has evidence of the rogue certificate.
  • Impact realization: Incident response cannot determine scope of compromise; cannot produce a list of certificates to revoke; compliance audit fails; legal proceedings lack evidence.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): SIEM log forwarding in real-time (< 60 second latency) from CA servers — all Windows Security events are forwarded to the SIEM before they can be deleted locally. Even if the local log is cleared, the SIEM retains all events.
  • Control 2 (Preventive): CA database audit log backed by an immutable log store (append-only) — either via SQL Server Audit writing to Azure Blob Storage with immutability policy, or via EJBCA's append-only signed audit log.
  • Control 3 (Detective): Windows Security Event 1102 (Audit log cleared) on CA servers → immediate P0 alert, regardless of who initiated the clear.
  • Control 4 (Detective): CA database integrity monitoring — daily hash of key tables (IssuedCertificates, RevokedCertificates) stored separately; any change to historical rows is detected at next hash comparison.
  • Control 5 (Detective): HSM audit log as independent source of truth — the HSM records every signing operation with a hardware-signed timestamp. Even if CA database and Windows log are wiped, the HSM log cannot be modified by software and proves what was signed.

Evidence and telemetry to validate controls

  • Windows Security Event 1102: Audit log cleared on CA server — immediate P0; contains the identity of who cleared it.
  • SIEM Log Completeness Check: Daily check: event count from CA server in SIEM vs expected baseline rate — significant drop in event volume from CA server indicates log forwarding interruption or local deletion.
  • CA Database Audit Hash: Daily hash comparison of IssuedCertificates row count and hash of serial number + issuance timestamp — any unexplained decrease in row count = deletion alert.
  • HSM Audit Log Cross-Reference: HSM signCertificate operation count vs CA database issued cert count — if HSM shows more signings than the database records, records were deleted.

T8 — OCSP Responder DoS (Soft-Fail Bypass)

STRIDE category: Denial of Service
Risk: High likelihood, High impact
Attacker objective: DDoS the OCSP responder so that TLS clients that implement OCSP soft-fail accept a revoked certificate as valid — allowing an attacker holding a revoked certificate to continue using it for authentication or MITM.

How the attacker would exploit it

  • Prerequisite: Attacker holds a certificate that has been revoked (e.g., a compromised TLS cert for a sensitive service, or a client auth cert that was revoked after credential theft). The attacker needs to prevent relying parties from learning about the revocation.
  • Initial action: Attacker launches a volumetric UDP/TCP flood against the OCSP responder's IP and port 80 (OCSP over HTTP). A 10 Gbps flood from a rented botnet is sufficient to saturate a typical OCSP responder. Alternatively, send millions of malformed OCSP requests to exhaust the responder's processing capacity.
  • Core exploitation: OCSP responder becomes unavailable. TLS clients attempting to validate the revoked certificate's status receive a connection timeout. Most browsers and load balancers implement OCSP soft-fail by default — they proceed with the connection rather than blocking it when OCSP is unreachable.
  • Escalation: While the OCSP responder is down, attacker uses the revoked certificate to authenticate to a sensitive service (MTLS client auth) or to serve fake content on an MITM proxy. The soft-fail window can be extended by maintaining the DDoS.
  • Impact realization: Revoked certificate treated as valid by relying parties; the revocation event that was supposed to terminate the attacker's access is bypassed for the DDoS duration.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): OCSP stapling on all internal TLS servers — the certificate is served with a pre-fetched, signed OCSP response stapled to the TLS handshake. Clients that support stapling do not query the OCSP responder; the stapled response is from before the attack. For newly revoked certs, the stapled response will expire and the server must refresh it (requiring OCSP responder availability).
  • Control 2 (Preventive): Redundant OCSP responders behind a DDoS-protected load balancer (Cloudflare, AWS Shield, Google Cloud Armor) — Layer 7 DDoS protection capable of absorbing multi-Gbps volumetric attacks.
  • Control 3 (Preventive): Reduce OCSP cache TTL for high-value certificate templates (code-signing, admin client auth) to 15 minutes — OCSP stapling responses expire quickly and must be refreshed, meaning a DDoS has a smaller window.
  • Control 4 (Detective): OCSP availability synthetic monitor — automated probe every 60 seconds to http://<ocsp-responder>/ocsp with a test certificate. Five consecutive failures → P1 alert (OCSP down; DDoS may be in progress).
  • Control 5 (Response): On OCSP outage alert: immediately investigate if any recently revoked high-value certificates exist (code-signing, admin auth); temporarily enforce hard-fail policy in CA configuration for those certificate types if possible; enable cloud DDoS scrubbing if under attack.

Evidence and telemetry to validate controls

  • OCSP Synthetic Monitor Logs: Consecutive failure count, response time — baseline: < 50ms; alert at > 500ms for 3 consecutive probes; page at 5 consecutive failures.
  • DDoS Detection Telemetry (NetFlow): Sudden spike in UDP/TCP traffic to OCSP server IP → DDoS mitigation appliance alert.
  • OCSP Access Log: Request volume spike: baseline ~1000 req/min; DDoS pattern = millions of req/min from diverse source IPs.
  • Revoked Certificate Query Log: High-value revoked serial numbers receiving OCSP queries post-revocation — indicates potential active use while OCSP is degraded.

T9 — CRL Distribution Point DoS / Stale CRL Replacement

STRIDE category: Denial of Service, Tampering
Risk: Medium likelihood, High impact
Attacker objective: Prevent relying parties from downloading the current CRL, or replace the current CRL with a stale version that does not contain newly revoked certificates, allowing revoked certificate holders to continue authenticating.

How the attacker would exploit it

  • Prerequisite: CDP is hosted on a standard HTTP web server (often IIS or Apache) that is reachable from external networks or has weak write controls. Attacker either DoSes the web server or, if write access to the CDP path is possible (misconfigured web root, compromised web server credential), replaces the CRL file.
  • Initial action: Option A (DoS): Flood the CDP web server, making GET http://<cdp>/pki/root.crl return connection refused. Option B (stale CRL): Gain write access to the web root (via compromised admin, web shell, or misconfigured file share) and replace the current CRL with an older version that predates the revocation of the attacker's certificate.
  • Core exploitation: Relying parties attempt to download the CRL and either receive connection errors (DoS path) or receive an older CRL that does not list the attacker's certificate serial as revoked (stale CRL path). In soft-fail clients, the certificate is accepted. In hard-fail clients (DoS path), service outage may occur.
  • Escalation: Attacker uses the "unrevoked" certificate for authentication during the CRL staleness window. For code-signing certificates, the stale CRL allows malware signed with a compromised certificate to be accepted by Windows as having a valid chain.
  • Impact realization: Revocation bypass for code-signed malware or TLS MITM; or service outage if hard-fail clients cannot download CRL.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): CDP write access is controlled by a dedicated service account with write-only permissions to the specific CRL directory — no interactive admin access; web server itself runs as a low-privilege read-only service account.
  • Control 2 (Preventive): CRL files served from a read-only CDN (Cloudflare R2, Azure Blob with Public Read + append-only write) — CRL updates are pushed by the CA automation process; the CDN cannot be modified by compromising the web server.
  • Control 3 (Preventive): OCSP stapling is the primary revocation mechanism; CRL is the fallback. Reducing reliance on CRL by ensuring OCSP stapling is deployed on all internal TLS servers limits CRL DoS impact.
  • Control 4 (Detective): Synthetic CRL download monitor — automated test every 5 minutes: download CRL, verify thisUpdate is within the last 24 hours and nextUpdate is in the future. If either check fails → P1 alert (CRL stale or unavailable).
  • Control 5 (Detective): File integrity monitoring on the CDP web root directory — any modification to CRL files outside of the scheduled CA update job triggers immediate P1 alert.

Evidence and telemetry to validate controls

  • CRL Synthetic Monitor Result: thisUpdate, nextUpdate, SHA-256 of CRL bytes — compare against expected values from CA automation platform; mismatch indicates stale or replaced CRL.
  • FIM Alert on CDP Web Root: File modification event for *.crl files outside the automated CA CRL publication schedule (e.g., at 02:00 daily by the CA service account) → immediate P1 alert.
  • Web Server Access Log: GET /*.crl request volume — sudden spike in failed requests or 404 responses indicates DoS or file deletion.
  • CA CRL Publish Audit: CA automation log confirming CRL was successfully published to CDP at the scheduled time; absence of successful publish = alert.

T12 — Subordinate CA Key Extraction via Software Exploit

STRIDE category: Information Disclosure, Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Exploit a remote code execution vulnerability in the subordinate CA's management software (ADCS web services, EJBCA administrative portal, or IIS) to achieve RCE on the CA server and extract the sub-CA private key from memory or from the HSM interface.

How the attacker would exploit it

  • Prerequisite: A CVE exists in the CA management software that allows unauthenticated or low-auth RCE. Examples: ADCS CES/CEP web services, EJBCA's admin web, or the underlying Windows server OS. The sub-CA server must be reachable from a compromised host on the network.
  • Initial action: Attacker runs nmap -sV <ca-server> from a compromised network host. Discovers Microsoft IIS 10.0 on port 443 and identifies the ADCS certificate enrollment service version. Cross-references against known CVEs.
  • Core exploitation: Attacker exploits the CVE to gain SYSTEM-level code execution on the CA server. With SYSTEM access, runs Mimikatz crypto::capi /secsupport or crypto::cng /export to attempt to export the CA's private key from the software CSP (if not stored in HSM) or to extract the HSM partition PIN from DPAPI-protected storage on disk.
  • Escalation: If the sub-CA private key is in software storage (CAPI) rather than an HSM, export succeeds immediately. If the key is in an HSM, the CA service runs as a service account that has the HSM partition PIN stored in DPAPI — extracting DPAPI secrets from SYSTEM yields the HSM PIN, which combined with the network-attached HSM grants key usage.
  • Impact realization: Sub-CA private key (or effective access to it via HSM PIN) allows the attacker to issue unlimited trusted certificates for any subject in the sub-CA's scope without triggering CA audit logs on the now-controlled CA server.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Sub-CA private key stored in FIPS 140-2 Level 2+ HSM — Mimikatz crypto::capi cannot export a key that is not in the software CAPI store; it is in the HSM's tamper-protected boundary.
  • Control 2 (Preventive): CA server patched within 72 hours of Critical CVEs in CA software, IIS, and underlying OS. Automated patch management (WSUS + SCCM) configured for CA servers with expedited patching SLA.
  • Control 3 (Preventive): CA server network isolation — accessible only from the PAW VLAN or a dedicated CA management VLAN, not from general enterprise networks. A compromised user workstation cannot reach the CA server's management ports.
  • Control 4 (Detective): EDR on CA servers — any attempt to dump DPAPI secrets, access LSA secrets, or call CryptExportKey on a CA server's key container triggers an immediate Critical alert.
  • Control 5 (Detective): CA server vulnerability scan — weekly authenticated scan using Tenable or Qualys; CA server CVE count compared against CISA KEV list; any Critical unpatched CVE triggers P1 remediation ticket.

Evidence and telemetry to validate controls

  • EDR CryptAPI Access Alert: CrowdStrike detects CryptExportKey API call or DPAPI decryption targeting CA key material on CA server → Critical alert.
  • Network Flow Log (CA Management Segment): Any connection from outside the CA management VLAN to CA server ports 443, 135, 445 → immediate alert (unexpected source).
  • CA Server Patch Status Report: Age of CA server OS and application patches compared against CISA KEV and vendor security bulletins — target: zero Critical CVEs > 72 hours.
  • HSM Key Export Audit: HSM audit log — any exportKey operation for the sub-CA key handle is anomalous and generates P0 alert.

T13 — OCSP Cache TTL Residual Window After Revocation

STRIDE category: Tampering, Information Disclosure
Risk: High likelihood, Medium impact
Attacker objective: Exploit the OCSP 4-hour cache TTL so that after a certificate is revoked, an attacker who holds the certificate continues to use it successfully for up to 4 hours before relying parties pick up the revocation — the "OCSP response caching blind spot."

How the attacker would exploit it

  • Prerequisite: Attacker holds a certificate (stolen TLS cert for a service, or a stolen client auth cert). The enterprise's PKI team has detected the compromise and revoked the certificate. OCSP responder's nextUpdate is set to now + 4 hours.
  • Initial action: Attacker's relying party (browser or load balancer) already has a cached OCSP response for the certificate showing "good" status, issued within the last 4 hours. The revocation event occurred after this cached response was issued.
  • Core exploitation: The cached "good" OCSP response remains valid until its nextUpdate timestamp. The attacker uses the revoked certificate during this window — for TLS client auth, API calls, or serving HTTPS traffic. The relying party's OCSP cache returns "good" without querying the OCSP responder again.
  • Escalation: A well-timed attacker who knows the OCSP cache refresh schedule can time the compromise to maximize the post-revocation window. For code-signing certificates, the cache TTL may be even longer (some clients cache for 24+ hours).
  • Impact realization: For a 4-hour window after revocation, the revoked certificate remains functionally valid for any relying party that cached its OCSP status before revocation. Meaningful for code-signing (Windows cache) and high-value client auth certificates.

Blue-team control review — what must stop the chain

  • Control 1 (Mitigation — Primary): Reduce OCSP nextUpdate TTL for high-value certificate templates: code-signing = 15 minutes, admin client auth = 15 minutes. TLS bulk certificates can remain at 4 hours. Shorter TTL means the post-revocation window is at most 15 minutes.
  • Control 2 (Mitigation): OCSP stapling with short staple lifetime — for TLS servers, the stapled OCSP response has its own TTL; setting it to 15 minutes for high-value certs means the server must refresh frequently and will pick up revocation on the next refresh.
  • Control 3 (Procedural): Emergency revocation procedure includes notification to all service owners to force-restart TLS services (which drops existing connections and forces new OCSP checks) and to purge browser caches via HSTS/HPKP mechanisms where possible.
  • Control 4 (Detective): Post-revocation OCSP query monitoring — after revoking a high-value certificate, monitor the OCSP access log for queries against that serial. Continued queries for 15+ minutes after revocation (beyond expected cache TTL) indicate active use of the revoked cert.
  • Control 5 (Compensating): Force OCSP hard-fail for high-value certificate categories on internal enterprise clients via Group Policy (IE/Chrome Enterprise Policy: OCSPEnforcedClients = all internal apps).

Evidence and telemetry to validate controls

  • OCSP Access Log: serial=<revoked-serial> queries after revocation timestamp → active use indicator. Query count × client IP helps identify where the revoked cert is still being used.
  • OCSP nextUpdate TTL by Template: Verify template-specific TTL configuration: certutil -getreg CA\OcspRespUrl and OCSP signing policy; confirm high-value templates have 15-minute TTL.
  • OCSP Staleness Alert: Any OCSP response returned by the responder with nextUpdate < now() → responder configuration error or stale response.

T14 — Unauthorized Subordinate CA Creation by Insider

STRIDE category: Elevation of Privilege, Tampering
Risk: Low likelihood, Critical impact
Attacker objective: Insider PKI admin with access to the Root CA activation process creates an unauthorized subordinate CA certificate during or outside of a ceremony, giving themselves a fully trusted CA they control personally — usable for long-term MITM, code signing, or client authentication attacks.

How the attacker would exploit it

  • Prerequisite: Malicious PKI insider holds a key custodian role or has access to a ceremony workstation. Alternatively, has achieved privileged access to the CA infrastructure and has identified a way to cause the Root CA to sign a CSR they generated.
  • Initial action: During a legitimate ceremony (e.g., sub-CA renewal), the insider secretly replaces one of the CSRs in the signing batch with their own CSR for an unauthorized sub-CA. The ceremony operators verify the authorized CSR's fingerprint — but if the verification step is skipped or the insider has pre-positioned a matching fingerprint, the substitution goes undetected.
  • Core exploitation: Root CA signs the unauthorized CSR. A new sub-CA certificate is issued with CA:TRUE, signed by the trusted Root CA. This sub-CA is trusted by all browsers, OS trust stores, and enterprise relying parties.
  • Escalation: Insider retains the unauthorized sub-CA's private key. Uses it to issue certificates for any domain, sign code, or create client auth certificates for any identity. No CA audit log records these because issuance happens entirely on the insider's systems using the offline sub-CA key.
  • Impact realization: Indefinite trusted certificate issuance authority outside the enterprise's control; undetectable until CT log monitoring or revocation checking identifies the rogue sub-CA.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): CSR fingerprint verification is a mandatory ceremony step — the CSR's SHA-256 fingerprint is published in the ceremony script 48 hours before the ceremony and verified independently by the auditor against the live ceremony CSR. Any substitution is detected.
  • Control 2 (Preventive): M-of-N quorum — a single insider cannot substitute a CSR without the other custodians and auditor witnessing the correct fingerprint mismatch. Multiple non-colluding parties verify independently.
  • Control 3 (Preventive): Comprehensive sub-CA registry — all authorized sub-CA serial numbers are published in an authoritative registry and in the enterprise's internal CT log. Any sub-CA not in the registry triggers an immediate P0 alert.
  • Control 4 (Detective): CT log monitoring for certificates with CA:TRUE extension issued by the Root CA OID — any new sub-CA that is not in the authorized registry is a P0 alert immediately.
  • Control 5 (Detective): Post-ceremony HSM audit log review — compare the list of CSRs signed during the ceremony (from HSM log) against the list in the ceremony script. Any additional signed CSR not in the script = unauthorized sub-CA.

Evidence and telemetry to validate controls

  • HSM Audit Log (Post-Ceremony): signCertificate operation count vs ceremony script expected CSR count — any extra signing operation = unauthorized sub-CA.
  • Sub-CA Registry: Authoritative list of sub-CA serial numbers and their OIDs — published in a version-controlled repo; any addition requires a PR merged by two PKI team leads.
