Scenario: Designing and rolling out a Privileged Access Workstation program for 150 administrators (Tier 0 DC/PKI/cloud-org admins, Tier 1 server/app admins, Tier 2 helpdesk/network admins) across on-prem AD, Azure, and GCP, with a constrained hardware budget requiring tiered deployment strategies.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

The review focuses on attack mechanics, prerequisites, command-level purple-team validation examples, detection evidence, and control validation. Commands use placeholder values and are intended for an owned lab or approved internal purple-team exercise only. High-risk techniques are described at the technique level and paired with safe defensive validation commands.

2. Highest-Risk Design Decision

Whether admin accounts are separated from daily-use accounts and whether device compliance is enforced at the Conditional Access layer for every privileged authentication. If an administrator uses a dual-use account (same UPN for email and admin tasks) on a non-hardened device, a phishing email or browser exploit on that device immediately yields admin credentials — and since there is no PAW device requirement at the token-issuance layer, those credentials work from anywhere. The correct design requires: (1) separate admin accounts with no mailbox, no SPN, in Protected Users group; (2) every admin account's Conditional Access policy requiring deviceCompliance = true AND deviceTrustType = AzureADJoined so that token issuance is hardware-gated to the enrolled PAW; and (3) FIDO2 hardware keys as the only permitted MFA method. Without all three, compromising the admin's daily-use laptop yields full domain admin access regardless of any other control.

6. Threat-by-Threat Technical Review

T1 — Admin on General-Purpose Workstation (No PAW)

STRIDE category: Spoofing, Information Disclosure
Risk: High likelihood, Critical impact
Attacker objective: Compromise the admin's daily-use laptop to steal cached admin credentials, then use those credentials to access domain controllers, cloud control planes, or other Tier 0 targets — bypassing the entire PAW program.

How the attacker would exploit it

  • Prerequisite: Admin uses a dual-use account (jsmith@corp.com with both email access and admin privileges) and sometimes performs admin tasks from their daily-use laptop "just this once."
  • Initial action: Spearphish targeting the admin's corporate email (jsmith@corp.com) with a malicious Office macro or a link to a browser exploit kit. High success rate because the admin's email account is publicly enumerable via LinkedIn.
  • Core exploitation: Macro or browser exploit executes on the daily-use laptop. Attacker runs Mimikatz sekurlsa::logonpasswords or sekurlsa::wdigest against LSASS. If the admin recently ran any privileged commands on this device, the NT hash, Kerberos TGT, or cleartext credentials are present in LSASS memory (unless Credential Guard is enabled).
  • Escalation: Attacker uses impacket-psexec or CrackMapExec with the stolen NT hash via pass-the-hash against a domain controller or admin server. If the account is in Protected Users, NTLM is blocked but Kerberos TGT may still be in memory — use Rubeus dump to extract TGT and pass-the-ticket.
  • Impact realization: Domain Admin or Global Admin access. Attacker can DCSync all credentials, issue certificates from Enterprise CA, modify GPOs for persistence, or destroy the cloud organization.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Separate admin accounts — jsmith-adm@corp.com with no mailbox, no SPN, no MX reachability. Admin account never touched on daily-use device. Enforced at AD account design: admin OU policy blocks interactive login on non-PAW workstations via GPO Deny logon locally for admin accounts on all workstations not in the PAW OU.
  • Control 2 (Preventive): Protected Users group for all admin accounts — blocks NTLM authentication, credential caching, Kerberos delegation, DES/RC4 encryption. Even if daily-use device is compromised, NTLM hash is unusable and credential cache is empty.
  • Control 3 (Preventive): Conditional Access: admin role + deviceCompliance = true + deviceTrustType = AzureADJoined required for token issuance to any admin-scoped app. Daily-use laptop fails this check.
  • Control 4 (Detective): Entra ID Sign-In alert: any admin-role account successfully signing in with deviceDetail.isCompliant != true → Critical alert (indicates CA policy failure or bypass).
  • Control 5 (Detective): SIEM rule: admin_account_signin WHERE src_device NOT IN paw_device_inventory — additional signal for cross-referencing device identity.

Evidence and telemetry to validate controls

  • Entra ID Sign-In Logs: deviceDetail.isCompliant, deviceDetail.trustType, conditionalAccessPolicies.result — all admin logins must show isCompliant=true and CA result success; any failure should immediately alert.
  • AD Security Event Log (on DCs): Event 4625 (logon failure) with LogonType=3 (Network) and AuthenticationPackage=NTLM from an admin account — NTLM from an admin account in Protected Users should be a 100% failure rate.
  • Protected Users Validation: Get-ADGroupMember "Protected Users" — confirm all Tier 0 and Tier 1 admin accounts are members.
  • Logon Rights GPO Audit: Verify Deny logon locally and Deny logon through RDP include admin account groups for all workstation OUs.

T2 — Browser / Email Execution on PAW

STRIDE category: Spoofing, Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Exploit the PAW's privileged context by tricking an admin into browsing the web or opening email on the PAW, then using a browser exploit, malicious document, or credential-harvesting page to steal the active admin session.

How the attacker would exploit it

  • Prerequisite: Admin finds it inconvenient to switch between their daily-use device and PAW, and uses the PAW's browser for "quick lookups" or to check email. Alternatively, admin opens a phishing email accidentally on the PAW thinking the hardened browser would protect them.
  • Initial action: Attacker sends spearphish to the admin's corporate email. Admin opens the email on the PAW (convenience failure). Malicious link leads to a browser exploit kit or credential harvesting page.
  • Core exploitation: If WDAC does not explicitly block msedge.exe and outlook.exe, the admin can launch these applications. A browser exploit (e.g., via a JavaScript engine CVE) achieves code execution within the Edge renderer process. Alternatively, admin enters admin credentials on a phishing page — credentials are captured while the admin has an active PIM-elevated session.
  • Escalation: Code execution in Edge renderer on PAW escalates to SYSTEM-level if a local privilege escalation CVE is combined. With SYSTEM access on the PAW, attacker can extract the TPM-bound device certificate (limited), dump LSASS for the active admin session's Kerberos tickets, or install a persistent C2 agent that survives within WDAC's allowed context.
  • Impact realization: Active admin session hijack — the PAW was in the middle of a PIM-elevated Tier 0 session; attacker inherits all privileges of that session.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): WDAC enforced-mode policy explicitly denying chrome.exe, msedge.exe, firefox.exe, iexplore.exe, outlook.exe, thunderbird.exe — kernel-level block, not bypassed by local admin. Policy deployed via Intune: Windows Defender Application Control profile → custom WDAC XML.
  • Control 2 (Preventive): PAW VLAN network blocks all internet-bound DNS and TCP/UDP traffic at the firewall — even if a browser somehow executes (WDAC bypass), it cannot reach external C2 or phishing sites.
  • Control 3 (Preventive): Separate daily-use device for email and browsing — admin physically separate from PAW when doing daily work. Physical and procedural separation enforced by PAW program policy.
  • Control 4 (Detective): EDR alert: process creation for msedge.exe, chrome.exe, firefox.exe, or outlook.exe on any device with MDM tag PAW → immediate Critical alert, regardless of WDAC status. If WDAC blocked it, it's an attempted violation; if WDAC did not block it, it's an active compromise.
  • Control 5 (Response): Automated MDM isolation on browser-execution alert: Intune compliance policy set to non-compliant → CA revokes all tokens for admin account within 5 minutes → admin session dropped.

Evidence and telemetry to validate controls

  • WDAC Audit Log (EventID 3076 - Audit; 3077 - Block): Event 3077 shows blocked execution of browser/email binaries on PAW — confirms WDAC is preventing execution. Absence of 3077 events combined with 3076 (audit-only mode) means WDAC is not fully enforced.
  • EDR Process Creation Events: CrowdStrike Falcon ProcessCreated event for browser binaries on PAW-tagged devices — should be absent in normal operation; any event = immediate alert.
  • Network Flow Logs (PAW VLAN): DNS query to public resolver or TCP connection to port 443 on non-admin IP from PAW VLAN — should never occur; any event = firewall misconfiguration or WDAC bypass.
  • MDM Compliance Isolation Log: Intune DeviceComplianceStateChange → non-compliant followed by Entra ID token revocation within 5 minutes.

T3 — Stolen FIDO2 Hardware Key

STRIDE category: Spoofing
Risk: Low likelihood, High impact
Attacker objective: Physically steal an admin's FIDO2 hardware key (YubiKey) and use it — along with the admin's PIN or password — to authenticate to the IdP and activate PIM roles from an attacker-controlled device.

How the attacker would exploit it

  • Prerequisite: Physical access to the admin's FIDO2 key (theft from desk, bag, or credential theft paired with key theft). FIDO2 requires user presence — a physical button press on the key — so remote-only attacks are insufficient.
  • Initial action: Attacker obtains both the FIDO2 key and the admin's PIN (via shoulder surfing, keylogging on non-PAW device, or social engineering). The attacker needs both factors: the physical key and the PIN.
  • Core exploitation: Attacker plugs the FIDO2 key into their device and attempts to authenticate to the IdP. The FIDO2 assertion is origin-bound to corp.idp.example.com — so it only works against the real IdP. User presence is required (button press on key), which the attacker provides manually. Attacker authenticates successfully from an attacker-controlled device (which is not the enrolled PAW).
  • Escalation: If Conditional Access only checks FIDO2 authentication and does not additionally require deviceCompliance = true, the attacker can authenticate from any device with the stolen key. They then activate PIM for Tier 0 roles.
  • Impact realization: Attacker gains an admin session with PIM-elevated Tier 0 role — full access to domain controllers, PKI, and cloud organization.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive — Critical): Conditional Access requires BOTH FIDO2 authentication AND deviceCompliance = true AND deviceTrustType = AzureADJoined. Stolen FIDO2 key on attacker's non-enrolled device fails the device compliance check regardless of successful FIDO2 assertion.
  • Control 2 (Preventive): FIDO2 key PIN mandatory (not optional) — without PIN, the key requires only physical presence; with PIN, attacker also needs the PIN. Configure via YubiKey Manager: ykman fido access change-pin.
  • Control 3 (Preventive): FIDO2 enterprise attestation — IdP verifies the YubiKey attestation certificate matches an approved hardware model from the enterprise key management inventory. Unathorized or cloned keys rejected.
  • Control 4 (Detective): PIM activation from new/unknown device ID — if attacker authenticated with stolen key from their own (non-enrolled) device, the deviceId in the PIM audit log will be absent or different from the admin's registered PAW device ID.
  • Control 5 (Response): Admin reports key lost/stolen → immediate FIDO2 key revocation in IdP (Entra: Remove security key via Authentication methods); all active sessions revoked; new key issued via secure in-person process.

Evidence and telemetry to validate controls

  • PIM Audit Logs: requestorDeviceId — every PIM activation should come from a registered PAW device ID. Activation from an unrecognized device ID triggers immediate alert.
  • Entra ID Sign-In Logs: authenticationDetail.AuthenticationMethod = FIDO2 SecurityKey with deviceDetail.isCompliant = false — FIDO2 on non-compliant device should be blocked by CA; any success here is a critical gap.
  • FIDO2 Key Inventory Log: Enterprise key inventory (YubiKey Manager or hardware token management system) — reported lost/stolen keys should be revoked within 15 minutes.
  • PIM Activation Geolocation: Activation from a country or city inconsistent with admin's expected location → alert.

T4 — Fake Device Enrolled as Compliant in MDM

STRIDE category: Spoofing
Risk: Medium likelihood, Critical impact
Attacker objective: Enroll an attacker-controlled device into the corporate MDM as a "compliant" PAW-equivalent, bypassing the device compliance gate in Conditional Access, and use it to authenticate as an admin from the attacker's device.

How the attacker would exploit it

  • Prerequisite: Attacker has compromised an admin account with sufficient privileges to enroll a new device in Intune (or knows the device enrollment URL and an enrollment token/credential). Alternatively, a compromised helpdesk account can enroll new devices on behalf of users.
  • Initial action: Attacker navigates to https://portal.manage.microsoft.com and initiates a device enrollment using the compromised admin or helpdesk credentials. On Windows, executes %windir%\system32\deviceenroller.exe /c /autoenrollmdm or uses a custom enrollment profile from the Intune Company Portal.
  • Core exploitation: If Intune enrollment does not require hardware attestation (TPM attestation of the device's hardware identity), any device can be enrolled and will receive a compliance certificate. The attacker's device becomes "compliant" in the MDM inventory.
  • Escalation: With a compliant enrolled device, the attacker's Conditional Access evaluation now passes — deviceCompliance = true — even though the device is attacker-controlled. Attacker uses admin credentials + FIDO2 key (if available) from this "compliant" device to access admin portals.
  • Impact realization: The device compliance gate — the primary PAW enforcement mechanism — is completely bypassed. Attacker can now authenticate with admin credentials from their own device as if it were an approved PAW.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Require TPM hardware attestation for Intune enrollment — Intune Enrollment restrictions → Windows enrollment → Require TPM attestation. Only devices with a genuine TPM chip that can produce a hardware attestation can enroll. Attacker's non-TPM device or VM is rejected at enrollment.
  • Control 2 (Preventive): Require existing admin approval for PAW device enrollment — enrollment for admin-tier devices requires a second authorized Tier 0 admin to approve the enrollment event in Intune before the device is marked compliant.
  • Control 3 (Preventive): Conditional Access device filter: device.extensionAttribute1 = PAW — devices are not just required to be compliant but also required to have a specific extension attribute set only by a Tier 0 PAW provisioning pipeline (not by the device itself).
  • Control 4 (Detective): Intune enrollment alert: any new device enrollment for a user in an admin role group → immediate alert to PAW program team. Enrollment inventory reconciled against physical asset register weekly.
  • Control 5 (Detective): Entra ID device object audit: any new AzureADJoined device for an admin user created outside the approved provisioning workflow (Autopilot or manual Tier 0 enrollment) → alert.

Evidence and telemetry to validate controls

  • Intune Enrollment Logs: enrollmentType, tpmAttestation field — confirm all PAW-tagged devices have tpmAttestation = success; devices without TPM attestation should not exist in PAW tier.
  • Entra ID Audit Logs: ActivityDisplayName = Register device — all device registrations for admin users should correlate with Autopilot provisioning events; ad-hoc enrollments are anomalous.
  • Conditional Access Device Filter Audit: Periodic query: all admin-account successful sign-ins where device.extensionAttribute1 != PAW — should be empty.
  • PAW Asset Inventory Reconciliation: Monthly comparison of Intune enrolled PAW devices vs physical hardware asset register — any Intune device without a matching physical serial number is suspect.

T5 — Pass-the-PRT (Primary Refresh Token Theft)

STRIDE category: Spoofing, Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Extract the Primary Refresh Token (PRT) from the PAW's LSASS process or Windows Cloud Authentication Broker to achieve silent SSO as the admin to all Entra ID-integrated applications, bypassing FIDO2 re-authentication.

How the attacker would exploit it

  • Prerequisite: Attacker has code execution on the PAW (via T2 browser exploit or T8 supply chain). The PAW is an Entra ID-joined Windows device; Windows stores the PRT in LSASS (within the CloudAP plugin).
  • Initial action: Attacker uses DSInternals PowerShell module or Mimikatz lsadump::cloudap to extract the PRT from LSASS. The PRT is a long-lived (14 days) opaque token that provides silent SSO to all Microsoft services including Entra ID admin portals.
  • Core exploitation: Extracted PRT can be imported into the attacker's Chrome browser using a DPAPI-protected Cookies database manipulation or via the RequestToken API. The BrowserCore.exe on Windows uses the PRT to silently issue access tokens without re-authenticating. Attacker uses this PRT to get access tokens for the Entra ID admin portal, PIM activation endpoint, and Azure Resource Manager.
  • Escalation: With a valid PRT, attacker silently activates PIM roles (if CA does not require device compliance at PIM activation time), modifies admin accounts, or issues new role assignments. The PRT-based SSO bypasses FIDO2 because the PRT was already issued after FIDO2 authentication on the legitimate PAW.
  • Impact realization: Full cloud control plane access under the admin's identity with a 14-day token — long enough to establish persistent access, create backdoor accounts, and exfiltrate data.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Enable Windows Credential Guard (HVCI mode) on all PAWs — moves LSASS into an isolated VSM (Virtual Secure Mode) container. DSInternals/Mimikatz cannot access the CloudAP credential material in VSM-protected LSASS. Verify via Get-ComputerInfo | Select-Object DeviceGuardVirtualizationBasedSecurityStatus.
  • Control 2 (Preventive): Conditional Access re-evaluation at PIM activation: configure a dedicated CA policy for the PIM activation endpoint (https://pim.azure.com) that requires device compliance at activation time — not just at initial sign-in. This means PIM activation from the attacker's non-PAW device is denied even with a valid PRT.
  • Control 3 (Preventive): Short PRT lifetime for admin accounts — not standard configuration, but achievable via Entra ID token lifetime policy (Sign-in frequency in CA: set to 4 hours for admin role accounts, requiring re-authentication). PRT is reissued only on fresh authentication.
  • Control 4 (Detective): EDR alert: any process other than lsass.exe itself or approved security tooling accessing lsass.exe memory via OpenProcess + PROCESS_VM_READ — Windows Security Event 10 (Sysmon) or CrowdStrike CredentialAccess detection.
  • Control 5 (Detective): Entra ID Sign-In Logs: token issued to a different deviceId than the one that last authenticated the admin — PRT is device-bound; if a token is somehow issued from a different device, it's an anomaly.

Evidence and telemetry to validate controls

  • Credential Guard Status: Get-ComputerInfo | Select-Object DeviceGuardVirtualizationBasedSecurityStatus returns Running on all PAW fleet devices.
  • Windows Security Event 10 (LSASS Access): Sysmon EventID 10: TargetImage = C:\Windows\System32\lsass.exe, GrantedAccess from a non-allowlisted process — immediate Critical alert.
  • Entra ID Sign-In Logs: tokenIssuancePolicy.signInFrequency — confirm sign-in frequency policy is applied to admin-role accounts.
  • CrowdStrike Falcon: Detection category CredentialAccess, technique T1003.001, on PAW-tagged devices.

T6 — PIM Activation from Non-PAW Device

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Use stolen admin credentials (without a PAW) to activate PIM roles — exploiting a gap where the Conditional Access policy enforcing device compliance does not cover the PIM activation endpoint specifically.

How the attacker would exploit it

  • Prerequisite: Attacker has the admin's UPN, password, and potentially an OTP from a phishing campaign. The Conditional Access policy covers normal sign-in to admin applications but does not include a separate policy for the Azure PIM portal (https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon).
  • Initial action: Attacker navigates to the Entra ID PIM portal. Authenticates with stolen credentials. If FIDO2 is required for admin apps but only "any MFA" is required for the PIM portal itself (a misconfiguration), attacker can bypass the hardware key requirement.
  • Core exploitation: Attacker activates a Tier 1 or even Tier 0 eligible role from their non-PAW device. No PAW device compliance check was applied to the PIM activation request. Justification field is filled with generic text.
  • Escalation: With an active admin role, attacker navigates to the Azure portal and begins modifying resources — creating new admin accounts, extracting secrets from Key Vault, or disabling diagnostic logs.
  • Impact realization: Unauthorized privilege escalation; potential cloud resource destruction or data exfiltration, depending on the activated role's permissions.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Dedicated Conditional Access policy for the PIM activation endpoint — Application = Azure AD PIM → require device compliance + FIDO2.   • Control 2 (Preventive): PIM requires justification text with valid ticket ID (validated via ITSM API at activation time). No ticket = no activation, regardless of device.
  • Control 3 (Preventive): Tier 0 roles require peer approval — activation does not take effect until a second Tier 0 admin approves. Approval workflow itself requires PAW access, breaking the chain.
  • Control 4 (Detective): PIM audit log alert: requestorDeviceId not in PAW device inventory, or authenticationMethod on the activation sign-in is not FIDO2 → immediate alert.
  • Control 5 (Detective): SIEM rule: PIM activation AND src_device NOT IN paw_inventory → Critical alert within 2 minutes.

  • PIM Audit Logs: requestorDeviceId, activationType, justification — every activation should have a known PAW device ID, FIDO2 MFA method, and a valid ticket ID.
  • Entra ID Sign-In Logs: Sign-in associated with PIM activation request — check authenticationRequirement was multiFactorAuthentication with authenticationMethod = FIDO2.
  • PIM Approval Workflow Logs (Tier 0): All Tier 0 activations have an approval record; absent approval record = activation bypassed workflow.
  • Role Activation Frequency Report: Monthly report on PIM activation counts per admin; spikes above baseline warrant investigation.

STRIDE category: Information Disclosure, Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Compromise the PAM (Privileged Access Management) platform — CyberArk, BeyondTrust, or Delinea — to dump the stored credentials vault, gaining access to all credentials managed by the PAM tool simultaneously.

How the attacker would exploit it

  • Prerequisite: Attacker identifies the PAM platform (CyberArk PAS, BeyondTrust Password Safe, or Delinea Secret Server) from network reconnaissance or OSINT (job postings mentioning PAM vendor). Alternatively, a compromised Tier 1 admin account has PAM access.
  • Initial action: Attacker exploits a vulnerability in the PAM web interface (CVE in the management portal), or uses compromised PAM API credentials to call the REST API. CyberArk's Central Policy Manager (CPM) API, for example, exposes credential retrieval endpoints.

Blue-team control review — what must stop the chain

  • Control 5 (Detective): PAM session recording reviewed for all Tier 0 credential retrievals — each retrieval is logged with actor, credential name, justification, and session recording. Out-of-scope or unusual retrieval patterns trigger review.

  • PAM Audit Log: event_type = credential_retrieve, actor, account_name, timestamp — every retrieval must have a corresponding change ticket; bulk retrievals (> 5 in 10 minutes) alert immediately.
  • PAM API Access Log: API calls to /Accounts/{id}/Password/Retrieve — baseline call frequency per API client; deviations alert.

STRIDE category: Tampering, Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Compromise a software update for an admin tool (RSAT, SSH client, network management utility, RMM agent) that runs on PAWs, delivering a backdoored version that executes in the admin context during privileged sessions.

How the attacker would exploit it

  • Prerequisite: Nation-state or highly capable attacker compromises a vendor's software signing key or update distribution channel. Target tools: RSAT extensions, PuTTY, WinSCP, network device management tools (SolarWinds, PRTG agent), or third-party PowerShell modules used by admins.

T10 — Admin Account Helpdesk Reset Social Engineering

STRIDE category: Spoofing
Risk: Medium likelihood, High impact
Attacker objective: Social-engineer the IT helpdesk to reset the FIDO2 key or MFA for an admin account, then use the reset window to authenticate without the hardware key requirement — bypassing the strongest control in the PAW architecture.

How the attacker would exploit it

  • Prerequisite: Attacker has the admin's name, title, email, and some personal information (gathered from LinkedIn, public conference talks, or previous phishing recon). Attacker calls the IT helpdesk impersonating the admin.
  • Initial action: "Hi, this is [admin name], I've lost my YubiKey and I'm traveling. I need to access the environment urgently for an incident. Can you reset my MFA so I can use my phone temporarily?" Helpdesk, under pressure, resets the MFA method to allow Microsoft Authenticator or SMS as a fallback.
  • Core exploitation: With MFA reset, Conditional Access now accepts a push notification or SMS code as satisfying authenticationStrength. Attacker, who has the admin's password from a prior phishing campaign, authenticates with password + SMS OTP — no FIDO2 hardware key required.
  • Escalation: Attacker now has an authenticated admin session from any device without the PAW requirement (because the device compliance check may not block a downgraded MFA session depending on CA policy specifics). Accesses PIM, activates roles.
  • Impact realization: The strongest link in the PAW architecture — FIDO2 hardware key — was bypassed via a 5-minute phone call. All technical controls are moot.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Admin account MFA reset requires in-person identity verification with physical ID and two-person authorization (admin's manager + IT security team). No MFA reset via phone or email ticket for admin-tier accounts. Documented in PAW program policy.
  • Control 2 (Preventive): Emergency access procedures — if admin loses FIDO2 key, the temporary access path uses a hardware backup key (each admin has a spare YubiKey stored in a sealed envelope in the PAM vault), not a software MFA downgrade.
  • Control 3 (Preventive): Conditional Access policy explicitly denies admin-role sign-ins with MFA methods other than FIDO2 (Authentication Strength policy: Allowed methods = FIDO2 security key only). Even if helpdesk resets to Microsoft Authenticator, CA blocks the sign-in.
  • Control 4 (Detective): Alert on admin account MFA method changes: Entra ID Audit Log ActivityDisplayName = Update authentication methods for any admin-role UPN → immediate alert to CISO and IT security lead.
  • Control 5 (Response): Any unplanned MFA method change on an admin account triggers an automatic 24-hour review hold — the account is temporarily suspended from PIM role activation until the change is verified as authorized.

Evidence and telemetry to validate controls

  • Entra ID Audit Logs: ActivityDisplayName = "Update user authentication methods" or "Delete user authentication methods" targeting admin UPNs → alert within 2 minutes with actor, target, and new/old authentication methods.
  • Entra ID Sign-In Logs Post-Reset: If sign-in succeeds with authenticationMethod = MicrosoftAuthenticator for an admin account that should only use FIDO2 → Critical alert (CA policy gap).
  • Helpdesk Ticket Audit: MFA reset tickets for admin-tier accounts that did NOT have a manager approval and IT security sign-off → control failure indicator.
  • Authentication Methods Policy Audit: Monthly report of authentication methods registered per admin UPN — any admin with SMS, phone call, or TOTP registered (in addition to FIDO2) represents a policy enforcement gap.

T11 — PAW VLAN Firewall Misconfiguration

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Exploit overly permissive firewall rules on the PAW VLAN boundary — whether allowing PAW to reach production user VLANs, the internet, or non-admin application servers — to use the PAW as a pivot point for lateral movement.

How the attacker would exploit it

  • Prerequisite: Attacker has gained code execution on a PAW (via T2 or T8) or is an insider admin who has discovered the misconfigured firewall rules. Alternatively, attacker has compromised a user workstation and discovers it has unexpected routes to the PAW VLAN.
  • Initial action: From the compromised PAW, attacker runs ip route / route print and nmap -sn 10.0.0.0/8 to enumerate reachable subnets. Discovers that the PAW VLAN can reach production workstation VLAN (10.20.0.0/24) — intended to be unreachable.
  • Core exploitation: From PAW, attacker can now SMB into production workstations (which have no LSASS Credential Guard), dump NTLM hashes from lsass, and use those for lateral movement within the production environment. Alternatively, PAW can reach internet (misconfigured egress rule), enabling C2 callback.
  • Escalation: From production workstations, attacker finds service account credentials, moves to database servers, or escalates to a different admin account that does not have PAW restrictions.
  • Impact realization: The PAW — which was supposed to be the most isolated and trusted segment — becomes the launch pad for attacks against the production environment.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Firewall policy review for PAW VLAN at minimum quarterly; all changes to PAW VLAN egress rules require security team approval and are tracked in change management. Default-deny outbound from PAW VLAN except explicit management ports to specific admin targets.
  • Control 2 (Preventive): PAW VLAN egress allowlist maintained as code (Terraform/Ansible) — network policy as code with peer review for any change to PAW VLAN ACLs. Prevents ad-hoc firewall rule additions.
  • Control 3 (Detective): Network flow monitoring on PAW VLAN egress: alert on any flow from PAW VLAN to a destination not in the approved admin management CIDR list. Any connection to user workstation VLAN or internet → Critical alert.
  • Control 4 (Detective): Firewall rule drift detection — automated comparison of deployed firewall rules against the approved IaC baseline every 15 minutes. Any drift triggers immediate alert and auto-revert.
  • Control 5 (Response): On detection of unexpected PAW VLAN egress flow: automated ACL addition blocking the specific flow; manual investigation of the rule source; if misconfiguration confirmed, full review of PAW VLAN ruleset.

Evidence and telemetry to validate controls

  • Network Flow Logs (PAW VLAN Egress): src_vlan = PAW_VLAN AND dst_ip NOT IN admin_management_cidr_list — alert threshold is one event (zero tolerance for unexpected PAW egress).
  • Firewall Rule Drift Report: Terraform plan output comparing deployed state against IaC baseline — any + (new rule) that wasn't committed to Git is a control gap.
  • SIEM PAW Egress Alert History: Baseline: zero alerts per week after tuning period. Any alert volume above zero requires investigation.
  • Penetration Test Results: Annual PAW segmentation test — pentest team attempts to reach production workstation VLAN from PAW VLAN; result must be "blocked."

T12 — Logging Gap — PAW Not Forwarding to SIEM

STRIDE category: Repudiation
Risk: Medium likelihood, High impact
Attacker objective: Exploit a gap in PAW telemetry forwarding to conduct malicious admin actions that are not logged in the centralized SIEM, making forensic reconstruction impossible and enabling deniability for insider threats.

How the attacker would exploit it

  • Prerequisite: Insider admin or attacker with PAW access discovers that the SIEM agent (CrowdStrike Falcon, Splunk UF, or Microsoft Monitoring Agent) on their PAW is not functional — misconfigured, uninstalled, or stopped. Or the attacker actively disables the agent.
  • Initial action: Attacker (insider) verifies that the PAW's EDR/SIEM agent is not running: Get-Service CSSensorService returns Stopped. Alternatively, attacker stops the agent: Stop-Service -Name CSSensorService (requires local admin — which PAW admins may have).
  • Core exploitation: With no local telemetry, the attacker's actions on the PAW (commands typed in privileged sessions, PowerShell executions, network connections) are not captured by the EDR or SIEM. Only the PAM session recording (if separate from the PAW) preserves evidence.
  • Escalation: Insider performs exfiltration, creates backdoor accounts, or modifies configurations — all invisible to the SIEM. Post-incident forensics finds no evidence of the actions.
  • Impact realization: Attacker's malicious actions are forensically invisible; compliance audits fail; incident response cannot reconstruct the attack timeline; insider escapes accountability.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): EDR agent (CrowdStrike Falcon) runs as a kernel-level driver with tamper protection enabled — cannot be stopped by local admin or even SYSTEM without specific anti-tamper bypass. Enable via Falcon console: Sensor Tamper Protection = Enabled.
  • Control 2 (Preventive): SIEM agent watchdog via MDM — Intune compliance policy includes a compliance check script that verifies the EDR/SIEM agent service is running; non-running agent = device non-compliant = CA revokes admin tokens.
  • Control 3 (Detective): SIEM heartbeat monitor — each PAW sends a heartbeat event every 60 seconds. Gap > 5 minutes during business hours → Critical alert (agent stopped, device offline, or network issue). SIEM inactivity = silent PAW = investigation.
  • Control 4 (Detective): Windows Security Event Forwarding (WEF) as a backup — Windows PAWs configured to forward Security events to a Windows Event Collector (WEC) server via GPO, independent of the EDR agent. Even if EDR is stopped, WEF continues forwarding via native Windows channels.
  • Control 5 (Response): On SIEM heartbeat gap alert: automated MDM compliance mark → CA revokes admin tokens for that PAW within 5 minutes; admin must investigate and restore agent before PIM activation is re-allowed.

Evidence and telemetry to validate controls

  • SIEM Heartbeat Log: PAW heartbeat events per device — count(heartbeat_events) by device_id, bin(TimeGenerated, 5m) — any device with zero events in a 5-minute window during business hours alerts.
  • MDM Compliance Script Result: Intune compliance script Check-EDRAgent.ps1 results for all PAWs — non-compliant result for any PAW.
  • EDR Anti-Tamper Audit: CrowdStrike Falcon console → Sensor anti-tamper protection status per device — all PAWs must show Enabled.
  • Windows Event Forwarding Subscription Log (WEC Server): All PAW device IDs should appear in the active subscriptions list; missing device = forwarding gap.

T13 — Privileged Session Outside Approved Maintenance Window

STRIDE category: Elevation of Privilege (Insider)
Risk: Medium likelihood, High impact
Attacker objective: Insider admin activates PIM roles and conducts malicious operations outside of business hours or approved maintenance windows — exploiting the lower scrutiny and reduced monitoring coverage during off-hours.

How the attacker would exploit it

  • Prerequisite: Insider admin (disgruntled employee or compromised admin account) wants to conduct malicious activity with lower detection probability. Knows that SOC is understaffed on weekends and that after-hours PIM activations generate fewer immediate responses.
  • Initial action: Insider activates a Tier 0 or Tier 1 PIM role at 2 AM on a Saturday with a generic justification ("system maintenance required"). No change ticket is linked.
  • Core exploitation: During the off-hours window, insider runs DCSync to extract all AD hashes, exports a KeyVault backup, or exfiltrates SAP data. SOC alert fires but is triaged by an on-call analyst who deprioritizes it as "likely legitimate maintenance."
  • Escalation: Extracted credentials are sold or used immediately. Exfiltrated data is transmitted out via encrypted channels. Activity is low-and-slow — no immediate signs of ransomware or destructive impact.
  • Impact realization: Data exfiltration completed undetected; credentials harvested; forensic investigation weeks later links the activity to the insider but by then the data is already gone.

Blue-team control review — what must stop the chain

  • Control 1 (Preventive): Tier 0 PIM activations outside business hours require a linked change ticket AND real-time peer approval from a second Tier 0 admin. If no second admin approves within 15 minutes, activation is automatically denied.
  • Control 2 (Preventive): PIM activation for Tier 0 roles between 20:00–06:00 or on weekends requires a manager emergency approval — a separate approval workflow that sends an automated phone call to the admin's manager.
  • Control 3 (Detective): SIEM rule: PIM activation for Tier 0 role AND activationTime.hour BETWEEN 20 AND 6 AND justification NOT MATCH CHG[0-9]{7} → Critical alert, pages on-call SOC analyst with 15-minute response SLA.
  • Control 4 (Detective): PAM session recording automatically flagged for human review when session occurs outside business hours. All off-hours Tier 0 sessions reviewed by security team within 24 hours.
  • Control 5 (Detective): UEBA baseline on admin activity hours per role — alert when admin activates a role at an hour/day outside their individual 30-day baseline (e.g., admin who always works 9-6 Monday-Friday activating at 2 AM Saturday).

Evidence and telemetry to validate controls

  • PIM Audit Logs: activationTime, roleName, requestorUPN, justification — all off-hours activations must have peer-approval records; absence of approval record is control failure.
  • SIEM Off-Hours Activation Alert History: Alert fires per week — baseline is 0-2 per week (emergency changes); spike to > 5 per week warrants investigation of change management compliance.
  • PAM Session Review Backlog: All flagged off-hours sessions reviewed within 24 hours — review completion tracked in security operations dashboard.
  • UEBA Individual Baseline Deviation: admin_activity_hour_anomaly signal per admin — first occurrence in off-hours triggers alert; repeated off-hours activity by same admin escalates to HR review.
