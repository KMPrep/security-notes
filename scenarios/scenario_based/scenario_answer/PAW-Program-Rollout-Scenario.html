Scenario 3 — Privileged Access Workstation (PAW) Program Rollout

Context: Designing a PAW program for 150 administrators who manage cloud, on-prem, and network infrastructure. Budget is constrained — dedicated hardware cannot be deployed for all admins.

1. ASSETS — What are we protecting?

Asset Sensitivity Why it Matters What Goes Wrong if Compromised
Admin credentials (Tier 0 — DC, PKI, cloud root/org admin) Critical Full control of identity fabric; complete enterprise takeover possible Attacker issues certificates, creates backdoor accounts, modifies GPOs, controls all cloud workloads
Privileged Identity Management (PIM) role assignments Critical Time-limited elevation; if tampered with, attacker grants themselves permanent DA Persistent admin access invisible to standard account auditing
Admin forest / ESAE trust relationship Critical Separate hardened forest for admin accounts; one-way trust to production; compromise collapses the separation model Admin accounts become equivalent to production accounts; full forest pivot
PAW device identity certificates / TPM attestation keys Critical Conditional Access evaluates device identity for admin login; fake or cloned device bypasses access control Attacker registers a fake compliant device and authenticates as admin
Tier 1 admin credentials (server, application, cloud service admins) High Lateral movement to production servers, databases, critical SaaS applications Data exfiltration, ransomware deployment, production manipulation
PAW MDM / Intune compliance state High Device compliance drives Conditional Access gate; tampering with compliance unlocks admin login from any device Compliance policy bypass = all device enforcement collapses
Session recordings (PAM tool) High Audit trail and forensic evidence; tamper = deniability for malicious admin actions Insider threat activity becomes unprovable; compliance audit failure
PAW operating system and application baseline High Compromised PAW is a beachhead with admin-level credentials cached in memory Direct credential theft; lateral movement to every admin target in scope
Network management interfaces (OOBM, IPMI, iDRAC, vCenter) High Out-of-band management channels often bypass host-level security; accessible only from admin VLAN Full host takeover without touching the OS; persistence even through re-imaging
Cloud control plane (AWS Organizations, GCP org, Azure management group) High Controls resource creation, IAM, billing; compromise at org level = unlimited blast radius Attacker creates shadow resources, exfiltrates data, destroys infrastructure
Logs and telemetry from PAW fleet Medium Only evidence of credential use, lateral movement, unusual admin activity Insider threat and external compromise become forensically invisible
Admin VLAN network topology Medium PAW VLAN segmentation is a key control; network map reveals attack paths Attacker can route around segmentation if topology is known

3. TRUST BOUNDARIES — Where does trust change?

3.1 High-level PAW Trust Boundary Architecture

The fundamental principle behind a PAW is the clean source principle: every admin account must authenticate from a device at least as trusted as the systems it manages. The PAW enforces this by making the device itself a trust boundary.

5. CONTROLS — Defense-in-Depth

5.1 Preventive Controls

Control Implementation Detail Enforcement Point Threat Mitigated Failure Mode / Bypass Risk
Device-based Conditional Access for admin accounts Entra ID / Okta CA policy: admin role requires deviceCompliance = true AND deviceTrustType = AzureADJoined AND authMethod = FIDO2. No exceptions. Per-account policy, not per-app. Entra ID Conditional Access engine — evaluated on every token issuance Admin auth from non-PAW device; stolen credentials used from attacker-controlled device MDM enrollment compromise; compliance misreporting — mitigated by hardware attestation
Windows Credential Guard (HVCI) Enable via Intune: DeviceGuard/EnableVirtualizationBasedSecurity = Enabled; LsaCfgFlags = 1 (Credential Guard). Requires UEFI Secure Boot + TPM 2.0. Verifiable via msinfo32 or Get-ComputerInfo. Windows kernel — LSASS process runs in VSM-isolated container LSASS credential dumping (Mimikatz, procdump, Task Manager); prevents NT hash and Kerberos TGT extraction from LSASS Pre-boot attacks on TPM; DMA attacks on older hardware without Kernel DMA Protection; requires HVCI-capable hardware
Protected Users security group for all admin accounts Add all Tier 0 and Tier 1 admin accounts to the Protected Users group in AD. This enforces: no NTLM auth, no Kerberos delegation, no DES/RC4 Kerberos, no credential caching, TGT lifetime capped at 4 hours. Active Directory KDC — enforced at authentication time Pass-the-hash (no NTLM); credential caching attacks; Kerberos downgrade; over-delegated service accounts Applications that require NTLM or delegation will break — audit before applying; must test thoroughly
Application allowlisting via WDAC Windows Defender Application Control policy in enforced mode. Allow: approved admin tools (RSAT, PowerShell, specific management consoles), signed by your internal CA or Microsoft. Deny all other executables, DLLs, scripts by default. Deploy via Intune MDM. Windows kernel (WDAC runs in VSM on HVCI-enabled systems) — cannot be bypassed by local admin Malware execution on PAW; browser execution on PAW; LOLBin abuse via unsigned paths; supply-chain malware Admins abusing allowed tools as LOLBAS; strict WDAC requires careful application inventory before deployment
Block browsers and email on PAW via WDAC + GPO WDAC policy: deny chrome.exe, msedge.exe, firefox.exe, iexplore.exe, outlook.exe, thunderbird.exe. GPO: disable Windows Store app installation. GPO: block PowerShell Download Cradles (AMSI-enforced). WDAC — kernel level; GPO — machine level Browser exploit compromising PAW during admin session; token/cookie theft; phishing landing pages on PAW Admins finding WDAC bypass via script interpreters — also add AppLocker as a second layer for script file type enforcement
PAW VLAN network segmentation Dedicated PAW VLAN (e.g., 10.0.0.0/26). Firewall policy: PAW VLAN → Admin targets only (specific management ports: TCP 22 SSH, TCP 443 HTTPS management, TCP 5986 WinRM, TCP 3389 RDP to jump only, TCP 88 Kerberos to DCs). Deny: PAW VLAN → Internet, PAW VLAN → user workstation VLANs, PAW VLAN → production application VLANs. DNS: PAW DNS resolvers resolve only internal names + admin portals. Network firewall — Layer 3 enforcement; NGFW with application inspection Internet access from PAW; lateral movement from compromised PAW to production workloads; NTLM relay across segments Firewall rule sprawl over time; require change review for any new PAW VLAN rule
Just-in-Time (JIT) role activation via PIM All Tier 0 and Tier 1 admin roles are eligible only, never permanent. Activation via Azure PIM or equivalent (CyberArk, BeyondTrust). Activation requires: business justification text, manager approval for Tier 0 roles, max duration 1–4 hours, role auto-expires. No standing admin access. PIM / PAM platform — enforced at directory level Standing privilege abuse; insider threat with always-on admin; dormant admin account abuse after credential theft PIM activation from non-PAW device if CA policy is not scoped to PIM activation endpoint; must verify CA covers PIM portal
Separate admin accounts (no dual-use accounts) Every admin has two accounts: jsmith@corp.com for daily use, jsmith-adm@corp.com (or in a separate admin forest) for admin tasks. Admin accounts: no mailbox, no browsing profile, no SPN, no delegation, no MX reachable. Admin accounts in Protected Users. Active Directory account design; admin forest (ESAE) for Tier 0 Credential theft from phishing/malware on daily-use account reaching admin privileges; admin account enumeration via email Over-complicated for Tier 2 admins; apply strictly to Tier 0/1; Tier 2 can use step-up MFA on existing account
Local Admin Password Solution (LAPS) for PAW local accounts Deploy Windows LAPS (v2, built into Windows Server 2019+). Unique 256-bit password per PAW, rotated every 24 hours, stored in AD with access controlled to PAW management team only. Active Directory — LAPS attribute on computer object Pass-the-hash via shared local admin credentials across PAW fleet; lateral movement between PAWs LAPS password accessible to compromised Tier 1 PAW management account — scope LAPS access to Tier 0 only
PAW provisioning from clean offline image PAWs provisioned offline from a golden image that has never been internet-connected. Image stored on internal golden-image server with content hash verification. Software updates applied in a controlled offline process before image deployment. Provisioning pipeline is air-gapped from production. Provisioning pipeline — isolated build environment Supply-chain compromise of admin tooling during provisioning; malicious pre-installed software Image drift if not re-enforced; quarterly re-image of all PAWs from refreshed golden image
FIDO2 hardware key for all admin authentication All admin accounts require FIDO2 hardware key (YubiKey 5 series or equivalent) for authentication. TOTP, SMS, and push-based MFA explicitly blocked for admin accounts via Conditional Access authentication strength policy. Entra ID CA — Authentication Strength policy: FIDO2 security key required for admin roles Phishing-resistant MFA requirement; TOTP bypass via AiTM proxy; SIM swapping Physical key loss — requires a documented, secure recovery procedure with identity verification

6.2 Detection Rules

Rule 1 — Admin Authentication from Non-Compliant Device

Alert when any Entra ID sign-in event for a user assigned to
any privileged role (Global Admin, Privileged Role Admin, Exchange Admin,
Tier 0 custom roles) has deviceDetail.isCompliant != true
OR conditionalAccessPolicies.result = failure within any 1-minute window.
  • Severity: Critical
  • MITRE: T1078.002
  • Source: Entra ID Sign-in Log
  • False positives: Near-zero if CA is properly scoped; legitimate exception = break-glass accounts (must be separately monitored)
  • Tuning: Zero-tolerance policy; any occurrence should be investigated; suppress only documented break-glass account usage with correlated ticket

Rule 2 — PAW Egress to Non-Admin Destination

Alert when any network flow originating from PAW VLAN CIDR (10.0.0.0/26)
has a destination IP not present in the approved admin management CIDR list
within any 1-minute window.
  • Severity: High
  • MITRE: T1071.001
  • Source: Firewall / NGFW flow log
  • False positives: Low after allowlist is tuned; initial noise from DNS lookups for admin SaaS portals
  • Tuning: Start in alert-only mode; build CIDR allowlist over 2 weeks; then enable auto-block

Rule 3 — PIM Role Activation with No Valid Change Ticket

Alert when PIM activation event for any Tier 0 or Tier 1 role
has justification text that does not match the regex
(CHG|INC|P[0-9])\d{6,8} within any event.
  • Severity: High
  • MITRE: T1098
  • Source: PIM Audit Log
  • False positives: Medium initially (admins not using ticket IDs); reduce via mandatory field format enforcement at activation form
  • Tuning: Add ticket ID API validation at PIM activation to auto-reject invalid IDs before they become alerts

Rule 4 — Browser Execution on PAW

Alert when EDR process creation event has processName in
[chrome.exe, msedge.exe, firefox.exe, iexplore.exe, MicrosoftEdge.exe]
on any device with MDM tag = PAW within any single event.
  • Severity: High
  • MITRE: T1185
  • Source: EDR telemetry
  • False positives: Low; WDAC should block execution — if this alert fires, WDAC was bypassed
  • Tuning: Zero-tolerance; WDAC bypass + browser execution is effectively a PAW compromise indicator

Rule 5 — LSASS Memory Access by Non-Allowlisted Process

Alert when Windows Security Event 10 (Process Access targeting lsass.exe)
is generated from a sourceProcess NOT in the approved security software allowlist
on any PAW-tagged device within any single event.
  • Severity: Critical
  • MITRE: T1003.001
  • Source: EDR / Windows Security Event Log
  • False positives: Low with accurate allowlist; maintain allowlist of EDR agent, AV scanner, specific approved tools
  • Tuning: Build allowlist from 2-week observe period; alert immediately on anything outside it

  1. Deploy EDR and log collection to PAWs in observe-only mode for 2 weeks before enabling blocking/alerting.
  2. Build the admin VLAN CIDR allowlist from observed legitimate flows before enabling firewall alerts.
  3. Baseline per-admin PAM session behavior (command count, target count, duration) before enabling anomaly alerts — use p90 as threshold, not a fixed number.
  4. Deploy PIM ticket validation in soft mode (warn but allow) for 2 weeks; move to hard block after 95% compliance.
  5. Enable Critical-severity alerts (non-compliant device auth, LSASS access) from day one — no tuning period for these.
  6. Avoid alert fatigue: PAW alerts should have a very low volume by design (controlled environment). If alert volume is high, it signals a policy configuration problem, not a tuning problem. Investigate the policy, not the alert.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and Policy-as-Code

At Google scale (10,000+ admins across global infrastructure), manual PAW management is impossible. Every PAW policy must be expressed, tested, and deployed as code.

  • PAW image as IaC: Golden PAW images defined in Packer templates stored in version control. Any change to the PAW image goes through code review, automated build, and canary deployment (10 PAWs → 50 PAWs → full fleet) before fleet-wide rollout.
  • WDAC policy as code: Application control policies expressed as XML, checked into git, reviewed, and deployed via Intune configuration profiles with automated policy verification tests.
  • Conditional Access policy as code: Entra ID CA policies defined as Terraform/Bicep templates. Policy changes go through PR review with automated test suite that validates policy logic against a set of synthetic sign-in scenarios before deployment.
  • PIM role assignments as IaC: Role eligibility assignments defined in Terraform; no manual assignment; every assignment change creates a PR; drift detection runs nightly.
  • Detection-as-code: All PAW detection rules in YAML/Sigma format checked into git; auto-deploy to SIEM; unit tests validate logic against synthetic log data before production deployment.
  • Continuous compliance scanning: Automated tool runs nightly against all PAWs; flags any configuration that has drifted from the golden state; auto-creates tickets for drift remediation.

ANSWER SUMMARY — 90-second version

The threat model that justifies a PAW program is straightforward: the cheapest path to your domain admin or cloud root credentials is not to attack your hardened infrastructure directly — it's to compromise the general-purpose laptop of the admin who manages that infrastructure, dump LSASS with Mimikatz, and laterally move with their credentials. A PAW breaks that chain by making the admin's device itself a trust boundary.

Given the budget constraint, I would design a tiered model. The fifteen or so Tier 0 admins who manage domain controllers, PKI, and cloud organization roots get physical dedicated PAWs that have never touched the internet, provisioned from an offline golden image. The next tier of server and cloud service admins gets cloud PAWs — Azure Virtual Desktop or GCP Cloud Workstations — which are cost-effective and can be provisioned on demand. The remaining helpdesk and workstation admins get a hardened managed device with PIM step-up, FIDO2, and Conditional Access enforcement, which is not a full PAW but significantly raises the bar.

The two controls I would not compromise on are: Windows Credential Guard with HVCI enabled, which protects LSASS from credential dumping even if the PAW OS is partially compromised; and device-based Conditional Access requiring PAW device compliance for every admin authentication token issuance, which means a stolen admin password alone is useless — the attacker also needs the PAW hardware. Layered on top is WDAC application allowlisting, which prevents browsers and email from running on the PAW, a PAW-dedicated VLAN with firewall rules that only allow admin management ports, and JIT role activation via PIM so there is no standing admin privilege even when admin credentials are valid.

For monitoring, my top signal is any admin authentication from a non-compliant device — that should never happen and is an immediate Critical alert. Secondary signals are browser execution on PAW, PAW VLAN egress to internet, and PIM activations outside business hours without a linked change ticket. The residual risk I am honest about is the Tier 2 admins on managed-but-not-fully-locked-down devices; they must be strictly scoped to low-blast-radius targets, and that scope boundary is as important as the device control itself.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What residual risk remains for admins who occasionally use PAWs for general browsing?

If WDAC policy is correctly enforced, browsers cannot execute on the PAW — so the risk is limited to WDAC bypass scenarios. However, if an admin circumvents WDAC (LOLBin-based browser launch, portable browser, or script interpreter gap), the PAW becomes a dual-purpose device: browser exploits can now reach a system with active admin credentials in memory, PIM tokens, and a session recording tool running with elevated rights. The entire threat model collapses because the clean source principle is violated. The defense-in-depth answer is that even if WDAC is bypassed, the PAW VLAN firewall still blocks internet access — so a browser can execute but cannot reach the internet. That second layer is why you need both WDAC AND network segmentation, not just one of them.

Follow-up 2: How do you handle a PAW where the admin also needs web-based management consoles (cloud portals, network management UI)?

This is a common gap. The answer is not to allow a general browser — it is to use a dedicated, isolated browser profile with zero extensions, pinned to only approved management FQDNs, with no password saving and no cross-site cookie sharing. On WDAC-enforced PAWs, this is implemented by having a custom-signed browser launcher application in the allowlist that invokes a locked-down Edge instance with specific --user-data-dir, --disable-extensions, and --host-rules arguments. The browser instance is allowed by WDAC only when launched by the signed launcher, not directly. Alternatively, for Tier 1 cloud admins, the cloud provider's browser-based console can be accessed through an ephemeral cloud PAW session that is destroyed after use — no browser state persists.

Follow-up 3: The budget was cut again — you can only implement one control from your entire PAW program. Which do you keep?

Device-based Conditional Access requiring PAW compliance for admin role authentication. This is the keystone control because it enforces the clean source principle at the identity plane — it makes the PAW enforcement real regardless of what happens at the endpoint layer. Without it, all the WDAC, Credential Guard, and VLAN segmentation are endpoint controls that an attacker can potentially work around; the credential remains usable from any device. With Conditional Access as the gate, a stolen admin credential is useless without the compliant PAW device. Every other control — WDAC, Credential Guard, VLAN, LAPS — makes the PAW itself harder to compromise, but Conditional Access is the control that enforces why the PAW exists at the authentication level. If I have only one control, it must be the one that makes stolen credentials non-replayable.

COMMON MISTAKES TO AVOID

Mistake Why It Is Weak Stronger Answer
Saying "deploy a jump server as the PAW solution" A jump server is a session broker, not a PAW. It centralizes access but does not solve credential theft from the admin's workstation — the admin's credentials are still typed on their general-purpose laptop to reach the jump server. The adversary steals them there, not from the jump server. "A jump server is a target, not a PAW. A PAW protects the credential at the source — the admin's workstation. The right answer combines a hardened PAW with a PAM tool that proxies sessions through the jump server for recording."
Treating PAW as a network control only (VLAN segmentation) Network segmentation prevents the PAW from being a lateral movement source, but it does not prevent credential theft from the PAW itself. An attacker who compromises the PAW while the admin is active still has valid credentials with which to authenticate from that same PAW (which is in the admin VLAN). "Network segmentation limits blast radius from a compromised PAW but does not prevent credential theft. Credential Guard, Protected Users group, FIDO2 enforcement, and WDAC are the controls that protect the credential on the PAW. Both layers are necessary."
Ignoring the budget constraint and proposing physical hardware for all 150 admins This is not answering the question asked. The reviewer specifically said hardware is constrained. Ignoring the constraint signals you do not operate in the real world. "With budget constraints, the answer is a tiered model: physical PAW only for Tier 0 (15 admins managing DCs and cloud org root); cloud PAWs via Azure VDI or GCP Cloud Workstations for Tier 1 (50 admins); hardened MDM device + PIM step-up for Tier 2 (85 admins). Allocate the strongest controls where the blast radius is highest."
Proposing TOTP-based MFA for PAW authentication TOTP is trivially bypassed by AiTM (adversary-in-the-middle) phishing proxies. An admin authenticating to a fake login page hands the attacker their TOTP code in real time. For privileged accounts accessing admin targets, the MFA must be phishing-resistant — FIDO2 only. "TOTP is not acceptable for admin accounts. FIDO2 hardware keys with a Conditional Access Authentication Strength policy that blocks all weaker MFA methods (TOTP, SMS, push) for privileged role authentication."
