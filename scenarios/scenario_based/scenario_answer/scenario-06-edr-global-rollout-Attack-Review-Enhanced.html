Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Deploying an EDR agent running as SYSTEM/root to 80,000 Windows, macOS, and Linux endpoints globally, with a cloud management console, automatic update pipeline, and SOC-driven response capability.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

The review focuses on attack mechanics, prerequisites, command-level purple-team validation examples, detection evidence, and control validation. Commands use placeholder values and are intended for an owned lab or approved internal purple-team exercise only. High-risk techniques are described at the technique level and paired with safe defensive validation commands.

2. Highest-Risk Design Decision

The single most consequential decision is whether the EDR agent auto-update pipeline enforces independent, pinned signature verification before applying any update — and whether deployment proceeds through staged canary rings with automated health gates.

An EDR agent running as SYSTEM/root on 80,000 endpoints is the highest-privilege code path in the entire enterprise. Without pinned binary signature verification, a single compromised vendor CDN or build pipeline produces fleet-wide arbitrary code execution at ring-3 (Windows user SYSTEM) before any detection alert can fire. The dangerous alternative — trusting the CDN TLS connection as the only authenticity check — leaves the entire fleet one CDN hijack or BGP hijack away from total compromise. The safe choice is: pin the vendor's code-signing certificate public key inside the agent, verify SHA-256 of every update package against a vendor-signed manifest, deploy to a 0.1% canary ring first, and enforce automated health gates (BSOD rate, crash rate, heartbeat continuity) before each subsequent ring.

6. Threat-by-Threat Technical Review

T1 — Supply-Chain Attack via Malicious EDR Update

STRIDE category: Tampering, Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Compromise the EDR vendor's build pipeline or CDN to deliver a malicious signed update that executes attacker code as SYSTEM/root on all 80,000 managed endpoints simultaneously.

How the attacker would exploit it

  • Prerequisite: Attacker gains access to the EDR vendor's build pipeline (CI/CD credentials, insider, or supply-chain compromise of a build dependency) — similar to the SolarWinds SUNBURST attack path.
  • Initial action: Attacker injects a malicious payload into the EDR agent build process or replaces a signed package on the vendor CDN after signing verification (CDN cache poisoning, BGP hijack, or compromised CDN credentials).
  • Core exploitation: The malicious update is signed with the vendor's legitimate code-signing certificate (because it was injected pre-signing or the signing key was stolen). The agent's signature verification passes. The agent applies the update and executes attacker-controlled code as SYSTEM/root. Tools: in-house implant compiled into agent update; SUNBURST-style DLL replacement; impacket for lateral movement from first compromised endpoint.
  • Escalation: With SYSTEM/root on 80,000 endpoints, attacker establishes C2, harvests credentials (Mimikatz, LaZagne), pivots to AD/domain controllers, and achieves domain dominance before any alert fires.
  • Impact realization: Full enterprise compromise: domain controller takeover, credential harvest, data exfiltration, or ransomware deployment across entire fleet.

Blue-team control review — what must stop the chain

  • Control 1: Pin vendor's code-signing certificate public key inside the agent (not just OS trust store) — agent rejects any update not signed by the pinned cert. GPO/MDM-enforced via Authenticode policy on Windows; macOS Gatekeeper with Team ID pinning; Linux IMA (Integrity Measurement Architecture) with pinned vendor key.
  • Control 2: Staged canary rollout: ring 0 = 0.1% (50 endpoints), health gate (BSOD rate, crash rate, heartbeat) must hold for 30 minutes before ring 1 proceeds. Automated halt and rollback if error rate exceeds 0.1%.
  • Control 3: Independent fleet health watchdog process outside EDR reports agent version and health to a separate SIEM pipeline — so a malicious update cannot suppress its own detection.
  • Control 4: Monitor CDN delivery log for SHA-256 hash mismatch between delivered package and vendor-published manifest. Alert on any delta immediately.
  • Control 5: Vendor contractual requirement: SOC 2 Type II attestation, annual penetration test of build pipeline, 4-hour breach notification SLA, and separate signing key escrow.

Evidence and telemetry to validate controls

  • EDR backend agent version telemetry: agent_id, agent_version, policy_version, update_ts — spike in unexpected version or version rollback is a canary signal.
  • CDN delivery log: signature_status = FAIL events — any non-zero count is a P1 alert.
  • Windows Event 7036 (service status changed), 7045 (new service installed) — baseline-deviant entries from EDR service path.
  • Canary ring health gate log: BSOD rate, crash rate, agent heartbeat continuity per ring per OS.

T2 — EDR Management Console Takeover

STRIDE category: Spoofing, Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Compromise an EDR admin account to gain console access, enabling global agent disablement, wildcard exclusion creation, and fleet-wide policy manipulation.

How the attacker would exploit it

  • Prerequisite: Attacker identifies EDR console admin email addresses via LinkedIn, GitHub (code commits referencing EDR console), or phishing reconnaissance (hunter.io, OSINT).
  • Initial action: Spear-phishing email targeting SOC admin with an AiTM (Adversary-in-the-Middle) proxy (evilginx2, Modlishka) to capture both credentials and OIDC session cookies, bypassing TOTP or push-notification MFA.
  • Core exploitation: Attacker replays captured session cookie to the EDR console. With TOTP or push MFA, AiTM succeeds; FIDO2 hardware keys stop AiTM because the key challenge is domain-bound. If FIDO2 is not enforced, attacker achieves console access with admin role.
  • Escalation: Once in console, attacker creates wildcard exclusion (C:\Users\Public\*), disables agents on high-value targets, or pushes a "do not alert" policy for specific process hashes (malware staging path).
  • Impact realization: EDR blind spot created for malware staging; fleet agents disabled before ransomware payload deployment; SOC loses visibility across enterprise.

Blue-team control review — what must stop the chain

  • Control 1: Enforce FIDO2 hardware security key (YubiKey 5 or equivalent) for all console admin accounts. Disable TOTP, SMS, and push notification MFA as fallback methods. IdP Conditional Access policy requires amr=fido2 claim in OIDC assertion — evilginx2 cannot replicate this because the challenge is domain-bound to the IdP domain.
  • Control 2: PAW (Privileged Access Workstation) requirement: console access only from managed, compliance-verified device. IdP Conditional Access requires device certificate (Entra ID/Okta device trust). Non-PAW device = login blocked.
  • Control 3: IP binding: EDR console accessible only from management VPN or PAW subnet. Any login from non-approved IP blocked before MFA challenge.
  • Control 4: IdP risk-based Conditional Access: risk score > 60 (new country, new device, anomalous login hour) forces step-up authentication or blocks login outright.
  • Control 5: Console session timeout: 8-hour maximum session with re-authentication required for any destructive action (exclusion create, agent disable, mass policy change).

Evidence and telemetry to validate controls

  • IdP sign-in log: amr field — any admin login without amr=fido2 is a P1 alert.
  • IdP sign-in log: country, risk_score, device_id — new country or non-compliant device triggers alert.
  • EDR console audit log: action=exclusion.create or action=agent.disable correlated with session_id from IdP login — within 10 minutes of login from anomalous source.
  • Windows Event 4624 + 4648 on PAW: PAW process creation events around console browser process — validates correct workstation is in use.

T3 — EDR Kernel Driver Exploitation (BYOVD / CVE)

STRIDE category: Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Exploit a vulnerability in the EDR's kernel driver (ring-0) to achieve kernel-mode code execution, disable EDR from within, install a rootkit, or crash the endpoint.

How the attacker would exploit it

  • Prerequisite: Attacker has local code execution on the target endpoint (user-land compromise via phishing, RCE, or malicious document). Attacker identifies the EDR kernel driver name and version via driverquery.exe or Get-WindowsDriver.
  • Initial action: For BYOVD (Bring Your Own Vulnerable Driver): attacker drops a known-vulnerable, legitimately signed kernel driver (e.g., a vulnerable graphics or storage driver from the LOLDrivers project) and loads it to gain ring-0 access. For direct CVE exploitation: attacker uses a published CVE in the EDR driver itself (e.g., type confusion, use-after-free in IOCTL handler) to escalate to kernel mode.
  • Core exploitation: From ring-0, attacker uses kernel primitives to: (a) unload the EDR kernel driver by patching its dispatch table, (b) blind the EDR by removing its kernel callbacks (PsSetLoadImageNotifyRoutine, ObRegisterCallbacks), or (c) install a kernel-mode rootkit that hides processes, files, and network connections from all EDR inspection.
  • Escalation: With EDR blind and ring-0 access, attacker installs persistent kernel rootkit, dumps LSASS from kernel space (bypassing EDR user-mode protection), and establishes persistent C2.
  • Impact realization: Host OS fully compromised at kernel level; forensic artifacts can be hidden; EDR and OS-level logging may be suppressed; endpoint unusable without reimaging.

Blue-team control review — what must stop the chain

  • Control 1: Windows Driver Blocklist (WDAC/Code Integrity) — enforce Microsoft Vulnerable Driver Blocklist via Windows Defender Application Control policy. This blocks known-vulnerable BYOVD drivers by hash. Update monthly from Microsoft MSRC and LOLDrivers feeds.
  • Control 2: Hypervisor-Protected Code Integrity (HVCI) enabled — prevents kernel memory page permissions being changed by attacker code; severely limits BYOVD technique effectiveness.
  • Control 3: EDR vendor patch SLA: critical kernel driver CVE must be patched and deployed to 100% of fleet within 24 hours of vendor patch availability. Automate via emergency ring (skip canary, but with immediate rollback capability).
  • Control 4: Monitor Windows Event 7045 (new service installed) for any driver not on the approved driver hash allowlist. Alert immediately.
  • Control 5: Detect EDR driver unload: EDR should emit a tamper event on driver unload; independently, Windows Event 7036 (service stopped) for the EDR driver service name should trigger a P1 alert.

Evidence and telemetry to validate controls

  • Windows Event 7045: new service/driver name and binary hash — compare against LOLDrivers database and approved driver allowlist.
  • Windows Event 7036: EDR service name transition to stopped state — P1 alert unconditionally.
  • Windows Event 4657 (registry value modified): WDAC policy registry keys — detect policy tampering.
  • EDR kernel callback audit log (if vendor supports): list of registered ObRegisterCallbacks — alert on removal.

T4 — Unauthorized or Malicious Exclusion Creation

STRIDE category: Tampering, Repudiation
Risk: Medium likelihood, High impact
Attacker objective: Create an exclusion zone in the EDR that covers a malware staging path, persistence directory, or lateral movement tool — making EDR invisible to subsequent attacker activity in that zone.

How the attacker would exploit it

  • Prerequisite: Attacker has either (a) compromised an EDR admin or security engineer account, (b) an insider with EDR-Policy-Admin role, or (c) social-engineered the exclusion approval process (e.g., claiming a build server requires an exclusion for a specific directory).
  • Initial action: Attacker researches EDR vendor-specific default exclusion paths documented in vendor support articles (e.g., CrowdStrike recommends excluding certain AV scan paths). Attacker submits exclusion request via ITSM using a convincing business justification referencing a build pipeline.
  • Core exploitation: If exclusion workflow lacks scope validation, attacker creates a wildcard exclusion (C:\ProgramData\Temp\* or /tmp/build/*) covering the intended malware staging path. Alternatively, if attacker has direct console access, they create the exclusion programmatically via the EDR API.
  • Escalation: Attacker drops malware into the excluded path. EDR does not alert on file creation, process launch, network connection, or memory injection from that path. C2 beacon, credential harvester, or lateral movement tools operate in the blind spot.
  • Impact realization: Long-term persistent access with no EDR detection; attacker operates freely within exclusion zone for months (average dwell time in environments with exclusion abuse is significantly longer than baseline).

Blue-team control review — what must stop the chain

  • Control 1: Exclusion scope validation at console API layer — reject any exclusion matching root-drive wildcards, temp directories, or all_processes scope. Block the API call before the exclusion is created; generate a P1 alert.
  • Control 2: ITSM-ticket gating: exclusion creation via API requires a valid approved ITSM ticket ID. SIEM correlates exclusion creation event against ITSM API in real time; exclusion with no matching approved ticket is auto-suspended within 5 minutes.
  • Control 3: SHA-256 hash-bound exclusions only (no directory wildcards) — policy enforced at console validation layer.
  • Control 4: Second-approver requirement for any exclusion >14 days or covering a server — EDR-Policy-Admin role cannot approve; requires security team co-approval.
  • Control 5: Auto-revocation at expiry (default 30 days, max 90 days) — no permanent exclusions without quarterly documented risk acceptance.

Evidence and telemetry to validate controls

  • EDR console audit log: exclusion.create event with exclusion_path_pattern, created_by, change_ticket_ref, expiry_ts.
  • ITSM API query: ticket_status for change_ticket_ref — mismatch triggers auto-suspend.
  • EDR console audit log: exclusion.export events — baseline and alert on unusual export volume.
  • SIEM rule: wildcard pattern regex on exclusion_path_pattern field — zero false positives in production.

T5 — EDR Management API Abuse via Stolen API Key

STRIDE category: Spoofing, Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Use a stolen or leaked EDR API key to programmatically disable agents, create exclusions, or export the exclusion list — at scale and without interactive console authentication.

How the attacker would exploit it

  • Prerequisite: Attacker discovers an EDR API key in a public GitHub repository, CI/CD pipeline log, environment variable dump, or Slack message (using TruffleHog, GitLeaks, or SIEM log search for bearer token patterns).
  • Initial action: Attacker tests the key against the EDR API: GET /api/v1/agents to enumerate all managed endpoints and verify the key is valid and its permission scope.
  • Core exploitation: Using the policy-write or response-action scoped key (or an over-privileged key), attacker calls POST /api/v1/agents/bulk-disable or POST /api/v1/exclusions from an attacker-controlled IP. If the key is not IP-bound, this succeeds from anywhere.
  • Escalation: Attacker disables agents on high-value targets (DCs, file servers, backup servers) before deploying ransomware via a separate access vector. Alternatively, creates exclusions covering ransomware staging directories.
  • Impact realization: Targeted agent disablement removes EDR visibility from critical servers before detonation; ransomware or data exfiltration tool executes without any EDR alert.

Blue-team control review — what must stop the chain

  • Control 1: Bind all EDR API keys to an approved source IP CIDR at the EDR API gateway. Reject any API call from a non-approved IP, regardless of key validity.
  • Control 2: Scope API keys with minimum necessary permissions: read-only, policy-write (no disable), response-action (isolate only, no bulk disable). No key has combined write-policy + disable-agent without dual approval.
  • Control 3: Store API keys exclusively in HashiCorp Vault (or cloud KMS equivalent); rotate every 90 days; never in environment variables, source code, or CI/CD logs. Deploy TruffleHog / GitLeaks in CI/CD to detect accidental commits.
  • Control 4: Alert on any API call to exclusion.create or agent.disable from an IP not in the management CIDR.
  • Control 5: Automatic key rotation on leak detection: integrate Vault with GitHub Advanced Security secret scanning; any detected key triggers immediate rotation and audit of all calls made with the compromised key.

Evidence and telemetry to validate controls

  • EDR API access log: api_key_id, caller_ip, action, response_code, timestamp — any caller_ip outside management CIDR is a P1 alert.
  • Secrets scanning CI/CD log: GitLeaks scan results — alert on any EDR token pattern in committed code.
  • Vault audit log: key access log — unexpected access to EDR API key path is an anomaly.
  • EDR console audit log: correlate API key ID to associated calls — enumerate blast radius on key compromise.

T6 — Agent-to-Backend Command Channel MitM

STRIDE category: Tampering, Spoofing
Risk: Low likelihood, High impact
Attacker objective: Intercept or inject commands on the TLS channel between EDR agents and the cloud backend to suppress telemetry, push false policy updates, or replay stale commands.

How the attacker would exploit it

  • Prerequisite: Attacker has a network-level position between the endpoint and the EDR cloud backend — achieved via rogue Wi-Fi, ARP spoofing on the same subnet, BGP hijack affecting the EDR CDN/API IP ranges, or a compromised corporate TLS inspection proxy.
  • Initial action: Attacker installs a rogue TLS inspection certificate in the enterprise's trusted root store (via GPO abuse or physical access) or exploits a misconfigured corporate SSL inspection proxy that inadvertently decrypts EDR traffic.
  • Core exploitation: With TLS decrypted, attacker injects a false command to the agent: {"command": "set_policy", "silent_mode": true} — suppressing telemetry upload without triggering a visible alert. Or attacker replays a stale isolate command to take an endpoint offline.
  • Escalation: With telemetry suppressed on targeted endpoints, attacker deploys payload and moves laterally without EDR visibility. Replayed isolation commands create operational disruption as a distraction.
  • Impact realization: Targeted telemetry blackout enables undetected lateral movement; Denial of Service via replay isolation of critical servers.

Blue-team control review — what must stop the chain

  • Control 1: Certificate pinning in EDR agent — agent pins the vendor's TLS certificate or public key fingerprint. Any TLS inspection proxy (even with a trusted enterprise CA) fails the pinned cert check and the agent refuses the connection, alerting on the failed connection.
  • Control 2: Require TLS 1.3 only for agent-to-backend communication — eliminates downgrade attacks and cipher negotiation weaknesses.
  • Control 3: Signed command envelopes — backend signs each command payload with a private key; agent verifies the signature before executing any command. A replay or injected command without a valid fresh signature is rejected.
  • Control 4: Heartbeat sequence numbers — agent and backend maintain monotonically increasing sequence numbers; replay of an old command is detected by sequence number mismatch.
  • Control 5: Alert on EDR agent connection failure rate spike — agents that cannot connect to backend generate a heartbeat gap alert.

Evidence and telemetry to validate controls

  • EDR agent connection log: tls_handshake_failure, cert_pin_mismatch events — any non-zero count is a P1 alert.
  • EDR backend command integrity log: signature_verification_failure on received command — alert immediately.
  • Network flow log: unexpected intermediate IPs in agent → EDR backend traffic path — BGP anomaly detection.
  • EDR agent heartbeat gap log: agents going silent in a specific subnet simultaneously (possible ARP poisoning scope).

T7 — Agent Kill / Telemetry Gap Before Payload Deployment

STRIDE category: Tampering, Repudiation
Risk: High likelihood, High impact
Attacker objective: Kill the EDR agent process or service before deploying a ransomware or destructive payload — eliminating the primary detection layer for the attack execution phase.

How the attacker would exploit it

  • Prerequisite: Attacker has local admin or SYSTEM privilege on the target endpoint (achieved via initial access + local privilege escalation).
  • Initial action: Attacker enumerates EDR service name via sc.exe query or Get-Service and EDR process name via tasklist. Researches vendor-specific service names and driver names (publicly available).
  • Core exploitation: Attacker attempts to stop the EDR service: sc.exe stop CrowdStrikeFalconSensor or net stop SentinelOne. If tamper protection is enabled, this fails. Attacker then escalates to: (a) BYOVD to kill the driver from ring-0, (b) use a vendor-specific uninstall token theft if token is cached on disk, or (c) exploit a tamper-protection bypass CVE.
  • Escalation: Once EDR is silenced, attacker deploys ransomware payload from a pre-staged path (possibly an exclusion zone). With no EDR telemetry, the encryption begins before any alert fires.
  • Impact realization: Ransomware encrypts file servers and domain controllers. Forensic gap during EDR-silent period limits post-incident reconstruction.

Blue-team control review — what must stop the chain

  • Control 1: Enable EDR tamper protection as mandatory policy on all agents — prevents service stop, process kill, and uninstall without management console authorization token. Enforce enrollment check: agents without tamper protection enabled are flagged as non-compliant.
  • Control 2: Independent OS-level log forwarding (separate from EDR agent): Windows Event Log forwarder (WEF to SIEM) and/or Sysmon must run as an independent service. If EDR is killed, this secondary telemetry layer survives.
  • Control 3: Windows Event 7036 (service status changed to stopped) for the EDR service name is forwarded to SIEM via WEF. Alert fires immediately, independently of EDR.
  • Control 4: Sysmon Event 1 (process create), 3 (network connect), 11 (file create) for known ransomware behavioral patterns (rapid file renames, shadow copy deletion) — detects attack even after EDR is killed.
  • Control 5: SOAR playbook: on EDR service stop event, automatically check tamper protection status on endpoint and attempt remote restart via management API. If restart fails, auto-isolate the endpoint network.

Evidence and telemetry to validate controls

  • Windows Event 7036 (service control manager): ServiceName = EDR service name, state = stopped — P1 alert, no false positives in production.
  • EDR tamper protection log: tamper_protection_bypass_attempt — critical alert even if attempt fails.
  • Sysmon Event 1: vssadmin.exe delete shadows or wbadmin.exe delete catalog — classic ransomware pre-encryption step visible after EDR kill.
  • WEF forwarding health: if Windows Event Log forwarder itself stops, this is a secondary alert channel.

T8 — Compromised SOC Analyst Account with Console Access

STRIDE category: Spoofing, Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Use a compromised SOC analyst account to access the EDR management console, create exclusions, suppress alerts, or export investigation data — leveraging the analyst's legitimate role to hide attacker activity.

How the attacker would exploit it

  • Prerequisite: Attacker compromises a SOC analyst's endpoint (via phishing, malicious document, or credential stuffing against a non-MFA-protected corporate email).
  • Initial action: With access to the analyst's machine, attacker extracts browser session cookies or saved credentials for the EDR console using SharpDPAPI, Mimikatz browser credential extraction, or cookie extraction tools.
  • Core exploitation: Attacker replays the session cookie or uses the analyst's saved password to access the EDR console (if the analyst was not using FIDO2 from their PAW, session can be replayed from a different machine). With SOC-Analyst role, attacker can view all endpoint telemetry — useful for understanding what the SOC knows about an active intrusion — and can create investigations that suppress or close alerts.
  • Escalation: If the analyst has IR-Engineer role, attacker can kill processes or remove files on endpoints from the console. If console RBAC allows investigation closure, attacker closes active alerts for their persistence paths.

  • Control 4: Sensitive action re-authentication: investigation closure, alert suppression, and exclusion creation require step-up re-authentication (FIDO2 re-prompt) even within an active session.

  • IdP session log: concurrent session alert for same user from different devices.

Risk: Medium likelihood, Medium impact

How the attacker would exploit it

T11 — Tamper-Protection Bypass by Local Administrator

STRIDE category: Tampering, Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Bypass EDR tamper-protection mechanisms using local SYSTEM or administrator privileges to uninstall, disable, or neuter the EDR agent on a specific endpoint.

How the attacker would exploit it

  • Prerequisite: Attacker has local administrator or SYSTEM-level access on the target endpoint (via initial access, LPE exploit, or pass-the-hash lateral movement with admin credentials).

  • Control 4: EDR tamper-protection violation events are forwarded to SIEM via a separate channel (not through the EDR itself, since the agent may be targeted). Use OS-native log forwarding.

  • EDR console: tamper_protection_status field per agent — any disabled or unknown triggers auto-remediation and P2 alert.
  • Windows Event 4657: registry value modification on HKLM\SYSTEM\CurrentControlSet\Services\<edr-service> — P1 alert.

T12 — Faulty Vendor Update Causing Fleet-Wide Crash

STRIDE category: Denial of Service
Risk: Low likelihood, Critical impact
Attacker objective (non-malicious variant): A non-malicious but faulty EDR vendor update (content file, configuration push, or agent binary) causes widespread kernel panic (BSOD) or agent crash across the fleet before canary health gates fire — mirroring the CrowdStrike July 2024 incident.

How the attacker would exploit it

  • Prerequisite: No attacker required — this is a vendor quality failure. For malicious variant: attacker with access to vendor's content update pipeline (lower barrier than binary signing access) pushes a malformed content file.
  • Initial action: Vendor pushes a content update (detection rule file, behavioral policy, or configuration file) that contains a parsing error, out-of-bounds memory access, or null pointer dereference in the EDR kernel driver's content parsing code path.
  • Core exploitation: When the EDR kernel driver parses the malformed content file, it triggers a kernel-mode exception (BSOD / kernel panic on Windows/Linux). Because the update is delivered simultaneously to all endpoints in the same ring, thousands of endpoints BSOD within minutes.
  • Escalation: Affected endpoints enter boot loop or require manual intervention (WinRE/safe mode). Remote remediation is impossible once the OS cannot boot. IT staff must touch each affected machine.
  • Impact realization: Thousands of endpoints offline simultaneously; hospital systems, payment terminals, or critical infrastructure unavailable; IT staff overwhelmed with physical remediation; estimated recovery time: days for large fleets (as seen in CrowdStrike incident).

Blue-team control review — what must stop the chain

  • Control 1: Staged canary rollout with automated health gates is the primary control — ring 0 = 0.1% (50 endpoints), monitor BSOD rate and crash rate for 30 minutes before ring 1. Automated halt if error rate > 0.1%.
  • Control 2: Separate rings per OS type — Linux ring 0 failures do not cascade to Windows ring 0. OS-segmented canary rings.
  • Control 3: Automatic rollback: if health gate fails, rollback to last-known-good content version automatically within 10 minutes. Rollback itself is canary-staged (roll back ring 4 first).
  • Control 4: Independent fleet health watchdog (outside EDR) monitors endpoint availability via ICMP/WMI heartbeat. Detects mass unavailability even before EDR backend notices.
  • Control 5: Contractual controls: vendor must provide a content update rollback API; vendor must maintain a clear channel emergency notification within 15 minutes of detecting a faulty update.

Evidence and telemetry to validate controls

  • Canary ring health gate log: BSOD rate, crash dump count, heartbeat failure rate per ring per OS.
  • Independent fleet health watchdog: endpoint reachability failure count per region per 5-minute window.
  • Windows Error Reporting (WER): BugcheckCode field correlated with EDR driver name in crash dump — identifies driver as crash source.
  • EDR backend delivery log: content update version correlated with BSOD spike timestamp.

T13 — FIDO2 Hardware Key Supply-Chain Compromise

STRIDE category: Spoofing, Elevation of Privilege
Risk: Very Low likelihood, Critical impact
Attacker objective: Compromise a batch of FIDO2 hardware security keys before they reach EDR admins — through a firmware backdoor or insecure attestation — allowing the attacker to clone or predict key material for EDR console authentication.

How the attacker would exploit it

  • Prerequisite: Nation-state-level attacker with access to a hardware security key manufacturer's supply chain, or ability to intercept keys in transit and replace firmware. This is an advanced supply-chain threat similar to the NSA ANT catalog hardware implant operations.
  • Initial action: Attacker compromises a batch of YubiKey or similar FIDO2 keys destined for a high-value organization. Inserts firmware backdoor that makes the ECDSA private key recoverable (predictable entropy) or creates a side-channel leakage path (similar to the Infineon RSA key generation flaw, CVE-2017-15361).
  • Core exploitation: When EDR admins enroll the compromised FIDO2 keys, the attacker who knows the backdoor can reconstruct the private key material. Using the cloned key material, attacker authenticates to the EDR console with a valid FIDO2 assertion, bypassing all authentication controls.
  • Escalation: Attacker authenticated with full admin FIDO2 credentials creates wildcard exclusions, disables agents on high-value targets, or maintains persistent access to the management console.
  • Impact realization: Authentication control — the last line of defense for console access — is silently bypassed. All console security assumptions are invalid.

Blue-team control review — what must stop the chain

  • Control 1: Verify FIDO2 attestation at enrollment: check each key's attestation certificate against the FIDO Alliance Metadata Service (FIDO MDS3). Reject keys with revoked, expired, or unrecognized attestation certificates.
  • Control 2: Use multiple hardware key vendors for EDR admin accounts — a supply-chain compromise of one vendor does not affect keys from a different vendor.
  • Control 3: Monitor for FIDO Alliance security advisories on enrolled key models. If a key model is flagged, immediately force re-enrollment with a different key model.
  • Control 4: Behavioral anomaly detection: even with valid FIDO2 authentication, alert on admin logins from new geographies, new devices, or anomalous hours — a key clone may be used from an attacker location.
  • Control 5: Periodic FIDO2 key firmware audit: check enrolled key firmware versions against vendor security advisories every 90 days.

Evidence and telemetry to validate controls

  • FIDO2 enrollment log: attestation_verified = true/false per key enrollment — any false is a blocking error.
  • FIDO Alliance MDS3 API: query enrolled key AAGUID (Authenticator Attestation GUID) against MDS3 revocation list — automated check during enrollment and weekly re-check.
  • IdP sign-in log: login from unusual geography with valid FIDO2 amr claim — triggers alert even though MFA succeeded.

T14 — EDR Vendor Cloud Backend Breach

STRIDE category: Tampering, Information Disclosure, Denial of Service
Risk: Very Low likelihood, Critical impact
Attacker objective: Compromise the EDR vendor's cloud backend infrastructure to gain the ability to push malicious commands to all managed agents, suppress telemetry globally, or exfiltrate telemetry data from all 80,000 managed endpoints.

How the attacker would exploit it

  • Prerequisite: Nation-state-level attacker with capability to compromise a major SaaS vendor's cloud infrastructure — via phishing of vendor engineers, exploiting a vendor service vulnerability, or supply-chain attack on vendor's cloud infrastructure dependencies.
  • Initial action: Attacker gains access to the EDR vendor's cloud backend (e.g., compromises a vendor engineer's account, exploits a vulnerability in the vendor's agent management API, or compromises a vendor cloud service dependency).
  • Core exploitation: From inside the vendor's backend, attacker can: (a) push a "silent mode" policy to all agents globally suppressing all alert uploads, (b) inject malicious configuration targeting specific customer tenants, or (c) access telemetry data from all managed endpoints (comprehensive enterprise intelligence collection).
  • Escalation: With global telemetry suppression, attacker's own intrusion operations across the customer's network are invisible. With access to all endpoint telemetry, attacker understands the full security posture of all customers.
  • Impact realization: Global telemetry blackout; all customers using the same EDR vendor simultaneously lose EDR visibility; nation-state intelligence collection across thousands of enterprises.

Blue-team control review — what must stop the chain

  • Control 1: Contractual controls: require vendor breach notification SLA (4 hours for suspected breach, 24 hours for confirmed breach). Annual penetration test report and SOC 2 Type II attestation shared with customer.
  • Control 2: Monitor EDR agent heartbeat and telemetry volume independently — if telemetry upload rate drops significantly (>50% reduction across fleet in 10 minutes), this is a vendor-side suppression indicator.
  • Control 3: Maintain an independent secondary detection layer (Sysmon + OS-native log forwarding) that does not route through the EDR vendor's cloud. If EDR vendor backend is compromised, this secondary layer is unaffected.
  • Control 4: Evaluate multi-vendor EDR strategy for critical server tier — if primary EDR is vendor X, deploy vendor Y on Tier-1 assets (DCs, domain-joined servers). Vendor X backend breach does not blind vendor Y.
  • Control 5: Customer tenant isolation: ensure EDR management API uses per-tenant authentication and that vendor's multi-tenant architecture prevents cross-tenant telemetry access. Validate in vendor penetration test scope.

Evidence and telemetry to validate controls

  • EDR backend telemetry volume dashboard: events-per-minute across all endpoints — sharp unexplained drop is a vendor-side issue indicator.
  • Independent Sysmon + WEF telemetry: compare event volume from Sysmon with EDR event volume — if EDR drops but Sysmon stays constant, vendor backend issue confirmed.
  • Vendor security advisory feed: subscribe to vendor's security bulletin RSS and CVE notification service; alert on any critical advisory within 1 hour.
