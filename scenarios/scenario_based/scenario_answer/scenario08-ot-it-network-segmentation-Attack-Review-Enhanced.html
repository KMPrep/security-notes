OT/IT Network Segmentation — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario08-ot-it-network-segmentation.md

Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Connect a manufacturing plant OT network (PLCs, SCADA, HMIs) to the corporate IT network for remote monitoring with bidirectional data flows through a Purdue Model segmentation — while nation-state APTs, ransomware operators, and malicious insiders target the IT/OT boundary.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

Commands use placeholder values and are intended for an owned lab or approved internal purple-team exercise only. High-risk OT techniques are described at the technique level with safe validation counterparts only — destructive PLC manipulation is not reproduced.

2. Highest-Risk Design Decision

The single most consequential design decision is whether the OT historian (Level 3.5 DMZ) is allowed outbound IP connectivity to the corporate IT network, and whether that path is governed by strict protocol allow-listing or permissive firewall rules.

The historian is the architectural linchpin: it sits in the DMZ with allowed read paths from SCADA (Level 3) and allowed push paths to ERP (Level 4). An attacker who compromises the historian has a network-layer path in both directions — toward SCADA and PLCs, and toward Active Directory and the corporate IT backbone. The safe design uses a hardware data diode for the OT-to-historian read path and an API gateway with cryptographic input validation for the MES write-back path. The dangerous alternative — a permissive bidirectional firewall rule between IT and the historian — means a single phished IT employee creates a straight lateral-movement path to Level 3 OT assets.

6. Threat-by-Threat Technical Review

T1 — Spear-Phishing IT Employee for Lateral Movement to OT

STRIDE category: Spoofing, Tampering
Risk: High likelihood, Critical impact
Attacker objective: Compromise an IT employee's workstation via spear-phishing, dump credentials with Mimikatz, then use those credentials to laterally move through the IT network to reach the OT DMZ historian — using the historian as a stepping stone into the Level 3 SCADA network.

How the attacker would exploit it

  • Prerequisite: Attacker has internet access and OSINT data on IT employees (LinkedIn, email conventions) to craft a convincing spear-phish.
  • Step 1: Targeted email with malicious Office macro or HTML smuggling attachment; victim opens document on IT workstation — Cobalt Strike beacon or similar C2 implant is established.
  • Step 2: Attacker runs Invoke-Mimikatz or uses LSASS dump via procdump.exe -ma lsass.exe to extract NTLM hashes and Kerberos tickets from the IT workstation.
  • Step 3: Uses stolen credentials with pass-the-hash or Kerberos pass-the-ticket via Rubeus.exe ptt /ticket:<base64-ticket> to authenticate to adjacent IT hosts.
  • Step 4: Identifies the OT historian IP via internal DNS lookup (nslookup pi-server.corp.internal) or network scanning of the DMZ subnet; establishes connection on the allowed firewall path (typically TCP 5450 for PI, or SMB/RDP to the historian host).
  • Step 5: From the historian, exploits the allowed OPC-UA read path toward SCADA, pivoting deeper into Level 3 OT systems.

Blue-team control review — what must stop the chain

  • Control 1: Microsoft Defender for Endpoint EDR on all IT workstations — alerts on LSASS access by non-OS processes (Event ID 10 in MDE: LSASS access with CallTrace pointing to unbacked memory), Cobalt Strike beacon injection patterns.
  • Control 2: Firewall deny-all between IT Level 4 and DMZ Level 3.5 with explicit allow-list — historian should only be reachable from approved IT analytics hosts on specific ports (PI TCP 5450, PI-Web-API TCP 443); any IT workstation reaching historian is a critical alert.
  • Control 3: Active Directory tiering: OT historian service account must not be an IT domain-joined service — use a separate OT Active Directory forest or standalone local authentication to prevent credential reuse across the IT/OT boundary.
  • Control 4: Claroty/Dragos: new IT host connecting to historian on any port — IT-Host-Connecting-To-OT detection rule fires within 1 minute.
  • Control 5: Network segmentation: workstations in the IT employee population cannot reach the DMZ subnet directly — only jump servers and analytics platforms in IT are in the historian firewall allow-list.

Evidence and telemetry to validate controls

  • MDE/EDR: Event ID 10 LSASS open process with GrantedAccess = 0x1010 or 0x1038 (credential dump access mask).
  • Active Directory Security Event 4769: Kerberos TGS request for historian service from non-approved source IP.
  • DMZ Firewall syslog: action=deny, src_zone=IT, dst_ip=<historian-ip>, dst_port=5450 — confirms deny-all default is working; action=allow on this flow from an unexpected source is the critical alert.
  • Claroty asset intelligence: new IP seen communicating with historian for the first time — asset_discovery_event, src_ip, dst_ip, protocol.

T2 — Vendor VPN with Shared or Reused Credentials

STRIDE category: Spoofing
Risk: High likelihood, Critical impact
Attacker objective: Use stolen, shared, or reused vendor VPN credentials to access the OT jump server in the DMZ — then pivot to SCADA or HMI systems under the guise of a legitimate maintenance session.

How the attacker would exploit it

  • Prerequisite: Attacker obtains vendor credentials via phishing the vendor, credential reuse from a breach database, or purchasing on the dark web. Many OT vendor accounts use shared team credentials.
  • Step 1: Authenticate to the remote access VPN or ZTNA gateway using the stolen credentials — if MFA is only TOTP and not FIDO2, AiTM phishing can bypass it.
  • Step 2: Lands on the jump server in the DMZ — if PAM (CyberArk) is not enforcing session proxying, attacker may have direct RDP or SSH to OT systems.
  • Step 3: From the jump server, initiates connection to SCADA server or HMI — connection is authorized by PAM policy for the vendor account.
  • Step 4: Executes unauthorized commands, uploads malicious configuration files, or modifies PLC parameters — all inside an ostensibly legitimate PAM session.
  • Step 5: Impact: attacker has authorized-looking access to OT systems; CyberArk session recording is the only forensic trail; if session recording is incomplete, the attack may not be discovered until physical impact.

Blue-team control review — what must stop the chain

  • Control 1: ZTNA vendor access gateway requiring FIDO2 hardware token + device certificate — shared/stolen passwords alone cannot authenticate; AiTM cannot relay FIDO2 origin-bound assertion.
  • Control 2: CyberArk Privileged Access Manager: all vendor sessions proxied through PAM — vendor never has direct credentials for OT systems; PAM enforces time-limited checkout with automatic rotation after session.
  • Control 3: Maintenance window enforcement: PAM policy restricts vendor account activation to pre-approved time windows in ServiceNow; any access request outside window requires break-glass approval with CISO notification.
  • Control 4: Session behavioral analytics on CyberArk recordings: keystroke timing, command sequences, and screen content analyzed; deviation from approved maintenance script flags the session for SOC review.
  • Control 5: Vendor device posture check via ZTNA: vendor laptop must have current AV, no unauthorized software, OS within patch compliance window — known-malware-infected vendor device is blocked at authentication.

Evidence and telemetry to validate controls

  • CyberArk PAS Audit: session_id, vendor_account, target_system, session_start_time, commands_executed, duration — compare against approved maintenance window schedule.
  • ZTNA access logs: auth_method, device_posture_score, mfa_type, source_ip, session_granted = true/false.
  • Windows Security Event 4624 on jump server: logon type 10 (RemoteInteractive) from vendor account outside maintenance window.
  • ServiceNow: maintenance window ticket lookup — cross-reference vendor session timestamp against open change requests for that vendor.

T3 — MES Bidirectional Write — Malicious SCADA Production Order Injection

STRIDE category: Tampering
Risk: Medium likelihood, Critical impact
Attacker objective: Compromise the MES server in the Level 3.5 DMZ and inject malicious production orders or process parameters via the allowed bidirectional write path into SCADA — causing the SCADA system to send unsafe setpoints to PLCs, producing physical damage or a safety event.

How the attacker would exploit it

  • Prerequisite: Attacker has compromised a host in the IT network (T1 path) and moved laterally to the MES server in the DMZ, which has an authorized write path to the SCADA API.
  • Step 1: From the compromised MES host, attacker enumerates the SCADA API endpoint (OPC-UA server address, REST API endpoint, or Modbus TCP gateway).
  • Step 2: Crafts malicious write commands: temperature setpoints set to 200% of normal operating range, motor speed ramp rates changed to exceed mechanical limits, or safety interlock thresholds modified.
  • Step 3: Injects the malicious write via the authorized MES-to-SCADA API path — the write appears to come from a known authorized source (the MES server IP is in the SCADA allow-list).
  • Step 4: SCADA propagates the malicious setpoints to PLCs at Level 1/2 via OPC-DA/UA or Modbus TCP.
  • Step 5: Impact: PLCs execute the malicious setpoints; physical process operates outside safe parameters — potential equipment destruction, fire, or worker safety event.

Blue-team control review — what must stop the chain

  • Control 1: SCADA API gateway with cryptographic input validation: all MES writes must include a signed change token (HMAC-SHA256 with a shared secret known only to the MES system and the SCADA API gateway); replayed or spoofed writes without valid signatures are rejected.
  • Control 2: Parameter range enforcement at the SCADA API gateway: every write value validated against engineering-defined safe operating range (min/max bounds per tag); values outside bounds rejected with alert, never propagated to PLC.
  • Control 3: Change window enforcement: MES write path active only during production schedule windows; any write outside the scheduled window requires an out-of-band approval token from the plant supervisor.
  • Control 4: Claroty/Dragos: alert on any SCADA setpoint write where the new value deviates more than 15% from the nominal operating baseline; alert fires before the PLC executes the command.
  • Control 5: Process historian anomaly baseline: SCADA alarm journal alert when setpoints change at a rate or magnitude inconsistent with normal recipe transitions — detects the injection in the SCADA layer before physical impact.

Evidence and telemetry to validate controls

  • SCADA API gateway logs: write_request, source_ip = MES-server, tag_name, value, signature_valid = true/false, within_operating_range = true/false.
  • Dragos/Claroty: SetpointChangeOutsideRange event — tag, new_value, operating_min, operating_max, source_host.
  • SCADA alarm journal: AlarmId, AlarmType = SetpointDeviation, OperatorId, AcknowledgeTime — unacknowledged anomalous setpoint alarms within seconds of a write are a strong indicator.
  • DMZ firewall: connection from MES server to SCADA on allowed port — timestamp correlation with anomalous setpoint event confirms MES as the write source.

T4 — Internet-Exposed HMI Exploitation

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Discover an HMI directly exposed to the internet (via Shodan search for port:102 siemens or port:20000 GE SRTP), exploit a known CVE in the HMI software, and gain remote code execution — then use that foothold to reach SCADA and PLC systems on the internal OT network.

How the attacker would exploit it

  • Prerequisite: HMI is running a Windows-based SCADA/HMI application (Wonderware InTouch, Ignition, FactoryTalk View) on a network with direct or NAT-exposed internet access due to misconfigured firewall or an IT team who added remote access without informing the OT security team.
  • Step 1: Attacker uses Shodan query port:102 country:US or "Siemens SIMATIC" to discover the exposed HMI. Fingerprint the application version from banner.
  • Step 2: Identifies a matching CVE (e.g., Ignition Gateway Remote Code Execution, FactoryTalk View Directory Traversal CVE-2020-12001). Downloads public PoC exploit.
  • Step 3: Executes exploit to achieve remote code execution on the HMI — typically a web interface on TCP 8088 or the SCADA historian web API.
  • Step 4: From the HMI, the attacker now has direct OPC-DA/UA access to the connected SCADA server at Level 3 — HMIs are by design authorized to issue control commands.
  • Step 5: Impact: attacker issues unauthorized control commands via the HMI; process operated in unsafe state; potential physical destruction.

Blue-team control review — what must stop the chain

  • Control 1: Perimeter firewall: block all inbound internet connections to OT subnet IPs — HMIs must never have a direct internet-facing port; remote access must route through the ZTNA/PAM jump server in the DMZ only.
  • Control 2: External Attack Surface Management (Shodan Monitor, Censys, Microsoft Defender EASM) — weekly scan of all corporate IP ranges; any discovered industrial protocol port (TCP 102, 20000, 44818, 2404, 47808) triggers critical alert.
  • Control 3: Application allow-listing on HMI (Tripwire Enterprise or McAfee Application Control) — prevents exploitation payloads from executing even if vulnerability is triggered; only signed OEM-approved executables run.
  • Control 4: Network access control: HMI VLAN has no route to internet or corporate IT — only allows OPC-UA/DA to SCADA server and engineering workstation access; Layer 3 routing rules enforce this.
  • Control 5: OT vulnerability management: register all HMI software versions in Dragos VulnDB or Claroty Continuous Threat Detection; alert when a new CVE matches a deployed HMI version.

Evidence and telemetry to validate controls

  • External scan results (Shodan Monitor API): any hit on OT-associated ports for corporate IP ranges — port, product, organization, ip.
  • Firewall logs: inbound internet connection to HMI VLAN IP on any port — src_zone=internet, dst_zone=OT-Level2 — should never appear; any occurrence is a critical misconfiguration.
  • Application allow-listing agent: blocked process execution on HMI — ProcessName, Hash, BlockReason = not in allow-list.
  • Claroty/Dragos: new external IP initiating communication with HMI via industrial protocol.

T5 — USB Media on Engineering Workstation

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Insert a malicious USB drive into an engineering workstation (intentionally or via a socially engineered "found USB" drop), execute malware that modifies PLC ladder logic files, and upload the tampered logic to connected PLCs via Siemens TIA Portal or Rockwell Studio 5000.

How the attacker would exploit it

  • Prerequisite: Physical access to a manufacturing plant (employee, contractor, visitor) or a malicious employee; engineering workstation has USB ports active.
  • Step 1: Attacker drops or hands a USB drive to an engineer — Autorun or a disguised executable runs when the drive is inserted (modern Windows has Autorun disabled, but a disguised PDF-icon malware can trick the user).
  • Step 2: Malware installs a backdoor or directly modifies PLC project files in TIA Portal's local project directory (.ap15 files for S7-1500, .ACD files for Rockwell).
  • Step 3: During the next maintenance window, the engineer opens the project in TIA Portal and downloads the logic to the PLC — the malicious function block executes at PLC runtime.
  • Step 4: Malicious logic can include a time-triggered sabotage routine (logic bomb), hidden setpoint manipulation, or SIS bypass commands that activate at a specific process state.
  • Step 5: Impact: malicious PLC logic executes at industrial-scale physical process — potential catastrophic damage, similar to Stuxnet targeting Siemens S7 PLCs.

Blue-team control review — what must stop the chain

  • Control 1: USB port lockdown on engineering workstations via Ivanti Device Control or Microsoft BitLocker To Go policy — only approved, company-issued, encrypted USB drives with a management certificate are writable; personal USB is blocked.
  • Control 2: Application allow-listing: Tripwire Enterprise or McAfee Application Control allows only signed, approved executables — any new unsigned binary from a USB drive is blocked before execution.
  • Control 3: PLC logic integrity monitoring via Dragos Platform: hash of PLC logic file backed up after each approved change window; any deviation from the last approved hash outside a change ticket fires a Critical alert.
  • Control 4: Dual-approval for PLC logic download: TIA Portal project download requires both the engineer and a second senior engineer to confirm the logic has been reviewed — change management gate.
  • Control 5: Physical security: engineering workstations locked in a separate room with badge-controlled access; visitor access to plant floor never includes engineering workstation area.

Evidence and telemetry to validate controls

  • Windows Security Event 4663: object access on USB drive path — ObjectName matches removable media letter, ProcessName = tia64.exe or Studio5000.exe — unusually, if TIA Portal reads from USB, that's a finding.
  • Application allow-list agent: Action = Blocked, FileName, FilePath = E:\ or F:\, Reason = Unsigned or Not in Allow-list.
  • Dragos: PLCLogicIntegrityViolation — PLC_ID, ExpectedHash, ActualHash, Timestamp, AssociatedChangeTicket = null.
  • Windows Security Event 4663 + 4688: USB insertion followed by executable launch from drive letter — correlation of USB device insert event (Event 2003 in Windows via USB audit policy) with process creation.

T7 — Unauthenticated Modbus Write from Compromised Level 2 Host

STRIDE category: Tampering
Risk: High likelihood, High impact
Attacker objective: From a compromised HMI or engineering workstation at Level 2, send unauthenticated Modbus TCP or DNP3 write commands directly to PLCs at Level 1 — bypassing SCADA entirely and directly manipulating PLC registers and coils.

How the attacker would exploit it

  • Prerequisite: Attacker has compromised a host at Purdue Level 2 (HMI or engineering workstation) that has direct network access to PLCs at Level 1 via Modbus TCP.
  • Step 1: Attacker uses pymodbus or ModbusPal to enumerate reachable PLCs: modbus_client.read_coils(address=0, count=10, unit=1) — scans Modbus unit IDs 1–247 on the Level 1 subnet.
  • Step 2: Identifies critical control registers by correlating process values with plant documentation or by observing normal traffic patterns via Claroty passive sensor baseline.
  • Step 3: Sends malicious write: modbus_client.write_coil(address=12, value=True, unit=5) — toggles a valve coil or motor control bit; Modbus TCP has no authentication, no source IP restriction at the PLC level.
  • Step 4: PLC executes the write immediately — Modbus is a pure request-response protocol with no authorization mechanism; any host on the same Layer 2 segment can issue writes.
  • Step 5: Impact: physical actuator commanded to unsafe state; process disruption or equipment damage; SCADA only learns of the change after the fact via historian polling.

Blue-team control review — what must stop the chain

  • Control 1: Micro-segmentation: PLCs on dedicated VLANs with firewall rules allowing Modbus TCP only from the specific SCADA server and engineering workstation IPs — no HMI or general workstation can reach PLC subnets directly.
  • Control 2: Dragos/Claroty: passive IDS detects Modbus Function Code 5 (Write Single Coil) or FC 16 (Write Multiple Registers) from any IP not in the approved source list — Critical alert within 30 seconds.
  • Control 3: Managed switches at Level 1/2 boundary: MAC-based port security limiting each switch port to the known MAC address of the authorized PLC or workstation — prevents IP spoofing and unauthorized device insertion.
  • Control 4: OPC-UA with certificate-based authentication between SCADA and PLCs: modern ICS deployments can use OPC-UA security policies (Basic256Sha256) — PLCs with OPC-UA support can be configured to reject connections without a valid client certificate.
  • Control 5: Process safety interlocks: PLCs programmed with hardcoded safety limits in ladder logic — setpoints outside the safe range are ignored by the PLC program regardless of the write source (defense-in-depth at the PLC logic level).

Evidence and telemetry to validate controls

  • Claroty/Dragos: ModbusWriteFromUnauthorizedSource — src_ip, dst_ip, function_code = 5 or 16, register_address, value, timestamp.
  • Layer 2 switch port security log: MAC address violation on PLC access port — port_id, expected_mac, seen_mac.
  • SCADA alarm: setpoint change event where source = Modbus and source_ip not matching the SCADA server IP — indicates out-of-band write directly to PLC.
  • Dragos: Function code frequency analysis — baseline FC15 (write multiple coils) frequency; spike beyond 3σ from baseline indicates automated write attack.

T12 — Insider PLC Logic Bomb During Maintenance

STRIDE category: Tampering
Risk: Low likelihood, Critical impact
Attacker objective: A malicious insider engineer (disgruntled employee or coerced by an external actor) inserts a time-triggered or condition-triggered logic bomb into a PLC program during an authorized maintenance window — designed to activate weeks or months later, making attribution difficult.

How the attacker would exploit it

  • Prerequisite: Insider has legitimate access to an engineering workstation and a scheduled maintenance window for PLC programming.
  • Step 1: During an authorized maintenance session, insider opens the PLC project in TIA Portal alongside the authorized changes being made.
  • Step 2: Adds a hidden function block that contains the logic bomb: a block that monitors a specific date/time condition or a specific sensor value combination, then triggers an unsafe actuator state.
  • Step 3: The logic bomb is carefully embedded to avoid casual review — hidden in a low-level utility function, disguised as a timing routine, or inserted with an obscure condition that will not trigger during test.
  • Step 4: Downloads the modified logic to the PLC alongside the legitimate changes — the change ticket shows the authorized modification; the malicious block is not documented.
  • Step 5: Insider resigns or is otherwise unavailable when the bomb triggers weeks later — attribution and forensics are complicated by time elapsed.

Blue-team control review — what must stop the chain

  • Control 1: Dual-approval for PLC logic download: two independent senior engineers must review and approve the full project file diff before any download — malicious function blocks are difficult to hide from a thorough code review.
  • Control 2: Automated logic diff: Git-based OT code repository with CI pipeline that diffs every version commit — any new function block or modification to an existing block that was not in the approved change ticket generates a review task.
  • Control 3: Dragos post-download integrity check: after every maintenance download, Dragos captures the running PLC logic hash and compares against the version in the OT code repository; any difference fires a Critical alert.
  • Control 4: PLC code static analysis: tool like Codesys Security Advisor or Claroty PLC analysis scans ladder logic for suspicious patterns — infinite loops, timed conditions in unexpected function blocks, undocumented variable access.
  • Control 5: Insider risk indicators: correlate HR signals (resignation notices, performance issues, unusual access to plant documentation) with PAM session activity and PLC change frequency — elevated risk individuals get enhanced monitoring of maintenance sessions.

Evidence and telemetry to validate controls

  • Dragos: PLCLogicHashMismatch post-download — ExpectedHash = repo-commit-hash, ActualHash = running-logic-hash — any mismatch requires rollback and investigation.
  • OT Git repository: git diff HEAD~1 HEAD -- <PLC_project_file> showing undocumented function block additions.
  • CyberArk session recording: screen recording of the maintenance session — forensic video replay shows exactly what the engineer modified; keylogger timestamps correlate with undocumented changes.
  • ServiceNow change ticket: authorized changes documented; logic bomb function block is not in the change description — the diff between the approved ticket and the actual download reveals the anomaly.

T13 — Compromised OT Vendor Software Update (Supply Chain)

STRIDE category: Tampering
Risk: Low likelihood, Critical impact
Attacker objective: Compromise the software update distribution channel for an OT vendor (Siemens, Rockwell, Honeywell) to deliver a backdoored TIA Portal, SCADA, or historian update to all plants using that software — gaining persistent access to every OT environment that applies the tampered update.

How the attacker would exploit it

  • Prerequisite: Attacker compromises the OT vendor's software distribution infrastructure (similar to SolarWinds) or a third-party update mirror used by the plant.
  • Step 1: Attacker injects a backdoored DLL into the vendor software installer — for example, a modified TIA_Portal_V16_Update.exe that includes a remote access implant alongside the legitimate update content.
  • Step 2: Plant engineers download and apply the "official" update to engineering workstations and SCADA servers during their next patch window.
  • Step 3: The backdoor establishes a C2 beacon from the engineering workstation or SCADA server to the attacker's infrastructure — using HTTPS to blend with normal traffic.
  • Step 4: Attacker uses the C2 to conduct reconnaissance across the OT network, then laterally moves to PLCs via TIA Portal.
  • Step 5: Impact: all plants that applied the same vendor update are simultaneously compromised; one operation achieves broad OT access across dozens of facilities.

Blue-team control review — what must stop the chain

  • Control 1: Cryptographic signature verification before any OT software installation: require SHA-256 hash verification against the vendor's published checksum and digital signature verification against the vendor's code signing certificate before any installer is run.
  • Control 2: Application allow-listing on engineering workstations and SCADA servers: any new unsigned or untrusted binary introduced by the update is blocked before execution; the allow-list only permits binaries signed by the approved OT vendor code signing certificate.
  • Control 3: OT software staging environment: apply vendor updates to an isolated OT test bench environment first; run for 14 days and monitor for unexpected network connections or process anomalies before applying to production.
  • Control 4: Network: block outbound internet from engineering workstations and SCADA servers — any C2 beacon attempt from a backdoored update is blocked at the Level 2/3 firewall.
  • Control 5: Vendor software update notification subscription: subscribe to vendor security bulletins (Siemens ProductCERT, Rockwell Security Advisory) — correlate update release timing and hash with ICS-CERT advisories before deployment.

Evidence and telemetry to validate controls

  • Application allow-list agent: Action = Blocked, FileName = TIA_Portal_backdoor.dll, Reason = Untrusted publisher or Hash not in allow-list.
  • Firewall: outbound connection from engineering WS to non-vendor external IP on TCP 443 during or after update installation — timing correlation with update deployment.
  • Dragos: new process spawning from TIA Portal or SCADA application process tree — ParentProcess = TIA64.exe, ChildProcess = powershell.exe or cmd.exe — unusual child process from OT application.
  • Hash verification log: update installer hash compared against vendor-published checksum — mismatch before installation prevents execution.

T14 — Ransomware Lateral Movement from IT to OT

STRIDE category: Denial of Service, Tampering
Risk: Medium likelihood, High impact
Attacker objective: Deploy ransomware across the IT network and use the IT/OT network convergence to extend encryption or destruction to the OT historian and SCADA servers — causing a production shutdown that multiplies ransom pressure beyond what IT-only ransomware achieves.

How the attacker would exploit it

  • Prerequisite: Attacker has compromised the IT network (T1 path) and deployed ransomware with OT-targeting capability (EKANS/SNAKE, Industroyer2).
  • Step 1: Ransomware enumerates the network for OT historian processes: wmic process list looking for PI System, OPCServer, RSLinx, FactoryTalkActivation — EKANS specifically terminates these services before encrypting.
  • Step 2: Uses the IT-to-DMZ firewall path (if misconfigured or previously tunneled via T1 lateral movement) to reach the historian server in the DMZ.
  • Step 3: Terminates OT historian service processes and encrypts historian data files (OSIsoft PI archive files, .pida, .pidat extensions) — causing loss of all recent process data.
  • Step 4: Attempts deeper penetration to SCADA at Level 3 — if allowed by firewall rules, encrypts SCADA server data and configuration files, causing total loss of process visibility.
  • Step 5: Impact: manufacturing plant loses process visibility and historical data; plant switches to manual operation; ransom demand includes both IT recovery and OT historian restoration.

Blue-team control review — what must stop the chain

  • Control 1: Firewall emergency isolation playbook: at first ransomware detection on IT network, SOAR automatically blocks ALL IT→DMZ firewall rules within 5 minutes — historian is isolated from the IT network before ransomware can reach it.
  • Control 2: OT historian offline backup on an air-gapped, read-only medium: daily snapshot of PI archive files to an isolated backup server not accessible from the IT or DMZ network — ensures recovery even if historian is encrypted.
  • Control 3: Micro-segmentation: historian server on its own VLAN with deny-all from IT subnets except the two approved analytics servers — reduces the blast radius of a ransomware outbreak to not automatically reaching the historian.
  • Control 4: Process isolation for OT historian services: run PI System as a non-elevated service account; historian files on a separate volume with an ACL that prevents the ransomware's service account from writing; VSS shadow copy enabled.
  • Control 5: OT-aware ransomware detection: Dragos/Claroty detects mass encryption events on OT historian — unusually high file modification rate on historian data directory triggers immediate Critical alert with auto-isolation of historian VLAN.

Evidence and telemetry to validate controls

  • EDR on IT: ransomware process behaviors — wmic process terminate targeting OT process names, mass file rename events, vssadmin delete shadows.
  • DMZ Firewall: connection from IT subnet to historian IP — should be in deny-all during ransomware response; SOAR confirms emergency block rule was applied.
  • Historian file integrity: file modification events on .pida and .pidat archives at a rate exceeding normal historian write speed (typically 1–5 new archives per day; mass modification is anomalous).
  • Windows Event 7036 on historian: OPC Server or PI System service stopped outside maintenance window — service stop is a precursor to ransomware encryption.
