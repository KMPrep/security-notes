Scenario 08 — Network Segmentation for OT/IT Convergence

Scenario: A manufacturing plant connects its OT network (PLCs, SCADA, HMIs) to the corporate IT network for remote monitoring. A Purdue Model segmentation is in place but a business requirement adds bidirectional data flows.

Candidate questions:

  • What does the threat model look like when IT/OT boundary has bidirectional flows?
  • How do you model a nation-state threat targeting SCADA systems via the IT network?
  • What controls replace traditional patching in OT environments?
  • What monitoring capabilities are realistic in OT environments given vendor restrictions?
  • What residual risk do you document and accept for unpatched PLCs?

3.4 Highest-risk boundary crossing

The Level 3.5 DMZ to Level 3 OT boundary is the single highest-risk crossing. When the business requirement adds bidirectional flows — specifically MES writing production orders back into SCADA, and the historian pulling data from Level 3 — the traditional one-way data diode model is broken. An attacker who compromises the historian or MES server in the DMZ now has a network path into the SCADA server at Level 3, which in turn communicates with HMIs and PLCs at Levels 1–2.

The critical failure mode is DMZ system compromise cascading downward. Traditional IT security instincts (patch the DMZ server, isolate it if breached) don't hold — a compromised historian can exfiltrate weeks of process data before detection, and a compromised MES can inject malicious production parameters into active processes. This boundary must be governed by strict protocol allow-listing (OPC-UA read-only from SCADA to historian; MES writes only via a validated API with input validation, not raw Modbus writes).

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

Manual firewall rule reviews and OT asset inventory audits fail at scale when a company operates 50+ manufacturing plants globally. At Google scale, you would codify the entire OT segmentation policy as Terraform or Ansible automation: firewall allow-lists generated from a CMDB asset registry, so any new OT device that appears without a CMDB entry automatically triggers a deny rule and an alert. Claroty or Dragos APIs would feed discovered assets into the CMDB automatically, and any delta (discovered but not registered, or registered but not discovered) would trigger a workflow in ServiceNow. Vendor access windows would be provisioned and expired via API calls to CyberArk, not human tickets.

8.2 Fleet-wide impact analysis

A single vulnerable historian software version across 50 plants is a fleet-wide critical vulnerability. If an attacker exploits the historian at one plant and it runs the same OSIsoft PI version as 49 others, lateral movement via the shared IT network backbone becomes a multi-plant incident. At Google scale, a single compromised OT DMZ system that communicates with corporate IT Active Directory can become a domain-wide incident. The blast radius is not one plant — it is every plant connected to the same AD forest and IT backbone.

8.3 Eliminating human bottlenecks

Today, "plant operator approves PLC change" is a human gate that takes hours or days. At scale with continuous deployment of production recipes, this becomes a bottleneck. The solution is a signed, policy-enforced change pipeline: production changes come from an MES system with cryptographically signed change tokens; the SCADA API gateway validates the signature before applying any setpoint change. No human approval needed for routine recipe changes — but any deviation from the signed parameter envelope requires a human gate. This mirrors Google's binary authorization model applied to OT.

8.4 Controls that break at scale

Manual OT asset inventory works for one plant with 200 devices. At 50 plants with 10,000+ devices, it becomes inaccurate within weeks. The control that breaks is the "approved device list maintained in spreadsheet" model — you need Claroty or Dragos performing continuous automated discovery feeding a CMDB. Similarly, manually reviewing 500 vendor access sessions per week across 50 plants is impossible; you need behavioral analytics on PAM session recordings (keystroke analytics, command risk scoring) to triage automatically.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What monitoring capabilities are realistic in OT environments given vendor restrictions?

Passive network monitoring via tap or SPAN port is the realistic baseline — Claroty or Dragos deployed as passive sensors can parse industrial protocols (Modbus, OPC-UA, DNP3, EtherNet/IP) without touching the OT devices. Active scanning is generally not acceptable because it can crash PLCs. For host-based visibility, you are limited to what the HMIs and engineering workstations (typically Windows) support — WEF/WEC for Windows Event Logs, and application allow-listing agents like Tripwire. PLCs themselves typically have no logging API; you infer their state from network traffic and historian data.

Follow-up 2: What controls replace traditional patching in OT environments?

The patching control stack for OT is compensating controls, not remediation: network micro-segmentation (attackers can't reach an unpatched PLC if it's on a VLAN with an allow-list of two authorized hosts), application allow-listing on hosts that interact with PLCs, integrity monitoring to detect any change to PLC firmware or logic, and virtual patching via the OT IDS/IPS (Claroty/Dragos can block exploit traffic matching known CVE signatures at the network level). Vendor-negotiated patching windows during planned maintenance shutdowns are the only path for actual firmware updates.

Follow-up 3: How do you model a nation-state threat targeting SCADA systems via the IT network?

I use the Dragos Year in Review threat intelligence combined with ICS ATT&CK to model the kill chain: initial access via spear-phishing an IT employee or compromising a VPN credential, lateral movement through AD to reach the OT DMZ, then pivoting to the historian, and finally reaching SCADA via the allowed data path. The key intelligence to layer in is that groups like VOLTZITE (Volt Typhoon) specifically target IT/OT converged environments and pre-position for years before activating. My detection strategy focuses on the historian as the IT-to-OT choke point — any unexpected connection from a new IT host to the historian is a critical alert.

Follow-up 4: You mentioned Claroty/Dragos as a key control. What happens if that control fails — what is your next layer?

If the OT monitoring platform fails to alert, my next layer is the firewall allow-list at the Level 3.5 boundary — any connection not in the explicit allow-list is denied and logged. Beyond that, PAM session recording catches malicious activity by authorized users. And for PLC integrity, I maintain offline snapshots of PLC logic that plant engineers compare against running logic during every maintenance window. The layered detection means an attacker has to defeat three independent detection mechanisms to reach a PLC undetected.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is compromising a vendor's laptop before they connect via VPN, since vendor endpoints are outside my security perimeter. The vendor authenticates with FIDO2 and device cert, but if their laptop already has a RAT installed, the attacker rides the legitimate session into the jump server and watches the CyberArk-proxied session to the SCADA system. My detection is the behavioral analytics on the PAM session (unusual commands, wrong timing, typing cadence anomaly) and the session recording for post-incident forensics. The control I'd harden further is requiring vendors to use a company-provisioned, managed tablet for all OT access — eliminating the BYOD vendor endpoint risk entirely.

COMMON MISTAKES TO AVOID

  • Treating OT security like IT security — Saying "just patch the PLCs" or "deploy EDR on all OT assets" signals you don't understand OT constraints. Patching PLCs requires vendor involvement, maintenance windows, and often physical access.
  • Recommending active scanning of OT networks — Active scanning tools (Nessus, Qualys) can cause PLCs and RTUs to crash. Passive monitoring is the only safe option.
  • Ignoring the Safety Instrumented System — Not mentioning SIS air-gap requirements, or treating SIS as just another OT device, signals shallow OT knowledge. SIS compromise is a safety-of-life risk.
  • Proposing bidirectional data diodes — Data diodes are by definition unidirectional. Saying you'd "configure a data diode for bidirectional flow" is a contradiction that immediately loses credibility.
  • Overlooking the vendor access vector — Shared credentials on a jump server is the scenario premise; jumping straight to network controls without addressing the identity and access problem misses the primary attack surface.
  • Generic SIEM rules that don't account for OT protocols — Proposing Windows Event Log-based detection as the primary OT monitoring strategy ignores that PLCs and RTUs generate no Windows events. OT-specific tools are required.
  • Not quantifying physical consequence — Failing to connect compromise to physical-world impact (production halt, safety event, equipment damage) misses the core reason OT security is different from IT security.
