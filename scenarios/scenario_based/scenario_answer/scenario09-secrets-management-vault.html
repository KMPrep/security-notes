Scenario 09 — Secrets Management Platform Deployment

Scenario: The enterprise is deploying HashiCorp Vault to replace hardcoded secrets in 400 applications across dev, staging, and production environments.

Candidate questions:

  • What is the bootstrapping problem in secrets management and how do you solve it?
  • How do you model a compromised CI/CD pipeline accessing Vault?
  • What controls prevent secret sprawl after Vault is deployed?
  • How do you monitor for unusual secret access patterns?
  • What residual risk remains for legacy apps that cannot be migrated to dynamic secrets?

3.4 Highest-risk boundary crossing

The highest-risk boundary is the CI/CD pipeline's authentication to Vault. CI/CD runners are high-value targets because they have broad code and secret access by design. If an attacker compromises a GitHub Actions workflow file (via a malicious PR, a compromised contributor account, or a poisoned dependency) they can exfiltrate the Vault token or AppRole credentials that the pipeline uses. The blast radius is proportional to the Vault policy scoped to that CI/CD role.

The critical failure mode is over-privileged CI/CD Vault policies — if the build pipeline has a Vault policy granting read access to production secrets "for convenience," a CI/CD compromise immediately becomes a production secret dump. The bootstrap chain (trusted orchestrator → wrapped secret_id → application) must be enforced architecturally, not by policy alone.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale, Vault policies, auth method configurations, and secret mounts must be 100% managed via Terraform with mandatory peer review and CI/CD pipelines. No human should have interactive admin write access to Vault policies in production. The Vault admin role is reserved for break-glass emergencies only, triggered via a PAM system that pages on-call and records every command. Application onboarding to Vault is automated via a self-service GitOps workflow: developers submit a PR with their app's secret requirements, the PR triggers a Terraform plan that creates the AppRole, scopes the policy, and configures the auth method — no human Vault admin required.

8.2 Fleet-wide impact analysis

A Vault cluster serving 400 applications is a single point of failure and a single point of compromise. If an attacker dumps all secrets from a single Vault cluster, 400 applications are simultaneously compromised. At Google scale, the architecture moves to Vault clusters partitioned by blast radius: separate clusters for different business units, sensitivity tiers, or geographic regions. A root token or unseal key compromise in one cluster cannot affect another. Additionally, dynamic secrets mean a Vault outage doesn't immediately break applications — leases are renewed lazily, and static KV secrets are cached — but the window is TTL-bounded. Fleet-wide Vault outage planning must account for the cascading application failure as TTLs expire.

8.3 Eliminating human bottlenecks

The bottleneck in most Vault deployments is "create a ticket for the Vault admin to add a new secret path to the policy." At Google scale with thousands of microservices, this is untenable. The solution is self-service secret path provisioning: each service owns a Vault path namespace (e.g., secret/data/svc-<service-name>/) and the service team can manage secrets within that namespace without admin intervention. Cross-service secret access requires a PR-reviewed policy change. Secret rotation is fully automated via Vault's dynamic secrets engines — no human initiates rotation.

8.4 Controls that break at scale

Manual quarterly secret rotation works for 20 applications. For 400+ applications, it fails — rotation windows get missed, app owners don't coordinate, and stale credentials accumulate. The control that breaks is human-initiated rotation; the replacement is dynamic secrets with auto-renewal enforced at the platform level. Similarly, manually auditing Vault policies for over-privilege every quarter is impossible at 400+ apps; you need an automated policy analyzer (HashiCorp's Vault Policy Advisor or custom tooling) that flags any policy with wildcard capabilities or overly broad paths.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What is the bootstrapping problem in secrets management and how do you solve it?

The bootstrapping problem is: to get a secret from Vault, the application needs to authenticate to Vault — but that initial credential is itself a secret. If you store it in an env var or config file, you've just moved the secret problem, not solved it. The solution is using a trusted platform identity as the first factor: on Kubernetes, the pod's K8s service account JWT is cryptographically issued by the Kubernetes API and Vault can verify it directly via the K8s auth method. On AWS, it's the EC2 instance's IAM role via the AWS auth method. The platform infrastructure becomes the trust anchor, not a pre-shared secret. For on-prem apps without a cloud identity, AppRole with response-wrapped secret_id delivery via a trusted orchestrator (Ansible, Terraform) provides a pragmatic solution.

Follow-up 2: How do you model a compromised CI/CD pipeline accessing Vault?

A compromised CI/CD pipeline has legitimate Vault credentials — it will authenticate as the CI identity, not as an attacker. Detection depends on behavioral anomalies: the CI role suddenly reading production database credentials when it normally only reads build-time config secrets; accessing a volume of secret paths far above the pipeline's normal baseline; or accessing secrets at an unusual time (3am on a Sunday). The preventive control is tight Vault policy scoping for the CI identity — CI should only read secrets needed at build time, never runtime production secrets. Dynamic OIDC tokens from GitHub Actions (scoped to repo and branch) limit exposure further, as each pipeline run gets a fresh short-lived token.

Follow-up 3: What controls prevent secret sprawl after Vault is deployed?

Secret sprawl is the tendency for secrets to proliferate in non-Vault stores (environment variables, K8s secrets, .env files, AWS SSM Parameter Store) even after Vault is deployed. Prevention requires three things: a secrets discovery scanner (truffleHog, Gitleaks) integrated into CI that blocks commits containing secret patterns; quarterly automated scans of cloud secret stores (AWS Secrets Manager, K8s secrets) to inventory non-Vault secrets; and a platform policy that application deployments fail if they mount non-Vault secrets without an approved exception. Making Vault easier to use than the alternative (a Vault sidecar injector in Kubernetes that auto-injects secrets without code changes) reduces the motivation for shortcuts.

Follow-up 4: You mentioned immutable audit logs. What happens if that control fails — what is your next layer?

If Vault audit log delivery fails, by default Vault blocks all requests rather than allowing secret access without audit logging — this is Vault's fail-safe behavior (configurable; should be set to block). So the audit log failing is itself a denial-of-service risk, not a silent security failure. The monitoring layer watches for audit log delivery gaps: if no Vault events reach the SIEM for 60+ seconds during business hours, an operational alert fires. The next detective layer is network flow analysis — bulk HTTPS traffic to port 8200 from unexpected hosts is visible even without application-layer logs. The compensating control for lost audit coverage is Vault metrics (token count, lease count) which don't capture individual operations but show anomalous aggregate patterns.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is targeting developer workstations or laptops that have Vault CLI configured with a long-lived personal token for development use. Developers frequently authenticate to Vault dev clusters with tokens that have broad read access for convenience, and these tokens often have TTLs measured in days or weeks. If I can compromise a developer's laptop (phishing, malicious npm package, browser exploit), I inherit their Vault token and can read anything their policy allows. The control I'd harden: enforce max token TTL of 8 hours for all human identities via Vault token policies, require MFA for all Vault CLI logins by mapping the Vault token to an OIDC session that requires re-authentication daily.

COMMON MISTAKES TO AVOID

  • Claiming Vault "solves" secret sprawl — Vault deployment does not automatically eliminate secrets in env vars, .env files, or K8s secrets. Secret sprawl requires active discovery, migration, and platform enforcement to eliminate.
  • Treating the root token as an admin credential — The root token should be generated once, used only to set up initial auth methods, and then revoked. Saying "use the root token for admin operations" signals a fundamental misunderstanding.
  • Ignoring the bootstrapping problem — Saying "store the AppRole role_id and secret_id in environment variables" just moves the problem; not addressing how those initial credentials are secured signals shallow thinking.
  • Proposing static long-lived secrets as the primary pattern — Vault's security value comes from dynamic secrets and short TTLs. Using Vault purely as a fancy .env file manager with long-lived KV secrets eliminates most of the security benefit.
  • Missing the CI/CD attack vector — Focusing only on application-to-Vault auth while ignoring the CI/CD pipeline as a high-privilege Vault client is a significant blind spot.
  • Forgetting audit log integrity — Not mentioning immutable audit log forwarding signals incomplete thinking. An attacker who can clear Vault audit logs can operate invisibly.
  • Not addressing Vault HA and disaster recovery — Vault is now business-critical infrastructure. A Vault outage can cascade to application failures as dynamic secrets expire. Not mentioning HA architecture, backup unseal key storage, and DR testing reveals operational immaturity.
