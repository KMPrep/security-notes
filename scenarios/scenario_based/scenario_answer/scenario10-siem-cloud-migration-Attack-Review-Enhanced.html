SIEM Migration from On-Prem Splunk to Cloud SIEM — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario10-siem-cloud-migration.md
Generated artifact: scenario10-siem-cloud-migration-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Migrate a 10-year-old on-prem Splunk deployment to a cloud-native SIEM while retaining 500+ detection rules, preserving 200 TB of historical logs, and keeping the 24/7 SOC operational without detection gaps.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

The review focuses on attack mechanics, control validation, and cloud SIEM migration-specific risk — including log injection, alert suppression, admin account compromise, and data sovereignty. Commands use placeholders and are intended for an owned lab or approved internal purple-team exercise only.

2. Highest-Risk Design Decision

The single most consequential design decision is whether the cloud SIEM is treated as a security-critical asset with its own hardened admin plane and immutable audit trail, or whether it is treated as a utility only for ingesting logs.

A cloud SIEM is not just a collector; it becomes the decision point for detection, alerting, and response. If the cloud SIEM admin console is left accessible without strong PAM, FIDO2, and out-of-band auditing, a compromise there can silence detection across the entire enterprise. The safe choice is to protect the SIEM admin plane with conditional access, PAM-checked privileged sessions, and separate immutable audit logging. The dangerous alternative is to rely on normal IAM access controls and assume the cloud provider will prevent all admin-plane misuse.

3. Attack-Tree Legend

Every attack tree uses this convention:

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Compromised SOC Analyst Account

STRIDE category: Spoofing, Repudiation
Risk: Critical likelihood, Critical impact
Attacker objective: Phish or otherwise compromise a SOC analyst account and use it to suppress alerts, disable detection rules, or modify response workflows — effectively blinding the SOC during the attacker's activity.

How the attacker would exploit it

  • Prerequisite: attacker obtains valid credentials for a cloud SIEM analyst account via spear-phishing, MFA fatigue, or credential stuffing.
  • Step 1: attacker logs into the cloud SIEM admin console using stolen credentials or an AiTM proxy session.
  • Step 2: attacker navigates to detection rule management and disables or deletes critical rules that would detect their C2 and lateral movement.
  • Step 3: attacker suppresses alerts or modifies SOAR playbooks to ignore specific alert categories or source IPs.
  • Step 4: attacker continues attack with reduced visibility while the SOC investigates fewer or no alerts.
  • Step 5: impact: attacker achieves extended dwell time and can exfiltrate data or deploy ransomware before detection.

Blue-team control review — what must stop the chain

  • Control 1: FIDO2 hardware MFA for all SOC analyst and SIEM admin accounts; password-only or TOTP-based logins are not sufficient.
  • Control 2: Privileged Access Management (PAM) for all SIEM admin sessions; direct console access is blocked, and every privileged session must be brokered through CyberArk/BeyondTrust.
  • Control 3: Conditional Access policy restricting SIEM admin console access to managed SOC devices or known SOC locations.
  • Control 4: Out-of-band immutable audit logging for all SIEM admin actions; logs are stored in a separate bucket or log tenant that SIEM admins cannot modify.
  • Control 5: Alert suppression workflow requiring dual analyst approval via SOAR; no single analyst can suppress critical alert categories without ticket correlation.

Evidence and telemetry to validate controls

  • Cloud SIEM admin audit log: operation = disable_rule OR delete_rule, principal = <analyst>, timestamp.
  • Conditional Access logs: sign-in to SIEM console from a non-managed device or unexpected IP.
  • PAM session audit: session_started, session_user, session_command correlated with SIEM admin actions.
  • SOAR audit logs: suppression action without approved change request ID.

Purple-team validation commands

# Validate FIDO2 enforcement for SIEM admin accounts (authorized lab)
# Query Azure AD conditional access policies for SIEM admin group
Get-AzureADMSConditionalAccessPolicy | Where-Object { $_.DisplayName -like "*SIEM Admin*" }

// Detect SIEM admin rule disable without approved SOAR ticket
CloudSIEMAdminAudit
| where operation in ("DisableRule", "DeleteRule")
| where not(AdditionalFields contains "ChangeRequestID")
| project TimeGenerated, principal, operation, targetRule
| order by TimeGenerated desc

T2 — Cloud SIEM Admin API Key Leak

STRIDE category: Spoofing, Information Disclosure
Risk: High likelihood, Critical impact
Attacker objective: Steal or discover a cloud SIEM admin API key from a repository, developer workstation, or forwarder config, then use it to modify rules, bulk-export logs, or suppress alerts via the cloud SIEM API.

How the attacker would exploit it

  • Prerequisite: attacker finds a service principal or API key in code, config file, or a leaked repository.
  • Step 1: attacker uses the API key to authenticate to the cloud SIEM API from an external host.
  • Step 2: attacker enumerates available API actions and identifies admin-level capabilities.
  • Step 3: attacker disables detection rules, exports historical logs, or changes retention settings.
  • Step 4: attacker exfiltrates the logs or uses the suppression capability to reduce detection.
  • Step 5: impact: the attacker gains the same control as a compromised admin account, with no interactive login trace.

Blue-team control review — what must stop the chain

  • Control 1: dedicated service accounts with narrow API scopes; admin API keys separated from ingestion and read-only keys.
  • Control 2: secret scanning on code repos, CI/CD pipelines, and artifact registries to detect leaked SIEM API keys.
  • Control 3: API key rotation policy — keys expire frequently and are replaced automatically.
  • Control 4: cloud provider IAM logs and SIEM audit logs that detect bulk export operations or excessive rule reads.
  • Control 5: key usage restrictions by source IP or service principal identity.

Evidence and telemetry to validate controls

  • Cloud provider IAM logs: API key used from unexpected principal or external IP.
  • SIEM API audit logs: bulk export call or admin policy modification call.
  • Repository scanning alerts: AWS_ACCESS_KEY_ID or SIEM_API_KEY pattern found in repo push.
  • CI/CD scan logs: secret detected for SIEM API key patterns.

Purple-team validation commands

# Validate cloud SIEM API key scope is narrow (authorized lab)
# Use an API key with limited scope and call the admin action
curl -H "Authorization: Bearer <api_key>" https://<cloud-siem>/api/v1/rules
# Expected: key can list rules only if scoped; cannot delete or modify admin settings if not permitted

// Detect SIEM API key bulk export or admin access from unexpected IP
CloudProviderAudit
| where ResourceType == "CloudSIEM" and OperationName in ("ExportLogs", "ModifyRule")
| where CallerIpAddress !in (<approved_api_ip_ranges>)
| project TimeGenerated, Caller, OperationName, CallerIpAddress
| order by TimeGenerated desc

T3 — Log Forwarder Compromise on Endpoint

STRIDE category: Denial of Service, Tampering
Risk: High likelihood, High impact
Attacker objective: Compromise a Splunk Universal Forwarder, Fluentd, or other log agent on an endpoint, then stop log forwarding to the cloud SIEM or alter log content to evade detection.

How the attacker would exploit it

  • Prerequisite: attacker has code execution or administrative privileges on a host.
  • Step 1: attacker stops or disables the forwarder service: sudo systemctl stop splunkforwarder or service td-agent stop.
  • Step 2: attacker may also modify forwarder configuration to exclude suspicious logs or inject false metadata.
  • Step 3: attacker uses the host as a pivot point while the SIEM receives no telemetry from that asset.
  • Step 4: if the forwarder is part of an endpoint kill chain, attacker can extend dwell time by suppressing logs from that host.
  • Step 5: impact: detection is blind for that host or host group; the attacker can move laterally or stage data exfiltration.

Blue-team control review — what must stop the chain

  • Control 1: forwarder health heartbeat monitoring and alerting; no log events from a critical host for >10 minutes triggers remediation.
  • Control 2: EDR protection on hosts prevents attacker from stopping the forwarder process or modifying its config.
  • Control 3: forwarder runs as a managed service account with least privilege; host users cannot disable it.
  • Control 4: network-based ingestion fallback; if the local agent is stopped, network devices still forward essential telemetry directly to the cloud SIEM.
  • Control 5: forwarder configuration file integrity monitoring; unexpected changes to /opt/splunkforwarder/etc/system/local/inputs.conf or /etc/td-agent/td-agent.conf trigger alerts.

Evidence and telemetry to validate controls

  • Forwarder heartbeat metrics: last_event_time older than 10 minutes for critical host.
  • EDR logs: process stop of splunkd or td-agent by a non-service account.
  • SIEM ingest metrics: source host absent from the expected host list in the last 15 minutes.
  • FIM alerts on forwarder config file changes.

Purple-team validation commands

# Validate forwarder health monitoring on a lab endpoint
# Stop the forwarder service and confirm SIEM health alert is triggered
Stop-Service -Name SplunkForwarder
# Wait 10 minutes and check the SIEM forwarder health dashboard or alert logs

// Detect critical host forwarder outage
ForwarderHeartbeat
| where HostName in (<critical_hosts>)
| where LastEventTime < ago(10m)
| project TimeGenerated, HostName, LastEventTime, Status
| order by LastEventTime asc

T4 — Detection Rule Deletion or Modification

STRIDE category: Tampering
Risk: High likelihood, Critical impact
Attacker objective: Delete or modify detection rules in the cloud SIEM so that the attacker's own TTPs no longer generate alerts, while keeping the rest of the environment looking normal.

How the attacker would exploit it

  • Prerequisite: attacker has SIEM admin access or a compromised CI/CD pipeline with permissions to deploy detection rules.
  • Step 1: attacker locates high-value rules such as C2 detection, lateral movement, or ransomware behavioral rules.
  • Step 2: attacker disables or deletes those rules in the SIEM admin console or via the API.
  • Step 3: attacker may also modify rule logic to reduce sensitivity for their specific IOC.
  • Step 4: attacker continues the attack; the SOC sees fewer alerts, or no alerts for the modified rule.
  • Step 5: impact: the attacker avoids detection for the duration of the suppressed rule activity and may use that gap to conduct data theft or ransomware.

Blue-team control review — what must stop the chain

  • Control 1: GitOps for detection rules; all rule changes must be deployed through a version-controlled CI/CD pipeline.
  • Control 2: rule drift detection; nightly comparison between deployed SIEM rules and Git HEAD triggers alerts for any manual changes.
  • Control 3: out-of-band rule audit; store rule deployment events in a separate log sink.
  • Control 4: rule modification approval process; manual changes require a documented ticket and dual approval.
  • Control 5: automatic rollback of rule changes that are not in Git within a recovery window.

Evidence and telemetry to validate controls

  • SIEM admin audit log: operation = DeleteRule OR UpdateRule and initiated_by service account.
  • GitOps drift monitor: deployed rule hash != Git HEAD rule hash.
  • CI/CD pipeline audit: rule change was not associated with a PR merge.
  • Out-of-band audit logs: rule modification recorded in a separate immutable store.

Purple-team validation commands

# Validate rule drift detection is active in lab
# Compare deployed SIEM rule export with Git HEAD rule file
# Expected: no drift; any deviation triggers rollback alert

// Detect deployed SIEM rules not matching Git repository state
RuleDriftAudit
| where DeployedRuleHash != GitHeadRuleHash
| project TimeGenerated, RuleName, DeployedRuleHash, GitHeadRuleHash, DeployedBy
| order by TimeGenerated desc

T5 — Log Injection Causing Parser Errors or False Positives

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Craft malicious log entries that either break the cloud SIEM parser or generate noisy false positives, causing the SOC to waste time and potentially miss real alerts.

How the attacker would exploit it

  • Prerequisite: attacker has access to an application or host that generates logs ingested by the SIEM.
  • Step 1: attacker inserts unexpected characters, malformed JSON, or invalid field values into log entries.
  • Step 2: parser errors may cause the SIEM to drop or misclassify logs, creating blind spots.
  • Step 3: attacker may also inject indicators that trigger low-value alerts, flooding the SOC.
  • Step 4: the SOC becomes distracted and less responsive to real alerts.
  • Step 5: impact: attacker can blend malicious activity into noisy logs and maintain persistence while the analyst team chases false positives.

Blue-team control review — what must stop the chain

  • Control 1: log normalization and validation at the pipeline layer before ingestion; bad logs are rejected or quarantined.
  • Control 2: parser resilience policies in the SIEM; unparseable logs are handled gracefully and forwarded to a quarantine index rather than dropped silently.
  • Control 3: alert noise detection; sudden spikes in low-confidence or low-fidelity alerts trigger a review of log quality rather than acceptance.
  • Control 4: source integrity rules; if a log source suddenly starts emitting malformed data, the source is quarantined and ticketed for investigation.
  • Control 5: separate low-fidelity index for untrusted log content; the SOC does not treat noisy logs as reliable detection data.

Evidence and telemetry to validate controls

  • Pipeline logs: ParseError count for a source rises above baseline.
  • SIEM ingest metrics: unparsed_event_count or droppped_event_count spikes.
  • Alert metrics: low-priority alert volume increases >200% for a source after a log schema change.
  • Quarantine index logs: malformed events captured with source metadata.

Purple-team validation commands

# Validate pipeline rejects malformed log events (authorized lab)
# Send intentionally malformed JSON log to the pipeline and confirm it lands in quarantine, not main SIEM

// Detect parser error spikes for a log source
PipelineMetrics
| where MetricName == "ParseErrorCount"
| summarize ErrorCount = sum(MetricValue) by Source, bin(TimeGenerated, 5m)
| where ErrorCount > 10
| order by ErrorCount desc

T6 — Migration Window Attack Exploiting Coverage Gaps

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Time an intrusion to the SIEM migration window when detection coverage is in flux and some rules are not yet tuned or translated, allowing the attacker to exploit the temporary gap between old and new detection contexts.

How the attacker would exploit it

  • Prerequisite: attacker has knowledge of the migration schedule or can infer it from reduced SOC activity or procurement notices.
  • Step 1: attacker begins reconnaissance or initial access during the migration phase.
  • Step 2: attacker identifies a gap in the new SIEM's rule set, such as a translated rule that is not yet deployed.
  • Step 3: attacker performs malicious actions that should have been detected by the old SIEM rule but are missed by the new SIEM.
  • Step 4: attacker may also leverage the old SIEM being deprioritized by the SOC during cutover.
  • Step 5: impact: attacker achieves longer dwell time and potentially completes data exfiltration or ransomware deployment.

Blue-team control review — what must stop the chain

  • Control 1: parallel ingestion of logs to both old and new SIEMs with dual alert monitoring.
  • Control 2: rule parity dashboard comparing alert counts and types between platforms.
  • Control 3: migration staging environment that validates translated rules against a historical log corpus before production promotion.
  • Control 4: extended parallel run period (minimum 90 days) for all critical rules and sources.
  • Control 5: dedicated migration SOC focus team that continues to monitor the legacy SIEM until the cutover is complete.

Evidence and telemetry to validate controls

  • Alert parity dashboard: per-rule match rate between old and new SIEMs.
  • Migration logs: source subset has 100% duplicated ingestion to both systems.
  • SOC incident queue: no increase in missed-detection incidents during migration.
  • Change control logs: each translated rule has a validation ticket or test record.

Purple-team validation commands

# Validate rule parity in lab migration environment
# Run the same test event through both legacy and new SIEMs and compare alert output

// Detect rule parity mismatches between legacy and new SIEM
RuleParityMetrics
| where LegacyAlertCount > 0
| extend Ratio = NewAlertCount * 1.0 / LegacyAlertCount
| where Ratio < 0.95 or Ratio > 1.05
| project RuleName, LegacyAlertCount, NewAlertCount, Ratio
| order by abs(Ratio - 1) desc

T7 — Cloud SIEM Tenant Misconfiguration

STRIDE category: Tampering, Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Exploit a misconfiguration in the cloud SIEM tenant — such as excluded sensitive sources, overly permissive retention settings, or inadequate access controls — to reduce visibility or expose sensitive log data.

How the attacker would exploit it

  • Prerequisite: attacker gathers intelligence on the SIEM architecture or finds cloud tenant misconfigurations via scanning.
  • Step 1: attacker identifies that certain sensitive sources are not ingested due to a bad inclusion filter or source tag.
  • Step 2: attacker uses those unmonitored assets as a pivot or exfiltration channel.
  • Step 3: attacker also looks for overly permissive access to the cloud storage backing the log archive.
  • Step 4: if retention is too short, attacker knows forensic evidence will disappear after the retention window.
  • Step 5: impact: attacker operates on assets that are effectively blind to the SOC, and historical evidence is unavailable.

Blue-team control review — what must stop the chain

  • Control 1: cloud SIEM source inventory reconciliation against the CMDB; every source must be accounted for.
  • Control 2: configuration drift detection for ingestion pipelines and retention policies.
  • Control 3: periodic access review for the cloud tenant and storage buckets.
  • Control 4: dedicated compliance rule that flags any source with sensitive=true that is not ingested.
  • Control 5: retention policy enforcement by policy-as-code and governance review.

Evidence and telemetry to validate controls

  • SIEM source inventory report: expected_sources - ingested_sources = missing_sources.
  • Cloud tenant configuration audit: changes to retention settings or source inclusion filters.
  • Storage bucket ACL and IAM policy audit: any non-approved principal with storage.objects.get on archive bucket.
  • Alert from compliance policy on missing sensitive sources.

Purple-team validation commands

# Validate source inventory reconciliation (authorized lab)
# Compare CMDB source list with SIEM ingested source list and report discrepancies

// Detect cloud SIEM sources missing from inventory
SourceInventory
| where Expected = true and Ingested == false
| project SourceName, SourceType, Owner
| order by SourceName

T8 — Cloud Provider Insider Access to Tenant Logs

STRIDE category: Information Disclosure
Risk: Low likelihood, High impact
Attacker objective: Leverage a cloud provider or third-party employee access to the SIEM tenant to read security telemetry, detection rules, or historical logs directly from the provider infrastructure.

How the attacker would exploit it

  • Prerequisite: attacker is a cloud provider insider or compromised provider support employee.
  • Step 1: attacker accesses the tenant through provider privileged access tooling.
  • Step 2: attacker reads ingested logs, admin audit logs, or the raw historical archive stored in provider storage.
  • Step 3: attacker exfiltrates detection rule definitions, alert content, and log details.
  • Step 4: attacker may combine this with knowledge of customer security controls to stage a more effective attack.
  • Step 5: impact: attacker learns the enterprise's detection coverage and can tailor future attacks to known gaps.

Blue-team control review — what must stop the chain

  • Control 1: contractual data access restrictions with the provider, including privileged access review and audit.
  • Control 2: if available, BYOK encryption of logs in cloud storage so provider staff cannot decrypt the data without the customer's key.
  • Control 3: provider access monitoring and access approval workflow for any privileged access to the tenant.
  • Control 4: minimize sensitive data in logs through tokenization or redaction prior to ingestion.
  • Control 5: periodic third-party audits of provider controls and SOC 2 Type II certification review.

Evidence and telemetry to validate controls

  • Provider access audit: employee access to tenant data and justification logs.
  • Key management logs: BYOK key usage only by authorized customer services.

STRIDE category: Repudiation
Risk: Medium likelihood, High impact

How the attacker would exploit it

  • Step 1: attacker edits a suppression query or archived alert filter to exclude their source IP, user, or event category.
  • Step 2: alerts matching the suppression query are still generated but not surfaced to the analyst queue.
  • Step 3: attacker continues activity while the SOC does not receive the relevant signal.
  • Step 4: suppression is maintained through the attack window until the abuse is discovered.

T12 — Log Pipeline MITM Between Forwarder and SIEM

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Intercept and modify logs in transit between the forwarder/pipeline and the cloud SIEM, either to redact attacker activity or to insert false data that misleads analysts.

How the attacker would exploit it

  • Prerequisite: attacker has control of a network segment or a proxy in the path from the forwarder to the SIEM ingestion endpoint.
  • Step 1: attacker intercepts the TLS stream and either terminates it with a forged certificate or exploits a weak cipher.
  • Step 2: attacker drops or modifies log events matching their activity.
  • Step 3: attacker reinjects the rest of the stream to the SIEM, leaving the pipeline intact.
  • Step 4: the SIEM never sees the malicious activity.
  • Step 5: impact: attacker achieves effective log deletion or alteration without touching the endpoint host.

Blue-team control review — what must stop the chain

  • Control 1: mTLS between forwarder/pipeline and cloud SIEM ingestion endpoint; both sides authenticate with certificates.
  • Control 2: certificate pinning or trust anchor verification on the forwarder and pipeline.
  • Control 3: network segmentation so the ingestion path is not easily intercepted by an on-path attacker.
  • Control 4: log integrity checks such as hash or HMAC sequence validation between forwarder and SIEM.
  • Control 5: forwarder-to-SIEM TLS configuration hardened against weak ciphers and improper certificate validation.

Evidence and telemetry to validate controls

  • Pipeline TLS logs: only valid client and server certificates exchanged over the ingestion path.
  • Forwarder logs: TLS handshake failures or certificate errors.
  • SIEM ingest logs: authenticated source identity matching the forwarder certificate.
  • Network flow logs: no unexpected intermediate device present on the ingestion path.

T13 — SIEM Forwarder Credential Reuse

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Extract a forwarder or ingestion service credential from a host or repository and use it to impersonate the forwarder, disable ingestion, or send bogus logs.

How the attacker would exploit it

  • Prerequisite: attacker finds a forwarder token/API key in plaintext in a config file, script, or repository.

  • Step 2: attacker may stop legitimate logs by exceeding rate limits, or inject false logs to confuse analysts.
  • Step 3: attacker can also use the token to enumerate the ingestion pipeline and find further misconfigurations.

  • Control 2: rotate forwarder credentials regularly and revoke old keys quickly.
  • Control 3: restrict forwarder API keys to specific source hosts or certificates.
  • Control 4: use short-lived credentials via workload identity or instance metadata where possible.
  • Control 5: secret scanning on forwarder config files and repositories.

Evidence and telemetry to validate controls

  • Forwarder config audit report: plaintext credentials found in host file system or repo.

Risk: Low likelihood, High impact

How the attacker would exploit it

  • Step 1: attacker injects bad IoCs (IPs, domains, hashes) into the feed.
  • Step 2: the SIEM ingest pipeline enriches alerts with those indicators.
  • Step 3: analysts receive a flood of low-quality alerts tied to the poisoned feed.
  • Step 4: real alerts may be missed or deprioritized due to analyst fatigue.
  • Step 5: impact: a denial-of-service effect on SOC attention, increasing attacker dwell time.

Blue-team control review — what must stop the chain

  • Control 1: threat intel source vetting and integrity validation before ingestion.
  • Control 2: thresholding on IOC-based alerts; do not automatically escalate low-confidence IOC matches.
  • Control 3: feed hash validation and anomaly detection — detect sudden feed volume or IOC churn spikes.
  • Control 4: analyst feedback loops to mark poisoned IOCs and remove them from active use.
  • Control 5: multiple independent threat feeds; do not rely on a single feed for high-confidence detection.

Evidence and telemetry to validate controls

  • Threat feed ingest logs: hash mismatches or output volume spikes.
  • SIEM alert metrics: sudden increase in alerts tied to a single threat feed.
  • Analyst feedback records: IOC false-positive markings.
