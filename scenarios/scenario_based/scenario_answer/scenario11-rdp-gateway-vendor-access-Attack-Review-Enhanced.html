Remote Desktop Gateway for Third-Party Vendor Access — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario11-rdp-gateway-vendor-access.md
Generated artifact: scenario11-rdp-gateway-vendor-access-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: 500 third-party vendors need time-limited access to internal systems for maintenance, but the current solution is a shared-credential jump server with no individual attribution, time-window enforcement, or session recording.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

The review focuses on third-party vendor access, shared-credential jump servers, PAM replacement architectures, session enforcement, and vendor endpoint compromise detection. Commands use placeholders and are intended for an owned lab or approved internal purple-team exercise only.

2. Highest-Risk Design Decision

The single most consequential design decision is whether vendor access is brokered through a PAM gateway with unique, time-limited identities and scoped target access, or whether it flows through a shared credential jump server that grants broad, unattributable internal reach.

The shared-credential jump server is dangerous because it removes attribution, prevents selective revocation, and means the jump server’s entire network adjacency becomes a single blast radius. The safe design is to use a PAM platform like CyberArk or BeyondTrust to provision per-vendor per-session access, enforce maintenance-window TTLs, record every session, and dynamically bind network access to the approved target systems defined in an ITSM change ticket. The dangerous alternative is to treat the jump server as a simple shared gateway and rely on vendor process discipline instead of technical enforcement.

3. Attack-Tree Legend

Every attack tree uses this convention:

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Shared Jump Server Credential Leak or Reuse

STRIDE category: Spoofing
Risk: Critical likelihood, Critical impact
Attacker objective: Use a leaked shared jump server password to access the corporate jump host and gain broad internal system access with no attribution and no selective revocation.

How the attacker would exploit it

  • Prerequisite: attacker obtains the shared credential from a vendor email, ticket system, dark web listing, or compromised vendor employee.
  • Step 1: attacker connects to the jump server via RDP/SSH using the shared account.
  • Step 2: the jump server accepts the credential because it is active and widely shared.
  • Step 3: the attacker uses the jump server’s network adjacency to connect to internal target systems.
  • Step 4: attacker performs reconnaissance and credential harvesting on internal systems.
  • Step 5: impact: attacker can move laterally or exfiltrate data while the organization cannot determine the identity of the active session.

Blue-team control review — what must stop the chain

  • Control 1: individual vendor identities provisioned through PAM; no shared accounts allowed.
  • Control 2: vendor access only through a managed PAM gateway that enforces authentication and session proxying.
  • Control 3: MFA for every vendor login; shared passwords alone are insufficient.
  • Control 4: automatic deprovisioning of vendor accounts after the maintenance window.
  • Control 5: SIEM alerts on concurrent sessions using the same vendor account or shared account pattern.

Evidence and telemetry to validate controls

  • PAM audit log: login_failure or login_success for shared account usage; unique vendor account required.
  • ZTNA logs: vendor endpoint identity and MFA status for each session.
  • SIEM: detection of same credential use from multiple vendor devices.
  • ITSM reconciliation: no active shared account should exist for vendor access.

| where TimeGenerated > ago(7d)
| project TimeGenerated, SourceIP, UserAgent, SessionID
| order by TimeGenerated desc

T2 — Compromised Vendor Endpoint Used in Authorized Session

STRIDE category: Spoofing, Elevation of Privilege
Risk: Critical likelihood, Critical impact
Attacker objective: Compromise a vendor laptop and use its valid PAM credentials/session context to perform malicious activity during an otherwise authorized vendor maintenance session.

How the attacker would exploit it

  • Prerequisite: attacker infects the vendor device with malware or steals the device after vendor authentication.
  • Step 1: vendor authenticates to PAM using valid credentials and MFA from the compromised device.
  • Step 2: attacker either hijacks the active session or waits for the vendor to initiate a new session.
  • Step 3: attacker uses the authenticated session to connect to approved target systems.
  • Step 4: attacker performs malicious actions while the session appears legitimate.
  • Step 5: impact: the attacker gains access to internal systems with valid vendor identity, making detection harder.

Blue-team control review — what must stop the chain

  • Control 1: ZTNA endpoint posture checks before session initiation, including AV status, patch level, disk encryption, and known bad processes.
  • Control 2: device identity binding; vendor device must be registered and approved for vendor access.
  • Control 3: session recording and command analytics to detect anomalous behavior even in authorized sessions.
  • Control 4: target-system process and file access monitoring during vendor sessions.
  • Control 5: automatic session termination on suspicious behavior such as unexpected process execution or file transfer.

Evidence and telemetry to validate controls

  • ZTNA logs: device posture pass/fail and device ID metadata.
  • PAM logs: device identifier associated with vendor account and session.
  • SIEM: unusual commands executed during the session compared to baseline vendor activity.
  • Endpoint logs on target systems: flagged process execution during a vendor session.

T3 — No Session Recording or Lost Audit Evidence

STRIDE category: Repudiation
Risk: Critical likelihood, Critical impact
Attacker objective: Conduct malicious activity during a vendor session and destroy or avoid session recording/audit evidence so forensic investigation cannot determine the exact actions taken.

How the attacker would exploit it

  • Prerequisite: vendor access is allowed and session recording is not enforced or recording storage fails silently.
  • Step 1: attacker performs sensitive actions during the session, such as credential harvesting or backdoor installation.
  • Step 2: attacker may also tamper with the recording configuration or storage if they can access PAM logs.
  • Step 3: no meaningful evidence exists after the session completes.
  • Step 4: the SOC is forced to rely on less granular system logs, making root cause analysis slower.
  • Step 5: impact: incident response is delayed and detection of the misuse is much harder.

Blue-team control review — what must stop the chain

  • Control 1: mandatory session recording for all vendor sessions; no bypass allowed.
  • Control 2: immutable storage for recordings with Object Lock or equivalent.
  • Control 3: health monitoring for recording delivery and storage.
  • Control 4: backup recording path and periodic recovery tests.
  • Control 5: alert on any session with no recording or incomplete recording data.

Evidence and telemetry to validate controls

  • PAM audit: session recording present and complete for each session ID.
  • Storage logs: recording object creation and retention metadata.
  • SIEM alert: session recording failures.
  • Periodic audit: sample sessions reviewed and recordings verified.

T4 — Vendor Access Outside Maintenance Window

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Use vendor credentials or an active session outside the approved maintenance window to access internal systems when no one is expecting it.

How the attacker would exploit it

  • Prerequisite: vendor credentials remain active beyond the approved maintenance window, or the vendor session is not forcibly terminated.
  • Step 1: attacker logs in or continues an existing session after the window end time.
  • Step 2: attacker takes advantage of reduced monitoring during off-hours.
  • Step 3: attacker performs malicious actions such as credential theft or data staging.
  • Step 4: attacker may maintain persistence by installing backdoors while fewer analysts are available.
  • Step 5: impact: attacker bypasses the control that limits vendor access to explicitly approved windows.

Blue-team control review — what must stop the chain

  • Control 1: PAM auto-terminates sessions at window end; requires no human action.
  • Control 2: service account TTLs are hard-coded to the approved maintenance window.
  • Control 3: SIEM alerts on any session after the ticket end time.
  • Control 4: time-based firewall rules or ZTNA policies enforce session access only during approved windows.
  • Control 5: ticket reconciliation engine automatically flags active sessions with expired or missing tickets.

Evidence and telemetry to validate controls

  • PAM log: session end time > ticket window end.
  • Change ticket log: approved window start/end values.
  • SIEM alert on out-of-window session start or continuation.
  • Firewall log: session terminated due to time-based rule.

T5 — Broad Lateral Movement from Jump Server on Flat Adjacency

STRIDE category: Lateral Movement, Elevation of Privilege
Risk: Critical likelihood, Critical impact
Attacker objective: Use the jump server’s broad internal network adjacency to reach multiple internal systems beyond the vendor’s approved target scope.

How the attacker would exploit it

  • Prerequisite: jump server is placed on a network segment with access to multiple internal systems and lacks per-session target restrictions.
  • Step 1: attacker logs into the jump server and scans internal IP ranges reachable from it.
  • Step 2: attacker uses RDP, SSH, SMB, or remote management tools to connect to adjacent systems.
  • Step 3: attacker steals credentials, dumps AD data, or compromises other servers.
  • Step 4: attacker may use the jump server as a persistence platform.
  • Step 5: impact: multiple internal systems are compromised from a single vendor access point.

Blue-team control review — what must stop the chain

  • Control 1: network micro-segmentation so each vendor session can only reach approved target IPs.
  • Control 2: dynamic firewall rules applied for each session based on the change ticket scope.
  • Control 3: jump server network placed in a dedicated DMZ with minimal adjacency.
  • Control 4: host-based firewalls on target systems only allow connections from the jump server for specific vendor service accounts.
  • Control 5: internal network flow monitoring detects unauthorized lateral connections from the jump server.

Evidence and telemetry to validate controls

  • Network flow logs: jump server connecting to target systems not in the approved target list.
  • Firewall logs: session-specific dynamic rules allowing only approved IPs.
  • PAM audit: session target list for the approved ticket.
  • Intrusion detection alerts on jump server lateral activity.

T6 — Over-Privileged Vendor Session with Admin-Level Access

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Use vendor session privileges that are broader than required for the maintenance task to install backdoors, harvest credentials, or modify sensitive configurations.

How the attacker would exploit it

  • Prerequisite: vendor is granted an account on the target system with administrative or broad permissions.
  • Step 1: attacker uses those permissions to enumerate local service accounts and credentials.
  • Step 2: attacker may create new privileged accounts or dump password hashes.
  • Step 3: attacker persists by modifying scheduled tasks, services, or startup scripts.
  • Step 4: attacker may also install monitoring or exfiltration agents.
  • Step 5: impact: internal systems are compromised beyond the approved maintenance action, and the attacker has a foothold for later campaigns.

Blue-team control review — what must stop the chain

  • Control 1: least-privilege target accounts; vendor accounts have only the permissions needed for the specific maintenance task.
  • Control 2: just-in-time escalation if higher privileges are temporarily required; elevated rights are automatically revoked after the task.
  • Control 3: command filtering and application control on target systems for vendor accounts.
  • Control 4: audit of privileged actions during vendor sessions; suspicious privilege-changing commands generate alerts.
  • Control 5: post-session review of target system account and ACL changes.

Evidence and telemetry to validate controls

  • PAM logs: target account privilege level and allowed commands.
  • Target system event logs: changes to local user accounts, service installs, and group membership.
  • SIEM: privilege escalation commands detected during vendor session.
  • Change ticket vs. actual actions comparison.

T7 — Former Vendor Employee with Unrevoked Access

STRIDE category: Spoofing
Risk: High likelihood, High impact
Attacker objective: Use residual vendor credentials after a contract has ended to access the jump server and internal systems.

How the attacker would exploit it

  • Prerequisite: vendor account provisioning and deprovisioning are not tightly integrated with contract lifecycle.
  • Step 1: former vendor employee retains credentials and uses them to log into the PAM or jump server.
  • Step 2: attacker may pass any MFA if tokens are still valid or if the account still has an active device registration.
  • Step 3: attacker uses the account to connect to internal systems.
  • Step 4: attacker performs unauthorized maintenance or data theft under the guise of a valid vendor account.
  • Step 5: impact: the attacker capitalizes on an access control gap created by poor offboarding.

Blue-team control review — what must stop the chain

  • Control 1: automated vendor account deprovisioning tied to contract end dates in ITSM.
  • Control 2: daily reconciliation of active vendor accounts against vendor contracts.
  • Control 3: revalidation of vendor device registrations and access certificates upon contract termination.
  • Control 4: SIEM alerts for vendor accounts used after contract end.
  • Control 5: mandatory short-lived vendor account lifetimes, requiring new access requests for each engagement.

Evidence and telemetry to validate controls

  • Identity governance report: vendor account active beyond contract end date.
  • PAM account audit: account last used after contract termination.
  • ITSM reconciliation results: orphan vendor accounts.
  • SIEM alert on stale vendor account usage.

T8 — Credential Harvesting During Vendor Session

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Use an authorized vendor session to retrieve internal credentials from target systems, then use those credentials for follow-on access outside the vendor session.

How the attacker would exploit it

  • Prerequisite: vendor target account has access to sensitive configuration files, service account stores, or local credential caches.
  • Step 1: attacker searches for credential files on the target system (e.g., appsettings.json, web.config, credentials.xml).
  • Step 2: attacker copies or types those credentials into an external location.
  • Step 3: attacker uses the harvested credentials to authenticate directly to other internal systems.
  • Step 4: the vendor session may remain legitimate while the attacker pivots with internal creds.
  • Step 5: impact: the attacker escalates beyond the vendor session and can maintain persistence with broader internal access.

Blue-team control review — what must stop the chain

  • Control 1: target system accounts for vendor sessions should not have access to sensitive credential stores unless necessary.
  • Control 2: file integrity and sensitive file access monitoring on target systems.
  • Control 3: command filtering for vendor sessions to prevent cat, type, or registry read operations on credential files.
  • Control 4: alert on vendor session actions touching high-value paths such as C:\ProgramData\Secrets or /etc/credentials.
  • Control 5: post-session credential rotation if sensitive data was accessed.

Evidence and telemetry to validate controls

  • Windows Event 4663 or Linux auditd logs on sensitive file reads during vendor session.
  • PAM command audit showing file access or copy commands.
  • SIEM alert on high-risk file access patterns during vendor sessions.
  • Credential vault access logs showing unusual target system access patterns.

T9 — Backdoor Persistence Installed During Maintenance

STRIDE category: Tampering
Risk: High likelihood, High impact
Attacker objective: Use vendor maintenance access to install a persistent backdoor on the target system, ensuring continued access after the vendor session ends.

How the attacker would exploit it

  • Prerequisite: vendor has sufficient permissions to create scheduled tasks, services, or startup items on the target system.
  • Step 1: attacker creates a new scheduled task, service, or rc.local entry during the session.
  • Step 2: attacker may also install a web shell or reverse shell payload.
  • Step 3: attacker uses the backdoor after the vendor session ends to regain access.
  • Step 4: attacker may cover tracks by deleting the task or cleaning logs.
  • Step 5: impact: persistence is established in the environment from an otherwise authorized maintenance session.

Blue-team control review — what must stop the chain

  • Control 1: only allow vendor commands required for the task; block creation of new services or scheduled tasks unless explicitly approved.
  • Control 2: host-based IDS on target systems detects new service or scheduled task creation.
  • Control 3: post-session integrity scan for new persistence mechanisms.
  • Control 4: SIEM correlation between vendor session activity and persistence creation events.
  • Control 5: periodic baseline and verification of allowed software on target systems.

Evidence and telemetry to validate controls

  • Windows Event 4698/4702: scheduled task created or changed during vendor session.
  • Windows Event 7045: new service installed.
  • PAM audit: vendor commands that created persistence objects.
  • SIEM alert on unexpected persistence mechanism creation.

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Transfer internal data from the target systems through the jump server to an external storage or command-and-control destination.

How the attacker would exploit it

  • Prerequisite: jump server or target systems have outbound egress to the internet.
  • Step 1: attacker uses the vendor session to copy data to the jump server.
  • Step 2: attacker initiates outbound transfer via HTTPS, SFTP, or DNS tunneling.
  • Step 3: attacker may use legitimate tools such as curl, scp, or cloud storage uploads.

Blue-team control review — what must stop the chain

  • Control 4: data loss prevention on the jump server for sensitive file transfers.
  • Control 5: audit of all file transfers performed during vendor sessions.

Evidence and telemetry to validate controls

  • PAM file transfer logs: upload/download events, file names, destination hosts.

T14 — Change Ticket Manipulation to Authorize Broader Access

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Modify or create a fraudulent change ticket that expands vendor access scope, then use that ticket to justify a broader PAM session.

How the attacker would exploit it

  • Prerequisite: attacker has access to the ITSM system or can social-engineer an approver.
  • Step 1: attacker submits a ticket with overbroad targets or requests out-of-scope access.
  • Step 2: attacker obtains approval from an unwitting manager or creates a fraudulent ticket.
  • Step 3: PAM provisions access based on the ticket scope.
  • Step 4: attacker performs malicious activity using the expanded scope.
  • Step 5: impact: the attack looks approved and may evade detection if ticket reconciliation is weak.

Blue-team control review — what must stop the chain

  • Control 1: dual approval for vendor change tickets affecting high-sensitivity systems.
  • Control 2: automated ticket scope validation against a whitelist of approved target systems for vendor type.
  • Control 3: SIEM correlation of ticket approval and session activity; mismatches flagged immediately.
  • Control 4: periodic audit of change tickets with scope versus actual session targets.
  • Control 5: vendor manager sign-off plus security review for any request outside standard maintenance templates.

Evidence and telemetry to validate controls

  • ITSM audit logs: change ticket edits, approvals, approver identities.
  • PAM provisioning logs: target list sourced from ticket fields.
  • SIEM correlation: session target IPs not matching ticket-approved targets.
  • Manual audit: ticket scope vs. session activity comparison.
