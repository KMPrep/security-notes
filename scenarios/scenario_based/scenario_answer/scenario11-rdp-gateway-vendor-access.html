Scenario 11 — Remote Desktop Gateway for Third-Party Vendor Access

Scenario: 500 third-party vendors need time-limited access to internal systems for maintenance. The current solution is a jump server with shared credentials.

Candidate questions:

  • What immediate risks does a shared-credential jump server create?
  • What controls replace shared credentials — walk me through your architecture?
  • How do you enforce time-limited, least-privilege, session-recorded access?
  • What monitoring detects a vendor using access outside its authorized maintenance window?
  • What residual risk remains if the vendor's own endpoint is compromised?

3.4 Highest-risk boundary crossing

The jump server to internal network boundary is the critical risk point. In the current shared-credential model, this boundary has no individual attribution — every session looks identical regardless of which of the 500 vendors (or an attacker impersonating them) initiated it. Once inside the jump server, there is no firewall enforcement of which internal systems each vendor is permitted to reach; the jump server's network adjacency determines the blast radius. A vendor contracted to maintain a file server can potentially reach the domain controller if they share the same subnet. The fix is not just replacing credentials — it's enforcing network micro-segmentation so that each vendor session's network reach is bounded to their specific authorized systems, not the entire adjacent network.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

Managing 500 vendors manually — provisioning accounts, defining scope, setting TTLs — is a human bottleneck that generates errors and stale access. At Google scale, vendor access is fully automated: ServiceNow intake triggers a Terraform or API-driven workflow that creates a time-limited account in CyberArk, sets the approved target list in the firewall policy, sends the vendor a one-time access link, and schedules automatic deprovisioning at window end. No PAM admin manually creates any account. The entire workflow is auditable, repeatable, and version-controlled. Access requests outside business hours go through a PagerDuty-integrated emergency approval workflow, not a "call the admin" process.

8.2 Fleet-wide impact analysis

A shared jump server credential for 500 vendors is a fleet-wide critical vulnerability. If that credential is compromised, every internal system reachable from the jump server is simultaneously at risk from a threat actor who can operate with complete impunity — no attribution, no audit trail, no obvious anomaly to detect. The blast radius is not "one vendor is compromised" — it is "the jump server becomes an attacker-controlled pivot into the entire internal network." At Google scale, with potentially thousands of vendor relationships, this problem is even more acute; individual accountability for every access is non-negotiable.

8.3 Eliminating human bottlenecks

The bottleneck in vendor access management is the approval workflow: vendor submits ticket, internal manager approves, PAM admin creates account, session proceeds. At 500 vendors with frequent maintenance windows, this creates a queue. The automation solution is risk-tiered self-service: low-risk vendors (HVAC, cleaning company with IT asset management system access) get pre-approved access windows via a policy-as-code rule without per-session human approval. High-risk vendors (network infrastructure, security tools) require named human approval. The system automatically classifies based on the target system's sensitivity tier in the CMDB.

8.4 Controls that break at scale

Manual audit of vendor session recordings is impossible at 500 vendors. You cannot have a human watch every session video. At scale, session recording review must be automated: ML-based anomaly detection on keystroke patterns (Securonix or similar) flags sessions with high-risk command sequences for human review. Only flagged sessions require analyst attention. Similarly, manually checking whether each vendor's access was deprovisioned after contract end fails at scale — automated ITSM-to-PAM reconciliation running hourly is the only way to maintain de-provisioning hygiene.

ANSWER SUMMARY — 90-second version

Use this to open your answer before going deep on any section.

"The shared-credential jump server is one of the most dangerous configurations in enterprise security — not because the jump server itself is complex, but because shared credentials eliminate the two things security depends on: attribution and revocation. When 500 vendors share a single password, you cannot tell which vendor was on the system at 3am, and you cannot revoke access from a specific compromised vendor without locking everyone out.

My immediate priorities: replace shared credentials with individual identities in a PAM platform like CyberArk, with time-limited accounts that auto-expire at the end of each maintenance window; require MFA for every vendor session; proxy all sessions so vendors never see target credentials; record every session keystroke and video to immutable storage.

For detection, the key signals are sessions outside the approved maintenance window, access to systems not in the approved scope, and suspicious command patterns like credential harvesting or reverse shell tools.

Residual risk I'd document: vendor endpoints are unmanaged, so a compromised vendor laptop can ride an authenticated session. I accept that risk with PAM session recording as the detective control and network micro-segmentation to limit the blast radius — even a compromised session can only reach the specific authorized targets, not the whole internal network."

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What immediate risks does a shared-credential jump server create?

Three critical risks: first, zero attribution — every action on every connected system is attributed to the shared account, making forensic investigation nearly impossible. Second, no revocation — if one vendor is compromised, you cannot revoke access from that vendor without locking out all 499 others. Third, no access boundary enforcement — the shared account's network reach is the jump server's network adjacency, meaning a vendor contracted to maintain a file server can potentially pivot to any system the jump server can reach. The shared credential also never expires because someone would always be using it for an active session, so it becomes a permanent, unrotated, high-privilege credential.

Follow-up 2: How do you enforce time-limited, least-privilege, session-recorded access?

Time-limited: the PAM platform (CyberArk, BeyondTrust) provisions a unique account per vendor per engagement with a hard TTL equal to the approved maintenance window plus a 30-minute buffer. The account auto-expires; there is no manual step to revoke it. Least-privilege: two-layer enforcement — the PAM session policy restricts which target systems the vendor can connect to (sourced from the change ticket); on the target system, the account has only the permissions required for the specific task (read-only for diagnostics, specific service restart permissions for maintenance). Session recording: CyberArk PSM or BeyondTrust records full video and keystroke for every session, forwarded to S3 with Object Lock. Vendors are contractually notified that sessions are recorded.

Follow-up 3: What monitoring detects a vendor using access outside its authorized maintenance window?

The detection chain: PAM has the approved window from the ServiceNow change ticket. If the vendor session continues past the window end, PAM automatically terminates it and fires an alert. If a new session is attempted after the window, the auto-expired account prevents login — that failed login attempt is the detection signal. The harder case is a vendor who authenticates during the window but remains active past it — PAM session duration monitoring covers this. A secondary control is a ServiceNow reconciliation job that runs every 15 minutes: any active PAM session with no corresponding open change ticket fires a Critical alert in the SIEM.

Follow-up 4: You mentioned PAM session recording as a key control. What happens if that control fails — what is your next layer?

If PAM session recording fails to capture a session (storage outage, PAM agent crash), my next forensic layer is the target system's own audit logs — Windows Security Event Logs (4624, 4688, 4663) and Linux syslog capture the logon event, process executions, and file access events during the session. These are shipped to the SIEM independently of the PAM platform. The third layer is network flow logging from the jump server VLAN: I may not know every command, but I can see which IPs were accessed, bytes transferred, and session duration. I'd also detect the session recording failure itself via a PAM health alert, investigate why, and treat any session without a recording as suspect until forensics are complete.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is compromising the vendor's endpoint before they connect. If I can install a keylogger or RAT on a target vendor's laptop — via a phishing email to the vendor employee or a supply-chain compromise in vendor-distributed software — I ride their legitimate authenticated session. From the enterprise's perspective, the session looks completely normal: valid credentials, valid MFA, valid change ticket, access to authorized systems. My detection is behavioral anomaly within the session: the vendor connected to do a "database index rebuild" but the session shows PowerShell executions, credential file reads, and a connection to an unexpected internal IP. Session recording and command analytics catch this even when the authentication chain is fully legitimate.

COMMON MISTAKES TO AVOID

  • Proposing a "better shared password" instead of individual accounts — Saying "just use a stronger shared password with rotation" fundamentally misunderstands the attribution and revocation problems. Individual identities are the only solution.
  • Treating MFA as the complete solution — MFA prevents credential-only attacks but doesn't help if the vendor's device is already compromised. MFA plus session recording plus least-privilege network access is the layered answer.
  • Not addressing network micro-segmentation — Replacing shared credentials is necessary but not sufficient if the jump server still has broad network access. A compromised session on an individual account can still pivot broadly if there's no network policy enforcing "this vendor can only reach IP 10.1.2.3."
  • Ignoring the vendor off-boarding problem — Focusing only on access provisioning while ignoring that vendor accounts that outlive contracts are a major source of unauthorized access. De-provisioning must be automated and tied to ITSM contract lifecycle.
  • Overlooking session recording as a forensic requirement — Without session recordings, a vendor insider can do almost anything during an authorized session with no forensic evidence. Recording is not optional for high-privilege vendor access.
  • Not mentioning the change ticket reconciliation — Any active vendor session must be correlated with an open, approved change ticket. Sessions without tickets are unauthorized access, not a policy violation. Failing to detect this gap means the attacker only needs to skip the ticket step, not bypass any technical control.
  • Underestimating the compromised vendor endpoint risk — Saying "ZTNA posture checks solve the vendor endpoint problem" overestimates posture check reliability. A sophisticated attacker will pass posture checks while being compromised. Session recording and behavioral detection are the realistic residual controls.
