Cloud Identity Federation — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario12-cloud-identity-federation.md
Generated artifact: scenario12-cloud-identity-federation-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Azure AD is the source of truth for identity while engineers authenticate to Google Workspace through cloud federation, SCIM provisioning, and OIDC trust.

1. Review Objective

This document reviews the threat model from two lenses:

  1. Attacker lens: How an attacker would realistically exploit each identified threat, including prerequisites, reconnaissance steps, tool usage, and escalation path.
  2. Blue-team lens: What controls must stop or detect the attack, where each control is enforced, and what telemetry proves the control is firing.

The review focuses on federation trust boundaries, Azure AD token theft, SCIM provisioning risk, deprovisioning lag, federation misconfiguration, break-glass accounts, and cross-system detection.

2. Highest-Risk Design Decision

The highest-risk design decision is trusting Azure AD as the authoritative issuer for Google Workspace sessions without independent Google-side control of device posture, token lifetime, or privileged claim mapping. If Google accepts any valid Azure AD JWT/SAML assertion as proof of identity, then compromise of Azure AD, the SCIM sync pipeline, or Azure AD federation app credentials directly results in cross-cloud account takeover. The safer design is to enforce short-lived tokens, explicit audience validation, per-app conditional access for Google federation, and Google-side context-aware access in addition to Azure AD controls.

3. Attack-Tree Legend

Every attack tree uses this convention:

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — AiTM Phishing for Azure AD Token Replay to Google

STRIDE category: Spoofing
Risk: Critical likelihood, Critical impact
Attacker objective: Capture a valid Azure AD federated token or session cookie and replay it to Google Workspace to obtain access without needing the user's credentials again.

How the attacker would exploit it

  • Prerequisite: attacker hosts an AiTM proxy such as Evilginx2 or Modlishka and lures a user to authenticate to Azure AD.
  • Step 1: victim enters credentials and completes MFA in the proxy site.
  • Step 2: AiTM proxy captures the Azure AD session token or id_token.
  • Step 3: attacker replays the valid token to Google Workspace federated login endpoint.
  • Step 4: attacker gains a Google Workspace session as the victim.
  • Step 5: impact: the attacker bypasses MFA and exfiltrates Drive/Docs data or performs unauthorized admin actions.

Blue-team control review — what must stop the chain

  • Control 1: phishing-resistant MFA (FIDO2, Windows Hello for Business) on Azure AD to prevent credential replay.
  • Control 2: Azure AD conditional access blocking unknown or risky sign-ins for federated users.
  • Control 3: browser-origin verification and anti-phishing protection on the IdP feed.
  • Control 4: Google Workspace BeyondCorp context-aware access requiring verified device posture.
  • Control 5: real-time detection of token replay or unusual federated login patterns.

Evidence and telemetry to validate controls

  • Azure AD sign-in logs: riskDetail contains atp or phishing and authenticationProcessingDetails flags.
  • Azure AD MFA logs: only FIDO2 authentications accepted for Google federation users.
  • Google Workspace login audit: federated login from unexpected IP or device fingerprint.
  • SIEM rule: repeated federated login attempts from same token or user within short interval.

Purple-team validation commands

# Validate Azure AD phishing-resistant MFA enforcement for federated users
Get-AzureADPolicy -Filter "DisplayName eq 'RequireFido2ForAll'"

// Detect suspicious federated sign-ins that may indicate replay or AiTM
SigninLogs
| where AppDisplayName == "Google Workspace"
| where RiskLevelAggregated == "high" or RiskDetail contains "tokenReplay"
| project TimeGenerated, UserPrincipalName, IPAddress, DeviceDetail, ConditionalAccessStatus
| order by TimeGenerated desc

T2 — Azure AD PRT Theft Used to Obtain Federated Google Token

STRIDE category: Spoofing
Risk: Critical likelihood, Critical impact
Attacker objective: Steal a Primary Refresh Token (PRT) from a compromised Windows device and use it to silently obtain a fresh Azure AD id_token that federates to Google Workspace.

How the attacker would exploit it

  • Prerequisite: attacker compromises a managed Windows device and extracts the Azure AD PRT from the device using tools like Mimikatz or proprietary memory dumping.
  • Step 1: attacker uses the PRT to request a fresh Azure AD access token and id_token silently.
  • Step 2: attacker sends the id_token to Google Workspace federated authentication.
  • Step 3: Google Workspace accepts the token as valid and creates a session.
  • Step 4: attacker operates in Google Workspace without interactive authentication.
  • Step 5: impact: attacker gains stealthy access and may stay within federated session lifetimes.

Blue-team control review — what must stop the chain

  • Control 1: require device-based conditional access and Windows Hello for Business for Azure AD logins.
  • Control 2: implement Microsoft Defender for Endpoint to detect PRT extraction techniques on endpoints.
  • Control 3: block refresh token use from anomalous devices or locations.
  • Control 4: enforce short token lifetimes and session revocation on risky detection.
  • Control 5: monitor for abnormal token issuance patterns in Azure AD.

Evidence and telemetry to validate controls

  • Microsoft Defender ATP alerts for suspicious LSA secrets access or PRT harvesting.
  • Azure AD sign-in logs: refresh token grant requests from an endpoint not previously seen for the user.
  • Conditional Access logs: token issuance denied for non-compliant device.
  • SIEM correlation: Azure AD refresh token use followed by Google federated login.

Purple-team validation commands

# Validate Azure AD refresh token use is blocked for non-compliant devices
Get-AzureADPolicy -Filter "DisplayName eq 'RequireCompliantDeviceForGoogle'"

SigninLogs
| where TokenIssuerType == "AzureAD"
| where AppDisplayName == "Google Workspace"
| where AuthenticationRequirement == "NotRequired"
| project TimeGenerated, UserPrincipalName, DeviceDetail, IPAddress, ConditionalAccessStatus

T3 — Azure AD Global Admin Compromise Alters Federation Config

STRIDE category: Elevation of Privilege
Risk: Critical likelihood, Critical impact
Attacker objective: Compromise an Azure AD Global Admin and change Google federation settings or claim mappings so that any Azure AD account can gain elevated Google Workspace privileges.

How the attacker would exploit it

  • Prerequisite: attacker obtains Global Admin credentials or session cookie via phishing, password spray, or session hijacking.
  • Step 1: attacker logs into Azure AD portal as Global Admin.
  • Step 2: attacker modifies federation application settings, claim mappings, or Conditional Access policy.
  • Step 3: attacker may change the roles or groups claim to map to Google admin roles.
  • Step 4: attacker logs in to Google Workspace using a low-privileged account with forged claims.
  • Step 5: impact: broad or high-privilege escalation across the Google environment.

Blue-team control review — what must stop the chain

  • Control 1: just-in-time Global Admin access via PIM with approval and MFA.
  • Control 2: Azure AD privileged access workstations for admin sign-in.
  • Control 3: audit and alert on federation app registration changes and claim mapping updates.
  • Control 4: limit user-assigned admin roles to minimal set and require security review for any changes.
  • Control 5: protect Global Admin with FIDO2 and require passwordless sign-in where possible.

Evidence and telemetry to validate controls

  • Azure AD audit logs: AddCredential, AddPassword, Update operations on app registrations and Conditional Access policies.
  • Azure AD PIM logs: Global Admin activation requests and approvals.
  • Google Workspace Admin audit: changes to SSO claim or IDP settings.
  • SIEM alert: unexpected federation mapping modification outside change control window.

Purple-team validation commands

# Validate Azure AD app registration change audit is enabled
Search-UnifiedAuditLog -StartDate (Get-Date).AddDays(-7) -EndDate (Get-Date) -RecordType AzureActiveDirectory | where { $_.Operations -match "AddPassword|Update|AddCredential" }

T4 — SCIM Provisioning Token Leak or Over-Privileged SCIM Service Account

STRIDE category: Elevation of Privilege
Risk: Critical likelihood, Critical impact
Attacker objective: Steal the Azure AD SCIM service account token and use it to create Google Workspace accounts with admin privileges or alter group memberships.

How the attacker would exploit it

  • Prerequisite: SCIM token is stored insecurely in source code, CI/CD variables, or a configuration file.
  • Step 1: attacker obtains the SCIM token or client secret via repo scan or compromised pipeline.
  • Step 2: attacker calls the Google Workspace SCIM API to create users or modify groups.
  • Step 3: attacker assigns admin groups or creates a rogue super admin account.
  • Step 4: attacker uses the admin account to disable audit trails or persist access.
  • Step 5: impact: the cloud provisioning pipeline becomes a direct path to admin takeover.

Blue-team control review — what must stop the chain

  • Control 1: SCIM service account scoped to minimal provisioning permissions, not admin rights.
  • Control 2: rotate SCIM secrets frequently and store them in vaults like Azure Key Vault.
  • Control 3: enforce least-privilege API access and restrict SCIM clients by IP/managed identity.
  • Control 4: monitor SCIM token use and alert on admin attribute changes.
  • Control 5: require separate approval and review for any elevated SCIM operations.

Evidence and telemetry to validate controls

  • SCIM provisioning logs: source IP, API client ID, action type, changedProperties.
  • Cloud secret vault access logs: reads to SCIM token secret.
  • Google Workspace audit log: account creation or group assignment with isAdmin=true.
  • SIEM alert: SCIM operation from unexpected environment or non-approved client.

Purple-team validation commands

# Validate SCIM secrets are stored in a vault and not in plain text repos
grep -R "google_scim" . | grep -E "client_secret|password"

AuditLogs
| where OperationName == "SCIMProvisioning" and tostring(Properties) contains "isAdmin"
| project TimeGenerated, TargetResources, InitiatedBy, Status

T5 — De-Provisioning Lag Between HR Termination and Google Suspension

STRIDE category: Spoofing
Risk: High likelihood, High impact
Attacker objective: Use a terminated employee’s still-active Google Workspace account during the provisioning lag window to access email and Drive data.

How the attacker would exploit it

  • Prerequisite: HR termination event is delayed or SCIM sync fails to immediately disable the Azure AD and Google accounts.
  • Step 1: attacker uses the terminated employee’s credentials or existing Google session.
  • Step 2: Azure AD disables either fails or takes hours to propagate to Google.
  • Step 3: attacker uses Google Workspace during the window of stale access.
  • Step 4: attacker exfiltrates data or covers tracks with legitimate-looking access.
  • Step 5: impact: a former employee or attacker maintains unauthorized access after separation.

Blue-team control review — what must stop the chain

  • Control 1: real-time HR webhook or Azure AD lifecycle policy triggers immediate account disable.
  • Control 2: SCIM provisioning success is monitored and any delay >1 hour alerts identity team.
  • Control 3: Google workspace session revocation occurs immediately when SCIM marking suspended=true.
  • Control 4: periodic reconciliation of HR, Azure AD, and Google auditing ensures account state matches.
  • Control 5: require manual account disable in Google if automated SCIM fails.

Evidence and telemetry to validate controls

  • HR audit logs: termination timestamp and effective date.
  • Azure AD account status change events and sign-in denial events after disable.
  • Google Workspace user status logs showing account suspended and recent login times.
  • SIEM reconciliation alerts for active Google accounts without a corresponding active HR record.

Purple-team validation commands

# Validate deprovisioning reconciliation across HR, Azure AD, and Google
# Ensure any terminated user is disabled in both Azure AD and Google within 1 hour

// Detect deprovisioning lag
HRTerminationEvents
| join kind=inner (AzureADUserStatus | where AccountEnabled == false) on UserPrincipalName
| join kind=leftouter (GoogleWorkspaceUsers | where Suspended == false) on UserPrincipalName
| where TimestampDiff('hour', HRTerminationEvents.TimeGenerated, GoogleWorkspaceUsers.LastSyncTime) > 1
| project UserPrincipalName, HRTerminationEvents.TimeGenerated, GoogleWorkspaceUsers.LastSyncTime

T6 — Federated Token Replay Due to Expired or Misvalidated JWT

STRIDE category: Spoofing
Risk: High likelihood, High impact
Attacker objective: Replay a stolen or expired federated token to Google Workspace when Google does not properly validate the token lifetime or audience.

How the attacker would exploit it

  • Prerequisite: attacker captures an Azure AD id_token or SAML assertion from a legitimate session.
  • Step 1: attacker reuses the token against Google Workspace auth endpoint.
  • Step 2: if Google accepts expired or wrong-audience tokens, access is granted.
  • Step 3: attacker may harvest data or create persistence before token expiry is checked.
  • Step 4: impact: token replay bypasses normal authentication.

Blue-team control review — what must stop the chain

  • Control 1: Google Workspace must enforce strict exp, aud, iss, and nonce validation.
  • Control 2: Azure AD token lifetimes must be short and refresh tokens revoked on risk.
  • Control 3: use Google Workspace session policies to reduce session reuse from stale tokens.
  • Control 4: alert on token replay behavior in login audit.
  • Control 5: periodically test federation using malformed or expired tokens to verify rejection.

Evidence and telemetry to validate controls

  • Google Workspace login audits: rejected logins due to invalid token claims.
  • Azure AD sign-in logs: token grant events and refresh token usage patterns.
  • Federation health checks: test failures from invalid token replay.
  • SIEM rules: repeated login attempts with same token ID from different client apps.

T7 — Azure AD Conditional Access Misconfiguration Bypasses Federation Security

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Exploit a misconfigured Conditional Access policy or legacy auth exemption in Azure AD to obtain federated Google access without required MFA or device compliance.

How the attacker would exploit it

  • Prerequisite: Azure AD CA policy has exclusions, legacy auth allowances, or incorrect user/device groups.
  • Step 1: attacker authenticates using a federated account from a bypassed client or device.
  • Step 2: Azure AD issues a token without enforcing security controls.
  • Step 3: attacker uses that token to access Google Workspace.
  • Step 4: impact: federated access is granted with reduced assurance.

Blue-team control review — what must stop the chain

  • Control 1: apply CA policies to all users in the Google federation app, not just selected groups.
  • Control 2: block legacy auth for federation and require modern authentication.
  • Control 3: remove policy exclusions and require MFA for federated logins.
  • Control 4: audit CA policy changes and exemptions.
  • Control 5: alert on any federated login where ConditionalAccessStatus is notApplied.

Evidence and telemetry to validate controls

  • Azure AD CA logs: policy decisions for the Google federation app and user scope.
  • Sign-in logs: conditionalAccessStatus for federated logins.
  • SIEM alert: federated login with policy bypass or no MFA.
  • Audit reports: CA policy exclusions or legacy auth detections.

Purple-team validation commands

SigninLogs
| where AppDisplayName == "Google Workspace"
| where ConditionalAccessStatus != "success"
| project TimeGenerated, UserPrincipalName, ConditionalAccessStatus, ClientAppUsed

T8 — Google Workspace Break-Glass Account Compromise Bypasses Federation

STRIDE category: Spoofing
Risk: Critical likelihood, Critical impact
Attacker objective: Compromise a local Google Workspace break-glass admin account that is not federated through Azure AD and use it to manage SSO, audit logs, and data access directly.

How the attacker would exploit it

  • Prerequisite: attacker discovers break-glass account credentials or a weak recovery path.
  • Step 1: attacker authenticates directly to Google Workspace with the local admin account.
  • Step 2: attacker changes federation settings, disables audit logging, or creates new admin accounts.
  • Step 3: attacker can maintain access independent of Azure AD state.
  • Step 4: impact: Google Workspace compromise bypasses the trusted Azure AD federation chain.

Blue-team control review — what must stop the chain

  • Control 1: require phishing-resistant MFA and hardware tokens for break-glass accounts.
  • Control 2: restrict break-glass logins to specific IP ranges and require just-in-time enablement.
  • Control 3: monitor and alert on any break-glass account use immediately.
  • Control 4: store credentials in a hardened vault with strict access controls.
  • Control 5: enforce rotating break-glass credentials after each use.

Evidence and telemetry to validate controls

  • Google Workspace login audit: break-glass account sign-ins and login contexts.
  • Google admin audit: direct changes to SSO or admin role assignments.
  • Vault access logs: retrieval of break-glass credentials.
  • SIEM alert: break-glass account use without declared incident.

T9 — Federation Misconfiguration Allows Claim Mapping Escalation

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Take advantage of a misconfigured Azure AD claim mapping in Google Workspace so that a normal user receives elevated Google privileges or admin roles.

How the attacker would exploit it

  • Prerequisite: federation claim rules are too permissive or map Azure AD groups/roles incorrectly to Google roles.
  • Step 1: attacker authenticates via Azure AD and receives a federated token.
  • Step 2: Google Workspace maps the token claims to an overly privileged role.
  • Step 3: attacker accesses sensitive data or admin controls.
  • Step 4: impact: privilege escalation occurs through configuration rather than credential theft.

Blue-team control review — what must stop the chain

  • Control 1: validate federation claim rules and audience values in Google Workspace SSO configuration.
  • Control 2: limit group-to-admin mapping to explicit approved groups only.
  • Control 3: test federation mappings in a staging environment before production.
  • Control 4: audit any mapping or SSO setting changes and require security review.
  • Control 5: monitor for unexpected privilege level changes after federated login.

Evidence and telemetry to validate controls

  • Google Workspace SSO configuration audit: claim mapping and group mapping changes.
  • Google login audit: role or privilege level assigned during federated login.
  • Azure AD token inspection logs: claims passed in the id_token.
  • SIEM correlation: user granted admin-level access without a matching identity group membership change.

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Use an Azure AD guest account with Google Workspace license and group membership to access data beyond the intended external collaboration scope.

How the attacker would exploit it

  • Prerequisite: Azure AD guest user is assigned to a Google Workspace group with broad access.
  • Step 1: attacker authenticates via Azure AD as the guest.
  • Step 2: Google Workspace federates the login and grants access based on group membership.
  • Step 3: attacker accesses internal engineer data or shared resources.
  • Step 4: impact: external guest account becomes a pivot point into sensitive content.

Blue-team control review — what must stop the chain

  • Control 1: treat guest accounts as high-risk and restrict their group memberships.
  • Control 2: enforce separate conditional access policies for guest users.
  • Control 3: audit Google Workspace licenses and ensure guests only get minimal access.
  • Control 4: monitor guest federated logins and access patterns.
  • Control 5: require periodic guest entitlement reviews.

Evidence and telemetry to validate controls

  • Azure AD guest account report: group memberships and external collaborator status.
  • Google Workspace audit: guest login and resource access events.
  • Conditional Access log: guest user access decisions.
  • SIEM alert: guest user accessing internal/shared engineer groups.

T12 — Compromised Azure AD Federation App Credentials Create Arbitrary Tokens

STRIDE category: Spoofing
Risk: Critical likelihood, Critical impact
Attacker objective: Compromise the Azure AD app registration credentials used by Google Workspace federation and use them to mint arbitrary id_tokens or modify token issuance settings.

How the attacker would exploit it

  • Prerequisite: attacker steals a client secret or certificate for the Azure AD app used in Google federation.
  • Step 1: attacker uses stolen app credentials to request tokens from Azure AD.
  • Step 2: attacker may also change the token issuance settings for the app.
  • Step 3: attacker obtains valid assertion tokens for arbitrary users.
  • Step 4: attacker logs into Google Workspace as those users.
  • Step 5: impact: federation trust is effectively broken and the attacker can impersonate any federated user.

Blue-team control review — what must stop the chain

  • Control 1: protect Azure AD app registration secrets in a hardware-backed key vault.
  • Control 2: rotate client secrets and certificates regularly.
  • Control 3: restrict app registration management to a few protected admins.
  • Control 4: monitor app registration secret creation and usage.
  • Control 5: require certificate-based authentication for federation apps when possible.

Evidence and telemetry to validate controls

  • Azure AD audit logs: AddPassword, AddKeyCredential, and secret rotation for the federation app.
  • App usage logs: token requests from unexpected client IPs or application IDs.
  • SIEM alert: new secret or certificate created on the Google federation app registration.
  • Conditional Access logs: token issuance events for the app.

STRIDE category: Spoofing
Risk: Medium likelihood, High impact
Attacker objective: Exploit an IT helpdesk error where a former employee is re-enabled in Azure AD and the Google Workspace account is reactivated or remains active through SCIM sync.

Blue-team control review — what must stop the chain

  • Control 4: retain audit trails of reactivation events in HR, Azure AD, and Google.
  • Control 5: require a safety check before federated access is restored.

Evidence and telemetry to validate controls

T14 — Log Gaps Between Azure AD and Google Audit Trails Hide Federated Abuse

STRIDE category: Repudiation
Risk: High likelihood, High impact
Attacker objective: Use the lack of cross-system log correlation or missing audit detail to hide the attack path between Azure AD token issuance and Google Workspace access.

How the attacker would exploit it

  • Prerequisite: the organization does not correlate Azure AD sign-ins, SCIM events, and Google Workspace login audits.
  • Step 1: attacker performs a federated login or creates an admin via SCIM.
  • Step 2: the suspicious event appears in one system but is not linked to the other.
  • Step 3: security analysts cannot reconstruct the full kill chain.
  • Step 4: attacker continues activity longer before detection.
  • Step 5: impact: investigation is delayed and remediation is incomplete.

Blue-team control review — what must stop the chain

  • Control 1: ingest Azure AD, Google Workspace, SCIM, and HR logs into a single SIEM.
  • Control 2: implement correlation rules that join federated login events with token issuance and account provisioning.
  • Control 3: preserve audit logs with strong retention and immutability.
  • Control 4: alert when a federated session lacks a preceding Azure AD token issuance event.
  • Control 5: perform regular attack-path reconstruction exercises across both systems.

Evidence and telemetry to validate controls

  • SIEM correlation alerts linking Azure AD sign-ins to Google Workspace federated sessions.
  • Audit log completeness reports for Azure AD and Google Workspace ingestion.
  • SCIM sync logs showing user provisioning events paired with HR actions.
  • Incident response timelines showing linked events across both systems.
