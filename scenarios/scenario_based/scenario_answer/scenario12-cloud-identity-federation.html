Scenario 12 — Cloud Identity Federation (Azure AD to Google Workspace)

Scenario: HR data in Azure AD is the source of truth. Engineers use Google Workspace. SCIM provisioning and OIDC federation are configured. Users authenticate to Google using Azure AD credentials.

Candidate questions:

  • What are the trust chain attack surfaces in this federated identity model?
  • How do you model a compromised Azure AD token being replayed against Google Workspace?
  • What controls detect de-provisioning lag between HR termination and account disablement?
  • How do you monitor for federated login anomalies (token replays, IdP misuse)?
  • What is your residual risk for break-glass accounts in each system?

3.4 Highest-risk boundary crossing

The highest-risk boundary is the trust relationship between Azure AD (IdP) and Google Workspace (SP). Google Workspace is configured to accept any valid JWT signed by Azure AD as proof of identity. This means that any attacker who can obtain a valid Azure AD token — via phishing, AiTM attack, Primary Refresh Token theft, or Azure AD compromise — can authenticate to Google Workspace without touching Google's own security controls. Google implicitly trusts Azure AD completely; it cannot independently verify whether the user's device was compliant, whether MFA was truly performed, or whether the token was issued in response to a legitimate user interaction. The security of Google Workspace is entirely downstream from Azure AD's security posture.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale, federation configuration is a change-controlled, version-controlled artifact. The Google Workspace OIDC federation config (issuer, client_id, claim mappings, group mappings) is stored in Terraform and deployed via CI/CD with mandatory PR review. Any drift between Terraform state and deployed config triggers an immediate alert. SCIM provisioning rules are also code: Azure AD attribute flow rules and Google Workspace schema mappings are version-controlled. No human manually edits federation config in the console — the UI is effectively read-only for production. This eliminates the misconfiguration attack surface entirely from human error.

8.2 Fleet-wide impact analysis

A single Azure AD Global Admin account compromise in a federated architecture is a full-fleet incident. At Google scale with 50,000+ engineers all using Google Workspace federated through Azure AD, a stolen Global Admin PRT can silently modify the federation trust to issue tokens that map all engineers to super-admin privilege. Before detection, every engineer's Workspace account could be accessed. The blast radius is not one compromised account — it is the entire Google Workspace tenant's data, configuration, and security controls. This is why the Global Admin count must be minimal (3–5 named individuals), each with FIDO2, PIM-gated access, and with Microsoft Defender for Identity monitoring their every action.

8.3 Eliminating human bottlenecks

The most critical human bottleneck in federated identity is off-boarding: "HR closes the ticket, Identity team gets the email, someone manually disables the Azure AD account, someone else checks if the Google account was suspended." At Google scale, this human chain is both too slow and too unreliable. The solution is a fully automated, event-driven provisioning pipeline: Workday fires a webhook on employee termination; Azure AD Entra ID Lifecycle Workflows process it within minutes; SCIM propagates to Google within the same pipeline run. The SLA is sub-one-hour, automated, with reconciliation monitoring. No human is in the loop for standard off-boarding.

8.4 Controls that break at scale

Manual review of SCIM provisioning errors works for 10 users per week. At Google scale with thousands of joiners/movers/leavers per week, SCIM errors become noise unless you have automated triage: critical errors (termination failures) are immediately escalated; non-critical errors (attribute sync delay) are batched for daily review. Similarly, the break-glass account usage review process ("CISO manually checks quarterly if break-glass was used") breaks at scale and in a high-velocity organization — automated alerting on every use is the only acceptable approach.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What are the trust chain attack surfaces in this federated identity model?

There are five trust chain links, each a potential attack surface: (1) HR system to Azure AD — if HR data is tampered with, wrong users get provisioned; (2) Azure AD as IdP — the entire Google Workspace trust depends on Azure AD's integrity; (3) the OIDC/SAML federation configuration itself — a misconfigured claim mapping can grant super-admin to any user; (4) the SCIM service account — over-privileged service account with static credentials can create arbitrary Google accounts; and (5) the token in transit — OIDC tokens are JWTs; if the Azure AD signing key is compromised, an attacker can forge tokens for any identity. Each link must be independently hardened because compromise of any single link breaks the entire chain.

Follow-up 2: How do you model a compromised Azure AD token being replayed against Google Workspace?

An attacker performs an AiTM phishing attack, proxying the Azure AD login flow and stealing the post-MFA session cookie or the OIDC id_token directly. They replay this to Google's federated login endpoint. Google validates the token signature (valid — it was legitimately issued by Azure AD), validates the expiry (valid — attack happens within the token's lifetime), and establishes a Google Workspace session for the victim. Detection: Azure AD Identity Protection flags the anomalous sign-in (new device, different IP than usual); Google Workspace login audit shows first-time login from new device/location. Prevention: FIDO2 MFA binds the authentication to the origin domain — an AiTM proxy cannot complete FIDO2 authentication because it cannot satisfy the origin check.

Follow-up 3: What controls detect de-provisioning lag between HR termination and account disablement?

A three-layer monitoring approach: (1) real-time SCIM sync monitoring — the SCIM provisioning log is checked every 15 minutes; any termination event that hasn't resulted in a Google account suspension within 1 hour fires a Critical alert; (2) daily reconciliation — automated job queries HR terminations in the last 24 hours and checks Azure AD and Google account status for each; any active account for a terminated employee is escalated; (3) weekly entitlement review — broader check that any Google Workspace account without a corresponding active Azure AD account is investigated (covers edge cases like SCIM missing a record entirely). The goal is zero accounts active more than 4 hours after HR termination for standard off-boarding.

Follow-up 4: You mentioned break-glass accounts. What is your residual risk position for them?

Break-glass accounts are a necessary risk — the alternative, having no recovery path if federation breaks, is worse. My risk acceptance conditions: Azure AD break-glass accounts use cloud-only identities with no Conditional Access (so they work even if CA policies are misconfigured); they use hardware FIDO2 keys stored in a physical safe with dual-person integrity for access; they are IP-restricted to the corporate office network; and any usage fires an immediate Critical alert to the CISO and incident response lead. Google break-glass accounts follow the same pattern. Both sets of accounts are tested quarterly with a witnessed drill. The acceptance condition is that these accounts are effectively single-use — after any use, the account is rotated before the next quarter.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is targeting a developer who uses TOTP-based MFA (not FIDO2) with an AiTM phishing kit. I'd send a convincing "Google Workspace security alert" email, proxy the Azure AD login through Evilginx2, capture the post-MFA session cookie, and have immediate access to the engineer's Google Workspace account. From there, I can access Google Drive, read email, and potentially find service account keys or other credentials in documents. I cannot reach Google admin settings with an engineer account, but IP theft and lateral movement opportunities are significant. The hardening I'd apply: mandate FIDO2 for all engineers (not just admins), implement Google Workspace context-aware access requiring device certificates so a stolen session from a non-enrolled device is blocked even if the token is valid.

COMMON MISTAKES TO AVOID

  • Treating federation as "just SSO" — Federation between two enterprise identity systems is a full trust relationship where the security of the service provider (Google Workspace) is entirely dependent on the IdP (Azure AD). Framing it as a convenience feature misses the security gravity.
  • Focusing only on Google Workspace security controls — Google Workspace's built-in controls are irrelevant if Azure AD is compromised. The security posture must be analyzed from the IdP outward; hardening only the SP is insufficient.
  • Ignoring de-provisioning lag as a risk — Many candidates address the provisioning (onboarding) flow but skip the de-provisioning (offboarding) timing problem. A one-hour window post-termination is a real and common attack vector.
  • Treating TOTP MFA as sufficient against AiTM — TOTP is phishable. For engineers with access to sensitive Google Workspace data, FIDO2 is the correct answer. Saying "we have MFA" without specifying the type signals shallow thinking.
  • Missing the SCIM service account as an attack vector — The SCIM provisioning account often has full user management rights in Google Workspace and is frequently a static service account with a long-lived secret. This is frequently overlooked.
  • Not addressing break-glass account risk — Every federated identity architecture needs break-glass accounts, but each one is a high-value target. Not mentioning how they are stored, monitored, and rotated is an incomplete answer.
  • Ignoring cross-system log correlation — A federated identity attack spans two systems. A candidate who proposes monitoring only Azure AD logs or only Google Workspace logs will miss attacks that are only visible when correlating both. The SIEM must ingest and correlate both.
