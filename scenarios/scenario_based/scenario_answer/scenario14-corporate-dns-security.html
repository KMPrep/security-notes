Scenario 14 — Corporate DNS Architecture Security Review

Scenario: The enterprise uses split-horizon DNS with internal resolvers, a public authoritative DNS, and recently added DNS-over-HTTPS for endpoints. Recursive resolvers are on-prem.

Candidate questions:

  • What threats does DNS tunneling present and what controls mitigate it?
  • How do you model a DNS cache poisoning attack in this architecture?
  • What is the threat model for DNS-over-HTTPS from a visibility perspective?
  • What controls prevent internal DNS zone data from being exfiltrated?
  • How do you monitor for DNS-based C2 communication patterns?

3.4 Highest-risk boundary crossing

The highest-risk boundary crossing is the path from internal recursive resolvers to the public internet. This boundary is traversed for every external DNS query and is bidirectional — internal resolvers send queries outbound and receive responses inbound. An attacker can exploit this boundary in two directions: inbound, by poisoning the cache with forged responses (Kaminsky-style attack targeting predictable transaction IDs); outbound, by running a malicious domain with DNS tunneling to receive queries that carry exfiltrated data encoded in subdomains. The introduction of DoH compounds this because it wraps DNS in HTTPS, making traditional UDP-53-based DNS inspection blind — if the DoH provider is not the enterprise's own controlled resolver, the enterprise loses all visibility into endpoint DNS traffic.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale (10,000+ domains, 1B+ DNS queries/day), DNS zone management must be entirely automated through Terraform or a proprietary zone management API — no manual record edits via console. All DNS record changes are committed to a Git repository, peer-reviewed, and applied via CI/CD pipeline with automated validation (no dangling CNAMEs, no wildcard records without justification, DNSSEC signing automation). DNSSEC key rollovers are automated via RFC 8901 CDS/CDNSKEY mechanism. RPZ feed updates are pulled automatically from threat intelligence APIs with a validation step before deployment.

8.2 Fleet-wide impact analysis

A poisoned internal resolver cache at Google scale doesn't affect 100 users — it affects 100,000 engineers making HTTPS connections to internal services. A single forged A record for an internal service hostname could redirect all engineers to an attacker-controlled server collecting credentials or SAML tokens. A registrar hijack replacing NS records for google.com would affect a billion users' ability to reach all Google services. At fleet scale, a DoH misconfiguration that routes all endpoint DNS to an attacker-controlled resolver creates a universal man-in-the-middle position for all HTTPS traffic (via DNS-rebinding or certificate mis-issuance if combined with CA compromise).

8.3 Eliminating human bottlenecks

Manual DNS change approval for 10,000 domains with hundreds of daily changes is impossible. Google-scale DNS management requires policy-as-code guardrails that enforce: no CNAME to external targets without allowlisting, mandatory DNSSEC for all public zones, mandatory TTL minimums for security-critical records, and automated alerts for any record change outside the CI pipeline. The human bottleneck for registrar changes (which require human verification) is addressed by pre-authorizing a specific set of domain management service accounts that are the only identities allowed to interact with the registrar API — with hardware token MFA and IP restriction.

8.4 Controls that break at scale

Manual AXFR access list management — manually maintaining per-zone AXFR ACLs across 10,000 domains fails; requires automated zone management platform that enforces AXFR restrictions as default. Per-domain CT log monitoring — at 10,000 domains, alerting on every certificate issuance creates unmanageable volume; requires tiered monitoring (critical domains = real-time alert, long-tail = daily digest + automated CNAME check). Static RPZ blocklists — at billion-query scale, static RPZ lists are stale within minutes of a new C2 domain appearing; requires streaming threat intelligence integration (VirusTotal API, Mandiant TI) with sub-minute RPZ update capability. Human review of DNS tunneling alerts — at 1B queries/day, per-query analysis requires ML-based DNS anomaly detection (Chronicle's ML models) rather than static threshold rules.

ANSWER SUMMARY — 90-second version

Use this to open your answer before going deep on any section.

"Split-horizon DNS with DoH creates a complex threat surface that touches confidentiality, integrity, and availability simultaneously. Let me walk through the key threats.

On confidentiality: the internal DNS zones contain a complete map of our infrastructure — every server name, IP, and service. Zone transfer restriction and split-horizon validation prevent that from leaking publicly. The bigger risk is DNS tunneling outbound — an attacker with a foothold can exfiltrate data through DNS queries to a malicious domain, bypassing DLP entirely. Controls: force all endpoints through a corporate recursive resolver, implement query length and entropy-based detection, and block direct outbound UDP/53 at the firewall.

On integrity: DNS cache poisoning and registrar hijacking are my top concerns. Cache poisoning is mitigated by DNSSEC validation on resolvers and source port randomization. Registrar hijacking is mitigated by domain lock, hardware MFA on the registrar account, and NS change monitoring.

The DoH wrinkle is that it moves DNS from a monitorable UDP protocol to encrypted HTTPS. My control is to deploy an internal DoH resolver and block external DoH providers by IP at the firewall — I keep the encryption benefit while maintaining visibility.

Residual risk I accept: a zero-day in the resolver software, which I mitigate with network segmentation, and a BGP hijack of my authoritative provider — mitigated with multi-provider DNS failover."

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What is the threat model for DNS-over-HTTPS from a visibility perspective?

DoH encrypts DNS queries in HTTPS, eliminating the enterprise's ability to perform DNS inspection at the network layer — a traditionally reliable detection and blocking point. An attacker can use a DoH provider not in the enterprise blocklist as a covert C2 channel. My control is threefold: deploy an internal DoH resolver (e.g., Cloudflare Gateway or internal Unbound with DoH enabled) and push its address via Managed Chrome / MDM policy, block all known external DoH provider IPs at the firewall (Google 8.8.8.8, Cloudflare 1.1.1.1, Quad9, etc.), and log all HTTPS connections to those IP ranges to catch bypass attempts. This preserves the encryption benefit for endpoint-to-resolver traffic while keeping the corporate resolver as the authoritative inspection point.

Follow-up 2: How do you model a DNS cache poisoning attack in this architecture?

A Kaminsky-style cache poisoning attack exploits predictable transaction IDs or source ports in the resolver's outbound queries. I model it as: attacker races forged UDP responses against the legitimate authoritative server response, winning if the transaction ID matches. Mitigations: ensure resolver uses randomized source ports (RFC 5452) and 0xffff-entropy transaction IDs (BIND 9+ defaults), enable DNSSEC validation so forged responses without valid signatures are rejected, and deploy DNS cookies (RFC 7873) for resolver-to-authoritative communication. Detection: any unexpected A/CNAME record change for a monitored hostname triggers an immediate alert, and I baseline resolver cache contents for high-value hostnames.

Follow-up 3: What controls prevent internal DNS zone data from being exfiltrated?

Zone transfer (AXFR) restriction is table stakes — only named secondary server IPs are authorized, enforced at the BIND ACL level. But the subtler risk is enumeration via brute-force subdomain queries, which doesn't require AXFR. Mitigations: enable BIND minimal-responses to prevent additional section data leakage, rate-limit queries per source IP on the authoritative server, do not publish PTR records for sensitive internal ranges externally, and validate split-horizon configuration weekly with automated external probes that query internal-only hostnames from outside the network and confirm NXDOMAIN responses. DNSSEC also helps here — it prevents zone walking on unsigned zones but actually enables it via NSEC on signed zones; use NSEC3 with opt-out to prevent zone enumeration.

Follow-up 4: You mentioned RPZ sinkholing. What happens if that control fails — what is your next layer?

If the RPZ feed is stale or the C2 domain is too new to be listed, the endpoint will successfully resolve and connect. The next layer is network-level blocking: a next-gen firewall or Zscaler proxy that does TLS inspection and URL categorization — even if DNS resolved, the HTTP connection to an uncategorized or newly-registered domain can be blocked. The layer after that is endpoint EDR (CrowdStrike Falcon), which detects the C2 beacon behavior at the process level regardless of whether DNS resolved successfully. Finally, SIEM anomaly detection on beaconing patterns — periodic connections from the same endpoint to the same IP at regular intervals — catches C2 even when the domain is clean. Defense-in-depth means no single control is a dependency.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is DNS-over-HTTPS to a provider not yet in the enterprise's blocklist — Nextdns.io, AdGuard DNS, or a self-hosted DoH server on a legitimate-looking domain over port 443. This bypasses the firewall UDP/53 block entirely. If MDM policy enforcement is weak (BYOD, non-managed apps), the endpoint may ignore the internal DoH configuration. The C2 traffic looks identical to HTTPS web browsing. My detection response is behavioral: even with DoH, the endpoint must connect somewhere over TCP/443, so I look for HTTPS connections to newly-registered domains, domains with short TTLs in CT logs, and endpoints with high HTTPS connection rates to low-reputation domains.

COMMON MISTAKES TO AVOID

  • Treating DNS security as purely a network problem. DNS security spans registrar, authoritative, recursive resolver, endpoint, and monitoring — candidates who only talk about firewall ACLs miss 80% of the attack surface.
  • Not addressing the DoH visibility gap. Candidates who recommend "block port 53" without addressing DoH/DoT encryption leave a significant blind spot. The enterprise resolver must support DoH and be enforced via MDM policy.
  • Conflating DNSSEC with encryption. DNSSEC provides integrity (authentication), not confidentiality. It does not encrypt DNS queries — that requires DoT or DoH. Many candidates say "enable DNSSEC for privacy," which is incorrect.
  • Forgetting the registrar as an attack surface. Domain registrar account compromise is a critical risk that bypasses all technical DNS controls. Most candidates don't mention registrar account security.
  • Not using NSEC3 for signed zones. Signing zones with NSEC (instead of NSEC3) enables zone walking — an attacker can enumerate all records in the zone by walking the NSEC chain. Always use NSEC3 with opt-out for large zones.
  • Proposing blocking all external DNS as a viable control. This sounds simple but breaks DNS over DoH/DoT in modern browsers, breaks split-tunnel VPN scenarios, and creates significant operational complexity. The right approach is policy enforcement on managed endpoints.
  • Ignoring the bootstrapping problem for DoH. How does the endpoint discover the internal DoH resolver address without first making a DNS query? This is solved via MDM/Group Policy pushing the DoH URL directly — candidates who don't acknowledge this gap reveal surface-level knowledge.
  • Not addressing subdomain takeover. A dangling CNAME pointing to a deprovisioned cloud resource is a common, high-severity finding that candidates rarely mention when discussing DNS security.
