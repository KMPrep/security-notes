Scenario 15 — Patch Management Program for Heterogeneous Fleet

Scenario: 80,000 endpoints (Windows, macOS, Linux, mobile) and 5,000 servers require patching within SLA. The team has a 30-day SLA for critical CVEs and 90 days for high CVEs.

Candidate questions:

  • How do you threat model the vulnerability exposure window for a zero-day?
  • What controls compensate when patching SLA cannot be met for a critical system?
  • How do you handle systems that break when patched (compatibility risk vs security risk)?
  • What is your monitoring strategy to confirm patch deployment and verify success?
  • What residual risk documentation do you maintain for exceptions?

3.4 Highest-risk boundary crossing

The highest-risk boundary crossing is between the public internet (where CVE PoC code is published) and the unpatched internal fleet. The critical insight is that the exposure window is not the 30-day patch SLA — it's the gap between PoC publication and patch deployment. For high-profile CVEs (Log4Shell, ProxyLogon, EternalBlue), weaponized exploit code is available within hours of disclosure. A 30-day SLA that begins at patch release is already a 30-day window after the vendor had time to develop a fix — the actual exposure window often starts weeks earlier when the vulnerability was being exploited as a zero-day. This means compensating controls must be in place immediately upon CVE disclosure, not only after patch deployment.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At 80,000 endpoints, manual patch approval and scheduling is impossible. Google's internal systems (Borgmon for fleet monitoring, automated patch deployment via Borg/Fleet Management) automate the entire pipeline: vulnerability scanner output feeds directly into a prioritization engine that computes risk scores (CVSS × asset criticality × exposure), generates patch tickets automatically, assigns owners from the CMDB, and triggers deployment via configuration management (Ansible/Puppet/Chef) without human initiation. Exceptions require automated justification workflow: system owner submits compensating controls, risk score is auto-calculated, CISO auto-approves below threshold or escalates above it. No human manually tracks 80,000 endpoints.

8.2 Fleet-wide impact analysis

A single unpatched critical CVE on an internet-facing server at Google scale isn't a single server problem — if the CVE is wormable (EternalBlue, Log4Shell), a single compromised host can laterally scan and exploit thousands of internal systems within hours. At 80,000 endpoints, a 5% unpatched rate for a critical CVE means 4,000 vulnerable machines — enough to sustain ransomware propagation even with network segmentation. A compromised WSUS server distributing a malicious payload to 80,000 machines simultaneously is a total fleet compromise event. The blast radius calculation for patch management decisions must account for wormability: a CVSS 9.8 wormable CVE must be treated differently from a CVSS 9.8 non-wormable CVE.

8.3 Eliminating human bottlenecks

The human bottleneck at scale is the patch approval process — security team manually reviewing and approving each patch before deployment. Replace with: automated trust in vendor-signed patches for OS-level patches (Microsoft, Apple, Red Hat), automated ring-based deployment with built-in quality gates, and automated rollback triggered by deployment failure metrics. Human involvement remains only for: emergency out-of-band patches with unusual risk profiles, patches affecting payment or regulatory-scoped systems requiring documented approval, and any patch to security infrastructure (SIEM, EDR, firewalls) which requires separate change management.

8.4 Controls that break at scale

Manual vulnerability triage — a security analyst reviewing 50,000 vulnerability findings per week is impossible; requires automated risk-based prioritization (Tenable Lumin, Qualys TruRisk) that surfaces the top 1% of findings requiring human action. Per-system patch scheduling — scheduling patches for 80,000 systems individually creates a coordination nightmare; replace with asset-group-based deployment waves driven by CMDB classification. Exception review meetings — weekly governance meetings for patch exceptions fail when there are 500 exceptions; requires automated exception management with auto-expiry, auto-escalation, and dashboard visibility for risk owners. Static scan cycles — weekly or monthly scans miss newly deployed systems and rapidly-deployed containers; requires continuous authenticated scanning with agent-based real-time inventory.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: How do you threat model the vulnerability exposure window for a zero-day?

A zero-day has three distinct phases with different threat models: pre-disclosure (attacker has working exploit, defender has no knowledge — detection only via behavioral EDR), post-disclosure pre-patch (CVE published, PoC may exist, no vendor fix — compensating controls critical), and post-patch pre-deployment (patch exists but not deployed — SLA clock running). The riskiest phase is post-disclosure pre-patch because the PoC is public and automated exploitation begins within hours. My compensating controls during this phase: WAF/IPS virtual patching (Palo Alto Threat Prevention rule for the specific CVE pattern), network segmentation of affected systems, emergency isolation of internet-facing instances of the vulnerable software, and enhanced EDR behavioral monitoring for exploitation artifacts.

Follow-up 2: What controls compensate when patching SLA cannot be met for a critical system?

The compensating control stack, in order of effectiveness: (1) network segmentation — place the system on an isolated VLAN with no inbound access except from known business-required sources; (2) WAF/IPS virtual patching — deploy a vendor-provided IPS signature that blocks the specific exploit pattern at the network layer; (3) enhanced monitoring — increase EDR alerting sensitivity on the system, add specific detection rules for the CVE's exploitation artifacts; (4) access restriction — require explicit approval for all interactive access, log and alert on all connections; (5) documented exception — record the compensating controls, expected patch date, and risk owner sign-off in the exception database with a mandatory expiry date. No exception is permanent.

Follow-up 3: How do you handle systems that break when patched?

This is a direct conflict between availability and security, and the default answer must be: security wins, with escalation. My process: the patch engineer documents the compatibility issue with reproduction steps and the affected system owner. If it's a critical CVE, the timeline for finding a workaround is 7 days maximum — after which the system is segmented or taken offline rather than left unpatched and internet-facing. For non-critical CVEs, a 90-day exception is granted while the compatibility issue is resolved. If the system cannot be patched due to vendor-imposed constraints (OT, medical devices), it goes on the formal exception register with compensating network controls and a replacement timeline. The exception is reported to the risk committee monthly.

Follow-up 4: You mentioned ring-based deployment. What happens if that control fails — what is your next layer?

If the ring-based deployment fails (pilot failures unreported, quality gate bypassed, WSUS misconfig), the next layer is the post-deployment vulnerability scan — Tenable/Qualys scans the fleet within 72 hours of a patch deployment and confirms CVE resolution. If the scan shows the CVE still open on systems that should have received the patch, an automated alert fires and triggers manual investigation. The layer after that is the SIEM SLA tracking rule, which independently monitors open CVE age from the vulnerability scanner and fires an alert at day 20 (10-day warning before SLA breach), regardless of the patch management system's reported status. Defense-in-depth means the scan is independent validation of the deployment system.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is targeting a system in the patch exception database — those are explicitly marked as unpatched and the compensating controls are often inadequate. I'd query the CMDB or vulnerability scanner output (if I have internal access) for systems with open critical CVEs and active exceptions, then cross-reference with internet-facing exposure. Alternatively, I'd target third-party libraries not tracked in the SBOM — Java, OpenSSL, Log4j — which are often present in production but not in the vulnerability scanner's scope because the team doesn't know they're installed. The fix is mandatory SBOM for all applications and agent-based software inventory (Tanium/CrowdStrike) to catch everything the scanner misses.

COMMON MISTAKES TO AVOID

  • Treating the 30-day SLA as the actual exposure window. The exposure window starts at CVE disclosure (or earlier for zero-days), not at patch release. The SLA clock running from patch release ignores the pre-patch exposure period.
  • Forgetting third-party libraries in the patch scope. Log4Shell hit organizations that were fully compliant on OS patching because they didn't track third-party library versions. SBOM and third-party software inventory are part of patch management.
  • Not having a compensating control plan for unpatched systems. "We'll patch it in 30 days" is not an acceptable answer without compensating controls (WAF, segmentation, enhanced monitoring) for the exposure window.
  • Ignoring the patch management infrastructure as an attack surface. WSUS/SCCM/Jamf compromise is a critical path to fleet-wide malicious payload distribution. Most candidates discuss patching endpoints but don't secure the patch delivery mechanism.
  • Not distinguishing wormable from non-wormable CVEs. A wormable RCE (EternalBlue pattern) at CVSS 9.8 requires emergency response within hours. A local privilege escalation at CVSS 9.8 requires patching but is not immediately externally exploitable. Treating them identically wastes emergency response resources.
  • Underestimating ring-based deployment risk. Candidates often propose ring deployment as a complete solution without noting that it adds 5-7 days to patch deployment time — during which the pilot ring is protected but the broad fleet is still exposed.
  • Not addressing patch rollback as an attack vector. An attacker with access to patch management infrastructure can rollback patches to reintroduce vulnerabilities. Rollback operations should be logged, alerted, and require explicit re-approval.
