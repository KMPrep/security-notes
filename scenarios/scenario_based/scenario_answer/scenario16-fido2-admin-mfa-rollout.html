Scenario 16 — Hardware Security Key (FIDO2) Rollout for Admins

Scenario: You are replacing TOTP-based MFA with FIDO2 hardware keys for 500 privileged administrators accessing critical infrastructure and production systems.

Candidate questions:

  • What residual phishing risk remains even with FIDO2 in place?
  • How do you handle account recovery when a hardware key is lost?
  • What controls prevent a cloned or counterfeit hardware key from being enrolled?
  • How do you monitor for suspicious FIDO2 enrollment events?
  • What is your threat model for a supply-chain attack on the hardware key batch?

3.4 Highest-risk boundary crossing

The highest-risk boundary crossing in FIDO2 is not the authentication itself — it is the account recovery and key re-enrollment process. FIDO2 hardware authentication is extremely strong; the vulnerability surface collapses to the recovery path. An attacker who can social engineer the IT helpdesk into resetting an admin's MFA and registering a new key has completely bypassed the hardware security model without ever touching a YubiKey. This is why Google's own Titan key deployment mandates that key re-enrollment for privileged admins requires in-person identity verification at a Google office with a government ID — no remote re-enrollment is permitted. The recovery boundary is where the hardware security model fails.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale, FIDO2 policy enforcement is driven by Terraform-managed IdP configuration — admin group authentication policies are version-controlled and deployed via CI/CD, with policy-as-code validation ensuring no MFA factors other than FIDO2 hardware keys are permitted for privileged roles. The FIDO2 key inventory is managed in the CMDB with an automated daily sync to IdP credential records; any discrepancy triggers an automated alert and quarantine. Key enrollment for new admins uses a self-service portal that enforces attestation validation automatically — no human manually checks the MDS3 database.

8.2 Fleet-wide impact analysis

At 500 privileged admins managing 100,000+ production systems, a single compromised admin account doesn't just affect one system — it provides entry into the IAM plane, enabling the attacker to elevate to additional admin accounts, modify audit logging, alter firewall rules, and access customer data across all tenants. A supply chain attack on a batch of 500 hardware keys simultaneously compromises all privileged admin credentials across the entire infrastructure. The blast radius is total: it's not a data breach, it's a full infrastructure takeover. This is why FIDO2 deployment for admins must be paired with PAW enforcement — the key is only as strong as the device it's plugged into.

8.3 Eliminating human bottlenecks

The human bottleneck is the MFA re-enrollment process — at 500 admins with frequent key rotation (annual), key losses (estimated 5%), and new admin onboarding, the in-person enrollment requirement creates operational friction. The solution is not to eliminate in-person verification (the security anchor) but to streamline it: designated enrollment stations at all offices with a trained ID verification officer, pre-scheduled enrollment appointments via ITSM self-service, and a secondary registered backup key (kept in secure storage by IT) for each admin that can be activated immediately without re-enrollment if the primary is lost.

8.4 Controls that break at scale

Manual key inventory tracking — spreadsheet-based serial number tracking fails at 500+ keys with constant turnover; requires CMDB integration with automated IdP sync and daily reconciliation job. Human review of all enrollment events — at 500 admins with annual key rotation, reviewing every enrollment event manually is impractical; requires automated attestation validation in the IdP enrollment flow that only pages security for anomalies (unrecognized model, attestation failure, enrollment outside approved window). In-person enrollment for all admin levels — in-person-only is correct for the most privileged roles but creates bottleneck for mid-tier admins in remote offices; tiered approach: top-50 highest-privilege admins require in-person, remaining 450 require video + manager approval.

ANSWER SUMMARY — 90-second version

Use this to open your answer before going deep on any section.

"Replacing TOTP with FIDO2 hardware keys is one of the highest-impact security improvements you can make for privileged accounts. The core security property of FIDO2 is that the private key never leaves the device, and the WebAuthn protocol binds the authentication to the origin — so even if an attacker runs a perfect AiTM phishing site, the browser will refuse to sign a challenge from the wrong origin. That's the phishing resistance.

But it's important to be clear about what FIDO2 does not protect against. First, it doesn't protect post-authentication session cookies — an AiTM proxy can steal the session token after the FIDO2 challenge succeeds. I address that with Device-Bound Session Credentials in Chrome Enterprise. Second, and more importantly, it doesn't protect the recovery path. If an attacker can social engineer the helpdesk into removing the enrolled key and registering a new one, the hardware security model is completely bypassed. My control there is in-person-only key re-enrollment for all privileged admins — no remote MFA reset under any circumstances.

For supply chain risk on the hardware batch: I enforce attestation validation via FIDO MDS3 at enrollment — any key from an unrecognized model or firmware version is rejected. I also source keys from a single verified reseller with chain-of-custody documentation.

Residual risk I accept: deepfake-enhanced social engineering of the helpdesk (mitigated with in-person only), and post-auth session hijacking (mitigated with DBSC). The biggest remaining risk is the break-glass accounts — those are the backdoor around FIDO2, and they need to be stored in a physical vault with dual-party authorization."

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What residual phishing risk remains even with FIDO2 in place?

FIDO2 eliminates credential phishing — the private key never leaves the device and the WebAuthn assertion is origin-bound, so it won't authenticate against a lookalike phishing domain. The residual risk is post-authentication session theft: an AiTM proxy (Evilginx2 pattern) can serve as a transparent proxy to the real IdP, relay the FIDO2 challenge through the victim's browser (which correctly authenticates against the real domain), and steal the resulting session cookie. The defense is Device-Bound Session Credentials (DBSC), which cryptographically binds the session to the PAW's TPM — the stolen cookie is useless without the device private key. Additionally, I enforce strict CORS and Secure cookie flags, short session TTLs (30 minutes for admin sessions), and re-authentication requirements for privileged actions.

Follow-up 2: How do you handle account recovery when a hardware key is lost?

This is the single highest-risk process in the entire FIDO2 deployment. My policy for privileged admins: no remote recovery under any circumstances. The process is: (1) admin reports lost key to IT security helpdesk, (2) account is immediately suspended pending in-person identity verification, (3) admin visits a designated office location with government ID, (4) IT officer verifies identity and issues a replacement key from secured inventory, (5) new key is enrolled with attestation validation, (6) account is re-activated. Each admin also has a backup key registered and stored in IT's physical secure storage — not with the admin — that can be activated immediately for emergency access while the replacement key is being issued, eliminating the productivity impact of key loss.

Follow-up 3: What controls prevent a cloned or counterfeit hardware key from being enrolled?

FIDO2 keys embed a hardware attestation certificate from the manufacturer. During enrollment, the IdP checks the attestation certificate against the FIDO Alliance Metadata Service (MDS3) — this proves the key is a known model with verified firmware. I configure the IdP to reject any FIDO2 credential with attestation type 'none' (software authenticators, some platform authenticators) or with an attestation certificate not in the MDS3 for approved key models. I maintain an allowlist of specific YubiKey 5 series model AAGUIDs — any other AAGUID is rejected. Note: attestation proves the key model and firmware, not individual key uniqueness — this is why physical chain-of-custody (serial number tracking) is also required.

Follow-up 4: You mentioned device-bound sessions. What happens if that control fails — what is your next layer?

If DBSC isn't available or fails (non-Chrome browser, policy gap), the next layer is short session TTLs — admin sessions expire after 30 minutes of inactivity, requiring re-authentication, which limits the useful window for a stolen cookie. The layer after that is step-up authentication for privileged actions: even within a valid session, any destructive action (IAM role change, production deploy, firewall modification) triggers a fresh WebAuthn challenge — the attacker needs the physical key in hand, not just a session cookie. Finally, SIEM behavioral analytics flags sessions where the IP or ASN changes mid-session — a stolen cookie being used from a different location generates an alert within minutes.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is targeting the helpdesk. I don't need to steal a YubiKey or compromise a PAW — I just need to convince a helpdesk analyst that I'm the admin whose key I want to replace. I'd research the admin via LinkedIn, find their manager's name on the org chart, spearphish the manager's email or spoof it, then call the helpdesk with the manager's endorsement requesting an emergency MFA reset for the admin who is "locked out while traveling." Without in-person verification enforced as a hard policy (not just a soft recommendation), the helpdesk is the weakest link. The fix is making in-person enrollment a non-negotiable hard policy written into the ITSM workflow — no manager override, no exception for travel — and training helpdesk staff to refuse and escalate any request that cites urgency or management pressure.

COMMON MISTAKES TO AVOID

  • Claiming FIDO2 eliminates all phishing risk. It eliminates credential phishing. Post-authentication session cookie theft via AiTM is still possible and is the primary residual attack vector.
  • Not addressing the recovery process as the highest-risk path. The authentication mechanism is strong; the recovery mechanism is the attacker's target. Most candidates discuss the enrollment and auth flow without analyzing the recovery workflow.
  • Treating attestation as proof of individual key integrity. Attestation proves key model and firmware (via FIDO MDS3) but does not prove individual key uniqueness. A malicious manufacturer could ship keys with shared private keys that all pass attestation. Physical chain-of-custody is a separate required control.
  • Not enforcing FIDO2-only for admin group — allowing TOTP fallback. The most common implementation mistake. If TOTP remains enabled as a fallback "for compatibility," the attacker phishes the TOTP, not the YubiKey.
  • Ignoring break-glass accounts as a FIDO2 bypass. Break-glass accounts typically have weaker authentication because they're designed for emergency access. They are the backdoor around FIDO2 and must be treated with at least equal protection.
  • Not specifying key model and attestation allowlist. "We use FIDO2" without specifying hardware-attested keys with MDS3 validation allows software authenticators (passkeys on a phone) or unverified keys — which don't have the same security properties as a dedicated hardware key.
  • Underestimating supply chain risk. Candidates often dismiss the supply chain threat as theoretical. The SolarWinds and XZ Utils attacks demonstrate that supply chain attacks are executed by nation-state actors as standard TTPs.
