Scenario 19 — Cloud Access Security Broker (CASB) Deployment — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario19-casb-deployment.md
Generated artifact: scenario19-casb-deployment-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Deploy inline CASB for DLP, threat protection, and shadow IT discovery for 20,000 users and 400+ cloud apps.

1. Review Objective

This review examines the CASB deployment from two lenses:

  1. Attacker lens: realistic exploitation paths against CASB features (TLS inspection, admin plane, API connectors), including prerequisites, TTPs, and escalation chains.
  2. Blue-team lens: specific controls that must stop or detect each attack, where to enforce them, and exact telemetry fields and queries that prove the controls are firing.

The output includes command-level purple-team validation examples (safe, with placeholders), detection queries, and mermaid attack trees with control breakpoints.

2. Highest-Risk Design Decision

Terminating TLS at the CASB (inline TLS inspection) is the single most consequential design decision. It gives the CASB plaintext access to every HTTPS session for 20,000 users and therefore makes the CASB's CA private key and admin plane the highest-value targets. The safe choices are HSM-backed key management, strict MDM-driven CA distribution, and minimizing bypass lists; the dangerous alternative is storing CA keys on general-purpose servers or allowing rapid, unreviewed bypasses.

6. Threat-by-Threat Technical Review

T1 — CASB TLS Inspection CA Private Key Theft

STRIDE category: Spoofing
Risk: Low likelihood, Critical impact
Attacker objective: Obtain CASB CA private key (HSM-export or vendor compromise) to generate rogue certificates and perform widespread MITM against managed endpoints.

How the attacker would exploit it

  • Prerequisite: Access to CASB key management system or backup that contains the CA private key.
  • Initial action: Exfiltrate key material or request HSM key-export via stolen admin credentials.
  • Core exploitation: Use stolen key to sign certificates for target domains and stand up rogue proxy to intercept traffic.
  • Escalation: Harvest session cookies, OAuth tokens, and credentials across 400+ cloud apps.
  • Impact realization: Mass account takeover and data exfiltration; attacker can bypass TLS protections globally.

Blue-team control review — what must stop the chain

  • Control 1: Store CA private key in an HSM with ExportDisabled and require Multi-Party Authorization (MPC/MEP) for any usage; only allow signing via HSM APIs.
  • Control 2: Enforce strong admin access controls: FIDO2-only MFA for CA operators, PAW (privileged admin workstation) and RBAC for key operations.
  • Control 3: Require signed, auditable certificate issuance workflow and offline root signing for the highest-trust artifacts.
  • Control 4: Monitor HSM audit logs and alert on any KeyExport or unusual Sign operations with KeyId=<CA_KEY_ID>.
  • Control 5: Maintain immutable backups of CA policy and rotate CA certs on anomaly with an automated fleet re-trust process via MDM.

Evidence and telemetry to validate controls

  • HSM audit logs: events KeyUsage, Sign, KeyExportAttempt with actor, nonce, timestamp.
  • CASB CA signing audit: certificate_request_id, issued_dn, signed_by, device_count.
  • SIEM rule: HSM_Audit | where EventType == 'KeyExport' and KeyId == '<CA_KEY_ID>'.
  • MDM logs: CertificateInstall events showing CA certificate distribution and re-trust attempts.

Purple-team validation commands

# Check HSM audit for recent key export attempts (placeholder)
curl -H "Authorization: Bearer <HSM_API_TOKEN>" "https://hsm.example.com/api/audit?keyId=<CA_KEY_ID>&since=24h" | jq '.'

# SIEM: alert on HSM key export attempts
HSMAudit
| where EventType == 'KeyExport' and KeyId == '<CA_KEY_ID>'

T2 — CASB Admin Console Compromise

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Compromise CASB admin account to modify DLP rules, add bypass entries, exfiltrate logs, or enumerate OAuth connectors.

How the attacker would exploit it

  • Prerequisite: Phishing, credential stuffing, or attacker-controlled MFA prompt on admin account.
  • Initial action: Acquire admin credentials or session via AiTM or stolen MFA device; log into admin console.
  • Core exploitation: Modify DLP policies, add apps to bypass list, export logs containing tokens and evidence.
  • Escalation: Create backdoor API connectors or rotate CASB API keys for persistent access.
  • Impact realization: Silent policy changes enabling large-scale exfiltration and loss of visibility.

Blue-team control review — what must stop the chain

  • Control 1: Require FIDO2 hardware keys only for CASB admin accounts, and restrict admin console access to PAW with IP allowlist.
  • Control 2: Enforce JIT admin elevation and short admin sessions; require ticketing approval for any DLP bypass changes.
  • Control 3: Enable CASB admin audit logging to immutable storage and stream to SIEM (CASB_Admin_Audit topic).
  • Control 4: Monitor admin activity for anomalous patterns (bulk bypass additions, changes outside business hours) with detection rules.
  • Control 5: Protect API keys in secrets manager with rotation and require approvals for new connector creation.

Evidence and telemetry to validate controls

  • CASB admin audit log: fields admin_user, action, policy_name, change_type, src_ip, timestamp.
  • IdP SigninLogs: user, client_ip, auth_method, success correlated to admin_user.
  • SIEM detection: CASBAdminLogs | where action in ('bypass_add','policy_delete').
  • Secrets manager audit: GetSecret events for CASB API keys and connector tokens.

Purple-team validation commands

# List recent CASB admin policy changes via API (placeholder)

T3 — CASB Proxy Outage (Availability Attack)

STRIDE category: Denial of Service
Risk: Medium likelihood, Critical impact
Attacker objective: Take down CASB inline proxies (DoS or resource exhaustion) to cause service interruption or force fail-open behaviour.

How the attacker would exploit it

  • Prerequisite: Discover CASB endpoints and load balancers; identify resource exhaustion vectors.
  • Initial action: Flood CASB public endpoints, exploit a vulnerability, or consume backend resources.
  • Core exploitation: CASB health degrades, failover triggered or if misconfigured, traffic allowed through uninspected.
  • Escalation: Combine outage with pre-placed exfil channels to move data during the outage.
  • Impact realization: Productivity outage or large window of uninspected traffic.

Blue-team control review — what must stop the chain

  • Control 1: Deploy CASB active-active across AZs/regions with autoscaling and DoS protections at the edge (WAF, rate-limiting).
  • Control 2: Health-check monitoring with automated failover to secondary POP and rapid scaling.
  • Control 3: Default fail-closed PAC policy to block traffic if all CASB nodes unhealthy; provide business-approved emergency allowlist for critical services.
  • Control 4: DDoS scrubbing service in front of CASB ingress to absorb volumetric attacks.
  • Control 5: Capacity planning and chaos-testing of CASB failover paths.

Evidence and telemetry to validate controls

  • CASB health logs: probe_result, latency_ms, node_id, region.
  • Edge WAF logs and cloud provider DoS alerts.
  • SIEM rule: CASBHealth | where probe_result == 'FAIL' and duration > 60.
  • SOAR runbook execution logs showing automatic failover and notification.

Purple-team validation commands

# Simulate CASB node failure in staging and validate failover
curl -X POST -H "Authorization: Bearer <ORCH_TOKEN>" "https://orchestrator.example.com/api/simulate/failnode?node=<CASB_NODE_ID>"

# Alert on sustained CASB health probe failures
CASBHealth
| where ProbeResult == 'FAIL' and DurationSeconds > 60

T4 — Fail-Open Misconfiguration

STRIDE category: Tampering
Risk: Medium likelihood, Critical impact
Attacker objective: Exploit a misconfiguration where CASB proxies are configured to fail-open, allowing traffic to bypass inspection during outages.

How the attacker would exploit it

  • Prerequisite: Identify environment where PAC fallback allows direct internet (e.g., legacy PAC file default route).
  • Initial action: Induce a CASB health failure or alter PAC distribution to point to uninspected routes.
  • Core exploitation: Traffic moves outside CASB inspection and DLP, enabling exfiltration and reduced detection.
  • Escalation: Use bulk uploads or command-and-control through cloud apps during the window.
  • Impact realization: Large undetected exfiltration window.

Blue-team control review — what must stop the chain

  • Control 1: Use fail-closed PAC configuration in MDM (ProxyAutoConfig with fallback to block page) for managed endpoints.
  • Control 2: Tamper-proof PAC deployment via signed PAC files and MDM enforcement to prevent local override.
  • Control 3: Monitor for policy_mode == FAIL_OPEN events and alert with high priority.
  • Control 4: Implement network egress controls that still restrict destinations even when CASB is offline.
  • Control 5: Regular PAC tamper tests and endpoint audits.

Evidence and telemetry to validate controls

  • MDM logs showing PAC distribution and any ProxyOverrideAttempt events.
  • CASB logs indicating policy_mode or fail_open transitions.
  • Proxy gateway logs showing direct-to-internet connections from managed endpoints.
  • SIEM correlation: MDM | where ProxyConfigChange == true join CASBHealth events.

Purple-team validation commands

# Detect devices with PAC overrides
Get-MDMDeviceReport | where { $_.ProxyOverride -eq $true }

# Detect fail-open events in CASB logs
CASBProxyLogs
| where PolicyMode == 'FAIL_OPEN' and TimeGenerated > ago(1d)

T5 — Over-Broad Certificate Bypass List

STRIDE category: Tampering
Risk: High likelihood, High impact
Attacker objective: Get sensitive apps added to the bypass list (certificate pinning or business exception) to create blind spots for exfiltration.

How the attacker would exploit it

  • Prerequisite: Convince admin via business justification or compromise admin session.
  • Initial action: Request bypass for specific app citing need to avoid breakage.
  • Core exploitation: Once bypassed, DLP and threat protection no longer inspect traffic for that app.
  • Escalation: Use bypassed app to exfiltrate sensitive data at scale.
  • Impact realization: Persistent blind spot for high-sensitivity data flows.

Blue-team control review — what must stop the chain

  • Control 1: Governance: bypass requires CISO approval, written justification, and expiry date in policy-as-code change (Git PR).
  • Control 2: Use API-mode connectors where inline inspection not possible, to monitor data at rest and sharing for bypassed apps.
  • Control 3: Implement automated risk scoring for bypass requests and require a canary staging test before fleet-wide approval.
  • Control 4: Alert on data flows from bypassed apps that match sensitive patterns via API-mode scanning.
  • Control 5: Maintain bypass list change audit with SIEM alerts on any bypass_add events.

Evidence and telemetry to validate controls

  • CASB bypass list change logs: admin_user, app, reason, expiry_date.
  • Git PR records for policy-as-code showing reviewer approvals.
  • CASB API-mode audit showing monitoring events for bypassed app uploads.
  • SIEM rule: CASBAdminLogs | where action == 'bypass_add' and expiry_date == null.

Purple-team validation commands

# Query bypass list changes and find non-expiring entries

T6 — Mobile Cellular Bypass (CASB Not Applied)

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Use cellular data or unmanaged mobile apps to upload sensitive data outside CASB inspection.

How the attacker would exploit it

  • Prerequisite: User has a personal device or corporate device with cellular enabled but without CASB agent.
  • Initial action: Upload sensitive files from mobile app (Dropbox, personal Drive) over cellular.
  • Core exploitation: Traffic bypasses inline CASB proxies; DLP not triggered.
  • Escalation: Combine with social engineering to exfiltrate high-value files.
  • Impact realization: Silent exfiltration to personal cloud storage.

Blue-team control review — what must stop the chain

  • Control 1: Enforce MDM and require CASB mobile agent (or Zscaler App) on corporate devices for corporate app access.
  • Control 2: IdP conditional access policy blocking cloud app access from unmanaged devices or from client app types flagged as personal.
  • Control 3: MDM cellular usage anomaly detection: alert on large cellular uploads by users with recent access to sensitive resources.
  • Control 4: DLP: restrict sharing permissions for sensitive files to only corporate domains via API-mode enforcement.
  • Control 5: User education and deterrence with automatic blocking for policy violations.

Evidence and telemetry to validate controls

  • MDM logs: cellular_bytes, device_compliance_state, agent_installed.
  • CASB ShadowIT and Mobile logs showing absent agent and direct connections.
  • SIEM correlation between sensitive data access on laptop and subsequent mobile uploads from separate device for same user.
  • IdP conditional access logs showing blocked login attempts from unmanaged device.

Purple-team validation commands

# Find devices with high cellular uploads and no CASB agent

T7 — Shadow IT via BYOD / Unmanaged Endpoints

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Encourage users to use unsanctioned SaaS or unmanaged endpoints to store or transfer sensitive data outside CASB visibility.

How the attacker would exploit it

  • Prerequisite: Lack of strict enforcement or visibility into BYOD usage.
  • Initial action: User installs or uses unsanctioned SaaS via browser on BYOD or personal device.
  • Core exploitation: Data flows to shadow IT without DLP inspection; attacker prompts user to upload sensitive data.
  • Escalation: Attacker obtains data via compromised unsanctioned app or harvests public links.
  • Impact realization: Data loss and intelligence about internal systems from usage patterns.

Blue-team control review — what must stop the chain

  • Control 1: Detect and inventory shadow IT via CASB discovery and block high-risk apps automatically.
  • Control 2: Implement conditional access that denies corporate resource access from unmanaged endpoints and requires device compliance.
  • Control 3: Integrate MDM and CASB discovery feeds to trigger automated remediation (block, notify, request sanctioning).
  • Control 4: Use browser isolation for unknown SaaS to prevent direct data copy/paste or downloads.
  • Control 5: Regularly run user education campaigns and automate user appeals for app approvals.

Evidence and telemetry to validate controls

  • CASB ShadowIT discovery records with app_name, risk_score, user_count, first_seen.
  • Conditional Access logs showing denied access from unmanaged devices.
  • SIEM rule: ShadowIT | where RiskScore > 7 and UserCount > 10.
  • Incident records of previously sanctioned apps to validate approval process.

T8 — Pre-CASB OAuth Tokens Granting Uninspected Access

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Use existing long-lived OAuth grants issued before CASB deployment to access cloud app data without inspection.

How the attacker would exploit it

  • Prerequisite: OAuth grants issued before CASB deployment or long-lived tokens that CASB does not control.
  • Initial action: Use token to access data or refresh token via API.
  • Core exploitation: Activity occurs without inline inspection; CASB only sees API-mode telemetry later (post-facto).
  • Escalation: Combine with compromised service accounts to move data to attacker-controlled storage.
  • Impact realization: Data theft using legitimate tokens that are invisible in inline logs.

Blue-team control review — what must stop the chain

  • Control 1: Perform a comprehensive OAuth grant audit (Graph API, Google Admin) and revoke tokens granted before CASB deployment for sensitive apps.
  • Control 2: Enforce conditional access to require device compliance for any token replay or new sessions for high-privilege scopes.
  • Control 3: Enable CASB API-mode connectors to monitor and revoke actions post-facto for suspicious activity.
  • Control 4: Implement short token lifetimes and refresh token rotation for high-sensitivity apps.
  • Control 5: Automated weekly reconciliation to identify tokens older than casb_deploy_date and force re-auth.

Evidence and telemetry to validate controls

  • OAuth audit logs: token_id, user, client_app, granted_date, scope from Graph API/Google Admin.
  • CASB API connector logs showing actions taken by pre-CASB tokens.
  • SIEM rule: OAuthAudits | where GrantedDate < <CASB_DEPLOY_DATE> and Scope contains 'write'.
  • IdP logs showing re-auth events after forced token revocation.

Purple-team validation commands

# Identify active tokens issued before CASB deployment

OAuthAudit
| where GrantedDate < datetime('<CASB_DEPLOY_DATE>') and Scopes contains 'Files.ReadWrite'

T9 — DLP Evasion via Encrypted Archives

STRIDE category: Information Disclosure
Risk: High likelihood, Medium impact
Attacker objective: Wrap sensitive content in password-protected ZIP/RAR archives or encrypt before upload to bypass CASB content inspection.

How the attacker would exploit it

  • Prerequisite: Knowledge that CASB DLP cannot inspect encrypted blobs; ability to upload to allowed destination.
  • Initial action: Create password-protected archive or encrypt payload and upload to allowed cloud app.
  • Core exploitation: CASB flags file type but cannot inspect contents; detection relies on behavioral signals.
  • Escalation: Use multiple small uploads to avoid large-file thresholds.
  • Impact realization: Exfiltration of sensitive content without content-level detection.

Blue-team control review — what must stop the chain
