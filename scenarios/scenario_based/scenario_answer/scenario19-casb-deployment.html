Scenario 19 — Cloud Access Security Broker (CASB) Deployment

Scenario: A CASB is deployed inline for sanctioned and unsanctioned cloud app discovery, DLP, and threat protection for 20,000 users connecting to 400+ cloud apps.

Candidate questions:

  • What trust issues arise from a CASB performing TLS inspection on all cloud traffic?
  • How do you handle apps that pin certificates and break under CASB inspection?
  • What controls prevent the CASB itself from becoming a single point of failure?
  • How do you detect shadow IT apps that bypass the CASB via mobile data?
  • What residual risk exists for OAuth tokens issued before CASB was deployed?

3.4 Highest-risk boundary crossing

The highest-risk boundary crossing is the CASB's TLS inspection function — specifically, the moment the CASB terminates the user's TLS session and decrypts it before re-encrypting to the destination. At this point, the CASB has access to all plaintext: passwords, session tokens, credit card numbers, healthcare data, API keys, and any other sensitive data in HTTPS traffic. This is a necessary capability for DLP and threat detection, but it means the CASB is effectively a trusted MITM for all 20,000 users' cloud traffic. The CASB's TLS inspection CA certificate, if stolen, allows an attacker to perform the exact same MITM attack against any endpoint that trusts the certificate — transforming a defensive capability into an offensive weapon. The CASB also becomes the ultimate single-point-of-failure: a compromise of the CASB is more severe than a compromise of any individual endpoint, because it affects all 20,000 users simultaneously.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale (20,000 users, 400+ cloud apps), CASB policy is managed as code — all DLP rules, URL categories, sanctioned app lists, and bypass lists are stored in Git and deployed via Terraform to the CASB platform. Policy changes require code review by the security team and are deployed through a staged rollout. Shadow IT discovery automatically populates a risk scoring database, and apps that breach a risk threshold trigger an automated ITSM ticket for the security team to make a sanctioning decision — no manual monitoring of shadow IT discovery reports. OAuth token audits are automated via Google Workspace Admin API and Microsoft Graph, running weekly with auto-revocation for tokens exceeding the risk threshold.

8.2 Fleet-wide impact analysis

A CASB proxy outage at 20,000 users isn't just a security control failure — it's a total productivity outage if fail-closed is configured, or a total security control failure if fail-open. At Google scale (1M+ users), a CASB providing TLS inspection for all Google Workspace traffic would see every Gmail message, every Google Drive upload, every Meet recording, and every Cloud Console action in plaintext. A compromise of the CASB TLS inspection CA would give an attacker a universal MITM capability against a billion users. This is why Google's BeyondCorp does not use TLS inspection as a primary security control — it uses context-aware access policies enforced at the application layer, avoiding the CASB MITM model entirely.

8.3 Eliminating human bottlenecks

The bottleneck in CASB management is the bypass list approval process — every app with certificate pinning needs an exception reviewed by a human. At Google scale with hundreds of apps, this becomes a backlog. Resolution: automated bypass list management where the CASB attempts inspection and auto-detects certificate pinning failures, automatically creates a bypass request with the app's risk score pre-computed, and routes for CISO approval only if risk score >5. Apps with risk score <5 are auto-approved for a 90-day bypass with automatic expiry. DLP policy tuning is also automated: false positive reports from users feed an ML model that continuously tunes sensitivity thresholds per data category per user group.

8.4 Controls that break at scale

Manual shadow IT approval workflow — at 400+ apps with dozens of new apps discovered weekly, manual review of every shadow IT app fails; requires automated risk scoring (VirusTotal, CASB vendor risk database) with auto-block for high-risk apps and auto-approve-for-monitoring for low-risk. TLS inspection CA single instance — a single HSM-backed CA is a bottleneck at high traffic volumes; requires distributed HSM infrastructure with certificate caching. Per-user OAuth token audit — manually reviewing 20,000 users' OAuth grants across 400 apps is impossible; requires automated Graph API queries with ML-based anomaly detection for unusual scope grants. Static DLP rules — fixed regex patterns for PII/IP detection fail at scale due to high false positives; requires ML-based sensitive data classification trained on enterprise-specific data patterns.

ANSWER SUMMARY — 90-second version

Use this to open your answer before going deep on any section.

"Deploying a CASB inline gives you powerful DLP and threat visibility, but it creates two critical security concerns that I want to address upfront.

First, TLS inspection makes the CASB a trusted MITM. The inspection CA certificate, if stolen, lets an attacker impersonate any HTTPS site to any managed endpoint. I protect that CA private key in an HSM — it never leaves the hardware — and I minimize the bypass list with a formal governance process requiring CISO approval and expiry dates for every bypass.

Second, the CASB is a single point of failure. I deploy it active-active across multiple availability zones with a fail-closed policy — if the CASB goes down, corporate internet traffic is blocked rather than uninspected. That's an operational tradeoff the business must explicitly accept.

For mobile CASB bypass: employees on cellular data bypass the proxy. I enforce the Zscaler ZIA agent on all corporate-managed mobile devices via MDM, and detect cellular bypass attempts via MDM data usage anomalies. BYOD is handled by blocking corporate app access from unmanaged devices via IdP conditional access.

The hardest residual risk is pre-CASB OAuth tokens — they grant ongoing cloud app access that was never inspected and can't be retroactively controlled. I address this with a quarterly OAuth token audit and force re-authentication through the CASB for any token older than 90 days. Tokens from before the CASB deployment get immediate revocation for sensitive apps."

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What trust issues arise from a CASB performing TLS inspection on all cloud traffic?

TLS inspection is technically a MITM attack — the CASB terminates the user's TLS session, reads the plaintext, and re-encrypts to the destination. The user's browser validates the CASB's dynamically generated certificate against the enterprise's CASB root CA (deployed via MDM), creating an explicit trust relationship. The concerns are: (1) the CASB root CA becomes a critical secret — if stolen, it enables MITM from any location against any enrolled endpoint; (2) the CASB sees all user data in plaintext, making it a high-value exfiltration target if compromised; (3) certificate pinned applications will reject the CASB's certificate and break; (4) healthcare and legal traffic may have regulatory constraints on inspection. I address (1) with HSM-backed CA, (2) with CASB admin access controls and encrypted log storage, (3) with a governed bypass list, and (4) with explicit DLP policy carve-outs for regulated data flows.

Follow-up 2: How do you handle apps that pin certificates and break under CASB inspection?

Certificate pinning means the app has hardcoded the expected leaf or intermediate certificate and rejects any other certificate — including the CASB's inspection cert. My approach: (1) identify pinned-cert apps during pre-deployment testing with a proxy canary environment; (2) add confirmed pinning apps to the CASB bypass list with documented business justification and a 6-month expiry that forces re-review; (3) require out-of-band API mode for these apps (use CASB's API connector to monitor data at rest and sharing via cloud app API, rather than inspecting in transit); (4) apply tighter egress controls for bypassed apps — scoped firewall rules restricting what data can flow to those destinations. The bypass list is a security debt register — every entry represents a DLP blind spot that must be compensated by API mode visibility.

Follow-up 3: What controls prevent the CASB itself from becoming a single point of failure?

Availability controls: active-active HA deployment across two availability zones (Zscaler POP redundancy, or Netskope multi-PoP); anycast-based traffic routing that automatically fails over to secondary PoP if primary is unhealthy; automated health probes every 30 seconds. Policy controls: fail-closed is the default — if CASB is unreachable, PAC file routes to a block page rather than passing traffic uninspected. Operations controls: CASB availability is a P0 incident with on-call response SLA under 15 minutes; separate DR region activated for extended outages. The fail-closed vs fail-open tradeoff must be an explicit business decision — I document the estimated hourly revenue impact of fail-closed vs the estimated security risk of fail-open, and present both to the CISO for a signed risk acceptance.

Follow-up 4: You mentioned pre-CASB OAuth tokens. What happens if that control fails — what is your next layer?

If the quarterly OAuth audit misses active tokens (incomplete coverage of all 400 apps, or tokens issued via APIs not covered by audit), the next layer is the CASB's out-of-band API mode — even for pre-CASB tokens, the CASB API connector can monitor cloud app activity (downloads, shares, admin actions) via the cloud app's native API. This means even if the token bypassed inline inspection, the activity it performs is visible to the CASB after the fact. Detection: CASB API mode alerts on anomalous behavior from API-connected apps — bulk downloads, unusual sharing to external addresses, off-hours large file access. The final backstop is the IdP conditional access policy: if I enforce that production cloud apps require device compliance, any device without the CASB agent fails compliance and is blocked — forcing re-auth with the CASB agent installed.

Follow-up 5: If you were the attacker, what is the cheapest path through your control set?

The cheapest path is using a personal device on cellular data. I open a personal SaaS app (Google Drive, Dropbox) on my iPhone, connect to corporate Wi-Fi on my laptop to access corporate data, copy it to my clipboard, switch to the iPhone, and upload it to personal cloud storage from the iPhone's cellular connection — never touching the CASB proxy. The CASB's inline mode has zero visibility into this. My detection for this: MDM cellular data usage anomaly (high cellular upload from a device whose user handles sensitive data), and IdP anomaly detection (user accessing corporate data from managed laptop immediately followed by new sign-in from personal device). The architectural fix is zero-trust data access: corporate data should only be accessible from MDM-compliant devices, and cloud app access should require device compliance check, making the iPhone on cellular unable to access corporate SaaS apps without the CASB agent running.

COMMON MISTAKES TO AVOID

  • Not addressing the CASB itself as an attack surface. The CASB performing TLS inspection creates a uniquely powerful MITM position. Candidates who treat the CASB as purely a defensive tool miss the offensive risk it creates.
  • Treating certificate bypass list as an edge case. Certificate pinning is common in modern applications (mobile apps, AWS SDK, Stripe APIs). A CASB without a well-governed bypass list will have a significant tail of bypassed traffic that is entirely uninspected.
  • Not having a fail-closed policy. The default for many CASB deployments is fail-open (if proxy fails, allow traffic). This is an explicit risk acceptance decision that must be documented — candidates who don't mention it miss a critical availability-vs-security tradeoff.
  • Ignoring mobile bypass on cellular data. The inline CASB proxy only works for traffic routed through it. Mobile devices on cellular data bypass the proxy entirely unless the CASB vendor's mobile agent is deployed via MDM. This is the most commonly exploited gap in CASB deployments.
  • Not addressing pre-CASB OAuth tokens. OAuth tokens issued before CASB deployment grant persistent cloud app access that the CASB has never inspected. These tokens are invisible to the inline CASB and must be audited and revoked separately.
  • Conflating CASB API mode and inline mode. API mode monitors data at rest via cloud app APIs; inline mode monitors data in transit. They have different coverage scopes and different capabilities. Candidates who use them interchangeably reveal shallow knowledge.
  • Underestimating the privacy and regulatory implications of TLS inspection. Many enterprises discover post-deployment that TLS inspection of healthcare portals, banking sites, or legal research tools has regulatory compliance implications (HIPAA, attorney-client privilege, EU GDPR Article 9).
