Source file reviewed: scenario20-insider-threat-detection.md

Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Launch an insider threat detection program using UEBA, DLP, and HR correlation for 15,000 employees with Legal and HR co-ownership.

1. Review Objective

This review maps realistic insider attack paths, the precise controls that must stop or detect them, the telemetry fields to validate controls, and safe purple-team validation steps. It emphasizes legal constraints, program governance, and detection-resistant techniques such as baseline manipulation and slow-burn exfiltration.

2. Highest-Risk Design Decision

Integrating sensitive HR attributes (disciplinary history, notice dates, medical leave) directly into the UEBA scoring engine is the single most consequential design decision. While HR signals improve prioritization for true positives, they also create legal risk (discrimination/GDPR) and magnify the privacy impact if the insider program is misused. The safe alternative is scoped HR signals (notice date, role, privileged status) with strict legal-approved filters and minimal retention; the dangerous alternative is broad HR feed ingestion without DPIA or oversight.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Bulk IP Download Before Departure

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Download substantial IP (source code, design docs) to local or cloud storage prior to termination.

How the attacker would exploit it

  • Prerequisite: Legitimate access to file shares or repo; opportunity during notice period.
  • Initial action: Use approved credentials to mass-download files over several sessions (to avoid rate limits).
  • Core exploitation: Copy files to local endpoint or cloud storage (personal Drive); remove or obfuscate audit trails.
  • Escalation: Use personal accounts or anonymized cloud storage to hold data post-termination.
  • Impact realization: Permanent exfiltration of IP and competitive leakage.

Blue-team control review — what must stop the chain

  • Control 1: DLP policy to block bulk downloads to personal cloud/USB during notice period; enforce automatic heightened monitoring for users flagged by HR.
  • Control 2: Endpoint DLP agent that logs and blocks mass file-copy operations and USB writes; create an EDRRule for FileCopy > X per minute.
  • Control 3: UEBA rule for rapid increase in file access volume relative to peer group and history.
  • Control 4: Offboarding automation: reduce access and require manager/HR sign-off for large downloads; implement read-only access for departing employees.
  • Control 5: Preserve forensic snapshots and run supervised export for legitimate data transfer requests.

Evidence and telemetry to validate controls

  • File access logs with user, file_count, bytes_transferred, timestamp.
  • DLP block logs indicating destination (personal cloud domain) and file_names.
  • EDR events for CreateFile/CopyFile and USB mount events.
  • HR event correlator showing notice_date and file access timeline.

Purple-team validation commands

# Simulate bulk download during notice period and verify DLP block
Invoke-DLPTest -User '<test-user>' -Files 100 -Destination 'https://personal-cloud.example.com/upload'

# SIEM: detect bulk download by departing users
FileAccessLogs
| where User in (departing_users_30d) and FileCount > 50 and TimeGenerated > ago(24h)

T2 — Email Forwarding to Personal Domains

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Create mailbox forwarding rules or manually forward sensitive emails to personal accounts to exfiltrate information.

How the attacker would exploit it

  • Prerequisite: Access to corporate mailbox and ability to create inbox rules or compose emails.
  • Initial action: Add forwarding rule to external address or manually forward sensitive emails.
  • Core exploitation: Persistent exfiltration via automated forwarding; use aliases to hide destination.
  • Escalation: Combine with auto-delete rules to remove evidence from corporate mailbox.
  • Impact realization: Ongoing leakage of sensitive communications.

Blue-team control review — what must stop the chain

  • Control 1: Email gateway DLP policy to block or quarantine messages with sensitive attachments to external domains and block mailbox forwarding to non-corporate addresses.
  • Control 2: SIEM rule to detect creation of forwarding rules and alert immediately.
  • Control 3: Mailbox audit logging retention and quick remediation scripts to remove unauthorized forwarding rules.
  • Control 4: Use mailbox access governance: require manager approval for forwarding rules or block by default.

Evidence and telemetry to validate controls

  • Exchange/Office365 audit logs showing Set-Mailbox or New-InboxRule events and ForwardingSMTPAddress values.
  • DLP logs for messages blocked/quarantined containing attachments matching sensitive categories.
  • SIEM correlation between New-InboxRule by a user and subsequent outbound messages to same external domain.

Purple-team validation commands

# Detect mailbox forwarding rules to external domains
Get-Mailbox | Get-InboxRule | where { $_.ForwardTo -like '*@gmail.com' }

# SIEM: alert when forwarding rule created and external messages follow
ExchangeAuditLogs
| where Operation == 'New-InboxRule' and Parameters contains 'RedirectTo'

T3 — Privileged Database Export

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Use admin DB access to export large datasets, schema, or PII for exfiltration.

How the attacker would exploit it

  • Prerequisite: Valid privileged DB credentials or JIT access granted.
  • Initial action: Execute large SELECT queries or bulk export utilities generating CSV/SQL dump.
  • Core exploitation: Transfer dump to local endpoint or cloud with sync agent.
  • Escalation: Create new user with export privileges or use existing service account.
  • Impact realization: Massive data exposure across customers and employees.

Blue-team control review — what must stop the chain

  • Control 1: JIT privileged sessions with approval workflow and session recording; limit export capabilities to approved sessions only.
  • Control 2: DB audit logging with row count threshold alerts: alert on queries returning unusually large row counts.
  • Control 3: Network egress controls preventing DB servers from initiating outbound uploads except to approved backup targets.
  • Control 4: DLP for database dumps: detect high-sensitivity schemas and block off-network export.

Evidence and telemetry to validate controls

  • DB audit logs: user, query, row_count, timestamp.
  • Network logs showing large outbound transfers from DB admin machine to external IPs.
  • JIT session logs showing approver and session duration.

Purple-team validation commands

-- Find DB queries with high row counts
SELECT username, query_text, rows_returned, timestamp FROM audit_log WHERE rows_returned > 100000;

# SIEM: alert on large DB query results
DBQueryLogs
| where RowCount > 100000 and User in (PrivilegedDBUsers)

T4 — UEBA Baseline Manipulation (Slow Normalization)

STRIDE category: Tampering
Risk: Low likelihood, High impact
Attacker objective: Gradually increase risky behavior to shift the UEBA baseline so large exfiltration appears normal.

How the attacker would exploit it

  • Prerequisite: Long-term access and knowledge of what signals the UEBA uses.
  • Initial action: Slowly increase download/upload volume or number of repo clones over months at rates below anomaly thresholds.
  • Core exploitation: Once baseline is elevated, perform larger exfiltration within new baseline without triggering alerts.
  • Escalation: Combine with token or credential theft to amplify impact.
  • Impact realization: Large exfiltration masquerading as normal behavior.

Blue-team control review — what must stop the chain

  • Control 1: UEBA baseline drift detection: alert when baseline increases >20% over 30 days for critical metrics.
  • Control 2: Cumulative long-window audits: monthly checks of total data accessed per user over 6 months to catch slow-burn patterns.
  • Control 3: Peer-group comparison and cross-attribute correlation.
  • Control 4: Randomized manual audits of privileged users regardless of UEBA score.

Evidence and telemetry to validate controls

  • UEBA baseline change logs with user, metric, baseline_old, baseline_new, delta%.
  • Long-term access summaries showing cumulative bytes and file counts over rolling 6 months.
  • SIEM alerts for baseline drift rules.

Purple-team validation commands

# Detect baseline drift >20% in 30 days
UEBA_Baseline
| where Baseline30dChange > 20 and Metric in ('download_volume','access_count')

# Simulate gradual increase of behavior in lab to test baseline drift detector
python simulate_behavior_increase.py --user '<test-user>' --metric downloads --days 60 --daily-increase 0.01

T5 — Logic Bomb Insertion in Code

STRIDE category: Tampering
Risk: Low likelihood, High impact
Attacker objective: Insert code that triggers destructive behavior post-departure or at predefined time to sabotage systems.

How the attacker would exploit it

  • Prerequisite: Commit privileges and ability to bypass code review or collude with reviewer.
  • Initial action: Add obfuscated backdoor or time-triggered payload in a low-review file.
  • Core exploitation: Payload activates in production causing data deletion or service disruption.
  • Escalation: Use as cover for data exfiltration or extortion.
  • Impact realization: Production outages, data loss, reputational damage.

Blue-team control review — what must stop the chain

  • Control 1: GitHub branch protection: require at least 2 reviewers, CI test coverage, and forbidden patterns detection before merge.
  • Control 2: Pre-deploy static analysis and secret scanning; post-deploy integrity checks and canary deployments.
  • Control 3: Enforce code ownership and separate dev/test versus prod deployer roles.
  • Control 4: Runtime integrity monitoring for unusual process behavior or scheduled tasks.

Evidence and telemetry to validate controls

  • Repo PR logs showing reviewer IDs, CI job outputs, and file diffs.
  • CI pipeline alerts for suspicious changes and failed policy checks.
  • Runtime monitoring alerts for unexpected process creation or scheduled job execution.

Purple-team validation commands

gh pr merge <number> --auto

| where Action == 'merge' and RequiredReviews == 0

T6 — Print / Photograph Sensitive Screens

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Photograph screens or print sensitive documents to physically exfiltrate data unseen by digital DLP.

How the attacker would exploit it

  • Prerequisite: Physical access to device or unattended workstation; lax physical controls.
  • Initial action: Use phone to photograph screen or print documents and take them off-site.
  • Core exploitation: Data moves out of digital control; photos are uploaded from personal device.
  • Escalation: Use micro-storage or physical smuggling to exfiltrate large volumes.
  • Impact realization: Data leakage that is hard to detect digitally.

Blue-team control review — what must stop the chain

  • Control 1: Physical security controls: restricted zones for high-sensitivity screens, privacy filters, and badge-controlled print pick-up.
  • Control 2: Print DLP: monitor and require authentication for high-volume or classified prints; quarantine prints for review.
  • Control 3: Behavioral detectors: monitor for unusual window focus changes, screen capture API usage, and use of camera hardware on managed devices.
  • Control 4: Regular training and clear desk policy enforcement with audits.

Evidence and telemetry to validate controls

  • Print server logs indicating large or classified print jobs with user, document_label, page_count.
  • Endpoint telemetry showing camera device usage or ScreenCapture API calls.
  • Physical access logs for secure zones and badge swipes correlated to sensitive access times.

Purple-team validation commands

# Detect large classified print jobs in last 24h

T7 — Investigator Abuse of Program Access

STRIDE category: Information Disclosure
Risk: Low likelihood, High impact
Attacker objective: Investigator with privileged access misuses the case database to surveil employees or leak sensitive data.

How the attacker would exploit it

  • Prerequisite: Investigator role with broad case database access and weak separation-of-duty controls.
  • Initial action: Query case DB for employees of interest, exfiltrate reports or investigator notes.
  • Core exploitation: Use access to widen targeting, plant false positives, or sell insights.
  • Escalation: Add peers to false-case pipeline to hide activity.
  • Impact realization: Privacy violations, legal risk, loss of trust, and insider program abuse.

Blue-team control review — what must stop the chain

  • Control 1: Strict role-based access with least-privilege and just-in-time investigator elevation; investigators only see minimal fields necessary.
  • Control 2: Dual-party investigation requirement for escalation and evidence preservation; no single investigator can unilaterally advance a case to discipline.
  • Control 3: Immutable audit logging of all investigator actions stored off-platform for periodic third-party review.
  • Control 4: Quarterly access recertification for investigator roles and automated anomaly detection on case access patterns.

Evidence and telemetry to validate controls

  • Case DB logs: investigator_id, case_id, fields_viewed, action, timestamp.
  • SIEM rule: detect unusual investigator queries (many profile views in short time).
  • Oversight board minutes showing access review results.

Purple-team validation commands

# Detect investigator accessing many distinct employee records quickly

| summarize CountDistinct = dcount(EmployeeId) by InvestigatorId
| where CountDistinct > 50

T8 — Credential Sharing with External Party

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Insider shares credentials with external party, enabling remote access under valid accounts.

How the attacker would exploit it

  • Prerequisite: Insider willingly shares credentials or stores them in shared location.
  • Initial action: External actor uses shared credentials to access cloud apps or systems.
  • Core exploitation: Activity appears as valid user events; detection depends on device and IP anomalies.
  • Escalation: External actor registers new devices or IPs that mimic corporate IPs via VPN.
  • Impact realization: Persistent unauthorized access and lateral movement.

Blue-team control review — what must stop the chain

  • Control 1: Enforce FIDO2 keys and disallow password-only auth for privileged accounts; require device-bound auth for critical systems.
  • Control 2: Monitor for new device registrations or anomalous IPs post-privileged session (IdP logs + device enrollment).
  • Control 3: Credential hygiene: rotate service account passwords, disable legacy auth, and enforce passwordless where possible.
  • Control 4: Detect concurrent sessions from geographically disparate IPs for same user.

Evidence and telemetry to validate controls

  • IdP logs: new device enrollments, OTP usage, FIDO2 registration events.
  • Session logs showing user, src_ip, device_id, and concurrent session timestamps.
  • SIEM rule: alert on User logged in from new device and performed sensitive actions within 1 hour.

Purple-team validation commands

# Detect concurrent sessions from different geolocations for privileged users
SigninLogs
| where User in (PrivilegedUsers)
| summarize distinct_ips = dcount(ClientIP) by SessionId
| where distinct_ips > 1

T9 — Data Staging in Personal Cloud then Exfiltration

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Stage sensitive files in personal cloud accounts and later transfer them to external recipients.

How the attacker would exploit it

  • Prerequisite: Access to cloud sync client or web upload from corporate endpoint.
  • Initial action: Upload select files to personal account using browser or sync client, often over multiple sessions.
  • Core exploitation: Use personal cloud as buffer to exfiltrate outside corporate controls.
  • Escalation: Use shared links or secondary accounts to relay files to buyer.
  • Impact realization: Long-lived exfiltration with plausible deniability.

Blue-team control review — what must stop the chain

  • Control 1: Endpoint DLP to block personal cloud sync clients and web uploads for sensitive file classifications.
  • Control 2: CASB discovery to detect personal cloud storage activity from managed endpoints; block or quarantine by policy.
  • Control 3: UEBA correlation: user accesses sensitive files then soon after personal cloud upload events detected.
  • Control 4: Network egress controls blocking direct uploads to known personal cloud domains from corporate network unless from approved devices.

Evidence and telemetry to validate controls

  • CASB/Proxy logs: user, dst_domain, bytes_out, file_names for uploads to personal cloud.
  • Endpoint logs showing process names googledrivesync, dropbox and blocked actions recorded by DLP.
  • SIEM correlation: FileAccess + ProxyUpload events within short time windows.

T10 — Supply Chain Insertion via Legitimate Role

STRIDE category: Tampering
Risk: Low likelihood, Critical impact
Attacker objective: Insider with commit privileges inserts backdoor into production code that exfiltrates data or opens persistence.

How the attacker would exploit it

  • Prerequisite: Developer access and ability to push commits; collusion or bypass of code review.
  • Initial action: Insert stealthy exfiltration code or backdoor disguised as refactor.
  • Core exploitation: Backdoor exfiltrates data to external endpoint triggered under certain conditions.
  • Escalation: Leak backdoor access or sell to external actor.
  • Impact realization: Persistent undetected backdoor with high blast radius.

Blue-team control review — what must stop the chain

  • Control 1: Enforce multi-person code review, static analysis, and runtime anomaly detection for new behavior before merge.
  • Control 2: Use reproducible builds and SBOM to detect unexpected dependencies or artifacts.
  • Control 3: Require all production merges to go through canary with behavioral monitors for exfil patterns.
  • Control 4: Periodic supply chain code audits and developer access certifications.

Evidence and telemetry to validate controls

  • Git commit history and PR metadata with reviewer IDs.
  • CI build logs with static analysis and dependency checks.
