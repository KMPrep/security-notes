Scenario 21 — Network Access Control (NAC) for Hospital IoT — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario21-nac-hospital-iot.md
Generated artifact: scenario21-nac-hospital-iot-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Deploy 802.1X NAC for wired/wireless access on a hospital campus with clinical IoT, imaging VLANs, staff workstations, and guest Wi-Fi; some medical devices lack 802.1X support.

1. Review Objective

This review enumerates attacker paths from guest or vendor access to life-critical clinical devices, maps each threat to specific controls and telemetry (event IDs, log fields), provides safe purple-team validation queries and a prioritized remediation list. Emphasis: patient safety impact, detection signals, and compensating controls for legacy IoMT devices.

2. Highest-Risk Design Decision

Allowing legacy medical devices onto the Clinical IoT VLAN using MAC Authentication Bypass (MAB) without strong multi-signal profiling is the single highest-risk design choice. MAB provides admission based on easily spoofable identifiers (MAC), so placing life-critical devices on the same VLAN without cryptographic identity or additional isolation creates a root cause that enables MAC spoofing, VLAN pivoting, and immediate patient-impacting attacks. The safe alternative: minimize MAB admission, enforce isolated management VLANs, and require compensating post-admit behavioral enforcement (Claroty/Medigate) and strict ACLs.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or safety impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — MAC Spoofing to Impersonate Medical Device

STRIDE category: Spoofing
Risk: High likelihood, Critical impact
Attacker objective: Impersonate an infusion pump or monitor to gain admission to Clinical IoT VLAN and issue malicious commands.

How the attacker would exploit it

  • Prerequisite: Passive network visibility (sniffing on Guest VLAN) or access to broadcast domain exposing device MACs.
  • Initial action: Capture target device MAC via ARP/mDNS/LLDP responses.
  • Core exploitation: Disconnect legitimate device, connect attacker host with spoofed MAC to a port or wireless.
  • Escalation: Use management interfaces to change dosing or push malicious firmware links.
  • Impact realization: Patient safety incident or persistent backdoor in clinical VLAN.

Blue-team control review — what must stop the chain

  • Control 1: Multi-signal NAC profiling (ISE): require OUI, DHCP fingerprint, TTL/HTTP UA match — not MAC-only.
  • Control 2: Port-security with violation shutdown and re-auth timer; restrict switchport port-security maximum 1 on clinical ports.
  • Control 3: Post-admit Claroty behavioral baseline: detect protocol deviations (DICOM/HL7 unexpected ports).
  • Control 4: Physical tamper controls: badge access + CCTV on wiring closets and patch panels.
  • Control 5: Maintain CMDB of device MAC: automated reconciliation with ISE and quarantine unknown devices.

Evidence and telemetry to validate controls

  • Cisco ISE auth logs: endpoint_mac, vlan_assigned, auth_result, port_id.
  • Switch syslog port-security events: event=PORT_SECURITY_VIOLATION, port_id, mac_addr.
  • Claroty/Medigate alert: device_id, protocol_deviation, risk_score.
  • NetFlow showing new host IP using medical device ports to unexpected destinations.

Purple-team validation commands

# Detect same MAC observed on multiple ports within 5 minutes
ISELogs | where EventTime > ago(1h) | summarize ports = dcount(PortId) by EndpointMAC | where ports > 1

# Check for recent port-security violations on clinical ports
Get-SwitchSyslog | where { $_.Event -eq 'PORT_SECURITY_VIOLATION' -and $_.Port -in $ClinicalPorts }

T2 — VLAN Hopping via DTP or Misconfigured Trunks

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Negotiate trunk or exploit misconfigured trunking to access multiple VLANs from a guest-facing port.

How the attacker would exploit it

  • Prerequisite: Access to switch port capable of DTP negotiation (auto trunking enabled) or a misconfigured trunk.
  • Initial action: Send DTP frames to trigger trunk negotiation or craft 802.1Q frames to exploit native VLAN.
  • Core exploitation: Receive traffic from clinical VLAN or inject frames to reach devices.
  • Escalation: Lateral movement into staff VLAN and EHR access.
  • Impact realization: Broad cross-VLAN compromise including patient-impacting devices.

Blue-team control review — what must stop the chain

  • Control 1: Disable DTP on all switch ports (switchport nonegotiate) and explicitly configure trunk/access modes.
  • Control 2: Set native VLAN to unused ID and drop native VLAN traffic on access ports.
  • Control 3: Configuration compliance scanning (Cisco DNA Center / Ansible) to detect unexpected trunk configs.
  • Control 4: VLAN change detection alerts from syslog and change-management enforcement.

Evidence and telemetry to validate controls

  • Switch config inventory and nightly compliance reports showing switchport mode per port.
  • Syslog DTP negotiation events: event=DTP_TRUNK_NEGOTIATED, port_id.
  • Firewall NetFlow showing guest subnet traffic to clinical IPs.
  • Change-management tickets correlated to VLAN config changes.

Purple-team validation commands

# Find ports configured for DTP/trunking unexpectedly
SwitchConfigs | where SwitchportMode != 'access' and PortType == 'edge'

# Detect DTP negotiation events on access ports
SwitchSyslog
| where Event == 'DTP_TRUNK_NEGOTIATED' and Port in AccessPortList

T3 — 802.1Q Double-Tagging VLAN Hop

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Use double-tagging to bypass VLAN filtering and reach clinical VLAN from an access port.

How the attacker would exploit it

  • Prerequisite: Access to a port with native VLAN configured that maps to clinical VLAN or misaligned native VLAN.
  • Initial action: Craft 802.1Q double-tagged frames to present inner tag as clinical VLAN.
  • Core exploitation: Frames forwarded to clinical VLAN devices.
  • Escalation: Execute commands on management interfaces or exfiltrate PHI.
  • Impact realization: Device compromise and patient harm.

Blue-team control review — what must stop the chain

  • Control 1: Set native VLAN to an unused ID and drop native VLAN on access ports.
  • Control 2: Enforce explicit VLAN tagging on all trunks; restrict access ports to single VLAN.
  • Control 3: IPS/IDS signatures for double-tagging anomalies and unknown tag lengths.
  • Control 4: Config drift detection and nightly audits for native VLAN misconfiguration.

Evidence and telemetry to validate controls

  • Firewall logs showing unexpected tagged frames or traffic originating from access ports.
  • Switch syslog for native VLAN mismatches and tag anomalies.
  • IDS/IPS alerts for malformed 802.1Q frames.
  • Claroty detection of unexpected source IPs initiating new protocols.

Purple-team validation commands

# Generate double-tagged frames in lab to verify native VLAN drop
scapy -c "send(Ether()/Dot1Q(vlan=999)/Dot1Q(vlan=300)/IP()/TCP())" --count 100

# SIEM: detect malformed 802.1Q frames or unknown tag events
SwitchSyslog | where Message contains '802.1Q' and Message contains 'double-tag'

T4 — RADIUS / Cisco ISE Compromise

STRIDE category: Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Compromise NAC control plane to admit arbitrary devices to any VLAN.

How the attacker would exploit it

  • Prerequisite: Admin credential theft, vulnerability in RADIUS/Ise, or lateral movement into NAC management network.
  • Initial action: Modify policies, add MAC allow-lists, or create backdoor access tokens.
  • Core exploitation: Admit attacker devices into clinical VLAN and disable port-security alarms.
  • Escalation: Persist by creating new admin accounts and removing logs.
  • Impact realization: Global fleet admission, mass patient safety impact, system-wide outage.

Blue-team control review — what must stop the chain

  • Control 1: Harden NAC management plane: separate management network, MFA for admin access, limited admin accounts.
  • Control 2: Immutable logging and SIEM ingestion for RADIUS admin actions; frequent integrity checks of policy bundles.
  • Control 3: RADIUS clustering with signed policy bundles and offline verification.
  • Control 4: Backup/restore policy validation and emergency fail-closed mode to quarantine unknown devices.

Evidence and telemetry to validate controls

  • RADIUS admin audit logs: admin_user, change_type, timestamp.
  • SIEM alerts for configuration changes pushed outside maintenance windows.
  • ISE accounting logs: sudden spike in Access-Accept events for Clinical VLAN.
  • Management network flows showing unusual connections to NAC controllers.

Purple-team validation commands

# Verify admin MFA and recent policy changes on ISE
Get-IseAdminPolicyChanges -Since (Get-Date).AddDays(-7)

# Detect spike in Access-Accept for clinical VLAN
ISEAccounting | where Vlan == 'clinical_iot' | summarize count() by bin(TimeGenerated,1h)

T5 — Rogue AP Bridging Guest and Clinical Networks

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Deploy a rogue wireless AP that bridges guest traffic into clinical VLAN or backdoor into wired network.

How the attacker would exploit it

  • Prerequisite: Physical proximity to hospital; access to a power outlet or hidden device placement.
  • Initial action: Deploy rogue AP configured with SSID similar to guest or create bridge to wired patch port.
  • Core exploitation: Bridge guest clients into clinical VLAN or capture traffic for MACs to spoof.
  • Escalation: Use bridge to launch lateral attacks into staff VLAN.
  • Impact realization: Unrestricted lateral movement and device compromise.

Blue-team control review — what must stop the chain

  • Control 1: WLC rogue AP detection with automated RF triangulation and immediate AP containment.
  • Control 2: AP allowed-listing and 802.11w management protection; block ad-hoc bridging behaviors.
  • Control 3: Periodic RF sweeps and physical site inspections for hidden APs.
  • Control 4: Port security on switch ports to prevent unknown AP uplinks.

Evidence and telemetry to validate controls

  • Wireless controller logs: rogue_ap_detected, ap_mac, rssi, first_seen.
  • NAC logs showing unusual client associations or new SSIDs.
  • Switch port-link up events correlated with unknown AP MACs.
  • Physical security camera footage aligned to rogue AP timelines.

Purple-team validation commands

# List rogue APs detected in last 24h

T6 — Unpatched Medical Device Worm / Ransomware

STRIDE category: Elevation of Privilege / Denial of Service
Risk: High likelihood, Critical impact
Attacker objective: Exploit known OS vulnerabilities on legacy devices to propagate worm/ransomware across clinical VLAN.

How the attacker would exploit it

  • Prerequisite: Vulnerable device OS (WinXP, embedded Linux) reachable on clinical VLAN; initial code execution via exposed service.
  • Initial action: Exploit known CVE to gain shell and deploy payload.
  • Core exploitation: Lateral movement to other devices using same vuln or default creds.
  • Escalation: Encrypt PACS/EHR or modify device behavior (alarm suppression, dosage changes).
  • Impact realization: Patient safety incidents, diagnostic outages, mass downtime.

Blue-team control review — what must stop the chain

  • Control 1: Network segmentation with allow-list firewall rules — limit protocols and destinations per device type.
  • Control 2: Passive IoMT monitoring (Claroty/Medigate) to detect lateral propagation and unusual SMB/SMBv1 or RPC traffic.
  • Control 3: Backup & rapid recovery playbooks for PACS/EHR with air-gapped backups.
  • Control 4: Device replacement roadmap and compensating controls until vendor patches are available.

Evidence and telemetry to validate controls

  • Claroty alerts: suspicious_scan, new_connection_pattern.
  • IDS signatures for known exploit payloads (EternalBlue, SMB exploit indicators).
  • Firewall logs showing unexpected lateral flows within clinical VLAN.
  • EDR events if agentable on workstations adjacent to devices.

Purple-team validation commands

T7 — Vendor Remote Maintenance Abuse

STRIDE category: Trusted Relationship / Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Use vendor remote access to pivot into clinical VLAN outside maintenance scope.

How the attacker would exploit it

  • Prerequisite: Vendor VPN credentials or compromised vendor workstation.
  • Initial action: Initiate remote maintenance session with broad scope.
  • Core exploitation: Access additional devices or pivot to adjacent systems via vendor access token reuse.
  • Escalation: Create persistent access, modify device configs.
  • Impact realization: Device misconfiguration, data exfiltration, or introduction of malicious firmware.

Blue-team control review — what must stop the chain

  • Control 1: Use PAM for vendor access (CyberArk): JIT, session recording, target-specific access, and automatic revocation.
  • Control 2: Network micro-segmentation: vendor VPN only allowed to management VLAN with no direct path to clinical VLAN.
  • Control 3: Vendor session monitoring with SOAR triggers on post-maintenance anomalous commands.
  • Control 4: Enforce vendor MFA and device posture checks before granting access.

Evidence and telemetry to validate controls

  • PAM logs showing vendor_id, target_ip, commands_executed, session_duration.
  • VPN logs for vendor IPs and session endpoints.
  • Claroty logs for unusual management commands or firmware pushes.
  • SIEM correlation between vendor session and subsequent device anomalies.

T8 — DICOM Protocol Exploitation (PACS Write Abuse)

STRIDE category: Tampering
Risk: Low likelihood, High impact
Attacker objective: Abuse unauthenticated DICOM transfers to write or replace imaging data on PACS or inject malicious study files.

How the attacker would exploit it

  • Prerequisite: Network path to PACS server and knowledge of DICOM endpoints.
  • Initial action: Send crafted DICOM objects or malformed headers to PACS.
  • Core exploitation: Overwrite imaging files, corrupt studies, or attach metadata that exfiltrates PHI.
  • Escalation: Disrupt diagnosis by corrupting critical images.
  • Impact realization: Misdiagnosis, delayed treatment, regulatory breach.

Blue-team control review — what must stop the chain

  • Control 1: Isolate PACS behind strict firewall with only known modality sources allowed by IP and DICOM AE title.
  • Control 2: DICOM-aware inspection (if available) to validate study integrity and origin.
  • Control 3: PACS WORM backup and integrity checks to restore altered studies.
  • Control 4: Monitor for unusual DICOM C-STORE/C-FIND activity and large writes from unknown senders.

Evidence and telemetry to validate controls

  • PACS audit logs: source_ae, study_uid, operation, file_hash.
  • Firewall logs: denied connections to PACS from non-allowlisted IPs.
  • DICOM proxy logs showing malformed or unexpected C-STORE attempts.
  • Claroty or IDS alerts for unexpected DICOM traffic patterns.

Purple-team validation commands

T9 — Physical Patch-Panel Recabling / Insider Mispatch

STRIDE category: Tampering
Risk: Medium likelihood, Critical impact
Attacker objective: Physically rewire patch panels to move clinical device port into a guest or staff VLAN, or vice versa, enabling unauthorized access.

How the attacker would exploit it

  • Prerequisite: Physical access to wiring closet or collusion with staff; ability to patch cables.
  • Initial action: Re-patch cable to different switch or port, changing VLAN exposure.
  • Core exploitation: Device now reachable from guest/staff VLAN and is attacked remotely.
  • Escalation: Use new path to pivot to other devices or administrative consoles.
  • Impact realization: Device compromise, patient harm, or data theft.

Blue-team control review — what must stop the chain

  • Control 1: Locked wiring closets with badge access and CCTV; require change ticket for any patch changes.
  • Control 2: Daily/weekly automated switch-port inventory checks comparing physical port mapping with CMDB.
  • Control 3: Set re-auth timer for 802.1X and enforce automatic port-level re-auth on link change.
  • Control 4: Physical tamper alarms for unauthorized cabinet opening outside scheduled maintenance.

Evidence and telemetry to validate controls

  • Badge access logs correlated to wiring closet door events and switch link-up times.
  • Switch LLDP/CDP neighbor changes and interface flap logs.
  • CMDB mismatch alerts when new MAC appears on different port than inventory.
  • CCTV footage for suspicious activity in wiring closets.

Purple-team validation commands

# Detect MAC observed on different port than CMDB expected
SwitchSyslog | where Event == 'PortChange' | join CMDB on MacAddr | where Port != ExpectedPort

# Verify wiring closet badge access outside maintenance hours
