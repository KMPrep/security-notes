Scenario 23 — HR System Integration with Identity Provider — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario23-hr-idp-integration.md
Generated artifact: scenario23-hr-idp-integration-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Workday is the authoritative JML source and pushes SCIM events to Okta. Provisioning flows to 80 downstream apps; failures or token abuse can create persistent ghost accounts.

1. Review Objective

This review maps attacker paths from Workday or SCIM token compromise to enterprise-wide access, provides concrete detection telemetry (Workday audit, Okta System Log, SCIM API logs, downstream app logs), safe purple-team validation queries, and prioritized remediation steps to harden the JML pipeline.

2. Highest-Risk Design Decision

Granting a broadly scoped, long-lived SCIM token to Workday with unmonitored write privileges to the Okta directory is the highest-risk design choice. That single credential can create, modify, or delete identities across the entire enterprise; exposure (in code, repo, or via a compromised admin) results in mass account creation or modification with immediate SSO access to all downstream apps. Preferred safe pattern: narrow-scoped, short-lived tokens stored in a secrets manager, event-driven near-real-time SCIM, and GitOps-controlled provisioning policies.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Malicious HR Admin Creating Ghost Accounts

STRIDE category: Spoofing
Risk: Medium likelihood, Critical impact
Attacker objective: Create a fabricated employee record in Workday that auto-provisions into Okta with role-based access to sensitive apps.

How the attacker would exploit it

  • Prerequisite: HR admin privileges in Workday or credential compromise.
  • Initial action: Create a ghost employee record with plausible metadata (name, manager, jobcode).
  • Core exploitation: Workday triggers SCIM POST /Users to Okta with the new user attributes.
  • Escalation: Assign role that maps to admin-level app assignments (executive role) or request elevated access via workaround.
  • Impact realization: Immediate SSO access to multiple sensitive apps; persistent backdoor.

Blue-team control review — what must stop the chain

  • Control 1: Dual-approval workflow for role-elevating joiners (HR admin + HRBP + requisition ID required).
  • Control 2: Workday change governance: create events flagged in audit and require JML event approval for high-privilege roles.
  • Control 3: SCIM event quarantine: new accounts auto-quarantined until downstream verification completes for privileged roles.
  • Control 4: Daily reconciliation job: disable Okta accounts not matched to Workday active records automatically.
  • Control 5: Behavioral baseline for new accounts: high-sensitivity downloads in first 24 hours trigger automatic disable and alert.

Evidence and telemetry to validate controls

  • Workday audit: actor_id, employee_record_id, role_assigned, timestamp.
  • Okta System Log: eventType, actor, target, source (SCIM) and ip.
  • SCIM API logs: method, path, token_id, source_ip.
  • Downstream app logs showing first-time admin actions from new account.

Purple-team validation commands

OktaSystemLog | where eventType == 'user.create' and source == 'SCIM' | join (OktaSystemLog | where eventType=='user.session.start') on userId | where sessionStartTimestamp - createTimestamp < 10m

workday-cli create-user --test --role 'Director' --expect-quarantine

T2 — SCIM Token Leak in GitHub or Config

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Find SCIM token in repo or config and use it to create Okta accounts or modify existing accounts.

How the attacker would exploit it

  • Prerequisite: Token committed to a public or internal repo, or stored in plaintext in config.
  • Initial action: Attacker harvests token and issues SCIM API calls to Okta.
  • Core exploitation: Create accounts, add to admin groups, or modify existing accounts.
  • Escalation: Rotate token usage to avoid detection, blend activity over hours/days.
  • Impact realization: Persistent unauthorized access across many apps.

Blue-team control review — what must stop the chain

  • Control 1: Secrets management: no token in code; token stored only in Vault with limited access and automated rotation.
  • Control 2: Token scope restriction: token limited to create/deactivate minimal fields; cannot add to admin groups.
  • Control 3: SCIM token rotation on any Git event containing credentials via secrets scanning (pre-commit & CI enforcement).
  • Control 4: IP allow-listing: accept SCIM calls only from Workday CIDR.
  • Control 5: Monitor Git repos for secrets with DLP and block merges with tokens.

Evidence and telemetry to validate controls

  • Git history alerts and DLP scans showing token leakage.
  • SCIM API logs showing calls from unexpected IPs or high-rate patterns.
  • Okta System Log events creating many users from same token.
  • Vault audit logs showing token issuance and rotation times.

Purple-team validation commands

SCIMApiLogs | where source_ip notin (WorkdayCIDR) | summarize count() by token_id, source_ip

git log --all -p | grep -E "SCIM_TOKEN|x-api-token" || echo 'No tokens found'

T3 — Leaver De-provisioning Failure (Connector)

STRIDE category: Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Exploit connector failure to keep downstream app accounts active after Okta disable.

How the attacker would exploit it

  • Prerequisite: Connector silently fails (API rate limit, auth token expired, transient error).
  • Initial action: Okta disables user but connector does not propagate the change.
  • Core exploitation: Former employee uses remaining app sessions or API keys to access data.
  • Escalation: Use persistent app sessions to create backdoors or exfiltrate data.
  • Impact realization: Data exfiltration or unauthorized access despite Okta disable.

Blue-team control review — what must stop the chain

  • Control 1: Treat connector failures as security incidents; immediate alerts to security + ops.
  • Control 2: Post-deprovision verification: automated API checks to confirm disabled state in critical apps within 1 hour.
  • Control 3: Retry logic and dead-letter queues for transient connector failures; escalate after retries.
  • Control 4: Token-less SSO enforcement (no local creds) for critical apps where possible.
  • Control 5: Emergency rapid deprovision playbook for manual removal when automation fails.

Evidence and telemetry to validate controls

  • Connector health logs and error codes; Dead-letter queue entries.
  • Downstream app audit logs showing active sessions post-Okta disable.
  • SIEM correlation between Okta disable event and downstream access events.
  • Ticketing system alerts for connector failures with SLA breaches.

T4 — Privilege Accumulation via Mover Events

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Leverage historical access retained across multiple role changes to gain excessive privileges.

How the attacker would exploit it

  • Prerequisite: Weak recertification and no automatic removal of old-role privileges.
  • Initial action: Mover event grants new role while old access is not removed.
  • Core exploitation: Attacker uses accumulated permissions to access sensitive systems.
  • Escalation: Combine with lateral movement to escalate to privileged roles.
  • Impact realization: Unauthorized access and potential data compromise.

Blue-team control review — what must stop the chain

  • Control 1: Automated role-to-access delta: on mover event, system auto-removes previous role entitlements unless explicitly approved.
  • Control 2: Continuous entitlement monitoring with excess-access scoring and auto-recognition of privilege spikes.
  • Control 3: Mandatory recertification after each mover event for high-risk roles.
  • Control 4: Identity governance rule that blocks simultaneous membership in mutually exclusive roles.
  • Control 5: Periodic access clean-up automation (orphan & excess removal workflows).

Evidence and telemetry to validate controls

  • Identity governance platform scores and change logs.
  • Provisioning history showing add/remove events per mover.
  • Manager recertification records.
  • SIEM alerts for privilege spikes after mover events.

Purple-team validation commands

IdentityGovernance | where RoleChangeCount > 2 and ExcessAccessScore > 0.6

simulate_mover_event --user testuser --from_role 'Sales' --to_role 'Finance' --expect-role-cleanup

T5 — Stale Contractor Accounts

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Use contractor account without expiry to access systems long after contract end.

How the attacker would exploit it

  • Prerequisite: Contractor provisioning without contract end date or expiry policy.
  • Initial action: Contractor account remains active and unused until exploited.
  • Core exploitation: Attacker takes over contractor account or insider abuses access.
  • Escalation: Lateral movement to sensitive systems.
  • Impact realization: Long-lived unauthorized access and potential data exfiltration.

Blue-team control review — what must stop the chain

  • Control 1: Mandatory contract_end_date field in Workday; auto-expire in Okta at contract end.
  • Control 2: Short-lived contractor accounts by default (max 12 months) with mandatory renewal.
  • Control 3: Periodic review of contractor accounts and enforced deletion for >30 days orphaned.
  • Control 4: Less-privileged baseline for contractor roles; require additional approval for elevated access.
  • Control 5: Monitor contractor login behavior and alert on lateral access attempts.

Evidence and telemetry to validate controls

  • Okta user profile fields showing contract_end_date and expiry status.
  • Reconciliation job showing contractors with expired contracts still active.
  • Access logs showing contractor account privileged actions post-expiry.
  • Manager certification records for contractor access.

Purple-team validation commands

T6 — SCIM Token Over-Permissioned (Admin Group Add)

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Use a SCIM token that can modify groups to add attacker-controlled users to admin groups.

How the attacker would exploit it

  • Prerequisite: Token grants group modification rights or Workday misconfigured to include group attributes.
  • Initial action: SCIM PUT to add new user to admin_group.
  • Core exploitation: New user gains admin privileges in Okta and downstream apps.
  • Escalation: Create API clients, add more accounts, or create persistent admin users.
  • Impact realization: Full identity plane compromise.

Blue-team control review — what must stop the chain

  • Control 1: Principle of least privilege on SCIM token scopes: disallow group modifications.
  • Control 2: Block SCIM group changes from Workday; require separate IT workflow for admin group membership.
  • Control 3: SIEM alert for any group.user.add event where group == admin_group and source == SCIM.
  • Control 4: Short-lived tokens and rotation on any anomalous group event.
  • Control 5: Two-person approval for any admin group membership changes.

Evidence and telemetry to validate controls

  • Okta System Log for group.user.add with source SCIM.
  • Workday event logs showing whether group attribute was present in user record.
  • SIEM rule hit for admin group modifications.
  • Vault rotation logs on suspicious token use.

STRIDE category: Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Compromise an Okta admin account to alter provisioning, bypass reconciliation, or create persistent admin accounts.

  • Control 4: Break-glass controls with human approval and notifications for any admin role creation.
  • Control 5: Periodic admin access recertification and minimum admin counts.

Evidence and telemetry to validate controls

  • Okta admin audit logs: admin_user, action, ip, user_agent.
  • PAM session recordings and access logs.
  • SIEM alerts for admin actions outside maintenance windows.
