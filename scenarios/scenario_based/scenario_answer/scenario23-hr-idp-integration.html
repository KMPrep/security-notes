Scenario 23 — HR System Integration with Identity Provider

Scenario: Workday HR is the authoritative source for joiner-mover-leaver (JML) events. It pushes user lifecycle events to Okta via SCIM. Provisioning flows to 80 downstream apps.

Candidate questions:

  • What is the threat model for a malicious HR admin creating ghost accounts?
  • How do you detect a leaver whose account remains active past their last day?
  • What controls enforce timely de-provisioning across all 80 downstream apps?
  • How do you handle provisioning failures that leave an account in an inconsistent state?
  • What monitoring detects SCIM token abuse or unauthorized provisioning calls?

3.4 Trust boundary analysis

The critical trust boundary is between Workday HR (data source) and Okta (enforcement point). Workday is trusted implicitly — whatever Workday says about an employee's status, Okta acts on immediately, without secondary verification. This trust relationship is correct in design but catastrophic if Workday is compromised or if an HR admin acts maliciously. The SCIM token is the literal key to the identity kingdom: it has write access to the Okta user directory with no user-level approval gate. The second critical boundary is between Okta (the SSO broker) and the 80 downstream apps — this boundary is only as strong as the provisioning connector for each app. Connector failures silently break the de-provisioning chain.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale (200,000+ employees, thousands of role changes per day), the JML pipeline must be fully automated with zero manual intervention for standard cases. All provisioning rules are defined as code (Okta Terraform provider, SCIM policy configurations), version-controlled, and audited. Role-to-access mappings are maintained in a centralized RBAC catalog that is the single source of truth for what access a given job family gets. Joiner provisioning is deterministic: given job family X, the system grants exactly the access defined in the RBAC catalog — no HR admin or IT admin judgment call needed.

8.2 Fleet-wide impact analysis

A SCIM token compromise at scale means an attacker can create accounts for all job families and roles across the entire enterprise. More critically, a malfunction in the SCIM pipeline (e.g., a bad Workday update that sends mass leaver events) could trigger mass de-provisioning of active employees — a productivity disruption affecting tens of thousands of employees simultaneously. Change management for SCIM configuration must include a rollback plan and testing in a non-production tenant before deployment.

8.3 Eliminating human bottlenecks

The human bottleneck is manager access certification. With thousands of managers certifying access for their reports across 80 apps, the process collapses without automation. The solution: machine learning access review assistance (Saviynt AI, SailPoint IdentityAI) that pre-scores each access item as "likely appropriate" or "likely excess" based on peer group analysis, reducing manager decisions to only the flagged excess items rather than all access. This reduces review time from days to hours per manager.

8.4 Controls that break at scale

Manual orphan account remediation — with 200,000 employees in flux, manual investigation of each orphaned account is impossible; requires automated remediation policies (orphan > 7 days → auto-disable; orphan > 30 days → auto-delete with 14-day archive). Per-app connector monitoring — monitoring 80 connector health states manually fails; requires automated connector health dashboard with SLA alerts on de-provisioning lag. Manual access recertification campaigns — quarterly certification of all access for all employees manually reviewed by managers does not scale; requires risk-based certification scope (only flagged excess access requires review).

ANSWER SUMMARY — 90-second version

"The JML pipeline is the foundation of enterprise identity governance — if Workday is the authority and Okta is the enforcement point, the integrity of that pipeline determines whether former employees retain access and whether unauthorized accounts can be created at scale.

My top concern is the SCIM token. This token has write access to the Okta directory and can create accounts for any role in the organization. It must never be in code or config files — stored in Vault, rotated quarterly, scoped to minimum permissions (create and deactivate users; no admin group modification). Any SCIM call from an IP outside the Workday IP range is an immediate critical alert.

For ghost account creation by a malicious HR admin, my detective control is the HR-to-Okta reconciliation job: any active Okta account without a corresponding active Workday record is flagged within 24 hours. Separately, any new account with an elevated role (VP, director) created outside of an approved headcount requisition triggers a dual-approval workflow.

For leavers, the biggest failure mode is silent connector failures — Okta says the user is disabled, but the Salesforce connector failed and that account is still active. I address this with a post-de-provisioning verification job that independently confirms disabled status in each high-risk app within 1 hour of Okta de-provisioning.

The residual risk I honestly document is mover privilege accumulation — employees who change roles 3 times in 3 years often retain access from all previous roles. Without quarterly access recertification backed by peer group analysis, this silently accumulates."

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: How do you detect a leaver whose account remains active past their last day?

Three complementary controls: (1) Real-time alerting — SCIM leaver event triggers a timer; if Okta account is not disabled within 60 minutes, an automated alert fires to IT Ops and security; (2) Daily reconciliation job — compares HR termination list to Okta active account list; any mismatch generates a report that is reviewed and remediated same-day; (3) Downstream app verification job — for high-risk apps (GitHub, AWS IAM, Salesforce), a daily job queries each app's user directory and compares active accounts to the Okta active account list; any account active in a downstream app but disabled in Okta triggers immediate manual de-provisioning. The combination of preventive (real-time SCIM), detective (daily reconciliation), and verification (downstream app check) provides defense-in-depth for the leaver scenario.

Follow-up 2: What controls enforce timely de-provisioning across 80 downstream apps?

The most important control is treating connector failure as a security event, not just an IT ops incident. When an Okta provisioning connector fails, it must alert the security team (not just IT ops), not silently retry for days. For critical apps (cloud, code, finance), I require independent verification within 1 hour of Okta de-provisioning. For lower-risk apps, a daily batch verification is sufficient. Architecturally, I prefer apps that use Okta SSO exclusively with no local credentials — if the Okta account is disabled, the user simply cannot log in regardless of whether the connector ran. Apps with local credential bypass (local admin accounts, API keys not tied to SSO) require explicit de-provisioning verification.

Follow-up 3: How do you handle provisioning failures that leave accounts in inconsistent states?

An account that exists in Okta but not in Salesforce (provisioning failure during join) is a known failure mode. My approach: (1) The provisioning connector logs the failure and immediately opens an IT ticket with app name, user, and error code; (2) Retry logic runs with exponential backoff for transient failures (API rate limit); (3) If retry fails after 3 attempts, a human IT admin is paged to manually complete provisioning; (4) The user is notified of the delay and given a temporary workaround if business-critical. The worst case for a joiner provisioning failure is an annoyed new hire; the worst case for a leaver de-provisioning failure is a former employee with active access — asymmetric risk, so leaver failures have a shorter SLA and are treated as security incidents.

Follow-up 4: What monitoring detects SCIM token abuse or unauthorized provisioning calls?

Two primary signals: (1) Source IP filtering — Okta logs the source IP of every SCIM API call. Workday's outbound SCIM calls come from known IP ranges. Any SCIM call (user create, group modify, user delete) from an IP not in the Workday range is a critical alert; (2) Unusual provisioning patterns — SCIM calls during overnight hours when no hiring is expected, bulk user creation events, or modification of high-privilege groups via SCIM. I also alert on any SCIM call that adds a user to an admin group — Workday should never do this; it's an explicit permission that should not be in the SCIM token scope, and any such event is likely either a misconfiguration or an attack.

Follow-up 5: Walk me through your threat model for a malicious HR admin creating ghost accounts.

The attack: the HR admin creates a fictitious employee record in Workday — real-looking name, valid manager, appropriate job title and business unit. Workday auto-fires a SCIM joiner event to Okta. Okta creates the user, assigns them to the group corresponding to their job title, and provisions access to all 80 downstream apps automatically. The HR admin then authenticates to Okta as the ghost user, has full access to everything a legitimate employee in that role would have, and can exfil data or maintain a persistent backdoor.

My detective controls: (1) Any new hire with a role above IC level triggers a dual-approval review (HR manager + HRBP must independently confirm the hire against the approved headcount list); (2) The daily reconciliation job flags any Okta account that doesn't have a matching approved headcount requisition in Workday; (3) Behavior analytics on the new account — a legitimate new hire takes days to ramp up; an account that immediately accesses sensitive systems and downloads data in the first 24 hours is anomalous. My preventive control is separating the ability to create employee records (HR admin) from the ability to grant elevated access (IT admin and RBAC catalog) — Workday SCIM only provisions role-standard access; any access above the role standard requires a separate IT access request workflow with approval.

COMMON MISTAKES TO AVOID

  • Treating SCIM as a one-way, always-correct fire-and-forget system. Connectors fail silently. De-provisioning is not complete until independently verified in each downstream app.
  • Over-trusting Workday as an attack-proof authoritative source. A malicious HR admin or a compromised Workday account can inject false data into the JML pipeline. Downstream verification controls are essential.
  • Not addressing mover privilege accumulation. Most candidates focus on joiner and leaver but miss that movers accumulate excessive access over multiple role changes — often the highest-risk residual identity risk in mature enterprises.
  • Storing the SCIM token in a config file or environment variable. The SCIM token is a credential with write access to the identity plane. It deserves the same protection as a root credential.
  • Missing the de-provisioning timing gap. Many enterprises run overnight SCIM batch jobs, leaving former employees with active accounts for 8–12 hours after their termination. This is a known risk that must be addressed.
  • Not considering the blast radius of a bad SCIM event. A misconfigured Workday SCIM push could trigger mass de-provisioning of active employees. Change management for SCIM configuration changes must include rollback procedures.
  • Forgetting about contractor and service account lifecycle. The JML pipeline typically focuses on full-time employees. Contractors and service accounts often have no automated expiry and accumulate indefinitely.
