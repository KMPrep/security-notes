Scenario 24 — Security Awareness Training Platform — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario24-security-awareness-platform.md
Generated artifact: scenario24-security-awareness-platform-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: A third-party phishing simulation platform is granted production email send access; if the vendor or its credentials are compromised, attackers can send authenticated phishing at enterprise scale.

1. Review Objective

This review evaluates how an attacker would exploit the security-awareness vendor integration (attacker lens) and what blue-team controls, telemetry, and runbooks must exist to prevent, detect, and contain each threat (blue-team lens). It focuses on command-level validation queries, specific log sources and fields, and prioritized remediation.

2. Highest-Risk Design Decision

Authorizing the vendor to send email that is perceived as originating from the primary corporate domain (by granting SPF/DKIM/DMARC equivalence or providing DKIM signing keys) is the single most consequential design decision. If the vendor is authorized on the primary domain, compromise results in attacker-sent email that will pass enterprise DMARC checks and look indistinguishable from legitimate comms — this enables large-scale spearphishing with minimal detection. Preferred alternative: enforce a dedicated simulation subdomain (e.g., phishsim.company.com) with isolated SPF/DKIM, send-only OAuth scopes, IP allow-lists, and automated revocation workflows.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Vendor SMTP Relay Credential Compromise

STRIDE category: Spoofing
Risk: Medium likelihood, Critical impact
Attacker objective: Use vendor SMTP credentials to send real phishing that bypasses filters and lands in employee inboxes.

How the attacker would exploit it

  • Prerequisite: Access to vendor SMTP credentials (leak, mismanaged secret, insider).
  • Initial action: Authenticate to vendor relay or vendor API and send emails as vendor's legitimate sender.
  • Core exploitation: Craft high-fidelity spearphish (CEO wire requests, credential reset) using vendor templates.
  • Escalation: Target high-value personnel (finance, executives) via tailored templates.
  • Impact realization: Credential harvest, fraudulent transactions, lateral compromise.

Blue-team control review — what must stop the chain

  • Control 1: Use a dedicated simulation subdomain (phishsim.company.com) — never authorize vendor to send from primary corporate domain company.com (DNS/TXT SPF + DKIM keys per subdomain).
  • Control 2: IP allow-listing for vendor SMTP relay and strict SPF record entries (exact IPs only), automated monitoring for SPF TXT changes via DNS monitoring service (Cloudflare/NS1 webhook).
  • Control 3: Exchange/Google mail flow policy (Transport Rule) that rejects or flags any message claiming to be from @company.com if SenderIp in vendor_list and From domain != simulation subdomain.
  • Control 4: DMARC aggregate (RUA) ingestion into SIEM and alert on any DMARC pass for primary domain with vendor IPs.
  • Control 5: Automated revocation playbook: revoke SMTP credentials (vendor API), remove vendor IPs from SPF via IaC, and emergency employee push notification.

Evidence and telemetry to validate controls

  • O365 Message Trace / Google Postmaster fields: SenderIP, FromAddress, RecipientAddress, MessageId, Received.
  • DMARC RUA reports: source_ip, header_from, spf_result, dkim_result.
  • Mail flow transport rule logs (Exchange Admin Center): RuleName, Action, MessageId.
  • DNS change logs from registrar / DNS provider showing SPF/DKIM updates.

Purple-team validation commands

Purpose: Detect vendor sends outside scheduled campaign window and from unexpected IPs.

Get-MessageTrace -SenderAddress "no-reply@phishsim.company.com" -StartDate (Get-Date).AddDays(-7) | Where-Object { $_.SenderIP -notin $VendorIpList -or $_.Received -lt $CampaignStart -or $_.Received -gt $CampaignEnd }

EmailEvents
| where SenderIP in ("<vendor_ip_1>","<vendor_ip_2>")
| where TimeGenerated < datetime(<campaign_start>) or TimeGenerated > datetime(<campaign_end>)
| summarize cnt=count() by SenderIP, bin(TimeGenerated,1h)

T2 — SPF/DKIM/DMARC Over-Authorization (Primary Domain)

STRIDE category: Spoofing
Risk: Low likelihood, Critical impact
Attacker objective: If vendor is authorized on the primary domain, attacker-sent email passes DMARC and looks indistinguishable from corporate email, enabling high-success spearphishing.

How the attacker would exploit it

  • Prerequisite: Vendor added to primary domain SPF or given DKIM signing key.
  • Initial action: Attacker crafts phishing email and sends via vendor channel.
  • Core exploitation: Emails pass DMARC checks; mail filters allow through to inboxes.
  • Escalation: Targeted credential harvest across org; lateral movement.
  • Impact realization: Widespread compromise and fraud.

Blue-team control review — what must stop the chain

  • Control 1: Policy: prohibit vendor DKIM keys or primary-domain signing for simulation vendors; use subdomain-only DKIM.
  • Control 2: DNS monitoring alerts for any TXT change to SPF/DKIM (Cloud DNS notifications; Registrar webhook) with automatic roll-back capability in IaC.
  • Control 3: DMARC policy p=reject for primary domain with strict RUA ingest and daily automated review.
  • Control 4: Use MTA-STS and TLS-RPT to reduce chances of interception; monitor TLS-RPT for unauthorized endpoints.

Evidence and telemetry to validate controls

  • DNS provider change logs showing SPF TXT edits and DKIM record additions.
  • DMARC aggregate reports (RUA) and report_id fields for forensic correlation.
  • TLS-RPT logs indicating unexpected TLS endpoints for outbound mail.

Purple-team validation commands

Purpose: Validate that primary domain DKIM/SPF remain unchanged and that only the simulation subdomain is authorized.

dig TXT company.com @8.8.8.8
dig TXT phishsim.company.com @8.8.8.8

DnsChangeLogs | where Resource == "company.com" and RecordType == "TXT" and TimeGenerated > ago(7d)

T3 — Employee List Export by Malicious Vendor Insider

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Export employee target lists and clicker data to enable targeted real phishing campaigns.

How the attacker would exploit it

  • Prerequisite: Vendor admin or insider with access to ResultsDB/target lists.
  • Initial action: Export or copy employee CSV or database table containing emails and click metadata.
  • Core exploitation: Sell or use the list to craft highly targeted campaigns.
  • Escalation: Combine with public OSINT to create convincing spearphish.
  • Impact realization: Targeted credential theft and confidentiality breaches.

Blue-team control review — what must stop the chain

  • Control 1: Limit vendor access: use per-campaign upload of target lists (no standing full-directory API) and short-lived, campaign-scoped tokens.
  • Control 2: Technical DLP on vendor storage (S3 bucket policies, encryption with customer-managed keys, audit logs) and deny bulk exports from ResultsDB.
  • Control 3: Contractual + technical controls: vendor must use SSE-C or KMS with customer-managed CMK and provide audit logs (CloudTrail/S3 access logs) to enterprise.
  • Control 4: Monitor vendor API access patterns and alert on large SELECT or EXPORT operations (CloudTrail / vendor audit logs) — require dual-approval for bulk exports.

Evidence and telemetry to validate controls

  • Vendor audit logs / CloudTrail: GetObject, ListObjects, StartExport events with actor_id and source_ip.
  • ResultsDB access logs: query_id, user_id, rows_returned.
  • S3 / object storage access logs showing large data transfer and unusual destination IPs.

Purple-team validation commands

Purpose: Detect large exports from vendor ResultsDB or object storage.

VendorAuditLogs | where OperationName in ('StartExport','GetObject') and BytesTransferred > 100000000 | project TimeGenerated, Actor, OperationName, BytesTransferred

aws s3api list-objects --bucket vendor-results --query "Contents[?Size>100000000].[Key,Size]"

T4 — Click-Through / Credential Data Exfiltration

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Steal credential submissions or click data from vendor ResultsDB and use it to target high-risk users.

How the attacker would exploit it

  • Prerequisite: Vendor stores click/credential data in ResultsDB or analytics store.
  • Initial action: Attacker gains access to ResultsDB (compromise, misconfigured ACLs, insider).
  • Core exploitation: Extract clickers and credential submissions; map to sensitive roles.
  • Escalation: Use harvested credentials or targeted campaigns.
  • Impact realization: Account takeover and data exfiltration.

Blue-team control review — what must stop the chain

  • Control 1: Reduce retention and avoid storing raw credentials. Configure vendor to record click events without storing submitted credentials; enable redaction and one-way hashes for PII.
  • Control 2: Encryption at rest with customer-managed keys (CMK) and access controls limiting read to named service accounts only.
  • Control 3: SIEM correlation for ResultsDB access anomalies and network egress detection to unknown IPs.
  • Control 4: Periodic integrity checks and penetration tests of vendor storage.

Evidence and telemetry to validate controls

  • ResultsDB access logs: query, actor, rows_returned, timestamp.
  • Vendor storage audit logs and CloudTrail events for object retrieval.
  • Network egress logs showing data flows to external IPs.

Purple-team validation commands

Purpose: Verify no plaintext credentials are retained and detect large retrievals.

ResultsDbLogs | where Operation == 'SELECT' and Table == 'submissions' and TimeGenerated > ago(30d) | summarize count() by Actor

T5 — Simulation Template Reuse in Real Attacks

STRIDE category: Tampering
Risk: Low likelihood, High impact
Attacker objective: Steal or reuse vendor templates so attackers can craft realistic, previously-seen simulation emails that employees are desensitized to.

How the attacker would exploit it

  • Prerequisite: Access to vendor template library (leak, insider, misconfigured S3 bucket).
  • Initial action: Exfiltrate templates matching high-value scenarios (exec impersonation, finance requests).
  • Core exploitation: Send real-world phishing using exact templates employees have seen in simulation.
  • Escalation: Increase success by timing real attacks when simulations are frequent (desensitization).
  • Impact realization: Higher click and compromise rates.

Blue-team control review — what must stop the chain

  • Control 1: Protect template storage with strict ACLs and customer-managed CMKs; no public S3 buckets; enable object-level logging.
  • Control 2: Rotate and vary templates regularly; use generated templates that avoid reuse patterns.
  • Control 3: Contract clause banning template reuse and right-to-audit vendor template access.
  • Control 4: Monitor threat intelligence feeds for reuse of known simulation templates in external campaigns.

Evidence and telemetry to validate controls

  • Object storage access logs and CloudTrail: GetObject events for template directories.
  • Threat intel matches: similarity hashes of phishing samples vs. vendor templates.
  • SIEM alerts correlating external phishing with internal template fingerprints.

T6 — Overprivileged Vendor OAuth App (Mail.Read / Mail.ReadAll)

STRIDE category: Information Disclosure / Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Use vendor OAuth app permissions to read employee mailboxes or directory data.

How the attacker would exploit it

  • Prerequisite: Admin consent granted to vendor app with excessive scopes (Mail.ReadAll, Directory.ReadAll).
  • Initial action: Attacker or malicious vendor uses token to call Graph API and read mail or contacts.
  • Core exploitation: Harvest PII, lead information, and privileged communications.
  • Escalation: Use sensitive mailbox data to craft targeted attacks or pivot to other systems.
  • Impact realization: Confidentiality breach and targeted compromise.

Blue-team control review — what must stop the chain

  • Control 1: Enforce least privilege for OAuth scopes: grant only Mail.Send for sending; deny Mail.Read and Mail.ReadWrite at tenant consent policies.
  • Control 2: Monitor Azure AD sign-in logs and conditional access for the vendor app (SignInLogs: ResourceDisplayName, AppId, ClientAppUsed).
  • Control 3: Use Azure AD Consent policies to require admin justification and block high-risk scopes (Azure AD portal / Conditional Access).
  • Control 4: SIEM rule for ApplicationConsent or unusual Graph API calls (e.g., GET /users/*/messages) with alerting and automated token revocation.

Evidence and telemetry to validate controls

  • Azure AD SignInLogs and AuditLogs: AppId, ResourceDisplayName, UserPrincipalName, IpAddress.
  • O365 Unified Audit Log entries for Graph API calls: Operation = MailItemsAccessed, UserId, ClientIP.
  • Conditional Access sign-in risk fields and risk detections.

Purple-team validation commands

Purpose: Detect vendor OAuth app usage of non-send permissions.

T8 — Vendor Supply-Chain Compromise Enabling Mass Spearphish

STRIDE category: Tampering / Spoofing
Risk: Low likelihood, Critical impact
Attacker objective: Compromise vendor infrastructure or pipeline to deliver authenticated spearphishing at scale to the entire organization.

How the attacker would exploit it

  • Prerequisite: Compromise of vendor CI/CD, credentials, or an insider threat at vendor.
  • Initial action: Push malicious campaign or steal keys; trigger mass sends via vendor platform.
  • Core exploitation: Use vendor's authorized infrastructure to deliver targeted spearphish bypassing filters.
  • Escalation: Long-term espionage or ransomware via social engineering.
  • Impact realization: Enterprise-wide compromise and data loss.

Blue-team control review — what must stop the chain

  • Control 1: Treat vendor as tier-1 supplier: require SOC 2 Type II, supply-chain security attestations, and incident notification SLA (1 hour).
  • Control 2: Implement canary campaigns and rate limits: bulk sends > X require manual out-of-band approval and signed manifests.
  • Control 3: Automated rollback and emergency revocation playbooks: IaC-managed SPF/DKIM updates and automated removal of vendor IPs.
  • Control 4: Threat intel and DMARC correlation to detect spikes in vendor-originated mail flow for off-hours.

Evidence and telemetry to validate controls

  • SIEM mail flow anomaly alerts and DMARC spike reports.
  • Vendor incident reports and CloudTrail-like logs showing mass send events.
  • Mail gateway logs with high send volume from vendor IPs.

T9 — Off-Schedule Vendor Sending (Credential Misuse)

STRIDE category: Spoofing / Tampering
Risk: Medium likelihood, High impact
Attacker objective: Use stolen credentials to send phishing outside scheduled campaign windows to maximize confusion and success.

How the attacker would exploit it

  • Prerequisite: Stolen vendor SMTP/API token or compromised vendor employee session.
  • Initial action: Send campaigns at unexpected times to avoid immediate comparison against scheduled campaigns.
  • Core exploitation: Send to targeted high-value employees with crafted templates.
  • Escalation: Blend activity to avoid volume-based detection.
  • Impact realization: Rapid credential theft before revocation.

Blue-team control review — what must stop the chain

  • Control 1: Enforce campaign schedule metadata and compare mail flow to scheduled windows; any vendor send outside windows triggers an automated incident.
  • Control 2: SIEM rule PHISH-001 (mail_trace where sender_ip in vendor_list and time not in campaign window) with high-priority alerting.
  • Control 3: Token binding: restrict token usage by IP or require mTLS between vendor and relay.
  • Control 4: Automated throttle and reject for unknown senders or rates exceeding configured max per hour.

Evidence and telemetry to validate controls

  • Mail flow trace vs. campaign schedule database.
  • SIEM alert logs for PHISH-001 and vendor-auth events.
  • TLS mutual auth logs and mTLS client cert rotation events.

T10 — Measurement Misuse & Desensitization

STRIDE category: Repudiation / Information Disclosure
Risk: Medium likelihood, Medium impact
Attacker objective: Not an external attacker per se — the program itself causes behavioral changes and legal/HR risk (e.g., employees penalized for clicks).

How the issue manifests

  • Prerequisite: Security program uses individual-level click data in HR or public reporting.
  • Initial action: Managers use click results for performance or disciplinary action.
  • Core exploitation: Employees hide evidence, share simulation emails, or ignore real reporting.
  • Escalation: Program effectiveness drops and real phishing success increases.
  • Impact realization: Reduced security posture and legal exposure.

Blue-team control review — what must stop the chain

  • Control 1: Policy: never share individual results outside security team; reports to HR are aggregate only.
  • Control 2: Use cohort-based KPIs (department/role) instead of individual names in external dashboards.
  • Control 3: Limit retention of individual-level data; access only by named analysts with audit logging.
  • Control 4: Design simulations as click-only where possible to avoid credential capture and legal risk.

Evidence and telemetry to validate controls

  • Platform access logs showing who viewed individual reports.
  • HR process docs and DPA demonstrating permitted uses of data.
  • SIEM alerts for bulk exports of individual-level datasets.
