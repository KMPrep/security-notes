Scenario 26 — Multi-Tenant SaaS Application on Google Cloud — Threat Exploitation and Blue-Team Control Review

Scenario: Logical multi-tenancy in a single GCP project for 500 enterprise customers. Focus: tenant isolation, IDOR, token abuse, admin control-plane risks, and large-scale export/exfiltration.

1. Review Objective

This document analyzes each threat to tenant isolation and the admin control plane through two lenses: attacker mechanics (how the adversary would exploit the weakness) and blue-team controls (specific preventive, detective, and corrective controls with telemetry and safe validation commands). Every threat has STRIDE, risk, concrete exploitation steps, control breakpoints, evidence sources, purple-team validation examples, and a small attack tree.

2. Highest-Risk Design Decision

The single highest-risk design decision is relying on purely logical isolation in a shared GCP project without enforcing defense-in-depth at the data plane (row-level predicates, signed tenant context propagation, and automated inventory enforcement). If tenant binding is not cryptographically or policy-enforced end-to-end, a single IDOR or token claim mismatch can enable cross-tenant reads and writes that scale to many customers. Preferred safe choice: combine logical isolation with enforceable runtime invariants (policy engine, DB RLS, signed context headers) and cell-based segmentation for high-regulated tenants.

6. Threat-by-Threat Technical Review

T1 — IDOR on data API

STRIDE category: Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Read or modify tenant B records using crafted object identifiers or query parameters against an endpoint missing tenant enforcement.

How the attacker would exploit it

  • Prerequisite: Valid tenant-scoped account or stolen token for tenant A.
  • Initial action: Enumerate object IDs and endpoints while monitoring 401/403/404 to 200 transitions.
  • Core exploitation: Find object IDs that return data for tenant B when tenant_id parameter is missing or ignored.
  • Escalation: Automate enumeration to retrieve bulk records across tenants.
  • Impact realization: Confidential data exfiltration and contractual breach.

Blue-team control review — what must stop the chain

  • Control 1: Enforce tenant_id binding at the authorization service (OPA/Cedar) and reject requests where binding is missing.
  • Control 2: DB row-level security (RLS) predicates in Cloud SQL that enforce tenant filter at the storage layer.
  • Control 3: API gateway invariants: reject requests lacking signed internal tenant context header.
  • Control 4: Rate-limit and detect sequential numeric ID access patterns at the gateway.
  • Control 5: Hardened CI tests that include property-based authz tests for object access invariants.

Evidence and telemetry to validate controls

  • API audit logs with request_id, sub, tenant_id, and status codes showing 404->200 pattern.
  • Cloud SQL query logs showing absence of tenant predicate in statements.
  • Authorization decision logs from policy engine showing enforcement decisions.
  • WAF/gateway metrics for id enumeration spikes.

Purple-team validation commands

KQL detection query to find 404 to 200 IDOR pattern:

APILogs
| where route startswith "/objects/" and statusCode in (200,404,403)
| summarize counts=make_list(statusCode) by caller=sub, route
| where array_length(list_filter(counts, x -> x==200))>0 and array_length(list_filter(counts, x -> x==404))>0

curl -H "Authorization: Bearer <token-tenant-a>" https://api.example.com/objects/<tenant-b-id>

T2 — Missing tenant predicate in analytics queries

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Use reporting or analytics endpoints to retrieve aggregated or raw rows from multiple tenants when queries omit tenant filters.

How the attacker would exploit it

  • Prerequisite: Access to reporting API or compromised internal analytics account.
  • Initial action: Submit complex queries or exports targeting shared tables.
  • Core exploitation: Reporting engine executes query without tenant predicate and writes output to staging path.
  • Escalation: Download exported file via staging signed URL.
  • Impact realization: Bulk cross-tenant data exposure.

Blue-team control review — what must stop the chain

  • Control 1: Enforce parameterized query templates in analytics engine that require tenant parameter.
  • Control 2: Job-level policy that validates and simulates query plan for tenant predicate presence before execution.
  • Control 3: Export staging paths are tenant-scoped and validated against requestor tenant_id.
  • Control 4: SIEM rule to alert on any analytics job that writes to staging with record counts above threshold.
  • Control 5: Require operator review for cross-tenant joins or ad-hoc queries via JIT approval.

Evidence and telemetry to validate controls

  • Analytics job logs showing query text and bound parameters.
  • Staging path creation logs with tenant_id tag.
  • GCS access logs for staging path reads and signed URL generation events.
  • SIEM alerts for export job size anomalies.

Purple-team validation commands

T3 — JWT audience or tenant claim misvalidation

STRIDE category: Spoofing
Risk: Medium likelihood, High impact
Attacker objective: Present a token with crafted aud or tenant claim to bypass gateway validation and perform unauthorized actions.

How the attacker would exploit it

  • Prerequisite: Knowledge of token claim schema or weak audience checks in gateway rules.
  • Initial action: Forge or reuse a token issued for a different client or tenant where audience checks are insufficient.
  • Core exploitation: Gateway forwards request; downstream services consume tenant claim and act incorrectly.
  • Escalation: Use accepted token to perform admin or export operations.
  • Impact realization: Unauthorized tenant access and lateral abuse.

Blue-team control review — what must stop the chain

  • Control 1: Strict token validation in the API gateway: issuer, audience, nonce, and tenant binding checks.
  • Control 2: Use token introspection for high-risk actions and verify token use context (client_id, origin, device).
  • Control 3: Short token TTL and rotation, combined with binding refresh tokens to client fingerprints.
  • Control 4: Downstream services must not trust opaque headers; validate tokens centrally or use signed internal headers.
  • Control 5: Detect token use from unexpected ASNs or new geos via token telemetry.

Evidence and telemetry to validate controls

  • Gateway token validation logs showing issuer and audience checks.
  • IdP introspection logs for token exchange and revocation events.
  • Token replay detection alerts and ASN/geolocation mismatches.
  • Authorization decision logs indicating tenant binding decisions.

Purple-team validation commands

KQL detect tokens used from multiple ASNs quickly:

TokenUsage
| where jti == "<jti>"
| summarize distinct_asns=dcount(asn) by jti, bin(TimeGenerated,10m)
| where distinct_asns > 1

T4 — Refresh token theft and replay

STRIDE category: Spoofing
Risk: Medium likelihood, High impact
Attacker objective: Use stolen refresh tokens to obtain new access tokens and maintain long-lived access to tenant resources.

How the attacker would exploit it

  • Prerequisite: Cross-site scripting, stolen browser storage, or compromised CI secret store containing refresh tokens.
  • Initial action: Redeem refresh token at IdP to obtain new access tokens.
  • Core exploitation: Use access tokens to call APIs and exfiltrate data or escalate actions.
  • Escalation: Rotate refresh tokens in victim account or request new refresh tokens via phishing flows.
  • Impact realization: Persistent account takeover and data theft.

Blue-team control review — what must stop the chain

  • Control 1: Bind refresh tokens to client fingerprint and rotate refresh family on suspicious use.
  • Control 2: Shorter refresh token lifetime for high-risk clients and use refresh token revocation endpoints.
  • Control 3: Detect refresh token redemption from unusual geos/ASNs and enforce step-up authentication.
  • Control 4: Monitor and protect developer secret stores and CI artifacts where refresh tokens could be stored.
  • Control 5: Use refresh token rotation and one-time use semantics.

Evidence and telemetry to validate controls

  • IdP logs showing refresh token redemption, jti, client_id, and source IP/ASN.
  • Token family revocation events and re-authentication records.
  • SIEM correlation for token redemption followed by high-volume data access.
  • CI/CD secret access logs for tokens used in build or deployment pipelines.

Purple-team validation commands

T5 — Support impersonation abuse

STRIDE category: Elevation of Privilege
Risk: Low likelihood, Critical impact
Attacker objective: Use legitimate support tooling and impersonation capabilities to access other tenants without consent.

How the attacker would exploit it

  • Prerequisite: Compromised or malicious support operator credentials or abused break-glass flow.
  • Initial action: Use support console impersonation to assume tenant context and perform reads/exports.
  • Core exploitation: Download exports or run high-privilege queries under impersonation.
  • Escalation: Alter tenant settings or inject backdoors into tenant objects.
  • Impact realization: Large-scale data breach from privileged operator abuse.

Blue-team control review — what must stop the chain

  • Control 1: JIT privileged access with approval, non-bypassable ticket ID, and time-limited tokens for support operations.
  • Control 2: Immutable audit trail of impersonation actions with tamper-evident checksums and out-of-band notifications to tenant owners.
  • Control 3: Dual control for very high-risk actions (two operators or manager approval) and MFA for support sessions.
  • Control 4: Monitor impersonation usage for anomalous exports and cross-check ticket metadata.
  • Control 5: Limit the surface area of impersonation to non-sensitive read-only operations by default.

Evidence and telemetry to validate controls

  • Admin API logs with operator_id, ticket_id, impersonated_tenant, and action details.
  • Tamper-evident audit chain logs and checksum verification events.
  • SOAR automated playbook runs that validate ticket authenticity.
  • Recipient download logs for exported files.

T6 — Over-permissioned service account

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Use a mis-scoped service account to directly read datastore or object storage bypassing app-layer authz.

How the attacker would exploit it

  • Prerequisite: Misconfigured IAM binding granting broad roles to service account or compromised workload identity.
  • Initial action: Use stolen keys or bound identity to call GCS or Cloud SQL directly.
  • Core exploitation: Access tenant objects without tenant-scoped checks, or modify data plane ACLs.
  • Escalation: Pivot to other services and issue long-lived credentials.
  • Impact realization: Platform-wide data access and potential ransomware staging.

Blue-team control review — what must stop the chain

  • Control 1: Least-privilege IAM with role boundaries and conditional bindings tied to resource attributes.
  • Control 2: Workload Identity Federation and removal of long-lived keys from dev workflows.
  • Control 3: Automated IAM drift detection and policy-as-code enforcement in PR gates.
  • Control 4: Service account usage alerts when calling high-risk APIs (Cloud SQL, GCS) from unexpected hosts.
  • Control 5: Enforce VPC-SC or private service perimeter for sensitive storage access.

Evidence and telemetry to validate controls

  • Cloud Audit Logs showing serviceAccount principal calls to data APIs.
  • IAM policy change logs and PR provenance for role grants.
  • VPC flow logs and egress patterns from abnormal hosts.
  • SIEM correlation for service account usage pattern shifts.

T7 — Insecure signed URL or bucket path access

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Obtain or guess signed URLs or misuse signed URL generation to access objects outside tenant prefix.

How the attacker would exploit it

  • Prerequisite: Weak signed URL generation logic or predictable object keys.
  • Initial action: Guess signed URL tokens or exploit token leakage from logs.
  • Core exploitation: Download tenant objects directly from GCS using signed URL.
  • Escalation: Automate scanning of signed URLs or abuse exposed staging buckets.
  • Impact realization: Object download and exposure of PII and secrets.

Blue-team control review — what must stop the chain

  • Control 1: Signed URL generation service enforces caller tenant_id and short TTL and embeds nonce bound to ticket.
  • Control 2: Signed URL tokens are unpredictable and never logged in plaintext; redaction of logs.
  • Control 3: Staging buckets enforce per-tenant prefixes and bucket policies to deny cross-prefix reads.
  • Control 4: Monitor signed URL creation and downloads and alert on high-volume or cross-tenant access.
  • Control 5: Use ephemeral, one-time download tokens for high-risk exports.

Evidence and telemetry to validate controls

  • GCS access logs showing object_name, requester, and referer.
  • Signed URL creation events with token metadata and TTL.
  • Application logs showing redaction of signed URL values.
  • SIEM alerts for cross-prefix reads.
