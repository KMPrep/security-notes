Scenario 27 — SaaS Authentication: OAuth2 + OIDC — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario27-saas-oauth2-oidc.md
Generated artifact: scenario27-saas-oauth2-oidc-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: SaaS platform exposing social login and enterprise SSO (OIDC/SAML) with long-lived API keys and programmatic access; focus on token/claim handling, redirect URI security, API key lifecycle, and admin-plane protections.

1. Review Objective

This review analyzes attacker paths that exploit OAuth2/OIDC and API-key implementation gaps and evaluates blue-team controls required to prevent, detect, and contain compromise. It emphasizes concrete attacker steps, telemetry fields, safe purple-team validation commands, and per-threat attack trees with control breakpoints.

2. Highest-Risk Design Decision

The single most consequential decision is allowing multiple trust sources and long-lived credentials (social IdPs, enterprise IdPs, API keys, refresh tokens) without a single enforceable account-binding and revocation model. If external assertions are accepted and linked to internal accounts without strict claim validation and a fast, comprehensive revoke mechanism, an attacker can convert a single assertion or leaked key into persistent cross-account access. Preferred safe choice: unify binding and revocation via a central token authority, require strict claim checks, and force short credentials or strong sender-binding for high-risk flows.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Open redirect URI abuse

STRIDE category: Spoofing
Risk: Critical likelihood, High impact
Attacker objective: Trick user into redirecting auth code to attacker-controlled callback, obtain authorization code and redeem it for tokens.

How the attacker would exploit it

  • Step 1: Register or identify a client that allows open or wildcard redirect URIs, or exploit a misconfigured redirect registry.
  • Step 2: Lure victim to authorize via crafted authorize URL with attacker redirect.
  • Step 3: Receive authorization code at attacker redirect endpoint.
  • Step 4: Redeem code for access/refresh tokens at token endpoint if PKCE or client authentication is weak.
  • Step 5: Use tokens to call resource APIs or persist via refresh token reuse.

Blue-team control review — what must stop the chain

  • Control 1: Exact redirect URI matching; disallow wildcards and require pre-registered full URIs.
  • Control 2: Require client authentication or PKCE for every exchange; block public clients without PKCE.
  • Control 3: Monitor authorize endpoint for redirect_uri values that are not in registry and block/alert.
  • Control 4: Require proof-of-possession or additional consent for high-scope grants to public clients.

Evidence and telemetry to validate controls

  • Authorization logs: client_id, redirect_uri, state, ip, user_agent.
  • Token endpoint logs: code_id, pkce_result, client_auth_method.
  • SIEM correlation: authorize event followed by token exchange from different IPs.

Purple-team validation commands

# Detect authorize requests with unknown redirect URIs
# Query placeholder; adapt to your logs
SELECT timestamp, client_id, redirect_uri, user_agent
FROM auth_authorize_logs
WHERE redirect_uri NOT IN (SELECT redirect_uri FROM oauth_client_registry)
ORDER BY timestamp DESC LIMIT 50;

# Simulate malicious redirect (lab-only)
# Use a test client registered with a safe test redirect
curl -v "https://auth.example.com/authorize?response_type=code&client_id=<client>&redirect_uri=https://attacker.example/cb&state=<s>"

T2 — Missing PKCE enforcement

STRIDE category: Elevation of Privilege
Risk: Critical likelihood, High impact
Attacker objective: Intercept authorization code (e.g., via open redirect) and redeem it without PKCE verifier, gaining tokens.

How the attacker would exploit it

  • Step 1: Intercept or induce redirect carrying authorization code (open redirect, network interception on captive portals, or malice on native platforms).
  • Step 2: Present code to token endpoint without PKCE verifier (or with incorrect verifier) when server does not require PKCE verification.
  • Step 3: Obtain access and refresh tokens from token endpoint.
  • Step 4: Use tokens to access APIs and escalate via refresh token reuse.

Blue-team control review — what must stop the chain

  • Control 1: Enforce PKCE for all public and native clients; reject token requests missing pkce_verifier.
  • Control 2: Reject authorization codes already redeemed (one-time use) and tie codes to PKCE challenge on issuance.
  • Control 3: Harden token endpoint with strict client authentication for confidential clients and strict CORS policies.
  • Control 4: Telemetry to detect authorization code replay patterns and raise immediate revoke actions.

Evidence and telemetry to validate controls

  • Token endpoint logs: grant_type, code_id, pkce_result, client_id, ip.
  • Authorization code issuance logs with challenge binding code_challenge and code_challenge_method.
  • SIEM rules for AuthCodeDoubleRedeem and missing pkce_result with accepted token response.

Purple-team validation commands

# Find token grants where pkce_result missing or failed but tokens issued
SELECT timestamp, code_id, client_id, pkce_result, status
FROM token_endpoint_logs
WHERE grant_type='authorization_code' AND (pkce_result IS NULL OR pkce_result!='OK') AND status='200'
ORDER BY timestamp DESC LIMIT 50;

# Lab validation: attempt redeeming code without PKCE verifier to ensure rejection
curl -X POST https://auth.example.com/token -d 'grant_type=authorization_code&code=<code>&client_id=<client>'

T3 — Weak audience/issuer validation (token confusion)

STRIDE category: Spoofing
Risk: High likelihood, High impact
Attacker objective: Use a valid token issued for a different audience/client to access protected APIs due to missing aud/iss checks.

How the attacker would exploit it

  • Step 1: Obtain a token from a social or enterprise IdP or another client with overlapping scopes.
  • Step 2: Present token to API or microservice that does not validate aud, iss, or azp correctly.
  • Step 3: Call privileged APIs or access tenant data that relies on token claims.
  • Step 4: Combine with account-linking flaws to escalate to higher-privilege internal account.

Blue-team control review — what must stop the chain

  • Control 1: API gateway validates iss, aud, azp, exp, and token signature against expected JWKS per client.
  • Control 2: Token introspection for cross-boundary operations and explicit mapping of client_id → allowed audiences.
  • Control 3: Reject tokens that contain unexpected extra scopes for the audience.
  • Control 4: Centralized JWKS rotation with strict caching policies to avoid stale keys.

Evidence and telemetry to validate controls

  • Gateway auth logs: token_iss, token_aud, validation_result, client_id.
  • IdP issuance logs mapping client_id to audience claims.
  • SIEM correlation for tokens accepted with unexpected aud values or from different issuers.

Purple-team validation commands

// Find accepted tokens whose aud is not expected for the endpoint
AuthGatewayLogs
| where httpStatus == 200 and token_aud != 'api.example.com'
| project timestamp, requestUrl, token_aud, token_iss, principal
| limit 50

# Test: present token from different IdP and ensure gateway rejects it
curl -H "Authorization: Bearer <token-from-other-iss>" https://api.example.com/private

T4 — Nonce/state mishandling (CSRF login swap)

STRIDE category: Tampering
Risk: High likelihood, High impact
Attacker objective: Swap authorization state or CSRF the login flow to bind the attacker's session to victim's account or vice versa.

How the attacker would exploit it

  • Step 1: Craft authorize URL and trick victim's browser into performing the flow (e.g., via CSRF or malicious link).
  • Step 2: If state and nonce are not validated per session, the attacker can swap tokens or cause session fixation.
  • Step 3: Victim's session becomes bound to attacker's credentials or attacker's session linked to victim identity.
  • Step 4: Attacker uses the resulting session or tokens to perform actions as the victim.

Blue-team control review — what must stop the chain

  • Control 1: Strong per-request state and nonce generation, stored server-side and validated on callback.
  • Control 2: Use SameSite cookies and CSRF protection on any endpoints that initiate auth flows.
  • Control 3: Limit exposed scopes for public clients and require user re-auth for account linking.
  • Control 4: Detect and alert on callback requests where state/nonce mismatch occurs frequently for specific client_id.

Evidence and telemetry to validate controls

  • Callback logs: state, nonce, session_id, and originating IP.
  • Metrics for state_mismatch and nonce_mismatch events with client_id and timestamp.
  • Browser logs or WAF logs showing CSRF attempts or suspicious referer headers.

Purple-team validation commands

-- Find state/nonce validation failures
SELECT timestamp, client_id, callback_url, error
FROM `project.dataset.callback_logs`
WHERE error LIKE '%state_mismatch%' OR error LIKE '%nonce_mismatch%'
ORDER BY timestamp DESC LIMIT 50;

# Lab: attempt CSRF-like flow and ensure server rejects callback due to state mismatch

T5 — Account-linking logic flaw

STRIDE category: Elevation of Privilege
Risk: Critical likelihood, High impact
Attacker objective: Link an attacker-controlled external identity (social or enterprise) to a victim's internal account and gain access.

How the attacker would exploit it

  • Step 1: Trigger account-link flow (email-confirmation, implicit link) where linkage checks are weak.
  • Step 2: Exploit race conditions or use previously granted access to claim ownership of an identifier.
  • Step 3: Link attacker identity to victim account (via weak email verification, predictable tokens, or flawed merge logic).
  • Step 4: Use linked identity to obtain tokens and impersonate the victim.

Blue-team control review — what must stop the chain

  • Control 1: Require strong proof-of-possession for linking (confirm via primary email with unique token and TTL, or require re-authentication with primary method).
  • Control 2: Disallow silent automatic linking across identity providers without explicit user action and audit trail.
  • Control 3: Validate federated subject identifiers (sub) and do not rely solely on displayName or email when linking.
  • Control 4: Rate-limit linking attempts and alert on multiple link operations for same account.

Evidence and telemetry to validate controls

  • Account-linking events: initiator, target_account, verification_sent, verification_completed.
  • Email verification logs and token issuance/consumption records.
  • SIEM rules for repeated link attempts or link completions from different IPs.

Purple-team validation commands

// Detect multiple account-link attempts for same account
AccountLinkLogs
| where action == 'link_attempt'
| summarize attempts = count(), by target_account
| where attempts > 3

T6 — Over-scoped OAuth client defaults

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: A misconfigured or default-overprivileged OAuth client obtains broad delegated access and exfiltrates tenant data.

How the attacker would exploit it

  • Step 1: Register or trick a client into requesting overly broad scopes during consent.
  • Step 2: Admin or user consents without reviewing scopes or consents via deceptive UI.
  • Step 3: Client uses token to call multiple APIs and extract data beyond intended scope.
  • Step 4: Persist access via refresh tokens or long-lived consent.

Blue-team control review — what must stop the chain

  • Control 1: Default-deny for client scopes; require explicit least-privilege configuration at registration.
  • Control 2: Consent UI that highlights high-risk scopes and requires step-up auth for admin-scoped grants.
  • Control 3: Automated vetting of newly registered clients and manual review for high-risk scopes.
  • Control 4: Monitor consent events and throttle or revoke clients that request or use broad scopes atypically.

Evidence and telemetry to validate controls

  • Consent logs: client_id, scopes, granted_by, grant_time.
  • API gateway logs showing calls with scope claim and endpoints accessed.
  • SIEM correlation for clients that access data outside expected scope patterns.

Purple-team validation commands

-- Find clients that were granted high-risk scopes
SELECT client_id, scopes, granted_by, grant_time
FROM `project.dataset.consent_logs`
WHERE scopes LIKE '%data:read_all%' OR scopes LIKE '%admin:%'
ORDER BY grant_time DESC LIMIT 50;

T7 — Long-lived API keys without TTL

STRIDE category: Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Use long-lived keys to automate exfiltration with minimal risk of rotation or detection.

How the attacker would exploit it

  • Step 1: Acquire API key through leakage, credential stuffing, or insider access.
  • Step 2: Execute automated calls across tenant resources, slowly to avoid rate limits.
  • Step 3: Persist access because keys do not expire and are not rotated automatically.
  • Step 4: Create secondary automation or backdoors leveraging the key's permissions.

Blue-team control review — what must stop the chain

  • Control 1: Default TTL on API keys with automatic expiration and forced rotation workflows.
  • Control 2: Per-key scoping and environment restriction (allowed IP ranges, ASN bind, or VPC restrictions).
  • Control 3: One-time reveal and hashed storage of key material; ability to revoke by key_id instantly.
  • Control 4: Analytics-driven anomaly detection for key usage (new ASN/country, high-volume trends) with automated revoke.

Evidence and telemetry to validate controls

  • API key gateway logs: key_id, source_ip, asn, requests_per_minute.
  • Key issuance logs and TTL metadata showing expiry and rotation events.
  • SIEM alerts for sudden data outflow tied to key_id.

Purple-team validation commands

// Detect keys used from new ASN

| where key_id == '<key_id>'

| where asns > 1

T8 — API key leakage in CI / build logs

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Extract keys from CI logs or artifacts and use them to access production APIs.

How the attacker would exploit it

  • Step 1: Scan public repos, CI artifacts, or leaked snapshots for key patterns.
  • Step 2: Extract key and test against API endpoints from diverse ASNs to validate usability.
  • Step 3: Use key for silent automation and exfiltration.
  • Step 4: Chain with other vulnerabilities to persist or escalate.

Blue-team control review — what must stop the chain

  • Control 1: Secrets scanning in CI with pre-commit and PR gates; block merge on high-confidence detection.
  • Control 2: Do not log secrets; mask outputs and use secret manager injectors for runners.

Evidence and telemetry to validate controls

  • CI runner logs with redaction or detection flags.
  • API gateway logs showing first-use patterns from unexpected IP ranges shortly after commit.

T9 — Refresh token replay from new device/IP

STRIDE category: Spoofing
Risk: High likelihood, High impact
Attacker objective: Redeem stolen refresh token from different ASN/geo and maintain long-lived access.

How the attacker would exploit it

  • Step 1: Obtain refresh token (XSS, stolen browser storage, breach, or leaked CI secret).
  • Step 2: Redeem it at IdP from new IP/ASN; if rotation or binding is not enforced, IdP issues new access tokens.
  • Step 3: Use new access tokens to call APIs, create artifacts, or persist deeper access.
  • Step 4: Evade detection by using similar user agent or throttling to mimic normal behavior.

Blue-team control review — what must stop the chain

  • Control 1: Refresh token rotation and bind refreshs to client fingerprint or DPoP header; reject out-of-band redeems without step-up.
  • Control 2: Detect refresh replay by jti across ASNs and force family revocation on suspicious reuse.
  • Control 3: Shorten refresh TTL for public clients and require interactive re-auth for sensitive scopes.
  • Control 4: SIEM rule to correlate refresh redeem events with immediate high-value API calls and trigger auto-revoke.

Evidence and telemetry to validate controls

  • IdP logs: refresh_redeem, refresh_jti, client_id, remote_ip, asn.
  • Access logs showing new tokens used to call sensitive endpoints shortly after redeem.
  • SIEM alerts for NewGeoRefreshReplay and token-family revocation events.

T10 — Revocation lag in edge caches

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Continue using a revoked token or key while caches still honor previously-issued credentials.

How the attacker would exploit it

  • Step 1: Use token or key that has recently been revoked by the control plane.
  • Step 2: Requests hit edge caches or gateways that haven’t yet refreshed revocation state.
  • Step 3: Perform actions during the revocation window before the cache invalidates.
  • Step 4: Evade detection by dispersing activity across edges to maximize window.

Blue-team control review — what must stop the chain

  • Control 1: Keep revocation propagation SLOs tight; push invalidations via pub/sub and immediate cache invalidation hooks.
  • Control 2: Edge honor short token TTLs and perform lightweight introspection for high-risk principals even if cache shows allow.
  • Control 3: Use token binding and per-request introspection for sensitive scopes rather than pure cache allow-lists.
  • Control 4: Monitor revocation propagation lag and escalate when SLO breached.

Evidence and telemetry to validate controls

  • Gateway cache hit/miss and revocation sync logs with timestamps.
  • Revocation events and subsequent request logs that still show accepted status.
  • SIEM rule RevocationDelayBreach when revoked tokens accepted after threshold.

Purple-team validation commands

-- Find accepted requests after revocation timestamp
SELECT r.timestamp, r.request_id, r.token_jti
FROM `project.dataset.gateway_logs` r
JOIN `project.dataset.revocations` v ON r.token_jti = v.jti
WHERE r.timestamp > v.revoked_at AND TIMESTAMP_DIFF(r.timestamp, v.revoked_at, SECOND) > 60
ORDER BY r.timestamp DESC LIMIT 50;

2. Highest-Risk Design Decision

The single most consequential decision is allowing heterogeneous trust sources (social IdPs, enterprise IdPs, and long-lived API keys) to map into a single internal account model without strict, provable binding at every enforcement point.
