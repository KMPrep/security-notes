SaaS Data Residency and Sovereignty — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario28-saas-data-residency.md
Generated artifact: scenario28-saas-data-residency-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Multi-region AWS SaaS serving EU and Japan customers while operating a shared global control plane in us-east-1; customers require regional data never leaves their geographic region.

1. Review Objective

This document reviews the scenario from two lenses:

  1. Attacker lens: Concrete attacker paths (prereqs, commands/tools, escalation) that would allow data to leave the region or be exposed to non-compliant jurisdictions.
  2. Blue-team lens: The specific preventive, detective and remedial controls (with product/event names and example log fields) that must stop each path, plus safe purple-team validation commands.

The goal is actionable validation: how to detect proof-of-failure, how to block the technique, and how to demonstrate evidence to customers.

2. Highest-Risk Design Decision

The single highest-risk design decision is operating a shared global control plane (us-east-1) that processes tenant identifiers or any PII. If the control plane accepts email addresses, user names, or other identifiers from EU or Japan tenants, those PII elements may be logged, aggregated, or processed in the US, creating a Chapter V / APPI cross-border processing event. The safe alternative is a minimal control plane contract: only pass opaque tenant_id and internal user_id (UUID) tokens; perform all PII processing and audit evidence generation inside the tenant's region. The dangerous alternative is allowing global services (billing, analytics, notifications) to accept or log PII from regional tenants.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Cross-region IAM role access to EU KMS

STRIDE category: Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Obtain plaintext of EU-protected data by invoking KMS decrypt or generating data keys from outside the EU region, then copy plaintext to non-EU storage.

How the attacker would exploit it

  • Prerequisite: attacker has credentials allowing sts:AssumeRole or a long-lived role with permissions that include kms:Decrypt or kms:GenerateDataKey scoped to EU CMK.
  • Initial action: attacker assumes the cross-region role from a US-based host or service and issues a KMS Decrypt request against the EU CMK (via region parameter --region eu-west-1).
  • Core exploitation: use AWS CLI or SDK to call Decrypt or GenerateDataKey and then write plaintext to an S3 bucket in us-east-1 (or stream it externally).
  • Escalation: rotate copy process into automated exfil pipeline, delete local traces, rotate credentials if possible.
  • Impact realization: full GDPR/APPI violation and unauthorized data disclosure.

Blue-team control review — what must stop the chain

  • Preventive: KMS key policy that denies kms:Decrypt or kms:GenerateDataKey unless aws:RequestedRegion = eu-west-1 and kms:CallerAccount matches tenant account.
  • Preventive: AWS Organizations SCP at EU OU: Deny kms:Decrypt/kms:GenerateDataKey when aws:RequestedRegion != eu-west-1.
  • Detective: CloudTrail alert rule — eventName=Decrypt AND awsRegion != eu-west-1 → critical alert (SIEM rule DR-001).
  • Corrective: AWS Config managed rule to validate KMS key policy and a Lambda auto-remediation that rotates or disables offending grants.

Evidence and telemetry to validate controls

  • CloudTrail: eventName = Decrypt or GenerateDataKey with awsRegion field and requestParameters.keyId.
  • AWS Config: KMS key policy missing RequestedRegion condition flagged as NON_COMPLIANT.
  • Security Hub / GuardDuty: finding for suspicious KMS activity or cross-region use.
  • SIEM alert: rule matching userIdentity.arn not in EU principals combined with eventName=Decrypt.

Purple-team validation commands

# Detect KMS Decrypts originating outside eu-west-1 via CloudTrail lookup
aws cloudtrail lookup-events --lookup-attributes AttributeKey=EventName,AttributeValue=Decrypt --start-time "2026-05-01T00:00:00Z" --end-time "2026-06-01T00:00:00Z"

# CloudWatch Logs Insights (example) — find Decrypt calls from non-EU regions
fields @timestamp, eventName, awsRegion, userIdentity.arn, requestParameters.keyId
| filter eventName = "Decrypt" and awsRegion != "eu-west-1"
| sort @timestamp desc
| limit 20

T2 — CloudTrail aggregation to us-east-1 containing EU metadata

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Cause EU CloudTrail events containing PII to be stored in us-east-1, making logs subject to US jurisdiction and exposing metadata.

How the attacker would exploit it

  • Prerequisite: account-level CloudTrail configuration points to a central S3 bucket in us-east-1 or a cross-account delivery role is misconfigured.
  • Initial action: change CloudTrail destination or create a new Trail that writes EU events to a US bucket (requires Write access to CloudTrail configuration or an admin role).
  • Core exploitation: once trails deliver to us-east-1, sift logs for EU tenant PII (emails, userIDs) and copy entries externally.
  • Escalation: couple log entries with other artifacts (S3 GetObject entries) to build a full audit of data accesses.
  • Impact realization: regulatory evidence of EU PII processed in US.

Blue-team control review — what must stop the chain

  • Preventive: AWS Config rule requiring CloudTrail S3 destination region = eu-west-1 for EU accounts; deny CloudTrail CreateTrail/UpdateTrail that point to non-EU buckets via SCP.
  • Detective: Config rule DR-004 + CloudTrail management event alert for CreateTrail/UpdateTrail actions where s3BucketRegion != eu-west-1.
  • Corrective: Auto-remediation Lambda that resets the trail destination to EU bucket and quarantines non-EU logs.
  • Audit: Provide tenant-owned CloudTrail S3 buckets for critical customers (customer-owned audit evidence).

Evidence and telemetry to validate controls

  • CloudTrail management events: CreateTrail, UpdateTrail with requestParameters.s3BucketName and s3BucketRegion.
  • AWS Config compliance state for CloudTrail resources.
  • S3 bucket inventory showing newly created objects in non-EU bucket names receiving CloudTrail logs.
  • SIEM rule matching eventSource = cloudtrail.amazonaws.com and eventName in (CreateTrail, UpdateTrail) with non-EU bucket.

Purple-team validation commands

# List CloudTrail configuration and check bucket region
aws cloudtrail describe-trails --query 'trailList[*].{Name:Name,S3Bucket: S3BucketName}'
aws s3api get-bucket-location --bucket <cloudtrail-bucket-name>

# CloudWatch Logs Insights: find Trail Create/Update events that set a non-EU bucket
fields @timestamp, eventName, requestParameters.s3BucketName, userIdentity.arn
| filter eventSource = "cloudtrail.amazonaws.com" and (eventName = "CreateTrail" or eventName = "UpdateTrail")
| filter requestParameters.s3BucketName not like /eu-*/
| sort @timestamp desc

T3 — S3 cross-region replication accidentally enabled

STRIDE category: Information Disclosure
Risk: Low likelihood, Critical impact
Attacker objective: Cause EU S3 buckets to replicate to a US destination (via misconfiguration or malicious replication role), then access replicated copies.

How the attacker would exploit it

  • Prerequisite: permission to modify S3 replication configuration (s3:PutBucketReplication) or an automated pipeline that applies replication rules incorrectly.
  • Initial action: enable replication with destination arn:aws:s3:::<us-bucket> or replicate to an account/region not in EU.
  • Core exploitation: confirm replication by observing s3:ObjectCreated:Put events in destination or by CloudTrail PutBucketReplication event.
  • Escalation: remove bucket-level logging or add lifecycle rules to hide replicated objects.
  • Impact realization: mass exfiltration of EU objects to non-EU storage.

Blue-team control review — what must stop the chain

  • Preventive: SCP deny s3:PutReplicationConfiguration when aws:RequestedRegion != eu-west-1 or destination region not in approved list.
  • Preventive: Bucket policy that denies s3:ReplicateObject or s3:PutObject unless aws:SourceAccount and region check pass.
  • Detective: AWS Config rule s3-bucket-replication-check (DR-003) that flags replication enabled to non-EU destinations.
  • Corrective: Auto-remediation that deletes replication configuration and quarantines replicated objects.

Evidence and telemetry to validate controls

  • CloudTrail: PutBucketReplication management event with replication destination.
  • S3 server access logs showing replication writes in destination bucket.
  • AWS Config finding for replication enabled to non-EU region.
  • SIEM alert for s3:ObjectCreated:Put events with destination region non-EU.

Purple-team validation commands

# Check bucket replication configuration
aws s3api get-bucket-replication --bucket <eu-bucket-name>

# CloudTrail lookup for PutBucketReplication events

T4 — CloudFront PoP routing EU traffic to US edge

STRIDE category: Information Disclosure
Risk: Medium likelihood, Medium impact
Attacker objective: Cause (or discover) CDN routing misconfiguration so EU requests terminate at a US PoP and any request-level PII is exposed.

How the attacker would exploit it

  • Prerequisite: misconfigured CloudFront geo-restriction or an origin failing to respect Origin selection logic.
  • Initial action: request content from a EU-sourced IP and inspect response headers / TTLs; use distributed measurement to see which PoP served requests.
  • Core exploitation: send specially crafted requests that cause application to include PII in headers or cookies that are stored in US PoP logs.
  • Escalation: combine with other telemetry to reconstruct user activity.
  • Impact realization: limited PII exposure; depending on workload, may trigger residency breach.

Blue-team control review — what must stop the chain

  • Preventive: CloudFront geo-restriction/Origin selection configured to prefer EU origins for EU geolocation; fail closed if origin not available.
  • Detective: CloudFront logs and GeoIP mapping showing any EU client served by non-EU PoP.
  • Corrective: CDN policy to scrub PII from headers at edge; ensure cookies not logged at PoP.
  • Audit: periodic sweep via synthetic clients to validate PoP routing.

Evidence and telemetry to validate controls

  • CloudFront access logs include x-edge-location and c-ip fields to show PoP and client IP.
  • Synthetic monitoring results and CDN metrics showing route anomalies.
  • SIEM rule correlating EU client IPs with non-EU PoP IDs.

Purple-team validation commands

# Check CloudFront logs or use curl with a EU exit to validate PoP
curl -H "Host: <tenant-host>" --resolve "<tenant-host>:443:<eu-edge-ip>" https://<tenant-host>/path -I

# Sample analysis (offline): grep CloudFront logs for EU client IPs served by non-EU PoP
grep "<eu-client-ip>" /path/to/cloudfront/logs | awk '{print $1,$2,$3,$4}'

T5 — DR backup replication from eu-west-1 to us-east-1

STRIDE category: Information Disclosure
Risk: Low likelihood, High impact
Attacker objective: Cause backups or snapshots to be copied to a non-EU region where they can be accessed by unauthorized principals.

How the attacker would exploit it

  • Prerequisite: permissions to create/copy snapshots (rds:CopyDBSnapshot, ec2:CopySnapshot, s3:PutObject for backup artifacts) or access to backup automation.
  • Initial action: schedule a backup job or trigger CopyDBSnapshot targeting us-east-1.
  • Core exploitation: after copy, snapshot can be shared or mounted in non-EU account and data extracted.
  • Escalation: create AMIs from snapshots or attach volumes to EC2 instances in non-EU region.
  • Impact realization: full dataset exposure; regulatory consequences and contractual breach.

Blue-team control review — what must stop the chain

  • Preventive: SCP deny snapshot copy actions to non-approved regions; guard the IAM permissions for backup automation.
  • Detective: CloudTrail rule DR-005 watching CopyDBSnapshot with destinationRegion not in approved EU list.
  • Corrective: Auto-revoke snapshot sharing, remove copied snapshot, rotate credentials if exposure detected.
  • Hardening: Restrict snapshot permissions (no public or cross-account sharing) by default.

Evidence and telemetry to validate controls

  • CloudTrail CopyDBSnapshot and CopySnapshot events with destinationRegion field.
  • AWS Config rule flagging shared or cross-region snapshots.
  • RDS event notifications and snapshot inventory.

T6 — Global control plane processing EU PII in US

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Cause control plane to accept or log EU PII (emails, user names), which is then persisted to US logs or accessible to US-region services.

How the attacker would exploit it

  • Prerequisite: control plane API accepts PII fields or engineers push code that sends email/name fields to global services.
  • Initial action: submit payloads or trigger flows that include PII fields (e.g., billing updates including email) or enable analytics pipeline that copies profile attributes to global analytics store.
  • Core exploitation: control plane logs or business events are stored in us-east-1 logging buckets or forwarded to global SIEM.
  • Escalation: exfiltrate logs or run queries in the global analytics cluster to extract EU PII.
  • Impact realization: PII in US jurisdiction, compliance and contractual breach.

Blue-team control review — what must stop the chain

  • Preventive: API schema contracts and server-side validation that reject PII fields (only tenant_id, internal_user_id allowed). Enforce via API gateway validation rules.
  • Detective: Application-level detection rule (DR-007) scanning control_plane logs for email patterns and alerting on any matches.
  • Preventive: Masking/pseudonymization helpers in SDKs used by services; static analysis to detect fields with email/name being passed to global services.
  • Corrective: CI/CD policy-as-code gate that denies PRs which add PII fields to control-plane payloads.

Evidence and telemetry to validate controls

  • Application logs from control_plane service containing fields: service=control_plane and message or structured fields showing presence of PII.
  • SIEM detections for regex matching email patterns in control plane logs (e.g., app_log WHERE service="control_plane" AND message =~ /[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/).
  • CloudTrail for API calls that include profile updates originating from EU tenants but logged in US.
  • CI/CD policy evaluation reports showing PR gating outcomes.

Purple-team validation commands

T8 — Console/API access from US-based terminal by developer

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Compromise developer credentials or social-engineer an engineer to run console/API queries from a US-based host that access EU data.

How the attacker would exploit it

  • Prerequisite: Phishing, credential theft, or stolen session tokens; or legitimate engineer operates from US IP with sufficient privileges.
  • Initial action: sign in via Console or use API keys to query RDS or S3 in eu-west-1 from a US IP (sourceIPAddress in CloudTrail shows US address).
  • Core exploitation: export query results or copy database dumps to S3 in us-east-1.
  • Escalation: create automation role in EU account that allows export to non-EU S3.
  • Impact realization: authorized-looking access that results in data export and compliance breach.

Blue-team control review — what must stop the chain

  • Preventive: Conditional access rules limiting Console/API management actions to corporate EU IPs for EU tenant operations; IAM session policies restricting sensitive actions to in-region management hosts.
  • Detective: CloudTrail ConsoleLogin events combined with sourceIPAddress geo-check; raise alerts on console logins from unexpected countries for privileged principals.
  • Corrective: Automated session revocation and temporary credential rotation when suspicious management-origin IPs are detected.
  • Hardening: Use short-lived credentials (OIDC or AWS SSO) and require MFA for sensitive operations.

Evidence and telemetry to validate controls

  • CloudTrail ConsoleLogin with sourceIPAddress and userIdentity.sessionContext.sessionIssuer fields.
  • CloudTrail data events showing GetObject, ExecuteStatement (RDS Data API), or DescribeDBInstances from non-EU IPs.
  • SIEM rule correlating MFA absence + console login from non-EU IP for privileged principals.

T9 — ETL/analytics pipeline running in wrong region

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Provision an analytics/ETL job in us-east-1 that reads EU databases and writes transformed data outside EU.

How the attacker would exploit it

  • Prerequisite: ability to create EMR/Glue/Batch jobs or misconfigured IaC pipeline that sets region to us-east-1.

T10 — Snapshot sharing to non-compliant account or region

STRIDE category: Information Disclosure
Risk: Low likelihood, Critical impact
Attacker objective: Share or copy RDS/S3 snapshots to accounts/regions that are outside compliance boundaries and extract data from them.

How the attacker would exploit it

  • Prerequisite: rds:ModifyDBSnapshotAttribute, rds:CopyDBSnapshot, or S3 snapshot share privileges.
  • Initial action: perform a snapshot copy/modify operation to share with another account or region.
  • Core exploitation: mount snapshot in non-EU account or spin up instance using leaked snapshot to read data.
  • Escalation: automate snapshot discovery and exfiltration across multiple tenants.
  • Impact realization: irreversible distribution of tenant data outside residency region.
