Scenario 28 — SaaS Data Residency and Sovereignty

Scenario: Enterprise customers in the EU and Japan require that their data never leaves their geographic region. Your SaaS is built on AWS multi-region with shared global control plane.

Candidate questions:

  • What threat does a shared control plane create for data residency enforcement?
  • How do you model an internal AWS admin bypassing data residency controls?
  • What technical controls enforce residency — not just policy controls?
  • How do you monitor and prove to customers that data has not left the region?
  • What residual risk remains for disaster recovery replication and backup data?

3.4 Highest-risk boundary: global control plane processing regional PII

The single highest-risk boundary in a data residency architecture is the interface between the shared global control plane and the regional data stores. A global control plane handles tasks like authentication, billing, feature flags, and configuration — all legitimate global functions. The risk arises when PII from EU tenant data flows into global control plane operations: user email addresses used in billing invoices, EU user profile data pulled by a global analytics job, or CloudTrail logs centralized to a US region that contain EU user identifiers. These flows can be invisible and unintentional — a developer adds EU user email to a billing notification and accidentally creates a GDPR Chapter V violation. Technical enforcement requires that the control plane operates exclusively on internal identifiers (UUID tenant IDs, not email addresses) and that any PII processing is explicitly gated to the regional tier.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale (thousands of GCP projects serving different regulatory regions), data residency enforcement cannot be manual. Every resource creation in a EU-designated project is governed by Organization Policy constraints (Google Cloud equivalent of AWS SCPs): constraints/gcp.resourceLocations restricts where resources can be created. VPC Service Controls create a logical perimeter around EU resources that prevents data exfiltration to non-EU services. All of this is defined as Terraform policy modules, applied automatically on project creation, and continuously validated by Config Validator.

8.2 Fleet-wide impact analysis

A misconfigured S3 replication rule or an AWS Config change that disables a critical residency control can affect all EU tenant data simultaneously. If eu-west-1 S3 buckets are suddenly replicating to us-east-1, all EU customer data is in motion outside the EU. At scale (100,000 EU customers), the GDPR notification obligation alone (Article 33: notify supervisory authority within 72 hours) is a significant operational challenge. The notification must include categories of data, approximate number of data subjects, and likely consequences — all requiring fast forensic capability.

8.3 Eliminating human bottlenecks

The human bottleneck is the data protection officer (DPO) reviewing every new service integration for EU data residency compliance. At scale, this is replaced by: automated data flow mapping (Privacera, BigID) that continuously discovers where EU tenant data flows; automated compliance tagging that requires engineers to declare data classification on every new resource; and a self-service DPA management portal that handles third-party vendor compliance assessment for common SaaS integrations.

8.4 Controls that break at scale

Manual AWS Config rule review — at 1,000 AWS accounts, reviewing compliance rules manually is impossible; requires AWS Security Hub aggregation with automated remediation. Manual residency audit reports — generating per-customer residency compliance evidence manually is not scalable; requires automated compliance evidence generation (AWS Artifact, custom Compliance-as-Code reports). Developer education about data residency — training 10,000 engineers on GDPR data residency requirements doesn't scale as a primary control; requires guardrails built into the deployment pipeline that fail the build if a new cross-region data flow is detected without explicit review approval.

ANSWER SUMMARY — 90-second version

"Data residency is fundamentally a technical enforcement problem disguised as a legal requirement. Contracts and policies alone don't keep data in a region — AWS S3 replication rules, IAM permissions, and application code can all move data outside the boundary without the legal team knowing.

My architecture for EU/Japan residency uses three layers: first, AWS Organizations SCPs that deny cross-region data operations at the IAM level — no S3 copy to non-EU, no KMS key replication, no RDS snapshot to US. Second, per-region KMS CMKs with key policies that deny decryption from outside the designated region — even if data is physically copied, it can't be decrypted. Third, the global control plane operates on internal UUIDs only, never EU PII — so the shared control plane processes only pseudonymized metadata.

The threat I model most carefully is the internal engineer with global AWS IAM access who can assume a cross-region role and directly query the EU database. My detection is CloudTrail monitoring for KMS Decrypt calls and S3 GetObject operations where the calling principal is in a non-EU region. Any such event is an immediate critical alert.

The residual risk I'm honest about is the US CLOUD Act — AWS is a US company and may be compelled by US legal process to produce EU customer data. BYOK with customer-controlled KMS keys in AWS EU regions is the only technical mitigation; the customer holds the keys, and AWS cannot decrypt the data even under a legal order."

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What technical controls enforce residency — not just policy controls?

Policy controls (contracts, DPAs) are necessary but insufficient — they establish liability, not prevention. Technical controls: (1) AWS Organizations SCP denying cross-region S3 copy, KMS replication, and RDS snapshot copy — this is IAM-enforced, not bypassable by application code; (2) KMS CMK key policy with kms:CallerAccount and kms:RequestedRegion conditions — data can be physically in EU but cryptographically bound to the EU region; (3) VPC Service Controls perimeter around EU resources — prevents data exfiltration to services outside the perimeter even by authenticated IAM principals; (4) S3 bucket policy denying GetObject from outside EU VPC.

Follow-up 2: How do you prove to customers that data has not left the region?

Three evidence types: (1) CloudTrail data access logs showing all S3 and KMS operations on EU resources, with no cross-region access — exportable to customer-controlled S3 bucket for customer audit; (2) AWS Config compliance history showing residency controls have been continuously in place — available in AWS Artifact; (3) Penetration test report specifically testing residency controls, conducted by accredited EU-based assessor, shared with customer. For high-value customers, I offer a customer-controlled CloudTrail S3 bucket in their own AWS account — they have direct read access to all audit logs without relying on my reporting.

Follow-up 3: How do you model an internal AWS admin bypassing data residency controls?

This is the BYOK argument. If the customer uses a KMS CMK stored in their own AWS account (not the SaaS vendor's account), the encryption key is cryptographically outside the SaaS vendor's control. An AWS employee at the SaaS vendor's account can copy the encrypted data, but they cannot decrypt it without the customer's KMS key. The customer controls key policy, so any decryption request outside their approved use cases can be detected via KMS CloudTrail in their own account. This is the only technical control that provides defense against both the SaaS vendor insider and a theoretical US legal order — the key is not in the SaaS vendor's custody.

Follow-up 4: What residual risk remains for disaster recovery replication and backup data?

DR is a genuine tension: GDPR requires data protection, but also requires business continuity. Pure in-region DR (multiple AZs within eu-west-1) satisfies both but provides lower RPO/RTO than multi-region DR. Cross-region DR (eu-west-1 + eu-central-1) is acceptable for GDPR if both regions are in the EU — Standard Contractual Clauses are not needed for intra-EU transfers. Cross-region DR to US regions requires SCCs with the customer. My approach: default to multi-AZ intra-region DR; offer multi-region EU-only DR as enterprise tier; document cross-region failover capability in DPA with explicit customer consent required before activation.

Follow-up 5: If you were the attacker, what is the cheapest path to access EU customer data?

The cheapest path is social engineering an internal engineer with global AWS IAM access. Most SaaS companies have engineers who can assume cross-region roles for operational purposes. A phishing attack against a developer, or a compromised laptop with AWS credentials cached, provides immediate cross-region access to EU data. The technical controls (KMS key policy, SCP) mitigate the risk but require careful configuration to be effective. A BYOK customer holds the key — even a compromised SaaS engineer can only access ciphertext. The second cheapest path is finding a misconfigured S3 bucket (no AWS Config rule yet deployed) with public access or overly broad bucket policy — AWS Security Hub's S3 checks catch this but only if enabled.

COMMON MISTAKES TO AVOID

  • Treating data residency as a legal/contractual problem only. Contracts establish liability but don't prevent data movement. Technical enforcement is required: IAM policies, KMS key conditions, SCPs.
  • Assuming the cloud provider's region choice is sufficient. Choosing eu-west-1 for EU customer data doesn't prevent cross-region API calls, replication, or CloudTrail aggregation to a US region. Each data flow must be explicitly controlled.
  • Forgetting CloudTrail and logging as data in scope. CloudTrail logs may contain EU user identifiers. Centralizing CloudTrail to a US region creates a residency violation for those log entries.
  • Not addressing the CLOUD Act / legal process risk. Candidates who claim data residency fully protects against US government access reveal unfamiliarity with the legal landscape.
  • Confusing pseudonymization with anonymization under GDPR. Pseudonymized data (UUID in place of email) is still personal data under GDPR if re-identification is possible. Control plane handling of UUIDs must be assessed for re-identification risk.
  • Missing the DR backup residency risk. Backup and DR replication are common sources of inadvertent residency violations that candidates often overlook.
  • Not knowing VPC Service Controls. For GCP-based architectures, VPC Service Controls are the primary technical enforcement mechanism for data residency. Candidates unfamiliar with this reveal shallow GCP security knowledge.
