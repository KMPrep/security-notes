Scenario 30 — SaaS CI/CD Pipeline Security — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario30-saas-cicd-pipeline.md
Generated artifact: scenario30-saas-cicd-pipeline-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: High-velocity SaaS CI/CD (GitHub Actions → GCR → GKE). Developers merge after peer review; workflows, runners, OIDC, artifact trust, and SBOMs are primary control points.

1. Review Objective

This review maps realistic attacker chains against the CI/CD pipeline, then prescribes specific preventive, detective and corrective controls tied to log sources and validation commands. All purple-team commands are safe, use placeholders, and intended for authorized lab or controlled environments only.

2. Highest-Risk Design Decision

The single highest-risk decision is allowing repository/ workflow content to execute in CI with privileged cloud identity (OIDC mapped service account) and broad artifact publish rights. If an attacker can change a workflow or a widely reused action and the CI identity can mint elevated credentials or push signed artifacts, the entire deployment pipeline becomes an attack vector.

Preferred safe choice: strict claim-based OIDC bindings, digest-pinned third-party actions, per-repo scoped service accounts, and admission controls requiring signed provenance. Dangerous alternative: permissive OIDC bindings, mutable action tags, and shared long-lived runner identities.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — Malicious workflow / workflow-file tamper

STRIDE category: Tampering
Risk: High likelihood, Critical impact
Attacker objective: Modify or add a workflow that executes attacker-controlled steps, exfiltrates secrets, or expands OIDC claims to increase cloud privileges.

How the attacker would exploit it

  • Prerequisite: compromised developer account, stolen maintainer keys, or social-engineered reviewer approval.
  • Initial action: commit or merge change touching .github/workflows/* to add new steps or modify existing ones.
  • Core exploitation: add a step that reads secrets, calls external C2, or mints broader OIDC tokens.
  • Escalation: use obtained tokens to push poisoned images or change deployment policies.
  • Impact realization: pipeline used as channel for code, secrets exfiltration, and direct production compromise.

Blue-team control review — what must stop the chain

  • SCM: Enforce branch protection with CODEOWNERS, required approvers, and block direct pushes to protected branches.
  • SCM: Require signed commits and enforced commit signature verification on push/gate.
  • CI Policy: Disallow workflow changes from low-trust contributors; require an additional security gate for workflow diffs.
  • Automation: Pre-merge policy-as-code check (Open Policy Agent / custom action) that rejects unpinned actions and disallowed steps.
  • Detective: GitHub audit webhook into SIEM for workflow_file modifications with automated alerting.

Evidence and telemetry to validate controls

  • GitHub audit log: action = push/workflow_dispatch, ref, actor, files_changed containing .github/workflows.
  • GitHub Actions run metadata: workflow_run events, job steps showing unexpected commands.
  • SIEM correlation: workflow file change + new OIDC token exchange + secret fetch within same timeline.
  • Code review approvals: required_pull_request_reviews events showing bypass or single approver.

Purple-team validation commands

# Find recent workflow file changes and actors
gh api repos/<org>/<repo>/commits --jq '.[] | select(.files[].filename|contains(".github/workflows")) | {sha: .sha, author: .commit.author.name}'

# SIEM: detect workflow-file changes followed by secret reads
GitHubAudit
| where action == "push" and tostring(file_names) contains ".github/workflows"
| join (CIEvents | where action == "GetSecret") on repo

T2 — Third-party action compromise / tag drift

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Introduce malicious code through an external GitHub Action referenced by workflows (mutable tag or unpinned tag).

How the attacker would exploit it

  • Prerequisite: workflows reference third-party action by tag (e.g., uses: owner/action@v1) not pinned to digest.
  • Initial action: attacker updates action tag or compromises action maintainer to ship malicious update.
  • Core exploitation: runner executes modified action code leading to in-job exfil or artifact tamper.
  • Escalation: altered action spreads across many repos that reuse it.
  • Impact realization: widespread pipeline compromise via a single action update.

Blue-team control review — what must stop the chain

  • Preventive: Require uses: owner/action@sha256:<digest> pinning or use internal action mirrors.
  • Preventive: Block actions from unapproved namespaces via enterprise policy or GitHub allowlist.
  • Detective: CI step-level provenance collection showing action digest and source; alert on digest drift.
  • Corrective: Replace compromised action with vetted internal fork and rotate any keys read during malicious runs.

Evidence and telemetry to validate controls

  • CI run metadata: action_ref including digest vs tag; step-level runtime metadata.
  • GitHub package events or action release logs showing owner change.
  • Internal mirror audit: mismatch between mirrored digest and referenced digest.

Purple-team validation commands

# List workflows that reference unpinned actions
rg "uses: .*@v[0-9]" -n .github/workflows || true

# Verify action digest used in a sample job (manual check)
echo "Check action digest matches internal mirror for <action>"

T3 — Compromised CI runner (self-hosted)

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Control a runner instance to exfiltrate secrets, tamper build artifacts, or persist a backdoor into the pipeline.

How the attacker would exploit it

  • Prerequisite: vulnerability in runner image, leaked runner credentials, or insufficient isolation of shared runners.
  • Initial action: attacker executes commands on runner to read env vars, download secrets, or replace build outputs.
  • Core exploitation: persist a listener or scheduled job to capture future builds and tokens.
  • Escalation: use runner to call artifact registry APIs and overwrite images if credentials permit.
  • Impact realization: sustained pipeline compromise across multiple repos.

Blue-team control review — what must stop the chain

  • Preventive: Use ephemeral runners for untrusted workflows and enforce node isolation (different VM per job) with zero persistent state.
  • Preventive: Use OIDC ephemeral creds with least privilege; do not store registry or cloud creds on runner host.
  • Detective: Runner telemetry (process spawn, outbound IPs) and artifact integrity checks for builds run on each runner.
  • Corrective: Auto-decommission and rebuild compromised runner image; rotate any credentials used by that runner.

Evidence and telemetry to validate controls

  • Runner host logs and orchestration (instance creation/teardown events) and CI job metadata linking job_id->runner_id.
  • Network egress logs showing outbound connections from runner to unknown domains.
  • Artifact registry push logs showing images pushed by runner identity.

Purple-team validation commands

# Check for long-lived runners or unexpected runner activity
gh api orgs/<org>/actions/runners --jq '.runners[] | {name: .name, status: .status, busy: .busy}'

T4 — OIDC mapping misconfiguration

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Exploit permissive OIDC claim-to-service-account bindings to mint over-privileged tokens.

How the attacker would exploit it

  • Prerequisite: OIDC trust configured with broad aud or insufficient sub/repo/ref constraints.
  • Initial action: craft a workflow or push trigger such that the runner's OIDC assertion satisfies the binding conditions.
  • Core exploitation: exchange assertion for a service account token with broader privileges than intended.
  • Escalation: use service account to push signed artifacts, alter IAM policies, or access secrets.
  • Impact realization: elevated cloud privileges without possession of long-lived keys.

Blue-team control review — what must stop the chain

  • Config: Bind identities using strict claim checks (repo, workflow, branch) and require additional context (e.g., job name claim) for high-privilege roles.
  • Preventive: Separate OIDC bindings for build-only vs deploy-only service accounts.
  • Detective: Cloud audit of token mint events: workload_identity_pool / token_exchange with unexpected aud or sub values.
  • Corrective: Revoke compromised binding and rotate service account keys or disable bindings.

Evidence and telemetry to validate controls

  • Cloud Audit Logs: token exchange events showing principalEmail, aud, and requestMetadata fields.
  • CI metadata: job SHA, ref, and workflow name correlated with token mint events.
  • SIEM rule: OIDCRoleAssumeMismatch alert when claims do not align with expected repo/branch.

Purple-team validation commands

# GCP: list IAM policy bindings for the platform service account
gcloud iam service-accounts get-iam-policy <sa>@<project>.iam.gserviceaccount.com --format=json

# SIEM: detect OIDC mappings used from unexpected repo or branch
CloudAudit
| where protoPayload.methodName contains "GenerateAccessToken"
| where protoPayload.request.metadata.claims.aud != "expected-audience"

T5 — Secrets leaked in CI logs/artifacts

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Harvest secrets exposed in job logs, build artifacts, or stack traces and reuse them against partners or cloud APIs.

How the attacker would exploit it

  • Prerequisite: build step prints secrets (debug echo), or artifact contains config files with tokens.
  • Initial action: attacker searches logs/artifacts storage for token patterns or known prefixes.
  • Core exploitation: validate tokens against partner APIs and use them to perform fraudulent actions or to access internal APIs.
  • Escalation: use tokens to access additional secrets or to impersonate services.
  • Impact realization: widespread credential misuse and data compromise.

Blue-team control review — what must stop the chain

  • Preventive: Mask secrets using CI provider secrets masking features and disallow printing of known secret patterns.
  • Preventive: Enforce build-time scrubbing and artifact hygiene (do not include config with secrets in artifacts).
  • Detective: Run DLP scanning on logs and artifacts (e.g., regex for token prefixes) prior to retention.
  • Corrective: Automated rotation playbook that revokes any leaked token and issues replacements.

Evidence and telemetry to validate controls

  • CI job logs and artifacts metadata showing job_id, actor, and suspected token strings.
  • Artifact storage access logs showing downloads of artifacts containing secrets.
  • SIEM DLP alerts and SOAR playbook logs for rotation actions.

Purple-team validation commands

# Artifact scan: list recent artifacts and grep for token-like strings (lab only)

T6 — Unsigned or unverified image deploy

STRIDE category: Tampering
Risk: Low likelihood, Critical impact
Attacker objective: Bypass admission gates to deploy an unsigned or tampered image into production.

How the attacker would exploit it

  • Prerequisite: admission controls not enforced or emergency bypass exists; registry policies allow tag mutation.
  • Initial action: push malicious image signed with attacker key or overwrite tag digest after signing window.
  • Core exploitation: CD controller pulls tampered image and deploys to GKE.
  • Escalation: run privileged workload or sidecar to exfiltrate tenant data.
  • Impact realization: production compromise at scale.

Blue-team control review — what must stop the chain

  • Deployment: Enforce Binary Authorization / admission controller that requires image signatures and provenance attestations.
  • Registry: Make tags immutable and restrict push permissions to a small set of CI service accounts.
  • Detective: Admission logs with policy_result and image digest mismatches trigger high-severity alerts.
  • Corrective: Global digest quarantine and automated rollback to last signed release in case of anomaly.

Evidence and telemetry to validate controls

  • GKE admission controller logs: admissionReview decision, image_digest, policy_result.
  • Artifact registry push events showing actor and digest mismatch or overwrite attempts.
  • SOAR runbook logs showing quarantine and rollback actions.

Purple-team validation commands

kubectl apply -f unsigned-deploy.yaml --dry-run=server

T7 — Dependency hijack / lockfile tamper

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Insert malicious dependency via lockfile tamper, typosquatting, or maintainer compromise.

How the attacker would exploit it

  • Prerequisite: weak lockfile verification, reliance on public registries without provenance.
  • Initial action: publish malicious package or alter lockfile to reference poisoned version.
  • Core exploitation: build pulls malicious package into image and executes code at runtime.
  • Escalation: use package to exfiltrate secrets or open backdoors in application code.
  • Impact realization: supply-chain compromise across releases.

Blue-team control review — what must stop the chain

  • Build: Enforce lockfile verification and signed lockfiles; use internal proxy caches with allowlist.
  • CI: Fail builds when lockfile integrity or provenance is absent (SBOM policy enforcement).
  • Detective: Dependency monitor alerts on publisher changes and unusual package updates.
  • Corrective: Rebuild images from verified base and rotate any secrets affected by malicious packages.

Evidence and telemetry to validate controls

  • SBOM and lockfile integrity scan results; CI failure logs when lockfile mismatch detected.
  • Dependency monitor events (e.g., new maintainer push, unexpected package owner change).
  • Build artifact differences and CVE scanner outputs.

Purple-team validation commands

# Verify lockfile integrity and SBOM presence in build artifacts
sbom-tool verify --file build/sbom.json || echo "SBOM missing or invalid"

T8 — Artifact registry overwrite / digest swap

STRIDE category: Spoofing / Tampering
Risk: Low likelihood, Critical impact
Attacker objective: Replace or swap digests for a given tag so that a previously trusted tag resolves to attacker-controlled image.

How the attacker would exploit it

  • Prerequisite: registry permissions overly permissive or tags mutable; attacker has push rights.
  • Initial action: push a new image with the same tag but different digest, or manipulate registry metadata.
  • Core exploitation: CD pulls the tag and deploys the new (malicious) digest.
  • Escalation: attacker executes arbitrary code in production workloads.
  • Impact realization: large-scale compromise via trusted release channels.

Blue-team control review — what must stop the chain

  • Registry: Make tags immutable by policy; require signed pushes and restrict push IAM to CI service accounts.
  • Admission: Verify pulled image digest against recorded signed digest; reject if mismatch.
  • Detective: Registry audit logs for tag mutations and unexpected pushes.
  • Corrective: Global blocklist of suspect digest and immediate rollback.

Evidence and telemetry to validate controls

  • Artifact registry events: push, overwrite, actor, and digest values.
  • CD logs showing resolved image:tag -> digest mapping and admission verification timestamps.
  • SIEM correlation for digest mismatch alerts.

Purple-team validation commands

# Check current digest for tag and compare to signed value (example)
artifact-cli describe --image gcr.io/<proj>/<image>:<tag> --format json | jq '.digest'

T9 — Canary bypass / emergency release abuse

STRIDE category: Denial of Service / Elevation
Risk: Medium likelihood, High impact
Attacker objective: Use emergency bypass or misconfigured canary gating to push a bad release to all users quickly.

How the attacker would exploit it

  • Prerequisite: policy allows emergency override or developer with release privileges abuses approval flow.
  • Initial action: mark release as emergency and skip or shorten canary checks.
  • Core exploitation: global rollout of malicious image before detection.
  • Escalation: remove rollback triggers or delay monitoring, maximizing damage window.
  • Impact realization: large-scale outage or data theft.

Blue-team control review — what must stop the chain

  • Process: Require multi-person approval and ticket linkage for any canary bypass; log and alert all bypass events.
  • Preventive: Enforce time-limited bypass tokens that expire quickly and require SOC sign-off.
  • Detective: CD audit logs with override flags and correlating automated canary metrics.
  • Corrective: Immediate global digest quarantine and automated rollback orchestration.

Evidence and telemetry to validate controls

  • CD controller approval logs and audit trails showing who initiated bypass and justification.
  • Canary monitoring metrics and alerts (error rates, latencies) with automated rollback triggers.
  • SIEM timeline showing bypass event correlated with deployment and abnormal metrics.

T10 — Build cache poisoning

STRIDE category: Tampering
Risk: Medium likelihood, Medium impact
Attacker objective: Insert attacker-controlled binaries into build cache so subsequent builds reuse poisoned artifacts.

How the attacker would exploit it

  • Prerequisite: shared cache with weak authentication or cache key collisions.
  • Initial action: insert malicious binary into cache with common cache key.
  • Core exploitation: subsequent builds retrieve poisoned binary, producing malicious image.
  • Escalation: persistence across rebuilds until cache is invalidated.
  • Impact realization: multiple releases include malicious binary.

Blue-team control review — what must stop the chain

  • Cache: Use signed cache entries or authenticated cache backends; segregate caches per repo or pipeline.
  • Build: Reproducible builds and artifact verification against known good hashes.
  • Detective: Cache integrity monitoring and alerts on unexpected cache writes.
  • Corrective: Cache invalidation playbooks and rebuilds from clean sources.

Evidence and telemetry to validate controls

  • CI cache logs: write events, actor, and key used; artifact hash comparisons during build.
  • SBOM comparisons between expected and produced artifacts.

T11 — SBOM / provenance absent or ignored

STRIDE category: Repudiation
Risk: Medium likelihood, High impact
Attacker objective: Reduce visibility so malicious dependencies or builds go undetected at admission.

How the attacker would exploit it

  • Prerequisite: SBOM generation optional or admission ignores provenance.
  • Initial action: modify workflow to skip SBOM generation or strip attestation during publish.
  • Core exploitation: deploy artifact lacking provenance to production; detection depends on audits.
  • Escalation: repeat across releases to hide changes.
  • Impact realization: harder forensic investigations and longer dwell time.

Blue-team control review — what must stop the chain

  • Build: Enforce SBOM generation as a hard failure; require signed attestation attached to image.
  • Admission: Reject any deploy missing attestation or with mismatching SBOM hash.
  • Detective: SBOM diff monitoring that alerts on new high-risk components.
  • Corrective: Quarantine any un-attested images and require manual review.

Evidence and telemetry to validate controls

  • Build logs: SBOM generation step output and attestation upload events.
  • Admission logs: policy checks showing missing attestation or SBOM mismatch.
  • Vulnerability scanner results compared against SBOM.

Purple-team validation commands

# Ensure SBOM present in an image's attestation (example)
cosign verify-attestation --type sbom gcr.io/<proj>/<image>@<digest>

T12 — Malicious insider commit

STRIDE category: Tampering / Repudiation
Risk: Low likelihood, Critical impact
Attacker objective: A developer with commit rights intentionally introduces backdoor or bypasses controls via collusion or deception.

How the attacker would exploit it

  • Prerequisite: sufficient privileges to push/merge or collude with approver.
  • Initial action: craft a subtle code change that exfiltrates secrets or enables later exploitation.

2. Highest-Risk Design Decision

The single highest-risk decision is allowing repository/ workflow content to execute in CI with privileged cloud identity (OIDC mapped service account) and broad artifact publish rights. If an attacker can change a workflow or a widely reused action and the CI identity can mint elevated credentials or push signed artifacts, the entire deployment pipeline becomes an attack vector.

T12 — Malicious insider commit

STRIDE category: Tampering / Repudiation
Risk: Low likelihood, Critical impact
Attacker objective: A developer with commit rights intentionally introduces backdoor or bypasses controls via collusion or deception.

How the attacker would exploit it

  • Prerequisite: sufficient privileges to push/merge or collude with approver.
  • Initial action: craft a subtle code change that exfiltrates secrets or enables later exploitation.
  • Core exploitation: hide payload in tests or rarely-executed code paths; merge under guise of maintenance.
  • Escalation: create stealthy persistence and delay detection by blending with legitimate commits.
  • Impact realization: high-confidence, long-lived compromise that is difficult to attribute.

Blue-team control review — what must stop the chain

  • Process: Enforce separation of duties; require security review for any code touching CI workflows, OIDC mappings, or deploy logic.

  • Detective: Heuristic and code-scanning for data exfil patterns during PR review (e.g., network calls to unusual domains, cryptic env usage).
  • Corrective: Immutable release provenance that allows rapid rollback and forensic capture of offending commits.
