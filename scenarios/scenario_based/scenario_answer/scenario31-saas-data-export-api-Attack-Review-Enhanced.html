Scenario 31 — SaaS Customer Data Export and API Access — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario31-saas-data-export-api.md
Generated artifact: scenario31-saas-data-export-api-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Enterprise bulk export API that stages large PII exports to S3, issues short-lived signed URLs, and supports tenant-scoped queries and downloads.

1. Review Objective

This review examines the export API from two lenses:

  1. Attacker lens — realistic exploitation paths for cross-tenant data exposure, signed-URL leakage, and export abuse.
  2. Blue-team lens — prescriptive controls (preventive, detective, corrective) with specific log sources, queries, and purple-team validation steps.

2. Highest-Risk Design Decision

The single most consequential decision is how tenant context is propagated and enforced across layers: API gateway authorizers, export job orchestrator, database queries, and object storage prefixes/ACLs. If any layer fails to enforce the tenant_id binding deterministically, legitimate-auth flows can be abused to access another tenant's data. Preferred safe choice: canonical tenant binding injected at the API gateway and carried as an immutable job claim used by RLS and storage key generation. Dangerous alternative: relying on developer-supplied filters or ad-hoc prefixing logic.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or materially contained.
  • Yellow nodes (:::impact): business or security impact if the chain succeeds.

6. Threat-by-Threat Technical Review

T1 — IDOR on export job id lookup

STRIDE category: Elevation of Privilege
Risk: Critical likelihood, High impact
Attacker objective: Retrieve metadata or download links for another tenant's export by guessing or enumerating job_id or export_id.

How the attacker would exploit it

  • Prerequisite: predictable or enumerable job IDs, no tenant binding on lookup endpoint.
  • Initial action: iterative requests to /exports/{job_id} probing for valid IDs and response differences.
  • Core exploitation: a successful lookup returns signed URL or storage key for victim tenant.
  • Escalation: use URL to download PII or chain to request additional exports.
  • Impact realization: bulk disclosure of another tenant's export metadata or files.

Blue-team control review — what must stop the chain

  • API gateway: validate authorizer.claims.tenant_id and compare to job ownership (requestContext.authorizer.claims.tenant_id == job.owner_tenant) at edge (e.g., AWS API Gateway + Lambda authorizer).
  • Job service: store immutable owner_tenant claim inside job record at creation and enforce lookup checks in service code and DB queries.
  • Rate limiting: per-IP and per-actor throttles for export status API (WAF + API Gateway throttling rules).
  • Detective: alert on high-volume 404->200 patterns and enumerate-like access (SIEM rule ExportIdEnumeration).

Evidence and telemetry to validate controls

  • API Gateway access logs: requestContext.authorizer.claims.tenant_id, path, statusCode.
  • Application logs: job record reads including job_id and owner_tenant fields.
  • SIEM rule: ExportIdEnumeration detecting many sequential job_id lookups from one actor.
  • DB audit: SELECT statements on exports table with job_id and returned owner_tenant.

Purple-team validation commands

# Find job lookups returning 200 for actors not matching owner_tenant
aws logs start-query --log-group-name /aws/apigateway/prod --start-time <start> --end-time <end> --query-string "fields requestContext.authorizer.claims.tenant_id, path, statusCode | filter path like '/exports/' and statusCode==200 | stats count() by requestContext.authorizer.claims.tenant_id"

// SIEM: detect job id enumeration pattern
ApiGatewayLogs
| where request_uri startswith "/exports/"
| summarize cnt=count() by src_ip, bin(timestamp, 5m)
| where cnt > 50

T2 — Missing tenant predicate in export query

STRIDE category: Information Disclosure
Risk: Critical likelihood, Critical impact
Attacker objective: Cause the export worker to run a query without a tenant filter and write other tenants' rows into the export file.

How the attacker would exploit it

  • Prerequisite: code path that accepts query filters from request and builds SQL without mandatory tenant predicate.
  • Initial action: craft export request with empty or manipulated filters that bypass filter builders.
  • Core exploitation: export job issues a query such as SELECT * FROM table WHERE <user_filters> missing tenant_id = X.
  • Escalation: request multiple exports covering different tables or time ranges.
  • Impact realization: export contains cross-tenant data including PII.

Blue-team control review — what must stop the chain

  • Data: Enforce database Row-Level Security (RLS) with policies requiring tenant_id = current_setting('app.current_tenant') (PostgreSQL RLS) or column-level filters in other DBs.
  • Job framework: canonicalize tenant_id in orchestration layer and inject as immutable binding in SQL parameterization — never concatenate filters.
  • CI/build: Static analysis unit tests that include negative tests validating missing predicate is rejected (policy-as-code tests run at PR stage).
  • Detective: Query-audit rule ExportQueryWithoutTenantFilter that alerts on queries against export workers without tenant_id predicate (DB audit or eBPF/pgAudit events).

Evidence and telemetry to validate controls

  • DB audit logs: statement text showing absence of tenant_id in WHERE clause (pgAudit, Oracle Fine-Grained Auditing) and query hashes.
  • Export job logs: recorded effective_query or parameter list showing tenant binding.
  • SIEM rule match ExportQueryWithoutTenantFilter with job_id and actor.
  • Unit test CI results showing policy-as-code failures for missing tenant predicates.

Purple-team validation commands

# DB audit: search for export queries missing tenant predicate (example for Postgres audit table)
psql -c "SELECT query, userid, logged_at FROM pgaudit.log WHERE query LIKE '%FROM exports%' AND query NOT LIKE '%tenant_id%';"

// SIEM: detect export jobs with missing tenant clause
DBAudit
| where statement contains "FROM" and statement contains "exports"
| where statement !contains "tenant_id"

T3 — Shared bucket prefix collision

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Read a neighboring tenant's staged export due to poor prefixing or ACL misconfiguration.

How the attacker would exploit it

  • Prerequisite: bucket layout uses shallow prefixes or shared buckets with weak object key policies.
  • Initial action: enumerate objects in bucket or observe predictable key naming and request access.
  • Core exploitation: read object via GetObject due to permissive ACL or incorrect prefix filtering.
  • Escalation: combine with leaked signed URLs to find accessible objects.
  • Impact realization: unauthorized download of other tenants' exports.

Blue-team control review — what must stop the chain

  • Storage: Use per-tenant bucket or enforce strict key prefix conditions in bucket policy (S3 condition StringLike on s3:prefix).
  • Access: Require signed URLs bound to object digest and one-time use options; do not rely solely on prefix isolation.
  • Detective: CloudTrail GetObject events with requester != owner or prefix mismatch rule.
  • Corrective: Automated object quarantine when ACL drift detected and immediate rotation of signed URLs.

Evidence and telemetry to validate controls

  • CloudTrail: eventName = GetObject and requestParameters.bucketName / requestParameters.key fields.
  • S3 server access logs: cs-uri-stem and x-amz-request-id with requester identity.
  • Storage config drift alerts from config management (e.g., AWS Config rule for S3 bucket public access or ACL drift).
  • SIEM correlation for cross-prefix access events.

Purple-team validation commands

# List objects in staging bucket and verify prefixes belong to tenant
aws s3api list-objects-v2 --bucket prod-exports --prefix "<tenant_prefix>/" --query 'Contents[].Key'

// CloudTrail-ish query: detect getobject from unexpected principal
CloudTrail
| where eventName=="GetObject"
| where requestParameters.bucketName=="prod-exports" and not startswith(requestParameters.key, "<tenant_prefix>/")

T4 — Leaked signed URL in logs or telemetry

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Obtain a time-limited signed URL from logs, error messages, or support channels and use it to download a staged export.

How the attacker would exploit it

  • Prerequisite: signed URLs written to application logs, stored in analytics snapshots, or delivered over insecure channels.
  • Initial action: search logs, SLO dashboards, chat transcripts, or email notifications for URL patterns (e.g., https://...signed...).
  • Core exploitation: reuse the URL before expiry or chain with DNS leaks or referer leaks.
  • Escalation: replay URL multiple times or share with third parties.
  • Impact realization: unauthorized download of full export file.

Blue-team control review — what must stop the chain

  • Delivery: Do not write signed URLs to logs; redact or token-hash them in telemetry. Use event IDs instead of URLs in logs.
  • URL policies: Enforce one-time-use signed URLs or require TLS client cert or referer-binding for downloads.
  • Detective: DLP on logs to detect URL patterns and trigger immediate revocation (SOAR playbook to invalidate URL and reissue).
  • Corrective: Invalidate leaked URL by rotating object key or toggling access policy immediately when leak detected.

Evidence and telemetry to validate controls

  • Application logs: ensure no signed_url or full url field present; check for url_redacted or url_id instead.
  • SIEM DLP alerts on log archives containing https://.*signed patterns.
  • Signed URL generation logs: creation timestamp, creator_id, TTL, and redemption counts.
  • Object storage access logs showing multiple redemptions of same URL-derived token.

Purple-team validation commands

# Search archived logs for signed URL patterns (lab only — search on backups)
rg "https?://[^"]+\?(?:X-Amz-|signature=)" /var/log/app/* || true

# Validate that signed URL creation logs contain only url_id and not the full URL
aws logs filter-log-events --log-group-name /app/export --filter-pattern "CreateSignedUrl" --query 'events[].message' | jq '. | select(.message|contains("url_id"))'

T5 — Excessive export request rate (DoS)

STRIDE category: Denial of Service
Risk: High likelihood, High impact
Attacker objective: Flood the export API with large requests, exhausting CPU, DB, and staging storage capacity, impacting other tenants.

How the attacker would exploit it

  • Prerequisite: public export endpoint or weak per-tenant quotas.
  • Initial action: submit many concurrent export requests or very-large-export filters to trigger heavy queries.
  • Core exploitation: overload database, saturate export job queue, exhaust storage IO or egress.
  • Escalation: orchestrate from multiple IPs to defeat simple rate limits.
  • Impact realization: platform degradation or denial for legitimate customers.

Blue-team control review — what must stop the chain

  • Throttling: enforce layered quotas — per-tenant concurrent jobs, per-tenant daily export rows, and global request rate via API Gateway throttling and WAF rules.
  • Queuing: admission queue that enqueues heavy jobs and schedules based on capacity, with priority bands per tenant tier.
  • Circuit breakers: automatic job back-pressure and operator-visible backoff with alerts to SRE.
  • Detective: rate-limiter metrics and anomaly alerts (CloudWatch metric ThrottledRequests or APIGW Count alarms).

Evidence and telemetry to validate controls

  • API gateway metrics: 4XX/5XX rates, ThrottledRequests and Latency percentiles.
  • Export job queue depth and worker CPU/memory metrics.
  • Storage ingress/ejection metrics for staging buckets and object count deltas.
  • SIEM rule: sustained exceedance of per-tenant request thresholds.

Purple-team validation commands

# Simulate high-rate export requests from test tenant to validate throttling (careful, lab only)
for i in {1..100}; do curl -s -X POST https://api.example.com/exports -H "Authorization: Bearer <test_admin_token>" -d '{"filters":{}}' & done

// Detection: find tenants with export rate > baseline
ApiGatewayMetrics | where name=="ThrottledRequests" | summarize sum(value) by tenant_id, bin(timestamp, 5m) | where sum_value > 100

T6 — Compromised customer admin token

STRIDE category: Spoofing
Risk: Medium likelihood, Critical impact
Attacker objective: Use stolen admin credentials or tokens to request and download large exports legitimately.

How the attacker would exploit it

  • Prerequisite: credential theft (phishing, credential stuffing) or token leakage.
  • Initial action: authenticate to API and request exports using victim's identity.
  • Core exploitation: because the request is authorized, it bypasses many guards and returns the export file.
  • Escalation: use exported secrets or keys to expand access or chain into other services.
  • Impact realization: high-fidelity, legitimate-appearing exfiltration of data.

Blue-team control review — what must stop the chain

  • Identity: monitor for abnormal session signals — token reuse from different ASN/geolocation and require step-up MFA for high-volume exports.
  • Rotations: implement short-lived tokens (OIDC) and automatic revocation upon suspicious activity.
  • Detective: Auth telemetry rule AdminTokenReplayAnomaly using auth logs fields sourceIp, userAgent, tokenId.
  • Corrective: Automated session invalidation and forced password/MFA reset for the compromised account.

Evidence and telemetry to validate controls

  • Auth provider logs: token issue and introspection events with token_id, client_ip, and user_agent fields.
  • SIEM correlation: login anomaly + export job creation within short time window.
  • Audit trail: job creation actor_id vs usual actor baseline.
  • SOAR runbook triggers for forced revocation and notification.

Purple-team validation commands

T7 — Weak role model for export scope

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Abuse coarse-grained roles that allow over-broad exports.

How the attacker would exploit it

  • Prerequisite: role export_admin that grants access to export all tenants or many datasets.
  • Initial action: use role privileges to run exports beyond intended scope or to create service-level keys.
  • Core exploitation: request exports that include sensitive tables or join across tenants.
  • Escalation: escalate privileges by requesting scoped service account tokens.
  • Impact realization: privileges allow broad data access across tenants.

Blue-team control review — what must stop the chain

  • Governance: implement least-privilege roles with field-level scoping and enforce export_scope constraints in RBAC (e.g., IAM policy specifying resource:arn:exports:<tenant>).
  • Preventive: Require justification and time-bound elevation for sensitive export roles (access request workflows).
  • Detective: Audit role usage with role_assume events and alert when export_admin runs against unexpected tenants.
  • Corrective: Revoke or reduce role scope and require reissuance through privileged access management (PAM).

Evidence and telemetry to validate controls

  • IAM audit logs: AssumeRole/GrantRole events with principal and resource details.
  • Job logs: actor_role recorded for each export job.
  • SIEM: rule for export_admin performing exports across >1 tenant in 24h.
  • PAM logs showing time-bound elevation events.

T8 — Support impersonation misuse

STRIDE category: Repudiation
Risk: Medium likelihood, High impact
Attacker objective: Insider or compromised support tooling used to initiate exports under false pretenses to exfiltrate data stealthily.

How the attacker would exploit it

  • Prerequisite: support tools that can operate on behalf of customers and lack full audit linking to tickets.
  • Initial action: invoke support impersonation flow to request or download an export without robust approval.
  • Core exploitation: use impersonation to hide actor identity and bypass tenant-level checks.
  • Escalation: chain to privileged support accounts to access multiple tenants.
  • Impact realization: stealthy exfiltration with plausible legitimate context.

Blue-team control review — what must stop the chain

  • Process: require ticket linkage and dual-approval for any support-initiated export and log ticket_id in job metadata.
  • Preventive: restrict support tooling to read-only by default; require explicit temporary escalate-to-export actions with justification.
  • Detective: alert on support exports lacking ticket_id or with ticket_id flagged as automated/empty.
  • Corrective: suspend support account and capture forensic snapshot when misuse detected.

Evidence and telemetry to validate controls

  • Support tool logs: actor, ticket_id, action, and target_tenant fields.
  • Job metadata: created_by should contain support:ticket:<id>.
  • SIEM: rule SupportExportWithoutTicket for support exports missing correlated ticket.
  • HR / access logs for post-incident investigation.

T9 — Delayed cleanup after TTL

STRIDE category: Information Disclosure
Risk: Medium likelihood, Medium impact
Attacker objective: Discover and download export objects that were not deleted due to lifecycle job failures.

How the attacker would exploit it

  • Prerequisite: failure of cleanup job or misconfigured TTL causing objects to persist longer than expected.
  • Initial action: enumerate staging bucket for objects older than expected or query metadata service for stale objects.
  • Core exploitation: retrieve objects via valid credentials or leaked URLs found elsewhere.
  • Escalation: build index of stale exports for later targeted downloads.
  • Impact realization: temporary window of extended exposure of PII.

Blue-team control review — what must stop the chain

  • Ops: implement robust lifecycle jobs with retries, monitoring, and alerting; require idempotent deletes and tombstones.
  • Detective: rule CleanupSLABreach alerting when objects older than TTL exist in staging.
  • Storage: encrypt artifacts with per-tenant keys and rotate when TTL breach detected to make retained objects unreadable.
  • Corrective: automated forced purge and notification to affected tenants.

Evidence and telemetry to validate controls

  • Lifecycle job metrics: success rate, error counts, and last run timestamp.
  • Object metadata: LastModified and x-amz-meta-tenant tags for each object.
  • SIEM: CleanupSLABreach rule matches objects older than TTL.
  • Audit logs for delete operations and returned errors.

T10 — Insecure callback / webhook interception

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Intercept or spoof export completion callbacks that contain signed URLs or object identifiers.

How the attacker would exploit it

  • Prerequisite: webhook endpoints delivered without signature verification or over non-TLS channels.
  • Initial action: observe network, use man-in-the-middle on poorly secured callback channel, or replay captured delivery.
  • Core exploitation: obtain the signed URL from callback payload and download export.
  • Escalation: substitute callback URL to redirect notifications to attacker-controlled endpoint and harvest multiple URLs.
  • Impact realization: automated exfiltration via intercepted callbacks.

Blue-team control review — what must stop the chain

  • Delivery: Use HMAC-signed webhook payloads (e.g., include X-Signature header with shared secret) and TLS for all deliveries.
  • Verification: Receiver must validate signature before acting on payload.
  • Detective: Monitor for callback destinations that are not on allowlist and alert on rapid retries or unusual recipient IPs.
  • Corrective: Rotate webhook secrets and revoke tokens associated with compromised endpoints.

Evidence and telemetry to validate controls

  • Webhook delivery logs: destination URL, X-Signature header presence, delivery status codes.
  • Network logs for callback traffic showing unexpected destinations or geographies.
  • SIEM rule for WebhookDestinationAnomaly mapping delivered to unknown endpoints.
  • Webhook replay detection metrics.

Purple-team validation commands

// Detect webhook deliveries to endpoints outside of allowlist

2. Highest-Risk Design Decision

The single most consequential decision is allowing an export request to transform unaudited tenant context into backend query predicates and storage prefixes (i.e., trusting incoming tenant claims end-to-end). If tenant scoping is enforced only at the API layer and not guaranteed in the job worker, database, and storage access controls, a valid authenticated request can cause cross-tenant data leakage and large-scale exfiltration.

Preferred safe choice: enforce tenant binding at every layer (API, job orchestrator, DB predicates, object keys, and KMS encryption context) with policy-as-code tests. Dangerous alternative: verification at API only, with workers or storage relying on ad-hoc prefixes.
