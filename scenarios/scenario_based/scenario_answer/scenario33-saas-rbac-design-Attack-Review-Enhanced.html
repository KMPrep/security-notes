Scenario 33 — SaaS Role-Based Access Control (RBAC) Design — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario33-saas-rbac-design.md
Generated artifact: scenario33-saas-rbac-design-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years purple-team experience
Scenario: Multi-tenant SaaS introducing custom roles and attribute-based policies, with risks from role creep, confused deputy service accounts, IdP sync blasts, and admin console tampering.

1. Review Objective

This review converts the scenario into attacker-focused exploit chains and actionable blue-team controls. For each threat we list STRIDE, attacker objectives, stepwise exploitation paths, precise controls (with log fields and rule names), safe purple-team validation commands, and mermaid attack trees with control breakpoints.

2. Highest-Risk Design Decision

The single most consequential decision is permitting flexible, tenant-defined roles and service-account scopes without centralized guardrails and automatic simulation. Allowing arbitrary permission expressions and broad service account Admin scopes enables privilege creep and confused-deputy attacks that can affect multiple tenants. Preferred safe choice: policy-as-code with hardened built-in templates, automated simulation and lint gating in CI, and strict defaults that remove Admin from automation identities.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or contained.
  • Yellow nodes (:::impact): business or security impact.

6. Threat-by-Threat Technical Review

T1 — Missing authz check on manager-only endpoint

STRIDE category: Elevation of Privilege
Risk: Critical likelihood, High impact
Attacker objective: Use a Manager credential or compromised token to call an endpoint that lacks centralized authz, performing Admin-level actions.

How the attacker would exploit it

  • Prerequisite: a Manager account or compromised manager token.
  • Initial action: enumerate role-management endpoints and test for permissive responses.
  • Core exploitation: call the missing-check endpoint to perform an Admin operation (create shadow admin account or change role bindings).
  • Escalation: use created Admin account to alter policies and access tenant data.
  • Impact realization: full tenant takeover and potential cross-tenant impact.

Blue-team control review — what must stop the chain

  • Preventive: Centralize policy checks in the PolicyEngine (no endpoint-specific bypass); enforce via Envoy RBAC / sidecar policy.
  • Preventive: Deny-by-default in the gateway and require policy_id on every sensitive API (reject on missing policy_id).
  • Detective: SIEM rule ManagerToAdminGrant triggered by RoleChangeLog where actor_role=Manager and new_role=Admin (Splunk/Sentinel/Chronicle alert).
  • Preventive: API integration tests + policy simulation in CI (policy-as-code gates using Open Policy Agent (OPA) and policy unit tests).
  • Corrective: Automated revert playbook (SOAR) to revoke unauthorized grants and invalidate sessions (SOAR playbook: Revert-Unauthorized-RoleChange).

Evidence and telemetry to validate controls

  • RoleChangeLog entries with fields actor_id, actor_role, target_id, old_role, new_role, tenant_id.
  • AuthZDecisionLog showing missing policy_id or policy_evaluation_result.
  • SIEM alert ManagerToAdminGrant correlating manager role + grant event.
  • API Gateway logs showing endpoint, authn_context, response_code for the request.

Purple-team validation commands

# Test: call role-change endpoint using Manager token and expect deny
curl -X POST -H "Authorization: Bearer <manager-token>" \
 https://api.example.com/v1/roles/bind -d '{"user":"victim","role":"Admin"}'

# Detect Manager->Admin grants
RoleChangeLog
| where actor_role == "Manager" and new_role == "Admin"
| where tenant_id == "<tenant>"

T2 — IDOR in role binding API

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Exploit insecure object references to modify another user's role bindings.

How the attacker would exploit it

  • Prerequisite: predictable role-binding IDs or weak access checks.
  • Initial action: enumerate role binding identifiers and attempt modifications across tenant objects.
  • Core exploitation: issue PATCH/PUT to binding resource of another user and set Admin.
  • Escalation: create persistent shadow accounts or escalate multiple users.
  • Impact realization: unauthorized privilege grants and data access.

Blue-team control review — what must stop the chain

  • Preventive: Authorize every role-binding action against tenant-scoped ownership and validate subject.tenant_id server-side.
  • Preventive: Use opaque UUIDs for binding resources and avoid sequential IDs.
  • Detective: API gateway WAF rules to detect anomalies and SIEM rule IDORRoleBindingAttempt logging source_ip and attempted binding_id pattern.
  • Preventive: Add unit tests in API PRs to check cross-tenant access attempts.
  • Corrective: Forensic snapshot of binding changes with rollback via RevertRoleBinding SOAR action.

Evidence and telemetry to validate controls

  • API_Gateway_Logs showing method, path, subject_claims, tenant_id mismatches.
  • RoleChangeLog with binding_id access patterns.
  • WAF alerts and blocked requests for unusual ID patterns.
  • SIEM correlation of atypical source_ip + binding accesses.

Purple-team validation commands

# Try to edit another user's role-binding by ID
curl -X PATCH -H "Authorization: Bearer <user-token>" https://api.example.com/v1/role_bindings/<other-binding-id> -d '{"role":"Admin"}'

# Detect cross-tenant binding modifications
API_Gateway_Logs
| where path contains "/role_bindings/" and claims.tenant_id != request.query.tenant_id

T3 — Confused deputy via service account Admin scope

STRIDE category: Elevation of Privilege
Risk: Critical likelihood, High impact
Attacker objective: Cause an automation/service flow (low-trust caller) to perform Admin actions via a service account with broad Admin scope.

How the attacker would exploit it

  • Prerequisite: service account used by lower-trust workflows has Admin permissions.
  • Initial action: invoke the workflow or API that uses the service account to perform actions on behalf of unprivileged callers.
  • Core exploitation: the service executes a privileged API (role change, data export) on behalf of attacker-controlled input.
  • Escalation: chain automated flows to create persistent admin accounts or export tenant data.
  • Impact realization: large-scale data exposure or role manipulation at machine speed.

Blue-team control review — what must stop the chain

  • Preventive: Apply least-privilege to service accounts; remove Admin by default and use scoped allowlists for actions.
  • Preventive: Implement caller-context checks (caller subject) before executing privileged actions; require proof-of-origin headers from authenticated services.
  • Detective: ServiceAccountUsage analytics (anomalous endpoint accesses, unusual call patterns) with SIEM rule ServiceAccountAdminPattern.
  • Preventive: Use token exchange (STS) with limited-lifetime delegated tokens instead of long-lived privileged tokens.
  • Corrective: Automated emergency rotation of service account keys and forced replay validation for actions executed in window.

Evidence and telemetry to validate controls

  • ServiceAccountUsage logs: service_account_id, caller_service, endpoint, args, tenant_id.
  • AuthZDecisionLog showing elevated decision for service principals.
  • SIEM ServiceAccountAdminPattern alert when service account performs disallowed role-write actions.
  • STS/token issuance logs to track delegated token flows.

Purple-team validation commands

# Simulate invoking a low-trust workflow that triggers privileged service call
curl -X POST -H "Authorization: Bearer <caller-token>" https://api.example.com/v1/workflow/run -d '{"action":"trigger_privileged"}'

# Detect service account performing role-write APIs
ServiceAccountUsage
| where endpoint contains "/roles" and method in ("POST","PATCH")
| summarize count() by service_account_id, bin(TimeGenerated, 5m)

T4 — Over-privileged default role template

STRIDE category: Information Disclosure / Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Exploit default templates that grant excessive access to new tenants or users.

How the attacker would exploit it

  • Prerequisite: onboarding process assigns permissive defaults.
  • Initial action: create accounts or onboard new tenants to inherit broad perms.
  • Core exploitation: use default access to reach sensitive APIs or data.
  • Escalation: pivot to automation flows or use default service accounts.
  • Impact realization: silent over-permission leading to data exposure.

Blue-team control review — what must stop the chain

  • Preventive: Harden built-in roles with least-privilege baselines and require explicit opt-in for expanded permissions.
  • Preventive: Policy linting for templates (CI) and automated staging tests that measure effective permission sets.
  • Detective: Onboarding audit logs and TemplateUsage monitoring for abnormal post-onboarding activities.
  • Preventive: Apply JIT elevation for higher privileges rather than permanent default grants.
  • Corrective: Auto-remediation to replace overly permissive defaults and notify tenant owners.

Evidence and telemetry to validate controls

  • OnboardingAudit showing template_id, assigned_roles, initiator.
  • Permission exposure scans reporting effective_permissions per tenant.
  • CI policy lint failure logs for role templates.
  • SIEM alerts for new tenant activity exceeding baseline usage patterns.

Purple-team validation commands

# Create a new tenant in staging to verify default roles are minimal

T5 — Bulk role sync from IdP without guardrails

STRIDE category: Tampering
Risk: High likelihood, High impact
Attacker objective: Cause a mass-grant of roles via IdP/SCIM sync errors or malicious group mappings.

How the attacker would exploit it

  • Prerequisite: IdP mapping to role templates and automated sync enabled.
  • Initial action: change IdP group membership or mapping to include broad permissions.
  • Core exploitation: SCIM sync applies changes at scale, granting many accounts elevated privileges.
  • Escalation: attacker gains access across many user accounts or creates backdoor memberships.
  • Impact realization: mass privilege expansion and rapid exposure.

Blue-team control review — what must stop the chain

  • Preventive: Enforce sync simulation and dry-run mode that surfaces delta changes before apply.
  • Preventive: Require risk-score evaluation for SCIM deltas and automatic pause above thresholds (bulk_threshold).
  • Detective: SCIMSync logs with delta_count, initiator, mapping_rule_id, and SIEM BulkRoleAssignmentSpike rule.
  • Corrective: Auto-rollback and staged application with approval for large deltas.
  • Preventive: Multi-stage rollout for group-to-role mappings and test harness for typical IdP changes.

Evidence and telemetry to validate controls

  • SCIMSync logs with fields mapping_rule_id, delta_count, changed_bindings.
  • Pre-apply simulation reports and SCIMDryRun audit artifacts.
  • SIEM alerts BulkRoleAssignmentSpike and mapping_rule correlation.
  • Tenant reports and monitoring of sudden permission growth.

Purple-team validation commands

# Trigger a dry-run SCIM sync with many group adds to ensure pause/alert
scim-cli sync --tenant <tenant> --dry-run --add-members bulk-list.json

# Detect large delta in SCIM sync

T6 — Unauthorized policy edit in admin console

STRIDE category: Tampering
Risk: High likelihood, High impact
Attacker objective: Modify policy objects in admin console to widen role permissions or to disable controls.

How the attacker would exploit it

  • Prerequisite: access to admin console (compromised admin/support user) or UI privilege escalation.
  • Initial action: edit policy document or publish new policy version with dangerous permission rules.
  • Core exploitation: policy engine uses altered policy, allowing elevated actions.
  • Escalation: attacker removes logging or throttle rules to avoid detection.
  • Impact realization: policy-wide privilege expansion and long-lived silent abuse.

Blue-team control review — what must stop the chain

  • Preventive: Require dual-approval for policy changes (two distinct approvers) and tie to a ticket/workflow system.
  • Preventive: Store policies as code in versioned repositories with CI checks and immutable deployment artifacts.
  • Detective: AdminAudit for ADMIN_POLICY_EDIT with diff snapshots and SIEM alert on policy changes outside CI deployment.
  • Preventive: Limit direct console edits to emergency-only with mandatory justification and post-facto review.
  • Corrective: Automated rollback to last known-good policy and full forensic snapshot preservation.

Evidence and telemetry to validate controls

  • AdminAudit events including actor_id, change_id, policy_diff, ticket_id.
  • Git commit metadata for policy-as-code pipelines and CI gate logs.
  • SIEM rule SuspiciousPolicyEdit triggered by console edit without matching CI deploy.
  • Immutable snapshot storage with hash checks (PolicySnapshotLog).

Purple-team validation commands

T7 — Break-glass misuse

STRIDE category: Repudiation
Risk: Medium likelihood, High impact
Attacker objective: Abuse emergency elevation workflows to gain sustained elevated access without proper incident justification.

How the attacker would exploit it

  • Prerequisite: break-glass mechanism exists with weak controls or long expiry.
  • Initial action: trigger emergency workflow or co-opt operator to request elevation.
  • Core exploitation: obtain elevated role that persists or bypasses audits.
  • Escalation: chain with other misconfigurations to mask activity.
  • Impact realization: long-lived elevated access with plausible deniability.

Blue-team control review — what must stop the chain

  • Preventive: Require contextual justification, time-bound TTL, and dual-approval for break-glass; require audit uploads of artifacts.
  • Detective: BreakGlassAudit with requestor, justification, expiry, and automatic SIEM correlation for suspicious use patterns.
  • Preventive: Short TTLs and mandatory session recording for actions performed during break-glass periods.
  • Corrective: Immediate automated revocation and post-incident review with reversion playbook.
  • Preventive: Limit break-glass scope to narrow emergency actions, not broad policy edits.

Evidence and telemetry to validate controls

  • BreakGlassAudit logs and session recordings tied to request IDs.
  • SIEM correlation of break-glass usage with anomalous admin activity.
  • TTL enforcement metrics and session termination logs.
  • Ticketing system linkage showing approvals.

Purple-team validation commands

T8 — Stale revoked roles due to cache delay

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Medium impact
Attacker objective: Continue to use revoked rights while authorization caches are stale.

How the attacker would exploit it

  • Prerequisite: revocation propagation delay across distributed caches.
  • Initial action: revoke a user's role via admin API and then use the old token during cache window.
  • Core exploitation: token accepted by cached policy decision or stale claims in gateway.
  • Escalation: combine with replay to extend window.
  • Impact realization: temporary unauthorized access to sensitive operations.

Blue-team control review — what must stop the chain

  • Preventive: Short cache TTLs for high-risk permissions and proactive invalidation APIs (push invalidation to gateways).
  • Preventive: Token revocation lists (TRL) and immediate session invalidation (InvalidateSessions API) on critical revokes.
  • Detective: CacheInvalidation metrics and AuthZCacheHit logs to monitor propagation lag.
  • Corrective: Force re-authentication for critical actions if revocation_time > token_issue_time.
  • Preventive: Design decision to favor real-time checks for highly sensitive actions.

Evidence and telemetry to validate controls

  • AuthZCache metrics: ttl, last_invalidation, hit_rate.
  • Session management logs: InvalidateSessions events.
  • SIEM rule RevocationWindowUse when token used post-revoke.
  • API gateway logs showing policy version used for decision.

Purple-team validation commands

# Revoke role and attempt action using cached token to verify deny

curl -X POST -H "Authorization: Bearer <old-token>" https://api.example.com/v1/admin/action

# Detect token usage after revocation
AuthZDecisionLog | where token_issue_time < revocation_time and decision == "Allow"

T9 — Weak audit log integrity / tampering

STRIDE category: Repudiation
Risk: Medium likelihood, Critical impact
Attacker objective: Modify or delete audit records to hide privileged changes.

How the attacker would exploit it

  • Prerequisite: insufficiently protected log storage or single-sink logging architecture.
  • Initial action: alter or delete role-change records or disable the logging pipeline.
  • Core exploitation: perform privileged changes without forensic trace.
  • Escalation: combine with backdated or altered audits to mislead investigators.
  • Impact realization: inability to detect or prove malicious admin activity.

Blue-team control review — what must stop the chain

  • Preventive: Append-only storage with tamper-evident hashing and replication to independent sinks (e.g., object store with immutability and KMS encryption).
  • Detective: AuditIntegrityChecker running hash-chain validation with SIEM alert on mismatch (AuditHashMismatch).
  • Preventive: Role separation for log management and no direct delete rights in log store.
  • Corrective: Switch to secondary trusted sink and initiate incident with preserved snapshots.
  • Preventive: Store a digest of audit streams in external transparency ledger (CT-like) for cross-checks.

Evidence and telemetry to validate controls

  • AuditIntegrityChecker output: validation_result, sequence_gap, mismatch_count.
  • Object store access logs with principals and operations (PUT/DELETE).
  • SIEM rule AuditHashMismatch triggered by hash-chain validation failures.
  • Cross-sink replication job logs showing last successful replication.

Purple-team validation commands

# Run audit integrity check in staging to ensure mismatch triggers alert
audit-checker --scan --since 24h --output findings.json

# Detect log deletions or unexpected writes
ObjectStoreAccess | where operation in ("Delete","Put") and bucket == "audit-logs" | summarize count() by principal

T10 — Excessive custom role complexity

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Leverage complex custom role combinations to create unintended allow paths.

How the attacker would exploit it

  • Prerequisite: customers can create roles with wildcard or combined permissions.
  • Initial action: craft role compositions that include subtle allow rules (e.g., resource wildcard with action include).
  • Core exploitation: combine role and group mappings to obtain broader effective permissions.
  • Escalation: chain across APIs to reach sensitive resources.
  • Impact realization: access outside documented intent leading to data leakage.

Blue-team control review — what must stop the chain

  • Preventive: Policy lint rules that forbid dangerous combinations and wildcards (CustomRoleDangerCombo policy) enforced in CI and admin UI.
  • Detective: Simulation tests showing effective permissions across APIs prior to publish.
  • Preventive: UI restrictions and templates that discourage excessive role composition.
  • Corrective: Automatic rollback of published custom roles that fail simulation or are flagged by tenant security teams.
  • Preventive: Require explicit justification and dual-approval for roles touching high-impact actions.

Evidence and telemetry to validate controls

  • CustomRoleLint reports and CI gate logs.
  • PolicySimulation outputs showing coverage and effective permissions.
  • SIEM alerts CustomRoleDangerCombo when a new role includes blocked combinations.
  • Support tickets and customer security review logs.

T11 — Token replay for privileged role changes

STRIDE category: Spoofing
Risk: High likelihood, High impact
Attacker objective: Replay or reuse tokens that perform role changes to make persistent unauthorized grants.

How the attacker would exploit it

  • Prerequisite: reusable or long-lived tokens with role-change permissions.
  • Initial action: capture token (via logging, misconfig, or intercept) and replay role-change API calls.
  • Core exploitation: perform role changes while token still valid.
  • Escalation: rotate through tokens or use replay to obfuscate origin.
  • Impact realization: unauthorized persistent role modifications.

Blue-team control review — what must stop the chain

  • Preventive: Short-lived tokens for role-change operations and enforce proof-of-possession where possible.
  • Preventive: Avoid logging tokens or secrets; scan code and logs for token leakage.

T12 — Privilege creep over lifecycle

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Over time, users accumulate privileges through role changes, group additions, and service grants leading to excessive access.

How the attacker would exploit it

  • Prerequisite: permissive role assignments and weak hygiene reviews.
  • Initial action: accumulate small privileges via legitimate operations or repeated grants.
  • Core exploitation: combine accumulated rights to access sensitive APIs or escalate further.
  • Escalation: exploit aged privileges that are no longer appropriate.
  • Impact realization: stealthy unauthorized access by legitimate accounts.
