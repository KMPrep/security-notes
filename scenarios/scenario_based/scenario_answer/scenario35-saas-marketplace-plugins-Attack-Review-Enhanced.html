# Scenario 35 — SaaS Marketplace / Plugin Architecture — Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario35-saas-marketplace-plugins.md
Generated artifact: scenario35-saas-marketplace-plugins-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead — 15+ years of purple-team experience
Scenario: A SaaS marketplace hosts ~300 third-party plugins that run in a permissioned runtime and hold tenant-scoped OAuth tokens. The platform must prevent supply-chain compromise, runtime escapes, token abuse, and cross-tenant data exfiltration while enabling safe updates at scale.

1. Review Objective

This review examines how an adversary would exploit marketplace and plugin attack surfaces (supply-chain, publisher account compromise, token theft, sandbox escape, webhook channels) and prescribes specific blue-team controls, detection telemetry (log fields and rules), safe purple-team validation snippets, and mermaid attack trees per threat.

2. Highest-Risk Design Decision

The single highest-risk design decision is allowing plugins to receive and act on long-lived, tenant-scoped delegated tokens combined with automatic or broad updates based on a single publisher signing key. If a developer account, signing key, or update channel is compromised, malicious code can reach thousands of tenants with token-based access already authorized — enabling stealthy exfiltration that mimics legitimate calls. Preferred safe choice: per-tenant authorization scoping, short TTLs, per-tenant or per-plugin KEKs for sensitive flows, staged rollouts, and multi-factor signing/endorsement for publish actions.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker actions or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision points.
  • Green nodes (:::blocked): attack stopped or contained.
  • Yellow nodes (:::impact): business or legal impact.

6. Threat-by-Threat Technical Review

T1 — Malicious initial plugin submission

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Ship a plugin whose advertised behavior appears legitimate but exfiltrates tenant data through allowed APIs.

How the attacker would exploit it

  • Prerequisite: Developer account able to publish plugins.
  • Step 1: Implement exfiltration logic hidden behind normal feature flags or triggered by tenant attributes.
  • Step 2: Pass static checks by obfuscating network destinations or using allowed APIs for exfil.
  • Step 3: Publish plugin; initial installs appear benign.
  • Step 4: Once installed at scale, activate exfiltration path.
  • Step 5: Use token refreshes or persistence to sustain access.

Blue-team control review — what must stop the chain

  • Control 1: Mandatory static analysis + sensitive-data-flow checks during submission (SBOM + taint flow enforcement).
  • Control 2: Enforce artifact signing and require two independent reviewer approvals for plugins requesting high-risk scopes.
  • Control 3: UI-level scope risk labels and deny-by-default for sensitive access combinations.
  • Control 4: Runtime DLP on plugin outbound channels and egress allowlists per plugin.

Evidence and telemetry to validate controls

  • PublishAudit: developer_id, plugin_id, version, review_status.
  • SBOMMonitor results, taint_flow scan outputs indicating sensitive sink usage.
  • DLP outbound events showing destination or payload classification.
  • InstallMetrics showing unusual rapid installs across tenants.

Purple-team validation commands

# Submit a staging plugin with a benign exfil test and ensure prepublish checks flag outbound sensitive sinks
curl -X POST https://marketplace.staging.example/publish -F "artifact=@test-plugin.zip" -H "Authorization: Bearer <dev-token>"

# Detect plugins that reference sensitive APIs in SBOM/taint flow
SBOMMonitor | where taint_flow contains "DataAPI" and taint_flow contains "sensitive"

T2 — Compromised publisher account pushes malicious update

STRIDE category: Tampering
Risk: Critical likelihood, High impact
Attacker objective: Use stolen developer credentials to publish updates that turn benign plugins malicious, rapidly expanding the blast radius via auto-update.

How the attacker would exploit it

  • Prerequisite: obtain publisher credentials or signing key.
  • Step 1: Authenticate to publisher portal or CI and push a malicious version.
  • Step 2: Bypass or exploit weak review gating (emergency publish path or missing checks).
  • Step 3: Trigger auto-update to installed tenant fleets.
  • Step 4: Use tenant tokens to call data APIs and exfiltrate.

Blue-team control review — what must stop the chain

  • Control 1: Enforce hardware-backed MFA for publisher accounts and require enrollment of recovery contacts.
  • Control 2: Per-publisher signing keys in HSM with usage logged and dual-approval for key usage (HSMPolicy:SignWithApproval).
  • Control 3: Require staged rollouts with canary monitoring and automatic pause on anomaly thresholds.
  • Control 4: Monitor publisher auth anomalies with alerting on new device/geography and immediate lockout on suspicious activity.

Evidence and telemetry to validate controls

  • PublisherAuth logs showing new device, IP, and publish_event correlation.
  • HSMKeyUsage audit entries for signing operations and requester principal.
  • RolloutLog indicating staged percent and canary alerts.
  • SIEM PublisherCompromise correlation rule combining auth anomaly + publish event.

Purple-team validation commands

# Simulate compromised publisher auth in staging and ensure publish is blocked or requires extra sign-off
curl -X POST -H "Authorization: Bearer <dev-token>" https://publisher.staging.example/publish -F "artifact=@evil.zip"

# Detect publisher publish from new geo/device
PublisherAuth | where action == "publish" and geo notin (ApprovedGeos)

T3 — Over-scoped consent accepted by tenant admins

STRIDE category: Elevation of Privilege
Risk: High likelihood, High impact
Attacker objective: Gain broader access by requesting wide scopes and relying on tenant admin consent or confusing UX.

How the attacker would exploit it

  • Prerequisite: plugin requests high-risk scopes and tenant admin approves.
  • Step 1: Present consent UI that obscures sensitive scopes (bundling, vague wording).
  • Step 2: Use social engineering or convincing feature set to induce consent.
  • Step 3: Use granted scopes to access tenant data beyond intended purpose.

Blue-team control review — what must stop the chain

  • Control 1: Deny-by-default consent for high-risk scope sets; require explicit justification and owner attestation.
  • Control 2: Scope combinator checks: disallow dangerous scope combinations and surface risk labels.
  • Control 3: Enforce re-consent on scope expansion (force approval flow when plugin updates request new scopes).
  • Control 4: Tenant-level scope monitoring with alerts on abnormal object types accessed by plugins.

Evidence and telemetry to validate controls

  • ConsentLog entries capturing approved_scopes, approver identity, and consent timestamp.
  • TokenService records: client_id, tenant_id, issued_scopes and refresh patterns.
  • PluginRuntime call logs showing attempts outside expected resource patterns.
  • SIEM rule OverConsentAbuse correlating consent events and immediate data spikes.

Purple-team validation commands

# Test consent UI by requesting high-risk scopes in staging and ensure UI blocks or surfaces risk (manual validation)
curl -X POST https://auth.staging.example/consent -d '{"client_id":"<plugin>","scopes":["read:all_users","export:full"] }'

# Alert when a token is issued with sensitive scopes and immediate high-volume access follows
TokenService | where issued_scopes contains "export:full" | join (PluginRuntime | where api_action == "read") on client_id

T4 — Token leakage in logs or repos

STRIDE category: Information Disclosure
Risk: High likelihood, Critical impact
Attacker objective: Obtain client secrets, tokens, or refresh tokens from logs, repo artifacts, or misconfigured debug outputs, then re-use them to access tenant data.

How the attacker would exploit it

  • Prerequisite: artifacts, logs, or repo contain tokens or secrets; token reuse allowed across scopes.
  • Step 1: Harvest tokens from public repos, staging logs, or leaked storage.
  • Step 2: Use tokens to call tenant APIs or exchange refresh tokens for access tokens.
  • Step 3: Maintain access with refresh tokens or obtain persistent credentials.

Blue-team control review — what must stop the chain

  • Control 1: Prevent logging of secrets via platform-wide logging policies and scanner that rejects artifacts with secret-like patterns (SecretScanner).
  • Control 2: Short-lived tokens with audience and scope restrictions and mandatory rotation on publish.
  • Control 3: Monitor for token usage from unexpected geos or IPs and alert on unusual refresh patterns.
  • Control 4: Use token binding and audience restriction (cannot be used from other origins) and rotated refresh tokens.

Evidence and telemetry to validate controls

  • SecretScanner findings for repo or artifact uploads.
  • TokenService logs with refresh_count, issue_ip, and client_audience.
  • SIEM TokenUseAnomaly rule for cross-geo or impossible travel patterns.
  • Artifact storage logs showing artifact uploads and access patterns.

Purple-team validation commands

# Upload sample artifact with a test token to staging and ensure pipeline rejects or redacts it

# Detect token use from unexpected IPs
TokenService | where client_id == "<plugin>" and issue_ip notin (KnownPluginIPs)

T5 — Sandbox escape / runtime breakout

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Exploit sandbox vulnerabilities or misconfigurations to execute code outside the plugin runtime and access platform internals or other tenants' data.

How the attacker would exploit it

  • Prerequisite: existence of a sandbox escape vulnerability or overly permissive runtime access controls.
  • Step 1: Trigger vulnerable code path or exploit container escape via syscall chain.
  • Step 2: Gain host-level access or access to shared services (metadata, key stores).
  • Step 3: Use host access to read tenant vaults or manipulate other plugin runtimes.

Blue-team control review — what must stop the chain

  • Control 1: Strong process isolation (microVMs, gVisor, or hardware virtualization) with least-privilege capabilities and seccomp/SECCOMP filters.
  • Control 2: Limit access to metadata services (IMDS) and require metadata service v2/IMDSv2-like protections.
  • Control 3: Host-based intrusion detection and kernel hardening; runtime deny-lists for syscalls.
  • Control 4: Regular runtime fuzzing and automated patching of underlying components.

Evidence and telemetry to validate controls

  • SandboxAlert entries: plugin_id, violation_type, syscall.
  • Host telemetry: kernel exception logs, container escape detection, abnormal process creation.
  • RuntimePolicy enforcement logs showing blocked capabilities.
  • Patch and vulnerability management records for runtime stack components.

Purple-team validation commands

# Run known sandbox escape PoC in isolated staging runtime to assert it is blocked/contained (authorized test only)
./run-poc --target runtime-staging --poc sandbox-escape-test

# Detect unexpected host-level process creation from plugin runtime
HostProcessEvents | where parent_process == "plugin-runtime" and process_name in ("/bin/sh","curl")

T6 — Webhook / callback exfiltration channel

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Use callback endpoints or webhook channels to push tenant data to attacker-controlled endpoints, often using allowed outbound flows.

How the attacker would exploit it

  • Prerequisite: ability to register or configure outbound endpoints or instruct plugin to call attacker-controlled callbacks.
  • Step 1: Plugin crafts outbound payloads to attacker endpoints disguised as telemetry or feature usage.
  • Step 2: Use allowed egress paths and mimic expected protocols to avoid DLP.
  • Step 3: Aggregate exfiltrated data ex-post for extraction.

Blue-team control review — what must stop the chain

  • Control 1: Outbound allowlists and egress proxy with enterprise DNS whitelisting and TLS inspection for high-risk destinations.
  • Control 2: Per-plugin sanctioned webhook registry with approval and signature verification of callback destinations.
  • Control 3: DLP inspection for sensitive fields and payload classification on outbound plugin traffic.
  • Control 4: Rate-limiting and destination anomaly detection for plugin outbound calls.

Evidence and telemetry to validate controls

  • OutboundProxy logs: plugin_id, dst_host, tls_subject and bytes_sent.
  • DLP logs showing matched sensitive payloads and payload_class.
  • Webhook registry changes and approval records.
  • SIEM OutboundEgressAnomaly when plugin sends traffic to unusual destinations.

Purple-team validation commands

T7 — Dependency / transitive package compromise

STRIDE category: Tampering
Risk: Medium likelihood, High impact
Attacker objective: Introduce malicious code via a compromised dependency in the plugin build chain (typosquatting, maintainer compromise) that evades static checks.

How the attacker would exploit it

  • Prerequisite: plugin depends on third-party package; dependency resolution is permissive.
  • Step 1: Compromise a transitive dependency or publish malicious package with similar name.
  • Step 2: Build process pulls malicious code and signs artifact.
  • Step 3: Malicious code runs in runtime with plugin privileges.

Blue-team control review — what must stop the chain

  • Control 1: SBOM generation and policy enforcement against unexpected dependency changes (SBOMPolicy:NoNewMaintainer).
  • Control 2: Dependency pinning and reproducible builds; require deterministic build outputs checked by verifier.
  • Control 3: Automated dependency vulnerability and maintainer change alerts with immediate blocking of versions failing provenance checks.
  • Control 4: Signed dependency provenance and supply-chain attestations (e.g., Sigstore/TUF-based verification).

Evidence and telemetry to validate controls

  • SBOMMonitor alerts on hash changes, unknown maintainers, or newly added packages.
  • CI build logs showing deterministic build hash and signature verification.
  • Dependency vulnerability scanner outputs and maintainer change events.
  • SIEM DependencyCompromise correlation for plugin builds.

STRIDE category: Tampering
Risk: Medium likelihood, High impact

How the attacker would exploit it

T10 — DLP evasion via allowed APIs

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Exfiltrate sensitive fields by splitting data across multiple allowed API calls or encoding payloads to avoid DLP signatures.

How the attacker would exploit it

  • Prerequisite: plugin has access to allowed APIs and can craft request payloads.
  • Step 1: Break sensitive content into small chunks or encode in innocuous fields.
  • Step 2: Send many small outbound requests to blend with normal traffic.
  • Step 3: Reassemble exfiltrated fragments offline.

Blue-team control review — what must stop the chain

  • Control 1: Field-level DLP that reconstructs multi-request exfil patterns and correlates by session or plugin id.
  • Control 2: Rate-limiting on sensitive resource reads per plugin and per tenant with spike detection.
  • Control 3: Context-aware anomaly detection that flags unusual sequences of API calls by plugin.
  • Control 4: Harden allowed payload formats and require schema enforcement on API inputs/outputs.

Evidence and telemetry to validate controls

  • DLP logs with session identifiers and matched fragments.
  • PluginRuntime sequence logs showing repeated small reads/exports.
  • SIEM FragmentedExfil rule correlating many small calls from same plugin.
  • Rate-limit logs indicating throttle events.

T12 — Kill-switch / revocation propagation failure

STRIDE category: Repudiation
Risk: Medium likelihood, High impact
Attacker objective: Continue exfiltration or operation after disablement because revocation did not propagate to all runtime nodes or tokens persisted.

How the attacker would exploit it

  • Prerequisite: slow or incomplete kill-switch propagation or stale tokens in caches.
  • Step 1: Malicious activity detected and central kill-switch invoked.
  • Step 2: Some runtime nodes or tenant caches still honor old tokens or execute cached plugin code.
  • Step 3: Exfiltration continues for window until propagation completes.

Blue-team control review — what must stop the chain

  • Control 1: Synchronous or strongly consistent revocation paths for tokens (CDS broadcast + per-region push) and immediate runtime enforcement checks.
  • Control 2: Short TTLs on tokens and binding to session to reduce time-to-kill.
  • Control 3: Health checks for kill-switch propagation and alert on region lag (KillSwitchHealth metric).
  • Control 4: Forced sandbox restart and node quarantine on kill events to evict cached state.

  • KillSwitch events with per-region propagation timestamps and nodes_pending counts.
  • TokenService revoke logs and token_revocation_latency metrics.
