Scenario 37 - SaaS Encryption Key Management (Bring Your Own Key) - Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario37-saas-byok-encryption.md
Prompt source: ../threat-modeling-prompt.md
Generated artifact: scenario37-saas-byok-encryption-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead - 15+ years of purple-team experience
Scenario: A SaaS platform adds BYOK support so enterprise customers can use AWS KMS, Azure Key Vault, or similar customer-managed keys to protect tenant data at rest. The platform must handle KMS grants, envelope encryption, revocation semantics, plaintext caching, decrypt monitoring, legacy backups, cross-region behavior, and customer self-lockout risk.

1. Review Objective

This review turns the BYOK scenario into attacker-centric exploitation chains and blue-team controls. It focuses on what a strong senior security answer should cover: cryptographic trust boundaries, operational failure modes, revocation behavior, telemetry, incident response, customer shared responsibility, and residual risk.

2. Highest-Risk Design Decision

The highest-risk design decision is treating BYOK as a pure encryption feature instead of a distributed authorization and lifecycle-control system.

BYOK only provides customer control if the SaaS data plane fails closed when the key is disabled, uses the correct tenant-scoped key every time, does not retain plaintext beyond a tightly bounded processing window, and provides auditable proof of KMS use. If the platform caches plaintext too long, keeps platform-key backups, has broad KMS grants, or lets internal support tooling bypass the BYOK path, the customer's key ownership becomes mostly cosmetic.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker action or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision point.
  • Green nodes (:::blocked): attack stopped or contained.
  • Yellow nodes (:::impact): confidentiality, availability, compliance, or customer trust impact.

6. Reference Architecture

flowchart LR
    TenantAdmin[Tenant admin] --> KeySetup[BYOK setup and validation]
    KeySetup --> KeyMap[Signed tenant-key mapping service]
    KeyMap --> EncryptSvc[Encryption service]
    EncryptSvc --> KMS[Customer KMS or Key Vault]
    EncryptSvc --> Store[Encrypted object store]
    Store --> Backup[Backup and replica stores]
    EncryptSvc --> Cache[Plaintext/key cache with TTL]
    KMS --> KMSLogs[KMS audit logs]
    EncryptSvc --> DecryptLogs[Decrypt telemetry]
    Cache --> CacheLogs[Cache eviction telemetry]
    KeyMap --> ChangeLogs[Key mapping audit]
    KMSLogs --> SIEM[SIEM and SOAR]
    DecryptLogs --> SIEM
    CacheLogs --> SIEM
    ChangeLogs --> SIEM

Security properties:

  • Tenant data uses envelope encryption: per-object or per-record DEKs are wrapped by customer-managed KEKs.
  • Tenant-to-key mapping is signed, versioned, and changed only through approved workflow.
  • Decrypt paths fail closed if key state is unknown, denied, disabled, or revoked.
  • Plaintext and unwrapped DEKs have short TTLs and are flushed on revocation.
  • Backup and replication key class is visible and reconciled.
  • KMS audit logs are correlated with SaaS decrypt logs.

7. Threat-by-Threat Technical Review

T1 - Over-broad customer KMS grants to SaaS principals

STRIDE category: Elevation of Privilege / Information Disclosure
Risk: High likelihood, Critical impact
Attacker objective: Abuse KMS grants that allow the SaaS principal to decrypt more data than intended.

How the attacker would exploit it

  • Prerequisite: customer KMS policy grants broad decrypt to a SaaS principal without tenant context, encryption context, or key conditions.
  • Step 1: Attacker compromises or abuses a SaaS workload identity.
  • Step 2: Call KMS decrypt with wrapped DEKs from multiple objects or tenants.
  • Step 3: Use returned plaintext DEKs to decrypt data.
  • Step 4: Blend KMS calls into normal platform decrypt volume.

Blue-team control review - what must stop the chain

  • Preventive: Require tenant-scoped KMS grants with encryption context conditions such as tenant_id, region, and key purpose.
  • Preventive: Validate key policy templates during BYOK onboarding and continuously after setup.
  • Preventive: Use separate workload identities per environment, region, and tenant risk class.
  • Detective: Alert on decrypt calls where encryption context does not match expected tenant-key mapping.
  • Corrective: Disable SaaS grant, rotate workload identity, and force customer key policy remediation.

Evidence and telemetry to validate controls

  • KMSGrantAudit: tenant_id, key_id, principal, conditions, validated.
  • KMSDecryptLog: key_id, principal, encryption_context, source_ip, result.
  • KeyPolicyValidationLog: tenant_id, key_id, finding, severity, remediation.
  • WorkloadIdentityLog: principal, workload, environment, attestation_result.

T2 - Tenant-to-key mapping tampering or key confusion

STRIDE category: Tampering / Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Change which key is used for a tenant, causing wrong-key decrypt, availability loss, or policy bypass.

How the attacker would exploit it

  • Prerequisite: control-plane API or config store can update tenant key references.
  • Step 1: Modify tenant key ARN, key version, vault URI, or region metadata.
  • Step 2: Map tenant data to an attacker-controlled or wrong customer key.
  • Step 3: Trigger rewrap, restore, or read operations using the incorrect mapping.
  • Step 4: Cause data exposure, lockout, or integrity confusion.

Blue-team control review - what must stop the chain

  • Preventive: Store key mappings in signed, append-only, versioned config with two-person approval for changes.
  • Preventive: Bind each encrypted object metadata to tenant ID, key ID, key version, and encryption context.
  • Preventive: Validate key ownership and customer proof-of-control before accepting a new key.
  • Detective: Alert on key mapping changes outside approved workflow or unusual volume of key map reads/updates.
  • Corrective: Roll back to previous signed mapping and freeze decrypt for affected tenant until verified.

Evidence and telemetry to validate controls

  • KeyMappingAudit: tenant_id, old_key, new_key, version, signed_by, approved_by.
  • ConfigChangeLog: actor, change_type, ticket_id, source, result.
  • ObjectCryptoMetadataLog: object_id, tenant_id, key_id, key_version, dek_id.
  • ProofOfControlLog: tenant_id, key_id, challenge_id, validated_at.

Purple-team validation commands

# In staging, attempt key mapping change without approval and verify it is rejected.

T3 - Key revocation race or fail-open decrypt behavior

STRIDE category: Information Disclosure / Denial of Service
Risk: Medium likelihood, Critical impact
Attacker objective: Continue decrypting after customer key disable or exploit error handling that serves data when key state is uncertain.

How the attacker would exploit it

  • Prerequisite: revocation state is polled slowly, cached locally, or KMS errors are treated as transient allow conditions.
  • Step 1: Customer disables key or attacker triggers key-state ambiguity.
  • Step 2: Platform continues using cached key state or retries from stale workers.
  • Step 3: Decrypt succeeds after it should have failed.
  • Step 4: Customer loses confidence in BYOK revocation semantics.

Blue-team control review - what must stop the chain

  • Preventive: Decrypt service must fail closed on disabled, denied, unknown, or stale key state.
  • Preventive: Subscribe to KMS/key lifecycle events where possible and poll as backup.
  • Preventive: Mark tenant as key-revoked or key-unknown in a strongly consistent state store.
  • Detective: Alert on any decrypt success after key disabled or revocation timestamp.
  • Corrective: Force tenant read lockout, flush caches, restart decrypt workers, and open incident.

Evidence and telemetry to validate controls

  • KeyStateLog: tenant_id, key_id, state, observed_at, source.
  • DecryptServiceLog: tenant_id, key_id, decrypt_result, key_state_seen, worker_id.
  • TenantLockoutLog: tenant_id, reason, started_at, ended_at.
  • KMSAvailabilityLog: provider, region, error_code, retry_policy.

Purple-team validation commands

# In staging, disable a test key and verify decrypt fails closed immediately.
byok-test revoke-key --tenant synthetic-tenant --key test-key --expect-decrypt-deny

DecryptServiceLog
| join kind=inner KeyStateLog on tenant_id, key_id
| where state in ("disabled", "revoked") and decrypt_result == "success"
| project TimeGenerated, tenant_id, key_id, worker_id, key_state_seen

T4 - Plaintext cache persists after key revocation

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Access cached plaintext data or unwrapped data keys after the customer has revoked the BYOK key.

How the attacker would exploit it

  • Prerequisite: data plane caches plaintext objects, query results, or unwrapped DEKs with long TTLs.
  • Step 1: Trigger high-value reads before key revocation.
  • Step 2: Customer disables key.
  • Step 3: Attacker or normal user reads data from cache, search index, queue, or worker memory.
  • Step 4: BYOK revocation does not actually stop access.

Blue-team control review - what must stop the chain

  • Preventive: Keep plaintext and unwrapped DEK TTLs short and bounded.
  • Preventive: Tag cache entries by tenant and key version for targeted purge.
  • Preventive: Exclude high-sensitivity plaintext from shared caches and search indexes where possible.
  • Detective: Require cache purge acknowledgments from every region and worker pool.
  • Corrective: Force tenant cache flush, worker restart, and memory/cache invalidation on key revocation.

Evidence and telemetry to validate controls

  • CacheEvictionLog: tenant_id, key_id, revocation_id, region, status.
  • MemoryTTLMetric: service, tenant_id, plaintext_ttl_ms, dek_ttl_ms.
  • CacheReadLog: tenant_id, key_id, cache_hit, key_state_at_read.
  • WorkerRestartLog: tenant_id, worker_pool, restart_reason, completed_at.

Purple-team validation commands

T5 - SaaS workload identity compromise enables decrypt abuse

STRIDE category: Spoofing / Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Use compromised service credentials to call customer KMS and SaaS data APIs.

How the attacker would exploit it

  • Prerequisite: workload identity, service account token, mTLS credential, or cloud role is stolen.
  • Step 1: Replay service credential from attacker-controlled environment or compromised workload.
  • Step 2: Call KMS decrypt and SaaS data APIs.
  • Step 3: Exfiltrate decrypted data while appearing as legitimate platform service.
  • Step 4: Continue until identity is revoked or anomalous behavior is detected.

Blue-team control review - what must stop the chain

  • Preventive: Use workload identity with hardware or platform attestation and short-lived credentials.
  • Preventive: Bind KMS access to workload, environment, region, tenant, and encryption context.
  • Preventive: Deny KMS access from unapproved networks and principals.
  • Detective: Alert on new workload identities, unusual regions, or decrypt from non-attested workloads.
  • Corrective: Revoke workload identity, rotate credentials, and trigger tenant secret/data exposure review.

Evidence and telemetry to validate controls

  • WorkloadIdentityLog: principal, workload, attestation, region, token_ttl.
  • KMSDecryptLog: principal, source_ip, user_agent, key_id, encryption_context.
  • NetworkEgressLog: service, destination, region, allowed.
  • IdentityRevocationLog: principal, revoked_at, reason, dependent_services.

T6 - Customer self-ransomware through key disable/delete

STRIDE category: Denial of Service / Tampering
Risk: High likelihood, High impact
Attacker objective: Abuse customer-controlled key lifecycle to lock the tenant out of their own data.

How the attacker would exploit it

  • Prerequisite: compromised or malicious customer admin can disable, schedule deletion, or revoke grants on the KMS key.
  • Step 1: Disable key or revoke SaaS decrypt grants.
  • Step 2: SaaS fail-closed behavior makes tenant data unavailable.
  • Step 3: Attacker demands ransom or sabotages business operations.
  • Step 4: Customer asks SaaS provider to recover data without the key.

Blue-team control review - what must stop the chain

  • Preventive: Provide customer-side guardrails: warning banners, break-glass roles, deletion waiting periods, and multi-admin approval guidance.
  • Preventive: Detect destructive KMS changes and notify tenant security contacts immediately.
  • Preventive: Document that SaaS cannot decrypt BYOK data after irrevocable key deletion.
  • Detective: Alert on unusual disable/delete/grant revoke events, especially outside business hours.
  • Corrective: Enter tenant lockout playbook, validate customer identity, and guide key restore or grant repair where possible.

Evidence and telemetry to validate controls

  • CustomerKMSLog: tenant_id, key_id, operation, actor, source_ip, result.
  • TenantLockoutLog: tenant_id, trigger, state, customer_contacts_notified.
  • CustomerAdminAlertLog: tenant_id, notification_type, delivered, acknowledged.
  • RecoveryRunbookLog: tenant_id, case_id, step, owner, status.

T7 - High-volume decrypt exfiltration under valid key

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Use valid decrypt permissions to access unusually high volumes of tenant data.

How the attacker would exploit it

  • Prerequisite: attacker has valid SaaS API session, service credential, or compromised automation.
  • Step 1: Enumerate high-value objects or export endpoints.
  • Step 2: Trigger many decrypt operations through normal read APIs.
  • Step 3: Keep requests below simple rate limits or spread across regions/workers.
  • Step 4: Exfiltrate data while KMS calls appear authorized.

Blue-team control review - what must stop the chain

  • Preventive: Apply per-tenant and per-principal decrypt budgets and sensitive export controls.
  • Preventive: Require step-up authorization for bulk export or unusually sensitive reads.
  • Detective: Baseline decrypt rate by tenant, service, data class, and time of day.
  • Detective: Correlate decrypt spikes with API reads, exports, and endpoint DLP.
  • Corrective: Throttle decrypt service, suspend session, and rotate exposed credentials.

Evidence and telemetry to validate controls

  • DecryptAnomalyLog: tenant_id, principal, decrypt_count, baseline, severity.
  • APIAccessLog: tenant_id, user_id, route, record_count, data_class.
  • DLPExportLog: tenant_id, export_id, data_class, volume, destination.
  • RateLimitLog: tenant_id, principal, budget_used, action_taken.

T8 - Legacy backups or replicas remain under platform keys

STRIDE category: Information Disclosure / Compliance
Risk: Medium likelihood, High impact
Attacker objective: Access data copies that were not re-encrypted under the customer's BYOK key.

How the attacker would exploit it

  • Prerequisite: data created before BYOK remains in backups, analytics stores, search indexes, or replicas encrypted with platform-managed keys.
  • Step 1: Target backup restore path, replica store, or legacy snapshot.
  • Step 2: Access data without customer KMS interaction.
  • Step 3: Bypass customer expectation that disabling BYOK blocks access.
  • Step 4: Create compliance and contractual dispute.

Blue-team control review - what must stop the chain

  • Preventive: Build a key-class inventory for every storage tier, backup, replica, index, and derived dataset.
  • Preventive: Re-encrypt or expire legacy backups according to a documented migration plan.
  • Preventive: Prevent restore of BYOK tenants into platform-key-only environments.
  • Detective: Alert on backup or restore jobs using platform keys for BYOK tenants.
  • Corrective: Re-encrypt affected sets and disclose residual exposure window to customer if required.

Evidence and telemetry to validate controls

  • BackupKeyInventory: backup_id, tenant_id, key_class, created_at, expires_at.
  • RestoreAudit: restore_id, tenant_id, source_key_class, target_key_class, approved.
  • DerivedDataInventory: dataset, tenant_id, contains_customer_data, key_class.
  • MigrationProgressLog: tenant_id, storage_tier, percent_reencrypted, deadline.

T9 - Cross-region or residency key mismatch

STRIDE category: Information Disclosure / Denial of Service
Risk: Medium likelihood, High impact
Attacker objective: Cause data to be encrypted, decrypted, replicated, or restored with a key outside the approved region or tenant policy.

How the attacker would exploit it

  • Prerequisite: SaaS replication or failover logic falls back to platform keys, wrong region keys, or wrong provider vault.
  • Step 1: Trigger region failover, backup restore, or migration.
  • Step 2: Data plane attempts decrypt using unavailable customer key.
  • Step 3: Fallback path uses wrong key or breaks tenant availability.
  • Step 4: Data residency or BYOK commitments fail.

Blue-team control review - what must stop the chain

  • Preventive: Encode tenant key region, data residency, and provider constraints in key mapping policy.
  • Preventive: Fail closed if the approved regional key is unavailable.
  • Preventive: Require customer-provided keys for every approved replica/failover region.
  • Detective: Alert on decrypt/encrypt requests where data region and key region differ from policy.
  • Corrective: Stop replication, freeze restore, and require explicit customer-approved remap.

10. Minimum Viable Control Set

If implementation time is constrained, the minimum viable control set is:

  1. Envelope encryption with tenant-scoped KMS grants and encryption context validation.
  2. Signed, versioned tenant-to-key mapping with two-person approval for changes.
  3. Fail-closed decrypt behavior for disabled, denied, unknown, or unavailable key state.
  4. Plaintext and DEK cache TTLs with revocation-triggered purge and per-region acknowledgments.
  5. KMS decrypt telemetry correlated with SaaS API, cache, backup, and support-access logs.

Do not ship BYOK if revocation does not stop access, support tooling can bypass key checks, or backups remain under platform keys without explicit residual-risk disclosure.

12. Google-Scale Considerations

At Google scale, BYOK must be operated like a critical control plane.

  • Policy-as-code onboarding: Generate and validate KMS/Key Vault policies automatically, including principals, conditions, region, and encryption context.
  • Continuous reconciliation: Reconcile key mapping, KMS grant state, object crypto metadata, backup key class, and revocation behavior continuously.
  • Revocation drills: Regularly test key disable scenarios across regions to prove fail-closed behavior and cache purge completeness.
  • Blast-radius modeling: A mapping bug or workload identity compromise should automatically identify affected tenants, keys, objects, backups, and regions.
  • Customer proof: Provide auditable reports showing key usage, decrypt counts, revocation tests, backup key class, and policy drift.
  • Paved-road support tooling: Internal support access must use the same BYOK enforcement path with JIT and ticket-bound audit.

13. Model Answer Summary - 90-second version

For BYOK, I would frame the core security question as: does customer key control actually govern access to tenant data, or only the primary storage path? The main risks are over-broad KMS grants, tampered tenant-to-key mapping, fail-open decrypt behavior, plaintext cache persistence after key revocation, and legacy backups outside BYOK coverage.

My design would use envelope encryption with per-object DEKs wrapped by tenant customer keys, tenant-scoped KMS grants with encryption context, signed and versioned key mapping, fail-closed decrypt behavior, short plaintext and DEK cache TTLs, and mandatory cache purge on revocation. Every decrypt should produce correlated evidence in customer KMS logs and SaaS data-plane logs.

For monitoring, I would alert on decrypt spikes, unauthorized KMS principals, key mapping changes without approval, decrypt success after revocation, backup jobs using platform keys for BYOK tenants, and support tools accessing data without BYOK checks. Residual risk remains for data in memory, pre-BYOK backups, KMS outages, and customer self-lockout; those need explicit owners, runbooks, and customer-facing documentation.

14. Likely Follow-Up Questions and Strong Answers

Follow-up 1: What happens to customer data access if the customer revokes the BYOK key?

Decrypt should fail closed immediately for data protected by that key. The tenant may lose read access, which is expected BYOK behavior. The platform must also flush plaintext caches, stop background jobs, block restores, and verify that no internal or support path can still read the data.

Follow-up 2: How do you model a customer using BYOK revocation as ransomware against themselves?

I model it as malicious or compromised customer admin behavior. The controls are customer-side guardrails, alerts to security contacts, deletion-delay guidance, multi-admin approval recommendations, and a recovery runbook. The SaaS provider should not keep a hidden bypass key, because that would undermine the BYOK promise.

Follow-up 3: What prevents plaintext caching after key revocation?

Cache entries must be tagged by tenant and key version, plaintext TTLs must be short, and revocation must trigger targeted purge across every region and worker pool. Completion must be measured with acknowledgments. Any cache hit after revocation is a critical incident.

Follow-up 4: How do you monitor key usage and anomalous decryption volume?

Correlate customer KMS decrypt logs with SaaS decrypt service logs, API route logs, data class, actor, workload identity, region, and baseline volume. Alert on unauthorized principals, encryption context mismatch, decrypt rate spikes, new regions, and decrypts after key disable.

Follow-up 5: What residual risk exists for data in transit, memory, or backups before BYOK?

BYOK primarily controls encryption at rest through customer key material. Data still exists in plaintext in memory during processing, travels over TLS in transit, and may exist in old backups or derived stores from before BYOK enablement. Those risks need compensating controls: short memory lifetime, TLS, backup inventory, re-encryption or expiry, and clear customer disclosures.

15. Common Mistakes to Avoid

  • Claiming BYOK means the SaaS provider can never access plaintext.
  • Keeping plaintext caches alive after customer key revocation.
  • Failing open when KMS is unavailable or key state is unknown.
  • Allowing broad KMS grants without tenant encryption context conditions.
  • Treating tenant-to-key mapping as ordinary config rather than a critical security control.
  • Forgetting backups, search indexes, analytics stores, and replicas.
  • Letting support/admin tools bypass BYOK enforcement.
  • Not reconciling customer KMS logs with SaaS decrypt logs.
  • Ignoring customer self-ransomware and key deletion recovery scenarios.
  • Migrating existing data to BYOK without a legacy data residual-risk plan.

16. Top 5 Immediate Actions

  1. Enforce tenant-scoped KMS grants - require encryption context and continuous policy validation.
  2. Make key mapping tamper-resistant - signed, versioned, approved, and tied to object crypto metadata.
  3. Prove revocation semantics - fail closed, flush caches, block internal bypasses, and drill regularly.
  4. Inventory all data copies - backups, replicas, search indexes, exports, and derived datasets must show key class.
  5. Correlate decrypt telemetry - join KMS logs, SaaS API logs, cache logs, and support-access logs for detection.
