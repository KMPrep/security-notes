Scenario 39 — SaaS DDoS Protection and Rate Limiting

Scenario: Your SaaS API handles 500M requests/day. A competitor DDoS attack or abusive customer could impact availability for all tenants.

Candidate questions:

  • What is the threat model for a volumetric DDoS vs a targeted API abuse attack?
  • How do you design rate limiting that is fair across tenants without punishing legitimate users?
  • What controls detect a low-and-slow API abuse attack below rate limit thresholds?
  • How do you monitor for tenant-specific abuse patterns that aggregate to platform-wide risk?
  • What residual risk do you accept during the mitigation window of a novel attack vector?

3.4 Highest-risk boundary crossing

The highest-risk boundary is between the public internet and the API gateway's origin servers. A volumetric attack that overwhelms edge scrubbing or exploits routing leaks bypasses all application-layer controls and directly exhausts compute capacity shared across all tenants.

The secondary risk is authenticated API abuse: a valid credential means the request passes DDoS filters but can still saturate shared resources. This requires behavioral detection at application layer, where standard L3/L4 protections are blind.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At 500M+ requests/day, all rate limit policies, quota tiers, and DDoS response playbooks must be codified and auto-deployed. Manual rule creation during an active flood is too slow. Use Terraform or similar infra-as-code for WAF and edge ACL rule management with automated merge and deployment pipelines.

8.2 Fleet-wide impact analysis

A misconfigured global rate limit can simultaneously block every tenant on the platform. Before any quota or WAF rule change, model blast radius by simulating against the request distribution of the top 1000 tenants. Use canary rollouts for rule changes, not global instant deploys.

8.3 Eliminating human bottlenecks

Human-in-the-loop approval for DDoS mitigation actions fails at scale. SOC operators should authorize entire runbook classes in advance, not individual block decisions. Automated SOAR playbooks should handle 95 percent of incident actions with humans reviewing outcomes asynchronously.

8.4 Controls that break at scale

Per-engineer WAF rule approval, manual IP blocklist updates, and chat-channel incident coordination collapse during sustained high-severity attacks. Replace with automated rule deployment, shared real-time observability dashboards, and pre-declared runbook authority boundaries.

ANSWER SUMMARY — 90-second version

Use this to open your answer before going deep on any section.

For a SaaS API handling 500M requests/day, the core availability risks are volumetric DDoS overwhelming edge infrastructure and authenticated API abuse that bypasses L3/L4 filters. The key assets are API availability, per-tenant rate-limit fairness, and the edge layer that protects origin capacity.

My architecture layers anycast CDN with DDoS scrubbing at network edge, WAF and bot management in front of the API gateway, and token-bucket per-tenant rate limiting at application layer. Monitoring uses CDN traffic anomalies, per-tenant quota hit rates, and cross-tenant correlated spike detection to catch both volumetric and low-and-slow patterns. Critical alerts auto-trigger SOAR playbooks for blackhole routing, key revocation, or origin scaling.

Residual risk is a genuine mitigation window before defenses activate, and novel attack vectors that outpace signature rules. Success requires automated playbooks, multi-CDN redundancy, and continuous tuning of behavioral baselines to shrink detection and response lag.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What is the difference in threat model between volumetric DDoS and targeted API abuse?

Volumetric DDoS attacks the network and transport layer — the goal is bandwidth exhaustion or connection saturation, and mitigation is primarily at edge and routing layers. Targeted API abuse is application-layer: it uses valid credentials and request structures, making it invisible to L3/L4 filters and requiring behavioral analysis, per-key quotas, and contextual anomaly detection.

Follow-up 2: How do you design rate limiting that is fair across tenants without punishing legitimate users?

Use per-credential token-bucket rate limits tied to commercial plan tiers, with smooth burst allowance above average before hard throttle. Instrument fairness by comparing actual versus expected quota utilization across tenants. Provide customers real-time quota dashboards and an automated raise path for legitimate burst use cases.

Follow-up 3: What controls detect a low-and-slow API abuse attack below individual rate limit thresholds?

Aggregate request volume across multiple dimensions: tenant, source ASN, endpoint, and time window. Use anomaly detection that baselines per-tenant and per-API behavior, and alert on correlated patterns across multiple tenants sharing common source infrastructure. Pair with auth failure rate and session entropy signals.

Follow-up 4: You mentioned SOAR playbooks for mitigation. What if the SOAR platform is also under attack?

SOAR should be isolated on separate infrastructure not in the data-path of the SaaS API. Pre-stage static emergency rule sets at CDN and WAF layer that can be activated with a single low-bandwidth signal, independent of SOAR availability. Document manual fallback procedures and keep CDN vendor escalation contacts current.

Follow-up 5: If you were the attacker, what is the cheapest path through your controls?

Acquire a small number of legitimate API keys via credential stuffing, then distribute abuse across them to stay below per-key thresholds while aggregating to platform-wide impact. This blends into normal traffic and evades volumetric filters. Defenses must therefore emphasize cross-key behavioral correlation, not just per-key quotas.

COMMON MISTAKES TO AVOID

  • Treating volumetric DDoS and application-layer abuse as the same problem with the same controls.
  • Setting static global rate limits without per-tenant fairness controls.
  • Relying only on IP-based blocking against modern botnets using residential proxies.
  • Not modeling the mitigation window as residual risk in SLA commitments.
  • Forgetting authenticated abuse paths that bypass network-layer filtering.
  • Manual WAF rule management during active incidents at production scale.
  • Missing cross-tenant correlation in detection — looking only at per-tenant signals.
  • No multi-CDN failover plan for attacks that saturate primary scrubbing capacity.
