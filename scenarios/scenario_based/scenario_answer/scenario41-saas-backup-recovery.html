Scenario 41 — SaaS Data Backup and Recovery Security

Scenario: Your SaaS takes nightly backups of all tenant data to a separate cloud region. Backups are retained for 30 days and can be restored on customer request.

Candidate questions:

  • What is the threat model for a ransomware attack that targets your backup infrastructure?
  • How do you ensure backups are encrypted separately from production data encryption keys?
  • What controls prevent a compromised production service account from deleting backups?
  • How do you verify backup integrity and test restores without accessing tenant data?
  • What residual risk remains for data created between the last backup and an incident?

3.4 Highest-risk boundary crossing

The highest-risk boundary crossing is from production service account credentials into the secondary-region backup storage. This crossing combines broad data access with destruction capability. An attacker who pivots from a compromised production identity to backup delete permissions can neutralize all recovery options simultaneously — the worst-case scenario for any ransomware event.

The secondary risk is the restore workflow: the same boundary that enables legitimate recovery also enables unauthorized data exfiltration or overwrite if restore access is not tightly scoped and audited.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale, backup policies, retention locks, cross-region replication configs, and integrity validation pipelines must be codified in infrastructure-as-code with automated compliance checks. Manual configuration of bucket policies or retention settings does not scale and creates drift risk across thousands of tenant namespaces.

8.2 Fleet-wide impact analysis

A ransomware attack targeting backup infrastructure with broad service account access could destroy the recovery option for every tenant simultaneously. Blast radius must be modeled by tenant count, total data volume, and recovery time from the last immutable copy. This drives the requirement for cross-account isolation — production account compromise should never reach backup account delete permissions.

8.3 Eliminating human bottlenecks

Manual restore approval, per-ticket integrity verification, and on-call engineer-driven ransomware containment fail at the time pressure and scale of a major incident. Pre-authorize backup accounts with automated WORM enforcement, automated credential revocation playbooks, and self-service restore with automated validation so that human decisions are only needed for scope and notification.

8.4 Controls that break at scale

Manual backup monitoring dashboards, per-tenant restore approval chains, and informal integrity spot-checks fail above a few hundred tenants. Replace with streaming completeness metrics, automated hash verification for every backup object, and policy-enforced WORM applied uniformly via IAM org policy rather than per-bucket configuration.

ANSWER SUMMARY — 90-second version

Use this to open your answer before going deep on any section.

For SaaS backup security, the critical threat is a ransomware operator who compromises a production identity and pivots to the backup layer to destroy all recovery options before extortion. The key assets are the backup data itself, the separate backup key hierarchy, the immutability controls on storage, and backup service account credentials.

My design isolates backup storage in a separate cloud account, enforces WORM object lock for the full retention period, uses a completely separate backup KMS key hierarchy, and grants production service accounts write-only access with no delete permissions. Every backup is hash-verified before acceptance. Monitoring watches for delete attempts on backup objects, backup job failures, integrity mismatches, and unexpected backup key usage.

Residual risk includes the 24-hour RPO gap for nightly backups and the edge case where both production and backup are compromised together. These are accepted with documented SLAs, cold storage air-gap copies, and forensic preservation of the last known-good backup at incident time.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What is the threat model for ransomware targeting backup infrastructure?

Ransomware attackers increasingly target backups first because destroying recovery options maximizes extortion leverage. The attack pattern is: credential theft or lateral movement from production environment, pivot to backup storage account, then bulk delete or encrypt backup objects before encrypting production data. Defense requires making backup deletion technically impossible from any production identity, not just policy-restricted.

Follow-up 2: How do you ensure backups are encrypted separately from production?

Maintain a completely separate KMS key hierarchy in a separate account, with no IAM path from production service accounts to backup key usage. Backup jobs authenticate to the backup account with dedicated short-lived credentials that cannot access production keys. This means even if production keys are compromised, backup data is protected by an independent cryptographic boundary.

Follow-up 3: What controls prevent a compromised production service account from deleting backups?

Cross-account isolation is the primary control: the backup storage bucket lives in a separate cloud account with its own IAM boundary. Production service accounts are granted only write-only access via cross-account roles, with no delete or overwrite permissions. WORM object lock provides a second enforcement layer that rejects deletes even if IAM is somehow misconfigured.

Follow-up 4: How do you verify backup integrity without accessing tenant data?

Use cryptographic hash verification at the object level. The backup job computes a hash for each encrypted object before writing, and an independent validation job verifies hashes against the manifest on a schedule. The validation service never needs to decrypt — it only compares ciphertext hashes. Integrity is proven without any access to plaintext tenant data.

Follow-up 5: If you were the attacker, what is the cheapest path through your controls?

I would look for a production service account that has cross-account permissions to the backup bucket, then silently corrupt backup objects by overwriting them with encrypted garbage over several days — below the per-object count alert threshold. The restore failure would only be discovered when recovery is needed. This is why WORM must prevent overwrites as well as deletes, and why integrity validation must run on every object, not just sampled.

COMMON MISTAKES TO AVOID

  • Storing backups in the same cloud account or under the same IAM boundary as production.
  • Using the same encryption keys for production data and backup data.
  • Relying on IAM policy alone without WORM object lock for backup protection.
  • Not running automated integrity verification on every backup object.
  • Forgetting the RPO gap as a residual risk requiring explicit SLA documentation.
  • Allowing restore operations without authorization workflow and audit trail.
  • Treating cross-region replication as equivalent to true cross-account isolation.
  • Ignoring the scenario where backups themselves contain already-compromised data.
