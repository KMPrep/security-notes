Scenario 44 - SaaS AI Feature with LLM Integration - Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario44-saas-llm-ai-feature.md
Prompt source: ../threat-modeling-prompt.md
Generated artifact: scenario44-saas-llm-ai-feature-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead - 15+ years of purple-team experience
Scenario: A SaaS product adds an AI assistant powered by a third-party LLM API such as OpenAI or Vertex AI. The assistant can retrieve tenant data to answer questions about the customer's own data. The platform must prevent prompt injection, cross-tenant disclosure, unsafe tool calls, data residency violations, provider retention exposure, and AI-assisted exfiltration.

1. Review Objective

This review converts Scenario 44 into attacker-centric exploitation chains and blue-team control guidance. It focuses on what a strong Google onsite answer should surface: trust boundaries, threat paths, control layering, telemetry, detection logic, residual risk, and how to validate assumptions safely through staging-only purple-team exercises.

2. Highest-Risk Design Decision

The highest-risk design decision is allowing the LLM to combine retrieved tenant data with broad tool-call privileges while relying on the system prompt to enforce security boundaries.

If the assistant can retrieve data and call internal APIs, prompt injection becomes more than text manipulation. A malicious user or malicious document can steer the model into querying, summarizing, exporting, or transforming sensitive data. The safe design is to make the LLM an untrusted reasoning component behind a policy-enforcing AI gateway: tenant isolation, authorization, tool scope, output filtering, and data residency must be enforced by deterministic services outside the model.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker action or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision point.
  • Green nodes (:::blocked): attack stopped or contained.
  • Yellow nodes (:::impact): business, privacy, legal, or platform impact.

6. Reference Architecture

flowchart LR
    User[Tenant user] --> UI[SaaS UI]
    UI --> Gateway[AI gateway]
    Gateway --> AuthZ[Policy and authorization service]
    Gateway --> PromptBuilder[Prompt assembly service]
    PromptBuilder --> DLP1[Pre-send DLP and minimization]
    PromptBuilder --> Retrieval[RAG retrieval service]
    Retrieval --> Vector[Per-tenant vector namespace]
    Retrieval --> Data[Tenant data stores]
    DLP1 --> Provider[Third-party LLM API]
    Provider --> Gateway
    Gateway --> ToolBroker[Tool broker]
    ToolBroker --> AuthZ
    ToolBroker --> APIs[Internal SaaS APIs]
    Gateway --> DLP2[Output DLP and safety validation]
    DLP2 --> UI
    Gateway --> Logs[Security telemetry stream]

Security properties:

  • The model is not the policy enforcement point.
  • Retrieval is tenant-scoped before data enters the prompt.
  • Tool calls are mediated by a broker that rechecks user, tenant, resource, action, and purpose.
  • Sensitive-data minimization runs before provider submission.
  • Output DLP and response validation run after model generation.
  • Telemetry is captured for query, retrieval, prompt assembly, provider route, tool call, and response events.

7. Threat-by-Threat Technical Review

T1 - Direct prompt injection against chat UI

STRIDE category: Tampering / Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Override system instructions, extract hidden context, reveal system prompts, or cause the assistant to return sensitive data.

How the attacker would exploit it

  • Prerequisite: valid access to the AI assistant as a tenant user.
  • Step 1: Submit prompts that instruct the model to ignore prior instructions, reveal hidden context, or disclose system policy.
  • Step 2: Ask the assistant to summarize all available context, enumerate retrieved documents, or output raw snippets.
  • Step 3: Iterate using encoding, role-play, translation, or chain-of-thought bait to bypass simple filters.
  • Step 4: Exfiltrate sensitive context through normal chat responses.

Blue-team control review - what must stop the chain

  • Preventive: Treat user input as untrusted data and wrap it in strict prompt templates with clear delimiters.
  • Preventive: Apply prompt-injection classifiers before prompt assembly.
  • Preventive: Never place secrets, credentials, API keys, or sensitive internal policy in the system prompt.
  • Detective: Log injection classifier scores, blocked patterns, and repeated attempts by user and tenant.
  • Corrective: Rate-limit or suspend sessions with repeated injection attempts and create SOC tickets for high-risk tenants.

Evidence and telemetry to validate controls

  • AIGatewayQueryLog: tenant_id, user_id, query_id, query_classification, risk_score.
  • PromptPolicyLog: policy_id, decision, matched_rule, redaction_action.
  • ResponseDLPLog: response_id, pii_count, secret_pattern_count, blocked.
  • SessionRiskLog: session_id, injection_attempt_count, action_taken.

Purple-team validation commands

# Submit a controlled prompt-injection test in staging and verify it is blocked or safely answered.

| where attempts > 3

T2 - Indirect prompt injection through RAG documents

STRIDE category: Tampering / Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Plant malicious instructions in tenant content so the RAG system retrieves them later and the model treats them as commands.

How the attacker would exploit it

  • Prerequisite: ability to upload, edit, or influence documents indexed by the assistant.
  • Step 1: Add hidden or visible instructions to a document, ticket, wiki page, email, or uploaded file.
  • Step 2: Wait for indexing into the vector store.
  • Step 3: Ask a query likely to retrieve the malicious document.
  • Step 4: The retrieved text instructs the model to call tools, disclose context, ignore policies, or return sensitive data.
  • Step 5: Exfiltrate output through the assistant or downstream tool action.

Blue-team control review - what must stop the chain

  • Preventive: Scan indexed documents for injection-like instructions and mark high-risk chunks.
  • Preventive: Treat retrieved content as untrusted context, never as instruction.
  • Preventive: Use prompt structure that separates system instruction, user question, and retrieved context.
  • Preventive: Tool broker must require deterministic authorization independent of model instruction.
  • Detective: Correlate document chunks with tool-call sequences and suspicious responses.
  • Corrective: Quarantine injected documents, remove embeddings, and rebuild affected tenant indexes.

Evidence and telemetry to validate controls

  • DocumentRiskScan: doc_id, tenant_id, risk_type, risk_score, chunk_id.
  • RetrievalAudit: query_id, tenant_id, doc_id, chunk_id, retrieval_score.
  • ToolCallAudit: query_id, tool_name, resource_id, decision, reason.
  • IndexLifecycleLog: doc_id, indexed_at, quarantined_at, embedding_version.

Purple-team validation commands

T3 - Cross-tenant retrieval from vector store

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Cause the RAG retrieval layer to return another tenant's documents through namespace, filter, cache, or index-isolation failure.

How the attacker would exploit it

  • Prerequisite: shared vector store or shared retrieval service with application-level tenant filters.
  • Step 1: Craft queries with ambiguous identifiers, tenant names, document IDs, or embeddings that collide semantically.
  • Step 2: Trigger a retrieval path that omits or misapplies tenant_id.
  • Step 3: Receive snippets or summaries from another tenant's data.
  • Step 4: Repeat across query variants to expand leakage.

Blue-team control review - what must stop the chain

  • Preventive: Enforce tenant isolation at storage/index layer, not only in application code.
  • Preventive: Use per-tenant namespaces, per-tenant encryption keys, and authorization-bound retrieval tokens.
  • Preventive: Deny any retrieval result whose doc_tenant_id does not match request_tenant_id.
  • Detective: Alert on any namespace mismatch, cache-key mismatch, or tenant filter omission.
  • Corrective: Disable AI retrieval for affected tenants and rebuild indexes from clean manifests.

Evidence and telemetry to validate controls

  • RetrievalAudit: request_tenant_id, doc_tenant_id, namespace, doc_id, query_id.
  • VectorStoreAuthLog: principal, allowed_namespace, requested_namespace, decision.
  • TenantIsolationTest: test_id, control_version, result, failed_case.
  • CacheAccessLog: cache_key, tenant_id, hit, owner_tenant_id.

Purple-team validation commands

T4 - Tool-call authorization bypass

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Make the assistant call internal tools or APIs with broader privileges than the user has.

How the attacker would exploit it

  • Prerequisite: assistant supports tools for search, export, ticket creation, user lookup, or admin actions.
  • Step 1: Inject instructions that ask the model to call a privileged tool.
  • Step 2: Exploit weak tool schemas, missing resource-level authorization, or confused-deputy behavior.
  • Step 3: Receive tool results through chat output or cause state-changing actions.
  • Step 4: Chain allowed tools into a forbidden outcome, such as bulk export.

Blue-team control review - what must stop the chain

  • Preventive: Tool broker must reauthorize each tool call using user, tenant, resource, action, and purpose.
  • Preventive: Separate read-only tools from state-changing tools and require explicit user confirmation for risky actions.
  • Preventive: Use narrow tool schemas that do not accept raw SQL, arbitrary URLs, or unsafely broad filters.
  • Detective: Alert on denied tool calls, unusual tool sequences, and high-volume result sets.
  • Corrective: Disable specific tools per tenant or globally through a kill switch.

Evidence and telemetry to validate controls

  • ToolCallAudit: tool_name, requested_action, tenant_id, user_id, authz_decision.
  • InternalAPIAccessLog: caller_service, resource_tenant_id, result_count, status.
  • RiskyActionConfirmationLog: action_id, confirmed_by, expires_at.
  • AIToolKillSwitchLog: tool_name, scope, disabled_by, reason.

T5 - Excessive sensitive data sent to third-party LLM

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Cause the platform to send unnecessary PII, secrets, financial data, or customer IP to the LLM provider.

How the attacker would exploit it

  • Prerequisite: prompt assembly includes broad context chunks or raw records.
  • Step 1: Ask broad questions that trigger large context retrieval.
  • Step 2: Force inclusion of raw tables, exports, or full documents.
  • Step 3: Sensitive data leaves the SaaS boundary and becomes subject to provider retention, logging, support access, or breach risk.

Blue-team control review - what must stop the chain

  • Preventive: Minimize context before provider submission; use smallest necessary chunks and field allowlists.
  • Preventive: Run DLP and secret scanning on assembled prompts before egress.
  • Preventive: Prefer redaction, tokenization, or customer-managed secure context references where feasible.
  • Detective: Log PII density, secret patterns, prompt size, and data categories sent per tenant.
  • Corrective: Block provider submission when sensitivity exceeds policy and return a safe refusal or ask for narrower scope.

Evidence and telemetry to validate controls

  • PromptAssemblyLog: query_id, context_doc_count, prompt_tokens, data_categories.
  • DLPPreSendLog: pii_count, secret_count, policy_action, redacted_fields.
  • ProviderRequestLog: provider, model, region, prompt_hash, payload_classification.
  • TenantAISettingsLog: tenant_id, allowed_data_classes, provider_opt_in.

T6 - Data residency or retention violation at provider

STRIDE category: Information Disclosure / Compliance
Risk: Medium likelihood, High impact
Attacker objective: Exploit weak routing or provider configuration so regulated tenant data is processed or retained outside the approved region or retention window.

How the attacker would exploit it

  • Prerequisite: provider endpoint selection is application-controlled or fallback routing is allowed.
  • Step 1: Trigger failover, model fallback, or region fallback during provider degradation.
  • Step 2: Prompt data is sent to a region outside tenant policy.
  • Step 3: Provider logs, support traces, or abuse-monitoring retention keep regulated data beyond contract.
  • Step 4: Customer or regulator discovers violation after the fact.

Blue-team control review - what must stop the chain

  • Preventive: Enforce egress routing with network policy, not only application flags.
  • Preventive: Map every tenant to approved provider, model, and region policy.
  • Preventive: Disable cross-region fallback for regulated tenants unless explicitly approved.
  • Detective: Log provider, endpoint, region, model, retention tier, and tenant residency policy on every request.
  • Corrective: Auto-disable AI feature for tenants when compliant provider route is unavailable.

Evidence and telemetry to validate controls

  • ProviderRouteLog: tenant_id, provider, model, region, route_decision.
  • EgressPolicyLog: destination, region, allowed, policy_id.
  • TenantResidencyPolicy: tenant_id, allowed_regions, allowed_providers, fallback_policy.
  • ProviderRetentionAudit: provider, model, retention_days, training_exclusion.

T7 - Conversation history or prompt log exposure

STRIDE category: Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Access stored prompts, responses, or conversation history containing sensitive tenant data.

How the attacker would exploit it

  • Prerequisite: prompt logs or conversation history are stored in a searchable index.
  • Step 1: Compromise a support user, engineer, or log-search role.
  • Step 2: Query conversation logs for secrets, PII, financial data, or sensitive customer intent.
  • Step 3: Export logs or screenshots and exfiltrate them.

Blue-team control review - what must stop the chain

  • Preventive: Do not store raw prompts and responses by default; store hashes or redacted text for debugging.
  • Preventive: Apply tenant-scoped encryption keys and strict access broker for conversation access.
  • Preventive: Enforce least privilege and just-in-time access for support/debug workflows.
  • Detective: Alert on broad log queries, sensitive term searches, and support access outside ticket context.
  • Corrective: Revoke access, rotate impacted secrets, and purge contaminated log partitions when possible.

Evidence and telemetry to validate controls

  • ConversationAccessLog: actor, tenant_id, conversation_id, ticket_id, access_reason.
  • LogIndexAccess: actor, query_text_hash, result_count, export_action.
  • DLPLogHygieneScan: index, sensitive_match_count, remediation_status.
  • JITAccessLog: actor, approved_by, expires_at, scope.

T8 - LLM provider API key compromise

STRIDE category: Elevation of Privilege / Information Disclosure
Risk: Medium likelihood, High impact
Attacker objective: Steal provider API keys or service credentials to call the provider directly, impersonate the SaaS platform, incur cost, or access provider-side logs.

How the attacker would exploit it

  • Prerequisite: API key stored in environment variables, CI/CD secrets, logs, or developer machines.
  • Step 1: Harvest key from logs, repo, container image, crash dump, or CI output.
  • Step 2: Use key to send arbitrary prompts or enumerate provider resources where supported.
  • Step 3: Burn quota, bypass tenant controls, or access provider logs/configuration.

Blue-team control review - what must stop the chain

  • Preventive: Store provider credentials in managed secrets service with workload identity and no static long-lived keys where possible.
  • Preventive: Restrict provider key by egress source, project, model, region, and quota.
  • Preventive: Scan repos, images, logs, and CI artifacts for leaked secrets.
  • Detective: Alert on provider usage from unexpected source, model, region, or token volume.
  • Corrective: Rotate key automatically and invalidate all affected sessions or provider connections.

Evidence and telemetry to validate controls

  • SecretScanner: location, secret_type, confidence, blocked.
  • LLMProviderUsageLog: api_key_id, source_ip, model, tokens, cost.
  • SecretsAccessLog: principal, secret_id, access_reason, workload_id.
  • KeyRotationLog: key_id, rotated_at, reason, old_key_disabled_at.

T9 - AI-assisted insider exfiltration

STRIDE category: Information Disclosure
Risk: High likelihood, High impact
Attacker objective: Use legitimate assistant access to summarize, transform, compress, or stage sensitive data for exfiltration.

How the attacker would exploit it

  • Prerequisite: insider has legitimate access to sensitive tenant data.
  • Step 1: Ask the assistant to aggregate data that would otherwise require many manual steps.
  • Step 2: Request summaries, CSV-like output, encoded chunks, or "executive digests" that hide sensitive detail density.
  • Step 3: Copy output, download conversation history, or route results to another tool.
  • Step 4: Repeat below per-query thresholds.

Blue-team control review - what must stop the chain

  • Preventive: Enforce same data-access policies as the underlying app; the assistant must not expand user privilege.
  • Preventive: Apply purpose-based and volume-based controls for sensitive aggregation.
  • Detective: UEBA for unusual query patterns, repeated broad summaries, and high sensitive-output density.
  • Detective: Correlate assistant usage with DLP endpoint, browser, email, and cloud-upload telemetry.
  • Corrective: Step-up auth, manager approval, or session suspension for high-risk aggregation.

Evidence and telemetry to validate controls

10. Minimum Viable Control Set

If budget or timeline is constrained, the minimum viable control set should include:

  1. AI gateway that enforces prompt policy, pre-send DLP, output DLP, and per-tenant quotas.
  2. Storage-layer tenant isolation for vector indexes and retrieval, with regression tests.
  3. Tool broker with deterministic authorization for every model-proposed tool call.
  4. Provider egress policy enforcing approved region, model, and retention settings.
  5. Telemetry correlation across query, retrieval, prompt assembly, provider request, tool call, and response.

Do not ship the feature if the model can call tools directly, if tenant isolation only exists in prompt text, or if raw prompts and responses are stored broadly accessible to support and engineering.

12. Google-Scale Considerations

At Google scale, the AI assistant must be governed as critical infrastructure rather than a feature-level integration.

  • Policy-as-code: Tenant isolation, tool permissions, provider routing, data class allowlists, and response limits should be declarative and enforced by shared infrastructure.
  • Continuous isolation testing: Every release of retrieval, vector index, prompt assembly, or caching logic should run synthetic cross-tenant leak tests.
  • Centralized AI gateway: Teams should not individually call providers from product services. A shared gateway enables consistent DLP, retention, routing, logging, and quota enforcement.
  • Blast-radius controls: Kill switches must exist per tenant, per model, per tool, per provider, and global feature.
  • Privacy-preserving logging: Prompt and response logs should be redacted by default. Access to raw traces should be rare, time-bound, and ticket-linked.
  • Fleet telemetry: Detection should correlate query intent, retrieved documents, tool calls, provider route, output sensitivity, and downstream copy/export events.

13. Model Answer Summary - 90-second version

For an LLM assistant with access to tenant data, I would treat the model as an untrusted component and enforce security outside the model. The main risks are direct prompt injection, indirect prompt injection through RAG content, cross-tenant retrieval bugs, unsafe tool calls, excessive sensitive data sent to the provider, and AI-assisted insider exfiltration.

The highest-risk design is letting the model retrieve data and call tools broadly while relying on the system prompt for enforcement. My design would put an AI gateway in front of the provider, enforce storage-layer tenant isolation in the vector store, run DLP and data minimization before provider submission, authorize every tool call through a deterministic tool broker, validate responses with output DLP, and log query, retrieval, tool, provider, and response events for detection.

Residual risk remains around novel indirect prompt injection, hallucinations, and provider-side retention, so I would document those risks with owners, test them through purple-team drills, and make high-risk use cases fail closed rather than degrade silently.

14. Likely Follow-Up Questions and Strong Answers

Follow-up 1: What is the prompt injection threat model for an LLM with access to tenant data?

A strong answer separates direct injection from indirect injection. Direct injection comes from the user's prompt. Indirect injection comes from retrieved content such as documents, tickets, emails, or wiki pages. The indirect path is usually more dangerous because the attacker can plant instructions in data that later becomes context for a different user's query. If tools are enabled, injection can become action execution, so tool authorization must be enforced outside the model.

Follow-up 2: How do you prevent a tenant user from accessing another tenant's data?

Tenant isolation must be enforced in the retrieval layer and internal APIs, not by asking the model to behave. The vector store should use per-tenant namespaces or equivalent storage-level isolation, and every retrieved result should carry a doc_tenant_id checked against the request tenant before prompt assembly. Tool calls must also recheck tenant, user, resource, and action authorization.

Follow-up 3: What controls govern data sent to the third-party LLM provider?

Use data minimization, pre-send DLP, tenant-specific data class policy, provider allowlists, and network-level egress controls. The provider contract should prohibit training on prompts and define retention, breach notification, and audit rights, but contractual controls are not enough. The technical control is to avoid sending unnecessary sensitive data in the first place.

Follow-up 4: How do you monitor for AI-assisted data exfiltration?

Correlate query intent, retrieval volume, response sensitivity, tool calls, and downstream copy/export behavior. Signals include repeated broad-summary queries, high response-token volume, many sensitive entities in output, denied tool calls, cross-tenant retrieval mismatches, encoded/chunked responses, and user behavior that deviates from peer group baselines.

Follow-up 5: What residual risk do you accept for hallucinations?

I would not accept hallucinations for high-stakes security or compliance actions without guardrails. For general assistant usage, I would accept residual hallucination risk only with clear citations, approved retrieval sources, confidence thresholds, user feedback loops, and escalation to human review for policy-changing or incident-response guidance.

15. Common Mistakes to Avoid

  • Treating prompt injection as only a user-input problem and ignoring RAG-based indirect injection.
  • Relying on system prompts for tenant isolation or authorization.
  • Giving the LLM broad tools that call internal APIs without a separate authorization broker.
  • Sending entire raw records to the provider when a small, redacted context chunk would be enough.
  • Storing raw prompts and responses in broadly searchable logs.
  • Assuming provider no-training clauses eliminate retention or breach risk.
  • Forgetting data residency when provider failover or model fallback occurs.
  • Missing AI-assisted insider exfiltration because each individual query looks legitimate.
  • Treating hallucinations as a UX issue only, even when the assistant gives security or compliance guidance.
  • Shipping without kill switches for tools, tenants, providers, models, and the global AI feature.

16. Top 5 Immediate Actions

  1. Put all LLM traffic behind an AI gateway - centralize prompt policy, DLP, provider routing, quota, logging, and kill switches.
  2. Enforce storage-layer tenant isolation for RAG - per-tenant namespaces, strict mismatch checks, and continuous isolation tests.
  3. Route every tool call through an authorization broker - the model proposes actions; deterministic services approve or deny them.
  4. Minimize and classify data before provider submission - redact PII/secrets and block sensitive classes that exceed tenant policy.
  5. Build correlation detections across query, retrieval, tool, provider, and response logs - prompt injection and exfiltration only become obvious when these signals are joined.
