Scenario 47 - Kubernetes Cluster Security Hardening - Threat Exploitation and Blue-Team Control Review

Source file reviewed: scenario47-kubernetes-hardening.md
Prompt source: ../threat-modeling-prompt.md
Generated artifact: scenario47-kubernetes-hardening-Attack-Review-Enhanced.md
Reviewer role: Cybersecurity Architect / Engineer Lead - 15+ years of purple-team experience
Scenario: A production Kubernetes cluster runs 200 microservices across dev, staging, and prod namespaces on the same cluster. Pod-to-pod communication is unrestricted. The platform must prevent a compromised workload from becoming cluster-wide compromise, enforce workload identity and mTLS, stop container-to-host privilege escalation, detect lateral movement, and manage residual risk from third-party Helm charts.

1. Review Objective

This review turns the Kubernetes hardening scenario into attacker-centric exploitation chains and blue-team controls. It is designed for study: each threat includes attacker path, defensive breakpoints, concrete telemetry, safe validation checks, and the sort of residual-risk reasoning expected in a senior security engineering onsite.

2. Highest-Risk Design Decision

The highest-risk design decision is running dev, staging, and prod workloads in one shared cluster with unrestricted pod-to-pod networking.

Namespaces are not a hard security boundary by default. If a dev pod is compromised and the cluster has no default-deny NetworkPolicy, no strict workload identity, and overbroad service account RBAC, the attacker can pivot inside the pod network, steal service account tokens, query the Kubernetes API, read secrets, and reach production services without crossing the external perimeter. The safer architecture is separate clusters or at least separate node pools and strict policy boundaries for prod, backed by default-deny network policy, Pod Security Admission restricted profile, workload identity, signed images, and admission policy-as-code.

3. Attack-Tree Legend

  • Red nodes (:::attack): attacker action or attacker-controlled state.
  • Blue nodes (:::control): defensive control decision point.
  • Green nodes (:::blocked): attack stopped or contained.
  • Yellow nodes (:::impact): business, privacy, availability, or platform impact.

6. Reference Architecture

flowchart LR
    Dev[Dev namespace] --> DevNP[Default-deny NetworkPolicy]
    Stage[Staging namespace] --> StageNP[Default-deny NetworkPolicy]
    Prod[Prod namespace] --> ProdNP[Default-deny NetworkPolicy]
    DevNP --> Mesh[Service mesh mTLS]
    StageNP --> Mesh
    ProdNP --> Mesh
    Mesh --> API[Kubernetes API server]
    API --> RBAC[RBAC and admission policy]
    API --> Etcd[etcd with KMS encryption]
    Pods[Workloads] --> WI[Workload identity]
    Pods --> Runtime[Runtime sandbox and Pod Security Admission]
    Runtime --> Nodes[Node pools]
    API --> Audit[K8s audit logs]
    Mesh --> Flows[Service and network telemetry]
    Runtime --> Falco[Falco or runtime events]
    Audit --> SIEM[SIEM and SOAR]
    Flows --> SIEM
    Falco --> SIEM

Security properties:

  • Prod should be isolated by cluster or, at minimum, by node pool, namespace, policy, and identity.
  • NetworkPolicy starts with default-deny ingress and egress.
  • Service-to-service calls use workload identity and strict mTLS.
  • Admission policy rejects privileged containers, host mounts, unsigned images, and dangerous capabilities.
  • RBAC is namespace-scoped and generated from workload needs.
  • Kubernetes audit, runtime, and network telemetry are correlated.

7. Threat-by-Threat Technical Review

T1 - Compromised pod lateral movement across namespaces

STRIDE category: Elevation of Privilege / Information Disclosure
Risk: High likelihood, Critical impact
Attacker objective: Use one compromised workload to discover and reach production workloads over the flat pod network.

How the attacker would exploit it

  • Prerequisite: RCE, SSRF, deserialization bug, exposed debug endpoint, or stolen app credential in any dev or staging pod.
  • Step 1: Enumerate environment variables, mounted tokens, DNS names, and cluster service discovery.
  • Step 2: Scan internal service DNS and pod CIDRs for reachable prod services.
  • Step 3: Connect directly to production services or databases that trust cluster-internal traffic.
  • Step 4: Exfiltrate data or pivot further using service credentials.

Blue-team control review - what must stop the chain

  • Preventive: Use separate prod clusters where feasible; otherwise separate prod node pools and strict namespace isolation.
  • Preventive: Apply default-deny ingress and egress NetworkPolicy to every namespace.
  • Preventive: Allow only declared service dependencies through policy-as-code generated from service ownership metadata.
  • Detective: Detect first-seen namespace-to-namespace flows and scans through Cilium Hubble, Calico flow logs, or service mesh telemetry.
  • Corrective: Quarantine source namespace, kill affected pods, and rotate credentials reachable by the compromised workload.

Evidence and telemetry to validate controls

  • NetworkFlowLog: src_namespace, src_pod, dst_namespace, dst_service, dst_port, bytes.
  • ServiceMeshTelemetry: source_workload, destination_workload, mtls, response_code.
  • K8sAuditLog: user.username, sourceIPs, verb, resource, namespace.
  • SOARActionLog: quarantine_namespace, pod_deleted, network_policy_applied.

Purple-team validation commands

# In staging, verify a dev pod cannot reach a prod-only service.
kubectl -n dev exec deploy/synthetic-client -- curl -sS --connect-timeout 3 http://prod-api.prod.svc.cluster.local/health

NetworkFlowLog
| where src_namespace != dst_namespace
| where not(has_declared_dependency)
| summarize flows=count(), bytes=sum(bytes) by src_namespace, src_pod, dst_namespace, dst_service, bin(TimeGenerated, 5m)

T2 - Service account token theft and overbroad RBAC

STRIDE category: Elevation of Privilege
Risk: High likelihood, Critical impact
Attacker objective: Steal a mounted service account token and use it to query the Kubernetes API for secrets, workloads, or RBAC changes.

How the attacker would exploit it

  • Prerequisite: compromised pod with automounted service account token.
  • Step 1: Read /var/run/secrets/kubernetes.io/serviceaccount/token.
  • Step 2: Call Kubernetes API server from inside the cluster.
  • Step 3: Test verbs like get, list, watch, create, and patch across namespaces.
  • Step 4: Read secrets, create pods, bind roles, or deploy a persistence workload.

Blue-team control review - what must stop the chain

  • Preventive: Disable automountServiceAccountToken unless the workload needs Kubernetes API access.
  • Preventive: Use short-lived projected service account tokens with audience restrictions.
  • Preventive: Enforce namespace-scoped least privilege; no default service account should have meaningful permissions.
  • Detective: Alert on service accounts reading secrets outside their namespace or using unusual verbs.
  • Corrective: Revoke token, delete service account, rotate impacted secrets, and redeploy workload with reduced RBAC.

Evidence and telemetry to validate controls

  • K8sAuditLog: user.username, verb, resource, namespace, responseStatus.code.
  • RBACInventory: service_account, role, verbs, resources, scope.
  • TokenRequestLog: audience, expirationSeconds, boundObjectRef.
  • SecretAccessAudit: actor, secret_name, secret_namespace, access_reason.

Purple-team validation commands

# In staging, verify default service accounts cannot list secrets.
kubectl auth can-i list secrets --as=system:serviceaccount:dev:default -n prod

K8sAuditLog
| where user_username startswith "system:serviceaccount:"
| where resource == "secrets" and verb in ("get", "list", "watch")
| where actor_namespace != namespace
| project TimeGenerated, user_username, verb, namespace, response_code

T3 - Privileged container or hostPath escape to node

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, Critical impact
Attacker objective: Escape the container boundary, gain host-level access, and reach all pods or node credentials.

How the attacker would exploit it

  • Prerequisite: workload runs privileged, has hostPath mounts, dangerous Linux capabilities, hostPID, hostNetwork, or vulnerable runtime.
  • Step 1: Inspect mounted host filesystems and container runtime sockets.
  • Step 2: Abuse hostPath, Docker/containerd socket, or kernel exploit to execute on the node.
  • Step 3: Read kubelet credentials, pod volumes, and node-level cloud credentials.
  • Step 4: Pivot to other pods on the node or cloud control plane.

Blue-team control review - what must stop the chain

  • Preventive: Enforce Pod Security Admission restricted profile for prod namespaces.
  • Preventive: Add OPA/Gatekeeper or Kyverno policies blocking privileged pods, hostPath, hostNetwork, hostPID, and dangerous capabilities.
  • Preventive: Use runtime sandboxing such as gVisor or Kata Containers for untrusted/high-risk workloads.
  • Detective: Use Falco or eBPF-based runtime detection for host file access, shell spawn, or container escape indicators.
  • Corrective: Cordon and drain node, preserve forensic disk, rotate node and workload credentials.

Evidence and telemetry to validate controls

  • AdmissionAudit: policy, resource, namespace, decision, violations.
  • FalcoRuntimeEvent: rule, pod, namespace, container, proc.name, file.path.
  • NodeAuditLog: host_pid, process, syscall, parent_process.
  • CordonDrainLog: node, reason, initiated_by, completed_at.

Purple-team validation commands

# In staging, attempt to apply a privileged pod manifest and verify admission rejects it.
kubectl apply -f tests/privileged-pod-deny.yaml --namespace prod

FalcoRuntimeEvent
| where rule in ("Terminal shell in container", "Write below root", "Launch privileged container", "Contact K8S API Server From Container")
| project TimeGenerated, namespace, pod, container, rule, proc_name

T4 - Kubernetes API server exposure or control-plane abuse

STRIDE category: Elevation of Privilege / Tampering
Risk: Medium likelihood, Critical impact
Attacker objective: Abuse exposed or weakly controlled API server access to control workloads and cluster state.

How the attacker would exploit it

  • Prerequisite: API server reachable from broad networks, weak auth, stolen kubeconfig, or excessive admin role.
  • Step 1: Discover API endpoint and test authentication paths.
  • Step 2: Use stolen kubeconfig or service account token to enumerate resources.
  • Step 3: Create privileged pods, modify deployments, or create ClusterRoleBindings.
  • Step 4: Persist through admission webhooks, daemonsets, or secrets.

Blue-team control review - what must stop the chain

  • Preventive: Restrict API server access to trusted networks or private endpoints.
  • Preventive: Require strong identity, MFA/JIT for human cluster admin, and certificate rotation.
  • Preventive: Enforce least-privilege RBAC and deny risky verbs for non-platform roles.
  • Detective: Alert on cluster role creation, webhook changes, privileged workload creation, and access from unusual IPs.
  • Corrective: Revoke kubeconfigs/certs, remove malicious roles/webhooks, and rebuild control-plane trust if needed.

Evidence and telemetry to validate controls

  • ControlPlaneAccessLog: source_ip, principal, auth_method, decision.
  • K8sAuditLog: verb, resource, objectRef.name, userAgent, sourceIPs.
  • RBACChangeLog: role, binding, subject, changed_by.
  • WebhookConfigAudit: webhook_name, failurePolicy, clientConfig, changed_by.

Purple-team validation commands

# In staging, verify non-admin identities cannot create ClusterRoleBindings.

| where verb in ("create", "patch", "update", "delete")
| project TimeGenerated, user_username, verb, resource, object_name, sourceIPs

T5 - etcd or Kubernetes secret exfiltration

STRIDE category: Information Disclosure
Risk: Medium likelihood, Critical impact
Attacker objective: Read secrets directly from Kubernetes API or etcd and reuse credentials outside the cluster.

How the attacker would exploit it

  • Prerequisite: API permission to read secrets, etcd credential exposure, or compromised control-plane host.
  • Step 1: List secrets across namespaces or connect to etcd.
  • Step 2: Decode secret values and extract DB credentials, API keys, TLS keys, or OAuth credentials.
  • Step 3: Use secrets against production databases, cloud services, or third-party APIs.
  • Step 4: Persist outside Kubernetes even after pod cleanup.

Blue-team control review - what must stop the chain

  • Preventive: Store high-value secrets in external secret manager and inject short-lived credentials at runtime.
  • Preventive: Enable etcd encryption at rest with KMS and restrict direct etcd network access.
  • Preventive: Deny broad list/watch secrets; prefer narrow get only where absolutely required.
  • Detective: Alert on secret reads by service accounts, bulk secret reads, and direct etcd clients.
  • Corrective: Rotate all secrets accessible to compromised identity and invalidate external sessions.

Evidence and telemetry to validate controls

  • SecretAccessAudit: actor, namespace, secret_name, verb, result.
  • EtcdAccessLog: client_ip, client_cert_subject, operation, key_prefix.
  • ExternalSecretLog: workload, secret_id, lease_id, ttl, renewal_count.
  • RotationLog: secret_id, rotated_at, reason, dependent_services_restarted.

Purple-team validation commands

# In staging, verify service account cannot list prod secrets.
kubectl auth can-i list secrets --as=system:serviceaccount:dev:synthetic-app -n prod

SecretAccessAudit
| where verb in ("list", "watch") or access_count > 10
| summarize reads=count() by actor, namespace, bin(TimeGenerated, 5m)

T6 - Cloud metadata service credential theft from pod

STRIDE category: Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Access node metadata service from a pod to steal cloud IAM credentials and pivot from Kubernetes into cloud infrastructure.

How the attacker would exploit it

  • Prerequisite: pods can reach cloud metadata endpoint and node service account has broad cloud permissions.
  • Step 1: From compromised pod, call metadata endpoint such as 169.254.169.254.
  • Step 2: Retrieve node identity token or temporary credentials.
  • Step 3: Use credentials against cloud APIs to access storage, container registry, KMS, or cluster control plane.
  • Step 4: Persist or exfiltrate outside Kubernetes.

Blue-team control review - what must stop the chain

  • Preventive: Use workload identity so pods receive scoped identities independent of node credentials.
  • Preventive: Block pod access to node metadata service except through controlled metadata proxy.
  • Preventive: Minimize node IAM permissions and separate node pools by sensitivity.
  • Detective: Alert on pod-origin requests to metadata endpoint and unusual cloud API calls from node identities.
  • Corrective: Rotate node credentials, cordon affected node, and review cloud IAM grants.

Evidence and telemetry to validate controls

  • IMDSAccessLog: pod_ip, namespace, pod, requested_path, decision.
  • CloudIAMLog: principal, api, resource, source_ip, user_agent.
  • WorkloadIdentityLog: pod, service_account, cloud_principal, token_audience.
  • NodeIAMInventory: node_pool, role, permissions, last_used.

T7 - Malicious third-party Helm chart deployment

STRIDE category: Tampering / Supply Chain
Risk: Medium likelihood, High impact
Attacker objective: Use a third-party chart to deploy privileged resources, backdoored images, broad RBAC, or exfiltration hooks.

How the attacker would exploit it

  • Prerequisite: team installs charts directly from public repositories or allows dynamic chart dependencies.
  • Step 1: Publish or compromise a Helm chart dependency.
  • Step 2: Add templates creating ClusterRoleBindings, privileged pods, hostPath mounts, or external callbacks.
  • Step 3: Chart passes through deployment pipeline without manifest review.
  • Step 4: Backdoor runs with tenant or cluster privileges.

Blue-team control review - what must stop the chain

  • Preventive: Mirror approved charts internally, pin versions and digests, and require provenance verification.
  • Preventive: Render charts in CI and scan the final Kubernetes manifests before deployment.
  • Preventive: Admission policies reject dangerous resources regardless of chart source.
  • Detective: Alert on new chart repositories, chart version drift, and high-risk manifests.
  • Corrective: Roll back Helm release, block chart digest, rotate credentials exposed to release.

Evidence and telemetry to validate controls

  • HelmReleaseAudit: release, chart, version, digest, values_hash, deployer.
  • ChartProvenanceLog: repo, signature_valid, approved, mirrored_at.
  • ManifestScanLog: release, risk_findings, blocked, policy_id.
  • AdmissionAudit: resource_kind, namespace, decision, violating_fields.

T8 - Unsigned or poisoned container image deployment

STRIDE category: Tampering / Supply Chain
Risk: Medium likelihood, High impact
Attacker objective: Deploy a malicious image through compromised CI/CD, registry, base image, or tag drift.

How the attacker would exploit it

  • Prerequisite: images are referenced by mutable tags, no signature verification, or registry credentials are compromised.
  • Step 1: Push malicious image to trusted registry or replace mutable tag.
  • Step 2: Deployment pulls malicious image during rollout or restart.
  • Step 3: Image executes credential theft, data exfiltration, or cryptomining.

Blue-team control review - what must stop the chain

  • Preventive: Require signed images with Cosign/Sigstore or equivalent, verified by admission policy.
  • Preventive: Pin images by digest, not mutable tags, for production workloads.
  • Preventive: Generate SBOM and scan vulnerabilities/malware before promotion.
  • Detective: Alert on image digest drift, unsigned images, and images from unapproved registries.
  • Corrective: Block digest, roll back release, rotate credentials exposed to container.

Evidence and telemetry to validate controls

  • ImageAdmissionLog: image, digest, signature_valid, admission_decision.
  • RegistryAudit: pusher, repo, tag, digest, source_ip.
  • SBOMScanLog: image_digest, critical_findings, malware_findings, approved.
  • DeploymentAudit: namespace, deployment, old_digest, new_digest, actor.

T9 - mTLS or workload identity bypass

STRIDE category: Spoofing / Elevation of Privilege
Risk: Medium likelihood, High impact
Attacker objective: Reach production services without a valid workload identity or impersonate a trusted service.

How the attacker would exploit it

  • Prerequisite: service mesh sidecar injection is optional, permissive mTLS is enabled, or identity issuance trusts weak labels.
  • Step 1: Deploy a pod without sidecar or with spoofed labels.
  • Step 2: Call a production service over plaintext or permissive mTLS.
  • Step 3: Abuse service behavior that trusts source IP, namespace, or headers rather than cryptographic identity.
  • Step 4: Perform unauthorized actions as a fake service.

Blue-team control review - what must stop the chain

  • Preventive: Enforce strict mTLS for prod namespaces with deny-by-default authorization policies.
  • Preventive: Bind workload identity to Kubernetes service account and namespace, not mutable labels.
  • Preventive: Require sidecar injection or ambient mesh enrollment through admission policy.
  • Detective: Alert on plaintext calls, missing identity, unexpected SPIFFE IDs, and authz denials.
  • Corrective: Disable insecure endpoint, apply mesh policy, and redeploy noncompliant workloads.

Evidence and telemetry to validate controls

  • ServiceMeshTelemetry: src_principal, dst_principal, mtls_mode, authz_decision.
  • SPIFFEIssuerLog: service_account, namespace, spiffe_id, issued_at.
  • MeshPolicyAudit: namespace, peer_authentication, authorization_policy, mode.
  • AdmissionAudit: sidecar_required, sidecar_present, decision.

10. Minimum Viable Control Set

If timeline is tight, the minimum viable hardening set is:

  1. Separate prod from dev/staging by cluster or at least by node pool and namespace with strong policy boundaries.
  2. Default-deny NetworkPolicy for every namespace, with explicit egress and ingress allowlists.
  3. Disable service account token automount by default and enforce namespace-scoped least-privilege RBAC.
  4. Pod Security Admission restricted profile plus OPA/Kyverno policies for privileged pods, hostPath, hostNetwork, hostPID, capabilities, and unsigned images.
  5. Kubernetes audit logs, runtime detection, and network flow telemetry correlated in SIEM.

Do not accept a design where dev can directly connect to prod services, default service accounts can read secrets, or prod can run unsigned and privileged workloads.

12. Google-Scale Considerations

At Google scale, Kubernetes hardening must be automated through shared infrastructure and policy-as-code.

  • Policy-as-code: RBAC, NetworkPolicy, Pod Security, image signatures, and chart provenance should be enforced in CI and admission, not reviewed manually.
  • Blast-radius reduction: Production workloads should be isolated by cluster and node pool; sensitive workloads should run in sandboxed runtimes.
  • Continuous graph analysis: Build a live graph of service dependencies, namespace boundaries, service account permissions, and reachable paths.
  • Automated remediation: Compromised pod response should cordon nodes, revoke tokens, rotate secrets, and block image digests with minimal human delay.
  • Telemetry completeness: Every node and namespace should report audit, runtime, and network signals; missing telemetry should be treated as a security incident.
  • Developer usability: Provide secure defaults and paved-road deployment templates so teams do not bypass controls for speed.

13. Model Answer Summary - 90-second version

For a shared Kubernetes cluster with dev, staging, and prod namespaces and unrestricted pod-to-pod traffic, I would first challenge the architecture. Namespaces are not a hard security boundary. A compromised dev pod can read its service account token, probe the API server, list secrets if RBAC is broad, and connect directly to production services over the flat cluster network.

My control set would be layered: separate prod into its own cluster or at least isolated node pools, default-deny NetworkPolicy in every namespace, strict service-to-service mTLS with workload identity, service account token automount disabled by default, least-privilege RBAC, Pod Security Admission restricted profile, OPA/Kyverno admission policies, signed images, internal Helm chart mirroring, and external secret management.

For detection, I would correlate Kubernetes API audit logs, runtime events from Falco or eBPF sensors, and network flow or service mesh telemetry. Critical alerts include cross-namespace secret reads, first-seen dev-to-prod connections, privileged pod admission, direct etcd access, metadata service access from pods, and prod exec/port-forward without JIT approval. Residual risks remain around runtime zero-days, Helm supply chain, and emergency human access, but each should have an explicit owner and acceptance condition.

14. Likely Follow-Up Questions and Strong Answers

Follow-up 1: What is the threat model for a compromised pod in this shared-namespace architecture?

A compromised pod is already inside the trusted cluster network and usually has a mounted service account token. If NetworkPolicy is default-allow, it can reach other namespaces directly. If RBAC is broad, it can query the API server and read secrets. The main chain is RCE in dev, token theft, API enumeration, secret access, direct prod service connection, and data exfiltration. I would reduce blast radius with separate prod clusters, default-deny network policy, least-privilege RBAC, and workload identity.

Follow-up 2: How do you enforce workload identity and service-to-service authentication?

Use a service mesh or equivalent workload identity system issuing cryptographic identities bound to Kubernetes service accounts and namespaces, such as SPIFFE/SVID identities. Enforce strict mTLS in prod and authorization policies that allow only declared caller-to-callee pairs. Applications should not trust source IP or headers alone, because those are weak signals inside the cluster.

Follow-up 3: What prevents privilege escalation from a container to the host?

Admission policy should block privileged containers, hostPath mounts, hostNetwork, hostPID, and dangerous capabilities. Pod Security Admission should enforce the restricted profile, with OPA or Kyverno as a second layer. For sensitive or untrusted workloads, use sandboxed runtimes such as gVisor or Kata. Detection should include runtime rules for shell execution, host file access, and container escape indicators.

Follow-up 4: How do you detect lateral movement?

Use three signals together: Kubernetes API audit logs for secret reads and RBAC changes, network flow logs or service mesh telemetry for first-seen cross-namespace connections, and runtime events for suspicious process execution. A high-confidence alert is a service account in dev reading secrets or making API calls outside its namespace followed by a new connection to a prod service.

Follow-up 5: What residual risk remains for third-party Helm charts?

The residual risk is that a chart or dependency can create dangerous manifests or run malicious code even if the chart appears legitimate. I would reduce that through internal chart mirroring, version and digest pinning, provenance verification, rendered-manifest scanning, and admission policies that reject privileged pods, broad ClusterRoleBindings, and unsigned images regardless of chart source.

15. Common Mistakes to Avoid

  • Treating namespaces as security boundaries without NetworkPolicy, RBAC, and admission controls.
  • Leaving pod-to-pod traffic default-allow across dev, staging, and prod.
  • Allowing default service accounts to carry useful API permissions.
  • Keeping automountServiceAccountToken enabled for workloads that do not need the API server.
  • Storing high-value secrets only in Kubernetes Secrets without external secret management or KMS.
  • Relying on service mesh mTLS while leaving direct pod or database paths open.
  • Allowing kubectl exec and port-forward in prod without JIT, ticket binding, and audit.
  • Installing third-party Helm charts directly from public repos without provenance or rendered-manifest scanning.
  • Setting security admission webhooks to fail open in production.
  • Enabling audit logs but missing runtime and network telemetry.

16. Top 5 Immediate Actions

  1. Break the flat network - separate prod or enforce default-deny NetworkPolicy for every namespace.
  2. Reduce service account blast radius - disable automount by default and enforce least-privilege RBAC.
  3. Block dangerous workloads at admission - enforce restricted Pod Security, OPA/Kyverno policies, and signed images.
  4. Enforce workload identity and mTLS - require authenticated service-to-service calls with explicit authorization.
  5. Correlate K8s audit, runtime, and network telemetry - detect cross-namespace movement, secret reads, privileged workloads, and telemetry gaps quickly.
