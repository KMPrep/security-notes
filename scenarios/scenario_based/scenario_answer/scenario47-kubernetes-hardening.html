Scenario 47 — Kubernetes Cluster Security Hardening

Scenario: A production Kubernetes cluster runs 200 microservices across dev, staging, and prod namespaces on the same cluster. Pod-to-pod communication is unrestricted.

Candidate questions:

  • What is the threat model for a compromised pod in this shared-namespace architecture?
  • How do you enforce workload identity and service-to-service authentication (mTLS)?
  • What controls prevent privilege escalation from a container to the host node?
  • How do you detect lateral movement within the cluster (unusual pod-to-pod calls)?
  • What residual risk remains for third-party Helm charts with unknown provenance?

3.4 Highest-risk boundary crossing

The highest-risk boundary crossing is between the dev or staging namespace and the production namespace when pod-to-pod communication is unrestricted and no NetworkPolicy objects exist. A compromised dev namespace pod can directly reach production services on their cluster-internal IP addresses and ports. Combined with service account tokens that have over-broad RBAC grants, an attacker with RCE in a low-privileged dev workload can pivot to read Kubernetes secrets in the production namespace and reach production databases via the flat pod network — all without leaving the cluster or triggering perimeter defenses.

8. GOOGLE-SCALE CONSIDERATIONS

8.1 Automation and policy-as-code

At Google scale, every Kubernetes security control must be defined as OPA/Gatekeeper policy enforced at admission time — no manual security reviews before deployment. NetworkPolicy, Pod Security Admission, image signing verification, RBAC least-privilege, and service account token disablement must all be enforced by admission webhooks with deny-by-default posture. Policy violations must fail CI/CD pipelines, not just trigger alerts in production.

8.2 Fleet-wide impact analysis

A container escape vulnerability in a shared runtime version affects every node in the cluster simultaneously. At Google scale with thousands of nodes and hundreds of microservices, a single exploited runtime escape in a dev namespace pod can cascade to all production workloads on that node within seconds. Blast radius reduction requires physical cluster separation between dev and prod, node isolation for sensitive workloads using dedicated node pools, and runtime sandboxing (gVisor) for untrusted workloads.

8.3 Eliminating human bottlenecks

Manual RBAC reviews, ad hoc NetworkPolicy configuration, and per-release Helm chart audits cannot scale across 200 microservices updated multiple times per day. Automate RBAC least-privilege enforcement via automated role generation from workload manifests, enforce NetworkPolicy via service mesh declarative config generated from the service dependency graph, and block unapproved Helm charts via internal chart mirroring and signing requirements enforced at the cluster level.

8.4 Controls that break at scale

Pod-by-pod security context review, manual secret rotation after each suspected compromise, and human review of every cluster role binding fail above a few dozen workloads. Replace with policy-as-code that prevents misconfiguration at source, automated secret rotation triggered by security events, and continuous RBAC drift detection with automated alerting for deviation from approved baseline.

ANSWER SUMMARY — 90-second version

Use this to open your answer before going deep on any section.

For a Kubernetes cluster with 200 microservices sharing a flat pod network across dev, staging, and prod namespaces, the core threat is cross-namespace lateral movement — a compromised dev pod can reach production services directly and steal Kubernetes secrets via an over-privileged service account token.

The architectural fix is immediate: the prod, staging, and dev workloads should run in separate clusters in a real production environment. Within a single cluster, the critical controls are NetworkPolicy with default-deny across namespaces, disable automountServiceAccountToken on all pods that don't need API server access, Pod Security Admission enforcing restricted profile, and OPA/Gatekeeper admission policies blocking privileged containers.

For service-to-service identity, deploy Istio with SPIFFE-based workload identity and enforce mTLS everywhere in the prod namespace so unauthenticated lateral movement calls are rejected.

Detection relies on Kubernetes API audit logs for cross-namespace secret reads, Falco runtime rules for container escape signals, and Cilium network flow monitoring for first-seen cross-namespace connections.

Residual risk I'd accept: unknown Helm chart vulnerabilities and zero-day container runtime escapes — mitigated with gVisor for sensitive workloads and chart provenance enforcement.

LIKELY FOLLOW-UP QUESTIONS & IDEAL ANSWERS

Follow-up 1: What is the threat model for a compromised pod in this shared-namespace architecture?

A compromised pod is an authenticated principal inside the cluster network with a mounted service account token. Without NetworkPolicy, it can reach any other pod on its CIDR range. Without RBAC restrictions, its service account token may have cluster-wide read on secrets. The attack chain is: RCE → read service account token → query API server → list secrets in prod namespace → extract database credentials → connect directly to production database. The blast radius of one compromised dev pod equals full production data access in this architecture.

Follow-up 2: How do you enforce workload identity and service-to-service authentication?

Deploy Istio or Linkerd with SPIFFE/SPIRE issuing X.509 SVIDs to each pod based on its Kubernetes service account. Configure PeerAuthentication policy to enforce STRICT mTLS mode in the prod namespace — any call without a valid SVID is rejected at the sidecar layer before reaching the application. This means even a pod with network access cannot impersonate a legitimate service without a valid cluster-issued identity certificate.

Follow-up 3: What controls prevent privilege escalation from a container to the host node?

Pod Security Admission enforcing the restricted profile blocks hostPath mounts, hostPID, hostNetwork, and privileged security contexts. OPA/Gatekeeper policies add a second enforcement layer. For highest-sensitivity workloads, replace the runc runtime with gVisor (runsc) which provides a user-space kernel boundary between the container and the host OS, so even a kernel exploit in the container does not escape to the host.

Follow-up 4: How do you detect lateral movement within the cluster?

Three signal sources: Kubernetes API audit logs detect cross-namespace secret reads and unusual verb patterns from service accounts. Cilium Hubble or Istio telemetry detect first-seen pod-to-pod connections between namespaces that have no declared service dependency. Falco runtime rules detect unexpected process execution inside containers — a shell spawning inside a Go application container is a strong escape signal. Correlate all three in the SIEM.

Follow-up 5: What residual risk remains for third-party Helm charts?

Helm charts are templated Kubernetes manifests that may request privileged security contexts, create ClusterRoleBindings, or include arbitrary container images. The residual risk is that malicious code in a chart runs in the cluster with the permissions of the deploying namespace. Mitigation: mirror all third-party charts in an internal registry, pin chart versions with SHA verification, require mandatory security review for new charts, and enforce admission policies that would reject any misconfigured workload even from a trusted chart source.

COMMON MISTAKES TO AVOID

  • Treating namespace isolation as a security boundary without NetworkPolicy enforcement — namespaces are organizational, not security, boundaries by default.
  • Not disabling automountServiceAccountToken on pods that have no legitimate API server access need.
  • Assuming PodSecurityPolicy (deprecated) or Pod Security Admission alone prevents container escapes without runtime sandboxing.
  • Not separating dev, staging, and prod into separate clusters and relying solely on RBAC for isolation.
  • Storing secrets directly in Kubernetes Secrets (etcd) without external secret management and KMS encryption.
  • Treating service mesh mTLS as optional or only for inter-service calls, while ignoring pod-to-database and pod-to-external-service paths.
  • Not having Kubernetes API server audit logging enabled — this is the single most important forensic evidence source.
  • Allowing cluster-admin ClusterRoleBindings for developer convenience in any environment that shares network or identity with production.
