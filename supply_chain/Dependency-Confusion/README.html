Dependency Confusion — Package Substitution and Registry Attacks

Domain: Supply-Chain Security
References: Alex Birsan "Dependency Confusion" (2021), CWE-1357, OWASP CI/CD-SEC, SLSA, SBOM

1. Overview

Dependency confusion is a class of software supply-chain attack in which a package manager is tricked into fetching an attacker-controlled package from a public registry instead of the intended internal package from a private registry. The root cause is not a memory bug or a cryptographic weakness — it is a name-resolution ambiguity. When an organization uses internally named packages (for example acme-internal-auth) that exist only in a private feed, and a package manager is configured to search both private and public sources, an attacker who publishes a same-named package to the public registry can win resolution because most default resolvers prefer the highest version number across all configured sources rather than the most trusted source.

The attack was popularized in 2021 by Alex Birsan, who used it to execute code inside Apple, Microsoft, Shopify, PayPal, and dozens of other companies by publishing benign proof-of-concept packages under internal names he harvested from leaked package.json files and error messages. It sits alongside two neighboring threats that share the same trust surface: typosquatting (registering a name close to a popular package, such as reqeusts for requests) and malicious package publication (uploading packages whose install scripts or code exfiltrate secrets). All three exploit the fact that installing a dependency is, in effect, executing untrusted code with the privileges of the build or developer.

These notes cover npm and PyPI concretely, but the pattern generalizes to any ecosystem that mixes trusted and untrusted sources under one resolution namespace: RubyGems, Maven, NuGet, Cargo, Go modules, and container base images. Defenses map directly onto SBOM (know exactly which package and version you shipped) and SLSA (prove where each artifact came from and that it was built as expected).

2. How It Works

2.1 The Resolution-Order Root Cause

The flaw lives in how a package manager decides which copy of a name to install when the same name is reachable through more than one configured source. Three design decisions combine to create the hole:

  1. Flat namespace with multiple sources. By default npm and pip treat a bare name like acme-utils as a single global identity. If you point them at both an internal registry and the public registry, both are candidates for that one name.
  2. Version-preference over source-preference. Given two candidates for the same name, the common default is to pick the highest semantic version, not the internal one. An attacker publishes acme-utils at version 99.99.99 publicly; it outranks the internal 1.4.0.
  3. Silent public fallback. Many configurations fall through to the public registry when a name is not found — or is found "better" — on the private feed, with no error and no signal to the developer that trust was crossed.

Concretely, on npm a repository whose .npmrc sets a single registry= that proxies to public, or uses no scope, will resolve unscoped internal names against the public registry. On PyPI, pip install --extra-index-url <private> is the classic trap: --extra-index-url tells pip to consider the private index in addition to PyPI, and pip then picks the highest version across both with no source priority. The attacker never needs to breach the private feed; they only need to know the internal name, which routinely leaks through committed lockfiles, public forks, build logs, and Docker layers.

The exploit payload usually runs at install time, not runtime. npm executes preinstall/postinstall lifecycle scripts automatically; Python executes arbitrary code in setup.py during a source build. So merely resolving and installing the malicious package is enough to get code execution on a developer laptop or a CI runner — the very machines that hold cloud credentials and signing keys.

Top 3 Critical Mitigations

1. Namespace Internal Packages and Reserve the Namespace

The single most effective structural fix is to give every internal package a name that a public attacker cannot own. On npm this means publishing under an organization scope (@acme/utils) and configuring that scope to resolve only from the private registry; an attacker cannot publish @acme/anything because your organization owns the @acme scope on the public registry too. On PyPI, where there is no scope mechanism, the equivalent is reserving your prefix by defensively registering internal names publicly as empty placeholders and, better, isolating internal packages behind an index that is never mixed with public PyPI. The trade-off is a one-time renaming and tooling cost, but it collapses the whole confusion vector because names stop being ambiguous.

2. Private-First Resolution with No Silent Public Fallback

Configuration should make trust, not version number, the deciding factor. Route all installs through a single internal proxy/mirror that pulls approved public packages on your behalf and serves internal packages authoritatively, so the client only ever talks to one trusted source. Crucially, avoid --extra-index-url style setups that ask the client to merge public and private candidates by version; prefer a proxy where a private name is never overridden by a public one, and where an unknown name fails closed rather than falling through to public. The trade-off is running and curating that proxy, but it removes the resolver's freedom to cross a trust boundary on its own.

3. Pin, Hash-Verify, and Verify Provenance

Even with good names and sources, you want cryptographic certainty about what installed. Commit a lockfile that pins exact versions and integrity hashes (npm package-lock.json with integrity, pip hashes via --require-hashes / a compiled requirements.txt), and run CI with npm ci / hash-enforced installs so any drift fails the build. Layer on verified pulls: check package signatures and, for high-value dependencies, require SLSA build provenance attesting the artifact was built from expected source by an expected builder. Feed the resolved set into an SBOM so you can answer "what exactly did we ship" and rapidly find exposure when a bad package is disclosed. The trade-off is stricter, occasionally noisier builds, but it converts trust from "hope" into a verifiable, auditable claim.

5. Practice Questions

Tier 1 — Fundamentals

  1. What is dependency confusion in one sentence? It is tricking a package manager into installing an attacker's public package in place of an intended private one because the resolver prefers version number over source trust. Asked to confirm you can define the core threat crisply.
  2. Why does merely installing a bad package cause harm? Package managers run install-time scripts (postinstall, setup.py) automatically, so resolution alone yields code execution. Asked to test whether you understand install-time versus runtime risk.
  3. What is typosquatting and how does it differ from confusion? Typosquatting relies on a human typo landing on a similarly named malicious package, while confusion needs no typo — it exploits automated resolution of a legitimate internal name. Asked to check you can separate two commonly conflated attacks.
  4. What leaks let attackers learn internal package names? Committed lockfiles, public forks, CI logs, stack traces, and Docker image layers. Asked to test awareness that internal names are effectively sensitive.
  5. What is an npm scope and why does it help? A scope like @acme/ is an owned namespace that only your org can publish under, so public attackers cannot register a colliding name. Asked to confirm you know the primary structural defense.

Tier 2 — System Design and Trade-offs

  1. Design a package-resolution policy that eliminates confusion for a large org. Reserve org scopes, route all installs through a single internal proxy that serves internal names authoritatively and mirrors only allowlisted public packages, disable public fallback, and enforce lockfiles in CI. Asked to test end-to-end control design, not point fixes.
  2. Why is pip install --extra-index-url dangerous and what replaces it? It tells pip to merge private and public indexes and pick the highest version across both, enabling confusion; replace it with a single internal index that proxies public packages. Asked to check concrete PyPI knowledge.
  3. How do lockfiles and integrity hashes defend against these attacks? They pin exact versions and content hashes so any substitution — public or malicious — produces a hash mismatch that fails the build. Asked to test understanding of verification versus mere pinning.
  4. What are the trade-offs of disabling install scripts by default? It removes the most direct install-time RCE path but can break packages that legitimately compile native code, so you allowlist those explicitly. Asked to test judgment about security-versus-usability tension.
  5. How would you detect an in-progress confusion attack? Alert on installs of internal names from a public source, unexpected new public packages matching your name prefixes, and lockfile hash mismatches in CI. Asked to test detection thinking, not just prevention.

Tier 3 — Expert and Deep-Dive

  1. Why is version-preference resolution the real root cause rather than the public registry itself? Because the registry is untrusted by design; the flaw is a client that lets an untrusted source win by version, so fixing the client's trust ordering removes the vector regardless of the public registry. Asked to test root-cause reasoning over surface blame.
  2. How does transitive dependency confusion evade top-level controls? A shadowed internal name can appear deep in the tree of a trusted top-level package, bypassing manifest review; only full-tree resolution and SBOM analysis surface it. Asked to test depth beyond direct dependencies.
  3. How do SBOM and SLSA each reduce this risk, and how do they differ? SBOM answers "what is in the build" so you can find exposure fast, while SLSA answers "where did each artifact come from and was it built as expected," giving provenance-based trust; SBOM is inventory, SLSA is verifiable lineage. Asked to test conceptual command of the modern supply-chain stack.
  4. What is starjacking and why is popularity a weak trust signal? Starjacking is a package falsely claiming another project's repository URL to inherit its stars, so popularity can be forged and must never substitute for verified provenance or signatures. Asked to test skepticism of social trust signals.
  5. A maintainer of a legitimate dependency is compromised and ships a trojaned release — how is this different from confusion, and what still protects you? It is not a naming trick but a trusted-source compromise, so namespacing does not help; provenance verification, delayed adoption, behavioral scanning, and pinned reviewed versions are the remaining defenses. Asked to test whether you can distinguish naming attacks from maintainer compromise and reason about defense in depth.

6. Key Takeaways

  • Dependency confusion is a name-resolution trust flaw, not a code bug: resolvers prefer highest version over most-trusted source.
  • Installing a package can execute code, so resolution alone is enough for compromise of dev and CI machines.
  • Typosquatting, starjacking, and install-script abuse share the same trust surface and often combine.
  • Internal package names are sensitive; they leak through lockfiles, forks, logs, and container layers.
  • Namespacing with reserved scopes is the strongest structural fix; PyPI needs isolation since it lacks scopes.
  • Private-first resolution through one internal proxy removes the resolver's freedom to cross trust boundaries.
  • Pinning plus integrity hashes plus verified pulls turn trust into a verifiable claim; SBOM and SLSA make it auditable.
