SLSA — Supply-chain Levels for Software Artifacts

Domain: Software Supply Chain Security — Build & Artifact Integrity
References: SLSA v1.0 Specification, in-toto Attestation Framework, Sigstore (cosign/Fulcio/Rekor), SSDF (NIST SP 800-218), Executive Order 14028

1. Overview

SLSA (pronounced "salsa") is a security framework — a checklist of controls and a common vocabulary — for hardening the process that turns source code into deployable artifacts. It exists because attackers have shifted left: instead of exploiting a running service, they compromise the build system or a dependency and let the victim's own trusted pipeline ship the malware. SolarWinds, Codecov, and the XZ Utils backdoor all followed this pattern, where a signed, "legitimate" artifact was in fact poisoned upstream of the signature.

SLSA answers one question: "Can I trust that this artifact was built from the source I think it was, by a build process I can inspect?" It does this by defining provenance — tamper-evident, machine-readable metadata describing how, where, and from what an artifact was produced — and a ladder of build levels (L0–L3) that describe how resistant that provenance is to forgery. SLSA is deliberately narrow: it is about build and artifact integrity, not about vulnerability scanning, secrets management, or runtime security. It composes with, rather than replaces, tools like in-toto (the attestation format) and Sigstore (the signing and transparency infrastructure).

2. How It Works

2.1 Core Concepts

Provenance. Provenance is a signed statement, emitted by the build platform, that records the inputs (source repo, commit digest, dependencies), the build entry point (which builder, which config), and the output artifact digests. The consumer verifies provenance before trusting the artifact. Crucially, provenance must be generated by the build system itself, not by the developer, so that a compromised developer workstation cannot forge it.

Levels. SLSA v1.0 defines a Build track with increasing guarantees:
- L0 — No guarantees. Ad-hoc, no provenance.
- L1 — Provenance exists. The build produces provenance describing how it was built. Catches mistakes and gives basic transparency, but provenance can still be forged because it is not cryptographically bound to a hardened builder.
- L2 — Signed provenance from a hosted build service. Provenance is authenticated and generated by a managed platform, so tampering after the build is detectable. Defends against a developer forging provenance on their laptop.
- L3 — Hardened builds. The build platform provides strong guarantees that provenance is unforgeable even against a malicious build-job author: builds run in isolated, ephemeral environments, signing keys are inaccessible to user-defined build steps, and there is no cross-build contamination.

Higher, still-maturing tracks (sometimes written "L3+" or discussed as future Source and Dependency tracks) add hermetic and reproducible builds and two-person review of source.

Build integrity. Two build properties underpin the higher levels:
- Hermetic build — all inputs (source, deps, toolchain) are declared up front and fetched before the build; the build itself has no network access. This eliminates "pull a mutable latest at build time" attacks and makes the input set fully enumerable in provenance.
- Reproducible build — the same inputs deterministically produce a bit-for-bit identical output, so independent rebuilders can confirm an artifact really came from the claimed source.

Attestation & the trust chain. Provenance is one kind of attestation — a signed claim about an artifact. in-toto defines the envelope format; Sigstore provides keyless signing (Fulcio issues short-lived certs tied to OIDC identity) and a transparency log (Rekor) so signatures are publicly auditable. cosign ties it together for container images.

3. What Can Go Wrong

# Attack Type Real Example What Happened Does SLSA Help?
1 Build system compromise SolarWinds Orion (2020) Malicious code injected into the build pipeline, then signed legitimately Yes — L3 hardened, isolated builds + provenance
2 CI secret / credential theft Codecov Bash Uploader (2021) Altered script exfiltrated CI env vars and tokens from thousands of orgs Partial — L2/L3 isolation limits blast radius
3 Malicious maintainer / dep XZ Utils backdoor (2024) Trusted maintainer slipped an obfuscated backdoor into release tarballs Partial — hermetic + reproducible expose tarball drift; Source track needed
4 Typosquatting / dependency confusion PyPI/npm confusion (2021) Attacker publishes package with similar or higher-version name No (out of Build track) — needs dep policy
5 Artifact tampering in registry Container image swap Attacker replaces a pushed image with a backdoored one Yes — signed digests + provenance verification
6 Provenance forgery Fake "we built this" claim Attacker fabricates provenance on their own machine Yes — L2 signed by builder, L3 unforgeable
7 Compromised source commit Injected commit to trusted repo Malicious commit merged without review Partial — Build track ok, Source track addresses
8 Toolchain / compiler backdoor Ken Thompson "trusting trust" Compiler injects backdoor invisible in source Partial — reproducible builds via diverse rebuild
9 Mutable dependency at build time curl \| bash, floating latest Build fetches unpinned input that changes underneath Yes — hermetic builds forbid undeclared fetches
10 Distribution channel MITM Mirror / CDN poisoning Artifact swapped between registry and consumer Yes — digest-pinned verification at consume time

4. Mitigation Controls

The SLSA levels themselves are the control ladder. Adopt them incrementally — each level defends against a strictly stronger adversary.

# Threat Addressed SLSA Control Level
1 No visibility into builds Emit provenance describing build inputs and outputs L1
2 Forged provenance on a laptop Generate + sign provenance on a hosted build service L2
3 Malicious build-job author Isolated, ephemeral builds; keys unreachable by user steps L3
4 Undeclared / mutable inputs Hermetic builds — declare and fetch all inputs up front L3+
5 Compiler / hidden tampering Reproducible builds verified by independent rebuilders L3+
6 Post-build artifact swap Sign artifact digests, verify at deploy via admission policy L2+
7 Audit / non-repudiation gaps Transparency log (Rekor) records every signature L2+

Top 3 Practices in Depth

1. Generate Unforgeable Provenance (L3 Builds)

The single highest-leverage control is moving provenance generation into a hardened, hosted builder where user-controlled build steps cannot touch the signing key. This is what would have caught SolarWinds: even though attackers held code-execution inside the build, an L3 platform running each build in an isolated ephemeral VM, with the provenance signer outside the build sandbox, prevents the attacker from silently producing "clean" provenance for the poisoned binary. The trade-off is real: teams must migrate off self-managed Jenkins-on-a-VM patterns onto platforms like GitHub Actions reusable-workflow generators, Google Cloud Build, or Tekton Chains. The payoff is that provenance becomes evidence, not decoration.

2. Hermetic and Reproducible Builds

Hermeticity closes the "the build reached out and grabbed something mutable" class of attack (Codecov's uploader, curl | bash, floating tags). By requiring every input to be declared and content-addressed before the build starts, and cutting network access during the build, the complete input set becomes enumerable and pinnable in provenance. Reproducibility layers on verification: if two independent builders produce byte-identical output from the same inputs, you gain confidence that no builder injected anything — the practical defense against "trusting trust" compiler backdoors. The cost is engineering discipline (vendoring, pinned toolchains, stripping timestamps and nondeterminism); many large orgs adopt Bazel or Nix precisely to get here.

3. Verify at the Consumption Boundary

Provenance is worthless if nobody checks it. The control that turns SLSA from paperwork into protection is policy-gated admission: at deploy time, a verifier (e.g. a Kubernetes admission controller, cosign policy, or in-toto layout check) fetches the artifact's signed provenance, confirms the transparency-log entry, and asserts policy — "this image must have provenance from our builder, built from our repo, at SLSA L3" — before the artifact runs. This is where distribution-tampering and artifact-swap attacks die. The trade-off is operational: you must run and maintain the verification infrastructure and handle key/identity rotation, and a too-strict policy can block legitimate deploys, so most teams start in warn-only mode before enforcing.

5. Practice Questions

Tier 1 — Fundamentals

  1. What problem does SLSA solve? It provides a framework and vocabulary for build and artifact integrity, so a consumer can verify an artifact came from the expected source via an inspectable build. Asked to confirm the candidate understands SLSA is about the pipeline, not runtime.
  2. What is provenance in SLSA? Signed, machine-readable metadata emitted by the build system describing the artifact's inputs, builder, and output digests. Asked to test the core primitive.
  3. Why must provenance be generated by the build system, not the developer? A compromised developer machine could otherwise forge it; the build platform is the trusted issuer. Asked to test threat-model reasoning.
  4. Name the SLSA build levels and the one-line difference between them. L0 none, L1 provenance exists, L2 signed provenance from a hosted builder, L3 unforgeable provenance from a hardened isolated builder. Asked as a baseline knowledge check.
  5. What is a hermetic build? A build whose inputs are all declared and fetched before build time, with no network access during the build. Asked to test build-integrity vocabulary.

Tier 2 — System Design and Trade-offs

  1. How would SLSA have changed the SolarWinds outcome? L3 isolated builds plus provenance signed outside the build sandbox would have made silently producing clean provenance for a poisoned binary far harder, and consumers verifying provenance could detect drift. Asked to connect the framework to a real incident.
  2. What does SLSA NOT protect against? Dependency confusion/typosquatting, source-review gaps (Build track only), vulnerabilities in correctly-built code, and secrets misuse — those need other controls or the Source/Dependency tracks. Asked to test honest scoping.
  3. How do in-toto and Sigstore relate to SLSA? in-toto is the attestation envelope format that carries provenance; Sigstore (Fulcio + Rekor + cosign) provides keyless signing and a transparency log. SLSA is the requirements framework; the other two are implementation machinery. Asked to test the ecosystem picture.
  4. Why are reproducible builds valuable even if you already have signed provenance? They let independent parties rebuild and confirm the artifact truly derives from the claimed source, defending against a compromised builder or compiler that signing alone cannot catch. Asked to test defense-in-depth thinking.
  5. How would you enforce SLSA in a Kubernetes deployment? A verifying admission controller (cosign policy / in-toto) checks signed provenance and policy — builder identity, source repo, level — before admitting the image. Asked to test operationalization.

Tier 3 — Expert and Deep-Dive

  1. Walk through the XZ Utils backdoor and where SLSA does and does not help. A trusted maintainer injected an obfuscated backdoor into the release tarball that differed from the git source. Hermetic + reproducible builds (build from source, not the shipped tarball) and a Source track with two-person review expose this; the Build track alone, which trusts the maintainer's source input, does not. Asked to probe nuanced framework limits.
  2. How does an L3 platform make provenance unforgeable against a malicious build-job author? By isolating each build in an ephemeral environment, keeping the provenance signing key outside the reach of user-defined build steps, and preventing cross-build contamination, so even code execution inside the build cannot mint arbitrary provenance. Asked to test deep understanding of the L2→L3 boundary.
  3. How would you design a large-org rollout of SLSA? Start with L1 provenance everywhere for visibility, migrate critical services onto a hosted L3-capable builder, add transparency-log verification in warn-only mode, then enforce policy at admission for P1 artifacts, and pursue hermeticity/reproducibility for the highest-risk builds. Asked to test staged, pragmatic execution.
  4. Where does the SLSA trust chain still root, and what is the residual risk? It roots in the build platform, its signing identity (OIDC/Fulcio), and the transparency log's integrity; residual risk includes compromise of that platform, of the OIDC provider, or of the initial source-review process. Asked to test that the candidate sees no framework is turtles-all-the-way-down.
  5. Contrast SLSA's Build track with SSDF and an SBOM — how do they compose? SLSA proves how an artifact was built (integrity), an SBOM enumerates what is inside it (composition, feeding vuln management), and SSDF (NIST 800-218) is the broader secure-development practice framework that references both. They are complementary layers, not substitutes. Asked to test where SLSA sits in the compliance landscape.

6. Key Takeaways

  • SLSA is a framework for build and artifact integrity — proving an artifact came from the expected source via an inspectable build.
  • Provenance (signed build metadata) is the core primitive, and it must be generated by the build system, not the developer.
  • Levels climb from L0 (nothing) to L1 (provenance exists), L2 (signed by a hosted builder), to L3 (unforgeable, hardened isolated builds).
  • Hermetic builds declare all inputs and cut build-time network access; reproducible builds let independent parties verify output byte-for-byte.
  • SolarWinds and Codecov are build-compromise cases SLSA directly targets; XZ needs hermetic/reproducible builds plus a Source track; dependency confusion is out of scope.
  • SLSA composes with in-toto (attestation format) and Sigstore (keyless signing + transparency log); it complements SBOMs and SSDF.
  • Provenance only protects you if it is verified at the consumption boundary through policy-gated admission.
