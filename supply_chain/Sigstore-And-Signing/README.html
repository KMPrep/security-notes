Sigstore and Artifact Signing — Keyless Signatures, Transparency, and Attestations

Domain: Supply-Chain Security
Ecosystem: Sigstore (cosign, Fulcio, Rekor), in-toto, SLSA
Related: SLSA provenance, Kubernetes admission control, OIDC

1. Overview

Sigstore is a set of open services that make signing and verifying software artifacts practical at scale, without asking every developer or CI system to guard a long-lived private key. The core idea is keyless signing: instead of holding a signing key, a signer proves its identity to an OIDC provider (GitHub Actions, Google, an enterprise IdP), Sigstore's certificate authority binds that identity to a short-lived signing certificate, and the signature plus certificate are recorded in a public, append-only transparency log. The private key exists only for the few seconds it takes to sign, then it is discarded.

This solves the two hardest problems in signing. First, key management: there is no key to rotate, store in a HSM, or leak in a build log, because the trust anchor becomes the identity (repo:org/app:ref:refs/heads/main) rather than a key fingerprint. Second, non-repudiation and detection: because every signature is written to a tamper-evident log, a compromised identity that signs a malicious artifact leaves a permanent, publicly auditable record. cosign is the CLI that ties this together for container images and arbitrary blobs, and it also carries in-toto attestations — signed statements about how an artifact was built, which is exactly the substrate SLSA provenance rides on. The downstream payoff is enforcement: a Kubernetes admission controller can refuse to run any image that was not signed by an expected identity, turning "we sign our stuff" into a hard runtime gate.

2. How It Works

2.1 cosign / Fulcio / Rekor Mechanics

The three moving parts play distinct roles:

  • Fulcio is a free certificate authority. It does not issue certificates to whoever asks; it issues a certificate only after validating an OIDC identity token. The resulting X.509 certificate embeds the OIDC identity (the SAN, e.g. an email or a workflow identity) and the issuer, and is valid for roughly ten minutes. This is a code-signing certificate whose subject is a human or workload identity, not a hostname.

  • cosign orchestrates the flow. To sign, it generates an ephemeral keypair in memory, obtains an OIDC token, sends the token plus the public key to Fulcio, and receives a signing certificate. It signs the artifact digest with the ephemeral private key, uploads the signature, certificate, and digest to Rekor, and then throws the private key away. For container images, the signature is stored as an OCI artifact alongside the image in the registry (a .sig tag derived from the image digest).

  • Rekor is the transparency log: an append-only, Merkle-tree-backed ledger. Each entry is a signed statement (signature + certificate + hashed payload). Rekor returns a signed entry timestamp (SET) proving the entry was included at a point in time. This inclusion is what makes short-lived certificates usable — at verification time the cert has long expired, but the Rekor entry proves the signature was created while the cert was valid. The Merkle structure means the log cannot be silently rewritten: any tampering breaks the tree's consistency proofs, which monitors continuously check.

  • Trust root: verifiers must trust Fulcio's CA and Rekor's public key. These are distributed via The Update Framework (TUF) — Sigstore's trusted_root.json is served through a TUF repository so the root of trust itself is rotatable and tamper-resistant. Enterprises running their own Sigstore stack pin their private Fulcio/Rekor keys instead of the public "Sigstore Public Good" instance.

Verification reverses the chain: fetch the signature and certificate, verify the certificate chains to the trusted Fulcio root, check that the certificate's identity (SAN + issuer) matches the expected signer, confirm the Rekor inclusion proof and that the signing time fell inside the certificate's validity window, and finally verify the signature over the artifact digest. Every one of these checks must pass; skipping the identity check in particular reduces "signed" to "signed by anyone," which is worthless.

Attestations extend signatures from "this digest is authentic" to "here is a signed claim about this digest." An in-toto attestation wraps a predicate (a build provenance document, an SBOM, a vulnerability scan result) in a DSSE envelope, signed the same keyless way and logged in Rekor. SLSA provenance is delivered as exactly this kind of attestation: a https://slsa.dev/provenance predicate describing the builder, source, and parameters, which a verifier can then evaluate against a policy ("built by our trusted GitHub Actions builder from our repo").

Top 3 Critical Mitigations

1. Pin the Identity, Not Just the Signature

The single most common failure is verifying that a Sigstore signature exists without checking whose identity signed it. Because Fulcio will issue a certificate to any valid OIDC identity, "is this signed via Sigstore?" is almost always true and almost always meaningless. Verification must assert the exact expected identity and issuer — for example, that the certificate SAN is https://github.com/org/app/.github/workflows/release.yml@refs/heads/main and the issuer is GitHub's OIDC endpoint. Only then does the signature mean "built and released by our own trusted pipeline." The trade-off is that identities must be enumerated and kept in sync as workflows change, but that maintenance cost is the whole point of the control.

2. Enforce at Admission and Fail Closed

Signing has no security value until something refuses to run what is not properly signed. In Kubernetes this belongs at admission control — a policy controller (Sigstore's policy-controller, Kyverno, or Connaisseur) that intercepts every pod, verifies signature and identity by digest, and denies the pod if verification fails. The controller should fail closed: if it cannot verify, it rejects rather than admits, so an outage or a missing signature never becomes an open door. The cost is operational risk — a broken verifier can block all deploys — which is why the mitigation pairs with cached inclusion proofs and tested break-glass procedures rather than a permissive fallback.

3. Verify Provenance, Not Just Authenticity

A valid signature proves an artifact is authentic and untampered; it says nothing about whether it was built safely. An attacker who compromises a build can produce a genuinely signed but malicious image. This is where SLSA provenance attestations close the gap: the policy should require an in-toto provenance predicate and evaluate it — the artifact must have been produced by the expected builder, from the expected source repository, on the expected trigger. Verifying signature and provenance together answers "is this the real thing, and was it made the right way?" The trade-off is a more complex policy and a mature build platform that emits trustworthy provenance, but it is what raises the assurance level from "signed" to genuinely tamper-resistant.

5. Practice Questions

Tier 1 — Fundamentals

  1. What problem does keyless signing solve? It removes long-lived signing keys, which are hard to store, rotate, and protect from leakage; trust shifts to a verifiable OIDC identity plus a transparency log. This is asked to confirm you understand why Sigstore exists, not just how to run the CLI.
  2. What do Fulcio and Rekor each do? Fulcio issues short-lived certificates that bind an OIDC identity, and Rekor is the append-only transparency log that records signatures. This is asked to check you can separate the CA from the log.
  3. Why are Sigstore signing certificates short-lived? The private key exists only briefly, so a short validity window limits misuse; the Rekor entry proves the signature was made while the cert was valid. This is asked to test the interplay between certs and the log.
  4. What is a transparency log and why does it matter here? A tamper-evident append-only ledger that gives non-repudiation and lets anyone detect a malicious signature after the fact. This is asked to test the detection-versus-prevention distinction.
  5. What is an in-toto attestation? A signed statement about an artifact — such as build provenance or an SBOM — wrapped in a DSSE envelope and logged like a signature. This is asked to see if you know signing extends beyond "this blob is authentic."

Tier 2 — System Design and Trade-offs

  1. How would you enforce image signing in Kubernetes? Deploy an admission controller (policy-controller, Kyverno, or Connaisseur) that verifies signature and identity by digest and fails closed. This is asked to connect signing to runtime enforcement.
  2. A team says "our images are Sigstore-signed, so we are safe." What is wrong? Signed by whom? Without asserting the expected identity and issuer, any Fulcio-issued signature passes, which is nearly meaningless. This is asked to catch the most common real-world mistake.
  3. How does verification work when the signing certificate has already expired? The Rekor inclusion proof and timestamp establish that the signature was created during the certificate's validity window. This is asked to test deep understanding of the trust model.
  4. How do you handle Rekor being unavailable during deployment? Bundle inclusion proofs with the artifact and cache signed entry timestamps so verification does not hard-depend on the live log. This is asked to test availability and failure-mode thinking.
  5. How do signing and SLSA provenance relate? Provenance is delivered as a signed in-toto attestation; the signature proves authenticity while the provenance predicate proves how it was built. This is asked to connect Sigstore to the broader supply-chain framework.

Tier 3 — Expert and Deep-Dive

  1. How is Sigstore's own trust root managed and rotated? Through TUF-distributed metadata (trusted_root.json), so the Fulcio and Rekor keys can be rotated and are themselves tamper-resistant. This is asked to test whether you understand the root of trust, not just the leaves.
  2. What does keyless signing not protect against? A compromised build or a poisoned dependency still yields a genuinely signed but malicious artifact; the identity is real, the content is bad. This is asked to test the limits of authenticity versus integrity of the build.
  3. How would you run Sigstore for an enterprise that cannot use the public-good instance? Stand up a private Fulcio and Rekor, wire them to the corporate IdP, and pin the private trust root in all verifiers. This is asked to test operational maturity and trust-boundary design.
  4. Why must verification be done by digest rather than tag? Tags are mutable, so a signed digest can be verified and then the tag repointed to an unsigned image before it is pulled. This is asked to test attention to a subtle but exploitable gap.
  5. How would you detect that a signer identity was compromised and abused? Continuously monitor Rekor for entries under your identities that do not correspond to legitimate releases, since the log makes every misuse permanently visible. This is asked to test how the transparency property turns into an actual detection.

6. Key Takeaways

  • Keyless signing replaces long-lived keys with short-lived certificates bound to an OIDC identity.
  • Fulcio issues the certificate, Rekor logs the signature, cosign ties it together, and TUF anchors the trust root.
  • The transparency log gives non-repudiation and after-the-fact detection, not prevention.
  • Verification is worthless unless it asserts the expected identity and issuer, not merely that a signature exists.
  • Enforce at Kubernetes admission control, fail closed, and always verify by digest.
  • Signatures prove authenticity; SLSA provenance attestations prove the artifact was built the right way — require both.
  • A real identity can still sign a malicious artifact, so build integrity and dependency hygiene remain essential.
