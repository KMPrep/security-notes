Data Flow Diagrams and Trust Boundaries

Domain: Threat Modeling — Methodology & Foundations
References: Shostack Threat Modeling: Designing for Security, Microsoft SDL, OWASP Threat Modeling

1. Overview

A data flow diagram (DFD) is the working model most threat modeling starts from. Instead of reasoning about a system in the abstract, you draw what data exists, where it moves, who touches it, and — critically — where control changes hands. Those hand-off points are trust boundaries, and they are where nearly every interesting threat lives. A DFD is deliberately simple: five element types, a few arrows, and dotted lines marking the boundaries. Its power is not in detail but in forcing you to enumerate every flow that crosses from something you control to something you do not.

DFDs are the on-ramp to structured analysis. Once the diagram exists, you walk each element and each boundary crossing and ask "what can go wrong here." That walk is usually structured with STRIDE (Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege), and the DFD is what makes STRIDE tractable — each element type suggests a predictable subset of the six categories. Getting the diagram right, and the boundaries honest, is the whole game: a wrong boundary hides a real attack surface, and a missing flow hides a real threat.

2. How It Works

2.1 DFD Elements and Trust Boundaries

A DFD uses exactly five building blocks. Keeping the vocabulary small is intentional — it keeps the diagram readable and keeps analysis repeatable.

  • External entity (drawn as a rectangle) — an actor or system outside your control that produces or consumes data: a browser user, an admin, a mobile client, a partner API. You cannot secure it; you can only distrust its input. External entities are the classic source of Spoofing threats.
  • Process (circle) — code that transforms or routes data: a web server, an API service, an auth handler, a Lambda function. Processes are where Tampering, Repudiation, DoS, and Elevation of privilege concentrate because they hold logic and privilege.
  • Data store (two parallel lines, an open-ended box) — data at rest: a relational DB, an object bucket, a cache, a message queue, a log file. Data stores attract Tampering, Information disclosure, and Repudiation (if logs can be altered).
  • Data flow (arrow) — data in motion between the above: an HTTPS request, a SQL query, a queue publish. Flows are where Tampering and Information disclosure happen on the wire.
  • Trust boundary (dotted line) — the line a data flow crosses when the level of trust changes: internet-to-DMZ, DMZ-to-internal, app-to-third-party, user-space-to-kernel, one tenant to another. A boundary is not a network device; it is a change in who you trust.

The key mental model: threats concentrate where flows cross boundaries. Data moving between two fully trusted processes inside the same trust zone is low-interest. The same data crossing from the internet into your app, or from your app out to a third party, is where you must authenticate, validate, encrypt, authorize, and log. When you enumerate threats, you are really walking the list of boundary-crossing flows.

Decomposition levels. DFDs are drawn at increasing detail:

  • Level 0 (context diagram) — the whole system as a single process with its external entities and top-level flows. Answers "what is the system and who talks to it."
  • Level 1 — the system broken into its major processes and data stores (web tier, API tier, DB, third-party). This is where most threat modeling happens — enough detail to see boundaries, not so much you drown.
  • Level 2+ — a single Level-1 process exploded into its internal components. Reserved for high-risk areas (auth, payment, crypto) where finer boundaries matter.

Match the level to the decision you are making. Over-decomposing early produces diagrams no one reads; the boundaries blur and the analysis stalls.

The five labeled crossings are the analysis surface. A flow entirely inside the Internal zone (say, API-to-cache) crosses no boundary and is lower priority; the five crossings above are where authentication, transport encryption, input validation, authorization, and logging must be proven. This is exactly the input a STRIDE walk consumes: for each crossing, ask which of the six STRIDE categories apply given the element types on each end.

3. What Can Go Wrong

Threats mapped to boundary crossings in the worked system, plus the common diagramming mistakes that hide them.

Threats at Boundary Crossings

# Crossing / Location Threat (STRIDE) Root Cause Impact
1 Internet → Web front end Spoofing, Tampering Unauthenticated or unvalidated input Account takeover, injection
2 Admin → Web front end Elevation of privilege Weak admin auth, no separation Full system compromise
3 Web → API (DMZ to internal) Spoofing, Elevation of privilege API trusts front end blindly Bypass of front-end checks
4 API → Database Tampering, Information disclosure Injection, over-broad DB creds Data theft or corruption
5 API ↔ Payment provider Tampering, Information disclosure Weak TLS, no response validation Fraud, leaked card data
6 API → Audit log Repudiation Logs writable or missing Actions cannot be attributed
7 Any flow on the wire Information disclosure Cleartext transport inside the network Sniffing, credential capture
8 Web front end (process) Denial of service No rate limiting at the edge Availability loss
9 Third-party response path Tampering Trusting external data as clean Injection via partner data
10 Cross-tenant / shared data Information disclosure, Elevation Missing authorization on data store One tenant reads another's

Common DFD Mistakes

# Mistake Why It Hurts
1 Drawing network topology, not data flow You model routers, miss the data and the real threats
2 Missing trust boundaries The most important lines are absent; threats invisible
3 Boundary drawn at the firewall only Ignores app-tier, tenant, and third-party trust changes
4 Over-decomposing (Level 3 everywhere) Diagram unreadable, analysis never finishes
5 Treating third party as fully trusted External responses become an unexamined attack surface
6 Omitting the return flows Response paths (e.g., payment result) carry real threats
7 No external entities shown Nothing marks where untrusted input originates
8 Stale diagram vs. real architecture You threat-model a system that no longer exists

4. Mitigation Controls

Controls are applied at the boundary crossing, matched to the threat the crossing exposes. The pattern is consistent: authenticate the source, validate the data, encrypt the flow, authorize the action, and log it.

# Crossing Control STRIDE Addressed
1 Internet → Web Strong auth, input validation, TLS, WAF, rate limiting S, T, D
2 Admin → Web MFA, separate admin plane, least privilege E
3 Web → API Service-to-service auth (mTLS/tokens), do not trust front end S, E
4 API → Database Parameterized queries, scoped DB creds, encryption at rest T, I
5 API ↔ Payment provider Pinned TLS, validate responses, treat as untrusted input T, I
6 API → Audit log Append-only, tamper-evident, integrity-protected logging R
7 Any internal flow Encrypt in transit even internally (zero trust) I
8 Data stores Authorization checks on every read/write, tenant isolation I, E

Top 3 Critical Controls

1. Authenticate and Authorize at Every Boundary

The most common architectural failure is a downstream service trusting an upstream one implicitly — the API assuming the web front end already validated the caller. Each boundary crossing must independently establish who is calling (authentication) and whether they may (authorization). Service-to-service identity via mTLS or signed tokens, plus authorization checks at the data store itself, prevents a single bypassed front-end control from collapsing the whole chain. The trade-off is added latency and key-management overhead, but defense in depth at boundaries is non-negotiable for a serious system.

2. Validate All Data Crossing Inward

Any data entering a more-trusted zone — from the internet, from an admin, or back from a third party — must be treated as hostile until validated. The third-party return path is the one teams routinely forget: a payment provider's response is external input and can carry injection just like a user request. Input validation, output encoding, and parameterized queries at each inward crossing directly kill Tampering and Information-disclosure threats. The trade-off is that strict validation can reject legitimate edge cases, so it must be maintained alongside the API contract.

3. Make the Boundaries Honest and the Diagram Current

Controls only work if the boundaries reflect reality. A DFD that omits the app-tier boundary, or marks a third party as trusted, produces a threat model that misses real attack surface entirely. Keep the diagram at Level 1 for most decisions, redraw it when architecture changes, and place a boundary wherever trust actually changes — not just at the firewall. The trade-off is diagramming effort, but an honest DFD is cheaper than the incident it prevents.

5. Practice Questions

Tier 1 — Fundamentals

  1. What are the five elements of a DFD? External entity, process, data store, data flow, and trust boundary. This is asked to confirm you know the vocabulary before going deeper.
  2. What is a trust boundary? A line a data flow crosses when the level of trust changes — not a network device, but a change in who you trust. This is asked to test whether you understand the core concept versus confusing it with topology.
  3. Why do threats concentrate at trust boundaries? Because that is where untrusted data or actors gain access to more-trusted components, so authentication, validation, and authorization must happen there. This is asked to check you know where to look.
  4. What is the difference between a Level 0 and Level 1 DFD? Level 0 is the whole system as one process with its external entities; Level 1 breaks it into major processes and data stores. This is asked to test decomposition literacy.
  5. How does a DFD feed STRIDE? Each element type maps to a predictable subset of STRIDE, so walking the diagram element by element enumerates candidate threats. This is asked to connect the diagram to structured analysis.

Tier 2 — System Design and Trade-offs

  1. Walk me through threat-modeling a web app with a payment provider. Draw a Level 1 DFD, mark boundaries at internet-to-DMZ, DMZ-to-internal, and internal-to-third-party, then STRIDE each crossing. This is asked to see your end-to-end process, not just definitions.
  2. Where would you place trust boundaries in a three-tier app? At every trust change: internet to web tier, web tier to API tier, API to data store, and app to any external service. This is asked because placing them only at the firewall is the classic mistake.
  3. How do you decide how deep to decompose? Match the level to the decision — Level 1 for most systems, Level 2 only for high-risk components like auth or payments. This is asked to test judgment about analysis cost versus coverage.
  4. A third-party API returns data to your service. Trusted or not? Untrusted — its response is external input and must be validated like any user data. This is asked because the return path is the most commonly missed boundary.
  5. How would you keep a DFD from going stale? Tie it to architecture reviews and CI/design-change gates so it is redrawn when the system changes. This is asked to test whether you treat threat modeling as continuous, not one-time.

Tier 3 — Expert and Deep-Dive

  1. Why is drawing network topology a common DFD failure mode? Topology shows devices and routes, not data and trust changes, so it hides the flows and boundaries where threats actually live. This is asked to separate real threat modeling from network diagramming.
  2. How do trust boundaries change in a multi-tenant system? Tenant-to-tenant becomes a first-class boundary at shared data stores, requiring authorization on every access to prevent cross-tenant disclosure. This is asked to test cloud-scale reasoning.
  3. How does a DFD-driven model handle service-to-service trust in microservices? Each service call is a boundary crossing needing its own identity (mTLS/tokens) and authorization — no implicit trust from upstream. This is asked to probe zero-trust maturity.
  4. What does a DFD not capture, and how do you compensate? It does not capture temporal state, business logic abuse, or attacker capability; supplement with attack trees, abuse cases, and STRIDE-per-interaction. This is asked to test awareness of the method's limits.
  5. How would you scale DFD-based threat modeling across hundreds of services? Standardize reusable DFD patterns and trust-boundary templates, automate diagram generation from IaC, and focus deep review on boundary-crossing and high-risk flows. This is asked to test how you operationalize methodology at scale.

6. Key Takeaways

  • A DFD models data flow and trust, not network topology.
  • Five elements only: external entity, process, data store, data flow, trust boundary.
  • Trust boundaries mark where trust changes — that is where threats concentrate.
  • Level 1 is the sweet spot for most threat modeling; decompose deeper only for high-risk areas.
  • Every inward boundary crossing needs authentication, validation, authorization, encryption, and logging.
  • Data returning from a third party is untrusted input and a commonly missed boundary.
  • The DFD is the input to STRIDE — element types predict which threat categories apply.
  • An honest, current diagram with correct boundaries is the difference between a real threat model and a false sense of security.
