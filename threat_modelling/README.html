Threat Modelling

Category: Security Design, Risk Assessment, Secure Software Development Lifecycle

Deep-dive notes (per-method): STRIDE · PASTA · Attack Trees · DFD & Trust Boundaries. This page is the overview; each note above is a full deep-dive with diagrams, worked examples, and tiered practice questions. For threat-modelling AI/agent systems specifically, see ai_security/Defenses/Secure-AI-SDLC-SAIF.

Overview

Threat modelling is a structured process for identifying and prioritising threats to a system before attackers find them. It answers four core questions:

  1. What are we building?
  2. What can go wrong?
  3. What are we going to do about it?
  4. Did we do a good enough job?

At large scale, threat modelling is not a one-time activity performed by a security team in isolation. It is embedded into design reviews, launch approval workflows, and ongoing detection engineering. At companies like Google, the threat model of a single service change can affect millions of users and dozens of downstream services, so the discipline must be systematic, repeatable, and proportional to risk.

When to Threat Model

Trigger Why It Matters
New feature design Cheapest time to fix; no code exists yet
Architecture change Trust boundaries shift; existing controls may not apply
New data type or sensitivity tier Data classification drives threat surface
Third-party integration You inherit their threat surface
Post-incident Prevent recurrence of the same root cause
Threat hunting / detection development Drives detection rules and alert logic
Periodic review of high-value services Threat landscape evolves even when code does not

Threat Modelling Process

flowchart TD
    A[Define Scope and Assets] --> B[Create Data Flow Diagram]
    B --> C[Identify Trust Boundaries]
    C --> D[Enumerate Threats per Component]
    D --> E[Rate Each Threat]
    E --> F[Propose Mitigations]
    F --> G[Validate Controls]
    G --> H[Document and Track]

Step 1 — Define Scope and Assets

Start by answering: what are we protecting and from whom?

  • List assets: user data, cryptographic keys, admin credentials, billing records, source code, ML models.
  • Define trust tiers: who is trusted, partially trusted, or untrusted (anonymous internet, authenticated user, internal service, admin, third-party partner).
  • Define scope boundary: what components are in scope for this review.

Step 2 — Create a Data Flow Diagram (DFD)

A DFD maps how data moves through the system. Use DFD Level 0 for overview, Level 1 for per-service detail.

DFD elements:

  • External entity (rectangle) — actor outside the system (user, partner API)
  • Process (circle/oval) — transformation of data (auth service, payment processor)
  • Data store (parallel lines) — persistent storage (database, cache, blob store)
  • Data flow (arrow) — data movement between elements
  • Trust boundary (dashed line) — the border where trust level changes

Example for a basic login flow:

flowchart LR
    U[User Browser] -->|HTTPS POST credentials| LB[Load Balancer]
    LB -->|Forward| AS[Auth Service]
    AS -->|Query| DB[(User DB)]
    AS -->|Emit event| EL[(Audit Log)]
    AS -->|Issue session token| U

Trust boundaries sit between: browser ↔ load balancer, load balancer ↔ auth service (if different network segments), auth service ↔ database.

Step 3 — Identify Trust Boundaries

A trust boundary is any place where data crosses from one privilege or trust tier to another:

  • Network perimeter (internet → DMZ → internal)
  • Process privilege boundary (user space → kernel space)
  • Authentication boundary (unauthenticated → authenticated)
  • Service-to-service boundary (stateless API caller → stateful data store)
  • Human-to-system boundary (operator input → automated pipeline)

Why this matters: Most exploitable vulnerabilities happen at trust boundaries where input is not re-validated or where the receiving component over-trusts the caller.

STRIDE Framework

STRIDE is the most widely used threat classification framework, developed at Microsoft. Each letter names a threat category and the security property it violates.

Letter Threat Property Violated Example
S Spoofing Authenticity Attacker forges a JWT to impersonate admin
T Tampering Integrity Attacker modifies a signed package in transit
R Repudiation Accountability User denies placing a fraudulent order; no audit log exists
I Information Disclosure Confidentiality Error message reveals SQL schema
D Denial of Service Availability Attacker floods login endpoint to lock out all users
E Elevation of Privilege Authorization IDOR vulnerability lets user access another user's data

Applying STRIDE to Each DFD Element

DFD Element Applicable STRIDE Threats
External entity S (spoofing identity)
Data flow T (tampering in transit), I (eavesdropping), D (flooding)
Process S, T, R, I, D, E (all six)
Data store T (tampering at rest), I (data leak), D (deletion/corruption)

Tip: Walk through each trust boundary on your DFD and ask: which STRIDE threats apply here? What is the control? What happens if the control fails?

DREAD Risk Scoring

DREAD assigns a numeric score to each threat to prioritize the backlog.

Dimension Question Score 1–10
Damage How bad is the damage if exploited? 10 = full data loss or RCE
Reproducibility How easy is it to reproduce? 10 = trivially reproducible
Exploitability How much skill/effort does exploitation require? 10 = script kiddie, no skill
Affected Users How many users are affected? 10 = all users
Discoverability How easy is it for an attacker to find the vulnerability? 10 = publicly documented

DREAD Score = (D + R + E + A + D) / 5

Thresholds: 0–3 Low, 4–6 Medium, 7–10 High.

Criticism of DREAD: Scores are subjective and vary between reviewers. DREAD was officially retired by Microsoft in 2008. However, the dimensions remain useful as a checklist even without formal scoring.

MITRE ATT&CK Framework

MITRE ATT&CK is a knowledge base of real-world adversary tactics, techniques, and procedures (TTPs) observed in the wild. It provides a common vocabulary for describing attacker behavior.

Structure:

  • Tactics (the "why") — 14 high-level goals, e.g. Initial Access, Persistence, Privilege Escalation
  • Techniques (the "how") — specific methods, e.g. T1566 Phishing, T1059 Command and Scripting Interpreter
  • Sub-techniques — granular variants, e.g. T1566.001 Spearphishing Attachment
flowchart LR
    TA[Tactics] --> TE[Techniques]
    TE --> ST[Sub-techniques]
    ST --> PR[Procedure Examples]
    PR --> MI[Mitigations and Detections]

Use: Map your system's components to ATT&CK techniques. "An attacker who compromises our frontend could use T1552 to look for credentials in environment variables, then T1078 to authenticate to our internal admin service."

Threat Matrix

A threat matrix is a working artefact produced during threat modelling. It lists threats, scores them, assigns owners, and tracks mitigations.

ID Component Threat STRIDE DREAD Score Mitigation Status
T001 Auth API Credential stuffing S 8.0 Rate limiting, CAPTCHA, anomaly detection Implemented
T002 Auth API JWT algorithm confusion (alg:none) S, E 9.2 Reject tokens unless alg is explicitly RS256 Open
T003 User DB SQL injection via username field T, I 7.8 Parameterised queries; WAF rule Implemented
T004 Audit log Log tampering by compromised admin T, R 6.5 Write-once log storage; SIEM export Partial

Mitigations by STRIDE Category

Threat Primary Mitigations
Spoofing Strong authentication (MFA, passkeys), mutual TLS, signed tokens, certificate pinning
Tampering Integrity checks (HMAC, digital signatures), TLS in transit, encryption at rest
Repudiation Immutable audit logs, WORM storage, non-repudiable signing
Information Disclosure Least-privilege access, encryption, sanitised error messages, DLP
Denial of Service Rate limiting, autoscaling, circuit breakers, WAF, CAPTCHA
Elevation of Privilege RBAC/ABAC enforcement, sandbox, principle of least privilege, code review

Common Failure Modes in Threat Modelling

Failure Mode Impact Better Approach
Threat model is only done once System evolves; threats accumulate Integrate into CI/CD and launch review
Only security team participates Engineers don't own the mitigations Engineers build the DFD; security facilitates
Too abstract, no DFD Threats are generic, mitigations are vague Always draw the data flow first
No scoring or prioritization Everything is "high"; nothing gets fixed Use DREAD or similar to force ranking
Focuses only on perimeter Internal threats and lateral movement missed Include internal service-to-service flows
No follow-through on open items Threats acknowledged but never fixed Track in bug tracker; require closure before launch

Practice Questions

Design / Process:

  • Walk me through how you would threat model a new payment processing API.
  • How would you embed threat modelling into a continuous delivery pipeline?
  • What is the difference between a threat and a vulnerability?
  • When should you stop doing threat modelling and ship? How do you know you've done "enough"?

Technical:

  • Apply STRIDE to a REST API that accepts a URL parameter and fetches data from that URL.
  • A new service will store user PII and be accessible by 50 internal microservices. What threats do you identify?
  • How does MITRE ATT&CK differ from a threat model? When would you use each?

Scenario:

  • You are reviewing a design for a logging pipeline that writes to a shared S3 bucket. Security engineering is told "logs only, no secrets." What threats do you model?
  • An intern suggests DREAD scoring is "too subjective to be useful." How do you respond?

Key Takeaways

  • Always start with a DFD. You cannot find threats in a system you have not drawn.
  • Trust boundaries are where most exploitable vulnerabilities live.
  • STRIDE gives you completeness; DREAD gives you priority.
  • MITRE ATT&CK grounds threat modelling in real-world attacker behavior.
  • Threat modelling without follow-through is theatre. Track every open item.
  • At scale, threat modelling must be delegated to engineering teams with security acting as advisor and reviewer.
