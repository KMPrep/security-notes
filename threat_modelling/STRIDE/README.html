STRIDE — Threat Modeling Framework

Domain: Threat Modeling / Secure Design
Origin: Microsoft (Loren Kohnfelder & Praerit Garg, 1999)
References: Microsoft SDL Threat Modeling, OWASP Threat Modeling, Shostack "Threat Modeling: Designing for Security"

1. Overview

STRIDE is a threat-classification framework that maps six categories of attack — Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, and Elevation of privilege — onto the six security properties they violate. Each category is the inverse of a desirable property: Spoofing breaks authentication, Tampering breaks integrity, Repudiation breaks non-repudiation, Information disclosure breaks confidentiality, Denial of service breaks availability, and Elevation of privilege breaks authorization. You apply STRIDE by walking a data-flow diagram (DFD) and, at each element that crosses a trust boundary, asking which of the six threats apply and what control neutralizes them. It gives structure to an otherwise open-ended "what could go wrong?" conversation, which is exactly why it works well under time pressure. STRIDE tells you what to look for; it does not rank risk, so it pairs with DREAD or a risk matrix for prioritization.

2. How It Works

2.1 Core Mechanics

STRIDE is applied against a data-flow diagram built from four element types: external entities (actors/clients), processes (services/code), data stores (databases, queues, caches), and data flows (the arrows between them). Trust boundaries are drawn wherever the level of trust changes — the browser-to-server edge, the service-to-database edge, the boundary between your VPC and a third-party API. Threats concentrate at these boundaries because that is where an attacker-controlled element meets a trusted one.

The practical technique is STRIDE-per-element: for each element you consult a susceptibility matrix, because not every category applies to every element type. External entities can be Spoofed and can Repudiate; processes are vulnerable to all six; data stores are prone to Tampering, Information disclosure, Denial of service, and (for logs) Repudiation; data flows are prone to Tampering, Information disclosure, and Denial of service. For every applicable cell you name a concrete threat, decide whether an existing control mitigates it, and if not, log it as a finding. The alternative, STRIDE-per-interaction, enumerates threats along each (source, flow, destination) tuple instead and scales better for large systems.

The mapping from category to property to control is the load-bearing idea:

STRIDE Violated Property Primary Control Family
Spoofing Authentication Identity proofing, MFA, mutual TLS
Tampering Integrity Signatures, hashes, access control, TLS

Worked example: an API-backed web app with a user database, a session cache, an audit log, and a third-party payment callback (Diagram A). Each row is a concrete threat surfaced by STRIDE-per-element.

4. Mitigation Controls

STRIDE Concrete Controls
Spoofing MFA, WebAuthn/passkeys, mutual TLS between services, OIDC/OAuth token validation, token binding, anti-replay nonces
Tampering TLS for flows, HMAC/signatures on messages, integrity hashes on stored objects, least-privilege write access, immutable infrastructure
Repudiation Append-only audit logs, log signing, tamper-evident storage (WORM), synchronized secure timestamps, forwarding logs off-host
Information disclosure TLS 1.2+/1.3, encryption at rest, field-level and column-level encryption, least privilege, data minimization, suppressing verbose errors
Denial of service Rate limiting and quotas, WAF, autoscaling, circuit breakers, request cost accounting, upstream DDoS scrubbing, timeouts
Elevation of privilege Deny-by-default authorization, per-object authZ checks, input validation, sandboxing, dropping OS privileges, RBAC/ABAC, avoiding confused-deputy patterns

Top 3 Controls — Deeper Context

1. Strong Authentication Against Spoofing

Spoofing is the entry point for most attack chains, so authentication controls carry the most leverage. Human identity should use phishing-resistant MFA (WebAuthn/passkeys) rather than SMS or TOTP, and service identity should use short-lived, automatically rotated credentials — mutual TLS or workload identity — instead of long-lived static keys. The design goal is that possession of a network position never equals possession of an identity. The trade-off is operational: certificate and key rotation add moving parts, but that is far cheaper than credential theft leading to full impersonation.

2. Authorization Against Elevation of Privilege

EoP is the highest-severity STRIDE category because a successful case usually collapses every other property at once. The durable defense is deny-by-default authorization checked at every request, including object-level checks — the single most common API flaw (BOLA/IDOR) is an endpoint that authenticates the caller but never confirms the caller owns the specific object being accessed. Authorization must live on the server and be re-evaluated per request; never trust a client-supplied role, tenant ID, or "is_admin" flag. The trade-off is that centralized, fine-grained authZ adds latency and code, so teams cache decisions and use policy engines (OPA, Cedar) to keep it maintainable.

3. Encryption and Least Privilege Against Information Disclosure

Confidentiality controls are layered because no single control is sufficient: TLS protects data in motion, encryption at rest protects stolen media, and least-privilege plus data minimization limit blast radius when the other two fail. The highest-value move is often reducing how much sensitive data exists and who can reach it, not adding another cipher. The trade-off is that key management, tokenization, and access reviews are ongoing operational burdens, but they cap the damage of the breaches that inevitably slip past prevention.

5. Practice Questions

Tier 1 — Fundamentals

  1. What does each letter of STRIDE stand for? Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege. This is asked to confirm you know the framework before they build on it.
  2. What security property does each STRIDE category violate? Authentication, integrity, non-repudiation, confidentiality, availability, and authorization respectively — STRIDE is literally the inverse of those six properties. This is asked to see if you understand why the categories exist rather than memorizing a list.
  3. What is a data-flow diagram and why does STRIDE need one? A DFD models external entities, processes, data stores, and flows, plus trust boundaries; STRIDE is applied per element, so without the diagram you have nothing to enumerate against. This is asked to test whether you can ground the framework in an artifact.
  4. What is a trust boundary and why does it matter? It is any edge where the level of trust changes, such as client-to-server or service-to-third-party; threats concentrate there because attacker-controlled input meets trusted logic. This is asked because boundary identification is the core skill of threat modeling.
  5. How is Spoofing different from Elevation of privilege? Spoofing is becoming someone you are not (breaking authentication); EoP is doing something your identity is not allowed to do (breaking authorization). This is asked to check that you separate authN from authZ cleanly.

Tier 2 — Design and Application

  1. Walk me through applying STRIDE to a login API. Draw browser to gateway to auth service to user DB; at the browser flow enumerate Spoofing (credential stuffing, token replay) and Tampering, at the DB enumerate Information disclosure and Tampering, then assign MFA, rate limiting, parameterized queries, and signed audit logs. This is asked to see if you can execute the method live, not just define it.
  2. Not every threat applies to every element — how do you know which to consider? Use the susceptibility matrix: external entities get S and R, data stores get T/I/D and R for logs, data flows get T/I/D, and processes get all six. This is asked to test whether you apply STRIDE efficiently instead of brute-forcing every cell.
  3. STRIDE gives you a list of threats but no priorities — how do you rank them? STRIDE only classifies; pair it with a risk method like DREAD, a likelihood-by-impact matrix, or CVSS-style scoring, and weight by business context and exploitability. This is asked to see that you know STRIDE's boundary and don't overclaim.
  4. How does Repudiation manifest in a system with no user-facing identity, like an internal batch job? It shows up as inability to attribute an action to a specific service or operator; the fix is signed, append-only audit logs with strong service identity and secure timestamps. This is asked because Repudiation is the category candidates most often skip or misunderstand.
  5. A third-party payment API calls back into your gateway — what STRIDE threats does that flow introduce? Spoofing (is the callback really from the provider?), Tampering (was the amount or status altered?), and Repudiation; controls are signature/HMAC verification of the webhook, allow-listing source, idempotency keys, and TLS. This is asked to test threat modeling of inbound integrations, a common real-world blind spot.
  6. How do you keep a threat model from going stale after the design changes? Store the DFD and threats as code/docs in the repo, tie updates to design-review gates, and re-run STRIDE on the diff when boundaries or data flows change. This is asked to test whether you treat threat modeling as a lifecycle activity, not a one-time document.
  7. When would you choose STRIDE-per-interaction over STRIDE-per-element? For large systems where per-element enumeration explodes; per-interaction walks each (source, flow, destination) tuple and produces fewer, more relevant threats. This is asked to probe how you scale the method to real architectures.

Tier 3 — Expert and Deep-Dive

  1. Where does STRIDE break down, and what would you use alongside or instead of it? It is weak on business-logic abuse, privacy harms, and multi-step attack chains; complement it with LINDDUN for privacy, attack trees or MITRE ATT&CK for chained adversary behavior, and PASTA for risk-centric depth. This is asked to see senior-level awareness of the framework's limits.
  2. How would you adapt STRIDE for an LLM-backed or agentic system? Map prompt injection to Tampering/EoP, training-data or context leakage to Information disclosure, model impersonation to Spoofing, and tool/plugin abuse to Elevation of privilege, then anchor findings to OWASP LLM Top 10 and MITRE ATLAS. This is asked to test whether you can stretch a classic framework onto modern attack surface.
  3. A confused-deputy vulnerability — which STRIDE category is it, and why is that subtle? It is Elevation of privilege: a trusted intermediary is tricked into using its authority on an attacker's behalf, so the attacker never directly authenticates as the higher role. It is subtle because authN succeeds and the flaw is purely in authZ/context handling. This is asked to test deep reasoning about authorization failures like SSRF and OAuth token confusion.

6. Key Takeaways

  • STRIDE is six threat categories that are the exact inverse of six security properties — authN, integrity, non-repudiation, confidentiality, availability, authorization.
  • You apply it against a data-flow diagram, and threats concentrate at trust boundaries where untrusted input meets trusted logic.
  • STRIDE-per-element with a susceptibility matrix keeps enumeration efficient; STRIDE-per-interaction scales better for large systems.
  • Map each category to its property, then to a concrete control family — that mapping is the whole point of the framework.
  • STRIDE classifies but does not prioritize; pair it with DREAD, CVSS, or a likelihood-by-impact matrix to rank risk.
  • Elevation of privilege and Spoofing carry the most leverage; object-level authZ and phishing-resistant authN are the highest-value controls.
  • Know its limits: complement STRIDE with LINDDUN (privacy), attack trees / MITRE ATT&CK (chains), and PASTA (risk-centric) for a complete picture.
