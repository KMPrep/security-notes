API Security

Category: APIs — Attack Surface, Authentication, Authorization, and Abuse
OWASP: OWASP API Security Top 10 (2023)
Applies To: REST APIs, GraphQL APIs, gRPC APIs, WebSocket APIs, and mobile backend APIs

Overview

APIs are the backbone of modern applications — mobile apps, SPAs, microservices, and third-party integrations all communicate via APIs. Unlike traditional web pages that render UI, APIs expose raw data and operations. APIs often have broader attack surfaces because they are designed for machine-to-machine communication and may lack security controls that UIs enforce (rate limiting, access control checks, input validation). An insecure API can expose far more data than any web page: bulk user records, PII, financial data, administrative functions, and internal infrastructure details.

At Google scale, public APIs serve billions of requests per day. The Google Cloud APIs, Google Maps API, YouTube Data API, and Gmail API collectively handle enormous volumes of requests. The OWASP API Security Top 10 addresses the unique threats that APIs face that go beyond the traditional OWASP Web Application Top 10.

How It Works

OWASP API Security Top 10 (2023)

API1: Broken Object Level Authorization (BOLA / IDOR)

The most common API vulnerability. An API exposes object IDs and does not verify that the authenticated user has access to that specific object.

GET /api/orders/12345          → Returns your order
GET /api/orders/12346          → Should fail, but returns another user's order
GET /api/users/987654/profile  → Returns any user's profile by ID

At scale, automated enumeration can extract millions of records by iterating IDs.

flowchart TD
    A[API receives GET /users/12345] --> B{Authenticated?}
    B -- Yes --> C{Does the authenticated user OWN user 12345?}
    C -- Check performed --> D[Allow or deny based on ownership]
    C -- Not checked - BOLA --> E[Return any user record for any ID]
    E --> F[Attacker enumerates 1 to N million user IDs]
    F --> G[Entire user database exposed]

API2: Broken Authentication

APIs with weak authentication:

  • Tokens that never expire
  • Weak API keys (short, guessable, no rotation)
  • No account lockout on failed auth attempts
  • JWT algorithm confusion (accepting alg: none)
  • Basic Auth over HTTP
  • Credentials in URL parameters (logged everywhere)

API3: Broken Object Property Level Authorization

Returning more object properties than the client should see:

GET /api/me
{
    "id": 123,
    "name": "Alice",
    "email": "alice@example.com",
    "internal_risk_score": 0.87,     // Should not be returned
    "password_hash": "bcrypt:$2b...", // Critical: should never be returned
    "admin": false,
    "stripe_customer_id": "cus_xxx"  // PCI concern
}

Also: Mass assignment — allowing clients to set server-managed properties:

PUT /api/profile
{"name": "Alice", "admin": true}  // Client sets admin=true; server blindly applies all fields

API4: Unrestricted Resource Consumption

APIs without rate limiting or resource quotas:

  • Unlimited requests per minute → DoS / abuse
  • No limit on response size → large queries dump entire databases
  • No timeout on expensive operations → CPU/memory exhaustion
  • No limit on concurrent operations per user

API5: Broken Function Level Authorization

Privileged operations not protected behind authorization checks:

GET /api/admin/users          → Admin-only endpoint; no auth check
DELETE /api/users/all         → Undocumented bulk delete endpoint
POST /api/admin/promote?uid=5 → Promote to admin via GET-accessible endpoint

API6: Unrestricted Access to Sensitive Business Flows

Business logic abuse at API level:

  • Buying expensive items with quantity overflow (integer wrap)
  • Referral code abuse: creating circular referral chains
  • Rate limiting on business logic only, not API calls
  • Coupon or voucher enumeration
  • Account creation spam

API7: Server-Side Request Forgery (SSRF)

Covered in the SSRF note; APIs that accept URLs and fetch them server-side.

API8: Security Misconfiguration

  • Default credentials on API management platforms
  • Unnecessary HTTP methods enabled (e.g., TRACE, OPTIONS without CORS)
  • Verbose error messages with stack traces
  • Outdated TLS versions accepted
  • CORS wildcard on authenticated endpoints
  • Swagger/OpenAPI schema exposed publicly with sensitive endpoint descriptions

API9: Improper Inventory Management

Shadow APIs and zombie APIs:

  • Old API versions still running (v1 without the security fixes of v3)
  • Internal APIs accidentally exposed to the internet
  • Debug or testing endpoints deployed to production
  • Third-party integrations with broader access than needed

API10: Unsafe Consumption of APIs

Trusting third-party API responses:

  • Storing unvalidated third-party API response data in database
  • Following redirects from third-party APIs without checking destination
  • Deserializing untrusted data from third-party response payloads

REST API Attack Surface

GraphQL adds unique security considerations:

    __schema {

# Deep nesting DoS

APIs often return fields the client does not render but that attackers can read:

  "user": {
    "id": 1,
    "email": "user@example.com",

}

Authorization Check Pattern for BOLA Prevention

@require_auth
def get_order(request, order_id):
    order = Order.query.get(order_id)

    # BOLA check: verify this user owns this order
    if order.user_id != request.user.id:
        return Response(403)  # Never 404 (to avoid info leakage on existence)

    return jsonify(order.to_dict())

Never do:

# INSECURE: assumes ID in URL is trustworthy
def get_order(request, order_id):
    return Order.query.get(order_id)  # Missing authorization check

Top 3 Critical Mitigations

  1. Enforce object-level authorization on every API endpoint. BOLA (IDOR) is the number one API vulnerability. Every time you implement an endpoint that retrieves a resource by ID, verify that the authenticated user is authorized for that specific resource. Build this as a framework-level check, not ad-hoc per-endpoint logic — at Google scale, missing one endpoint is enough for a massive data breach.
  2. Use explicit field allowlists in API request handlers. Mass assignment bypasses privileges by allowing clients to set server-managed fields (admin: true, balance: 1000000). Use DTO (Data Transfer Objects) or serializer whitelist patterns that explicitly declare which request fields are accepted and mapped. Never blindly map all request fields to a database model.
  3. Implement rate limiting and enumeration defense on all endpoints. Without rate limits, any BOLA or auth weakness is amplified from an individual user issue to a full dataset breach. Rate limiting per user + per IP + alerting on sequential ID access patterns transforms a single vulnerability from a full database dump into a detectable, limited-impact incident.

Answers to Practice Questions

API Security Model Answers — Deep Dive

1. What is BOLA / IDOR and how is it different from a broken authentication vulnerability?

BOLA stands for Broken Object Level Authorization. It is also commonly called IDOR, or Insecure Direct Object Reference.

BOLA happens when an API verifies that the caller is logged in, but fails to verify whether that caller is allowed to access the specific object being requested. OWASP ranks BOLA as API1:2023 because APIs commonly expose object identifiers in paths, query parameters, headers, and request bodies. Attackers exploit this by changing object IDs in requests. (OWASP Foundation)

Simple example

User A is authenticated and sends:

GET /api/v1/accounts/1001
Authorization: Bearer token_for_user_A

The attacker changes the object ID:

GET /api/v1/accounts/1002
Authorization: Bearer token_for_user_A

If the API returns account 1002, that is BOLA.

The user was authenticated, but not authorized for that object.

BOLA vs broken authentication

Area Broken Authentication BOLA / IDOR
Core question “Who are you?” “Are you allowed to access this specific object?”
Failure type Login/session/token validation failure Object-level authorization failure
Example Accepting expired JWTs, weak password reset, session fixation User changes /orders/123 to /orders/124 and accesses another user’s order
Authentication required? Often bypasses or compromises authentication Usually happens after successful authentication
Primary control Strong authN, MFA, session security, token validation Per-object authorization checks
Common mistake Trusting unauthenticated or forged identity Trusting authenticated identity too broadly

Model answer

Broken authentication means the system failed to correctly establish or protect identity. BOLA means the system knows who the user is, but fails to check whether that user is allowed to access the specific object. Authentication answers “who are you?” Authorization answers “what are you allowed to access?”

2. How would you design an authorization framework that prevents BOLA across thousands of API endpoints in a microservices architecture?

For thousands of APIs, I would not rely on each developer remembering to write object checks manually. I would design centralized policy, decentralized enforcement, and mandatory object-scoped authorization.

The design goal is:

Every endpoint that reads, updates, deletes, exports, or acts on an object must prove that the authenticated principal has permission over that object.

High-level architecture

flowchart TD
    A[Client] --> B[API Gateway]
    B --> C[Authentication Layer]
    C --> D[Token Validation / Identity Context]
    D --> E[Service Mesh / Microservice]
    E --> F[Authorization Middleware]
    F --> G[Policy Decision Point]
    G --> H[Policy Store]
    F --> I[Resource Ownership / Relationship Lookup]
    I --> J[Database / Entitlement Service]
    F --> K[Business Handler]
    K --> L[Data Access Layer]
    L --> M[Database]

    N[Audit Logs] --> O[SIEM]
    F --> N
    G --> N

Core components

1. Strong identity context

Every request should carry a trusted identity context:

{
  "subject": "user_123",
  "tenant_id": "tenant_abc",
  "roles": ["employee"],
  "groups": ["finance-apac"],
  "auth_time": 1710000000,
  "client_id": "mobile-app",
  "scopes": ["orders:read"]
}

But identity context alone is not enough.

The dangerous mistake is:

if (user.isAuthenticated()) {
    return orderRepository.findById(orderId);
}

The correct model is:

if (authz.canRead(user, "order", orderId)) {
    return orderRepository.findById(orderId);
}

2. Object-level authorization middleware

Every route that contains object identifiers should pass through an authorization guard.

Example:

@GetMapping("/orders/{orderId}")
public OrderDto getOrder(
    @AuthenticatedUser UserPrincipal user,
    @PathVariable String orderId
) {
    authorizationService.check(user, Action.READ, ResourceType.ORDER, orderId);
    return orderService.getOrder(orderId);
}

The important idea is that authorization is not optional business logic. It is a framework-level requirement.

3. Policy Decision Point

Use a central policy engine or service, for example:

  • OPA / Open Policy Agent
  • Cedar-style policy model
  • Zanzibar-style relationship-based authorization
  • Custom entitlement service
  • Cloud IAM-style model

Policy example:

allow {
  input.action == "read"
  input.resource.type == "invoice"
  input.principal.tenant_id == input.resource.tenant_id
  input.principal.id == input.resource.owner_id
}

For enterprise systems, I prefer a model supporting:

Model Use case
RBAC Coarse role-level permission
ABAC Attribute-based decisions, e.g. region, department, risk tier
ReBAC Relationship-based access, e.g. user owns document, belongs to project, manages employee
Contextual authZ Device trust, location, step-up status, transaction risk

4. Resource ownership and relationship lookup

BOLA prevention requires knowing who owns or can access the object.

For example:

SELECT tenant_id, owner_user_id, project_id, classification
FROM orders
WHERE order_id = :orderId;

Then authorization checks:

request.user.tenant_id == order.tenant_id
AND (
  request.user.id == order.owner_user_id
  OR request.user has relationship "project_admin" on order.project_id
)

5. Secure data access patterns

A very strong pattern is to avoid fetching by object ID alone.

Bad:

SELECT * FROM orders WHERE id = :orderId;

Better:

SELECT * FROM orders
WHERE id = :orderId
AND tenant_id = :userTenantId
AND owner_user_id = :userId;

For multi-tenant systems, always include tenant boundaries in queries.

6. Deny by default

Default behavior should be:

No policy match = deny
No resource found = generic 404
No tenant match = deny
No ownership proof = deny

Avoid returning:

{
  "error": "You are not allowed to access invoice 12345 owned by tenant XYZ"
}

That leaks object existence.

7. Standardized route metadata

Each endpoint should declare what resource and action it operates on.

Example:

endpoint: GET /api/orders/{orderId}
resource_type: order
resource_id_param: orderId
action: read
authorization_required: true

This can be enforced through:

  • API gateway config
  • Service annotations
  • OpenAPI extensions
  • CI/CD checks
  • Contract tests

8. CI/CD authorization checks

For thousands of endpoints, add automated controls:

Control Purpose
Static analysis Detect handlers reading object IDs without authZ guard
OpenAPI linting Require x-authorization metadata
Unit tests Verify owner, non-owner, tenant-crossing cases
Integration tests Replay IDOR attack cases
Fuzzing Mutate IDs in requests
Negative tests Confirm userA cannot access userB objects
Production canaries Detect abnormal object access patterns

BOLA prevention test matrix

Test case Expected result
Owner reads own object Allow
User reads another user’s object in same tenant Deny
User reads object from another tenant Deny
Admin reads object within authorized tenant Allow
Admin reads object outside authorized scope Deny
Deleted object accessed Deny or 404
Sequential ID enumeration Block / alert
UUID guessing Deny / alert
Batch request with mixed authorized and unauthorized IDs Return only authorized objects or fail safely

Model answer

At scale, BOLA prevention must be systematic. I would not trust individual endpoint developers to remember object checks. I would enforce authorization through shared middleware, a central policy decision point, route metadata, tenant-aware data access, mandatory negative tests, and continuous detection for object enumeration. Every object access must be authorized against the caller, tenant, action, and resource relationship.

3. What is mass assignment and how do DTO patterns prevent it?

Mass assignment happens when an application automatically binds user-controlled input into internal objects or database models, allowing attackers to set fields they should not control.

OWASP recommends allow-listing bindable fields and using DTOs so that only intended user-editable properties can be assigned. (OWASP Cheat Sheet Series)

Vulnerable example

Assume this API updates a user profile:

PATCH /api/users/me
Content-Type: application/json

{
  "displayName": "Kumar",
  "phone": "91234567",
  "role": "admin",
  "isVerified": true,
  "accountBalance": 999999
}

Bad backend code:

@PostMapping("/users/me")
public User updateUser(@RequestBody User userInput) {
    User user = userRepository.findCurrentUser();
    BeanUtils.copyProperties(userInput, user);
    return userRepository.save(user);
}

If User contains sensitive fields like role, isAdmin, tenantId, creditLimit, or isVerified, the attacker may modify them.

Why this happens

Developers bind request JSON directly into domain models:

class User {
    String displayName;
    String phone;
    String role;
    Boolean isAdmin;
    String tenantId;
    BigDecimal accountBalance;
}

The framework does what it is told: it maps all matching input fields.

DTO-based prevention

Use a request DTO with only fields the user may update:

class UpdateProfileRequest {
    @Size(max = 100)
    private String displayName;

    @Pattern(regexp = "^[0-9]{8,15}$")
    private String phone;
}

Then map explicitly:

public User updateProfile(UserPrincipal currentUser, UpdateProfileRequest request) {
    User user = userRepository.findById(currentUser.getUserId());

    user.setDisplayName(request.getDisplayName());
    user.setPhone(request.getPhone());

    return userRepository.save(user);
}

Now even if the attacker sends:

{
  "displayName": "Kumar",
  "role": "admin",
  "isVerified": true
}

those fields are ignored because they do not exist in the DTO.

DTO pattern benefits

Benefit Explanation
Allow-listing Only approved fields exist in the request model
Separation of concerns API input model is separate from database model
Validation Input validation lives close to API contract
Safer evolution Adding internal fields does not automatically expose them
Clear review boundary Security reviewers can inspect DTOs for assignable fields

Model answer

Mass assignment is an input-binding vulnerability where attackers submit extra fields and the framework maps them into sensitive internal object properties. DTOs prevent it by creating an explicit allow-list of fields that the client is allowed to control. Never bind external JSON directly to persistence entities.

4. A developer argues that their GraphQL API is safe because it requires authentication. What GraphQL-specific vulnerabilities still apply?

Authentication only proves who the caller is. It does not automatically solve authorization, abuse, data exposure, or denial-of-service risks.

OWASP’s GraphQL guidance highlights controls such as depth limiting, amount limiting, pagination, timeouts, query cost analysis, and rate limiting. (OWASP Cheat Sheet Series)

GraphQL-specific risks

1. Broken object-level authorization in resolvers

Example:

query {
  user(id: "victim-user-id") {
    id
    email
    phone
    address
  }
}

The caller is authenticated, but the resolver must still check whether the caller can access that specific user object.

Bad resolver:

user: (_, args) => db.users.findById(args.id);

Good resolver:

user: (_, args, ctx) => {
  const user = db.users.findById(args.id);
  authz.check(ctx.principal, "read", user);
  return user;
};

2. Field-level authorization failure

A user may be allowed to read a profile but not sensitive fields.

query {
  employee(id: "123") {
    name
    salary
    performanceRating
    nationalId
  }
}

The resolver must enforce authorization at the field or object projection level.

3. Excessive data exposure

GraphQL allows clients to choose fields. If sensitive fields exist in the schema, a client can ask for them even if the frontend never does.

query {
  me {
    id
    name
    email
    internalRiskScore
    mfaSecret
  }
}

4. Query depth attack

query {
  user {
    friends {
      friends {
        friends {
          friends {
            friends {
              id
            }
          }
        }
      }
    }
  }
}

This can cause expensive recursive resolver execution.

5. Query complexity abuse

A shallow query can still be expensive:

query {
  searchUsers(first: 10000) {
    nodes {
      posts(first: 10000) {
        comments(first: 10000) {
          text
        }
      }
    }
  }
}

Controls:

  • Query cost calculation
  • Maximum page size
  • Timeouts
  • Resolver budgets
  • Per-user quotas

6. Batching abuse

GraphQL can batch many operations into one HTTP request. Attackers can use this for brute force or enumeration.

Example:

[
  { "query": "query { user(id: \"1\") { email } }" },
  { "query": "query { user(id: \"2\") { email } }" },
  { "query": "query { user(id: \"3\") { email } }" }
]

7. Introspection and schema discovery

Introspection can reveal:

  • Object types
  • Hidden fields
  • Deprecated fields
  • Admin mutations
  • Internal naming conventions
  • Sensitive relationships

Disabling introspection in production does not replace authorization, but it reduces attacker discovery.

8. Mutation abuse

Example:

mutation {
  updateUser(input: { id: "victim-id", role: "admin" }) {
    id
    role
  }
}

This combines BOLA and mass assignment.

Model answer

Authentication is table stakes. GraphQL still needs resolver-level authorization, field-level authorization, query depth limits, query cost analysis, pagination, batching controls, introspection governance, and sensitive-field review. Otherwise a logged-in low-privilege user can still extract or mutate data they should never access.

5. Why are API tokens in URL parameters dangerous, even over HTTPS?

API tokens in URLs are dangerous because URLs leak into many places outside the encrypted HTTPS connection.

HTTPS protects the token in transit between client and server. It does not prevent the URL from being stored, logged, cached, copied, bookmarked, or leaked after the request is made.

Example

Bad:

GET /api/download?access_token=eyJhbGciOi...

Better:

GET /api/download
Authorization: Bearer eyJhbGciOi...

Where URL tokens leak

Leak location Example
Web server logs Full request path logged
Reverse proxy logs NGINX, Envoy, ALB logs
Browser history URL saved locally
Referer header Token sent to third-party site if user clicks away
Analytics tools Page URL captured
Monitoring/APM tools Full URL indexed
Screenshots Users share URL
Bookmarks Token stored indefinitely
SIEM logs Security tools ingest token
Error reports URLs sent to crash-reporting systems
CDN logs Edge logs capture query string
Email/chat Users paste full links

Why HTTPS does not solve it

HTTPS protects:

Client <---- encrypted transit ----> Server

But after termination:

Server logs
Proxy logs
Browser history
Monitoring tools
Referrer leakage

are outside TLS protection.

Safer design

Use:

Authorization: Bearer <token>

For one-time download links, use:

  • Short expiration
  • One-time use
  • Audience binding
  • IP/device binding where appropriate
  • No long-lived bearer tokens
  • Server-side revocation
  • Token redaction in logs

Model answer

API tokens in URLs are dangerous because URLs are treated as metadata and are widely logged, cached, shared, and leaked. HTTPS protects the request in transit, but it does not stop proxies, logs, browser history, referer headers, or monitoring systems from capturing the token.

6. How would you audit an API that returns more fields than the frontend renders? What risks does this create?

This is an excessive data exposure problem. The frontend may hide fields, but attackers do not use only the frontend. They call the API directly.

Example

Frontend displays:

{
  "name": "John Tan",
  "department": "Finance"
}

But API returns:

{
  "id": "user_123",
  "name": "John Tan",
  "department": "Finance",
  "salary": 185000,
  "nationalId": "S1234567A",
  "riskScore": 92,
  "managerNotes": "Performance issue",
  "isPrivileged": true
}

Even if the UI hides those fields, the API exposed them.

Audit approach

1. Build an endpoint inventory

Sources:

  • OpenAPI / Swagger specs
  • API gateway logs
  • Frontend network traces
  • Mobile app traffic
  • Postman collections
  • GraphQL schema
  • Service code
  • Runtime traffic capture
  • API management platform

2. Compare rendered fields vs returned fields

For each frontend feature:

Field Returned by API? Rendered by UI? Sensitive? Needed?
name Yes Yes Low Yes
salary Yes No High No
nationalId Yes No High No
isAdmin Yes No Medium No
internalRiskScore Yes No High No

3. Classify sensitive fields

Look for:

  • PII
  • Credentials
  • Tokens
  • Session IDs
  • Internal flags
  • Entitlements
  • Risk scores
  • Audit notes
  • Financial data
  • Health data
  • Customer identifiers
  • Admin-only metadata

4. Test with different roles

Use:

  • Anonymous user
  • Normal user
  • Premium user
  • Support agent
  • Tenant admin
  • Cross-tenant user
  • Deactivated user
  • Read-only user

Check whether lower-privilege users receive hidden sensitive fields.

5. Inspect backend serialization

Common causes:

return userEntity;

Instead of:

return UserProfileDto.from(userEntity);

Risks often come from returning ORM entities directly.

6. Check GraphQL schema and REST DTOs

For GraphQL:

type User {
  id: ID!
  name: String!
  email: String!
  salary: Float
  mfaSecret: String
  internalRiskScore: Int
}

If sensitive fields exist in the schema, clients may request them.

For REST, review response DTOs.

Risks created

Risk Explanation
PII exposure Hidden fields may contain personal data
Privilege discovery Fields like isAdmin, role, tenantId reveal access model
Business logic abuse Internal states can help attackers manipulate flows
Compliance breach GDPR, MAS TRM, PCI, HIPAA-style obligations depending on data
Reconnaissance Internal IDs and flags help attackers plan attacks
Token leakage Some APIs accidentally return secrets
Competitive leakage Pricing, risk scores, credit limits, salaries

Remediation

  • Use response DTOs.
  • Return minimum required fields.
  • Apply field-level authorization.
  • Use role-specific projections.
  • Add contract tests for response schemas.
  • Add sensitive-field scanners in CI.
  • Do not serialize database entities directly.
  • Use API schema governance.
  • Add logging for sensitive field exposure.
  • Implement GraphQL field allowlists or persisted queries where appropriate.

Model answer

I would audit API responses by comparing actual wire responses against frontend-rendered fields and classifying every extra field by sensitivity and business need. The core risk is that the frontend is not a security boundary. If the API returns sensitive data, an attacker can read it even if the UI never displays it.

7. What makes a zombie/shadow API particularly dangerous, and how would you discover them in a large organization’s infrastructure?

A shadow API is an API that exists outside official inventory or governance. A zombie API is an old, deprecated, forgotten, or supposedly retired API that still responds to traffic.

They are dangerous because security teams often do not know they exist.

Why they are dangerous

Risk Explanation
No ownership Nobody patches or monitors it
Weak auth Old endpoints may use legacy auth
Missing authorization Older APIs may predate modern controls
No rate limiting Not onboarded to API gateway
Sensitive data exposure Old response schemas may expose too much
Forgotten dependencies Still connected to production databases
Compliance gap Not included in data inventory
Attack surface expansion Internet-exposed but unmanaged
Bypass modern controls May avoid WAF, gateway, service mesh, or IAM
Old vulnerabilities Runs outdated frameworks or libraries

Realistic example

A company migrates from:

/api/v1/customers/{id}

to:

/api/v3/customers/{id}

The frontend no longer uses v1. But v1 still exists behind:

legacy-api.company.com/api/v1/customers/123

It has weaker auth and returns:

{
  "name": "Alice",
  "email": "alice@example.com",
  "nationalId": "S1234567A",
  "creditLimit": 50000
}

That is a zombie API.

8. How does disabling GraphQL introspection improve security without breaking legitimate use?

GraphQL introspection allows clients to query the schema itself.

Example:

query {
  __schema {
    types {
      name
      fields {
        name
      }
    }
  }
}

This is useful for developers and tooling, but in production it helps attackers discover the API.

OWASP’s GraphQL guidance recommends controls such as limiting query depth, amount, cost, pagination, timeouts, and rate limiting; introspection governance is commonly used as part of reducing schema discovery in production. (OWASP Cheat Sheet Series)

What introspection can reveal

  • Types
  • Queries
  • Mutations
  • Subscriptions
  • Hidden fields
  • Deprecated fields
  • Admin operations
  • Internal object relationships
  • Naming conventions
  • Sensitive business concepts
  • Input structures
  • Enum values
  • Error shapes

Why disabling it helps

It does not make the API secure by itself, but it reduces attacker speed and visibility.

With introspection enabled, an attacker can quickly discover:

mutation {
  promoteUserToAdmin(userId: ID!)
}

or:

type User {
  id: ID!
  email: String!
  role: String!
  mfaSecret: String
  internalRiskScore: Int
}

Without introspection, attackers must infer schema from traffic, documentation leaks, JS bundles, or brute force.

How to avoid breaking legitimate use

Use environment-aware controls.

Environment Introspection setting
Local development Enabled
Dev / test Enabled with authentication
Staging Restricted to developers / VPN / admin roles
Production public API Disabled or restricted
Production internal API Restricted based on identity and network

Better production patterns

1. Persisted queries

Clients send an operation ID instead of arbitrary GraphQL.

{
  "operationId": "GetUserProfile_v3",
  "variables": {
    "userId": "123"
  }
}

Server executes only approved operations.

2. Schema registry

Developer tools use a schema registry, not production introspection.

CI/CD publishes schema → schema registry → developer tooling

3. Role-based introspection

Allow introspection only for authenticated developer/admin roles.

4. Separate internal and external schemas

Do not expose internal admin fields in public schema.

Important caveat

Disabling introspection is defense-in-depth, not a primary authorization control.

You still need:

  • Resolver authorization
  • Field-level authorization
  • Query depth limits
  • Complexity limits
  • Rate limits
  • Input validation
  • Sensitive field review

Model answer

Disabling introspection reduces schema reconnaissance in production. It prevents attackers from trivially enumerating types, fields, and mutations. It should not break legitimate use if developers rely on schema registries, persisted queries, staging introspection, or role-restricted introspection. But it is defense-in-depth, not a substitute for resolver-level authorization.

9. What is a JWT algorithm confusion attack, including alg: none, and how do you prevent it in JWT library configuration?

A JWT has three parts:

base64url(header).base64url(payload).base64url(signature)

The header contains the algorithm:

{
  "alg": "RS256",
  "typ": "JWT"
}

A JWT algorithm confusion attack happens when the server trusts the token’s alg header too much instead of enforcing the expected algorithm from server-side configuration.

OWASP’s REST security guidance specifically says not to allow unsecured JWTs using {"alg":"none"} and to ensure JWTs are integrity protected by a signature or MAC. (OWASP Cheat Sheet Series)

alg: none attack

In older or misconfigured libraries, an attacker could change:

{
  "alg": "RS256"
}

to:

{
  "alg": "none"
}

Then remove the signature and modify claims:

{
  "sub": "attacker",
  "role": "admin"
}

If the server accepts alg: none, it treats an unsigned token as valid.

RSA/HMAC confusion attack

Another classic algorithm confusion issue happens when a server expects RS256, but accepts HS256.

Intended model

With RS256:

  • Private key signs token.
  • Public key verifies token.

Attack

Attacker changes header:

{
  "alg": "HS256"
}

Then signs the token using the server’s public key as the HMAC secret.

If the server incorrectly uses the same key material and trusts the header algorithm, it may accept the forged token.

Bad verification pattern

jwt.verify(token, key);

This may be dangerous if the allowed algorithm is not explicitly restricted.

Safer configuration

jwt.verify(token, publicKey, {
  algorithms: ["RS256"],
  issuer: "https://idp.example.com",
  audience: "payments-api",
  clockTolerance: 30,
});

Prevention checklist

Control Explanation
Explicitly allow algorithms Example: only RS256, not whatever header says
Reject none Never accept unsigned JWTs for auth
Separate symmetric/asymmetric verification Do not mix HS and RS paths
Validate issuer iss must match trusted IdP
Validate audience aud must match your API
Validate expiry Enforce exp, nbf, iat
Validate key ID safely Do not allow arbitrary kid file path or URL lookup
Use JWKS allowlist Fetch keys only from trusted IdP
Cache and rotate keys safely Avoid accepting stale or attacker-controlled keys
Do not trust claims before verification Parse only after signature validation
Use mature libraries Avoid custom JWT parsing

Strong Java-style example

Algorithm algorithm = Algorithm.RSA256(rsaPublicKey, null);

JWTVerifier verifier = JWT.require(algorithm)
    .withIssuer("https://idp.example.com")
    .withAudience("payments-api")
    .acceptLeeway(30)
    .build();

DecodedJWT jwt = verifier.verify(token);

Key point: the server specifies the algorithm. The token does not get to choose.

Model answer

JWT algorithm confusion happens when the verifier lets the token header decide how the token should be verified. An attacker may switch RS256 to HS256 or none and forge claims. The fix is to configure the JWT library with a strict algorithm allowlist, reject none, separate symmetric and asymmetric verification, validate issuer and audience, and never trust claims before signature verification.

10. Design an API rate-limiting strategy for a payment API that must handle 1 million requests per day while detecting and blocking credential-stuffing and enumeration attacks.

A payment API needs more than simple request-per-minute throttling. It needs multi-dimensional, risk-aware rate limiting.

1 million requests/day averages to:

1,000,000 / 86,400 ≈ 11.6 requests per second

But traffic is not evenly distributed. We need to handle peak bursts while detecting abuse.

Design goals

Goal Explanation
Protect availability Prevent DoS and expensive abuse
Protect accounts Stop credential stuffing
Protect data Stop account/card/payment enumeration
Preserve legitimate traffic Avoid blocking real customers
Support business SLAs Payment APIs need low latency
Provide explainability SOC and fraud teams need reason codes
Fail safely Avoid catastrophic outage from limiter dependency failure

High-level architecture

flowchart TD
    A[Client] --> B[CDN / Edge WAF]
    B --> C[API Gateway]
    C --> D[Rate Limiting Layer]
    D --> E[Risk Engine]
    E --> F[Auth Service]
    F --> G[Payment API]
    G --> H[Payment Processor]

    D --> I[Redis / Distributed Counter]
    E --> J[Feature Store]
    E --> K[SIEM / Fraud Platform]
    G --> L[Audit Logs]

Layered rate limits

1. Global protection limit

Protect the whole platform.

Global payment API capacity: 2,000 RPS
Soft alert: 1,200 RPS
Hard shed: 1,800 RPS

Used for volumetric attacks.

2. Per-IP limit

Useful but not sufficient because attackers use botnets and proxies.

Unauthenticated:
- 60 requests/minute/IP
- 500 requests/hour/IP

Authenticated:
- 300 requests/minute/IP, adjusted by client type

3. Per-account limit

Prevents one account from being abused.

Payment creation:
- 10 payment attempts/minute/account
- 100 payment attempts/day/account

4. Per-device/session limit

Useful for credential stuffing and automation detection.

Login attempts:
- 5 failed attempts/minute/device
- 20 failed attempts/hour/device

5. Per-credential limit

Credential stuffing requires rate-limiting by username/email, not just IP.

Failed login for same username:
- 5 failures/15 minutes
- Step-up or temporary lock

But avoid hard lockouts that let attackers DoS users. Prefer progressive friction.

6. Per-merchant / API client limit

For B2B payment APIs:

merchant_id: merchant_abc
client_id: checkout-service
limit: 100 RPS sustained, 300 RPS burst

Use contractual limits and service tiers.

7. Per-endpoint limit

Different endpoints have different risk.

Endpoint Risk Example limit
GET /payment-methods/{id} Enumeration Strict
POST /payments Financial impact Strict + risk scoring
POST /login Credential stuffing Very strict
POST /otp/verify OTP brute force Very strict
GET /transactions Data scraping Moderate
GET /health Low Higher

Token bucket design

Use token bucket for normal traffic bursts.

merchant_id: 100 requests/sec refill
burst: 300 tokens

This allows short bursts but controls sustained abuse.

Sliding window for abuse detection

For credential stuffing, use sliding windows:

failed_login_count(username, 15 minutes)
failed_login_count(ip, 15 minutes)
failed_login_count(device_id, 15 minutes)
distinct_usernames_per_ip(10 minutes)
distinct_ips_per_username(10 minutes)

Credential stuffing detection

Credential stuffing pattern:

Many usernames attempted from same IP / ASN / device fingerprint
Same username attempted from many IPs
High failed-login ratio
Known leaked credentials
Automation-like timing
Suspicious user agents
Residential proxy / hosting ASN
No normal browsing behavior before login

Detection features

Feature Signal

Username velocity Many usernames from same source

Enumeration attack detection

Payment APIs are vulnerable to enumeration of:

  • User IDs
  • Account IDs
  • Payment IDs
  • Invoice IDs
  • Card BINs
  • Promo codes
  • Transaction references
  • Phone numbers
  • Email addresses
  • OTPs

Enumeration signals

Signal Example
High 404/403 ratio Many guessed IDs
Sequential ID access /payments/1001, /1002, /1003
High distinct object IDs One user tries many payment IDs
Cross-tenant access attempts Tenant A requesting Tenant B resources
Repeated validation failures Many card/account lookups
Low success ratio Mostly misses
Uniform timing Automation

Enumeration-specific limits

GET /payments/{paymentId}
- Max 30 distinct payment IDs/minute/account
- Max 100 distinct payment IDs/hour/account
- Alert on sequential patterns

POST /verify-account
- Max 5 failed validations/minute/user
- Max 20 failed validations/day/user
- Do not reveal whether account exists

Error response hardening

Avoid enumeration-friendly responses.

Bad:

{
  "error": "Payment ID exists but belongs to another user"
}

Better:

{
  "error": "Not found"
}

For login:

Bad:

{
  "error": "User exists but password is wrong"
}

Better:

{
  "error": "Invalid username or password"
}

Abuse-focused pseudocode

def authorize_request(request):
    identity = extract_identity(request)
    endpoint = classify_endpoint(request)
    risk = risk_engine.score(request, identity)

    keys = build_rate_limit_keys(request, identity, endpoint)

    for key, limit in keys:
        if exceeds_limit(key, limit):
            log_rate_limit_event(request, key, risk)
            return deny_or_challenge(risk)

    if endpoint == "login" and looks_like_credential_stuffing(request):
        return step_up_or_block(request)

    if endpoint == "object_read" and looks_like_enumeration(request):
        return throttle_or_block(request)

    return allow()

Model answer

For a payment API handling 1 million requests per day, I would use layered, risk-aware rate limiting: global, per-IP, per-user, per-account, per-device, per-merchant, and per-endpoint. Credential stuffing needs username, IP, device, ASN, and failure-ratio signals. Enumeration detection needs distinct-object counting, 403/404 ratios, sequential ID detection, and generic error responses. The limiter should use token buckets for normal traffic, sliding windows for abuse detection, and integrate with fraud, SIEM, and step-up authentication.

Final Summary

The common theme across all these questions is:

flowchart LR
    A[Authentication] --> B[Authorization]
    B --> C[Object-Level Checks]
    C --> D[Field-Level Controls]
    D --> E[Input Allowlisting]
    E --> F[API Inventory]
    F --> G[Abuse Detection]
    G --> H[Continuous Monitoring]

A strong senior security engineer answer is:

Authentication is only the start. Secure APIs require object-level authorization, field-level authorization, input allowlisting, minimal responses, governed API inventory, safe token handling, secure JWT verification, GraphQL-specific controls, and abuse-aware rate limiting. Most serious API breaches happen when teams assume that being logged in means being allowed to access everything behind the endpoint.

Key Takeaways

  • BOLA (IDOR) is the number one API vulnerability: always verify the authenticated user owns the accessed object by ID.
  • APIs return raw data; extra fields in responses expose PII, internal logic, and credentials even if the frontend doesn't display them.
  • Mass assignment blindly applying request body to model allows privilege escalation; use explicit DTO whitelists.
  • Rate limiting is essential: without it, any information disclosure or enumerable ID becomes a full-database-dump threat.
  • Never expose Swagger/OpenAPI, enable GraphQL introspection, or return verbose errors in production.
  • Shadow and zombie APIs (old versions, test endpoints, undocumented routes) carry full-severity vulnerabilities without the visibility of the main API.
  • JWT alg: none and weak algorithm acceptance are auth bypass bugs that exist at the library configuration level.
