Chrome Extension Development Guide

Spec: Manifest V3 (MV3) — required for new Chrome Web Store submissions since June 2024
Runtime: Chromium — Chrome, Edge, Opera, Brave
Languages: HTML, CSS, JavaScript (no server-side code runs inside the extension)

1. Core Concepts

A Chrome extension is a ZIP package (.crx) containing:

File Purpose
manifest.json Required. Declares metadata, permissions, and entry points
Background service worker Long-running event handler; no DOM access
Content scripts JS/CSS injected into web pages
Popup / Options pages HTML UI shown in the toolbar or extension settings
Icons Required at multiple sizes (16, 48, 128 px)
_locales/ Internationalization messages

Extension Contexts

Context DOM Access Network Access Chrome APIs Lifetime
Background service worker No Yes Full Event-driven; terminated when idle
Content script Yes (page DOM) Fetch only Limited subset Per page load
Popup / Options page Yes (own DOM) Yes Full While page is open
Offscreen document (MV3) Yes (hidden) Yes Limited Explicitly managed

2. Project Structure

my-extension/
├── manifest.json          # Required — extension declaration
├── background.js          # Service worker
├── content.js             # Content script
├── popup.html             # Toolbar popup UI
├── popup.js               # Popup logic
├── options.html           # Extension settings page
├── options.js
├── styles/
│   └── content.css        # CSS injected into pages
└── icons/
    ├── icon16.png
    ├── icon48.png
    └── icon128.png

3. manifest.json — MV3

{
  "manifest_version": 3,
  "name": "My Extension",
  "version": "1.0.0",
  "description": "A brief description of what this extension does.",

  "icons": {
    "16": "icons/icon16.png",
    "48": "icons/icon48.png",
    "128": "icons/icon128.png"
  },

  "action": {
    "default_popup": "popup.html",
    "default_title": "Open My Extension",
    "default_icon": {
      "16": "icons/icon16.png",
      "48": "icons/icon48.png"
    }
  },

  "background": {
    "service_worker": "background.js",
    "type": "module"
  },

  "content_scripts": [
    {
      "matches": ["https://example.com/*"],
      "js": ["content.js"],
      "css": ["styles/content.css"],
      "run_at": "document_idle"
    }
  ],

  "permissions": ["storage", "tabs", "activeTab"],

  "host_permissions": ["https://api.example.com/*"],

  "options_page": "options.html",

  "web_accessible_resources": [
    {
      "resources": ["images/*.png"],
      "matches": ["https://example.com/*"]
    }
  ]
}

Key manifest fields

Field Notes
manifest_version Must be 3 for new submissions
action Toolbar button; omit if no popup is needed
background.service_worker Single JS file; supports ES modules with "type": "module"
content_scripts[].matches URL patterns — use specific patterns, not <all_urls>
content_scripts[].run_at document_start, document_end, or document_idle (default)
permissions Chrome API permissions — see Permission Reference below
host_permissions URL patterns the extension can fetch from or inject into
web_accessible_resources Files inside the extension that web pages can load

Permission Reference

API Permissions (permissions field)

These grant access to Chrome extension APIs. Declared in the permissions array in manifest.json.

Permission What It Grants Sensitivity
activeTab Temporary access to the currently active tab when user invokes the extension (click, keyboard shortcut) — preferred over broad host permissions Low
alarms Create and listen to chrome.alarms for scheduled/periodic tasks Low
bookmarks Read, create, modify, and delete bookmarks Med
clipboardRead Read the system clipboard via document.execCommand('paste') High
clipboardWrite Write to the system clipboard Low
contentSettings Read and modify per-site content settings (images, JavaScript, cookies, etc.) Med
contextMenus Add items to the browser's right-click context menu Low
cookies Read, set, and delete cookies for any URL matching host_permissions High
debugger Attach to tabs as a debugger; access the Chrome DevTools Protocol High
declarativeContent Show the page action based on page content without reading page content Low
declarativeNetRequest Block or modify network requests using static/dynamic rules Med
declarativeNetRequestWithHostAccess Same as above but also allows redirecting requests (requires host permissions) High
downloads Manage downloads — monitor, open, remove download items Med
favicon Access the favicon URLs of pages via chrome.tabs Low
geolocation Access the user's geographic location High
history Read and modify the browser's browsing history High
identity Get OAuth2 access tokens via the Chrome identity API High

unlimitedStorage Remove the 10 MB quota cap for chrome.storage.local and IndexedDB Low

webRequest Observe (but not block) network requests — MV3 non-blocking only High

Host Permissions (host_permissions field)

Host permissions control which URLs the extension can access — inject content scripts into, make cross-origin fetch requests to, or read cookies from.

Pattern Matches
https://example.com/* All pages on example.com over HTTPS
https://*.example.com/* All subdomains of example.com over HTTPS
https://example.com/api/* Only paths under /api/ on example.com
https://*/* All HTTPS URLs (broad — triggers manual review)
http://*/* All HTTP URLs (broad + insecure)
<all_urls> Every URL including HTTP, HTTPS, file, and ftp (maximum scope — almost always rejected or flagged)

Prefer activeTab over host permissions. activeTab gives temporary access to the current tab only when the user explicitly invokes the extension, without requiring broad standing host permissions.

Optional Permissions

Permissions can be requested at runtime rather than at install time — users grant them on demand, reducing install-time friction:

// manifest.json
{
  "optional_permissions": ["history", "bookmarks"],
  "optional_host_permissions": ["https://*.example.com/*"]
}

// Request optional permission at runtime
const granted = await chrome.permissions.request({
  permissions: ["history"],
  origins: ["https://*.example.com/*"],
});

if (granted) {
  // Permission was granted by the user
}

// Check if permission is currently held
const hasIt = await chrome.permissions.contains({ permissions: ["history"] });

// Release a permission no longer needed
await chrome.permissions.remove({ permissions: ["history"] });

4. Background Service Worker

The service worker replaces MV2's persistent background page. It is event-driven and may be terminated at any time when idle.

// background.js

// Listen for extension install
chrome.runtime.onInstalled.addListener(({ reason }) => {
  if (reason === chrome.runtime.OnInstalledReason.INSTALL) {
    console.log("Extension installed");
    chrome.storage.local.set({ count: 0 });
  }
});

// Listen for toolbar button click (when no popup is defined)
chrome.action.onClicked.addListener((tab) => {
  console.log("Action clicked on tab:", tab.url);
});

// Listen for messages from content scripts or popups
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.type === "GET_DATA") {
    chrome.storage.local.get("count", ({ count }) => {
      sendResponse({ count });
    });
    return true; // Keep message channel open for async sendResponse
  }
});

// Alarm API — replaces setInterval (service workers cannot use setInterval reliably)
chrome.alarms.create("periodic-check", { periodInMinutes: 1 });
chrome.alarms.onAlarm.addListener((alarm) => {
  if (alarm.name === "periodic-check") {
    doPeriodicWork();
  }
});

Service Worker Gotchas

Issue Explanation Fix
setInterval / setTimeout unreliable Service worker is killed when idle; timers stop Use chrome.alarms API instead
Global state lost Variables reset when service worker restarts Use chrome.storage for persistence
Async sendResponse Must return true in onMessage listener to keep channel open Always return true when sending response asynchronously
No DOM Cannot access document or window Use offscreen documents for DOM tasks

5. Content Scripts

Content scripts are injected into matching web pages. They run in an isolated world — separate JS environment from the page, but sharing the same DOM.

// content.js

// Read and modify the page DOM
const heading = document.querySelector("h1");
if (heading) {
  heading.style.outline = "2px solid red";
}

// Send a message to the background service worker
chrome.runtime.sendMessage({ type: "PAGE_LOADED", url: location.href });

// Listen for messages from the popup or background
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.type === "HIGHLIGHT") {
    document.querySelectorAll(message.selector).forEach((el) => {
      el.style.backgroundColor = "yellow";
    });
    sendResponse({ done: true });
  }
});

Programmatic Injection (from popup or background)

// Inject a content script on demand (requires activeTab or host permission)
await chrome.scripting.executeScript({
  target: { tabId: tab.id },
  files: ["content.js"],
});

// Or inject an inline function
await chrome.scripting.executeScript({
  target: { tabId: tab.id },
  func: () => {
    document.body.style.background = "lightblue";
  },
});

Content Script vs Page Script Communication

Content scripts cannot access the page's JS variables. To bridge the two worlds:

// content.js — inject a <script> into the page context
const script = document.createElement("script");
script.src = chrome.runtime.getURL("page-script.js");
script.onload = () => script.remove();
document.documentElement.appendChild(script);

// Listen for messages from the page context
window.addEventListener("message", (event) => {
  if (event.source !== window || !event.data?.type) return;
  if (event.data.type === "FROM_PAGE") {
    chrome.runtime.sendMessage(event.data);
  }
});

// page-script.js (loaded into page context via web_accessible_resources)
window.postMessage({ type: "FROM_PAGE", payload: "hello" }, "*");

6. Popup

<!-- popup.html -->
<!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <style>
      body {
        width: 300px;
        padding: 12px;
        font-family: sans-serif;
      }
    </style>
  </head>
  <body>
    <h2>My Extension</h2>
    <p id="status">Loading...</p>
    <button id="btn-action">Run Action</button>
    <script src="popup.js"></script>
  </body>
</html>

// popup.js

// Get the active tab
const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });

// Send a message to the content script on the active tab
chrome.tabs.sendMessage(
  tab.id,
  { type: "HIGHLIGHT", selector: "h2" },
  (response) => {
    document.getElementById("status").textContent = response?.done
      ? "Done"
      : "Error";
  },
);

// Send a message to the background service worker
chrome.runtime.sendMessage({ type: "GET_DATA" }, ({ count }) => {
  document.getElementById("status").textContent = `Count: ${count}`;
});

// React to button click
document.getElementById("btn-action").addEventListener("click", async () => {
  await chrome.scripting.executeScript({
    target: { tabId: tab.id },
    func: () => alert("Hello from extension!"),
  });
});

7. Storage API

Always use chrome.storage (not localStorage) — it works in service workers, persists across restarts, and can be synced across devices.

// Write
await chrome.storage.local.set({ key: "value", count: 42 });

// Read
const { key, count } = await chrome.storage.local.get(["key", "count"]);

// Remove
await chrome.storage.local.remove("key");

// Clear all
await chrome.storage.local.clear();

// Sync storage (shared across user's signed-in Chrome instances, 100KB limit)
await chrome.storage.sync.set({ theme: "dark" });
const { theme } = await chrome.storage.sync.get("theme");

// Listen for changes
chrome.storage.onChanged.addListener((changes, area) => {
  for (const [key, { oldValue, newValue }] of Object.entries(changes)) {
    console.log(`${area}::${key}: ${oldValue} → ${newValue}`);
  }
});
Storage type Quota Synced Available in service worker
chrome.storage.local 10 MB No Yes
chrome.storage.sync 100 KB total, 8 KB per item Yes (Chrome account) Yes
chrome.storage.session 10 MB No Yes (cleared on browser restart)

8. Message Passing

Content Script → Background

// content.js
const response = await chrome.runtime.sendMessage({
  type: "FETCH_DATA",
  url: "https://api.example.com/data",
});
console.log(response.data);

// background.js
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.type === "FETCH_DATA") {
    fetch(message.url)
      .then((r) => r.json())
      .then((data) => sendResponse({ data }))
      .catch((err) => sendResponse({ error: err.message }));
    return true; // Required for async response
  }
});

Background → Content Script (specific tab)

// background.js
chrome.tabs.sendMessage(
  tabId,
  { type: "UPDATE", payload: "hello" },
  (response) => {
    console.log("Content script responded:", response);
  },
);

Long-lived connections (ports)

// content.js
const port = chrome.runtime.connect({ name: "my-channel" });
port.postMessage({ type: "START" });
port.onMessage.addListener((msg) => console.log("From background:", msg));

// background.js
chrome.runtime.onConnect.addListener((port) => {
  if (port.name !== "my-channel") return;
  port.onMessage.addListener((msg) => {
    port.postMessage({ type: "ACK", received: msg.type });
  });
});

10. Network Requests (declarativeNetRequest)

MV3 replaces webRequest blocking with the declarative API for performance and privacy. Background scripts can still use fetch.

// manifest.json
{
  "permissions": ["declarativeNetRequest"],
  "declarative_net_request": {
    "rule_resources": [
      {
        "id": "ruleset_1",
        "enabled": true,
        "path": "rules.json"
      }
    ]
  }
}

// rules.json
[
  {
    "id": 1,
    "priority": 1,
    "action": { "type": "block" },
    "condition": {
      "urlFilter": "*://*.ads.example.com/*",
      "resourceTypes": ["script", "image", "xmlhttprequest"]
    }
  }
]

Dynamic rules (at runtime):

await chrome.declarativeNetRequest.updateDynamicRules({
  addRules: [
    {
      id: 100,
      priority: 1,
      action: { type: "block" },
      condition: { urlFilter: "*://tracker.example.com/*" },
    },
  ],
  removeRuleIds: [],
});

11. Offscreen Documents (MV3)

For tasks requiring a DOM in the background (audio playback, clipboard, canvas), use offscreen documents:

// background.js
await chrome.offscreen.createDocument({
  url: "offscreen.html",
  reasons: [chrome.offscreen.Reason.CLIPBOARD],
  justification: "Reading clipboard text",
});

// Send a message to the offscreen document
chrome.runtime.sendMessage({ type: "READ_CLIPBOARD" });

// offscreen.js
chrome.runtime.onMessage.addListener(async (msg) => {
  if (msg.type === "READ_CLIPBOARD") {
    const text = await navigator.clipboard.readText();
    chrome.runtime.sendMessage({ type: "CLIPBOARD_DATA", text });
  }
});

12. Loading and Testing Locally

  1. Open Chrome and navigate to chrome://extensions
  2. Enable Developer mode (toggle, top right)
  3. Click Load unpacked and select your extension folder
  4. The extension appears in the toolbar and extensions list
  5. Click the refresh icon after making code changes
  6. Use Inspect views: service worker link to open DevTools for the background worker
  7. Right-click the popup → Inspect to debug popup JS

Useful DevTools tips

Task How
Debug background service worker chrome://extensions → click "service worker" link
Debug content script Open DevTools on the target page → Sources → Content scripts
View extension storage DevTools → Application → Extension Storage
Reload extension from CLI Not built-in; use chrome.runtime.reload() from background or a build tool

13. Build Tooling

For larger extensions, use a bundler:

Vite (recommended for MV3):

npm create vite@latest my-extension -- --template vanilla
npm install -D @crxjs/vite-plugin

// vite.config.js
import { defineConfig } from "vite";
import { crx } from "@crxjs/vite-plugin";
import manifest from "./manifest.json";

export default defineConfig({
  plugins: [crx({ manifest })],
});

Webpack:

npm install -D webpack webpack-cli copy-webpack-plugin

TypeScript support: Add tsconfig.json and change entry files to .ts; all Chrome APIs are typed via @types/chrome:

npm install -D typescript @types/chrome

14. Publishing to Chrome Web Store

  1. Create a developer account at chrome.google.com/webstore/devconsole ($5 one-time fee)
  2. ZIP the extension (do not include node_modules, .git, source maps in production):
    bash zip -r my-extension.zip . -x "*.git*" -x "node_modules/*"
  3. Upload the ZIP in the Developer Dashboard → New Item
  4. Fill in store listing: description, screenshots (1280×800 or 640×400), category, privacy policy URL
  5. Declare permissions with justifications — the review team checks that requested permissions match declared functionality
  6. Submit for review — new extensions typically reviewed in 1–3 business days

Review tips

  • Use the minimum necessary permissions; overly broad permissions (<all_urls>, cookies, history) trigger manual review
  • Include a privacy policy if the extension handles any user data
  • Remote code execution is prohibited: all JS must be bundled (no eval, no external script tags loading code from remote URLs)
  • Single purpose policy: the extension must have one clear, primary function

15. Security Best Practices for Developers

Practice Why
Declare minimum permissions Each permission is attack surface if the extension is compromised via supply chain
Never use eval() or new Function() CSP in MV3 blocks it by default; also a XSS risk in extension pages
Sanitize DOM manipulation Use textContent instead of innerHTML to avoid XSS in extension pages
Validate all messages Check sender.origin or sender.id in onMessage listeners
Store secrets server-side Never bundle API keys in extension source — extension ZIP is publicly downloadable
Use https:// only for external requests Never fetch over HTTP from background or content scripts
Enable CSP in extension pages Add to manifest.json: "content_security_policy": { "extension_pages": "script-src 'self'; object-src 'self'" }
Scope host permissions tightly https://api.example.com/* instead of https://*/*
