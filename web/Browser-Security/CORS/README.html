CORS — Cross-Origin Resource Sharing

Category: Browser Security — Controlled SOP Relaxation
Specification: Fetch Standard (WHATWG), W3C CORS
RFC: RFC 6454 (origins)
Applies To: All browsers; all web APIs that are called cross-origin

Overview

CORS is the browser's mechanism for allowing servers to declare which foreign origins may read their responses. Same-Origin Policy blocks cross-origin reads by default; CORS is the controlled exception. When a page at app.example.com calls api.example.com/data, the browser negotiates permission via CORS headers before exposing the response to JavaScript.

At Google-scale, CORS is the gating mechanism for every public API, cross-service fetch, CDN asset load, and analytics integration. Misconfiguring CORS is one of the most common ways production systems accidentally expose sensitive data or authenticated resources to untrusted origins. A single bad wildcard or un-validated origin reflection can turn a correctly-built backend into an open data tap for any attacker-controlled page.

How It Works

Core Mechanics

CORS uses HTTP headers to establish permission. The key request header is Origin, and the key response header is Access-Control-Allow-Origin. The browser enforces the match; the server only provides the declaration.

Key CORS HTTP Headers:

Header Direction Purpose
Origin Request The origin making the cross-origin request
Access-Control-Allow-Origin Response Origins allowed to read the response
Access-Control-Allow-Credentials Response Whether cookies and auth headers are allowed
Access-Control-Allow-Methods Response (preflight) Methods permitted for cross-origin access
Access-Control-Allow-Headers Response (preflight) Request headers the JS is allowed to set
Access-Control-Expose-Headers Response Response headers JS is allowed to read
Access-Control-Max-Age Response (preflight) How long to cache preflight result
Access-Control-Request-Method Preflight request Method the actual request will use
Access-Control-Request-Headers Preflight request Custom headers the request will send

Simple vs Preflighted Requests

Not every cross-origin request requires a preflight OPTIONS check:

Simple requests (no preflight) are sent directly when ALL of:

  • Method is GET, POST, or HEAD
  • Headers are only Accept, Accept-Language, Content-Language, Content-Type
  • Content-Type is only application/x-www-form-urlencoded, multipart/form-data, or text/plain

Preflighted requests require an OPTIONS check first when:

  • Method is PUT, PATCH, DELETE, or any non-simple method
  • Custom headers like Authorization, X-Custom-Header are set
  • Content-Type is application/json

Preflighted CORS Request

sequenceDiagram
    participant JS as JS on app.example.com
    participant Browser as Browser CORS Layer
    participant API as api.example.com

    JS->>Browser: fetch with DELETE method + Authorization header
    Browser->>API: OPTIONS /resource\nOrigin: https://app.example.com\nAccess-Control-Request-Method: DELETE
    API-->>Browser: 204 No Content\nAccess-Control-Allow-Origin: https://app.example.com\nAccess-Control-Allow-Methods: DELETE\nAccess-Control-Max-Age: 86400
    Browser->>Browser: Preflight approved, cache result
    Browser->>API: DELETE /resource\nOrigin: https://app.example.com\nAuthorization: Bearer token
    API-->>Browser: 200 OK\nAccess-Control-Allow-Origin: https://app.example.com
    Browser-->>JS: Response body exposed

    H -- Yes --> J[JS can read response]

Credentials And CORS

By default, CORS requests do not include cookies or HTTP authentication. To include credentials:

  1. JS must set credentials: 'include' on the fetch
  2. Server must return Access-Control-Allow-Credentials: true
  3. Server must not use Access-Control-Allow-Origin: * — it must specify the exact origin

This triple requirement exists because credentials give substantial power. If the server allowed * with credentials, any attacker page could silently make authenticated requests to your API.

Null Origin

Requests from file://, sandboxed iframes with no allow-same-origin, and redirected cross-origin requests sometimes have Origin: null. A server that responds with Access-Control-Allow-Origin: null will allow any sandboxed page to access it, including attacker-controlled sandboxed iframes.

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Wildcard with credentials Never combine * with credentials; always specify exact origin CORS middleware config
2 Reflecting Origin verbatim Validate Origin against a hard-coded allowlist before echoing Server middleware / API gateway
3 Overly broad regex Use exact string matching, not regex on origin allowlist; include full origin with scheme and port CORS allowlist implementation
4 Null origin trust Never add null to the allowed origins list CORS middleware
5 Missing Vary header Always return Vary: Origin when Access-Control-Allow-Origin is dynamically set CDN and origin server
6 Exposing sensitive headers Audit Access-Control-Expose-Headers to include only non-sensitive response headers API design review
7 Over-permissive methods Only allow methods actually needed; combine with auth at the endpoint level CORS + endpoint auth
8 Preflight bypass Do not rely solely on CORS for authorization; always apply auth checks on the actual request Server-side auth middleware
9 CORS on internal services Internal services should not have CORS configured at all unless they legitimately serve browsers Network design + CORS policy
10 Subdomain takeover Audit all allowed origins for abandoned subdomains; enforce subdomain hygiene Origin allowlist + domain monitoring

Top 3 Critical Mitigations

  1. Never reflect the Origin header without an allowlist check. This is the single most common critical CORS misconfiguration in production. An allowlist must be a static list of exact strings (scheme + host + port), validated on every request. Even a regex shortcut can be bypassed by an attacker who registers evil-yourcompany.com.
  2. Add Vary: Origin to all dynamically CORS-enabled responses. Without this, CDNs and shared caches will serve the cached CORS headers from one origin's request to another origin's request, effectively either locking out or unlocking origins incorrectly. This is a production-wide cache poisoning bug.
  3. Never combine wildcard with credentials; enforce auth server-side. CORS does not protect API endpoints on its own. Even correct CORS prevents the browser from exposing the response to JS, but the server still processed the request. Authorization checks must live server-side regardless of CORS configuration.

Key Takeaways

  • CORS is a controlled relaxation of Same-Origin Policy via HTTP response headers; the browser enforces it, not the server.
  • Reflecting the Origin header without an allowlist is a critical misconfiguration that defeats SOP for the endpoint.
  • Wildcard (*) must never be combined with Access-Control-Allow-Credentials: true.
  • Always set Vary: Origin when CORS headers are dynamically generated to prevent CDN cache poisoning.
  • CORS does not replace server-side authorization; the server still processes the request regardless of CORS outcome.
  • Preflight caching with Access-Control-Max-Age reduces latency at scale but must not be used to skip authorization on actual requests.
