SameSite Cookie Attribute

Category: Browser Security — CSRF Defense, Cookie Scoping
Applies To: All browser-based web applications
Related: CSRF, Session Management, OAuth redirect flows

What Is SameSite?

SameSite is a cookie attribute that controls whether the browser sends a cookie on cross-site requests. It is the primary declarative CSRF defense at the cookie level — it tells the browser "only send this cookie when the request origin matches (or is close enough to) the site that set it."

Without SameSite, any site on the internet can trigger requests to your server and the browser will automatically attach your cookies. That is the root cause of Cross-Site Request Forgery (CSRF).

Same-Site vs Same-Origin — Key Distinction

SameSite does not use origin matching. It uses registrable domain (eTLD+1) matching.

Pair Same-Origin? Same-Site?
app.example.com vs api.example.com No (different host) Yes (same eTLD+1: example.com)
example.com vs example.com (HTTP vs HTTPS) No (different scheme) Yes (schemeful: No — see below)
example.com vs attacker.com No No
example.co.uk vs other.co.uk No No (.co.uk is a public suffix)

Schemeful Same-Site (modern browsers): HTTP and HTTPS versions of the same host are now treated as cross-site for SameSite purposes. http://example.com → https://example.com is cross-site.

The Three Values

SameSite=Strict

Cookie is sent only on same-site requests — no exceptions, not even top-level navigations.

Set-Cookie: session=abc; SameSite=Strict; Secure; HttpOnly

Sent on:

  • Any request originating from the same site (same eTLD+1)

NOT sent on:

  • User clicks a link from google.com → bank.com (cross-site navigation, even GET)
  • Any cross-site form submission
  • Any cross-site AJAX/fetch

Implication: If a user has bank.com open and clicks a link to bank.com/dashboard from an email, the cookie will not be sent on the first request. The user lands on the page as if logged out, then gets redirected to login (where the cookie is then sent as same-site). This "double-click" friction breaks OAuth/OIDC redirect flows and email confirmation links.

Use when: The resource never needs to be linked to from external sites in an authenticated state. Admin panels, internal tools.

SameSite=Lax

Cookie is sent on same-site requests plus safe top-level cross-site navigations.

Set-Cookie: session=abc; SameSite=Lax; Secure; HttpOnly

The rule: send the cookie if both conditions are true:

  1. The request is a top-level navigation (the browser's address bar changes — not a subresource load)
  2. The HTTP method is safe (GET or HEAD — no intended side effects per RFC 7231)

Decision table:

Scenario Method Top-level nav? Cookie sent?
User clicks <a href="https://bank.com/dashboard"> from Gmail GET Yes Yes
User types URL directly in browser GET Yes Yes
HTML <form method="POST" action="https://bank.com/transfer"> on evil.com POST Yes No
fetch("https://bank.com/api/data") from evil.com JS GET No (subresource) No
<img src="https://bank.com/pixel"> on evil.com GET No (subresource) No
<iframe src="https://bank.com/widget"> on evil.com GET No (subresource) No
AJAX PUT/DELETE from evil.com PUT/DELETE No No
<form method="GET" action="https://bank.com/search"> on evil.com GET Yes Yes ⚠️

⚠️ Cross-site GET form submission does send the Lax cookie. This is intentional (search forms, link-style navigation), but means GET endpoints must not have state-changing side effects.

Browser default: Since Chrome 80 (2020) and Firefox, Lax is the default when SameSite is not specified.

Use when: Most authenticated web applications. Good balance between CSRF protection and usability.

SameSite=None

Cookie is sent on all requests, including cross-site subresource loads. Requires the Secure flag (HTTPS only). Browsers reject SameSite=None without Secure.

Set-Cookie: tracker=xyz; SameSite=None; Secure

Sent on:

  • Everything: cross-site navigations, subresource loads, AJAX, iframes, forms

Use when:

  • Third-party widgets embedded in other sites (payment iframes, comment boxes)
  • Federated authentication using <iframe> or cross-site redirects
  • Cross-site API access where the calling site is trusted (though OAuth tokens are better here)
  • CDN or analytics cookies that must work across sites

Risk: No CSRF protection from the SameSite attribute alone. Must use CSRF tokens or other mitigations.

Concrete Attack Scenarios

Scenario 1 — Classic CSRF (Lax blocks this)

Attacker hosts evil.com with:

<form id="f" method="POST" action="https://bank.com/transfer">
  <input name="to" value="attacker_account" />
  <input name="amount" value="10000" />
</form>
<script>
  document.getElementById("f").submit();
</script>
  • Victim visits evil.com while logged into bank.com
  • Browser sends POST to bank.com/transfer
  • With SameSite=Lax: Cookie is NOT sent (cross-site POST). Attack fails.
  • With no SameSite (legacy): Cookie IS sent. Attack succeeds.

Scenario 2 — GET-based CSRF (Lax does NOT fully protect)

If bank.com/transfer?to=attacker&amount=10000 is a state-changing GET endpoint:

<!-- On evil.com -->
<img src="https://bank.com/transfer?to=attacker&amount=10000" />
  • <img> is a subresource load — Lax cookie NOT sent. Safe.

But if the attacker constructs a top-level GET navigation:

<a href="https://bank.com/transfer?to=attacker&amount=10000"
  >Click here for free money</a
>
  • Top-level GET navigation — Lax cookie IS sent. Attack succeeds if endpoint has side effects.

Lesson: SameSite=Lax does not protect state-changing GET endpoints. GET must be idempotent.

Scenario 3 — Strict breaks OAuth login

Flow: User is on myapp.com, clicks "Login with Google". Google redirects back to:

https://myapp.com/auth/callback?code=xyz
  • This redirect is a top-level GET cross-site navigation
  • With SameSite=Strict: The myapp.com session cookie is NOT sent on this callback request. The server cannot associate the callback with the pending login state.
  • With SameSite=Lax: Cookie IS sent. OAuth works correctly.

Lesson: SameSite=Strict on session cookies can break OAuth/OIDC and any flow involving redirects from external origins.

Scenario 4 — Third-party iframe (None required)

payments.com embeds a checkout widget in shop.com:

<!-- On shop.com -->
<iframe src="https://payments.com/checkout/widget"></iframe>
  • payments.com needs to read its own session cookie inside the iframe
  • The iframe request is a cross-site subresource load
  • Lax/Strict: Cookie NOT sent. Widget shows as logged out.
  • None + Secure: Cookie sent. Widget works.

SameSite and CSRF Token Comparison

Defense What It Does Covers GET side-effects? Works Without JS?
SameSite=Lax Block cross-site non-safe requests No Yes
SameSite=Strict Block all cross-site requests Yes Yes
CSRF token (synchronizer) Validate hidden token in request body Yes (if in query param too) Yes
Double Submit Cookie Compare cookie vs request param token Yes Yes
Origin/Referer check Validate source origin server-side Yes Yes

Best practice: layer defenses — use SameSite=Lax or Strict as a first line, CSRF tokens as a second line.

Lax 2-Minute Window (Chrome Only)

Chrome has a special exception for cookies set without a SameSite attribute: they are treated as Lax but for the first 2 minutes after being set, they are also sent on cross-site POST requests. This was a compatibility hack to avoid breaking OAuth POST-based flows during the rollout of Lax-by-default.

Implication: An attacker can time a CSRF attack within 2 minutes of a victim logging in to exploit this window against sites that still rely on default (no SameSite) cookies.

Fix: Always explicitly set SameSite=Lax or SameSite=Strict. Explicit SameSite=Lax does NOT have the 2-minute exception.

Request Type Reference

Cross-site request types and whether SameSite=Lax sends the cookie:

Top-level navigations (address bar changes):
  <a href="...">               GET   → Lax: YES  | Strict: NO
  window.location.href = ...   GET   → Lax: YES  | Strict: NO
  <form method="GET">          GET   → Lax: YES  | Strict: NO
  <form method="POST">         POST  → Lax: NO   | Strict: NO
  window.open(...)             GET   → Lax: YES  | Strict: NO

Subresource / embedded:
  <img src="...">              GET   → Lax: NO   | Strict: NO
  <script src="...">           GET   → Lax: NO   | Strict: NO
  <iframe src="...">           GET   → Lax: NO   | Strict: NO
  fetch(...) / XHR             ANY   → Lax: NO   | Strict: NO
  CSS background-image         GET   → Lax: NO   | Strict: NO

Quick-Reference Decision Guide

You are building... Recommended SameSite
Standard session cookie for a web app Lax
Admin panel / internal tool Strict

API-only service (tokens in headers) N/A — don't use cookies
Federated SSO with cross-site POST binding (SAML) None; Secure + CSRF token
