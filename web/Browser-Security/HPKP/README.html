HPKP — HTTP Public Key Pinning

Category: Browser Security — Certificate Binding (Deprecated)
RFC: RFC 7469
Status: Deprecated and removed by Google Chrome in Chrome 72 (January 2019). Removed from Firefox in Firefox 72.
Replaced By: Certificate Transparency (CT) + CAA DNS Records

Overview

HTTP Public Key Pinning was a browser security mechanism that allowed websites to declare which specific public keys (or CA chains) should be trusted for their domain. The idea: if a rogue CA issued a fraudulent certificate for bank.com, a browser with a pin for bank.com would reject it because the rogue certificate would not match the pinned keys. HPKP was the HTTP-header-based successor to certificate pinning concepts used in apps.

Google Chrome deprecated and removed HPKP in 2018 because it created an unacceptably high risk of site operator self-induced DoS. One operator misconfiguration — a wrong pin, an unrenewed backup key — made a site completely unreachable for all users for the entire max-age period with no recovery path. This happened in production multiple times. The risk-to-benefit ratio was unsustainable at scale. Certificate Transparency provides the mis-issuance detection benefit without the catastrophic self-DoS risk.

This note is preserved for historical understanding, comparison with CT, and historical context.

How It Works

Core Mechanics

When a browser received an HSTS header, it stored a pin set — one or more public key hashes — and a validity period. On every future visit, the browser computed the hash of each certificate in the delivered chain and checked whether at least one matched the stored pin set. If no match was found, the browser refused the connection hard, with no user override.

Header:

Public-Key-Pins: pin-sha256="<base64-hash-1>"; pin-sha256="<base64-hash-2>"; max-age=5184000; includeSubDomains; report-uri="https://example.com/hpkp-report"
Directive Purpose
pin-sha256 Base64 SHA-256 hash of a certificate's SubjectPublicKeyInfo (SPKI) structure
max-age Seconds to remember the pin (up to 60 days recommended; longer is dangerous)
includeSubDomains Apply pins to all subdomains
report-uri Endpoint to receive pin failure reports (also Public-Key-Pins-Report-Only)

At least two pins are required: a live pin (matching the current cert or CA in the chain) and a backup pin (for a key not currently in use, stored securely offline). This is because if you only pin one key and lose it, you can never recover.

The Self-DoS Problem

The attack surface of HPKP was not from malicious outsiders — it was the site operators themselves:

  1. Certificate renewal that changes the key pair without updating pins → All users locked out
  2. CDN or load balancer change that uses a different CA path → Pin mismatch for path
  3. Backup pin stored insecurely alongside live key (both compromised simultaneously)
  4. max-age set too high → Long lockout window if anything goes wrong

This failure mode is called HPKP Suicide (accidental self-lockout) or HPKP Ransom (attacker pins your domain with a high max-age via XSS that causes your 200 response to set a bad pin header).

What Can Go Wrong

# Failure Root Cause Impact Likelihood
1 Key rotation without updating pins Cert renewed with new key pair, old pin no longer matches Entire user base locked out for max-age duration High
2 Loss of backup key Backup pin key is lost or compromised; live key rotates; no valid pin exists Site permanently unreachable until max-age expires Med
3 HPKP Ransom via XSS Attacker exploits XSS to inject a response header with attacker's pin Site locked out for max-age, holding domain hostage Med
4 Including SubDomains carelessly Subdomain loses TLS or changes key All subdomains unreachable to pinned browsers Med
5 Over-long max-age Setting max-age to 1+ year means any mistake has 1-year impact Extremely long service disruption High
6 Pinning to leaf cert not CA key Leaf cert rotations happen more frequently; must re-pin every renewal Frequent pin management overhead; outage if missed High
7 Backup key disclosed with live key Both stored together, both compromised Pin protection lost; rogue cert bypass possible Med

Mitigation Controls (Historical, For Awareness)

# Failure What Was Required Notes
1 Key rotation Pin to CA public key, not leaf cert; rotate backup pin before live pin Possible but complex
2 Backup key loss Store backup key in hardware HSM; never co-locate with live key Operational requirement
3 HPKP Ransom Strict CSP to prevent XSS that injects response headers Defense-in-depth
4 includeSubDomains Only use after confirming all subdomains have valid, pinned TLS Pre-deployment audit
5 Over-long max-age Keep max-age at 60 days or less Conservative operational policy
6 Leaf pinning Pin intermediate or root CA public key Design guidance
7 Backup key exposure Separate key storage with HSM and separate access controls Key management

Why CT Is Better

Certificate Transparency achieves the same goal (detecting rogue CA-issued certificates) without any of the self-DoS risk:

  • If a CA issues a fraudulent cert, it must be logged in CT → you detect it via monitoring
  • Your own certificate renewal has no impact on CT → no lockout risk
  • CT is automatic and managed by CAs → operational overhead near zero for site operators

Key Takeaways

  • HPKP was a browser header that forced TLS connections to match pre-declared public key hashes.
  • Deprecated by Chrome in 2019 because operational mistakes caused catastrophic self-induced DoS; no recovery path.
  • HPKP Suicide (accidental misconfiguration lockout) and HPKP Ransom (attacker-injected bad pin via XSS) were fatal design flaws.
  • Certificate Transparency replaces HPKP for the rogue CA use case — same detection, zero self-DoS risk.
  • Certificate pinning is still relevant in mobile apps (where the app controls the trust store) but not in browsers.
  • Modern browser defense against rogue CA issuance is: HSTS + CT + CAA DNS + monitoring.
