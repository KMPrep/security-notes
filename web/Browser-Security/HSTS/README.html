HSTS — HTTP Strict Transport Security

Category: Browser Security — Transport Enforcement
RFC: RFC 6797
Applies To: All HTTPS-serving origins; preload list maintained by browser vendors

Overview

HSTS is a security policy header that instructs browsers to access a site exclusively over HTTPS, for a declared period of time. Once a browser has seen an HSTS header from a site, it will internally rewrite any future http:// requests to https:// before sending them to the network, and will refuse to connect if the TLS certificate is invalid rather than presenting a warning the user can click through. HSTS eliminates the first-visit downgrade window (the "TOFU" problem — Trust On First Use) and SSL stripping attacks, where a network-level attacker intercepts the initial HTTP request before the user ever gets to HTTPS.

At Google Chrome scale, HSTS preloading means tens of thousands of domains get forced-HTTPS treatment from the very first visit, requiring no prior exchange. Misconfiguring HSTS (wrong max-age, missing subdomains, bad preload entries) can make entire domains unreachable if TLS fails.

How It Works

Core Mechanics

When a browser receives a response with Strict-Transport-Security header from an HTTPS origin, it records the policy in its HSTS store. From that point, any http:// URL for that host is internally rewritten to https:// before the request leaves the browser. No HTTP request ever hits the network.

Header Syntax:

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Directive Required Meaning
max-age=<seconds> Yes How long to remember the HSTS policy (31536000 = 1 year)
includeSubDomains Recommended Apply policy to all subdomains of the host
preload Optional Signal intent to be included in built-in browser preload lists

HSTS Store Behavior:

Condition Browser Action
URL is http:// for known HSTS host Internally rewrite to https:// before any network activity
HTTPS cert invalid for known HSTS host Hard error; no click-through override
max-age=0 received Delete HSTS record for host (used to opt out)
HSTS header received over HTTP Ignored; only valid over HTTPS

HSTS Preload List

The preload list is a hardcoded list of domains shipped inside Chrome, Firefox, Edge, and Safari. Domains on this list get HSTS enforcement from the very first visit to any user who has never been to the site before, eliminating the TOFU window entirely.

To qualify for preloading (per hstspreload.org):

  1. Serve a valid certificate
  2. Redirect all HTTP to HTTPS
  3. Serve HSTS header with max-age >= 31536000
  4. Include includeSubDomains directive
  5. Include preload directive

Warning: Preloading is very difficult to reverse. Removal from the list takes months to propagate and requires all subdomains to support HTTPS. Never add a domain to the preload list unless TLS is fully operational on every subdomain.

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 SSL stripping on first visit (TOFU window) HSTS not preloaded; attacker intercepts first HTTP request Credentials or session exposed over HTTP High
2 Short or missing max-age max-age=0 or very short value effectively disables HSTS Attacker can wait out the policy Med
3 Missing includeSubDomains Subdomains are reachable over HTTP without HSTS protection Cookie theft via HTTP subdomain Med
4 Preloading a not-fully-HTTPS domain All subdomains must support HTTPS; one HTTP-only subdomain breaks it Subdomains become unreachable High
5 HSTS received over HTTP (ignored) Server sends header on HTTP response (invalid) Policy silently not enforced Med
6 Removal from preload list very slow Once preloaded, removal takes months TLS outage makes domain unreachable for months High
7 Mixed content after HSTS HTTPS page loads HTTP subresources Active mixed content bypasses TLS; passive leaks data Med
8 HSTS supercookie / privacy abuse includeSubDomains combined with crafted subdomains enables state tracking User tracking across sites Low
9 Misconfigured redirect chains HTTP to HTTPS redirect does not set HSTS on the HTTPS response HSTS never stored Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 SSL stripping on first visit Submit to HSTS preload list for high-value domains HSTS preload submission
2 Short or missing max-age Set max-age to at least 31536000 (1 year) HTTP response header config
3 Missing includeSubDomains Always include includeSubDomains when all subdomains support HTTPS HTTP response header config
4 Preloading not-fully-HTTPS domain Audit all subdomains for HTTPS coverage before adding preload directive Pre-submission audit
5 HSTS over HTTP Only set HSTS header on HTTPS responses; configure at load balancer or web server Web server or LB config
6 Removal difficulty Treat preload as permanent; do not add domains unless TLS is stable long-term Process and planning
7 Mixed content Enforce CSP upgrade-insecure-requests; scan for HTTP subresource references CSP header + build scanning
8 HSTS supercookie Use SameSite cookie and Partitioning features to limit tracking surface Cookie design
9 Redirect chain does not set HSTS Ensure HSTS header is set on the final HTTPS 200 response, not only on redirects Server configuration

Top 3 Critical Mitigations

  1. Preload high-value domains. The TOFU window is a real attack surface. An attacker on a coffee shop network can strip TLS from the first HTTP request before HSTS is ever established. Preloading eliminates this entirely. For login domains, payment pages, and OAuth endpoints, preload is essential.
  2. Always set max-age to at least one year with includeSubDomains. A short max-age means protection expires quickly. Missing includeSubDomains means an attacker-controlled subdomain (via DNS hijack or takeover) can serve content over HTTP and steal cookies that are scoped to the parent domain.
  3. Audit before preloading; treat it as a permanent commitment. Prelude is effectively irreversible on any human timescale. If a single legacy subdomain still runs HTTP-only, it becomes unreachable after preloading. Run a full subdomain audit and TLS coverage check before submitting.

Key Takeaways

  • HSTS forces browsers to use HTTPS exclusively for a domain for the declared max-age period.
  • Without HSTS preload, the first HTTP request in a new session exposes a downgrade attack window.
  • includeSubDomains is critical; without it, subdomains remain HTTP-reachable and can expose parent-domain cookies.
  • Preloading is essentially permanent — never add domains to the preload list unless TLS is stable and all subdomains are covered.
  • HSTS errors are hard errors; the user cannot click through, which is by design and eliminates social engineering bypasses.
  • Combined with CT (Certificate Transparency), HSTS ensures both that HTTPS is used and that the certificate used is legitimately issued.
