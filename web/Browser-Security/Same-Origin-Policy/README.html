Same-Origin Policy (SOP)

Category: Browser Security — Isolation Primitive
Specification: HTML Living Standard, Fetch Standard, RFC 6454
Applies To: All major browsers (Chrome, Firefox, Safari, Edge)

Overview

The Same-Origin Policy is the foundational isolation mechanism built into every web browser. It prevents JavaScript running on one origin from reading data returned by a different origin. Without SOP, a malicious page at evil.com could silently read your Gmail inbox, your bank balance, or your company's internal dashboards using your authenticated session cookies. SOP is not optional — it is the principal security boundary between web origins. Every CORS exception, postMessage bridge, or cross-origin API surface represents a deliberate, controlled hole in this boundary.

At browser scale (billions of page loads per day), SOP enforcement is the single most critical isolation guarantee. Getting it wrong even once in Chrome can expose data across hundreds of millions of users without any action on their part.

How It Works

Core Mechanics

An origin is a tuple of three components:

Component Example
Scheme (protocol) https
Host (registered domain + subdomain) api.example.com
Port 443

Two URLs share an origin only if all three match exactly.

URL A URL B Same Origin? Reason
https://example.com https://example.com/path Yes Path does not affect origin
https://example.com http://example.com No Scheme differs
https://example.com https://www.example.com No Subdomain differs
https://example.com:443 https://example.com Yes 443 is implied for HTTPS
https://example.com:8080 https://example.com No Port differs

What SOP Allows and Blocks

Action Cross-Origin Allowed? Notes
Read XMLHttpRequest / fetch response body No (unless CORS) Core SOP restriction
Read <iframe> document content No contentDocument access blocked
Write to <iframe> (navigate) Yes Navigation is generally allowed
Read cookies of another origin No Cookie origin is scheme + host + path
<img> tag loading cross-origin image Yes But JS cannot read pixel data
<script> tag loading cross-origin JS Yes Script executes but cannot be read
<link> loading cross-origin CSS Yes Limited timing side-channel exists
window.postMessage Yes Explicit message passing across origins
localStorage / sessionStorage No Scoped to origin

SOP For Subdomains and document.domain

Sites can relax SOP between parent and subdomain by setting document.domain = "example.com" on both sides. This is considered legacy and dangerous. Modern best practice is to use CORS or postMessage instead. Chrome is in the process of restricting document.domain relaxation.

CORS As Controlled SOP Relaxation

SOP is enforced at read time. CORS headers on the server tell the browser it is safe to expose the response to the requesting origin. Without CORS headers, the browser still sends the request but silently discards the response before JavaScript can read it. See the CORS note for full details.

Why Requests Still Reach The Server

SOP blocks reads, not writes. The browser does send the request, and the server does process it. This is why CSRF is separately needed — SOP alone does not prevent state-changing side effects. A <form> POST or <img> tag can trigger a cross-origin request, but JavaScript cannot read the response unless CORS permits it.

Opaque Responses

Resources loaded with no-cors mode (as used by <img>, <script>, <link>) return an opaque response — a data blob with no visible headers, status, or body. Browsers still enforce that JavaScript cannot read pixel data from cross-origin images via <canvas> (the canvas is "tainted"). Service workers can cache opaque responses but cannot inspect them.

What Can Go Wrong

# Failure / Attack Root Cause Impact Likelihood
1 CORS misconfiguration Server echoes Origin header unconditionally or allows * with credentials Cross-origin data theft High
2 document.domain relaxation abuse Both subdomains lower to shared parent domain Same-origin access across subdomains Med
3 Spectre-class side-channel CPU speculative execution leaks cross-origin data via timing Memory content disclosure High (historically)
4 Sandbox bypass via plugin or extension Legacy plugins (Flash, Java) had their own origin handling SOP bypass, cross-origin read Low (plugins gone)
5 PostMessage origin not validated Receiver does not check event.origin Cross-origin message injection High
6 Cross-origin timing attacks via CSS/JS loading Timing of cached vs uncached resources reveals browsing history User tracking, browsing history leakage Med
7 Cross-origin redirect leakage Final URL after redirect can be inferred URL disclosure to third-party origin Med
8 Dangling markup injection HTML injected without script can exfiltrate data cross-origin Data exfiltration via <img>, <form> Med
9 Confused deputy in postMessage Service uses postMessage as trusted IPC without verifying sender Logic bypass, unauthorized actions Med

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 CORS misconfiguration Validate Origin against a strict allowlist; never reflect raw Origin dynamically; never combine * with credentials: true Server CORS middleware
2 document.domain abuse Disable document.domain modification; use CORS or postMessage for cross-subdomain comms Browser policy + app design
3 Spectre side-channel Enable Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Embedder-Policy: require-corp to enable site isolation and SharedArrayBuffer restrictions HTTP response headers
4 Plugin-based bypass Block plugins entirely; use CSP to restrict object-src CSP + browser policy
5 PostMessage origin not validated Always validate event.origin against expected origin before processing message JS message handler
6 Timing side-channels Use Cross-Origin-Resource-Policy: same-origin on sensitive endpoints; Vary cache behavior less on un-auth'd state Server headers + CDN config
7 Redirect leakage Use Referrer-Policy: no-referrer or same-origin; return opaque responses for sensitive redirects HTTP response headers
8 Dangling markup injection Strict Content Security Policy; HTML encode all output in contexts where untrusted data appears CSP + output encoding
9 PostMessage confused deputy Validate both event.origin and message structure; use nonces or capabilities in messages Application layer

Top 3 Critical Mitigations

  1. Strict CORS allowlists, never dynamic reflection: The single most common SOP bypass in production is a server that echoes back whatever Origin header it receives. This turns SOP into dead letter. CORS allowlists must be static and validated on every origin-bearing request.
  2. Enable site isolation via COOP and COEP headers: Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Embedder-Policy: require-corp are required to unlock full process-level isolation (used by Chrome's Site Isolation feature), which defends against Spectre-class attacks that can bypass SOP through hardware timing channels.
  3. Always validate event.origin in postMessage handlers: Many application logic bypasses come from code that uses postMessage as an internal communication bus but forgets to validate who the sender is. An attacker iframe or popup can send crafted messages to an application that trusts all postMessage events.

Key Takeaways

  • An origin is scheme + host + port; all three must match for same-origin equality.
  • SOP blocks JavaScript from reading cross-origin responses, not from sending requests.
  • CORS is the standard mechanism for relaxing SOP; misconfigured CORS is the most common SOP bypass.
  • Spectre-class hardware attacks require additional isolation headers (COOP + COEP) beyond what SOP alone provides.
  • PostMessage is an explicit channel through SOP; always validate event.origin in every message handler.
  • SOP is the root of browser security. Every web security control builds on or compensates for gaps in this primitive.
