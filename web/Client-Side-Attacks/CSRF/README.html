CSRF — Cross-Site Request Forgery

Category: Client-Side Attacks — Session-Based Abuse
OWASP: A01:2021 (Broken Access Control); historically A08
CWE: CWE-352
Applies To: Web applications that use cookie-based sessions for authenticated state-changing operations

Overview

Cross-Site Request Forgery is an attack that tricks an authenticated user's browser into making unintended requests to a web application, using the user's existing session credentials. The key insight: browsers automatically attach cookies to every matching HTTP request, regardless of which site initiated the request. If evil.com can cause the victim's browser to send a POST request to bank.com/transfer, and the victim has an active bank.com session cookie, the bank sees an authenticated request from the legitimate user.

The attacker does not need to steal the session cookie — they exploit the browser's automatic cookie-sending behavior. CSRF is fundamentally a trust confusion attack: the server trusts the request because it comes with a valid session cookie, but the genuine user did not initiate it.

At Google scale, CSRF on critical account operations (password change, email change, OAuth grants, payment authorization) could let attackers take over millions of accounts. Google's SameSite cookie default (Chrome 80+) was a major ecosystem-level fix that reduced the CSRF attack surface for all sites, not just Google's.

How It Works

Core Mechanics

Why CSRF Works:

  • HTTP cookies are automatically sent by the browser on every request to a matching origin
  • The browser makes no distinction between "request initiated by me" and "request initiated by a cross-site page"
  • The server sees a valid session cookie and has no way to tell which site caused the request (without additional defenses)

CSRF Attack Surface:

  • State-changing operations: fund transfers, password change, email change, account deletion
  • OAuth grant flows (consent to authorize)
  • Administrative actions: user creation, permission grants
  • Any POST, PUT, DELETE, PATCH endpoint using cookie-based auth

Attack Vectors

Vector 1: Automatic GET CSRF via Image Tag

<!-- On attacker page evil.com -->
<img src="https://admin.example.com/users/delete?id=42" style="display:none" />

When the victim loads evil.com, the browser silently makes a GET request to the admin URL with the victim's cookies. Only works if the application performs state changes on GET requests (anti-pattern).

Vector 2: Silent Form Auto-Submit (POST)

<form action="https://bank.example.com/transfer" method="POST" id="csrf">
  <input type="hidden" name="amount" value="5000" />
  <input type="hidden" name="to_account" value="attacker_account" />
</form>
<script>
  document.getElementById("csrf").submit();
</script>

Auto-submits on page load. Victim's session cookie is sent with the POST to the bank.

Vector 3: CSRF via JSON API (Requires text/plain Content-Type trick)
Most JSON APIs require Content-Type: application/json, which triggers a CORS preflight. But an attacker can sometimes use text/plain with a crafted body if the server ignores the content type header.

SameSite Cookie Attribute and CSRF

SameSite is the modern browser-level CSRF defense built into cookies:

SameSite Value CSRF Impact
Strict Cookie not sent on any cross-site request; eliminates CSRF completely
Lax Cookie sent on safe top-level navigations (GET); not on cross-site POST/PUT/DELETE; eliminates most CSRF
None Cookie always sent cross-site; CSRF fully possible if used for session
Not set (old default) Treated as None in old browsers; Lax is now the Chrome default

Chrome 80+ Change: Cookies without an explicit SameSite attribute default to SameSite=Lax. This was a significant ecosystem-wide CSRF reduction without requiring any application changes.

CSRF Token Pattern

The classic defense: the server embeds a secret, unpredictable token in forms and API requests. The attacker cannot read this token due to SOP (cannot read another origin's responses). Valid requests include the token; forged cross-site requests cannot.

Synchronizer Token Pattern:

  1. Server generates a random CSRF token per session (or per form)
  2. Token is embedded as a hidden form field or custom header
  3. Server validates token on every state-changing request
  4. If token is absent or wrong, request is rejected

Double Submit Cookie Pattern:

  1. Server sets a random value as a cookie
  2. App JS reads the cookie and sends it as a custom header (e.g., X-CSRF-Token) or form field
  3. Server checks that cookie value equals header/field value
  4. Attacker cannot read the cookie (SOP) so cannot construct valid header

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Missing CSRF token Use framework CSRF middleware (Django CSRF, Rails authenticity token, Spring Security CSRF); every state-changing request must require token Application framework
2 Token not validated Automated tests to verify CSRF validation; middleware that always validates on mutating verbs Backend middleware
3 Reused token Implement per-request (per-form) tokens or at minimum per-session tokens with proper invalidation Token generation logic
4 Token in URL Never put CSRF tokens in URLs; use hidden form fields or custom headers only Form design
5 GET state changes Never perform state-changing operations in GET handlers; follow HTTP semantics strictly API design
6 SameSite=None without CSRF If SameSite=None is required (cross-site embed), add CSRF token defense as compensating control Cookie design + CSRF middleware
7 XSS bypasses CSRF Eliminate XSS first; CSP; CSRF cannot fully protect against XSS in the same origin XSS mitigations
8 Content type abuse Validate Content-Type header on JSON endpoints; reject text/plain or application/x-www-form-urlencoded for JSON endpoints API input validation
9 CORS misconfiguration Fix CORS allowlist; don't allow cross-origin credential access for internal APIs CORS configuration
10 Clickjacking Set X-Frame-Options: DENY or Content-Security-Policy: frame-ancestors 'none' HTTP response headers

Top 3 Critical Mitigations

  1. Set SameSite=Lax or Strict on all session cookies. Introduced as the Chrome default in Chrome 80, this immediately eliminates cross-site POST CSRF for most applications with no code changes. SameSite=Strict is the strongest option. This is the single most impactful ecosystem-level CSRF fix.
  2. Use server-side CSRF tokens on every state-changing request. Never trust the browser's automatic cookie to mean "the user consciously initiated this request." A synchronizer CSRF token (random per-session value embedded in the form, validated server-side) proves the request originated from the legitimate page. Most frameworks provide this out of the box.
  3. Never perform state changes in response to GET requests. <img>, bookmarks, browser pre-fetching, and cross-site links all trigger GET requests with session cookies. GET must be safe (read-only) per HTTP semantics. Any GET endpoint that changes state is a CSRF vector that bypasses both SameSite=Lax and some token schemes.

Key Takeaways

  • CSRF exploits the browser's automatic cookie-sending behavior to forge authenticated requests from attacker-controlled pages.
  • The server sees a valid session cookie and cannot distinguish intentional from forged requests without additional controls.
  • SameSite=Lax (Chrome 80 default) eliminates cross-site POST CSRF for most applications transparently.
  • CSRF tokens prove the request originated from the legitimate page by embedding an unguessable secret the attacker cannot read.
  • APIs using Bearer tokens (not cookies) are inherently CSRF-resistant because the browser doesn't auto-attach Authorization headers.
  • XSS defeats CSRF tokens: an attacker with XSS can read the token and build a valid forged request.
  • GET endpoints that change state are the most dangerous CSRF exposure because they bypass SameSite=Lax protection.
