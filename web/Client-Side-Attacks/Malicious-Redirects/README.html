Malicious Redirects and Open Redirect Vulnerabilities

Category: Client-Side Attacks — URL Manipulation
OWASP: A01:2021 (Broken Access Control) — Open Redirect is a component
CWE: CWE-601 (URL Redirection to Untrusted Site)
Applies To: Any web application that uses URL-based redirects, especially authentication and OAuth flows

Overview

Malicious redirects are attacks where users are navigated — either transparently or as part of a legitimate-looking flow — to attacker-controlled destinations. The most common form is an open redirect vulnerability: an application that accepts a redirect_to or next URL parameter and performs a redirect to it without validating that the destination is safe. Open redirects enable phishing (lending the trusted domain's credibility to a malicious link), OAuth token theft, and account takeover when combined with authentication flows.

At Google scale, open redirects in Google's authentication endpoints could be used to steal OAuth authorization codes and access tokens. For example, a crafted link that looks like accounts.google.com/...?continue=evil.com/steal-token would be highly trusted by users because of the google.com domain in the URL. Google's auth team validates redirect URIs very strictly against pre-registered allowlists — a key lesson for all OAuth implementors.

How It Works

Core Mechanics

An open redirect exists when an application:

  1. Accepts a URL parameter containing a destination URL
  2. Redirects the user to that destination
  3. Does not validate that the destination is within the application's domain or an approved list

Typical Vulnerable Code Pattern:

# Vulnerable: no validation
next_url = request.args.get('next', '/')
return redirect(next_url)

Attack URL:

https://trusted.example.com/login?next=https://evil.com/phish

User sees a URL starting with trusted.example.com, completes login, then gets silently redirected to evil.com. The redirect happens so smoothly after login that many users do not notice the destination changed.

Open Redirect in OAuth 2.0 Token Theft

Common URL Bypass Techniques For Open Redirect Filters

Attackers craft URLs to bypass naive validation:

Bypass Technique Example Why It Might Work
Protocol-relative URL //evil.com/ Browser infers https: or http:; filter only blocks http: and https:
@ character abuse https://trusted.com@evil.com Browser uses evil.com as host; trusted.com is username
URL encoding https://evil%2Ecom Decoded after filter runs
Unicode normalization https://eⅵl.com (Unicode lookalike) Passes ASCII filter
Open redirect chain https://trusted.com/r?u=https://trusted2.com/r?u=evil.com Filter validates one hop; chained redirect reaches evil.com
Fragment with scheme https://trusted.com#https://evil.com (then JS redirect) Fragment check bypassed
JavaScript URI javascript:document.location='evil.com' (in href context) If input used as href without scheme validation

Redirect After Authentication

Many applications redirect users to a next or continue URL after login. This is a legitimate feature (deep linking) but creates the most dangerous open redirect vector:

https://app.example.com/login?next=https://evil.com/steal-cookies

Steps:

  1. User visits malicious link; gets sent to the legitimate login page
  2. User logs in normally (correctly enters credentials)
  3. Application redirects to https://evil.com/steal-cookies as post-login destination
  4. Victim's browser loads the attacker page with a valid fresh session

This is especially dangerous because the user's guard is down after having completed a genuine login.

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Open redirect Validate redirect destination against a strict allowlist of approved paths/domains; prefer path-only redirects Server-side redirect handling
2 OAuth auth code theft Strictly validate redirect_uri against pre-registered list; exact match only; reject partial matches OAuth server implementation
3 Token in URL / Referer Never put tokens or auth codes in URL parameters; use POST body or fragment; set Referrer-Policy: no-referrer Auth flow design + response headers
4 Redirect chain bypass Validate final destination, not just first hop; reject any redirect that passes through external domains Server-side validation
5 Protocol-relative bypass Normalize URL before validation; use URL parsing library, not string matching Input validation library
6 @ character abuse Use URL parsing library to extract host component; check host not full URL string Input validation
7 JavaScript URI Block javascript: scheme explicitly; ensure redirect only follows https:// or relative paths Scheme allowlisting
8 Malvertising chains Restrict ad content via Content Security Policy; use strict-dynamic; limit third-party scripts CSP header
9 SEO abuse Add RelCanonical and X-Robots-Tag to prevent redirect pages from being indexed SEO controls

Safe Redirect Implementation

Best Practice — Allowlist Based:

ALLOWED_REDIRECT_HOSTS = {'app.example.com', 'api.example.com'}

def safe_redirect(url):
    parsed = urllib.parse.urlparse(url)
    # Only allow relative paths OR explicitly allowlisted hosts
    if not parsed.netloc:  # Relative path: OK
        return redirect(url)
    if parsed.netloc in ALLOWED_REDIRECT_HOSTS and parsed.scheme in ('https',):
        return redirect(url)
    return redirect('/')  # Default to home on invalid input

Best Practice — Relative Path Only:

def safe_redirect(url):
    # Only allow paths starting with /; not protocol or host
    if url.startswith('/') and not url.startswith('//'):
        return redirect(url)
    return redirect('/')

Top 3 Critical Mitigations

  1. Validate redirect destinations against an allowlist of exact host+scheme values. String contains/substring matching is easily bypassed. Use a proper URL parsing library to extract the host component, then compare it against a static allowlist. Reject anything else with a safe default redirect.
  2. For OAuth redirect_uri, use exact-match validation only. The OAuth spec requires exact match. Wildcard or prefix matching in redirect_uri validation has historically led to auth code theft. Register exact URIs for each client and reject any deviation.
  3. Prefer relative-path-only redirects wherever possible. If the application only ever needs to redirect to its own pages (post-login next parameter, etc.), restrict the input to paths starting with / but not //. This eliminates any possibility of external redirect without complex URL parsing.

Key Takeaways

  • Open redirect vulnerabilities allow attackers to use a trusted domain as a stepping stone for phishing and token theft.
  • The most dangerous variant is open redirect in OAuth flows: it turns a benign-looking auth URL into an auth code interception attack.
  • String-based validation is easily bypassed by protocol-relative URLs, @ abuse, URL encoding, and redirect chains.
  • Always use a URL parsing library to extract the host component, then compare host against an explicit allowlist.
  • Never put tokens or auth codes in URL parameters — they leak via Referer headers, browser history, and server logs.
  • Prefer relative-path-only redirects whenever the application only needs to redirect within its own domain.
