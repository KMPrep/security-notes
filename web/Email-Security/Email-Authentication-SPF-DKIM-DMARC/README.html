Email Authentication — SPF, DKIM, DMARC (+ BIMI, MTA-STS, TLS-RPT, ARC)

Category: Email Security — Sender Authentication & Anti-Spoofing
RFCs: RFC 5321/5322 (SMTP & message format), RFC 7208 (SPF), RFC 6376 (DKIM), RFC 7489 (DMARC), RFC 8617 (ARC), RFC 8461 (MTA-STS), RFC 8460 (TLS-RPT), RFC 8463 (Ed25519 DKIM)
MITRE ATT&CK: T1566 (Phishing), T1566.001/.002 (Spearphishing Attachment/Link), T1534 (Internal Spearphishing), T1585.002 (Email Accounts), T1656 (Impersonation)
Cross-refs: network/Layer7/DNS/README.md (all three controls live in DNS TXT records), web/Email-Security/Phishing-and-BEC/README.md (the attacks these controls defend against)

Overview

SMTP was designed in 1982 (RFC 821) for a small, mutually trusting network of academic mail hosts and has no built-in authentication of the sender. Any host that can open TCP/25 to a receiver can claim to be sending mail from any domain — MAIL FROM:<ceo@yourbank.com> and a From: header reading CEO <ceo@yourbank.com> are accepted verbatim. SPF, DKIM, and DMARC are three layered DNS-published controls bolted onto this trust-by-default protocol to answer three questions: Is this IP allowed to send for the domain? (SPF), Was this message cryptographically signed by the domain and unmodified in transit? (DKIM), and Do those results align with the domain the human actually sees, and what should the receiver do if not? (DMARC).

At scale this is existential. Google, Microsoft, and Yahoo each process hundreds of billions of messages daily; without sender authentication, spoofed mail purporting to be from paypal.com, irs.gov, or your own corporate domain would flood inboxes. As of February 2024, Google and Yahoo require bulk senders (5,000+ messages/day) to publish SPF, DKIM, and a DMARC policy or have their mail rejected/spam-foldered — turning what was a best practice into a deliverability mandate. DMARC is also the single most effective control against exact-domain spoofing, the technique behind Business Email Compromise (BEC) that the FBI's IC3 attributes to over $2.9 billion in reported losses in 2023 alone.

TL;DR
- SPF authorizes sending IPs for the MAIL FROM (envelope/Return-Path) domain via a DNS TXT record; it breaks on forwarding and has a hard 10 DNS-lookup limit (exceeding it = permerror).
- DKIM attaches a cryptographic signature (DKIM-Signature header) over selected headers + body; the verifier fetches the public key from <selector>._domainkey.<domain> in DNS. It survives forwarding but not body modification.
- DMARC ties SPF/DKIM back to the visible From: header domain via alignment, publishes a policy (p=none|quarantine|reject), and requests aggregate (rua) and forensic (ruf) reports. DMARC passes if at least one of aligned-SPF or aligned-DKIM passes.
- Roll out p=none → quarantine → reject using aggregate reports to find and authorize every legitimate sender before enforcing.
- ARC preserves authentication results across trusted intermediaries (mailing lists, forwarders) that break SPF/DKIM; MTA-STS/TLS-RPT enforce and report on transport TLS.

How It Works

2.1 Core Mechanics

The critical, notable subtlety is that email has two different "from" identities, and each control checks a different one:

Identity Where it lives RFC Who sees it Checked by
Envelope sender / Return-Path / MAIL FROM SMTP envelope (transport) RFC 5321 Mail servers only SPF
Header From: (a.k.a. RFC5322.From) Message headers RFC 5322 The human recipient DMARC alignment
d= domain in DKIM-Signature Message header, signed RFC 6376 Signature covers it DKIM
DKIM-Signature selector s= Header, points to DNS key RFC 6376 — DKIM key lookup

Spoofing exploits the gap: an attacker can pass SPF using a domain they own in the envelope while forging a header From of your bank. SPF and DKIM alone don't require these to match — DMARC's entire purpose is to require the authenticated identity to align with the header From the user sees.

Concept Definition Notes

SPF qualifier + pass (default), - fail (hardfail), ~ softfail, ? neutral -all vs ~all is the key policy choice
SPF result pass, fail, softfail, neutral, none, permerror, temperror permerror from >10 lookups
DKIM selector Label choosing which key pair; enables multiple/rotated keys s=selector1 → selector1._domainkey.d
DKIM canonicalization simple or relaxed for header/body (c=relaxed/relaxed) relaxed tolerates whitespace/case changes
DKIM bh= Base64 hash of the (canonicalized, l=-limited) body Body-integrity check
DKIM b= The RSA/Ed25519 signature over signed headers h= lists which headers are signed

DMARC policy p=none|quarantine|reject; sp= for subdomains; pct= phased % Receiver enforces

Forensic report (RUF) Per-message redacted failure sample (AFRF) Rare today (privacy)

Evaluation/decision logic at a receiving MTA (e.g., Gmail's inbound path) proceeds: (1) accept the SMTP transaction and record the connecting IP + MAIL FROM domain; (2) evaluate SPF against that IP; (3) verify each DKIM-Signature by fetching the public key and recomputing hashes; (4) look up the DMARC record for the header From org domain; (5) test whether SPF-domain or DKIM-d= aligns with the header From; (6) DMARC passes if any aligned method passed; (7) if it fails, apply the published policy (subject to pct and any local override such as ARC or a known-forwarder allowlist); (8) emit an aggregate report row.

2.2 Concrete Configuration

SPF record — a single TXT record at the domain apex. There must be exactly one SPF record per domain (multiple v=spf1 records = permerror):

; example.com apex TXT record
example.com.   IN TXT "v=spf1 ip4:203.0.113.0/24 ip4:198.51.100.7 include:_spf.google.com include:sendgrid.net -all"
  • ip4:203.0.113.0/24 — authorize your own mail egress range directly (costs 0 DNS lookups; prefer this).
  • include:_spf.google.com — delegate to Google Workspace's SPF (counts as 1 lookup, and recursively expands its own includes, each counted).
  • -all — hardfail: anything not listed is unauthorized. Use -all in production; ~all (softfail) is a safe stepping stone but does not itself cause rejection.

The 10-lookup limit (RFC 7208 §4.6.4): SPF evaluation must not require more than 10 DNS-querying mechanisms (include, a, mx, ptr, exists, and the redirect modifier). ip4/ip6/all cost nothing. Exceed 10 and the result is permerror, which under DMARC counts as an SPF failure. Real records blow this budget fast because each include recursively expands — a single include:spf.protection.outlook.com can consume 2-3, and marketing tools chain further. Fix by flattening (replacing includes with the resolved ip4 ranges, at the cost of manual maintenance) or using an SPF-flattening/macro service. Inspect with:

# Count SPF lookups and see the expansion tree
dig +short TXT example.com
# Tooling: use an SPF checker, or a script that recursively resolves includes
python3 -c "import checkdmarc; import json; print(json.dumps(checkdmarc.get_spf_record('example.com'), default=str))"

DKIM — generate a key pair, publish the public half in DNS under a selector, configure the signer with the private half. With OpenDKIM/Postfix:

# Generate a 2048-bit key with selector "s1" for example.com
opendkim-genkey -b 2048 -d example.com -s s1 -v
# Produces s1.private (keep secret on the MTA) and s1.txt (the DNS record)

; Publish the public key. Selector "s1" -> queried at s1._domainkey.example.com
s1._domainkey.example.com. IN TXT ( "v=DKIM1; k=rsa; t=y; "
  "p=MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA1x...snip...QIDAQAB" )

A resulting outbound signature header looks like:

DKIM-Signature: v=1; a=rsa-sha256; c=relaxed/relaxed; d=example.com; s=s1;
    t=1720310400; h=from:to:subject:date:message-id;
    bh=2jUSOH9NhtVGCQWNr9BrIAPreKQjO6Sn7XIkfJVOzv8=;
    b=Cg8bW...base64signature...==

Key operational points: t=y marks the key as testing (verifiers should not treat failures as significant) — remove it once confident. h=from:... must include from (DMARC requires the From header be signed). l= (body-length limit) is a footgun — it lets an attacker append content below the signed portion; omit l=. Rotate keys by publishing a new selector (s2) first, switching the signer to sign with s2, then removing s1 after the mail-in-flight window (a few days). Never reuse a selector for a new key.

DMARC — a TXT record at the _dmarc subdomain of the org domain:

; Monitoring phase — collect data, enforce nothing
_dmarc.example.com. IN TXT "v=DMARC1; p=none; rua=mailto:dmarc-agg@example.com; ruf=mailto:dmarc-forensic@example.com; fo=1; adkim=r; aspf=r; pct=100"

; Enforcement phase — full reject with strict DKIM alignment and subdomain reject
_dmarc.example.com. IN TXT "v=DMARC1; p=reject; sp=reject; rua=mailto:dmarc-agg@example.com; adkim=s; aspf=r; pct=100"
  • p= policy for the org domain; sp= policy for subdomains (if omitted, subdomains inherit p). Set sp=reject explicitly to block spoofing of unused subdomains like no-reply.marketing.example.com.
  • adkim/aspf: r (relaxed — org domain match) or s (strict — exact FQDN). Relaxed is the pragmatic default; strict prevents subdomain-based alignment tricks.
  • fo=1 requests a forensic report if any underlying mechanism fails (default fo=0 = only if all fail).
  • pct= samples the policy — pct=25 applies p to 25% of failing mail (rest falls back to the next-lower policy). Useful when moving to reject. Note: pct was removed in the DMARCbis draft (RFC 7489 successor) because it caused inconsistent enforcement — modern advice is to use pct briefly if at all.

If a receiver at a different org hosts your rua, add authorization at their domain: example.com._report._dmarc.thirdparty.com IN TXT "v=DMARC1".

MTA-STS (RFC 8461) — enforces TLS for inbound SMTP so an active attacker can't strip STARTTLS. It needs a DNS TXT record and an HTTPS-served policy file:

_mta-sts.example.com. IN TXT "v=STSv1; id=20240601000000Z"

# Served at https://mta-sts.example.com/.well-known/mta-sts.txt
version: STSv1
mode: enforce
mx: mx1.example.com
mx: mx2.example.com
max_age: 604800

TLS-RPT (RFC 8460) — asks partners to report TLS-delivery failures so you detect downgrade attacks or broken certs:

_smtp._tls.example.com. IN TXT "v=TLSRPTv1; rua=mailto:tls-reports@example.com"

BIMI — publishes a brand logo (and, with a Verified Mark Certificate, a checkmark) shown by supporting clients only when DMARC is at enforcement (quarantine/reject):

default._bimi.example.com. IN TXT "v=BIMI1; l=https://example.com/bimi/logo.svg; a=https://example.com/bimi/vmc.pem"

Case study — Ubiquiti Networks (2015, $46.7M). Attackers impersonated executives and an outside entity, emailing the finance department with wire-transfer instructions to attacker-controlled overseas accounts. This is textbook BEC (ATT&CK T1656/T1566). The technical enabler is that the recipient's client rendered a trusted-looking From:, and either the domain had no DMARC enforcement (allowing exact-domain spoofing) or a lookalike domain (ubnt-corp.com) was used. DMARC at p=reject stops exact-domain spoofing outright. Lookalike domains require complementary controls: brand/domain monitoring, and receiver-side display-name/cousin-domain heuristics — DMARC does not protect domains you don't own. See web/Email-Security/Phishing-and-BEC/README.md.

Case study — "EchoSpoofing" via Proofpoint (2024). Researchers found that a permissive Proofpoint hosted-email relay configuration allowed spammers to route millions of spoofed messages (impersonating Disney, IBM, Nike, etc.) through the relay, where they were DKIM-signed and passed SPF/DMARC because the relay was an authorized sender for those tenants. Millions of "perfectly authenticated" phishing messages reached inboxes daily. Root cause: the relay didn't verify that the connecting tenant was actually authorized to send as the claimed domain. Lesson: authentication is only as strong as the weakest authorized path. A DMARC pass means "an authorized sender signed this," not "the legitimate business intended it" — you must lock down every relay/ESP that can sign for you, and scope third-party authorizations tightly.

Case study — DMARC's DHS success (2017-2018). US DHS Binding Operational Directive 18-01 mandated all federal .gov domains publish DMARC and reach p=reject within one year. Adoption jumped from ~34% to ~85% of agencies at enforcement, and Agari/Valimail reporting showed a measurable drop in .gov-spoofing phishing. Lesson: DMARC works at scale when enforcement is mandated and reports are actually monitored — the same playbook Google/Yahoo now impose on bulk senders.

Top 5 Controls in Depth

1. DMARC at p=reject with monitored aggregate reports. This is the keystone. Technically, when a receiver evaluates a message whose header-From org domain publishes p=reject, and neither aligned-SPF nor aligned-DKIM passes, the receiver rejects (5xx) or discards. Implementation is the phased record evolution shown in §2.2 (none → quarantine → reject), driven entirely by RUA data. Trade-off: enforcement can break legitimate mail you didn't know about (a marketing tool, a helpdesk system, a forwarder). This is why you never jump straight to reject — you spend weeks at p=none inventorying senders from reports, authorizing each in SPF/DKIM, then tighten. Limit: DMARC protects only domains you own and publish records for; it does nothing against lookalike domains or display-name spoofing.

2. Keep SPF under the 10-lookup ceiling. Each DNS-querying mechanism counts; blow the budget and SPF returns permerror, which DMARC treats as a fail — potentially rejecting your own mail. Implementation: audit with checkdmarc/spfquery, replace expensive include: chains with direct ip4:/ip6: ranges ("flattening"), and remove decommissioned ESPs. Trade-off: flattened records are brittle — if an ESP changes its egress IPs, mail breaks silently until you re-flatten, so this needs monitoring or an automated flattening service. Limit: flattening can leave you dependent on a vendor whose IPs change without notice; the cleanest fix is fewer senders and using DKIM (which is lookup-free) as the primary alignment method.

3. DKIM signing everything, 2048-bit, from signed, l= omitted, rotated. DKIM is more robust than SPF because it survives IP changes and simple forwarding. The signer canonicalizes (relaxed/relaxed recommended) and signs h=from:to:subject:date:... plus the body hash. Oversigning (listing a header twice in h=) prevents an attacker from adding a second From header the verifier ignores. Rotate via new selectors quarterly. Trade-off: DKIM proves the domain signed the message, not that the domain intended this recipient — hence replay abuse; mitigate with short x= expiry and per-message controls. Limit: body modification by mailing lists breaks DKIM; that gap is what ARC fills.

4. ARC (Authenticated Received Chain) for intermediaries. Forwarders and mailing lists legitimately break SPF (new source IP) and often DKIM (they rewrite subjects/footers). ARC lets each hop seal the authentication results it observed (ARC-Authentication-Results), cryptographically chained (ARC-Seal, ARC-Message-Signature) so a downstream receiver can see "when this arrived at the trusted list server, DMARC passed" and choose to honor that despite a current DMARC fail. Implementation is on the MTA (Gmail, Office 365 both seal and evaluate). Trade-off: ARC is only as trustworthy as the sealer — you must maintain an allowlist of intermediaries you trust; blindly honoring any ARC chain reintroduces spoofing. Limit: adoption is uneven, so ARC reduces but doesn't eliminate forwarding false-positives.

5. MTA-STS + TLS-RPT (transport integrity). SPF/DKIM/DMARC authenticate content and sender, not the channel. Opportunistic STARTTLS can be stripped by an on-path attacker (T1557), enabling interception. MTA-STS publishes an enforce-mode policy (DNS TXT pointer + HTTPS-hosted policy file listing valid MX and requiring TLS with a valid cert); compliant senders refuse to deliver if TLS can't be negotiated to a listed MX. TLS-RPT collects failure reports so you spot downgrade attempts or expired certs. Trade-off: a broken MTA-STS policy or lapsed cert can block inbound mail — start in testing mode and watch TLS-RPT before enforce. Limit: MTA-STS trusts the web PKI (a mis-issued cert defeats it); DANE (DNSSEC-anchored TLSA) is stronger but requires DNSSEC (see network/Layer7/DNS/README.md).

Detection & Response

What to log / telemetry sources

  • DMARC aggregate (RUA) reports — daily XML, one <record> per source IP with <row><policy_evaluated> (disposition, dkim, spf) and <auth_results>. This is the primary source for finding spoofing and unauthorized senders.
  • DMARC forensic (RUF) reports — per-message redacted failure samples (AFRF); rich but privacy-sensitive and rarely sent by major receivers.
  • Inbound MTA logs / Authentication-Results headers — your own receivers' spf=, dkim=, dmarc= verdicts per message (Gmail, Postfix+OpenDMARC, Exchange headers).
  • TLS-RPT reports — JSON summaries of TLS delivery success/failure per sending MTA.
  • SIEM email telemetry — sender IP, header From vs envelope From, auth verdicts, message-id, subject; correlate with URL/attachment detonation.
  • DNS change logs — alert on modifications to _dmarc, _domainkey, and apex SPF records (an attacker who can edit DNS can neuter all three).

Sample detections

A — SQL over parsed DMARC aggregate reports: find IPs spoofing you. Assume RUA XML is parsed into a table dmarc_agg(report_date, source_ip, header_from, count, disposition, spf_result, spf_aligned, dkim_result, dkim_aligned, sending_domain).

-- Sources sending as our domain that FAIL DMARC (unauthorized or misconfigured legit sender)
SELECT source_ip,
       header_from,
       SUM(count)                          AS volume,
       MAX(report_date)                    AS last_seen,
       SUM(CASE WHEN spf_aligned  THEN count ELSE 0 END) AS spf_ok,
       SUM(CASE WHEN dkim_aligned THEN count ELSE 0 END) AS dkim_ok
FROM   dmarc_agg
WHERE  report_date >= CURRENT_DATE - INTERVAL '7' DAY
  AND  disposition IN ('quarantine','reject')       -- or dkim/spf both != pass at p=none
GROUP  BY source_ip, header_from
HAVING SUM(count) > 50                                -- tune threshold
ORDER  BY volume DESC;

Triage: an unknown IP with high failing volume and no alignment → likely spoofing (or a forgotten legit sender to authorize). A known corporate/ESP IP failing → misconfiguration to fix before enforcement.

B — Sigma rule: inbound message failing DMARC while spoofing an internal/high-value domain.

title: Inbound mail fails DMARC impersonating protected domain
id: 4a3f9e21-emailauth-dmarc-fail

  service: gateway

  - source_ip

  - header_from
  - subject
  - message_id
level: high

C — KQL (Microsoft 365 / Defender) for display-name impersonation of executives.

EmailEvents
| where Timestamp > ago(1d)
| where SenderFromDomain !in ("example.com")            // external sender...
| extend DisplayName = tostring(SenderDisplayName)
| where DisplayName has_any ("CEO","CFO","Jane Doe","John Smith")  // exec name list
| where AuthenticationDetails has "dmarc" and AuthenticationDetails !has "dmarc=pass"

Response playbook

  1. Contain: if aggregate reports show sustained spoofing of your domain, confirm your DMARC is at p=reject; if still at none/quarantine, move to enforcement (after verifying no legit sender is collateral). Notify the SEG/mail team to block the spoofing source IPs and quarantine matching in-flight messages.
  2. Identify blast radius: query mail logs for delivered messages matching the spoof pattern (source IP, subject, header From) during the window before enforcement; list recipients who received/opened.
  3. Eradicate: for a compromised authorized sender (ESP, relay), revoke/rotate its DKIM selector and remove it from SPF; for a dangling include, delete it; for a compromised DNS account, restore records and rotate DNS provider credentials.
  4. Recover: send targeted user warnings to recipients of spoofed mail; force password resets if credential phishing; verify any financial transaction touched by BEC out-of-band before it settles (recall wires within the 72-hour window — the FBI IC3 Financial Fraud Kill Chain).
  5. Harden: raise pct to 100 and sp=reject; add -all; deploy MTA-STS enforce; register defensive lookalike domains; add SEG anti-impersonation rules for exec display names and cousin domains.
  6. Report & learn: file with IC3 for BEC losses; feed the spoofing IPs/domains into TI; add a detection for the specific pattern; schedule a quarterly SPF/DKIM/DMARC posture review.

Practice Questions

Tier 1 — Fundamentals

Q1. Why does email need SPF/DKIM/DMARC at all — what's wrong with SMTP?
SMTP (RFC 5321) has no authentication of the sender: any host can issue MAIL FROM and set a From: header for any domain, and the receiver historically accepted it. This lets attackers spoof trusted brands and internal executives for phishing and BEC. SPF, DKIM, and DMARC are DNS-published controls layered on top: SPF authorizes sending IPs, DKIM cryptographically signs the message, and DMARC ties those results to the visible From header and tells receivers what to do on failure.
Signal: Do they understand these are bolt-ons to an inherently trust-based protocol, not part of SMTP itself.

Q2. What identity does SPF check, and what identity does DMARC care about?
SPF authenticates the envelope sender (MAIL FROM / Return-Path), which mail servers see but users don't. DMARC cares about the header From: — the address the human recipient actually reads. The two can legitimately differ, so SPF passing doesn't mean the visible From is genuine; DMARC's alignment requirement is what forces the authenticated identity to match the header From.
Signal: The single most important conceptual distinction; separates people who've operated these from those who've read a blog.

Q3. Explain the SPF 10-lookup limit and one consequence of exceeding it.
SPF evaluation may trigger at most 10 DNS-querying mechanisms (include, a, mx, ptr, exists, redirect); ip4/ip6/all are free. Nested includes expand recursively and count too. Exceeding 10 yields permerror, which DMARC treats as an SPF failure — meaning a bloated SPF record can cause your own legitimate mail to fail DMARC and be rejected.
Signal: Knowing it's a real operational failure mode, not trivia.

Q4. What is a DKIM selector and why do selectors exist?
A selector is a label in the DKIM-Signature (s=) that tells the verifier which public key to fetch — from <selector>._domainkey.<d-domain>. Selectors let a domain publish multiple keys simultaneously: for different ESPs/systems and, crucially, for key rotation (publish a new selector's key, switch signing to it, retire the old one) without a flag-day cutover.
Signal: Understands DKIM key lifecycle, not just the concept.

Q5. What are the three DMARC policies and what does each instruct receivers to do?
p=none = monitor only; deliver as normal but send reports (used to gather data). p=quarantine = treat failing mail as suspicious, typically route to spam/junk. p=reject = refuse failing mail outright (5xx). Alongside sits sp= for subdomains and pct= for phased sampling.
Signal: Baseline fluency and awareness of the phased-rollout intent behind none.

Tier 2 — System Design / Trade-offs

Q6. Design a safe DMARC rollout for a company with 20 sending systems and no current records. Walk the phases.
Start by publishing DKIM on every legitimate sending system and a correct SPF (≤10 lookups), then publish p=none with rua pointing to a monitored mailbox/analytics platform. Spend 2-6 weeks parsing aggregate reports to build a complete inventory of sources sending as your domain, authorizing each legitimate one (add to SPF, enable DKIM) and investigating unknowns. Once essentially all legitimate volume passes and aligns, move to p=quarantine — optionally with pct to sample — and watch reports for false positives (forwarders, forgotten tools). When quarantine is stable, go to p=reject, add sp=reject, harden SPF to -all, and keep the reporting pipeline running permanently. The whole point of the phased approach is that enforcement blocks legitimate mail you didn't know existed, so reports drive every promotion.
Signal: Do they treat reports as the engine and refuse to jump to reject; do they mention alignment, not just pass.

Q7. A partner says your legitimate mail lands in their spam after they enabled DMARC on their side — actually your mail to a mailing list gets rejected. Why, and how do you fix it?
Mailing lists and forwarders break both underlying mechanisms: they relay from their own IP (so SPF fails for the original domain) and often rewrite the subject or add footers (so the body hash breaks DKIM). With no aligned pass, DMARC fails and a strict receiver rejects. Fixes: the list can rewrite the From to its own domain (From: "Alice via list" <list@lists.example>) so DMARC applies to the list's domain; forwarders can implement SRS to rewrite the envelope so SPF passes; and the ecosystem's real answer is ARC, where the list seals the authentication results it saw so downstream receivers can honor the original pass despite the current failure. On your side, ensure DKIM signs a minimal, stable header set and omit l=.
Signal: Deep understanding of the forwarding problem and ARC/SRS as the fixes.

Q8. SPF vs DKIM — if you could only rely on one for DMARC alignment, which and why?
DKIM, because it's far more robust: the signature travels with the message and survives IP changes, relaying, and simple forwarding, whereas SPF fails the moment mail is forwarded or the sending IP isn't in the record, and it's fragile against the 10-lookup limit. DKIM also doesn't consume the SPF lookup budget. SPF still has value (it's cheap and catches direct spoofing from unauthorized IPs), and you want both for redundancy since DMARC passes if either aligns — but if forced to pick the primary alignment method, DKIM wins.
Signal: An opinion backed by the forwarding and lookup-limit realities.

Q9. How would you architect a pipeline to consume DMARC aggregate reports for 500 domains and surface spoofing?
Receive rua at a dedicated mailbox or provider endpoint; ingest the (often gzip/zip) XML, validate and parse each <record> into a normalized table keyed by report_date, domain, source_ip, header_from, disposition, spf/dkim result and alignment, and count. Enrich source_ip with ASN/geo/rDNS and cross-reference known-good sender inventories (your ESPs, egress ranges). Build dashboards for per-domain pass-rate trends and alerts for new high-volume failing sources, sudden pass-rate drops (a broken key or SPF change), and enforcement readiness (percentage of aligned volume). The goal is to (a) find unauthorized senders/spoofing and (b) safely advance each domain's policy. Retain history for trend analysis and compliance.
Signal: Systems thinking: parsing, enrichment, inventory correlation, alerting — not just "read the XML."

Q10. Google/Yahoo's 2024 bulk-sender rules — what do they require and why does it matter architecturally?
As of February 2024, senders of 5,000+ messages/day to Gmail/Yahoo must: authenticate with both SPF and DKIM, have a DMARC record (at minimum p=none) with alignment, keep spam complaint rates low (under ~0.3%), support one-click unsubscribe (RFC 8058), and send from domains with valid forward/reverse DNS over TLS. Architecturally this forces every marketing/transactional system to have proper DKIM signing and be inventoried in SPF, and it effectively mandates DMARC for anyone doing bulk mail — non-compliant mail is spam-foldered or rejected. It codified best practice into a deliverability requirement.
Signal: Awareness of the current mandate and its practical push toward enforcement.

Tier 3 — Expert / Deep-Dive

Q11. Explain the DKIM replay attack and why a valid signature doesn't imply a legitimate message.
DKIM proves that the signing domain's key signed the covered headers and body — it says nothing about the intended recipient or the sender's intent. In a replay attack, an adversary obtains one legitimately DKIM-signed message (e.g., by getting a benign email through a large provider that signs all outbound), then re-sends that exact message — signature intact and valid — to millions of new recipients. Because the signature verifies and aligns, it passes DMARC at the big provider's expense. Mitigations: sign a tight header set and include recipient-context where feasible, set a short signature expiry (x=), rate-limit and monitor for the same bh=/message being sent at scale, avoid l=, and providers must add anti-abuse heuristics beyond DKIM. The 2023-24 Gmail replay wave made this concrete.
Signal: Understands DKIM's semantics precisely — origin proof, not intent or authorization.

Q12. Walk through DKIM canonicalization and the l= tag risks.
Canonicalization normalizes the message before hashing so trivial in-transit changes don't break the signature. simple is strict (almost no changes tolerated); relaxed tolerates header case/whitespace folding and body trailing-whitespace/empty-line changes — c=relaxed/relaxed is the practical choice for surviving benign MTA rewrites. The bh= body hash covers the canonicalized body, optionally limited to the first l= bytes. l= is dangerous: it signs only a prefix of the body, so an attacker can append arbitrary content (malicious links, different message) after the signed portion and the signature still verifies — a content-injection vector. Best practice: omit l= entirely and oversign headers (list from twice in h=) so an attacker can't add an unsigned duplicate From that some clients display.
Signal: Cryptographic-protocol depth and knowledge of real footguns.

Q13. Compare MTA-STS and DANE for SMTP transport security. When would you choose each?
Both prevent STARTTLS downgrade, but their trust anchors differ. MTA-STS (RFC 8461) publishes a policy via DNS TXT pointer plus an HTTPS-hosted policy file; senders cache it and refuse cleartext/invalid-cert delivery to listed MX. It relies on the web PKI and TOFU-style caching — deployable without DNSSEC, which is why Google/Microsoft adopted it, but a mis-issued CA cert or a first-contact attacker can undermine it. DANE (RFC 7672) publishes a TLSA record binding the MX cert/key, anchored in DNSSEC — cryptographically stronger and not dependent on the CA ecosystem, but it requires DNSSEC on both sides, which many operators lack. Choose MTA-STS for broad interoperability today (especially with the big mailbox providers); choose DANE where you control DNSSEC end-to-end and want the stronger guarantee. Pair either with TLS-RPT to detect failures. See network/Layer7/DNS/README.md for DNSSEC.
Signal: Knows both, their PKI vs DNSSEC trust models, and the pragmatic deployment reality.

Q14. DMARC gives you p=reject on your owned domains. What classes of email impersonation does it NOT stop, and how do you cover them?
DMARC only protects domains that publish records and are the header From — so it does nothing against (1) lookalike/cousin domains (paypa1.com, example-support.com) which are attacker-owned and can have their own passing DMARC; (2) display-name spoofing ("Your CEO" <random@gmail.com>), which passes DMARC for gmail.com; (3) homograph/IDN visual spoofs; and (4) compromised legitimate accounts sending real authenticated mail. Coverage requires complementary controls: defensive domain registration and brand/domain monitoring for lookalikes, secure-email-gateway anti-impersonation rules keyed on executive display names and cousin domains, IDN/punycode detection, external-sender banners, and account-takeover detection plus MFA. DMARC is necessary but not sufficient — it's the anti-exact-domain-spoof layer only.
Signal: Realistic scoping of DMARC's limits and the surrounding defense-in-depth.

Q15. How does ARC actually preserve authentication, and what's the trust model risk?
ARC (RFC 8617) adds three header sets at each participating hop: ARC-Authentication-Results (the SPF/DKIM/DMARC verdict that hop observed), ARC-Message-Signature (a DKIM-like signature over the message as that hop saw it), and ARC-Seal (a signature over the chain of prior ARC headers, giving tamper-evidence for the whole chain). A downstream receiver that would otherwise DMARC-fail a forwarded message can validate the ARC chain, see that DMARC passed when the message entered a trusted intermediary (e.g., a mailing list), and override the failure to deliver. The trust-model risk is that ARC is only meaningful if you trust the sealer: any MTA can seal a chain asserting "DMARC passed," so blindly honoring arbitrary ARC chains reintroduces the spoofing DMARC prevents. Receivers therefore honor ARC only from an allowlist of vetted intermediaries and still weigh reputation. ARC reduces forwarding false-positives but must never be a blanket bypass.
Signal: Deep protocol knowledge plus the security-critical caveat about sealer trust.

Key Takeaways

  • SMTP trusts by default; SPF/DKIM/DMARC are DNS-published bolt-ons that add sender authorization, message integrity, and a policy tying both to the visible From.
  • SPF checks the envelope; DMARC checks the header From — alignment is the linchpin. SPF or DKIM passing alone doesn't protect the identity the user sees.
  • SPF has a hard 10-DNS-lookup limit; blowing it causes permerror and can reject your own mail. Prefer direct ip4: and -all in production.
  • DKIM is the more robust alignment method — it survives forwarding and IP changes — but proves origin, not intent (hence replay abuse). Use 2048-bit/Ed25519, sign from, omit l=, rotate via selectors.
  • DMARC passes if either aligned-SPF or aligned-DKIM passes; roll out p=none → quarantine → reject driven entirely by aggregate reports. Set sp=reject too.
  • Forwarding and mailing lists break SPF and DKIM; ARC repairs this by sealing prior authentication results — but only trust sealers on an allowlist.
  • MTA-STS + TLS-RPT (or DANE) secure the transport channel, which the content-authentication controls do not.
  • DMARC stops exact-domain spoofing only — cousin domains, display-name spoofing, and account takeover need SEG anti-impersonation, brand monitoring, and MFA.
  • Google/Yahoo's Feb 2024 bulk-sender rules made SPF+DKIM+DMARC a deliverability mandate for anyone sending 5,000+ messages/day.
  • Aggregate-report parsing is the operational heart of the program — without it you're blind to spoofing and stuck at p=none.
