Secure Email Gateways (SEG), ICES & Anti-Spoofing Architecture

Domain: Web / Email Security
Standards & refs: RFC 5321 (SMTP), RFC 5322 (IMF), RFC 7208 (SPF), RFC 6376 (DKIM), RFC 7489 (DMARC), RFC 8461 (MTA-STS), RFC 8460 (TLS-RPT), RFC 8617 (ARC), M³AAWG best-practice docs
Related notes: ../Email-Authentication/README.md (SPF/DKIM/DMARC internals) · ../Phishing-and-BEC/README.md (social-engineering tradecraft) · ../../../dlp/README.md and ../../../dlp/Enforcement-Channels/README.md (outbound DLP) · ../../../dlp/CASB/README.md
MITRE ATT&CK: T1566 (Phishing), T1566.001/.002/.003, T1114 (Email Collection), T1137 (Office App Startup), T1585/T1586 (Establish/Compromise Accounts), T1534 (Internal Spearphishing)

1. Overview

Email remains the single highest-volume initial-access vector in enterprise breaches: it is the transport for credential phishing, business email compromise (BEC), malware droppers, and internal lateral phishing after account takeover (ATO). A Secure Email Gateway (SEG) is the traditional control point — an MX-based mail relay that sits inline between the internet and the mailbox, inspecting every message before delivery. It performs connection reputation, anti-spam, anti-malware, attachment detonation, URL rewriting, impersonation analysis, DMARC enforcement, and outbound DLP/TLS. The modern architecture is bifurcating: as mailboxes moved into Microsoft 365 and Google Workspace (which ship native filtering — EOP/Defender and Google's Gmail security), a second class of product emerged — Integrated Cloud Email Security (ICES) — that connects via API behind the native filter, sees mail post-delivery, and can remediate messages already sitting in inboxes. A senior engineer must reason about both models, where each catches what the other misses, and how anti-spoofing (SPF/DKIM/DMARC/ARC + MTA-STS) is enforced across the chain.

TL;DR
- A SEG is an inline MX relay (blocks pre-delivery); an ICES is an API/journal add-on (detects and claws back post-delivery). They are complementary, not either/or.
- Anti-spoofing = SPF (path) + DKIM (signature) + DMARC (alignment + policy) + ARC (preserve auth across forwarding); the gateway enforces DMARC and adds impersonation/lookalike-domain analysis that DMARC cannot (cousin domains pass their own DMARC).
- Time-of-click URL rewriting and attachment sandbox detonation defeat weaponize-after-delivery and zero-day payloads; both have bypasses (CAPTCHA gates, geofencing, password-protected archives).
- MTA-STS + TLS-RPT stop opportunistic-TLS downgrade/MITM on the SMTP path; DANE is the DNSSEC-based alternative.
- Post-ATO threats (malicious inbox rules, internal phishing, mass mailbox export) are invisible to an inline gateway and are exactly what API-based detection exists to catch.

2. How It Works

2.1 Core mechanics

An email's journey crosses several trust boundaries, each an inspection opportunity. Inbound, the sending MTA connects to your published MX record, which points at the SEG (or directly at EOP/Gmail if you run ICES-only). The gateway terminates the SMTP conversation, applies connection- and content-layer verdicts, and either relays the accepted message to the mailbox provider or rejects/quarantines it. Outbound, the mailbox provider routes send-connectors through the gateway (or a cloud smart host) so DLP and encryption policy apply before mail leaves the org.

Decision logic (inbound, simplified): connection reputation → SMTP envelope checks (SPF at MAIL FROM, recipient validity) → DATA received → header/auth evaluation (DKIM verify, DMARC alignment, ARC) → content scoring (spam ML, impersonation, URL extraction + reputation, attachment static analysis) → sandbox detonation for unknowns (may hold the message) → verdict (deliver / spam-fold / quarantine / reject / redirect to admin). Verdicts are score-and-threshold, not binary: a message can accumulate risk points from many weak signals and cross a quarantine threshold even if no single rule fires.

2.2 Concrete configuration

Anti-spoofing DNS records (the foundation the gateway enforces). Publish a strict SPF that ends in a hard fail, a DKIM selector, and a DMARC policy that starts at none for monitoring then ratchets to reject:

; SPF — authorize only your senders; -all = hardfail everything else
example.com.        IN TXT "v=spf1 include:_spf.google.com include:spf.protection.outlook.com ip4:203.0.113.0/26 -all"

; DKIM — public key for selector "s1" (private key signs at the outbound MTA)
s1._domainkey.example.com. IN TXT "v=DKIM1; k=rsa; p=MIIBIjANBgkqh...QAB"

; DMARC — enforce, aggregate + forensic reporting, 100% coverage, strict alignment
_dmarc.example.com. IN TXT "v=DMARC1; p=reject; sp=reject; adkim=s; aspf=s; pct=100; rua=mailto:dmarc-agg@example.com; ruf=mailto:dmarc-fx@example.com; fo=1"

Commentary: -all (not ~all) is what lets a receiver reject forged path spoofing; adkim=s;aspf=s require the signing/return-path domain to match From: exactly (strict alignment) — this closes subdomain-relaxed abuse. sp=reject protects unused subdomains from being spoofed. See ../Email-Authentication/README.md for the full alignment math.

MTA-STS — force TLS on inbound SMTP and stop downgrade MITM. Publish a policy file and a DNS record; senders that support MTA-STS will refuse to deliver over cleartext or to a cert-mismatched MX:

_mta-sts.example.com. IN TXT "v=STSv1; id=20260707T000000Z"
_smtp._tls.example.com. IN TXT "v=TLSRPTv1; rua=mailto:tls-rpt@example.com"

# https://mta-sts.example.com/.well-known/mta-sts.txt  (served over HTTPS)
version: STSv1
mode: enforce
mx: mx1.example.com
mx: mx2.example.com
max_age: 604800

Commentary: start mode: testing (failures reported via TLS-RPT but mail still flows), watch the rua reports for a week, then flip to enforce. DANE (TLSA records + DNSSEC) is the alternative used heavily in the EU/German ecosystem; MTA-STS is easier because it rides on WebPKI instead of requiring DNSSEC.

Microsoft 365 / Exchange Online Protection — anti-phishing + impersonation (PowerShell). EOP/Defender policy that protects named VIPs and your own domains from impersonation and turns on mailbox intelligence:

New-AntiPhishPolicy -Name "VIP-AntiPhish" `
  -EnableTargetedUserProtection $true `
  -TargetedUsersToProtect "CEO Jane;jane@example.com","CFO Ravi;ravi@example.com" `
  -EnableTargetedDomainsProtection $true -TargetedDomainsToProtect "example.com" `
  -EnableOrganizationDomainsProtection $true `
  -EnableMailboxIntelligence $true -EnableMailboxIntelligenceProtection $true `
  -EnableSpoofIntelligence $true `
  -TargetedUserProtectionAction Quarantine `
  -TargetedDomainProtectionAction Quarantine `
  -PhishThresholdLevel 3   # 1=standard ... 4=most aggressive

# Safe Links (time-of-click URL protection) + Safe Attachments (detonation, Block mode)
New-SafeLinksPolicy -Name "SL-Default" -EnableSafeLinksForEmail $true `
  -ScanUrls $true -DeliverMessageAfterScan $true -EnableForInternalSenders $true
New-SafeAttachmentPolicy -Name "SA-Default" -Enable $true -Action Block -Redirect $false

Commentary: DeliverMessageAfterScan $true holds the message until detonation finishes (adds latency but prevents a "deliver-then-detonate" gap). EnableForInternalSenders $true is critical — it makes Safe Links rewrite internal mail so a compromised internal account's phishing links are still checked (defends T1534). PhishThresholdLevel/PhishThresholdLevel 3 trades false positives for catch rate; tune against your quarantine review capacity.

DMARC reject as an inbound rule at the SEG. On EOP, honor external senders' p=reject instead of the default "quarantine anyway" behavior:

Set-HostedContentFilterPolicy -Identity Default `
  -HonorDmarcPolicy $true `        # respect p=quarantine/p=reject from senders
  -SpoofQuarantineTag "SpoofItems"

Detect a malicious inbox rule via Microsoft Graph (post-ATO hunting). Attackers create hidden rules that auto-forward, delete, or move security alerts to hide their tracks (T1114.003 / T1564.008). Enumerate rules across mailboxes:

Connect-MgGraph -Scopes "MailboxSettings.Read","Mail.Read"
$suspect = @()
foreach ($u in (Get-MgUser -All)) {
  $rules = Get-MgUserMailFolderMessageRule -UserId $u.Id -MailFolderId "inbox" -ErrorAction SilentlyContinue
  foreach ($r in $rules) {
    $a = $r.Actions
    if ($a.ForwardTo -or $a.RedirectTo -or $a.Delete -or `
        ($r.Conditions.SubjectContains -match "invoice|password|payment|hacked|virus")) {
      $suspect += [pscustomobject]@{User=$u.UserPrincipalName; Rule=$r.DisplayName; `
        Fwd=$a.ForwardTo.EmailAddress.Address; Delete=$a.Delete; Move=$a.MoveToFolder}
    }
  }
}
$suspect | Format-Table -Auto

Commentary: the classic ATO tell is a rule named . or (single space) that moves anything containing "phish"/"invoice" straight to RSS Feeds or Deleted Items and forwards externally — this is what an inline SEG cannot see because the mailbox already accepted the mail. This is the core argument for API-based (ICES/journaling) coverage.

Case study — EchoSpoofing (2024). Researchers found that a large number of tenants routing outbound mail through Proofpoint had a permissive configuration: Proofpoint's shared outbound infrastructure would accept and re-send mail claiming to be from big brands because the "authorized sending" allowlist was too broad and SPF/DKIM were applied by Proofpoint on egress. Attackers relayed millions of spoofed messages that then carried legitimate Proofpoint DKIM signatures, sailing past DMARC. The fix was tenant-scoped allowlisting (X-headers / connector validation) so only genuinely authorized senders can use a given outbound path. Lesson: a gateway that signs on egress must strictly bind who is allowed to inject mail, or it becomes a spoofing laundromat.

Case study — CVE-2023-2868 (Barracuda ESG). A command-injection flaw in how the Email Security Gateway parsed the filenames inside .tar archives let UNC4841 (a China-nexus actor) achieve remote code execution simply by sending an email with a crafted attachment. Because the SEG opens every attachment to inspect it, the inspection engine became the exploit target. Barracuda ultimately advised replacing affected physical appliances rather than patching. Lesson: the detonation/parsing tier is high-value, internet-exposed, and must be sandboxed, network-segmented, and treated as an assume-breach zone — an SEG is not just a control, it is an asset that itself needs threat-modeling.

Case study — CVE-2023-23397 (Outlook zero-click). A malicious calendar/reminder message set a UNC path in PidLidReminderFileParameter; when Outlook processed the reminder it reached out over SMB and leaked the user's Net-NTLMv2 hash to the attacker — no click required. Content-scoring and URL rewriting do nothing here because there is no user-facing link; the payload is a property that triggers an outbound protocol call. Controls that actually helped: blocking outbound TCP/445 at the perimeter, Windows patch, and stripping/flagging messages with UNC-path properties at the gateway. Lesson: some "email attacks" are protocol-exploit delivery, and defense must include network egress control, not just content filtering.

Top 5 in depth

1. DMARC enforcement (p=reject with strict alignment). DMARC ties From: header identity to SPF/DKIM results via alignment: SPF aligns when the MAIL FROM domain matches From:; DKIM aligns when the d= signing domain matches From:. A message passes DMARC if at least one aligned mechanism passes, and the receiver applies the domain owner's policy (none/quarantine/reject) when it fails. Technically you deploy in stages: publish p=none; rua=..., collect aggregate XML reports for 2–4 weeks to enumerate every legitimate sender (marketing SaaS, ticketing, ERP), fix their SPF/DKIM, then move p=quarantine; pct=25 → pct=100 → p=reject with adkim=s; aspf=s. Trade-off: DMARC only protects the exact From: domain — it does nothing against lookalike/cousin domains (which pass their own DMARC), so it must be paired with impersonation analysis. Forwarding also breaks SPF (path changes) and can break DKIM (body rewrite), which is why ARC exists to carry a trusted chain of prior auth results. Full internals live in ../Email-Authentication/README.md.

2. Time-of-click URL rewriting (Safe Links / URL Defense). At delivery the gateway extracts every URL and replaces it with a proxy URL carrying an opaque token. When the user clicks — potentially days later — the request hits the vendor's click proxy, which re-evaluates reputation live, optionally detonates the destination, and either transparently redirects or shows a block page. This defeats the extremely common "clean at send, weaponized after delivery" pattern where an attacker sends a benign Google-Docs-hosted link then swaps the payload once the message has cleared filters. Concrete: EOP New-SafeLinksPolicy -ScanUrls $true -DeliverMessageAfterScan $true. Trade-offs: (a) it breaks link preview and some legitimate tracking, (b) attackers hide behind CAPTCHA/geofencing so the proxy sees a benign page, and (c) the proxy becomes a single point of latency and a juicy target. Pair with browser isolation for high-risk clicks.

3. Attachment sandbox detonation + Content Disarm & Reconstruction (CDR). Unknown files are executed in an instrumented VM/container that watches for malicious behavior (process injection, C2 beacons, persistence, macro auto-exec). Verdict feeds the deliver/block decision. Because detonation is slow and evadable, layer it with CDR (a.k.a. "file sanitization"): rather than judging a document, CDR rebuilds it — stripping macros, embedded objects, and active content to produce a flat, safe copy. Concrete: EOP New-SafeAttachmentPolicy -Enable $true -Action Block; Dynamic Delivery delivers the body immediately and swaps the attachment in after the verdict. Trade-offs: detonation adds delivery latency and is beaten by environment-aware malware (checks for VM artifacts, sleeps, requires human interaction); CDR can break legitimate macros/embedded content and cannot detonate URLs. Best practice: block password-protected archives you can't inspect, and back everything with endpoint EDR because no email control is 100%.

4. Impersonation & lookalike-domain protection. This ML/heuristic layer is the primary technical defense against payload-less BEC that DMARC cannot touch. Signals: display-name matches a protected VIP but the address doesn't; sending domain is a homoglyph/typo of yours (examp1e.com, IDN аpple.com), or newly registered (age from WHOIS/CT logs); anomalous header/routing vs the sender's history (mailbox intelligence); first-contact "high risk" between two parties. Concrete: EOP -EnableTargetedUserProtection, -EnableMailboxIntelligenceProtection, plus an external-sender safety tip banner. Defensive-registration of your own lookalikes and monitoring CT logs for cousin-domain registration are cheap wins. Trade-off: high false-positive potential on legitimate look-similar partners; needs a VIP list that stays current and a fast release/allow workflow.

5. Outbound DLP + transport encryption (MTA-STS/TLS). Egress is where insider risk and accidental leakage are caught. Outbound DLP applies regex/dictionaries (credit-card, SSN/national-ID patterns), exact-data-match fingerprinting of known-sensitive datasets, and ML classifiers; policy verdicts are encrypt / block / quarantine-for-review / log-only. Cross-reference ../../../dlp/Enforcement-Channels/README.md (email as a DLP channel) and ../../../dlp/Content-Inspection/README.md. Transport-wise, MTA-STS enforce + TLS-RPT guarantees mail to partners can't be silently downgraded to cleartext by a STARTTLS-stripping MITM, and forced-TLS connectors ensure regulated data never traverses plaintext SMTP. Trade-offs: DLP false positives frustrate users and drive shadow-IT workarounds (personal webmail/USB), so tune with allow-contexts and business-justification prompts; MTA-STS enforce can block legitimate mail to a partner whose TLS breaks, so monitor TLS-RPT before enforcing.

6. Detection & Response

What to log / telemetry sources

  • SEG/ICES message logs: verdict, spam score, impersonation flags, URL-click events, detonation verdicts, sender IP/ASN, DMARC/SPF/DKIM result, quarantine actions.
  • M365: Unified Audit Log — New-InboxRule/Set-InboxRule, Set-Mailbox ForwardingSmtpAddress, Add-MailboxPermission, UpdateInboxRules; Defender EmailEvents, EmailUrlInfo, UrlClickEvents, EmailAttachmentInfo; sign-in logs (impossible travel, risky sign-in).
  • Google Workspace: Admin/Investigation logs, Gmail log events (quarantine, link_clicked), rule/forwarding change events, drive/exfil correlation.
  • DNS/transport: DMARC rua aggregate reports, TLS-RPT reports, MX/SPF change monitoring.
  • Auth/IdP: MFA challenges, token issuance (OAuth grant to unusual app = consent phishing).

Sample detections

KQL (Microsoft Sentinel / Defender) — suspicious inbox rule creation right after a risky sign-in:

let riskySignins = SigninLogs
  | where RiskLevelDuringSignIn in ("high","medium")
  | project UserPrincipalName, riskTime = TimeGenerated, IPAddress;
OfficeActivity
| where Operation in ("New-InboxRule","Set-InboxRule","UpdateInboxRules")
| extend params = tostring(Parameters)
| where params has_any ("ForwardTo","RedirectTo","DeleteMessage","MoveToFolder")
| where params matches regex @"(?i)(invoice|payment|password|phish|hacked|virus|helpdesk)"
   or params has_any ("RSS Feeds","Deleted Items","Conversation History")
| join kind=inner riskySignins on $left.UserId == $right.UserPrincipalName
| where TimeGenerated between (riskTime .. riskTime + 24h)
| project TimeGenerated, UserId, Operation, params, IPAddress

SQL — cluster inbound messages from newly registered lookalike domains (BEC hunt):

SELECT sender_domain,
       COUNT(*)                         AS msg_count,
       COUNT(DISTINCT recipient)        AS recips,
       MIN(received_at)                 AS first_seen,
       AVG(levenshtein(sender_domain,'example.com')) AS edit_distance
FROM email_events
WHERE received_at > NOW() - INTERVAL '7 days'
  AND domain_age_days < 30                       -- freshly registered
  AND levenshtein(sender_domain,'example.com') BETWEEN 1 AND 2   -- cousin/homoglyph
GROUP BY sender_domain
HAVING COUNT(DISTINCT recipient) > 3             -- targeting multiple staff
ORDER BY msg_count DESC;

Response playbook (ATO + malicious inbox rule)

  1. Contain identity: revoke sessions/tokens (Revoke-MgUserSignInSession), force password reset, require re-registration of MFA (assume the attacker enrolled their own authenticator or consented an OAuth app).
  2. Kill persistence: enumerate and delete malicious inbox rules and forwarding (Remove-InboxRule, clear ForwardingSmtpAddress); revoke suspicious OAuth app grants (consent phishing).
  3. Claw back mail: use ICES/Search-Mailbox/Defender "Soft delete/Move to junk" to remediate messages the compromised account sent internally (internal phishing) and any inbound phish that seeded the ATO — across all mailboxes, not just the victim.
  4. Scope the blast radius: query audit logs for mailbox exports, Add-MailboxPermission, and Graph Mail.Read API pulls; correlate with sign-in IPs/ASNs to find other compromised accounts.
  5. Eradicate delivery infra: block the sender domain/IP, add URLs to the click-proxy blocklist, submit samples to detonation for signatures, and update impersonation VIP lists if an exec was spoofed.
  6. Recover & harden: restore normal mail flow, verify DMARC/MTA-STS still enforced, and if a lookalike domain was used, file takedown (registrar/CERT) and defensively register the variant.
  7. Post-incident: tune the detection thresholds that missed/over-fired, add a rule-creation alert, and run a user-awareness reset for the targeted group.

7. Practice Questions

Tier 1 — Fundamentals

Q1. What is the difference between a Secure Email Gateway and an ICES?
A SEG is an inline, MX-based mail relay: DNS MX points at it, so it inspects and blocks messages before they reach the mailbox (pre-delivery accept/reject/quarantine). An ICES connects via API (Microsoft Graph / Gmail API) or journaling and reads messages after the provider's native filter delivered them, letting it detect and remediate post-delivery — including threats an inline device never sees, like malicious inbox rules and internal phishing. The two are complementary: SEG gives pre-delivery blocking and outbound control; ICES gives post-delivery detection and claw-back with no MX change.
Signal: Do they understand the architectural shift and why both layers exist?

Q2. How do SPF, DKIM, and DMARC combine to stop spoofing?
SPF authorizes which IPs may send for a domain (checked against the MAIL FROM path); DKIM cryptographically signs headers/body so tampering and origin can be verified via a DNS-published public key; DMARC ties both to the visible From: header through alignment and tells receivers what to do on failure (none/quarantine/reject) plus where to send reports. A message passes DMARC if at least one aligned mechanism passes. Together they stop exact-domain spoofing, but none of them stop lookalike/cousin domains, which is why impersonation analysis is still needed.
Signal: Can they connect the three and know DMARC's alignment concept, not just name them?

Q3. What is time-of-click URL protection and what does it defend against?
The gateway rewrites URLs at delivery so clicks route through a proxy that re-checks reputation and can detonate the destination at click time. It defends against "weaponize-after-delivery," where a link is benign when the message passes filters but is swapped to a malicious payload afterward. Static receive-time checks miss this entirely. Limits: attackers hide behind CAPTCHA/geofencing so the proxy sees a clean page.
Signal: Understands the temporal gap in static scanning.

Q4. Why block or specially handle password-protected archives?
An encrypted .zip/.7z can't be opened by the AV or sandbox, so a malicious payload is invisible to content inspection — a classic evasion. Options: block them outright, quarantine for manual review, or attempt password extraction from the email body (many phish helpfully include the password). It's a deliberate detection-gap you must policy around rather than pretend to scan.
Signal: Recognizes inspection blind spots.

Q5. What is MTA-STS and what problem does it solve?
SMTP TLS is opportunistic by default — a MITM can strip STARTTLS and force cleartext, or present a bad cert, with no failure. MTA-STS publishes a policy (via DNS + an HTTPS .well-known file) declaring that senders must use TLS with a valid cert to your MX; compliant senders refuse to downgrade. TLS-RPT gives you reports of failures. DANE (TLSA + DNSSEC) is the alternative. It closes SMTP downgrade/interception.
Signal: Knows transport security is separate from content/auth.

Tier 2 — System design / trade-offs

Q6. Design email security for a 50,000-seat company on Microsoft 365. SEG, ICES, or both?
I'd run layered: keep EOP/Defender as the native baseline (it's already inline and free with the license), and add an ICES via Graph API for post-delivery detection, internal-phishing catch, and one-click claw-back without changing MX or adding mail-flow latency. Whether to also put a third-party SEG inline depends on requirements: a SEG adds richer outbound DLP, mature encryption/large-file handling, and vendor-diverse detonation, but doubles mail-flow complexity and creates a second outage domain. For most M365-native shops in 2026 I'd lead with Defender + ICES and only add an inline SEG if outbound DLP/compliance or a specific detection gap demands it. I'd enforce DMARC p=reject, MTA-STS, Safe Links/Attachments with internal-sender scanning, and wire ICES remediation into SOAR. The deciding trade-off is latency/complexity of inline vs the post-delivery visibility of API.
Signal: Balances cost, latency, coverage, and operational blast radius rather than dogmatically picking one.

Q7. How would you roll out DMARC to p=reject without breaking legitimate mail?
Publish p=none; rua=...; ruf=...; fo=1 first and ingest aggregate reports (via a DMARC analyzer) for 2–4 weeks to enumerate every legitimate sender — marketing SaaS, ticketing, payroll, ERP, third parties sending "as" you. Fix each to align (add their IPs to SPF or, better, get them to DKIM-sign with your domain). Then ratchet: p=quarantine; pct=25 → pct=100 → p=reject, watching reports at each step for a spike in failures. Set adkim=s; aspf=s once you're confident and sp=reject to protect subdomains. The hard part is discovery and the long tail of shadow senders; ARC handles forwarding/mailing-list breakage. Rushing to reject without the monitoring phase is how you black-hole invoices.
Signal: Knows it's a phased, data-driven rollout, not a one-line change.

Q8. Where do you enforce outbound DLP — endpoint, gateway, or provider — and why?
Email is one channel; I'd enforce DLP in defense-in-depth with email as one enforcement point. At the email egress (SEG outbound connector or provider DLP) I catch regulated data leaving via mail with policy verdicts encrypt/block/quarantine. But email DLP alone misses uploads, USB, and pastes, so I pair it with endpoint DLP and CASB/SSE for web/SaaS channels (cross-ref ../../../dlp/CASB/README.md and ../../../dlp/Enforcement-Channels/README.md). The trade-off is false positives driving users to shadow channels — so tune with exact-data-match fingerprinting over crude regex, add business-justification prompts, and log-only before hard-block. Encryption gateways let legitimate sensitive mail flow securely instead of being blocked.
Signal: Systems view of DLP across channels, not just email.

Q9. Your ICES catches a phish that Defender delivered. What does that tell you and what do you change?
It tells me native filtering has a real-time gap the API layer covered post-delivery — the value proposition of ICES. I'd analyze why Defender missed it (novel URL not yet in reputation, HTML smuggling, look-alike sender passing spoof intelligence), submit the sample to improve signatures, and check whether the message was clicked before remediation (UrlClickEvents) to scope impact. Longer term I'd tune Defender policy (raise PhishThresholdLevel, enable internal-sender Safe Links), add the pattern to detection rules, and if this class recurs, evaluate an inline SEG for pre-delivery blocking so I'm not relying on claw-back speed. The key metric is time-to-remediate vs time-to-click.
Signal: Uses the miss to drive layered improvement, understands claw-back is a race.

Q10. Design detection for account takeover manifesting as internal phishing.
Correlate three signal families: (1) identity — risky/impossible-travel sign-in, new MFA method, OAuth app consent; (2) mailbox config — new inbox rule forwarding/deleting, added delegate/permission, forwarding address set; (3) mail behavior — a normally-external-facing user suddenly blasting internal recipients, reply-chain hijacking, or a burst of similar messages. I'd build a KQL/SIEM rule joining risky sign-ins to rule-creation within 24h (see §6), enable internal-sender content scanning so east-west phish is inspected, and auto-trigger SOAR: revoke sessions, disable rules, claw back the internal messages across all recipients. Detection must span identity and mail planes because either alone has too many false positives.
Signal: Cross-plane correlation and automated response design.

Tier 3 — Expert / deep-dive

Q11. Explain how forwarding breaks DMARC and how ARC fixes it.
When a message is forwarded (mailing list, auto-forward, .forward), the forwarder becomes the new sending IP, so SPF fails at the final receiver (path no longer matches the origin domain). DKIM may survive if headers/body are untouched, but lists that add footers, subject tags, or re-encode MIME break the DKIM signature too — so both aligned mechanisms fail and a legit message hits p=reject. ARC (RFC 8617) solves this: each intermediary adds an ARC-Seal, ARC-Message-Signature, and ARC-Authentication-Results recording the auth results it saw before modifying the message. A receiver that trusts the forwarder can validate the ARC chain, see that DMARC passed upstream, and choose to honor that instead of failing. The catch: ARC is trust-based — you only honor chains from forwarders you trust, or an attacker forges a "passed upstream" claim.
Signal: Deep grasp of the forwarding failure mode and the trust model of the fix.

Q12. How does sandbox detonation get evaded, and how do you harden it?
Malware fingerprints the analysis environment: checks for VM artifacts (MAC OUI, disk size, drivers), low CPU/RAM, absence of user files/browser history, no mouse movement, or specific process names; then sleeps past the detonation window, requires human interaction (click, close-dialog), or geofences/CAPTCHA-gates the payload so it only fires for the real target. It may also stage — the attachment is benign and only fetches the payload later (which time-of-click helps with). Hardening: use bare-metal or hardened hypervisors with realistic artifacts and user activity simulation, randomize detonation duration, run multiple OS/patch environments, re-detonate URLs periodically, and combine detonation with CDR (rebuild the file safely instead of judging it) so evasion of judgment doesn't matter. Ultimately back it with endpoint EDR — no sandbox is complete.
Signal: Concrete evasion tradecraft plus a layered mitigation strategy.

Q13. Walk through the EchoSpoofing-class abuse of a shared outbound SEG and how you'd prevent it.
A cloud SEG that applies SPF/DKIM on egress signs mail as it leaves. If the "authorized to send through this path" binding is too broad — e.g., any customer's connector can inject mail claiming any tenant's domain — an attacker relays spoofed messages that then carry the SEG's legitimate DKIM signature and pass DMARC at the recipient. The 2024 Proofpoint case saw millions of spoofed brand messages laundered this way. Prevention is strict tenant-scoped authorization on injection: bind each outbound connector to a specific tenant/domain set, require authenticated/validated headers (e.g., X-OriginatorOrg checks), don't sign for domains the injecting tenant doesn't own, and monitor egress volume anomalies. It's fundamentally an authorization flaw in a signing intermediary — the same class as an open relay, just with DKIM making it worse.
Signal: Understands that a signing intermediary is an authorization boundary that can be abused.

Q14. The SEG's own attachment parser is a target (CVE-2023-2868). How do you threat-model the gateway as an asset?
The gateway opens, parses, and detonates hostile input from the entire internet by design — it is a maximum-exposure asset, not just a control. I'd threat-model it as assume-breach: (1) segment the detonation/parsing tier into an isolated network zone with no path to production identity or mailbox data, egress-filtered so a compromised node can't C2; (2) treat parser/appliance CVEs as P1 patch with rapid vendor-advisory monitoring (Barracuda advised replacing appliances — have a swap plan); (3) run the management plane behind strong auth/MFA and never expose it to the internet; (4) rotate any certs the gateway uses for connector auth (Mimecast 2021 cert theft); (5) monitor the appliance's own outbound connections and process behavior with EDR. The mental model: the security tool is part of your attack surface and supply chain.
Signal: Sees the control as an asset; supply-chain and blast-radius thinking.

Q15. How would you detect and defend against QR-code phishing (quishing) and HTML smuggling that bypass URL and attachment scanning?
Both evade classic content scanning by not presenting a scannable URL or file. Quishing embeds the malicious URL in a QR image the user scans with a phone (off the managed network, no endpoint controls) — defeated by OCR/QR-decoding of inbound images so the embedded URL is extracted and reputation-checked/rewritten, plus flagging mails that are image-heavy with little text and a QR from an external first-contact sender. HTML smuggling ships a benign-looking HTML attachment containing an encoded blob that JavaScript reassembles into a malicious file in the browser, past the gateway which only saw HTML — defeated by CDR/stripping active content from HTML attachments, blocking or sandboxing HTML/.svg attachments, and endpoint controls (MOTW, AMSI, SmartScreen) that catch the reconstructed file on write/exec. Both underscore that gateway content inspection has structural blind spots and must be backed by endpoint detection and user reporting.
Signal: Awareness of current (2024–2026) evasion techniques and multi-layer mitigation.

9. Key Takeaways

  • SEG = pre-delivery inline block; ICES = post-delivery API detection/claw-back. They cover different failure modes; mature orgs layer both on top of native EOP/Gmail filtering.
  • Anti-spoofing is a stack: SPF (path) + DKIM (signature) + DMARC (alignment + policy) + ARC (forwarding) + MTA-STS/TLS (transport). Each covers a distinct gap; none alone is sufficient.
  • DMARC p=reject stops exact-domain spoofing but not lookalike/cousin domains — those pass their own DMARC, so impersonation/anomaly ML is mandatory for BEC.
  • Time-of-click rewriting and detonation defeat weaponize-after-delivery and unknown payloads, but both are evadable (CAPTCHA/geo, VM-aware malware, encrypted archives) — back with CDR and endpoint EDR.
  • Post-ATO threats (malicious inbox rules, internal phishing, mailbox export) are invisible to inline gateways — API-based monitoring of rule/forwarding changes correlated with risky sign-ins is the detection.
  • The gateway itself is an internet-exposed asset (Barracuda CVE-2023-2868, Mimecast cert theft) — segment, patch as P1, and threat-model it as attack surface and supply chain.
  • A signing outbound intermediary is an authorization boundary — over-broad authorized-sender binding turns it into a spoofing laundromat (EchoSpoofing 2024).
  • Roll DMARC out in monitored stages (none→quarantine→reject) driven by rua report analysis; enforce MTA-STS in testing before enforce using TLS-RPT.
  • Outbound DLP on email is one channel — combine with endpoint/CASB DLP and tune to avoid pushing users to shadow channels (cross-ref dlp/).
  • Response to ATO is a race: revoke tokens, kill inbox-rule persistence, and claw back internal messages across all mailboxes before the phish spreads.
