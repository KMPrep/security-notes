Directory Traversal (Path Traversal)

Category: Server-Side Attacks — Filesystem Exposure
OWASP: A01:2021 — Broken Access Control
CWE: CWE-22
Applies To: Any application that reads, serves, or processes files using paths derived from user input

Overview

Directory traversal (also called path traversal) is an attack where an attacker manipulates file path inputs with ../ sequences to navigate outside the intended directory and access arbitrary files on the server. The target is the server's filesystem: configuration files, credentials, private keys, source code, password databases, and OS files — anything the web server process can read.

The attack is simple in concept but broad in impact. A vulnerable file-download endpoint, a log viewer, a template renderer, or even an image-serving route can become a mechanism for reading /etc/passwd, ~/.ssh/id_rsa, web.config, .env, or C:\Windows\System32\config\SAM. Because the web server process often runs with elevated permissions, the accessible file scope is wide.

Directory traversal is frequently a stepping stone: exposed credentials or SSH keys lead to full server compromise. Combined with Local File Inclusion (LFI), traversal enables remote code execution via log poisoning.

How It Works

Core Mechanics

A vulnerable application constructs file paths by appending user input to a base directory:

# VULNERABLE
base_dir = "/var/www/uploads/"
filename = request.args.get("file")
filepath = base_dir + filename
with open(filepath) as f:
    return f.read()

If filename is ../../../../etc/passwd, the resolved path is /etc/passwd.

The ../ Sequence:
Each ../ moves one directory level up in the filesystem:

/var/www/uploads/../../../etc/passwd
→ /var/www/etc/passwd        (after first ../)
→ /var/etc/passwd            (after second ../)
→ /etc/passwd                (after third ../)

Attack Variants and Bypass Techniques

Variant Payload Why Used
Basic ../../../../etc/passwd Direct traversal on Linux
Windows ..\..\..\Windows\System32\drivers\etc\hosts Windows path separator
URL-encoded %2e%2e%2f%2e%2e%2f%2e%2e%2fetc%2fpasswd Bypass filters on literal ../
Double URL-encoded %252e%252e%252f Bypass single-decode filters
Unicode / UTF-8 ..%c0%af../etc/passwd Legacy IIS / Java traversal
Null byte ../../../../etc/passwd%00.jpg Bypass extension checks (PHP < 5.3)
Mixed slashes ....//....//etc/passwd Bypass naive ../ stripping
Absolute path /etc/passwd When app prepends nothing, direct absolute path

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 Basic traversal Canonicalize the resolved path and verify it starts with the intended base directory before opening File access layer in every handler
2 Encoding bypasses Resolve the real path after URL decoding and normalization; compare the canonical absolute path to the allowed base Path resolution logic
3 Arbitrary file read Use an allowlist of permitted filenames or file IDs; map user-facing IDs to server paths server-side, never accept paths directly API design — indirect reference maps
4 Extension bypass Do not rely on extension checks as a security control; check canonical path against base directory instead File access layer
5 Sensitive files accessible Run the web server process with a dedicated low-privilege user; chroot or containerize to limit accessible filesystem scope OS / deployment hardening
6 Directory listings Disable directory listing on web server (Apache Options -Indexes, Nginx autoindex off) Web server configuration
7 Source code exposure Store application source and config outside the web root; never serve files from the project root Application deployment layout
8 Cloud credential exposure Use IMDSv2 on AWS (requires session token, not directly reachable via simple traversal); apply network-level metadata IP blocking Cloud configuration

Secure Path Validation Pattern

import os

BASE_DIR = "/var/www/uploads"

def safe_file_read(user_filename: str) -> bytes:
    # Resolve the canonical absolute path (resolves .., symlinks, etc.)
    requested = os.path.realpath(os.path.join(BASE_DIR, user_filename))

    # Verify the resolved path is still within the allowed base directory
    if not requested.startswith(BASE_DIR + os.sep):
        raise PermissionError("Access denied: path traversal attempt")

    with open(requested, "rb") as f:
        return f.read()

Better: indirect reference map — never expose filesystem paths at all:

ALLOWED_FILES = {
    "q1-report": "/var/www/uploads/q1-report.pdf",
    "q2-report": "/var/www/uploads/q2-report.pdf",
}

def safe_file_read(file_id: str) -> bytes:
    filepath = ALLOWED_FILES.get(file_id)
    if not filepath:
        raise ValueError("Unknown file ID")
    with open(filepath, "rb") as f:
        return f.read()

Top 3 Critical Mitigations

  1. Canonicalize the resolved path and compare it to the base directory. After resolving .. sequences and symlinks (os.path.realpath() or equivalent), verify the resulting absolute path starts with the intended base directory. This catches all encoding variants and ../ permutations because canonicalization happens at the OS level, not via string matching.
  2. Use indirect reference maps instead of user-supplied paths. Never accept a file path from user input. Instead, accept an opaque file ID (a UUID, a slug, a numeric ID) and map it server-side to the actual path. The user never touches the filesystem path — traversal is impossible by design.
  3. Run the web server process with minimal filesystem permissions. Use a dedicated low-privilege OS user. Containerize or chroot the application so the accessible filesystem is scoped to only what the application legitimately needs. Even if traversal succeeds, the attacker can only read what the process can read — eliminating access to SSH keys, shadow files, and secrets outside the container boundary.

Key Takeaways

  • Directory traversal uses ../ sequences to escape the intended directory and read arbitrary files on the server filesystem.
  • Filtering ../ strings is insufficient — URL encoding, double encoding, and Unicode variants bypass naive string-based filters.
  • Canonical path resolution followed by base directory prefix check is the correct technical fix.
  • Indirect reference maps (mapping user-supplied IDs to server paths server-side) make traversal impossible by design.
  • High-value targets include /etc/passwd, SSH private keys, .env files, application configs, and logs (which enable RCE via log poisoning with LFI).
  • Run with minimal OS permissions and in containers to limit blast radius even if traversal is exploited.
