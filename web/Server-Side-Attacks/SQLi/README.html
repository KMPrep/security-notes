SQL Injection (SQLi)

Category: Server-Side Attacks — Database Exploitation
OWASP: A03:2021 — Injection (historically #1 for over a decade)
CWE: CWE-89
Applies To: Any application that constructs SQL queries using user-supplied input

Overview

SQL Injection is an attack where user-controlled input is interpreted as SQL syntax, allowing the attacker to manipulate database queries. Instead of treating input as data, the database engine executes it as code. SQLi can expose the entire database, bypass authentication, modify or delete data, and in some configurations escalate to OS-level command execution.

SQLi has been the most prevalent and damaging web vulnerability for decades. Notable breaches caused by SQLi include the 2008 Heartland Payment Systems breach (130 million card records), the 2012 LinkedIn breach (117 million credentials), and the 2011 Sony PlayStation Network breach (77 million accounts). Despite being well-understood and preventable with parameterized queries, SQLi continues to appear in production systems due to legacy code, dynamic query building, and ORM misuse.

At Google scale, any application that builds SQL queries from user input — search boxes, filters, URL parameters, API fields — is a potential SQLi surface. The fix is universal and non-negotiable: parameterized queries or prepared statements everywhere.

How It Works

Core Mechanics

SQLi occurs when string concatenation is used to build SQL queries:

# VULNERABLE: user input concatenated directly into SQL
query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'"

If username is set to admin' --, the query becomes:

SELECT * FROM users WHERE username = 'admin' --' AND password = '...'

The -- comments out the password check — authentication bypassed with no credentials.

Types of SQL Injection

Type Mechanism Output Visible? Use Case
Classic / Error-based Malformed syntax triggers DB error with data Yes (in error) Fast enumeration via error messages
UNION-based Appends UNION SELECT to extract data into normal response Yes (in response) Dump arbitrary tables into response
Blind Boolean Changes query condition; infers data from true/false responses No Extract data bit by bit via branching
Blind Time-based Uses SLEEP() / WAITFOR DELAY; infers from response time No Works when no output visible at all
Out-of-band Triggers DNS or HTTP request to attacker-controlled server External Exfiltrate data via DNS/HTTP when blind
Second-order Payload stored in DB; executed when retrieved and used later Deferred Bypasses input validation on initial insert

Attack Examples

Authentication Bypass:

-- Input: username = admin'--
SELECT * FROM users WHERE username = 'admin'--' AND password = ''
-- Password check commented out; logs in as admin

UNION-based Data Extraction:

-- Injected: ' UNION SELECT username, password, null FROM users--
SELECT id, name, email FROM products WHERE id = '' UNION SELECT username, password, null FROM users--'
-- Returns username/password from users table in product response

Blind Boolean Inference:

-- Is first char of admin password 'a'?
' AND SUBSTRING((SELECT password FROM users WHERE username='admin'),1,1)='a'--
-- True response = page loads normally; False = different behavior

Time-based Blind:

-- MySQL: if admin exists, delay 5 seconds
'; IF (SELECT COUNT(*) FROM users WHERE username='admin')>0 WAITFOR DELAY '0:0:5'--

OS Command Execution (MSSQL xp_cmdshell):

'; EXEC xp_cmdshell('whoami'); --
-- Runs OS command on the database server if xp_cmdshell is enabled

Second-Order SQLi

Input is safely stored but becomes an injection vector when later retrieved and used in another query without sanitization:

flowchart TD
    A[Attacker registers username: admin'--] --> B[App escapes on insert, stores safely in DB]
    B --> C[Later: admin changes password]
    C --> D[App fetches username from DB, uses it raw in UPDATE query]
    D --> E["UPDATE users SET password='x' WHERE username='admin'--'"]
    E --> F[Password of real admin account changed without auth]

Mitigation Controls

# Failure Mitigation Layer / Where Applied
1 All SQLi types Use parameterized queries (prepared statements) for every SQL query — this is the primary fix ORM / database access layer
2 All SQLi types Use an ORM (SQLAlchemy, Hibernate, ActiveRecord) correctly; never call .raw() or equivalent with user input Application code
3 Auth bypass Never build login queries via concatenation; use parameterized auth queries Authentication module
4 Privilege escalation via DB Principle of least privilege: app DB user should have only SELECT/INSERT/UPDATE on needed tables; never DBA rights Database configuration
5 OS command execution Disable xp_cmdshell on MSSQL; audit DB server permissions; do not run DB as root Database hardening
6 Error-based enumeration Suppress database error details in production responses; log errors server-side only Error handling middleware
7 Second-order SQLi Treat data from DB as untrusted if it will be used in subsequent queries; parameterize those too All query construction sites
8 WAF bypass WAF is defense-in-depth, not a primary fix; fix the root cause (parameterized queries) first API gateway / WAF layer
9 Blind extraction Enforce query timeouts to limit time-based blind attacks; rate-limit endpoints Database + application layer
10 Schema exposure Remove access to information_schema for app DB user where possible Database access controls

Secure vs Insecure Query Patterns

# INSECURE: string concatenation
query = "SELECT * FROM users WHERE email = '" + email + "'"

# INSECURE: f-string / format string
query = f"SELECT * FROM users WHERE email = '{email}'"

# SECURE: parameterized query (Python DB-API)
cursor.execute("SELECT * FROM users WHERE email = %s", (email,))

# SECURE: SQLAlchemy ORM
user = session.query(User).filter(User.email == email).first()

# INSECURE ORM (raw): still vulnerable if input interpolated
session.execute(f"SELECT * FROM users WHERE email = '{email}'")

# SECURE ORM (raw with bind parameter):
session.execute(text("SELECT * FROM users WHERE email = :email"), {"email": email})

Top 3 Critical Mitigations

  1. Use parameterized queries everywhere, without exception. Parameterized queries (prepared statements) separate SQL code from data at the protocol level — the database engine never interprets user input as syntax. This single control prevents all classic SQLi. Any codebase that concatenates user input into SQL strings is vulnerable regardless of other controls.
  2. Apply least privilege to database accounts. The application's database user should have only the permissions it needs — SELECT, INSERT, UPDATE on specific tables. It should never have DROP, CREATE, GRANT, EXECUTE (for stored procs), or DBA rights. This limits the blast radius if SQLi does occur: an attacker who injects into a read-only connection cannot drop tables or run OS commands.
  3. Never expose database error messages to users. Error-based SQLi is the fastest attack path — detailed error messages tell attackers exactly which table, column, and query structure to target. Return generic error codes to clients and log the full error server-side. This converts error-based SQLi into slower blind SQLi, significantly raising the attack cost.

Practice Questions

  1. What is the difference between error-based, UNION-based, and blind SQL injection? When would you use each?
  2. Why do parameterized queries prevent SQL injection at a fundamentally different level than input sanitization or escaping?
  3. What is second-order SQL injection and why does it bypass defenses that work for first-order injection?
  4. An application uses an ORM. A developer says "we can't have SQLi because we use SQLAlchemy." Under what conditions is this still false?
  5. How would you exploit SQLi on a site that returns the same response for both true and false conditions (no visible error, no visible data difference)?
  6. MSSQL's xp_cmdshell is disabled. How else could an attacker achieve OS-level impact via SQLi on MSSQL?
  7. Describe the steps you would take to perform a time-based blind SQL injection attack to extract an admin password hash.
  8. A WAF is blocking your UNION SELECT attempts. What bypass techniques would you try, and why do they work?
  9. How would you design a secure data access layer for a multi-tenant SaaS application to prevent both SQLi and tenant data isolation failures?
  10. What database-level controls complement application-level parameterized queries to reduce SQLi blast radius?

Key Takeaways

  • SQLi occurs when user input is concatenated into SQL queries, allowing the database to interpret input as code.
  • Parameterized queries (prepared statements) are the definitive fix — they separate SQL code from data at the protocol level.
  • Attack types escalate from auth bypass and data exposure to OS command execution depending on DB type and permissions.
  • Second-order SQLi bypasses input-time defenses by exploiting data stored safely but later used unsafely in another query.
  • Least-privilege DB accounts limit blast radius: a read-only connection cannot drop tables or run OS commands.
  • Never expose DB error details to users — they are the primary enabler of fast, automated enumeration.
  • ORMs are not a silver bullet: raw query escapes, .raw() calls, and string-formatted ORM queries remain vulnerable.
