SSRF — Server-Side Request Forgery

Category: Server-Side Attacks — Internal Network Pivoting
OWASP: A10:2021 — SSRF (newly promoted to standalone category)
CWE: CWE-918
Applies To: Any application that fetches URLs or makes outbound HTTP requests based on user input

Overview

Server-Side Request Forgery is an attack where the attacker causes the server to make HTTP requests to arbitrary destinations — including internal network addresses, cloud metadata services, and loopback interfaces — that the attacker could not reach directly. The server acts as an unintentional proxy, forwarding requests from the attacker to targets inside the private network or to cloud metadata APIs that expose credentials.

SSRF became a critical vulnerability category in cloud environments. The most well-known exploitation of SSRF was the 2019 Capital One breach, where SSRF was used to access the AWS EC2 instance metadata service (169.254.169.254) and retrieve IAM role credentials, ultimately leading to the exfiltration of over 100 million customer records. The attack was so impactful that AWS redesigned their IMDSv2 to require session-oriented authentication specifically to mitigate SSRF-based attacks on the metadata service.

At Google Cloud scale, SSRF in a GCP application exposes the GCP metadata server (metadata.google.internal), which can return Google service account tokens, project data, and SSH keys — instantly providing full GCP project access.

How It Works

Core Mechanics

SSRF exists when an application:

  1. Accepts a URL or domain from user input
  2. Makes an outbound HTTP request to that URL from the server
  3. Does not validate that the URL is limited to expected external destinations

Common SSRF Entry Points:

Cloud Metadata Service — The Critical Target

AWS IMDSv1 is the canonical SSRF-to-credential-theft path:

    WebApp->>IMDS: GET /latest/meta-data/ (no auth required in IMDSv1)
    IMDS-->>WebApp: ami-id, hostname, iam/ (list of available paths)

    IMDS-->>WebApp: AccessKeyId, SecretAccessKey, Token (full IAM creds)

    Attacker->>AWS: Use stolen IAM creds to call S3, RDS, etc.

SSRF Bypass Techniques

Naive allowlist/blocklist-based SSRF filters are commonly bypassed:

Bypass Technique Method
Localhost aliases http://127.1/, http://127.0.0.1/, http://[::1]/, http://0x7f000001/
DNS to internal http://attacker.com resolves to 192.168.1.x (DNS rebinding)
Decimal encoding http://2130706433/ (2130706433 decimal = 127.0.0.1)
URL shorteners http://short.ly/x → http://internal-host/
Redirect chain Attacker server returns 301 redirect to 169.254.169.254
IPv6 http://[::ffff:169.254.169.254]/
URL scheme abuse file:///etc/passwd, gopher://localhost:6379/_FLUSHALL
CNAME to private IP Attacker controls DNS; CNAME maps to internal IP
IMDSv1 header bypass Some filters check Host but not X-Forwarded-Host

    App->>App: Filter check: 1.2.3.4 is external, OK

    App->>Attacker: DNS resolve attacker.com? (refetch)
    Attacker-->>App: TTL=1, A=192.168.1.1 (internal IP)

    Internal-->>App: Internal data
    App-->>Attacker: Returns internal data

Blind SSRF

Sometimes the application makes the internal request but does not return the response to the user (blind SSRF). Detection methods:

  • Out-of-band DNS/HTTP callback (e.g., Burp Collaborator, interactsh)
  • Timing-based detection (internal services respond faster or slower)
  • Error message differences when internal vs external hosts are hit
flowchart TD
    A[User provides URL] --> B[Server fetches URL internally]
    B --> C{Response returned to user?}
    C -- Yes --> D[Regular SSRF - data exfiltrated directly]
    C -- No --> E[Blind SSRF]
    E --> F[Use OOB interaction detector]
    F --> G[Server reaches out-of-band endpoint]
    G --> H[DNS/HTTP callback received by attacker]
    H --> I[Confirms SSRF exists; pivot to extract data via side channels]

IMDSv2 — The AWS Defense

AWS IMDSv2 adds a session token requirement to every metadata request:

# IMDSv1 - trivially accessible via SSRF with one request
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/role-name

# IMDSv2 - requires two requests; SSRF must follow redirects AND send PUT first
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/

Simple SSRF tools (basic fetch() calls) do not follow this two-step pattern; they fail to get the token, so the metadata API returns 401. The attacker must use a more sophisticated SSRF exploit that sends PUT first and follows redirects — much harder to execute.

Safe SSRF Mitigation Library Pattern

import ipaddress
import socket
from urllib.parse import urlparse

ALLOWED_SCHEMES = {'https', 'http'}
BLOCKED_PREFIXES = ['169.254.', '10.', '192.168.', '172.16.', '172.17.', '172.18.', '172.19.',
                    '172.20.', '172.21.', '172.22.', '172.23.', '172.24.', '172.25.', '172.26.',
                    '172.27.', '172.28.', '172.29.', '172.30.', '172.31.']

def safe_fetch(url):
    parsed = urlparse(url)

    # 1. Validate scheme
    if parsed.scheme not in ALLOWED_SCHEMES:
        raise ValueError("Blocked scheme")

    # 2. Resolve DNS
    try:
        ip = socket.gethostbyname(parsed.hostname)
    except socket.gaierror:
        raise ValueError("Could not resolve hostname")

    # 3. Block private IPs
    addr = ipaddress.ip_address(ip)
    if addr.is_private or addr.is_loopback or addr.is_link_local or addr.is_reserved:
        raise ValueError("Blocked private IP range")

    # 4. Re-resolve at fetch time (use a DNS resolver that blocks private IPs)
    return requests.get(url, timeout=5, allow_redirects=False)

Top 3 Critical Mitigations

  1. Enforce AWS IMDSv2 on all EC2 instances and equivalent on GCP/Azure. The cloud metadata service is the single highest-impact SSRF target. IMDSv2's session-oriented token requirement breaks simple SSRF exploitation by requiring a state-sensitive two-step interaction. Enable IMDSv2 on all instances; set HttpTokens: required in instance metadata options.
  2. Validate outbound URL destinations against a strict allowlist; block RFC-1918 and loopback ranges. Never allow the server to make requests to 10.x, 172.16-31.x, 192.168.x, 127.x, 0.x, or 169.254.x as a result of user input. An allowlist of known external APIs is safer than a blocklist of private ranges. Also block file://, gopher://, and dict:// schemes.
  3. Implement network-level egress filtering. Deny the application server's outbound access to metadata services and internal admin ports at the network level (VPC security groups, iptables, network policies). Defense-in-depth means that even if SSRF validation is bypassed, the network route is blocked.

Key Takeaways

  • SSRF causes the server to make HTTP requests to attacker-controlled destinations, pivoting into internal networks and cloud metadata services.
  • The most critical SSRF target in AWS is the EC2 IMDSv1 metadata service — it provides temporary IAM credentials with no authentication.
  • IMDSv2 mitigates metadata SSRF by requiring session-oriented token exchange; enable it on all EC2 instances.
  • DNS rebinding, localhost aliases, decimal encoding, and URL redirect chains are common SSRF filter bypass techniques.
  • Blind SSRF requires out-of-band detection but can have the same impact as regular SSRF.
  • Defense layers: URL allowlist + private IP blocklist + DNS resolver validation + network egress filtering + IMDSv2.
  • file://, gopher://, and dict:// schemes extend SSRF beyond HTTP to filesystem access and raw TCP interactions.
