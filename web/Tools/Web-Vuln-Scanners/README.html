Web Vulnerability Scanners

Category: Tools — Automated Discovery and Exploitation
Applies To: Security testing, penetration testing, bug bounty, DevSecOps CI/CD scanning

Overview

Web vulnerability scanners are tools that automate the discovery, testing, and often exploitation of security vulnerabilities in web applications. They range from passive reconnaissance tools (crawling and fingerprinting) to active scanners (injecting payloads to trigger and confirm vulnerabilities) to purpose-built exploitation frameworks (sqlmap for SQL injection). At Google scale, automated scanning is a required component of the security pipeline — manual testing alone cannot cover the surface area of constantly-deployed microservices and thousands of API endpoints. Understanding how these tools work is equally important for attackers using them and defenders detecting and defending against automated scanning.

Core Tool Categories

Category Tools Purpose
Full web app scanners Burp Suite Pro, OWASP ZAP, Nikto, Nessus Web Broad vulnerability discovery across OWASP Top 10

Tool Deep Dives

1. SQLMap

Purpose: Fully automated SQL injection detection and database extraction.

Key Capabilities:

Feature Description
Detection Automatically detects injectable parameters across GET, POST, cookie, headers
Techniques Tests all 5 SQLi types: error-based, union-based, boolean-blind, time-blind, out-of-band
Extraction Dumps entire database: schema, tables, columns, rows
OS interaction On vulnerable DBMS: --os-shell (if permissions allow), read/write files
Authentication Supports form-based auth, cookie auth, HTTP basic auth
Evasion Tamper scripts to bypass WAFs: case randomization, whitespace insertion, char encoding

Example Commands:

# Basic GET parameter injection
sqlmap -u "https://target.com/search?q=1" --dbs

# POST body injection
sqlmap -u "https://target.com/login" --data "username=admin&password=test" --dbs

# Injection in Cookie header
sqlmap -u "https://target.com/" --cookie "session=abc123" --dbs

# Dump specific database
sqlmap -u "https://target.com/search?q=1" -D target_db --tables
sqlmap -u "https://target.com/search?q=1" -D target_db -T users --dump

# OS-level interaction (if DB has file/exec privileges)
sqlmap -u "https://target.com/search?q=1" --os-shell

# WAF bypass with tamper scripts
sqlmap -u "https://target.com/search?q=1" --tamper="space2comment,randomcase,charencode"

# Blind injection with threading
sqlmap -u "https://target.com/search?q=1" --technique=T --threads=10 --dbms=mysql

How SQLMap Detects Injection:

  1. Sends baseline request; records response time and content hash
  2. Injects payloads for each technique sequentially
  3. Compares response: error text, response time delta, content difference, or OOB callback
  4. If any technique confirms injection, classifies the vulnerability type
  5. Proceeds to extract database schema and data automatically
flowchart TD
    A[sqlmap started with target URL] --> B[Crawl and identify injectable parameters]
    B --> C[For each parameter: inject test payloads]
    C --> D{Response shows injection?}
    D -- Error message changed --> E[Error-based confirmed]
    D -- Response size/content diff --> F[Boolean-blind confirmed]
    D -- Response timing delta >= threshold --> G[Time-blind confirmed]
    D -- OOB DNS callback received --> H[Out-of-band confirmed]
    E --> I[Dump database schema]
    F --> I
    G --> I
    H --> I
    I --> J[Extract tables and rows automatically]

2. Burp Suite

Purpose: Comprehensive web application penetration testing platform; both intercepting proxy and active scanner.

Key Components:

Component Purpose
Proxy Intercept, inspect, and modify HTTP/HTTPS traffic between browser and server
Repeater Manually modify and replay individual requests
Intruder Automated fuzzing engine for payloads, brute force, and enumeration
Scanner (Pro) Active vulnerability scanner: SQLi, XSS, XXE, SSRF, path traversal, IDOR
Decoder Encode/decode data in base64, URL encoding, HTML encoding, etc.
Comparer Diff two responses or requests
Collaborator Out-of-band service to detect blind SSRF, blind SQLi, blind XSS (like a private DNS/HTTP listener)
Extensions (BApp Store) Hundreds of community extensions: active scan checks, passive detectors, auth handlers

Workflow:

flowchart TD
    A[Configure browser to use Burp Proxy port 8080] --> B[Browse application normally]
    B --> C[Proxy captures all HTTP/S requests and responses]
    C --> D[Send interesting requests to Repeater for manual testing]
    C --> E[Send parameter-rich requests to Scanner for active scan]
    C --> F[Send auth endpoints to Intruder for brute force / fuzzing]
    D --> G[Manually test for SQLi, XSS, IDOR, auth bypass]
    E --> H[Scanner reports vulnerabilities with evidence]
    F --> I[Intruder identifies valid credentials or valid IDs]

Burp Collaborator for Blind Vulnerabilities:

Burp Collaborator is a hosted OOB interaction server. When testing for blind SQLi, SSRF, or blind XSS, Burp generates a unique payload pointing to the Collaborator domain. If the target server makes a DNS lookup or HTTP request to the Collaborator domain, Burp records it and reports the blind vulnerability. This is essential because blind vulnerabilities leave no visible trace in the response.

3. OWASP ZAP (Zed Attack Proxy)

Purpose: Free, open-source web app scanner; excellent for CI/CD integration and beginners.

Key Features:

Feature Description
Active Scan Tests for OWASP Top 10 vulnerabilities
Passive Scan Analyzes traffic without sending additional requests; lower risk
Fuzzer Similar to Burp Intruder; parameter fuzzing
Spider / Ajax Spider Crawls application to discover endpoints
API Import Accepts OpenAPI/Swagger specs to auto-generate test cases
CI/CD Mode Headless mode with Docker for pipeline integration
HUD Browser extension overlay for in-browser scanning

ZAP in CI/CD:

# GitHub Actions example
- name: Run ZAP baseline scan
  uses: zaproxy/action-baseline@v0.7.0
  with:
    target: "https://app.example.com"
    rules_file_name: ".zap/rules.tsv"
    cmd_options: "-a -j" # -a: ajax spider; -j: active scan

4. Nikto

Purpose: Fast web server misconfiguration and known vulnerability scanner.

What Nikto Tests:

Test Category Examples
Server banners Reveals software version in headers
Default files /etc/passwd exposure, default admin pages, test files
Dangerous files Backup files (admin.php.bak), config files
Outdated software Apache/nginx/IIS version with known CVEs
HTTP methods TRACE, DELETE enabled
Default credentials Checks common admin/admin, root/root on web interfaces
Cookie flags Missing HttpOnly, Secure flags
Directory listing Enabled directory indexing 
# Basic Nikto scan
nikto -h https://target.com

# With authentication
nikto -h https://target.com -id admin:password

# Save to HTML report
nikto -h https://target.com -Format html -output results.html

# Scan non-standard port
nikto -h https://target.com -p 8443

5. Nuclei

Purpose: Fast, template-driven vulnerability scanner for known CVEs and misconfigurations.

Key Concept: Nuclei uses YAML templates. Any researcher can write a template for a known CVE, and nuclei runs that template against any target at web scale.

# Example Nuclei template: detect exposed .env files
id: exposed-env-file
info:
  name: Exposed .env file
  severity: high
requests:
  - method: GET
    path:
      - "{{BaseURL}}/.env"
    matchers-condition: and
    matchers:
      - type: word
        words:
          - "APP_KEY"
          - "DB_PASSWORD"
      - type: status
        status:
          - 200

Use Cases:

# Scan for common CVEs
nuclei -u https://target.com -t cves/

# Scan for misconfigurations
nuclei -u https://target.com -t misconfigurations/

# Scan a list of targets
nuclei -l targets.txt -t technologies/ -t exposures/

# Run specific severity templates
nuclei -u https://target.com -severity critical,high

6. Directory and Content Enumeration Tools

gobuster

# Directory enumeration
gobuster dir -u https://target.com -w /wordlists/common.txt

# DNS subdomain enumeration
gobuster dns -d example.com -w /wordlists/subdomains.txt

# Virtual host discovery
gobuster vhost -u https://example.com -w /wordlists/vhosts.txt

FFUF (Fuzz Faster U Fool)

# Directory fuzzing
ffuf -u https://target.com/FUZZ -w /wordlists/common.txt

# Parameter fuzzing
ffuf -u https://target.com/api?FUZZ=value -w /wordlists/params.txt

# Multiple parameter fuzzing
ffuf -u https://target.com/api?user=FUZZ&action=W2 -w users.txt:FUZZ -w actions.txt:W2

# Filter by response size/status
ffuf -u https://target.com/FUZZ -w wordlist.txt -fs 1234 -mc 200,301,302

Detecting and Defending Against Scanner Traffic

Scanner Detection Signals

Signal Characteristic
Request rate Hundreds or thousands of requests per minute from single IP
Path patterns Sequential paths, dictionary wordlist patterns (/admin, /backup, /test)
User-Agent Nikto, sqlmap, Burp Scanner have recognizable UAs (or try to fake them)
Payload patterns SQLi payloads (' OR '1'='1), XSS payloads (<script>), traversal sequences (../)
404 rate Scanners generate high 404 error rates
Parameter probing Same endpoint with thousands of different parameter values
HTML form changes Intruder-based attacks modify expected form fields

Mitigation Against Automated Scanning

Control Description
WAF with rate limiting Block scanning rate patterns; block known scanner signatures
CAPTCHA on sensitive forms Anti-automation control on login, registration
IP reputation blocking Block IPs in scanner/proxy reputation lists (Cloudflare, Akamai)
Rate limiting per endpoint 429 Too Many Requests when patterns detected
Honeypot endpoints Add fake endpoints to trigger alert when visited (/admin-backup.php)
Log monitoring and alerting Alert on high 404 rate, SQLi payload patterns in WAF logs
Realistic error responses Return 404 for both "not found" and "not authorized"; reduces information leakage 
flowchart TD
    A[Incoming HTTP requests] --> B[WAF checks]
    B --> C{Known scanner UA or payload?}
    C -- Yes --> D[Block request; log event]
    C -- No --> E{Rate limit exceeded?}
    E -- Yes --> F[429 response; add to watch list]
    E -- No --> G{IP in threat intelligence list?}
    G -- Yes --> H[Challenge or block]
    G -- No --> I[Forward to application]
    I --> J{Honeypot endpoint hit?}
    J -- Yes --> K[Alert security team; investigate IP]
    J -- No --> L[Normal application response]

What Can Go Wrong (For Organizations)

# Issue Impact Likelihood
1 Scanner finds SQLi before internal team does Database compromise by external attacker High
2 Nikto finds exposed backup files / .env Database creds and API keys exposed High
3 Directory brute force reveals admin panel Admin access attempts on exposed panel High
4 Nuclei finds known CVE in outdated library Automated exploitation of known vulnerability High
5 Burp Collaborator detects blind SSRF Internal service access proven Med
6 Scanner causes resource exhaustion Active scanner overwhelms staging or low-capacity service Med
7 Automated scan on production Scanner traffic degrades production performance Med

Key Takeaways

  • web vulnerability scanners automate discovery of OWASP Top 10 and many more vulnerability classes.
  • sqlmap automates full SQLi exploitation including database extraction and OS interaction; parameterized queries are the only defense.
  • Burp Suite is the industry-standard proxy and scanner; Collaborator enables blind vulnerability detection.
  • Nuclei provides template-based scanning that can run known CVE checks across entire asset inventories at scale.
  • Directory enumeration tools (gobuster, FFUF) discover hidden endpoints and files that are not linked from the UI.
  • Scanners should be run in authorized testing only; unauthorized scanning is illegal in most jurisdictions.
  • Defending against scanners requires WAF rate limiting + payload detection + honeypot endpoints + real-time alerting — but fixing the underlying vulnerabilities is the only complete defense.
